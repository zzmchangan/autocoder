# 《gRPC 设计与实现深入浅出:为什么把方法调用变成一条流》—— 目录与导读

> 一本写给"天天用 gRPC、甚至翻过 grpc 源码,却总觉得一知半解"的人的小书。
>
> **一句话主旨**:如何让一次跨网络、跨语言的方法调用像本地调用一样自然,又不假装网络不存在?——核心是把一次方法调用变成 HTTP/2 上的一条可控的"流"。
>
> **二分法**(迷路时回到它):**协议层**(把方法调用编码成网络上流动的字节:IDL/HTTP/2/HPACK/flow control) vs **框架层**(把网络字节变回可调用、可治理的方法:call/filter/channel/resolver/balancer/retry)。
>
> **主比喻**:直球为主、比喻点睛。开篇用"打一次电话"做一次性点睛(拨号选号 = resolver、选线路 = balancer、通话 = stream、保活 = keepalive、挂机 = 连接管理),全书不再建立贯穿重比喻。

每章一行:**一句话钩子** —— 技巧标签 —— 二分法归属(`协议` / `框架` / `治理` / `可用` / `衔接` / `总览`)。

## 第 0 篇 · 开篇:为什么需要 gRPC

- [P0-01 · 第一性原理:为什么需要 gRPC](P0-01-第一性原理-为什么需要gRPC.md) —— 远程调用本该像本地一样自然,可"假装成本地"会掩盖网络的不可靠;gRPC 用"IDL 契约 + HTTP/2 流 + protobuf 编码"三件套,把网络的异步性暴露成一等公民的"流"。 —— 远程调用的根本矛盾 + 流的第一性 —— `总览`

## 第 1 篇 · 契约:用 protobuf 定义跨语言契约

> 这一篇立起"跨语言"的根:一份语言无关的契约。protobuf 是后面一切的基础,**建议顺序读**。

- [P1-02 · IDL 与 .proto:语言无关的契约](P1-02-IDL与proto-语言无关的契约.md) —— 跨语言调用的根,是先有一份"语言无关"的描述:protobuf 用**字段号**而非字段名定位字段,所以加字段不破坏老代码。 —— 字段号定位 + 向前/向后兼容 —— `协议`
- [P1-03 · protobuf 编码:为什么又小又快](P1-03-protobuf编码-为什么又小又快.md) —— JSON 慢在文本、大在冗余;protobuf 用 varint + tag + wire format 把每个字段压到最少字节,还天生前后兼容。 —— varint + zigzag + packed repeated + unknown fields —— `协议`
- [P1-04 · 代码生成与 Stub](P1-04-代码生成与Stub.md) —— 一份 `.proto` 怎么变出 C++/Java/Go 的 Stub,还一次给齐 sync / async / callback 三套 API。 —— protoc 插件 + 三模 API 生成 —— `框架`

## 第 2 篇 · 传输:把方法调用变成 HTTP/2 上的流(全书最硬)

> 这一篇是 gRPC C++ core 的心脏:它**自己用 C 实现了一整套 HTTP/2**。是本书区别于 grpc-go / grpc-java(它们复用语言库 HTTP/2)的核心资本。HTTP/2 协议层招牌密集,**建议顺序读**。

- [P2-05 · HTTP/2 基础:为什么是 gRPC 的天然底座](P2-05-HTTP2基础-为什么是天然底座.md) —— 为什么 gRPC 放弃 HTTP/1.1(队头阻塞)和私有 TCP(割裂生态),选 HTTP/2:二进制分帧 + 流 + 多路复用,把一次"请求-响应"升级成"一条流"。 —— 多路复用消队头阻塞 + 流是第一性 —— `协议`
- [P2-06 · chttp2 transport 全貌:一条流的一生](P2-06-chttp2-transport全貌-一条流的一生.md) —— 一个 TCP 连接怎么同时跑成千上万个调用?stream 是身份、stream 表是花名册、写循环是班车。 —— stream 表 + 写循环攒批发 —— `协议/衔接`
- [P2-07 · HPACK:头部压缩的招牌](P2-07-HPACK-头部压缩的招牌.md) —— 每次调用重复的 `:path` / `content-type` 凭什么压到几乎零字节?静态表 + 动态表 + Huffman 三重压缩。 —— 静态/动态表 + Huffman + 敏感头 never-indexed —— `协议(招牌)`
- [P2-08 · gRPC framing 与帧解析](P2-08-gRPC-framing与帧解析.md) —— protobuf 消息怎么塞进 DATA 帧?读侧 parsing 状态机、写侧 writing 攒批,各帧各司其职。 —— 5 字节 Length-Prefixed + 双状态机 + SETTINGS 协商 —— `协议`
- [P2-09 · 流量控制:不淹不饿](P2-09-流量控制-不淹不饿.md) —— 生产者比消费者快会淹死连接;HTTP/2 双层 window 给信用,gRPC 再叠 BDP 估计和 ping 限速。 —— 双层 window + BDP 自适应 + ping 防滥用 —— `协议(招牌)`

## 第 3 篇 · 调用:一条流的生命周期

- [P3-10 · call 与 completion queue:经典异步模型](P3-10-call与completion-queue-经典异步模型.md) —— gRPC 的招牌异步:所有异步操作都从 completion queue 出口,一个线程就能服务海量调用。 —— CQ 事件队列 + 三模 API + Next/Pluck —— `框架`
- [P3-11 · filter stack:可插拔架构的灵魂](P3-11-filter-stack-可插拔架构的灵魂.md) —— 鉴权 / 日志 / 压缩怎么不侵入业务?一次调用穿过一串 filter,新架构还能把它们编译期融合成一条流水线。 —— filter 责任链 + filter fusion 编译期合并 —— `框架(招牌)`
- [P3-12 · 四种调用模式与流状态机](P3-12-四种调用模式与流状态机.md) —— unary / server-stream / client-stream / bidi 本质都是流,只是谁先发、谁多发;call 把这条流编排成 send/recv 三段。 —— send/recv 三段 + call spine 的 promise 编排 —— `框架`

## 第 4 篇 · 客户端治理:地址、负载均衡、重试

> 微服务场景下 gRPC 的看家本领。调用怎么发出去、发去哪、失败了怎么办。

- [P4-13 · Resolver:名字到地址](P4-13-Resolver-名字到地址.md) —— `dns:///user-svc` 怎么变成地址列表?解析器异步出结果,名字服务还顺带把 LB 策略和 retry 配置下发。 —— URI scheme + 异步解析 + service config 下发 —— `治理`
- [P4-14 · SubChannel:一条后端连接](P4-14-SubChannel-一条后端连接.md) —— 一个后端怎么抽象成可建连 / 复用 / 重连的 SubChannel,多个 channel 还能共享同一条后端连接省连接数。 —— SubChannel 复用池 + 连接状态机 + 退避 —— `治理/衔接`
- [P4-15 · 负载均衡:挑这条流去哪](P4-15-负载均衡-挑这条流去哪.md) —— 多个 SubChannel 这次挑哪个?round_robin / 加权 / 一致性哈希各有用场,Picker 用无锁 fast path 每次原子挑一个。 —— Picker 无锁 fast path + ring_hash 一致性哈希 —— `治理(招牌)`
- [P4-16 · 重试与对冲](P4-16-重试与对冲.md) —— 失败了什么时候重试、怎么退避?后端连续失败时令牌桶自动降重试防雪崩,急了还能对冲并发发第二个。 —— retry throttle 令牌桶 + hedging 对冲 + transparent retry —— `治理(招牌)`

## 第 5 篇 · 生产可用:keepalive、健康检查、安全

- [P5-17 · keepalive:探测死连接](P5-17-keepalive-探测死连接.md) —— 云负载均衡器空闲超时让连接"假活",keepalive ping 探活,但服务端要限速防客户端打爆。 —— keepalive ping/pong + ping 限速防滥用 —— `可用`

- [P5-19 · 安全:TLS / mTLS / ALTS / tsi](P5-19-安全-TLS-mTLS-ALTS-tsi.md) —— 跨网络怎么认证加密?tsi 抽象屏蔽 SSL/ALTS,channel creds 管连接级 TLS、call creds 叠调用级 OAuth。 —— tsi 抽象 + ALTS + channel/call creds 分离 + handshaker —— `可用`
