# 第 2 篇 · 第 9 章 · 流量控制:不淹不饿

> **核心问题**:gRPC 把海量并发调用塞进同一条 TCP 连接(P2-05/06),如果生产者(发送方)比消费者(接收方)快,未消费的字节会堆在接收方内存里,直到 OOM 崩溃——这叫"**淹死**"。反过来,如果流控太保守,窗口给小了,网络明明空着却发不出去,吞吐上不去——这叫"**饿死**"。HTTP/2 给了**双层 window**(连接级 + 流级)的信用模型来防淹死,但 RFC 默认窗口是静态的,扛不住高带宽延迟积(BDP)的网络。gRPC 在 HTTP/2 流控之上叠了哪些工程,既做到"不淹死"又做到"不饿死"?具体三个问题:① 双层 window 为什么是两层,不是一层或三层?② gRPC 的 BDP 估计怎么用 PING 测 RTT、用字节数估带宽,从而自动调窗?③ keepalive ping 怎么既探活又防被滥用打爆服务端 CPU?

> **读完本章你会明白**:
> 1. HTTP/2 双层 window(连接级 + 流级)的**信用模型**到底怎么运作:发送方扣 window,接收方消费完才回 WINDOW_UPDATE 加信用——为什么是两层(防一条慢流饿死整连接,也防整连接背压波及无辜流),以及为什么两层各自独立扣减。
> 2. gRPC 的 **BDP 估计**(bandwidth-delay product,带宽时延积)的直觉:窗口大小要 ≈ 在途字节数 ≈ 带宽 × RTT 才能打满链路;gRPC 怎么用 PING 测 RTT、用 DATA 帧累积字节数估带宽,从而**自动**把窗口从默认 64 KB 推到 MB 级。
> 3. BDP 估计出的窗口怎么落地——**通过 SETTINGS_INITIAL_WINDOW_SIZE 调整,而不是 WINDOW_UPDATE**。这两条路径的区别是本章最容易混的点。
> 4. gRPC 的三个流控 adjunct 策略:**write_size_policy**(每次攒批多少字节)、**ping_rate_policy**(出站 PING 限速)、**ping_abuse_policy**(入站 PING 防滥用,strike 制度可关连接)——它们防的是 keepalive 滥用打爆服务端 CPU 这种攻击面。
> 5. 经典版 `flow_control.cc` 的流控逻辑与 Promise 版 `http2_client_transport.cc` 的实现差异——尤其是 Promise 版 BDP 还没接上、用固定 8 秒定时器替代的诚实现状。

> **如果一读觉得太难**:先只记住三件事——① **双层 window = 连接级总额度 + 每条流独立额度**,发送 DATA 两层都扣,接收方消费完才给两层分别加信用;② **BDP = 带宽 × RTT**,窗口设成 BDP 才能打满链路,gRPC 用 PING 测 RTT、用 DATA 字节估带宽,自动把窗口从 64 KB 推到 MB 级;③ **gRPC 在流控之上加了 write_size_policy(攒批上限)+ ping_rate/abuse_policy(防 ping 滥用)**,这是 HTTP/2 之上 gRPC 的招牌工程。具体怎么算、怎么落地,是给"想看穿实现"的读者准备的。

## 〇、一句话点破

> **HTTP/2 流控是一个"信用制双层 window":发送方发 DATA 扣信用,接收方消费完才回 WINDOW_UPDATE 加信用,两层(连接级 + 流级)独立,防一条慢流饿死整连接、也防整连接背压波及无辜流。gRPC 在它之上做了三件 RFC 没规定的事:① 用 BDP 估计(带宽 × RTT)自动调窗口,从默认 64 KB 推到 MB 级,让高 BDP 链路也能打满;② write_size_policy 动态调攒批上限,找延迟 vs 吞吐的甜点;③ ping_rate/abuse_policy 限速 PING,防 keepalive 滥用打爆服务端 CPU。**

这是结论,不是理由。本章倒过来拆:先让读者"感到痛"——为什么没有流控会被淹死;再讲 HTTP/2 双层 window 的信用模型怎么防淹死;然后讲双层窗口在高 BDP 网络下会"饿死",gRPC 的 BDP 估计怎么救;接着讲 ping 滥用的攻击面和限速;最后诚实交代经典版与 Promise 版的实现差异。

## 一、为什么必须流控:不淹死的硬道理

讲双层 window 之前,先让读者真的"感到痛"。设想一个没有流控的世界。

### 一个会淹死连接的场景

客户端调服务端的 `StreamLogs(request, response_stream)`——一个 server-streaming 调用,服务端会持续推送日志。假设:

- 服务端在万兆机房,网卡 10 Gbps,产日志的速度 1 GB/s。
- 客户端在偏远机房,通过一条 100 Mbps 的专线连过来,且客户端把日志写磁盘,磁盘 IO 只能吃 50 MB/s。

服务端产 1 GB/s,客户端吃 50 MB/s——服务端比客户端快 20 倍。如果没有任何流控,服务端发了 1 GB,客户端只来得及处理 50 MB,**剩下 950 MB 全堆在客户端的内核 socket 缓冲、gRPC 内存池、应用队列里**。几秒后客户端 OOM 崩溃。

```
   没有流控的世界:生产者淹死消费者
   ┌──────────────┐                                  ┌──────────────┐
   │  服务端       │ ──1 GB/s──────────────────────→ │  客户端       │
   │  产 1 GB/s    │ ──万兆网卡,飞快发──→           │  吃 50 MB/s   │
   └──────────────┘                                  └──────────────┘
                                                       ↓
                                              ┌──────────────────┐
                                              │ 950 MB 堆积/OOM  │
                                              └──────────────────┘
```

### TCP 自己有流控,为什么还不够

TCP 自己有流控——TCP 的滑动窗口(TCP window)。但 TCP 窗口的粒度是**整条连接**:

- 服务端的字节发给客户端,TCP 层的 ACK 会带"客户端还能收 N 字节"的窗口信息。
- 客户端 TCP 缓冲满了,会回 window=0,服务端 TCP 停止发送。

所以"淹死客户端的 TCP 缓冲"这件事,TCP 自己挡得住。但 gRPC 跑在 HTTP/2 上,**HTTP/2 把一条 TCP 连接切成多条并发流**,每条流是一个独立的 RPC。TCP 窗口管的是"字节进了 TCP 缓冲没有",但**字节从 TCP 缓冲读到应用层、由应用层消费,这中间还有一道**——这道 HTTP/1.1 不管(HTTP/1.1 一连接一请求,TCP 窗口够了),HTTP/2 必须管(一连接多请求,某个慢请求堆积的字节会占满 TCP 缓冲,影响其他快请求)。

> **不这样会怎样**:如果只有 TCP 流控,没有 HTTP/2 流控,会发生两件事。① **慢请求饿死快请求**:一条 HTTP/2 连接上 stream 1(server-streaming 日志,客户端慢)堆积 1 GB 在客户端 TCP 缓冲,把 TCP 缓冲占满 → 服务端的 TCP 窗口变 0 → 服务端连 stream 2(一个普通 GetUser,快请求)的字节也发不出来。② **应用层 OOM**:TCP 缓冲虽然满了,但客户端内核把字节 read 上来交给 gRPC,gRPC 又交给应用层——如果应用层处理慢,字节在 gRPC 的内存池里堆积,绕过了 TCP 流控,最终 OOM。

HTTP/2 的双层 window 就是为了解决这两件事——它把流控从 TCP 层(整条连接)提升到 HTTP/2 层(连接级 + 流级),让"慢流不影响快流"成为可能。

## 二、双层 window 的信用模型:HTTP/2 的核心发明

### 双层 window 的物理形态

HTTP/2 规定(对得上 RFC 9113 §6.9):流控有**两层 window**,各自独立:

```
   HTTP/2 双层 window(对得上 RFC 9113 §6.9)
   ┌─────────────────────────────────────────────────────────┐
   │  连接级 window(整条连接的总额度)                        │
   │  remote_window_(transport)  /  announced_window_(transport)│
   ├─────────────────────────────────────────────────────────┤
   │  流级 window:每条流独立                                  │
   │  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐ │
   │  │ stream 1 │  │ stream 3 │  │ stream 5 │  │  ...     │ │
   │  │ window   │  │ window   │  │ window   │  │          │ │
   │  └──────────┘  └──────────┘  └──────────┘  └──────────┘ │
   └─────────────────────────────────────────────────────────┘
```

每层 window 都是一对:**remote window**(对方的,我能发多少)+ **announced window**(我告诉对方能收多少)。HTTP/2 的初始 window 大小由 SETTINGS 的 `INITIAL_WINDOW_SIZE` 协商,RFC 9113 默认是 **65535 字节**(64 KB)。

### gRPC 的物理实现:TransportFlowControl + StreamFlowControl

在 gRPC 的 `flow_control.h` 里,这两层 window 分别是两个类的成员。

**连接级** `TransportFlowControl`(在 [`flow_control.h`](../grpc/src/core/ext/transport/chttp2/transport/flow_control.h#L202-L485))的关键字段(行 469-477):

```cpp
int64_t remote_window_ = kDefaultWindow;        // 对方的 window(我能发多少)
int64_t announced_window_ = kDefaultWindow;     // 我告诉对方能收多少
uint32_t acked_init_window_ = kDefaultWindow;   // 对方 ACK 过的 INITIAL_WINDOW_SIZE
uint32_t sent_init_window_ = kDefaultWindow;    // 我发出去的 INITIAL_WINDOW_SIZE
```

`kDefaultWindow = 65535`(行 50),对得上 RFC 9113。

**流级** `StreamFlowControl`(在 [`flow_control.h`](../grpc/src/core/ext/transport/chttp2/transport/flow_control.h#L489-L650))的关键字段(行 643-647):

```cpp
TransportFlowControl* const tfc_;
int64_t min_progress_size_ = 0;                 // 应用层需要的最小进度字节
int64_t remote_window_delta_ = 0;               // 流级出站 window(相对 transport 的 delta)
int64_t announced_window_delta_ = 0;            // 流级入站 window(相对 transport 的 delta)
std::optional<int64_t> pending_size_;           // 已收但应用层还没读的字节
```

注意一个**关键设计**:流级 window 不是绝对值,而是**相对 transport 的 delta**——流级出站 window = `remote_window_delta_ + peer_settings.initial_window_size()`(对得上 [`flow_control_manager.h`](../grpc/src/core/ext/transport/chttp2/transport/flow_control_manager.h#L67-L75) 的 `GetStreamFlowControlTokens`)。为什么这样?因为 `INITIAL_WINDOW_SIZE` 可以中途改(SETTINGS 协商),如果流级 window 存绝对值,每次 `INITIAL_WINDOW_SIZE` 变了要遍历所有流改值;存 delta,只要 transport 的 `initial_window_size` 一改,所有流的"有效 window"自动跟着变——这是把 O(N) 的更新变成 O(1) 的工程巧思。

### 信用模型:发送方扣 window,接收方消费完才加信用

HTTP/2 流控的核心是**信用制**(credit-based):发送方每发一个 DATA 帧,**两层 window 都扣**;接收方消费完(应用层 read 了),才回 WINDOW_UPDATE 给对应层加信用。

看 gRPC 源码:`StreamFlowControl::OutgoingUpdateContext::SentData`(在 [`flow_control.h`](../grpc/src/core/ext/transport/chttp2/transport/flow_control.h#L571-L574) 的实现,逻辑在 OutgoingUpdateContext):

```cpp
// 逻辑简化(实际在 StreamFlowControl::OutgoingUpdateContext 内)
void SentData(int64_t outgoing_frame_size) {
  tfc_upd_.StreamSentData(outgoing_frame_size);   // 连接级 remote_window_ -= size
  sfc_->remote_window_delta_ -= outgoing_frame_size;  // 流级 remote_window_delta_ -= size
}
```

**两层同时扣**。这意味着,一条流要发数据,必须同时满足"连接级 window 还有"和"流级 window 还有"——任一层为 0,这条流就被卡住。

接收方收到 DATA,反向操作——`StreamFlowControl::IncomingUpdateContext::RecvData`(在 [`flow_control.cc`](../grpc/src/core/ext/transport/chttp2/transport/flow_control.cc#L146-L163)):

```cpp
absl::Status StreamFlowControl::IncomingUpdateContext::RecvData(
    int64_t incoming_frame_size) {
  return tfc_upd_.RecvData(incoming_frame_size, [this, incoming_frame_size]() {
    int64_t acked_stream_window =
        sfc_->announced_window_delta_ + sfc_->tfc_->acked_init_window();
    if (incoming_frame_size > acked_stream_window) {
      return absl::InternalError("...FLOW_CONTROL_ERROR...");   // 超流控,连接错
    }
    // 两层都扣(我告诉对方能收的额度)
    tfc_upd_.UpdateAnnouncedWindowDelta(&sfc_->announced_window_delta_,
                                        -incoming_frame_size);
    sfc_->min_progress_size_ -= std::min(sfc_->min_progress_size_,
                                         incoming_frame_size);
    return absl::OkStatus();
  });
}
```

注意:**收数据时,两层 announced_window 也都扣**(我告诉对方能收的额度减少)。等应用层把数据读走了(consume),才考虑回 WINDOW_UPDATE 把额度补回去——这是关键的"消费驱动"。

> **钉死这件事**:HTTP/2 流控是**信用制双层 window**——发送方发 DATA 两层 remote_window 都扣;接收方收 DATA 两层 announced_window 都扣,**消费完**(应用层 read)才回 WINDOW_UPDATE 加信用。任一层 window=0,发送方停止发送。**这是"不淹死"的硬保证**——接收方没消费,信用不回来,发送方自然停。

### 为什么是两层:防饿死与防波及

现在回答最关键的问题——**为什么是两层,而不是一层或三层?**

**一层(只有连接级 window)会怎样**:

```
   一层 window(只有连接级):慢流饿死快流
   ┌──────────────────────────────────────────────────┐
   │  连接级 window = 65535                            │
   │  stream 1(慢流,日志推送)发了 64 KB → window=0  │
   │  stream 2(快流,GetUser)→ 发不出!              │  ← 慢流吃光连接额度
   └──────────────────────────────────────────────────┘
```

如果只有连接级 window,一条慢流(stream 1)把 64 KB 连接额度吃完后,**同连接的快流(stream 2)也被卡住**——因为连接级 window 没了。这就是"一条慢流饿死整连接"。

**加流级 window 解决了它**:

```
   双层 window:流级额度独立,慢流卡自己不卡别人
   ┌──────────────────────────────────────────────────┐
   │  连接级 window = 65535                            │
   │  ┌─────────────────┐  ┌─────────────────┐         │
   │  │ stream 1 window │  │ stream 2 window │         │
   │  │   = 0(慢流卡) │  │   = 64 KB(可发)│         │
   │  └─────────────────┘  └─────────────────┘         │
   │  stream 1 卡(自己的流级 window=0)               │
   │  stream 2 可以发(连接级 + 流级都还有)           │  ← 慢流只卡自己
   └──────────────────────────────────────────────────┘
```

流级 window 让每条流有独立额度,慢流吃光自己的流级 window 后只卡自己,**不影响其他流**——只要连接级 window 还有,其他流照发。

**那为什么还要保留连接级 window**?因为如果只有流级 window,假设一条连接上有 10000 条流,每条流 64 KB,**总额度 640 MB**——接收方要预先准备 640 MB 内存,容易 OOM。连接级 window 给整条连接一个总额度上限(64 KB),保证接收方总内存可控。

**所以两层各有职责**:

- **连接级 window**:限制整条连接的总在途字节,保接收方总内存不爆。
- **流级 window**:让每条流的背压独立,慢流不波及快流。

> **不这样会怎样**:只有连接级 → 慢流饿死快流;只有流级 → 总额度爆炸,接收方 OOM。两层缺一不可。**这是 HTTP/2 流控设计的精髓**——双层各自独立,既保总内存可控,又让流间背压隔离。

### 死锁与饿死:双层 window 的反面教材

双层 window 看起来完美,但它**有潜在的死锁/饿死陷阱**。考虑这个场景:

- 连接级 window = 64 KB,有 1000 条流,每条流级 window = 64 KB。
- 每条流各发 65 字节,1000 条流共发 65 KB,连接级 window 还剩 64 KB - 65 KB < 0?

不,实际上每条流发的时候**两层都扣**:1000 条流各发 65 字节,连接级扣 65 KB,每条流级扣 65 字节。连接级 window 是 64 KB,所以最多发 64 KB / 65 字节 ≈ 1000 条流各发一点。但 1000 条流每条想发 64 KB,总需求 64 MB——连接级 window 只有 64 KB,**99.9% 的流被连接级卡住**。

这不算"死锁",但算"饿死"——连接级 window 太小,海量流的总需求远超它,大量流被卡。这就是为什么静态的 64 KB 默认窗口扛不住高并发高 BDP 网络——需要 BDP 估计动态调窗,下一节拆。

至于真死锁:HTTP/2 双层 window 本身不会死锁(因为没有循环依赖),但如果实现有 bug——比如 WINDOW_UPDATE 丢失或窗口计算错——可能导致"两边都等对方的信用"的假死。gRPC 的实现通过 `DesiredAnnounceSize` 和 `UpdateAction` 的阈值触发(下面 §三 拆),保证 window 不会被"扣到很负还不发 WINDOW_UPDATE"。

> **钉死这件事**:双层 window 是 HTTP/2 流控的核心发明——连接级保总内存可控,流级让背压隔离。静态默认窗口 64 KB 在高 BDP 网络下会"饿死"海量流的总需求,需要 BDP 估计动态调窗。双层本身不会死锁,但实现 bug 可能导致假死,gRPC 用阈值触发保证及时回 WINDOW_UPDATE。

## 三、WINDOW_UPDATE 的触发:不是收到就回,而是消费完才回

讲完了双层 window 的模型,现在看 gRPC 的实现——WINDOW_UPDATE 帧什么时候发出去。

### 朴素直觉:收到 DATA 立刻回 WINDOW_UPDATE?

很多人第一次理解 HTTP/2 流控,会以为"接收方收到 DATA 就立刻回 WINDOW_UPDATE 补信用"。**这是错的**。如果这样,流控就失去意义了——接收方收了 64 KB 立刻回 WINDOW_UPDATE,然后应用层还没处理,下一批 64 KB 又到,又回 WINDOW_UPDATE……字节全堆在接收方内存里,淹死。

正确的语义是:**接收方把 DATA 交给应用层消费后,才回 WINDOW_UPDATE**。消费多少,补多少。

但 gRPC 的实现更精细——它不是"每消费一字节就回一字节 WINDOW_UPDATE"(那 WINDOW_UPDATE 帧太多,本身浪费带宽),而是**攒到一定阈值再回**。

### gRPC 的阈值触发:DesiredAnnounceSize 和 hurry_up_size

gRPC 的流控核心逻辑在 `StreamFlowControl::DesiredAnnounceSize` 和 `UpdateAction`。看源码,在 [`flow_control.cc`](../grpc/src/core/ext/transport/chttp2/transport/flow_control.cc#L363-L405):

```cpp
uint32_t StreamFlowControl::DesiredAnnounceSize() const {
  int64_t desired_window_delta = [this]() {
    if (min_progress_size_ == 0) {        // 没有读者在等
      if (pending_size_.has_value() &&
          announced_window_delta_ < -*pending_size_) {
        return -*pending_size_;           // 补到 pending_size
      } else {
        return announced_window_delta_;   // 不需要补
      }
    } else {                              // 有读者在等
      return std::min(min_progress_size_, kMaxWindowDelta);
    }
  }();
  return Clamp(desired_window_delta - announced_window_delta_, int64_t{0},
               kMaxWindowUpdateSize);
}

FlowControlAction StreamFlowControl::UpdateAction(FlowControlAction action) {
  const int64_t desired_announce_size = DesiredAnnounceSize();
  if (desired_announce_size > 0) {
    FlowControlAction::Urgency urgency = FlowControlAction::Urgency::QUEUE_UPDATE;
    // hurry_up_size:半窗或 8 KB,取大
    const int64_t hurry_up_size = std::max(
        static_cast<int64_t>(tfc_->queued_init_window()) / 2, int64_t{8192});
    if (desired_announce_size > hurry_up_size) {
      urgency = FlowControlAction::Urgency::UPDATE_IMMEDIATELY;
    }
    // 有读者在等 + 已扣到半窗以下 → 立刻发
    if (min_progress_size_ > 0) {
      if (announced_window_delta_ <=
          -static_cast<int64_t>(tfc_->sent_init_window()) / 2) {
        urgency = FlowControlAction::Urgency::UPDATE_IMMEDIATELY;
      }
    }
    action.set_send_stream_update(urgency);
  }
  return action;
}
```

关键阈值是 `hurry_up_size`——`max(queued_init_window / 2, 8 KB)`。**意思是:当接收方消费的字节攒够"半窗"或"8 KB"(取大),才立刻发 WINDOW_UPDATE**。少于这个阈值,只标 QUEUE_UPDATE(下次写循环顺便发)。

为什么是半窗?这是个工程折中:

- **阈值太小**(比如 1 字节就发):WINDOW_UPDATE 帧泛滥,本身吃带宽。
- **阈值太大**(比如窗口空了才发):发送方早就停下来等了,吞吐下降。

半窗是个甜点——发送方在窗口用完前就能收到补给,不会停;同时 WINDOW_UPDATE 帧数量可控(每个流最多每"半窗"发一次)。

连接级 window 的触发逻辑类似,`TransportFlowControl::UpdateAction` 在 [`flow_control.cc`](../grpc/src/core/ext/transport/chttp2/transport/flow_control.cc#L187-L196):

```cpp
FlowControlAction TransportFlowControl::UpdateAction(FlowControlAction action) {
  const int64_t target = target_window();
  const int64_t send_threshold = (target + 1) / 2;   // 半窗
  if (announced_window_ < send_threshold) {
    action.set_send_transport_update(
        FlowControlAction::Urgency::UPDATE_IMMEDIATELY);
  }
  return action;
}
```

同样是半窗阈值。

### Urgency 三级:什么时候立刻发,什么时候顺便发

gRPC 的 `FlowControlAction` 用三级 `Urgency`(在 [`flow_control.h`](../grpc/src/core/ext/transport/chttp2/transport/flow_control.h#L77-L85)):

- `NO_ACTION_NEEDED`:什么都不用做。
- `QUEUE_UPDATE`:下次写循环顺便发(和别的帧一起攒批)。
- `UPDATE_IMMEDIATELY`:**立刻触发一次写循环**(不等别的)。

`UPDATE_IMMEDIATELY` 是关键——它会让 transport 立刻发包,即使本来没打算写。这是为了保证流控信用及时补给,不让发送方卡住。

`FlowControlAction::AnyUpdateImmediately()`(行 100-107)OR 五种 action(stream update / transport update / initial_window update / max_frame_size update / preferred_rx_crypto_frame_size update),只要有一个 `UPDATE_IMMEDIATELY`,就触发立刻写。Promise 版 transport 在这点上有个特殊行为——它会**暂停读循环**(`SetPauseReadLoop`),优先处理写(发 WINDOW_UPDATE),避免"读了一堆 DATA 但来不及发 WINDOW_UPDATE"导致连接级 window 卡死。

> **钉死这件事**:WINDOW_UPDATE 不是"收到 DATA 就回",而是**"消费完 + 攒到半窗或 8 KB"才回**。这个阈值是延迟 vs 带宽的甜点——既不让 WINDOW_UPDATE 帧泛滥,也不让发送方卡住。`UPDATE_IMMEDIATELY` 让 transport 立刻触发写循环,保证信用及时补给。

## 四、双层 window 的天花板:静态窗口在高 BDP 网络下会饿死

到目前为止,双层 window 看起来完美——不淹死、背压隔离、及时补给。但它有个致命的天花板:**静态窗口太小,在高 BDP 链路上会"饿死"吞吐**。

### 什么是 BDP:带宽时延积

**BDP(Bandwidth-Delay Product,带宽时延积)= 链路带宽 × 往返时延(RTT)**。

它有个直观的物理意义:**"要把链路塞满,在途字节数必须 ≈ BDP"**。

```
   一条链路:带宽 1 Gbps,RTT 100 ms
   ┌──────────────────────────────────────────────────────────────┐
   │  客户端 ──→ (字节在飞 100 ms) ──→ 服务端                       │
   │                                                              │
   │  BDP = 1 Gbps × 100 ms = 1 Gb × 0.1 s = 100 Mb ≈ 12.5 MB    │
   │                                                              │
   │  意思:要让这条链路跑满 1 Gbps,在途字节必须有 12.5 MB        │
   │       (因为第一个 ACK 要 100 ms 才回来,这 100 ms 里          │
   │        发送方必须不停发,共发 1 Gbps × 0.1 s = 12.5 MB)       │
   └──────────────────────────────────────────────────────────────┘
```

如果窗口 < BDP,会发生什么?发送方在窗口用完时停下来等 ACK,等 ACK 的时间里链路空着——**吞吐上不去**。

### 静态 64 KB 窗口在高 BDP 链路上的灾难

HTTP/2 默认 INITIAL_WINDOW_SIZE = 64 KB。在跨太平洋链路(带宽 1 Gbps,RTT 100 ms,BDP = 12.5 MB)上:

- 发送方发 64 KB,窗口用完,停下来等 WINDOW_UPDATE。
- 接收方消费完回 WINDOW_UPDATE,但 ACK 要飞 100 ms 回来。
- 这 100 ms 里,发送方什么都不能发——**链路空着**。
- 实际吞吐 ≈ 64 KB / 100 ms = 640 KB/s ≈ 5 Mbps——**远低于链路的 1 Gbps**。

```
   静态 64 KB 窗口在 BDP=12.5 MB 链路上的吞吐
   ┌──────────────────────────────────────────────────────────┐
   │  发送方发 64 KB → 窗口用完,停 → 等 100 ms                │
   │  收到 ACK,补 64 KB → 再发 64 KB → 停 → 等 100 ms         │
   │  ...                                                     │
   │  实际吞吐:64 KB / 100 ms = 640 KB/s(链路能跑 125 MB/s)│
   │                                                          │
   │  链路利用率:640 KB/s ÷ 125 MB/s ≈ 0.5%                  │  ← 饿死!
   └──────────────────────────────────────────────────────────┘
```

**这就是"饿死"**——连接明明能跑 1 Gbps,因为窗口太小,实际只跑 0.5%。

> **不这样会怎样**:静态 64 KB 窗口在高 BDP 链路上,吞吐只有链路能力的 BDP 分之一。跨太平洋 RPC、跨可用区调用、卫星链路——这些高 RTT 场景下,静态窗口让 gRPC 的吞吐惨不忍睹。**必须把窗口设成 ≈ BDP,才能打满链路**。但 BDP 因链路而异(局域网 1 ms RTT vs 跨洋 100 ms RTT 差 100 倍),静态设个固定值不行——必须自适应。这就是 gRPC 加 BDP 估计的根。

## 五、gRPC 的 BDP 估计:用 PING 测 RTT、用 DATA 估带宽

这是本章的招牌。gRPC 在 HTTP/2 流控之上加的第一个工程,就是 **BDP 估计**——自动测量链路的 BDP,动态把窗口推到 ≈ BDP。

### BDP 估计的直觉

BDP = 带宽 × RTT。要估 BDP,得分别估带宽和 RTT。

- **RTT 怎么测**:发一个 PING,等 PING ACK,这段时间就是 RTT(HTTP/2 的 PING 帧天生适合干这个——它有 opaque 配对,ACK 原样回填,见 P2-08)。
- **带宽怎么估**:在 PING 飞出去到 ACK 回来的这段时间(RTT 窗口)里,记录接收方收到了多少字节(DATA 帧的 payload 累积)。这段时间收的字节数 / RTT ≈ 带宽。**更妙的是,这段时间收的字节总数本身就是一个 BDP 估计**——因为在 RTT 窗口里"能收到的字节数"恰好反映"链路在飞的字节数"。

### gRPC 的 BdpEstimator 源码

gRPC 的 BDP 估计器是 [`BdpEstimator`](../grpc/src/core/lib/transport/bdp_estimator.h)(不是 chttp2 目录,是更通用的 `src/core/lib/transport/`,因为 BDP 概念跨 transport 复用)。它被 `TransportFlowControl` 持有为成员 `bdp_estimator_`([`flow_control.h`](../grpc/src/core/ext/transport/chttp2/transport/flow_control.h#L467))。

关键字段在 [`bdp_estimator.h`](../grpc/src/core/lib/transport/bdp_estimator.h#L78-L86):

```cpp
int64_t accumulator_;           // 本次 PING 周期内收到的字节数
int64_t estimate_;              // BDP 估计值(字节)
gpr_timespec ping_start_time_;  // PING 发出时刻(测 RTT 用)
Duration inter_ping_delay_;     // 下次探测的间隔(自适应)
int stable_estimate_count_;     // 连续稳定估计次数
PingState ping_state_;          // UNSCHEDULED / SCHEDULED / STARTED
double bw_est_;                 // 带宽估计(bytes/sec)
```

构造函数的初始值在 [`bdp_estimator.cc`](../grpc/src/core/lib/transport/bdp_estimator.cc#L34-L42):

```cpp
BdpEstimator::BdpEstimator(absl::string_view name)
    : accumulator_(0),
      estimate_(65536),                         // 初始 BDP 估计 64 KB
      inter_ping_delay_(Duration::Milliseconds(100)),  // 初始 100 ms 探测一次
      stable_estimate_count_(0),
      ping_state_(PingState::UNSCHEDULED),
      bw_est_(0),
      name_(name) {}
```

核心算法在 `CompletePing`([`bdp_estimator.cc`](../grpc/src/core/lib/transport/bdp_estimator.cc#L44-L81)),PING ACK 回来时调用:

```cpp
Timestamp BdpEstimator::CompletePing() {
  gpr_timespec now = gpr_now(GPR_CLOCK_MONOTONIC);
  gpr_timespec dt_ts = gpr_time_sub(now, ping_start_time_);
  double dt = static_cast<double>(dt_ts.tv_sec) +
              (1e-9 * static_cast<double>(dt_ts.tv_nsec));
  double bw = dt > 0 ? (static_cast<double>(accumulator_) / dt) : 0;  // 带宽 = 字节 / RTT
  ...
  if (accumulator_ > 2 * estimate_ / 3 && bw > bw_est_) {
    // 这次收到的字节数超过上次估计的 2/3,且带宽创新高 → BDP 在涨
    estimate_ = std::max(accumulator_, estimate_ * 2);   // BDP 估计翻倍或取实际
    bw_est_ = bw;
    inter_ping_delay_ /= 2;                              // 加快探测
  } else if (inter_ping_delay_ < Duration::Seconds(10)) {
    // 估计稳定 → 慢慢降低探测频率
    stable_estimate_count_++;
    if (stable_estimate_count_ >= 2) {
      inter_ping_delay_ +=
          Duration::Milliseconds(absl::Uniform<int>(SharedBitGen(), 100, 200));
    }
  }
  ...
  ping_state_ = PingState::UNSCHEDULED;
  accumulator_ = 0;
  return Timestamp::Now() + inter_ping_delay_;   // 下次探测时刻
}
```

逐段拆这个算法:

1. **算 RTT**:`dt = now - ping_start_time_`。这就是 PING 一来回的时间,即 RTT。
2. **算带宽**:`bw = accumulator_ / dt`——PING 飞行期间收到的字节数 ÷ RTT = 带宽。
3. **更新 BDP 估计**:注意这里 `estimate_` **不是 `bw * dt`**(虽然那也是 BDP),而是直接用 `accumulator_`(本次 RTT 窗口收的字节数)。这有个微妙的好处:`accumulator_` 是**实际测量值**,而 `bw * dt` 是计算值,直接用 `accumulator_` 避免浮点误差,且更接近"链路真正在飞的字节数"。
4. **涨得快就探测得快**:`if (accumulator_ > 2 * estimate_ / 3 && bw > bw_est_)`——如果这次测的字节数比上次估计的 2/3 还多(说明 BDP 在涨),`inter_ping_delay_ /= 2`(探测频率翻倍),以便快速跟上 BDP 变化。
5. **稳定就慢慢退探**:`stable_estimate_count_ >= 2`(连续两次稳定),`inter_ping_delay_ += 100~200 ms`(带抖动地降低频率),避免无谓探测。抖动(`absl::Uniform`)是为了避免多条连接同步探测形成"惊群"。

### 探测周期:SchedulePing → StartPing → AddIncomingBytes → CompletePing

BDP 估计的完整生命周期:

1. **SchedulePing**:重置 `accumulator_ = 0`,状态转 SCHEDULED。
2. **StartPing**:PING 帧真正发出时,记录 `ping_start_time_ = now`,状态转 STARTED。
3. **AddIncomingBytes**:每收到一个 DATA 帧,调 `bdp_estimator->AddIncomingBytes(frame_size)`,`accumulator_ += frame_size`。这在 `parsing.cc` 的 `init_data_frame_parser`(行 565-575)里——每个 DATA 帧 `accumulator_` 都在涨。
4. **CompletePing**:PING ACK 回来,算 RTT 和带宽,更新 `estimate_`,安排下次探测。

这是一个**周期性被动测量**——gRPC 不发额外的探测流量(复用 keepalive 的 PING 或普通 PING),只是"借 PING 飞行这段时间,顺便数收到的字节数"。这是把"测量 BDP"这件事**几乎零开销**地嵌入正常数据流的工程巧思。

### BDP 估计怎么落地:通过 SETTINGS,不是 WINDOW_UPDATE

这里是最容易混的点。BDP 估计出了一个 `estimate_`(比如 12 MB),它怎么变成实际的窗口大小?

答案是:**通过 SETTINGS_INITIAL_WINDOW_SIZE 帧,不是 WINDOW_UPDATE**。

- **WINDOW_UPDATE**:补充已消费的信用(我消费了 64 KB,回 64 KB WINDOW_UPDATE)。它**不改变窗口的初始大小**,只是把扣掉的补回来。
- **SETTINGS_INITIAL_WINDOW_SIZE**:改变窗口的**初始大小**(从 64 KB 改成 12 MB)。这是一次"窗口扩容"。

BDP 估计的目的是扩容窗口(从默认 64 KB 推到 BDP 大小),所以它走 SETTINGS 路径。具体在 `TransportFlowControl::PeriodicUpdate`(在 [`flow_control.cc`](../grpc/src/core/ext/transport/chttp2/transport/flow_control.cc#L289-L336)):

```cpp
FlowControlAction TransportFlowControl::PeriodicUpdate() {

  if (enable_bdp_probe_) {
    // 算目标窗口(结合 BDP 和内存压力)

    UpdateSetting(Http2Settings::max_frame_size_name(), &target_frame_size_,
                  Clamp(target, Http2Settings::min_max_frame_size(),

```

`UpdateSetting`(行 252-273)的逻辑很简单:如果 `new_desired_value != *desired_value`,标记一个 action(QUEUE_UPDATE 或 UPDATE_IMMEDIATELY),稍后 transport 把这个 SETTINGS 发出去。

### target_window 的三段式:内存压力 + BDP

`TargetInitialWindowSizeBasedOnMemoryPressureAndBdp`(在 [`flow_control.cc`](../grpc/src/core/ext/transport/chttp2/transport/flow_control.cc#L198-L250))是 BDP 落地的关键函数。它不是简单"窗口 = BDP",而是结合**内存压力**做三段式调整:

```cpp
const double bdp = bdp_estimator_.EstimateBdp() * 2.0;   // BDP 的 2 倍(留余量)
const double memory_pressure = memory_owner_->GetPressureInfo().pressure_control_value;
// 三段式(源码注释里的 ASCII 图):
//   ▲
//   │
// 4mb ────┤---------x----
//   │              -----
// BDP ────┤            ----x---
//   │                    ----
//   ├─────────┬─────────────┬────────────────┬─────►
//   │Anything │Adjust to    │Drop to zero    │
//   │Goes     │BDP          │                │
//   0%        20%           50%              100% memory pressure
const double kAnythingGoesWindow = std::max(4.0 * kOneMegabyte, bdp);
if (memory_pressure < 0.2) {
  return kAnythingGoesWindow;                 // 低压力:给 max(4 MB, BDP)
} else if (memory_pressure < 0.5) {
  return lerp(memory_pressure, 0.2, 0.5,
              kAnythingGoesWindow, bdp);      // 中压力:从 max(4MB,BDP) 线性降到 BDP
} else if (memory_pressure < 1.0) {
  return lerp(memory_pressure, 0.5, 1.0, bdp, 0);  // 高压力:从 BDP 线性降到 0
} else {
  return 0;                                    // 压力爆表:窗口压到 0
}
```

逐段:

- **内存压力 < 20%**("Anything Goes"):内存充裕,窗口给 `max(4 MB, 2×BDP)`——保底 4 MB,链路 BDP 大就给 BDP。这是"放开了跑"。
- **20% ~ 50%**("Adjust to BDP"):内存开始紧张,窗口从 `max(4MB, 2×BDP)` 线性降到 `2×BDP`——还是给 BDP,但不再给多余的 4 MB 保底。
- **50% ~ 100%**("Drop to zero"):内存严重紧张,窗口从 `2×BDP` 线性降到 0——开始主动限流,优先保命。
- **>= 100%**:内存爆表,窗口 0——完全停收。

**这是个非常精妙的设计**——它把"网络 BDP"(链路能力)和"本机内存压力"(本机能力)结合,动态找窗口的甜点。网络好但内存紧,主动降窗口保命;网络差但内存宽,给大窗口也无所谓(反正发不快)。这是 gRPC 在 HTTP/2 之上加的**自适应工程**,远超 RFC 9113 规定的静态窗口。

> **钉死这件事**:gRPC 的 BDP 估计用 PING 测 RTT、用 DATA 帧字节累积估带宽,**几乎零开销**地嵌入正常数据流(复用 PING,不发额外探测流量)。BDP 估计出的窗口**通过 SETTINGS_INITIAL_WINDOW_SIZE 落地,不是 WINDOW_UPDATE**(WINDOW_UPDATE 只补信用,不扩容)。`TargetInitialWindowSizeBasedOnMemoryPressureAndBdp` 把网络 BDP 和本机内存压力三段式结合,**网络好内存紧就降窗口保命**——这是 gRPC 在 HTTP/2 流控之上的招牌自适应工程。

## 六、攒批上限:write_size_policy 找延迟 vs 吞吐的甜点

BDP 估计解决了"窗口多大"的问题,但还有个相关问题——**每次写循环发多少字节**。这是 `write_size_policy` 的职责(P2-06 已经点过,这里拆透)。

### 为什么攒批上限要动态

P2-06 讲过,写循环把多条流的帧攒成一批发出去(省系统调用)。攒批上限 `target_write_size` 决定"攒够多少字节就发"。这是个两难:

- **攒太少**(每次几百字节):系统调用频繁,CPU 跑满。
- **攒太多**(每次几 MB):数据在 outbuf 等很久才发,**延迟上升**;且一旦连接断了,损失大(攒的几 MB 全废)。

gRPC 的 `Chttp2WriteSizePolicy`([`write_size_policy.h`](../grpc/src/core/ext/transport/chttp2/transport/write_size_policy.h#L26-L61) + [`write_size_policy.cc`](../grpc/src/core/ext/transport/chttp2/transport/write_size_policy.cc))动态调整这个值。核心常量(行 29-40):

```cpp
static constexpr size_t MinTarget() { return 32 * 1024; }        // 下限 32 KB
static constexpr size_t MaxTarget() { return 16 * 1024 * 1024; } // 上限 16 MB
static constexpr Duration FastWrite() { return Duration::Milliseconds(100); }  // <100ms=快
static constexpr Duration SlowWrite() { return Duration::Seconds(1); }         // >1s=慢
size_t current_target_ = 128 * 1024;   // 初始 128 KB
int8_t state_ = 0;                     // 范围 -2..+2,记录连续快/慢次数
```

算法在 [`write_size_policy.cc`](../grpc/src/core/ext/transport/chttp2/transport/write_size_policy.cc#L27-L58):

```cpp
void Chttp2WriteSizePolicy::BeginWrite(size_t size) {
  if (size >= current_target_ * 7 / 10) {     // 攒够 70% 才开始计时
    ... start timing ...
  } else {
    state_ = std::max<int8_t>(state_, 0);     // 攒太少,重置负 state
  }
}

void Chttp2WriteSizePolicy::EndWrite(bool success) {
  ... compute elapsed ...
  if (elapsed < FastWrite()) {                // 写得快(<100ms)
    --state_;
    if (state_ == -2) {                       // 连续两次快
      state_ = 0;
      current_target_ = std::min(current_target_ * 3 / 2, MaxTarget());  // 涨 50%
    }
  } else if (elapsed > SlowWrite()) {         // 写得慢(>1s)
    ++state_;
    if (state_ == 2) {                        // 连续两次慢
      state_ = 0;
      current_target_ = std::max(current_target_ / 3, MinTarget());      // 降到 1/3
    }
  } else {
    state_ = 0;                               // 正常,重置
  }
}
```

关键设计:

1. **要连续两次同方向才调整**(`state_ == -2` 或 `+2`)。这是**去噪**——单次写快或慢可能是抖动(系统调度、GC 等),连续两次才确认是网络趋势。
2. **涨 50%、降到 1/3**:涨慢、降快。这是 AIMD(Additive Increase, Multiplicative Decrease)的变体——网络好时慢慢涨,网络差时快速降。和 TCP 拥塞控制同源。
3. **上下限保底**:`MinTarget=32 KB`(再差也不能小于 32 KB,否则系统调用太频繁)、`MaxTarget=16 MB`(再好也不能大于 16 MB,否则延迟太高)。

> **钉死这件事**:write_size_policy 用 AIMD 变体(连续两次快涨 50%、连续两次慢降到 1/3)动态调攒批上限,在延迟 vs 吞吐之间找甜点。涨慢降快、去噪(连续两次才调),这是从 TCP 拥塞控制借鉴的成熟策略。它和 BDP 估计一起,构成 gRPC 在 HTTP/2 流控之上的两大自适应工程。

## 七、PING 的攻击面与限速:ping_rate_policy 和 ping_abuse_policy

BDP 估计和 keepalive 都依赖 PING(P2-08 讲过 PING 帧,P5-17 会拆 keepalive)。但 PING 是个潜在的**攻击面**——恶意客户端可以狂发 PING,把服务端 CPU 打爆(每个 PING 都要处理、回 ACK、更新统计)。RFC 9113 §6.7 明确说"PING 不应被滥用",gRPC 用两个策略限速。

### ping_rate_policy:出站 PING 限速(管自己)

`Chttp2PingRatePolicy`([`ping_rate_policy.h`](../grpc/src/core/ext/transport/chttp2/transport/ping_rate_policy.h) + [`ping_rate_policy.cc`](../grpc/src/core/ext/transport/chttp2/transport/ping_rate_policy.cc))管**自己往外发 PING 的频率**——客户端用,避免自己不小心 ping 太频繁惹服务端反感。关键逻辑在 `RequestSendPing`([`ping_rate_policy.cc`](../grpc/src/core/ext/transport/chttp2/transport/ping_rate_policy.cc#L58-L91)):

```cpp
enum class SendPingResult { SendGranted, TooManyRecentPings, TooSoon };
SendPingResult RequestSendPing();

// 核心逻辑(简化):
// 1. 在飞 PING 数 > max_inflight_pings_ → TooManyRecentPings
// 2. 距上次发 PING < next_allowed_ping_interval → TooSoon
// 3. max_pings_without_data_sent_ != 0 且 pings_before_data_sending_required_ == 0
//    → 强制 1 分钟 throttle(kThrottleIntervalWithoutDataSent)
```

关键参数:

- `max_pings_without_data_sent_`:默认 2,来自 channel arg `GRPC_ARG_HTTP2_MAX_PINGS_WITHOUT_DATA`。意思是"没有发过数据的情况下,最多发 2 个 PING"——防止"连接没数据却狂 ping"。
- `kThrottleIntervalWithoutDataSent = 1 minute`:没数据时,PING 之间至少间隔 1 分钟。

> 这个策略不直接惩罚——它返回 `TooSoon` / `TooManyRecentPings`,让调用方决定"延迟再发"还是"放弃"。Promise 版的 `PingManager` 会启动一个 `DelayedPingPromise`(`Sleep(next_allowed_interval)` 然后再试)。

### ping_abuse_policy:入站 PING 防滥用(管对方)

`Chttp2PingAbusePolicy`([`ping_abuse_policy.h`](../grpc/src/core/ext/transport/chttp2/transport/ping_abuse_policy.h) + [`ping_abuse_policy.cc`](../grpc/src/core/ext/transport/chttp2/transport/ping_abuse_policy.cc))管**对方发来的 PING 频率**——服务端用,防恶意客户端狂 ping。这是**带惩罚**的策略,关键在 `ReceivedOnePing`([`ping_abuse_policy.cc`](../grpc/src/core/ext/transport/chttp2/transport/ping_abuse_policy.cc#L53-L62)):

```cpp
bool Chttp2PingAbusePolicy::ReceivedOnePing(bool transport_idle) {
  const Timestamp now = Timestamp::Now();
  const Timestamp next_allowed_ping =
      last_ping_recv_time_ + RecvPingIntervalWithoutData(transport_idle);
  last_ping_recv_time_ = now;
  if (next_allowed_ping <= now) return false;      // 间隔够,OK
  // 间隔不够(对方 ping 太快)→ 加 strike
  ++ping_strikes_;
  return ping_strikes_ > max_ping_strikes_ && max_ping_strikes_ != 0;  // 超额 → GOAWAY
}
```

关键参数:

- `min_recv_ping_interval_without_data_`:默认 **5 分钟**,来自 `GRPC_ARG_HTTP2_MIN_RECV_PING_INTERVAL_WITHOUT_DATA_MS`。客户端没发数据时,PING 之间至少 5 分钟。
- `max_ping_strikes_`:默认 **2**,来自 `GRPC_ARG_HTTP2_MAX_PING_STRIKES`。连续违反 2 次后,服务端 GOAWAY 关连接。
- `RecvPingIntervalWithoutData(transport_idle)`:如果 transport 完全空闲(没活跃流),间隔放大到 **2 小时**(对得上 RFC 1122 的 TCP keep-alive 默认 2 小时)——避免"无业务却频繁 ping"的可疑行为。

**返回 `true` 意味着"对方 ping 太频繁,该关连接"**。服务端会发 GOAWAY(带 `ENHANCE_YOUR_CALM` 错误码,debug data 含 `"too_many_pings"`),关闭这条连接。

### 两个策略的对照

| | ping_rate_policy | ping_abuse_policy |
|---|---|---|
| 管谁 | 自己出站 PING | 对方入站 PING |
| 用在哪 | 客户端 | 服务端 |
| 惩罚 | 不惩罚(返回 TooSoon,延迟再发) | **惩罚**(strike 累积,关连接) |
| 关键参数 | `GRPC_ARG_HTTP2_MAX_PINGS_WITHOUT_DATA`(默认 2) | `GRPC_ARG_HTTP2_MIN_RECV_PING_INTERVAL_WITHOUT_DATA_MS`(默认 5 分钟) |
| RFC 依据 | RFC 9113 §6.7(建议) | RFC 9113 §6.7 + ENHANCE_YOUR_CALM |

这两个策略组合,gRPC 既能用 PING 做 BDP 估计和 keepalive(自己的 PING 有限速,不滥用),又能防御恶意客户端 ping 攻击(对方的 PING 有 strike 制度)。

> **钉死这件事**:gRPC 用两个 PING 策略防滥用——**ping_rate_policy(出站,延迟再发,不惩罚)和 ping_abuse_policy(入站,strike 制度,超额关连接)**。前者管自己别 ping 太频,后者管对方别 ping 太频。ping_abuse 的 strike 制度(默认 5 分钟间隔、2 次违规关连接)是 gRPC 对 RFC 9113 §6.7 "PING 不应滥用"的具体落地,关连接时带 ENHANCE_YOUR_CALM + "too_many_pings" 反馈给客户端。

## 八、经典版 vs Promise 版:流控实现的差异(诚实交代)

本章最后必须诚实交代:经典版和 Promise 版的流控实现**不一样**,Promise 版有个明显的缺口。

### 共享核心:flow_control.cc

`TransportFlowControl` / `StreamFlowControl` / `FlowControlAction` 这三个类在 `flow_control.{h,cc}` 里,**经典版和 Promise 版共享**。这是"协议核心稳定,外围换骨架"的体现——无论 transport 是 callback 还是 promise,流控的信用模型、阈值触发、BDP 估计都一样。

### 经典版:BDP 完整接入

经典版(`chttp2_transport.cc`)把 BDP 估计**完整接入**:

- `schedule_bdp_ping_locked`(行 3168):安排一次 BDP PING。
- `start_bdp_ping_locked`(行 3188):PING 发出时,`bdp_estimator->StartPing()` 记录时刻。
- 每收到 DATA 帧,`parsing.cc` 行 574 调 `bdp_est->AddIncomingBytes(frame_size)` 累积字节。
- `finish_bdp_ping_locked`(行 3213):PING ACK 回来,`bdp_estimator->CompletePing()` 算 RTT 和带宽,然后 `PeriodicUpdate()` 应用新估计。
- 下次 BDP PING 在 `CompletePing` 返回的时刻(`Now() + inter_ping_delay_`)再安排——**自适应间隔**。

经典版的 BDP 估计是个完整的闭环:发 PING → 累积字节 → ACK → 算估计 → 调窗口 → 安排下次探测。

### Promise 版:BDP 还没接,用固定 8 秒定时器

Promise 版(`http2_client_transport.cc` / `http2_server_transport.cc`)的流控有个**诚实标注的缺口**——BDP 估计**还没接**。看 `FlowControlPeriodicUpdateLoop`([`http2_client_transport.cc`](../grpc/src/core/ext/transport/chttp2/transport/http2_client_transport.cc#L740-L773)):

```cpp
auto Http2ClientTransport::FlowControlPeriodicUpdateLoop() {
  return AssertResultType<absl::Status>(
    Loop([this]() {
      return TrySeq(
        Sleep(chttp2::kFlowControlPeriodicUpdateTimer),   // ★ 固定 8 秒
        [this]() -> Poll<absl::Status> {
          chttp2::FlowControlAction action = flow_control_.PeriodicUpdate();
          ActOnFlowControlAction(action, nullptr);
          ...
        }
      );
    })
  );
}
```

`kFlowControlPeriodicUpdateTimer = 8 seconds`([`flow_control.h`](../grpc/src/core/ext/transport/chttp2/transport/flow_control.h#L62))。Promise 版用**固定 8 秒**定时器调 `PeriodicUpdate`,而经典版是用 BDP PING 的自适应间隔(`inter_ping_delay_`,100 ms ~ 10 秒)调。

更关键的是,Promise 版**从不调** `AddIncomingBytes`、`SchedulePing`、`StartPing`、`CompletePing`——BDP 估计器在 Promise 版里**闲置**,estimate 永远是构造默认值 65536(64 KB)。源码里有明确的 TODO,在 [`http2_client_transport.cc`](../grpc/src/core/ext/transport/chttp2/transport/http2_client_transport.cc#L748-L762):

```
// TODO [PH2][P2][BDP] Remove this static sleep when the BDP code is done.
```

意思是"Promise 版的 BDP 接入是 P2 优先级的待办,目前用固定 8 秒定时器占位"。这是 gRPC "换骨架"过程中的诚实状态——新架构先把骨架立起来,协议核心(信用模型、WINDOW_UPDATE、阈值触发)接入完整,但 BDP 这种"高级自适应"还在 TODO 里。

### 其他 Promise 版差异

除了 BDP 没接,Promise 版还有几个差异:

1. **pending_size 用 hack**:`HackIncrementPendingSize`(在 [`flow_control.cc`](../grpc/src/core/ext/transport/chttp2/transport/flow_control.cc#L407-L417))每个 DATA 帧直接加 pending_size,而不是等消息组装完才 `SetPendingSize`。这是临时方案(函数名带 `Hack` 前缀就是诚实标注),有过度估计的副作用。
2. **transport WINDOW_UPDATE 总是 writing_anyway=true**:Promise 版的 `MaybeAddTransportWindowUpdateFrame`([`http2_transport.cc`](../grpc/src/core/ext/transport/chttp2/transport/http2_transport.cc#L410-L438))总传 `writing_anyway=true`,意味着"只要有写循环就发 WINDOW_UPDATE",不精细判断"是否必须发"。源码注释明确标了这是 known limitation。
3. **读循环可暂停**:遇到 `UPDATE_IMMEDIATELY`,Promise 版调 `read_context_.SetPauseReadLoop()`([`http2_client_transport.cc`](../grpc/src/core/ext/transport/chttp2/transport/http2_client_transport.cc#L804))暂停读,优先发 WINDOW_UPDATE——避免"读了一堆 DATA 但来不及回信用"导致死锁。

> **钉死这件事**:Promise 版流控的协议核心(信用模型、WINDOW_UPDATE、阈值触发)**完整接入**,但 **BDP 估计还没接**——用固定 8 秒定时器替代经典版的自适应 BDP PING 周期。BDP 估计器在 Promise 版里闲置,estimate 永远是默认 64 KB。这是 gRPC "换骨架"过程的诚实状态,源码有明确 TODO。**生产环境默认还是经典版**,完整 BDP 能力在经典版里。

## 九、技巧精解

本章挑两个最硬核的技巧单独拆透。

### 技巧一:流级 window 存 delta 而非绝对值——把 O(N) 更新变成 O(1)

第一个技巧是 `StreamFlowControl` 的字段设计——**流级 window 存相对 transport 的 delta,而不是绝对值**。回头看字段(行 643-647):

```cpp
int64_t remote_window_delta_ = 0;          // 不是 remote_window_,是 delta
int64_t announced_window_delta_ = 0;       // 不是 announced_window_,是 delta
```

有效流级出站 window = `remote_window_delta_ + peer_settings.initial_window_size()`(对得上 [`flow_control_manager.h`](../grpc/src/core/ext/transport/chttp2/transport/flow_control_manager.h#L67-L75) 的 `GetStreamFlowControlTokens`)。

为什么这样设计?考虑一个场景:transport 有 10000 条流,BDP 估计把 `INITIAL_WINDOW_SIZE` 从 64 KB 改成 12 MB。

- **存绝对值**:要遍历 10000 条流,每条流的 `remote_window` 都加 (12 MB - 64 KB)——**O(N) 更新**。10000 条流,这是个可观开销,且流控 SETTINGS 调整很频繁(BDP 周期性更新)。
- **存 delta**:**只改 transport 的 `target_initial_window_size_`**——所有流的有效 window 自动跟着变(`delta + new_init_window`)——**O(1) 更新**。

这是把"全局参数变化导致批量更新"的 O(N) 操作,通过"存相对偏移"变成 O(1)。代价是每次算有效 window 要做一次加法(`delta + init_window`),但加法远比遍历便宜。

这个技巧的精髓在于:**流级 window 的"初始值"是 transport 级参数(`INITIAL_WINDOW_SIZE`),而不是每条流独立**。把"初始值"抽到 transport 上,流级只存"相对偏移",全局参数变化时只改一处。

> **不这样会怎样**:如果流级 window 存绝对值,每次 BDP 推高 `INITIAL_WINDOW_SIZE`(高频操作),要遍历所有流改值——O(N) 开销,在万级并发流下是可观 CPU。存 delta 把这个 O(N) 变成 O(1),是海量并发下流控可扩展的物理基础。

### 技巧二:BDP 估计的"零开销测量"——借 PING 飞行期顺便数字节

第二个技巧是 BDP 估计的测量方式——**借 PING 飞行的那段 RTT,顺便数收到的字节数**,几乎零开销。

朴素的 BDP 测量方案是:发专门的探测流量(比如一段已知大小的数据),测它一来回的时间。但这有代价——探测流量本身吃带宽,且增加复杂度。

gRPC 的方案更妙:**复用已有的 PING**(keepalive PING 或普通 PING),在 PING 飞行的那段 RTT 里,记录"收到的 DATA 帧字节数"。这段时间收的字节数,本身就反映"链路在飞的字节数"——**这就是 BDP 的物理定义**。

```
   BDP 估计的零开销测量(借 PING 飞行期)
   ┌──────────────────────────────────────────────────────────┐
   │  t=0:  发 PING(ping_start_time_ = t=0),accumulator_ = 0  │
   │  t=10ms: 收到 DATA 5 KB → accumulator_ += 5 KB           │
   │  t=20ms: 收到 DATA 8 KB → accumulator_ += 8 KB           │
   │  ...                                                     │
   │  t=100ms: 收到 PING ACK,CompletePing()                   │
   │    RTT = 100ms,accumulator_ = 100 MB                     │
   │    带宽 = 100 MB / 100ms = 1 GB/s                        │
   │    BDP 估计 = 100 MB(就是 accumulator_)                 │
   │                                                          │
   │  关键:PING 是本来就要发的(keepalive 用),数据是本来就在│
   │  收的(DATA 帧),BDP 估计没有额外开销                   │
   └──────────────────────────────────────────────────────────┘
```

这个设计的妙处:

1. **零额外流量**:不发探测数据,复用 PING(本来就要发)。
2. **被动测量**:不主动发探测,只在正常数据流期间"旁观"。
3. **物理正确**:`accumulator_` 在 RTT 窗口里的累积值,恰好就是"链路在飞的字节数"——这就是 BDP 的定义。

配合自适应间隔(`inter_ping_delay_`,BDP 在涨就探测快、稳定就探测慢),这个方案在几乎零开销下,持续跟踪链路 BDP 变化。

> **不这样会怎样**:如果用主动探测(发已知大小数据测时间),探测流量本身吃带宽,在低速链路上探测流量占比可观;且探测要额外代码路径,增加复杂度。gRPC 的"借 PING 飞行期数字节"把测量开销压到几乎零,是协议工程"借力打力"的精品。**这是把"网络测量"无侵入地嵌入正常数据流的典范**。

> **钉死这件事**:本章的两个技巧——**流级 window 存 delta 把 O(N) 更新变 O(1)**(支持海量并发流的流控可扩展)、**BDP 估计借 PING 飞行期零开销测量**(不开销地持续跟踪链路能力)——都是 gRPC 在 HTTP/2 流控之上的精品工程。它们让 gRPC 在高并发、高 BDP、动态网络下都能"不淹不饿"。

## 十、章末小结

### 回扣主线

本章拆的是 gRPC 的流量控制,属于二分法的**协议层招牌**那一面(让一次方法调用变成一条"可控"的流——流控是"可控"的核心)。从层次看,本章是第 2 篇(HTTP/2 传输)的压轴——P2-05 讲 HTTP/2 五件发明,P2-06 讲 transport,P2-07 讲 HPACK(头部压缩),P2-08 讲 framing,本章讲流控。**流控 + HPACK 是 gRPC 协议层的两大招牌**。

回到全书主线:**把一次方法调用变成 HTTP/2 上的一条可控的流**。本章拆的是这条流的"**节奏控制**"——HTTP/2 双层 window 的信用模型怎么防淹死,gRPC 的 BDP 估计怎么防饿死,ping 限速怎么防滥用。这三件加起来,让 gRPC 在任何网络下都能"不淹不饿"。第 2 篇到此结束,下一章 P3-10 进入第 3 篇——从协议层转向框架层,拆 call 与 completion queue,看一次方法调用在两端怎么被"组织"起来。

### 五个为什么

1. **为什么 HTTP/2 流控要双层 window?**——连接级 window 保整条连接总内存可控,流级 window 让每条流的背压独立(慢流不波及快流)。只有连接级会饿死快流,只有流级会总额度爆炸。两层缺一不可。

2. **为什么 WINDOW_UPDATE 不是"收到 DATA 就回"?**——流控的目的是防淹死。如果收到 DATA 立刻回 WINDOW_UPDATE,字节全堆在接收方内存里,流控失效。正确语义是**消费完才回信用**——消费多少补多少。gRPC 加了半窗阈值(攒够半窗或 8 KB 才回),在 WINDOW_UPDATE 帧数量和发送方卡顿之间找平衡。

3. **为什么静态默认窗口(64 KB)在高 BDP 链路上会饿死?**——BDP(带宽时延积)= 带宽 × RTT。窗口 < BDP 时,发送方在窗口用完时停下来等 ACK,等 ACK 的时间里链路空着,吞吐上不去。64 KB 在跨洋链路(BDP 12.5 MB)上只跑到链路能力的 0.5%。

4. **为什么 BDP 估计落地走 SETTINGS_INITIAL_WINDOW_SIZE 而非 WINDOW_UPDATE?**——WINDOW_UPDATE 只补已消费的信用(扣 64 KB 补 64 KB),不改变窗口初始大小。BDP 估计的目的是**扩容窗口**(从 64 KB 推到 12 MB),必须改 `INITIAL_WINDOW_SIZE`,这通过 SETTINGS 帧协商。两条路径职责不同,不能混。

5. **为什么 gRPC 要 ping_rate 和 ping_abuse 两个策略?**——PING 是 BDP 估计和 keepalive 的依赖,但也是攻击面(狂 ping 打爆 CPU)。ping_rate_policy(出站,延迟再发,不惩罚)管自己别 ping 太频;ping_abuse_policy(入站,strike 制度,超额关连接)管对方别 ping 太频。两者组合,gRPC 既能用 PING 做正事,又能防滥用。
