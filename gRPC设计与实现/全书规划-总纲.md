## 一、这本书的定位

### 目标读者

写过 `.proto`、用 `stub->GetUser()` 调过远程方法、配过 keepalive 或负载均衡,甚至翻过 `grpc/grpc` 源码,但**一知半解**:知道有 channel、call、completion queue、balancer、resolver、chttp2,却讲不清一次 `GetUser` 调用从客户端发出到服务端 handler 收到、再到客户端拿到结果,中间到底穿过了哪些层、每一层为什么这么设计、源码里那些 HPACK 动态表、flow control window、filter stack、call spine 到底在干什么。和"看过 LevelDB 源码但不理解"、"读过 tokio 源码但一知半解"是同一种病。

### 全书一句话主旨

> **从"一次本地的函数调用",到"跨网络、跨语言、可治理的一次方法调用"——gRPC 是怎么把一次方法调用变成 HTTP/2 上的一条可控的"流"的,每一步的设计动机和实现技巧都讲透。**

每个组件,都是这次旅程上的一个**驿站**。读完一本书,读者不是"多记了几个 API",而是能**在脑子里放映出一次 gRPC 调用的全过程**:客户端 stub 怎么把方法签名变成一次调用,怎么穿过 filter 栈,怎么被 resolver 解析成地址、被 balancer 挑中一条 SubChannel,怎么在 chttp2 transport 里变成一条 HTTP/2 stream,怎么被 HPACK 压缩头部、被 flow control 背压,怎么在服务端被解析回方法、交给 handler,响应再原路返回——以及每一步底下用了什么巧妙的手段。

### 为什么用"一次方法调用的旅程"当骨架(而非按模块平铺)

gRPC 的模块(`call` / `transport` / `client_channel` / `resolver` / `load_balancing` / `filters` / `tsi`)高度耦合:一次调用要穿过 filter 栈、被 client_channel 派发、被 balancer 选路、被 SubChannel 承载、被 chttp2 编码成帧。按模块百科式平铺,读者会觉得每块都是孤岛、不知道**为什么要学这个**。用"一次方法调用从客户端到服务端再回来"这条旅程当骨架,每个模块都是"为了让旅程继续而必须出现的角色"——读者永远知道**我在哪、为什么需要下一站**。

## 三、全书主线、二分法、主比喻

### 主线

> **如何让一次跨网络、跨语言的方法调用,像本地调用一样自然,又不假装网络不存在?——核心是把一次方法调用变成 HTTP/2 上的一条可控的"流"。**

任何一处看不懂某个机制,回到这句问:"这是在**把方法调用编码成网络上流动的字节(协议层)**,还是在**把网络字节变回可调用、可治理的方法(框架层)**?"

### 二分法

本书的二分法:

> **协议层(把方法调用编码成网络上流动的字节) vs 框架层(把网络字节变回可调用、可治理的方法)。**

- **协议层这一面**:IDL 契约(protobuf)、HTTP/2 帧与流、HPACK 头部压缩、gRPC framing、flow control——这些决定"字节怎么过线"。
- **框架层这一面**:call、completion queue、filter stack、channel、resolver、balancer、SubChannel、retry、keepalive、security——这些决定"调用怎么发起、怎么治理、怎么可用"。

gRPC C++ core 的每个模块都落在这两面之一(或两者衔接处:`client_channel` 是框架层通往协议层的枢纽)。任何时候迷路,回到这个二分法。

### 主比喻决策:直球为主,比喻点睛(⚠️ 区别于 Tokio,待总设计师拍板)

Tokio 那本用了贯穿全书的"餐厅服务员"重比喻体系,因为 async 概念(task/scheduler/reactor)抽象、需要落地。gRPC 的概念相对具体(方法调用、连接、流、地址),而且**本系列自《LevelDB》起,用户已拍板"直球为主、比喻点睛"**(《Lua/LuaJIT》更进一步到"纯直球")。因此本书基调:

> **直球为主:绝大多数地方第一性原理直给,不套比喻。比喻点睛:仅在开篇(P0-01)用"打一次电话"做一次性点睛(拨号选号 = resolver、选线路 = balancer、通话 = stream、保活 = keepalive、挂机 = 连接管理),全书不建立贯穿的重比喻体系,不得喧宾夺主。**

> 这是本书与《Tokio》最显著的风格差异。如果你希望本书也用贯穿比喻(如"电话交换机"全程),告诉我,定下再开工。见第七节拍板点①。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线是一条旅程:**远程调用的根本矛盾 → 用契约抹平语言 → 用 HTTP/2 把调用变成流 → 用 filter/CQ 让调用可发 → 客户端怎么选路治理 → 生产怎么可用 → 性能与生态 → 收束**。

### 第 0 篇 · 开篇:为什么需要 gRPC(定调,1 章)

- **第 1 章 · 第一性原理:为什么需要 gRPC**(`P0-01`):远程调用的根本矛盾——Birrell & Nelson 1984 的 RPC 初心("让远程调用像本地调用一样自然")与"伪装成本地调用会掩盖网络的不可靠"(Waldo 的"八条谬误")。gRPC 用"IDL 契约 + HTTP/2 流 + protobuf 编码"三件套回答:不假装网络是本地的,而是把网络的异步性暴露成一等公民的"流"。为什么这三件套让它成了跨语言 RPC 的事实标准。点出 C++ core 的两层(协议 / 框架)与正在进行的 Promise-based 重构。**【样本章节,全书风格标杆,见后】**

### 第 1 篇 · 契约:用 protobuf 定义跨语言契约(3 章)

> 先解决"跨语言"的根:一份语言无关的中间契约。这一篇是全书地基。

- **第 2 章 · IDL 与 .proto:语言无关的契约**(`P1-02`)——为什么跨语言调用必须先有一份"语言无关"的描述?.proto 怎么用 message / service / rpc / 字段号 定义接口。技巧:为什么用**字段号**而非字段名定位(向后兼容的根)、message 嵌套、oneof/map、proto2/proto3/editions 的取舍。归属:`协议(契约层)`。
- **第 3 章 · protobuf 编码:为什么又小又快**(`P1-03`)——JSON/XML 文本编码为什么慢且大?protobuf 的 **varint 变长 + tag-length-value wire format** 凭什么又小又快。技巧:**varint**(每 7 位 + 1 续位)、**zigzag**(有符号整数)、**向前/向后兼容**(unknown fields 透传)、packed repeated。源码:protobuf wire format,对照 `src/core/ext/transport/chttp2/transport/varint.cc`(HTTP/2 侧的变长整数,设计同源)。归属:`协议`。

### 第 2 篇 · 传输:把方法调用变成 HTTP/2 上的流(5 章,全书最硬)

> 这一篇是 gRPC C++ core 的心脏:它**自己用 C 实现了一整套 HTTP/2**。是本书区别于 grpc-go/grpc-java(它们复用语言库 HTTP/2、讲不透)的核心资本。

- **第 5 章 · HTTP/2 基础:为什么是 gRPC 的天然底座**(`P2-05`)——为什么 gRPC 不用 HTTP/1.1(队头阻塞、文本、无流)、也不用私有 TCP(割裂生态)?HTTP/2 的**二进制分帧、流、多路复用、HPACK、流量控制**各解决什么。技巧:多路复用如何消除 HTTP/1.1 的队头阻塞、为什么"流"是 gRPC 的第一性概念。对照 RFC 9113(HTTP/2 bis,2022,取代老的 RFC 7540——第一波 agent 核实整个 chttp2 目录对 7540 引用 0 处、对 9113 引用 19 文件)+ `src/core/ext/transport/chttp2/transport/frame_*.cc`(DATA/SETTINGS/PING/GOAWAY/RST_STREAM/WINDOW_UPDATE 各帧)。归属:`协议`。
- **第 6 章 · chttp2 transport 全貌:一条流的一生**(`P2-06`)——一个 TCP 连接怎么同时承载成千上万个并发调用?stream 怎么创建、管理、销毁?技巧:**stream 表(`stream.h` / `stream_lists.cc`)**、一次写循环怎么把多条流的数据攒成一批发出去(`writing.cc` / `write_cycle.cc`)。源码:`chttp2_transport.cc`、`http2_client_transport.cc` / `http2_server_transport.cc`、`transport_common.cc`。归属:`协议/衔接`。
- **第 7 章 · HPACK:头部压缩的招牌**(`P2-07`)——每次调用重复的 `:path` / `content-type` / `te: trailers` 文本冗余,HPACK 怎么压到几乎零字节?技巧:**静态表(61 项常用头)+ 动态表(本次连接学到的头)+ Huffman 编码**三重压缩;动态表尺寸协商(`SETTINGS_HEADER_TABLE_SIZE`);敏感头(never-indexed)为什么不进动态表。源码:`hpack_encoder.cc` + `hpack_encoder_table.cc`(编码端)、`hpack_parser.cc` + `hpack_parser_table.cc`(解码端)、`hpack_constants.h` / `hpack_tables.txt`(静态表)、`decode_huff.cc` + `huffsyms.cc`(Huffman)、`bin_encoder.cc` / `bin_decoder.cc`。归属:`协议(招牌)`。
- **第 8 章 · gRPC framing 与帧解析**(`P2-08`)——protobuf 消息怎么塞进 HTTP/2 的 DATA 帧?gRPC 的 **Length-Prefixed-Message(1 字节压缩位 + 4 字节长度 + 消息体)**;各种控制帧怎么处理。技巧:**parsing(读侧状态机)与 writing(写侧)双状态机**、帧处理与流的解耦、初始 SETTINGS 协商、`grpc-status` / `grpc-message` 走 trailer。源码:`parsing.cc`、`writing.cc` / `write_cycle.cc`、`frame_data.cc`、`frame_settings.cc` / `http2_settings.cc` / `http2_settings_manager.cc`、`frame_ping.cc` / `ping_callbacks.cc`、`frame_goaway.cc` / `goaway.cc`、`frame_rst_stream.cc`、`frame_window_update.cc`。归属:`协议`。
- **第 9 章 · 流量控制:不淹不饿**(`P2-09`)——生产者比消费者快会"淹死"连接,flow control 怎么背压?gRPC 在 HTTP/2 流控之上又加了什么?技巧:**HTTP/2 window(连接级 + 流级两层信用)**、**gRPC BDP 估计**(按带宽时延积自动调窗)、**write_size_policy**(每次写多少)、**ping_rate_policy / ping_abuse_policy**(限速 ping、防客户端滥用 keepalive 打爆服务端)。源码:`flow_control.cc` / `flow_control_manager.h`、`write_size_policy.cc`、`ping_rate_policy.cc` / `ping_abuse_policy.cc`、`bdp_estimator` 思路。归属:`协议(招牌)`。

### 第 3 篇 · 调用:一条流的生命周期(3 章)

> 一次方法调用在两端怎么被"组织"起来:从 API 到流。

- **第 10 章 · call 与 completion queue:经典异步模型**(`P3-10`)——gRPC core 的招牌异步机制:为什么所有异步操作都通过 completion queue 交付?callback API 怎么从 sync/async 演进而来?技巧:**CQ 的事件队列与多路复用**(一个线程 poll 一个 CQ 服务海量调用)、sync/async/callback 三模的取舍、`Pluck` vs `Next`。源码:`src/cpp/common/completion_queue_cc.cc`、`completion_queue` 内部、call 生命周期。归属:`框架`。
- **第 11 章 · filter stack:可插拔架构的灵魂**(`P3-11`)——鉴权 / 日志 / 压缩 / 限流 / census 这些横切关注点怎么不侵入业务?gRPC 的 filter stack(滤镜栈)。技巧:**filter 责任链**(一次调用穿过一串 filter,每个 filter 处理一类横切)、**filter fusion**(新 Promise 架构下,多个 filter 的 promise 在编译期合并成一条流水线,省掉层层回调)、`StartBatch` 批量操作。源码:`src/core/ext/filters/`(http/rbac/logging/fault_injection/message_size/census/channel_idle 各 filter)、`src/core/call/call_filters.cc`、`src/core/call/filter_fusion.h`、`src/core/filter/` 框架。归属:`框架(招牌)`。
- **第 12 章 · 四种调用模式与流状态机**(`P3-12`)——unary / server-streaming / client-streaming / bidi-streaming 为什么本质都是流?一次 call 的生命周期。技巧:**send-metadata → send-message → recv-message → recv-status** 的三段式状态机;新架构下 `call_spine`(call 主干,promise 编排)怎么把这条流组织起来;`metadata_batch`(一次打包多个 header,而非逐个)。源码:`src/core/call/call_spine.cc` / `client_call.cc` / `server_call.cc` / `message.cc` / `metadata_batch.cc` / `interception_chain.cc`。归属:`框架`。

### 第 4 篇 · 客户端治理:地址、负载均衡、重试(4 章)

> 调用怎么发出去、发去哪、失败了怎么办。这一篇是微服务场景下 gRPC 的看家本领。

- **第 13 章 · Resolver:名字到地址**(`P4-13`)——`dns:///user-svc:8080` 怎么变成 `[ip:port]` 列表?解析器怎么订阅后端变更?技巧:**URI scheme**(dns:///、passthrough:///、xds:///、unix:)、异步解析与结果回灌、**service config 下发**(名字服务顺带下发 LB 策略 / retry 配置 / 超时)。源码:`src/core/resolver/`(dns / passthrough / sockaddr 等)、`client_channel` 里 resolver 的接入、`src/core/service_config/`。归属:`治理`。
- **第 14 章 · SubChannel:一条后端连接**(`P4-14`)——一个后端实例怎么被抽象成可建连 / 复用 / 重连的 SubChannel?技巧:**SubChannel 复用池**(global / local pool,多个 channel 共享同一后端的连接,省连接数)、连接状态机(IDLE → CONNECTING → READY → TRANSIENT_FAILURE)、connection attempt 与指数退避。源码:`subchannel.cc`、`global_subchannel_pool.cc` / `local_subchannel_pool.cc` / `subchannel_pool_interface.cc`、`connector.h`。归属:`治理/衔接`。
- **第 15 章 · 负载均衡:挑这条流去哪**(`P4-15`)——多个 SubChannel,这次调用挑哪个?技巧:**pick_first / round_robin / weighted_round_robin / least_request / ring_hash** 各解决什么;**Picker 的无锁 fast path**(每次调用原子选一个 SubChannel);**ring_hash**(一致性哈希,XDS 粘性会话用);新架构 `load_balanced_call_destination.cc`(call 目的地在 LB 之后)。源码:`src/core/load_balancing/`(各策略)、`load_balanced_call_destination.cc`。归属:`治理(招牌)`。
- **第 16 章 · 重试与对冲**(`P4-16`)——调用失败了什么时候该重试、怎么退避、怎么防雪崩?技巧:**retry policy**(可重试的状态码 / 最大次数 / 退避抖动)、**retry throttle**(令牌桶:当后端连续失败时自动降低重试,防雪崩)、**hedging 对冲**(不取消原请求就并发发第二个,换延迟换吞吐)、**transparent retry vs application retry**(建连前失败可透明重试)。源码:`retry_filter.cc` / `retry_filter_legacy_call_data.cc` / `retry_interceptor.cc`、`retry_service_config.cc` / `retry_throttle.cc`。归属:`治理(招牌)`。

### 第 5 篇 · 生产可用:keepalive、健康检查、安全(3 章)

- **第 17 章 · keepalive:探测死连接**(`P5-17`)——TCP 半开、云负载均衡器空闲超时,会让连接"假活",keepalive 怎么探测?技巧:**keepalive ping/pong**(`GRPC_ARG_KEEPALIVE_TIME` 等)、**ping 限速**(`ping_rate_policy`:服务端限制客户端 ping 频率,防 keepalive 滥用打爆 CPU)、无活跃调用时的特殊处理。源码:`src/core/ext/transport/chttp2/transport/keepalive.cc`、`ping_callbacks.cc` / `ping_promise.cc`、`ping_rate_policy.cc` / `ping_abuse_policy.cc`、`frame_ping.cc`。归属:`可用`。
- **第 18 章 · 健康检查与服务反射**(`P5-18`)——客户端怎么知道服务端能不能接活?gRPC 的 health check 协议。技巧:**health check over gRPC**(复用 gRPC 自己传健康状态)、streaming health(状态变更推送)、server reflection(运行时反射服务定义,便于调试 / 动态调用);SubChannel 用 `subchannel_stream_client.cc` 探健康。源码:`src/cpp/server/health/`、`reflection`、`src/core/client_channel/subchannel_stream_client.cc`。归属:`可用`。
- **第 19 章 · 安全:TLS / mTLS / ALTS / tsi**(`P5-19`)——跨网络怎么认证加密?gRPC 的凭证体系。技巧:**tsi(Transport Security Interface)抽象**(屏蔽 SSL/ALTS/local 后端)、**ALTS**(Google 内部 mTLS 替代)、handshaker 握手状态机、**channel creds(连接级)vs call creds(调用级)** 分离(连接用 TLS、调用用 OAuth/JWT 叠加)。源码:`src/core/tsi/`、`src/core/credentials/`、`src/core/ext/transport/chttp2/alpn/`、`src/core/call/auth_context`、`src/core/handshaker/`。归属:`可用`。

### 第 6 篇 · 可观测、性能、生态(3 章)

- **第 20 章 · 元数据、channelz、stats / otel**(`P6-20`)——一次调用的上下文(鉴权令牌 / trace-id / deadline)怎么传?线上怎么观测?技巧:**metadata_batch**(把多个 header 打包成一批,一次解析)、**channelz**(运行时把 channel / SubChannel / socket 的状态做成一棵诊断树)、OpenTelemetry tracer。源码:`src/core/call/metadata_batch.cc` / `parsed_metadata.cc`、`src/core/channelz/`、`src/core/telemetry/`、`src/cpp/ext/otel/`、`src/cpp/server/channelz/`。归属:`可用/可观测`。
- **第 21 章 · 性能:slice、arena、零拷贝、压缩**(`P6-21`)——gRPC core 怎么在跨语言、跨网络下还快?技巧:**arena**(一次 call 内的内存一次性分配、call 结束整体丢弃,无 per-free)、**slice 引用计数零拷贝**(消息在 filter 栈里传递时只挪指针不拷字节)、压缩(gzip/zstd,`message_compress` filter)、连接 / 流复用调优。源码:`src/core/call/call_arena_allocator.cc`、`message.cc`、`message_compress` filter、slice(引用计数缓冲,新版位置待 clone 核实)。归属:`协议/框架(招牌)`。
- **第 22 章 · xDS 与服务网格**(`P6-22`)——gRPC 怎么和 Envoy / xDS 配合,动态下发 LB / 路由 / 熔断?技巧:**xDS 协议**(LDS / RDS / CDS / EDS)、**gRPC 作为 xDS client**(不依赖 Envoy sidecar,client 内置 xDS 解析)、bootstrap 配置、与 Envoy / Istio 互通。源码:`src/core/xds/`、xds 解析器与 balancer、`src/core/resolver/` xds。**承接《Envoy 设计与实现》那本**。归属:`治理/衔接`。

### 第 7 篇 · 收束(1 章)

- **第 23 章 · 全书收束:RPC 的演化与"流"的代价**(`P7-23`)——把一次方法调用变成一条流,得到了什么(多路复用、背压、流式、跨语言)、付出了什么(HTTP/2 的复杂度、调试难、人肉 curl 不友好);**gRPC vs REST/HTTP1 vs Dubbo vs Thrift 对照总表**;展望 gRPC core 的 Promise-based 重构(call spine / filter fusion / chaotic_good 新 transport)把这套带向何方。归属:`总览`。

## 六、写作约定(沿用《Tokio》系列,微调适配 gRPC)

1. **动机 + 技巧双线**(本书命脉,见第二节):每个关键代码点讲"为什么这么设计" + "用了什么技巧、不这么写会怎样";每章单开"技巧精解"小节。

5. **每章标准结构**:① 章首(核心问题 + 读完你会明白 + 难度高的加逃生阀);② 正文若干小节(招牌四段式:**提问 → 不这样会怎样 → 所以这样设计 → 源码/技巧佐证**);③ 技巧精解;④ 章末小结(回扣"协议层 vs 框架层"主线 + 五个为什么清单 + 想深入往哪钻 + 引出下一章)。
6. **回扣全局**:每章结尾回到二分法,说明本章服务"协议层"还是"框架层"那一面,衔接前后章。
7. **语言**:中文,技术名词保留英文(gRPC、HTTP/2、HPACK、call、completion queue、channel、SubChannel、balancer、resolver、chttp2、filter、stream、slice、arena 等)。
