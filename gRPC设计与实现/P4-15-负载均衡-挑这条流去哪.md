# 第 4 篇 · 第 15 章 · 负载均衡:挑这条流去哪(招牌章)

> **核心问题**:Resolver 给了一组地址,P4-14 把每个地址变成了一条 SubChannel(可能 READY、可能 IDLE、可能 TRANSIENT_FAILURE)。现在来了一条调用,要发去哪条 SubChannel?朴素做法是"挑一条能用的就行",但微服务场景对负载均衡有更细的要求:能不能轮询、能不能按权重、能不能粘性会话、能不能剔除异常后端、能不能按请求 hash 路由?而且,挑后端这一步是**每次调用都要走的超热路径**——它必须**快到极致**。本章拆透 gRPC 各 LB 策略各解决什么,以及一个 gRPC 在性能上的招牌技巧——**Picker 的无锁 fast path:每次调用选后端只用一条原子指令,不加任何锁**。

> **读完本章你会明白**:
> 1. `pick_first`、`round_robin`、`weighted_round_robin`、`ring_hash`、`outlier_detection`、`priority`、`weighted_target` 各解决什么问题,以及一个**关键修正**——**C 核心里没有 `least_request` 策略**(全树 grep 零命中),Java/Go 那套不能直接搬过来。
> 2. 为什么所有策略都遵循"**控制平面 / 数据平面分离**":控制平面(SubChannel 状态变化时)重建 Picker,数据平面(每次调用)只调 `Pick()`——这是 Picker 能做到无锁的根。
> 3. ★**Picker 无锁 fast path** 的全部真相:round_robin 的 Picker 只有一条 `last_picked_index_.fetch_add(1, std::memory_order_relaxed)` 原子指令,"只选 READY"在控制平面构造 Picker 时就做完了。
> 4. 新架构 `LoadBalancedCallDestination` 和经典 Picker 模型的关系——**Picker 没被取代,只是被 Promise 化包装**。

> **如果一读觉得太难**:先只记住三件事——① LB 策略 = 控制平面(管 SubChannel)+ 数据平面(Picker 选后端);② round_robin 的 Picker 用一条 `fetch_add(relaxed)` 原子指令无锁选下标;③ "只选 READY"不在 Pick 里做,在构造 Picker 时就过滤掉了。

## 〇、一句话点破

> **gRPC 的负载均衡,核心是"控制平面/数据平面分离":SubChannel 状态变化时,在 work_serializer 里(慢路径、串行)重建一个新 Picker;每次调用来时,在调用线程(热路径、并发)只调一次 `Picker::Pick()` 选一个 SubChannel——这一步是无锁的,因为所有"该过滤谁、该按什么顺序"的复杂决策,都在慢路径构造 Picker 时做完了。**

这是结论,不是理由。本章倒过来拆:先讲为什么必须分离,再看各策略的定位,然后拆透 round_robin 的无锁 fast path,最后看新架构怎么把 Picker 包进 Promise。

## 一、为什么"挑后端"这么难

### 1.1 朴素做法的墙

设想你写一个最朴素的 LB:resolver 给了 N 个 SubChannel,每次调用随机选一个。看起来没问题,实际上处处撞墙:

1. **SubChannel 状态会变**:有些是 READY、有些是 IDLE、有些是 TF。直接随机可能选到一条根本没连接的 SubChannel,调用立刻失败。
2. **并发调用极多**:一个高 QPS 客户端每秒可能发起几十万次调用。每次选后端如果加锁、做复杂决策,锁竞争会成瓶颈。
3. **不同场景要不同策略**:无状态服务要轮询;有 CPU 反馈的服务要按权重;带 session 的要粘性;有故障的节点要剔除。一种策略打天下满足不了。
4. **后端列表会变**:resolver 随时推新地址,要平滑切换,不能让正在飞的调用挂掉。

gRPC 用一个统一架构解决这四件事,核心就是**控制平面/数据平面分离**。

### 1.2 控制平面/数据平面分离(整个 LB 设计的中枢)

`lb_policy.h:78-85` 文件注释把这件事说得直白:

> An LB policy has two parts:
>   - A LoadBalancingPolicy, which deals with the control plane work of managing subchannels.
>   - A SubchannelPicker, which handles the data plane work of determining which subchannel a given call should be sent on.

具体来说:

- **控制平面 = `LoadBalancingPolicy` 本体**:管 SubChannel 创建、订阅它们的状态变化、维护策略内部数据结构。所有这些操作都在 **work_serializer** 里串行执行(P4-13 讲过 work_serializer 是 gRPC 的"逻辑单线程")。慢、但安全。
- **数据平面 = `SubchannelPicker`**:每次调用要选 SubChannel 时,只调它的 `Pick(PickArgs)` 方法。这个方法**必须快、必须可并发调用**——它是每条调用的必经之路。

两者的桥梁是:**每当控制平面观察到 SubChannel 状态变化(或地址列表变化),就构造一个新 Picker,通过 `UpdateState` 原子替换全局可见的 Picker**。数据平面的并发调用要么用旧 Picker、要么用新 Picker,不会撞上中间态。

> **不这样会怎样**:如果让 `Pick()` 自己去检查所有 SubChannel 的状态、再决策,那么:① 每次调用都要遍历 SubChannel、读它们的状态,慢且要加锁;② SubChannel 状态在 `Pick()` 执行期间可能变化,要处理一致性问题;③ 不同策略的逻辑全挤在 `Pick()` 里,代码乱。
>
> 把"该选谁"的决策提前到"构造 Picker"这一步(慢路径),`Pick()` 只剩"按既定规则选一个",无锁、零分支、O(1)。这是用**空间换时间**的极致——同时可能存在多个 Picker(旧的还在被使用),但每个 Picker 都极简、独立、不可变。

### 1.3 接口定义

`SubchannelPicker` 基类(`lb_policy.h:282-290`):

```cpp
// src/core/load_balancing/lb_policy.h:282
class SubchannelPicker : public DualRefCounted<SubchannelPicker> {
 public:
  SubchannelPicker();
  virtual PickResult Pick(PickArgs args) = 0;   // ★唯一数据平面入口
 protected:
  void Orphaned() override {}
};
```

注意几个细节:

1. **`Pick()` 是唯一的纯虚方法**——数据平面就这么一个入口。控制平面事件(resolver 更新、SubChannel 状态)绝不进 `Pick`。
2. **`SubchannelPicker` 继承 `DualRefCounted`**(双引用计数),不是普通 RefCounted。这是为了支持"原子替换后,旧调用方安全持有旧 Picker 直到调用结束"——`DualRefCounted` 的强/弱分离让"已发布但还在被用"的 Picker 不会被销毁。
3. **`PickArgs`(`lb_policy.h:148-159`)没有锁、没有回调、没有 I/O 句柄**,只有 `{ path, initial_metadata, call_state }` 这些只读元数据。这是 `Pick()` 能无锁的前提——它不依赖任何会变化的共享状态。

**`PickResult`**(`lb_policy.h:201-270`)是一个 `std::variant<Complete, Queue, Fail, Drop>`:

- `Complete{subchannel, subchannel_call_tracker, metadata_mutations, authority_override}`:挑中了一个 SubChannel。
- `Queue`:暂时挑不了(没有 READY),让调用方排队等新 Picker。
- `Fail{status}`:策略明确拒绝(比如全 TF 时 `TransientFailurePicker`)。
- `Drop{status}`:主动丢弃(比如熔断、限流)。

### 2.3 各策略的定位

**`pick_first`**(最简单):连第一个能用的 SubChannel,所有调用都走它。

- 适用:直连固定后端、本地 socket、只有一个后端的场景。
- 行为:从地址列表第一个开始尝试,某个 SubChannel 进 READY 就锁定它(`pick_first.cc:704-723`),Picker 永远返回这个 SubChannel(`pick_first.cc:351-362`)。还实现了 Happy Eyeballs(IPv4/IPv6 并发竞速,`pick_first.cc:981-1006`)。
- 优点:连接数最少(只一条)。
- 缺点:多后端时不能负载均衡,后端故障要重新选。

**`round_robin`**(招牌,2.4 详述):READY 的 SubChannel 轮询。

- 适用:同质无状态后端。
- 行为:控制平面收集所有 READY 的 SubChannel,构造一个轮询 Picker;每次调用原子地轮到下一个。
- 优点:实现极简、无锁、负载均匀。
- 缺点:不考虑后端实际负载。

**`weighted_round_robin`**(WRR):按权重轮询,权重来自后端 ORCA 反馈。

- 适用:异构后端(不同 CPU/内存配置)、要按真实负载调流量。
- 权重公式(`weighted_round_robin.cc:513-546`):`weight = qps / (utilization + eps/qps * error_penalty)`。`qps`、`eps`(errors/sec)、`utilization`(CPU/内存/应用)都来自 `BackendMetricData`(服务端 ORCA 上报)。
- 两条上报通道:inline(每次 RPC 结束 `SubchannelCallTracker::Finish`,`weighted_round_robin.cc:585-604`)和 out-of-band(按 `weight_update_period_` 定时拉,`weighted_round_robin.cc:894-898`)。
- 算法核心是 `StaticStrideScheduler::Pick`(`weighted_round_robin/static_stride_scheduler.cc:162-195`),用 `scheduler_state_.fetch_add(1)` 原子序号源 + 概率跳过实现加权。

**`ring_hash`**(一致性哈希,XDS 粘性会话用):按请求 hash 映射到一致性哈希环。

- 适用:粘性会话(session sticky)、缓存亲和性。
- 行为:把所有 SubChannel 按 hash key 生成 N 个副本(replica,数 ∝ 权重),按 hash 排序成环(`ring_hash.cc:437-535`)。Picker 收到调用时,取请求 hash(从 `RequestHashAttribute` 或配置的 `request_hash_header_` 读),二分查找环上最近的副本(`ring_hash.cc:333-385`)。
- hash 算法是 **XXH64**(`ring_hash.cc:519`),副本 key = `地址 + "_" + 序号`。
- 加减后端只影响相邻段,这是"一致性"哈希的核心价值。

**`outlier_detection`**(装饰器):剔除异常后端,包裹一个 child policy。

- 适用:对故障后端做主动检测和剔除。
- 行为:周期性统计每个 SubChannel 的成功率/失败率(`EjectionTimer`,`outlier_detection.cc:824-1001`),按 `success_rate_ejection`(均值-stdev 阈值)和 `failure_percentage_ejection` 决定 eject。被 eject 的在控制平面就剔除出 child picker。
- 它的 Picker(`outlier_detection.cc:530-557`)委派给 child,只注入一个 call tracker 统计成败。

**`priority`**(分层 failover):按优先级维护多个 child,主 child 全 TF 时切到下一个。

- 适用:主备机房、灰度发布。
- 自己没有 Pick 逻辑,直接复用当前活跃 child 的 picker。

**`weighted_target`**(加权多目标):和 round_robin 类似维护多 child,按权重选 child 的 picker。

- 适用:多个目标按权重分流(比如 90% 流量到 A、10% 到 B)。

**`grpclb`**(已过时):老的服务端负载均衡方案,通过单独的 LB 服务器分发地址。

- 现在被 xDS 取代了,新场景不用。

**`rls_experimental`**:Route Lookup Service,按请求元数据查路由表(类似 Envoy 的 RLS)。

> **钉死这件事**:这些策略是**可组合**的——`priority` 和 `weighted_target` 的 child 可以是任意其他策略,`outlier_detection` 可以包任何策略。gRPC 在 xDS 场景下大量用这种组合来表达"按优先级 failover + 每层加权 + 异常剔除"的复杂路由(P6-22 详述)。

## 三、★Picker 无锁 fast path(本章核心技巧)

这一节拆透 round_robin 的 Picker。它只有 4 行核心代码,但这 4 行背后是 gRPC 在性能上最精妙的设计之一。

### 3.1 round_robin 的 Picker 完整实现

`round_robin.cc:141-198`。先看 Picker 类的字段(`round_robin.cc:141-155`):

```cpp
// src/core/load_balancing/round_robin/round_robin.cc:141
class Picker final : public SubchannelPicker {
 public:
  Picker(RoundRobin* parent,
         std::vector<RefCountedPtr<LoadBalancingPolicy::SubchannelPicker>>
             pickers);
  PickResult Pick(PickArgs args) override;
 private:
  RoundRobin* parent_;                                              // 不持 ref
  std::atomic<size_t> last_picked_index_;                          // ★唯一的共享状态
  std::vector<RefCountedPtr<LoadBalancingPolicy::SubchannelPicker>> pickers_;
};
```

**`Pick()` 方法完整代码**(`round_robin.cc:191-198`)——整个无锁 fast path:

```cpp
// src/core/load_balancing/round_robin/round_robin.cc:191
RoundRobin::PickResult RoundRobin::Picker::Pick(PickArgs args) {
  size_t index = last_picked_index_.fetch_add(1, std::memory_order_relaxed) %
                 pickers_.size();
  GRPC_TRACE_LOG(round_robin, INFO) << ...;
  return pickers_[index]->Pick(args);    // 委派给被选中 endpoint 的 child picker
}
```

逐原子操作解读:

1. **`last_picked_index_.fetch_add(1, std::memory_order_relaxed)`**——这是整个 fast path 唯一的同步原语。`fetch_add` 原子地"读旧值并 +1"。
2. **`std::memory_order_relaxed`**——关键。它只保证该原子操作本身的原子性,**不插入任何内存屏障、不建立 happens-before 序**。这是合法的,因为:
   - 选下标这一步不需要和其他内存写有序——`pickers_` 这个 vector 在 Picker 构造完成后从不修改(只读),读 `pickers_[index]` 不需要 fence 来"看见"最新值。
   - 多线程并发调用时,每个线程拿到的 `fetch_add` 返回值天然两两不同——这就是"轮流"语义,无需任何锁协调。
3. **`% pickers_.size()`**——取模回绕成环形下标。
4. **`return pickers_[index]->Pick(args);`**——委派给被选中 endpoint 的 child picker。child policy 一般是 `pick_first` 单连接。

> **钉死这件事(无锁的核心)**:整个 fast path **唯一的同步原语是一条 `fetch_add(relaxed)`**。在 x86 上它编译成单条 `lock xadd` 指令(无 fence);多核并发时,核间只在该 cache line 上做原子协议(MESI),没有锁、没有内核参与、没有上下文切换。这就是 gRPC 客户端能在每秒几十万次调用下,挑后端这一步几乎不耗时的根。

### 3.2 反例:朴素加锁会撞什么墙

> **不这样会怎样**:如果把 `Pick()` 改成加锁版本:
>
> ```cpp
> PickResult Pick(PickArgs args) {
>   MutexLock lock(&mu_);                        // 加锁
>   size_t index = last_picked_index_++ % pickers_.size();
>   return pickers_[index]->Pick(args);
> }
> ```
>
> 在高并发下会撞三堵墙:
>
> 1. **锁竞争**:QPS 高时所有调用线程串行化在一个临界区,吞吐被锁的拿/放开销主导,scaling 随核数变差(8 核不一定比 4 核快,因为都堵在锁上)。
> 2. **cache line 弹跳**:互斥量在核间传递时让 cacheline 反复失效,比单条原子指令(`lock xadd` 在自己的 cache line 上做 MESI)贵一两个数量级。
> 3. **抢占开销**:竞争激烈时操作系统会调度(上下文切换),远比原子指令慢。
>
> `relaxed fetch_add` 在 x86 上是单条 `lock xadd`,无 fence、无调度器参与。这就是 gRPC 选 atomic 而非 mutex 的全部理由——在每秒数十万次调用的热路径上,这个差距是数倍到数十倍吞吐的差别。

### 3.3 ★"只选 READY"不在 fast path,在构造 Picker 时就做完了

这是 round_robin 设计最巧的一点。**`Pick()` 里完全没有"检查 SubChannel 是不是 READY"的代码**——那 `pickers_` vector 里全是 READY 的 SubChannel 吗?

答案是:**控制平面构造 Picker 时就已经过滤好了**(`round_robin.cc:376-386`):

```cpp
// src/core/load_balancing/round_robin/round_robin.cc:376(简化示意)
std::vector<RefCountedPtr<LoadBalancingPolicy::SubchannelPicker>> pickers;
for (const auto& endpoint : endpoints()) {
  auto state = endpoint->connectivity_state();
  if (state.has_value() && *state == GRPC_CHANNEL_READY) {   // ★只挑 READY
    pickers.push_back(endpoint->picker());
  }
}
GRPC_CHECK(!pickers.empty());                                // 至少一个 READY 才到这
round_robin->channel_control_helper()->UpdateState(
    GRPC_CHANNEL_READY, absl::OkStatus(),
    MakeRefCounted<Picker>(round_robin, std::move(pickers)));
```

> **钉死这件事**:round_robin 的 fast path 之所以能零分支、零锁,**根本原因是所有"该不该选这条 SubChannel"的判断都提前到构造 Picker 时做了**。每当任何 SubChannel 状态变化(READY ↔ IDLE ↔ TF),控制平面就重建一个新 Picker,通过 `UpdateState` 原子替换。数据平面的并发调用要么用旧 Picker(里面是上一批 READY)、要么用新 Picker(里面是新一批 READY),永远不会撞上"中间态"。
>
> 这是一种**典型的不可变数据结构 + 写时复制**思路:Picker 一旦构造就不变,要变就整个换一个。这是函数式编程里常见的"持久化数据结构"思想在系统编程里的落地——和 P4-13 里 GlobalSubchannelPool 的 AVL 不可变快照是同一思路。

### 3.4 两级 Picker 结构(容易踩坑)

很多人(包括我)一开始以为 round_robin 的 Picker 持的是 `vector<SubchannelInfo>`。读完源码才知道**它持的是 `vector<RefCountedPtr<SubchannelPicker>>`**(`round_robin.cc:154`)。这意味着 round_robin 是**两级结构**:

- `RoundRobin`(top-level policy)→ 管理一个 `RoundRobinEndpointList`(`round_robin.cc:69`,继承自通用 `EndpointList`)。每个 endpoint 对应 resolver 给的一个 `EndpointAddresses`(可能含多个地址)。
- 每个 endpoint 内部又有自己的 **child policy**(见 `endpoint_list.h:165` `OrphanablePtr<LoadBalancingPolicy> child_policy_`)。child policy 默认是 `pick_first`(从 endpoint 的多个地址里挑一个能连上的)。
- endpoint 提供 child 的 picker(`endpoint_list.h:117-119` `picker()`)。
- round_robin 自己的 Picker 收集所有 READY endpoint 的 child picker,组成 `pickers_` vector。

为什么两级?因为现代 resolver(尤其 xDS)给的一个"endpoint"可能对应多个地址(比如同一服务的 IPv4 + IPv6,或同一 backend 的多副本)。round_robin 在 endpoint 层做轮询,endpoint 内部用 pick_first 选具体哪条连接。这种"组合 policy"的设计在 gRPC 里很常见。

### 3.5 随机起点:防 thundering herd

构造 Picker 时有一个细节(`round_robin.cc:182`):

```cpp
// src/core/load_balancing/round_robin/round_robin.cc:182
size_t index = absl::Uniform<size_t>(SharedBitGen(), 0, pickers_.size());
last_picked_index_.store(index, std::memory_order_relaxed);
```

`last_picked_index_` 的初始值是**随机的**,不是 0。注释(`round_robin.cc:180-181`)指向 grpc-go#2580:

> For discussion on why we generate a random starting index for the picker, see https://github.com/grpc/grpc-go/issues/2580.

> **不这样会怎样**:如果所有客户端都从 index 0 开始轮询,那么一批客户端同时启动时,它们的第一次调用**全部命中 backend 0**,造成 backend 0 瞬时过载、其他 backend 闲置。这就是 thundering herd。随机起点让每个客户端的起始位置散开,流量从一开始就分布均匀。

## 四、weighted_round_robin 的加权无锁算法

WRR 的 fast path **不是纯无锁**(注意与 round_robin 的差别),但真正的加权挑选用了一个巧妙的原子序号源。

### 4.1 fast path 的短锁

`WeightedRoundRobin::Picker::Pick`(`weighted_round_robin.cc:647-666`):

```cpp
// src/core/load_balancing/weighted_round_robin/weighted_round_robin.cc:647
WeightedRoundRobin::PickResult WeightedRoundRobin::Picker::Pick(PickArgs args) {
  size_t index = PickIndex();                              // 加权挑选
  ...
  auto result = endpoint_info.picker->Pick(args);          // 委派给 child
  if (!config_->enable_oob_load_report()) {
    auto* complete = std::get_if<PickResult::Complete>(&result.result);
    if (complete != nullptr) {
      complete->subchannel_call_tracker =
          std::make_unique<SubchannelCallTracker>(...);    // 挂上拿反馈
    }
  }
  return result;
}
```

`PickIndex`(`weighted_round_robin.cc:668-680`):

```cpp
// src/core/load_balancing/weighted_round_robin/weighted_round_robin.cc:668
size_t WeightedRoundRobin::Picker::PickIndex() {
  std::shared_ptr<StaticStrideScheduler> scheduler;
  {
    MutexLock lock(&scheduler_mu_);                        // ★短锁,只拷贝 shared_ptr
    scheduler = scheduler_;
  }
  if (scheduler != nullptr) return scheduler->Pick();
  return last_picked_index_.fetch_add(1) % endpoints_.size();  // fallback 纯 RR
}
```

> **钉死这件事(WRR vs RR 的差别)**:WRR 的 fast path **有锁**,但锁内只做一次 `shared_ptr` 拷贝(几纳秒),临界区极短。真正的加权挑选在锁外的 `StaticStrideScheduler::Pick()` 里。这种"锁内拿句柄、锁外干活"的模式在并发编程里很常见——把不可变的共享对象(`scheduler`)用 shared_ptr 保护,锁只保护指针赋值,不保护实际工作。

### 4.2 StaticStrideScheduler:加权无锁核心

真正的加权挑选在 `static_stride_scheduler.cc:162-195`:

```cpp
// src/core/load_balancing/weighted_round_robin/static_stride_scheduler.cc:162
size_t StaticStrideScheduler::Pick() const {
  while (true) {
    const uint32_t sequence = next_sequence_func_();        // ★原子递增
    const uint64_t backend_index = sequence % weights_.size();
    const uint64_t generation = sequence / weights_.size();
    const uint64_t weight = weights_[backend_index];
    const uint16_t kOffset = kMaxWeight / 2;
    const uint16_t mod =
        (weight * generation + backend_index * kOffset) % kMaxWeight;
    if (mod < kMaxWeight - weight) {
      continue;                                            // 按概率跳过
    }
    return backend_index;
  }
}
```

`next_sequence_func_` 在构造时绑定为(`weighted_round_robin.cc:711`):

```cpp
[this]() { return wrr_->scheduler_state_.fetch_add(1); }   // ★原子序号源
```

`scheduler_state_` 是 `std::atomic<uint32_t>`(`weighted_round_robin.cc:497-498`)。算法本质(注释在 :166-184):sequence 拆成低位的 `backend_index` 和高位的 `generation`。每个 backend 在 `kMaxWeight(=65535)` 代里被命中 `weight` 次。跳过概率 = `1 - mean/max`,典型约 20%。

这是一套**确定性加权 + 概率跳过**的混合算法——既保证长期权重比例精确(每个 generation 内每个 backend 被命中固定次数),又通过跳过避免短期固定模式。比朴素的"按 weight 累加选"更平滑。

## 五、ring_hash 的一致性哈希

ring_hash 适合需要粘性的场景。它的 Picker **没有锁也没有原子计数器**——因为下标完全由请求 hash 决定,是确定性映射,并发调用天然不冲突。

### 5.1 环的构建

`Ring::Ring`(`ring_hash.cc:437-535`)。关键副本生成(`ring_hash.cc:508-525`):

```cpp
// src/core/load_balancing/ring_hash/ring_hash.cc:508(简化示意)
for (size_t i = 0; i < endpoints.size(); ++i) {
  const std::string& hash_key = endpoint_weights[i].hash_key;
  ...
  target_hashes += scale * endpoint_weights[i].normalized_weight;
  size_t count = 0;
  while (current_hashes < target_hashes) {
    // 副本 key = hash_key + "_" + count
    const uint64_t hash = XXH64(hash_key_with_count.data(), ...);
    ring_.push_back({hash, i});                            // ★副本
    ++count; ++current_hashes;
  }
}
std::sort(ring_.begin(), ring_.end(), ...);                // 按 hash 排序成环
```

- hash 算法是 **XXH64**(非加密,极快)。
- 副本数 ∝ `scale × normalized_weight`,scale 在 `[min_ring_size, max_ring_size]`(默认 1024/4096)之间。
- 末尾 `std::sort` 得到有序环。

### 5.2 Picker:按请求 hash 二分查找

`RingHash::Picker::Pick`(`ring_hash.cc:333-385`)。请求 hash 来源(`ring_hash.cc:337-356`)两种模式:

- **XDS 模式**(无 header):从 call attribute `RequestHashAttribute` 取。
- **独立模式**:从配置的 `request_hash_header_` 读 header 再 XXH64;缺失则随机。

二分查找(`ring_hash.cc:358-385`,注释明确要求用有符号整数,移植自 ketama):

```cpp
// src/core/load_balancing/ring_hash/ring_hash.cc:358(简化示意)
const auto& ring = ring_->ring();
int64_t lowp = 0;
int64_t highp = ring.size();
int64_t index = 0;
while (true) {
  index = (lowp + highp) / 2;
  if (index == static_cast<int64_t>(ring.size())) { index = 0; break; }
  uint64_t midval = ring[index].hash;
  uint64_t midval1 = index == 0 ? 0 : ring[index - 1].hash;
  if (request_hash <= midval && request_hash > midval1) break;
  if (midval < request_hash) lowp = index + 1; else highp = index - 1;
  if (lowp > highp) { index = 0; break; }
}
```

之后从 `index` 起环形扫描第一个**已连接**的 endpoint(`ring_hash.cc:388-396`)——这处理"hash 命中的 endpoint 恰好 TF"的情况,顺延到环上下一个。

> **钉死这件事(ring_hash 的无锁)**:ring_hash 的 Picker 没有任何原子操作或锁——因为下标由请求 hash 完全确定(同一 hash 永远命中同一 backend),并发调用天然不冲突。这是 round_robin(动态轮询,要原子序号)和 ring_hash(静态映射,无需协调)的本质差别。

## 六、新架构 LoadBalancedCallDestination:不取代 Picker,是 Promise 化包装

很多人(包括一些 gRPC 内部文档)说新架构的 `LoadBalancedCallDestination`(LBDC)取代了 Picker。读完源码才知道**没有**——它把同步的 Picker 包成了 Promise/call_destination 形态。

### 6.1 LBDC 的形态

`load_balanced_call_destination.h:33-44`:

```cpp
// src/core/client_channel/load_balanced_call_destination.h:33
class LoadBalancedCallDestination final : public UnstartedCallDestination {
 public:
  explicit LoadBalancedCallDestination(ClientChannel::PickerObservable picker)
      : picker_(std::move(picker)) {}
  void StartCall(UnstartedCallHandler unstarted_handler) override;
 private:
  ClientChannel::PickerObservable picker_;
};
```

它持有 `PickerObservable = Observable<RefCountedPtr<SubchannelPicker>>`(`client_channel.h:42-43`)——**picker 还是那个 picker,只是变成了可观察的流**。

### 6.2 StartCall:Promise 形态调 Picker

核心在 `PickSubchannel`(`load_balanced_call_destination.cc:94-186`),真正调 Picker 的地方(`load_balanced_call_destination.cc:108`):

```cpp
// src/core/client_channel/load_balanced_call_destination.cc:108
auto result = picker.Pick(pick_args);    // ★这行就是 fast-path 入口,和经典一模一样
```

返回值通过 `HandlePickResult`(`load_balanced_call_destination.cc:60-85`)处理 Complete/Queue/Fail/Drop:

- **Complete** → 从 subchannel 取 `call_destination()`(`.cc:124-126`),若为空(SubChannel 已掉 READY 但 LB 还没换 picker)则 `return Continue{}` 排队重试(`.cc:131-137`)。
- **Queue** → `Continue{}`(`.cc:152-157`)。
- **Fail** → 非 wait_for_ready 直接返回错误(`.cc:159-175`)。
- **Drop** → 返回带 `kLbPolicyDrop` 标记的状态(`.cc:177-185`)。

整个调用怎么挂在 Promise 上(`StartCall` 的核心,`load_balanced_call_destination.cc:190-254`,简化示意):

```cpp
// 简化示意,非源码原文
unstarted_handler.SpawnGuardedUntilCallCompletes("lb_pick", [...]() {
  return Map(
    CheckDelayed(Loop(
      [last_picker, picker]() mutable {
        return Map(
          picker.Next(last_picker),              // ★等"新 picker"到达
          [...](RefCountedPtr<...Picker> picker) {
            last_picker = std::move(picker);
            return PickSubchannel(*last_picker, ...);   // 调 Pick
          });
      })),
    [...](pick_result) {
      (*call_destination)->StartCall(unstarted_handler);  // 交给 transport
      return absl::OkStatus();
    });
});
```

> **钉死这件事**:LBDC **没有取代 Picker**。`picker->Pick()` 这一行无论经典还是新架构都是 fast-path 的实际执行点。LBDC 做的是**把同步的 Pick 调用包进 Promise/Loop 形态**——`picker.Next(last_picker)` 等"新 picker 到达",Queue 时 `Loop` 返回 `Continue{}` 由新 picker 唤醒。这让 LB 这一步融入了 gRPC 新的 Promise-based call pipeline(P3-12 详述 call spine)。

### 6.3 经典 vs 新形态的对照

| 维度 | 经典(client_channel_filter) | 新(LoadBalancedCallDestination) |
|---|---|---|
| 调用线程 | 同步调 `picker->Pick()` | 在 Promise/Activity 里调 `picker->Pick()` |
| Queue 时 | 用 `grpc_lb_pick_data` 挂链表等新 picker | `Loop` 返回 `Continue{}`,由 `picker.Next()` 唤醒 |
| Picker 接口 | `SubchannelPicker::Pick`(同步) | **完全相同的接口** |
| 选中后 | SubChannel 自己有 transport | SubChannel 暴露 `call_destination()`,LBDC 转 call 给它 |
| 控制/数据分离 | 有 | 有 |

所以新架构**没有改变 LB 的核心设计**——控制平面/数据平面分离、Picker 无锁 fast path 都保留下来了。新形态只是把"调用链路"Promise 化,让 LB Pick 这一步能和 filter fusion(P3-11)等新机制融合。这是 gRPC 渐进式重构的典型风格——核心抽象稳定,外层包装演进。

### 6.4 picker 的原子替换

无论新旧架构,picker 的替换都是原子的。控制平面通过 `ChannelControlHelper::UpdateState` 最终调 `ClientChannel::UpdateStateAndPickerLocked`(`client_channel.cc:1434-1440`):

```cpp
// src/core/client_channel/client_channel.cc:1434(简化示意)
void ClientChannel::UpdateStateAndPickerLocked(
    grpc_connectivity_state state, const absl::Status& status,
    const char* reason,
    RefCountedPtr<LoadBalancingPolicy::SubchannelPicker> picker) {
  UpdateStateLocked(state, status, reason);
  picker_.Set(std::move(picker));    // ★写进 observable,唤醒所有等 Next 的 LBDC Loop
}
```

`picker_.Set(...)` 会唤醒所有正在 `picker.Next(last_picker)` 等"新 picker"的 LBDC Loop——被 Queue 的调用就用新 picker 重新 Pick。这是一个事件驱动的循环,不需要调用方阻塞或轮询。

## 七、技巧精解:Picker 无锁 fast path 的全部秘密

本章的核心技巧,值得单独钉死——它背后是几个深层设计的合力。

### 技巧核心:为什么 `fetch_add(relaxed)` 就够了

很多人看到 `std::memory_order_relaxed` 会紧张——"relaxed 不会出问题吗?会不会读到脏数据?"答案是**在这个场景下完全安全**,理由有三条:

1. **`pickers_` vector 在 Picker 构造后不变**。所以读 `pickers_[index]` 不需要 fence 来"看见最新值"——所有线程看到的都是同一个完整的 vector。
2. **`fetch_add` 本身保证原子性**。多线程并发调用时,每个线程拿到的返回值天然两两不同(原子指令的协议保证),所以不会有两个线程选到同一个 index(除非取模后碰巧相同,但那是轮询语义允许的)。
3. **不需要 happens-before 序**。"第 N 次调用"和"第 N+1 次调用"之间没有需要同步的其他内存操作——它们各自独立地选一个 SubChannel、发自己的调用。所以 relaxed 足够。

如果用 `seq_cst`(强一致)会怎样?在 x86 上没区别(x86 的原子操作默认有 seq_cst 语义),但在 ARM 等弱内存模型架构上,`seq_cst` 会插入额外的 fence,白白增加开销。`relaxed` 是**最少且足够的**内存序——这是 C++ 内存模型的精妙用法。

### 三层合力:为什么 fast path 能这么简

round_robin 的 `Pick()` 能简化到一条原子指令,是三层设计合力的结果:

1. **控制平面提前过滤**:"只选 READY"在构造 Picker 时就做完了(`round_robin.cc:376-382`),fast path 拿到的 vector 全是可用的。
2. **不可变 Picker + 写时替换**:Picker 一旦构造就不变,要变就整个换一个(通过 `UpdateState` 原子替换)。数据平面的并发调用要么用旧、要么用新,不会撞中间态。
3. **DualRefCounted 的强/弱分离**:旧 Picker 即使已经被替换,只要还有调用在用(strong ref > 0),就不会被销毁。`fetch_add` 在 Picker 的最后一个 strong ref 释放前都安全。

> **不这么设计会怎样**:三层里任何一层缺失,fast path 都简不下来:
> - 没有"控制平面过滤",`Pick()` 里要遍历 SubChannel、检查状态——慢且要加锁。
> - 没有"不可变 + 写时替换",`Pick()` 里要处理 vector 被并发修改的问题——要么加锁,要么用复杂的并发数据结构。
> - 没有"DualRefCounted",旧 Picker 可能在 `Pick()` 执行到一半被销毁——UB。
>
> 这三层合力,把"挑后端"这一最热的操作压到了**一条原子指令**。这是 gRPC 客户端能在百万 QPS 级别下,负载均衡本身不成为瓶颈的根。

### 一句话总结

> **Picker 的无锁 fast path,本质是"把所有该决策的事提前到慢路径做,快路径只执行既定决策"——这是 high-throughput 系统设计的通用范式**。Linux 内核的网络包处理、数据库的查询计划执行、Tokio 的 task 调度,都是同一思路:慢路径准备数据结构,快路径无锁执行。gRPC 的 Picker 是这个范式在客户端 RPC 负载均衡上的落地。

## 八、章末小结

### 回扣主线

本章是第 4 篇(客户端治理)的第三章,讲的是治理链的**决策核心**:多个 SubChannel 摆在面前,这次调用挑哪个。它服务二分法的**框架层(治理)**——LB 策略自己不碰协议层(不编 HTTP/2 帧、不发包),但它决定"这次调用发去哪条连接",这是治理的核心决策。

P4-14 给了 SubChannel(地址 → 可用连接),本章讲"多个连接挑哪个";下一章 P4-16 讲"挑了发出去了,失败了怎么办"。

### 五个为什么

1. **为什么 LB 要分控制平面和数据平面?**——控制平面(管 SubChannel)慢但安全,数据平面(每次调用选 SubChannel)快且并发。把复杂决策提前到控制平面,数据平面才能简化到一条原子指令。这是 high-throughput 系统设计的通用范式。
2. **为什么 round_robin 的 Pick 只用 `fetch_add(relaxed)`?**——`pickers_` 构造后不变,选下标不需要和其他内存写有序;`fetch_add` 原子性保证多线程选到不同下标。relaxed 是最少且足够的内存序,在 ARM 等弱内存模型上比 seq_cst 省掉 fence。
3. **为什么"只选 READY"不在 fast path 做?**——状态过滤提前到构造 Picker 时(round_robin.cc:376-382),fast path 拿到的 vector 全是 READY。这把 O(N) 状态检查压到 O(1) 取模,且让 Picker 不可变。
4. **为什么 C 核心没有 least_request?**——历史上没实现,语义替代品是 `weighted_round_robin`(基于后端 ORCA 反馈动态调权重)。Java/Go 才有 least_request,不要从那边搬过来。
5. **为什么 LoadBalancedCallDestination 不取代 Picker?**——Picker 的设计(控制/数据分离、无锁 fast path)是稳定的;LBDC 只是把同步 Pick 包进 Promise/Loop,让 LB 这一步融入新的 call spine 架构。这是渐进式重构:核心抽象稳定,外层包装演进。
