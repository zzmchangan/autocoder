# 《Linux 容器与命名空间深入浅出:cgroup v2 + namespace 怎么造出容器》—— 全书总纲

> 一本写给"用过 Docker / K8s、甚至翻过 `kernel/cgroup/` 和 `kernel/nsproxy.c`,却总觉得一知半解"的人的小书。
>
> 本文是全书的**总纲**:定主线、定方法、定分篇分章、定源码策略、定写作约定。
> 与《Tokio》《LevelDB》《etcd》《Go runtime》《内存分配器》《Linux mm》《Linux 调度器》同源,都是"**第一性原理 + 真实源码**";命脉同样是**动机(why) + 技巧(how) 双线**;基调延续《LevelDB》——**直球讲透为主,比喻只在反直觉处点睛**。
> **本书独有特色:★ 对照 runc / 容器运行时(用户态编排内核能力造容器)、Docker / K8s**——内核只提供"积木(namespace + cgroup)",真正造出容器的是用户态的运行时(runc 按 OCI 规范,clone+unshare+setns+写 cgroup 文件把它们组装起来)。两本(内核能力 vs 运行时)合一,才是"容器全栈"。

## 一、这本书的定位

### 目标读者

会写 C、用过 Linux、敲过 `docker run`/`kubectl`、读过容器原理博客,甚至翻过 `kernel/cgroup/cgroup.c`、`kernel/nsproxy.c`,但**一知半解**:知道 namespace 能隔离、cgroup 能限额,却讲不清一次 `clone(CLONE_NEWNS | CLONE_NEWPID | CLONE_NEWNET | ...)` 内核到底造出了什么、`task_struct` 里的 `nsproxy` 怎么切换、`css_set` 为什么是一组指针、`cgroup.procs` 写一个 PID 后内核走的是哪条迁移链、memcg 的 charge 怎么落到 page、cpu cgroup 的 `cpu.max` 怎么变成 throttle、user namespace 的 uid_map 凭什么让"容器里 root = 宿主 nobody"。和"读过 tokio/go runtime/mm/sched 源码但不理解"是同一种病。

### 全书一句话主旨

> **容器不是一个新东西,而是一个"看起来独占整机"的进程——靠 namespace 改它的视图(看不到别人)、靠 cgroup 限它的资源(吃不到别人),两者一夹,就是一个盒子。namespace 管它"看见什么",cgroup 管它"能用多少"。**

每个组件,都是这条主线上的一个**驿站**。读完一本书,读者能**在脑子里放映出容器从无到有的全过程**:一次 `clone(CLONE_NEW*)` 怎么造出 7 种命名空间、`nsproxy` 怎么挂到 `task_struct` 上、`unshare`/`setns` 怎么改它的视图、cgroup 怎么把 CPU/内存/IO/PID 限额层层套到一个目录上、`cgroup.attach_task` 怎么把一个进程迁过去——以及每一步底下用了什么巧妙的手段。

### 为什么用"cgroup + namespace 怎么造出容器"当骨架(而非按文件平铺)

容器涉及两个**完全独立**的子系统:cgroup(资源控制,在 `kernel/cgroup/` + 各 controller)和 namespace(视图隔离,分散在 `kernel/nsproxy.c`、`kernel/utsname.c`、`kernel/pid_namespace.c`、`kernel/user_namespace.c`、`net/core/net_namespace.c`、`fs/namespace.c`、`ipc/namespace.c`)。按文件百科式平铺,读者会觉得 namespace 七种 + cgroup 七个 controller 是一堆散点。用"一个进程怎么被关进盒子"这条主线当骨架——先讲视图隔离(namespace:它看到的世界)、再讲资源控制(cgroup:它能用多少)、最后讲组装(运行时把它们拼起来造容器)——每个机制都是"为了让这个盒子成立"而出现的角色,读者永远知道**我在哪、为什么需要下一站**。

## 二、写作方法

### 命脉:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与一般容器原理书、源码导读最关键的分野,所有写作者必读、必守。**

Linux 容器源码充斥内核级技巧:`nsproxy` 一次性聚合 7 种命名空间指针(`copy_namespaces` 检测 `CLONE_NEW*` 决定拷还是共享)、`switch_task_namespaces` 用 `task_lock` 而非全局锁切视图、cgroup 用 `css_set`(一组 css 指针)做"任务↔cgroup 多对多"的去重表、`css_set` 用哈希表 + refcount 让相同归属的进程共享一个 cset、`cgroup_attach_task` 的四步迁移(add_src → prepare_dst → migrate → finish)用 `cgroup_threadgroup_rwsem` 保证 fork 不抢跑、memcg 的 charge 走 page→memcg 反查、cpu cgroup 复用调度器的 `sched_entity` 组调度 + bandwidth throttle、user namespace 的 uid_map 把"内核里的 kuid_t"和"用户态看到的 uid"解耦。读者"一知半解"的缺口往往**不在**"它要解决什么"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?(沿用系列一贯的"为什么优先")
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果不用这个技巧、朴素地写,会撞上什么墙",让妙处显形。

一句话:**讲不清技巧,等于没讲 Linux 容器。**

### 基调:直球讲透为主,比喻只在反直觉处点睛 ⚠️ 延续《LevelDB》/《Linux mm》

容器有一部分反直觉(namespace 切了视图但没切物理、uid_map 让 root 变 nobody、cgroup 的层级 vs flat),这部分**允许适度比喻点睛**(盒子、视图、限额、映射表)。但容器的大部分(`nsproxy` 字段、`css_set` 哈希、`cgroup_attach_task` 的迁移链、memcg charge 路径)**具体可见,直球讲清最好**。

- **默认直球**:能用表 / 流程图 / 源码说清的,不上比喻。
- **比喻只点睛**:namespace 可点"像给进程戴上一副眼镜,只看见自己盒子里的东西";cgroup 可点"像给进程发一张配额卡,超了就 throttle"。点到为止。
- **判断标准**:直球讲不清才上比喻。

### ★ 对照 runc / Docker / K8s:本书独有特色

> 本书和用户态的《runc / 容器运行时》是一对——内核只提供"积木"(namespace 系统调用 + cgroup 文件接口),真正把这些积木组装成"容器"的是用户态运行时。runc 按 OCI 规范,大致这么造一个容器:① `clone(CLONE_NEWNS | CLONE_NEWPID | CLONE_NEWNET | CLONE_NEWIPC | CLONE_NEWUTS | CLONE_NEWUSER)` 创出新命名空间;② `unshare`/`setns` 微调视图;③ 把进程的 PID 写进各 cgroup 的 `cgroup.procs`(触发 `cgroup_attach_task`);④ pivot_root 换根文件系统。Docker/K8s 在更上层做镜像、编排、调度。

本书在 namespace 各章、cgroup 各章末用轻量"★对照"小栏,告诉读者"这个内核能力,runc 是怎么用的";收尾章给总表——内核能力 → 运行时接口 → 容器效果。对照要点:`clone(CLONE_NEW*)` ↔ runc 的 `create`;`cgroup.procs` 写 ↔ runc 的资源限额设置;`pivot_root`/`set_mnt_ns` ↔ runc 的 rootfs 切换;`setns` ↔ `docker exec`/`kubectl exec` 进入容器。

## 三、全书主线、二分法

### 主线

> **容器 = 一个被关进"看起来独占整机"盒子的进程。namespace 改它的视图(看不见别人),cgroup 限它的资源(吃不到别人),两者一夹,就是容器。**

任何一处看不懂某个机制,回到这句问:"这是在**改它的视图(namespace:它看到什么)**,还是在**限它的资源(cgroup:它能用多少)**,或是在**把它们组装起来造容器(clone/unshare/setns + 写 cgroup 文件)**?"把答案写进"为什么"里。

### 二分法(迷路时回到它)

本书的二分法:

> **资源控制(cgroup:cpu/memory/io/pids/freeze 等限额) vs 视图隔离(namespace:mnt/pid/net/uts/ipc/user/cgroup 看到什么)**。

- **资源控制(cgroup)**:`cgroup_attach_task`(把进程归到某 cgroup)、cpu 子系统(`cpu.max`/`cpu.weight` → 调度器 throttle,回扣《调度器》P6-19)、memory 子系统(memcg,charge 每个 page → OOM,回扣《mm》)、io 子系统(`io.max` 限 IOPS/BPS)、pids(进程数上限)、freezer(冻结整组)、cpuset(绑核/绑内存节点)。这些回答**"能用多少"**。
- **视图隔离(namespace)**:`nsproxy`(总入口,聚合 7 种 ns 指针)、mnt namespace(挂载视图)、pid namespace(进程号视图,容器里 PID 1)、net namespace(独立网络栈:网卡/路由/iptables/socket,回扣《网络栈》)、uts namespace(hostname)、ipc namespace(SysV IPC/POSIX 消息队列)、user namespace(uid 映射,安全)。这些回答**"看到什么"**。

支撑这两者的地基:`task_struct->nsproxy`(视图指针)和 `task_struct->cgroups`(`css_set` 指针)、`struct nsproxy`(7 种 ns 指针聚合)、`struct css_set`(去重表,一组 css 指针)、`copy_namespaces`/`cgroup_post_fork`(fork 时的拷贝/挂靠)、`switch_task_namespaces`/`cgroup_attach_task`(运行时切换)。

容器每个机制都落在这两面之一或组装层。任何时候迷路,回到这个二分法。

### 第 0 篇 · 开篇:为什么需要容器(定调,1 章)

- **第 1 章 · 第一性原理:为什么需要容器与命名空间**(`P0-01`):云原生基石(隔离 + 限额 + 可迁移),朴素虚拟机太重(全机器仿真)、裸进程太散(没隔离没限额)→ 内核提供两类原语:**namespace(视图隔离)+ cgroup(资源控制)**→ 立起"视图 vs 资源"二分法 + ★对照 runc(用户态造容器)。**【样本章节】**

### 第 1 篇 · namespace 视图隔离:进程看到什么(7 章)

> 源码主体 `kernel/nsproxy.c`(总入口)、各 namespace 的 `kernel/utsname.c`/`kernel/pid_namespace.c`/`kernel/user_namespace.c`/`net/core/net_namespace.c`/`fs/namespace.c`/`ipc/namespace.c`、`include/linux/nsproxy.h`/各 `*_namespace.h`。

- **第 2 章 · `nsproxy`:namespace 的总入口**(`P1-02`)——`task_struct->nsproxy` 一个指针聚合 7 种 ns;`copy_namespaces`(fork 时按 `CLONE_NEW*` 决定拷/共享)、`switch_task_namespaces`(运行时切视图)、`create_new_namespaces`(逐个调 `copy_*_ns`)。源码:`include/linux/nsproxy.h`、`kernel/nsproxy.c`。**技巧精解:`nsproxy` 聚合 + `CLONE_NEW*` 标志位驱动**(一次系统调用切多种视图,反面对比"每种 ns 单独系统调用")。归属:`视图(支撑)`。
- **第 3 章 · mnt namespace:挂载视图**(`P1-03`)——容器看到自己的根文件系统;`copy_mnt_ns` 复制挂载树(`copy_tree`/`clone_mnt`)、`pivot_root`/`chroot` 换根、挂载传播(shared/slave/private)。源码:`fs/namespace.c`、`fs/mount.h`。**技巧精解:`copy_tree` 整树复制 + 挂载传播类型**(反面对比"无传播类型,A 容器挂个卷 B 容器也看见")。归属:`视图`。
- **第 4 章 · pid namespace:进程号视图**(`P1-04`)——容器里 PID 1;`struct pid_namespace` 层级(父子链)、`copy_pid_ns`、进程在多层 pid ns 里各有不同 PID、孤儿进程过继给同 ns 的 subreaper。源码:`kernel/pid_namespace.c`、`include/linux/pid_namespace.h`、`kernel/pid.c`。**技巧精解:pid ns 层级 + `pid->numbers[]` 多层 PID**(一个进程在不同层 ns 有不同号,反面对比"全局唯一 PID 容器里冲突")。归属:`视图`。
- **第 5 章 · net namespace:网络栈视图**(`P1-05`)——容器有独立网卡/路由表/iptables/socket;`copy_net_ns`/`setup_net`、`struct net` 是网络栈的根、veth pair 跨 ns 连通、per-netns 的 pernet_ops 列表初始化各协议栈。源码:`net/core/net_namespace.c`、`include/linux/net_namespace.h`。**技巧精解:pernet_ops 链表批量初始化网络子系统**(每个 net ns 创建时遍历所有协议模块的 init,反面对比"每加一个协议要改 copy_net_ns")。回扣《网络栈》。归属:`视图`。
- **第 6 章 · uts namespace:hostname 视图**(`P1-06`)——容器有独立 hostname/domainname;`struct new_utsname`(nodename/release/version 等)、`copy_utsname`。源码:`kernel/utsname.c`、`include/linux/utsname.h`。**技巧精解:uts ns 为何独立于其他**(简单到几乎没有逻辑,但揭示了"ns 是按视图维度切分"的设计哲学)。归属:`视图`。
- **第 7 章 · ipc namespace:System V IPC 视图**(`P1-07`)——独立的消息队列/信号量/共享内存;`copy_ipcs`、`struct ipc_namespace`。源码:`ipc/namespace.c`、`include/linux/ipc_namespace.h`。**技巧精解:ipc ns 的 `ids` 表隔离**(每个 ns 一组 IPC `ids`,反面对比"全局 IPC key 冲突")。归属:`视图`。
- **第 8 章 · user namespace:uid 映射与安全**(`P1-08`)——★ 容器安全的基石;容器里 uid 0 = 宿主 uid 100000;`create_user_ns`、`struct user_namespace` 的 `uid_map`/`gid_map`、`map_write`、kuid_t 内核全局 uid 与 ns 局部 uid 的转换(`make_kuid`/`from_kuid`)、capability 在 user ns 内有效。源码:`kernel/user_namespace.c`、`include/linux/user_namespace.h`。**技巧精解:kuid_t 解耦 + uid_map 映射表**(反面对比"全局 uid,容器 root 直接是宿主 root,容器逃逸即接管整机")。归属:`视图(安全)`。

### 第 2 篇 · cgroup 资源控制:进程能用多少(6 章)

> 源码主体 `kernel/cgroup/cgroup.c`(cgroup v2 核心)、`include/linux/cgroup-defs.h`(`struct cgroup`/`css_set`/`cgroup_subsys_state`/`cgroup_subsys`)、`include/linux/cgroup_subsys.h`(controller 列表)、各 controller(`mm/memcontrol.c`、`kernel/sched/`(cpu 复用调度器)、`block/blk-cgroup.c`(io)、`kernel/cgroup/pids.c`、`kernel/cgroup/cpuset.c`、`kernel/cgroup/freezer.c`)。

- **第 9 章 · cgroup v2 概览:统一层级与 `css_set`**(`P2-09`)——cgroup v2 单一根(unified hierarchy)、`struct cgroup`(层级树)、`struct css_set`(一组 css 指针的任务去重表)、`struct cgroup_subsys_state`(每个 cgroup×controller 一个 css)、`cgroup_subsys.h` 列出 15 个 controller、`cgroup_init`/`cgroup_init_subsys`。源码:`kernel/cgroup/cgroup.c`、`include/linux/cgroup-defs.h`、`cgroup_subsys.h`。**技巧精解:`css_set` 去重表 + 哈希查找**(任务↔cgroup 多对多用一张去重表,反面对比"每个任务每个 controller 一组指针,内存爆炸")。归属:`资源(支撑)`。
- **第 10 章 · cgroup_attach_task:进程怎么迁进 cgroup**(`P2-10`)——写 `cgroup.procs` 触发 `__cgroup_procs_write` → `cgroup_attach_task`;四步迁移:add_src(找源 css_set)→ prepare_dst(`find_css_set` 找/建目标 css_set)→ migrate(`css_set_move_task`)→ finish(放引用);`cgroup_threadgroup_rwsem` 保证 fork/exit 不抢跑;`can_attach`/`attach`/`post_attach` 回调。源码:`kernel/cgroup/cgroup.c`(L2866 `cgroup_attach_task`、L1163 `find_css_set`、L870 `css_set_move_task`)。**技巧精解:四步迁移 + `find_css_set` 复用**(同类归属的任务共享一个 css_set,反面对比"每次迁移都新建 css_set,内存/锁爆炸")。归属:`资源(机制)`。
- **第 11 章 · cpu 子系统:cpu.max 与 bandwidth throttle**(`P2-11`)——★ 回扣《调度器》P6-19;`cpu.max`(quota/period)→ cfs_bandwidth → 超额 `throttle_cfs_rq`;`cpu.weight` → 调度权重;组调度复用 `sched_entity`。源码:`kernel/sched/core.c`(cgroup 接口)、`kernel/sched/fair.c`(`task_group`/`sched_entity` 组调度/`throttle_cfs_rq`)、`kernel/sched/cpuacct.c`。**技巧精解:组调度复用 `sched_entity` + bandwidth throttle**(任务和组用同一个 `sched_entity`,反面对比"两套数据结构维护两套调度逻辑")。归属:`资源`。
- **第 12 章 · memory 子系统(memcg):memory.current 与 charge/OOM**(`P2-12`)——★ 回扣《mm》;每个 page 分配时 `try_charge`(memcg),超 `memory.max` → OOM kill(容器内杀进程);`memory.swap.max`、`memory.oom.group`;层级化 charge(子 cgroup 用量累加到父)。源码:`mm/memcontrol.c`(`mem_cgroup_try_charge`/`mem_cgroup_object_signed`)、`include/linux/memcontrol.h`。**技巧精解:page→memcg 反查 + 层级累加**(每个 page 记录所属 memcg,反面对比"按进程记账,共享内存/fork 后页算不清")。归属:`资源`。
- **第 13 章 · io 子系统:io.max 与 IOPS/BPS 限额**(`P2-13`)——★ 回扣《块设备》;`io.max`(rbps/wbps/riops/wiops)→ blk-iocost/blk-iolatency/throttle;per-device 限额;`io.stat` 统计。源码:`block/blk-cgroup.c`、`block/blk-iocost.c`、`block/blk-iolatency.c`、`include/linux/blk-cgroup.h`。**技巧精解:per-device per-cgroup 限额 + iocost 模型**(把 IO 代价抽象成 token,反面对比"朴素按字节数限,SSD 的随机 4K 和顺序 1M 不公平")。归属:`资源`。
- **第 14 章 · pids/freezer/cpuset:进程数、冻结、绑核**(`P2-14`)——`pids.max` 限进程数(`css->counter`)、`freezer` 冻结整组(SIGSTOP 不到 cgroup 子进程,freezer 走任务冻结)、cpuset 绑核/绑内存节点(`cpuset.cpus`/`cpuset.mems`)。源码:`kernel/cgroup/pids.c`、`kernel/cgroup/freezer.c`、`kernel/cgroup/cpuset.c`。**技巧精解:freezer 的 task_lock + TIF_FROZEN 状态机**(整组冻结,反面对比"逐个 kill -STOP,race 期间新 fork 不被冻")。归属:`资源`。

### 第 3 篇 · 容器组装:把它们拼起来(3 章)

> 源码主体 `kernel/fork.c`(clone)、`kernel/nsproxy.c`(unshare/setns)、`kernel/cgroup/cgroup.c`(procs_write)、`fs/namespace.c`(pivot_root)。

## 七、内核 C 技巧侧重(对标 Tokio Rust / Go runtime / LevelDB C++ / mm / 调度器 内核 C)

本书要重点拆的 Linux 内核容器技巧(写作时遇到这些,务必钻"怎么做到的 + 为什么 sound"):

- **`nsproxy` 聚合 + `CLONE_NEW*` 标志位**:一次 clone 创多种 ns,中间没有"半新半旧"状态——视图切换的原子性靠 `create_new_namespaces` 全成或全回滚。
- **`switch_task_namespaces` 用 `task_lock`**:切视图只需锁任务本身(`task_struct->alloc_lock`),不是全局锁——切换并发性。
- **`css_set` 去重表**:任务↔cgroup 多对多用一张哈希表(`css_set_table`)去重,相同归属的任务共享一个 `css_set`——内存 + 迁移性能。
- **`css_set` 引用计数 + RCU**:`put_css_set` 异步释放,迭代器(`css_task_iter`)用 RCU 安全遍历——并发迁移 + 迭代不互锁。
- **`cgroup_attach_task` 四步迁移**:add_src → prepare_dst → migrate → finish,配 `cgroup_threadgroup_rwsem` 保证 fork/exit 不抢跑——迁移的原子性。
- **`cgroup_subsys` 函数指针多态**:每个 controller 一个 `struct cgroup_subsys`,实现 css_alloc/css_online/attach/can_fork 等回调——controller 可插拔(cgroup v2 的核心架构)。
- **memcg 的 page→memcg 反查**:每个 page 记 `page->mem_cgroup`,分配时 charge,释放时 uncharge——共享内存/fork 后的页也能正确记账。
- **memcg 层级累加**:子 cgroup 用量自动累加到所有祖先——树状资源约束。
- **cpu cgroup 复用 `sched_entity` 组调度**:任务和组用同一个 `sched_entity`,`cpu.max` throttle 走 `throttle_cfs_rq`——和调度器一套数据结构。
- **user ns 的 kuid_t 解耦**:内核全局 kuid_t 与 ns 局部 uid 用 `uid_map`/`gid_map` 映射,容器 root 在宿主是 nobody——容器安全的基石。
- **net ns 的 pernet_ops 链表**:每个 net ns 创建时遍历所有协议模块的 init/exit 回调——网络协议可插拔。
- **mnt ns 的挂载传播**:shared/slave/private 三种传播类型,`copy_tree` 整树复制——容器挂载隔离 + 共享卷双向可见的精细控制。
- **cgroup v2 单一层级 + no internal process**:所有 controller 共用一棵树,cgroup 要么有子进程要么启用 controller——避免 v1 的归属矛盾。
- **setns 的两阶段**:prepare_nsset 全部成功才 commit_nsset——切视图不会半新半旧。
- **freezer 的 task_lock + TIF_FROZEN**:整组冻结靠每个任务的 frozen 状态机——比 SIGSTOP 安全(race 期间新 fork 也被冻)。
