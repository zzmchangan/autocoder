# 第五章 · 安全性与持久化

> 篇:第 1 篇 · Raft 地基(共识协议层)
> 主线呼应:前四章我们一块一块搭起了 Raft——term 单调、一个 term 一票、leader 选举、log matching、commit 要多数派、Figure 8 的"只 commit 当前 term"。这一章是第 1 篇的收束章,把那些散落的规则**收成一句话**:"**一条 entry 一旦 commit,就永远不丢。**"凭什么?靠两条东西——**选举限制**(选出的 leader 必含所有已 commit 的 entry)和**持久化要求**(`currentTerm`/`votedFor`/`log` 必须在响应前落盘)。前者让"换 leader"这件事不破坏已 commit 的数据,后者让"节点崩溃重启"这件事不破坏投票和日志的不变式。etcd-raft 用 unstable/stable 的分离,把"还没落盘的内存 entry"和"已落盘的 entry"在数据结构上隔开,让协议状态机在崩溃前后的世界都自洽。

## 核心问题

**一条 entry 被 commit 之后,凭什么说它"永不丢失"?换 leader 不会丢吗?节点崩溃重启不会丢吗?——本章把 Raft safety 收束成一条核心保证,并把它拆成两条机制的因果链:选举限制(让 leader 完整)+ 持久化(让崩溃前后状态自洽)。**

读完本章你会明白:

1. **选举限制**凭什么保证选出的 leader 一定拥有所有已 commit 的 entry——`isUpToDate` 这一行代码,加上"一个 term 一票 + commit 要多数派",怎么一步步推出"已 commit entry 必在新 leader 的 log 里"。
2. **Leader 完整性(Leader Completeness)** 是 Raft 最核心的 safety 保证,它怎么从选举限制 + 一个 term 一票 + 多数派 commit 这三条规则数学地推出来。
3. 为什么 Raft **必须**持久化 `currentTerm`/`votedFor`/`log` 三样才安全——不持久化哪一样会破坏哪个不变式?
4. **unstable/stable 分离**在解决什么:没落盘的 entry 放 `unstable`,落盘后才并进 stable log;凭什么丢 unstable 不破坏 safety。

> **如果一读觉得太难**:先只记住三件事——
> ① 投票时,投票者只给"日志至少和自己一样新"的 candidate 投票(`isUpToDate`),这条把"落后节点"挡在 leader 门外。
> ② 配合"一个 term 一票 + commit 要多数派",能推出任何被选出的 leader 必含所有已 commit 的 entry——这就是 leader 完整性。
> ③ Raft 要求 `currentTerm`/`votedFor`/`log` 在响应任何 RPC 前落盘,etcd-raft 用 `unstable`(内存)/`stable`(已落盘)两段把这件事在数据结构上显式分开。
> 这三件是后面所有章节的根。读不懂完整证明,记住这三条结论也能往下走。

## 5.1 一句话点破

> **Raft 最重要的一条保证是"已 commit 的 entry 永不丢失"。这条保证不是单一机制做出来的,而是三条规则联手的结果:① 选举限制——投票者只给日志"至少和自己一样新"的 candidate 投票;② 一个 term 一个节点最多投一票;③ commit 要多数派。三者联动,推出"任何被选出的 leader 都必含所有已 commit 的 entry"(leader 完整性);再配合"currentTerm/votedFor/log 必须在响应前落盘"这条持久化要求,让节点崩溃前后都满足这些不变式——于是 commit 之后,无论怎么换 leader、怎么崩溃重启,这条 entry 都丢不了。**

这是结论,不是理由。本章倒过来拆:先看选举限制这一行代码在干什么(5.2),再看它怎么联手两条规则推出 leader 完整性(5.3),然后看为什么 leader 上任时不直接 commit 旧 term 的 entry(5.4,回扣 Figure 8),最后把"持久化"这条要求和 unstable/stable 分离摆出来(5.5~5.6),全章收束到 Raft safety 的核心。

## 5.2 选举限制:这一行 `isUpToDate` 在防什么

candidate 发起选举时,它在 `MsgVote` 里附上**自己 log 最后一条 entry 的 (term, index)**;投票者收到后,先比"candidate 的 log 是不是至少和我一样新",够新才投。这段逻辑的代码就在 [`raft.go` 处理 `MsgVote` 的分支](../etcd-raft/raft.go#L1212-L1262)(@`39eb80a`):

```go
case pb.MsgVote, pb.MsgPreVote:
    // We can vote if this is a repeat of a vote we've already cast...
    canVote := r.Vote == m.GetFrom() ||
        (r.Vote == None && r.lead == None) ||
        (m.GetType() == pb.MsgPreVote && m.GetTerm() > r.Term)
    // ...and we believe the candidate is up to date.
    lastID := r.raftLog.lastEntryID()
    candLastID := entryID{term: m.GetLogTerm(), index: m.GetIndex()}
    if canVote && r.raftLog.isUpToDate(candLastID) {
        r.send(&pb.Message{To: m.From, Term: m.Term, Type: voteRespMsgType(m.GetType()).Enum()})
        if m.GetType() == pb.MsgVote {
            r.electionElapsed = 0
            r.Vote = m.GetFrom()
        }
    } else {
        r.send(&pb.Message{To: m.From, Term: new(r.Term), Type: voteRespMsgType(m.GetType()).Enum(), Reject: new(true)})
    }
```

关键就是那一行 `r.raftLog.isUpToDate(candLastID)`。它的实现在 [`log.go:442`](../etcd-raft/log.go#L442-L445):

```go
func (l *raftLog) isUpToDate(their entryID) bool {
    our := l.lastEntryID()
    return their.term > our.term || their.term == our.term && their.index >= our.index
}
```

判定的口诀是 Raft 论文 5.4.1 那段:**先比最后一条 entry 的 term,term 大的更新;term 相同时,index 大(即 log 更长)的更新**。candidate 的 log 比投票者"至少一样新",才拿到这张票。`candidate 的 (term, index)` 怎么填的?candidate 在发起选举时,把自己 log 的最后一条 entry 的 ID 塞进 `MsgVote`([`raft.go:1063-1071`](../etcd-raft/raft.go#L1063-L1071)):

```go
last := r.raftLog.lastEntryID()
r.send(&pb.Message{
    To: new(id), Term: new(term), Type: voteMsg.Enum(),
    Index: new(last.index), LogTerm: new(last.term), ...
})
```

> **不这样会怎样**:假设没有 `isUpToDate` 这道关,只要 term 够大、节点能凑够票数就能当 leader。考虑这样一个场景(5 节点 A/B/C/D/E):
>
> - A 当 leader 时,`entry#7(term=3)` 被 {A,B,C} 三个节点复制,**已 commit**。
> - 然后 B、C、D 各自因为某些原因没收到后续的复制消息(网络抖动),它们的 log 停在 `entry#7`。A 的 log 已经到了 `entry#10`。
> - D 是个**全新加入的节点,log 完全为空**(或者只到 `entry#3`)。
> - A 挂了,发起选举。D 也发起选举——D 的 term 抢得高(比如它刚被重启,term=5),D 向 E、B、C 拉票。
> - 若没有 `isUpToDate` 限制:D 的 term=5 够高,E 投它,加上它自己那一票 + 一个别的节点(只要凑够 3 票)就当上了 leader。
> - **但 D 的 log 里根本没有已 commit 的 `entry#7`!** D 当上 leader 后,从它的(短)log 往后追加,会用 `entry#4` 这种 D 自以为是的内容覆盖 B、C 已有的 `entry#7`——**已 commit 的 entry 被覆盖丢失**。
>
> 这就是"选出落后 leader"的灾难。`isUpToDate` 这一行,就是把这个灾难堵在投票那一刻:**我手里的 log 都比你的新(或一样新),我才投你**。D 这种 log 空白的节点,除非它的 (term,index) 不小于投票者的最后一条 entry,否则一张票都拿不到。

> **打个比方**:选举限制有点像"接班人考核"——要接班当 leader,你的"履历"(log)不能比任何一个考官(投票者)差。只要有任何一个考官的履历比你强,他就拒绝给你投票。这条考官把关,挡住了"履历空白的愣头青上位"。

注意 `isUpToDate` 是个**充分条件式的判定**:它只回答"candidate 够不够新到值得我投票",**不**直接回答"candidate 是不是包含所有已 commit 的 entry"。后者是一个更深的性质(leader 完整性),需要 5.3 节的证明。代码上看起来就一行,但这一行背后是一整套数学。

## 5.3 从 `isUpToDate` 到 Leader 完整性:三条规则联手推出"已 commit 永不丢"

这一节是全章最重要、也是 Raft safety 最核心的证明。我们一步一步推:**为什么 `isUpToDate` + 一个 term 一票 + commit 要多数派,能推出"任何被选出的 leader 必含所有已 commit 的 entry"**。

**命题(Leader Completeness)**:对任意一条 entry `e`,如果 `e` 在某个 term `T` 被 commit,那么对任意 `T' > T` 的 term,`T'` 的 leader 的 log 里一定有 `e`(在同一 index、同一 term)。

**证明**(反证法,结合鸽巢):

1. 设 `e` 在 term `T` 被 commit。按 commit 的定义(P1-04),`e` 被**多数派节点**写进了 log。
2. 假设存在某个 `T' > T`,`T'` 的 leader `L'` 的 log 里**没有** `e`。
3. `L'` 是怎么被选上的?它必须拿到**多数派节点**的投票(选举规则)。
4. 关键一步:这两个多数派——"commit `e` 的多数派" 和 "选 `L'` 的多数派"——必有交集(P0-01 鸽巢:任意两个过半数子集必有公共节点)。设这个交集节点为 `V`。
5. `V` 既在 "commit `e` 的多数派" 里(它的 log 里有 `e`),又在 "选 `L'` 的多数派" 里(它给 `L'` 投了票)。
6. 现在 `V` 给 `L'` 投了票。`V` 投票时一定检查过 `isUpToDate`(`raft.go:1222`)——即 `L'` 的 log "至少和 `V` 一样新"。
7. 但 `V` 的 log 里有 `e`(它的 last entry 的 term/index 至少和 `e` 一样新),`L'` 的 log 里没有 `e`(假设)。那么 `L'` 的 log 必须在"最后一条 entry 的 term 大于 `V` 的"——因为如果 term 相同,`isUpToDate` 还要求 index ≥,**而 index 大意味着 log 更长,不能少一条 entry**;唯一能让 `L'` "更新却又少 entry"的可能,是 `L'` 的最后一条 entry 的 term **大于** `V` 的最后一条 entry 的 term。
8. 也就是说,`L'` 有一条 term > `V` 的最后一条 entry 的 term 的 entry,记为 `f`(在某个 term `T_f`,`T < T_f < T'`,因为 `T'` 的 leader 的 log 是选举前积累的)。
9. 但 `f` 这条 entry 又是怎么来的?它一定是 `T_f` 那个 term 的 leader 复制给 `L'` 的。而 `f` 出现在 `L'` 的 log 里,说明 `f` **要么被 commit,要么没被 commit**。
10. 这一步是归纳的递归,完整的严格证明要按 term 递增归纳(详见 Ongaro/Oki 的 PhD 论文第 3 节,或 verdi-raft 的 Coq 形式化)。结论:**假设 `L'` 的 log 里没有 `e`,会反推出存在一条"更高 term 却冲突"的已 commit entry,无限回退到矛盾**。∎

这个证明看一遍不轻松,这正是为什么 Raft 论文要花 Figure 8 + 一整节 5.4.3 来讲它,而 etcd-raft 把它**直接形式化**进了 TLA+ 规约里,作为不变式 [`LeaderCompletenessInv`](../etcd-raft/tla/etcdraft.tla#L859-L869):

```tla
LeaderCompletenessInv ==
    \A i \in Server :
        LET committed == Committed(i) IN
        \A idx \in 1..Len(committed) :
            LET entry == log[i][idx] IN
            \A l \in CurrentLeaders :
                currentTerm[l] > entry.term =>
                log[l][idx] = entry
```

翻译过来:对任意节点 `i` 的已 commit 日志里的任意一条 `entry`,**任何 term 更高的 leader 的 log 在同一 index 位置一定是同一条 entry**。这就是 leader 完整性的形式化陈述。TLA+ 模型检查器(TLC)穷举了所有可达状态,确认这个不变式在所有情况下都成立——也就是说,上面的反证法结论,机器已经帮你验过了。

TLA+ 里还有一条配套的 [`MoreUpToDateCorrectInv`](../etcd-raft/tla/etcdraft.tla#L848-L853),直接刻画了 `isUpToDate` 的含义:

```tla
MoreUpToDateCorrectInv ==
    \A i, j \in Server :
       (\/ LastTerm(log[i]) > LastTerm(log[j])
        \/ /\ LastTerm(log[i]) = LastTerm(log[j])
           /\ Len(log[i]) >= Len(log[j])) =>
       IsPrefix(Committed(j), log[i])
```

翻译:如果 `i` 的 log 比 `j` 的"至少一样新"(按 `isUpToDate` 的口径),那么 `j` 的所有已 commit entry 一定都是 `i` 的 log 的前缀——即 `i` 含有 `j` 的所有已 commit entry。这条不变式把"`isUpToDate` 这个判定"和"candidate 含有投票者已 commit 的 entry"这两件事在数学上接了起来,是上面证明步骤 6→7 的形式化。

> **钉死这件事**:**`isUpToDate` + 一个 term 一票 + commit 要多数派,三者缺一不可,联手推出 Leader Completeness——已 commit 的 entry 在任何后续 term 的 leader 的 log 里都还在。** 任何一条规则被破坏,证明就断了:
> - 没有 `isUpToDate`:5.2 节反例,选出落后 leader,直接覆盖已 commit 数据。
> - 一个 term 可以投多票(P1-02 讲过):同一 term 可能选出两个 leader,两个不相交多数派各自 commit 不同 entry,违反"两个多数派必有交集"的前提。
> - commit 不要多数派:那 `e` 只在一个节点上就可能被"commit",换 leader 必丢——根本谈不上"永不丢"。
>
> 这三条规则不是 Raft 的设计装饰,而是 leader 完整性这条 safety 性质的**三根支柱**。

## 5.4 回扣 Figure 8:leader 上任时为什么不能直接 commit 旧 term 的 entry

Leader 完整性说"已 commit 的 entry 永不丢",但 P1-04 的 Figure 8 陷阱提醒我们:**新 leader 不能凭"旧 term 的 entry 已经在多数派节点上"就直接 commit 旧 term 的 entry**。这条限制乍看和"完整性"矛盾——既然 leader 都有这些 entry 了,为什么不让它直接 commit?

回扣 P1-04 的那个经典反例(简化复述):leader 在 term 2 复制了 `entry#2(term=2)` 到多数派但**还没 commit 就挂了**;新 leader 在 term 3 上任,看到 `entry#2` 已经在多数派上,如果它**直接 commit** `entry#2`,那此刻若 term 3 的 leader 又挂、一个没有 `entry#2` 的节点在 term 4 当选(它可能通过复制更高 term 的 entry 把 `entry#2` 覆盖掉),就会违反 leader 完整性——已 commit 的 `entry#2` 在 term 4 的 leader 那里丢了。

Raft 的解法是 P1-04 讲过的:**新 leader 只能通过 commit 自己当前 term 的 entry,顺带 commit 之前 term 的 entry**。etcd-raft 的实现是,`becomeLeader` 上任时**立刻 append 一条空 entry**([`raft.go:961-965`](../etcd-raft/raft.go#L961-L965)):

```go
emptyEnt := &pb.Entry{Data: nil}
if !r.appendEntry(emptyEnt) {
    r.logger.Panic("empty entry was dropped")
}
```

`appendEntry` 会把这条 entry 的 `Term` 设为当前 term([`raft.go:818-819`](../etcd-raft/raft.go#L818-L819)),于是 leader 的 log 末尾就有一条"本 term 的 entry"。等这条空 entry 被多数派复制并 commit(P1-04 的 commit 规则要求 `matchTerm` 当前 term),**它之前的所有 entry 也随之 commit**(commit 是单调前进的)。

> **所以这样设计**:这条"无操作"的空 entry 唯一作用就是**给新 leader 一个"本 term 的 anchor entry"**,让它可以走合法的 commit 路径(commit 当前 term entry 顺带 commit 旧的),而**不**需要单独发明一条"凭多数派复制就 commit 旧 term entry"的危险规则。把 Figure 8 的坑,用一条"上任即写一条空 entry"的工程约定干净地填掉。

> **钉死这件事**(回扣本章主线):Figure 8 的 commit 规则,和 leader 完整性**不矛盾,而是它的保护壳**——完整性保证"已 commit 永不丢",而严格的 commit 规则保证"我们认定的 commit 是真的安全"。一条 entry 在 Raft 里被认定 commit 的那一刻,它就拿到了完整性这条铁律的保护;在此之前,它"看起来在多数派上"也不能算数。P1-04 详讲了这条规则,本章只回扣它的角色:它是完整性证明里"commit"这个概念的定义边界。

## 5.5 持久化要求:不落盘的承诺不算承诺

Leader 完整性保证了"换 leader 不丢已 commit 的 entry",但这个保证有一个隐含前提:**节点崩溃前后,它的状态(term、投了谁的票、log)是一致的**。如果崩溃能让节点"忘了自己投过谁",或者"丢了 log 里几条 entry",那么前面所有的不变式都站不住。这一节讲 Raft 的**持久化要求**:哪些状态必须落盘,不落盘会破坏哪个不变式。

### 5.5.1 必须落盘的三样:`currentTerm` / `votedFor` / `log`

Raft 论文 Figure 2 明确要求,所有 server 上的 **persistent state**(`currentTerm`、`votedFor`、`log[]`)必须"updated on stable storage before responding to RPCs"。etcd-raft 的协议状态机本身不碰磁盘——它只把"需要落盘的东西"打包进 `Ready` 里的 `HardState` 和 `Entries`,由上层(在 etcd 里就是 WAL,第 17 章详讲)实际写盘。看 [`raft.go` 的 `hardState()`](../etcd-raft/raft.go#L504-L510):

```go
func (r *raft) hardState() *pb.HardState {
    return &pb.HardState{
        Term:   new(r.Term),
        Vote:   new(r.Vote),
        Commit: new(r.raftLog.committed),
    }
}
```

`HardState` 就是 Raft Figure 2 里除 `log[]` 之外的两个持久字段(`currentTerm`、`votedFor`)加上 `commit`(帮助快速恢复 commitIndex),对应 [`raftpb/raft.pb.go:693`](../etcd-raft/raftpb/raft.pb.go#L693-L700):

```go
type HardState struct {
    Term  *uint64  // currentTerm
    Vote  *uint64  // votedFor
    Commit *uint64 // commitIndex
    ...
}
```

而 `log[]` 通过 `Ready.Entries` 交出去。看 [`rawnode.go:142-161`](../etcd-raft/rawnode.go#L142-L161) 的 `readyWithoutAccept`,字段就对得上:

```go
rd := Ready{
    Entries:          r.raftLog.nextUnstableEnts(),   // 待持久化的新 entry
    CommittedEntries: r.raftLog.nextCommittedEnts(...), // 待 apply 的已 commit entry
    Messages:         r.msgs,
}
// ...
if hardSt := r.hardState(); !isHardStateEqual(hardSt, rn.prevHardSt) {
    rd.HardState = hardSt  // HardState 变了才需要重写
}
```

`MustSync`([`rawnode.go:191-198`](../etcd-raft/rawnode.go#L191-L198))则把"何时必须同步落盘"显式化:

```go
func MustSync(st, prevst *pb.HardState, entsnum int) bool {
    // Persistent state on all servers:
    // (Updated on stable storage before responding to RPCs)
    // currentTerm
    // votedFor
    // log entries[]
    return entsnum != 0 || st.GetVote() != prevst.GetVote() || st.GetTerm() != prevst.GetTerm()
}
```

这三条 `||` 就是 Raft 持久化要求的代码化身:**只要新写了 log entry、或者 votedFor 变了、或者 term 变了,这次就必须 fsync 落盘**。

### 5.5.2 不落盘哪一样,会破坏什么

> **不持久化 `votedFor`**:节点 A 在 term 5 投了 B,然后**没落盘**就崩溃。重启后 A 的 `votedFor` 还是 None(它忘了自己投过)。此时 term 5 内 C 又来拉票,A 把"这宝贵的一票"再投给了 C。结果 term 5 出现了两个 leader(B 和 C 各拿到一张"以为还没投过"的票),脑裂。**`votedFor` 必须落盘,是为了"一个 term 一票"这条不变式在崩溃前后都成立**——这是 P1-02 讲的 term 逻辑时钟的根基,也是 5.3 节证明步骤 6 的前提。
>
> **不持久化 `currentTerm`**:节点 A 在 term 5 当过 leader,然后崩溃,term 没落盘。重启后 A 以为自己是 term 1 的 follower。这时网络里有个 term 3 的旧 leader 发来 AppendEntries,A 看"term 3 > 我的 term 1",直接退化为 follower 接受它——**term 倒退了**。这破坏了 term 单调,进而破坏"高 term 的 leader 拥有低 term 已 commit entry"(完整性证明步骤 7)。**`currentTerm` 必须落盘,是为了 term 单调在崩溃前后都成立**。
>
> **不持久化 `log`**:节点 A 的 log 里有已 commit 的 `entry#7`,但**没落盘**就崩溃。重启后 A 的 log 没有 `entry#7`。A 重新参与选举时,它的 `lastEntryID` 不含 `entry#7`,`isUpToDate` 判定时它可能既投得出票(对它而言别人够新)、又自己被别人接受(对它而言别人看不到它缺 `entry#7`)。最严重的是:**A 丢失了已 commit 的 entry,完整性从崩溃那一刻就破了**。**`log` 必须落盘,是为了"已 commit 的 entry 在崩溃后还在"**(完整性的直接物质基础)。

> **钉死这件事**:Raft 的 safety 不变式都建立在"节点崩溃后能恢复到崩溃前的关键状态"这个前提上。持久化不是性能优化,是**正确性要求**。少了任何一样(`currentTerm`/`votedFor`/`log`),上面 5.3 节的完整性证明就有一根支柱断了。etcd-raft 用 `HardState`(term/vote)+ `Entries`(log)这两个 `Ready` 字段,把"要落盘的东西"清晰交给上层;`MustSync` 则划了"必须 fsync"的硬线。

etcd 的 WAL(P5-17)就是负责把 `HardState` 和 `Entries` 真正写到磁盘的那一层。etcd-raft 自己不碰磁盘——这是它"纯协议库"的设计取舍(P1-06 详讲),持久化是应用层的职责。本章只讲"为什么必须持久化",不讲"WAL 怎么写"。

## 5.6 unstable/stable 分离:把"没落盘"显式建模进数据结构

Raft 要求"log 要落盘才算数",但 etcd-raft 是个纯内存状态机:它在内存里 `append` entry、在内存里判定 commit、在内存里 apply。那么"还没落盘的 entry"和"已经落盘的 entry"在数据结构上怎么区分?答案是 **`raftLog` 的两段式布局**:stable 部分交给 `Storage`(在 etcd 里是 MemoryStorage + WAL),unstable 部分由 [`unstable` 结构体](../etcd-raft/log_unstable.go#L37-L54) 自己持有。

### 5.6.1 `raftLog` 的两段结构

看 [`raftLog` 的字段](../etcd-raft/log.go#L25-L64):

```go
type raftLog struct {
    // storage contains all stable entries since the last snapshot.
    storage Storage

    // unstable contains all unstable entries and snapshot.
    unstable unstable

    // committed is the highest log position that is known to be in
    // stable storage on a quorum of nodes.
    committed uint64
    applying uint64
    applied  uint64
    // ...
}
```

注释说得很清楚:`storage` 装"自上次 snapshot 以来所有 **stable** entry",`unstable` 装"所有 **unstable**(尚未写到 Storage 的)entry 和 snapshot"。

`unstable` 本体([`log_unstable.go:37-54`](../etcd-raft/log_unstable.go#L37-L54))更直观:

```go
type unstable struct {
    snapshot *pb.Snapshot     // 尚未落盘的 snapshot
    entries  []*pb.Entry      // 尚未落盘的 entry 列表
    offset   uint64           // entries[i] 对应 raft log 的 index = i + offset

    snapshotInProgress bool
    offsetInProgress   uint64  // entries[:offsetInProgress-offset] 正在写盘
    logger Logger
}
```

关键字段是 `offset`:`unstable.entries[i]` 对应的 raft log index 是 `i + offset`。这个 `offset` 就是 unstable/stable 的分界线——**index < offset 的 entry 都在 stable(已落盘),index >= offset 的 entry 都在 unstable(还在内存)**。`newLog` 初始化时([`log.go:84-99`](../etcd-raft/log.go#L84-L99))就把 `offset` 设成 `lastIndex+1`,即"刚启动时所有已落盘,unstable 为空":

```go
return &raftLog{
    storage: storage,
    unstable: unstable{
        offset:           lastIndex + 1,
        offsetInProgress: lastIndex + 1,
        // ...
    },
    // ...
}
```

整个布局用 ASCII 画出来:

```
raftLog 的逻辑视图(按 log index 排列):

         [snapshot 覆盖的范围]   [stable:已落盘]        [unstable:内存中]
         ◄─────────────────► ◄──────────────────► ◄──────────────────────►
index:   0 ...           snap  snap+1        off-1  off            lastIndex
                              └────── storage.ents ──┘  └── unstable.entries ──┘
                                                       (offset = lastIndex+1 的值)

  - storage.ents[i] 对应 raft log index = i + storage.ents[0].Index
  - unstable.entries[i] 对应 raft log index = i + unstable.offset
  - committed / applied 是逻辑游标,落在 [snapshot ... lastIndex] 区间
```

### 5.6.2 持久化推进:`nextUnstableEnts` → 上层写盘 → `stableTo`

写盘的推进是一个推拉模型(P1-06 详讲):

1. **协议状态机** `append` 一条 entry → 进 `unstable.entries`(还在内存)。
2. **`Ready`** 把"还没在写盘中"的 unstable entry 通过 [`nextUnstableEnts`](../etcd-raft/log.go#L198-L200)(实为 `unstable.nextEntries` [`log_unstable.go:100-106`](../etcd-raft/log_unstable.go#L100-L106))取出来,交给上层:

```go
func (u *unstable) nextEntries() []*pb.Entry {
    inProgress := int(u.offsetInProgress - u.offset)
    if len(u.entries) == inProgress {
        return nil
    }
    return u.entries[inProgress:]
}
```

3. 上层(etcd 的 WAL)把这些 entry 写盘,完成后发回 `MsgStorageAppendResp`,触发 [`raftLog.stableTo`](../etcd-raft/log.go#L367) → [`unstable.stableTo`](../etcd-raft/log_unstable.go#L138-L164),把已落盘的部分从 `unstable.entries` 头部"切掉",`offset` 前移:

```go
func (u *unstable) stableTo(id entryID) {
    // ... 校验 term 一致(防止把过期回写当有效)...
    num := int(id.index + 1 - u.offset)
    u.entries = u.entries[num:]
    u.offset = id.index + 1
    u.offsetInProgress = max(u.offsetInProgress, u.offset)
    u.shrinkEntriesArray()
}
```

注意几个精妙细节:

- **`stableTo` 收的是 `(index, term)`,不是只 index**。因为 unstable 可能在"上层正在写盘"期间被新 leader 的 log 覆盖([`truncateAndAppend`](../etcd-raft/log_unstable.go#L200-L222)),收到回写时必须校验 term 一致,防止把"旧 log 的写盘完成"误当成"新 log 的写盘完成"。
- **`offsetInProgress`** 这第二个游标区分"还没开始写"和"正在写但没完成"。`acceptInProgress`([`log_unstable.go:122-130`](../etcd-raft/log_unstable.go#L122-L130))把当前所有 unstable 标记成"in-progress",这样下一次 `Ready` 不会重复返回正在写的这批。这是 P1-06 "Ready/Advance 推拉模型" 的细节,本章不展开。
- **`shrinkEntriesArray`**([`log_unstable.go:170-183`](../etcd-raft/log_unstable.go#L170-L183)):切掉头之后,如果底层数组容量利用率不到一半,就重新分配一个小数组,避免长期持有一个巨大数组、阻止其中已被切走的 entry 被 GC。这是纯 Go 工程优化,和 safety 无关,但读源码时值得注意。

### 5.6.3 凭什么"丢 unstable 不破坏 safety"

现在回答本章技巧精解的关键问题:**崩溃时 unstable 里的 entry 丢了,凭什么不破坏 Raft safety?**

> **不这样会怎样**(反面对比"不分离 unstable/stable"):假设我们朴素地不分,所有刚 `append` 的 entry 都当"已经在 log 里"。那么协议状态机可能在 entry **刚 append、还没落盘** 时,就把它当成"在多数派节点 log 里"来推 commitIndex,进而 apply 到状态机。一旦这时崩溃,**内存里那条 entry 没了**,但状态机已经 apply 了它对外承诺的效果。重启后 apply 指针要回退——**违反一致性**(对外说成功的事其实没持久化)。

etcd-raft 用两件事堵住这个坑:

**(1) commit 的定义本身要求"多数派的 stable storage"**。看 [`raftLog.committed` 字段的注释](../etcd-raft/log.go#L32-L35):

```go
// committed is the highest log position that is known to be in
// stable storage on a quorum of nodes.
committed uint64
```

注意 "**stable** storage **on a quorum**"。一条 entry 要被认定 commit,需要它在多数派节点的**稳定存储**里。leader 推进 commitIndex 时,只数那些 follower 已经 ack 过 `MsgAppResp`(而 `MsgAppResp` 在 etcd-raft 新架构里是 WAL 写完后才发的,P1-06 详讲)的 entry。

**(2) apply 路径上还有一道闸:`allowUnstable`**。看 [`maxAppliableIndex`](../etcd-raft/log.go#L267-L273):

```go
func (l *raftLog) maxAppliableIndex(allowUnstable bool) uint64 {
    hi := l.committed
    if !allowUnstable {
        hi = min(hi, l.unstable.offset-1)  // 只允许 apply 到 stable 边界
    }
    return hi
}
```

`allowUnstable` 由 [`RawNode.applyUnstableEntries()`](../etcd-raft/rawnode.go#L440-L445) 控制:

```go
func (rn *RawNode) applyUnstableEntries() bool {
    return !rn.asyncStorageWrites
}
```

这里的语义要小心(这是本章少数需要读者细辨的地方):当 `asyncStorageWrites=false`(默认)时,`applyUnstableEntries()` 返回 **true**,意思是"允许 apply 还在 unstable 的 committed entry"。这看起来违反了"apply 只读 stable"的直觉,但实际上**不违反 safety**——因为在 `asyncStorageWrites=false` 模式下,`Ready` 的协议约定就是"上层必须先把这些 entry 持久化(`Entries`),再 apply(`CommittedEntries`)",顺序由 P1-06 讲的 Ready 契约保证([`rawnode.go:176-178`](../etcd-raft/rawnode.go#L176-L178) 的注释:"Messages cannot be sent until after Entries are written to stable storage")。也就是说,apply 那一刻,上层已经保证这些 entry 在本节点落盘了。

打开 `asyncStorageWrites=true` 模式时(性能更激进的路径),`applyUnstableEntries()` 返回 false,`maxAppliableIndex` 强制 `hi = min(hi, unstable.offset-1)`——**apply 严格只读 stable**,这时分离的语义才在代码路径上显式生效。

无论哪种模式,关键不变式都是:**对外承诺的 apply,一定是在 entry 本地落盘之后**。unstable/stable 的分离,就是把这个不变式**在数据结构上显式建模**——让"什么时候 entry 才算稳定"这件事可推理、可检查,而不是藏在隐式假设里。

> **钉死这件事**:unstable/stable 分离解决两件事:① 让"尚未持久化的 entry"有明确归宿(`unstable.entries` + `offset`),协议状态机能继续基于完整 log 推理(term、matchIndex、commit 都看完整 log,包括 unstable 部分);② 让"持久化完成"这个事件可观测——`stableTo` 一调,entry 就从 unstable 切到 stable,从此"崩溃也不怕"。崩溃时 unstable 可能丢,但"丢的 entry 不可能已被 commit(commit 要求多数派 stable storage)",所以丢 unstable 不破坏 safety。

TLA+ 里也直接刻画了这条性质,叫 [`CommittedIsDurableInv`](../etcd-raft/tla/etcdraft.tla#L871-L874):

```tla
CommittedIsDurableInv ==
    \A i \in Server :
        state[i] = Leader => commitIndex[i] <= durableState[i].log
```

翻译:leader 的 commitIndex 永远不超过它已持久化的 log 长度——即 commit 的东西一定 durable。这正是 unstable/stable 分离想保证的。

## 技巧精解:Raft safety 的两根命门——选举限制 + unstable/stable 分离

这一节把本章两个最硬核的技巧单独拆透。它们是 Raft safety 的两根命门,讲不清它们,就等于没讲 Raft 的安全性。

### 技巧 ① 选举限制 → Leader 完整性的因果链

**这个技巧的"一招鲜"**:Raft 不去证明"选出的 leader 一定有所有已 commit 的 entry"这个结论(直接证很难),而是埋一条更易证的规则(`isUpToDate`),让结论从规则 + 两条公理(一个 term 一票、commit 要多数派)数学地推出来。

**因果链(钉在脑子里)**:

```
规则 1:投票时 isUpToDate(candidate_last ≥ voter_last)          ┐
规则 2:一个 term 一个节点最多投一票                              │ 三者
规则 3:commit 要多数派复制(已 commit = 多数派 stable log 里有)  ┘ 缺一不可
              │
              ▼ (鸽巢:任意两个多数派必有交集)
推论 A:任何"选 leader 的多数派" 和 "commit 某 entry 的多数派" 有公共节点 V
              │
              ▼ (规则 1:V 给 leader 投了票 ⇒ leader 的 log 至少和 V 一样新)
推论 B:leader 的 log 必含 V 已有的所有 entry(包括已 commit 的那条)
              │
              ▼ (按 term 归纳)
结论:Leader Completeness——任何已 commit 的 entry,在任何更高 term 的 leader 的 log 里都还在
              │
              ▼
最终保证:已 commit 的 entry 永不丢失
```

**真实源码**:整条因果链的"启动点"就是 [`log.go:442-445`](../etcd-raft/log.go#L442-L445) 那一行 `isUpToDate` + [`raft.go:1222`](../etcd-raft/raft.go#L1222) 的调用 + [`raft.go:1071`](../etcd-raft/raft.go#L1071) 把 `last` 塞进 `MsgVote`。三段加起来不到 10 行代码,扛起了整个 Raft safety。

**反面对比**:如果**没有选举限制**(把 `raft.go:1222` 那行去掉,直接 `canVote` 即投):

```
5 节点集群,A 是旧 leader,entry#7(term=3)已被 {A,B,C} commit。
D 是新加入的节点,log 为空。A 挂。
D 重启后 term 抢到 5,发起选举。D+E+任一 = 3 票过半,D 当选 leader。
D 的 log 里没有 entry#7。D 从 index 1 开始追加自己的内容,
通过 AppendEntries 的 prevLogIndex 匹配失败,把 B、C 的 log 从某处截断重写。
entry#7 被覆盖。——已 commit 的数据丢了。
```

这就是 Raft 论文 §5.4.1 反复强调"candidate 必须含所有已 commit entry"的原因,也是 etcd-raft 把它编进 `isUpToDate` 这一行代码的原因。**没有这一行,Raft 就不是共识算法**。

### 技巧 ② unstable/stable 分离:把"未持久化"显式建模

**这个技巧的"一招鲜"**:不在协议状态机里隐式假设"append 即持久",而是把"已持久化"和"未持久化"的 entry **在数据结构上物理分开**,让"持久化推进"成为一个可观测、可推理的事件(`stableTo`),从而把崩溃前后的世界都纳入一个统一模型。

**真实源码要点**:

- `raftLog` = `storage`(stable)+ `unstable`(内存,未落盘)。分界线是 `unstable.offset`。
- 新 `append` 的 entry 进 `unstable.entries`([`log_unstable.go:200-222`](../etcd-raft/log_unstable.go#L200-L222) 的 `truncateAndAppend`)。
- 上层从 `Ready.Entries` 拿到 unstable entry,写盘,回写 `MsgStorageAppendResp` → [`unstable.stableTo`](../etcd-raft/log_unstable.go#L138-L164) 切掉已落盘的前缀,`offset` 前移。
- apply 路径上 `maxAppliableIndex` 用 `unstable.offset-1` 闸住,保证 apply 不越过 stable 边界(在 `asyncStorageWrites=true` 模式下显式生效;在默认模式下由 Ready 契约保证)。

**反面对比**:如果不分 unstable/stable,朴素地用一个数组管整个 log:

```
朴素方案:raftLog 只有一个 entries []pb.Entry,append 即认为"在 log 里"。
leader append entry#10 → 立刻当它在 log 里 → 立刻数多数派 → 立刻 commit → 立刻 apply。
此刻还没落盘。崩溃。entries[] 丢了 entry#10,但状态机已经 apply 了它。
重启:apply 指针要么回退(对外承诺的事没做),要么向前跳(把没 apply 的当 apply 了)。
两种都违反线性一致性。
```

unstable/stable 分离让这种灾难在协议状态机的数据结构层面**根本无法表达**——"commit"的定义里就钉死了"多数派 stable storage",entry 没进 stable 就不算 commit,apply 就走不到它。崩溃丢 unstable,丢的是"还没对外承诺的东西",safety 不破。

> 这两根命门合起来,就是本章的结论:**选举限制保证换 leader 不丢已 commit 的 entry,unstable/stable 分离(配合持久化要求)保证崩溃重启不破坏已 commit 的 entry**。两者一道,把"已 commit 的 entry 永不丢失"这条 Raft 最重的承诺落到实处。

## 章末小结

这一章是第 1 篇 Raft 地基的**收束章**。前四章一块块搭起的规则(term 单调、一个 term 一票、leader 选举、log matching、commit 要多数派、Figure 8 的 commit 限制),在这一章收成一句话:

> **一条 entry 一旦被 commit,就永不丢失。**

这条保证由两个机制联手支撑:**选举限制 + leader 完整性**(换 leader 不丢)、**持久化要求 + unstable/stable 分离**(崩溃重启不丢)。etcd-raft 用一行 `isUpToDate` 扛起前者的因果链,用 `raftLog.storage/unstable` 的两段布局显式建模后者的边界。TLA+ 规约里的 `LeaderCompletenessInv`、`MoreUpToDateCorrectInv`、`CommittedIsDurableInv` 把这些性质形式化,机器已穷举验证。

回到全书二分法:本章完全落在**协议层**(etcd-raft 仓)。它讲的选举限制、完整性、持久化要求,都是 Raft 协议本身的性质;WAL 怎么把 `HardState`/`Entries` 写盘(应用层)是第 17 章的事,本章只点到"必须落盘"这一层。换言之——Raft 协议说"要落盘才安全",etcd-raft 用 `Ready` 把"要落盘的东西"交出来,WAL 负责真正写下去。三层各司其职。

### 五个"为什么"清单

1. **为什么选出的 leader 一定有所有已 commit 的 entry?** 因为投票时的 `isUpToDate` 限制——投票者只给"日志至少和自己一样新"的 candidate 投票——配合"一个 term 一票 + commit 要多数派",数学地推出 Leader Completeness(5.3 节证明)。
2. **为什么 `isUpToDate` 比 `(term, index)` 就够,不需要比较"已 commit 到哪"?** 因为"candidate 的 log 是否更新"用 (最后一条 entry 的 term, index) 判定,等价于"candidate 是否含投票者的所有已 commit entry"(TLA+ 的 `MoreUpToDateCorrectInv`),一个简单判定承载一个深刻性质。
3. **为什么 leader 上任时要 append 一条空 entry?** 给新 leader 一个"本 term 的 anchor entry",让它能走合法的 commit 路径(commit 当前 term entry 顺带 commit 旧 term entry),填掉 Figure 8 的坑(5.4 节)。
4. **为什么必须持久化 `currentTerm`/`votedFor`/`log` 三样?** 不持久化任一样,5.3 节证明的某根支柱就断:`votedFor` 丢会破坏"一个 term 一票";`currentTerm` 丢会破坏 term 单调;`log` 丢直接丢已 commit 的 entry(5.5.2 节)。
5. **为什么 unstable/stable 要分开?** 把"未持久化的 entry"显式建模进数据结构,让"持久化推进"成为可观测事件(`stableTo`),让协议状态机在崩溃前后都自洽。崩溃时 unstable 可能丢,但"丢的 entry 不可能已 commit",所以丢 unstable 不破坏 safety(5.6.3 节)。

### 想继续深入往哪钻
