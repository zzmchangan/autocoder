# 《etcd 设计与实现深入浅出:一条 Put 如何被多数派共识》—— 全书总纲

> 一本写给"用过 etcd(或 Kubernetes)、甚至翻过源码,却总觉得一知半解"的人的小书。
>
> 本文是全书的**总纲**:定主线、定方法、定分篇分章、定源码策略、定写作约定。
> 与《Tokio》《LevelDB》《数据库内核》《容器》同源,都是"**第一性原理 + 真实源码**";命脉同样是**动机(why) + 技巧(how) 双线**;基调延续《LevelDB》——**直球讲透为主,比喻只在反直觉处点睛**。本书的特别之处:etcd 跨**三个源码仓库**(etcd 主仓 + etcd-io/raft + bbolt),而 Raft 共识协议是全书重头戏。

## 一、这本书的定位

### 目标读者

会写 Go、用过 etcd(`etcdctl put/get`、Kubernetes 里把它当配置中心/服务发现),甚至翻过 `server/etcdserver/` 或 `etcd-raft/raft.go`,但**一知半解**:知道有 Raft、leader、mvcc、watch、lease,却讲不清一条 `Put` 怎么从客户端走到多数派共识、怎么落到带版本的存储、怎么被 watch 推出去;也讲不清 `etcd-raft` 里那些 channel 驱动、`Ready`/`Advance`、`ProgressTracker`、`inflights` 到底在干什么。和"读过 tokio/leveldb 源码但不理解"是同一种病。

### 全书一句话主旨

> **一条 `Put` 请求,如何跨多节点达成"多数派共识、不丢不乱",再被应用到带多版本的存储、推给所有 watcher——共识(Raft)保证安全,工程优化(批量、pipeline、读优化、租约)保证它在真实集群里够快够活。**

每个组件,都是这条旅程上的一个**驿站**。读完一本书,读者能**在脑子里放映出一条 Put 的全过程 + 一次 leader 选举的全过程**:客户端 gRPC 请求怎么到 leader、leader 怎么 propose 进 Raft 日志、日志怎么被多数派复制并 commit、commit 后怎么 apply 到 mvcc 多版本存储、mvcc 怎么推给 watcher——以及故障时怎么重新选主、怎么恢复一致。每一步底下都用了什么巧妙的手段,都要讲透。

### 为什么用"一条 Put 的共识之旅"当骨架(而非按模块平铺)

etcd 的模块高度耦合:gRPC API → etcdserver → Raft(共识)→ apply → mvcc(存储)→ bbolt(底座),watch 又从 mvcc 订阅,lease 给 key 续命,WAL 给 Raft 兜底。按模块百科式平铺,读者会觉得每块都是孤岛、不知道**为什么要学这个**。用"一条 Put 怎么进来、怎么被共识、怎么落地、怎么被订阅、故障时怎么不丢不乱"这条旅程当骨架,每个模块都是"为了让旅程继续而必须出现的角色"——读者永远知道**我在哪、为什么需要下一站**。

## 二、写作方法

### 命脉:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与 API 文档、一般源码导读最关键的分野,所有写作者必读、必守。**

etcd 源码里充斥 Go 系统级与协议级技巧:channel 驱动的状态机(node.go 的 `recvc`/`propc`/`readyc`)、`Ready`/`Advance` 推拉模型、`ProgressTracker` + `inflights` 环形缓冲追踪复制进度、log 的 unstable/stable 分离、term 单调的不变式、PreVote 防惊群、ReadIndex 线性一致读、mvcc 的全局单调 revision + treeIndex 版本链、watch 的 synced/unsynced 分组、bbolt 的 COW 事务 + mmap、batch 批量提交。用户自己读过源码,"一知半解"的缺口往往**不在**"它要解决什么问题"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?(沿用系列一贯的"为什么优先")
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果不用这个技巧、朴素地写,会撞上什么墙",让妙处显形。共识算法尤其如此:Raft 的每一条规则(term、一个 term 一票、log matching、选举限制)都是**为了堵住某个具体的正确性漏洞**,讲不清"不这条会怎样",就等于没讲 Raft。

一句话:**讲不清技巧,等于没讲 etcd。**

### 基调:直球讲透为主,比喻只在反直觉处点睛 ⚠️ 延续《LevelDB》

> **这一节延续《LevelDB》由用户拍板的基调。**

《Tokio》用"餐厅服务员"贯穿,因为 async 运行时反直觉。etcd 的主体(Raft 共识)也有一部分**反直觉**(为什么多数派就够?为什么 term?为什么一个 term 只能投一票?leader 挂了数据会不会丢?)——**这部分允许适度比喻点睛**(Raft 论文本身就是为"可理解性"设计的,它用"投票立法"等类比)。但 Raft 之外的大部分(mvcc 存储结构、bbolt B+tree、WAL 编码、channel 驱动)都**具体可见,直球讲清最好**。

所以本书基调:
- **默认直球**:把数据结构、协议流程、源码本身讲透。能用一张表、一张时序图、一段源码说清的,**不上比喻**。
- **比喻只点睛,不贯穿,不另起体系**:只在 Raft 共识的反直觉处(多数派、term、选举)用一两句点一个比喻帮助"啊哈",点到为止。例如多数派可点一句"像议会投票,过半数即生效";leader 选举可点一句"像值班长挂了,大家随机等一会儿再抢着接班,避免同时都抢"。**绝不**展开成对照表、**绝不**强制每章出现。
- **判断标准**:直球讲不清的,才上比喻;讲得清的,就直球。

## 三、全书主线、二分法

### 主线

> **如何让一条写操作,跨多台可能同时故障的机器,达成"多数派一致、不丢不乱",并尽快对客户端返回成功——共识(Raft)管安全,工程优化管快和活。**

任何一处看不懂某个机制,回到这句问:"这是在**让多节点对操作达成共识**(协议层),还是在**把共识结果落地成可查可订阅的状态**(应用层)?"把答案写进"为什么"里。

### 二分法(迷路时回到它)

本书的二分法,正是 Raft 自身的架构分野:

> **共识协议层(Raft:让多数派对日志顺序达成一致、leader 主导复制) vs 状态机应用层(mvcc/bbolt/watch:把共识结果落地成可查、可订阅的多版本状态)。**

- **协议层这一面**:Raft 状态机、leader 选举、日志复制、安全性、`etcd-raft` 的驱动模型(Node/Ready/Step/Tick)、读一致性(ReadIndex)。这些都在 `etcd-raft` 仓。
- **应用层这一面**:etcdserver 的 apply 流程、mvcc 多版本存储(revision/treeIndex/watch)、backend + bbolt、WAL 持久化与恢复、lease。这些在 etcd 主仓(存储底座在 bbolt 仓)。

etcd 每个机制都落在这两面之一(或两者衔接处——etcdserver 的 raft.go 就是衔接)。任何时候迷路,回到这个二分法。

> Raft 的本质就是"**复制日志(Replicated Log)+ 应用到状态机**"。这个二分法不是我们硬套的,是 Raft 天然的骨架。etcd 把"协议层"做成了可复用的 `etcd-raft` 库(被 Kubernetes、TiKV、CockroachDB 等共享),把"应用层"留在 etcdserver——这个工程切分本身就是一条主线。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线是一条旅程:**为什么需要共识 → Raft 怎么达成 → etcdserver 怎么用 Raft → mvcc 怎么存 → bbolt 底座 → WAL/恢复不丢 → lease/成员 → 收束**。
> 章号全局连续;命名 `P{篇}-{章}-标题.md`,开篇第 1 章 = `P0-01`,第 1 篇第 2 章 = `P1-02`,以此类推。**全书 22 章**(聚焦共识核心,对标《Tokio》《LevelDB》体量)。

### 第 0 篇 · 开篇:为什么需要共识(定调,1 章)

- **第 1 章 · 第一性原理:为什么需要共识**(`P0-01`):单机为什么不够(宕机即不可用、数据一份不安全)→ 多副本引入的新问题(副本间怎么一致?谁是最新?)→ 脑裂(split-brain)与多数派(quorum)→ CAP 与线性一致性 → Raft 一句话(选主 + 复制日志 + 多数派 commit)→ etcd 全貌与全书旅程。立起"协议层 vs 应用层"二分法。**【样本章节】**

### 第 1 篇 · Raft 地基:共识协议层(5 章)⚠️ 全书重头戏

> 这一篇讲透 Raft。源码主体在 `etcd-raft` 仓。这是 etcd 的灵魂。

- **第 2 章 · Raft 状态机与 term**(`P1-02`)——三个角色(Follower/Candidate/Leader)、term 单调递增、为什么"一个 term 一个节点最多投一票"(防同一 term 多主)。源码:`etcd-raft/raft.go`、`etcd-raft/types.go`。**技巧精解:term 作为逻辑时钟**(为什么不用物理时间;term 单调 + 一个 term 一票,凭什么堵住"同一任期多个 leader")。
- **第 3 章 · Leader 选举**(`P1-03`)——heartbeat、随机化选举超时(为什么随机化:防多个 candidate 同时抢票分裂)、PreVote(防网络分区里的节点用高 term 打断正常集群)、checkQuorum(leader 主动确认自己还能联系多数派)。源码:`etcd-raft/raft.go`(`tickElection`/`becomeCandidate`/`stepCandidate`/`Step` 的 `MsgPreVote`)。**技巧精解:PreVote 防惊群 + 随机选举超时**——反面对比"不随机/不 PreVote"会怎样(活锁、分区扰动主集群)。
- **第 4 章 · 日志复制与 commit**(`P1-04`)——AppendEntries、log matching property(prevLogIndex/term 匹配才追加)、commitIndex 推进(只 commit 当前 term 的 entry 才安全,Figure 8 陷阱)、`ProgressTracker` + `inflights`(环形缓冲追踪每个 follower 的复制进度与流水线窗口)。源码:`etcd-raft/raft.go`(`stepLeader`/`handleHeartbeat`/`handleAppendEntries`)、`etcd-raft/log.go`(`maybeAppend`/`slice`)、`etcd-raft/tracker/`。**技巧精解:Figure 8——为什么只能 commit 当前 term 的 entry**(反面对比"commit 旧 term entry"会丢已提交数据)+ `inflights` 环形窗口控流水线。
- **第 5 章 · 安全性与持久化**(`P1-05`)——选举限制(candidate 的 log 必须至少和投票者一样新,保证选出的 leader 拥有所有已提交 entry)、leader 完整性(被提交的 entry 永不丢失)、必须**持久化** `currentTerm`/`votedFor`/`log` 才算安全。源码:`etcd-raft/raft.go`(`isUpToDate`)、`etcd-raft/log_unstable.go`、`etcd-raft/storage.go`。**技巧精解:选举限制 → leader 完整性**(这条因果链是 Raft safety 的核心)+ unstable/stable 分离(没持久化的 entry 单独放在 `unstable`,持久化后才并进 stable log)。
- **第 6 章 · etcd raft 的驱动模型:Node 与 Ready**(`P1-06`)——`etcd-raft` 把"协议状态机"和"上层驱动"切开:`raft.go` 是纯状态机(`Step`/`Tick` 步进、`Advance` 推进),`Node`/`RawNode` 用 channel(`propc`/`recvc`/`readyc`)把状态机的输出(`Ready`:要持久化的日志、要发的消息、要 apply 的 entry、要存的 snapshot)批量交给上层。这是"库 + 应用"解耦的典范。源码:`etcd-raft/node.go`、`etcd-raft/rawnode.go`。**技巧精解:`Ready`/`Advance` 推拉模型 + channel 解耦**——凭什么一个纯状态机能被嵌进任何应用(对比"把协议和网络 IO 焊死"会怎样)。

### 第 2 篇 · 写读路径:etcdserver(3 章)

> Raft 达成共识后,谁把它应用成 etcd 的 KV?这一篇讲 etcdserver 怎么驱动 raft + apply。

- **第 7 章 · etcdserver 架构与 gRPC 入口**(`P2-07`)——一条 `Put` 怎么从 gRPC 进来、`EtcdServer` 的两条主通道(raft 通道发 propose、apply 通道收已 commit 的 entry)、`raft.go` 怎么把 etcdserver 和 `etcd-raft` 的 Node 粘起来。源码:`etcd/server/etcdserver/server.go`、`etcd/server/etcdserver/raft.go`。**技巧精解:etcdserver 与 raft.Node 的衔接**(propose 走 channel、apply 走 channel,两条通道异步解耦)。
- **第 8 章 · 写路径全流程**(`P2-08`)——`Put` → 序列化成 `Request` → `Propose` 进 raft log → 多数派复制 commit → 从 apply 通道取出 → `applierV3` 应用到 mvcc → 返回客户端。一步步拆穿。源码:`etcd/server/etcdserver/raft.go`(`Propose`)、`etcd/server/etcdserver/applier.go`、`etcd/server/etcdserver/server.go`(`applyEntries`)。**技巧精解:propose 的"等 apply 才返回"**(为什么客户端要等到 apply 而非 commit——线性一致性)。
- **第 9 章 · 线性一致读与吞吐优化**(`P2-09`)——读也要线性一致(不能读到旧 leader 的旧数据):ReadIndex(读前问 leader 当前 commitIndex,leader 再确认自己还是 leader)、lease-based read(靠 leader 租约省一轮确认)。以及 pipeline propose、batch apply 怎么把吞吐拉起来。源码:`etcd-raft/read_only.go`、`etcd/server/etcdserver/raft.go` 的读路径。**技巧精解:ReadIndex 的正确性**(为什么朴素读会违反线性一致)+ lease read 的取舍(快但有租约过期的风险)。

### 第 3 篇 · 存储 mvcc:多版本的世界(4 章)

> apply 把 entry 落到 mvcc。这一篇讲 etcd 怎么存"一个 key 的多次修改"、怎么 watch、怎么压缩。

- **第 10 章 · revision 与 treeIndex/keyIndex**(`P3-10`)——etcd 不存"当前值",存"全部历史":每次修改分配全局单调递增的 `revision`(`main revision` + `sub revision`);`treeIndex`(B-tree)按 key 索引,每个 key 一条 `keyIndex`(它的各代 generation、每个 revision)。源码:`etcd/server/storage/mvcc/index.go`、`etcd/server/storage/mvcc/key_index.go`、`etcd/server/storage/mvcc/revision.go`。**技巧精解:全局单调 revision + keyIndex 版本链**(为什么不直接 key→value——为了 MVCC 多版本与 watch)。
- **第 11 章 · kvstore 与事务**(`P3-11`)——`watchableStore` 是核心;写走 `TxnWrite`(批量)、读走 `TxnRead`/`view`;怎么把 revision 对应的 value 存进 backend(bbolt)。源码:`etcd/server/storage/mvcc/kvstore.go`、`etcd/server/storage/mvcc/kvstore_txn.go`、`etcd/server/storage/mvcc/kv.go`。**技巧精解:treeIndex(存 revision)与 backend(存 value)分离**——索引在内存、值在 bolt,各取所长。
- **第 12 章 · watch:基于 revision 的事件推送**(`P3-12`)——watch 本质是"从某个 revision 开始订阅后续变更";`watcher_group` 把 watcher 按 synced(已追上)/unsynced(要从历史补)分组;事件怎么从写事务推给 watcher。源码:`etcd/server/storage/mvcc/watchable_store.go`、`etcd/server/storage/mvcc/watcher.go`、`etcd/server/storage/mvcc/watcher_group.go`。**技巧精解:synced/unsynced 分组**(追上实时 vs 补历史,两条路径)+ revision 作为 watch 的"游标"。
- **第 13 章 · compaction:压缩历史**(`P3-13`)——历史不能无限长,compaction 删掉某个 revision 之前的旧版本、收缩 keyIndex;以及定期 hash 校验数据一致性。源码:`etcd/server/storage/mvcc/kvstore_compaction.go`、`etcd/server/storage/mvcc/hash.go`。**技巧精解:compaction 的安全边界**(不能压掉 watcher 还需要的 revision)+ hash 一致性校验。

### 第 4 篇 · backend 与 bbolt:存储底座(3 章)

> mvcc 的 value 最终落在 bbolt。这一篇讲 backend 的批事务 + bbolt 的单文件 B+tree。

- **第 14 章 · backend:批事务**(`P4-14`)——`backend` 把 bbolt 的 `Tx` 包成 `batch_tx`(攒一批写、一次 commit,削 WAL/锁开销)、`read_tx`(读)、`tx_buffer`(缓冲未提交的写让读能看见)。源码:`etcd/server/storage/backend/backend.go`、`etcd/server/storage/backend/batch_tx.go`、`etcd/server/storage/backend/tx_buffer.go`。**技巧精解:batch 攒批提交**(反面对比"每次写一个 bolt 事务"会怎样——吞吐被 bolt 的 COW 拖垮)。
- **第 15 章 · bbolt 之一:单文件 B+tree + mmap**(`P4-15`)——bbolt 把整个库放在一个文件里,页(page)是基本单位;B+tree(branch/leaf node);mmap 只读映射把文件映射进内存省拷贝。源码:`bbolt/db.go`、`bbolt/node.go`、`bbolt/bucket.go`。**技巧精解:单文件 + 定长页 + mmap 只读映射**(为什么这样设计读快、且崩溃一致性好做)。
- **第 16 章 · bbolt 之二:COW 事务 + freelist**(`P4-16`)——bbolt 用 Copy-On-Write:写事务不改原页,复制新页写,提交时原子翻转 meta 页指针;`freelist` 管理空闲页回收;读写并发靠"写时读者用旧 meta 页"。源码:`bbolt/tx.go`、`bbolt/freelist*.go`、`bbolt/db.go`(`meta` 翻转)。**技巧精解:COW + meta 页双缓冲原子提交**(凭什么一个写事务能和多个读事务并发而不锁)+ freelist 回收。

### 第 5 篇 · 不丢不乱:WAL、Snapshot 与恢复(3 章)

> Raft 的 entry 要持久化(否则崩溃丢已 commit 数据);日志不能无限长,要靠 snapshot 截断。

- **第 17 章 · WAL:Raft 日志的持久化**(`P5-17`)——`etcd-raft` 只管协议、不碰磁盘,持久化由 etcd 的 `storage/wal` 做;WAL 的 record 格式(encoder/decoder)、预分配文件流水线(`file_pipeline`)、损坏修复(`repair`)。源码:`etcd/server/storage/wal/wal.go`、`etcd/server/storage/wal/encoder.go`、`etcd/server/storage/wal/decoder.go`、`etcd/server/storage/wal/repair.go`。**技巧精解:WAL 的 record 编码 + 预分配**(为什么预分配文件——避免运行时 fallocate 抖动)+ 损坏修复。
- **第 18 章 · Snapshot 与日志截断**(`P5-18`)——日志太长,定期把"当前状态机快照"存盘、截断旧日志(snapshot 之后的不用留);snapshot 触发、`Save`/`Release`、log truncation。源码:`etcd/server/storage/wal/wal.go`(`SaveSnapshot`)、`etcd-raft/log.go`(`stableTo`/compaction)。**技巧精解:snapshot 与 log 的衔接**(截到哪、follower 怎么用 snapshot 追上)。
- **第 19 章 · 启动恢复**(`P5-19`)——etcd 启动怎么 `Recover`:读 WAL 重放未 snapshot 的 entry、加载 snapshot、重建 raft 状态、bootstrap cluster 成员。源码:`etcd/server/storage/wal/wal.go`(`ReadAll`)、`etcd/server/etcdserver/server.go`(`load`/recover 路径)。**技巧精解:两层持久化分工**(WAL 兜 raft 日志、snapshot 兜状态机,崩溃后各司其职恢复到一致)。

### 第 6 篇 · lease 与成员变更(2 章)

- **第 20 章 · lease:给 key 续命**(`P6-20`)——`Put` 可以带 TTL,靠 lease:lease 有到期时间、客户端定期 `KeepAlive` 续约;lease 过期,挂在它名下的 key 被回收;`Lessor` 管理 lease、lease 与 key 的反向索引。源码:`etcd/server/lease/lease.go`、`etcd/server/lease/lessor.go`。**技巧精解:lease 与 key 的解耦 + 续约**(为什么 lease 和 key 分开——一个 lease 续命一批 key,省开销)+ 过期回收。
- **第 21 章 · 成员变更与 cluster**(`P6-21`)——怎么安全地增删节点(不能让两个不相交的多数派同时存在):Joint Consensus(`confchange` 两阶段)、member add/remove、leader transfer、membership 管理。源码:`etcd-raft/confchange/`、`etcd/server/etcdserver/membership.go`、`etcd-raft/quorum/`。**技巧精解:成员变更的安全性**(为什么单步变更是安全的——每次增删一个,两个旧新多数派必有交集)+ Joint Consensus 两阶段。

### 第 7 篇 · 收尾:权衡哲学(1 章)

- **第 22 章 · 共识 vs 性能:etcd 的权衡哲学**(`P7-22`)——把全书收束成几条哲学(共识换一致、多数派换可用、Raft 的可理解性优先、COW 换并发、batch 换吞吐);quorum 的数学(N/2+1 为什么够);`etcd-raft/tla/` 的 TLA+ 形式化规约怎么佐证 Raft 正确性;etcd 4.0 相对老版本演进(raft 拆分、cache 层)。**技巧精解:TLA+ 形式化验证**(工业级共识算法怎么被证明对——不只靠测试,靠数学)。

### 附录

## 七、Go 技巧侧重(对标 Tokio 的 Rust unsafe/原子/Pin、LevelDB 的 C++ 无锁/引用计数)

本书要重点拆的 Go 系统级与协议级技巧(写作时遇到这些,务必钻"怎么做到的 + 为什么 sound"):

- **channel 驱动的状态机**:`node.go` 的 `recvc`/`propc`/`readyc`/`applyc` 等 channel,凭什么用 channel 解耦协议状态机与上层 IO(对比"协议和网络焊死")。
- **`Ready`/`Advance` 推拉模型**:raft 状态机的输出(`Ready`:待持久化日志、待发消息、待 apply entry、待存 snapshot)批量交给上层,上层处理完调 `Advance` 推进——这是"库 + 应用"解耦的核心。
- **term 单调 + 一个 term 一票的不变式**:Raft safety 的根基,怎么靠这两条堵住"同任期多主"。
- **`ProgressTracker` + `inflights`**:追踪每个 follower 复制进度的数据结构,inflights 环形缓冲怎么控流水线窗口。
- **log 的 unstable/stable 分离**:没持久化的 entry 放 unstable,持久化后才并进 stable——保证崩溃一致。
- **mvcc 的全局单调 revision + treeIndex/keyIndex**:多版本存储的骨架,为什么这样能同时支持读当前值、读历史、watch。
- **bbolt 的 COW 事务 + meta 页双缓冲**:凭什么写事务能和读事务并发不锁。
- **batch 攒批提交**(backend)+ **预分配文件**(WAL):吞吐与稳定的工程手段。
- **TLA+ 形式化规约**(`etcd-raft/tla/`):工业级共识算法怎么被数学证明对,不只靠测试。
