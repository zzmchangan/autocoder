## 一、这本书的定位

### 目标读者

用 axum 写过 Web、用 reqwest 调过 HTTP、知道 Rust 异步底层是 Tokio+hyper、看过几篇 hyper 源码分析,但**一知半解**:知道有 Service、HTTP/1 解析、HTTP/2(h2 crate)、连接池、Body、bytes,却讲不清一次 axum 请求从 TCP 字节进来、被 HTTP/1 状态机解析、交给 Service(一个 Future)、响应 body 怎么以 Stream 形式写出去,中间怎么和 Tokio 的 reactor/scheduler/AsyncRead 无缝拼起来。和"读过 Tokio 源码但不理解"、"翻过 gRPC 源码却一知半解"是同一种病。

### 全书一句话主旨

> **hyper 做的事,本质是把"HTTP 协议机"(HTTP/1 状态机 + HTTP/2 多路复用)无缝地建在 Tokio 的异步运行时之上——每个连接一个 task,协议机用 AsyncRead/AsyncWrite 读写字节,Service 把"处理一个请求"抽象成一个 Future。它是 Tokio 之上第一层,也是 axum/reqwest/tonic/Pingora 的共同地基。**

每个组件,都是"协议机 × 运行时"拼合的一个环节。读完一本书,读者不是"多记了几个 hyper API",而是能**在脑子里放映出一次 HTTP 请求在 hyper 里的全过程**:TCP 字节被 Tokio reactor 唤醒 → hyper 的 buffered IO(`common/io`)读进来 → HTTP/1 状态机(`proto/h1/decode`)切成请求行/头/body → 交给 Service(一个 `Future<Output=Response>`)→ 响应 body 作为 `Stream` 经编码(`proto/h1/encode`)写回 → 全程跑在一个 Tokio task 上,靠 budget 让出、靠 timer 控超时——以及这每一步底下,Tokio 是怎么被用起来的(承接)。

### 为什么用"协议侧 vs 框架侧"当骨架(而非按模块平铺)

hyper 的模块(proto/service/client/server/body/common)高度耦合:一次请求要穿过协议机、被 Service 调度、body 要以 Stream 流。按模块百科式平铺,读者会觉得每块都是孤岛。用"**协议侧(HTTP 协议怎么解析编码:HTTP/1 状态机 + HTTP/2 via h2)vs 框架侧(怎么把协议机组织成可用的 client/server:Service/连接池/body/IO)**"这条主轴当骨架,每个模块都是"为了让 HTTP 跑起来而必须出现的角色"。

## 三、全书主线、二分法、主比喻、承接

### 主线

> **把 HTTP 协议机(HTTP/1 状态机 + HTTP/2 多路复用)无缝建在 Tokio 异步运行时之上——每个连接一个 task,IO 用 AsyncRead/AsyncWrite,Service 把请求处理抽象成 Future。**

任何一处看不懂某个机制,回到这句问:"这是在**协议侧**(HTTP 协议怎么解析编码:HTTP/1 状态机 / HTTP/2),还是在**框架侧**(怎么组织成 client/server:Service / 连接池 / body / IO)?这个机制 Tokio 是怎么支撑的?"

### 二分法

本书的二分法:

> **协议侧(HTTP/1 解析编码状态机 + HTTP/2 多路复用) vs 框架侧(Service trait + 连接池 + Body Stream + buffered IO)。**

- **协议侧这一面**:`proto/h1/`(conn/decode/dispatch/encode/io/role 的 HTTP/1 状态机)、`proto/h2/`(委托 h2 crate 的 HTTP/2)、帧/流/keep-alive/chunked/升级——这些决定"HTTP 字节怎么切、怎么编"。
- **框架侧这一面**:`service/`(Service trait)、`client/`(连接池与分发)、`server/`(连接接受与生命周期)、`body/`(Body as Stream)、`common/io`+`common/buf`(buffered IO 与 bytes 零拷贝)、`rt/`(运行时适配)——这些决定"怎么把协议机组织成可用的 client/server"。

### 承接《Tokio》(本书最大特色,务必遵守)

hyper 直接长在 Tokio 上,承接最紧密。**承接铁律**:Tokio 已讲透的,一句带过 + 指路 [[tokio-source-facts]],篇幅全留 hyper 独有。具体承接点:

- **每个连接一个 Tokio task**:hyper 每条 HTTP 连接 spawn 一个 task,task 内跑协议机循环。Tokio 的 task/调度/预算让出([[tokio-source-facts]] budget=128)一句带过。
- **IO 用 AsyncRead/AsyncWrite**:hyper 的 `common/io` 在 Tokio 的 AsyncRead/AsyncWrite 之上做 buffered IO。Tokio 的 reactor/mio(epoll/kqueue,[[tokio-source-facts]] mio 1.2.1 edge-triggered)一句带过。
- **Body 是 Stream**:hyper Body 基于 `Stream` trait,承接 Tokio Stream 模型。
- **timer 用 tokio::time**:超时/keepalive 用 tokio 时间轮([[tokio-source-facts]] runtime/time/wheel)一句带过。
- **Service 返回 Future**:Service trait 是 `Fn(Request) -> Future<Output=Response>`,承接 Tokio Future/Poll 模型(标准库 core::future,不在 tokio 仓)。

**禁止把 Tokio 讲透的(reactor/scheduler/park/time wheel/task 状态位/Cell 内存布局)当新东西重复**——那是承接铁律的硬伤,违者返工。

### 承接《gRPC》(HTTP/2 实现对照)

gRPC C++ core 自己用 C 实现了 HTTP/2(chttp2,《gRPC》第 2 篇招牌);hyper 委托 `h2` crate 实现 HTTP/2。两种语言、两套实现,对照"HTTP/2 在真实库里怎么落地"。第 3 篇对照,HTTP/2 的帧/流/HPACK/流控在《gRPC》已拆透,一句带过,篇幅留 hyper 怎么用 h2 把请求映射成 stream。

### 横连《内存分配器》/《Envoy》

- `bytes::Bytes` 引用计数零拷贝,对照《内存分配器》和 gRPC slice。
- HTTP/1 状态机对照 Envoy HCM(你已立项的《Envoy》第 P3 篇)。

### 主比喻决策:直球为主,比喻点睛(沿用系列)

> **直球为主**:绝大多数地方第一性原理直给,不套比喻。**比喻点睛**:仅在开篇(P0-01)用"流水线"做一次性点位睛——Tokio 是"车间(运行时,工人/调度)",hyper 是"流水线(把 HTTP 这道工序装进车间)":协议机是"模具"(把字节压成请求/响应)、Service 是"工人"(处理一个请求)、连接池是"料车复用"(keep-alive 省握手)。全书不建立贯穿重比喻。沿用《RocksDB》《gRPC》定调。

## 六、写作约定(沿用《gRPC》《TiKV》《RocksDB》《JVM》系列)

1. **动机 + 技巧双线**(本书命脉,见第二节):每个关键代码点讲"为什么这么设计 + Tokio 怎么支撑/不这样会怎样 + hyper 怎么实现";每章单开"技巧精解"小节。
2. **"为什么"永远先于"怎么做"**:先讲它解决什么本质问题、Tokio/gRPC 怎么做、不这样会怎样,再讲 hyper 实现。
3. **大白话 + 比喻点睛**(直球为主):绝大多数地方第一性原理直给;仅在开篇用"流水线"做一次性点睛,不建立贯穿重比喻。
4. **关键路径配真实源码**:挑本章最核心的 1~2 处,贴真实片段、逐段讲。**技巧精解**固定在正文后、小结前。
5. **每章标准结构**:① 章首(核心问题 + 读完你会明白 + 难度高的加逃生阀);② 一句话点破;③ 正文若干小节(招牌四段式:**提问 → Tokio/gRPC 怎么做/不这样会怎样 → 所以 hyper 这么设计 → 源码/技巧佐证**);④ 技巧精解;⑤ 章末小结(回扣"协议侧 vs 框架侧"主线 + 五个为什么清单 + 想深入往哪钻 + 引出下一章)。
6. **回扣全局**:每章结尾回到二分法,说明本章服务"协议"还是"框架",衔接前后章;并交代"Takio 怎么支撑/对照 gRPC 怎么做"。
7. **语言**:中文,技术名词保留英文(hyper、Tokio、Service trait、AsyncRead/AsyncWrite、HTTP/1 状态机、h2、Stream、bytes、Body、连接池、keep-alive 等)。
8. **标点**:**半角 ASCII 逗号 `,`、冒号 `:`、括号 `()`、问号 `?`;全角顿号 `、`、破折号 `——`、引号 `""`、句号 `。`**;中英文之间留空格。
9. **文件命名**:`P{篇}-{章}-标题.md`。第 0 篇第 1 章 = `P0-01-…`,第 2 篇第 6 章 = `P2-06-…`。
