## 一、这本书的定位

### 目标读者

写过 Rust 异步、知道 Pingora 是 Cloudflare 开源的反向代理框架(撑起每秒 4000 万请求)、看过几篇 Pingora 源码分析,但**一知半解**:知道有 `ProxyHttp` trait、upstream、filter、连接池、负载均衡、Session,却讲不清一次 HTTP 请求从进 Pingora、被钩子链处理、选 upstream、经连接池转发、响应原路返回的整个过程;更讲不清 `ProxyHttp` trait 那 30 来个 filter 钩子(request_filter/upstream_peer/response_filter…)到底按什么顺序触发、为什么这样切;以及 Pingora 的自研运行时(NoSteal 多单线程池不做 work stealing)、L4/L7 连接池、Ketama 一致性哈希、零拷贝转发这些招牌到底在干什么。和"用过 Nginx 但不懂它"、"读过 Envoy 源码却一知半解"是同一种病。

### 全书一句话主旨

> **Pingora 把"代理一条 HTTP 请求"拆成一条可挂载钩子的请求生命周期——`ProxyHttp` trait 提供一串 filter 钩子(early_request_filter/request_filter/upstream_peer/upstream_request_filter/upstream_response_filter/response_filter/…),框架管 upstream 连接池(`TransportConnector` + `pingora-pool`)与负载均衡(`pingora-load-balancing`),业务只在钩子里写逻辑,全程跑在 Tokio 异步上(自研 `NoStealRuntime` 多单线程池不做 work stealing)。它是 Rust 异步网络栈"应用层·反向代理/网关"的代表。**

每个组件,都是这条钩子链上的一个**关卡**。读完一本书,读者不是"多记了几个 Pingora API",而是能**在脑子里放映出一次 HTTP 请求在 Pingora 里的全过程**:TCP 字节被 Tokio reactor 唤醒 → listener 接连接、每连接 spawn 一个 task → HTTP/1 状态机(自研,基于 `httparse`)或 HTTP/2(委托 `h2` crate)切成结构化请求 → 进入 `ProxyHttp` 钩子链(`early_request_filter` → `request_filter` 可直接短路响应 → `upstream_peer` 选后端 → `upstream_request_filter` 改请求 → `TransportConnector` 经连接池取/建到 upstream 的连接并透传 → `upstream_response_filter`/`response_filter` 改响应 → `logging`)——以及每一步底下,Tokio 是怎么被用起来、对照 Envoy 的 filter chain / Nginx 怎么做。

### 为什么用"钩子链(数据面) vs 框架设施(转发面)"当骨架(而非按模块平铺)

Pingora 是一个 multi-crate workspace(~16 个 crate:`pingora-core`/`pingora-proxy`/`pingora-load-balancing`/`pingora-cache`/`pingora-runtime`/`pingora-pool`/…),模块高度耦合:一条请求要穿过钩子链、被连接池转发、负载均衡挑后端。按 crate 百科式平铺,读者会觉得每块都是孤岛。用"**钩子链(`ProxyHttp` trait 的 filter 生命周期,业务挂载点)vs 转发设施(upstream 连接池/负载均衡/协议解析/缓存/运行时,框架自管)**"这条主轴当骨架,每个模块都是"为了让请求在钩子之间流转而必须出现的角色"——这正是用户提示词里的二分法种子,经核实源码后确认成立。

## 二、写作方法:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与数学系列、其他源码系列最关键的方法论,所有写作者必读、必守。**

Pingora 源码(Rust)里充斥高级技巧:`ProxyHttp` trait 用 `async_trait` 把每个 filter 钩子做成 async(承接 Tokio Future,且每请求一个 `CTX` 泛型贯穿全链做状态共享)、自研 `NoStealRuntime`(多个 current-thread runtime 池,避免 work-stealing 开销,与 Tokio 多线程 runtime 对照)、`TransportConnector` 的连接池(`Arc<Mutex<Stream>>` + 空闲 `idle_poll` + `test_reusable_stream` 用 1 字节探测连接死活)、HTTP/1 自研解析(基于 `httparse`)、HTTP/2 委托 `h2`(与 hyper 同根)、Ketama 一致性哈希(与 Nginx `hash consistent` 字节级兼容)、`HttpTask` 枚举统一 header/body/trailer 的透传、retry buffer(缓存请求体以便失败重试)、zero-copy 转发(Bytes 引用计数)、TLS 多后端可插换(openssl/boringssl/rustls/s2n)……用户自己读过源码,"一知半解"的缺口往往**不在**"它要解决什么问题"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书的每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?Tokio/Envoy/Nginx 是怎么做的、不这样会怎样?
2. **实现技巧(how 的精妙)**:Pingora 用了什么 Rust 手段?为什么妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号(已 clone(0.8.1 tag) 后钉死),单独拆透。技巧解释必须配**反面对比**——"如果朴素地实现 / Envoy(C++)/Nginx 怎么做",让妙处显形。

一句话:**讲不清 `ProxyHttp` trait 的钩子链、`TransportConnector` 连接池、NoStealRuntime、HTTP/1 自研解析,等于没讲 Pingora。**

> **⚠️ 深度标杆(用户强调,本书命脉)**:本书篇幅标杆拉到对标《SQLite》《RocksDB》《gRPC》《Envoy》《hyper》《MySQL》实际深度——**每章正文约 50000~68000 字符(招牌章奔 70000+)**。用户原话:"不要吝啬文字和篇幅,内容可以更丰富一点,要把问题讲解清楚,深入浅出,好理解又有深度。"所以每个机制至少讲透三层(动机 → Tokio/Envoy/Nginx 怎么做/不这样会怎样 → Pingora 怎么实现 + 为什么 sound),宁可多铺反例、对照(Envoy filter chain vs Pingora ProxyHttp、Nginx worker vs NoStealRuntime、hyper+h2 vs Pingora+h2),把问题讲到读者"原来如此",也不要一笔带过。与《hyper》《Envoy》《gRPC》《TiKV》一致。

## 三、全书主线、二分法、主比喻、承接

### 主线

> **把"代理一条 HTTP 请求"做成一条可挂载钩子的请求生命周期——业务实现 `ProxyHttp` trait 在钩子里写逻辑,框架管 upstream 连接池/负载均衡/协议解析/缓存/运行时,全程跑在 Tokio 异步上。**

任何一处看不懂某个机制,回到这句问:"这是**钩子链**(`ProxyHttp` 的 filter,业务挂载点),还是**转发设施**(upstream 连接池/负载均衡/协议/缓存/运行时,框架自管)?"这个机制 Tokio 是怎么支撑的、Envoy/Nginx 怎么做?"

### 二分法

本书的二分法(用户种子,经核实源码确认):

> **钩子链(控制面:`ProxyHttp` trait 的 filter 生命周期,业务在此挂载) vs 转发设施(数据面:upstream 连接池 / 负载均衡 / HTTP 解析 / 缓存 / 运行时 / TLS)。**

- **钩子链这一面(控制面)**:`pingora-proxy/src/proxy_trait.rs` 的 `ProxyHttp` trait,~30 个 filter 钩子:`early_request_filter` → `request_filter`(可短路)→ `request_body_filter` → `upstream_peer`(选后端)→ `proxy_upstream_filter` → `upstream_request_filter` → `connected_to_upstream` → `upstream_response_filter`/`response_cache_filter` → `upstream_response_body_filter` → `response_filter`/`response_body_filter` → `logging`;以及 `CTX` 泛型如何贯穿、`init_downstream_modules` 配置模块、`fail_to_proxy`/`error_while_proxy`/`fail_to_connect` 的错误处理钩子、retry 与 `persist_connection_context`/`on_connection_reuse` 的 keepalive 钩子。这些决定"业务在哪、按什么顺序介入请求"。
- **转发设施这一面(数据面)**:`pingora-core/src/connectors/`(`TransportConnector` L4/TLS 连接池 + HTTP connector L7)、`pingora-pool/`(底层连接池 LRU)、`pingora-load-balancing/`(`LoadBalancer` + 服务发现 + 健康检查 + Ketama 一致性哈希)、`pingora-core/src/protocols/http/{v1,v2}/`(HTTP/1 自研 + HTTP/2 委托 h2)、`pingora-cache/`(HTTP 缓存)、`pingora-runtime/`(NoStealRuntime)、`pingora-core/src/listeners/`(listener + connection_filter)、`pingora-core/src/upstreams/peer.rs`(`HttpPeer`/`Peer` trait)、TLS 多后端(`pingora-{openssl,boringssl,rustls,s2n}`)。这些决定"框架怎么把请求从 downstream 转到 upstream 再转回来字节级机制"。

> 命名说明:用户种子里写"数据面 vs 控制面/钩子"。本书为对齐承接,采用更精确的口径——**钩子链(业务介入点,偏控制) vs 转发设施(框架自管的字节流转发,偏数据)**。两者实为同义,只是把"钩子"提到主位(因为 Pingora 的灵魂就是 `ProxyHttp`)。读者迷路时,问一句"这块是业务写的钩子,还是框架自己管的转发?"即可定位。

### 承接《Tokio》(本书最大特色之一,务必遵守)

Pingora 全程跑在 Tokio 上,承接最紧密。**承接铁律**:Tokio 已讲透的,一句带过 + 指路 [[tokio-source-facts]],篇幅全留 Pingora 独有。具体承接点:

- **每个连接一个 Tokio task**:Pingora 的 `ListeningService` accept 一个连接就 spawn 一个 task,task 内跑钩子链 + 转发循环。Tokio 的 task/调度/预算让出([[tokio-source-facts]] budget=128)一句带过。
- **IO 用 AsyncRead/AsyncWrite**:Pingora 的 `Stream`(`Box<dyn AsyncRead + AsyncWrite + Send>`)直接建立在 Tokio 的 AsyncRead/AsyncWrite 上。Tokio 的 reactor/mio(epoll/kqueue,[[tokio-source-facts]] mio 1.2.1 edge-triggered)一句带过。
- **timer 用 tokio::time + pingora-timeout**:`pingora-timeout` 提供 `fast_timeout`(无锁,比 tokio::time::timeout 轻),底层仍是 tokio 时间轮([[tokio-source-facts]] runtime/time/wheel)一句带过。
- **Future/Poll 模型**:`ProxyHttp` 的 async 钩子就是 Future(标准库 core::future,不在 tokio 仓)。一句带过。
- **NoStealRuntime 是 Pingora 独有(承接后讲差异)**:`pingora-runtime` 的 `NoStealRuntime` 用多个 `current_thread` runtime 池替代 Tokio 的多线程 work-stealing runtime——这是 Pingora 自己的取舍(避免 work-stealing 开销),**这是本书独有,要详讲,对照 Tokio 多线程 runtime**。

**禁止把 Tokio 讲透的(reactor/scheduler/park/time wheel/task 状态位/Cell/budget/mio edge-triggered)当新东西重复**——那是承接铁律的硬伤,违者返工。

### ★ 同层对照《hyper》(本书核实的关键发现,务必讲清)

> **核实结论(改变凭记忆的旧印象)**:Pingora 在运行时**不依赖 hyper**。`pingora-core/Cargo.toml` 里 `hyper = "0.14"` 只出现在 `[dev-dependencies]`(测试/benchmark 用),运行时依赖里**没有 hyper**。Pingora 自己用 `httparse` crate 实现 HTTP/1(`pingora-core/src/protocols/http/v1/`),HTTP/2 委托 `h2` crate(`pingora-core/src/protocols/http/v2/`,与 hyper 用的是同一个 `h2` crate)。

所以 **Pingora 和 hyper 是 Tokio 之上的同级库(都建在 Tokio + h2 上),不是 Pingora 建在 hyper 上**。本书对照而非承接:
- HTTP/1:Pingora 自研(基于 `httparse`)vs hyper 自研(基于 `httparse`,但实现独立)——两套独立 HTTP/1 状态机,对照"HTTP/1 在不同库里怎么落地"。
- HTTP/2:Pingora 和 hyper **都用 `h2` crate**——这部分是**强同源**,讲 Pingora 怎么用 h2(对照 hyper 怎么用 h2,《hyper》P3-09~11 讲透的一句带过指路)。
- hyper 的 `Service` trait(Fn(Request)->Future<Response>)= HTTP 库抽象;Pingora 的 `ProxyHttp` trait(一串 filter 钩子)= 代理抽象。两者设计哲学对照(一个是"处理一个请求",一个是"代理一个请求,钩子贯穿")。

### ★ 强对照《Envoy》(同为反向代理,Rust 版 vs C++ 版)

Envoy 讲透的 filter chain(《Envoy》第 3 篇 HCM/http filter 一句带过),Pingora 的 `ProxyHttp` 钩子链是 Rust 异步版的同类设计。对照点:
- **Envoy filter chain**(listener filter → network filter → HCM → http filter → router)是多段链 + decoder/encoder 两向,**C++ 虚函数**驱动;**Pingora `ProxyHttp`** 是单一 trait 的一串 async 方法,Rust `async_trait` 驱动,顺序固定(early→request→upstream_peer→…→logging)。
- **Envoy xDS 控制面**(LDS/RDS/CDS/EDS 动态下发)是 Envoy 的灵魂;**Pingora 没有内置 xDS**(配置是 Rust 代码 + YAML,动态性靠业务自己在钩子里实现服务发现)。这是**根本差异**,本书讲清"Pingora 选择把动态性留给业务代码(实现 `ProxyHttp`),Envoy 选择把动态性做成 xDS 协议"。
- **Envoy worker + thread-local 无锁**(每 worker 一个 libevent dispatcher);**Pingora NoStealRuntime**(多个 current-thread runtime 池,无 work stealing)。两者都是"少量线程扛海量连接",但实现路径不同。

### 横连《gRPC》《内存分配器》

- HTTP/2 帧/流/HPACK/流控:Pingora 和 hyper 一样用 `h2`,与 gRPC C++ core 自实现 chttp2 是同协议不同实现。HTTP/2 协议在《gRPC》第 2 篇已拆透,一句带过。
- `bytes::Bytes` 引用计数零拷贝,对照《内存分配器》和 gRPC slice、hyper bytes。

### 主比喻决策:直球为主,比喻点睛(沿用系列)

> **直球为主**:绝大多数地方第一性原理直给,不套比喻。**比喻点睛**:仅在开篇(P0-01)用"**关卡/收费站**"做一次性点位睛——一条 HTTP 请求像一辆车开过 Pingora 这条公路,沿途是一串**关卡**(`ProxyHttp` 的 filter 钩子),业务在关卡上设岗(实现钩子):有的关卡直接放行、有的关卡改车牌(改 header)、有的关卡指路(选 upstream);而公路本身(连接池、负载均衡、协议解析、运行时)是框架修好的,业务不动。Pingora = "一条布满可挂载关卡的转发公路"。全书不建立贯穿重比喻。沿用《RocksDB》《gRPC》《hyper》《Envoy》定调。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线:开篇定调 → 钩子链(`ProxyHttp` trait)→ 转发设施(连接池/负载均衡/协议/运行时/TLS/缓存)→ 生产特性 → 收束。
> 章号全局连续;命名 `P{篇}-{章}-标题.md`。规模 **20 章 + 附录 AB**。

### 第 0 篇 · 开篇:为什么需要 Pingora(定调,1 章)

- **第 1 章 · 第一性原理:为什么用 Rust 异步写反向代理**(`P0-01`):Nginx(C,配置驱动)的痛——配置写死、动态性靠 lua/OpenResty 打补丁、内存安全靠人工;Envoy(C++,filter chain + xDS)的强与重。Pingora 的回答:用 Rust 内存安全 + 全异步 Tokio,把代理做成"实现 `ProxyHttp` trait 在钩子里写逻辑"的可编程框架。承 Tokio(运行时)、★同层对照 hyper(都建在 Tokio+h2 上,Pingora 自研 HTTP/1)、★强对照 Envoy(Rust 钩子 vs C++ filter chain + xDS)、对照 Nginx。点出 Cloudflare 撑起 40M+ req/s。**【样本章节,全书风格标杆】**

### 第 1 篇 · 钩子链:`ProxyHttp` trait 的请求生命周期(4 章,Pingora 灵魂)

> 这一篇是 Pingora 区别于"写死的代理"的根本:`ProxyHttp` trait。建议顺序读。

- **第 2 章 · `ProxyHttp` trait:一串 async filter 钩子**(`P1-02`,★招牌章)——Pingora 怎么把"代理一条请求"抽象成一个 trait 的一串 async 方法?技巧:★**`ProxyHttp` trait 的钩子全貌**(`pingora-proxy/src/proxy_trait.rs`,~30 个方法)、`type CTX` 泛型如何让每请求状态贯穿全链、`async_trait` 把每个 filter 做成 Future(承 Tokio Future)、为什么钩子是 async(重逻辑可 offload 到 blocking 线程不阻塞请求线程)。归属:`钩子链`。
- **第 3 章 · 请求前半段钩子:early/request_filter 与短路响应**(`P1-03`)——请求进来、还没选 upstream 之前能做什么?技巧:**`early_request_filter`(在模块前)/ `request_filter`(可返回 `Ok(true)` 短路响应,如鉴权拒绝/限流)/ `request_body_filter`(逐块处理请求体)/ `init_downstream_modules`(注册 compression 等 module)**;短路语义、与 Envoy decoder filter 的 `stop` 语义对照。归属:`钩子链`。
- **第 4 章 · upstream 选择与请求改写钩子**(`P1-04`)——怎么选后端、怎么改要发出去的请求?技巧:★**`upstream_peer`(返回 `HttpPeer`,核心选后端钩子)/ `proxy_upstream_filter`(cache miss 后决定是否真的转发)/ `upstream_request_filter`(改 upstream 请求头)**、`HttpPeer` 结构(sni/alpn/tls/path)。源码:`proxy_trait.rs`、`pingora-core/src/upstreams/peer.rs`。归属:`钩子链`。
- **第 5 章 · 响应与收尾钩子**(`P1-05`)——响应回来怎么改、请求结束做什么?技巧:**`upstream_response_filter`(改响应头,在缓存前)/ `response_filter`(改响应头,在缓存后)/ `upstream_response_body_filter`/`response_body_filter`(逐块改响应体)/ `response_trailer_filter`/`logging`(收尾打访问日志)/ `connected_to_upstream`(连接建立回调,记时延)**;filter 的顺序为何这么排(缓存前后区分)。归属:`钩子链`。

### 第 2 篇 · 转发设施·upstream 连接池(3 章,数据面招牌)

> 选好后端,怎么把连接建起来、怎么复用、怎么在 HTTP 层和 L4 层管池。

- **第 6 章 · `TransportConnector`:L4/TLS 连接与复用**(`P2-06`,★招牌章)——到 upstream 的 TCP/TLS 连接怎么建、怎么复用?技巧:★**`TransportConnector`**(`pingora-core/src/connectors/mod.rs`)、连接池(`pingora-pool` 的 `ConnectionPool<Arc<Mutex<Stream>>>`)、`get_stream`(先 `reused_stream` 再 `new_stream`)、`release_stream`(归还池 + `idle_poll` 空闲探测)、★**`test_reusable_stream`**(用 1 字节非阻塞读探测连接死活,避免把死连接复用出去)、`PreferredHttpVersion`(记住某 peer 该用 h1 还是 h2)、`offload_threadpool`(把建连这种 CPU 密集活 offload 到独立线程池)。归属:`转发设施(招牌)`。
- **第 7 章 · HTTP connector:L7 连接与 h1/h2 协商会话**(`P2-07`)——L4 stream 之上怎么建 HTTP 会话?技巧:**L7 connector**(`connectors/http/{v1.rs,v2.rs,mod.rs}`)、h1/h2 ALPN 协商、HTTP/1 连接的 keepalive 循环、HTTP/2 单连接多 stream 的复用、`HttpSession`(downstream 侧)/ 与 upstream 的会话对象。源码:`pingora-core/src/connectors/http/`、`protocols/http/{client.rs,server.rs}`。归属:`转发设施`。
- **第 8 章 · 零拷贝转发:`HttpTask` 与 body 流**(`P2-08`)——请求/响应字节怎么在 downstream 和 upstream 之间透传、怎么零拷贝?技巧:★**`HttpTask` 枚举**(Header/Body/Trailer/UpgradedBody/Done/Failed,统一 header/body/trailer 的透传单位)、**`response_duplex_vec`**(在 downstream 和 upstream 之间双向 pump)、`Bytes` 引用计数零拷贝(对照《内存分配器》/hyper bytes)、retry buffer(缓存请求体到 `FixedBuffer` 以便失败重试,`BODY_BUF_LIMIT=64KB`)。归属:`转发设施`。

### 第 3 篇 · 转发设施·负载均衡与服务发现(3 章)

> upstream_peer 钩子里怎么选后端、后端怎么发现、怎么知道健康。

- **第 9 章 · `LoadBalancer` 与 Backend 选择**(`P3-09`)——`upstream_peer` 里怎么从一堆后端挑一个?技巧:★**`LoadBalancer<S: BackendSelection>`**(`pingora-load-balancing/src/lib.rs`)、`select`/`select_with`(按 key 选,带 `max_iterations` 限界)、`UniqueIterator`(去重 + 限步)、`Backend`(addr + weight + ext)、`ArcSwap` 无锁更新 selector。归属:`转发设施(招牌)`。
- **第 10 章 · 选择算法:RoundRobin / Random / Ketama 一致性哈希**(`P3-10`,★招牌章)——具体怎么挑?技巧:★**`BackendSelection` trait**、`RoundRobin`/`Random`/`FNVHash`(都基于 `Weighted` 包装)、★**`Consistent` = `KetamaHashing`**(与 Nginx `hash consistent` 字节级兼容,`pingora-ketama` crate)、为什么一致性哈希能粘性。源码:`pingora-load-balancing/src/selection/{mod.rs,algorithms.rs,consistent.rs,weighted.rs}`、`pingora-ketama/src/lib.rs`。归属:`转发设施(招牌)`。
- **第 11 章 · 服务发现与健康检查**(`P3-11`)——后端列表哪来、怎么知道活没活?技巧:**`ServiceDiscovery` trait**(Static/DNS)、**`HealthCheck` trait**(`TcpHealthCheck`)、`Backends` 的 `do_update`(`ArcSwap` 原子更新)、`run_health_check`(可并行,`observe_health` 带阈值翻转)、`BackgroundService` 周期跑发现与检查。源码:`pingora-load-balancing/src/{discovery.rs,health_check.rs,background.rs}`。归属:`转发设施`。

### 第 4 篇 · 转发设施·HTTP 协议解析(3 章,协议招牌)

> Pingora 的 HTTP/1 自研、HTTP/2 委托 h2。这一篇讲字节怎么切成 HTTP。

- **第 12 章 · HTTP/1 自研解析(基于 httparse)**(`P4-12`,★招牌章)——Pingora 的 HTTP/1 为什么自己写、不用 hyper?技巧:★**HTTP/1 自研**(`pingora-core/src/protocols/http/v1/`,基于 `httparse` crate,`header.rs`/`client.rs`/`server.rs`/`body.rs`)、逐字节解析状态机、keep-alive 循环、chunked/100-continue、★**为什么不直接用 hyper**(对照 hyper HTTP/1,两套独立实现,Pingora 要对代理场景做 smuggling 防护——见 RUSTSEC-2026-0034)。归属:`转发设施(招牌)`。

- **第 14 章 · HTTP/1↔HTTP/2 协议转换**(`P4-14`)——downstream 是 h1、upstream 是 h2(或反过来)怎么转?技巧:**协议转换**(`bridge/`、`proxy_h1.rs`/`proxy_h2.rs`)、header 改写(h2 去 hop header `connection`/`transfer-encoding`/`keep-alive`)、body 与 trailer 的映射、`HttpTask::UpgradedBody`(websocket 等升级)。源码:`pingora-core/src/protocols/http/bridge/`、`pingora-proxy/src/{proxy_h1.rs,proxy_h2.rs}`。归属:`转发设施`。

### 第 5 篇 · 转发设施·运行时与 TLS(2 章)
