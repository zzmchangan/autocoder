## 一、写作铁律(命脉,不可违反)

### 1. 动机(why) + 技巧(how) 双线

每个关键代码点讲**两件事**:
- **设计动机(why)**:它解决什么本质问题?Tokio/hyper/Envoy/Nginx 是怎么做的、不这样会怎样?
- **实现技巧(how 的精妙)**:Pingora 用了什么 Rust 手段?为什么妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> 用户读过源码,"一知半解"的缺口往往**不在**动机(好懂),**在**技巧(看不懂)。所以技巧必须配**反面对比**——"朴素实现 / Tokio/hyper/Envoy 怎么做",让妙处显形。

### 2. "为什么"永远先于"怎么做"

先讲它解决什么本质问题、Tokio/hyper/Envoy 怎么做、不这样会怎样,**再**讲 Pingora 实现。禁止倒序。

### 3. 直球为主,比喻点睛

- **直球**:绝大多数地方第一性原理直给,不套比喻。
- **点睛**:仅在开篇(P0-01)用"关卡/收费站"做一次性点位睛(请求 = 车,`ProxyHttp` 钩子 = 沿途关卡,业务在关卡设岗;公路本身是框架修的)。
- **禁止**建立贯穿重比喻(不学《Tokio》餐厅服务员全程贯穿)。沿用《RocksDB》《gRPC》《hyper》《Envoy》定调。

### 4. 关键路径配真实源码

挑本章最核心的 1~2 处,贴真实片段、逐段讲。**严禁凭记忆写源码/行号**——每处引用写前用 Grep/Read(本地 `../pingora/`)核实;不确定精确行号时,宁可只标文件不标行号。简化展示必须标注"(简化示意,非源码原文)"。

## 二、每章标准结构(五段,固定)

### ① 章首(核心问题 + 读完你会明白)

- 开门见山抛**核心问题**(一句话):本章要回答什么。
- "读完你会明白":列 3~5 个具体收获(读者能讲清什么)。
- 难度高的章加**逃生阀**("如果只读一节,读第三节")。

### ② 一句话点破

用一两句话,把本章的核心机制点透(承主线"钩子链 vs 转发设施")。

### ③ 正文若干小节(招牌四段式)

每个关键机制按**招牌四段式**写:

1. **提问**:这个机制要解决什么问题?为什么需要它?(从读者能感知的痛点切入)
2. **承接方怎么做 / 不这样会怎样**:Tokio/hyper/Envoy/Nginx 怎么做?如果 Pingora 朴素地实现/照搬,会撞上什么墙?
3. **所以 Pingora 这么设计**:Pingora 的具体设计,为什么这么选(动机 + 取舍)。
4. **源码/技巧佐证**:贴真实源码片段(文件路径 + 行号),逐段拆实现技巧,配反面对比。

> 一个机制至少讲透这三层:动机 → 承接方/反例 → Pingora 实现 + 为什么 sound。宁可多铺反例、对照、图,把问题讲到读者"原来如此",也不要一笔带过。

### ④ 技巧精解(固定位置:正文之后、小结之前)

- 挑本章**最硬核的 1~2 个技巧**,单独拆透。

- **必须配反面对比**:"如果朴素地实现 / Tokio/hyper/Envoy 怎么做"。

## 三、承接铁律(篇幅分配的命脉)

> **承接铁律**:对方讲透的,一句带过 + 指路,篇幅全留 Pingora 独有。违反 = 稀释本书价值。

### 强承接《Tokio》(最紧密)

Tokio 讲透的,**一句带过 + 指路 [[tokio-source-facts]]**:
- reactor / mio(epoll/kqueue,edge-triggered)
- scheduler / work-stealing / park
- time wheel(runtime/time/wheel)
- task 状态位 / Cell 内存布局
- budget=128 让出

**但 NoStealRuntime 是 Pingora 独有取舍**:要详讲——Pingora 用多个 current-thread runtime 池替代 Tokio 多线程 work-stealing runtime,为什么(避免 work-stealing 开销)、怎么做(`NoStealRuntime`/`get_handle`/`init_pools`)、对照 Tokio 多线程 runtime。这是本书独有,不能一句带过。

### 同级对照《hyper》(不承接,讲同级)

核实结论:**Pingora 运行时不依赖 hyper**(hyper 只在 `[dev-dependencies]`)。Pingora 与 hyper 是 Tokio 之上的同级库。

- **HTTP/1**:Pingora 自研(基于 httparse,`protocols/http/v1/`)vs hyper 自研(基于 httparse,独立实现)——两套独立 HTTP/1 状态机,讲差异(尤其 smuggling 防护)。
- **HTTP/2**:Pingora 和 hyper **都用 `h2` crate**——强同源,讲 Pingora 怎么用 h2(对照《hyper》P3-09~11 讲透的一句带过指路)。
- hyper `Service`(Fn(Request)->Future<Response>)vs Pingora `ProxyHttp`(一串 filter 钩子)——设计哲学对照。

### 强对照《Envoy》(代理设计对照)

Envoy 讲透的(filter chain、xDS),**一句带过 + 指路**:
- Envoy filter chain(listener filter → network filter → HCM → http filter → router)→ 一句带过,讲 Pingora `ProxyHttp` 钩子链如何对应。
- Envoy xDS(LDS/RDS/CDS/EDS)→ 一句带过,讲 Pingora **没有内置 xDS**(动态性靠业务实现 `ProxyHttp` + 服务发现),这是根本差异。
- Envoy worker + thread-local 无锁 → 一句带过,讲 Pingora NoStealRuntime。

### 承接《gRPC》《内存分配器》

- HTTP/2 帧/流/HPACK/流控:《gRPC》第 2 篇拆透,一句带过指路,Pingora 怎么用 h2。
- `bytes::Bytes` 零拷贝:《内存分配器》拆透,一句带过。
