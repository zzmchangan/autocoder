# 《LevelDB 设计与实现深入浅出:为什么把写都变成追加》—— 目录与导读

> 一本写给"用过 LevelDB(或 RocksDB)、甚至翻过源码,却总觉得一知半解"的人的小书。
>
> **一句话主旨**:如何用"只追加、不原地改"换来写入的极致吞吐,代价是读/写/空间三笔放大,而 Compaction 是不断把它们收敛回来的会计。
>
> **二分法**(迷路时回到它):**前台路径**(写秒回、读多路归并拿正确最新值) vs **后台 Compaction**(把追加堆积的碎片、放大、过期版本一层层合并压缩掉)。
>
> **基调**:直球讲透为主,比喻只在反直觉处点睛——LevelDB 的机制具体可见,不需要贯穿比喻兜着。

每章一行:**一句话钩子** —— 技巧标签 —— 二分法归属(`前台` / `后台` / `衔接` / `一致性` / `收束`)。

## 第 0 篇 · 开篇:为什么把写都变成追加

- [P0-01 · 第一性原理:为什么把写都变成追加](P0-01-第一性原理-为什么把写都变成追加.md) —— 磁盘的真相是"顺序写快、随机写慢、原地改更慢",这一条物理事实逼出了 LSM:写只追加、改=盖标记、删=墓碑,后台 Compaction 慢慢收拾。B-tree 与 LSM 的根本分野在哪。 —— LSM/B-tree 分野、seek 的代价 —— `总览`

## 第 1 篇 · 写入的前台:WAL + MemTable

> 这一篇立起"一次写怎么进来"。先讲 API 基石、多版本键编码,再讲 MemTable 的骨架(跳表 + 内存池),最后讲并发写的批处理。**建议顺序读**。

- [P1-02 · Slice、Status、Comparator:API 三基石](P1-02-Slice-Status-Comparator-API三基石.md) —— 为什么不直接传 `std::string`(拷贝太贵);`Slice` 怎么零拷贝;`Status` 怎么一个值塞下状态码+消息;`Comparator` 怎么定义键序。 —— Slice 零拷贝 + 生命周期陷阱 —— `前台`
- [P1-03 · InternalKey 与 SequenceNumber:多版本键的编码](P1-03-InternalKey与SequenceNumber-多版本键的编码.md) —— 改了又改、删了又写,怎么区分新旧?把 `(user_key, seq, type)` 打包成一个可排序的 internal key,"取最新版本"退化成"取排序后第一个"。 —— seq+type 尾部打包、比较器降序 —— `前台`
- [P1-04 · SkipList:MemTable 的骨架](P1-04-SkipList-MemTable的骨架.md) —— 为什么 MemTable 用跳表不用红黑树?因为跳表能"写者加锁、读者无锁"。几何概率层数怎么来的。 —— `std::atomic<Node*>` 无锁读 + 内存序 —— `前台`
- [P1-05 · Arena 与 MemTable:bump 分配](P1-05-Arena与MemTable-bump分配.md) —— MemTable 的内存从哪来、为什么不逐块 free、为什么"只进不出"反而更快更省。 —— bump allocator + 整块回收 —— `前台`
- [P1-06 · WriteBatch 与写组:group commit](P1-06-WriteBatch与写组-group-commit.md) —— 一次 `Write` 到底先写 WAL 还是先写 MemTable?多线程并发写时,LevelDB 怎么让一个 leader 代表整组把 N 次写合并成 1 次 WAL 追加。 —— leader/follower 写组批处理 —— `前台`

## 第 2 篇 · SSTable 格式:文件长什么样

> MemTable 满了要落盘。落盘后的文件怎么紧凑、怎么被快速定位。

- [P2-07 · SSTable 文件格式:四级布局](P2-07-SSTable文件格式-四级布局.md) —— 一个 `.ldb` 文件:data block、filter block、index block、footer,为什么 footer 固定在文件末尾当"锚"。 —— footer 固定锚 + BlockHandle 变长偏移 —— `衔接`
- [P2-08 · Block 的内部:前缀压缩 + restart point](P2-08-Block的内部-前缀压缩与restart-point.md) —— 一个 block 内相邻 key 前缀重复,怎么共享前缀省空间,又怎么用 restart point 保住二分查找。 —— 共享前缀 + restart point —— `前台`
- [P2-09 · Filter Block:布隆过滤器](P2-09-Filter-Block-布隆过滤器.md) —— 读一条 key,先问"这个 block 八成有没有它",挡掉绝大多数无效查询。布隆的 k 个 hash 其实只用一个 hash。 —— DoubleHashing + 误判率数学 —— `前台`
- [P2-10 · TableBuilder 与 Table:构建与打开](P2-10-TableBuilder与Table-构建与打开.md) —— 一堆 KV 怎么边排序边写成文件;打开文件时 `two_level_iterator` 怎么让 index 懒指向 data。 —— 两级迭代器懒加载 —— `衔接`

## 第 3 篇 · 读取:多路归并的艺术

> 怎么把散在 MemTable + Immutable + 多层 SSTable 里的版本,拼成一条正确的最新结果。

- [P3-11 · Iterator 抽象与 DBIter](P3-11-Iterator抽象与DBIter.md) —— 为什么所有数据源都实现同一个 `Iterator`?`DBIter` 怎么把 internal key 翻译回 user 眼里的 key、跳过被删的。 —— 统一接口 + 装饰器叠加 —— `前台`
- [P3-12 · MergingIterator:多路归并](P3-12-MergingIterator-多路归并.md) —— 读一条要同时翻 memtable、imm、每层若干 SSTable,怎么归并成一条有序流、取到最新版本就停。 —— k 路归并取最新 —— `前台`
- [P3-13 · 读路径全流程:Get 怎么找到那条记录](P3-13-读路径全流程-Get怎么找到那条记录.md) —— 一次 `Get` 的完整路径:memtable → imm → 每层 SSTable(布隆 → index 二分 → data block),每一道"提前剪枝"如何联手压低读放大。 —— 多级剪枝 + 采样触发压实 —— `前台`

## 第 4 篇 · Compaction:后台的灵魂

> 全书重头戏。把追加堆积的碎片、放大、过期版本一层层合并压缩掉。

- [P4-14 · Version 与 VersionSet:层级状态机](P4-14-Version与VersionSet-层级状态机.md) —— "数据库此刻长什么样"怎么表达?compaction 切换 Version 时,正在读的旧 reader 为什么不被打扰。 —— Version 引用计数 + 整体可见性 —— `后台`
- [P4-15 · Compaction 的触发与选取](P4-15-Compaction的触发与选取.md) —— 什么时候触发(level-0 文件数 / level-N 大小)、选哪个文件压(按上次结束 key 轮转,公平覆盖)、level-0 为什么特殊。 —— 轮转选取 + 10x 层级约束 —— `后台`

## 第 5 篇 · 不丢不乱:WAL、Manifest 与崩溃恢复

> 进程崩了、机器挂了,怎么恢复到一致状态。

- [P5-17 · WAL 的格式与读写](P5-17-WAL的格式与读写.md) —— WAL 为什么用 32KB block、一条 batch 跨 block 怎么分片、CRC32C 怎么防撕裂。 —— 32KB 对齐 + record 分片 + CRC —— `一致性`
- [P5-18 · Manifest 与崩溃恢复全流程](P5-18-Manifest与崩溃恢复全流程.md) —— `CURRENT` 指向当前 Manifest,Manifest 是"版本的编辑日志";打开 DB 时怎么 `Recover`:读 CURRENT → 重放 Manifest → 重放 WAL。 —— 两层日志分工 + CURRENT 指针 —— `一致性`
