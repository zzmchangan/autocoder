# 《LevelDB 设计与实现深入浅出:为什么把写都变成追加》—— 全书总纲

> 一本写给"用过 LevelDB(或 RocksDB)、甚至翻过源码,却总觉得一知半解"的人的小书。
>
> 本文是全书的**总纲**:定主线、定方法、定分篇分章、定源码策略、定写作约定。
> 与《Tokio》《数据库内核》《容器》同源,都是"**第一性原理 + 真实源码**";命脉同样是**动机(why) + 技巧(how) 双线**;但本书有一条与 Tokio 截然不同的基调:**直球讲透为主,比喻只在反直觉处点睛**——因为 LevelDB 的机制大多具体可见,不需要一个贯穿全书的比喻来"兜住理解"。

## 一、这本书的定位

### 目标读者

会写 C++、用过 LevelDB / RocksDB 的 API(`Open` / `Put` / `Get` / `Delete` / `Iterator` / `CompactRange`),甚至翻过 `db_impl.cc`,但**一知半解**:知道有 memtable、SSTable、compaction、WAL,却讲不清它们怎么协作、为什么这么设计、源码里那些 `std::atomic<Node*>`、`Arena` 的 bump 分配、`varint`、写组(group commit)、`GUARDED_BY` 到底在干什么。和"读过 tokio 源码但不理解"是同一种病。

### 全书一句话主旨

> **磁盘的物理真相是"顺序写快、随机写慢、原地改更慢"。LevelDB 把每一次写都变成顺序追加——先追加到 WAL 防丢,再写进内存 MemTable 立即可读;内存满了整块刷成 SSTable 文件;文件多了,后台 Compaction 一层层把它们合并、去重、排序。这样换来写入的极致吞吐,代价是读放大、写放大、空间放大三笔账,而 Compaction 就是那个不断把这三笔账收敛回来的会计。**

每个组件,都是这条旅程上的一个**驿站**。读完一本书,读者不是"多记了几个 API",而是能**在脑子里放映出一次 `Put` 的一生 + 后台 Compaction 的全过程**:一条写怎么进来、先进 WAL 还是先进 MemTable、内存满了怎么变文件、文件长什么样、读的时候怎么把散在内存和多层文件里的版本拼成一条正确结果、文件多了后台怎么收拾——以及每一步底下用了什么巧妙的手段。

### 为什么用"一次 Put 的一生 + 后台收拾"当骨架(而非按模块平铺)

LevelDB 的模块(`db` / `table` / `util` / `include`)高度耦合:写要先进 memtable 再进 SSTable,读要同时翻 memtable 和多层 SSTable,compaction 又要把 SSTable 重写、还要靠 Manifest 记账、靠 WAL 兜底。按模块百科式平铺,读者会觉得每块都是孤岛、不知道**为什么要学这个**。用"一条写怎么进来、怎么不丢、怎么变成文件、怎么被读出来、怎么被后台收拾"这条旅程当骨架,每个模块都是"为了让旅程继续而必须出现的角色"——读者永远知道**我在哪、为什么需要下一站**。

## 二、写作方法

### 命脉:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与 API 文档、与一般源码导读最关键的分野,所有写作者必读、必守。**

LevelDB 源码里充斥 C++ 系统级技巧:无锁原子读(SkipList 的 `std::atomic<Node*>` + 内存序)、引用计数(cache handle、`Version`)、零拷贝(`Slice`)、内存池(`Arena` 的 bump 分配)、位运算(`varint` 解码、bloom 的 bit 数组、`CRC32C`)、写组(group commit 的 leader/follower 批处理)、`GUARDED_BY` 并发注解、`no_destructor` 延迟析构。用户自己读过源码,"一知半解"的缺口往往**不在**"它要解决什么问题"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?(沿用系列一贯的"为什么优先")
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果不用这个技巧、朴素地写,会撞上什么墙",让妙处显形。

一句话:**讲不清技巧,等于没讲 LevelDB。**

### 基调:直球讲透为主,比喻只在反直觉处点睛 ⚠️ 本书与 Tokio 的关键区别

> **这一节是本书的方法论基调,由用户拍板("也不用非要用比喻,如果不是很抽象的话,能讲清楚就行")。**

Tokio 的 async 运行时**反直觉**(轮询为何高效、为何不阻塞、为何把等待从占用解放),所以必须用"餐厅服务员"这种贯穿比喻来兜住理解。**LevelDB 不一样**:它的机制大多**具体可见**——跳表就是跳表、SSTable 就是排好序的文件、compaction 就是归并排序。这些直球讲清楚最清楚,硬套一个贯穿比喻反而啰嗦、喧宾夺主、还可能不准。

所以本书基调:

- **默认直球**:把数据结构、流程、源码本身讲透。能用一张表、一张图、一段源码说清的,**不上比喻**。
- **比喻只点睛,不贯穿,不另起体系**:只在少数**反直觉或本质抽象**处,用一两句话点一个比喻帮助"啊哈",点到为止。例如:
  - LSM 的整体心智("写都变追加、改=盖标记、删=墓碑、后台慢慢收拾"):可点一句"像先记流水账、后台再清账归档"。
  - Compaction:可点一句"像档案员一层层把散册合并、去重、归档"。
  - 但**绝不**展开成术语对照表、**绝不**强制每章出现、**绝不**和别的比喻混用。
- **判断标准**:直球讲不清的,才上比喻;讲得清的,就直球。比喻服务于理解,不是装饰,更不是 KPI。

> 这一条对标《数据库内核》系列的"弱化比喻直球"。读者要的是"它凭什么这么设计、源码里那段 C++ 在干什么",不是"再来一个比喻"。

## 三、全书主线、二分法

### 主线

> **如何用"只追加、不原地改"换来写入的极致吞吐,代价是读/写/空间三笔放大,而 Compaction 是不断把它们收敛回来的会计。**

任何一处看不懂某个机制,回到这句问:"这是在让**前台更快更准**(写秒回、读拿到最新正确值),还是在让**后台把堆积的代价收敛**(合并、去重、压缩、回收)?把答案写进"为什么"里。

### 二分法(迷路时回到它)

本书的二分法:

> **前台路径(让写秒回、让读拿到正确最新值) vs 后台 Compaction(把追加堆积的碎片、放大、过期版本,一层层合并压缩掉)。**

- **前台这一面**:WAL + MemTable(写)、MemTable + 多层 SSTable 多路归并(读)、Iterator 体系、缓存。
- **后台这一面**:Immutable 刷盘、Compaction 合并、`VersionSet` 版本管理、Manifest 记账、`RemoveObsoleteFiles` 垃圾回收。

LevelDB 每个机制都落在这两面之一(或两者衔接处)。任何时候迷路,回到这个二分法。

> 注:WAL 与 Manifest 既服务前台(不丢、可读),也服务后台(崩溃恢复)。它们是衔接处,讲的时候要说明"它兜的是前台的哪一面、后台的哪一面"。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线是一条旅程:**为什么这么设计 → 一次写怎么进来、怎么不丢 → 落成什么样的文件 → 怎么读出来 → 后台怎么收拾 → 崩了怎么不丢不乱 → 性能基建 → 收束**。
> 章号全局连续;命名 `P{篇}-{章}-标题.md`,开篇第 1 章 = `P0-01`,第 1 篇第 2 章 = `P1-02`,以此类推。**全书 21 章**,对标《Tokio》。

### 第 0 篇 · 开篇:为什么把写都变成追加(定调,1 章)

- **第 1 章 · 第一性原理:为什么把写都变成追加**(`P0-01`):磁盘的物理真相(顺序写快、随机写慢、原地改更慢,seek 的代价)→ B-tree 的取舍(读优写劣,原地更新=随机写)→ LSM 的取舍(写优、读/空间需放大,改=盖标记、删=墓碑)→ LevelDB 三件事(WAL 不丢 + MemTable 立即可读 + Compaction 收敛放大)。立起"前台 vs 后台 Compaction"二分法。**【样本章节,见后】**

### 第 1 篇 · 写入的前台:WAL + MemTable(5 章)

> 一次写怎么进来、怎么不丢、怎么在内存里排序好。这一篇是全书地基。

- **第 2 章 · Slice、Status、Comparator:API 三基石**(`P1-02`)——为什么不直接传 `std::string`(拷贝太贵)、`Slice` 怎么做到零拷贝;`Status` 怎么用一个值同时表达"ok/NotFound/ Corruption/..."(状态码 + 消息打包);`Comparator` 怎么定义键序(多态基类)。源码:`include/leveldb/slice.h`、`include/leveldb/status.h`、`util/status.cc`、`include/leveldb/comparator.h`。**技巧精解:`Slice` 的 `{data_, size_}` 零拷贝语义 + 生命周期陷阱**(Slice 不拥有内存);**`Status` 的状态打包**(一个 `Status` 怎么塞进 code + 两个长度消息)。
- **第 3 章 · InternalKey 与 SequenceNumber:多版本键的编码**(`P1-03`)——为什么不能只存 user_key(改了又改、删了又写,得有版本);怎么把 `(user_key, sequence_number, type)` 打包成一个**可排序**的 internal key,让"同 key 取最新版本"退化成"取排序后第一个"。源码:`db/dbformat.h`、`db/dbformat.cc`。**技巧精解:internal key 的尾部追加编码**(64 位 seq+type 打包,比较器先比 user_key 再比 seq 降序)——这是 MVCC 与多版本的根。
- **第 4 章 · SkipList:MemTable 的骨架**(`P1-04`)——为什么 MemTable 用跳表不用红黑树(并发友好:写者加锁、读者**无锁**);几何概率层数怎么来的(`1/4` 提升概率);插入/查找 O(log n)。源码:`db/skiplist.h`、`db/skiplist_test.cc`。**技巧精解:单写多读的无锁读**——写者 `std::atomic` store、读者 `std::atomic` load + 合适内存序,凭什么不会数据竞争(`Insert` 先 new 节点再用 release store 挂上去,reader acquire load 看到的要么是旧要么是完整新)。
- **第 5 章 · Arena 与 MemTable:bump 分配**(`P1-05`)——MemTable 的内存从哪来;`Arena` 为什么不逐块 free(bump allocator 只进不出,整块回收);`ApproximateMemoryUsage` 怎么算。源码:`util/arena.h`、`util/arena.cc`、`db/memtable.h`、`db/memtable.cc`。**技巧精解:bump allocator + 不释放单块**——为什么这么"浪费"反而更快、更省(减少 malloc/free 碎片、减少锁);`memtable` 怎么把跳表 + 引用计数装进一个内存表。
- **第 6 章 · WriteBatch 与写组:group commit**(`P1-06`)——一次 `Write` 到底先写 WAL 还是先写 MemTable(先 WAL);`WriteBatch` 怎么把一批操作打包成一条原子记录;多线程并发 `Write` 时,LevelDB 怎么用**写组**(leader 代表整组写、follower 干等)把 N 次写合并成 1 次 WAL 追加 + 1 次 MemTable 插入。源码:`db/db_impl.cc`(`Write` / `MakeRoomForWrite` / `BuildBatchGroup`)、`db/write_batch.cc`。**技巧精解:写组的 leader/follower 批处理**(group commit)——写入吞吐的核心手段;反面对比"每个 writer 各写各的 WAL"会怎样。

### 第 2 篇 · 持久化的第一道:SSTable 的格式(4 章)

> MemTable 满了要落盘。落盘后的文件长什么样、怎么紧凑、怎么被快速定位。

- **第 7 章 · SSTable 文件格式:四级布局**(`P2-07`)——一个 `.ldb` 文件怎么组织:data block、(可选)filter block、metaindex block、index block、footer;为什么 footer 固定在文件末尾(打开文件先读 footer 反向定位)。源码:`table/format.h`、`table/format.cc`、`doc/table_format.md`。**技巧精解:footer 作为"固定锚"**——文件末尾的固定大小块,让打开时一次 seek 就拿到整张文件的索引根;BlockHandle 的变长偏移编码。
- **第 8 章 · Block 的内部:前缀压缩 + restart point**(`P2-08`)——一个 block 内的 key 为什么不做完整存储(相邻 key 前缀重复多);共享前缀压缩怎么省空间;`restart point` 怎么让压缩后的 block 仍能二分查找(每隔 K 条存一个完整 key 重启点)。源码:`table/block_builder.cc`、`table/block.cc`。**技巧精解:前缀压缩 + restart point**——压缩省空间 vs 二分要求随机访问的矛盾,用 restart point 妥协;反面对比"每条存完整 key"会怎样(空间爆炸)。
- **第 9 章 · Filter Block:布隆过滤器**(`P2-09`)——读一条 key,为什么要先问"这个 block 里八成有没有它"(挡掉绝大多数无效查询,削读放大);布隆过滤器的 k 个 hash、bit 数组、误判率。源码:`util/bloom.cc`、`util/filter_policy.cc`、`table/filter_block.cc`。**技巧精解:布隆的 k 个 hash 实际只用一个 hash + 位段切分**(`util/bloom.cc` 的 `DoubleHashing`)、误判率与 bits/key、k 的关系(数学);反面对比"不用布隆、每次都读 block"会怎样(读放大爆炸)。
- **第 10 章 · TableBuilder 与 Table:构建与打开**(`P2-10`)——`TableBuilder::Add` 怎么把一堆 KV 一边排序一边写成文件、何时 flush block、何时写 index/filter;`Table::Open` 怎么打开;`two_level_iterator` 怎么让"index 指向 data"懒加载。源码:`table/table_builder.cc`、`table/table.cc`、`table/two_level_iterator.cc`。**技巧精解:两级迭代器(Two-Level Iterator)**——上层 index 迭代器定位、下层 data 迭代器懒打开,缓存友好。

### 第 3 篇 · 读取:多路归并的艺术(3 章)

> 怎么把"散在 MemTable + Immutable + 多层 SSTable"里的版本,拼成一条正确的最新结果。

- **第 11 章 · Iterator 抽象与 DBIter**(`P3-11`)——为什么所有数据源(MemTable、SSTable)都实现同一个 `Iterator` 接口(组合);`DBIter` 怎么把 internal key`(user_key, seq, type)`翻译回 user 眼里的 key、跳过被删的。源码:`include/leveldb/iterator.h`、`db/db_iter.cc`。**技巧精解:统一的 `Iterator` 接口 + 装饰器叠加**(MergingIterator / TwoLevelIterator / DBIter 像洋葱层层包),凭什么能组合出任意读路径。
- **第 12 章 · MergingIterator:多路归并**(`P3-12`)——读一条 `Get`,要同时翻 memtable、imm、每层的若干 SSTable,怎么把它们归并成一条有序流、取到最新版本就停。源码:`table/merger.cc`。**技巧精解:多路归并的优先队列/胜者逻辑**——k 路有序流取最小,同 key 取 seq 最大的(internal key 已编码好),拿到第一个有效值即返回;反面对比"把所有层读出来再全局排序"会怎样。
- **第 13 章 · 读路径全流程:Get 怎么找到那条记录**(`P3-13`)——一次 `Get` 的完整路径:memtable → imm → 每层 SSTable(table_cache 打开 → 布隆过滤 → index block 二分 → data block → 读出);`Version::Get` 怎么按层找。源码:`db/db_impl.cc`(`Get`)、`db/version_set.cc`(`Version::Get`)。**技巧精解:读路径上每一道"提前剪枝"**(布隆、index、缓存)如何联手压低读放大;`RecordReadSample` 怎么据此触发 compaction(自动压实热点)。

### 第 4 篇 · 后台的灵魂:Compaction(3 章)

> 全书真正的重头戏。把追加堆积的碎片、放大、过期版本,一层层合并压缩掉。

- **第 14 章 · Version 与 VersionSet:层级状态机**(`P4-14`)——某一刻"数据库长什么样"怎么表达(`Version`:每层一组 `FileMetaData`,带 key range);`VersionSet` 管理版本更替;为什么读用旧 Version、写产生新 Version(读不阻塞写)。源码:`db/version_set.h`、`db/version_set.cc`。**技巧精解:Version 的引用计数 + 整体可见性**——compaction 产出一批新文件后,原子地"切换 Version",旧 reader 还在用旧 Version,用完析构回收;这就是"前台读不被后台写打扰"的根。
- **第 15 章 · Compaction 的触发与选取**(`P4-15`)——什么时候触发(level-0 文件数 > 阈值 / level-N 总大小 > 10^N MB);选哪个文件压(`PickCompaction`:按上次 compaction 的结束 key 轮转,公平覆盖 key 空间);level-0 特殊(文件间可能重叠,要一起压)。源码:`db/version_set.cc`(`PickCompaction`、`Compaction`)、`db/db_impl.cc`(`MaybeScheduleCompaction`)。**技巧精解:轮转选取策略 + 10x 层级大小约束**——为什么是 10 倍(放大与层数的权衡)、轮转避免某段 key 反复压。
- **第 16 章 · Compaction 的执行:归并生成新文件**(`P4-16`)——拿到输入文件后,用 `MergingIterator` 归并,边归并边写新 SSTable(每 2MB 切一个文件,或 key range 涨到重叠 > 10 个下层文件时切);丢弃被覆盖的旧值、在合适时机丢弃 tombstone。源码:`db/db_impl.cc`(`DoCompactionWork`、`OpenCompactionOutputFile` / `FinishCompactionOutputFile`)。**技巧精解:归并排序 + 何时丢弃 tombstone**(只有所有更深层都没有重叠 key 时才能扔墓碑,否则会"复活"旧值)+ 写放大的来源(同一条数据被反复重写)。

### 第 5 篇 · 不丢不乱:WAL、Manifest 与崩溃恢复(2 章)

> 进程崩了、机器挂了,怎么恢复到一致状态。

- **第 17 章 · WAL 的格式与读写**(`P5-17`)——WAL 为什么用 32KB 的 block、record 怎么分片(一条 batch 可能跨 block,分 `kFirstType` / `kMiddleType` / `kLastType`);CRC32C 校验;`log::Writer` 追加、`log::Reader` 逐 record 读。源码:`db/log_format.h`、`db/log_writer.cc`、`db/log_reader.cc`。**技巧精解:32KB block 对齐 + record 分片**——为什么是 32KB(留 6 字节 header 后是 2 的幂,对齐开销小)、CRC 防止撕裂(torn write);反面对比"一个 record 一行、不管对齐"会怎样。
- **第 18 章 · Manifest 与崩溃恢复全流程**(`P5-18`)——`CURRENT` 指向当前 Manifest、Manifest 是"版本的编辑日志"(`VersionEdit` 一条条追加:加文件/删文件/改层级);打开 DB 时怎么 `Recover`:读 CURRENT → 重放 Manifest 重建 VersionSet → 重放未归档的 WAL 重建 memtable。源码:`db/version_edit.cc`、`db/version_set.cc`(`Recover` / `WriteLevel0Table` / `RecoverLogFile`)、`db/db_impl.cc`(`Recover`)。**技巧精解:两层日志的分工**——WAL 兜"用户数据"未刷盘部分、Manifest 兜"文件结构"变更,崩溃后各自重放回一致状态;`CURRENT` 单文件指针解决"哪个 Manifest 是最新的"。

### 第 6 篇 · 性能基础设施(2 章)

> 支撑前面一切的性能基建。

- **第 19 章 · ShardedLRUCache:block cache 与 table cache**(`P6-19`)——LevelDB 为什么手写 LRU 不用 STL(要支持容量按字节 `charge`、handle 内联引用计数、pin 语义);16 路分片降低锁争用;block cache(缓存解压后的 block)与 table cache(缓存打开的 Table 对象)各缓存什么。源码:`util/cache.cc`、`util/cache_test.cc`。**技巧精解:LRU 的双向链表 + 哈希 + handle 引用计数**(一个 `Cache::Handle*` 同时是链表节点、带 `refs`/`charge`,被引用时不进 LRU 淘汰);反面对比"用 std::list + unordered_map"会怎样(做不到 pin、做不到字节预算)。
- **第 20 章 · Env 抽象:文件、线程、时间**(`P6-20`)——为什么把所有平台相关操作抽象成 `Env`(依赖注入、可测试);`Env::Posix` 与 `Env::Windows` 各自实现;`helpers/memenv/memenv.cc` 怎么用纯内存实现 `Env` 供测试。源码:`include/leveldb/env.h`、`util/env_posix.cc`、`util/env_windows.cc`、`helpers/memenv/memenv.cc`。**技巧精解:依赖注入做平台/测试隔离**——`Env` 虚函数表替换,业务代码零改动即可在内存 env 上跑测试;`no_destructor`(`util/no_destructor.h`)怎么避免全局单例的析构顺序坑。

### 第 7 篇 · 收尾:快照与权衡哲学(1 章)

- **第 21 章 · Snapshot/MVCC 与三放大哲学**(`P7-21`)——`GetSnapshot` 怎么靠 SequenceNumber 实现快照隔离(读时只看 <= 该 seq 的版本);把全书收束成几条哲学(只追加不原地改、前台快后台收、一把大锁换简单、用空间/读放大换写吞吐、version 引用计数让读写不互斥)。源码:`db/snapshot.h`、全书回顾。**技巧精解:SequenceNumber 实现 MVCC 快照**(零开销——只是读迭代器多带一个 seq 上界);全书**读放大 / 写放大 / 空间放大**三笔账的总账单与权衡。

## 七、C++ 技巧侧重(对标 Tokio 的 Rust `unsafe`/原子/`Pin`)

本书要重点拆的 C++ 系统级技巧(写作时遇到这些,务必钻"怎么做到的 + 为什么 sound"):

- **并发模型**:一把大锁 `mutex_` 保护几乎所有状态 + 后台线程 + `GUARDED_BY(mutex_)` 注解(`port/thread_annotations.h`)+ `std::atomic`(如 `has_imm_`、`shutting_down_`)。LevelDB 用"一把大锁"换简单,这是它和 Tokio(无锁优先)的鲜明对照。
- **无锁读**:SkipList 的 `std::atomic<Node*>` + 内存序(`acquire`/`release`),单写多读凭什么不会数据竞争。
- **内存布局与所有权**:`Slice` 零拷贝(不拥有内存,生命周期陷阱)、`Arena` bump 分配、cache `Handle` 引用计数、`Version` 引用计数、`no_destructor` 延迟析构(规避全局对象析构顺序坑)。
- **位运算**:`varint` 变长解码(`util/coding.h`)、bloom 的 bit 数组与 `DoubleHashing`、`CRC32C`(`util/crc32c.cc`)、internal key 的 seq+type 打包。
- **模板与泛型**:`SkipList` 模板、`Comparator` / `FilterPolicy` / `Env` 虚函数多态、`Iterator` 装饰器组合。
