## 一、这本书的定位

### 目标读者

写过 Java、用 `jstat`/`jmap` 看过堆、配过 `-Xmx`/`-XX:+UseG1GC`、调过 GC 或看过 JIT 相关文章,甚至翻过 `openjdk/jdk` 的 `src/hotspot/`,但**一知半解**:知道有类加载、字节码、解释器、C1/C2、堆、GC、TLAB、Mark Word、偏向锁、safepoint,却讲不清一次 `new Object()` 在堆里怎么布局分配、一段 Java 方法怎么从解释执行被分层编译成机器码、G1 的 Region 和 SATB 怎么做到可预测停顿、ZGC 凭什么 sub-ms 暂停、`synchronized` 怎么从偏向锁升级到重量级锁、Java 21 虚拟线程和 goroutine 到底差在哪。和"读过 LuaJIT 源码但不理解"、"翻过 Go runtime 却一知半解"是同一种病。

### 全书一句话主旨

> **Java 用"字节码跨平台 + JIT 即时编译 + GC 自动内存"三件套,把"一次编写处处运行"的跨平台抽象开销,用 JIT 赚回性能、用 GC 赚回生产力——HotSpot 就是那个把抽象代价赚回来的 C++ 运行时。**

每个组件,都是这套"赚回来"的一个环节。读完一本书,读者不是"多记了几个 JVM 参数",而是能**在脑子里放映出一段 Java 代码的一生**:字节码怎么被类加载器加载、解释器怎么按操作数栈执行、热点方法怎么被探测、C1/C2 怎么分层编译成机器码、`new` 出来的对象在堆里怎么布局、TLAB 怎么无锁分配、GC 怎么三色标记回收、`synchronized` 怎么锁升级、虚拟线程怎么把阻塞从占用解放——以及这每一步底下,Go runtime / LuaJIT 是怎么做的对照(承接成网)。

### 为什么用"执行子系统 vs 内存子系统"当骨架(而非按模块平铺)

HotSpot 的模块(类加载/解释器/C1/C2/对象模型/堆/GC/锁/JMM)高度耦合:一段代码的执行穿过类加载→解释器→JIT,它 new 的对象要分配进堆、由 GC 回收,它的锁要靠对象头 Mark Word。按模块百科式平铺,读者会觉得每块都是孤岛。用"**执行子系统(代码怎么跑起来:类加载→字节码→解释器→JIT)vs 内存子系统(数据怎么存怎么收:对象布局→堆分配→GC)**"这条主轴当骨架,横切 JMM/锁/虚拟线程,每个模块都是"为了让代码跑得又快又安全而必须出现的角色"。

## 二、写作方法:动机(why) + 技巧(how)双线 ⚠️ 本系列命脉

> **这一节是本书与数学系列、其他源码系列最关键的方法论,所有写作者必读、必守。**

HotSpot 源码(几百万行 C++)里充斥高级技巧:template interpreter 的汇编模板、C2 的 Sea-of-Nodes IR(逃逸分析/标量替换/锁消除/内联)、uncommon trap 逆优化、分层编译 tiered、对象头 Mark Word 的位复用、compressed oops、TLAB 无锁 bump-pointer 分配、G1 的 Region 化 + SATB + Pause Prediction Model、ZGC 的染色指针 + 读屏障、synchronized 的锁升级(偏向→轻量→重量)、AQS 的 CLH 队列、safepoint 全局协调……用户自己读过源码,"一知半解"的缺口往往**不在**"它要解决什么问题"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书的每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?Go runtime / LuaJIT / 操作系统是怎么做的对照、不这样会怎样?
2. **实现技巧(how 的精妙)**:HotSpot 用了什么 C++ 手段?为什么妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果朴素地实现 / 别的运行时怎么做的",让妙处显形。

一句话:**讲不清 C2 Sea-of-Nodes、G1 SATB、ZGC 染色指针、锁升级,等于没讲 JVM。**

> **⚠️ 深度标杆(用户强调,本书命脉)**:本书篇幅标杆拉到对标《LuaJIT》深入浅出第一性原理版——**每章约 12000~18000 字**。用户原话:"不要吝啬文字和篇幅,内容可以更丰富一点,要把问题讲解清楚,深入浅出,好理解又有深度。"所以每个机制至少讲透三层(动机 → 别的运行时怎么做的反例 → HotSpot 怎么做到 + 为什么 sound),宁可多铺反例、对照(Go GC vs JVM GC、goroutine vs 虚拟线程、LuaJIT trace JIT vs C2 method JIT)、图,把问题讲到读者"原来如此",也不要一笔带过。与《gRPC》《TiKV》《RocksDB》一致。

## 三、全书主线、二分法、主比喻、承接

### 主线

> **Java 用"字节码跨平台 + JIT + GC"三件套,把跨平台抽象的代价赚回来——HotSpot 是那台赚回代价的 C++ 运行时。**

任何一处看不懂某个机制,回到这句问:"这是在**执行子系统**(让代码跑起来:类加载/字节码/解释器/JIT),还是在**内存子系统**(让数据存得住收得回:对象/堆/GC)?这个机制 Go runtime / LuaJIT 是怎么做的对照?"

### 二分法

本书的二分法:

> **执行子系统(类加载→字节码→解释器→JIT 编译) vs 内存子系统(对象布局→堆分配→GC 回收)。**

- **执行子系统这一面**:类加载与双亲委派、字节码格式与常量池、template interpreter 解释执行、C1/C2 分层编译与 JIT、uncommon trap 逆优化——这些决定"代码怎么跑、跑得多快"。
- **内存子系统这一面**:对象内存布局(Mark Word/Klass Pointer)、堆与 TLAB 分配、Metaspace、各种 GC(Serial/G1/ZGC/Shenandoah)——这些决定"对象怎么存、谁回收"。
- **横切两面**:JMM(happens-before)、synchronized 锁升级、JUC/AQS、虚拟线程、safepoint(GC 与 JIT 的全局协调点)。

### 承接成网(本书最大特色,务必遵守)

JVM 不是孤立运行时,它是运行时线的**承重墙**,一本横连多条线。**承接铁律**:相关概念在对应系列已讲透的,一句带过 + 指路,篇幅全留 HotSpot 独有。

- **JIT ← [[luajit-series-project]]**:LuaJIT 是 **trace JIT**(录热循环路径),HotSpot C2 是 **method JIT**(整方法编译);LuaJIT 用线性 IR + 反向 LSRA,C2 用 **Sea-of-Nodes IR** + 图着色寄存器分配;LuaJIT "sink" ≠ C2 标量替换(memory 已钉死);LuaJIT guard/side exit ↔ C2 uncommon trap 逆优化;LuaJIT 不做向量化,C2 有 SuperWord。第 2 篇处处对照。
- **GC ← [[alloc-series-project]] + [[go-runtime-series-project]]**:Go GC 是并发三色标记非分代;JVM GC 是**分代 + 多算法**(G1/ZGC)。Go mcache 的 tiny allocator ↔ JVM TLAB。第 4 篇对照。
- **锁 ← [[linux-sync-series-project]]**:JVM `synchronized` 重量级锁底层是 `ObjectMonitor` + futex/park;AQS 是 CLH 变种队列锁。对照 Linux futex/mutex。第 5 篇对照。
- **虚拟线程 ← [[go-runtime-series-project]] + [[tokio-series-project]]**:Java 21 virtual thread ↔ Go goroutine(都是 M:N 用户态调度)↔ Tokio task(异步)。第 6 篇三栈对照。
- **托底 Java 分布式线**:RocketMQ/Dubbo/Kafka 都跑在 JVM 上,本书讲透 JVM,读那些系统时"这段 Java 代码在 JVM 里怎么跑"有答案。

### 主比喻决策:直球为主,比喻点睛(沿用系列)

> **直球为主**:绝大多数地方第一性原理直给,不套比喻。**比喻点睛**:仅在开篇(P0-01)用"翻译官"做一次性点位睛——字节码是"世界语",JVM 是"随身翻译官"(跨平台靠它),解释器是"口译"(慢但即时)、JIT 是"笔译成母语"(慢一次但后快)、GC 是"保洁阿姨"(你不用管,自动收)。全书不建立贯穿重比喻。沿用《RocksDB》《gRPC》定调。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线:代码怎么跑(执行)→ 数据怎么存收(内存)→ 怎么并发(横切)→ 怎么生产可用 → 收束。

### 第 1 篇 · 执行地基:类加载与字节码(3 章)

> 代码进 JVM 的第一关:加载、字节码、解释执行。

- **第 2 章 · 类加载与双亲委派**(`P1-02`)——`.class` 怎么被加载进 JVM?双亲委派为什么这么设计?技巧:**双亲委派**(安全 + 一致性)、**类加载器命名空间**、破坏双亲委派(SPI/Tomcat)。源码:`src/hotspot/share/classfile/classLoader.cpp`、`systemDictionary.cpp`。归属:`执行`。
- **第 3 章 · 字节码格式与常量池**(`P1-03`)——`.class` 文件长什么样?技巧:**字节码指令集**(操作码 + 操作数)、**常量池**(符号引用)、栈式字节码(操作数栈 + 局部变量表)。对照 LuaJIT 带编号栈槽字节码(memory 已钉:LuaJIT 非匿名栈顶)。源码:`src/hotspot/share/interpreter/bytecode.cpp`、`classfile/classFileParser.cpp`。归属:`执行`。
- **第 4 章 · 解释器与字节码执行**(`P1-04`)——字节码怎么被执行?技巧:★**template interpreter**(模板汇编,每条字节码对应一段汇编模板,而非 switch 解释)、**操作数栈**计算模型、invokedynamic。源码:`src/hotspot/share/interpreter/templateTable_*`、`interpreter_x86*.cpp`。归属:`执行`。

### 第 2 篇 · JIT 编译:把字节码变机器码(4 章,性能招牌)

> 这一篇是 JVM 性能的核心,也是与 LuaJIT 双璧对照的主战场。

- **第 5 章 · 分层编译 tiered**(`P2-05`)——为什么要分层(解释器→C1→C2)?技巧:**分层编译**(C2 等"足够热"再编译、C1 快速出码占位)、**热点探测**(方法调用计数 + 回边计数)、**method JIT vs LuaJIT trace JIT 对照**(整方法 vs 录热循环)。源码:`src/hotspot/share/compiler/`、`compileBroker.cpp`。归属:`执行(招牌)`。
- **第 6 章 · C1 编译器**(`P2-06`)——client 编译器怎么快速出码?技巧:**C1 的线性 IR + SSA**、优化等级(不含逃逸分析)、占位作用。源码:`src/hotspot/share/c1/`(c1_Compiler/c1_IR.cpp 等)。归属:`执行`。
- **第 7 章 · C2 与 Sea-of-Nodes IR**(`P2-07`,★全书皇冠招牌章)——C2 凭什么把 Java 编到接近 C 的性能?技巧:★**Sea-of-Nodes IR**(数据流 + 控制流显式建模,对照 LuaJIT 线性 IR)、★**逃逸分析与标量替换**(对象不上堆,对照 LuaJIT sink —— memory 已钉 LuaJIT sink≠标量替换)、**锁消除/粗化**、**内联**、**SuperWord 自动向量化**(LuaJIT 不做向量化,这是 method JIT 的优势)。源码:`src/hotspot/share/opto/`(compile.cpp、escape.cpp、superword.cpp)。归属:`执行(招牌)`。
- **第 8 章 · 逆优化 deoptimization**(`P2-08`)——JIT 编译基于的假设(类层次、类型)失效了怎么办?技巧:★**uncommon trap + 逆优化**(假设失败回退解释器,对照 LuaJIT guard/side exit)、**去优化到解释器**。源码:`src/hotspot/share/runtime/deoptimization.cpp`。归属:`执行`。

### 第 3 篇 · 内存地基:对象与堆(3 章)

> 数据怎么存。这一篇是 GC 的地基(对象布局看不懂,GC 无从讲起)。

- **第 9 章 · 对象内存布局**(`P3-09`)——一个 Java 对象在堆里长什么样?技巧:★**对象头(Mark Word + Klass Pointer)+ 实例数据 + 对齐填充**、Mark Word 的**位复用**(锁状态/GC 分代年龄/hash 都挤进一个字)、**compressed oops**(压缩指针省内存)。源码:`src/hotspot/share/oops/markWord.hpp`、`oop.hpp`。归属:`内存`。
- **第 10 章 · 堆分配与 TLAB**(`P3-10`)——`new` 一个对象怎么进堆?技巧:★**TLAB(Thread-Local Allocation Buffer)**无锁 bump-pointer 分配(对照 Go mcache tiny allocator / 内存分配器 thread-local cache)、**指针碰撞 vs 空闲列表**、逃逸分析不上堆(承接 P2-07)。源码:`src/hotspot/share/gc/shared/`、`threadLocalAllocBuffer.hpp`。归属:`内存(招牌)`。
- **第 11 章 · Metaspace 与直接内存**(`P3-11`)——元数据(类信息)存哪?NIO 的堆外内存怎么管?技巧:**Metaspace(替代永久代)**、**直接内存与 Cleaner**。源码:`src/hotspot/share/memory/metaspce/`。归属:`内存`。

### 第 4 篇 · GC:自动内存管理全家桶(5 章,灵魂招牌)

> 这一篇是全书最重。JVM GC 是工业级自动内存管理的标杆,与 Go GC 形成对照。

- **第 12 章 · GC 基础:可达性与三色标记**(`P4-12`)——怎么判断对象该回收?技巧:**GC Roots 可达性分析**、★**三色标记 + 漏标问题**(白/灰/黑,SATB/增量更新的根)、**safepoint**(GC 的全局暂停协调)。对照 Go GC 三色标记。源码:`src/hotspot/share/gc/shared/`。归属:`内存`。
- **第 13 章 · 分代假说与 Minor GC**(`P4-13`)——为什么分代?技巧:**分代假说**(朝生夕死)、**Young/Old 区**、**复制算法**(Eden/Survivor)、**卡表 Card Table**(跨代引用)。源码:`src/hotspot/share/gc/shared/generation.cpp`、`src/hotspot/share/gc/serial/`。归属:`内存`。
- **第 14 章 · G1 GC**(`P4-14`,★招牌章)——G1 凭什么做到可预测停顿?技巧:★**Region 化堆**(不用固定分代物理划分)、★**SATB(Snapshot-At-The-Beginning)**、**混合回收 Mixed GC**、★**Pause Prediction Model**(用户设停顿目标,GC 算这次收多少 Region)。源码:`src/hotspot/share/gc/g1/`(g1CollectedHeap.cpp、g1SATBMarkQueueSet.cpp)。归属:`内存(招牌)`。
- **第 15 章 · ZGC**(`P4-15`,★招牌章)——ZGC 凭什么 sub-ms 暂停?技巧:★**染色指针(Colored Pointers)**(指针高位存 GC 状态)、★**读屏障(Load Barrier)**(读引用时自行处理)、**并发标记/转移/重定位**、generational ZGC(JDK 21+).源码:`src/hotspot/share/gc/z/`(zBarrier.cpp、zCollectedHeap.cpp)。归属:`内存(招牌)`。
- **第 16 章 · Shenandoah 与 GC 选型**(`P4-16`)——Shenandoah 怎么并发压缩?各 GC 怎么选?技巧:**转发指针(Brooks Pointer)/ load reference barrier**、并发整理(对照 G1/ZGC)、**GC 选型决策表**(Serial/Parallel/G1/ZGC/Shenandoah 各适用场景)。源码:`src/hotspot/share/gc/shenandoah/`。归属:`内存`。

### 第 5 篇 · 并发与同步(3 章,横切)

- **第 17 章 · Java 内存模型 JMM**(`P5-17`)——多线程下共享变量怎么保证可见/有序?技巧:★**happens-before**、**volatile**(可见性 + 禁重排)、**final 语义**。对照 Linux 同步原语的内存序([[linux-sync-source-facts]] smp_mb 等)。源码:`src/hotspot/share/runtime/orderAccess.hpp`。归属:`横切`。
- **第 18 章 · synchronized 与锁升级**(`P5-18`,★招牌章)——`synchronized` 到底怎么实现?技巧:★**锁升级(偏向→轻量→重量)**、★**Mark Word 编码锁状态**(承接 P3-09 位复用)、**ObjectMonitor**(重量级锁,接 Linux futex/park,[[linux-sync-source-facts]] futex_wake)、偏向锁在 JDK 15+ 弃用。源码:`src/hotspot/share/runtime/objectMonitor.cpp`、`synchronizer.cpp`。归属:`横切(招牌)`。
- **第 19 章 · JUC 与 AQS**(`P5-19`)——ReentrantLock/CountDownLatch 怎么实现?技巧:★**AQS(AbstractQueuedSynchronizer)的 CLH 变种队列**、**CAS + park/unpark**、volatile state。接 Linux 同步原语 CAS/futex。源码:JUC 在 JDK 类库(`src/java.base/share/classes/java/util/concurrent/`),AQS 是 Java 实现(非 HotSpot C++)。归属:`横切`。

### 第 6 篇 · 生产可用(3 章)

- **第 20 章 · 线程模型与虚拟线程**(`P6-20`,★招牌章)——Java 21 虚拟线程和普通线程、goroutine 差在哪?技巧:**平台线程(1:1 内核线程)→ 虚拟线程(M:N 用户态调度)**、**continuation + scheduler**、虚拟线程在 IO 阻塞时 unmount。★对照 Go goroutine([[go-runtime-source-facts]] GMP)/Tokio task(三栈对照)。源码:`src/java.base/share/classes/java/lang/VirtualThread.java`、`src/hotspot/share/runtime/continuation.cpp`。归属:`横切(招牌)`。
