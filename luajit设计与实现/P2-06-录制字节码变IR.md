# P2-06 录制:字节码变 IR(record)

> **本书主线**:把动态执行安全变成机器码。**二分法**:解释器侧 ↔ JIT 侧。本章是 **JIT 侧流水线的第一个真实动作**——P2-05 把 trace 生命周期讲成了状态机,RECORD 是其中最长的一段;这一章深入 RECORD:解释器再跑一遍热点时,怎么同时把每条字节码翻译成 IR、怎么在假设处插 guard、怎么遇到循环就收尾。**★对照**:官方 Lua(没录制)+ JVM/V8(从字节码/AST 静态构建 IR vs LuaJIT 运行时录制)。**源码**:LuaJIT 2.1.ROLLING,`lj_record.c`(2940 行,JIT 侧最大的源文件)+ `lj_record.h` + `lj_ir.c`/`lj_ir.h`。**基调**:纯直球,不用比喻;从第一性原理一步步推导。

---

## 引子:从 P2-05 的状态机,到 RECORD 的内部

P2-05 把一条 trace 的一生讲成了一条状态机:`START → RECORD → END → ASM → 安装`。其中 RECORD 是这么一句(`lj_trace.c:680` 的 `trace_state`,P2-05 §3 贴过):

```c
case LJ_TRACE_RECORD:
  trace_pendpatch(J, 0);
  setvmstate(J2G(J), RECORD);
  ...
  lj_record_ins(J);
  break;
```

整条状态机外层套着 `do { ... } while (J->state > LJ_TRACE_RECORD)`,条件是"状态值大于 RECORD(=2)才继续转"。这意味着:**RECORD 每转一圈只录一条字节码,然后 `break` 退出 switch,`while` 判定状态还是 RECORD(不大于它),于是退出循环返回 NULL,把控制权交还解释器**。解释器去执行下一条字节码,又触发回调,再进来录一条……

所以 RECORD 阶段的真正工作,压在了那个被反复调用的 `lj_record_ins(J)` 上(`lj_record.c:2226`)。这一章就是把这个函数的内部拆开:解释器执行的一条字节码,是怎么被同时"翻译"成一条或几条 IR、怎么"假设"出具体类型、怎么在假设处插上 guard、怎么在循环回边处收尾。

P2-05 §3.1 还讲过一个关键性质:**RECORD 期间程序其实在正常前进**——它不是离线编译,而是"让解释器再跑一遍热点路径,边跑边记"。这个"边跑边记"是 trace JIT 的精髓,也是这一章为什么叫"录制(record)"而不是"翻译(translate)"或"编译(compile)"。这一章要把"录制"这两个字彻底讲透。

## §1 第一性原理:为什么是"录制"

动笔看源码之前,我们必须先把这一章要回答的核心问题推导清楚。这个问题是:**热点被发现之后,要把它变成机器码,第一步为什么必须是"录制"这一步?能不能跳过它,直接拿字节码编译?**

这一节全部用来回答这个问题。我们不假设读者懂编译原理,从最基础的事实一步步推。

### 1.1 障碍一:静态分析拿不到运行时类型

回忆 P0-01 §4 的核心障碍:Lua 是动态类型语言,看代码看不出一个变量下一秒是什么类型。

```lua
function add(a, b) return a + b end
add(1, 2)       -- 这次 a,b 是整数
add(1.5, 2.5)   -- 这次 a,b 是浮点
add("x", "y")   -- 这次 a,b 是字符串,会报错
```

`a + b` 在这三次调用里语义完全不同。要把 `a + b` 翻译成机器码,你得先知道是哪一种。

那能不能像 C 那样,**读一遍代码**就推断出来?C 行,因为类型写在源码里(`int add(int a, int b)`)。但 Lua 不行——Lua 的变量没有类型声明,而且同一份代码可能被不同类型的数据反复调用。

你可能想:**那做"类型推断"呢?** 像某些静态分析工具那样,顺着控制流和数据流分析,推断出"这里大概是整数"。理论上可以,但对动态语言很难做准确,而且**推断出来的结论可能跟实际运行情况不一致**——分析说 `a` 大概率是整数,但实际跑到第 100 万次时 `a` 突然变成字符串。

如果基于"大概率的推断"生成了机器码,而推断错了,机器码就会算错。这又退回 P0-01 §7 的核心难题了。

### 1.2 障碍二:静态分析不知道哪条分支真走

还有第二个障碍。看 P0-01 §10 那个例子:

```lua
for i = 1, 1000000 do
  if i < 900000 then
    x = x + 1       -- 这条路,跑了 90 万次
  else
    x = x - 1       -- 这条路,只跑了 10 万次
  end
end
```

循环里有个 if/else。读一遍代码,你能看到两个分支,但你看不出"实际运行时,绝大多数时候走哪一条"。静态分析只能把两个分支都当成"可能走",于是要同时为两条路生成机器码,或者生成带条件判断的通用机器码——这就是 JVM/V8 的 method JIT 干的事(P0-01 §14 对照过)。

但 LuaJIT 选了 trace 路线:**只录实际在走的那条线性路径,把它编成机器码**。这就要求它必须知道"实际在走的是哪一条"。读代码读不出来——只有让程序真跑一遍才知道。

### 1.3 唯一可靠的解:让程序真跑一遍,边跑边记

把上面两个障碍合起来,结论非常自然:

> 既然编译时(读代码时)既不知道类型、也不知道实际走哪条路,那唯一的可靠信息来源,就是**程序真正运行时的那次执行**。

这就是"录制"的本质:**让解释器把热点路径从头到尾再跑一遍(注意是再跑一遍,不是凭空分析),但这一次,每执行一条字节码,LuaJIT 同时把它"翻译"成一条或几条 IR 指令,追加到正在构建的 trace 里**。

这样做的三个好处,正好对应 P0-01 主线的三股张力:

**第一,类型来自真实运行。** 录制时,解释器真正取到了变量的值,这个值是什么类型,录制器就照着这个类型生成对应的 IR(整数就生成整数 IR、浮点就生成浮点 IR)。**这是真实发生的事实,不是推断**——所以它一定正确(至少对这一次执行正确)。这就是 trace JIT "看真实类型"的精髓。

**第二,路径来自真实分支。** 录制时,if/else 真实走了某一条路(因为解释器在真实执行),录制器就把这条路的字节码依次录进 IR。另一条没走的路,自然不进 IR。这就是 P0-01 §10 说的"只录一条线性路径"。

**第三,假设都配 guard,所以安全。** 类型只对录制那一次执行正确,那万一以后某次类型变了呢?录制器在生成"假设 a 是整数"的 IR 时,**同时**插入一条 guard IR("运行时检查 a 真的是整数")。检查通过就继续(P0-01 §7 的"乐观"部分),检查失败就退出(P0-01 §8 的 side exit)。所以"录的真实类型"+"配的 guard"=既快又安全。

这就是为什么这一步叫**录制**而不是翻译或编译:**翻译/编译是从一种静态形式到另一种静态形式的转换,而录制是"让程序动起来,边动边记"**。它是 trace JIT 区别于 method JIT 的根本动作。

### 1.4 一个最小例子:for 循环里的 `x = x + i`,怎么被一条条录成 IR

为了把"录制"看具体,我们回到 P0-01 §12 那个贯穿全书的例子,这次把"录制"那一步放大:

```lua
local x = 0
for i = 1, 1000000 do
  x = x + i
end
```

假设 hotcount 已经触发,`trace_start` 已经初始化好录制环境(P2-05 §5.1),RECORD 状态启动。这时**解释器又开始跑这个循环体一圈**,但这次每执行一条字节码,就回调 `lj_record_ins` 录一条。我们逐条看(这里只看概念,具体 IR 操作码 §2 细讲):

循环体大致被前端(`lj_parse.c`)编译成这样的字节码序列(简化,真实顺序见 §2):

```
; x 在某个栈槽,i 是循环变量
FORL      循环回边(其实录制从 FORL 的下一条开始)
... 循环体 ...
ADDVV  x_slot, x_slot, i_slot    ; x = x + i  ← 这是关键一条
... 调整 i、回边 ...
```

录制开始时,录制器要做几件事:

1. **把入口字节码的栈状态初始化成 IR 引用**。`x` 和 `i` 这两个栈槽,被翻译成 IR 里的"加载这个栈槽"指令(`IR_SLOAD`,§2.3 细讲)。这一步同时观察运行时的类型——这次跑,`x` 和 `i` 都是整数,所以 `IR_SLOAD` 的类型标记成 `IRT_INT`,并在它上面附带一个 guard("运行时检查这个槽真的是整数")。这两条 `IR_SLOAD` 就是 trace 的"输入"。

2. **录到 `x = x + i` 那条 `ADDVV`**。解释器这次执行它,真的取了 `x` 和 `i` 的值(都是整数),做了整数加法。录制器**同时**看到这两个槽的 IR 引用(都是 `IRT_INT`),于是发射一条整数加法的 IR:`IR_ADD`,类型 `IRT_INT`,操作数是上一步的两个 `IR_SLOAD` 引用。**这次录的加法,和解释器这次实际做的加法,是同一种**——这就是"录制忠实记录真实类型"。

3. **循环变量 `i` 的递增和回边**也被录成对应 IR(§2.7 讲循环时细讲)。

4. **录到循环回边**(即将跳回循环开头),录制器发现"这条路径从头到尾走完了,而且回到了起点",于是调用 `lj_record_stop(J, LJ_TRLINK_LOOP, ...)`,把这条 trace 标记成"循环回自己",结束录制。

录完之后,IR 里大致是这样(简化,真实还有循环优化展开的细节):

```
IR_SLOAD  x_slot   (IRT_INT, 带 guard)
IR_SLOAD  i_slot   (IRT_INT, 带 guard)
... 循环变量的步进 ...
IR_ADD    = (SLOAD x) + (SLOAD i)   (IRT_INT)
... 把结果存回 x 的槽、回边 ...
```

注意三件事,它们是这一章反复要强调的核心:

- **每条 IR 都对应一次真实的解释器执行**。`IR_ADD` 那条不是凭空想的,是解释器这次真的做了整数加法,录制器照着录的。
- **类型 `IRT_INT` 来自这次真实执行**。如果这次 `x` 是浮点,录的就是 `IR_ADD` + `IRT_NUM`。
- **`IR_SLOAD` 上带了 guard**。这条 trace 装上机器码跑起来后,每次都检查 `x` 槽还是不是整数;某次变了,guard 触发,side exit 退回解释器(P0-01 §8)。

这就是录制的全貌。剩下这一章,是把上面这四步的每一步,在源码里逐行指出来。

## §2 源码印证:录制的主干 lj_record_ins

读者已经被 §1 扶到了"录制是什么、为什么这么干"。现在用源码**印证**前面讲清的道理。

录制的入口是 `lj_record_ins`(`lj_record.c:2226`),它是整个 JIT 侧最大的函数,也是 2940 行 `lj_record.c` 的核心。它的主体是一个**巨型 switch**,每个 case 对应一种字节码操作码(BC op)。我们分几层把它拆开。

### 2.1 双轨执行:解释器跑、录制器记

先看函数头部到 switch 之前的预处理(`lj_record.c:2226-2345`):

```c
void lj_record_ins(jit_State *J)
{
  cTValue *lbase;
  RecordIndex ix;
  const BCIns *pc;
  BCIns ins;
  BCOp op;
  TRef ra, rb, rc;

  /* Perform post-processing action before recording the next instruction. */
  if (LJ_UNLIKELY(J->postproc != LJ_POST_NONE)) {
    switch (J->postproc) {
    case LJ_POST_FIXCOMP:  /* Fixup comparison. */
      ...
    case LJ_POST_FIXGUARD:  /* Fixup and emit pending guard. */
    case LJ_POST_FIXGUARDSNAP:  /* Fixup and emit pending guard and snapshot. */
      ...
    }
    J->postproc = LJ_POST_NONE;
  }

  /* Need snapshot before recording next bytecode (e.g. after a store). */
  if (J->needsnap) {
    J->needsnap = 0;
    if (J->pt && bc_op(*J->pc) < BC_FUNCF) lj_snap_purge(J);
    lj_snap_add(J);
    J->mergesnap = 1;
  }

  /* Skip some bytecodes. */
  if (LJ_UNLIKELY(J->bcskip > 0)) {
    J->bcskip--;
    return;
  }

  /* Record only closed loops for root traces. */
  pc = J->pc;
  if (J->framedepth == 0 &&
     (MSize)((char *)pc - (char *)J->bc_min) >= J->bc_extent)
    lj_trace_err(J, LJ_TRERR_LLEAVE);
  ...
```

(`lj_record.c:2226-2301`,有删节)

这一段在做录制一条字节码**之前**的准备工作。我们关注两个关键点,它们体现了录制的"双轨"本质:

**第一,`J->pc` 指向的是"解释器即将执行的字节码"**。看 `lj_trace.c:770` 的 `lj_trace_ins`(P2-05 §4.1 贴过):它在调 `trace_state` 之前设了 `J->pc = pc`。这个 `pc` 是解释器 dispatch 循环传进来的"下一条要执行的字节码"。所以 `lj_record_ins` 一进来,`pc = J->pc` 拿到的,就是**解释器这一刻真正要执行的那条字节码**。这就是"双轨"的第一轨——解释器正在真实执行什么,录制器就处理什么。两者看的是同一条字节码,只是解释器真去执行它(更新真实栈状态),录制器把它翻译成 IR(追加到 `J->cur.ir`)。

**第二,`lbase = J->L->base` 拿到解释器真实的栈基址**(`lj_record.c:2316`)。后面 `ix.valv`、`ix.tabv`、`ix.keyv` 这些"v 结尾"的字段(`lj_record.h:14-26` 的 `RecordIndex` 结构体),存的就是**解释器这次执行时这些槽里真实的 TValue**。录制器不光看字节码长什么样,还要去解释器的真实栈里取这次的真实值——比如判断这次 `x` 是整数还是浮点,就是看 `lbase[ra]` 这个 `TValue` 的类型标记。

这两个机制合起来,精确地编码了"双轨执行":**解释器执行一条字节码(改真实状态)+ 录制器把同一条字节码翻成 IR(基于真实状态)**。两者共享同一份"这次执行的真实事实"——pc、栈值、类型——所以录出来的 IR 永远忠实于这一次执行。

继续往下,switch 之前还有一段把字节码的 a/b/c 操作数**统一翻译成 TRef**(`lj_record.c:2311-2345`)。这段是关键,因为它处理了"栈槽 → IR 引用"这个映射,我们单独用一节讲。

### 2.2 栈槽 → IR 引用:getslot 与 sload

字节码是栈式的——它的操作数大多是"栈槽号"(`BCReg`,一个表示栈位置的整数)。比如 `ADDVV ra, rb, rc` 意思是"把栈槽 rb 和 rc 相加,结果放栈槽 ra"。但 IR 不是栈式的,IR 是 SSA 形式的(P2-07 详讲)——每条 IR 指令有一个唯一的编号(`IRRef`),操作数直接引用这个编号。

所以录制时必须维护一个映射:**栈槽号 → 当前对应的 IR 引用**。这个映射表存在 `jit_State` 里(`lj_jit.h:425,429,479`):

```c
typedef struct jit_State {
  ...
  TRef *base;        /* Current frame base, points into J->slots. */
  BCReg maxslot;     /* Relative to baseslot. */
  BCReg baseslot;    /* Current frame base, offset into J->slots. */
  ...
  TRef slot[LJ_MAX_JSLOTS+LJ_STACK_EXTRA];  /* Stack slot map. */
  ...
} jit_State;
```

`J->slot[]` 是一个大数组,下标是栈槽的**绝对位置**(相对于整个 Lua 栈底部),每个元素是一个 `TRef`(tagged IR 引用,P2-07 讲 TRef 的位布局)。`J->base` 是个指针,指向"当前函数帧在 slot 数组里的起点"——`J->base = J->slot + J->baseslot`(`lj_record.c:2838` 在 `lj_record_setup` 里设)。这样 `J->base[s]`(相对栈基址的栈槽 s)就取到这个槽对应的 `TRef`。

录制开始时,这些槽是空的(0)。**第一次**某个栈槽被用到时,录制器要"加载"它——生成一条 `IR_SLOAD` 指令,把"从解释器栈的这个位置加载一个值"翻成 IR,然后把这条 `IR_SLOAD` 的引用存进 `J->base[s]`。以后这个槽再用到,直接拿 `J->base[s]` 里那条 `IR_SLOAD` 的引用就行,不用重复加载。

这个逻辑封装在一个宏里(`lj_record.c:211`):

```c
/* Get TRef from slot. Load slot and specialize if not done already. */
#define getslot(J, s)	(J->base[(s)] ? J->base[(s)] : sload(J, (int32_t)(s)))
```

**一句话:槽已经在 IR 里有引用就直接返回,没有就调 `sload` 加载并特化**。这个宏在 `lj_record_ins` 里满天飞——每次字节码的操作数是 BCMvar(栈槽变量)时,就调它。

`sload` 本体(`lj_record.c:200`):

```c
/* Specialize a slot to the runtime type. Note: slot can be negative! */
static TRef sload(jit_State *J, int32_t slot)
{
  IRType t = itype2irt(&J->L->base[slot]);
  TRef ref = emitir_raw(IRTG(IR_SLOAD, t), (int32_t)J->baseslot+slot,
                        IRSLOAD_TYPECHECK);
  if (irtype_ispri(t)) ref = TREF_PRI(t);  /* Canonicalize primitive refs. */
  J->base[slot] = ref;
  return ref;
}
```

逐行看,每一行都对应 §1 讲的道理:

1. **`IRType t = itype2irt(&J->L->base[slot])`**——这一行是"录制看真实类型"的核心。`J->L->base[slot]` 是解释器这次执行时这个槽里真实的 `TValue`,`itype2irt` 把它的类型标记(`LJ_TINT`/`LJ_TNUM`/`LJ_TSTR`/...)翻成对应的 IR 类型(`IRT_INT`/`IRT_NUM`/`IRT_STR`/...)。**这是真实发生的事实**,不是推断。如果这次这个槽是整数,`t` 就是 `IRT_INT`,后面这条 trace 就专门录成"假设这个槽永远是整数"。

2. **`emitir_raw(IRTG(IR_SLOAD, t), ...)`**——发射一条 `IR_SLOAD` 指令。注意是 `IRTG`(不是 `IRT`):`IRTG` 宏在类型上额外或一个 `IRT_GUARD` 位(`lj_ir.h:361` 的 `#define IRTG(o, t) (IRT((o), IRT_GUARD|(t)))`),`IRT_GUARD = 0x80`(`lj_ir.h:346`)。**这个位标记这条 IR 是 guard**——意思是它生成的机器码要带运行时检查,检查失败就退出。换句话说,`sload` 隐式地发射了一条 guard:**运行时检查这个槽的类型,如果不是 `t` 指定的那种,就 side exit**。

   这就是 §1.3 说的"假设都配 guard"在源码里的体现。`IR_SLOAD` 那条 IR 看起来只是"加载一个栈槽",但因为带了 `IRT_GUARD`,它同时是一条类型假设的 guard。

3. **`IRSLOAD_TYPECHECK`**(`lj_ir.h:235` 的 SLOAD 模式位 `0x04`)——传给 `IR_SLOAD` 的 op2,告诉后端"这次加载要做类型检查"。和 `IRT_GUARD` 配合,精确指定这条 SLOAD 是带类型 guard 的。

4. **`J->base[slot] = ref`**——把这条 `IR_SLOAD` 的引用存进 slot 映射表。以后这个槽再用 `getslot` 取,就直接返回这个 ref,不再加载第二次。

   **这是个关键设计**:同一个栈槽在一条 trace 里只加载一次。这保证了 SSA 的"单次定义"性质(P2-07 详讲)——栈槽可能被反复读写(循环里 `x = x + i` 每圈都改 x),但录制器只录一次"从输入栈加载 x",后续每次改 x 都是新的 IR 指令产生新的引用。

把这一节合起来:字节码的栈槽号(BCReg),经过 `getslot → sload`,变成一条 `IR_SLOAD` 指令的引用(IRRef)。这条 SLOAD 既"加载了栈槽",又"特化了类型"(来自真实执行),又"插了类型 guard"(IRT_GUARD)。**三个动作合一**,非常紧凑。这就是 trace 录制处理栈式字节码的精髓。

### 2.3 字节码操作数解码:那段统一的预处理

回到 `lj_record_ins` 的 switch 之前(`lj_record.c:2311-2345`):

```c
  /* Keep a copy of the runtime values of var/num/str operands. */
#define rav	(&ix.valv)
#define rbv	(&ix.tabv)
#define rcv	(&ix.keyv)

  lbase = J->L->base;
  ins = *pc;
  op = bc_op(ins);
  ra = bc_a(ins);
  ix.val = 0;
  switch (bcmode_a(op)) {
  case BCMvar:
    copyTV(J->L, rav, &lbase[ra]); ix.val = ra = getslot(J, ra); break;
  default: break;  /* Handled later. */
  }
  rb = bc_b(ins);
  rc = bc_c(ins);
  switch (bcmode_b(op)) {
  case BCMnone: rb = 0; rc = bc_d(ins); break;  /* Upgrade rc to 'rd'. */
  case BCMvar:
    copyTV(J->L, rbv, &lbase[rb]); ix.tab = rb = getslot(J, rb); break;
  default: break;  /* Handled later. */
  }
  switch (bcmode_c(op)) {
  case BCMvar:
    copyTV(J->L, rcv, &lbase[rc]); ix.key = rc = getslot(J, rc); break;
  case BCMpri: setpriV(rcv, ~rc); ix.key = rc = TREF_PRI(IRT_NIL+rc); break;
  case BCMnum: { cTValue *tv = proto_knumtv(J->pt, rc);
    copyTV(J->L, rcv, tv); ix.key = rc = tvisint(tv) ? lj_ir_kint(J, intV(tv)) :
    tv->u32.hi == LJ_KEYINDEX ? (lj_ir_kint(J, 0) | TREF_KEYINDEX) :
    lj_ir_knumint(J, numV(tv)); } break;
  case BCMstr: { GCstr *s = gco2str(proto_kgc(J->pt, ~(ptrdiff_t)rc));
    copyTV(J->L, rcv, s); ix.key = rc = lj_ir_kstr(J, s); } break;
  default: break;  /* Handled later. */
  }
```

(`lj_record.c:2311-2345`)

这段在 switch 之前,把字节码的 a/b/c 三个操作数**统一**翻译成 `TRef`(IR 引用),同时把它们的运行时真实值存进 `ix.valv/tabv/keyv`(为了后面 metamethod 查找等用)。理解这段要抓住 `bcmode_a/b/c`(字节码操作数模式)这套机制:

LuaJIT 的字节码是 32 位定长(`lj_bc.h`),分成 a/b/c/d 几个字段。每个操作码(BC op)的操作数模式是固定的——a 字段可能是"目标栈槽 BCMdst"、可能是"栈槽变量 BCMvar"、可能是"无 BCMnone";b/c 字段类似。`bcmode_a/b/c(op)` 查表返回这个字段是什么模式(`lj_bcdef.h` 里宏定义)。

这段预处理对每种模式做对应处理:

- **BCMvar(栈槽变量)**:调 `getslot`(§2.2),把这个栈槽翻成 `IR_SLOAD` 引用。同时 `copyTV` 把解释器栈里这个槽的真实 TValue 复制一份到 `ix.*v`,后面要查 metamethod、判类型时用。
- **BCMpri(原始值 nil/true/false)**:直接生成一个固定的原始引用 `TREF_PRI`(`lj_ir.h:534`,`TREF(REF_NIL-(t), (t))`)。这些值在 IR 里有固定的引用号(`REF_NIL`/`REF_FALSE`/`REF_TRUE`,`lj_ir.h:465-467`),不需要新建。
- **BCMnum(数字常量)**:从 prototype 的常量表(`proto_knumtv`)取这个数字常量,然后调 `lj_ir_kint` 或 `lj_ir_knumint`(§2.4 讲常量录制)把它注册进 IR 常量表。
- **BCMstr(字符串常量)**:从 prototype 的 GC 常量表取字符串,调 `lj_ir_kstr` 注册进 IR。

这段做完,字节码的三个操作数 a/b/c,凡是"栈槽"或"常量"的,都已经翻成了对应的 `TRef`(IR 引用)。剩下的"目标字段"(BCMdst,纯数字字面量,比如"结果存到第几号栈槽")保持原样——它们不需要转 IR,只是告诉录制器"结果该写到 `J->base[ra]`"。

这一段是整个录制流程的"前置统一翻译层"。它把栈式字节码的异质操作数(栈槽/常量/字面量)统一成 IR 世界里的同质引用(TRef)。后面的 switch 各个 case 拿到 `ra`/`rb`/`rc` 时,它们已经是干净的 TRef 了,case 主体只需要关心"这个操作码翻译成什么 IR"。

### 2.4 常量的录制:数字/字符串进 IR 常量表

§2.3 看到 `BCMnum` 和 `BCMstr` 操作数要调 `lj_ir_kint`/`lj_ir_kstr` 注册进 IR。这一节讲清楚"常量怎么进 IR"。

先看 IR 数组的布局。LuaJIT 的 IR 是一个连续数组 `J->cur.ir`(实际指向可增长缓冲区 `J->irbuf`,P2-05 §5.1 讲过),但它的下标有**偏移**——中间有个 `REF_BIAS = 0x8000`(`lj_ir.h:464`)。引用号从 `REF_BASE = REF_BIAS` 往上是普通指令区,从 `REF_BIAS` 往下是常量区:

```c
enum {
  REF_BIAS =	0x8000,
  REF_TRUE =	REF_BIAS-3,
  REF_FALSE =	REF_BIAS-2,
  REF_NIL =	REF_BIAS-1,	/* \--- Constants grow downwards. */
  REF_BASE =	REF_BIAS,	/* /--- IR grows upwards. */
  REF_FIRST =	REF_BIAS+1,
  REF_DROP =	0xffff
};
```

(`lj_ir.h:462-471`)

也就是说:**普通指令从中间往大下标长(`nins` 每发射一条指令加一),常量从中间往小下标长(`nk` 每加一个常量减一)**。两者背向生长,直到 `nins` 追上 `nk` 才需要扩容(`lj_ir_growtop`/`lj_ir_growbot`,`lj_ir.c:73/90`)。这是个非常紧凑的内存布局——同一段数组既装指令又装常量,互不干扰。

录制一个数字常量,比如 `BCMnum` 那个槽里存的是整数 5,调 `lj_ir_kint(J, 5)`(`lj_ir.c:200`):

```c
/* Intern int32_t constant. */
TRef LJ_FASTCALL lj_ir_kint(jit_State *J, int32_t k)
{
  IRIns *ir, *cir = J->cur.ir;
  IRRef ref;
  for (ref = J->chain[IR_KINT]; ref; ref = cir[ref].prev)
    if (cir[ref].i == k)
      goto found;
  ref = ir_nextk(J);
  ir = IR(ref);
  ir->i = k;
  ir->t.irt = IRT_INT;
  ir->o = IR_KINT;
  ir->prev = J->chain[IR_KINT];
  J->chain[IR_KINT] = (IRRef1)ref;
found:
  return TREF(ref, IRT_INT);
}
```

读这段,关键要理解一个叫 **chain(链)** 的机制,它是 LuaJIT IR 的核心数据结构,后面 CSE 等优化都靠它(`lj_jit.h:478` 的 `IRRef1 chain[IR__MAX]`)。

`J->chain[op]` 是一个数组,**每个操作码 op 对应一个链表头**,指向"目前为止最后一条这个 op 的 IR 指令的引用"。每条 IR 指令的 `prev` 字段(`lj_ir.h:563` 的 `IRRef1 prev`)指向前一条同 op 的指令。于是所有 `IR_KINT` 指令串成一条链,链头是 `J->chain[IR_KINT]`。

`lj_ir_kint` 做两件事:

1. **查重**:顺着 `J->chain[IR_KINT]` 这条链,看有没有哪条 `IR_KINT` 的值(`cir[ref].i`)已经是 `k`。有就 `goto found` 直接返回它的引用——**同一个常量只录一次**。
2. **新建**(没查到):`ir_nextk(J)` 在常量区(`nk` 往下)分配一个新槽,填上 `ir->i = k`、类型 `IRT_INT`、op `IR_KINT`,然后把这个新指令**插到 KINT 链头**(`J->chain[IR_KINT] = ref`、`ir->prev = 旧链头`)。

返回值 `TREF(ref, IRT_INT)`——把引用号 `ref` 和类型 `IRT_INT` 打包成一个 TRef(`lj_ir.h:502` 的 `#define TREF(ref, t) ((TRef)((ref) + ((t)<<24)))`,高 8 位放类型,低 16 位放引用)。

这种"按 op 串链 + 查重"的设计叫 **interning(常量去重)**。它保证:**同一条 trace 里,值相同的常量只出现一次**。这点很重要——后面优化 pass(CSE)做冗余消除时,两条引用如果指向同一个常量引用,就直接判等;如果同一个常量被录成多条 IR,优化就做不干净。所以常量在录制时就 intern 好,是后面所有优化的基础。

字符串常量走类似的路子:`lj_ir_kstr`(`lj_iropt.h:72` 的宏,展开成 `lj_ir_kgc`),走 `IR_KGC` 链查重(`lj_ir.c:266`)。浮点数走 `lj_ir_knum_u64` → `lj_ir_k64`(`lj_ir.c:219`),用 64 位比较(`ir_k64(&cir[ref])->u64 == u64`)。

> **★一个 2.1 ROLLING 的细节**:`lj_ir_knumint`(`lj_ir.c:252`)有个微妙处理——它先尝试把 `lua_Number` 转成 int32,能转就用 `lj_ir_kint`(省一个 IR 槽,因为 int32 常量只占一个 IRIns,而浮点常量占两个,见 `ir_nextk` vs `ir_nextk64`,`lj_ir.c:173/184`)。而且 `-0.0` **不被**当作整数(注释明确写 `-0 is NOT considered an int`,因为 `-0` 在 IEEE 754 里是特殊值,当整数 0 会丢信息)。这种边界处理在老资料里常被简化掉,2.1 ROLLING 是显式拒绝的。

### 2.5 guard 在录制时插入:那条带 IRT_GUARD 的 IR

§2.2 已经埋了 guard 这个伏笔——`sload` 发射的 `IR_SLOAD` 带 `IRT_GUARD`。这一节专门把 guard 怎么"自然出现在录制流里"讲透,因为这是"假设 + 运行时检查"在源码层的精确落点。

guard 在 LuaJIT 里不是一个独立的"插 guard"动作。它的本质是:**某些 IR 指令被标记成"guard",意味着它生成的机器码要带运行时检查,检查失败就退出**。这个标记就是类型字段里的 `IRT_GUARD` 位(`lj_ir.h:346`,`IRT_GUARD = 0x80`)。

哪些 IR 该是 guard?由两个机制决定:

**第一,录制器主动发 guard**。看 `sload`——它发 `IRTG(IR_SLOAD, t)`,这里的 `IRTG` 宏强制带 `IRT_GUARD`。这意味着"从栈槽加载一个值"默认带类型检查:加载时假设槽是 `t` 类型,运行时检查不是就退出。这是 trace 的"输入假设"——trace 假设入口栈的每个槽都是这次录到的类型。

类似地,看 `BC_ISLT`(小于比较)的录制(`lj_record.c:2351` 之后):

```c
  case BC_ISLT: case BC_ISGE: case BC_ISLE: case BC_ISGT:
    ...
    emitir(IRTG(irop, ta), ra, rc);
    rec_comp_fixup(J, J->pc, ((int)op ^ irop) & 1);
    break;
```

`emitir(IRTG(irop, ta), ra, rc)` 发射一条比较 IR(比如 `IR_LT`),而且用 `IRTG`,所以是 guard。**这就是"分支用 guard 兜底"在源码里的体现**——一个 `if a < b` 在录制时,因为只录实际走的那条路,比较本身被翻成一条 guard IR:`IR_LT`,机器码生成时会带上"如果 a < b 不成立就退出"。退出后回解释器走另一条没录的路。

**第二,IR 操作码的 mode 里自带 guard 性质**。看 `lj_ir.h:284` 的 IR mode 位定义:

```c
/* Mode bits: Commutative, {Normal/Ref, Alloc, Load, Store}, Non-weak guard. */
#define IRM_C			0x10

#define IRM_N			0x00
#define IRM_R			IRM_N
#define IRM_A			0x20
#define IRM_L			0x40
#define IRM_S			0x60

#define IRM_W			0x80
```

和 `lj_ir.h:602` 的判定:

```c
  return (((ir->t.irt | ~IRT_GUARD) & lj_ir_mode[ir->o]) >= IRM_S);
```

这个表达式判定一条 IR 是不是"需要 guard"。逻辑是:IR 操作码本身有个 mode(在 `lj_ir_mode[]` 表里),这个 mode 可能含 `IRM_W` 位(=0x80,正好等于 `IRT_GUARD`)。如果操作码 mode 含 W,而且这条 IR 的类型字段也含 `IRT_GUARD`,那么它**必须**生成 guard 机器码。

也就是说:**有些 IR 操作码天生是 guard**——比如 `IR_EQ`、`IR_LT`、`IR_ULT` 这些比较操作(因为它们的结果只有"成立/不成立"两种,录的时候录了"成立"这条路,机器码必须检查"不成立就退")。它们的 mode 在 `lj_ir_mode[]` 表里写好了带 W,后端 `lj_asm` 见到 W+IRT_GUARD 就生成带跳转的机器码。

**两条机制合起来,guard 是"自然"出现在录制流里的**——录制器在翻译字节码时,凡是遇到"假设"或"分支"的地方,就发 `IRTG`(`IRT` + `IRT_GUARD`)或发那些 mode 自带 W 的比较 IR。这些 guard IR 在 IR 数组里看起来和普通 IR 没区别(就是 op + 类型 + 操作数),只是类型里多了 `IRT_GUARD` 位、op 的 mode 带了 W。**没有单独的"插 guard"步骤,guard 就是录制流的一部分**。这是 LuaJIT 设计的一个优雅之处。

### 2.6 guard 和 snapshot 的耦合:lj_snap_add

guard 还要配 snapshot(P0-01 §8)——guard 触发退出时要靠 snapshot 恢复解释器状态。那 snapshot 是在录制的什么时候加的?

看 §2.1 引的这段(`lj_record.c:2283`):

```c

    J->needsnap = 0;
    if (J->pt && bc_op(*J->pc) < BC_FUNCF) lj_snap_purge(J);
    lj_snap_add(J);
    J->mergesnap = 1;

```

`J->needsnap` 是个标志(`lj_jit.h:432`),表示"下一条字节码录之前必须先加一个 snapshot"。它在什么时候被置位?看 `lj_ir_call`(`lj_ir.c:145`):

```c

    J->needsnap = 1;  /* Need snapshot after call with side effect. */
```

有副作用的调用(`IR_CALLS`)之后,置 `needsnap`。还有各种 store 操作、可能退出的点,都会置 needsnap。

`lj_snap_add`(`lj_snap.c:181`)做的事:把当前栈状态(`J->base[]` 里每个槽的 TRef)压缩成一组 `SnapEntry`,追加到 `snapmap`,`snap` 数组加一项。它的合并逻辑(`lj_snap.c:186`):

```c
  /* Merge if no ins. inbetween or if requested and no guard inbetween. */
  if ((nsnap > 0 && J->cur.snap[nsnap-1].ref == J->cur.nins) ||
      (J->mergesnap && !irt_isguard(J->guardemit))) {
    ...merge...
```

关键看 `!irt_isguard(J->guardemit)`——`J->guardemit`(`lj_jit.h:433`)是个累积字段,记录"自上次 snapshot 以来,有没有发射过 guard"。如果没发射过 guard,而且允许合并(`mergesnap`),那连续的两个 snapshot 就合成一个(因为中间没有可能退出的点,前一个 snapshot 的状态对后一个也有效)。

**这个机制保证:snapshot 只在需要的地方加(每个 guard 之后),没有冗余**。所以 IR 数组里 snapshot 的分布和 guard 的分布耦合——guard 密的地方 snapshot 也密。录制器不需要显式管"哪里加 snapshot",通过 `needsnap` + `guardemit` 这两个标志自然驱动。

### 2.7 录制的终止:循环回边与 abort

前面 §2.1-§2.6 讲的都是"录一条字节码"。录制还要有个**终止条件**——什么时候算录完了。LuaJIT 有几种终止方式,对应几种 trace 结局。

**最常见:循环回边 → 录制结束,trace 回自己**。

看 `lj_record_ins` 的 `case BC_FORL`(`lj_record.c:2656`):

`lj_record_stop`(`lj_record.c:299`)做收尾:

```c
static void rec_loop_jit(jit_State *J, TraceNo lnk, LoopEvent ev)

  if (J->parent == 0 && J->exitno == 0) {  /* Root trace hit an inner loop. */
    /* Better let the inner loop spawn a side trace back here. */
    lj_trace_err(J, LJ_TRERR_LINNER);

    J->instunroll = 0;  /* Cannot continue across a compiled loop op. */
    if (J->pc == J->startpc && J->framedepth + J->retdepth == 0)
      lj_record_stop(J, LJ_TRLINK_LOOP, J->cur.traceno);  /* Form extra loop. */
    else
      lj_record_stop(J, LJ_TRLINK_ROOT, lnk);  /* Link to the loop. */
  }
```

如果是 root trace 撞到一个已编译的内层循环,直接 abort(`LJ_TRERR_LINNER`),让那条内层循环自己长出 side trace 来接;如果是 side trace 撞到,就 `LJ_TRLINK_ROOT` 链过去。

**第三种终止:遇到无法录制的操作 → abort**。

`case BC_UCLO`、`BC_FNEW` 等(`lj_record.c:2727`):

```c
  case BC_UCLO:
  case BC_FNEW:
    setintV(&J->errinfo, (int32_t)op);
    lj_trace_err_info(J, LJ_TRERR_NYIBC);
    break;
```

`LJ_TRERR_NYIBC` = "not yet implemented bytecode"(尚不支持的字节码)。`UCLO`(关闭 upvalue)和 `FNEW`(创建函数)在 root trace 里目前不录,直接 abort。abort 走 P2-05 §6 讲的 ERR 流程——失败发生在装上机器码之前,所以无害,解释器继续按它原本的方式执行,程序结果和纯解释器一致。

**第四种:函数调用 → 进入子帧,继续录(stitch)**。

录到 `BC_CALL`(`lj_record.c:2614`),调 `lj_record_call`(`lj_record.c:863`)——它不结束 trace,而是**把栈帧往下推一层**(`J->base += func+1+LJ_FR2`),`framedepth++`,继续在子函数里录。这叫 trace 跨函数调用——trace 不止于一个函数,它可以顺着调用链一直录下去。但被调函数必须能录;如果被调函数没被 JIT(`PROTO_NOJIT`)或太复杂,会走 stitch(P2-05 §4.3)或 abort。

四种终止合起来,覆盖了录制会遇到的所有收尾情况。最常见的还是第一种——循环回边收尾,trace 回自己,这正是 root trace 最典型的形态。

## §3 几个典型 rec_* 函数:字节码怎么翻成 IR

§2 讲了录制的主干。这一节挑几个有代表性的字节码,看它们在 `lj_record_ins` 的 switch 里**具体**怎么被翻成 IR。读者看完这一节,应该能举一反三,自己读懂别的 case。

### 3.1 算术:rec_mm_arith 与 lj_opt_narrow_arith

看 `BC_ADDVN`(常量加变量)、`BC_ADDVV`(变量加变量)的录制(`lj_record.c:2482`):

```c
  case BC_ADDVN: case BC_SUBVN: case BC_MULVN: case BC_DIVVN:
  case BC_ADDVV: case BC_SUBVV: case BC_MULVV: case BC_DIVVV: {
    MMS mm = bcmode_mm(op);
    if (tref_isnumber_str(rb) && tref_isnumber_str(rc))
      rc = lj_opt_narrow_arith(J, rb, rc, rbv, rcv,
                               (int)mm - (int)MM_add + (int)IR_ADD);
    else
      rc = rec_mm_arith(J, &ix, mm);
    break;
    }
```

这是个非常清晰的二分:

- **如果两个操作数都是 number 或 string**(快路径),调 `lj_opt_narrow_arith`——这是**类型窄化**的核心函数(P2-08 专章),它根据这次真实执行的类型(rbv/rcv 里的 TValue),决定生成 `IRT_INT` 的整数加法 IR 还是 `IRT_NUM` 的浮点加法 IR,还可能插一些 guard(比如"这个浮点真的是整数"的检查)。最终发射的 IR 操作码由 `(int)mm - (int)MM_add + (int)IR_ADD` 算出——把 metamethod 枚举(`MM_add`/`MM_sub`/...)映射到 IR 操作码(`IR_ADD`/`IR_SUB`/...)。所以 `BC_ADDVN` 录成 `IR_ADD`,`BC_SUBVN` 录成 `IR_SUB`,以此类推。
  
- **否则**(有 table/udata 等非数字操作数),走 `rec_mm_arith`(`lj_record.c:1160`)——查 metamethod(`__add`/`__sub`/...),录成对 metamethod 的调用。这就是 Lua 的运算符重载:对没有原生的 number 行为的对象,调它的 `__add` 元方法。

注意这里 `lj_opt_narrow_arith` 名字带 "opt",但它在录制期(RECORD 阶段)就被调了——它是**录制时决定类型**的关键,不是事后优化。这点容易混:narrow 看起来像优化,其实是录制的一部分,因为它要做"这次真实类型决定 IR 类型"这件事,必须在录制期完成(P2-08 详讲)。

`rec_mm_arith`(`lj_record.c:1160`)本身的结构也值得看,因为它展示了"metamethod 怎么录成调用":

```c
static TRef rec_mm_arith(jit_State *J, RecordIndex *ix, MMS mm)
{
  /* Set up metamethod call first to save ix->tab and ix->tabv. */
  BCReg func = rec_mm_prep(J, mm == MM_concat ? lj_cont_cat : lj_cont_ra);
  TRef *base = J->base + func;
  TValue *basev = J->L->base + func;
  base[1+LJ_FR2] = ix->tab; base[2+LJ_FR2] = ix->key;
  ...
  if (!lj_record_mm_lookup(J, ix, mm)) {  /* Lookup mm on 1st operand. */
    if (mm != MM_unm) {
      ix->tab = ix->key;
      ...
      if (lj_record_mm_lookup(J, ix, mm))  /* Lookup mm on 2nd operand. */
        goto ok;
    }
    lj_trace_err(J, LJ_TRERR_NOMM);
  }
ok:
  base[0] = ix->mobj;
  ...
  lj_record_call(J, func, 2);
  return 0;  /* No result yet. */
}
```

它干的事:在栈上准备好一个"调 metamethod"的调用帧(`base[0]` 放 metamethod 函数,`base[1]`/`base[2]` 放两个操作数),然后查 metamethod(先查第一个操作数的元表,没有再查第二个),找到就调 `lj_record_call`——这等于录了一次函数调用(§3.3)。所以 `a + b` 当 a 是 table 时,录出来的是"调 a 的 `__add` 方法"的 IR 序列,而不是整数加法。

### 3.2 表访问:lj_record_idx

`BC_TGETS`(字符串 key 取表)、`BC_TGETB`(整数 key 取表)、`BC_TGETV`(变量 key 取表)都进同一个录制路径(`lj_record.c:2568`):

```c
  case BC_TGETB: case BC_TSETB:
    setintV(&ix.keyv, (int32_t)rc);
    ix.key = lj_ir_kint(J, (int32_t)rc);
    /* fallthrough */
  case BC_TGETV: case BC_TGETS: case BC_TSETV: case BC_TSETS:
    ix.idxchain = LJ_MAX_IDXCHAIN;
    rc = lj_record_idx(J, &ix);
    break;
```

(注意 `TGETB` 把整数 key 先存进 `ix.keyv` 并录成 `IR_KINT` 常量,然后 fallthrough 到通用路径。)

核心是 `lj_record_idx`(`lj_record.c:1530`)。这个函数长(近 200 行),因为它要处理表查找的所有细节:数组部分 vs 哈希部分、key 类型、metamethod(`__index`/`__newindex`)、metatable 链。我们只看它录出来的 IR 长什么样。

先看开头(`lj_record.c:1591`):

```c
  /* Record the key lookup. */
  xref = rec_idx_key(J, ix, &rbref, &rbguard);
  xrefop = IR(tref_ref(xref))->o;
  loadop = xrefop == IR_AREF ? IR_ALOAD : IR_HLOAD;
```

`rec_idx_key`(`lj_record.c:1437`,在 `rec_idx_bump` 之后)做"找 key 在表的哪个位置"——对数组 key,生成 `IR_AREF`(数组引用,指向数组某个槽);对哈希 key,生成 `IR_HREFK`(哈希引用)。这个 xref 是个"指向表内某个槽位的 IR 引用",它本身带 guard(检查 key 真的在那个位置、表结构没变)。

然后(`lj_record.c:1597`,indexed load 分支):

```c
  if (ix->val == 0) {  /* Indexed load */
    IRType t = itype2irt(oldv);
    TRef res;
    if (oldv == niltvg(J2G(J))) {
      emitir(IRTG(IR_EQ, IRT_PGC), xref, lj_ir_kkptr(J, niltvg(J2G(J))));
      res = TREF_NIL;
    } else {
      res = emitir(IRTG(loadop, t), xref, 0);
    }
```

`oldv` 是这次真实执行时这个槽里的值(`ix->oldv`,`rec_idx_key` 填的)。`itype2irt(oldv)` 把它的真实类型翻成 IR 类型。然后:

- 如果槽里是 nil(指向全局的 niltv),发一条 `IR_EQ` guard("检查这个槽真的还是 nil")——因为表可能被别处改,这个检查保证 trace 装上后运行时,如果槽不再是 nil 就退出。
- 否则,发 `IRTG(loadop, t)`——`loadop` 是 `IR_ALOAD`(数组加载)或 `IR_HLOAD`(哈希加载),带 guard,从 xref 指向的位置加载一个类型为 `t` 的值。

所以一个 `t.k`(TGETS)录出来大概是:

```
IR_HREFK   = hash lookup(t, "k")   (带 guard:检查 key 在哈希表的位置)
IR_HLOAD   = load(HREFK 结果)       (带 guard:类型是这次录到的 t)
```

两条 IR,两条 guard。两条 guard 各自兜底——位置变了 HREFK 退出,值类型变了 HLOAD 退出。**这是"动态语言的表访问怎么安全录成机器码"的完整答案**:每一步假设都配 guard。

store(TSETS/TSETB)走 `else` 分支(`lj_record.c:1614`),类似但发的是 `IR_ASTORE`/`IR_HSTORE`,把值写回 xref 指向的位置。store 还有额外的 GC write barrier 处理(因为写一个 GC 对象进表要通知 GC)。

### 3.3 函数调用:lj_record_call 与 rec_call_setup

`BC_CALL` 录制(`lj_record.c:2614`)调 `lj_record_call`(`lj_record.c:863`):

```c
void lj_record_call(jit_State *J, BCReg func, ptrdiff_t nargs)
{
  rec_call_setup(J, func, nargs);
  /* Bump frame. */
  J->framedepth++;
  J->base += func+1+LJ_FR2;
  J->baseslot += func+1+LJ_FR2;
  if (J->baseslot + J->maxslot >= LJ_MAX_JSLOTS)
    lj_trace_err(J, LJ_TRERR_STACKOV);
}
```

`rec_call_setup`(`lj_record.c:830`)干关键的活:先 `getslot` 确保函数和所有参数都有 IR 引用(§2.2),如果函数本身不是函数类型就查 `__call` metamethod,然后 `rec_call_specialize` 特化——它发一条 guard(`lj_record.c:825`):

```c
  /* Otherwise specialize to the function (closure) value itself. */
  kfunc = lj_ir_kfunc(J, fn);
  emitir(IRTG(IR_EQ, IRT_FUNC), tr, kfunc);
  return kfunc;
```

`emitir(IRTG(IR_EQ, IRT_FUNC), tr, kfunc)`——发一条 `IR_EQ` guard:**运行时检查这个函数槽真的还是这个具体的函数对象 `fn`**。这是个非常细的特化——trace 不光假设"这是函数",还假设"是这个特定的函数"。为什么?因为不同函数有不同的字节码、不同的 prototype,trace 接下来要录这个函数的字节码,必须确定就是这一个函数,不能是别的同类型函数。如果运行时这个槽换成了别的函数(比如闭包变量改了),guard 触发,退出。

`lj_record_call` 的尾巴做"推栈帧":`J->base += func+1+LJ_FR2`(`lj_jit.h:425` 的 base 是指针,推进去相当于进入被调函数的栈帧),`framedepth++`。从此录制器在子函数的栈帧里继续录——它录的每条字节码,都是子函数的字节码。这就是 trace 跨函数调用:trace 不止于 caller,而是顺着真实调用链一直录进 callee。

返回(`BC_RET`)走 `lj_record_ret`(`lj_record.c:922`),它把栈帧收回来:`J->base -= ...`,`framedepth--`,把返回值搬到 caller 的栈槽。如果 `framedepth` 减到 0(已经返回到 trace 的根帧),还要判定 trace 该不该结束(P2-05 §2 的 down-rec / return-to-interp)。

## §4 录制的初始化:lj_record_setup

录制开始之前,`trace_start`(P2-05 §5.1)会调 `lj_record_setup`(`lj_record.c:2823`)初始化录制环境。这个函数是录制"从无到有"的第一步,理解它能把 §2-§3 的零散机制串起来。

看主体(`lj_record.c:2823`):

```c
void lj_record_setup(jit_State *J)
{
  uint32_t i;

  /* Initialize state related to current trace. */
  memset(J->slot, 0, sizeof(J->slot));
  memset(J->chain, 0, sizeof(J->chain));
#ifdef LUAJIT_ENABLE_TABLE_BUMP
  memset(J->rbchash, 0, sizeof(J->rbchash));
#endif
  memset(J->bpropcache, 0, sizeof(J->bpropcache));
  J->scev.idx = REF_NIL;
  setmref(J->scev.pc, NULL);

  J->baseslot = 1+LJ_FR2;  /* Invoking function is at base[-1-LJ_FR2]. */
  J->base = J->slot + J->baseslot;
  J->maxslot = 0;
  J->framedepth = 0;
  J->retdepth = 0;

  J->instunroll = J->param[JIT_P_instunroll];
  J->loopunroll = J->param[JIT_P_loopunroll];
  J->tailcalled = 0;
  J->loopref = 0;

  J->bc_min = NULL;  /* Means no limit. */
  J->bc_extent = ~(MSize)0;

  /* Emit instructions for fixed references. Also triggers initial IR alloc. */
  emitir_raw(IRT(IR_BASE, IRT_PGC), J->parent, J->exitno);
  for (i = 0; i <= 2; i++) {
    IRIns *ir = IR(REF_NIL-i);
    ir->i = 0;
    ir->t.irt = (uint8_t)(IRT_NIL+i);
    ir->o = IR_KPRI;
    ir->prev = 0;
  }
  J->cur.nk = REF_TRUE;

  J->startpc = J->pc;
  setmref(J->cur.startpc, J->pc);
```

逐段看,每一件都对应 §2-§3 讲过的机制:

1. **`memset(J->slot, 0, ...)`**——清空栈槽映射表(§2.2)。新 trace 开始,所有栈槽都没加载过。
2. **`memset(J->chain, 0, ...)`**——清空所有 IR 操作码的链头(`lj_jit.h:478`)。新 trace 没有任何 IR,所有链都是空的。这个清空很关键——常量 interning(§2.4)和后面的 CSE 都靠这些链,不清空会串到上一条 trace 的 IR。
3. **`J->base = J->slot + J->baseslot`**——设栈帧基址。`baseslot = 1+LJ_FR2`(`LJ_FR2` 在 GC64/FR2 模式下是 1,否则 0),意思是"调用者在 `base[-1-LJ_FR2]`"(栈帧布局,frame pointer 在那)。这样 `J->base[0]` 对应被调函数的第一个本地变量。
4. **`emitir_raw(IRT(IR_BASE, IRT_PGC), J->parent, J->exitno)`**——发射 trace 的第一条 IR:`IR_BASE`。这是个特殊的 IR,代表"trace 的基址指针"(指向解释器栈的当前位置)。后面所有 `IR_SLOAD` 都用它作为基址去加载栈槽。`emitir_raw` 是不经过优化的发射(`lj_record.c:45` 的宏,直接调 `lj_ir_emit`),因为 `IR_BASE` 是固定指令,不需要折叠。
5. **填三个固定引用 `REF_NIL`/`REF_FALSE`/`REF_TRUE`**——这三个引用号(`lj_ir.h:465-467`)是预留给 nil/false/true 的,在 trace 开头就填好,后面所有原始值引用它们(`TREF_PRI`,`lj_ir.h:534`)。`IR_KPRI` 是"原始常量"操作码。
6. **`J->cur.nk = REF_TRUE`**——常量区顶部设到 `REF_TRUE`(= REF_BIAS - 3)。意味着常量区从这里往下长(`ir_nextk` 每次 `--nk`)。

然后处理 side trace vs root trace 的分歧(`lj_record.c:2864`):

```c
  if (J->parent) {  /* Side trace. */
    GCtrace *T = traceref(J, J->parent);
    ...
    lj_snap_replay(J, T);
    ...
  } else {  /* Root trace. */
    J->cur.root = 0;
    J->cur.startins = *J->pc;
    J->pc = rec_setup_root(J);
    ...
    lj_snap_add(J);
    ...
  }
```

**side trace**(从某个退出点录)调 `lj_snap_replay`——把父 trace 的 snapshot "重放"进当前 trace 的栈槽映射,这样当前 trace 的入口栈状态就和父 trace 退出时一致。**root trace**(从循环回边录)调 `rec_setup_root`(`lj_record.c:2758`)——根据起始字节码类型(`BC_FORL`/`BC_ITERL`/`BC_LOOP`/...)算出循环的字节码范围(`bc_min`/`bc_extent`,§2.1 的 root trace 范围检查用),然后 `lj_snap_add` 加 snapshot #0(trace 入口状态)。

`rec_setup_root` 的细节(`lj_record.c:2758`)对理解 root trace 录制的"循环范围"很关键:

```c
  case BC_FORL:
    J->bc_extent = (MSize)(-bc_j(ins))*sizeof(BCIns);
    pc += 1+bc_j(ins);
    J->bc_min = pc;
    break;
```

`bc_j(ins)` 是 `BC_FORL` 字节码里的跳转偏移(负数,跳回循环头)。`J->bc_min` 设成"循环头字节码的位置",`J->bc_extent` 设成"循环体的字节码长度"。这俩字段后面 §2.1 的 `lj_record.c:2298` 那个检查用——`if (J->framedepth == 0 && (pc - J->bc_min) >= J->bc_extent) lj_trace_err(J, LJ_TRERR_LLEAVE)`。意思是:**root trace 录制时,如果当前 pc 跑出了循环范围,就 abort**。这保证 root trace 真的是"一个循环体",不是乱跑出来的路径。

理解了 `lj_record_setup`,录制环境的全貌就清楚了:`slot[]` 映射表(空)、`chain[]` 链头(空)、栈帧基址(入口)、`IR_BASE` 第一条 IR、固定原始引用、循环范围。一切就绪,回到 `trace_state` 的 `case LJ_TRACE_RECORD`,开始录第一条字节码。

## §5 为什么 sound:双轨执行与 guard 兜底

讲完了机制,这一节专门论证"为什么这样录制不会算错"。这是 P0-01 §9 "解释器保底,机器码加速"在录制层面的兑现。

录制 sound 的根基有三层。

### 5.1 第一层:忠实记录真实路径与真实类型

录制的 IR 不是凭空构造的,是**解释器这次真实执行的忠实镜像**。具体怎么保证?

- **`J->pc` 是解释器真正要执行的 pc**(`lj_trace_ins` 设的)。录制器处理的字节码,就是解释器正在执行的那条。
- **`lbase = J->L->base` 是解释器真实的栈**(`lj_record.c:2316`)。录制器查类型(`itype2irt`),查的是这次执行的真实 TValue。
- **`IR_SLOAD` 的类型来自这次真实执行**(`sload` 的 `itype2irt(&J->L->base[slot])`)。这次槽里是整数,录的就是整数 IR。
- **算术 IR 的操作码由真实类型决定**(`lj_opt_narrow_arith` 根据操作数的真实类型选 INT 还是 NUM)。
- **表访问的值类型来自这次真实加载**(`lj_record_idx` 的 `itype2irt(oldv)`)。

合起来:**录出来的 IR 序列,和解释器这次执行的状态序列,是一一对应的**。如果机器码严格按 IR 执行(假设全成立),结果和解释器这次执行完全一样。这是 sound 的第一道——"忠实"。

但忠实只对"这次执行"成立。万一下次执行类型变了呢?第二层。

### 5.2 第二层:每个假设都配 guard

录制的 IR 里,所有"假设"都配了 guard:

- **栈槽类型假设**:`IR_SLOAD` 带 `IRT_GUARD`(`sload` 的 `IRTG`),运行时检查槽类型没变。
- **比较/分支**:`IR_LT`/`IR_LE`/`IR_EQ` 等比较 IR,mode 自带 `IRM_W`(`lj_ir.h:602`),运行时检查条件成立。
- **函数特化**:`IR_EQ` + 函数引用(`rec_call_specialize`),运行时检查函数对象没换。
- **表结构**:`IR_HREFK` 带 guard,运行时检查 key 还在哈希表的同一位置(表没被 rehash)。
- **值类型**:`IR_HLOAD`/`IR_ALOAD` 带 guard,运行时检查加载的值类型没变。

每一个乐观假设,都有对应的 guard 兜底。假设再多再大胆,只要都配了 guard,就没有一个假设能"偷偷"失效——任何一个破了,对应的 guard 一定触发。guard 触发就 side exit(P0-01 §8,§5-16 详讲):照 snapshot 恢复解释器状态,退回解释器继续。

所以机器码执行的结果只有两种:**要么和解释器完全一样(所有 guard 不触发,忠实执行 IR);要么退回解释器(某个 guard 触发,解释器按真实类型算)**。它永远不会比解释器更错。这是 sound 的第二道——"兜底"。

### 5.3 第三层:guard 不会漏

第二层要求"每个假设都配了 guard"。怎么保证录制器没漏掉某个假设?这要靠 IR 操作码 mode 的系统性。

LuaJIT 的每个 IR 操作码,在 `lj_ir_mode[]` 表里都标好了它的性质——是否 commutative(`IRM_C`)、是否 alloc/load/store(`IRM_A`/`IRM_L`/`IRM_S`)、是否必须 guard(`IRM_W`)。这些性质是**操作码本身的契约**,不是录制器每次手决定的。

具体看 `lj_ir.h:602`:

```c
  return (((ir->t.irt | ~IRT_GUARD) & lj_ir_mode[ir->o]) >= IRM_S);
```

这个表达式判定"一条 IR 是否需要 guard 机器码"。逻辑:如果操作码 mode 含 W(`IRM_W` = 0x80),而且这条 IR 的类型字段也含 `IRT_GUARD`(也是 0x80),那么 `(t.irt | ~IRT_GUARD) & mode` 的最高位会保留,结果 ≥ `IRM_S`(0x60),判为需要 guard。

意思是:**所有 mode 带 W 的操作码,只要类型字段被设了 IRT_GUARD,就一定生成 guard 机器码**。后端 `lj_asm` 不加选择地按这个判定走——它见到需要 guard 就生成检查指令,不存在"某条该 guard 的 IR 被漏掉没生成检查"的可能。

而录制器发 IR 时,凡是涉及假设的,要么用 `IRTG`(强制 IRT_GUARD),要么发的是那些 mode 自带 W 的比较 IR。这两种情况下,guard 标记都会到位。所以"假设配 guard"是**录制器发射约定 + IR 操作码 mode 契约 + 后端按 mode 生成**三层共同保证的,不是靠录制器每次记得插——所以漏不掉。

这三层合起来,就是录制 sound 的完整论证:**忠实记录(对应解释器这次执行)+ 每个假设配 guard(运行时检查)+ guard 不漏(mode 契约驱动)**。因此录出的 IR,语义和解释器一致——机器码要么和解释器一样,要么退回解释器。这就是"动态执行被安全变成机器码"在录制这一步的精确兑现。

## §6 ★对照:官方 Lua 与 JVM/V8

把 LuaJIT 的录制和两个对象对照,取舍会更清晰。

### 6.1 对照一:官方 Lua(切"有没有录制")

官方 Lua 是纯解释器,**根本没有"录制"这个概念**。它的字节码(`lj_parse.c` 那一步前端编译,官方 Lua 也有等价的)编译完之后,就直接进解释器执行,永远不翻译成 IR,永远不编译成机器码。

所以对照官方 Lua,LuaJIT 多出来的,是 §1-§4 这一整套:**栈槽映射表**(`J->slot`)、**IR 发射与常量 interning**(`lj_ir_kint` 等)、**guard 标记机制**(`IRT_GUARD`/`IRM_W`)、**snapshot 与 guard 的耦合**(`needsnap`/`guardemit`)、**录制的终止判断**(`rec_loop_interp`/`lj_record_stop`)、**录制环境的初始化**(`lj_record_setup`)。

这整套机制的体积——`lj_record.c` 2940 行 + `lj_ir.c` 487 行 + `lj_iropt.h`/`lj_ir.h` 里的大量定义——就是 LuaJIT 相对官方 Lua"快几十倍"的代价之一。官方 Lua 不需要任何这些东西,它的字节码对象一旦编译完,执行就是简单的"取指-译码-执行"循环,没有任何 IR、guard、snapshot 概念。

一个具体的对照点:**类型处理**。官方 Lua 解释器每条字节码执行时,都**当场**判断类型——`ADDVV` 执行时,取两个操作数的 TValue,看 itype,选整数加法或浮点加法或字符串拼接或报错。每次都判,每次都对,但每次都花这个判断开销。LuaJIT 录制时,**一次**把类型看清楚(录成 `IRT_INT` 的 `IR_ADD`),以后机器码就按这个类型走,不再每次判——这就是"省掉反复判断"的来源。代价是万一类型变了要 guard 兜底退出。

### 6.2 对照二:JVM/V8(切"静态构建 IR vs 运行时录制")

JVM 和 V8 也有 JIT,它们也把代码翻成 IR(HotSpot 的 Sea-of-Nodes、V8 的 TurboFan 节点图)。但它们构建 IR 的方式和 LuaJIT 根本不同——这是这一章最有讲头的对照点。

**JVM/V8:从字节码/AST 静态构建 IR**。

HotSpot 的 C1/C2 编译器,接到"编译这个方法"的请求后,做的事是:**读这个方法的字节码,从头到尾构建 IR**。构建时,它把方法里**所有**分支都纳入 IR——if/else 两个分支都建,循环的所有可能路径都建。IR 是一个图,节点是操作,边是控制流和数据流。

那类型呢?JVM/V8 不靠"这次执行的真实类型",而是靠**类型 profile(类型画像)**——解释器执行时,顺便记录"这个调用点见过的参数类型分布"(比如"99% 是 int,1% 是 string")。编译时,基于 profile 推断"这里大概率是 int",生成偏向 int 的机器码,但保留兜底(string 那条路通过 deoptimization 处理)。

**LuaJIT:运行时录制**。

LuaJIT 完全不这么做。它不读字节码构建 IR 图,而是**让解释器真跑一遍热点路径,边跑边记**。录出来的 IR 不是"方法里所有分支的图",而是"这次实际走过的那条线性路径的链"——if/else 没走的那条路,根本不进 IR。类型也不是基于 profile 推断,而是**这次执行的真实类型**——录到啥类型就是啥类型,配上 guard 保证以后类型变了能退出。

这个根本区别带来几个具体差异:

**第一,IR 形状不同。** JVM/V8 的 IR 是**图**(有分支、有汇合),因为整个方法的所有路径都在里面;LuaJIT 的 IR 是**线性链**(就是一条路径,中间没有分叉,分叉被 guard 代替了)。线性的好处是简单——寄存器分配、优化、代码生成都比图简单(整个 P3/P4 都因此简化);代价是一条 trace 覆盖不了一个方法的所有情况,要靠 side trace 树补齐(P5-18)。

**第二,类型依据不同。** JVM/V8 用 profile(统计分布),允许"虽然 99% 是 int,但偶尔 string,机器码里保留 string 的处理路径";LuaJIT 用"这次真实类型",假设更激进——这次是 int 就假设永远是 int,string 那条路完全不录,只在 guard 触发时退回解释器。LuaJIT 的假设更乐观、机器码更精简(没冗余分支),但退出更频繁(类型一变就退)。

**第三,处理"没见过的分支"不同。** JVM/V8 因为构建了所有分支的 IR,即使编译时没见过的分支,机器码里也有处理(虽然可能走慢路径)。LuaJIT 录制时没走的分支,IR 里完全没有,只能靠 guard + side exit + 后续 side trace 补——这是个"渐进覆盖"的过程(P2-05 §11.2 对照过)。

**第四,录制的"忠实"代价。** LuaJIT 录制必须让程序真跑一遍——这"一遍"是真实执行,有副作用(改了真实状态)。所以录制不能"想录几遍录几遍",它就是程序自然执行的一圈。如果这一圈恰好走了不典型的路径(比如第一次录就走到一个冷分支),录出的 trace 可能质量不高。JVM/V8 的 profile 是多次执行的累积统计,更稳健。这是 LuaJIT "录制"路线的一个固有弱点——它依赖"这次执行"的代表性,而 JVM/V8 依赖"多次执行的统计"。

下面这个表把两种构建 IR 的方式对照清楚:

| 维度 | LuaJIT(运行时录制) | JVM/V8(静态构建) |
|---|---|---|
| IR 来源 | 解释器真实执行一遍,边跑边记 | 读字节码,构建控制流图 |
| IR 形状 | 线性链(一条路径) | 图(所有分支) |
| 类型依据 | 这次执行的真实类型 | 类型 profile(统计分布) |
| 分支处理 | 没走的分支不录,guard 兜底 | 所有分支都进 IR |
| 假设激进程度 | 高(这次啥样假设啥样) | 中(基于 profile,留兜底) |
| 覆盖完整路径 | 渐进(side trace 树补齐) | 一次编译覆盖整个方法 |
| 编译速度 | 快(IR 短,线性) | 慢(IR 大,图优化复杂) |

trace JIT 的录制路线,换来的是更快的编译、更精简的机器码、对热点更聚焦;代价是覆盖要靠 side trace 渐进补齐、假设更激进退出更频繁、依赖"这次执行"的代表性。这些取舍,全部从"录制"这个根本动作里长出来。

## §7 回扣主线 + 后续地图

回到本书的主线:**把动态执行安全变成机器码**。

这一章讲的是这条主线在 **JIT 侧流水线的第一个真实动作**——录制。一个热点路径要变成机器码,第一步就是把它的动态执行"固化"成静态的、可优化的 IR,而"固化"的方式是录制:

- **"动态"** 体现在录制的输入是解释器这次真实执行的 pc、栈值、类型——不是静态读代码,而是让程序动起来边跑边记。这是 trace JIT 区别于 method JIT 的根本。
- **"变成 IR"** 体现在 `lj_record_ins` 的大 switch——每条字节码,根据它的操作码和这次真实类型,翻译成对应的 IR 指令,追加到 `J->cur.ir`。栈槽经 `getslot`/`sload` 变成 `IR_SLOAD`,常量经 `lj_ir_kint`/`lj_ir_kstr` 进常量区,算术经 `lj_opt_narrow_arith` 变成 `IR_ADD` 等,表访问经 `lj_record_idx` 变成 `IR_HREFK`+`IR_HLOAD`,调用经 `lj_record_call` 跨帧继续录。
- **"安全"** 体现在每个乐观假设都配 guard(`IRT_GUARD`/`IRM_W`),guard 和 snapshot 耦合(`needsnap`/`guardemit`),终止判断保证 root trace 真的是循环(`rec_loop_interp`/`lj_record_stop`)。录出的 IR 要么忠实于这次执行,要么 guard 兜底退出——永远不会比解释器更错。

录制结束时,`J->cur.ir` 里装着这条 trace 的完整 IR 序列——线性、带 guard、带 snapshot、类型已特化。这份 IR 是后面所有阶段(OPT 优化、ASM 生成机器码)的输入。P2-05 §3.2 那张"状态机每个状态调用的模块"表里,RECORD 状态调的就是 `lj_record_ins`,它的产出物 IR 就是下一章的主题。

录制的 IR 序列虽然线性,但它的内部结构遵循一个叫 SSA(静态单一赋值)的形式——每条 IR 指令定义一个唯一的值,操作数引用这个值,不重复定义。SSA 是后面所有优化的基础(常量折叠、CSE、死代码消除都假设 SSA)。下一章就来把这份 IR 的 SSA 形式、引用编号、指令格式彻底讲清。

| 本章讲的机制 | 后续章节 | 讲什么 |
|---|---|---|
| **IR_SLOAD 的类型特化** | [P2-07 IR 与 SSA](P2-07-IR与SSA.md) | IR 的 SSA 形式、REF_BIAS 偏移、IRIns 64 位格式、TRef 位布局 |
| **lj_opt_narrow_arith 的类型决策** | P2-08 类型窄化 narrowing | 怎么从动态类型推断出具体类型(整数/浮点),决定生成哪种 IR |
| **guard IR 的 mode 契约** | P5-16 guard 运行时检查与 side exit | guard 在机器码里长什么样,怎么跳到 exit stub |
| **lj_snap_add 与 guard 的耦合** | P5-17 snapshot 恢复解释器状态 | snapshot 怎么编码栈状态,退出时怎么恢复 |
| **rec_loop_jit / 链接到已编译 trace** | P5-19 trace 链接(TraceLink 9 种) | trace 跑完去哪:LOOP/ROOT/STITCH/... |
| **rec_mm_arith 录 metamethod 调用** | P6-20/P6-21 FFI | metamethod 调用怎么录,FFI 调 C 怎么录(`lj_crecord.c`) |

读完这一章,你应该能在脑子里看见录制的那条"双轨":**解释器执行一条字节码(改真实状态)的同时,录制器把它翻译成 IR(基于真实状态),两边共享这次执行的真实事实**。这条双轨不是凭空设计的,它是"动态语言编译时类型未知"这个根本障碍逼出来的——既然静态分析拿不到类型,就让程序真跑一遍,边跑边记,记下的就是真实发生的那条路径和真实类型。这就是"录制",trace JIT 的心脏动作。

录制没有魔法。它的全部精妙,在于"让解释器再跑一遍"这个朴素的动作——一遍跑完,一条线性路径就固化成了 IR,带着真实类型、带着 guard 兜底、带着 snapshot 退路。后面所有的优化和代码生成,都在这份 IR 上做。下一章,我们就来把这份 IR 的内部结构——SSA 形式、引用编号、指令格式——彻底拆开。
