# P1-02 字节码与手写汇编 VM

> **本书主线**:把动态执行安全变成机器码。**二分法**:解释器侧 ↔ JIT 侧。**★对照**:官方 Lua + JVM/V8。**源码**:LuaJIT 2.1.ROLLING。**基调**:纯直球，不用比喻；从第一性原理一步步推导。
>
> **本章在主线里的位置**:上一章(P0-01)讲清了 JIT 的全貌——先在解释器里跑，发现热点，录成 trace,编译成机器码，失败退回解释器。那句话里"先在解释器里跑"和"失败退回解释器",都把解释器摆在了**基座**的位置：它是 JIT 的起点，也是 JIT 的退路。这一章就来把这个基座本身看清楚——LuaJIT 的解释器为什么即便不编译，也已经快得惊人。答案浓缩成一句：它的解释器是**手写汇编**写的，跑的是**栈式字节码**。

## 引子:一个让人意外的数字

P0-01 我们留下了一个数字:LuaJIT 跑同样的 Lua 代码，可以比官方 Lua 快几十倍。这个"几十倍",等到后面讲完 JIT(trace + guard + 机器码),是容易理解的——机器码本来就比解释执行快得多。

但有一个事实会让人意外:**哪怕关掉 JIT(LuaJIT 用 `-j off` 启动，只跑解释器),LuaJIT 的解释器本身就比官方 Lua 的解释器快 2 到 5 倍。**

这就奇怪了。JIT 还没上场，光凭解释器，LuaJIT 就已经拉开身位了。这说明 LuaJIT 的解释器本身就不是个"普通"解释器——它在 JIT 出现之前，就已经为"快"做了极致的设计。

这一章要回答的问题就是:**LuaJIT 的解释器，凭什么自己就这么快?**

答案有两个关键词，摆在标题里:**字节码**和**手写汇编**。我们一个一个来，先把"为什么是字节码"从第一性原理推导出来，再把"为什么是手写汇编"推导出来。两条线索合起来，你就看懂了这个基座的全貌。

## §1 第一性原理:CPU 执行一段脚本，中间发生了什么

我们从最基础的地方开始。

你写了这么一段 Lua:

```lua
local x = 1
local y = x + 2
```

P0-01 讲过:CPU 只懂机器码，你的 Lua 代码必须先被**翻译**成 CPU 能执行的东西。问题是，这个翻译分几步做、做成什么样?

最朴素的想法是:**一步到位**,直接把 Lua 源码翻译成机器码。每次运行，读一句 `local y = x + 2`,当场分析它的语法、查 x 的类型、生成对应的机器码，然后执行。

这个想法听起来直接，实际上有个致命问题:**源码分析太慢了**。

源码是什么?是一串字符:`local y = x + 2`。要从这串字符里搞清楚"这是一个赋值语句，左边叫 y,右边是 x 加 2",你得做词法分析(把字符切成 `local`/`y`/`=`/`x`/`+`/`2` 这些 token)、语法分析(把这些 token 拼成一棵语法树)、语义分析(查 x 是哪个变量、+ 在这个上下文什么含义)。这套流程叫**前端编译**,它非常吃 CPU——光是把一串字符读懂，就要跑成千上万条机器指令。

而这套前端编译的产出是什么?是"这条语句要做的事"。比如"把 x 加 2 存到 y"。**这个产出，每次运行都是一样的。** 你第一次读懂 `local y = x + 2` 得到"y = x + 2",第二次、第三次读到同一行，得到的结果一模一样。

那如果每次运行都重新从字符分析一遍，就亏大了——明明结果不变，却一遍遍重复最慢的"读懂字符"的活儿。

这就引出了一个自然的拆分:**把前端编译和执行分开。**

### 1.1 字节码:把"读懂字符"的结果存下来

把前端编译(词法 + 语法 + 语义分析)做**一次**,把结果存成一种**紧凑的、解释器能快速消费的中间格式**。以后每次运行，不再碰源码字符，只消费这个中间格式。

这个中间格式，就叫**字节码(bytecode)**。

为什么叫字节码?因为它的基本单位是一个 32 位的字(LuaJIT 里),或者若干字节。每一条字节码，就代表一个**最小操作**——"取某槽的值""加一个常量""存到某槽""跳转"等等。它比源码字符紧凑得多(不用再分析语法),又比机器码高级(还保留"操作"这个语义，能被解释器理解)。

所以完整的链条是:

```text
源码字符  --[前端编译,做一次]-->  字节码  --[解释器,每次运行]-->  执行
```

字节码站在中间，是个**折中点**:它替解释器挡掉了"反复读懂字符"的开销，但又没走到"机器码"那一步(那是 JIT 干的活，需要类型信息，P0-01 讲过动态语言不能提前做)。

这个折中，就是所有解释型语言(Python、Ruby、官方 Lua、LuaJIT)共同的架构选择:**先编译成字节码，再用解释器跑字节码。**

### 1.2 为什么不直接跳到机器码?

你可能会问：既然要翻译，为什么不一步到位翻译成机器码，省掉字节码这一层?

P0-01 已经回答了:**动态语言不能提前编译成机器码**,因为类型在运行时才知道。`a + b` 在你看到真实运行时 a、b 是整数之前，根本不知道该生成整数加法还是浮点加法还是别的。所以"提前编译成机器码"这条路，对动态语言走不通。

那字节码这一层就躲不掉了——它是动态语言能提前做到的最远一步。把源码结构化成字节码，留给运行时(解释器或 JIT)去处理类型、去执行。

讲清了"为什么需要字节码",我们来看 LuaJIT 的字节码长什么样。

## §2 LuaJIT 字节码:一条 32 位的指令长什么样

我们终于要看源码了。但在贴源码之前，先用第一性原理想想：一条字节码，应该装哪些东西?

一条指令要做一件事，至少需要两样:

1. **做什么**(操作码 opcode):加法?赋值?跳转?调用函数?
2. **对谁做**(操作数 operand):加法的两个数从哪来?结果存哪?跳转跳到哪?

LuaJIT 把这两样打包进一个 **32 位**的字里。源码在 `lj_bc.h`,先看格式说明(`lj_bc.h:12-22`):

```c
/* Bytecode instruction format, 32 bit wide, fields of 8 or 16 bit:
**
** +----+----+----+----+
** | B  | C  | A  | OP | Format ABC
** +----+----+----+----+
** |    D    | A  | OP | Format AD
** +--------------------
** MSB               LSB
**
** In-memory instructions are always stored in host byte order.
*/
```

这张图是整章的基础，看懂它，后面全都顺。我们从最低位(右边)往上拆:

- **OP(8 位)**:操作码。占最低 8 位，所以最多能编码 256 种操作(实际用了 97 种，后面讲)。这 8 位告诉解释器"这是一条什么指令"。
- **A(8 位)**:第一个操作数。几乎每条指令都有 A,通常是**目的槽**(结果存到栈上的第几个位置)或**基址槽**。
- 接下来分两种格式:

  - **ABC 格式**:A 之后是 C(8 位)、B(8 位),共三个 8 位操作数。适合"两个源 + 一个目的"的指令，比如加法 `dst = B + C`。
  - **AD 格式**:A 之后是 D(16 位),把 B 和 C 合并成一个 16 位的大操作数。适合只需要一个宽操作数的指令，比如加载一个常量(常量表下标可能超过 255)、或者跳转(跳转距离用 16 位编码)。

为什么搞两种格式?因为不同指令需要的操作数个数和宽度不一样。加法需要两个源操作数，各 8 位够用(最多 256 个槽);但跳转的距离可能很大，需要 16 位。一个 32 位字，既要有 8 位的灵活性，又要有 16 位的容量，那就设计成"要么 ABC 三个 8 位、要么 AD 一个 8 位加一个 16 位"。这是对 32 位空间的最经济利用。

### 2.1 为什么是 32 位，不是 16 位或 64 位?

值得停下来想：为什么偏偏是 32 位?

- **16 位太窄**:操作码占 8 位后只剩 8 位，操作数宽度严重不足(比如常量表下标只能到 256,跳转距离极短),装不下真实程序的需要。
- **64 位太宽**:每条指令占 8 字节，一个函数的字节码体积翻倍，占内存、占缓存。而且大多数指令根本用不满 64 位(很多指令的 B、C 都是小数字),纯属浪费。
- **32 位是甜点**:操作码 8 位 + 操作数 24 位(ABC)或 8+16(AD),操作数范围足够覆盖一个函数的局部变量数(一个函数的栈槽数通常几十个)和常量表大小，又保持每条指令只占 4 字节，紧凑、省缓存。

这个 32 位的选取，是"够用"和"紧凑"的平衡。后面你会看到，解释器主循环取一条指令，就是 `mov reg, [PC]` 读一个 4 字节字，然后 `PC += 4`,非常整齐。

### 2.2 拆解一条指令:源码里的宏

光看格式图还不够，得看源码怎么把这条指令的各个字段取出来。`lj_bc.h` 里有一组宏，把 32 位字拆成 OP/A/B/C/D(`lj_bc.h:34-39`):

```c
/* Macros to get instruction fields. */
#define bc_op(i)	((BCOp)((i)&0xff))
#define bc_a(i)		((BCReg)(((i)>>8)&0xff))
#define bc_b(i)		((BCReg)((i)>>24))
#define bc_c(i)		((BCReg)(((i)>>16)&0xff))
#define bc_d(i)		((BCReg)((i)>>16))
#define bc_j(i)		((ptrdiff_t)bc_d(i)-BCBIAS_J)
```

这组宏就是"字节码解码器"。逐行看:

- `bc_op(i)`:取最低 8 位(`i & 0xff`),解释成操作码类型 `BCOp`。
- `bc_a(i)`:取第 8-15 位(`(i>>8) & 0xff`),解释成"寄存器/槽号"`BCReg`。
- `bc_b(i)`:取最高 8 位(`i>>24`)。注意没有 `& 0xff`,因为右移 24 位后，高 8 位已经移到最低位(符号扩展会处理),结果就是那 8 位。
- `bc_c(i)`:取第 16-23 位(`(i>>16) & 0xff`)。
- `bc_d(i)`:取第 16-31 位(`i>>16`),也就是 C 和 B 拼起来的 16 位。
- `bc_j(i)`:跳转专用。`bc_d` 减去一个偏移 `BCBIAS_J`,把无符号的 D 转成有符号的跳转距离(向前跳还是向后跳)。

操作数类型 `BCReg` 是什么?查 `lj_obj.h:151-153`:

```c
typedef uint32_t BCIns;  /* Bytecode instruction. */
typedef uint32_t BCPos;  /* Bytecode position. */
typedef uint32_t BCReg;  /* Bytecode register. */
```

三条都注释得很清楚。`BCIns` 是一条完整的字节码指令(那个 32 位字),`BCPos` 是字节码位置(第几条指令),`BCReg` 是"字节码寄存器"——注意这个命名，它叫 register,但稍后你会看到，LuaJIT 的"寄存器"其实就是**栈上的槽位编号**。这是理解"栈式字节码"的关键，我们在 §4 展开。

还有一组对应的"组装"宏，把字段拼回成 32 位指令(`lj_bc.h:53-57`):

```c
#define BCINS_ABC(o, a, b, c) \
  (((BCIns)(o))|((BCIns)(a)<<8)|((BCIns)(b)<<24)|((BCIns)(c)<<16))
#define BCINS_AD(o, a, d) \
  (((BCIns)(o))|((BCIns)(a)<<8)|((BCIns)(d)<<16))
#define BCINS_AJ(o, a, j)	BCINS_AD(o, a, (BCPos)((int32_t)(j)+BCBIAS_J))
```

`BCINS_ABC(o,a,b,c)` 把操作码放最低 8 位、A 左移 8 位、B 左移 24 位、C 左移 16 位，或起来，就是一条 ABC 格式的指令。`BCINS_AD` 同理。这两个宏在**编译器前端**(`lj_parse.c`)生成字节码时大量使用——比如 `lj_parse.c:434-435`:

```c
#define bcemit_ABC(fs, o, a, b, c)	bcemit_INS(fs, BCINS_ABC(o, a, b, c))
#define bcemit_AD(fs, o, a, d)		bcemit_INS(fs, BCINS_AD(o, a, d))
```

前端分析完源码，调用 `bcemit_ABC`/`bcemit_AD` 把一条条字节码写进函数的字节码段。字节码段紧跟在 `GCproto`(函数原型)结构体后面，通过 `proto_bc` 宏取出(`lj_obj.h:420`):

```c
#define proto_bc(pt)	((BCIns *)((char *)(pt) + sizeof(GCproto)))
```

也就是说，一个 Lua 函数在内存里长这样:`GCproto` 头(含参数数、栈帧大小、行号表等信息)紧跟着一段 `BCIns[]` 字节码数组。解释器拿到 `GCproto`,就能找到它的字节码，开始一条条执行。

到这里，字节码的**格式**讲清了。但"格式"只是容器，真正有意思的是容器里装了哪些操作码、它们怎么对应到"操作"。下一节看操作码全集。

## §3 操作码全集:97 种操作能干些什么

LuaJIT 一共有多少种操作码?数一下 `lj_bc.h` 里 `BCDEF` 宏的条目。`BCDEF` 是一个巨大的宏(`lj_bc.h:71-197`),每行定义一个操作码，格式是:

```c
/* (name, filler, Amode, Bmode, Cmode or Dmode, metamethod) */
```

五个字段：名字、填充、A 的模式、B 的模式、C(或 D)的模式、对应的元方法。我数过，一共 **97 条**(可以用 `grep -cE '^\s*_\(' lj_bc.h` 验证，得 97)。这 97 条覆盖了一门动态语言需要的全部操作。

这个 `BCDEF` 宏用得很巧妙。它本身只是"声明",不带实现。靠不同的"展开方式",同一份 `BCDEF` 能生成多种东西：操作码枚举、模式表、解释器派发入口等。看 `lj_bc.h:200-205`:

```c
/* Bytecode opcode numbers. */
typedef enum {
#define BCENUM(name, ma, mb, mc, mt)	BC_##name,
BCDEF(BCENUM)
#undef BCENUM
  BC__MAX
} BCOp;
```

`BCDEF(BCENUM)` 把每条展开成 `BC_ISLT,`、`BC_ISGE,`……最后是 `BC__MAX`(代表操作码总数，也就是 97)。这个枚举 `BCOp` 就是操作码的类型。后面你会看到，解释器的派发表就是用 `BC_XXX` 当下标索引的——`jmp aword [DISPATCH+OP*8]` 里那个 `OP`,取值就是 `BC_XXX` 之一。

97 个操作码怎么分类?`BCDEF` 里的注释已经分好了组，我们按组看，挑有代表性的讲(不逐条列，那没意义；讲清每类"解决什么问题")。

### 3.1 比较与测试(ISLT/ISEQV/IST/...)

```c
/* Comparison ops. ORDER OPR. */
_(ISLT,	var,	___,	var,	lt) \
_(ISGE,	var,	___,	var,	lt) \
...
_(ISEQV,	var,	___,	var,	eq) \
_(ISNEV,	var,	___,	var,	eq) \
```

`ISLT` = "is less than",比较 A 和 D 两个槽的值，小于则继续、不小于则跳转。`ISEQV` = "is equal value",相等判断。这类指令的特点是:**它们本身不产生结果，只决定下一步走哪条路**(配合后面的 `JMP`)。这是分支的底层实现。

注意 `BCDEF` 里相邻的操作码经过了精心排序，源码里有一堆 `LJ_STATIC_ASSERT` 守着这些顺序(`lj_bc.h:207-228`):

```c
LJ_STATIC_ASSERT(((int)BC_ISEQV^1) == (int)BC_ISNEV);
LJ_STATIC_ASSERT(((int)BC_ISEQS^1) == (int)BC_ISNES);
...
LJ_STATIC_ASSERT((int)BC_ISLT^1) == (int)BC_ISGE);
```

`ISEQV` 异或 1 等于 `ISNEV`——也就是说，"相等"和"不等"这两个语义相反的操作码，编号只差 1(最低位不同)。为什么?这样在汇编里，要把"相等"翻转成"不等",只要把操作码的最低位翻一下(`xor op, 1`),一条指令搞定。这种"把相关操作码排得只差一位"的技巧，在手写汇编里能省掉分支，后面讲 dispatch 会看到它的价值。这是 LuaJIT 把每条指令开销压到极致的一个缩影。

### 3.2 算术(ADDVN/SUBVN/.../ADDVV)

这是本章反复用的例子 `y = x + 2` 会落到的指令。LuaJIT 的算术指令分三类(`lj_bc.h:102-118`):

```c
/* Binary ops. ORDER OPR. VV last, POW must be next. */
_(ADDVN,	dst,	var,	num,	add) \
_(SUBVN,	dst,	var,	num,	sub) \
...
_(ADDNV,	dst,	var,	num,	add) \
_(SUBNV,	dst,	var,	num,	sub) \
...
_(ADDVV,	dst,	var,	var,	add) \
_(SUBVV,	dst,	var,	var,	sub) \
```

看后缀:**VN** 表示 "var + num"(一个操作数是栈上的槽，一个是常量池里的数字),**NV** 是 "num + var"(常量在前),**VV** 是 "var + var"(两个都是栈上的槽)。为什么要分三种?

因为 `y = x + 2` 这种"变量加常量"的代码极其常见。如果不分，统一用 ADDVV,那么加 2 这个常量每次都得先从常量池搬到栈上，再相加——多一次栈写。LuaJIT 干脆为"加常量"专门造了 ADDVN 指令：一个操作数直接指向常量表下标，解释器直接从常量表读，省掉一次入栈。

这是一种典型的"为高频场景特化指令"的设计。代价是指令数变多(加法就有 3×5=15 条:VN/NV/VV × 加减乘除模),换来的是每条指令更精炼、执行更快。手写汇编允许做这种特化——C 写的解释器如果搞这么多变种，维护成本会很高；而汇编每个变种就一小段代码，复制粘贴改几个寻址方式，不增加复杂度。

### 3.3 常量加载(KSHORT/KNUM/KSTR/KPRI/KNIL)

```c
_(KSHORT,	dst,	___,	lits,	___) \
_(KNUM,	dst,	___,	num,	___) \
_(KSTR,	dst,	___,	str,	___) \
_(KPRI,	dst,	___,	pri,	___) \
_(KNIL,	base,	___,	base,	___) \
```

把常量加载到栈槽。`KSHORT` 加载一个 16 位有符号整数字面量(直接编码在指令里，`lits` = signed literal short),`KNUM` 从常量池加载一个浮点数，`KSTR` 加载一个字符串常量，`KPRI` 加载原始类型(nil/true/false),`KNIL` 把一段槽批量置 nil。

`y = x + 2` 里的 `2`,如果是个小整数，会被编译成 `KSHORT` 指令(直接把 2 编进指令的操作数),不用查常量表——又是一个为小整数特化的快路径。

### 3.4 表操作(TGETS/TGETB/TGETV/TNEW/...)

```c
_(TGETV,	dst,	var,	var,	index) \
_(TGETS,	dst,	var,	str,	index) \
_(TGETB,	dst,	var,	lit,	index) \
```

表查找。`TGETS` = 用字符串 key 查表(`t.name`),`TGETB` = 用小整数 key 查表(`t[1]`),`TGETV` = 用另一个变量当 key 查表(`t[k]`)。又是按 key 类型特化：字符串 key 最常见(对象属性访问),给它专门的指令和快速路径。

### 3.5 调用与返回(CALL/RETM/RET0/...)

```c
_(CALL,	base,	lit,	lit,	call) \
_(RET,	rbase,	___,	lit,	___) \
_(RET0,	rbase,	___,	lit,	___) \
_(RET1,	rbase,	___,	lit,	___) \
```

`CALL` 调用函数，A 是函数所在基址槽，D 编码参数个数和返回值个数。返回类指令分 RET/RET0/RET1:RET0 返回 0 个值(最常见，函数结尾的 `return`),RET1 返回 1 个值，RET 返回多个值。又是按"返回值个数"特化——绝大多数函数返回 0 或 1 个值，给它们专门的、更精简的指令。

### 3.6 循环与跳转(FORL/JMP/LOOP)

```c
_(FORL,	base,	___,	jump,	___) \
_(IFORL,	base,	___,	jump,	___) \
_(JFORL,	base,	___,	lit,	___) \
_(LOOP,	rbase,	___,	jump,	___) \
_(JLOOP,	rbase,	___,	lit,	___) \
_(JMP,	rbase,	___,	jump,	___) \
```

这里藏着一个对全书主线至关重要的设计。注意循环指令分三种前缀:**无前缀**(FORL/ITERL/LOOP)= 普通解释器执行；**I 前缀**(IFORL/IITERL/ILOOP)= interpreter 专用变体；**J 前缀**(JFORL/JITERL/JLOOP)= **JIT 接管后的入口**。

`JLOOP` 后面跟的操作数不是跳转目标，而是一个 `lit`(字面量)——这个字面量是**已编译的 trace 的编号**。也就是说，当 JIT 把一段循环编译成机器码后，解释器会在原来循环回边的地方，把 `LOOP` 指令替换成 `JLOOP`,这样下次循环回到这里，解释器不再解释执行循环体，而是**直接跳进那条 trace 的机器码**。

这就是 P0-01 §12 讲的"第 6 阶段：安装，机器码接管"在字节码层面的落点——JIT 不是凭空接管代码，而是通过把字节码里的 `LOOP` 换成 `JLOOP`,让解释器的派发机制自然地把控制权交给机器码。这个设计极其优雅:JIT 和解释器共享同一套字节码派发，切换是无缝的。

类似的还有函数头指令 FUNCF/IFUNCF/JFUNCF(`lj_bc.h:190-192`):一个函数被 JIT 编译后，它的入口字节码从 FUNCF 换成 JFUNCF,下次调用直接进 JIT 代码。

`BCDEF` 末尾那些 `LJ_STATIC_ASSERT` 守着这套编码的规律(`lj_bc.h:219-228`):

```c
LJ_STATIC_ASSERT((int)BC_FORL + 1 == (int)BC_IFORL);
LJ_STATIC_ASSERT((int)BC_FORL + 2 == (int)BC_JFORL);
LJ_STATIC_ASSERT((int)BC_LOOP + 1 == (int)BC_ILOOP);
LJ_STATIC_ASSERT((int)BC_LOOP + 2 == (int)BC_JLOOP);
```

普通/I/J 三种变体，编号依次 +1、+2。这样在汇编里，要从 FORL 切换到 IFORL,只要操作码 +1;切到 JFORL,+2。又是为了在汇编里用最简单的算术完成变体切换。

97 个操作码就讲这些代表。核心思想是一致的:**每种高频操作都有专门的、特化的指令，让解释器用最少的步骤完成它**。这是"快"的一个来源。但还有另一个、也是更惊人的来源——这些指令不是用 C 实现的，而是用**手写汇编**。在展开那个之前，我们先解决一个更根本的问题：这些操作数(槽号)到底指的是什么?为什么是"栈式"?

## §4 栈式字节码:操作数是"栈位置"

这是本章一个关键概念，也是 LuaJIT 解释器和官方 Lua 解释器的第一个重大分野。

回看 §2:`BCReg` 是"字节码寄存器",操作数 A/B/C/D 都是 `BCReg` 类型。但这个"寄存器"不是 CPU 寄存器，而是**值栈上的槽位编号**。

什么叫值栈?Lua 是一门支持函数调用、局部变量的语言。每个函数运行时，需要一块内存放它的参数、局部变量、临时中间结果。LuaJIT 给每个函数调用分配一段连续的内存，叫**值栈(value stack)**,栈上的每个位置(每个槽)能放一个 Lua 值(一个 `TValue`,8 字节，稍后细看)。

操作数 A、B、C,指的就是"栈上的第几个槽"。比如 `MOV` 指令(复制一个值),A 是目的槽号，D 是源槽号——意思是"把栈上第 D 槽的值，复制到第 A 槽"。

### 4.1 用 y = x + 2 走一遍

我们把 `local y = x + 2` 从源码走到字节码，看栈是怎么动的。

假设进入这个函数时，值栈布局是这样的(槽号从 0 开始):

- 槽 0:函数对象自己(func,占一个槽，这是惯例)
- 槽 1:`x`(x 是个局部变量，分配在槽 1)
- 槽 2:`y`(y 还没赋值，槽 2 留给它)

现在要执行 `y = x + 2`。前端编译后，大致生成这样的字节码序列(简化示意，真实的寄存器分配会略有不同):

```text
KSHORT  3, 2        ; 把常量 2 加载到槽 3(临时槽)
ADDVN   2, 1, 0     ; 槽2 = 槽1(x) + 常量表[0](也是2),结果存槽2(y)
```

这里 `ADDVN 2,1,0` 的三个操作数:A=2(目的，即 y 所在槽)、B=1(源，即 x 所在槽)、C=0(常量表下标，指向常量 2)。执行后，槽 2(y)就有了 x+2 的值。

注意整个过程:**操作数全部是槽号(栈位置),没有任何"压栈/弹栈"的显式动作**。为什么?

这就是"栈式字节码"的精髓——但这里要澄清一个常见的误解。很多人把"栈式字节码"想象成 JVM 那种"压一个、弹一个"的纯栈机器:`push x; push 2; add; store y`。**LuaJIT 不是这种。** LuaJIT 的字节码虽然叫"栈式"(因为操作数指向一个栈),但它的操作数是**带编号的栈槽**(槽 1、槽 2、槽 3),不是"栈顶那个匿名的值"。

更准确的说法是:LuaJIT 的字节码是**基于栈的、但栈槽有编号**的格式。每个局部变量在编译时就被分配了一个固定的栈槽号(槽 1 是 x,槽 2 是 y),操作数直接引用这些编号。这和 JVM 那种"操作数栈匿名压弹"不同，反而更像"寄存器号",只不过这些"寄存器"物理上排成一段连续内存(值栈)。

那为什么还叫"栈式"?因为它和"真正的寄存器式"(官方 Lua 那种)有一个本质区别，这个区别要在 §6 的对照里才看得最清楚。这里先记住:LuaJIT 的操作数是**值栈槽号**,值栈是一段连续内存，栈顶的位置由一个专门的寄存器(BASE)指着。

### 4.2 TValue:一个栈槽装什么

既然每个槽是一个 Lua 值，我们得看清楚这个值长什么样。源码在 `lj_obj.h:174-213`:

```c
/* Tagged value. */
typedef LJ_ALIGN(8) union TValue {
  uint64_t u64;		/* 64 bit pattern overlaps number. */
  lua_Number n;		/* Number object overlaps split tag/value object. */
#if LJ_GC64
  GCRef gcr;		/* GCobj reference with tag. */
  int64_t it64;
  struct {
    LJ_ENDIAN_LOHI(
      int32_t i;	/* Integer value. */
    , uint32_t it;	/* Internal object tag. Must overlap MSW of number. */
    )
  };
#else
  struct {
    LJ_ENDIAN_LOHI(
      union {
	GCRef gcr;	/* GCobj reference (if any). */
	int32_t i;	/* Integer value. */
      };
    , uint32_t it;	/* Internal object tag. Must overlap MSW of number. */
    )
  };
#endif
  ...
} TValue;
```

一个 `TValue` 是 8 字节(64 位),它是个 union——同一块 8 字节内存，可以按不同方式解读。关键设计是**带标签的值(tagged value)**:

- 如果这个值是个**数字**(浮点 double),这 8 字节就是一个完整的 IEEE 754 双精度浮点数(字段 `n`)。
- 如果这个值是**别的类型**(字符串、表、函数、nil、true/false、整数),那么低 4 字节放"值"(比如 GC 对象的指针、整数值),高 4 字节放**类型标签**(`it`,internal tag)。

为什么这样设计?这是解释器能快速判断类型的基础。前面 P0-01 讲过，动态语言每条指令执行前都要判断操作数的类型。LuaJIT 把类型标签和值打包在同一个 8 字节里，判断类型只要读高 4 字节比对一下(`cmp dword [BASE+reg*8+4], tp`),一条汇编指令搞定。

类型标签具体是什么值?看 `lj_obj.h:264-285` 附近:

```c
#define LJ_TSTR			(~4u)
#define LJ_TTAB			(~11u)
...
#define LJ_TISNUM		0xfffeffffu
```

注意这些标签是**小的负数**(`~4u` 就是 -5,`~11u` 就是 -12)。为什么用负数?源码注释解释了(`lj_obj.h:228-232`):

```c
/* Internal tags overlap the MSW of a number object (must be a double).
** ... Small negative numbers are used to shorten the
** encoding of type comparisons (reg/mem against sign-ext. 8 bit immediate). */
```

用小负数当标签，是因为在 x86 汇编里，和一个 8 位有符号立即数比较，编码比和 32 位立即数比较更短、更快。比如 `cmp dword [...], LJ_TISNUM`,因为 `LJ_TISNUM` 是 `0xfffeffff`,符号扩展后能用更紧凑的指令编码。这种"为汇编指令编码长度优化数据布局"的设计，只有在手写汇编、对每条指令精打细算的语境下才会做——C 编译器根本不会替你想到这一层。

`LJ_TISNUM` 还有个特别的值 `0xfffeffff`。在非 GC64 模式下，它是一个"特殊的 NaN"——当这 8 字节被当成 double 解读时，它是个 NaN(不是数字);当当成带标签的值解读时，高 4 字节是 `0xfffeffff` 表示"这是个整数"。这种"标签和 double 的位模式重叠"的技巧叫 **NaN-boxing**,用一套位模式同时容纳 double 和带标签的其他类型，不浪费一个字节。LuaJIT 的 NaN-boxing 比官方 Lua 的更紧凑、判断更快。

讲清了栈槽和 TValue,我们有了看手写汇编的全部前置知识。现在进入本章的核心问题:**为什么这些字节码指令，要用汇编来写?**

## §5 核心问题:为什么要手写汇编

这是 LuaJIT 解释器快的根源，也是这一章最重要的一节。

回到 §1 的结论：解释器的工作，是"反复地理解并执行字节码"。每条字节码，解释器都要走一遍"取指 → 译码 → 执行 → 下一条"的循环。这个循环跑得有多快，直接决定了解释器有多快。

现在问:**用 C 写这个循环，和用汇编写这个循环，差在哪?**

### 5.1 C 写解释器的开销

假设我们用 C 写解释器。一个最朴素的 C 解释器主循环长这样(伪代码):

```c
for (;;) {
  BCIns ins = *pc++;           // 取指
  BCOp op = bc_op(ins);        // 译码
  switch (op) {                // 派发
    case BC_MOV: {             // 执行 MOV
      BCReg a = bc_a(ins), d = bc_d(ins);
      stack[a] = stack[d];     // 复制 TValue
      break;
    }
    case BC_ADDVN: { ... }
    ...
  }
}
```

看起来清晰、好维护。但这段 C 代码，经过 C 编译器(gcc/clang)编译成机器码后，会背上很多"解释器不需要、但 C 的抽象模型强加"的开销:

**第一项开销：寄存器分配不在你手里。**

C 编译器要管理通用寄存器。它看到 `stack[a] = stack[d]`,得决定 a、d、stack 的值放哪个寄存器。但通用寄存器有限( x86 只有 8 个，x64 也只有 16 个),编译器要在"解释器主循环的状态"(pc、当前栈基址、派发表指针)和"每条指令的临时变量"之间分配，经常不得不把一些值在寄存器和内存(栈)之间来回搬(store/load)。这些搬动是纯开销，对解释逻辑毫无贡献。

而手写汇编，可以把解释器最关键的几个状态**永久钉死**在固定寄存器里。LuaJIT 就是这么做的(下一节看源码):值栈基址永远在某个寄存器里、字节码指针 pc 永远在某个寄存器里、派发表指针永远在某个寄存器里。这些寄存器从头到尾不挪窝，解释器主循环里完全不出现"把 pc 存回内存、下一条再取出来"的动作。

**第二项开销：函数调用的栈帧开销。**

如果 C 代码里把每条字节码的处理写成一个函数(`void do_mov(...)`、`void do_addvn(...)`),那么每条字节码都要经历一次函数调用:push 参数、push 返回地址、建栈帧、执行、拆栈帧、ret。这些 push/pop 和栈帧操作，每条字节码都要来一遍，开销惊人。

即便不写成函数、写成 switch case,编译器生成的代码里，每个 case 块仍然可能有冗余的栈调整(为了对齐、为了保存调用者保存寄存器)。

而手写汇编，可以让每条字节码的处理**就是一段直线代码**,没有函数调用、没有栈帧、处理完直接跳到下一条指令的取指。每条字节码的开销被压到最低。

**第三项开销：派发的实现方式。**

C 的 `switch(op)`,编译器通常编译成一堆比较+跳转，或者一个跳转表。但 C 编译器对 switch 的优化是"通用"的，它不知道每个 case 出现的频率、不知道怎么让分支预测器更舒服。而且 switch 的每个 case 结尾要有 break(跳到循环末尾再回顶部),多一次跳转。

而手写汇编可以用**计算跳转(computed goto)**:每条字节码处理完，直接 `jmp [派发表 + op*8]` 跳到下一条指令的处理代码，一步到位，没有"回循环顶部"的中间跳转。GCC 后来加了 computed goto 扩展(`&&label`)也是受了这种思路启发，但手写汇编能做得更彻底——下一节你会看到 LuaJIT 的 `ins_next` 宏就是这么干的。

**第四项开销：无法为每条指令做极致特化。**

前面 §3 讲过，LuaJIT 有 ADDVN/ADDNV/ADDVV 三种加法、TGETS/TGETB/TGETV 三种表查找。用 C 写，每多一个变种就多写一个 case,代码膨胀，维护痛苦。而且 C 编译器不一定能把每个变种都优化到最优(它对"这个 case 里 B 一定是常量表下标"这种语义不敏感)。

手写汇编，每个变种就是一小段精心调过的代码，复制粘贴改几个寻址方式，不增加心智负担，却能把每种情况都压到最快。

### 5.2 把开销加起来:几倍的差距

上面四项开销，单独看每项都不大(几十纳秒、几个时钟周期)。但解释器是什么?是**每条字节码都要走一遍这个循环**。一个程序如果跑一亿次字节码(很常见，一个百万次循环里每次循环体十几条指令就上亿了),那么每条字节码省下的几十纳秒，乘以一亿，就是几秒的差距。

所以 C 写的解释器和手写汇编写的解释器，在"每条指令的开销"上能差 2-5 倍。这就是为什么 LuaJIT 解释器(关掉 JIT)就比官方 Lua(C 写的解释器)快 2-5 倍——**每条指令省下的那点开销，被亿次执行放大成了几倍的总体差距。**

这就是"为什么要手写汇编"的全部理由。用一句话概括:

> **C 编译器是为"通用代码"优化的，它不知道解释器主循环的特殊性(固定状态、直线代码、计算跳转)。手写汇编绕开编译器的通用策略，把解释器循环的开销压到物理极限。**

代价当然有：汇编难写、难读、难维护，而且每种 CPU 架构(x86、ARM、MIPS、PPC)的汇编完全不同，得为每种架构各写一遍。这个代价怎么解决?LuaJIT 用了一个叫 **dynasm** 的工具。但在看 dynasm 之前，我们先把 LuaJIT 手写汇编的成果——`vm_x86.dasc`——看清楚，印证上面讲的每一项设计。

## §6 源码印证:vm_x86.dasc 怎么实现"每条指令开销最小化"

LuaJIT 的解释器汇编源码，在 `src/vm_x86.dasc`(x86/x64 架构，5947 行)。还有 `vm_arm64.dasc`、`vm_mips.dasc` 等对应其他架构。我们以 x86 版本为例。

先看这个文件开头最重要的东西:**固定的寄存器分配**(`vm_x86.dasc:26-58`)。

```asm
|// Fixed register assignments for the interpreter.
|// This is very fragile and has many dependencies. Caveat emptor.
|.define BASE,		edx		// Not C callee-save, refetched anyway.
|.if not X64
|.define KBASE,		edi		// Must be C callee-save.
|.define KBASEa,	KBASE
|.define PC,		esi		// Must be C callee-save.
|.define PCa,		PC
|.define DISPATCH,	ebx		// Must be C callee-save.
|.elif X64WIN
...
|.define KBASE,		r15d		// Must be C callee-save.
|.define KBASEa,	r15
|.define PC,		ebx		// Must be C callee-save.
|.define PCa,		rbx
|.define DISPATCH,	r14d		// Must be C callee-save.
|.endif
|
|.define RA,		ecx
|.define RAH,		ch
|.define RAL,		cl
|.define RB,		ebp		// Must be ebp (C callee-save).
|.define RC,		eax		// Must be eax.
|.define RCW,		ax
|.define RCH,		ah
|.define RCL,		al
|.define OP,		RB
|.define RD,		RC
```

这段把"解释器的核心状态"永久钉死在固定寄存器上。逐个看:

- **`BASE`(edx)**:值栈基址。指向当前函数值栈的起点。所有"槽 A、槽 B"的访问，都是 `[BASE + A*8]` 这种形式——BASE 加上槽号乘 8(每个 TValue 8 字节)。BASE 永远不挪窝，解释器随时能用它访问任何栈槽。
- **`PC`(esi)**:字节码指针(program counter)。指向**下一条要执行的字节码**。取指就是 `mov reg, [PC]`,然后 `PC += 4`。PC 也永远不挪窝。
- **`KBASE`(edi/r15)**:常量表基址。指向当前函数的常量池。像 `ADDVN` 里"加常量"的那个常量，就是从 `[KBASE + C*8]` 取。给常量表一个专门寄存器，省得每次访问常量都重新算基址。
- **`DISPATCH`(ebx/r14)**:派发表指针。指向那张"操作码 → 处理代码地址"的跳转表。这个寄存器是计算跳转的关键。
- **`RA`(ecx)/`RB`(ebp)/`RC`(eax)**:指令操作数的临时寄存器。译码后，A、B、C 的值就放在这几个里，后面的执行代码直接用它们寻址栈槽。`OP` 和 `RD` 是 `RB`/`RC` 的别名(同一物理寄存器复用，因为译码阶段用完后，执行阶段可以覆盖)。

注释里反复出现"Must be C callee-save"(必须是调用者保存的寄存器)。为什么?因为解释器在执行字节码的过程中，偶尔要调用 C 函数(比如遇到元方法、要 GC)。调用 C 函数时，被调用者(callee)可能会破坏调用者保存的寄存器(eax/edx/ecx 等)。如果 PC、BASE 这些状态放在调用者保存的寄存器里，每次调 C 都得先存起来、调完再恢复——又是开销。所以把它们放在**被调用者保存的寄存器**(esi/edi/ebx/ebp)里，C 函数会自动帮你保存恢复，解释器就不用操心了。

这是一个"为减少跨 C 调用的保存恢复开销"的精心选择。注释那句"This is very fragile and has many dependencies. Caveat emptor.(买者自慎)"也点明了：这种固定寄存器分配极度脆弱，改一处牵动全身，但换来的是极致的速度——LuaJIT 作者认为这个交换值得。

### 6.1 取指/译码/派发:ins_next 宏

现在看解释器的心脏——取下一条指令并派发。这是 `ins_NEXT` 宏(`vm_x86.dasc:336-347`):

```asm
|// Instruction decode+dispatch. Carefully tuned (nope, lodsd is not faster).
|.macro ins_NEXT
|  mov RC, [PC]           // 取指:把 PC 指向的 4 字节字节码读进 RC
|  movzx RA, RCH          // 译码 A:取 RC 的高字节(第 8-15 位)放 RA
|  movzx OP, RCL          // 译码 OP:取 RC 的低字节(第 0-7 位)放 OP
|  add PC, 4              // PC 前进一条(PC 指向下一条)
|  shr RC, 16             // 把 RC 右移 16 位,现在 RC 低 16 位是 D(或 C+B)
|.if X64
|  jmp aword [DISPATCH+OP*8]   // 计算跳转:跳到派发表[OP]项指向的代码
|.else
|  jmp aword [DISPATCH+OP*4]
|.endif
|.endmacro
```

就这 7 行，完成了"取指、译码、PC 前进、派发"全套。逐行拆:

1. `mov RC, [PC]`:PC 寄存器指向下一条字节码(4 字节)。这条指令把它整体读进 RC 寄存器。现在 RC 里装着完整的 32 位字节码。
2. `movzx RA, RCH`:`RCH` 是 RC 的高字节(ah)。回顾字节码格式:`| B | C | A | OP |`,最低字节是 OP,第二个字节是 A。x86 是小端，所以读进 RC 后，最低字节(RCL)是 OP,第二字节(RCH)是 A。这条把 A 取出来放 RA。
3. `movzx OP, RCL`:把 OP 取出来放 OP(就是 RB 寄存器)。现在 RB 里是操作码。
4. `add PC, 4`:PC 前进 4 字节，指向下一条字节码。**注意这一步在执行当前指令之前就做了**——这是个小优化，这样当前指令执行中如果出错(比如抛异常),PC 已经指向下一条，处理起来更方便。
5. `shr RC, 16`:RC 右移 16 位。原来 RC 是 `|B|C|A|OP|`,右移 16 位后变成 `|0|0|B|C|`,也就是低 16 位是 C 和 B(或者说 D)。现在 RC(也是 RD)里是 B/C/D 操作数，后面的执行代码可以直接用。
6. `jmp aword [DISPATCH+OP*8]`:**计算跳转**。DISPATCH 是派发表基址，`OP*8` 是偏移(每个表项 8 字节，即一个 64 位指针),`[DISPATCH+OP*8]` 取出该表项里存的地址，然后跳过去。这个地址就是"操作码 OP 对应的那段处理代码"的起始地址。

这 7 行就是 P0-01 §3 讲的"解释器为了搞清楚该做什么而做的额外工作"的物理实现。注意它有多紧凑:

- **没有函数调用**:不 call 任何函数，全程直线代码 + 一个间接跳转。
- **没有栈帧**:不 push/pop 任何东西(除了偶尔调 C 时)。
- **寄存器全复用**:RC 既装原始指令、又装译码后的 D 操作数；RA/RB 在译码阶段用、执行阶段复用。
- **计算跳转一步到位**:不经过 switch 的"比较+跳转"或"回循环顶部",直接一个间接 jmp 到目标。

注释那句"Carefully tuned (nope, lodsd is not faster)"也很有意思:`lodsd` 是 x86 一条"读 [esi] 到 eax 并 esi+=4"的指令，看起来正好能替代 `mov RC,[PC]` + `add PC,4` 两行。但作者实测发现 `lodsd` 并不更快(它在现代 CPU 上是微码实现的，反而慢)。这种"连一条指令的取舍都要实测"的态度，正是手写汇编的日常。

派发宏有两种模式，由 `vm_x86.dasc:350-365` 选择:

```asm
|// Instruction footer.
|.if 1
  // Replicated dispatch. Less unpredictable branches, but higher I-Cache use.
  .define ins_next, ins_NEXT
  .define ins_next_, ins_NEXT
|.else
  // Common dispatch. Lower I-Cache use, only one (very) unpredictable branch.
  // Affects only certain kinds of benchmarks (and only with -j off).
  // Around 10%-30% slower on Core2, a lot more slower on P4.
  .macro ins_next
    jmp ->ins_next
  .endmacro
  ...
|.endif
```

默认选 `.if 1` 分支，即**复制式派发(replicated dispatch)**:每个字节码处理代码的末尾，都内联一份完整的 `ins_NEXT`(那 7 行)。代价是代码体积大(每条指令末尾都有这 7 行，占用指令缓存 I-Cache),好处是**少了一次跳转**——不用先 `jmp ->ins_next` 跳到公共的取指代码、再跳回来，直接就地取指派发。

另一种"公共派发"是所有指令末尾都 `jmp ->ins_next`,跳到唯一一份取指代码。省 I-Cache,但多一次跳转，而且这个跳转目标固定，分支预测器很难猜(因为前一条指令是什么完全不确定),注释说在 Core2 上慢 10-30%,在 P4 上慢更多。

LuaJIT 选了复制式——**用 I-Cache 换分支预测的准确性**。这是手写汇编才能做的权衡:C 编译器不会替你做这种"把同一段代码复制 97 份"的决定。

### 6.2 派发表:GG_State 里的函数指针数组

`jmp [DISPATCH+OP*8]` 跳到的地址，存在哪?在派发表里。派发表是 `GG_State` 结构体的一个字段(`lj_dispatch.h:107`):

```c
typedef struct GG_State {
  lua_State L;				/* Main thread. */
  global_State g;			/* Global state. */
  ...
  ASMFunction dispatch[GG_LEN_DISP];	/* Instruction dispatch tables. */
  BCIns bcff[GG_NUM_ASMFF];		/* Bytecode for ASM fast functions. */
} GG_State;
```

`dispatch` 是一个 `ASMFunction`(函数指针)数组，长度 `GG_LEN_DISP`(`lj_dispatch.h:84-86`):

```c
#define GG_LEN_DDISP	(BC__MAX + GG_NUM_ASMFF)
#define GG_LEN_SDISP	BC_FUNCF
#define GG_LEN_DISP	(GG_LEN_DDISP + GG_LEN_SDISP)
```

`GG_LEN_DDISP = BC__MAX + GG_NUM_ASMFF`——前 `BC__MAX`(97)项就是 97 个操作码对应的处理代码地址，后面 `GG_NUM_ASMFF`(57)项是 ASM 快速函数(内置库的高频函数，也在汇编里实现)。所以 `jmp [DISPATCH+OP*8]` 里 `OP` 是 0-96 时，正好落到 `dispatch[OP]`,取出的就是"操作码 OP 的处理代码"的地址。

`GG_State` 这个结构体设计得很有讲究：它把 `lua_State`(主线程)、`global_State`(全局状态)、`jit_State`(JIT 状态)、`hotcount`(热点计数器)、`dispatch`(派发表)全部**打包分配在一起**。为什么?因为它们全都要被汇编频繁访问，放在一起，用一个 DISPATCH 寄存器(指向 dispatch 数组)就能用相对偏移访问到所有这些字段。看 `lj_dispatch.h:119-123`:

```c
#define GG_DISP2G	(GG_OFS(g) - GG_OFS(dispatch))
#define GG_DISP2J	(GG_OFS(J) - GG_OFS(dispatch))
#define GG_DISP2HOT	(GG_OFS(hotcount) - GG_OFS(dispatch))
#define GG_DISP2STATIC	(GG_LEN_DDISP*(int)sizeof(ASMFunction))
```

`GG_DISP2G` 是"从 dispatch 到 global_State 的偏移",`GG_DISP2J` 是到 jit_State 的偏移，`GG_DISP2HOT` 是到 hotcount 数组的偏移。汇编里要用全局状态，就 `mov reg, [DISPATCH + GG_DISP2G + 字段偏移]`;要用 JIT 状态，就 `[DISPATCH + GG_DISP2J + ...]`;要访问热点计数器，就 `[DISPATCH + GG_DISP2HOT + ...]`。一个 DISPATCH 寄存器，一把梭访问所有运行时状态。这是"把所有热数据集中、用一个基址寄存器覆盖"的空间布局优化。

### 6.3 一条具体指令:BC_MOV 的汇编

宏和派发表讲完了，看一条最简单的指令——MOV(复制一个值)——的完整汇编(`vm_x86.dasc:3831-3843`):

```asm
  case BC_MOV:
    |  ins_AD	// RA = dst, RD = src
    |.if X64
    |  mov RBa, [BASE+RD*8]       // 读源槽(整个 8 字节 TValue)
    |  mov [BASE+RA*8], RBa       // 写目的槽
    |.else
    |  mov RB, [BASE+RD*8+4]      // 32 位下分两次搬(高 4 字节:类型标签)
    |  mov RD, [BASE+RD*8]        //  (低 4 字节:值)
    |  mov [BASE+RA*8+4], RB
    |  mov [BASE+RA*8], RD
    |.endif
    |  ins_next_
    break;
```

逐行:

- `ins_AD`:这是译码宏(`vm_x86.dasc:328`),把指令的 A 和 D 操作数取出来放 RA 和 RD。对于 AD 格式，`ins_AD` 实际上是空的(`.macro ins_AD; .endmacro`)——因为 `ins_NEXT` 里 `shr RC,16` 已经把 D 放进 RC(=RD)了，而 A 也已经放进 RA 了，这里不需要再做任何事，只是个占位/文档。
- x64 下:`mov RBa, [BASE+RD*8]` 一次性读源槽的 8 字节(整个 TValue,`RBa` 是 64 位版本),`mov [BASE+RA*8], RBa` 一次性写目的槽。两条指令完成一次 TValue 复制。
- x86(32 位)下：寄存器只有 32 位，得拆成两次：先搬高 4 字节(类型标签),再搬低 4 字节(值)。四次 mov。
- `ins_next_`:派发到下一条指令。

这就是 MOV 的全部：从栈槽 RD 读 8 字节、写到栈槽 RA。**没有任何类型判断**(MOV 不关心值是什么类型，原样搬就行)、**没有任何函数调用**、**没有栈帧**。从取指到派发下一条，总共可能就十几条机器指令，几纳秒。

对比一下如果用 C 写:`stack[a] = stack[d]` 这一行，编译器看到的是两个 `TValue` 的赋值，可能生成一串 load/store,中间还可能插寄存器溢出；外面还套着 switch case 的跳转。同样的语义，C 版本可能多出几条指令。

### 6.4 一条带类型检查的指令:BC_ADDVN

MOV 太简单，没体现"动态语言要判断类型"。我们看 `y = x + 2` 真正会走到的指令——ADDVN(变量加常量)。它通过一个 `ins_arithpre` 宏实现(`vm_x86.dasc:3934-3961`):

```asm
|.macro ins_arithpre, sseins, ssereg
|  ins_ABC
||vk = ((int)op - BC_ADDVN) / (BC_ADDNV-BC_ADDVN);
||switch (vk) {
||case 0:                          // VN: var + num
|   checknum RB, ->vmeta_arith_vn  // 检查 RB 是不是数字,不是就跳元方法
|   .if DUALNUM
|     cmp dword [KBASE+RC*8+4], LJ_TISNUM; jae ->vmeta_arith_vn
|   .endif
|   movsd xmm0, qword [BASE+RB*8]    // 从栈槽 RB 读双精度数到 xmm0
|   sseins ssereg, qword [KBASE+RC*8] // 从常量表 RC 读双精度数,与 xmm0 运算
||  break;
||case 1:                          // NV: num + var
|   checknum RB, ->vmeta_arith_nv
|   ...
|   movsd xmm0, qword [KBASE+RC*8]
|   sseins ssereg, qword [BASE+RB*8]
||  break;
||default:                         // VV: var + var
|   checknum RB, ->vmeta_arith_vv
|   checknum RC, ->vmeta_arith_vv
|   movsd xmm0, qword [BASE+RB*8]
|   sseins ssereg, qword [BASE+RC*8]
||  break;
||}
|.endmacro
```

这是算术指令的"预处理":译码(`ins_ABC`)、判断变体(VN/NV/VV,用 `vk` 区分)、类型检查(`checknum`)、把两个操作数读进 SSE 寄存器(`xmm0` 和 `ssereg`)。

关键看 VN 变体(我们的 `y = x + 2`):

- `checknum RB, ->vmeta_arith_vn`:`checknum` 是个宏(`vm_x86.dasc:392`),展开成 `cmp dword [BASE+RB*8+4], LJ_TISNUM; jae target`——检查栈槽 RB 的高 4 字节(类型标签),如果标签值 >= `LJ_TISNUM`(说明不是数字，可能是字符串、表等),就跳到 `->vmeta_arith_vn`(元方法处理，慢路径)。
- 如果是数字(标签 < LJ_TISNUM),继续:`movsd xmm0, [BASE+RB*8]` 把 RB 槽的 double 读进 xmm0。
- `sseins ssereg, [KBASE+RC*8]`:`sseins` 是个参数，对 ADDVN 它会被替换成 `addsd`。所以这条是 `addsd xmm0, [KBASE+RC*8]`——从常量表 RC 槽读一个 double,加到 xmm0 上。
- 结果在 xmm0 里。后面 `ins_arithpost` 宏(`vm_x86.dasc:3995-3997`)把它写回:`movsd qword [BASE+RA*8], xmm0`。
- 最后 `ins_next` 派发。

把 ADDVN 的完整执行串起来:

```asm
; 译码
movzx RB, RCH        ; B(源槽)放 RB
movzx RC, RCL        ; C(常量下标)放 RC
; 类型检查(快路径)
cmp dword [BASE+RB*8+4], LJ_TISNUM
jae ->vmeta_arith_vn ; 不是数字?跳元方法
; 加法(SSE)
movsd xmm0, qword [BASE+RB*8]    ; x
addsd xmm0, qword [KBASE+RC*8]   ; x + 常量(2)
movsd qword [BASE+RA*8], xmm0    ; 存到 y
; 派发下一条
<ins_NEXT 那 7 行>
```

十几条指令，完成"判断 x 是数字、做浮点加法、存结果、取下一条指令并派发"。**类型检查只有一条 cmp + 一条条件跳转**(而且因为标签是小负数，这条 cmp 编码很紧凑);**真正的算术只用 SSE 的 addsd 一条**;**派发是一个间接 jmp**。

对比 C 写的等价代码:`if (isnum(b)) { setnum(a, numval(b) + constnum(c)); } else { call_metamethod(...); }`——编译器要为 isnum 判断生成 load+cmp、为 numval/constnum 生成多次内存读、可能还有临时的栈保存。手写汇编把这些全压成最小。

还有个细节：类型检查失败时跳到 `->vmeta_arith_vn`,这是个**元方法处理入口**,用 C 实现(在 `lj_meta.c`)。也就是说，LuaJIT 的策略是:**常见情况(数字加数字)用汇编快路径直接处理，罕见情况(字符串拼接、表运算调元方法)退回 C 慢路径**。这是"为热路径特化、冷路径走通用"的典型设计，和 P0-01 讲的 JIT 思路(热路径编译、冷路径退回解释器)是同一个哲学在不同层面的体现。

### 6.5 hotcount 的嵌入:FORL 指令

最后看一条把解释器和 JIT 串起来的指令——FORL(数值 for 循环回边)。这正好回扣全书主线(`vm_x86.dasc:5239-5244`):

```asm
  case BC_FORL:
    |.if JIT
    |  hotloop RB
    |.endif
    | // Fall through. Assumes BC_IFORL follows and ins_AJ is a no-op.
    break;
```

`hotloop` 是个宏(`vm_x86.dasc:415-421`):

```asm
|.macro hotloop, reg
|  mov reg, PC
|  shr reg, 1
|  and reg, HOTCOUNT_PCMASK
|  sub word [DISPATCH+reg+GG_DISP2HOT], HOTCOUNT_LOOP
|  jb ->vm_hotloop
|.endmacro
```

它做的事，就是 P0-01 §6 讲的 hotcount 机制在汇编里的落地:

- `mov reg, PC; shr reg, 1; and reg, HOTCOUNT_PCMASK`:用当前 PC 算一个哈希(PC 右移 1 位再掩码),落到 hotcount 数组的某个槽。`HOTCOUNT_PCMASK` 是 `63*2`(64 个槽，每个 2 字节),所以哈希到 64 个槽之一。
- `sub word [DISPATCH+reg+GG_DISP2HOT], HOTCOUNT_LOOP`:把那个槽的计数器减去 `HOTCOUNT_LOOP`(值是 2,见 `lj_dispatch.h:78`)。注意这是 `[DISPATCH + GG_DISP2HOT + ...]`,用 DISPATCH 寄存器加上偏移就访问到 hotcount 数组——印证了 §6.2 讲的"一个 DISPATCH 覆盖所有运行时状态"。
- `jb ->vm_hotloop`:如果减完结果为负(借位),说明计数器到 0 了，这个循环够热了，跳到 `->vm_hotloop`(触发 trace 录制，进 JIT)。

所以每次 for 循环回边，FORL 指令除了做循环判断(下一条 IFORL 干这个),还会顺手把 hotcount 减一，减到 0 就启动 JIT。**解释器和 JIT 的衔接点，就埋在这条字节码的汇编实现里**。这是本章和 P1-03(dispatch 与热点)的接缝。

到这里，手写汇编解释器的全部关键——固定寄存器分配、计算跳转派发、每条指令的极致精简、hotcount 嵌入——都从源码印证了。接下来解决一个工程问题：这么多架构的汇编，怎么维护?

## §7 dynasm:手写汇编又保持可移植

手写汇编有个显而易见的难处:x86、ARM、MIPS、PPC、ARM64,每种 CPU 架构的汇编完全不同。给每种架构各手写一遍 6000 行解释器，工作量惊人，而且容易写错、难维护。

LuaJIT 的解法是一个叫 **dynasm(Dynamic Assembler)** 的小工具。它在 `luajit2/dynasm/` 目录下，是一套 Lua 写的程序(`dynasm.lua` 加上各架构后端 `dasm_x86.lua`、`dasm_arm64.lua` 等)。

dynasm 的思路是:**用一种带宏的、架构无关的汇编描述语言(.dasc 文件),由 dynasm 工具生成具体架构的汇编(.s 文件),再由汇编器汇编成机器码。** 这样，解释器的"源码"是 .dasc,人只需要维护 .dasc 这一份(虽然也有分架构的 vm_x86.dasc/vm_arm64.dasc,但它们共享大量宏和结构)。

### 7.1 .dasc 文件长什么样

前面贴的所有 `vm_x86.dasc` 代码，都是 dynasm 格式。它的特征:

- 每行汇编以 `|` 开头(`| mov RC, [PC]`)。dynasm 看到 `|`,就把这行当成汇编指令处理。
- `|.` 开头的是 dynasm 的元指令(`.define`、`.macro`、`.if`、`.else`),用来定义宏、做条件编译。
- `||` 开头的是**嵌入的宿主端 Lua/C 代码**。比如前面看到的 `||vk = ((int)op - BC_ADDVN) / (BC_ADDNV-BC_ADDVN);`,这是在 dynasm 生成阶段(构建时)执行的 C 代码，用来根据当前在生成哪个操作码，选择生成哪段汇编。
- 标签用 `|->label:` 或 `|=>local:` 定义，前者是全局标签(如 `->BC_MOV_Z`),后者是局部标签。

这种设计的好处：汇编里可以混入逻辑判断(用 `||` 嵌入 C),实现"根据参数生成不同的汇编"。比如算术指令那套宏，ADDVN/ADDNV/ADDVV 用同一个宏模板，靠 `||switch(vk)` 在生成时选不同分支，生成三套不同的汇编代码。这就是为什么 .dasc 文件里有 `case BC_ADDVN: case BC_ADDNV: case BC_ADDVV:` 这种 C 的 switch——它不是运行时的 switch,而是**构建时的 switch**,决定 dynasm 生成哪些机器码。

### 7.2 构建:从 .dasc 到机器码

整个构建流程(`build_backend` 函数，`vm_x86.dasc:5623-5631`):

```c
static int build_backend(BuildCtx *ctx)
{
  int op;
  dasm_growpc(Dst, BC__MAX);       // 给 97 个操作码分配标签空间
  build_subroutines(ctx);          // 生成各种子例程(vm_call 等)
  |.code_op
  for (op = 0; op < BC__MAX; op++)
    build_ins(ctx, (BCOp)op, op);  // 为每个操作码生成处理代码
  return BC__MAX;
}
```

`build_ins`(`vm_x86.dasco:3499`)对每个操作码调用一次，里面是巨大的 `switch(op)`,每个 `case BC_XXX:` 生成那段指令的汇编。最后 dynasm 把所有生成的指令编码成机器码字节流，输出成 `lj_vm.s`(或直接编进目标文件)。

这套机制的关键洞察:**解释器的机器码，是构建时一次性生成的，不是运行时生成的**。.dasc 是"源",`lj_vm.s`/`lj_vm.o` 是"产物",运行时直接用。这和 JIT(运行时生成机器码)是两回事——dynasm 是个"静态的"代码生成工具，只是它的输入是高层次描述、输出是手工质量的汇编。

这给了 LuaJIT 两个好处:

1. **可维护**:.dasc 用宏和条件编译，一份描述能衍生出多种变体(32/64 位、有无 DUALNUM、有无 JIT),比纯手写 .s 易读易改。
2. **可移植**:换架构只要换 dynasm 的后端(`dasm_arm64.lua`),.dasc 里大部分逻辑可以复用(虽然 vm_x86.dasc 和 vm_arm64.dasc 是分开的，但它们共享设计理念)。

dynasm 本身是个独立的小项目(MIT 许可，`dynasm.lua` 文件头 `dynasm/dynasm.lua:1-43` 标明 version 1.5.0,作者 Mike Pall),除了 LuaJIT 自己用，也被其他需要运行时生成代码的项目(比如别的 JIT)拿去用。在 LuaJIT 里，它同时服务于两件事:**构建时生成解释器**(本章),**运行时为 JIT 生成的 trace 机器码做辅助**(后端 `lj_asm` 那条线，P4 篇)。

讲完了字节码、手写汇编、dynasm 三件事，我们回头审视这个设计的"为什么 sound"。

## §8 为什么这个设计是 sound 的:每条指令开销最小化，但不丢正确性

"sound"在这里的意思是：这个解释器虽然为了快做了各种激进的优化(固定寄存器、计算跳转、类型特化指令),但它**仍然是一个语义完整的、正确的 Lua 解释器**——任何合法的 Lua 程序，它都能跑出和官方 Lua 完全一样的结果。快，绝不以牺牲正确性为代价。

我们逐项审视那些"激进"的设计，看它们为什么不破坏正确性:

**固定寄存器分配(BASE/PC/DISPATCH 永久占着寄存器)**

这不影响正确性，因为这些寄存器只被解释器自己用。当解释器要调用 C 函数时，那些"被调用者保存"的寄存器(esi/edi/ebx/ebp)C 函数会自动保存恢复；而 BASE 用的是 edx(调用者保存),所以解释器在调 C 前会把它存到 `lua_State->base`(看 `vm_call` 里 `mov BASE, L:RB->base` 就是重新从 L 里取回 BASE)。也就是说，**固定寄存器只是"解释器内部的约定",跨 C 调用时该保存的都保存了**,语义上和 C 版本没区别，只是少了运行时的来回搬运。

**计算跳转(jmp [DISPATCH+OP*8])**

这只是"用跳转表实现 switch"的一种方式，语义上等价于 `switch(op){case BC_MOV:...}`——每个操作码都派发到它对应的处理代码，一个不漏。派发表在 `GG_State.dispatch` 里，构建时由 `build_ins` 为所有 97 个操作码都填了入口(漏一个就会跳到野指针崩溃，而 LuaJIT 显然没漏)。所以计算跳转快，但不丢任何一个操作码的处理。

**类型特化指令(ADDVN 假设操作数是数字)**

注意 ADDVN 不是"假设操作数一定是数字然后盲目做加法"。它有一条 `checknum RB, ->vmeta_arith_vn`——**先检查类型，不对就跳元方法**。所以无论操作数是什么类型，都有对应的处理：是数字走快路径(浮点加法),不是数字走慢路径(元方法，可能调 `__add`)。两类路径合起来，覆盖了 Lua 加法的全部语义。快路径只是"把最常见的数字情况特化了",慢路径兜底所有其他情况。

**NaN-boxing 标签(类型标签和 double 位模式重叠)**

这个设计要小心：它假设"用作标签的那些位模式，当被当成 double 解读时，都是 NaN"。这样，一个 TValue 要么是个合法的 double(非 NaN),要么是个带标签的其他类型(特定的 NaN 模式),两者不会混淆。判断类型只要看高位是不是某个标签值。这个不变式由标签的取值保证(`LJ_TISNUM = 0xfffeffff` 等，都是 NaN 范围内的位模式),只要不变式成立，类型判断就是准确的。LuaJIT 的 `itype()` 宏和 `tvisnum()` 等检查(`lj_obj.h:804-806`)都建立在这个不变式上。

**复制式派发(每条指令末尾都内联 ins_NEXT)**

这只是"把同一段取指派发代码复制 97 份",每份做的事完全一样(取 PC、译码、跳转表派发)。复制不影响语义，只影响代码体积和 I-Cache。所以它纯粹是个性能优化，正确性上等价于"所有指令共用一份取指代码"。

把这些合起来，结论是:**LuaJIT 解释器的每一个"快"的设计，都对应一个等价的、正确的语义实现；快是在"减少不必要的开销"上做文章，不是在"省略必要的步骤"上做文章。** 每条字节码该做的(取指、译码、类型判断、执行、派发)一样不少，只是每一步都用最精炼的汇编实现。这就是它能"又快又对"的根基。

这也回扣了 P0-01 §9 讲的那个关键不变式:**解释器保底，机器码加速**。JIT 生成的机器码，无论多么激进地假设，失败时都要退回解释器。而解释器本身——就是这一章讲的这个手写汇编 VM——是一个完整正确的 Lua 实现，它接过机器码的烂摊子(通过 snapshot 恢复状态，下一章 P1-03 和 P5-17 细讲),按真实类型一步步执行，给出正确结果。**解释器越快，这个"保底"就越廉价**,退回时的代价就越小。所以把手写汇编解释器做快，不只是为了让没编译的代码跑得快，更是为了让 JIT 失败回退时的代价可控。这是基座对整个 JIT 体系的支撑。

## §9 ★对照:官方 Lua 与 JVM/V8

讲清了 LuaJIT 解释器的全貌，把它和两个对象放在一起对照，设计取舍会更清楚。

### 9.1 对照一:官方 Lua(切"栈式 vs 寄存器式"和"汇编 vs C")

官方 Lua(5.x)也是个字节码解释器，也有字节码格式、也有解释器主循环。但它和 LuaJIT 有两个根本差异:

**差异一：字节码格式——官方 Lua 是寄存器式，LuaJIT 是栈式(带编号槽)。**

官方 Lua 的字节码，操作数是**寄存器号**。一条指令 `GETTABLE R1, R2, R3` 意思是"用 R2 和 R3 的值做表查找，结果放 R1"——R1/R2/R3 是寄存器号，但这里的"寄存器"和 LuaJIT 的"槽号"在本层面上是一回事(都是值栈的位置)。

那真正的区别在哪?在于**指令能引用的寄存器范围**和**指令的编码**。官方 Lua 的指令也是 32 位，格式是 A(8位)/B(9位)/C(9位)/Opcode(7位)。它的 B 和 C 是 9 位(能编 512 个寄存器),比 LuaJIT 的 8 位(256 个槽)宽。为什么官方要 9 位?因为寄存器式希望尽量把值留在"寄存器"里复用，需要更多寄存器号；而 LuaJIT 的栈式设计倾向于值用完就丢(临时槽),需要的槽号范围小一些。

更深的区别在**指令密度**。寄存器式的优点是**指令少**:每个操作显式指定源和目的，不需要额外的"压栈/弹栈"或"加载常量"指令。比如 `a = b + c` 寄存器式可能就一条 `ADD Ra, Rb, Rc`;而纯栈式(像 JVM 字节码)要先 `load b; load c; add; store a` 四条。官方 Lua 凭这点，同样的程序生成的字节码条数通常比 JVM 字节码少。

但 LuaJIT 不是纯栈式(它的操作数是带编号的槽，不是匿名栈顶),所以它兼具两者的特点：像寄存器式一样能直接引用任意槽(不需要一串压栈),又像栈式一样把所有"寄存器"排在连续内存里(用 BASE 寄存器统一寻址)。LuaJIT 选这个折中，核心原因是**配合手写汇编**:只要一个 BASE 寄存器，所有槽都能用 `[BASE+槽号*8]` 访问到，汇编极其简洁；如果是真正的寄存器堆(分散的寄存器),寻址反而复杂。

**差异二：解释器实现——官方 Lua 用 C 写，LuaJIT 用手写汇编。**

这是拉开 2-5 倍差距的主因。官方 Lua 的解释器(`lvm.c` 里的 `luaV_execute`)是一个巨大的 C 函数，里面是 `switch(opcode)` 的主循环。它清晰、可读、跨平台(一份 C 代码在所有平台编译运行),但背上了 §5 讲的全部 C 开销：编译器分配寄存器、switch 派发的多次跳转、每个 case 的栈帧调整、无法为每条指令特化。

LuaJIT 为了快几倍，放弃了 C 的可维护性，为每种架构手写汇编。代价是:vm_x86.dasc/vm_arm64.dasc 这些文件极难读(满屏 `| mov`/`| jmp`/`| ins_next`,还有 `||` 嵌的构建期逻辑),改一处要懂整套寄存器约定，新加一种 CPU 架构要从头写一遍。但换来的是每条字节码只占十几条机器指令、几纳秒——这个性能差距，在亿次执行的循环里被放大成几倍。

一句话对照:**官方 Lua 用 C 换可维护性，LuaJIT 用汇编换速度。** 同样的字节码语义，两种实现策略，差几倍性能。

### 9.2 对照二:JVM / V8(切"解释器的角色"和"派发方式")

JVM 和 V8 也有解释器(JVM 的模板解释器、V8 的 Ignition),它们的解释器和 LuaJIT 的对照，能看清"解释器在 JIT 体系里的定位"。

**JVM 的模板解释器**:HotSpot JVM 的解释器也是"汇编级别"的——它用"模板"(每个字节码对应一段手写的或半手写的汇编),思路和 LuaJIT 几乎一样：固定寄存器分配、计算跳转派发、每条指令精简。实际上 LuaJIT 的设计与 JVM 模板解释器有异曲同工之妙(都认识到"C 写解释器不够快，要下沉到汇编")。区别是 JVM 字节码是纯栈式(匿名栈顶压弹),指令更密集；而 LuaJIT 是带编号槽，指令更稀疏但每条做的事更明确。

**V8 的 Ignition**:V8 后来也回归了解释器(Ignition,用寄存器式字节码 + C++ 写的解释器，配合代码生成),目的和 LuaJIT 一样：在 JIT 上场前先有个快的基座。Ignition 也用了"寄存器分配 + 计算跳转"的思路。

**派发方式的演进**:早期解释器用 switch 派发(慢),后来都转向直接线程化代码(direct threading,即计算跳转)。LuaJIT 的 `ins_next` 就是这个思路的极致——每条指令末尾内联派发，不回公共入口。JVM 和 V8 的解释器也这么做。这是现代高性能解释器的共识:**派发的开销是解释器的头号瓶颈，必须用计算跳转 + 复制派发把它压到最低。**

**解释器在 JIT 体系里的角色**:JVM/V8/LuaJIT 都遵循同一个架构——**解释器跑冷代码和保底，JIT 接管热点**。这印证了 P0-01 的主线：解释器是 JIT 的基座，不是替代品。三者都认识到：即使有 JIT,也不能没有解释器(冷代码太多，全编译不现实；JIT 失败要能退回)。而解释器本身越快，这个基座就越称职——这正是 LuaJIT 下沉到手写汇编的理由。

| 维度 | LuaJIT | 官方 Lua | JVM(V8) |
|---|---|---|---|
| 字节码格式 | 栈式(带编号槽) | 寄存器式 | 纯栈式(JVM)/寄存器式(V8 Ignition) |
| 解释器实现 | 手写汇编(dynasm) | C | 模板汇编(JVM)/C++(V8) |
| 派发方式 | 计算跳转 + 复制派发 | switch | 计算跳转 |
| 每条指令开销 | 极低(十几条机器指令) | 较高(C 开销) | 低(汇编级) |
| 有无 JIT | 有(trace JIT) | 无 | 有(method JIT) |
| 解释器角色 | 基座 + 保底 | 全部 | 基座 + 保底 |

这张表把 LuaJIT 解释器的定位讲清楚了:**它是"手写汇编级的、栈式带编号槽的、计算跳转派发的"解释器，在"快"上做到了和 JVM 模板解释器一个档次，而体积和复杂度远小于后者**。这是 LuaJIT 能作为"小而精的 JIT 教学样本"的原因——它的解释器就 6000 行 .dasc,却体现了高性能解释器的全部关键设计。

## §10 回扣主线:基座已就位，接下来发现热点

我们把这一章收束到全书主线上。

P0-01 讲了 JIT 的全貌：解释器先跑，发现热点，录 trace,编译机器码，失败退回。这一章我们深挖了那条主线里反复出现的"解释器",看清了它的真面目:

- 它跑的是**字节码**——一种 32 位紧凑的中间格式，把源码结构化的结果存下来，省掉反复解析字符的开销。字节码是动态语言能提前做到的最远一步(再往前就是机器码，需要类型信息，动态语言做不了)。
- 它的每条字节码指令，对应一小段**手写汇编**:固定寄存器分配(BASE/PC/DISPATCH 永久占着)、计算跳转派发(`jmp [DISPATCH+OP*8]`)、为常见情况特化指令(ADDVN/KSHORT/RET0...)。每条指令只占十几条机器指令、几纳秒，把"解释器的开销"压到了物理极限。
- 它本身是**完整正确**的：每个操作码都有处理、每个类型检查都有快路径和慢路径兜底、跨 C 调用时该保存的状态都保存。快，但不丢正确性。

这个基座做了两件事，直接服务于主线:

**第一，它让"还没被 JIT 接管的代码"也跑得不慢。** 程序刚启动时，所有代码都在解释器跑；JIT 只接管热点，冷代码永远在解释器。如果解释器慢(像官方 Lua 那样),冷代码就拖后腿。LuaJIT 的手写汇编解释器，让冷代码也保持合理的速度，整体性能才高。

**第二，它让"JIT 失败回退"的代价可控。** P0-01 §8 讲过，JIT 假设失败时要 side exit + snapshot 退回解释器。退回的频率不低(尤其是 trace 树还没长全的早期),每次退回都要在解释器里跑一段。解释器越快，这种退回的代价越小，JIT 体系就越健壮。**所以手写汇编解释器不只是"为了冷代码快",更是"为了让 JIT 敢于做激进的假设"——反正失败了退回来也不亏。** 这是基座对整个 JIT 冒进策略的底层支撑。

讲清了基座，接下来的问题是:**这个基座怎么发现热点?** 解释器一边跑字节码，一边要数"哪段代码跑得多了"——这就是 §6.5 里 `hotloop` 宏做的事，把 hotcount 减一、到 0 触发 JIT。但 hotcount 的完整机制(计数器怎么布局、阈值怎么定、触发后怎么进入录制、JIT 开关怎么动态影响派发表)还没展开。

下一章 [P1-03 dispatch 表与热点检测](P1-03-dispatch表与热点检测.md) 就来讲这个：那张 `dispatch[]` 表不只是操作码到处理代码的映射，它还会被动态改写(把普通指令换成 JIT 入口、把派发钩进热点检测);hotcount 不只是个计数器，它和 dispatch 表、和 trace 录制的启动，组成一套完整的热点发现机制。基座已就位，接下来看它怎么长出"发现热点"的眼睛。

---

*下一章 [P1-03 dispatch 表与热点检测](P1-03-dispatch表与热点检测.md):解释器怎么一边跑、一边数热点；dispatch 表怎么在运行时被改写，把热点代码无缝交给 JIT。*
