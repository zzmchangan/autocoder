# 《LuaJIT设计与实现深入浅出》·全书规划总纲

## 一、书名与定位

**《LuaJIT设计与实现深入浅出:把动态执行安全变成机器码》**

定位:"源码精解大书"语言运行时/JIT 线。以 **LuaJIT 2.1.ROLLING** 官方实现为唯一主干,逐文件、逐函数、逐行号讲清一个 trace JIT 编译器是怎么造出来的。深度对标《Tokio设计与实现深入浅出》,承接《Lua虚拟机设计与实现》(官方 Lua 那本)。

本书回答一个核心问题:**一段动态执行的脚本,怎么被安全地变成机器码。**

不是:Lua/LuaJIT 语法教程、LuaJIT 性能调优手册、FFI 用法指南(FFI 只作为"JIT 能调 C"的一章)。

## 二、灵魂主线

LuaJIT 如何把一段动态执行的脚本,**安全地**变成机器码。核心张力有三股,彼此拉扯:

- **快**:把热点编译成原生机器码,摆脱解释执行的取指/译码开销。
- **安全**:Lua 是动态类型,编译时不能像静态语言那样把类型定死;过度假设会在运行时出错。
- **省**:编译本身有开销,不能编译每段代码,也不能编译太久。

LuaJIT 用三个机制平衡这三股张力,也是全书主线:

1. **trace(线性热路径)**:不编译整个方法,只录制实际跑得最多的那条线性路径(循环体/调用链)。省——只编译热点。
2. **guard(把假设编进机器码)**:编译时做乐观假设(如"这里总是整数"),并在生成的机器码里插入运行时检查。安全——假设错就当场发现。
3. **side exit + snapshot(失败回退)**:假设失败时,trace 退出,靠 snapshot 把当前的寄存器/栈状态恢复成解释器能继续的样子,回到解释器执行。快——大部分时候假设成立,直接跑机器码;少数失败也不崩。

**核心一句话:乐观假设 + 运行时检查 + 失败可回退,让动态语言也能享受机器码的速度。**

## 三、二分法

**解释器侧 ↔ JIT 侧**。

- **解释器侧(P1)**:LuaJIT 自带一个极快的解释器(`lj_vm.s` 手写汇编、栈式字节码)+ dispatch 表 + 热点计数。这是 JIT 的基座:代码先在解释器跑,JIT 只接管热点。
- **JIT 侧(P2–P5)**:trace 录制(`lj_record`)→ IR(`lj_ir`,SSA)→ 优化(`lj_opt_*`)→ 后端代码生成(`lj_asm`)→ 运行时支持(snapshot `lj_snap`/side trace)。

P6(FFI/GC)是 JIT 与外部的协作。

## 四、★双对照

每章固定一栏 ★对照,两个对象,凸显 LuaJIT trace JIT 的取舍:

1. **官方 Lua(前一本)**:对照"有没有 JIT"。同样是 Lua 语言,官方 Lua 是纯解释器(寄存器式字节码 + 增量 GC);LuaJIT 是栈式字节码 + trace JIT + 增量 GC。对照出"JIT 到底加了什么、改了什么"。
2. **JVM / V8(method JIT)**:对照"trace vs method"。JVM/V8 以方法为单位编译(整个方法变 IR 再优化);LuaJIT 以 trace 为单位(只录一条热路径)。这是 JIT 两大流派的根本分歧——trace 对动态语言的热点更聚焦、编译更快,但假设更乐观、退出处理更复杂。

trace vs method 是全书最有讲头的一条对照线。

## 六、基调:纯直球,不用比喻

延续《Lua虚拟机》那本用户拍板的基调。JIT 技术性强(IR/SSA/寄存器分配/guard),更要靠:

- **第一性原理**:每个机制为什么必须这样,不这样会怎样。
- **真实源码**:函数名、行号、结构体字段、宏、汇编片段。
- **精确机制描述**:SSA 构建、别名分析、线性扫描、snapshot 编码。

全角中文标点(代码/行号/变量名半角)。
