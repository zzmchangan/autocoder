## 一、主线再回扣:快、安全、省,三股张力与三个解药

我们再回到 P0-01 那个起点。一本书的结尾,应该把开头的命题重新讲一遍——讲得更深。

LuaJIT 要做的事,可以用一句话概括:**把一段动态执行的 Lua 脚本,安全地变成机器码。** 这句话里有三个词彼此拉扯,构成了贯穿全书的核心张力:

- **快**。目标就是快。CPU 只懂机器码(P0-01 §1),解释器慢在"反复理解同一段指令"(P0-01 §3),要快就得让代码直接以机器码形态在 CPU 上飞奔,摆脱取指、译码、查表、分发的开销。
- **安全**。但 Lua 是动态类型,编译时类型未知(P0-01 §4)。`a + b` 到底是整数加、浮点加还是字符串拼,运行前谁也不知道。机器码必须确定——一旦生成了"整数加法",它不会自己判断。过度假设会在运行时算错、崩溃。
- **省**。编译本身有开销(P0-01 §6)。把每一段代码都编译,编译时间会超过它省下的时间。编译要花在刀刃上。

这三股张力看似无解:要快就得提前编译成机器码,但提前编译时类型未知(安全崩);要安全就保守一点现场判断类型,但现场判断就是解释器(快没了);要省就只编译热点,但热点之外的代码怎么办(安全/快覆盖不全)。JIT 的全部精妙,就是在这三角里找到平衡点。

LuaJIT 的解药是三个机制,正好一一对应(P0-01 §5–§10):

**解药一:trace(只录一条热路径,对应"省")。** 不编译整个函数,只在运行时录制"实际在走的那条线性路径"。一个循环里即使有十个分支,也只录跑得最多的那一条。trace 是线性的、没有分叉的,比整个方法小一两个量级,编译飞快。这把 method JIT(编译整个函数)的编译量,压缩到了实际热的那条路径。这是"省"的来源。

**解药二:guard(把假设编进机器码,对应"安全")。** trace 录制时观察到"x 每次都是整数",就乐观地按整数生成机器码——但**额外插一条检查**:运行时如果不是整数,立刻跳走。guard 本质是机器码里一句"cmp + jcc"。只要检查通过,所有乐观假设都成立,结果一定对;检查不通过,假设当场破产,机器码不会带着错误假设往下算。这是"安全"的兜底。

**解药三:side exit + snapshot(失败回退,对应"快"不被安全拖累)。** guard 触发后,机器码不能继续,但程序必须接着跑出正确结果。snapshot 事先记录了"机器码跑到这个 guard 时,寄存器里的值对应解释器栈的哪个槽",照着恢复,把状态摆成解释器能认的样子,退回解释器继续。解释器是完整正确的 Lua 实现,它算出的结果和官方 Lua 一模一样。这次慢一下,但不出错。而那个经常失败的退出点,如果够热,会再录一条 side trace——下次从那里直接走机器码,不再退回解释器。

把三个解药合起来,得到全书最重要的不变式(P0-01 §9):

> **机器码产生的结果,要么和解释器完全一样(所有 guard 通过),要么退回解释器(guard 触发)。它永远不会比解释器更错。**

正确性由解释器保底,速度由机器码提供。这就是"把动态执行**安全**变成机器码"里"安全"两个字的真正含义——也是 LuaJIT 比官方 Lua 快几十倍、却保证语义完全一致的根基。

一句话浓缩全书的命脉,这句话从 P0-01 写到 P6-21,每一章都在落地它的某一个环节:

> **乐观假设 + 运行时检查(guard)+ 失败可回退(side exit / snapshot),让动态语言也能享受机器码的速度。**

## 三、解释器侧 ↔ JIT 侧:21 章二分地图

全书的二分法是**解释器侧 ↔ JIT 侧**。这一节把 21 章铺进一张表,看每一章落在二分法的哪一边、它的主线落点是什么。

| 章节 | 标题 | 二分位置 | 主线落点(对应"快/安全/省"哪一股,或哪一段流水线) |
|---|---|---|---|
| P0-01 | 第一性原理:JIT 凭什么这么快 | 总纲 | 提出三股张力 + 三个解药;全书命题的推导 |
| P1-02 | 字节码与手写汇编 VM | **解释器侧** | 解释器本身要快(快):手写汇编把每条指令的 dispatch 省到极致;冷代码不慢 + JIT 失败回退代价可控(省/安全) |
| P1-03 | dispatch 表与热点检测 | **解释器侧**(跨 JIT 的开关) | 零开销发现热点(省):hotcount 计数器 + dispatch 模式切换不改字节码(sound) |
| P1-04 | 值·Table·状态 | **解释器侧** | 数据表示紧凑(快/省):NaN-boxing 8 字节 TValue + colo 同置 + table 哈希 |

| P4-13 | 线性扫描寄存器分配 | **JIT 侧·后端首章** | O(n log n) 快分配(省);与 snapshot 精密配合(预分配 / 弱引用 / rename 检查保 sound) |
| P4-14 | 汇编生成 asm | **JIT 侧·后端** | 三层(asm_ir 分发 → asm_* 翻译 → emit_* 发字节)对应三股张力;guard 在这里变成真实 cmp+jcc(假设→机器检查的临界点) |
| P4-15 | 后端目标与机器码内存 | **JIT 侧·后端收尾** | 五种 CPU 多后端 + W^X 可执行内存 + mcode area 管理(机器码安全放出来跑的工程问题) |
| P5-16 | guard 运行时检查与 side exit | **JIT 侧·运行时** | guard 的 cmp+jcc 怎么执行、失败怎么跳 exit stub(安全的运行时落地) |
| P5-17 | snapshot 恢复解释器状态 | **JIT 侧·运行时** | slot→IR ref 映射 + 差量压缩(快+省);机器码状态无损翻译回解释器栈(安全交接) |
| P5-18 | side trace:从退出点录制新 trace | **JIT 侧·运行时** | "越跑越快"的根源:增量优化 + snapshot 重用 + 热补丁不改父逻辑 |
| P5-19 | trace 链接(TraceLink 9 种) | **JIT 侧·运行时收尾** | trace 跑完去哪:LOOP 回自己 / STITCH 缝合 / 退回解释器;织成机器码跳转网 |

| P6-21 | FFI 录制 + GC 与 JIT 协作 | **JIT 侧**(压轴) | FFI 调用能被 JIT 编译(快的"一半")+ GC 追踪所有 JIT 造的对象(内存安全) |

读这张表的方法:左边 P1 系列(02/03/04)都在解释器侧——它们回答"JIT 还没启动时,代码怎么跑、热点怎么被发现、数据怎么表示"。从 P2-05 开始全部进入 JIT 侧,沿着流水线一路向下:P2 是录制和 IR(把路径变成半成品),P3 是优化(把半成品打磨),P4 是后端(把半成品变成机器码),P5 是运行时(机器码怎么安全跑、怎么退出、怎么链接),P6 是 FFI 和 GC(JIT 的边界扩展和内存安全兜底)。

P0-01 不在这条流水线上,它是流水线的"说明书"——把流水线为什么这样设计讲清。

## 四、关键设计取舍清单

LuaJIT 在 trace JIT 这条路线上,做了大量非显然的设计决策。每一个决策都有一个明确的"为什么",不这么做会怎样。这一节把全书出现的重要取舍汇总,每条一句话讲 why + 出处章节。一共 18 条。

| # | 设计决策 | 为什么这么做(不这么做的代价) | 出处 |
|---|---|---|---|
| 1 | **trace 而非 method** | 只录一条线性热路径,编译量比整个函数小一两个量级(省);假设可以做得非常乐观非常细(快)。method JIT 编译慢、要做复杂类型推断 | P0-01 §10 / P2-05 |
| 2 | **guard + snapshot 而非全面分支** | 大胆假设 + 每次检查,机器码本体尽情乐观,guard 兜底。比"把所有可能都编进带分支机器码"更精简 | P0-01 §7–§8 / P5-16 |
| 3 | **反向线性扫描寄存器分配(非图着色)** | trace 是线性的,区间结构简单,线性扫描 O(n log n) 够用且快。图着色虽更优但慢、对 trace 这种线性结构没必要 | P4-13 |
| 4 | **复制-替换做循环优化(非传统 LICM)** | 动态语言 guard 太密,PHI 节点和 guard 交织,传统 LICM 的不变性证明会破坏 guard 的 soundness。用复制-替换绕开 | P3-12 |
| 5 | **手写汇编 VM(`lj_vm.s`)** | 解释器是 JIT 的基座,每条字节码的 dispatch 都要省到极致(快);JIT 失败回退代价也要小(安全)。编译器生成的汇编达不到这种粒度 | P1-02 |
| 6 | **栈式字节码(非寄存器式)** | 栈式每条指令隐式操作栈顶,指令短、dispatch 表小,适合手写汇编优化。官方 Lua 用寄存器式是为了减少指令数,但 LuaJIT 用手写汇编把栈式也优化到极快 | P1-02 |
| 7 | **NaN-boxing(8 字节 TValue)** | 把类型 tag 编码进 double 的 NaN 空间,值用 8 字节而非官方 Lua 的 16 字节 tagged TValue。省一半内存、缓存友好(快) | P1-04 |
| 8 | **hotcount 是步长(递减计数)而非阈值累加** | 计数器从初值递减到 0 触发,每次只做一条减法指令(零额外开销)。比"累加 + 比较"省一条比较指令。HOTCOUNT_SIZE=64,pc>>2 哈希 | P1-03 |
| 9 | **dispatch 表模式切换(不改字节码)** | INTERP / RECORD / JIT 之间切换靠改 dispatch 表入口指针,字节码本身不动。保证三种模式跑的是同一段代码(sound) | P1-03 |
| 10 | **IR 用线性 SSA + REF_BIAS 偏移编码** | trace 是线性的,IR 指令按生成顺序线性排列,用 REF_BIAS 把 ref 分正负(常量负 / 指令正)。不需要 Sea-of-Nodes 的图状 IR(以形式换简单) | P2-07 |
| 11 | **类型窄化保 number 动态性** | 不强行把 number 拆成 int/double 两种类型(那破坏 Lua 语义),而是编译时乐观假设 int + overflow guard,失败回退 double 重算。类型推断不必非此即彼 | P2-08 |
| 12 | **fold 在线 + 哈希查表(规则内嵌)** | 常量折叠规则在 2.1 内嵌进 `lj_opt_fold.c`,由 `buildvm_fold.c` 生成 `lj_folddef.h`(2.0 有独立 def 文件)。在线 + 哈希做到低开销高频次 | P3-09 |
| 13 | **别名分析三值逻辑保守取向** | 两个引用"一定同一对象 / 一定不同 / 可能同",只要"可能同"就放弃优化(load 带 guard)。宁可少优化不可错优化(soundness 优先) | P3-10 |
| 14 | **sink 用完备副作用表 + 三道闸** | 分配消除很激进(短命 table 不分配),但必须 sound。完备副作用表 + 三道闸保证不消除有副作用的分配 | P3-11 |
| 15 | **asm_guardcc 在寄存器分配后发射** | guard 的 cmp+jcc 不在 IR 层处理,而在汇编生成时、寄存器分配确定后发射。因为 guard 检查的值要先知道在哪个寄存器 | P4-14 / P5-16 |
| 16 | **mcode 用独立可执行内存区(W^X),不由单 trace 释放** | 机器码要可执行,数据要可写,两者权限分开(W^X 防 shellcode 注入)。mcode area 是全局管理的,单 trace 销毁不释放内存(避免频繁 mprotect 抖动),留给后续 trace 复用 | P4-15 |
| 17 | **snapshot 用差量压缩 + slot→IR ref 映射** | 每 guard 一个 snapshot,但相邻 snapshot 只存差异(增量)。slot 映射到 IR ref,恢复时照 ref 取值填栈。预计算 + 差量压缩把 snapshot 开销压到极低 | P5-17 |
| 18 | **side trace 用热补丁(patchexit)而非重编译父 trace** | 父 trace 已经在跑,不能停;从退出点录新 trace 后,用 `lj_asm_patchexit` 改父机器码里那个 exit 的跳转目标,下次失败直接跳进 side trace。增量优化,不改父逻辑 | P5-18 |

这 18 条,是 LuaJIT 之所以是 LuaJIT 的骨架。把它们合起来,就是一个 trace JIT 编译器的全部关键决策。每一条都是为了应对某一股张力——快、安全、省——而做的工程权衡。

## 五、★双对照总表:LuaJIT vs 官方 Lua vs JVM/V8

全书每一章都有一栏 ★对照,两个对象:**官方 Lua**(切"有没有 JIT")和 **JVM / V8**(切"trace vs method")。这一节把 21 章的对照汇总成一张大表,按维度组织。读完这张表,你能一眼看清 LuaJIT 在 JIT 谱系里的位置。

### 对照一:LuaJIT vs 官方 Lua(有没有 JIT,加了什么)

| 维度 | 官方 Lua | LuaJIT(2.1 ROLLING) | 出处 |
|---|---|---|---|
| 整体定位 | 纯解释器 | 解释器 + trace JIT + 增量 GC | P0-01 |
| 字节码形式 | 寄存器式 | 栈式 | P1-02 |
| 解释器实现 | C 写的 `luaV_execute` | 手写汇编 `lj_vm.s` | P1-02 |
| 热点检测 | 无 | hotcount 计数器(pc>>2 哈希,64 槽) | P1-03 |
| 值表示 | 16 字节 tagged TValue | 8 字节 NaN-boxing TValue | P1-04 |
| Table | tagged 数组 | NaN-boxing + colo 同置 + LJ_KEYINDEX(2.1) | P1-04 |
| 编译单位 | 不编译 | 一条线性 trace | P0-01 / P2-05 |
| IR | 无中间层 | 线性 SSA(`IRIns` + REF_BIAS) | P2-07 |
| 类型处理 | 运行时动态判(只有 number) | 类型窄化 number→int(乐观)+ guard | P2-08 |
| 优化 pass | 无 | fold / mem / sink+dce / loop+split 四道 | P3 全篇 |
| 寄存器分配 | 无(纯栈式解释) | 反向线性扫描 | P4-13 |
| 机器码生成 | 无 | `lj_asm.c` 三层翻译 | P4-14 |
| 可执行内存 | 无 | W^X mcode area + 多后端 | P4-15 |
| 假设失效 | —(每步现场判,不会失效) | side exit + snapshot 退回解释器 | P5-16 / P5-17 |
| 失败后恢复 | — | snapshot restore + side trace | P5-17 / P5-18 |
| 代码衔接 | 函数调用 / 循环回边 | TraceLink 9 种织成跳转网 | P5-19 |
| 调 C | Lua C API(慢) | FFI 直调(可被 JIT) | P6-20 / P6-21 |
| GC | 增量 GC(无 JIT 协作) | 增量 GC + 追踪 trace/cdata/mcode | P6-21 |

这张表右边那一列,就是"加上 JIT 之后多了什么"。官方 Lua 那一列大半是"无",正是这些"无"让它慢——但慢得诚实、简单、永远正确。LuaJIT 把这些"无"全部补成了"有",代价是实现了整整一套编译器(4.9 万行 C + 手写汇编)。

### 对照二:LuaJIT vs JVM/V8(trace vs method,JIT 两大流派)

| 维度 | LuaJIT(trace JIT) | JVM / V8(method JIT) | 出处 |
|---|---|---|---|
| 编译单位 | 一条线性热路径(trace) | 整个函数(method) | P0-01 / P2-05 |
| 编译时机 | 运行时,热点触发 | 运行时,方法热点触发 | P1-03 |
| 编译开销 | 低(trace 短,几十条指令) | 较高(整函数,可能上千指令) | P2-05 |
| IR 形式 | 线性 trace 级 SSA | Sea-of-Nodes / 方法级图状 SSA | P2-07 |
| IR 构建 | 运行时录制(边执行边录) | 从字节码/AST 静态构建 | P2-06 |
| 类型处理 | 乐观假设 + guard(基于运行时观察) | 类型推断 / profile / hidden class | P2-08 |
| 失败处理 | side exit + snapshot(局部退出) | deoptimization(整个方法退回) | P5-16 |
| 失败后恢复 | 退回解释器,退出点热则录 side trace | 退回解释器,可能基于新 profile 重编译整个方法 | P5-17 / P5-18 |
| 寄存器分配 | 线性扫描 | 图着色(V8 较新也用线性扫描,但传统是图着色) | P4-13 |
| 分配消除 | allocation sinking | 逃逸分析 + 标量替换 | P3-11 |
| 循环优化 | 复制-替换(非 LICM,绕开 guard 密集) | method 级 LICM / 向量化 | P3-12 |
| 代码衔接 | TraceLink 9 种 + side trace 树 | 方法间内联 + 调用链接 | P5-19 |
| 一次编译覆盖 | 一条路径 | 方法内所有分支 | P2-05 |
| 生命周期终态 | GC 回收 / flush | method 卸载 / code cache LRU | P2-05 |
| FFI / 外部调用 | FFI 调用能被 JIT 内联(`lj_crecord`) | JNI 不能被 JIT 内联 | P6-21 |
| GC 协作 | 增量 GC + JIT 显式标记 trace | 分代 / 并发 GC | P6-21 |
| 实现规模 | 4.9 万行 C + 手写汇编,一人(Mike Pall)实现 | 几百万行,大团队 | P6-21 |

这张表是全书最有讲头的一条对照线。trace vs method 是 JIT 的两大流派根本分歧:trace 对动态语言的热点更聚焦、编译更快、假设更乐观,代价是退出处理更复杂(snapshot、side trace 树);method 覆盖全面、假设相对保守,代价是编译慢、要做大量类型推断。LuaJIT 选 trace,正因为它小到能被一人独立实现、小到能被一本书一行行讲清——这是它作为"最佳 trace JIT 教学样本"的根本理由。

### 两张表合读

把上面两张表并起来看:LuaJIT 相对官方 Lua,是"从无到有加上一套编译器";相对 JVM/V8,是"在同一套 JIT 思想里选了不同的流派"。这两条对照线交织在一起,定义了 LuaJIT 的全部独特性——一个**小而完整的 trace JIT**,既有解释器保底的安全,又有机器码的速度,还简单到能被读懂。

## 七、收尾:回到 P0-01 的命题

写到这里,我们把 21 章串成了一张图。最后,回到 P0-01 开篇那个命题,重新讲一遍——讲得更完整。

P0-01 的命题是:

> **Lua 是动态类型语言,编译时根本不知道一个变量下一秒是什么类型。这种语言,怎么可能安全地编译成机器码?**

这本书用 21 章回答了这个问题。答案的骨架,就是开头主线那一句话:

> **乐观假设 + 运行时检查(guard)+ 失败可回退(side exit / snapshot),让动态语言也能享受机器码的速度。**

但这句骨架,在 21 章里被展开成了具体的机制、具体的源码、具体的不变式。读完这本书,你应该在脑子里建立这样几层认知:

**第一层:动态语言能 JIT,不是因为绕开了类型未知,而是因为换了个编译时机。** 不在编译时编译(那时类型未知),而在运行时、看到真实类型之后再编译(P0-01 §5)。这是 JIT 区别于 AOT(提前编译)的根本。LuaJIT 的 trace 录制(P2-06),就是"运行时观察真实类型 + 基于观察做乐观假设"的具体落地。

**第二层:乐观假设之所以敢乐观,是因为有 guard 兜底。** 每一个假设都配一条运行时检查,检查通过假设成立、结果对,检查不通过当场破产、不会带着错往下算(P0-01 §7 / P5-16)。guard 是"乐观"和"安全"之间的契约。

**第三层:guard 失败之所以不致命,是因为有 snapshot + 解释器保底。** snapshot 事先记录好机器码状态到解释器栈的映射,失败时照表恢复,退回那个完整正确的解释器(P0-01 §8 / P5-17)。解释器是 JIT 的安全网——它让 JIT 敢于激进,因为最坏情况只是退回去慢一下。

**第四层:热点之所以越跑越快,是因为有 side trace + trace 链接。** 经常失败的退出点会被录成新 trace,trace 之间用 9 种链接方式织成跳转网(P5-18 / P5-19)。程序跑得越久,trace 树越密,退回解释器的次数越少——这是 JIT"越跑越快"的根源。

**第五层:整套机制的 soundness,建立在"解释器永远正确"这个不变式上。** 机器码产生的结果,要么和解释器完全一样(guard 全过),要么退回解释器(guard 触发),它永远不会比解释器更错(P0-01 §9)。这是 LuaJIT 比 Lua 快几十倍却保证语义完全一致的根基。

把这五层合起来,就是一个完整的 trace JIT 编译器的认知模型。LuaJIT 没有魔法——trace 是"录制实际跑的那条",guard 是"每次都检查",snapshot 是"事先记录映射表",side trace 是"从失败点重新录",FFI 录制是"基于类型表生成精确 IR",GC 协作是"显式标记所有 JIT 造的对象"。每一步都有明确的理由、精确的源码、严谨的不变式。

LuaJIT 比 V8 / HotSpot 小一两个量级,正因为它选择的 trace 路线足够简单——简单到能被一个程序员(Mike Pall)独立实现,简单到能被一本书一行行讲清。但这个"简单"的设计,却能跑出比纯解释器快几十倍的速度。这是工程上的优雅,也是这本书想带你看到的整体认知:

> **好的设计,是把复杂的权衡(快 vs 安全 vs 省),用简单的机制(trace + guard + side exit)解决。**

这就是"把动态执行安全变成机器码"这句话的全部含义。21 章讲完,你看到的不是一个黑箱,而是一台每一个齿轮都看得清的机器——它怎么发现热点、怎么录路径、怎么做乐观假设、怎么插检查、怎么生成机器码、怎么安全退出、怎么越跑越快。每一个环节你都知道为什么这么设计、不这么设计会怎样、源码在哪一行。

这就是读完这本书,应该得到的整体认知。
