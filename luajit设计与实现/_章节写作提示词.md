# _章节写作提示词(给写作 agent)

本文件是写每一章前的标准纪律。每个写作 agent 动笔前必须读完并照做。

---

## 一、写每章前必填的五要素

```
本章主线落点:这章在"把动态执行安全变成机器码"里负责讲哪一环(trace/guard/side exit/优化/后端)?
二分法位置:解释器侧 / JIT 侧?
对照栏对象:官方 Lua(哪个点) + JVM/V8(哪个点)?
源码版本与锚点:LuaJIT 2.1.ROLLING,本章涉及文件与起始函数(带 file:line)
基调:纯直球,不用比喻
```

## 二、固定四段结构(★深入浅出·第一性原理·不吝篇幅)

> 用户明确要求:"直面第一性原理,尽量详细,深入浅出,不要吝啬篇幅"。纯直球不用比喻 **≠** 精简快上结构体。深入浅出靠第一性原理推导 + 最小例子反复演示。

1. **这章解决什么问题(第一性原理推导,占大头)**:不假设读者已知。从最基础的事实一步步推导到本章主题。每个概念**先讲动机**(为什么需要它?不这样会怎样?)再给定义。用问句引导("那怎么办?""为什么?")。用一个最小代码片段在多个阶段反复走。这一段可以很长(几千字),把读者"扶"到能看懂源码的程度。
2. **源码怎么实现**:读者已经被第①段扶上来,这里用源码**印证**前面讲清的道理。贴关键结构体/函数(带 `file:line`),逐字段讲为什么这么设计(对应第①段的哪个动机)。慢,不跳。
3. **为什么这样设计是 sound 的**:讲清楚这个设计为什么不丢正确性(guard 为什么不漏检查 / snapshot 为什么能完整恢复 / 窄化为什么不改变语义)。
4. **★对照(官方 Lua + JVM/V8)+ 回扣主线**:对照两个对象,点出 LuaJIT trace 路线的取舍;结尾回扣"把动态执行安全变成机器码"。

**篇幅硬要求:每章 12000–20000 字**(不是 8000–12000)。宁可详尽啰嗦,不要简略跳步。每个"为什么"讲透。

## 三、纯直球纪律

- 不用比喻。不写"想象……""好比……"。直接讲数据结构、算法、不变式、汇编。
- 讲设计靠第一性原理:这个机制解决什么问题?约束是什么?为什么是这个解?
- 讲实现靠真实源码:函数名、行号、字段、宏、汇编片段,贴出来走一遍。
- JIT 涉及汇编处,贴真实汇编片段并逐行解释(对照 IR 与机器码)。

## 四、源码引用规范

- 版本:**LuaJIT 2.1.ROLLING**(唯一主干,`深入浅出系列\luajit2\src\`)。
- 引用格式:`lj_asm.c:1200`(`lj_asm_trace`)。关键函数给 file:line。
- 代码块用三反引号标 `c` 或 `asm`。贴的代码必须与 2.1.ROLLING 逐字一致(可省略号跳过无关行,保留行必须真实)。
- 交付前 Grep/Read 核对每个行号。2.1 ROLLING 与老版(2.0/讲老版的资料)有差异处显式标注。

## 五、对照栏规范

- 每章有 ★对照栏,**双对照**:官方 Lua(承接前一本)+ JVM/V8(method JIT)。
- 官方 Lua 对照点:解释器(栈式 vs 寄存器式)、有没有 JIT、值/Table 表示异同。
- JVM/V8 对照点:trace vs method、IR 形式、寄存器分配、优化 pass、deoptimization(对应 LuaJIT 的 side exit)。
- 对照要具体,不要泛泛说"JVM 更复杂"。

## 六、行号核实清单(交付前自检)

- [ ] 每个 `file:line` 已 Grep/Read 核实存在于 LuaJIT 2.1.ROLLING
- [ ] 代码块与源码逐字一致
- [ ] 标点全角(代码/汇编/行号除外)
- [ ] 有 ★对照栏(官方 Lua + JVM/V8)
- [ ] 结尾回扣主线
- [ ] 2.1 ROLLING vs 老版差异已标注
