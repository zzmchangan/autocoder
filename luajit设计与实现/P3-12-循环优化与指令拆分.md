# P3-12 循环优化与指令拆分

> **本书主线**:把动态执行安全变成机器码。**二分法**:解释器侧 ↔ JIT 侧。本章是 **JIT 侧优化篇(P3)的收尾**——讲 trace 在变成机器码之前,最后两道优化 pass:**循环优化**(`lj_opt_loop`)与**指令拆分**(`lj_opt_split`)。循环优化处理"trace 多是循环"这个事实,让循环体更精简;指令拆分处理"32 位平台做不了 64 位运算"这个限制,把一条 64 位 IR 拆成两条 32 位 IR。**★对照**:官方 Lua(不做任何循环优化)+ JVM/V8(method 级 LICM/向量化 vs trace 级 loop 优化)。**源码**:LuaJIT 2.1.ROLLING。**基调**:纯直球,不用比喻;从第一性原理一步步推导。

## 引子:优化篇还差两件事

P3-09 讲了常量折叠,它吃掉 `x+0`、`x*1` 这类常量冗余。P3-10 讲了内存优化,它让前一次存储能被后一次加载直接复用(load/store forwarding)。P3-11 讲了分配消除和死代码消除,前者把"分配了却只赋值不读"的对象干脆不分配,后者删掉"算出来没人用"的指令。这三道 pass 合起来,已经把一条 trace 的 IR 压缩、精简了一大截。

但到此为止,还有两类问题没有处理。这两个问题在 trace 这种"只录一条线性路径"的形式下,各自需要一道专门的 pass。

**第一类:循环。** 回想 P0-01 §10 和 P2-07 §7 反复强调的一个事实——**绝大多数 trace 是循环**。热点是循环(一个 `for` 跑了一百万次,它的回边才是热点,才会触发录制),所以录制下来的 trace 几乎总是带一个回边、跑完一圈要跳回开头再跑。这条循环 trace 的 IR,P3-09~11 已经优化过一轮,但循环自身的特殊性还没被专门处理:循环体里有些计算每次循环结果都一样(循环不变量),它们本该只算一次,现在却每一圈都重新算;循环变量在回边前后的值,在 SSA 里还没被正确地合并成一个闭环。这两件事需要一道**循环优化 pass**——`lj_opt_loop`。

**第二类:64 位运算在 32 位平台上做不了。** 这一类听起来很具体,但非常实际。在 32 位 CPU(或软浮点目标)上,CPU 没有原生的 64 位加法、64 位乘法指令;可 IR 里偏偏有 64 位整数(`IRT_I64`/`IRT_U64`,FFI 的 `int64_t`/`uint64_t` 会产生)和 64 位浮点(`IRT_NUM`,软浮点目标上没有硬件 FPU)。后端汇编生成器(P4)要为每条 IR 选一条机器指令,遇到一条 `i64 ADD`(64 位整数加),它在 32 位 x86 上找不到对应的单条指令——它得用两条 32 位 `ADD` 加一条 `ADC`(带进位加)凑出来。如果把"一条 64 位 IR 怎么映射成多条 32 位机器指令"的逻辑全塞进后端,后端会变得极其臃肿:每条 64 位指令都要特判、都要分配寄存器对、都要处理进位传播。LuaJIT 的选择是:在 IR 层面先**拆**——把每条 64 位 IR 拆成"低 32 位部分 + 高 32 位部分"两条 32 位 IR(配一条 `IR_HIOP` 标记高位),拆完之后,后端只面对 32 位 IR,干干净净。这道 pass 就是 `lj_opt_split`。

所以这一章讲两道 pass,它们各自解决优化篇最后遗留的两个问题,共同把 IR 准备到"后端可以直接逐条翻机器码"的程度。下一章 P4-13 开始,就是把这些 IR 一条条分配寄存器、生成机器码。优化篇到此收束,后端篇由此开始。

我们先从更基础、也更核心的循环优化讲起——它是 LuaJIT 处理循环 SSA 的招牌技巧,也是 P2-07 §7 留下的"循环怎么闭环"那个问题的完整答案。

## §1 第一性原理:循环里为什么有浪费

在讲 LuaJIT 怎么优化循环之前,我们必须先把"循环里到底有什么浪费"这件事看清楚。不看清浪费,就看不懂优化在干什么。我们从最朴素的循环开始推导。

### 1.1 一个最朴素的循环

看这段几乎每个 JIT 教程都会用的循环:

```lua
local sum = 0
local step = 1
for i = 1, 1000000 do
  sum = sum + i + step
end
```

这个循环跑一百万次。我们假设它已经被录制成了一条 trace(因为回边触发热点),录下来的 IR 是循环体的一圈。在没有任何循环优化时,这一圈 IR 的机器码,会在每一圈循环里都执行一次。

现在盯着循环体 `sum = sum + i + step` 看三秒钟,问一个问题:**这一圈里,哪些东西每一圈都在变,哪些东西每一圈都不变?**

- `sum`:每一圈都在变(每轮加一个新值)。**变。**
- `i`:每一圈都在变(每轮加 step)。**变。**
- `step`:它在循环体外面被赋成 1,循环体里没有任何地方改它。**每一圈都是 1。不变。**

再看 `i + step` 这个子表达式。`i` 每圈变,`step` 每圈都是 1,所以 `i + step` 每圈的结果确实在变(`i` 变了)。**但是**,这里有一个更细的观察:`step` 是循环不变量,只有 `i` 在变。也就是说,`i + step` 这个加法,它的"被加数"每一圈都不一样(`i` 在涨),但它的"加数"每一圈都一样(`step` 永远是 1)。

那这个加法本身能省掉吗?不能——因为结果在变。但是,加法里有一个操作数(`step`)是常量性质的不变量,这个事实能不能被利用?比如,能不能把"`step` 是 1"这件事在循环外确定一次,循环体里就不用每次重新加载 `step`、判断它的类型?

这就是循环优化的核心动机:**识别循环里哪些东西不变,把和它们相关的开销尽量挪到循环外。**

### 1.2 循环不变量外提(LICM):最经典的循环优化

上面这个动机,有一个经典的、教科书式的名字:**LICM(Loop-Invariant Code Motion,循环不变量外提)**。它的定义是:

> **如果一条指令的计算结果,在整个循环的每一圈都一样(它的所有操作数都是循环不变量),那么这条指令可以挪到循环外面,只算一次,循环体里直接用这次算好的结果。**

举一个最直白的例子(伪 IR):

```
循环前:
  step = SLOAD slot=step        ← step 的值(假设循环里没人改它)

循环体(每一圈都执行):
  0001  ADD i, step             ← i + step
  ...
```

`step` 在循环体里不变。那 `SLOAD slot=step`(从栈槽加载 step)这件事,每一圈加载到的都是同一个值——它在循环外算一次就够了。LICM 会把这条 SLOAD 挪到循环前面:

```
循环前:
  step = SLOAD slot=step        ← 只算一次

循环体:
  0001  ADD i, step             ← 直接用循环前算好的 step
```

更激进一点:如果 `i + step` 里 `step` 是常量 1,而循环里 `i` 的初值也已知,有些编译器甚至能把 `i + 1` 这种"加常量"识别成"递增",用更便宜的递增指令。这些都是 LICM 家族做的事。

LICM 在静态语言(C、Fortran)的循环优化里立了大功。几十年来,教科书讲循环优化,第一个讲的就是 LICM。所以当你听到"LuaJIT 有循环优化 pass"时,自然会以为它做的就是 LICM。

### 1.3 为什么传统 LICM 对动态语言基本没用

但 LuaJIT 的循环优化 pass(`lj_opt_loop.c`),文件开头那段长达 70 行的注释(`lj_opt_loop.c:22-90`),开篇第一段就把这个预期打破了。我把它逐句翻译过来:

> 传统的循环不变量外提(LICM),把循环的指令分成"不变的"和"变的"两类。不变的指令被提到循环外面,循环体里只剩变的指令。
>
> **不幸的是,LICM 对编译动态语言基本没用。** IR 里有大量的 guard,而大多数后续指令都控制依赖于这些 guard。第一个不能外提的 guard,就会阻止后面所有指令的外提。

这段话信息量很大,我们逐层拆开。

第一层:**动态语言的 IR 里全是 guard。** 回想 P0-01 §7 和 P2-07 §3.2:LuaJIT 编译动态 Lua,靠的是"乐观假设 + guard 检查"。每一条从栈槽加载的 `SLOAD`、每一次加法的溢出检查 `ADDOV`、每一个比较 `LT/LE`,凡是基于"假设这里类型/值不变"的,都会带一个 `IRT_GUARD` 标志。一条循环 trace 里,可能有大大小小十几个 guard。这是动态语言 JIT 的常态——guard 是安全的基础,但它也意味着 IR 里几乎每条指令都挂着一个"失败就退出"的检查。

第二层:**大多数指令控制依赖于这些 guard。** 一条指令"控制依赖"于某个 guard,意思是"这条指令只有在那个 guard 成立时才该执行"。为什么?因为这条指令假设的前提,正是那个 guard 保证的东西。比如 `ADD i, step` 假设 `i` 和 `step` 都是整数——而"`i` 是整数"这件事,是由前面某条 SLOAD 的类型检查 guard 保证的;如果那个 guard 失败了(说明 i 不是整数),那这条 ADD 根本不该执行(因为它是按整数加法生成的机器码,对非整数会算错)。

第三层,也是最致命的一层:**第一个不能外提的 guard,就堵死了后面所有指令的外提。** 为什么?因为控制依赖是"传染"的。假设循环体里第一个 guard 是"检查 i 是整数",它不能外提(因为 i 在循环里变,每圈都要重新检查)。那么,所有依赖这个 guard 的指令——也就是所有用到 i 的指令——也都不能外提(因为它们在"i 不是整数"时本就不该跑,不能提到循环外只算一次)。而用到 i 的指令,它们的输出又会被后续指令用到……于是,从这个不能外提的 guard 出发,沿着数据依赖链往下,几乎所有指令都被牵连,统统不能外提。

结果就是:传统 LICM 在动态语言上,外提不了几条指令。循环体里的绝大多数指令,因为挂在 guard 的控制依赖链上,只能老老实实留在循环体里每圈重算。LICM 的收益,在动态语言上接近于零。

**这就是为什么 LuaJIT 不做传统 LICM。** 它走了完全不同的路。

### 1.4 LuaJIT 的路:展开两圈,让优化 pipeline 自动处理

LuaJIT 的循环优化思路,文件注释(`lj_opt_loop.c:34-41`)这样描述:

> 我们用一种特殊的展开形式,结合复制-替换(copy-substitution)和冗余消除:
>
> 录制的指令流,被用"替换了操作数"的方式,重新喂给编译器的优化 pipeline。替换表里填的,是"重新发射每条指令后返回的引用"。这件事可以边录边做,因为 IR 是严格的 SSA 形式——每个引用都在它被使用之前定义。

这段话很技术,我们先不钻细节。先用一句话抓住它的精神:

> **不显式做 LICM,而是把循环体复制一遍,让原本就写好的常量折叠/CSE/forwarding 这些优化 pipeline,自动识别出哪些是循环不变的、哪些是循环携带的,该外提的外提、该留的留。**

这个想法,第一次读会觉得"绕"。所以我们用一个具体的、纸上的演示,把它一步步走通。走完你就会恍然大悟:原来它巧妙地利用了 trace 的特性,把循环优化变成了一个"已经会做的事"的副产品。

### 1.5 纸上演示:展开两圈怎么自动外提

我们用 §1.1 那个 `sum = sum + i + step` 循环,把 `i + step` 这个子表达式单独拎出来演示(为了聚焦,忽略 sum 的累加)。

**录制阶段**:录制器录了一圈循环体。假设录到的 IR(极度简化)是:

```
(录制段 / pre-roll,引用号用 0xxx)
0001  SLOAD slot=i          ← 加载 i
0002  SLOAD slot=step       ← 加载 step
0003  ADD 0001, 0002        ← i + step  (这就是那个"step 不变、i 变"的子表达式)
0004  SSTORE slot=tmp, 0003 ← 存回 tmp(假设循环体用到这个和)
...   (循环回边相关:更新 i、边界检查)
0007  LOOP                  ← 循环回边标记
```

注意:这 0001~0007 都是录制阶段录下来的,它们组成 trace 的"pre-roll"(预转)段。机器码生成后,这一段会被执行**一次**(在进入循环之前先跑一圈,把状态带上正轨)。

现在 LuaJIT 的循环优化要做什么?它要在这段录制的 IR 后面,**再追加一段"循环体"的 IR**,这段循环体会在每一圈循环里执行。怎么追加?它的办法是:**把录制段复制一遍,但操作数用替换表替换。**

**复制阶段**:对录制段的每一条指令,用一张"替换表" `subst[]` 替换它的操作数,然后重新喂给优化 pipeline(FOLD/CSE)。替换表的语义是:`subst[旧引用] = 新引用`——原指令 0001 复制后,变成一个新引用,记进 `subst[0001]`。

关键是:复制时,重新喂给 pipeline,pipeline 会做它最擅长的事——**常量折叠和 CSE**。如果复制后的某条指令,经过折叠/CSE 后,发现它的结果和原来录制段的某条指令**完全一样**(操作数相同),pipeline 会直接返回那条旧指令的引用,不真正发射新指令。这时 `subst[旧引用] = 旧引用`——复制后的这条指令"折叠回"了原指令,等于没有复制。

**这就是不变量自动外提的机制**:如果一条指令的所有操作数都是循环不变的(复制后操作数没变),pipeline 就让它折叠回原指令——这条指令实际上只在 pre-roll 段存在一次,循环体里没有它的副本,于是它"自动"被提到了循环外(pre-roll 就在循环外)。

走一遍那个 `i + step` 的例子。复制阶段开始,替换表初始化为空,逐条复制:

复制 0001 `SLOAD slot=i`:
- 操作数是字面量 `slot=i`(不是引用),不需要替换。
- 重新喂给 pipeline。pipeline 看到一条新的 SLOAD,但 i 在循环里是变的——这条 SLOAD 在循环体里是新的、不同的(它加载的是"这一圈的 i")。所以 pipeline 真正发射它,得到新引用,比如 0008。`subst[0001] = 0008`。

复制 0002 `SLOAD slot=step`:
- 操作数 `slot=step`,字面量,不替换。
- 重新喂给 pipeline。pipeline 沿着 `IR_SLOAD` 的同 opcode 链(P2-07 §3.3 讲过 `prev` 字段串的链)往前找,发现前面已经有一条一模一样的 `SLOAD slot=step`(就是 0002)——而且 step 是循环不变量,这一圈加载到的 step 和上一圈完全一样。**CSE 命中**:pipeline 直接返回 0002 的引用,不发射新指令。`subst[0002] = 0002`。
- **这一步,就是 step 这个循环不变量被自动外提的瞬间**:循环体里不会再有一条加载 step 的 SLOAD,它复用了 pre-roll 段的 0002。

复制 0003 `ADD 0001, 0002`:
- 替换操作数:`op1 = subst[0001] = 0008`(复制后的新 i),`op2 = subst[0002] = 0002`(原 step,因为折叠回去了)。
- 操作数变了(op1 从 0001 变成 0008),重新喂给 pipeline。pipeline 看到 `ADD 0008, 0002`,沿 `IR_ADD` 链找——有没有一条 `ADD 0008, 0002`?没有(0008 是新的 i,和上一圈的 i 不同)。所以真正发射新指令,得到引用 0009。`subst[0003] = 0009`。
- 这条 ADD 留在循环体里,因为 i 每圈变。

复制完成后,IR 长这样(简化):

```
(pre-roll,循环外,执行一次)
0001  SLOAD slot=i
0002  SLOAD slot=step       ← step 只在这里加载一次
0003  ADD 0001, 0002
0004  SSTORE slot=tmp, 0003
...
0007  LOOP

(循环体,每圈执行)
0008  SLOAD slot=i          ← 复制后的新 i(注意:step 没有复制,被外提了)
0009  ADD 0008, 0002        ← 复制后的 ADD,用新 i 和外提的 step
...   (循环体的其他指令)
```

**看明白了吗?** step 这个循环不变量,没有任何一行代码显式地"判断它不变然后外提"。是复制-替换 + CSE 这套组合拳,让"step 这条 SLOAD 复制后操作数没变 → CSE 命中 → 折叠回原指令"这件事自然发生,step 就被"外提"到了 pre-roll。**整个外提过程,完全是已有优化 pipeline 的副产品。**

这就是 LuaJIT 循环优化的精髓。我们再用一节,把它的几个妙处讲透,然后才去看源码。

### 1.6 这个方法的三个妙处

文件注释(`lj_opt_loop.c:54-89`)总结了这套"展开 + 复制替换 + pipeline 复用"方法的几个好处。挑最重要的三条讲。

**妙处一:控制依赖被隐式保留,而且不变 guard 也被外提。**

回忆 §1.3 的难题:传统 LICM 因为 guard 的控制依赖而寸步难行。LuaJIT 的方法怎么绕开的?

复制阶段把录制段的每一条指令——包括 guard——都复制一遍。复制后,guard 也走 pipeline。如果一条 guard 检查的是循环不变量(比如"检查 step 是整数",step 不变),那么复制后这条 guard 的操作数也没变,CSE 让它折叠回原 guard——这条 guard 被外提到了 pre-roll,**循环体里不再检查它**。文件注释(`lj_opt_loop.c:73-75`)专门强调这一点:

> 由于控制依赖被隐式保留,CSE 也适用于各种 guard。主要的好处是,**所有不变的 guard 也都能被外提**。

这一条极其关键。传统 LICM 不敢外提 guard(因为外提 guard 意味着"循环里不再检查这个条件",万一条件在循环里变了怎么办?)。但 LuaJIT 的逻辑是:如果 guard 检查的东西是循环不变的(操作数每圈都一样),那么"在循环外检查一次"和"每圈都检查"在语义上完全等价——检查的东西不变,检查结果就不变,检查一次就够了。所以不变 guard 可以安全外提。这是 LuaJIT 循环优化一个不显眼但很重要的收益:它把循环体里的 guard 数量也压减了,循环体更精简、更快。

而变的 guard(检查 i 这种每圈变的东西),复制后操作数变了,不会被 CSE 命中,老老实实留在循环体里每圈检查。控制依赖被保留得严严实实——循环体里的指令,依然只在它们的 guard 成立时才执行。没有任何一个 guard 被错误地外提。

**妙处二:所有优化 pipeline 白嫖。**

复制段重新喂给 FOLD/CSE/FWD,这些 pass 本来就写好了(P3-09、P3-10),循环优化不需要重新实现常量折叠、CSE、forwarding 的任何逻辑。`lj_opt_loop.c` 全文只有 453 行,其中真正的循环逻辑(`loop_unroll`)不到 120 行——因为它把所有重活都甩给了已有的 pipeline。文件注释(`lj_opt_loop.c:62-65`)说:

> 这个方法最妙的一点是,编译器 pipeline 的所有优化(FOLD、CSE、FWD 等等)都能被复用,只有很少的限制(比如不应该跨循环携带依赖做折叠)。

这个"限制"是指:不能把一个循环携带依赖(每圈变的值)折叠成不变量——那会算错。LuaJIT 在 pipeline 里加了一个小约束(下一节看源码怎么做的),保证不会跨循环携带依赖误折叠。

**妙处三:循环携带依赖靠 PHI 显式建模,PHI 在循环体下方发射。**

循环携带依赖(loop-carried dependency)是"每圈变的、且每一圈用到上一圈值的"依赖。最典型:`sum = sum + i`,这一圈的 sum 依赖上一圈的 sum。这种依赖在 SSA 里(P2-07 §7)必须用 PHI 节点显式合并。

LuaJIT 的循环优化,在复制阶段顺手收集所有潜在的 PHI 候选(复制后引用又指回录制段的那些值),复制结束后调 `loop_emit_phi` 发射 PHI 指令。这个我们 §3 专门讲。这里只先点出:PHI 的处理和循环优化是同一次 pass 完成的,它们共享同一套复制-替换机制。

把三个妙处合起来,LuaJIT 的循环优化可以一句话总结:

> **把循环体复制一遍,让已有的 FOLD/CSE/FWD pipeline 自动区分"循环不变的"(折叠回 pre-roll,等于外提)和"循环携带的"(留在循环体,用 PHI 闭环)。不显式做 LICM,却达到了 LICM 的效果,还顺带外提了不变的 guard。**

理解了这个精髓,源码就很好读了。下一节我们钻进 `lj_opt_loop.c`,逐函数印证。

## §2 源码印证:lj_opt_loop.c 怎么落地

P2-07 §7 已经从概念上讲过 `lj_opt_loop` 的复制-替换。这一节我们补全 P2-07 没细讲的工程细节:它怎么管理替换表、怎么处理 snapshot、怎么处理类型不稳定、失败时怎么回滚。每一处都对应 §1 讲的某个动机。

### 2.1 入口与保护调用:lj_opt_loop

先看 `lj_opt_loop.c` 的入口函数(`lj_opt_loop.c:415`):

```c
/* Loop optimization. */
int lj_opt_loop(jit_State *J)
{
  IRRef nins = J->cur.nins;
  SnapNo nsnap = J->cur.nsnap;
  MSize nsnapmap = J->cur.nsnapmap;
  LoopState lps;
  int errcode;
  lps.J = J;
  lps.subst = NULL;
  lps.sizesubst = 0;
  errcode = lj_vm_cpcall(J->L, NULL, &lps, cploop_opt);
  lj_mem_freevec(J2G(J), lps.subst, lps.sizesubst, IRRef1);
  if (LJ_UNLIKELY(errcode)) {
    lua_State *L = J->L;
    if (errcode == LUA_ERRRUN && tvisnumber(L->top-1)) {  /* Trace error? */
      int32_t e = numberVint(L->top-1);
      switch ((TraceError)e) {
      case LJ_TRERR_TYPEINS:  /* Type instability. */
      case LJ_TRERR_GFAIL:  /* Guard would always fail. */
	/* Unrolling via recording fixes many cases, e.g. a flipped boolean. */
	if (--J->instunroll < 0)  /* But do not unroll forever. */
	  break;
	L->top--;  /* Remove error object. */
	loop_undo(J, nins, nsnap, nsnapmap);
	return 1;  /* Loop optimization failed, continue recording. */
      default:
	break;
      }
    }
    lj_err_throw(L, errcode);  /* Propagate all other errors. */
  }
  return 0;  /* Loop optimization is ok. */
}
```

(`lj_opt_loop.c:415-447`)

逐段讲:

**记录回滚点**。函数一进来,先把三个"当前进度"存起来:`nins`(IR 指令数)、`nsnap`(snapshot 数)、`nsnapmap`(snapshot map 大小)。为什么?因为循环优化要往 IR 里追加大量指令(复制段)、要改 snapshot,如果中途失败(PHI 太多、类型不稳定),得**回滚**到这个点,把追加的东西全撤掉,当作循环优化没发生。回滚逻辑在 `loop_undo`(`lj_opt_loop.c:384`),它把 `J->cur.nins/nsnap/nsnapmap` 恢复,再调 `lj_ir_rollback(J, ins)` 把 IR 数组里 `ins` 之后的指令作废,还要清掉循环优化加上的 PHI/mark 标志。这是"失败可回退"在优化阶段的体现——循环优化失败,绝不留下半成品 IR。

**保护调用**。真正的循环优化在 `cploop_opt`(`lj_opt_loop.c:407`),它通过 `lj_vm_cpcall` 调用。`lj_vm_cpcall` 是 Lua 的"受保护调用"(类似 pcall):如果 `cploop_opt` 里抛了 trace error(比如 `lj_trace_err(J, LJ_TRERR_PHIOV)`),这个 error 会被 `lj_vm_cpcall` 捕获,变成一个 errcode 返回,而不是直接 longjmp 出去。这样 `lj_opt_loop` 能在 errcode 里判断错误类型,决定怎么处理。这是 LuaJIT 用 Lua 自身的异常机制做 JIT 编译错误处理的惯用手法。

**失败的分类处理**。errcode 非 0 时,分两种 trace error 特殊处理(`lj_opt_loop.c:432-433`):

- `LJ_TRERR_TYPEINS`(persistent type instability,持久类型不稳定):循环变量在两圈之间类型不一致(一圈整数一圈浮点之类)。
- `LJ_TRERR_GFAIL`(guard would always fail,guard 必然失败):折叠器(P3-09)发现某个 guard 的条件在编译时就已经能判定为假(比如 `x < 0` 而 x 已知是 5),这种 guard 留着必失败,等于这条 trace 白录。

这两个错误的处理是:**退回录制,多展开几圈试试**。`--J->instunroll < 0` 是计数器(`lj_jit.h:120` 默认 `instunroll = 4`,即最多重试 4 次)。为什么多展开能修?文件注释里举了个例子("a flipped boolean",翻转的布尔):有些循环,第一圈和第二圈的类型/值表现不同(比如第一圈走冷分支、第二圈才走热分支),多录几圈,让录制器看到稳定后的状态,类型不稳定就消失了。`loop_undo` 回滚掉失败的循环优化,然后 `return 1` 告诉调用者"循环优化失败,请继续录制"。调用者是 `lj_trace.c:727`,它收到返回值 1 后,把状态退回 `LJ_TRACE_RECORD`,继续录:

```c
	if (lj_opt_loop(J)) {  /* Loop optimization failed? */
	  J->cur.link = 0;
	  J->cur.linktype = LJ_TRLINK_NONE;
	  J->loopref = J->cur.nins;
	  J->state = LJ_TRACE_RECORD;  /* Try to continue recording. */
	  break;
	}
```

(`lj_trace.c:727-733`)

这是 trace JIT 的容错:循环优化失败不等于 trace 作废,先试多展开几次,实在不行(`instunroll` 用完)才彻底放弃走 `lj_err_throw`。`LJ_MAX_PHI = 64`(`lj_def.h:86`)是另一道硬限制——一条循环 trace 最多 64 个 PHI,超过报 `LJ_TRERR_PHIOV`(PHI 溢出),这种是结构性太复杂,多展开也救不回来,直接 throw。

### 2.2 loop_unroll:复制-替换的主循环

`cploop_opt` 调 `loop_unroll`(`lj_opt_loop.c:265`),这是循环优化的主体。我们分段读。

**第一段:准备替换表、发射 LOOP 标记。**

```c
  /* Allocate substitution table.
  ** Only non-constant refs in [REF_BIAS,invar) are valid indexes.
  */
  invar = J->cur.nins;
  lps->sizesubst = invar - REF_BIAS;
  lps->subst = lj_mem_newvec(J->L, lps->sizesubst, IRRef1);
  subst = lps->subst - REF_BIAS;
  subst[REF_BASE] = REF_BASE;

  /* LOOP separates the pre-roll from the loop body. */
  emitir_raw(IRTG(IR_LOOP, IRT_NIL), 0, 0);
```

(`lj_opt_loop.c:276-286`)

`invar = J->cur.nins` 记录"录制段的边界"——录制到此为止,已经录了 `nins` 条指令(从 `REF_FIRST` 到 `nins-1`)。这个 `invar` 是个关键边界:复制段只复制 `invar` 之前的指令;复制后新发射的指令,引用号都 `>= invar`。后面 PHI 收集时,判断"是不是循环携带依赖"就靠"复制后的引用是不是 `< invar`(指回录制段)"。

替换表 `subst` 是个 `IRRef1`(16 位)数组,大小是 `invar - REF_BIAS`。为什么减 `REF_BIAS`?回忆 P2-07 §4.2:LuaJIT 的引用号从 `REF_BIAS`(=0x8000=32768)开始往上。`subst` 数组从 0 开始索引,但引用号从 32768 开始,所以做一个指针偏移 `subst = lps->subst - REF_BIAS`,这样 `subst[32768]` 就访问到数组的第 0 个元素——用引用号直接当下标,省去减法。这是个常见的 C 技巧(负偏移指针)。

`subst[REF_BASE] = REF_BASE`:`REF_BASE`(=32768)是栈基址(P2-07 §4.6),它在循环里不变(栈基址不会变),所以映射到自身。这是替换表的初始种子。

然后 `emitir_raw(IRTG(IR_LOOP, IRT_NIL), 0, 0)` 发射 `IR_LOOP` 指令。注意它带 `IRTG`(guard 标志)——`IR_LOOP` 是个特殊的标记指令,它本身不生成实际计算,但在后端标记"循环回边在这里",汇编生成器靠它定位循环体的起止。`emitir_raw` 是直接发射、不经过 FOLD pipeline(因为 LOOP 不该被折叠)。发射后,`IR_LOOP` 这条指令的引用号就是 pre-roll 和循环体的分界线——它之前的指令是 pre-roll,之后是复制的循环体。

**第二段:扩展 snapshot 缓冲区。**

```c
  /* Grow snapshot buffer and map for copy-substituted snapshots.
  ** Need up to twice the number of snapshots minus #0 and loop snapshot.
  ** Need up to twice the number of entries plus fallback substitutions
  ** from the loop snapshot entries for each new snapshot.
  ** Caveat: both calls may reallocate J->cur.snap and J->cur.snapmap!
  */
  onsnap = J->cur.nsnap;
  lj_snap_grow_buf(J, 2*onsnap-2);
  lj_snap_grow_map(J, J->cur.nsnapmap*2+(onsnap-2)*J->cur.snap[onsnap-1].nent);
```

(`lj_opt_loop.c:288-296`)

循环优化不只复制 IR,还要**复制 snapshot**。为什么?因为 pre-roll 段的每个 snapshot(对应每个 guard 的退出点),在循环体里也要有一个对应的 snapshot(循环体里的 guard 也要能退出)。如果不复制,循环体里的 guard 失败时没有 snapshot 可用,side exit 就没法恢复解释器状态(P5-17 讲 snapshot 恢复)。

复制 snapshot 会让 snapshot 数量最多翻倍(每个 pre-roll 的 snapshot,循环体里可能对应一个),所以 `lj_snap_grow_buf(J, 2*onsnap-2)` 预留两倍空间。注释里的 "-2" 是因为 #0 snapshot(根 trace 的空 snapshot)和 loop snapshot(循环回边的 snapshot)不需要复制。`lj_snap_grow_map` 同理,预留 snapshot map 的空间。注释的 "Caveat" 提醒:这两个调用可能 realloc `J->cur.snap` 和 `J->cur.snapmap`,所以之前拿的指针会失效——这也是为什么后面访问 snapshot 都用 `&J->cur.snap[...]` 重新取地址,不缓存指针。

snapshot 复制的细节(`loop_subst_snap`,`lj_opt_loop.c:208`)比较琐碎,核心是:对老 snapshot 的每个条目,用 `subst[]` 替换它的引用(指向复制后的新引用),合并 loop snapshot 的条目(循环体里新增的栈槽),生成新 snapshot。这一节我们先不钻 snapshot 复制的每个分支,抓住"snapshot 也被复制替换,和 IR 同步"这个要点就够。P5-17 讲 snapshot 恢复时再回头看。

**第三段:逐条复制录制段的指令。**


这是 §1.5 纸上演示对应的真实代码。逐行对应:

- `for (ins = REF_FIRST; ins < invar; ins++)`:遍历录制段的每一条指令(从 `REF_FIRST`=32769 到 `invar-1`)。
- `if (ins >= osnap->ref) loop_subst_snap(...)`:如果当前指令越过了某个 snapshot 的边界(`osnap->ref` 是这个 snapshot 对应的 IR 引用),就先复制替换那个 snapshot。这保证 snapshot 和 IR 同步推进。
- **替换操作数**:`op1 = ir->op1; if (!irref_isk(op1)) op1 = subst[op1];`——对每个操作数,如果不是常量(`irref_isk`,P2-07 §4.3 讲过 `< REF_BIAS` 判断),就用 `subst[]` 替换。这是"复制-替换"的"替换"部分。
- **判断是否不变**:`if (irm_kind(...) == IRM_N && op1 == ir->op1 && op2 == ir->op2)`——如果这条指令的模式是 `IRM_N`(无副作用的普通指令,比如 ADD),**而且替换后两个操作数都没变**(说明它的操作数都是循环不变的),那就走 shortcut:`subst[ins] = ins`,直接指向原指令,不复制。**这一行,就是 §1.5 说的"不变量折叠回原指令、被外提"在代码里的落点。** 注意 shortcut 只对 `IRM_N` 模式的指令适用——有副作用的指令(store、guard)即使操作数不变,也不能简单 shortcut,因为它们有顺序敏感的副作用,需要走 pipeline 正确处理。
- **否则重新喂给 pipeline**:`emitir(ir->ot & ~IRT_ISPHI, op1, op2)`。`emitir` 是 `lj_opt_loop.c:96` 定义的本地宏,等于 `lj_ir_set` + `lj_opt_fold`——把替换后的指令送进 FOLD/CSE pipeline。`& ~IRT_ISPHI` 把原指令的 ISPHI 标志清掉(复制时不该带 PHI 标志,PHI 标志由后面的 PHI 收集阶段统一设)。pipeline 返回新引用 `ref`,`subst[ins] = ref`。

`IRType1 t = ir->t;` 这行注释很重要:`emitir` 可能扩容 IR 数组(P2-07 §4.4 的 `lj_ir_growtop`),一旦扩容,`ir` 指针(指向旧数组)就失效了。所以先把 `ir->t`(类型)取出来存到局部变量 `t`,后面用 `t` 而不是 `ir->t`。这是 C 里"调用可能 realloc 的函数前,先保存指针指向的数据"的常见防御。

**第四段:PHI 候选收集。**

pipeline 返回 `ref` 后,紧接着判断它是不是潜在的 PHI 候选(代码在第三段的 `else` 分支里):


(`lj_opt_loop.c:332-371`,有删节)

这段是 §1.6 妙处三(PHI 显式建模)的代码。逻辑:

- `if (ref != ins)`:复制后的引用 `ref` 和原引用 `ins` 不同(说明这条指令没有被 shortcut 外提,它是变的)。
- `if (ref < invar)`:**复制后的引用,指向了录制段范围内**(`ref < invar`)。这是循环携带依赖的标志——为什么?因为复制段的指令引用号都 `>= invar`(它们是新发射的),如果 `ref < invar`,说明 pipeline 把这条指令折叠回了**录制段的某条旧指令**。这意味着:循环体里这个值,等于 pre-roll 段某个值——而这个 pre-roll 段的值,在循环里会被更新(它是循环携带的)。所以这是个潜在的 PHI。
- 收集 PHI:`irt_setphi(irr->t)` 给这个 ref 标上 `IRT_ISPHI` 标志(P2-07 §3.2 讲过这个标志位),然后 `phi[nphi++] = ref` 把它加进 PHI 候选数组。`if (nphi >= LJ_MAX_PHI)` 检查 PHI 数量上限(64),超了报 `LJ_TRERR_PHIOV`。

**类型不稳定检查**。同一个循环携带依赖,在 pre-roll 段的类型(`t`,原指令类型)和在复制段的类型(`irr->t`,pipeline 返回的 ref 的类型)可能不同。比如 `sum` 第一圈是整数,第二圈因为某次加法溢出变成了浮点。这就是 §2.1 提到的 `LJ_TRERR_TYPEINS` 的来源。代码处理:

- 两边都是整数(`irt_isinteger(t) && irt_isinteger(irr->t)`):OK,`continue`。整数之间(比如 int 和 i32)不算类型不稳定。
- pre-roll 是浮点、复制段是整数(`irt_isnum(t) && irt_isinteger(irr->t)`):插一个 `IR_CONV` 把整数转成浮点(`IRCONV_NUM_INT`),让两边统一成浮点。注意这个 CONV **不带 guard**(`IRTN`,N 表示非 guard),因为 int→num 是无损的,不需要检查。
- pre-roll 是整数、复制段是浮点(`irt_isnum(irr->t) && irt_isinteger(t)`):插一个 `IR_CONV` 把浮点转成整数,**带 guard**(`IRTGI`,G 表示 guard,`IRCONV_CHECK` 表示要检查)。为什么带 guard?因为 num→int 可能丢失精度(浮点 3.7 转整数 3),必须检查转换是否安全,不安全就退出 trace。
- 其他情况(比如一边表一边数):类型差太远,`lj_trace_err(J, LJ_TRERR_TYPEINS)` 报错,触发 §2.1 的退回录制逻辑。

这段代码体现了 LuaJIT 对动态语言类型不稳定的态度:**能修的(整数/浮点互转)就地修,不能修的(类型差太远)报错重试,绝不生成类型不匹配的 PHI**。这是 sound 的体现——类型不稳定的 PHI 会让后端用错机器指令(整数指令处理浮点),算错。

### 2.3 loop_emit_phi:发射真正的 PHI

`loop_unroll` 复制循环结束后,调 `loop_emit_phi`(`lj_opt_loop.c:104`)处理收集到的 PHI 候选。这个函数有 5 趟(pass)扫描,逻辑比较密。我们抓住它的核心目标——**去掉冗余 PHI,发射真正需要的 PHI**——讲清楚每趟在干嘛,不逐行展开。

**Pass #1:标记冗余和潜在冗余的 PHI。**

```c
  /* Pass #1: mark redundant and potentially redundant PHIs. */
  for (i = 0, j = 0; i < nphi; i++) {
    IRRef lref = phi[i];
    IRRef rref = subst[lref];
    if (lref == rref || rref == REF_DROP) {  /* Invariants are redundant. */
      irt_clearphi(IR(lref)->t);
    } else {
      phi[j++] = (IRRef1)lref;
      if (!(IR(rref)->op1 == lref || IR(rref)->op2 == lref)) {
	/* Quick check for simple recurrences failed, need pass2. */
	irt_setmark(IR(lref)->t);
	passx = 1;
      }
    }
  }
  nphi = j;
```

(`lj_opt_loop.c:110-125`)

对每个 PHI 候选 `lref`(pre-roll 段的引用),查它的替换 `rref = subst[lref]`(复制段的引用):

- 如果 `lref == rref`(复制后指向自己)或 `rref == REF_DROP`(被 DCE 删了),说明这个值其实是循环不变的(或者已经不需要了),**PHI 冗余,清掉它的 ISPHI 标志**,不保留这个候选。
- 否则,这个 PHI 候选保留(`phi[j++] = lref`)。再做一个"快速检查":如果 `rref` 的操作数里**直接包含** `lref`(`IR(rref)->op1 == lref || IR(rref)->op2 == lref`),说明这是个"简单递推"(simple recurrence)——比如 `sum_new = sum_old + i`,新值直接由旧值算来,这种 PHI 一定不冗余。如果快速检查失败(`!(...)`),说明 rref 间接依赖 lref,可能是冗余的(比如经过一串计算后又回到 lref),需要 Pass #2 精确判断,先打个 mark(`irt_setmark`)。

**Pass #2:遍历循环体,清除"被使用的"PHI 的 mark。**

```c
  /* Pass #2: traverse variant part and clear marks of non-redundant PHIs. */
  if (passx) {
    SnapNo s;
    for (i = J->cur.nins-1; i > invar; i--) {
      IRIns *ir = IR(i);
      if (!irref_isk(ir->op2)) irt_clearmark(IR(ir->op2)->t);
      if (!irref_isk(ir->op1)) {
	irt_clearmark(IR(ir->op1)->t);
	...
      }
    }
    for (s = J->cur.nsnap-1; s >= onsnap; s--) {
      ...
      for (n = 0; n < nent; n++) {
	IRRef ref = snap_ref(map[n]);
	if (!irref_isk(ref)) irt_clearmark(IR(ref)->t);
      }
    }
  }
```

(`lj_opt_loop.c:127-155`,有删节)

Pass #1 打了 mark 的 PHI,是"可能冗余"的。怎么确认它到底冗不冗余?方法是:**看循环体里有没有任何指令(或 snapshot)用了它**。如果被用了,它就不冗余(删了会断数据依赖);如果完全没人用,它就是冗余的(可以删)。Pass #2 从后往前扫循环体(`i > invar`)的每条指令的操作数,以及循环体的所有 snapshot 条目,把"被作为操作数或被 snapshot 引用"的 PHI 的 mark 清掉。清完 mark 之后,还带着 mark 的 PHI,就是真正冗余的。

**Pass #3:为"在栈槽里但没有对应 SLOAD"的值补 PHI。**

```c
  /* Pass #3: add PHIs for variant slots without a corresponding SLOAD. */
  nslots = J->baseslot+J->maxslot;
  for (i = 1; i < nslots; i++) {
    IRRef ref = tref_ref(J->slot[i]);
    while (!irref_isk(ref) && ref != subst[ref]) {
      IRIns *ir = IR(ref);
      irt_clearmark(ir->t);  /* Unmark potential uses, too. */
      if (irt_isphi(ir->t) || irt_ispri(ir->t))
	break;
      irt_setphi(ir->t);
      if (nphi >= LJ_MAX_PHI)
	lj_trace_err(J, LJ_TRERR_PHIOV);
      phi[nphi++] = (IRRef1)ref;
      ref = subst[ref];
      if (ref > invar)
	break;
    }
  }
```

(`lj_opt_loop.c:157-173`)

这一趟处理一种特殊情况:有些循环携带的值,存在栈槽里(`J->slot[]`,P2-07 §6.3 讲过 slot 到 TRef 的映射),但循环体里**没有对应的 SLOAD**(因为循环体不重新加载它,直接用 pre-roll 的值)。这种值也需要 PHI——因为它在循环里变,但不经过 SLOAD 就不会被前面的候选收集抓到。Pass #3 遍历所有栈槽,顺着 `subst[]` 链追,把漏掉的 PHI 候选补上。

**Pass #4:传播非冗余 PHI。**

```c
  /* Pass #4: propagate non-redundant PHIs. */
  while (passx) {
    passx = 0;
    for (i = 0; i < nphi; i++) {
      IRRef lref = phi[i];
      IRIns *ir = IR(lref);
      if (!irt_ismarked(ir->t)) {  /* Propagate only from unmarked PHIs. */
	IRIns *irr = IR(subst[lref]);
	if (irt_ismarked(irr->t)) {  /* Right ref points to other PHI? */
	  irt_clearmark(irr->t);  /* Mark that PHI as non-redundant. */
	  passx = 1;  /* Retry. */
	}
      }
    }
  }
```

(`lj_opt_loop.c:175-188`)

如果 PHI A 的替换值,指向另一个 PHI B(而 B 被 mark 成"可能冗余"),那么 A 不冗余 ⇒ A 用了 B ⇒ B 也不冗余。Pass #4 传播这个关系:只要一个非冗余 PHI 的替换值指向另一个被 mark 的 PHI,就把那个 mark 清掉(它也不冗余)。这是个不动点迭代(`while (passx)`),直到没有新的 mark 被清除。

**Pass #5:发射 PHI 指令或消除 PHI。**

```c
  /* Pass #5: emit PHI instructions or eliminate PHIs. */
  for (i = 0; i < nphi; i++) {
    IRRef lref = phi[i];
    IRIns *ir = IR(lref);
    if (!irt_ismarked(ir->t)) {  /* Emit PHI if not marked. */
      IRRef rref = subst[lref];
      if (rref > invar)
	irt_setphi(IR(rref)->t);
      emitir_raw(IRT(IR_PHI, irt_type(ir->t)), lref, rref);
    } else {  /* Otherwise eliminate PHI. */
      irt_clearmark(ir->t);
      irt_clearphi(ir->t);
    }
  }
```

(`lj_opt_loop.c:190-202`)

最后一趟,对每个剩下的 PHI 候选:

- 如果没被 mark(非冗余),发射真正的 `IR_PHI` 指令:`emitir_raw(IRT(IR_PHI, irt_type(ir->t)), lref, rref)`。PHI 的两个操作数是 `lref`(pre-roll 段的值,循环入口旧值)和 `rref`(复制段的值,回边新值)。类型用原指令的类型 `irt_type(ir->t)`。注意 PHI 用 `emitir_raw` 直接发射,**不经过 FOLD pipeline**——PHI 不能被折叠(P2-07 §7.4 提过,折叠 PHI 会破坏循环携带依赖)。
- 如果被 mark(冗余),消除:清掉 mark 和 ISPHI 标志,不发射 PHI。

`IR_PHI` 的指令定义(`lj_ir.h:41`):`_(PHI, S, ref, ref)`——两个操作数都是引用,模式是 `IRM_S`(有副作用,不能被 DCE 删)。PHI 在 IR 里**发射在循环体下方**(因为它是复制完才发射的,位置在所有循环体指令之后)。这个位置很特别——传统 SSA 的 PHI 在基本块开头,而 LuaJIT 的 PHI 在循环体结尾。文件注释(`lj_opt_loop.c:84-89`)解释:这不改变 PHI 本身的语义(PHI 还是合并两个值),但大大简化了即时 IR 生成和机器码生成——后端看到 PHI,就知道"循环结束时把这个值更新到对应位置",顺序自然。

### 2.4 小结:循环优化的完整流程

把 §2.2、§2.3 合起来,LuaJIT 的循环优化 `lj_opt_loop` 做了这几件事:

1. 发射 `IR_LOOP` 标记,分开 pre-roll 和循环体。
2. 逐条复制录制段指令,用 `subst[]` 替换操作数,重新喂给 FOLD/CSE pipeline。
   - 操作数都没变的(`IRM_N` 模式),shortcut 折叠回原指令——**不变量自动外提**。
   - 操作数变了的,真正发射新指令到循环体。
3. 复制过程中,发现 `ref < invar`(指回录制段)的,收集为 PHI 候选;发现类型不稳定的,插 CONV 修复或报错退回。
4. snapshot 同步复制替换。
5. `loop_emit_phi` 五趟扫描:去冗余、补漏、传播、最后发射 `IR_PHI` 指令在循环体下方。
6. 失败(PHI 溢出、类型不稳定、guard 必失败)时,`loop_undo` 回滚,退回录制多展开(`instunroll` 次)。

整个过程没有一行显式的"LICM 算法",但 LICM 想做的事(外提不变量、保留循环携带依赖)全部由"复制-替换 + pipeline 复用 + PHI 收集"这套机制达成。这是 LuaJIT 在 trace 框架下的独门技巧,也是它能把循环优化做到 453 行 C 的根本原因。

讲完了循环优化,我们转向本章的第二个主题——指令拆分。它和循环优化没有直接关系(它处理的是 64 位运算在 32 位平台的问题),但它同样是优化篇的收尾 pass,和循环优化一起,把 IR 准备到后端可用的状态。

## §3 第一性原理:为什么要把一条 IR 拆成两条

指令拆分(split)要解决的问题,和循环优化完全不同。我们重新从第一性原理推导:为什么需要把一条 IR 拆成两条?

### 3.1 32 位 CPU 没有 64 位指令

这件事听起来像是历史遗留问题,但在 2026 年依然现实。LuaJIT 支持的目标平台里,有一类是 **32 位 CPU**——比如 32 位 x86(x86)、某些 32 位 ARM。在这些 CPU 上:

- **没有原生的 64 位整数运算指令**。CPU 的寄存器是 32 位的(`eax`、`ebx` 等),ALU 算术逻辑单元按 32 位设计。要做 64 位加法,得用两条 32 位 `ADD`(分别加低 32 位和高 32 位)加一条 `ADC`(带进位加,处理低 32 位的进位)凑出来。
- **可能没有硬件浮点单元(FPU)**。有些嵌入式 ARM(软浮点目标,soft-float)没有 FPU,做浮点运算(`double` 的加减乘除)全靠软件模拟——调用一组 C 函数(`softfp_add`、`softfp_mul` 等)来完成。

而 LuaJIT 的 IR 里,偏偏有 64 位类型:

- `IRT_I64` / `IRT_U64`:64 位有符号/无符号整数。**FFI 的 `int64_t` / `uint64_t` 会产生这两个类型**。LuaJIT 的 FFI 让 Lua 代码能直接操作 C 的 64 位整数,这些操作录制成 IR,就是 64 位整数运算。
- `IRT_NUM`:双精度浮点(64 位)。在软浮点目标上,这个类型的运算没有硬件支持。

所以,在 32 位平台 + FFI(或软浮点目标)的组合下,IR 里会出现"一条 64 位运算指令",而后端找不到对应的单条 32 位机器指令来生成。

### 3.2 两条路:后端特判 vs IR 层拆分

面对"一条 64 位 IR 对应多条 32 位机器指令"这件事,有两种实现策略。

**策略一:后端特判。** 后端的汇编生成器(P4-14,`lj_asm.c`)在为每条 IR 选机器指令时,遇到 64 位 IR 就特判:`i64 ADD` 不是一条 `ADD` 指令,而是 `low.ADD + high.ADC`;`num MUL` 在软浮点上不是一个 `MUL` 指令,而是一次 `softfp_mul` 函数调用。这意味着:

- 后端要为每种 64 位指令写一套"拆成多条 32 位"的特殊逻辑。`ADD`/`SUB`/`MUL`/`DIV`/`MOD`/移位/位运算……每种都要特判。
- **寄存器分配也要特判**:64 位值要占两个 32 位寄存器(低字+高字),寄存器分配器(P4-13)得知道"这条指令需要分配一对寄存器"。这让寄存器分配逻辑复杂化——本来线性扫描(P4-13)是按单寄存器算的,现在要处理寄存器对。
- 后端代码量爆炸。`lj_opt_split.c` 文件头注释(`lj_opt_split.c:29-34`)的原话是:`"This is much easier than adding support for allocating register pairs to each backend (believe me, I tried)."`——"这比给每个后端加寄存器对分配的支持容易得多(相信我,我试过)。" Mike Pall 亲口说他试过后端特判的路线,放弃了。

**策略二:IR 层拆分。** 在 IR 还没进后端之前,先做一道 pass:把每条 64 位 IR,**拆成两条 32 位 IR**(低字一条 + 高字一条),让后端只面对 32 位 IR。后端完全不需要知道"曾经有过 64 位指令"——它看到的都是它本来就支持的 32 位指令,干干净净。

LuaJIT 选了策略二。这就是 `lj_opt_split` 这道 pass 存在的理由。文件头注释(`lj_opt_split.c:22-28`)说得很直白:

> 这个 pass 把 64 位 IR 指令拆成多条 32 位 IR 指令。它只在软浮点目标、或缺乏原生 64 位整数操作的 32 位 CPU(目前 FFI 是 64 位整数指令的唯一产生者)上激活。
>
> 在单独一个 pass 里拆分 IR,让每个 32 位 IR 的汇编后端保持简单。只需要实现少量额外功能。这比给每个后端加寄存器对分配容易得多。

策略二的好处清晰:**把复杂度集中在一道 pass 里,让后端简单**。这是"分阶段处理"的工程思想——每个 pass 只做一件事,后端只管"把 32 位 IR 翻成机器码",不用操心 64 位。

### 3.3 拆分的基本模式:原指令 + HIOP

那具体怎么拆?注释(`lj_opt_split.c:36-46`)讲了基本模式:

> 基本想法是,把每条 64 位 IR 指令替换成它的 32 位等价物,加上一条额外的 `HIOP` 指令。拆分后的 IR 不经过 FOLD 或任何其他优化,所以每条 HIOP 保证紧跟在它的对应指令后面。HIOP 的实际功能由前一条指令推断。
>
> HIOP 的操作数持有"高字输入引用"。HIOP 的输出是"高字输出引用",也用来存高字的寄存器或溢出槽信息。寄存器分配器把这条指令当作独立于其他指令的指令处理,这比使用固定寄存器对改善了代码质量。

提炼成要点:

**拆分模式**:一条 64 位 IR `op`(比如 `i64 ADD op1, op2`),拆成:

```
原指令(改成 32 位):  int ADD  op1_lo, op2_lo     ← 低 32 位加法
HIOP 指令:          int HIOP op1_hi, op2_hi     ← 高 32 位带进位加法
```

- 原指令的操作码不变(`ADD` 还是 `ADD`),但**类型从 `i64` 改成 `int`**(32 位整数)。它的操作数换成"低字"引用(`op1_lo`、`op2_lo`)。
- 紧跟一条 `IR_HIOP` 指令。HIOP 的操作数是"高字"引用(`op1_hi`、`op2_hi`)。HIOP 的语义由前一条指令推断——前一条是 `ADD`,那 HIOP 就表示"高字的带进位加"(机器码生成 `ADC`);前一条是 `MUL`,HIOP 就表示"高字的乘法相关运算"。

**关键约束**:HIOP 必须**紧跟**原指令。为什么?因为 HIOP 的语义靠"前一条指令是什么"推断——如果它们被别的指令隔开,或者顺序被打乱,后端就无法知道这条 HIOP 对应的是 ADD 还是 MUL。为了保证这一点,拆分后的 IR **不再经过 FOLD/CSE pipeline**——`split_emit`(`lj_opt_split.c:99`)直接写 IR 数组(`ir->ot/op1/op2 = ...`),不走 `lj_opt_fold`。这和循环优化里的 `emitir`(走 pipeline)形成对比。注释(`lj_opt_split.c:38-39`)明确:"The splitted IR is not passed through FOLD or any other optimizations, so each HIOP is guaranteed to immediately follow it's counterpart."

**寄存器分配独立**。HIOP 是一条独立的 IR 指令,寄存器分配器给它独立分配寄存器(P4-13)。这比"固定寄存器对"(比如强制 64 位值占 `eax:edx`)灵活——分配器可以给低字和高字分别选最合适的寄存器。注释(`lj_opt_split.c:44-46`)说这改善了代码质量。

### 3.4 一个完整的拆分示例

文件头注释(`lj_opt_split.c:59-93`)给了一个非常好的示例,我们直接用它。考虑这样一段 Lua FFI 代码(操作一个有两个 `int64_t` 字段的结构体):

```lua
x.b = x.a + 1   -- x.a, x.b 都是 int64_t
```

这段代码录制成 IR(在 64 位平台上,不需要拆分),长这样:

```
0100    p32 ADD    base  +8        ← 算 x.a 的地址(base+8)
0101    i64 XLOAD  0100            ← 加载 x.a(64 位)
0102    i64 ADD    0101  +1        ← x.a + 1(64 位加法)
0103    p32 ADD    base  +16       ← 算 x.b 的地址(base+16)
0104    i64 XSTORE 0103  0102      ← 存到 x.b(64 位存储)
```

对应的 x64 机器码(64 位平台,原生支持 64 位):

```asm
        mov rax, [esi+0x8]        ; 加载 x.a
        add rax, +0x01            ; x.a + 1
        mov [esi+0x10], rax       ; 存到 x.b
```

干净利落,三条指令。

但同样这段 IR,在 32 位 x86 上,经过 `lj_opt_split` 拆分后,变成:

```
0100    p32 ADD    base  +8        ← x.a 地址(低字),不变
0101    int XLOAD  0100            ← 加载 x.a 的低 32 位(类型从 i64 改成 int)
0102    p32 ADD    base  +12       ← 算 x.a 高字地址(base+12,重关联!)
0103    int XLOAD  0102            ← 加载 x.a 的高 32 位
0104    int ADD    0101  +1        ← 低字加法
0105    int HIOP   0103  +0        ← 高字带进位加(加 0,接低字的进位)
0106    p32 ADD    base  +16       ← x.b 地址(低字)
0107    int XSTORE 0106  0104      ← 存 x.b 低字
0108    int HIOP   0106  0105      ← 存 x.b 高字
```

对应的 32 位 x86 机器码:

```asm
        mov eax, [esi+0x8]        ; 加载 x.a 低字
        mov ecx, [esi+0xc]        ; 加载 x.a 高字(0x8+4=0xc)
        add eax, +0x01            ; 低字加 1
        adc ecx, +0x00            ; 高字加进位(adc = add with carry)
        mov [esi+0x10], eax       ; 存 x.b 低字
        mov [esi+0x14], ecx       ; 存 x.b 高字
```

观察几件事,对应 §3.3 讲的拆分模式:

1. **每条 64 位指令拆成"32 位原指令 + HIOP"**。`0102 i64 ADD` 拆成 `0104 int ADD`(低字)+ `0105 int HIOP`(高字)。HIOP 紧跟在原指令后面。
2. **HIOP 的语义靠前一条指令推断**。`0105` 是 HIOP,前一条 `0104` 是 ADD,所以 `0105` 对应机器码 `adc`(带进位加)。`0108` 是 HIOP,前一条 `0107` 是 XSTORE,所以 `0108` 对应"存高字"(`mov [esi+0x14], ecx`)。
3. **地址计算被重关联(reassociate)**。原本 `i64 XLOAD 0100` 只需要一个地址(base+8)。拆分后,低字 XLOAD 用 `base+8`,高字 XLOAD 需要 `base+12`(0x8+4)。`split_ptr`(`lj_opt_split.c:183`)专门处理这个——它算出"另一个 32 位字的地址",并且做重关联优化:如果原地址是 `base+8`,高字地址就是 `base+8+4 = base+12`,可以合并成 `base+12` 一条 ADD。注释(`lj_opt_split.c:91-92`)专门提到:"You may notice the reassociated hiword address computation, which is later fused into the mov operands by the assembler."——这个重关联的高字地址计算,后来会被汇编器融合进 `mov` 的内存操作数里(`mov ecx, [esi+0xc]` 直接把 0xc 编进寻址),不占独立指令。这是 split pass 顺手做的一个小优化。
4. **64 位常量也拆**。`+1` 这个加数,在 64 位 ADD 里是 64 位常量 1。拆分后,低字用 `+1`,高字用 `+0`(因为 1 的高 32 位是 0)。`split_bitop`/常量拆分逻辑(`lj_opt_split.c:346-359`)处理这个。
5. **机器码条数变多,但每条都是 32 位**。64 位平台 3 条指令,32 位平台 6 条指令。这是 32 位平台做 64 位运算的固有代价——拆分只是把这个代价"提前到 IR 层显式化",并没有凭空消除它。但拆分让后端简单了:后端只面对 32 位 IR,每条翻一条 32 位机器指令,不需要特判。

这个示例把 split 的全部精髓展示了。下一节我们看源码怎么落地。

### 3.5 软浮点的特殊处理:浮点变调用

上面讲的是 64 位整数拆分。还有一类更特殊:软浮点(soft-float / softfp)目标上的浮点运算。

在软浮点目标上(`LJ_SOFTFP` 定义),CPU 没有 FPU,`double`(IRT_NUM)的加减乘除没有硬件指令。怎么办?**把浮点运算直接翻成 C 函数调用**。`lj_opt_split.c` 在 `LJ_SOFTFP` 条件下(`lj_opt_split.c:109` 的 `#if LJ_SOFTFP`),对 `IRT_NUM` 类型的运算指令,直接替换成 `IR_CALLN`(函数调用指令),调用 `softfp_add`/`softfp_sub`/`softfp_mul`/`softfp_div` 这些软浮点库函数。

看代码(`lj_opt_split.c:389-404`):

```c

```

`IRT_NUM` 的 `ADD`,被替换成 `CALLN`(调用 `softfp_add`)。`split_call_ll`(`lj_opt_split.c:162`)做的事:把两个 64 位操作数(各拆成低字+高字)打包成 CARG(调用参数),调 `softfp_add`,返回值是 64 位结果(低字 + 一个 `IRT_SOFTFP` 类型的 HIOP 表示高字)。

注意一个细节(`lj_opt_split.c:388`):`nir->t.irt = IRT_INT | (... IRT_GUARD)`——原指令的类型被改成 `IRT_INT`。因为软浮点上,`double` 用两个 32 位字表示(低字 + 高字),低字被当成 `IRT_INT` 处理。HIOP 的类型是 `IRT_SOFTFP`(`lj_opt_split.c:57` 注释:"HIOP for number results has the type IRT_SOFTFP")——这是一个特殊类型,告诉后端"这个 HIOP 是软浮点结果的高字,要和低字一起表示一个 double"。

注释里还有一句(`lj_opt_split.c:387`):"Note: hi ref = lo ref + 1!"——软浮点结果的高字引用号 = 低字引用号 + 1。这个约束是给 snapshot 逻辑用的(`SNAP_SOFTFPNUM`),P5-17 会讲。这里先知道:软浮点 double 的两个字的引用号是连续的,方便 snapshot 编码。

比较和 MIN/MAX 不拆成调用(注释 `lj_opt_split.c:56`:"except for comparisons and MIN/MAX")——它们有专门的软浮点比较函数,走另一条路径。

### 3.6 拆分的触发条件

不是所有平台都需要拆分。`lj_opt_split` 只在特定条件下编译和激活。

**编译条件**(`lj_opt_split.c:11`):

```c
#if LJ_HASJIT && (LJ_SOFTFP32 || (LJ_32 && LJ_HASFFI))
```

只有 `LJ_HASJIT`(启用 JIT)且(`LJ_SOFTFP32`(32 位软浮点目标)或(`LJ_32`(32 位平台)且 `LJ_HASFFI`(启用 FFI)))时,`lj_opt_split.c` 才被编译进二进制。其他情况(64 位平台、或不带 FFI 的 32 位),这个文件根本不参与编译。

相应的,`lj_iropt.h:159-163` 用条件编译提供 `lj_opt_split` 的声明或空宏:

```c
#if LJ_SOFTFP32 || (LJ_32 && LJ_HASFFI)
LJ_FUNC void lj_opt_split(jit_State *J);
#else
#define lj_opt_split(J)		UNUSED(J)
#endif
```

在不拆分的平台上,`lj_opt_split(J)` 被宏替换成 `UNUSED(J)`(什么也不做)。所以 `lj_trace.c:736` 的 `lj_opt_split(J)` 调用,在 64 位平台上是个空操作,零开销。

**运行时触发**。即使在会编译 split 的平台上,也不是每条 trace 都需要拆分——只有当 trace 的 IR 里**真的有 64 位指令**时才需要。这个判断由 `J->needsplit` 标志(`lj_jit.h:469`,`uint8_t needsplit; /* Need SPLIT pass. */`)记录。

`needsplit` 在录制阶段被设置。当录制器(`lj_record.c`、`lj_crecord.c`、`lj_ffrecord.c`)发射一条 64 位 IR(比如 FFI 的 `int64_t` 运算)时,它会调 `lj_needsplit(J)` 宏(`lj_jit.h:402`):

```c
#define lj_needsplit(J)		(J->needsplit = 1)
```

把 `needsplit` 置 1。到优化阶段,`lj_opt_split`(`lj_opt_split.c:822`)检查这个标志:

```c
/* SPLIT pass. */
void lj_opt_split(jit_State *J)
{
#if LJ_SOFTFP
  if (!J->needsplit)
    J->needsplit = split_needsplit(J);
#else
  lj_assertJ(J->needsplit >= split_needsplit(J), "bad SPLIT state");
#endif
  if (J->needsplit) {
    int errcode = lj_vm_cpcall(J->L, NULL, J, cpsplit);
    if (errcode) {
      /* Completely reset the trace to avoid inconsistent dump on abort. */
      J->cur.nins = J->cur.nk = REF_BASE;
      J->cur.nsnap = 0;
      lj_err_throw(J->L, errcode);  /* Propagate errors. */
    }
  }
}
```

(`lj_opt_split.c:822-839`)

逻辑:

- 软浮点目标(`LJ_SOFTFP`):如果 `needsplit` 没被录制阶段置位,再调 `split_needsplit`(`lj_opt_split.c:797`)扫一遍 IR 确认是否真的需要。软浮点上,有些浮点相关指令(比如 `SLOAD` 带 `IRSLOAD_CONVERT`、`TOBIT`)不一定触发 `lj_needsplit`,所以要兜底扫一遍。
- 非软浮点的 32 位 + FFI:`lj_assertJ(J->needsplit >= split_needsplit(J), ...)`,断言 `needsplit` 的值不低于实际扫描结果——即"如果扫描说需要,那录制阶段一定已经置位了"。这是个一致性检查。
- `if (J->needsplit)`:真的需要才执行 `cpsplit` → `split_ir`(真正的拆分逻辑)。不需要就跳过,零开销。
- 失败处理:如果拆分抛错(`errcode` 非 0),**彻底重置 trace**(`J->cur.nins = J->cur.nk = REF_BASE; J->cur.nsnap = 0;`),然后 `lj_err_throw` 传播错误。注意这和循环优化不同——循环优化失败可以退回录制重试,split 失败直接作废整条 trace(重置成空)。因为 split 失败通常是结构性问题(比如 IR 里有 split 处理不了的指令),重试也救不回来。

`split_needsplit`(`lj_opt_split.c:797`)的扫描逻辑:

```c
static int split_needsplit(jit_State *J)
{
  IRIns *ir, *irend;
  IRRef ref;
  for (ir = IR(REF_FIRST), irend = IR(J->cur.nins); ir < irend; ir++)
    if (LJ_SOFTFP ? irt_is64orfp(ir->t) : irt_isint64(ir->t))
      return 1;
  if (LJ_SOFTFP) {
    for (ref = J->chain[IR_SLOAD]; ref; ref = IR(ref)->prev)
      if ((IR(ref)->op2 & IRSLOAD_CONVERT))
	return 1;
    if (J->chain[IR_TOBIT])
      return 1;
  }
  for (ref = J->chain[IR_CONV]; ref; ref = IR(ref)->prev) {
    IRType st = (IR(ref)->op2 & IRCONV_SRCMASK);
    if ((LJ_SOFTFP && (st == IRT_NUM || st == IRT_FLOAT)) ||
	st == IRT_I64 || st == IRT_U64)
      return 1;
  }
  return 0;  /* Nope. */
}
```

(`lj_opt_split.c:797-818`)

它扫三类情况:

1. IR 里有没有 64 位或浮点类型的指令(`irt_is64orfp` 软浮点 / `irt_isint64` 32+FFI)。有就返回 1。
2. 软浮点上,有没有带 `IRSLOAD_CONVERT`(int↔num 转换的 SLOAD)或 `IR_TOBIT`(转位运算整数)。这些在软浮点上需要拆分。
3. 有没有源类型是 64 位/浮点的 `IR_CONV`(类型转换)。有就返回 1。

都没有,返回 0,不需要拆分。

这个扫描只在软浮点目标的兜底路径里调用(非软浮点的 32+FFI 走 assert 不实际调用),所以不是每条 trace 都扫——大多数 trace 的 `needsplit` 由录制阶段的 `lj_needsplit` 直接确定。

## §4 源码印证:split_ir 的主循环

上一节讲了 split 的第一性原理和触发条件。这一节我们钻进 `split_ir`(`lj_opt_split.c:325`)的主循环,看它怎么逐条拆分 IR。这个函数比 `loop_unroll` 长(因为它要处理几十种 IR 指令的拆分),但结构是清晰的:遍历老 IR,对每条指令,根据它的类型和操作码,拆分成新 IR。

### 4.1 整体结构:复制老 IR,逐条改写

`split_ir` 的整体策略和 `loop_unroll` 类似——**把老 IR 复制出来,然后逐条改写成拆分后的新 IR**。开头(`lj_opt_split.c:327-343`):

```c
static void split_ir(jit_State *J)
{
  IRRef nins = J->cur.nins, nk = J->cur.nk;
  MSize irlen = nins - nk;
  MSize need = (irlen+1)*(sizeof(IRIns) + sizeof(IRRef1));
  IRIns *oir = (IRIns *)lj_buf_tmp(J->L, need);
  IRRef1 *hisubst;
  IRRef ref, snref;
  SnapShot *snap;

  /* Copy old IR to buffer. */
  memcpy(oir, IR(nk), irlen*sizeof(IRIns));
  /* Bias hiword substitution table and old IR. Loword kept in field prev. */
  hisubst = (IRRef1 *)&oir[irlen] - nk;
  oir -= nk;

  /* Remove all IR instructions, but retain IR constants. */
  J->cur.nins = REF_FIRST;
  J->loopref = 0;
```

关键步骤:

- `memcpy(oir, IR(nk), irlen*sizeof(IRIns))`:把整个老 IR(从常量 `nk` 到指令 `nins`)复制到临时缓冲区 `oir`。后面基于 `oir` 改写。
- `hisubst = (IRRef1 *)&oir[irlen] - nk`:分配"高字替换表"`hisubst`,和 `loop_unroll` 的 `subst` 类似,用负偏移指针,让 `hisubst[ref]` 直接用引用号当下标。`hisubst[ref]` 存的是"老引用 `ref` 对应的高字新引用"。低字替换存在 `oir[ref].prev` 字段里(复用 `prev`,P2-07 §3.3 讲过它在寄存器分配前可复用)——所以低字替换不另开数组,直接借 `prev`。
- `J->cur.nins = REF_FIRST`:把当前 IR 的指令计数器重置到起点(常量保留)。接下来会重新发射所有指令到 `J->cur.ir`。
- `J->loopref = 0`:`loopref`(P2-07 提过,标记循环位置)清零,后面遇到 `IR_LOOP` 时重新设。

注意一个重要区别:`loop_unroll` 是**追加**(在老 IR 后面追加复制段),而 `split_ir` 是**重建**(把老 IR 全复制出来,然后清空当前 IR,逐条重新发射)。为什么不同?因为 split 要拆分 64 位指令,拆分后指令数量变多(一条变两条或多条),而且 64 位常量也要拆,整个 IR 的布局会变——所以干脆重建。而循环优化只是追加一段循环体,老 IR 不动。

### 4.2 常量和固定引用的拆分

重建从常量和固定引用开始(`lj_opt_split.c:346-359`):

```c
  /* Process constants and fixed references. */
  for (ref = nk; ref <= REF_BASE; ref++) {
    IRIns *ir = &oir[ref];
    if ((LJ_SOFTFP && ir->o == IR_KNUM) || ir->o == IR_KINT64) {
      /* Split up 64 bit constant. */
      TValue tv = *ir_k64(ir);
      ir->prev = lj_ir_kint(J, (int32_t)tv.u32.lo);
      hisubst[ref] = lj_ir_kint(J, (int32_t)tv.u32.hi);
    } else {
      ir->prev = ref;  /* Identity substitution for loword. */
      hisubst[ref] = 0;
    }
    if (irt_is64(ir->t) && ir->o != IR_KNULL)
      ref++;
  }
```

对每个常量:

- 64 位常量(`IR_KNUM` 浮点,软浮点上;或 `IR_KINT64` 64 位整数):拆成两个 32 位常量。`ir->prev`(低字替换)指向新的 `lj_ir_kint(低32位)`,`hisubst[ref]`(高字替换)指向 `lj_ir_kint(高32位)`。`ir_k64`(`lj_ir.h:592`,P2-07 §3.5 讲过)取 64 位常量的值,拆成 `tv.u32.lo` 和 `tv.u32.hi`。
- 32 位常量:低字就是自己(`ir->prev = ref`),没有高字(`hisubst[ref] = 0`)。
- `if (irt_is64(ir->t) && ir->o != IR_KNULL) ref++;`:64 位常量占两个 IRIns 槽(P2-07 §3.5 讲过 KNUM/KINT64 占两槽),所以 `ref++` 跳过第二个槽。

### 4.3 主循环:逐条改写指令

主循环(`lj_opt_split.c:364-771`)是最长的部分。结构:

```c
  for (ref = REF_FIRST; ref < nins; ref++) {
    IRIns *ir = &oir[ref];

      case IR_SUB:
	...
      case IR_MUL: ... split_call_ll(...); break;

      case IR_BSHL: case IR_BSHR: case IR_BSAR: case IR_BROL: case IR_BROR:

    else if (ir->o == IR_CONV) { ... }

```

每条老指令的处理:

1. **分配新指令槽**:`nref = lj_ir_nextins(J)`,`nir = IR(nref)`。
2. **snapshot 同步**:如果越过 snapshot 边界,调 `split_subst_snap`(`lj_opt_split.c:312`)替换 snapshot 里的引用(和 loop 的 snapshot 替换同理)。
3. **复制-替换操作数**:`nir->op1 = oir[ir->op1].prev`——用老指令的操作数,去 `oir[].prev`(低字替换表)查它对应的新引用。这是"低字操作数替换"。如果操作数是常量(`ir->op1 < nk`),直接用原引用(常量已经在 §4.2 拆好了,但 32 位常量的 `prev` 就是自己,所以直接用也行)。
4. **记录低字替换**:`ir->prev = nref`,让后续指令引用这条指令时,能找到它的低字新引用。
5. **复制操作码和类型**:`nir->o = ir->o; nir->t.irt = ir->t.irt & ~(IRT_MARK|IRT_ISPHI)`。清掉 `IRT_MARK` 和 `IRT_ISPHI` 标志(split 不需要它们)。
6. **按类型/操作码拆分**:这是核心。分三大块:
   - `LJ_SOFTFP` 下 `irt_isnum`:浮点指令,§3.5 讲过,翻成 `softfp_*` 调用。
   - `LJ_32 && LJ_HASFFI` 下 `irt_isint64`:64 位整数指令,拆成低字 + HIOP。各种位运算(`BAND`/`BOR`/`BXOR`/移位)有专门优化(`split_bitop`/`split_bitshift`)。
   - 其他:`SLOAD` 带 CONVERT、`CONV` 类型转换、`CALLXS`/`CARG`(调用参数)、`CNEWI`、`LOOP` 等,各有处理。
7. **记录高字替换**:`hisubst[ref] = hi`,把这条指令的高字新引用存进替换表。如果这条指令不需要高字(`hi = 0`),就是 0。

### 4.4 位运算的拆分优化:split_bitop 和 split_bitshift

64 位位运算(`BAND`/`BOR`/`BXOR`/`BSHL`/`BSHR`/`BSAR`/`BROL`/`BROR`)在 32 位平台上的拆分,有些可以优化——`split_bitop`(`lj_opt_split.c:271`)和 `split_bitshift`(`lj_opt_split.c:200`)就做了这些优化。

**`split_bitop`**(按位与/或/异或):

按位运算的好处是**高低字独立**——64 位 `BAND` 的高字,就是两个操作数高字的 32 位 `BAND`;低字同理。没有进位传播。所以 `BAND`/`BOR`/`BXOR` 的拆分很直接:低字 `BAND op1_lo, op2_lo`,高字 `BAND op1_hi, op2_hi`。

但 `split_bitop` 还顺手做常量优化(`lj_opt_split.c:276-307`):如果某个操作数是常量 0 或 -1(全 0 或全 1),可以简化。比如 `BAND x, 0` = 0(与 0 得 0),`BOR x, 0` = x(或 0 不变),`BXOR x, 0` = x(异或 0 不变),`BAND x, -1` = x(与全 1 不变)等等。这些在 32 位位运算里是经典优化,split 在高低字上分别做。

**`split_bitshift`**(移位):

移位比按位运算复杂,因为移位可能跨越高低字边界。比如 64 位左移 `BSHL x, 33`:低 32 位全移出去了,原来的低字变成新的高字,还要补 0。`split_bitshift`(`lj_opt_split.c:200`)对常量移位(`irref_isk(kref)`)做了精细优化:

- 移位量 `k = 0`:passthrough,不变。
- `k < 32`:移位不跨边界,高低字各自移位,再补一点跨字部分(`BSHL` 高字补低字的高位,`BSHR` 低字补高字的低位)。
- `k >= 32`:移位跨边界,高低字对调(`BSHL k>=32`:低字移成高字,新低字是 0;`BSHR k>=32`:高字移成低字)。
- `k == 32`:特殊情况,正好一整个字。
- 非常量移位:调 `lj_carith_shl64` 等 64 位移位库函数(`split_call_li`)。

这些优化把 64 位移位在 32 位平台上的各种情况都覆盖了,而且对常量移位生成最优的 32 位指令序列。这是 split pass"顺手做小优化"的体现(文件注释 `lj_opt_split.c:33-34` 提到:"A few simple, but important optimizations can be performed by the SPLIT pass, which would be tedious to do in the backend")。

### 4.5 PHI 在 split 里的处理

split 还要处理循环优化留下的 PHI 指令。因为 PHI 的两个操作数可能是 64 位值(循环携带的 64 位变量),拆分后 PHI 也要拆成"低字 PHI + 高字 PHI"。

`lj_opt_split.c:487-493`(软浮点)/`604-613`(32+FFI):

```c
      case IR_PHI:
	if (nir->op1 == nir->op2)
	  J->cur.nins--;  /* Drop useless PHIs. */
	if (hisubst[ir->op1] != hisubst[ir->op2])
	  split_emit(J, IRT(IR_PHI, IRT_SOFTFP),
		     hisubst[ir->op1], hisubst[ir->op2]);
	break;
```

逻辑:

- 低字 PHI:如果两个操作数的低字相同(`nir->op1 == nir->op2`),这个 PHI 冗余(合并的两个值低字一样),删掉(`J->cur.nins--` 撤回刚才分配的槽)。否则保留低字 PHI(已经在前面的通用逻辑里发射了)。
- 高字 PHI:如果两个操作数的高字不同(`hisubst[ir->op1] != hisubst[ir->op2]`),发射一个高字 PHI(`IRT_SOFTFP` 软浮点 / `IRT_INT` 64 位整数)。如果高字相同,不需要高字 PHI。

这样,一个 64 位 PHI,拆成最多两个 32 位 PHI(低字一个、高字一个)。循环携带的 64 位依赖,在拆分后依然正确闭环。

split 结束后,还有一道收尾(`lj_opt_split.c:778-783`):给新 IR 里所有 PHI 指令的操作数重新打上 ISPHI 标志(因为 split 重建 IR 时清掉了标志,这里补回去,给后端用):

```c
  /* Add PHI marks. */
  for (ref = J->cur.nins-1; ref >= REF_FIRST; ref--) {
    IRIns *ir = IR(ref);
    if (ir->o != IR_PHI) break;
    if (!irref_isk(ir->op1)) irt_setphi(IR(ir->op1)->t);
    if (ir->op2 > J->loopref) irt_setphi(IR(ir->op2)->t);
  }
```

从后往前扫,遇到非 PHI 就停(PHI 都在 IR 末尾),给每个 PHI 的两个操作数打 ISPHI 标志。`ir->op2 > J->loopref` 的判断:op2 是循环回边后的新值(在循环体里),给它打 ISPHI;op1 是 pre-roll 的旧值,如果非常量也打。这个标志给后端寄存器分配用——ISPHI 的值在循环里要保持在固定位置(寄存器或溢出槽),不能随意移动。

## §5 为什么 sound:循环优化和拆分都不会算错

讲完了机制,这一节回答"为什么这样设计是 sound 的"——也就是,循环优化和指令拆分,为什么不改变程序语义,不会算错结果。

### 5.1 循环优化的 sound 性:复制-替换保持数据流等价

循环优化的核心是"复制录制段、替换操作数、重新过 pipeline"。这套机制凭什么不改变语义?

**第一,复制-替换保证了数据流等价。** 复制段的每条指令,操作数用 `subst[]` 替换——`subst[old]` 指向"old 这条指令复制后的新引用"。这意味着复制段的指令,用的都是"复制后的值",数据依赖在复制段内自洽。从数据流角度,复制段等价于"把循环体再跑一遍"——每个值都正确地来自上一条指令的结果。没有依赖断裂。

**第二,pipeline 复用保证了优化正确。** 复制段重新喂给 FOLD/CSE/FWD,这些 pass 本身就是 sound 的(P3-09、P3-10 论证过):常量折叠只化简"结果确定的运算",CSE 只合并"输入完全相同的指令",forwarding 只复用"未被覆盖的加载"。它们用在复制段上,和在原段上一样正确——不会化简出错误的指令。唯一的额外约束(P2-07 §7.4 提过)是"不跨循环携带依赖折叠",这个约束在 pipeline 里通过 ISPHI 标志实现:带 ISPHI 的指令(循环携带依赖)不被折叠。

**第三,PHI 正确闭环。** §2.3 讲了 `loop_emit_phi` 的五趟扫描,目标是正确识别所有循环携带依赖并发射 PHI。PHI 的两个操作数 `lref`(pre-roll 旧值)和 `rref`(循环体新值)正确对应"循环入口值"和"回边值"。后端(P4)把 PHI 翻译成"循环外初始化 + 循环内更新"的寄存器使用,保证每一圈用的值都和"不优化、每圈重新算"一致。

**第四,guard 全程保留。** 这是最关键的安全保障。复制段保留了录制段的所有 guard(包括 shortcut 外提的不变 guard,和留在循环体的变 guard)。guard 失败就 side exit 退回解释器(P0-01 §9 的不变式)。所以即使循环优化基于的假设(类型不变、值不变)在某圈被打破,guard 会抓住,退回解释器按真实情况算——结果照样对。循环优化只是"在 guard 都成立时让循环跑得更快",它不削弱 guard 的任何一道检查。

具体到不变 guard 的外提(§1.6 妙处一):外提一个 guard,意味着"在循环外检查一次,循环体里不再检查"。这安全吗?安全——因为这个 guard 检查的东西是循环不变的(操作数每圈一样),那么"检查一次"和"每圈检查"的结果完全相同。检查的东西不变,检查结果就不变,所以检查一次就够了。如果检查的东西在循环里变了(操作数变),那它就不是不变 guard,不会被外提,老老实实留在循环体每圈检查。所以外提的 guard 一定是"外提不影响安全"的 guard——这是 CSE 的正确性保证的(CSE 只合并完全相同的指令,操作数不同的 guard 不会被合并)。

**第五,失败兜底。** 即使循环优化整体失败(PHI 溢出、类型不稳定),`loop_undo` 回滚所有修改,退回录制多展开。失败时,这条 trace 根本不会被装上去,解释器继续跑,结果完全正确。所以循环优化"要么成功且语义不变,要么失败且不影响结果"。

### 5.2 指令拆分的 sound 性:拆分前后位级等价

指令拆分的 sound 性,核心是"拆分前后,计算结果在位级别(bit-level)完全相同"。

**64 位整数拆分**。一条 64 位整数运算,拆成"低 32 位运算 + 高 32 位运算(+ 进位)"。这在计算机算术里是标准的——64 位加法 = 低 32 位无进位加 + 高 32 位带进位加(`ADD` + `ADC`)。LuaJIT 的拆分,严格遵循这个等价关系。比如 §3.4 的例子,`i64 ADD x.a, 1` 拆成 `int ADD x.a_lo, 1`(低字加 1)+ `int HIOP x.a_hi, 0`(高字加 0 带进位)。如果低字加 1 产生进位(x.a_lo = 0xFFFFFFFF 时),HIOP 的 `ADC` 会把进位加到高字——结果和 64 位原生加法完全一致。这就是为什么 HIOP 紧跟原指令——它的语义(`ADC`、`SBB`、无进位等)由前一条指令决定,必须配对。

位运算拆分更简单——按位运算高低字独立,不存在进位传播,拆分天然等价。

移位拆分(`split_bitshift`)的等价性,靠它对各种移位量(`k=0`、`k<32`、`k>=32`、`k==32`)的完整覆盖。每种情况生成的 32 位指令序列,都和 64 位原生移位的结果一致。对于常量移位,`split_bitshift` 用精细的分支保证最优且等价;对于非常量移位,退到调用 64 位移位库函数(`lj_carith_shl64` 等),这些库函数本身就是正确的 64 位实现。

**软浮点拆分**。`double` 的加减乘除,在软浮点上替换成 `softfp_add` 等库函数调用。这些库函数是标准的软浮点实现(遵循 IEEE 754),它们的计算结果和硬件 FPU 完全一致(都按 IEEE 754 规则)。所以替换不改变语义,只是把硬件做的事搬到软件做。

**snapshot 同步**。拆分改变了 IR 布局(一条变两条,引用号全变),snapshot 里记录的引用也必须同步更新。`split_subst_snap`(`lj_opt_split.c:312`)负责这个——它用 `oir[].prev`(低字替换表)更新 snapshot 里的每个引用。这样,side exit 时,snapshot 恢复的还是正确的值(低字+高字组合成完整的 64 位值)。如果 snapshot 不同步,side exit 会恢复出错误的值(只拿到低字,丢高字),导致解释器算错。这是拆分 sound 性的一个容易被忽略的点——`split_subst_snap` 保证它不出错。

**PHI 拆分**。64 位 PHI 拆成低字 PHI + 高字 PHI(§4.5)。这两个 PHI 各自闭环(低字 PHI 合并低字循环携带值,高字 PHI 合并高字),组合起来等价于原 64 位 PHI 的合并。拆分不会破坏循环携带依赖的闭环。

综上,指令拆分的 sound 性,靠的是"拆分前后位级等价"这个数学事实,加上 snapshot/PHI 的同步更新。它不引入任何近似或假设——拆完的 IR 计算的结果,和没拆的 IR 在每个 bit 上都一样。

## §6 ★对照:官方 Lua 与 JVM/V8

把 LuaJIT 的循环优化和指令拆分,和两个对象对照,取舍会更清晰。

### 6.1 对照一:官方 Lua(切"有没有循环优化")

官方 Lua **完全不做任何循环优化**。它是纯解释器,每条字节码逐条执行,循环跑一百万次就解释一百万次。在官方 Lua 里:

- **没有循环不变量外提**。`sum = sum + i + step` 这个循环,step 是循环不变量,但官方 Lua 每一圈都重新从栈槽加载 step、重新判断它的类型。明明 step 不变,却一百万次重复加载和判断。
- **没有循环展开**。循环体一百万次执行,每次都是完整的取指、译码、执行流程。
- **没有 PHI、没有 SSA**。官方 Lua 的字节码层面,变量就是栈槽,反复赋值,没有数据流分析,没有循环携带依赖的显式建模。
- **没有指令拆分**——因为官方 Lua 根本没有 IR,没有后端,不存在"64 位运算在 32 位平台怎么做"这个问题。解释器遇到 `int64_t` 运算(通过 FFI,虽然官方 Lua 没有 FFI,这里是假设性的),直接调对应的 C 库函数,不涉及拆分。

所以官方 Lua 慢,除了慢在"每条字节码要解释"(P0-01 §3),还慢在"它根本不做任何循环优化"——同样的循环体,它每一圈都从头算,不识别不变量、不消除冗余。这是"没有 IR、没有优化 pass"的直接代价。

LuaJIT 的循环优化,本质上就是**在 IR 这个舞台上,把循环里能省的计算省掉**。它不靠显式的 LICM 算法,而是靠"复制-替换 + pipeline 复用"让循环不变量自动外提。这个能力,官方 Lua 完全没有——这是 JIT 相对解释器的又一大加速来源(除了生成机器码之外)。

### 6.2 对照二:JVM/V8(method 级循环优化 vs trace 级)

JVM(HotSpot)和 V8(TurboFan)也做循环优化,但它们是 method JIT,循环优化的粒度和方式都和 LuaJIT 不同。这一对照很有讲头,因为它体现了 trace 和 method 两大流派在循环优化上的根本分歧。

**JVM C2 的循环优化:method 级 LICM + 循环剥离 + 向量化**

HotSpot 的 C2 编译器,以整个方法为单位编译,循环优化是它的强项。C2 的循环优化包括:

- **LICM(循环不变量外提)**:C2 在 Sea-of-Nodes IR(P2-07 §9.2 讲过)上,显式地做 LICM——识别循环不变量节点,把它们在图上挪到循环外面。C2 用 GCM(Global Code Motion,全局代码移动)算法在图上移动节点。这是教科书式的 LICM,因为它面对的是静态类型语言(Java),IR 里的 guard(deoptimization 节点)比动态语言少得多(Java 编译时类型已确定,大部分假设靠类型推断保证,不需要运行时检查),所以 §1.3 说的"guard 阻碍 LICM"的问题在 Java 上没那么严重。
- **循环剥离(loop peeling)**:把循环的第一圈单独拎出来处理(处理边界条件、冷启动),剩下的循环体更规整、更好优化。
- **循环展开(loop unrolling)**:把循环体复制几遍(比如 4 圈变一圈),减少循环开销(回边判断、跳转)。C2 会做不同程度展开。
- **向量化(auto-vectorization)**:识别循环里可以并行的数组操作,用 CPU 的 SIMD 指令(比如 x86 的 SSE/AVX)一次处理多个元素。比如 `for i: a[i] = b[i] + c[i]`,C2 可以生成 `padd` 指令一次加 4 个或 8 个 float。这是 C2 循环优化的重头戏,对数值计算循环提速巨大。
- **逃逸分析 + 标量替换**:虽然不是专门的循环优化,但对循环里创建的对象(比如 `for i: local obj = Point.new(...)`),如果 obj 不逃逸出循环,C2 会把它标量替换成几个局部变量,消除分配。这和 LuaJIT 的 sink(P3-11)类似。

**V8 TurboFan 的循环优化**:类似 C2,在 TurboFan IR(Sea-of-Nodes 变体)上做 LICM、展开、向量化。V8 因为是 JS(动态语言),deoptimization 节点比 Java 多,循环优化的激进程度受 deopt 约束。

**LuaJIT 的循环优化:trace 级,复制-替换,无显式 LICM**

LuaJIT 的循环优化,和 C2/V8 形成鲜明对比:

| 维度 | LuaJIT(trace 级) | JVM C2 / V8(method 级) |
|---|---|---|
| 编译单位 | 一条 trace(一条线性循环路径) | 整个方法(含所有分支) |
| LICM | 不显式做,靠复制-替换+CSE 自动外提 | 显式 LICM 算法(GCM 在图上移动节点) |
| guard 影响 | 大(IR 全是 guard),但不阻碍(靠复制保留控制依赖) | 小(Java 类型确定) / 中(JS deopt 节点) |
| 循环展开 | 隐式(复制一圈=展开 2 次) | 显式(可展开多次,如 4×、8×) |
| 向量化 | **不做** | 做(SIMD,SSE/AVX/NEON) |
| 循环剥离 | 不做(复制-替换天然处理首圈) | 做 |
| PHI 密度 | 极少(只循环回边,≤64) | 多(凡控制流合并处) |
| 实现复杂度 | ~453 行(lj_opt_loop.c) | 几千到几万行 |

几个关键差异值得展开:

**第一,LICM 的实现方式完全不同。** C2/V8 显式建控制流图,显式跑 LICM 算法,在图上挪节点。LuaJIT 不显式做——它复制一圈,让 CSE 自动识别不变量。LuaJIT 的方式更轻(复用已有 pipeline),但展开次数固定(2 次,复制一圈),不像 C2 能展开 8 次。而且 LuaJIT 不做循环剥离——它的 pre-roll 段天然就是"第一圈的剥离"(注释 `lj_opt_loop.c:46-49` 明确 pre-roll "performs exactly one loop iteration"),复制-替换天然包含了剥离的效果。

**第二,向量化是 LuaJIT 的短板。** C2/V8 的 auto-vectorization 对数值循环(数组加、点积、矩阵运算)提升巨大——一条 SIMD 指令顶 4-8 条标量指令。LuaJIT **不做向量化**。这是 trace JIT 的一个固有局限——trace 是线性的,向量化需要识别循环的并行性(多个独立迭代),而 trace 只看到一条路径,识别并行性的能力弱。所以纯数值计算的重循环(科学计算、图像处理),LuaJIT 通常跑不过 V8/C2。但 LuaJIT 的强项不在数值计算,而在通用逻辑(游戏脚本、网络服务、配置处理),这些循环体通常无法向量化(有分支、有间接调用),LuaJIT 的"trace 短、编译快、guard 精准"反而更合适。

**第三,guard 对循环优化的影响,是 trace vs method 的一个深层分歧。** 我们在 §1.3 讲过:动态语言 IR 全是 guard,传统 LICM 被 guard 的控制依赖阻碍。LuaJIT 用复制-替换绕开(控制依赖隐式保留),C2 因为 Java 类型确定不受影响,V8(JS,动态)受 deopt 约束但比 Lua 乐观一些(因为 V8 的类型反馈 type feedback 比 LuaJIT 的单次观测更鲁棒)。这个差异,本质上是"动态程度"的差异:Lua 比 JS 更动态(JS 至少有 V8 的 hidden class 做类型反馈),所以 LuaJIT 的 guard 更密,循环优化的激进空间更小。LuaJIT 用巧妙的设计(复制-替换)在这个约束下榨取了最大收益,但它注定做不到 C2 那种激进的向量化、展开。

**指令拆分:LuaJIT 特有的 concern**

指令拆分(split)这件事,C2/V8 **不需要做**。为什么?因为 C2/V8 运行在 64 位平台上(现代 JVM/V8 主要目标 64 位 x86/ARM),64 位运算有原生硬件支持。而且 JVM/V8 即使在 32 位平台上,也倾向于在后端处理 64 位运算(用寄存器对分配,虽然复杂但 C2/V8 有这个工程量)。

LuaJIT 做 split,是因为它要支持 32 位/软浮点的嵌入式目标(物联网设备、老旧 32 位系统),而且它追求后端简单(每个 32 位后端只实现 32 位指令)。split 把 64 位运算的复杂度集中在 IR 层一道 pass,换后端的简单。这是 LuaJIT "小而美"哲学的体现——用一道 843 行的 pass,换所有 32 位后端的简洁。

这个取舍,JVM/V8 不需要做(它们不主打 32 位/嵌入式),官方 Lua 不需要做(它没有 IR)。split 是 LuaJIT 在"支持广泛平台 + 保持后端简单"这两个目标下的特有设计。

### 6.3 对照小结

- **vs 官方 Lua**:LuaJIT 的循环优化,是在 IR 上识别循环不变量、外提、闭环循环携带依赖。官方 Lua 完全不做,每圈从头算。这是 JIT 相对解释器的又一大加速来源。
- **vs JVM/V8**:LuaJIT 的循环优化是 trace 级的,靠复制-替换 + pipeline 复用,不显式 LICM,不做向量化(短板)。C2/V8 是 method 级的,显式 LICM + 展开 + 向量化(强项),但编译更重。guard 密度的差异(动态程度),决定了 LuaJIT 的循环优化注定比 C2 保守——但它用巧妙设计在约束下榨取了最大收益。
- **指令拆分**:LuaJIT 特有(为 32 位/软浮点后端的简洁),C2/V8 不需要(64 位原生),官方 Lua 不涉及(没有 IR)。

## §7 回扣主线

回到本书的主线:**把动态执行安全变成机器码。**

这一章讲的是这条主线在 **JIT 侧优化篇(P3)的收尾**。两道 pass——循环优化和指令拆分——各自完成优化篇的最后一块拼图,把 IR 准备到"后端可以直接逐条翻机器码"的程度。

- **"变成机器码"** 体现在:循环优化让循环 trace 的机器码更精简(不变量外提、不变 guard 外提),循环跑得更快;指令拆分让 32 位平台也能生成正确的 64 位运算机器码(拆成 32 位 + HIOP)。两道 pass 都服务于"生成高质量机器码"这个最终目标。
- **"安全"** 体现在:循环优化的复制-替换保持数据流等价,所有 guard(包括外提的不变 guard)都保留并正确触发;指令拆分保证位级等价,snapshot 和 PHI 同步更新。两道 pass 都不改变语义——要么成功且结果一致,要么失败回滚/作废,绝不生成算错的机器码。失败兜底(loop_undo 退回录制、split 失败重置 trace)保证"没做成也不做错"。
- **"动态"** 体现在:LuaJIT 的循环优化刻意不做传统 LICM(因为动态语言 guard 太密),而是用复制-替换绕开这个障碍——这是它针对动态语言特性做的专门设计。guard 的密集既是动态语言的负担,也是安全的保障(每个假设都有检查),LuaJIT 的循环优化巧妙地让 guard 成为"可外提的不变量标记"(不变 guard 折叠回原指令),把负担变成了优化机会。
- **"省"** 体现在:循环优化复用已有 pipeline(FOLD/CSE/FWD 白嫖),不重写优化逻辑,`lj_opt_loop.c` 只有 453 行;指令拆分集中处理 64 位运算,让每个 32 位后端保持简单(`lj_opt_split.c` 843 行换所有 32 位后端的简洁)。两道 pass 都体现了 LuaJIT "以最小的工程量榨取最大收益"的哲学。

LuaJIT 的循环优化,还体现了一个贯穿全书的思想:**以形式换简单**。trace 的线性,让循环 SSA 不需要传统 PHI 插入算法(复制-替换自然收集 PHI);trace 的线性,让循环优化不需要传统 LICM(复制-替换 + CSE 自动外提);trace 的线性,让控制依赖不需要显式建模(复制保留)。每一个简化,都源自"只录一条线性热路径"这个根本选择。method JIT(C2/V8)就没有这个便利——它们必须处理整个方法的所有分支,所以必须用图状 IR、显式 LICM、显式控制依赖建模,复杂度自然高出一个量级。

讲到这里,P3 优化篇的全部 pass 都讲完了。从 P3-09 的常量折叠,到 P3-10 的内存优化,到 P3-11 的分配消除和死代码消除,再到本章的循环优化和指令拆分——这五道 pass 协同工作,把录制器(P2)产出的原始 IR,逐步精简、优化、规整,最终变成一份"后端可以直接逐条翻译成机器码"的 IR。

优化篇到此结束。下一章 P4-13,我们跨入后端篇——看这份优化好的 IR,怎么被**分配寄存器**(线性扫描)、**生成机器码**(汇编生成)。从"优化 IR"转入"把 IR 变成机器码",这是 JIT 流水线的后半段。优化让机器码更精简,后端让机器码真正落地——两者合起来,才完成了"把动态执行变成机器码"的全过程。
