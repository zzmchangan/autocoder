# P4-15 后端目标与机器码内存

> **本书主线**:把动态执行安全变成机器码。**二分法**:解释器侧 ↔ JIT 侧。本章在 **JIT 侧(P4 后端)**,讲后端最后两件事:怎么让一套代码支持 x86/x64/ARM/ARM64/MIPS 五种 CPU,以及生成的机器码该放在什么样的内存里、怎么管理。**★对照**:官方 Lua(无 JIT,无机器码) + JVM/V8(多后端架构支持、代码缓存)。**源码**:LuaJIT 2.1.ROLLING。**基调**:纯直球,从第一性原理推导。

---

## §0 本章在 P4 里的位置

P4 一路讲下来,到了 P4-14,我们已经把一条 IR 指令变成了机器码:寄存器分配(P4-13)决定值放哪个寄存器,汇编生成(P4-14)把 IR 操作翻译成具体的机器指令。但 P4-14 留了两个尾巴没讲,它们正好是本章的全部内容:

第一个尾巴:**P4-14 里那些"生成机器码"的函数,生成的是哪种 CPU 的机器码?** 我们贴的 `emit_rr`、`emit_jcc` 是 x86 的。但 LuaJIT 要跑在 x86、x64、ARM、ARM64、MIPS 上,这些 CPU 的寄存器完全不一样(ARM64 有 X0~X30, x86 只有 EAX~EDI),指令编码也完全不一样(ARM64 一条指令固定 4 字节, x86 是变长 1~15 字节)。一套源码怎么同时支持五种 CPU?

第二个尾巴:**生成出来的机器码,放哪?** 这听起来是个无聊的问题——放内存里呗。但事情没那么简单:普通 `malloc` 出来的内存,CPU 是**不能执行**的(现代操作系统会拒绝,直接抛段错误)。而且就算能执行,把"可写"和"可执行"同时开在一块内存上,又有安全风险。LuaJIT 要专门处理这块内存。

这两个尾巴,前一个是"多架构抽象"(target 抽象层),后一个是"可执行内存"(mcode 管理)。本章分这两大块讲透。讲完,P4 篇就完整了:从 IR 到任意架构的、放在正确内存里的、真正可执行的机器码,整条链路打通。

## §1 第一性原理:机器码不是一种东西,是五种东西

先讲第一个问题:多架构。

### §1.1 同一段逻辑,五种机器码

回想 P0-01 说过的事实:CPU 只懂机器码。但"机器码"不是一个统一的东西——**每一种 CPU 架构,有自己的机器码**。

拿一个最简单的操作举例:把寄存器 r1 和 r2 相加,结果放 r1。这个操作,在不同 CPU 上的机器码完全不同:

- **x86/x64**:`ADD r1, r2`,编码成 1~3 字节的变长指令。比如 `add eax, ecx` 是两个字节 `01 c8`。
- **ARM64**:`ADD X0, X0, X1`,编码成**固定 4 字节**的指令。寄存器编号嵌在 4 字节里的固定位段。
- **ARM(32位)**:`ADD R0, R0, R1`,也是 4 字节,但编码格式和 ARM64 不同。
- **MIPS**:又是另一套 4 字节编码,而且是 R 型指令格式(opcode+rs+rt+rd+shamt+funct)。

而且不光是编码不同,**寄存器本身也不同**:

- x86(32位)有 8 个通用寄存器(EAX/ECX/EDX/EBX/ESP/EBP/ESI/EDI)。
- x64 把它们扩展到 16 个(RAX~R15)。
- ARM64 有 31 个通用寄存器(X0~X30)加一个固定为零的 ZR。
- ARM(32位)有 16 个(R0~R15)。
- MIPS 有 32 个($0~$31,$0 恒为零)。

这意味着:你为 x86 生成的机器码,放到 ARM64 上跑,就是一堆毫无意义的字节,ARM64 的 CPU 会把它译码成乱七八糟的指令然后崩溃。反之亦然。

### §1.2 那怎么用一套代码支持五种 CPU?

现在问题清楚了:LuaJIT 的汇编生成器(P4-14 那些 `emit_*` 函数),如果直接写死成 x86 的,那它就只能支持 x86。要支持 ARM64,就得再写一份完全不同的代码。五种 CPU,写五份?

能写,但太蠢了。五份代码里有大量逻辑是**相同的**:

- "决定值放哪个寄存器"(寄存器分配,P4-13)——这部分和 CPU 无关,只和 IR 有关。
- "哪些寄存器是 caller-saved、哪些是参数寄存器"——结构相同,只是具体寄存器不同。
- "生成一条加法指令"——抽象层面都是"把两个寄存器相加",只是编码不同。

只有"把抽象操作变成具体字节"这最后一步,才是每种 CPU 独有的。所以聪明的做法是:**把相同的逻辑抽到架构无关的层,只把不同的部分留到架构相关的层。**

这就是 LuaJIT(也是所有跨平台 JIT)的设计:

- **架构无关层**(`lj_target.h`、`lj_asm.c` 的主体、寄存器分配):定义一套抽象的"寄存器""寄存器集合""操作",不关心具体是 x86 还是 ARM。
- **架构相关层**(`lj_target_x86.h`/`lj_target_arm64.h`/...、`lj_emit_x86.h`/`lj_emit_arm64.h`/...、`lj_asm_x86.h`/`lj_asm_arm64.h`/...):把抽象映射到具体,真正生成某种 CPU 的字节。

编译时,根据目标 CPU 选一个架构相关层包含进来。运行时,架构无关层调用的那些"生成加法"函数,实际上是选中那个架构的实现。一套上层逻辑,配多个底层实现——这是经典的**抽象屏障(abstraction barrier)**。

我们来看 LuaJIT 具体怎么做这道屏障。

## §2 抽象层 lj_target.h:用一套名字说所有 CPU 的寄存器

### §2.1 为什么要抽象"寄存器"

寄存器分配(P4-13)是这样工作的:它扫一遍 IR,给每个 IR 值挑一个物理寄存器(或者决定溢出到栈)。但这里有个问题——它挑的"物理寄存器",该怎么表示?

如果它直接说"挑 EAX",那就绑死 x86 了。在 ARM64 上根本没 EAX,有的是 X0。

所以寄存器分配需要一个**中性的表示**:不叫 EAX 也不叫 X0,就叫"第 0 号寄存器""第 1 号寄存器"……一个抽象编号。至于"第 0 号到底是 EAX 还是 X0",那是架构相关层的事,寄存器分配不关心。

这个抽象编号,在 LuaJIT 里叫 **RID(Register ID)**。它就是一个整数:

```c
/* lj_target.h:14 */
typedef uint32_t Reg;
```

`Reg` 类型就是个 32 位整数,装的就是 RID。RID_NONE=0x80 表示"没有寄存器",低位 0~0x7f 是真正的寄存器编号。

寄存器分配输出 `Reg`,emit 层(架构相关)拿到 `Reg` 再翻译成具体字节。中间这一层完全是数字,不带任何架构色彩。这就是第一道抽象。

### §2.2 寄存器集合 RegSet:寄存器分配的核心数据结构

寄存器分配时,要频繁做一件事:"现在有哪些寄存器是空闲的?"最自然的表示是**位图**:一个寄存器对应一个 bit,bit 为 1 表示这个寄存器在集合里(比如空闲)。

LuaJIT 叫它 `RegSet`:

```c
/* lj_target.h:55 */
/* Bitset for registers. 32 registers suffice for most architectures.
** Note that one set holds bits for both GPRs and FPRs.
*/
#if LJ_TARGET_PPC || LJ_TARGET_MIPS || LJ_TARGET_ARM64
typedef uint64_t RegSet;
#define RSET_BITS		6
#define rset_picktop_(rs)	((Reg)lj_fls64(rs))
#define rset_pickbot_(rs)	((Reg)lj_ffs64(rs))
#else
typedef uint32_t RegSet;
#define RSET_BITS		5
#define rset_picktop_(rs)	((Reg)lj_fls(rs))
#define rset_pickbot_(rs)	((Reg)lj_ffs(rs))
#endif
```

注意这里的精妙之处:**抽象层定义了 `RegSet` 是位图、怎么 pick、怎么 test,但"用几个 bit"由架构决定。** x86 只有 16 个寄存器(8 GPR + 8 FPR),32 位够装(`RSET_BITS=5`,即 2^5=32 个位);ARM64/MIPS/PPC 有 32+ 个寄存器,得用 64 位(`RSET_BITS=6`,即 2^6=64 个位)。

这是一道**有架构分支的抽象**:接口统一(都是位图操作 `rset_test`/`rset_set`/`rset_clear`),实现按架构分支(uint32 还是 uint64)。但这个分支被封装在几个宏里,上层的寄存器分配代码只管用 `rset_test(rs, r)` 这种统一调用,不知道底下是 32 位还是 64 位。

配套的一套操作宏(`lj_target.h:70`):

```c
#define RID2RSET(r)		(((RegSet)1) << (r))
#define RSET_EMPTY		((RegSet)0)
#define RSET_RANGE(lo, hi)	((RID2RSET((hi)-(lo))-1) << (lo))

#define rset_test(rs, r)	((int)((rs) >> (r)) & 1)
#define rset_set(rs, r)		(rs |= RID2RSET(r))
#define rset_clear(rs, r)	(rs &= ~RID2RSET(r))
#define rset_exclude(rs, r)	(rs & ~RID2RSET(r))
```

`RSET_RANGE(lo, hi)` 生成一个"从 lo 到 hi 的所有寄存器"的位图。`rset_picktop_`/`rset_pickbot_` 从位图里挑出最高/最低位那个寄存器(用 `lj_fls`/`lj_ffs` 这种"找最高/最低置位 bit"的快速指令)。寄存器分配就靠这套宏工作:从空闲集合里挑一个、从可用集合里去掉某个、测试某个在不在集合里。

到这里,寄存器分配需要的"工具"齐了:统一的 `Reg`(编号)、统一的 `RegSet`(位图)、统一的位操作。它对架构一无所知。

### §2.3 分发到具体架构:lj_target.h 的 #include 分发

抽象层定义完接口,接下来要把"具体架构的实现"接进来。LuaJIT 用编译期的 `#include` 分发(`lj_target.h:136`):

```c
#if LJ_TARGET_X86ORX64
#include "lj_target_x86.h"
#elif LJ_TARGET_ARM
#include "lj_target_arm.h"
#elif LJ_TARGET_ARM64
#include "lj_target_arm64.h"
#elif LJ_TARGET_PPC
#include "lj_target_ppc.h"
#elif LJ_TARGET_MIPS
#include "lj_target_mips.h"
#else
#error "Missing include for target CPU"
#endif
```

`LJ_TARGET_X86ORX64` 这些宏在 `lj_arch.h` 里根据编译时目标 CPU 定义。编译 LuaJIT 时,你只能为一种 CPU 编译(比如在 x86 机器上编,`LJ_TARGET_X86ORX64` 就是 1,其余是 0)。所以编译完的 LuaJIT 只支持一种 CPU——但**同一份源码**能编出五种 CPU 的版本。这就是"一套源码多架构"的含义:不是运行时切换,是编译时选择。

接下来每个 `lj_target_*.h` 干两件事:定义这套 CPU 的寄存器(把抽象 RID 落实)、定义这套 CPU 的指令编码格式。我们以 x86 为例看。

## §3 架构相关层 lj_target_x86.h:把抽象落实成具体

### §3.1 寄存器定义:GPRDEF/FPRDEF 宏的妙用

x86 的寄存器怎么和抽象 RID 对应?LuaJIT 用了一个很巧的宏技巧(`lj_target_x86.h:11`):

```c
#if LJ_64
#define GPRDEF(_) \
  _(EAX) _(ECX) _(EDX) _(EBX) _(ESP) _(EBP) _(ESI) _(EDI) \
  _(R8D) _(R9D) _(R10D) _(R11D) _(R12D) _(R13D) _(R14D) _(R15D)
#define FPRDEF(_) \
  _(XMM0) _(XMM1) _(XMM2) _(XMM3) _(XMM4) _(XMM5) _(XMM6) _(XMM7) \
  _(XMM8) _(XMM9) _(XMM10) _(XMM11) _(XMM12) _(XMM13) _(XMM14) _(XMM15)
#else
#define GPRDEF(_) \
  _(EAX) _(ECX) _(EDX) _(EBX) _(ESP) _(EBP) _(ESI) _(EDI)
#define FPRDEF(_) \
  _(XMM0) _(XMM1) _(XMM2) _(XMM3) _(XMM4) _(XMM5) _(XMM6) _(XMM7)
#endif
```

`GPRDEF` 是一个"寄存器名字列表"的宏模板。它接收一个宏参数 `_`,把每个寄存器名传给 `_` 调用一次。`FPRDEF` 同理是浮点寄存器列表。

然后(`lj_target_x86.h:27`):

```c
#define RIDENUM(name)	RID_##name,

enum {
  GPRDEF(RIDENUM)		/* General-purpose registers (GPRs). */
  FPRDEF(RIDENUM)		/* Floating-point registers (FPRs). */
  RID_MAX,
  ...
};
```

`RIDENUM(EAX)` 展开成 `RID_EAX,`,所以 `GPRDEF(RIDENUM)` 展开成一串 `RID_EAX, RID_ECX, ...`。放进 enum 里,就自动得到了 `RID_EAX=0, RID_ECX=1, RID_EDX=2, ...` 这样的编号。

**这就是抽象 RID 和具体寄存器的对应关系:RID 0 = EAX,RID 1 = ECX,以此类推。** 寄存器分配输出的"RID 0",在 x86 上就是 EAX;同样的"RID 0",在 ARM64 上就是 X0(因为 ARM64 的 `GPRDEF` 第一个是 `X0`)。抽象层和架构层靠这个 enum 接口对接。

这个宏技巧的好处:加寄存器只要改 `GPRDEF` 一个地方,enum、数量统计(`RID_NUM_GPR`)都自动更新,不会漏。

### §3.2 寄存器的"角色":调用约定相关的特殊 RID

寄存器除了编号,还有"角色":哪个是栈指针、哪个放返回值、哪几个是参数寄存器、哪个是解释器专用的 BASE/PC。这些和 **ABI(调用约定)** 强相关,也是架构独有的。x86 的定义(`lj_target_x86.h:36`):

```c
  /* Calling conventions. */
  RID_SP = RID_ESP,
  RID_RET = RID_EAX,
#if LJ_64
  RID_FPRET = RID_XMM0,
#endif
  RID_RETLO = RID_EAX,
  RID_RETHI = RID_EDX,

  /* These definitions must match with the *.dasc file(s): */
  RID_BASE = RID_EDX,		/* Interpreter BASE. */
#if LJ_64 && !LJ_ABI_WIN
  RID_LPC = RID_EBX,		/* Interpreter PC. */
  RID_DISPATCH = RID_R14D,	/* Interpreter DISPATCH table. */
#else
  RID_LPC = RID_ESI,		/* Interpreter PC. */
  RID_DISPATCH = RID_EBX,	/* Interpreter DISPATCH table. */
#endif
```

注意几个要点:

1. **`RID_SP = RID_ESP`**:x86 的栈指针是 ESP。ARM64 的栈指针是 SP(它 `RID_ZERO = RID_SP`)。抽象层只要知道"有个 RID_SP",具体是谁由架构决定。
2. **`RID_RET = RID_EAX`**:函数返回值放 EAX。x86 的调用约定就这么定的。ARM64 的返回值放 X0(`RID_RET = RID_X0`)。
3. **`RID_BASE = RID_EDX`**:这是**解释器**用的——LuaJIT 解释器(P1-02)把"当前栈基址"固定放 EDX。注意注释 `must match with the *.dasc file(s)`:解释器是 dynasm 手写的(P1-02 讲过),那里写死了 BASE 用哪个寄存器,这里必须一致,否则 JIT 生成的代码和解释器对接不上。

这些"角色寄存器"是抽象层和架构层、JIT 和解释器的契约。寄存器分配知道这些角色,会避开它们(不把 BASE 分配给普通 IR 值)。

### §3.3 寄存器集合:RSET_GPR / RSET_FPR / RSET_SCRATCH

有了寄存器编号,还要定义"哪些寄存器可分配"。x86 的(`lj_target_x86.h:66`):

```c
/* Make use of all registers, except the stack pointer (and maybe DISPATCH). */
#define RSET_GPR	(RSET_RANGE(RID_MIN_GPR, RID_MAX_GPR) \
				 - RID2RSET(RID_ESP) \
				 - LJ_GC64*RID2RSET(RID_DISPATCH))
#define RSET_FPR	(RSET_RANGE(RID_MIN_FPR, RID_MAX_FPR))
#define RSET_ALL	(RSET_GPR|RSET_FPR)
#define RSET_INIT	RSET_ALL
```

`RSET_GPR` 是"所有通用寄存器,但去掉栈指针 ESP(不能占用栈指针,否则栈就乱了),GC64 模式下再去掉 DISPATCH"。这就是 §2.2 那套位图宏的实际应用:用 `RSET_RANGE` 圈一个范围,再用减法抠掉不能用的。

还有"调用者保存寄存器集合"`RSET_SCRATCH`(`lj_target_x86.h:84`),分 Windows x64 和 System V x64 两套 ABI——因为两种 ABI 的调用约定不同(哪些寄存器调用时会被 clobber)。

```c
#if LJ_ABI_WIN
/* Windows x64 ABI. */
#define RSET_SCRATCH \
  (RSET_ACD|RSET_RANGE(RID_R8D, RID_R11D+1)|RSET_RANGE(RID_XMM0, RID_XMM5+1))
...
#else
/* The rest of the civilized x64 world has a common ABI. */
#define RSET_SCRATCH \
  (RSET_ACD|RSET_RANGE(RID_ESI, RID_R11D+1)|RSET_FPR)
...
#endif
```

为什么要有 `RSET_SCRATCH`?因为 JIT 生成的代码里会调用 C 函数(比如 `lj_gc_step`、`lj_strscan_num`),调用 C 函数按 ABI 规定会 clobber(破坏)一批"调用者保存"寄存器。寄存器分配时,要提前把这些寄存器里还活着的值挪走(evict),否则一调用就丢。`RSET_SCRATCH` 就是"会被 clobber 的寄存器集合",给 evict 用。

注意所有这些集合定义,都是用 §2.2 的抽象宏(`RSET_RANGE`、`RID2RSET`、减法)组合出来的。**架构层只提供"具体寄存器名单",集合运算逻辑是抽象层共享的。** 这是抽象屏障发挥作用的实例。

### §3.4 指令编码:x86Op 枚举(把指令变成字节)

寄存器搞定了,接下来是指令。x86 的指令怎么表示?

x86 指令是变长的(1~15 字节),编码很复杂:可能有前缀(REX/66/F2/F3)、操作码(1~3 字节)、ModRM 字节、SIB 字节、位移、立即数。要生成一条指令,得把这些字节按正确顺序拼出来。

LuaJIT 没有用字符串或结构体表示指令,而是用一个**紧凑的 32 位整数**编码一条指令的全部信息(`lj_target_x86.h:187`):

```c
/* Macros to construct variable-length x86 opcodes. -(len+1) is in LSB. */
#define XO_(o)		((uint32_t)(0x0000fe + (0x##o<<24)))
#define XO_FPU(a,b)	((uint32_t)(0x00fd + (0x##a<<16)+(0x##b<<24)))
#define XO_0f(o)	((uint32_t)(0x0f00fd + (0x##o<<24)))
#define XO_66(o)	((uint32_t)(0x6600fd + (0x##o<<24)))
#define XO_660f(o)	((uint32_t)(0x0f66fc + (0x##o<<24)))
#define XO_f20f(o)	((uint32_t)(0x0ff2fc + (0x##o<<24)))
#define XO_f30f(o)	((uint32_t)(0x0ff3fc + (0x##o<<24)))
```

这套 `XO_*` 宏把一条 x86 指令压成一个 32 位整数:低字节存"指令长度取反"(所以 `-(len+1)`,用补码,emit 时 `int8_t` 取出来就是负数),高位存实际的操作码字节。

举例,`XO_MOV = XO_(8b)`(`lj_target_x86.h:250`),展开就是一个 32 位数,里面编码了"`8b` 是 MOV 指令的操作码,长度 2 字节"。`XO_ADDSD = XO_f20f(58)`(`lj_target_x86.h:290`)编码了"`f2 0f 58` 是 ADDSD(双精度浮点加法),3 字节加 1 字节前缀"。

为什么这么设计?因为 emit 层生成指令时,要处理"指令多长""前缀是什么""操作码字节是什么"——把这些塞进一个 32 位整数,emit 函数就能用一个统一的逻辑(§4 会讲 `emit_op`)处理所有指令,不用为每条指令写专门代码。这是把 x86 指令的复杂性**数据化**:不是用代码分支处理各种指令格式,而是用数据(32 位整数里的位段)描述各种格式。

配套的还有条件码枚举(`lj_target_x86.h:352`):

```c
/* x86 condition codes. */
typedef enum {
  CC_O, CC_NO, CC_B, CC_NB, CC_E, CC_NE, CC_BE, CC_NBE,
  CC_S, CC_NS, CC_P, CC_NP, CC_L, CC_NL, CC_LE, CC_NLE,
  CC_C = CC_B, CC_NAE = CC_C, CC_NC = CC_NB, CC_AE = CC_NB,
  CC_Z = CC_E, CC_NZ = CC_NE, CC_NA = CC_BE, CC_A = CC_NBE,
  ...
} x86CC;
```

`CC_E` 是"相等则跳"(condition code for jcc),`CC_NE` 是"不等则跳",等等。生成 guard(条件跳转)时要用到——`asm_guardcc(as, CC_NE)` 就是"生成一条:不等则跳到 exit 的指令"。这又是抽象:P4-14 讲过 guard 在 IR 层是 `IR_NE`/`IR_LT` 这种抽象比较,到了 emit 层才变成具体的 `CC_NE`/`CC_L`。

还有 ModRM 操作数的编码(`lj_target_x86.h:171`):

```c
typedef enum {
  XM_OFS0 = 0x00, XM_OFS8 = 0x40, XM_OFS32 = 0x80, XM_REG = 0xc0,
  XM_SCALE1 = 0x00, XM_SCALE2 = 0x40, XM_SCALE4 = 0x80, XM_SCALE8 = 0xc0,
  XM_MASK = 0xc0
} x86Mode;

typedef struct {
  int32_t ofs;		/* Offset. */
  uint8_t base;		/* Base register or RID_NONE. */
  uint8_t idx;		/* Index register or RID_NONE. */
  uint8_t scale;	/* Index scale (XM_SCALE1 .. XM_SCALE8). */
} x86ModRM;
```

x86 的内存操作数是 `段:[base + index*scale + offset]` 这种形式,ModRM 字节编码其中一部分。`x86ModRM` 结构装一个"变量内存操作数"(base/index/scale/ofs),emit 层据此生成 ModRM+SIB 字节。这也是 x86 独有的——ARM64 没有这种复杂的寻址模式。

到此,架构层提供了:**寄存器(RID 枚举)、寄存器集合(RSET_*)、指令编码(x86Op)、条件码(x86CC)、寻址模式(x86ModRM)**。这套"词汇表"是 emit 层的输入:emit 层拿到抽象 IR 操作,用这套词汇表拼出 x86 机器码字节。

ARM64 的 `lj_target_arm64.h` 提供的是另一套词汇表(ARM64 的寄存器 X0~X30、ARM64 的条件码、ARM64 的指令格式),但**对外接口和 x86 一样**:都是给 emit 层提供"这台机器的词汇表"。

## §4 emit 层 lj_emit_x86.h:架构相关的最后一层

### §4.1 emit 层在编译流水线里的位置

抽象层(lj_asm.c 主体)做完寄存器分配、决定"这条 IR 要生成什么操作"后,调 emit 层。emit 层是**架构相关的最后一层**——它把"ADD r1, r2"这种抽象操作,变成具体的字节 `01 c8` 写进机器码缓冲区。

P4-14 已经讲过 emit 层怎么和 IR 衔接。这里重点讲 emit 层**本身**怎么工作——它怎么把一条 x86 指令的字节拼出来。

### §4.2 机器码缓冲区:从顶向下写

先看一个容易困惑的点:**LuaJIT 的机器码是从高地址往低地址写的。**

为什么?看 `lj_emit_x86.h` 里的写法:

```c
#define emit_i8(as, i)		(*--as->mcp = (MCode)(i))
#define emit_i32(as, i)	(*(int32_t *)(as->mcp-4) = (i), as->mcp -= 4)
```

`as->mcp` 是"当前写指针",每写一个字节就 `*--as->mcp = ...`,即**先减再写**(指针往低地址走)。

这是因为 P4-14 讲过的原因:代码生成是**反向**的(从最后一条 IR 往前生成),因为反向才能做寄存器分配的死区间断点判断。反向生成,机器码就自然从高往低长。`as->mcp` 从 `mctop`(机器码区顶部)开始,一路减到 `mcbot`(底部)。最后 `T->mcode = as->mcp`,记录"机器码起点"(也就是写完后的最低地址)。

### §4.3 emit_op:把 32 位整数指令展开成字节

核心函数是 `emit_op`(`lj_emit_x86.h:37`),它把 §3.4 那个 32 位整数 `x86Op` 展开成实际字节:

```c
static LJ_AINLINE MCode *emit_op(x86Op xo, Reg rr, Reg rb, Reg rx,
				 MCode *p, int delta)
{
  int n = (int8_t)xo;
  if (n == -60) {  /* VEX-encoded instruction */
#if LJ_64
    xo ^= (((rr>>1)&4)+((rx>>2)&2)+((rb>>3)&1))<<13;
#endif
    *(uint32_t *)(p+delta-5) = (uint32_t)xo;
    return p+delta-5;
  }
#if defined(__GNUC__) || defined(__clang__)
  if (__builtin_constant_p(xo) && n == -2)
    p[delta-2] = (MCode)(xo >> 24);
  else if (__builtin_constant_p(xo) && n == -3)
    *(uint16_t *)(p+delta-3) = (uint16_t)(xo >> 16);
  else
#endif
    *(uint32_t *)(p+delta-5) = (uint32_t)xo;
  p += n + delta;
  ...
}
```

回忆 §3.4:`x86Op` 的低字节是 `-(len+1)`,即指令长度的编码。`(int8_t)xo` 取出来是个负数(`-2` 表示 1 字节操作码、`-3` 表示 2 字节、`-5` 表示 4 字节...)。`emit_op` 用这个负数算出"这条指令多长",然后把高位存的操作码字节写到缓冲区。

`__builtin_constant_p` 是个优化:如果 `xo` 是编译期常量(大多数情况),直接按已知长度写,不用走通用的 4 字节写入。这能让编译器生成更紧凑的写指令代码。

`rr/rb/rx` 是寄存器编号,用来生成 REX 前缀(x64 扩展的寄存器需要 REX 前缀指明):

```c
#if LJ_64
  {
    uint32_t rex = 0x40 + ((rr>>1)&(4+(FORCE_REX>>1)))+((rx>>2)&2)+((rb>>3)&1);
    if (rex != 0x40) {
      rex |= (rr >> 16);
      if (n == -4) { *p = (MCode)rex; rex = (MCode)(xo >> 8); }
      else if ((xo & 0xffffff) == 0x6600fd) { *p = (MCode)rex; rex = 0x66; }
      *--p = (MCode)rex;
    }
  }
#endif
```

x64 下,如果用到了 R8~R15 这些扩展寄存器(编号 ≥ 8),就要在指令前面加 REX 前缀(0x40~0x4f)。REX 字节里的位编码"操作数寄存器、基址寄存器、变址寄存器是不是扩展的"。`emit_op` 根据 `rr/rb/rx` 算出要不要 REX、REX 是什么,写进去。

这是 x86/x64 独有的麻烦事——ARM64 寄存器直接嵌在 4 字节指令的位段里,没有前缀这一说。所以 `emit_op` 这种函数,完全是 x86 emit 层独有的。

### §4.4 高层 emit 函数:emit_rr / emit_rmro / emit_jcc

在 `emit_op` 之上,封装了一批"语义化"的 emit 函数,对应 x86 的各种指令形式:

**emit_rr**——寄存器到寄存器(`lj_emit_x86.h:92`):

```c
/* op r1, r2 */
static void emit_rr(ASMState *as, x86Op xo, Reg r1, Reg r2)
{
  MCode *p = as->mcp;
  as->mcp = emit_opm(xo, XM_REG, r1, r2, p, 0);
}
```

`emit_rr(as, XO_ADD, r1, r2)` 生成 `ADD r1, r2`。`XM_REG` 表示"ModRM 的 mode 字段是 11(寄存器-寄存器)"。

**emit_rmro**——寄存器到内存(`lj_emit_x86.h:110`),生成 `op r, [base+ofs]`:

```c
/* op r, [base+ofs] */
static void emit_rmro(ASMState *as, x86Op xo, Reg rr, Reg rb, int32_t ofs)
{
  MCode *p = as->mcp;
  x86Mode mode;
  if (ra_hasreg(rb)) {
    ...
    if (ofs == 0 && (rb&7) != RID_EBP) {
      mode = XM_OFS0;          /* 无偏移 */
    } else if (checki8(ofs)) {
      *--p = (MCode)ofs;
      mode = XM_OFS8;          /* 8位偏移 */
    } else {
      p -= 4;
      *(int32_t *)p = ofs;
      mode = XM_OFS32;         /* 32位偏移 */
    }
    ...
  }
  as->mcp = emit_opm(xo, mode, rr, rb, p, 0);
}
```

这里能看到 x86 变长编码的体现:偏移量是 0 就用 `XM_OFS0`(省掉偏移字节)、能放进 8 位就用 `XM_OFS8`(1 字节偏移)、否则用 `XM_OFS32`(4 字节偏移)。emit 函数自动选最短的编码——这是 LuaJIT 机器码紧凑的一个原因。

**emit_jcc**——条件跳转(`lj_emit_x86.h:502`),guard 用:

```c
/* jcc target */
static void emit_jcc(ASMState *as, int cc, MCode *target)
{
  MCode *p = as->mcp;
  *(int32_t *)(p-4) = jmprel(as->J, p, target);
  p[-5] = (MCode)(XI_JCCn+(cc&15));
  p[-6] = 0x0f;
  as->mcp = p - 6;
}
```

生成 `0f 8x xx xx xx xx`(6 字节的近条件跳转)。`cc` 是条件码(§3.4 的 `x86CC`),`target` 是跳转目标(通常是 exit stub,即 guard 失败要去的地方)。`jmprel` 算"目标地址减当前地址"的相对偏移(`lj_emit_x86.h:493`):

```c
static LJ_AINLINE int32_t jmprel(jit_State *J, MCode *p, MCode *target)
{
  ptrdiff_t delta = target - p;
  UNUSED(J);
  lj_assertJ(delta == (int32_t)delta, "jump target out of range");
  return (int32_t)delta;
}
```

**关键点:x86 的跳转是"相对跳转"**——指令里存的是"跳多远"(目标地址减当前地址),不是绝对地址。这就引出了本章后半部分的一个大问题:如果跳转目标和当前指令隔得太远(超过 32 位能表示的范围),就跳不过去了。这个范围限制,直接决定了机器码内存要怎么分配(§6 会讲)。

### §4.5 emit_call:调用 C 函数

JIT 代码经常要调用 C 函数(GC、字符串扫描、错误处理)。`emit_call`(`lj_emit_x86.h:521`):

```c
/* call target */
static void emit_call_(ASMState *as, MCode *target)
{
  MCode *p = as->mcp;
#if LJ_64
  if (!jmprel_ok(target, p)) {
    /* Assumes RID_RET is never an argument to calls and always clobbered. */
    emit_rr(as, XO_GROUP5, XOg_CALL, RID_RET);
    emit_loadu64(as, RID_RET, (uint64_t)target);
    return;
  }
#endif
  *(int32_t *)(p-4) = jmprel(as->J, p, target);
  p[-5] = XI_CALL;
  as->mcp = p - 5;
}
```

x86 的 `CALL` 也是相对调用(存偏移)。但 x64 下,被调用的 C 函数可能离当前机器码超过 2GB(比如解释器代码在另一块内存),32 位偏移够不着。这时候 `jmprel_ok` 返回 false,走 fallback:先把目标地址用 `emit_loadu64` 装进一个寄存器(借用了 `RID_RET`,因为返回值寄存器在调用时必然被 clobber,拿来临时存地址不冲突),再用间接 call(`call [reg]`)。

这个 fallback 又一次提醒我们:**跳转范围是个硬约束,它逼着 LuaJIT 在内存分配上做文章。**

到此,emit 层讲完了。它是一组"把抽象操作 + 寄存器编号,翻译成具体架构字节"的函数。架构无关层(lj_asm.c)做完决策后,调这些函数;这些函数是架构相关的,但接口统一(都是 `emit_xxx(as, 操作, 寄存器, ...)`)。换个架构,换一套 emit 函数实现,上层不动。

## §5 dynasm:解释器的"可移植汇编",和 JIT 后端的关系

### §5.1 两个不同的"生成机器码"需求

讲完 emit 层,要澄清一个容易混淆的点:LuaJIT 里有**两套**"生成机器码"的机制,服务于不同目的。

**第一套:解释器(P1)的机器码。** LuaJIT 的解释器本身是手写汇编写的(`lj_vm.s` 或 `.dasc` 文件),它是一段**预先编好**的、固定的机器码,负责执行字节码。这段机器码在 LuaJIT 编译时就生成了,运行时不变。

但手写汇编有个问题:要支持五种 CPU,就得手写五份汇编——而且 x86 汇编语法、ARM 汇编语法、MIPS 汇编语法都不一样,用各自架构的汇编器写,工作量巨大且易错。

Mike Pall 写了个工具解决这个问题,叫 **dynasm**(Dynamic Assembler)。它是一种"可移植的汇编语言":你用 dynasm 的语法写一份 `.dasc` 文件,里面用架构无关的宏描述指令,dynasm 工具(`dynasm.lua`)在编译时把它翻译成对应架构的 C 代码(一个生成机器码字节的函数),再编译进 LuaJIT。一份 `.dasc`,五种 CPU。

dynasm 目录下有这些文件(`luajit2/dynasm/`):

```
dynasm.lua      主工具
dasm_x86.lua    x86/x64 后端
dasm_x64.lua
dasm_arm.lua    ARM 后端
dasm_arm64.lua  ARM64 后端
dasm_ppc.lua    PowerPC 后端
dasm_mips.lua   MIPS 后端
dasm_mips64.lua MIPS64 后端
dasm_x86.h / dasm_arm.h / ...  各后端的 C 运行时支持
```

每种架构一个 lua 后端 + 一个 C 头。`.dasc` 文件用 dynasm 宏写,编译时按目标架构选后端翻译。P1-02 讲解释器时已经详细讲过这套机制。

**第二套:JIT 后端(P4)的机器码。** 这是我们本章讲的——JIT 在**运行时**动态生成机器码(trace 的机器码)。这套用的是 §3、§4 讲的 `lj_target_*.h` + `lj_emit_*.h` + `lj_asm_*.h`,**不**用 dynasm。

### §5.2 为什么 JIT 后端不用 dynasm

一个自然的问题:既然 dynasm 能"一份代码多架构",JIT 后端为什么不用它,而要自己搞一套 target/emit 抽象?

因为两者的需求根本不同:

- **dynasm(解释器)的需求:静态、完整、可读。** 解释器代码是固定的,在编译时一次性生成。dynasm 适合写一大段完整的、人能读的汇编程序。它的输入是汇编源码,输出是机器码字节序列。
- **JIT 后端的需求:动态、增量、数据驱动。** JIT 要在运行时,根据实际的 IR(每条 IR 指令、每个寄存器分配结果都是运行时才知道的),**一条一条地**生成机器码。它的输入不是汇编源码,而是 IR + 寄存器分配结果这种数据结构。

dynasm 不擅长"根据数据结构动态拼指令"——它的范式是"写汇编源码"。而 JIT 需要的是"IR 操作 → 字节"的程序化映射,这正是 §3/§4 那套 `x86Op` 枚举 + `emit_op` 函数干的事:用数据(32 位整数)描述指令,用函数把数据变成字节。

所以 LuaJIT 有两套机制,各司其职:

| 维度 | dynasm(解释器) | lj_target + lj_emit(JIT 后端) |
|---|---|---|
| 服务对象 | 解释器(`lj_vm`) | JIT 生成的 trace 机器码 |
| 生成时机 | 编译时(静态) | 运行时(动态) |
| 输入 | `.dasc` 汇编源码 | IR + 寄存器分配结果 |
| 输出 | 固定的机器码字节 | 运行时拼出的机器码字节 |
| 多架构方式 | 编译时选 dynasm 后端翻译 | 编译时选 `lj_target_*.h`/`lj_emit_*.h` |

两者有一点是共通的:**都是"一份逻辑,配多个架构后端,编译时选择"**。这是跨平台 JIT 的通用模式。

## §6 第一性原理:机器码必须放在"可执行"的内存里

现在讲本章第二大块:可执行内存。这是和"多架构"完全独立的一个问题——不管哪种架构的机器码,都要放对地方才能跑。

### §6.1 普通内存不能执行

回到 §0 的那个问题:生成出来的机器码,放哪?

直觉答案:放内存里,`malloc` 一块,把字节写进去,然后把指针当函数调用。代码大概这样:

```c
char *buf = malloc(1024);
buf[0] = 0xc3;  /* ret 指令 */
((void(*)())buf)();  /* 调用它 */
```

这在 20 年前的系统上能跑。但在**现代操作系统**上,这段代码会崩溃——通常是段错误(SIGSEGV)。

为什么?因为现代操作系统对内存实行**页保护**:每一页内存(通常 4KB)都有一组权限位,标记这页"可读/可写/可执行"。`malloc` 出来的内存,默认权限是"可读可写",**没有可执行权限**。CPU 尝试执行这块内存里的字节时,内存管理单元(MMU)发现"这页不可执行",触发一个保护异常,操作系统把进程杀掉。

这个机制叫 **DEP/NX Bit**(Data Execution Prevention / No-eXecute)。它的目的是防止安全漏洞:很多攻击手法是把恶意代码塞进数据缓冲区(比如栈上的字符串),然后跳过去执行。如果数据内存不可执行,这种攻击就失效了。

### §6.2 JIT 的困境:必须往可执行内存里写代码

DEP 对普通程序是好事,但对 JIT 是个大麻烦。JIT 的核心功能就是**在运行时生成机器码并执行**——这恰恰是 DEP 想阻止的"往内存里写代码然后执行"。

所以 JIT 编译器不能复用普通的 `malloc`,必须用操作系统提供的特殊接口,申请一块**明确标记为可执行**的内存。不同操作系统的接口不同:

- **POSIX(Linux/macOS/BSD)**:`mmap` 系统调用,带 `PROT_EXEC` 标志。
- **Windows**:`VirtualAlloc` API,带 `PAGE_EXECUTE_READ` 或 `PAGE_EXECUTE_READWRITE` 标志。

来看 LuaJIT 怎么封装(`lj_mcode.c:80` 起)。

### §6.3 Windows 版:mcode_alloc_at

```c
#if LJ_TARGET_WINDOWS

#define MCPROT_RW	PAGE_READWRITE
#define MCPROT_RX	PAGE_EXECUTE_READ
#define MCPROT_RWX	PAGE_EXECUTE_READWRITE

static void *mcode_alloc_at(uintptr_t hint, size_t sz, DWORD prot)
{
  return LJ_WIN_VALLOC((void *)hint, sz,
		       MEM_RESERVE|MEM_COMMIT|MEM_TOP_DOWN, prot);
}

static void mcode_free(void *p, size_t sz)
{
  UNUSED(sz);
  VirtualFree(p, 0, MEM_RELEASE);
}

static void mcode_setprot(jit_State *J, void *p, size_t sz, DWORD prot)
{
#if LUAJIT_SECURITY_MCODE != 0
  DWORD oprot;
  if (!LJ_WIN_VPROTECT(p, sz, prot, &oprot)) mcode_protfail(J);
#else
  UNUSED(J); UNUSED(p); UNUSED(sz); UNUSED(prot);
#endif
}
```

`mcode_alloc_at` 调 `VirtualAlloc` 申请内存,`prot` 参数决定权限(可读可写 `MCPROT_RW` / 可读可执行 `MCPROT_RX` / 全开 `MCPROT_RWX`)。`MEM_TOP_DOWN` 是从地址空间顶部往下分配(后面 §7 会讲为什么)。`mcode_setprot` 后面用来**改权限**(把已分配的内存从"可写"切到"可执行")。

### §6.4 POSIX 版:mmap

```c
#elif LJ_TARGET_POSIX

#include <sys/mman.h>

#define MCPROT_RW	(PROT_READ|PROT_WRITE)
#define MCPROT_RX	(PROT_READ|PROT_EXEC)
#define MCPROT_RWX	(PROT_READ|PROT_WRITE|PROT_EXEC)
...

static void *mcode_alloc_at(uintptr_t hint, size_t sz, int prot)
{
  void *p = mmap((void *)hint, sz, prot|MCPROT_CREATE, MAP_PRIVATE|MAP_ANONYMOUS|MCMAP_CREATE, -1, 0);
  if (p == MAP_FAILED) return NULL;
#if MCMAP_CREATE
  pthread_jit_write_protect_np(0);
#endif
  return p;
}

static void mcode_free(void *p, size_t sz)
{
  munmap(p, sz);
}

static void mcode_setprot(jit_State *J, void *p, size_t sz, int prot)
{
#if LUAJIT_SECURITY_MCODE != 0
#if MCMAP_CREATE
  UNUSED(J); UNUSED(p); UNUSED(sz);
  pthread_jit_write_protect_np((prot & PROT_EXEC));
  return 0;
#else
  if (mprotect(p, sz, prot)) mcode_protfail(J);
#endif
#else
  UNUSED(J); UNUSED(p); UNUSED(sz); UNUSED(prot);
#endif
}
```

`mmap` 申请匿名内存,`prot` 控制权限。注意 `MCMAP_CREATE`:这是 macOS Hardened Runtime(强化运行时)的特殊处理——这种模式下,要用 `MAP_JIT` 标志申请一块特殊的"JIT 内存",并通过 `pthread_jit_write_protect_np` 在"可写"和"可执行"之间切换(不能同时)。这是 Apple 的 W^X 强化版,后面 §8 讲。

### §6.5 缓存同步:lj_mcode_sync

还有一个细节。有些 CPU 架构(ARM/ARM64/PPC)有**分离的指令缓存(I-cache)和数据缓存(D-cache)**。你往内存写机器码字节,写的是 D-cache;但 CPU 执行时从 I-cache 取指令。如果不同步,可能执行到 I-cache 里的旧数据(而不是你刚写的机器码)。x86/x64 是统一的缓存,没这个问题。

所以写完机器码、执行之前,要刷新 I-cache。`lj_mcode_sync`(`lj_mcode.c:42`):

```c
/* Synchronize data/instruction cache. */
void lj_mcode_sync(void *start, void *end)
{
#ifdef LUAJIT_USE_VALGRIND
  VALGRIND_DISCARD_TRANSLATIONS(start, (char *)end-(char *)start);
#endif
#if LJ_TARGET_X86ORX64
  UNUSED(start); UNUSED(end);
#elif LJ_TARGET_WINDOWS
  FlushInstructionCache(GetCurrentProcess(), start, (char *)end-(char *)start);
#elif LJ_TARGET_IOS
  sys_icache_invalidate(start, (char *)end-(char *)start);
#elif LJ_TARGET_PPC
  lj_vm_cachesync(start, end);
#elif defined(__GNUC__) || defined(__clang__)
  __clear_cache(start, end);
#else
#error "Missing builtin to flush instruction cache"
#endif
}
```

x86/x64 直接 `UNUSED`(不需要);Windows 用 `FlushInstructionCache`;GCC/Clang 用 `__clear_cache` 内建函数。每次写完机器码(生成 trace 时、patch 时)都要调它,确保 CPU 执行的是最新写的字节。

## §7 第一性原理:为什么机器码内存要集中(跳转范围)

讲完"怎么申请可执行内存",还有一个关键约束:**申请出来的内存,位置不能随便放。**

### §7.1 跳转范围的硬约束

回忆 §4.4:x86 的跳转(`jmp`/`jcc`/`call`)是**相对跳转**,指令里存的是 32 位偏移(近跳转)。32 位有符号偏移能覆盖 ±2GB。

但 x86 还有一种**短跳转**(8 位偏移,±127 字节)和**近跳转**(32 位偏移,±2GB)。LuaJIT 生成的 guard 用近跳转(`emit_jcc` 那个 6 字节的 `0f 8x`)。

±2GB 听起来很大,但在 64 位地址空间里,两块内存完全可能隔得超过 2GB。如果 JIT 生成的 trace 机器码在一块内存,而它要跳转的目标(另一个 trace、或解释器代码、或 exit stub)在另一块内存,两者距离超过 2GB,跳转指令就够不着了——机器码会生成失败。

不同架构的跳转范围不同(`lj_arch.h`):

```c
/* x86/x64 */
#define LJ_TARGET_JUMPRANGE	31	/* +-2^31 = +-2GB */
/* ARM(32位) */
#define LJ_TARGET_JUMPRANGE	25	/* +-2^25 = +-32MB */
/* ARM64 */
#define LJ_TARGET_JUMPRANGE	27	/* +-2^27 = +-128MB */
/* MIPS */
#define LJ_TARGET_JUMPRANGE	28	/* 2^28 = 256MB-aligned region */
```

`LJ_TARGET_JUMPRANGE` 是"这种架构的跳转指令能覆盖多少范围"的位数。x64 是 31(±2GB),ARM 只有 25(±32MB),更紧张。

### §7.2 LuaJIT 的对策:把机器码集中在一个范围内

为了让所有跳转都能成功,LuaJIT 把**所有 JIT 生成的机器码,集中申请在一个有限的地址范围内**,这个范围以"解释器代码地址"为中心。这样 trace 机器码、解释器、exit stub 互相之间都不会太远,跳转一定够得着。

这个逻辑在 `mcode_alloc`(`lj_mcode.c:295`):

```c
#ifdef LJ_TARGET_JUMPRANGE

#define MCODE_RANGE64	((1u << LJ_TARGET_JUMPRANGE) - 0x10000u)

/* Set a memory range for mcode allocation with addr in the middle. */
static void mcode_setrange(jit_State *J, uintptr_t addr)
{
#if LJ_TARGET_MIPS
  /* Use the whole 256MB-aligned region. */
  J->mcmin = addr & ~(uintptr_t)((1u << LJ_TARGET_JUMPRANGE) - 1);
  J->mcmax = J->mcmin + (1u << LJ_TARGET_JUMPRANGE);
#else
  /* Every address in the 64KB-aligned range should be able to reach
  ** any other, so MCODE_RANGE64 is only half the (signed) branch range.
  */
  J->mcmin = (addr - (MCODE_RANGE64 >> 1) + 0xffffu) & ~(uintptr_t)0xffffu;
  J->mcmax = J->mcmin + MCODE_RANGE64;
#endif
  /* Avoid wrap-around and the 64KB corners. */
  if (addr < J->mcmin || !J->mcmin) J->mcmin = 0x10000u;
  if (addr > J->mcmax) J->mcmax = ~(uintptr_t)0xffffu;
}
```

`mcode_setrange` 以某个地址(通常是解释器代码地址 `lj_vm_exit_handler`)为中心,算出一个 `[mcmin, mcmax]` 的范围,所有机器码都只能申请在这个范围里。

注意注释的关键点:`MCODE_RANGE64 is only half the (signed) branch range`——范围只取跳转范围的一半。为什么?因为跳转是"从这个范围里的任意一点,跳到这个范围里的任意一点"。如果范围取满跳转范围,那从范围的一头跳到另一头可能就超了(因为有符号偏移)。取一半,保证范围里任意两点之间的距离都不超过跳转范围的一半,双向都能跳到。

`mcode_alloc` 接着用 hint(提示地址)试探性地申请,检查申请到的地址是否在范围内,不行就换地方(`lj_mcode.c:295` 主体):

```c
static void *mcode_alloc(jit_State *J, size_t sz)
{
  uintptr_t hint;
  int i = 0, j;
  if (!J->mcmin)  /* Place initial range near the interpreter code. */
    mcode_setrange(J, (uintptr_t)(void *)lj_vm_exit_handler);
  else if (!J->mcmax)  /* Switch to a new range (already flushed). */
    goto newrange;
  /* First try a contiguous area below the last one (if in range). */
  hint = (uintptr_t)J->mcarea - sz;
  if (!mcode_inrange(J, hint, sz))  /* Also takes care of NULL J->mcarea. */
    goto probe;
  for (; i < 16; i++) {
    void *p = mcode_alloc_at(hint, sz, MCPROT_GEN);
    if (mcode_inrange(J, (uintptr_t)p, sz))
      return p;  /* Success. */
    else if (p)
      mcode_free(p, sz);  /* Free badly placed area. */
  probe:
    /* Next try probing 64KB-aligned pseudo-random addresses. */
    j = 0;
    do {
      hint = J->mcmin + (lj_prng_u64(&J2G(J)->prng) & MCODE_RANGE64);
      if (++j > 15) goto fail;
    } while (!mcode_inrange(J, hint, sz));
  }
fail:
  ...
}
```

策略是:先试"上一块机器码旁边"(连续分配,提高局部性),操作系统返回的地址不在范围内就释放掉重试;连续 16 次不行,就改成"在范围内随机挑地址"试探,最多 15 次;全不行,就放弃当前范围,新开一个范围(`newrange`)。

`mcode_inrange`(`lj_mcode.c:288`)检查地址是否落在 `[mcmin, mcmax]`:

```c
static LJ_AINLINE int mcode_inrange(jit_State *J, uintptr_t addr, size_t sz)
{
  /* Take care of unsigned wrap-around of addr + sz, too. */
  return addr >= J->mcmin && addr + sz >= J->mcmin && addr + sz <= J->mcmax;
}
```

这套机制保证:无论申请多少块机器码,它们都集中在一个跳转能互相够得着的范围内。这是 §4.5 那个 `emit_call` fallback(地址太远走间接调用)之外的另一道保障——尽量让直接跳转就够用,不用 fallback。

对于没有跳转范围限制的架构(理论上没有,但代码里 `#else` 分支兜底),`mcode_alloc` 退化成简单的"随便申请一块"(`lj_mcode.c:340`)。

## §8 第一性原理:W^X 安全(Write XOR Execute)

讲完"申请可执行内存"和"集中在一个范围",还有第三个、也是最微妙的一个约束:**可写和可执行,不能同时开。**

### §8.1 为什么不能同时可写可执行

设想一块内存同时是 `RWX`(可读可写可执行)。这意味着:

1. 程序可以往这块内存写任意字节(因为可写)。
2. 然后 CPU 可以执行这些字节(因为可执行)。

这正是 DEP/NX(§6.1)要防的——如果攻击者找到一个普通的内存破坏漏洞(比如缓冲区溢出),他可以把恶意机器码写进这块 RWX 内存,然后跳过去执行。有了 RWX 内存,攻击门槛大幅降低。

所以现代操作系统和安全规范要求:**一块内存,要么可写(写完切走写权限),要么可执行(执行时不可写),不能两者同时。** 这叫 **W^X(Write XOR Execute)**。OpenBSD 最早强制,现在 macOS Hardened Runtime、Linux PaX/SELinux 都跟进了。

### §8.2 JIT 怎么在 W^X 下工作

JIT 必须**生成机器码**(要写)然后**执行**(要执行)。在 W^X 下,它不能一块内存同时干这两件事。怎么办?

答案是**分时复用**:同一块内存,在"生成机器码"阶段设为可写(不可执行),写完、准备执行时,切换成可执行(不可写)。这两个阶段不重叠,所以任何时刻内存要么可写要么可执行,不同时。

LuaJIT 的实现(`lj_mcode.c:238`):

```c
/* This is the default behaviour and much safer:
**
** Most of the time the memory pages holding machine code are executable,
** but NONE of them is writable.
**
** The current memory area is marked read-write (but NOT executable) only
** during the short time window while the assembler generates machine code.
*/
#define MCPROT_GEN	MCPROT_RW
#define MCPROT_RUN	MCPROT_RX

/* Change protection of MCode area. */
static void mcode_protect(jit_State *J, int prot)
{
  if (J->mcprot != prot) {
    mcode_setprot(J, J->mcarea, J->szmcarea, prot);
    J->mcprot = prot;
  }
}
```

两个常量定义了两种状态:

- `MCPROT_GEN = MCPROT_RW`(生成态:可读可写,不可执行)——正在写机器码时用。
- `MCPROT_RUN = MCPROT_RX`(运行态:可读可执行,不可写)——机器码跑的时候用。

`mcode_protect(J, prot)` 切换当前机器码区的权限。注意它有个缓存 `J->mcprot`:如果请求的权限和当前一样,就不调系统调用(系统调用有开销),只有不同才切。这是性能优化——大部分时间机器码在运行态,不需要切。

### §8.3 W^X 在 trace 生成周期里的切换时机

来看 W^X 在一个 trace 的生命周期里怎么切换。trace 生成的入口(`lj_asm.c` 的 `lj_asm_trace`,`lj_mcode_reserve`):

```c
/* lj_mcode.c:391 */
MCode *lj_mcode_reserve(jit_State *J, MCode **lim)
{
  if (!J->mcarea)
    mcode_allocarea(J);
  else
    mcode_protect(J, MCPROT_GEN);  /* 切到生成态(可写) */
  *lim = J->mcbot;
  return J->mctop;
}
```

`lj_mcode_reserve` 在汇编生成前调用:把当前机器码区切到**可写**(MCPROT_GEN)。从此刻起,emit 层(§4)往这块内存写字节是合法的。

写完之后(`lj_mcode_commit`,`lj_mcode.c:402`):

```c
/* Commit the top part of the current MCode area. */
void lj_mcode_commit(jit_State *J, MCode *top)
{
  J->mctop = top;
  mcode_protect(J, MCPROT_RUN);  /* 切到运行态(可执行) */
}
```

`lj_mcode_commit` 把权限切回**可执行**(MCPROT_RUN)。从此刻起,这块机器码可以被 CPU 执行了,但不能再写(直到下次生成时再切回可写)。

所以一个 trace 的 W^X 周期是:

1. **reserve**:切可写。
2. **emit 一堆字节**:写机器码(此时可写,不可执行——就算有 bug 让 CPU 跳进来也不会执行半成品)。
3. **commit**:切可执行。
4. **运行**:CPU 执行这块机器码(此时可执行,不可写——就算有攻击者想改它也改不了)。

这是个干净的"写时不执行,执行时不写"的分时。任何时刻都满足 W^X。

`lj_mcode_abort`(`lj_mcode.c:409`)是生成失败时的清理:也切回可执行(虽然生成失败,但权限要恢复到安全态)。

### §8.4 LUAJIT_SECURITY_MCODE 开关

W^X 不是强制的。`lj_arch.h:763`:

```c
#ifndef LUAJIT_SECURITY_MCODE
/* Machine code page protection: 0 = insecure RWX, 1 = secure RW^X. */
#define LUAJIT_SECURITY_MCODE	1
#endif
```

默认 `LUAJIT_SECURITY_MCODE=1`(开 W^X)。但如果设成 0(`lj_mcode.c:216`):

```c
#if LUAJIT_SECURITY_MCODE == 0
...
#define MCPROT_GEN	MCPROT_RWX
#define MCPROT_RUN	MCPROT_RWX

static void mcode_protect(jit_State *J, int prot)
{
  UNUSED(J); UNUSED(prot);
}
...
#else
```

`MCPROT_GEN` 和 `MCPROT_RUN` 都变成 `MCPROT_RWX`(全开),`mcode_protect` 变成空函数(不切换)。这是"不安全模式":内存一直可写可执行,省掉了切换权限的系统调用开销,但有安全风险。

源码注释(`lj_mcode.c:218`)明确警告:

> Define this ONLY if page protection twiddling becomes a bottleneck.
> It's generally considered to be a potential security risk to have pages with simultaneous write and execute access in a process.
> Do not even think about using this mode for server processes or apps handling untrusted external data.
> The security risk is not in LuaJIT itself -- but if an adversary finds any other flaw in your C application logic, then any RWX memory pages simplify writing an exploit considerably.

意思是:关掉 W^X 纯粹是为了性能(省 `mprotect` 系统调用)。风险不在 LuaJIT 本身,而在于:如果你的程序有**别的**漏洞,RWX 内存会让攻击者更容易利用那个漏洞。所以只在"完全信任的环境、且性能确实瓶颈在权限切换"时才考虑关。默认开 W^X 是安全的做法。

### §8.5 macOS Hardened Runtime 的特殊处理

macOS 的 Hardened Runtime(强化运行时)对 W^X 有更严格的版本:它用 `MAP_JIT` 标志申请一种特殊的"JIT 内存",这种内存的"可写/可执行"状态由一个**进程级**的开关 `pthread_jit_write_protect_np` 控制——整个进程同一时刻,所有 JIT 内存要么全可写、要么全可执行,不能混合。

`lj_mcode.c:117` 处理这个:

```c
/* Check for macOS hardened runtime. */
#if defined(LUAJIT_ENABLE_OSX_HRT) && LUAJIT_SECURITY_MCODE != 0 && defined(MAP_JIT) && __ENVIRONMENT_MAC_OS_X_VERSION_MIN_REQUIRED__ >= 110000
#include <pthread.h>
#define MCMAP_CREATE	MAP_JIT
#else
#define MCMAP_CREATE	0
#endif
```

满足条件(macOS 11+、开了 Hardened Runtime、有 `MAP_JIT`)时,用 `MAP_JIT` 申请内存,并通过 `pthread_jit_write_protect_np(0/1)` 切换:

```c
static void *mcode_alloc_at(uintptr_t hint, size_t sz, int prot)
{
  void *p = mmap((void *)hint, sz, prot|MCPROT_CREATE, MAP_PRIVATE|MAP_ANONYMOUS|MCMAP_CREATE, -1, 0);
  if (p == MAP_FAILED) return NULL;
#if MCMAP_CREATE
  pthread_jit_write_protect_np(0);  /* 申请后默认切到可写 */
#endif
  return p;
}

static void mcode_setprot(jit_State *J, void *p, size_t sz, int prot)
{
#if LUAJIT_SECURITY_MCODE != 0
#if MCMAP_CREATE
  UNUSED(J); UNUSED(p); UNUSED(sz);
  pthread_jit_write_protect_np((prot & PROT_EXEC));  /* 按权限切开关 */
  return 0;
#else
  if (mprotect(p, sz, prot)) mcode_protfail(J);
#endif
#else
  ...
#endif
}
```

注意 Hardened Runtime 模式下,`mcode_setprot` 不再调 `mprotect`(因为权限是进程级开关控制的,不是 per-page),而是调 `pthread_jit_write_protect_np`。这是 Apple 的设计:牺牲灵活性(不能混合状态)换更强的安全性(整个进程的 JIT 内存状态单一)。

如果 W^X 切换失败(比如操作系统不让),`mcode_protfail`(`lj_mcode.c:68`)报错退出:

```c
#if LUAJIT_SECURITY_MCODE != 0
/* Protection twiddling failed. Probably due to kernel security. */
static LJ_NORET LJ_NOINLINE void mcode_protfail(jit_State *J)
{
  lua_CFunction panic = J2G(J)->panic;
  if (panic) {
    lua_State *L = J->L;
    setstrV(L, L->top++, lj_err_str(L, LJ_ERR_JITPROT));
    panic(L);
  }
  exit(EXIT_FAILURE);
}
#endif
```

注释 `Probably due to kernel security`——内核的安全策略(比如某些 SELinux 配置)可能禁止进程把自己的内存改成可执行。这时候 LuaJIT 没法工作,只能 panic 退出。这是 W^X 强制的代价:在极端安全配置下,JIT 直接不可用。

## §9 机器码区的组织:MCLink 链与多 trace 管理

讲完了单块机器码的申请和权限,现在看**多块**机器码怎么管理。一个程序运行久了,会编译出成百上千个 trace,每个 trace 都有自己的机器码。这些机器码怎么组织?

### §9.1 MCLink:机器码区拉成链表

LuaJIT 把所有已申请的机器码区,拉成一个**链表**。链表节点叫 `MCLink`(`lj_jit.h:174`):

```c
/* Linked list of MCode areas. */
typedef struct MCLink {
  MCode *next;		/* Next area. */
  size_t size;		/* Size of current area. */
} MCLink;
```

每块机器码区的**开头**放一个 `MCLink` 结构,记录"下一块在哪""这块多大"。这样从当前块就能遍历到所有块。

`mcode_allocarea`(`lj_mcode.c:358`)申请新块时,把新块接到链表头:

```c
/* Allocate a new MCode area. */
static void mcode_allocarea(jit_State *J)
{
  MCode *oldarea = J->mcarea;
  size_t sz = (size_t)J->param[JIT_P_sizemcode] << 10;
  J->mcarea = (MCode *)mcode_alloc(J, sz);
  J->szmcarea = sz;
  J->mcprot = MCPROT_GEN;
  J->mctop = (MCode *)((char *)J->mcarea + J->szmcarea);
  J->mcbot = (MCode *)((char *)J->mcarea + sizeof(MCLink));
  ((MCLink *)J->mcarea)->next = oldarea;   /* 新块指向旧块 */
  ((MCLink *)J->mcarea)->size = sz;
  J->szallmcarea += sz;
  J->mcbot = (MCode *)lj_err_register_mcode(J->mcarea, sz, (uint8_t *)J->mcbot);
}
```

注意几个字段(都在 `jit_State` 里,`lj_jit.h:505`):

```c
  int mcprot;		/* Protection of current mcode area. */
  MCode *mcarea;	/* Base of current mcode area. */
  MCode *mctop;		/* Top of current mcode area. */
  MCode *mcbot;		/* Bottom of current mcode area. */
  size_t szmcarea;	/* Size of current mcode area. */
  size_t szallmcarea;	/* Total size of all allocated mcode areas. */
  uintptr_t mcmin, mcmax;	/* Mcode allocation range. */
```

- `mcarea`:当前正在用的那块机器码区的基址。
- `mctop`/`mcbot`:当前块的上界/下界。新 trace 的机器码从 `mctop` 往下长(§4.2),exit stub 和常量从 `mcbot` 往上长(§10.3 会讲)。中间是空闲空间。
- `szmcarea`:当前块大小。
- `szallmcarea`:所有块的总大小(用来限制总机器码量,§9.3)。

新块申请时,`mctop = mcarea + sz`(整个块都是空闲的,从顶开始),`mcbot = mcarea + sizeof(MCLink)`(底部留一个 MCLink 头,从它上面开始放东西)。`MCLink` 的 `next` 指向旧块,链表就建起来了。

### §9.2 一块机器码区装多个 trace

关键理解:**一块机器码区(mcode area),可以装多个 trace 的机器码。** 不是一个 trace 一块内存——那样太浪费(每块至少一页 4KB,小 trace 才几十字节)。

实际做法:当前块从 `mctop` 往下分配空间给新 trace。一个 trace 用完一段,`mctop` 下移;下一个 trace 接着用。直到当前块快满了(`mcp` 逼近 `mcbot`),才申请新块。

这个"快满了"的检测靠 `checkmclim`(`lj_asm.c:146`,§10.1 详讲)。一旦逼近,触发 `lj_mcode_limiterr`,申请新块,把当前 trace 在新块里重汇编。

所以 `szallmcarea`(总量)是所有块加起来,而 `szmcarea`(当前块)只是当前在用的那块。多个 trace 共享一块,提高内存利用率。

### §9.3 总量限制与 flush

机器码不能无限涨。LuaJIT 有两个参数(`lj_mcode.c:444`):

```c
/* Limit of MCode reservation reached. */
void lj_mcode_limiterr(jit_State *J, size_t need)
{
  size_t sizemcode, maxmcode;
  lj_mcode_abort(J);
  sizemcode = (size_t)J->param[JIT_P_sizemcode] << 10;
  maxmcode = (size_t)J->param[JIT_P_maxmcode] << 10;
  if (need * sizeof(MCode) > sizemcode)
    lj_trace_err(J, LJ_TRERR_MCODEOV);  /* Too long for any area. */
  if (J->szallmcarea + sizemcode > maxmcode)
    lj_trace_err(J, LJ_TRERR_MCODEAL);
  mcode_allocarea(J);
  lj_trace_err(J, LJ_TRERR_MCODELM);  /* Retry with new area. */
}
```

`JIT_P_sizemcode` 是单块大小(默认 32KB 或 64KB),`JIT_P_maxmcode` 是总量上限(默认 512MB 或可配)。一个 trace 的机器码如果超过单块大小(`need > sizemcode`),直接报错"太长"(这种 trace 没法编译);如果总量快超了(`szallmcarea + sizemcode > maxmcode`),报错"分配失败"。

当机器码区需要全部回收时(比如 JIT 被 `jit.off()` 关掉,或者内存压力),`lj_mcode_free`(`lj_mcode.c:374`)遍历链表全释放:

```c
/* Free all MCode areas. */
void lj_mcode_free(jit_State *J)
{
  MCode *mc = J->mcarea;
  J->mcarea = NULL;
  J->szallmcarea = 0;
  while (mc) {
    MCode *next = ((MCLink *)mc)->next;
    size_t sz = ((MCLink *)mc)->size;
    lj_err_deregister_mcode(mc, sz, (uint8_t *)mc + sizeof(MCLink));
    mcode_free(mc, sz);
    mc = next;
  }
}
```

沿 `next` 链走,每块 `mcode_free`(调 `VirtualFree` 或 `munmap`)。这就是 flush 的底层——所有 trace 的机器码一次性清空。flush 之后,热点会重新被发现、重新编译(P2-05 讲过 trace 生命周期)。

## §10 checkmclim 与机器码容量的实时检查

### §10.1 红区(red zone)机制

emit 层每写几个字节,机器码缓冲区就少一点空间。什么时候该检查"是不是快满了"?

最笨的办法:每写一个字节检查一次。但写机器码是高频操作,每字节检查太慢。

LuaJIT 用**红区(red zone)**机制(`lj_asm.c:139`):

```c
/* Sparse limit checks using a red zone before the actual limit. */
#define MCLIM_REDZONE	64

static LJ_NORET LJ_NOINLINE void asm_mclimit(ASMState *as)
{
  lj_mcode_limiterr(as->J, (size_t)(as->mctop - as->mcp + 4*MCLIM_REDZONE));
}

static LJ_AINLINE void checkmclim(ASMState *as)
{
#ifdef LUA_USE_ASSERT
  if (as->mcp + MCLIM_REDZONE < as->mcp_prev) {
    IRIns *ir = IR(as->curins+1);
    lj_assertA(0, "red zone overflow: %p IR %04d  %02d %04d %04d\n", as->mcp,
      as->curins+1-REF_BIAS, ir->o, ir->op1-REF_BIAS, ir->op2-REF_BIAS);
  }
#endif
  if (LJ_UNLIKELY(as->mcp < as->mclim)) asm_mclimit(as);
#ifdef LUA_USE_ASSERT
  as->mcp_prev = as->mcp;
#endif
}
```

思路:`mclim` 设在 `mcbot + MCLIM_REDZONE`(即离底部还有 64 字节的地方)。emit 层不需要每字节检查,只需要**隔一段**检查一次 `mcp < mclim`。只要两次检查之间写的字节不超过 64(red zone 大小),就不会越过真正的底部 `mcbot`(因为还有 64 字节缓冲)。

`asm_exitstub_gen`(`lj_asm_x86.h:9`)生成 exit stub 时也检查(因为 exit stub 从 `mcbot` 往上长,也可能撞到 `mctop`):

```c
static MCode *asm_exitstub_gen(ASMState *as, ExitNo group)
{
  ...
  if (mxp + ((2+2)*EXITSTUBS_PER_GROUP +
	     (LJ_GC64 ? 0 : 8) +
	     (LJ_64 ? 6 : 5)) >= as->mctop)
    asm_mclimit(as);
  ...
}
```

### §10.2 checkmclim 的调用密度

`checkmclim` 在 emit 层到处调(`lj_asm_x86.h` 和 `lj_asm.c` 里有几十处)。基本上,每生成几条指令、每处理一条 IR,都会调一次 `checkmclim`。这保证了 red zone 不会被越过——任何时刻 `mcp` 离 `mcbot` 至少还有 `MCLIM_REDZONE` 的余量。

一旦 `mcp < mclim`(快到底了),`asm_mclimit` 触发,它会:

1. 调 `lj_mcode_limiterr`(§9.3)。
2. `lj_mcode_limiterr` 调 `mcode_allocarea` 申请新块。
3. 抛 `LJ_TRERR_MCODELM` 错误。

这个错误被 `lj_asm_trace` 的外层捕获(§10.3),它会**在新块里重新汇编整个 trace**。这就是源码注释(`lj_asm.c:2510` 那段长注释)说的 "MCode may have to be (re-)assembled multiple times" 的一种情况:机器码区不够,换块重来。

### §10.3 emit 的两个生长方向:mctop 和 mcbot

再强调一个容易忽略的细节:机器码区有两个生长方向。

- **trace 机器码**从 `mctop` **往下**长(§4.2,`as->mcp` 从 `mctop` 减)。
- **exit stub 和常量**从 `mcbot` **往上**长。

为什么有两个方向?因为 trace 机器码的长度在汇编前不确定(反向生成,边生成边长),而 exit stub 和某些常量(64 位浮点常量,§10.4)是**边生成边追加**的、且需要被 trace 机器码引用(跳转/读取)。从两边往中间长,中间是空闲,谁先涨谁先用,直到相遇。

`mclim` 就是 `mcbot + MCLIM_REDZONE`——它跟着 `mcbot` 走。`mcbot` 涨(追加了 exit stub 或常量),`mclim` 也涨,留给 trace 机器码的空间就更小。所以 `checkmclim` 检查的是"trace 机器码(mcp 从上往下)有没有撞到 exit stub/常量区(mcbot 从下往上 + red zone)"。

### §10.4 64 位常量内联(emit_loadk64)

顺便看一个 `mcbot` 往上涨的实际场景:64 位常量内联(`lj_emit_x86.h:391`):


## §11 side exit 的热修改:lj_asm_patchexit

最后讲一个把本章所有东西串起来的机制:**机器码的热修改(patchexit)**。P2-05 提过,这里讲透它的机器码层面。

### §11.1 为什么要 patch 父 trace 的机器码

回忆 side trace(P0-01 §13、P2-05):一个 trace(root)的某个 guard 经常失败,退回解释器。失败得多了,LuaJIT 觉得"这条失败路径也挺热",就从那个退出点**录制一条新的 trace**(side trace)。

side trace 编译好之后,有一个问题:**怎么让程序跑到那个退出点时,不再退回解释器,而是直接跳进新的 side trace?**

原来的退出点,机器码里是一条"条件跳转:guard 失败就跳到 exit stub"(§4.4 的 `emit_jcc`)。exit stub 会把控制权交给解释器。

现在要让这个跳转改成"跳到 side trace 的入口"。怎么做?**改父 trace 的机器码**——把那条 `jcc` 的跳转目标,从 exit stub 改成 side trace 入口。

这就是 patchexit:**在父 trace 已经编译好、甚至正在运行的机器码上,动态修改跳转目标。** 这是对"已生成机器码"的热修改。

### §11.2 patchexit 的源码

`lj_asm_patchexit`(`lj_asm_x86.h:3127`):

```c
/* Patch exit jumps of existing machine code to a new target. */
void lj_asm_patchexit(jit_State *J, GCtrace *T, ExitNo exitno, MCode *target)
{
  MCode *p = T->mcode;
  MCode *mcarea = lj_mcode_patch(J, p, 0);   /* ① 切到可写 */
  MSize len = T->szmcode;
  MCode *px = exitstub_addr(J, exitno) - 6;  /* 原 exit stub 地址 - 6(jcc 指令长 6) */
  MCode *pe = p+len-6;
  MCode *pgc = NULL;
  ...
  /* ② 先处理尾部可能的无条件 jmp */
  if (len > 5 && p[len-5] == XI_JMP && p+len-6 + *(int32_t *)(p+len-4) == px)
    *(int32_t *)(p+len-4) = jmprel(J, p+len, target);
  /* Do not patch parent exit for a stack check. Skip beyond vmstate update. */
  /* ③ 跳过 vmstate 更新指令,避免误 patch */
  for (; p < pe; p += asm_x86_inslen(p)) {
    intptr_t ofs = LJ_GC64 ? (p[0] & 0xf0) == 0x40 : LJ_64;
    if (*(uint32_t *)(p+2+ofs) == statei && p[ofs+LJ_GC64-LJ_64] == XI_MOVmi)
      break;
  }
  lj_assertJ(p < pe, "instruction length decoder failed");
  /* ④ 扫描所有 jcc,把指向原 exit stub 的改成指向 side trace */
  for (; p < pe; p += asm_x86_inslen(p)) {
    if ((*(uint16_t *)p & 0xf0ff) == 0x800f && p + *(int32_t *)(p+2) == px &&
	p != pgc) {
      *(int32_t *)(p+2) = jmprel(J, p+6, target);
    } else if (*p == XI_CALL &&
	      (void *)(p+5+*(int32_t *)(p+1)) == (void *)lj_gc_step_jit) {
      pgc = p+7;  /* Do not patch GC check exit. */
    } else if (LJ_64 && *p == 0xff &&
				 p[1] == MODRM(XM_REG, XOg_CALL, RID_RET) &&
				 p[2] == XI_NOP) {
      pgc = p+5;  /* Do not patch GC check exit. */
    }
  }
  lj_mcode_sync(T->mcode, T->mcode + T->szmcode);  /* ⑤ 刷新 I-cache */
  lj_mcode_patch(J, mcarea, 1);                     /* ⑥ 切回可执行 */
}
```

这个函数是本章所有概念的总汇。逐步看:

**① `lj_mcode_patch(J, p, 0)`**——把父 trace 的机器码区切到**可写**(W^X 的写阶段)。回忆 §8:平时机器码是可执行的不可写,要改它必须先切权限。`lj_mcode_patch(J, ptr, finish)`,`finish=0` 表示"开始 patch,切可写",返回这块机器码区所属的 mcarea 基址(用于后面切回)。`lj_mcode_patch` 的实现(`lj_mcode.c:416`)会先查"这块机器码是不是当前 mcarea",是就直接 `mcode_protect`,不是就遍历 MCLink 链找到它再 `mcode_setprot`。

**② 处理尾部的无条件跳转**——如果父 trace 末尾有个 `jmp` 正好跳到这个 exit stub(比如循环回跳),直接改它的目标。

**③ 跳过 vmstate 更新**——trace 的机器码开头会更新 `vmstate`(标记当前在跑哪个 trace,exit 时要用)。这段不能 patch。`asm_x86_inslen(p)` 是一个 x86 指令长度解码器(靠 `map_op1`/`map_op2` 那两个查表,§11.3 讲),逐条指令往前走,直到找到更新 vmstate 的 `MOVmi` 指令,从它之后才开始扫 jcc。

**④ 扫描所有 jcc**——这是核心。逐条指令走(`p += asm_x86_inslen(p)`),找形如 `0f 8x`(近条件跳转)且跳转目标正好是 `px`(原 exit stub)的指令,把它的跳转偏移改成 `target`(side trace 入口)。`jmprel(J, p+6, target)` 算新的相对偏移。

注意 `pgc` 的处理:有些 jcc 是 **GC 检查**的退出(调用 `lj_gc_step_jit` 后,内存不够就退出触发 GC)。这种退出**不能** patch 成跳到 side trace——因为它退出是为了做 GC,不是为了走另一条路径。所以扫到 GC 检查的 call,标记 `pgc`,后面遇到它关联的 jcc 就跳过。

**⑤ `lj_mcode_sync`**——改完字节,刷新 I-cache(§6.5),让 CPU 执行时看到新机器码。

**⑥ `lj_mcode_patch(J, mcarea, 1)`**——切回**可执行**(W^X 的执行阶段)。`finish=1`。

### §11.3 指令长度解码器:asm_x86_inslen

patchexit 要逐条指令扫描机器码,但 x86 是变长指令,要知道每条指令多长才能走到下一条。这需要一个**指令长度解码器**。

LuaJIT 没有用完整的 x86 译码器(那太复杂),而是用一个**简化的长度表**——`map_op1` 和 `map_op2`(`lj_asm_x86.h:3026` 起,两个 256 项的 `uint8_t` 数组)。每个操作码字节对应一个长度编码(数组里的值编码了"这条指令多长、有没有 ModRM、有没有立即数"等)。`asm_x86_inslen` 查这两个表,算出一条指令的长度。

这个解码器是**专为 patchexit 设计的简化版**——它只需要算长度(走到下一条),不需要完整译码。但它必须对 LuaJIT 实际生成的所有指令都正确,否则扫描会错位。这是个微妙的地方:如果 LuaJIT 将来用了新指令(长度表没覆盖),patchexit 会出错。所以加新指令时,要同步更新 `map_op1`/`map_op2`。

### §11.4 patchexit 在 side trace 创建时的调用

patchexit 在哪被调用?side trace 编译完成、要安装时(`lj_trace.c:528`,处理 `BC_JMP` 退出点):

```c
  case BC_JMP:
    /* Patch exit branch in parent to side trace entry. */
    lj_assertJ(J->parent != 0 && J->cur.root != 0, "not a side trace");
    lj_asm_patchexit(J, traceref(J, J->parent), J->exitno, J->cur.mcode);
    /* Avoid compiling a side trace twice (stack resizing uses parent exit). */
    {
      SnapShot *snap = &traceref(J, J->parent)->snap[J->exitno];
      snap->count = SNAPCOUNT_DONE;   /* 标记这个退出点已编译 side trace */
      ...
    }
    ...
    break;
```

side trace 录制完,它的"父 trace 在哪个退出点(J->exitno)退出"是已知的。调 `lj_asm_patchexit`,把父 trace 里那个退出点的 jcc,改成跳到 side trace 入口。然后把 snapshot 的 `count` 设成 `SNAPCOUNT_DONE`(`lj_jit.h:190`),标记"这个退出点已经有 side trace 了,别再编译第二次"。

之后,所有 patch 改完,才 `lj_mcode_commit`(`lj_trace.c:558`):

```c
  /* Commit new mcode only after all patching is done. */
  lj_mcode_commit(J, J->cur.mcode);
```

注释强调顺序:**先 patch 父 trace,再 commit side trace 自己的机器码**。因为 side trace 的机器码入口地址(`J->cur.mcode`)是 patch 的目标——必须先确定这个地址(已经在 §4 汇编时定好了),patch 才能指向它;而 commit(切 side trace 机器码为可执行)要在 patch 完成后,确保 side trace 准备好被跳进。

这就是 side trace 长出来的完整机器码流程:父 trace 已有机器码 → side trace 录制汇编出自己机器码 → patchexit 改父 trace 的跳转目标指向 side trace → commit side trace 可执行 → 此后跑到那个退出点,直接进 side trace,不退解释器。

## §12 为什么这样设计是 sound 的

本章涉及三块相互独立但都关乎正确性的设计:架构抽象、W^X 安全、机器码热修改。逐个看它们为什么不破坏正确性。

### §12.1 架构抽象的 sound:抽象层和架构层契约对齐

架构抽象(§2-§4)的 sound 在于:**抽象层和架构层的契约严格对齐**。具体表现为三个不变式:

**不变式一:每个抽象 RID 在架构层都有唯一定义。** 抽象层用 `Reg`(整数 0~31)表示寄存器。架构层的 `GPRDEF`/`FPRDEF` 把这些整数映射到具体寄存器(RID 0 = EAX 或 X0...)。这个映射在编译时就固定了(enum),不会运行时变。所以抽象层说"RID 3",架构层无歧义地知道是哪个寄存器。

**不变式二:寄存器集合的位运算在位宽内正确。** `RegSet` 是位图,`rset_set`/`rset_clear` 等操作。sound 的前提是"寄存器编号不超过位图位宽"。这由编译期检查保证:`RID_NUM_GPR + RID_NUM_FPR <= RSET_BITS * ...`(ARM64 用 64 位 RegSet 就是因为它有 32+ 个寄存器)。如果某种新架构寄存器超过 64 个,这套位图就不够用了——但现有架构都没超。

**不变式三:特殊角色寄存器(RID_SP/RID_BASE/RID_RET)在架构层和解释器层一致。** 注释 `must match with the *.dasc file(s)` 反复强调这点。如果 JIT 生成代码假设 BASE 在 EDX,但解释器(dynasm 生成)实际把 BASE 放在 EBX,JIT 和解释器交接时会读错栈——这是致命 bug。靠"两边手动保持一致 + 注释提醒"来保证。这是这套设计的一个**弱点**(没有编译期自动校验),但实践中靠纪律守住。

### §12.2 W^X 的 sound:写时不执行,执行时不写

W^X(§8)的 sound 在于:**任意时刻,一块机器码内存不会同时处于"可写"和"可执行"状态**。这通过 `mcode_protect` 的状态机保证:

- `MCPROT_GEN`(可写不可执行):只在 `lj_mcode_reserve` 到 `lj_mcode_commit`/`lj_mcode_abort` 之间。这段时间 CPU 不会执行这块内存(正在生成,还没装好)。
- `MCPROT_RUN`(可执行不可写):其余所有时间。这段时间 emit 层不会写它(要写得先 reserve 切回 GEN)。

切换的原子性由 `mcode_setprot`(调 `mprotect` 或 `VirtualProtect`)保证——系统调用要么成功(权限切了)要么失败(`mcode_protfail` 退出)。不会出现"切了一半"的中间态。

这个设计的隐含假设是:**操作系统真的尊重 `mprotect`/`VirtualProtect` 设置的权限**。现代操作系统都尊重(MMU 硬件强制)。唯一例外是某些嵌入式或老系统可能忽略——那种环境 LuaJIT 文档会警告。

W^X 对正确性没影响(它不影响机器码内容),但对**安全**有影响:它缩小了攻击面。即使 LuaJIT 本身或宿主程序有别的漏洞,W^X 也让攻击者更难利用(不能往可执行内存写恶意代码)。这是"defense in depth"。

### §12.3 patchexit 的 sound:改跳转不破坏机器码语义

patchexit(§11)是最需要小心 sound 的部分——它在**正在运行的机器码**上动刀。为什么不会破坏正确性?

关键在于:**patchexit 只改跳转的"目标地址",不改指令本身的结构**。

一条 `jcc`(近条件跳转)的编码是 `0f 8x [4字节偏移]`——指令长度固定 6 字节,只有那 4 字节偏移是变量。patchexit 把这 4 字节偏移从"指向 exit stub"改成"指向 side trace 入口",指令的操作码字节(`0f 8x`)、条件码(`x`)都不变。

这意味着:

1. **指令长度不变**(还是 6 字节)——所以后续指令的地址不偏移,不会破坏别的指令的相对跳转。
2. **条件不变**——原来"相等则跳",patch 后还是"相等则跳",只是跳的地方变了。语义上,guard 失败(条件成立)时的行为从"退解释器"变成"进 side trace"。而 side trace 正是从这个退出点录制的,它的入口状态(snapshot,P5-17 讲)和这个退出点的状态一致——所以跳进去能正确继续。
3. **改的时机安全**——patch 时机器码不可执行(W^X 切到可写),改完切回可执行。没有"改到一半 CPU 跑过来执行半成品"的窗口。

唯一的微妙点:**如果 patch 时,正好有别的线程在执行这条 jcc 怎么办?** 这是多核机器上的真实问题。LuaJIT 的处理:`lj_mcode_sync` 刷 I-cache 确保可见性;而且 patch 是原子的(写 4 字节偏移,在 x86 上对齐的 4 字节写是原子的)。最坏情况是某个核已经过了这条 jcc 的取指阶段(执行旧目标),这没关系——旧目标是 exit stub,退解释器也正确(只是没用到 side trace 的优化)。下一次再跑到这,就是新目标了。所以 patch 是**最终一致**的:可能有一两次执行还是走老路,但不会错。

### §12.4 跳转范围的 sound:mcode 集中保证跳转可达

§7 的跳转范围分配,它的 sound 在于:**所有 mcode 都在 `[mcmin, mcmax]` 范围内,而这个范围的宽度取跳转范围的一半,保证任意两点距离不超过跳转范围**。

这个不变式由 `mcode_alloc` 保证:每次申请都 `mcode_inrange` 检查,不在范围就释放重试或开新范围。所以 emit 层生成相对跳转时,目标一定在范围内,`jmprel` 算出的偏移一定放得进 32 位(或对应架构的位宽)。如果万一超了(理论上不该发生),`jmprel` 里的 `lj_assertJ(delta == (int32_t)delta, ...)` 会抓住。

## §13 ★对照:官方 Lua 与 JVM/V8

### §13.1 对照官方 Lua:没有 JIT,就没有这些问题

官方 Lua 是纯解释器,它**完全没有本章讲的任何东西**:

- **没有多架构后端**:官方 Lua 用 C 写解释器,C 编译器(gcc/clang)负责跨架构。官方 Lua 不需要自己生成机器码,所以不需要 target 抽象、不需要 emit 层。跨架构的复杂性被甩给了 C 编译器。
- **没有可执行内存**:解释器代码是编译时就生成的(在 `.text` 段,天生可执行),运行时不需要申请新的可执行内存。没有 mmap/VirtualAlloc、没有 W^X、没有 mcode 管理。
- **没有热修改**:解释器不修改自己的代码,没有 patchexit。

所以官方 Lua 的实现简单得多——它不需要应付"动态生成执行代码"带来的一堆问题。这是 JIT 的**复杂度代价**:为了快,要造一整套机器码生成和管理设施。官方 Lua 选择不付这个代价,慢但简单。

这也反过来说明:**LuaJIT 加在官方 Lua 之上的,不只是"一个编译器",还有一整套和操作系统、CPU 架构打交道的运行时设施**(可执行内存、W^X、缓存同步、跳转范围)。这些都是 trace JIT 路线躲不开的工程负担。

### §13.2 对照 JVM/V8:多后端架构支持与代码缓存

JVM(HotSpot)和 V8 也是 JIT,也跨多种 CPU。它们怎么处理本章的两个问题?

**多后端架构**:

HotSpot 和 V8 都有自己的"架构抽象层",思路和 LuaJIT 类似(抽象 + 多个架构后端),但规模大得多:

- **HotSpot** 有完整的后端模块(`src/hotspot/cpu/x86/`、`src/hotspot/cpu/arm/`、`src/hotspot/cpu/ppc/`...),每个架构一套匹配器(matcher,把 IR 匹配到具体指令)、一套编码器。HotSpot 用的是 **method JIT**,IR 是基于 Sea-of-Nodes 的图(比 LuaJIT 的线性 SSA 复杂),后端要做指令选择、指令调度(指令流水线优化),这些都和架构强相关。HotSpot 的后端代码量是 LuaJIT 的几十倍。
- **V8**(TurboFan)同样有多架构后端(`src/compiler/backend/x64/`、`src/compiler/backend/arm64/`...),用 CodeStubAssembler/Torque 抽象。V8 还额外支持 WASM(又一套指令集)。

LuaJIT 的多后端**轻量**得多:每个架构就一个 `lj_target_*.h`(寄存器/指令定义)+ `lj_emit_*.h`(发射)+ `lj_asm_*.h`(IR 到机器码)。因为 LuaJIT 的 IR 简单(线性 SSA)、不做复杂指令调度、以 trace 为单位(代码量小),后端可以做得精简。这是 trace JIT 路线的优势之一:**后端简单,易于多架构**。代价是优化深度不如 HotSpot/V8(没有指令调度、没有复杂的指令选择)。

**可执行内存与代码缓存**:

HotSpot 和 V8 也要申请可执行内存生成机器码,机制类似(`mmap`/`VirtualAlloc` + W^X)。但它们多了一个 LuaJIT 没有的层次:**代码缓存(Code Cache)管理**。

- **HotSpot** 有专门的 CodeCache 子系统,管理编译出的机器码(nmethod)。它支持分层编译(Tiered Compilation:解释 → C1(client compiler)→ C2(server compiler)),同一方法可能有多版机器码,要管理它们的生命周期、去优化(deoptimization,把机器码"作废"退回解释器)。CodeCache 满了会触发清理。这套系统比 LuaJIT 的 MCLink 链复杂得多。
- **V8** 有自己的代码缓存(基于 CodeSpace,在垃圾回收的堆里管理机器码),也支持去优化(deoptimization,通过"埋点"的机器码检查)。V8 还会把编译出的代码序列化到磁盘(代码缓存序列化),加速下次启动。

LuaJIT 的 mcode 管理相比之下极简:一个 MCLink 链、两个指针(mctop/mcbot)、W^X 切换、flush 时全释放。没有分层编译(只有一个 JIT 编译器)、没有代码缓存序列化、deoptimization 退化为 side exit/snapshot(P5)。这是 trace JIT + 小团队的取舍:**够用就行,不搞复杂的代码缓存层**。

**对照表**:

| 维度 | LuaJIT(trace JIT) | 官方 Lua | JVM/V8(method JIT) |
|---|---|---|---|
| 机器码生成 | 运行时,trace 触发 | 不生成 | 运行时,方法热点触发 |
| 多架构方式 | 编译时选 `lj_target_*.h`/`lj_emit_*.h` | C 编译器负责 | 完整后端模块(matcher/encoder) |
| 后端复杂度 | 轻(每架构 3 个文件) | 无 | 重(每架构一整套,含指令调度) |
| 可执行内存 | mmap/VirtualAlloc + W^X | 不需要 | 同 mmap/VirtualAlloc + W^X |
| 代码缓存 | MCLink 链,简单 | 无 | CodeCache/CodeSpace,分层管理 |
| 去优化 | side exit + snapshot(P5) | — | deoptimization(埋点机器码) |
| 热修改 | patchexit(改跳转目标) | 无 | deoptimization(改机器码为陷阱) |

trace vs method 在这里的体现:trace 代码量小,后端可以轻量多架构;mmethod 代码量大,要复杂的代码缓存和去优化。LuaJIT 选了前者,用简洁换深度。

## §14 回扣主线

回到本书的主线:**把动态执行安全变成机器码。**

本章讲的是这条主线的"落地"环节——抽象的 IR 操作,经过 P4-13(寄存器分配)和 P4-14(汇编生成)之后,变成了具体的机器码指令。但"变成机器码指令"还不够,这段机器码还得:

1. **是正确架构的机器码**(§1-§5):x86 的字节在 ARM64 上是垃圾。靠 target 抽象层 + emit 层,一套上层逻辑配多个架构后端,保证生成的字节对得上目标 CPU。
2. **放在可执行的内存里**(§6):普通内存不可执行,CPU 拒绝跑。靠 mmap/VirtualAlloc 申请专门的执行内存。
3. **集中在一个跳转能互相够得着的范围**(§7):机器码之间要互相跳转,距离不能太远。靠 mcode_setrange 把所有机器码圈在一个窗口里。
4. **遵守 W^X 安全约束**(§8):可写和可执行不能同时开。靠 mcode_protect 在生成时切可写、运行时切可执行,分时复用。
5. **能被高效管理**(§9-§10):成百上千个 trace 的机器码,要组织、要检查容量、要能释放。靠 MCLink 链 + 红区 checkmclim + flush。
6. **能被热修改**(§11):side trace 长出来时,要改父 trace 的跳转目标。靠 patchexit,在 W^X 保护下改跳转偏移,不破坏指令结构。

这六件事,都是"把机器码安全地放出来让它跑"必须解决的工程问题。它们不涉及 IR、不涉及优化、不涉及类型——纯粹是"机器码生成出来后,怎么让它正确、安全、高效地存在于进程的地址空间里"。这是 trace JIT 路线躲不开的最后一公里。

P4 篇到这里就完整了:从 IR(P2)出发,经过优化(P3)、寄存器分配(P4-13)、汇编生成(P4-14)、后端目标与机器码内存(本章),一条 IR 最终变成了一段放在正确内存里、遵守所有架构和安全约束、真正可执行的机器码。

但机器码能跑还不够——它跑的时候,guard 可能失败(P0-01 §7)。guard 失败后怎么安全地退回解释器?退回时怎么恢复状态?这是 P5 篇的事。下一章,我们看 guard 在机器码里到底长什么样、side exit 的完整机制。
