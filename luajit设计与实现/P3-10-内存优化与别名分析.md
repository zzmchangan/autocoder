## 引子:fold 处理不了的指令——带"记忆"的运算

上一章(P3-09)讲完了 fold。我们看到 fold 引擎的核心是"规则表驱动的模式匹配",它最擅长的是纯运算——加、减、乘、移位、按位逻辑。这些运算有一个共同性质:**同样的输入永远给出同样的输出,而且不改变别的任何东西**。正因为如此,fold 能放心地用 `1 + 2 → 3`、`x + 0 → x` 这类恒等式去化简它们。

但程序里还有一类指令,fold 表面对它们只能摊手。看这两行 IR:

```text
%10 = ALOAD  %8         ; 从表数组槽读一个值
%12 = ALOAD  %8         ; 又读同一个槽
```

按 fold 的逻辑,这是不是该走 CSE——"两次相同的运算,复用第一次的结果"?不行。`ALOAD`(数组加载)不像 `ADD`:它的结果不取决于操作数 `%8` 这一个引用本身,而取决于**那一刻内存里那个槽里装的是什么**。如果两次 `ALOAD` 之间,有别的指令把那个槽改了,第二次读出来的值就和第一次不同——这时候复用第一次的结果就错了。

所以内存相关的指令(load 读内存、store 写内存)是 fold 处理不了的特例。它们带"记忆":**结果取决于内存的当前状态,而内存状态会被别的指令改写**。fold 的常量折叠、代数化简在这里统统失灵——内存里当下是什么,编译期不知道;`x + 0` 在任何时刻都等于 `x`,但"读内存地址 p"在两次之间可能给出不同结果。

那么,这类带记忆的指令,就只能老老实实地每条都生成机器码、每次都真的去访问内存吗?

当然不是。仔细想就会发现,即便结果取决于内存状态,**很多时候内存状态其实没变**。如果两次读同一个地址,而中间没有任何指令改这个地址,那第二次读必然得到和第一次一样的值——这时候第二次读取完全是多余的,可以直接复用第一次的结果。这就是一类巨大的优化机会。

但"中间没有指令改这个地址"这个前提,要怎么验证?这就引出了本章的核心难题:**怎么判断两条内存访问指令,指的是不是同一个内存位置?** 这个问题有个专门的名字——**别名分析(alias analysis)**。判断"两个引用是不是指向同一地址"叫做"别名分析",因为如果两个引用可能指向同一地址,它们就互为"别名"。

别名分析是整个 load/store 优化的命脉。判断准了,才能安全地消除冗余的读(load forwarding,读转发)和冗余的写(store elimination,写消除);判断不准(或判断不了),就只能保守地当作"可能别名",不优化,以保证正确。

LuaJIT 把这件事放在一个专门的文件里:`lj_opt_mem.c`,992 行。它的文件头注释(`lj_opt_mem.c:1-7`)把职责说得明明白白:

```c
/*
** Memory access optimizations.
** AA: Alias Analysis using high-level semantic disambiguation.
** FWD: Load Forwarding (L2L) + Store Forwarding (S2L).
** DSE: Dead-Store Elimination.
*/
```

三件事:**AA**(别名分析)、**FWD**(转发,包括 load 到 load 和 store 到 load)、**DSE**(死存储消除)。本章就从"为什么判断两次内存访问是否同一地址这么难"这个第一性问题开始,一步步把这三件事讲透。

而且,这件事对 Lua **尤其**重要,值得在引子里就点明:Lua 里唯一的复合数据结构就是 table,而 table 的字段访问(`t.k`、`t[i]`)是 Lua 程序里最频繁的操作之一。一段处理数据的 Lua 代码,常常一行里就要访问好几次表的字段。如果每次访问都老老实实走一遍完整的查表逻辑(算哈希、找槽、读值),开销惊人。LuaJIT 把表访问翻成 IR 后,能不能消掉那些重复的访问,直接决定了大量 Lua 程序的性能上限。所以别名分析不是个可有可无的锦上添花,它是 LuaJIT 优化层里和 fold 同等重要的另一根支柱。

## §1 第一性原理:为什么内存优化这么难

在直接看 LuaJIT 怎么做别名分析之前,我们必须先把"这件事为什么难"想清楚。不搞清楚难在哪,就看不懂 LuaJIT 那一堆判断为什么写成那样、为什么有时候明明看起来能优化却放弃了。

### 1.1 从最朴素的优化机会说起:冗余的 load

先看一个最小、最直白的优化机会。考虑这段 Lua:

```lua
local x = t.k
local y = t.k      -- 紧接着又读了一次同一个字段
```

翻译成 IR(具体的录制过程 §6 会细讲,这里先看大致形状):

```text
%5  = HREF   %1, KSTR("k")    ; 算出 t.k 这个槽的地址
%6  = HLOAD  %5               ; 从那个槽读值 → x
%8  = HREF   %1, KSTR("k")    ; 又算一次同一个槽的地址
%9  = HLOAD  %8               ; 又从那个槽读值 → y
```

人眼一看就知道:`%6` 和 `%9` 读的是同一个地方,而且中间没有任何东西改过 `t.k`,所以 `%9` 的结果必然等于 `%6`。理想情况下,`%8`、`%9` 这两条都该省掉,直接用 `%6`:

```text
%6  = HLOAD  %5
y = %6      ; 直接复用,不再读内存
```

机器码里就少了一次哈希查找、少了一次内存读取。在跑一百万次的循环里,这就是一百万次省下来的查表。

这个优化叫 **load forwarding(读转发)**,更准确地说是 load-to-load forwarding——后面一次 load 转发复用前面一次 load 的结果。它看似简单,但要落地,必须回答一个致命问题:**凭什么确信中间没有指令改过 `t.k`?**

### 1.2 难点:内存是"全局可变"的

我们来认真审视"中间没有指令改过 `t.k`"这个前提。它看似显然,实则极难精确判断。

**第一层难:改内存的指令不止是显式的 store。** 最直白的改写是 `ASTORE`/`HSTORE`(数组/哈希存储)——往槽里写值。但除此之外还有:`NEWREF`(往表里插入一个新键,可能触发 rehash,把所有键的位置都挪一遍)、`table.clear`(清空整个表)、`CALLS`(调用一个可能修改内存的函数,比如 `string.gsub` 的回调往表里写)、FFI 的 `XSTORE`(往裸内存地址写)。任何一种都可能让"中间没改过 `t.k`"失效。所以"搜索中间有没有冲突 store",搜索范围远不止同一种 store 指令。

**第二层难:地址是运行时算出来的。** 这是最根本的难处。算术运算的输入就是它的操作数,操作数相同,运算就相同——这是 SSA 给的硬保证。但内存访问的"地址",本身是另一条指令算出来的结果。比如 `ALOAD %8`,真正访问的地址是 `%8` 这条 `AREF` 指令算出来的(表数组基址 + 下标)。而 `%8` 的操作数里可能有一个是运行时变量(比如循环下标 `i`),那么两次 `AREF` 算出的地址,**只有在运行时才知道**到底一不一样。

举一个循环里的典型例子:

```lua
for i = 1, n do
  local a = t[i]      ; 第 i 次循环:读 t[i]
  local b = t[i+1]    ; 读 t[i+1]
  local c = t[i]      -- 又读 t[i]
end
```

这里 `a` 和 `c` 都是 `t[i]`,看起来该复用。但编译时 `i` 是个循环变量(它的值在循环里变),编译器怎么知道第一次的 `t[i]` 和第三次的 `t[i]` 指的是同一个下标?如果判断不了,就不敢优化。

**第三层难:别名是"可能"而非"确定"。** 即使两个引用的地址在编译时部分已知,也往往只能判断"可能别名"而无法判断"一定不别名"或"一定别名"。比如两个指向**不同表**的 `HREF`,能不能断定它们不别名?如果这两个表是同一次 `TNEW` 分配出来的两个不同表,那肯定不别名(两个不同的分配永远不重叠);但如果其中一个是外部传进来的表,编译器根本不知道它和另一个表是不是同一个对象——只能保守地当"可能别名",放弃优化。

这三层难合起来,构成了别名分析的根本困境:**你想证明"两次访问不冲突",但地址是运行时算的、改内存的指令五花八门、而且很多情况下根本拿不到足够的信息去证明"不冲突"。**

### 1.3 解法:保守是底线,精确是追求

面对这个困境,编译器只有一条底线绝不能破:**宁可漏掉优化机会(性能损失),也绝不能做错误的优化(正确性损失)**。

为什么这条底线如此重要?因为错误的别名判断会导致错误的优化,而错误的优化会改变程序结果。比如误判"两次 `t.k` 不会冲突",复用了第一次的值,但中间其实有个 `t.k = 42` 改写了它——那第二次读本该是 42,却用了旧值,程序就算错了。对一个 JIT 来说,算错比慢更致命(回想 P0-01 §9 的关键不变式:机器码的结果绝不能比解释器更错)。

所以别名分析的判断,遵循一个三值逻辑(而不是非黑即白)。LuaJIT 在 `lj_opt_mem.c:35-39` 明确定义了三个返回值:

```c
/* Return values from alias analysis. */
typedef enum {
  ALIAS_NO,	/* The two refs CANNOT alias (exact). */
  ALIAS_MAY,	/* The two refs MAY alias (inexact). */
  ALIAS_MUST	/* The two refs MUST alias (exact). */
} AliasRet;
```

- **`ALIAS_NO`(一定不别名)**:精确判断两个引用绝不指向同一地址。这是最有价值的结论——它意味着两次访问互不干扰,可以放心优化。
- **`ALIAS_MUST`(一定别名)**:精确判断两个引用必然指向同一地址。这同样有价值——比如一个 store 之后紧跟一个同地址的 load,可以确信 load 读到的就是刚才 store 的值(这叫 store-to-load forwarding)。
- **`ALIAS_MAY`(可能别名)**:判断不了,保险起见当作可能冲突。这种情况下,**安全的选择是不优化**(或只做有限优化)。

注意 `ALIAS_MAY` 的注释是 "inexact"(不精确),意思是这个结论不是确凿的——它不代表"真的会冲突",只代表"我判断不了,得按可能冲突处理"。这是一个**保守**的结论:宁可放弃优化,也不冒险。

这套三值逻辑是整个 `lj_opt_mem.c` 的基石。文件里每一个 `aa_xxx` 函数(aa 是 alias analysis 的缩写)的返回值都是这三个之一。理解了"保守是底线",后面看到那些函数里各种"拿不准就返回 `ALIAS_MAY`"的写法,就不会觉得是偷懒,而是 sound 的保障。

### 1.4 三类优化的分工:AA / FWD / DSE

有了三值逻辑,我们就能讲清楚 `lj_opt_mem.c` 做的三类优化各自要什么前提:

**Load Forwarding(FWD,读转发)**:当一个 load 出现时,去搜索在它之前有没有"同一个地址的 load"或"同一个地址的 store"。
- 如果找到同地址的 **store**(且中间没别的冲突),直接用 store 写进去的那个值——这叫 **store-to-load forwarding(S2L)**,最理想,连内存都不用读。
- 如果找到同地址的 **load**(且中间没冲突),复用它读到的值——这叫 **load-to-load forwarding(L2L)**。
- 搜索时只要遇到一个 `ALIAS_MAY` 的 store,就得停下来(它可能改了这个地址),把搜索下限缩到这里。

**Dead-Store Elimination(DSE,死存储消除)**:当一个 store 出现时,去搜索在它之前有没有"同一个地址的 store"。
- 如果之前的 store 写的值和现在一样(`store->op2 == val`),那现在这个 store 是多余的(写了一遍又写一遍同样的值),直接删掉(`DROPFOLD`)。
- 如果之前的 store 写的是不同的值,但**中间没有任何指令读过这个地址**(也没有会触发副作用的 guard),那么之前那个 store 是"死的"(写了从没被读过,马上又被覆盖),可以把它改成 NOP,只留现在这个。
- 如果遇到 `ALIAS_MAY` 的 store 且值不同,说明无法判断,保守地都保留。

**Alias Analysis(AA,别名分析)**:是上面两类的地基。FWD 和 DSE 在搜索时,每遇到一个候选的 store/load,都要调 `aa_xxx` 判断它和当前指令是不是同地址——这就是别名分析。

这三类在 `lj_opt_mem.c` 里泾渭分明:文件按"ALOAD/HLOAD"(§前半)、"ULOAD"(§中)、"FLOAD"(§中后)、"XLOAD"(§后)分块,每块都是"A A 分析函数 + FWD 函数 + DSE 函数"的三件套。AA 是私有的 `static` 函数,FWD 和 DSE 是公开的 `lj_opt_fwd_xxx`/`lj_opt_dse_xxx`,被 fold 引擎通过 `LJFOLDX` 调用。

### 1.5 和 fold/CSE 的关系:谁先谁后

讲清楚三件套后,我们要把它和上一章的 fold、CSE 的关系理顺,免得读者混淆。

回忆 P3-09 §7.3:fold 引擎在处理 load/store 这类有副作用的指令时,不会交给 CSE(那是给纯运算用的),而是通过 `LJFOLDX` 转交给 `lj_opt_mem.c` 里的对应函数。比如 `lj_opt_fold.c:2124-2125`:

```c
LJFOLD(ALOAD any)
LJFOLDX(lj_opt_fwd_aload)
```

意思是:**任何 `ALOAD` 指令,fold 引擎都把它转交给 `lj_opt_fwd_aload`**。这个函数先做别名分析和转发;如果转发成功了(找到一个可复用的旧值),直接返回那个值,不再发射新指令;如果没成功,返回 `EMITFOLD`(原样发射)。所以别名分析是"夹"在 fold 引擎和真正发射之间的一个 pass——它先于 CSE 处理 load/store。

这个分工有个细节值得注意:`lj_opt_mem.c:30-32` 的文件头注释专门提醒——

```c
/*
** Caveat #1: return value is not always a TRef -- only use with tref_ref().
** Caveat #2: FWD relies on active CSE for xREF operands -- see lj_opt_fold().
*/
```

Caveat #2 说"FWD 依赖 CSE 对 xREF 操作数生效"。意思是:别名分析判断两个 `ALOAD` 是否同地址时,前提是它们的 `AREF`(地址计算指令)已经被 CSE 归并过——如果同一地址的 `AREF` 没被 CSE 成同一个引用,别名分析看到的会是两个看起来不同的 `AREF`,可能误判。所以 load/store 优化是一个**协作链**:`AREF`/`HREF` 这种地址计算先经 CSE 归并,然后 `ALOAD`/`HLOAD` 再经别名分析转发。两者缺一不可。

好,第一性原理铺垫完了。我们已经知道:内存优化难在地址运行时才算、改写源五花八门、判断只能保守。下面三节,我们用真实源码,把 LuaJIT 怎么对**表访问**(Lua 里最频繁的内存操作)做别名分析、做转发、做死存储消除,逐一讲透。表访问是 `lj_opt_mem.c` 的重头戏(占了文件前一半多),也是本章的讲解重点;之后我们会简要覆盖 upvalue、字段、FFI 内存这几类。

## §2 表访问的 IR 形状:别名分析的对象

别名分析的对象是"内存引用"。在 LuaJIT 里,一次表访问 `t[k]` 在 IR 里长什么样?这是我们看懂 `aa_ahref` 等函数的前提。这一节先把表访问的 IR 形状讲清楚。

### 2.1 一次 `t.k` 翻成什么:AREF / HREF / HREFK + ALOAD / HLOAD

Lua 的 table 分两部分存:**数组部分**(下标 1..asize 的整数键)和**哈希部分**(其他键,包括字符串键、非连续整数键等)。所以 `t[k]` 的访问,录制时(`lj_record.c` 的 `rec_idx_key` 函数,`lj_record.c:1437`)会先判断 `k` 落在哪部分:

- 如果 `k` 是小整数且当前在数组范围内,录制成 `AREF`(数组引用):`lj_record.c:1463` 的 `emitir(IRT(IR_AREF, IRT_PGC), arrayref, ikey)`。`AREF` 的两个操作数是:数组基址(由 `FLOAD` 从表里取出 `array` 字段得到)和下标。
- 如果 `k` 是字符串常量(或编译期已知的常量键),且它在哈希表里的位置 `hslot` 能算出来且足够小,录制成 `HREFK`(哈希引用,带常量槽位):`lj_record.c:1505` 的 `emitir(IRTG(IR_HREFK, IRT_PGC), node, kslot)`。`HREFK` 的两个操作数是:节点基址(由 `FLOAD` 取出 `node` 字段)和一个 `KSLOT`(常量键 + 槽位)。
- 否则(哈希查找,位置运行时才算),录制成 `HREF`(哈希引用,动态查找):`lj_record.c:1509` 的 `emitir(IRT(IR_HREF, IRT_PGC), ix->tab, key)`。

拿到这个"地址引用"(`AREF`/`HREFK`/`HREF`)之后,再发一条加载指令真正读值——`lj_record.c:1593`:

```c
loadop = xrefop == IR_AREF ? IR_ALOAD : IR_HLOAD;
```

即:`AREF` 之后跟 `ALOAD`(数组加载),`HREF`/`HREFK` 之后跟 `HLOAD`(哈希加载)。所以一次 `t.k` 读,IR 大致是两到三条:`FLOAD`(取 array/node 字段)+ `AREF`/`HREF`/`HREFK`(算地址)+ `ALOAD`/`HLOAD`(读值)。

这些指令在 IR 定义里(`lj_ir.h:88-116`)按"内存引用"和"加载/存储"分组,而且有个关键约束——**load 和 store 必须成对、按固定顺序排列**:

```c
/* Memory references. */
_(AREF,	R , ref, ref) \
_(HREFK, R , ref, ref) \
_(HREF,	L , ref, ref) \
_(NEWREF, S , ref, ref) \
...
/* Loads and Stores. These must be in the same order. */
_(ALOAD,	L , ref, ___) \
_(HLOAD,	L , ref, ___) \
_(ULOAD,	L , ref, ___) \
_(FLOAD,	L , ref, lit) \
_(XLOAD,	L , ref, lit) \
_(SLOAD,	L , lit, lit) \
_(VLOAD,	L , ref, lit) \
_(ALEN,	L , ref, ref) \

_(ASTORE,	S , ref, ref) \
_(HSTORE,	S , ref, ref) \
_(USTORE,	S , ref, ref) \
_(FSTORE,	S , ref, ref) \
_(XSTORE,	S , ref, ref) \
```

注意那个注释 "These must be in the same order." 以及紧跟着的 `LJ_STATIC_ASSERT`(`lj_ir.h:170-175`):

```c
/* Delta between xLOAD and xSTORE. */
#define IRDELTA_L2S		((int)IR_ASTORE - (int)IR_ALOAD)

LJ_STATIC_ASSERT((int)IR_HLOAD + IRDELTA_L2S == (int)IR_HSTORE);
LJ_STATIC_ASSERT((int)IR_ULOAD + IRDELTA_L2S == (int)IR_USTORE);
LJ_STATIC_ASSERT((int)IR_FLOAD + IRDELTA_L2S == (int)IR_FSTORE);
LJ_STATIC_ASSERT((int)IR_XLOAD + IRDELTA_L2S == (int)IR_XSTORE);
```

这是 `lj_opt_mem.c` 一个核心设计的依据:**load 和它对应的 store 在 opcode 枚举里相差一个固定的常数 `IRDELTA_L2S`**。`ALOAD + IRDELTA_L2S == ASTORE`,`HLOAD + IRDELTA_L2S == HSTORE`,以此类推。这个常数让代码能从一条 load 的 opcode 直接算出"它对应的 store 是哪种"——比如 `fwd_ahload` 里 `lj_opt_mem.c:169`:

```c
ref = J->chain[fins->o+IRDELTA_L2S];
```

`fins->o` 是当前 load(`ALOAD` 或 `HLOAD`),`fins->o + IRDELTA_L2S` 就是它对应的 store(`ASTORE` 或 `HSTORE`)。`J->chain[...]` 是按 opcode 分的指令链表(上一章 §7.1 讲过)。所以这一行就是"取到所有同类型的 store 组成的链表头",然后顺着链表往前找冲突的 store。这个"load 加一个常数得到 store"的技巧,让 `fwd_ahload` 一个函数就能同时处理 `ALOAD` 和 `HLOAD` 两种加载,不必为每种写一遍。

### 2.2 三种"引用"的语义:R / L / S

再细看那几个引用指令的 IR mode(第二个字段):`AREF` 和 `HREFK` 是 `R`,`HREF` 是 `L`,`NEWREF` 是 `S`。这些 mode 标记定义在 `lj_ir.h:287-303`:

```c
#define IRM_N			0x00
#define IRM_R			IRM_N
#define IRM_A			0x20
#define IRM_L			0x40
#define IRM_S			0x60
...
#define irm_kind(m)		((m) & IRM_S)
```

`irm_kind` 取出的是"大类":`N`(纯运算,无副作用)、`R`(引用,纯计算地址,本身不读不写内存)、`L`(加载,有读副作用)、`S`(存储,有写副作用)。理解这几个分类对看 fold 引擎的分派很关键:

- `AREF`/`HREFK`/`FREF` 是 `R`——它们只**算出一个地址**,本身不访问内存,所以是纯运算,可以走 CSE(两个相同的 `AREF` 当然可以合并成一个)。
- `HREF`/`UREF`/`LREF` 是 `L`——它们在"算地址"的同时隐含了一次内存读(`HREF` 要查哈希表才能定位槽位),所以算加载,不能简单 CSE。
- `NEWREF` 是 `S`——它往表里插入新键,**可能触发 rehash**,会改写表里所有键的位置,所以是存储(而且是个很"重"的存储,影响面大)。
- `ALOAD`/`HLOAD`/...是 `L`(加载);`ASTORE`/`HSTORE`/...是 `S`(存储)。

这套分类让 fold 引擎能按"是否纯运算、是否读、是否写"来分派:`R` 的交给 CSE,`L`/`S` 的交给 `lj_opt_mem.c`。我们在 §1.5 看到的 `LJFOLD(ALOAD any) LJFOLDX(lj_opt_fwd_aload)`,就是因为 `ALOAD` 是 `L` 类,不能走 CSE,必须走别名分析。

### 2.3 一个完整的 `t.k` 读取:四条 IR

把上面的合起来,一次 `t.k`(k 是字符串常量、且哈希位置已知)读访问,录出来的 IR 是这样的(假设 `t` 在 `%1`):

```text
%2 = FLOAD  %1, IRFL_TAB_NODE     ; 取表的 node 字段(哈希节点数组基址)
%3 = FLOAD  %1, IRFL_TAB_HMASK    ; 取 hmask(哈希表大小掩码),配 guard 检查
%4 = EQ     %3, KINT(hmask)       ; guard:哈希表大小没变(否则槽位会错)
%5 = HREFK  %2, KSLOT(KSTR("k"), slot)   ; 算出 t.k 的节点地址(常量槽位)
%6 = HLOAD  %5                    ; 从那个节点读值
```

(实际的录制还有 snapshot、guard 类型等细节,这里简化了,抓主要形状。)

看,一次"读 `t.k`"就发了五条 IR(两条 `FLOAD` 取表字段、一条 guard 检查大小没变、一条 `HREFK` 算地址、一条 `HLOAD` 读值)。如果紧接着又读一次 `t.k`,又会发五条。这十条 IR 里,后五条和前五条几乎一模一样——只要别名分析能确认"中间没人改过 `t`",后五条全都可以复用前五条的结果,一条都不用真发射。

这就是 load forwarding 的战场。它的收益是巨大的:把重复的表访问压成零开销。下面我们就来看 LuaJIT 怎么打这场仗。

## §3 别名分析的核心:aa_ahref 怎么判断两个表引用是否同地址

本节是本章的技术核心。我们逐行读 `aa_ahref` 这个函数(`lj_opt_mem.c:104-159`),看 LuaJIT 怎么对两个表引用(数组或哈希)做别名分析。这个函数是 `fwd_ahload`(读转发)和 `lj_opt_dse_ahstore`(死存储消除)共用的判断器,所有表访问的优化都建立在它的结论之上。

### 3.1 函数签名和两个引用

```c
/* Alias analysis for array and hash access using key-based disambiguation. */
static AliasRet aa_ahref(jit_State *J, IRIns *refa, IRIns *refb)
{
  IRRef ka = refa->op2;
  IRRef kb = refb->op2;
  IRIns *keya, *keyb;
  IRRef ta, tb;
  if (refa == refb)
    return ALIAS_MUST;  /* Shortcut for same refs. */
```

`refa` 和 `refb` 是两个内存引用指令(`AREF`/`HREF`/`HREFK` 之一)。函数要判断它们是否指向同一地址。

第一步是个快捷判断:如果 `refa == refb`——两个引用是**同一条 IR 指令**(同一个指针),那它们当然是同一个地址,直接返回 `ALIAS_MUST`。这是 CSE 已经把相同 `AREF` 归并之后的最理想情况:两个 `ALOAD` 共用一个 `AREF`,别名分析一眼就能确认它们访问同一槽。

接下来取出两个引用的"键":

```c
  keya = IR(ka);
  if (keya->o == IR_KSLOT) { ka = keya->op1; keya = IR(ka); }
  keyb = IR(kb);
  if (keyb->o == IR_KSLOT) { kb = keyb->op1; keyb = IR(kb); }
```

`ka = refa->op2` 是引用的第二个操作数——也就是**键**。回忆 §2.1:`AREF` 的 op2 是下标,`HREF` 的 op2 是键,`HREFK` 的 op2 是个 `KSLOT`(常量键 + 槽位打包)。这里有个处理:如果键是个 `KSLOT`(`HREFK` 的情况),要穿透一层——`KSLOT` 的 op1 才是真正的键(`lj_ir.h` 里 `KSLOT` 的定义是把常量键和槽位打包,取键要解一层)。所以 `keya` 最后指向真正的键指令。

### 3.2 取出表引用:穿透 FLOAD

```c
  ta = (refa->o==IR_HREFK || refa->o==IR_AREF) ? IR(refa->op1)->op1 : refa->op1;
  tb = (refb->o==IR_HREFK || refb->o==IR_AREF) ? IR(refb->op1)->op1 : refb->op1;
```

这两行取**表引用**(`ta`、`tb`),但逻辑有点绕,值得细讲。

回忆 §2.3:`AREF` 和 `HREFK` 的 op1 不是表本身,而是从表里 `FLOAD` 出来的 `array`/`node` 字段。也就是说,`AREF` 的形状是 `AREF(FLOAD(tab, TAB_ARRAY), ikey)`——op1 是那条 `FLOAD`,而 `FLOAD` 的 op1 才是表。所以要拿到表引用,得穿透两层:`IR(refa->op1)->op1`——先到 `FLOAD` 那条指令,再取它的 op1(表)。

而 `HREF` 不一样:它的 op1 直接就是表(回忆 `lj_record.c:1509`:`emitir(IRT(IR_HREF, IRT_PGC), ix->tab, key)`),不用穿透。所以条件表达式区分了这两种情况。

为什么要费劲拿表引用?因为下一步要判断"两个引用是不是指向同一个表"。如果连表都不一样,那俩引用大概率不别名;如果表一样,还得看键。表引用是别名分析的关键输入之一。

### 3.3 第一道判断:键相同吗

```c
  if (ka == kb) {
    /* Same key. Check for same table with different ref (NEWREF vs. HREF). */
    if (ta == tb)
      return ALIAS_MUST;  /* Same key, same table. */
    else
      return aa_table(J, ta, tb);  /* Same key, possibly different table. */
  }
```

如果两个键的 IR 引用相同(`ka == kb`),意味着它们是同一个键(SSA 保证相同引用即相同值)。这时候:
- 如果表也相同(`ta == tb`),那俩引用**必然**指向同一地址——同一个表的同一个键,槽位一样。返回 `ALIAS_MUST`。
- 如果表不同,调 `aa_table`(下一节讲)判断两个表是不是其实是同一个对象(可能一个是 `TNEW`、一个是外部传入,碰巧指向同一张表)。注意这里不能直接说"表不同所以不别名"——因为 `ta`/`tb` 是 IR 引用,两个不同的引用可能指向运行时同一个表对象(SSA 只保证"相同引用即相同值",不保证"不同引用即不同值")。

### 3.4 第二道判断:两个常量键不同吗

```c
  if (irref_isk(ka) && irref_isk(kb))
    return ALIAS_NO;  /* Different constant keys. */
```

走到这里说明 `ka != kb`(键引用不同)。但如果两个键都是**编译期常量**(`irref_isk` 判断),而且引用不同,那就说明它们是**两个不同的常量**——比如一个是 `KSTR("k")`,另一个是 `KSTR("m")`。两个不同的常量键,即使落在同一个表里,也必然对应不同的槽位(哈希表按键的值哈希)。所以俩引用**绝不别名**,返回 `ALIAS_NO`。

这是别名分析里最有价值的一类结论:`t.k` 和 `t.m`,只要 `k`、`m` 是不同的字符串常量,就一定不冲突。这正是 Lua 程序里"同时访问一个对象的多个不同字段"这种极常见模式的安全保障。

### 3.5 数组引用:基于下标算术的判断

```c
  if (refa->o == IR_AREF) {
    /* Disambiguate array references based on index arithmetic. */
    int32_t ofsa = 0, ofsb = 0;
    IRRef basea = ka, baseb = kb;
    lj_assertJ(refb->o == IR_AREF, "expected AREF");
    /* Gather base and offset from t[base] or t[base+-ofs]. */
    if (keya->o == IR_ADD && irref_isk(keya->op2)) {
      basea = keya->op1;
      ofsa = IR(keya->op2)->i;
      if (basea == kb && ofsa != 0)
        return ALIAS_NO;  /* t[base+-ofs] vs. t[base]. */
    }
    if (keyb->o == IR_BADD && irref_isk(keyb->op2)) {
      baseb = keyb->op1;
      ofsb = IR(keyb->op2)->i;
      if (ka == baseb && ofsb != 0)
        return ALIAS_NO;  /* t[base] vs. t[base+-ofs]. */
    }
    if (basea == baseb && ofsa != ofsb)
      return ALIAS_NO;  /* t[base+-o1] vs. t[base+-o2] and o1 != o2. */
  }
```

这段处理数组引用(两个都是 `AREF`)。回忆 §1.2 的难点:数组下标常常是运行时变量,但很多时候它是"变量 ± 常量"的形式——比如 `t[i]`、`t[i+1]`、`t[i-1]`。这段代码就是要把这种 `base ± ofs` 的形式拆开,看两个下标的 base 和 offset。

逻辑是:如果键是一条 `ADD` 指令(`keya->o == IR_ADD`)且第二个操作数是常量(`irref_isk(keya->op2)`),那它就是 `base + ofs` 的形式——`basea = keya->op1`(base,通常是个变量),`ofsa = IR(keya->op2)->i`(常量偏移)。

拆出 base 和 ofs 之后,做三种判断:
- `t[base±ofs]` vs `t[base]`:如果一个是 `base` 另一个是 `base±ofs`(且 `ofs != 0`),它们必然是不同的下标,返回 `ALIAS_NO`。
- `t[base±o1]` vs `t[base±o2]`(`o1 != o2`):base 相同但 offset 不同,下标不同,返回 `ALIAS_NO`。

这就解决了 §1.2 那个 `t[i]` vs `t[i+1]` 的例子:拆出 base 都是 `i`,offset 一个是 0 一个是 1,不同,判定不别名。两次访问互不干扰,可以各自独立优化。

注意一个隐含的保守:如果 base 不同(比如 `t[i]` vs `t[j]`,`i`、`j` 是不同变量),代码**不返回 `ALIAS_NO`**——因为编译时不知道 `i` 和 `j` 运行时是不是相等。这种情况会落到后面的 `ALIAS_MAY`。这是 sound 的体现:不武断地说"不同变量一定不等"。

### 3.6 哈希引用:基于键类型的判断

```c
  } else {
    /* Disambiguate hash references based on the type of their keys. */
    lj_assertJ((refa->o==IR_HREF || refa->o==IR_HREFK || refa->o==IR_NEWREF) &&
	       (refb->o==IR_HREF || refb->o==IR_HREFK || refb->o==IR_NEWREF),
	       "bad xREF IR op %d or %d", refa->o, refb->o);
    if (!irt_sametype(keya->t, keyb->t))
      return ALIAS_NO;  /* Different key types. */
  }
```

哈希引用的判断更简单:看两个键的**类型**。如果一个是字符串键、另一个是数字键,它们在哈希表里必然落在不同的桶(哈希函数依赖类型),所以不别名,返回 `ALIAS_NO`。

这是个很实用的判断:Lua 程序里同时用字符串键和整数键访问同一个表很常见(`t.name` 是字符串键,`t[1]` 如果落在哈希部分是整数键转数字键),它们互不干扰。

但如果键类型相同(都是字符串,或都是数字),就没法靠类型区分了——两个不同的字符串键可能哈希到同一个桶(哈希冲突),所以不能说"不别名"。这时候落到下面的兜底。

### 3.7 兜底:表相同时的 MAY,表不同时交给 aa_table

```c
  if (ta == tb)
    return ALIAS_MAY;  /* Same table, cannot disambiguate keys. */
  else
    return aa_table(J, ta, tb);  /* Try to disambiguate tables. */
```

走到这里,说明:键的引用不同、不是两个不同的常量、数组的 base/offset 或哈希的键类型都没能区分开。这时候:
- 如果表相同(`ta == tb`),只能说"可能别名"——同一个表里,键运行时可能碰巧相等(比如两个变量键 `i`、`j` 运行时可能相同)。返回 `ALIAS_MAY`,保守处理。
- 如果表不同,最后再试一次 `aa_table`,看能不能证明两个表其实是不同的对象(下一节讲)。证明不了就还是 `ALIAS_MAY`。

把整个 `aa_ahref` 串起来,它的判断策略是一个**从易到难、能定则定、定不了就保守**的漏斗:先看是不是同一个引用(`MUST`),再看键是否相同(`MUST`/交 `aa_table`),再看两个常量键是否不同(`NO`),再按数组/哈希各自的下标/类型规则尝试区分(`NO`),最后兜底 `MAY`/`aa_table`。每一步都尽可能给出精确结论,给不出才放宽。这个漏斗的每一层,都对应一种 Lua 程序里常见的访问模式,精准地把那些"明显不冲突"的情况挑出来,让优化能下手。

### 3.8 aa_table:两个表引用是不是同一个对象

顺带讲 `aa_table`(`lj_opt_mem.c:56-73`),它判断两个表引用(`ta`、`tb`)是不是同一个表对象:

```c
/* Alias analysis for two different table references. */
static AliasRet aa_table(jit_State *J, IRRef ta, IRRef tb)
{
  IRIns *taba = IR(ta), *tabb = IR(tb);
  int newa, newb;
  lj_assertJ(ta != tb, "bad usage");
  lj_assertJ(irt_istab(taba->t) && irt_istab(tabb->t), "bad usage");
  /* Disambiguate new allocations. */
  newa = (taba->o == IR_TNEW || taba->o == IR_TDUP);
  newb = (tabb->o == IR_TNEW || tabb->o == IR_TDUP);
  if (newa && newb)
    return ALIAS_NO;  /* Two different allocations never alias. */
  if (newb) {  /* At least one allocation? */
    IRIns *tmp = taba; taba = tabb; tabb = tmp;
  } else if (!newa) {
    return ALIAS_MAY;  /* Anything else: we just don't know. */
  }
  return aa_escape(J, taba, tabb);
}
```

核心思路:**新分配的表,和别的表不会别名**。如果 `ta` 是一个 `TNEW`/`TDUP`(本 trace 内新分配的表),`tb` 是另一个 `TNEW`/`TDUP`,那它俩是两次不同的分配,必然是两个不同的表对象,返回 `ALIAS_NO`。

如果只有一个表是新分配的(比如 `ta` 是 `TNEW`,`tb` 是外部传入的),代码把新分配的那个挪到 `taba`,然后调 `aa_escape`——这是个"简化版逃逸分析",看这个新表有没有"逃"到可能被 `tb` 引用的地方(`lj_opt_mem.c:44-53`):

```c
/* Simplified escape analysis: check for intervening stores. */
static AliasRet aa_escape(jit_State *J, IRIns *ir, IRRef stop)
{
  IRRef ref = (IRRef)(ir - J->cur.ir);  /* The ref that might be stored. */
  for (ir++; ir < stop; ir++)
    if (ir->op2 == ref &&
	(ir->o == IR_ASTORE || ir->o == IR_HSTORE ||
	 ir->o == IR_USTORE || ir->o == IR_FSTORE))
      return ALIAS_MAY;  /* Reference was stored and might alias. */
  return ALIAS_NO;  /* Reference was not stored. */
}
```

它从新分配点往后扫,看有没有任何 store 把这个新表的引用存到别的地方(`op2 == ref`)。如果有,说明这个新表可能"逃"出去了,外部传进来的 `tb` 可能就指向它,返回 `MAY`;如果没有任何 store 把它存出去,说明它一直待在本地,外部 `tb` 不可能是它,返回 `NO`。

这是个非常简化的逃逸分析——它只看"有没有 store 把这个表存出去",不看更复杂的逃逸路径(比如作为函数调用的参数传出去)。注释里 `aa_table` 上方写的是 "Simplified escape analysis",诚实地标注了简化程度。但就是这个简化版,已经能抓住 LuaJIT trace 里最常见的情形:一个局部新建的表,在 trace 内部用,没逃出去——这时候它和任何外部表都不别名,可以放心优化。

如果两个表都不是新分配的(都是外部传入的),`aa_table` 直接返回 `ALIAS_MAY`——"Anything else: we just don't know"。这是诚实的承认:对外部表,LuaJIT 不做更精确的过程间分析,一律当可能别名。这是个刻意的取舍:精确的过程间分析代价高(要做调用图、上下文敏感分析等),而 LuaJIT 的设计哲学是"编译要快",所以这里选择保守。

## §4 读转发:fwd_ahload 怎么消除冗余的表读取

有了 `aa_ahref` 这个判断器,我们来看它怎么用在读转发上。核心函数是 `fwd_ahload`(`lj_opt_mem.c:162-255`),它是 `lj_opt_fwd_aload` 和 `lj_opt_fwd_hload` 共用的实现。这个函数有点长,我们分两段读:前半段做 store-to-load 转发和 load-to-load 转发,后半段(从"const-fold loads from allocations"开始)处理从新分配的表里读初值这种特殊情况。

### 4.1 第一段:搜索冲突的 store

```c
/* Array and hash load forwarding. */
static TRef fwd_ahload(jit_State *J, IRRef xref)
{
  IRIns *xr = IR(xref);
  IRRef lim = xref;  /* Search limit. */
  IRRef ref;

  /* Search for conflicting stores. */
  ref = J->chain[fins->o+IRDELTA_L2S];
  while (ref > xref) {
    IRIns *store = IR(ref);
    switch (aa_ahref(J, xr, IR(store->op1))) {
    case ALIAS_NO:   break;  /* Continue searching. */
    case ALIAS_MAY:  lim = ref; goto cselim;  /* Limit search for load. */
    case ALIAS_MUST: return store->op2;  /* Store forwarding. */
    }
    ref = store->prev;
  }
```

`xref` 是当前 load 要读的那个地址引用(`AREF`/`HREF`/`HREFK`)。`lim = xref` 是搜索下限——我们只找在 `xref` **之后**(即在它之前发射、SSA 里引用号更大)的指令。回忆 SSA 的引用号是递增的,`ref > xref` 意味着 `ref` 是比 `xref` 更晚定义的指令——但这里我们要找的是"在 load 之前的 store",为什么条件是 `ref > xref`?

这里要澄清一个 SSA 编号的方向问题。LuaJIT 的 IR 引用号是**从大到小**增长的——新指令的引用号更小(`nins` 递减,`lj_jit.h` 里 `IRRef nins; /* Next IR instruction. Biased with REF_BIAS. */`)。所以"`ref > xref`"指的是 `ref` 的引用号更大,即 `ref` 是**更早**发射的指令。这个方向和直觉相反,但和 `J->chain[op]` 链表的"从新到旧"遍历方向一致。所以这个循环是在"从最近的 store 往回找,找所有定义在 `xref` 之前的 store"。

循环体里,对每一个候选的 store,调 `aa_ahref(xr, IR(store->op1))` 判断它和当前 load 的地址是否别名。`store->op1` 是 store 写的那个地址引用(回忆 `ASTORE xref val` 的形状:op1 是地址,op2 是值)。按三值逻辑分支:

- **`ALIAS_NO`**:这个 store 写的不是我们要读的地方,无关。`break` 继续找下一个。
- **`ALIAS_MAY`**:这个 store **可能**写了我们要读的地方。这时候不能转发(万一真写了呢),但也不能就此放弃整个优化——还要去看看有没有更早的同地址 load 可以复用。所以把搜索下限 `lim` 收紧到这个 store(`lim = ref`),然后跳到 `cselim` 去找 load。这个收紧的含义是:"在找 load 时,只能找比这个 store 更早的 load——因为比它晚的 load 可能已经被这个 store 改了语义"。
- **`ALIAS_MUST`**:这个 store **一定**写了我们要读的地方。那 load 读到的值,就是 store 写进去的值——直接返回 `store->op2`(store 的值操作数)。这就是 **store-to-load forwarding(S2L)**,最理想的情形:连内存都不用读,直接用编译期已知的那个值。

注意 `ALIAS_MUST` 这一分支的妙处:它不光省了 load,还把"读内存"变成了"用一个 IR 引用"——如果 `store->op2` 本身是个常量或简单运算,后端能进一步优化。比如 `t.k = 42; ...; x = t.k`,store 的值是常量 42,转发后 load 直接变成常量 42,机器码里连访问内存都省了。

### 4.2 第二段:从新分配的表里读初值

如果第一段循环没找到 `ALIAS_MUST` 的 store(也没遇到 `ALIAS_MAY` 跳走),代码进入第二段——处理"从新分配的表里读"这种特殊情况(`lj_opt_mem.c:180-243`)。

为什么要单独处理新分配的表?因为 `TNEW`(新建空表)和 `TDUP`(从模板复制表)是编译期就知道内容的表:
- `TNEW` 出来的表,所有槽位都是 nil。
- `TDUP` 出来的表,内容是模板表(一个编译期已知的 `GCtab`)的拷贝。

所以从这两种表里读,结果往往是编译期就能确定的(全 nil,或者模板里的某个值),可以折叠成常量,根本不用真的 load。

这段代码比较长,我们看主干:

```c
  /* No conflicting store (yet): const-fold loads from allocations. */
  {
    IRIns *ir = (xr->o == IR_HREFK || xr->o == IR_AREF) ? IR(xr->op1) : xr;
    IRRef tab = ir->op1;
    ir = IR(tab);
    if ((ir->o == IR_TNEW || (ir->o == IR_TDUP && irref_isk(xr->op2))) &&
	fwd_aa_tab_clear(J, tab, tab)) {
```

先穿透拿到表引用 `tab`(同样的逻辑:`AREF`/`HREFK` 要穿透 `FLOAD`,`HREF` 不用),然后判断这个表是不是 `TNEW` 或 `TDUP`(且键是常量)。如果是,还要调 `fwd_aa_tab_clear`(`lj_opt_mem.c:76-87`)确认中间没有 `table.clear` 把它清空了——`table.clear` 是个 `CALLS` 调用,会清空整个表,如果有就得放弃。

确认安全后,分两种情况:
- 如果是 `TNEW`:新表全是 nil,直接返回 `TREF_NIL`(`lj_opt_mem.c:222`)。读一个空表的任何字段都是 nil,编译期就知道。
- 如果是 `TDUP`:新表是模板的拷贝,用 `lj_tab_get` 在**编译期**查模板表里那个键的值(`lj_opt_mem.c:223-240`),把查到的值折叠成对应的 IR 常量(`lj_ir_knum`/`lj_ir_kint`/`lj_ir_kstr` 等)。

第二种尤其精妙:它相当于**在编译期把一次表查找执行掉了**,把结果直接嵌进 IR 当常量。运行时这次表访问完全不存在。

不过这里有个细节值得注意——`lj_opt_mem.c:187-205` 那段注释和代码处理了一个微妙的边界:**`NEWREF` 插入数字键时,可能触发 rehash,把原本在哈希部分的数字键挪到数组部分(或反过来)**。所以如果中间有 `NEWREF` 且键是数字,这个"从 TDUP 读初值"的优化就得放弃(代码里 `goto cselim`),因为 rehash 可能让槽位对不上。这是个很 subtle 的情况,注释说得明白:"A NEWREF with a number key may end up pointing to the array part... Or a NEWREF may rehash the table and move unrelated number keys"。这种对边界情况的细致处理,是 sound 的保障——宁可放过优化,也不冒险。

### 4.3 第三段:cselim,找同地址的旧 load

如果前面两段都没成功(遇到了 `ALIAS_MAY` 的 store,或者不是新分配的表),代码到达 `cselim` 标号(`lj_opt_mem.c:245-254`):

```c
cselim:
  /* Try to find a matching load. Below the conflicting store, if any. */
  ref = J->chain[fins->o];
  while (ref > lim) {
    IRIns *load = IR(ref);
    if (load->op1 == xref)
      return ref;  /* Load forwarding. */
    ref = load->prev;
  }
  return 0;  /* Conflict or no match. */
```

这是 **load-to-load forwarding(L2L)**:在当前 load 之前(`ref > lim`,且不超过前面 `ALIAS_MAY` store 划定的下限 `lim`)找有没有"读同一个地址"的旧 load。判断条件是 `load->op1 == xref`——旧 load 的地址引用和当前的一样。

为什么这里只用"地址引用相等"(`==`)判断,而不调 `aa_ahref`?因为 L2L 要求**确定性**的复用——只有当地址引用完全相同(SSA 同一个引用)时,才能保证两次 load 读的是同一个地方、且(在 `lim` 限定的范围内)中间没被改过。如果用 `aa_ahref` 的 `ALIAS_MUST`(地址不同但必然同一),那两次 load 的地址引用不同,虽然指向同一槽,但无法保证它们之间没有别的 load 已经基于"旧值"做了运算——复用会破坏语义。所以 L2L 用最严格的"同一引用"判断。

这里也能看出 §1.5 那个 Caveat #2 的意义:L2L 要求两个 load 的 `op1`(地址引用)相同,而这又要求 `AREF`/`HREF` 这种地址计算已经被 CSE 归并成同一个引用。如果 CSE 没把两个相同的 `AREF` 合并,L2L 就会看到两个不同的 `op1`,错过优化。所以 FWD 和 CSE 是协作的——CSE 先归并地址,FWD 再转发 load。

找到匹配的旧 load,返回它的引用 `ref`——当前 load 直接复用旧 load 的结果,不用再发射。找不到返回 0,表示没法优化。

### 4.4 入口函数:lj_opt_fwd_aload / lj_opt_fwd_hload

`fwd_ahload` 是个 `static` 函数,对外暴露的是两个包装(`lj_opt_mem.c:281-297`):

```c
/* ALOAD forwarding. */
TRef LJ_FASTCALL lj_opt_fwd_aload(jit_State *J)
{
  IRRef ref;
  if ((ref = fwd_ahload(J, fins->op1)) ||
      (ref = fwd_aload_reassoc(J)))
    return ref;
  return EMITFOLD;
}

/* HLOAD forwarding. */
TRef LJ_FASTCALL lj_opt_fwd_hload(jit_State *J)
{
  IRRef ref = fwd_ahload(J, fins->op1);
  if (ref)
    return ref;
  return EMITFOLD;
}
```

两者都调 `fwd_ahload`(传当前 load 的地址引用 `fins->op1`)。如果能转发(返回非 0),返回那个引用;否则返回 `EMITFOLD`——意思是"没法优化,原样发射这条 load"。

注意 `lj_opt_fwd_aload` 多了一步 `fwd_aload_reassoc`(下一小节讲),这是因为数组访问有个特殊的"跨循环重结合"情形,值得单独处理。`HLOAD` 没有这个。

这两个函数就是 fold 引擎通过 `LJFOLDX` 调用的入口(`lj_opt_fold.c:2124-2137`)。每录制到一条 `ALOAD`/`HLOAD`,fold 引擎就调它们,尝试转发;转发成功就省一条 load,失败就老实发射。整个过程对录制器透明——录制器只管发 `ALOAD`,优化是 fold+别名分析在背后做的。

### 4.5 ALOAD 的跨循环重结合:fwd_aload_reassoc

`fwd_aload_reassoc`(`lj_opt_mem.c:258-278`)处理一个循环里很常见的模式,值得单独一看:

```c
/* Reassociate ALOAD across PHIs to handle t[i-1] forwarding case. */
static TRef fwd_aload_reassoc(jit_State *J)
{
  IRIns *irx = IR(fins->op1);
  IRIns *key = IR(irx->op2);
  if (key->o == IR_ADD && irref_isk(key->op2)) {
    IRIns *add2 = IR(key->op1);
    if (add2->o == IR_ADD && irref_isk(add2->op2) &&
	IR(key->op2)->i == -IR(add2->op2)->i) {
      IRRef ref = J->chain[IR_AREF];
      IRRef lim = add2->op1;
      if (irx->op1 > lim) lim = irx->op1;
      while (ref > lim) {
	IRIns *ir = IR(ref);
	if (ir->op1 == irx->op1 && ir->op2 == add2->op1)
	  return fwd_ahload(J, ref);
	ref = ir->prev;
      }
    }
  }
  return 0;
}
```

注释说得很清楚:"handle t[i-1] forwarding case"。考虑这个循环:

```lua
for i = 2, n do
  s = s + t[i-1]     -- 第 i 次循环读 t[i-1]
end
```

第 `i` 次循环读 `t[i-1]`。但循环变量 `i` 是个 PHI(循环头汇合的变量),`i-1` 经过窄化后会变成 `(i-1)` 这种形式。下一轮循环进来,`i` 变成 `i+1`(PHI 的迭代),`t[i]` 其实就是上一轮的 `t[(i-1)+1]`——理论上能复用上一轮的 load。

这个函数识别的模式是:当前 `AREF` 的 key 是 `ADD(base, k1)`,而 `base` 又是 `ADD(idx, k2)`,且 `k1 == -k2`(即 `k1 + k2 == 0`)。这意味着当前 key 等价于 `idx`(偏移互相抵消)。它就去链表里找一个 key 是 `idx` 的旧 `AREF`,把它的 load 结果转发过来。

这是个跨 PHI 的重结合(reassociation across PHI)——上一章 P3-09 §8.3 讲 fold 时强调了"不跨 PHI 折叠"的铁律,但这里是 `lj_opt_mem.c` 里一个**受控的例外**:它不是在算术层跨 PHI(那会破坏 SSA 结构),而是在内存引用层,通过识别"偏移抵消"的代数模式,找到一个语义等价的旧引用来转发。注释里那条 `if (irx->op1 > lim) lim = irx->op1;` 是在确保不会跨过不安全的边界。

这个函数体现了 LuaJIT 在循环优化上的一个细致努力:循环里的表访问(尤其是滑动窗口式的 `t[i-1]`、`t[i]`、`t[i+1]` 这种)极其常见,值得专门写一段重结合逻辑去抓住它。虽然代码不长,但收益在数值计算、字符串处理等循环密集的 Lua 代码里很可观。

## §5 死存储消除:lj_opt_dse_ahstore 怎么删掉多余的写

读转发讲完了,本节讲它的对偶——**死存储消除(DSE,Dead-Store Elimination)**。读转发针对 load(消除多余的读),DSE 针对 store(消除多余的写)。函数是 `lj_opt_dse_ahstore`(`lj_opt_mem.c:352-392`),处理 `ASTORE` 和 `HSTORE`。

### 5.1 三种死存储的模式

什么样的 store 是"死"的、可以删掉?有三种情形:

**情形一:重复写同一个值(`x = v; ...; x = v`,值相同)**。一个槽先写了值 `v`,后来又写了一次同样的值 `v`。第二次写完全是多余的——槽里的值没变。删掉第二次。

**情形二:写完从没被读,就被覆盖(`x = v1; ...; x = v2`,中间没人读 `x`)**。第一次写 `v1` 之后,从来没有任何 load 读过这个槽,紧接着第二次写 `v2` 把它覆盖了。那么第一次写是"死"的——它的结果从来没被用过。删掉第一次(改成 NOP),只留第二次。

**情形三:可能冲突但值相同(`MAY` alias 且值相同)**。两个 store 可能写同一个地方(不确定),但写的值一样。这种情况下,即使它们真的写了同一个地方,后一个也是多余的(值没变)。可以删后一个。

DSE 就是识别这三种情形,安全地删掉冗余的 store。下面看代码怎么实现。

### 5.2 代码:搜索冗余或冲突的 store

```c
/* ASTORE/HSTORE elimination. */
TRef LJ_FASTCALL lj_opt_dse_ahstore(jit_State *J)
{
  IRRef xref = fins->op1;  /* xREF reference. */
  IRRef val = fins->op2;  /* Stored value reference. */
  IRIns *xr = IR(xref);
  IRRef1 *refp = &J->chain[fins->o];
  IRRef ref = *refp;
  while (ref > xref) {  /* Search for redundant or conflicting stores. */
    IRIns *store = IR(ref);
    switch (aa_ahref(J, xr, IR(store->op1))) {
    case ALIAS_NO:
      break;  /* Continue searching. */
    case ALIAS_MAY:	/* Store to MAYBE the same location. */
      if (store->op2 != val)  /* Conflict if the value is different. */
	goto doemit;
      break;  /* Otherwise continue searching. */
    case ALIAS_MUST:	/* Store to the same location. */
      if (store->op2 == val)  /* Same value: drop the new store. */
	return DROPFOLD;
      /* Different value: try to eliminate the redundant store. */
      if (ref > J->chain[IR_LOOP]) {  /* Quick check to avoid crossing LOOP. */
	IRIns *ir;
	/* Check for any intervening guards (includes conflicting loads).
	** Note that lj_tab_keyindex and lj_vm_next don't need guards,
	** since they are followed by at least one guarded VLOAD.
	*/
	for (ir = IR(J->cur.nins-1); ir > store; ir--)
	  if (irt_isguard(ir->t) || ir->o == IR_ALEN)
	    goto doemit;  /* No elimination possible. */
	/* Remove redundant store from chain and replace with NOP. */
	*refp = store->prev;
	lj_ir_nop(store);
	/* Now emit the new store instead. */
      }
      goto doemit;
    }
    ref = *(refp = &store->prev);
  }
doemit:
  return EMITFOLD;  /* Otherwise we have a conflict or simply no match. */
}
```

逐段拆。`fins` 是当前正在处理的 store(`ASTORE`/`HSTORE`)。`xref = fins->op1` 是它写的地址,`val = fins->op2` 是它写的值。`refp` 指向当前 store 链表的头指针(`J->chain[ASTORE]` 或 `J->chain[HSTORE]`),`ref = *refp` 是链表头——即最近的旧 store。循环往前找所有 `ref > xref`(定义在当前 store 之前的)旧 store。

对每个旧 store,调 `aa_ahref` 判断,分三种:

**`ALIAS_NO`**(`lj_opt_mem.c:362`):旧 store 写的不是当前地址,无关。`break` 继续找下一个。注意这里 `refp = &store->prev` 这一步(在循环末尾 `ref = *(refp = &store->prev)`)——它把 `refp` 指向前一个 store 的 `prev` 字段地址。这是个技巧:如果后面要删除某个 store(把它的 `prev` 摘链),`refp` 指向"指向它的那个指针",删除时 `*refp = store->prev` 就能跳过它。这是单链表删除的标准手法。

**`ALIAS_MAY` 且值不同**(`lj_opt_mem.c:364-366`):旧 store 可能写同一地址,而且写的值和当前不同。这意味着——如果它们真的写同一地址,那顺序很重要(后写覆盖先写),不能动它们。保守处理,`goto doemit` 直接发射当前 store,不做任何消除。

**`ALIAS_MAY` 且值相同**(`lj_opt_mem.c:366-367`):旧 store 可能写同一地址,但值一样。即使真的写同一地址,当前 store 也是多余的(值没变)。但这种情况下不能立刻 `DROPFOLD`——因为后面可能还有别的 `ALIAS_MUST` store 能做更强的消除。所以 `break` 继续找,看有没有更好的机会。这是个保守但不会错过优化的策略。

**`ALIAS_MUST`**(`lj_opt_mem.c:368-386`):旧 store 一定写同一地址。这里又分两种:
- 如果值相同(`store->op2 == val`):当前 store 是完全多余的(同一个地方写了两遍同样的值)。`return DROPFOLD` 直接丢弃当前 store。这就是情形一。
- 如果值不同:这是情形二——同一个地址先后写了不同的值。**前面那个旧 store 可能是死的**(如果中间没人读过)。代码尝试消除它:先检查 `ref > J->chain[IR_LOOP]`(不在同一循环迭代内,避免破坏循环 SSA),然后扫描旧 store 到当前点之间有没有 **guard 或 ALEN**(行 378-380)——如果有任何 guard,说明中间可能有个 side exit 依赖了那个旧 store 的效果(比如 snapshot 里恢复了那个槽的值),不能删;如果中间有 `ALOAD`/`HLOAD`(它们是 guard 的,见 §6.2 关于 `IRTG`),说明有人读过那个槽,旧 store 不是死的,不能删。确认安全后,行 382-383 把旧 store 从链表摘除并改成 NOP(`lj_ir_nop`),然后 `goto doemit` 发射当前 store。净效果:删了旧 store,留了新 store。

### 5.3 为什么 DSE 要小心 guard

§5.2 那段"扫描中间有没有 guard"特别值得强调,因为它是 DSE sound 的关键。

考虑这个情形:

```text
%10 = ASTORE %5, KINT(42)    ; 写 t.k = 42
... (中间有个 guard,在某条件下 side exit)
%20 = ASTORE %5, KINT(99)    ; 又写 t.k = 99
```

如果中间那个 guard 触发了 side exit,snapshot 会把当前各个槽的值恢复给解释器——而 `t.k` 此刻的值应该是 **42**(第一个 store 写的)。如果把第一个 store 删了,snapshot 恢复出来的 `t.k` 就不是 42 了,解释器拿到的状态就错了——这就是为什么中间有 guard 就不能删旧 store。

行 378-380 那个扫描就是在检查这个:

```c
for (ir = IR(J->cur.nins-1); ir > store; ir--)
  if (irt_isguard(ir->t) || ir->o == IR_ALEN)
    goto doemit;  /* No elimination possible. */
```

`irt_isguard(ir->t)` 判断一条指令是不是 guard。只要有任何 guard 夹在两个 store 之间,就不能删——因为 side exit 可能依赖旧 store 的效果。`ALEN`(`#t` 这种取表长度的指令)也被算进来,因为它隐式依赖表的状态。注释还提到 `lj_tab_keyindex` 和 `lj_vm_next`(next 迭代)不需要 guard——因为它们后面总跟着一个带 guard 的 `VLOAD`,扫描时那个 `VLOAD` 会挡住,所以不用单独检查。

这个细节体现了 DSE 的 sound 纪律:**删除一个 store,不光要确认"它的值后来被覆盖了",还要确认"在它被覆盖之前,没有任何 side exit 依赖过它"**。Snapshot 是 JIT 正确性的命脉(P0-01 §8、P5 篇),任何可能影响 snapshot 的优化都得小心翼翼。DSE 通过扫描 guard,把这道关守住了。

### 5.4 LJFOLD 入口

DSE 函数的入口也在 fold 表里(`lj_opt_fold.c:2471-2474`):

```c
LJFOLD(ASTORE any any)
LJFOLD(HSTORE any any)
LJFOLDX(lj_opt_dse_ahstore)
```

`ASTORE` 和 `HSTORE` 共用一个 DSE 函数(因为 `aa_ahref` 同时处理数组和哈希)。每录制到一条 `ASTORE`/`HSTORE`,fold 引擎调 `lj_opt_dse_ahstore`,能消除就消除(`DROPFOLD` 或改旧 store 为 NOP),不能就 `EMITFOLD` 原样发射。

注意这里的 `any any`——两个操作数都是通配符,意味着这条规则兜底**所有** `ASTORE`/`HSTORE`。P3-09 §7.3 引用的文件头注释说"all loads, stores and allocations must have an any/any rule to avoid being passed on to CSE"——这条 `ASTORE any any` 就是兜底,确保 store 不会漏到 CSE 那里被当纯运算错误去重。

## §6 为什么这些优化是 sound 的:别名分析的保守纪律

讲了 AA、FWD、DSE 的机制,我们要专门用一节回答贯穿全章的核心问题:**凭什么相信这些优化不会改变程序结果?** 这一节是本章的"安全"支柱,对应全书主线里"安全"两个字。

### 6.1 纪律一:三值逻辑的保守取向

第一道保障是 §1.3 讲的三值逻辑,以及它在每个分支里的保守取向。回顾一下三个值的语义:

- `ALIAS_NO`:精确的"不"。只有确凿证明两个引用不重叠才返回。
- `ALIAS_MUST`:精确的"是"。只有确凿证明两个引用同一地址才返回。
- `ALIAS_MAY`:不精确的"可能"。**凡是判断不了的,一律落到这里**。

关键在于 `ALIAS_MAY` 的**取向**——它代表"可能冲突",而不是"可能不冲突"。在 FWD 里,遇到 `ALIAS_MAY` 的 store,代码选择**停止转发**(`lim = ref; goto cselim`),把搜索范围收紧;在 DSE 里,遇到 `ALIAS_MAY` 且值不同,代码选择**直接放弃消除**(`goto doemit`)。两种选择都是"宁可不优化,也不冒险"。

这种取向是 sound 的根本。设想如果反过来——`ALIAS_MAY` 当作"可能不冲突"去优化,那就会在两个引用其实重叠(运行时碰巧指向同一地址)的时候,错误地复用旧值或删除 store,改变结果。三值逻辑的保守取向杜绝了这种错误。

### 6.2 纪律二:load 带 guard,类型必须对

第二道保障藏在表访问的录制里。回忆 §2.3:一次 `t.k` 读,录出来的 `HLOAD` 是带 guard 的(录制时用 `IRTG`,即 IR type guard,`lj_record.c:1604` 的 `emitir(IRTG(loadop, t), xref, 0)`)。这个 guard 检查的是**读出来的值的类型**——如果运行时类型和编译期假设的不一样(比如编译期认为 `t.k` 是整数,运行时却是字符串),guard 触发 side exit。

这意味着:每一次成功的 load 转发,本质上是"复用了一次已经通过 guard 验证的值"。被复用的那个旧 load,它读出来的类型已经被它的 guard 确认过了;只要中间没有冲突 store 改过这个槽,槽里的值(包括类型)就不变,复用就是安全的。

更深一层,这也解释了 §5.3 为什么 DSE 扫描 guard:如果一个旧 store 之后有个 load(带 guard)读过它,说明那个 load 依赖了 store 写的值;如果中间有 side exit,snapshot 里那个槽的值应该是 store 写的。所以有 guard 就不能删 store。load 带 guard 这件事,是 FWD 和 DSE 都依赖的正确性基石。

### 6.3 纪律三:NEWREF 和 table.clear 的全面防御

第三道保障针对表结构本身可能变化的情况。Lua 的 table 不是静态结构——插入新键(`NEWREF`)可能触发 rehash,把所有键的位置挪一遍;`table.clear` 会清空整个表。这两种情况会让所有已经算出来的地址引用(基于旧的 array/node 布局)全部失效。

LuaJIT 对此有专门的防御:
- `fwd_aa_tab_clear`(`lj_opt_mem.c:76-87`):在做"从 TDUP 读初值"这种优化前,扫描中间有没有 `table.clear` 调用(`IRCALL_lj_tab_clear`),如果有就放弃。
- `fwd_ahload` 第二段(`lj_opt_mem.c:187-205`):处理 `NEWREF` 插入数字键可能 rehash 的边界,遇到就 `goto cselim` 放弃折叠。
- `lj_opt_fwd_tptr`(`lj_opt_mem.c:90-101`):检查 `NEWREF` 和 `table.clear` 对表指针的影响,被 `HREFK` 转发等场景调用。
- `aa_ahref` 里 `HREF` 的兜底返回 `ALIAS_MAY`(`lj_opt_mem.c:155-158`):动态哈希查找的地址,运行时才算,一律当可能别名。

这些防御合起来,确保了"表结构变化"这种灾难性的情况不会被别名分析漏掉。漏掉一个 `NEWREF` 导致的 rehash,会让所有基于旧布局的地址判断全错——这是不可接受的。LuaJIT 用"宁可放过优化"的保守策略,把这个风险压到零。

### 6.4 纪律四:SSA 的支配性保障转发安全

第四道保障来自 SSA 本身。回忆 P3-09 §9 讲的 SSA 性质:每个引用唯一对应一条定义,且该定义支配所有使用它的地方。

这个性质对 FWD 至关重要。当 `fwd_ahload` 找到一个同地址的旧 load 准备复用时,SSA 保证:那个旧 load 已经被执行过(它的定义支配当前点),它的值是确定的(单赋值,不会变)。所以复用它,等价于"用同一个确定的值",绝不会拿到一个未初始化或被改过的值。

同样,DSE 删除一个旧 store 时,SSA 保证那个 store 的地址和值都是确定的(操作数引用不变),删除它不会影响别的依赖(因为 SSA 里没有"别的指令隐式依赖这个 store 的副作用"这种事——side exit 的依赖通过 guard 扫描单独处理了)。

所以 FWD/DSE 的 sound,是"三值保守逻辑 + load 带 guard + NEWREF/clear 防御 + SSA 支配性"四道保障合起来的结果。每一道都对应一类可能出错的情况,合起来覆盖了所有已知的正确性风险。这就是 LuaJIT 敢自动消除冗余内存访问的底气。

## §7 其他几类内存访问:ULOAD / FLOAD / XLOAD

表访问(A/H)是 `lj_opt_mem.c` 的重头戏,但不是全部。文件还覆盖了三类内存访问:**upvalue**(U)、**字段**(F)、**外部内存/FFI**(X)。它们各自有独立的 AA 函数和 FWD/DSE 函数,结构上和表访问的三件套一样,只是别名判断的依据不同。这一节简要覆盖,重点讲它们和表访问的区别。

### 7.1 ULOAD:upvalue 的简化别名分析

Upvalue 是 Lua 闭包捕获的外层局部变量。它的别名分析比表简单,因为 upvalue 的"身份"很明确——由(所属函数, upvalue 索引)唯一确定。`aa_uref`(`lj_opt_mem.c:465-485`):

```c
static AliasRet aa_uref(IRIns *refa, IRIns *refb)
{
  if (refa->op1 == refb->op1) {  /* Same function. */
    if (refa->op2 == refb->op2)
      return ALIAS_MUST;  /* Same function, same upvalue idx. */
    else
      return ALIAS_NO;  /* Same function, different upvalue idx. */
  } else {  /* Different functions, check disambiguation hash values. */
    if (((refa->op2 ^ refb->op2) & 0xff)) {
      return ALIAS_NO;  /* Upvalues with different hash values cannot alias. */
    } else if (refa->o != refb->o) {
      ...
      return ALIAS_NO;
    } else {
      return ALIAS_MAY;
    }
  }
}
```

同一个函数的不同 upvalue 索引,必然是不同的 upvalue(`ALIAS_NO`)——这是个很强的结论,因为 Lua 的 upvalue 在函数创建时就固定了。不同函数的 upvalue,用一个 8 位的"消歧哈希"(`op2` 的低 8 位)来判断——哈希不同则不别名,哈希相同则可能别名。

文件头的注释(`lj_opt_mem.c:457-464`)诚实地说明这是个**非常简化**的分析:"only disambiguates between the unique upvalues of the same function... A more precise analysis would be feasible with the help of the parser"。也就是说,跨函数的 upvalue 别名判断是粗略的,但够用(因为大多数 upvalue 是只读的)。这是个刻意的简化,换取编译速度。

`lj_opt_fwd_uload`(`lj_opt_mem.c:488-518`)和 `lj_opt_dse_ustore`(`lj_opt_mem.c:521-568`)的结构和表访问的对应函数几乎一样:搜索冲突 store,按 `ALIAS_NO/MAY/MUST` 三分支处理。DSE 还多处理了一个 `OBAR`(write barrier)的细节(`lj_opt_mem.c:550-559`)——upvalue store 可能涉及 GC 屏障,删除 store 时要把关联的屏障也一起处理。

### 7.2 FLOAD:字段的类型消歧

字段访问(`FLOAD`/`FSTORE`)针对的是结构体的命名字段(比如表的 `metatable`、`asize`、`hmask`,字符串的 `len` 等)。它的别名分析 `aa_fref`(`lj_opt_mem.c:576-586`)极其简洁:

```c
static AliasRet aa_fref(jit_State *J, IRIns *refa, IRIns *refb)
{
  if (refa->op2 != refb->op2)
    return ALIAS_NO;  /* Different fields. */
  if (refa->op1 == refb->op1)
    return ALIAS_MUST;  /* Same field, same object. */
  else if (refa->op2 >= IRFL_TAB_META && refa->op2 <= IRFL_TAB_NOMM)
    return aa_table(J, refa->op1, refb->op1);  /* Disambiguate tables. */
  else
    return ALIAS_MAY;  /* Same field, possibly different object. */
}
```

字段访问的"地址"由(对象, 字段 ID)决定。`op2` 是字段 ID(一个编译期常量,见 `lj_ir.h:193-215` 的 `IRFLDEF`)。判断逻辑:
- 字段 ID 不同 → 必然是不同的字段 → `ALIAS_NO`。这个结论极强——`t.metatable` 和 `t.asize` 是同一个对象的不同字段,内存里不重叠(结构体字段不重叠),绝不别名。
- 字段 ID 相同、对象相同 → 同一个对象的同一个字段 → `ALIAS_MUST`。
- 字段 ID 相同、对象不同,且是表字段(`IRFL_TAB_META` 到 `IRFL_TAB_NOMM`)→ 调 `aa_table` 看两个对象是不是同一个表。
- 其他(同字段、不同对象、非表字段)→ `ALIAS_MAY`。

注释(`lj_opt_mem.c:572-575`)说"Field loads are cheap and field stores are rare. Simple disambiguation based on field types is good enough."——字段访问本身便宜(就是结构体偏移访问),store 又少,所以用简单的字段 ID 消歧就够,不必像表访问那样做精细的键分析。这是个务实的取舍。

`lj_opt_fwd_fload`(`lj_opt_mem.c:589-618`)还处理了一个特殊优化:从 `TNEW`/`TDUP` 读 `metatable` 字段,新表的 metatable 一定是 NULL,直接返回 `lj_ir_knull(J, IRT_TAB)`(`lj_opt_mem.c:609-613`)——把一次字段读折叠成常量 NULL。

### 7.3 XLOAD:FFI 内存的严格别名规则

最复杂的是 `XLOAD`/`XSTORE`,针对 FFI 的裸内存访问(通过指针读写 C 数据)。它的别名分析 `aa_xref`(`lj_opt_mem.c:714-760`)是最长的,因为要处理指针算术、类型双关(type punning)、cdata 分配等复杂情况。

核心思路有两层:

**第一层:基于偏移的消歧**(`lj_opt_mem.c:721-753`)。两个指针引用,如果都是 `base ± offset` 的形式(偏移是编译期常量),且 base 相同,那么看 offset 和访问的大小:
- offset 相同、大小相同、类型类别相同(都是整数或都是浮点)→ `ALIAS_MUST`(同一地址同一类型)。
- 两个访问区间不重叠(`ofsa + sza <= ofsb || ofsb + szb <= ofsa`)→ `ALIAS_NO`(比如读 `p` 的低 4 字节和写 `p+8` 的 8 字节,不重叠)。
- 否则(重叠或类型双关)→ `ALIAS_MAY`,强制重新加载。

**第二层:严格别名规则(strict aliasing)**(`lj_opt_mem.c:739-757`)。注释说得很明白:"This implements (very) strict aliasing rules. Different types do NOT alias, except for differences in signedness."。也就是说:**不同类型的访问不别名**(除了有符号/无符号的差异)。比如一个 `int *` 和一个 `float *`,只要类型不同,编译器就认为它们不会指向同一地址(C 的严格别名规则,和 GCC 的 `-fstrict-aliasing` 一个道理)。这让 FFI 代码里不同类型的访问能各自独立优化。

`aa_xref` 还调 `aa_cnew`(`lj_opt_mem.c:701-711`)做 cdata 的逃逸分析(类似 `aa_escape`),判断两个指针是不是指向不同的 cdata 分配。

`lj_opt_fwd_xload`(`lj_opt_mem.c:824-889`)还处理了 `IRXLOAD_READONLY`(只读内存,直接 CSE)和 `IRXLOAD_VOLATILE`(volatile 内存,每次都重新读,不优化)两个标志,以及一个跨循环的 `reassoc_xref`(类似 `fwd_aload_reassoc`,处理 `p[i-1]` 这种)。

特别值得一提的是 `lj_opt_fwd_xload` 里 store-to-load 转发时的**类型转换处理**(`lj_opt_mem.c:847-861`):如果 store 写的类型和 load 读的类型不同(但有符号/无符号差异或截断关系),转发时要插一条 `CONV` 指令做转换,然后 `RETRYFOLD` 让 fold 引擎重新处理。这个细节保证了"转发跨类型的值"也能正确——读出来的位模式按目标类型解释。

## §8 别名分析之外:几个辅助优化

`lj_opt_mem.c` 里还有几个不属于"标准 AA/FWD/DSE 三件套"的辅助函数,值得简要提一下,它们处理的是表访问优化的一些边角但重要的情形。

### 8.1 HREFK 转发:lj_opt_fwd_hrefk

`HREFK` 是"带常量槽位的哈希引用"。它的转发 `lj_opt_fwd_hrefk`(`lj_opt_mem.c:300-321`)有点特殊,因为它要处理 `NEWREF` 的影响:

```c
TRef LJ_FASTCALL lj_opt_fwd_hrefk(jit_State *J)
{
  IRRef tab = fleft->op1;
  IRRef ref = J->chain[IR_NEWREF];
  while (ref > tab) {
    IRIns *newref = IR(ref);
    if (tab == newref->op1) {
      if (fright->op1 == newref->op2 && fwd_aa_tab_clear(J, ref, tab))
	return ref;  /* Forward from NEWREF. */
      else
	goto docse;
    } else if (aa_table(J, tab, newref->op1) != ALIAS_NO) {
      goto docse;
    }
    ref = newref->prev;
  }
  /* No conflicting NEWREF: key location unchanged for HREFK of TDUP. */
  if (IR(tab)->o == IR_TDUP && fwd_aa_tab_clear(J, tab, tab))
    fins->t.irt &= ~IRT_GUARD;  /* Drop HREFK guard. */
docse:
  return CSEFOLD;
}
```

`HREFK` 的特殊之处在于它带一个 guard(检查哈希表大小 `hmask` 没变,否则槽位会错)。但如果表是从 `TDUP` 来的,且中间没有 `NEWREF` 改过它、也没有 `table.clear`,那么表的结构是固定的,这个 guard 可以**去掉**(`fins->t.irt &= ~IRT_GUARD`,行 318)。去掉一个 guard 意味着少一条运行时检查,机器码更精简。这是个很有价值的优化——很多从常量表读字段的场景都能受益。

另外,如果一个 `NEWREF` 刚好插入了 `HREFK` 要找的那个键,`HREFK` 可以直接转发成那个 `NEWREF` 的结果(`return ref`,行 308)——刚插入的键的地址就是 `NEWREF` 返回的。

### 8.2 ALEN 转发:lj_opt_fwd_alen

`ALEN` 是取表长度(`#t`)。它的优化 `lj_opt_fwd_alen`(`lj_opt_mem.c:395-453`)处理两类情形:
- 如果中间有往数字键的 `HSTORE`(可能改变数组长度),收紧搜索下限。
- 识别 `t[#t+1] = x` 这种"尾部追加"的惯用法(行 432-439):如果检测到这种模式,设置一个 "hint",帮助后端生成更高效的代码(直接追加,不用重新算长度)。

### 8.3 wasnonnil 分析:lj_opt_fwd_wasnonnil

`lj_opt_fwd_wasnonnil`(`lj_opt_mem.c:948-983`)是个辅助分析,判断"某个表槽的旧值是不是非 nil"。这个信息被录制阶段用来决定要不要插入某些 guard(`lj_record.c:1657`)。如果已知旧值非 nil(比如之前有个 store 写了非 nil 值),就可以省掉一些"值是否 nil"的检查。它的逻辑是:搜索同地址的旧 store 和旧 load,如果它们写/读的是非 nil 值,就返回 1(确定非 nil);否则返回 0(可能 nil)。

## §9 把整个过程串起来:一次 `t.k + t.k` 的完整优化

讲完了所有机制,我们用一个最小例子,把别名分析和 load forwarding 的完整流程走一遍,让你看见它**作为一个整体**怎么运作。

例子:

```lua
local function f(t)
  return t.k + t.k     -- 读两次 t.k
end
```

**第 1 步:录制第一个 `t.k`**

录制器进入 `lj_record_idx`(`lj_record.c:1530`),处理 `t.k`。`rec_idx_key`(`lj_record.c:1437`)判断 `k` 是字符串常量,算出它在哈希表里的槽位,发:

```text
%2 = FLOAD  %1, IRFL_TAB_NODE     ; 取 node 基址
%3 = FLOAD  %1, IRFL_TAB_HMASK    ; 取 hmask
%4 = EQ     %3, KINT(h0)          ; guard: hmask 没变
%5 = HREFK  %2, KSLOT(KSTR("k"), slot)   ; 算出 t.k 的地址
```

然后发 load 读值(`lj_record.c:1604`):

```text
%6 = HLOAD  %5      ; 读 t.k 的值(带类型 guard)
```

这些 IR 每发一条都过 fold 引擎。`FLOAD`/`HREFK`/`HLOAD` 各自走自己的 fold 规则(§2.2、§8.1)。假设这里是第一次访问,没有旧指令可转发,都正常发射。

**第 2 步:录制第二个 `t.k`**

录制器又处理一次 `t.k`(同一个表达式)。同样发:

```text
%8 = FLOAD  %1, IRFL_TAB_NODE
%9 = FLOAD  %1, IRFL_TAB_HMASK
%10 = EQ    %9, KINT(h0)
%11 = HREFK %8, KSLOT(KSTR("k"), slot)
%12 = HLOAD %11
```

但这次,每条 IR 过 fold 时,优化机会出现了。

**第 3 步:FLOAD 的 CSE**

`%8 = FLOAD %1, IRFL_TAB_NODE` 进 fold。`FLOAD` 走到 `lj_opt_fold.c:2275-2278` 的 `LJFOLD(FLOAD any IRFL_TAB_ARRAY)` 等规则(这些走的是 `CSEFOLD`,因为 `FLOAD` 的字段引用是纯运算)。CSE 发现前面已经有 `%2 = FLOAD %1, IRFL_TAB_NODE`(同样的对象、同样的字段),直接返回 `%2`。`%8` 不发射。

同理 `%9`、`%10` 也被 CSE 掉,复用 `%3`、`%4`。

**第 4 步:HREFK 的转发**

`%11 = HREFK %8, KSLOT(...)` 进 fold。现在 `%8` 已经被 CSE 成 `%2` 了,所以这条实际是 `HREFK %2, KSLOT(KSTR("k"), slot)`。它走 `lj_opt_fwd_hrefk`(`lj_opt_fold.c:2209`)。这个函数检查中间没有冲突的 `NEWREF`(`lj_opt_mem.c:303-315`),然后走 `docse` 走 `CSEFOLD`(`lj_opt_mem.c:319-320`)。CSE 发现前面有 `%5 = HREFK %2, KSLOT(KSTR("k"), slot)`(现在 `%8`=`%2`,操作数完全相同),返回 `%5`。`%11` 不发射。

**第 5 步:HLOAD 的 load forwarding**

`%12 = HLOAD %11` 进 fold(`lj_opt_fold.c:2136`,`lj_opt_fwd_hload`)。现在 `%11` 已经是 `%5` 了,所以这是 `HLOAD %5`。`fwd_ahload`(`lj_opt_mem.c:162`)被调用:

- 第一段(`lj_opt_mem.c:168-178`):搜索 `HSTORE` 链表(`fins->o + IRDELTA_L2S = HLOAD + delta = HSTORE`)。假设中间没有 `HSTORE`(这个函数体里没有写表),循环找不到冲突 store。
- 第二段(`lj_opt_mem.c:180-243`):`%5` 的表不是 `TNEW`/`TDUP`(是外部传入的 `%1`),跳过。
- 第三段 `cselim`(`lj_opt_mem.c:245-254`):搜索 `HLOAD` 链表,找有没有 `op1 == xref`(即 `op1 == %5`)的旧 load。找到了——`%6 = HLOAD %5`!返回 `%6`。

`%12` 转发成 `%6`,不发射。

**最终结果**

原本要发 10 条 IR(两次完整的 `t.k` 读),经过 CSE + 别名分析 + load forwarding,实际只发了 6 条(第一次的 `%2`-`%6`),第二次的 4 条全部复用第一次的结果。机器码里,`t.k + t.k` 变成"读一次内存 + 一次加法",而不是"读两次 + 加法"。

这就是别名分析和 load forwarding 的威力。在真实的 Lua 程序里,一个函数里访问同一个字段好几次(比如 `self.x`、`self.x` 反复读)是常态,这个优化把那些重复访问压成了零开销。收益巨大。

而且整个过程对程序员完全透明——录制器老老实实发 IR,fold + 别名分析在背后把冗余消掉。这正是优化 pass 存在的意义:让程序员写出自然的代码,编译器负责把它压榨到最快。

## §10 ★对照:官方 Lua 与 JVM/V8

讲完了 LuaJIT 的内存优化,我们把它和两个对象放在一起对照。

### 10.1 对照一:官方 Lua(切"每次访问都走完整查表")

官方 Lua 是纯解释器,它**没有任何内存优化 pass**。每一次 `t.k` 访问,都老老实实走完整的查表逻辑——算哈希、找槽、比较键、读值。即使两次 `t.k` 紧挨着,第二次也完全不记得第一次的结果,从头查一遍。

具体地,官方 Lua 的表访问走的是 `lj_tab_get`/`luaH_get` 这种运行时函数(C 实现):对字符串键,算哈希定位桶,遍历桶里的节点比较键。这一套下来,即使哈希命中,也要好几十个时钟周期。一段代码里读同一个字段十次,就是十次完整的查表。

对比之下,LuaJIT 录制时也发完整的查表 IR(§2.3 那五条),但别名分析在编译期就把后九次全消掉了——因为编译器能"看到"这十次访问的是同一个地方、中间没人改。机器码里只剩一次真正的内存读。这就是内存优化的收益:把解释器里"每次都查表"的开销,在编译期压成"只查一次"。

这条对照的结论:**官方 Lua 没有 AA/FWD/DSE,因为它没有可分析的 IR——每次访问都是独立的运行时查表;LuaJIT 有,因为它把访问翻成了 IR,而 IR 上的别名分析能识别冗余**。内存优化是 JIT 相对解释器的又一项独立增益,和"机器码比解释快"叠加。

### 10.2 对照二:JVM / V8(切"trace 级 AA vs 方法级逃逸分析")

更有意思的对照是 LuaJIT 的别名分析和 JVM/V8 的内存优化。两者都做 load/store 优化,但**分析的精度、范围、核心抓手**截然不同。

**精度对照**。LuaJIT 的 AA 是**保守的、局部的**。我们在 §3 看到,它的判断基于"键是否相同/是否常量/下标算术/键类型"这些**结构化的、针对 Lua table 的规则**。遇到拿不准的(外部表、动态键),一律 `ALIAS_MAY` 放弃。而 JVM 的 C2 和 V8 的 TurboFan,做的是**逃逸分析(escape analysis)+ 标量替换(scalar replacement)**——它们能分析一个对象"有没有逃出方法",没逃的对象,根本不在堆上分配,而是把它的字段拆成一个个局部变量(标量化),这样所有对该对象字段的访问都变成对局部变量的访问,别名问题整个消失(局部变量没有别名)。

这是两种根本不同的思路:
- **LuaJIT**:table 已经在堆上(是 Lua 的唯一复合结构,无法避免堆分配),所以 AA 的任务是"判断两次访问是不是同一个堆位置"——在堆存在的前提下做精确判断。
- **JVM/V8**:很多对象其实不需要在堆上(它们没逃出方法),所以优化是"把对象消灭掉"——逃逸分析识别出不逃的对象,标量替换把它的字段变成局部变量,从根上消灭别名。

LuaJIT **不做**逃逸分析(table 是 Lua 的核心结构,几乎总是逃出当前 trace——trace 太窄,看不到对象的完整生命周期),所以它的 AA 停留在"判断堆位置别名"这一层。这是 trace JIT 的固有限制:trace 是一条线性路径,看不到方法的完整控制流,做不了方法级的逃逸分析。代价是那些"本可以标量化的对象",在 LuaJIT 里仍然是堆上的 table,每次字段访问仍然是内存读(虽然有 AA 消除冗余)。

**范围对照**。LuaJIT 的 AA 只在**一条 trace 内**有效。§9 那个 `t.k + t.k` 的优化,前提是两次访问在同一条 trace 里。如果它们分散在不同的 trace(或一个在 trace、一个在解释器),优化无效。而 JVM/V8 的优化在**整个方法**内有效——它能看到方法的所有控制流,跨分支、跨循环地分析对象生命周期和字段访问。

**核心抓手对照**。这个对照最能说明问题:
- **LuaJIT 的抓手是"Lua table 的结构化语义"**。它的 AA 规则(数组 vs 哈希、常量键、下标算术)是专门为 Lua table 设计的,利用了 table"数组+哈希两部分、键有类型"这些特性。这是 trace JIT 的优势——针对一种特定数据结构做精准优化。
- **JVM/V8 的抓手是"对象的分配点 + 控制流"**。它们的逃逸分析基于"对象在哪里分配、指针怎么流动、有没有跨方法边界",是通用的、面向所有对象类型的分析。

用一个具体的对照点收尾:**消除重复的 `obj.field` 访问**。LuaJIT 用 `aa_ahref` + `fwd_ahload` 判断"两次 `t.k` 是否同一槽位",能消除同 trace 内的重复访问。JVM C2 用逃逸分析判断"`obj` 是否逃逸",没逃就标量化,所有 `obj.field` 访问都变成寄存器访问——不光消除重复,连"访问"这个动作本身都消灭了(字段变成了寄存器里的局部变量)。**LuaJIT 让访问更快(消冗余),JVM 让访问消失(标量化)**——这背后的差距,就是 trace 级 AA 和方法级逃逸分析的差距。

### 10.3 对照表

| 维度 | LuaJIT(lj_opt_mem) | 官方 Lua | JVM / V8 |
|---|---|---|---|
| 是否有内存优化 | 有(AA+FWD+DSE) | 无 | 有(逃逸分析+标量替换等) |
| 优化范围 | 单条 trace(线性路径) | — | 整个方法(全控制流) |
| 别名分析基础 | table 结构化语义(数组/哈希/键类型) | — | 对象分配点 + 控制流 |
| 逃逸分析 | 简化版(aa_escape,仅看 store) | — | 完整方法级逃逸分析 |
| 标量替换 | 无(trace 太窄) | — | 有(核心优化) |
| load forwarding | 有(L2L + S2L) | — | 有 |
| 死存储消除 | 有(含 guard 扫描) | — | 有 |
| 表/对象字段访问 | 消冗余(仍走内存) | 每次完整查表 | 标量化(走寄存器) |
| 实现规模 | 992 行 | — | 数万行(C2 escape analysis 等) |

## §11 回扣主线:别名分析让"变成机器码"在内存层也精简

回到全书的主线:**把动态执行安全变成机器码**。别名分析在这条主线里扮演什么角色?

我们要看清:**安全变成机器码,在算术层和内存层都要"值"**。上一章的 fold 让算术运算精简——消掉 `x + 0`、把 `*8` 换成 `<<3`。但如果只精简了算术,内存访问还是一团冗余(每次读表都从头查、每次写表都老老实实写),那机器码仍然背着沉重的内存访问包袱——而内存访问恰恰是 Lua 程序里最频繁、最昂贵的操作。别名分析的工作,就是**把内存访问这一层也压到最精简**,让最终生成的机器码在内存操作上也每一拍都用在刀刃上。

具体地,`lj_opt_mem.c` 通过三类优化贡献于"快":

- **load forwarding**:同地址已读过、中间无冲突,复用旧结果,少一次内存读。对 Lua 这种"反复访问表字段"的代码,收益巨大。
- **store-to-load forwarding**:同地址刚写过、紧跟着读,直接用写进去的值,连内存都不用读。
- **dead-store elimination**:冗余的 store(写同样的值、或写了从没被读)删掉,少一次内存写。

而它通过四道保障贡献于"安全":

- **三值逻辑的保守取向**(`ALIAS_MAY` 宁可放弃优化)。
- **load 带 guard**,复用的值已通过类型检查。
- **NEWREF/table.clear 的全面防御**,表结构变化不被漏掉。
- **SSA 的支配性**,被复用的旧值确定可用。

再加上它是**和 fold 协作**的——fold 通过 `LJFOLDX` 把 load/store 转交给它,它处理完再交回 fold/CSE——这让内存优化无缝嵌入整个优化流水线,几乎不增加额外的编译延迟。

所以别名分析完美地体现了主线的张力平衡:**它追求快(激进地消除每一处可消除的冗余内存访问),但用保守的三值逻辑和全面的副作用防御保障安全(绝不消除不该消除的),同时用和 fold 的协作与 SSA 链表做到省(优化开销小)**。快、安全、省,在 `lj_opt_mem.c` 这一个文件里达成了统一。

而且,这件事对 Lua **尤其关键**——因为 table 是 Lua 唯一的复合结构,字段访问是性能热点。LuaJIT 能比官方 Lua 快几十倍,很大一部分增益就来自这里:官方 Lua 每次表访问都完整查表,LuaJIT 在编译期把冗余访问消掉,机器码里只剩最少的、必要的内存操作。可以说,**别名分析是 LuaJIT 性能优势的核心支柱之一**,和 trace 录制、guard、机器码生成同等重要。

讲到这里,优化篇的两大 pass——fold(算术)和 mem(内存)——都讲完了。但 IR 里还有一类指令我们没讲怎么优化:**分配**(`TNEW`/`TDUP`/`CNEW`)。分配是创建新对象的指令,它看起来必不可少(程序要建表嘛),但很多时候一个分配出来的对象,在 trace 内部用完就扔了,根本没人外部看到它——这种分配可以"沉到循环外"甚至完全消除。这就是下一章的主题:**分配消除(sink)与死代码消除(dce)**。如果说 fold 精简算术、mem 精简内存访问,那么 sink 精简的就是"对象的生灭"——它把那些"生了但白生"的对象消灭掉,让机器码连分配都省了。这是优化篇的第三根支柱,也是 trace JIT 在对象密集代码上保持高性能的关键。
