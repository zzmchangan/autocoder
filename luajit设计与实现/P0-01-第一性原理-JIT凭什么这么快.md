# P0-01 第一性原理:JIT 凭什么这么快

> **本书主线**:把动态执行安全变成机器码。**二分法**:解释器侧 ↔ JIT 侧。**★对照**:官方 Lua + JVM/V8。**源码**:LuaJIT 2.1.ROLLING。**基调**:纯直球,不用比喻;从第一性原理一步步推导。

## 引子:这本书要回答的问题

你大概听说过:LuaJIT 跑同样的 Lua 代码,可以比官方 Lua 快几十倍。

几十倍是个很吓人的数字。同样的语言、同样的语法、同样的程序,换一个实现就快这么多,这中间到底发生了什么?

答案只有一句话:**LuaJIT 把正在运行的代码,编译成了机器码,直接交给 CPU 执行。** 官方 Lua 没有做这件事。

这件事叫 JIT(Just-In-Time,即时编译)。这本书的全部内容,就是讲清楚它**到底是怎么做到的**。

但"把代码编译成机器码"这件事,一旦你认真去想,就会冒出一个看起来无解的难题——而整个 LuaJIT 的设计,都是为了绕开这个难题。所以这本书的真正主线,不是"怎么编译",而是:

> **Lua 是动态类型语言,编译时根本不知道一个变量下一秒是什么类型。这种语言,怎么可能安全地编译成机器码?**

这一章不急着回答。我们先从最基础的地方开始,一步步把这个难题**推导**出来。等你看清了难题,答案就会自然浮现。

## §1 第一性原理:CPU 只懂机器码

一切要从 CPU 说起。

你写的 Lua 代码、C 代码、Python 代码,长这样:

```lua
local x = 1
local y = x + 2
```

但 CPU 一个字都不认识。CPU 这块硅片,只认识一种东西:**机器码**——一串串的二进制数字,每一段对应一条指令,比如"把内存地址 0x100 的数取到寄存器""把两个寄存器里的数相加""把结果写回内存"。

所以,不管你用什么语言写程序,最后都必须**翻译**成机器码,CPU 才会执行。没有翻译,你的代码就是一堆字符,什么都不会发生。

那翻译这件事,什么时候做?这就是一切分歧的起点。

---

## §2 翻译代码的两条路:解释 vs 编译

翻译时机不同,分出两条路。

**第一条路:编译(提前翻译)**

C、Rust 走这条路。你写完代码,运行**之前**,先用编译器把整个程序翻译成机器码,存成一个可执行文件(Windows 上的 `.exe`)。等到运行的时候,CPU 直接读这个机器码文件,飞速执行。

- 优点:运行时不用再翻译,快。
- 代价:必须提前知道所有类型信息(C 里你写 `int x`,编译器才知道用整数指令);改了代码要重新编译。

**第二条路:解释(现场翻译)**

Python、Ruby、官方 Lua 走这条路。不提前翻译,而是写一个叫**解释器**的程序。运行的时候,解释器**一条一条地**读你的代码,读一条、理解一条、执行一条,实时进行。

拿这段代码举例:

```lua
local y = x + 2
```

解释器在运行时会做这些事(大致):

1. 读到"取 x"——去查 x 这个变量存在哪,把它的值取出来。
2. 读到"加 2"——确定要做加法,把 2 也准备好。
3. 判断 x 和 2 的类型(是整数?浮点?字符串?不同类型加法不一样),选对加法方式。
4. 执行加法,得到结果。
5. 读到"赋给 y"——把结果存到 y。
6. 下一条……

- 优点:灵活,不用提前编译,改了立刻能跑;而且因为每一步都现场判断类型,动态语言天然适合。
- 代价:**每执行一条,都要现场"理解"它一遍**。这个"理解"的过程本身就要花时间。

**官方 Lua 就是纯解释器。** 它很灵活、很正确、实现也简单,但它慢——慢就慢在"每一条都要现场理解"。

---

## §3 解释器到底慢在哪

我们来把这件事看清楚:解释器慢,到底慢在哪一步?

很多人以为是"算术慢"。不是。CPU 做一次整数加法,只要不到一纳秒。真正慢的,是**解释器为了搞清楚该做什么算术,而做的那些额外工作**。

还是那句 `y = x + 2`。CPU 如果直接执行,其实只需要三条机器指令:取 x、加 2、存 y,可能只要 3 纳秒。

但解释器不能直接这么干,因为它在运行时**才知道** x 是整数还是别的。所以它每遇到这条指令,都要走一遍上面那 6 步:取指、译码(这条指令是什么意思?)、查表、判断类型、分发到对应的处理逻辑。这些"理解指令"的工作,可能花掉几十甚至上百纳秒——是真正计算的好几十倍。

**也就是说:解释器的时间,绝大多数花在"搞清楚要做什么"上,而不是"真正去做"上。**

而且更要命的是:这种开销是**重复**的。如果这段代码在一个跑 100 万次的循环里,那这 100 万次,每一次解释器都要重新"理解"一遍同样的指令——明明第 1 次已经理解过了,第 2 次到第 100 万次的理解结果完全一样,却还要一遍遍重来。

这就是解释器慢的根源:**反复地、一模一样地理解同一段代码。**

讲到这里,一个自然的想法就冒出来了。

---

## §4 那把 Lua 也编译了,不就行了?

既然慢在"反复理解",那像 C 那样,提前把代码翻译成机器码,不就不用反复理解了吗?运行时 CPU 直接跑机器码,3 纳秒搞定,快几十倍。

想法很美。但一旦动手,立刻撞上一堵墙。

来看这段代码:

```lua
function add(a, b)
  return a + b
end
```

现在问一个致命的问题:`a + b` 这条加法,该翻译成**哪种**机器指令?

- 如果 a、b 是整数,该用 CPU 的**整数加法指令**。
- 如果 a、b 是浮点数,该用**浮点加法指令**(和整数加法是完全不同的指令)。
- 如果 a、b 是字符串,`+` 根本不是加法,得报错(或拼接)。

机器码必须**确定**——你生成"整数加法"的机器码,CPU 就老老实实做整数加法,它不会自己判断。所以编译时,你必须知道 a、b 是什么类型,才能生成对的机器指令。

C 语言没问题,因为你写 `int add(int a, int b)`,编译器一眼就知道是整数。

**但 Lua 不行。** Lua 是动态类型,变量没有类型声明:

```lua
add(1, 2)        -- 这次 a,b 是整数
add(1.5, 2.5)    -- 这次 a,b 是浮点
add("x", "y")    -- 这次 a,b 是字符串
```

同一个 `add` 函数,每次调用,参数可能是不同类型。在程序**真正运行之前**,谁也不知道 `a + b` 到底是什么类型的加法。

**这就是动态语言无法像 C 那样提前编译的根本障碍:编译时类型未知,生成不出确定的机器码。**

你可能会想:那生成"什么类型都能处理"的通用机器码不就行了?——可以,但那种机器码里要塞进"判断类型→分支处理"的逻辑,结果又退化成了类似解释器的东西,没快多少。

所以,动态语言的传统选择就是:干脆不编译,老老实实解释,用灵活换速度。官方 Lua 就是这么做的。

那 LuaJIT 凭什么能把动态的 Lua 编译成机器码,还快几十倍?它一定用了什么聪明办法。这个办法,就是 JIT。

---

## §5 JIT 的核心思想:等运行起来再编译

LuaJIT 的关键洞察是一句话:

> **既然编译时不知道类型,那就不要在编译时编译,等程序运行起来、看到真实类型了,再编译。**

这就叫 JIT——Just-In-Time,即时编译,在程序**运行的过程中**编译。

它的逻辑是这样的:程序一开始,还是用解释器跑(慢但正确)。但解释器一边跑,一边在**观察**:这段代码跑了多少次?它的变量实际都是什么类型?

等观察到某个循环跑了很多次(热点),而且观察到"这个循环里的 x,每次都是整数"——好,这时候类型已知了,**现在可以编译了**。LuaJIT 就把这个循环编译成"假设 x 是整数"的整数加法机器码。从此以后,这个循环就跑机器码,飞快。

这就绕开了 §4 的障碍:**不是编译时编译(那时类型未知),而是运行时、看到真实类型之后再编译(那时类型已知)。** 动态语言从此也能享受机器码的速度。

听起来很完美。但这个想法一落地,立刻冒出两个大问题。LuaJIT 的全部精妙设计,都是在回答这两个问题。

---

## §6 第一个问题:什么时候编译?(热点)

第一个问题:程序里那么多代码,难道每一段都要编译吗?

不行。因为**编译本身是有开销的**。要把一段代码分析、翻译成机器码,比直接解释它慢得多(可能慢几百倍)。如果你编译了一段只跑两次的冷代码,那编译花的时间,远远超过它省下来的时间——净亏。

那该编译什么?

程序员的经验告诉我们一个普遍规律:**程序的大部分时间,花在很少一部分代码上。** 一个程序可能有几万行代码,但真正吃 CPU 的,往往就是那么几个循环、那么几个频繁调用的函数——它们可能跑成千上万次,而其余代码只跑一两次。

这个规律有个名字,叫"热点"。**只有热的代码,才值得编译。** 冷代码编译了是浪费,让它继续解释就好。

那怎么知道哪段代码热?最直接的办法:**数它跑了多少次。** 给每段可能成为热点的代码(循环的回边、函数调用)配一个计数器,每跑一次就加一(或减一),等计数器到达一个阈值,就认定它够热了,触发编译。

LuaJIT 用的就是这个办法,叫 hotcount。在源码里(`lj_dispatch.h:74`):

```c
#define HOTCOUNT_SIZE    64
#define HOTCOUNT_LOOP    2
#define HOTCOUNT_CALL    1
...
HotCount hotcount[HOTCOUNT_SIZE];  /* Hot counters. */
...
#define hotcount_get(gg, pc)  (gg)->hotcount[(u32ptr(pc)>>2) & (HOTCOUNT_SIZE-1)]
```

先不用看懂每一行,抓住三件事:

1. 它有一个计数器数组 `hotcount`,共 64 个槽。
2. 每条字节码指令(用 `pc` 表示位置)通过 `pc>>2` 算出一个哈希值,落到这 64 个槽里的某一个。哪条指令热,就去查它对应的那个槽。
3. `HOTCOUNT_LOOP=2`、`HOTCOUNT_CALL=1` 是每次自减的**步长**(循环回边减 2、函数调用减 1),真正决定"何时触发"的是参数 `hotloop`(默认 56)——计数器从 `hotloop×步长` 一路递减到 0,才真正触发录制。(P1-03 会把这个机制讲透,这里先建立印象:它就是"数热点"的计数器。)

这个机制回答了"什么时候编译":**数据驱动,谁跑得多就编译谁。** 不是人工标注,是运行时统计。这也保证了 LuaJIT 只把编译开销花在真正值得的地方——这就是主线里"省"那一股张力。

第一个问题解决了。但第二个问题,才是真正的硬骨头。

## §7 第二个问题:编译时的假设,运行时失效了怎么办?

回忆 §5:LuaJIT 观察到"这个循环里的 x 每次都是整数",于是编译成"假设 x 是整数"的整数加法机器码。

现在请你想一个可怕的情况:**如果某一次,x 不是整数了呢?**

Lua 是动态语言,程序完全可能在第 100 万次循环时,突然让 x 变成字符串(也许是某个分支、某次外部输入)。但编译好的机器码,是按"x 一定是整数"生成的——它只会做整数加法。如果这时 x 真是字符串,机器码就会**算错**,甚至把内存搞坏、程序崩溃。

这才是 JIT 真正的核心难题,比"什么时候编译"难得多:

> **编译时基于观察做了假设,但动态语言的假设随时可能被打破。一旦打破,机器码就会出错。**

那怎么办?有两条路。

**第一条路:把所有可能都考虑进去。**

编译时不只假设"x 是整数",而是生成"如果 x 是整数走这条机器码、如果是浮点走那条、如果是字符串走第三条"的带分支的机器码。这样无论 x 是什么,都有对应的处理。

JVM、V8 基本走的是这条路(它们以整个函数为单位编译,叫做 method JIT)。好处是全面、稳;代价是机器码复杂、编译慢、要做大量类型推断。

**第二条路:大胆假设,但每次都检查。**

编译时就大胆假设"x 一定是整数",生成最快的整数加法机器码。但是——在机器码里**额外插一条检查**:"如果 x 不是整数,就别往下走了,跳出去"。

只要 x 是整数(绝大多数时候),检查通过,机器码全速运行,快。万一某次 x 不是整数,检查发现,立刻跳出去——不做整数加法(那会错),而是退回去,用别的办法处理。

LuaJIT 走的是这第二条路。这条插在机器码里的检查,有个专门的名字:**guard**(守卫)。

我们来把 guard 看清楚。它本质上就是机器码里的一句"如果……就跳走"。

假设编译器基于观察,认定变量 x 是整数。那它在生成的机器码里,会做两件事:

1. 生成整数加法指令(乐观的部分,负责快)。
2. 在加法之前(或之后),插入一条检查指令:检查 x 的类型标记,如果不是整数,就跳到一个叫"退出点"的地方(悲观的部分,负责安全)。

```text
(机器码,大致示意)
  检查 x 的类型标记   ← 这就是 guard
  如果不是整数,跳到 EXIT
  把 x 当整数,做加法  ← 乐观假设成立时执行
  ...
EXIT:
  (退出处理)
```

guard 这个设计,把"乐观假设"和"安全保障"分开了:**机器码本体尽情乐观(所以快),guard 负责兜底(所以安全)。** 只要 guard 没被触发,就说明所有假设都成立,结果一定对。

好,假设不成立时,guard 抓住了。但"跳到 EXIT"之后呢?程序不能停啊。这就引出下一个机制。

---

## §8 假设失败了怎么办:snapshot 与 side exit

guard 触发后,程序跳到退出点(EXIT)。现在的处境是:机器码不能继续了(它是按假设生成的,假设已经破产),但程序必须接着跑出正确结果。

怎么办?**退回解释器。** 解释器是完整的、正确的 Lua 实现,它什么类型都能处理。只要能把当前的状态(各个变量的值)交给解释器,解释器就能从这里接着跑,给出正确结果。

但这又有一个麻烦:机器码运行的时候,数据都在 **CPU 寄存器**里(寄存器是 CPU 内部极快的存储);而解释器期望数据在它自己的**值栈**上(一段内存)。两边的"数据存放位置"对不上。直接把控制权丢给解释器,解释器会一脸懵:我要的值在哪?

所以,退出的时候,需要**把机器码这一刻的状态(寄存器里的值),翻译、搬运成解释器期望的样子(栈上的值)**,然后才能交给解释器继续。

这个"状态翻译表",在 LuaJIT 里叫 **snapshot**(快照)。

为什么叫快照?因为它就像给机器码运行到这一刻的状态**拍了张照片**:记录下"这个寄存器对应栈上的第几个槽""那个值是什么"。等 guard 触发要退出时,就照着这张照片,把状态恢复出来,摆成解释器能认的样子。

而且,每条机器码里可能有多个 guard(每个假设一个),每个 guard 退出时需要恢复的状态不同。所以一个 trace(一段编译出来的机器码)会带**一组** snapshot,每个 guard 对应一个。在源码里,trace 的结构 `GCtrace` 就有一个字段 `snap` 存这组快照(`lj_jit.h:250`,稍后细看)。

退出、靠 snapshot 恢复、退回解释器继续——这个完整的动作,叫 **side exit**(侧出口)。

现在把 guard 和 side exit 合起来看,整个图景就清楚了:

- **大部分时候**:所有假设成立,guard 都不触发,机器码全速跑到底。**快。**
- **偶尔**:某个假设破了,guard 触发,side exit 靠 snapshot 恢复状态,退回解释器。**正确,虽然慢一下。**

## §9 关键不变式:解释器保底,机器码加速

把前面几节合起来,我们能得到 LuaJIT 最重要的一条性质。它值得单独一节,因为它是整个设计"为什么不会算错"的根基。

这条性质是:

> **机器码产生的结果,要么和解释器完全一样(假设成立),要么退回解释器(假设失败)。它永远不会比解释器更错。**

为什么能保证这一点?因为有两道保险:

1. **每一个乐观假设,都有一个 guard 兜底。** 假设再多再大胆,只要它们都配了 guard,就没有一个假设能"偷偷地"失效而不被发现。任何一个假设破了,对应的 guard 一定触发。
2. **guard 一触发,就退回解释器。** 而解释器本身是一个完整、正确的 Lua 实现——它不做任何假设,每一步都按真实类型处理,它给出的结果永远是**对的**。

所以,从程序员的视角看:LuaJIT 跑出来的结果,和官方 Lua 跑出来的结果,**一模一样**(语义完全一致)。区别只在于:LuaJIT 在那些假设成立的热点上,用机器码飞奔,快了几十倍;在假设偶尔失效的地方,退回解释器,慢一下,但结果照样对。

**正确性由解释器保底,速度由机器码提供。** 两者各司其职,这就是"安全"和"快"如何共存——也是主线"把动态执行**安全**变成机器码"里"安全"两个字的真正含义。

讲到这里,JIT 的核心难题已经被解决了。剩下的,是 LuaJIT 把这些想法落地的**具体做法**,以及它为什么做得这么快、这么省。从下一节开始,我们进入 LuaJIT 的具体设计。

## §10 trace:为什么是"一条路径",而不是"整个方法"

还有最后一个大设计选择要讲清楚。它解释了 LuaJIT 和 JVM/V8 的根本区别。

编译的时候,以**多大范围**为单位编译?

JVM 和 V8 的选择是:以**整个函数(方法)**为单位。一个函数里可能有 if/else、循环、各种分支,它们把整个方法的代码结构看成一个图(控制流图),全部翻译成 IR(中间表示),再做优化、生成机器码。这叫 **method JIT**。

好处是覆盖全面(整个函数都优化了);代价是编译量大(整个函数,哪怕里面大部分分支根本没跑到)、要做复杂的类型推断、编译慢。

LuaJIT 走了一条完全不同的路。它的观察是:

> **热点代码在运行时,实际只走其中一条路径。**

什么意思?看这个循环:

```lua
for i = 1, 1000000 do
  if i < 900000 then
    x = x + 1       -- 这条路,跑了 90 万次
  else
    x = x - 1       -- 这条路,只跑了 10 万次
  end
end
```

这个循环里有 if/else 两个分支。但运行时,绝大多数时候(90 万次)走的是 `x = x + 1` 这条路。那条 `x = x - 1` 是冷分支。

method JIT 会把整个循环(两个分支都)编译。但 LuaJIT 想:**既然 90% 的时候只走 `x = x + 1` 这一条线性路径,那我干嘛要编译整个循环?我只把这条实际在走的路径录下来、编译了不就行了吗?**

这条**实际在运行的、线性的、没有分叉的路径**,就叫一个 **trace**(轨迹)。

为什么这么做好?三个理由,正好对应主线的三股张力:

1. **省**:trace 是一条线性路径,比整个函数小得多,编译快、编译开销小。
2. **快**:因为只录一条路,假设可以做得非常乐观非常细(这条路看到的全是整数,就假设全是整数),生成的机器码极简极快。
3. **安全依然有保障**:分支怎么处理?用 guard。`if i < 900000` 这个判断,在 trace 里就变成一个 guard:条件成立就继续(走录的那条路),不成立就 side exit 退回解释器(走另一条没编译的路)。

所以 trace 不是"丢掉分支",而是"只编译主流分支,其他分支用 guard 兜底、失败时退回解释器"。这把 method JIT 要处理的复杂分支,简化成了"一条主线 + 若干 guard"。

**这就是 trace compiler(跟踪式编译器)的核心:以运行时实际走过的一条线性路径为编译单位,而不是整个方法。** LuaJIT 是这条路线最著名的实现。

## §11 一个 trace 长什么样:GCtrace 的三件套

现在我们终于可以看源码了。一个 trace,在 LuaJIT 里是 `GCtrace` 结构(`lj_jit.h:250`):

```c
typedef struct GCtrace {
  GCHeader;
  uint16_t nsnap;     /* Number of snapshots. */
  IRRef nins;         /* Next IR instruction. Biased with REF_BIAS. */
  GCRef gclist;
  IRIns *ir;          /* IR instructions/constants. Biased with REF_BIAS. */
  IRRef nk;           /* Lowest IR constant. Biased with REF_BIAS. */
  uint32_t nsnapmap;  /* Number of snapshot map elements. */
  SnapShot *snap;     /* Snapshot array. */
  SnapEntry *snapmap; /* Snapshot map. */
  ...
  MSize szmcode;      /* Size of machine code. */
  MCode *mcode;       /* Start of machine code. */
  MSize mcloop;       /* Offset of loop start in machine code. */
  uint16_t nchild;    /* Number of child traces (root trace only). */
  TraceNo1 traceno;   /* Trace number. */
  TraceNo1 link;      /* Linked trace (or self for loops). */
  TraceNo1 root;      /* Root trace of side trace (or 0 for root traces). */
  TraceNo1 nextroot;  /* Next root trace for same prototype. */
  TraceNo1 nextside;  /* Next side trace of same root trace. */
  ...
} GCtrace;
```

不用怕这个结构体长。我们已经把概念都讲透了,现在只是用源码**印证**。它装的核心是三样东西,正好对应我们前面讲的:

**第一样:`ir`——这条路径的中间表示(IR)**

`ir` 是一个数组,存的是 IR 指令(`IRIns`)。IR 就是 trace 的"半成品":把那条线性路径,翻译成一种比机器码高级、比 Lua 代码低级的中间形式(后面 P2-07 会讲它是一种叫 SSA 的形式)。优化和代码生成都在 IR 上做。

为什么需要 IR 这个中间层,而不是直接 Lua→机器码?因为直接翻译太难优化。先翻成 IR,在 IR 上做常量折叠、消除冗余等优化(整个 P3 篇),再把优化后的 IR 翻成机器码。多一层抽象,换来强大的优化能力。这是所有现代编译器的标准做法。

**第二样:`snap` / `snapmap`——一组快照**

这就是 §8 讲的 snapshot。一个数组,每个 guard 对应一个,记录"机器码跑到这个 guard 时,状态该怎么恢复成解释器的样子"。guard 一触发,照着快照恢复,退回解释器。

**第三样:`mcode`——最终生成的机器码**

`mcode` 指向一段真正的、可执行的机器码(存在一块专门申请的可执行内存里)。`szmcode` 是它的大小,`mcloop` 标记循环的起点(为了把循环首尾接起来)。

**一句话:`GCtrace` = IR(半成品)+ 快照(退路)+ 机器码(成品)。** 它把一条 trace 从录好到能跑所需的全部信息,装在一个结构里。

剩下的字段(`traceno`/`link`/`root`/`nextroot`/`nextside`)是管理"trace 之间关系"的,下一节讲。

## §12 把整个过程串起来:一个 for 循环的完整旅程

概念都讲过了,源码也看了一眼。现在用一个最小例子,把整条流水线走一遍,让你看见它**作为一个整体**怎么运作。

例子:

```lua
local x = 0
for i = 1, 1000000 do
  x = x + i
end
```

**第 1 阶段:解释器先跑(慢,但正确)**

程序启动,`x = x + i` 这个循环体先由解释器执行。解释器是完整的 Lua 实现,每一步都判断类型、现场执行,结果一定对。只是慢——每条指令都要"理解"一遍。

**第 2 阶段:热点被发现(hotcount 计数)**

循环每回边一次,LuaJIT 就把这条循环回边对应的 hotcount 计数器减一(或加一)。跑到阈值(比如 `HOTCOUNT_LOOP`)时,LuaJIT 认定:这段循环够热了,值得编译。

**第 3 阶段:录制 trace(把路径录成 IR)**

LuaJIT 进入"录制"模式。它让解释器再跑一遍这个循环体,但这一遍,解释器每执行一条字节码,LuaJIT 就**同时**把它翻译成一条 IR 指令,追加到正在构建的 trace 的 `ir` 数组里。

录制过程中,它观察到:`x` 和 `i` 这次都是整数。于是它**乐观地**假设"这条路径上 x、i 永远是整数",并:
- 把 `x + i` 翻译成"整数加法"的 IR。
- 在前面插一个 guard:"检查 x 是整数""检查 i 是整数"(对应 IR 里的检查指令)。

遇到分支(这里循环没有 if,但概念上)或循环回边,录制就结束——一条**线性**路径录完了。

**第 4 阶段:优化 IR(P3 篇)**

录好的 IR 送去优化:常量折叠(`x + 0` 这种消掉)、循环不变量外提(`i` 不变的部分提到循环外)等等。优化让机器码更精简。

**第 5 阶段:生成机器码(P4 篇)**

优化后的 IR,经过**寄存器分配**(决定哪些值放哪个 CPU 寄存器)和**汇编生成**,变成真正的机器码,存进 `mcode`。同时,每个 guard 对应的 snapshot 也生成好,存进 `snap`。

**第 6 阶段:安装,机器码接管(快)**

这个 trace(连同它的机器码)被"安装"好。从此,解释器再跑到这个循环回边时,不再解释执行,而是**直接跳进 `mcode` 去跑机器码**。机器码假设 x、i 是整数,做整数加法,飞快。剩下 99 万多次循环,几乎都在机器码里飞奔。

**第 7 阶段:万一假设失败(正确,退回)**

假设某次 x 真的不是整数了(这个例子里不会,但换个有外部输入的程序就可能)。机器码里的 guard"检查 x 是整数"发现不对,触发 side exit:照着对应的 snapshot 恢复状态,退回解释器。解释器按真实类型处理,结果照样对。这次慢一点,但不出错。

这就是 JIT 的完整一生。你看,**没有任何魔法**:先解释,发现热点,录一条路径,假设+检查地编译成机器码,失败就退回。每一步都有明确的理由,都是为了应对某一股张力。

## §13 trace 之间织成网:TraceLink 与 side trace

最后还有一个细节,因为它解释了 LuaJIT 怎么把多条 trace 接起来,跑得更久。

一个 trace 录完、机器码生成后,机器码跑到底会怎样?有几种结局:

- **回到自己**(最常见):这是个循环 trace,跑完一次循环体,直接跳回开头再跑。机器码首尾相接,形成闭环,可以无限跑下去。
- **链到另一个 trace**:这条 trace 跑完,正好接上另一条已编译的 trace,直接跳过去,不用退回解释器。
- **退回解释器**:这条 trace 跑完没法接别的,老老实实回解释器。

LuaJIT 把这些结局分成 9 种,叫 TraceLink(`lj_jit.h:237`):

```c
typedef enum {
  LJ_TRLINK_NONE,      /* Incomplete trace. No link, yet. */
  LJ_TRLINK_ROOT,      /* Link to other root trace. */
  LJ_TRLINK_LOOP,      /* Loop to same trace. */
  LJ_TRLINK_TAILREC,   /* Tail-recursion. */
  LJ_TRLINK_UPREC,     /* Up-recursion. */
  LJ_TRLINK_DOWNREC,   /* Down-recursion. */
  LJ_TRLINK_INTERP,    /* Fallback to interpreter. */
  LJ_TRLINK_RETURN,    /* Return to interpreter. */
  LJ_TRLINK_STITCH     /* Trace stitching. */
} TraceLink;
```

不用记全,记住最常见的 `LOOP`(循环回自己)和 `STITCH`(把几条 trace 缝在一起)就够了。

还有一个更重要的概念:**side trace**(侧轨迹)。

回想 §8:某个 guard 经常失败(比如那个 `if i < 900000`,有 10 万次走另一条路)。每次失败都退回解释器,太亏了——那条"另一条路"其实也跑了 10 万次,也挺热的。

那能不能把那条失败后走的路径,也录成一条新的 trace?能。LuaJIT 会从一个经常失败的 guard 退出点,**重新录制一条 trace**,这就是 side trace。它和原来的 root trace(主轨迹)组成一棵树(`GCtrace` 里的 `root`/`nextroot`/`nextside` 字段就在管这棵树)。

这样,程序跑得越久,trace 树长得越完整,越来越多的路径被编译成机器码,退回解释器的次数越来越少,整体越来越快。**这是 JIT"越跑越快"的根源。**

## §14 ★对照:官方 Lua 与 JVM/V8

讲完了 LuaJIT 的全貌,我们把它和两个对象放在一起对照,你会看得更清楚。

**对照一:官方 Lua(切"有没有 JIT")**

官方 Lua 是纯解释器。它没有 §5 之后的任何东西——没有热点检测、没有 trace、没有机器码。每条指令永远在现场解释,循环跑 100 万次就解释 100 万次。

所以官方 Lua 慢,但慢得"诚实":实现简单、永远正确、没有编译开销。LuaJIT 把这一切都加上了:解释器(还是要有,负责保底和冷代码)+ 热点检测 + trace 编译。LuaJIT 快几十倍,代价是实现了整整一套编译器。

**对照二:JVM / V8(切"trace vs method")**

JVM 和 V8 也有 JIT,但它们是 **method JIT**:以整个函数为单位编译,做类型推断,生成覆盖所有分支的机器码。假设相对保守、编译较慢、用 deoptimization(去优化)处理假设失效。

LuaJIT 是 **trace JIT**:只录一条线性热路径,假设大胆乐观、编译飞快、用 side exit + snapshot 处理失效。

| 维度 | LuaJIT(trace JIT) | 官方 Lua | JVM / V8(method JIT) |
|---|---|---|---|
| 编译单位 | 一条线性热路径(trace) | 不编译 | 整个函数(method) |
| 编译时机 | 运行时,热点触发 | — | 运行时,方法热点触发 |
| 类型处理 | 乐观假设 + guard | 运行时动态判 | 类型推断/profile |
| 假设失效 | side exit + snapshot 回解释器 | — | deoptimization 回解释器 |
| 编译开销 | 低(trace 短) | 无 | 较高(整函数) |

trace vs method 是 JIT 的两大流派。LuaJIT 选 trace,是因为它对动态语言的热点更聚焦、编译更轻、假设更激进——但也因此需要更精巧的退出处理(snapshot、side trace 树)。这些正是后面 P2–P5 要逐行讲清的东西。

## §15 全书怎么读 + 回扣主线

到这里,你应该已经抓住了 JIT 的全貌。我们把全书的结构再梳理一遍,每一篇都对应你刚学到的一个环节:

- **P1 解释器与热点**:讲解释器本身(它极快,手写汇编)和热点检测(hotcount)。这是 JIT 的基座——代码先在这里跑,热点也在这里被发现。
- **P2 trace 录制与 IR**:讲怎么把一条热路径录成 IR(`lj_record`),以及怎么把动态类型"窄化"成具体类型(`lj_opt_narrow`)。
- **P3 优化 pass**:讲在 IR 上做哪些优化(常量折叠、内存优化、分配消除等)。
- **P4 后端**:讲怎么把优化后的 IR 变成机器码(线性扫描寄存器分配、汇编生成、可执行内存)。
- **P5 运行时**:讲机器码怎么安全运行(guard、snapshot 恢复、side trace 长出来、trace 之间链接)。
- **P6 FFI 与 GC**:讲 LuaJIT 连调 C 函数都能 JIT,以及 GC 怎么和 JIT 协作。

建议读法:P0-01 到 P1 先读,把"解释器 + 热点"吃透,这是后面一切的前提。之后 P2(录制)和 P4(后端)是流水线的两端,可以对照读;P3(优化)和 P5(运行时)相对独立,可按兴趣选读。

每一篇,我们都会回到这一章讲的那条主线:

> **用乐观假设 + 运行时检查(guard)+ 失败可回退(side exit/snapshot),把一段动态执行的脚本,安全地变成机器码。**

JIT 没有魔法。它的快,来自一个朴素而精巧的权衡:在能乐观的地方尽情乐观(生成最快的机器码),在可能出错的地方老老实实检查(guard),实在不行就退回那个永远正确的解释器。LuaJIT 把这个权衡做到了极致——这就是它比纯解释器快几十倍的根本原因,也是这本书要带你一行行看清的东西。
