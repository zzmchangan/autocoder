## 引子：从 IR 到机器码，卡在寄存器上

前面三章(P2 录制、P3 优化)讲完之后，我们手里有了一段优化好的 IR。IR 是 LuaJIT 内部的中间表示：一条 trace 被录成一串 SSA 形式的 IR 指令，每条指令算出一个"值",用它的引用号(ref)来指代。优化 pass 已经把能折叠的常量折叠了、能消除的死代码消除了、能下沉的分配下沉了。剩下的事情很直接：把这段 IR 翻译成 CPU 能跑的机器码。

这一步听起来像是"一对一翻译"——每条 IR 指令对应一段机器指令，有什么难的?真正动手翻译，立刻撞上一堵墙。这堵墙，就是这一章要拆掉的。

IR 里有无数个值。前面讲过,SSA 的特性是"每个值只定义一次",所以 IR 里每算出一个新结果，就是一个新的 ref。一个稍微长一点的 trace,几百个 ref 是常事。这几百个值，在 IR 里都活得好好的——ref 只是个编号，想要多少有多少。

但 CPU 不认 ref。CPU 认的是**寄存器**——它内部那屈指可数的、极快的存储单元。机器码里所有的运算，操作的都是寄存器:`add eax, ecx`、`mov rdi, rax`、`mulsd xmm0, xmm1`。寄存器是机器码唯一能直接算术运算的对象(内存也可以参与，但慢，而且很多指令不支持)。

问题来了:**x86-64 的通用寄存器，满打满算只有 16 个**(EAX/ECX/EDX/EBX/ESP/EBP/ESI/EDI/R8D–R15D),其中 ESP 是栈指针不能动，某些还要留给 ABI 用，真正能自由分配的更少。浮点寄存器(XMM)也是 16 个。而 IR 里有几百个值。

**怎么把几百个虚拟的 ref,塞进十几个物理寄存器里?** 这就是寄存器分配(register allocation)。它是后端代码生成的第一道、也是最关键的一道工序。这一章，我们讲清楚 LuaJIT 怎么做这件事。

LuaJIT 用的是**线性扫描**(linear scan)寄存器分配，不是教科书里最经典的**图着色**(graph coloring)。为什么?这一章会回答。但在回答"为什么"之前，我们要先把问题本身看清楚——寄存器分配到底是个什么样的问题，为什么难，不分配会怎样。

## §1 第一性原理：虚拟值无穷，物理寄存器有限

### 1.1 寄存器是什么，为什么机器码离不了它

先把基础事实钉死，不假设读者都清楚。

CPU 内部有一组寄存器。以 x86-64 为例，通用寄存器(general-purpose register, GPR)有 16 个，每个 64 位(也叫 64 位模式下 RAX/RBX...,32 位模式下 EAX/EBX...)。另外还有一组浮点/SIMD 寄存器 XMM0–XMM15,共 16 个，每个 128 位。

寄存器是 CPU 内部**最快**的存储。访问一个寄存器，基本不花时间(和一条指令的执行同步完成)。相比之下，访问内存(哪怕是 L1 缓存)要慢几个数量级。所以，任何想在 CPU 上高速运行的代码，都必须尽量把正在用的数据放在寄存器里，运算直接在寄存器之间进行。

机器码的运算指令，几乎全是"寄存器之间"的。比如:

```asm
add eax, ecx      ; eax = eax + ecx,两个都是寄存器
imul rdi, rsi      ; rdi = rdi * rsi
movsd xmm0, xmm1  ; xmm0 = xmm1(浮点拷贝)
```

内存当然也能参与(比如 `add eax, [rsp+8]`),但一来慢，二来不是所有指令形态都支持。所以编译器生成的机器码，主导形态是：把数据 load 进寄存器，在寄存器里反复算，最后再 store 回内存。

**结论：机器码的世界，数据主要住在寄存器里。寄存器就是机器码的"工作台"。**

### 1.2 IR 的世界:ref 无穷，值自由

现在看 IR 的世界。前面 P2-07 讲过,LuaJIT 的 IR 是 SSA(静态单赋值)形式。SSA 的核心规矩是:**每个值只被定义一次，之后永远不变**。每定义一个新值，就给它一个新的引用号(ref)。

拿一段最简单的循环举例:

```lua
local x = 0
for i = 1, 100 do
  x = x + i
end
```

录成 trace 之后，循环体里 `x = x + i` 这条加法，在 IR 里大致长这样(简化，真实 IR 有更多细节):

```
%012 = ADD %008, %009     ; x_prev + i,得到新的 x
```

`%012` 就是这次加法算出的那个新值。注意 SSA 的规矩：这个值**只此一次**。下一次循环再算 `x + i`,会产生另一个 ref `%xxx`,和 `%012` 是不同的值(SSA 里哪怕值相等也是不同的定义点)。

这意味着什么?**一个长 trace,ref 会越来越多。** 几百条 IR 指令，就有几百个 ref。每个 ref 代表一个"值",这个值在 IR 的世界里是抽象的、自由的——它就是一个编号，不占任何具体的物理资源(除了 IR 数组里那一格)。

IR 的世界里有几百个自由的值。机器码的世界里只有十几个寄存器。**要把 IR 翻译成机器码，就必须给每个"需要被放进寄存器的值",指定一个具体的物理寄存器。** 这就是寄存器分配。

### 1.3 为什么不是每个值都要寄存器

先澄清一个容易混淆的点：并不是 IR 里**每一个** ref 都需要一个寄存器。

有些值是**常量**。IR 里有一类指令叫 `KINT`、`KNUM`、`KGC` 等，就是常量。常量不需要长期占一个寄存器——用到的时候，从内存(或立即数)load 进来就行。

有些值是**一次性的中间结果**。比如 `a + b + c`,IR 是 `%t1 = ADD a, b; %t2 = ADD %t1, c`。`%t1` 算出来马上就用，用完就再也不碰了。这种值，生命周期极短，占一个寄存器算完就行。

有些值是**长寿命的**。比如循环里那个累加器 `x`,从循环开始到结束，每次循环都要用。这种值最好一直待在同一个寄存器里，别老换地方。

所以寄存器分配不是"无脑给每个 ref 配一个寄存器",而是:**在有限的物理寄存器里，安排好每个活着的值，让它们各得其所；安排不下的，想办法存到别处。** 关键词是"活着的"——这就引出下一个概念：活跃区间。

### 1.4 活跃区间：一个值从生到死

寄存器分配的核心数据结构，是**活跃区间**(live range)。

一个值的活跃区间，是它**从被定义，到最后一次被使用，这一整段范围**。在这段范围内，这个值"活着"——后续还要用，必须保留着(在寄存器里，或者在内存里随时能取回来)。超出活跃区间，这个值就"死"了，寄存器可以拿去给别人用。

举个具体的例子。假设有这样一个简化的 IR 序列(ref 用数字表示，越小越早):

```
%10 = ADD  %1, %2     ; 定义 %10
%11 = MUL  %10, %3    ; 用 %10,定义 %11
%12 = ADD  %11, %4    ; 用 %11,定义 %12
...                    ; %10 再也没被用过
```

- `%10` 的活跃区间：从第 1 条(定义)到第 2 条(最后一次使用)。过了第 2 条,`%10` 死了。
- `%11` 的活跃区间：从第 2 条到第 3 条。
- `%12` 的活跃区间：从第 3 条往后，直到它最后一次被使用。

为什么要关心活跃区间?因为**两个值，如果它们的活跃区间不重叠，就可以共用同一个物理寄存器**。

比如 `%10` 和 `%12`:`%10` 在第 2 条之后就死了,`%12` 在第 3 条才出生。它俩的活跃区间不重叠(一个死了另一个才生),所以可以共用一个寄存器——第 1、2 条用这个寄存器装 `%10`,第 3 条开始用同一个寄存器装 `%12`。

这就是寄存器分配的本质:**找出所有值的活跃区间，把不重叠的区间尽量塞进同一个物理寄存器，让有限的寄存器被复用到极致。**

### 1.5 分配不下怎么办:spill(溢出)

但不管你怎么复用，物理寄存器就那么几个。如果一个时刻，同时活着的值(活跃区间相互重叠的值)超过了物理寄存器的数量，就**分配不下**了。

比如 x86-64 有 14 个可分配通用寄存器(扣掉 ESP,加上 ABI 约束实际更少)。如果某个时刻，有 20 个值同时活着，那就有 6 个值无处可去。

怎么办?**把它们存到内存里。** 内存(具体说是栈)很大，想要多少有多少。把分配不下的值，写到栈上的某个槽(spill slot),这个动作叫 **spill**(溢出)。等以后要用这个值的时候，再从栈上 load 回寄存器。

spill 是有代价的：每次 spill 和 reload 都是一次内存访问(虽然栈通常在缓存里，但还是比寄存器慢)。所以寄存器分配的质量，很大程度上取决于"**少 spill**"——把那些 spill 代价最高的值(比如循环里反复用的累加器)留在寄存器里，把那些 spill 代价低的(比如用一次就死的常量)优先 spill。

到这里，寄存器分配这个问题，被我们完整地刻画出来了:

> **给定一组值(每个有一个活跃区间)和有限的物理寄存器，给每个值分配一个物理寄存器(让不重叠的区间复用同一寄存器),实在分配不下的 spill 到栈上，目标是让程序跑得尽量快(少 spill、少 load/store)。**

这是一个经典的组合优化问题。怎么解它?有两类主流算法。下一节讲。

## §2 两种算法：图着色 vs 线性扫描

寄存器分配的算法，历史上有两大流派:**图着色**(graph coloring)和**线性扫描**(linear scan)。LuaJIT 选了后者。要理解为什么选后者，得先把两者都看清楚。

### 2.1 图着色：经典但慢

图着色是最经典的寄存器分配算法，从 1981 年 Chaitin 的论文开始，被 GCC、早期 JVM(C1/C2 的部分)、很多教学编译器采用。

它的思路是这样的:

**第一步：建干扰图(interference graph)。** 把每个值当成图里的一个节点。如果两个值的活跃区间**重叠**(即某个时刻同时活着，不能共用寄存器),就在它们之间连一条边。这条边叫"干扰边"。建完图，所有"互相冲突"的值对，都由边连起来了。

**第二步：给图着色。** "着色"就是给每个节点涂一个颜色，要求**有边相连的两个节点，颜色必须不同**。每种颜色代表一个物理寄存器。如果只有 K 个物理寄存器，那就是"用 K 种颜色给图着色"。

着色问题有个尴尬的性质：它是 **NP-hard** 的——没有已知的多项式时间算法能保证找到最优解(用 K 种颜色能不能着出来，本身就可判定为 NP 完全)。所以实际编译器用的都是**启发式**(heuristic):贪心地挑一个节点，给它一个不冲突的颜色，失败了就回溯或者 spill,近似着色。

图着色的优点是**质量高**。因为它精确地建模了"谁和谁冲突",着色出来的方案 spill 往往比较少。尤其在做了一些优化(比如 coalescing,把不必要的 mov 合并掉)之后，生成的代码质量很好。

缺点是**慢**。建干扰图要 O(n²)(n 是值个数，两两检查活跃区间是否重叠);着色虽然是启发式，但遍历图、维护栈、反复试色，也要不少时间。对于 AOT 编译器(GCC 编译 C 程序),慢一点没关系，反正只编译一次。但对于 JIT,慢就是大问题。

### 2.2 线性扫描：快，够用

线性扫描是 1999 年 Massimiliano Poletto 和 Vivek Sarkar 提出的算法，专门为了**快**而设计。后来被 LLVM、HotSpot 的 client compiler(C1)、Graal、LuaJIT 广泛采用。

它的核心思路极其朴素:**不建图，直接扫一遍。**

**第一步：算出所有值的活跃区间。** 每个值有一个 [起点, 终点] 的区间(起点是定义处，终点是最后一次使用处)。

**第二步：把所有区间按起点排序。** 从小到大排好。

**第三步：线性扫描。** 维护一个"当前已分配的寄存器列表"。按起点顺序，逐个处理每个区间:
- 遇到一个新区间(一个值出生):看有没有空闲寄存器。有就分配给它；没有就选一个"当前占用着寄存器、但马上要过期或 spill 代价最低的",把它 spill 掉，腾出寄存器给当前这个。
- 扫到一个区间的终点(一个值死了):释放它占的寄存器，标记为空闲。

就这么扫一遍，完了。每个值要么拿到了寄存器，要么被 spill。整个过程是 **O(n log n)**(排序是主要开销，扫描本身 O(n))——比图着色的 O(n²) 快得多。

线性扫描的缺点是**质量略低**。因为它不建干扰图，只看区间的起止，有些本来可以更优的复用机会它会错过。但实践证明，这个质量损失通常很小(几个百分点的 spill 多一点),换来的是**编译速度大幅提升**。

### 2.3 一个最小例子：走一遍两种算法

为了把两者看清，我们用一个极简的例子，分别走一遍。

假设有 3 个物理寄存器(R1, R2, R3),有 5 个值，活跃区间如下:

```
A: [1, 6]   (第1条定义,第6条最后一次用)
B: [2, 4]
C: [3, 8]
D: [5, 7]
E: [6, 9]
```

**图着色怎么搞:**

先建干扰图——区间重叠的连边:
- A[1,6] 和 B[2,4] 重叠(2~4 段)→ 连边
- A 和 C[3,8] 重叠(3~6)→ 连边
- A 和 D[5,7] 重叠(5~6)→ 连边
- A 和 E[6,9] 在 6 点重叠(端点)→ 看是否含端点，假设重叠 → 连边
- B 和 C 重叠(3~4)→ 连边
- C 和 D 重叠(5~7)→ 连边
- C 和 E 重叠(6~8)→ 连边
- D 和 E 重叠(6~7)→ 连边
- B 和 D 不重叠(B 死在 4,D 生在 5)→ 不连
- B 和 E 不重叠 → 不连

得到图：每个节点和很多节点相连。然后试着用 3 种颜色着色。你会发现 A、C、D、E 互相都连着边(两两冲突),这是一个 4-clique(完全图 K4),4 个互相冲突的值,3 种颜色根本不够——必然有一个要 spill。图着色会发现这一点，选一个 spill 掉(比如选 spill 代价最低的那个),剩下 3 个着色。

**线性扫描怎么搞:**

按起点排序(已经排好了):A(1), B(2), C(3), D(5), E(6)。

扫到 A[1,6]:R1 空，分配 R1。占用:{A=R1}。
扫到 B[2,4]:R2 空，分配 R2。占用:{A=R1, B=R2}。
扫到 C[3,8]:R3 空，分配 R3。占用:{A=R1, B=R2, C=R3}。
扫到 D[5,7]:先检查有没有过期的——B 在 4 就死了!B 的区间已过期，释放 R2。现在 R2 空。分配 R2 给 D。占用:{A=R1, C=R3, D=R2}。
扫到 E[6,9]:检查过期——A 在 6 死了(假设端点 6 是最后一次用,A 的区间到此为止)。释放 R1。分配 R1 给 E。占用:{C=R3, D=R2, E=R1}。

结果:A=R1, B=R2, C=R3, D=R2, E=R1。**没有一个 spill!** 线性扫描巧妙地利用了"前面的死了腾出寄存器",把 5 个值全塞进了 3 个寄存器。

这个例子说明：线性扫描虽然简单，但在很多情况下效果并不差(甚至这个例子里比图着色判断得更乐观，因为图着色把端点重叠也当冲突，而线性扫描按"死了一个再生一个"更灵活)。当然，也有线性扫描不如图着色的反例——但总体上，质量损失有限，速度优势巨大。

### 2.4 为什么 LuaJIT 选线性扫描

现在回答核心问题:LuaJIT 为什么选线性扫描?

理由有三层，层层递进:

**第一层:JIT 必须快。** 这是表面的、也是最强的理由。寄存器分配是后端代码生成里耗时占比很大的一步。图着色的 O(n²) 建图 + 启发式着色，在一个要"现场编译、马上就要跑"的 JIT 里，太贵了。LuaJIT 的目标是"编译开销小到用户感觉不到",线性扫描的 O(n log n) 正合适。

**第二层:trace 天然是线性的。** 这是深一层的理由，也是 trace JIT 的结构性优势。回忆 P0-01:trace 是**一条线性路径**,没有分叉(分叉都变成了 guard,失败就 side exit)。所以 trace 的 IR 也是线性的——一条直线序列，从开头到结尾。

这种线性结构，对图着色没有任何好处(图着色要的就是处理任意控制流图),但对线性扫描是**天然契合**。线性扫描本来就假设"按一个线性顺序扫",trace 恰好就是线性的。可以说,**trace JIT 和线性扫描，是天生一对。**

**第三层:trace 短。** 一个 trace 通常只有几十到几百条 IR 指令(不像一个完整方法可能有几千条)。trace 短，意味着值的活跃区间少、重叠情况简单，线性扫描的"质量略低"这个缺点被进一步稀释——本来就没几个值要分配，图着色能优化的空间也很有限。

所以 LuaJIT 选线性扫描，不是将就，是**正合适**。trace 的线性结构 + 短 + JIT 要快，三股力量都指向线性扫描。

还有第四个理由，藏在线性扫描的一个变体里，下一节会讲到:LuaJIT 用的不是普通的"正向"线性扫描，而是**反向线性扫描**(reverse linear scan)。这个反向，带来了一个精妙的简化。我们先看源码怎么落地。

## §3 源码印证:lj_asm.c 的 ra_* 函数族

讲清了原理，现在进源码。LuaJIT 的寄存器分配，全部实现在 `lj_asm.c` 的 `ra_*` 函数族里(`ra` = register allocator)。这一节，我们逐个看这些函数，看它们怎么把线性扫描落地。

但看源码之前，要先交代一个 LuaJIT 寄存器分配的关键特征，它和教科书里的线性扫描有一个重要区别——**反向**。

### 3.1 关键特征：反向线性扫描(R-LSRA)

教科书里的线性扫描是**正向**的：按区间起点从小到大扫(从程序开头扫到结尾)。LuaJIT 的寄存器分配是**反向**的：从 IR 的最后一条指令，往前扫，扫到第一条。

为什么反向?这和 LuaJIT 代码生成的整体策略有关。LuaJIT 生成机器码时，机器码是**从高地址往低地址生长**的——每生成一条机器指令,`mcp`(machine code pointer)就减一(`--as->mcp`)。这是因为 LuaJIT 用了一种"反向发射"的技巧：先知道循环结尾要跳回哪里，再往前生成，跳转目标自然就定好了。

既然机器码是反向生成的，寄存器分配也跟着反向扫 IR,是最自然的——**生成到哪条 IR,就分配哪条 IR 用到的寄存器**。这样寄存器分配和代码生成同步进行，不需要先扫一遍 IR 算好分配、再扫一遍生成代码。

反向扫描带来一个精妙的简化，源码注释里写得很清楚(`lj_target.h:79-120`,这段注释非常重要，值得整段读):

```
The allocator first checks the free mask for a hit. Otherwise an (unrolled)
linear search for the minimum cost is used. ...
Without the cost-model this degenerates to the standard heuristics for
(reverse) linear-scan register allocation. Since code generation is done
in reverse, a live interval extends from the last use to the first def.
For an SSA IR the IR reference is the first (and only) def and thus
trivially marks the end of the interval. The LSRA heuristics says to pick
the register whose live interval has the furthest extent, i.e. the lowest
IR reference in our case.
```

这段话点出了反向线性扫描在 SSA 上的妙处。正向线性扫描里，一个值的活跃区间是 [定义点, 最后使用点]。要算"最后使用点"得扫一遍所有指令才行。但反向扫描里，扫到一个值的时候，它的"定义点"还没扫到(定义点在更前面),而它的所有"使用点"已经扫过了(使用点在更后面，反向扫已经经过了)。**更妙的是,SSA 里每个值只定义一次，而 LuaJIT 的 IR ref 是按定义顺序编号的**——ref 越小，定义得越早(越靠前)。

所以反向扫描时，遇到一个正在用的值，它的 ref 就是它的"活跃区间起点"(定义点)。这个信息是**免费**的——不用专门算,ref 本身就编码了。"选活跃区间最远的那个 spill",在反向扫描里就是"选 ref 最小的那个 spill"。这是一个极大的简化：不用维护区间数据结构,ref 本身就是区间信息。

这就是 LuaJIT 寄存器分配的精髓:**反向线性扫描 + 利用 SSA 的 ref 编码活跃区间 + 成本模型加权**。下面看它怎么用源码落地。

### 3.2 寄存器和寄存器集合:RID 与 RSET

先看最基本的类型定义，在 `lj_target.h`。

寄存器用一个 `Reg` 类型表示(`lj_target.h:15`),实际是 `uint32_t`。这条分配结果最终记到 IR 指令的 `r` 字段——`IRIns` 结构(`lj_ir.h:556`)里有个 union,`prev` 字段在寄存器分配前用作链表指针，分配后复用为 `r`(分配的寄存器)+ `s`(spill 槽)两个 8 位字段。每个物理寄存器有一个 ID(RID, register id),从 0 开始编号。几个特殊的 ID(`lj_target.h:21-25`):

```c
#define RID_NONE        0x80
#define RID_MASK        0x7f
#define RID_INIT        (RID_NONE|RID_MASK)
#define RID_SINK        (RID_INIT-1)
#define RID_SUNK        (RID_INIT-2)
```

`RID_NONE`(0x80)表示"还没有分配寄存器"——注意它的最高位是 1。`ra_noreg(r)` 就是检查最高位(`(r) & RID_NONE`,`lj_target.h:27`),`ra_hasreg(r)` 是反过来(`!((r) & RID_NONE)`)。这个设计的妙处注释里说了(`lj_target.h:17-20`):**已分配的寄存器，高 bit 是 0**,这样它的值可以直接拿来用，不用 mask。高 bit 是 1 的那些(`RID_INIT`/`RID_SINK`/`RID_SUNK`)是特殊标记，不是真正的寄存器——`RID_SINK` 和 `RID_SUNK` 是给"分配消除"(sink,见 P3-11)用的，表示这个值被下沉了不需要真的分配寄存器。

每个架构有自己的寄存器列表，用宏 `GPRDEF` 和 `FPRDEF` 定义。以 x86/x64 为例(`lj_target_x86.h:12-23`):

```c
#if LJ_64
#define GPRDEF(_) \
  _(EAX) _(ECX) _(EDX) _(EBX) _(ESP) _(EBP) _(ESI) _(EDI) \
  _(R8D) _(R9D) _(R10D) _(R11D) _(R12D) _(R13D) _(R14D) _(R15D)
#define FPRDEF(_) \
  _(XMM0) _(XMM1) _(XMM2) _(XMM3) _(XMM4) _(XMM5) _(XMM6) _(XMM7) \
  _(XMM8) _(XMM9) _(XMM10) _(XMM11) _(XMM12) _(XMM13) _(XMM14) _(XMM15)
#else
#define GPRDEF(_) \
  _(EAX) _(ECX) _(EDX) _(EBX) _(ESP) _(EBP) _(ESI) _(EDI)
#define FPRDEF(_) \
  _(XMM0) _(XMM1) _(XMM2) _(XMM3) _(XMM4) _(XMM5) _(XMM6) _(XMM7)
#endif
```

x86-64 有 16 个 GPR + 16 个 FPR;32 位 x86 只有 8 个 GPR + 8 个 FPR。这套宏被展开成枚举(`lj_target_x86.h:29-62`),`RID_EAX`=0, `RID_ECX`=1, ..., `RID_MAX` 是总数。还有几个特殊的别名:`RID_SP = RID_ESP`(栈指针，不能分配)、`RID_BASE = RID_EDX`(解释器的 BASE 寄存器)、`RID_RET = RID_EAX`(函数返回值寄存器)等。

寄存器集合用 `RegSet` 表示(`lj_target.h:58-68`)。它是一个**位图**(bitset),每个 bit 对应一个寄存器。x86/ARM 用 32 位(`uint32_t`),PPC/MIPS/ARM64 寄存器多，用 64 位(`uint64_t`)。一组操作宏(`lj_target.h:70-77`):

```c
#define RID2RSET(r)     (((RegSet)1) << (r))     /* 单个寄存器变集合 */
#define RSET_EMPTY      ((RegSet)0)
#define RSET_RANGE(lo, hi)  ((RID2RSET((hi)-(lo))-1) << (lo))  /* 连续区间 */
#define rset_test(rs, r)    ((int)((rs) >> (r)) & 1)   /* 测试某寄存器在不在集合里 */
#define rset_set(rs, r)     (rs |= RID2RSET(r))        /* 加入 */
#define rset_clear(rs, r)   (rs &= ~RID2RSET(r))       /* 移除 */
#define rset_exclude(rs, r) (rs & ~RID2RSET(r))        /* 排除某寄存器 */
```

位图的好处是：判断"有没有空闲寄存器"、从集合里挑一个寄存器，都是位运算，飞快。`rset_picktop`(取集合里编号最大的)和 `rset_pickbot`(取编号最小的)用的是 `lj_fls`(find last set)和 `lj_ffs`(find first set)这两个位操作指令，在 x86 上对应 `bsr`/`bsf` 单条指令。

几个关键的预定义集合(`lj_target_x86.h:67-72`):

```c
#define RSET_GPR    (RSET_RANGE(RID_MIN_GPR, RID_MAX_GPR) \
                     - RID2RSET(RID_ESP) \
                     - LJ_GC64*RID2RSET(RID_DISPATCH))
#define RSET_FPR    (RSET_RANGE(RID_MIN_FPR, RID_MAX_FPR))
#define RSET_ALL    (RSET_GPR|RSET_FPR)
#define RSET_INIT   RSET_ALL
```

`RSET_GPR` 是所有可用通用寄存器(扣掉 ESP 栈指针；在 GC64 模式下还扣掉 DISPATCH 寄存器)。`RSET_FPR` 是所有浮点寄存器。`RSET_ALL` 是全部。`RSET_INIT` 是初始状态——一开始，所有寄存器都"空闲"。

还有个 `RSET_SCRATCH`(`lj_target_x86.h:86-98`),是"调用者保存"(caller-saved)的寄存器集合——这些寄存器在函数调用时会被破坏。代码生成时，凡是会调用 C 函数的地方，这些寄存器里的值都要预先处理(spill 或 save)。

### 3.3 ASMState:寄存器分配的全局状态

寄存器分配不是孤立的一步，它和代码生成交织在一起，共享一个状态结构 `ASMState`(`lj_asm.c:41-112`)。这个结构体很大，我们只看和寄存器分配直接相关的字段:

```c
typedef struct ASMState {
  RegCost cost[RID_MAX];  /* 每个寄存器的分配成本(ref + 加权得分) */
  ...
  RegSet freeset;   /* 空闲寄存器集合 */
  RegSet modset;    /* 本次循环内被修改过的寄存器集合 */
  RegSet weakset;   /* 弱引用寄存器集合(可优先 spill) */
  RegSet phiset;    /* PHI 寄存器集合 */
  ...
  int32_t evenspill;  /* 下一个偶数 spill 槽 */
  int32_t oddspill;   /* 下一个奇数 spill 槽(或 0) */
  IRRef curins;       /* 当前正在处理的 IR 指令的 ref */
  IRRef snapref;      /* 当前 snapshot 生效的 ref 位置 */
  ...
  IRRef1 phireg[RID_MAX];  /* 每个寄存器对应的 PHI ref */
  ...
} ASMState;
```

四个 `RegSet` 是核心:

- **`freeset`**:当前哪些寄存器是空闲的、可以分配。这是寄存器分配的"主账本"。分配一个寄存器，就从 `freeset` 里移走；释放一个，就加回来。
- **`modset`**:循环内被修改过的寄存器。这个集合用于处理循环：循环体里改过的寄存器，在循环回边时要保证状态正确(因为下次循环还要用)。
- **`weakset`**:弱引用寄存器。这是 snapshot 相关的——某些值只是为了"万一 guard 失败要恢复"而临时放进寄存器(见 §4.3),它们是"弱"的，优先被 spill。
- **`phiset`**:PHI 寄存器。PHI 是循环回边处"合并"多个版本值的指令(见 P3-12),它的寄存器分配有特殊处理(见 §3.7)。

`cost[RID_MAX]` 是成本模型的核心：每个寄存器有一个 `RegCost`(uint32),低 16 位存的是**当前占用这个寄存器的值的 ref**,高 16 位存的是**加权成本**。spill 时挑成本最低的寄存器下手。

`evenspill`/`oddspill` 管理 spill 槽:64 位值要用偶数对齐的槽,32 位值可以用奇数槽，这样不浪费空间。

### 3.4 ra_setup:初始化

寄存器分配从 `ra_setup` 开始(`lj_asm.c:358-369`):

```c
static void ra_setup(ASMState *as)
{
  Reg r;
  /* Initially all regs (except the stack pointer) are free for use. */
  as->freeset = RSET_INIT;
  as->modset = RSET_EMPTY;
  as->weakset = RSET_EMPTY;
  as->phiset = RSET_EMPTY;
  memset(as->phireg, 0, sizeof(as->phireg));
  for (r = RID_MIN_GPR; r < RID_MAX; r++)
    as->cost[r] = REGCOST(~0u, 0u);
}
```

一开始，所有寄存器都空闲(`RSET_INIT`)。所有集合清空。每个寄存器的成本初始化成最大值(`~0u`,即无穷大)——表示"这个寄存器还没被占用,spill 它代价无穷大(因为它根本没东西可 spill)"。

注意一个细节:`RSET_INIT` 已经排除了 ESP(和 GC64 下的 DISPATCH),所以"所有寄存器"本来就是"所有可分配的寄存器"。栈指针从一开始就不在游戏里。

### 3.5 成本模型:cost 数组与 REGCOST

LuaJIT 的寄存器分配不是纯粹的"最远区间优先",它还叠加了一个**成本模型**。源码注释(`lj_target.h:81-120`)详细解释了这个模型，我们抓要点。

每个寄存器的 `cost` 是一个 32 位数，拆成两部分(`lj_target.h:127-132`):

```c
#define REGCOST(cost, ref)  ((RegCost)(ref) + ((RegCost)(cost) << 16))
#define regcost_ref(rc)     ((IRRef1)(rc))
#define REGCOST_T(t) \
  ((RegCost)((t)&IRT_ISPHI) * (((RegCost)(REGCOST_PHI_WEIGHT)<<16)/IRT_ISPHI))
#define REGCOST_REF_T(ref, t)  (REGCOST((ref), (ref)) + REGCOST_T((t)))
```

- 低 16 位(`regcost_ref`):当前占用这个寄存器的值的 ref。ref 越小，定义越早，活跃区间越远(反向扫描里"远"意味着下次用还很远,spill 它代价低)。
- 高 16 位：加权得分。目前唯一的权重是 **PHI 权重**(`REGCOST_PHI_WEIGHT = 64`,`lj_target.h:121`)。如果一个值是 PHI(循环合并点),它的成本会被加上 64 个单位的权重——意思是"PHI 值特别不想被 spill"(因为 spill PHI 会破坏循环的寄存器复用，代价高)。

spill 时(见下面的 `ra_evict`),挑 `cost` 最小的寄存器下手。由于 ref 在低 16 位、权重在高 16 位，比较 `cost` 的时候:**先比权重(高 16 位),权重小的优先 spill;权重相同再比 ref,ref 小的(活跃区间远的)优先 spill。** 这就把"PHI 不 spill"和"远区间优先 spill"两个启发式统一在一个 32 位整数比较里，极其紧凑。

注释还提到一个设计巧合(`lj_target.h:108-114`):由于 IR 的精心设计,**常量的 ref 天然比不变量小，不变量的 ref 天然比变量小**。所以"ref 小优先 spill"这条启发式，天然就把常量优先 spill(常量 spill 了随时能重新 load 进来，代价几乎为零)、把变量尽量留在寄存器——这正好是想要的结果，不用额外区分。

### 3.6 核心函数族:ra_allocref / ra_evict / ra_restore / ra_spill

现在看寄存器分配的核心函数。这几个函数互相调用，组成一个协作的小体系。

**分配一个寄存器给一个 ref:`ra_allocref`(`lj_asm.c:670-711`)**

这是最核心的分配函数。它给一个还没分配寄存器的 ref,分配一个物理寄存器。

```c
static Reg ra_allocref(ASMState *as, IRRef ref, RegSet allow)
{
  IRIns *ir = IR(ref);
  RegSet pick = as->freeset & allow;   /* 空闲且允许的寄存器 */
  Reg r;
  lj_assertA(ra_noreg(ir->r), "IR %04d already has reg %d", ...);
  if (pick) {
    /* 1. 先看有没有 hint(从 PHI 或传播来的提示) */
    if (ra_hashint(ir->r)) {
      r = ra_gethint(ir->r);
      if (rset_test(pick, r)) goto found;  /* hint 寄存器可用,直接用 */
      /* hint 寄存器被占着,但占着的是常量,重新物化比错过 hint 还便宜 */
      if (rset_test(allow, r) && emit_canremat(regcost_ref(as->cost[r]))) {
        ra_rematk(as, regcost_ref(as->cost[r]));
        goto found;
      }
    }
    /* 2. 不变量优先用没被修改过的寄存器 */
    if (ref < as->loopref && !irt_isphi(ir->t)) {
      if ((pick & ~as->modset)) pick &= ~as->modset;
      r = rset_pickbot(pick);   /* 从小编号挑(减少冲突) */
    } else {
      /* 3. 变量:寄存器多的话,优先用 callee-save(不被调用破坏) */
      if (RID_NUM_GPR > 8 && (pick & ~RSET_SCRATCH)) pick &= ~RSET_SCRATCH;
      r = rset_picktop(pick);   /* 从大编号挑 */
    }
  } else {
    r = ra_evict(as, allow);   /* 没有空闲的,evict 一个 */
  }
found:
  ir->r = (uint8_t)r;                  /* 记录分配结果 */
  rset_clear(as->freeset, r);          /* 标记为占用 */
  ra_noweak(as, r);                    /* 清除弱标记 */
  as->cost[r] = REGCOST_REF_T(ref, irt_t(ir->t));  /* 更新成本 */
  return r;
}
```

这个函数的逻辑清晰地体现了反向线性扫描的启发式:

1. **hint 优先**。如果这个 ref 有一个"建议寄存器"(来自 PHI 合并或寄存器传播，见后面),优先用建议的那个。用建议的寄存器可以省掉一次 `mov`(值的搬运),所以优先满足 hint。
2. **不变量 vs 变量不同策略**。`ref < as->loopref` 判断这个值是不是在循环外定义的(不变量)。不变量优先用"没被循环修改过"的寄存器(`~as->modset`),这样循环回边时不用管它。变量则优先用 callee-save 寄存器(不被 C 调用破坏)。
3. **空闲不够就 evict**。如果 `freeset & allow` 是空的(没有既空闲又允许的寄存器),调 `ra_evict` 挑一个占用着的、spill 掉，腾出来。

注意 `pickbot`(小编号优先)和 `picktop`(大编号优先)的差别：不变量小编号优先、变量大编号优先，这是为了"让两类寄存器在不同的编号区间，减少互相挤占"。源码注释 `rset_pickbot(pick); /* Reduce conflicts with inverse allocation. */` 点出了这点。

**evict(驱逐)一个寄存器:`ra_evict`(`lj_asm.c:501-521`)**

当没有空闲寄存器时，要挑一个占用着的，把它 spill 走，腾出寄存器。

```c
#define MINCOST(name) \
  if (rset_test(RSET_ALL, RID_##name) && \
      LJ_LIKELY(allow&RID2RSET(RID_##name)) && as->cost[RID_##name] < cost) \
    cost = as->cost[RID_##name];

static Reg ra_evict(ASMState *as, RegSet allow)
{
  IRRef ref;
  RegCost cost = ~(RegCost)0;   /* 初始最大 */
  lj_assertA(allow != RSET_EMPTY, "evict from empty set");
  if (RID_NUM_FPR == 0 || allow < RID2RSET(RID_MAX_GPR)) {
    GPRDEF(MINCOST)   /* 遍历所有 GPR,找成本最小的 */
  } else {
    FPRDEF(MINCOST)   /* 或遍历所有 FPR */
  }
  ref = regcost_ref(cost);   /* 成本最小的那个,它占的寄存器就是要 evict 的 */
  ...
  /* 优先 evict 弱引用(只为 snapshot 临时占用的) */
  if (!irref_isk(ref) && (as->weakset & allow)) {
    IRIns *ir = IR(ref);
    if (!rset_test(as->weakset, ir->r))
      ref = regcost_ref(as->cost[rset_pickbot((as->weakset & allow))]);
  }
  return ra_restore(as, ref);   /* 把这个 ref 处理掉(restore 或 remat) */
}
```

`MINCOST` 宏被 `GPRDEF`/`FPRDEF` 展开，展开后是一串"if (EAX 允许 && cost[EAX] < 当前最小) 更新最小"的判断——遍历所有寄存器，找成本最低的。这是个**展开的线性搜索**,源码注释说不用记最小值的位置，只记最小值的成本，最后从成本里反解出 ref(`regcost_ref(cost)`)。

挑出成本最低的之后，还有个优化：如果这个 ref 不是常量，而且 `weakset` 里有允许的寄存器，优先 evict 弱引用(弱引用是只为 snapshot 临时占的,spill 它代价最低)。

最后调 `ra_restore` 真正处理这个被 evict 的值。

**restore(恢复)一个被 evict 的值:`ra_restore`(`lj_asm.c:468-486`)**

```c
static Reg ra_restore(ASMState *as, IRRef ref)
{
  if (emit_canremat(ref)) {
    return ra_rematk(as, ref);   /* 常量:重新物化(重新 load) */
  } else {
    IRIns *ir = IR(ref);
    int32_t ofs = ra_spill(as, ir);   /* 变量:强制 spill 到栈 */
    Reg r = ir->r;
    ra_sethint(ir->r, r);   /* 保留 hint,以后可能还分配到这个寄存器 */
    ra_free(as, r);
    if (!rset_test(as->weakset, r)) {   /* 非弱引用才真的生成 reload 指令 */
      ra_modified(as, r);
      emit_spload(as, ir, r, ofs);   /* 生成从栈 load 回寄存器的指令 */
    }
    return r;
  }
}
```

这里有个关键的分流:**常量走 remat(重新物化),变量走 spill。**

`emit_canremat(ref)` 判断这个 ref 能不能重新物化。它的定义很简单(`lj_emit_x86.h:589`):

```c
#define emit_canremat(ref)  ((ref) <= REF_BASE)
```

`REF_BASE` 是一个特殊 ref,常量的 ref 都比它小。所以"能 remat"的就是各种常量(KINT/KNUM/KGC/KPTR 等)和 BASE。常量为什么能 remat?因为常量的值是固定的,spill 到栈再 load 回来，不如直接从原始来源(立即数、全局指针)重新生成——反正值不变，重新 load 一次和从栈 load 一次，代价差不多甚至更便宜(立即数 load 不用访问内存)。所以常量"spill"不是真的写到栈，而是**直接丢弃寄存器，用的时候重新 load**。这叫 rematerialization(重新物化),是比 spill 更省的技巧。

变量不能 remat(它的值是算出来的，没法重新生成),只能老老实实 spill 到栈，用的时候从栈 load 回来。

`emit_spload` 和 `emit_spstore` 是架构相关的宏，在 x86 上展开成 `mov reg, [rsp+ofs]` 和 `mov [rsp+ofs], reg` 这样的内存访问指令。

**spill(强制溢出):`ra_spill`(`lj_asm.c:429-451`)**

```c
static int32_t ra_spill(ASMState *as, IRIns *ir)
{
  int32_t slot = ir->s;
  if (!ra_hasspill(slot)) {   /* 还没分配 spill 槽 */
    if (irt_is64(ir->t)) {
      slot = as->evenspill;       /* 64 位:偶数槽 */
      as->evenspill += 2;
    } else if (as->oddspill) {    /* 32 位,有奇数槽可用 */
      slot = as->oddspill;
      as->oddspill = 0;
    } else {                      /* 32 位,没奇数槽,新开一对 */
      slot = as->evenspill;
      as->oddspill = slot+1;
      as->evenspill += 2;
    }
    if (as->evenspill > 256)
      lj_trace_err(as->J, LJ_TRERR_SPILLOV);   /* spill 槽太多,报错 */
    ir->s = (uint8_t)slot;
  }
  return sps_scale(slot);   /* 返回字节偏移 */
}
```

这个函数管理 spill 槽的分配。spill 槽在栈上，每个槽 4 字节(`sps_scale(slot) = 4 * slot`,`lj_target_x86.h:151`)。64 位值要 8 字节，用偶数对齐的槽(占两个)。32 位值可以用奇数槽(填补 64 位值留下的空隙),避免浪费。

这是个聪明的打包策略:64 位值占偶数对齐的两个槽,32 位值塞进奇数槽。`evenspill` 和 `oddspill` 两个游标分别管理偶数和奇数的下一个空闲位置。如果 spill 槽超过 256 个(`LJ_TRERR_SPILLOV`),直接报错放弃这个 trace——这种情况极少见，通常意味着代码出了问题。

### 3.7 把分配串起来:ra_dest / ra_left / ra_scratch

上面是底层原语。代码生成时，高层一点的是 `ra_dest`、`ra_left`、`ra_scratch` 这几个，它们组合底层原语完成"为一条 IR 指令分配寄存器"的任务。

**ra_dest:给一条指令的结果分配目的寄存器(`lj_asm.c:766-784`)**

```c
static Reg ra_dest(ASMState *as, IRIns *ir, RegSet allow)
{
  Reg dest = ir->r;
  if (ra_hasreg(dest)) {      /* 已经有寄存器了(从 hint 来的) */
    ra_free(as, dest);
    ra_modified(as, dest);
  } else {
    if (ra_hashint(dest) && rset_test((as->freeset&allow), ra_gethint(dest))) {
      dest = ra_gethint(dest);   /* 用 hint */
    } else {
      dest = ra_scratch(as, allow);  /* 拿一个 scratch 寄存器 */
    }
    ir->r = dest;
  }
  if (LJ_UNLIKELY(ra_hasspill(ir->s))) ra_save(as, ir, dest);  /* 有 spill 槽就存一份 */
  return dest;
}
```

每条会算出结果的 IR 指令，都要调 `ra_dest` 给结果找个家。逻辑：如果之前已经有寄存器了(从 PHI 或传播来的 hint),直接用；否则拿一个 scratch 寄存器。如果这个值同时有 spill 槽(被 snapshot 要求要 spill),还要生成一条 store 指令把值存到栈上(`ra_save`)。

**ra_left:处理两操作数指令的左操作数(`lj_asm.c:801-846`)**

x86 的运算指令大多是**两操作数**(two-operand):`add eax, ecx` 是 `eax = eax + ecx`,结果写回第一个操作数。这和 IR 的三操作数语义(`%c = ADD %a, %b`)对不上。`ra_left` 就是处理这个差异的(`lj_asm.c:797-846` 注释:"This is a required fixup step for all 2-operand machine instructions")。

它的逻辑：如果左操作数还没有寄存器，给它分配一个；如果左操作数的寄存器和目的寄存器不一样，生成一条 `mov` 把左操作数搬到目的寄存器，这样后面的运算就能"在目的寄存器上累加"。但有个优化：如果左操作数是 PHI 值，且目的寄存器正好是它的 PHI 寄存器，就用 **rename**(重命名)而不是 mov——见下节。

**ra_scratch:拿一个临时寄存器(`lj_asm.c:534-540`)**

```c
static Reg ra_scratch(ASMState *as, RegSet allow)
{
  Reg r = ra_pick(as, allow);   /* 从空闲里挑一个,没有就 evict */
  ra_modified(as, r);           /* 标记为已修改 */
  return r;
}
```

有些指令需要临时寄存器(比如算复杂地址、调用函数时存中间值)。`ra_scratch` 就是拿一个临时的，用完就丢(标记为 modified,后续可以重新分配)。

### 3.8 PHI 与 rename:循环回边的特殊处理

讲到现在，有一个情况我们绕开了:**循环**。

循环的寄存器分配有个特殊难题。考虑一个循环:

```
loop:
  %x_new = ADD %x_old, %i    ; x 每次循环加上 i
  ...
  goto loop                   ; 回到开头,%x_old 现在是上一轮的 %x_new
```

这里 `%x_old` 和 `%x_new` 是同一个逻辑变量在不同循环轮次的两个版本。SSA 用 **PHI** 节点表达这种"合并":在循环开头插一个 `%x = PHI(%x_init, %x_new)`,表示"`%x` 的值，第一轮来自初始值 `%x_init`,后续轮次来自上一轮的 `%x_new`"。

PHI 给寄存器分配带来一个约束:**`%x`(PHI 的结果)和 `%x_new`(回边来的那个版本),最好用同一个物理寄存器。** 为什么?因为循环回边时,`%x_new` 的值要变成下一轮 `%x` 的值。如果它俩用同一个寄存器，回边什么都不用做(值已经在正确的寄存器里);如果用不同的寄存器，回边处要生成一条 `mov` 搬运——白白多一条指令。

所以 LuaJIT 的策略是:**尽量让 PHI 和它的回边版本共用寄存器**,这叫"PHI 寄存器合并"(PHI coalescing)。

这体现在 `ra_rename`(`lj_asm.c:734-760`):

```c
static void ra_rename(ASMState *as, Reg down, Reg up)
{
  IRRef ref = regcost_ref(as->cost[up] = as->cost[down]);  /* 把 down 的所有权转给 up */
  IRIns *ir = IR(ref);
  ir->r = (uint8_t)up;            /* 值现在在 up 寄存器 */
  as->cost[down] = 0;
  ...
  ra_free(as, down);              /* down 空闲了 */
  ra_modified(as, down);
  rset_clear(as->freeset, up);    /* up 被占用 */
  ra_noweak(as, up);
  emit_movrr(as, ir, down, up);   /* 反向代码生成,要发反向的 mov */
  if (!ra_hasspill(IR(ref)->s)) {
    ra_addrename(as, down, ref, as->snapno + as->snapalloc);  /* 记录到 IR,供 snapshot 用 */
  }
}
```

`ra_rename` 的语义是：把一个值从寄存器 `down` "改名"到寄存器 `up`。这不是真的搬数据(数据还在 `down` 里),而是**在逻辑上把这个值的归属从 `down` 改成 `up`**,同时生成一条 `mov` 指令把数据真正搬过去(因为是反向代码生成,mov 的方向是 `up → down` 的反向，即 `mov up, down` 的反向 emit)。

为什么要这么绕?因为反向代码生成里,"回边"对应的机器码位置，是在更后面(已经生成完了)。要在那个位置插入一条 mov,不能直接插(已经过去了),所以用一个 trick:把这个 rename 记录到 IR 里(加一条 `IR_RENAME` 指令,`ra_addrename`,`lj_asm.c:724-731`),等 snapshot 处理时再根据 RENAME 调整。这样，循环回边的寄存器搬运，就被妥善安排在了正确的位置。

PHI 的处理集中在 `asm_phi`(`lj_asm.c:1647-1681`)。它在扫描到 PHI 指令时调用:

```c
static void asm_phi(ASMState *as, IRIns *ir)
{
  RegSet allow = ((!LJ_SOFTFP && irt_isfp(ir->t)) ? RSET_FPR : RSET_GPR) & ~as->phiset;
  RegSet afree = (as->freeset & allow);
  IRIns *irl = IR(ir->op1);   /* PHI 的左操作数(初始值) */
  IRIns *irr = IR(ir->op2);   /* PHI 的右操作数(回边来的) */
  if (ir->r == RID_SINK) return;   /* 被 sink 了,不分配 */
  ...
  if ((afree & (afree-1))) {   /* 有两个以上空闲寄存器 */
    Reg r;
    if (ra_noreg(irr->r)) {
      r = ra_allocref(as, ir->op2, allow);   /* 给右操作数分配 */
    } else {
      r = ra_scratch(as, allow);             /* 右操作数已有寄存器,复制一份 */
      emit_movrr(as, irr, r, irr->r);
    }
    ir->r = (uint8_t)r;
    rset_set(as->phiset, r);                 /* 标记为 PHI 寄存器 */
    as->phireg[r] = (IRRef1)ir->op1;         /* 记录左操作数,供后面合并 */
    irt_setmark(irl->t);
    if (ra_noreg(irl->r)) ra_sethint(irl->r, r);  /* 给左操作数设 hint,尽量用同一个寄存器 */
  } else {   /* 空闲寄存器不够,PHI 走 spill(罕见) */
    ...
    ra_spill(as, ir);
    irr->s = ir->s;
  }
}
```

逻辑：如果空闲寄存器够(至少两个，留一个给非 PHI),给 PHI 的右操作数分配一个寄存器，把 PHI 结果也指向它(这样回边时不用搬),再给左操作数设个 hint 让它也尽量用这个寄存器(这样初始值进入循环时也不用搬)。如果寄存器实在不够,PHI 只能 spill(这种情况注释说是"synthetic code"才会触发，正常代码不会)。

这套 PHI 处理，加上 `asm_phi_shuffle`(循环回边处的寄存器搬运)和 `asm_phi_copyspill`(spill 槽同步),组成了循环寄存器分配的完整方案。核心目标就一个:**让循环回边尽量不产生 mov 指令。**

### 3.9 主循环:lj_asm_trace 的反向扫描

最后，把所有这些串起来的是 `lj_asm_trace`(`lj_asm.c:2471-2639`),它是整个汇编生成的入口。它的核心是一个**反向扫描 IR 的循环**(`lj_asm.c:2565-2579`):

```c
/* Assemble a trace in linear backwards order. */
for (as->curins--; as->curins > as->stopins; as->curins--) {
  IRIns *ir = IR(as->curins);
  /* 64 bit types handled by SPLIT for 32 bit archs. */
  lj_assertA(!(LJ_32 && irt_isint64(ir->t)), ...);
  asm_snap_prev(as);                              /* 跨过 snapshot 边界 */
  if (!ra_used(ir) && !ir_sideeff(ir) && (as->flags & JIT_F_OPT_DCE))
    continue;                                     /* 死代码消除 */
  if (irt_isguard(ir->t))
    asm_snap_prep(as);                            /* 这条是 guard,准备 snapshot */
  RA_DBG_REF();
  checkmclim(as);
  asm_ir(as, ir);                                 /* 真正生成这条 IR 的机器码 */
}
```

`as->curins` 从最后一条 IR 指令开始，往前递减，直到 `stopins`(第一条)。每条指令:
1. `asm_snap_prev`:如果跨过了一个 snapshot 的边界，更新当前 snapshot(见 §4.3)。
2. 死代码消除：如果这条指令的结果没人用(`!ra_used`),也没副作用(`!ir_sideeff`),而且开了 DCE 优化，直接跳过——连机器码都不生成。
3. `asm_snap_prep`:如果这条是 guard(带 `IRT_GUARD` 标记),准备对应的 snapshot——保证 guard 失败时能正确恢复状态。
4. `asm_ir`:这是分发函数，根据 IR 的 opcode,调用对应的 `asm_xxx` 函数(比如 `ASM_ADD` 调 `asm_add`),那个函数里会调 `ra_dest`/`ra_left` 等完成寄存器分配和机器码生成。

注意这个循环没有显式的"分配寄存器"步骤——**寄存器分配是融合在代码生成里的**。每生成一条 IR 的机器码，就顺带把它用到的寄存器分配了(`ra_dest` 给结果分配,`ra_left`/`ra_alloc1` 给操作数分配)。这是 LuaJIT 寄存器分配的一个特点:**它不是一个独立的 pass,而是和代码生成交织在一起，边扫边分。** 这正是反向线性扫描的精髓——扫描本身就是分配。

扫描完之后，还要处理 trace 的头部(`asm_head_root` 或 `asm_head_side`,`lj_asm.c:2594-2597`)和尾部，这些我们留到 P4-14 讲汇编生成时细看。本章关注的是寄存器分配本身。

最后,`lj_asm_trace` 有个断言(`lj_asm.c:2623`):

```c
if (as->freeset != RSET_ALL)
  lj_trace_err(as->J, LJ_TRERR_BADRA);  /* Ouch! Should never happen. */
```

扫完之后，所有寄存器应该都回归空闲(`freeset == RSET_ALL`)。如果不是，说明寄存器分配有 bug(有寄存器没释放),直接报错。这是寄存器分配正确性的一个运行时检查。

## §4 为什么这样设计是 sound 的

讲完了源码，这一节我们要回答一个关键问题:**这套寄存器分配，为什么是正确的?为什么不会算错?** 这是"把动态执行安全变成机器码"里"安全"两个字的又一处体现。

寄存器分配的正确性，要保证三件事:**每个活着的值都有地方放(寄存器或 spill)、每次访问都能取到正确的值、guard 失败时 snapshot 能正确恢复。** 我们逐一验证。

### 4.1 每个活着的值都有地方放

这是最基本的。反向扫描时，每遇到一个值的使用，就调 `ra_alloc1` 确保它有寄存器(`lj_asm.c:714-721`):

```c
static Reg ra_alloc1(ASMState *as, IRRef ref, RegSet allow)
{
  Reg r = IR(ref)->r;
  if (ra_noreg(r)) r = ra_allocref(as, ref, allow);  /* 没寄存器就分配 */
  ra_noweak(as, r);
  return r;
}
```

`ra_allocref` 的逻辑是：先找空闲的(`freeset & allow`),没有就 evict 一个。evict 又是：挑成本最低的，要么 remat(常量)、要么 spill(变量，写到栈)。**无论寄存器多紧张，这个流程总能给出一个寄存器**——大不了把别的值 spill 到栈上，栈是无限的(直到 256 个 spill 槽的上限)。

所以，只要 spill 槽没超 256 个(超了报 `LJ_TRERR_SPILLOV` 放弃 trace),每个活着的值，要么在寄存器里，要么在栈上，绝不会"无处可去"。这是 soundness 的第一条保证。

### 4.2 每次访问都能取到正确的值

这一条看似显然，其实有几个陷阱,LuaJIT 都妥善处理了:

**陷阱一:spill 之后用错版本。** 一个值 spill 到栈之后，如果它又被修改了(在寄存器里改了新值),栈上的还是旧值。下次从栈 load,会拿到错的。

LuaJIT 怎么避免?它的 spill 是和代码生成同步的。`ra_save`(`lj_asm.c:489-493`)在值被修改后，生成一条 store 把新值写回栈;`emit_spload` 在值被 evict 后需要用时，生成一条 load 从栈读回。由于代码生成是反向的、和扫描同步的,store 和 load 的顺序在最终机器码里是正确的——总是"先 store 新值，后 load"。 SSA 的"每个值只定义一次"在这里帮了大忙：不会出现一个寄存器同时装着两个不同版本的同一个值。

**陷阱二：常量 remat 之后，值变了。** 常量本来不变,remat(重新 load)没问题。但如果误把一个变量当成常量 remat 了，就会拿到错值。`emit_canremat(ref)`(`ref <= REF_BASE`)严格限制了只有真正的常量(和 BASE 这种特殊"伪常量")才能 remat,变量永远走 spill,不会误 remat。

**陷阱三：循环回边，值在错误的寄存器。** 循环回边时，上一轮的 `%x_new` 要变成下一轮的 `%x`。如果它俩寄存器不一样，回边要 mov。`asm_phi_shuffle` 负责在循环回边处生成正确的 mov 序列，保证每个 PHI 值都搬到正确的寄存器。如果它俩一样(PHI 合并成功),就什么都不用做——这正是 PHI 合并优化的目的。

这三条都处理好之后，机器码运行时，每条指令访问的寄存器/栈位置，装的都是正确的值。这是 soundness 的第二条保证。

### 4.3 guard 失败时,snapshot 能正确恢复

这是最微妙的一条，也是 trace JIT 特有的正确性要求。

回忆 P0-01:trace 里到处是 guard(乐观假设的运行时检查)。guard 失败时，要 side exit——退出机器码，靠 snapshot 把状态恢复成解释器期望的样子，退回解释器继续跑。

snapshot(`SnapShot` 结构,`lj_jit.h:180`)里记录的是"这个 guard 退出时，每个对解释器有意义的值，应该恢复到哪个栈槽"。但这些值，在机器码运行时，分散在各个寄存器里(或者 spill 在栈上)。side exit 时，要把它们从"机器码的状态"(寄存器 + spill 栈)搬运到"解释器的状态"(值栈)。这就要求:**snapshot 记录的每个值，在 guard 退出那一刻，都必须是"可取到的"——要么在寄存器里，要么在 spill 栈上。**

寄存器分配怎么保证这一点?靠 **snapshot 驱动的预分配**。

在主扫描循环里，每遇到一个 guard 指令(`irt_isguard(ir->t)`),就调 `asm_snap_prep`(`lj_asm.c:1055-1070`):

```c
static void asm_snap_prep(ASMState *as)
{
  if (as->snapalloc) {
    /* 第一次遇到这个 snapshot,预分配 */
    as->snapalloc = 0;
    asm_snap_alloc(as, as->snapno);
    as->snaprename = as->T->nins;
  } else {
    /* 检查之后有没有 rename 影响这个 snapshot */
    for (; as->snaprename < as->T->nins; as->snaprename++) {
      IRIns *ir = &as->T->ir[as->snaprename];
      if (asm_snap_checkrename(as, ir->op1))
        ir->op2 = REF_BIAS-1;  /* 杀掉这个 rename(已处理) */
    }
  }
}
```

`asm_snap_alloc`(`lj_asm.c:1014-1033`)遍历这个 snapshot 里记录的所有值，对每个值调 `asm_snap_alloc1`(`lj_asm.c:953-1011`)。`asm_snap_alloc1` 的核心逻辑:

```c
static void asm_snap_alloc1(ASMState *as, IRRef ref)
{
  IRIns *ir = IR(ref);
  if (!irref_isk(ref)) {   /* 常量不用管(随时能 remat) */
    ...
    if (ra_used(ir)) return;   /* 已经有寄存器或 spill 槽了,不用管 */
    ...
    /* 这个值还没分配,给它分配一个"弱"寄存器,或者 spill 槽 */
    if ((as->freeset & allow) || ...) {
      Reg r = ra_allocref(as, ref, allow);
      if (!irt_isphi(ir->t)) ra_weak(as, r);   /* 标记为弱引用 */
    } else {
      ra_spill(as, ir);   /* 没空闲寄存器,spill 到栈 */
    }
  }
}
```

这段逻辑是:**对 snapshot 里每个还没着落的值，要么给它分配一个"弱"寄存器，要么给它一个 spill 槽。** 这样保证:guard 退出那一刻,snapshot 里的每个值都"可取到"。

注意"弱寄存器"(`ra_weak`)这个标记。它的含义是：这个寄存器**只是为了 snapshot 而临时占用**,不是程序正常运算需要的。所以当后续需要寄存器时，弱引用的寄存器会被**优先 evict**(`ra_evict` 里 `if (!irref_isk(ref) && (as->weakset & allow))` 那段)。弱引用被 evict 后会 spill 到栈——这没问题，反正它本来就是为了 snapshot 准备的,spill 到栈正好满足 snapshot"可取到"的要求。

还有一个微妙的点:**rename 和 snapshot 的一致性。** 前面讲 PHI 时提到 `ra_rename` 会改一个值的寄存器归属。如果一个值被 rename 了，而它正好出现在某个 snapshot 里，那 snapshot 记录的"这个值在哪个寄存器"就和实际不符了——side exit 时会从错误的寄存器取值。

LuaJIT 怎么处理?`asm_snap_checkrename`(`lj_asm.c:1042-1052`):每次有 rename 发生，检查这个被 rename 的值是不是出现在当前 snapshot 里。如果是,**强制给它一个 spill 槽**(`ra_spill(as, ir)`)。这样，即使它在寄存器之间搬来搬去，栈上始终有一份正确的拷贝,snapshot 从栈取就行——稳了。

这套 snapshot 驱动的预分配 + 弱引用 + rename 检查，保证了:**无论寄存器分配怎么折腾,snapshot 里记录的每个值，在 guard 退出那一刻，都是可取到的。** 这是 soundness 的第三条保证，也是寄存器分配和 trace JIT 的 guard/side exit 机制精密配合的体现。

### 4.4 小结：三层保证

合起来，寄存器分配的正确性有三层保证:

1. **每个活着的值都有地方放**(寄存器或 spill 栈，靠 `ra_allocref`/`ra_evict`/`ra_spill` 保证)。
2. **每次访问取到正确的值**(靠 SSA 单次定义、同步的 store/load、严格的 remat 判断、PHI 回边搬运保证)。
3. **guard 失败时 snapshot 可恢复**(靠 snapshot 驱动的预分配、弱引用、rename 检查保证)。

这三层加起来，寄存器分配不会破坏机器码的正确性——机器码算出的结果，和解释器算出的结果，一模一样(假设成立时),或者能正确退回解释器(假设失败时)。这正是主线"把动态执行**安全**变成机器码"在后端这一环的落地。

## §5 ★对照：官方 Lua 与 JVM/V8

把 LuaJIT 的寄存器分配放回两个对照对象里，取舍会更清晰。

### 5.1 对照一：官方 Lua——根本没有寄存器分配

官方 Lua 是**寄存器式字节码**的解释器(P1-02 讲过,LuaJIT 自己倒是栈式，这是两套实现的差异)。它的"寄存器"是**字节码层面的虚拟寄存器**——每个 Lua 函数有一组虚拟寄存器(局部变量各占一个),解释器执行字节码时，操作这些虚拟寄存器。但这些"虚拟寄存器",在底层其实还是存在**内存里**(Lua 栈的一块),由解释器的 C 代码(或 LuaJIT 解释器的手写汇编)来读写。

也就是说，官方 Lua **根本没有"把虚拟寄存器映射到物理寄存器"这一步**——因为它不生成机器码。字节码操作虚拟寄存器(在内存里),解释器的机器码(固定的那段循环)负责读懂字节码、搬弄虚拟寄存器。物理寄存器的使用，是解释器自己的事，和用户的代码无关。

所以，寄存器分配是 **JIT 特有的工序**。一旦你要把用户的代码变成机器码，你就必须面对"虚拟值怎么映射到物理寄存器"。官方 Lua 没这步，因为它不编译;LuaJIT 有这步，因为它要编译。这是"加 JIT 到底加了什么"的又一个具体体现——加了一整套后端，寄存器分配是其中第一关。

还有一个对照点:**LuaJIT 的解释器(手写汇编 VM)对物理寄存器的使用，是人工精心安排的**。比如 `RID_BASE = RID_EDX`(解释器的 BASE 指针固定用 EDX)、`RID_DISPATCH = RID_R14D`(dispatch 表指针固定用 R14)。这些是解释器作者手写汇编时定死的，不需要算法分配。只有 JIT 生成的机器码，才需要算法(线性扫描)来分配。这也是为什么 `RSET_GPR` 要减掉 `RID_DISPATCH`(GC64 模式)——那个寄存器被解释器占着,JIT 不能用。

### 5.2 对照二:JVM/V8——图着色 vs 线性扫描

JVM 和 V8 也有寄存器分配，因为它们也是 JIT。但它们的算法选择，和 LuaJIT 不同，这背后是 **method JIT vs trace JIT** 的根本分歧。

**HotSpot JVM(C2 编译器):历史上用图着色，后来部分改用线性扫描。** C2 是 HotSpot 的"服务端"编译器(优化重),历史上用的是**图着色**(Chaitin 风格，带 coalescing 等优化),追求生成代码的质量——因为 C2 编译的是热点方法，值得花时间优化。后来 HotSpot 的"客户端"编译器(C1)用的是**线性扫描**,因为 C1 要快(编译开销小)。所以 JVM 内部其实是**两套寄存器分配并存**:C1 线性扫描(快)、C2 图着色(质量高)。一个方法先用 C1 编译(快但糙),跑得足够热了再用 C2 重新编译(慢但精)。

**GraalVM(Graal 编译器):线性扫描。** Graal 是用 Java 写的现代 JIT,用的也是线性扫描(带一些扩展，如 linear scan with intervals)。它的选择和 LuaJIT 类似——追求编译速度。

**V8(TurboFan):图着色(基于 IR 的 register allocator)。** V8 的 TurboFan 用的是更接近图着色的方法(基于 SSA 的 register allocator,带 coalescing)。V8 也是 method JIT,编译整个函数，值得花时间做图着色。

**LuaJIT:线性扫描(反向)。** 前面讲过，选线性扫描是因为 trace 短、线性、JIT 要快。

把这几个放一起，能看出一个规律:**编译单位越大、优化越重，越倾向图着色；编译单位越小、越追求编译速度，越倾向线性扫描。**

| 维度 | LuaJIT(trace) | HotSpot C1(method,快) | HotSpot C2/Graal(method,重) | V8 TurboFan(method) |
|---|---|---|---|---|
| 编译单位 | 一条线性 trace | 整个方法 | 整个方法 | 整个方法 |
| 寄存器分配 | 反向线性扫描 | 线性扫描 | 图着色 / 线性扫描 | 图着色 |
| 编译速度 | 极快 | 快 | 较慢 | 较慢 |
| 代码质量 | 够用 | 够用 | 高 | 高 |
| 活跃区间 | ref 直接编码(SSA) | 显式 interval | 干扰图 | 干扰图 |

LuaJIT 在这张表里是最"轻"的——反向线性扫描，连活跃区间都不用显式存(SSA 的 ref 直接当区间)。这种极简，是 trace JIT 哲学的体现:**用最简单的工具，解决恰好匹配的问题。** trace 的线性结构，让复杂的图着色变成杀鸡用牛刀；反向扫描 + SSA ref,让活跃区间几乎免费。这是 LuaJIT 在"省"这一股张力上的又一次胜利。

还有一个对照点值得提:**deoptimization vs side exit 的寄存器分配影响。** JVM/V8 的 deoptimization(去优化，假设失效时退回解释器),也需要恢复状态。但 JVM 的 deopt 通常是从一个**预先计算的 deoptimization 表**里查"这个位置每个值在哪",这个表的构建和寄存器分配是配合的。LuaJIT 的 snapshot 机制(§4.3 讲的预分配 + 弱引用 + rename 检查)是类似的思想，但更轻量——因为 trace 短,snapshot 也短，处理起来快。两者本质都在解决同一个问题(退出时状态恢复),只是 trace 的线性结构让 LuaJIT 的方案更紧凑。

## §6 回扣主线

这一章我们拆了后端的第一道门槛：寄存器分配。

回到主线"把动态执行安全变成机器码"。IR 是"半成品",机器码是"成品",从半成品到成品，寄存器分配是必经的一步——因为 IR 的世界有无数自由的值，而 CPU 只有十几个物理寄存器，必须做映射。

LuaJIT 选择**反向线性扫描**,这个选择的每一条理由，都呼应着主线的三股张力:

- **快**:线性扫描 O(n log n),比图着色的 O(n²) 快得多，编译开销小(对应"省")。
- **契合 trace**:trace 是线性结构，反向扫描天然契合，活跃区间用 SSA 的 ref 免费编码(对应"省"和"快")。
- **安全**:寄存器分配的正确性有三层保证(每个值有着落、每次访问取对值、snapshot 可恢复),不破坏机器码的正确性(对应"安全")。

特别值得强调的是第三点——寄存器分配不是孤立地追求"少 spill",它还要**和 guard/snapshot 机制精密配合**。snapshot 驱动的预分配、弱引用、rename 检查，都是为了保证"假设失败时能正确退回"。这是 trace JIT 的寄存器分配和普通编译器最大的不同：它不仅要算对，还要**为可能的中断(side exit)做好准备**。这是"乐观假设 + 运行时检查 + 失败可回退"这条主线，在后端的具体落地。

寄存器分配解决了"值放哪"的问题。但机器码生成还有下一步：这些值参与的**运算**,具体翻译成哪些机器指令?加法翻译成 `add` 还是 `lea`?内存访问怎么编码地址?guard 怎么翻译成条件跳转?这些，是下一章的内容。
