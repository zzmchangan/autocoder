# 第 16 章 · 锁:表锁、行锁与死锁

> **前置**:你需要先读过[第 15 章《隔离级别与并发异常》](P4-15-隔离级别与并发异常.md)。上一章讲了**快照**——它让读不阻塞写、写不阻塞读(MVCC 的根基),挡住了脏读、不可重复读、幻读。但快照不是万能的:两个事务要**改同一行**,光靠快照解决不了谁先谁后;一个事务要 `ALTER TABLE` 改表结构,快照也挡不住"一边改结构一边读"的混乱。这些**写写冲突、结构变更冲突**,必须靠**锁**来排队。本章就把 PG 的锁体系从顶到底拆开。

> **核心问题**:PG 的锁分哪几类、各保护什么?表锁为什么有 **8 个强度级别**,它们之间谁和谁冲突?给某一行加锁时,为什么先要给整张表加一把"意向锁(IS/IX)"——多粒度锁到底解决什么问题?**PG 的行锁和 MySQL 有什么本质不同**(关键:PG 行锁不在数据页上留物理锁标记,而是借助 `xmax` 标志位 + 共享的多粒度锁表实现,所以行锁不会撑大表、崩溃也不会留下脏锁)?**死锁**怎么产生、PG 怎么检测(`deadlock.c` 的等待图找环)、检测到后怎么破?

> **读完本章你会明白**:PG 有三类性质完全不同的锁——**常规锁(heavyweight lock)**保护表/对象等"逻辑资源"、**轻量锁(LWLock)**保护内存里的共享数据结构、**行锁**用 MVCC 的 `xmax` 标记实现;表锁的 8 级构成一张冲突矩阵(本质是张规则表);多粒度锁用意向锁解决"表锁和行锁互相感知"的问题;死锁靠"等待图找环 + 延时检测"破解,检测到后回滚代价较小那个事务。读完这章,你对"并发写怎么正确串行化"就有了完整的内核图景。

> **如果一读觉得太难**:先只记住三件事——① 锁分三类:常规锁(到事务结束)、轻量锁(临界区,极短)、行锁(到事务结束,但靠 xmax 标记,开销极小);② 表锁 8 级的核心是冲突矩阵,`AccessShareLock`(读)最弱、`AccessExclusiveLock`(改结构)最强,互斥关系查表即可;③ 死锁就是"A 等 B、B 等 A"的环,PG 默认等 1 秒(`deadlock_timeout`)就开始检测,检测到就回滚自己。其余细节第二遍配合源码再抠。

## 一、先看清"锁"在解决什么:快照管不了的冲突

第 15 章我们讲快照解决了"读写并发"——读者看旧版本,写者写新版本,各取所需。但你立刻会发现:有些冲突快照根本管不着。

### 快照管不了的两种冲突

**冲突一:写写冲突。** 两个事务同时要 `UPDATE` 同一行。

```text
事务 A                                事务 B
BEGIN;                                BEGIN;
UPDATE accounts SET balance=balance-100
        WHERE id=1;   -- A 要改 id=1
                                      UPDATE accounts SET balance=balance+500
                                              WHERE id=1;   -- B 也要改 id=1
COMMIT;                               COMMIT;
```

A 和 B 都基于"旧值 100"算新值——A 算成 0、B 算成 600。最后哪个落地?如果两个都落地,**其中一个的改动就丢了**(这叫**丢失更新 lost update**,第 15 章提过)。快照挡不住它:快照只决定"读谁",决定不了"两个写谁先谁后"。这里必须有人排队——要么 A 先改完 B 再改,要么 B 先 A 后。

**冲突二:结构变更冲突。** 一个事务在读一张表,另一个事务在 `DROP` 这张表。

```text
事务 A                                事务 B
BEGIN;                                BEGIN;
SELECT * FROM big_table;              DROP TABLE big_table;
-- A 正在扫这张表                     -- B 要把表整个删掉
COMMIT;                               COMMIT;
```

如果 B 删表时 A 还在扫,结果就是"A 扫到一半表没了",数据错乱。快照能挡住这个吗?不能——`DROP` 是改**表结构**和**文件**,不是改某一行数据,快照(管的是数据版本可见性)管不着。

> **不这样会怎样**:如果没有锁来排队这些冲突,后果是灾难性的:丢失更新让数据悄悄变错(账面少钱、库存超卖),结构变更中途读会让查询崩溃或读到残缺数据。这两类冲突,必须有一个机制让它们**串行化**(谁先谁后,而不是同时发生)。这个机制就是**锁**。

### 锁的本质:把"本会同时发生"的操作,强制排队

锁的核心思想非常朴素:**给一个资源(表、行、内存结构)加上一把"门闩",谁要动它,先看门闩在不在;在,就等;不在,自己挂上,做完摘掉**。两把门闩如果互斥(比如一把"写闩"、一把"独占改结构闩"),后到的人就得排队。

锁和快照的分工可以这样记:

| | 快照(MVCC) | 锁 |
|---|---|---|
| 解决什么 | 读写并发(读不阻塞写、写不阻塞读) | 写写冲突、结构变更冲突、并发改内存结构 |
| 代价 | 旧行不立刻消失(要 VACUUM) | 持有期间会挡住别人(并发度下降) |

PG 的大部分读写并发靠 MVCC(下一章详讲),**锁主要用来管写写和结构**。但锁不止一种,PG 内部有**三类性质完全不同**的锁,服务完全不同的对象。下面我们一个个看。

## 二、三类锁:常规锁、轻量锁、行锁

这是本章最重要的一个分类,先把三者摆到桌面上对比清楚。很多人混不清"锁",根子就是没分清这三类。

| 锁类 | 英文 | 保护什么 | 持续多久 | 实现机制 | 例子 |
|---|---|---|---|---|---|
| **常规锁** | regular lock / heavyweight lock | 逻辑资源:表、对象、行(的强锁) | 一般**到事务结束**才释放 | 共享锁表(`LOCK`/`PROCLOCK`/`LOCALLOCK`)+ 等待队列 | `SELECT` 拿 `AccessShareLock`,`ALTER TABLE` 拿 `AccessExclusiveLock` |
| **轻量锁** | lightweight lock / LWLock | 内存里的共享数据结构(BufferDesc、锁表本身、控制结构) | **极短的临界区**,用完即放 | 自旋(spin)+ 等待队列,基于原子操作 | `BufferDesc` 的 `io_in_progress` 锁、保护 `ProcArray` 的 `ProcArrayLock` |
| **行锁** | tuple lock | 单行数据(的"有人在改这行"标记) | 到事务结束 | 借 `xmax` 标志位 + 多粒度锁表,**不在数据页留物理锁** | `SELECT FOR UPDATE`、`UPDATE`/`DELETE` 隐式加的行锁 |

这三类锁的**设计哲学完全不同**,逐个讲。

### 1. 常规锁:保护逻辑资源,持续到事务结束

常规锁是"读者最熟悉的那个锁"——它就是你 `LOCK TABLE` 或者 `SELECT FOR UPDATE` 时拿到的锁,也是 PG 内部自动加的表锁。它的特点:

- **保护的是逻辑资源**(表、对象、事务等),而不是某个内存地址。
- **持续到事务结束**(`COMMIT`/`ROLLBACK` 时统一释放),除非是显式的 session lock。
- **有 8 个强度级别**(下一节详讲),级别之间有"兼容/冲突"的关系。
- 实现上用**共享内存里的三张哈希表**(`LOCK` 表 + `PROCLOCK` 表 + 每个 backend 的 `LOCALLOCK` 表)管理,本章后半会精读。

常规锁是本章的主角——"表锁的 8 级"、"多粒度锁"、"死锁检测"主要都发生在它身上。

### 2. 轻量锁(LWLock):保护内存结构,用完就放

轻量锁是另一个世界。它保护的是**内存里的共享数据结构**——比如 Buffer Pool 里某个 buffer 的描述符(`BufferDesc`)、全局的事务数组(`ProcArray`)、锁表自己的哈希分区锁。它的特点和常规锁截然相反:

- **持有时间极短**:只在访问某个内存结构的"临界区"内持有,代码段结束立刻释放,**不会持续到事务结束**(否则就退化成常规锁了)。
- **不参与死锁检测**(后面讲为什么),靠"严格的使用顺序约定"避免死锁。
- **没有 8 个级别**,只有三种模式:`LW_SHARED`(共享,多人可读)、`LW_EXCLUSIVE`(排他,独占)、`LW_WAIT_UNTIL_FREE`(等到没人持有就退出,不真锁)。

看它的定义:

> [src/include/storage/lwlock.h:41-50](../postgresql-17.0/src/include/storage/lwlock.h#L41-L50)

```c
typedef struct LWLock
{
	uint16		tranche;		/* tranche ID */
	pg_atomic_uint32 state;		/* state of exclusive/nonexclusive lockers */
	proclist_head waiters;		/* list of waiting PGPROCs */
#ifdef LOCK_DEBUG
	pg_atomic_uint32 nwaiters;	/* number of waiters */
	struct PGPROC *owner;		/* last exclusive owner of the lock */
#endif
} LWLock;
```

`state` 是一个原子变量,记录"现在有几个共享持有者、有没有排他持有者"。加锁时,先用原子操作试着改 `state`(共享锁是给计数 +1,排他锁是看计数是否为 0),改成功就拿到;改不成就**自旋(spin)一会儿**再试,自旋够了就把自己挂到 `waiters` 队列上睡眠,等持有者释放时被唤醒。

> [src/include/storage/lwlock.h:112-119](../postgresql-17.0/src/include/storage/lwlock.h#L112-L119)

```c
typedef enum LWLockMode
{
	LW_EXCLUSIVE,
	LW_SHARED,
	LW_WAIT_UNTIL_FREE,			/* A special mode used in PGPROC->lwWaitMode,
								 * not expected to be used by Callers. */
} LWLockMode;
```

**为什么轻量锁不参与死锁检测?** 因为它持有时间太短(临界区,几微秒),如果为它跑全套死锁检测(要锁住所有锁分区、构建等待图、找环),开销比临界区本身还大。所以 PG 对轻量锁采取的策略是:**靠代码约定加锁顺序来避免死锁**(比如"拿多把 LWLock 时,按地址升序拿"),而不是检测。这是设计上的权衡——轻量锁要快,所以不能背死锁检测的包袱。

> **一个关键的联动**:常规锁的内部实现,**自己就大量使用轻量锁**。比如锁表被分成 16 个分区(`NUM_LOCK_PARTITIONS = 16`),每个分区由一把轻量锁保护。当 backend 要改锁表(加锁、释放锁),先拿对应分区的 LWLock 排他锁,改完释放。这种"用轻量锁保护共享结构"的模式在 PG 内核里无处不在。

### 3. 行锁:PG 的特色——靠 xmax 标记,不在数据页留物理锁

行锁是 PG 最有特色、也最容易被误解的部分。先说结论:

> **PG 的行锁,本质不是"在行上加一个物理锁标记",而是借助 MVCC 的 `xmax` 字段 + 多粒度锁表实现的"逻辑锁"。** 这和 MySQL InnoDB(在数据页/行存锁信息)是根本不同的设计。

具体说,PG 的行锁分两层落地:

**第一层:行上的 `xmax` 标志位。** 当一个事务对某行加 `SELECT FOR UPDATE` / `SELECT FOR SHARE` 这种"强行锁"(或 UPDATE/DELETE 隐式加的锁),PG 会在这一行的 tuple header 里写下**自己的事务号到 `xmax`**,并打上几个标志位,表明"这一行被某个事务锁住了,锁的强度是几级"。

这些标志位定义在 tuple header 的 `t_infomask` 里:

> [src/include/access/htup_details.h:194-203](../postgresql-17.0/src/include/access/htup_details.h#L194-L203)

```c
#define HEAP_XMAX_KEYSHR_LOCK	0x0010	/* xmax is a key-shared locker */
...
#define HEAP_XMAX_EXCL_LOCK		0x0040	/* xmax is exclusive locker */
#define HEAP_XMAX_LOCK_ONLY		0x0080	/* xmax, if valid, is only a locker */

 /* xmax is a shared locker */
#define HEAP_XMAX_SHR_LOCK	(HEAP_XMAX_EXCL_LOCK | HEAP_XMAX_KEYSHR_LOCK)

#define HEAP_LOCK_MASK	(HEAP_XMAX_SHR_LOCK | HEAP_XMAX_EXCL_LOCK | \
						 HEAP_XMAX_KEYSHR_LOCK)
```

逐个拆:

- **`HEAP_XMAX_LOCK_ONLY`**:最关键的一个标志。它的意思是"`xmax` 这个字段记的不是'删除/更新这一行的事务',而是'仅仅锁住这一行的事务'"。这是行锁和 UPDATE/DELETE 的分界线——UPDATE/DELETE 会让旧行的 `xmax` 指向新版本的事务(伴随 `HEAP_UPDATED` 标志),而**纯行锁(`SELECT FOR UPDATE` 这种)只设 `LOCK_ONLY`,不产生新版本**。
- **`HEAP_XMAX_KEYSHR_LOCK`**:key-shared 锁(`SELECT FOR KEY SHARE`,最弱的强锁,允许别人改非键列)。
- **`HEAP_XMAX_EXCL_LOCK`**:exclusive 锁(`SELECT FOR UPDATE`、UPDATE 改键列、DELETE)。
- **`HEAP_XMAX_SHR_LOCK` = `EXCL_LOCK | KEYSHR_LOCK`**:shared 锁(`SELECT FOR SHARE`),用两个位的组合表示。

有了这三个标志位,一行被谁锁、锁多强,就**全写在 tuple 自己的 header 里**了。判定一个 tuple 是不是"被锁住",直接查这几个位即可。PG 还提供了便捷宏:

> [src/include/access/htup_details.h:231-233](../postgresql-17.0/src/include/access/htup_details.h#L231-L233)

```c
#define HEAP_XMAX_IS_LOCKED_ONLY(infomask) \
	(((infomask) & HEAP_XMAX_LOCK_ONLY) || \
	 (((infomask) & (HEAP_XMAX_IS_MULTI | HEAP_LOCK_MASK)) == HEAP_XMAX_EXCL_LOCK))
```

**第二层:多粒度锁表的"强锁"。** `xmax` 标志位只能记"单一事务锁了这一行"。但如果**多个事务**要同时锁同一行(比如两个 `SELECT FOR SHARE`),`xmax` 只有一个槽位怎么办?PG 用一个叫 **MultiXact** 的机制:把"多个锁持有者"打包成一个 MultiXactId 存到 `xmax`(配合 `HEAP_XMAX_IS_MULTI` 标志),真正的持有者列表存到共享的 multixact 表里。这是进阶细节,本章点到为止。

此外,对于 `SELECT FOR UPDATE` 这种**需要排队的强行锁**,PG 还会走常规锁的 `LockTuple`(`SET_LOCKTAG_TUPLE`),在共享锁表里登记一把 tuple 级的常规锁。这是为了**等待和唤醒**:当一行被锁住,后来者要能"睡在"这把锁上,被唤醒——这个等待/唤醒机制,常规锁有,纯 `xmax` 标志位没有。

看 `LockTuple` 的封装:

> [src/backend/storage/lmgr/lmgr.c:557-569](../postgresql-17.0/src/backend/storage/lmgr/lmgr.c#L557-L569)

```c
/*
 * Obtain a tuple-level lock.  This is used in a less-than-intuitive fashion
 * because we can't afford to keep a separate lock in shared memory for every
 * tuple.  See heap_lock_tuple before using this!
 */
void
LockTuple(Relation relation, ItemPointer tid, LOCKMODE lockmode)
{
	LOCKTAG		tag;

	SET_LOCKTAG_TUPLE(tag,
					  relation->rd_lockInfo.lockRelId.dbId,
					  relation->rd_lockInfo.lockRelId.relId,
					  ItemPointerGetBlockNumber(tid),
					  ItemPointerGetOffsetNumber(tid));

	(void) LockAcquire(&tag, lockmode, false, false);
}
```

注意这条注释——**"we can't afford to keep a separate lock in shared memory for every tuple"**。这是 PG 行锁设计的核心出发点:**不可能给每行都在共享内存里留一把锁**(一张表几亿行,锁表会被撑爆)。所以 PG 的策略是:绝大多数行锁,只在 `xmax` 标志位记一下(开销接近零,就是几个 bit);只有真正需要"等待/唤醒"的强行锁,才在共享锁表里登记,而且**只在冲突发生时**(后来者要排队),才升级到锁表登记。

> **这带来的三个红利**(为下一章 MVCC 埋伏笔):
> 1. **行锁不会撑大表**:锁信息就藏在 tuple header 的几个 bit 里,不需要额外的锁空间。
> 2. **崩溃不会留下脏锁**:行锁的"持有者"是事务号(`xmax`),事务回滚了,`xmax` 标记自然失效;不像某些数据库会在数据页留持久锁标记,崩溃后还得专门清理。
> 3. **行锁和 MVCC 天然耦合**:行锁用的 `xmax`、`HEAP_XMAX_*` 标志,正是 MVCC 用来追踪"这一行被谁改过"的同一套字段。锁和版本,在 PG 里共用一套机制——这是它和 InnoDB 最深的区别。

行锁有四种强度,对应 SQL 的四种锁定子句:

> [src/include/nodes/lockoptions.h:49-59](../postgresql-17.0/src/include/nodes/lockoptions.h#L49-L59)

```c
typedef enum LockTupleMode
{
	/* SELECT FOR KEY SHARE */
	LockTupleKeyShare,
	/* SELECT FOR SHARE */
	LockTupleShare,
	/* SELECT FOR NO KEY UPDATE, and UPDATEs that don't modify key columns */
	LockTupleNoKeyExclusive,
	/* SELECT FOR UPDATE, UPDATEs that modify key columns, and DELETE */
	LockTupleExclusive,
} LockTupleMode;
```

从弱到强:`KEY SHARE`(允许别人改非键列)→ `SHARE`(只读共享)→ `NO KEY UPDATE`(改非键列)→ `EXCLUSIVE`(改键列/删除,最强)。为什么要有"是否改键"的区分?因为**外键检查只需要锁键列**——一个事务 `SELECT FOR KEY SHARE` 锁住父表的一行(防止它被删/改键),子表就能安全地引用它;而另一个事务想改这一行的非键列(比如改个备注),完全不冲突。这种细分让并发度更高。

行锁的具体落地在 `heap_lock_tuple`(显式 `SELECT FOR SHARE/UPDATE`)和 `heap_update`/`heap_delete`(隐式行锁)里:

> [src/backend/access/heap/heapam.c:4310-4313](../postgresql-17.0/src/backend/access/heap/heapam.c#L4310-L4313)

```c
heap_lock_tuple(Relation relation, HeapTuple tuple,
				CommandId cid, LockTupleMode mode, LockWaitPolicy wait_policy,
				bool follow_updates,
				Buffer *buffer, TM_FailureData *tmfd)
```

它做的事:读 buffer → 加 buffer 排他轻量锁 → 检查这行的 `HeapTupleSatisfiesUpdate` 状态 → 如果没冲突,在 tuple header 写 `xmax` + 锁标志位 → 释放 buffer 锁。整个过程的"锁"开销极小(就是改几个 bit),这也是为什么 PG 的行锁性能很好。

三类锁讲完,做个收束:

> **常规锁**保护逻辑资源(到事务结束,有 8 级,参与死锁检测);**轻量锁**保护内存结构(临界区,极短,靠加锁顺序避死锁);**行锁**保护单行数据(到事务结束,但靠 `xmax` 标志位 + 必要时升级到锁表,开销极小)。本章后面讲的"8 级表锁"、"多粒度"、"死锁",主要发生在常规锁上。

## 三、表锁的 8 个级别:一张冲突矩阵

现在专门讲常规锁里最常用的一种——**表锁(relation lock)**。这是你写 SQL 时最常碰到的锁。

### 为什么有 8 级,而不是简单的"读锁/写锁"两级

很多系统只有两级锁:共享锁(读,S)和排他锁(写,X)。但数据库的并发场景丰富得多:

- `SELECT` 只读,几个 `SELECT` 一起跑完全没问题——需要**最弱的读锁**。
- `INSERT/UPDATE/DELETE` 改数据,但改的是行,不是表结构——几个写操作同时改一张表的不同行,也应该允许——需要**行级排他锁**(强度中等)。
- `CREATE INDEX` 要重建索引,不能有别人在改数据,但可以有读——需要**共享锁**。
- `ALTER TABLE / DROP TABLE` 改表结构,这时连读都不能有——需要**最强的排他锁**。

如果只有两级,要么太松(允许了不该允许的并发,出错),要么太紧(挡住了本可以并发的操作,性能差)。**8 级就是在这道"松紧光谱"上切出的精细刻度**,让每种操作拿"刚好够用"的锁,不挡住不必要的并发。

这 8 级用宏定义,数字就是强度(数字越大越强):

> [src/include/storage/lockdefs.h:33-48](../postgresql-17.0/src/include/storage/lockdefs.h#L33-L48)

```c
/* NoLock is not a lock mode, but a flag value meaning "don't get a lock" */
#define NoLock					0

#define AccessShareLock			1	/* SELECT */
#define RowShareLock			2	/* SELECT FOR UPDATE/FOR SHARE */
#define RowExclusiveLock		3	/* INSERT, UPDATE, DELETE */
#define ShareUpdateExclusiveLock 4	/* VACUUM (non-FULL), ANALYZE, CREATE
									 * INDEX CONCURRENTLY */
#define ShareLock				5	/* CREATE INDEX (WITHOUT CONCURRENTLY) */
#define ShareRowExclusiveLock	6	/* like EXCLUSIVE MODE, but allows ROW
									 * SHARE */
#define ExclusiveLock			7	/* blocks ROW SHARE/SELECT...FOR UPDATE */
#define AccessExclusiveLock		8	/* ALTER TABLE, DROP TABLE, VACUUM FULL,
									 * and unqualified LOCK TABLE */

#define MaxLockMode				8	/* highest standard lock mode */
```

每级的注释直接告诉你"谁用它",我们逐个过(这是本章最该背的一张表):

| 级别 | 宏 | 谁用它 | 强度直觉 |
|---|---|---|---|
| 1 | `AccessShareLock` | **所有 `SELECT`** | 最弱,只读,几个 SELECT 共存 |
| 2 | `RowShareLock` | `SELECT FOR UPDATE/FOR SHARE` | 行级强锁,但允许别人也来读 |
| 3 | `RowExclusiveLock` | `INSERT/UPDATE/DELETE` | 改数据,允许别人也改不同行 |
| 4 | `ShareUpdateExclusiveLock` | `VACUUM`(非 FULL)、`ANALYZE`、`CREATE INDEX CONCURRENTLY` | 维护操作,和它自己冲突 |
| 5 | `ShareLock` | `CREATE INDEX`(非 CONCURRENTLY) | 建索引,挡写不挡读 |
| 6 | `ShareRowExclusiveLock` | 较少用,类似 EXCLUSIVE 但允许 ROW SHARE | — |
| 7 | `ExclusiveLock` | 只允许读,挡掉所有写和 FOR UPDATE | 很强,只读能进 |
| 8 | `AccessExclusiveLock` | `ALTER TABLE/DROP TABLE/VACUUM FULL` | 最强,挡掉一切,包括读 |

### 冲突矩阵:谁和谁互斥

光知道每级是谁用还不够,关键是**哪两级会冲突(后到的要等)**。这是规则表,直球讲——本质上就是一张位掩码表。

PG 把这张表叫 `LockConflicts`,定义在 `lock.c` 开头:

> [src/backend/storage/lmgr/lock.c:64-104](../postgresql-17.0/src/backend/storage/lmgr/lock.c#L64-L104)

```c
static const LOCKMASK LockConflicts[] = {
	0,

	/* AccessShareLock */
	LOCKBIT_ON(AccessExclusiveLock),

	/* RowShareLock */
	LOCKBIT_ON(ExclusiveLock) | LOCKBIT_ON(AccessExclusiveLock),

	/* RowExclusiveLock */
	LOCKBIT_ON(ShareLock) | LOCKBIT_ON(ShareRowExclusiveLock) |
	LOCKBIT_ON(ExclusiveLock) | LOCKBIT_ON(AccessExclusiveLock),

	/* ShareUpdateExclusiveLock */
	LOCKBIT_ON(ShareUpdateExclusiveLock) |
	LOCKBIT_ON(ShareLock) | LOCKBIT_ON(ShareRowExclusiveLock) |
	LOCKBIT_ON(ExclusiveLock) | LOCKBIT_ON(AccessExclusiveLock),

	/* ShareLock */
	LOCKBIT_ON(RowExclusiveLock) | LOCKBIT_ON(ShareUpdateExclusiveLock) |
	LOCKBIT_ON(ShareRowExclusiveLock) |
	LOCKBIT_ON(ExclusiveLock) | LOCKBIT_ON(AccessExclusiveLock),

	/* ShareRowExclusiveLock */
	LOCKBIT_ON(RowExclusiveLock) | LOCKBIT_ON(ShareUpdateExclusiveLock) |
	LOCKBIT_ON(ShareLock) | LOCKBIT_ON(ShareRowExclusiveLock) |
	LOCKBIT_ON(ExclusiveLock) | LOCKBIT_ON(AccessExclusiveLock),

	/* ExclusiveLock */
	LOCKBIT_ON(RowShareLock) |
	LOCKBIT_ON(RowExclusiveLock) | LOCKBIT_ON(ShareUpdateExclusiveLock) |
	LOCKBIT_ON(ShareLock) | LOCKBIT_ON(ShareRowExclusiveLock) |
	LOCKBIT_ON(ExclusiveLock) | LOCKBIT_ON(AccessExclusiveLock),

	/* AccessExclusiveLock */
	LOCKBIT_ON(AccessShareLock) | LOCKBIT_ON(RowShareLock) |
	LOCKBIT_ON(RowExclusiveLock) | LOCKBIT_ON(ShareUpdateExclusiveLock) |
	LOCKBIT_ON(ShareLock) | LOCKBIT_ON(ShareRowExclusiveLock) |
	LOCKBIT_ON(ExclusiveLock) | LOCKBIT_ON(AccessExclusiveLock)

};
```

`LOCKBIT_ON(mode)` 是 `1 << mode`,即把锁级别转成一个 bit。`LockConflicts[i]` 是一个位掩码,**第 j 位为 1 就表示"级别 i 和级别 j 冲突"**。比如 `LockConflicts[AccessShareLock] = LOCKBIT_ON(AccessExclusiveLock)`——意思是"`AccessShareLock` 只和 `AccessExclusiveLock` 冲突"。读这表的规则:**查第 i 行,看哪些 bit 亮着,亮的就是和它冲突的级别**。

把这张位掩码表翻译成易读的冲突矩阵(✗ = 冲突,✓ = 兼容),行是"已持有的锁"、列是"新请求的锁":

| 已持\新请求 | AS(1) | RS(2) | RE(3) | SUE(4) | S(5) | SRE(6) | E(7) | AE(8) |
|---|---|---|---|---|---|---|---|---|
| **AccessShareLock**(1) | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✗ |
| **RowShareLock**(2) | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✗ | ✗ |
| **RowExclusiveLock**(3) | ✓ | ✓ | ✓ | ✓ | ✗ | ✗ | ✗ | ✗ |
| **ShareUpdateExclusiveLock**(4) | ✓ | ✓ | ✓ | ✗ | ✗ | ✗ | ✗ | ✗ |
| **ShareLock**(5) | ✓ | ✓ | ✗ | ✗ | ✓ | ✗ | ✗ | ✗ |
| **ShareRowExclusiveLock**(6) | ✓ | ✓ | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ |
| **ExclusiveLock**(7) | ✓ | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ |
| **AccessExclusiveLock**(8) | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ |

读这张表的几个关键直觉:

1. **`AccessShareLock`(读锁)几乎不挡任何人**,只和最强的 `AccessExclusiveLock`(改表结构)冲突。这就是为什么 `SELECT` 几乎不会被阻塞——除非有人正在 `DROP TABLE`。
2. **`AccessExclusiveLock`(最强)挡掉一切**,包括其他 `SELECT`。所以 `ALTER TABLE`、`DROP TABLE` 时,整张表谁都碰不了。
3. **两个 `RowExclusiveLock` 兼容**——这就是为什么两个 `UPDATE` 改同一张表的不同行可以并发(表锁层面不冲突,真正冲突的行再靠行锁解决)。
4. **`ShareUpdateExclusiveLock` 和它自己冲突**(`LockConflicts[4]` 的第 4 位亮)——这是为什么两个 `VACUUM` 不能同时跑在同一张表上。
5. **从左上到右下,冲突越来越多**——级别越高,挡的人越多。这正是"强度"的含义。

> **为什么这张表是位掩码而不是函数?** 因为冲突判定是锁系统里**最高频的操作**(每次加锁都要查),必须极快。位掩码的"按位与"(`conflictTab[mode] & grantMask`)是一条 CPU 指令,O(1)。这是性能驱动的数据结构选择——回扣全书主线"让数据快"。

判定冲突的代码就在 `LockCheckConflicts` 里,核心就是这一句:

> [src/backend/storage/lmgr/lock.c:1451-1455](../postgresql-17.0/src/backend/storage/lmgr/lock.c#L1451-L1455)

```c
	if (!(conflictMask & lock->grantMask))
	{
		PROCLOCK_PRINT("LockCheckConflicts: no conflict", proclock);
		return false;
	}
```

`conflictMask` 是新请求锁的冲突掩码(从 `LockConflicts[mode]` 查),`lock->grantMask` 是当前这把锁上**已经授予的所有锁级别的并集掩码**。两者按位与——如果结果非零,说明有冲突;为零,说明完全兼容,直接授予。这就是冲突判定的全部:**一次按位与**。

## 四、多粒度锁:意向锁为什么必要

讲完 8 级表锁,你可能有个疑问:**表锁这么粗,那行锁呢?** 一个事务只想锁一行,另一个事务想锁整张表——它们怎么互相感知?这就引出**多粒度锁(multi-granularity locking)**和**意向锁(intention lock)**。

### 问题:表锁和行锁怎么互相发现

假设没有意向锁,看看会出什么乱子:

```text
事务 A                                事务 B
BEGIN;                                BEGIN;
-- A 想给 id=1 这一行加 X 锁         -- B 想给整张表加 S 锁(比如建索引)
SELECT ... FOR UPDATE WHERE id=1;     LOCK TABLE accounts IN SHARE MODE;
COMMIT;                               COMMIT;
```

A 在某一行上加了排他锁,B 想给整张表加共享锁。**这两个是冲突的**——B 的表级 S 锁意味着"我要读整张表的所有行",而 A 锁的某一行正在被改,B 不应该和它共存。

> **不这样会怎样**:如果 B 不知道 A 在某一行上有锁,直接授予了表级 S 锁,那 B 就会"看到"A 正在改的那一行的中间状态(或者 B 建的索引会漏掉 A 的改动)——数据不一致。所以 B 在拿表级 S 锁之前,**必须先确认"表里没有任何一行被排他锁住"**。

但"扫遍表里所有行的锁"是不现实的——表可能有几亿行。怎么让 B 高效地知道"A 在某行上有锁"?

### 解法:意向锁(IS/IX)——表上的"占位符"

PG(和 SQL 标准)的做法是:**事务在给某一行加锁之前,先在表上挂一把"意向锁"**。意向锁不真的锁表,它只是一个**信号**:"我在这个表里有行级锁,你想锁整张表,得先过我这一关。"

意向锁有两种(对应行锁的两种基本模式):

- **IS(Intention Share,意向共享)**:我准备给某些行加共享锁(`SELECT FOR SHARE`)。
- **IX(Intention Exclusive,意向排他)**:我准备给某些行加排他锁(`SELECT FOR UPDATE`、`UPDATE`、`DELETE`)。

在 PG 的 8 级表锁里,`RowShareLock` 充当 IS 的角色,`RowExclusiveLock` 充当 IX 的角色——这也解释了它们名字里"Row"的含义:**"我准备在行上做事,先在表上挂个意向"**。

现在再看刚才的例子:

1. A 要给 `id=1` 加行级 X 锁,先给表加 `RowExclusiveLock`(=IX)。再给行加 X 锁。
2. B 要给表加 S 锁(`ShareLock`),查冲突矩阵:`ShareLock` 和 `RowExclusiveLock`(IX)**冲突**(看上面的矩阵,行 `ShareLock`(5) 和列 `RowExclusiveLock`(3) 是 ✗)。所以 B 被挡住,等 A 释放。

**意向锁让"表锁"和"行锁"通过表级的意向锁互相感知**,不需要扫所有行。这是多粒度锁的核心价值。

### 多粒度锁的完整层级

PG 的锁其实有多个粒度(从粗到细):

## 五、常规锁的内部:三张哈希表 + 等待队列

现在钻进常规锁的内部实现。PG 用**三张哈希表**管理常规锁,这是本章最核心的数据结构。

### 为什么是三张表,不是一张

直觉上,一把锁就该有一个对象存着。但 PG 拆成了三张,各有分工:

1. **共享的 `LOCK` 表**(在共享内存):每个"被锁的对象"一条记录。key 是 `LOCKTAG`,记录"这个对象上现在有哪些级别的锁被持有、有哪些在等待"。
2. **共享的 `PROCLOCK` 表**(在共享内存):每个"(对象, 持有者)"组合一条记录。因为同一个对象可能被多个 backend 同时持有(比如多个 SELECT),需要记"谁持有了什么"。
3. **每个 backend 私有的 `LOCALLOCK` 表**(在后端进程内存):本地缓存,记"我自己加过哪些锁、加了几次"。这是性能优化——避免每次查锁都去碰共享内存。

三张表的定义:

`LOCK`(被锁对象的全局状态):

> [src/include/storage/lock.h:308-322](../postgresql-17.0/src/include/storage/lock.h#L308-L322)

```c
typedef struct LOCK
{
	/* hash key */
	LOCKTAG		tag;			/* unique identifier of lockable object */

	/* data */
	LOCKMASK	grantMask;		/* bitmask for lock types already granted */
	LOCKMASK	waitMask;		/* bitmask for lock types awaited */
	dlist_head	procLocks;		/* list of PROCLOCK objects assoc. with lock */
	dclist_head waitProcs;		/* list of PGPROC objects waiting on lock */
	int			requested[MAX_LOCKMODES];	/* counts of requested locks */
	int			nRequested;		/* total of requested[] array */
	int			granted[MAX_LOCKMODES]; /* counts of granted locks */
	int			nGranted;		/* total of granted[] array */
} LOCK;
```

逐字段:

- **`tag`**:这个锁对应的对象(哪张表、哪个 tuple)。
- **`grantMask`**:已经授予的锁级别的并集掩码(查冲突时和它做按位与)。
- **`waitMask`**:正在等待的锁级别的并集掩码(新来者要先和它比,避免"插队",保证公平)。
- **`procLocks`**:所有持有/等待这个对象的 PROCLOCK 记录组成的链表。
- **`waitProcs`**:正在等待的 PGPROC(进程)组成的队列——唤醒时从这里挑。
- **`requested[]` / `granted[]`**:按级别细分的"请求数 / 授予数"计数。

`PROCLOCK`(单个持有者在这个对象上的状态):

> [src/include/storage/lock.h:369-380](../postgresql-17.0/src/include/storage/lock.h#L369-L380)

```c
typedef struct PROCLOCK
{
	/* tag */
	PROCLOCKTAG tag;			/* unique identifier of proclock object */

	/* data */
	PGPROC	   *groupLeader;	/* proc's lock group leader, or proc itself */
	LOCKMASK	holdMask;		/* bitmask for lock types currently held */
	LOCKMASK	releaseMask;	/* bitmask for lock types to be released */
	dlist_node	lockLink;		/* list link in LOCK's list of proclocks */
	dlist_node	procLink;		/* list link in PGPROC's list of proclocks */

```

`holdMask` 是"这个 backend 在这个对象上持有了哪些级别的锁"的掩码。一条 PROCLOCK 把"一个 backend + 一个对象"绑定起来。

`LOCALLOCK`(backend 本地缓存):

> [src/include/storage/lock.h:426-441](../postgresql-17.0/src/include/storage/lock.h#L426-L441)

```c
typedef struct LOCALLOCK
{
	/* tag */
	LOCALLOCKTAG tag;			/* unique identifier of locallock entry */

	/* data */
	uint32		hashcode;		/* copy of LOCKTAG's hash value */
	LOCK	   *lock;			/* associated LOCK object, if any */
	PROCLOCK   *proclock;		/* associated PROCLOCK object, if any */
	int64		nLocks;			/* total number of times lock is held */
	int			numLockOwners;	/* # of relevant ResourceOwners */
	int			maxLockOwners;	/* allocated size of array */
	LOCALLOCKOWNER *lockOwners; /* dynamically resizable array */
	bool		holdsStrongLockCount;	/* bumped FastPathStrongRelationLocks */
	bool		lockCleared;	/* we read all sinval msgs for lock */
} LOCALLOCK;
```

关键在 `nLocks`:它记的是**"这个 backend 在这个对象上,以这个模式加过几次锁"**。为什么需要它?因为同一个事务里,可能多次加同一把锁(比如嵌套函数调用都 `LockRelation`)。如果每次都去共享内存操作,太贵。所以 `LOCALLOCK` 让"重复加同一把锁"变成**本地计数 +1**(`nLocks++`),根本不碰共享内存。只有第一次加、和最后一次释放,才去共享内存操作。

> **这就是为什么 `LOCALLOCK` 是性能优化**:它让"已持有的锁再加一次"变成 O(1) 的本地操作,绕开共享内存的 LWLock 争用。在海量并发小事务的场景下,这至关重要。

### 加一把锁的全过程:`LockAcquireExtended`

把这些串起来,看 `LockAcquireExtended` 的核心流程:

它的主要步骤(我们挑关键的看):

**步骤二:走 fast-path(快路径)优化。**

```c
		if (FastPathStrongRelationLocks->count[fasthashcode] != 0)
			acquired = false;
		else
			acquired = FastPathGrantRelationLock(locktag->locktag_field2,
												 lockmode);
		LWLockRelease(&MyProc->fpInfoLock);
```

**fast-path 是一个针对"弱表锁"的进一步优化**。它的思路:大多数表锁是 `AccessShareLock`(SELECT)或 `RowExclusiveLock`(INSERT/UPDATE/DELETE),这些锁很少冲突。为它们在共享内存建 LOCK/PROCLOCK 条目太浪费。所以 PG 给每个 backend 在自己的 `PGPROC` 里留了 **16 个 fast-path 槽位**(`FP_LOCK_SLOTS_PER_BACKEND = 16`),把这种弱锁直接记在本地,根本不入共享锁表。

fast-path 的资格条件:

> [src/backend/storage/lmgr/lock.c:213-218](../postgresql-17.0/src/backend/storage/lmgr/lock.c#L213-L218)

```c
#define EligibleForRelationFastPath(locktag, mode) \
	((locktag)->locktag_lockmethodid == DEFAULT_LOCKMETHOD && \
	(locktag)->locktag_type == LOCKTAG_RELATION && \
	(locktag)->locktag_field1 == MyDatabaseId && \
	MyDatabaseId != InvalidOid && \
	(mode) < ShareUpdateExclusiveLock)
```

只对"本数据库内的 relation 锁,且强度 < `ShareUpdateExclusiveLock`(即只对 AS/RS/RE 三级)"开放。当某个 backend 想加强锁(≥ SUE),它得先扫描所有 backend 的 fast-path 槽,把可能相关的弱锁"提升"到共享锁表,再继续——这就是 `FastPathStrongRelationLocks` 计数器的作用(标志某个分区有没有强锁在进行中)。这是 PG 锁系统里相当精巧的一笔,在海量并发 SELECT 的场景下大幅降低了锁表争用。

**步骤三:fast-path 没成功,走"正规流程"——拿分区轻量锁,在共享锁表里建/找 LOCK 和 PROCLOCK,查冲突。**

> [src/backend/storage/lmgr/lock.c:996-1046](../postgresql-17.0/src/backend/storage/lmgr/lock.c#L996-L1046)

```c
	partitionLock = LockHashPartitionLock(hashcode);

	LWLockAcquire(partitionLock, LW_EXCLUSIVE);

	...
	proclock = SetupLockInTable(lockMethodTable, MyProc, locktag,
								hashcode, lockmode);
	...
	locallock->proclock = proclock;
	lock = proclock->tag.myLock;
	locallock->lock = lock;

	/*
	 * If lock requested conflicts with locks requested by waiters, must join
	 * wait queue.  Otherwise, check for conflict with already-held locks.
	 */
	if (lockMethodTable->conflictTab[lockmode] & lock->waitMask)
		found_conflict = true;
	else
		found_conflict = LockCheckConflicts(lockMethodTable, lockmode,
											lock, proclock);

	if (!found_conflict)
	{
		/* No conflict with held or previously requested locks */
		GrantLock(lock, proclock, lockmode);
		GrantLockLocal(locallock, owner);
	}
```

这里有两层冲突检查,顺序很讲究:

1. **先查 `waitMask`**:如果这把锁的等待队列里已经有人在等,而且新请求和那些等待者冲突,**就必须排队**(不能插队)。这是**公平性**保证——避免后来者插队饿死等待者。
2. **再查 `LockCheckConflicts`**:和已持有的锁查冲突(就是前面讲的那次按位与)。
3. 没冲突就 `GrantLock` 授予(更新 LOCK 和 PROCLOCK 的掩码);有冲突就 `WaitOnLock`(把自己挂到等待队列睡眠)。

**步骤四(冲突时):睡眠等待。** 这一段调 `WaitOnLock`,它会把当前进程的 `PGPROC` 挂到 `LOCK.waitProcs` 队列上,然后进入睡眠(latch 等待)。同时启动一个 `deadlock_timeout` 定时器——这是死锁检测的触发点(下一节详讲)。

整个加锁流程,可以简化为:**LOCALLOCK 查重 → fast-path 快速通道 → 共享锁表建条目 → 查冲突(等队列优先 + 持有锁)→ 授予或睡眠**。这条链路上的每一步优化(LOCALLOCK、fast-path、位掩码冲突判定),都是为了"让锁快"——回扣全书主线。

## 六、死锁:等待图找环 + 延时检测

讲完加锁,最后讲一个所有锁系统都绕不开的难题:**死锁(deadlock)**。

### 死锁怎么产生:经典的"A 等 B、B 等 A"

最经典的死锁时序:

```text
时刻    事务 A                          事务 B
T1     BEGIN;                           BEGIN;
T2     UPDATE accounts SET ...
              WHERE id=1;               -- A 锁住行1
T3                                      UPDATE accounts SET ...
                                              WHERE id=2;  -- B 锁住行2
T4     UPDATE accounts SET ...          
              WHERE id=2;               -- A 想锁行2,被B挡住,A 睡眠等B
T5                                      UPDATE accounts SET ...
                                              WHERE id=1;  -- B 想锁行1,被A挡住,B 睡眠等A
       -- A 等 B 释放行2,B 等 A 释放行1 —— 死锁!两个都动不了
```

T4 时刻 A 等 B(T2 的事务),T5 时刻 B 等 A——形成了一个**等待的环**。没有外力介入,两个事务会永远等下去。

> **不这样会怎样**:如果没有死锁检测,这两个事务会**永久挂起**,占着锁和资源,把连接池耗尽,最终整个系统僵死。更阴险的是:这种死锁往往是**间歇性**的(只在特定时序下发生),测试时很难复现,生产上突然爆发,定位极痛苦。所以数据库必须能**自动检测并打破死锁**。

死锁的本质是**等待图(wait-for graph)里出现了环**:节点是事务,边是"A 在等 B"。有环就是死锁。检测死锁,就是在这个图里找环。

### PG 怎么检测:延时触发 + 构建等待图找环

PG 的死锁检测策略是**被动检测**——不实时维护等待图(太贵),而是在**某个事务等锁超过一定时间**后,才触发一次检测。

触发时机由参数 `deadlock_timeout` 控制:

> [src/backend/utils/misc/guc_tables.c:2137-2147](../postgresql-17.0/src/backend/utils/misc/guc_tables.c#L2137-L2147)

```c
	{
		/* This is PGC_SUSET to prevent hiding from log_lock_waits. */
		{"deadlock_timeout", PGC_SUSET, LOCK_MANAGEMENT,
			gettext_noop("Sets the time to wait on a lock before checking for deadlock."),
			NULL,
			GUC_UNIT_MS
		},
		&DeadlockTimeout,
		1000, 1, INT_MAX,
		NULL, NULL, NULL
	},
```

**默认 1000 毫秒(1 秒)**。意思是:一个事务等锁超过 1 秒,就触发死锁检测。

> **为什么是延时检测而不是实时?** 因为绝大多数锁等待**不是死锁**,只是别人还没做完(比如一个长 UPDATE)。实时维护等待图(每次加锁/等待都更新图)开销巨大。延时检测的策略是:**先等一会儿,大概率自己就被放行了;如果等了 1 秒还没动静,再花一次代价做完整的死锁检测**。这是"检测成本"和"死锁滞留时间"的权衡——1 秒是个经验值,够长(避免误触发),又够短(死锁不会滞留太久)。

信号处理函数 `CheckDeadLockAlert` 在超时触发时被调用,它只设一个标志位:

> [src/backend/storage/lmgr/proc.c:1835-1844](../postgresql-17.0/src/backend/storage/lmgr/proc.c#L1835-L1844)

```c

 * CheckDeadLockAlert - Handle the expiry of deadlock_timeout.
 *
 * NB: Runs inside a signal handler, be careful.
 */
void
CheckDeadLockAlert(void)
{
	got_deadlock_timeout = true;
```

(信号处理函数里不能干重活,所以只设标志,等进程从睡眠中醒来时再检查。)

进程在睡眠循环里检查这个标志,一旦发现就调 `CheckDeadLock`:

> [src/backend/storage/lmgr/proc.c:1754-1791](../postgresql-17.0/src/backend/storage/lmgr/proc.c#L1754-L1791)

```c
static void
CheckDeadLock(void)
{
	int			i;

	/*
	 * Acquire exclusive lock on the entire shared lock data structures. Must
	 * grab LWLocks in partition-number order to avoid LWLock deadlock.
	 */
	for (i = 0; i < NUM_LOCK_PARTITIONS; i++)
		LWLockAcquire(LockHashPartitionLockByIndex(i), LW_EXCLUSIVE);

	/*
	 * Check to see if we've been awoken by anyone in the interim.
	 */
	if (MyProc->links.prev == NULL ||
		MyProc->links.next == NULL)
		goto check_done;

	...

	/* Run the deadlock check, and set deadlock_state for use by ProcSleep */
	deadlock_state = DeadLockCheck(MyProc);
```

注意开头:**它把全部 16 个锁分区的 LWLock 都按顺序排他地拿住**。这是为了在检测期间**冻结整个锁系统的状态**(没有人能改 LOCK/PROCLOCK),否则等待图会一边检测一边变,出错。按分区号顺序拿,是为了避免自己陷入 LWLock 死锁(就是前面讲的轻量锁靠加锁顺序避死锁)。

然后检查"自己是不是在检测期间已经被别人唤醒放行了"(如果已经脱离等待队列,就没事,直接返回)。最后调 `DeadLockCheck(MyProc)` 做真正的检测。

### `DeadLockCheck`:构建等待图找环

这是死锁检测的核心:

> [src/backend/storage/lmgr/deadlock.c:216-243](../postgresql-17.0/src/backend/storage/lmgr/deadlock.c#L216-L243)

```c
DeadLockState
DeadLockCheck(PGPROC *proc)
{
	/* Initialize to "no constraints" */
	nCurConstraints = 0;
	nPossibleConstraints = 0;
	nWaitOrders = 0;

	/* Initialize to not blocked by an autovacuum worker */
	blocking_autovacuum_proc = NULL;

	/* Search for deadlocks and possible fixes */
	if (DeadLockCheckRecurse(proc))
	{
		/*
		 * Call FindLockCycle one more time, to record the correct
		 * deadlockDetails[] for the basic state with no rearrangements.
		 */
		int			nSoftEdges;

		TRACE_POSTGRESQL_DEADLOCK_FOUND();

		nWaitOrders = 0;
		if (!FindLockCycle(proc, possibleConstraints, &nSoftEdges))
			elog(FATAL, "deadlock seems to have disappeared");

		return DS_HARD_DEADLOCK;	/* cannot find a non-deadlocked state */
	}
```

它的核心是 `DeadLockCheckRecurse`——从当前进程出发,**递归地沿着"我在等谁、他在等谁"这条链找环**。这本质就是在等待图上做 DFS 找环。

这里有个精妙的设计:PG 区分**硬边(hard edge)**和**软边(soft edge)**:

- **硬边**:已经持有的锁造成的阻塞(比如 A 持有行1,B 要行1,B 等 A)。这种边**没法靠重排队列消除**。
- **软边**:等待队列里的顺序造成的阻塞(比如 A、B 都在等同一把锁,排在前面的挡住后面的)。这种边**可能靠重排队列消除**(把后面的提到前面,就不挡了)。

所以 `DeadLockCheck` 找到环时,不是立刻报死锁,而是**先尝试重排等待队列**(把软边换顺序),看能不能解开。只有所有重排都解不开(剩下全是硬边构成的环),才返回 `DS_HARD_DEADLOCK`——这是真正的死锁,只能靠回滚打破。

返回 `DS_HARD_DEADLOCK` 后,`CheckDeadLock` 做的事是:

> [src/backend/storage/lmgr/proc.c:1793-1808](../postgresql-17.0/src/backend/storage/lmgr/proc.c#L1793-L1808)

```c
	if (deadlock_state == DS_HARD_DEADLOCK)
	{
		/*
		 * Oops.  We have a deadlock.
		 *
		 * Get this process out of wait state. (Note: we could do this more
		 * efficiently by relying on lockAwaited, but use this coding to
		 * preserve the flexibility to kill some other transaction than the
		 * one detecting the deadlock.)
		 */
		Assert(MyProc->waitLock != NULL);
		RemoveFromWaitQueue(MyProc, LockTagHashCode(&(MyProc->waitLock->tag)));
```

注意这条注释——**"preserve the flexibility to kill some other transaction than the one detecting the deadlock"**。理论上 PG 可以选择回滚环里**代价较小**的那个事务(比如改动最少的),而不是非得回滚"触发检测的那个"。当前实现是回滚检测者自己(因为它已经在信号处理里、最容易抛错),但设计上留了余地。回滚后,这个事务抛出错误 `ERROR: deadlock detected`,客户端收到详细的事务等待链信息。

### 检测到死锁后:回滚,然后应用重试

死锁被打破的方式是**回滚其中一个事务**。被回滚的事务释放它持有的所有锁,环就断了,另一个事务得以继续。

被回滚的事务会收到一个错误:

```text
ERROR:  deadlock detected
DETAIL:  Process ... waits for ShareLock on transaction ...; blocked by process ....
         Process ... waits for ShareLock on transaction ...; blocked by process ....
HINT:  See server log for query details.
```

DETAIL 里就是那条等待链(谁等谁),帮你定位。**关键点:死锁不是 bug,是并发事务的正常可能性**——应用代码必须**捕获这个错误并重试**整个事务。这是写并发数据库应用的基本素养:

```python
for attempt in range(max_retries):
    try:
        with db.transaction():
            ...  # 你的业务 SQL
        break
    except DeadlockDetected:
        continue  # 重试
```

> **`deadlock_timeout` 默认 1 秒,意味着什么?** 意味着一旦发生死锁,最坏要等 1 秒才会被检测到(然后回滚、重试)。如果你的业务对延迟敏感(比如支付),可以把 `deadlock_timeout` 调小(比如 200ms),让死锁更快被发现。但要注意:调太小会让"正常的锁等待"误触发检测(检测本身有开销,锁住所有锁分区),反而拖慢系统。1 秒是个平衡点。另外,`log_lock_waits = on` 时,任何超过 `deadlock_timeout` 的锁等待都会被记进日志(即使不是死锁),这对排查"为什么我的查询慢"很有用——它会把阻塞者的事务 ID、查询都打出来。

**阶段⑤ 授予后的收尾。**


把这五阶段对应到前面的设计:

| 阶段 | 对应的设计点 |
|---|---|
| ① 校验 + ResourceOwner | 锁的生命周期(事务 vs 会话) |
| ② LOCALLOCK 查重 | 本地缓存优化,绕开共享内存 |
| ③ fast-path | 弱表锁的进一步加速(16 槽位) |
| ④ 共享锁表正规流程 | 三张表 + 位掩码冲突判定 + 等队列公平 |
| ⑤ 收尾 | 分区 LWLock 释放 |

读完这条链路,你就理解了"PG 加一把锁"的全部内部逻辑——从"为什么有 LOCALLOCK"到"fast-path 加速什么"到"冲突怎么判定"到"等队列怎么保证公平",都在这一个函数里。

## 章末小结

### 用一段话回顾本章

PG 的锁分三类,各司其职:**常规锁**(heavyweight lock)保护表、对象等逻辑资源,有 8 个强度级别(从最弱的 `AccessShareLock` 到最强的 `AccessExclusiveLock`),它们之间的冲突关系由一张位掩码表 `LockConflicts` 定义——判定冲突就是一次按位与,O(1);**轻量锁(LWLock)**保护内存里的共享数据结构(BufferDesc、ProcArray、锁表本身),只在极短的临界区持有,用自旋 + 等待队列实现,靠加锁顺序约定避死锁、不参与死锁检测;**行锁**最特别——它不在数据页上留物理锁标记,而是借助 tuple header 的 `xmax` 字段 + 几个标志位(`HEAP_XMAX_LOCK_ONLY`、`HEAP_XMAX_*_LOCK`)实现,只在需要排队时才升级到共享锁表。这套设计让 PG 的行锁开销极小、不会撑大表、崩溃也不留脏锁。多粒度锁靠**意向锁**(`RowShareLock`/`RowExclusiveLock` 充当 IS/IX)让表锁和行锁互相感知。死锁靠**延时检测**(`deadlock_timeout` 默认 1 秒触发)+ **等待图找环**(`DeadLockCheck`)破解,检测到就回滚其中一个事务,应用需要捕获错误并重试。常规锁内部用三张表(`LOCK` 共享对象状态 + `PROCLOCK` 持有者状态 + `LOCALLOCK` 本地缓存)管理,加上 fast-path 弱锁加速,让加锁路径极快。

### 回扣主线

本章服务的是全书主线里"**让数据不丢不乱**"那一侧的"**不乱**"。具体说,锁保证的是**并发写的正确串行化**——多个事务同时改,锁让它们排成某个先后顺序,谁先谁后无所谓(只要结果等价于某个串行执行),但不能"乱"(丢失更新、读到改一半的结构)。这是 ACID 里 **I(隔离性)** 的"写写冲突"那一半(MVCC 管"读写并发",锁管"写写冲突",两者合起来才是完整的 I)。

回扣三个本性:锁对付的是第二条——**"数据要共享,但共享会冲突"**。共享就是并发访问,冲突就是同时改,锁就是把冲突排队化、串行化的机制。没有锁,共享就会变成混乱。

而 PG 锁系统里所有的优化——LOCALLOCK 本地缓存、fast-path 快通道、位掩码冲突判定、LWLock 自旋——都是在"保证不乱"的前提下,拼命"让数据快"。这又一次印证了全书二分法:**不乱是底线,快是追求**,PG 在锁这个看似拖慢并发的子系统里,也塞满了让它变快的精巧设计。

### 三类锁的对比收束(再贴一次,记住这张表)

| | 常规锁 | 轻量锁(LWLock) | 行锁 |
|---|---|---|---|
| 保护对象 | 表、对象(逻辑资源) | 内存共享结构 | 单行数据 |
| 持续时间 | 到事务结束 | 临界区(极短) | 到事务结束 |
| 级别数 | 8 级 | 3 模式(S/X/WAIT_UNTIL_FREE) | 4 级 |
| 实现载体 | 共享 LOCK/PROCLOCK 表 + LOCALLOCK | 原子变量 + 等待队列 | xmax 标志位(+ 必要时锁表) |
| 死锁检测 | 参与(DeadLockCheck) | 不参与(靠加锁顺序) | 借常规锁的通道 |
| 典型开销 | 中(首次入表,后续本地) | 极小(原子操作) | 极小(改几个 bit) |
