# 第 1 篇 · 查询引擎:一条 SQL 怎么进来、被理解、被规划、被执行

> 读者递进门的便条,到馆员决定怎么去取档——SQL 的前半生。

---

## 篇定位

本篇讲一条 SQL 的**前半生**:从客户端敲下回车、连接进数据库,到这句字符串被**读懂(parse)**、被**规划(optimizer 选最省的执行计划)**、被**跑出结果(executor)**——到这一篇结束,数据还**没真正落到磁盘页和索引上**(那是 P2/P3 的事),但"一句声明式 SQL 是怎么一步步变成对存储层的具体调用"这条链路,已经完整走完。

在"一条 SQL 的旅程"这条主线上,本篇覆盖旅程的**第 1~4 站**(连接 → 解析 → 优化 → 执行),是全书**最靠前、也最贴近应用层**的一篇——读者天天写的 `SELECT/INSERT/UPDATE/DELETE`,在这一篇里第一次被拆开看个底朝天。

回到全书主线**二分法**(让数据"快" vs 让数据"不丢不乱"):本篇整篇都服务**"快"**那一侧,更精确地说,服务数据的第三个本性——**"用户说'要什么',机器只会'按位置读'"**。SQL 是声明式的,它和物理存取之间存在一道鸿沟;这道鸿沟,就由本篇的四章接力填平:连接章把 SQL 送进门,解析章把它翻译成一棵结构化的 `Query` 树(只懂语法、不懂语义,再到查表落实的 `Query`),优化章从这棵"声明意图"的树出发枚举出所有"说得通"的物理路径并选最省的,执行章把最优计划按火山模型一行行跑出来。

> 一句话:**没有本篇,声明式 SQL 就只是一串字符;有了本篇,它才变成一组对存储层的具体读写指令**。本篇也是"快"这一侧的第一块拼图——后面 P2/P3 把"数据怎么高效存在磁盘与内存""怎么用索引快速定位"接上,整条"让查询快起来"的链路才闭合。

## 篇内分章清单

> 每章都围绕一个**核心问题**展开。先读问题,带着问题读正文。所有源码引用指向本仓库 `postgresql-17.0/`(PostgreSQL 17.0),可点击跳转。

### 第 2 章 · 连接与会话:客户端怎么"进门"

- **核心问题**:你敲下 `psql -h ... -U ...`,到能输入 SQL 之间,发生了什么?为什么 PostgreSQL 给**每个客户端连接单独 fork 一个进程**,而不是像很多数据库那样用线程池?一个 backend 崩了会怎样?你按 Ctrl+C 取消一个跑飞的查询,数据库是怎么停下来的?
- **要讲清的"为什么"**:
  - "每连接一进程"换来的**崩溃隔离(fault isolation)**:一个 backend 崩了(空指针、C 扩展 bug)只死它自己,不拖垮别的连接——OS 在硬件层面保证进程地址空间隔离。
  - 一个 backend 崩了为什么宁可**全库重启**也不只重启那一个:共享内存(锁表、缓冲池、ProcArray)可能在崩溃瞬间被改到一半,带脏状态苟活是数据正确性的灾难,所以"宁崩勿脏"。
  - 认证为什么放在 **fork 之后、backend 子进程里**做,以及认证这段最不可信时间里 backend 刻意**不碰共享内存**(出问题可直接 `_exit`)。
  - cancel key 为什么用**密码学级随机数**——它是"取消令牌",防止任意客户端凭 pid 就能取消别人的查询(DoS)。
  - **扩展查询协议**(Parse/Bind/Execute)为什么又快(一次 Parse 多次复用)又安全(参数结构化,根本性防 SQL 注入)。
- **关键源码**:
  - [`ServerLoop`](../postgresql-17.0/src/backend/postmaster/postmaster.c#L1624)(postmaster 主循环,接客/fork)、[`BackendStartup`](../postgresql-17.0/src/backend/postmaster/postmaster.c#L3544)(生成 cancel key + fork)、[`processCancelRequest`](../postgresql-17.0/src/backend/postmaster/postmaster.c#L1835)(cancel key 校验,含那句 `no way, Jose`)。
  - [`BackendMain`](../postgresql-17.0/src/backend/tcop/backend_startup.c#L56) → [`BackendInitialize`](../postgresql-17.0/src/backend/tcop/backend_startup.c#L122)(读 startup packet、早期拒绝)。
  - [`InitPostgres`](../postgresql-17.0/src/backend/utils/init/postinit.c#L736)(认证 / 选库 / relcache / 起事务快照,会话初始化总入口)、[`ClientAuthentication`](../postgresql-17.0/src/backend/libpq/auth.c#L381)(按 pg_hba.conf 选认证方式)。
  - [`PostgresMain`](../postgresql-17.0/src/backend/tcop/postgres.c#L4239)(backend 主循环)、`sigsetjmp` 锚点([L4426](../postgresql-17.0/src/backend/tcop/postgres.c#L4426),一条 SQL 出错 backend 不死的根基)。
- **难度 / 阅读提示**:★★☆ 入门级。本章是旅程**第一站**,概念都贴近日常(`psql`、Ctrl+C、密码登录),没有数据结构门槛。重点抓"每连接一进程"这个架构决策的**换来了什么 / 付出了什么**。

### 第 3 章 · SQL 解析器:从字符串到解析树

- **核心问题**:你敲下的 `SELECT id, name FROM users WHERE id = 1`,到 backend 手里就是一个字符数组 `'S','E','L','E','C','T',...`。数据库怎么"读懂"它?为什么必须先把它翻译成一棵**解析树(parse tree)**,而不是边读字符串边执行?词法分析、语法分析、语义分析这**三步**各自在干什么、为什么不能合并成一步?
- **要讲清的"为什么"**:
  - 字符串和执行器之间有鸿沟:字符串是人写的、线性的,执行器要的是结构化、可逐节点处理的指令。**树**是"声明式语句最自然的载体"(嵌套天然用树表达)。
  - 为什么"边读字符串边执行"不行:没法做检查(表/列存不存在)、没法优化(选哪条执行计划)、没法重写(视图展开)——这些都需要"完整理解整句 SQL 之后"才能做。
  - 为什么分**三步**:词法用正则(flex)、语法用上下文无关文法(bison)、语义需要查表,三种不同性质的逻辑各自由专门工具处理最高效;更关键的是,**纯字符解析必须和"需要查表"的语义分析分开**,否则事务 abort 状态下连 `ROLLBACK` 都解析不了。
  - 为什么语义分析在最后:语义依赖是单向的(列引用依赖表引用、表达式类型依赖列类型),必须先把 FROM(表)落实,才能解析目标列和 WHERE。
- **关键源码**:
  - [`exec_simple_query`](../postgresql-17.0/src/backend/tcop/postgres.c#L1017)(backend 主循环里处理 SQL 的入口)→ [`pg_parse_query`](../postgresql-17.0/src/backend/tcop/postgres.c#L615)(纯字符解析,注释明确点出"analysis and rewriting cannot be done in an aborted transaction")。
  - 词法:[`scan.l`](../postgresql-17.0/src/backend/parser/scan.l#L1080)(flex,`{identifier}` 规则,关键字与标识符同源、转小写);语法:[`gram.y`](../postgresql-17.0/src/backend/parser/gram.y#L12790)(bison,`simple_select` 产生式)、[`raw_parser`](../postgresql-17.0/src/backend/parser/parser.c#L41)(把词法和语法串起来)。
  - 语义:[`parse_analyze_fixedparams`](../postgresql-17.0/src/backend/parser/analyze.c#L103) → [`transformStmt`](../postgresql-17.0/src/backend/parser/analyze.c#L311) → [`transformSelectStmt`](../postgresql-17.0/src/backend/parser/analyze.c#L1337)(依次 transform FROM / 目标列 / WHERE)。
  - 关键节点:`SelectStmt`([parsenodes.h#L2116](../postgresql-17.0/src/include/nodes/parsenodes.h#L2116))、`RangeVar`([primnodes.h#L71](../postgresql-17.0/src/include/nodes/primnodes.h#L71),只记表名字符串、无 OID)。
- **难度 / 阅读提示**:★★☆ 入门偏中。需要一点点对"树/递归"的直觉,但不需要懂编译原理(词法=正则、语法=文法,点到即止)。重点抓三阶段分界线:**前两步纯看字符串、第三步才开始查表**。

### 第 4 章 · 查询优化器:怎么决定"用哪条路最省"

- **核心问题**:同一句 SQL 为什么会有多种执行计划?优化器怎么从解析树出发,枚举出所有"说得通"的执行路径,又怎么在其中挑出**代价最小**的那条?这个"代价(Cost)"到底由哪几部分加起来?统计信息(直方图、最常见值、distinct 数)是怎么把"这一行到底符不符合条件"这种**不确定**的事,变成一个**可以拿去算**的数字的?以及最反直觉的一点——为什么优化器算出来的代价本质上是个**有根据的猜测**,而不是精确计算?它因此会在什么场景下出错?
- **要讲清的"为什么"**:
  - 优化器的存在,是被 SQL 的**声明式本性**逼出来的(SQL 只说"要什么",优化器负责填上"怎么取");PG 选**基于代价的优化(CBO)**而非"基于规则(RBO)",是因为"快不快取决于数据长什么样",而代价模型是唯一能把"数据分布"纳入考量的方式。
  - 代价是一个**无量纲的相对分数**(单位 = "一次顺序读页"),= 顺序读页代价 + 随机读页代价 + 每行 CPU 代价,分 `startup_cost` / `total_cost` 两段。
  - 选择率(selectivity)→ 基数(cardinality)→ 每层算子代价,这条**自底向上的估算链**是优化器的脊柱;统计信息过期时,误差会沿着这条链一路放大。
  - 两阶段:**逻辑等价重写**(`pull_up_subqueries` 等,只动 Query 树结构)在前,**物理路径枚举 + 代价比较**(动态规划,表多则退化 GEQO)在后,分两阶段是为了先变形暴露更多最优路径、再比较代价。
  - **等值选择率用 MCV + distinct 均摊,不用直方图**(直方图只服务范围查询)——这是最常见的误解。
- **关键源码**:
  - 代价模型:[`DEFAULT_SEQ_PAGE_COST` 等常数](../postgresql-17.0/src/include/optimizer/cost.h#L24)、[`cost_seqscan`](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L284)(磁盘代价 + CPU 代价的加和范本,逐行拆解过)、[`set_baserel_size_estimates`](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L5239)(`nrows = tuples × clauselist_selectivity`,本章脊梁公式)。
  - 统计信息:[`pg_class.relpages/reltuples`](../postgresql-17.0/src/include/catalog/pg_class.h#L62)(表级,注释明说 "not always up-to-date")、[`pg_statistic`](../postgresql-17.0/src/include/catalog/pg_statistic.h#L37)(列级,`stadistinct` 带符号的小聪明)。
  - 选择率:[`var_eq_const`](../postgresql-17.0/src/backend/utils/adt/selfuncs.c#L294)(等值查询:命中 MCV 精确、未命中则 `(1 - sumcommon - nullfrac) / otherdistinct` 均摊)。
  - 统计收集:[`analyze_rel`](../postgresql-17.0/src/backend/commands/analyze.c#L111)、[`acquire_sample_rows`](../postgresql-17.0/src/backend/commands/analyze.c#L1158)(为什么采样而非全表扫)、Haas-Stokes 估计器([L2562](../postgresql-17.0/src/backend/commands/analyze.c#L2562))。
  - 两阶段入口:[`subquery_planner`](../postgresql-17.0/src/backend/optimizer/plan/planner.c#L633)(重写在 L730、`grouping_planner` 在 L1092);路径枚举:[`make_one_rel`](../postgresql-17.0/src/backend/optimizer/path/allpaths.c#L170)、[`standard_join_search`](../postgresql-17.0/src/backend/optimizer/path/allpaths.c#L3410)(动态规划)、[`geqo`](../postgresql-17.0/src/backend/optimizer/geqo/geqo_main.c#L71)(遗传算法逃生阀)。
- **难度 / 阅读提示**:★★★★ 全篇最难也最精华的一章。需要接受"代价是个相对分数、不是秒"这个反直觉设定,以及"优化器在猜不在算"这个本质。建议配合 `EXPLAIN` 实操读——看完本章再去看任何 `EXPLAIN` 输出,都会有"豁然开朗"的通透感。如果一读觉得太难,正文开头有三件事的"速记版",不影响理解第 5 章。

### 第 5 章 · 执行器:把计划跑出来

- **核心问题**:执行计划(Plan 树)怎么被一步步跑出结果?为什么几乎所有关系数据库(MySQL、Oracle、PostgreSQL)的执行器都用同一套**「火山模型 / Volcano 迭代器」**——每个算子暴露一个 `next()` 接口,上层来拉、下层被动吐?顺序扫描、索引扫描、三种连接(嵌套循环 / Hash / 归并)分别在哪类场景最优、各自的代价来自哪里?
- **要讲清的"为什么"**:
  - 火山模型为什么是**"拉(pull)"而不是"推(push)"**:接口统一(组合自由)、流式产出(中间结果不必物化、内存可控)、能短路 `LIMIT`(根节点拉够就停,子树自然停止)、适合流水线(边算边返回第一行)。
  - 拉模型的代价:逐行函数调用,在 OLAP 重扫场景显现,于是有向量化改进;PG 没改向量化,但用 `ExecProcNodeFirst` 一次性检查 + 自我替换、`MultiExecProcNode` 逃生舱、`pg_attribute_always_inline` 等细节抠热路径。
  - 三种连接的本质取舍:NestedLoop(O(外×内),适合外表小+内表有索引、或非等值)、Hash(O(内+外),等值连接首选,内表超内存用**分批 batching**)、MergeJoin(两边有序时 O(内+外) 最省,无序要先排)。
  - Hash 节点不走标准 `ExecProcNode` 协议、用 `MultiExecProcNode` 一次性消费整个内表——这是火山模型为"天然需要物化"的算子留的逃生舱。
- **关键源码**:
  - 火山模型统一入口:[`ExecProcNode`](../postgresql-17.0/src/include/executor/executor.h#L267)(内联包装,核心是 `node->ExecProcNode(node)` 虚函数分发)、[`ExecProcNodeFirst`](../postgresql-17.0/src/backend/executor/execProcnode.c#L442)(一次性栈深度检查 + 自我替换)、`PlanState` 骨架([execnodes.h#L1112](../postgresql-17.0/src/include/nodes/execnodes.h#L1112))。
  - 叶子算子:[`ExecSeqScan`](../postgresql-17.0/src/backend/executor/nodeSeqscan.c#L107) + [`SeqNext`](../postgresql-17.0/src/backend/executor/nodeSeqscan.c#L49)、[`ExecIndexScan`](../postgresql-17.0/src/backend/executor/nodeIndexscan.c#L519) + [`IndexNext`](../postgresql-17.0/src/backend/executor/nodeIndexscan.c#L80)(runtime keys + lossy recheck)、通用扫描框架 [`ExecScan`](../postgresql-17.0/src/backend/executor/execScan.c#L193)。
  - 三种连接:[`ExecNestLoop`](../postgresql-17.0/src/backend/executor/nodeNestloop.c#L99)(双层 for,`ExecReScan` 重扫内表)、[`ExecHashJoinImpl`](../postgresql-17.0/src/backend/executor/nodeHashjoin.c#L254)(五状态机,`HJ_BUILD_HASHTABLE` 用 `MultiExecProcNode`)、`ExecHash` 拒绝逐行([nodeHash.c#L90](../postgresql-17.0/src/backend/executor/nodeHash.c#L90))、[`ExecChooseHashTableSize`](../postgresql-17.0/src/backend/executor/nodeHash.c#L675)(分批决策)、[`nodeMergejoin.c`](../postgresql-17.0/src/backend/executor/nodeMergejoin.c#L105)(11 状态机)。
  - 顶层三段:[`ExecutorStart`](../postgresql-17.0/src/backend/executor/execMain.c#L124)(建 EState)、[`ExecutorRun` → `ExecutePlan`](../postgresql-17.0/src/backend/executor/execMain.c#L299)(`for(;;)` 拉根节点的循环,火山模型发动机)、`ExecutorFinish`(触发器,独立于 `ExecutorEnd` 是为 `EXPLAIN ANALYZE` 统计窗口完整)。
- **难度 / 阅读提示**:★★★ 中等。火山模型本身很直觉(就是迭代器模式),难点在三种连接的状态机和内存控制细节。建议先抓"拉模型 + 三种连接适用场景"的主干,状态机细节配合源码第二遍再抠。

## 篇内衔接线索:连接 → 解析 → 优化 → 执行的旅程

这四章不是平铺的四个话题,而是一条**严格接力**的数据流水线——每一步的输出,都是下一步的输入,中间任何一环断了,后面的章节都无从谈起。

```text
   客户端发 SQL
        │
        ▼
┌──────────────────────────────────────────────┐
│ 第 2 章 · 连接与会话                              │
│   TCP → startup packet → fork backend          │
│   → 认证 → InitPostgres 会话初始化 → 主循环       │
│   产出:一个就位的 backend,阻塞在 ReadCommand     │
└──────────────────┬───────────────────────────┘
                   │  客户端发来一句 SQL(字符串)
                   ▼
┌──────────────────────────────────────────────┐
│ 第 3 章 · SQL 解析器                              │
│   字符串 → 词法(token) → 语法(raw parse tree) │
│   → 语义(查系统目录)→ Query 树                  │
│   产出一棵"声明意图"的树:只说"要什么"           │
│   (要 id=1 的行、要把两表连起来)                  │
└──────────────────┬───────────────────────────┘
                   │  Query 树(只懂语义,不懂"怎么取")
                   ▼
┌──────────────────────────────────────────────┐
│ 第 4 章 · 查询优化器                              │
│   逻辑重写 → 单表路径 → DP 枚举连接顺序           │
│   → 每条路径估代价(统计信息→选择率→基数→代价)    │
│   → 选最省 → create_plan 物化                    │
│   产出一棵"选定算法、估好代价"的 Plan 树          │
└──────────────────┬───────────────────────────┘
                   │  Plan 树(图纸,一行数据还没动)
                   ▼
┌──────────────────────────────────────────────┐
│ 第 5 章 · 执行器                                  │
│   ExecutorStart(建 EState)→ ExecutorRun        │
│   → ExecutePlan 循环拉根节点                     │
│   火山模型:上层 next() 拉、下层吐               │
│   叶子算子调 table_scan / index_getnext         │
│   ──▶ 把活儿下放给存储引擎(P2/P3)              │
└──────────────────┬───────────────────────────┘
                   │
                   ▼
        结果一行行返回给客户端
```

**为什么章与章必须紧挨着:**

- **第 2 → 第 3 章**:第 2 章结尾,backend 在 `PostgresMain` 主循环里阻塞在 `ReadCommand`,等客户端发 SQL。客户端一旦发来一句,backend 读到的就是一串字符——这串字符它还**看不懂**。所以第 3 章必须紧接着讲"怎么把这串字符变成树"。第 3 章的入口 `exec_simple_query`,正是第 2 章主循环 `switch(firstchar)` 里 `PqMsg_Query` 那个 case 调进来的。

- **第 3 → 第 4 章**:第 3 章结尾,产出的 `Query` 树只表达了"**要什么**"(要 `id=42` 的行、要把两张表连起来),但**故意没说**"怎么去取"。这棵"声明意图"的树,正是优化器的**输入**。优化器的全部职责,就是从这棵树出发,决定走哪条物理路径最省。

- **第 4 → 第 5 章**:第 4 章结尾,优化器通过 `create_plan` 把最优 Path 物化成一棵 **Plan 树**(执行器认识的形态)。但这棵树还只是"图纸",一行数据都还没动。第 5 章的执行器,就是负责把这棵图纸**真正跑起来**,一行一行把结果吐出来。

- **第 5 → P2**:第 5 章执行器在叶子算子(`SeqScan`/`IndexScan`)里调 `table_scan_getnextslot` / `index_getnext_slot` 取数据——但这些数据**住在哪里**、磁盘上怎么摆放、为什么按 8KB 页整块搬?这就把活儿**下放给存储引擎**,自然引出 P2 第 6 章(Page)。

> 这条接力链的每一环,都对应着回扣主线的同一个本性——**"用户说要什么,机器只会按位置读"**。解析器把"要什么"从字符串变成结构化树(跨鸿沟第一步),优化器把"要什么"翻译成"怎么取"的多个候选并选最省(跨鸿沟第二步),执行器把"怎么取"的图纸跑成对存储层的具体调用(鸿沟落地)。本篇四章,就是这道鸿沟被一步步填平的全过程。

## 阅读建议

### 顺序

**强烈建议按 2 → 3 → 4 → 5 的顺序读**,不要跳。原因有二:

1. **数据结构上严格接力**:`Query` 树(第 3 章产出)是优化器(第 4 章)的输入;`Plan` 树(第 4 章产出)是执行器(第 5 章)的输入。跳着读会在中间环节断链,看不懂"这棵树从哪来、到哪去"。
2. **概念难度递进**:第 2 章最贴近日常(连接、认证),第 3 章引入"树/parse"概念,第 4 章是全篇最难(代价模型 + 统计信息),第 5 章回到直觉的迭代器模式。这个顺序让难度爬坡最平滑。

唯一可以单独抽出来读的是**第 2 章**——它和后三章的耦合最松(主要讲进程模型和协议,不依赖"树"的概念),如果你想先建立"PG 整体长什么样"的全景,可以先读它。

### 哪章是精华

- **如果只读一章,读第 4 章(优化器)**。它是本篇的灵魂,也是全书"快"那一侧的理论巅峰。读完它,你对任何 `EXPLAIN` 输出都会有全新的理解,也会明白为什么 DBA 的两条铁律之一是"定期 `ANALYZE`"。
- **如果只读两章,读第 4 + 第 5 章**。第 4 章讲"怎么决策",第 5 章讲"决策怎么落地",合起来就是"一句 SQL 是怎么被高效执行的"完整答案。
- **第 2 章和第 3 章**虽然不是精华,但它们是**地基**:第 2 章建立的"进程模型 + 会话状态"概念会在 P4(事务/MVCC)再次出现(`ProcArray`、快照);第 3 章建立的"`Query` 树"是本篇后续所有讨论的起点。跳过它们,后面会缺一块。

### 前置依赖

- **必须先读**:第 0 篇的[全书导言](全书导言-一篇看懂数据库内核全貌.md)(建立"一条 SQL 的旅程"全景 + "快 vs 不丢不乱"主线)和[第 1 章 · 第一性原理](P0-01-第一性原理-为什么需要数据库.md)(数据的三个本性,尤其第三个"声明式查询 vs 物理存取"——本篇整篇都在对付这个本性)。
- **不需要**:本篇不预设你读过 P2/P3/P4/P5 任何一章。执行器调存储引擎的接口(`table_scan_getnextslot` 等),本篇只点到"活儿下放给存储引擎",具体实现留到 P2/P3 讲;MVCC 的快照在第 2 章 `InitPostgres` 里只是"backend 第一次访问数据库前要拿快照"一句话带过,细节在 P4。
- **建议配合**:一边读一边在自己机器上跑 PG,用 `psql` 连上,随手 `EXPLAIN` 看计划。尤其是第 4 章,光读文字不够,自己造一张表、插不同分布的数据、`ANALYZE` 前后对比 `EXPLAIN` 输出,体感最深。

### 读完本篇你应该具备的能力

- 看懂任何一句 SQL 从客户端到结果返回的**完整调用链**(能说出"这一步在哪个文件、哪个函数")。
- 看懂 `EXPLAIN` 输出里每个算子的代价数字**怎么来的**(能指出"`cost=1234.56` = 顺序读多少页 + 随机读多少页 + 处理多少行 × 每行代价")。
- 能诊断"为什么优化器选了看起来更慢的计划"——根子大概率在**统计信息过期**,第一味药是 `ANALYZE`。
- 理解为什么 PG 是"每连接一进程"、为什么崩溃隔离这么重要、为什么连接多了要上 Pgbouncer——这些是后续 P4(事务/锁/MVCC)和 P6(进阶)的地基。
