# 第 2 篇 · 存储引擎:数据在磁盘与内存里长什么样

> 一条 SQL 走到"取/存数据"这一站时,数据库的视角从"SQL 怎么被理解、被执行"(P1)转到"这些数据到底以什么形态躺在磁盘上、又怎么搬进内存"。
>
> 本篇钉死四个层次:**页(page,8KB 容器)** → **元组(tuple,页里的一行行内容)** → **Buffer Pool(内存里的中转站)** → **FSM/VM(两张辅助地图)**。它把数据三本性里"**磁盘慢、内存快但易失**"接住一半:**高效靠这里的内存中转,持久要靠 P5 的日志**——所以本篇另一只手会反复和 P5 的 WAL 握着(最典型的一处:第 8 章 `FlushBuffer` 里那句"刷数据前先 `XLogFlush` 刷日志")。
>
> 全书主线"**让数据快 vs 让数据不丢不乱**"在本篇尤其清晰:页布局/tuple 紧凑/Buffer Pool 缓存/FSM 找空页,全在追求"快";而页校验和、`pd_lsn`、脏页刷盘前的 WAL 铁律、`LP_DEAD` 与 VACUUM 的交接,守的全是"不丢不乱"。

## 篇内衔接线索:一条数据从落盘到被读到

```text
   INSERT/SELECT 要动一行数据
            │
            ▼
   ┌─────────────────────────────────────┐
   │ ① 页 Page(8KB 容器)                 │ 第 6 章
   │   两头往中间长:头部行指针+尾部 tuple │ 磁盘 I/O 的最小单位
   └──────────────┬──────────────────────┘
                  │ 页尾塞的就是一行行
                  ▼
   ┌─────────────────────────────────────┐
   │ ② 元组 Tuple(页里的内容)            │ 第 7 章
   │   23B 头 + null 位图 + 对齐 + 数据    │ 一行数据怎么落盘
   │   t_xmin/t_xmax/t_ctid(为 MVCC 埋伏笔)│ TOAST 切片大字段
   └──────────────┬──────────────────────┘
                  │ 每次读写不能真打磁盘
                  ▼
   ┌─────────────────────────────────────┐
   │ ③ Buffer Pool(内存中转站)           │ 第 8 章
   │   固定数组 + hash 表 + Clock-Sweep    │ 用内存换速度
   │   脏页刷盘前先 XLogFlush(接 P5 WAL)  │
   └──────────────┬──────────────────────┘
                  │ 插入要找空页 / 清理要跳页
                  ▼
   ┌─────────────────────────────────────┐
   │ ④ FSM 与 VM(两张辅助地图)           │ 第 9 章
   │   FSM:树形找空页 O(log n)            │
   │   VM :每页 2 bit,VACUUM/仅索引扫跳页 │ 接 P4 MVCC、P3 仅索引扫
   └─────────────────────────────────────┘
```

**层次口诀**:页是**容器**,tuple 是**内容**,Buffer Pool 是**内存中转**,FSM/VM 是**辅助地图**。前两者管"数据物理长什么样",后两者管"怎么高效地读写它"。

## 篇内分章清单

### 第 6 章 · 页面 Page:磁盘 I/O 的最小单位

- **核心问题**:数据库存几千万行数据,为什么不是"一行一行"存,而是切成一个个 8KB 的**页(page)**?这 8KB 在磁盘上到底长什么样——页头 24 字节记了什么、行指针怎么编码、空闲空间在哪、校验和怎么保住"不乱"?
- **要讲清的"为什么"**:
  - I/O 粒度必须匹配硬件的块粒度(磁盘扇区 / OS 4KB 页 / CPU cache line),不按页存会有"寻道摊不开 / read-modify-write / 无法寻址缓存"三个灾难;
  - 8KB 是页头开销、行容量、缓冲池粒度、以及 `lp_off`/`lp_len` **15 位位域上限(32KB)**之间的权衡;
  - 页**两头往中间长**(头部行指针从前往后、尾部 tuple 从后往前),让加新行是 O(1)、不搬动已有数据;
  - 行指针 `ItemId` 用位域压成 4 字节、四种状态(`LP_DEAD`/`LP_REDIRECT` 分别预告 P6 的 VACUUM 和 HOT);
  - 索引通过 `(block, offset)` 经行指针**间接寻址** tuple,tuple 页内移动不影响索引;
  - 页带 `pd_checksum`(FNV-1a 改良、混入页号、防静默腐败)和 `pd_lsn`(锁住"WAL 先 flush 才能刷脏页"的 P5 铁律)。
- **关键源码**:

### 第 7 章 · 堆表与元组:一行数据怎么落盘

- **核心问题**:一行数据(一个 tuple)在磁盘的 8KB 页里到底长什么样?行指针怎么让页不必顺序存也能快速定位某行?定长字段(int)和变长字段(TEXT)各自怎么存、为什么要对齐(padding)?`HeapTupleHeader` 里 `t_xmin`/`t_xmax`/`t_cid`/`t_ctid` 这些"看不见的列"是干嘛的?超过 ~2KB 的大字段为什么要被"切片"存到单独的 TOAST 表?
- **要讲清的"为什么"**:
  - 一行 = **固定 23 字节头** + 可选 null 位图 + 对齐填充 + 用户数据区;行指针 `lp_off` 指向它;
  - 四个"看不见的列"服务 MVCC:`t_xmin`(谁插入)/`t_xmax`(谁删)/`t_cid`(事务内第几条命令)/`t_ctid`(指向更新后的新版本,串起版本链);
  - 标志位**拆两套** `t_infomask`(可见性 + 锁,含 `HEAP_XMIN_COMMITTED` 等**提示位**,把 CLOG 查询缓存进 tuple 头)和 `t_infomask2`(属性数 + HOT/更新),按变更语义分组;
  - **NULL 用位图**,只有 `HEAP_HASNULL` 才存,数据区不占字节(几乎零成本);
  - **定长按 attalign 对齐**前补 padding(列顺序影响空间占用);**变长 varlena** 有 4 字节/1 字节两种 header,短 varlena(<127B)用 1 字节还免去对齐;
  - **TOAST** 在行超 ~2KB(`TOAST_TUPLE_THRESHOLD`)时触发,大字段切片(每片 ~2KB)挪到独立 TOAST 表,主表留 `varatt_external` 指针;四种策略 PLAIN/EXTERNAL/EXTENDED(默认)/MAIN 按固定四步算法生效;
  - `LP_DEAD` = 死元组待 VACUUM 清理——MVCC 的代价是死元组,VACUUM 来还债。
- **关键源码**:
  - 事务信息字段:[`HeapTupleFields`](../postgresql-17.0/src/include/access/htup_details.h#L122-L132)(`t_xmin`/`t_xmax`/`t_field3`);

### 第 8 章 · Buffer Pool:内存里的中转站

- **核心问题**:为什么不能每次读写都直接打磁盘?Buffer Pool 怎么用一块固定大小的共享内存缓存磁盘页?缓冲池满了该淘汰谁——**Clock-Sweep(时钟扫描)** 怎么工作,为什么不用 LRU?一个页被读到内存要经过几道映射?**脏页(dirty page)** 为什么既是性能命脉又是风险点,它什么时候被写回、由谁写?
- **要讲清的"为什么"**:
  - 内存随机访问几十纳秒 vs 磁盘寻道几毫秒,差十万倍——必须用**内存换速度**;`shared_buffers` 默认 128MB,生产经验值是物理内存 1/4;
  - Buffer Pool = **NBuffers 个固定槽位的数组** + 一张 `BufferTag → buf_id` 的 **hash 表**(分 128 个分区,把锁争用摊薄 128 倍);
  - `state` 把**标志位 + refcount + usage count 三合一打包进 32 位整数**,便于无锁 CAS;
  - 读页路径:查 hash 表 → 命中 pin 了用 / 未命中挑 victim(脏的先刷)→ 改 tag → 登记 → 从磁盘读入;
  - **pin(refcount)防淘汰,lock 防内容冲突**;锁序铁律:先 pin 后 lock,先 unlock 后 unpin;
  - **Clock-Sweep 近似 LRU 但更廉价还抗扫描**:指针转圈,usage count>0 减 1 给二次机会、usage count=0 且 refcount=0 淘汰;usage count 上限 5;
  - 脏页攒内存(快)但掉电就丢——`FlushBuffer` 里 `XLogFlush(recptr)` **强制日志先于数据落盘**(和 P5 WAL 咬合);
  - `shared_buffers` 不是越大越好:脏页刷盘压力、checkpoint 抖动、和 OS 缓存争内存。

### 第 9 章 · FSM 与 VM:两个辅助地图

- **核心问题**:`INSERT` 一行,数据库怎么**快速**知道几百万个堆页里哪一个还塞得下,而不必逐页翻找?**FSM** 怎么用一棵**树**把"每页剩余空间"压缩成 O(log n) 查询?**VM** 又是什么——它为每个堆页只存一个 bit,为什么能让 VACUUM 跳过整页扫描,又为什么能让"仅索引扫描"成为可能?
- **要讲清的"为什么"**:
  - FSM 把每个堆页的空闲空间压成**一个字节(0~255 类别,32 字节一档,255 特殊表示 ≥ MaxHeapTupleSize)**;再组织成**页内二叉树**(父=子之 max)+ **跨页三层树**(默认 3 层覆盖 2³² 堆页);
  - **为什么用树不用数组**:数组找"≥ X"是 O(n),树让"判断有没有"变 O(1)(看根)、"找具体哪个"变 O(log n);
  - 更新走**冒泡**:改叶子后逐层重算父节点 max,值没变就提前终止;底层变动传到上层靠 VACUUM 的 `FreeSpaceMapVacuum`;
  - **VM 每页 2 个 bit**:`ALL_VISIBLE`(全可见,让 VACUUM 和仅索引扫描跳页)、`ALL_FROZEN`(全冻结,让防回卷 VACUUM 也跳页);一页 1/4 字节,比 FSM 更省;
  - **一次 INSERT/UPDATE/DELETE 必清 VM 位**:页里一出现对某些事务不可见的 tuple,这页就不再"全可见",`PD_ALL_VISIBLE` 和 VM bit 成对清——这是 MVCC 不变量的直接体现(接 P4);
  - **设位写 WAL、清位不单独写**:VM 位虽是 hint,但"设 `ALL_VISIBLE`"因和 `PD_ALL_VISIBLE` 配对、涉及崩溃后一致性,必须 WAL;
  - FSM 和 VM 都是 **hint**:坏了只损失效率不损失正确性,权威数据只在堆表 main fork。

## 阅读建议

### 顺序

**强烈建议按 6 → 7 → 8 → 9 的顺序读**,因为层次是严格递进的:第 6 章的页布局是第 7 章 tuple 的容器、是第 8 章 Buffer Pool 的缓存单位、是第 9 章 FSM/VM 记录的对象;跳读会反复回头查"页是什么"。

### 精华章(时间紧可只读)

- **第 6 章(页)** + **第 8 章(Buffer Pool)**:这两章是本篇骨架,也是后续所有章节反复引用的地基。第 8 章尤其精华——它是全书主线"快 vs 不丢不乱"在单个机制(`FlushBuffer` 的 `XLogFlush`)上交汇的瞬间。
- 第 7 章 tuple 和第 9 章 FSM/VM 偏"字节级细节 / 辅助算法",可第二遍精读。

### 前置依赖

- **必读**:[第 1 章《第一性原理》](P0-01-第一性原理-为什么需要数据库.md)——数据的三个本性(尤其"磁盘慢、内存快但易失")是本篇全部设计的动因;第 1 章末尾已点过 `pd_lsn` 和 `xmin/xmax`。
- **可选**:[P1 第 5 章《执行器》](P1-05-执行器.md)——知道执行器把计划跑完、得到一批行之后,自然引出"这些行从哪取/往哪存"。P1 没读也不影响本篇,本篇从"数据物理形态"切入,和 SQL 路径解耦。

### 后续衔接

- **P3 索引篇**(第 10-13 章):第 6 章的行指针间接寻址、第 9 章 VM 的 `ALL_VISIBLE` 位,是 P3"仅索引扫描"能省掉回表的直接前提——本篇和索引篇在 VM 这一个 bit 上汇合。
- **P4 事务并发篇**(第 14-17 章):第 7 章的 `t_xmin`/`t_xmax`/`t_ctid`/提示位、第 9 章"改页必清 VM 位",都是 MVCC 的伏笔,P4 会把它们讲透。
- **P5 持久恢复篇**(第 18-20 章):第 6 章的 `pd_lsn`、第 8 章 `FlushBuffer` 的 `XLogFlush`,是"WAL write-ahead rule"在存储层的两个强制点——P5 会从日志侧把它们补全。
- **P6 进阶篇**(第 21 章 VACUUM):第 6/7 章的 `LP_DEAD`、第 9 章的 `ALL_FROZEN`,直接通向 VACUUM 和 anti-wraparound。
