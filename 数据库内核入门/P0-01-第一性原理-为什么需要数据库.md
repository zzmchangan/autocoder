# 第 1 章 · 第一性原理:为什么需要数据库

> **前置**:这是全书的起点,前置是[全书导言](全书导言-一篇看懂数据库内核全貌.md)——它已经用"一条 SQL 的旅程"把数据库全貌串了一遍,并立下"让数据快 / 让数据不丢不乱"这条主线和数据的三个本性。本章就把导言里那句轻描淡写的"数据库必须存在",**追问到底**:凭什么必须存在?没有它会怎样?

> **核心问题**:为什么需要数据库?它到底解决了什么"不解决就完全没法用"的问题?
>
> 这一章我们不碰任何具体机制(不解析、不优化、不开事务),只问一件事:**如果应用程序直接读写磁盘上的文件来存数据,到底会出什么乱子?这个乱子,又为什么非得靠一个专门的"数据库"程序站在应用和磁盘之间来解决?**
>
> **读完本章你会明白**:数据库不是"存数据的工具"(那只是文件系统),而是"让多人能同时、可靠、快速地操作一份数据"这个局能成立的**先决条件**;它的全部复杂性,都是被"用文件存数据的四个乱子"和"数据的三个本性"**逼**出来的。理解了这一章,你在后面任何一章遇到"为什么搞这么复杂",都能自己找到答案。

---

## 章首·一句话点破

导言里那句话:

> **数据库站在应用程序和磁盘之间,让应用可以放心地以为:我的 SQL 总能拿到正确结果、改的数据掉电不丢、和别人同时改也不打架。**

但这句话是**结论**,不是**理由**。这一章要倒过来:先把结论收起来,从最底层问起——

- 一群应用程序,**直接用文件存数据**,到底长什么样?
- 如果就这么干,会出什么事?
- "管数据"这件事,为什么非得是一个**专门的、独立的程序**(数据库),而不能"每个应用自己管自己的文件"?

我们一块一块拆。

---

## 一、先看清"没有数据库"的世界长什么样

要理解为什么需要数据库,得先看清"没有数据库"是个什么处境。

很多新手的第一反应是:"存数据?那不就是写文件吗?我用 `fopen`/`fwrite` 把数据写进一个文件,要用的时候 `fread` 读出来,不就完了?为什么还要搞个这么复杂的数据库?"

这话听起来有道理——而且**在单机、单人、数据量小**的时候,它确实没问题。可一旦数据量大起来、一旦有多个人(多个应用、多个请求)同时操作、一旦机器有可能崩溃——用裸文件存数据,就会暴露出四个要命的乱子。我们一个个看。

### 乱子一:写一半机器挂了,数据半残

第一个、也最基础的麻烦:**应用程序写文件,不是原子的**。

假设你要把"给用户 A 的余额减 100、给用户 B 的余额加 100"这个操作写进文件。它至少要写两处:改 A 的记录、再改 B 的记录。这两次写之间,如果**机器突然断电**(或者进程崩了、磁盘满了),会发生什么?

> **不这样会怎样**:A 的记录已经改成"减了 100",但 B 的记录还没来得及改。于是 100 块钱凭空消失了——文件里留下一个**半成品状态**:既不是"转账前",也不是"转账后",而是一个**谁都不想要的中间态**。更糟的是,这个半残状态会被下次启动的程序当成"真实数据"读出来,你的账上永远差了 100 块,而且没有任何线索告诉你"那次转账其实没完成"。

这就是用裸文件存数据的第一个致命问题:**没有"原子性"**。一次业务操作往往涉及多处修改,而文件系统只保证"每一次 `write` 调用"成功,不保证"一组相关的 `write` 要么全成功、要么全不做"。一旦中途出错,数据就停在半空。

### 乱子二:两人同时改同一份数据,互相覆盖

第二个麻烦,在有并发的时候爆发。

两个请求同时来了:请求 1 要把某商品的库存从 100 减到 90(卖出 10 个),请求 2 也要把库存从 100 减到 95(卖出 5 个)。如果它们各自读文件、各自计算、各自写回:

> **不这样会怎样**:两个请求**都读到"库存 = 100"**,请求 1 算出 90 写回,请求 2 算出 95 写回。最后文件里是 90 或 95——**反正不是正确的 85**。卖出去 15 个商品,库存却只减了 5 或 10。这就是经典的"**丢失更新(lost update)**"。如果这是银行账户或库存系统,每次并发都会出错,系统根本没法用。

用裸文件,你得自己在应用代码里加锁(`flock`、分布式锁),自己保证"同一份数据同一时刻只有一个写入者"。这套锁逻辑写在每个应用里,既重复又容易写错(谁能保证所有开发者都记得加锁?谁能保证锁的范围对?),而且跨多台机器时几乎没法正确实现。

### 乱子三:找一条数据,要扫遍整个文件

第三个麻烦,在数据量大的时候爆发。

你的用户表存了 1000 万行,存在一个文件里。现在要查 `id = 42` 的那个用户。怎么办?

> **不这样会怎样**:你只能**从头到尾把文件读一遍**,逐行比对,直到找到 id = 42 的那行。1000 万行,可能要扫几百 MB 甚至几个 GB 的数据,耗时几十秒。而绝大多数查询都只关心其中寥寥几行——为了找 1 行,读 1000 万行,99.9999% 的 I/O 全是浪费。

文件系统没有"按内容快速定位"的能力。你当然可以自己维护一个"目录"(id 到文件位置的映射),但那就等于**自己重新发明索引**——而且要保证它和数据文件始终一致(改了数据要同步改索引),又是另一摊麻烦。

### 乱子四:改了一半的数据,被别人读走

第四个麻烦,还是并发,但方向相反:这次是**读**的问题。

请求 1 正在执行那个转账(给 A 减 100、给 B 加 100),改了 A 还没改 B。就在这个**中间瞬间**,请求 2 来查"所有用户余额总和"做报表。它读到的是"A 已经减了、B 还没加"的状态——**总和凭空少了 100**。

> **不这样会怎样**:请求 2 拿到了一个**根本不存在的中间状态**当作"真实数据"去生成报表。这份报表是错的,而且没人能发现它错——因为请求 2 读到的那一刻,数据确实就是那个样子的。脏数据就这样无声无息地流进了业务系统。

用裸文件,你没法对"一次还没完成的修改"说"等它做完了再让别人读"——文件里的字节就是字节,谁打开都能看到当前内容,不管它是不是一个完整的、一致的状态。

### 小结:四个乱子,指向同一个缺口

把四个乱子叠在一起:

| 乱子 | 缺的是什么 |
|------|-----------|
| 写一半机器挂了,数据半残 | **原子性**——一组操作要么全做、要么全不做 |
| 两人同时改,互相覆盖 | **隔离性**——并发操作互相不能踩,要像排队 |
| 找条数据要扫全表 | **高效查询**——能按内容快速定位,不用遍历 |
| 改一半的数据被读走 | **一致性**——别人读到的必须是完整的、一致的状态 |

这四个缺口,合起来指向同一个需求:**应用程序和磁盘之间,需要一个专门的程序,替所有应用管好这份数据——保证操作原子、并发隔离、查询高效、状态一致,还要掉电不丢。**

这个"专门的程序",就是**数据库**。

## 二、四个乱子,逼出 ACID

上一节的四个缺口,前人早就踩过无数遍,并把"一个合格的数据管理程序必须满足什么"提炼成了四个字母:**ACID**。这四个字母不是凭空发明的,每一个都是上面某个乱子的**解药**。

### A — Atomicity(原子性):一组操作,要么全做,要么全不做

**治的乱子**:乱子一(写一半半残)。

数据库怎么做到?它给你一个叫**事务(transaction)**的东西:用 `BEGIN` 开始、`COMMIT` 确认、`ROLLBACK` 撤销。在 `COMMIT` 之前,这次事务的所有改动都还"悬着";一旦中途出错,一个 `ROLLBACK` 就把所有改动**全部撤销**,数据回到事务开始前的样子,绝不留半残状态。

> 这背后的核心机制(预写日志 WAL),是 P5 的主角。本章你只需记住:**数据库敢于承诺"一组操作要么全做、要么全不做",这个承诺就是原子性。**

### C — Consistency(一致性):数据永远满足约束

**治的乱子**:乱子四的一致性那一面。

一致性指的是:无论怎么操作,数据都必须**满足预先定义的规则**(主键唯一、外键存在、CHECK 约束、余额不能为负……)。事务不能把数据改成一个违反约束的状态。数据库通过约束检查,加上原子性和隔离性,共同保证事务执行前后,数据从一个"合法状态"变成另一个"合法状态",绝不会停在"非法状态"。

> 注意:一致性有一部分是**应用层**的责任(比如"转账前后总金额不变"这个业务规则,数据库并不直接知道)。数据库保证的是**它能强制执行的约束**(主键、外键、CHECK 等)不被破坏,并为应用层一致性提供原子性和隔离性这两块地基。

### I — Isolation(隔离性):并发事务,互相看不见对方的中间状态

**治的乱子**:乱子二(互相覆盖)和乱子四(读到中间状态)的隔离那一面。

隔离性说:多个事务并发执行时,**每个事务都应该感觉不到其他事务的存在**——仿佛整个数据库只有自己在用。这样请求 2 就不会读到请求 1 改了一半的中间数据,两个并发的"减库存"也不会互相覆盖。

> 完全的隔离(让并发事务的效果完全等同于"一个一个串行执行")代价很大,所以现实里数据库提供不同档位的**隔离级别**,让用户在"隔离得多干净"和"并发得多高"之间权衡。这是 P4 的主角,也是全书最难、最精华的部分——尤其 PostgreSQL 用的 **MVCC**(多版本并发控制)那一套,是 P4 第 17 章的核心。

### D — Durability(持久性):提交了,就掉电也不丢

**治的乱子**:乱子一的后半部分(掉电丢数据)。

持久性说:一旦事务 `COMMIT` 成功,数据库向你**保证**这次改动已经永久生效——哪怕下一秒整栋机房断电,重启后这份数据也还在。

> 这个保证怎么兑现?靠的还是 **WAL**:数据库在 `COMMIT` 时,先把这次改动的日志**强制写到磁盘**,然后才返回"成功"。只要日志落了盘,哪怕数据本身还没写盘,掉电后也能靠日志重放恢复。详见 P5。

### ACID 一句话:四个乱子的总账单

| 乱子 | ACID 解药 | 兑现它的机制 |
|------|-----------|--------------|
| 写一半半残 | **A** 原子性 | 事务 + WAL 回滚 |
| 互相覆盖 | **I** 隔离性 | 锁 / MVCC |
| 读到中间状态 | **I** + **C** | MVCC / 约束 |
| 掉电丢数据 | **D** 持久性 | WAL 先落盘 |

而乱子三(找数据慢)不在 ACID 里——它是"性能"问题,由**索引和查询引擎**解决(那是"让数据快"那一侧的故事)。所以完整的图景是:**ACID(不丢不乱)+ 索引/查询(快)= 一个数据库的全部职责。**

## 三、数据的三个本性,逼出三大子系统

ACID 和索引是"需求"。现在追问一层:**为什么兑现这些需求,会那么复杂?为什么数据库要分成那么多个子系统?** 答案在数据本身的**三个本性**。这三个本性,是全书一切复杂设计的总根源。

### 本性一:磁盘慢、内存快但易失

这一条最要命,因为它自相矛盾。

- **磁盘慢**:随机访问一次磁盘,要几毫秒;而内存只要几十纳秒。差了**十万倍**。所以数据库绝不能每条 SQL 都去读磁盘,否则慢得没法用。
- **内存快,但易失**:内存一掉电就全没了。所以你不能把数据只放在内存里,否则断电就全完了。

> **不这样会怎样**:只用磁盘,慢;只用内存,丢。要又快又不丢,就得**两个都用**:平时读写都在内存里(Buffer Pool,快),同时把改动记到磁盘上的日志(WAL,不丢)。这一对矛盾,直接逼出了**存储引擎(Buffer Pool)和持久性机制(WAL / Checkpoint)**两大块——也就是 P2 存储篇和 P5 持久篇的全部内容。

### 本性二:数据要共享,但共享会冲突

数据库从诞生起就是**多用户**的:几十上百个连接同时读写同一份数据。

- "共享"是刚需:数据集中存、大家都能用,这正是数据库相对于"每个应用自己存文件"的核心价值。
- 但"共享"会"冲突":两个事务同时改同一行,谁先谁后?一个事务改一半,另一个能读吗?(就是第二节的乱子二、四。)

> **不这样会怎样**:如果不管并发,任由大家一起改,就是丢失更新、脏读、不可重复读满天飞(第二节那几个乱子)。要管,就得有一套**并发控制**机制——让并发事务既打得开(高并发),又不打架(隔离正确)。这套机制就是**事务(ACID)、锁、隔离级别、MVCC**,也就是 P4 的全部。这是数据库区别于文件系统最深刻的地方,也是全书最复杂的部分。

### 本性三:用户说"要什么",机器只会"按位置读"

这一条最容易被忽略,却贯穿数据库的"查询"那一半。

- 应用程序用 **SQL** 说的是"**我要什么**":`SELECT * FROM users WHERE id = 42`——它只描述了"要 id = 42 的行",**完全没说**这个数据在磁盘哪个位置、怎么取。
- 但磁盘只会**"按位置读"**:你要给它一个文件偏移量,它才给你那块字节。

> **不这样会怎样**:这个"要什么"和"怎么取"之间的鸿沟,如果让应用程序自己填,那每个应用都得知道"users 表存在哪个文件、id = 42 在第几页第几行"——这就把存储细节漏给了应用,一旦换存储格式,所有应用都得改。所以要有一层专门的东西,负责把"SQL 描述的意图"翻译成"磁盘上的具体操作",并尽量翻译得**省**(用索引、走最优计划)。这层就是**查询引擎(解析 → 优化 → 执行)+ 索引**,也就是 P1 和 P3 的全部。

### 三个本性 → 三大子系统

把三个本性对应到全书的篇:

| 数据的本性 | 逼出的子系统 | 对应的篇 |
|---|---|---|
| 磁盘慢、内存易失 | 存储引擎(Buffer Pool)+ 持久性(WAL) | P2 存储、P5 持久 |
| 共享会冲突 | 事务 + 锁 + MVCC | P4 事务并发 |
| 声明式查询 vs 物理存取 | 查询引擎 + 索引 | P1 查询、P3 索引 |

**这就是全书 P1~P5 五大篇的总根源。** 你后面在任何一章觉得"这设计怎么这么绕",去找这三个本性里的哪个在作怪——准能找到。

## 四、主线二分法:让数据"快" vs 让数据"不丢不乱"

把前面两节合起来,就得到了贯穿全书的主线:

> **数据库一半在"让数据快"(用内存换速度、用索引换查找),一半在"让数据不丢、不乱"(用日志换持久、用事务/MVCC 换一致)。**

这两半,分别对应全书的两组篇:

- **"快"那一侧**(对付"磁盘慢"和"声明式查询"):
  - **P1 查询引擎**:把 SQL 翻译成执行计划,选最省的路径。
  - **P2 Buffer Pool**:把热数据搬进内存,别等磁盘。
  - **P3 索引**:O(log n) 定位,别扫全表。

- **"不丢不乱"那一侧**(对付"易失"和"共享冲突"):
  - **P4 事务并发**:ACID + 锁 + MVCC,让多人同时改也不乱。
  - **P5 持久恢复**:WAL + Checkpoint,让数据掉电不丢。

> 后面每一章,你要做的只是不断回到这个二分法,问一句:**"这章是在让数据快,还是在让数据不丢不乱?"** 就不会迷路。

## 关键源码精读:三个奠基结构,三个本性的代码化身

本章是概念章,代码点到为止。但我们看三个 PostgreSQL 里最奠基的结构体——它们正好分别对应数据的三个本性,让你提前在代码里"摸到"全书要讲的三大子系统。**这一节不求懂细节,只求建立"本性 ↔ 代码"的对应感。**

### 1. 元组头 `HeapTupleHeaderData`:并发本性的化身

第二节我们说,要让并发事务不打架、让每个事务看到"自己该看到的版本"。这个承诺怎么落到代码里?答案是:**PostgreSQL 给每一行数据(tuple)都配上了一个"元组头"**,里面记着这行的来历。

先看这个头里最关键的部分——记录事务信息的 `HeapTupleFields`:

> [src/include/access/htup_details.h:122-132](../postgresql-17.0/src/include/access/htup_details.h#L122-L132)

```c
typedef struct HeapTupleFields
{
	TransactionId t_xmin;		/* inserting xact ID */
	TransactionId t_xmax;		/* deleting or locking xact ID */

	union
	{
		CommandId	t_cid;		/* inserting or deleting command ID, or both */
		TransactionId t_xvac;	/* old-style VACUUM FULL xact ID */
	}			t_field3;
} HeapTupleFields;
```

注意这两个字段:

- **`t_xmin`**——"**插入这行的事务 ID**"。每一行都知道自己是被哪个事务"生"出来的。
- **`t_xmax`**——"**删除(或锁定)这行的事务 ID**"。一行被删,不是真的从磁盘抹掉,而是给它打个"已被某某事务删除"的标记。

这个 `HeapTupleFields` 嵌在完整的元组头 `HeapTupleHeaderData` 里:

> [src/include/access/htup_details.h:153-181](../postgresql-17.0/src/include/access/htup_details.h#L153-L181)

```c
struct HeapTupleHeaderData
{
	union
	{
		HeapTupleFields t_heap;
		DatumTupleFields t_datum;
	}			t_choice;

	ItemPointerData t_ctid;		/* current TID of this or newer tuple ... */
	...
	uint16		t_infomask2;	/* number of attributes + various flags */
	uint16		t_infomask;		/* various flag bits, see below */
	uint8		t_hoff;			/* sizeof header incl. bitmap, padding */
	...
	bits8		t_bits[FLEXIBLE_ARRAY_MEMBER];	/* bitmap of NULLs */
};
```

**为什么每行都要带 `t_xmin`/`t_xmax`?** 因为这是 **MVCC(多版本并发控制)** 的地基。PostgreSQL 的做法是:**更新一行,不是原地改,而是写一个新版本;旧行不马上删,而是靠 `t_xmax` 标记。** 这样,不同事务根据自己的"快照",能各自看到合适的版本——读的人看旧版本、写的人写新版本,读不阻塞写、写不阻塞读。第二节乱子二、四(并发冲突),就是被这小小的 `t_xmin`/`t_xmax` 化解的。

> 这是 P4 第 17 章(MVCC)的主角。本章你只需记住:**"让并发事务各自看到正确版本"这个承诺,在代码里浓缩成每行自带的 `t_xmin`/`t_xmax` 两个事务 ID。**

### 2. 日志记录 `XLogRecord`:易失本性的化身

第四节说,要让数据掉电不丢,靠的是 WAL——**改数据之前,先把改动记成一条日志**。这条日志在代码里长什么样?

> [src/include/access/xlogrecord.h:41-50](../postgresql-17.0/src/include/access/xlogrecord.h#L41-L50)

```c
typedef struct XLogRecord
{
	uint32		xl_tot_len;		/* total len of entire record */
	TransactionId xl_xid;		/* xact id */
	XLogRecPtr	xl_prev;		/* ptr to previous record in log */
	uint8		xl_info;		/* flag bits, see below */
	RmgrId		xl_rmid;		/* resource manager for this record */
	/* 2 bytes of padding here, initialize to zero */
	pg_crc32c	xl_crc;			/* CRC for this record */

	/* XLogRecordBlockHeaders and XLogRecordDataHeader follow, no padding */
```

每一行字段都对应一个设计意图:

- **`xl_xid`**:这条记录属于哪个事务。崩溃恢复时,数据库靠它判断"这个事务到底提交了没有"。
- **`xl_rmid`** + **`xl_info`**:**"这条记录是什么类型的改动"**——是改了某个堆表页、还是建了索引、还是别的事。PostgreSQL 把不同子系统的日志交给各自的"资源管理器(resource manager)"解读,`xl_rmid` 就是这个分工的代号。
- **`xl_tot_len`** + **`xl_crc`**:整条记录多长、校验和。崩溃恢复要能正确地一条条解析日志,长度和 CRC 是保证日志没被写坏的护栏。

**为什么先记这个、再改数据?** 因为**日志是顺序追加写**(往文件末尾接着写,极快),而改数据是**随机写**(改散落在各处的页,慢)。数据库赌的是:只要日志落了盘,数据哪怕还没写盘,掉电后也能靠重放日志(把每条 `XLogRecord` 重新执行一遍)把改动补回来。所以 `COMMIT` 时,数据库真正等的是"**日志写盘**",而不是"数据写盘"——这就是持久性(D)能兑现的真相。

> 这是 P5 第 18 章(WAL)的主角。本章你只需记住:**"掉电不丢"这个承诺,在代码里就是这一条条追加写的 `XLogRecord`。**

### 3. 页头 `PageHeaderData`:存储地基,快与持久的交汇点

最后看数据真正"住"的容器——**页(page)**。PostgreSQL 把所有数据(堆表、索引)都切成一个个 **8KB 的页**,磁盘按页整块读写。每个页的开头,是这个页的"身份证":

> [src/include/storage/bufpage.h:155-168](../postgresql-17.0/src/include/storage/bufpage.h#L155-L168)

```c
typedef struct PageHeaderData
{
	/* XXX LSN is member of *any* block, not only page-organized ones */
	PageXLogRecPtr pd_lsn;		/* LSN: next byte after last byte of xlog
								 * record for last change to this page */
	uint16		pd_checksum;	/* checksum */
	uint16		pd_flags;		/* flag bits, see below */
	LocationIndex pd_lower;		/* offset to start of free space */
	LocationIndex pd_upper;		/* offset to end of free space */
	LocationIndex pd_special;	/* offset to start of special space */
	uint16		pd_pagesize_version;
	TransactionId pd_prune_xid; /* oldest prunable XID, or zero if none */
	ItemIdData	pd_linp[FLEXIBLE_ARRAY_MEMBER]; /* line pointer array */
} PageHeaderData;
```

留意三个字段,它们分别牵出全书后面的几条主线:

- **`pd_lsn`**:这个页**最后一次被改动时,对应的 WAL 日志位置**。注意——它把"页"和前面讲的 `XLogRecord` 日志**勾连**起来了!崩溃恢复时,数据库比较每个页的 `pd_lsn` 和日志位置,就知道"这个页的改动有没有进日志、要不要重放"。**存储和持久性,在这里交汇。**
- **`pd_lower` / `pd_upper`**:页内空闲空间的起止位置。页是从两头往中间长的(头部放行指针 `pd_linp`,尾部放实际行数据),中间的空隙就是还能塞新行的地方。Buffer Pool 和堆表的增删改,天天在摆弄这两个指针。
- **`pd_linp[]`**:**行指针数组**。页里每一行,在这里有个条目,指向这行在页里的实际位置——所以"按页读"之后,还要靠行指针定位到具体某行。

**为什么数据要按 8KB 页存,而不是按行?** 因为磁盘(机械盘尤其,SSD 也一样)按块读写的效率远高于按字节;8KB 是 PostgreSQL 在"I/O 粒度"和"内存占用"之间权衡出的默认大小。这个设计决策,是 P2 第 6 章(页面)的主角。

> 本章你只需记住:**数据按 8KB 页存;每个页有 `pd_lsn` 指向它的日志、有 `pd_lower/pd_upper` 标着空闲空间、有 `pd_linp` 索引着页里的每一行。** 它是"快"(Buffer Pool 按页缓存)和"持久"(页通过 `pd_lsn` 关联 WAL)共同的地基。

### 三个结构,一张全景

把这三个奠基结构放一起,你其实已经看到了全书要讲的三层,以及它们怎么彼此勾连:

```text
   应用看到:   SQL(声明式:要什么)             ← 本性三 → P1 查询引擎
                   │ (优化器翻译成"怎么取")
   存储单元:   页 Page( pd_lsn ↔ WAL )         ← 本性一 → P2 存储
                   │ (页里塞着一行行 tuple)
   每一行:     tuple( t_xmin/t_xmax ↔ 事务 )   ← 本性二 → P4 事务
                   │ (改之前先记)
   持久保证:   XLogRecord( xl_xid ↔ 事务 )      ← 本性一 → P5 持久
```

页的 `pd_lsn` 指向 WAL 记录,元组的 `t_xmin`/`t_xmax` 关联事务,WAL 的 `xl_xid` 也关联事务——**页、tuple、事务、WAL 四个概念,通过几个 ID 字段彼此勾连,织成了数据库内核的整张网。** 后面二十多章,就是把这网的每一根线拆开讲透。

## 章末小结

### 用一句话回顾本章

这一章我们做了一件纯粹的事:**搞清楚"数据库为什么非存在不可"**。

答案不在任何一个机制里,而在"用裸文件存数据"的**四个乱子**里:

1. **写一半机器挂了,数据半残** → 缺原子性;
2. **两人同时改,互相覆盖** → 缺隔离性;
3. **找条数据要扫全表** → 缺高效查询;
4. **改一半的数据被读走** → 缺一致性的可见性。

这四个乱子,逼出了 **ACID**(原子性、一致性、隔离性、持久性)加上**索引/查询**——这就是一个数据库的全部职责。而兑现这些职责之所以复杂,是因为数据带着**三个本性**:磁盘慢且内存易失、共享会冲突、声明式查询对不上物理存取。这三个本性,逼出了全书的 **P1~P5 五大篇**。

最后,这一切可以浓缩成一句主线:**数据库一半在"让数据快",一半在"让数据不丢、不乱"。**

### 本章在全书主线中的位置

这一章我们把全书主线**从根上立住了**:

- **ACID + 索引**,是数据库要兑现的**承诺**(需求侧);
- **三个本性 → 五大篇**,是兑现承诺的**全部手段**(供给侧)。

后面每一章,你要做的只是不断回到这俩锚,问一句:"这章是在**让数据快**,还是在**让数据不丢不乱**?它对付的是三个本性里的哪一个?"

### 五个"为什么"清单

如果你只能记五件事,记这五件:

1. **为什么需要数据库**:用裸文件存数据有四个乱子(半残、覆盖、扫全表、脏读),逼出一个专门管数据的程序。
2. **ACID 各治什么**:A 治半残、I 治并发冲突(覆盖/脏读)、C 治一致性、D 治掉电丢失;加上索引治"慢"。
3. **复杂性从哪来**:数据的三个本性——磁盘慢且易失、共享会冲突、声明式对不上物理。
4. **全书主线**:让数据"快"(Buffer Pool / 索引 / 优化器)vs 让数据"不丢不乱"(WAL / 事务 / MVCC)。
5. **三个奠基结构在代码里长啥样**:`HeapTupleHeaderData`(xmin/xmax)、`XLogRecord`、`PageHeaderData`——分别对应并发、易失、存储三个本性。

### 想继续深入,该往哪钻
