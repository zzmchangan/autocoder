# 第 13 章 · 索引与堆的配合:回表、覆盖索引、仅索引扫描

> **前置**:前面三章(P3 第 10~12 章)讲了索引本身——B 树怎么 O(log n) 定位、Hash/GiST/GIN/BRIN 各为哪类查询而生。但有一个事实一直被我们**轻描淡写地滑过去**:PostgreSQL 的索引叶子节点里,除了索引键值,通常只存一个**指向堆行位置的 TID**(第 7 章讲过,TID = `(块号, 行指针偏移)`)。索引能告诉你"满足条件的行**在哪**",却**给不出这一行的完整内容**。本章就讲清这最后一步:**从索引里的 TID 到用户拿到手的整行数据,中间这条"回表"之路有多贵,以及怎么把它省掉**。

> **核心问题**:普通索引扫描(Index Scan)拿到一个 TID 后,为什么几乎总是要**回表**——回到堆表里把这个 TID 指向的页读进来、取出完整 tuple?这一步是**随机 I/O**(命中的行散落在不同堆页里),代价能有多大?什么情况下能干脆**不回表**,只用索引里的数据就把查询答完——也就是**仅索引扫描(Index-Only Scan)**?它要两个苛刻前提:**① 索引里就有查询要的全部列;② 这些行所在的堆页对当前事务全可见(第 9 章讲过,即 VM 的 `ALL_VISIBLE` 位为真)**。第二点最反直觉——"数据明明在索引里,凭什么还要去查堆页的可见性?"本章把这条逻辑彻底讲透。再顺带讲**覆盖索引(Covering Index)**:`CREATE INDEX ... INCLUDE (列)` 怎么把额外列塞进索引、帮仅索引扫描达成第一个前提;以及为什么 `SELECT` 少选几列,执行计划有时就从 Index Scan 变成了 Index-Only Scan。

> **读完本章你会明白**:三种走索引的扫描(Index Scan / Bitmap Scan / Index-Only Scan)各自的 I/O 路径和适用场景;"回表"在源码里就是一次 `index_fetch_heap`,为什么它是随机 I/O 的大头;仅索引扫描为什么还要查 VM——索引里的 tuple 是"历史快照",不知道堆里对应行是否已被删改,只有 VM 的 `ALL_VISIBLE` 才能"免回堆担保可见性";EXPLAIN ANALYZE 里那个 `Heap Fetches` 数字到底在告诉你什么;覆盖索引的 INCLUDE 列为何不参与排序键、却能被仅索引扫描直接取用。

> **如果一读觉得太难**:先记三件事——① 普通索引扫描=索引页定位 TID + **每个 TID 回堆表取整行**(随机 I/O);仅索引扫描=**只读索引页就把查询答完**,前提是索引里有全部列且堆页全可见。② "全可见"靠 VM 的 `ALL_VISIBLE` 位担保(第 9 章),它一旦被清(页里有 INSERT/UPDATE/DELETE 的痕迹),仅索引扫描就只能老老实实回堆验证可见性,这就是 `Heap Fetches` 的来源。③ 覆盖索引用 `INCLUDE` 把非键列塞进索引叶子,让"索引里有全部列"这个前提成立;Bitmap 扫描则是"先在索引里收集所有 TID 位图、再按页批量回堆",把随机 I/O 改造成"近乎顺序"。

## 一、索引叶子只存 TID:回表这道"最后一公里"

把第 12 章的一个结论再钉死一次:**PostgreSQL 的索引,默认不存完整行数据**。一个 B 树索引 `idx_users_id ON users(id)`,叶子节点里每条记录大致是 `(id 的值, TID)`——比如 `(42, (5, 7))` 表示"id=42 的那行在块 5 的第 7 个行指针处"。它**不**存这行的 `name`、`email`、`created_at`。

为什么这么设计?因为 PG 的堆表是"无序、可变"的——一行被 UPDATE 会产生新版本、挪到别的页(第 17 章详讲),而索引要尽量稳定。如果每个索引都把整行数据冗余存一份,一张表上有 5 个索引,数据就被存了 5 份,存储膨胀、写入放大都不可接受。所以索引只存"路标"(TID),真正的数据**只在堆表里有一份**。

### 不这样会怎样:有了 TID,但用户要的是整行

> **不这样会怎样**:你执行 `SELECT * FROM users WHERE id = 42`。优化器决定走 `idx_users_id`。索引叶子给出 `TID = (5, 7)`。可你要的是整行(`*`),光一个 TID 不够——`name`、`email` 这些值索引里根本没有。**执行器必须拿着 TID 回到堆表,把块 5 读进 Buffer Pool,沿着行指针取出完整 tuple,才能把这一行返回给用户。** 这一步,就是"回表"(英文叫 heap fetch,或 table fetch by TID)。

这条路径在执行器源码里看得清清楚楚。普通索引扫描的主循环是 `IndexNext`:

> [src/backend/executor/nodeIndexscan.c:130-150](../postgresql-17.0/src/backend/executor/nodeIndexscan.c#L130-L150)

```c
	while (index_getnext_slot(scandesc, direction, slot))
	{
		CHECK_FOR_INTERRUPTS();

		/*
		 * If the index was lossy, we have to recheck the index quals using
		 * the fetched tuple.
		 */
		if (scandesc->xs_recheck)
		{
			econtext->ecxt_scantuple = slot;
			if (!ExecQualAndReset(node->indexqualorig, econtext))
			{
				/* Fails recheck, so drop it and loop back for another */
				InstrCountFiltered2(node, 1);
				continue;
			}
		}

		return slot;
	}
```

看起来就是个 `while` 循环不断取下一个 tuple。但"取下一个 tuple"的 `index_getnext_slot` 内部,藏着回表的真相:

> [src/backend/access/index/indexam.c:672-702](../postgresql-17.0/src/backend/access/index/indexam.c#L672-L702)

```c
bool
index_getnext_slot(IndexScanDesc scan, ScanDirection direction, TupleTableSlot *slot)
{
	for (;;)
	{
		if (!scan->xs_heap_continue)
		{
			ItemPointer tid;

			/* Time to fetch the next TID from the index */
			tid = index_getnext_tid(scan, direction);

			/* If we're out of index entries, we're done */
			if (tid == NULL)
				break;

			Assert(ItemPointerEquals(tid, &scan->xs_heaptid));
		}

		/*
		 * Fetch the next (or only) visible heap tuple for this index entry.
		 * If we don't find anything, loop around and grab the next TID from
		 * the index.
		 */
		Assert(ItemPointerIsValid(&scan->xs_heaptid));
		if (index_fetch_heap(scan, slot))
			return true;
	}

	return false;
}
```

这段是普通索引扫描的"心脏",两步交替:

1. **`index_getnext_tid`**:问索引要下一个 TID。这一步在索引的 B 树/Hash 里走,只读**索引页**,顺序性强、缓存友好。
2. **`index_fetch_heap`**:拿着 TID 回堆表取整行。这就是**回表**。

`index_fetch_heap` 内部最终调到 `table_index_fetch_tuple`,它会**按 TID 的块号去读那个堆页**(可能要先从磁盘读到 Buffer Pool,如果不在缓存里),再沿行指针取出 tuple,还要做可见性判断。注意注释里那句 `Fetch the next (or only) visible heap tuple`——回表不只是"取数据",还要确认这行**对当前事务可见**(下一节讲为什么这点要紧)。

> [src/backend/access/index/indexam.c:631-655](../postgresql-17.0/src/backend/access/index/indexam.c#L631-L655)

```c
bool
index_fetch_heap(IndexScanDesc scan, TupleTableSlot *slot)
{
	bool		all_dead = false;
	bool		found;

	found = table_index_fetch_tuple(scan->xs_heapfetch, &scan->xs_heaptid,
									scan->xs_snapshot, slot,
									&scan->xs_heap_continue, &all_dead);

	if (found)
		pgstat_count_heap_fetch(scan->indexRelation);
	...
	return found;
}
```

那个 `pgstat_count_heap_fetch` 就是 EXPLAIN ANALYZE 里 `Heap Fetches` 计数的来源(后面细讲)。

### 回表的代价:为什么是随机 I/O 的大头

关键在于:索引返回的 TID 序列,对应到堆页上是**散乱**的。假设你要 `WHERE age BETWEEN 20 AND 30`,索引按 `age` 排序返回 TID,但这些行的物理存储位置和 `age` 毫无关系——它们是当初 `INSERT` 时被塞进"当时有空位的页"(第 9 章 FSM 选页)的,散落在表的几百几千个页里。

> **不这样会怎样**:于是每取一个 TID 回一次堆,读的就是一个**可能完全没缓存、和上一个 TID 八竿子打不着的页**。如果一次查询命中 1000 行,最坏就是 1000 次**随机 I/O**(每次读一个 8KB 页,只为取其中一行)。随机 I/O 在机械盘上是毫秒级、比顺序 I/O 慢一到两个数量级;就算 SSD,随机访问的延迟也远高于顺序。这就是为什么"走索引"并不总是比"顺序扫表"快——当命中行数占总行数的比例较高时,随机回表的代价会超过直接顺序扫一遍堆表。第 10 章讲过这个"索引选择性"的权衡,本章从执行路径上把代价落实。

这就是回表:索引给了你"在哪",但"取出来"这一步,是随机 I/O 的代价大头。后面三节,分别讲三种办法对付它。

## 二、Bitmap 扫描:先把 TID 攒成位图,再批量有序回堆

对付"随机回表"的第一个办法,是**把回堆的顺序理顺**。

普通索引扫描的毛病在于:索引按键值顺序给 TID,可堆页的物理顺序和键值顺序无关,导致回堆时东一页西一页。**Bitmap 扫描**的做法分两段:**① 先在索引里把所有满足条件的 TID 全取出来,塞进一个"位图"(TIDBitmap),位图按堆页组织——记录"块 5 上有第 3、7、12 行命中,块 9 上有第 1 行……";② 然后按块号从小到大的顺序,逐个堆页批量回表。** 这样回堆就从"随机跳"变成了"按物理顺序一页页走",I/O 模式从随机改造为近乎顺序。

源码上,这分两个执行器节点:`BitmapIndexScan` 负责第①步(只碰索引,产 TID 位图),`BitmapHeapScan` 负责第②步(按位图回堆)。注意 `BitmapIndexScan` 不像普通扫描那样"一次吐一个 tuple",它是个 `MultiExec` 节点——一次性把整个位图算完交给上层:

> [src/backend/executor/nodeBitmapIndexscan.c:99-114](../postgresql-17.0/src/backend/executor/nodeBitmapIndexscan.c#L99-L114)

```c
	/*
	 * Get TIDs from index and insert into bitmap
	 */
	while (doscan)
	{
		nTuples += (double) index_getbitmap(scandesc, tbm);

		CHECK_FOR_INTERRUPTS();

		doscan = ExecIndexAdvanceArrayKeys(node->biss_ArrayKeys,
										   node->biss_NumArrayKeys);
		if (doscan)				/* reset index scan */
			index_rescan(node->biss_ScanDesc,
						 node->biss_ScanKeys, node->biss_NumScanKeys,
						 NULL, 0);
	}
```

`index_getbitmap` 让索引 AM 把所有命中 TID 灌进 `tbm`(TIDBitmap)。位图内部用两种表示:命中行少时用"精确模式"(exact,记下每个 TID);命中行多到放不进 `work_mem` 时,退化成"有损模式"(lossy,**只记到页粒度**——"块 5 上至少有一行命中",具体哪几行忘了),到时回堆后得**逐行重新检查 WHERE 条件**。这就是 `BitmapHeapNext` 里这段的来历:

> [src/backend/executor/nodeBitmapHeapscan.c:213-313](../postgresql-17.0/src/backend/executor/nodeBitmapHeapscan.c#L213-L313)

```c
	for (;;)
	{
		bool		valid_block;

		CHECK_FOR_INTERRUPTS();

		/*
		 * Get next page of results if needed
		 */
		if (tbmres == NULL)
		{
			...
			node->tbmres = tbmres = tbm_iterate(tbmiterator);
			...
			if (tbmres == NULL)
			{
				/* no more entries in the bitmap */
				break;
			}
			...
			valid_block = table_scan_bitmap_next_block(scan, tbmres);

			if (tbmres->ntuples >= 0)
				node->exact_pages++;
			else
				node->lossy_pages++;
			...
		}
		...
		/*
		 * Attempt to fetch tuple from AM.
		 */
		if (!table_scan_bitmap_next_tuple(scan, tbmres, slot))
		{
			/* nothing more to look at on this page */
			node->tbmres = tbmres = NULL;
			continue;
		}

		/*
		 * If we are using lossy info, we have to recheck the qual conditions
		 * at every tuple.
		 */
		if (tbmres->recheck)
		{
			econtext->ecxt_scantuple = slot;
			if (!ExecQualAndReset(node->bitmapqualorig, econtext))
			{
				/* Fails recheck, so drop it and loop back for another */
				InstrCountFiltered2(node, 1);
				ExecClearTuple(slot);
				continue;
			}
		}

		/* OK to return this tuple */
		return slot;
	}
```

`tbm_iterate` 按**块号升序**产出位图条目,所以回堆是顺序的。`exact_pages`/`lossy_pages` 两个计数器就是 EXPLAIN ANALYZE 里 `Exact Heap Bitmaps` / `Lossy Heap Bitmaps` 的来源——`lossy` 不为 0 说明位图太大被压缩到页粒度了,得逐行 recheck。

### 什么时候 Bitmap 比普通 Index Scan 更优

- **命中行数较多**(比如几千几万行)、且**散布在大量页上**:普通 Index Scan 每行一次随机回表,代价爆炸;Bitmap 把回表顺序化,I/O 模式改善明显。
- **要回堆**(查询需要索引里没有的列):Bitmap 的全部价值在于"理顺回表顺序",如果根本不用回表(仅索引扫描,见下节),Bitmap 反而是白费功夫——它多了一次"先攒位图"的开销,却没有可省的随机 I/O。所以**Bitmap 扫描和仅索引扫描是互斥的**:优化器在 `check_index_only` 成立时优先仅索引扫描,不成立才考虑 Bitmap。
- **多个索引可以 AND/OR 组合**:Bitmap 的位图能做集合运算,把多个索引的命中 TID 取交/并。这是普通 Index Scan 做不到的。

反过来,命中行数极少(比如主键等值查一行),普通 Index Scan 直接回表一次就完事,Bitmap 的"先攒位图"反而是额外开销。这是优化器代价模型要权衡的。

> Bitmap 扫描**没有消除回表**,只是把回表的随机 I/O 改造成顺序 I/O。真正能消除回表的,是下一节。

## 三、仅索引扫描:索引里就有全部列,凭什么还要查 VM

最彻底地对付回表的办法,是**根本不回表**。如果查询要的所有列,索引里恰好都有,那理论上从索引里直接取数据就行,何必再去读堆页?

这就是**仅索引扫描(Index-Only Scan)**。听起来简单,但它的成立条件里藏着一个反直觉的硬约束,必须讲透。

### 前提一:索引里要有查询要的全部列

`SELECT id FROM users WHERE id < 100`,而 `users` 上有个 `idx_users_id ON (id)` 的 B 树索引。`id` 既是查询条件(`WHERE id < 100`)又是输出列(`SELECT id`)——它**就在索引里**。这一条满足。

但 `SELECT id, name FROM users WHERE id < 100` 就不行了——`name` 不在 `idx_users_id` 里,光读索引拿不到 `name`,还得回表。

优化器怎么判断"索引里是否有全部列"?靠 `check_index_only`:

> [src/backend/optimizer/path/indxpath.c:1729-1800](../postgresql-17.0/src/backend/optimizer/path/indxpath.c#L1729-L1800)

```c
static bool
check_index_only(RelOptInfo *rel, IndexOptInfo *index)
{
	bool		result;
	Bitmapset  *attrs_used = NULL;
	Bitmapset  *index_canreturn_attrs = NULL;
	...
	/*
	 * First, identify all the attributes needed for joins or final output.
	 */
	pull_varattnos((Node *) rel->reltarget->exprs, rel->relid, &attrs_used);

	/*
	 * Add all the attributes used by restriction clauses; ...
	 */
	foreach(lc, index->indrestrictinfo)
	{
		RestrictInfo *rinfo = (RestrictInfo *) lfirst(lc);
		pull_varattnos((Node *) rinfo->clause, rel->relid, &attrs_used);
	}

	/*
	 * Construct a bitmapset of columns that the index can return back in an
	 * index-only scan.
	 */
	for (i = 0; i < index->ncolumns; i++)
	{
		int			attno = index->indexkeys[i];
		...
		if (index->canreturn[i])
			index_canreturn_attrs =
				bms_add_member(index_canreturn_attrs,
							   attno - FirstLowInvalidHeapAttributeNumber);
	}

	/* Do we have all the necessary attributes? */
	result = bms_is_subset(attrs_used, index_canreturn_attrs);
	...
	return result;
}
```

逻辑直白:**把"查询用到的所有列"(`attrs_used`,来自输出列表和过滤条件)和"索引能返回的列"(`index_canreturn_attrs`)做集合包含判断——前者是后者的子集,仅索引扫描就可行**。这就是为什么"SELECT 少选几列,计划可能从 Index Scan 变成 Index-Only Scan"——少选一列,`attrs_used` 就可能变成 `index_canreturn_attrs` 的子集,`check_index_only` 翻盘为真。

注意 `canreturn[i]` 这个判断:它问的是"索引的第 i 列能不能在仅索引扫描时被取出来"。对 B 树,所有列(包括下文 INCLUDE 的列)都能返回;但对 GiST/GIN 等某些 AM,部分列可能取不出原值(只能用于匹配),那就不能仅索引扫描。这个 `canreturn` 数组在加载索引信息时逐列调 `index_can_return` 填充:

> [src/backend/optimizer/util/plancat.c:296-302](../postgresql-17.0/src/backend/optimizer/util/plancat.c#L296-L302)

```c
			info->canreturn = (bool *) palloc(sizeof(bool) * ncolumns);

			for (i = 0; i < ncolumns; i++)
			{
				info->indexkeys[i] = index->indkey.values[i];
				info->canreturn[i] = index_can_return(indexRelation, i + 1);
			}
```

循环到 `ncolumns`(即 `indnatts`,**包括 INCLUDE 列**),所以 INCLUDE 列也进了 `canreturn` 检查——下一节覆盖索引就靠这个。

### 前提二(反直觉的重点):堆页必须对当前事务全可见——查 VM

前提一满足,是不是就能放心"只读索引、不回堆"了?**不行**。这里有个根本性的障碍,必须想通。

> **不这样会怎样**:假设索引里说"id=42 的行 TID=(5,7)"。但就在你查询的这一刻,另一个事务刚刚 `DELETE FROM users WHERE id = 42` 并提交了(或者 UPDATE 了它,旧版本已失效)。堆表里块 5 的第 7 行,对**你当前这个事务**已经**不可见**了。可索引呢?索引里的 `(42, (5,7))` 这条记录**还在**(删除的索引清理是 VACUUM 延后做的,见第 21 章)。如果仅索引扫描盲目相信索引、直接返回 "id=42",你就**读到了一行其实对你已经不可见的数据**——脏读,数据不一致,比慢还可怕。

问题的根源:**索引里的 tuple 是"历史快照",它不知道自己指向的堆行是否还活着、对当前事务是否可见**。可见性信息只存在堆 tuple 的 `xmin`/`xmax` 里(第 7、17 章),索引里没有。所以,要确认可见性,**原则上必须回堆**去查 `xmin`/`xmax`。

> 这下矛盾了:仅索引扫描的本意是不回堆,可确认可见性又必须回堆。如果每条都回堆确认,那和普通索引扫描有何区别?——这正是第 9 章埋下的伏笔:**VM(可见性映射)的 `ALL_VISIBLE` 位,就是用来解开这个死结的**。

VM 给每个堆页存一个 `ALL_VISIBLE` 位(第 9 章详讲)。这个位为真,含义是:**这一页里所有 tuple,对当前及之后所有事务都可见**——没有死元组、没有未提交的新版本。换句话说,`ALL_VISIBLE` 是一份"整页免检"的担保:只要这个堆页的 `ALL_VISIBLE` 亮着,那这页里任何一行对当前事务**一定可见**,不必再回堆逐行查 `xmin`/`xmax`。

于是仅索引扫描的逻辑成形了:**拿到一个 TID,先看它指向的堆页的 `ALL_VISIBLE` 位。亮着,就不回堆,直接用索引里的数据;没亮,就必须回堆确认这行到底可不可见**(因为它可能已被删改)。

源码在 `nodeIndexonlyscan.c` 的主循环里,正是这段:

> [src/backend/executor/nodeIndexonlyscan.c:120-190](../postgresql-17.0/src/backend/executor/nodeIndexonlyscan.c#L120-L190)

```c
	while ((tid = index_getnext_tid(scandesc, direction)) != NULL)
	{
		bool		tuple_from_heap = false;

		CHECK_FOR_INTERRUPTS();

		/*
		 * We can skip the heap fetch if the TID references a heap page on
		 * which all tuples are known visible to everybody.  In any case,
		 * we'll use the index tuple not the heap tuple as the data source.
		 *
		 * Note on Memory Ordering Effects: visibilitymap_get_status does not
		 * lock the visibility map buffer, and therefore the result we read
		 * here could be slightly stale.  However, it can't be stale enough to
		 * matter.
		 ...
		 * It's worth going through this complexity to avoid needing to lock
		 * the VM buffer, which could cause significant contention.
		 */
		if (!VM_ALL_VISIBLE(scandesc->heapRelation,
							ItemPointerGetBlockNumber(tid),
							&node->ioss_VMBuffer))
		{
			/*
			 * Rats, we have to visit the heap to check visibility.
			 */
			InstrCountTuples2(node, 1);
			if (!index_fetch_heap(scandesc, node->ioss_TableSlot))
				continue;		/* no visible tuple, try next index entry */
			...
			tuple_from_heap = true;
		}

		/*
		 * Fill the scan tuple slot with data from the index. ...
		 */
		if (scandesc->xs_hitup)
		{
			...
			ExecForceStoreHeapTuple(scandesc->xs_hitup, slot, false);
		}
		else if (scandesc->xs_itup)
			StoreIndexTuple(node, slot, scandesc->xs_itup, scandesc->xs_itupdesc);
		else
			elog(ERROR, "no data returned for index-only scan");
		...
		return slot;
	}
```

逐句对应:

1. `index_getnext_tid` 从索引取下一个 TID(只读索引页,不碰堆)。
2. **`VM_ALL_VISIBLE(...)` 查 VM**:这页全可见吗?
3. **全可见**:跳过回堆,直接走到下面的 `StoreIndexTuple`,**从索引 tuple 里取出列值**填进 slot 返回。这一条命中就是"真正的仅索引"——零回堆。
4. **没全可见**:`InstrCountTuples2(node, 1)` 把计数器加一(**这个计数器 ntuples2,就是 EXPLAIN 里 `Heap Fetches` 的值,见下节**),然后老老实实 `index_fetch_heap` 回堆确认可见性。回堆发现不可见就 `continue` 换下一个;可见的话,虽然回堆了,但**数据仍然优先用索引里的**(注释 `we'll use the index tuple not the heap tuple as the data source`)。

注意一个微妙点:`VM_ALL_VISIBLE` 读 VM 时**不加锁**(注释 `Note on Memory Ordering Effects` 那一大段解释了为什么不加锁也安全)。核心论据是:VM 位被清(因为 INSERT/UPDATE/DELETE)总是**先于**对应堆页的修改,而修改堆页后才会去更新索引;读写索引页时的锁构成了内存屏障。所以仅索引扫描读到的 TID,如果对应堆页刚被改过、VM 位已清,它一定能看到那个清掉的位(不会误以为还全可见)。这套论证让 VM 查询免去了锁争用——对高并发仅索引扫描的性能很关键。

### 仅索引扫描到底省了什么

把上面串起来,仅索引扫描的真实收益是:**当堆页全可见(VM `ALL_VISIBLE` 亮)时,完全跳过回表**。它省的是随机 I/O。

但它有个"不省"的地方:回堆确认可见性那一步,如果 VM 没亮,还得读堆页。所以仅索引扫描的"纯度"取决于数据的更新活跃度:

- 一张**只读/很少更新**的表,VACUUM 之后大部分页 `ALL_VISIBLE` 都亮,仅索引扫描近乎零回表,极快。
- 一张**频繁 UPDATE/DELETE** 的表,大量页的 `ALL_VISIBLE` 被清(第 9 章讲过,任何写都会清位),仅索引扫描退化成"还是得回堆",收益大打折扣。这就是为什么 VACUUM 对仅索引扫描性能至关重要——VACUUM 清理死元组后会把页重新标记 `ALL_VISIBLE`。

### EXPLAIN 里的 `Heap Fetches`:回表多少,一眼可见

仅索引扫描到底回了几次表,EXPLAIN ANALYZE 会直接告诉你。看 explain.c 里输出这一行的条件:

> [src/backend/commands/explain.c:1992-1994](../postgresql-17.0/src/backend/commands/explain.c#L1992-L1994)

```c
			if (es->analyze)
				ExplainPropertyFloat("Heap Fetches", NULL,
									 planstate->instrument->ntuples2, 0, es);
```

`ntuples2` 这个计数,正是上面源码里 `InstrCountTuples2(node, 1)` 累加的——**每次因为 VM 没亮而被迫回堆确认可见性,就 +1**。所以 `Heap Fetches` 的含义非常精确:**仅索引扫描中,有多少行不得不回堆**。它为 0,说明所有命中页都全可见,真正零回表;它很大,说明数据更新频繁、VM 位被清得多,仅索引扫描"名不副实",这时该考虑 VACUUM 了。

> 这也回答了一个常见困惑:"为什么我的 Index Only Scan 还是很慢?"——看 `Heap Fetches`,如果它和行数差不多,说明几乎每行都在回堆,仅索引扫描的收益没兑现。根子在可见性,不在索引本身。

## 四、覆盖索引:用 INCLUDE 把额外列塞进索引

仅索引扫描的前提一是"索引里有全部列"。可很多查询是这样的:`SELECT id, name FROM users WHERE id = 42`——查询条件走 `id` 索引,但还想顺带拿 `name`。`name` 不在索引里,就只能回表。

最朴素的解法是把索引建成 `idx ON (id, name)`——把 `name` 也加进索引键。但这有两个副作用:**① 索引变大了**(每个叶子节点都多存一列);**② `name` 进了排序键**,B 树要按 `(id, name)` 联合排序,影响索引结构、也影响某些查询能否用这个索引。

**覆盖索引(Covering Index)** 提供了一个更精细的工具:`INCLUDE` 子句。

```sql
CREATE INDEX idx_users_id_cover ON users (id) INCLUDE (name);
```

它的语义是:**`id` 是索引的键列(key columns),参与 B 树排序;`name` 是"被包含列"(included columns),只存在索引叶子节点里,但不参与排序**。这样:

- 索引仍按 `id` 排序,结构和一个普通 `idx ON (id)` 完全一样(B 树内部节点不含 `name`)。
- 但叶子节点里每条记录除了 `(id, TID)`,还额外存了 `name` 的值。
- 现在 `SELECT id, name FROM users WHERE id = 42` 要的列(`id`、`name`)索引里全有了,仅索引扫描的前提一成立。

源码上,PG 用 `indnkeyatts`(键列数)和 `indnatts`(总列数,键列 + INCLUDE 列)区分这两类列:

> [src/include/catalog/pg_index.h:33-36](../postgresql-17.0/src/include/catalog/pg_index.h#L33-L36)

```c
	int32		indnatts;		/* total number of columns in index */
	int16		indnkeyatts;	/* number of key columns in index */
```

`indnkeyatts` 是参与排序的键列数;`indnatts - indnkeyatts` 就是 INCLUDE 列数。B 树的代码处处用 `IndexRelationGetNumberOfKeyAttributes`(= `indnkeyatts`)来界定"键空间":

> [src/include/utils/rel.h:521-525](../postgresql-17.0/src/include/utils/rel.h#L521-L525)

```c
/*
 * IndexRelationGetNumberOfKeyAttributes
 */
#define IndexRelationGetNumberOfKeyAttributes(relation) \
		((relation)->rd_index->indnkeyatts)
```

关键在于:**INCLUDE 列不参与排序,所以 B 树的内部节点(用于导航的分隔键)里不会出现它们**。这发生在 `_bt_truncate`——B 树页面分裂时生成新的"枢轴 tuple"(pivot,即内部节点里用来分隔左右子树的键)。枢轴 tuple 只保留键列,INCLUDE 列被截断掉:

> [src/backend/access/nbtree/nbtutils.c:4617-4671](../postgresql-17.0/src/backend/access/nbtree/nbtutils.c#L4617-L4671)

```c
/*
 *	_bt_truncate() -- create tuple without unneeded suffix attributes.
 *
 * Returns truncated pivot index tuple allocated in caller's memory context,
 * with key attributes copied from caller's firstright argument.  If rel is
 * an INCLUDE index, non-key attributes will definitely be truncated away,
 * since they're not part of the key space.  More aggressive suffix
 * truncation can take place when it's clear that the returned tuple does
 * not need one or more suffix key attributes. ...
 */
IndexTuple
_bt_truncate(Relation rel, IndexTuple lastleft, IndexTuple firstright,
			 BTScanInsert itup_key)
{
	TupleDesc	itupdesc = RelationGetDescr(rel);
	int16		nkeyatts = IndexRelationGetNumberOfKeyAttributes(rel);
	int			keepnatts;
	IndexTuple	pivot;
	...
	/* Determine how many attributes must be kept in truncated tuple */
	keepnatts = _bt_keep_natts(rel, lastleft, firstright, itup_key);
	...
	pivot = index_truncate_tuple(itupdesc, firstright,
								 Min(keepnatts, nkeyatts));
	...
}
```

注释 `If rel is an INCLUDE index, non-key attributes will definitely be truncated away, since they're not part of the key space` 一语道破:**INCLUDE 列不在键空间里,内部节点的枢轴 tuple 不存它们**。`Min(keepnatts, nkeyatts)` 保证截断后最多只保留到 `nkeyatts`(键列数)——INCLUDE 列(`indnkeyatts+1` 到 `indnatts`)一律被砍掉。

> 这样设计的好处:INCLUDE 列只增加**叶子节点**的体积(每个叶子多存几列),**不增加**内部节点的体积、不改变 B 树的排序行为。索引比"全塞进键列"的方案更紧凑,而且对查询优化器友好——`WHERE id = ?` 用这个索引和用纯 `idx ON (id)` 的代价估算几乎一样(键空间没变)。

### INCLUDE 列怎么被仅索引扫描取用

回到前提一的 `check_index_only`。它构造 `index_canreturn_attrs` 时,循环到 `index->ncolumns`(= `indnatts`,包括 INCLUDE 列),并对每列查 `canreturn`。对 B 树,INCLUDE 列的 `canreturn` 是 true(B 树的 `amcanreturn` 对所有列返回真)。所以 INCLUDE 列会被加入 `index_canreturn_attrs`,`SELECT id, name` 的 `name`(假设在 INCLUDE 里)就能被认作"索引可返回",仅索引扫描可行。

这就是覆盖索引的完整价值链:**INCLUDE 让"索引里有全部列"成立 → `check_index_only` 通过 → 仅索引扫描被选中 → 配合 VM 全可见 → 真正省掉回表**。前提二(VM 全可见)依然不能少——覆盖索引解决的是前提一,可见性还得靠 VM。

### 什么时候用 INCLUDE,什么时候直接加进键

- 查询需要对额外列**做范围/排序**(如 `WHERE id = ? ORDER BY name`):把 `name` 加进键列 `(id, name)`,索引能直接支持排序。
- 查询只是**顺带取出来**(如 `SELECT id, name WHERE id = ?`,不要 `ORDER BY name`):用 `INCLUDE (name)`,不污染键空间,索引更小。
- 一个经典场景:**主键/唯一约束 + 额外取列**。`UNIQUE INDEX ON (a, b) INCLUDE (c, d)` 保证 `(a,b)` 唯一,又让 `SELECT a, b, c, d WHERE a=? AND b=?` 能仅索引扫描——`c, d` 不影响唯一性约束(约束只看键列),又能被扫描直接取出。

## 五、关键源码精读:`StoreIndexTuple` 与三种扫描的 I/O 路径对照

把本章最核心的一段——仅索引扫描"从索引 tuple 取数据填进 slot"的 `StoreIndexTuple`,逐行看清。它回答了"不回堆时,数据到底从哪来":

> [src/backend/executor/nodeIndexonlyscan.c:267-282](../postgresql-17.0/src/backend/executor/nodeIndexonlyscan.c#L267-L282)

```c
static void
StoreIndexTuple(IndexOnlyScanState *node, TupleTableSlot *slot,
				IndexTuple itup, TupleDesc itupdesc)
{
	/*
	 * Note: we must use the tupdesc supplied by the AM in index_deform_tuple,
	 * not the slot's tupdesc, in case the latter has different datatypes
	 * (this happens for btree name_ops in particular).  They'd better have
	 * the same number of columns though, as well as being datatype-compatible
	 * which is something we can't so easily check.
	 */
	Assert(slot->tts_tupleDescriptor->natts == itupdesc->natts);

	ExecClearTuple(slot);
	index_deform_tuple(itup, itupdesc, slot->tts_values, slot->tts_isnull);
	...
}
```

`itup` 是索引 AM 返回的索引 tuple(`scandesc->xs_itup`),`itupdesc` 是索引的 tuple 描述(列定义)。`index_deform_tuple` 把这个紧凑的索引 tuple **解包**成一组 Datum,填进 `slot->tts_values`(列值数组)和 `tts_isnull`(是否为空数组)。**注意:这里的数据来源是索引 tuple,不是堆 tuple**——这就是"仅索引"的体现。

关键细节在注释:`itupdesc->natts` 必须等于 slot 的列数。对一个覆盖索引,`itupdesc` 包含键列 + INCLUDE 列,所以 `StoreIndexTuple` 能把 INCLUDE 列也解包出来。这印证了上节"INCLUDE 列能被仅索引扫描直接取用"——数据真真切切地从索引 tuple 里 deform 出来,没碰堆。

### 三种扫描的 I/O 路径对照(本章核心总结)

把三种走索引的扫描放在一起对照(这是本章的"全景图"):

```text
┌─────────────────────────────────────────────────────────────────────┐
│ 普通 Index Scan                                                      │
│   ① 读索引页 → 顺次拿到 TID₁, TID₂, TID₃ ...                       │
│   ② 对每个 TIDᵢ → 回堆表读对应页(随机!)→ 取整行 → 返回           │
│   I/O 模式:索引顺序读 + 每行一次随机堆读                            │
│   适用:命中行数少(选择性高);需要索引里没有的列                    │
├─────────────────────────────────────────────────────────────────────┤
│ Bitmap Scan                                                          │
│   ① 读索引页 → 把所有命中 TID 攒进位图(按堆页组织)                │
│   ② 按块号升序,逐页批量回堆 → 每页取出本页所有命中行               │
│   I/O 模式:索引一次扫完 + 堆页近似顺序读(随机→顺序改造)          │
│   适用:命中行数多、散布广;多个索引 AND/OR 组合                     │
│   注意:仍要回表(除非退化);lossy 时还得逐行 recheck               │
├─────────────────────────────────────────────────────────────────────┤
│ Index-Only Scan                                                      │
│   ① 读索引页 → 拿到 TIDᵢ 和该行的全部列(含 INCLUDE 列)            │
│   ② 查 VM:该堆页 ALL_VISIBLE?                                       │
│        是 → 直接用索引里的列值返回(零回堆!)                       │
│        否 → 回堆确认可见性,但数据仍用索引里的                       │
│   I/O 模式:理想情况下只读索引页(前提:VM 全可见 + 索引含全部列)   │
│   适用:索引覆盖了查询全部列 + 数据少更新(VM 位常亮)               │
└─────────────────────────────────────────────────────────────────────┘
```

记住一条判别:**回表是随机 I/O 的代价大头,三种扫描分别是"老老实实回表"(Index Scan)、"把回表顺序化"(Bitmap)、"干脆不回表"(Index-Only)**。仅索引扫描最理想,但门槛最高——既要索引里有全部列(靠覆盖索引达成),又要堆页全可见(靠 VM 担保、靠 VACUUM 维护)。

## 章末小结

- **索引叶子只存 TID**:PG 的索引不存完整行,只存"路标"。用户要整行,就得拿 TID 回堆表取——这一步是**回表(heap fetch)**,且因为命中行物理上散乱,回表是**随机 I/O** 的代价大头。
- **普通 Index Scan**:`index_getnext_slot` 交替"取 TID + 回堆取整行",每行一次随机堆读。命中行多时代价爆炸。
- **Bitmap Scan**:先把所有 TID 攒进位图(按堆页组织),再按块号顺序批量回堆,把随机 I/O 改造为近似顺序。仍要回表,只是顺序化了;命中太多放不进 `work_mem` 会退化成 lossy(只记页、逐行 recheck)。
- **Index-Only Scan**:两个前提——① 索引含查询全部列(`check_index_only` 判定);② 堆页对当前事务全可见(VM 的 `ALL_VISIBLE` 位)。前提二最反直觉:索引 tuple 是历史快照,不知堆里对应行是否被删改,只有 VM 担保"整页全可见"才能免回堆。VM 没亮时仍回堆确认可见性,计数器 `ntuples2` 就是 EXPLAIN 的 `Heap Fetches`。
- **覆盖索引**:`INCLUDE` 把非键列塞进索引叶子,不参与排序(键空间不变,内部节点枢轴 tuple 截断掉它们,见 `_bt_truncate`),却能让仅索引扫描的前提一成立。
- **SELECT 少选几列**:`check_index_only` 比较"用到的列 ⊆ 索引能返回的列",少选一列就可能让这个包含关系成立,计划从 Index Scan 变 Index-Only Scan。

### 回扣主线

本章是 P3 索引篇的收尾,它把索引和堆表真正"接"了起来。回到全书的二分法:

- 这一章整体服务"**快**"。回表是随机 I/O 的大头,仅索引扫描和覆盖索引存在的全部意义,就是**消除回表**;Bitmap 扫描则是把回表的随机 I/O 改造成顺序——三种扫描都在和"磁盘慢、随机 I/O 贵"这个本性较劲。
- 但本章也深深牵动"**不丢不乱**"那一侧。仅索引扫描最反直觉的约束——"数据在索引里,却还要查堆页的可见性"——根子就在 MVCC:**PG 的 UPDATE/DELETE 不原地改,而是造新版本、留旧版本,所以一行到底对当前事务可不可见,是个运行时才能回答的问题**。VM 的 `ALL_VISIBLE` 位,正是第 9 章讲的、为高效回答这个问题而设的"整页免检"担保。一次 UPDATE 清掉 VM 位(第 9 章讲过),直接导致仅索引扫描被迫回表——存储层(P2)和事务层(P4)在 VM 这个 bit 上紧紧咬合。可以说,没有 MVCC,就不需要查可见性,仅索引扫描会简单得多;但也正因为 PG 坚持 MVCC(读不阻塞写),才有了 VM 这层"用空间换免检"的精巧设计。

至于数据的三个本性:本章主要对付"**用户说要什么,机器只会按位置读**"——索引弥合"按值找"和"按位置存"的错位,而仅索引扫描/覆盖索引进一步抹平了"索引按值、数据按位置"之间那道回表的沟。同时它又承接"**共享会冲突**":可见性问题的全部来源,就是多人并发改数据。
