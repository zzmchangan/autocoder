# 第 5 章 · 执行器:把计划跑出来

> **前置**:上一章《查询优化器》给出了一棵**执行计划(Plan 树)**——优化器算出了"先扫哪张表、用不用索引、用哪种连接、按什么顺序最省"。但那只是"决策",一行数据都还没动。本章就负责把这棵 Plan 树**真正跑起来**,一行一行地把结果吐出来。

> **核心问题**:执行计划(Plan 树)怎么被一步步跑出结果?为什么几乎所有关系数据库(MySQL、Oracle、PostgreSQL)的执行器都用同一套**「火山模型/Volcano 迭代器」**——每个算子暴露一个 `next()` 接口,上层来拉、下层被动吐?顺序扫描、索引扫描、三种连接(嵌套循环 / Hash / 归并)分别在哪类场景最优、各自的代价来自哪里?

> **读完本章你会明白**:火山模型为什么是"拉(pull)"而不是"推(push)";为什么这个选择让接口统一、能流式产出、能短路 `LIMIT`、内存可控;它的代价是什么、PG 为此做了哪些批量化改进;三种连接算法各自的"适用场景 + 代价来源 + 不这样会怎样";以及 `ExecutorStart → ExecutorRun → ExecutorFinish` 这三段顶层接口是怎么把一棵计划树驱动起来的。

> **如果一读觉得太难**:先记三件事——① 火山模型 = 每个算子一个 `next()`,上层拉、下层吐,像迭代器;② 三种连接:小表全装内存用 Hash、有索引或内表可复用用 NestedLoop、两边已有序用 MergeJoin;③ 代价都是"逐行函数调用 + 流式一行行传"——这也是后续批量化改进要治的痛。其余细节配合源码第二遍再抠。

## 一、执行器要解决的根本问题:怎么"运行"一棵树

先回到上一章的结尾:优化器吐出的不是一个"动作",而是一棵 **Plan 树**。比如下面这个查询:

```sql
SELECT u.name, o.amount
FROM users u JOIN orders o ON u.id = o.user_id
WHERE u.city = '北京';
```

它对应的 Plan 树大概长这样(简化示意,非 EXPLAIN 真实输出):

```text
                    NestLoop          (顶层算子:连接)
                   /        \
            SeqScan(users)  IndexScan(orders, user_id)   (两个子算子)
            Filter: city='北京'
```

树的**叶子**是扫描算子(从表/索引取数据),**中间**是连接、过滤、聚合、排序等算子,**根**是最终结果。优化器已经决定了"每个节点用什么算子、子节点是谁",但**怎么把一棵静态的树变成一堆流动的结果行**——这就是执行器的事。

> **不这样会怎样**:如果执行器没有一套统一的"运行方式",那每加一种新算子(比如新增一种连接),就得为它单独写一套"怎么从子节点拿数据、怎么把数据交给父节点"的逻辑。算子一多,组合爆炸,代码完全没法维护。更糟的是:整棵树的数据什么时候物化(materialize,即把中间结果整个存下来)、什么时候流式处理,会变成一笔糊涂账——要么全物化(内存爆炸),要么写死(没法短路 `LIMIT`)。

所以执行器需要一个**统一的数据驱动模型**。这个模型,就是数据库界近四十年几乎没变过的标准答案:**火山模型(Volcano / Iterator Model)**。

## 二、火山模型:为什么是"拉(pull)"而不是"推(push)"

### 火山模型的核心:每个算子一个 `next()`

火山模型一句话:**每个算子都暴露一个"给我下一行"的接口,父算子主动调用子算子的这个接口去拉数据。** 整棵树的数据流向,是自上而下"拉"出来的。

抽象成伪代码,每个算子都有这么一个函数:

```text
Tuple next() {
    // 返回这个算子产出的下一行;
    // 没有了就返回 NULL(表示结束)。
}
```

数据**不会**自己从叶子往上"涌"。而是:最顶层(根算子)被外部调用 `next()`,它为了产出一行,就去调用它子节点的 `next()`;子节点为了产出,又去调用**它**的子节点……一直递归到叶子算子,叶子从磁盘/缓冲池读一行返回。这一行沿着调用链一层层返回上去,最终被根算子交给客户端。

> **本质**:火山模型就是**迭代器模式(iterator pattern)** 应用到查询执行上。会写 `for (x in list)` 的程序员,已经懂迭代器了——`list` 给你一个 `hasNext()/next()` 的接口,你只管拉,不用管它内部怎么实现。执行器里的算子,就是这样一个个嵌套的迭代器。

### 为什么几乎所有数据库都用"拉"模型

这是本章最关键的一个"为什么"。我们逐条看"拉"模型换来了什么。

**1. 接口统一,组合自由。** 每个算子只认一个协议:子节点能 `next()`。所以一个 `NestLoop` 的左边可以是 `SeqScan`,也可以是另一个 `HashJoin`,还可以是带 `LIMIT` 的子树——上层完全不用改。优化器可以自由地把算子像积木一样拼成任意形状的树,执行器照单全收。这就是为什么数据库能支持任意复杂的 SQL:算子之间的接口是统一的,组合性天然成立。

**2. 流式产出,不必物化中间结果。** "拉"模型意味着**只有当父节点需要一行时,子节点才去算一行**。中间结果不需要整批存下来,而是一行产出、一行消费、一行丢弃。对一个返回百万行的连接,执行器从头到尾可能同时在内存里只持有**几行**(每个算子各一行),而不是一百万行。这对内存是救命稻草。

> **不这样会怎样(如果用"推"或"全物化")**:设想一个"推"模型——叶子算子扫到一行就主动往上送。问题来了:顶层是 `LIMIT 10`,但叶子不知道上面只要 10 行,它会把整张表扫完往上推,白白做了海量无用功。而"拉"模型下,根 `Limit` 算子拉满 10 行就停止再调用子节点的 `next()`,子节点的扫描**自然就停了**——这叫**短路(short-circuit)**,免费获得。

**3. 内存可控。** 流式 + 短路,直接带来内存可控:执行器不会因为结果集大而把内存撑爆。需要"记住全部数据"的算子(比如 Hash 连接要先建哈希表、Sort 要先收齐),是**特例**,由该算子自己负责管理那块内存;绝大多数算子(扫描、过滤、投影、嵌套循环连接)都是无状态的逐行处理,内存占用恒定。这种"默认流式、特例物化"的设计,让执行器的内存行为可预测。

**4. 适合流水线(pipelining)。** 一行从叶子到根的旅程是一气呵成的,中间不落地。这意味着结果可以**边算边返回给客户端**——你执行一个 `SELECT`,第一行几乎立刻就能出现在屏幕上,不用等全部算完。这是"快"那一侧的一个重要体验。

### 拉模型的代价:逐行函数调用,以及 PG 的批量化改进

拉模型不是没代价。它最大的痛点是:**每产出一行,就要跨好几个算子做一次函数调用**。一棵 5 层深的树,每行要 5 次虚函数分发(virtual call)。对扫描一亿行的查询,那就是五亿次函数调用——CPU 分支预测被打乱、指令缓存频繁失效、虚函数表查找的开销累积起来相当可观。

> **这个代价在 OLAP(分析型)场景尤其刺眼**。OLTP 一条 SQL 通常只碰几十几百行,逐行调用的开销可忽略;但 OLAP 动辄扫几亿行做聚合,逐行就成了瓶颈。这就是为什么近十几年数据库界一直在讨论**向量化执行(vectorized execution)**:把"每次 next 返回一行"改成"每次 next 返回一批(比如 1024 行)",让每个算子对一批数据连续处理,摊薄函数调用开销、还能让 CPU SIMD 指令派上用场。DuckDB、ClickHouse 这些分析型引擎走的就是这条路。

PostgreSQL 主线目前仍是经典火山模型(逐行),但并非没动过批量化心思。源码里就能看到几处为减少逐行开销做的设计:

- **Hash 节点不走标准 `ExecProcNode` 协议**,而是用一个 `MultiExecProcNode` 一次性把内表全部读进哈希表(后面精读会看到)。这是火山模型的一个"逃生舱":当某个算子天然需要一次性消费整个输入时,允许它绕开逐行协议。
- `ExecProcNodeFirst` 在节点**第一次**执行时做一次栈深度检查,之后就把它替换成无包装的直接调用(`node->ExecProcNodeReal`),省掉每行一次的检查开销(见下文源码精读)。
- `ExecHashJoinImpl` 用了 `pg_attribute_always_inline` 强制内联,省去一次函数调用开销——在热路径上,这种细节是值得抠的。

> **一句话**:火山模型用"逐行函数调用"换来了"接口统一、流式、短路、内存可控"。对 OLTP 这笔交易非常划算;对重 OLAP 才显现出代价,于是有了向量化。理解这个权衡,你就理解了执行器四十年设计的主轴。

## 关键源码精读(一):火山模型的统一入口 `ExecProcNode`

我们顺着源码,把"上层拉、下层吐"这条机制钉死。这是本章最核心的一段。

### 1. 拉取的统一入口:`ExecProcNode`

PG 里,父算子拉取子算子的下一行,统一调用 `ExecProcNode`。它其实是个**内联包装函数**,定义在头文件里:

> [src/include/executor/executor.h:267-L276](../postgresql-17.0/src/include/executor/executor.h#L267-L276)

```c
#ifndef FRONTEND
static inline TupleTableSlot *
ExecProcNode(PlanState *node)
{
	if (node->chgParam != NULL) /* something changed? */
		ExecReScan(node);		/* let ReScan handle this */

	return node->ExecProcNode(node);
}
#endif
```

这就是火山模型的全部"协议"。逐行看:

- 入参 `PlanState *node` —— 要拉取的那个算子的**运行状态**。注意是 `PlanState`(运行态)不是 `Plan`(静态计划):`Plan` 是优化器给的"图纸",`PlanState` 是执行器按图纸建出来的"实例",里面带着扫描游标、哈希表指针等所有运行时状态。
- `chgParam` —— 如果上层传下来的参数变了(典型场景:相关子查询、`NestLoop` 把外层当前行的值传给内层扫描),就触发一次 `ExecReScan`,让这个算子用新参数重新扫描。
- `return node->ExecProcNode(node);` —— **关键一行**:每个 `PlanState` 里有一个函数指针 `ExecProcNode`,指向"这个算子真正的 next 实现"。这里就是一次**虚函数调用**:根据算子类型,分派到 `ExecSeqScan`、`ExecNestLoop`、`ExecHashJoin`……各自的实现。

函数指针类型叫 `ExecProcNodeMtd`,定义在 `execnodes.h`:

> [src/include/nodes/execnodes.h:L1103](../postgresql-17.0/src/include/nodes/execnodes.h#L1103)

```c
typedef TupleTableSlot *(*ExecProcNodeMtd) (struct PlanState *pstate);
```

签名一目了然:吃一个 `PlanState`,吐一个 `TupleTableSlot`(就是"一行"的载体,可能为空表示结束)。**全数据库所有算子的 next 函数,签名都是这一个**——这就是"统一接口"在 C 层面的落点。

### 2. `PlanState`:每个算子运行态的公共骨架

`PlanState` 是所有算子运行状态的"抽象基类",每个具体算子(SeqScanState、NestLoopState……)都以它打头:

> [src/include/nodes/execnodes.h:L1112-L1126](../postgresql-17.0/src/include/nodes/execnodes.h#L1112-L1126)

```c
typedef struct PlanState
{
	pg_node_attr(abstract)

	NodeTag		type;

	Plan	   *plan;			/* associated Plan node */

	EState	   *state;			/* at execution time, states of individual
								 * nodes point to one EState for the whole
								 * top-level plan */

	ExecProcNodeMtd ExecProcNode;	/* function to return next tuple */
	ExecProcNodeMtd ExecProcNodeReal;	/* actual function, if above is a
										 * wrapper */

	Instrumentation *instrument;	/* Optional runtime stats for this node */
	...
```

两个函数指针需要解释清楚,这是 PG 为减少逐行开销做的小机关:

- `ExecProcNodeReal` —— 这个算子**真正**的 next 函数(比如 `ExecSeqScan`)。
- `ExecProcNode` —— 当前实际被调用的那个。它可能等于 `ExecProcNodeReal`(直接调用,最快),也可能是个**包装函数**。

为什么要包装?因为节点**第一次**执行时要做一些一次性的事:检查栈深度(防止递归过深的 Plan 树把栈撑爆)、挂上性能统计(instrumentation)。但这些事只需要做一次,之后每行都做就太浪费。PG 的做法是:初始化时把 `ExecProcNode` 指向 `ExecProcNodeFirst`(带检查的包装),第一次调用时它做完一次性检查,然后**把自己替换掉**。

### 3. 一次性检查 + 自我替换:`ExecProcNodeFirst`

> [src/backend/executor/execProcnode.c:L442-L465](../postgresql-17.0/src/backend/executor/execProcnode.c#L442-L465)

```c
static TupleTableSlot *
ExecProcNodeFirst(PlanState *node)
{
	/*
	 * Perform stack depth check during the first execution of the node.  We
	 * only do so the first time round because it turns out to not be cheap on
	 * some common architectures (eg. x86).  This relies on the assumption
	 * that ExecProcNode calls for a given plan node will always be made at
	 * roughly the same stack depth.
	 */
	check_stack_depth();

	/*
	 * If instrumentation is required, change the wrapper to one that just
	 * does instrumentation.  Otherwise we can dispense with all wrappers and
	 * have ExecProcNode() directly call the relevant function from now on.
	 */
	if (node->instrument)
		node->ExecProcNode = ExecProcNodeInstr;
	else
		node->ExecProcNode = node->ExecProcNodeReal;

	return node->ExecProcNode(node);
}
```

读这段注释能品出工程味儿:栈深度检查在 x86 上"turns out to not be cheap",所以只在第一次做,之后默认假设同一个节点的后续调用栈深度不变(因为递归结构固定)。第一次做完检查,就直接把 `node->ExecProcNode` 改写:

- 如果开启了 `EXPLAIN ANALYZE` 之类的性能统计(`node->instrument` 非空),换成 `ExecProcNodeInstr`——这个包装在每次调用前后记账(多少行、耗时多少);
- 否则,**直接指向 `ExecProcNodeReal`**,从此每行都是一次干净利落的直接函数调用,没有任何额外开销。

> 这就是"火山模型逐行开销大,PG 尽力抠细节"的一个活样本:它没有改成向量化,但把一切非必要的、只该做一次的工作从热路径上挪走了。

## 三、叶子算子:从表和索引取数据

叶子算子是数据的源头——它们不调用任何子算子的 `next()`,而是直接去存储层读数据。最基本的两类:**顺序扫描(SeqScan)**和**索引扫描(IndexScan)**。

### 顺序扫描 SeqScan:老老实实从头扫到尾

顺序扫描就是最朴素的取法:从表的第一页第一行开始,一行一行往后读,直到表尾。

PG 里 `ExecSeqScan` 的实现极其简短:

> [src/backend/executor/nodeSeqscan.c:L107-L115](../postgresql-17.0/src/backend/executor/nodeSeqscan.c#L107-L115)

```c
static TupleTableSlot *
ExecSeqScan(PlanState *pstate)
{
	SeqScanState *node = castNode(SeqScanState, pstate);

	return ExecScan(&node->ss,
					(ExecScanAccessMtd) SeqNext,
					(ExecScanRecheckMtd) SeqRecheck);
}
```

它把脏活全交给了通用的 `ExecScan` 框架(所有扫描类算子共享),自己只提供两个回调:`SeqNext`(怎么取下一行)和 `SeqRecheck`(怎么重新校验一行,SeqScan 用不上,直接返回 true)。

真正的取行逻辑在 `SeqNext`:

> [src/backend/executor/nodeSeqscan.c:L49-L83](../postgresql-17.0/src/backend/executor/nodeSeqscan.c#L49-L83)

```c
static TupleTableSlot *
SeqNext(SeqScanState *node)
{
	...
	scandesc = node->ss.ss_currentScanDesc;
	...

	{
		/*
		 * We reach here if the scan is not parallel, or if we're serially
		 * executing a scan that was planned to be parallel.
		 */
		scandesc = table_beginscan(node->ss.ss_currentRelation,
								   estate->es_snapshot,
								   0, NULL);
		node->ss.ss_currentScanDesc = scandesc;
	}

	 * get the next tuple from the table
	 */

```

`table_beginscan` / `table_scan_getnextslot` 是表访问层(table AM)的接口——这是把"取下一行"的活儿进一步下放给存储引擎(P2 会讲)。对执行器而言,SeqScan 就是"给我下一行,直到没有为止"。

> **SeqScan 适合什么场景、代价来自哪里**:
> - **适合**:表很小(整个表本来就没几页,扫一遍比走索引还快);或者查询要返回**大部分行**(`SELECT * FROM big WHERE status = 'active'`,而 90% 的行都符合——这时候走索引反而要一次次回表,比直接扫全表慢)。
> - **代价**:O(n),n 是表的总行数。无论你要的数据是 1 行还是 100 万行,SeqScan 都得把整张表过一遍。所以当结果集很小(比如 `WHERE id = 42` 只要 1 行)而表很大时,SeqScan 是灾难——这时候该用索引。

这就是"扫描算子内部的小循环":拿到一行 → 检查过滤条件(`ExecQual`) → 不满足就丢掉继续拿下一行 → 满足就往上返回。注意它依然是火山模型的一部分:外层每调用一次 `ExecSeqScan`,这个循环可能跑很多次内部 fetch 才返回一行满足条件的——但从外层看,就是"返回了下一行"。

### 索引扫描 IndexScan:按索引直接定位

索引扫描走的是另一条路:先在索引里找到满足条件的条目(通常是 B 树,详见 P3),拿到对应的表行位置(TID),再去堆表里取那一行。

入口 `ExecIndexScan`:

> [src/backend/executor/nodeIndexscan.c:L519-L537](../postgresql-17.0/src/backend/executor/nodeIndexscan.c#L519-L537)

```c
ExecIndexScan(PlanState *pstate)
{
	IndexScanState *node = castNode(IndexScanState, pstate);

	/*
	 * If we have runtime keys and they've not already been set up, do it now.
	 */
	if (node->iss_NumRuntimeKeys != 0 && !node->iss_RuntimeKeysReady)
		ExecReScan((PlanState *) node);

	if (node->iss_NumOrderByKeys > 0)
		return ExecScan(&node->ss,
						(ExecScanAccessMtd) IndexNextWithReorder,
						(ExecScanRecheckMtd) IndexRecheck);
	else
		return ExecScan(&node->ss,
						(ExecScanAccessMtd) IndexNext,
						(ExecScanRecheckMtd) IndexRecheck);
}
```

和 SeqScan 一样套用 `ExecScan`,但回调换成 `IndexNext`。注意开头那段 **runtime keys** 处理:索引扫描的"搜索键"可能要到运行时才能确定(比如 `WHERE user_id = $1` 的 `$1`,或 NestLoop 传下来的外层当前行值),这种键要在每次扫描前重新计算并 `ExecReScan`。这正是 NestedLoop 连接能"对每一个外层行,用新参数重扫一次内层索引"的底层支撑(后面讲 NestLoop 会回扣这里)。

取行核心 `IndexNext`:

> **IndexScan 适合什么场景、代价来自哪里**:
> - **适合**:结果集小而表大(典型 `WHERE id = 42` 只取 1 行);或者需要**按索引顺序**输出(`ORDER BY indexed_column`,直接省掉一个 Sort 算子)。
> - **代价**:每次取一行要两步——查索引 + 回表取行(随机 I/O)。如果结果集占了表的很大比例(比如 30%),这两步的随机 I/O 累积起来,反而比 SeqScan 的顺序扫描慢。这就是为什么 P3 会专门讲"什么时候索引反而更慢":**索引不是万能的**,优化器要靠统计信息估算"选索引还是选全表扫"更省。

## 四、连接算子:两张表怎么拼起来

连接(Join)是执行器里最有看头的部分。同一句 `JOIN`,优化器有三种完全不同的物理实现可选,各自的适用场景和代价天差地别。这一节我们把三种连接挨个拆透。

先给一个算子树示意(简化,非真实 EXPLAIN):

```text
查询: SELECT u.name, o.amount
      FROM users u JOIN orders o ON u.id = o.user_id
      WHERE u.city = '北京' AND o.amount > 100;

可能的一棵计划树(用 HashJoin):
                           HashJoin              (用 user_id 做 hash key)
                          /          \
                   SeqScan(users)    Hash         (把 orders 全部灌进哈希表)
                   Filter: city='北京'  |
                                      IndexScan(orders)   (或 SeqScan)
                                      Filter: amount>100

数据流(拉模型):
  HashJoin.next() ──拉──▶ 左子 SeqScan 拿一个 user(北京)
                  ──用 user.id 在哈希表里查匹配的 orders ──▶ 逐对产出
  哈希表在计划启动时由 Hash 节点把 orders 全量读入建成(一次性)
```

### 1. 嵌套循环连接 NestedLoop:最朴素的双层 for

NestedLoop 是最直白的连接算法,直接对应两层嵌套循环的直觉:对外表(outer)的每一行,把内表(inner)从头扫一遍,找到所有匹配的。

PG 的实现 `ExecNestLoop` 主体就是一个 `for(;;)` 大循环:

> [src/backend/executor/nodeNestloop.c:L99-L160](../postgresql-17.0/src/backend/executor/nodeNestloop.c#L99-L160)

```c

	{
		/*
		 * If we don't have an outer tuple, get the next one and reset the
		 * inner scan.
		 */

		{
			ENL1_printf("getting new outer tuple");
			outerTupleSlot = ExecProcNode(outerPlan);

			/*
			 * if there are no more outer tuples, then the join is complete..
			 */

			{

				return NULL;
			}
			...
			/*
			 * now rescan the inner plan
			 */
			ENL1_printf("rescanning inner plan");
			ExecReScan(innerPlan);
		}

		/*
		 * we have an outerTuple, try to get the next inner tuple.
		 */
		ENL1_printf("getting new inner tuple");

		econtext->ecxt_innertuple = innerTupleSlot;

			...

		}
```

注意它怎么用火山模型的接口:`ExecProcNode(outerPlan)` 拿外表下一行;`ExecReScan(innerPlan)` 重置内表游标;然后 `ExecProcNode(innerPlan)` 反复拿内表行。内表扫完(`TupIsNull`)就置 `nl_NeedNewOuter=true`,去拿下一个外表行,再重扫内表。这正是两层 `for` 嵌套,只不过用的是迭代器协议。

> **NestedLoop 适合什么场景、代价来自哪里**:
> - **适合**:① 内表上有索引,且外表较小。注意看上面 `IndexScan` 那段——`nestParams` 把外表当前行的值传给内层 IndexScan 当 runtime key,**内表不用从头扫,而是每次按索引直接定位**。这种"`NestLoop` + 内层 `IndexScan`"的组合,对"外表几行、内表巨大"的点查询连接,效率极高。② 没有等值条件只有不等值/范围条件的连接(Hash 和 MergeJoin 都要求等值可哈希/可排序)。
> - **代价来源**:朴素 NestedLoop 是 O(外 × 内)。如果内表没索引,每来一个外表行都要把内表全扫一遍——外表 1 万行、内表 10 万行,就是 10 亿次比较,灾难。所以**没有索引支撑的 NestedLoop 只适合至少一边很小**的情况。

> **不这样会怎样**:如果对一个"外表大、内表大、又没有等值条件"的场景硬上 NestedLoop,就是 O(n×m) 的全量笛卡尔积扫描。这就是为什么有等值条件时优化器更偏爱 Hash 或 MergeJoin。

### 2. Hash 连接:把内表装进哈希表,O(1) 查匹配

Hash 连接专门对付**等值连接**(`ON a.x = b.y`),思路分两阶段:

1. **Build 阶段**:把较小的那张表(内表)整个读进内存,按连接键建一张哈希表。
2. **Probe 阶段**:逐行扫描外表,对每行用连接键算哈希,去哈希表里 O(1) 找匹配。

总代价 O(内 + 外),远胜 NestedLoop 的 O(内 × 外)。

PG 的实现是 `ExecHashJoin`,用一个**状态机**驱动(注意 Hash 连接不是简单的两层循环,因为它要在建表和探测之间切换,还要处理内存装不下的分批):

### 3. 归并连接 MergeJoin:两边都排好序,顺水推舟

MergeJoin 也要求**等值连接**,但它不建哈希表,而是要求**两边输入都按连接键有序**。然后像合并两个有序链表一样,两个游标齐头并进:外键 = 内键就配对产出,谁小谁前进。

这个思路的前提是"有序"。有序从哪来?要么是索引天然有序(B 树索引按键排序),要么是下面挂一个 Sort 算子现排。源码文件开头的注释把机制讲得很清楚:

> [src/backend/executor/nodeMergejoin.c:L21-L48](../postgresql-17.0/src/backend/executor/nodeMergejoin.c#L21-L48)

```c
 *		Merge-join is done by joining the inner and outer tuples satisfying
 *		join clauses of the form ((= outerKey innerKey) ...).
 ...
 *		However, the query executor needs to know whether an outer
 *		tuple is "greater/smaller" than an inner tuple so that it can
 *		"synchronize" the two relations.  For example, consider the following
 *		relations:
 *
 *			outer: (0 ^1 1 2 5 5 5 6 6 7)	current tuple: 1
 *			inner: (1 ^3 5 5 5 5 6)			current tuple: 3
 *
 *		To continue the merge-join, the executor needs to scan both inner
 *		and outer relations till the matching tuples 5.  It needs to know
 *		that currently inner tuple 3 is "greater" than outer tuple 1 and
 *		therefore it should scan the outer relation first to find a
 *		matching tuple and so on.
```

> **MergeJoin 适合什么场景、代价来自哪里**:
> - **适合**:① 两边输入**已经有序**(比如两边都走按同一键排序的索引扫描,省掉额外排序);② 查询本身需要 `ORDER BY` 连接键,这时 MergeJoin 顺带把排序做了,一石二鸟;③ 数据特别大、Hash 连接需要分太多 batch 时,MergeJoin + 外排可能更稳。
> - **代价来源**:如果输入**不**有序,要在下面挂 Sort 算子现排,排序代价 O(n log n)。所以"两边都需现排"时,MergeJoin 往往不如 Hash;但"两边天然有序"时,MergeJoin 只需各扫一遍 O(内 + 外),且无需建哈希表的内存,非常省。
> - **不这样会怎样**:对已有序输入硬上 Hash,白白浪费了"有序"这个免费优势,还要额外建哈希表占内存;反过来对无序输入硬上 MergeJoin 而不排,结果是错的(MergeJoin 依赖有序性来保证不漏配对)。

### 三种连接速查表

| 连接 | 前提 | 适合 | 代价 | 不该用时 |
|---|---|---|---|---|
| NestedLoop | 无 | 外表小 + 内表有索引;非等值连接 | O(外×内),无索引时灾难 | 两边都大且无索引 |
| Hash | 等值、可哈希 | 大表等值连接(默认首选) | O(内+外) + 建表内存,超内存要分批 | 非等值;内表极大且无法分批 |
| MergeJoin | 等值、两边有序 | 两边已有序(索引或外排需求) | O(内+外),需排序时额外 O(n log n) | 无序且不值得排序 |

```c
					queryDesc->plannedstmt->parallelModeNeeded,
					operation,
					sendTuples,
					count,
					direction,
					dest,
					execute_once);
	}
	...
```

核心是调用 `ExecutePlan`,而 `ExecutePlan` 就是那个不断拉根节点的循环:

> [src/backend/executor/execMain.c:L1640-L1655](../postgresql-17.0/src/backend/executor/execMain.c#L1640-L1655)

```c
	for (;;)
	{
		/* Reset the per-output-tuple exprcontext */
		ResetPerTupleExprContext(estate);

		/*
		 * Execute the plan and obtain a tuple
		 */
		slot = ExecProcNode(planstate);

		/*
		 * if the tuple is null, then we assume there is nothing more to
		 * process so we just end the loop...
		 */
		if (TupIsNull(slot))
			break;
```

**这就是火山模型的发动机**:一个 `for(;;)`,每次调用 `ExecProcNode(planstate)`——`planstate` 是根算子——拉一行出来,交给目标接收器(`DestReceiver`,可能是发回客户端、也可能是写进临时表)。拉到 NULL 就结束。

注意 `count` 参数:它允许调用方说"我只要 N 行就停"。这就是前面讲的"短路"在顶层的体现——`ExecutePlan` 拉够 `count` 行就 break,根节点的子树自然停止被调用。流式 + 短路,在这里合流。

### 3. ExecutorFinish 与 ExecutorEnd:收尾

> [src/backend/executor/execMain.c:L409](../postgresql-17.0/src/backend/executor/execMain.c#L409) 之后是 `ExecutorFinish`,负责触发 AFTER 触发器等收尾动作(对 `INSERT/UPDATE/DELETE`)。再之后 `ExecutorEnd` 释放 EState、关闭扫描、释放内存。

为什么 Finish 和 End 要分开?注释说得很明白:**`EXPLAIN ANALYZE` 要把 AFTER 触发器的耗时也算进总运行时间**,所以触发器执行(Finish)必须独立于资源释放(End),这样统计窗口才完整。这是一个为可观测性服务的细节。

## 五、小结与回扣

### 本章讲了什么

- **执行器的根本任务**:把优化器给的 Plan 树"运行"出结果行。
- **火山模型**:每个算子一个 `next()`(PG 里是 `ExecProcNode`),上层拉、下层吐,本质是迭代器模式。好处是接口统一、流式产出、内存可控、能短路 `LIMIT`;代价是逐行函数调用开销,在 OLAP 重扫场景显现,于是有向量化改进。
- **叶子算子**:SeqScan(全表顺序扫,O(n),适合表小或返回大部分行);IndexScan(走索引定位,适合返回少量行或需有序输出,代价是回表的随机 I/O)。
- **三种连接**:NestedLoop(双层循环,适合外表小+内表有索引);Hash(等值连接首选,建哈希表 O(1) 查,内表超内存用分批);MergeJoin(两边有序时高效,无序则要先排)。
- **顶层三段**:`ExecutorStart`(建 EState)→ `ExecutorRun`(驱动 `ExecutePlan` 循环拉根节点)→ `ExecutorFinish`/`End`(触发器 + 释放)。

### 回扣主线:快 vs 不丢不乱

执行器主要服务"**快**"那一侧:它要把优化器选出的最优计划,用尽量少的 CPU 和内存跑出来。火山模型的流式产出和短路(`LIMIT` 提前停),都是在"别做无用功";Hash 连接的分批机制,则是在"结果集再大也别把内存撑爆"——这是流式产出的**内存控制**,本质也是服务"快而稳"。

> 注意:执行器只负责"算",不负责"存"和"不丢"。它取数据时调用的 `table_scan_getnextslot`、`index_getnext_slot`,都是把活儿下放给存储引擎(P2)。执行器本身不直接碰磁盘页、不写 WAL、不管事务可见性(它通过 EState 里的快照 `es_snapshot` 间接享受 MVCC 的成果,P4 讲)。所以执行器是"快"的执行者,但"快"的根基(数据怎么高效存在内存/磁盘)在后面几篇。
