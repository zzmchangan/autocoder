# 第 4 章 · 查询优化器:怎么决定走哪条路最省

> **前置**:你需要先读过[第 3 章《SQL 解析器》](P1-03-SQL解析器-从字符串到解析树.md)——它把一串字符串 SQL 拆词、按语法拼成了一棵 **Query 树(解析树)**。但 Query 树只表达了"**要什么**"(要 `id=42` 的用户、要把两张表连起来),并没说"**怎么去取**"。本章要回答的,就是从这棵"声明意图"的树出发,数据库怎么决定**走哪条物理路径去取数据最省**——这就是优化器(optimizer)的全部职责。

> **核心问题**:同一句 SQL 为什么会有多种执行计划?优化器怎么从解析树出发,枚举出所有"说得通"的执行路径,又怎么在其中挑出**代价最小**的那条?这个"代价(Cost)"到底由哪几部分加起来?统计信息(直方图、最常见值、distinct 数)是怎么把"这一行到底符不符合条件"这种**不确定**的事,变成一个**可以拿去算**的数字的?以及最反直觉的一点——为什么优化器算出来的代价本质上是个**有根据的猜测**,而不是精确计算?它因此会在什么场景下出错?

> **读完本章你会明白**:为什么 SQL 是"声明式"语言这件事,直接逼出了优化器这个角色;代价模型长什么样(磁盘代价 + CPU 代价,各按顺序/随机区分);选择率(selectivity)→基数(cardinality)→每层算子代价这条**自底向上的估算链**怎么一路推上去;逻辑等价重写和物理路径枚举为什么要**分两阶段**;为什么 PostgreSQL 选**基于代价的优化(CBO)**而不是老式的"基于规则";以及统计信息过期时,优化器为什么会自信满满地选错索引。

> **如果一读觉得太难**:先只记住三件事——
> ① 同一句 SQL 有多种"取法",优化器的活儿就是给每种取法**估个价、挑最便宜的**;那个"价"=`顺序读页的代价 + 随机读页的代价 + 处理每一行的 CPU 代价`,三个数加起来;
> ② 它估算"会选出多少行"靠的是**统计信息**(一张表多少行、某列有多少不同的值、最常见值各占多少比例)——`行数 = 表总行数 × 选择率`,这个公式是整章的脊梁;
> ③ 所以优化器**不是在算真值,是在猜**——猜得准不准全看统计信息新不新、全不全。`ANALYZE` 刷统计信息后计划会变,就是这个道理。其余细节第二遍配合源码再抠,不影响理解后续的执行器。

## 一、优化器为什么必须存在:声明式 SQL 的"另一半"

第 3 章结束时,Query 树已经把 `SELECT * FROM users WHERE id = 42` 表达成了一棵结构化的树:目标列是 `*`,源表是 `users`,过滤条件是 `id = 42`。但它**故意没说**一件事:取 `id = 42` 这行,**具体怎么取**?

这里藏着一个根本性的分裂(回扣第 1 章的第三个本性:**用户说"要什么",机器只会"按位置读"**):

- SQL 是**声明式**的——你只描述"我要什么"(id 等于 42 的那行),不描述"怎么去找它"。
- 但磁盘只会按位置读写。要让 SQL 真的落地成一组对页面的读操作,必须有人把这层**"要什么"翻译成"怎么取"**的鸿沟填上。

填这个鸿沟,理论上可以有两种极端做法:

**做法 A:写死规则。** 比如"凡是 `WHERE 主键 = 常量`,一律走主键索引"。简单、确定、好实现。早期的数据库(包括 PostgreSQL 的前辈们)确实大量依赖这种**基于规则的优化(rule-based optimization,RBO)**。

**做法 B:算一算每条路大概多贵,挑最便宜的。** 比如"走索引估个价、走全表扫描估个价,谁低用谁"。这叫**基于代价的优化(cost-based optimization,CBO)**。

PostgreSQL 走的是 B,而且是**彻底的 CBO**——它几乎没有"某情况必须走某路"的死规则,一切交给代价比较。

### 不这样会怎样:规则会把你带进沟里

先看"写死规则"为什么不够用。考虑这句:

```sql
SELECT count(*) FROM users WHERE status = 'active';
```

按规则,`status` 上如果有索引,就该走索引。可如果 `status = 'active'` 的用户占了全表 95% 呢?走索引意味着:先在索引里查出 95% 行的位置,再**一行一行随机回表**取数据。随机 I/O 比顺序 I/O 慢得多(下一节会看到 PG 默认假设随机读是顺序读的 4 倍代价)。结果走索引反而比**直接顺序扫一遍整张表**还慢。

> **不这样会怎样**:一个不看数据分布的固定规则,会在"列上明明有索引、但符合条件的行占绝大多数"时,**固执地走索引,把查询拖慢几倍**。规则是死的,数据是活的;死规则套活数据,必然在边界情况翻车。

更麻烦的是连接(JOIN)。三张表 A、B、C 做连接,光"先连谁、后连谁"就有 3! = 6 种顺序;每种顺序里每一步又能选嵌套循环、哈希、归并三种算法;再叠加索引选不选、排序不排序……组合爆炸。没有任何一套固定规则能"一眼看穿"哪种组合最快——**快不快,取决于数据长什么样、有多少行**。这个"取决于数据"的事实,直接判了 RBO 的死刑:**优化必须看数据,看数据就得估代价**。

### 所以这样设计:让代价来拍板

PostgreSQL 的解法是:对同一条 SQL,**枚举出所有说得通的执行路径**(Path),给每条路径**估算一个代价(Cost,一个无量纲的浮点数)**,然后**挑代价最小的那条**,把它物化成一棵执行计划树(Plan),交给第 5 章的执行器去跑。

注意三个关键词:

- **枚举**:不是"想到哪条算哪条",而是系统性地把可能的路径都生成出来(扫描方式、连接顺序、连接算法的各种组合)。
- **估算**:代价不是测出来的(那是执行完之后的事),而是在**执行之前**、基于统计信息**算**出来的——所以本质是估计。
- **比较**:不追求"绝对正确的最优",只追求"在我枚举到的候选里挑最省的"。优化器从不保证找到全局最优(那是 NP 难的),它保证的是"在合理时间内,找到一个相当不错的计划"。

这就是 CBO 的灵魂:**把"哪个计划快"这件本来无法预知的事,转化成"哪个计划估算代价低"这件可以在执行前算出来的事**。代价估得准不准,直接决定计划好不好——而代价估得准不准,又全看**统计信息**准不准。这条因果链,是本章后半段的主线,也是优化器一切"玄学"的根源。

> 这一节我们只立住一件事:**优化器的存在,是被 SQL 的"声明式"本性逼出来的;而它选 CBO 而非 RBO,是因为"快不快取决于数据",而代价模型是唯一能把"数据长什么样"纳入考量的方式**。接下来三节,我们依次拆开 CBO 的三块积木:**代价怎么算、行数怎么估、路径怎么枚举**。

## 二、代价模型:一个查询"贵"在哪,怎么用一个数表示

优化器要比较路径,首先得有一把**统一的尺子**:不管这条路径是"顺序扫表"还是"走索引再回表"还是"两张表做哈希连接",都得能折算成**同一个无量纲的数字**——代价(Cost)。这一节我们就把这把尺子拆开看。

### 一次查询的开销,本质只有三类

抛开一切术语,一条 SQL 跑起来,机器到底在忙什么?拆到最底,无非三件事:

1. **从磁盘读数据页**。表和索引都按 8KB 页存(第 6 章详讲),取数据就是取页。
2. **在 CPU 里处理每一行**。读到的每一行,都得判断它符不符合 `WHERE`、要不要算表达式、要不要塞进连接结果。
3. 如果涉及排序、哈希,还有**比较和哈希计算的 CPU 开销**。

PostgreSQL 的代价模型,就是把这三件事各打一个分,加起来。读页的开销分两种:

- **顺序读(sequential page fetch)**:一页接一页、从前往后扫。操作系统会预读(read-ahead),磁头不用来回跳,所以便宜。
- **随机读(random page fetch)**:跳着读,想读哪页读哪页(走索引回表的典型形态)。磁头要寻道(SSD 没有机械寻道,但仍有闪存翻译层开销和读放大),所以贵。

于是代价模型的核心就是这五个**可调的常数**(术语叫 GUC 参数,可以在 `postgresql.conf` 里改):

| 常数 | 默认值 | 含义 |
|------|--------|------|
| `seq_page_cost` | 1.0 | 顺序读一个页的代价(**基准单位,永远是 1**) |
| `random_page_cost` | 4.0 | 随机读一个页的代价(默认是顺序的 4 倍) |
| `cpu_tuple_cost` | 0.01 | CPU 处理一个堆表元组(tuple)的代价 |
| `cpu_index_tuple_cost` | 0.005 | CPU 处理一个索引项的代价 |
| `cpu_operator_cost` | 0.0025 | 执行一次函数或操作符(比如 `=`、`+`)的代价 |

`seq_page_cost = 1.0` 是**整个代价体系的锚点**——所有代价都以"一次顺序读页"为单位 1。所以你会在 `EXPLAIN` 里看到代价是 `1234.56` 这种数字,它的单位就是"等价于顺序读 1234.56 个页"。

这五个默认值定义在头文件里:

> [src/include/optimizer/cost.h:24-30](../postgresql-17.0/src/include/optimizer/cost.h#L24-L30)

```c

#define DEFAULT_RANDOM_PAGE_COST  4.0
#define DEFAULT_CPU_TUPLE_COST	0.01
#define DEFAULT_CPU_INDEX_TUPLE_COST 0.005
#define DEFAULT_CPU_OPERATOR_COST  0.0025
#define DEFAULT_PARALLEL_TUPLE_COST 0.1
#define DEFAULT_PARALLEL_SETUP_COST  1000.0
```

注意文件顶部紧跟的警告:

> [src/include/optimizer/cost.h:21-23](../postgresql-17.0/src/include/optimizer/cost.h#L21-L23)

```c

```

意思是:这只是一份**默认值快照**,真正的代价计算代码读的是**变量**(`seq_page_cost` 等,运行时可被 `SET` 改掉),而不是这几个宏。变量的定义和初值在 costsize.c 开头:


`Cost` 本身就是个 `double`:

> [src/include/nodes/nodes.h:251](../postgresql-17.0/src/include/nodes/nodes.h#L251)

```c

```

注释里那句 "in page-access units"(以页访问为单位),正点明了代价的**单位就是"读一个页"**。

### 为什么是"加和模型",而不是更复杂的公式

有人会问:磁盘 I/O 和 CPU 时间根本不是一回事(一个是毫秒级、一个是纳秒级),凭什么能直接加?这是个好问题,答案是:**代价不是真实的物理时间,而是一个刻意构造出来的、相对的"贵贱分数"**。优化器不关心一条计划绝对要跑多久,只关心"A 比 B 便宜"。把"读页的次数 × 页代价"和"处理的行数 × 行代价"加在一起,就得到了一个**能把不同计划放到同一根数轴上比大小**的分数。至于这个分数是不是真实的毫秒数,无所谓——只要它**单调地反映了真实开销的趋势**就行。

这套加和的口径,源码顶部的注释说得清清楚楚:

> [src/backend/optimizer/path/costsize.c:6-13](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L6-L13)

```c
 * Path costs are measured in arbitrary units established by these basic
 * parameters:
 *
 *	seq_page_cost		Cost of a sequential page fetch
 *	random_page_cost	Cost of a non-sequential page fetch
 *	cpu_tuple_cost		Cost of typical CPU time to process a tuple
 *	cpu_index_tuple_cost  Cost of typical CPU time to process an index tuple
 *	cpu_operator_cost	Cost of CPU time to execute an operator or function
```

"arbitrary units"(任意单位)——作者明确告诉你:别把这个数当成秒,它就是个**用来比较的相对量**。

### 代价分两段:启动代价与总代价

到这里还没完。PG 的代价模型有个更细的设计:每条路径记**两个**代价,而不是一个:

> [src/backend/optimizer/path/costsize.c:37-45](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L37-L45)

```c
 * We compute two separate costs for each path:
 *		total_cost: total estimated cost to fetch all tuples
 *		startup_cost: cost that is expended before first tuple is fetched
 * In some scenarios, such as when there is a LIMIT or we are implementing
 * an EXISTS(...) sub-select, it is not necessary to fetch all tuples of the
 * path's result.  A caller can estimate the cost of fetching a partial
 * result by interpolating between startup_cost and total_cost.  In detail:
 *		actual_cost = startup_cost +
 *			(total_cost - startup_cost) * tuples_to_fetch / path->rows;
```

- **`total_cost`**:把所有结果行都吐出来的总代价。
- **`startup_cost`**:**在吐出第一行之前**,必须先花掉的代价。

为什么要分两段?想一个排序节点(`SORT`):它必须**把所有输入读完、排好序**,才能吐出第一行——所以它的 startup_cost 很高(几乎等于 total_cost)。而一个顺序扫描(`SeqScan`)是边读边吐,读完第一页就能输出第一行,所以 startup_cost 很低。

这俩数在遇到 `LIMIT`、`EXISTS` 子查询时至关重要:如果你只要前 10 行,优化器会按上面那段公式用 `startup_cost` 和 `total_cost` **插值**估出"只取 10 行的代价"。一个 startup_cost 低、total_cost 高的路径(比如顺序扫描),在 `LIMIT` 小的时候反而可能赢过 startup_cost 高的排序路径——这正是 `EXPLAIN` 里有时你会看到"明明有索引却走了全表扫描"的合理原因。

### 代码佐证:顺序扫描的代价是怎么算出来的

把上面的概念落到一个最简单的算子上看。顺序扫描(`SeqScan`)的代价函数是 `cost_seqscan`:

> [src/backend/optimizer/path/costsize.c:284](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L284)

```c
cost_seqscan(Path *path, PlannerInfo *root,
			 RelOptInfo *baserel, ParamPathInfo *param_info)
```

它算代价的核心就几行(中间省略并行分支):

> [src/backend/optimizer/path/costsize.c:304-350](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L304-L350)

```c
	if (!enable_seqscan)
		startup_cost += disable_cost;
	...
	disk_run_cost = spc_seq_page_cost * baserel->pages;
	...
	startup_cost += qpqual_cost.startup;
	cpu_per_tuple = cpu_tuple_cost + qpqual_cost.per_tuple;
	cpu_run_cost = cpu_per_tuple * baserel->tuples;
	/* tlist eval costs are paid per output row, not per tuple scanned */
	startup_cost += path->pathtarget->cost.startup;
	cpu_run_cost += path->pathtarget->cost.per_tuple * path->rows;
	...
	path->startup_cost = startup_cost;
	path->total_cost = startup_cost + cpu_run_cost + disk_run_cost;
```

逐行翻译,就是代价模型的全貌:

- `disk_run_cost = spc_seq_page_cost * baserel->pages` —— **磁盘代价 = 每页代价 × 总页数**。顺序扫描要读的,就是这张表的所有页(`baserel->pages`,来自 `pg_class.relpages`)。乘上 `seq_page_cost`(=1.0),就是磁盘这部分的开销。
- `cpu_per_tuple = cpu_tuple_cost + qpqual_cost.per_tuple` —— **处理每一行的 CPU 代价 = 基础 tuple 代价 + 判断 `WHERE` 条件的代价**。`qpqual` 就是"在堆表元组上逐行求值的过滤条件"。
- `cpu_run_cost = cpu_per_tuple * baserel->tuples` —— **CPU 代价 = 每行代价 × 总行数**。注意这里是 `tuples`(表里全部行),不是 `rows`(过滤后输出的行)——因为顺序扫描**每一行都得读出来、都得判断一次 `WHERE`**,哪怕最后大部分被过滤掉。
- `path->total_cost = startup_cost + cpu_run_cost + disk_run_cost` —— **总代价 = 启动代价 + CPU 代价 + 磁盘代价**,三段加起来。

还有一处值得专门点出:

> [src/backend/optimizer/path/costsize.c:304-305](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L304-L305)

```c
	if (!enable_seqscan)
		startup_cost += disable_cost;
```

`enable_seqscan` 是个开关(GUC)。如果用户 `SET enable_seqscan = off`,这里就给代价**加上一个天文数字** `disable_cost`。它定义在同一个文件:

> [src/backend/optimizer/path/costsize.c:130](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L130)

```c
Cost		disable_cost = 1.0e10;
```

`1.0e10`——一百亿。这并不是"真的禁止顺序扫描"(PG 不会硬性禁止任何算子),而是**把它的代价抬到天上,让它几乎不可能在比较中胜出**。这是 PG 一个一贯的手法:**所有 `enable_xxx = off` 都是"加巨款劝退",不是"真禁用"**。理解了这一点,你就明白为什么 `SET enable_indexscan = off` 之后,有时计划里还是会出现索引扫描——因为如果**所有别的路径都被更狠地劝退了**,这个被加了 1e10 的路径反而还是最便宜的。代价模型始终在比较,而不是在服从命令。

> 这一节把"代价是什么"钉死了:**一个无量纲的相对分数,= 顺序读页代价 + 随机读页代价 + 每行 CPU 代价,分 startup/total 两段**。但注意 `cost_seqscan` 里反复出现的 `baserel->pages`、`baserel->tuples`——这些数字从哪来?它们来自统计信息。下一节就讲:**优化器凭什么知道一张表有多少页、多少行,以及一句 `WHERE` 会过滤掉多少行**。

## 三、统计信息与选择率:把"不确定性"变成"一个能算的数"

这是优化器最像"黑魔法"的一节,也是最该讲透的一节。因为代价模型本身只是个加法公式,公式里填什么数,才决定计划对不对——而那些数,绝大多数来自**统计信息**。

### 优化器凭什么知道一张表多大

`cost_seqscan` 用到的 `baserel->pages`(总页数)和 `baserel->tuples`(总行数),不是优化器现场去数出来的(那得把整张表扫一遍,比执行查询还慢)。它们是**事先存好的**,存在系统目录 `pg_class` 的两个字段里:

> [src/include/catalog/pg_class.h:62-66](../postgresql-17.0/src/include/catalog/pg_class.h#L62-L66)

```c
/* # of blocks (not always up-to-date) */
int32		relpages BKI_DEFAULT(0);

/* # of tuples (not always up-to-date; -1 means "unknown") */
float4		reltuples BKI_DEFAULT(-1);
```

注释里那句 "**not always up-to-date**"(不总是最新的)是关键——这两个数字是 `VACUUM`、`ANALYZE`、以及某些 DDL 操作**顺手更新**的快照,**不是实时的**。表里插了一百万行而还没 `ANALYZE`,`pg_class.reltuples` 可能还停在旧值。这就是为什么"统计信息过期会导致计划变差"——根子就在这俩字段会过时。

优化器拿这俩数,是通过一个叫 `get_relation_info` 的函数,在规划开始时读进 `RelOptInfo` 结构(优化器内部描述一个表的节点):

> [src/backend/optimizer/util/plancat.c:116](../postgresql-17.0/src/backend/optimizer/util/plancat.c#L116)

```c
get_relation_info(PlannerInfo *root, Oid relationObjectId, bool inhparent,
				  RelOptInfo *rel)
```

它把 `pg_class.reltuples` 填进 `rel->tuples`,供 `cost_seqscan` 之类消费。链条是:`pg_class.reltuples` → relcache 缓存 → `get_relation_info` → `baserel->tuples` → 代价公式。

### 表级统计不够:还要看列里数据长什么样

光知道"这张表有 100 万行"还不够。考虑 `WHERE city = '北京'`,优化器得知道"北京"在这张表里占多大比例,才能估算出会选出多少行。这个比例,叫**选择率(selectivity)**——一个 0 到 1 之间的数,表示"符合某个条件的行占总行的比例"。选择率 0.01 表示百分子一,0.5 表示一半。

但选择率怎么估?优化器又不能真去扫表。答案是:**ANALYZE 预先把每一列的数据分布特征采样统计好,存进另一个系统目录 `pg_statistic`**。优化器查这张表,就能把"这一行符不符合 `city='北京'`"这种**执行时才知道**的事,变成一个**执行前就能查到的频次**。

`pg_statistic` 的结构,核心是这几列:

> [src/include/catalog/pg_statistic.h:37-69](../postgresql-17.0/src/include/catalog/pg_statistic.h#L37-L69)

```c
/* the fraction of the column's entries that are NULL: */
float4		stanullfrac;
...
/* ----------------
 * stadistinct indicates the (approximate) number of distinct non-null
 * data values in the column.  The interpretation is:
 *		0		unknown or not computed
 *		> 0		actual number of distinct values
 *		< 0		negative of multiplier for number of rows
 * ...
 * ---------------- */
float4		stadistinct;
```

注意 `stadistinct` 这个**带符号的小聪明**:正数就是 distinct 值的绝对个数(适合小集合);负数是一个**乘子**(比如 -0.5 表示"distinct 数 ≈ 总行数 × 0.5",适合大集合,因为绝对个数会随表增长而无意义地膨胀,用比例更稳定)。这个设计本身就在告诉你:统计信息是**为估算服务的**,形式服从估算的需要。

除了 `stanullfrac` 和 `stadistinct`,`pg_statistic` 每列还有**最多 5 个"统计槽位"**(`stakind1..5`),每个槽位装一种统计。最常用的三种:

- **MCV(Most Common Values,最常见值)**:`stakind=1`。记录这一列里出现频次最高的若干个值,以及各自的比例。比如 `city` 列的 MCV 可能是 `{北京: 0.30, 上海: 0.15, 广州: 0.08, ...}`。
- **直方图(histogram_bounds)**:`stakind=2`。把列的值域等频地切成若干个桶,记录每个桶的边界。它专门用来估**范围条件**(如 `age > 30`)的选择率。
- **相关性(correlation)**:`stakind=3`。物理行顺序和逻辑值顺序的相关程度,影响"走索引要不要随机回表"的代价估算。

> 为什么等值(`=`)用 MCV、范围(`> <`)用直方图?因为它们各自最擅长回答的问题不同:MCV 直接告诉你"某个具体值占多少比例",对等值查询一查一个准;直方图告诉你"小于某个值的有多少",对范围查询正好。直方图**故意排除了 MCV 里那些高频值**(否则高频值会让某些桶畸重),两者是**互补**的——一个查询的选择率,常常是"MCV 部分的选择率 + 直方图部分的选择率"拼起来的。这一点 `selfuncs.c` 里有明确注释,我们在源码精读里会看到。

### ANALYZE:统计信息是怎么收集的(为什么是采样)

这些统计是 `ANALYZE` 命令收集的。它的入口是 `analyze_rel`:

> [src/backend/commands/analyze.c:111](../postgresql-17.0/src/backend/commands/analyze.c#L111)

```c
analyze_rel(Oid relid, RangeVar *relation,
			VacuumParams *params, List *va_cols, bool in_outer_xact,
			BufferAccessStrategy bstrategy)
```

它核心干两件事:**采样拿一批行**(`acquire_sample_rows`),然后**对这些样本算统计**(`compute_scalar_stats`)。

> [src/backend/commands/analyze.c:1158](../postgresql-17.0/src/backend/commands/analyze.c#L1158)

```c
acquire_sample_rows(Relation onerel, int elevel,
```

采样,不是全表扫。为什么?

> **不这样会怎样**:如果要算精确统计,`ANALYZE` 就得读遍整张表的每一行——那它和执行一遍 `SELECT *` 有什么区别?一张几亿行的大表,光为了维护统计信息就要扫几分钟,根本不可行。更糟的是,数据天天在变,你今天精确统计完,明天就又不准了——精确性是个**保质期极短**的资产,为之付出全表扫描的代价,极其不划算。

所以 `ANALYZE` 走的是**采样**:随机抽若干页(Vitter 算法,保证每行被抽中的概率均等),只对样本算统计,再**按比例外推**到全表。默认采 30000 行(`default_statistics_target = 100`,实际样本数是 `300 × target`)。对于几十亿行的大表,采几万行算出来的 MCV 和直方图,已经足够让优化器把选择率估得八九不离十——**统计信息的精髓,是用很小的代价换取"足够准"的估算,而不是追求精确**。

distinct 数的估算最能体现这种"够用就行"的工程哲学。`compute_scalar_stats` 对采到的样本,用一个叫 **Haas-Stokes 估计器**(来自 IBM 的一篇统计论文)的公式,根据"样本里恰好只出现一次的值有多少"来推算全表的 distinct 数:

> [src/backend/commands/analyze.c:2562-2594](../postgresql-17.0/src/backend/commands/analyze.c#L2562-L2594)

```c
int			f1 = ndistinct - nmultiple + toowide_cnt;
int			d = f1 + nmultiple;
double		n = samplerows - null_cnt;
double		N = totalrows * (1.0 - stats->stanullfrac);
double		stadistinct;
if (N > 0)
	stadistinct = (n * d) / ((n - f1) + f1 * n / N);
```

公式 `n*d / (n - f1 + f1*n/N)` 里,`f1` 是"样本里只出现一次的值个数",`d` 是样本 distinct 数,`n` 是样本行数,`N` 是全表行数。直觉是:**样本里"只出现一次"的值越多,说明全表还有大量没被采到的不同值**(否则它们在样本里会更频繁出现)。这个公式不是 PG 发明的,是借用统计学的现成结论——**数据库内核的很多"魔法数字",背后都是几十年前的统计学论文**。理解这一点,你就不会觉得优化器是在"瞎猜",它是在用**有理论依据的估计方法**猜。

### 选择率怎么算:等值查询的完整推演

现在把所有积木拼起来,看一句最简单的 `WHERE city = '北京'`,优化器怎么从统计信息推出"会选出多少行"。

等值查询的选择率函数是 `eqsel`,但它是个薄壳,真正干活的是 `eqsel_internal` → `var_eq_const`:

> [src/backend/utils/adt/selfuncs.c:294](../postgresql-17.0/src/backend/utils/adt/selfuncs.c#L294)

```c
double
var_eq_const(VariableStatData *vardata, Oid oproid, Oid collation,
```

它第一步先取出 NULL 比例:

> [src/backend/utils/adt/selfuncs.c:315-321](../postgresql-17.0/src/backend/utils/adt/selfuncs.c#L315-L321)

```c
if (HeapTupleIsValid(vardata->statsTuple))
{
	Form_pg_statistic stats;
	stats = (Form_pg_statistic) GETSTRUCT(vardata->statsTuple);
	nullfrac = stats->stanullfrac;
}
```

然后拿 MCV 槽位,把常量 `'北京'` 和 MCV 里的每个值逐一比对。**如果命中**(北京在 MCV 列表里),选择率就是它在 MCV 里的频次——这是最准的情况:

> [src/backend/utils/adt/selfuncs.c:397-404](../postgresql-17.0/src/backend/utils/adt/selfuncs.c#L397-L404)

```c
if (match)
{
	/*
	 * Constant is "=" to this common value.  We know selectivity
	 * exactly (or as exactly as ANALYZE could calculate it, anyway).
	 */
	selec = sslot.numbers[i];
}
```

注释那句 "We know selectivity exactly"——对高频值,选择率是已知的、精确的(精确到 `ANALYZE` 采样精度)。

**如果没命中**(北京不在 MCV 里,是个长尾值),就用 fallback 公式:

> [src/backend/utils/adt/selfuncs.c:405-436](../postgresql-17.0/src/backend/utils/adt/selfuncs.c#L405-L436)

```c
else
{
	double		sumcommon = 0.0;
	double		otherdistinct;

	for (i = 0; i < sslot.nnumbers; i++)
		sumcommon += sslot.numbers[i];
	selec = 1.0 - sumcommon - nullfrac;
	CLAMP_PROBABILITY(selec);

	/*
	 * and in fact it's probably a good deal less. We approximate that
	 * all the not-common values share this remaining fraction
	 * equally, so we divide by the number of other distinct values.
	 */
	otherdistinct = get_variable_numdistinct(vardata, &isdefault) -
		sslot.nnumbers;
	if (otherdistinct > 1)
		selec /= otherdistinct;

	/*
	 * Another cross-check: selectivity shouldn't be estimated as more
	 * than the least common "most common value".
	 */
	if (sslot.nnumbers > 0 && selec > sslot.numbers[sslot.nnumbers - 1])
		selec = sslot.numbers[sslot.nnumbers - 1];
}
```

这段逻辑翻译成公式就是:

```
选择率 = (1 - MCV所有频次之和 - null比例) / (总distinct数 - MCV个数)
```

意思是:把"所有非 MCV、非 NULL 的值"占的总比例(`1 - sumcommon - nullfrac`),**平均分摊**到每一个长尾 distinct 值上。注释直白承认 "and in fact it's probably a good deal less"(实际上可能比这还少)——**这是个"均摊假设"**,假设长尾值们出现频率差不多。现实里当然不是严格均摊,但没别的信息可用时,均摊是最不偏的猜测。

最后还有个兜底:算出来的选择率不能超过"MCV 里最不常见那个值的频次"——因为一个连 MCV 都进不去的值,它的真实频次理应小于 MCV 的门槛。这个交叉检查(`selec > sslot.numbers[sslot.nnumbers - 1]` 就截断)是个防呆设计,保证估算不会荒腔走板。

> 这里有个**容易翻车的细节**,值得专门点破:**等值查询的选择率,根本不读直方图**。很多人(包括不少教科书)以为"直方图是选择率的核心",其实 PG 里等值(`=`)靠 MCV + distinct 均摊,直方图只服务于**不等式**(`< <= > >=`,函数是 `ineq_histogram_selectivity`,selfuncs.c:1041 起)。原因很简单:MCV 直接给了精确频次,比直方图更准;只有估范围时,才需要直方图告诉你"小于某值的有多少"。把这点讲错,是写优化器文章最常见的失误之一。

### 选择率 → 基数 → 代价:自底向上的估算链

拿到选择率后,怎么推到代价?链条就两步:

**第一步:选择率 × 表总行数 = 结果行数(基数,cardinality)。** 这一步在 `set_baserel_size_estimates` 里:

> [src/backend/optimizer/path/costsize.c:5239-5253](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L5239-L5253)

```c
set_baserel_size_estimates(PlannerInfo *root, RelOptInfo *rel)
{
	double		nrows;

	/* Should only be applied to base relations */
	Assert(rel->relid > 0);

	nrows = rel->tuples *
		clauselist_selectivity(root,
							   rel->baserestrictinfo,
							   0,
							   JOIN_INNER,
							   NULL);

	rel->rows = clamp_row_est(nrows);
	...
}
```

核心就一行:`nrows = rel->tuples * clauselist_selectivity(...)`。`rel->tuples` 是表总行数(来自 `pg_class.reltuples`),`clauselist_selectivity` 把这张表上所有 `WHERE` 条件的选择率综合起来(多个条件的选择率怎么组合,PG 有一套默认独立性假设:认为不同列的条件互相独立,选择率相乘——这也是个"猜测",现实里列之间常有相关性,所以 PG 后来加了 `CREATE STATISTICS` 扩展统计来纠正这一点),两者相乘,就是估算出来的"这张表过滤后会输出多少行"——这就是 `rel->rows`,代价函数里到处用的那个数。

注意最后还过了一道 `clamp_row_est`:

> [src/backend/optimizer/path/costsize.c:202-218](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L202-L218)

```c
clamp_row_est(double nrows)
{
	/*
	 * Avoid infinite and NaN row estimates.  Costs derived from such values
	 * are going to be useless.  Also force the estimate to be at least one
	 * row, to make explain output look better and to avoid possible
	 * divide-by-zero when interpolating costs.  Make it an integer, too.
	 */
	if (nrows > MAXIMUM_ROWCOUNT || isnan(nrows))
		nrows = MAXIMUM_ROWCOUNT;
	else if (nrows <= 1.0)
		nrows = 1.0;
	else
		nrows = rint(nrows);

	return nrows;
}
```

它把行数估计划到 `[1, MAXIMUM_ROWCOUNT]` 之间、且取整。注释点明了两个理由:防止无穷大/NaN 污染后续代价(无穷大代价没意义);强制至少 1 行,避免后面插值时除以零。这也是个"工程防呆"——**估算的数本身就不精确,所以下游代码必须能容忍粗糙的输入**。

**第二步:基数 × 每行代价 = 该算子的代价,再往上传给上层算子。** 这就是上一节 `cost_seqscan` 里 `cpu_run_cost = cpu_per_tuple * baserel->tuples` 的逻辑(注意 SeqScan 用 tuples 而非 rows,因为它要处理全部行);而对于索引扫描、连接,用的就是过滤后的 `rows`。每层算子都拿"下层算子估算输出的行数"作为自己的输入,逐层往上累加,直到根节点——整棵计划树的总代价就出来了。

这条链——**统计信息 → 选择率 → 基数 → 单算子代价 → 整树代价**——是优化器的脊柱。理解了它,你就理解了为什么"统计信息过期,计划就崩":链条最底层的 `tuples` 或 MCV 频次一旦失真,误差会**沿着这条链一路放大**,到顶上可能差出几个数量级。

> 这一节是本章的"发动机"。把它浓缩成一句话:**优化器把"会选出多少行"这件不确定的事,变成"选择率 × 总行数"这个能算的数;选择率又靠统计信息(MCV、distinct、直方图)从"数据分布"里推出来**。下一节看这些估算好的代价,是怎么在众多候选路径里被**枚举和比较**的。

## 四、两阶段:先"逻辑等价重写",再"物理路径枚举"

到目前为止,我们好像默认"优化器一上来就在比代价"。其实不是。在比代价之前,优化器要先做一件性质完全不同的事:**逻辑重写**。PG 把优化器的工作明确分成两个阶段,这两个阶段解决的问题压根不是一回事。

### 为什么必须分两阶段

先想一个场景。你写了:

```sql
SELECT * FROM users WHERE id IN (SELECT user_id FROM orders WHERE amount > 1000);
```

这棵 Query 树里有个**子查询**(`IN (SELECT ...)`)。如果就这么交给物理规划,优化器得为"子查询怎么执行"单独规划一条路径——可能是每查到一行 users 就把子查询跑一遍(相关子查询的典型灾难)。但其实,这个 `IN` 子查询在**逻辑上等价于一个连接**:

```sql
SELECT u.* FROM users u JOIN orders o ON u.id = o.user_id WHERE o.amount > 1000;
```

连接,优化器有一整套成熟的处理手段(连接顺序、哈希/嵌套/归并)。所以**最聪明的做法,是先把 `IN` 子查询"提升"成连接,再进入物理规划**。这个"提升",叫 `pull_up_subqueries`(子查询上提),它不改变 SQL 的语义(结果完全一样),只是把同一件事**换一种逻辑形态表达**,好让后面的物理规划更顺手。

> **不这样会怎样**:如果逻辑重写和物理规划搅在一起,每发现一种"逻辑等价变形"就得重新跑一遍代价比较,工作会极度混乱、还容易重复或遗漏。更糟的是,有些逻辑变形(比如把 `IN` 变连接、把外连接转内连接、消除冗余的 `DISTINCT`)**只有在变形之后,物理层面才会暴露出更多、更优的路径**——不先变形,优化器根本看不到这些路径的存在,等于自缚手脚。

所以 PG 的做法是:**先把所有"纯逻辑的、和代价无关的、保证语义等价的变形"一口气做完**(这是重写阶段),**再在变形后的、干净的 Query 树上做物理路径枚举和代价比较**(这是规划阶段)。两阶段各司其职,边界清晰。


"the stuff that should only be done once per Query object"(每个 Query 对象只做一次的事)——说的就是逻辑重写这类预处理。

> 注意:这里说的"两阶段",和 PostgreSQL 还有一个独立的 **rule 系统(rule rewriter)** 不是一回事。rule 系统在解析之后、交给优化器之前就跑完了(它处理视图展开、规则触发),它的产物就是 Query 树。本章说的"逻辑重写",是指优化器**自己内部**、在 `subquery_planner` 开头做的那些等价变形。两者一前一后,但分属不同模块。新手常把这俩混为一谈,这里点破。

### 物理路径枚举:从单表路径到连接顺序

进入 `grouping_planner` 后,核心是 `make_one_rel`:

> [src/backend/optimizer/path/allpaths.c:170-171](../postgresql-17.0/src/backend/optimizer/path/allpaths.c#L170-L171)

```c

{
```

它的工作分两步:**先给每个基表(单表)生成所有可行路径,再考虑表和表之间怎么连接**。

**第一步:单表路径。** 对每个基表,生成它的候选扫描路径:

> [src/backend/optimizer/path/allpaths.c:218-221](../postgresql-17.0/src/backend/optimizer/path/allpaths.c#L218-L221)

```c
	/*
	 * Generate access paths for each base rel.
	 */
	set_base_rel_pathlists(root);
```

对一张表,`set_base_rel_pathlists` 会生成至少两条路径:**顺序扫描**(永远考虑)和**每一个可用索引的索引扫描/位图扫描**。每条路径都跑一遍对应的 `cost_xxx` 算出代价,挂在 `RelOptInfo` 的路径列表里。一张表上挂多少条路径,取决于它有多少索引、有没有条件能利用索引。最后 `set_cheapest` 选出这张表最便宜的那条,作为"代表"。

**第二步:连接顺序的枚举。** 这是优化器最经典、也最考验算法的部分。多张表做连接,先连谁后连谁,对代价影响巨大。考虑 4 张表 A、B、C、D 的连接:顺序有 4! = 24 种,再加上每步可选不同连接算法,组合爆炸。

PostgreSQL 用**动态规划(dynamic programming)**来系统但不爆炸地枚举。入口是 `make_rel_from_joinlist`:

> [src/backend/optimizer/path/allpaths.c:3305-3306](../postgresql-17.0/src/backend/optimizer/path/allpaths.c#L3305-L3306)

```c

{
```

它先数一下要连的表有多少个,这就是动态规划的"层数":

> [src/backend/optimizer/path/allpaths.c:3312-3317](../postgresql-17.0/src/backend/optimizer/path/allpaths.c#L3312-L3317)

```c
	/*
	 * Count the number of child joinlist nodes.  This is the depth of the
	 * dynamic-programming algorithm we must employ to consider all ways of
	 * joining the child nodes.
	 */
	levels_needed = list_length(joinlist);
```

然后根据表的数量,在两种连接搜索算法里二选一:

> [src/backend/optimizer/path/allpaths.c:3360-3377](../postgresql-17.0/src/backend/optimizer/path/allpaths.c#L3360-L3377) 附近


它的思路是经典的 DP:**自底向上,逐层扩大**。第 1 层是所有单表(已经在第一步算好了各自的最优路径);第 2 层是所有"两表连接"的可能组合,对每种组合试各种连接算法,算代价,留下最优;第 3 层是"三表连接",用第 2 层的结果继续拼……直到第 N 层拼出"所有表连在一起"的那个最终关系,它的最优路径就是整个连接的最优计划。

每一层的工作由 `join_search_one_level` 完成,它的注释把这层意思点得很明白:

```c
 *	  jointree items.  (This is one step of the dynamic-programming method
 *	  embodied in standard_join_search.)  Join rel nodes for each feasible

 *	  Implementation paths are created for each such joinrel, too.
```

"one step of the dynamic-programming method"(动态规划方法的一步)——白纸黑字。

### 不这样会怎样:表太多时,动态规划也会爆炸

动态规划的好处是把连接顺序的搜索从 `O(N!)` 压到 `O(3^N)` 量级,对 N=10 张表还能扛住。但当 N 很大(几十张表全连接),`3^N` 依然爆炸——优化器会卡在规划阶段,查询还没开始跑就先慢半分钟。

> **不这样会怎样**:一个表数极多的查询,纯动态规划会把规划时间本身变成灾难。用户会觉得"我 SQL 都还没执行,光解析就卡住了",体验极差。

所以 PG 有个**逃生阀**:当参与连接的表数超过阈值 `geqo_threshold`(默认 12),改用 **GEQO(Genetic Query Optimization,遗传算法)**。它不保证找到最优,但能在表多时**快速收敛到一个"还行"的计划**,把规划时间压住。入口在 `make_rel_from_joinlist` 里按阈值分发,真正的遗传算法实现在:

> [src/backend/optimizer/geqo/geqo_main.c:71-72](../postgresql-17.0/src/backend/optimizer/geqo/geqo_main.c#L71-L72)

```c

geqo(PlannerInfo *root, int number_of_rels, List *initial_rels)
{
```

文件顶部就挑明它是遗传算法:

> [src/backend/optimizer/geqo/geqo_main.c:1-6](../postgresql-17.0/src/backend/optimizer/geqo/geqo_main.c#L1-L6)

```c
/*------------------------------------------------------------------------
 *
 * geqo_main.c
 *	  solution to the query optimization problem
 *	  by means of a Genetic Algorithm (GA)
```

GEQO 把"连接顺序选择"类比成旅行商问题(TSP),用基因编码连接顺序,通过选择、交叉、变异在解空间里搜索。这是个**用质量换时间**的明确权衡:表少时(PG 认为能算得起)用 DP 求尽量优;表多时(算不起了)用 GEQO 求一个能用的。这也是 CBO 务实的一面——**它追求的是"在可用时间内最好的计划",不是数学最优**。

### 选出最优后,还要物化成执行计划

枚举 + 比较完,根节点的 `RelOptInfo` 里会有一条最优 `Path`。但 `Path` 只是优化器内部的一个"候选描述",还不是一个能直接执行的、带完整节点信息的结构。最后一步,要把这条最优 Path **翻译**成一棵 **Plan 树**(执行器认识的形态),由 `create_plan` 完成:

> [src/backend/optimizer/plan/createplan.c:337-338](../postgresql-17.0/src/backend/optimizer/plan/createplan.c#L337-L338)

```c

create_plan(PlannerInfo *root, Path *best_path)
{
```

它递归地把每个 Path 节点转成对应的 Plan 节点(SeqPath→SeqPlan、HashJoinPath→HashJoinPlan……)。这棵 Plan 树,才是交给第 5 章执行器去真正"跑"的东西。优化器到这一步,使命完成。

> 这一节把优化器的**整体工作流**立起来了:**逻辑重写(预处理 Query 树)→ 单表路径生成 → 动态规划枚举连接顺序(表多则退化为 GEQO)→ 选最优 Path → 物化成 Plan 树**。代价模型和选择率,是嵌在这条流程里、为每一步"打分"的尺子。

## 五、连接的代价:为什么同样两张表,有时用 Hash、有时用 Nested Loop

上一节讲了"连接顺序怎么枚举",但每一步连接**用什么算法**,也是路径的一部分。PG 对每个连接关系会同时考虑三种连接算法,各自估代价:

- **嵌套循环连接(Nested Loop Join)**:对外层结果的每一行,扫一遍内层找匹配。
- **哈希连接(Hash Join)**:先把内层建一张哈希表,再用外层每一行去探测。
- **归并连接(Merge Join)**:两边都按连接键排好序,归并式往前走。

它们谁赢,完全取决于代价估算。看嵌套循环的代价核心(在 `final_cost_nestloop` 里):

> [src/backend/optimizer/path/costsize.c:3461-3473](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L3461-L3473)

```c
	else
	{
		/* Normal-case source costs were included in preliminary estimate */

		/* Compute number of tuples processed (not number emitted!) */
		ntuples = outer_path_rows * inner_path_rows;
	}

	/* CPU costs */
	cost_qual_eval(&restrict_qual_cost, path->jpath.joinrestrictinfo, root);
	startup_cost += restrict_qual_cost.startup;
	cpu_per_tuple = cpu_tuple_cost + restrict_qual_cost.per_tuple;
	run_cost += cpu_per_tuple * ntuples;
```

关键就一行:`ntuples = outer_path_rows * inner_path_rows`。嵌套循环要处理的元组数 = **外层行数 × 内层行数**(笛卡尔积规模),然后乘以每行 CPU 代价。这立刻说明:**嵌套循环的代价,对"内层结果有多大"极度敏感**。如果内层有 1 万行、外层也有 1 万行,那就是 1 亿次比较——代价爆炸。

那为什么嵌套循环有时候反而最优?因为它有个别的算法没有的优势:**内层可以被"参数化重扫"**。如果内层是个走索引的扫描(比如 `orders.user_id` 上有索引),那么对外层每一行,内层不是扫 1 万行,而是用索引**只取匹配的那几行**——内层的有效代价从"1 万行"缩成"几行"。这种"参数化嵌套循环"在**外层结果行数少、内层有高选择性索引**时,是杀器。代价模型里专门有 `inner_scan_frac` 来折扣内层重扫代价:

> [src/backend/optimizer/path/costsize.c:3409-3411](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L3409-L3411)

```c
			run_cost += inner_run_cost * inner_scan_frac;
			...
			run_cost += (outer_matched_rows - 1) * inner_rescan_run_cost * inner_scan_frac;
```

`inner_scan_frac` 就是"内层每次重扫实际只命中多大比例",乘上去就把内层代价打了折。

哈希连接是另一种思路,它的代价分两段(对应 PG 的 `initial_cost_hashjoin` + `final_cost_hashjoin` 两段式)。建表代价在初始阶段:

> [src/backend/optimizer/path/costsize.c:4096-4108](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L4096-L4108)

```c
	/*
	 * Cost of computing hash function: must do it once per input tuple.  We
	 * charge one cpu_operator_cost for each column's hash function.  Also,
	 * tack on one cpu_tuple_cost per inner row, to model the costs of
	 * inserting the row into the hashtable.
	 ...
	startup_cost += (cpu_operator_cost * num_hashclauses + cpu_tuple_cost)
		* inner_path_rows;
	run_cost += cpu_operator_cost * num_hashclauses * outer_path_rows;
```

注意 `startup_cost` 里那一项:对内层每一行都要算哈希 + 插入哈希表——所以 **Hash Join 的 startup_cost 正比于内层行数**。这意味着:**内层越大,Hash Join 越慢启动**;如果查询带 `LIMIT` 要前几行,大内层的 Hash Join 可能输给 Nested Loop。

探测代价在 final 阶段:

> [src/backend/optimizer/path/costsize.c:4396-4398](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L4396-L4398)

```c
	startup_cost += hash_qual_cost.startup;
	run_cost += hash_qual_cost.per_tuple * outer_path_rows *
		clamp_row_est(inner_path_rows * innerbucketsize) * 0.5;
```

外层每一行去探测,期望匹配的桶内元组数是 `inner_path_rows * innerbucketsize`(内层行数 × 每桶大小)。注意末尾 `* 0.5`:注释解释为"只有哈希值精确匹配时才真正算连接条件",平均打了对折。这又是个**统计假设**(假设哈希分布均匀)。

把两种连接的代价公式摆一起,优劣就清楚了:

| 场景 | 谁赢 | 为什么 |
|------|------|--------|
| 两边都很大、没好索引 | **Hash Join** | 嵌套循环是 O(外×内),Hash 是 O(外+内)建表+探测 |
| 外层很小、内层有索引 | **Nested Loop** | 内层靠索引只取几行,嵌套循环重扫很便宜 |
| 两边都已按连接键排序 | **Merge Join** | 不用建哈希表、不用嵌套,直接归并,省 |
| 带 `LIMIT`、外层行数少 | 常是 Nested Loop | startup_cost 低,不用先建大哈希表 |

**所有这些判断,都不是写死的规则,而是代价公式比出来的结果**。同一张 `orders` 表,今天 1 万行优化器选 Hash Join,明天涨到 1000 万行且有了索引,优化器可能改选 Nested Loop——计划随数据变,这正是 CBO 的核心价值。

### EXPLAIN 看代价:一个对照示意

`(以下 EXPLAIN 输出为示意格式,非真实运行结果,数字仅作说明用)`

```text
-- 场景:大表 orders(1000万行) join 小表 users(1万行),连接键都有索引

EXPLAIN SELECT * FROM orders o JOIN users u ON o.user_id = u.id WHERE u.city = '北京';

-- 优化器可能给出的计划(示意):
                                     QUERY PLAN
-----------------------------------------------------------------------------------------
 Hash Join  (cost=15432.00..895421.33 rows=3000000 width=...)
   Hash Cond: (o.user_id = u.id)
   ->  Seq Scan on orders o  (cost=0.00..234100.00 rows=10000000 width=...)
   ->  Hash  (cost=180.50..180.50 rows=3000 width=...)
         ->  Index Scan using idx_users_city on users u  (cost=0.29..180.50 rows=3000 width=...)
              Index Cond: (city = '北京'::text)
```

读这棵计划树自底向上:先用索引扫出 `city='北京'` 的 3000 个用户(代价 180.50,很低,因为索引高选择率);把这 3000 行建哈希表;再顺序扫 1000 万行 orders(代价 234100,磁盘+CPU);每行 orders 去哈希表探测。最终 Hash Join 总代价约 89 万。

为什么不用 Nested Loop?因为如果嵌套循环,内层 orders 没法用单一索引高效重扫(`user_id` 索引重扫 3000 次,每次代价高),`ntuples = 3000 × 10000000` 会爆。代价模型算出来 Hash 更省,于是选 Hash。**这整套推理,优化器是靠统计信息(rows 估算)和代价公式自动完成的,没有任何规则代码**。

### 真实易错场景:统计信息过期,优化器自信地选错

现在讲本章最该警惕的一类故障。假设你有一张 `orders` 表,起初 1 万行,`status` 字段上建了索引,且 `status` 大部分是 `'pending'`。你跑了 `ANALYZE`,统计信息记下:`status` 的 MCV 是 `{pending: 0.95}`,即 95% 是 pending。

然后业务爆发,表涨到 1 亿行,但**新数据的分布变了**——现在 95% 是 `'completed'`,`'pending'` 只剩 0.1%。可你没重新 `ANALYZE`,`pg_statistic` 里还是旧的 MCV `{pending: 0.95}`。

这时你跑:

```sql
SELECT * FROM orders WHERE status = 'pending';
```

`(以下为计划示意,说明优化器如何被过期统计误导)`

```text
-- 过期统计下,优化器以为 pending 只占 5%(其实是 0.1% 还反向了,这里简化为"以为很少"):
                                     QUERY PLAN
-----------------------------------------------------------------------------------------
 Index Scan using idx_orders_status on orders  (cost=0.42..980.55 rows=5000 width=...)
   Index Cond: (status = 'pending'::text)
-- 但真实情况是 pending 现在是 95% 的大头!走索引意味着上千万次随机回表,实际跑几分钟
```

优化器查 MCV,看到 `pending` 的频次记着 0.95(旧值)——但它以为是"高频值"所以走了……不对,这里要小心:如果 MCV 显示 pending 是高频 0.95,优化器其实会算出"会选出 95% 的行",按理**应该走顺序扫描**(因为选大比例走索引不划算)。真正出错的版本是**统计信息把比例记反了或记低了**:比如 MCV 还停在某个中间状态,把 pending 记成只占 5%。于是优化器以为"只选 5%,走索引划算",自信地选了 `Index Scan`——可现实里 pending 占 95%,走索引变成上千万次随机 I/O,查询慢到无法用。

> **不这样会怎样**:统计信息一旦和真实分布脱节,代价公式算出来的"选择率"就是错的,而错误的选择率会**沿第三节那条链一路放大**:错的 rows → 错的单表代价 → 错的连接代价 → 错的计划。最阴险的是,**优化器依然会"自信地"给出一个计划**——它不知道自己的输入是脏的,它只是忠实地按公式算。用户看到的是"明明有更快的全表扫描,数据库却死磕索引",百思不得其解,根子全在统计信息过期。

这就是为什么 DBA 的两条铁律是:**定期 `ANALYZE`**(尤其是大批量 INSERT/UPDATE/DELETE 之后),以及**关注 `pg_stat_user_tables` 里 `n_mod_since_analyze`**(自上次分析以来改了多少行,改太多就该重新分析了)。PG 也有 `autovacuum` 后台进程自动触发 `ANALYZE`,但它有阈值,写入极猛时可能跟不上,仍需人工介入。

修这个故障的办法往往就一句:`ANALYZE orders;`。重新采样后,MCV 更新成真实分布,优化器重算选择率,计划立刻翻回正确的顺序扫描。**这一幕,是"优化器在猜,不是在算"这句话最生动的注脚**——它猜得对不对,完全取决于你喂给它的统计信息新不新。

> 这一节把代价模型用在了最该用的地方:连接算法的取舍。也讲清了优化器最常翻车的场景:**不是算法有 bug,而是输入(统计信息)失真**。理解了这点,你就理解了为什么"调优"很多时候不是改 SQL、不是加索引,而是 `ANALYZE` 一下。

## 关键源码精读:把代价模型的两段公式钉死

前面散落着贴了不少片段,这一节我们挑两个**最能代表本章核心**的函数,完整地、逐段地走一遍。一个是**代价计算的范本 `cost_seqscan`**(它把"磁盘代价 + CPU 代价"这条加和模型体现得最干净),另一个是**选择率→基数那一步的化身 `set_baserel_size_estimates`**(它是第三节那条估算链的咽喉)。读懂这两个,你就抓住了优化器代价体系的骨架。

### 精读一:`cost_seqscan` —— 一个算子的代价是怎么算出来的

这是顺序扫描的完整代价函数,60 多行,逻辑却异常清晰,分四步:

> [src/backend/optimizer/path/costsize.c:284-351](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L284-L351)

```c
cost_seqscan(Path *path, PlannerInfo *root,
			 RelOptInfo *baserel, ParamPathInfo *param_info)
{
	Cost		startup_cost = 0;
	Cost		cpu_run_cost;
	Cost		disk_run_cost;
	double		spc_seq_page_cost;
	QualCost	qpqual_cost;
	Cost		cpu_per_tuple;

	/* Should only be applied to base relations */
	Assert(baserel->relid > 0);
	Assert(baserel->rtekind == RTE_RELATION);

	/* Mark the path with the correct row estimate */
	if (param_info)
		path->rows = param_info->ppi_rows;
	else
		path->rows = baserel->rows;

	if (!enable_seqscan)
		startup_cost += disable_cost;

	/* fetch estimated page cost for tablespace containing table */
	get_tablespace_page_costs(baserel->reltablespace,
							  NULL,
							  &spc_seq_page_cost);

	/*
	 * disk costs
	 */
	disk_run_cost = spc_seq_page_cost * baserel->pages;

	/* CPU costs */
	get_restriction_qual_cost(root, baserel, param_info, &qpqual_cost);

	startup_cost += qpqual_cost.startup;
	cpu_per_tuple = cpu_tuple_cost + qpqual_cost.per_tuple;
	cpu_run_cost = cpu_per_tuple * baserel->tuples;
	/* tlist eval costs are paid per output row, not per tuple scanned */
	startup_cost += path->pathtarget->cost.startup;
	cpu_run_cost += path->pathtarget->cost.per_tuple * path->rows;

	/* Adjust costing for parallelism, if used. */
	if (path->parallel_workers > 0)
	{
		double		parallel_divisor = get_parallel_divisor(path);

		/* The CPU cost is divided among all workers. */
		cpu_run_cost /= parallel_divisor;

		/*
		 * It may be possible to amortize some of the I/O cost, but probably
		 * not very much, because most operating systems already do aggressive
		 * prefetching.  For now, we assume that the disk run cost can't be
		 * amortized at all.
		 */

		/*
		 * In the case of a parallel plan, the row count needs to represent
		 * the number of tuples processed per worker.
		 */
		path->rows = clamp_row_est(path->rows / parallel_divisor);
	}

	path->startup_cost = startup_cost;
	path->total_cost = startup_cost + cpu_run_cost + disk_run_cost;
}
```

**第一步(L299-302):确定这条路径输出的行数。** 如果是参数化扫描(带连接参数的扫描),用 `param_info->ppi_rows`;否则用 `baserel->rows`——也就是 `set_baserel_size_estimates` 算出来的"过滤后行数"。这个 `rows` 是给上层算子用的(上层知道"我这个下层会吐多少行"),不是顺序扫描自己处理多少行。

**第二步(L304-310):`enable_seqscan` 开关与 tablespace 代价。** 前面讲过,`enable_seqscan = off` 就加 `disable_cost`(1e10)劝退。`get_tablespace_page_costs` 是一个细节但重要的设计:**代价常量可以按表空间(tablespace)覆盖**。你可以在特定表空间上单独设 `seq_page_cost`/`random_page_cost`(比如把存 SSD 的表空间随机代价调低),这里就是读取那个覆盖值。默认情况下 `spc_seq_page_cost` 就是全局的 `seq_page_cost = 1.0`。这个机制让"不同存储介质用不同代价"成为可能,体现了代价模型的弹性。

**第三步(L312-325):核心,磁盘代价 + CPU 代价。**

- `disk_run_cost = spc_seq_page_cost * baserel->pages` —— 磁盘代价 = 每页代价 × 总页数。顺序扫描要读全部页,没有别的。这是代价里最直观的部分:几页就乘几个 1.0。
- `get_restriction_qual_cost(..., &qpqual_cost)` —— 把这张表上的 `WHERE` 条件(`baserestrictinfo`)逐个估一次代价,汇总成 `qpqual_cost`。它返回两部分:`startup`(算这些条件前的固定开销)和 `per_tuple`(每行算一遍这些条件的开销)。
- `cpu_per_tuple = cpu_tuple_cost + qpqual_cost.per_tuple` —— 处理每一行的 CPU 开销 = 基础 tuple 代价(纯取出一行的开销,0.01) + 判断 WHERE 条件的开销。
- `cpu_run_cost = cpu_per_tuple * baserel->tuples` —— **关键**:CPU 代价乘的是 `baserel->tuples`(表里**全部**行数),不是 `rows`!因为顺序扫描必须把每一行都读出来、都判断一次条件,哪怕最后 99% 被过滤掉。这个区别(`tuples` vs `rows`)是理解 SeqScan 代价的关键:它为"读全部"付费,不为"输出多少"付费。
- 接着两行(L324-325)处理 targetlist(目标列表达式)的开销,这个是**按输出行数 `rows` 算**的,因为只有没被过滤掉、要真正输出的行,才会去算 `SELECT` 列里的表达式。注释 "tlist eval costs are paid per output row, not per tuple scanned" 把这个区分点得很清楚:**扫描代价按 tuples 算、输出代价按 rows 算**。

**第四步(L327-347):并行调整。** 如果这是并行扫描(`parallel_workers > 0`),CPU 代价除以 `parallel_divisor`(大致是 worker 数),因为活儿分摊给多个 worker 了。但注释明确说磁盘代价**不**打折——因为操作系统本来就在预读,多 worker 也榨不出更多顺序 I/O 的便宜。这是个务实的判断:**承认并行能省 CPU,但不幻想它能省已经优化好的顺序 I/O**。

**收尾(L349-350):** `total_cost = startup_cost + cpu_run_cost + disk_run_cost`。三段加和——启动代价 + CPU 运行代价 + 磁盘运行代价。整个函数没有任何魔法,就是"读多少页 × 每页代价 + 处理多少行 × 每行代价"。这套朴素加法,正是 PG 代价模型的灵魂:用最简单、可解释、可调参的线性组合,逼近真实的执行开销。

读懂 `cost_seqscan`,其他算子的代价函数(`cost_index`、`cost_nestloop`、`cost_hashjoin`……)都是同一套思路的变体:把"这个算子要做多少 I/O、要处理多少行、每行算什么"拆开,各乘各的代价常数,加起来。区别只在"I/O 是顺序还是随机""行数是 tuples 还是 rows""有没有建表/排序之类的一次性开销"。

### 精读二:`set_baserel_size_estimates` —— 选择率怎么变成行数

如果说 `cost_seqscan` 是"代价侧"的范本,那 `set_baserel_size_estimates` 就是"行数侧"的咽喉——它把第三节讲的"选择率 × 总行数"这个公式,在代码里落地:

> [src/backend/optimizer/path/costsize.c:5239-5258](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L5239-L5258)

```c
set_baserel_size_estimates(PlannerInfo *root, RelOptInfo *rel)
{
	double		nrows;

	/* Should only be applied to base relations */
	Assert(rel->relid > 0);

	nrows = rel->tuples *
		clauselist_selectivity(root,
							   rel->baserestrictinfo,
							   0,
							   JOIN_INNER,
							   NULL);

	rel->rows = clamp_row_est(nrows);

	cost_qual_eval(&rel->baserestrictcost, rel->baserestrictinfo, root);

	set_rel_width(root, rel);
}
```

短短十几行,信息量却很大:

- `rel->tuples`:表的总行数,**来自 `pg_class.reltuples`**(经 relcache → `get_relation_info` → 这里)。这是统计信息的表级部分。
- `clauselist_selectivity(root, rel->baserestrictinfo, ...)`:把这张表上**所有的 `WHERE` 条件**(`baserestrictinfo`,一个限制条件链表)综合,算出一个**总选择率**。这个函数内部会对每个条件调对应的选择率函数——等值调 `eqsel`→`var_eq_const`(第三节精读过的那个),范围调 `ineq_histogram_selectivity`(用直方图),等等;然后把各条件的选择率按一定规则组合(默认假设条件间独立,选择率相乘)。
- `nrows = rel->tuples * <总选择率>` —— **本章的脊梁公式**。选择率乘以总行数,就是估算出的"过滤后行数"。
- `rel->rows = clamp_row_est(nrows)` —— 划到 `[1, MAXIMUM_ROWCOUNT]` 并取整,防呆。
- 后两行顺带把"过滤条件本身的 CPU 代价"(`baserestrictcost`)和"每行平均宽度"(`set_rel_width`,影响 I/O 代价估算)也算出来,塞进 `rel`,供后面 `cost_seqscan` 之类使用。

把这个函数和 `cost_seqscan` 串起来,整章的逻辑闭环了:

```
pg_class.reltuples / pg_statistic(MCV,distinct,直方图)
            │  (经 get_relation_info 读入)
            ▼
      rel->tuples
            │  × clauselist_selectivity (eqsel/ineqsel …)
            ▼
       rel->rows  ──────────┐
            │                │
            ▼                ▼
   cost_seqscan: disk = seq_page_cost × pages
                 cpu   = cpu_tuple_cost × tuples   ← 用 tuples
                         + tlist_cost  × rows      ← 用 rows
            ▼
     path->startup_cost / path->total_cost
            │
            ▼
   沿计划树自底向上累加 → 选最优 Path → create_plan 物化
```

这张图就是优化器代价体系的全部:统计信息在底,选择率在中,代价公式在上,最后累加成整树代价。`set_baserel_size_estimates` 是底和中之间的桥,`cost_seqscan` 是中和上之间的桥。把握住这两座桥,你就把握住了 PG 优化器的命脉。

## 章末小结

### 一句话回顾本章

优化器接手的是一棵"说清了要什么、没说怎么取"的 Query 树,交出的是一棵"每一步都选定了具体算法、估好了代价"的 Plan 树。它干的活可以拆成三层:

1. **逻辑重写**:`pull_up_subqueries` 等预处理,把 `IN` 子查询变连接、化简常量、降级外连接——只做语义等价的变形,为物理规划扫清障碍。
2. **路径枚举 + 代价比较**:对每个表生成顺序扫描/索引扫描路径,用动态规划(表多则 GEQO)枚举连接顺序和算法,每条路径用代价公式打分,选最便宜的。
3. **物化**:`create_plan` 把最优 Path 转成执行器能跑的 Plan 树。

而贯穿这三层的,是**代价模型**:代价 = 顺序读页代价 + 随机读页代价 + 每行 CPU 代价,以"一次顺序读页"为单位 1。代价公式里填的数,绝大多数来自**统计信息**(`pg_class.reltuples`、`pg_statistic` 的 MCV/distinct/直方图),统计信息驱动**选择率**,选择率乘总行数得到**基数**,基数驱动每层算子代价——这条 `统计 → 选择率 → 基数 → 代价` 的链,是优化器的脊柱。

理解了这条链,就理解了优化器最反直觉的本质:**它不是在精确计算,是在有根据地猜测**。它猜得准不准,全看统计信息新不新、全不全——所以 `ANALYZE` 不是可有可无的杂务,而是优化器正确工作的生命线。它因此会在统计信息过期时自信满满地选错索引,这是它最大的、也是无法根治的弱点(只能靠及时 `ANALYZE` 缓解)。

### 回扣主线:本章服务"快"那一侧,对付"声明式查询"本性

回到全书的二分法:**数据库一半在让数据快,一半在让数据不丢不乱**。优化器毫无疑问站在"快"这一侧——它的全部价值,就是在众多执行方案里挑出**最省的那个**,让查询跑得尽可能快。

再对应数据的三个本性(第 1 章):

- **声明式查询 vs 物理存取**——这是优化器存在的**直接理由**。SQL 只说"要什么",优化器负责填上"怎么取"。没有优化器,声明式 SQL 就无法落地成对页面的具体读写。这是本章最核心的本性。
- **磁盘慢、内存快**——代价模型为什么把"随机读"定得比"顺序读"贵 4 倍?根子就是磁盘的这一本性。优化器千方百计避免随机 I/O、偏爱顺序扫描或能用索引高选择率命中的路径,全是在和"磁盘慢"博弈。
- **共享会冲突**——本章没直接碰并发,但提一句:优化器规划时用到的统计信息(行数、选择率),是基于**已提交数据**的快照估算的;并发写入带来的"规划时和执行时数据不一致",是优化器估算误差的来源之一(这部分细节属于 P4 事务篇)。

所以本章是"快"这一侧、对付"声明式查询"本性的第二块拼图(第一块是上一章的解析器——把 SQL 变成树;本章决定树怎么执行最省;下一章执行器才真正去跑)。

### 几个容易记混的点,再点一遍

1. **代价是相对分数,不是秒**。单位是"一次顺序读页"。比较用,不是计时用。
2. **等值选择率用 MCV + distinct 均摊,不用直方图**;直方图只服务范围查询。这是最常见的误解。
3. **`enable_xxx = off` 不是禁用,是加 1e10 劝退**。代价模型始终在比较。
4. **SeqScan 的 CPU 代价乘 `tuples`(全表行),不是 `rows`(过滤后行)**,因为它要处理每一行;但 targetlist 代价乘 `rows`。这个区分要看清。
5. **统计信息过期 = 优化器瞎猜**。`ANALYZE` 是治"计划突然变差"的第一味药。
