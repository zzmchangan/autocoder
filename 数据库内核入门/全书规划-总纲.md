# 《数据库内核:从一条 SQL 说起》—— 全书总纲

> 一本写给"天天用数据库、却没读过它的源码;想搞懂每个设计**为什么**这么干,而不是只会写 SQL 和调参数"的人的第一本书。
>
> 本文是全书的**总纲**:定主线、定写作风格、定分篇分章、定写作路线。
> 本书源码基于 **PostgreSQL 17.0**(官方 release,固定版本,行号稳定,全书统一引用)。

---

## 一、这本书的定位

### 目标读者

会写 SQL、天天和数据库打交道(后端开发 / DBA / 架构师),但没读过数据库内核源码;**想搞懂"为什么这么设计",而不是"SQL 怎么写、参数怎么调"**。和《Linux 内核》系列是同一类读者。

### 全书一句话主旨

> **从你敲下一句 `SELECT`,到屏幕上返回结果——这中间这个数据库到底发生了什么。**

每个子系统,都是这条旅程上的一个**驿站**。读完一本书,读者不是"背下了一堆机制",而是**亲眼跟着一条 SQL 走完它的一生**,在每个驿站都搞懂了"这站为什么必须存在、它解决了什么不解决就不行的问题"。

### 为什么用"一条 SQL 的旅程"当骨架(而不是按子系统平铺)

和 Linux 内核系列用"从上电到 `ls`"是同一个道理:数据库的子系统高度耦合(查询要走存储、存储要靠日志保不丢、并发要靠锁/MVCC 不打架)。如果按"存储篇、索引篇、事务篇…"百科式平铺,读者会觉得每篇都是孤岛,而且不知道**为什么要学这个**。

用"一条 SQL 的旅程"当骨架,每个子系统都是"为了让这条 SQL 走下去而必须出现的角色"——读者永远知道**我在哪、为什么需要下一站**。

## 二、写作风格:第一性原理直球为主,比喻按需点缀

> ⚠️ **这一节是本书与 Linux 内核系列最大的不同,所有写作者必读、必守。**

Linux 内核系列用了"机器=城市,内核=市政府"的**全局统一比喻体系**。那是因为操作系统概念(虚拟内存、特权环、页表、中断陷入)对应用层程序员是**不可见且反直觉**的,必须靠一个贯穿全书的比喻把抽象拽到地面。

**数据库不一样。** SQL、表、索引、事务这些,读者天天在用,本来就有直觉;"索引像书的目录""事务像转账"几乎是常识。如果硬套一个全局比喻体系(比如"档案馆"),反而会让行文绕、信息密度被稀释——比如讲 B+ 树的页面分裂,直接讲数据结构和算法,比绕到"档案柜怎么挪格子"清楚十倍。

所以本书的风格是:

1. **第一性原理直球为主**。每个机制先讲它解决什么本质问题(**不这样会怎样** → **所以这样设计**)、再讲源码怎么落地。这是最高原则,与 Linux 系列完全一致。
2. **比喻按需点缀,不搞全局体系**。只在概念**确实反直觉**时,用一个**一次性小比喻**点一下(用完即弃,不追求全书统一映射);直观的概念直接讲。

**这个"度"怎么把握:**

| 直接讲更好(本质是数据结构 / 算法) | 偶尔点一下就通(反直觉的机制) |
|---|---|
| B+ 树的页面分裂与合并 | MVCC:"每个事务看到自己进门那一刻拍的照片" |
| 8KB 页的字节布局 | WAL:"先记账后动钱,所以掉电也不丢" |
| 优化器的代价模型 | 隔离级别:脏读 / 不可重复读 / 幻读这些并发异常 |
| Tuple 的字段与变长存储 | Buffer Pool 的 Clock-Sweep 替换策略 |

**一句话原则:比喻服务于理解,不是装饰。用不用,看那个概念是不是反直觉——直觉够用的,就别绕。**

## 三、全书主线:让数据"快" vs 让数据"不丢不乱"

Linux 内核系列的主线是"管资源 vs 造幻觉"。本书的主线是:

> **数据库一半在"让数据快"(用内存换速度、用索引换查找),一半在"让数据不丢、不乱"(用日志换持久、用事务/MVCC 换一致)。**

任何时候看不懂某个机制,回到这句问:"这是在追求**快**,还是在追求**不丢不乱**?"

而所有复杂性,都来自数据的**三个本性**(对标 Linux 内存篇的"四个本性"):

| 数据的本性 | 逼出的子系统 | 典型机制 |
|---|---|---|
| 磁盘慢、内存快但**易失** | 存储与持久性 | Buffer Pool、WAL、Checkpoint |
| 数据要**共享**,但共享会**冲突** | 事务与并发控制 | ACID、锁、MVCC、隔离级别 |
| 用户说"要**什么**",机器只会"按**位置**读" | 查询引擎与索引 | 解析器、优化器、B+ 树 |

这三大子系统 = 全书的 **P1~P5 五篇**。后面你看到任何一处"为什么搞这么复杂",去找是这三个本性中的哪个在作怪。

## 四、全书结构(分篇 + 分章)

> 每章只给**章名 + 核心问题**(一句话),细节在各自篇的 `PX-00-目录.md` 里展开。

### 第 0 篇 · 开篇:数据库是什么,为什么必须存在

- **导言 · 一篇看懂数据库内核全貌**:用"一条 SQL 的旅程"把所有子系统串一遍,立住"快 vs 不丢不乱"二分法。
- **第 1 章 · 第一性原理:为什么需要数据库**:直接用文件存数据到底哪里不行?ACID 是怎么被四个乱子**逼**出来的?

### 第 1 篇 · 查询引擎:一条 SQL 怎么进来、被理解、被规划、被执行

> 读者递进门的便条,到馆员决定怎么去取档——SQL 的前半生。

- **第 2 章 · 连接与会话**:客户端怎么"进门"(PG 的进程模型:为什么是"每连接一进程"而不是线程)。
- **第 3 章 · SQL 解析器**:从字符串到解析树(词法/语法分析,为什么 SQL 要先被翻译成一棵树)。
- **第 4 章 · 查询优化器**:怎么决定"用哪条路最省"(代价模型、统计信息、为什么同一句 SQL 有多种执行计划)。
- **第 5 章 · 执行器**:把计划跑出来(火山模型/迭代器、顺序扫描、嵌套循环/Hash/归并连接)。

### 第 2 篇 · 存储引擎:数据在磁盘与内存里长什么样

> 数据真正"住"的地方。这一篇把"易失的本性"接住一半(持久靠 P5 的日志,高效靠这里的内存中转)。

- **第 6 章 · 页面 Page**:磁盘 I/O 的最小单位(8KB 页布局,为什么不按字节存取)。
- **第 7 章 · 堆表与元组 Heap/Tuple**:一行数据怎么落盘(行指针、变长字段、TOAST)。
- **第 8 章 · Buffer Pool**:内存里的中转站(缓冲池、Clock-Sweep 替换、脏页,为什么是性能命脉)。
- **第 9 章 · FSM 与 VM**:空闲空间映射与可见性映射(两个辅助结构为什么必要)。

### 第 3 篇 · 索引:怎么快速找到数据

> "用户说要什么,机器只会按位置读"——这个错位,靠索引弥合。

- **第 10 章 · 为什么需要索引**:顺序扫描 O(n) vs 索引扫描 O(log n),什么时候反而索引更慢。
- **第 11 章 · B 树**:平衡的多路查找(PG 默认索引,为什么不是二叉树/红黑树/哈希)。
- **第 12 章 · 索引家族**:Hash/GiST/GIN/BRIN(PG 特色,各自为哪类查询而生)。
- **第 13 章 · 索引与堆的配合**:回表、覆盖索引、仅索引扫描。

### 第 4 篇 · 事务与并发控制:多人同时写怎么不打架

> "数据要共享,但共享会冲突"——这一篇是全书最难、也最精华的部分。

- **第 14 章 · 事务与 ACID**:事务是什么,每个字母各防什么灾难。
- **第 15 章 · 隔离级别与并发异常**:脏读/不可重复读/幻读,为什么隔离是"正确性 vs 性能"的权衡。
- **第 16 章 · 锁**:表锁/行锁/死锁(PG 的锁层级、死锁检测)。
- **第 17 章 · MVCC**:多版本并发控制(PG 的精华——读不阻塞写、写不阻塞读,xmin/xmax/快照)。

### 第 5 篇 · 持久性与崩溃恢复:掉电怎么不丢

> "磁盘慢、内存快但易失"——内存里的改动怎么保证掉电不丢。

- **第 18 章 · WAL**:预写日志(先记日志再改数据,这个顺序为什么是持久性的命根子)。
- **第 19 章 · Checkpoint**:日志与数据的同步点(检查点为什么必要,崩溃后要重放多少)。
- **第 20 章 · 崩溃恢复**:从日志重建一致性(PG 的 redo-only 设计)。

### 第 6 篇 · 进阶选讲(按精力决定)

- **第 21 章 · VACUUM**:MVCC 的代价(死元组清理,为什么 PG 需要"打扫")。
- **第 22 章 · 复制与高可用**:流复制、主从一致性。
- **第 23 章 · 并行查询**:多核怎么一起跑一条 SQL。

### 附录

- **A · 怎么读 PG 源码**:目录结构、阅读路线、cscope/ctags。
- **B · 观测与调试工具**:`EXPLAIN`、`pg_stat_*`、`pageinspect`、`pg_waldump`。

## 五、PostgreSQL 源码引用约定

- 本书源码基于 **PostgreSQL 17.0** 官方 release(固定版本,行号稳定,多 agent 引用天然一致)。
- 源码在工作区 `postgresql-17.0/` 目录下。各章节 md 在 `数据库内核入门/` 下,**引用格式**:`[描述](../postgresql-17.0/路径#L行号)`;**区间一律用 `#L起始-L结束`**(如 `#L150-L168`),不要只标起点。
- **严禁凭记忆引用源码**(最易翻车的硬伤,务必严格遵守):
  1. 每个要引用的函数/结构体,**先 grep 出真实行号再写**。
  2. 每个代码块必须是真实文件**摘录**(允许 `...` 省略无关部分,但保留的部分逐字一致)。**绝对不要把意译/改写/脑补的代码当真实源码贴。**
  3. 简化展示必须**明确标注**"(以下为简化示意,非源码原文)"。
  4. 不确定精确行号时,**宁可只标文件不标行号,也不要猜**。

## 六、写作路线与分工

### 路线(推荐顺序)

1. **先写第 0 篇**(导言 + 第 1 章第一性原理)——它定调,后续所有篇向它对齐。
2. **沿旅程顺序写**:P1 查询引擎 → P2 存储引擎 → P3 索引 → P4 事务并发 → P5 持久恢复 → P6 进阶。

### 每篇的"启动包"(三件套)

每写一个新篇,先产出三样东西(和 Linux 内核系列完全对应):
1. **该篇目录**(篇内分章 + 每章核心问题 + 关键源码)—— `PX-00-目录.md`
2. **该篇导言**(全局直白解释,用"一条 SQL 的旅程"串一遍本篇)—— `PX-00-导言.md`
3. **逐章正文**(用 `_章节写作提示词.md` 模板驱动)。

### 多 agent 并行

`_章节写作提示词.md` 是驱动各写作 agent 的**统一基准**。每个 agent 拿到一篇/一章任务,按提示词产出初稿。**风格策略(弱化比喻、直球为主)、主线二分法、源码引用规范、章节结构**,全部以提示词为准,保证全书统一、可并行。

> **命名约定**:全书所有篇一律用 `PX-0Y-标题.md`(第 0 篇=P0、第 1 篇=P1…,Y 为两位章号)。例:第 0 篇第 1 章 = `P0-01-…`,第 4 篇第 17 章 = `P4-17-…`。

## 七、本书与 Linux 内核系列的关系

两本书同源——都是"**为什么优先 + 真实源码 + 固定章节结构 + 回扣主线**"。读者可以当它是 Linux 内核系列的姊妹篇。

唯一的风格差异:**Linux 系列用全局比喻(城市/市政府),本书弱化比喻、直球为主**(理由见第二节)。这个差异是被两个领域概念抽象度的不同逼出来的,不是随意为之。
