# 第 7 章 · 堆表与元组:一行数据怎么落盘

> **前置**:你需要先读过[第 6 章《页面 Page:磁盘 I/O 的最小单位》](P2-06-页面Page-磁盘IO的最小单位.md)——它讲了 8KB 页这个容器:页两头往中间长,头部是行指针 `ItemId` 数组,尾部塞一行行数据,中间是空闲空间。本章就钻进**页的尾部**,把那一行行数据(tuple)自己到底长什么样、怎么被行指针找到、定长和变长字段各怎么存,一次讲透。这是 P2 存储引擎的第二章。

> **核心问题**:一行数据(一个 tuple)在磁盘的 8KB 页里到底长什么样?行指针(line pointer/ItemId)怎么让页不必顺序存也能快速定位某行?定长字段(int/bigint)和变长字段(VARCHAR/TEXT)各自怎么存、为什么要对齐(padding)?还有 `HeapTupleHeader` 里 `t_xmin/t_xmax/t_cid` 这些"看不见的列"是干嘛的(为 MVCC 埋伏笔)?超过 ~2KB 的大字段为什么要被"切片"存到单独的 TOAST 表?
>
> **读完本章你会明白**:tuple 的字节布局(固定 23 字节头 + null 位图 + 对齐填充 + 数据区);`ItemId` 的三种状态(本章只讲 `LP_NORMAL/LP_DEAD/LP_UNUSED`,REDIRECT 留给 P6)和 VACUUM 的关系;为什么可见性标志要拆成 `t_infomask` 和 `t_infomask2` 两套;变长字段(varlena)的 1 字节/4 字节两种 header 和短 varlena 优化;以及 TOAST 的四种策略(PLAIN/EXTERNAL/EXTENDED/MAIN)各自是什么、谁会被切片。

> **如果一读觉得太难**:先只记三件事——① 一行 = 23 字节固定头 + 可选 null 位图 + 对齐 padding + 真实数据,行指针(ItemId)指向它;② 变长字段(VARCHAR/TEXT)有 1 字节和 4 字节两种 header,短的省一半;③ 超过 ~2KB 的大字段会被 TOAST 切片挪到单独的表,主表只留一个"指针"。xmin/xmax 这些"看不见的列"是为 MVCC 准备的,本章只埋伏笔,细节留给 P4。

## 一、先看清一行数据在页里的位置

第 6 章我们已经画过页内布局:头部 `PageHeaderData` + 行指针数组 `pd_linp`,尾部塞一行行 tuple,中间空闲。一个 tuple 的"地址",由头部那个 4 字节行指针 `ItemId` 给出——它编码了三个字段:`lp_off`(这行在页里的偏移)、`lp_len`(这行多长)、`lp_flags`(状态)。

> [src/include/storage/itemid.h:25-30](../postgresql-17.0/src/include/storage/itemid.h#L25-L30)

```c
typedef struct ItemIdData
{
	unsigned	lp_off:15,		/* offset to tuple (from start of page) */
				lp_flags:2,		/* state of line pointer, see below */
				lp_len:15;		/* byte length of tuple */
} ItemIdData;
```

这一节我们只关心一件事:**那个 `lp_off`/`lp_len` 指向的"行数据区域"里,字节是怎么排的**。先上一张全景图,再逐块拆。

### 一行 tuple 的字节布局(示意)

```text
   ItemId(行指针,在页头部,4 字节)──┐
                                     │ lp_off 指向这里 ↓
   ┌─────────────────────────────────┴───────────────────────────┐
   │ HeapTupleHeaderData(固定 23 字节,MAXALIGN 到 24)             │
   │  ┌────────────────────────────────────────────────────────┐ │
   │  │ t_choice(union)  t_heap: t_xmin(4) t_xmax(4) t_field3(4) │ │ ← 12
   │  │ t_ctid  ItemPointerData(6)                              │ │ ← 6
   │  │ t_infomask2(2)   t_infomask(2)   t_hoff(1)              │ │ ← 5
   │  └────────────────────────────────────────────────────────┘ │  = 23
   │  null 位图 t_bits[ ]  (仅当 t_infomask 有 HEAP_HASNULL)      │  ← 可变
   │  对齐填充 padding      (让用户数据 MAXALIGN 对齐,通常补到 24) │
   ├──────────────────────────────────────────────────────────────┤
   │ 用户数据区(t_hoff 之后开始)                                  │
   │   col0(int4)  col1(int8)  pad  col2(text varlena)  col3...   │
   └──────────────────────────────────────────────────────────────┘
```

> 这张图标注为**示意**:真实字节序受平台对齐影响,23 是固定头的"逻辑大小",`t_hoff` 把它补齐到 `MAXALIGN`(64 位平台=8 的倍数,常见 24)。来源见下方源码注释 `^ - 23 bytes - ^`。

整张图分三大块:**固定头(服务并发/可见性)、null 位图+对齐填充(服务紧凑存储)、用户数据区(服务"快"——紧凑对齐 + 变长压缩)**。下面三节分别拆。

## 二、看不见的列:t_xmin / t_xmax / t_cid / t_ctid

第 1 章我们埋过一个伏笔:每一行 tuple 都自带一个"来历",浓缩成 `t_xmin`/`t_xmax` 两个事务 ID。现在把它讲透。

### 不这样会怎样:没有这些字段,MVCC 根本无从谈起

PostgreSQL 解决"两人同时改同一行不打架"的核心机制叫 **MVCC(多版本并发控制)**,它的做法是:**更新一行,不是原地改,而是写一个新版本;旧行不马上删,而是打个标记。** 那么问题来了——

> **不这样会怎样**:一个事务扫到某行,它怎么知道"这行我该不该看见"?如果 tuple 里不记事务信息,事务只能问全局锁表"这行现在归谁、改了没"——每一行读写都要查全局表,并发越高锁表越热,性能崩塌。更糟的是,新写的版本和旧版本物理上都在页里,没有事务标记就无法区分"谁是谁"。

**所以这样设计**:每行 tuple 自带事务信息,谁插入、谁删除、谁更新,写在 tuple 头里。读的时候,事务拿自己的"快照"和 tuple 头里的 `t_xmin`/`t_xmax` 一比,当场就知道这行对自己可不可见——**不用查全局状态,每行自决可见性**。这是 MVCC 能做到"读不阻塞写"的地基。

### HeapTupleFields:t_xmin / t_xmax / t_field3

固定头里记录事务信息的部分,是 `HeapTupleFields`:

> [src/include/access/htup_details.h:122-132](../postgresql-17.0/src/include/access/htup_details.h#L122-L132)

```c
typedef struct HeapTupleFields
{
	TransactionId t_xmin;		/* inserting xact ID */
	TransactionId t_xmax;		/* deleting or locking xact ID */

	union
	{
		CommandId	t_cid;		/* inserting or deleting command ID, or both */
		TransactionId t_xvac;	/* old-style VACUUM FULL xact ID */
	}			t_field3;
} HeapTupleFields;
```

四个"看不见的列",其实是三个物理字段:

- **`t_xmin`**——插入这行的事务 ID。每行都知道自己是被哪个事务"生"出来的。
- **`t_xmax`**——删除或锁定这行的事务 ID。一行被删,不是抹掉,而是给 `t_xmax` 打上"已被某某事务删"。
- **`t_cid`(CommandId)**——事务内第几条命令。一个事务里可能连续执行多条 SQL,`t_cid` 用来区分"自己事务里前面的命令改的,还是后面的命令改的"——这关系到"一个事务能不能看到自己前面命令的修改"。
- **`t_xvac`**——和老式 VACUUM FULL 配合(挪动 tuple 的事务 ID),和 `t_cid` 共用一个 union,因为两者不会同时需要。

> **注意"五个逻辑字段塞进三个物理字段"**:源码注释明说,"Xmin/Cmin/Xmax/Cmax/Xvac 五个逻辑字段,存在三个物理字段里"。`Cmin`(插入命令号)和 `Cmax`(删除命令号)复用 `t_cid`——因为一行在被插入的那个事务里,只关心 Cmin;在被删除的那个事务里,只关心 Cmax,两者生命周期不重叠(除非插入和删除发生在同一事务,这时用 combo CID 映射,见 combocid.c)。这是典型的"为省空间做时间换空间":每一行、每个字段都在省。

### t_ctid:版本链的桥

固定头里还有个关键字段 `t_ctid`(current TID):

> [src/include/access/htup_details.h:161-162](../postgresql-17.0/src/include/access/htup_details.h#L161-L162)

```c
	ItemPointerData t_ctid;		/* current TID of this or newer tuple (or a
								 * speculative insertion token) */
```

`t_ctid` 初始指向自己(自己的 TID = 块号 + 行号)。一旦这行被更新,新版本写在别处,旧版本的 `t_ctid` 就改成指向新版本。这样,旧行 → 新行串成一条链,顺着 `t_ctid` 就能找到"这一行的最新版本"。源码注释把规则说得很清楚:

> [src/include/access/htup_details.h:86-94](../postgresql-17.0/src/include/access/htup_details.h#L86-L94)
> "a tuple is the latest version of its row iff XMAX is invalid or t_ctid points to itself"

也就是:**判断一行是不是"最新版本",看 `t_xmax` 无效 或 `t_ctid` 指向自己**。`t_ctid` 这条链,是 MVCC 多版本导航的路标,也是第 17 章(MVCC)和 P6(HOT)的主线。

> 这一节只要记住:tuple 头里 `t_xmin/t_xmax/t_cid/t_ctid` 这四个"看不见的列",是为 P4 事务并发准备的。本章埋下伏笔,细节(快照怎么和它们比、可见性怎么判)留给第 17 章。

## 三、固定头全貌:HeapTupleHeaderData 与 23 字节

把固定头整个看清。这是后续所有字段引用的总账。

> [src/include/access/htup_details.h:153-181](../postgresql-17.0/src/include/access/htup_details.h#L153-L181)

```c
struct HeapTupleHeaderData
{
	union
	{
		HeapTupleFields t_heap;
		DatumTupleFields t_datum;
	}			t_choice;

	ItemPointerData t_ctid;		/* current TID of this or newer tuple ... */
	...
	uint16		t_infomask2;	/* number of attributes + various flags */

	uint16		t_infomask;		/* various flag bits, see below */

	uint8		t_hoff;			/* sizeof header incl. bitmap, padding */
	...
	bits8		t_bits[FLEXIBLE_ARRAY_MEMBER];	/* bitmap of NULLs */

	/* MORE DATA FOLLOWS AT END OF STRUCT */
};
```

注意三点:

1. **`t_choice` 是个 union**——`t_heap`(落盘时填事务信息)和 `t_datum`(在内存里当复合类型值用时填类型/长度信息)共用。落盘前用 `t_datum` 字段,`heap_prepare_insert` 里会把事务信息写进去覆盖。一行两种身份,省一份空间。

2. **`t_hoff` 是头部长度的"导航员"**——它记录"头部(含 null 位图和填充)一共多少字节"。用户数据从 `t_hoff` 之后开始。源码注释:

> [src/include/access/htup_details.h:117-119](../postgresql-17.0/src/include/access/htup_details.h#L117-L119)
> "Note that t_hoff must be a multiple of MAXALIGN."

3. **固定头逻辑大小 = 23 字节**——源码里直接写死标注 `^ - 23 bytes - ^`:

> [src/include/access/htup_details.h:175](../postgresql-17.0/src/include/access/htup_details.h#L175)
```c
	/* ^ - 23 bytes - ^ */
```

`SizeofHeapTupleHeader` 就是 `offsetof(HeapTupleHeaderData, t_bits)`,等于 23。但 `t_hoff` 必须 `MAXALIGN`,所以在 64 位平台(对齐 8),没有 null 位图时 `t_hoff = 24`(补 1 字节 padding);有 null 位图时按 null 数量算再对齐。

> **为什么头部只有 23 字节这么抠?** 因为一行数据通常也就几十到几百字节,头部多一个字节,每行都多一个字节,一页几百行、全表上亿行,放大出来是天文数字的浪费。把"看不见的列"压进 23 字节(事务信息 + 状态标志 + 头长都塞下),是存储引擎对空间锱铢必较的典型。

## 四、两套标志位:t_infomask 与 t_infomask2

固定头里有两个 `uint16` 的标志位字段:`t_infomask` 和 `t_infomask2`。乍看多此一举,其实是有意拆分。先看 `t_infomask`(可见性 + 存储特征):

> [src/include/access/htup_details.h:190-209](../postgresql-17.0/src/include/access/htup_details.h#L190-L209)

```c
#define HEAP_HASNULL			0x0001	/* has null attribute(s) */
#define HEAP_HASVARWIDTH		0x0002	/* has variable-width attribute(s) */
#define HEAP_HASEXTERNAL		0x0004	/* has external stored attribute(s) */
#define HEAP_HASOID_OLD			0x0008	/* has an object-id field */
#define HEAP_XMAX_KEYSHR_LOCK	0x0010	/* xmax is a key-shared locker */
#define HEAP_COMBOCID			0x0020	/* t_cid is a combo CID */
#define HEAP_XMAX_EXCL_LOCK		0x0040	/* xmax is exclusive locker */
#define HEAP_XMAX_LOCK_ONLY		0x0080	/* xmax, if valid, is only a locker */
...
#define HEAP_XMIN_COMMITTED		0x0100	/* t_xmin committed */
#define HEAP_XMIN_INVALID		0x0200	/* t_xmin invalid/aborted */
#define HEAP_XMIN_FROZEN		(HEAP_XMIN_COMMITTED|HEAP_XMIN_INVALID)
#define HEAP_XMAX_COMMITTED		0x0400	/* t_xmax committed */
#define HEAP_XMAX_INVALID		0x0800	/* t_xmax invalid/aborted */
#define HEAP_XMAX_IS_MULTI		0x1000	/* t_xmax is a MultiXactId */
#define HEAP_UPDATED			0x2000	/* this is UPDATEd version of row */
```

再看 `t_infomask2`(属性数 + HOT 相关):

> [src/include/access/htup_details.h:277-284](../postgresql-17.0/src/include/access/htup_details.h#L277-L284)

```c
#define HEAP_NATTS_MASK			0x07FF	/* 11 bits for number of attributes */
/* bits 0x1800 are available */
#define HEAP_KEYS_UPDATED		0x2000	/* tuple was updated and key cols
										 * modified, or tuple deleted */
#define HEAP_HOT_UPDATED		0x4000	/* tuple was HOT-updated */
#define HEAP_ONLY_TUPLE			0x8000	/* this is heap-only tuple */

#define HEAP2_XACT_MASK			0xE000	/* visibility-related bits */
```

### 不这样会怎样:为什么拆两套,不塞进一个 uint16?

> **不这样会怎样**:假设把"属性数"和"可见性标志"全塞进一个 `uint16`。属性数(`HEAP_NATTS_MASK`)占了 11 位,可见性标志又有一大堆(`XMIN_COMMITTED/INVALID/FROZEN`、`XMAX_COMMITTED/INVALID/IS_MULTI`、各种锁标志……),一个 16 位字段根本塞不下。要么牺牲属性数上限(但 11 位才支持到 2047 列,已经紧),要么牺牲标志位(把锁信息丢到别处),要么用 `uint32`(每行多 2 字节,放大浪费)。

**所以这样设计**:拆成两个 `uint16`,各管一摊:

- **`t_infomask`**——管**可见性与并发状态**。`XMIN_COMMITTED/XMIN_INVALID` 是"提示位"(hint bit),记录"插入这行的事务到底提交没提交",这样下次扫到这行就不用去查 CLOG(P4 第 14 章)了,直接看位。锁的强度信息(`KEYSHR/SHR/EXCL`)、`LOCK_ONLY`(只锁不改)也在这里。
- **`t_infomask2`**——管**属性数(`HEAP_NATTS_MASK`,11 位)+ HOT/更新相关**。`HEAP_HOT_UPDATED`(这行是 HOT 更新的旧版本)、`HEAP_ONLY_TUPLE`(这是 HOT 链上的中间版本,没被索引指向)、`HEAP_KEYS_UPDATED`(键列被改了)。

> **拆开的本质理由**:这两类标志的**生命周期和修改频率不同**。可见性标志会随事务推进被反复改写(提交了就把 COMMITTED 位点上),而属性数一旦 tuple 建好就再也不变。把它们分到两个 16 位字段,代码里修改一类时不必担心污染另一类,也方便用掩码 `HEAP_XACT_MASK`/`HEAP2_XACT_MASK` 一次性清/查。这是"按变更语义分组"的小心思。

`HEAP_XMIN_COMMITTED` 这些"提示位"尤其关键——它们把"查 CLOG 才知道的事务状态"**缓存进了 tuple 头**。第一次扫到这行,查 CLOG 确认提交了,就把位点上;之后所有事务扫到这行,看位就行,省掉一次 CLOG 查询。这是 MVCC 性能的一个隐形加速器,P4 第 17 章细讲。

## 五、null 位图:NULL 怎么不占空间

如果一行里某列是 NULL,数据库怎么存?

### 不这样会怎样:NULL 也占一整个字段位置?

> **不这样会怎样**:如果 NULL 像普通值一样占一个字段位置,那一个"几乎全 NULL"的稀疏表(比如 100 列只填了 3 列),每行白白占 97 列的空间,放大到上亿行,浪费惊人。

**所以这样设计**:用一个**位图(bitmap)**,每位对应一列,置 1 表示"这列有值",置 0 表示"这列是 NULL"。位图紧接在固定头之后(从 `t_bits` 开始),只有当这行至少有一个 NULL 时才存(`HEAP_HASNULL` 位点上才存,否则省掉整个位图)。位图大小 = `BITMAPLEN(列数)`,8 列才 1 字节。

> [src/include/access/htup_details.h:114-116](../postgresql-17.0/src/include/access/htup_details.h#L114-L116)
> "The bitmap is *not* stored if t_infomask shows that there are no nulls in the tuple."

位图的状态机在 `fill_val` 里:

> [src/backend/access/common/heaptuple.c:297-303](../postgresql-17.0/src/backend/access/common/heaptuple.c#L297-L303)

```c
		if (isnull)
		{
			*infomask |= HEAP_HASNULL;
			return;
		}

		**bit |= *bitmask;
```

某列是 NULL,只把 `HEAP_HASNULL` 位点上、对应 bit 保持 0,**这一列在数据区就完全不占任何字节**(下面 `fill_val` 直接 return)。位图把"有没有值"和"值多少字节"解耦,让 NULL 几乎零成本。

## 六、定长字段与对齐:为什么 int 后面要 padding

讲完头部和 null,看用户数据区。最简单的是定长字段:一个 `int4` 就存 4 字节,一个 `int8` 存 8 字节。但有件麻烦事——**对齐(alignment)**。

### 不这样会怎样:不对齐,CPU 访问会崩或变慢

CPU 访问内存,对"数据的起始地址是不是它大小的整数倍"很敏感。一个 8 字节的 `int8`,如果起始地址不是 8 的倍数(叫"未对齐"),在很多 CPU 架构上要么直接触发硬件异常(老架构),要么要分两次访问拼起来(慢一倍)。

> **不这样会怎样**:如果 tuple 里字段紧挨着排、完全不管对齐,那 CPU 读一行几乎每个字段都要受惩罚。更危险的是某些架构(如 SPARC、老的 ARM)直接总线错误,进程崩。磁盘上的数据按不对齐存,读进内存还是不对齐——这个问题会贯穿所有读写路径。

**所以这样设计**:tuple 里每个字段,按它类型的对齐要求(`attalign`),在前面补 padding 字节(填 0),让它的起始地址对齐。对齐要求由类型决定:`int4` 对齐 4,`int8`/`float8` 对齐 8(64 位平台),`int2` 对齐 2,`char` 对齐 1。tuple 头本身就先 `MAXALIGN` 对齐,保证数据区起点是对齐的。`heap_form_tuple` 算头长时直接 MAXALIGN:

> [src/backend/access/common/heaptuple.c:1155](../postgresql-17.0/src/backend/access/common/heaptuple.c#L1155)

```c
	hoff = len = MAXALIGN(len); /* align user data safely */
```

`MAXALIGN` 的定义:

> [src/include/c.h:811](../postgresql-17.0/src/include/c.h#L811)

```c
#define MAXALIGN(LEN)			TYPEALIGN(MAXIMUM_ALIGNOF, (LEN))
```

### 列顺序的隐性成本

对齐填充的字节是"白送"的——不存任何信息,只为让下一个字段对齐。这就引出一个工程实践:**表的列顺序会影响空间占用**。比如 `(int8, int4, int2)` vs `(int4, int2, int8)`,因为 int8 要对齐 8,如果排在中间,前后都可能要补 padding。PG 不自动重排列顺序,所以建表时把定长大字段、对齐要求高的放前面,是个减少 padding 的技巧。这是"快"那一侧(省空间=一页塞更多行=I/O 次数少)的一个细节。

## 七、变长字段 varlena:1 字节还是 4 字节 header

定长好说,变长(VARCHAR/TEXT/BYTEA 等)就复杂了——这行数据到底多长,得有个"长度头"。PG 用一套叫 **varlena** 的机制,长度 header 有两种:1 字节和 4 字节。

### 不这样会怎样:统一 4 字节 header,短字符串白送 3 字节

> **不这样会怎样**:如果所有变长字段都用 4 字节长度头(最多表示到 1GB),那一堆短字符串(`'abc'`、`'hi'`)每个都白送 3 字节。一个存用户名、状态码、短备注的表,变长字段几十个,每行浪费几十字节,放大到上亿行又是天文数字。

**所以这样设计**:varlena 有**三种 header**:

- **4 字节 header**(标准变长,`va_4byte`)——长度占 30 位,可表示到 ~1GB,数据需 4 字节对齐。
- **1 字节 header**(短变长,`va_1b` / `VARATT_IS_SHORT`)——长度占 7 位,最多 127 字节,**不对齐**(直接紧挨前一个字段)。这是给短字符串的优化。
- **1 字节 + tag 的 TOAST 指针**(`va_1b_e`)——长度头里标记"这值不在行内,去 TOAST 表找",见下节。

位布局源码注释写得极清楚(以大端为例):

> [src/include/varatt.h:142-154](../postgresql-17.0/src/include/varatt.h#L142-L154)

```c
 * 00xxxxxx 4-byte length word, aligned, uncompressed data (up to 1G)
 * 01xxxxxx 4-byte length word, aligned, *compressed* data (up to 1G)
 * 10000000 1-byte length word, unaligned, TOAST pointer
 * 1xxxxxxx 1-byte length word, unaligned, uncompressed data (up to 126b)
```

关键在**最高位(或最低位,取决于字节序)**:

- 0 开头 → 4 字节 header(第 2 位再区分压不压缩);
- 1 开头 → 1 字节 header(再细分是普通短 varlena 还是 TOAST 指针)。

结构体定义:

> [src/include/varatt.h:111-131](../postgresql-17.0/src/include/varatt.h#L111-L131)


## 八、TOAST:大字段为什么要被切片

变长字段最多到 1GB,但一页才 8KB,一行装不下怎么办?这就是 **TOAST**(The Oversized-Attribute Storage Technique, oversized 太大属性的存储技术)。

### 不这样会怎样:大字段撑爆一页

> **不这样会怎样**:假设你在 `users` 表存了一个 100KB 的简历字段。一个堆页才 8KB,这一行根本塞不进任何一页。如果硬塞,要么把页撑大(破坏 8KB 约定,I/O/Buffer Pool 全乱套),要么这行跨多页存(每次读这行要多次 I/O,且行指针那 4 字节指向哪个偏移都不够用)。更糟的是,一行 100KB 把一整页的其它行位置都挤没了,热点页里塞个大字段,其它行全得搬家。

**所以这样设计**:PG 给每个含变长列的表,自动配一个**独立的 TOAST 表**。当一行太大(超过一个阈值,约 2KB),就把其中的大字段**切片**(chunk,默认每个约 2KB)存到 TOAST 表里;主表这一行原位置只留一个**TOAST 指针**(几字节的 `varatt_external`),记录"去哪个 TOAST 表、按什么 ID 找、原始多大、压没压"。

阈值不是写死 2KB,而是"一页能塞 4 行的最大尺寸"算出来的:

> [src/include/access/heaptoast.h:46-50](../postgresql-17.0/src/include/access/heaptoast.h#L46-L50)

```c
#define TOAST_TUPLES_PER_PAGE	4

#define TOAST_TUPLE_THRESHOLD	MaximumBytesPerTuple(TOAST_TUPLES_PER_PAGE)

#define TOAST_TUPLE_TARGET		TOAST_TUPLE_THRESHOLD
```

`MaximumBytesPerTuple(N)` = (页大小 - 页头 - N 个 ItemId) / N 后对齐。8KB 页算下来,阈值约 2032 字节(具体随平台对齐微调)。也就是说:**一行超过 ~2KB,就触发 TOAST**。选"一页 4 行"是经验值——保证一页至少能塞下几行正常 tuple,不让一行独占一页。

TOAST 指针在主表里的样子(`varatt_external`,只占十几字节):

> [src/include/varatt.h:32-39](../postgresql-17.0/src/include/varatt.h#L32-L39)

```c
typedef struct varatt_external
{
	int32		va_rawsize;		/* Original data size (includes header) */
	uint32		va_extinfo;		/* External saved size (without header) and
								 * compression method */
	Oid			va_valueid;		/* Unique ID of value within TOAST table */
	Oid			va_toastrelid;	/* RelID of TOAST table containing it */
}			varatt_external;
```

- `va_rawsize`——原始多大(没压、没切之前)。
- `va_extinfo`——存到 TOAST 表的实际大小 + 压缩方法(高 2 位编码压缩方法,低 30 位是大小)。如果实际大小 < 原始大小,说明被压缩过。
- `va_valueid`——这个值在 TOAST 表里的唯一 ID。
- `va_toastrelid`——TOAST 表本身的 OID。

读这行时,PG 发现这字段是 TOAST 指针(`VARATT_IS_EXTERNAL`),就去 TOAST 表按 `va_valueid` 把所有切片捞回来,按序拼好(如果压缩过再解压),才得到原始值。这套对应用透明——`SELECT` 时你拿到的还是完整的大字段。

### TOAST 切片多大

切片大小也按"一页能塞 4 个"算:

> [src/include/access/heaptoast.h:80-89](../postgresql-17.0/src/include/access/heaptoast.h#L80-L89)

```c
#define EXTERN_TUPLES_PER_PAGE	4	/* tweak only this */

#define EXTERN_TUPLE_MAX_SIZE	MaximumBytesPerTuple(EXTERN_TUPLES_PER_PAGE)

#define TOAST_MAX_CHUNK_SIZE	\
	(EXTERN_TUPLE_MAX_SIZE -							\
	 MAXALIGN(SizeofHeapTupleHeader) -					\
	 sizeof(Oid) -										\
	 sizeof(int32) -									\
	 VARHDRSZ)
```

也就是:每个 TOAST 切片,扣掉 TOAST 表自己的 tuple 头、ID 字段、长度头之后,净荷约 2KB。一个 100KB 的字段,会被切成约 50 片,各自是 TOAST 表里的一行(带个 chunk 序号),读的时候按序号拼回。

### 四种策略:谁会被切片

不是所有变长字段都一视同仁。每个变长列有个**存储策略**(`attstorage`),由类型的 `TYPSTORAGE` 决定,建表时可改(`ALTER TABLE ... ALTER COLUMN ... SET STORAGE`)。四种策略:

> [src/include/catalog/pg_type.h:307-310](../postgresql-17.0/src/include/catalog/pg_type.h#L307-L310)

```c
#define  TYPSTORAGE_PLAIN		'p' /* type not prepared for toasting */
#define  TYPSTORAGE_EXTERNAL	'e' /* toastable, don't try to compress */
#define  TYPSTORAGE_EXTENDED	'x' /* fully toastable */
#define  TYPSTORAGE_MAIN		'm' /* like 'x' but try to store inline */
```

- **PLAIN(`p`)**——不参与 TOAST。定长类型(int、float、date)和部分小变长(如 uuid)用这个。永远就地存,放不下就直接报错(不会被挪走)。所以一个全定长的大行,如果超过 `MaxHeapTupleSize`,INSERT 直接失败。
- **EXTERNAL(`e`)**——可以挪到 TOAST 表(行外存),但**不压缩**。适合"我不在乎体积、但访问时要随机读片"的场景(比如已经压过的图片、视频)。挪走省行内空间,但不重复压一遍。
- **EXTENDED(`x`)**——**默认策略**。VARCHAR/TEXT/BYTEA/JSON 默认都是它。**先尝试压缩,压完还太大就挪到 TOAST 表**。最"激进"的省空间策略,大多数变长字段用它。
- **MAIN(`m`)**——和 EXTENDED 一样先压后挪,但**尽量留在行内**:只有前面三步都做完、行还是太大,才把 MAIN 的字段挪走。适合"我希望它尽量和别的一起读、少一次 TOAST 表的 I/O"的字段。

### 四步 TOAST 算法

四种策略在 `heap_toast_insert_or_update` 里按固定顺序生效。源码注释把四步说得明明白白:

> [src/backend/access/heap/heaptoast.c:160-168](../postgresql-17.0/src/backend/access/heap/heaptoast.c#L160-L168)

```c

	 *
	 *	1: Inline compress attributes with attstorage EXTENDED, and store very
	 *	   large attributes with attstorage EXTENDED or EXTERNAL external
	 *	   immediately
	 *	2: Store attributes with attstorage EXTENDED or EXTERNAL external
	 *	3: Inline compress attributes with attstorage MAIN

```

```c
			break;
```

| 步 | 策略命中 | 动作 |
|---|---|---|
| 1 | EXTENDED | 先**压缩**;若压完仍超阈值,**挪到 TOAST 表** |
| 2 | EXTENDED / EXTERNAL | 还超?把没挪走的 EXTENDED/EXTERNAL 字段**挪走** |
| 3 | MAIN | 还超?把 MAIN 字段也**压缩** |
| 4 | MAIN | 还超(此时目标阈值放宽到"一页 1 行")?把 MAIN 字段**挪走** |

> [src/backend/access/heap/heaptoast.c:258](../postgresql-17.0/src/backend/access/heap/heaptoast.c#L258)


## 九、行指针状态:LP_NORMAL / LP_DEAD / LP_UNUSED

最后回到页头部那个行指针 `ItemId`。第 6 章讲过它四种状态(`LP_UNUSED/normal/REDIRECT/DEAD`),本章只重申三种(REDIRECT 是 HOT 优化,留给 P6):

> [src/include/storage/itemid.h:38-41](../postgresql-17.0/src/include/storage/itemid.h#L38-L41)

```c
#define LP_UNUSED		0		/* unused (should always have lp_len=0) */
#define LP_NORMAL		1		/* used (should always have lp_len>0) */
#define LP_REDIRECT		2		/* HOT redirect (should have lp_len=0) */
#define LP_DEAD			3		/* dead, may or may not have storage */
```

- **`LP_NORMAL`(1)**——正常,`lp_off`/`lp_len` 指向页里一处活着的 tuple。
- **`LP_UNUSED`(0)**——这个槽位空着,可被新 tuple 复用。
- **`LP_DEAD`(3)**——这行 tuple 已死(没人能看见),但**行指针和 tuple 数据都还在页里占着**。

### 不这样会怎样:为什么 tuple 死了不立刻抹掉?

> **不这样会怎样**:MVCC 下,一行被更新或删除后,旧版本不能马上删——可能还有别的并发事务要读它(它们的事务开始得早,快照里这行还"活着")。如果一删就抹,正在读它的事务就读到错乱。但旧版本总归要清掉,否则页被死元组塞满,空间永远涨。如果每次删除/更新都立刻回收,又得频繁搬动页内数据、加锁,影响并发。

**所以这样设计**:tuple 死了,先标成 `LP_DEAD`(行指针状态 + tuple 仍占着),**等一个独立的、不阻塞业务的 VACUUM 进程来统一清理**。VACUUM 扫到 `LP_DEAD`,确认"确实没有任何事务还需要它了",才把它的空间收回(标成 `LP_UNUSED`,或整页压缩)。这是 MVCC 的代价——读不阻塞写的代价是产生死元组,VACUUM 来还债。`LP_DEAD` 这个状态,是 MVCC 和 VACUUM 之间的交接点。

> 这是 P6 第 21 章(VACUUM)的直接前置。本章只要记住:`LP_DEAD` = 死元组待清理,VACUUM 来收拾;`LP_UNUSED` = 空位可复用。

## 关键源码精读:tuple 的组装与拆解

正文讲完了布局,这一节挑两个最核心的函数——**组装**(`heap_form_tuple` + `heap_fill_tuple`)和**拆解**(`heap_deform_tuple`)——把"为什么这么排"对应到代码。

### 1. 组装一行:heap_form_tuple

`heap_form_tuple` 是把"一组值 + 一组 isnull"装成一个完整 tuple 的入口。

> [src/backend/access/common/heaptuple.c:1116-1192](../postgresql-17.0/src/backend/access/common/heaptuple.c#L1116-L1192)

```c
heap_form_tuple(TupleDesc tupleDescriptor,
				const Datum *values,
				const bool *isnull)
{
	...
	bool		hasnull = false;
	int			numberOfAttributes = tupleDescriptor->natts;
	...

	/*
	 * Check for nulls
	 */
	for (i = 0; i < numberOfAttributes; i++)
	{
		if (isnull[i])
		{
			hasnull = true;
			break;
		}
	}

	/*
	 * Determine total space needed
	 */
	len = offsetof(HeapTupleHeaderData, t_bits);

	if (hasnull)
		len += BITMAPLEN(numberOfAttributes);

	hoff = len = MAXALIGN(len); /* align user data safely */

	data_len = heap_compute_data_size(tupleDescriptor, values, isnull);

	len += data_len;
	...
	tuple->t_len = len;
	...
	HeapTupleHeaderSetNatts(td, numberOfAttributes);
	td->t_hoff = hoff;

	heap_fill_tuple(tupleDescriptor,
					values,
					isnull,
					(char *) td + hoff,
					data_len,
					&td->t_infomask,
					(hasnull ? td->t_bits : NULL));

	return tuple;
}
```

逐段对应前面的设计:

1. **先扫一遍看有没有 NULL**(L1138-L1145)——决定要不要留 null 位图。
2. **算头部总长**(L1150-L1155):固定头 23 字节 + (有 NULL 时)位图 `BITMAPLEN(列数)`,再 `MAXALIGN` 对齐成 `t_hoff`。这就是"23 字节固定头 + 可选位图 + 对齐填充"的算法化身。
3. **算数据区长度** `data_len`(`heap_compute_data_size`,内部会按每个字段的 attlen/attalign 算 padding)——这是对齐填充算到每列的体现。
4. **填头**:`t_hoff`、属性数、然后调 `heap_fill_tuple` 把数据区一行行填进去。

注意 `heap_compute_data_size`(没贴,但被 L1157 调用)会**预先把每个字段的 padding 都算进总长**——所以 `len` 已经包含了对齐填充,后面 `heap_fill_tuple` 才能放心按"前面对齐过的指针"挨个 memcpy。

### 2. 逐列填数据:heap_fill_tuple → fill_val

`heap_fill_tuple` 循环调 `fill_val` 逐列处理:

> [src/backend/access/common/heaptuple.c:426-442](../postgresql-17.0/src/backend/access/common/heaptuple.c#L426-L442)

```c
	*infomask &= ~(HEAP_HASNULL | HEAP_HASVARWIDTH | HEAP_HASEXTERNAL);

	for (i = 0; i < numberOfAttributes; i++)
	{
		Form_pg_attribute attr = TupleDescAttr(tupleDesc, i);

		fill_val(attr,
				 bitP ? &bitP : NULL,
				 &bitmask,
				 &data,
				 infomask,
				 values ? values[i] : PointerGetDatum(NULL),
				 isnull ? isnull[i] : true);
	}

	Assert((data - start) == data_size);
```

`fill_val`(第七节贴过主体)的精髓:

- **定长 by-value**(`attbyval`):`att_align_nominal` 对齐指针 → `store_attbyval` 写值 → 长度 = `attlen`。
- **varlena**(`attlen == -1`):按是否短 varlena/可压缩/外部分四种走法(见第七节),需要时给 `*infomask` 点上 `HEAP_HASVARWIDTH` 或 `HEAP_HASEXTERNAL`。
- **cstring**(`attlen == -2`):从不对齐,memcpy 到 `\0`。
- **定长 by-reference**(`attlen > 0`):对齐后 memcpy。

最后那句 `Assert((data - start) == data_size)` 是硬约束——实际填进去的字节数必须等于预算的 `data_size`。如果对齐算法和数据长度算法不一致(比如改了一个忘了改另一个),这个断言会炸。这是"对齐 padding 在两处独立计算、必须一致"的代码守护。

### 3. 拆解一行:heap_deform_tuple

`heap_deform_tuple` 是 `heap_form_tuple` 的逆运算,把磁盘上的 tuple 拆回 `values`/`isnull` 数组:

> [src/backend/access/common/heaptuple.c:1344-1420](../postgresql-17.0/src/backend/access/common/heaptuple.c#L1344-L1420)

```c
void
heap_deform_tuple(HeapTuple tuple, TupleDesc tupleDesc,
				  Datum *values, bool *isnull)
{
	HeapTupleHeader tup = tuple->t_data;
	bool		hasnulls = HeapTupleHasNulls(tuple);
	...
	char	   *tp;				/* ptr to tuple data */
	uint32		off;			/* offset in tuple data */
	bits8	   *bp = tup->t_bits;	/* ptr to null bitmap in tuple */
	bool		slow = false;	/* can we use/set attcacheoff? */

	natts = HeapTupleHeaderGetNatts(tup);
	...
	tp = (char *) tup + tup->t_hoff;

	off = 0;

	for (attnum = 0; attnum < natts; attnum++)
	{
		Form_pg_attribute thisatt = TupleDescAttr(tupleDesc, attnum);

		if (hasnulls && att_isnull(attnum, bp))
		{
			values[attnum] = (Datum) 0;
			isnull[attnum] = true;
			slow = true;		/* can't use attcacheoff anymore */
			continue;
		}

		isnull[attnum] = false;

		if (!slow && thisatt->attcacheoff >= 0)
			off = thisatt->attcacheoff;
		else if (thisatt->attlen == -1)
		{
			...
			off = att_align_pointer(off, thisatt->attalign, -1,
									tp + off);
			slow = true;
		}
		else
		{
			/* not varlena, so safe to use att_align_nominal */
			off = att_align_nominal(off, thisatt->attalign);
			...
		}

		values[attnum] = fetchatt(thisatt, tp + off);

		off = att_addlength_pointer(off, thisatt->attlen, tp + off);

		if (thisatt->attlen <= 0)
			slow = true;		/* can't use attcacheoff anymore */
	}
	...
}
```

几个关键点:

1. **`tp = (char *) tup + tup->t_hoff`**——数据区从 `t_hoff` 开始,这就是 `t_hoff` 存在的意义:不用每次重新算头部多长,直接跳过去。
2. **NULL 列直接 `continue`**,数据区不读任何字节(`off` 不动)——这就是位图让 NULL 零成本的逆向体现。
3. **`slow` 优化**:如果前面所有列都是定长且无 NULL,PG 会缓存每列的偏移(`attcacheoff`),后续读同一 tupleDesc 的别的 tuple 直接复用,省重复对齐计算。一旦遇到 varlena(`attlen == -1`)或 NULL,缓存失效(`slow = true`)——因为 varlena 用 1 字节 header 时不对齐,偏移会变,没法缓存。这是"短 varlena 优化省空间"换来的"拆解时不能缓存偏移"的小代价。

> 这三个函数是 tuple 存储的"组装/拆解"核心。读它们你会明白:**对齐、null 位图、varlena header 选择,全在 `fill_val` 一个函数里落地**;`t_hoff` 是头部和数据区的分界;`attcacheoff` 是定长无 NULL 路径的加速器。

## 章末小结

- 一行 tuple 在页尾排布:**固定 23 字节头**(`t_xmin/t_xmax/t_cid` + `t_ctid` + `t_infomask/t_infomask2` + `t_hoff`)→ 可选 null 位图 → 对齐填充 → 用户数据区;行指针 `ItemId` 的 `lp_off` 指向它。
- **四个"看不见的列"** `t_xmin/t_xmax/t_cid/t_ctid` 服务 MVCC:`t_xmin` 谁插入、`t_xmax` 谁删除、`t_cid` 事务内第几条命令、`t_ctid` 指向更新后的新版本——读不阻塞写的地基,细节 P4。
- **标志位拆两套** `t_infomask`(可见性/锁,含 `HEAP_XMIN_COMMITTED` 等提示位)和 `t_infomask2`(属性数 + HOT/更新),按变更语义分组。
- **NULL 用位图**(`t_bits`),只有 `HEAP_HASNULL` 才存,每列 1 bit,数据区不占字节。
- **定长字段按 attalign 对齐**,前补 padding(填 0),换 CPU 访问效率;列顺序影响 padding 占用。
- **变长 varlena** 有 4 字节 / 1 字节两种 header,短 varlena(<127B)用 1 字节还免去对齐——给短字符串省空间。
- **TOAST** 在行超过 ~2KB(`TOAST_TUPLE_THRESHOLD`)时触发,把大字段切片(每片 ~2KB)挪到独立 TOAST 表,主表留个 `varatt_external` 指针。四种策略 PLAIN(不动)/EXTERNAL(挪不压)/EXTENDED(先压后挪,默认)/MAIN(尽量留行内),按固定四步算法生效。
- **行指针 `LP_DEAD`** = 死元组待 VACUUM 清理,`LP_UNUSED` = 空位可复用——MVCC 的代价是死元组,VACUUM 来还债。

### 回扣主线

本章同时服务"快"和"不丢不乱"两侧:

- **服务"快"**:tuple 字段布局的本质是数据结构优化——**紧凑对齐、null 位图零成本、变长 1 字节 header、TOAST 切片**,都在让一行尽可能小、一页塞尽可能多行、I/O 次数尽可能少。这是"让数据快"在存储粒度的体现。
- **服务"不丢不乱"**:`t_xmin/t_xmax/t_cid/t_ctid` 这一整套事务标记,加上 `HEAP_XMIN_COMMITTED` 等提示位,是 MVCC 可见性判断的地基;`LP_DEAD` 状态则是 MVCC 产生死元组、VACUUM 回收的交接点。这些"看不见的列"和状态位,让"读不阻塞写、写不阻塞读"成为可能——这是"让数据不乱"的核心。
- **对应三个本性**:TOAST 和对齐,治"磁盘慢"(省空间=少 I/O);`t_xmin/t_xmax`,治"共享会冲突"(MVCC 版本可见性)。
