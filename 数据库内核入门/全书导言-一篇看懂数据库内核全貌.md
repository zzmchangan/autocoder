# 全书导言 · 一篇看懂数据库内核全貌

> 这篇是整本书的"正门"。读任何一篇之前先读它,你会对数据库是个什么东西、每个子系统在干嘛、为什么必须存在,建立一个结实的直觉。
>
> 全文只讲两件事:**数据库在干什么**,以及**为什么非得这么干**。
> 我们跟着**一条 SQL 从发出到返回**的旅程,把整个数据库一口气走一遍。

---

## 一、一句话说清数据库在干嘛

> **数据库站在应用程序和磁盘之间,它让应用程序可以放心地以为:我发出的 SQL 总能拿到正确的结果、我改的数据掉电也不丢、我和别人同时改也不会互相打架——而底下真实的磁盘有多慢、多不可靠、多容易冲突,应用程序完全不用操心。**

这就是数据库存在的全部理由。后面所有机制——连接、解析、优化、执行、存储、索引、事务、日志——都是为了兑现上面这个承诺。

你任何时候看不懂某个机制,就回来问自己一个问题:

> **"这个机制,是在让数据'快',还是在让数据'不丢不乱'?"**

整个数据库也就这两大块活儿。记住这个二分法,你就永远不会迷路:

- **"快"那一侧**:磁盘比内存慢十万倍,所以要用 Buffer Pool 把热数据搬进内存、用索引避免扫全表、用优化器选最省的执行路径——一切为了"别等磁盘、别做无用功"。
- **"不丢不乱"那一侧**:内存一掉电就清空,所以要用 WAL 把改动先记进日志;多人同时改同一份数据会打架,所以要用事务、锁、MVCC 协调——一切为了"数据最终是对的"。

---

## 二、所有复杂性,来自数据的三个本性

二分法是表象,根子在数据本身的**三个本性**。这三条,是全书所有复杂设计的总根源(可以类比操作系统那本姊妹篇里"资源有限 / 大家共享 / 硬件各异"的三个现实):

| 数据的本性 | 它带来的麻烦 | 逼出的子系统 |
|---|---|---|
| 磁盘慢、内存快但**易失** | 快和持久不可兼得 | 存储引擎 + 持久性(Buffer Pool、WAL) |
| 数据要**共享**,但共享会**冲突** | 多人同时改会乱 | 事务与并发控制(ACID、锁、MVCC) |
| 用户说"要**什么**",机器只会"按**位置**读" | 想法和存法对不上 | 查询引擎 + 索引(解析、优化、B 树) |

后面你看到任何一处"为什么搞这么复杂",去找这三个本性里的哪个在作怪——准能找到。

## 三、跟着旅程走一遍:一条 SQL 的一生

整本书的主线,就是这条旅程。每个子系统,都是旅程上的一个驿站。

### 第 1 站:进门(连接与会话)

你用 psql 或 JDBC 连上数据库。这一步看似简单,背后是个重要的架构决策:PostgreSQL 为**每个连接 fork 一个独立的 backend 进程**来服务它。为什么是进程而不是线程?为什么客户端不能直接去碰数据文件?——P1 第 2 章。

### 第 2 站:读懂便条(SQL 解析)

你的 `SELECT * FROM users WHERE id = 42` 到了,但它只是一串字符串。数据库先用**解析器**把它拆成词、按语法组装成一棵**解析树**,并检查"users 表存在吗?id 列存在吗?"。这一步把"人话"翻译成"机器能处理的结构"。——P1 第 3 章。

### 第 3 站:规划取法(查询优化器)

解析树只说"要干什么",没说"怎么干"。取 `id = 42` 这条记录,可以扫遍整张表,也可以走索引——哪种快?**优化器**基于统计信息估算各种执行方案的代价,选最省的那条。同一句 SQL,数据量不同、有没有索引,执行计划可能完全不同。——P1 第 4 章。

### 第 4 站:按计划跑(执行器)

优化器给出的是一棵**计划树**。**执行器**按这棵树一层层取数据(这套"一层层往上吐数据"的模型有个名字,叫**火山模型**)。最底层是扫描(顺序扫表 or 索引扫描),往上可能是连接、聚合、排序。——P1 第 5 章。

### 第 5 站:数据住哪(页面与堆表)

执行器要取数据,数据在哪?磁盘上。但磁盘不是按"行"读写的,而是按**页(page,PostgreSQL 里是 8KB)**整块搬。一张表就是一堆页,每个页里塞着若干行(tuple)。行怎么在页里排?变长字段怎么存?——P2 第 6、7 章。

### 第 6 站:内存中转(Buffer Pool)

每次取数据都读磁盘太慢。所以数据库在内存里辟一块**缓冲池(Buffer Pool)**,常用的页留在内存,读写都先在内存里搞。磁盘比内存慢十万倍,Buffer Pool 是性能的命脉。——P2 第 8 章。

### 第 7 站:快速定位(索引)

`WHERE id = 42`,如果每次都扫遍几百万行,太慢。**索引**(默认是 B 树)让你像查字典目录一样,O(log n) 就定位到。索引和堆表怎么配合?什么时候索引反而更慢?——P3 第 10-13 章。

> 走到这里,"让数据**快**"那一侧的骨干就齐了:**解析 → 优化 → 执行 →(Buffer Pool)→(索引)→ 页面**。这一路,所有机制都在对付"磁盘慢"和"声明式查询对不上物理存取"这两个本性。

### 第 8 站:多人同时改(事务与并发)

但现实里,不止你一个人在用数据库。两个事务同时改同一行,谁先谁后?一个事务改了一半,另一个事务能读到吗?——这就是"共享会冲突"这个本性爆发的时刻。数据库用**事务(ACID)**、**锁**、尤其是 **MVCC(多版本并发控制)** 来摆平:让读不阻塞写、写不阻塞读,同时每个人看到的都是一致的数据。——P4 第 14-17 章(全书最难、最精华)。

### 第 9 站:掉电不丢(WAL 与恢复)

内存里的改动,掉电就没了。你怎么敢相信"数据库说更新成功了"就真的成功了?因为数据库有条铁律:**改数据之前,先把这次改动记进日志(WAL)**。日志是顺序写、比随机改数据快得多,而且先落盘。哪怕这一秒断电,重启后数据库重放日志,就能把没来得及写盘的改动补回来。这就是"易失"这个本性被治住的办法。——P5 第 18-20 章。

### 终点:结果返回

执行器拿到数据,穿过连接,回到你的客户端。一句 SQL 的一生结束。

> **注意第 8、9 站是同时发生的**:一条 SQL 走到"改数据"时,数据库必须**同时**让并发的事务不打架(第 8 站)、又让改动掉电不丢(第 9 站)。这两件事,就是全书主线"快 vs 不丢不乱"里"不丢不乱"那一半的全部来源。

## 四、收束:二分法 + 三句话

走完整条旅程,你会发现数据库毕生精力花在两件事上:

1. **让数据快**(Buffer Pool、索引、优化器)——对付"磁盘慢"和"声明式查询"。
2. **让数据不丢、不乱**(WAL、事务、锁、MVCC)——对付"易失"和"共享冲突"。

### 记住三句话,就抓住了全书

1. **数据库一半在"让数据快",一半在"让数据不丢不乱"。** 迷路时回到这个二分法,问"这站在让数据快,还是让数据不丢不乱"。
2. **所有复杂机制都是被三个本性逼出来的**:磁盘慢且内存易失、数据共享会冲突、用户说"要什么"机器只会"按位置读"。当你觉得某个设计"为什么搞这么复杂",去找是哪个本性在作怪。
3. **一条 SQL 的旅程是全书骨架**:进门 → 解析 → 优化 → 执行 → 存储/索引 → 事务/日志 → 返回。每个子系统都是这条路上的一个驿站。
