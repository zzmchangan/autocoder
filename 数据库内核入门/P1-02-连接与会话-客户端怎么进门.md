# 第 2 章 · 连接与会话:客户端怎么"进门"

> **前置**:你需要先读过[第 1 章《第一性原理》](P0-01-第一性原理-为什么需要数据库.md)——它立下了"一条 SQL 的旅程"这条主线,并解释了数据为什么要被一个**专门的程序**管起来。本章是旅程的**第一站**:SQL 还没开始解析,我们先把"客户端和数据库怎么连上"这件事讲透——包括 PG 为什么给每个连接单独 fork 一个进程、认证发生在哪一步、怎么取消一个跑飞的查询、以及两种查询协议。

> **核心问题**:你敲下 `psql -h ... -U ...`,到能输入 SQL 之间,发生了什么?为什么 PostgreSQL 给**每个客户端连接单独 fork 一个进程**,而不是像很多数据库那样用一个进程里的线程池处理所有连接?一个 backend 崩了会怎样?你按 Ctrl+C 取消一个跑飞的查询,数据库是怎么停下来的?
>
> **读完本章你会明白**:一条连接的完整旅程(TCP 三次握手 → startup packet → fork backend → 认证 → InitPostgres 会话初始化 → 进入主循环);为什么 PG 选"每连接一进程",以及它换来的"崩溃隔离"和付出的"连接成本";为什么内核不内置连接池、社区用 pgbouncer 之类外部池化方案补强;认证有哪几种方式、它在源码里走哪条调用链;cancel key 怎么实现查询取消;以及"简单查询"和"扩展查询"两种协议的区别。

> **如果一读觉得太难**:先记三件事——① 每连接一进程,为的是崩溃隔离;② 进门前先认证(用户/密码/权限),认证通过后这个连接由一个专属 backend 进程一直服务到断开;③ 查询取消靠一个单独的 cancel key + 新连接,而不是在原连接上喊停。

## 一、"进门"这件事,比你以为的复杂

很多新手以为:客户端连数据库,就是建个 TCP 连接,然后往里面发 SQL。差不多对,但漏了关键的中间环节。

先看一条连接在 PG 里**真实的**完整旅程(下面几节会逐段展开,这里先建立全景):

```text
   ┌─────────┐                                              ┌──────────────┐
   │ 客户端   │                                              │  PostgreSQL  │
   │  psql   │                                              │   服务端     │
   └────┬────┘                                              └──────┬───────┘
        │                                                          │
        │  ① TCP 三次握手 (SYN/SYN-ACK/ACK)                          │
        │ ────────────────────────────────────────────────────────► │  postmaster 在 5432 监听
        │                                                          │  accept() 拿到一个已连接套接字
        │                                                          │
        │  ② SSL/GSS 协商(可选)                                     │
        │ ◄──►                                                      │
        │                                                          │
        │  ③ StartupPacket(协议版本 / user / database / GUC)        │
        │ ────────────────────────────────────────────────────────► │  ProcessStartupPacket 解析
        │                                                          │
        │                                           ★ 这里 postmaster fork 出 backend 子进程 ★
        │                                                          ┌──────────────┐
        │                                                          │ backend 子进程 │ ← 从这一刻起,
        │                                                          └──────┬───────┘   这个套接字归它
        │                                                                 │
        │  ④ AuthenticationRequest(要求密码/SCRAM challenge…)             │
        │ ◄────────────────────────────────────────────────────────────  │  ClientAuthentication
        │  ⑤ 认证应答(密码/SCRAM response)                                │   按 pg_hba.conf 选方式
        │ ─────────────────────────────────────────────────────────────► │
        │  ⑥ AuthenticationOk + BackendKeyData(pid + cancel key)         │
        │ ◄────────────────────────────────────────────────────────────  │
        │                                                                 │
        │                                              InitPostgres(选库/relcache/起事务快照)
        │                                                                 │
        │  ⑦ ReadyForQuery ('Z' 消息)                                     │
        │ ◄────────────────────────────────────────────────────────────  │  进入 PostgresMain 主循环
        │                                                                 │
        │  ⑧ 客户端发 SQL('Q' 简单查询 / Parse-Bind-Execute 扩展查询)     │
        │ ─────────────────────────────────────────────────────────────► │  解析→优化→执行→回结果
        │  ⑨ 结果集 / CommandComplete / ReadyForQuery                     │
        │ ◄────────────────────────────────────────────────────────────  │  循环 ⑧⑨,直到客户端断开
```

把这张图浓缩成一句话:

> **客户端发起 TCP 连接,postmaster 在约定端口 `accept`;postmaster 让客户端先自报家门并做认证,然后 `fork` 一个子进程(backend),把这个连接的套接字交给它;从此这条连接的所有 SQL 都由这个专属 backend 处理,postmaster 自己回前台继续等下一个客人。**

注意第 ④ 步之后那个 ★:**postmaster 自己不处理任何 SQL,也不做完整的认证握手**。它的角色更像"前台接待"——接客、读 startup packet、fork 一位专员、把客人领过去,然后立刻回前台等下一个。真正干活(包括跑认证握手、跑 SQL)的是专员。

## 二、为什么是"每连接一进程",而不是线程?

这是 PG 架构里最经典、也最常被讨论的一个决策。很多数据库(MySQL、Oracle、SQL Server)用"线程池"——一堆线程共享处理所有连接;或者像 MySQL 那样"每连接一线程",但所有线程挤在**同一个进程地址空间**里。PG 却坚持"一个连接一个 OS 进程",每个 backend 都是 postmaster 的**子进程**,拥有**独立的地址空间**。为什么?

这是本章的灵魂问题,我们拆三层来回答:换来了什么(崩溃隔离)、附带了什么好处(IPC 与 COW)、付出了什么代价(连接数受限)。

### 不这样会怎样:进程隔离 = 崩溃隔离(fault isolation)

先想"每连接一进程"到底换来了什么。数据库 backend 跑的是**用户提交的 SQL**——这玩意儿不可信:可能有 bug(用户写了触发 PG 自己缺陷的 SQL)、可能 OOM、可能在扩展模块(C 函数)里访问非法地址、可能在执行计划生成的瞬间踩到内存边界。如果所有连接共享一个地址空间(线程模型):

> **不这样会怎样**:一个连接跑了一条触发空指针解引用、或一次非法内存访问的 SQL。整个进程**整个崩掉**——OS 给它发 SIGSEGV,进程当场消失。代价是什么?**所有其他正在处理的连接、所有正在跑的事务、所有未提交的修改,全部陪葬。** 一个用户写的烂 SQL,或者一个写得不严谨的 PL/pgSQL 函数,把全库所有人的服务都打断了。对于数据库这种"多用户共享"系统,这是不可接受的单点失败。

而"每连接一进程":那条烂 SQL 所在的 backend 进程崩溃了,**只有这一个连接死掉**,其他 backend 进程毫无察觉、继续干活——因为 OS 在硬件层面(MMU、页表)保证一个进程崩了不会污染另一个进程的内存。

> **进程模型最根本的好处:崩溃隔离(fault isolation)。** 一个进程的地址空间是独立的,OS 保证它崩了不会牵连别的进程。这份隔离,正是用"进程"而非"线程"换来的最贵重的东西。

PG 把这件事做到了源码里。看 postmaster 文件顶上的那个 `FatalError` 标志:

> [src/backend/postmaster/postmaster.c:271](../postgresql-17.0/src/backend/postmaster/postmaster.c#L271)

```c
static bool FatalError = false; /* T if recovering from backend crash */
```

它就是 postmaster 这个"监工"用来记录"是不是已经有 backend 崩了、现在是不是处于崩溃恢复状态"的开关。一旦某个 backend 异常退出,postmaster 的 `HandleChildCrash` 被触发,把这个标志置 true,然后开始全库重启流程(下一节细讲)。**进程隔离的价值,在这个标志被置位的瞬间体现得最清楚:一个崩了的 backend 不会静悄悄地污染别人,而是被监工大声喊出来、统一处置。**

### 进程相对线程的另外两个好处

除了崩溃隔离,PG 选进程还有两个现实的工程理由:

1. **共享内存做 IPC,简单且安全**。多个 backend 要共享的东西(锁表、缓冲池、WAL 缓冲、ProcArray),PG 用一段显式的**共享内存**段;各自私有的东西(本连接的事务状态、本地内存、解析缓存)就真私有,不会误踩。线程模型里**所有内存默认共享**,一个线程写错地方就可能污染另一个,调试噩梦——尤其难复现的并发 bug,在进程模型里压根不会出现。共享内存是"显式约定才能共享",线程是"默认共享、不想要才要加锁",前者出错面小得多。

2. **`fork()` + 写时复制(COW),启动一个 backend 比想象中便宜**。子进程 fork 出来时,并不真的复制父进程的全部内存,而是和父进程**共享物理页**,只有谁改了哪页才复制那一页。postmaster 启动后,共享内存段、relcache 初始化数据、各类全局表都已经在它的地址空间里;fork 出来的 backend 几乎不触发实际复制(因为它读多写少,大部分共享数据是只读的)。所以"每连接一进程"的启动开销,远低于"复制一份完整进程内存"的直觉开销。

### 崩溃了之后:为什么宁可全库重启?

到这里有个微妙的问题:既然进程隔离这么好,为什么一个 backend 崩了,PG 通常还要**全库重启**(所有 backend 被强制断开、共享内存重新初始化、走一遍崩溃恢复),而不是"只重启那一个 backend"?

> **不这样会怎样**:假设只重启那一个崩溃的 backend、其余 backend 继续干活,会发生什么?**共享内存里的状态可能已经不可信了。** 崩溃的那个 backend,在死之前可能正在改共享内存:也许正在持有一把 LWLock、锁表里登记着它持有某把表锁、缓冲池里某个脏页正被它写到一半、ProcArray 里它的 slot 标着"当前事务正在执行"。它一崩,这些共享状态**没有任何机会被正常释放**——锁没人解、脏页写到一半、ProcArray slot 残留。如果其余 backend 继续在这套**半残的共享状态**上服务,迟早会出现:死锁检测把一个早死了的进程当作"还在持锁"、缓冲池元数据和实际内容不一致、可见性判断错误——**那才是真正的灾难,而且是会悄悄腐蚀数据正确性的灾难**。

所以 PG 的策略是:**一旦共享状态可疑,宁可大声喊出来、停服重启,绝不带着脏状态苟活。** 看源码里 `HandleChildCrash` 的注释:

> [src/backend/postmaster/postmaster.c:2867-2872](../postgresql-17.0/src/backend/postmaster/postmaster.c#L2867-L2872)

```c
 * HandleChildCrash -- cleanup after failed backend, bgwriter, checkpointer,
 * walwriter, autovacuum, archiver, slot sync worker, or background worker.
 *
 * The objectives here are to clean up our local state about the child
 * process, and to signal all other remaining children to quickdie.
```

注意最后那个词:**`quickdie`**。postmaster 不是"礼貌地请其他 backend 退出",而是发 SIGQUIT 让它们 `quickdie`——快速、粗暴地自我了断,因为此刻已经不能信任任何共享状态。每个 backend 收到 SIGQUIT 后会走 [`quickdie`](../postgresql-17.0/src/backend/tcop/postgres.c#L4285) 这个处理器,简单通知客户端"server 要崩了"然后立刻 `_exit`,绝不再碰共享内存。然后 postmaster 自己重新走一遍启动流程、做崩溃恢复(P5 的主角)、把共享内存重置干净,再恢复接客。

> 这正是"崩溃隔离"另一面的诚实:**进程隔离保护了"别的 backend 的私有内存",但共享内存是大家共用的、一旦可疑就必须整体重建。** 所以 PG 的隔离不是"一个崩了完全不影响别人",而是"一个崩了不会**悄悄**污染别人——出事了就大声喊出来(全库重启),绝不带着脏状态继续服务"。这种"宁崩勿脏"的设计哲学,会贯穿 P5 持久性那一篇。

### 代价:连接数大时,进程开销显现

进程模型不是没代价。这恰恰是为什么 PG 在"超高连接数"场景下需要外部池化。

每个 backend 是一个完整的 OS 进程:

- 有自己的**地址空间**(页表、虚拟内存布局)、内核栈、文件描述符表;
- **即使什么都不干(刚连上就 idle)**,也要占几 MB 内存——PG 给每个连接预留的工作内存(work_mem、maintenance_work_mem、temp_buffers)上限是按连接数翻倍的;
- 进程**上下文切换比线程略贵**(要切页表寄存器、刷 TLB),连接数千时调度开销会显现。

> **不这样会怎样**:如果你天真地"每个 Web 请求开一个新连接、用完关掉",然后业务高峰同时来了 2 万个请求——你会 fork 出 2 万个 backend 进程,光是它们空跑占的内存就几十 GB,而 OS 调度 2 万个进程会让 CPU 大半时间花在切换上,真正干 SQL 的活反而不多。这就是为什么**不能无脑开几万个连接**。

PG 的应对分两层:

- **内核侧**:`max_connections` 参数(默认 100)限制 backend 总数,超了直接拒连(下面 cancel key 那节会看到 `CAC_TOOMANY` 这个状态);还预留了 `superuser_reserved_connections` 给管理员救场用。
- **生态侧**:**外部连接池**(Pgbouncer、pgcat、Pgpool-II)——在客户端和 PG 之间架一个轻量代理,把成千上万的客户端连接"复用"到 PG 后端几十个真 backend 上。客户端连的是 pgbouncer,pgbouncer 维护一个到 PG 的 backend 池,把客户端的 SQL 转发过去。

> **为什么 PG 内核不内置连接池?** 这是个常被问的问题。历史上社区有过多次讨论,核心顾虑有二:① 内置池化要做"事务结束后把 backend 借给下一个客户端"的语义切换,涉及会话状态、临时表、prepared statement、`SET` 变量等一大堆"会话级状态"的清理,稍有不慎就出 bug——这种 bug 极其隐蔽(用户的状态被下一个人看到);② PG 的进程模型天然适合"一进程一会话"的清晰语义,内置池化会破坏这套语义。所以社区选择**把池化交给外部组件**(Pgbouncer 是事实标准),内核保持简洁可靠。近几年也有 in-core 的"connection concentrator"提案和插件式方案在讨论,但截至 PG 17,生产环境主流仍是 Pgbouncer。这是 P6 进阶的话题。

## 关键源码精读(上):从 fork 到主循环

我们顺着源码,把"进门 → fork → 主循环"这条线钉死。先看一个常被忽略的事实:**整个 PG 服务端从 `main` 开始就是个分发器**。

### 0. 程序入口:`main` 怎么决定自己是谁

PG 只有一个可执行文件 `postgres`,但根据启动参数它会扮演完全不同的角色——postmaster、单用户模式、bootstrap 模式、子进程……这个分发在 [`main`](../postgresql-17.0/src/backend/main/main.c#L58) 里:

> [src/backend/main/main.c:179-197](../postgresql-17.0/src/backend/main/main.c#L179-L197)

```c
	/*
	 * Dispatch to one of various subprograms depending on first argument.
	 */

	if (argc > 1 && strcmp(argv[1], "--check") == 0)
		BootstrapModeMain(argc, argv, true);
	else if (argc > 1 && strcmp(argv[1], "--boot") == 0)
		BootstrapModeMain(argc, argv, false);
#ifdef EXEC_BACKEND
	else if (argc > 1 && strncmp(argv[1], "--forkchild", 11) == 0)
		SubPostmasterMain(argc, argv);
#endif
	else if (argc > 1 && strcmp(argv[1], "--describe-config") == 0)
		GucInfoMain();
	else if (argc > 1 && strcmp(argv[1], "--single") == 0)
		PostgresSingleUserMain(argc, argv,
							   strdup(get_user_name_or_exit(progname)));
	else
		PostmasterMain(argc, argv);
	/* the functions above should not return */
	abort();
```

注意三点:

- 正常 `postgres -D /data` 启动,落到最后一行 `PostmasterMain`——这是 **postmaster 进程**的入口。
- `--single` 是单用户模式(DBA 维护用),走 `PostgresSingleUserMain`,没有 postmaster、不 fork。
- `--forkchild`(只在 `EXEC_BACKEND` 平台如 Windows 上有)是 fork/exec 模式下子进程的入口——这种平台不能用纯 `fork()`,只能 fork 后再 exec 一个新的 `postgres` 进程,用参数告诉它"你是一个 backend 子进程"。在 Linux 上默认是纯 fork 模式,不走这条分支。

`PostmasterMain` 完成共享内存初始化、读配置、起辅助进程(checkpointer、bgwriter、walwriter、autovacuum launcher…)后,最终进入 `ServerLoop`——postmaster 的主循环。

### 1. postmaster 的主循环:`ServerLoop`

postmaster 启动后的生命,几乎全在这个无限循环里:

> [src/backend/postmaster/postmaster.c:1624-1625](../postgresql-17.0/src/backend/postmaster/postmaster.c#L1624-L1625)

```c
ServerLoop(void)
{
	...
	for (;;)
	{
		...
		nevents = WaitEventSetWait(pm_wait_set,
								   DetermineSleepTime(),
								   events,
								   lengthof(events),
								   0 /* postmaster posts no wait_events */ );
```

postmaster 在这里 **`WaitEventSetWait` 阻塞等待**——它注册了一组监听套接字(每个 `listen_addresses` + 端口一个),有新连接进来时被唤醒。除了接客,它还要处理几类事件:子进程退出(`pending_pm_child_exit`)、配置重载(`pending_pm_reload_request`)、关库请求(`pending_pm_shutdown_request`)、其他子进程发来的信号(`pending_pm_pmsignal`)。看循环体里关键的几行:

> [src/backend/postmaster/postmaster.c:1669-1682](../postgresql-17.0/src/backend/postmaster/postmaster.c#L1669-L1682)

```c
			if (events[i].events & WL_SOCKET_ACCEPT)
			{
				ClientSocket s;

				if (AcceptConnection(events[i].fd, &s) == STATUS_OK)
					BackendStartup(&s);

				/* We no longer need the open socket in this process */
				if (s.sock != PGINVALID_SOCKET)
				{
					if (closesocket(s.sock) != 0)
						elog(LOG, "could not close client socket: %m");
				}
			}
```

读懂这段:`AcceptConnection` 完成 TCP 三次握手的 `accept()`,拿到一个已连接套接字(封装成 `ClientSocket`);成功就 `BackendStartup(&s)`——**fork 一个 backend 来接管**;然后 postmaster **自己关掉这个套接字**(它不再用),回去继续 `WaitEventSetWait` 等下一个。

> 这就是"前台接待"的代码化身:接客 → 把客人交给专员 → 自己关掉对讲机 → 回前台等下一个。postmaster 自始至终不读一条 SQL。

### 2. fork 路径:`BackendStartup` 生成 cancel key 并 fork

`BackendStartup` 是 fork 一个新 backend 的入口。它的开头有个细节,揭示了我们后面 cancel key 那节的设计:

> [src/backend/postmaster/postmaster.c:3544-3546](../postgresql-17.0/src/backend/postmaster/postmaster.c#L3544-L3546)

```c
BackendStartup(ClientSocket *client_sock)
{
	Backend    *bn;				/* for backend cleanup */
	pid_t		pid;
	BackendStartupData startup_data;
```

第一步先在 postmaster 内存里建一个 `Backend` 结构体记录这个子进程。这个结构体长这样:

> [src/backend/postmaster/postmaster.c:168-177](../postgresql-17.0/src/backend/postmaster/postmaster.c#L168-L177)

```c
typedef struct bkend
{
	pid_t		pid;			/* process id of backend */
	int32		cancel_key;		/* cancel key for cancels for this backend */
	int			child_slot;		/* PMChildSlot for this backend, if any */
	int			bkend_type;		/* child process flavor, see above */
	bool		dead_end;		/* is it going to send an error and quit? */
	bool		bgworker_notify;	/* gets bgworker start/stop notifications */
	dlist_node	elem;			/* list link in BackendList */
} Backend;
```

留意 `cancel_key`——postmaster 要用它来识别"哪个客户端有权取消哪个 backend"。fork **之前**,postmaster 先生成一个随机 cancel key:

> [src/backend/postmaster/postmaster.c:3563-3580](../postgresql-17.0/src/backend/postmaster/postmaster.c#L3563-L3580)

```c
	/*
	 * Compute the cancel key that will be assigned to this backend. The
	 * backend will have its own copy in the forked-off process' value of
	 *MyCancelKey, so that it can transmit the key to the frontend.
	 */
	if (!RandomCancelKey(&MyCancelKey))
	{
		pfree(bn);
		ereport(LOG,
				(errcode(ERRCODE_INTERNAL_ERROR),
				 errmsg("could not generate random cancel key")));
		return STATUS_ERROR;
	}

	/* Pass down canAcceptConnections state */
	startup_data.canAcceptConnections = canAcceptConnections(BACKEND_TYPE_NORMAL);
	bn->dead_end = (startup_data.canAcceptConnections != CAC_OK);
	bn->cancel_key = MyCancelKey;
```

注意两个细节:

- **`RandomCancelKey` 用的是 `pg_strong_random`**(看 [L3868-3870](../postgresql-17.0/src/backend/postmaster/postmaster.c#L3868-L3870)),即密码学级强随机数,不是普通 `rand()`。这很关键——cancel key 等同于一个"取消令牌",弱随机会让攻击者能猜出来。
- **`canAcceptConnections(BACKEND_TYPE_NORMAL)` 在 fork 之前再检查一遍**数据库当前状态:正在 shutdown?正在 recovery?连接数已满?这个结果通过 `startup_data` 传给子进程,子进程据此决定是直接拒绝(`CAC_SHUTDOWN`/`CAC_TOOMANY`…)还是继续。**fork 之后再检查、再拒绝会浪费一次 fork,所以提前算好。**

然后才是真正的 fork:

> [src/backend/postmaster/postmaster.c:3593-3595](../postgresql-17.0/src/backend/postmaster/postmaster.c#L3593-L3595)

```c
	pid = postmaster_child_launch(B_BACKEND,
								  (char *) &startup_data, sizeof(startup_data),
								  client_sock);
```

`postmaster_child_launch` 是 PG 17 对 fork 的封装,在 [`launch_backend.c`](../postgresql-17.0/src/backend/postmaster/launch_backend.c#L231) 里:

> [src/backend/postmaster/launch_backend.c:244-278](../postgresql-17.0/src/backend/postmaster/launch_backend.c#L244-L278)

```c
	pid = fork_process();
	if (pid == 0)				/* child */
	{
		/* Close the postmaster's sockets */
		ClosePostmasterPorts(child_type == B_LOGGER);

		/* Detangle from postmaster */
		InitPostmasterChild();

		/* Detach shared memory if not needed. */
		if (!child_process_kinds[child_type].shmem_attach)
		{
			dsm_detach_all();
			PGSharedMemoryDetach();
		}

		/*
		 * Enter the Main function with TopMemoryContext.  The startup data is
		 * allocated in PostmasterContext, so we cannot release it here yet.
		 * The Main function will do it after it's done handling the startup
		 * data.
		 */
		MemoryContextSwitchTo(TopMemoryContext);

		if (client_sock)
		{
			MyClientSocket = palloc(sizeof(ClientSocket));
			memcpy(MyClientSocket, client_sock, sizeof(ClientSocket));
		}

		/*
		 * Run the appropriate Main function
		 */
		child_process_kinds[child_type].main_fn(startup_data, startup_data_len);
		pg_unreachable();		/* main_fn never returns */
	}
```

fork 返回后的分叉至关重要:

- **父进程(postmaster)**:`fork` 返回子进程 pid > 0。它把这个 pid 记到 `BackendList`,然后**立刻返回 `ServerLoop` 继续接客**。它绝不碰 SQL。
- **子进程(新 backend)**:`fork` 返回 0。它做三件关键的事:① `ClosePostmasterPorts` 关掉 postmaster 打开但不属于它的文件描述符(包括 postmaster 自己的监听套接字);② `InitPostmasterChild` 脱离 postmaster 的"亲子关系"清理(设置信号处理器、初始化错误栈);③ 把传进来的 `client_sock` 复制一份保存到全局 `MyClientSocket`——**从这一刻起,这个 TCP 套接字归它独有**。最后调用 `child_process_kinds[B_BACKEND].main_fn`,对 backend 来说就是 `BackendMain`。

### 3. backend 子进程的开场:`BackendMain` → `BackendInitialize`

fork 出来的子进程入口是 [`BackendMain`](../postgresql-17.0/src/backend/tcop/backend_startup.c#L56)。读它的代码,你会看到一条连接进门后最关键的几步依次发生:

> [src/backend/tcop/backend_startup.c:56-106](../postgresql-17.0/src/backend/tcop/backend_startup.c#L56-L106)

```c
void
BackendMain(char *startup_data, size_t startup_data_len)
{
	BackendStartupData *bsdata = (BackendStartupData *) startup_data;
	...
	/* Perform additional initialization and collect startup packet */
	BackendInitialize(MyClientSocket, bsdata->canAcceptConnections);

	/*
	 * Create a per-backend PGPROC struct in shared memory.  We must do this
	 * before we can use LWLocks or access any shared memory.
	 */
	InitProcess();

	/*
	 * Make sure we aren't in PostmasterContext anymore.  (We can't delete it
	 * just yet, though, because InitPostgres will need the HBA data.)
	 */
	MemoryContextSwitchTo(TopMemoryContext);

	PostgresMain(MyProcPort->database_name, MyProcPort->user_name);
}
```

读懂这个函数,你就读懂了 backend 一生的"开场白":

1. **`BackendInitialize`**——初始化这条连接(libpq 状态)、读 startup packet、做早期拒绝检查(库不存在、连接数已满等)。注意此时**还没认证、还没碰共享内存**。
2. **`InitProcess`**——在共享内存里给自己分配一个 `PGPROC` 结构(把这个 backend 注册到 ProcArray,从此"对其他 backend 可见")。**这一步之后,backend 才真正算共享内存的"成员"。**
3. **`PostgresMain`**——backend 的主循环入口,从此在这里循环到断开。

为什么顺序这么排?看 `BackendInitialize` 里那行注释:

> [src/backend/tcop/backend_startup.c:145-146](../postgresql-17.0/src/backend/tcop/backend_startup.c#L145-L146)

```c
	/* This flag will remain set until InitPostgres finishes authentication */
	ClientAuthInProgress = true;	/* limit visibility of log messages */
```

> **不这样会怎样**:如果在认证完成前就把自己注册进共享内存(ProcArray)、就允许访问系统目录,会发生什么?一个**还没证明自己是谁**的客户端,理论上可能触发 backend 的 bug、可能让 backend 崩溃——而一旦它在共享内存里留下了脏状态,这个脏状态就会污染所有其他 backend。所以 PG 刻意把"读 startup packet + 认证"这段最不可信的时间,放在**碰共享内存之前**。如果这段时间 backend 因为恶意/畸形输入要退出,它直接 `_exit(1)`,**不会留下任何共享状态需要清理**。看 `BackendInitialize` 里 SIGTERM 处理器的注释:

> [src/backend/tcop/backend_startup.c:167-176](../postgresql-17.0/src/backend/tcop/backend_startup.c#L167-L176)

```c
	/*
	 * We arrange to do _exit(1) if we receive SIGTERM or timeout while trying
	 * to collect the startup packet; while SIGQUIT results in _exit(2).
	 * Otherwise the postmaster cannot shutdown the database FAST or IMMED
	 * cleanly if a buggy client fails to send the packet promptly.
	 *
	 * Exiting with _exit(1) is only possible because we have not yet touched
	 * shared memory; therefore no outside-the-process state needs to get
	 * cleaned up.
	 */
```

这段注释把"为什么顺序这么排"讲得淋漓尽致:**"能用 `_exit` 直接退"是"还没碰共享内存"的特权。** 一旦碰了共享内存,退出就必须走完整的清理流程(释放锁、从 ProcArray 摘掉自己、刷脏状态…)——这是后面 `InitProcess` 之后的规则。**这种"按是否碰共享内存分阶段"的设计,贯穿整个 backend 启动流程。**

### 4. 读 startup packet:`ProcessStartupPacket`

`BackendInitialize` 里最实质的一步是读 startup packet:


`ProcessStartupPacket` 干四件事,都和协议安全相关:

> [src/backend/tcop/backend_startup.c:526-548](../postgresql-17.0/src/backend/tcop/backend_startup.c#L526-L548)

```c

	 * The first field is either a protocol version number or a special
	 * request code.
	 */

	if (proto == CANCEL_REQUEST_CODE)
	{
		CancelRequestPacket *canc;
		int			backendPID;
		int32		cancelAuthCode;
```

## 进门先认证:你是谁,你有权吗

连接不是建上就能用,中间还有**认证**这一关——这是数据库安全的**第一道闸**。

### 认证发生在哪一步:不在 postmaster,在 backend 子进程里

一个常见误解:以为认证是 postmaster 做的。其实不是——postmaster 只读 startup packet 拿到 user/database 名字,**真正的认证握手(SCRAM 挑战、密码校验)发生在 fork 出来的 backend 子进程里**,具体在 `InitPostgres` → `PerformAuthentication` → `ClientAuthentication` 这条调用链上。看 `PostgresMain` 入口处:

> [src/backend/tcop/postgres.c:4324-4327](../postgresql-17.0/src/backend/tcop/postgres.c#L4324-L4327)

```c
	InitPostgres(dbname, InvalidOid,	/* database to connect to */
				 username, InvalidOid,	/* role to connect as */
				 (!am_walsender) ? INIT_PG_LOAD_SESSION_LIBS : 0,
				 NULL);			/* no out_dbname */
```

`InitPostgres` 是 backend 会话初始化的总入口(下一节细讲)。在它内部,走到认证这一步:

> [src/backend/utils/init/postinit.c:920-925](../postgresql-17.0/src/backend/utils/init/postinit.c#L920-L925)

```c
	else
	{
		/* normal multiuser case */
		Assert(MyProcPort != NULL);
		PerformAuthentication(MyProcPort);
		InitializeSessionUserId(username, useroid, false);
```

`PerformAuthentication` 又委托给真正的认证总入口 [`ClientAuthentication`](../postgresql-17.0/src/backend/libpq/auth.c#L381):

> [src/backend/libpq/auth.c:381-393](../postgresql-17.0/src/backend/libpq/auth.c#L381-L393)

```c
void
ClientAuthentication(Port *port)
{
	int			status = STATUS_ERROR;
	const char *logdetail = NULL;

	/*
	 * Get the authentication method to use for this frontend/database
	 * combination.  Note: we do not parse the file at this point; this has
	 * already been done elsewhere.  hba.c dropped an error message into the
	 * server logfile if parsing the hba config file failed.
	 */
	hba_getauthmethod(port);
```

`hba_getauthmethod` 根据 **pg_hba.conf**(host-based authentication,基于主机的认证)的配置,查出"这个 IP 的这个用户、要连这个库,该用哪种认证方式"。然后 `ClientAuthentication` 进入一个 `switch (port->hba->auth_method)`,按方式分发。

### 认证方式:pg_hba.conf 让你精细到 IP × 用户 × 库

pg_hba.conf 是一个文本配置文件,每行一条规则,形如 `连接类型 数据库 用户 IP 认证方式 [选项]`。它让你能精细到"哪个 IP 段的哪个用户、用哪种方式、连哪个库"。常见几种认证方式:

- **trust**:完全信任,不问密码。**只用于绝对可信的内网或本地测试,生产环境禁用**——任何能连到这个端口的人都能以任意用户身份进库。
- **password**:用明文密码认证(必须配合 SSL,否则密码在网络上裸奔)。
- **md5**:对密码做 MD5 哈希挑战。比明文强,但 MD5 算法本身已不抗碰撞,不推荐新部署使用。
- **scram-sha-256**:**现代默认**。基于 SCRAM-SHA-256(Salted Challenge Response Authentication Mechanism)协议,服务器永远不接触明文密码,客户端发的是用 salt + nonce 挑战后的应答,既能防重放、又能防密码字典泄露后被反推。PG 10+ 推荐、PG 14+ 默认就是它。
- **cert**:基于客户端 SSL 证书认证(需要 SSL 连接 + 客户端证书被服务端 CA 信任)。
- **peer**:依赖操作系统判断连接来自哪个 OS 用户(本地 Unix socket 连接常用,Linux 上内核直接告诉 PG 这个 socket 对端是哪个 uid)。

> **为什么认证这么重要?** 因为数据库存的是最值钱的数据(用户信息、交易记录、商业机密)。一个认证漏洞,等于把全库数据拱手送人——而且数据库通常对内网"信任",一旦被穿透,横向移动到其他系统也容易。所以认证是数据库安全的**第一道闸**,pg_hba.conf 是 DBA 必须配对的配置(默认的 pg_hba.conf 模板会反复提醒不要用 trust)。

### 认证失败会怎样

认证失败(密码错、用户不存在、pg_hba.conf 没匹配到规则),`ClientAuthentication` 内部会 `ereport(FATAL, ...)`,backend 进程退出。客户端收到的是一条 `ErrorResponse` 消息(带 `FATAL` 严重级别和具体的 SQLSTATE 错误码,如 `28P01` = invalid password)。

注意这里又一个安全设计:**认证失败的日志默认不会告诉客户端"用户名不存在"还是"密码错"**——它统一返回"authentication failed",避免攻击者通过差异响应枚举出哪些用户名存在。这种"统一错误响应"是认证系统的标准防 enumeration 实践。

## 会话初始化:InitPostgres 在认证后还做了什么

认证通过只是 backend 一生的开场。`InitPostgres` 这个函数名会让人以为它只做"初始化",其实它做的事**决定了这条会话之后能看到什么、能干什么**。我们把它做的关键事情列一下(对照源码):

> [src/backend/utils/init/postinit.c:736-740](../postgresql-17.0/src/backend/utils/init/postinit.c#L736-L740)

```c
InitPostgres(const char *in_dbname, Oid dboid,
			 const char *username, Oid useroid,
			 bits32 flags,
			 char *out_dbname)
{
```

它做的事情包括(按顺序):

1. **`InitProcessPhase2()`**——把自己加进 ProcArray(其他 backend 从此能看到"我在这")。注释明确点出:"Once I have done this, I am visible to other backends!"([L749-754](../postgresql-17.0/src/backend/utils/init/postinit.c#L749-L754))。

2. **`SharedInvalBackendInit` + `ProcSignalInit`**——注册到共享失效管理器和进程信号数组,让"别人改了缓存、需要通知我失效"这套机制能工作。

3. **`RelationCacheInitialize` + `InitCatalogCache` + `InitPlanCache`**——初始化 **relcache(关系缓存)** 和系统目录缓存的哈希表骨架。relcache 是 PG 在内存里缓存"表/索引的元信息(OID、列定义、约束、统计信息)"的机制——后面每条 SQL 都要反复查这些元信息,缓存在内存里避免反复读 `pg_class`/`pg_attribute` 等系统表。**这一步只建空骨架,真正加载表元信息要等后面第一次访问该表时按需填入。**

4. **`StartTransactionCommand` + `GetTransactionSnapshot`**——**起一个事务、拿一个快照**。这一步的注释揭示了一个深层原因:

> [src/backend/utils/init/postinit.c:853-860](../postgresql-17.0/src/backend/utils/init/postinit.c#L853-L860)

```c
	/*
	 * Start a new transaction here before first access to db, and get a
	 * snapshot.  We don't have a use for the snapshot itself, but we're
	 * interested in the secondary effect that it sets RecentGlobalXmin. (This
	 * is critical for anything that reads heap pages, because HOT may decide
	 * to prune them even if the process doesn't attempt to modify any
	 * tuples.)
	 */
```

这里牵出 P4 才会细讲的 MVCC/HOT 机制,本章你只需记一个直觉:**在 PG 里,即便是"读"操作,也要先有一个事务快照,才能安全地读 heap 页**(否则后台的 HOT 清理可能在你眼皮底下把你要读的行 pruning 掉)。所以 backend 在第一次访问数据库前,必须先起事务、先拿快照。

5. **`PerformAuthentication`**——这里才真正做认证(前面那节讲过)。

6. **`GetDatabaseTuple(in_dbname)`**——查 `pg_database` 系统表,确认要连的数据库**真的存在**、拿到它的 OID。不存在就 `FATAL: database "xxx" does not exist`:

> [src/backend/utils/init/postinit.c:1024-1036](../postgresql-17.0/src/backend/utils/init/postinit.c#L1024-L1036)

```c
	else if (in_dbname != NULL)
	{
		HeapTuple	tuple;
		Form_pg_database dbform;

		tuple = GetDatabaseTuple(in_dbname);
		if (!HeapTupleIsValid(tuple))
			ereport(FATAL,
					(errcode(ERRCODE_UNDEFINED_DATABASE),
					 errmsg("database \"%s\" does not exist", in_dbname)));
		dbform = (Form_pg_database) GETSTRUCT(tuple);
		dboid = dbform->oid;
	}
```

7. **锁数据库对象 + 把 dboid 设到全局**——给自己连的这个数据库加一个 `RowExclusiveLock`,防止别人 concurrently DROP 它;然后设置 `MyDatabaseId`、`MyDatabaseTableSpace` 等全局变量。从此这条会话"绑定"到这个数据库。

8. **加载这个数据库自己的系统目录到 relcache**——前面第 3 步加载的是共享的系统目录(`pg_database`、`pg_authid` 这些跨库共享的),这一步加载这个库**特有**的目录(`pg_class` 里属于这个库的部分等)。从这一刻起,backend 才能解析 `SELECT * FROM users` 这种引用本库表的 SQL。

9. 后面还有加载 GUC、process startup options、设置 client encoding、报告 backend 启动到统计系统、提交初始化事务等收尾。

`InitPostgres` 返回后,backend 才真正"准备好接 SQL"了。这一连串步骤的意义:**它把一个刚 fork 出来的、对数据库一无所知的子进程,变成了一个"知道自己是谁、连的哪个库、用哪个角色、能看到哪些表"的有状态会话**。这套状态(relcache、事务快照、会话 GUC、角色 OID)就是会话(session)的全部内涵——它从这一刻起跟着 backend,直到 backend 退出。

## 关键源码精读(下):backend 的主循环与崩溃恢复

### 1. backend 的一生:`PostgresMain` 主循环

`InitPostgres` 返回后,backend 进入 [`PostgresMain`](../postgresql-17.0/src/backend/tcop/postgres.c#L4239)——backend 的"主循环"。一个 backend 进程几乎全部的生命都耗在这个函数里:

> [src/backend/tcop/postgres.c:4239-4240](../postgresql-17.0/src/backend/tcop/postgres.c#L4239-L4240)

```c

{
	sigjmp_buf	local_sigjmp_buf;
```

> [src/backend/tcop/postgres.c:4363-4372](../postgresql-17.0/src/backend/tcop/postgres.c#L4363-L4372)

```c
	/*
	 * Send this backend's cancellation info to the frontend.

	if (whereToSendOutput == DestRemote)
	{
		StringInfoData buf;

		pq_sendint32(&buf, (int32) MyProcPid);
		pq_sendint32(&buf, (int32) MyCancelKey);
		pq_endmessage(&buf);
		/* Need not flush since ReadyForQuery will do it. */
```

**这段代码就是 cancel key 机制的协议层化身。** backend 把自己的 pid 和 cancel key 通过 `BackendKeyData` 消息发给客户端,客户端存起来——以后要取消这个 backend 的查询,就拿这两个值开新连接发 cancel request。然后发一条 `ReadyForQuery` 表示"我准备好接活了",进入主循环:

> [src/backend/tcop/postgres.c:4549-4552](../postgresql-17.0/src/backend/tcop/postgres.c#L4549-L4552)

```c
	for (;;)
	{
		int			firstchar;
		StringInfoData input_message;
```

这个无限循环的骨架是:

> [src/backend/tcop/postgres.c:4688-4699](../postgresql-17.0/src/backend/tcop/postgres.c#L4688-L4699)

```c

		 * (2) Allow asynchronous signals to be executed immediately if they
		 * come in while we are waiting for client input. (This must be
		 * conditional since we don't want, say, reads on behalf of COPY FROM
		 * STDIN doing the same thing.)

```

`ReadCommand` 阻塞等待客户端发来的下一个消息([它在 L492](../postgresql-17.0/src/backend/tcop/postgres.c#L492))。客户端发来一条 SQL,`firstchar` 是消息类型的第一个字节(协议消息都以一个字节标识类型),循环体里的 `switch (firstchar)` 把它分发:

> [src/backend/tcop/postgres.c:4749-4773](../postgresql-17.0/src/backend/tcop/postgres.c#L4749-L4773)

```c

		{
			case PqMsg_Query:
				{

					pq_getmsgend(&input_message);
```

`PqMsg_Query` 就是简单查询协议的 `'Q'` 消息——一串 SQL 文本,backend 调 [`exec_simple_query`](../postgresql-17.0/src/backend/tcop/postgres.c#L1017) 把它解析→优化→执行→返回结果(这是第 3 章的主角)。扩展查询协议的 `Parse`/`Bind`/`Execute` 各自对应 `case PqMsg_Parse`/`PqMsg_Bind`/`PqMsg_Execute`,走的是另一套函数(`exec_parse_message` 等)。

```text
   客户端 ──TCP──▶ postmaster (ServerLoop)

                        ▼

                   回 ServerLoop                 ▼
                   接下一个客               BackendMain
                                               │

                                               │
                                          InitProcess (注册到 ProcArray)
                                               │
                                          PostgresMain
                                               │
                                          InitPostgres (认证/选库/relcache/起事务)
                                               │
                                          ┌────sigsetjmp 锚点────┐
                                          │                       │
                                          ▼                       │ (出错时 longjmp 回来)
                                   for(;;){                       │
                                     ReadCommand                  │
                                       → exec_simple_query        │
                                       → 解析→优化→执行→回写      │
                                       → ReadyForQuery            │
                                   }                              │
                                          │                       │
                                   客户端断开 → 进程退出 ─────────┘
```

### 2. `sigsetjmp`:一条 SQL 出错,为什么 backend 不死

主循环开头那个 `sigsetjmp` 是 backend 韧性的核心。看它怎么用:

> [src/backend/tcop/postgres.c:4426-4441](../postgresql-17.0/src/backend/tcop/postgres.c#L4426-L4441)

```c
	if (sigsetjmp(local_sigjmp_buf, 1) != 0)
	{
		/*
		 * NOTE: if you are tempted to add more code in this if-block,
		 * consider the high probability that it should be in
		 * AbortTransaction() instead.  The only stuff done directly here
		 * should be stuff that is guaranteed to apply *only* for outer-level
		 * error recovery, such as adjusting the FE/BE protocol status.
		 */

		/* Since not using PG_TRY, must reset error stack by hand */
		error_context_stack = NULL;

		/* Prevent interrupts while cleaning up */
		HOLD_INTERRUPTS();
```

读法:`sigsetjmp` 第一次执行返回 0,正常往下走到主循环。但如果 backend 在执行 SQL 过程中调用 `ereport(ERROR, ...)`,PG 的错误机制会 `longjmp` 回到这里——这时 `sigsetjmp` 返回非零,进入 `if` 块执行清理:`EmitErrorReport`(把错误发给客户端和日志)、`AbortCurrentTransaction()`(回滚当前事务)、重置各种状态,然后**重新进入主循环等下一条命令**。

> **不这样会怎样**:如果没有这套 longjmp 机制,任何一条 SQL 里的错误(语法错、除零、违反约束)都会让 backend 进程退出——客户端的连接直接断掉,用户得重连。这在实际使用中是灾难:你敲错一个列名,整个 psql 会话就废了。有了 sigsetjmp,**一条 SQL 出错只影响这一条**:回滚它的事务、给客户端报错、然后 backend 若无其事地继续等下一条。这就是为什么你在 psql 里写错 SQL,看到 ERROR 后还能继续敲——backend 没死,只是把这一条的事务回滚了。

注意源码注释里那句关键的话:

> [src/backend/tcop/postgres.c:4407-4417](../postgresql-17.0/src/backend/tcop/postgres.c#L4407-L4417)

```c
	/*
	 * POSTGRES main processing loop begins here
	 *
	 * If an exception is encountered, processing resumes here so we abort the
	 * current transaction and start a new one.
	 *
	 * You might wonder why this isn't coded as an infinite loop around a
	 * PG_TRY construct.  The reason is that this is the bottom of the
	 * exception stack, and so with PG_TRY there would be no exception handler
	 * in force at all during the CATCH part.  By leaving the outermost setjmp
	 * always active, we have at least some chance of recovering from an error
	 * during error recovery.
	 */
```

读这段注释你会理解一个微妙的设计选择:**为什么用裸 `sigsetjmp` 而不是封装好的 `PG_TRY`/`PG_CATCH`?** 因为这是错误栈的"最底层"——CATCH 段如果再出错,必须有外层 setjmp 接住。`PG_TRY` 是栈式的,栈底必须是裸的 `sigsetjmp` 才能保证"错误恢复过程中又出错"也有兜底。这种细节体现了 PG 团队对错误处理路径的深思熟虑。

**注意区分两件事**:这种 `ereport(ERROR)` + longjmp 是**语句级错误**(一条 SQL 出错),backend 不死、回滚继续;而上一节讲的"backend 进程崩溃"(SIGSEGV、C 扩展里的非法访问)是**进程级崩溃**,backend 直接死,要靠 postmaster 全库重启。两者层次不同:**语句错误靠 longjmp 软恢复,进程崩溃靠 postmaster 硬重启。** 这是 PG 韧性设计的两层防线。

## cancel key:怎么取消一个跑飞的查询

一个真实场景:你执行了一个 `SELECT` 扫几亿行,要跑十分钟。你按 Ctrl+C 想取消它——数据库是怎么"停"掉那个正在 backend 里狂跑的查询的?

关键洞察:PG **不能用原来的连接发"取消"**——因为那个连接的 backend 正阻塞在执行那条 SQL,根本不会去读你新发的消息(它的 `ReadCommand` 根本没回到)。所以取消走的是**另一条路**:

1. 连接建立时,backend 通过 `BackendKeyData` 消息把自己的 pid + cancel key 发给客户端存着(前面 [L4363-4372](../postgresql-17.0/src/backend/tcop/postgres.c#L4363-L4372) 那段)。
2. 你按 Ctrl+C 时,客户端(libpq)**新开一条 TCP 连接**到 postmaster,发一个 **cancel request packet**,里面装"要取消哪个 pid 的 backend + 那个 backend 的 cancel key"。
3. postmaster 收到这种特殊包(我们在 `ProcessStartupPacket` 里看到过 `CANCEL_REQUEST_CODE` 分支),调 [`processCancelRequest`](../postgresql-17.0/src/backend/postmaster/postmaster.c#L1835) 验证 key 对得上,然后给目标 backend 发一个 **SIGINT**。
4. 目标 backend 的 `StatementCancelHandler` 把中断标志位置位;backend 在执行循环的检查点(`CHECK_FOR_INTERRUPTS`)看到"有中断待处理",主动中断当前查询、回滚、返回错误。

`processCancelRequest` 的代码完美展示了"为什么用随机 key":

> [src/backend/postmaster/postmaster.c:1859-1874](../postgresql-17.0/src/backend/postmaster/postmaster.c#L1859-L1874)

```c
		if (bp->pid == backendPID)
		{
			if (bp->cancel_key == cancelAuthCode)
			{
				/* Found a match; signal that backend to cancel current op */
				ereport(DEBUG2,
						(errmsg_internal("processing cancel request: sending SIGINT to process %d",
										 backendPID)));
				signal_child(bp->pid, SIGINT);
			}
			else
				/* Right PID, wrong key: no way, Jose */
				ereport(LOG,
						(errmsg("wrong key in cancel request for process %d",
								backendPID)));
			return;
		}
```

读法:postmaster 先在 `BackendList` 里按 pid 找到那个 backend,**然后比对 cancel key**——pid 对、key 也对,才发 SIGINT;pid 对、key 错,记一条日志拒绝。

> **为什么用随机 key,而不是直接"按 pid 取消"?** 防止恶意取消:如果谁都能凭 pid 取消任意 backend,那一个攻击者连上数据库(甚至不需要密码,只要能连到 5432 端口),就能把别人的长查询全杀掉——DoS 攻击。随机 cancel key 相当于"取消令牌",**只有持有它的客户端(即当初建立了连接、拿到 key 的那个)才能取消那个 backend**。pid 是公开信息(`pg_stat_activity` 里能看到),cancel key 是秘密,两者一起构成"知道 pid + 持有 key"的双重凭证。

代码里那句 `/* Right PID, wrong key: no way, Jose */` 注释,堪称 PG 源码里最俏皮的一句——把"门都没有"的安全态度写得明明白白。

## 简单查询 vs 扩展查询:backend 怎么处理你的 SQL

backend 在 `PostgresMain` 循环里读到客户端发来的命令后,有两种处理 SQL 的**协议模式**。看主循环的 `switch` 就一目了然——简单查询是一个 case,扩展查询是三个 case(Parse/Bind/Execute)。

### 简单查询协议

客户端发一个 `PqMsg_Query`(就是 `'Q'` 消息,内容是一串 SQL 文本),backend 调 `exec_simple_query` 把它**一次性**解析→优化→执行→返回结果。一来一回,简单。你用 `psql` 敲 SQL 走的就是这个。

**问题**:它把 SQL 当纯文本,每次都要重新解析(无 prepared statement 复用);而且应用层如果直接拼字符串(`"SELECT ... WHERE id = " + userInput`),容易**SQL 注入**——用户输入的 `' OR 1=1 --` 直接被当成 SQL 语法执行。

### 扩展查询协议

进阶模式,把一次查询拆成三步,可以反复复用:

- **Parse**:发送 SQL 模板(带 `$1`/`$2` 占位符),backend 解析成 prepared statement,返回一个语句句柄。一次解析,多次复用。
- **Bind**:把具体参数值绑定到占位符,生成一个可执行的 portal。
- **Execute**:执行这个 portal,可以分批返回结果。

好处:

1. **一次 Parse、多次 Bind / Execute**:同一条 SQL 反复执行(如循环里批量插入 10000 行)只解析一次,快得多——尤其复杂 SQL 的解析/优化开销很大时,差距明显。
2. **参数和 SQL 分离**:Bind 阶段的参数是结构化的二进制数据(一个 int 就是 4 字节 int,不是字符串),**不会被当成 SQL 语法**。哪怕用户输入 `' OR 1=1`,它也只是被当作一个普通字符串参数值,**从根本上防 SQL 注入**。

> **为什么 JDBC / psycopg / Npgsql 等现代驱动默认用扩展协议?** 因为它又快又安全。你写 `conn.prepareStatement("... WHERE id = ?").setInt(1, 42)`,底层就是 Parse + Bind + Execute。理解这两个协议,能解释很多性能和安全现象:为什么 prepared statement 快、为什么参数化查询能防注入、为什么 ORM 默认都用 prepared statement——根子都在协议的这套拆分上。

### 两种协议在源码里的分叉

回到 `PostgresMain` 主循环,看 `switch (firstchar)`:

- `PqMsg_Query` → `exec_simple_query`(简单协议,一条龙)。
- `PqMsg_Parse` → `exec_parse_message`(扩展协议第一步)。
- `PqMsg_Bind` → `exec_bind_message`(扩展协议第二步)。
- `PqMsg_Execute` → `exec_execute_message`(扩展协议第三步)。
- 还有 `PqMsg_Describe`、`PqMsg_Sync`、`PqMsg_Close` 等扩展协议配套消息。

**无论走哪条,最终落到执行的都是同一套执行器**(P1 后面章节的主角),只是 SQL 进入执行器之前经过的"预处理流水线"不同。简单协议把 Parse/Bind/Execute 揉成一步(每次都重新走完整流程);扩展协议把它们拆开,允许复用 Parse 的结果。

## 章末小结

### 用一句话回顾本章

一条连接进门,要走过 **TCP → startup packet → fork backend → 认证 → InitPostgres 会话初始化 → 主循环** 这条链路。其中 postmaster 是个"前台接待",自己不碰 SQL,只负责接客、读 startup packet、fork backend、把客人交给专员;真正干活的是 fork 出来的 backend 子进程,它从 `BackendMain` → `BackendInitialize` → `InitProcess` → `PostgresMain` → `InitPostgres` 走完一长串初始化,然后进入 `for(;;)` 主循环,在 `ReadCommand` 上阻塞等 SQL、处理、回结果、循环到断开。

### 五个"为什么"清单

1. **为什么"每连接一进程"?** 为崩溃隔离:一个 backend 崩了(空指针、非法内存访问、C 扩展 bug)只死它自己,不会拖垮其他连接。OS 在硬件层面保证进程地址空间隔离,这是线程模型给不了的。附带好处:共享内存 IPC 比"全部内存共享"安全;`fork()` + COW 让 fork 开销远低于直觉。
2. **为什么一个 backend 崩了要全库重启,而不是只重启那一个?** 因为共享内存(锁表、缓冲池、ProcArray)可能在崩溃瞬间被改到一半,已经不可信。带着可疑的共享状态继续服务是数据正确性的灾难,所以宁可全库重启也不带脏状态苟活。这是"宁崩勿脏"的诚实设计。
3. **为什么认证放在 fork 之后、放在 backend 子进程里做?** 因为认证握手涉及网络往返和密码学计算,放在 postmaster 会阻塞它接客;放在 backend 子进程里则不影响其他连接。更深一层:认证这段最不可信的时间里,backend 刻意**不碰共享内存**——出问题可以直接 `_exit`,不留脏状态。
4. **为什么查询取消要新开一条连接、用 cancel key?** 因为原连接的 backend 阻塞在执行 SQL,读不到任何消息;只能用新连接发 cancel request。cancel key 是随机令牌,防止任意客户端凭 pid 就能取消别人的查询(DoS)。
5. **为什么扩展查询协议又快又安全?** Parse 一次复用多次(省去反复解析优化);参数结构化传输不被当 SQL 语法(根本性防注入)。现代驱动默认用它。

### 回扣主线

这一站是**旅程的地基**:无论后面追求"快"还是"不丢不乱",都得先把连接建立、backend 拉起来、会话状态初始化好。

它和"快 vs 不丢不乱"这条主线的关联:

- **进程模型的崩溃隔离、宁崩勿脏、cancel key 的安全凭证**——这些都在服务**"不乱"**(数据不被错误连接污染、查询能被安全取消、共享状态出问题宁可停服也不带病运行),呼应第 1 章的"共享"本性。
- **连接成本、连接池、扩展查询协议**——这些偏向**"快"**(控制连接开销、复用 prepared statement),呼应"性能"那一侧。
- 认证这道闸,则是数据库作为"多人共享"系统必须把守的关隘——谁能连、能做什么、能不能被取消,这些"权限与隔离"问题,本质上是"共享会冲突"这个本性在连接层的化身(呼应第 1 章三大本性里的第二个)。
