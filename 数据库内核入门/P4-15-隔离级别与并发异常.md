# 第 15 章 · 隔离级别与并发异常:并发事务之间到底能看到什么

> **前置**:你需要先读过[第 14 章《事务与 ACID》](P4-14-事务与ACID-一组操作要么全做要么全不做.md)。上一章把 ACID 的 I(隔离性)只点了一句——"并发事务不打架,兑现它的是锁 + MVCC"。但"隔离到什么程度"本身就是一道选择题:**隔离越严,正确性越高,可并发度越低**。本章就是把这条权衡曲线彻底讲透。

> **核心问题**:多个事务并发跑时,它们**互相能看到什么**?SQL 标准定义了四种隔离级别(读未提交 / 读已提交 / 可重复读 / 可串行化)和三类并发异常(脏读 / 不可重复读 / 幻读 / 序列化异常),它们是什么关系?为什么"隔离级别越高越安全,但并发越低"?PG 的实现又有什么独到之处(默认读已提交;它的"可重复读"其实是快照隔离 SI,能防住标准认为可重复读防不住的幻读,但防不住写偏序;要真正可串行化得靠 SSI)?
>
> **读完本章你会明白**:并发异常不是"实现得烂",而是"严格隔离的代价承受不起"逼出的权衡;快照(snapshot)是 MVCC 用来回答"我能看到哪些版本"的核心数据结构——它就是事务进门那一刻拍下的一张"已提交数据的照片";以及为什么 PG 默认读已提交、而非可重复读(性能!快照的作用范围决定了开销)。

> **如果一读觉得太难**:先只记住三件事——① 隔离级别 = 正确性 vs 并发性能的权衡,没有免费午餐;② "快照 = 进门时拍的照片",可重复读整事务共用一张,读已提交每条语句拍一张;③ PG 的"可重复读"是快照隔离 SI,比 SQL 标准的可重复读更严(防幻读),但不是真正可串行化(防不住写偏序)。其余细节第二遍配合源码再抠。

## 一、先把问题摆到桌面上:并发事务互相能看到什么

第 14 章我们说隔离性(I)是"并发事务感觉不到彼此的存在"。这句话听起来痛快,做起来却暗藏一个根本难题:**"感觉不到彼此"到底要到什么程度?**

把场景摆具体。假设账户表里有一行 `id=1, balance=100`。现在两个事务并发:

```text
事务 A (转账查询)              事务 B (独立改动)
                               BEGIN;
                               UPDATE accounts SET balance=balance+50 WHERE id=1;  -- balance 变 150
                               COMMIT;
BEGIN;
SELECT balance FROM accounts WHERE id=1;   -- A 这次读,看到什么?
SELECT balance FROM accounts WHERE id=1;   -- A 再读一次,又看到什么?
COMMIT;
```

焦点全在**事务 A 两次 `SELECT` 能读到什么**。这件事取决于"隔离到什么程度":

- 如果 A 第一次读到 100、第二次读到 150——它"看到了" B 中途提交的改动。这叫**不可重复读(non-repeatable read)**:同一个事务里读同一行,值会变。
- 如果 A 两次都读到 100——它"看不见" B 中途提交的改动,仿佛 B 不存在。这才像"感觉不到彼此"。

直觉上你会说:"当然要后者啊,谁愿意读到飘忽不定的数据?" 但事情没这么简单——**让 A 两次读到同一个值,是要付代价的**。最彻底的做法是"整个事务期间,A 把它读过的所有行都上锁,不许 B 改"——可这样一来 B 就被挡住了,并发度暴跌。如果每张表都有几百个事务在并发读写,这种"严格隔离"会让数据库退化成"一个事务一个事务排队跑",那还要并发干什么?

这就是隔离级别要解决的核心矛盾。它不是"对错题",而是**一道取舍题**:

> **隔离越严,数据越一致(正确性高),但事务之间互相挡路越狠(并发性能低);隔离越松,并发越高,但事务可能读到"中途变了"的数据(正确性风险大)。**

SQL 标准把这道选择题**标准化**成了四个档位,让你按业务需要挑。下面我们先看清四个档位本身,再看清每个档位能挡住哪些"乱子"。

## 二、四档隔离级别:从严到松的一道光谱

SQL 标准(ANSI SQL-92)定义了四个隔离级别,从严到松排列:

| 隔离级别 | 英文 | 俗称 | 一句话特征 |
|---|---|---|---|
| 读未提交 | Read Uncommitted | RU | 连别人没提交的改动都能看到 |
| 读已提交 | Read Committed | RC | 只看得到别人**已提交**的改动(但同一行可能被改两次) |
| 可重复读 | Repeatable Read | RR | 同一事务里读同一行,值不变 |
| 可串行化 | Serializable | — | 并发执行的效果,等同于一个一个串行跑 |

PG 用四个宏表示这四档(数字就是严的程度):

> [src/include/access/xact.h:36-39](../postgresql-17.0/src/include/access/xact.h#L36-L39)

```c
#define XACT_READ_UNCOMMITTED	0
#define XACT_READ_COMMITTED		1
#define XACT_REPEATABLE_READ	2
#define XACT_SERIALIZABLE		3
```

PG 当前事务用的隔离级别存在全局变量 `XactIsoLevel`,它的默认值是 `XACT_READ_COMMITTED`:

> [src/backend/access/transam/xact.c:76-77](../postgresql-17.0/src/backend/access/transam/xact.c#L76-L77)

```c
int			DefaultXactIsoLevel = XACT_READ_COMMITTED;
int			XactIsoLevel = XACT_READ_COMMITTED;
```

`DefaultXactIsoLevel` 是"新事务的默认档位",`XactIsoLevel` 是"当前事务实际在用的档位"。用户可以用 `SET TRANSACTION ISOLATION LEVEL ...` 或配置 `default_transaction_isolation` 改它(后面讲)。

**先看一眼这四档分别防什么**,再逐个讲。这有一张标准对照表(SQL 标准定义 + PG 实际行为):

### 隔离级别 × 并发异常对照表

| 隔离级别 | 脏读 dirty read | 不可重复读 non-repeatable read | 幻读 phantom read | 序列化异常 serialization anomaly |
|---|---|---|---|---|
| 读未提交 RU | **可能出现** | 可能 | 可能 | 可能 |
| 读已提交 RC | 不可能 | 可能 | 可能 | 可能 |
| 可重复读 RR(SQL 标准) | 不可能 | 不可能 | **可能** | 可能 |
| 可重复读 RR(**PG 实际**) | 不可能 | 不可能 | **不可能** | 可能 |
| 可串行化 Serializable | 不可能 | 不可能 | 不可能 | 不可能 |

这张表有三个地方要特别留意,也是本章要讲清的关键:

1. **从上往下,每严一档就多挡住一种异常。** 但每挡住一种异常,都要付出相应的并发代价(后面讲)。
2. **"PG 实际"那一行和"SQL 标准"那一行不一样**——PG 的可重复读能挡住幻读,这比标准定义的 RR 更严。这是 PG 的实现选择(它用的是快照隔离 SI),后面专门讲。
3. **只有可串行化挡住了"序列化异常"**——这是四种异常里最隐蔽、也最难理解的一种(写偏序 write skew)。它没法用"读同一行"来描述,需要单独讲。

下面我们把四种异常一个一个摆开,配上具体时序,讲清"它怎么发生、为什么可怕、哪一档能挡住它"。

## 三、四种并发异常:一个一个看时序

### 异常一:脏读(dirty read)

脏读是一个事务读到了**另一个事务还没提交**的改动。

```text
事务 A                            事务 B
                                  BEGIN;
                                  UPDATE accounts SET balance=balance+1000 WHERE id=1;
                                  -- 改了,但还没 COMMIT
SELECT balance FROM accounts WHERE id=1;
-- A 读到了 1100(基于 B 没提交的改动!)
                                  ROLLBACK;   -- B 反悔了,改动作废
```

> **不这样会怎样**:A 基于一个**根本没存在过**的数据(1100)做了决策——比如它以为账户有钱,放了一笔贷款出去。等 B 回滚,账户其实只有 100,但 A 已经基于"幻影余额"行动了。这种错误比"读到旧数据"严重得多:旧数据至少是真的,脏数据是**从未提交、即将消失**的。

脏读是所有异常里最不可接受的,所以连最宽松的、数据库默认的隔离级别都基本会挡住它。SQL 标准里"读未提交(RU)"允许脏读,但 **PG 的实现里,即使你设成 RU,行为也等同于 RC**(PG 不支持真正的脏读,这是它刻意的选择)。所以在 PG 里,你**永远碰不到脏读**。

### 异常二:不可重复读(non-repeatable read)

不可重复读是同一事务里,两次读**同一行**,值变了——因为别的并发事务在中间改了这一行并提交。

```text
事务 A                            事务 B
BEGIN;                            BEGIN;
SELECT balance FROM accounts WHERE id=1;   -- 读到 100
                                  UPDATE accounts SET balance=150 WHERE id=1;
                                  COMMIT;            -- B 改完提交了
SELECT balance FROM accounts WHERE id=1;   -- 又读到 150!
COMMIT;
```

> **不这样会怎样**:假设 A 第一次读 100,基于它做了某个判断("余额够,放款");中间 B 把余额改成 150;A 第二次读发现变成 150——它第一次基于的判断**已经在脚下变了**。如果 A 这两次读之间还夹着计算,前后数据对不上,逻辑就乱了。这种"同一事务内数据自相矛盾"的情况,在报表、对账、审计这类场景里尤其致命。

挡住它的就是"可重复读(RR)":在 RR 下,A 两次读都会得到同一个值(它第一次读时拍下的快照),B 的提交 A 看不见。读已提交(RC)挡不住它——这正是 RC 和 RR 的分界线。

### 异常三:幻读(phantom read)

幻读比不可重复读更微妙。它不是"同一行的值变了",而是"同一个范围查询,两次查出来的**行数**变了"——因为别的并发事务在中间**插入或删除**了符合条件的行。

```text
事务 A                                   事务 B
BEGIN;                                   BEGIN;
SELECT count(*) FROM accounts WHERE balance > 1000;
-- 查到 5 行(5 个千万富翁)
                                         INSERT INTO accounts VALUES(..., balance=2000);
                                         COMMIT;          -- B 新增了一个千万富翁
SELECT count(*) FROM accounts WHERE balance > 1000;
-- 查到 6 行!凭空多出来一个
COMMIT;
```

> **不这样会怎样**:幻读的可怕在于它破坏的是"集合"的一致性。比如 A 想统计"存款超过 1000 万的客户名单"用于发邀请函,第一次查到 5 人,基于它印了 5 张邀请函;中间 B 新增了一个符合条件的人并提交;A 第二次查变成 6 人——邀请函数量对不上名单,漏发了。这种"行数会变"的异常,在范围查询、聚合统计、存在性检查(`WHERE NOT EXISTS`)里最容易咬人。

幻读和不可重复读的区别很关键:**不可重复读针对的是"同一行被改",幻读针对的是"集合里多/少了行(插入或删除)"**。挡住幻读,严格说比挡住不可重复读更难——你得锁住的不只是某一行,而是"某个范围内可能插入新行的空隙"。这正是 SQL 标准认为"可重复读挡不住幻读"的原因:标准定义的 RR 只锁已存在的行,锁不住"还不存在的行"。

但 **PG 的可重复读挡得住幻读**——因为它用的是快照隔离 SI(下一节详讲),整个事务看到的是同一张"照片",B 新插入的行对 A 始终不可见。这是 PG 比标准更严的地方。

### 异常四:序列化异常(serialization anomaly,含写偏序 write skew)

这是四种异常里最难、也最反直觉的一个。它不能用"读"来描述,而是一种**写与写之间的隐蔽冲突**。最典型的形态叫**写偏序(write skew)**:

```text
事务 A                                事务 B
-- 规则:值班室至少留 1 人。现在有 2 人值班(Alice、Bob),有人想请假
-- A 想 Alice 请假,B 想 Bob 请假;单独看都满足"留 1 人"
BEGIN;                                BEGIN;
SELECT count(*) FROM on_call WHERE on_duty=true;  -- A 查到 2
                                     SELECT count(*) FROM on_call WHERE on_duty=true;  -- B 也查到 2
-- "有 2 个人,我走还剩 1,符合规则"      -- "有 2 个人,我走还剩 1,符合规则"
UPDATE on_call SET on_duty=false WHERE name='Alice';
                                     UPDATE on_call SET on_duty=false WHERE name='Bob';
COMMIT;                              COMMIT;
-- 结果:两个人都走了,值班室空了!违反业务规则!
```

> **不这样会怎样**:A、B 各自看自己读到的数据,判断都正确(各自走人后还剩 1 人);可它们都基于**同一份快照**(都读到"2 人在岗")做决策,又各自改的是**不同的行**(A 改 Alice、B 改 Bob),谁也没挡住谁——最终两个"看起来都对"的决策叠加,产生了一个谁都不想要的结果(值班室空了)。这就是写偏序。

写偏序的棘手之处:

- A、B 改的是**不同的行**,所以传统行锁挡不住它(行锁只在改同一行时才冲突)。
- 它不能用"快照隔离"挡住——因为 A、B 进门时都看到"2 人在岗",都各自做"合法"的决策;快照隔离能保证"我看到的数据不被别人改",但保证不了"我的改动和别人合起来还合法"。
- 真正要挡住它,数据库必须**追踪事务读了什么、写了什么,检测出"A 读的依赖 B 写的,B 读的依赖 A 写的"这种循环依赖**——这套机制叫 SSI(可串行化快照隔离),代价昂贵(下一节讲)。

这就是为什么对照表里"序列化异常"只有**可串行化**档位能挡住:RU/RC/RR 都挡不住,PG 的 RR(=SI)也挡不住。

## 四、隔离级别是怎么实现的:快照是核心

上面讲了四种异常分别是什么、哪一档能挡住。现在问"**怎么挡**?"——这就是 MVCC 和快照登场的地方。

### 快照:事务进门时拍的一张"照片"

MVCC(多版本并发控制,第 17 章详讲)的核心思想是:**每条记录都可能同时存在多个版本,每个事务根据自己的"快照"决定能看到哪些版本**。快照(snapshot)就是回答"我能看到哪些版本"的那张清单。

> 用个一次性比喻:快照就像事务**进门那一刻拍下的一张照片**——照片里记录了"这一瞬间,哪些事务已经提交、哪些还在跑"。一个事务拿着这张照片去翻数据:凡是在照片拍摄时**已经提交**的改动,它看得见;在照片拍摄时**还在跑**或**还没开始**的事务的改动,它看不见。照片一旦拍下就定格了,后面新提交的事务,对拿这张照片的人来说等于不存在。

快照在 PG 里就是 `SnapshotData` 这个结构(本章关键源码精读会逐字段讲),它核心记着三件事:

- **`xmin`**:一个事务号阈值。"比 `xmin` 小的事务号,都已提交,我看得见。"(优化:免得对每个事务号都查一遍)
- **`xmax`**:**"所有事务号 ≥ `xmax` 的改动,我都看不见。"**(因为它们是在我拍快照之后才开始的)
- **`xip[]`(xid in progress)**:**"在 `xmin` 和 `xmax` 之间,这几个事务号还在跑(没提交),它们的改动我也看不见。"**

有了这三件,任意一条记录的版本对"拿这张快照的事务"是否可见,就完全确定了。

### RC vs RR:快照什么时候拍、用多久

理解了快照,RC 和 RR 的区别就豁然开朗——它**不是机制的区别,而是"快照拍几张、用多久"的区别**:

| | 读已提交 RC | 可重复读 RR |
|---|---|---|
| 快照何时拍 | **每条 SQL 语句**开始时各拍一张 | 事务**第一条语句**开始时拍一张,贯穿整事务 |
| 看到别人的新提交吗 | 看到(只要拍新照片时它已提交) | 看不到(整事务用老照片) |
| 代价 | 每条语句都要重新扫一遍"当前谁在跑" | 拍一次,后续免单 |

PG 这套逻辑的核心在 `GetTransactionSnapshot()` 函数:

> [src/backend/utils/time/snapmgr.c:215-283](../postgresql-17.0/src/backend/utils/time/snapmgr.c#L215-L283)

```c
Snapshot
GetTransactionSnapshot(void)
{
	if (HistoricSnapshotActive())
	{
		Assert(!FirstSnapshotSet);
		return HistoricSnapshot;
	}

	/* First call in transaction? */
	if (!FirstSnapshotSet)
	{
		...
		if (IsolationUsesXactSnapshot())
		{
			/* First, create the snapshot in CurrentSnapshotData */
			if (IsolationIsSerializable())
				CurrentSnapshot = GetSerializableTransactionSnapshot(&CurrentSnapshotData);
			else
				CurrentSnapshot = GetSnapshotData(&CurrentSnapshotData);
			/* Make a saved copy */
			CurrentSnapshot = CopySnapshot(CurrentSnapshot);
			FirstXactSnapshot = CurrentSnapshot;
			...
		}
		else
			CurrentSnapshot = GetSnapshotData(&CurrentSnapshotData);

		FirstSnapshotSet = true;
		return CurrentSnapshot;
	}

	if (IsolationUsesXactSnapshot())
		return CurrentSnapshot;
	...
	CurrentSnapshot = GetSnapshotData(&CurrentSnapshotData);

	return CurrentSnapshot;
}
```

这段代码的分支结构,就是"RC 和 RR 实现差异"的精确答案。我们逐段拆:

**第一层判断:`!FirstSnapshotSet`(事务里第一次要快照)**。如果这是事务第一次要快照,就去拍一张——无论 RC 还是 RR,第一次都得拍。拍完把 `FirstSnapshotSet` 置 true。

**第二层判断:`IsolationUsesXactSnapshot()`**——这就是区分 RC 和 RR 的开关。看它的定义:

> [src/include/access/xact.h:51-52](../postgresql-17.0/src/include/access/xact.h#L51-L52)

```c
#define IsolationUsesXactSnapshot() (XactIsoLevel >= XACT_REPEATABLE_READ)
#define IsolationIsSerializable() (XactIsoLevel == XACT_SERIALIZABLE)
```

一句话:**隔离级别 ≥ RR 时,`IsolationUsesXactSnapshot()` 为真**——也就是"整事务共用一张快照"。这就解释了 RR 这个名字:"事务级快照(transaction-snapshot mode)",一事务一张照片。

把这个宏套回 `GetTransactionSnapshot()`:

- **RR / Serializable(`IsolationUsesXactSnapshot()` 为真)**:第一次拍快照时,**额外做一份拷贝**(`CopySnapshot`),存进 `FirstXactSnapshot`,以后整事务复用它。注意第二次进函数时直接 `return CurrentSnapshot`——**根本不再拍新快照**。这就是"可重复读"的真相:整事务锁死一张照片。
- **RC(`IsolationUsesXactSnapshot()` 为假)**:每次进函数都走到最后的 `CurrentSnapshot = GetSnapshotData(&CurrentSnapshotData)`——**每条语句都重拍**。所以 RC 下你能看到别人在两次语句之间新提交的改动。

> **为什么 PG 默认 RC 而不是 RR?** 看上面代码就知道:RR 要求整事务**锁死一张快照**,意味着这张快照覆盖的事务(那些在它拍照时还在跑的)的改动,整事务期间对它都不可见——哪怕它们早就提交了。这有两个代价:① 这张老快照必须**一直被持有**(它的 `xmin` 阻碍 VACUUM 回收那些它"还可能要看"的旧版本,第 21 章详讲);② 长事务在 RR 下会拖累整个系统(它持有的老快照让一大堆死元组迟迟清不掉)。RC 每条语句用完即弃,快照生命周期短,系统包袱小。所以 PG 默认 RC,把"要不要更严的隔离"交给应用按需开启。这是**性能**驱动的选择——回扣全书主线,这是"快 vs 不丢不乱"里偏向"快"的那一侧。

### 可串行化:在 RR 之上加 SSI

可串行化档位在快照层面和 RR 一样——它也是整事务共用一张快照(看上面代码里 `IsolationIsSerializable()` 分支,同样走 `CopySnapshot`)。但它**多干了一件事**:在拿快照时,调的是 `GetSerializableTransactionSnapshot()` 而不是直接的 `GetSnapshotData()`:

```c
if (IsolationIsSerializable())
    CurrentSnapshot = GetSerializableTransactionSnapshot(&CurrentSnapshotData);
```

> [src/backend/utils/time/snapmgr.c:256-257](../postgresql-17.0/src/backend/utils/time/snapmgr.c#L256-L257)

`GetSerializableTransactionSnapshot` 干什么?它在拿快照之外,**给这个事务注册一套"谓词锁(predicate lock)"追踪机制**——这是 SSI(Serializable Snapshot Isolation)的入口。

> [src/backend/storage/lmgr/predicate.c:1667-1693](../postgresql-17.0/src/backend/storage/lmgr/predicate.c#L1667-L1693)

```c
GetSerializableTransactionSnapshot(Snapshot snapshot)
{
	Assert(IsolationIsSerializable());
	...
	if (RecoveryInProgress())
		ereport(ERROR, ... "cannot use serializable mode in a hot standby" ...);

	/*
	 * A special optimization is available for SERIALIZABLE READ ONLY
	 * DEFERRABLE transactions -- we can wait for a suitable snapshot and
	 * thereby avoid all SSI overhead once it's running.
	 */
	if (XactReadOnly && XactDeferrable)
		return GetSafeSnapshot(snapshot);

	return GetSerializableTransactionSnapshotInt(snapshot,
												 NULL, InvalidPid);
}
```

SSI 的核心思路是:**不仅给事务一张快照,还偷偷记下它读了哪些数据(谓词锁);等它要写时,检查"我写的这些数据,会不会和某个并发事务读过的数据形成依赖环"——如果形成,就回滚其中一个,报"序列化失败"**。这种"先乐观跑、发现冲突再回滚"的策略,让可串行化比传统的"两阶段锁"并发度高得多,但仍比 RR 贵得多(要维护一堆谓词锁、要追踪读写依赖)。

我们看一下谓词锁锁的是什么——它锁的不是"某一行",而是"读过的范围"(可能是整张关系、某个页、某一行,粒度可变):

> [src/include/storage/predicate_internals.h:361-367](../postgresql-17.0/src/include/storage/predicate_internals.h#L361-L367)

```c
typedef enum PredicateLockTargetType
{
	PREDLOCKTAG_RELATION,
	PREDLOCKTAG_PAGE,
	PREDLOCKTAG_TUPLE,
	/* TODO SSI: Other types may be needed for index locking */
} PredicateLockTargetType;
```

注意这三种粒度:**关系级、页级、元组级**。SSI 会尽量用细粒度(元组级),但当锁太多时自动"升级"到更粗的粒度(页级、关系级)以节省内存。这种粒度调整是谓词锁管理开销的一部分,也是为什么可串行化代价最高。

> **可串行化的代价**:(1) 内存——每个可串行化事务的读写都要在共享哈希表里登记谓词锁;(2) CPU——每次读写都要查这套表、检测冲突;(3) **回滚**——一旦检测到"危险结构"(可能产生写偏序的循环依赖),SSI 会主动回滚其中一个事务,客户端收到 `could not serialize access due to read/write dependencies among transactions` 错误,**应用必须重试**。这意味着用可串行化的代码必须写好重试逻辑,否则用户会看到莫名其妙的失败。这就是为什么可串行化是"最严但最贵"的档位——它用性能和复杂度换来了"绝对正确"。

## 五、PG 实现的几个特别之处(对照标准)

讲完机制,把 PG 的几个"独到选择"摆出来,免得读者拿标准套 PG 时踩坑:

### 1. PG 没有"真正的读未提交"

SQL 标准里 RU 允许脏读。但 **PG 的 RU 行为等同于 RC**——它不让你读到未提交的改动。这是 PG 的刻意决定:脏读几乎没有正当用途,实现真脏读反而要额外开销(要绕过快照机制),不划算。所以你在 PG 里即使 `SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED`,拿到的也是 RC 行为。

### 2. PG 的"可重复读"实际是快照隔离(SI)

这是本章最关键的一个事实。SQL 标准的可重复读**允许幻读**(标准认为 RR 只保证"同一行可重复读",不保证"范围查询结果稳定")。但 **PG 的 RR 实现是快照隔离**——整事务用同一张快照,所有新插入的行对它都不可见,所以**幻读也被挡住了**。这就是对照表里"PG 实际"那行比"SQL 标准"那行多挡一个幻读的原因。

但 SI **挡不住写偏序**(第四节那个值班室的例子)——因为它只能保证"我看到的数据稳定",保证不了"我和别人的改动合起来还合法"。要挡写偏序,必须上 SSI(可串行化)。

> 这有个实用结论:**在 PG 里,可重复读比大多数数据库的可重复读更安全(挡幻读),但如果你有写偏序类业务逻辑(基于读做条件写,且并发改不同行),仍要用可串行化或加显式锁**。很多迁移到 PG 的应用,从别的数据库迁过来发现"以前会出幻读的查询现在不出了",就是 SI 的功劳。

### 3. 可串行化用 SSI,不是传统两阶段锁

传统的可串行化用"两阶段锁(2PL)":读加共享锁、写加排他锁,事务结束后才释放。它绝对正确,但并发度极低(读也会挡写)。PG 用的是 **SSI(Serializable Snapshot Isolation)**——一种乐观策略:让事务照常用 MVCC 快照跑,但额外追踪读写依赖,**只在检测到真正会破坏可串行性的"危险结构"时才回滚**。这让可串行化的并发度比 2PL 高得多,代价是实现极其复杂(`predicate.c` 是 PG 里最难的源码之一)。

## 六、关键源码精读:`SnapshotData` ——快照到底装了什么

前面多次提到快照,现在把它彻底拆开。`SnapshotData` 是 PG 里回答"一条记录的版本对一个事务是否可见"的核心结构。

> [src/include/utils/snapshot.h:142-217](../postgresql-17.0/src/include/utils/snapshot.h#L142-L217)

```c
typedef struct SnapshotData
{
	SnapshotType snapshot_type; /* type of snapshot */

	/*
	 * The remaining fields are used only for MVCC snapshots, and are normally
	 * just zeroes in special snapshots.  (But xmin and xmax are used
	 * specially by HeapTupleSatisfiesDirty, and xmin is used specially by
	 * HeapTupleSatisfiesNonVacuumable.)
	 *
	 * An MVCC snapshot can never see the effects of XIDs >= xmax. It can see
	 * the effects of all older XIDs except those listed in the snapshot. xmin
	 * is stored as an optimization to avoid needing to search the XID arrays
	 * for most tuples.
	 */
	TransactionId xmin;			/* all XID < xmin are visible to me */
	TransactionId xmax;			/* all XID >= xmax are invisible to me */

	/*
	 * For normal MVCC snapshot this contains the all xact IDs that are in
	 * progress, unless the snapshot was taken during recovery in which case
	 * it's empty. For historic MVCC snapshots, the meaning is inverted, i.e.
	 * it contains *committed* transactions between xmin and xmax.
	 *
	 * note: all ids in xip[] satisfy xmin <= xip[i] < xmax
	 */
	TransactionId *xip;
	uint32		xcnt;			/* # of xact ids in xip[] */

	/*
	 * For non-historic MVCC snapshots, this contains subxact IDs that are in
	 * progress (and other transactions that are in progress if taken during
	 * recovery). For historic snapshot it contains *all* xids assigned to the
	 * replayed transaction, including the toplevel xid.
	 *
	 * note: all ids in subxip[] are >= xmin, but we don't bother filtering
	 * out any that are >= xmax
	 */
	TransactionId *subxip;
	int32		subxcnt;		/* # of xact ids in subxip[] */
	bool		suboverflowed;	/* has the subxip array overflowed? */

	bool		takenDuringRecovery;	/* recovery-shaped snapshot? */
	bool		copied;			/* false if it's a static snapshot */

	CommandId	curcid;			/* in my xact, CID < curcid are visible */
	...
	uint32		active_count;	/* refcount on ActiveSnapshot stack */
	uint32		regd_count;		/* refcount on RegisteredSnapshots */
	...
	TimestampTz whenTaken;		/* timestamp when snapshot was taken */
	XLogRecPtr	lsn;			/* position in the WAL stream when taken */

	/*
	 * The transaction completion count at the time GetSnapshotData() built
	 * this snapshot. Allows to avoid re-computing static snapshots when no
	 * transactions completed since the last GetSnapshotData().
	 */
	uint64		snapXactCompletionCount;
} SnapshotData;
```

这个结构字段不少,但抓住三组就够:

**第一组:可见性的核心——`xmin` / `xmax` / `xip[]`**。这三个字段,共同定义了"这张快照能看见哪些事务的改动"。注释把规则说得很清楚:

- 凡是事务号 `< xmin` 的,都已提交,**可见**;
- 凡是事务号 `≥ xmax` 的,在拍快照之后才开始,**不可见**;
- 在 `xmin` 和 `xmax` 之间的,要查 `xip[]`(in progress 列表):在列表里的(还没提交),**不可见**;不在列表里的(已提交),**可见**。

`xmin` 是个优化:大多数事务号都比 `xmin` 小(早就提交了),不必逐个查 `xip[]`,直接判 `< xmin` 就可见。`subxip[]` 是子事务(subtransaction)的在跑列表,逻辑类似但针对子事务(第 14 章讲过的 savepoint 产生的子事务)。`suboverflowed` 表示子事务太多、数组装不下溢出了(此时只能保守处理)。

**第二组:命令级可见性——`curcid`**。`CommandId` 是事务内每条 SQL 的编号。同一个事务里,你自己刚 `INSERT` 的行,你下一条 `SELECT` 应该能看到——这个"自己的改动对自己可见"靠的就是 `curcid`:CID 比当前命令号小的(也就是这条命令之前产生的改动),对自己可见。这让一个事务内的多条 SQL 能看到自己前面的改动。

**第三组:缓存优化——`snapXactCompletionCount`**。这个字段是 PG 一个很巧的优化。每次有事务提交或回滚,全局计数器 `xactCompletionCount` 加 1。`GetSnapshotData()` 拍快照时记下当时的计数。下次再要拍快照时,先比一下计数:

> [src/backend/storage/ipc/procarray.c:2095-2135](../postgresql-17.0/src/backend/storage/ipc/procarray.c#L2095-L2135)

```c

{

	Assert(LWLockHeldByMe(ProcArrayLock));

	if (unlikely(snapshot->snapXactCompletionCount == 0))
		return false;

	curXactCompletionCount = TransamVariables->xactCompletionCount;
	if (curXactCompletionCount != snapshot->snapXactCompletionCount)
		return false;

	/*
	 * If the current xactCompletionCount is still the same as it was at the
	 * time the snapshot was built, we can be sure that rebuilding the
	 * contents of the snapshot the hard way would result in the same snapshot
	 * contents:
	 *
	 * As explained in transam/README, the set of xids considered running by
	 * GetSnapshotData() cannot change while ProcArrayLock is held. Snapshot
	 * contents only depend on transactions with xids and xactCompletionCount
	 * is incremented whenever a transaction with an xid finishes (while
	 * holding ProcArrayLock exclusively). Thus the xactCompletionCount check
	 * ensures we would detect if the snapshot would have changed.
	 */

```

逻辑是:**如果自从上次拍快照以来,没有任何事务结束(计数没变),那么"现在谁在跑"和上次一模一样,直接复用旧快照,不用重新扫一遍进程数组**。这在并发不激烈、或快照频繁拍的场景下(RC 每条语句都拍!)省下大量开销。这是 RC 虽然每条语句都拍快照、却仍很快的重要原因之一。

### 这张快照是怎么填出来的:`GetSnapshotData`

光有结构不够,得看它怎么被填出来。`GetSnapshotData()`(注意:实现在 `procarray.c`,不在 `snapmgr.c`)做的事是——**扫一遍当前所有正在运行的后端进程,收集它们的事务号,据此算出 `xmin`、`xmax`、`xip[]`**。

它最核心的两步:

> [src/backend/storage/ipc/procarray.c:2249-2260](../postgresql-17.0/src/backend/storage/ipc/procarray.c#L2249-L2260)

```c
	/* xmax is always latestCompletedXid + 1 */
	xmax = XidFromFullTransactionId(latest_completed);
	TransactionIdAdvance(xmax);
	Assert(TransactionIdIsNormal(xmax));

	/* initialize xmin calculation with xmax */
	xmin = xmax;
```

**`xmax` = 最近已完成事务号 + 1**。为什么?因为"最近已完成的事务 +1"就是"下一个即将开始的事务号"——比这个号大的事务,在我拍快照时还没开始,它们的改动我当然看不见。所以 `xmax` 就卡在"现在"这个时间点上。

然后 `xmin` 初始化为 `xmax`,再往下扫,凡是在跑的事务号比当前 `xmin` 小的,就把它降下来。这一段在循环里:

> [src/backend/storage/ipc/procarray.c:2322-2326](../postgresql-17.0/src/backend/storage/ipc/procarray.c#L2322-L2326)

```c
			if (NormalTransactionIdPrecedes(xid, xmin))
				xmin = xid;

			/* Add XID to snapshot. */
			xip[count++] = xid;
```

每个还在跑的事务,它的事务号进 `xip[]`(标记为"对我不可见"),同时如果它比当前 `xmin` 小,就把 `xmin` 拉到它这儿。循环走完,`xmin` 就是"所有在跑事务里最小的事务号"——比它还小的事务都已提交完(没在跑列表里),所以 `< xmin` 直接判可见。

最后把算好的值写进快照:

> [src/backend/storage/ipc/procarray.c:2503-2510](../postgresql-17.0/src/backend/storage/ipc/procarray.c#L2503-L2510)

```c
	snapshot->xmin = xmin;
	snapshot->xmax = xmax;
	snapshot->xcnt = count;
	snapshot->subxcnt = subcount;
	snapshot->suboverflowed = suboverflowed;
	snapshot->snapXactCompletionCount = curXactCompletionCount;

	snapshot->curcid = GetCurrentCommandId(false);
```

> **把这段和"快照=照片"的比喻对应起来**:`GetSnapshotData` 按下快门的那一瞬间,它锁住 `ProcArrayLock`(进程数组锁),扫一遍所有后端进程,把"现在谁在跑"这张照片定格。照片里:`xmax` 是快门按下的时刻(之后开始的事务,照片里没有),`xip[]` 是照片里那些正在跑的人(他们的改动对你不可见),`xmin` 是个加速查找的索引。一张照片拍完,以后无论外面世界怎么变(新事务开始、老事务提交),你拿的始终是这张照片——这就是 MVCC 让"读不阻塞写、写不阻塞读"的根基。

## 七、事务号从哪来:顺带讲清 xid 的分配

快照里全是事务号(xid),那事务号本身是怎么发的?补一句,免得读者卡在这里。

每个事务开始(且它确实需要写数据时)会从全局计数器领一个唯一的事务号。这发生在 `GetNewTransactionId()`:

> [src/backend/access/transam/varsup.c:77-108](../postgresql-17.0/src/backend/access/transam/varsup.c#L77-L108)

```c
GetNewTransactionId(bool isSubXact)
{
	...
	/* safety check, we should never get this far in a HS standby */
	if (RecoveryInProgress())
		elog(ERROR, "cannot assign TransactionIds during recovery");

	LWLockAcquire(XidGenLock, LW_EXCLUSIVE);

	full_xid = TransamVariables->nextXid;
	xid = XidFromFullTransactionId(full_xid);
	...
```

核心动作:**加 `XidGenLock` 排他锁 → 读取全局的 `nextXid` 作为本事务的 id → 后面会把它加 1,留给下一个事务**。`TransamVariables->nextXid` 是全局共享的事务号发生器,所有后端进程都从这里领号。注意领号要加排他锁——因为事务号必须全局唯一、单调递增。

> 这里有个 PG 的优化细节:只读事务(只 SELECT 不改数据)默认**不领 xid**——它没必要占一个号(它不产生任何别人需要看见的改动)。只有当事务真的要写(INSERT/UPDATE/DELETE)时,才"惰性"地领一个 xid。这减少了事务号的消耗(事务号是 32 位,会用完、会触发 wraparound,第 21 章 VACUUM 会讲)。所以前面 `GetSnapshotData` 扫进程数组时,很多进程的 xid 是 `InvalidTransactionId`(没领),会被跳过。

## 八、章末小结

### 用一段话回顾本章

隔离级别是**正确性与并发性能之间的权衡曲线**。SQL 标准把它切成四档:读未提交(RU)、读已提交(RC)、可重复读(RR)、可串行化(Serializable),每严一档就多挡住一种并发异常(脏读 → 不可重复读 → 幻读 → 序列化异常),但也要多付出并发代价。挡异常的核心机制是**快照(snapshot)**——事务进门时拍下的一张"已提交数据照片",决定它能看到哪些版本。RC 每条语句拍一张(能看到中途新提交的),RR/Serializable 整事务共用一张(中途新提交的看不见),这是它们的本质区别。PG 有三处偏离标准值得牢记:它的 RU 等同 RC(不做真脏读);它的 RR 是快照隔离 SI(比标准更严,挡幻读);它的 Serializable 用 SSI(乐观追踪读写依赖,挡写偏序,但代价最高、可能回滚)。

### 回扣主线

本章服务的是全书主线里"**让数据不丢不乱**"那一侧的"**不乱**"。隔离级别是"不乱"的尺度——**越严越不乱,但也越慢**。这是"正确性 vs 性能"权衡最典型的战场。具体到三个本性,隔离级别服务的是第二条——**"数据要共享,但共享会冲突"**:并发事务就是共享,隔离级别就是给这种共享定规矩,让冲突不至于破坏数据一致性。

PG 默认选 RC 而不是 RR,是**性能优先**的选择(回扣"快"那一侧):RC 的快照生命周期短,系统包袱小。但 RC 会让同一事务里的两次读到不同的值——应用如果需要"事务内数据稳定"(对账、审计、基于读做条件写),就得主动升到 RR 或 Serializable。这永远是一道业务驱动的选择题,没有银弹。
