# 《数据库内核:从一条 SQL 说起》—— 第 6 篇 · 进阶选讲

> 本篇是"按精力决定"的进阶三章。前五篇把"一条 SQL 的旅程"从进门讲到落盘、从并发讲到恢复,主线已经收束。本篇不引入新的主线,只挑前五篇留下的**三个尾巴**各写一章:**VACUUM 收 MVCC 的尾(P4 留下的死元组与事务号回绕)、复制把"不丢"从单机扩到机房(P5 单点故障的解药)、并行查询把单核到顶的瓶颈交给多核(P1 执行器的延伸)**。三章相对独立、可选读,各自回扣前五篇的某一处伏笔。

## 一、篇定位

> 本篇三章,每一章都是对前面某一篇结尾那个"钩子"的回收——
>
> - **第 21 章 VACUUM** 回扣 P4 第 17 章《MVCC》结尾留下的钩子:**MVCC 的旧版本(dead tuple)和会回卷的 32 位事务号,谁来收拾?**
> - **第 22 章 复制** 回扣 P5 三章结尾的那句话:**单机本身是单点——整机损毁,WAL 和数据页同归于尽,P5 的全部努力归零。** 复制是它的解药。
> - **第 23 章 并行查询** 回扣 P1 第 5 章《执行器》的火山模型:**一棵 Plan 树从头到尾只有一个进程在跑,32 核机器上 31 个核晾着——能不能切开分给多个进程?**

## 二、篇内分章清单

### 第 21 章 · VACUUM:MVCC 的代价与死元组清理

- **核心问题**:第 17 章讲清了 MVCC 为什么"读不阻塞写"——靠的是 UPDATE/DELETE 不就地改、保留旧版本。但旧版本(dead tuple)会无限堆积,PG 的 32 位事务号又会回绕(wraparound)。**这两件事数据库自己不会自愈——谁负责打扫?答案是 VACUUM。** 它是 MVCC 的内置收尾,不是可选运维。
- **要讲清的"为什么"**:
  - 为什么 PG 比 MySQL InnoDB 更需要显式清理(InnoDB 靠 undo 链即时 purge,PG 走"简单写路径 + 延迟扫表清理"的另一条路)。
  - 从不 VACUUM 的两笔账:**慢(表膨胀 bloat)** 和 **致命(事务号回绕让数据"凭空消失")**。
  - XID wraparound 这场 PG 最怕的灾难的完整因果链,以及"四道闸 + 强制只读 + anti-wraparound VACUUM 不认配置"三层防线。
  - 一次 lazy VACUUM 实际做的六件事(prune / freeze / 更新 VM / 清索引 / 清堆 / 推进 relfrozenxid),以及为什么必须"先清索引再清堆"的铁律。
  - lazy VACUUM 与 VACUUM FULL 的取舍(一个不锁表不还空间、一个锁表重写还空间)。
  - autovacuum 的阈值公式(`threshold + scale_factor × reltuples`)为什么是"基数 + 比例"的混合。
  - page-level prune 与 HOT update 作为 VACUUM 的"前哨",怎么让高频 UPDATE 表的死元组基本不堆积。
- **关键源码**(均已在前文各章引用处核验):
  - 单表入口 [`heap_vacuum_rel`](../postgresql-17.0/src/backend/access/heap/vacuumlazy.c#L284-L296) 与 workhorse [`lazy_scan_heap`](../postgresql-17.0/src/backend/access/heap/vacuumlazy.c#L780-L815)(六件事的总纲注释)。
  - 命令入口 [`ExecVacuum`](../postgresql-17.0/src/backend/commands/vacuum.c#L147)、[`vacuum`](../postgresql-17.0/src/backend/commands/vacuum.c#L478-L479)(含"不能在事务块里跑"的约束 L491-L498)、跨表收尾 [`vac_truncate_clog`](../postgresql-17.0/src/backend/commands/vacuum.c#L1766-L1786)。
  - autovacuum launcher/worker 与阈值公式 [`relation_needs_vacanalyze`](../postgresql-17.0/src/backend/postmaster/autovacuum.c#L3034-L3055)、anti-wraparound 的 `force_vacuum` 判定 [autovacuum.c:2989-3014](../postgresql-17.0/src/backend/postmaster/autovacuum.c#L2989-L3014)。
  - 事务号回绕的四道闸 [`GetNewTransactionId`](../postgresql-17.0/src/backend/access/transam/varsup.c#L123-L167)、强制只读报错 [varsup.c:1009-1021](../postgresql-17.0/src/backend/access/transam/varsup.c#L1009-L1021)、`SetTransactionIdLimit` 的阈值公式 [varsup.c:389-406](../postgresql-17.0/src/backend/access/transam/varsup.c#L389-L406)。
  - 页内清理 [`heap_page_prune_opt`](../postgresql-17.0/src/backend/access/heap/pruneheap.c#L180-L246)、冻结判定 [`heap_prepare_freeze_tuple`](../postgresql-17.0/src/backend/access/heap/heapam.c#L6550)。
- **难度 / 阅读提示**:★★★★(偏难)。**前置依赖**:第 17 章《MVCC》(xmin/xmax/冻结/快照)、第 9 章《FSM 与 VM》(VM 的 `ALL_VISIBLE`/`ALL_FROZEN` 两位)、第 7 章《堆表与元组》(行指针状态 `LP_NORMAL`/`LP_DEAD`/`LP_UNUSED`)。本章是 P4 第 17 章的直接续章,没读 17 章不要先读这里。
- **读完你会明白**:为什么"装了 PG 就别关 autovacuum"是第一条运维铁律;XID wraparound 为什么是 PG 最怕的灾难、PG 怎么用三层防线挡住它;以及为什么"VACUUM 不删数据、不还空间给 OS"——它只是把死元组的槽位标成可复用。

### 第 22 章 · 复制与高可用:流复制、主从一致性

- **核心问题**:P5 三章把单机的"易失"治住了(WAL 兜底、checkpoint 划 redo 锚点、崩溃恢复重放)。但单机本身是**单点**——整机损毁,WAL 和数据页同归于尽。**怎么把"不丢"从一台机器扩展到一个机房?** PostgreSQL 的答案是**物理流复制**:把主机产生的同一份 WAL,流式搬到另一台机器上去重放。
- **要讲清的"为什么"**:
  - 复制的是 WAL(物理日志、字节级重放)而不是 SQL——为什么这样最简单一致。
  - **备机 apply 的本质就是崩溃恢复**(WAL 来源从本地文件换成网络),这是物理复制"简单且一致"的根因。
  - 异步复制 vs 同步复制:异步的"数据丢失窗口"长什么样、同步怎么消除它、各自的代价(延迟、备机挂了主也卡)。
  - `synchronous_commit` 五档同步级别(off / local / remote_write / on / remote_flush / remote_apply)的精确语义。
  - 复制槽(replication slot)为什么必不可少——钉住 WAL 删除边界,断线备机重连还能追上;以及它的反面风险(备机永不回来则 WAL 堆积)。
  - 主从切换(failover)为什么有"脑裂"风险,为什么 PG 物理复制本身不防脑裂、要靠外部 HA 软件 + fencing。
  - 物理备机为什么必须只读、必须同版本同架构;逻辑复制能做什么、代价是什么。
- **关键源码**(均已在前文各章引用处核验):
  - 备机进 standby 模式的信号文件判定 [`readRecoverySignalFile`](../postgresql-17.0/src/backend/access/transam/xlogrecovery.c#L1056-L1094)。
  - 全量基线 [`perform_base_backup`](../postgresql-17.0/src/backend/backup/basebackup.c#L234)。
  - 同步复制等待 [`SyncRepWaitForLSN`](../postgresql-17.0/src/backend/replication/syncrep.c#L148-L180)、反馈唤醒 [syncrep.c:508-521](../postgresql-17.0/src/backend/replication/syncrep.c#L508-L521)、三档水位常量 [syncrep.h:22-27](../postgresql-17.0/src/include/replication/syncrep.h#L22-L27)。
  - 备机反馈消息构造 [walreceiver.c:1140-1146](../postgresql-17.0/src/backend/replication/walreceiver.c#L1140-L1146)。
  - 复制槽的 `restart_lsn` [slot.h:92-93](../postgresql-17.0/src/include/replication/slot.h#L92-L93)、`ReplicationSlotsComputeRequiredLSN` [slot.c:1105-1138](../postgresql-17.0/src/backend/replication/slot.c#L1105-L1138)。
  - WAL 段回收联动复制槽 [`KeepLogSeg`](../postgresql-17.0/src/backend/access/transam/xlog.c#L7931-L7946)(含 `max_slot_wal_keep_size` 兜底)。
  - walsender 主循环 [`WalSndLoop`](../postgresql-17.0/src/backend/replication/walsender.c#L2780-L2865)、`XLogSendPhysical` 的"只发已落盘 WAL"铁律 [walsender.c:3158-3170](../postgresql-17.0/src/backend/replication/walsender.c#L3158-L3170)。
- **难度 / 阅读提示**:★★★★(偏难)。**前置依赖**:第 18 章《WAL》、第 19 章《Checkpoint》、第 20 章《崩溃恢复》(尤其第 20 章——本章反复回扣"备机 apply = 崩溃恢复的 redo")。没读 P5 三章不要先读这里。
- **读完你会明白**:物理流复制是"主机 walsender 流式发 WAL、备机 walreceiver 写盘、startup 进程持续 apply"的三方协作;同步复制的代价是延迟升 + 备机挂了主也卡;复制槽是断线备机的命脉、用完必须 drop;以及为什么"复制 + WAL + VACUUM"三者合起来才是"让数据不丢"的完整答案。

### 第 23 章 · 并行查询:多核一起跑一条 SQL

- **核心问题**:一条 SQL,凭什么能用上多个 CPU 核一起跑?为什么 PostgreSQL **默认就不开并行**——同样的查询,小表上并行反而比串行还慢?优化器怎么判断"这次并行划算不划算",又是怎么把一个单进程的火山模型计划树改造成"几个 worker 各跑一份 + 一个 Gather 节点收拢"的形态的?
- **要讲清的"为什么"**:
  - 并行的根本动机:2005 年后单核到顶、核数越来越多,单进程执行把多核晾着是浪费——但只对**大查询**值得。
  - 并行有三类固定 overhead(启动 worker 进程、建共享内存 DSM、tuple 跨进程传递),所以对 OLTP 小查询是**纯亏**,PG 默认不并行。
  - 代价模型给并行加了两个常数(`parallel_setup_cost = 1000`、`parallel_tuple_cost = 0.1`),并行计划必须比串行更省才会被选。
  - 怎么并行:leader + N 个 worker + 共享内存队列 shm_mq + Gather / GatherMerge 节点;worker 数由 `compute_parallel_worker` 按表大小对数增长决定。
  - 并行安全三级标签(SAFE / RESTRICTED / UNSAFE),为什么写操作(CRUD)天生不能并行——它们碰的是全局共享状态。
  - 三道 GUC 闸门(`max_worker_processes` / `max_parallel_workers` / `max_parallel_workers_per_gather`)怎么把单查询并行度死死限住。
- **关键源码**(均已在前文各章引用处核验):
  - Gather 节点的拉取主循环 [`ExecGather`](../postgresql-17.0/src/backend/executor/nodeGather.c#L137-L209)、`gather_getnext` [nodeGather.c:256-298](../postgresql-17.0/src/backend/executor/nodeGather.c#L256-L298)、`gather_readnext` 的非阻塞轮询 [nodeGather.c:303-345](../postgresql-17.0/src/backend/executor/nodeGather.c#L303-L345)。
  - 并行代价常数 [cost.h:24-30](../postgresql-17.0/src/include/optimizer/cost.h#L24-L30)、`cost_gather` [costsize.c:435-461](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L435-L461)、`cost_gather_merge` [costsize.c:463-528](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L463-L528)、SeqScan 的并行分摊 [costsize.c:327-347](../postgresql-17.0/src/backend/optimizer/path/costsize.c#L327-L347)。
  - worker 数启发式 [`compute_parallel_worker`](../postgresql-17.0/src/backend/optimizer/path/allpaths.c#L4203-L4279)、并行否决检查 `set_rel_consider_parallel` [allpaths.c:607-651](../postgresql-17.0/src/backend/optimizer/path/allpaths.c#L607-L651)。
  - 共享内存队列 `shm_mq_send` [shm_mq.c:329-338](../postgresql-17.0/src/backend/storage/ipc/shm_mq.c#L329-L338)、`shm_mq_receive` [shm_mq.c:571-573](../postgresql-17.0/src/backend/storage/ipc/shm_mq.c#L571-L573)。
  - 并行基础设施 `ExecInitParallelPlan` [execParallel.c:586-629](../postgresql-17.0/src/backend/executor/execParallel.c#L586-L629)、`LaunchParallelWorkers` [parallel.c:552-606](../postgresql-17.0/src/backend/access/transam/parallel.c#L552-L606)、worker 入口 `ParallelWorkerMain` [parallel.c:1271-1299](../postgresql-17.0/src/backend/access/transam/parallel.c#L1271-L1299)。
  - 并行安全 `max_parallel_hazard` [clauses.c:722-743](../postgresql-17.0/src/backend/optimizer/util/clauses.c#L722-L743)、`PROPARALLEL_SAFE/RESTRICTED/UNSAFE` 常量 [pg_proc.h:169-175](../postgresql-17.0/src/include/catalog/pg_proc.h#L169-L175)。
  - 三道 GUC 闸门 [guc_tables.c:3409-3429](../postgresql-17.0/src/backend/utils/misc/guc_tables.c#L3409-L3429)、`min_parallel_table_scan_size` 默认 8MB [guc_tables.c:3510-3519](../postgresql-17.0/src/backend/utils/misc/guc_tables.c#L3510-L3519)。
- **难度 / 阅读提示**:★★★★(偏难)。**前置依赖**:第 5 章《执行器》(火山模型 / `ExecProcNode` 是本章全部地基)、第 4 章《查询优化器》(并行要不要开、开几个 worker 是优化器在 costsize.c 里比代价比出来的)。本章没有另起执行框架,而是给火山模型多挂一个 Gather 算子,所以必须先理解第 5 章。
- **读完你会明白**:并行查询不是"能并行就并行",而是"划算才并行"——这个划算由优化器的代价模型量化;它复用了第 5 章的全部算子,只是把计划树下段切成几份、由 leader 用 Gather 节点合并;以及为什么写操作天生不并行。

## 三、篇内衔接线索:三章各自补一处尾巴

> 三章虽然相对独立,但顺着读会发现一条隐线——**每一章都在把前五篇某一处"到此为止"的边界往外推一格**。

```text
   │  尾巴 1:旧版本堆积 +   │         │  尾巴 2:单机本身是单点——    │

        第 21 章 VACUUM                    第 22 章 复制
   "收 MVCC 的两笔账"                  "把 WAL 搬到备机重放——
   (死元组 + wraparound)                备机 apply 就是崩溃恢复"
               │                                    │
               └──────────────┬─────────────────────┘
                              │
                              ▼
                   前五篇主线已收束,但 P1 执行器留了第三个尾巴:
                   "一棵 Plan 树只有一个进程在跑,多核被晾着"
                              │
                              ▼

                   leader + worker + Gather 合并"
                              │

                   全书主线"一条 SQL 的旅程"收尾:
            进门 → 解析 → 优化 → (并行)执行 → 存储 → 索引
                  → 事务并发 → 持久恢复 → 进阶补尾
```

**衔接的关键回扣点**:

- **VACUUM → MVCC**(第 21 → 第 17 章):第 17 章讲 MVCC 用"留旧版本"换"读写不阻塞",第 21 章还这笔账——旧版本谁来清、32 位事务号回绕怎么防。**没有 VACUUM,MVCC 用不了几次就把数据库拖垮;有了 VACUUM,MVCC 才能长期可持续运转**。VACUUM 是 MVCC 的闭环。
- **复制 → 崩溃恢复**(第 22 → 第 20 章):第 22 章反复强调"**备机 apply 的本质就是崩溃恢复**"——备机的 startup 进程和主机崩溃重启后跑的 startup 是同一套 redo 代码,只是 WAL 来源从本地 `pg_wal/` 换成网络。所以读第 22 章前必须读懂第 20 章,否则看不清"为什么物理复制这么简单一致"。
- **并行 → 执行器**(第 23 → 第 5 章):第 23 章明确说**并行没有另起一套执行框架**,而是给火山模型多挂一个叫 Gather 的算子——Gather 节点对父节点就是个普通的 `ExecProcNode` 来源,父节点完全不知道底下是 N 个进程在并行干活。这是 PG 并行设计最优雅的地方,也是为什么第 5 章的火山模型是本章全部地基。

> 三章合起来,还呼应了全书主线"**快 vs 不丢不乱**"的二分法:**VACUUM 同时服务两侧**(清死元组让查询不读垃圾=快,冻结老元组防回绕=不乱);**复制专攻"不丢不乱"**(把持久性承诺从单机磁盘延伸到另一台机器,同步复制连"不乱"也兜住);**并行专攻"快"**(单核到顶的 CPU 瓶颈靠多核打破),但它处处被"不乱"约束(默认不开避免拖慢小查询、worker 限数避免占满机器、写不并行避免破坏共享状态)。

- **第 23 章 并行查询**:必读 **P1 第 5 章《执行器》**(火山模型),辅读 **第 4 章《查询优化器》**(并行决策是代价模型的一个应用)。**没读懂火山模型不要先读这里。**

### 推荐阅读顺序

精力充沛的读者建议按 **21 → 22 → 23** 的顺序读,正好沿着"MVCC 的代价 → 单点故障 → 单核瓶颈"这条隐线走完,最后用第 23 章收束全书"一条 SQL 的旅程"。精力有限的读者,可以只挑与当前工作最相关的一章——**DBA 必读 21(VACUUM)和 22(复制),做 OLAP / 分析查询的工程师优先读 23(并行)**。

### 与全书主线的关系

本篇不引入新主线,只是把全书"**快 vs 不丢不乱**"二分法在三个进阶维度上各自兑现一次:**VACUUM 让 MVCC 长期可持续(不乱)、复制让单机不再是致命点(不丢)、并行让大查询用满多核(快)**。读完本篇,你就走完了"一条 SQL 的完整一生"——从进门、被理解、被规划、被(并行)执行,到数据被高效存储、被索引加速、在并发中保持一致、在掉电和崩溃中幸存、在整机损毁时被复制备份、在大查询时被多核加速。**附录**会教你两件事:怎么自己开始读 PG 源码,以及怎么用 `EXPLAIN`、`pg_stat_*`、`pageinspect`、`pg_waldump` 这些工具把本书讲的每一条原理在你自己的机器上亲眼看见。
