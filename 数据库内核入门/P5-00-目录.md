# 《数据库内核:从一条 SQL 说起》—— 第 5 篇 · 持久性与崩溃恢复

> **篇定位**:本篇治数据的"**易失**"本性——内存里的改动怎么保证掉电不丢。它服务全书主线"**快 vs 不丢不乱**"里"**不丢不乱**"那一侧:用日志的持久换内存的易失,把"已承诺给用户的 `COMMIT`"在崩溃后真兑现。三章合成一个闭环:**WAL 先记日志(持久性的根基)→ Checkpoint 划定 redo 锚点(让 WAL 不无限涨、恢复不无限久)→ 崩溃恢复从锚点重放重建(只 redo 不 undo)**。

### 第 19 章 · Checkpoint:日志与数据的同步点

- **核心问题**:为什么数据库必须周期性地做 checkpoint?它到底干了哪几件事、为什么干这些事就足以让"崩溃后恢复"既快又对?checkpoint 本身是个把所有脏页刷盘的"重活",怎么避免它一启动就把磁盘 I/O 打满、把查询拖卡?checkpoint 的节奏(多久一次、刷多快)怎么调,调偏了各有什么代价?
- **读完你会明白**:checkpoint 为何是"WAL 与数据的同步锚点"——它决定了崩溃后要从 WAL 的哪个位置开始重放(redo 起点为什么必须存进 checkpoint 记录);一次 checkpoint 在源码里的完整生命周期(定 redo 点 → 刷所有脏页 → fsync 数据文件 → 写一条 checkpoint WAL 记录并 fsync → 更新 ControlFile → 截断/回收旧 WAL 段);spread checkpoint(配合 `checkpoint_completion_target` 默认 0.9)怎么把刷脏页摊到整个时间窗里、避免 I/O 尖峰;bgwriter 和 checkpointer 怎么分工把脏页平滑落盘;调 `checkpoint_timeout` / `max_wal_size` 的代价权衡。

### 第 20 章 · 崩溃恢复:从日志重建一致性

- **核心问题**:数据库崩溃重启后,内存里的脏页(没落盘的改动)没了——怎么靠已经落盘的 WAL 把它们"变回来"?为什么 PostgreSQL 的崩溃恢复是"**只 redo、不 undo**"——它不需要把未提交事务的改动撤销?这条 redo 路径在源码里是怎么走的:从读 ControlFile 找最近 checkpoint、取出 redo 点、逐条重放 WAL 到末尾?重放时怎么保证"幂等"(重放已经部分应用过的记录不会算错)?
- **读完你会明白**:崩溃恢复的本质——从最近 checkpoint 的 redo 起点开始,逐条重放 WAL 到末尾,把没落盘的改动补回数据页;为什么 PG 只要 redo、不要 undo(MVCC 让未提交事务的改动对别人本来就不可能可见,abort 只是 `pg_xact` 打标记、不擦物理改动);完整的恢复流程(读 ControlFile → 找最近 checkpoint 记录 → 取 redo 点 → 循环读 WAL record → 按 `xl_rmid` 分发到对应 `rm_redo` → 遇到 commit/abort 记录处理事务状态 → 恢复 2PC prepared 事务);redo 幂等性怎么保证(LSN 比较:`page->pd_lsn >= record->lsn` 就跳过),以及第 18 章埋的 FPI 为什么是这套幂等性能成立的前提(防 torn page);为什么崩溃恢复重放到 WAL 末尾就能开门,而备机 apply 是同一套代码。

## 阅读建议

### 必须顺序:WAL → Checkpoint → 崩溃恢复

本篇三章是**严格的逻辑接力链**,严禁跳读:

- **第 18 章 WAL** 立铁律"先记日志后改数据",定义了 `XLogRecord` / rmgr / LSN / FPI 这些全套术语。第 19、20 章全建立在它们之上。
- **第 19 章 Checkpoint** 收第 18 章的钩子(脏页怎么落盘、WAL 怎么截),焊下 redo 起点,把它写进 ControlFile。第 20 章开局就是读 ControlFile 找这个起点。
- **第 20 章 崩溃恢复** 收第 18、19 章的所有钩子,从 redo 起点重放 WAL。`XLogReadBufferForRedo` 的 LSN 比较、FPI restore,都是第 18 章概念在恢复侧的兑现。

跳过 18 直接读 19,你不知道 redo 起点为何物;跳过 19 直接读 20,你不知道 ControlFile 里的 `checkPoint.redo` 是谁写的、为什么可信。

### 精华在三处

1. **第 18 章的 WAL 铁律落地**:"先记日志后改数据"不是道德约束,而是"脏页刷盘前先 `XLogFlush(pd_lsn)`"这个 LSN 比较 + 一次 fsync。这是持久性根基。
2. **第 19 章的六件事 + spread checkpoint**:一次 checkpoint 从定 redo 点到回收旧 WAL 段的完整链条;`checkpoint_completion_target = 0.9` 把刷盘摊匀避免 I/O 风暴——这是生产 DBA 最该吃透的设计。
3. **第 20 章的"只 redo 不 undo"**:PG 崩溃恢复的灵魂。MVCC 让未提交事务的改动天然不可见,所以 abort 只是 `pg_xact` 打标记、不擦物理改动——这把 InnoDB 那套 undo 日志整个省掉了,代价是旧版本堆积交给 VACUUM(下一篇第 21 章)。这是全书"快 vs 不丢不乱"二分法最精彩的体现之一。

### 前置依赖

- **必须读过**:第 1 章(数据的"易失"本性,以及"四个乱子"里"写一半掉电,数据半残"那个乱子)、第 8 章《Buffer Pool》(脏页从哪来——内存里的页改了之后是"脏"的,迟早要落盘)、第 14 章《事务与 ACID》(那一章 `RecordTransactionCommit` 写完 commit record 后调 `XLogFlush` 等落盘——第 18 章把这一步彻底讲透)。
- **强烈建议读过**:第 6 章《页面 Page》(页头的 `pd_lsn`,这是 LSN 体系的物理载体)、第 17 章《MVCC》(xmin/xmax、旧版本仍在——第 20 章"为什么只 redo 不 undo"的答案全在 MVCC 里)。
- **顺带衔接**:本篇是第 22 章《复制与高可用》的前奏——第 20 章末尾点明"备机 apply 就是崩溃恢复,同一套 `PerformWalRecovery` 主循环、同一套 `CheckRecoveryConsistency`",只是 WAL 来源(本地文件 vs 网络)和停止条件不同。读完本篇再读 P6 第 22 章,复制的一致性保证就顺理成章。
