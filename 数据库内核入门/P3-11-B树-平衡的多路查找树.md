# 第 11 章 · B 树:平衡的多路查找树

> **前置**:你需要先读过[第 10 章《为什么需要索引》](P3-10-为什么需要索引-O(n)与O(logn)的较量.md)——它立了"索引为何存在":把查找从 O(n) 降到 O(log n),并预告了 B 树这个默认角色;以及[第 6 章《页面 Page》](P2-06-页面Page-磁盘IO的最小单位.md)——它讲了 8KB 页的布局,本章会反复用到"页、`pd_special`、行指针"这些术语。本章钻进 PostgreSQL 默认索引 **B-tree** 的内部:看它凭什么扛起 O(log n),以及为什么是它,而不是二叉搜索树、红黑树、哈希表。

> **核心问题**:为什么 PG 默认索引是 B 树,而不是二叉搜索树 / 红黑树 / 哈希表?核心在三个字——**多路(multi-way)+ 平衡 + 页对齐**:每个节点正好是一页(8KB),一页能装上百个有序键,树高 = 页层数,3~4 层就能覆盖亿级行,每层只一次页 I/O。为什么必须"平衡"(否则退化成链表,O(n))?页满了怎么**分裂(split)**、空了怎么回收(delete/recycle),分裂时又怎么**自底向上**把分裂一路传播到父页直到根(根一分裂,树就长高一层)?PG 的 B 树在教科书 B+ 树之上做了哪些增强——**高键(highkey)** 让并发分裂不出错、**TID 去重(dedup)** 压缩重复键、**包含列(INCLUDE)** 与**增量排序**——这些增强各自为哪类负载而生?

> **读完本章你会明白**:
> - 为什么"多路 + 页对齐"是 B 树的灵魂——把 1000 万行从二叉树的 23 层压到 B 树的 3 层,每层 = 1 次磁盘页 I/O,差距碾压。
> - 一棵 PG B 树的物理形态:根页 / 枝页 / 叶页三种角色,叶页存(有序键 + TID)、枝页存路由键(pivot)与 downlink、所有叶页用 `btpo_next` 串成双向链表支持范围扫描。
> - 一次**点查询**的根到叶路径(`_bt_search`)与一次**范围扫描**如何沿叶链表走。
> - 页满了为什么是**自底向上分裂**、根分裂为何等于树长高一层、分裂点为什么默认按 fillfactor(叶 90 / 内部 70)选。
> - PG 在教科书 B+ 树之上加的几把刷子:高键 / 去重 / 包含列 / 增量排序。

> **如果一读觉得太难**:先只记三件事——① **B 树 = 多路 + 平衡 + 页对齐**:一个节点就是一页,所以"树高几层"等于"要读几个页",3~4 层搞定亿级行。② 为什么不是二叉树:二叉树一节点两路,1000 万行要 23 层 = 23 次页 I/O;B 树一节点上百路,只要 3 层。③ 页满了就**分裂成左右两半**,分裂会向上传,传到根就长高一层——这保证了永远平衡。其余 dedup、高键、`_bt_binsrch` 二分的细节,第二遍配合源码再抠,不影响理解后续章节。

## 一、为什么不是二叉树:多路 + 页对齐是灵魂

第 10 章结尾说,B 树是"平衡的**多路**查找树",扇出 F 典型 100~200,所以树高 h ≈ log_F(n)。但那只是把结论摆出来。本章要回答更根本的问题:**为什么必须"多路"?为什么 PG 偏偏选 B 树,而不是数据结构课上学的二叉搜索树(BST)、平衡二叉树(AVL / 红黑树),或者哈希表?**

答案的根,在"磁盘 I/O 的最小单位是页"这件事上(第 6 章讲过)。这逼出了 B 树最关键的设计:**让树的每一个节点,正好等于一个页。**

### 不这样会怎样:二叉树的层数 = 行数的对数,层太多 = I/O 太多

先做一个"算账"。假设有一张 1000 万行的表,在某个列上建索引。

**如果用二叉搜索树(BST,或自平衡的 AVL / 红黑树)**:每个节点只有两个子指针(左、右),所以扇出 F = 2。树高 h ≈ log₂(n) = log₂(10,000,000) ≈ **23 层**。在内存里,23 次指针跳转是纳秒级,无所谓;但索引可能比内存大得多,大部分节点在磁盘上——**每往下一层,就是一次"读一个页"的 I/O**(因为节点分散在堆表之外的索引文件里,跳到子节点往往要现读那一页)。23 层 = **23 次随机页 I/O**。按机械盘一次随机 I/O 约 5~10 毫秒算,单次点查询要 100~200 毫秒——和顺序扫描没拉开数量级的差距,索引白建了。

更扎心的对比:红黑树一个节点通常存一个键 + 两个指针 + 颜色位,二三十字节。一个 8KB 页能塞下二三百个这样的节点。但二叉树**根本不利用页**——它的逻辑结构是"一个节点一个键",节点之间靠指针乱跳,这些节点在物理上可能散落在任意页里。你为了读一个节点,常常要把整个 8KB 页读进来(磁盘最小粒度就是页,第 6 章),读完只用到其中二三十字节——**页里另外 99% 的数据全浪费了**。

> **不这样会怎样**:二叉树有两个致命问题叠加——① 层数太多(log₂ n,1000 万行 23 层),每层一次 I/O;② 节点和磁盘页粒度不匹配,一次 I/O 读一整页却只用一个键,浪费巨大。两个问题合起来,二叉树在"数据量远大于内存、存在磁盘上"的场景里,是个糟糕透顶的索引结构。

红黑树、AVL 这些"自平衡二叉树"只解决了"别退化成链表"(下面会讲),但**没解决"层太多"和"节点不匹配页"**——它们是为纯内存设计的。

### 所以:B 树让一个节点装满一整页,把扇出拉到上百

B 树的解法直球:**既然磁盘一次最少读一个页,那就让一个树节点正好占一个页,并且在一页里尽可能多地塞键。** 一页 8KB,一个索引项(键 + 子指针 / TID)算它几十到一百字节,一页能塞 **100~200 个键**。这意味着一个节点有 100~200 路分叉(扇出 F ≈ 100~200)。

再算一次树高:h ≈ log_F(n) = log₁₀₀(10,000,000) = log(10⁷) / log(10²) = 7/2 = **3.5 层**,取整 **4 层**。把上一章那张表填满:

| 表行数 n | 二叉树树高 log₂ n | **B 树树高 log_F n (F=100)** | B 树点查询的索引页 I/O |
|---|---|---|---|
| 1 万 | 14 | 2 | 2 |
| 1 百万 | 20 | 3 | 3 |
| **1 千万** | **23** | **4** | **4** |
| 1 亿 | 27 | 4 | 4 |
| 10 亿 | 30 | 5 | 5 |

这张表是本章的灵魂。看右边两列:**B 树的树高几乎不随数据量增长**——10 亿行也只要读 5 个索引页。而每读一个索引页就是一次 I/O(命中 Buffer Pool 的话连 I/O 都省了,第 8 章)。**树高 = 页层数 = I/O 次数**,这是 B 树把"找一行"压到微秒级的全部秘密。

> 这就是"多路 + 页对齐"两个词合在一起的力量:**多路**让一层能筛掉 99% 的候选(扇出 100,每往下一层剩下 1/100);**页对齐**让"访问一个节点"刚好等于"读一个页",不浪费任何 I/O。两个缺一不可——多路但不页对齐(节点塞不下一页或跨页),或者页对齐但只两路(退化成二叉树),都达不到这个效果。

### 顺带回答:为什么也不是哈希表?

哈希表点查询是 O(1),比 B 树的 O(log n) 还快,为什么 PG 默认不是它?三个硬伤:

1. **哈希只支持等值查询**(`=`)。`WHERE id = 42` 能用,但 `WHERE id > 42`、`WHERE id BETWEEN 10 AND 20`、`ORDER BY id`——哈希表全废。哈希把键打散到各个桶里,桶之间**无序**,没法做范围扫描、排序。而真实业务里范围查询和排序极常见。
2. **哈希不支持多列联合索引的前缀匹配**。`(a, b)` 上的 B 树能加速 `WHERE a = 1`(用前缀),也能加速 `WHERE a = 1 AND b > 5`;哈希只能整体哈希,`WHERE a = 1` 用不上。
3. **哈希冲突 + 扩容麻烦**。哈希桶满了要 rehash 整张表,是个大开销。

B 树因为键在页内**有序**,天然支持等值、范围、排序、前缀匹配——它是"全能选手"。PG 把它设为默认(`CREATE INDEX` 不写类型就是 B 树),正是因为绝大多数查询它都能扛。哈希索引(`CREATE INDEX ... USING HASH`)只在"纯等值、且值分布均匀"的窄场景才用得上(第 12 章会讲 PG 还支持它)。

> **一句话**:B 树赢在"有序 + 多路 + 页对齐"三合一。有序让它能范围扫描和排序;多路让它层数极少;页对齐让每次节点访问等于一次页 I/O,不浪费。

## 二、B 树长什么样:根、枝、叶三种页 + 叶链表

讲清了"为什么是 B 树",现在看它**物理上长什么样**。先上一张示意图(以一个 3 层的 B 树为例,键是数字):

```text
                         ┌─────────────────────────────┐
                 Level 2 │  根页 (root, 也是枝页)        │   ← 入口
                         │ [30] [60] [90]  (pivot 路由键)│
                         │  │    │    │                 │
                         │  ↓    ↓    ↓   (downlink 指向子页块号)
                         └──────┼────┼───┼─────────────┘
                   ┌────────────┘    │   └────────────┐
                   ▼                 ▼                 ▼
          ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
  Level 1 │  枝页 A       │  │  枝页 B       │  │  枝页 C       │
          │ [10][20][30] │  │ [40][50][60] │  │ [70][80][90] │
          │  │ │ │        │  │  │ │ │        │  │  │ │ │        │
          └──┼─┼─┼────────┘  └──┼─┼─┼────────┘  └──┼─┼─┼────────┘
             ▼ ▼ ▼              ▼ ▼ ▼              ▼ ▼ ▼
   叶页:  [1..30] ←next→ [31..60] ←next→ [61..90] ←next→ [91..120]
   Level 0 (leaf)
            └─ 叶页里存:(键值, TID) 对,按键有序 ─┘
            └─ btpo_next 把所有叶页串成→单向链表(向右);
               btpo_prev 串成 ← 反向链表(向左,支持反向扫描)
```

把这张图拆开看,有三种角色的页:

### 叶页(leaf page,Level 0):存(键值, TID)

叶页是 B 树的"最底层",真正存索引数据的地方。一个叶页里装着一组**按键有序排列的索引项**,每一项是:

> **(键值, TID)**

TID 就是第 6、7 章讲过的"块号 + 行指针偏移",指向堆表里的那一行。注意一个关键点(第 10 章预告过):

> **PG 的 B 树叶页,只存(键值, TID),不存整行。** 真正的完整数据只在堆表里有一份。索引拿到 TID 后,通常还要"回表"去堆里读那一行(第 13 章详讲回表与怎么免回表)。

这个设计是 PG 索引"二级结构"的体现——堆表和索引分离,索引只记路标。后面会看到,它和某些引擎(MySQL InnoDB 聚簇索引)的区别,直接决定了 PG 的回表路径。

### 枝页 / 内部页(internal page,Level ≥ 1):存路由键 + downlink

枝页(包括根页,如果树高于一层)不存实际数据,只存**路由信息**:一组**路由键(pivot tuple)** + 每个键下面挂一个 **downlink(指向子页的块号)**。查找时,在枝页里比较"我要找的键落在哪个路由键的左边/右边",从而决定往下走哪个子页。

注意枝页里的路由键是**截断的(truncated)**——它不需要存完整的键值,只需要存"够区分往左还是往右"的那部分(下面讲高键和 suffix truncation 时细说)。这让枝页能塞下更多路由键 = 更大的扇出 = 更矮的树。

### 根页(root page):特殊的枝页,也是树的入口

根页是**最顶层**的页,只有一个。它的特殊在于:**它是所有查找的起点**,而且它"没有父页"——别的页满了分裂,分裂信息往上传给父页;但根页分裂时没有父页可传,这时要**新建一个根**(下面讲分裂时会详细说,这是树长高的唯一时刻)。根页的块号记在一个叫**元页(metapage)**的特殊页里(永远是第 0 页),这样每次查找都能找到根。

### 叶链表:用 `btpo_next` / `btpo_prev` 把叶页串起来

这是 B 树(准确地说是 **B+ 树**——PG 用的是 B+ 树变体,所有数据只在叶页,枝页只路由)最实用的一个结构特征:**所有叶页,从左到右用指针串成一条双向链表。**

每个叶页的页尾(在 special 区,第 6 章讲过 `pd_special`)存着两个兄弟指针:`btpo_prev`(左兄弟页块号)和 `btpo_next`(右兄弟页块号)。这条链表是**范围扫描的生命线**:

- 点查询 `WHERE id = 42`:从根走到叶,在一个叶页里二分找到 42,结束。
- 范围查询 `WHERE id BETWEEN 10 AND 80`:从根走到叶,定位到"第一个 ≥ 10"的位置,然后**沿着 `btpo_next` 一路向右走**,边走边把满足 ≤ 80 的项吐出去,直到超过 80。叶页之间不必再回到根——**链表让范围扫描变成顺序地走一串页**,极其高效(尤其配合 Buffer Pool 的预读)。

反向扫描(`ORDER BY id DESC`)则沿着 `btpo_prev` 向左走。

> 这条叶链表,是 B 树"既快又全能"的另一半理由:多路 + 平衡让点查询 O(log n);叶链表让范围查询 O(命中行数 + 叶页数),且是顺序 I/O。哈希表永远做不到范围扫描,根因就是它没有这条有序链表。

## 三、为什么必须平衡:不平衡就退化成 O(n) 的链表

上面一直说"B 树是平衡的"——从根到任何叶页的路径长度都一样(等于树高 h)。但为什么必须平衡?不平衡会怎样?

### 不这样会怎样:二叉搜索树按有序数据插入,退化成链表

经典反例是朴素的二叉搜索树(BST)。BST 的规则是:小的往左、大的往右。如果你**按递增顺序插入**键(比如自增主键 `id = 1, 2, 3, 4, ...`),每个新键都比所有已有的都大,于是它一路往右走,最后挂在最右下角——整棵树退化成一条**向右歪的链表**:

```text
    1
     \
      2
       \
        3
         \
          4   ← 树高 = n,查找 O(n)
```

这棵"树"的高度等于节点数,查找一个键要走的层数 = 它在序列里的位置——平均 n/2,最坏 n。**退化成链表 = 退化回 O(n),索引形同虚设。** 这就是"不平衡"的灾难:数据插入顺序如果碰巧有序(而自增主键、时间戳这种业务极常见,本身就是递增的!),BST 就废了。

红黑树、AVL 树就是为治这个病发明的——它们在插入/删除后做"旋转",强制保持"任意节点的左右子树高度差不超过 1",从而保证树高 = O(log n)。但如第一节所说,它们是二叉的,扇出只有 2,层数还是 log₂ n,在磁盘 I/O 层面层数太多。

### B 树怎么保持平衡:分裂向上传,根一分裂就整体长高一层

B 树用一种更彻底的方式保证平衡:**所有叶页到根的距离都严格相等。** 怎么做到?靠"分裂向上传播"这个机制。

- B 树的每一页,数据是**按键有序**塞的。新键插入时,找到它该在的叶页,塞进正确位置(页内可能要搬动后面的项,但都在一个 8KB 页内,内存操作,快)。
- 如果这个叶页**满了**(空闲空间塞不下新项),就把它**分裂(split)**成左右两个叶页:把一半的项搬到新分配的右页,原页变成左页。分裂点两边的数据依然各自有序。
- 分裂产生了一个新页(右页),它需要一个"入口"——也就是父页(枝页)里要加一个**新的路由键 + downlink** 指向这个新页。**所以分裂会向上传播**:叶页分裂 → 要在父枝页里插一个 downlink → 父枝页可能也满了 → 父枝页也分裂 → ... 一路向上。
- **关键**:这个传播是**自底向上**的(从叶往根走),而且**只在分裂时发生**。平时插入(页没满)根本不碰上层结构。
- 传到根页怎么办?根页没有父页。这时**新建一个根页**(比原根高一层),把原根和分裂出的右页都挂在它下面。**根页分裂 = 整棵树长高一层。** 因为新根同时是所有叶的新公共祖先,长高一层意味着**所有叶页到根的距离同时 +1**——平衡依旧保持。

> **所以平衡的根因**:B 树从不"只让一条路径变长"。它要么在一层内塞进某页(不影响高度),要么分裂、且分裂最终可能导致根分裂让**全树统一长高**。没有任何一条路径能单独变长——这就是"从根到任何叶子距离相等"的保证。教科书上叫"B 树是高度平衡的(height-balanced)",代码里靠的就是这套自底向上的分裂传播。

正因为这样,B 树不会像 BST 那样被有序插入数据搞退化。你往 B 树里插 `1,2,3,...,1000万`,它只会让最右叶页不断分裂、链表不断向右长,但**树高始终是 log_F(当前行数)**——永远不会退化成链表。这就是为什么自增主键、时间戳这种天然有序的列,在 B 树上也能正常工作(虽然会集中打最右页,有并发热点,但那是另一回事,不破坏平衡)。

## 四、页面满了怎么办:分裂(split)

平衡靠分裂保证,那分裂本身怎么干?这是 B 树最核心、也最需要讲清的写入操作。我们分两步:先看**单个页怎么裂成两半**(分裂点怎么选),再看**分裂怎么向上传播到父页**。

### 4.1 分裂点怎么选:默认 50/50,最右页按 fillfactor

一页满了要裂成左右两页,问题是:**从哪里切开?** 把哪些项留给左页(原页),哪些搬到右页(新页)?

最朴素的想法是对半切——左右各分一半的项。PG 基本也是这个思路,但有个重要的细化,叫 **fillfactor(填充因子)**。

先看 fillfactor 的定义。PG 给 B 树设了几个默认的填充因子:

> [src/include/access/nbtree.h:200-202](../postgresql-17.0/src/include/access/nbtree.h#L200-L202)

```c
#define BTREE_DEFAULT_FILLFACTOR	90
#define BTREE_NONLEAF_FILLFACTOR	70
#define BTREE_SINGLEVAL_FILLFACTOR	96
```

- **`BTREE_DEFAULT_FILLFACTOR = 90`**:叶页默认填充到 90%。也就是说,平时叶页最多塞到 90% 满,留 10% 空闲给将来的插入(避免刚分裂完又马上满)。
- **`BTREE_NONLEAF_FILLFACTOR = 70`**:枝页(内部页)默认只填到 70%。枝页留更多空闲,是因为枝页分裂的代价更大(会触发上层再分裂),多留点余地。
- **`BTREE_SINGLEVAL_FILLFACTOR = 96`**:这是给"低基数列"(大量重复值,见第 10 章场景三)的特殊策略用的,下面讲。

分裂点选择的核心逻辑在 `_bt_findsplitloc`。看它的设计目标(注释讲得很直白):

> [src/backend/access/nbtree/nbtsplitloc.c:87-102](../postgresql-17.0/src/backend/access/nbtree/nbtsplitloc.c#L87-L102)

```c
 *	_bt_findsplitloc() -- find an appropriate place to split a page.
 *
 * The main goal here is to equalize the free space that will be on each
 * split page, *after accounting for the inserted tuple*.  (If we fail to
 * account for it, we might find ourselves with too little room on the page
 * that it needs to go into!)
 *
 * If the page is the rightmost page on its level, we instead try to arrange
 * to leave the left split page fillfactor% full.  In this way, when we are
 * inserting successively increasing keys (consider sequences, timestamps,
 * etc) we will end up with a tree whose pages are about fillfactor% full,
 * instead of the 50% full result that we'd get without this special case.
 * This is the same as nbtsort.c produces for a newly-created tree.  Note
 * that leaf and nonleaf pages use different fillfactors.
```

这段注释点出两种策略:

1. **一般页(非最右页):目标是"左右两半的空闲空间尽量相等"**——也就是接近 50/50 切。因为这种页的插入是"从四面八方来的"(键值分布),左右都可能继续被插,所以两边留一样多的空闲最公平。
2. **最右页(rightmost page):目标是"让左页填到 fillfactor%(90%)满"**。为什么特殊?因为最右页专门接"递增的键"(自增 id、时间戳总是往最右边插)。如果把左页填到 90%、右页留 10% 空闲,那么右页继续接递增键,能撑更久才再次分裂;而左页填到 90% 后基本不会再被插(新键都比它大),满了就满了。这样长期下来,树的页平均填充率高(接近 90%),而不是每次都 50% 空着浪费空间。

> **不这样会怎样(最右页不特殊处理)**:如果最右页也按 50/50 切,那每次分裂左页就剩 50% 空闲再也不会被用(因为新键永远往右走),整个索引的空间利用率只有 50%——一张 100GB 表的索引会膨胀到接近 200GB。按 fillfactor 切,利用率回到 90%,省一半空间。

此外,`_bt_findsplitloc` 还有一个"次要目标":在有多个候选分裂点(都满足空间均衡)时,**优先选能让路由键截断(suffix truncation)更有效的点**——也就是让父页里要插的那个新路由键尽量短。短的路由键 = 枝页塞得更多 = 扇出更大 = 树更矮。这是个以小博大的优化。

### 4.2 分裂向上传播:`_bt_insert_parent` 与根分裂

页切好两半后,要在父页里插一个 downlink 指向新页。如果父页也满了,父页也分裂,继续向上。这个"向上传播"由 `_bt_insert_parent` 完成。

> [src/backend/access/nbtree/nbtinsert.c:2083-2097](../postgresql-17.0/src/backend/access/nbtree/nbtinsert.c#L2083-L2097)

```c
 * _bt_insert_parent() -- Insert downlink into parent, completing split.
 *
 * On entry, buf and rbuf are the left and right split pages, which we
 * still hold write locks on.  Both locks will be released here.  We
 * release the rbuf lock once we have a write lock on the page that we
 * intend to insert a downlink to rbuf on (i.e. buf's current parent page).
 * The lock on buf is released at the same point as the lock on the parent
 * page, since buf's INCOMPLETE_SPLIT flag must be cleared by the same
 * atomic operation that completes the split by inserting a new downlink.
 *
 * stack - stack showing how we got here.  Will be NULL when splitting true
 *			root, or during concurrent root split, where we can be inefficient
 * isroot - we split the true root
 * isonly - we split a page alone on its level (might have been fast root)
 */
```

注意这里提到的 `INCOMPLETE_SPLIT` 标志——这是 B 树并发控制的关键(下面讲高键时会展开)。分裂分两步:① 物理上把页切成两半;② 在父页插入 downlink。这两步**不是原子的**(中间可能被打断)。如果只完成了第一步、第二步还没做,这个页就处于"分裂未完成"状态,用 `BTP_INCOMPLETE_SPLIT` 标志标记。别的进程碰到这种页,要先帮它把分裂补完(`_bt_finish_split`),这保证了即使分裂中途崩溃,重启后也能恢复一致。

最关键的是**根分裂**——这是树长高的唯一时刻。看 `_bt_insert_parent` 怎么处理:

> [src/backend/access/nbtree/nbtinsert.c:2122-2134](../postgresql-17.0/src/backend/access/nbtree/nbtinsert.c#L2122-L2134)

```c
	if (isroot)
	{
		Buffer		rootbuf;

		Assert(stack == NULL);
		Assert(isonly);
		/* create a new root node one level up and update the metapage */
		rootbuf = _bt_newlevel(rel, heaprel, buf, rbuf);
		/* release the split buffers */
		_bt_relbuf(rel, rootbuf);
		_bt_relbuf(rel, rbuf);
		_bt_relbuf(rel, buf);
	}
```

当发现分裂的是根页(`isroot`),它不往"父页"插 downlink(因为根没有父页),而是调用 `_bt_newlevel` **凭空造一个新的根页**,把原根页和分裂出的右页都挂到新根下面。新根比旧根高一层——**整棵树长高一层**。同时要更新元页(metapage)里的根指针(`btm_root` / `btm_fastroot`),让以后所有查找都从新根开始。

> 这就是"树长高"的全过程:它从最底层(叶页满了)开始,分裂像波浪一样向上传,传到根,根造个新爸爸,全树高一层。**整个过程,所有叶页到根的距离同时 +1——平衡不破。** 这和 BST 的"某条路径单独变长"有本质区别。

### 4.3 空了怎么办:删除与回收(delete / recycle)

有插入就有删除。当叶页里的项被删(比如堆表行被 VACUUM 清掉,索引项也跟着删,第 21 章),叶页可能变空。空的页不会立刻还给操作系统,而是先标记为 `BTP_HALF_DEAD`(半死)→ 再标记为 `BTP_DELETED`(已删除)→ 最后由 VACUUM 回收复用。这个过程叫**页删除(page deletion)**,在 `nbtpage.c` 里(`_bt_pagedel` 等)。

页删除比分裂复杂得多,因为它要保证:删掉一个页后,它的 downlink 要从父页移除、它的左右兄弟要重新连起来(`btpo_prev`/`btpo_next` 重新指向)。如果删的是某个枝页,更麻烦(要确保它下面没有子页了)。所以 PG 对页删除很保守——**只有当一页完全空了,且删除它不会破坏树的结构时才删**。大部分时候,部分空的页就让它留着(分裂时已经按 fillfactor 留了余量),等 VACUUM 统一处理。

> 这里和"合并(merge)"要澄清一个概念:教科书 B 树有时讲"页太空了就和兄弟合并"。**PG 的 B 树不主动做合并**——它宁可让页半空着(空间换简单),只做"页彻底空了才删"。这是工程取舍:合并要搬数据、改 downlink、重连兄弟,代价高且容易有并发问题;而半空的页占的空间,相对整张表通常微不足道。VACUUM 会定期清理完全空的页回收空间。

## 五、PG 在教科书 B+ 树之上的增强

到这里讲的是一个标准的 B+ 树。但 PG 的 nbtree 实现几十年迭代,加了不少教科书没有的增强。挑四个最值得讲的。

### 5.1 高键(highkey):让并发分裂不出错

这是 PG B 树(以及所有现代 B+ 树实现)的基石,源自 1981 年 Lehman & Yao 的经典论文。问题是这样:

**分裂时的并发难题**:假设事务 A 正在从根往叶走,走到某个枝页,决定往下走"第 2 个 downlink"指向的子页 P。就在它要读 P 之前,另一个事务 B 把 P **分裂**了——P 的一半数据被搬到了新页 P' (P 的右兄弟),P 上现在的数据只是左半部分。此时 A 去读 P,但它要找的键可能在 P'(右半部分,已经不 P 里了)。A 怎么知道该去 P' 找?

**高键的解法**:在每个页里存一个"高键(high key)"——它表示**这个页能容纳的键的上界**(即"本页所有键都 ≤ highkey")。高键放在叶页/枝页的第一个特殊位置(`P_HIKEY`)。查找时,如果要找的键 **> 这个页的高键**,说明它不在这个页里(被分裂到右兄弟了),就顺着 `btpo_next` 走到右兄弟继续找。

这就是 `_bt_moveright` 干的事(下面源码精读会看)——它拿着要找的键,和当前页的高键比;如果键比高键大(或页处于删除/半死等异常状态),就"向右移"到兄弟页,直到停在一个"高键 ≥ 要找的键"的正常页上。**有了高键,查找时即使目标页正在分裂,也能靠高键 + 右兄弟链表找到正确的页,不需要锁住整棵树。** 这是 B 树能高并发(多个事务同时读写不同页)的关键——Lehman-Yao 算法的核心就是"页级锁 + 高键 + 右移",而不是"锁整棵树"。

> 枝页里存的路由键(pivot),本质上也是一类"截断的高键"。分裂时,左页的高键变成父页里那个新 downlink 的键(经过 suffix truncation 截断)。这样父页的路由键和子页的高键语义一致:父页说"走这个 downlink,因为你要的键 ≤ 这个路由键";子页的高键说"我这里所有键都 ≤ 这个值"。两边对得上。

### 5.2 TID 去重(dedup):压缩大量重复键

第 10 章讲过"低基数列不适合 B 树"(大量重复值会让索引巨大且没用)。但现实里,有些列**就是**有很多重复值(比如 `status` 只有几种状态,但表有几亿行)。PG 从 v13 起,给 B 树加了 **dedup(去重)** 缓解这个问题。

思路:如果叶页里有大量**键值完全相同**的项(只是 TID 不同),就把它们**合并成一个"posting list tuple"**——存一份键值,后面挂一个 **TID 列表**(posting list)。这样 100 个 `(status='paid', TID_i)` 项,合并成 1 个 `(status='paid', [TID_1, TID_2, ..., TID_100])`,省掉 99 份重复的键值存储。索引能小很多。

去重的核心在 `_bt_dedup_one_page`(`nbtdedup.c`)。它在页满要分裂**之前**先试一把去重——如果去重后腾出了足够空间,就**避免一次分裂**(分裂代价大)。看它的触发场景:

> [src/backend/access/nbtree/nbtdedup.c:34-44](../postgresql-17.0/src/backend/access/nbtree/nbtdedup.c#L34-L44)

```c
 * Perform a deduplication pass.
 *
 * The general approach taken here is to perform as much deduplication as
 * possible by merging together groups of duplicate index tuples that share
 * the same key values.  Each group is merged into a posting list tuple.
 *
 * This must be called when a page is full of duplicate tuples to avoid a
 * page split.  It can also be called when the page is not really full of
 * duplicates, to implement bottom-up index deletion.  (Bottom-up deletion
 * merges suffix tuples from the page to "give a hint" to VACUUM.)
```

注意它有两个用途:① 页满时去重避免分裂(主要);② 配合"自底向上删除"(bottom-up deletion)给 VACUUM 提示(第 21 章)。

dedup 不是万能的。它要求**索引列的所有 opclass 都是"equalimage"**(即"值相等则其二进制表示长度不变",用于安全地复用 posting list),元页里有个 `btm_allequalimage` 标志记录这一点。`text`、`varchar` 这类变长但相等即同长的类型可以;某些自定义类型不行。`INCLUDE` 列(下面讲)不参与去重(它们不是键)。

> dedup 把"低基数列上 B 树索引巨大"这个老痛点大幅缓解了——同样是 `status` 列几亿行,去重后索引可能从几十 GB 缩到几 GB。但别忘了第 10 章的核心:索引慢的根在"回表的随机 I/O",去重只省了**索引本身的存储**,没省回表。所以低基数列上,该用 BRIN 还是用 BRIN(第 12 章)。

### 5.3 包含列(INCLUDE):不用回表就能取额外列

第 13 章会详讲"仅索引扫描(index-only scan)"——如果查询要的列**全在索引里**,就不用回表。但 B 树的键是参与排序的,你不想把一个只用于 `SELECT` 显示、不参与 `WHERE`/`ORDER BY` 的列也塞进键里(会拖慢插入、增大扇出损失)。

`INCLUDE` 列解法:`CREATE INDEX idx ON t (a) INCLUDE (b, c)`。这样 `a` 是键(参与排序和路由),`b`、`c` 是**额外附带的列**,只存在**叶页**里(不进枝页的路由键,不参与排序)。查询 `SELECT a, b FROM t WHERE a = 5` 就能仅索引扫描拿到 `b`,不用回表。

PG 的 B 树声明支持这个能力,看 `bthandler` 里:

> [src/backend/access/nbtree/nbtree.c:121](../postgresql-17.0/src/backend/access/nbtree/nbtree.c#L121)

```c
	amroutine->amcaninclude = true;
```

`amcaninclude = true` 表示这个索引 AM 支持 INCLUDE 列。B 树的枝页只存截断的键(pivot),INCLUDE 列被截断掉不进枝页——所以它们不影响树高和扇出,只在叶页占点空间换"免回表"。这是"用空间换时间"的又一例,细节留到第 13 章。

### 5.4 增量排序(incremental sort)

这是 PG 优化器和 B 树配合的一个优化。如果查询要 `ORDER BY a, b`,而 `a` 上有 B 树索引,那么按索引扫描拿到的数据**已经按 `a` 有序**了(叶链表有序)。优化器可以利用这一点,只对"每组相同的 `a` 内部"按 `b` 排序,而不必对全量数据排序——这叫**增量排序**。B 树的"叶页内有序 + 叶链表有序"是它能配合增量排序的前提。细节在执行器/优化器(P1),这里只点出 B 树的有序性是基础。

## 六、点查询与范围扫描的完整路径

有了上面的结构,我们把"一次点查询"和"一次范围扫描"在 B 树上走的路径完整走一遍。

### 6.1 点查询 `WHERE id = 42`:从根到叶,逐层二分

1. **找根**:从元页(metapage,第 0 页)读出根页块号(`btm_fastroot`)。优化:PG 把元页内容缓存在内存的 `rel->rd_amcache` 里,不用每次都读元页这一页 I/O(下面 `_bt_getroot` 会看到)。
2. **根页内二分**:在根页里,用二分查找(`_bt_binsrch`)比较 `id = 42`,找到"最后一个路由键 < 42"的位置,顺着它的 downlink 往下走。
3. **枝页内二分**:到了枝页,再二分,再顺着 downlink 往下。
4. **到达叶页**:到了叶页(标志是 `BTP_LEAF`),二分找到 `id = 42` 的项,拿到它的 TID。
5. **回表**:用 TID 去堆表读那一行(第 13 章)。

整个过程,读的索引页数 = 树高(3~4 页),加上 1 次回表堆页。如果这些页都在 Buffer Pool(第 8 章)里,连物理 I/O 都没有,纳秒到微秒级。

### 6.2 范围扫描 `WHERE id BETWEEN 10 AND 80`:定位起点,沿叶链表走

1. **定位起点**:和点查询一样,从根走到叶,但这次定位的是"第一个 ≥ 10"的项(用 `_bt_binsrch` 的 `nextkey=false` 语义,找第一个 ≥ key 的位置)。
2. **沿 `btpo_next` 向右走**:从这个位置开始,边走边把满足 ≤ 80 的项吐出去。当前叶页走完了,顺着 `btpo_next` 跳到右兄弟叶页,继续。
3. **遇到 > 80 就停**:一旦某项 > 80,扫描结束。

反向范围扫描则定位"最后一个 ≤ 80",沿 `btpo_prev` 向左走。这套机制全靠叶链表——没有它,范围扫描就得在树里上上下下,慢且复杂。

> 这两段路径,在代码里就是 `btgettuple` 调 `_bt_first`(定位起点,内含一次从根到叶的 `_bt_search`)+ 反复调 `_bt_next`(沿当前叶页/链表推进)。下面源码精读会看。

## 关键源码精读:从根到叶的查找 `_bt_search` 与页内二分 `_bt_binsrch`

前面讲了 B 树的结构和操作,现在钻进代码,看"一次从根到叶的查找"到底怎么落地。核心是两个函数:`_bt_search`(在树里逐层下降)和 `_bt_binsrch`(在某一页里二分定位)。

### 1. `_bt_search`:从根开始,一层一层往下走

> [src/backend/access/nbtree/nbtsearch.c:95-179](../postgresql-17.0/src/backend/access/nbtree/nbtsearch.c#L95-L179)

```c
BTStack
_bt_search(Relation rel, Relation heaprel, BTScanInsert key, Buffer *bufP,
		   int access)
{
	BTStack		stack_in = NULL;
	int			page_access = BT_READ;

	/* heaprel must be set whenever _bt_allocbuf is reachable */
	Assert(access == BT_READ || access == BT_WRITE);
	Assert(access == BT_READ || heaprel != NULL);

	/* Get the root page to start with */
	*bufP = _bt_getroot(rel, heaprel, access);

	/* If index is empty and access = BT_READ, no root page is created. */
	if (!BufferIsValid(*bufP))
		return (BTStack) NULL;

	/* Loop iterates once per level descended in the tree */
	for (;;)
	{
		Page		page;
		BTPageOpaque opaque;
		OffsetNumber offnum;
		ItemId		itemid;
		IndexTuple itup;
		BlockNumber child;
		BTStack		new_stack;

		/*
		 * Race -- the page we just grabbed may have split since we read its
		 * downlink in its parent page (or the metapage).  If it has, we may
		 * need to move right to its new sibling.  Do that.
		 */
		*bufP = _bt_moveright(rel, heaprel, key, *bufP, (access == BT_WRITE),
							  stack_in, page_access);

		/* if this is a leaf page, we're done */
		page = BufferGetPage(*bufP);
		opaque = BTPageGetOpaque(page);
		if (P_ISLEAF(opaque))
			break;

		/*
		 * Find the appropriate pivot tuple on this page.  Its downlink points
		 * to the child page that we're about to descend to.
		 */
		offnum = _bt_binsrch(rel, key, *bufP);
		itemid = PageGetItemId(page, offnum);
		itup = (IndexTuple) PageGetItem(page, itemid);
		Assert(BTreeTupleIsPivot(itup) || !key->heapkeyspace);
		child = BTreeTupleGetDownLink(itup);

		/*
		 * We need to save the location of the pivot tuple we chose in a new
		 * stack entry for this page/level.  If caller ends up splitting a
		 * page one level down, it usually ends up inserting a new pivot
		 * tuple/downlink immediately after the location recorded here.
		 */
		new_stack = (BTStack) palloc(sizeof(BTStackData));
		new_stack->bts_blkno = BufferGetBlockNumber(*bufP);
		new_stack->bts_offset = offnum;
		new_stack->bts_parent = stack_in;

		...
		/* drop the read lock on the page, then acquire one on its child */
		*bufP = _bt_relandgetbuf(rel, *bufP, child, page_access);

		/* okay, all set to move down a level */
		stack_in = new_stack;
	}
	...
	return stack_in;
}
```

逐段读这个函数,它把"从根到叶"写得极清晰:

- **`*bufP = _bt_getroot(...)`**:先拿到根页。`_bt_getroot` 会优先用缓存的元页数据(`rel->rd_amcache`)直接定位根,省一次 I/O。
- **`for (;;)` 主循环:每轮下降一层**。循环体里干四件事:
  1. **`_bt_moveright`**:处理"我刚拿到的页可能已经被别人分裂了"的竞态——这是高键机制的落点(下面单独看)。
  2. **`P_ISLEAF(opaque)` 判断**:如果到叶页了,`break` 跳出。注意它判断的是页的 `BTP_LEAF` 标志位,而不是层数——`btpo_level == 0` 等价于叶页。
  3. **`_bt_binsrch` 二分定位**:在当前枝页里二分,找到该走哪个 downlink(`offnum`),取出那个 pivot tuple,拿到它的 downlink(`child = BTreeTupleGetDownLink(itup)`)。
  4. **压栈 + 下降**:`new_stack` 把"我在这一层选的 pivot 位置"记下来(块号 + 偏移),压进 `stack`。这个栈是给**插入**用的——如果下面叶页分裂了,要回到这个位置在父页插 downlink(就是前面讲的分裂向上传播)。然后 `_bt_relandgetbuf` 释放当前页、锁住子页,下降一层。
- **返回值是 `stack`**:一条从根到叶的父页路径。查找用不到它(查找只要叶页 buffer),但**插入**会用到——这是 `_bt_search` 同时服务查询和插入的精妙之处。

> 注意一个细节:下降时,父页的锁**会被释放**(`_bt_relandgetbuf` 里 rel 释放、get 新页)。也就是说,查找途中不持有上层页的锁——这是为了并发(别阻塞别人访问上层页)。代价是:如果叶页要分裂,父页的锁已经没了,得靠 stack 里记的位置重新找到父页并加锁(在 `_bt_getstackbuf` 里做)。这是 Lehman-Yao 并发 B 树的标准做法:**只锁当前操作的页,不锁路径上的页**。

### 2. `_bt_moveright`:高键在手的"向右漂移"

上面循环里每层都先调 `_bt_moveright`。它就是高键机制的代码化身——看它的核心判断:

> [src/backend/access/nbtree/nbtsearch.c:266-307](../postgresql-17.0/src/backend/access/nbtree/nbtsearch.c#L266-L307)

```c
	cmpval = key->nextkey ? 0 : 1;

	for (;;)
	{
		page = BufferGetPage(buf);
		opaque = BTPageGetOpaque(page);

		if (P_RIGHTMOST(opaque))
			break;

		/*
		 * Finish any incomplete splits we encounter along the way.
		 */
		if (forupdate && P_INCOMPLETE_SPLIT(opaque))
		{
			...
			if (P_INCOMPLETE_SPLIT(opaque))
				_bt_finish_split(rel, heaprel, buf, stack);
			...
			continue;
		}

		if (P_IGNORE(opaque) || _bt_compare(rel, key, page, P_HIKEY) >= cmpval)
		{
			/* step right one page */
			buf = _bt_relandgetbuf(rel, buf, opaque->btpo_next, access);
			continue;
		}
		else
			break;
	}
```

关键这一行:

```c
if (P_IGNORE(opaque) || _bt_compare(rel, key, page, P_HIKEY) >= cmpval)
```

- `_bt_compare(rel, key, page, P_HIKEY)` 把要找的 `key` 和**当前页的高键**(`P_HIKEY` 是高键的位置)比。
- 如果 `key >= 高键`(普通情况 `cmpval=1`,即 `key > 高键`),说明要找的键**不在这个页**(它被分裂到右兄弟了),于是 `buf = _bt_relandgetbuf(rel, buf, opaque->btpo_next, ...)`——**顺着 `btpo_next` 跳到右兄弟页**,继续判断。
- 循环往复,直到停在一个"高键 ≥ key"的正常页上,才返回,交给 `_bt_binsrch` 去页内二分。

还有两种情况也触发右移:`P_IGNORE(opaque)`(页处于已删除/半死状态,跳过它)和 `P_INCOMPLETE_SPLIT`(分裂未完成,先帮它补完 `_bt_finish_split`)。`P_RIGHTMOST` 是最右页(没有右兄弟了),到这就停——最右页的高键是正无穷,任何 key 都不会超过它。

> 这个函数虽小,却是"B 树能并发"的命根。没有它,分裂时就得锁整棵树(防止别人走到半分裂的页);有了高键 + 右移,分裂时只需要锁**正在分裂的那一页**和它的父页,其他事务靠高键自己找到正确的页。这是高并发的关键。

### 3. `_bt_binsrch`:页内二分查找

到了正确的页,要在页内定位。一个页里有上百个有序键,用二分。`_bt_binsrch` 是教科书二分的实现:

> [src/backend/access/nbtree/nbtsearch.c:356-397](../postgresql-17.0/src/backend/access/nbtree/nbtsearch.c#L356-L397)

```c
	low = P_FIRSTDATAKEY(opaque);
	high = PageGetMaxOffsetNumber(page);

	...
	if (unlikely(high < low))
		return low;

	...
	high++;						/* establish the loop invariant for high */

	cmpval = key->nextkey ? 0 : 1;	/* select comparison value */

	while (high > low)
	{
		OffsetNumber mid = low + ((high - low) / 2);

		/* We have low <= mid < high, so mid points at a real slot */

		result = _bt_compare(rel, key, page, mid);

		if (result >= cmpval)
			low = mid + 1;
		else
			high = mid;
	}
```

标准的二分:`low`/`high` 夹逼,`mid = low + (high-low)/2`,`_bt_compare` 比较 key 和 mid 位置的项,根据结果缩范围。`P_FIRSTDATAKEY` 是"第一个真实数据键的位置"——叶页是第 1 项,枝页是第 2 项(第 1 项是 negative infinity 项,即"-∞",保证至少有一个 downlink 指向最左子页)。

注意函数开头那个 `P_FIRSTDATAKEY(opaque)` 和"页可能只有高键没有数据项"的处理(`high < low` 时返回 `low`)——这对应"页被 VACUUM 清空了数据只剩高键"的边界情况。

这个二分返回什么,分两种语义(函数头注释讲得清楚):

> [src/backend/access/nbtree/nbtsearch.c:320-328](../postgresql-17.0/src/backend/access/nbtree/nbtsearch.c#L320-L328)

```c
 * On an internal (non-leaf) page, _bt_binsrch() returns the OffsetNumber
 * of the last key < given scankey, or last key <= given scankey if nextkey
 * is true.  (Since _bt_compare treats the first data key of such a page as
 * minus infinity, there will always be at least one key < scankey, so the
 * result
 * always points at one of the keys on the page.)
 *
 * On a leaf page, _bt_binsrch() returns the final result of the initial
 * positioning process that started with _bt_first's call to _bt_search.
```

- **枝页**:返回"最后一个 < scankey 的键"的位置——它的 downlink 指向"键可能所在的子页"。这是"往哪走"的答案。
- **叶页**:返回"第一个 ≥ scankey 的项"的位置(前向扫描)——这是扫描的起点。

至此,`_bt_search` + `_bt_moveright` + `_bt_binsrch` 三件套,把"从根到叶,逐层二分,顺便处理分裂竞态"讲完了。一次点查询在 B 树里的全部代码路径就是它们——加起来就是"树高次循环,每次一个页内二分 + 可能的右移",对应前面说的 O(log n) 次页访问。

### 补充:`_bt_getroot` 与元页缓存

`_bt_search` 第一步 `_bt_getroot` 怎么找根?它有个重要优化——缓存元页:

> [src/backend/access/nbtree/nbtpage.c:361-394](../postgresql-17.0/src/backend/access/nbtree/nbtpage.c#L361-L394)

```c
	if (rel->rd_amcache != NULL)
	{
		metad = (BTMetaPageData *) rel->rd_amcache;
		...
		rootblkno = metad->btm_fastroot;
		Assert(rootblkno != P_NONE);
		rootlevel = metad->btm_fastlevel;

		rootbuf = _bt_getbuf(rel, rootblkno, BT_READ);
		rootpage = BufferGetPage(rootbuf);
		rootopaque = BTPageGetOpaque(rootpage);

		/*
		 * Since the cache might be stale, we check the page more carefully
		 * here than normal.  We *must* check that it's not deleted. If it's
		 * not alone on its level, then we reject too --- this may be overly
		 * paranoid but better safe than sorry.  Note we don't check P_ISROOT,
		 * because that's not set in a "fast root".
		 */
		if (!P_IGNORE(rootopaque) &&
			rootopaque->btpo_level == rootlevel &&
			P_LEFTMOST(rootopaque) &&
			P_RIGHTMOST(rootopaque))
		{
			/* OK, accept cached page as the root */
			return rootbuf;
		}
		_bt_relbuf(rel, rootbuf);
		/* Cache is stale, throw it away */
		if (rel->rd_amcache)
			pfree(rel->rd_amcache);
		rel->rd_amcache = NULL;
	}

	metabuf = _bt_getbuf(rel, BTREE_METAPAGE, BT_READ);
	metad = _bt_getmeta(rel, metabuf);
```

注意它用的是 `btm_fastroot`(fast root),不是 `btm_root`(true root)。区别:**"fast root"是当前实际有效的根**——如果真正的根下面只有唯一一个子页(因为 VACUUM 删空了别的子页),那这个根就是多余的,真正的查找可以从它的子页开始(降一层省一次 I/O)。`btm_fastroot` 指向这种"瘦身后的有效根"。只有当 fast root 校验失败(缓存过期),才回退去读真正的元页(`BTREE_METAPAGE`,块号 0)。

这套缓存 + fast root,把"找根"这步的 I/O 从"读元页 + 读根页"两次,压到"读根页"一次(缓存命中时)。对每次查询都从根开始的 B 树,这是个实打实的优化。

## 关键源码精读:插入与分裂 `_bt_insertonpg`

查是读,写是另一回事。往 B 树插一个新项,核心是 `_bt_insertonpg`(on page 的缩写)。它的职责注释写得极全:

> [src/backend/access/nbtree/nbtinsert.c:1075-1102](../postgresql-17.0/src/backend/access/nbtree/nbtinsert.c#L1075-L1102)

```c
 *	_bt_insertonpg() -- Insert a tuple on a particular page in the index.
 *
 *		This recursive procedure does the following things:
 *
 *			+  if postingoff != 0, splits existing posting list tuple
 *			   (since it overlaps with new 'itup' tuple).
 *			+  if necessary, splits the target page, using 'itup_key' for
 *			   suffix truncation on leaf pages (caller passes NULL for
 *			   non-leaf pages).
 *			+  inserts the new tuple (might be split from posting list).
 *			+  if the page was split, pops the parent stack, and finds the
 *			   right place to insert the new child pointer (by walking
 *			   right using information stored in the parent stack).
 *			+  invokes itself with the appropriate tuple for the right
 *			   child page on the parent.
 *			+  updates the metapage if a true root or fast root is split.
```

它列出了插入的全部子步骤。看核心的"要不要分裂"判断:

> [src/backend/access/nbtree/nbtinsert.c:1210-1242](../postgresql-17.0/src/backend/access/nbtree/nbtinsert.c#L1210-L1242)

```c
	if (PageGetFreeSpace(page) < itemsz)
	{
		Buffer		rbuf;

		Assert(!split_only_page);

		/* split the buffer into left and right halves */
		rbuf = _bt_split(rel, heaprel, itup_key, buf, cbuf, newitemoff, itemsz,
						 itup, origitup, nposting, postingoff);
		PredicateLockPageSplit(rel,
							   BufferGetBlockNumber(buf),
							   BufferGetBlockNumber(rbuf));

		/*----------
		 * By here,
		 *
		 *		+  our target page has been split;
		 *		+  the original tuple has been inserted;
		 *		+  we have write locks on both the old (left half)
		 *		   and new (right half) buffers, after the split; and
		 *		+  we know the key we want to insert into the parent
		 *		   (it's the "high key" on the left child page).
		 *
		 * We're ready to do the parent insertion.
		 */
		_bt_insert_parent(rel, heaprel, buf, rbuf, stack, isroot, isonly);
	}
```

逻辑干净利落:

- **`PageGetFreeSpace(page) < itemsz`**:`PageGetFreeSpace` 返回页的空闲空间(第 6 章的 `pd_upper - pd_lower`)。如果塞不下新项,就分裂。
- **`_bt_split`**:真正干分裂——调用 `_bt_findsplitloc` 选分裂点,把页切成左右两半(分配新页 `rbuf` 作右页),把项分到两边,把新项塞进该去的那一半。**注意注释里那句 "we know the key we want to insert into the parent (it's the 'high key' on the left child page)"——分裂后,左页的高键,就是要插入父页的那个新路由键。** 这是高键和路由键的连接点:分裂产生高键,高键变成父页的 pivot。
- **`_bt_insert_parent`**:把 downlink 插进父页(就是上一节看的那个函数,它会判断是不是根分裂)。这里形成了**递归**:`_bt_insertonpg` 分裂后调 `_bt_insert_parent`,`_bt_insert_parent` 在父页里又要插入,如果父页满了又调 `_bt_insertonpg`(对父页)分裂……自底向上传播就是这么递归起来的。

注意还有个 dedup 的前置:如果 `postingoff != 0`,先拆 posting list(去重相关的边界情况);以及,在真正调 `_bt_split` 之前,`_bt_doinsert` 的调用链里其实已经试过一次 `_bt_dedup_one_page`(去重避免分裂)。只有去重也腾不出空间,才走 `_bt_split`。所以分裂是最后的手段。

> 把 `_bt_search`(找位置)+ `_bt_insertonpg`(插,满了就分裂)+ `_bt_insert_parent`(向上传播)串起来,就是一次 B 树插入的全貌:**先像查询一样从根走到目标叶页(顺带压了栈),在叶页插入;满了就分裂,分裂信息顺着栈向上传,传到根就造新根长高。** 整个过程,平衡始终成立——因为长高是全树统一长一层,不是某条路径单独长。

## 章末小结

### 本章讲了什么

- **为什么是 B 树(不是二叉树/红黑树/哈希)**:核心是"**多路 + 平衡 + 页对齐**"。多路让一层筛掉 99% 候选(扇出 F≈100);平衡让树高 = log_F(n);页对齐让一个节点 = 一个 8KB 页,每次访问 = 一次页 I/O,不浪费。1000 万行:二叉树 23 层(23 次 I/O)vs B 树 4 层(4 次 I/O),碾压。哈希虽 O(1) 但只支持等值、不支持范围/排序/前缀,所以不是默认。
- **B 树的三种页**:叶页存(键值, TID),只存路标不存整行(回表第 13 章);枝页存截断的路由键(pivot)+ downlink;根页是入口,块号记在元页里。**叶页用 `btpo_next`/`btpo_prev` 串成双向链表**,这是范围扫描的生命线。
- **为什么必须平衡**:不平衡(如 BST 按有序数据插入)退化成链表 O(n)。B 树靠**分裂向上传播**保证平衡——叶页满了分裂,分裂传到父页,父页满了再分裂,传到根就造新根、全树长高一层。所有叶页到根距离始终相等。
- **分裂**:分裂点默认按"左右空闲均衡"选(50/50);**最右页特殊**——按 fillfactor 让左页填到 90%(叶)/70%(枝),因为最右页专接递增键。叶 fillfactor 默认 90,枝 70。PG 不主动合并,只删彻底空的页(VACUUM 回收)。
- **PG 的增强**:① **高键(highkey)** + `_bt_moveright` 右移,让分裂时不必锁整棵树(Lehman-Yao 并发 B 树);② **TID 去重(dedup)** 把大量重复键合并成 posting list,压缩索引、避免分裂;③ **INCLUDE 列**只存叶页、不进枝页路由键,支持仅索引扫描免回表;④ 配合优化器做**增量排序**(叶链表已按前缀有序)。
- **点查询** = 从根到叶逐层二分(`_bt_search` + `_bt_binsrch`),读树高个页 + 1 次回表;**范围扫描** = 定位起点后沿 `btpo_next` 顺序走叶链表。

### 回扣主线:快 vs 不丢不乱 + 三个本性

本章整章服务"**快**"那一侧——B 树的全部意义,是把"找数据"从 O(n) 压到 O(log n),并且让范围扫描走顺序 I/O。它直接对付的是数据的**第三个本性**:"用户说**要什么**(id=42 / id BETWEEN 10 AND 80),机器只会**按位置读**"——B 树就是这个错位的翻译官里最高效的一位:多路把层数压到 3~4,页对齐让每层就是一次 I/O,叶链表把范围查询变成顺序走。

> 注意一个张力(和第 10 章一脉相承):B 树在服务"快",但分裂、维护高键、dedup 都引入了**写入的代价**。每次 INSERT 可能触发分裂(分裂要分配新页、搬数据、改父页 downlink、记 WAL),写放大明显。这是"快"内部的权衡——查询越快(树越矮、扇出越大),写入时维护这棵精巧的树代价越高。第 21 章会看到,VACUUM 还要定期清理 B 树里的死页,这是平衡的长期代价。

另一点:本章讲的高键、`INCOMPLETE_SPLIT`、页删除的保守策略,这些其实都在服务"**不丢不乱**"——分裂是个多步操作,中途崩溃不能让树结构损坏;高键让并发事务看到的树始终一致。所以 B 树既是"快"的工具,其内部机制又要保证"不乱"(并发一致、崩溃可恢复,靠 WAL 记每一次页修改,第 18 章)。
