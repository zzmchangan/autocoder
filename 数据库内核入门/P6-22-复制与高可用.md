# 第 22 章 · 复制与高可用:流复制、主从一致性

> **前置**:你需要先读过 [第 18 章《WAL》](P5-18-WAL-预写日志-掉电不丢的命根子.md) 和 [第 19 章《Checkpoint》](P5-19-Checkpoint-日志与数据的同步点.md)——P5 持久篇把一台机器里的数据做到了"掉电不丢":改动先记进 WAL,脏页靠 checkpoint 兑现成磁盘持久,崩溃后从 redo 起点重放就能恢复一致。但单机本身是一个**单点(single point of failure)**——一旦整机损毁(机房失火、磁盘物理损坏、所在机架断电),WAL 和数据页同归于尽,P5 的全部努力归零。本章做一件事:**把主机产生的同一份 WAL,搬到另一台机器上去重放**,于是"不丢"从一台机器扩展到了一个机房。这就是 PostgreSQL 高可用的根基——**物理流复制**。

> **核心问题**:为什么需要复制(单机是单点故障,整机没了数据就全没)?PostgreSQL 的物理流复制怎么工作——主机每产生一条 WAL,怎么流式传给备机、备机怎么把它"apply"成和主一致的数据?为什么复制的是 WAL(物理日志)而不是 SQL?异步复制和同步复制各自防什么灾难、各有什么代价?复制槽(replication slot)为什么必不可少?主从切换(failover)为什么有"脑裂"风险?
>
> **读完本章你会明白**:① 物理流复制的全貌——主机 walsender 流式发 WAL、备机 walreceiver 接收落盘、startup 进程持续 apply,**备机 apply 的本质就是"一直在做的崩溃恢复"**(WAL 来源从本地文件换成网络);② 异步复制的"数据丢失窗口"长什么样、同步复制怎么消除它、以及代价(延迟、备机挂了主也卡住);③ `synchronous_commit` 五档同步级别的精确语义;④ 复制槽如何为断线备机保留 WAL,以及"备机永不回来则主机 WAL 堆积"这个风险;⑤ 主备是物理复制 → 备机只读、必须同版本同架构,逻辑复制可跨版本但有局限。

> **如果一读觉得太难**:先记三件事——① 复制 = 把主机的 WAL 流式传给备机,备机像崩溃恢复一样持续重放,从而和主保持一致;② 异步:主不等备(快,但主机挂了可能丢已提交数据);同步:主等备确认(不丢,但慢、备挂主也卡);③ 复制槽让主机为断线备机保留 WAL,否则备机追不上。其余细节第二遍配合源码再抠。

## 一、为什么必须有复制:单机是单点

P5 三章讲下来,单机的"易失"本性看似被治住了:WAL 兜底持久性,checkpoint 划定 redo 锚点,崩溃恢复把没落盘的改动补回来。但这里藏着一个前提——**WAL 和数据页必须在同一台机器的磁盘上**。

### 不这样会怎样:整机一毁,全盘皆输

> **不这样会怎样**:假设你只有一台数据库服务器。WAL、数据页、`pg_control` 全在这一台机器的磁盘上。现在机房失火(或者磁盘控制器烧了、所在机架掉电、误删了数据目录)——**整台机器物理损毁**。WAL 写得再勤、checkpoint 做得再标准,有什么用?磁盘都没了,P5 那套"从 redo 起点重放"根本没有 WAL 可重放。所有数据,包括上一秒刚 `COMMIT` 成功的,全没了。这就是**单点故障**:一台机器,无论它内部多可靠,都是整个系统的单一致命点。

P5 解决的是"机器没坏、只是掉电/崩溃后内存丢了"这种场景;它**没有**解决"机器本身没了"。而"机器本身没了"在生产里并不罕见——硬件会坏、机房会断电、火灾水灾会损毁物理设备、运维会误操作 `rm -rf`。对任何严肃的生产系统,**把数据冗余到另一台(甚至另一个机房)的机器上,是和 WAL 一样硬的刚需**。这就是**复制(replication)**存在的全部理由:**把主机上的数据搬到备机上,主机没了,备机顶上**。

复制还顺带带来一个"读扩展"的好处:既然备机有一份和主一样的数据,只读查询可以分流到备机去跑,减轻主机的压力。但要注意——**复制的首要动机是高可用/容灾,读扩展是它的副产品,不是它的本职**。下面我们紧扣"容灾"这条主线来讲。

## 二、复制什么:为什么复制 WAL 而不是 SQL

知道了"要把数据搬到备机",下一个问题是:**搬什么**?有两种截然不同的思路,PostgreSQL 两种都支持,但本章的主角是第一种。

### 思路一:物理复制——搬 WAL,字节级重放

第 18 章讲过,主机上的每一次数据改动,都被忠实地记成一条 WAL 记录("改了哪个块、改成什么样")。物理复制的做法是:**把主机产生的 WAL,原封不动地传给备机;备机拿到这些 WAL 记录后,像主机崩溃恢复一样,一条条重放(redo),把同样的改动施加到自己的数据页上**。

为什么这样做能保证主备一致?因为 WAL 是**字节级、确定性的改动说明书**:同一条 WAL 记录,无论在主机上重放还是在备机上重放,施加的效果完全一样(改的是同一个 block、同样的增量)。只要备机从一个和主一致的起点(全量基线,见第四节 basebackup)开始,按同样的顺序重放同样的 WAL,它就会一步一步地走到和主机相同的状态。

> **关键洞察(本章最重要的一句话):备机在持续做的事,就是崩溃恢复。** 第 20 章会专门讲崩溃恢复——主机崩溃重启后,从 redo 起点重放 WAL,把没落盘的改动补回来,直到追上 WAL 末尾。备机的 apply 过程,和这个**完全是一回事**:都是"读 WAL → 调资源管理器的 redo 函数 → 把改动施加到数据页"。唯一的区别是——**主机崩溃恢复时,WAL 来源是本地 `pg_wal/` 里已经落盘的段文件;备机 apply 时,WAL 来源是 walreceiver 刚从网络收到、写进本地 `pg_wal/` 的段**。除此之外,redo 的逻辑代码是同一套。这就是为什么物理复制"简单且一致"——它没有发明新机制,而是把已有的崩溃恢复机制,套到一个"WAL 从网络源源不断来"的循环上。

### 思路二:逻辑复制——搬"变更",按表/行重放

另一种思路是**逻辑复制**:主机不只传 WAL 原始字节,而是先把 WAL **解码(decode)**成"对哪张表的哪一行做了 INSERT/UPDATE/DELETE"这样的逻辑变更,再传给备机;备机收到后,像执行 SQL 一样把这些变更应用到自己表上。

> **为什么物理复制是 PG 高可用的首选?** ① **简单一致**:WAL 是字节级重放,不需要理解语义,天然保证主备字节级一致,不会有"解码漏了某种操作"的问题;② **复用机制**:不用为复制单独造一套,直接拿崩溃恢复的 redo 来用;③ **延迟低**:主机产生 WAL 的同时就能流式发出去,几乎实时。逻辑复制则要额外做解码这一步,复杂且开销大。
>
> 但逻辑复制有物理复制做不到的本事:它可以**跨大版本**(主 13 备 15)、**跨平台/架构**(主 Linux 备 Windows)、**选择性复制**(只复制某些表)、**双向/多向**。物理复制因为重放的是字节级 WAL,要求**主备必须是相同大版本、相同架构(尤其 `--with-blocksize` 等编译选项要一致)**,否则字节对不上。所以物理复制用于"同构主备容灾"(本章重点),逻辑复制用于"异构、跨版本、数据订阅"等场景(本章末尾概览)。

### 物理复制的一个硬约束:备机只读

物理流复制下,备机处于**持续恢复状态(archive recovery / standby mode)**——它一直在 apply WAL,数据页随时可能被 redo 改写。如果允许同时在外部修改备机的数据,就会和 redo 打架(刚改的页被下一条 WAL 重放覆盖),一致性立刻崩溃。所以 **PG 的物理备机是只读的**:可以查询(这叫 hot standby),但不能写。这也是"读写分离"只能把读分流到备机、写必须走主机的根本原因。

## 三、物理流复制怎么工作:walsender + walreceiver + startup 三方协作

现在讲机制。物理流复制涉及三个进程角色,它们跨网络协作:

```
   主机(primary)                          备机(standby)
   ┌─────────────────┐   WAL 字节流    ┌──────────────────┐
   │  walsender 进程  │ ──────────────> │ walreceiver 进程  │
   │  (每连接一个)    │  <────────────  │ (写进本地 pg_wal) │
   │  读 pg_wal 段    │   备机反馈       │                  │
   └─────────────────┘  (write/flush/   └────────┬─────────┘
         ↑ 读              apply 位置)           │ 通知
   WAL 落盘位置                                  ↓
                                       ┌──────────────────┐
                                       │ startup 进程      │
                                       │ (持续 apply WAL)  │
                                       └──────────────────┘
```

1. **walsender(主机侧)**:主库上,每有一个备机连上来,就 fork 一个 **walsender 进程**专门服务这个连接。它的活儿是——读主机 `pg_wal/` 里的 WAL 段(或直接从 WAL 缓冲),通过连接**流式**发给备机,并接收备机反馈回来的进度(write/flush/apply 到哪了)。
2. **walreceiver(备机侧)**:备库上常驻一个 **walreceiver 进程**(由 startup 进程拉起)。它从网络收 WAL 字节,**写进备机本地的 `pg_wal/` 段文件**、fsync,然后给主机回一个"我收到/落盘/回放到哪了"的反馈消息。
3. **startup(备机侧)**:这就是崩溃恢复用的那个 startup 进程。它在备机上**持续 apply(重放)**本地 `pg_wal/` 里的 WAL——和第 20 章崩溃恢复的 redo 完全一样,只不过这里 WAL 不会"用完",而是源源不断地被 walreceiver 写进来。每当 walreceiver flush 了一批新 WAL,它就唤醒 startup 继续重放。

> **整个链条的节奏**:主机产生 WAL → walsender 发出 → walreceiver 收下写盘并反馈 → startup 重放 → 备机状态追上主机。只要这个循环不停,备机就一直在追赶主机,延迟通常在毫秒到秒级。这就是"流复制"里"流(streaming)"的含义——WAL 像水流一样源源不断地从主流向备,而不是攒一批再传(攒一批传是 archive restore 的模式)。

### 备机怎么标识自己是备机:standby.signal

备机启动时怎么知道自己该进"持续恢复"模式、而不是当普通主库?靠一个**信号文件**。备机的数据目录里如果存在 `standby.signal` 文件,启动时就会进入 standby 模式(持续恢复,等 WAL 来,不主动提升为主)。这个判断在 `readRecoverySignalFile` 里:

> [src/backend/access/transam/xlogrecovery.c:1056-1094](../postgresql-17.0/src/backend/access/transam/xlogrecovery.c#L1056-L1094)

```c
	if (stat(STANDBY_SIGNAL_FILE, &stat_buf) == 0)
	{
		...
		standby_signal_file_found = true;
	}
	else if (stat(RECOVERY_SIGNAL_FILE, &stat_buf) == 0)
	{
		...
		recovery_signal_file_found = true;
	}

	StandbyModeRequested = false;
	ArchiveRecoveryRequested = false;
	if (standby_signal_file_found)
	{
		StandbyModeRequested = true;
		ArchiveRecoveryRequested = true;
	}
	else if (recovery_signal_file_found)
	{
		StandbyModeRequested = false;
		ArchiveRecoveryRequested = true;
	}
```

注意注释里那句:**"If present, standby signal file takes precedence"**——`standby.signal` 优先于 `recovery.signal`。两者的区别:`standby.signal` 表示"我要一直当备机(持续恢复,追上后继续等新 WAL)";`recovery.signal` 表示"我要做一次性的归档恢复(追上 archive 后就提升为主,停下来)"。所以搭物理备机,放一个空的 `standby.signal` 文件就行。这俩文件本身内容为空,它们的存在与否就是全部信号——这是 PG 设计里典型的"用文件存在性当开关"的手法(和 checkpoint 里 `pg_control` 的原子替换写是同一种朴素的可靠性偏好)。

## 四、初始全量同步:basebackup——复制的起点

流复制有一个先决条件:**备机得有一个和主机一致的起点**,然后才能从这个起点开始追 WAL。否则备机是一张白纸,主机发来的 WAL 记录里说"改 block 1234",备机根本不知道 block 1234 现在长什么样,redo 无从下手。

这个"起点"怎么来?靠 **basebackup(基础备份)**——主机把整个数据目录拍一个**一致性快照**(某个 WAL 位置那一刻的全量数据),传给备机;备机解压后就有了一份和主机在那个时刻一致的数据,然后从快照对应的 WAL 位置开始流式追赶之后的增量。

basebackup 的核心函数是 `perform_base_backup`([src/backend/backup/basebackup.c:234](../postgresql-17.0/src/backend/backup/basebackup.c#L234))。它大体做这些事:① 调 `do_pg_backup_start` 记录备份起始的 WAL 位置(这个位置就是备机要开始追的 redo 点);② 遍历数据目录,把每个文件通过 `sendFile` 传过去([basebackup.c:88](../postgresql-17.0/src/backend/backup/basebackup.c#L88),实际读取由 `basebackup_read_file` 完成);③ 同时确保备份期间产生的必要 WAL 也一起传(否则备机追不上从起点到备份结束这段);④ 传一个 `backup_label` 文件,里面记着"这个备份从哪个 WAL 位置开始"——备机启动时读它,就知道从哪开始 redo。

> **为什么不直接 `cp` 数据目录?** 因为数据目录是个"活的"状态——后台进程在不断改 buffer、写 WAL、移动文件。你 `cp` 的时候拷到的可能是一致性破损的中间态(某个文件拷了一半、checkpoint 刚推进、`pg_control` 和数据页不匹配)。basebackup 的关键是它和 checkpoint 配合,记录了一个**确定的、一致的 WAL 起点**,并保证拷出来的文件集合对应那个起点的一致状态。所以搭备机的标准流程是:`pg_basebackup -h primary -D /data/standby`(它内部就是调 `BASE_BACKUP` 复制协议命令,触发主机的 basebackup),然后放一个 `standby.signal`,启动即可。

## 五、异步复制 vs 同步复制:数据丢失窗口的权衡

到这里流复制的骨架清楚了,但有一个关键问题悬而未决:**主机 `COMMIT` 一个事务时,要不要等备机也收到/落盘这条 WAL 才返回成功给客户端?** 这个选择,把复制分成"异步"和"同步"两种模式,它们防的灾难完全不同。

### 异步复制:主不等备,快但有丢失窗口

默认情况下是**异步复制**:主机 `COMMIT` 时,**只等本地 WAL fsync 落盘就返回成功**(就是第 18 章讲的 `synchronous_commit = on` 的默认行为),**不等备机**。WAL 照常流式发给备机,但提交不卡在备机身上。

> **不这样会怎样(异步的丢失窗口)**:异步模式下,存在一个**数据丢失窗口**——主机已经 `COMMIT` 成功并返回客户端"提交了",但这批 WAL **还没传到备机**。就在这个瞬间主机损毁了(比如主机磁盘突然坏了、整机掉电后磁盘也废了)。由于异步,备机此时还没收到这批 WAL,于是**这批已经"提交成功"的事务,在备机上不存在**。failover 把备机提升为新主后,客户端会发现:明明数据库返回过"提交成功"的几笔交易,凭空消失了。这就是异步复制的代价:为了不卡提交、保持主机高性能,牺牲了"主机整机损毁时不丢已提交数据"这个保证。

异步复制是绝大多数场景的默认选择,因为:① 性能好,主机提交延迟不受备机网络/磁盘影响;② 大多数业务能容忍"整机损毁时丢最后几毫秒数据"这个小概率事件(整机损毁本身就罕见)。但如果你不能容忍(比如金融账务),就要用同步复制。

### 同步复制:主等备确认,不丢但有代价

**同步复制**让主机的 `COMMIT` 多等一步——**等备机确认收到/落盘这条 WAL 之后,才返回成功**。这样,只要主机返回了"提交成功",就保证备机已经有了这条 WAL;主机整机损毁,failover 到备机,这批数据不会丢。同步复制**消除了丢失窗口**。

> **代价(必须看清楚)**:① **延迟上升**——每次提交都要等一个网络往返 + 备机磁盘 fsync,提交延迟明显变大(尤其备机在远端机房时,网络 RTT 直接加在每次提交上);② **备机挂了,主也卡住**——这是最要命的。如果同步备机宕机或网络断了,主机的提交会**一直等**备机确认,等不到就卡住,整个主库的写操作全停。这就是为什么同步复制通常要配多个备机(`synchronous_standby_names` 可以要求"至少 N 个备机确认"),只要还有一个同步备机活着,主就能继续;③ 如果所有同步备机都挂了且没配降级,主库会完全停摆。

注意源码里这个细节——同步等待发生在提交路径上,`SyncRepWaitForLSN` 接收两个参数:`lsn`(要等到哪个 WAL 位置)和 `commit`(是不是提交操作):

> [src/backend/replication/syncrep.c:148-180](../postgresql-17.0/src/backend/replication/syncrep.c#L148-L180)

```c
SyncRepWaitForLSN(XLogRecPtr lsn, bool commit)
{
	int			mode;
	...
	/* Fast exit if user has not requested sync replication ... */
	if (!SyncRepRequested() ||
		!((volatile WalSndCtlData *) WalSndCtl)->sync_standbys_defined)
		return;

	/* Cap the level for anything other than commit to remote flush only. */
	if (commit)
		mode = SyncRepWaitMode;
	else
		mode = Min(SyncRepWaitMode, SYNC_REP_WAIT_FLUSH);
```

读这段有两个关键点:① **快速退出**——如果用户根本没请求同步复制(`!SyncRepRequested()`)或没定义同步备机名(`!sync_standbys_defined`),直接 `return`,不等待。这让同步复制在未启用时对普通提交路径**零开销**(提交路径每次都会经过这里,必须能快速跳过);② **非提交操作只等到 flush**——`commit` 为 false 时(比如某些非关键的内务操作),最多等到备机 flush,不等到 apply,因为这些操作不需要 apply 完成那么强的保证。这是个精细的分级。

### 等到备机的哪一步:`synchronous_commit` 五档同步级别

同步复制"等备机等到哪一步"不是非黑即白,而是一系列**水位**。第 18 章已经见过 `synchronous_commit` 的本地档位(off/local/on),这里补齐它的**远程档位**——这五档合起来,精确刻画了"主机提交愿意为持久性等到备机的哪个动作":

| `synchronous_commit` | 主机 `COMMIT` 等到 | 主机整机损毁后丢不丢已提交数据 | 性能 |
|---|---|---|---|
| `off` | WAL 进**本地内存缓冲**(不等 fsync) | 丢(连本地都没落盘) | 最快 |
| `local` | 本地 WAL **fsync 落盘**(不等备机) | 主机磁盘没坏则不丢;整机坏则丢 | 快 |
| `remote_write` | 本地 fsync + 备机**写到 OS 缓存**(write,未 fsync) | 备机随后掉电可能丢(数据在备机 OS 缓存里没落盘) | 中 |
| `on`(默认) | 本地 fsync(若无同步备机,等同 local) | 取决于有无同步备机 | 中 |
| `remote_flush` | 本地 fsync + 备机 **fsync 落盘** | 不丢(备机已落盘) | 慢 |
| `remote_apply` | 本地 fsync + 备机**已回放完**(apply) | 不丢,且备机已可读到该数据 | 最慢 |

(严格说 `on` 在有同步备机时按 `synchronous_standby_names` 配置的级别等待;上表把语义讲清即可。)

这几个水位在源码里就是三个常量,代表备机反馈的三种进度:

> [src/include/replication/syncrep.h:22-27](../postgresql-17.0/src/include/replication/syncrep.h#L22-L27)

```c
#define SYNC_REP_NO_WAIT		(-1)
#define SYNC_REP_WAIT_WRITE		0
#define SYNC_REP_WAIT_FLUSH		1
#define SYNC_REP_WAIT_APPLY		2

#define NUM_SYNC_REP_WAIT_MODE	3
```

备机的 walreceiver 每次落盘/回放后,会给主机发一个**反馈消息**,里面带三个 LSN:写到哪(`writePtr`)、flush 到哪(`flushPtr`)、apply 到哪(`applyPtr`)。看它怎么构造这个反馈:

> [src/backend/replication/walreceiver.c:1140-1146](../postgresql-17.0/src/backend/replication/walreceiver.c#L1140-L1146)

```c
	resetStringInfo(&reply_message);
	pq_sendbyte(&reply_message, 'r');
	pq_sendint64(&reply_message, writePtr);
	pq_sendint64(&reply_message, flushPtr);
	pq_sendint64(&reply_message, applyPtr);
	pq_sendint64(&reply_message, GetCurrentTimestamp());
	pq_sendbyte(&reply_message, requestReply ? 1 : 0);
```

主机的 walsender 收到这个反馈后,更新全局的"同步备机已确认到哪",然后**唤醒**所有等到这个 LSN 的事务:

> [src/backend/replication/syncrep.c:508-521](../postgresql-17.0/src/backend/replication/syncrep.c#L508-L521)

```c
	if (walsndctl->lsn[SYNC_REP_WAIT_WRITE] < writePtr)
	{
		walsndctl->lsn[SYNC_REP_WAIT_WRITE] = writePtr;
		numwrite = SyncRepWakeQueue(false, SYNC_REP_WAIT_WRITE);
	}
	if (walsndctl->lsn[SYNC_REP_WAIT_FLUSH] < flushPtr)
	{
		walsndctl->lsn[SYNC_REP_WAIT_FLUSH] = flushPtr;
		numflush = SyncRepWakeQueue(false, SYNC_REP_WAIT_FLUSH);
	}
	if (walsndctl->lsn[SYNC_REP_WAIT_APPLY] < applyPtr)
	{
		walsndctl->lsn[SYNC_REP_WAIT_APPLY] = applyPtr;
		numapply = SyncRepWakeQueue(false, SYNC_REP_WAIT_APPLY);
	}
```

读这段就能把同步复制的机制彻底看懂:**备机反馈三个进度(write/flush/apply)→ 主机更新三个对应的全局水位 → 唤醒等待队列里所有"我等的水位已经被某备机达到"的事务**。一个事务在 `SyncRepWaitForLSN` 里把自己挂到 `waitLSN = 我的提交 LSN`、`syncRepState = SYNC_REP_WAITING` 上去睡;walsender 这边一旦发现某档水位追上了它,就把它改成 `SYNC_REP_WAIT_COMPLETE` 唤醒,事务返回提交成功。

> **`remote_write` vs `remote_flush` 这个细微差别为什么重要?** `remote_write` 等的是备机把 WAL `write()` 到 OS 页缓存——快(不用等备机磁盘 fsync),但备机随后若掉电,OS 缓存里的数据丢了,等于没收到。`remote_flush` 等的是备机 fsync 真正落盘——慢一点,但备机掉电也不丢。生产里大多选 `remote_flush`(够安全且延迟可接受);只有对延迟极敏感、能容忍"备机也掉电则丢"的场景才选 `remote_write`。这正是第 18 章"持久性是一系列水位"那句话在复制层面的延伸。

## 六、复制槽(replication slot):为断线备机保留 WAL

到这里还有一个致命问题没解决:**备机断线时,主机会不会把备机还没收的 WAL 删掉**?

### 不这样会怎样:断线备机追不上,数据永久丢失

> **不这样会怎样(没有复制槽)**:回忆第 19 章,主机靠 checkpoint 划定"WAL 可以安全截断到哪"——redo 点之前的 WAL 可以回收。这个截断边界是**按主机自己的需要**算的(checkpoint 推进了就删旧 WAL)。现在假设一个备机断线了(网络故障、停机维护)几个小时。这期间主机一直在产生新 WAL、做 checkpoint、回收旧 WAL。等备机修好重连,它要的"断线期间那批 WAL"——**已经被主机删了**。备机彻底追不上,只能重新做一次全量 basebackup。更糟的是,如果这批被删的 WAL 里恰好有备机还没 apply 的关键数据,备机的状态就和主永久分叉了。

这个问题的本质是:**主机在决定"哪些 WAL 可以删"时,只考虑了自己(崩溃恢复需要),没考虑备机(它可能还要那些 WAL)**。需要一种机制,让主机**知道"还有个备机没收到某个 LSN 之前的 WAL,这些 WAL 不能删"**——这就是**复制槽(replication slot)**。

### 复制槽怎么工作:restart_lsn 钉住 WAL 删除边界

复制槽是一个**具名的、持久化的状态记录**,每个槽代表"一个消费者(某个备机、某个逻辑订阅)的消费进度"。槽里最关键字段是 **`restart_lsn`**——"这个消费者确认收到的最老 WAL 位置",意思是"`restart_lsn` 之前的 WAL 它已经消费完了,可以删;之后的它还要,不能删"。

> [src/include/replication/slot.h:92-93](../postgresql-17.0/src/include/replication/slot.h#L92-L93)

```c
	/* oldest LSN that might be required by this replication slot */
	XLogRecPtr	restart_lsn;
```

槽的数据存在 `ReplicationSlotPersistentData` 里(磁盘上也有一份,重启不丢)。每当备机(walreceiver)确认消费到新位置,就更新对应槽的 `restart_lsn` 往前推。主机在算"WAL 能删到哪"时,**会把所有复制槽的 `restart_lsn` 取最小值,作为删除边界的下限**——看 `ReplicationSlotsComputeRequiredLSN`:

> [src/backend/replication/slot.c:1105-1138](../postgresql-17.0/src/backend/replication/slot.c#L1105-L1138)


### 复制槽的风险:备机永不回来,WAL 堆积

> **不这样会怎样(槽的反面风险)**:复制槽是个双刃剑。如果某个备机**永远不再回来**(比如退役了、忘了删槽),它的槽还钉着一个很老的 `restart_lsn`,主机就会**为这个永不回来的消费者无限保留 WAL**——`pg_wal/` 目录会一直涨,最终撑爆磁盘,主库挂掉。这就是为什么用槽要负责:不用的槽必须手动 drop 掉(`pg_drop_replication_slot`)。

源码里这个风险有个兜底——`max_slot_wal_keep_size`(注释里的 "Cap by max_slot_wal_keep_size"):它限制"为单个槽最多保留多少 WAL",超过就**允许删**(此时那个慢槽会变成 invalidated,它的消费者将追不上,但保住了主库不被 WAL 撑爆)。这是个"宁可让慢备机失效,也要保主库活着"的取舍。默认值 `-1` 表示无限保留(老行为,危险),生产环境强烈建议设一个上限。

> **物理复制槽 vs 逻辑复制槽**:物理槽钉的是 `restart_lsn`(用于物理备机);逻辑槽还多钉一个 `confirmed_flush`(逻辑解码的消费进度)和 `catalog_xmin`(防止主机的 catalog 被清理掉,导致解码不出旧行版本)。两者机制类似,本章聚焦物理槽。

## 七、主从切换与脑裂:failover 的风险

当主机真的挂了,需要把备机提升为新主——这叫 **failover(故障切换)**,PG 里叫 **promote(提升)**。promote 后备机退出恢复模式,变成可读可写的主库。

> **脑裂(split-brain)风险**:failover 最怕的就是**同时出现两个主**。想象这个场景——原主机其实没死,只是网络瞬断了一下,监控误判它挂了,把备机 promote 成新主。现在原主机又恢复了,网络上同时存在两个都能写的主库。两边的写各走各的,数据彻底分叉,且**无法自动合并**(物理复制不支持双向)。这就是**脑裂**——两个"大脑"各自为政,是分布式系统里最灾难性的故障。

> **PG 物理流复制本身不防脑裂**。它没有内置的"分布式仲裁/quorum"机制来保证任何时刻只有一个主。防脑裂要靠**外部的高可用管理软件**(如 Patroni、repmgr、pg_auto_failover)配合**fencing**——promote 夈备之前,必须先**彻底隔离(fence)旧主机**(把它关电、隔离网络、`pg_ctl stop -m immediate`),确保它绝不可能再接受写,然后才提升备机。这一步"先 fence 后 promote"是所有 PG HA 方案的生命线,绕不过去。

promote 之后还有一个**时间线(timeline)**的概念:每次 promote,新主会**切换到一个新的时间线**(timeline ID + 1),并写一个 timeline history 文件记录"我在旧时间线的哪个 LSN 分叉出去的"。这样新主产生的 WAL 用新 timeline 标记,和旧主的 WAL 不会混。其他备机要追随新主,得切到新 timeline。timeline 机制是 PG 处理"分叉历史"的基础,也是为什么 `pg_wal` 文件名里带 timeline ID 的原因。

## 关键源码精读:walsender 主循环——主机怎么流式发 WAL

这一章最值得精读的是 walsender 的核心循环:`WalSndLoop`([src/backend/replication/walsender.c:2767](../postgresql-17.0/src/backend/replication/walsender.c#L2767))和它调用的 `XLogSendPhysical`([walsender.c:3066](../postgresql-17.0/src/backend/replication/walsender.c#L3066))。它把"主机怎么决定发到哪、怎么发、怎么等"讲得清清楚楚。

### WalSndLoop:发数据 + 处理反馈 + 等待

`WalSndLoop` 是 walsender 的主循环骨架。它接收一个回调 `send_data`(物理复制时就是 `XLogSendPhysical`):

> [src/backend/replication/walsender.c:2780-2865](../postgresql-17.0/src/backend/replication/walsender.c#L2780-L2865)(节选)

```c
	for (;;)
	{
		ResetLatch(MyLatch);
		CHECK_FOR_INTERRUPTS();

		/* Process any requests or signals received recently */
		if (ConfigReloadPending)
		{
			ConfigReloadPending = false;
			ProcessConfigFile(PGC_SIGHUP);
			SyncRepInitConfig();
		}

		/* Check for input from the client */
		ProcessRepliesIfAny();
		...
		/*
		 * If we don't have any pending data in the output buffer, try to send
		 * some more.
		 */
		if (!pq_is_send_pending())
			send_data();
		else
			WalSndCaughtUp = false;

		/* Try to flush pending output to the client */
		if (pq_flush_if_writable() != 0)
			WalSndShutdown();

		/* If nothing remains to be sent right now ... */
		if (WalSndCaughtUp && !pq_is_send_pending())
		{
			if (MyWalSnd->state == WALSNDSTATE_CATCHUP)
			{
				...
				WalSndSetState(WALSNDSTATE_STREAMING);
			}
			...
		}

		/* Check for replication timeout. */
		WalSndCheckTimeOut();
		/* Send keepalive if the time has come */
		WalSndKeepaliveIfNecessary();
		...
		if ((WalSndCaughtUp && send_data != XLogSendLogical &&
			 !streamingDoneSending) ||
			pq_is_send_pending())
```

读这段循环,它的每一拍做四件事:

1. **`ProcessRepliesIfAny()`**:先处理备机发来的反馈消息。walsender 是**双向**的——不光发 WAL,还收备机的 write/flush/apply 进度反馈(就是第五节那套同步复制用到的)。这里把反馈解析掉,更新同步复制的水位。
2. **`send_data()`(即 `XLogSendPhysical`)**:如果没有待发数据,调一次 `send_data` 去抓新 WAL 发出去。这是真正"发 WAL"的动作。
3. **超时与保活**:`WalSndCheckTimeOut()` 检查备机是不是太久没响应(可能挂了);`WalSndKeepaliveIfNecessary()` 在空闲时发 keepalive,让备机知道主还活着。
4. **状态机**:从 `WALSNDSTATE_CATCHUP`(追赶中)切到 `WALSNDSTATE_STREAMING`(已追上,实时同步)。注释特意点出——catchup 到 streaming 这个切换很重要,因为"在 catchup 完成前,主机挂了 failover 到备机会丢数据"。这就是异步复制丢失窗口的代码体现。

### XLogSendPhysical:决定发到哪、发多少

真正抓 WAL 发送的是 `XLogSendPhysical`。它的核心是先算出"能安全发到哪",再读那段 WAL 发出去。算安全边界的逻辑很值得读:

> [src/backend/replication/walsender.c:3158-3170](../postgresql-17.0/src/backend/replication/walsender.c#L3158-L3170)

```c
	else
	{
		/*
		 * Streaming the current timeline on a primary.
		 *
		 * Attempt to send all data that's already been written out and
		 * fsync'd to disk.  We cannot go further than what's been written out
		 * given the current implementation of WALRead().  And in any case
		 * it's unsafe to send WAL that is not securely down to disk on the
		 * primary: if the primary subsequently crashes and restarts, standbys
		 * must not have applied any WAL that got lost on the primary.
		 */
		SendRqstPtr = GetFlushRecPtr(NULL);
	}
```

注意这条铁律的注释:**"it's unsafe to send WAL that is not securely down to disk on the primary"**——主机只能发**已经 fsync 落盘**的 WAL,不能发还在 WAL 缓冲里没落盘的。为什么?因为如果发了没落盘的 WAL,备机 apply 了它,然后主机崩溃重启——这条 WAL 在主机上可能丢失(没落盘),于是主备分叉:备机 apply 了一条主机自己都没有的 WAL。所以发送边界严格卡在 `GetFlushRecPtr()`(已 flush 的位置)。这和第 18 章"日志先落盘"的精神一脉相承:**持久的东西才能复制**。

算好 `SendRqstPtr` 后,它按 `MAX_SEND_SIZE` 一批批读 WAL 发出。读的时候有个细节值得注意——优先从 **WAL 缓冲**读,读不到再读文件:

> [src/backend/replication/walsender.c:3289-3307](../postgresql-17.0/src/backend/replication/walsender.c#L3289-L3307)(节选)

```c

	rbytes = WALReadFromBuffers(&output_message.data[output_message.len],
								startptr, nbytes, xlogreader->seg.ws_tli);
	...
	/* now read the remaining WAL from WAL file */

		!WALRead(xlogreader,
```

这个"缓冲优先"的优化很巧妙——刚产生、还在 WAL buffer 里的热 WAL(还没被 write 出去),walsender 可以直接从内存读来发,省一次磁盘读。这对"主机刚提交、备机立刻要追"的低延迟场景很有帮助。

发完后更新共享内存里这个 walsender 的 `sentPtr`,供同步复制状态机查询:

> [src/backend/replication/walsender.c:3353-3358](../postgresql-17.0/src/backend/replication/walsender.c#L3353-L3358)

```c

		SpinLockAcquire(&walsnd->mutex);
		walsnd->sentPtr = sentPtr;
		SpinLockRelease(&walsnd->mutex);
	}
```

> **把 WalSndLoop + XLogSendPhysical 串起来看**:walsender 是一个"算边界(只发已落盘的)→ 批量读(缓冲优先)→ 发出去 → 收反馈 → 等下一拍"的循环。它既是 WAL 的发送方,又是同步复制反馈的接收方,一头连着主机的 WAL 产生,一头连着备机的消费进度。理解了这个循环,物理流复制的运行时行为就彻底清楚了。

作为对照,备机一侧的 `WalReceiverMain`([src/backend/replication/walreceiver.c:183](../postgresql-17.0/src/backend/replication/walreceiver.c#L183))是个对称的循环:收 WAL → `XLogWalRcvWrite` 写进本地段([walreceiver.c:910](../postgresql-17.0/src/backend/replication/walreceiver.c#L910),用 `pg_pwrite`)→ `XLogWalRcvFlush` fsync([walreceiver.c:992](../postgresql-17.0/src/backend/replication/walreceiver.c#L992),调 `issue_xlog_fsync`)→ `XLogWalRcvSendReply` 发反馈([walreceiver.c:1140](../postgresql-17.0/src/backend/replication/walreceiver.c#L1140))→ 唤醒 startup 重放(`WakeupRecovery()`)。两边一唱一和,WAL 就这样从主流向备。

## 八、逻辑复制概览:物理复制做不到的事

本章主角是物理流复制,但 PostgreSQL 还有一套**逻辑复制(logical replication)**(PG 10 引入),它解决的是物理复制覆盖不了的场景。这里做一个概览,不展开。

物理复制的硬约束是:**必须同大版本、同架构、备机只读、整库复制**。但现实里有这些需求:① **跨版本升级**(主 14 备 16,滚动升级不停服);② **选择性复制**(只把某些表同步到另一个库,做报表或数据分发);③ **异构环境**(主在 Linux,目标在云上另一个平台);④ **多主写入**(多个节点都能写,逻辑复制做双向同步)。

逻辑复制的工作方式:主机上为"发布(publication)"建一个逻辑复制槽,一个 **walsender 进程**读 WAL,但不是原样发字节,而是先用**逻辑解码插件(logical decoding,在 `src/backend/replication/logical/` 目录)**把 WAL 解码成"对表 X 的行做了 INSERT/UPDATE/DELETE"的变更流,再发出去;备机(订阅方)上有一个 **apply worker 进程**(`src/backend/replication/logical/worker.c`),像执行 SQL 一样把这些变更应用到本地表。

> **代价**:① **复杂、开销大**——解码要理解表的 schema、要处理 catalog 版本(所以逻辑槽要钉 `catalog_xmin`),比物理重放贵得多;② **不支持所有 DDL**——逻辑复制只搬 DML(增删改),不搬表结构变更(CREATE/ALTER TABLE)。所以跨版本升级时,新版本加的列、改的类型,得手动在订阅端先建好结构;③ **只搬 logged 表**,临时表、序列等有限制;④ 一致性是"逻辑级"的(行级),不是字节级,主备数据可能不是逐字节相同(但语义一致)。

逻辑复制是 PG 生态里一块大内容,够单独写好几章。本章你只需记住:**物理复制=字节级 WAL 重放,同构容灾,简单一致;逻辑复制=解码后的变更重放,异构灵活但有局限**。生产里两者常常共存——物理复制做同城高可用容灾,逻辑复制做跨地域数据分发/升级。

## 章末小结

- **复制把"不丢"从单机扩展到机房**:单机是单点,整机损毁则 P5 的持久性全部归零;复制把主机的 WAL 搬到备机重放,实现异地冗余,主机没了备机顶上,治的是"整机物理损毁"这种 P5 管不到的灾难。
- **物理流复制 = 备机在持续做崩溃恢复**:walsender 流式发 WAL、walreceiver 收下写进本地 `pg_wal/`、startup 进程持续 apply。**备机的 apply 本质就是 redo**,和第 20 章崩溃恢复是同一套代码,只是 WAL 来源从本地文件变成网络。这是物理复制"简单且一致"的根因。
- **复制 WAL 而不是 SQL**:WAL 是字节级、确定性的改动说明书,同条记录在主备重放效果相同,天然一致;逻辑复制要解码成变更,灵活但复杂、不支持所有 DDL。物理复制要求主备同版本同架构、备机只读。
- **初始全量同步靠 basebackup**:备机要先有一份和主一致的起点(`perform_base_backup` 拍一致性快照 + `backup_label` 记起点 WAL 位置),才能从该点流式追后续 WAL。备机用 `standby.signal` 标识自己进持续恢复模式。
- **异步 vs 同步**:异步(默认)主不等备,快但有**丢失窗口**(主机提交后、备机收到前整机损毁则丢这批已提交数据);同步主等备确认(write/flush/apply 三档),不丢但延迟升、备机挂了主也卡。`synchronous_commit` 五档(off/local/remote_write/on/remote_flush/remote_apply)精确控制等到备机哪一步。
- **复制槽钉住 WAL 删除边界**:无槽时主机按自身需要删 WAL,断线备机会因 WAL 被删而追不上;槽的 `restart_lsn` 让主机为该消费者保留 WAL(`ReplicationSlotsComputeRequiredLSN` 取最小值进 `KeepLogSeg`)。风险:备机永不回来则 WAL 堆积(靠 `max_slot_wal_keep_size` 兜底)。
- **failover 有脑裂风险**:PG 物理复制本身不防脑裂,需外部 HA 软件(Patroni 等)配合 fencing(先隔离旧主再 promote 备机);promote 会切新 timeline 避免主备 WAL 混淆。

### 回扣主线

这一章是"**不丢不乱**"那一侧在**单机之外**的延伸,治的还是"**易失**"本性——但这次的"易失"指的是**整机级的易失**(机器本身会损毁),比 P5 治的"内存易失"更严重一层。

P5 的 WAL 把持久性承诺从"内存"延伸到"单机的磁盘";**复制把持久性承诺从"单机磁盘"再延伸到"另一台机器的磁盘"**——这就是"易失的纵深防御":内存易失靠 WAL,单机易失靠复制,机房易失靠跨地域复制。每一层都把"不丢"的边界往外推一格。同步复制更进一步,把"**不乱**"(主备一致)也兜住了——异步只保证"最终一致"(有延迟窗口),同步保证"提交即可见、不丢"。

至于三个本性:本章主攻"**易失**"(容灾);同时同步复制涉及的等待/反馈机制,和 P4 事务并发、P5 的 group commit 在"提交路径上的协调"上一脉相承——你会在 `SyncRepWaitForLSN` 里看到和锁等待、latch 唤醒同样的套路。物理复制只读这一约束,则源自"共享会冲突"——备机若同时被外部写,会和 redo 打架(第 20 章会详细讲 redo 的并发约束)。
