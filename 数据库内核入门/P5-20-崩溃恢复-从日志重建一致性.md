# 第 20 章 · 崩溃恢复:从日志重建一致性

> **前置**:你需要先读过 [第 18 章《WAL》](P5-18-WAL-预写日志-掉电不丢的命根子.md)(WAL 铁律、`XLogRecord`、资源管理器 rmgr、FPI)和 [第 19 章《Checkpoint》](P5-19-Checkpoint-日志与数据的同步点.md)(checkpoint 在 WAL 里焊下一个 redo 起点,并把它写进 ControlFile)。本章就是兑现这两章埋下的所有钩子:数据库真的崩溃了、重启了,它怎么靠那条 redo 起点把内存里丢失的脏页"变回来"。此外建议读过 [第 17 章《MVCC》](P4-17-MVCC-读不阻塞写写不阻塞读.md)(xmin/xmax、旧版本仍在)——本章的灵魂一问"为什么 PG 只要 redo 不要 undo",答案全在 MVCC 里。

>
> **读完本章你会明白**:① 崩溃恢复的本质——从最近 checkpoint 的 redo 起点开始,逐条重放 WAL 到末尾,把没落盘的改动补回数据页;② 为什么 PG 只要 redo、不要 undo——这是 MVCC 设计的直接后果(未提交事务的改动对别人本来就不可能可见,把它当 abort 处理即可,而 abort 在 MVCC 下是"打个标记"、不是"擦掉改动");③ 完整的恢复流程:读 ControlFile → 找最近 checkpoint 记录 → 取 redo 点 → 循环读 WAL record → 按 `xl_rmid` 分发到对应 `rm_redo` → 遇到 commit/abort 记录处理事务状态 → 恢复 2PC prepared 事务;④ redo 幂等性怎么保证(LSN 比较:`page->pd_lsn >= record->lsn` 就跳过这条),以及第 18 章埋的 FPI 为什么是这套幂等性能成立的前提(防 torn page);⑤ 为什么崩溃恢复重放到 WAL 末尾就能开门营业,而备机 apply 是**同一套代码**(第 22 章已预告,本章兑现)。

> **如果一读觉得太难**:先记三件事——① 崩溃恢复 = 从最近 checkpoint 的 redo 起点开始,逐条重放 WAL 到末尾;② PG 只要 redo 不要 undo,因为 MVCC 让"未提交的改动本来就没人看得见",把它当 abort 处理就行;③ 重放时先比 LSN:`page->pd_lsn >= record->lsn` 说明这页已经比这条记录新,跳过,这样重放才幂等。其余细节第二遍配合源码再抠。

## 一、崩溃恢复要解决什么:内存脏页没了,但 WAL 还在

第 18 章立下 WAL 铁律后,持久性似乎解决了:`COMMIT` 只等日志 fsync,数据页可以安心脏在内存里慢慢刷。第 19 章又用 checkpoint 把"脏页批量兑现"做成了周期动作,并在 WAL 里焊下一个 redo 起点。可这两个机制都没回答最关键的一问——**真的崩溃了,内存清空了,那些还没落盘的脏页,到底怎么回来?**

先把"崩溃"这件事写具体。数据库正常运行时,数据页被改的位置在两处:① 已经 `write + fsync` 到磁盘的(在 checkpoint 范围里的,或被 backend/bgwriter 顺手刷掉的);② 还脏在内存 Buffer Pool 里的。崩溃(掉电、`kill -9` postmaster、内核 panic)发生时,**第②类全没了**——内存一清空,这些"已经承诺给用户"的改动蒸发。这时磁盘上的状态分两种:

- **redo 点之前**:数据页是新的、一致的(checkpoint 保证,见第 19 章那六件事)。
- **redo 点之后**:可能有一批脏页的改动没落盘(还在内存里就崩溃了),也可能落了一部分(checkpoint 之后某个 backend 顺手刷过)。具体哪些落了哪些没落,数据库**无从知晓**——它只知道"redo 点之后,数据页的状态是不可信的"。

> **不这样会怎样(没有崩溃恢复)**:假设数据库崩溃重启后,直接当"redo 点之前的磁盘状态"就是最终状态开门营业。后果:用户上一秒 `COMMIT` 成功的转账,磁盘上其实没落盘(redo 点之后的脏页丢了),重启后这笔转账**凭空消失**。持久性(D)彻底破产——你向用户承诺了"`COMMIT` 成功 = 永久生效",可崩一次就丢了已提交的事务。这是不可接受的灾难。

那怎么救?靠 WAL。第 18 章埋下的伏笔在这里兑现:**WAL 是已 fsync 的、持久化的**。`COMMIT` 成功时,这条事务产生的所有 WAL 记录(包括 commit record 本身)都已经 `fsync` 到磁盘(第 18 章第九节讲的三道水位)。所以崩溃那一刻——

- 内存里的脏页**丢了**(易失本性);
- 但磁盘上的 WAL **还在**(已 fsync,持久),它忠实地记录了"redo 点之后,每个事务对每个数据页做了什么改动"。

**崩溃恢复做的事,就是用还在的 WAL,把丢失的脏页改动重新施加到磁盘的数据页上。** 这就是"重放(redo)":按 WAL 记录的顺序,一条一条地把改动"再做一遍",做到 WAL 末尾,数据页就回到了崩溃前那一刻的状态(至少是所有已 fsync 的 WAL 反映的状态)。

> 一句话:**崩溃恢复 = 从 redo 起点,逐条重放 WAL 到末尾**。这就是 redo。第 18、19 章铺的所有垫子(WAL 记录格式、rmgr 分发、FPI、LSN、redo 起点),都是为了让这"逐条重放"既对又快。

## 二、灵魂一问:为什么只要 redo、不要 undo

这一节是本章最关键、也是数据库内核设计里最有"为什么"味道的一问。先抛结论:

> **PostgreSQL 的崩溃恢复只做 redo(重做已提交的改动),不做 undo(撤销未提交的改动)。**

很多人第一次听到会愣住:崩溃时,内存里有些事务的改动**还没提交**(比如一个事务刚 INSERT 了几行,正准备 `COMMIT` 就崩了)。这些未提交的改动,有些可能已经产生 WAL 记录(在 redo 点之后,backend 的 `XLogInsert` 是边改边记的,不是攒到 COMMIT 才记),重放 WAL 时这些改动会被重做出来。**不撤销它们,数据不就脏了吗?**

答案是 PG 根本不需要撤销,因为 **MVCC 让"未提交事务的改动对别人本来就不可能可见"**。我们把这个机制掰开。

### MVCC 的核心:旧版本仍在,新版本标无效

第 17 章讲过,PG 的每一行(tuple)在页里都带一个 tuple 头,头里有两个关键字段:

> [src/include/access/htup_details.h:124-125](../postgresql-17.0/src/include/access/htup_details.h#L124-L125)

```c
	TransactionId t_xmin;		/* inserting xact ID */
	TransactionId t_xmax;		/* deleting or locking xact ID */
```

- `t_xmin`:**创建(插入)这条 tuple 的事务 ID**。
- `t_xmax`:**删除(或锁定)这条 tuple 的事务 ID**——如果这条 tuple 还在,`t_xmax` 通常是 `InvalidTransactionId`。

MVCC 的精髓是:**改一行,不是把旧行原地覆盖,而是插一条新行,把旧行标记为"被某事务删除"**。一个事务读这行时,看自己的快照(事务开始时拍的照片),对比 tuple 的 `t_xmin/t_xmax` 和提交状态,决定这条 tuple 对自己可不可见——读到的永远是自己事务开始那一刻的一致版本,中间别人正在改、还没提交的版本,对它根本不存在。

这就引出 PG 处理 abort 的关键性质:**abort 一个事务,不需要"擦掉"它做的改动**。为什么?因为它的改动全在那些"新 tuple"上,而新 tuple 的 `t_xmin = 这个事务`。这个事务 abort 了,`t_xmin` 这个事务的提交状态在 `pg_xact`(提交状态位图,旧名 CLOG)里被标成"aborted"。**任何读它的人,看到 `t_xmin` 对应的事务是 aborted,立刻知道这条 tuple 无效,跳过它**——根本不会读到它的内容。也就是说,**未提交/已 abort 的改动,留在数据页里也没人看;abort 只是在 `pg_xact` 里打个"aborted"标记,不需要回滚物理改动**。

tuple 头里还有加速这个判断的 hint bits:

> [src/include/access/htup_details.h:204-208](../postgresql-17.0/src/include/access/htup_details.h#L204-L208)

```c
#define HEAP_XMIN_COMMITTED		0x0100	/* t_xmin committed */
#define HEAP_XMIN_INVALID		0x0200	/* t_xmin invalid/aborted */
...
#define HEAP_XMAX_COMMITTED		0x0400	/* t_xmax committed */
#define HEAP_XMAX_INVALID		0x0800	/* t_xmax invalid/aborted */
```

`HEAP_XMIN_INVALID` 这个 hint bit 标着"创建这条 tuple 的事务是 abort 的",读的时候一眼跳过,不用再去查 `pg_xact`。

### 崩溃时,未提交事务怎么处理

有了上面这个铺垫,崩溃恢复的处理就非常优雅了。崩溃那一刻,有这几类事务:

1. **已经 `COMMIT` 成功的事务**:它的 commit record 已经 fsync(第 18 章第九节)。重放 WAL 时会遇到这条 commit record,redo 会把它的提交状态在 `pg_xact` 里标成 `committed`——它的改动**保留**。
2. **已经显式 `ROLLBACK` 或出错 abort 的事务**:它的 abort record 也在 WAL 里(abort 也要记一条 WAL,见 `xact_redo_abort`)。重放时遇到 abort record,把它的状态在 `pg_xact` 里标成 `aborted`——它的改动**留在页里但被标记为无效**,读时跳过。
3. **崩溃那一刻还在跑、既没 commit 也没 abort 的事务**:它的 commit/abort record **不在 WAL 里**(还没来得及产生)。重放时**永远不会遇到它的提交记录**,所以它在 `pg_xact` 里永远停留在"未提交"状态。PG 恢复结束时,会把所有"redo 点之后、没看到 commit record"的事务统一**当作 abort 处理**(标记成 aborted)。

> **关键洞察**:第③类事务的物理改动(那些新 tuple)其实被 redo 重放出来了——它们留在数据页里。但因为它们的事务被标成 aborted,这些 tuple 的 `t_xmin` 对应 aborted,任何读它们的人都跳过。**它们就像一堆"看不见的幽灵 tuple",躺在页里不碍事**,等 VACUUM(下一章)来清理。PG 完全不需要 undo——把物理改动擦掉。

### 对比 Undo-based 设计:为什么 PG 的选择是"省一个 undo 日志"

不是所有数据库都这么干。MySQL InnoDB、Oracle 这类用的是 **undo-based 设计**:它们的数据页是"原地更新"(改一行就覆盖旧行),所以必须有**一个独立的 undo 日志(回滚段)**,记录"改之前是什么样",用来在 abort 或崩溃恢复时把改动**物理撤销**回去。

> **不这样会怎样(InnoDB 那种原地更新,却没有 undo)**:InnoDB 改一行,是直接把页里那条 tuple 的内容覆盖掉。如果一个事务改了一行、还没 commit 就崩了,页里现在是"新内容"。要是不撤销,这条改动就永久污染了数据——别的查询会直接读到这条未提交的改动(MVCC 的旧版本被覆盖了,读不到旧值)。所以 InnoDB **必须** undo:回滚时按 undo 日志,把页里那条 tuple 改回旧值。

对比一下两种设计的根本差异:

| | PG(MVCC + 仅 redo) | InnoDB/Oracle(原地更新 + redo + undo) |
|---|---|---|
| 改一行怎么做 | 插一条新 tuple,旧行打删除标记 | 原地覆盖旧行 |
| undo 日志 | **不需要**(旧版本就是数据页里的旧行) | **必须有**(undo 表空间) |
| abort 怎么办 | 在 `pg_xact` 标 aborted,物理改动留着当幽灵 | 按 undo 日志把改动擦回旧值 |
| 崩溃恢复 | 只 redo(已 commit 的重做,未 commit 的留幽灵) | redo + undo(已 commit 的重做,未 commit 的撤销) |
| 代价 | 旧版本堆积,需要 VACUUM 清理 | undo 表空间要管理,但旧版本不堆积 |

PG 这个选择是**用空间换简单**:多存几个旧版本(占空间,要 VACUUM 清),换来崩溃恢复"只 redo 不 undo"的简洁——redo 永远是"往前做",没有"往后退"这一步,逻辑统一,实现也更不容易出 bug。而 VACUUM(下一章第 21 章的主题)就是 PG 为这个选择付出的代价。

> 这是全书"快 vs 不丢不乱"二分法的一次精彩体现:InnoDB 选择"原地更新 + undo"是为了**省空间**(旧版本不堆积),代价是 undo 表空间的复杂性和崩溃恢复的双向逻辑;PG 选择"MVCC + 仅 redo"是为了**省逻辑**(崩溃恢复只 redo,统一),代价是旧版本堆积要 VACUUM。两种都"不丢不乱",但走的路截然不同。这也是为什么 PG 体系里 VACUUM 那么重要——它是 MVCC 这套选择的清场机制。

## 三、恢复的全流程:从 ControlFile 找到 redo 起点,然后重放

讲清"为什么只 redo",现在看"redo 怎么走"。崩溃恢复的入口是 `StartupXLOG`(PG 启动时 startup 进程的主函数):

> [src/backend/access/transam/xlog.c:5401](../postgresql-17.0/src/backend/access/transam/xlog.c#L5401)

xlog.c 的文件头注释把它定位得明明白白:

> [src/backend/access/transam/xlog.c:18-22](../postgresql-17.0/src/backend/access/transam/xlog.c#L18-L22)

```c
 * StartupXLOG() is the main entry point of the startup process.  It
 * coordinates database startup, performing WAL recovery, and the
 * transition from WAL recovery into normal operations.
```

`StartupXLOG` 是 startup 进程的"总指挥":它先判断要不要恢复、确定 redo 起点,然后调 redo 主循环,最后把数据库从"恢复中"切换到"正常运行"。整个流程分几个阶段。

### 第一步:读 ControlFile,判断是不是干净关闭的

`StartupXLOG` 一进来,第一件事是检查 `ControlFile`——第 19 章第五件事讲过,checkpoint 完成时会把最近 checkpoint 记录的位置写进 `ControlFile->checkPoint`。ControlFile 在内存里是一份 `ControlFileData` 结构体:

> [src/include/catalog/pg_control.h:104-135](../postgresql-17.0/src/include/catalog/pg_control.h#L104-L135)(节选)

```c
typedef struct ControlFileData
{
	uint64		system_identifier;
	...
	DBState		state;			/* see enum above */
	pg_time_t	time;			/* time stamp of last pg_control update */
	XLogRecPtr	checkPoint;		/* last check point record ptr */

	CheckPoint	checkPointCopy; /* copy of last check point record */
	...
```

关键是 `state` 字段,它是个枚举,记录数据库上次是怎么结束的:

> [src/include/catalog/pg_control.h:89-98](../postgresql-17.0/src/include/catalog/pg_control.h#L89-L98)

```c
typedef enum DBState
{
	DB_STARTUP = 0,
	DB_SHUTDOWNED,				/* cleaned shutdown */
	DB_SHUTDOWNED_IN_RECOVERY,
	DB_SHUTDOWNING,
	DB_IN_CRASH_RECOVERY,
	DB_IN_ARCHIVE_RECOVERY,
	DB_IN_PRODUCTION,
} DBState;
```

`StartupXLOG` 根据 `state` 打日志、判断要不要恢复:

> [src/backend/access/transam/xlog.c:5425-5480](../postgresql-17.0/src/backend/access/transam/xlog.c#L5425-L5480)

```c
	switch (ControlFile->state)
	{
		case DB_SHUTDOWNED:
			/* 干净关闭,这是最常见也最希望的 case */
			ereport(IsPostmasterEnvironment ? LOG : NOTICE,
					(errmsg("database system was shut down at %s",
							str_time(ControlFile->time))));
			break;
		...
		case DB_SHUTDOWNING:
			ereport(LOG,
					(errmsg("database system shutdown was interrupted; last known up at %s",
							str_time(ControlFile->time))));
			break;
		case DB_IN_CRASH_RECOVERY:
			ereport(LOG,
					(errmsg("database system was interrupted while in recovery at %s",
							str_time(ControlFile->time)),
					 errhint("This probably means that some data is corrupted and"
							 " you will have to use the last backup for recovery.")));
			break;
		...
		case DB_IN_PRODUCTION:
			ereport(LOG,
					(errmsg("database system was interrupted; last known up at %s",
							str_time(ControlFile->time))));
			break;
```

`DB_SHUTDOWNED` 是干净关闭(checkpoint 里第 19 章讲过,shutdown checkpoint 会把 state 设成这个)——这种情况通常**不需要崩溃恢复**(只要没有新 WAL 累积)。`DB_SHUTDOWNING`、`DB_IN_PRODUCTION`、`DB_IN_CRASH_RECOVERY` 都意味着上次没干净关——**必须恢复**。

> **为什么要先 fsync 整个数据目录(崩溃场景下)**:注释([xlog.c:5510-5526](../postgresql-17.0/src/backend/access/transam/xlog.c#L5510-L5526))讲了一个微妙点:崩溃后,数据目录里可能有些之前 `write()` 过、但**还没 fsync** 的数据(在 OS 页缓存里)。如果不处理,掉电时这些可能丢。所以崩溃恢复开始前,PG 会调 `SyncDataDirectory()` 把整个数据目录 fsync 一遍,固化已有的写。这是防"二次崩溃丢更多"的保险。

### 第二步:InitWalRecovery——找到最近 checkpoint,确定 redo 起点

接下来 `StartupXLOG` 调 `InitWalRecovery`:

> [src/backend/access/transam/xlog.c:5539-5541](../postgresql-17.0/src/backend/access/transam/xlog.c#L5539-L5541)

```c
	InitWalRecovery(ControlFile, &wasShutdown,
					&haveBackupLabel, &haveTblspcMap);
	checkPoint = ControlFile->checkPointCopy;
```

`InitWalRecovery` 是 PG17 从 `xlog.c` 拆出来的恢复专用函数集合(文件 [src/backend/access/transam/xlogrecovery.c](../postgresql-17.0/src/backend/access/transam/xlogrecovery.c))。它的核心动作是:**用 `ControlFile->checkPoint` 这个 LSN,去 WAL 里读出那条 checkpoint 记录,拿到里面的 `CheckPoint` 结构体(含 redo 起点)**。

`ControlFile->checkPoint` 指向"最近 checkpoint 记录在 WAL 里的位置"。`InitWalRecovery` 用 `ReadCheckpointRecord` 读这条记录:

> [src/backend/access/transam/xlogrecovery.c:775-790](../postgresql-17.0/src/backend/access/transam/xlogrecovery.c#L775-L790)

```c

		CheckPointLoc = ControlFile->checkPoint;
		CheckPointTLI = ControlFile->checkPointCopy.ThisTimeLineID;
		RedoStartLSN = ControlFile->checkPointCopy.redo;
		RedoStartTLI = ControlFile->checkPointCopy.ThisTimeLineID;
		record = ReadCheckpointRecord(xlogprefetcher, CheckPointLoc,
									  CheckPointTLI);
		if (record != NULL)
		{
			ereport(DEBUG1,
					(errmsg_internal("checkpoint record is at %X/%X",
									 LSN_FORMAT_ARGS(CheckPointLoc))));
		}
		else
		{
			/*
			 * We used to attempt to go back to a secondary checkpoint record
			 * here, but only when not in standby mode. We now just fail if we
			 * can't read the last checkpoint because this allows us to
			 * simplify processing around checkpoints.
			 */
			ereport(PANIC,
					(errmsg("could not locate a valid checkpoint record at %X/%X",
							LSN_FORMAT_ARGS(CheckPointLoc))));
		}
		memcpy(&checkPoint, XLogRecGetData(xlogreader), sizeof(CheckPoint));
		wasShutdown = ((record->xl_info & ~XLR_INFO_MASK) == XLOG_CHECKPOINT_SHUTDOWN);
```

注意几个要点:

1. `CheckPointLoc`(checkpoint 记录的位置)和 `RedoStartLSN`(redo 起点)是**两个不同的 LSN**。checkpoint 记录本身在 WAL 流里某个位置,而 redo 起点是这条记录里**存的字段**(`checkPoint.redo`,第 19 章第一节讲过它怎么定)——它指向"checkpoint 之前、所有脏页都已落盘的那一刻"。**重放从 redo 起点开始,不是从 checkpoint 记录本身开始**。
2. 如果连这条 checkpoint 记录都读不到(`record == NULL`),PG 直接 `PANIC`——注释说"现在不再尝试回退到上一个 checkpoint,直接失败"。因为 ControlFile 里存的是最近 checkpoint 的位置,它读不出来意味着元数据严重损坏,硬撑下去只会越错越远。
3. `wasShutdown` 判断这条 checkpoint 是不是 shutdown checkpoint(`XLOG_CHECKPOINT_SHUTDOWN`)——shutdown checkpoint 意味着上次是干净关闭的。

`ReadCheckpointRecord` 自己做了一串校验,确保读到的是真正的 checkpoint 记录:

> [src/backend/access/transam/xlogrecovery.c:4049-4099](../postgresql-17.0/src/backend/access/transam/xlogrecovery.c#L4049-L4099)

```c
	ReadCheckpointRecord(XLogPrefetcher *xlogprefetcher, XLogRecPtr RecPtr,
						 TimeLineID replayTLI)
	{
		XLogRecord *record;
		uint8		info;
		...
		XLogPrefetcherBeginRead(xlogprefetcher, RecPtr);
		record = ReadRecord(xlogprefetcher, LOG, true, replayTLI);
		...
		if (record->xl_rmid != RM_XLOG_ID)
		{
			ereport(LOG, (errmsg("invalid resource manager ID in checkpoint record")));
			return NULL;
		}

		if (info != XLOG_CHECKPOINT_SHUTDOWN &&
			info != XLOG_CHECKPOINT_ONLINE)
		{

			return NULL;
		}
```

校验三件:`xl_rmid` 必须是 `RM_XLOG_ID`(checkpoint 记录是 XLOG 资源管理器产生的);`xl_info` 必须是 shutdown 或 online 两种 checkpoint 之一;记录长度必须正好是"记录头 + 数据头 + `sizeof(CheckPoint)`"。任何一个不对,这条记录就不是合法 checkpoint——返回 NULL,调用方 `PANIC`。这种"宁可不恢复也不要瞎恢复"的严格,是数据库内核的底线:恢复错了比不恢复还糟。

### 第三步:决定要不要恢复——`checkPoint.redo < CheckPointLoc` 这个判据

这是判断"是否需要崩溃恢复"的核心逻辑。读完 checkpoint 记录,`InitWalRecovery` 用一条很精妙的判据决定要不要恢复:

> [src/backend/access/transam/xlogrecovery.c:905-918](../postgresql-17.0/src/backend/access/transam/xlogrecovery.c#L905-L918)

```c
	/*
	 * Check whether we need to force recovery from WAL.  If it appears to
	 * have been a clean shutdown and we did not have a recovery signal file,
	 * then assume no recovery needed.
	 */
	if (checkPoint.redo < CheckPointLoc)
	{
		if (wasShutdown)
			ereport(PANIC,
					(errmsg("invalid redo record in shutdown checkpoint")));
		InRecovery = true;
	}
	else if (ControlFile->state != DB_SHUTDOWNED)
		InRecovery = true;
	else if (ArchiveRecoveryRequested)
	{
		/* force recovery due to presence of recovery signal file */
		InRecovery = true;
	}
```

逐条读这三个判据:

1. **`checkPoint.redo < CheckPointLoc`**:redo 起点**严格小于** checkpoint 记录的位置——意思是 checkpoint 记录产生时,WAL 里已经有 redo 点之后的记录了(那些是 redo 点之后、checkpoint 记录之前产生的 WAL,对应 checkpoint 进行期间还在跑的事务)。这种情况意味着 checkpoint 是 **online checkpoint**(运行中做的),redo 点之后肯定有需要重放的 WAL——`InRecovery = true`。如果是 shutdown checkpoint 但 redo 却小于 CheckPointLoc,那是数据损坏(shutdown checkpoint 的 redo 应该等于自身位置),`PANIC`。
2. **`ControlFile->state != DB_SHUTDOWNED`**:上次不是干净关闭的(`DB_SHUTDOWNING`、`DB_IN_PRODUCTION`、`DB_IN_CRASH_RECOVERY` 等都意味着中断)——必须恢复。
3. **`ArchiveRecoveryRequested`**:数据目录里有 `recovery.signal` 或 `standby.signal` 文件(第 22 章详讲)——用户明确要求做归档恢复,强制进恢复。

> **为什么不直接看 `state` 就够了?** 因为正常运行的数据库,checkpoint 都是 online 的(`XLOG_CHECKPOINT_ONLINE`),它的 `redo < CheckPointLoc` 总成立——但 `state` 此时是 `DB_IN_PRODUCTION`。第②条已经覆盖了这种情况(强制恢复)。第①条覆盖的是更微妙的场景:哪怕 `state` 恰好是 `DB_SHUTDOWNED`(干净关),如果 redo 点居然在 CheckPointLoc 之前(不该发生但发生了),也得恢复探查。三个判据互为补充,把所有"可能有未重放 WAL"的情况都兜住。

一旦 `InRecovery = true`,PG 就更新 ControlFile 的 state:

> [src/backend/access/transam/xlogrecovery.c:925-945](../postgresql-17.0/src/backend/access/transam/xlogrecovery.c#L925-L945)

```c
	if (InRecovery)
	{
		if (InArchiveRecovery)
		{
			ControlFile->state = DB_IN_ARCHIVE_RECOVERY;
		}
		else
		{
			ereport(LOG,
					(errmsg("database system was not properly shut down; "
							"automatic recovery in progress")));
			...
			ControlFile->state = DB_IN_CRASH_RECOVERY;
		}
```

崩溃恢复(没有 recovery.signal 文件)进 `DB_IN_CRASH_RECOVERY`,归档恢复(有信号文件)进 `DB_IN_ARCHIVE_RECOVERY`。这条日志"database system was not properly shut down; automatic recovery in progress"——你 `kill -9` 重启数据库时一定见过。

### 第四步:初始化全局计数器,准备 redo

回到 `StartupXLOG`,恢复决定后,它从 checkpoint 记录里恢复各种全局状态:

> [src/backend/access/transam/xlog.c:5543-5552](../postgresql-17.0/src/backend/access/transam/xlog.c#L5543-L5552)

然后设 redo 起点这个关键变量:

> [src/backend/access/transam/xlog.c:5661](../postgresql-17.0/src/backend/access/transam/xlog.c#L5661)

```c

```

`RedoRecPtr`(本地)、`XLogCtl->RedoRecPtr`(共享)——两个 redo 起点副本都设成 `checkPoint.redo`。这个值后续会被 FPI 判断用到(第 18 章第五节:`page_lsn <= RedoRecPtr` 就要加 FPI)。

### 第五步:PerformWalRecovery——真正的 redo 主循环

准备工作做完,真正的重放在这里:

> [src/backend/access/transam/xlog.c:5819-5821](../postgresql-17.0/src/backend/access/transam/xlog.c#L5819-L5821)

```c
		PerformWalRecovery();
		performedWalRecovery = true;
```

`PerformWalRecovery` 是 redo 主循环(下一节"关键源码精读"详讲)。它从 redo 起点读第一条 record,然后循环:读一条 → 分发到 `rm_redo` 重放 → 读下一条,直到 WAL 末尾(`ReadRecord` 返回 NULL,意味着读不到下一条了,说明重放到了已 fsync 的 WAL 的尽头)。下一节专门精读。

### 第六步:恢复 2PC prepared 事务,切换到正常运行

redo 主循环结束后,还有几件收尾的事。其中最重要的是处理 **2PC(两阶段提交)的 prepared 事务**。

2PC 是分布式事务用的协议:一个事务先 `PREPARE`(写一条 PREPARE record 到 WAL,把事务状态持久化成"prepared"),之后某个时刻再 `COMMIT PREPARED` 或 `ROLLBACK PREPARED`。崩溃时,如果某个事务处于 prepared 状态(已 PREPARE 但还没 COMMIT/ROLLBACK),它的 PREPARE record 在 WAL 里——重放时会重建这个 prepared 事务。重启后这个事务**仍然处于 prepared 状态**,等应用层来决定 COMMIT 还是 ROLLBACK。

PG 在恢复结束时调 `RecoverPreparedTransactions` 把这些 prepared 事务重新"挂起来"——恢复它们的锁、占用的资源,让它们继续等:

> [src/backend/access/transam/xlog.c:6076-6078](../postgresql-17.0/src/backend/access/transam/xlog.c#L6076-L6078)

```c
	/*
	 * Reload shared-memory state for prepared transactions.  This needs to
	 * happen before renaming the last partial segment of the old timeline as
	 * it may be possible that we have to recovery some transactions from it.
	 */
	RecoverPreparedTransactions();
```

2PC 恢复的细节下面第六节专讲。最后,把数据库从"恢复中"切到"正常运行"——`InRecovery = false`:

> [src/backend/access/transam/xlog.c:6028-6030](../postgresql-17.0/src/backend/access/transam/xlog.c#L6028-L6030)

```c
	/*
	 * Okay, we're officially UP.
	 */
	InRecovery = false;
```

到这一行,数据库就算"开门营业"了,可以接受连接。这一行之前发生的一切(读 ControlFile → 找 checkpoint → redo 起点 → 重放 WAL → 恢复 2PC),就是崩溃恢复的全过程。

> **崩溃恢复 vs 干净重启的快慢差异**:如果 `state == DB_SHUTDOWNED` 且 redo 点正好在 CheckPointLoc(意味着 checkpoint 之后没有任何新 WAL),`InRecovery` 保持 false,跳过整个 `PerformWalRecovery`——重启非常快,基本只花在初始化共享内存上。这就是为什么干净关库再启动很快,而崩溃重启要慢——后者要重放 redo 点之后的所有 WAL。第 19 章讲"checkpoint 越近崩溃恢复越快"的根因就在这里:要重放的 WAL 越少,重启开门越快。

## 四、崩溃恢复的判断时机:为什么 redo 不会"重过头"也不会"漏一条"

讲完流程,这里要专门抠一个新手最容易困惑的细节:**redo 从 redo 起点开始重放,但 redo 点之后、崩溃之前,有些脏页可能已经被 backend 或 bgwriter 顺手刷盘了(只是不完整)**。重放这些已经在磁盘上的改动,会不会"做多了"?答案是不会——靠 LSN 比较实现 redo 的幂等性。

回忆第 18 章第五节讲过的 LSN 的第三个用途:"redo 时的判据"。崩溃恢复重放某条 record 时,要先读它影响的页,比较 `page->pd_lsn` 和 `record->lsn`——如果 `page->pd_lsn >= record->lsn`,说明这页的改动**已经在磁盘上了**(这页的 LSN 比这条记录还新,意味着这条记录对应的改动早已应用),**跳过这条 redo**;否则才重放。这是 redo "不重复劳动、且幂等"的关键。

这个判据在源码里是 `XLogReadBufferForRedo` 系列函数。每个资源管理器的 redo(比如 `heap_redo`),处理一条 record 时,对 record 里引用的每个 block,先调 `XLogReadBufferForRedo` 把页读进来并判断要不要 redo:

> [src/backend/access/transam/xlogutils.c:277-313](../postgresql-17.0/src/backend/access/transam/xlogutils.c#L277-L313)

```c
/*
 * XLogReadBufferForRedo
 *		Read a page during XLOG replay
 *
 * Reads a block referenced by a WAL record into shared buffer cache, and
 * determines what needs to be done to redo the changes to it.  If the WAL
 * record includes a full-page image of the page, it is restored.
 *
 * 'record.EndRecPtr' is compared to the page's LSN to determine if the record
 * has already been replayed.  'block_id' is the ID number the block was
 * registered with, when the WAL record was created.
 *
 * Returns one of the following:
 *
 *	BLK_NEEDS_REDO	- changes from the WAL record need to be applied
 *	BLK_DONE		- block doesn't need replaying
 *	BLK_RESTORED	- block was restored from a full-page image included in
 *					  the record
 *	BLK_NOTFOUND	- block was not found (because it was truncated away by
 *					  an operation later in the WAL stream)
 ...
 */
XLogRedoAction
XLogReadBufferForRedo(XLogReaderState *record, uint8 block_id,
					  Buffer *buf)
{
	return XLogReadBufferForRedoExtended(record, block_id, RBM_NORMAL,
										 false, buf);
}
```

注释里那句"'record.EndRecPtr' is compared to the page's LSN to determine if the record has already been replayed"——这就是 LSN 比较的精髓。返回值有四种,关键是 `BLK_NEEDS_REDO`(需要重放)和 `BLK_DONE`(这页已经够新,跳过):

> [src/include/access/xlogutils.h:71-74](../postgresql-17.0/src/include/access/xlogutils.h#L71-L74)

```c
	BLK_NEEDS_REDO,				/* changes from the WAL record need to be applied */
	BLK_DONE,					/* block is already up-to-date */
	BLK_RESTORED,				/* block was restored from a full-page image */
	BLK_NOTFOUND,				/* block was not found (and hence does not need
								   redo) */
```

`XLogReadBufferForRedoExtended` 里这个比较的代码:

> [src/backend/access/transam/xlogutils.c:435-444](../postgresql-17.0/src/backend/access/transam/xlogutils.c#L435-L444)

```c
			if (lsn <= PageGetLSN(BufferGetPage(*buf)))
				return BLK_DONE;
			else
				return BLK_NEEDS_REDO;
```

`lsn` 是当前 record 的 `EndRecPtr`,`PageGetLSN(...)` 是这页的 `pd_lsn`。如果 `record.lsn <= page.lsn`(这页的 LSN 已经 >= 这条记录),返回 `BLK_DONE`——这条 record 对这页的改动已经在磁盘上,跳过。否则 `BLK_NEEDS_REDO`——还没应用,重放。

> **这就回答了"会不会 redo 多了"**:哪怕某条 record 对应的改动崩溃前已经被刷盘了,redo 时一比 LSN,发现这页的 LSN 已经超过这条 record,直接跳过(`BLK_DONE`)。**redo 不会重复应用,天然幂等**。这也是为什么崩溃恢复可以放心地"从 redo 起点无脑重放到末尾"——靠 LSN 比较自动跳过已应用的部分,不需要精确知道"哪条刷了哪条没刷"。

## 五、FPI:幂等性能成立的隐含前提,以及为什么 redo 必须幂等

上一节讲"靠 LSN 比较跳过已应用的 record"。可这里藏着一个深层前提:**这个比较只有在"磁盘上的页是完整且正确"时才成立**。如果磁盘上的页是破损的(torn page,见第 18 章第七节),`pd_lsn` 可能是个乱七八糟的值,LSN 比较就不可靠——这正是 FPI 要救的场。

### 崩溃恢复可能重放"已经部分应用过"的 record

崩溃恢复并不总是一次性、从头到尾顺顺当当跑完的。考虑这些场景:

1. **崩溃恢复中途又崩了**:恢复到一半,机器又掉电。重启时再次恢复——这次又要从 redo 点开始重放,会重放**上次恢复已经重放过一遍的 record**。
2. **崩溃前 backend 已经刷了一部分脏页**:这些刷过的页,对应的 record 还在 redo 范围里(它们在 redo 点之后产生),恢复时会再次遇到——但页已经有了这些改动。
3. **2PC prepared 事务的 redo**:PREPARE record 可能在前一次恢复里已经被处理过。

这些场景的共同点是:**同一个 record 可能被重放不止一次**。所以 redo **必须幂等(idempotent)**——重放一条 record 任意多次,结果都和重放一次一样。

> **不这样会怎样(redo 不幂等)**:假设 redo 不是幂等的——某条 record "对 block X 加 1",重放两次就变成加 2。崩溃恢复里这会立刻出错:重放一遍 +1,中途崩了,再恢复又重放一遍又 +1,数据彻底错乱。所以 PG 的 redo 必须设计成幂等。

PG 的 redo 幂等性来自两层:

**第一层:LSN 比较跳过已应用的**(上一节讲过)。这一层要求页是完整的——能正确读出 `pd_lsn`。

**第二层:FPI 兜底破损页**。如果崩溃前 backend 刷盘时发生 partial write(8KB 页只写了 4KB),`pd_lsn` 在破损页上可能是错的。LSN 比较会失效。这时 FPI 接管——这条 record(它是 checkpoint 后首次改这页时产生的,带整页镜像)重放时,直接**用 FPI 把整页重置成 checkpoint 时的样子**,然后在干净的页基础上重放之后的增量。

这就是 `XLogReadBufferForRedoExtended` 里 FPI 分支干的事:

> [src/backend/access/transam/xlogutils.c:388-408](../postgresql-17.0/src/backend/access/transam/xlogutils.c#L388-L408)

```c
	/* If it has a full-page image and it should be restored, do it. */
	if (XLogRecBlockImageApply(record, block_id))
	{
		Assert(XLogRecHasBlockImage(record, block_id));
		*buf = XLogReadBufferExtended(rlocator, forknum, blkno,
									  get_cleanup_lock ? RBM_ZERO_AND_CLEANUP_LOCK : RBM_ZERO_AND_LOCK,
									  prefetch_buffer);
		page = BufferGetPage(*buf);
		if (!RestoreBlockImage(record, block_id, page))
			ereport(ERROR,
					(errcode(ERRCODE_INTERNAL_ERROR),
					 errmsg_internal("%s", record->errormsg_buf)));

		/*
		 * The page may be uninitialized. If so, we can't set the LSN because
		 * that would corrupt the page.
		 */
		if (!PageIsNew(page))
		{
			PageSetLSN(page, lsn);
		}

		MarkBufferDirty(*buf);
		...
		return BLK_RESTORED;
	}
```

这段把第 18 章讲的 FPI 落地:如果这条 record 带了 FPI(`XLogRecHasBlockImage`),并且 FPI 需要 apply(`XLogRecBlockImageApply`),就用 `RestoreBlockImage` 把 record 里存的整页镜像**直接覆盖**到数据页上(不管磁盘上原本是什么——破损页也好、新页也好,统统被 FPI 重置成 checkpoint 时的干净状态)。然后 `PageSetLSN(page, lsn)` 把页的 LSN 设成这条 record 的 LSN,返回 `BLK_RESTORED`。

> **关键洞察:FPI 让 redo 跳过了"破损页"这个失败模式**。没有 FPI,partial write 后,基于破损页做增量 redo 会算错;有了 FPI,首次改某页的 record 自带"这一整页应该是什么样",直接重置页到已知干净状态,然后再增量 redo——破损页这个不确定性被消除了。

`XLogReadBufferForRedo` 的文档注释把这个权衡讲得最透:

> [src/backend/access/transam/xlogutils.c:301-313](../postgresql-17.0/src/backend/access/transam/xlogutils.c#L301-L313)

```c
 * Note: when a backup block is available in XLOG with the BKPIMAGE_APPLY flag
 * set, we restore it, even if the page in the database appears newer.  This
 * is to protect ourselves against database pages that were partially or
 * incorrectly written during a crash.  We assume that the XLOG data must be
 * good because it has passed a CRC check, while the database page might not
 * be.  This will force us to replay all subsequent modifications of the page
 * that appear in XLOG, rather than possibly ignoring them as already
 * applied, but that's not a huge drawback.
```

**这段注释是 FPI 设计思想的精华**:哪怕数据库页看起来比 record 新,只要 record 带了 FPI 该 apply,就**强制用 FPI 覆盖**。理由是 WAL 通过了 CRC 校验(可信),而数据页可能是 partial write 写坏的(不可信)。"宁可 redo 多一点(后面的增量改动都要重放),也不要基于一个可能破损的页省那点 redo"——这是可靠性的优先。

> **回扣第 18、19 章**:第 18 章讲 FPI 时埋下"防 torn page"的钩子,第 19 章讲"checkpoint 越频繁 FPI 越多"。这一节把这两个钩子都兑现了:崩溃恢复里,FPI 是幂等性成立的隐含前提(没有它,LSN 比较在破损页上失效);而 checkpoint 间隔越短,FPI 触发越频繁——这是用 WAL 体积换恢复可靠性。整条 WAL-FPI-redo 的设计是一个自洽的闭环。

## 六、2PC prepared 事务的恢复:那些"挂着"的事务怎么办

第二节讲未提交事务时,我们说"崩溃那一刻还在跑的事务当 abort 处理"。但有一类事务是例外:**已经 `PREPARE` 但还没 `COMMIT PREPARED`/`ROLLBACK PREPARED` 的 2PC 事务**。这种事务在崩溃时**既不能当 abort 也不能当 commit**——它的状态是"prepared",应用层(通常是分布式事务协调者)还没下最终决定。

PG 的处理是:**重放 PREPARE record 时,把这个事务重建为 prepared 状态,继续挂着等应用层决定**。这是 2PC 协议的核心保证——prepared 事务必须**跨崩溃存活**。

`xact_redo` 处理 PREPARE record 的分支:

> [src/backend/access/transam/xact.c:6349-6362](../postgresql-17.0/src/backend/access/transam/xact.c#L6349-L6362)

```c
	else if (info == XLOG_XACT_PREPARE)
	{
		/*
		 * Store xid and start/end pointers of the WAL record in TwoPhaseState
		 * gxact entry.
		 */
		LWLockAcquire(TwoPhaseStateLock, LW_EXCLUSIVE);
		PrepareRedoAdd(XLogRecGetData(record),
					   record->ReadRecPtr,
					   record->EndRecPtr,
					   XLogRecGetOrigin(record));
		LWLockRelease(TwoPhaseStateLock);
	}
```

`PrepareRedoAdd` 把这个 prepared 事务加进 `TwoPhaseState`(两阶段事务的全局表)。注意它只记了 WAL 位置,真正的 prepared 事务数据(事务内容、锁、子事务等)存在 2PC 文件里(数据目录 `pg_twophase/` 下,按事务号命名的文件)——崩溃前 PG 会把这些 prepared 状态写盘,恢复时从盘上读回来。

PG17 在 `StartupXLOG` 里很早就调 `restoreTwoPhaseData`,把磁盘上的 2PC 文件预扫进 `TwoPhaseState`:

> [src/backend/access/transam/xlog.c:5633-5642](../postgresql-17.0/src/backend/access/transam/xlog.c#L5633-L5642)

```c
	/*
	 * Before running in recovery, scan pg_twophase and fill in its status to
	 * be able to work on entries generated by redo.  Doing a scan before
	 * taking any recovery action has the merit to discard any 2PC files that
	 * are newer than the first record to replay, saving from any conflicts at
	 * replay.  This avoids as well any subsequent scans when doing recovery
	 * of the on-disk two-phase data.
	 */
	restoreTwoPhaseData();
```

注释里那个"discard any 2PC files that are newer than the first record to replay"——很重要:**比 redo 起点还新的 2PC 文件直接丢弃**。为什么?因为 redo 会重放这些事务的 PREPARE record,会重新生成对应的 2PC 状态;如果盘上还留着旧版的(可能是 partial write 写坏的),会冲突。先清场,后重放,避免冲突。

恢复结束、redo 跑完后,调 `PrescanPreparedTransactions`(扫描所有 prepared 事务,确定事务号范围)和 `RecoverPreparedTransactions`(真正重建它们的锁、子事务状态等):

> [src/backend/access/transam/xlog.c:5893-5898](../postgresql-17.0/src/backend/access/transam/xlog.c#L5893-L5898)

```c
	/*
	 * Pre-scan prepared transactions to find out the range of XIDs present.
	 * This information is not quite needed yet, but it is positioned here so
	 * as potential problems are detected before any on-disk change is done.
	 */
	oldestActiveXID = PrescanPreparedTransactions(NULL, NULL);
```

`RecoverPreparedTransactions` 在更靠后的位置调用,把每个 prepared 事务恢复成"挂着等决定"的状态:

> [src/backend/access/transam/twophase.c:2083-2130](../postgresql-17.0/src/backend/access/transam/twophase.c#L2083-L2130)(节选)

```c
RecoverPreparedTransactions(void)
{
	int			i;

	LWLockAcquire(TwoPhaseStateLock, LW_EXCLUSIVE);
	for (i = 0; i < TwoPhaseState->numPrepXacts; i++)
	{
		TransactionId xid;
		char	   *buf;
		GlobalTransaction gxact = TwoPhaseState->prepXacts[i];
		...
		buf = ProcessTwoPhaseBuffer(xid,
									gxact->prepare_start_lsn,
									gxact->ondisk, true, false);
		if (buf == NULL)
			continue;

		ereport(LOG,
				(errmsg("recovering prepared transaction %u from shared memory", xid)));
		...
		MarkAsPreparingGuts(gxact, xid, gid,
							hdr->prepared_at,
							hdr->owner, hdr->database);

		/* recovered, so reset the flag for entries generated by redo */
		gxact->inredo = false;

		GXactLoadSubxactData(gxact, hdr->nsubxacts, subxids);
		MarkAsPrepared(gxact, true);
		...
		/*
		 * Recover other state (notably locks) using resource managers.
		 */
		ProcessRecords(bufptr, xid, twophase_recover_callbacks);
```

读这段:`RecoverPreparedTransactions` 遍历每个 prepared 事务,读它的 2PC 文件内容,用 `MarkAsPreparingGuts` 重建它的 `GlobalTransaction` 结构,加载子事务数据,标记为 prepared,然后用 `ProcessRecords` 配合资源管理器回调恢复它的**锁**等状态。重放完成后,这些事务就和正常运行时 prepared 的事务一样——锁还占着、状态是 prepared,等应用层来 `COMMIT PREPARED`/`ROLLBACK PREPARED`。

> **2PC 的崩溃恢复保证**:这是 2PC 协议"原子提交"能跨崩溃成立的根基。prepared 事务跨崩溃存活,意味着分布式事务的协调者崩了重启,依然能找到当时已经 prepared 的事务,继续决定提交还是回滚——不会出现"prepare 成功但事务丢了"的中间态。这个保证比单事务的持久性更强(单事务要么 commit 要么 abort,2PC 多了一个"prepared"这个能跨崩溃的中间态)。

## 七、什么算"恢复完成":minimal recovery point 与开门营业

讲完重放,回答最后一个问题:**重放到哪算"完成"?什么时候可以接受连接?**

崩溃恢复(没有 `standby.signal`/`recovery.signal`)的答案很简单——**重放到 WAL 末尾就算完成**。redo 主循环 `PerformWalRecovery` 里,`ReadRecord` 返回 NULL(读不到下一条 record 了,意味着已 fsync 的 WAL 到此为止),循环结束,恢复完成。然后 `StartupXLOG` 把 state 设回正常、`InRecovery = false`、可以接受连接。

但归档恢复和备机恢复(第 22 章详讲)就复杂——它们要重放到一个**特定的"一致点"**才能开门。这个一致点叫 **minimal recovery point(最小恢复点)**,存在 ControlFile 里:

> [src/include/catalog/pg_control.h:135-155](../postgresql-17.0/src/include/catalog/pg_control.h#L135-L155)(节选)

```c
	/*
	 * These two values determine the minimum point we must recover up to
	 * before starting up:
	 *
	 * minRecoveryPoint is updated to the latest replayed LSN whenever we
	 * flush a data change during archive recovery. That guards against
	 * starting archive recovery, aborting it, and restarting with an earlier
	 * stop location. If we've already flushed data changes from WAL record X
	 * to disk, we mustn't start up until we reach X again. Zero when not
	 * doing archive recovery.
	 */
	XLogRecPtr	minRecoveryPoint;
	TimeLineID	minRecoveryPointTLI;
```

注释解释了 `minRecoveryPoint` 的作用:防止"恢复到一半中断,重启后从一个更早的位置认为已经一致了"。每重放一条改动数据的 record,就把 `minRecoveryPoint` 推进到这条 record 的 LSN。这样,如果恢复中断重启,必须至少重放到上次中断的位置,才允许开门——否则可能用半残状态提供服务。

判断"是否达到一致、可以开门"的逻辑在 `CheckRecoveryConsistency`:

> [src/backend/access/transam/xlogrecovery.c:2175-2190](../postgresql-17.0/src/backend/access/transam/xlogrecovery.c#L2175-L2190)

```c
CheckRecoveryConsistency(void)
{
	XLogRecPtr	lastReplayedEndRecPtr;
	TimeLineID	lastReplayedTLI;

	/*
	 * During crash recovery, we don't reach a consistent state until we've
	 * replayed all the WAL.
	 */
	if (XLogRecPtrIsInvalid(minRecoveryPoint))
		return;

	Assert(InArchiveRecovery);
```

注意第一段注释:"During crash recovery, we don't reach a consistent state until we've replayed all the WAL."——**崩溃恢复(非归档恢复)时,重放完所有 WAL 才一致**;归档恢复则有 `minRecoveryPoint` 这个明确的目标点。`minRecoveryPoint` 无效(崩溃恢复时)就直接返回,不做一致性判断——因为崩溃恢复的"一致"就是"WAL 末尾",由主循环自然达到。

> [src/backend/access/transam/xlogrecovery.c:2247-2262](../postgresql-17.0/src/backend/access/transam/xlogrecovery.c#L2247-L2262)

```c
	if (!reachedConsistency && !backupEndRequired &&
		minRecoveryPoint <= lastReplayedEndRecPtr)
	{
		/*
		 * Check to see if the XLOG sequence contained any unresolved
		 * references to uninitialized pages.
		 */
		XLogCheckInvalidPages();

		/*
		 * Check that pg_tblspc doesn't contain any real directories.
		 */
		CheckTablespaceDirectory();

		reachedConsistency = true;
		ereport(LOG,
				(errmsg("consistent recovery state reached at %X/%X",
						LSN_FORMAT_ARGS(lastReplayedEndRecPtr))));
	}
```

`minRecoveryPoint <= lastReplayedEndRecPtr`——重放进度追上了最小恢复点,就 `reachedConsistency = true`,打日志"consistent recovery state reached"。备机到了这个点,就可以接受只读连接(hot standby,第 22 章)。

> **崩溃恢复是归档恢复的特例**:看这一节你会发现,崩溃恢复其实是"恢复到一致点"机制的一个**特例**——它的"一致点"就是 WAL 末尾。归档恢复(从 basebackup 起步,重放归档 WAL)和备机恢复(持续 apply 主机来的 WAL)用的是**同一套 redo 主循环、同一套一致性判断**,只是"什么时候算完"的判据不同。这就是为什么本章开头强调"备机 apply 就是崩溃恢复"——同一个 `PerformWalRecovery`,同一个 `CheckRecoveryConsistency`,只是 WAL 来源(本地文件 vs 归档 vs 网络)和停止条件不同。第 22 章会从这个连接点展开讲复制。

## 关键源码精读:`PerformWalRecovery`——redo 主循环

这一章最值得精读的是 `PerformWalRecovery`([src/backend/access/transam/xlogrecovery.c:1652](../postgresql-17.0/src/backend/access/transam/xlogrecovery.c#L1652))——它是崩溃恢复的"心脏",所有 redo 都在这里循环。它也是第 22 章备机 apply 的心脏(同一套代码)。

### 起手:把进度计数器初始化到 redo 起点

`PerformWalRecovery` 一进来,先把"已重放到哪"的共享计数器初始化到 redo 起点(或 checkpoint 本身,看哪种):

> [src/backend/access/transam/xlogrecovery.c:1658-1685](../postgresql-17.0/src/backend/access/transam/xlogrecovery.c#L1658-L1685)

```c
PerformWalRecovery(void)
{
	XLogRecord *record;
	bool		reachedRecoveryTarget = false;
	TimeLineID	replayTLI;

	/*
	 * Initialize shared variables for tracking progress of WAL replay, as if
	 * we had just replayed the record before the REDO location (or the
	 * checkpoint record itself, if it's a shutdown checkpoint).
	 */
	SpinLockAcquire(&XLogRecoveryCtl->info_lck);
	if (RedoStartLSN < CheckPointLoc)
	{
		XLogRecoveryCtl->lastReplayedReadRecPtr = InvalidXLogRecPtr;
		XLogRecoveryCtl->lastReplayedEndRecPtr = RedoStartLSN;
		XLogRecoveryCtl->lastReplayedTLI = RedoStartTLI;
	}
	else
	{
		XLogRecoveryCtl->lastReplayedReadRecPtr = xlogreader->ReadRecPtr;
		XLogRecoveryCtl->lastReplayedEndRecPtr = xlogreader->EndRecPtr;
		XLogRecoveryCtl->lastReplayedTLI = CheckPointTLI;
	}
	...
	SpinLockRelease(&XLogRecoveryCtl->info_lck);
```

`lastReplayedEndRecPtr` 是"已重放的最后一条 record 的结尾 LSN"——这是恢复进度的"水位线",备机的 walsender 反馈、minRecoveryPoint 推进、一致性判断都看它。这里把它初始化到 redo 起点(或 checkpoint),意味着"重放即将从这里开始"。

### 找第一条 record:从 redo 点开始,可能要先回退到 redo record

> [src/backend/access/transam/xlogrecovery.c:1703-1735](../postgresql-17.0/src/backend/access/transam/xlogrecovery.c#L1703-L1735)

```c
	/*
	 * Find the first record that logically follows the checkpoint --- it
	 * might physically precede it, though.
	 */
	if (RedoStartLSN < CheckPointLoc)
	{
		/* back up to find the record */
		replayTLI = RedoStartTLI;
		XLogPrefetcherBeginRead(xlogprefetcher, RedoStartLSN);
		record = ReadRecord(xlogprefetcher, PANIC, false, replayTLI);

		/*
		 * If a checkpoint record's redo pointer points back to an earlier
		 * LSN, the record at that LSN should be an XLOG_CHECKPOINT_REDO
		 * record.
		 */
		if (record->xl_rmid != RM_XLOG_ID ||
			(record->xl_info & ~XLR_INFO_MASK) != XLOG_CHECKPOINT_REDO)
			ereport(FATAL,
					(errmsg("unexpected record type found at redo point %X/%X",
							LSN_FORMAT_ARGS(xlogreader->ReadRecPtr))));
	}
	else
	{
		/* just have to read next record after CheckPoint */
		Assert(xlogreader->ReadRecPtr == CheckPointLoc);
		replayTLI = CheckPointTLI;
		record = ReadRecord(xlogprefetcher, LOG, false, replayTLI);
	}
```

注意 `RedoStartLSN < CheckPointLoc` 这个分支:如果 redo 点在 checkpoint 记录之前(online checkpoint 的常见情况),要**回退**到 redo 点去读第一条 record。第 19 章讲过,PG17 用 `XLOG_CHECKPOINT_REDO` 这条特殊 record 来标记 redo 起点——这里校验读到的第一条 record 必须是 `XLOG_CHECKPOINT_REDO`,否则 FATAL(数据不一致)。

`ReadRecord` 是从指定 LSN 读下一条 WAL record 的函数。它处理 record 跨段、CRC 校验、跨页边界等所有底层细节,返回完整的 `XLogRecord *`。读不到(到 WAL 末尾)就返回 NULL。

### 主循环:读一条 → 分发到 rm_redo → 读下一条

找到第一条 record 后,进入主循环:

> [src/backend/access/transam/xlogrecovery.c:1748-1830](../postgresql-17.0/src/backend/access/transam/xlogrecovery.c#L1748-L1830)(节选)

```c
	if (record != NULL)
	{
		...
		InRedo = true;

		RmgrStartup();

		ereport(LOG,
				(errmsg("redo starts at %X/%X",
						LSN_FORMAT_ARGS(xlogreader->ReadRecPtr))));
		...
		/*
		 * main redo apply loop
		 */
		do
		{
			...
			/* Handle interrupt signals of startup process */
			HandleStartupProcInterrupts();
			...
			/*
			 * Have we reached our recovery target?
			 */
			if (recoveryStopsBefore(xlogreader))
			{
				reachedRecoveryTarget = true;
				break;
			}
			...
			/*
			 * Apply the record
			 */
			ApplyWalRecord(xlogreader, record, &replayTLI);

			/* Exit loop if we reached inclusive recovery target */
			if (recoveryStopsAfter(xlogreader))
			{
				reachedRecoveryTarget = true;
				break;
			}

			/* Else, try to fetch the next WAL record */
			record = ReadRecord(xlogprefetcher, LOG, false, replayTLI);
		} while (record != NULL);
```

逐段读这个循环:

1. **`InRedo = true`**:打"正在 redo"标记。第 18 章讲过 `XLogBeginInsert` 里有 `if (!XLogInsertAllowed()) elog(ERROR, "cannot make new WAL entries during recovery")`——`XLogInsertAllowed` 内部就检查 `InRedo`。**redo 期间不允许产生新 WAL**,这是铁律(否则会循环:重放产生新 record,新 record 又要重放)。
2. **`RmgrStartup()`**:启动所有资源管理器。每个 rmgr 在 `RmgrTable` 里登记了 `rm_startup` 钩子,redo 开始前调用——比如 B 树 redo 前要初始化一些状态(`btree_xlog_startup`)。`RmgrStartup` 遍历整张表逐个调用:

   > [src/backend/access/transam/rmgr.c:58-70](../postgresql-17.0/src/backend/access/transam/rmgr.c#L58-L70)

   ```c
   void
   RmgrStartup(void)
   {
       for (int rmid = 0; rmid <= RM_MAX_ID; rmid++)
       {
           if (!RmgrIdExists(rmid))
               continue;

           if (RmgrTable[rmid].rm_startup != NULL)
               RmgrTable[rmid].rm_startup();
       }
   }
   ```

3. **`"redo starts at %X/%X"`** 日志:你崩溃重启数据库时一定见过这条日志——它告诉你 redo 从哪个 LSN 开始。
4. **每轮循环**:`HandleStartupProcInterrupts`(处理信号,比如 recovery 暂停请求)→ `recoveryStopsBefore`(检查是否到达恢复目标,PITR 用)→ `ApplyWalRecord`(真正重放这条 record,下面详讲)→ `recoveryStopsAfter`(恢复目标检查)→ `ReadRecord`(读下一条)。
5. **循环退出条件**:`record == NULL`,即 `ReadRecord` 读不到下一条了——意味着已 fsync 的 WAL 到此为止,重放完成。日志会打"redo done at ..."。

### ApplyWalRecord:按 xl_rmid 分发到 rm_redo

每条 record 的重放在 `ApplyWalRecord`:

> [src/backend/access/transam/xlogrecovery.c:2016-2022](../postgresql-17.0/src/backend/access/transam/xlogrecovery.c#L2016-L2022)

```c
	/*
	 * Some XLOG record types that are related to recovery are processed
	 * directly here, rather than in xlog_redo()
	 */
	if (record->xl_rmid == RM_XLOG_ID)
		xlogrecovery_redo(xlogreader, *replayTLI);

	/* Now apply the WAL record itself */
	GetRmgr(record->xl_rmid).rm_redo(xlogreader);
```

这两行是整个崩溃恢复的精华,逐字读:

1. **`if (record->xl_rmid == RM_XLOG_ID) xlogrecovery_redo(...)`**:`RM_XLOG_ID` 是 XLOG 资源管理器(管 checkpoint、timeline 切换、backup end 这类系统级 record)。这类 record 的处理涉及恢复流程本身(比如遇到 backup_end record 时推进 backup 状态),所以在通用的 `rm_redo` 分发**之前**先单独处理。`xlogrecovery_redo` 处理的就是 `XLOG_BACKUP_END`、`XLOG_OVERWRITE_CONTRECORD` 这些。

2. **`GetRmgr(record->xl_rmid).rm_redo(xlogreader)`**:**这是崩溃恢复与资源管理器的接口,是第 18 章那节"rmgr"的兑现**。`GetRmgr(rmid)` 从 `RmgrTable` 取出这个 rmid 对应的 `RmgrData`,然后调它的 `rm_redo` 函数指针。第 18 章讲过:

   - `RM_HEAP_ID` 的 `rm_redo` 是 `heap_redo`(重放 heap 表的 INSERT/DELETE/UPDATE);
   - `RM_XACT_ID` 的 `rm_redo` 是 `xact_redo`(重放 commit/abort/prepare);
   - `RM_BTREE_ID` 的 `rm_redo` 是 `btree_redo`……

   每类对象的改动,由它自己的 rm_redo 重放。这就是为什么 WAL 能通用地记录所有对象的改动——它只负责把 record 读出来按 rmid 分发,具体怎么 redo 各归各家。

   `GetRmgr` 本身就是个带边界检查的数组索引:

   > [src/include/access/xlog_internal.h:376-383](../postgresql-17.0/src/include/access/xlog_internal.h#L376-L383)

   ```c
   static inline RmgrData
   GetRmgr(RmgrId rmid)
   {
       if (unlikely(!RmgrIdExists(rmid)))
           RmgrNotFound(rmid);
       return RmgrTable[rmid];
   }
   ```

   `RmgrId` 是个 `uint8`(见 [src/include/access/rmgr.h:11](../postgresql-17.0/src/include/access/rmgr.h#L11)),所以最多 256 个 rmgr,够用。`RmgrIdExists` 检查这个 rmid 是否已注册(没注册的 rmid 调用是 bug)。

3. 之后还有一致性检查(`XLR_CHECK_CONSISTENCY` 标志位)、错误回栈更新、唤醒 walsender 等收尾。

### `heap_redo`:资源管理器 redo 的典型样子

看一个具体的 rm_redo——`heap_redo`,管堆表(普通表)的改动重放:

> [src/backend/access/heap/heapam.c:9880-9926](../postgresql-17.0/src/backend/access/heap/heapam.c#L9880-L9926)

```c
heap_redo(XLogReaderState *record)
{
	uint8		info = XLogRecGetInfo(record) & ~XLR_INFO_MASK;

	/*
	 * These operations don't overwrite MVCC data so no conflict processing is
	 * required. The ones in heap2 rmgr do.
	 */

	switch (info & XLOG_HEAP_OPMASK)
	{
		case XLOG_HEAP_INSERT:
			heap_xlog_insert(record);
			break;
		case XLOG_HEAP_DELETE:
			heap_xlog_delete(record);
			break;
		case XLOG_HEAP_UPDATE:
			heap_xlog_update(record, false);
			break;
		case XLOG_HEAP_TRUNCATE:
			/*
			 * TRUNCATE is a no-op because the actions are already logged as
			 * SMGR WAL records.  TRUNCATE WAL record only exists for logical
			 * decoding.
			 */
			break;
		case XLOG_HEAP_HOT_UPDATE:
			heap_xlog_update(record, true);
			break;
		case XLOG_HEAP_CONFIRM:
			heap_xlog_confirm(record);
			break;
		case XLOG_HEAP_LOCK:
			heap_xlog_lock(record);
			break;
		case XLOG_HEAP_INPLACE:
			heap_xlog_inplace(record);
			break;
		default:
			elog(PANIC, "heap_redo: unknown op code %u", info);
	}
}
```

模式很清楚:**用 `info`(记录的 `xl_info` 低 4 位,回忆第 18 章第三节的 `XLOG_HEAP_OPMASK`)区分操作类型,调对应的 `heap_xlog_*` 函数**。INSERT 调 `heap_xlog_insert`,DELETE 调 `heap_xlog_delete`,等等。每个 `heap_xlog_*` 函数会:① 调 `XLogReadBufferForRedo` 读页并判断要不要 redo(`BLK_NEEDS_REDO` 才做);② 按 record 里的 payload 把 tuple 插入/删除/更新到页里;③ 标脏页。

> 注意那个注释:"These operations don't overwrite MVCC data"——heap rmgr 的操作都是**追加新 tuple**(MVCC 的多版本),不覆盖老 tuple。而 `heap2` rmgr 里的操作(如 prune/freeze、visible)会原地改页,需要额外的冲突处理。这个区分正是 MVCC 设计在 redo 层面的体现:heap 的 INSERT/UPDATE/DELETE redo 是"追加",幂等性好处理(重放一个 INSERT,要么 LSN 已超跳过,要么真插入一遍)。

### `xact_redo`:commit/abort/prepare 怎么重放

最后看 `xact_redo`——重放事务提交记录,它是 redo 里**最关键的一类**(决定哪些事务算提交):

> [src/backend/access/transam/xact.c:6301-6318](../postgresql-17.0/src/backend/access/transam/xact.c#L6301-L6318)

```c
xact_redo(XLogReaderState *record)
{
	uint8		info = XLogRecGetInfo(record) & XLOG_XACT_OPMASK;

	/* Backup blocks are not used in xact records */
	Assert(!XLogRecHasAnyBlockRefs(record));

	if (info == XLOG_XACT_COMMIT)
	{
		xl_xact_commit *xlrec = (xl_xact_commit *) XLogRecGetData(record);
		xl_xact_parsed_commit parsed;

		ParseCommitRecord(XLogRecGetInfo(record), xlrec, &parsed);
		xact_redo_commit(&parsed, XLogRecGetXid(record),
						 record->EndRecPtr, XLogRecGetOrigin(record));
	}
	...
	else if (info == XLOG_XACT_ABORT)
	{
		xl_xact_abort *xlrec = (xl_xact_abort *) XLogRecGetData(record);
		xl_xact_parsed_abort parsed;

		ParseAbortRecord(XLogRecGetInfo(record), xlrec, &parsed);
		xact_redo_abort(&parsed, XLogRecGetXid(record),
						record->EndRecPtr, XLogRecGetOrigin(record));
	}
```

注释 "Backup blocks are not used in xact records"——commit/abort record **不带数据页引用**(它改的不是数据页,而是 `pg_xact` 这个提交状态位图)。所以 redo 它不需要走 `XLogReadBufferForRedo` 那套 LSN 比较,直接处理。

`XLOG_XACT_COMMIT` 分支调 `xact_redo_commit`——它把事务在 `pg_xact` 里标成 committed(调用 `TransactionIdCommitTree`)。`XLOG_XACT_ABORT` 调 `xact_redo_abort`——把事务标成 aborted(`TransactionIdAbortTree`)。

> [src/backend/access/transam/xact.c:6095-6102](../postgresql-17.0/src/backend/access/transam/xact.c#L6095-L6102)

```c
	if (standbyState == STANDBY_DISABLED)
	{
		/*
		 * Mark the transaction committed in pg_xact.
		 */
		TransactionIdCommitTree(xid, parsed->nsubxacts, parsed->subxacts);
	}
```

这就是第二节"为什么只 redo 不 undo"在源码层面的兑现:**遇到 commit record,把事务标 committed;遇到 abort record,标 aborted;两者都没有的(崩溃时还挂着的事务),恢复结束时统一当 abort**。事务的状态决定它产生的 tuple 可不可见(MVCC)——tuple 本身的物理改动可能已经被 redo 重放出来了,但只要事务是 aborted,这些 tuple 就成了"看不见的幽灵",等 VACUUM 清理。

> 把 `PerformWalRecovery` + `ApplyWalRecord` + `heap_redo` + `xact_redo` 串起来,崩溃恢复的全貌就清楚了:**主循环读 record → 按 `xl_rmid` 分发到 `rm_redo` → 每个 rm_redo 用 `XLogReadBufferForRedo` 读页、比 LSN、FPI 兜底破损页 → 把改动重施到页**。这套循环不光用于崩溃恢复,也是备机 apply(第 22 章)的同一套代码——这是本章和第 22 章最重要的连接点。

## 章末小结

- **崩溃恢复要解决什么**:内存里的脏页(没落盘的改动)崩溃时丢了,但 WAL 是已 fsync 的、持久化的。靠重放 WAL,把丢失的改动重新施加到磁盘的数据页上。
- **本质**:从最近 checkpoint 记录里的 redo 起点开始,逐条重放 WAL 到末尾。这就是 redo。redo 点之前的数据页已经在磁盘上(checkpoint 保证),redo 点之后的靠重放补回。
- **只要 redo、不要 undo——这是 PG 的灵魂设计**:MVCC 让"未提交事务的改动对别人本来就不可能可见"。abort 一个事务,只要在 `pg_xact` 标 aborted,它的 tuple 因为 `t_xmin` 对应 aborted 自动被读时跳过——**不需要物理擦除**。所以 PG 的崩溃恢复遇到 commit record 标 committed、遇到 abort record 标 aborted、两者都没有的统一当 abort,事务的状态决定 tuple 可见性,redo 永远"往前做"不"往后退"。对比 InnoDB/Oracle 的 undo-based 设计:它们原地覆盖改行,必须有 undo 日志回滚;PG 选择 MVCC 多版本,代价是旧版本堆积要 VACUUM(下一章)。
- **完整恢复流程**:读 ControlFile(`StartupXLOG`)→ 找最近 checkpoint 记录(`InitWalRecovery` + `ReadCheckpointRecord`)→ 判断要不要恢复(`checkPoint.redo < CheckPointLoc` 等三个判据)→ 取 redo 点 → 初始化全局计数器 → redo 主循环(`PerformWalRecovery`)→ 收尾恢复 2PC prepared 事务(`RecoverPreparedTransactions`)→ `InRecovery = false` 开门营业。
- **redo 幂等性**:靠 LSN 比较——`page->pd_lsn >= record->lsn` 就跳过这条(`BLK_DONE`),只有 `BLK_NEEDS_REDO` 才重放。这让 redo "可以重放已经部分应用过的 record 而不出错"——崩溃恢复中途又崩、二次恢复也能正确。
- **FPI 是幂等性能成立的隐含前提**:LSN 比较在破损页(torn page)上失效,FPI 救场——带 FPI 的 record 用整页镜像强制覆盖数据页(`RestoreBlockImage`),消除 partial write 这个不确定性。`XLogReadBufferForRedo` 里"哪怕页看起来更新也强制 restore FPI"的设计,体现了"宁可多 redo 也不要基于破损页"的可靠性优先。
- **2PC prepared 事务的恢复**:PREPARE record 在 redo 时重建事务为 prepared 状态(`PrepareRedoAdd`),恢复结束时 `RecoverPreparedTransactions` 重建它们的锁和子事务状态——让 prepared 事务**跨崩溃存活**,这是 2PC 原子提交的根基。
- **恢复完成的判据**:崩溃恢复重放到 WAL 末尾就算完成;归档恢复/备机恢复要重放到 `minRecoveryPoint`(`CheckRecoveryConsistency` 里判断),这个判据就是"恢复到一致点"。崩溃恢复是这个机制的特例(一致点 = WAL 末尾)。
- **崩溃恢复 = 备机 apply(第 22 章预告)**:`PerformWalRecovery` 这套 redo 主循环、`CheckRecoveryConsistency` 这套一致性判断,被备机 apply **原样复用**——唯一区别是 WAL 来源(本地文件 vs 网络)。这就是第 22 章那句"备机 apply 就是崩溃恢复"的全部含义。

### 回扣主线

这一章是"**不丢不乱**"那一侧的收官——它把 P5 持久篇的三章(WAL → Checkpoint → 崩溃恢复)收成一个闭环:**先记 WAL(第 18 章)→ checkpoint 划锚点(第 19 章)→ 崩溃后从锚点 redo 重建(第 20 章)**。这一闭环治的是"**易失**"这个本性:内存的脏页会丢,但只要 WAL 在、redo 起点在,丢的就能变回来。

PG 选择"只 redo 不 undo"的设计,本质上是 MVCC(第 17 章)在持久性侧的延伸——MVCC 让 abort 不需要物理擦除,所以崩溃恢复只需要"往前做"。这个选择的代价是旧版本堆积,需要 VACUUM(下一章)清场。所以 P5(持久)和 P4(并发)通过 MVCC + redo-only 这条线紧密咬合:并发控制用 MVCC,持久性用 WAL+redo,而 MVCC 的多版本让 redo-only 成为可能——两个子系统是配套设计的,不是各自为政。

至于数据的三个本性:本章直接服务"**易失**"(用 WAL 的持久换内存的易失);同时 redo 的幂等性设计、FPI 的"宁可多 redo"取舍,又体现了对"**可靠性**"——这是"不丢不乱"那一侧的极致追求,宁可慢一点(多 redo、记 FPI 多耗 WAL),也不能错。
