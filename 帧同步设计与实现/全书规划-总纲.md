# 《帧同步设计与实现深入浅出:怎么让千万台机器算出同一个结果》—— 全书总纲

> 一本写给"会写代码、懂数学(微积分、线性代数、概率)、懂系统(网络/内核/数据库读过或写过),**但没做过帧同步联机游戏**"的工程师的书。
>
> 本文是全书的**总纲**:定主线、定方法、定比喻、定分篇分章、定源码策略、定写作约定。本书与系列里那些"读开源大项目源码"的书最大的不同——**源码项目(LockstepSdk)是作者自己设计的**,所以本书带一个独特视角:**设计复盘**(作者亲述"我为什么这么定")+ **客观精解**(像读开源那样把 how 讲透)。本书另两条命脉:① **确定性 bug 怎么定位**——帧同步最难的不是写出来,是"为什么不同步了,怎么找";② **怎么把帧同步做成独立 SDK**——市面罕见的定位(多数帧同步是引擎内嵌模块或教程 demo)。

## 一、这本书的定位

### 目标读者

会写程序、懂数学、懂系统,但**帧同步联机游戏是零经验**。听说过"帧同步""lockstep""定点数""预测回滚""延迟补偿"这些词,但讲不清:

- 联机游戏里,两台配置不同的手机,为什么能在同一时刻看到同一个游戏局面?是服务器把局面算好发下来的吗(那是状态同步)?——**帧同步不是**。帧同步只广播**玩家输入**,局面由每台机器**各自算**。那凭什么两台机器算出的局面一样?
- 既然各算各的,为什么不能直接用 `float`?`float` 在两台机器上算出来不一样会怎样?——会**不同步(desync)**,而且这种不同步是**逐位累积**的,一帧差一个最低位,几千帧后局面天差地别。所以帧同步**必须用定点数**,而且**所有可能影响局面的运算**(随机数、容器遍历顺序、时间)都要确定性。
- 玩家按下方向键到屏幕上坦克动起来,中间要过网络(几十毫秒往返),怎么做到"不卡"?——**预测回滚**:客户端不等服务器,本地先按猜测的输入算下去(预测),等服务器权威输入回来了,如果猜对了就直接确认,猜错了就**回滚**到出错那一帧,用正确输入重演。这套机制是帧同步的招牌。
- 网络会丢包,丢了一帧输入怎么办?所有客户端卡住等吗?——不等。靠**冗余历史帧**补,丢一包靠下一包捎带的前几帧补上,零往返恢复。
- 断线重连呢?退出再进来怎么追上这几十秒的战局?——服务器存**快照 + 增量帧**,重连时发一份快照,再把快照之后的帧补给你,你本地飞速重演追上。
- 逻辑帧只有 20fps,屏幕却是 60fps,画面为什么不顿?——**逻辑/表现分离**:渲染层读逻辑状态做插值;回滚时位置会瞬跳,用**视觉偏移补偿**(Visual Offset)吸收跳变,平时却保持零延迟手感。
- 最要命的:**线上跑着跑着两台客户端不同步了,怎么找原因?**两份都在跑、都"看起来正常",但局面慢慢分叉。这种 bug 怎么定位到是哪一行代码、哪一次运算的最低位不一样?——这是帧同步工程化的真正难点,也是本书的独门主题。

读者不是不聪明,是**没做过这个领域**。本书不假设读者懂任何帧同步术语,但**假设读者懂编程、懂数值表示(整数/浮点)、懂网络基础(UDP/TCP、RTT)**。这样讲到定点数的精度范围、预测回滚的时间线、网络时钟的平滑算法时,读者接得住。

### 全书一句话主旨

> **帧同步的本质是一句契约:相同初始状态 + 相同输入序列 + 确定性运算 = 相同结果。全书围绕"确定性"展开——先讲怎么造一台单机就确定的机器(定点数/确定性随机/有序 ECS/字节级序列化),再讲怎么把它接到网络上(预测回滚/网络时钟/丢帧补偿/断线重连),最后讲怎么把它做成独立 SDK、以及不同步了怎么定位。**

读完一本书,读者能在脑子里放映出帧同步的完整发生过程:采集输入 → 本地预测执行 → 输入发往服务器 → 服务器聚合并按固定节拍广播权威帧 → 客户端收到后比对,猜对了确认/猜错了回滚重演 → 渲染层读逻辑状态插值平滑 → 全程每帧用哈希对账抓 desync——以及每一步**为什么必须存在、原理是什么、源码怎么实现、出问题了怎么找**。

### 为什么用"确定性内核 vs 同步机制"当骨架(而非按模块平铺)

很多帧同步文章的写法是"按概念罗列":定点数、ECS、回滚、网络……本书**不这么写**。因为那样讲,读者记的是"有哪些组件",而不是"帧同步在解决什么层层递进的问题"。本书用帧同步**真正的两半**当骨架:

- **确定性内核**(上篇)——回答"怎么造一台单机就确定的机器"。即使不联网,一台机器上,相同输入跑两遍也必须位级相同。这是地基。定点数、确定性随机、有序遍历 ECS、字节级序列化,都是为它服务。
- **同步机制**(下篇)——回答"怎么把这台确定的机器接到不可靠的网络上"。网络有延迟、抖动、丢包,怎么在这些干扰下还让所有客户端最终一致。预测回滚、网络时钟、冗余帧、断线重连,都是为它服务。

中间一座桥:**预测回滚**——它既依赖确定性内核(回滚后重演必须确定),又是同步机制的核心手段。**表现平滑**紧随其后——逻辑侧能倒带了,表现侧怎么画得平滑。读者永远知道**我在哪、为什么需要下一步**。

### ★本书的四个独特优势(别的系列没有的)

1. **作者视角 / 设计复盘**:LockstepSdk 是作者本人设计的。每个关键决策(Q48.16 而非 Q32.16、双引擎 ECS、KCP 做成 stub、双 TFM 双版本号、物理时钟节拍器),作者能亲述"我当时为什么这么定、踩过什么坑、考虑过什么替代方案"。这是读开源项目永远拿不到的深度。写作时用第一人称"我为什么这么定"穿插,但**不喧宾夺主**,主线仍是客观原理。
2. **bug 定位是真主题**:帧同步工程化的真正难点不是"写出来",是"线上 desync 了怎么找"。本书专设第六篇"确定性调试",并把项目里 `issues_found.md` 与工业级审计报告里的血泪 bug fix(P0-1 跨 TFM 舍入分叉、双轨哈希漂移即覆盖、进程级重连根因链、BufferPool 双倍归还……)做成案例,讲清"从现象到根因"的定位方法,还专设**"假问题教学集"**教读者鉴别真假 bug。
3. **双 TFM 跨平台确定性活教材**:LockstepSdk 同时编译 `net8.0` 和 `netstandard2.1`(为了 Unity 等旧运行时兼容)。同一个定点数乘法,在两个运行时上**会因为舍入语义不同而差一个最低位**,导致跨平台 desync。这种"确定性的确定性"——连"算同一个数学题"本身都要保证跨平台一致——是别处讲不到的硬核点(第 2 章 P1-03 招牌)。
4. **SDK 化是独特定位**:市面帧同步多是"引擎内嵌模块"(某 Unity 插件里的同步器)或"教程 demo",**独立、SDK 级、多引擎可集成、带 Builder API 和可注入依赖的帧同步框架几乎没有**。竞品(Photon Fusion / Mirror / Fish-Net)清一色状态同步+预测,无确定性保证。本书专设一章讲"怎么把帧同步做成 SDK",这本身是稀缺内容。

## 二、写作方法:动机(why)/ 技巧(how)双线 + 作者复盘 + bug 贯穿 + 直球比喻点睛 ⚠️ 本书命脉

### 命脉一:"为什么"永远先于"怎么做"

帧同步的每个机制都有强烈的"为什么必须如此"。每个关键点必须先回答:**它解决什么本质问题?最朴素的笨办法撞什么墙?** 然后才讲实现。

> **一句话**:任何一章,读者读完还不懂"这一块凭什么存在、不这样会怎样",这章就是失败的——哪怕源码贴得再多。

### 命脉二:动机(why) + 技巧(how)双线

帧同步的源码充斥精妙技巧:**定点数乘法的三路径 + 双路径等价性测试**、**跨 TFM 的有符号算术移位**、**回滚安全组件池的保序删除**、**Jacobson 算法平滑 RTT + 硬边界防回滚风暴**、**冗余历史帧抗 UDP 丢包**、**单调钟替换墙钟防 NTP 回跳**、**Visual Offset 吸收回滚瞬跳**……每个关键点要讲两件事:① 设计动机(它解决什么本质问题?朴素做法撞什么墙?);② 实现技巧(用了什么数学/数据结构把问题漂亮解决?为什么妙?)。

> **每章单开"技巧精解"小节**(正文之后、小结之前),挑本章最硬核 1~2 个技巧,配源码/数学推导 + 行号,单独拆透,且必须配**反面对比**("最朴素的笨办法撞什么墙")。

### 命脉三:作者复盘视角(本书独有)

LockstepSdk 是作者设计的,关键决策处插入"**作者复盘**"小框:讲当时为什么这么选、考虑过什么替代方案、后来有没有后悔、踩过什么坑。最有戏的素材来自 `RECONSTRUCTION_LOG.md`:早期服务器由玩家输入驱动,结果坦克无规则瞬移、乱开炮("发疯的坦克"),由此悟出"物理时钟节拍器 + 强制空输入填充"这个定型架构——这种"从病态里悟出正道"的复盘,是读开源拿不到的。

> **作者复盘 · 物理时钟节拍器**:早期服务器"收到输入就广播",客户端一断开服务器就停顿、一密集输入就广播失控,坦克满屏乱瞬移。根因是"服务器节奏被玩家输入绑架"。改成服务器按固定 20Hz 物理节拍推进、无论玩家在不在都强制填空输入广播——所有客户端收到的包节奏绝对恒定,"发疯的坦克"立刻治好。这个认知(服务器该不该被输入驱动)是帧同步服务端的分水岭。

这种"亲述"是本书深度天花板高于读开源的关键。但**节制使用**——只在真正有"为什么这么定"的非显然决策处用。

### 命脉四:bug 定位贯穿(本书独有)

确定性 bug 是帧同步的幽灵。本书不是把 bug 藏在最后,而是**贯穿全书**:每个机制讲完,顺带讲"这块出过什么 bug、表现是什么、怎么定位的"。第六篇再系统总结定位方法论,其中专设**"假问题教学集"**——`issues_found.md` 四轮审查里被排除的 8 个"假 bug"(如"LFloat(long) 构造溢出"其实全代码库从未调用、"LFloat/int 除零"其实 C# `long/0` 必抛跨平台一致),教读者**鉴别真假 bug 的能力**——这本身就是工程成熟度的标志。

### ⚠️ 深度标杆(用户强调,本书命脉)

篇幅对标系列里《LuaJIT》《SQLite》《RocksDB》级——**每章约 12000~18000 字,招牌章可更厚**。用户原话:"深入肯定要对得起深入浅出系列""定点数必须讲哈"。宁可展开多讲几层 why、多铺数学推导(定点数精度范围、移位舍入、Jacobson 算法系数)、多给循序渐进的小例子、多对比反例(浮点怎么 desync、朴素回滚怎么爆炸、"发疯的坦克"怎么乱),也不一笔带过。

### ★配图用 matplotlib 生成 PNG(不用 ASCII)

帧同步的图大多是**时间线 / 数值 / 结构图**:预测回滚的时间线(预测帧 → 服务器帧 → 回滚重演)、定点数的精度范围示意、环形缓冲的环绕、网络时钟的 EWMA 平滑、冗余帧抗丢包、逻辑帧/渲染帧的节拍分离、Visual Offset 吸收瞬跳。matplotlib 画得清晰;图内**英文标注**(避中文字体方块),正文中文解释。流程图用 mermaid。**每章至少 2~3 张图**。

## 三、全书主线、二分法、主比喻、承接

### 主线

> **帧同步 = 相同初始状态 + 相同输入序列 + 确定性运算 = 相同结果。全书围绕"确定性"展开:上篇造一台单机就确定的机器,下篇把它接到不可靠的网络上。**

任何一处看不懂某个机制,回到这句问:"这是在保证'确定性'(单机位级一致:定点数/随机/遍历/序列化),还是在解决'同步'(多机最终一致:预测回滚/时钟/丢帧/重连)?它和最朴素的笨办法比,妙在哪?"

### 二分法

本书的二分法,对应帧同步两大主题:

> **确定性内核(单机就能同步) vs 同步机制(多机才算同步)。中间一座桥:预测回滚 + 表现平滑。**

- **确定性内核**(第 1-2 篇):即使不联网,一台机器上相同输入跑两遍也必须位级相同。包括:**定点数 LFloat**(为什么不能用浮点)、**确定性数学库**(乘法三路径、三角函数查表、确定性 Sqrt)、**确定性随机 LRandom**(Xorshift128+)、**有序 ECS**(为什么 Dictionary 不能遍历、组件池删除为什么不能 swap-and-pop、防呆体系)、**字节级序列化**(强制排序、小端序、FNV-1a 哈希)。
- **预测回滚 + 表现平滑**(第 3 篇):单机的"时间机器" + 它在屏幕上的平滑呈现。**预测**(不等服务器先猜着算)+ **回滚**(猜错了倒带重演,以及**回滚反向施加给整个代码库的编程纪律**)+ **LockstepController 实现** + **表现平滑**(逻辑/渲染帧分离、插值、回滚视觉补偿)。
- **同步机制**(第 4 篇):把确定的机器接到网络上。包括:**LockstepDriver**(SDK 主循环、跨线程命令队列)、**网络时钟**(Jacobson 算法、硬边界防回滚风暴)、**Relay vs Authoritative 双服务器模式**(核心二分)、**GameRoom 与多房间**(顶号重连、哈希防作弊、中毒快照熔断)、**消息处理/协议/反作弊**(策略模式、双轨版本号、拦截器)、**传输层**(TCP/UDP/WebSocket/KCP 为什么是 stub)。
- **横切全书**:SDK 化与工程化(第 5 篇)、确定性调试(第 6 篇)。

任何时候迷路,回到这个二分法:**"这是在保证单机的确定性,还是在解决多机的同步?"**

### 主比喻决策:乐谱 + 指挥 + 时间机器 + 对账,直球为主、比喻点睛

> **直球为主**:绝大多数地方第一性原理 + 数学推导直给。**比喻点睛**:仅在序章(P0-01)用一组比喻做一次性点睛——
> - **确定性像乐谱**:所有乐手(客户端)拿同一份乐谱(初始状态 + 输入序列),按相同节拍演奏,出来的曲子必然一样。任何一个乐手把 `float` 当成"差不多就行",就像偷偷改了一个音符,几百小节后整首曲子就乱了。
> - **服务器像指挥**:不替乐手演奏(不算局面),只负责打节拍(固定 20Hz 广播权威帧),确保所有乐手在同一拍——而且不管某个乐手有没有跟上,节拍器自顾自走(强制空输入填充)。
> - **预测回滚像时间机器**:乐手不等指挥拍子(网络延迟),先按自己猜的节奏往下吹(预测);指挥的拍子到了发现猜错了,倒回到那一拍重吹(回滚)。
> - **哈希校验像对账**:每个乐手每拍结束算一个"这一拍的总音量指纹"(状态哈希),互相核对,指纹对不上就是有人吹错了(desync)。

**全书其他章节不得沿用这套比喻做主线**。新概念用第一性原理 + 数学 + 真实源码讲。

### 承接(本书特色,务必遵守)

- **★承《物理引擎》**(强承接):帧同步游戏里的碰撞检测(Lockstep.Collision 的 QuadTree + Circle/AABB/OBB),原理和《物理引擎》那本讲的完全一样。本书第 7 篇碰撞章节**一句带过指路** [[physics-engine-source-facts]],篇幅留给"确定性物理"特有的(定点数碰撞、为什么物理也必须确定性)。
- **★承数学线(《数学分析》《线性代数》《概率论》)**:定点数是《数学分析》"精确 vs 逼近"的极致——浮点是逼近(且跨平台不一致),定点数是精确(且跨平台位级一致);三角函数查表是数值逼近;LRandom 的 Xorshift128+ 是《概率论》伪随机序列;Jacobson 算法是 EWMA 平滑。讲到时**指路数学线**,篇幅留帧同步特有的。
- **承《Lua》《LuaJIT》**(神似对照):帧同步的本质就是一台"确定性虚拟机"——相同输入相同输出,这和 Lua VM 的确定性执行一脉相承。可做点睛对照,弱承接。
- **承网络相关系列**(《Tokio》《gRPC》等):TCP/UDP/WebSocket 这些传输层概念是通用的,本书讲到传输层时可一句带过指路,篇幅留帧同步特有的(冗余帧抗丢包、为什么 KCP 做成 stub)。

### 反哺 / 横连

- 本书讲透确定性后,**反哺**读者对"跨平台数值一致性"的理解——同一思想用在区块链共识、分布式 SIM 仿真、CI 复现等场景。
- 帧同步 vs 状态同步的对照(序章 + FAQ 级话题),帮读者建立"联机游戏同步方案"的全景判断。

### 第 1 篇 · 确定性内核·上:定点数与确定性数学(招牌,2 章)

- **第 2 章 · 定点数 LFloat:为什么不能用浮点**(`P1-02`)——IEEE 浮点为什么跨平台不一致(FPU 舍入、扩展精度、FMA、溢出截断),定点数怎么用整数模拟小数。Q48.16 用 long 的精度范围权衡(±1.4×10¹⁴ 范围 / 1/65536 精度)。RawValue 的位结构,加减直接、乘除要移位。`LFloat.One` 类型陷阱(bug 案例:早期 const int → 现 const long,但 One≠LFloat 语义陷阱仍在)。FromFloat 边界溢出静默截断(bug 案例 P0)。归属:`确定性内核`。

### 第 2 篇 · 确定性内核·下:随机、ECS、序列化(4 章)

> 确定性不只指数,还有随机数、容器遍历、序列化——任何一个不确定都会 desync。本篇还讲框架的**防呆体系**(把确定性纪律做成编译/加载期硬约束)。

- **第 4 章 · 确定性随机 LRandom:Xorshift128+ 与状态序列化**(`P2-04`)——为什么 `System.Random` 不能用(跨平台/跨版本算法不一),LRandom 用 Xorshift128+(Vigna 算法,长周期高质量),splitmix64 播种,状态就是两个 ulong(回滚零成本)。随机数状态的序列化与回滚(常见坑:忘存随机状态)。RestoreState 零状态检查的对称性 bug。归属:`确定性内核`。
- **第 5 章 · 有序 ECS 与 World:逻辑数据分离、防呆体检**(`P2-05`)——ECS 的逻辑/数据分离,System 执行顺序怎么保证确定(**稳定插入排序而非 List.Sort**,bug 案例:非稳定排序破坏确定性),实体代数(Generation)机制,World 的 SaveState 字节级格式(VersionMagic/SerializationVersion=2/组件池按类型 Ordinal 排序)。★**防呆体系**:SystemStateValidator 反射体检(DEBUG 拦截 System 里的 Dictionary/HashSet/Task/Random/静态字段)、`[AllowUnsafeField]`(Rust unsafe 式免责声明)、`[AutoSerialize]` Source Generator + DEBUG 往返校验。Smart Query 最小集遍历(优先遍历规模最小的池)。归属:`确定性内核`。
- **第 6 章 · 组件池与回滚安全:双引擎的取舍**(`P2-06`)——★招牌章。SafeECS(ComponentPool 保序标记删除,回滚友好,BinarySearch 维护有序)vs UnsafeECS(UnsafeComponentPool Sparse Set + swap-and-pop + **裸内存 fixed 指针拷贝序列化**,快照快 200 倍)。为什么组件池删除不能 swap-and-pop(破坏回滚后遍历顺序)。★**序列化脏标记缓存**(IsDirty + 高频组件 `[HighFrequencyComponent]` 跳缓存,把快照成本和组件变化频率解耦)。诚实标注:UnsafeECS 未接入 World.SaveState(高级逃生舱)。归属:`确定性内核(招牌)`。
- **第 7 章 · 字节级序列化:BitWriter、强制排序、FNV-1a**(`P2-07`)——序列化为什么必须确定性(相同状态产生相同字节流,否则哈希永远误报 desync),小端序跨平台一致,WriteDictionarySorted 强制 Key 排序,溢出安全防 OOM DoS(BitReader count 校验,bug 案例:FrameData PlayerCount 无边界检查)。归属:`确定性内核`。

### 第 3 篇 · 预测回滚与表现平滑:时间机器与平滑呈现(4 章)

> 把确定性机器变成"可以倒带"的机器,再把它平滑地画出来。本篇还不涉及网络,是纯客户端侧。

- **第 8 章 · 预测:本地不等服务器先算下去**(`P3-08`)——为什么要预测(等服务器确认再算会卡顿,操作"粘手"),预测窗口与预测深度(基于 RTT 提前算几帧),预测的前提(本地输入实时注入 + 非本地输入靠猜)。输入预测器 IInputPredictor 三实现(LastInput 保持上一帧 / Neutral 空输入 / Trend 趋势外推),可插拔(不同游戏类型最优预测不同)。本地与非本地玩家的输入掩码隔离(防混入外部状态)。预测错了的代价预告(引出下一章回滚)。归属:`预测回滚·预测侧`。
- **第 9 章 · 回滚:猜错了怎么倒带重演**(`P3-09`)——★招牌章。**回滚到底怎么发生**:检测预测错误(本地预测输入 ≠ 服务器权威输入)→ 找最近快照点 → 恢复状态 → 用正确输入重演到当前帧 → 继续预测。为什么用"快照 + 重演"而不是"每帧存完整状态"(空间代价对比)。回滚的代价(一次回滚 = 重演 N 帧的 CPU),回滚风暴(连续频繁回滚),快照间隔 SnapshotInterval 的权衡(每帧存快照 = 回滚快但内存大 / 隔几帧存 = 省内存但回滚要重演更多帧)。**★回滚反向施加给整个代码库的编程纪律(本章精髓)**:回滚不是"加个功能",而是一套渗透到每行逻辑代码的编程纪律——① 所有逻辑状态必须可序列化可恢复(组件严禁持有表现层对象引用,用 EntityID 间接引用);② 逻辑层严禁不可逆副作用,音效/粒子/UI 必须用 IsReplaying/IsPredicting 门控(C-6 bug 根源:标志位没复位导致重演时副作用重播);③ 实体创建/销毁不能在遍历中直接做,要命令缓冲;④ 组件池删除不能 swap-and-pop;⑤ 随机数状态必须进快照;⑥ 每帧临时状态必须重置;⑦ 首帧状态全端一致。本质:为了能倒带重演,游戏逻辑必须写成"纯函数式 + 可快照"的风格。归属:`预测回滚·回滚侧(招牌)`。
- **第 10 章 · LockstepController:预测回滚的实现主控**(`P3-10`)——把前两章落地成一个组件。DoUpdate 两阶段:ConfirmServerFrames(确认服务器帧,不一致就触发第 9 章的回滚)+ PredictAhead(本地向前预测,为保证操作即时性不限速)。RollbackTo 流程、追帧动态限速、MaxSimulationMsPerFrame 防卡死。RingBuffer(2 的幂 + 位与取模,无分支)存帧历史,★时效性契约 C-5(纯槽数组越界静默环绕到陈旧槽,靠调用方纪律维系)与负索引映射的设计意图之争。快照池化零分配。归属:`预测回滚·实现`。
- **第 11 章 · 表现平滑:逻辑帧/渲染帧分离与回滚视觉补偿**(`P3-11`)——★用户点名章。逻辑帧固定 20fps、渲染帧可变 30-144fps,渲染层**只读**逻辑状态。为什么直接画逻辑状态会顿(20fps 跳变),用 `Interpolation`(0~1)在上一帧/当前帧状态间线性插值。OnLogicStep 采样 + 双缓冲(_lastStates/_currStates)保证追帧/回滚时插值不断层(为什么不能用 OnLogicStepComplete)。★**视觉偏移补偿 Visual Offset**:回滚时逻辑位置瞬跳,直接 Lerp 会有常态操作延迟感("手感肉");方案是平时逻辑与视觉 1:1 同步,仅在回滚瞬间用 Offset 吸收跳变并指数衰减归零——常态零延迟、回滚不闪现。追帧进度 OnPursueProgress。归属:`表现平滑`。

### 第 4 篇 · 同步机制:把确定的机器接到网络(6 章)

> 网络有延迟、抖动、丢包。在这些干扰下还让所有客户端最终一致。

- **第 12 章 · LockstepDriver:SDK 主循环与跨线程纪律**(`P4-12`)——Driver 把什么封装了(模拟器 + 网络 + 控制器 + 时钟 + 回放 + 渲染插值),Update 主循环真实流程(消费跨线程命令队列 → 帧累加器 → 模式分支 → Controller.DoUpdate → 渲染插值),跨线程命令队列(网络回调先入队、主线程消费,保证 World 线程亲和,DEBUG CheckThreadAffinity 断言),启动停止与资源释放。归属:`同步机制`。
- **第 13 章 · 网络时钟 NetworkClock:Jacobson 算法与硬边界防回滚风暴**(`P4-13`)——★招牌章。SRTT/RTTVAR 真实系数(alpha=0.125, beta=0.25,RFC6298 经典),ClockOffset 三档 EWMA(冷启动 0.5 → 常规 0.1 → 强修正 0.3),PreSendCount 不对称迟滞(变差立即增深/变好缓慢衰减),★硬边界 Clamp 防回滚风暴(smoothedTarget 触界即放弃平滑,hardMin 防本地跑太慢触发无谓回滚,hardMax 防无限制预测爆炸)。时钟偏差从 Pong(主源高精度)/ServerFrame(安全网,>150ms 才介入)双向校准。架构演进:从整数除法算帧号(不稳定)到累加器驱动。归属:`同步机制(招牌)`。
- **第 14 章 · Relay vs Authoritative:两种服务器模式 + 固定节拍**(`P4-14`)——★招牌章(核心二分)。**Relay(中继)**:只收集输入+广播,不跑逻辑,无 desync 校验,无快照重连,反作弊弱;**Authoritative(权威)**:注入 ISimulation 跑完整逻辑,有 desync 校验,有快照重连,反作弊强。两种模式的配置(UseInput vs UseSimulation)、适用场景、权衡。★**物理时钟节拍器**(承序章"发疯的坦克"复盘):服务器按固定 20Hz 推进、输入聚合不等人(迟到用 nullInput 填)、100Hz 调度 + 每 tick 512 消息预算(防单房洪泛)、★单调钟 Stopwatch 替墙钟防 NTP 回跳(P1 血泪:UtcNow 算 tick 会让房间永久冻结且超时检测同时失效)、冗余历史帧抗 UDP 丢包(零往返恢复)、单线程泵保证 World 线程亲和。归属:`同步机制(招牌)`。
- **第 15 章 · GameRoom 与多房间:顶号重连、哈希防作弊、中毒快照熔断**(`P4-15`)——玩家生命周期与房间四态机(Waiting→Playing→Finished→Recyclable),★顶号重连必须自证 ReconnectToken(P0-2 身份劫持防御),HashReport 全员到齐才比对(众数基准问题:基准取 player[0] 可能本身错),★中毒快照熔断(权威 sim 故障后停用 + 清空旧快照,故障降级而非崩溃)。多房间并发 + 房间回收。归属:`同步机制`。
- **第 16 章 · 消息处理、协议与反作弊**(`P4-16`)——9 个 Handler 策略模式(文档说 7 个,真实 9 个),MessageType 17 个枚举值,★双轨版本号(ProtocolVersion 1.1 握手线协议 vs SerializationVersion 2 快照格式,两个独立兼容性边界)。★反作弊:IMessageInterceptor 拦截器(Priority 排序,业务层实现具体校验)、HashReport 状态篡改检测、加速挂天然防御(服务器控帧率)、协议版本强制对齐。为什么 SDK 不内置具体校验逻辑(每个游戏输入语义不同——SDK 边界设计)。归属:`同步机制`。
- **第 17 章 · 传输层抽象:TCP/UDP/WebSocket,以及 KCP 为什么是 stub**(`P4-17`)——INetworkClient/IServerTransport 抽象,TCP 大端长度前缀分帧(含 1MB 上限防 slowloris)vs UDP 无可靠性 vs WebSocket 消息边界,★KCP 是"接口齐全算法空实现"的 stub(接口先行实现后补,默认行为 = UDP + 4 字节 conv 会话过滤),为什么冗余帧让 UDP 已经够用(应用层输入冗余 10% 丢包下连续丢 4 包概率 0.01%)。归属:`同步机制`。
