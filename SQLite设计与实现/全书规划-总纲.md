# 《SQLite 设计与实现深入浅出:为什么把 SQL 编译成字节码再执行》—— 全书总纲

> 一本写给"天天用 SQLite(或在 App/浏览器里不知不觉用它)、却没想过它内部怎么跑"的人的小书。
>
> 本文是全书的**总纲**:定主线、定方法、定比喻、定分篇分章、定源码策略、定写作约定。
> 与《Tokio》《gRPC》《TiKV》《MySQL》《Lua》《LevelDB》同源,都是"**第一性原理 + 真实源码**";本书最大的特色是**一个灵魂、四重承接**:**VDBE 字节码虚拟机(灵魂——SQLite 不直接执行 SQL,而是把 SQL 编译成 opcode、用虚拟机执行)**——这个思想和《Lua》那本的"源码→字节码→虚拟机执行"是同一个根,这是本书最强的承接;再加上 B-tree/WAL(承《MySQL·InnoDB》)、嵌入式单文件(承《LevelDB》)、C/S 对照(承《PG/MySQL》)。

## 一、这本书的定位

### 目标读者

用过 SQLite(或不知不觉在手机/浏览器/App 里用了它)、写过 `sqlite3_exec`、知道它"轻量、单文件、不用装服务",甚至翻过 `sqlite/sqlite` 的 amalgamation(`sqlite3.c` 那个超大单文件),但**一知半解**:知道 SQLite 有 B-tree、WAL、pager,却讲不清一条 `SELECT` 到底怎么变成执行结果;更讲不清 SQLite 最独特的招牌——**VDBE 虚拟机**(它把 SQL 编译成 opcode,用虚拟机执行),以及它和 Lua 字节码虚拟机、InnoDB B+树的根本异同。和"用过 SQLite 但不懂它"是同一种病。

### 全书一句话主旨

> **SQLite 把一条 SQL,先编译成 VDBE 字节码(opcode),再用虚拟机执行;执行时从单文件 B-tree 读数据、经 pager 缓存与 WAL/journal 保证 ACID——"编译成字节码 + 虚拟机执行 + 单文件 B-tree",是它小而美、能塞进每个 App 的根本。每一步的设计动机和实现技巧都讲透。**

每个组件,都是这条旅程上的一个**工位**。读完一本书,读者不是"多记了几个 SQL 用法",而是能**在脑子里放映出一条 `SELECT` 在 SQLite 里的全过程**:SQL 字符串 → Tokenizer 切词 → Parser 建 AST → Code Generator 产出 VDBE opcode 流 → VDBE 虚拟机逐条执行 opcode(OpenRead/Column/Next/ResultRow…)→ 经 B-tree 读页 → pager 缓存页 → WAL/journal 保证原子。以及每一步底下用了什么巧妙的手段。

### 为什么用"一条 SQL 的旅程"当骨架(而非按模块平铺)

SQLite 的模块(编译前端 / VDBE / B-tree / Pager / WAL / VFS)是一条清晰的流水线:一条 SQL 要从编译前端一路流到 B-tree 读数据。按模块百科式平铺,读者会觉得每块都是孤岛、不知道**为什么要学这个**。用"一条 SQL 从字符串到结果"这条旅程当骨架,每个模块都是"为了让 SQL 继续往下走而必须出现的角色"——读者永远知道**我在哪、为什么需要下一站**。

## 二、写作方法:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与数学系列、其他源码系列最关键的方法论,所有写作者必读、必守。**

SQLite 是 C 写的(D. Richard Hipp 主导,几十年演进,Public Domain),源码(尤其 amalgamation `sqlite3.c`)以**紧凑、高密度、技巧密集**著称:VDBE 字节码虚拟机(opcode 循环 + 寄存器)、SQL 编译成 opcode(code generator)、B-tree(表和索引都是 B-tree,非 B+树)、pager 页缓存、WAL 读写并发、rollback journal 原子提交、VFS 可移植抽象、动态类型(type affinity)、多种内存分配可切换(mem0-5)、多种 mutex 可切换……用户用过 SQLite,"一知半解"的缺口往往**不在**"它要解决什么问题"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书的每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?(沿用系列一贯的"为什么优先")
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果不用这个技巧、朴素地写,会撞上什么墙",让妙处显形。

一句话:**讲不清 VDBE 怎么把 opcode 串成执行流、讲不清 B-tree/WAL 怎么保证 ACID,等于没讲 SQLite。**

> **⚠️ 深度标杆(用户强调,本书命脉)**:本书篇幅标杆**以《RocksDB 设计与实现深入浅出》的实际深度为准**——**每章正文约 50000~68000 字符**(含代码、英文术语、标点;比纯中文字数口径更厚,与 RocksDB/MySQL/gRPC/Envoy/TiKV 六本同档)。用户原话:"不要吝啬文字和篇幅,内容可以更丰富一点,要把问题讲解清楚,深入浅出,好理解又有深度。"(同 gRPC/TiKV/Envoy/Dubbo/MySQL 六本)所以每个机制至少讲透三层(动机 → 反例 → 技巧 + 为什么 sound),宁可多铺反例、对比、图、选型历史(为什么 SQLite 用 B-tree 不是 B+树、为什么 VDBE 而不是直接解释 AST、SQLite vs MySQL/PG),把问题讲到读者"原来如此",也不要一笔带过。

## 三、全书主线、二分法、主比喻

### 主线

> **SQLite 把 SQL 编译成 VDBE 字节码(opcode)、用虚拟机执行,执行时从单文件 B-tree 读数据、经 pager + WAL/journal 保证 ACID——"编译成字节码 + 虚拟机执行 + 单文件 B-tree",是它的本质。**

任何一处看不懂某个机制,回到这句问:"这是在**把 SQL 翻译成可执行的字节码(编译与执行)**,还是在**把数据存好、保证不丢(存储与事务)**?"

### 二分法

本书的二分法:

> **编译与执行(上半:SQL 怎么变成字节码并执行) vs 存储与事务(下半:数据怎么存、怎么不丢)。**

- **编译与执行这一面**:Tokenizer(切词)、Parser(建 AST)、Code Generator(产出 opcode)、VDBE(虚拟机执行 opcode)——这些决定"SQL 怎么变成执行结果"。这是 SQLite 最独特的地方(它是个"SQL 编译器 + 字节码虚拟机")。
- **存储与事务这一面**:B-tree(表/索引存 B-tree)、Pager(页缓存)、WAL/journal(原子提交、ACID)、VFS(OS 抽象)——这些决定"数据怎么存、怎么不丢"。

SQLite 每个模块都落在这两面之一(或两者衔接处:VDBE 的 opcode 调 B-tree 接口,是编译执行通往存储的枢纽)。任何时候迷路,回到这个二分法。

### 主比喻决策:直球为主,比喻点睛(⚠️ 沿用系列定调)

沿用系列定调:**直球为主**——绝大多数地方第一性原理直给,不套比喻。**比喻点睛**——仅在开篇(P0-01)用"自带发动机的小车"做一次性点睛(MySQL/PG 是"专门的车队 + 加油站(C/S 服务)",SQLite 是"一辆自带发动机、加满油就能跑的小车,钥匙给你(链接库 + 单文件),开哪是哪"),全书不建立贯穿的重比喻体系。这是本书与《Tokio》(餐厅服务员全程贯穿)最显著的风格差异。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线是一条旅程:**为什么需要 SQLite → SQL 怎么编译成字节码 → VDBE 怎么执行 → B-tree 怎么存 → Pager/WAL 怎么保 ACID → VFS/并发 → 收束**。
> 章号全局连续;命名 `P{篇}-{章}-标题.md`,开篇第 1 章 = `P0-01`,第 1 篇第 2 章 = `P1-02`,以此类推。

### 第 0 篇 · 开篇:为什么需要 SQLite(定调,1 章)

- **第 1 章 · 第一性原理:为什么需要 SQLite**(`P0-01`):嵌入式场景(手机/浏览器/App 内置数据库)为什么需要 SQLite 而不是 MySQL/PG(serverless、零配置、单文件)。SQLite 的本质:**一个把 SQL 编译成字节码的虚拟机(VDBE)+ 一个单文件 B-tree 存储**。★四重承接(↔《Lua》VDBE 字节码虚拟机、↔《MySQL·InnoDB》B-tree/WAL、↔《LevelDB》嵌入式、↔《PG/MySQL》C/S 对照)。**【样本章节,全书风格标杆,见后】**

### 第 1 篇 · 编译:SQL 怎么变成字节码(3 章)—— 承接《Lua》编译器

> 先搞清楚 SQLite 的上半——SQL 怎么从字符串变成可执行的字节码。

- **第 2 章 · SQLite 架构全景:八层流水线**(`P1-02`)——SQLite 怎么切成 Interface / Tokenizer / Parser / Code Generator / VDBE / B-tree / Pager / OS Interface 八层?一条 SQL 怎么从顶层 API 一路流下来?技巧:**八层流水线**(每层只依赖下层接口)、**prepared statement**(SQL 编译一次、执行多次)。源码:`src/main.c`、`src/prepare.c`、`src/sqlite.h.in`。归属:`编译与执行(总览)`。
- **第 3 章 · Tokenizer 与 Parser:SQL → AST**(`P1-03`)——SQL 字符串怎么变成语法树?技巧:**Tokenizer**(`tokenize.c` 切词)、**Parser**(`parse.y` yacc 文法,生成 AST)、为什么用 yacc 生成而不是手写。源码:`src/tokenize.c`、`src/parse.y`。归属:`编译与执行`。
- **第 4 章 · Code Generator:AST → VDBE 字节码**(`P1-04`)——AST 怎么变成 opcode?技巧:**code generator**(`select.c`/`insert.c`/`update.c`/`delete.c`/`build.c` 各语句产出 opcode)、为什么"编译成字节码"而不是直接解释 AST(执行时可重复、可优化、虚拟机统一)。承接《Lua》编译器(源码→字节码)。源码:`src/select.c`、`src/where.c`、`src/vdbeaux.c`。归属:`编译与执行(招牌·承《Lua》)`。

### 第 2 篇 · 执行:VDBE 虚拟机(3 章)—— SQLite 灵魂,承接《Lua》虚拟机

> 这是 SQLite 最独特的部分:**opcode 在虚拟机里怎么执行**。

- **第 5 章 · VDBE 虚拟机:执行字节码**(`P2-05`)——opcode 怎么被执行?技巧:**opcode 循环**(`vdbe.c` 一个大 switch-case 循环逐条执行 opcode)、**寄存器/栈**、**游标(cursor)**。承接《Lua》虚拟机(字节码循环执行,只是 Lua 是寄存器式、SQLite 的 VDBE 也大量用寄存器)。源码:`src/vdbe.c`(SQLite 最核心文件)。归属:`编译与执行(招牌·承《Lua》)`。
- **第 6 章 · opcode 详解:OpenRead / Column / Next / ResultRow**(`P2-06`)——都有哪些 opcode、各干什么?技巧:**核心 opcode 拆解**(`OpenRead` 打开表读、`Column` 取列、`Next` 下一条、`ResultRow` 返回一行、`Rewind`/`Seek` 定位),一条 SELECT 的 opcode 长什么样。源码:`src/vdbe.c`(各 opcode 的 case)、`src/vdbe.h`。归属:`编译与执行`。
- **第 7 章 · 一条 SELECT 怎么执行**(`P2-07`)——从 SQL 到结果的全过程串起来。技巧:**SELECT 的编译 + 执行全流程**(WHERE 怎么变 opcode、JOIN 怎么变嵌套循环 opcode、索引怎么选)、`EXPLAIN` 看 opcode。源码:`src/select.c`、`src/where.c`、`src/vdbe.c`。归属:`编译与执行(招牌)`。

### 第 3 篇 · 存储:B-tree(3 章)—— 承接《MySQL·InnoDB》《LevelDB》

> VDBE 执行时怎么读写数据?数据存在单文件的 B-tree 里。

- **第 8 章 · B-tree 存储:表和索引都是 B-tree**(`P3-08`)——SQLite 怎么存一张表?技巧:**表和索引都是 B-tree**(注意 SQLite 用 **B-tree 不是 B+树**——叶子页和内部页都存数据)、对照《MySQL·InnoDB》B+树、对照《LevelDB》LSM、**单文件多 B-tree**(一个 .db 文件里有多棵 B-tree,用根页号区分)。源码:`src/btree.c`、`src/btreeInt.h`。归属:`存储与事务(招牌)`。
- **第 9 章 · 记录格式与动态类型(type affinity)**(`P3-09`)——一行数据在 B-tree 页里长什么样?SQLite 为什么"类型宽松"?技巧:**Record 格式**(变长记录)、**动态类型**(一个值可存 INTEGER/REAL/TEXT/BLOB/NULL,SQLite 靠 type affinity 而非强类型)、对照 MySQL 强类型。源码:`src/btree.c`、`src/util.c`、`src/sqliteInt.h`。归属:`存储与事务`。
- **第 10 章 · 索引与查询:怎么用 B-tree 加速**(`P3-10`)——索引怎么建、怎么用?技巧:**索引 B-tree**、索引选择、`WHERE` 怎么走索引。源码:`src/where.c`、`src/btree.c`。归属:`存储与事务`。

### 第 4 篇 · 页管理与事务:Pager(4 章)—— 承接《MySQL·InnoDB》WAL

> B-tree 页在磁盘上,pager 负责把它们缓存进内存、保证写不丢。

- **第 11 章 · Pager:页缓存**(`P4-11`)——页怎么缓存、怎么读写?技巧:**pager**(页管理,`pager.c`)、**pcache 页缓存**(`pcache.c`/`pcache1.c`)、脏页、对照《MySQL》buffer pool、《Linux内存管理》page cache。源码:`src/pager.c`、`src/pcache.c`、`src/pcache1.c`。归属:`存储与事务(承《MySQL》《Linux mm》)`。
- **第 12 章 · rollback journal:默认原子提交**(`P4-12`)——SQLite 默认怎么保证原子?技巧:**rollback journal**(改页之前,先把页的**原内容**写进 journal;crash 后用 journal 把页改回去)、为什么默认用它(简单、单写)。源码:`src/pager.c`、`src/memjournal.c`。归属:`存储与事务(招牌)`。
- **第 13 章 · WAL 模式:读写并发**(`P4-13`)——WAL 怎么让读写并发?技巧:**WAL**(改先写 WAL 不动数据文件、读读数据文件+WAL)、**读不阻塞写、写不阻塞读**(对照 rollback journal 的"写时独占")、checkpoint(把 WAL 合并回数据文件)。源码:`src/wal.c`。归属:`存储与事务(招牌)`。
- **第 14 章 · crash recovery 与 ACID**(`P4-14`)——crash 了怎么恢复?技巧:**ACID 保证**(rollback journal 模式 / WAL 模式各自的 recovery)、对照《MySQL》redo/undo。源码:`src/pager.c`、`src/wal.c`。归属:`存储与事务`。

### 第 5 篇 · VFS 与并发(3 章)

- **第 15 章 · VFS:OS 抽象**(`P5-15`)——SQLite 怎么跨平台?技巧:**VFS(Virtual File System)**(可移植 OS 抽象,`os_unix.c`/`os_win.c`/`os_kv.c`)、**自己定义文件接口**(不直接调 POSIX,套一层 VFS)、为什么这样设计(可移植 + 可替换后端,如 os_kv 存到 KV 存储)。源码:`src/os.c`、`src/os_unix.c`、`src/os_win.c`、`src/os_kv.c`。归属:`存储与事务(承《Linux内核》VFS)`。
- **第 16 章 · 动态类型与 type affinity**(`P5-16`)——SQLite 类型为什么"宽松"?技巧:**type affinity**(列有"亲和性"而非强类型)、值存什么类型由值决定、对照 MySQL 强类型,为什么这样设计(灵活、兼容)。源码:`src/util.c`、`src/resolve.c`。归属:`存储与事务`。
- **第 17 章 · 并发模型:database lock**(`P5-17`)——多个连接/线程怎么并发?技巧:**文件锁**(database-level lock,5 种锁状态:unlocked/shared/reserved/pending/exclusive)、**reader/writer**、为什么 SQLite 并发比 MySQL 弱(单文件锁,不是行锁)。源码:`src/os_unix.c`、`src/pager.c`、`src/btmutex.c`。归属:`存储与事务`。

### 第 6 篇 · 进阶(3 章)

- **第 18 章 · 事务隔离与 BEGIN/COMMIT**(`P6-18`)——事务怎么开始结束、隔离怎么样?技巧:**BEGIN/COMMIT**(deferred/immediate/exclusive 三种 BEGIN)、SQLite 的隔离级别(默认 SERIALIZABLE-ish)。源码:`src/main.c`、`src/pager.c`。归属:`存储与事务`。
- **第 19 章 · 触发器 / 视图 / FTS**(`P6-19`)——高级 SQL 特性怎么在 VDBE 实现?技巧:**触发器**(编译进 opcode 流)、**视图**(展开成子查询)、**FTS 全文索引**(`ext/fts`)。源码:`src/trigger.c`、`src/select.c`、`ext/fts3/`。归属:`编译与执行`。
- **第 20 章 · prepared statement 与复用**(`P6-20`)——SQL 怎么编译一次、执行多次?技巧:**prepared statement**(编译成 opcode 后反复 bind 参数执行)、参数绑定、对照《MySQL》prepared。源码:`src/prepare.c`、`src/vdbeapi.c`。归属:`编译与执行`。

### 第 7 篇 · 收束(1 章)

- **第 21 章 · 全书收束:SQLite vs MySQL/PG vs LevelDB**(`P7-21`)——VDBE + B-tree + 单文件这套,得到了什么(零配置、嵌入式、稳定)、付出了什么(单写者、并发弱、无网络);**SQLite vs MySQL/PG(C/S) vs LevelDB(KV)对照总表**;展望 SQLite 在端侧 AI、KV 后端的新角色。归属:`总览`。

  - **B-tree + WAL → 承接《MySQL·InnoDB》那本**。SQLite 用 B-tree(非 B+树)、WAL/rollback journal,和 InnoDB 的 B+树/redo 同源思想,本书对照"嵌入式单文件 vs C/S 引擎"。
  - **嵌入式存储 → 承接《LevelDB》那本**。SQLite 和 LevelDB 都是嵌入式(链接进应用、单文件),对照"SQL 接口 vs KV 接口""B-tree vs LSM"。
  - **C/S 对照 → 承接《PG/MySQL》那本**。SQLite 嵌入式 vs PG/MySQL 的 C/S 架构,两种数据库部署形态对照。
  - **(附带)多种内存分配可切换(mem0-5)→ 承接《内存分配器》**。
- **引用规范**:本地相对路径 `[描述](../sqlite/src/文件.c#L起-L止)`;区间 `#La-Lb`、单行 `#L`;VSCode 预览可点击跳转。
- **版本钉死**:全书以 `sqlite @ 07607c6 (3.5x dev)` 为准,行号统一;若引用处 master 已变动,标注 commit。
- **⚠️ 架构演进交代(本书独有,务必处理)**:SQLite 演进相对稳定(几十年 Hipp 主导),但仍要交代几处:
  1. **WAL(3.7+,2010)**:老资料讲 rollback journal 是默认(仍对),但 WAL 模式是现代高并发首选,讲清两者。
  2. **JSON(3.9+)**、**窗口函数(3.25+)**、**UPSERT(3.24+)**:这些是较新加的 SQL 特性,老资料可能没有。
  3. **os_kv(KV VFS,新)**:把 SQLite 存到 KV 存储后端的新特性,涉及处讲清。
  4. **memdb(内存 db)**、**mem0-5 多种内存分配**:体现 SQLite 的可裁剪/可替换。

## 六、写作约定(沿用《gRPC》《TiKV》《MySQL》系列,LuaJIT 级标杆)

1. **动机 + 技巧双线**(本书命脉,见第二节):每个关键代码点讲"为什么这么设计" + "用了什么技巧、不这么写会怎样";每章单开"技巧精解"小节。
2. **"为什么"永远先于"怎么做"**:先讲它解决什么本质问题、不这样会怎样,再讲实现。
3. **大白话 + 比喻点睛**(直球为主):绝大多数地方第一性原理直给;仅在开篇用"自带发动机的小车"做一次性点睛,不建立贯穿重比喻。
4. **关键路径配真实源码**:挑本章最核心的 1~2 处,贴真实片段、逐段讲。**技巧精解**固定在正文后、小结前。
5. **每章标准结构**:① 章首(核心问题 + 读完你会明白 + 难度高的加逃生阀);② 正文若干小节(招牌四段式:**提问 → 不这样会怎样 → 所以这样设计 → 源码/技巧佐证**);③ 技巧精解;④ 章末小结(回扣"编译与执行 vs 存储与事务"主线 + 五个为什么清单 + 想深入往哪钻 + 引出下一章)。
6. **回扣全局**:每章结尾回到二分法,说明本章服务"编译与执行"还是"存储与事务"那一面,衔接前后章。
7. **承接前作**:VDBE 对照《Lua》虚拟机、B-tree/WAL 对照《MySQL·InnoDB》、嵌入式对照《LevelDB》、C/S 对照《PG/MySQL》、内存分配对照《内存分配器》,不重复造轮子。
8. **语言**:中文,技术名词保留英文(SQLite、VDBE、opcode、B-tree、WAL、rollback journal、pager、pcache、VFS、type affinity、prepared statement、cursor 等)。
9. **标点**:**半角 ASCII 逗号 `,`、冒号 `:`、括号 `()`、问号 `?`;全角顿号 `、`、破折号 `——`、引号 `""`、句号 `。`**;中英文之间留空格。
10. **文件命名**:`P{篇}-{章}-标题.md`。第 0 篇第 1 章 = `P0-01-…`,第 2 篇第 5 章 = `P2-05-…`。
