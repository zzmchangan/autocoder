# 第 1 篇 · 第 2 章 · 架构全景:八层流水线

> **核心问题**:上一章我们用"一条 SELECT 的旅程"把 SQLite 的三个本质串起来了——SQL 编译成 opcode、VDBE 虚拟机执行、单文件 B-tree 存、WAL/journal 保 ACID。可这只回答了"它做了什么",没回答"它内部怎么切成块的"。现在把镜头拉近:当你调 `sqlite3_prepare_v2()` 传进一句 SQL,到 `sqlite3_step()` 把第一行结果吐出来,中间这条链路到底经过了哪几层?每一层的职责边界画在哪?为什么 SQLite 要把它切成 **Interface / Tokenizer / Parser / Code Generator / VDBE / B-tree / Pager / OS Interface(VFS)** 这八层,而不是一个大一统的"数据库引擎"?为什么是**单向调用**(上层只依赖下层接口、下层不回调上层)?这条单向链路,是怎么做到"换掉最底下一层 VFS,上面七层一行不用改"的?

> **读完本章你会明白**:
> 1. SQLite 的**八层流水线**分别是什么、对应的源码文件是哪些(`main.c`/`tokenize.c`/`parse.y`/`select.c`/`vdbe.c`/`btree.c`/`pager.c`/`os_unix.c`)、每层的输入输出是什么。
> 2. **prepared statement**(编译一次、执行多次)是怎么落到源码里的——`sqlite3_stmt` 其实就是 `Vdbe` 结构体的别名,编译期产物和执行期状态怎么在这一个对象里共存。
> 3. **三段式对象生命周期**:`sqlite3_prepare_v2`(编译)→ `sqlite3_step`(执行)→ `sqlite3_finalize`(销毁),三步分别在八层的哪几层干活,以及 `sqlite3_reset` 为什么是"回到 READY 不重新编译"。
> 4. 为什么 VFS 是"可裁剪/可替换的接缝"——`sqlite3_vfs` 这张函数指针表,怎么让 SQLite 一套代码同时跑在 POSIX/Windows/内存/KV 后端上,以及它和《MySQL·InnoDB》C/S 架构里"handler→engine"那条接缝的本质区别。
> 5. 为什么分层解耦的代价是值得的(可裁剪、可移植、可换实现),以及为什么 SQLite 选八层(不是六层也不是十层)。

> **如果一读觉得太密**:先抓住三件事——① 一条 SQL 永远是**先编译(prepare)成 opcode、再执行(step)opcode**,编译和执行是分开的两步,中间产物就是 prepared statement;② 八层从上到下**单向依赖**,上层只看下层接口,不感知下层实现;③ 最底下两层(Pager / VFS)是"存储与事务",上面六层是"编译与执行",这两大块在 Code Generator → VDBE 那个接缝处交汇。本章是**地图章**,细节会在后面 18 章逐层深挖,这里只把全景和调用链立起来。

## 〇、一句话点破

> **SQLite 是一条单向流水线:SQL 字符串从最上层的 Interface 进来,被 Tokenizer 切词、Parser 建树、Code Generator 翻译成 opcode,产物 VDBE(program)在执行时调 B-tree 读页、B-tree 调 Pager 缓存、Pager 调 VFS 落盘——每一层只依赖下层的接口、绝不反向调用,这让最底下的 VFS 可以整层换掉而上面七层一行不改。**

这是结论,不是理由。本章倒过来拆:先看清八层各自是什么、对应的源码文件在哪,然后跟着一条真实的 `prepare → step → finalize` 调用链,逐层拆"这一步到底落到了哪层的哪个函数、传了什么、返回了什么",再讲透 prepared statement 这个对象在源码里到底是什么(`sqlite3_stmt` ↔ `Vdbe` 的别名关系),最后回答最关键的设计问题——为什么要分八层、为什么单向、为什么 VFS 是可替换的接缝。

## 一、八层流水线:先看清全貌

SQLite 官方文档里有一张著名的"Architecture of SQLite"图,画的就是这八层。本书把它落成一张带源码文件标注的图,这样你一边读一边能跳进真实文件里对。

### 八层从上到下

这张图有四个要点先钉死,后面每一节都是在展开它们:

1. **从上往下单向依赖**。第 N 层只能调第 N+1 层(或者更下)的接口,绝不能反向回调。VFS 永远不知道上面是 Pager 还是别的;pager 永远不知道上面是 B-tree 还是 dbpage(`src/dbpage.c`,一个直接看页的调试接口);B-tree 永远不知道上面是 VDBE 还是 vacuum。**每一层对上层是个黑盒接口,对下层是个调用者**。

2. **每一层有清晰的职责边界,且职责互不重叠**。Tokenizer 只管切词不懂语法;Parser 只管建树不懂执行;Code Generator 只管产出 opcode 不懂 B-tree 怎么读页;VDBE 只管执行 opcode 不懂页怎么缓存;B-tree 只管页之间的树形导航不懂缓存替换;Pager 只管缓存和事务不懂文件怎么读写;VFS 只管把文件操作映射到具体 OS。**没有一层"什么都管"的上帝层**。

3. **编译与执行的接缝在 Code Generator ↔ VDBE 之间**。第 1~4 层(Interface/Tokenizer/Parser/Code Generator)合起来叫**编译前端**,它们的产物是一个 opcode 流(prepared statement)。第 5 层 VDBE 是**执行器**,它消费 opcode。这两者之间是一条明确的接缝:**SQL 先被完全编译成 opcode,编译完之后才轮到执行**。这正是 SQLite 最核心的架构决策,也是它承接《Lua》"源码→字节码→虚拟机"那条线的地方。

4. **存储与事务的接缝在 VDBE ↔ B-tree 之间**。第 5 层 VDBE 执行 opcode 时,凡是"读数据/写数据",都通过 cursor 调 B-tree 的接口(`sqlite3BtreeTableMoveto`/`sqlite3BtreeInsert`/`sqlite3BtreeDelete`…)。从 B-tree 往下(B-tree/Pager/VFS)这三层,合起来是**存储与事务**子系统。VDBE↔B-tree 这个接缝,就是全书二分法"编译与执行 vs 存储与事务"在源码里的落点。

> **钉死这件事**:这八层不是任意切的,而是**按"职责单一、接口稳定、可独立替换"的原则分层**。后面你会看到:换 VFS 不用动 Pager,换内存分配器(`mem0~5`)不用动任何上层,关掉 shared cache 只改一个 flag。这种可裁剪性,是 SQLite 能塞进任何设备的根。如果 SQLite 是"一锅炖"的大函数,这些都不可能。

### 八层和"编译与执行 vs 存储与事务"二分法的对应

把八层映射回上一章立的二分法:

| 层 | 编译与执行 / 存储与事务 | 备注 |
|----|------------------------|------|
| 1. Interface | **编译与执行(总入口)** | 既管编译(prepare)又管执行(step)的分发 |
| 2. Tokenizer | **编译与执行** | 编译前端第一站 |
| 3. Parser | **编译与执行** | 编译前端第二站 |
| 4. Code Generator | **编译与执行** | 编译前端最后一站,产出 opcode |
| 5. VDBE | **编译与执行 ↔ 存储与事务(枢纽)** | 执行 opcode(编译侧),执行时调 B-tree(存储侧) |
| 6. B-tree | **存储与事务** | 存储子系统第一站 |
| 7. Pager | **存储与事务** | 缓存 + WAL/journal + ACID |
| 8. VFS | **存储与事务** | 可移植 OS 抽象 |

记住这张表。后面任何一章,你都能根据"这章讲的层落在哪一行"立刻判断它服务二分法的哪一面。

## 二、跟着一条真实的 SQL 走一遍:prepare / step / finalize 三段式

光看分层图是抽象的。现在我们用一条最朴素的 SQL——

```c
sqlite3 *db;
sqlite3_stmt *stmt;
sqlite3_open_v2("test.db", &db, SQLITE_OPEN_READWRITE, 0);
sqlite3_prepare_v2(db, "SELECT name FROM users WHERE id=10", -1, &stmt, 0);
while( sqlite3_step(stmt)==SQLITE_ROW ){
    printf("%s\n", sqlite3_column_text(stmt, 0));
}
sqlite3_finalize(stmt);
sqlite3_close_v2(db);
```

——一行一行追,看 `open / prepare / step / finalize` 这四步分别落在八层的哪几层。**这一节的目标,是让你在脑子里建起一条"API 调用 → 八层 → 源码函数"的精确映射**。

下面这张时序图先给全景,然后我们逐层拆。


注意时序图上三个关键的"分界":

- **第 2~6 步(open)**:只走存储侧(Interface→Pager→VFS),不碰编译。`open` 的本质是"把数据库文件和连接对象关联起来"。
- **第 7~13 步(prepare)**:走编译侧(Interface→Tokenizer→Parser→Code Generator),产物是一个装满 opcode 的 `Vdbe` 对象。**prepare 不碰 B-tree 的数据**(它要读 schema,但那是元数据,不是用户数据)。
- **第 14~24 步(step)**:走执行 + 存储(VDBE→B-tree→Pager→VFS),这是真正读用户数据的阶段。

下面把这三段逐段拆透。

### 2.1 `sqlite3_open_v2`:建连接,只走存储侧

很多讲 SQLite 架构的文章会跳过 `open`,直接讲 `prepare`。但 `open` 其实很有讲头——它揭示了 SQLite 的**初始化路径**和**VFS 的选定时机**。

`sqlite3_open_v2` 只有一行,直接转给内部函数 `openDatabase`:

```c
/* src/main.c */
int sqlite3_open_v2(
  const char *filename,   /* Database filename (UTF-8) */
  sqlite3 **ppDb,         /* OUT: SQLite db handle */
  int flags,              /* Flags */
  const char *zVfs        /* Name of VFS module to use */
){
  return openDatabase(filename, ppDb, (unsigned int)flags, zVfs);
}
```

([sqlite3_open_v2 转给 openDatabase](../sqlite/src/main.c#L3694-L3701))

`openDatabase` 是个 600 多行的大函数(`src/main.c` 第 3325 行起),它干三件事,每一件都对应八层里的某一层:

1. **第一件事:库级初始化**。`openDatabase` 第一行就是 `rc = sqlite3_initialize();`([openDatabase 调 sqlite3_initialize](../sqlite/src/main.c#L3343))。`sqlite3_initialize` 是 SQLite 全局只跑一次的初始化([sqlite3_initialize](../sqlite/src/main.c#L190))——它注册默认 VFS、设置默认内存分配器(mem0/pcache1)、初始化 mutex 子系统。如果库已经初始化过,这次调用就是个 no-op。**这是"库级"初始化,不是"连接级"**——多个连接共享一次初始化。

   > **钉死这件事**:`sqlite3_initialize` 是惰性的。你不显式调它,SQLite 在第一次 `open` 时也会帮你调。这就是为什么 SQLite 号称"零配置"——你什么都不用做,第一次 `open` 它自己把所有全局状态建好。这是嵌入式数据库和 C/S 数据库(要 `mysqld --initialize`、`initdb`)的鲜明对比。

2. **第二件事:选定 VFS**。`openDatabase` 根据 `zVfs` 参数(传 `NULL` 就是默认 VFS)找到对应的 `sqlite3_vfs` 对象。在 Unix 上默认是 "unix" VFS(`src/os_unix.c`),Windows 上是 "win32" VFS(`src/os_win.c`)。**VFS 在这一刻就被钉死进连接对象,后面所有文件操作都走这个 VFS**。这就是为什么 VFS 可以"整层换掉"——只要在 open 时指定一个自定义 VFS(传 `zVfs="myvfs"`),上面七层完全无感。

3. **第三件事:构造 `sqlite3` 连接对象**(就是那个 `db`)。`struct sqlite3` 是 SQLite 里最大的结构体之一,定义在 `src/sqliteInt.h` 第 1669 行([struct sqlite3](../sqlite/src/sqliteInt.h#L1669)),里面装着:VFS 指针、所有打开的 B-tree(`aDb[]`)、缓存配置、mutex、当前活跃的 prepared statement 链表(`pVdbe`)、错误码……这个对象的生命周期 = 整个连接的生命周期,从 `open` 到 `close`。

注意 `open` 这一步**没有真正读数据库文件**。它只是把 `sqlite3` 对象建好、把 VFS 挂上、把 pager 的壳子搭起来。**第一次真正读文件,是在 `prepare` 读 schema 的时候**(见下)。

> **不这样会怎样**:如果 `open` 就把整个数据库文件 mmap/read 进内存,那对于几十 GB 的库,光 `open` 就要几秒——这对"嵌入式、按需读"的场景是灾难。SQLite 的设计是**惰性**:连接对象先建好,数据按需(执行到某条 opcode 要读某页时)才进缓存。这个"惰性加载"的思想,和《LevelDB》那本讲的"打开 DB 不读数据、查到才读"是一致的(承《LevelDB》嵌入式)。

### 2.2 `sqlite3_prepare_v2`:编译期,产出 opcode 流

这是八层流水线的**上半场**——SQL 从字符串变成 opcode。这是本书"编译与执行"这一面的核心,本节先把主流程立起来,后面 P1-03 / P1-04 会拆到 Tokenizer / Parser / Code Generator 的内部细节。

`sqlite3_prepare_v2` 的官方声明在 `src/sqlite.h.in`([sqlite3_prepare_v2 声明](../sqlite/src/sqlite.h.in#L4620)):

```c
int sqlite3_prepare_v2(
  sqlite3 *db,            /* Database handle */
  const char *zSql,       /* SQL statement, UTF-8 encoded */
  int nByte,              /* Maximum length of zSql in bytes. */
  sqlite3_stmt **ppStmt,  /* OUT: Statement handle */
  const char **pzTail     /* OUT: Pointer to unused portion of zSql */
);
```

它本身只有两行([sqlite3_prepare_v2 实现](../sqlite/src/prepare.c#L943-L960))——直接转给 `sqlite3LockAndPrepare`,多传一个 `SQLITE_PREPARE_SAVESQL` 标志。这个标志的含义后文讲(它让 prepared statement 记住原始 SQL,以便 schema 变了能自动重编译)。

`sqlite3LockAndPrepare` 是"加锁 + 重试"的包装层([sqlite3LockAndPrepare](../sqlite/src/prepare.c#L842-L881)):

```c
static int sqlite3LockAndPrepare(
  sqlite3 *db,
  const char *zSql, int nBytes, u32 prepFlags,
  Vdbe *pOld, sqlite3_stmt **ppStmt, const char **pzTail
){
  int rc;  int cnt = 0;
  *ppStmt = 0;
  if( !sqlite3SafetyCheckOk(db)||zSql==0 ) return SQLITE_MISUSE_BKPT;
  sqlite3_mutex_enter(db->mutex);      // 连接级 mutex
  sqlite3BtreeEnterAll(db);            // 进所有 B-tree 的子锁
  do{
    /* 编译可能因 schema 变化需要重试,最多重试一次 */
    rc = sqlite3Prepare(db, zSql, nBytes, prepFlags, pOld, ppStmt, pzTail);
    if( rc==SQLITE_OK || db->mallocFailed ) break;
    cnt++;
  }while( (rc==SQLITE_ERROR_RETRY && ALWAYS(cnt<=SQLITE_MAX_PREPARE_RETRY))
       || (rc==SQLITE_SCHEMA && (sqlite3ResetOneSchema(db,-1), cnt)==1) );
  sqlite3BtreeLeaveAll(db);
  rc = sqlite3ApiExit(db, rc);
  db->busyHandler.nBusy = 0;
  sqlite3_mutex_leave(db->mutex);
  return rc;
}
```

注意三件事:

- **加锁**。`sqlite3_mutex_enter(db->mutex)` 是**连接级 mutex**(默认 `SQLITE_OPEN_FULLMUTEX` 时,一个连接同时只能有一个线程在 prepare/step)。`sqlite3BtreeEnterAll` 是进所有 B-tree 的子锁(shared cache 模式下用来保护 schema)。这两层锁是 SQLite 多线程安全的根——后面 P5-17 会专门拆并发模型,这里只要记住"prepare 整个过程在 mutex 保护下"。
- **重试循环**。`do-while` 处理的是一种叫 `SQLITE_SCHEMA` 的情况——**你正 prepare 的时候,别的连接改了 schema**(加了列、删了表),你的 AST 可能基于过期 schema,这时要 reset schema 重新 prepare 一次。这种"schema 变了自动重试"的容错,是 SQLite 稳定性的一个细节。
- **真正干活的是 `sqlite3Prepare`**。`sqlite3Prepare` 是编译主流程([sqlite3Prepare](../sqlite/src/prepare.c#L688-L842)),它把 SQL 字符串交给 `sqlite3RunParser`(这是编译前端的统一入口),里面依次跑 Tokenizer 和 Parser,然后 Code Generator 把 AST 翻成 opcode。

`sqlite3Prepare` 的关键一段是这一行——它把整个编译前端一气呵成:

```c
/* src/prepare.c, 在 sqlite3Prepare 里 */
if( nBytes>=0 && (nBytes==0 || zSql[nBytes-1]!=0) ){
    // ... 拷贝 SQL 字符串 ...
    sqlite3RunParser(&sParse, zSqlCopy);
}else{
    sqlite3RunParser(&sParse, zSql);
}
```

([sqlite3Prepare 调 sqlite3RunParser](../sqlite/src/prepare.c#L785-L793))

**`sqlite3RunParser` 就是八层的第 2~4 层的合体入口**——它住在 `src/tokenize.c` 第 600 行([sqlite3RunParser](../sqlite/src/tokenize.c#L600)),内部循环:每次调用 Tokenizer 拿一个 token、把 token 喂给 Parser、Parser 边建 AST 边触发 Code Generator 产出 opcode。整个过程是**增量、流式**的(token 一边切,树一边建,opcode 一边生成),不是"先切完所有 token 再建树再产 opcode"。这个流式设计让 SQLite 编译大 SQL 时不需要先存住所有 token。

编译完之后,`sqlite3Prepare` 把产出的 `Vdbe` 对象(里面 `aOp[]` 装满了 opcode)通过 `*ppStmt` 返回给调用者。**这就是 `sqlite3_prepare_v2` 的产物——一个 `sqlite3_stmt*`,它在源码里就是一个 `Vdbe*`**(下面技巧精解会拆这个别名)。

> **钉死这件事**:`sqlite3_prepare_v2` 走完,八层的第 1~4 层(Interface/Tokenizer/Parser/Code Generator)全部跑完,产物是一个"装满 opcode 的 Vdbe 对象"。**B-tree/Pager/VFS 这时候基本没动**(只有读 schema 时碰了一下)。opcode 还没执行——执行是 `step` 的事。这就是"编译与执行分离"在 prepare/step 两步上的具体体现。

### 2.3 `sqlite3_step`:执行期,VDBE 跑 opcode,逐层下沉到 B-tree

`sqlite3_prepare_v2` 给你一个装满 opcode 的 `sqlite3_stmt`。但这些 opcode 此刻一行也没跑。要让它跑,调 `sqlite3_step`:

```c
/* src/sqlite.h.in */
#define SQLITE_ROW         100  /* sqlite3_step() has another row ready */
#define SQLITE_DONE        101  /* sqlite3_step() has finished executing */
int sqlite3_step(sqlite3_stmt*);
```

([SQLITE_ROW/DONE 定义](../sqlite/src/sqlite.h.in#L479-L480) · [sqlite3_step 声明](../sqlite/src/sqlite.h.in#L5330))

`sqlite3_step` 的实现在 `src/vdbeapi.c`([sqlite3_step](../sqlite/src/vdbeapi.c#L978))。它本身是个外壳,负责"schema 变了自动重 prepare"的重试循环,真正执行 opcode 的核心是 `sqlite3Step`([sqlite3Step](../sqlite/src/vdbeapi.c#L836))。`sqlite3Step` 最重要的几行:

```c
static int sqlite3Step(Vdbe *p){
  sqlite3 *db;  int rc;
  db = p->db;
  if( p->eVdbeState!=VDBE_RUN_STATE ){
    restart_step:
    if( p->eVdbeState==VDBE_READY_STATE ){
      /* ... 准备执行:重置中断标志、计数器 ... */
      db->nVdbeActive++;
      if( p->readOnly==0 ) db->nVdbeWrite++;
      if( p->bIsReader ) db->nVdbeRead++;
      p->pc = 0;                          // 程序计数器清零
      p->eVdbeState = VDBE_RUN_STATE;     // 进入 RUN 态
    }else if( ALWAYS(p->eVdbeState==VDBE_HALT_STATE) ){
      /* ... 自动 reset 再重来 ... */
      sqlite3_reset((sqlite3_stmt*)p);
      goto restart_step;
    }
  }
  /* ... */
  rc = sqlite3VdbeExec(p);   // ★ 真正执行 opcode 的地方
```

([sqlite3Step 进入 RUN 态并调 sqlite3VdbeExec](../sqlite/src/vdbeapi.c#L836-L928))

`sqlite3VdbeExec` 是 SQLite 最核心、最大的函数,住在 `src/vdbe.c` 第 881 行([sqlite3VdbeExec](../sqlite/src/vdbe.c#L881))。它是一个**巨大的 switch-case 循环**——读当前 opcode(`p->aOp[p->pc]`)、根据 opcode 类型跳到对应 case、执行逻辑、更新 `pc`、读下一条。下面是它的"骨架"(只保留框架,真实函数有几千行 case):

```c
int sqlite3VdbeExec(Vdbe *p){
  Op *aOp = p->aOp;     // 整条 opcode 流
  Op *pOp = aOp;        // 当前 opcode 指针
  int rc = SQLITE_OK;
  sqlite3 *db = p->db;
  Mem *aMem = p->aMem;  // 寄存器组
  /* ... 各种局部寄存器缓存 ... */

  assert( p->eVdbeState==VDBE_RUN_STATE );
  /* ... 初始化 ... */

  for(;;){                         // 主循环
    /* ... */
    switch( pOp->opcode ){         // 按 opcode 分派
      case OP_OpenRead:            // 打开表读
        /* ... 调 sqlite3BtreeCursor ... */
        break;
      case OP_SeekRowid:           // 按 rowid 定位
        /* ... 调 sqlite3BtreeTableMoveto ... */
        break;
      case OP_Column:              // 取某列
        /* ... 从 cursor 取当前行某列到寄存器 ... */
        break;
      case OP_ResultRow:           // 返回一行结果
        /* ... 设置 pResultRow, 跳出循环回 SQLITE_ROW ... */
        goto vdbe_return;
      case OP_Next:                // 游标前进
        /* ... */
        break;
      /* ... 几十个其他 opcode ... */
      case OP_Halt:                // 执行结束
        goto vdbe_return;
    }
    pOp++;   // pc 前进(部分 opcode 会改 pc 做跳转)
  }
vdbe_return:
  /* ... 收尾、写回 p->pc ... */
  return rc;
}
```

([sqlite3VdbeExec 开头的局部变量缓存](../sqlite/src/vdbe.c#L881-L940))

这个 switch-case 循环是 SQLite 的灵魂——它和《Lua》那本的 `luaV_execute` 主循环**是同一个东西**(`OP_*` opcode + 寄存器 + program counter)。本书 P2-05 会把这个循环逐字拆透,这里只要抓住:**opcode 在这个循环里被一条条吃掉,吃掉每一条时会调到下层(B-tree /Pager / VFS)的接口**。

举一个最具体的例子。一条 `SELECT name FROM users WHERE id=10` 编译出的 opcode 大概长这样(用 `EXPLAIN` 能看到):

```
addr  opcode      p1    p2    p3     注释
----  ----------  ----  ----  ----   ---------------------------------
0     OpenRead    0     2     0      打开表 users(根页 2)读
1     Integer     10    1     0      常量 10 → 寄存器 1
2     SeekRowid   0     3     1      表 0 按 rowid(=寄存器1 的 10)定位
3     Column      0     1     2      取第 1 列(name)→ 寄存器 2
4     ResultRow   2     1     0      寄存器 2 作为结果行返回
5     Close       0     0     0      关闭表
6     Halt        0     0     0      结束
```

`sqlite3VdbeExec` 执行这串 opcode 时,每条 opcode 触发的下层调用:

| opcode | VDBE 内部干的活 | 调到的下层接口 |
|--------|----------------|----------------|
| `OpenRead 0 2 0` | 打开根页 2 的 B-tree,建一个 cursor | `sqlite3BtreeCursor` → `pager->readPage(2)` → `vfs->xRead` |
| `Integer 10 1 0` | 把 10 写进寄存器 1 | 纯寄存器操作,不碰下层 |
| `SeekRowid 0 3 1` | 在 cursor 0 里按 rowid 定位 | `sqlite3BtreeTableMoveto` → 可能读多个页 |
| `Column 0 1 2` | 从当前行取列 1(name)解码 | 解码 cursor 缓存里的页,不碰下层 |
| `ResultRow 2 1 0` | 把寄存器 2 标为结果行,跳出循环 | 纯寄存器操作 |
| `Close 0` / `Halt` | 关游标,结束 | `sqlite3BtreeCloseCursor` |

**这就是八层流水线"逐层下沉"的真实样子**:VDBE 跑 opcode → opcode 要读页就调 B-tree → B-tree 要页就调 Pager → Pager 要读盘就调 VFS → VFS 调 POSIX `pread`。每一层只调下一层的接口,绝没有"VFS 反过来通知 VDBE"这种反向调用。

> **钉死这件事**:`sqlite3_step` 一次只推进到下一个 `ResultRow`(或 `Halt`)就返回。所以一条 SELECT 有 100 行结果,你就要 `step` 101 次(100 次 `SQLITE_ROW` + 1 次 `SQLITE_DONE`)。VDBE 的 program counter 会记住上次跑到哪,下次 `step` 从 `pc` 接着跑。这种"一次跑一段、用 pc 记位置"的设计,让 SQLite 可以**流式吐结果**——不需要先把所有结果算出来再返回,这对大结果集极其友好。

### 2.4 `sqlite3_column_text`:读结果,纯寄存器操作

`sqlite3_step` 返回 `SQLITE_ROW` 之后,你的应用调 `sqlite3_column_text(stmt, 0)` 取当前行的第 0 列。这一步**完全在 VDBE 内部**,不碰任何下层——因为 `ResultRow` opcode 已经把结果写进了 `Vdbe` 对象的寄存器(`pResultRow`),`sqlite3_column_text` 只是从那个寄存器读出来转成 `const unsigned char*`。

这是为什么 `sqlite3_column_*` 系列函数特别快:它不读盘、不解码 B-tree 页,纯粹是"从内存里的寄存器取一个值"。代价是:**返回的指针在下次 `step`/`reset`/`finalize` 后就失效了**(寄存器会被覆盖)——SQLite 文档反复强调的"不要把 column_text 的结果存下来"就是这个道理。

### 2.5 `sqlite3_finalize`:销毁,关游标 + 释放 opcode

最后,`sqlite3_finalize(stmt)` 销毁 prepared statement:

```c
int sqlite3_finalize(sqlite3_stmt *pStmt){
  int rc;
  if( pStmt==0 ) return SQLITE_OK;       // NULL 是 no-op
  Vdbe *v = (Vdbe*)pStmt;
  sqlite3 *db = v->db;
  sqlite3_mutex_enter(db->mutex);
  checkProfileCallback(db, v);
  rc = sqlite3VdbeReset(v);              // ★ 关所有游标、回滚未提交事务
  sqlite3VdbeDelete(v);                  // ★ 释放 opcode/寄存器/cursor
  rc = sqlite3ApiExit(db, rc);
  sqlite3LeaveMutexAndCloseZombie(db);
  return rc;
}
```

([sqlite3_finalize 实现](../sqlite/src/vdbeapi.c#L99-L118))

两步:

- `sqlite3VdbeReset`:如果 VDBE 还在 RUN 态,先停下来——关闭所有打开的 cursor、把脏页处理掉(回滚未提交的事务)、把状态从 `VDBE_RUN_STATE` 退回 `VDBE_HALT_STATE`。这个函数住在 `src/vdbeaux.c` 第 3593 行([sqlite3VdbeReset](../sqlite/src/vdbeaux.c#L3593))。
- `sqlite3VdbeDelete`:释放 `Vdbe` 对象本身——`aOp[]`(opcode 流)、`aMem[]`(寄存器)、`apCsr[]`(游标指针)全部 free 掉,把这个 Vdbe 从 `db->pVdbe` 链表里摘下来。

`finalize` 之后,这个 `sqlite3_stmt` 再也不能用了。如果你想重用一个 prepared statement(再执行一次,不销毁),调 `sqlite3_reset` 而不是 `finalize`——`reset` 只把状态从 HALT 退回 READY(关游标、清寄存器),**不重新编译、不释放 opcode**:

```c
int sqlite3_reset(sqlite3_stmt *pStmt){
  /* ... */
  Vdbe *v = (Vdbe*)pStmt;
  rc = sqlite3VdbeReset(v);
  sqlite3VdbeRewind(v);   // ← 关键:把 pc 清零、寄存器清零,但 opcode 不动
  /* ... */
}
```

([sqlite3_reset 实现](../sqlite/src/vdbeapi.c#L128-L138))

`reset` vs `finalize` 的区别,是 prepared statement "编译一次、执行多次"的关键:

| 操作 | 状态变化 | opcode | 游标/寄存器 | 对象本身 |
|------|---------|--------|-------------|----------|
| `prepare` | —→READY | 生成 | 分配 | 创建 |
| `step` | READY→RUN→HALT | 执行 | 用 | 在 |
| `reset` | HALT→READY | **保留** | 清零 | 在 |
| `finalize` | *→销毁 | **释放** | 释放 | 删除 |

> **钉死这件事**:prepared statement 的"复用"靠的是 `reset`——它把状态退回 READY 但 opcode 一字不动,下次 `step` 直接从 `pc=0` 重跑。这就是"编译一次、执行多次"在源码里的实现细节。如果你用 `finalize` 代替 `reset`,每次都要重新 prepare(重新跑 Tokenizer/Parser/Code Generator),性能差几个数量级。**`reset` 而不是 `finalize`,是 SQLite 高性能编程的第一条铁律。**

## 三、八层为什么是八层:设计动机逐层拆

到这里你已经能把 API 调用映射到八层了。但还有一个更根本的问题没回答:**为什么是八层?为什么不把 Tokenizer+Parser 合并、为什么不把 Pager 并进 B-tree?** 这一节回答这个"为什么"。每层都遵循四段式:**它解决什么问题 → 不这样会怎样 → 所以这样设计 → 源码佐证**。

### 3.1 Interface 层:为什么需要一个"API 分发层",而不是让用户直接调 Tokenizer

> **不这样会怎样**:如果让用户直接调 `sqlite3RunParser`,会出现两个灾难。第一,内部 API(`sqlite3RunParser` 等)会随着重构频繁变化,用户代码今天能编过明天就编不过——SQLite 一向的承诺是**公共 API 极度稳定**(`sqlite3_open/prepare/step` 几十年没动过签名),内部 API 每个版本都可能变。第二,每次调内部 API 都得自己加 mutex、加错误处理、加 schema 重试——这些"通用胶水"散落到用户代码里,既难写又容易错。

> **所以这样设计**:Interface 层(`src/main.c` / `src/legacy.c` / `src/prepare.c` / `src/vdbeapi.c`)是一个**薄薄的分发层**——它把公共 API(`sqlite3_open_v2`/`prepare_v2`/`step`/`finalize`/`exec`)翻译成内部调用,同时统一处理 mutex、错误码转换、schema 重试、API armor(参数校验)。用户只看到稳定的公共 API,内部怎么重构都不影响用户代码。

证据就是 `sqlite3_prepare_v2` 那两行——它只加一个 `SQLITE_PREPARE_SAVESQL` 标志就转给 `sqlite3LockAndPrepare`,而 `sqlite3LockAndPrepare` 才是干 mutex/重试/BtreeEnterAll 这些"胶水"的地方。Interface 层的薄,正是它的设计目的。

> **钉死这件事**:Interface 层不是"功能层",是"边界层"。它的存在,是为了把"稳定的公共 API"和"频繁变化的内部实现"隔开。这是分层架构最朴素也最值钱的动机。

### 3.2 Tokenizer vs Parser:为什么切词和建树要分开

> **不这样会怎样**:为什么不把切词和建树合并成一个阶段?朴素的做法是一个大函数:边读字符、边判断是不是关键字、边建树。这有两个问题。第一,**职责纠缠**——词法规则(什么是合法标识符、字符串怎么转义、数字字面量格式)和语法规则(SELECT 的结构、WHERE 怎么挂)是完全不同的两套规则,混在一起会让代码变成意大利面。第二,**无法复用**——SQLite 还要在别的地方用 Tokenizer(比如 `complete.c` 判断一个 SQL 字符串是否完整、`shell.c` 高亮语法),如果把切词嵌进 Parser,这些场景都得重新写一遍。

> **所以这样设计**:Tokenizer(`src/tokenize.c`)只负责"字符 → token",不懂任何语法;Parser(`src/parse.y` + 生成的 `parse.c`)只负责"token → AST",不懂字符细节。两层的接口是一个简单的"给我下一个 token"函数。这种切分是编译器的教科书结构(承《Lua》那本的 `llex.c`/`lparser.c` 也是这么分的),好处是:Tokenizer 可以独立测试、独立复用,Parser 只关心语法结构。

证据是 `sqlite3RunParser` 的结构——它在一个循环里反复调 Tokenizer 拿 token,把 token 喂给 Parser(`yyparse`,Lemon 生成的入口),Parser 在建树过程中遇到需要展开的语法节点,就调 Code Generator 产出 opcode。**三层(Token/Parser/CodeGen)是流式协作的,但代码边界清晰**。P1-03 会把这个循环逐行拆。

### 3.3 Parser vs Code Generator:为什么建 AST 和生成 opcode 分开

> **不这样会怎样**:SQLite 的 Parser 是用 Lemon(yacc 的变种)生成的,Parser 的产物本来可以直接是 opcode——很多教学型数据库就是这么干的(Parser 边识别语法、边直接 emit opcode)。但 SQLite 让 Parser 产出的是一个中间表示(AST,在 SQLite 里它不完全是经典 AST,而是挂在 `Parse` 对象上的语法节点 + 触发的 Code Generator 调用),然后由 `select.c`/`insert.c`/`update.c`/`delete.c`/`build.c` 这些文件里的 Code Generator 把它翻成 opcode。

> **所以这样设计**:把"识别 SQL 语法"(Parser)和"生成 opcode"(Code Generator)分开,有两个关键好处:

1. **Parser 可以用工具生成**。SQLite 用 Lemon 生成 Parser,自动处理左递归、冲突消解。Lemon 生成的代码只负责"语法对不对、AST 长什么样",不管"opcode 怎么排"——这样 Lemon 可以保持简单、稳定。
2. **Code Generator 可以做优化**。Code Generator(`select.c` 等文件)是手写的、复杂的——它要做 WHERE 化简、索引选择(`where.c`)、JOIN 顺序优化、子查询展开。这些优化只能在"AST 已经建好、能整体分析"的前提下做。如果 Parser 直接 emit opcode,这些优化都做不了。

证据:`src/select.c` 是 SQLite 最大的源码文件之一(几千行),它干的活全是"针对 SELECT 的 AST 节点,生成高效的 opcode"——`sqlite3Select`([sqlite3Select](../sqlite/src/select.c))就是这个分发中心。它绝不是"机械翻译 AST",里面有大量的优化逻辑。P1-04 会专门拆它。

> **钉死这件事**:Code Generator 是 SQLite 编译前端里**最复杂、最值得读**的一层。它是"声明式 SQL"和"命令式 opcode"之间的翻译官——你写的是 `SELECT ... WHERE ...`,出来的是一串 `OpenRead/SeekRowid/Column/Next` 这样的命令式指令。**这个翻译是 SQLite 把 SQL 的"做什么"翻译成机器能跑的"怎么做"的地方,是整个编译前端的高潮。** 这一层和《Lua》那本的 `lcode.c`(Lua 编译器把 AST 翻成字节码)是同构的——承《Lua》编译器。

### 3.4 Code Generator vs VDBE:为什么"编译成 opcode"而不是"直接执行 AST"

这是 SQLite 最核心的架构决策,上一章的"一句话点破"就立在这里。这一章从分层的角度再钉一遍:

> **不这样会怎样**:如果 SQLite 不编译成 opcode、而是直接遍历 AST 执行(很多教学型数据库这么干),会有三个问题:

1. **prepared statement 不可能**。AST 遍历执行每次都要从头走一遍树,无法"编译一次、缓存、多次 bind 参数执行"。SQLite 的招牌 prepared statement 模型直接垮掉。
2. **优化困难**。AST 是嵌套树,在树上做优化(选索引、重排 JOIN)极其别扭;opcode 是扁平指令流,优化器可以像分析汇编一样在上面做数据流分析。
3. **执行模型不统一**。SELECT/INSERT/UPDATE/DELETE 各自的 AST 长得完全不同,如果直接遍历 AST 执行,执行逻辑要写四套;而编译成 opcode 之后,**所有语句最后都是一串 opcode,由同一个 VDBE 执行**——执行器只有一套。

> **所以这样设计**:Code Generator 产出 opcode 流,VDBE 是一个统一的 opcode 执行器。**这条接缝是"声明式世界(SQL/AST)通往命令式世界(opcode/寄存器)"的单向门**——走过这道门,后面所有的事(执行、优化、复用)都在命令式世界里完成。这个思想和《Lua》把源码编译成字节码、用 Lua VM 执行**完全同构**(承《Lua》虚拟机)。

证据就是 `sqlite3VdbeExec` 那个巨大的 switch-case——它不区分 SELECT 还是 INSERT,所有 opcode 都在同一个循环里执行。这就是"执行模型统一"的代码体现。

> **钉死这件事**:这条"Code Generator → VDBE"的接缝,是全书最重要的接缝。它把 SQLite 切成了"编译前端"和"执行 + 存储"两大块,这正是全书二分法"编译与执行 vs 存储与事务"在源码里的精确落点。

### 3.5 VDBE vs B-tree:为什么执行器和存储要用 cursor 抽象隔开

> **不这样会怎样**:VDBE 执行 `OpenRead`/`Column` 这些 opcode 时,要读 B-tree 的数据。如果 VDBE 直接操纵 B-tree 的内部结构(页、cell、指针),会出现灾难:VDBE 要懂 B-tree 的二进制布局、要懂页缓存、要懂事务——执行器和存储器彻底耦合,任何一个改动都要两边一起改。

> **所以这样设计**:VDBE 和 B-tree 之间通过 **cursor(游标)** 抽象隔开。VDBE 说"打开表 users 的一个游标、把游标定位到 rowid=10、从游标取当前行的第 1 列"——它完全不知道 B-tree 内部长什么样。B-tree 这边收到 cursor 操作,自己去管页的加载、cell 的解码、树的遍历。**cursor 是个抽象指针,VDBE 拿着它就能读数据,不感知底层**。

证据:`OP_OpenRead` 这个 opcode 的 case 在 `src/vdbe.c` 第 4421 行([case OP_OpenRead](../sqlite/src/vdbe.c#L4421)),它内部调 `sqlite3BtreeCursor`(在 `btree.c`),建一个 `BtCursor` 结构体,然后 VDBE 把这个 cursor 的句柄存进自己的 `apCsr[]` 数组(`Vdbe.apCsr`,见上面 Vdbe 结构体的第 476 行)。后续 `SeekRowid`/`Column`/`Next` 都通过这个句柄调 B-tree。

> **钉死这件事**:cursor 抽象让"执行"和"存储"解耦——VDBE 可以独立演进(加新 opcode、做新优化),B-tree 也可以独立演进(换页格式、改缓存策略),两边只通过 cursor 这个稳定的接口对话。这种解耦,是 SQLite 能在几十年里持续演进而不破坏内部架构的根。

### 3.6 B-tree vs Pager:为什么树形导航和页缓存要分开

> **不这样会怎样**:B-tree 的本质是"在页之间做树形导航"(从根页开始、按 key 比较走到子页、直到叶子页)。但页在磁盘上,每次读都要进内存。如果 B-tree 自己管"哪个页在缓存、哪个页要读盘",B-tree 的代码会被缓存逻辑彻底污染——它既要管树形结构,又要管 LRU 替换、脏页标记。

> **所以这样设计**:B-tree(`src/btree.c`)只管"页之间的树形导航",它向 Pager 要页时只说"给我第 N 页";Pager(`src/pager.c`)负责"哪个页在缓存(pcache)、哪个页要读盘、哪个页脏了要写回、怎么保证 crash 不丢(WAL/journal)"。**B-tree 把 Pager 当成一个"按页号取页的数组"用,完全不感知页是从磁盘读的还是从缓存来的**。

这个分层极其关键——它让 B-tree 的代码可以纯粹地讲算法(树形结构),把"缓存和事务"这个复杂的工程问题完全甩给 Pager。** Pager 这个名字(Page-er)就是"管页的"的意思,它和《MySQL·InnoDB》的 buffer pool、《Linux内存管理》的 page cache 是同一类东西**(承《MySQL》buffer pool、承《Linux mm》page cache),P4-11 会专门拆。

### 3.7 Pager vs VFS:为什么"管页"和"读写文件"要分开

> **不这样会怎样**:Pager 要读写页,如果它直接调 POSIX `pread`/`pwrite`,SQLite 就只能跑在 POSIX 上——Windows、嵌入式 RTOS、内存 mock、KV 存储后端全都跑不了。

> **所以这样设计**:Pager 不直接调 OS,它通过 VFS(Virtual File System)这个抽象层调。"读第 N 页"对 Pager 来说是"调 VFS 的 xRead 方法,从文件偏移 N×pageSize 处读 pageSize 字节";**xRead 在 Unix VFS 里是 `pread` 的包装,在 Windows VFS 里是 `ReadFile` 的包装,在 `memdb`(内存 db)里是 `memcpy`,在 `os_kv`(KV 后端)里是一个 KV get**。VFS 是一张函数指针表([struct sqlite3_vfs](../sqlite/src/sqlite.h.in#L1513-L1542)):

```c
struct sqlite3_vfs {
  int iVersion;            /* 结构体版本 */
  int szOsFile;            /* 子类化 sqlite3_file 的大小 */
  int mxPathname;          /* 最大路径长度 */
  sqlite3_vfs *pNext;      /* 注册链表 */
  const char *zName;       /* 这个 VFS 的名字("unix"/"win32"/…) */
  void *pAppData;
  int (*xOpen)(sqlite3_vfs*, sqlite3_filename, sqlite3_file*, int, int*);
  int (*xDelete)(sqlite3_vfs*, const char *zName, int syncDir);
  int (*xAccess)(sqlite3_vfs*, const char *zName, int flags, int *pResOut);
  int (*xFullPathname)(sqlite3_vfs*, const char *zName, int nOut, char *zOut);
  /* ... 动态加载(xDlOpen 等)、随机数、时间 ... */
  int (*xCurrentTimeInt64)(sqlite3_vfs*, sqlite3_int64*);
  /* ... version 3: xSetSystemCall/xGetSystemCall/xNextSystemCall ... */
};
```

([struct sqlite3_vfs 完整定义](../sqlite/src/sqlite.h.in#L1513-L1542))

这张函数指针表就是"VFS 接缝"的全部——每个 VFS 实现(`unix`/`win32`/`memdb`/`kv`)就是填一张这样的表。**Pager 只看这张表,不关心表背后是什么 OS**。这是 SQLite 跨平台的根,也是它能塞进任何设备的根。P5-15 会专门拆 VFS。

> **钉死这件事**:VFS 是八层流水线最底下、也是最重要的接缝。它把"页级读写"和"OS 系统调用"彻底切开——换 OS、换存储后端,只换 VFS,上面七层一行不改。**这是分层架构"可替换"的极致体现。** 对照《MySQL·InnoDB》:`InnoDB` 也有自己的"文件抽象"(os/file),但 InnoDB 是 C/S 引擎,跑在服务端、OS 固定(典型 Linux),它的文件抽象远不如 SQLite 的 VFS 这么"可热插拔"——因为 SQLite 要适应从手机到航电到 KV 后端的各种环境,VFS 必须做成可替换的接缝。

### 3.8 八层是不是"刚好"

你可能问:为什么是八层,不是六层(合并 Tokenizer+Parser)、也不是十层(再拆出优化器)?SQLite 的回答是——**层数是职责自然长出来的,不是预设的**。Tokenizer 和 Parser 分开,是因为词法和语法是两套规则(3.2);Code Generator 独立,是因为优化太复杂(3.3);Pager 和 B-tree 分开,是因为缓存和树形导航是两个工程问题(3.6)。每一条接缝的存在,都能讲清"为什么这层独立存在"。如果某条接缝没有清晰的职责边界,SQLite 就不会硬拆——比如 SQLite 没有独立的"优化器层"(优化散在 Code Generator 里,`where.c` 就是优化器的一部分),因为 SQLite 的优化和代码生成是交织的,硬拆反而别扭。

> **钉死这件事**:八层不是教条,是"职责单一 + 接口稳定 + 可独立替换"原则的自然产物。SQLite 还允许你在编译时**裁掉某些层**——`SQLITE_OMIT_*` 宏能关掉触发器、视图、FTS 等特性,让 SQLite 进一步瘦身;`SQLITE_OS_OTHER` 能让你用一个完全自定义的 OS 层。**这种可裁剪性,只有在分层清晰的前提下才可能**——这是分层架构的真正回报。

## 四、prepared statement 的对象生命周期:`sqlite3_stmt` 到底是什么

prepare/step/finalize 三段式我们已经走过一遍了。现在把镜头对准那个贯穿三段的**对象**——`sqlite3_stmt`。这一节回答四个问题:它在源码里是什么?编译期产物和执行期状态怎么在一个对象里共存?它的状态机长什么样?bind 参数(bind 阶段)在哪?

### 4.1 `sqlite3_stmt` 就是 `Vdbe`

这是 SQLite 里一个让初学者迷惑的点:`sqlite3_prepare_v2` 的产物类型叫 `sqlite3_stmt`,但你翻遍 `src/sqlite.h.in` 找不到 `struct sqlite3_stmt` 的定义——因为**`sqlite3_stmt` 是 `Vdbe` 的别名**。看 `src/sqlite.h.in` 的注释:

> The prepared statement object [...] is an opaque object ...

公开头文件里它是个**不透明指针**——用户只知道它叫 `sqlite3_stmt*`,不知道里面有什么。真正定义在内部的 `src/vdbeInt.h` 第 458 行([struct Vdbe](../sqlite/src/vdbeInt.h#L458-L529)):

```c
struct Vdbe {
  sqlite3 *db;            /* 拥有这个语句的连接 */
  Vdbe **ppVPrev,*pVNext; /* 同一个 db 上的 Vdbe 链表 */
  Parse *pParse;          /* 创建它的编译上下文 */
  ynVar nVar;             /* 参数个数(? 的个数) */
  int nMem;               /* 寄存器个数 */
  int nCursor;            /* 游标个数 */
  int pc;                 /* 程序计数器(program counter) */
  int rc;                 /* 返回码 */
  Mem *aMem;              /* 寄存器组 */
  VdbeCursor **apCsr;     /* 游标指针数组 */
  Mem *aVar;              /* 参数寄存器(bind 的值放这) */
  Op *aOp;                /* opcode 流 ★编译期产物 */
  int nOp;                /* opcode 条数 */
  u8 eVdbeState;          /* INIT/READY/RUN/HALT 四态 */
  /* ... 还有几十个字段 ... */
  char *zSql;             /* 原始 SQL(prepare_v2 保存,以便 schema 变了重编译) */
};
```

([struct Vdbe 完整定义](../sqlite/src/vdbeInt.h#L458-L529))

这个结构体里**同时装着编译期产物和执行期状态**:

- **编译期产物**(prepare 时填好,执行时不改):`aOp[]`(opcode 流)、`nOp`(opcode 条数)、`nMem`(寄存器数)、`nVar`(参数数)、`nCursor`(游标数)、`zSql`(原始 SQL)。
- **执行期状态**(step 时不断变):`pc`(程序计数器)、`aMem[]`(寄存器值)、`apCsr[]`(当前打开的游标)、`aVar[]`(bind 进来的参数)、`rc`(当前返回码)、`eVdbeState`(当前状态)。

**一个对象两副面孔**:prepare 之后它是"一段编译好的程序",step 之后它是"一个跑了一半的进程"。这就是 prepared statement 的本质——**它是把"程序"(opcode)和"进程状态"(pc/寄存器)打包在一起的对象**。

> **钉死这件事**:`sqlite3_stmt` 不是什么神秘的包装,它就是 `Vdbe`。这个命名的"障眼法"(对外叫 statement,对内叫 Vdbe)是 SQLite API 设计的典型手法——对外提供不透明指针(只暴露 `sqlite3_stmt*`),对内用 `Vdbe` 这个实现类型。这种"接口/实现分离"是 C 语言做面向对象的标准技巧(承《Lua》的 `lua_State` 也是这么设计的)。

### 4.2 四态状态机:INIT / READY / RUN / HALT

`Vdbe.eVdbeState` 字段记录这个 prepared statement 当前在哪个状态。定义在 `src/vdbeInt.h` 第 534 行([VDBE 状态定义](../sqlite/src/vdbeInt.h#L534-L537)):

```c
#define VDBE_INIT_STATE     0   /* prepared statement 正在构造(prepare 还没返回) */
#define VDBE_READY_STATE    1   /* 准备好但还没开始(可以 step) */
#define VDBE_RUN_STATE      2   /* 正在执行(step 调用中) */
#define VDBE_HALT_STATE     3   /* 跑完了(等 reset 或 finalize) */
```

状态流转(对照 prepare/step/reset/finalize):

```
                prepare                 step(首次)
   INIT  ───────────────▶  READY  ───────────────▶  RUN
   (构造中)                 │  ▲                      │
                           │  │  step 中途             │
                           │  │  pc 停在 ResultRow     │
                           │  └─────────────────────  │
                           │       (回到 READY)        │
                           │                            │ step 跑到 Halt
                           │   reset / finalize         ▼
                           │   ┌───────────────────  HALT
                           │   │                     (跑完)
                           └───┘
                              reset
```

详细说明:

- **INIT→READY**:`prepare` 在构造 Vdbe(opcode 一条条 append 进去),构造完返回前把状态设成 READY。`VDBE_INIT_STATE` 在用户代码里几乎看不到(prepare 返回时已经不是 INIT 了),它只在 prepare 内部短暂存在。
- **READY→RUN**:第一次 `step` 时,`sqlite3Step` 检测到 `eVdbeState==VDBE_READY_STATE`,做执行前准备(清零 pc、计数 `nVdbeActive`),把状态设成 RUN,然后调 `sqlite3VdbeExec`。
- **RUN→READY**:`sqlite3VdbeExec` 跑到 `ResultRow` opcode 时,**不退出 RUN 态**——它把 pc 停在 ResultRow 之后,返回 `SQLITE_ROW`,但状态还是 RUN。下次 `step` 直接接着 RUN 跑(这是"一次 step 一行"的关键)。只有 `Halt` opcode 才把状态设成 HALT。
- **RUN→HALT**:跑到 `Halt` opcode(或出错),状态变 HALT,返回 `SQLITE_DONE`(或错误码)。
- **HALT→READY**:`reset` 把状态从 HALT 退回 READY(关游标、清寄存器、清 pc),可以再 step。
- **\*→销毁**:`finalize` 不管在哪个状态都先 reset(退回 READY→HALT),再 `sqlite3VdbeDelete` 释放整个对象。

> **钉死这件事**:这个四态状态机是 prepared statement 复用的基础。`reset` 不是"销毁重建",它是"HALT→READY 的状态转换"——opcode 一字不动,只清执行期状态(pc/寄存器/游标)。这就是为什么 reset 之后 step 极快(没有重新编译)。理解了这个状态机,你就理解了 `prepare/step/reset/finalize` 四个 API 的全部语义。

### 4.3 bind 参数:`?` 是怎么挂进 opcode 的

`SELECT name FROM users WHERE id=?` 里那个 `?`,在编译期就变成了一个 `OP_Variable` opcode——它的意思是"执行时从参数寄存器取值"。看 `Vdbe` 结构体里的 `aVar[]`(参数寄存器数组,第 477 行):**`sqlite3_bind_int(stmt, 1, 10)` 就是往 `aVar[0]` 里写 10**(参数编号从 1 开始,数组下标从 0 开始)。执行时 `OP_Variable` 把 `aVar[i]` 的值拷到某个工作寄存器,后续 opcode 用那个寄存器。

这就是 prepared statement 的精髓:

1. **prepare**:SQL 里每个 `?` 编译成一个 `OP_Variable` opcode,记录"执行时去 aVar 的第几格取"。`aVar` 数组在 prepare 时分配好,大小 = `?` 的个数。
2. **bind**:`sqlite3_bind_*` 往 `aVar[i]` 写值。这一步**不重新编译**——opcode 早就编好了,bind 只是填数据。
3. **step**:执行时碰到 `OP_Variable`,从 `aVar[i]` 取值用。
4. **reset + 再 bind + 再 step**:换个参数值,再跑一次。opcode 流一字不动,只是 `aVar` 换了值。

> **不这样会怎样**:如果没有 prepared statement,每次执行 `WHERE id=10` 都要把 "10" 当字面量编进 opcode(`OP_Integer 10`)——换 id 就得重新编译整条 SQL。prepared statement 把"SQL 结构(opcode)"和"参数值(aVar)"解耦,结构编一次,值随便换。这是 SQLite 高性能的核心机制,P6-20 会专门拆。

## 五、八层流水线 vs MySQL/InnoDB 的分层:为什么 SQLite 是"函数调用链"而不是"网络协议栈"

这一节做一次跨架构对照,把 SQLite 的八层流水线放回"数据库架构谱系"里看,你会更清楚它为什么这么切。

### 5.1 SQLite 是同进程函数调用链,MySQL 是跨进程网络协议栈

上一章讲过,SQLite 是嵌入式、MySQL 是 C/S。这个差异在分层上长什么样?

```
   MySQL/InnoDB(C/S)——跨进程,通过网络协议:
   ┌──────────┐                    ┌──────────────────────────────┐
   │ 客户端    │ ──MySQL 协议──▶    │ 服务端进程                    │
   │ (你的App)│                    │  ┌────────────────────────┐  │
   │  libmysql│                    │  │ protocol 层(收发包)    │  │
   └──────────┘                    │  │  ↓                      │  │
                                   │  │ parser(解析)           │  │
                                   │  │  ↓                      │  │
                                   │  │ optimizer(优化)        │  │
                                   │  │  ↓                      │  │
                                   │  │ executor(执行)         │  │
                                   │  │  ↓                      │  │
                                   │  │ handler 接口 ←接缝       │  │
                                   │  │  ↓                      │  │
                                   │  │ InnoDB 引擎(buffer pool │  │
                                   │  │   / redo / undo / B+树) │  │
                                   │  └────────────────────────┘  │
                                   └──────────────────────────────┘
   每一层之间是函数调用,但客户端↔服务端是网络往返。

   SQLite(嵌入式)——同进程,纯函数调用:
   ┌────────────────────────────────────────────────────┐
   │ 你的 App 进程                                       │
   │  ┌──────────────────────────────────────────────┐ │
   │  │ Interface → Tokenizer → Parser → CodeGen     │ │
   │  │  → VDBE → B-tree → Pager → VFS ── 文件       │ │
   │  └──────────────────────────────────────────────┘ │
   │  每一层之间都是直接函数调用,零网络往返。            │
   └────────────────────────────────────────────────────┘
```

关键差异:

- **MySQL 的"分层"跨进程**。客户端调 `mysql_query`,这个调用要先走 MySQL 客户端协议库、序列化成网络包、TCP 传到服务端、服务端 protocol 层解包、再走 parser/optimizer/executor。**每一层都有序列化/反序列化的开销**,更别说网络往返延迟。
- **SQLite 的"分层"全在同进程**。`sqlite3_prepare_v2` → `sqlite3LockAndPrepare` → `sqlite3Prepare` → `sqlite3RunParser` → Tokenizer → Parser → Code Generator,每一步都是**直接的 C 函数调用**,参数是内存指针,没有序列化、没有网络。这就是为什么 SQLite 在嵌入式场景里比 MySQL 快几个数量级——它省掉了 C/S 那一整条协议栈的开销。

### 5.2 "handler 接缝" vs "VDBE/B-tree 接缝":都是"执行↔存储"的边界,但哲学不同

MySQL/InnoDB 有一个著名的 **handler 接口**(`handler` 类)——它是"SQL 执行层"和"存储引擎层"之间的接缝,执行层通过 handler 接口调 InnoDB/MyISAM/其他引擎。这个接缝让 MySQL 可以**插件式换存储引擎**。

SQLite 也有一个类似的接缝——**VDBE/B-tree 之间的 cursor 接口**。但两者哲学不同:

| 维度 | MySQL handler 接缝 | SQLite VDBE/B-tree 接缝 |
|------|--------------------|-------------------------|
| 目的 | 插件式换存储引擎 | 单一 B-tree,但和执行解耦 |
| 谁能换 | InnoDB/MyISAM/MyRocks… | 理论上也能换,但 SQLite 只内置 B-tree |
| 真正可换的是 | 存储引擎 | **OS 抽象(VFS)**——更下面的接缝 |
| 设计动因 | C/S 服务端要支持多引擎 | 嵌入式只要一个够好的 B-tree |

> **钉死这件事**:MySQL 把"可替换"放在存储引擎层(handler 接缝),因为它是 C/S 服务端,要支持不同业务用不同引擎;SQLite 把"可替换"放在 OS 抽象层(VFS 接缝),因为它是嵌入式,存储引擎只要一个 B-tree 就够(OS 不固定才是它的真问题)。**两者的"接缝位置"反映了它们各自的核心矛盾——MySQL 的矛盾是"多业务多引擎",SQLite 的矛盾是"多 OS 多设备"。** 这是嵌入式 vs C/S 在分层哲学上的根本差异。

## 六、技巧精解:两个最硬核的分层技巧

这一章讲的是分层架构,有两个技巧最值得单独钉死。它们不是"某一层的实现细节",而是**整个分层架构赖以成立的两个工程支柱**。

### 技巧一:VFS 函数指针表——一张表换掉一整层

八层流水线最底下的 VFS,是怎么做到"整层换掉上面七层无感"的?答案是**函数指针表 + 编译时/运行时选择**。

#### 函数指针表:VFS 就是一张"方法表"

看 `struct sqlite3_vfs` 的定义([struct sqlite3_vfs](../sqlite/src/sqlite.h.in#L1513-L1542))——它里面全是函数指针:`xOpen`/`xDelete`/`xAccess`/`xRead`(在 sqlite3_file 的方法表里)/`xWrite`/`xCurrentTime`… 这是一张典型的"面向对象的方法表"(C 语言没有类,用函数指针表模拟 vtable)。

每个 VFS 实现就是填一张这样的表:

- `unix` VFS(`src/os_unix.c`):`xOpen` 指向 `unixOpen`,`xRead` 指向 `unixRead`(`pread` 包装)…
- `win32` VFS(`src/os_win.c`):`xOpen` 指向 `winOpen`,`xRead` 指向 `winRead`(`ReadFile` 包装)…
- `memdb` VFS(`src/memdb.c`):`xOpen` 指向 `memdbOpen`,`xRead` 指向 `memdbRead`(`memcpy`)…
- `kv` VFS(`src/os_kv.c`):`xOpen` 指向 `kvOpen`,`xRead` 指向 `kvRead`(KV get)…

Pager 拿着一个 `sqlite3_vfs*` 指针,调 `pVfs->xRead(...)`——它根本不知道 xRead 背后是 `pread` 还是 `ReadFile` 还是 `memcpy`。**这就是"一张表换掉一整层"的全部魔法**。

#### 注册机制:多个 VFS 可以共存

SQLite 维护一个全局的 VFS 链表(`sqlite3_vfs.pNext` 串起来),每个 VFS 有个名字(`zName`)。`sqlite3_open_v2` 的第 4 个参数 `zVfs` 就是按名字找:

- `zVfs=NULL`:用默认 VFS(Unix 上是 "unix",Windows 上是 "win32")。
- `zVfs="unix"`:强制用 Unix VFS。
- `zVfs="memdb"`:用内存 VFS(整个库在内存里,常用于测试)。
- `zVfs="mycustom"`:用你自己注册的自定义 VFS(通过 `sqlite3_vfs_register` 注册)。

这就是为什么 SQLite 能"一套代码跑在任何地方"——只要那个平台有人填了一张 `sqlite3_vfs` 表。**VFS 是 SQLite 可移植性的全部根**。

#### 双层指针表:sqlite3_vfs 和 sqlite3_file

VFS 实际上有两层函数指针表:

- **`sqlite3_vfs`**:全局 VFS 的方法(`xOpen` 打开文件、`xDelete` 删文件、`xAccess` 检查文件存在…)。一个 VFS 一张表。
- **`sqlite3_file`**:每个打开的文件的方法(`xRead`/`xWrite`/`xClose`/`xSync`/`xTruncate`…)。`sqlite3_vfs.xOpen` 会创建一个 `sqlite3_file` 对象,这个对象里又有一张方法表(`pMethods`)。这是两层抽象——VFS 管"文件系统级操作"(打开/删除),`sqlite3_file` 管"文件级操作"(读写/sync)。

> **不这样会怎样**:如果 SQLite 把"读文件"硬编码成 `pread`,那它只能跑在 POSIX 上,Windows/嵌入式/KV 后端全用不了——它就当不成"全球部署量最大的数据库"。函数指针表这个技巧,让 SQLite 用一份主干代码,通过换一张表,就能跑在从手机到航电到 KV 后端的任何地方。**这是 C 语言做"可移植抽象"的最经典手法**(承《Linux内核》的 VFS、承《Lua》的 `lua_Load`),SQLite 把它用到了极致。

#### 反面对比:直接调 POSIX 会怎样

假设我们朴素地写——在 Pager 里直接调 `pread`:

```c
/* 朴素(错误)写法 */
rc = pread(fd, pageBuf, pageSize, pageNo*pageSize);
```

这能跑在 Linux 上。但:

- Windows 没有 `pread`,要改成 `ReadFile`——要么 `#ifdef _WIN32`,要么写两套 Pager。
- 测试时想 mock 一个"读失败"的场景,改不了——`pread` 是系统调用,没法 mock。
- 想把整个库放进内存(memdb),做不到——`pread` 是读真文件。
- 想存到 KV 后端(os_kv),做不到。

换一张函数指针表,以上全部解决。**这就是"分层 + 函数指针表"的价值**。

### 技巧二:Vdbe 状态机——一个字段管住 prepare/step/reset/finalize 全部语义

`Vdbe.eVdbeState` 这个一字段(`u8`,只占一个字节),管住了 prepare/step/reset/finalize 四个 API 的全部合法状态转换。这是 SQLite 用**最小状态**覆盖复杂 API 语义的典范。

#### 为什么这个字段这么值钱

`sqlite3_step` 是个对外 API,用户可能在任何时刻调它——刚 prepare 完调、step 跑一半再调、step 返回 DONE 之后还调、reset 之后调、甚至 finalize 之后(虽然这是 use-after-free)。如果没有状态机,这些边界情况会变成一堆 if-else 的噩梦。有了 `eVdbeState`,`sqlite3Step` 的开头就这么几行:

```c
if( p->eVdbeState!=VDBE_RUN_STATE ){
  restart_step:
  if( p->eVdbeState==VDBE_READY_STATE ){
    /* 准备执行 */
    ...
    p->eVdbeState = VDBE_RUN_STATE;
  }else if( ALWAYS(p->eVdbeState==VDBE_HALT_STATE) ){
    /* 自动 reset 再重来 */
    sqlite3_reset((sqlite3_stmt*)p);
    goto restart_step;
  }
}
/* 此刻必定是 RUN 态,执行 */
rc = sqlite3VdbeExec(p);
```

([sqlite3Step 的状态机分派](../sqlite/src/vdbeapi.c#L842-L928))

这几行处理了三种情况:

1. **READY 态调 step**:正常首次执行,做初始化,转 RUN。
2. **HALT 态调 step**:用户忘了 reset,SQLite **自动 reset** 再重新 step(这是 3.7.0 引入的 "automatic-reset" 行为,见上面注释)。
3. **RUN 态调 step**:接着上次 ResultRow 之后跑(这是"一次 step 一行"的实现)。

一个 `eVdbeState` 字段 + 一个 `if-else`,就把"用户可能在任何时刻调 step"这个复杂的边界问题,收敛成了清晰的三种情况。**这就是状态机的力量——用最小的状态空间,覆盖最大的行为空间**。

#### 反面对比:不用状态机会怎样

假设没有 `eVdbeState`,我们要用什么判断"现在能不能 step"?可能要靠一堆散落的 flag:`isPrepared`/`isRunning`/`isHalted`/`pc>0`… 这些 flag 之间会出现不一致的组合(比如 `isPrepared=true && isRunning=true` 到底啥意思?),调试时极难定位。状态机把这些 flag 收敛成**互斥的四个状态**,任何时刻只有一个为真——这是用"枚举状态"代替"布尔标志组合"的经典手法(承《Lua》的 `lua_State` 状态、承《LevelDB》的 DB 状态机)。

> **钉死这件事**:Vdbe 状态机展示了"用最小状态覆盖复杂 API 语义"的工程美学。一个 `u8` 字段,四个互斥状态,管住了 prepared statement 从构造到销毁的全部生命周期。这种"状态最小化"的设计,是 SQLite 代码紧凑、bug 少的一个根——状态越少,能出错的组合越少。

## 七、章末小结

### 回扣主线

本章是**编译篇的总览章**。它把上一章"一条 SELECT 的旅程"那三个本质,落成了源码里**可定位、可点击、可逐行读**的八层结构。记住三个回扣:

1. **二分法的源码落点**。全书主线"编译与执行 vs 存储与事务",在源码里的精确落点是 **Code Generator ↔ VDBE** 这条接缝——1~4 层(Interface/Tokenizer/Parser/Code Generator)是编译前端,VDBE 往下(B-tree/Pager/VFS)是存储与事务。这条接缝,就是全书二分法在架构图上的那条竖线。
2. **prepared statement = Vdbe 对象**。SQLite 的灵魂"编译成 opcode + 虚拟机执行",在源码里就是"`sqlite3_prepare_v2` 产出 `Vdbe`(装满 opcode),`sqlite3_step` 跑 `Vdbe` 里的 opcode"。一个 `Vdbe` 结构体,装着编译期产物(aOp)和执行期状态(pc/aMem/apCsr),用四态状态机(INIT/READY/RUN/HALT)管住全部生命周期。**这个对象是本书后续所有章节的主角**。
3. **八层单向依赖 = 可裁剪 = 可移植**。八层之所以是八层,是因为每条接缝都有清晰的职责边界。这种分层换来了三大回报:**VFS 可换**(跨平台)、**内存/mutex 可换**(可裁剪)、**特性可关**(`SQLITE_OMIT_*`)。这是 SQLite 能塞进任何设备的工程根。

本章服务二分法的**编译与执行**这一面(它讲的是编译前端的分层),但因为八层覆盖了存储侧,也顺带把存储侧的接缝(Pager/VFS)立起来了——后续 P3/P4/P5 会从 B-tree 开始逐层深挖存储侧。

### 五个为什么

1. **为什么 SQLite 分八层,而不是一个大一统引擎?**——职责单一 + 接口稳定 + 可独立替换。每条接缝都有清晰的"为什么这层独立存在",分层换来了可裁剪、可移植、可换实现的三大回报。
2. **为什么八层单向依赖(上层只调下层)?**——反向依赖会破坏可替换性。如果 VFS 反过来调 Pager,换 VFS 就会牵动 Pager,可裁剪性垮掉。单向依赖是"接缝可换"的前提。
3. **为什么 prepared statement 是"编译一次执行多次"?**——prepare 产出 opcode 流(存进 `Vdbe.aOp`),bind 只填参数(`aVar`),reset 只清执行状态(pc/寄存器)不动 opcode。结构编一次,值和执行状态随便换。
4. **为什么 `sqlite3_reset` 比 `sqlite3_finalize` 快几个数量级?**——reset 是"HALT→READY 的状态转换",opcode 一字不动;finalize 是"销毁整个对象",下次要重新跑 Tokenizer/Parser/Code Generator。复用 prepared statement 必须用 reset。
5. **为什么 VFS 是整层可换的接缝?**——VFS 是一张函数指针表(`sqlite3_vfs`),每个 OS 后端填一张表。Pager 只看这张表,不感知背后是 POSIX/Win32/内存/KV。换 OS 只换 VFS,上面七层一行不改。对照 MySQL 的"可换"放在 handler(存储引擎)层——SQLite 的核心矛盾是"多 OS",所以可换放在 VFS。
