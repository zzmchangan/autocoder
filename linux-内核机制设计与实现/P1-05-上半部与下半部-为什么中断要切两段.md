# 第五章 · 上半部与下半部:为什么中断要切两段

> 篇:P1 中断与软中断
> 主线呼应:上一章(P1-04)我们钉死了一件事——**中断上下文不是进程**,它没有可调度的 `task_struct`,所以拿不到能挂起的"身体",一旦睡眠就再也醒不过来,系统当场死锁。这一章顺着这个硬约束推下去:既然中断上下文这么"窄",那中断处理函数就**只能干最紧急的事**,剩下那些可推迟的重活——把网卡包推进协议栈、把磁盘 IO 后续走完、唤醒等待的进程——必须挪到一个约束更松的上下文里去做。这就是 Linux 把中断处理**切成两段**的根本理由:**上半部(hardirq)快收快放,下半部(deferred work)延后做重活**。下半部不是一种机制,而是三种(softirq/tasklet/workqueue),它们的差别全在"约束放松到什么程度"。

## 核心问题

**既然事件已经把 CPU 拉进了内核,为什么中断处理还要切两段?为什么不能在中断里一口气把包推给协议栈、把磁盘 IO 处理完?三种下半部(softirq/tasklet/workqueue)到底差在哪、各自该用在哪?**

读完本章你会明白:

1. 上半部/下半部切分的**本质动机**:不是"为了好看",而是**性能与功能的切分**——上半部抢着 CPU、可能屏蔽其他中断、不可睡眠,只能干最紧急的;下半部放开约束,才能做重活。
2. 为什么"全在 hardirq 里干完"会**卡死系统**:长时间屏蔽中断、其他事件饿死、无法调阻塞 API。
3. 三种下半部的**横向对比**:softirq(轻量、per-CPU、不可睡眠)、tasklet(softirq 的包装,BH 时代遗物,6.x 正在废弃)、workqueue(有进程上下文、可睡眠)。
4. `request_threaded_irq` 给出的**第四条路**:线程化中断(handler 在内核线程里跑,可睡眠)——为什么 RT-Linux 离不开它。
5. ★ 对照:上半部/下半部切分 ↔ Tokio 的 `mio` 只取 epoll 就绪事件 + 异步 task 接力做后续,两层切分同构。

> **逃生阀**:如果你已经知道"中断里不能 `malloc`/`mutex`、要靠 softirq 接力",可以直接跳到 5.4(三种下半部对比表)和 5.6(技巧精解)。但 5.2~5.3 的"为什么不能全在 hardirq 做"的推导,是后面 softirq/workqueue 章的地基,建议读。

## 5.1 一句话点破

> **中断处理切两段,不是设计上的洁癖,是被两个硬约束逼出来的:中断上下文不能睡眠(上一章),中断处理时往往还会本地屏蔽其他中断。所以上半部只做"非它不可"的事(应答硬件、拷最少的数据、置一个'待会再处理'的标记),把所有可推迟的重活挪到下半部——下半部要么在中断退出后立刻接力跑(softirq,仍不能睡眠但中断已开),要么交给一个真正的内核线程去睡到能跑为止(workqueue)。一句话:切分是为了把中断对系统的'打扰'压到最小。**

这是结论,不是理由。本章倒过来拆:先看"全在 hardirq 里干完"会撞什么墙,再看切分后两段各自承担什么,然后把三种下半部摆上桌横向比,最后钻进源码看切分的边界到底画在哪。

---

## 5.2 不这样会怎样:全在 hardirq 里干完会撞什么墙

先做个思想实验。假设我们没有下半部,网卡收到一个包,中断处理函数(hardirq)一口气把所有事做完:

1. 从网卡的 ring buffer 把包拷进内核内存。
2. 调 `netif_receive_skb` 把包推进协议栈,过 `ip_rcv`/`tcp_v4_rcv`,做路由查表、连接跟踪、socket 查找。
3. 如果是 `tcp` 数据,还要唤醒等待在这个 socket 上的进程(调 `wake_up_interruptible`),可能触发一次调度。

乍一看没毛病——逻辑完整、一次跑完。但放进真实系统里,这堵墙是这样的:

**墙一:长时间屏蔽其他中断,高优先级事件被饿死。** 大多数驱动在 hardirq 里会本地关中断(`local_irq_disable`,或中断控制器自动 mask 这条 IRQ 线),避免同一个 CPU 上中断嵌套把栈撑爆。如果你在 hardirq 里跑 `tcp` 协议栈——查路由表要遍历 trie、连接跟踪要查 hash、socket 查找要拿自旋锁——这一跑就是几万条指令。这段时间内,**这个 CPU 上所有其他中断都被屏蔽**:键盘敲击没人响应、定时器 tick 被延迟、别的网卡包堆积。哪怕你的 hardirq 不主动关中断,CPU 在响应一条中断时硬件也会自动屏蔽同级和更低级中断( x86 的中断优先级机制),所以"hardirq 跑得越久,其他中断越惨"是硬件层面的硬约束。

**墙二:无法调任何阻塞 API,功能被阉割。** hardirq 是中断上下文,`in_interrupt()` 为真。所有会睡眠的 API——`mutex_lock`(会睡眠)、`kmalloc(GFP_KERNEL)`(可能睡眠换页)、`wait_event`(必睡眠)、磁盘块的同步读——开头都有 `might_sleep()` 断言,在调试配置下直接 `BUG_ON(in_interrupt())`。但很多下半部工作天然要阻塞:磁盘 IO 完成后要走文件系统层,文件系统要拿 `inode` 锁(可能是 `mutex`);USB 设备的中断后续要和 USB 栈通信,USB 栈会睡眠等传输完成。全在 hardirq 做,这些路**根本走不通**。

**墙三:中断里拿自旋锁要 `spin_lock_irqsave`,锁竞争被放大。** 协议栈里那些自旋锁,在进程上下文里拿是 `spin_lock`,在 hardirq 里拿必须 `spin_lock_irqsave`(关中断再拿,防止中断里自旋锁嵌套死锁)。`spin_lock_irqsave` 比普通 `spin_lock` 贵得多(要操作中断标志),而且持有期间关中断,把"锁临界区"和"中断屏蔽区"绑死。如果一个 hardirq 持有协议栈锁 1ms,这 1ms 内该 CPU 所有中断全断。

> **不这样会怎样**:把所有中断处理塞进 hardirq,系统在高负载下会**实时性崩溃**——网卡中断跑 `tcp` 协议栈花掉 2ms,这 2ms 里定时器 tick 漏了几次、键盘鼠标的点击堆积、其他网卡包因为 ring buffer 满被丢弃;想调个阻塞 API 直接 `panic`(调试态)或死锁(生产态)。老内核(2.0 之前)确实在 hardirq 里干了不少重活,后来被一次次"挪到下半部",就是因为这堵墙。

## 5.3 所以上半部/下半部这样切

把中断处理切两段,就是把这堵墙拆掉:

```
 一次网卡 RX 中断,切两段(简化):

   ┌──────────────────────────────────────────────────────────┐
   │ 上半部 hardirq(在硬中断上下文,关中断/抢 CPU/不可睡眠)        │
   │   ├─ 应答硬件:给网卡发 EOI / mask 这条 IRQ 线               │
   │   ├─ 拷最少的数据:把包从网卡 ring buffer 拷到内核 skb        │
   │   ├─ 置标记:raise_softirq(NET_RX_SOFTIRQ)                  │
   │   └─ 立刻返回(目标:微秒级)                                  │
   └──────────────────────┬───────────────────────────────────┘
                          │ 中断退出(irq_exit)→ 开中断
                          ▼
   ┌──────────────────────────────────────────────────────────┐
   │ 下半部 softirq(中断退出后接力跑,中断已开/不可睡眠)            │
   │   └─ net_rx_action:把 skb 推进协议栈(ip_rcv → tcp_v4_rcv)  │
   │      唤醒等待的进程,可能触发调度                              │
   └──────────────────────────────────────────────────────────┘
```

切分的判断标准极朴素:**这件事是不是"非 hardirq 不可"?**

- **必须 hardirq 做的**:应答硬件(不给网卡 EOI 它就不发下一个中断)、从硬件 FIFO/ring buffer 把数据拷走(硬件缓冲区小,不拷会丢)、置"待处理"标记。这些都有硬件时间约束——你不应答硬件就卡着,你不拷数据硬件缓冲区就溢出。
- **可以推迟的**:协议栈处理、文件系统后续、设备状态机推进、唤醒进程、触发调度。这些**只关系软件状态**,早一毫秒晚一毫秒做不影响正确性。

> **所以这样设计**:上半部只做"非它不可"的最少的事,把所有软件层面的重活标记成"待处理",然后立刻退出中断、交还 CPU(要么回被中断的进程,要么进下半部)。下半部在中断退出后接力跑,**此时中断已经打开**(别的中断能进来)、**抢占已经允许**(时间片到了能切换)、**CPU 已经不在硬件中断栈上**(栈空间充足)。约束放松了,能做的事就多了。

但"下半部"不是一种机制,而是三种,差别就在**约束放松到什么程度**:

- **softirq**:中断退出后立刻在中断上下文里接力跑,中断已开但**仍不可睡眠**。最快、延迟最低,但能干的事有限(不能调阻塞 API)。
- **tasklet**:softirq 的薄包装,提供"同一个 tasklet 不会在多核上并发执行"的语义。BH(bottom-half)时代的遗物,6.x 正在废弃,新代码不该用。
- **workqueue**:把延迟工作丢给一个真正的内核线程(worker),worker 在**进程上下文**里跑,**可以睡眠、可以调任何阻塞 API**。最灵活,但比 softirq 慢一点(要走一次调度)。

还有一种和切分同构、但**不属于"下半部三选一"**的机制:**线程化中断(threaded IRQ)**。它把整个中断处理(连上半部)都丢到一个内核线程里跑,用 `request_threaded_irq` 注册。我们放最后讲。

> **钉死这件事**:切分不是为了"分层好看",是为了**把中断对系统的打扰压到最小**。上半部抢 CPU 的时间压到微秒级,系统实时性才有保障;下半部放开约束做重活,功能才完整。这是性能(上半部要快)与功能(下半部要做重活)的切分。

## 5.4 三种下半部横向对比

把三种下半部摆上桌,从"能不能睡眠""并发度""延迟""典型用途"四个维度比:

| 维度 | softirq | tasklet | workqueue |
|------|---------|---------|-----------|
| **执行上下文** | 软中断上下文(`in_softirq()` 真) | 软中断上下文(被 softirq 包装) | 进程上下文(worker 内核线程) |
| **能否睡眠** | **否**(`might_sleep()` 会崩) | **否**(同 softirq) | **能**(随便睡,`mutex`/`GFP_KERNEL`/磁盘 IO 都行) |
| **能否被抢占** | 不能(软中断上下文不可抢占) | 不能 | 能(它是普通进程) |
| **并发度** | per-CPU,同一种 softirq 可在多核**同时**跑 | 同一个 tasklet 同时只在一个 CPU 跑(自串行化) | 由 worker_pool 管控,可配置并发度 |
| **延迟** | 最低(`irq_exit` 立刻接力) | 低(同 softirq) | 较高(要走一次调度,微秒~毫秒级) |
| **注册/调度 API** | `open_softirq`/`raise_softirq`(内核内部,驱动基本不碰) | `tasklet_schedule`/`DECLARE_TASKLET` | `INIT_WORK`/`queue_work`/`schedule_work` |
| **典型用途** | 网络收发(`NET_RX/TX_SOFTIRQ`)、定时器、`RCU`、块设备 | 历史遗留(网卡老驱动、声卡) | 磁盘 IO 后续、文件系统、电源管理、`USB` 栈 |
| **6.9 状态** | 中坚,活跃使用 | **正在废弃**,新代码禁止用 | 中坚,活跃使用 |

这张表是本章的"答案页"。几条关键结论:

**结论一:softirq 是"内核内部用"的下半部,普通驱动基本碰不到。** softirq 的种类是**编译期写死的**——`include/linux/interrupt.h` 里那个 `enum`(`HI_SOFTIRQ`/`TIMER_SOFTIRQ`/`NET_RX_SOFTIRQ`/...,见 [interrupt.h#L551-L565](../linux/include/linux/interrupt.h#L551-L565)),一共 10 种(`NR_SOFTIRQS`),普通驱动**不能新增** softirq(头文件里有句警告:"PLEASE, avoid to allocate new softirqs")。你要用 softirq 提供的能力,走的是它包装出来的 tasklet(历史)或者直接调 `schedule_work`(现代)。softirq 留给子系统级的高频路径(网络、定时器、`RCU`)。

**结论二:tasklet 是"被 softirq 包装的、自串行化的"轻量下半部,但它在死。** tasklet 用 `TASKLET_SOFTIRQ`/`HI_SOFTIRQ` 两个 softirq 实现:`tasklet_schedule` 把 tasklet 挂到 per-CPU 链表、raise 对应 softirq;softirq 处理函数遍历链表逐个跑 tasklet。它的卖点——"同一个 tasklet 不会在多核并发跑"——正是它被废弃的导火索(下面 5.5 详讲)。

**结论三:workqueue 是"需要睡眠时的唯一选择"。** 任何下半部工作,只要可能调阻塞 API(拿 `mutex`、做磁盘 IO、`kmalloc(GFP_KERNEL)`),就只能用 workqueue。workqueue 的 worker 是真正的内核线程(`kworker/uX:N`),在进程上下文里跑你的工作函数,所有进程上下文能干的事它都能干。CMWQ(Concurrency Managed Workqueue)还能按需唤醒/休眠 worker、管制并发度,既不爆炸又能并发(下一章 P1-07 详讲)。

## 5.5 tasklet 为什么在死

tasklet 是 BH(Bottom Half,2.0 时代的下半部机制,全局只允许一个 BH 跑)被 softirq 取代后的过渡产物。它的定位是"比 softirq 易用、比 BH 并发度好":多个不同 tasklet 可以在多核同时跑(不像 BH 全局串行),但**同一个 tasklet 同时只在一个 CPU 上跑**(自串行化,免得用户自己加锁)。

听上去很美,但它在 6.x 正被逐步移除,理由有三:

**理由一:per-CPU 调度 + 全局串行化的组合,带来不必要的复杂和竞争。** tasklet 的实现是:每个 CPU 一个 `tasklet_vec` 链表,`tasklet_schedule` 把 tasklet 挂到**当前 CPU** 的链表上并 raise `TASKLET_SOFTIRQ`;softirq 处理函数(`tasklet_action_common`,见 [softirq.c#L763](../linux/kernel/softirq.c#L763))把链表摘下来逐个跑,用 `tasklet_trylock`(`test_and_set_bit(TASKLET_STATE_RUN)`,见 [interrupt.h#L694-L697](../linux/include/linux/interrupt.h#L694-L697))保证同一个 tasklet 不会被两个 CPU 同时执行。这意味着:一个 tasklet 被调度到 CPU0,但 CPU0 上 softirq 处理慢,CPU1 想"接力跑"也跑不了——它得 `tasklet_trylock` 失败、把 tasklet 重新挂回链表等下次。这种"调度是 per-CPU、执行是全局串行"的组合,引入了 `TASKLET_STATE_SCHED`/`TASKLET_STATE_RUN` 两套状态位和重新调度的活锁风险。源码头文件里的注释把话说得很直白:

> `/* Tasklets --- multithreaded analogue of BHs. This API is deprecated. Please consider using threaded IRQs instead */`
> 见 [interrupt.h#L622-L643](../linux/include/linux/interrupt.h#L622-L643)

**理由二:线程化中断(`request_threaded_irq`)是更好的替代品。** tasklet 当年解决的是"我要一段延迟执行的、自串行化的代码",但 `threaded IRQ` + workqueue 已经能更干净地满足这个需求:线程化中断天然在一个内核线程里跑(自串行化),还能睡眠;workqueue 也能配成 ordered(`alloc_ordered_workqueue`,串行执行)。tasklet 能做的,这两个都能做,还做得更好(可睡眠、并发度可配)。

**理由三:RT-Linux(`PREEMPT_RT`)和 tasklet 不兼容。** RT-Linux 要把尽可能多的中断处理线程化(让中断处理可被抢占),但 tasklet 死绑在 softirq 上、跑在软中断上下文不可抢占,成了 RT 化的障碍。所以 RT 树早就动手改 tasklet,主线也在跟进。

> **钉死这件事**:本书讲 tasklet,是为了让你看懂老代码(很多网卡、声卡、`SCSI` 驱动还在用),但**新代码不要用 tasklet**。要么用 workqueue(可睡眠、推荐),要么用线程化中断。tasklet 在 6.x 已标记 deprecated,迟早会移除。所以下面我们把笔墨主要花在 softirq(下一章 P1-06)和 workqueue(P1-07)上,tasklet 讲清它是 softirq 的薄包装、为什么被废弃即可。

## 5.6 源码:切分的边界画在哪

讲清楚动机和对比后,我们看切分在源码里长什么样。挑两个最关键的点:**① 上半部怎么"应答硬件 + 拷数据 + raise softirq";② 中断退出怎么"接力跑下半部"**。

### 5.6.1 中断处理函数的返回值:`IRQ_HANDLED` vs `IRQ_WAKE_THREAD`

先看上半部(handler)和下半部的接口契约。一个注册的中断处理函数长这样(`irq_handler_t` 是函数指针类型,见 [interrupt.h#L103](../linux/include/linux/interrupt.h#L103)):

```c
typedef irqreturn_t (*irq_handler_t)(int, void *);
```

它返回一个 `irqreturn_t`,这是一个枚举,定义极简([irqreturn.h#L11-L18](../linux/include/linux/irqreturn.h#L11-L18)):

```c
enum irqreturn {
    IRQ_NONE        = (0 << 0),  // 这个中断不是我的设备产生的
    IRQ_HANDLED     = (1 << 0),  // 我处理完了,就到此为止
    IRQ_WAKE_THREAD = (1 << 1),  // 处理完了,请唤醒我的线程化下半部
};
```

三个返回值的含义正是切分的契约:

- **`IRQ_NONE`**:这个 handler 被调用,但它判断出中断不是自己的设备产生的(共享 IRQ 线上常见),返回 `IRQ_NONE`,让 IRQ 核心知道"这次没人认领"。
- **`IRQ_HANDLED`**:最常见,handler 在 hardirq 上下文里把该做的都做完了(应答硬件 + 拷数据 + 如果需要再 `raise_softirq`/`schedule_work` 把后续工作排出去),告诉核心"处理完了,不必唤醒线程"。
- **`IRQ_WAKE_THREAD`**:线程化中断的专用返回值。handler 只做了"判断是不是我的设备 + 屏蔽这条 IRQ 线"这点最紧急的事,然后返回 `IRQ_WAKE_THREAD`,核心会唤醒驱动注册时提供的 `thread_fn` 内核线程,把真正的处理工作交给它(在进程上下文里跑,可睡眠)。

IRQ 核心在 `__handle_irq_event_percpu` 里处理这三个返回值([handle.c#L139-L187](../linux/kernel/irq/handle.c#L139-L187)):

```c
irqreturn_t __handle_irq_event_percpu(struct irq_desc *desc)
{
    irqreturn_t retval = IRQ_NONE;
    unsigned int irq = desc->irq_data.irq;
    struct irqaction *action;

    for_each_action_of_desc(desc, action) {
        irqreturn_t res;
        /* ... */
        res = action->handler(irq, action->dev_id);   // 调用驱动的 hardirq handler

        if (WARN_ONCE(!irqs_disabled(), ...))          // handler 退出时中断必须是关的
            local_irq_disable();

        switch (res) {
        case IRQ_WAKE_THREAD:                          // handler 要求唤醒线程化下半部
            if (unlikely(!action->thread_fn)) {
                warn_no_thread(irq, action);
                break;
            }
            __irq_wake_thread(desc, action);           // 唤醒 thread_fn 对应的内核线程
            break;
        default:
            break;
        }
        retval |= res;
    }
    return retval;
}
```

注意那个 `WARN_ONCE(!irqs_disabled(), ...)`([handle.c#L161](../linux/kernel/irq/handle.c#L161))——它是一条**正确性契约**:handler 退出时,本地中断必须是关闭的。中断核心代码依赖这一点,因为 `__handle_irq_event_percpu` 是在关中断的临界区里被调用的,handler 不能偷偷把中断打开(打开了就可能嵌套进来同一个 IRQ,破坏 `IRQS_INPROGRESS` 状态)。这是"上半部要快"的另一面——它跑在中断屏蔽区里,跑完交还,不能破坏这个区。

`for_each_action_of_desc` 是**共享中断**的支持:多个设备共享一条 IRQ 线时,它们的 handler 挂在 `irq_desc` 的 action 链表上,核心挨个调,谁的设备返回了 `IRQ_HANDLED`/`IRQ_WAKE_THREAD` 就算这次中断被认领了。这也是 `dev_id` 参数存在的理由——共享时核心要把 `dev_id` 传回 handler,让它判断"是不是我的设备"。

### 5.6.2 上半部的"应答 + raise"模式:以网卡中断为例

一个典型的网卡 RX 中断 handler(简化示意,非源码原文)长这样:

```c
/* 简化示意,非真实驱动源码 */
static irqreturn_t my_nic_irq_handler(int irq, void *dev_id)
{
    struct nic_dev *nic = dev_id;
    int budget = 64;  /* hardirq 里只拷有限个包,剩下的丢给 NAPI 轮询 */

    /* 1. 应答硬件:mask 这条 IRQ 线,告诉网卡"先别给我发中断了" */
    nic->irq_mask(nic);

    /* 2. 拷最少的数据:从 ring buffer 拿若干包到内核 skb,推进 NAPI 轮询表 */
    napi_schedule(&nic->napi);   /* 内部 raise_softirq(NET_RX_SOFTIRQ) */

    /* 3. 立刻返回,剩下的交给下半部 */
    return IRQ_HANDLED;
}
```

三步:**应答硬件(mask 这条线,防止它在 softirq 跑完前再来捣乱)** → **拷最少的数据 / 把设备挂到 NAPI 轮询表** → **raise 一个 softirq 标记"待会处理",然后 `IRQ_HANDLED` 返回**。

> 这里有个更精巧的设计:**NAPI(New API)**。网卡中断里**不拷包**,只做"禁用这个网卡的 RX 中断 + 把网卡挂到 per-CPU 的 NAPI 轮询表 + raise `NET_RX_SOFTIRQ`"。真正的拷包发生在下半部 `net_rx_action`(softirq)里,而且**用轮询而非中断**——一次轮询最多拿 64 个包,拿完如果还有就再轮询,直到队列空了才重新打开这个网卡的 RX 中断。这样在高 pps(包/秒)场景下,中断不会风暴(每来一个包就一次中断),而是"中断一次,然后靠 softirq 轮询批量吃光"。NAPI 是"上半部尽量轻、下半部尽量批量"的极致体现,第 6 章(P1-06)讲 softirq 会再回扣。

### 5.6.3 中断退出怎么接力下半部:`irq_exit` → `invoke_softirq`

上半部返回后,CPU 怎么从中断上下文过渡到下半部?关键路径是 `__irq_exit_rcu`(6.9 的中断退出核心,[softirq.c#L627-L640](../linux/kernel/softirq.c#L627-L640)):

```c
static inline void __irq_exit_rcu(void)
{
#ifndef __ARCH_IRQ_EXIT_IRQS_DISABLED
    local_irq_disable();
#else
    lockdep_assert_irqs_disabled();
#endif
    account_hardirq_exit(current);
    preempt_count_sub(HARDIRQ_OFFSET);                    // 退出 hardirq 上下文
    if (!in_interrupt() && local_softirq_pending())       // 不在中断里 + 有 softirq pending
        invoke_softirq();                                  // 就地跑 softirq(下半部接力)

    tick_irq_exit();
}
```

这段代码短小但极其关键,逐行拆:

1. `preempt_count_sub(HARDIRQ_OFFSET)`:[softirq.c#L635](../linux/kernel/softirq.c#L635)。把 `preempt_count` 的 hardirq 计数减一(进 hardirq 时 `irq_enter_rcu` 加过 `HARDIRQ_OFFSET`)。这一减,`in_interrupt()` 的判断就变了——如果之前没有嵌套中断,现在 hardirq 退出后,`preempt_count` 的 hardirq 段归零。
2. `if (!in_interrupt() && local_softirq_pending())`:[softirq.c#L636](../linux/kernel/softirq.c#L636)。**两个条件缺一不可**:
   - `!in_interrupt()`:**只有"不在任何中断上下文"才就地跑 softirq**。如果你是嵌套中断(比如 hardirq 里又来了个 NMI,或者中断退出时发现还在 softirq 里),那就不就地跑——避免 softirq 嵌套,改交给 ksoftirqd 内核线程稍后跑。
   - `local_softirq_pending()`:**本 CPU 的 softirq pending 位图非零**,即有人 raise 过 softirq(`raise_softirq` 会置位)。
3. `invoke_softirq()`:[softirq.c#L419-L440](../linux/kernel/softirq.c#L419-L440)。**这是接力的核心**。看它的实现:

```c
static inline void invoke_softirq(void)
{
    if (!force_irqthreads() || !__this_cpu_read(ksoftirqd)) {
#ifdef CONFIG_HAVE_IRQ_EXIT_ON_IRQ_STACK
        __do_softirq();                  // 在硬中断栈上就地跑 softirq
#else
        do_softirq_own_stack();          // 切到独立栈再跑,防栈溢出
#endif
    } else {
        wakeup_softirqd();               // 强制线程化:唤醒 ksoftirqd
    }
}
```

`invoke_softirq` 给出了下半部的两条路:

- **默认路:就地跑**(`__do_softirq`)。中断退出时如果不在强制线程化配置下,直接在当前栈(或独立 softirq 栈)上跑 softirq——延迟最低。`__do_softirq` 在 6.9 已瘦身成 `handle_softirqs(false)` 的薄包装([softirq.c#L586-L589](../linux/kernel/softirq.c#L586-L589)):
  ```c
  asmlinkage __visible void __softirq_entry __do_softirq(void)
  {
      handle_softirqs(false);
  }
  ```
  主体逻辑全在 `handle_softirqs`([softirq.c#L511-L584](../linux/kernel/softirq.c#L511-L584)),下一章 P1-06 会逐行讲。

- **强制线程化路:唤醒 ksoftirqd**(`wakeup_softirqd`)。`force_irqthreads()` 为真时(`PREEMPT_RT` 配置或启动参数 `threadirqs`),**所有 softirq 都不就地跑,而是丢给 ksoftirqd 内核线程**。这是 RT-Linux 的刚需——softirq 就地跑不可抢占,RT 要把一切都线程化才能保证实时性。

这样,中断退出就完成了从"上半部 hardirq"到"下半部 softirq"的接力。注意此时**中断已经打开**(中断退出时硬件会恢复 EFLAGS 的 IF 标志,`__irq_exit_rcu` 里 `local_irq_disable` 只是这一段的保护,退出 `irq_exit` 后会恢复),所以 softirq 里可以被打断(被新的硬中断),但 softirq 本身仍不可睡眠(它在软中断上下文)。

### 5.6.4 raise_softirq:上半部怎么"挂号"

最后看上半部是怎么"置标记"的——`raise_softirq`([softirq.c#L670-L701](../linux/kernel/softirq.c#L670-L701)):

```c
inline void raise_softirq_irqoff(unsigned int nr)
{
    __raise_softirq_irqoff(nr);

    /* 如果当前不在任何中断里,顺便唤醒 ksoftirqd 兜底 */
    if (!in_interrupt() && should_wake_ksoftirqd())
        wakeup_softirqd();
}

void raise_softirq(unsigned int nr)
{
    unsigned long flags;
    local_irq_save(flags);            // 关中断再操作位图
    raise_softirq_irqoff(nr);
    local_irq_restore(flags);
}

void __raise_softirq_irqoff(unsigned int nr)
{
    lockdep_assert_irqs_disabled();
    trace_softirq_raise(nr);
    or_softirq_pending(1UL << nr);    // 置本 CPU 位图的第 nr 位
}
```

**核心就一句**:`or_softirq_pending(1UL << nr)`——把本 CPU 的 `__softirq_pending` 位图的第 `nr` 位置 1。这个位图是 per-CPU 的 32 位整数(见上一章 P1-04 的技巧精解),每一位对应一种 softirq(`NET_RX_SOFTIRQ`/`TIMER_SOFTIRQ`/...)。

注意三个细节:

1. **操作位图必须关中断**(`raise_softirq` 用 `local_irq_save` 包起来,`raise_softirq_irqoff` 要求调用者已关中断且有 `lockdep_assert_irqs_disabled`)。因为 `or_softirq_pending` 是"读-改-写"操作,不能被同一个 CPU 上的硬中断打断(中断里的中断会读到旧值再写回,丢更新)。
2. **`raise_softirq_irqoff` 里那个 `if (!in_interrupt())`**:[softirq.c#L683](../linux/kernel/softirq.c#L683)。如果你是在**进程上下文**里 raise softirq(比如某个内核线程主动要求"待会跑一下 RCU 回收"),那就不仅要置位,还要唤醒 ksoftirqd——因为进程上下文不会经过 `irq_exit` 的接力路径,不唤醒 ksoftirqd 的话这个 softirq 永远没人跑。如果你是在**中断上下文**里 raise(hardirq 里 `raise_softirq(NET_RX_SOFTIRQ)` 是最常见的场景),那就只置位不唤醒——因为马上要走的 `irq_exit` → `invoke_softirq` 会就地跑掉。
3. **`or` 而非 `set`**:`or_softirq_pending` 用的是按位或,不是赋值。这意味着**同一种 softirq 被 raise 多次,位图只置 1 一次**——softirq 处理函数一次把所有 pending 都处理掉,不保证"raise 几次就跑几次"。这是 softirq 的语义:它是"事件类型"级别的延迟处理,不是"事件计数"级别的(实时信号才有计数,见第 17 章)。

> **钉死这件事**:上半部和下半部的接口,就是**一个 32 位 per-CPU 位图 + 一组 `softirq_vec[]` 处理函数**。上半部用 `raise_softirq` 置位(挂号),`irq_exit` 退出时检查位图有位就 `invoke_softirq` 接力跑下半部。这套接口短小、无锁(per-CPU)、O(1),是 softirq 全部性能优势的根源。下一章 P1-06 会把 `handle_softirqs` 的主循环、`MAX_SOFTIRQ_RESTART` 防饿死、ksoftirqd 兜底全讲透。

## 5.7 线程化中断:第四条路

讲完三种下半部,还有一种和切分同构、但**不属于"下半部三选一"**的机制:**线程化中断**。它由 `request_threaded_irq` 注册([manage.c#L2147](../linux/kernel/irq/manage.c#L2147)),签名是:

```c
int request_threaded_irq(unsigned int irq, irq_handler_t handler,
                         irq_handler_t thread_fn, unsigned long irqflags,
                         const char *devname, void *dev_id);
```

它把中断处理拆成**两个函数**:

- `handler`:仍在 hardirq 上下文跑。它只做"判断是不是我的设备 + 必要的应答",然后返回 `IRQ_WAKE_THREAD`(`__handle_irq_event_percpu` 见到这个返回值就调 `__irq_wake_thread` 唤醒线程,[handle.c#L166-L177](../linux/kernel/irq/handle.c#L166-L177))。
- `thread_fn`:在一个专用的内核线程里跑。它做"剩下的所有事",**在进程上下文里,可以睡眠、可以调阻塞 API**。

和传统"hardirq + softirq/workqueue"切分比,线程化中断的特点是:**下半部不是 softirq(不可睡眠)也不是独立 work(要走 workqueue 调度),而是这个 IRQ 专用的内核线程**。好处有三:

1. **天然串行化**:同一个 IRQ 的 `thread_fn` 不会并发(一个线程就一个执行流),驱动不用自己加锁。
2. **可睡眠**:`thread_fn` 能调任何阻塞 API,功能完整。
3. **可被抢占/可设优先级**:线程化后,中断处理可以参与调度优先级管理,RT-Linux 借此实现确定性延迟。

它的代价是**延迟更高**(`thread_fn` 要等调度,不像 softirq 在 `irq_exit` 立刻接力)。所以默认配置下,大多数驱动仍用 `request_irq`(它就是 `thread_fn = NULL` 的 `request_threaded_irq`,见 [interrupt.h#L167-L172](../linux/include/linux/interrupt.h#L167-L172))走 hardirq + softirq;只有**RT-Linux** 或**中断工作天然需要睡眠**的驱动(USB、某些 SPI/I2C 设备)才用线程化中断。

`request_threaded_irq` 的注释把"为什么这么设计"写得很清楚([manage.c#L2126-L2132](../linux/kernel/irq/manage.c#L2126-L2132)):

> `@handler is still called in hard interrupt context and has to check whether the interrupt originates from the device. If yes it needs to disable the interrupt on the device and return IRQ_WAKE_THREAD which will wake up the handler thread and run @thread_fn. This split handler design is necessary to support shared interrupts.`

注意那句"This split handler design is necessary to support shared interrupts"——共享中断线上,handler 必须先在 hardirq 里判断"是不是我的设备"(否则 thread_fn 会被无关中断频繁唤醒),所以 handler 这一层省不掉。这是切分在"线程化中断"模式下的另一种画法。

## 5.8 技巧精解:上半部/下半部切分的"在哪画线"

讲清动机和实现后,我们钻一个最容易被忽略、却最关键的技巧:**切分的线到底画在哪?** 这不是一个"按规定画"的问题,而是**驱动作者每次写 handler 都要做的工程权衡**。

### 技巧:用"是否阻塞""是否硬件约束"两条尺子切分

朴素地写,驱动作者会把"能做完的都在 hardirq 里做完",理由是"省事,不用写 softirq 注册代码"。这堵墙前面讲过(5.2):hardirq 跑久了屏蔽中断、无法调阻塞 API。

更糟的反面是**切分错位**——把"必须 hardirq 做"的事推迟了,或把"可以推迟"的事留在 hardirq 了:

```c
/* 反面一(错位):把"应答硬件"推迟到 softirq,hardirq 里只拷数据 */
static irqreturn_t bad_handler_a(int irq, void *dev_id)
{
    struct nic_dev *nic = dev_id;
    copy_packets_to_skb(nic);          /* 拷数据 */
    napi_schedule_irqoff(&nic->napi);   /* 置 softirq 位,把"应答硬件"丢给 softirq */
    return IRQ_HANDLED;
}
/* 问题:应答硬件(EOI/mask)被推迟到 softirq,这期间网卡以为中断没被处理,
   可能疯狂重发中断(电平触发 IRQ),或者 ring buffer 溢出(新包没地方放)。
   硬件约束(应答)必须在 hardirq 里立刻做。 */

/* 反面二(错位):把"协议栈处理"留在 hardirq,不走 softirq */
static irqreturn_t bad_handler_b(int irq, void *dev_id)
{
    struct nic_dev *nic = dev_id;
    struct sk_buff *skb = copy_one_packet(nic);
    nic->irq_eoi(nic);                 /* 应答硬件 */
    netif_receive_skb(skb);            /* ←在 hardirq 里跑协议栈! */
    return IRQ_HANDLED;
}
/* 问题:netif_receive_skb 走 ip_rcv/tcp_v4_rcv/连接跟踪/socket 查找,
   几万条指令,hardirq 里跑这么久屏蔽所有中断;还不能调阻塞 API,
   如果协议栈路径上要 mutex 直接 BUG_ON。 */
```

Linux 给出的正确切分尺子有两条:

**尺子一:硬件约束的事,必须在 hardirq 做。** "硬件约束"指那些**硬件有时间窗口**的操作:应答中断(不给 EOI 硬件就不消中断)、mask IRQ 线(电平触发的 IRQ 不 mask 就会一直触发)、从硬件 FIFO/ring buffer 拷数据(硬件缓冲区小,不拷会溢出)。这些事**晚一毫秒做都会出问题**(硬件状态机卡死、数据丢失),所以必须 hardirq。

**尺子二:会阻塞或耗时长的软件工作,必须推迟。** "软件工作"指那些**只关系内核数据结构**的操作:协议栈处理、文件系统层、设备状态机推进、唤醒进程、触发调度。这些事晚一毫秒做完全没问题(数据已经在内核内存里了,跑一下软件逻辑而已),但它们要么会阻塞(`mutex`/磁盘 IO),要么耗时太长(几万条指令),所以必须推迟到 softirq 或 workqueue。

两条尺子合起来:**"非 hardirq 不可"的事放 hardirq,"可推迟"的事放下半部**。这就是 `request_threaded_irq` 注释里那句"split handler design"的本质——`handler` 只做尺子一的事,`thread_fn`(或 softirq、或 workqueue)做尺子二的事。

> **反面对比**:如果驱动作者不按这两条尺子切,要么硬件状态机出问题(应答推迟),要么系统实时性崩(协议栈留 hardirq)。老内核的很多驱动都经历过这种"切分重构"——最典型就是网络栈从"hardirq 里跑 BH"到"hardirq + NET_RX_SOFTIRQ"再到"NAPI 轮询"的演进。每一次重构,都是把更多工作从 hardirq 挪到下半部,把 hardirq 压到更短。

### 这套切分为什么 sound

切分的正确性保证(sound)来自三件事:

1. **数据所有权不冲突**:hardirq 拷数据进内核内存后,数据已经"脱离硬件",softirq 处理它时不需要再碰硬件寄存器——所以 hardirq 退出(开中断)不影响 softirq 拿到完整数据。如果数据还在硬件 FIFO 里就退出 hardirq,数据可能被新数据冲掉,这就违反了"硬件约束的事必须 hardirq 做"。
2. **标记传递不丢**:`raise_softirq` 置位 + `irq_exit` 检查位图,这套接口保证 hardirq raise 的 softirq 一定会被跑(要么 `irq_exit` 就地跑,要么 ksoftirqd 稍后跑)。位图是 per-CPU 关中断操作的,不会丢更新。
3. **约束放松有界**:softirq 比 hardirq 放松了"中断屏蔽"(softirq 跑时中断已开),但**没有放松"不可睡眠"**——softirq 仍在中断上下文(`in_softirq()` 真),`might_sleep()` 还是会崩。所以 softirq 能做"软件工作里不阻塞的那部分",真正要睡眠的(磁盘 IO、`mutex`)还得再往下一层挪到 workqueue。切分是**分层的约束放松**,不是一刀切。

> **钉死这件事**:上半部/下半部切分,是 Linux 中断处理的工程美学核心。它不是"为切而切",而是被两个硬约束(中断上下文不能睡眠、hardirq 跑久屏蔽中断)逼出来的分层。驱动作者每次写 handler 都要按"硬件约束 vs 可推迟软件工作"的尺子切,切错了就出问题。这是后面 P1-06(softirq)、P1-07(workqueue)两章的共同地基——softirq 和 workqueue 本质上都是"为可推迟的工作提供更宽松的执行上下文",差别只在放松到哪一层。

## 5.9 ★ 对照:上半部/下半部 ↔ Tokio mio + 异步 task

把内核的切分和用户态运行时的事件模型放一起,你会发现**两层切分是同构的**:

| 层 | 机制 | 上半部(快路径) | 下半部(后续) |
|---|---|---|---|
| 内核(本书) | 中断处理 | hardirq:应答硬件 + 拷数据 + `raise_softirq` | softirq/workqueue:协议栈、IO 后续 |
| 用户态运行时 | Tokio | `mio` 只调 `epoll_wait` 取就绪事件 | 异步 task:解析、业务、落盘 |

对照要点:

- **Tokio 的 `mio` 只做"取就绪事件"**(`epoll_wait` 返回哪些 fd 就绪),绝不在事件循环里跑业务逻辑。这和内核 hardirq "只做应答 + 拷数据 + 标记"一模一样——快路径只取"有事发生了"这个信号,重活丢给后续。
- **Tokio 的异步 task 接力**,和内核 softirq/workqueue 接力同构。task 在 worker 线程上跑,可以 `await`(让出执行流,等未来就绪)——这相当于内核 workqueue 的"可睡眠"。Tokio 没有"不可睡眠的下半部"(它的 task 都在用户态进程上下文),这是用户态运行时比内核简单的地方:它不用面对"中断上下文不能睡眠"这个硬约束,因为它的"事件"本来就是 `epoll` 这种系统调用返回的,不在中断里。
- **NAPI 的"中断一次 + 轮询批量吃光"**,和 Tokio 的 "一次 `epoll_wait` 批量取所有就绪 fd + 逐个 spawn task" 是同一种"用批量换中断/唤醒次数"的思路。两者的目的都是:**高负载下别让事件触发次数把 CPU 烧光**。

> **钉死这件事**:内核的中断切分(上半部快、下半部接力)和用户态运行时的事件切分(mio 取事件、task 接力)是**同一种工程模式**——把"快路径"和"重活"分层,快路径只取信号、重活丢给更宽松的执行流。内核的切分被"中断上下文不能睡眠"逼出来,Tokio 的切分被"事件循环不能阻塞"逼出来,动机同构。这是为什么读内核机制和读 Tokio 源码会反复"似曾相识"——它们解决的是同一类问题(把事件处理的快慢路径分层)。

---

## 章末小结

这一章我们钉死了 Linux 中断处理的工程核心:**上半部/下半部切分**。它不是设计洁癖,是被两个硬约束逼出来的分层——hardirq 不能睡眠(P1-04)、hardirq 跑久了屏蔽其他中断。所以上半部只做"非它不可"的最紧急的事(应答硬件、拷最少数据、置标记),下半部(softirq/tasklet/workqueue)放开约束做重活。三种下半部的差别全在"约束放松到什么程度":softirq 放松了中断屏蔽但仍不可睡眠,tasklet 是 softirq 的自串行化包装(6.x 在废弃),workqueue 放松到进程上下文可睡眠。线程化中断是第四条路,把整个 handler 丢给内核线程。

**回扣二分法**:本章服务"进内核"那一面。下半部机制(softirq/workqueue)本身不是新的事件入口,它服务的是"事件已经把控制权拉进内核后,怎么把处理做完"——所以它属于"进内核"那一面的事件处理链路。上半部是入口路径的终点,下半部是入口路径的延伸。

### 五个"为什么"清单

1. **为什么中断处理要切两段?** hardirq 不能睡眠(P1-04)、跑久了屏蔽其他中断(硬件层面中断优先级),所以只能干最紧急的事。可推迟的重活挪到下半部,下半部放开约束(softirq 开中断、workqueue 进程上下文可睡眠)才能做。切分把中断对系统的"打扰"压到最小。
2. **softirq、tasklet、workqueue 到底怎么选?** 不能睡眠且要最快→softirq(但驱动基本用不到,种类编译期写死);要自串行化又不能睡眠→tasklet(但 6.x 在废弃,别用);要睡眠或调阻塞 API→workqueue(唯一选择)。线程化中断(`request_threaded_irq`)是第四条路,适合"中断工作天然需要睡眠"或 RT 场景。
3. **为什么 tasklet 在被废弃?** 它"per-CPU 调度 + 全局串行化"的组合引入不必要的复杂(`TASKLET_STATE_SCHED`/`RUN` 双状态位 + 重新调度活锁),线程化中断和 ordered workqueue 能更干净地满足同样需求,且和 RT-Linux 兼容。源码头文件已标 deprecated。
4. **`irq_exit` 怎么把控制权交给下半部?** `__irq_exit_rcu` 把 `preempt_count` 的 hardirq 段减一后,检查 `!in_interrupt() && local_softirq_pending()`——不在中断里且有 softirq pending,就 `invoke_softirq` 就地跑(或唤醒 ksoftirqd 线程化)。这是 hardirq 到 softirq 的接力点。
5. **`IRQ_HANDLED` 和 `IRQ_WAKE_THREAD` 的区别?** `IRQ_HANDLED` 表示 handler 在 hardirq 里把事做完了(或已 raise softirq 排出后续),核心到此为止;`IRQ_WAKE_THREAD` 表示只做了最紧急的判断和应答,核心应唤醒 `thread_fn` 内核线程去做剩下的(线程化中断模式)。
