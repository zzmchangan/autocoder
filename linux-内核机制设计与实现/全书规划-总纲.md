# 《Linux 内核机制深入浅出:中断·系统调用·时钟·信号》—— 全书总纲

> 一本写给"写过 Linux 用户态程序、读过操作系统课、甚至翻过 `kernel/irq/`/`softirq.c`/`signal.c`/`kernel/time/`,却总觉得一知半解"的人的小书。
>
> 本文是全书的**总纲**:定主线、定方法、定分篇分章、定源码策略、定写作约定。
> 与《Tokio》《LevelDB》《etcd》《Go runtime》《内存分配器》《Linux mm》《Linux 调度器》同源,都是"**第一性原理 + 真实源码**";命脉同样是**动机(why) + 技巧(how) 双线**;基调延续《LevelDB》——**直球讲透为主,比喻只在反直觉处点睛**。
> **本书独有特色:★ 三向对照**——中断↔io_uring 完成事件、时钟↔Tokio 时间轮、信号↔Go channel/panic、系统调用↔Go runtime 的 systrap。把内核的"事件跨越边界"机制和用户态运行时的事件模型拼成完整图景。

## 一、这本书的定位

### 目标读者

会写 C、读过操作系统课、用过 `read`/`write`/`signal`/`sigaction`/`setitimer`/`request_irq`、甚至翻过 `kernel/irq/handle.c`、`kernel/softirq.c`、`kernel/entry/common.c`、`kernel/time/hrtimer.c`、`kernel/signal.c`,但**一知半解**:知道有中断、softirq、workqueue、系统调用、hrtimer、NOHZ、信号,却讲不清一次网卡中断从硬件触发到 NAPI 轮询的完整旅程、为什么 `SYSCALL` 比 `int 0x80` 快、hrtimer 凭什么做到纳秒精度、NOHZ idle 怎么让 CPU 睡下去又不丢 tick、信号为什么"延迟到返回用户态才真正处理"。和"读过 tokio/go runtime/mm/sched 源码但不理解"是同一种病。

### 全书一句话主旨

> **中断/异常把控制权从用户态拉进内核、系统调用是用户态合法进内核的入口、时钟驱动调度与定时、信号是内核向进程的异步通知——这四个机制都是"事件跨越用户/内核边界"或"内核主动驱动/通知",合起来构成内核的事件驱动骨架。**

每个组件,都是这条旅程上的一个**驿站**。读完一本书,读者能**在脑子里放映出内核的事件全貌**:一个网卡中断怎么把 CPU 拉进内核、上半部怎么快收快放、softirq 怎么在 IRQ 退出后接力、一个 `read()` 怎么走过 SYSCALL 入口和 `sys_call_table`、hrtimer 怎么在红黑树上挑最早到期的、NOHZ idle 怎么停 tick 又被下一个事件唤醒、一个 `kill -9` 怎么挂到目标进程的 pending 队列、等到它返回用户态时才真正跑 handler——以及每一步底下用了什么巧妙的手段。

### 为什么用"四个机制合成一本"当骨架(而非按子系统各写一本)

中断、系统调用、时钟、信号各自只有 4~6 章的体量,**单拎都撑不满一本 21 章的大书**(不像 mm/调度器/网络栈)。但它们**主题高度同构**:都是**事件跨越用户/内核边界**(中断/异常/系统调用是"进内核",时钟/信号是"内核向外")。合成本书,既够厚又成体系,而且**互为彼此的地基**——中断驱动网络/块 IO,时钟驱动调度器(回扣第 11 本 sched P1-04),系统调用是一切用户接口,信号是 IPC/异常通道。这是一本"小机制合,立事件驱动骨架"的书。

> **不这样会怎样**:如果按子系统各写一本,中断只有 6 章、系统调用只有 4 章,每本都太薄不成体系;如果按文件平铺(`kernel/irq/` 一章、`softirq.c` 一章、`hrtimer.c` 一章),读者会觉得四块是孤岛。用"事件跨越边界"这条主线串起来,每个机制都是为了让"内核能被事件驱动"而出现的角色,读者永远知道**我在哪、为什么需要下一站**。

## 二、写作方法

### 命脉:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与一般操作系统教材、源码导读最关键的分野,所有写作者必读、必守。**

Linux 这四个机制源码充斥内核级技巧:中断上下文不能睡眠(`in_interrupt()`/`preempt_count` 计数)、上半部/下半部切分(快路径 vs 延后处理)、softirq 的 per-CPU pending 位图 + `__do_softirq`/`handle_softirqs` 循环 + `MAX_SOFTIRQ_RESTART` 防饿死、workqueue 的 worker 池 + `process_one_work`、`SYSCALL` 指令 vs `int 0x80`(CPU 直切 vs 软中断)、VDSO 把 `gettimeofday` 留在用户态、hrtimer 的红黑树 + softexpires 区间 + 高精度 clockevent、NOHZ idle 停 tick 省电、信号的延迟投递(pending 队列 + `_TIF_SIGPENDING` 返回用户态检查)、信号栈切换的 `sigreturn` 回跳、内存序屏障(`smp_mb`/`smp_wmb`/`smp_rmb`)。用户自己读过源码,"一知半解"的缺口往往**不在**"它要解决什么"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?(沿用系列一贯的"为什么优先")
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果不用这个技巧、朴素地写,会撞上什么墙",让妙处显形。

一句话:**讲不清技巧,等于没讲 Linux 内核机制。**

### 基调:直球讲透为主,比喻只在反直觉处点睛 ⚠️ 延续《LevelDB》/《Linux mm》/《调度器》

这四个机制有一部分反直觉(为什么中断上下文不能睡眠、为什么信号要延迟到返回用户态才处理、NOHZ 为什么能停 tick 又不丢),这部分**允许适度比喻点睛**。但大部分(中断向量号、`sys_call_table` 数组、hrtimer 红黑树、pending 位图)**具体可见,直球讲清最好**。

- **默认直球**:能用表 / 流程图 / 源码说清的,不上比喻。
- **比喻只点睛**:softirq 可点"像中断的『续集』,在 IRQ 退出后才演";信号可点"像一封挂号信,内核先帮你签收(pending),等你回家(返回用户态)才当面交给你(handler)"。点到为止。
- **判断标准**:直球讲不清才上比喻。

### ★ 对照(Tokio/Go/io_uring):本书独有特色

> 本书是"事件跨越边界"机制的合集,和用户态运行时(Tokio/Go runtime)的事件模型天然一对——内核的中断/时钟/信号 vs Tokio 的事件循环/时间轮 vs Go 的 channel/panic。把它们放一起,就拼出了"一个事件怎么被操作系统处理"的**完整全栈**。

本书在中断下半部、时钟 hrtimer、信号投递、系统调用入口等关键章末用轻量"★对照"小栏,和 Tokio/Go/io_uring 对读;收尾章给总表。对照要点:

- **中断上半部/下半部** ↔ **Tokio 的 ` mio` epoll 事件就绪 + 异步任务后续处理**(内核硬中断只做最少的事,softirq/workqueue 接力,和 mio 只读事件、上层 task 接力是同构的)。
- **hrtimer 红黑树 + softexpires** ↔ **Tokio 时间轮(`tokio::time::wheel`)**(都是按到期时间组织、挑最早一个),但内核用红黑树(精确)、Tokio 用层级时间轮(批量、O(1))——不同的取舍。
- **NOHZ idle 停 tick** ↔ **Tokio 进程空闲时 park 线程**(都为"没事干就别空转")。
- **信号投递延迟到返回用户态** ↔ **Go channel 投递(异步、延迟)、panic(同步杀死 goroutine)**;信号的 "内核先记账、用户态才跑 handler" 模型,和 channel 的"先入队、收方稍后取"是同一思路。
- **`SYSCALL` 指令入口** ↔ **Go runtime 的 `runtime.systemstack_switch`/`asmsysdyn` 的 systrap**(都是"用户态合法进内核/进 runtime"的受控通道)。
- **中断↔io_uring 完成事件**:传统中断是"内核主动通知用户"(异步抢占);io_uring 的 CQE 是"用户主动轮询完成队列"(同步可批),两种事件模型的代际差。

## 三、全书主线、二分法

### 主线

> **中断/异常把控制权拉进内核、系统调用是用户态的合法入口、时钟驱动调度与定时、信号是内核向进程的异步通知——这四个机制都是"事件跨越用户/内核边界",合起来构成内核的事件驱动骨架。**

任何一处看不懂某个机制,回到这句问:"这是在**把控制权拉进内核/让用户合法进内核(进内核:中断/异常/系统调用入口)**,还是在**内核主动向外驱动/通知(时钟触发调度、信号投递)**,或是在**支撑这两件事(IRQ 抽象、时钟源、pending 队列、`task_struct` 标志)**?"把答案写进"为什么"里。

### 二分法(迷路时回到它)

本书的二分法:

> **进内核(中断/异常/系统调用入口,事件把控制权拉进内核) vs 内核主动驱动/通知(时钟触发调度、信号投递,内核向外)。**

- **进内核这一面**:硬件中断(interrupt,异步外部事件)、异常/陷阱(exception/trap,同步 CPU 事件)、系统调用(syscall,用户态主动合法入口)。这些都"在用户态跑着 → 突然进内核执行"。代表机制:中断入口 `handle_arch_irq` → `handle_irq_event`、softirq 上半部接力、`SYSCALL` 指令 → `do_syscall_64` → `sys_call_table`。
- **内核主动驱动/通知这一面**:时钟驱动(时钟硬件周期触发,内核主动推进调度/timer/墙上时间)、信号投递(内核把"有事找你"挂到目标进程的 pending 队列)。这些都"内核内部产生事件,要么推进自己(调度/timer),要么通知用户进程(信号)"。代表机制:`hrtimer_interrupt` → 调度器 hrtick、`complete_signal` → `_TIF_SIGPENDING`。

支撑这两者的地基:IRQ domain/中断控制器抽象、clocksource/clockevent 硬件抽象、`task_struct` 的 `pending`/`sighand`、`preempt_count`/`thread_flags`、per-CPU 数据。

每个机制都落在这两面之一或支撑地基。任何时候迷路,回到这个二分法。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线是一条旅程:**为什么内核要管这四个机制 → 中断怎么把控制权拉进内核 → 系统调用怎么合法进内核 → 时钟怎么驱动调度与定时 → 信号怎么通知进程 → 哲学收束**。
> 章号全局连续;命名 `P{篇}-{章}-标题.md`。**全书 21 章 6 篇**(核心全覆盖,4 个机制各 4-6 章 + 开篇收尾)。

### 第 0 篇 · 开篇:为什么内核要管中断/系统调用/时钟/信号(1 章)

- **第 1 章 · 第一性原理:为什么内核要管中断与系统调用**(`P0-01`)——用户态↔内核态边界是 OS 命脉,事件驱动 vs 轮询的本质权衡,中断是"被动响应"的唯一方式、系统调用是"用户合法进入"的唯一入口、时钟是"调度与定时的心跳"、信号是"进程间异步通知";立起"进内核 vs 内核主动"二分法 + ★对照 Tokio/Go/io_uring。**【样本章节】**

### 第 1 篇 · 中断与软中断:把控制权拉进内核(6 章)⚠️ 用户点名

> 源码主体 `kernel/irq/`(`manage.c`/`handle.c`/`chip.c`/`irqdomain.c`/`internals.h`)、`kernel/softirq.c`、`kernel/workqueue.c`、`include/linux/interrupt.h`/`irq.h`。建议顺序读(硬件中断 → 抽象 → 上下文 → 上半部/下半部 → softirq → workqueue)。

- **第 2 章 · 硬件中断与中断向量:CPU 怎么被拉进内核**(`P1-02`)——中断电气本质(CPU 每条指令后查中断引脚)、x86 中断向量号(0-255)、IDT(中断描述符表)、IRQ 号 vs 向量号、`handle_arch_irq` 入口。源码:`kernel/irq/handle.c`(`set_handle_irq`)、arch 描述。**技巧精解:IDT 表 + CPU 自动保存上下文**(硬件做的工作,反面对比"软件轮询网卡会浪费 CPU")。归属:`进内核`。
- **第 3 章 · IRQ domain 与中断控制器抽象:`irq_chip`/`irq_domain`**(`P1-03`)——Linux 怎么把五花八门的中断控制器(GIC/LAPIC/IOAPIC/MSI)抽象成统一接口,`struct irq_chip`(硬件操作)、`struct irq_domain`(硬件中断号 → Linux IRQ 号映射)、`irq_desc` 中断描述符。源码:`kernel/irq/irqdomain.c`、`kernel/irq/chip.c`、`include/linux/irq.h`(`struct irq_chip`@L501、`struct irq_data`@L179)。**技巧精解:`irq_domain` 的层级映射**(硬件 hwirq → Linux IRQ,反面对比"每驱动自己维护映射表会乱套")。归属:`支撑`。
- **第 4 章 · 中断上下文:为什么不能睡眠**(`P1-04`)——中断上下文(hardirq context)的本质:不是进程,没有可调度实体,`current` 是被中断的进程但不是它;`preempt_count` 的 `HARDIRQ_OFFSET`/`SOFTIRQ_OFFSET` 计数、`in_interrupt()`、为什么不能 `sleep`/`mutex`(没有 task 结构可挂起)。源码:`kernel/softirq.c`(`irq_enter_rcu`@L594/`__irq_exit_rcu`@L627)、`include/linux/preempt.h`。**技巧精解:`preempt_count` 嵌套计数**(用一位段表示 hardirq/softirq/nmi/抢占屏蔽,反面对比"用一个布尔标志无法处理嵌套")。归属:`支撑`。
- **第 5 章 · 上半部与下半部:为什么中断要切两段**(`P1-05`)——上半部(hardirq)快收快放、下半部(deferred work)延后处理;为什么不能在中断里干重活(占用 CPU、可能屏蔽其他中断、不可睡眠);三种下半部:softirq(轻量、per-CPU、不可睡眠)、tasklet(softirq 包装,BH 时代的遗物,6.x 在逐步废弃)、workqueue(可睡眠、有进程上下文)。源码:`include/linux/interrupt.h`、`kernel/softirq.c`、`kernel/workqueue.c`。**技巧精解:上半部/下半部切分**(性能与功能的切分,反面对比"全部在 hardirq 做会卡死其他中断")。归属:`进内核`(下半部机制仍服务"进内核"那一面的事件处理)。★对照 Tokio(` mio` 只读事件,异步 task 接力)。
- **第 6 章 · softirq 软中断:per-CPU 的延迟工作**(`P1-06`)——softirq 是中断的"续集":hardirq 退出时若 pending 有位,`irq_exit` 调 `invoke_softirq` → `__do_softirq` → `handle_softirqs`;32 位的 pending 位图、`softirq_vec[]` 数组、`open_softirq` 注册、`raise_softirq` 标记、`MAX_SOFTIRQ_RESTART`/`MAX_SOFTIRQ_TIME` 防饿死、ksoftirqd 兜底线程。源码:`kernel/softirq.c`(`softirq_vec[]`@L60、`open_softirq`@L703、`raise_softirq`@L687、`__do_softirq`@L586、`handle_softirqs`@L511、`irq_exit` 路径)。**技巧精解:per-CPU pending 位图 + ffs 找第一个置位**(无锁、O(1) 查 pending;`MAX_SOFTIRQ_RESTART` 防止 softirq 自激饿死普通进程,反面对比"无限制循环会让 softirq 永远占着 CPU")。归属:`进内核`(下半部)。★对照 Tokio(` mio` 就绪事件 → task 调度)。
- **第 7 章 · workqueue:可睡眠的延迟工作**(`P1-07`)——softirq/tasklet 不能睡眠,但有些延后工作要调阻塞 API(如 `mutex`/磁盘 IO);workqueue 提供**有进程上下文**的延迟工作:`work_struct` + `worker_pool` + worker 内核线程、`queue_work` 入队、`process_one_work` 执行;concurrency managed(CMWQ)保证并发又不爆炸。源码:`kernel/workqueue.c`(`__queue_work`@L2313、`queue_work_on`@L2439、`process_one_work`@L3166、`worker_thread`@L3374)。**技巧精解:CMWQ 的 worker 池管理**(按需唤醒 worker、并发度管制,反面对比"每 work 一个线程会爆炸")。归属:`进内核`(下半部)。

### 第 2 篇 · 系统调用:用户态合法进内核(4 章)⚠️ 用户点名

> 源码主体 `kernel/entry/common.c`(通用入口框架)、`kernel/sys.c`(系统调用实现集合)、arch/x86/entry 描述(`entry_64.S`/`syscall_64.c`,未 sparse clone 则描述 + 引通用框架)。

- **第 8 章 · 系统调用入口:`SYSCALL` 指令与 `sys_call_table`**(`P2-08`)——x86_64 的 `SYSCALL` 指令 vs 老 `int 0x80`(CPU 直切 vs 软中断,`SYSCALL` 不压完整的 trap frame 更快)、`do_syscall_64` 入口、`sys_call_table[]` 函数指针数组(用系统调用号索引)、参数传递(rdi/rsi/rdx/r10/r8/r9 六寄存器)、返回值。源码:arch/x86/entry 描述 + `kernel/entry/common.c`(`syscall_enter_from_user_mode_prepare`@L74、`syscall_exit_to_user_mode`@L215)。**技巧精解:`SYSCALL` 指令 vs `int 0x80` 的快路径**(`SYSCALL` 不查 IDT、不切栈硬件压完整 frame,CPU 用 MSR 直接跳到内核入口,反面对比"`int 0x80` 走 IDT 查询 + 完整 trap frame,慢一个数量级")。归属:`进内核`。★对照 Go runtime 的 systrap。
- **第 9 章 · 参数传递与系统调用返回**(`P2-09`)——用户态指针怎么进内核(`copy_from_user`/`copy_to_user` 的边界检查)、参数寄存器约定、返回值(0 表示成功、负数表示 `-errno`)、`errno` 的用户态映射。源码:`kernel/sys.c`(`SYSCALL_DEFINE*` 宏展开)、`arch/x86/lib/copy_user_*` 描述。**技巧精解:`copy_from_user` 的 page fault 容错**(用户态指针可能非法,内核访问会触发 page fault,内核用 `__get_user` 的 fixup 表把 fault 路径回到 `-EFAULT` 返回,反面对比"直接解引用非法指针会内核 panic")。归属:`进内核`。
- **第 10 章 · VDSO:不进内核的"系统调用"**(`P2-10`)——`gettimeofday`/`clock_gettime` 每秒可能被调百万次,真走系统调用开销巨大;VDSO(Virtual Dynamic Shared Object)把内核映射的共享页 + 用户态函数让"读时间"完全不进内核:内核把墙上时间写在共享页、用户态直接读。源码:`arch/x86/entry/vdso/` 描述 + `kernel/time/timekeeping.c`(`tk_update_offset`)。**技巧精解:VDSO 共享页 + seqlock**(内核写时 seqlock 奇数、用户态读到偶数才用,无锁又一致;反面对比"每次读时间都进内核,百万级 QPS 直接拖垮 CPU")。归属:`进内核`(VDSO 是"避免进内核"的优化,服务系统调用这一面)。
- **第 11 章 · 系统调用追踪:seccomp 与 ftrace**(`P2-11`)——seccomp(BPF 过滤系统调用,沙箱核心)、ftrace/`trace_events`(观测系统调用延迟、参数、返回值)、`/proc/<pid>/syscall`。源码:`kernel/seccomp.c`、`kernel/trace/`。**技巧精解:seccomp BPF**(在系统调用入口前用 BPF 过滤,反面对比"系统调用之后才检查已经被系统调用执行开销")。归属:`进内核`。

### 第 3 篇 · 时钟与定时器:内核主动驱动的心跳(5 章)

> 源码主体 `kernel/time/`(`hrtimer.c`/`tick-sched.c`/`timekeeping.c`/`clocksource.c`/`clockevents.c`/`posix-cpu-timers.c`/`posix-timers.c`)。这一篇与调度器第 11 本 P1-04(hrtick)回扣。

- **第 12 章 · clocksource/clockevent:硬件时钟抽象**(`P3-12`)——时钟硬件五花八门(TSC/HPET/PIT/ACPI PM),内核抽象成两套:`struct clocksource`(只读、单调、提供 `read()` 返回纳秒,墙上时间用)、`struct clock_event_device`(可编程定时中断,触发 tick/hrtimer 用)。源码:`kernel/time/clocksource.c`(`struct clocksource`@include/linux/clocksource.h#L96)、`kernel/time/clockevents.c`。**技巧精解:clocksource 的 `read()` 抽象 + watchdog**(多个时钟源互校,反面对比"硬件直接暴露会让驱动各写一份")。归属:`支撑`。
- **第 13 章 · timekeeping:墙上时间怎么维护**(`P3-13`)——`struct timekeeper`(墙上时间 xtime、单调时间 monotonic、NTP 校正),`ktime_get`/`ktime_get_ts`/`ktime_get_real`、NTP 校正(`adjtimex`/`ntp.c`)、时间向前修正。源码:`kernel/time/timekeeping.c`、`include/linux/timekeeper_internal.h`(`struct timekeeper`@L92)。**技巧精解:timekeeper 的 seqlock + per-CPU 缓存**(写时 seqlock 奇数、读者重试,反面对比"全局锁会让 `gettimeofday` 在多核上排队")。归属:`支撑`。
- **第 14 章 · hrtimer:高精度红黑树定时器**(`P3-14`)⚠️ **核心重头戏**——hrtimer 取代老的低精度 timer wheel:每个 CPU 一个 `hrtimer_cpu_base`,内含多个 `hrtimer_clock_base`(MONOTONIC/REAL/BOOT/TAI),每个 base 一棵红黑树(实际是 timerqueue,基于红黑树)按到期时间排序;`hrtimer_start` 入队、`__hrtimer_run_queues` 扫到期、`hrtimer_interrupt` 处理;**softexpires 区间**允许合并唤醒减少中断。源码:`kernel/time/hrtimer.c`(`__hrtimer_run_queues`@L1724、`hrtimer_interrupt`@L1788、`__hrtimer_next_event_base`@L505)、`include/linux/hrtimer_types.h`(`struct hrtimer`@L39)。**技巧精解:红黑树 + softexpires 区间**(挑最早到期 O(log n);softexpires 让多个 timer 的"最早可被唤醒"对齐,避免每次加一个 timer 都重新编程 clockevent,反面对比"朴素按到期唤醒会有 thundering herd 的中断风暴")。归属:`内核主动`(时钟驱动)。★对照 Tokio 时间轮(`tokio::time::wheel`):内核用红黑树求精确、Tokio 用层级时间轮求 O(1) 批量,不同的取舍。
- **第 15 章 · tick 与 NOHZ:动态时钟省电**(`P3-15`)——周期性 tick(每 HZ 一次)驱动统计/调度,但 idle 时空跑 tick 浪费电;NOHZ(NO_HZ_IDLE)让 CPU 进 idle 时停掉下一个 tick、由最近一个 hrtimer 到期时唤醒;NOHZ_FULL(给 CPU 跑单一 CPU 密集任务的极端场景,几乎完全无 tick)。源码:`kernel/time/tick-sched.c`(`tick_nohz_idle_enter`@L1250、`tick_nohz_irq_exit`@L1287、`tick_nohz_restart_sched_tick`@L1086、`tick_sched_do_timer`@L206)。**技巧精解:NOHZ idle 停 tick 的"借下一个 hrtimer"**(进 idle 时把下一个 tick 取消、改让最近的 hrtimer 唤醒 CPU,反面对比"无 NOHZ 的 CPU idle 时还在每秒 100/250/1000 次空转,服务器功耗与虚拟化噪声巨大")。归属:`内核主动`。★对照 Tokio 进程空闲 park 线程。
- **第 16 章 · POSIX timer/itimer:用户态定时器 API**(`P3-16`)——`setitimer`/`timer_create`/`timer_settime` 的用户接口,内核映射成 hrtimer 或 CPU timer(进程 CPU 时间消耗的定时器,`posix-cpu-timers.c`);SIGALRM/SIGPROF 投递;与调度器 PELT 的关系。源码:`kernel/time/posix-timers.c`、`kernel/time/posix-cpu-timers.c`、`kernel/time/itimer.c`。**技巧精解:POSIX timer 复用 hrtimer**(老 itimer 是 hrtimer 的薄包装,反面对比"独立实现会让定时器机制碎片化")。归属:`内核主动`(连接时钟和信号,呼应第 4 篇)。

### 第 4 篇 · 信号:内核向进程的异步通知(4 章)

> 源码主体 `kernel/signal.c`、`include/linux/sched/signal.h`(`struct signal_struct`/`sighand_struct`)、`arch/x86/kernel/signal.c` 描述。

- **第 17 章 · 信号投递:`send_sig`/`complete_signal`**(`P4-17`)——`kill`/`tgkill`/`raise`/`rt_sigqueueinfo` 的入口、`do_send_sig_info` → `complete_signal`,挂到目标进程的 `pending` 队列(`struct sigpending` + `sigqueue`);进程级 vs 线程组级信号;信号编号与默认动作。源码:`kernel/signal.c`(`do_send_sig_info`@L1294、`complete_signal`@L995)。**技巧精解:`sigpending` 队列 + 信号合并**(普通信号同编号只挂一次、实时信号排队不丢,反面对比"无队列的全局位图无法表达信号数据 `siginfo`,且实时信号会丢")。归属:`内核主动`(信号投递)。★对照 Go channel(投递延迟到收方)。
- **第 18 章 · 处理入口:返回用户态检查 `_TIF_SIGPENDING`**(`P4-18`)——信号投递只是"挂账",真正跑 handler 在**返回用户态前**:`exit_to_user_mode_loop` 检查 `thread_info->flags` 的 `_TIF_SIGPENDING`/`_TIF_NOTIFY_RESUME` 位,有则调 `arch_do_signal_or_restart` → `get_signal` 取信号 → `handle_signal` 跳到用户态 handler。源码:`kernel/entry/common.c`(`exit_to_user_mode_loop`@L90、`syscall_exit_to_user_mode`@L215、`arch_do_signal_or_restart`@L83)、`kernel/signal.c`(`get_signal`@L2675)。**技巧精解:信号延迟到返回用户态才处理**(内核只挂 pending,handler 必须在用户态跑——因为 handler 可能要访问用户栈/用户数据;反面对比"内核态直接跑 handler 会破坏用户上下文、且 handler 自身就是用户代码")。归属:`内核主动`。★对照 Go 的 panic(同步进 runtime)、channel(收方主动取)。
- **第 19 章 · `sigaction` 与 handler 栈切换:`rt_sigreturn`**(`P4-19`)——`sigaction` 注册 handler;信号到达时内核在**用户栈上**构建 `ucontext` + `siginfo` 帧(handler 返回时能恢复原上下文)、把返回地址改成 `rt_sigreturn` 系统调用入口;handler 跑完自动调 `rt_sigreturn` 回内核恢复被打断的现场;可选 `SA_ONSTACK` + `sigaltstack`(自定义信号栈,主栈溢出时还能跑 handler)。源码:arch/x86/kernel/signal.c 描述(`__setup_rt_frame`)+ `kernel/signal.c`。**技巧精解:用户栈上构建 sigframe + `rt_sigreturn` 回跳**(让 handler 像普通函数调用,但"返回"被劫持成恢复原上下文的系统调用;反面对比"内核保存完整用户上下文会引入复杂状态机")。归属:`内核主动`。
- **第 20 章 · 信号与异常的关系**(`P4-20`)——CPU 异常(缺页/除零/SIGFPE、非法指令/SIGILL、段错误/SIGSEGV)本质上也是"内核向进程发信号";异常处理路径:CPU 触发 → 内核异常入口 → 判断是否可修复(如缺页)→ 不可修复则 `force_sig` 投递;与硬件中断的区别(异常同步、中断异步);与信号的统一(都是"事件通知进程")。源码:`kernel/signal.c`(`force_sig_info` 等)、arch/x86 异常入口描述。**技巧精解:异常路径与信号投递的统一**(异常不可恢复时统一走 `force_sig`,让用户态用同一套信号 handler 处理硬件错误;反面对比"异常单独机制会让用户接口碎片化")。归属:`进内核`(异常) + `内核主动`(信号)的桥梁。

## 七、内核 C 技巧侧重(对标 Tokio Rust / Go runtime / LevelDB C++ / mm/sched 内核 C)

本书要重点拆的 Linux 内核机制技巧(写作时遇到这些,务必钻"怎么做到的 + 为什么 sound"):

- **上半部/下半部切分**:hardirq 快收快放、softirq/workqueue 接力——性能与功能的切分(防止中断嵌套阻塞、长处理让其他中断饿死)。
- **softirq 的 per-CPU pending 位图**:每个 CPU 一个 `__softirq_pending` 位图,`__do_softirq` → `handle_softirqs` 用 `ffs` 找第一个置位、`softirq_vec[]` 数组分发——per-CPU 无锁、O(1) 找待处理。
- **`MAX_SOFTIRQ_RESTART`/`MAX_SOFTIRQ_TIME` 防饿死**:softirq 循环限制次数/时间,超了就唤醒 ksoftirqd 内核线程接管,防止 softirq 自激(网络 RX 触发 TX 触发 RX)永远占着 CPU。
- **`preempt_count` 嵌套计数**:用一段 bit 位同时表示 hardirq/softirq/nmi/抢占屏蔽/RCU,`in_interrupt()`/`in_softirq()`/`in_irq()` 用位段判断——嵌套中断的关键。
- **CMWQ(concurrency managed workqueue)的 worker 池**:按需唤醒 worker、并发度管制,既不爆炸又能并发执行阻塞工作。
- **`SYSCALL` 指令快路径**:CPU 用 MSR 直接跳到内核入口,不查 IDT、不压完整 trap frame,比 `int 0x80` 快一个数量级。
- **VDSO + seqlock**:内核把时间写在共享页,用户态无系统调用直接读;seqlock 保证读到一致(奇数版本号=正在写,读者重试)。
- **`copy_from_user` 的 fixup 表**:用户态指针非法触发 page fault,内核 fixup 表把 fault 路径回 `-EFAULT`,不 panic。
- **clocksource 的 `read()` 抽象 + watchdog**:多时钟源互校,自动降级不可靠的源。
- **timekeeper 的 seqlock 写**:写墙上时间时 seqlock 奇数版本号,`ktime_get` 读者重试,无锁高并发。
- **hrtimer 红黑树(timerqueue)+ softexpires 区间**:O(log n) 找最早到期;softexpires 允许多个 timer 对齐唤醒,减少 clockevent 重编程。
- **NOHZ idle "借下一个 hrtimer"**:进 idle 时把周期 tick 取消,让最近的 hrtimer 充当唤醒源,CPU 真正睡眠。
- **信号的延迟投递**:内核只挂 `pending` 队列(`sigqueue` 链表 + 位图),真正 handler 在返回用户态前 `_TIF_SIGPENDING` 检查时跑——handler 必须在用户态执行(访问用户上下文)。
- **sigframe + `rt_sigreturn` 回跳**:内核在用户栈上构建 `ucontext` + `siginfo`,handler 像普通函数返回,但"返回地址"被劫持成 `rt_sigreturn` 系统调用。
- **per-CPU 数据无锁化**:softirq pending、hrtimer cpu_base、`__get_cpu_var`——避免多核锁竞争。
- **内存序屏障**:`smp_mb`/`smp_wmb`/`smp_rmb` 在 pending 位图发布、信号投递、唤醒的正确使用。
