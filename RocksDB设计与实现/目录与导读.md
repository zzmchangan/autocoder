# 《RocksDB 设计与实现深入浅出:为什么把 LSM 的每个旋钮都交给你》—— 目录与导读

> 一本写给"用过 RocksDB、调过几个 Options、甚至翻过 rocksdb 源码,却总觉得一知半解"的人的小书。
>
> **一句话主旨**:LevelDB 把读写放大钉死在一个固定点;RocksDB 把这个点上每个写死的决策,都做成了可调旋钮,让你在"写快 / 读快 / 省空间"三角里,精确定位到你的 workload 想要的那个点。
>
> **二分法**(迷路时回到它):**写路径**(WriteBatch→WAL→MemTable→Flush→Compaction,产生写放大) vs **读路径**(Cache→Index/Filter→多层 SST 归并,产生读放大);Column Family / Snapshot / Transaction / BlobDB 是横切两面。
>
> **承接**:《LevelDB 设计与实现》是本书的"基线篇"——LevelDB 讲透的(SkipList 无锁读/WAL header/SST footer/MergingIterator/Bloom/Manifest/Snapshot/写组)一句话带过 + 指路,篇幅全留 RocksDB 独有;P7 反哺《TiKV》(TiKV 的单机引擎就是 RocksDB)。
>
> **主比喻**:直球为主、比喻点睛。开篇用"调音台"做一次性点位睛——LevelDB 是旋钮焊死的收音机,RocksDB 把每个旋钮都接出来。

每章一行:**一句话钩子** —— 技巧标签 —— 二分法归属(`写路径` / `读路径` / `格式` / `调控` / `横切` / `总览`)。

旅程:从"一次本地的 Put/Get",一路走到"工业级 LSM 怎么把读写放大调到你想要的点"。每篇都是这次旅程上的一个驿站——读完你能在脑子里放映出一次 Put 的全过程(WriteBatch→WriteGroup 攒批→WAL→InlineSkipList MemTable→Flush 成 L0 SST→Compaction 逐层下压)和一次 Get 的全过程(Block Cache→Index 二分→Bloom 早退→多层归并),以及 LevelDB 哪里写死、RocksDB 打开成了什么旋钮。

## 第 1 篇 · 写路径:从 WriteBatch 到 L0

> 写进去怎么秒回、怎么攒批、怎么落盘成 SST。这一篇是写路径地基,**建议顺序读**。

- [P1-02 · WriteBatch 与写组](P1-02-WriteBatch与写组.md) —— 一次 Put 怎么进 WriteBatch、被 WriteGroup 攒成一批一次写 WAL,leader 写、followers 零拷贝 piggyback。 —— WriteGroup 批写合并 + 批链接 —— `写路径`
- [P1-03 · WAL:落盘与回收](P1-03-WAL-落盘与回收.md) —— 写组怎么落 WAL,log file 怎么复用、sync 策略、自动裁剪。 —— log file 复用 + sync 策略 + wal_ttl —— `写路径`
- [P1-04 · MemTable:并发跳表](P1-04-MemTable-并发跳表.md) —— MemTable 长什么样:InlineSkipList 支持并发无锁插入(承 LevelDB 单写跳表),还能换成 hash/vector。 —— InlineSkipList 并发插 + 可插拔 MemTableRep —— `写路径(招牌)`
- [P1-05 · Flush:MemTable 变 SST](P1-05-Flush-MemTable变SST.md) —— MemTable 满了怎么 Flush 成 L0 SST,流水线写 WAL 与 Flush,触发阈值怎么定。 —— Flush 调度 + Pipelined Flush + 阈值 —— `写路径`

## 第 2 篇 · 静态格式:Block-based Table 与 Manifest

> 数据在磁盘上长什么样。这是读路径的地基(SST 看不懂,读路径无从讲起),**建议顺序读**。

- [P2-06 · Block-based Table 格式](P2-06-Block-based-Table格式.md) —— 一个 SST 内部怎么分块:data / index / filter / metaindex / footer,承 LevelDB footer 但更丰富。 —— 多 block 布局 + properties block —— `格式`
- [P2-07 · Index/Filter 分离与分区](P2-07-Index-Filter分离与分区.md) —— 为什么 index/filter 要能从 data 分离、甚至自己分区:大索引不一次载入、cache 按角色分。 —— Partitioned index/filter + data block hash + cache 角色 —— `格式(招牌)`
- [P2-08 · Bloom 与 Ribbon Filter](P2-08-Bloom与Ribbon-Filter.md) —— point lookup 怎么避免无谓读 block:Bloom 承 LevelDB,Ribbon 省 ~30% 内存。 —— Bloom DoubleHashing + Ribbon 可重建 —— `格式`
- [P2-09 · Version Set 与 Manifest](P2-09-Version-Set与Manifest.md) —— 所有 SST 的花名册怎么维护、多 CF 元数据怎么组织。 —— VersionSet/Edit + CF MANIFEST —— `格式`

## 第 3 篇 · 读路径:从 Get 到多层归并

- [P3-10 · Block Cache](P3-10-Block-Cache.md) —— 读 block 怎么不每次碰磁盘:分片 LRU 承 LevelDB + 多档 pin + cache 按角色分。 —— 多档 pin + row_cache + cache 角色 —— `读路径`
- [P3-11 · Get 路径与读放大](P3-11-Get路径与读放大.md) —— 一次 Get 怎么穿透 memtable→immutable→L0→Ln,Bloom 早退、Index 二分、读放大怎么压。 —— point lookup 全链路 + 早退 —— `读路径`
- [P3-12 · Iterator 与多路归并](P3-12-Iterator与多路归并.md) —— 范围扫描怎么合并多层 SST,承 LevelDB MergingIterator + Prefix seek。 —— 合并迭代 + Prefix seek + pinning —— `读路径`

## 第 4 篇 · Compaction:三种策略与收敛(全书灵魂招牌)

> 写放大靠 Compaction 收敛。RocksDB 的灵魂:把 LevelDB 那一种 Compaction 做成三种,适配不同 workload。**建议顺序读**。

- [P4-13 · Compaction 框架](P4-13-Compaction框架.md) —— Compaction 怎么调度执行:job/picker/iterator 三件套 + subcompaction 并发拆分。 —— compaction 三件套 + subcompaction —— `写路径(招牌)`
- [P4-14 · Level Compaction](P4-14-Level-Compaction.md) —— 默认分层策略,承 LevelDB 但 level 大小可调,适合读多写少。 —— max_bytes_for_level 可调 —— `写路径(招牌)`
- [P4-15 · Universal Compaction](P4-15-Universal-Compaction.md) —— 写优化策略,宽 L0 合并相似文件,用空间放大换写放大小,SSD 友好。 —— size ratio/age 触发 —— `写路径(招牌)`
- [P4-16 · FIFO Compaction 与 CompactionFilter](P4-16-FIFO-Compaction与CompactionFilter.md) —— 时序先进先出删旧 + 用户在 compaction 时过滤改 KV。 —— FIFO TTL + CompactionFilter —— `写路径`

## 第 5 篇 · 可调控面:反压、限速、隔离

- [P5-19 · Column Family](P5-19-Column-Family.md) —— 多逻辑库共享一个引擎实例又隔离:共享 WAL/MANIFEST、独立 MemTable/SST/Options。 —— 共享 WAL + 独立 SST 隔离 —— `调控(架构招牌)`
