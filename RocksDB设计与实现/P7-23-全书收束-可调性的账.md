## 〇、一句话点破

> **LevelDB 是一台旋钮焊死的收音机——音质出厂固定,你拧不动;RocksDB 是把每个旋钮都接出来的调音台。这本书讲的不是"调音台怎么调",而是"每个旋钮背后是什么机制,转一下读写放大三角上的点会怎么动"。走完 22 章的驿站,这章把所有旋钮收成一张账,告诉你:这套可调性的代价有多大,以及在 TiKV / MySQL / Kafka 里它真的撑起了什么。**

这是结论,不是理由。本章倒过来收:先把 22 章的旋钮汇总成一张总账,再讲这套可调性在工业系统里怎么落地,最后讲它的代价与未来。

## 一、从 P6-22 接过来:横切篇讲完,全书机制讲完

横切篇(P6-20 Snapshot/MVCC,P6-21 Transaction/Merge,P6-22 BlobDB/运维)讲完,本书 22 个机制章全部走完。回头看你走过的路,每篇都拆了一组焊点:

- **第 1 篇(写路径地基)**:WriteBatch/WriteGroup 攒批(把 LevelDB 的 leader-follower 写组,演进成支持高并发攒批的 WriteGroup)→ WAL 落盘与回收(把 LevelDB 写死的 sync 策略、log file 复用打开)→ MemTable 的 InlineSkipList(把 LevelDB 单写者 SkipList 打开成多写者无锁插)→ Flush 成 L0 SST(把 LevelDB 写死的 MemTable 大小/数量打开,Pipelined Flush 流水线化)。**这一篇拆的焊点:写路径上"多大、几个、怎么攒、怎么落盘"全是常数**。
- **第 2 篇(静态格式)**:Block-based Table 多 block 布局(比 LevelDB 的 footer 更丰富,加 properties block)→ Index/Filter 分离与分区(把 LevelDB 写死的一个大 index/filter block,打开成可分区、可按角色缓存)→ Bloom/Ribbon(把 LevelDB 写死的 Bloom DoubleHashing,补一个省 30% 内存的 Ribbon)→ VersionSet/Manifest(多 CF 的元数据组织)。**这一篇拆的焊点:磁盘格式上"索引怎么组织、filter 用哪种、元数据怎么记"全可换**。
- **第 3 篇(读路径)**:Block Cache 多档 pin(把 LevelDB 的分片 LRU 一档,打开成 in-use/high_pri 多档 + row_cache + persistent_cache + HyperClockCache)→ Get 穿透多层(Bloom/Ribbon 早退、Index 二分、`max_sequential_skip_in_iterations`)→ Iterator 多路归并(Prefix seek、Iterator pinning)。**这一篇拆的焊点:读路径上"缓存几档、扫多少、怎么早退"全可调**。
- **第 4 篇(Compaction 三策略,招牌)**:框架三件套(job/picker/iterator)+ subcompaction 并发 → Level(读优化,层大小可调)→ Universal(写优化,用空间放大换写放大)→ FIFO + CompactionFilter(时序 + 用户过滤)。**这一篇拆的焊点,是全书最重的——LevelDB 写死只有一种 Compaction,RocksDB 打开成三种,这是"用空间放大换写放大"这个选项第一次出现**。
- **第 5 篇(可调控面)**:Write Stall/Delay 两档反压(把 LevelDB 没有"写太快怎么办"补上)→ Rate Limiter 令牌桶限速(后台 IO 不挤前台)→ Column Family 共享 WAL 又隔离(架构级演进)。**这一篇拆的焊点:LevelDB 没有"反压、限速、多逻辑库"这三件工业必备**。
- **第 6 篇(生产可用)**:Snapshot MVCC(承 LevelDB 但与 compaction 交互更精细)→ Transaction/Merge(跨 key 原子写 + 读时合并)→ BlobDB/运维(大 value 分离 + Checkpoint/Backup/CompactRange/Stats)。**这一篇拆的焊点:LevelDB 没有"事务、大 value 分离、运维工具"**。

每一章,都在重复同一件事——**LevelDB 这里焊死,RocksDB 把它打开成旋钮**。可这个"打开"到底打开了多少个?每个旋钮在读写放大三角上动了哪个点?解决什么、代价什么?现在把它们收成一张账。

### 2.1.1 怎么读这张表:三种"打开"的姿态

把表里几十个旋钮再归归类,会发现 RocksDB"打开焊点"的姿态有三种,这三种姿态也是全书 22 章的展开方式:

**姿态一:把常数变成可配参数(最常见)**。LevelDB 写死一个数字,RocksDB 把它做成 Options 字段,你填数字。这是最大一类——`write_buffer_size`(4MB→可调 64MB)、`max_bytes_for_level_base`(10MB→可调 256MB)、`max_bytes_for_level_multiplier`(10→可调 10)、`target_file_size_base`(2MB→可调 64MB)、`level0_slowdown_writes_trigger`(不存在→可调 20)、各种 trigger/limit 全是这一类。这类旋钮的本质是**"LevelDB 替你选了一个值,RocksDB 让你自己选"**。P1 篇的 MemTable/Flush 大小、P4 篇的 Compaction 层级、P5 篇的反压阈值,大多数是这种姿态。

**姿态二:把单一策略变成多选(最体现灵魂)**。LevelDB 写死"就这么做",RocksDB 给你几种做法让你选。最典型的是 Compaction:LevelDB 只有一种(类 Level),RocksDB 给你 Level / Universal / FIFO / None 四选(`advanced_options.h:29`)。再如 MemTable 底层结构:LevelDB 写死 SkipList,RocksDB 给你 SkipList / hash_linklist / hash_skiplist / vector 可插拔(`memtable/`)。Filter 也是:LevelDB 写死 Bloom,RocksDB 给你 Bloom / Ribbon 可选。这类旋钮的本质是**"LevelDB 替你选了一种机制,RocksDB 让你按 workload 换机制"**。P4 篇三种 Compaction、P1-04 可插拔 MemTableRep、P2-08 Ribbon,是这种姿态。这也是为什么 P4 篇是全书灵魂招牌——三种 Compaction 不是"三个参数",是"三种机制"。

**姿态三:从无到有,加一个 LevelDB 没有的架构(最重)**。LevelDB 根本没这个东西,RocksDB 加了,而且加的是架构级的东西。最典型的是 **Column Family**(P5-19)——LevelDB 一个 DB 一个逻辑库,RocksDB 加了 CF 架构,让一个实例扛多个逻辑库。**Transaction**(P6-21)、**BlobDB**(P6-22)、**WriteController 反压**(P5-17)、**Rate Limiter**(P5-18)都是这一类——LevelDB 没有事务、没有 value 分离、没有反压、没有限速。这类旋钮的本质是**"LevelDB 假设你不需要,RocksDB 发现工业场景需要,于是加了一个新机制"**。

> **钉死这件事**:三种姿态——常数变参数、单策略变多选、无中生有加架构——对应三种不同的"打开"力度。力度越大,改动越深(CF 是动整个引擎架构),但解决的工作负载越极端(没 CF,TiKV 落不了地)。**本书 22 章的难度排序,基本就是这三种姿态的力度排序**。

### 2.2 三笔放大上的账:每个旋钮动了哪个点

光列旋钮不够,还得看每个旋钮在读写放大三角上动了哪个点。把上面几十个旋钮归归类,按"它主要压哪一笔放大、付出哪一笔"分:

```
                  写放大(write amp)
                  ╱╲
                 ╱  ╲   ← 压写放大的旋钮(往左下挪):
                ╱    ╲     Universal Compaction / 大 MemTable / 宽 L0
               ╱  LSM ╲     target_file_size 大 / 减层数
              ╱  在这  ╲
             ╱  个三角 ╲   ← 压读放大的旋钮(往右上挪):
            ╱   里挪点 ╲     Level Compaction / 压层数 / 大 Block Cache
           ╱            ╲    Bloom/Ribbon 密 / Partitioned index+pin
          ╱______________╲
     读放大              空间放大
   (read amp)          (space amp)
                       ← 压空间放大的旋钮(往右下挪):
                          勤 Compaction / 尽早丢旧版本 /
                          CompactionFilter / BlobDB(对大 value)
```

| 你想压哪一笔 | 主要拧这些旋钮 | 付出哪一笔 |
|---|---|---|
| **写放大小**(SSD 寿命、写吞吐) | Universal Compaction、放大 MemTable(`write_buffer_size`↑)、减层数(`num_levels`↓)、大 `target_file_size_base` | 读放大↑、空间放大↑ |
| **读放大小**(点查延迟、范围扫) | Level Compaction、压层数、大 Block Cache、Partitioned index+filter、Ribbon 密、`row_cache` | 写放大↑、内存↑ |
| **空间放大小**(省盘) | 勤 Compaction(调小 trigger)、CompactionFilter 丢旧版本、BlobDB 分离大 value | 写放大↑ |
| **大 value 不拖累** | BlobDB 开启、`blob_file_size` 调 | 多一套 blob 管理 + GC |

> **LevelDB 是写死的,RocksDB 打开成了旋钮**:LevelDB 替你在这三角上选了一个点(偏读放大小、写放大中等、空间放大靠 Compaction 收敛),然后焊死。RocksDB 把这几十个旋钮都交给你,让你按 workload 在三角上挪点。**这就是全书一切设计的总开关**。

## 三、RocksDB 在工业界怎么落地:反哺 TiKV / MySQL / Kafka

讲清账了,但这些旋钮在真实系统里真的有人拧、有人用吗?这一节是本书最重要的反哺——尤其反哺《TiKV》。讲法诚实:**讲框架,不编集成细节**。涉及具体集成版本/配置的细节,我点到为止,你要深究去读对应系统的源码。

### 3.1 TiKV:单机引擎就是 RocksDB(★本书最强反哺)

> **钉死这件事**:TiKV 是分布式 KV 数据库,它的**单机存储引擎就是 RocksDB**——在 TiKV 源码里叫 `engine_rocks`(实现 `engine_traits` trait)。你在《TiKV》那本书里看到的"存储层",拆到底就是 RocksDB。本书讲透 RocksDB 后,你回看 TiKV 的存储层会有"原来如此"。

TiKV 怎么用 RocksDB?框架层面的几个事实(诚实讲,不展开 TiKV 内部调度细节):

**① 三个 Column Family 划分**。TiKV 把一个 Region 的数据按语义分到三个 CF:

- **default CF**:存真正的 KV 数据(key 是 `data_key`,value 是 MVCC 多版本数据)。
- **lock CF**:存 Percolator 事务的锁(2PC 的 prewrite 阶段写锁)。
- **write CF**:存 MVCC 的 commit 记录(每个 key 最新提交的时间戳和指针)。

这三个 CF **共享一个 RocksDB 实例**(共享 WAL、Compaction 统一调度),又**独立配置 Options**(写极多的 default CF 和要快速查的 lock/write CF 各自调 MemTable、Compaction 策略)。这正是本书 P5-19 讲的"Column Family 共享 WAL 又隔离数据"在工业界的真实落地——**如果没有 CF,TiKV 要么开三个 RocksDB 实例(WAL/Compaction/内存各管各,资源浪费),要么把三种数据混在一个 LSM 里(没法分别调优)**。

**② RocksDB 的 Snapshot 支撑 TiKV 的 MVCC**。TiKV 用 Percolator 模型做多版本事务,读请求需要一个一致的视图。TiKV 在 RocksDB 上拿一个 `Snapshot`(基于 SequenceNumber),所有读都走这个 snapshot——这就是本书 P6-20 讲的"Snapshot MVCC"。RocksDB 的 snapshot 几乎零开销(只是把一个 seq 登记进链表),这让 TiKV 的 MVCC 读很便宜。

**③ RocksDB 的调优直接影响 TiKV 的性能**。TiKV 的写延迟、Compaction 抖动、空间放大,本质上都是 RocksDB 的旋钮在动。TiKV 的 `engine_rocks` 把 RocksDB 的 Options 暴露成 TiKV 配置,运维调 TiKV 时调的 `rocksdb.write_buffer_size`、`rocksdb.max_bytes_for_level_base`、`rocksdb.defaultcf.compression`,拧的都是本书讲的旋钮。**本书讲的每一个旋钮,在 TiKV 生产环境里都可能被拧过**。

**④ Region 分裂与 RocksDB 的关系**。TiKV 的 Region 默认 256MB(8.3.0+),分裂时产生新 Region。每个 TiKV 节点上,所有 Region 的数据共用少数几个 RocksDB 实例(按 CF 划分),Region 分裂是逻辑层面的 key range 切分,在 RocksDB 层面只是 CompactRange 的边界变了——不是每 Region 一个 RocksDB 实例(那样实例爆炸)。

> **不这样会怎样**:如果 TiKV 自己从零写一个 LSM,它要重新趟过 RocksDB 十年踩过的所有正确性坑(无锁并发、Manifest 恢复、Compaction 边界、BlobDB、Transaction),还要自己实现 SSD 海量写的调优旋钮。**用 RocksDB,TiKV 把"单机存储"这件事外包给了一个十年验证的引擎,自己专注分布式(Raft、事务、调度)**。这正是"为什么分布式数据库的单机引擎几乎都选 RocksDB"的根——它在读写放大三角上的可调性,让上层系统不用关心单机 LSM,只关心业务。

### 3.2 MySQL:MyRocks,把 RocksDB 当存储引擎替代 InnoDB

MySQL 的存储引擎是可插拔的(InnoDB / MyISAM / Memory……)。**MyRocks 是把 RocksDB 作为一个 MySQL 存储引擎**,替代 InnoDB。这是 Facebook(现 Meta)在 2010 年代中后期主推的——他们的 MySQL 数据库规模巨大,InnoDB(B+树)的写放大大(随机写)在 SSD 海量写场景下吃 SSD 寿命,RocksDB(LSM 顺序写)的写放大小更划算。

框架层面的几个事实(诚实讲,MyRocks 的现状有变,细节去查 Meta/Percona 的资料):

**① MyRocks 的历史与现状**。Facebook 内部大规模用 MyRocks(UDB,用户数据库),写放大从 InnoDB 的 ~10x 降到 ~3x,省了海量 SSD。开源社区 Percona 也维护了 Percona MyRocks。但**近年(MySQL 8.x / 9.x 时代)MyRocks 在主线 MySQL 的维护有所收缩**——MySQL 官方主推 InnoDB,MyRocks 更多作为特定场景(写密集、要省 SSD)的可选引擎存在,不是默认。

**② MyRocks 怎么把 MySQL 的语义映射到 RocksDB**。MySQL 的行(row)→ RocksDB 的 KV(MyRocks 用主键索引组织,二级索引单独存),MySQL 的事务 → RocksDB 的 Transaction(pessimistic,加锁),MySQL 的 MVCC → RocksDB 的 Snapshot。**MyRocks 本质上是"在 RocksDB 之上,重建 MySQL 的关系语义"**。

**②'. 为什么 InnoDB(B+树)写放大大,LSM(RocksDB)写放大小**。这是 MyRocks 存在的根,值得展开。InnoDB 是 B+树聚簇索引,一次 `UPDATE` 要**找到那个数据页(随机读)、在页内原地改、再把整个脏页写回(doublewrite 还要写两份)**——一次逻辑写,实际随机写好几页,写放大常达 10x 以上,SSD 寿命吃得很凶。RocksDB 是 LSM,**一次 `Put` 只追加**(WAL 一次 + MemTable),顺序写;旧版本由后台 Compaction 异步收敛,一个 key 被重写的次数 ≈ 层数(Level Compaction 7~8 遍,Universal 更少)。**顺序写 + 后台收敛,让 LSM 的写放大在写密集场景显著低于 B+树**——这正是 Facebook 愿意花大力气做 MyRocks 的经济账:海量 UDB 写入,SSD 省下的钱抵得过维护一个新引擎的成本。代价是 LSM 的读放大(多层归并)比 B+树(一次树根到叶)大,所以 MyRocks 读密集场景不如 InnoDB——**天下没有免费的午餐,只是把放大从写挪到了读**。

**③ 对照本书的旋钮**。MyRocks 调的就是本书讲的那些:`write_buffer_size`、`max_bytes_for_level_base`、`compaction_style`(MyRocks 大量用 Universal 降写放大)、BlobDB(大字段分离)。你在本书学的每个旋钮,在 MyRocks 运维里都用得上。

> **钉死这件事**:MyRocks 是"RocksDB 替代 InnoDB"的工业级实证——它证明 LSM 的可调性,在写密集的关系数据库场景里,比 B+树(InnoDB)更省 SSD。代价是 MyRocks 要自己重建 MySQL 的复杂语义(事务隔离、二级索引、外键等),维护成本高,这也是它没能取代 InnoDB 成默认的原因之一。

### 3.3 Kafka Streams 与其他:嵌入式状态存储

Kafka Streams(以及 Apache Flink 的 RocksDB state backend)把 RocksDB 用作**嵌入式状态存储**(state store)。流处理需要在本地持久化状态(窗口聚合、join 状态、物化视图),RocksDB 作为嵌入式 KV 库正好——比 hash 表省内存(冷数据落盘),比纯文件随机读写快(LSM 的读路径有 Cache + Bloom)。

框架层面:

- **Kafka Streams 的 state store** 底层是 RocksDB(`RocksDBStore`),key-value 状态、窗口状态都存在本地 RocksDB 实例。
- **Flink 的 RocksDBStateBackend** 同理,把算子状态存 RocksDB。
- **其他**:Cassandra 内部不是直接用 RocksDB(它有自研的 LSM,但思路相通),但很多时序/图数据库(GraphBase、ArangoDB 部分引擎)、ZooKeeper 替代品(如某些 etcd-like 系统选 RocksDB)都用了 RocksDB。

共同点:**这些系统把"本地持久化状态"这件事交给 RocksDB,自己专注业务/分布式**。和 TiKV 的逻辑一模一样——**单机 LSM 用现成的工业级轮子,分布式/语义层自己写**。

### 3.4 反哺小结:为什么"工业级分布式系统的单机引擎几乎都是 RocksDB"

把上面三个例子合起来看,有个共同的洞察:

> **钉死这件事**:RocksDB 之所以成为"工业级分布式系统的单机引擎事实标准",不是因为它最快,而是因为它**在读写放大三角上的可调性最强**——每个上层系统(TiKV / MySQL / Kafka Streams / Flink)的 workload 不同,需要的平衡点不同,RocksDB 的几十个旋钮让每个系统都能把引擎挪到自己想要的点。**不可调的引擎(哪怕单点性能更好)反而落不了地**,因为上层系统的 workload 千差万别,一个焊死的平衡点满足不了所有人。

这是本书反哺《TiKV》的核心:你回看 TiKV 的 `engine_rocks`,会发现它不是一个"随便选的 KV 库",而是**TiKV 的整个存储层调优能力,都建立在 RocksDB 的可调旋钮之上**。本书讲的每个旋钮,都是 TiKV 运维手里的扳手。

## 四、可调性的代价:旋钮多为什么是负担

讲完了可调的好处和工业落地,必须诚实讲它的代价——这是 P0-01 立的"代价"那节的收口。

### 4.1 代价一:Options 组合爆炸

RocksDB 有**几百个 Options**。你在 `options.h` / `advanced_options.h` 里能数到几百个字段。它们不是独立的,而是**互相耦合**:

- 你加大 `write_buffer_size` → Flush 频率↓ → L0 文件产生慢 → `level0_file_num_compaction_trigger` 更久才触发 → L0→L1 Compaction 间隔拉长 → 可能撞 `level0_slowdown_writes_trigger`(L0 堆太多)反压。
- 你换 Universal Compaction → `max_bytes_for_level_base` / `max_bytes_for_level_multiplier` 的语义变了(Universal 不按层大小倍数)→ Cache 的命中率曲线变了 → 读延迟模型完全不同。
- 你开 BlobDB → 大 value 不进 LSM → Compaction 重写的字节数↓ → 写放大的计算公式要改 → 但 blob 文件有自己的 GC,空间放大要算两份。

> **不这样会怎样**:这种组合爆炸的直接后果是——**没人能凭直觉调好 RocksDB**。官方文档自己都写"Tuning is an art"。一个新手面对几百个旋钮,要么全用默认(可能不适合 workload),要么乱调(可能比默认还差)。这也是为什么 RocksDB 提供 `Options::IncreaseParallelism()`、`Options::OptimizeForSmallDb()` 这类"一键预设"——面对几百个旋钮,绝大多数用户只想说"我是个普通 SSD 单机 KV",让 RocksDB 替你把旋钮拧到一组合理的默认值。

### 4.2 代价二:每个旋钮都假设你懂底层

更要命的是,每个旋钮的文档说明,**都假设你懂它背后的 LSM 机制**。举几个本书讲过的:

- `level0_slowdown_writes_trigger`(默认 20):你得懂"L0 文件数过多 → 读要扫多个 L0 文件 → 读放大↑ → 所以 L0 堆到 20 个要反压 delay 写"(P5-17),才能调它。
- `max_bytes_for_level_multiplier`(默认 10):你得懂 Level Compaction 的"每层是上一层 10 倍"规则(P4-14),才能调它。
- `compaction_options_universal`:你得懂 Universal 的"size ratio / age 触发"(P4-15),才能配它。
- `cache_index_and_filter_blocks`:你得懂 Partitioned index 的"大索引分页不一次载入"(P2-07),才会知道为什么要开它。

> **钉死这件事**:RocksDB 把"调的能力"给了你,代价是**它要求你懂 LSM 的每一个机制**。这正是本书存在的理由——**不是教你调哪个参数**(那是手册的事),而是**把每个旋钮背后的机制讲到源码级**,让你知道拧它会发生什么、为什么。这本书 22 章,就是 22 个"为什么这个旋钮存在、它动哪个放大、为什么 sound"的拆解。

### 4.3 代价三:可观测与调参的循环成本

旋钮多,意味着出问题时要排查的维度多。线上 RocksDB 写延迟突增,可能是:

- L0 文件堆太多(查 `level0_slowdown_writes_trigger` 是不是触发了 delay)。
- Compaction 跟不上(查 `soft_pending_compaction_bytes_limit` 是不是触发了 stall)。
- Rate Limiter 限太狠(查 `rate_limiter` 的 `bytes_per_second`)。
- Block Cache 太小(查 cache hit rate,读放大变大拖慢前台)。
- ……(几十种可能)

每个可能都对应一组旋钮和一组 Stats。**调参不是一次性的事,是"压测 → 看 Stats → 调旋钮 → 再压测"的循环**。本书 P6-22 讲的 Stats / PerfContext / Trace,以及附录 B 的调优方法论,就是帮你缩短这个循环的工具。

举一个真实的调参循环例子,让你感受这个代价有多实在:假设线上写延迟突增,你打开 Stats 发现 `WRITE_STALL` 和 `WRITE_DELAY` 计数都在涨。你推断是反压触发了。接下来要定位是哪个 trigger 撞的——是 `level0_slowdown_writes_trigger`(L0 堆太多)?是 `soft_pending_compaction_bytes_limit`(后台 Compaction 积压)?两者排查方向完全不同:前者要看 L0 文件数(`rocksdb.num-files-at-level0`),后者要看 `compaction-pending-bytes`。如果是前者,你还要往下追:为什么 L0 堆多了?是 Flush 太慢(查 `write_buffer_size` 是不是太大导致单次 Flush 久)?是 Compaction 来不及把 L0 压到 L1(查 Compaction 线程数 `max_background_compactions`、Rate Limiter 是不是限太狠)?是 L0→L1 的 `max_bytes_for_level_base` 设小了导致 L1 一直满?每一层追问都指向一个旋钮,而调这个旋钮可能又引发别的问题(比如放开 Rate Limiter → Compaction 吃 IO → 前台读延迟抖)。**一个"写延迟突增"的排查,可能要在七八个旋钮之间来回试,花几天才收敛**。这就是可调性的真实成本——不是"读文档调个参数"那么轻巧。

### 4.4 RocksDB 怎么降低这个代价

RocksDB 自己也意识到可调性的代价,这些年一直在努力降低:

- **预设方法**:`IncreaseParallelism` / `OptimizeForSmallDb` / `OptimizeForPointLookup` / `OptimizeLevelStyleCompaction` / `OptimizeUniversalStyleCompaction`,一键拧一组旋钮。
- **自适应**:**动态 level bytes**(`level_compaction_dynamic_level_bytes`,根据实际数据量自动调每层大小,不用手算 `max_bytes_for_level_base`)、自适应 `max_write_buffer_number`。
- **更好的默认值**:RocksDB 这些年不断提高默认值的合理性(比如默认 `write_buffer_size` 从早期到 64MB,`format_version` 推进到 6),让新手用默认就能跑得不错。
- **可观测增强**:Stats / PerfContext / Trace / Perf Context(本书 P6-22)越来越细。

> **所以这样设计**:RocksDB 的方向是"**底层可调性拉满,上层易用性渐进**"——旋钮一个不撤(工业级用户要),但预设和自适应让普通用户不用碰旋钮也能跑好。这是"可调"与"易用"的工程平衡。

## 五、RocksDB 演进展望:这十年往哪走

诚实讲,RocksDB 这十年(LevelDB 2011 fork,2012 至今约 13 年)的工业级演进,有几条主线。讲框架不编细节(具体版本号/年份记不准的,我只讲趋势):

### 5.1 演进主线一:在三角上增加可点的位置

RocksDB 的核心价值是"读写放大三角上的可调性"。这些年的演进,大部分是在**三角上增加更多可点的位置**——给新的 workload 一个旋钮组合。**为什么这是主线?**因为工业界的 workload 在不断分化:SSD 海量写需要写放小的点(TiKV)、时序数据需要 FIFO 的点、大 value workload 需要 BlobDB 的点、高并发读需要无锁 cache 的点……每种新 workload 出现,RocksDB 就在三角上给它加一个可点的位置。**演进的驱动力是 workload 的分化,不是技术的猎奇**。

- **Ribbon Filter**(本书 P2-08):比 Bloom 省 ~30% 内存。这让"读放大小"的点(密 Bloom)成本更低,三角右上角的点更便宜。源码在 `util/ribbon_impl.h`、`util/ribbon_alg.h`。
- **Universal Compaction**(P4-15):给"写放大小"的 workload 一个点(三角左下角)。
- **FIFO + CompactionFilter**(P4-16):给"时序数据只留最新"的点。
- **BlobDB**(P6-22):给"大 value workload"一个点(大 value 不拖累 LSM)。
- **HyperClockCache**(`cache/clock_cache.h`):给"高并发读"的点(cache 无锁,吞吐↑)。

> **钉死这件事**:RocksDB 的演进不是"换一个更好的 LSM",而是**在同一个 LSM 上,给更多 workload 类型提供可点的位置**。每个新特性(Ribbon / BlobDB / HyperClockCache),都是三角上一个新点位的"插件"。这印证了 P0-01 那句——**RocksDB 是把固定点变成可调曲线,这条曲线上能点的位置越来越多**。

### 5.2 演进主线二:把"调"做得更省心

第二条主线是降低可调性的代价(呼应第四节)。**为什么这是主线?**因为第四节讲的代价(组合爆炸、调参难、假设懂底层)正在劝退用户——如果 RocksDB 难到没人调得动,它的可调性就变成了"纸面优势"。所以 RocksDB 这些年一直在"可调性拉满"和"易用性渐进"之间找平衡:**底层一个旋钮都不撤,但上层加预设、加自适应、加更好的默认值,让不懂 LSM 的用户也能跑好**。这是"专业用户拉满 + 普通用户省心"的双层设计。

- **动态 level bytes**(`level_compaction_dynamic_level_bytes`):不用手算每层大小,RocksDB 按实际数据量自动调。
- **更好的默认值**:这些默认值在 `DEFAULT_OPTIONS_HISTORY.md` 里有变更记录,RocksDB 不断把"对大多数 workload 更优"的值设为默认。
- **Integrated BlobDB**:早期 BlobDB 是 Experimental(`utilities/blob_db/`),近年迁进主路径(`db/blob/`),成为 Integrated BlobDB,默认可用,不用单独开。
- **自适应反压**:WriteController 的 stall/delay 阈值,近年有基于实际负载的自适应调整。

### 5.3 演进主线三:正确性与新特性的扩展

第三条主线是补 LevelDB 没有的能力。**为什么这是主线?**因为 LevelDB 的设计假设是"单机、中等负载、简单 KV",而工业场景要的事务、MVCC、用户时间戳、远程 Compaction,LevelDB 全没有。上层系统(TiKV / MySQL)要这些能力,要么自己在 RocksDB 之上重建(成本高、易错),要么 RocksDB 直接内置。RocksDB 选了内置——**把"工业场景普遍需要的能力"做进引擎,减少上层系统的重复造轮子**。这是从"KV 库"向"工业级存储引擎"的定位升级。

- **User Defined Timestamp**(用户自定义时间戳):近年加的,允许用户给每个 key 带自定义时间戳,做更灵活的 MVCC(不止 RocksDB 内部的 SequenceNumber)。`include/rocksdb/` 多个头文件支持。这对分布式系统(需要物理时间戳的 MVCC)很有用。
- **Transaction / Optimistic / Pessimistic**(P6-21):跨 key 原子写,LevelDB 没有。
- **Compaction Service**:把 Compaction 卸载到远程节点(近年加的,适合计算存储分离)。
- **更细的 Cache 角色**(`cache_entry_roles`):data / index / filter / write-buffer 各自配额,精细化 cache 管理。

### 5.4 演进时间线(诚实示意,年份为大致)

```
2011  LevelDB 发布(Google)
       │
2012  RocksDB 从 LevelDB fork(Facebook)
       │  ── 加并发(InlineSkipList)、WriteGroup
2014~ ── Column Family(核心架构演进)
       │  ── Universal / FIFO Compaction
2016~ ── Transaction、BlobDB(早期 experimental)
       │  ── Partitioned index/filter
2018~ ── Ribbon Filter(替/补 Bloom)
       │  ── 动态 level bytes、subcompaction
2020~ ── Integrated BlobDB(从 experimental 进主路径)
       │  ── HyperClockCache(无锁 cache)
2022~ ── User Defined Timestamp
       │  ── Compaction Service(远程 Compaction)
2024+ ── 持续演进(format_version 6、自适应、易用性)
       │
现在  RocksDB 11.6.0(本书版本,commit 8ba4204b)
```

> **诚实声明**:上图年份是大致,具体某特性哪年进 main、哪年 GA,以 RocksDB 的 `HISTORY.md` / `DEFAULT_OPTIONS_HISTORY.md` 为准。本书讲的是"趋势与方向",不是"某特性精确时间表"。

> **钉死这件事**:RocksDB 这十年的演进,印证了 P0-01 的总开关——**它没有换一套更好的 LSM,而是把 LevelDB 的焊点一个个拆成旋钮,并且在三角上不断增加新的可点位置,同时把"调"做得更省心**。这条演进路径,本身就是"可调性"哲学的胜利。

## 六、技巧精解:两个收束级洞察

本章是收束章,技巧密度比正文章低,但有两个值得钉死的收束级洞察。

### 洞察一:为什么"拆焊点"比"重写"聪明——继承正确性的杠杆

RocksDB 是 LevelDB 的 fork,核心 LSM 结构(MemTable/SST/WAL/Compaction)和 LevelDB 一脉相承。这不是偶然,是**深思熟虑的工程选择**:

- **继承十年验证的正确性**:LevelDB 的无锁读 sound(`std::atomic` + release/acquire)、Manifest 恢复正确性、Compaction 边界处理、Snapshot MVCC——这些都是 Google 内部 + 外部多年验证过的。RocksDB fork 它,白拿了这些正确性。
- **只改"可调"的部分**:RocksDB 的演进,基本是"在 LevelDB 已 sound 的机制上,加旋钮、加并发、加新策略",而不是推翻核心数据结构。InlineSkipList 承 SkipList(并发化)、WriteGroup 承 leader-follower 写组(批链接)、三种 Compaction 承 LevelDB 那一种(加 Universal/FIFO)——都是"加法",不是"重写"。

> **不这样会怎样**:如果 RocksDB 选"重写一个更好的 LSM"(比如换一种全新的数据结构),它要重新趟过 LevelDB 踩过的所有正确性坑——无锁读的 memory order、Manifest 的崩溃恢复、Compaction 的并发安全——任何一个错了都是数据丢失。RocksDB 的明智在于:**LSM 的机制本身没问题,有问题的是"焊死"——那就只拆焊点,不动机制**。这让它既继承了正确性,又获得了灵活性。这是"演进优于革命"在系统设计里的经典案例。

### 洞察二:Column Family 是 RocksDB 最成功的架构演进

旋钮那么多,但有一个旋钮不是"调参数",而是"重构架构"——**Column Family**(P5-19)。它是 RocksDB 相对 LevelDB 最成功的架构演进,理由:

- **LevelDB 没有 CF**:一个 DB 一个逻辑库。多 workload 要开多个 DB 实例,WAL/Compaction/内存各管各。
- **RocksDB 加 CF**:多个逻辑库共享一个引擎实例(共享 WAL/MANIFEST,Compaction 统一调度),又独立(MemTable/SST/Options 各自)。源码核心是 `db/column_family.h:299` 的 `ColumnFamilyData` 类,管理每个 CF 的独立状态;`dropped_` 字段(`db/column_family.h:649`)支持 CF 的 drop(删 CF 不影响别的)。
- **CF 让 TiKV 成为可能**:TiKV 的 default/lock/write 三 CF,就是 CF 架构的工业级实证。**没有 CF,TiKV 要么开三个 RocksDB 实例(资源浪费、调度打架),要么三种数据混一个 LSM(没法分别调优)**。CF 是 TiKV 这类"多 workload 共享引擎"系统能落地的前提。

> **钉死这件事**:Column Family 不是一个"调参旋钮",它是一个"架构旋钮"——它改变了"一个引擎实例能干什么"。本书其他旋钮(write_buffer_size、compaction_style……)是在一个 CF 内调参数,CF 是让你在一个实例里开多个"逻辑库"。**CF 是 RocksDB 从"单库 KV"走向"多 workload 引擎"的关键一跃**。

## 七、章末小结:全书合上

### 回扣主线

本章是全书最后一章,它做了三件事:

1. **总账**:把 22 章的旋钮收成一张表(第二节),每行写清"LevelDB 写死 / RocksDB 打开成什么 / 解决什么 / 代价什么 / 第几章"。这张表是全书的索引。
2. **反哺**:讲清 RocksDB 在 TiKV / MySQL / Kafka 怎么落地——TiKV 单机引擎就是 RocksDB(`engine_rocks`,三 CF),MyRocks 替 InnoDB,Kafka Streams 做 state store。**工业级分布式系统的单机引擎几乎都是 RocksDB**,因为它的可调性最强。
3. **代价与展望**:可调性的代价是组合爆炸 + 调参难 + 假设用户懂底层;RocksDB 的演进展望是在三角上增加可点位置 + 把调做得更省心 + 补正确性扩展。

回到全书的主线——**写路径(WriteBatch→WAL→MemTable→Flush→Compaction,产生写放大)vs 读路径(Cache→Index/Filter→多层归并,产生读放大),Column Family / Snapshot / Transaction / BlobDB 横切两面**。这 22 章,每一章都是这两条路径上的一个驿站,每个驿站都是"LevelDB 写死、RocksDB 打开成旋钮"。

### 五个为什么

1. **为什么要把旋钮收成一张总账?**——因为 22 章的细节容易忘,但"LevelDB 写死的每个决策,RocksDB 都打开了旋钮"这个对应关系是全书的骨架。总账表让你一眼看到骨架,忘了某章翻回去看。
2. **为什么 TiKV 的单机引擎选 RocksDB?**——因为 TiKV 是分布式 KV,单机 LSM 用现成的工业级轮子(RocksDB 的可调旋钮覆盖了 TiKV 各种 workload),自己专注 Raft/事务/调度。RocksDB 的 CF 架构正好支撑 TiKV 的 default/lock/write 三 CF 划分。
3. **为什么 MyRocks 没能取代 InnoDB 成默认?**——因为 MyRocks 要自己重建 MySQL 的复杂语义(事务隔离/二级索引/外键),维护成本高;InnoDB 的 B+树在通用关系数据库场景成熟稳定。MyRocks 更适合"写密集、要省 SSD"的特定场景。
4. **为什么可调性是代价?**——因为几百个 Options 组合爆炸、互相耦合,没人能凭直觉调好;每个旋钮的文档假设你懂底层 LSM 机制。可调性要求用户懂每个机制,这正是本书存在的理由。
5. **为什么 RocksDB 的演进是"拆焊点"而不是"重写"?**——因为 LSM 机制本身没问题,问题在焊死;fork LevelDB 拆焊点,既继承十年验证的正确性,又获得灵活性。这是"演进优于革命"的经典。
