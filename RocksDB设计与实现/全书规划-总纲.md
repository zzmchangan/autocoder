## 一、这本书的定位

### 目标读者

写 Java/Go 服务时把 RocksDB 当黑盒存储用过,调过 `write_buffer_size` / `max_write_buffer_number` / `level0_file_num_compaction_trigger` / block cache,甚至翻过 `facebook/rocksdb` 源码,但**一知半解**:知道有 MemTable、SST、Compaction、Column Family、Write Stall,却讲不清一次 `Put` 从 WriteBatch 到落盘、一次 `Get` 怎么穿透多层 SST、WriteGroup 怎么把并发写攒成一批、三种 Compaction 各解决什么、Column Family 怎么共享 WAL 又隔离数据、Write Stall 在什么阈值触发。和"看过 LevelDB 源码但不理解"、"读过 gRPC 源码但一知半解"是同一种病。

### 全书一句话主旨

> **LevelDB 是"够用就好"的 LSM-tree——它把读写放大钉死在一个固定点;RocksDB 把那个固定点上的每一个设计决策,都做成了可调旋钮,让你在"写快 / 读快 / 省空间"三角里,精确定位到你的 workload 想要的那个点。**

每个组件,都是这个"可调性"的一个旋钮。读完一本书,读者不是"多记了几个 Options",而是能**在脑子里放映出一次 Put/Get 的全过程**:WriteBatch 怎么进 WriteGroup 攒批、WAL 怎么落盘、MemTable(InlineSkipList)怎么并发写、Immutable 怎么 Flush 成 L0 SST、Compaction 怎么按 Level/Universal/FIFO 三种策略把数据逐层下压收敛写放大、Get 怎么靠 Block Cache + Bloom + 多层归并以最小读放大穿透——以及这每一步底下,LevelDB 是写死的、RocksDB 是怎么把它打开成旋钮的。

### 为什么用"读写放大的两条路径"当骨架(而非按模块平铺)

LSM-tree 的本质矛盾是**读写放大**:写快(只追加)就要读慢(多层归并),读快(原地更新)就要写慢(B+树随机写)。RocksDB 整本书都是围绕"把读写放大调到你想要的点"。用"写路径(产生写放大:WriteBatch→WAL→MemTable→Flush→Compaction)vs 读路径(产生读放大:Cache→Index→Filter→多层归并)"这条主轴当骨架,每个模块都是"为了让某条路径继续而必须出现的角色"——读者永远知道**我在哪、为什么需要下一站**。

## 二、写作方法:动机(why) + 技巧(how)双线 ⚠️ 本系列命脉

> **这一节是本书与数学系列、其他源码系列最关键的方法论,所有写作者必读、必守。**

RocksDB 源码里充斥高级技巧:InlineSkipList 的无锁并发插入(承 LevelDB SkipList 但支持多写者)、WriteGroup 的批写合并(leader 写 WAL,folowers 零拷贝 piggyback)、Block-based Table 的 index/filter/data block 分离与分区、Ribbon Filter(比 Bloom 省 ~30%)、双指针环形 Block Cache 的多档 pin、Compaction 的 subcompaction 并发、WriteController 的 stall/delay 反压、Rate Limiter 的令牌桶限速、Column Family 共享 WAL 又隔离 MemTable……用户自己读过源码,"一知半解"的缺口往往**不在**"它要解决什么问题"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书的每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?LevelDB 是怎么写死的、不这样会怎样?
2. **实现技巧(how 的精妙)**:RocksDB 用了什么手段把它做成了可调旋钮?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果像 LevelDB 那样写死、或朴素地写,会撞上什么墙",让妙处显形。

一句话:**讲不清 WriteGroup 批写、三种 Compaction、Write Stall 反压,等于没讲 RocksDB。**

> **⚠️ 深度标杆(用户强调,本书命脉)**:本书篇幅标杆拉到对标《LuaJIT》深入浅出第一性原理版——**每章约 12000~18000 字**。用户原话:"不要吝啬文字和篇幅,内容可以更丰富一点,要把问题讲解清楚,深入浅出,好理解又有深度。"所以每个机制至少讲透三层(动机 → LevelDB 怎么写死的反例 → RocksDB 怎么打开成旋钮 + 为什么 sound),宁可多铺反例、对比、图、调优经验(每种 Compaction 适用什么 workload、Write Stall 各档阈值怎么定),把问题讲到读者"原来如此",也不要一笔带过。与《gRPC》《TiKV》保持一致。

## 三、全书主线、二分法、主比喻、承接

### 主线

> **LevelDB 把读写放大钉死在一个固定点;RocksDB 把这个点上每个写死的决策,都做成了可调旋钮,让你在"写快 / 读快 / 省空间"三角里精确定位到你的 workload。**

任何一处看不懂某个机制,回到这句问:"这是在**写入到落盘的写路径**(产生写放大),还是在**读请求穿透的读路径**(产生读放大)?而这个机制在 LevelDB 里是写死的、RocksDB 把它打开成了什么旋钮?"

### 二分法

本书的二分法:

> **写路径(WriteBatch→WAL→MemTable→Flush→Compaction,产生写放大) vs 读路径(Cache→Index/Filter→多层 SST 归并,产生读放大)。**

- **写路径这一面**:WriteBatch/WriteGroup、WAL、MemTable(InlineSkipList)、Flush、Compaction(三种策略)、Write Stall/Rate Limiter(写路径的反压)——这些决定"写进去有多快、写放大多大、后台怎么收敛"。
- **读路径这一面**:Block Cache、Block-based Table 的 index/filter、Bloom/Ribbon、Get/Iterator 的多层归并——这些决定"读出来有多快、读放大多大"。
- **横切两面**:Column Family(共享 WAL 又隔离 MemTable/SST 的架构)、Snapshot/MVCC、Transaction、BlobDB、可观测(Stats/PerfContext)。

任何时候迷路,回到这个二分法。

### 承接《LevelDB》(本书最大特色,务必遵守)

本书是《LevelDB 设计与实现深入浅出》的"工业级续篇"。**承接铁律**:LevelDB 已讲透的概念(SkipList 无锁读、WAL 7字节 header、SST footer 48字节、MergingIterator 线性扫、Bloom DoubleHashing、Manifest 重放、Snapshot 链表 MVCC、leader-follower 写组),在本书**一句话带过 + 指路 [[leveldb-source-facts]]**,篇幅全留给 RocksDB 独有的演进:

- **InlineSkipList**(承 LevelDB SkipList):LevelDB 的 SkipList 单写者;RocksDB 用 InlineSkipList 支持并发无锁插入。
- **WriteGroup**(承 LevelDB 写组):LevelDB 的 leader-follower 批写;RocksDB 的 WriteThread/WriteGroup 更精细(链接批、零拷贝 piggyback)。
- **三种 Compaction**:LevelDB 只有一种(类 Level);RocksDB 加 Universal(写优化)、FIFO(TTL)。
- **Column Family**:LevelDB 没有;RocksDB 的核心架构演进。
- **可插拔**:MemTable rep、TableFactory、Cache、CompactionFilter 全可插拔(LevelDB 全写死)。

### 反哺《TiKV》

TiKV 的单机引擎就是 RocksDB(`engine_rocks`),写《TiKV》时按承接铁律只能"一句话带过 RocksDB"。本书讲透后,**反哺 TiKV**:读者回看 TiKV 的存储层(CF 划分 default/lock/write、RocksDB 调优、RocksDB 的 snapshot 怎么支撑 TiKV 的 MVCC)会有"原来如此"。本书 P7 收束专设"RocksDB 在 TiKV/MySQL/Kafka 里怎么落地"。

### 主比喻决策:直球为主,比喻点睛(沿用系列)

> **直球为主**:绝大多数地方第一性原理直给,不套比喻。**比喻点睛**:仅在开篇(P0-01)用"调音台"做一次性点位睛——LevelDB 是个旋钮焊死的收音机(音质固定),RocksDB 是把每个旋钮都接出来的调音台(你自己在写快/读快/省空间三档里找平衡)。全书不建立贯穿的重比喻体系,不得喧宾夺主。沿用《LevelDB》《etcd》《gRPC》定调,区别于《Tokio》重比喻。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线是读写两条路径:**写进去怎么落盘收敛(写路径)→ 怎么读出来(读路径)→ 怎么调(可调控面)→ 怎么生产可用(横切)→ 收束**。
> 章号全局连续;命名 `P{篇}-{章}-标题.md`,开篇第 1 章 = `P0-01`,第 1 篇第 2 章 = `P1-02`。

### 第 0 篇 · 开篇:为什么 LevelDB 不够用(定调,1 章)

- **第 1 章 · 第一性原理:为什么 LevelDB 不够用**(`P0-01`):LevelDB 的"够用就好"假设(单机、中等负载、SSD 友好度无所谓、单 CF)在工业场景(SSD 海量写、低延迟读、多 workload 混跑、要 TTL/事务/备份)下撞墙。RocksDB 的回答不是重写,而是**把 LevelDB 写死的每个决策做成可调旋钮**。读写放大三角、可调性的代价(调参难)。承 LevelDB 三件套(只追加/SST/Compaction)一句话带过。点出 RocksDB 的读写两条路径与横切调控。**【样本章节,全书风格标杆】**

### 第 1 篇 · 写路径:从 WriteBatch 到 L0(4 章)

> 写进去怎么秒回、怎么攒批、怎么落盘成 SST。这一篇是写路径地基。

- **第 2 章 · WriteBatch 与写组**(`P1-02`)——一次 Put 怎么进 WriteBatch、怎么被 WriteThread/WriteGroup 攒成一批一次写 WAL?技巧:**WriteGroup 批写合并**(leader 写 WAL,followers 的 batch 链接进去零拷贝 piggyback)、写组并发模型。承 LevelDB leader-follower 写组(LevelDB 一句话带过),讲 RocksDB 的精细(批链接、JoinBatchGroup、AwaitState)。源码:`db/write_thread.cc`、`db/db_impl_write.cc`(Grep 核实路径)。归属:`写路径`。
- **第 3 章 · WAL:落盘与回收**(`P1-03`)——写组怎么落 WAL?log file 怎么复用/回收?sync 策略?技巧:**log file 复用**( recycled_log)、**sync 策略**(kSkip/kNoSync/kAllSync)、`wal_ttl_seconds` / `wal_ttl_mb` 自动裁剪。承 LevelDB WAL 7字节 header + CRC(一句话带过)。源码:`db/log_writer.cc`、`db/db_impl_open.cc`/`db/wal_*`。归属:`写路径`。
- **第 4 章 · MemTable:并发跳表**(`P1-04`)——写进 WAL 同时进 MemTable,MemTable 长什么样?技巧:★**InlineSkipList 无锁并发插入**(承 LevelDB SkipList 单写者,RocksDB 支持多写者无锁插)、**可插拔 MemTableRep**(skiplist/hash_linklist/hash_skiplist/vector)、多 MemTable + Immutable 队列、Prefix Bloom(读路径的早退伏笔)、`write_buffer_manager` 全局内存预算。源码:`memtable/inlineskiplist.h`、`memtable/skiplistrep.cc`、`memtable/write_buffer_manager.cc`、`db/memtable.cc`、`db/memtable_list.cc`。归属:`写路径(招牌)`。
- **第 5 章 · Flush:MemTable 变 SST**(`P1-05`)——MemTable 满了怎么变成 L0 SST?技巧:**Flush 调度**(flush_scheduler/flush_job)、**Pipelined Flush**(写 WAL 与 Flush 流水线)、L0 文件产出、Flush 触发阈值(`write_buffer_size`/`max_write_buffer_number`/`level0_file_num_compaction_trigger` 的关系)。源码:`db/flush_job.cc`、`db/flush_scheduler.cc`、`db/db_impl_write.cc`。归属:`写路径`。

### 第 2 篇 · 静态格式:Block-based Table 与 Manifest(4 章)

> 数据在磁盘上长什么样。这一篇是读路径的地基(SST 看不懂,读路径无从讲起)。

- **第 6 章 · Block-based Table 格式**(`P2-06`)——一个 SST 文件内部怎么分块?技巧:**data block / index block / filter block / metaindex / footer** 的布局(承 LevelDB SST footer 48字节,一句话带过),RocksDB 的丰富(多 index 类型、多 filter、properties block)。源码:`table/block_based/block_based_table_reader.cc`、`block.cc`、`block_builder.cc`、`table/format.cc`。归属:`格式`。
- **第 7 章 · Index / Filter 分离与分区**(`P2-07`)——为什么 index 和 filter 要能从 data block 分离、甚至自己分区?技巧:★**Partitioned index / Partitioned filter**(大索引自己分页,不一次载入)、**data block hash index**(块内 hash 加速)、**cache 角色**(block cache 按 data/index/filter 分角lot,`cache_entry_roles`)。讲清"朴素一个大 index block 会撞什么墙"(大 SST 索引载入慢、挤 cache)。源码:`table/block_based/binary_search_index_reader.cc`、`block_prefix_index.cc`、`data_block_hash_index.cc`、`cache/cache_entry_roles.cc`。归属:`格式(招牌)`。
- **第 8 章 · Bloom 与 Ribbon Filter**(`P2-08`)——point lookup 怎么避免无谓的读 block?技巧:**Bloom Filter**(承 LevelDB DoubleHashing,一句话带过)、★**Ribbon Filter**(比 Bloom 省 ~30% 内存,基于 Ribbon 矩阵的可重建过滤器)。讲清 Ribbon 凭什么省、sound 在哪。源码:`util/bloom.cc`、`util/ribbon.cc`(Grep 核实)、`include/rocksdb/filter_policy.h`。归属:`格式`。
- **第 9 章 · Version Set 与 Manifest**(`P2-09`)——所有 SST 文件的"花名册"怎么维护、怎么持久化?技巧:VersionSet/VersionEdit(承 LevelDB,一句话带过)、★**Column Family 的 MANIFEST 演进**(多 CF 的元数据怎么组织)、Version 引用计数。源码:`db/version_set.cc`、`db/version_edit.cc`、`db/version_builder.cc`、`db/column_family.cc`。归属:`格式`。

### 第 3 篇 · 读路径:从 Get 到多层归并(3 章)

> 一次 Get 怎么穿透 MemTable + 多层 SST 找到最新值,读放大怎么压到最小。

- **第 10 章 · Block Cache**(`P3-10`)——读 block 怎么不每次都碰磁盘?技巧:**分片 LRU**(承 LevelDB ShardedLRUCache,一句话带过)、★**多档 pin**(block cache 的 in-use/high_pri 两档、`row_cache`、`persistent_cache`)、**cache 角色**。源码:`cache/cache.cc`、`cache/cache_entry_roles.cc`、`cache/cache_key.cc`、`cache/lru_cache.h`(Grep 核实)。归属:`读路径`。
- **第 11 章 · Get 路径与读放大**(`P3-11`)——一次 Get 怎么穿透 active memtable → immutable → L0(多文件)→ L1..Ln?技巧:**point lookup 全链路**、**Bloom/Ribbon 早退**、**Index 二分定位**、**max_sequential_skip_in_iterations**。讲清读放大怎么算、怎么压。源码:`db/db_impl.cc` 的 Get、`table/block_based/block_based_table_reader.cc` 的 Get/Get_Entry。归属:`读路径`。
- **第 12 章 · Iterator 与多路归并**(`P3-12`)——范围扫描怎么合并多层 SST?技巧:**MergingIterator**(承 LevelDB 线性扫,一句话带过)、RocksDB 的**合并迭代优化**(heap/IteratorWrapper 缓存)、**Prefix seek**(`prefix_extractor` + `total_order_seek` 取舍)、**Iterator pinning**。源码:`table/merger.cc`、`table/block_based/block_based_table_iterator.cc`、`db/db_iter.cc`。归属:`读路径`。

### 第 4 篇 · Compaction:三种策略与收敛(4 章,全书灵魂招牌)

> 写放大靠 Compaction 收敛。RocksDB 的灵魂是把 LevelDB 那一种 Compaction 做成三种,适配不同 workload。

- **第 13 章 · Compaction 框架**(`P4-13`)——Compaction 怎么调度、怎么执行?技巧:**compaction_job / compaction_picker / compaction_iterator** 三件套、**subcompaction 并发**(一个大 compaction 拆多个子任务并行)、Compaction 时机与触发。源码:`db/compaction_job.cc`、`db/compaction_picker.cc`、`db/compaction_iterator.cc`(Grep 核实确切名)。归属:`写路径(招牌)`。
- **第 14 章 · Level Compaction**(`P4-14`)——默认策略:类 LevelDB 的分层。技巧:**level 大小可调**(`max_bytes_for_level_base`/`max_bytes_for_level_multiplier`)、L0→L1 的特殊处理、与 LevelDB MaxBytesForLevel 的对照(承 LevelDB L0/L1 共享基数,一句话带过)。讲清 Level 适合"读多写少、要读放大小"。源码:`db/compaction_picker_level.cc`(Grep 核实)。归属:`写路径(招牌)`。
- **第 15 章 · Universal Compaction**(`P4-15`)——写优化策略:宽 L0、合并相似大小文件。技巧:**Universal 的合并规则**(trigger by size ratio / age)、为什么 SSD 适合(顺序写、可接受空间放大换写放大小)、`compaction_options_universal`。讲清 Universal 适合"写多读少、能接受空间放大"。源码:`db/compaction_picker_universal.cc`(Grep 核实)。归属:`写路径(招牌)`。
- **第 16 章 · FIFO Compaction 与 CompactionFilter**(`P4-16`)——时序策略:先进先出删旧。技巧:**FIFO 的 TTL 语义**(`ttl` 场景)、**CompactionFilter**(用户在 compaction 时过滤/改 KV,删旧版本/打 TTL 的利器)。讲清 FIFO 适合"时序数据、只要最新的"。源码:`db/compaction_picker_fifo.cc`(Grep 核实)、`include/rocksdb/compaction_filter.h`。归属:`写路径`。

### 第 5 篇 · 可调控面:反压、限速、隔离(3 章)

> 写太快怎么办、后台 IO 怎么不挤占前台、多 workload 怎么隔离。这一篇是"可调性"集中展示。

- **第 17 章 · Write Stall 与 Write Delay**(`P5-17`)——写太快、MemTable/L0 堆积会"淹死"系统,怎么反压?技巧:★**WriteController 的 stall/delay 两档**(L0 文件数 / memtable 数 / level 大小触发,先 delay 拖慢再 stall 拒绝)、各档阈值(`level0_slowdown_writes_trigger`/`level0_stop_writes_trigger`/`soft_pending_compaction_bytes_limit`)。讲清"朴素地不反压会撞什么墙"(OOM / 写延迟雪崩)。承 [[grpc-series-project]] P2-09"不淹不饿"的同类思想。源码:`db/write_controller.cc`、`db/db_impl_write.cc`。归属:`调控(招牌)`。
- **第 18 章 · Rate Limiter 与 IO 调度**(`P5-18`)——Compaction/Flush 的后台 IO 怎么不挤占前台写/读?技巧:**令牌桶限速**(`rate_limiter` 的 bytes_per_second、refill)、**IO 优先级**(前台 user reads/writes vs 后台 compaction/flush 的优先级队列)、`rate_limit_bg` / `tuned_for`。讲清"不限速会撞什么墙"(后台 compaction 抢占前台延迟)。源码:`util/rate_limiter.cc`、`util/rate_limiter_impl.h`。归属:`调控`。
- **第 19 章 · Column Family**(`P5-19`)——多个"逻辑库"怎么共享一条物理引擎实例又互相隔离?技巧:★**共享 WAL 与 MANIFEST、独立 MemTable/SST/Options**(CF 的核心架构)、CF 的 drop(删 CF 不影响别的)、CF 与 Compaction 的关系。讲清"朴素地开多个 DB 实例会撞什么墙"(连接/内存/Compaction 各自为政)。源码:`db/column_family.cc`、`db/db_impl_column_family.cc`。归属:`调控(架构招牌)`。

### 第 6 篇 · 生产可用:快照、事务、Blob、运维(3 章)

- **第 20 章 · Snapshot 与 MVCC**(`P6-20`)——怎么读一个历史时刻的数据?技巧:SequenceNumber/Snapshot(承 LevelDB SnapshotList,一句话带过)、WriteBatch 的 seq 分配、快照与 Compaction 的交互(compaction 不能丢被快照引用的版本)。源码:`db/snapshot_impl.cc`、`db/write_batch.cc`。归属:`横切`。
- **第 21 章 · Transaction 与 MergeOperator**(`P6-21`)——怎么做跨 key 的原子写、怎么读改写?技巧:**Transaction**(optimistic 写时验冲突 / pessimistic 写前加锁)、**MergeOperator**(读时合并,避免读改写的放大)。源码:`utilities/transactions/`(Grep 核实)、`include/rocksdb/merge_operator.h`、`db/merge_helper.cc`。归属:`横切`。
- **第 22 章 · BlobDB 与运维**(`P6-22`)——大 value 怎么不撑爆 LSM?线上怎么备份/观测?技巧:★**BlobDB**(value 分离:大 value 存 blob 文件,LSM 只存指针,扛大 value workload)、**Checkpoint / Backup / Secondary instance**、**CompactRange 手动 compaction**、Stats/PerfContext/Trace。源码:`db/blob/`(blob_file_builder/cache/meta)、`include/rocksdb/utilities/checkpoint.h`、`monitoring/`。归属:`横切(招牌)`。

### 第 7 篇 · 收束(1 章)

- **第 23 章 · 全书收束:可调性的账**(`P7-23`)——把 LevelDB→RocksDB 每个旋钮的演进做成总账(每个旋钮解决什么、代价什么);**RocksDB 在 TiKV/MySQL/Kafka 里的落地**(反哺 [[tikv-series-project]]);可调性的代价(调参难、Options 组合爆炸);RocksDB 演进展望(Ribbon/BlobDB/InlineSkipList 等近年的工业级演进)。归属:`总览`。
