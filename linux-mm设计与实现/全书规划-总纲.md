# 《Linux 内存管理设计与实现深入浅出:内核怎么把物理内存分出去又收回来》—— 全书总纲

> 一本写给"写过 Linux 用户态程序、甚至翻过 `mm/`,却总觉得一知半解"的人的小书。
>
> 本文是全书的**总纲**:定主线、定方法、定分篇分章、定源码策略、定写作约定。
> 与《Tokio》《LevelDB》《etcd》《Go runtime》《内存分配器》同源,都是"**第一性原理 + 真实源码**";命脉同样是**动机(why) + 技巧(how) 双线**;基调延续《LevelDB》——**直球讲透为主,比喻只在反直觉处点睛**。

## 一、这本书的定位

### 目标读者

会写 C、读过操作系统课、用过 `malloc`/`mmap`、甚至翻过 `mm/page_alloc.c`、`mm/slub.c`,但**一知半解**:知道有 buddy、slab、页表、缺页、swap,却讲不清一次 `kmalloc` 怎么拿到内存、一次用户态访问触发缺页后内核怎么建映射、内存紧张时 kswapd 怎么决定回收谁、buddy 和 slab 怎么分工。和"读过 tokio/go runtime 源码但不理解"是同一种病。

### 全书一句话主旨

> **内核怎么管理"物理内存"这片共享资源——把物理页交给 buddy(伙伴系统)按页分配,把小对象交给 slab/slub 在页上切,把用户进程的虚拟地址通过页表映射到物理页,内存紧张时回收/规整/换出。每一层都在"快"和"省"之间找平衡。**

每个组件,都是这条旅程上的一个**驿站**。读完一本书,读者能**在脑子里放映出内核内存管理的全过程**:一次 `kmalloc`(或用户 `malloc`→`brk`/`mmap`→缺页)怎么一步步拿到物理内存、buddy 怎么用 order 拆分合并页、slab 怎么在页上切对象并靠 per-cpu partial 加速、页表怎么把虚拟映射到物理、内存紧张时 kswapd/vmscan 怎么回收——以及每一步底下用了什么巧妙的手段。

### 为什么用"分配出去又收回来"当骨架(而非按文件平铺)

mm/ 子系统有一百多个 `.c`,高度耦合:buddy 管物理页、slab 在 buddy 给的页上切对象、用户 mmap 建虚拟映射靠页表指向物理页、回收(vmscan)又要从页表/rmap 找到页来收回。按文件百科式平铺,读者会觉得每块都是孤岛。用"内存怎么分出去(buddy/slab/mmap)、紧张时怎么收回来(vmscan/compaction/swap)"这条主线当骨架,每个模块都是"为了让这条流走得通"而出现的角色——读者永远知道**我在哪、为什么需要下一站**。

## 二、写作方法

### 命脉:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与一般内核文档、源码导读最关键的分野,所有写作者必读、必守。**

Linux mm 源码里充斥内核级技巧:buddy 的 order 拆分合并(对抗外碎片)、`__alloc_pages` 的快路径(rmqueue)+ 慢路径(水位/直接回收)分级、per-cpu pageset(无锁快路径)、slab/slub 的 freelist + per-cpu partial + `cmpxchg` 无锁 + frozen slab、多级页表(pgd/p4d/pud/pmd/pte)+ `mmu_gather` 批量 TLB 刷新、rmap 反向映射(PTE 反查物理页)、LRU/workingset 的回收启发、compaction 的迁移凑连续。用户自己读过源码,"一知半解"的缺口往往**不在**"它要解决什么"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?(沿用系列一贯的"为什么优先")
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果不用这个技巧、朴素地写,会撞上什么墙",让妙处显形。

一句话:**讲不清技巧,等于没讲 Linux mm。**

### 基调:直球讲透为主,比喻只在反直觉处点睛 ⚠️ 延续《LevelDB》

mm 有一部分反直觉(为什么 buddy 用 order 2^N、为什么 slab 要 per-cpu、为什么回收要 LRU 而非随机),这部分**允许适度比喻点睛**。但 mm 的大部分(buddy 算法、slab 对象布局、页表结构)**具体可见,直球讲清最好**。

- **默认直球**:能用表 / 流程图 / 源码说清的,不上比喻。
- **比喻只点睛**:buddy 可点"像把大块不断对半切、还回来能合回去";slab 可点"像在一个 4KB 托盘里摆满同型号零件"。点到为止。
- **判断标准**:直球讲不清才上比喻。

### ★ 对照《内存分配器》(第 8 本):本书独有特色

> 本书和第 8 本《内存分配器设计与实现深入浅出》是一对——内核 mm 是"内核态分配",tcmalloc/jemalloc 是"用户态分配"。slab 篇对照 tcmalloc 的 thread cache(per-cpu partial vs thread cache)、buddy 对照 tcmalloc 的 span(页级管理),两本合成"内存分配全栈"。

本书在 slab/buddy 等关键章末用轻量"★对照"小栏,和第 8 本的对应物对读;收尾章给总表。

## 三、全书主线、二分法

### 主线

> **内核怎么把物理内存这片共享资源"分出去"(buddy 分页、slab 切对象、mmap 建用户映射),又在紧张时"收回来"(vmscan 回收、compaction 规整、swap 换出、OOM 杀进程)——每一层都在"快"和"省"之间平衡。**

任何一处看不懂某个机制,回到这句问:"这是在**把内存分出去**,还是在**紧张时把内存收回来**,或是在**支撑这两件事(页表/rmap/数据结构)**?"把答案写进"为什么"里。

### 二分法(迷路时回到它)

本书的二分法:

> **分配路径(给内存:buddy 分页、slab 切对象、mmap 建映射) vs 回收路径(紧张时:vmscan 回收、compaction 规整、swap 换出、OOM 杀进程)。**

- **分配路径**:buddy(`__alloc_pages`/`__free_pages`)、slab(`kmem_cache_alloc`/`kmalloc`)、vmalloc、用户地址空间(mmap/VMA/页表/缺页)。这些要"快"——快路径无锁、per-cpu 缓存。
- **回收路径**:watermark/kswapd、LRU/vmscan、compaction、swap/OOM。这些要"稳"和"省"——内存紧张时的兜底。

支撑这两者的地基:物理内存组织(node/zone/pgdat)、页表与 rmap(把虚拟反查到物理)、各种锁与 RCU。

mm 每个机制都落在这两面之一或支撑地基。任何时候迷路,回到这个二分法。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线是一条旅程:**为什么内核管内存 → buddy 怎么分页 → slab 怎么切对象 → vmalloc/percpu → 用户地址空间怎么映射 → 紧张时怎么回收 → 进阶 → 收束**。
> 章号全局连续;命名 `P{篇}-{章}-标题.md`。**全书 21 章**(核心全覆盖,对标已有系列)。

### 第 0 篇 · 开篇:为什么内核要自己管内存(定调,1 章)

- **第 1 章 · 第一性原理:为什么内核要管内存**(`P0-01`):物理内存是稀缺共享资源 + 多进程要隔离 + 硬件 MMU/页表是地基 → 内核必须管分配、管映射、管回收 → mm 全貌(buddy/slab/页表/回收)→ 立起"分配 vs 回收"二分法 + ★对照第 8 本(用户态 vs 内核态)。**【样本章节】**

### 第 1 篇 · buddy 伙伴系统:分配·物理页(5 章)⚠️ 核心重头戏之一

> 源码主体 `mm/page_alloc.c`(4000+ 行)、`include/linux/mmzone.h`、`include/linux/gfp.h`。建议顺序读。

- **第 2 章 · 物理内存模型 + node/zone/pgdat**(`P1-02`)——物理页怎么组织(`struct page`/`folio`)、NUMA 的 `pg_data_t`、zone(DMA/NORMAL/HIGH/MOVABLE)、watermark 字段。源码:`page_alloc.c`、`mmzone.h`。**技巧精解:`struct page` 的紧凑布局**(一个页几十字节,几百万页如何省内存)+ zone 为什么这么分。
- **第 3 章 · buddy 算法:order、free_area、拆分合并**(`P1-03`)——伙伴系统核心:按 order(2^N 页)管理空闲、`free_area[]`、分配时拆分、释放时合并伙伴;对抗外碎片。源码:`page_alloc.c`(`__rmqueue`/`__free_one_page`)。**技巧精解:buddy 的合并**(为什么按 2 的幂——O(1) 找伙伴 + 合并抗碎片;反面对比"按字节分配"会怎样)。
- **第 4 章 · 分配路径 __alloc_pages:快路径与慢路径**(`P1-04`)——一次页分配的全流程:`alloc_pages`→`__alloc_pages_nodemask`→快路径(rmqueue,水位满足)→慢路径(水位不足时唤醒 kswapd/直接回收/compaction)。GFP 标志怎么影响分配。源码:`page_alloc.c`(`get_page_from_free_area`/`__alloc_pages_slowpath`)。**技巧精解:快慢路径分级 + watermark**(快路径无锁-ish、慢路径兜底;反面对比"无水印直接分配"会 OOM 抖动)。
- **第 5 章 · 释放路径 __free_pages + per-cpu pageset**(`P1-05`)——释放怎么合伙伴、per-cpu pageset(pcp)缓存热页避免锁竞争。源码:`page_alloc.c`(`free_unref_page`/`__free_one_page`)、`zone` 的 `pageset`。**技巧精解:per-cpu pageset**(每 CPU 一个热页缓存,释放/小量分配走 pcp 无锁;反面对比"每次释放都锁 zone"会怎样)。
- **第 6 章 · migrate types、pageblock 与 CMA**(`P1-06`)——页按可迁移性分类型(MOVABLE/UNMOVABLE/RECLAIMABLE)、pageblock、CMA(连续内存分配,给设备大块)。源码:`page_alloc.c`、`page_isolation.c`、`cma.c`。**技巧精解:migrate types 抗碎片**(把会动的和不会动的分开,给大块连续留余地;反面对比"不分类型"会碎片化到无法分配大页)。

### 第 2 篇 · slab/slub:分配·小对象(4 章)⚠️ 核心重头戏之二 ★对照第8本

> 源码 `mm/slub.c`(默认 SLUB)、`mm/slab_common.c`(kmalloc)、`mm/slab.h`。这一篇和第 8 本《内存分配器》强对照。

- **第 7 章 · kmem_cache、对象布局、freelist**(`P2-07`)——slab 的核心抽象:`kmem_cache`(一种固定大小的对象池)、一个 slab(page 上)怎么摆满同型号对象 + freelist、`kmem_cache_create`。源码:`slub.c`(`kmem_cache_alloc`/`__slab_alloc`)、`slab.h`。**技巧精解:对象布局 + freelist**(同型号对象紧凑摆放、空闲对象用 freelist 串;反面对比"通用 buddy 分配小对象"会碎到爆)。
- **第 8 章 · alloc/free 路径 + per-cpu partial(快路径)**(`P2-08`)——SLUB 的快路径:per-cpu 的"当前 slab"(frozen)+ partial 队列,`cmpxchg` 无锁分配;慢路径回 per-node partial。源码:`slub.c`(`___slab_alloc`/`__slab_free`)。**技巧精解:per-cpu frozen slab + cmpxchg 无锁**(每 CPU 一个当前 slab,分配几乎无锁;反面对比"每次分配锁 node"会怎样)。
- **第 9 章 · kmalloc:size class 与全局 cache**(`P2-09`)——`kmalloc` 怎么按大小分到一组全局 `kmalloc-<size>` cache、`kmalloc_caches`、`kmem_cache_zalloc`。源码:`slab_common.c`(`kmalloc`/`create_kmalloc_caches`)。**技巧精解:size class**(把任意大小归并到有限的几个 cache,平衡碎片与命中)。
- **第 10 章 · ★对照第 8 本:slab vs tcmalloc**(`P2-10`)——内核 slab 和用户态 tcmalloc/jemalloc 的对照:per-cpu partial vs thread cache、slab(page 上切)vs span/中央堆、内核无 `sbrk`(直接 buddy)。**对照小栏 + 总表**。

### 第 3 篇 · vmalloc 与 percpu:虚拟连续(1 章)

- **第 11 章 · vmalloc + percpu**(`P3-11`)——vmalloc(虚拟地址连续、物理页离散,靠页表缝起来,给内核大块虚拟连续区)、percpu(per-CPU 数据,每 CPU 一份,无锁访问)。源码:`vmalloc.c`、`percpu.c`。**技巧精解:vmalloc 的取舍**(虚拟连续但慢(改页表 + TLB),只在需要大块连续时用;percpu 的无锁 per-CPU 副本)。

### 第 4 篇 · 用户地址空间:进程内存(4 章)

> 用户 `malloc`→`brk`/`mmap`→访问触发缺页→内核建映射。这一篇讲进程地址空间。

- **第 12 章 · VMA 与 mmap**(`P4-12`)——进程地址空间怎么组织(`mm_struct` + VMA 红黑树/ maple tree)、`mmap` 怎么建一个虚拟区间(还不动物理页)。源码:`mmap.c`(`do_mmap`)、`vma.c`。**技巧精解:VMA 抽象 + maple tree**(虚拟区间惰性分配,访问才给物理页)。
- **第 13 章 · 多级页表与 mmu_gather**(`P4-13`)——x86 多级页表(pgd/p4d/pud/pmd/pte)、为什么多级(省内存)、`mmu_gather` 批量收集 TLB 刷新。源码:`memory.c`、`pgtable-generic.c`、`mmu_gather.c`、`include/asm/pgtable.h`。**技巧精解:多级页表**(稀疏地址空间用多级省内存 + mmu_gather 批量刷 TLB 抗抖动)。
- **第 14 章 · 缺页中断:从虚拟到物理**(`P4-14`)——访问一个还没映射的虚拟地址,触发缺页,`handle_mm_fault`→分配物理页→建页表项;匿名页/文件页/写时复制 COW。源码:`memory.c`(`handle_mm_fault`/`do_anonymous_page`/`do_wp_page`)。**技巧精解:惰性分配 + COW**(访问才给页;fork 的页共享到写时才复制)。
- **第 15 章 · rmap 反向映射 + gup**(`P4-15`)——rmap(从物理页反查所有映射它的 PTE,回收/迁移要用)、`get_user_pages`(内核钉住用户页,给 DMA/IO)。源码:`rmap.c`、`gup.c`。**技巧精解:rmap**(物理页→PTE 反查,让回收能找到"谁映射了这页";反面对比"扫描所有页表"会怎样)。

### 第 5 篇 · 回收与规整:紧张时收回来(4 章)

- **第 16 章 · watermark 与 kswapd**(`P5-16`)——每个 zone 的水位(high/low/min)、kswapd 后台线程在低于 low 时开始回收、高于 high 停。源码:`vmscan.c`(`balance_pgdat`/`kswapd`)、`mmzone.h` 水位字段。**技巧精解:watermark + kswapd 后台回收**(趁还没到 min 提前回收,避免业务触发直接回收的抖动)。
- **第 17 章 · LRU + workingset + vmscan**(`P5-17`)——页按活跃度分 LRU(active/inactive,anon/file)、workingset 估算页的"价值"、vmscan 的回收决策(先回收谁的页)。源码:`vmscan.c`(`shrink_lruvec`)、`workingset.c`、`swap.c`。**技巧精解:LRU + workingset**(近似 LRU 识别冷页;workingset 用访问频率/时间估算价值)。
- **第 18 章 · compaction:内存规整**(`P5-18`)——碎片化导致拿不到大块连续页时,compaction 迁移页凑出连续(给大页/高阶分配)。源码:`compaction.c`。**技巧精解:compaction 的迁移**(把 MOVABLE 页搬走凑连续;反面对比"不规整,碎片化后大块永久拿不到")。
- **第 19 章 · swap 与 OOM**(`P5-19`)——anon 页换出到 swap(swapfile/zswap)、文件页丢弃/回写;实在没了,OOM killer 按 score 挑进程杀。源码:`swapfile.c`、`zswap.c`、`vmscan.c`(`shrink_folio_list`)、`oom_kill.c`。**技巧精解:swap 换出 + OOM 兜底**(冷 anon 页换出到慢设备换内存;OOM 是最后的极端兜底)。

### 第 6 篇 · 进阶(精选,1 章)

- **第 20 章 · 大页 / KSM / memcg / NUMA(精选)**(`P6-20`)——大页(hugetlb 预留 + THP 透明大页,省 TLB)、KSM(同页合并,虚拟化省内存)、memory cgroup(分组限额/统计)、NUMA mempolicy(跨节点策略)。源码:`hugetlb.c`/`huge_memory.c`/`khugepaged.c`、`ksm.c`、`memcontrol.c`、`mempolicy.c`。**技巧精解:THP 透明大页**(自动合并小页为大页省 TLB,与 compaction 协作)。

### 第 7 篇 · 收尾:哲学与对照总表(1 章)

- **第 21 章 · Linux mm 的哲学 + ★对照第 8 本总表**(`P7-21`)——全书收束:快慢路径分级、per-cpu 无锁快路径、惰性分配、buddy 抗碎片、回收的启发式;给一张**内核 mm vs 用户态分配器(tcmalloc/jemalloc)对照总表**,把本书和第 8 本钉成"内存分配全栈"。源码:全书回顾。

## 七、内核 C 技巧侧重(对标 Tokio Rust / Go runtime / LevelDB C++)

本书要重点拆的 Linux 内核技巧(写作时遇到这些,务必钻"怎么做到的 + 为什么 sound"):

- **buddy 算法**:order 2^N、`free_area[]`、O(1) 找伙伴合并——抗外碎片的核心。
- **per-cpu 无锁快路径**:pageset(buddy)、frozen slab + partial(slab)——`cmpxchg` 无锁分配。
- **slab/slub**:freelist、对象布局、frozen slab、partial 队列。
- **多级页表 + mmu_gather**:稀疏地址空间省内存 + 批量刷 TLB 抗抖动。
- **rmap 反向映射**:物理页→PTE 反查,让回收/迁移能定位映射。
- **watermark + kswapd**:后台预回收,避免业务直接回收抖动。
- **LRU/workingset**:近似 LRU + 访问频率估算页价值。
- **RCU / 锁分级**:mm 大量用 RCU(`read_lock`)和细粒度锁。
- **`struct page`/folio 紧凑布局**:海量页的元数据如何省内存。
