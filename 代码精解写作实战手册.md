# 代码精解写作实战手册:从看不懂到讲透

> 一份给"面对陌生代码库、想真正读懂它并能讲透"的人的实战手册。
>
> 本手册沉淀自 11 本"源码精解大书"的实战(etcd / Tokio / LevelDB / Linux 内存管理 / Linux 调度器 / 内存分配器 / RocketMQ / Go runtime / 容器...),把那些"用户读完盛赞"的写法,提炼成可照做的流程、模板、清单。
>
> **写给你**:刚入职、对公司项目代码不熟、想用"写精解"倒逼自己读懂、并产出一份能长期沉淀的文档。本手册第 10、11 节专门讲"陌生公司代码库 + 入职者"的特殊打法。

---

## 0. 先回答一个问题:为什么"写精解"是读懂代码最快的方式

你刚入职,面对几十万行陌生代码,常规做法是"打开 IDE 从头读"——这几乎一定失败:代码是网状的,从哪读?读到哪算懂?读完就忘。

**"写精解"是一种倒逼机制**(本质是费曼学习法):

- **能讲清,才算真懂**。你必须把"这个模块为什么存在、怎么工作、哪里精妙"写明白,写到另一个新人能看懂——写不出来,就是没懂。
- **输出倒逼输入**:为了写清楚一个函数,你会去 grep 它的调用方、读它的测试、追它的 git 历史、问写它的人。这个"追"的过程,就是建立心智模型的过程。
- **双线(动机+技巧)是探照灯**:当你试图回答"它解决什么问题"和"用了什么巧妙手段",你会立刻发现自己哪里没看懂——这两个问题专治"一知半解"。

> **钉死这件事**:你不是在"写文档给别人看",你是在"用写文档的方式让自己读懂"。产物是副产品,读懂才是目的。带着这个心态,你会写得诚实(不懂就追到懂),而不是堆术语装懂。

## 1. 命脉:动机(why)+ 技巧(how)双线 ⚠️ 这是整本书的命脉

这是本手册最重要的一节。所有其他技巧都为这条服务。

读完一段代码,你脑子里要回答**两个问题**,缺一不可:

1. **设计动机(why)**:它解决什么本质问题?**不这样会怎样?**
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

为什么是双线?因为"一知半解"的人,缺口几乎都不在 why(动机好懂:"这是为了做订单幂等""这是为了限流"),而在 **how**(技巧看不懂:"它为什么这里用了双重检查锁""这个状态机凭什么不会死锁")。

> **一句话**:只讲 why 不讲 how,等于没讲透实现;只讲 how 不讲 why,读者不知道这玩意为啥存在。两条都要,且都要配**反面对比**("朴素地写会撞什么墙")。

### 双线的具体写法

每个关键代码点,这样组织:

```
> **不这样会怎样**: <朴素方案会撞的墙:性能损失/正确性漏洞/维护地狱>
> **所以这样设计**: <这个手段怎么化解了上面的墙>
> **钉死这件事**: <一句话结论,带走的那个 insight>
```

然后贴真实源码佐证。**反面对比是让妙处显形的唯一办法**——光说"它用了 X 技巧很妙",读者无感;说"如果不用 X,朴素地写会 Y(具体多惨)",读者才"啊哈"。

### 例子(取自 etcd 系列)

讲 Raft 的"一个 term 一个节点最多投一票",不能只说"这是规则"。要讲:

- **不这样会怎样**:同一个 term 可能选出两个 leader(脑裂),两个 leader 各自接受写,数据冲突。
- **所以这样设计**:投票记录(votedFor)+ term 单调,保证一个 term 内一个节点只投一票,两个候选者不可能都拿到多数派。
- **源码佐证**:贴 `reset` 里 `if r.Term != term { r.Vote = None }` 这一行,讲清"是条件清票,不是无条件"。

**每一条规则、每一个技巧,都要能回答"不这条会破坏什么"。** 回答不了,说明你自己还没懂——这正是精解逼你读懂的地方。

## 2. 基调:直球讲透为主,比喻只在反直觉处点睛

默认**直球**:把数据结构、流程、源码本身讲透。能用一张表、一张时序图、一段源码说清的,**不上比喻**。

**比喻只点睛,不贯穿,不另起体系**:只在某个机制反直觉、直球讲不清的地方,用一两句点一个比喻帮读者"啊哈"。点到为止。

判断标准:**直球讲不清的,才上比喻;讲得清的,就直球。**

### 反例(不要这样)

- ❌ 全篇用一个贯穿比喻(餐厅、工厂、政府...),把每个技术点硬套进去——读者要不停做"比喻→实际"的翻译,累且失真。
- ❌ 为了"生动"在每个概念上都糊一个比喻。
- ❌ 用比喻代替解释("这就像一个管家"然后不解释管家具体干什么)。

### 正例(可以这样)

- ✅ 多数派:"像议会投票,过半数即生效"——一句话点透"过半"的语义,然后立刻回到数学(鸽巢证明)。
- ✅ leader 选举随机超时:"像值班长挂了,大家随机等一会儿再抢着接班,避免同时都抢"——点透"随机化防同时抢票",然后讲代码。

> **一个测试**:把比喻删掉,文章还能不能懂?如果能,比喻是点睛;如果不懂,说明你在用比喻代替讲解,要补直球。

## 3. 结构:每章的标准骨架(严格照此组织)

每章用这个五段骨架。读者一旦熟悉,每章都能快速定位;你写的时候也有模板可依。

```
1. 章首
   - 核心问题(一句话):这一章到底回答什么?
   - 读完本章你会明白(3~4 条,每条戳一个 why/how)
   - 逃生阀(难度高的章才加):"如果一读觉得太难,先只记住三件事——①…②…③…"

2. 一句话点破
   - 给结论 / 金句,然后说"这是结论不是理由,本章倒过来拆"

3. 正文 · 若干小节(每节用四段式,见下)

4. 技巧精解(固定位置:正文之后、小结之前)
   - 挑本章最硬核的 1~2 个技巧
   - 配真实源码片段 + 行号 + 反面对比
   - 单独拆透

5. 章末小结
   - 回扣全书主线(你定的那条"旅程")
   - 五个"为什么"清单(本章五个最关键的 why)
   - 想继续深入往哪钻(对应源码文件、延伸方向)
   - 一句话引出下一章
```

### 正文小节的四段式招牌句式

每个小节都按这个节奏走,反复用这几句招牌话:

```
提出问题 → 不这样会怎样 → 所以这样设计 → 源码/技巧佐证
```

招牌句式(让全书语气统一):

- `> **不这样会怎样**:…`
- `> **所以这样设计**:…`
- `> **钉死这件事**:…`
- (反直觉处才用)`> **打个比方**:…`

这套句式不是八股,是**强制你每个点都走完"问题→反证→方案→佐证"的闭环**。走不完,就是没讲透。

## 4. 第一性原理开篇:从"没有它会怎样"切入

每个系列、每个篇章的第一章,**不要直接讲实现**,要从第一性原理讲"为什么需要这个东西"。

套路:**先讲没有它会怎样(痛点)→ 朴素方案为什么不行 → 所以需要这个机制 → 它一句话是什么。**

这给读者一个"为什么要学"的理由,也给你自己一个"确认我真懂这个模块存在的意义"的机会。

> 例子:etcd 系列第一章不讲 Raft,先讲"单机为什么不够(宕机/数据丢)→ 多副本为什么更难(一致性)→ 脑裂为什么致命 → 多数派凭什么救场 → 所以需要共识算法"。讲完这些,Raft 的每条规则都有了"它要堵的漏洞"。

## 5. 真实源码:严禁凭记忆 ⚠️ 这是最大的硬伤

写精解最大的坑是**凭记忆写源码和行号**。代码细节极多,记忆必然出错,而读者照着错行号去翻源码会一脸懵。

### 铁律

1. **每个要引用的函数/结构体/行号,写前用 Grep / Read 核实**。不确定精确行号时,**宁可只标文件不标行号**,也不要猜。
2. **代码块从真实文件摘录**(允许 `...` 省略,保留部分逐字一致);简化展示必须标注"(简化示意,非源码原文)"。
3. **鼓励质疑自己最初的印象**。你会惊讶地发现,你"记得"的代码,核实后常常是错的——**发现并修正这些错,正是你从"读过"到"真懂"的关键一跃**。

> 实战中,11 本大书的写作 agent 每一本都主动修正了 5~20 处"最初印象不准"的源码事实(比如 etcd 的 `new(r.Term)` 是 Go 1.26 泛型 new 而非老的 new(LevelDB 的 freelist 不在顶层文件而在子包;mvcc 的 revision Main 是事务级而非每次 Put +1)。这些修正全是高质量信号——**敢质疑自己最初印象的,才是真读了源码**。

### 核查工具

- 本地代码:Grep(找符号)+ Read(核行号/上下文)。
- 没有本地代码的开源项目:用 GitHub 的 `zread`/WebFetch,或 `gh`。
- 公司代码:就在你公司仓库,直接 Grep/Read。

## 6. 配图策略

图是精解的半条命。原则:**几何/坐标才需要图片;流程/结构/布局用矢量图或文本框图**。

| 想表达的东西 | 用什么 |
|---|---|
| 时序(请求一来一回、状态推进) | **mermaid `sequenceDiagram`** |
| 状态转换(Follower→Candidate→Leader) | **mermaid `stateDiagram`** |
| 流程/分支(compaction、读路径) | **mermaid `flowchart`** |
| 内存/数据结构布局(B+tree 页、环形缓冲、版本链、结构体字段) | **ASCII 框图或表格** |
| 真正的坐标曲线/分布 | matplotlib 之类出 PNG |

**铁律**:不要用 ASCII 画时序(用 mermaid);不要用图片画能文本表达的流程。

## 7. 标点与语言

虽是小事,但统一标点让全书像一个人写的、读起来不累。

- **标点**:半角 ASCII 逗号 `,`、冒号 `:`、括号 `()`、问号 `?`;全角顿号 `、`、破折号 `——`、引号 `""`、句号 `。`。(简单记:**逗号冒号括号问号用半角,顿号破折号引号句号用全角**)
- **中英文之间留空格**:`使用 Raft 协议` 而非 `使用Raft协议`。
- **技术名词保留英文**:leader / follower / commit / apply / revision / WAL 等,不硬译。
- 全书写完扫一遍:全角逗号 `，` 和全角冒号 `：` 应该**零命中**(代码块外)。

## 8. 多 agent 编排:规模化量产高质量章节 ⚠️ 如果你用 AI 辅助

如果你想用 AI(agent)辅助写一本多章的精解,**正确编排是质量的关键**。这是踩过坑总结的。

### 8.1 地基层串行 + 应用层篇间并行

- **地基层(概念强依赖)必须串行**:后章依赖前章立起的概念(如 Raft 的 term→选举→复制→安全→驱动)。用**多个 agent 接力**(每章一个 agent,前章落盘后起下章 agent 读它衔接)。
- **应用层(各篇相对独立)篇间并行**:不同篇章可同时开多个 agent。
- **不要让一个 agent 写整篇(4 万字以上)**:单 agent 上下文/token 压力大,后段会偷工。**每章一个 agent** 是质量基本盘。

### 8.2 每个写作 agent 的 prompt 五要素(给全才不跑偏)

1. **必读文件**(绝对路径,按序):全书总纲 + 写作提示词 + 风格样板章 + 前驱章(地基/篇内上一章)。子 agent 看不到你的上下文,必读要写死。
2. **本章专属任务**:标题 / 核心问题 / 服务主线哪面 / 关键源码起点文件 / **钦定技巧**(为技巧精解备料)/ 前后衔接句 / 概念边界(防与同篇他章重复)。
3. **铁律速查**:四段式招牌句式、技巧精解固定位置 + 反面对比、直球基调、mermaid + ASCII 配图、标点、五段结构。
4. **源码核查硬要求**:写前 Grep/Read 核实行号,**严禁凭记忆**;不确定只标文件;简化代码标"(简化示意)";**明确鼓励 agent 质疑/修正你最初不准的源码印象**(这是高质量信号)。
5. **输出**:逐章 Write 落盘到 `P{篇}-{章}-标题.md`;回报路径 + 字数 + 引用的源码文件 + 是否修正了源码印象。

### 8.3 主控核验三板斧(每波交付后做,堵硬伤)

1. **Grep 结构关键词一次性扫全章节**:`技巧精解`、`五个为什么`、`章末小结` 等,立刻发现漏写/格式不一。
2. **抽查每篇 1~2 处关键源码行号真实性**(最大硬伤是编造行号):Read 源码对照 agent 引用,逐字吻合才放行。
3. **看 agent 是否主动修正源码印象**:敢质疑、按真实源码改写 = 真读了源码,可信度高;照搬你给的 = 可能凭记忆,重点抽查。

> 并行要点:同一批要并行的 agent,**放在一条消息里多个 Agent 调用**,才真并发。

## 9. 配套文件:框架四件套(开写前先搭)

动笔前,先搭这四份"脚手架",后面每章 agent 都靠它们对齐:

1. **全书规划-总纲.md**:主线、二分法、分篇分章、源码策略、写作约定、技巧侧重。这是宪法。
2. **目录与导读.md**:每章一句话钩子 + 技巧标签 + 二分法归属;推荐阅读路线;速查表(按目标读哪几章)。
3. **_章节写作提示词.md**:写作执行手册(角色、双线命脉、基调、四段式、技巧精解、自检清单、附全章清单与并行分组)。这份**直接复用本手册**的内容即可。
4. **P0-01 样本章**:第一性原理开篇章,写好后作为**风格样板**,后面所有 agent 对齐它的文风/结构/源码链接格式。

### 定主线和二分法(总纲的核心)

- **主线**:用"一条什么旅程"当骨架(如 etcd 的"一条 Put 怎么被多数派共识"),而非按模块平铺。读者永远知道"我在哪、为什么需要下一站"。
- **二分法**:给读者一个迷路时回到的锚(如 etcd 的"协议层 vs 应用层")。每个机制都归到两面之一。

## 10. ⭐ 写一个新系列的完整流程(从 0 到 1)

这是你入职后写公司代码精解的操作流程。

### Step 1 · 摸清项目(入职者最关键,别跳)

陌生代码库,先建心智模型,别急着写:

- **跑起来**:本地把项目跑起来,跑通一个最小流程(下一个断点,跟一遍)。**能断点调试,胜过读十小时代码**。
- **找入口**:main 函数 / API 入口 / 消息消费入口 / 定时任务入口——从"一个请求进来"开始追。
- **画调用链**:追一个典型请求,画出"入口 → ... → 落库/返回"的调用链(mermaid 时序图)。这条链就是你的"主线"候选。
- **摸模块边界**:看目录结构、package/模块划分,搞清有哪些子系统、它们怎么依赖。
- **读文档**:README、设计文档、wiki、内部 Confluence、PRD——有任何文档先读。
- **问人**(入职者最大优势):你的 mentor / 写这段代码的同事 / code owner。**"这个模块主要解决什么问题""为什么要这么设计"——问 why,答案直接进你的精解**。git blame + git log 看某段代码的提交历史和 PR 描述,常常藏着"为什么这么写"。
- **看测试**:测试用例是最好的"使用说明书",告诉你这玩意预期怎么用、边界在哪。

> 这一步产出:一张全局调用链图 + 一份"模块职责清单"+ 一堆"为什么"的答案(来自文档和同事)。

### Step 2 · 定主线 + 二分法

- 从 Step 1 的调用链里选一条**最有代表性的旅程**当主线(如"一个下单请求从 API 到落库""一条消息从生产到消费")。
- 找一个**二分法**(迷路时的锚):常见的是"同步路径 vs 异步路径""核心逻辑 vs 边界处理""业务层 vs 基础设施层""写路径 vs 读路径"。

### Step 3 · 定基调

默认**直球为主,比喻点睛**(本手册第 2 节)。业务代码大多具体可见,直球讲清最好。只在反直觉处(并发竞态、状态机、缓存一致性等)点一两句比喻。

### Step 4 · 搭框架四件套

按第 9 节搭:总纲、目录、写作提示词、样本章(P0-01 第一性原理)。**样本章一定先手写一章**,定调子,后面 agent 才有对齐基准。

### Step 5 · 多 agent 铺开(地基串行 + 篇间并行)

按第 8 节编排。地基篇(概念强依赖的)先串行写完,应用层各篇再并行。

### Step 6 · 核验三板斧

每波交付后:Grep 扫结构 + 抽查行号 + 看是否修正印象。

### Step 7 · 迭代

自己通读,标出"这里我还讲不清"的地方——那就是你还没真懂,回去追源码/问人,补上。**精解是活的,随着你理解加深持续修订**。

## 11. ⭐ 公司代码场景的特殊适配(区别于开源大系统)

你写公司代码精解,和写 etcd/Tokio 这种开源大系统,有几处关键不同。下面是适配清单。

### 11.1 主线选"一条业务请求的旅程"

公司代码是**业务驱动**的,不像开源系统有清晰的"协议/存储/调度"分层。所以主线用**一条业务请求的端到端旅程**(一个 API 请求 / 一条消息 / 一个订单),最贴合实际,也最能让同事(读者)代入。

### 11.2 why 的来源:问人和 git 历史,而非论文

- 开源系统:why 常来自论文/规范/对比(Raft 论文、B+tree 教科书)。
- 公司代码:why 常来自**业务需求 + 历史包袱**。去问 code owner、看 PR 描述、git blame。**"为什么这么设计"的答案,经常是"当时有个 bug/性能问题/合规要求"——这种故事比论文更生动,且只有你能挖到**。

### 11.3 技巧侧重:业务领域的工程技巧

公司代码不一定有无锁队列/Pin/RCU 这种系统级硬核技巧,但有大量**业务工程技巧**,同样值得精解:

- **状态机**:订单状态流转、工单状态机——讲清状态、迁移条件、为什么不会到非法状态。
- **并发控制**:库存扣减的乐观锁/悲观锁、防超卖、幂等键。
- **缓存**:多级缓存、缓存一致性、击穿/穿透/雪崩的防范。
- **事务**:分布式事务(Saga/TCC/本地消息表)、最终一致性。
- **幂等/去重**:接口幂等、消息去重、重试安全。
- **配置/开关**:灰度、feature flag、动态配置。
- **限流/降级/熔断**:稳定性手段。

这些是公司代码的"硬核技巧",讲透它们,你的精解就有含金量。

### 11.4 规模:从一个小模块练手,别一上来写大书

开源系列是 21 章大书。你刚入职,**先从一个小模块(3~5 章)练手**:挑一个你最近在碰的模块(比如"订单状态机""支付回调处理"),用本手册方法写 3~5 章。写完你就把这个模块彻底吃透了,也验证了方法。后面再扩。

### 11.5 入职者的独特优势:新人视角

你现在的"不懂",是**最有价值的资产**——它精准标记了"哪里难懂"。老员工早已习以为常、觉得"理所当然"的地方,恰恰是新读者最需要解释的地方。**把你踩过的每个坑、每个"这是啥"的瞬间,都写进精解**。这是老员工写不出来的东西。

### 11.6 ⚠️ 保密红线

- 公司代码**不外发**:如果你用 AI agent 辅助,**只用本地/合规的 agent**,不要把公司代码贴到外部服务。
- 精解文档本身可能含敏感信息(业务逻辑、架构细节)——**写之前确认公司对内/对外文档的合规要求**,内部精解放公司内网 wiki 即可,别往公共 GitHub 推。
- 涉密数据(真实用户数据、密钥、内部价格)绝不进文档,示例用脱敏的假数据。

### 11.7 代码版本:钉死 commit

公司代码持续迭代。你的精解**钉死某个 commit/版本号**,行号以那个版本为准。代码变了,文档标"以 vX.Y.Z 为准"。否则三个月后行号全错。

## 12. 常见坑(踩过的,别再踩)

1. **凭记忆写源码/行号**(最大硬伤)→ 每处引用 Grep/Read 核实。
2. **只讲 why 不讲 how**(读者最大抱怨)→ 每个点双线,技巧精解不可省。
3. **滥用比喻**→ 直球为主,删掉比喻还能懂才对。
4. **单 agent 写整篇**(后段偷工)→ 每章一个 agent。
5. **示例数字/输出与声明不符**(我踩过:蒙特卡洛模拟的输出和 seed 对不上)→ 代码示例交付前原样重跑核对。
6. **编造行号**→ 不确定只标文件。
7. **篇内并行衔接打架**(同篇两章并行,概念重复或矛盾)→ prompt 里钉死每章"概念边界",串行写地基篇。
8. **跳过"不这样会怎样"直接讲实现**→ 四段式不能省,反证是讲透的关键。
9. **堆术语装懂**("显然""众所周知"是禁词)→ 每个术语第一次出现就解释。

## 13. 交付前自检清单(逐条过完再算完成)

- [ ] 每个小节先讲"解决什么问题/不这样会怎样",再讲实现?
- [ ] 每个关键代码点都讲了"动机 + 技巧"两面?
- [ ] "技巧精解"挑了 1~2 个最硬核技巧,配真实源码 + 反面对比?
- [ ] 章首有"核心问题 + 读完你会明白(3~4 条)";难度高的加了逃生阀?
- [ ] 是否直球讲清为主,比喻只在反直觉处点睛、没堆砌?
- [ ] 所有源码引用都 Grep/Read 核过,行号与真实文件一致?
- [ ] 所有并发/协议技巧都讲了"为什么 sound"(不会数据竞争/不会破坏不变式)?
- [ ] 时序/状态/流程图用了 mermaid,数据结构布局用了 ASCII 框图,没用 ASCII 画时序?
- [ ] 标点:半角逗号冒号括号问号 + 全角顿号破折号引号句号,中英文间空格?
- [ ] 章末有"五个为什么清单"+"想继续深入往哪钻"+引出下一章?回扣了主线?
- [ ] 通读一遍:有没有"显然""众所周知"这类跳过解释的词?有没有和前文重复?

## 14. 可复用模板:章节写作 agent 提示词骨架

把下面填空,就能驱动一个 agent 写一章(本地/合规 agent)。`【】` 里填你的内容。

```
你是《【项目名】代码精解》的章节写作 agent(资深工程师 + 技术作者)。

【动手前必读,用 Read 通读】:
1. 【总纲绝对路径】
2. 【写作提示词绝对路径】 ← 全部写作铁律(双线/直球/四段式/技巧精解/自检)
3. 【样本章 P0-01 绝对路径】 ← 风格样板,文风/结构/源码链接格式一律对齐它
4. 【同篇前一章绝对路径,若已写】 ← 衔接,不重复不矛盾
5. 【目录与导读绝对路径】

【源码,本地已就位。每处引用前 Grep/Read 核实行号,严禁凭记忆】:
- 【代码仓库根路径】(主要模块: ...)

【铁律摘要】
- 动机(why)+技巧(how)双线;每章单开"技巧精解"小节(正文后小结前),
  1~2 个最硬核技巧 + 真实源码 + 行号 + 反面对比("朴素地写会撞什么墙")。
- 直球为主,比喻只在反直觉处点一两句。
- 正文每节四段式:提问→不这样会怎样→所以这样设计→源码佐证;
  招牌句式反复用 > **不这样会怎样**:…、> **所以这样设计**:…、> **钉死这件事**:…。
- 配图:时序/状态/流程用 mermaid;数据结构布局用 ASCII 框图;不用 ASCII 画时序。
- 标点:半角逗号,冒号:括号()问号?;全角顿号、破折号——引号""句号。;中英文间空格。
- 每章标准结构:①章首(核心问题+读完你会明白3~4条,难度高的加逃生阀);
  ②一句话点破;③正文四段式;④技巧精解;⑤章末小结(回扣主线+五个为什么+
  想继续深入往哪钻+引出下一章)。
- 篇幅 8000~12000 字。严禁凭记忆写源码/行号;不确定行号只标文件;
  简化代码标注"(简化示意)"。鼓励质疑最初印象不准处并按真实源码修正。

【输出】Write 到:【P{篇}-{章}-{标题}.md 绝对路径】。
完成后回报:文件路径+字数+引用的关键源码文件+是否修正了源码印象。

【本轮任务】写 第 {篇号} 篇 · 第 {章号} 章 · 《{标题}》。
- 核心问题: ...
- 主线归属: ...(这条旅程的哪一站)
- 关键源码起点(自行核实行号): ...
- 钦定技巧精解(必写,配真实源码+行号+反面对比): ...
- 前置衔接: 上一章 ... ,从"..."接;读者已知 ...,不重述。
- 概念边界(防与同篇他章重复): 本章只讲 ...,不讲 ...(那章讲)。
```

本手册的方法,在以下系列实战中验证(用户原话"写的太棒了""就是要这样的代码精解系列"):
