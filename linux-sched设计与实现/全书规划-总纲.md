## 一、这本书的定位

### 目标读者

会写 C、读过操作系统课、用过 `fork`/`pthread`/`nice`/`chrt`、甚至翻过 `kernel/sched/fair.c`、`kernel/sched/core.c`,但**一知半解**:知道有 CFS、运行队列、负载均衡、上下文切换,却讲不清一次 `sched_yield` 内核做了什么、一个任务被唤醒后多久能跑、EEVDF 怎么挑下一个、nice +1 到底改了什么、ksoftirqd 怎么和你的进程抢 CPU、负载均衡为什么把任务搬走又搬回。和"读过 tokio/go runtime/mm 源码但不理解"是同一种病。

### 全书一句话主旨

> **少量物理 CPU 要公平、高效地驱动成千上万的任务——调度器决定"下一个跑谁、跑多久、何时被打断、负载怎么在多核间均衡"。它在一个时钟节拍、一次唤醒、一次阻塞的瞬间反复做决策,在公平(每个任务都该有份)、优先级(重要的先跑)、吞吐(CPU 别闲着)、响应(交互快)之间平衡。**

每个组件,都是这条旅程上的一个**驿站**。读完一本书,读者能**在脑子里放映出内核调度的全过程**:一个任务从 `fork` 创建、进运行队列、被 EEVDF 选中、跑到时间片耗尽被抢占、上下文切换走、在 SMP 上被负载均衡迁到另一个核——以及每一步底下用了什么巧妙的手段。

### 为什么用"少量 CPU 怎么驱动海量任务"当骨架(而非按文件平铺)

`kernel/sched/` 有十几个 `.c`、一个巨大的内部头 `sched.h`,高度耦合:任务表示(`task_struct`/`sched_entity`)、运行队列(`rq`/`cfs_rq`/`rt_rq`)、选任务策略(CFS/EEVDF、RT、deadline)、抢占与切换、SMP 均衡、cgroup 限额。按文件百科式平铺,读者会觉得每块都是孤岛。用"少量 CPU 怎么驱动海量任务"这条主线当骨架——任务怎么表示、怎么入队、调度器怎么选下一个、怎么切换上去、怎么在核间均衡、怎么被 cgroup 管住——每个模块都是"为了让这条流走得通"而出现的角色,读者永远知道**我在哪、为什么需要下一站**。

## 二、写作方法

### 命脉:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与一般操作系统教材、源码导读最关键的分野,所有写作者必读、必守。**

Linux 调度器源码充斥内核级技巧:EEVDF 的 lag + virtual deadline(公平的核心数学)、PELT 的几何级数负载跟踪、nice → weight 查表、per-CPU runqueue + `rq->lock` 细粒度锁、`preempt_count` 抢占计数、`TIF_NEED_RESCHED` 延迟抢占、`switch_to` 的栈切换与寄存器交接、`load_balance` 的调度域层级、cgroup bandwidth 的 throttle、内存序屏障。用户自己读过源码,"一知半解"的缺口往往**不在**"它要解决什么"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?(沿用系列一贯的"为什么优先")
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果不用这个技巧、朴素地写,会撞上什么墙",让妙处显形。

一句话:**讲不清技巧,等于没讲 Linux 调度器。**

### 基调:直球讲透为主,比喻只在反直觉处点睛 ⚠️ 延续《LevelDB》

调度器有一部分反直觉(为什么 EEVDF 用 virtual deadline、为什么时间片不是固定值、为什么 nice 不是线性),这部分**允许适度比喻点睛**。但调度器的大部分(runqueue 结构、EEVDF 公式、上下文切换的寄存器流转)**具体可见,直球讲清最好**。

- **默认直球**:能用表 / 流程图 / 源码说清的,不上比喻。
- **比喻只点睛**:EEVDF 可点"像给每个任务发一个截止时间,谁快到了谁先跑";runqueue 可点"像每核一个候车厅,乘客按各自的紧迫度排队"。点到为止。
- **判断标准**:直球讲不清才上比喻。

### ★ 对照《Go runtime》(第七本):本书独有特色

> 本书和第七本《Go runtime 设计与实现深入浅出》是一对——Linux 调度器是"内核级、抢占式、通用"调度(对所有任务,内核强制切换),Go runtime 的 GMP 是"语言级、协作式 + 异步抢占、面向 goroutine"调度(Go 自己管 goroutine,少量地搭在内核线程上)。两本合成"调度全栈"。

本书在调度类、上下文切换、负载均衡等关键章末用轻量"★对照"小栏,和 Go 的 GMP 对读;收尾章给总表。对照要点:内核 `rq` ↔ Go 的 `p`(P)本地队列;`switch_to`(内核切换线程)↔ Go 的 `gogo`(切换 goroutine 栈);Linux `load_balance`(核间拉任务)↔ Go 的 work-stealing(偷 goroutine);nice/权重 ↔ goroutine 无优先级(默认全公平);Linux 抢占(任何内核态可抢占点)↔ Go 协作抢占(函数调用安全点)+ 6.x 异步抢占(信号强制)。

## 三、全书主线、二分法

### 主线

> **少量物理 CPU 要公平、高效地驱动海量任务——调度器决定下一个跑谁、跑多久、何时被打断、负载怎么在多核间均衡,在公平、优先级、吞吐、响应之间平衡。**

任何一处看不懂某个机制,回到这句问:"这是在**决定跑谁(策略)**,还是在**落实跑这件事(机制:切换/抢占/均衡/限额)**,或是在**支撑这两件事(任务表示/运行队列/时钟)**?"把答案写进"为什么"里。

### 二分法(迷路时回到它)

本书的二分法:

> **策略层(给定 runqueue,下一个跑谁:CFS/EEVDF 公平、RT 实时、deadline 截止期) vs 机制层(怎么落实:任务进出队列、上下文切换、抢占时机、SMP 负载均衡、cgroup 限额)。**

- **策略层**:`pick_next_task`(选下一个)、`enqueue_task`/`dequeue_task`(进/出队列的记账)、EEVDF 的 lag/virtual deadline、RT 按 prio、deadline 按 EDF。这些回答"谁该跑"。
- **机制层**:`__schedule`/`context_switch`/`switch_to`(切换上去)、抢占点与 `TIF_NEED_RESCHED`(何时打断)、`load_balance`(核间搬任务)、cgroup cpu bandwidth(限额)。

支撑这两者的地基:`task_struct`/`sched_entity`(任务表示)、`rq`/`cfs_rq`/`rt_rq`(运行队列)、时钟(`sched_clock`/`hrtick`)、`preempt_count`(抢占计数)、锁(`rq->lock`)。

调度器每个机制都落在这两面之一或支撑地基。任何时候迷路,回到这个二分法。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线是一条旅程:**为什么内核要调度 → 任务与运行队列怎么表示 → EEVDF 怎么选下一个 → 抢占与上下文切换怎么落实 → SMP 怎么均衡 → 实时/cgroup 怎么管 → 哲学收束**。
> 章号全局连续;命名 `P{篇}-{章}-标题.md`。**全书 21 章**(核心全覆盖,对标已有系列)。

### 第 0 篇 · 开篇:为什么内核要调度(定调,1 章)

- **第 1 章 · 第一性原理:为什么内核要调度**(`P0-01`):CPU 是稀缺共享资源 + 多任务要并发幻觉 + 公平/优先级/吞吐/响应的矛盾 → 必须有仲裁者决定下一个跑谁 → 调度器全貌(policy/mechanism/支撑)→ 立起"策略 vs 机制"二分法 + ★对照 Go runtime(内核抢占 vs 语言协作)。**【样本章节】**

### 第 1 篇 · 任务与运行队列:调度的账本(4 章)

> 源码主体 `kernel/sched/core.c`、`kernel/sched/sched.h`(内部头,含 `rq`/`cfs_rq`/`sched_entity`/`sched_class`)、`include/linux/sched.h`(`task_struct`)。建议顺序读。

- **第 2 章 · `task_struct` 与 `sched_entity`:任务怎么表示**(`P1-02`)——一个任务在内核里长什么样、`sched_class`(调度类多态)、`sched_entity`(调度实体,既表示任务也表示组)、`prio`/`nice`/`policy` 字段。源码:`include/linux/sched.h`、`kernel/sched/sched.h`。**技巧精解:`sched_class` 多态 + `sched_entity` 嵌入式结构体**(用 `container_of` 反查 `task_struct`,组调度复用同一套)。归属:`支撑`。
- **第 3 章 · `rq`、`cfs_rq`、`rt_rq`:运行队列**(`P1-03`)——每个 CPU 一个 `rq`,内含 `cfs_rq`(公平)/`rt_rq`(实时)/`dl_rq`(deadline)子队列,`rq->lock`、`rq->curr`/`rq->idle`。源码:`kernel/sched/sched.h`、`core.c`。**技巧精解:per-CPU `rq` + 每队列独立 `rq->lock`**(细粒度锁,反面对比"全局一把大锁")。归属:`支撑`。
- **第 4 章 · 时钟:`sched_clock`、tick 与 hrtick**(`P1-04`)——调度器的心跳:`sched_clock`(纳秒级、可能不稳定)、定期 tick(`scheduler_tick`)、`hrtick`(高精度单次定时器,EEVDF 精确抢占)。源码:`kernel/sched/clock.c`、`core.c`(`scheduler_tick`/`hrtick`)。**技巧精解:hrtick**(到点精确抢占,反面对比"只靠粗 tick,时间片误差一个 tick")。归属:`支撑`。
- **第 5 章 · 任务的入队出队:`enqueue`/`dequeue`/`wakeup`**(`P1-05`)——任务生命周期:`fork` 创建→`activate_task`(enqueue)、阻塞睡眠→`deactivate_task`、`try_to_wake_up` 唤醒。源码:`core.c`(`try_to_wake_up`/`activate_task`/`enqueue_task`)、`fair.c`(`enqueue_entity`)。**技巧精解:`try_to_wake_up` 的选核**(唤醒时挑哪个 CPU 跑,`select_task_rq`/`select_idle_sibling`,cache 亲和)。归属:`机制`。

### 第 2 篇 · EEVDF 公平调度:下一个跑谁(5 章)⚠️ 核心重头戏之一

> 源码主体 `kernel/sched/fair.c`(全书最大、最核心的文件)、`kernel/sched/features.h`。这一篇是 6.6 EEVDF 替代 CFS 的重头。

- **第 6 章 · 从 CFS 到 EEVDF:为什么换**(`P2-06`)——CFS(完全公平调度器)统治 20 年的原理(虚拟运行时间 `vruntime` + 红黑树选最小 vruntime)及其局限(延迟保障弱、对体重任务不公平、cgroup 场景失真)→ 6.6 引入 EEVDF(Earliest Eligible Virtual Deadline First)。源码:`fair.c`(`pick_next_entity`/`entity_eligible`)。**技巧精解:CFS vruntime 的直觉与缺陷**(为讲 EEVDF 铺垫)。归属:`策略`。
- **第 7 章 · EEVDF 算法:lag、eligibility、virtual deadline**(`P2-07`)——EEVDF 核心:`lag`(任务"欠"多少 CPU 时间)、`eligible`(lag ≥ 0 才有资格跑)、`virtual deadline`(按权重算的截止虚拟时间),pick = 在 eligible 任务里选最早 deadline 的。源码:`fair.c`(`entity_eligible`/`vdelay`/`pick_eevdf`/`update_curr`)。**技巧精解:EEVDF 的三件套**(lag 表公平、eligible 表资格、deadline 表先后;反面对比"CFS 只看 vruntime 无法保证延迟")。归属:`策略`。
- **第 8 章 · `nice` 与权重:从 nice -20..19 到 timeslice**(`P2-08`)——`nice` 值怎么映射成权重(`prio_to_weight[]` 查表)、权重怎么决定时间片占比、`renice` 改了什么。源码:`fair.c`(`set_load_weight`/`__calc_delta`)、`sched.h`(`prio_to_weight`)。**技巧精解:nice → weight 的非线性查表**(每级差约 10% CPU,反面对比"线性映射会让 nice 差异不明显")。归属:`策略`。
- **第 9 章 · PELT 负载跟踪:per-entity 的 `load_avg`/`util_avg`**(`P2-09`)——PELT(Per-Entity Load Tracking):每个调度实体维护衰减的负载/利用率(`load_avg`/`util_avg`/`runnable_avg`),几何级数按 1024us 衰减,驱动负载均衡与 CPU 频率调节。源码:`fair.c`(`__update_load_avg_blocked`/`__update_load_avg_se`/`accumulate_sum`)、`sched.h`(`sched_avg`)。**技巧精解:PELT 几何级数衰减**(Y^32 的衰减系数,反面对比"瞬时采样会被尖峰误导")。归属:`支撑`(为均衡/频调服务)。
- **第 10 章 · EEVDF 的 `pick_next_fair` 与时间片**(`P2-10`)——CFS/EEVDF 的 `pick_next_task_fair`:遍历 cfs_rq 选最早 deadline 的 entity、算本周期时间片(`calc_delta_fair`)。源码:`fair.c`(`pick_next_task_fair`/`pick_next_entity`)。**技巧精解:时间片的动态计算**(按权重在 period 内的比例,反对比"固定时间片")。归属:`策略`。

### 第 3 篇 · 抢占与上下文切换:怎么落实跑(3 章)

> 选下一个谁,只是决策;真正切过去、还要让被打断的任务之后能恢复,是机制层的硬骨头。

- **第 11 章 · 抢占点与 `TIF_NEED_RESCHED`**(`P3-11`)——抢占的本质:`TIF_NEED_RESCHED` 标志 + 在"抢占点"(`preempt_enable`/中断返回/唤醒)检查它、`preempt_count` 控制是否允许抢占。源码:`core.c`(`resched_curr`/`check_preempt_curr`/`preempt_schedule`)、`include/linux/preempt.h`。**技巧精解:延迟抢占(`TIF_NEED_RESCHED`)**(标记要抢、等到安全点才真切,反面对比"立即抢占会在持锁时崩溃")。归属:`机制`。
- **第 12 章 · `__schedule` 与 `pick_next_task`**(`P3-12`)——主调度函数 `__schedule`:关抢占、拿 `rq->lock`、`pick_next_task`(按 dl > rt > fair > idle 的优先级遍历 `sched_class`)、`context_switch`。源码:`core.c`(`__schedule`/`pick_next_task`/`schedule`)。**技巧精解:`sched_class` 链表按优先级遍历**(dl/rt/fair/idle 一条链,谁有就绪任务谁胜出)。归属:`机制`。
- **第 13 章 · 上下文切换 `switch_to`**(`P3-13`)——真正切换:`context_switch`→`switch_to`(宏,切内核栈 + 切寄存器 + 切 FPU/向量)+ `finish_task_switch`。x86 的 `__switch_to_asm`(栈指针切换)。源码:`core.c`(`context_switch`/`switch_to` 宏)、`arch/x86/kernel/process_32.c`/`process_64.c`(未 sparse clone,描述其作用)。**技巧精解:`switch_to` 的栈切换 + 两个返回点**(切 `rsp` 后控制权交给新任务,`prev` 参数通过 `current` 传递的精妙)。归属:`机制`。

### 第 4 篇 · SMP 负载均衡:核间搬任务(3 章)

> 多核场景:一个核忙死、一个核闲死不行,要把任务在核间均衡。

- **第 14 章 · 调度域与调度组:`sched_domain`/`sched_group`**(`P4-14`)——SMP 拓扑怎么抽象成调度域层次(核 → 物理 CPU → NUMA node),`sched_domain`/`sched_group`/`sched_domain_topology`. 源码:`kernel/sched/topology.c`、`sched.h`。**技巧精解:调度域分层**(按 cache/NUMA 距离分域,近域优先均衡,反面对比"全局平铺均衡会破坏 cache 局部性")。归属:`机制`。
- **第 15 章 · `load_balance`:周期均衡与 idle 均衡**(`P4-15`)——负载均衡主循环:`load_balance`(周期触发,从忙域 pull 任务)、`idle_balance`(CPU idle 时主动 pull)、`find_busiest_group`/`find_busiest_queue`。源码:`fair.c`(`load_balance`/`sched_balance_rq`/`detach_tasks`/`attach_tasks`)。**技巧精解:`load_balance` 的 pull 模型 + 迁移代价评估**(反面对比"盲目迁移导致 cache 失效比不均衡还亏")。归属:`机制`。
- **第 16 章 · 任务迁移与 CPU 亲和**(`P4-16`)——迁移一个任务的开销(cache 冷)、`sched_setaffinity` 设亲和掩码、`cpuset` cgroup、migration/IPI 把任务踢到目标核。源码:`core.c`(`set_cpus_allowed_ptr`/`migrate_task`)、`stop_task.c`(migration 线程)。**技巧精解:迁移的 cache 代价 + 亲和掩码**(反面对比"任意迁,cache 抖动抵消均衡收益")。归属:`机制`。

### 第 5 篇 · 实时与特殊调度类(2 章)

> CFS/EEVDF 只管 `SCHED_NORMAL`/`SCHED_BATCH`;实时和特殊任务有专门调度类。

- **第 17 章 · RT 实时调度:`rt_rq` 与 RT throttling**(`P5-17`)——`SCHED_FIFO`/`SCHED_RR`:按优先级(0-99)绝对抢占普通任务,`rt_rq` 用位图+队列 O(1) 选最高优先级、RT throttling(默认 RT 占 95% CPU 上限,留 5% 给系统避免锁死)。源码:`kernel/sched/rt.c`(`enqueue_task_rt`/`pick_next_task_rt`/`sched_rt_runtime_exceeded`)。**技巧精解:RT 的位图 O(1) 选 prio + RT throttling**(反面对比"无 throttle,失控 RT 进程能挂死整机")。归属:`策略`。
- **第 18 章 · `deadline` 调度 + `idle` + `stop`**(`P5-18`)——`SCHED_DEADLINE`(EDF 最早截止期 + CBS 带宽预留,最高优先级调度类)、`idle_task`(没事干的兜底)、`stop_sched_class`(特权,迁移/在线性热插拔用,总是可抢占任何任务)。源码:`kernel/sched/deadline.c`、`idle.c`、`stop_task.c`。**技巧精解:`SCHED_DEADLINE` 的 EDF + CBS**(基于 CBS 适配的 EDF,保证可调度性;`stop_sched_class` 的特殊地位)。归属:`策略`。

### 第 6 篇 · cgroup 与调度进阶(2 章)

- **第 19 章 · cgroup cpu:组调度与 bandwidth 限额**(`P6-19`)——`cpu` 子系统把任务分组,每组一个调度实体(`sched_entity` 充当组),`cpu.max`/`cpu.weight` 限额;bandwidth:`cpu` cgroup 的 period/quota 配额 + 超额 throttle。源码:`fair.c`(`task_group`/`sched_entity` 组层次/`throttle_cfs_rq`)、`core.c`。**技巧精解:组调度(任务和组复用 `sched_entity`)+ bandwidth throttle**(层级化公平,反面对比"无 cgroup,一个进程的几千线程能独占整机")。归属:`机制`。
- **第 20 章 · NUMA balancing + 调度可观测与调参**(`P6-20`)——NUMA balancing(自动把页和任务迁到本 NUMA 节点,见 mm 系列的 `do_numa_page` 回扣)、调度可观测(`/proc/sched_debug`/`/proc/<pid>/sched`/`schedstat`)、调参与工具(`chrt`/`renice`/`taskset`/`tuna`),并展望 `sched_ext`(eBPF 可编程调度器,注:6.9 未合入主线,6.12 起,作延伸)。源码:`fair.c`(`task_numa_placement`)、`kernel/sched/stats.c`、`debug.c`。**技巧精解:NUMA balancing 的故障率采样**(以页 protnone 触发采样,估算最优节点)。归属:`机制`。

### 第 7 篇 · 收尾:哲学与对照总表(1 章)

- **第 21 章 · Linux 调度器的哲学 + ★对照 Go runtime 总表**(`P7-21`)——全书收束:延迟抢占、per-CPU 队列细粒度锁、`sched_class` 多态、PELT 几何衰减、调度域分层均衡、组调度复用 `sched_entity`;给一张**Linux 调度器 vs Go runtime GMP 对照总表**,把本书和第 7 本钉成"调度全栈"(内核通用抢占 vs 语言轻量协作)。源码:全书回顾。

## 七、内核 C 技巧侧重(对标 Tokio Rust / Go runtime / LevelDB C++ / mm 内核 C)

本书要重点拆的 Linux 内核调度技巧(写作时遇到这些,务必钻"怎么做到的 + 为什么 sound"):

- **EEVDF 算法**:lag(欠账)、eligibility(资格)、virtual deadline(截止),三件套实现公平 + 延迟保障——CFS 之后调度公平性的核心数学。
- **PELT 几何级数衰减**:`load_avg`/`util_avg` 用 Y^32 衰减系数跟踪 per-entity 负载,驱动均衡与频调。
- **`sched_class` 多态**:dl/rt/fair/idle/stop 五个调度类一条链,`pick_next_task` 按优先级遍历——调度策略的插件化。
- **`sched_entity` 嵌入 + `container_of`**:任务和组复用同一个 `sched_entity`,CFS 的组调度靠它。
- **per-CPU `rq` + 细粒度 `rq->lock`**:每核一个 runqueue、每队列独立自旋锁,避免全局锁。
- **延迟抢占(`TIF_NEED_RESCHED` + `preempt_count`)**:标记要抢、延迟到安全点(可抢占点)才真切——避免在持锁/原子区切换的崩溃。
- **`switch_to` 栈切换**:切 `rsp` 移交控制权,`prev` 通过特殊方式传递,两个返回点——上下文切换的硬核。
- **调度域分层均衡**:`sched_domain` 按 cache/NUMA 距离分层,近域优先 pull——SMP 均衡的 cache 友好。
- **RT throttling**:限制 RT 任务占 CPU 上限(默认 95%),防失控 RT 进程挂死系统。
- **`SCHED_DEADLINE` 的 CBS + EDF**:适配 CBS 的最早截止期,基于利用率判定可调度性。
- **组调度 + bandwidth throttle**:cgroup cpu 用 `sched_entity` 做层级公平 + period/quota 超额 throttle。
- **NUMA balancing 故障率采样**:以 protnone PTE 采样、估算最优 NUMA 节点。
- **内存序**:唤醒(`wake_up_process`)、抢占、`rq->lock` 的屏障,smp_mb/smp_wmb 的正确使用。
