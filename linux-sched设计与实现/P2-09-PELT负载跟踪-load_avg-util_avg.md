# 第九章 · PELT 负载跟踪:per-entity 的 load_avg/util_avg

> 篇:第 2 篇 · EEVDF 公平调度:下一个跑谁
> 主线呼应:前面三章讲了 EEVDF 的算法核心——lag/deadline/权重。但 EEVDF 回答的只是"在**一个** `cfs_rq` 上,下一个跑谁"。多核场景下还有个更难的问题:**哪个核忙、哪个核闲?一个任务到底"有多忙"?** 这不是 EEVDF 管的,是 **PELT(Per-Entity Load Tracking,每实体负载跟踪)** 管的。PELT 给每个 `sched_entity` 维护一组按几何级数衰减的负载/利用率指标(`load_avg`/`util_avg`/`runnable_avg`),回答"这个任务最近有多忙",驱动负载均衡(第 15 章把忙核的任务 pull 到闲核)和 CPU 频率调节(cpufreq 看核忙不忙决定升降频)。这一章讲清 PELT 的几何衰减原理和 6.9 实现。

## 核心问题

**PELT(Per-Entity Load Tracking,2013 年合入)给每个 `sched_entity` 维护三个衰减的累计量:`load_avg`(负载,带权重)、`util_avg`(利用率,不带权重,0..1024 表示 0..100% CPU)、`runnable_avg`(在队时间占比)。它们都按同一套几何级数衰减:**每 1024µs(≈1ms)一个周期,每过 32 个周期(≈32ms)历史贡献衰减一半**(衰减系数 y 满足 y^32 = 1/2)。这给每个任务一个"近期有多忙"的平滑指标,不会被瞬时尖峰误导、也不会对历史包袱恋恋不舍。PELT 的 `load_avg` 驱动负载均衡(第 15 章),`util_avg` 驱动 CPU 频率调节(cpufreq)和任务放置(EAS 能量感知调度)。**

读完本章你会明白:

1. PELT 的三个量(`load_avg`/`util_avg`/`runnable_avg`)各表什么、存哪些字段(`struct sched_avg`)。
2. 几何级数衰减的数学:`y^32 = 0.5`,32ms 半衰期,`runnable_avg_yN_inv[]` 查表常数时间衰减。
3. `accumulate_sum` 的三段累积(d1/d2/d3),为什么这样设计(避免每个 1024µs 周期都更新,只在跨周期时一次性补算)。
4. `load_avg` vs `util_avg` 的区别(带不带权重),它们各自驱动什么(均衡 vs 频调)。
5. PELT 的局限性:跨 CPU 迁移时的 `sync_entity_load_avg` 同步、`util_est` 预测补偿(避免 PELT 对突发任务的慢响应)。

> **逃生阀**:如果你只关心"下一个跑谁"(EEVDF),可以略读本章。但 PELT 是第 4 篇(负载均衡)和 CPU 频率调节的输入,跳过它你会看不懂第 15 章的"负载"到底指什么。PELT 的几何衰减是它的招牌技巧,务必看一眼 9.3 节的衰减曲线。

## 9.1 一句话点破

> **PELT 把"任务最近有多忙"建模成一个按几何级数衰减的累计和——最近 1ms 的贡献权重 1,32ms 前的贡献权重 1/2,64ms 前 1/4……这样得到一个平滑、对尖峰不敏感、对历史会遗忘的负载/利用率指标。它比"瞬时采样"稳,比"长期平均"灵,正好适合调度器要做"近期决策"的需求。**

这是结论,不是理由。本章倒过来拆:先看 PELT 要解决什么问题(瞬时采样 vs 长期平均的两难),再看几何衰减的数学,然后走读 6.9 的实现,最后讲它的两个下游(负载均衡、频调)和一个补丁(util_est)。

## 9.2 提出问题:负载跟踪为什么难

负载均衡和 CPU 频率调节都依赖一个量:"这个任务/这个核有多忙"。最朴素的两个方案都撞墙:

### 9.2.1 朴素方案一:瞬时采样

每次 tick 看一眼"任务在不在跑",跑就记 1,不跑记 0,最近 N 个 tick 求平均。

> **不这样会怎样**:瞬时采样被尖峰误导。一个任务平时几乎不跑(平均 1%),偶尔连跑 5ms(尖峰),瞬时采样会在这 5ms 内把它判为"100% busy",触发不必要的负载迁移或 CPU 升频;尖峰一过又判回 0,决策抖动。调度器要的是"平滑趋势",不是"瞬时状态"。

### 9.2.2 朴素方案二:长期平均

从任务创建起累计总运行时间 / 总存活时间,得到一个终身平均。

> **不这样会怎样**:长期平均对历史包袱恋恋不舍。一个任务前 1 小时 100% 跑满,之后转 idle,长期平均会很久才降下来——调度器在这段时间一直误判它很忙,把它当成"重负载"拒绝迁移、CPU 持续高频。调度器要的是"近期趋势",历史包袱要快速遗忘。

### 9.2.3 PELT 的答案:几何级数衰减——"平滑 + 遗忘"两手抓

PELT 的解法是**几何级数衰减**:近期贡献权重大、远期权重小,按固定比例连续衰减。具体:

- 把时间切成 1024µs(≈1ms)一个周期。
- 每个周期内,任务"可运行"(runnable,在 `cfs_rq` 上排队或在跑)的时间比例记为 `u_i`(0..1)。
- 整体负载 = `u_0 + u_1*y + u_2*y^2 + ...`(无穷级数),其中 `y^32 = 1/2`——即 32 个周期(≈32ms)前的贡献权重减半,64ms 前减到 1/4,以此类推。

这样:近期的"忙"被充分体现(权重接近 1),历史尖峰被快速遗忘(32ms 衰减一半),整体平滑(几何级数连续衰减)。一个稳态 50% 忙的任务,`load_avg` 稳定在 50%;一个尖峰之后,32ms 内衰减到一半,64ms 到 1/4——既反映了尖峰,又不让它持久影响决策。

> **钉死这件事**:PELT 的核心是"几何级数衰减"——用 `y^32 = 1/2` 的衰减系数,得到一个 32ms 半衰期的负载指标。它既不像瞬时采样那样被尖峰误导,也不像长期平均那样被历史拖累。32ms 这个半衰期是经验值:太短(比如 1ms)会让负载指标抖动剧烈,太长(比如 1s)会让任务负载变化响应太慢。32ms 大约是几个调度 slice(0.75ms * 几十),正好覆盖"近期调度行为"的尺度。

## 9.3 几何衰减的数学:y^32 = 1/2

### 9.3.1 衰减系数 y

PELT 选定 `y^32 = 1/2`,即 `y = 2^(-1/32) ≈ 0.9785`。每过一个周期(1024µs),所有历史贡献乘以 y(衰减约 2.15%)。32 个周期(≈32ms)后,历史贡献累计乘以 `y^32 = 1/2`(半衰期)。

源码不直接用浮点 y(内核避免浮点),而是用 32 个预计算的定点常数 `runnable_avg_yN_inv[]`,表示 `y^n * 2^32`(n=0..31),见 [`kernel/sched/sched-pelt.h:4`](../linux/kernel/sched/sched-pelt.h#L4):

```c
/* Generated by Documentation/scheduler/sched-pelt; do not modify. */
static const u32 runnable_avg_yN_inv[] __maybe_unused = {
	0xffffffff, 0xfa83b2da, 0xf5257d14, 0xefe4b99a, 0xeac0c6e6, 0xe5b906e6,
	0xe0ccdeeb, 0xdbfbb796, 0xd744fcc9, 0xd2a81d91, 0xce248c14, 0xc9b9bd85,
	0xc5672a10, 0xc12c4cc9, 0xbd08a39e, 0xb8fbaf46, 0xb504f333, 0xb123f581,
	0xad583ee9, 0xa9a15ab4, 0xa5fed6a9, 0xa2704302, 0x9ef5325f, 0x9b8d39b9,
	0x9837f050, 0x94f4efa8, 0x91c3d373, 0x8ea4398a, 0x8b95c1e3, 0x88980e80,
	0x85aac367, 0x82cd8698,
};
```

读法:第 0 项是 `y^0 = 1`(即 `2^32` 的全 1 表示),第 32 项是 `y^31`(因为 `y^32 = 0.5`,第 32 项不存在——表里只到 `y^31`,因为再下一个周期就把"32 个周期前的贡献"右移一位即除以 2)。验证:第 0 项 `0xffffffff` = 4294967295 ≈ 2^32 ✓;半衰期处大约在 0x85aac367 ≈ 2.24e9,约一半 ✓(实际 n=30)。

衰减函数 [`decay_load`](../linux/kernel/sched/pelt.c#L31)([pelt.c:31](../linux/kernel/sched/pelt.c#L31)):

```c
/*
 * Approximate:
 *   val * y^n,    where y^32 ~= 0.5 (~1 scheduling period)
 */
static u64 decay_load(u64 val, u64 n)
{
	unsigned int local_n;

	if (unlikely(n > LOAD_AVG_PERIOD * 63))
		return 0;

	/* after bounds checking we can collapse to 32-bit */
	local_n = n;

	/*
	 * As y^PERIOD = 1/2, we can combine
	 *    y^n = 1/2^(n/PERIOD) * y^(n%PERIOD)
	 * With a look-up table which covers y^n (n<PERIOD)
	 *
	 * To achieve constant time decay_load.
	 */
	if (unlikely(local_n >= LOAD_AVG_PERIOD)) {
		val >>= local_n / LOAD_AVG_PERIOD;            /* 整除 32 的部分用右移(除以 2^(n/32)) */
		local_n %= LOAD_AVG_PERIOD;                   /* 余数部分查表 */
	}

	val = mul_u64_u32_shr(val, runnable_avg_yN_inv[local_n], 32);
	return val;
}
```

精妙之处:`y^n` 拆成 `y^(32k+r) = (y^32)^k * y^r = (1/2)^k * y^r`。`(1/2)^k` 直接用右移 k 位实现(常数时间),`y^r` 查表(常数时间)。所以无论 n 多大,`decay_load` 都是 O(1)——这就是注释说的 "constant time decay_load"。如果朴素地乘 n 次 y,会是 O(n),对几千个周期的衰减会慢。

### 9.3.2 衰减曲线的形状

```
 PELT 几何衰减曲线(y^32 = 0.5,32ms 半衰期):

  权重 y^n
  (相对 1.0)
   1.0 │●                              ← n=0(最近 1ms 内),权重 1.0
       │ ●
       │  ●
   0.5 │   ●●●●                         ← n=32(32ms 前),权重 0.5(半衰期)
       │       ●●●
       │          ●●●
   0.25│              ●●●●              ← n=64(64ms 前),权重 0.25
       │                   ●●●●
       │                        ●●●●●●● ← 拖尾(n>96),渐近 0
   0.0 │
       └─────────────────────────────────► n(周期,每周期 1024µs)
       0       32      64      96      ...

  → "近期 32ms" 占了一半权重,"前 64ms" 占 3/4,"前 96ms" 占 7/8
  → 一个稳态 X% 的任务,load_avg 收敛到 X(因为 u_i 恒定,级数和 = X/(1-y) 归一化到 X)
  → 一个尖峰之后,32ms 内衰减一半,~200ms 后基本无影响
```

> **钉死这件事**:衰减系数 `y^32 = 1/2`(32ms 半衰期)是 PELT 的核心参数。它让负载指标"近期敏感、远期遗忘"——既能反映任务的近期忙度变化,又不被远古历史拖累。这个半衰期和调度时间尺度(几个 slice)匹配,是经验上的甜蜜点。`decay_load` 用"右移 + 查表"做到 O(1) 衰减,无论衰减多少周期都是常数时间。

## 9.4 `struct sched_avg`:PELT 的账本

每个 `sched_entity`(和每个 `cfs_rq`)有一个 `struct sched_avg` 字段(见 [`include/linux/sched.h:471`](../linux/include/linux/sched.h#L471),[sched.h:471](../linux/include/linux/sched.h#L471)):

```c
struct sched_avg {
	u64				last_update_time;   /* 上次更新的 PELT 时钟时刻 */
	u64				load_sum;           /* 加权累计和(分子,衰减) */
	u64				runnable_sum;       /* 可运行累计和(无权重,衰减) */
	u32				util_sum;           /* 在跑累计和(无权重,衰减,左移 SCHED_CAPACITY_SHIFT) */
	u32				period_contrib;     /* 当前周期已累积的 1024ns 单位数 */
	unsigned long			load_avg;           /* load_sum / divider,带权重负载 */
	unsigned long			runnable_avg;       /* runnable_sum / divider */
	unsigned long			util_avg;           /* util_sum / divider,CPU 利用率 0..1024 */
	unsigned int			util_est;           /* util_avg 的预测补偿(见 9.6) */
} ____cacheline_aligned;
```

读法:

- `*_sum`(load_sum/runnable_sum/util_sum)是衰减的累计和(几何级数),随时间衰减 + 累积。
- `*_avg`(*_avg)是 `*_sum / divider`,即归一化后的负载/利用率。`divider = LOAD_AVG_MAX - 1024 + period_contrib`(见 [`pelt.h:40`](../linux/kernel/sched/pelt.h#L40) 的 `get_pelt_divider`)。`LOAD_AVG_MAX = 47742`(几何级数在 y^32=0.5 下的无穷和上限,见 [`sched-pelt.h:14`](../linux/kernel/sched/sched-pelt.h#L14))。
- `load_avg` 带 `se->load.weight`(权重),用于负载均衡——nice -20 的任务和 nice 0 的任务都跑 50%,前者的 `load_avg` 是后者的 ~87 倍(权重比)。
- `util_avg` 不带权重(用 `1` 作为 load 系数),范围 0..1024(1024 = 100% 一个 CPU),用于 CPU 频率调节和 EAS——一个 nice 19 和 nice -20 的任务都跑满一个核,`util_avg` 都是 1024(它们对 CPU 频率的需求一样)。

```
 load_avg vs util_avg 的区别(两个任务都跑满一个核 100%):

   任务 A (nice -20, weight 88761):  load_avg ≈ 88761   util_avg ≈ 1024
   任务 B (nice  0, weight  1024):   load_avg ≈  1024   util_avg ≈ 1024
   任务 C (nice 19, weight    15):   load_avg ≈    15   util_avg ≈ 1024

   load_avg:带权重 → 用于负载均衡(权重大的任务迁移代价评估)
   util_avg:不带权重(0..1024) → 用于 CPU 频率调节(满载就该升频,不管 nice)
```

> **钉死这件事**:`load_avg`(带权重)和 `util_avg`(不带权重)是 PELT 的两个一等输出,服务不同下游。`load_avg` 给负载均衡——它知道"这个任务有多重"(nice -20 任务搬走代价大);`util_avg` 给 CPU 频率调节——它知道"这个核有多满"(满载就升频,不管任务 nice)。两者都用同一套几何衰减累积,只是累加时乘不乘权重。

## 9.5 PELT 的更新路径:accumulate_sum 的三段累积

每次任务入队/出队/tick,调度器调 `update_load_avg`(见 [`fair.c:4741`](../linux/kernel/sched/fair.c#L4741))更新这个任务的 PELT 指标。它最终调到 [`___update_load_sum`](../linux/kernel/sched/pelt.c#L180)([pelt.c:180](../linux/kernel/sched/pelt.c#L180))→ [`accumulate_sum`](../linux/kernel/sched/pelt.c#L102)。

### 9.5.1 `___update_load_sum`:把时间差切成周期

[`___update_load_sum`](../linux/kernel/sched/pelt.c#L180) 先算"距离上次更新过了多少 1024ns 单位":

```c
static __always_inline int
___update_load_sum(u64 now, struct sched_avg *sa,
		  unsigned long load, unsigned long runnable, int running)
{
	u64 delta;

	delta = now - sa->last_update_time;
	...
	delta >>= 10;                          /* 除以 1024:把 ns 换成 1024ns 单位 */
	if (!delta)
		return 0;

	sa->last_update_time += delta << 10;

	if (!load)
		runnable = running = 0;
	...
	if (!accumulate_sum(delta, sa, load, runnable, running))
		return 0;

	return 1;
}
```

`delta >>= 10` 把纳秒换算成 1024 纳秒单位(≈1µs)。然后调 `accumulate_sum` 做跨周期的累积。

### 9.5.2 `accumulate_sum`:三段累积(d1/d2/d3)

[`accumulate_sum`](../linux/kernel/sched/pelt.c#L102)([pelt.c:102](../linux/kernel/sched/pelt.c#L102))的注释画了一张精彩的时间轴,把时间差分成三段:

```
           d1          d2           d3
           ^           ^            ^
           |           |            |
         |<->|<----------------->|<--->|
 ... |---x---|------| ... |------|-----x (now)
                          (上次周期边界)

   d1: 上次周期剩余部分(不满一个周期)
   d2: 中间完整的周期数
   d3: 当前周期已过部分(不满一个周期)
```

累积公式(注释 L92-99):

```
                           p-1
   u' = (u + d1) y^p + 1024 \Sum y^n + d3 y^0
                           n=1

      = u y^p +                    (Step 1: 衰减旧 sum)

                        p-1
        d1 y^p + 1024 \Sum y^n + d3 y^0     (Step 2: 加新贡献)
                        n=1
```

源码实现([`pelt.c:102`](../linux/kernel/sched/pelt.c#L102) 简化):

```c
static __always_inline u32
accumulate_sum(u64 delta, struct sched_avg *sa,
	       unsigned long load, unsigned long runnable, int running)
{
	u32 contrib = (u32)delta;       /* p == 0 -> delta < 1024 */
	u64 periods;

	delta += sa->period_contrib;
	periods = delta / 1024;          /* 跨了多少个周期 */

	/* Step 1: 衰减旧 sum */
	if (periods) {
		sa->load_sum = decay_load(sa->load_sum, periods);
		sa->runnable_sum = decay_load(sa->runnable_sum, periods);
		sa->util_sum = decay_load((u64)(sa->util_sum), periods);

		/* Step 2: 累加跨过的完整周期(d1/d2 段) */
		delta %= 1024;
		if (load) {
			contrib = __accumulate_pelt_segments(periods,
					1024 - sa->period_contrib, delta);
		}
	}
	sa->period_contrib = delta;      /* d3 留到下次 */

	if (load)
		sa->load_sum += load * contrib;
	if (runnable)
		sa->runnable_sum += runnable * contrib << SCHED_CAPACITY_SHIFT;
	if (running)
		sa->util_sum += contrib << SCHED_CAPACITY_SHIFT;

	return periods;
}
```

精妙之处:**不每周期更新一次,只在有事件(tick/入队/出队)时一次性补算**。两次更新之间可能跨过几十个周期(任务睡眠了),`accumulate_sum` 用 `decay_load`(O(1))衰减旧 sum,再用 `__accumulate_pelt_segments` 用闭式公式(`LOAD_AVG_MAX - decay_load(LOAD_AVG_MAX, periods) - 1024`)一次算出跨过那段的累积贡献——避免逐周期循环。这是 PELT 性能的关键:无论跨多少周期,更新都是 O(1)。

> **钉死这件事**:`accumulate_sum` 的三段累积(d1/d2/d3)是 PELT 的性能核心。它不在每个 1ms 周期都更新(那要每核每秒 1000 次中断,开销巨大),而是在事件触发(tick/入队/出队)时一次性补算跨过的所有周期。`decay_load` O(1) 衰减旧 sum,`__accumulate_pelt_segments` 用闭式公式算中间段的累积——无论跨多少周期都是常数时间。这是"用数学闭式替代循环"的典范,和 LevelDB 的 bloom filter 闭式哈希是同一类思路。

### 9.5.3 `___update_load_avg`:sum → avg

累积完 sum,调 [`___update_load_avg`](../linux/kernel/sched/pelt.c#L257) 把 sum 归一化成 avg:

```c
static __always_inline void
___update_load_avg(struct sched_avg *sa, unsigned long load)
{
	u32 divider = get_pelt_divider(sa);

	/* Step 2: update *_avg. */
	sa->load_avg = div_u64(load * sa->load_sum, divider);
	sa->runnable_avg = div_u64(sa->runnable_sum, divider);
	WRITE_ONCE(sa->util_avg, sa->util_sum / divider);
}
```

`divider = LOAD_AVG_MAX - 1024 + period_contrib`(见 [`pelt.h:42`](../linux/kernel/sched/pelt.h#L42))。`LOAD_AVG_MAX = 47742` 是几何级数在 y^32=0.5 下的无穷和上限(归一化常数)。这里有一次除法——但 `update_load_avg` 不在 tick 的最热路径上(它和 EEVDF 的决策分离),所以可以接受。

## 9.6 PELT 的两个下游:负载均衡与 CPU 频率调节

PELT 算出的 `load_avg`/`util_avg` 不是给自己看的,是给两个下游消费的:

### 9.6.1 下游一:负载均衡(load_balance,第 15 章)

`load_balance`(第 15 章)决定"把任务从忙核 pull 到闲核"。它怎么判断"忙"和"闲"?用每个核 `cfs_rq` 的 `load_avg`(或 `runnable_avg`)——所有任务的 `load_avg` 加起来。一个核上跑着两个 nice -20 的重任务,它的 `cfs_rq->load_avg` 远高于另一个跑两个 nice 0 任务的核,均衡器就把其中一个任务迁过去。

这里 `load_avg`(带权重)是对的:我们要均衡的是"CPU 占用压力",权重大的任务占压力大,理应被优先迁。如果用 `util_avg`(不带权重),两个 nice -20 任务和两个 nice 0 任务都会显示 200% util,均衡器看不出区别——但前者实际占 CPU 是后者几十倍。

### 9.6.2 下游二:CPU 频率调节(schedutil cpufreq governor)

cpufreq 的 schedutil governor 决定 CPU 升降频。它看 `cfs_rq->util_avg`(不带权重,0..1024)——一个核的 util_avg 接近 1024(满载)就升频,低于某阈值就降频。

这里 `util_avg`(不带权重)是对的:CPU 频率是按"这个核有多满"调的,不管跑的是 nice -20 还是 nice 19——只要核满载就该升频。如果用 `load_avg`(带权重),nice -20 任务会触发虚假升频(它权重大但可能实际只用了一部分 CPU),nice 19 任务该升频却不升。

> **钉死这件事**:`load_avg`(带权重)和 `util_avg`(不带权重)的分工反映了一个深刻区别:**负载均衡关心"任务的相对重量"(权重重要),CPU 频率调节关心"核的绝对占用"(权重无关)**。PELT 同时维护这两个量,用同一套几何衰减累积,只是累加时乘不乘权重。这是 PELT 设计的精髓——一套衰减机制,服务两个语义不同的下游。

## 9.7 技巧精解:PELT 的几何衰减——为什么是 y^32 = 1/2

这一章的招牌技巧是 PELT 的几何衰减本身。我们拆透三个点:为什么是几何级数(不是别的衰减)、为什么是 y^32=1/2(不是别的系数)、为什么用 1024µs 周期(不是别的粒度)。

### 技巧一:为什么几何级数——常数时间衰减的关键

几何级数衰减有个独一无二的性质:**新状态可以由旧状态乘以 y 得到,不需要重新遍历历史**。具体:

```
旧 load_avg = u_0 + u_1*y + u_2*y^2 + ...
新 load_avg = u_0' + y*(u_0 + u_1*y + ...)    (u_0' 是新周期的贡献)
            = u_0' + y * 旧 load_avg
```

更新只需"乘 y + 加新贡献"——O(1)。如果用别的衰减(比如线性衰减、滑动窗口平均),每次更新要重新求和,O(N) 或 O(窗口大小)。几何级数是唯一能 O(1) 增量更新的衰减形式。

> **反面对比**:如果用滑动窗口平均(最近 N 个周期的算术平均),每次窗口滚动要把最老的样本剔除、把新样本加入,要么 O(N) 重算,要么维护一个 O(N) 的环形缓冲。任务数一多(系统里几百个任务),每个任务的 PELT 维护成本累加,显著拖慢调度路径。几何级数用 O(1) 增量更新,几百个任务的 PELT 维护总成本可控。这是 PELT 选几何级数的根本理由——不是数学美感,是常数时间更新的工程优势。

### 技巧二:为什么 y^32 = 1/2——32ms 半衰期的甜蜜点

衰减系数的选择是个权衡:y 大(接近 1)衰减慢,负载指标滞后;y 小(远小于 1)衰减快,负载指标抖动。`y^32 = 1/2` 对应 32ms 半衰期,是个经验甜蜜点:

- **覆盖几个调度 slice**:一个 slice 是 0.75ms,32ms ≈ 43 个 slice——正好覆盖"近期调度行为"的尺度。
- **对尖峰遗忘够快**:一个 5ms 的尖峰,32ms 后衰减到一半,64ms 到 1/4,~200ms 后基本无影响——既反映尖峰,又不让它持久扭曲决策。
- **对稳态响应够稳**:一个稳态 50% 的任务,大约 100-200ms 收敛到 50%(几个半衰期)——不会因为启动慢就长期低估。

如果半衰期太短(比如 1ms),任务一 sleep 一 wake,负载指标剧烈抖动,均衡器和频调疲于奔命;如果太长(比如 1s),任务负载变化要几秒才反映出来,均衡和频调响应迟钝。32ms 是经验上的折中。

### 技巧三:为什么 1024µs 周期——和 SCHED_FIXEDPOINT_SHIFT 对齐

周期粒度 1024µs(1ms)选得也巧:**1024 = 2^10**,所以 `delta >>= 10`(除以 1024)是快速移位(见 [`___update_load_sum`](../linux/kernel/sched/pelt.c#L199) 的 `delta >>= 10`)。如果周期是 1000µs,要除以 1000(慢除法)。1024 这个数和内核的 `SCHED_FIXEDPOINT_SHIFT = 10`(见 [`include/linux/sched.h:414`](../linux/include/linux/sched.h#L414))、PELT 的几何级数实现都协调——所有定点运算都用 2 的幂移位,避免除法。

> **钉死这件事**:PELT 的三个参数(几何级数衰减形式、y^32=1/2 系数、1024µs 周期)都是精心选的工程甜蜜点:几何级数给 O(1) 增量更新,y^32=1/2 给合适的"近期敏感/远期遗忘"平衡,1024µs 给快速移位运算。这套设计让 PELT 在不显著拖慢调度路径的前提下,给每个任务一个平滑、对尖峰不敏感、对历史会遗忘的负载/利用率指标。这是内核"用数学换性能"的典范。

## 9.8 PELT 的局限与补丁:util_est

PELT 有个先天局限:**慢启动**。一个任务刚唤醒开始 100% 跑,它的 `util_avg` 从 0 开始累积,要几个半衰期(~200ms)才收敛到 1024。在这段时间里,cpufreq 看到的 util_avg 偏低,CPU 不升频,任务实际跑得慢(频率不够)——这对交互式/突发任务不友好(用户感受到卡顿)。

为弥补这个,Linux 加了 `util_est`(见 [`struct sched_avg`](../linux/include/linux/sched.h#L480) 的 `unsigned int util_est;` 字段,和 ` fair.c` 的 `util_est_update`):它记录任务**历史峰值** util_avg 的指数移动平均(EMA),作为 util_avg 的"下限"。一个任务之前跑到过 util_avg=800,睡醒后即使 PELT 还没追上,util_est 也给它至少 800 的估计——cpufreq 立刻升频,避免慢启动卡顿。

这是个"在 PELT 之上打补丁"的工程,反映 PELT 的固有局限(几何衰减对启动慢)。类似的补丁还有 `sync_entity_load_avg`(跨 CPU 迁移时同步 PELT 时钟,避免迁移后 PELT 跳变)。

> **钉死这件事**:PELT 不是银弹——它的几何衰减对"任务启动"响应慢(util_avg 从 0 爬到稳态要几百毫秒)。`util_est` 用历史峰值的 EMA 给一个下限补偿,缓解这个问题。这是 Linux 调度器"PELT + util_est"组合拳的来由——PELT 给平滑趋势,util_est 给快速响应。

## 章末小结

这一章是 EEVDF 的"负载支撑",我们立清了四样东西:

1. **PELT 解决的问题**:瞬时采样被尖峰误导、长期平均被历史拖累——PELT 用几何级数衰减(32ms 半衰期)得到平滑且会遗忘的负载指标。
2. **三个量**:`load_avg`(带权重,服务负载均衡)、`util_avg`(不带权重 0..1024,服务 CPU 频调)、`runnable_avg`(在队时间占比)。存在 `struct sched_avg` 的 `*_sum`(衰减累计和)和 `*_avg`(归一化值)字段里。
3. **几何衰减的实现**:`y^32=1/2`,`runnable_avg_yN_inv[]` 32 项查表 + 右移实现 O(1) 衰减;`accumulate_sum` 三段累积(d1/d2/d3)用闭式公式跨多周期补算,O(1) 更新。
4. **两个下游 + util_est 补丁**:`load_avg` 给负载均衡(第 15 章),`util_avg` 给 CPU 频率调节(schedutil)。PELT 慢启动的局限由 `util_est`(历史峰值 EMA 下限)缓解。

本章还讲了招牌技巧——**PELT 几何级数衰减**,三个参数(几何级数形式、y^32=1/2、1024µs 周期)都是工程甜蜜点,核心优势是 O(1) 增量更新。

本章服务二分法的**支撑**面:PELT 不决定"下一个跑谁"(那是 EEVDF 的策略),它为负载均衡(第 15 章,机制)和 CPU 频率调节提供输入——它是策略和均衡机制之间的数据桥梁。

### 五个"为什么"清单

1. **PELT 解决什么问题?** "任务最近有多忙"这个量,瞬时采样会被尖峰误导,长期平均会被历史拖累。PELT 用几何级数衰减(32ms 半衰期)得到平滑且会遗忘的指标。
2. **load_avg 和 util_avg 有什么区别?** load_avg 带权重(`se->load.weight`),服务负载均衡(权重大的任务迁走代价大);util_avg 不带权重(0..1024 = 0..100% 一个 CPU),服务 CPU 频率调节(满载就升频,不管 nice)。两者都用同一套几何衰减累积,只是累加时乘不乘权重。
3. **为什么用几何级数?** 几何级数是唯一能 O(1) 增量更新的衰减形式(新状态 = y * 旧状态 + 新贡献)。其他衰减(滑动窗口)要 O(N) 重算。任务数多时,O(1) 是必须的。
4. **y^32 = 1/2 怎么实现?** 32 项查表 `runnable_avg_yN_inv[]`(y^0..y^31 的 2^32 定点表示),整周期部分用右移(除以 2^(n/32)),余数部分查表——O(1) 衰减,无论 n 多大。
5. **PELT 慢启动怎么补?** PELT 的 util_avg 从 0 爬到稳态要几百毫秒。`util_est` 记录历史峰值 util_avg 的 EMA 作为下限,任务睡醒后即使 PELT 没追上,util_est 也给一个保底估计,cpufreq 立刻升频,避免慢启动卡顿。
