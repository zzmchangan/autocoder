# 全书第一性原理(贯穿始终,反复回扣)

> **RocketMQ 把所有 Topic 的所有消息一股脑顺序追加进一个 CommitLog,用纯顺序写换极致写入吞吐(Topic/Queue 再多也不退化为随机写);代价是消费端面对"混写的大文件",于是用 ConsumeQueue 重建逻辑队列、IndexFile 建 key 索引、mmap+sendfile 零拷贝把随机读代价收回;再靠 NameServer 路由、主从复制、刷盘策略守住"不丢",靠 Rebalance 与消费位点守住"不重不漏"。**

每写一章,先问自己:这一章讲的机制,是在**让存储内核把消息高效地只追加一次、又高效地读出来**(存储内核),还是在**让消息在集群里可靠流转、不丢不重不漏**(分布式骨架)?把答案写进"为什么"里。读者要的不是"这个 API 怎么用",而是"它凭什么这么设计、源码里那段 `putMessageLock` / `ReputMessageService` / `RouteInfoManager` / `PullMessageProcessor` 在干什么"。

# 写作方法命脉:动机(why) + 技巧(how) 双线 ⚠️

这是本书区别于 API 文档、一般源码导读的关键。RocketMQ 源码充斥 Java 系统级技巧(`MappedByteBuffer` mmap、`transferTo` sendfile、`TransientStorePool` 堆外内存池、`putMessageLock` 三选一、`AdaptiveBackOffSpinLock`、`ThreadLocal` 编码器、`ReputMessageService` 异步分发、`ConcurrentHashMap`+`ReadWriteLock` 分层并发、`CompletableFuture` 全链路异步、`SemaphoreReleaseOnlyOnce` CAS、Netty 主从 Reactor、`HashedWheelTimer` 时间轮、5.x `TimerWheel` 延时)。读者"一知半解"的缺口**不在动机**(好懂),**在技巧**(看不懂)。所以每个关键代码点必须讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?
2. **实现技巧(how 的精妙)**:用了什么手段?为什么妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

**每章单开"技巧精解"小节**(正文后、小结前),挑本章**最硬核 1~2 个技巧**,配真实源码 + 行号,单独拆透,且必须有**反面对比**("朴素地写会撞什么墙")。**讲不清技巧,等于没讲 RocketMQ。**

# 基调:直球讲透为主,比喻只在反直觉处点睛 ⚠️ 延续《LevelDB》《etcd》

《Tokio》用"餐厅服务员"贯穿,因为 async 运行时整体反直觉。RocketMQ 的主体机制大多**具体可见**——CommitLog 就是顺序追加的大文件、ConsumeQueue 就是 20 字节一单元的紧凑数组、Rebalance 就是对两个排好序的列表做分配——这些**直球讲清楚最清楚**,硬套贯穿比喻反而啰嗦。

但有几处**反直觉**,允许适度比喻点睛(点到为止,不贯穿、不另起体系、不展开成对照表):
- **为什么所有 Topic 混写一个 CommitLog**(可点一句"像超大的流水账本,所有业务往一本账上记,极快;想查某类账靠后面的分类索引")。
- **Push 本质是 Pull**(可点一句"像订的报纸不是报社塞进门,而是报童每隔一会儿去问'要新的吗'")。
- **零拷贝**(可点一句"像快递不经过中转仓,从产地车直接换挂到配送车")。
- **时间轮**(对标《Tokio》时间轮那章的比喻)。

**判断标准**:直球讲不清,才上比喻;讲得清,就直球。

# 配图策略(mermaid 为主,ASCII 布局为辅)

RocketMQ 的图是**流程 / 时序 / 状态 / 文件与内存布局**:

- **时序图**(一次 send:Producer → Remoting → SendMessageProcessor → CommitLog.asyncPutMessage → Reput 分发;一次 Pull 长轮询;同步双写 GroupTransferService 等待)、**状态图**(broker Live/Dead;master/slave 切换;延时消息 SCHEDULE_TOPIC → 到期写回)、**流程图**(Reput 分发 dispatcher 责任链;Rebalance doRebalance;主从复制推送):用 **mermaid**(`sequenceDiagram` / `stateDiagram` / `flowchart`),VSCode 预览 + GitHub 原生渲染。
- **文件 / 内存 / 数据结构布局**(一条消息在 CommitLog 的字节布局、ConsumeQueue 的 20 字节单元、IndexFile 的 slot+链表、CommitLog 一组 1GB MappedFile 的滚动、时间轮 slot):用 **ASCII 框图或表格**(对标 LevelDB/数据库系列画文件布局的风格)。例:
  ```
   ConsumeQueue 一个单元 = 20 字节定长:
   ┌──────────────┬──────────┬──────────────┐
   │ offset(8B)    │ size(4B) │ tagHash(8B)   │  ← 回 CommitLog 取消息体
   └──────────────┴──────────┴──────────────┘
   consumeOffset 即数组下标 → 偏移 = offset × 20,O(1) 定位
  ```
- **铁律**:几何 / 坐标才需要 PNG;流程 / 结构 / 布局用 mermaid 或 ASCII 框图。不要用 ASCII 画时序(用 mermaid)。
