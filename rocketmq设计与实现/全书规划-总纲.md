## 一、这本书的定位

### 目标读者

会写 Java、用过 RocketMQ(或 Kafka)的 `DefaultMQProducer.send` / `DefaultMQPushConsumer`、配过 NameServer/Broker、甚至翻过 `CommitLog.java`、`DefaultMessageStore.java`,但**一知半解**:知道有 CommitLog、ConsumeQueue、IndexFile、ReputMessageService、主从复制、Rebalance,却讲不清一条消息怎么从 Producer 到 Broker 的 CommitLog(为什么所有 Topic 混写一个文件)、怎么被后台线程异步分发成 ConsumeQueue 让消费端能快速定位、怎么靠 mmap+sendfile 零拷贝送到网卡;也讲不清 `putMessageLock` 那把"自旋/重入/自适应三选一"的锁、`AdaptiveBackOffSpinLock`、`TransientStorePool` 堆外内存池、`ReputMessageService`、`RouteInfoManager` 五张表、`PullMessageProcessor` 的长轮询到底在干什么。和"读过 tokio/leveldb/etcd 源码但不理解"是同一种病。

### 全书一句话主旨

> **RocketMQ 把所有 Topic 的所有消息一股脑顺序追加进一个 CommitLog(不像 Kafka 每个 Partition 一个文件),用纯顺序写换极致写入吞吐(Topic/Queue 再多也不退化为随机写);代价是消费端面对的是"混写的大文件",于是用 ConsumeQueue 重建逻辑队列、IndexFile 建 key 索引、mmap 写 + sendfile 读零拷贝,把这条随机读的代价收回来;再靠 NameServer 路由、主从复制、刷盘策略守住"不丢",靠 Rebalance 与消费位点守住"不重不漏"。**

每个组件,都是这条旅程上的一个**驿站**。读完一本书,读者能**在脑子里放映出一条消息的一生**:Producer 怎么选 queue 发、Broker 的 `SendMessageProcessor` 怎么接、`CommitLog.asyncPutMessage` 怎么加锁串行化追加(为什么锁内要重新设 storeTimestamp 保证全局有序)、后台 `ReputMessageService` 怎么异步把它分发进 ConsumeQueue 和 IndexFile、`PullMessageProcessor` 怎么靠长轮询把消息递给 Consumer、`RebalanceImpl` 怎么把 queue 分给消费组内的 consumer、消费位点怎么上报——以及故障时 NameServer 怎么发现死 broker、主从怎么复制、自动怎么切换。每一步底下都用了什么巧妙的手段,都要讲透。

### 为什么用"一条消息的旅程"当骨架(而非按模块平铺)

RocketMQ 的模块高度耦合:`client`(生产/消费)→ `remoting`(通信)→ `broker`(Processor 路由)→ `store`(CommitLog/ConsumeQueue/IndexFile 存储内核)→ `namesrv`(路由发现)→ `ha`(复制)→ `timer`/`transaction`(特性)。按模块百科式平铺,读者会觉得每块都是孤岛、不知道**为什么要学这个**。用"一条消息怎么进来、怎么落地、怎么被高效读出、怎么被消费、故障时怎么不丢不乱"这条旅程当骨架,每个模块都是"为了让旅程继续而必须出现的角色"——读者永远知道**我在哪、为什么需要下一站**。横向的基础设施(Remoting 通信、NameServer 路由、HA 复制)和增强特性(顺序/延时/事务),则作为旅程两翼的支撑。

## 二、写作方法

### 命脉:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与 API 文档、一般源码导读最关键的分野,所有写作者必读、必守。**

RocketMQ 源码里充斥 Java 系统级与并发技巧:`MappedByteBuffer`(mmap 内存映射)与 `FileChannel.transferTo`(sendfile 零拷贝)、`TransientStorePool` 堆外内存池(写堆外再 commit,避开页缓存锁竞争)、`putMessageLock` 三选一(`PutMessageSpinLock` 自旋 / `PutMessageReentrantLock` 重入 / `AdaptiveBackOffSpinLock` 自适应退避)、`TopicQueueLock` 分段锁、`beginTimeInLock` 锁内重设 storeTimestamp 保全局有序、`ThreadLocal<PutMessageThreadLocal>` 编码器避免竞争、`ReputMessageService` 异步分发解耦写与建索引、`ConcurrentHashMap` + `ReadWriteLock` 分层并发(RouteInfoManager)、`CompletableFuture` 全链路异步、`SemaphoreReleaseOnlyOnce` CAS 一次性释放、`LengthFieldBasedFrameDecoder` 解决粘包、Netty 主从 Reactor 三组线程、`channelWritabilityChanged` 背压反压、`HashedWheelTimer` 时间轮做超时扫描、5.x `TimerWheel` 时间轮延时消息。用户自己读过源码,"一知半解"的缺口往往**不在**"它要解决什么问题"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?(沿用系列一贯的"为什么优先")
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果不用这个技巧、朴素地写,会撞上什么墙",让妙处显形。

一句话:**讲不清技巧,等于没讲 RocketMQ。**

### 基调:直球讲透为主,比喻只在反直觉处点睛 ⚠️ 延续《LevelDB》《etcd》

> **这一节延续《LevelDB》《etcd》由用户拍板的基调。**

《Tokio》用"餐厅服务员"贯穿,因为 async 运行时整体反直觉。RocketMQ 的主体机制大多**具体可见**——CommitLog 就是顺序追加的大文件、ConsumeQueue 就是 20 字节一单元的索引、ConsumeQueue 就是个紧凑数组、Rebalance 就是对两个排好序的列表做分配、HA 就是 master 往 slave 推 CommitLog——这些**直球讲清楚最清楚**,硬套贯穿比喻反而啰嗦。

但 RocketMQ 有几处**反直觉**,允许适度比喻点睛:
- **为什么所有 Topic 混写一个 CommitLog**(反直觉:消息不该按 Topic 分文件吗?)——可点一句"像一个超大的流水账本,所有业务往一本账上记,极快;想查某类账,靠后面单独做的分类索引"。
- **Push 本质是 Pull**(反直觉:用户调的是 push)——可点一句"像你订的报纸不是报社主动塞进门,而是报童每隔一会儿去问你'要新的吗'"。
- **零拷贝**(反直觉:数据怎么不经过 JVM 就到了网卡)——可点一句"像快递不经过中转仓,从产地车直接换挂到配送车"。
- **时间轮延时**(反直觉:海量定时任务怎么不爆炸)——对标 Tokio 时间轮那章的比喻,点到为止。

**判断标准**:直球讲不清的,才上比喻;讲得清的,就直球。比喻只点睛,**不贯穿,不另起体系,不展开成对照表,不强制每章出现**。

## 三、全书主线、二分法

### 主线

> **如何用"所有消息混写一个 CommitLog、纯顺序追加"换极致写入吞吐,代价是消费端的随机读,靠 ConsumeQueue/IndexFile/零拷贝把代价收敛回来,再靠 NameServer/HA/刷盘守住不丢、靠 Rebalance/位点守住不重不漏。**

任何一处看不懂某个机制,回到这句问:"这是在**让存储内核把消息高效地只追加一次、又高效地读出来**(存储内核),还是在**让消息在集群里可靠流转、不丢不重不漏**(分布式骨架)?"把答案写进"为什么"里。

### 二分法(迷路时回到它)

本书的二分法,正是 RocketMQ 存储引擎与集群架构的天然分野:

> **存储内核(CommitLog/ConsumeQueue/IndexFile:消息怎么只追加一次、又怎么被各种姿势高效读出)vs 分布式骨架(NameServer/Remoting/HA/刷盘/Rebalance:消息怎么可靠流转、不丢不重不漏)。**

- **存储内核这一面**:CommitLog(全局顺序追加)、ConsumeQueue(逻辑队列重建)、IndexFile(key 哈希索引)、零拷贝(mmap/sendfile/堆外内存池)、Reput 异步分发、刷盘。源码主体在 `store` 模块。
- **分布式骨架这一面**:Remoting(Netty 主从 Reactor + 协议)、NameServer(路由发现)、HA(主从复制/DLedger/Controller)、Rebalance(queue 分配)、消费位点、长轮询。源码在 `remoting` / `namesrv` / `broker` / `client` / `store/ha` 模块。

RocketMQ 每个机制都落在这两面之一(或两者衔接处——`SendMessageProcessor` / `PullMessageProcessor` 就是衔接)。任何时候迷路,回到这个二分法。

> 注:`store` 模块既是存储内核的大本营,也藏着 `ha/`(复制,分布式骨架)和 `timer/`(延时,特性)两个子目录。讲的时候要说明"它在二分法的哪一面"。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线是一条旅程:**为什么混写一个 CommitLog → 一条消息怎么编码进 CommitLog → 怎么顺序追加、怎么刷盘 → 后台怎么异步分发成 ConsumeQueue/Index → 消费端怎么靠索引快速读出 → 消费者怎么拉、怎么分 queue、怎么记位点 → 底层通信 Remoting → 路由 NameServer → 高可用 HA → 特性消息 → 收束**。
> 章号全局连续;命名 `P{篇}-{章}-标题.md`,开篇第 1 章 = `P0-01`,第 1 篇第 2 章 = `P1-02`,以此类推。**全书 24 章**(HA 拆 3 章 + 5.x 新架构单独成章),对标《Tokio》《LevelDB》《etcd》体量。

### 第 0 篇 · 开篇:为什么把所有消息堆进一个 CommitLog(定调,1 章)

- **第 1 章 · 第一性原理:为什么把所有消息堆进一个 CommitLog**(`P0-01`):消息中间件要解决什么(生产消费解耦、削峰填谷、异步)→ Kafka 的取舍(每 Partition 一个 log,多 Partition 并发写多文件,读快但要管理海量文件、Partition 数受限于并发顺序写)→ RocketMQ 的取舍(所有 Topic 混写一个 CommitLog,**纯顺序写吞吐极致,Topic/Queue 再多也不退化为随机写**,代价是消费要重建逻辑队列)→ 三个文件各司其职(CommitLog 落地 / ConsumeQueue 逻辑队列 / IndexFile key 索引)→ RocketMQ 四角色(Producer/Broker/Consumer/NameServer)全貌 → 立起"存储内核 vs 分布式骨架"二分法。**【样本章节,见后】**

### 第 1 篇 · 存储内核之写入:CommitLog 与刷盘(4 章)

> 一次写怎么进来、怎么编码、怎么顺序追加进 CommitLog、怎么刷盘、后台怎么异步分发。这一篇是全书地基,源码主体在 `store` 模块。

- **第 2 章 · 消息的编码:一条消息在 CommitLog 里长什么样**(`P1-02`)——为什么不直接存裸 body(要带 topic/queueId/offset/storeTimestamp/bornHost/properties 等元信息);一条消息在 CommitLog 里的字节布局(TOTALSIZE / MAGICCODE / BODYCRC / QUEUEOFFSET / PHYSICALOFFSET / SYSFLAG / BORN TIMESTAMP/HOST / STORE TIMESTAMP/HOST / RECONSUMETIMES / ... / BODY / TOPIC / PROPERTIES);`MessageExtEncoder` 怎么把它编成这段字节。源码:`store/.../MessageExtEncoder.java`、`common/.../message/MessageDecoder.java`。**技巧精解:`MessageExtEncoder` + `ThreadLocal<PutMessageThreadLocal>`**——每线程一个编码器 + 复用 ByteBuffer,避免高并发下的编码器竞争与对象分配;为什么字段顺序这么排(queueOffset/physicalOffset 紧挨着,方便消费端 ConsumeQueue 索引)。
- **第 3 章 · CommitLog:全局顺序追加**(`P1-03`)——为什么所有 Topic 的消息都往一个 `MappedFileQueue`(一组 1GB 的 `MappedFile`)末尾追加;`CommitLog.asyncPutMessage` 的流程(`topicQueueLock` 分配 queueOffset → encode → `putMessageLock.lock()` 锁内 `getLastMappedFile` + `appendMessage`);**为什么锁内要重新设 storeTimestamp**(`beginTimeInLock`,"in order to guarantee an orderly global"——全局顺序);`MappedFileQueue` 怎么滚动建新文件。源码:`store/.../CommitLog.java`(`asyncPutMessage` :969、`putMessageLock` :100/:1057、`beginTimeInLock` :98)、`store/.../MappedFileQueue.java`、`store/.../logfile/DefaultMappedFile.java`。**技巧精解:`putMessageLock` 三选一(自旋/重入/`AdaptiveBackOffSpinLock` 自适应退避)+ `topicQueueLock` 分段锁**——为什么 CommitLog 写要加锁(全局顺序追加注定串行化),三种锁各自的取舍(自旋省唤醒但有 CPU 空转、重入无空转但有 park/unpark 开销、自适应在两者间切换);反面对比"不加锁会怎样"(storeTimestamp 乱序、物理偏移冲突)。
- **第 4 章 · 刷盘策略:同步刷盘 vs 异步刷盘**(`P1-04`)——为什么写到 mmap 的 `MappedByteBuffer` 还不算落盘(只是页缓存,掉电会丢);同步刷盘(`FlushDiskType.SYNC_FLUSH`)怎么靠 `GroupCommitService` 等待 force 完成、为什么不阻塞前台(用等待队列);异步刷盘(`ASYNC_FLUSH`)的 `FlushRealTimeService` 定时 force;`FlushManager` 抽象;为什么 RocketMQ 区分"commit(堆外→页缓存)"与"flush(页缓存→磁盘)"两步。源码:`store/.../FlushManager.java`、`store/.../config/FlushDiskType.java`、CommitLog 内部 `GroupCommitService` / `FlushRealTimeService`。**技巧精解:同步刷盘的等待队列(`GroupCommitService`)**——前台把"待刷 offset + CountDownLatch"塞队列,刷盘线程 force 完逐个唤醒,凭什么前台不阻塞 IO 线程又能等到落盘;反面对比"前台自己 force"会怎样(每条都同步系统调用,吞吐崩塌)。
- **第 5 章 · ReputMessageService:从 CommitLog 异步分发到 ConsumeQueue/Index**(`P1-05`)——**全书最反直觉的转折**:写只写 CommitLog,ConsumeQueue 和 IndexFile 不是写时同步建,而是后台 `ReputMessageService` 线程顺着 CommitLog 一条条读、解析成 `DispatchRequest`、再分发给 `CommitLogDispatcherBuildConsumeQueue` 和 `CommitLogDispatcherBuildIndex`;为什么这么设计(写路径只管纯顺序追加,极快;建索引的脏活留后台,不拖慢写)。源码:`store/.../DefaultMessageStore.java`(`ReputMessageService` :2657、`CommitLogDispatcherBuildConsumeQueue` :2240、`CommitLogDispatcherBuildIndex` :2257)、`store/.../CommitLogDispatcher.java`(:25 接口)、`store/.../DispatchRequest.java`。**技巧精解:Reput 的"写-建索引"解耦 + dispatcher 列表**——`dispatcherList`(Build ConsumeQueue + Build Index + 可选 Compaction)是责任链,加一种索引只加一个 dispatcher,写路径零改动;反面对比"写时同步建 ConsumeQueue"会怎样(写退化成"顺序追加 + 随机写索引",吞吐被拖垮)。

### 第 2 篇 · 存储内核之读取:ConsumeQueue、IndexFile、零拷贝(3 章)

> 消费端怎么从"混写的大文件 CommitLog"里,快速定位到某个 topic-queue 的消息、按 key 查消息、把消息零拷贝送到网卡。

- **第 6 章 · ConsumeQueue:逻辑队列的重建**(`P2-06`)——CommitLog 里所有消息混写,消费端"我要 topic=T 的 queue=Q 从 offset=O 开始的消息"怎么找?答案:每个 topic-queue 一个 ConsumeQueue,每条消息在 ConsumeQueue 里占固定 **20 字节**(8 字节 CommitLog 物理偏移 + 4 字节消息总长 + 8 字节 tag hashcode);`ConsumeQueue` 怎么按 consumeOffset 二分定位、再回 CommitLog 取消息体。源码:`store/.../ConsumeQueue.java`、`store/.../queue/ConsumeQueueStore.java`、`store/.../queue/CqUnit.java`。**技巧精解:ConsumeQueue 的 20 字节定长单元 + ConsumeOffset 即数组下标**——凭什么 consumeOffset 能 O(1) 定位(定长 → 乘法算偏移);tag hashcode 怎么做服务端 tag 过滤的第一道筛(不用读 CommitLog 的消息体);反面对比"不定长索引"会怎样(不能直接算偏移,要二分/扫描)。
- **第 7 章 · IndexFile:按 key 的哈希索引**(`P2-07`)——除了按 topic-queue-offset 顺序消费,RocketMQ 还要支持"按 msgId / 业务 key 查消息";`IndexFile` 怎么用一个 hash 索引(500w hash slot + 2000w 条目)+ 链表解决冲突;`IndexService` 管多个 IndexFile、按时间范围查。源码:`store/.../index/IndexFile.java`、`store/.../index/IndexService.java`、`store/.../index/IndexHeader.java`。**技巧精解:IndexFile 的 hash + 链地址法 + 单文件定长**——为什么 slot 指针和链表节点都存在同一个文件里(slot table 固定在前、链表条目在后),既省内存又能 mmap;反面对比"用 HashMap"会怎样(内存放不下海量消息的 key,重启即丢)。
- **第 8 章 · 零拷贝:mmap 写 + sendfile 读 + 堆外内存池**(`P2-08`)——写消息用 `MappedByteBuffer`(mmap)直接写页缓存;读消息给消费端时用 `FileChannel.transferTo`(底层 sendfile)把页缓存直接送到 socket,**全程不进 JVM 堆**;`TransientStorePool` 堆外内存池怎么让写"先写堆外 DirectByteBuffer、后台再 commit 进 mappedFile",避开高并发下页缓存的锁竞争;`SelectMappedBufferResult` 怎么切片返回。源码:`store/.../logfile/DefaultMappedFile.java`(`appendMessage` mmap 写、`selectMappedBuffer` 切片)、`store/.../TransientStorePool.java`、broker 端 `PullMessageProcessor` 触发 sendfile 处。**技巧精解:mmap vs sendfile 的分工 + 堆外内存池**——为什么写用 mmap(用户态直接写页缓存,少一次拷贝)、读发给消费者用 sendfile(页缓存→网卡,零 CPU 拷贝、零进 JVM);反面对比"读写都走 ByteBuffer 进 JVM 堆"会怎样(两次拷贝 + GC 压力 + 吞吐腰斩)。

### 第 3 篇 · 消费:拉取、Rebalance、位点(3 章)

> 消息存好了、能高效读了。这一篇讲消费端:Push 为什么本质是 Pull 长轮询、消费组内的 consumer 怎么分 queue、消费到哪了怎么记。

- **第 9 章 · 消费模型:Push 本质是 Pull 长轮询**(`P3-09`)——用户调的是 `DefaultMQPushConsumer`(push 语义),底层却是 `PullMessageService` 单线程不断拉(本质 Pull);没有消息时 broker 端 `PullRequestHoldService` 把请求挂起(长轮询),消息到达由 `NotifyMessageArrivingListener` 唤醒返回;`PullMessageProcessor` 怎么处理 pull 请求、怎么算从哪个 consumeOffset 开始、怎么把消息零拷贝回去。源码:`client/.../impl/consumer/DefaultMQPushConsumerImpl.java`、`client/.../impl/consumer/PullMessageService.java`、`broker/.../processor/PullMessageProcessor.java`、`store/.../MessageArrivingListener.java`。**技巧精解:长轮询(suspend timeout)**——反面对比"短轮询(消费者不停问)"会怎样(空转刷爆 CPU/网络)、"纯推送(broker 主动发)"会怎样(消费慢时 broker 要维护每消费者状态、背压难做);长轮询兼顾实时性与无状态。
- **第 10 章 · Rebalance:Queue 怎么分配给 Consumer**(`P3-10`)——一个消费组有多个 consumer,topic 的多个 queue 怎么分给它们;`RebalanceImpl.doRebalance` → `rebalanceByTopic`(CLUSTERING 模式:取 `mqAll` 全部 queue + `cidAll` 全部 consumer id,都 `Collections.sort` 保证全网一致,再调 `AllocateMessageQueueStrategy.allocate`);默认 `AllocateMessageQueueAveragely`(均分)、还有 `AveragelyByCircle`(轮分)、`ConsistentHash`(一致性哈希 + 虚拟节点);consumer 增减时 `RebalanceService` 20s 一次触发重算。源码:`client/.../impl/consumer/RebalanceImpl.java`(`doRebalance` :168、`rebalanceByTopic` :195)、`client/.../consumer/rebalance/AllocateMessageQueueAveragely.java`、`common/.../consistenthash/`。**技巧精解:无中心协调的确定性分配 + 一致性哈希**——凭什么全 consumer 各自算出互斥结果(双排序 + 同一确定性算法,无需协调者);反面对比"需要中心协调分配"会怎样(单点 + 通信开销);一致性哈希 + 虚拟节点怎么让扩缩容只迁移少量 queue。
- **第 11 章 · 消费位点:ConsumeOffset 与幂等**(`P3-11`)——消费到哪了(offset)怎么记、存哪、重启怎么恢复;CLUSTERING 模式 `RemoteBrokerOffsetStore`(内存 offsetTable + 定时上报 broker)、广播模式 `LocalFileOffsetStore`(本地文件);broker 端 `ConsumerOffsetManager` 持久化;`increaseOnly` 为什么只增不减;RocketMQ 的"至少一次"语义→消息可能重复消费→为什么业务要做幂等。源码:`client/.../consumer/store/RemoteBrokerOffsetStore.java`、`broker/.../offset/ConsumerOffsetManager.java`、`client/.../impl/factory/MQClientInstance.java`(`persistAllConsumerOffset` 定时任务)。**技巧精解:offset 的内存缓冲 + 批量上报 + increaseOnly**——凭什么 5s 上报一次够用(崩了最多丢 5s 的进度,从 broker 上次位点重拉,业务幂等兜底);increaseOnly 防什么(并发上报乱序导致位点回退、重复消费爆炸)。

### 第 4 篇 · Remoting:协议、Netty、Processor(3 章)

> 前三篇讲存储与消费,这一篇沉到底层:所有跨进程通信都走 `remoting` 模块。讲透 RocketMQ 自研的 RPC 协议、Netty 主从 Reactor 线程模型、Processor 路由与三种调用语义。

- **第 12 章 · RemotingCommand:协议的字节布局与零拷贝编码**(`P4-12`)——所有请求/响应都是一个 `RemotingCommand`;wire 格式 = `[4字节 totalLength][4字节 headerLength(高1位塞 SerializeType 编码)][header][body]`;header 用 `extFields`(HashMap) + 反射互转 `CommandCustomHeader`;`fastEncodeHeader` 零拷贝直写 Netty `ByteBuf`;`RequestCode` 是请求类型的整数枚举。源码:`remoting/.../protocol/RemotingCommand.java`、`remoting/.../protocol/RocketMQSerializable.java`。**技巧精解:headerLength 高位塞 SerializeType + 反射缓存 `CLASS_HASH_MAP`**——凭什么一个 4 字节同时表达长度和序列化类型(位域复用);反面对比"单独一个字节标序列化类型"会怎样(破坏 4 字节对齐、多一次解析);`CLASS_HASH_MAP` 怎么缓存反射结果避免每次扫字段。
- **第 13 章 · Netty 主从 Reactor 线程模型与背压**(`P4-13`)——`NettyRemotingServer` 三组线程:`eventLoopGroupBoss`(1 个,accept)、`eventLoopGroupSelector`(N 个,read/write,Linux 用 epoll)、`defaultEventExecutorGroup`(M 个,跑编解码和业务 handler);ChannelHandler pipeline 顺序(Handshake → Encoder → Decoder → IdleState → ConnectManage → ServerHandler);背压:不可写时 `setAutoRead(false)`。源码:`remoting/.../netty/NettyRemotingServer.java`(`buildEventLoopGroupBoss` :125、`configChannel` :273、`channelWritabilityChanged` :510)、`remoting/.../netty/NettyEncoder.java`、`remoting/.../netty/NettyDecoder.java`(继承 `LengthFieldBasedFrameDecoder` 解决粘包)。**技巧精解:主从 Reactor 三组线程分工 + LengthFieldBasedFrameDecoder 粘包 + autoRead 背压**——为什么把 accept / IO / 业务分三组线程(职责隔离、IO 不被业务拖慢);反面对比"单组线程全包揽"会怎样(慢业务拖死 IO);`LengthFieldBasedFrameDecoder` 凭什么解决 TCP 粘包(前 4 字节长度定界)。
- **第 14 章 · Processor 路由、线程池隔离与三种调用语义**(`P4-14`)——`NettyRemotingAbstract` 怎么按 `RequestCode` 查 `processorTable`(RequestCode → Pair<Processor, ExecutorService>)路由;每个 Processor 配独立线程池做请求隔离(Send / Pull / Heartbeat 互不饿死);三种调用语义 `invokeSync` / `invokeAsync` / `invokeOneway`;`ResponseFuture` + `SemaphoreReleaseOnlyOnce`(CAS 保证信号量只释放一次);`HashedWheelTimer` 扫超时请求。源码:`remoting/.../netty/NettyRemotingAbstract.java`(`processorTable` :97、`processRequestCommand` :290、`responseTable` :90、`scanResponseTable` :380)、`remoting/.../netty/ResponseFuture.java`、`broker/.../BrokerController.java`(`registerProcessor`)。**技巧精解:每 Processor 独立线程池隔离 + `SemaphoreReleaseOnlyOnce` CAS 一次性释放 + HashedWheelTimer 超时扫描**——为什么每 Processor 独立线程池(一个慢不拖垮别的);反面对比"共享线程池"会怎样(一个慢请求饿死全局);信号量 CAS 释放凭什么应对"超时与正常回调并发"。

### 第 5 篇 · NameServer:路由与服务发现(2 章)

> Producer 怎么知道消息往哪个 broker 发、Consumer 怎么知道从哪拉——靠 NameServer。讲透它的五张路由表,以及那个著名的"为什么不用 ZooKeeper"。

- **第 15 章 · RouteInfoManager:五张路由表与分层并发**(`P5-15`)——`RouteInfoManager` 五张表(`topicQueueTable` / `brokerAddrTable` / `clusterAddrTable` / `brokerLiveTable` / `filterServerTable`)各自存什么;`registerBroker`(broker 心跳注册)、`pickupTopicRouteData`(client 拉路由)、`scanNotActiveBroker`(扫死 broker)、`unRegisterBroker`(级联清理)四条主路径。源码:`namesrv/.../routeinfo/RouteInfoManager.java`(`registerBroker` :113、`pickupTopicRouteData` :480、`scanNotActiveBroker` :560)。**技巧精解:`ConcurrentHashMap` + 粗粒度 `ReadWriteLock` 分层并发**——CHM 管单表原子性、ReadWriteLock 管跨表事务性(registerBroker 要同时改 5 张表);读用 readLock 并发、写用 writeLock 互斥;反面对比"纯 CHM"会怎样(跨表操作非原子,读到半成品路由)、"纯大锁"会怎样(读路径排队,拉路由吞吐崩)。
- **第 16 章 · 为什么不用 ZooKeeper:AP 心跳注册中心**(`P5-16`)——Broker 每 1s 向所有 NameServer 心跳注册(上报 topic 配置 + DataVersion)、NameServer 各节点**完全独立互不同步**(无共识)、`brokerLiveTable` 靠心跳 TTL(~120s)判活、client 每 30s 拉路由;为什么 RocketMQ 选择这种"AP 心跳"而非 ZK 的"CP 共识":无共识开销(低延迟高吞吐)、运维极简(无状态单进程)、最终一致性对 MQ 场景够用(容忍 30s 路由延迟)。源码:`namesrv/.../routeinfo/RouteInfoManager.java`、`broker/.../BrokerController.java`(心跳定时任务)、`client/.../impl/factory/MQClientInstance.java`(`updateTopicRouteInfoFromNameServer` 30s)。**技巧精解:AP 心跳软状态 + DataVersion 增量仲裁**——为什么各 NameServer 不用同步还安全(broker 向全部注册,任一 NameServer 都有全量路由)、DataVersion 单调版本号怎么做冲突仲裁;反面对比"用 ZK"会怎样(每次注册走 quorum、leader 切换脑裂、ZK 集群本身要重运维)。

### 第 6 篇 · 高可用:主从复制与自动切换(3 章)

> 存储内核只在单 broker。这一篇讲消息怎么跨节点复制不丢、master 挂了怎么自动切换。RocketMQ 高可用三条演进路,正好一章一条:第 17 章传统主从 → 第 18 章 DLedger(Raft)→ 第 19 章 5.x Controller。

- **第 17 章 · HA 主从复制:同步双写与异步复制**(`P6-17`)——master 的 `DefaultHAService` accept slave 连接、把 CommitLog 数据推过去;slave 的 `DefaultHAClient` 连 master 拉取、写本地 CommitLog;同步复制(SYNC_MASTER)靠 `GroupTransferService` 等待 slave ACK 到位才告诉 producer 成功、异步复制(ASYNC_MASTER)master 落盘即返回;`FlowMonitor` 复制限流。源码:`store/.../ha/DefaultHAService.java`、`store/.../ha/DefaultHAConnection.java`、`store/.../ha/DefaultHAClient.java`、`store/.../ha/GroupTransferService.java`。**技巧精解:`GroupTransferService` 同步双写的等待机制**——前台把"等 slave ACK 到 offset N"塞等待队列、HA 推送线程每轮检查 slave 进度、到位唤醒;凭什么前台不阻塞 IO 线程又能等同步;反面对比"前台自己等 slave"会怎样(每条都阻塞、吞吐崩)、"纯异步"会怎样(master 落盘成功但还没复制就挂,数据丢)。
- **第 18 章 · DLedger:基于 Raft 的自动选主**(`P6-18`)——传统主从(第 17 章)的致命伤:master 挂了要人工切换、不可用窗口大;`DLedger`(openmessaging/dledger)用 Raft 做**自动选主 + 日志复制**,`DLedgerCommitLog` 把 CommitLog 和 DLedger 的 Raft 日志结合——每条消息既走 CommitLog 存储、又走 DLedger 多数派复制,master 挂了集群自动选出新 master、slave 已有的多数派数据不丢。源码:`store/.../dledger/DLedgerCommitLog.java`、DLedger 库的 `DLedgerServer`/`DLedgerLeaderElector`。**技巧精解:DLedgerCommitLog 如何把 CommitLog 嵌进 Raft 日志**——既要保住 CommitLog 的顺序追加吞吐、又要满足 Raft 的复制语义,凭什么兼容;反面对比"传统主从 master 挂了"会怎样(人工切、数据可能丢、长不可用窗口)。本章呼应《etcd》的 Raft,讲清 RocketMQ 为什么不用 etcd-raft 而自研 DLedger、二者取舍。
- **第 19 章 · Controller:5.x 自动主备切换**(`P6-19`)——DLedger(第 18 章)的代价:每条消息都走全量 Raft 多数派复制,写入开销大;5.x 的 `Controller` 取其精华——只用 DLedger/Raft **选主**(轻量),数据复制仍走第 17 章的 HA 通道(不把 CommitLog 塞进 Raft),配 `autoswitch/` 的 `AutoSwitchHAService` + `EpochFileCache`(基于 epoch 的协议)实现 master 挂了自动切、且切换时不丢已确认数据。三种高可用方案取舍总账。源码:`store/.../ha/autoswitch/AutoSwitchHAService.java`、`store/.../ha/autoswitch/AutoSwitchHAConnection.java`、`store/.../ha/autoswitch/EpochFileCache.java`、`controller/.../DLedgerController.java`、`controller/.../ReplicasInfoManager.java`。**技巧精解:epoch 协议怎么保证切换不丢数据**——为什么 Controller 不让数据走 Raft(开销大),而用 epoch 标记每任 master 的进度边界、切换时新 master 从 epoch 边界对齐;反面对比"三种方案各自的代价"(传统主从=人工切+不可用窗口、DLedger=全量复制开销、Controller=架构复杂度但性能最优)。

### 第 7 篇 · 特性消息:顺序、延时、事务(3 章)

> 在基本旅程(发-存-消)之上,RocketMQ 的三张特性王牌。每一张都是在 CommitLog/ConsumeQueue 基本机制上做的精巧增强。

- **第 20 章 · 顺序消息:分区有序与消费端锁**(`P7-20`)——全局顺序(整个 topic 一把锁,吞吐极低)vs 分区顺序(同一业务 key 的消息进同一 queue,queue 内有序);发送端 `MessageQueueSelector`(按 hash(key) 选 queue)保证同 key 进同 queue;消费端 `MessageListenerOrderly`(单 queue 串行消费 + `ProcessQueue` 锁)vs `MessageListenerConcurrently`;`RebalanceImpl` 里锁 queue 的机制(分布式锁,保证同一 queue 同时只被一个 consumer 顺序消费)。源码:`client/.../producer/DefaultMQProducerImpl.java`(`selectMessageQueue`)、`client/.../impl/consumer/ConsumeMessageOrderlyService.java`、`broker` 端 order message lock。**技巧精解:分区有序 = 发送端 hash 选 queue + 消费端串行 + queue 锁**——为什么全局顺序不可行(单 queue 串行吞吐极低)、分区顺序怎么在"有序"和"并发"间平衡;消费端锁 queue 凭什么保证 Rebalance 时不会两个 consumer 同时消费同一 queue。
- **第 21 章 · 延时消息:5.x 时间轮**(`P7-21`)——延时消息进 broker 先改 topic 为 `SCHEDULE_TOPIC_XXXX`(暂时不进业务 ConsumeQueue)、存进时间轮;5.x 用 `TimerWheel`(每个 slot 一个时间刻度)+ `TimerLog`(延时消息记录),到期 scan 对应 slot 写回原 topic;对比 4.x 固定 18 个 delayLevel(`scheduleMessageService`)——5.x 时间轮怎么支持任意延时;`TimerMessageStore` / `TimerLog` / `TimerWheel` / `Slot` 的协作。源码:`store/.../timer/TimerMessageStore.java`、`store/.../timer/TimerWheel.java`、`store/.../timer/TimerLog.java`、`store/.../timer/Slot.java`。**技巧精解:时间轮(slot + 指针)+ TimerLog/TimerWheel 双文件**——凭什么海量延时任务不爆炸(slot 数固定、按到期时间分桶、指针轮转);反面对比"每个延时任务一个 ScheduledFuture / 一条定时扫描"会怎样(任务数爆炸、扫描 O(n));TimerWheel(按时间的 slot 表)与 TimerLog(按写入的明细)为什么要拆两个文件。
- **第 22 章 · 事务消息:half message 与回查**(`P7-22`)——两阶段:① producer 发 half message(写 `RMQ_SYS_TRANS_HALF_TOPIC`,对消费者不可见);② 执行本地事务;③ 本地事务成功 → commit(写回业务 topic)/ 失败 → rollback(标记删除);④ 未决(producer 没返回 commit/rollback)靠 broker 定期回查 producer 的本地事务状态(`checkLocalTransaction`)。源码:`broker/.../transaction/queue/TransactionalMessageService.java`、`broker/.../transaction/queue/TransactionalMessageBridge.java`、`broker/.../processor/EndTransactionProcessor.java`、`client/.../TransactionListener.java`。**技巧精解:half message 两阶段 + 回查保证最终一致**——为什么不能"先执行本地事务再发消息"(本地事务成功但发消息失败,不一致)、为什么不能"先发消息再执行本地事务"(消息发了但本地事务失败,下游收到脏数据);half message 怎么把"本地事务"和"发消息"做成原子(先 half 占位、本地事务决定 commit/rollback、超时回查兜底);对标分布式事务的 saga/TCC。

### 第 8 篇 · 5.x 新架构:Proxy、RocksDB 存储、分级存储(1 章)

> 5.x 在经典架构(前 7 篇)之上做的架构演进,单独成章。讲清每项新架构解决的老痛点、以及它和经典实现的对照。

- **第 23 章 · 5.x 新架构:Proxy 与 gRPC、RocksDB 存储、分级存储**(`P8-23`)——**Proxy**(基于 gRPC 的新接入层,与 Remoting 并存:为什么 5.x 引入 Proxy——多语言友好、协议更清晰、计算与存储分离、可独立扩缩);**RocksDB 存储**(`RocksDBMessageStore` 作为 `CommitLog + ConsumeQueue` 的可选替代实现:用 RocksDB 的 LSM 换"海量 Topic/Queue 不再受 ConsumeQueue 文件数拖累",呼应《LevelDB》《Linux 内存管理》的 LSM 与写放大);**TieredStore 分级存储**(冷数据下沉到对象存储 / 低成本介质、热数据留本地,省钱又不降热路径性能)。源码:`proxy/`、`store/.../RocksDBMessageStore.java`、`store/.../rocksdb/`、`tieredstore/`。**技巧精解:5.x 为什么在某些场景用 LSM(RocksDB)替掉 CommitLog + ConsumeQueue**——经典 CommitLog 是"一个大文件纯顺序写"(第 1~2 篇),RocksDB 是"LSM 多层归并"(对标《LevelDB》),凭什么在"海量 Queue"场景 LSM 反而更优(经典 ConsumeQueue 每 queue 一个文件,百万 Queue 时文件数爆炸,LSM 收敛之);反面对比"经典 CommitLog 在百万 Queue 下的痛点"。

### 第 9 篇 · 收尾:权衡哲学(1 章)

- **第 24 章 · RocketMQ 的权衡哲学与全景**(`P9-24`)——把全书收束成几条哲学(混写一个 CommitLog 换纯顺序写、写与建索引异步解耦、用 ConsumeQueue 把混写消息重新按队列组织、零拷贝把磁盘到网卡的拷贝清零、AP 心跳注册中心换可用性、长轮询换实时与无状态、至少一次 + 业务幂等换实现简单);RocketMQ vs Kafka 的根本分野(混写 vs 分 Partition、NameServer vs ZK、Push 长轮询 vs Pull);5.x 相对 4.x 的演进回顾(Proxy、Controller、RocksDB 存储、时间轮延时、分级存储)。全书一张端到端时序总图。

## 七、Java 技巧侧重(对标 Tokio 的 Rust `unsafe`/原子/`Pin`、LevelDB 的 C++ 无锁/引用计数、etcd 的 Go channel/`Ready`)

本书要重点拆的 Java 系统级与并发技巧(写作时遇到这些,务必钻"怎么做到的 + 为什么 sound"):

- **并发锁模型**:CommitLog 写路径的 `putMessageLock` 三选一(`PutMessageSpinLock` 自旋 / `PutMessageReentrantLock` 重入 / `AdaptiveBackOffSpinLockImpl` 自适应退避),为什么全局顺序追加注定要加锁、三种锁的取舍与自适应切换;`TopicQueueLock` 分段锁(按 topic-queue 分桶,降低 queueOffset 分配的锁粒度);`beginTimeInLock` 锁内重设 storeTimestamp 保全局有序。
- **内存映射与零拷贝**:`MappedByteBuffer`(mmap,写路径)、`FileChannel.transferTo`(sendfile,读路径送消费者)、`TransientStorePool` 堆外内存池(写堆外 DirectByteBuffer 再 commit,避开页缓存锁竞争);`ThreadLocal<PutMessageThreadLocal>` 编码器复用。
- **Netty 主从 Reactor**:三组线程(Boss/Selector/Worker)分工、`ChannelPipeline` 顺序、`LengthFieldBasedFrameDecoder` 粘包、`channelWritabilityChanged` + `setAutoRead` 背压反压、池化 `ByteBuf`。
- **异步编程**:`CompletableFuture` 全链路异步(从 `invokeAsync` 到 `asyncPutMessage` 到客户端回调,线程不阻塞)、`SemaphoreReleaseOnlyOnce` CAS 一次性释放信号量、`ResponseFuture` + `CountDownLatch`。
- **并发容器与分层并发**:`ConcurrentHashMap`(路由表、位点表、在途请求表)+ 粗粒度 `ReadWriteLock`(跨表事务性)的分层并发设计(RouteInfoManager)。
- **后台线程模型**:`ServiceThread`(ReputMessageService / FlushRealTimeService / GroupCommitService / HAService / PullRequestHoldService / RebalanceService)+ `ScheduledExecutorService` 定时任务(MQClientInstance 的 30s/5s/20s),写路径与后台脏活的解耦。
- **时间轮**:`HashedWheelTimer`(remoting 超时扫描)+ 5.x `TimerWheel`(延时消息 slot),对标《Tokio》层级时间轮,讲 slot/指针/双文件设计。
- **责任链与解耦**:`CommitLogDispatcher` 列表(Build ConsumeQueue + Build Index + 可选 Compaction),加索引类型零改动写路径。
- **AP 分布式设计**:NameServer 各节点独立 + broker 向全部注册 + 心跳 TTL 软状态,放弃 ZK 的 quorum 共识;Rebalance 无中心协调(双排序 + 确定性算法);DataVersion 单调版本号仲裁。
