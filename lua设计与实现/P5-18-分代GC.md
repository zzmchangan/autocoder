# P5-18 分代 GC:增量之外另一种摊平成本的策略

> **本书主线**:统一与精简换小而快。**二分法**:编译侧(源码→字节码)↔ 执行侧(VM 执行 + 管值生命周期)。**本章位置**:执行侧·P5 垃圾回收收尾,接 [P5-16 三色标记与增量步进](P5-16-三色标记与增量步进-GC必须可中断.md)、[P5-17 对象回收与弱表](P5-17-对象回收与弱表.md)。**★对照**:CPython(三代分代 + 引用计数主力 vs Lua 整堆分代可选)。**源码**:lua-5.5.0 `lgc.c`/`lgc.h`/`lstate.h`。**基调**:纯直球,不用比喻。

## 一、这章解决什么问题

P5-16 讲了 Lua 的增量式三色标记:把一次完整回收切成无数小步,每执行一小段字节码顺手做一小步 GC,靠 `GCdebt` 控制每步工作量。这套机制对所有工作负载都稳健,是 5.5 的默认 GC。但它有一个挥之不去的开销特征:**每一轮回收都要把整个堆从头到尾扫一遍**。哪怕堆里 90% 的对象是早就存活、长期不死的"老住户",每轮回收也要重新把它们标记一遍、扫一遍。

真实程序里的对象生命周期往往极度不均衡。大量对象是"朝生夕死"的:循环里的临时 Table、字符串拼接的中间结果、一次请求里构造的局部数据,用完即弃;只有少数对象(全局表、模块、长期持有的缓存)活得久。如果能把"长期不死的对象"和"刚创建、大概率很快死掉的对象"分开,让 GC 频繁地只去清理后者,偶尔才去打扰前者,单次回收的成本就能大幅下降。

这就是**分代垃圾回收**(generational GC)的出发点。它建立在一个经验假设上:

> **多数对象朝生夕死。**

基于这个假设,把堆里的对象按"年龄"分成几代:**年轻代**(刚创建、存活轮数少)里对象密度大、死亡率高,GC 频繁地只扫这一小片区域,回收快、成本低;**老年代**(存活多轮、已经被多次回收验证过"不是垃圾")里对象死亡率低,GC 偶尔才做一次全量扫描。

Lua 5.5 同时提供了两套 GC 策略:

- **增量模式**(`KGC_INC`):P5-16 讲的三色标记增量,每轮扫全堆,对任何工作负载都稳健。**这是默认**。
- **分代模式**(`KGC_GEN`):基于年龄的分代回收,在"对象朝生夕死"特征明显的程序上更省。

本章回答三个问题:

1. Lua 的分代 GC 在 5.5 源码里到底怎么实现的?对象怎么分代、minor collection(年轻代回收)怎么走、老年代什么时候才扫?
2. 分代模式怎么从增量模式切过来、又怎么切回去?为什么 5.5 默认还是增量?
3. 分代 GC 有一个经典难题——**老对象指向年轻对象的跨代引用**:minor 只扫年轻代,如果某个老对象持有一个年轻对象的引用,而老对象这轮不参与扫描,岂不是会把那个还活着的年轻对象当垃圾回收掉?Lua 怎么解决这个"不漏扫"的问题?

读完本章会看到:Lua 的分代实现没有搞独立的年轻代堆,而是在**同一条 `allgc` 链表**上靠"年龄位"切出代边界,用一组指针标记每代的起点;跨代引用靠**写屏障**实时记录到 `grayagain` 链表,minor 时据此把被老对象引用的年轻对象救回来。这是一套相当精巧的、复用增量三色基础设施的分代设计。

> **5.5 与老资料的关键差异(先说在前面)**:讲 5.1/5.3 的老资料常说 Lua 的 GC 模式是 `KGC_INC` 和 `KGC_GEN` 两种。**5.5 实际是三种**:`KGC_INC`(0)、`KGC_GENMINOR`(1)、`KGC_GENMAJOR`(2),见 `lstate.h:162-164`。分代内部又分 minor(常规年轻代回收)和 major(周期性全量回收)两个子模式,major 子模式在实现上**复用增量三色的步进逻辑**,只是名字挂在分代名下。此外,5.5 的年龄模型是 **7 个态**(`G_NEW`/`G_SURVIVAL`/`G_OLD0`/`G_OLD1`/`G_OLD`/`G_TOUCHED1`/`G_TOUCHED2`,`lgc.h:110-116`),不是老资料说的简单的 young/survived/old 三态。这些差异本章会逐一落到源码。

## 二、源码怎么实现

### (一)三种模式与模式切换

先看 GC 的"模式"是怎么表示的。`global_State` 里有一个 `gckind` 字段(`lstate.h:341`):

```c
lu_byte gckind;  /* kind of GC running */
```

它取三个值(`lstate.h:162-164`):

```c
#define KGC_INC		0	/* incremental gc */
#define KGC_GENMINOR	1	/* generational gc in minor (regular) mode */
#define KGC_GENMAJOR	2	/* generational in major mode */
```

`KGC_INC` 是增量三色;`KGC_GENMINOR` 是分代的常规态(做年轻代回收);`KGC_GENMAJOR` 是分代退化成"做一次完整增量回收"的态(用于周期性清理老年代)。后两者合起来才是用户眼里的"分代 GC"。

用户通过 `collectgarbage("generational")` 或 C API `lua_gc(L, LUA_GCGEN)` 开启分代。这个调用最终走到 `lapi.c:1219`:

```c
case LUA_GCGEN: {
  res = (g->gckind == KGC_INC) ? LUA_GCINC : LUA_GCGEN;
  luaC_changemode(L, KGC_GENMINOR);
  break;
}
case LUA_GCINC: {
  res = (g->gckind == KGC_INC) ? LUA_GCINC : LUA_GCGEN;
  luaC_changemode(L, KGC_INC);
  break;
}
```

`res` 返回的是"切换前的旧模式"(便于用户事后切回去)。真正的切换在 `luaC_changemode`(`lgc.c:1440`):

```c
void luaC_changemode (lua_State *L, int newmode) {
  global_State *g = G(L);
  if (g->gckind == KGC_GENMAJOR)  /* doing major collections? */
    g->gckind = KGC_INC;  /* already incremental but in name */
  if (newmode != g->gckind) {  /* does it need to change? */
    if (newmode == KGC_INC)  /* entering incremental mode? */
      minor2inc(L, g, KGC_INC);  /* entering incremental mode */
    else {
      lua_assert(newmode == KGC_GENMINOR);
      entergen(L, g);
    }
  }
}
```

两处要点:

1. 如果当前正处在 `KGC_GENMAJOR`(分代的 major 子态),先把它"正名"成 `KGC_INC`——因为 major 子态底层就是增量步进,只是名义上挂分代。这样后续的 `newmode != g->gckind` 比较才正确。
2. 进入分代调 `entergen`,进入增量调 `minor2inc`。

`entergen`(`lgc.c:1428`)是进入分代的正门:

```c
static void entergen (lua_State *L, global_State *g) {
  luaC_runtilstate(L, GCSpause, 1);  /* prepare to start a new cycle */
  luaC_runtilstate(L, GCSpropagate, 1);  /* start new cycle */
  atomic(L);  /* propagates all and then do the atomic stuff */
  atomic2gen(L, g);
  setminordebt(g);  /* set debt assuming next cycle will be minor */
}
```

它先把当前(增量的)GC 周期跑到 `GCSpause`(收尾),再开一个新周期跑到 `GCSpropagate`,然后一口气做完 `atomic`(原子标记阶段)。为什么要先做一轮完整的原子标记?因为分代模式要求"进入分代那一刻,堆里所有活对象都已经是老对象"。只有先做一次全量标记,才能确定哪些是活的、把它们统统划为老年代。这个"把所有存活对象变老"的动作在 `atomic2gen` 里。

`atomic2gen`(`lgc.c:1389`)是进入分代的核心转换:

```c
static void atomic2gen (lua_State *L, global_State *g) {
  cleargraylists(g);
  /* sweep all elements making them old */
  g->gcstate = GCSswpallgc;
  sweep2old(L, &g->allgc);
  /* everything alive now is old */
  g->reallyold = g->old1 = g->survival = g->allgc;
  g->firstold1 = NULL;  /* there are no OLD1 objects anywhere */

  /* repeat for 'finobj' lists */
  sweep2old(L, &g->finobj);
  g->finobjrold = g->finobjold1 = g->finobjsur = g->finobj;

  sweep2old(L, &g->tobefnz);

  g->gckind = KGC_GENMINOR;
  g->GCmajorminor = g->GCmarked;  /* "base" for number of bytes */
  g->GCmarked = 0;  /* to count the number of added old1 bytes */
  finishgencycle(L, g);
}
```

`sweep2old`(`lgc.c:1136`)扫整条链,把死的释放、活的统统设成 `G_OLD`:

```c
static void sweep2old (lua_State *L, GCObject **p) {
  GCObject *curr;
  global_State *g = G(L);
  while ((curr = *p) != NULL) {
    if (iswhite(curr)) {  /* is 'curr' dead? */
      lua_assert(isdead(g, curr));
      *p = curr->next;  /* remove 'curr' from list */
      freeobj(L, curr);  /* erase 'curr' */
    }
    else {  /* all surviving objects become old */
      setage(curr, G_OLD);
      if (curr->tt == LUA_VTHREAD) {  /* threads must be watched */
        lua_State *th = gco2th(curr);
        linkgclist(th, g->grayagain);  /* insert into 'grayagain' list */
      }
      else if (curr->tt == LUA_VUPVAL && upisopen(gco2upv(curr)))
        set2gray(curr);  /* open upvalues are always gray */
      else  /* everything else is black */
        nw2black(curr);
      p = &curr->next;  /* go to next element */
    }
  }
}
```

注意进入分代时,线程(`LUA_VTHREAD`)被挂到 `grayagain`、开放 upvalue 保持灰、其余对象都设黑。这是因为线程和开放 upvalue 的内容会随执行变化,必须每轮重新访问;其余老对象一旦变黑,只要不被写屏障打回灰色,就不再参与遍历。

回到 `atomic2gen` 的指针赋值:`g->reallyold = g->old1 = g->survival = g->allgc`——刚进入分代时,所有对象都是老的,所以"年轻代的起点"(`survival`)、"old1 的起点"(`old1`)、"真正老的起点"(`reallyold`)全指向链表头。这意味着此刻"年轻代"是空的。之后新创建的对象会插到链表头(见 `luaC_newobjdt@lgc.c:297` 的 `o->next = g->allgc; g->allgc = o;`),它们天然落在 `allgc` 头部、`survival` 指针之前,构成年轻代。

### (二)对象的年龄位:7 个态与 `nextage` 升级表

分代的核心数据是每个 GC 对象 `marked` 字段低 3 位的"年龄"。`lgc.h:109-122`:

```c
/* object age in generational mode */
#define G_NEW		0	/* created in current cycle */
#define G_SURVIVAL	1	/* created in previous cycle */
#define G_OLD0		2	/* marked old by frw. barrier in this cycle */
#define G_OLD1		3	/* first full cycle as old */
#define G_OLD		4	/* really old object (not to be visited) */
#define G_TOUCHED1	5	/* old object touched this cycle */
#define G_TOUCHED2	6	/* old object touched in previous cycle */

#define AGEBITS		7  /* all age bits (111) */

#define getage(o)	((o)->marked & AGEBITS)
#define setage(o,a)  ((o)->marked = cast_byte(((o)->marked & (~AGEBITS)) | a))
#define isold(o)	(getage(o) > G_SURVIVAL)
```

低 3 位能表示 0–7,这里用到 0–6 共 7 个态(`AGEBITS=7` 即二进制 `111`)。`isold` 的判据是"年龄大于 `G_SURVIVAL`",也就是 `G_OLD0` 及以后都算"老"。`lgc.h` 第 125–160 行有一大段注释详细解释每个态的含义,翻译成要点:

- **`G_NEW`(0)**:当前周期里新建的对象。年轻。
- **`G_SURVIVAL`(1)**:上一周期创建、经历过一次 minor 还活着的对象。仍算年轻。
- **`G_OLD0`(2)**:被前向写屏障在本周期内"强行催老"的对象(后面讲屏障时解释为什么需要这个中间态)。老,但可能还指向年轻对象。
- **`G_OLD1`(3)**:作为老对象经历的第一个完整周期。老,但不能指向 `G_NEW`。
- **`G_OLD`(4)**:真正老。不再被遍历。
- **`G_TOUCHED1`(5)**:本周期被后向屏障触碰的老对象。挂在 `grayagain`,本轮结尾要重访。
- **`G_TOUCHED2`(6)**:上一周期被触碰的老对象。

年龄怎么"升级"(从年轻变老)?在分代的 sweep 阶段,`sweepgen`(`lgc.c:1172`)用一张静态表 `nextage` 决定每个对象的下一站:

```c
static GCObject **sweepgen (lua_State *L, global_State *g, GCObject **p,
                            GCObject *limit, GCObject **pfirstold1,
                            l_mem *paddedold) {
  static const lu_byte nextage[] = {
    G_SURVIVAL,  /* from G_NEW */
    G_OLD1,      /* from G_SURVIVAL */
    G_OLD1,      /* from G_OLD0 */
    G_OLD,       /* from G_OLD1 */
    G_OLD,       /* from G_OLD (do not change) */
    G_TOUCHED1,  /* from G_TOUCHED1 (do not change) */
    G_TOUCHED2   /* from G_TOUCHED2 (do not change) */
  };
  l_mem addedold = 0;
  int white = luaC_white(g);
  GCObject *curr;
  while ((curr = *p) != limit) {
    if (iswhite(curr)) {  /* is 'curr' dead? */
      lua_assert(!isold(curr) && isdead(g, curr));
      *p = curr->next;  /* remove 'curr' from list */
      freeobj(L, curr);  /* erase 'curr' */
    }
    else {  /* correct mark and age */
      int age = getage(curr);
      if (age == G_NEW) {  /* new objects go back to white */
        int marked = curr->marked & ~maskgcbits;  /* erase GC bits */
        curr->marked = cast_byte(marked | G_SURVIVAL | white);
      }
      else {  /* all other objects will be old, and so keep their color */
        lua_assert(age != G_OLD1);  /* advanced in 'markold' */
        setage(curr, nextage[age]);
        if (getage(curr) == G_OLD1) {
          addedold += objsize(curr);  /* bytes becoming old */
          if (*pfirstold1 == NULL)
            *pfirstold1 = curr;  /* first OLD1 object in the list */
        }
      }
      p = &curr->next;  /* go to next element */
    }
  }
  *paddedold += addedold;
  return p;
}
```

`sweepgen` 一边扫链表,一边对每个对象做两件事之一:

- **白色对象 = 死**:直接从链表摘除并释放(`freeobj`)。注意断言 `!isold(curr)`——分代模式下,老对象不会是白色(它们要么黑要么灰),白色的一定是年轻的、且确实死了。
- **非白对象 = 活**:按 `nextage[age]` 升级年龄。

`nextage` 表是年龄状态机的转移函数,值得逐行看清:

| 当前年龄 | `nextage` 后 | 含义 |
|---|---|---|
| `G_NEW`(0) | `G_SURVIVAL`(1) | 新对象熬过这轮 → 升 survival |
| `G_SURVIVAL`(1) | `G_OLD1`(3) | survival 再熬一轮 → 升 old1 |
| `G_OLD0`(2) | `G_OLD1`(3) | 屏障催老的 old0 → 下一轮变 old1 |
| `G_OLD1`(3) | `G_OLD`(4) | old1 再熬一轮 → 真正变 old |
| `G_OLD`(4) | `G_OLD`(4) | 老,不变 |
| `G_TOUCHED1`(5) | `G_TOUCHED1`(5) | touched1 的升级在 `correctgraylist` 另算 |
| `G_TOUCHED2`(6) | `G_TOUCHED2`(6) | touched2 同上 |

注意 `G_NEW` 走的是特例分支(第 1195 行):它不仅改年龄到 `G_SURVIVAL`,还把整个 GC 位(颜色位 + 年龄位)清掉重设为 `G_SURVIVAL | white`——也就是重新变白。为什么?因为年轻对象在 mutator 运行期间一直保持白色(下一小节讲),sweep 时要把它的标记清干净,准备下一轮。

还有一个细节:升到 `G_OLD1` 的对象会累加 `addedold`(`objsize(curr)`,`lgc.c:113`),并记录 `firstold1` 指针。这个字节数是后面判断"要不要触发 major 回收"的依据。

### (三)minor collection:年轻代回收的完整流程

分代模式的常规回收叫 **minor collection**(年轻代回收),由 `youngcollection`(`lgc.c:1335`)完成。它是分代模式的心脏:

```c
static void youngcollection (lua_State *L, global_State *g) {
  l_mem addedold1 = 0;
  l_mem marked = g->GCmarked;  /* preserve 'g->GCmarked' */
  GCObject **psurvival;  /* to point to first non-dead survival object */
  GCObject *dummy;  /* dummy out parameter to 'sweepgen' */
  lua_assert(g->gcstate == GCSpropagate);
  if (g->firstold1) {  /* are there regular OLD1 objects? */
    markold(g, g->firstold1, g->reallyold);  /* mark them */
    g->firstold1 = NULL;  /* no more OLD1 objects (for now) */
  }
  markold(g, g->finobj, g->finobjrold);
  markold(g, g->tobefnz, NULL);

  atomic(L);  /* will lose 'g->marked' */

  /* sweep nursery and get a pointer to its last live element */
  g->gcstate = GCSswpallgc;
  psurvival = sweepgen(L, g, &g->allgc, g->survival, &g->firstold1, &addedold1);
  /* sweep 'survival' */
  sweepgen(L, g, psurvival, g->old1, &g->firstold1, &addedold1);
  g->reallyold = g->old1;
  g->old1 = *psurvival;  /* 'survival' survivals are old now */
  g->survival = g->allgc;  /* all news are survivals */

  /* repeat for 'finobj' lists */
  dummy = NULL;  /* no 'firstold1' optimization for 'finobj' lists */
  psurvival = sweepgen(L, g, &g->finobj, g->finobjsur, &dummy, &addedold1);
  /* sweep 'survival' */
  sweepgen(L, g, psurvival, g->finobjold1, &dummy, &addedold1);
  g->finobjrold = g->finobjold1;
  g->finobjold1 = *psurvival;  /* 'survival' survivals are old now */
  g->finobjsur = g->finobj;  /* all news are survivals */

  sweepgen(L, g, &g->tobefnz, NULL, &dummy, &addedold1);

  /* keep total number of added old1 bytes */
  g->GCmarked = marked + addedold1;

  /* decide whether to shift to major mode */
  if (checkminormajor(g)) {
    minor2inc(L, g, KGC_GENMAJOR);  /* go to major mode */
    g->GCmarked = 0;  /* avoid pause in first major cycle (see 'setpause') */
  }
  else
    finishgencycle(L, g);  /* still in minor mode; finish it */
}
```

走一遍这个流程。注意 `allgc` 是一条单向链表,按"年龄从年轻到老"的顺序排列——链表头是最新创建的对象,链表尾是最老的对象。代边界靠几个指针切出:`survival` 指向"经历过一轮的对象"的起点,`old1` 指向 old1 段的起点,`reallyold` 指向真正老对象的起点。所以链表逻辑上是四段:

```
allgc头 -> [新生对象 G_NEW] -> survival -> [G_SURVIVAL] -> old1 -> [G_OLD1] -> reallyold -> [G_OLD / G_TOUCHED*]
```

`youngcollection` 的步骤:

**第一步,`markold`**(`lgc.c:1276`):把上一轮升上来的 `G_OLD1` 对象标记一遍。

```c
static void markold (global_State *g, GCObject *from, GCObject *to) {
  GCObject *p;
  for (p = from; p != to; p = p->next) {
    if (getage(p) == G_OLD1) {
      lua_assert(!iswhite(p));
      setage(p, G_OLD);  /* now they are old */
      if (isblack(p))
        reallymarkobject(g, p);
    }
  }
}
```

为什么要单独标记 old1?因为 old1 对象是"上一轮刚从 survival 升上来的",它们可能还持有 survival 对象的引用(`lgc.h` 注释:`old1` can still point to survival objects)。minor collection 只扫年轻代,如果不把 old1 也遍历一遍,old1 指向的 survival 对象就会漏扫。所以 `markold` 把 old1 提升为 `G_OLD`,并对其黑色实例调 `reallymarkobject` 把它引用的对象也标记上。`firstold1` 是个优化指针,记住 old1 段从哪里开始,免去全段扫描;用完置 `NULL`。

对 `finobj`(带 finalizer 的对象链)和 `tobefnz`(待 finalize 的对象链)也各调一次 `markold`,处理它们里面的 old1。

**第二步,`atomic`**(`lgc.c:1543`):原子标记阶段。这一步把所有灰色对象(包括上一步 markold 新加的、写屏障挂进 `grayagain` 的)全部遍历完,处理弱表、ephemeron 表的收敛。这一步是 STW(停顿 mutator)的,但因为它只遍历年轻代 + old1 + 被记录的跨代引用,工作量比全堆 atomic 小得多。

**第三步,sweepgen 三段扫**。这是真正回收年轻代的地方。注意 `youngcollection` 对 `allgc` 链调了**两次** `sweepgen`:

- 第一次:`sweepgen(L, g, &g->allgc, g->survival, ...)`——从链表头扫到 `survival` 指针,即扫**新生段(G_NEW)**。返回 `psurvival`,指向新生段最后一个活对象的位置。
- 第二次:`sweepgen(L, g, psurvival, g->old1, ...)`——从新生段尾部扫到 `old1` 指针,即扫**survival 段(G_SURVIVAL)**。

为什么不一次扫到 `old1`?因为两段的年龄升级结果不同(新生→survival,survival→old1),而且需要更新不同的代边界指针。扫完后,代边界指针要"整体后移":

```c
g->reallyold = g->old1;
g->old1 = *psurvival;  /* 'survival' survivals are old now */
g->survival = g->allgc;  /* all news are survivals */
```

含义:原本的 old1 段并入 reallyold(它们又老了一轮);原本的 survival 段(刚被 sweepgen 升级成 old1)成为新的 old1 段;新生段(刚被 sweepgen 升级成 survival)成为新的 survival 段;而 `survival = allgc`(链表头),因为新一轮的"新生对象"此时还没创建,新生段为空。这就是"年轻代整体往后挪一格"。

对 `finobj` 链做完全对称的操作(它有自己的 `finobjsur`/`finobjold1`/`finobjrold` 三段指针),再扫一遍 `tobefnz`。

**第四步,更新字节计数**:`g->GCmarked = marked + addedold1`。`addedold1` 是这一轮 sweepgen 累计的"新升为 old1 的对象字节数"。这个数字累计下去,就是判断"要不要触发 major"的依据。

**第五步,`checkminormajor` 决策**:判断要不要从 minor 切到 major。

```c
static int checkminormajor (global_State *g) {
  l_mem limit = applygcparam(g, MINORMAJOR, g->GCmajorminor);
  if (limit == 0)
    return 0;  /* special case: 'minormajor' 0 stops major collections */
  return (g->GCmarked >= limit);
}
```

`GCmajorminor` 是上一次 major 回收结束时记录的"活对象字节数"(基准)。`MINORMAJOR` 参数默认 70(`lgc.h:173`),即基准的 70%。当"累计新增的 old1 字节"超过基准的 70% 时,说明老年代积攒了太多对象,该做一次全量清理了,返回真,触发 `minor2inc(L, g, KGC_GENMAJOR)`——切到 major 子态(本质是切回增量,做一轮完整回收)。否则调 `finishgencycle` 收尾,继续留在 minor。

`finishgencycle`(`lgc.c:1292`)做收尾:校正灰表(`correctgraylists`,把 touched1 升 touched2、touched2 升 old 等)、检查字符串表大小(`checkSizes`)、把 gcstate 重置为 `GCSpropagate`、调用待执行的 finalizer。

### (四)字节计数器:`GCmarked` 与 `GCmajorminor` 的双重角色

分代模式的节奏控制,靠两个 `l_mem` 计数器:`GCmarked` 和 `GCmajorminor`(`lstate.h:332-333`)。它们在不同模式下含义不同,`lgc.c:1101-1114` 有一段专门注释:

```
* KGC_INC:
*     GCmarked: number of marked bytes during a cycle.
*     GCmajorminor: not used.
* KGC_GENMINOR
*     GCmarked: number of bytes that became old since last major collection.
*     GCmajorminor: number of bytes marked in last major collection.
* KGC_GENMAJOR
*     GCmarked: number of bytes that became old since last major collection.
*     GCmajorminor: number of bytes marked in last major collection.
```

也就是说:

- **增量模式**:`GCmarked` 是本轮已标记的字节数(`reallymarkobject@lgc.c:339` 里 `g->GCmarked += objsize(o)` 就是累加点)。`GCmajorminor` 不用。
- **分代 minor / major**:`GCmajorminor` 固定是"上一次 major 回收结束时全堆活对象字节数"(基准锚点);`GCmarked` 是"自上次 major 以来,累计新升为 old1 的字节数"。

有了这两个数,minor 何时转 major、major 何时转回 minor,就有了客观依据:

- minor → major:当 `GCmarked`(新增 old 字节)≥ `GCmajorminor`(基准)× `MINORMAJOR`(70%),说明老年代膨胀太快,该全量清理了。见 `checkminormajor@lgc.c:1323`。
- major → minor:做了一轮 major(增量全量回收)后,如果这轮回收掉的垃圾够多,就回 minor。见下面 `checkmajorminor`。

major 子态本质是"挂分代名、走增量逻辑"。一旦切到 `KGC_GENMAJOR`,后续的 `luaC_step` 会把它当 `KGC_INC` 一样调 `incstep` 跑增量步进(见后面 step 分派)。做完一轮增量回收、到达 atomic 时,`checkmajorminor`(`lgc.c:1471`)检查要不要回 minor:

```c
static int checkmajorminor (lua_State *L, global_State *g) {
  if (g->gckind == KGC_GENMAJOR) {  /* generational mode? */
    l_mem numbytes = gettotalbytes(g);
    l_mem addedbytes = numbytes - g->GCmajorminor;
    l_mem limit = applygcparam(g, MAJORMINOR, addedbytes);
    l_mem tobecollected = numbytes - g->GCmarked;
    if (tobecollected > limit) {
      atomic2gen(L, g);  /* return to generational mode */
      setminordebt(g);
      return 1;  /* exit incremental collection */
    }
  }
  g->GCmajorminor = g->GCmarked;  /* prepare for next collection */
  return 0;  /* stay doing incremental collections */
}
```

逻辑:在 major 子态做了一轮增量 atomic 之后,看"当前还能回收的字节数(`numbytes - GCmarked`,即总量减去本轮标记存活量)"是否大于"`addedbytes`(自上次 major 以来新增字节)× `MAJORMINOR`(默认 50,`lgc.h:179`)"。如果回收潜力够大,就 `atomic2gen` 切回 minor;否则继续留 major 做下一轮增量。

这套计数器机制把 minor 和 major 的切换节奏完全交给"字节数"驱动,不靠固定周期,能自适应工作负载:对象死得快就频繁 minor、很少 major;对象累积多就及时 major 清理。

### (五)跨代引用:写屏障如何记录 old → young

分代 GC 的经典难题是**跨代引用**。minor collection 只扫年轻代,但如果某个**老对象**持有一个**年轻对象**的引用,而老对象这轮不参与扫描,那个年轻对象虽然在别处没人引用、却被这个老对象拽着不能死,扫描器怎么会知道它还活着?

漏扫的后果是致命的:把活对象当垃圾释放,程序立刻崩溃。所以分代 GC 必须有一套机制,在"老对象新指向年轻对象"发生时,把这个关系记录下来,minor 时据此把被引用的年轻对象救回。

Lua 复用了 P5-16 讲的增量三色写屏障来做这件事。写屏障分前向(forward)和后向(backward)两种,触发时机都是"一个指向关系的建立打破了三色不变式"——具体到分代,就是"黑(老)对象指向白(年轻)对象"。

**前向屏障 `luaC_barrier_`**(`lgc.c:246`):

```c
void luaC_barrier_ (lua_State *L, GCObject *o, GCObject *v) {
  global_State *g = G(L);
  lua_assert(isblack(o) && iswhite(v) && !isdead(g, v) && !isdead(g, o));
  if (keepinvariant(g)) {  /* must keep invariant? */
    reallymarkobject(g, v);  /* restore invariant */
    if (isold(o)) {
      lua_assert(!isold(v));  /* white object could not be old */
      setage(v, G_OLD0);  /* restore generational invariant */
    }
  }
  else {  /* sweep phase */
    lua_assert(issweepphase(g));
    if (g->gckind != KGC_GENMINOR)  /* incremental mode? */
      makewhite(g, o);  /* mark 'o' as white to avoid other barriers */
  }
}
```

当老对象 `o` 指向年轻(白)对象 `v` 时,前向屏障直接把 `v` 标记(`reallymarkobject`),让它进入扫描视野。但这里有个分代特有的麻烦:`v` 被标记后,如果它的年龄还是 `G_NEW`,下一轮 minor 还是会扫它——可它现在实际上是被老对象拽着的,把它当年轻对象反复扫没意义。所以屏障额外做 `setage(v, G_OLD0)`:把 `v` 的年龄设成 `G_OLD0`。

为什么是 `G_OLD0` 而不是直接 `G_OLD`?因为 `v` 此刻可能还指向别的年轻对象(它自己刚被创建不久)。如果直接设 `G_OLD`,`lgc.h` 注释里说老对象不能指向 `G_NEW` 对象,这会破坏分代不变式。`G_OLD0` 是个过渡态——"已经被屏障催老,但还可能指向年轻对象",它会在下一个 `sweepgen` 周期经 `nextage[G_OLD0] = G_OLD1` 升级,再一轮才到 `G_OLD`。这个两级过渡保证了"`v` 自己变老"和"`v` 引用的对象也跟着被遍历"不会脱节。

注意 `else` 分支(sweep 阶段):只有 `g->gckind != KGC_GENMINOR`(即非分代 minor 态)才把 `o` 重新变白。分代模式下不能这么干——分代的 sweep 不区分白和死(`lgc.c:1162-1165` 注释:`any white object must be dead`),把老对象变白会让它被当垃圾释放。所以分代 minor 的 sweep 阶段对前向屏障是空操作。

**后向屏障 `luaC_barrierback_`**(`lgc.c:268`):

```c
void luaC_barrierback_ (lua_State *L, GCObject *o) {
  global_State *g = G(L);
  lua_assert(isblack(o) && !isdead(g, o));
  lua_assert((g->gckind != KGC_GENMINOR)
          || (isold(o) && getage(o) != G_TOUCHED1));
  if (getage(o) == G_TOUCHED2)  /* already in gray list? */
    set2gray(o);  /* make it gray to become touched1 */
  else  /* link it in 'grayagain' and paint it gray */
    linkobjgclist(o, g->grayagain);
  if (isold(o))  /* generational mode? */
    setage(o, G_TOUCHED1);  /* touched in current cycle */
}
```

后向屏障的思路相反:不标记被指向的对象,而是把**指向者 `o` 本身**打回灰色,挂进 `grayagain` 链表。这样下一次 atomic 会重新遍历 `o`,它指向的所有对象(包括那个年轻对象)都会被标记到。

分代模式下,后向屏障额外把 `o` 的年龄设成 `G_TOUCHED1`(`lgc.c:278`)。`G_TOUCHED1` 的语义是"本周期被触碰的老对象",它必须留在 `grayagain` 里,等本轮 minor 的 atomic 重访。重访之后,`correctgraylist`(`lgc.c:1227`)会把它升成 `G_TOUCHED2`(`lgc.c:1236`);再下一轮如果没再被触碰,就升成 `G_OLD`(`lgc.c:1246`)。这条 `touched1 → touched2 → old` 的路径,保证了"偶尔被改的老对象"会被多盯几轮,确认它确实稳定后才放回不再遍历的 `G_OLD`。

**`genlink`**(`lgc.c:470`)是遍历结束时的收尾钩子,在弱表/ephemeron 表遍历完(`traverseweakvalue` 等)时调用,把年龄处理对:

```c
static void genlink (global_State *g, GCObject *o) {
  lua_assert(isblack(o));
  if (getage(o) == G_TOUCHED1) {  /* touched in this cycle? */
    linkobjgclist(o, g->grayagain);  /* link it back in 'grayagain' */
  }  /* everything else do not need to be linked back */
  else if (getage(o) == G_TOUCHED2)
    setage(o, G_OLD);  /* advance age */
}
```

`genlink` 只对 `G_TOUCHED1` 和 `G_TOUCHED2` 有动作:`G_TOUCHED1` 重新挂回 `grayagain`(保证下一轮还能被找到);`G_TOUCHED2` 升级为 `G_OLD`。其余年龄态无需处理。`lgc.c:467-468` 注释点明:它本可以把所有老对象都挂进链表、把判断全丢给 `correctgraylist`,但那样会加入大量马上又被摘除的元素,不划算,所以只挂 `G_TOUCHED1`。

把这套屏障串起来看,跨代引用的处理是:**老对象一旦新指向年轻对象,要么把年轻对象催老(前向屏障 → `G_OLD0`),要么把老自己打回灰、挂进 `grayagain` 等下轮重访(后向屏障 → `G_TOUCHED1`)**。无论哪条路,minor collection 的 atomic 阶段都能据此找到那个年轻对象,不漏扫。这就是分代 GC "不丢活对象" 的保障。

> **写屏障的触发入口**:`luaC_objbarrier`/`luaC_barrierback` 两个宏(`lgc.h:241-252`)是大多数赋值点的入口,它们先判断"指向者黑、被指向者白",命中才调真正的 `luaC_barrier_`/`luaC_barrierback_`。这与 P5-16 讲的增量写屏障是同一套,分代只是复用。

### (六)`luaC_step` 的三态分派

前面看了 minor 和模式切换,现在看 GC 的总入口 `luaC_step`(`lgc.c:1740`)怎么根据 `gckind` 分派:

```c
void luaC_step (lua_State *L) {
  global_State *g = G(L);
  lua_assert(!g->gcemergency);
  if (!gcrunning(g)) {  /* not running? */
    if (g->gcstp & GCSTPUSR)  /* stopped by the user? */
      luaE_setdebt(g, 20000);
  }
  else {
    luai_tracegc(L, 1);  /* for internal debugging */
    switch (g->gckind) {
      case KGC_INC: case KGC_GENMAJOR:
        incstep(L, g);
        break;
      case KGC_GENMINOR:
        youngcollection(L, g);
        setminordebt(g);
        break;
    }
    luai_tracegc(L, 0);  /* for internal debugging */
  }
}
```

三个分支:

- `KGC_INC`(增量)和 `KGC_GENMAJOR`(分代 major 子态)都走 `incstep`(`lgc.c:1710`)——做一小段增量步进。这印证了前面说的"major 子态底层就是增量"。`incstep` 内部会检测 `singlestep` 的返回值,如果返回 `step2minor`(从 major 切回 minor 了)就提前返回(`lgc.c:1717-1718`)。
- `KGC_GENMINOR` 走 `youngcollection` + `setminordebt`——**一次性做完一整轮 minor 回收**。

这里有个重要的设计取舍:**分代的 minor 回收不是增量的,而是一次性跑完的**。为什么?因为年轻代通常很小(只有"自上次 minor 以来新创建的对象"),一次性扫完的开销本就微乎其微,没必要再切成增量小步徒增状态机复杂度。这也呼应了 `sweepgen` 注释(`lgc.c:1162-1165`)说的"分代的 sweep 没有增量,所以不会有 new white 对象,任何白色对象都是死的"——正因为 minor 是原子的,mutator 不会在 sweep 中途往年轻代塞新对象。

`setminordebt`(`lgc.c:1417`)设下一次 minor 的触发债务:

```c
static void setminordebt (global_State *g) {
  luaE_setdebt(g, applygcparam(g, MINORMUL, g->GCmajorminor));
}
```

`MINORMUL` 默认 20(`lgc.h:185`)。含义:当总分配字节比基准 `GCmajorminor` 增长 20% 时,触发下一次 minor。也就是"每新增 20% 基准字节的新对象,就做一次年轻代回收"。这个比例越小,minor 越频繁(年轻代越小、单次越快);越大,minor 越稀(年轻代积攒得多、单次回收成本高)。20% 是个兼顾两者的默认值。

### (七)`luaC_fullgc`:强制全量回收

用户调 `collectgarbage("collect")` 或 `luaC_fullgc`(`lgc.c:1786`)时,无论当前什么模式,都强制做一次全量回收:

```c
void luaC_fullgc (lua_State *L, int isemergency) {
  global_State *g = G(L);
  lua_assert(!g->gcemergency);
  g->gcemergency = cast_byte(isemergency);  /* set flag */
  switch (g->gckind) {
    case KGC_GENMINOR: fullgen(L, g); break;
    case KGC_INC: fullinc(L, g); break;
    case KGC_GENMAJOR:
      g->gckind = KGC_INC;
      fullinc(L, g);
      g->gckind = KGC_GENMAJOR;
      break;
  }
  g->gcemergency = 0;
}
```

分代 minor 态下调 `fullgen`(`lgc.c:1458`):先 `minor2inc` 切回增量、做一轮全量增量回收(`fullinc`),再 `entergen` 重新进入分代。也就是说,分代模式下的"全量回收"是"临时切回增量、扫完、再切回分代"。这再次体现了 Lua 分代与增量共享同一套核心实现。

### (八)新对象的默认年龄与白色

最后补一个细节:新创建对象的初始状态。`luaC_newobjdt`(`lgc.c:297`):

```c
GCObject *luaC_newobjdt (lua_State *L, lu_byte tt, size_t sz, size_t offset) {
  global_State *g = G(L);
  char *p = cast_charp(luaM_newobject(L, novariant(tt), sz));
  GCObject *o = cast(GCObject *, p + offset);
  o->marked = luaC_white(g);
  o->tt = tt;
  o->next = g->allgc;
  g->allgc = o;
  return o;
}
```

`o->marked = luaC_white(g)` 只设了白色位(WHITE0BIT 或 WHITE1BIT,`lgc.h:79-80`,位 3 和位 4),年龄位(低 3 位)默认全 0,也就是 `G_NEW`。新对象插到 `allgc` 链表头(`g->allgc = o`),天然落在 `survival` 指针之前——这就是它进入"年轻代"的方式。没有单独的"年轻代分配区",年轻代就是 `allgc` 链表头部、`survival` 指针之前的那一段。

`lgc.h:152-159` 注释说得很清楚:分代模式下,mutator 运行期间年轻对象**始终保持白色**。这与增量模式不同(增量模式存活对象会被标记成黑)。分代里年轻对象保持白色,意味着它们在 minor 的 sweep 阶段要么被 atomic 标记过(变非白,存活)、要么仍是白色(死,被回收)。这正是 `sweepgen` 能用"白色即死"判定的前提。

## 三、为什么这样设计是 sound 的

### (一)minor 不漏扫活对象

这是分代 GC 正确性的命脉。minor 只扫年轻代,凭什么保证不被某个老对象拽着的年轻对象误回收?靠三层保障:

1. **进入分代时全堆变老**(`atomic2gen` → `sweep2old`):所有存活对象设 `G_OLD` 且变黑。此后老对象默认不指向白(年轻)对象——这是进入分代时建立的不变式起点。
2. **运行期写屏障记录每一次"老→新"**:`luaC_barrier_`(前向)把被指向的年轻对象催老(`G_OLD0`);`luaC_barrierback_`(后向)把老指向者打回灰、挂 `grayagain`(`G_TOUCHED1`)。任何破坏"老不指新"不变式的赋值,都会被屏障捕获并记录。
3. **minor 的 atomic 阶段遍历所有记录**:`youngcollection` 先 `markold`(处理上轮的 old1),再 `atomic`(把 `grayagain` 里所有 touched 对象、markold 新加的对象全部遍历完)。任何被老对象引用的年轻对象,要么已被前向屏障催老、要么其老指向者在 `grayagain` 里等着被重访——两条路都通向 atomic 的标记。

所以,minor 的标记根集不只是常规根(注册表、栈、全局表),还包括"所有被写屏障记录的老对象"。这就是跨代引用不漏扫的 soundness 保障。

### (二)年龄升级的正确性

年龄升级靠 `nextage` 表。`sweepgen` 只对非 `G_NEW` 对象调 `setage(curr, nextage[age])`,而 `G_NEW` 走特例(重设为 `G_SURVIVAL | white`)。为什么 `G_NEW` 要特殊处理?因为它要保持白色(下一轮还是年轻对象,仍可能被当"白色即死"判定);而其他年龄态的对象颜色不变(老对象保持黑/灰)。

`nextage` 表的关键性质:**每一步升级都是"单调变老"**——`G_NEW → G_SURVIVAL → G_OLD1 → G_OLD`,以及屏障介入的 `G_OLD0 → G_OLD1 → G_OLD`、`... → G_TOUCHED1 → G_TOUCHED2 → G_OLD`。没有任何路径让对象"变年轻"。这保证了:一个对象一旦经历过足够多轮 minor 还活着,必然收敛到 `G_OLD`,从此退出年轻代扫描。

`markold` 的断言 `age != G_OLD1`(在 `sweepgen` 里,`lgc.c:1200`)也很关键:old1 对象的年龄升级**不在 sweepgen 里做**,而是在 minor 开始时的 `markold` 里被提前升成 `G_OLD`(`lgc.c:1281`)。这是因为 old1 必须在 atomic 之前就被遍历(它可能指向 survival),所以它的升级时机和别的对象不同。这个分工保证了"old1 在被 sweepgen 扫到时,已经不是 old1 了"。

### (三)为什么 minor 可以不是增量的

P5-16 强调增量是为了"不卡宿主"。那分代的 minor 一次性跑完(`youngcollection` 在 `luaC_step` 里一次调完),会不会卡宿主?

不会,因为**年轻代很小**。minor 的触发由 `setminordebt` 控制:每新增 20% 基准字节(`MINORMUL=20`)才触发一次。这意味着每次 minor 时,年轻代大约就是"基准的 20%"——如果程序用了 10MB,一次 minor 扫的年轻代也就 2MB 上下,微秒到毫秒级就跑完。相比增量每步切得更细但总和一样,minor 用"频率高、单次小"换"单次稍重但极短",对宿主的延迟影响同样可控,且省去了增量切步的状态机开销。

但这也带来一个边界条件:**如果程序的年轻代异常大**(比如一次循环里瞬间创建海量临时对象),minor 的一次性开销可能不低。这正是 5.5 把分代设成**可选**、默认仍用增量的原因之一——增量对所有工作负载都稳健,分代在极端工作负载下可能不如增量平滑。

### (四)增量 vs 分代的取舍

两种模式各有适用场景:

- **增量**(`KGC_INC`):每轮扫全堆,对所有工作负载表现稳健。无论对象生命周期是否均衡,回收延迟都可预测、可切细。代价是每一轮都要重新标记那些长期存活的对象,有重复开销。
- **分代**(`KGC_GEN`):利用"多数对象朝生夕死",频繁便宜地回收年轻代,老年代偶尔才扫。在对象生命周期明显分化的程序上(典型:请求处理型,每请求构造大量临时对象),单次回收成本显著低于增量。代价是:年轻代异常大时 minor 一次性开销可能失控;写屏障在"老对象频繁改"的程序上(record-heavy workload)记录成本不低。

5.5 选择**默认增量**,分代作为可选优化,是一个稳健性优先的取舍。`lstate.c:361` 在状态初始化时:

```c
g->gckind = KGC_INC;
```

Lua 作为嵌入式脚本语言,无法预知宿主的负载特征。游戏引擎、Redis、nginx 这些宿主里,Lua 脚本的生命周期各异:有的短促(一次请求)、有的长寿(整个进程)。默认增量保证最坏情况下也不会有糟糕的停顿;需要分代的用户(清楚自己的负载是"朝生夕死"型)可以显式 `collectgarbage("generational")` 切过去。这个选择和 Lua 一贯的"默认稳健、高级可选"哲学一致——和它默认不自动开 JIT、默认 table 不预分配哈希区是同一种克制。

## 四、★对照 CPython + 回扣主线

### CPython 的分代 GC:三代 + 引用计数主力

CPython 的 GC 体系与 Lua 截然不同。它的**主力是引用计数**:每个对象有个 `ob_refcnt`,每次指针赋值都增减计数,归零即回收。引用计数的好处是回收极即时(对象一不可达就释放),代价是每次指针赋值都有计数开销,且**无法处理循环引用**——两个对象互相引用,各自的引用计数永远不为零,即使外部已经不可达。

为了补这个漏洞,CPython 在引用计数之上叠了一层**分代标记清除**,只用来找循环引用。它的分代结构(`Modules/gcmodule.c`):

- **三代**:generation 0、1、2。新对象进 generation 0。
- **阈值触发**:每代有个分配计数阈值(默认 0 代是 700)。generation 0 的分配数超过阈值,触发 0 代回收;0 代回收后存活的对象升入 1 代;1 代回收(更稀)后存活升入 2 代;2 代最稀。
- **只管循环引用**:分代回收时,CPython 暂时把目标代里所有对象的引用计数"减去容器内引用",减完仍大于零的是外部可达(活),归零的是循环垃圾(死)。这套算法叫 "trial deletion" / "concatenate and subtract"。

关键差异:

| 维度 | Lua 5.5 分代 | CPython 分代 |
|---|---|---|
| **定位** | 可选的完整 GC 策略,独立于增量 | 引用计数的补丁,只管循环引用 |
| **代数** | 软分代(年龄位 7 态,无独立堆) | 硬三代(0/1/2,各有独立链表) |
| **触发** | 字节增长率(`MINORMUL=20%`) | 分配计数阈值(0 代默认 700 次分配) |
| **回收范围** | 年轻代对象 + 被记录的跨代引用 | 目标代及更年轻代的所有容器对象 |
| **跨代引用处理** | 写屏障实时记录到 `grayagain` | 回收时临时遍历所有 older 代对象(无写屏障) |
| **主力** | 标记清除(分代或增量二选一) | 引用计数(每次赋值增减) |

最深刻的差别在**跨代引用处理**。Lua 用写屏障在运行期实时记录"老→新"关系,minor 时只扫年轻代 + 记录集,不用碰老年代。CPython 不用写屏障——它在回收 generation N 时,会把 generation 0..N 的所有对象都拉进来一起算引用计数减法(`gc.collect` 的 `concatenate_and_subtract`)。也就是说,CPython 的"0 代回收"实际上要遍历 0 代所有对象,而它的"2 代回收"要遍历 0+1+2 三代所有对象,代价随代数升高。

两种思路反映了两种语言的不同取向。Lua 没有"每次赋值增减计数"的开销(它的 GC 完全靠标记清除,赋值零额外成本),所以可以把分代做得精细——写屏障记录跨代引用、年轻代原子回收、老年代按字节增长率触发。CPython 因为引用计数已经是主力(每次赋值都付了计数代价),分代层只需轻量地补循环引用漏洞,不必也无力再做精细的跨代追踪。

### 回扣主线

分代 GC 是 Lua 在"摊平 GC 成本"这条路上的第二种策略。P5-16 的增量用"切细步"摊平;本章的分代用"按年龄分工、频繁便宜地回收年轻代"摊平。两者都服务于同一个核心目标:**GC 永远不成为宿主的负担**——而这正是 Lua "适合嵌入任何宿主"的命脉(P0-01 第三招)。

Lua 分代的实现处处体现"精简":它没有独立的年轻代堆,而是在同一条 `allgc` 链表上靠年龄位和几个指针切出代边界(`survival`/`old1`/`reallyold`);它没有独立的分代写屏障,而是复用增量三色的前向/后向屏障,只额外加几个年龄态(`G_OLD0`/`G_TOUCHED1`/`G_TOUCHED2`)来维护分代不变式;它的 major 子态甚至直接复用增量步进逻辑,只是名义上挂分代名下。一套三色标记 + 写屏障的基础设施,同时支撑增量和分代两种模式——这又是"用更少的机制换更多的能力"的具体落地。

而 5.5 默认仍选增量,把分代留作可选,体现了 Lua 对"嵌入友好"的另一种坚守:**默认行为必须在最坏情况下也稳健**。宿主的负载特征千差万别,Lua 无法预判,于是把"需要了解自己工作负载才能用好"的分代,交给用户显式开启。这种克制,和它把 Table 做成唯一复合结构、把字节码做成寄存器式,是同一种设计品味——不堆机制,让每一种机制都同时服务多个目的,剩下的交给用户按需打开。

至此,P5 垃圾回收的三章讲完:三色标记与增量步进(P5-16)、对象回收与弱表(P5-17)、分代 GC(本章)。Lua 的值生命周期管理,从"标记什么、怎么不卡宿主"到"怎么对付弱引用"再到"怎么按年龄分而治之",构成了一套完整而精简的体系。接下来 P6 转入 Lua 作为一门完整语言的另三块基石:元表与元方法(Table 如何充当对象)、协程(协作式多任务的栈切换)、C API(宿主与脚本的双向桥)。
