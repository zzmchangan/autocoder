## 一、主线再回扣:三招如何各自又共同化解「小 ↔ 全/快」

全书第一篇(P0-01)就立了一个核心张力:Lua 必须小到能塞进游戏引擎、Redis、nginx、嵌入式设备(一个 `lua_State` 就是一个完整且互相隔离的 VM 实例),又必须快到不像脚本、全到能写生产逻辑(闭包、协程、元表、垃圾回收,一应俱全)。小和全/快天然打架——功能多就大,要快就得为每种情况写专用代码(又变大)。Lua 不靠堆功能,而靠三招把这对矛盾同时化解。

这三招,每一招在源码里都有明确的物理根基,在全书里都对应一个篇:

### 第一招·统一:一切复合数据都是 Table

`Table`(`lobject.h:776`)是 Lua 唯一的复合结构,数组部分 `array` 与哈希部分 `node` 并存于同一个结构体。它一个结构同时承担数组、哈希、对象、命名空间、全局环境、模块、注册表。源码里没有 `list`/`dict`/`tuple`/`set` 的分立——只有 `Table`。一套创建、遍历、扩容、GC 的代码,服务所有复合数据。这是「小」在数据结构层面的根源:内核直接少了一大半类型代码。

而它没有因此变慢:整数键命中 `array` 部分时,访问代价和 C 数组完全一样(O(1) 直接下标)。统一的代价(都可能落进哈希)只发生在离散键上,而哈希部分本身用链式散列 + Brent 变体,装填因子到 100% 仍可用(P1-05)。rehash 时,`computesizes` 用 `2^k` 区间统计贪心地选出「数组部分利用率最高」的切分点,让整数键和哈希键自动在两半之间重新分配。没有这套算法,「一切复合数据都是 Table」就会变成「一切复合数据都慢」——是哈希探测和 rehash 让统一真正可行。

这一招在 P1(值的统一)展开:P1-02 讲 `TValue` 怎么把九种类型装进同一种 C 表示,P1-03 讲字符串的统一接口 + 精简分工(短串驻留/长串惰性),P1-04/P1-05 讲 Table 的合体与 rehash。

### 第二招·精简①:寄存器式字节码

Lua 的字节码是寄存器式的,操作数直接是寄存器号。`a + b` 一条 `ADDI R1 R0 2` 就完事,不像栈式那样要 `LOAD`/`LOAD`/`BINARY_OP`/`STORE` 来回压弹。指令少,意味着取指、译码、分发的次数都少——这是「快」的来源。同时指令更紧凑,字节码体积更小——这也是「小」。

P3-12 用同一段代码同时反编译 Lua 5.5 和 CPython 3.11+,六个对照实验稳定显示 Lua 指令数是 CPython 的 1/3 到 1/6,省下来的全是和业务无关的搬运指令。差异的机制根源是操作数隐式(栈式)vs 显式(寄存器式)——栈式每跨一次求值栈边界就要一条指令,寄存器式的值从不跨边界。

代价是编译器必须做寄存器分配。但 Lua 不做图着色(NP 难),而用线性分配:局部变量按声明顺序占低槽位,临时表达式结果从 `freereg` 往后顺序占用,用完即释放(P2-09)。再配合延迟表示(`expdesc`),一个表达式在被真正消费前不急着落进寄存器,而是记录「它现在是什么形态」,等消费时一次性 discharge,很多表达式可以就地优化掉(`a > 0` 直接 `GTI` 一条,不必先 `LOADI 0` 再 `LT`)。

这一招横跨编译侧和执行侧:P2-09 讲编译器怎么做线性寄存器分配,P3-10 讲 32 位指令格式怎么把操作码和三个操作数域挤进 32 位,P3-11 讲 `luaV_execute` 主循环怎么取指-译码-执行-分发,P3-12 做对照实证。

### 第三招·精简②:增量式 GC,可中断

Lua 的 GC 是增量式三色标记清除。它不一次性扫完整个堆(那样会卡住宿主几百毫秒),而把一次完整回收切成无数小步,每执行一小段字节码顺手做一小步 GC,由 `GCdebt` 控制每步工作量,停顿摊平到微秒级。

这一招对「嵌入」几乎生死攸关。游戏引擎每帧 16 毫秒,绝不能接受某帧 Lua GC 停顿 200 毫秒;Redis 用 Lua 跑事务脚本,更不能在单线程里被 GC 堵住。增量让 GC 永远不成为宿主的负担——这恰好是「适合嵌入」的核心要求。

可中断的代价是更复杂的状态机:三色不变式(白未达/灰已达未扫完/黑已扫完)、写屏障(增量期间对象引用关系会变,要回溯染灰)、多条灰链(`gray`/`grayagain`/`weak`/`ephemeron`)。P5-16 把这套机制一行行讲清楚,P5-17 讲各种对象的标记和弱表,P5-18 讲分代模式(增量之外另一种摊平成本的策略)。

### 三招的共同指向

三招不是各管一头,而是相互成全。Table 既是数组又是哈希又是对象,一套代码服务多种用途(小),同时整数键走数组 O(1)(快);一条 `ADDI` 既取操作数又算又存结果(快),指令少也就紧凑(小);增量 GC 既回收又不阻塞(友好),按需推进(平滑)。三招的共同指向是同一件事:**砍掉一切可砍的机制,让剩下每一种机制都同时服务多个目的。** 这就是「用更少的机制换更多的能力」,也是全书主线在源码层面的具体含义。

而且这三招不是孤立的工程优化,它们落到同一个东西上:`lua_State`。Lua 的所有运行时状态——值栈、调用帧、upvalue、错误恢复点、调试钩子——全装在 `struct lua_State`(`lstate.h:285`)这一个结构里;所有线程共享的全局状态(GC、字符串表、注册表、内存分配器)在 `global_State`(`lstate.h:327`)里,由 `l_G` 指针指向。宿主链接几个 `.c` 文件、调 `luaL_newstate()` 就得到一个全新、独立的 VM 实例。这个布局是三招能落地、Lua 能嵌入的物理根基。

## 二、数据流全景:一段 Lua 代码从源码到执行的完整旅程

把全书各章串成一条数据流。取一段最常见的代码:

```lua
local t = {}
t[1] = "a"
t.name = "lua"
local s = t.name
```

它从源码到执行,经历下面每一步,每一步都对应一篇章节:

```
        源码文本 (字节流)
              │
              │  ① 词法分析 llex            ── P2-06
              ▼
         token 流 (TK_LOCAL / TK_NAME("t") / '=' / '{' / ...)
              │
              │  ② 语法分析 lparser          ── P2-07 / P2-08
              │     (递归下降 + 算符优先,不建 AST,边解析边生成)
              │     · 作用域:BlockCtx 链分配栈槽      ── P2-08
              │     · upvalue:singlevaraux 静态登记  ── P2-08
              ▼
         表达式中间表示 (expdesc:延迟表示)
              │
              │  ③ 代码生成 lcode            ── P2-09
              │     · 线性寄存器分配 (freereg)
              │     · 常量表去重 (kcache)
              │     · 跳转回填
              ▼
      32 位寄存器式字节码 (Proto->code[])
      (NEWTABLE / SETLIST / SETFIELD / GETFIELD / RETURN0)
              │
              │  ④ 指令编码                   ── P3-10
              │     32 位 = 操作码 7 位 + A/B/C 三域
              ▼
         luaV_execute 主循环                 ── P3-11
              │  取指 → 译码 → 执行 → 分发
              │
              │  ⑤ 访问 Table                ── P1-04 / P1-05
              │     · t[1] 走 array 部分 O(1)
              │     · t.name 走 hash 部分
              │       (name 是短串驻留,指针比较)
              │                              ── P1-03
              ▼
         函数调用 (如果有 OP_CALL)            ── P4-13
              │  · 建 CallInfo 帧
              │  · 实参就地变被调的 R0..
              │  · 闭包/upvalue 的开合        ── P4-14
              │  · 尾调用/可变参数/to-be-closed ── P4-15
              ▼
         元方法回退 (如果操作未直接命中)       ── P6-19
              │  __index / __newindex / __add ...
              ▼
         执行过程中不断分配对象
         (Table / 字符串 / 闭包 / upvalue)
              │
              │  ⑥ GC 回收                   ── P5-16 / P5-17 / P5-18
              │     增量三色标记,可中断
              ▼
         内存被回收,VM 继续跑
```

把这条流对应到章节,就是下表:

| 步骤 | 发生的事 | 对应章节 | 数据形态 |
|---|---|---|---|
| ① 词法 | 字符流切 token | P2-06 | token(`LexState`) |
| ② 语法 | token 流变字节码(不建 AST) | P2-07 / P2-08 | `expdesc` + `FuncState` |
| ③ 代码生成 | 寄存器分配 + 指令发射 | P2-09 | `Proto->code[]` |
| ④ 指令编码 | 32 位拆解 | P3-10 | `Instruction`(32 位) |
| ⑤ VM 执行 | 取指-译码-执行-分发 | P3-11 / P3-12 | 值栈槽(`TValue`) |
| Table 访问 | 数组/哈希两半 | P1-04 / P1-05 | `Table`(`array`/`node`) |
| 字符串 | 短串驻留/长串惰性 | P1-03 | `TString` |
| 值表示 | tagged 编码 | P1-02 | `TValue`(16 字节) |
| 函数调用 | CallInfo 帧压栈 | P4-13 | `CallInfo` 链 |
| 闭包 | upvalue 开合 | P4-14 | `UpVal`(open/closed) |
| 边界语义 | 尾调用/vararg/tbc | P4-15 | TAILCALL / `PF_VAHID` / `tbclist` |
| 元方法 | Table 充当对象 | P6-19 | metatable + flags 缓存 |
| 协程 | 栈切换 | P6-20 | 独立 `lua_State` |
| C API | 宿主桥接 | P6-21 | 栈式 API |
| GC 回收 | 增量三色标记 | P5-16 / P5-17 / P5-18 | `global_State` 的灰链 |

这张表是全书 21 章的一次性速查。读任何一章卡住时,回到这张表看它在数据流的哪个位置,就能立刻定位它上下游是什么。

注意一个贯穿的事实:数据流的每一站,操作的都是同一种东西——**值**(`TValue`)。词法器把字面量认成值,编译器把值塞进常量表,VM 从寄存器(值栈槽)读值、对值做运算,Table 存的是 `TValue`,GC 扫的是值栈上的 `TValue` 和堆上 GC 对象里的 `TValue`。P1 立下的「万物皆值」是整条数据流的统一货币。这也是为什么 P1 必须先读、必须吃透——后面所有章节都建立在 `TValue` 和 `lua_State` 这两个结构上。

## 三、编译侧 ↔ 执行侧 二分地图

全书按「编译侧(源码→字节码)↔ 执行侧(VM 执行 + 管值生命周期)」的二分法组织,数据根基(P1)是两边共用的基础语言。下表把 21 章铺进这张地图:

| 篇 | 章 | 标题 | 二分法位置 | 主线落点 |
|---|---|---|---|---|
| **P0** | P0-01 | 第一性原理 | 全书入口 | 立主线:三招化解「小↔全/快」 |
| **P1 数据根基** | P1-02 | lua_State 与 TValue | 共用根基 | 统一:九种类型一种 C 表示 |
| | P1-03 | 字符串 | 共用根基 | 统一接口 + 精简分工 |
| | P1-04 | Table(上) | 共用根基 | 统一的皇冠:数组+哈希合体 |
| | P1-05 | Table(下) | 共用根基 | 统一可行:rehash 自动切分 |
| **P2 编译侧** | P2-06 | 词法分析 llex | 编译侧入口 | 精简:查找表 + switch,760 行 |
| | P2-07 | 语法分析(上) | 编译侧 | 精简:不建 AST,边解析边生成 |
| | P2-08 | 语法分析(下) | 编译侧 | 精简:作用域/upvalue 前移到编译期 |

| | P3-11 | 解释器循环 luaV_execute | 执行侧核心 | 精简:一个 for 循环 + 一张操作码表 |
| | P3-12 | 寄存器式精解 | 执行侧收尾 | 精简换快的实证(★对照 CPython) |
| **P4 执行侧** | P4-13 | 调用栈与调用约定 | 执行侧·调用 | 精简:CallInfo 一结构吃两种函数 |

几个观察:

**P1 是共用的地基,不能跳。** P2 编译器和 P3 VM 都得读写 `TValue`;P4 闭包用 `UpVal` 指向值栈槽;P5 GC 靠 `iscollectable` 扫值栈;P6 C API 在值栈顶推取 `TValue`。每一篇都回到 `TValue` 和 `lua_State` 这两个结构。所以全书建议的读法是:P0-01 到 P1 先顺序读,把「值」的表示吃透,之后 P2(编译侧)和 P3(执行侧)可以并行读,它们正好对应「源码怎么进来」和「字节码怎么跑出去」两条线。

**编译侧只有 4 章(P2-06 到 P2-09),执行侧有 13 章(P3 到 P6)。** 这不是篇幅失衡,是 Lua 的真实重心分布。Lua 的编译器极小(`lparser.c` 2193 行 + `lcode.c` 1971 行,且不建 AST,一遍解析直接出字节码),执行侧才是运行时的命脉——VM 循环、调用栈、GC、元表、协程、C API,每一块都比编译器重。这也是「小」的另一种体现:Lua 把编译期压到最短,执行期承担一切。

**P4(调用)、P5(GC)、P6(语言基石)都是执行侧,但角度不同。** P4 讲值在栈上的生命周期(调用、闭包、返回),P5 讲值在堆上的生命周期(GC 回收),P6 讲值怎么被组织成一门完整语言(元表给 Table 加类型行为、协程给 VM 加并发、C API 给宿主加桥)。三层合起来,才是一门「完整的语言」。

## 四、关键设计取舍清单

全书散落着许多「Lua 为什么这么选」的设计决策。这里把它们汇总成一份清单,每条一句话讲 why,附上出处章节:

1. **tagged value 内联(P1-02)**:基本类型(整数/布尔/浮点)用 16 字节 `TValue` 直接内联值,不堆分配。why:嵌入场景每帧创建海量临时值,零 malloc 才能快;访问零解引用(值和标签同缓存行)。

2. **Table 数组+哈希合体(P1-04)**:一个结构里 `array` 和 `node` 并存。why:一套创建/遍历/GC 代码服务所有复合数据(小),整数键走数组 O(1)(快);代价是离散键落哈希。

3. **Node 键字段拆开存放 + 5.5 倒排数组(P1-04)**:`Node` 的值和键 next 偏移分开存,5.5 进一步把数组的 tag 和 value 分离成两个平行数组。why:消灭 padding,大数组省内存;tag 单字节判空快。

4. **dummynode 共享空表哈希(P1-04)**:空表的哈希部分指向一个全局共享的 `dummynode`,不分配。why:空表零哈希内存,热路径少一次分支——「不为空状态付费」。

5. **flags 元方法缓存(P1-04)**:`Table.flags` 一个字节,每位表示某个元方法「不存在」。why:99% 的字段访问 `checknoTM` 一步短路,不必每次都去查 metatable。

6. **链式散列 + Brent 变体(P1-05)**:哈希冲突用 `Node.next` 偏移链 + Brent 变体(把 main position 的占有者挪走让新键归位)。why:装填因子到 100% 仍可用,短链;4 字节偏移换紧凑。

7. **rehash 时数组/哈希自动切分(P1-05)**:`computesizes` 统计每个 `2^k` 区间的整数键数,贪心选「数组部分利用率最高」的切分点。why:让统一 Table 不付性能税——该进数组的进数组,该进哈希的进哈希。

8. **短串全局驻留 + 长串惰性哈希(P1-03)**:短串(≤40 字节)intern 到全局表,指针比较即相等;长串不驻留,惰性算哈希。why:短串当 key 比较退化成 O(1) 指针比较(字段访问接近 C struct 速度);长串不驻留免得驻留表被撑大。

9. **external string(P1-03,5.5 新)**:`LSTRFIX`/`LSTRMEM` 让宿主把不归 Lua 管的内存包装成 Lua 字符串。why:嵌入场景零拷贝(Redis 把网络字节直接当 Lua 字符串)。

10. **单遍编译,不建 AST(P2-07)**:递归下降 + 算符优先,解析一遍直接产 `Proto->code`。why:编译期内存 O(深度)而非 O(源码大小);`loadstring` 延迟接近配置解析;砍掉一整个中间表示层。

11. **算符优先表(18 行)而非多层递归下降(P2-07)**:一张优先级表 + 一个 `subexpr` 函数搞定所有二元运算符。why:折叠了传统递归下降十几层子程序,代码更紧凑;代价是不如 PEG 直观。

12. **全局变量统一进 upvalue(P2-08)**:`_ENV` 只是个普通 upvalue,全局访问走 `GETTABUP`(即一次 table indexed)。why:没有专门的全局表查询机制,机制更少;代价是全局访问也走一次哈希。

13. **upvalue 编译期静态登记 + 运行期开合(P2-08 + P4-14)**:`singlevaraux` 在编译期沿 `FuncState` 链把外层局部变量登记成 upvalue(两个字节 `instack`/`idx`);运行期 `UpVal` 有 open(指向栈)/closed(拷出)两态。why:运行时不查名字、不沿调用链回溯;外层访问自己局部变量零间接,内层访问 upvalue 一次间接;只有返回那一刻付一次拷贝。

14. **线性寄存器分配 + 延迟表示(P2-09)**:局部变量按声明顺序占低槽,临时结果从 `freereg` 往后;`expdesc` 延迟到消费时 discharge。why:不做图着色(NP 难),编译器小;很多表达式就地优化(`a > 0` 直接 `GTI` 一条)。

15. **32 位指令 + 多种编码域(P3-10)**:一条指令 32 位,操作码 7 位,其余是 A/B/C(8/8/8)或 ABx/AsBx/iABx/iAsBx/iAx/sJ 等多种编码;常量可编码进操作数(K 位变体)。why:挤进尽可能少、尽可能紧凑的指令;5.5 把 RK 的 K 位挪到操作码域(`luaP_isOT`/`isIT`)。

16. **解释器一个 for 循环 + goto jumptab 分发(P3-11)**:`luaV_execute` 是 `for(;;)` + 一张操作码表,computed goto(可用时)或 switch 分发;`Protect`/`ProtectNT`/`halfProtect` 三档 GC 钩子穿插。why:没有线程化、没有 JIT,极简;`RETURN0`/`RETURN1` 在无钩子时手写快路径不调 `luaD_poscall`。

17. **CallInfo 一结构吃 Lua 和 C 两种函数(P4-13)**:`CallInfo.u` 联合让同一个结构既跑 Lua 函数(用 `savedpc`/`trap`)又跑 C 函数(用 `k`/`ctx`),靠 `CIST_C` 一位区分。why:一套调用机制(`luaD_precall`/`poscall`)服务两种函数,没有为 C 函数另造调用栈。

18. **实参就地变寄存器,零拷贝(P4-13)**:实参压在 `func+1..`,新帧 `base = func+1`,实参就地成为被调的 R0..。why:没有专门的参数传递步骤、没有跨数组搬运——寄存器式 VM 在调用约定上的红利。

19. **Lua→Lua 调用不压 C 栈(P4-13)**:`goto startfunc`/`returning` 把调用链做成数据结构(CallInfo 链),深递归便宜。why:调用链深度只受值栈容量限制,不受 C 栈限制;一个 `CIST_FRESH` 位区分逻辑帧和物理 C 帧。

20. **proper tail call(P4-15)**:`return f()` 把 `f` 的帧直接盖在当前帧位置,复用栈空间。why:尾调用不增长调用栈,可以无限递归;一个判断(`!insidetbc`)+ 一个搬运(`luaD_pretailcall`)省掉一整层调用帧。这是 Lua 函数式编程(递归下降、状态机)的物理基础。

21. **可变参数两套机制按需切换(P4-15,5.5 新)**:`PF_VAHID`(实参留栈,转发场景不付表分配代价)和 `PF_VATAB`(实参打包成 Table),编译器根据函数体怎么用 `...` 自动二选一;新增 `OP_GETVARG` 取单个可变参数。why:一套语法两种实现,常见场景(转发)走零分配快路径。

22. **to-be-closed 用变量属性 + 栈槽复用链表(P4-15)**:`tbclist` 把 to-be-closed 变量串成链表复用栈槽,`luaF_close` 受保护关闭。why:不引入新语句(只是变量属性)、不分配节点、错误隔离(多个 `__close` 错误串成链不互相挡)——用最少新机制拿下 RAII。

23. **增量三色标记,可中断(P5-16)**:GC 切成无数小步,`GCdebt` 控制每步工作量,写屏障维持增量期正确性。why:停顿摊平到微秒级,绝不卡宿主——嵌入对延迟敏感场景(游戏、Redis)的命脉。

24. **弱表寄生在 Table + metatable(P5-17)**:弱表不是新类型,是普通 Table 加 `__mode` 元字段后的回收策略;弱引用清理推迟到 atomic 一个瞬间。why:不为弱引用引入新用户可见类型;atomic 集中清理避免了增量期维护「弱引用对象可能死」的复杂状态。

25. **finalizer 的复活机制(P5-17)**:带 `__gc` 的对象被标死后复活(挂回 allgc),下轮才真正回收,和对象的「逻辑死亡」分开。why:复活成为一等公民,Lua 程序惯用 `__gc` 做清理且知道对象此时还在内存里可被安全访问;finalizer 只在 GC 专属阶段跑,不在任意分配点抢控制权。

26. **分代模式复用三色基础设施(P5-18)**:没有独立年轻代堆,在同一条 `allgc` 链上靠年龄位切代边界;没有独立分代写屏障,复用增量三色的前向/后向屏障,只加几个年龄态。why:一套三色标记 + 写屏障支撑增量和分代两种模式——用更少的机制换更多的能力;5.5 默认仍选增量(最坏情况稳健),分代可显式开启。

27. **元表复用 Table(P6-19)**:元方法是约定好名字的字段,VM 在固定时机去查 metatable。why:不专门造 `Type` 结构体(否则破坏「一切复合数据都是 Table」的统一,要多一套类型系统代码);metatable 也是 Table,GC 照常回收;代价是类型行为查询比内建类型系统慢(哈希查找),靠 flags 缓存弥补。

28. **协程 = 独立 lua_State(P6-20)**:一个协程就是一个 `lua_State`(类型 `LUA_TTHREAD`),resume/yield 就是 C 函数调用进入 + longjmp 弹出。why:不引入 OS 线程、调度器、就绪队列;`lua_State` 这个已有结构服务「主 VM 实例」和「协程」两种角色;`nCcalls` 高 16 位一个计数器把「不能跨 C 边界 yield」物化成一行检查(sound 的代价)。

29. **栈式 C API(P6-21)**:宿主通过压值、调函数、取结果的栈式 API 和 VM 交互。why:`lua_State` 本就有值栈,复用它做宿主桥,不必另造接口;宿主语言(C/C++/Rust/Go)都能对接。

30. **错误用 setjmp/longjmp(P4-13)**:错误发生时直接撕开 C 栈。why:快但暴力(C 栈析构不跑,所以 `luaD_pcall` 要手动 `luaF_close`);对照 CPython 用返回码 + 异常对象逐层传播(慢但干净)。Lua 选快,CPython 选稳。

每一条都是「用更少的机制换更多的能力」在某个具体机制上的落地。把它们叠起来,就是 Lua 又小又快、又能塞进任何宿主的全部根由。

## 五、★对照 CPython 总表

全书每章固定一栏 ★对照 CPython。这里把各章的对照栏汇总成一张大表,按维度分:

| 维度 | Lua 5.5 | CPython | 出处 |
|---|---|---|---|
| **值表示** | tagged `TValue`,16 字节内联(小整数/布尔/浮点零堆分配) | `PyObject` 堆 box + `ob_refcnt` + `ob_type`(每值一次 malloc) | P1-02 |
| **复合数据** | 只有 `Table` 一个结构,数组/哈希/对象/模块全是它 | `list`/`dict`/`tuple`/`set`/`frozenset`/`bytes` 多种内建类型,各自一套 C 实现 | P1-04 |
| **字符串驻留** | 硬阈值 40 一刀切,驻留表拉链哈希 | 启发式规则(identifier-like),版本间不同,复杂度高 | P1-03 |
| **外部内存** | `LSTRFIX`/`LSTRMEM` 可包装宿主内存,零拷贝 | 无,字符串内容归对象所有 | P1-03 |
| **哈希实现** | 链式 + Brent 变体,装填因子到 100% 仍可用,tag array 倒排紧凑 | 开放寻址,装填因子控制在 2/3 左右 | P1-05 |
| **指令模型** | 寄存器式(操作数即寄存器号) | 栈式(操作数隐含在栈顶) | P3-10/P3-12 |
| **代码生成** | 线性寄存器分配 + 延迟表示(`expdesc`) | 几乎不做分配,直接压栈弹栈 | P2-09 |
| **中间表示** | 不建 AST,单遍编译直接产字节码 | 先建 AST(`parser.c`),再 `compile.c` 遍历 AST 发射 | P2-07 |
| **算符优先** | 一张 18 行表 + 一个 `subexpr` | PEG 文法,每个优先级一条独立规则 | P2-07 |
| **全局变量** | 统一进 upvalue(`_ENV`),`GETTABUP` 即 indexed | 单独一套 `LOAD_NAME`/`STORE_NAME`(查 `f_locals` miss 才查 globals) | P2-08 |
| **闭包捕获** | upvalue 开合两态(open 指向栈/closed 拷出) | `cell` 对象,一开始就上堆,没有开合两态 | P4-14 |
| **解释器分发** | `for(;;)` + goto jumptab(可用时)/switch | switch + specializing adaptive interpreter(3.11+ 内联缓存) | P3-11 |
| **指令数** | 同一段代码指令数 1/3 到 1/6 | 指令多,但 3.11+ 专用化指令部分弥补 | P3-12 |
| **调用栈** | `CallInfo` 一结构吃 Lua/C 两种函数,实参零拷贝变寄存器,Lua→Lua 不压 C 栈 | 建 frame / `PyCFunction` 走 C | P4-13 |
| **尾调用** | proper tail call,不增长栈,可无限递归 | 无尾调用优化 | P4-15 |
| **可变参数** | 两套机制按需切换(`PF_VAHID`/`PF_VATAB`),5.5 新增命名可变参数 `OP_GETVARG` | `*args` 打包成 tuple | P4-15 |
| **资源管理** | to-be-closed 变量属性 + `__close` 元方法 + 受保护关闭链 | `with` 语句 + `__exit__` 协议 + `ExitStack` | P4-15 |
| **错误处理** | `setjmp`/`longjmp`,快但暴力 | 返回码 + 异常对象逐层传播,慢但干净 | P4-13 |
| **GC 主力** | 追踪式三色标记(增量或分代二选一),赋值零额外成本 | 引用计数(每次赋值增减),即时回收 | P5-16 |
| **循环引用** | 三色标记天然处理 | refcnt 处理不了,靠独立分代周期(那是一次 STW) | P5-16 |
| **停顿特征** | 增量小步 + 一次 atomic,停顿摊平微秒级 | 每次赋值 refcnt 开销(纳秒级但极频繁);分代周期是完整停顿 | P5-16 |
| **对嵌入友好度** | 高:停顿可控、可调参 | 中:refcnt 散布每次赋值;分代 STW 大堆上几十毫秒 | P5-16 |
| **分代实现** | 复用三色 + 写屏障,allgc 链上靠年龄位切代,minor 只扫年轻代 + 记录集 | 不用写屏障,回收 N 代时把 0..N 代全拉进来算 refcnt 减法 | P5-18 |
| **弱引用** | 弱表(`__mode` 元字段),寄生在 Table | `weakref` 单独类型 + `__del__` finalizer | P5-17 |
| **finalizer** | `__gc` 在专属 `GCScallfin` 阶段,复活是一等公民 | `__del__` 在 refcount 归零瞬间调,复活棘手 | P5-17 |
| **类型系统** | metatable + 元方法,Table 充当对象,继承靠 `__index` 递归单链 | `class`/`type` 内建,MRO C3 线性化,双下划线方法 | P6-19 |
| **并发** | 协程(独立 `lua_State` + resume/yield + longjmp),无 event loop | generator + asyncio(有 event loop / 就绪队列) | P6-20 |
| **C API** | 栈式 API(压值/调函数/取结果) | 引用计数式对象协议 | P6-21 |

每一个对照都揭示一种取舍。把它们横着看,会发现一个规律:**Lua 几乎在每个维度都选了「机制更少、更紧凑、把代价集中到少数关键点」的那一边,CPython 几乎在每个维度都选了「机制更专用、更内建、代价散布」的那一边。** 这不是偶然,是两个 VM 服务不同生态位的结果:Lua 为嵌入而生(每帧 16 毫秒的预算、零容忍长停顿、内核要小),CPython 为通用脚本而生(能力优先、生态优先、标准库优先)。两套都 sound,但服务的目标不同。

读懂这张总表,就读懂了「Lua 为什么这么选」的相对位置——每一个设计决策,都不是凭空做的,而是和 CPython 摆在一起时,它主动站到了「精简换小换快」的那一侧。

## 六、5.5 vs 老资料关键演进汇总

这是本书相对旧书(讲 5.3/5.4 的书和博客)的核心增量。Lua 5.5.0(2025 年 PUC-Rio)相对 5.3/5.4 有一批实质演进,讲老版本的书完全没提。本书所有讲解以 5.5.0 源码行号为准,凡与老资料冲突处显式标注。这里把全书标注的 5.5 演进集中成一张表:

| # | 演进点 | 5.5 实况 | 老资料(5.3/5.4) | 出处 |

| 25 | **字符串驻留阈值** | 5.5 短串阈值是 40(`LUAI_MAXSHORTLEN`) | 老资料数值一致但 variant/kind 机制是 5.4 后引入 | P1-03 |
| 26 | **字符串 variant/kind** | 5.5 `TString` 有 variant(短/长)和 kind(内部/external)两维标记 | 老资料只有单一的 `TString` 标记 | P1-03 |
| 27 | **`lenhint` 藏数组中缝** | 5.5 Table 的长度提示藏在数组部分中缝,不占 `Table` 字段 | 老资料 `#t` 实现不同 | P1-04 |
| 28 | **词法器行数** | 5.5 `llex.c` 约 760 行 | 与老版本接近,但 5.5 token 种类随语法增加 | P2-06 |

这 28 条就是本书相对旧书的核心增量。讲老版本的书(不管是讲 5.3 还是 5.4)在这些点上要么没提、要么讲的是已经过时的实现。如果你读过其他 Lua 内核的书或博客,再读本书,这 28 条是你最该更新的认知。

其中几条尤其值得单独点出,因为它们是「读老资料会得到错误印象」的硬差异:

- **`StkIdRel` 相对表示(#1-2)**:这是 5.5 为栈搬家优化做的根本性改动。老资料讲 `StkId` 绝对指针,会让人以为栈扩容要遍历更新所有指针——5.5 已经不需要了。
- **可变参数双机制 + 命名可变参数(#7-8)**:5.5 的可变参数语义和老版本完全不同。讲 5.4 的书讲的是单一 `OP_VARARG`,5.5 是两套机制 + 全新的命名可变参数和 `OP_GETVARG`。
- **指令编码 6 种 + `sJ` + RK 改 k 位(#9-11)**:5.5 对 32 位指令的编码做了重排,新增了 `sJ` 跳转域,把 K 位挪到操作码域。这是 5.5 指令编码重排的一部分,直接影响怎么读 `lopcodes.h`。
- **GC 三模式 + 年龄 7 态 + 参数数组化(#17-21)**:5.5 的 GC 子系统改动很大,讲老版本 GC 的资料在这些点上会误导。5.5 分代是复用增量基础设施实现的,年龄态更细,默认还选增量。
- **算术 + MMBIN 指令对(#13)**:5.3 的算术指令内部自己判元方法,5.4/5.5 是算术指令 + 紧跟一条 MMBIN,失败时回看上一条。这是 5.4 引入、5.5 沿用的优化,讲 5.3 的资料看不到 `pi = *(pc - 2)` 这种写法。

## 七、收尾:用更少的机制换更多的能力

回到 P0-01 立的那个命题:**Lua 用「统一」和「精简」,同时拿下「小」和「快」。**

读完这 21 章,应该建立起一个整体认知:Lua 的每一个设计——从 `TValue` 的 16 字节内联,到 Table 的数组哈希合体,到单遍编译不建 AST,到线性寄存器分配,到一个 for 循环的解释器,到可中断的增量三色 GC,到寄生在 Table 上的元表,到切 `lua_State` 的协程——都不是孤立的优化,而是同一种设计品味的反复落地。这种品味浓缩成一句话:

**砍掉一切可砍的机制,让剩下每一种机制都同时服务多个目的。**

- Table 既是数组又是哈希又是对象,一套代码服务所有复合数据。
- 一条 `ADDI` 既取操作数又算又存结果,省掉栈式来回压弹。
- 增量 GC 既回收又不阻塞,把停顿摊平到微秒级。
- upvalue 编译期静态登记 + 运行期开合,运行时不查名字。
- metatable 复用 Table,让 Table 拿到类型系统的能力而不引入新结构。
- 协程复用 `lua_State`,让「并发」压成几次 C 函数调用 + 一次 longjmp。

每一种机制都在身兼数职。代价当然有:Table 的统一让离散键不如专用 dict 紧凑;寄存器式让编译器要做分配;增量 GC 需要三色状态机和写屏障;upvalue 要处理开合两态;metatable 的类型查询比内建类型系统慢(靠 flags 缓存弥补);协程不能跨 C 边界 yield。但每一个代价都被刻意集中到少数关键点,让常见路径尽量短。这是 Lua 和 CPython 最根本的气质差别:CPython 把代价散布(每次赋值改 refcnt、每个值堆 box),Lua 把代价集中(编译期一次分配、返回时一次拷贝、GC 专属阶段一次 finalizer)。

这种「把代价集中到少数关键点,让常见路径尽量短」的取舍,不是某个功能的局部优化,而是一种贯穿全书的设计哲学。它让 Lua 用 32106 行 C 源码(5.5.0 全量),造出了一门完整的、相当快的、能塞进任何宿主的脚本语言。这是「用更少的机制换更多的能力」最干净的工程样本,也是 Lua 内核比 V8/CPython/JVM 都小、却最适合拿来讲清「脚本语言 VM 是怎么搭起来的」的根本原因。

读完本书,你应该得到的不是一堆函数名和行号,而是这三个判断力:

1. **看到一个设计,能判断它在「小↔全/快」张力里站在哪。** 比如看到一个 VM 用栈式指令,你能立刻知道它选了「编译器简单、执行期指令多」这一侧,和 Lua 相反。
2. **看到一个机制,能问出「它身兼几职、代价集中在哪里」。** 比如 Lua 的 `UpVal` 兼了 open/closed/搬家三态,代价集中在返回那一刻的关闭逻辑。
3. **看到一个 Lua 实现,能分辨它是 5.5 还是老版本。** 比如 `StkIdRel` 相对表示、可变参数双机制、指令编码 6 种、GC 三模式——这些是 5.5 的硬特征,讲老资料的书没有。

这三条判断力,就是这本书相对一本语法教程或一本 API 手册的全部增量。Lua 的内核小到能完整读懂,大到能讲清一门脚本语言 VM 的所有命脉——这正是它作为「最佳教学样本」的魅力,也是这本书一行行带你走完它的意义。
