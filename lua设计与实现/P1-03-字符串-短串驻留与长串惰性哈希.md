# P1-03 字符串:短串驻留与长串惰性哈希

> **本书主线**:统一与精简换小而快。**二分法**:编译侧(源码→字节码)↔ 执行侧(VM 执行 + 管值生命周期);本章在**数据根基(P1)**。**本章落点**:统一接口下的精简分工——所有字符串用同一种 `TString` 表示,但内部按使用频率分两种策略(短串驻留 / 长串惰性),同时服务"快"(key 比较变指针比较)和"小"(去重省内存)。**★对照**:CPython 的 `PyUnicodeObject`(也 intern,但粒度/机制不同;Python 字符串带引用计数)。**源码**:lua-5.5.0。**基调**:纯直球,不用比喻。

---

## 一、这章解决什么问题

先回到一个最普通的 Lua 写法:

```lua
local t = { name = "alice", age = 30 }
print(t.name)
```

`t.name` 这一行,VM 要做的事是:拿字符串 `"name"` 当 key,到 table `t` 里查它对应的值。这个查的过程,在 P1-04 会细讲,核心是一句"在哈希桶的链上找 key 相等的节点"。这里有一个看似不起眼、却决定性能的关键问题:**"key 相等"怎么判定**?

最朴素的答案是从 table 的那个 node 里取出 key、和当前这个 `"name"` 逐字节 `memcmp`。但字符串 key 在 Lua 里出现得极其频繁——对象的每个字段名、全局变量的每个名字、每个表的每个键,几乎都是字符串。如果每一次 table 访问都要逐字节比较,这笔开销是付不起的。

Lua 的解法是:把字符串**全局去重**。所有内容相同的短字符串,在 VM 里只存在一个对象,所有用到 `"name"` 的地方,手里拿的都是**同一个 `TString` 指针**。这样一来,key 相等的判定退化成**指针相等**——一次 `==` 比较,O(1),不必看内容。这就是"短串驻留(interning)"。

但驻留不是没代价的:每创建一个串都要去一张全局哈希表里查一遍,长字符串(几 KB、几十 KB 的源码、配置、JSON)如果也驻留,这张表会被撑得极大,而长串重复的概率又很低(两段完全一样的长文本很少见),驻留的收益微乎其微。所以 Lua 做了分工:

- **短串(长度 ≤ 40)**:驻留。全局唯一,key 比较变指针比较。
- **长串(长度 > 40)**:不驻留。每次 `luaS_newlstr` 都新建一个对象,hash 也只在需要时才算(惰性哈希)。

这一刀切下去,把"快"(短串指针比较)和"省"(短串去重 + 长串不占驻留表)同时拿到。而对外,无论是短串还是长串,都是一个 `TString`,所有读它内容的接口(`getstr`/`tsslen`)都统一——这就是"统一与精简"在字符串上的体现:**一种表示,两套策略,一个接口**。

这章要讲清楚四件事:① `TString` 这个统一表示长什么样,短串和长串在同一个结构里怎么区分;② 短串是怎么驻留进那张全局表的(`luaS_newlstr`/`internshrstr` 流程);③ 长串为什么不驻留、hash 为什么惰性算、那个随机 `seed` 在防什么;④ GC 怎么把短串从表里清掉。最后对照 CPython 的 intern,看两种语言在这同一问题上各自的选择。

## 二、源码怎么实现

### 2.1 统一表示:`TString` 结构

所有 Lua 字符串,无论短长,都是一个 `TString`。它定义在 `lobject.h:405`:

```c
typedef struct TString {
  CommonHeader;
  lu_byte extra;  /* reserved words for short strings; "has hash" for longs */
  ls_byte shrlen;  /* length for short strings, negative for long strings */
  unsigned int hash;
  union {
    size_t lnglen;  /* length for long strings */
    struct TString *hnext;  /* linked list for hash table */
  } u;
  char *contents;  /* pointer to content in long strings */
  lua_Alloc falloc;  /* deallocation function for external strings */
  void *ud;  /* user data for external strings */
} TString;
```

逐字段看。

`CommonHeader` 是所有可回收对象都有的头(`lobject.h:301`):`struct GCObject *next; lu_byte tt; lu_byte marked;`。`tt` 是带 variant 的类型 tag,字符串有两种 variant(`lobject.h:373`):

```c
#define LUA_VSHRSTR  makevariant(LUA_TSTRING, 0)  /* short strings */
#define LUA_VLNGSTR  makevariant(LUA_TSTRING, 1)  /* long strings */
```

短串的 `tt` 是 `LUA_VSHRSTR`,长串是 `LUA_VLNGSTR`。同属 `LUA_TSTRING`,只是 variant 不同。

`extra` 是个复用字段。对短串,它标"这是不是保留字"(reserved word),保留字 `extra > 0`(`lstring.h:48` 的 `isreserved` 宏);对长串,它标"hash 算过了没",0 表示还没算、1 表示已算(`luaS_hashlongstr` 里会改它,见后)。同一个 `lu_byte`,两种语义,靠 variant 区分。

`shrlen` 是个带符号字节(`ls_byte`)。对短串,它存的是字符串长度(0 到 40,所以一个字节够);对长串,它**存负数**——而且不是随便的负数,是长串的"种类"。看 `lobject.h:397`:

```c
/* Kinds of long strings (stored in 'shrlen') */
#define LSTRREG  -1  /* regular long string */
#define LSTRFIX  -2  /* fixed external long string */
#define LSTRMEM  -3  /* external long string with deallocation */
```

这是 5.5 一个需要专门讲的变化:**长串分三种**。普通的、内容在对象自己内存里的长串是 `LSTRREG`;通过 C API `lua_pushexternalstring` 挂进来的、内容不归 Lua 管、不可回收外部内存的是 `LSTRFIX`;同样外部内容但可以由用户提供的 `falloc`/`ud` 在 GC 时回收的是 `LSTRMEM`。老资料(讲 5.3/5.4 的)只讲"长串就是长串"一种,5.5 把外部串(external string)这件事正式化了,多了 `falloc`/`ud` 两个字段和三种 kind。`shrlen >= 0` 就是短串,`< 0` 就是长串,判断用 `strisshr` 宏(`lobject.h:420`):`#define strisshr(ts) ((ts)->shrlen >= 0)`。

`hash` 是哈希值。短串在创建时就算好(驻留要用它),长串创建时先放一个随机 seed 进去(见 2.4),真正的内容 hash 惰性算。

`u` 是个 union。短串驻留在全局哈希表里,要用 `hnext` 串成链;长串不驻留,不需要 `hnext`,改用 `lnglen` 存长度(长串长度可能很大,放 `size_t`)。两者互斥:驻留的短串用 `hnext`,不驻留的长串用 `lnglen`。

`contents` 是 5.5 的关键字段。对长串,它指向字符串的实际内容——可能是对象自己的内存后面(`LSTRREG`),也可能是外部内存(`LSTRFIX`/`LSTRMEM`)。对短串,`contents` 字段本身不存指针,**短串的内容紧跟在 `contents` 字段后面**(也就是结构体的末尾),用 `rawgetshrstr` 宏取地址:

```c
#define rawgetshrstr(ts)  (cast_charp(&(ts)->contents))
```

`&(ts)->contents` 是 `contents` 字段的地址;因为短串内容紧跟其后,这个地址就是字符串首字节。这种"结构体末尾跟柔性内容"的技巧,省掉一次指针解引用。`getstr` 宏根据短长二选一:

```c
#define getstr(ts)  (strisshr(ts) ? rawgetshrstr(ts) : (ts)->contents)
```

注意一个与老资料的硬差异:5.3/5.4 的 `TString` 结构里没有 `contents` 字段,短串内容跟在 `union u` 后面,长串内容跟在整个结构体末尾。5.5 引入 external string 后,长串内容可能在外部,所以加了 `contents` 指针来统一指向;短串的存放位置也相应调整为跟在 `contents` 字段之后(用 `offsetof(TString, contents)` 计算大小,见 `sizestrshr` 宏 `lstring.h:37`)。这一点,凡是照着 5.3/5.4 资料画 `TString` 内存布局的,都会和 5.5 对不上。

把短串对象的总大小算清楚:`sizestrshr(l)` 是 header 加上内容(含结尾 `\0`):

```c
#define sizestrshr(l)  \
  (offsetof(TString, contents) + ((l) + 1) * sizeof(char))
```

也就是从结构体开头到 `contents` 字段的字节数,加上 `(长度+1)` 字节内容。短串对象是**一次性分配**的:header 和内容在一块连续内存里。

### 2.2 全局驻留表:`stringtable`

所有驻留的短串,统一登记在 `global_State` 里的 `strt` 字段(`lstate.h:334`):

```c
stringtable strt;  /* hash table for strings */
```

`stringtable` 定义在 `lstate.h:167`:

```c
typedef struct stringtable {
  TString **hash;  /* array of buckets (linked lists of strings) */
  int nuse;  /* number of elements */
  int size;  /* number of buckets */
} stringtable;
```

这是一个**拉链法哈希表**。`hash` 是一个指针数组,每个桶是一个 `TString *` 链头;短串通过自己的 `u.hnext` 串进对应桶的链里。桶数 `size` 永远是 2 的幂(初始化 `MINSTRTABSIZE = 128`,`lstring.c:37`),所以取模用位与(`lmod` 宏,`lobject.h:824`)。`nuse` 是当前驻留的短串总数,用于判断要不要扩容。

这个表挂在 `global_State` 上,而不是 `lua_State` 上,意味着:**所有协程(共享同一个 `global_State` 的线程)共用一张短串表**。两个协程里写 `"name"`,拿到的是同一个 `TString`。这是驻留的全局性所在,也是 GC 必须扫这张表的根因(见 2.5)。

### 2.3 短串驻留流程:`luaS_newlstr` / `internshrstr`

创建字符串的总入口是 `luaS_newlstr`(`lstring.c:249`):

```c
TString *luaS_newlstr (lua_State *L, const char *str, size_t l) {
  if (l <= LUAI_MAXSHORTLEN)  /* short string? */
    return internshrstr(L, str, l);
  else {
    TString *ts;
    if (l_unlikely(l * sizeof(char) >= (MAX_SIZE - sizeof(TString))))
      luaM_toobig(L);
    ts = luaS_createlngstrobj(L, l);
    memcpy(getlngstr(ts), str, l * sizeof(char));
    return ts;
  }
}
```

长度 `l` 和 `LUAI_MAXSHORTLEN` 比,短走 `internshrstr`(驻留),长走 `luaS_createlngstrobj`(新建,见 2.4)。`LUAI_MAXSHORTLEN` 在 `lstring.h:28` 定义为 **40**:

```c
#if !defined(LUAI_MAXSHORTLEN)
#define LUAI_MAXSHORTLEN  40
#endif
```

注释解释了这个值的下限来源:不能比保留字和元方法名短(`#("function") = 8, #("__newindex") = 10`),这些字符串**必须**能被驻留(VM 依赖它们全局唯一来快速匹配)。40 这个值是经验值,覆盖绝大多数会反复出现的标识符和 key 名。

驻留的核心是 `internshrstr`(`lstring.c:214`),逐行走:

```c
static TString *internshrstr (lua_State *L, const char *str, size_t l) {
  TString *ts;
  global_State *g = G(L);
  stringtable *tb = &g->strt;
  unsigned int h = luaS_hash(str, l, g->seed);
  TString **list = &tb->hash[lmod(h, tb->size)];
  lua_assert(str != NULL);
  for (ts = *list; ts != NULL; ts = ts->u.hnext) {
    if (l == cast_uint(ts->shrlen) &&
        (memcmp(str, getshrstr(ts), l * sizeof(char)) == 0)) {
      /* found! */
      if (isdead(g, ts))  /* dead (but not collected yet)? */
        changewhite(ts);  /* resurrect it */
      return ts;
    }
  }
  /* else must create a new string */
  if (tb->nuse >= tb->size) {  /* need to grow string table? */
    growstrtab(L, tb);
    list = &tb->hash[lmod(h, tb->size)];  /* rehash with new size */
  }
  ts = createstrobj(L, sizestrshr(l), LUA_VSHRSTR, h);
  ts->shrlen = cast(ls_byte, l);
  getshrstr(ts)[l] = '\0';  /* ending 0 */
  memcpy(getshrstr(ts), str, l * sizeof(char));
  ts->u.hnext = *list;
  *list = ts;
  tb->nuse++;
  return ts;
}
```

四步。

**第一步,算 hash**。`luaS_hash(str, l, g->seed)`(`lstring.c:53`):

```c
static unsigned luaS_hash (const char *str, size_t l, unsigned seed) {
  unsigned int h = seed ^ cast_uint(l);
  for (; l > 0; l--)
    h ^= ((h<<5) + (h>>2) + cast_byte(str[l - 1]));
  return h;
}
```

初值 `seed ^ l`(把长度混进 hash,这样不同长度的串即便内容前缀相同也不会撞到同一个起点);然后对每个字节做 `h ^= ((h<<5) + (h>>2) + byte)`。这是一个简单的增量 hash,`<<5` 和 `>>2` 的组合让高位低位都参与运算,抗碰撞性对短串足够。注意混进了 `seed`,这个 seed 的来历在 2.4 讲,是防 DoS 的关键。

**第二步,查桶**。桶号是 `lmod(h, tb->size)`,因为 `size` 是 2 的幂,`lmod` 退化成位与(`lobject.h:824`)。`list` 指向桶头。

**第三步,沿链找**。对链上每个 `ts`,先比长度(`l == ts->shrlen`),长度不等直接跳过(省掉 `memcmp`);长度相等才 `memcmp` 内容。这是经典优化:**长度不同必然不等,先比长度这一步过滤掉绝大多数**。命中就返回已有的 `ts`——这就是去重:相同内容的短串,拿到的是同一个指针。命中时还有个小细节:`isdead(g, ts)` 检查这个串是不是已经被 GC 标成"死但还没释放"(GC 是增量的,中间状态会存在),如果是,`changewhite(ts)` 把它"复活"——这个串现在又有人引用了,不能回收。这是驻留和增量 GC 协同的一个点。

**第四步,没命中就建**。先看要不要扩容:`tb->nuse >= tb->size` 就 `growstrtab`(见 2.6),扩容后桶号变了,要重算 `list`。然后 `createstrobj` 分配新对象(传 `LUA_VSHRSTR` tag 和算好的 hash `h`),写长度、拷内容、写结尾 `\0`,最后**头插**进桶链(`ts->u.hnext = *list; *list = ts;`)、`nuse++`。

`createstrobj`(`lstring.c:167`)本身很简单:调 `luaC_newobj` 拿到一个挂在 allgc 链上的 GCObject,转成 `TString *`,写 `hash` 和 `extra=0`。`luaC_newobj` 顺手把它链进 GC 的 allgc 表(它是可回收对象),并按 `tag` 设置 `tt`。

到这里,驻留的核心就讲完了:**相同内容的短串,全局只有一个 `TString`**。任何拿着 `"name"` 这个字面量或 `lua_tostring` 出来的短串的地方,`TString *` 都指向同一块内存。

### 2.4 长串:不驻留 + 惰性哈希

长串走的是另一条路。回到 `luaS_newlstr`,长度 > 40 时:

```c
ts = luaS_createlngstrobj(L, l);
memcpy(getlngstr(ts), str, l * sizeof(char));
return ts;
```

`luaS_createlngstrobj`(`lstring.c:179`):

```c
TString *luaS_createlngstrobj (lua_State *L, size_t l) {
  size_t totalsize = luaS_sizelngstr(l, LSTRREG);
  TString *ts = createstrobj(L, totalsize, LUA_VLNGSTR, G(L)->seed);
  ts->u.lnglen = l;
  ts->shrlen = LSTRREG;  /* signals that it is a regular long string */
  ts->contents = cast_charp(ts) + offsetof(TString, falloc);
  ts->contents[l] = '\0';  /* ending 0 */
  return ts;
}
```

几个要点。

**第一,不查表**。直接 `createstrobj` 建新对象,**不去 `strt` 里找有没有一样的**。所以两段内容完全相同的长串,会各自占一块内存,是两个不同的 `TString`。

**第二,hash 字段先放 seed**。`createstrobj` 的第四个参数是初始 hash 值,这里传的是 `G(L)->seed`。也就是说,长串创建时**不算内容 hash**,只把随机 seed 搁在 `hash` 字段里。真正的内容 hash 等到需要时才算——这就是"惰性哈希"。算法在 `luaS_hashlongstr`(`lstring.c:61`):

```c
unsigned luaS_hashlongstr (TString *ts) {
  lua_assert(ts->tt == LUA_VLNGSTR);
  if (ts->extra == 0) {  /* no hash? */
    size_t len = ts->u.lnglen;
    ts->hash = luaS_hash(getlngstr(ts), len, ts->hash);
    ts->extra = 1;  /* now it has its hash */
  }
  return ts->hash;
}
```

`extra == 0` 表示还没算过;算一次,把 `extra` 置 1,之后再调直接返回缓存的 `hash`。注意这里 `luaS_hash` 的第三个参数是 `ts->hash`,也就是创建时放进去的随机 seed——这就把 seed 带进了长串的 hash 计算。什么时候会触发算长串 hash?**只有当长串被当成 table 的 key 时**。看 `ltable.c:202` 的 `mainpositionTV`:

```c
case LUA_VLNGSTR: {
  TString *ts = tsvalue(key);
  return hashpow2(t, luaS_hashlongstr(ts));
}
```

用长串查 table,才调 `luaS_hashlongstr`。一个长串如果从来不当 key(比如只是个被打印的字符串),它永远不算 hash——这就是惰性的收益:为绝大多数长串(只被读取、从不做 key)省掉一笔 O(n) 的扫描。

**第三,为什么 hash 要混 seed——防 DoS**。这是关键的安全考量。如果 hash 函数是固定的(初值恒为 0 或常数),攻击者可以构造大量 hash 值相同的长字符串作为 HTTP 请求参数、POST body,塞进一个 Lua table。所有同 hash 的 key 会挤在同一个桶的链上,table 查找退化成 O(n) 链表扫描,n 个 key 的表查一次是 O(n),插 n 个就是 O(n²)——几 MB 的恶意请求就能把服务器 CPU 耗尽。这是 2011-2012 年横扫多种语言(包括 Python、Ruby、PHP)的"hash flooding"攻击。

Lua 的对策:每次 `lua_newstate` 生成一个**随机 seed**,所有短串和长串的 hash 都混进这个 seed(`internshrstr` 里 `luaS_hash(str, l, g->seed)`,长串算 hash 时初值也是 seed)。攻击者无法预测 seed,就无法构造出一组必然碰撞的 key。这个 seed 存在 `global_State.seed`(`lstate.h:337`),是 5.5 公开 API 的一部分:

```c
unsigned int seed;  /* randomized seed for hashes */
```

`lua_newstate` 的签名 5.5 里多了一个 `seed` 参数(`lua.h:163`):

```c
LUA_API lua_State *(lua_newstate) (lua_Alloc f, void *ud, unsigned seed);
```

这是 5.5 相对老资料的一个 API 变化(老版本 seed 是内部生成的,不在 `lua_newstate` 签名里)。宿主可以自己传一个真随机的 seed 进来;标准的 `luaL_newstate` 会调 `luaL_makeseed` 帮你生成一个(`lauxlib.c:1185`):

```c
LUALIB_API lua_State *(luaL_newstate) (void) {
  lua_State *L = lua_newstate(luaL_alloc, NULL, luaL_makeseed(NULL));
  ...
```

`luaL_makeseed`(`lauxlib.c:1155`)的熵源是两个:本地栈变量的地址(受 ASLR 影响,每次进程启动地址不同)和时间戳 `time(NULL)`。两者异或揉进一个 `unsigned int`。这不是密码学级别的随机,但对防 hash flooding 足够——攻击者远程猜不到进程内存布局。

### 2.5 字符串与 GC:扫描、缩容、移除

驻留的短串挂在 `strt` 上,GC 必须照顾到它们。这一节看三件事:扫描、回收时的移除、表的缩容。

**扫描短串表**。增量三色 GC 在"原子阶段"(atomic phase)会遍历整个 `strt`,把还活着的短串标黑、把没人引用的标死。这部分细节属于 P5-16,这里只点出关键:GC 不是单独管理短串的,短串作为 `LUA_TSTRING` 类型的 GC 对象,和 table、closure 一样进 allgc 链;但因为它额外挂在 `strt` 上,GC 还要在原子阶段扫一遍 `strt` 来处理那些"在 strt 里但已经白色(死)"的串(注释见 `lgc.c` 里 `traverse` 相关函数)。短串不进 gray 链(它没有指向其他对象的字段),直接由扫描 `strt` 处理。

**回收时移除**。一个短串被判定为死、要释放时,走 `freeobj` 的 `LUA_VSHRSTR` 分支(`lgc.c:866`):

```c
case LUA_VSHRSTR: {
  TString *ts = gco2ts(o);
  luaS_remove(L, ts);  /* remove it from hash table */
  luaM_freemem(L, ts, sizestrshr(cast_uint(ts->shrlen)));
  break;
}
case LUA_VLNGSTR: {
  TString *ts = gco2ts(o);
  if (ts->shrlen == LSTRMEM)  /* must free external string? */
    (*ts->falloc)(ts->ud, ts->contents, ts->u.lnglen + 1, 0);
  luaM_freemem(L, ts, luaS_sizelngstr(ts->u.lnglen, ts->shrlen));
  break;
}
```

短串释放前,必须先 `luaS_remove` 把它从 `strt` 的链里摘掉(`lstring.c:190`):

```c
void luaS_remove (lua_State *L, TString *ts) {
  stringtable *tb = &G(L)->strt;
  TString **p = &tb->hash[lmod(ts->hash, tb->size)];
  while (*p != ts)  /* find previous element */
    p = &(*p)->u.hnext;
  *p = (*p)->u.hnext;  /* remove element from its list */
  tb->nuse--;
}
```

因为表是拉链法,释放一个节点要改它前驱的 `hnext`,所以得先沿链找到前驱(`*p != ts` 时一直走),再把前驱的 `hnext` 跳过自己。`nuse--` 维护计数。这是个 O(链长) 操作,但只在 GC 真正回收时发生,不在热路径。

长串释放更简单:它不在 `strt` 里,不用 remove;但 `LSTRMEM` 这种外部串要回调用户的 `falloc` 让外部内存被释放(`lgc.c:874`),这是 5.5 external string 的回收契约。

**表的缩容**。短串大量被 GC 回收后,`strt` 可能空了一大半。GC 在每个周期的原子阶段结尾调 `checkSizes`(`lgc.c:935`):

```c
static void checkSizes (lua_State *L, global_State *g) {
  if (!g->gcemergency) {
    if (g->strt.nuse < g->strt.size / 4)  /* string table too big? */
      luaS_resize(L, g->strt.size / 2);
  }
}
```

`nuse` 跌到 `size` 的四分之一以下,就把表**缩一半**。这是怕驻留表长期占着大内存不还。注意只在非紧急 GC(`gcemergency` 为假)时缩,紧急 GC(内存不足触发的全量回收)不折腾表,先保证能分配出内存再说。

### 2.6 扩容:`luaS_resize` 与 `tablerehash`

`internshrstr` 里 `nuse >= size` 时调 `growstrtab`(`lstring.c:200`):

```c
static void growstrtab (lua_State *L, stringtable *tb) {
  if (l_unlikely(tb->nuse == INT_MAX)) {  /* too many strings? */
    luaC_fullgc(L, 1);  /* try to free some... */
    if (tb->nuse == INT_MAX)  /* still too many? */
      luaM_error(L);  /* cannot even create a message... */
  }
  if (tb->size <= MAXSTRTB / 2)  /* can grow string table? */
    luaS_resize(L, tb->size * 2);
}
```

扩容就是 `size * 2`。先有个极限保护:`nuse == INT_MAX` 时先做一次全量 GC 看能不能清出点,还满就直接报错(连错误信息都建不出来,因为错误信息本身也是个字符串)。`MAXSTRTB` 是 `INT_MAX` 能装下的 `TString*` 数上限(`lstring.c:28`)。

真正的搬运在 `luaS_resize`(`lstring.c:95`):

```c
void luaS_resize (lua_State *L, int nsize) {
  stringtable *tb = &G(L)->strt;
  int osize = tb->size;
  TString **newvect;
  if (nsize < osize)  /* shrinking table? */
    tablerehash(tb->hash, osize, nsize);  /* depopulate shrinking part */
  newvect = luaM_reallocvector(L, tb->hash, osize, nsize, TString*);
  if (l_unlikely(newvect == NULL)) {  /* reallocation failed? */
    if (nsize < osize)  /* was it shrinking table? */
      tablerehash(tb->hash, nsize, osize);  /* restore to original size */
    /* leave table as it was */
  }
  else {  /* allocation succeeded */
    tb->hash = newvect;
    tb->size = nsize;
    if (nsize > osize)
      tablerehash(newvect, osize, nsize);  /* rehash for new size */
  }
}
```

注意缩容和扩容的顺序不同。**缩容**先 `tablerehash` 把所有串重新分布到更少的桶里(把高位的桶清空,把里面的串搬到保留区),再 `realloc` 把数组缩短——这样 realloc 不会丢数据。**扩容**反过来:先 `realloc` 把数组拉长,再 `tablerehash` 把旧桶里的串散到新桶里。`realloc` 失败时(`newvect == NULL`),如果是缩容,要把刚才的 rehash 撤回(还原成原 size 的分布),保持表能用——注释明说"any non-zero size should work correctly",即任何非零大小都能正确工作,只是性能差。

`tablerehash`(`lstring.c:72`)做的事:清空新桶、遍历旧桶、每个串用 `lmod(p->hash, nsize)` 算新桶号、头插进新桶。整个过程不重算 hash(hash 值在串创建时就定死了,和表大小无关,只靠 `lmod` 取模),所以 rehash 是 O(n) 一次遍历。

### 2.7 一个隐藏的加速:`luaS_new` 的 API 缓存

除了 `strt` 这张驻留表,Lua 还给 C API 入口 `lua_pushstring` 等加了一层小缓存,在 `luaS_new`(`lstring.c:269`):

```c
TString *luaS_new (lua_State *L, const char *str) {
  unsigned int i = point2uint(str) % STRCACHE_N;  /* hash */
  int j;
  TString **p = G(L)->strcache[i];
  for (j = 0; j < STRCACHE_M; j++) {
    if (strcmp(str, getstr(p[j])) == 0)  /* hit? */
      return p[j];  /* that is it */
  }
  /* normal route */
  for (j = STRCACHE_M - 1; j > 0; j--)
    p[j] = p[j - 1];  /* move out last element */
  /* new element is first in the list */
  p[0] = luaS_newlstr(L, str, strlen(str));
  return p[0];
}
```

这是针对"宿主反复用同一个 C 字符串字面量调 API"的场景(比如 Redis 里反复 `lua_pushstring(L, "GET")`)。它用**这个 C 字符串的内存地址**(`point2uint(str)`)做 hash,而不是内容——因为同一个字面量在编译产物里地址是固定的。缓存是 `STRCACHE_N=53` 个组、每组 `STRCACHE_M=2` 个槽的 LRU。命中就直接返回缓存的 `TString`,连 `luaS_hash` 和 `strt` 查找都省了。没命中才走 `luaS_newlstr` 进驻留表。这层缓存(`global_State.strcache`,`lstate.h:368`)在 GC 时由 `luaS_clearcache`(`lstring.c:120`)清理:凡是缓存里的串已经变白(将被回收),就用那个永不回收的 `memerrmsg` 顶上。

这是"统一与精简"的一个小注脚:驻留表 `strt` 解决"Lua 内部所有短串的去重";API 缓存 `strcache` 解决"宿主反复传同一 C 字符串进来的去重"。两层各管一段,接口统一在 `TString *`。

## 三、为什么这样设计是 sound 的

讲完实现,回头问几个"为什么"。这一节是命脉。

### 3.1 为什么短串驻留、长串不驻留

驻留的本质是把"判定相等"从 O(n) 的内容比较,提前换算成 O(1) 的指针比较。这笔换算的**成本**是:每个串创建时要去全局表查一次(O(链长)),并且全局表本身占内存(每个桶一个指针 + 链节点复用 `hnext` 不额外占)。

- 对**短串**:它会被反复当 key 用(字段名、变量名、保留字),相等判定的频次极高;短串内容小,重复概率高(`"name"`、`"id"`、`"__index"` 这些会被用无数次),去重的内存收益大;全局表里挂几万个短串也才几十 KB。**收益远大于成本**,驻留。
- 对**长串**:它很少被当 key(谁会把一段 10KB 的文本当 table key?);内容大,重复概率极低(两段一样的长文本几乎不出现),去重几乎无收益;如果把长串也挂进 `strt`,表会被几 KB 一个的串撑到几百 MB,而查表时为了去重还得对每段长文本算 hash + memcmp——**成本远大于收益**,不驻留。

40 这个分界线是经验值,但它对齐了一个事实:**绝大多数实际的字符串 key 都很短**。Lua 标准库的元方法名(`__index`/`__newindex`/`__add`...)、保留字(`function`/`local`/`return`...)、常见字段名,全在 10 个字节以内。把分界线放在 40,既覆盖了所有必须驻留的(注释里点名 `__newindex` 是 10),又留了余量给稍长的标识符。

这个分工的 sound 体现在:**它没有为了"统一"而强求所有字符串走同一条路**。`TString` 是统一的表示,但创建策略(intern vs new)和 hash 策略(即时 vs 惰性)按使用特征分治。这正是"精简"——用最少的两种策略,覆盖了两个极端的特征区间,中间不存在第三种需要特殊处理的情况。

### 3.2 为什么短串比较是 sound 的

驻留保证了"内容相同 ⇔ 指针相同"。但要 sound,必须保证**没有两个内容相同的不同短串对象**漏过驻留。

`internshrstr` 是唯一的短串创建路径(`luaS_newlstr` 对短串只调它)。它每次都查 `strt`,命中就返回已有对象。所以只要 `strt` 这张表始终包含所有活着的短串,这个双射就成立。什么情况下会破?

- **编译期产生的字符串常量**:加载 chunk 时由 `luaO_pushfstring` 等走 `luaS_new`,进 `luaS_newlstr`,进驻留表。
- **运行期 `tostring`、字符串拼接**:`luaV_concat`(`lvm.c:684`)拼出来的串,如果结果 ≤ 40 走 `luaS_newlstr` 驻留,> 40 才走长串。看 `lvm.c:711`:

```c
if (tl <= LUAI_MAXSHORTLEN) {  /* is result a short string? */
  char buff[LUAI_MAXSHORTLEN];
  copy2buff(top, n, buff);
  ts = luaS_newlstr(L, buff, tl);
}
else {  /* long string; copy strings directly to final result */
  ts = luaS_createlngstrobj(L, tl);
  ...
```

注意 concat 这里有个讲究:结果长度 ≤ 40 时用栈上一个 `char buff[40]` 临时拼,然后 `luaS_newlstr`;> 40 时直接 `luaS_createlngstrobj` 建长串、把内容拷进去。也就是说,**concat 的产物严格按 40 分界走驻留或长串**,和 `luaS_newlstr` 的分界一致,没有第三条路。这保证了"任何内容 ≤ 40 的字符串都必然在 strt 里"。

- **C API**:`lua_pushstring`/`lua_pushlstring` 最终都走 `luaS_newlstr`(经 `luaS_new` 缓存或直调)。没有绕过驻留的旁路。

所以"短串内容相同 ⇔ 指针相同"这条不变式,在 Lua 内核里是**全局成立**的。`eqshrstr(a, b)` 宏(`lstring.h:54`)就敢直接写成 `a == b`:

```c
#define eqshrstr(a,b)  check_exp((a)->tt == LUA_VSHRSTR, (a) == (b))
```

`check_exp` 在 release 编译是空断言,所以这宏展开就是 `(a) == (b)`——一次指针比较。`ltable.c` 的注释(`ltable.c:237`)把这层依赖写明了:"It is assumed that 'eqshrstr' is simply pointer equality, so that short strings are handled in the default case."。整个 VM 对短串相等的信任,建立在驻留这条全局不变式上。

### 3.3 为什么长串 hash 惰性是 sound 的

惰性要求:**没算 hash 的长串,在被需要 hash 的场景出现之前,不能被用到 hash**。长串什么时候需要 hash?只有当它当 table key 时(`mainpositionTV` 的 `LUA_VLNGSTR` 分支调 `luaS_hashlongstr`)。其他场景(读内容、`#`取长度、拼接)都不碰 hash。

- 读内容走 `getlngstr` → `ts->contents`,不碰 hash。
- 取长度 `#s` 走 `luaV_objlen` → `tsslen` → `ts->u.lnglen`,不碰 hash。
- 拼接 `..` 走 `luaV_concat` → `tsslen` 取各段长度 + `getlngstr` 取各段内容,不碰 hash。

所以一个长串如果从不被当 key,它的 hash 永远是 seed(未计算状态),`extra` 永远是 0,完全不影响正确性。一旦要当 key,`luaS_hashlongstr` 算一次、缓存进 `hash`、`extra` 置 1,之后都用缓存值。sound。

`extra` 这个 `lu_byte` 同时承担短串的"是否保留字"和长串的"是否已算 hash"两个语义,是复用,不冲突——因为短串和长串是 variant 互斥的两种对象,同一个 `TString` 不可能同时是两者。

### 3.4 seed 防 DoS 为什么 sound

sound 的要求:**攻击者无法预先构造出一组在受害进程里必然 hash 碰撞的字符串**。

如果 seed 是固定的(比如硬编码 0),`luaS_hash` 就是确定函数,攻击者可以本地跑同样的算法,构造出一组内容 hash 全相同的字符串,远程投递。5.5 的 seed 来自 `luaL_makeseed`:进程地址空间随机(ASLR)+ 启动时间,这两个攻击者远程都不可知。所以同一个字符串,在受害进程 A 和进程 B 里 hash 值不同,攻击者无法批量构造碰撞组。

更严格地说,seed 把 hash flooding 从"确定性漏洞"降级为"概率性事故"——某个特定串碰不碰撞取决于 seed,而 seed 每次启动都变。攻击成本从"构造一次,处处可用"变成"对每个目标都要重试,且重试也猜不准"。对脚本嵌入场景(Redis、nginx)足够。

为什么 Lua 不直接用更强的 SipHash 之类?一是 Lua 内核追求小,`luaS_hash` 那几行(`<<5`/`>>2`/异或)编译出来就十几个时钟周期,SipHash 大得多;二是 seed 已经把"远程可预测"这个根因堵住了,对 Lua 的威胁模型(嵌入宿主,攻击面是脚本输入)够用。这是典型的"精简换够用"——不追求密码学强度,只追求堵住已知攻击向量。

## 四、★对照 CPython + 回扣主线

### 4.1 两种 intern 策略

CPython 也有字符串驻留(intern),但机制和粒度都不同。对照如下。

| 维度 | Lua 5.5 | CPython |
|---|---|---|
| **字符串表示** | 统一 `TString`,variant 区分短/长(`LUA_VSHRSTR`/`LUA_VLNGSTR`) | `PyUnicodeObject`,内部按字符宽度分 1/2/4 字节三种表示,加 compact 形式 |
| **驻留判据** | 长度 ≤ 40(`LUAI_MAXSHORTLEN`)一刀切 | 多重规则:看起来像标识符的(只含字母数字下划线)、长度短的、编译期常量、被 dict key 用过的...各版本规则在变 |
| **驻留表** | 全局 `stringtable strt`,拉链法,所有协程共享 | 进程级 `interned` dict |
| **生命周期** | GC 管理。短串无引用时被增量标记清除回收,从 strt 摘除 | **引用计数**为主。intern 的字符串 PyUnicode 对象有 `ob_refcnt`,引用归零才出 interned 表;环引用走分代标记 |
| **长串** | 不驻留,内容跟在对象内存后(或外部),hash 惰性 | 是否 intern 看规则(不是单纯看长度);长串通常不 intern,但内部表示仍是 `PyUnicodeObject` |
| **hash 防碰撞** | 每进程随机 seed,`luaS_hash` 混 seed | Python 3 也有 `PYTHONHASHSEED`,SipHash(对 bytes)或 fnv(老版本),默认随机 |
| **外部内容** | 5.5 有 external string(`LSTRFIX`/`LSTRMEM`),内容可在外部内存 | 无对应。所有 `PyUnicodeObject` 内容都归对象管 |

最本质的差异在**生命周期管理**。Lua 的短串由增量三色 GC 统一管,和 table、closure 一样进 allgc 链,GC 扫 `strt` 时统一判定生死——这对"嵌入宿主不能长时间卡顿"是友好的(GC 可中断,见 P0-01 第三招)。CPython 的 intern 字符串靠引用计数,对象一不可达就立刻回收(出 interned 表),及时性好,但每次指针赋值都有 `Py_INCREF`/`Py_DECREF` 的开销,且环引用(虽然 intern 的字符串通常不成环,但 PyUnicode 作为 key/value 参与的容器可能成环)还得靠周期性的分代标记——那是一次完整 STW。这是"引用计数 vs 增量标记"这一对张力在字符串上的具体投影。

第二个差异是**驻留判据的复杂度**。Lua 用一个硬阈值(40)一刀切,简单、可预测、源码就一个 `if`。CPython 的 intern 规则是启发式的(`PyUnicode_InternInPlace` 及其调用点里多处判断"这是不是个 identifier-like string"),不同 Python 版本规则不同,复杂度高。这正对应了主线的对照三刀之一:**数据模型上,Lua 用精简换简单**——一个数字 40 解决"谁该驻留",CPython 用一套规则追求"把更可能重复的串挑出来驻留",各有所得,但 Lua 的代码量明显更小。

第三个差异是 **external string**。5.5 的 `LSTRFIX`/`LSTRMEM` 让宿主可以把一块**不归 Lua 管的内存**包装成 Lua 字符串,零拷贝。这在嵌入场景很有用(比如 Redis 把网络收到的字节直接当 Lua 字符串喂进去,不用先 memcpy 到 Lua 自己的 TString 内存)。CPython 没有对应能力,所有 Python 字符串内容都归对象所有——这又一次体现 Lua 为"嵌入"做得更深。

### 4.2 回扣主线

把这一章放进全书主线"统一与精简换小而快"里看:

- **统一**:所有字符串——无论是 3 个字节的 `"age"`、还是 3 万字节的源码、还是宿主外部内存里的一段——都是一个 `TString`。读内容用同一个 `getstr`,取长度用同一个 `tsslen`,当 key 时用同一套 `mainpositionTV`。对编译器、VM、GC 来说,字符串是**一种**对象,不是两种(短/长)或四种(内部/外部/驻留/非驻留)。variant 和 kind 只是内部策略,不破坏接口统一。
- **精简**:这一种表示内部,只分**两种**策略(短串驻留 / 长串惰性),分界线是一个数字 40。没有"中等长度串"的第三种特例,没有"特殊符号串"的第四种规则。驻留表 `strt` 是拉链法哈希表,一份数据结构服务所有短串;API 缓存 `strcache` 是 53×2 的 LRU,一份数据结构服务所有 C API 入口。每个机制的代码量都很小(`internshrstr` 30 行,`luaS_hash` 5 行),但各自把一类问题堵死。
- **小**:短串去重,`"name"` 在整个 VM 里只存一份,内存省;驻留表本身是稀疏指针数组,开销小。
- **快**:`a.name` 这种 table 字段访问,因为短串驻留,key 比较退化成指针比较(O(1))。这是 Lua 字段访问接近 C struct 访问速度的根因之一。长串不参与驻留,免得驻留表被撑大、拖慢所有短串的查表——这是用"不快"(长串逐字节比较)保"快"(短串指针比较)的整体最优。

字符串这章是"统一接口 + 精简分工"这一主线思想最干净的样本:它不像 Table(P1-04/05)那样把"统一"做到极致(一个结构当数组/哈希/对象),而是把"统一表示 + 分治策略"做得最清晰。`TString` 是统一的壳,短/长是壳内精简的两套策略。读到这里,应该建立起一个直觉:**Lua 的每一个"快"和"省",都不是靠堆机制,而是靠把机制切到正确的粒度,让一种表示覆盖尽量多的情况、让少数几种策略覆盖特征的极端**。这个直觉在后面的 Table、寄存器式字节码、增量 GC 里会反复出现。
