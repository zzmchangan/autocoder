# P2-07 语法分析(上):递归下降与算符优先

> **本书主线**:统一与精简换小而快。**二分法**:编译侧(源码→字节码)↔ 执行侧。**★对照**:CPython。**源码**:lua-5.5.0。**基调**:纯直球,不用比喻。
>
> **本章落点**:编译侧 P2 第二环——词法吐出的 token 流,怎么被递归下降一步步吃掉、并同时吐出寄存器式字节码。Lua 编译器**不建 AST**,解析一遍就直接生成 `Proto->code`。这是"精简"在编译期的最高体现:砍掉一整个中间表示层。

---

## 一、这章解决什么问题:token 流怎么变字节码,为什么没有"语法树"

上一章(P2-06)讲完词法分析:`llex` 把源码切成一串 token(`TK_NAME`、`TK_INT`、`'+'`、`'='`……),装在 `LexState` 里供消费者按需读取。本章回答下一个问题:**这串 token,怎么变成 VM 能跑的寄存器式字节码?**

很多脚本语言的回答是分两步走:先把 token 组织成一棵**抽象语法树(AST)**,再遍历 AST 生成字节码。CPython 走的就是这条路:`parser.c` 先建出一棵 CST/AST,再由 `compile.c` 遍历 AST 发射字节码。两步之间有一个完整的、和字节码并列的中间产物——AST 节点。

Lua 砍掉了这一步。它的语法分析是**单遍(single pass)**的:递归下降解析器一边读 token、一边通过 `lcode` 的接口直接往当前函数的 `Proto->code` 数组里塞 `Instruction`。**整个编译过程里,不存在任何 AST 节点对象。** 解析器手里只有一个轻量的"表达式描述符"`expdesc`,它描述"刚刚读到的这个表达式,目前该到哪个寄存器、是不是常量、能不能延迟求值",一旦某个表达式完整,就立刻把对应的字节码发射出去。

这个选择直接落在主线"精简换小而快"上:

- **小**:省掉了 AST 节点这个中间层的全部内存分配。解析一个函数,从开始到结束,手里的中间状态只有 `FuncState`(一个结构体)和一个全局的 `Dyndata`(动态数组),没有任何按 token 数量线性增长的树。
- **快**:少一遍遍历。两遍式要先建树、再走树;单遍式读 token 的同时就把字节码写好了,编译延迟更低。Lua 经常被宿主在运行时 `loadstring` 编译小段脚本,编译速度本身是体感指标。
- **适合嵌入**:编译期的峰值内存占用低。嵌入式设备、游戏运行时编译一段脚本,不必为 AST 预留大块堆内存。

代价是解析器和代码生成器**耦合得很紧**:理解表达式解析,就必须同时理解 `lcode` 的 `luaK_infix`/`luaK_prefix`/`luaK_posfix` 三段式——它们就是嵌在递归下降里的代码生成钩子。本章会把这条链路从 `luaY_parser` 一路走到 `expr`/`subexpr`,再用一个具体例子 `local a = b + c * 2` 把"读 token → 发指令"的全过程走一遍。

(关于作用域、local 变量的栈位分配、upvalue 的静态收集,是下一章 P2-08 的主题;本章遇到时一句话指向,不展开。)

## 二、源码怎么实现

### 2.1 编译的总产物:`Proto`

先看编译到底产出什么。一个 Lua 函数被编译后,产物是一个 `Proto` 结构(lobject.h:602):

```c
typedef struct Proto {
  CommonHeader;
  lu_byte numparams;  /* number of fixed (named) parameters */
  lu_byte flag;
  lu_byte maxstacksize;  /* number of registers needed by this function */
  int sizeupvalues;  /* size of 'upvalues' */
  int sizek;  /* size of 'k' */
  int sizecode;
  int sizelineinfo;
  int sizep;  /* size of 'p' */
  int sizelocvars;
  int sizeabslineinfo;  /* size of 'abslineinfo' */
  int linedefined;  /* debug information  */
  int lastlinedefined;  /* debug information  */
  TValue *k;  /* constants used by the function */
  Instruction *code;  /* opcodes */
  struct Proto **p;  /* functions defined inside the function */
  Upvaldesc *upvalues;  /* upvalue information */
  ls_byte *lineinfo;  /* information about source lines (debug information) */
  AbsLineInfo *abslineinfo;  /* idem */
  LocVar *locvars;  /* information about local variables (debug information) */
  TString  *source;  /* used for debug information */
  GCObject *gclist;
} Proto;
```

这是 VM 执行侧(P3)的全部输入。本章最关心的几个字段:

- `code`:`Instruction *`,字节码指令数组。一条 Lua 指令是 32 位的 `Instruction`(编码见 P3-10)。整个 P2 编译侧的所有工作,最终就是往这个数组里写东西。
- `k`:`TValue *`,常量表。源码里的字面量(`1`、`3.14`、`"hello"`),以及编译期需要落表的大常量,都进这里。指令通过 `K[C]` 这种操作数引用它们。
- `p`:`struct Proto **`,嵌套函数。Lua 里每个 `function ... end` 都是一个嵌套的 `Proto`,挂在父 `Proto->p` 上,形成一个树状结构(注意:这是**函数之间的嵌套树**,不是表达式的 AST)。
- `upvalues`:`Upvaldesc *`,本函数捕获的外层局部变量。怎么收集的见 P2-08。
- `numparams` / `is_vararg`(`flag` 的 `PF_VAHID` 位):参数个数、是否变参。
- `maxstacksize`:这个函数最多需要多少个寄存器。VM 运行前据此分配值栈槽位。
- `lineinfo` / `locvars`:调试信息(每条指令对应的源码行、局部变量名范围)。出错信息和 debug 库靠它。

记住一点:`code`、`k`、`p` 三者是并列的数组,不是树形嵌套。单遍编译的全部"产物"就是这三个数组被填满。

### 2.2 编译一个函数的瞬时状态:`FuncState`

解析器在编译一个函数期间,手里的全部状态装在 `FuncState`(lparser.h:166)里:

```c
typedef struct FuncState {
  Proto *f;  /* current function header */
  struct FuncState *prev;  /* enclosing function */
  struct LexState *ls;  /* lexical state */
  struct BlockCnt *bl;  /* chain of current blocks */
  Table *kcache;  /* cache for reusing constants */
  int pc;  /* next position to code (equivalent to 'ncode') */
  int lasttarget;   /* 'label' of last 'jump label' */
  int previousline;  /* last line that was saved in 'lineinfo' */
  int nk;  /* number of elements in 'k' */
  int np;  /* number of elements in 'p' */
  int nabslineinfo;  /* number of elements in 'abslineinfo' */
  int firstlocal;  /* index of first local var (in Dyndata array) */
  int firstlabel;  /* index of first label (in 'dyd->label->arr') */
  short ndebugvars;  /* number of elements in 'f->locvars' */
  short nactvar;  /* number of active variable declarations */
  lu_byte nups;  /* number of upvalues */
  lu_byte freereg;  /* first free register */
  lu_byte iwthabs;  /* instructions issued since last absolute line info */
  lu_byte needclose;  /* function needs to close upvalues when returning */
} FuncState;
```

逐字段看它的职责:

- `f`:当前正在编译的函数的 `Proto`。所有 `code`/`k`/`p` 的写入都通过它。
- `prev`:外层函数的 `FuncState`。嵌套函数定义时(`function ... end`)会建一个新的 `FuncState`,把外层的链进来。upvalue 沿着这条链往外找外层局部变量(P2-08)。
- `ls`:词法状态。解析器从这里读 token。
- `bl`:`BlockCnt` 链。Lua 的作用域是块结构(`do...end`、`while...end`、函数体都是块),每个块在进入时压一个 `BlockCnt`、退出时弹。`bl` 记录当前最内层块。块链的细节在 P2-08,这里只记它存在。
- `kcache`:一个 `Table`,用来去重常量。同一个字面量 `"abc"` 在函数里出现两次,只往 `k` 里存一份——查 `kcache` 命中就复用,不命中才插入。
- `pc`:下一条指令要写到 `code` 的哪个下标。等价于"已发射的指令数"。
- `nk` / `np` / `nabslineinfo` / `ndebugvars` / `nups`:各自数组(`k`/`p`/`abslineinfo`/`locvars`/`upvalues`)的已用长度。
- `firstlocal` / `firstlabel`:本函数在全局 `Dyndata` 里的起始下标。注意:所有局部变量和标签不是存在 `FuncState` 里,而是存在一个全局的 `Dyndata`(`ls->dyd`)的连续数组里。每个 `FuncState` 用 `firstlocal` 圈出自己那段。这样设计是为了 goto/label 跨块解析时能在同一个数组里线性操作。
- `nactvar`:当前**活跃**的局部变量数。进入一个 `local x, y = ...`,这两个变量就进 `nactvar`;块结束时减回去。它是寄存器分配的基本依据(下一段)。
- `freereg`:下一个空闲寄存器号。表达式求值的中间结果、临时变量都从这里分配。
- `needclose`:本函数返回时是否需要关闭 upvalue(把开放 upvalue 闭合成 closed)。P2-08 详讲。

`FuncState` 是编译器的"工作内存"。它极其紧凑——没有 AST 节点、没有表达式树,只有计数器和几个指针。

### 2.3 入口:`luaY_parser`

整个编译的入口是 `luaY_parser`(lparser.c:2168):

```c
LClosure *luaY_parser (lua_State *L, ZIO *z, Mbuffer *buff,
                       Dyndata *dyd, const char *name, int firstchar) {
  LexState lexstate;
  FuncState funcstate;
  LClosure *cl = luaF_newLclosure(L, 1);  /* create main closure */
  setclLvalue2s(L, L->top.p, cl);  /* anchor it (to avoid being collected) */
  luaD_inctop(L);
  lexstate.h = luaH_new(L);  /* create table for scanner */
  sethvalue2s(L, L->top.p, lexstate.h);  /* anchor it */
  luaD_inctop(L);
  funcstate.f = cl->p = luaF_newproto(L);
  luaC_objbarrier(L, cl, cl->p);
  funcstate.f->source = luaS_new(L, name);  /* create and anchor TString */
  luaC_objbarrier(L, funcstate.f, funcstate.f->source);
  lexstate.buff = buff;
  lexstate.dyd = dyd;
  dyd->actvar.n = dyd->gt.n = dyd->label.n = 0;
  luaX_setinput(L, &lexstate, z, funcstate.f->source, firstchar);
  mainfunc(&lexstate, &funcstate);
  lua_assert(!funcstate.prev && funcstate.nups == 1 && !lexstate.fs);
  /* all scopes should be correctly finished */
  lua_assert(dyd->actvar.n == 0 && dyd->gt.n == 0 && dyd->label.n == 0);
  L->top.p--;  /* remove scanner's table */
  return cl;  /* closure is on the stack, too */
}
```

几个要点:

1. **返回类型是 `LClosure *`,不是 `Proto *`**。这是 5.5 的实况,很多老资料(讲 5.3/5.4)会说返回 `Proto*` 或 `LClosure*` 含糊不清。实际 5.5 创建了一个 `LClosure`(Lua 闭包),它的 `p` 指向编译产物主 `Proto`,返回的就是这个闭包。这样上层(`luaD_protectedparser`→`luaL_loadbufferx`)可以直接拿一个可调用的闭包,不必再包一层。
2. 创建主闭包、扫描器用的哈希表(`lexstate.h`,词法分析里字符串驻留用)、源码名字符串,都被**锚定到值栈**(`setclLvalue2s`/`sethvalue2s` + `luaD_inctop`)。因为这些是 GC 对象,编译期间 GC 可能被触发(每写一条指令后 `luaC_checkGC`),必须保证它们不被回收。
3. `funcstate.f = cl->p = luaF_newproto(L)`:主 `Proto` 创建出来,挂在闭包上,同时作为 `funcstate.f`——接下来的 `mainfunc` 就往这个 `Proto` 里写。
4. `mainfunc(&lexstate, &funcstate)`:进入主函数体的解析。
5. 收尾的两个 `lua_assert`:`funcstate.nups == 1`——主函数必定有一个 upvalue,就是环境表 `_ENV`(下一节看怎么来的)。三个 `Dyndata` 计数归零,确认所有作用域、goto、label 都正确闭合。

### 2.4 主函数体:`mainfunc`

`mainfunc`(lparser.c:2150)编译"主函数"——即整个源文件作为一个隐式的 vararg 函数:

```c
static void mainfunc (LexState *ls, FuncState *fs) {
  BlockCnt bl;
  Upvaldesc *env;
  open_func(ls, fs, &bl);
  setvararg(fs);  /* main function is always vararg */
  env = allocupvalue(fs);  /* ...set environment upvalue */
  env->instack = 1;
  env->idx = 0;
  env->kind = VDKREG;
  env->name = ls->envn;
  luaC_objbarrier(ls->L, fs->f, env->name);
  luaX_next(ls);  /* read first token */
  statlist(ls);  /* parse main body */
  check(ls, TK_EOS);
  close_func(ls);
}
```

三件事:

1. `open_func`:初始化 `FuncState` 的所有字段,进入最外层 `BlockCnt`,给 `f->maxstacksize` 设初值 `2`(`open_func` 里 `f->maxstacksize = 2`,因为寄存器 0、1 永远有效——VM 约定,见 P3)。
2. `setvararg`:主函数永远是变参的。源文件级 `...` 拿到的是命令行参数(宿主传入)。这会发射一条 `OP_VARARGPREP` 指令。
3. **创建第一个 upvalue,名为 `_ENV`(代码里是 `ls->envn`,通常是字符串 `LUA_ENV` 即 `"_ENV"`)**。`instack=1, idx=0` 表示"它指向调用者栈上的第 0 个槽位"。这就是 Lua 里 `_ENV` 的物理来源:它不是一个关键字,而是一个**普通的第一个 upvalue**。所有"全局变量"访问 `x`,实际编译成 `_ENV.x`(`_ENV` 是 upvalue 0,`x` 是字符串常量)。这点后面 `singlevar` 还会再讲。
4. `luaX_next`:读第一个 token(此前 token 流还没启动)。
5. `statlist`:循环解析语句,直到遇到文件尾 `TK_EOS`。
6. `close_func`:收尾——发射最终 `return`、`luaK_finish` 把跳转目标全部 patch、把各数组 shrink 到精确大小、pop kcache 表、触发一次 GC 检查。

### 2.5 函数体的循环:`statlist` 与 `statement`

`statlist`(lparser.c:875)是个简单循环:

```c
static void statlist (LexState *ls) {
  /* statlist -> { stat [';'] } */
  while (!block_follow(ls, 1)) {
    if (ls->t.token == TK_RETURN) {
      statement(ls);
      return;  /* 'return' must be last statement */
    }
    statement(ls);
    ...
  }
}
```

读到块结束符(`TK_END`/`TK_ELSE`/`TK_EOS`/`TK_UNTIL`)就停。`return` 必须是块里最后一条语句——遇到 `TK_RETURN` 就解析完立刻 return,不再继续。

`statement`(lparser.c:2049)是**递归下降的总分发器**,按当前 token(lookahead)分派:

```c
static void statement (LexState *ls) {
  int line = ls->linenumber;  /* may be needed for error messages */
  enterlevel(ls);
  switch (ls->t.token) {
    case ';': {  /* stat -> ';' (empty statement) */
      luaX_next(ls);  /* skip ';' */
      break;
    }
    case TK_IF: {  /* stat -> ifstat */
      ifstat(ls, line);
      break;
    }
    case TK_WHILE: {  /* stat -> whilestat */
      whilestat(ls, line);
      break;
    }
    case TK_DO: {  /* stat -> DO block END */
      luaX_next(ls);  /* skip DO */
      block(ls);
      check_match(ls, TK_END, TK_DO, line);
      break;
    }
    case TK_FOR: {  /* stat -> forstat */
      forstat(ls, line);
      break;
    }
    case TK_REPEAT: {  /* stat -> repeatstat */
      repeatstat(ls, line);
      break;
    }
    case TK_FUNCTION: {  /* stat -> funcstat */
      funcstat(ls, line);
      break;
    }
    case TK_LOCAL: {  /* stat -> localstat */
      luaX_next(ls);  /* skip LOCAL */
      if (testnext(ls, TK_FUNCTION))  /* local function? */
        localfunc(ls);
      else
        localstat(ls);
      break;
    }
    case TK_GLOBAL: {  /* stat -> globalstatfunc */
      globalstatfunc(ls, line);
      break;
    }
    case TK_DBCOLON: {  /* stat -> label */
      luaX_next(ls);  /* skip double colon */
      labelstat(ls, str_checkname(ls), line);
      break;
    }
    case TK_RETURN: {  /* stat -> retstat */
      luaX_next(ls);  /* skip RETURN */
      retstat(ls);
      break;
    }
    case TK_BREAK: {  /* stat -> breakstat */
      breakstat(ls, line);
      break;
    }
    case TK_GOTO: {  /* stat -> 'goto' NAME */
      luaX_next(ls);  /* skip 'goto' */
      gotostat(ls, line);
      break;
    }
    ...
    default: {  /* stat -> func | assignment */
      exprstat(ls);
      break;
    }
  }
  lua_assert(ls->fs->f->maxstacksize >= ls->fs->freereg &&
             ls->fs->freereg >= luaY_nvarstack(ls->fs));
  ls->fs->freereg = luaY_nvarstack(ls->fs);  /* free registers */
  leavelevel(ls);
}
```

这是教科书式的**递归下降 + 前看一个 token 分发**。每个语法结构对应一个 `xxxxstat` 函数:`ifstat`、`whilestat`、`forstat`、`repeatstat`、`funcstat`(函数定义语句 `function a.b.c() ... end`)、`localstat`(`local x = ...`)、`localfunc`(`local function f() ... end`,和 `localstat` 不同:它要先声明 `f` 再编译函数体,这样函数体里能递归引用自己)、`retstat`、`breakstat`、`gotostat`、`labelstat`、`globalstatfunc`(5.5 新增的 `global`/`global <const>` 声明,见后)。

`default` 分支处理两种最普遍的语句:**函数调用语句**(`f()`)和**赋值**(`x = 1`、`a.b = 2`、`t[1], t[2] = y, z`)。两者都以一个表达式开头,`exprstat` 先解析这个表达式,再看后面是 `=`/`,` 还是直接结束来区分。

注意 `statement` 末尾:

```c
ls->fs->freereg = luaY_nvarstack(ls->fs);  /* free registers */
```

每条语句处理完,临时寄存器(`freereg` 以上分配给中间结果的那些)全部释放——下一条语句从 `nvarstack`(当前活跃 local 占用的寄存器数)重新开始分配。这是 Lua 编译器"寄存器即用即弃、线性分配"的关键不变式,寄存器分配的具体机制在 P2-09。

**5.5 新增:`TK_GLOBAL` 分支**。老版本(5.3/5.4)没有显式的全局变量声明,全局变量就是"赋值给 `_ENV` 的字段"。5.5 引入了 `global` 关键字(以及 `global <const>`、`global *`、`global function ...`)用来显式声明全局变量,并支持词法层面的全局/局部区分。`statement` 里因此多出 `case TK_GLOBAL`。当 `LUA_COMPAT_GLOBAL` 定义时,还有一段在 `TK_NAME` 分支里用 `ls->glbn` 字符串识别非保留字的 `global`,本章不深入这块兼容逻辑。

**语句层面就是这些**。本章对每个 `xxxxstat` 不展开,只确认它们的入口和分发位置。真正的解析密度集中在表达式——下一节。

### 2.6 表达式入口:`expr` 与递归子程序链

表达式解析的入口是 `expr`(lparser.c:1404):

```c
static void expr (LexState *ls, expdesc *v) {
  subexpr(ls, v, 0);
}
```

它把当前读到的完整表达式解析出来,描述符写进 `v`。注意它接受一个 `expdesc *v`:这是表达式解析全程在手里倒腾的"中间描述"。先看 `expdesc`(lparser.h:78):

```c
typedef struct expdesc {
  expkind k;
  union {
    lua_Integer ival;    /* for VKINT */
    lua_Number nval;  /* for VKFLT */
    TString *strval;  /* for VKSTR */
    int info;  /* for generic use */
    struct {  /* for indexed variables */
      short idx;  /* index (R or "long" K) */
      lu_byte t;  /* table (register or upvalue) */
      lu_byte ro;  /* true if variable is read-only */
      int keystr;  /* index in 'k' of string key, or -1 if not a string */
    } ind;
    struct {  /* for local variables */
      lu_byte ridx;  /* register holding the variable */
      short vidx;  /* index in 'actvar.arr' */
    } var;
  } u;
  int t;  /* patch list of 'exit when true' */
  int f;  /* patch list of 'exit when false' */
} expdesc;
```

`expdesc` 是单遍编译的**核心数据结构**。它不是 AST 节点——它只有一个,在表达式递归过程中不断被改写、覆盖,描述"当前这个子表达式求值后值在哪里、是什么形态"。关键字段:

- `k`:`expkind` 枚举,说明这个表达式现在是什么(常量 nil/true/false、整数常量 `VKINT`、浮点常量 `VKFLT`、字符串常量 `VKSTR`、常量表项 `VK`、已经在某寄存器 `VNONRELOC`、可以重定位到任意寄存器 `VRELOC`、局部变量 `VLOCAL`、upvalue `VUPVAL`、函数调用 `VCALL`、表索引 `VINDEXED`/`VINDEXI`/`VINDEXSTR`/`VINDEXUP`、跳转测试 `VJMP`、可变参数 `VVARARG` 等)。
- `u`:按 `k` 的不同,用不同的字段。整数用 `ival`,浮点用 `nval`,字符串用 `strval`,索引用 `ind.{t,idx,keystr,ro}`,局部变量用 `var.{ridx,vidx}`。
- `t` / `f`:跳转 patch 列表。`and`/`or` 短路求值、比较运算生成的条件跳转用它们(详见 P2-09)。

`expdesc` 是个**栈上局部变量**——`expr`/`subexpr` 的调用者声明一个 `expdesc v`,把地址传进来,函数内部改写它。没有堆分配,没有 AST 节点对象。这是单遍编译"省内存"最具体的一处。

#### 算符优先的核心:`priority` 表

二元运算符的优先级和结合性,在 lparser.c:1351 的一张静态表里:

```c
static const struct {
  lu_byte left;  /* left priority for each binary operator */
  lu_byte right; /* right priority */
} priority[] = {  /* ORDER OPR */
   {10, 10}, {10, 10},           /* '+' '-' */
   {11, 11}, {11, 11},           /* '*' '%' */
   {14, 13},                  /* '^' (right associative) */
   {11, 11}, {11, 11},           /* '/' '//' */
   {6, 6}, {4, 4}, {5, 5},   /* '&' '|' '~' */
   {7, 7}, {7, 7},           /* '<<' '>>' */
   {9, 8},                   /* '..' (right associative) */
   {3, 3}, {3, 3}, {3, 3},   /* ==, <, <= */
   {3, 3}, {3, 3}, {3, 3},   /* ~=, >, >= */
   {2, 2}, {1, 1}            /* and, or */
};

#define UNARY_PRIORITY	12  /* priority for unary operators */
```

这张表的下标是 `BinOpr` 枚举(`OPR_ADD`、`OPR_SUB`、……、`OPR_AND`、`OPR_OR`)的值,顺序和 `getbinopr`(下文)里的 `switch` 一一对应。每项两个数:`left` 是这个运算符作为左结合时的优先级,`right` 是作为右结合时的优先级。

注意几处**左右不相等**的项:

- `^`:`{14, 13}`。`right`(13)<`left`(14),意味着右结合。`a ^ b ^ c` 解析成 `a ^ (b ^ c)`。
- `..`:`{9, 8}`。同样右结合,`a .. b .. c` 解析成 `a .. (b .. c)`。
- 其它算术/比较运算符左右相等,即左结合。

`UNARY_PRIORITY = 12`:一元运算符(`not`、`-`、`~`、`#`)的优先级是固定的 12。把它放在 `*`/`/`(11)和 `^`(14)之间——这意味着 `-a^b` 是 `-(a^b)`(因为 `^` 的 left=14 > 一元的 12,`^` 先吃掉 `a`),而 `-a * b` 是 `(-a) * b`(一元 12 > `*` 的 left=11,一元先吃掉 `a`)。这是符合数学直觉的结合。

这张表是 Lua 表达式语法的全部"优先级规则"。没有文法产生式、没有 BNF,优先级就是这 18 行数字。

#### `subexpr`:算符优先的递归实现

`subexpr`(lparser.c:1374)是算符优先算法的递归实现。算法的精华在这一段:

```c
static BinOpr subexpr (LexState *ls, expdesc *v, int limit) {
  BinOpr op;
  UnOpr uop;
  enterlevel(ls);
  uop = getunopr(ls->t.token);
  if (uop != OPR_NOUNOPR) {  /* prefix (unary) operator? */
    int line = ls->linenumber;
    luaX_next(ls);  /* skip operator */
    subexpr(ls, v, UNARY_PRIORITY);
    luaK_prefix(ls->fs, uop, v, line);
  }
  else simpleexp(ls, v);
  /* expand while operators have priorities higher than 'limit' */
  op = getbinopr(ls->t.token);
  while (op != OPR_NOBINOPR && priority[op].left > limit) {
    expdesc v2;
    BinOpr nextop;
    int line = ls->linenumber;
    luaX_next(ls);  /* skip operator */
    luaK_infix(ls->fs, op, v);
    /* read sub-expression with higher priority */
    nextop = subexpr(ls, &v2, priority[op].right);
    luaK_posfix(ls->fs, op, v, &v2, line);
    op = nextop;
  }
  leavelevel(ls);
  return op;  /* return first untreated operator */
}
```

这是经典的 **Floyd 算符优先算法**(Operator-Precedence Parsing)的递归形式。关键变量是 `limit`:**"只吃掉优先级严格高于 `limit` 的运算符"**。

逐段拆:

1. **先看是不是一元运算符前缀**(`not`/`-`/`~`/`#`,`getunopr` 识别)。若是,跳过运算符,递归 `subexpr` 读它的操作数——但传入的 `limit` 是 `UNARY_PRIORITY`(12)。意思是:一元的操作数,只吃掉优先级 > 12 的二元运算符(`^` 的 left=14,所以 `-a^b` 的 `a^b` 会被吃进一元的操作数)。读完操作数,调 `luaK_prefix` 发射一元运算的字节码。
2. **否则,读一个 simpleexp**(下一段讲)。
3. **然后进入 while 循环**:只要当前 token 是二元运算符,且它的 `left` 优先级**严格大于** `limit`,就处理它:
   - `luaX_next` 跳过运算符;
   - `luaK_infix(fs, op, v)`:**这是代码生成的第一钩**。它告诉 `lcode`"我马上要算 `v op ?`,先处理左操作数 `v`"——比如对算术运算,如果 `v` 还只是个常量描述符,这里决定是把它装进寄存器、还是保留为常量待折叠。
   - 递归 `subexpr(ls, &v2, priority[op].right)`:**读右操作数**,但传入的 `limit` 是 `priority[op].right`。这是关键:
     - 对左结合运算符(`left == right`,比如 `+` 是 `{10,10}`):右操作数只吃掉优先级 > 10 的(`*` 是 11,会吃;另一个 `+` 是 10,不会吃)。所以 `a + b + c` 解析成 `(a + b) + c`(左结合)。
     - 对右结合运算符(`right < left`,比如 `^` 是 `{14,13}`):右操作数吃掉优先级 > 13 的(另一个 `^` 的 left=14 > 13,会被吃)。所以 `a ^ b ^ c` 解析成 `a ^ (b ^ c)`(右结合)。
   - `luaK_posfix(fs, op, v, &v2, line)`:**代码生成第三钩**。左右操作数都齐了,发射二元运算的字节码,把结果描述写回 `v`。
   - `op = nextop`:把右操作数解析途中返回的"未处理的运算符"作为下一个要处理的运算符(它已经 > 当前的 `limit`,所以 while 循环条件可能仍成立)。

4. 当遇到一个优先级 ≤ `limit` 的运算符,或不是运算符时,`while` 退出,返回当前 `op` 给上层。

注意整个算法**没有把表达式构造成树**。它就是"读一个操作数、看下一个运算符、决定要不要吃它、吃了就立刻发射字节码"。表达式 `a + b * c - d` 的解析过程是一边读、一边算,字节码按 `* → + → -` 的求值顺序逐条产生。最终的产物是 `code` 数组里几条 `OP_MUL`/`OP_ADD`/`OP_SUB` 指令,不是一棵 AST。

`expr` 调用 `subexpr(ls, v, 0)`——传入 `limit=0`,意味着所有优先级 > 0 的运算符都会被吃掉(所有运算符优先级都 ≥ 1,即整个表达式)。

`enterlevel`/`leavelevel`(lparser.c:570,展开成 `luaE_incCstack`/`nCcalls--`)是**递归深度保护**:Lua 用 C 函数递归实现下降,深度受 C 栈限制。`enterlevel` 每深一层递增一个全局计数器,超过 `LUAI_MAXCCALLS`(默认 200)就报 "C stack overflow"。这保护的是"嵌套极深的表达式/语句"导致的 C 栈爆掉。

### 2.7 simpleexp:表达式的"原子"

`subexpr` 读到不是一元运算符时,调 `simpleexp`(lparser.c:1254):

```c
static void simpleexp (LexState *ls, expdesc *v) {
  /* simpleexp -> FLT | INT | STRING | NIL | TRUE | FALSE | ... |
                  constructor | FUNCTION body | suffixedexp */
  switch (ls->t.token) {
    case TK_FLT: {
      init_exp(v, VKFLT, 0);
      v->u.nval = ls->t.seminfo.r;
      break;
    }
    case TK_INT: {
      init_exp(v, VKINT, 0);
      v->u.ival = ls->t.seminfo.i;
      break;
    }
    case TK_STRING: {
      codestring(v, ls->t.seminfo.ts);
      break;
    }
    case TK_NIL: {
      init_exp(v, VNIL, 0);
      break;
    }
    case TK_TRUE: {
      init_exp(v, VTRUE, 0);
      break;
    }
    case TK_FALSE: {
      init_exp(v, VFALSE, 0);
      break;
    }
    case TK_DOTS: {  /* vararg */
      FuncState *fs = ls->fs;
      check_condition(ls, isvararg(fs->f),
                      "cannot use '...' outside a vararg function");
      init_exp(v, VVARARG, luaK_codeABC(fs, OP_VARARG, 0, fs->f->numparams, 1));
      break;
    }
    case '{' /*}*/: {  /* constructor */
      constructor(ls, v);
      return;
    }
    case TK_FUNCTION: {
      luaX_next(ls);
      body(ls, v, 0, ls->linenumber);
      return;
    }
    default: {
      suffixedexp(ls, v);
      return;
    }
  }
  luaX_next(ls);
}
```

注意几个点:

- **5.5 区分 `TK_FLT` 和 `TK_INT`**(`VKFLT` 和 `VKINT` 两种 `expkind`)。讲 5.3 之前的资料常把它们说成 `TK_NUMBER`,5.3 起 Lua 整数和浮点是两种类型,词法和语法这里都分开处理——整数常量保留为 `lua_Integer`(`ival`),浮点常量保留为 `lua_Number`(`nval`)。这让编译期常量折叠能区分 `1` 和 `1.0`(`3 / 2` 是 `1.5`,`3 // 2` 是 `1`)。
- 字面量在 `simpleexp` 里**不立刻进常量表 `k`**,而是以 `VKINT`/`VKFLT`/`VKSTR` 的"裸"形式留在 `expdesc` 里。只有当这个值真的需要进寄存器时(比如作为非立即数二元运算的操作数),才由 `luaK_exp2K` 落进 `k`。这是**延迟落表**:很多字面量(比如折叠掉的、或被立即数指令吸收的)根本不进 `k`,省空间。
- `...`(`TK_DOTS`)直接发射 `OP_VARARG` 指令,并检查当前函数是不是 vararg(`isvararg(fs->f)`)。
- `{...}` 表构造走 `constructor`。
- `function ... end` 匿名函数走 `body`——它会**创建一个新的 `FuncState` 和新的 `Proto`**,递归编译函数体,结束后挂到父 `Proto->p`,父函数里发射一条 `OP_CLOSURE` 指令创建闭包。注意 `body` 后面直接 `return`,不走末尾的 `luaX_next`。
- 默认分支走 `suffixedexp`:处理变量名、字段访问、方法调用、索引、函数调用这一整类"带后缀"的表达式。

末尾的 `luaX_next(ls)` 对前 6 个字面量分支统一跳过当前 token(它们都是单 token 的字面量)。

### 2.8 `suffixedexp`:变量与调用的累积

`suffixedexp`(lparser.c:1217)解析一个"主表达式 + 一串后缀":

```c
static void suffixedexp (LexState *ls, expdesc *v) {
  /* suffixedexp ->
       primaryexp { '.' NAME | '[' exp ']' | ':' NAME funcargs | funcargs } */
  FuncState *fs = ls->fs;
  primaryexp(ls, v);
  for (;;) {
    switch (ls->t.token) {
      case '.': {  /* fieldsel */
        fieldsel(ls, v);
        break;
      }
      case '[': {  /* '[' exp ']' */
        expdesc key;
        luaK_exp2anyregup(fs, v);
        yindex(ls, &key);
        luaK_indexed(fs, v, &key);
        break;
      }
      case ':': {  /* ':' NAME funcargs */
        expdesc key;
        luaX_next(ls);
        codename(ls, &key);
        luaK_self(fs, v, &key);
        funcargs(ls, v);
        break;
      }
      case '(': case TK_STRING: case '{' /*}*/: {  /* funcargs */
        luaK_exp2nextreg(fs, v);
        funcargs(ls, v);
        break;
      }
      default: return;
    }
  }
}
```

`primaryexp`(lparser.c:1195)读一个"原子":要么是 `( expr )`(括号表达式,`luaK_dischargevars` 把它强制求值到一个寄存器),要么是一个名字 `TK_NAME`(`singlevar` 处理)。

`singlevar`(lparser.c:538)就一行:

```c
static void singlevar (LexState *ls, expdesc *var) {
  buildvar(ls, str_checkname(ls), var);
}
```

`buildvar` 的逻辑很关键(lparser.c:520):先用 `singlevaraux` 在当前 `FuncState` 链上找这个名字。`singlevaraux`(lparser.c:476)沿 `fs->prev` 一层层往外找:

- 在当前函数的活跃局部变量里找到 → `VLOCAL`(如果这个变量将来会被内层函数当 upvalue 捕获,这里 `markupval` 标记它的块需要 close,P2-08)。
- 没找到,但外层函数有 → 沿 `prev` 往上找,沿途在外层各 `FuncState` 里登记一个 upvalue(`newupvalue`),最终在当前函数里得到 `VUPVAL`。
- 一直找到主函数都没找到 → 它是**全局变量**,走 `buildglobal`。

`buildglobal`(lparser.c:502)做的事是前面提到的 `_ENV` 机制的落点:

```c
static void buildglobal (LexState *ls, TString *varname, expdesc *var) {
  FuncState *fs = ls->fs;
  expdesc key;
  init_exp(var, VGLOBAL, -1);  /* global by default */
  singlevaraux(fs, ls->envn, var, 1);  /* get environment variable */
  ...
  luaK_exp2anyregup(fs, var);  /* _ENV could be a constant */
  codestring(&key, varname);  /* key is variable name */
  luaK_indexed(fs, var, &key);  /* 'var' represents _ENV[varname] */
}
```

它先把 `_ENV`(`ls->envn`)本身作为一个变量解析(走 `singlevaraux`,在主函数层会找到那个第一个 upvalue),再把变量名作为字符串 key,生成一个"表索引"表达式 `_ENV[varname]`。后续 `luaK_indexed` 会根据情况发射 `GETTABUP`/`GETFIELD`/`GETI` 等指令(具体在 P2-09 讲)。这就是 Lua 全局变量的物理实现:**没有单独的全局表查询路径,全局变量访问和 `t.field` 是同一条代码路径**,只是表是 `_ENV` 这个 upvalue。这是"统一"(P1 主线)在编译期的体现:全局变量、模块成员、对象字段,全是 `VINDEXED` 一类表达式。

回到 `suffixedexp` 的循环。它把后缀**累积**到 `v` 上:

- `.NAME`(`fieldsel`):`v.NAME`。把 `v` 求值到寄存器/upvalue,生成索引表达式 `v["NAME"]`,后续发射 `GETFIELD`。
- `[exp]`(`yindex`):`v[exp]`。括号里读一个表达式作为 key,生成 `VINDEXED`,后续根据 key 类型(`GETI` 整数 key,或 `GETFIELD` 字符串 key)。
- `:NAME funcargs`:**方法调用** `v:NAME(args)`。`luaK_self` 发射一条 `OP_SELF` 指令(把 `v` 和 `NAME` 拼成 `v.NAME`,同时把 `v` 自己作为第一个参数压栈——这就是 `self` 的物理来源),然后 `funcargs` 处理参数。
- 直接 `(`/字符串/`{`:**函数调用** `v(args)` / `v"str"` / `v{...}`。先把 `v` 求值到下一个寄存器,再 `funcargs` 处理参数,最终发射 `OP_CALL`。

整个循环把 `a.b.c[1]:m(x, y).n` 这种链式访问**逐步折叠**进一个 `expdesc`:每一步都把前一步的表达式作为表/对象,生成一个新的索引或调用表达式,覆盖回 `v`。最终 `v` 描述的是整条链最末端的结果。注意中间没有 AST——每一步都已经在内存里准备好了下一步指令所需的全部信息(表在哪个寄存器、key 是什么),真正发射哪条 `GETFIELD`/`GETI`/`CALL` 由 `lcode` 在需要把 `v` 落到寄存器时(`luaK_exp2nextreg` 等)决定。

### 2.9 函数调用参数:`funcargs`

`funcargs`(lparser.c:1138)处理三种调用形式:

```c
static void funcargs (LexState *ls, expdesc *f) {
  ...
  switch (ls->t.token) {
    case '(': {  /* funcargs -> '(' [ explist ] ')' */
      ...
      explist(ls, &args);
      ...
      break;
    }
    case '{' /*}*/: {  /* funcargs -> constructor */
      constructor(ls, &args);
      break;
    }
    case TK_STRING: {  /* funcargs -> STRING */
      codestring(&args, ls->t.seminfo.ts);
      luaX_next(ls);
      break;
    }
    ...
  }
  ...
  init_exp(f, VCALL, luaK_codeABC(fs, OP_CALL, base, nparams+1, 2));
  ...
  fs->freereg = cast_byte(base + 1);
}
```

`f("str")` 和 `f{...}` 是 `f("str")` / `f({...})` 的语法糖,这里展开。最后一条 `OP_CALL` 指令把整个调用变成 `VCALL` 表达式(值是调用结果,在 `base` 寄存器)。注意末尾 `fs->freereg = base + 1`——调用完成后,参数占用的寄存器全部释放,只留函数和结果共用的 `base` 槽。这是寄存器分配"调用即清理"的具体规则。

### 2.10 走一遍:`local a = b + c * 2`

把上面的链路串起来。源码:

```lua
local a = b + c * 2
```

假设 `b`、`c` 都是当前函数的局部变量(分别在 R0、R1),主函数已开始编译。token 流:`TK_LOCAL TK_NAME(a) '=' TK_NAME(b) '+' TK_NAME(c) '*' TK_INT(2) TK_EOS`。

**第 1 步:`statement` 分发**。当前 token 是 `TK_LOCAL`,走 `case TK_LOCAL`,跳过 `LOCAL`,下一个 token 不是 `TK_FUNCTION`,走 `localstat`。

**第 2 步:`localstat` 前半段(声明)**。读 `TK_NAME(a)`,没有属性(`<const>`/`<close>`),`new_varkind` 把 `a` 登记进 `Dyndata->actvar`(此时还没分配寄存器,只是登记名字)。`nvars=1`。

**第 3 步:`localstat` 后半段(初始化表达式)**。读到 `=`,调 `explist` 读右边表达式列表。`explist` 调 `expr`。

**第 4 步:`expr → subexpr(limit=0)`**。
- 当前 token `TK_NAME(b)`。`getunopr` 返回 `NOUNOPR`,不是一元前缀。走 `simpleexp`。
- `simpleexp` 的 default 分支走 `suffixedexp → primaryexp → singlevar("b")`。`singlevaraux` 在当前 `FuncState` 的活跃变量里找到 `b`,它是 `VLOCAL`,`u.var.ridx=0`(R0)。`suffixedexp` 的循环看到下一个 token 是 `+`,不是后缀,退出。回到 `simpleexp` default 分支直接 return(注意这里 default 不走末尾的 `luaX_next`,因为 `singlevar` 内部已经 `luaX_next` 跳过了名字)。
- 当前 `v.k = VLOCAL, v.u.var.ridx = 0`。
- 回到 `subexpr` 的 while:`getbinopr('+')` 返回 `OPR_ADD`。`priority[OPR_ADD].left = 10 > 0`,进入循环。
- `luaX_next` 跳过 `+`。
- `luaK_infix(fs, OPR_ADD, v)`:**第一钩**。对算术运算,`luaK_infix`(lcode.c:1718)看到 `v` 是 `VLOCAL` 不是 numeral,调 `luaK_exp2anyreg(fs, v)` 把 `b` "固化"到寄存器——但 `b` 本来就在 R0,这里实际上不发射任何指令,只把 `v` 标记成 `VNONRELOC, u.info=0`(`lcode` 的 `discharge2reg` 看到 local 已在寄存器就直接复用)。
- 递归 `subexpr(ls, &v2, priority[OPR_ADD].right=10)` 读右操作数。
  - 当前 token `TK_NAME(c)`。同样路径,`v2.k = VLOCAL, u.var.ridx=1`。
  - while:`getbinopr('*')` 返回 `OPR_MUL`。`priority[OPR_MUL].left=11 > 10`,进入循环。
  - `luaX_next` 跳过 `*`。
  - `luaK_infix(fs, OPR_MUL, v2)`:把 `c` 固化到 R1,`v2` 变 `VNONRELOC, u.info=1`。
  - 递归 `subexpr(ls, &v2b, priority[OPR_MUL].right=11)` 读 `*` 的右操作数。
    - 当前 token `TK_INT(2)`。`simpleexp` 走 `case TK_INT`,`v2b.k = VKINT, v2b.u.ival = 2`。**注意:这里 2 没有进常量表 `k`,只是 `expdesc` 里的 `VKINT`。** `luaX_next` 跳过 `2`。
    - while:`getbinopr(<下一个 token>)`——下一个是 `TK_EOS`(假设这是函数体最后一条),返回 `NOBINOPR`,while 不进入。返回 `NOBINOPR`。
  - `luaK_posfix(fs, OPR_MUL, v2, &v2b, line)`:**第三钩**。`v2` 是 R1(`VNONRELOC`),`v2b` 是 `VKINT(2)`。`codearith`→`codearith` 看右操作数是整常量且在立即数范围,走 `codebinK` 还是 `codebini`?

    这里要细看 `lcode` 的逻辑:`OPR_MUL` 走 `codecommutative`(lcode.c:1598),它先尝试交换操作数让常量在右边,然后调 `codearith`。`codearith`(lcode.c:1584)对右操作数 `v2b(VKINT)`:`tonumeral(v2b, NULL) && luaK_exp2K(fs, v2b)`——`luaK_exp2K` 尝试把 `2` 转成 K 常量(进 `k` 表)。如果成功,走 `codebinK` 发射 `OP_MULK R_dest R1 K[2]`。但 5.5 对小整数还有 `codebini` 路径(`OP_MULK` vs 立即数 `OP_MULI`?实际查 `lopcodes.h`:5.5 的乘法**没有** `OP_MULI`,只有 `OP_MUL`/`OP_MULK`——立即数形式只对 `+`/`-` 有 `OP_ADDI`/`OP_SUBI`)。所以 `c * 2` 编译成 `OP_MULK` 或 `OP_MUL`(取决于 `2` 能否进 K 表的小整数槽位)。为简化,假设走 `OP_MULK`:发射 `MULK R2 R1 K[0]`(把 `2` 放进 `k[0]`,结果放新分配的 R2)。`v2` 变成 `VRELOC, u.info=<MULK 的 pc>`(结果可以重定位)。
  - 递归返回的 `nextop` 是 `NOBINOPR`,while 退出,返回 `NOBINOPR`。
- 回到外层 `subexpr` 的 while。`luaK_posfix(fs, OPR_ADD, v, &v2, line)`:左 `v` 是 R0(`b`),右 `v2` 是 `MULK` 结果(`VRELOC`,会被 `luaK_exp2anyreg` 固化到 R2)。`OPR_ADD` 走 `codecommutative`。两个操作数都不是整常量(右边是寄存器),走 `codebinexpval` 发射 `ADD R3 R0 R2`。`v` 变 `VRELOC` 指向 `ADD`。
- 外层 while 的 `nextop` 是 `NOBINOPR`,退出。`expr` 返回,`v` 是 `ADD` 的结果(`VRELOC`)。

**第 5 步:回到 `localstat`,赋值**。`adjust_assign(nvars=1, nexps=1, &e)`——变量数等于表达式数,不需要调整。`adjustlocalvars(1)`:给 `a` 分配寄存器 R3(下一个空闲),登记调试信息。然后 `luaK_storevar` 把 `e`(`VRELOC` 的 `ADD`)的"结果寄存器"**重新指派**到 `a` 的寄存器——`lcode` 的 `luaK_freeexp` + `luaK_exp2nextreg` 机制会把那条 `ADD` 指令的目标寄存器直接改成 R3(因为 `VRELOC` 还没固定,可以重定位到任意空闲寄存器),省掉一条 `MOVE`。

最终 `code` 数组大致是:

```
MULK  R2  R1  K[0]   ; R2 = R1 * 2     (c * 2)
ADD   R3  R0  R2     ; R3 = R0 + R2    (b + (c*2))
RETURN1 R3           ; 函数尾的隐式 return a
```

(实际 `a` 的寄存器分配可能因 `freereg` 状态而异,这里示意。)

整个过程:**没有 AST**。从读第一个 token 到最后一条 `ADD` 写进 `code`,手里只有一个 `expdesc v` 被反复改写,加上 `FuncState` 的几个计数器。`b + c * 2` 这棵"表达式树"从未在内存里作为对象存在过——它只在递归调用栈上短暂存在(对应 `subexpr` 的嵌套调用),代码生成是边读边发的。

### 2.11 表构造 `constructor`、语句细节

**表构造** `{...}`(lparser.c:1028)值得单独看,因为它体现了单遍编译的另一个特点——**两遍扫描的替代**:

```c
static void constructor (LexState *ls, expdesc *t) {
  ...
  int pc = luaK_codevABCk(fs, OP_NEWTABLE, 0, 0, 0, 0);
  ...
  luaK_code(fs, 0);  /* space for extra arg. */
  ...
  init_exp(t, VNONRELOC, fs->freereg);  /* table will be at stack top */
  luaK_reserveregs(fs, 1);
  ...
  checknext(ls, '{' /*}*/);
  cc.maxtostore = maxtostore(fs);
  do {
    if (ls->t.token == /*{*/ '}') break;
    if (cc.v.k != VVOID)  /* is there a previous list item? */
      closelistfield(fs, &cc);  /* close it */
    field(ls, &cc);
    ...
  } while (testnext(ls, ',') || testnext(ls, ';'));
  check_match(ls, /*{*/ '}', '{' /*}*/, line);
  lastlistfield(fs, &cc);
  luaK_settablesize(fs, pc, t->u.info, cc.na, cc.nh);
}
```

注意 `luaK_codevABCk(fs, OP_NEWTABLE, 0, 0, 0, 0)` 在**解析表内容之前**就先发射了一条 `OP_NEWTABLE` 指令(后面跟一个 extra-arg 占位)。因为单遍编译不知道这个表最终有多少数组元素(`na`)和多少哈希记录(`nh`),但指令必须现在就发射(否则顺序错乱)。所以先发一个占位的 `OP_NEWTABLE`,最后用 `luaK_settablesize` **回填**真实的 `na`/`nh`(VM 用这两个数预分配表大小,避免边插边 rehash)。

这就是单遍编译处理"需要整体信息才能决定的字段"的标准手法:**先占位、后回填**。跳转目标、表大小、`OP_FORPREP` 的循环回跳点,都是这么处理的。

字段解析 `field`(lparser.c:990)区分 `listfield`(`exp` 形式的数组项)、`recfield`(`name = exp` 或 `[exp] = exp` 的哈希项),靠一个 lookahead 区分:`TK_NAME` 后面是 `=` 就是 recfield,否则是 listfield。这是少数几处需要两 token 前看的地方(Lua 的 `LexState` 有一个 `lookahead` 字段,llex.h:69,`luaX_lookahead` 提前读一个 token)。

**赋值与多重赋值**走 `exprstat`/`restassign`(lparser.c:1999、1498)。`exprstat` 先解析一个 `suffixedexp`(左值),再看后面是 `=`/`,` 还是直接结束。直接结束就是函数调用语句——它会把那条 `OP_CALL` 的 C 操作数(返回值数)改成 1(丢弃返回值,`SETARG_C(*inst, 1)`)。是 `=`/`,` 则是赋值,递归 `restassign` 处理 `x, y, z = a, b, c` 这种多重赋值。多重赋值的关键:`adjust_assign` 调整左右数量(多了截断,少了补 nil,最后一个表达式若是多返回值调用则展开为 `LUA_MULTRET`)。

**`return`**(`retstat`,lparser.c:2017)有个优化:如果 `return f()` 的 `f()` 是函数调用(多返回值),且不在 to-be-closed 作用域内,会把那条 `OP_CALL` 直接改成 `OP_TAILCALL`——尾调用优化在编译期就标记好(P4-15)。

### 2.12 语句总览

把 `statement` 能分发到的语句种类列全(本章只给框架,细节分散在各 `xxxxstat`):

- 空语句 `;`
- `if ... then ... elseif ... else ... end`(`ifstat`)
- `while cond do ... end`(`whilestat`)
- `do ... end`(`block`)
- `for i = e1, e2, e3 do ... end` / `for k, v in pairs(t) do ... end`(`forstat`,数值 for 和泛型 for 共用入口)
- `repeat ... until cond`(`repeatstat`)
- `function a.b.c(...) ... end`(`funcstat`)
- `local name = ...` / `local function f(...) ... end`(`localstat`/`localfunc`)
- `global ...`(5.5 新增,`globalstatfunc`)
- `::label::`(`labelstat`)
- `return explist`(`retstat`)
- `break`(`breakstat`,实际是 `goto ::break::`)
- `goto label`(`gotostat`)
- 函数调用语句 / 赋值(`exprstat`,default 分支)

**块结构**:`block`(lparser.c:1419)进入时压一个 `BlockCnt`,解析 `statlist`,退出时 `leaveblock` 弹出并关闭本块作用域的局部变量(把它们的 `endpc` 记到调试信息)。`BlockCnt` 链(`bl`)是 goto/label 解析和 upvalue close 决策的基础,P2-08 详讲。

## 三、为什么这样设计是 sound 的

### 3.1 单遍编译为什么不丢正确性

单遍编译最大的担忧是:**没有 AST 兜底,怎么保证生成的字节码语义和源码一致?** Lua 的回答是三层保障:

1. **递归下降本身保证语法正确**。每个非终结符对应一个函数,函数入口检查当前 token 是否合法(`check`/`checknext`/`check_match`),不合法就 `luaX_syntaxerror`。语法错的程序根本编译不过。
2. **算符优先算法保证求值顺序正确**。`priority` 表 + `limit` 参数,严格按"优先级高的先吃、同优先级按结合性吃"的规则展开表达式。`a + b * c` 必然先算 `b * c`,因为 `subexpr(limit=10)` 读 `*` 时 `priority[MUL].left=11 > 10`,会递归吃掉 `b * c` 才返回。这个不变式是数学上可证明的(算符优先文法的标准结论)。
3. **`expdesc` 的延迟求值保证代码生成不重复、不遗漏**。一个子表达式解析完,它的"值在哪"由 `expdesc.k` 精确描述(常量/寄存器/upvalue/待重定位/调用结果)。只有当这个值真的需要落到某个具体位置(比如作为二元运算的操作数、作为赋值的右值)时,`luaK_dischargevars`/`luaK_exp2nextreg` 才把它"放电"成具体指令。延迟求值让常量折叠(`constfolding`)、立即数优化(`OP_ADDI`)、寄存器复用(local 已在寄存器就不重新加载)这些优化自然发生,且不会因为"先发射再优化"导致指令顺序错乱。

### 3.2 算符优先为什么 sound:结合性的数学保证

`priority[op].left > limit` 这个判断是 sound 的核心。证明思路:

- `subexpr(ls, v, limit)` 的契约是:"读一个操作数,然后吃掉所有优先级**严格大于** `limit` 的后续二元运算符,返回时 `v` 是整个被吃掉的表达式的求值结果"。
- 对左结合运算符 `op`(`left == right == p`):处理完 `a op`,递归读右操作数时传 `limit = right = p`。右操作数只会吃掉优先级 > p 的运算符,不会吃掉另一个 `op`(它的 left=p,不 > p)。所以 `a op b op c` = `(a op b) op c`——左结合。
- 对右结合运算符 `op`(`left = p, right = q, p > q`):递归读右操作数时传 `limit = q`。右操作数会吃掉优先级 > q 的运算符,包括另一个 `op`(它的 left=p > q)。所以 `a op b op c` = `a op (b op c)`——右结合。

关键在于 `>` 而非 `>=`:`>=` 会导致左结合变成右结合(右操作数会吃掉同优先级的下一个运算符)。Lua 选 `>` 配合 `left==right` 实现左结合,配合 `left>right` 实现右结合,数学上完备。

### 3.3 优先级驱动结合性,省掉了递归子程序的层级

传统递归下降处理表达式,要为每个优先级写一层函数:`expr → addexpr → mulexpr → ... → primary`。每层一个函数,层级等于优先级数。Lua 的算符优先把这些层级**折叠进一个 `subexpr` + 一张 `priority` 表**:同一个函数靠 `limit` 参数区分层级。好处:

- 代码量小(一张 18 行的表 vs 十几个层层调用的函数)。
- 加一个新运算符只需在 `getbinopr`/`getunopr` 加一个 case、在 `priority` 表加一行,不必改文法结构。
- 递归深度受实际运算符链长度控制,而不是固定优先级层数。

这是"精简换小而快"在解析算法层面的落地:用一张表换掉了一整套递归子程序层级。

### 3.4 `expdesc` 的延迟求值不丢优化机会

`expdesc` 把"值在哪"这件事抽象成几种状态(`VKINT` 裸常量、`VKSTR` 裸字符串、`VLOCAL` 局部变量、`VRELOC` 可重定位、`VNONRELOC` 已固定寄存器、`VCALL` 调用结果……)。这个抽象让 `lcode` 能在最后一刻做最优选择:

- `b + 2`:`2` 是 `VKINT`,到 `posfix` 时若在立即数范围,走 `OP_ADDI` 不进 `k` 表;若不在,走 `OP_ADDK` 进 `k` 表;若左操作数也是常量,直接 `constfolding` 折叠掉,不发射任何指令。
- `b + c`(都在寄存器):`c` 已在 R1,`discharge2reg` 不发 `MOVE`,直接用 R1 作为 `ADD` 的操作数。

这些优化如果先建 AST 再遍历做,需要额外的 AST 节点遍历和标记;单遍编译里它们发生在 `expdesc` 状态转换的瞬间,零额外开销。

## 四、★对照 CPython + 回扣主线

### CPython 的两遍式编译

CPython 编译一段 Python 源码,是两步走:

1. **`Parser/parser.c`(PEG parser,3.9+ 取代了旧的 LL(1) parser)**:读 token 流,构建一棵 **CST(具体语法树)** 或直接 **AST**。AST 节点是 `stmt_ty`/`expr_ty` 等 C 结构体,每个表达式、语句都是一个堆上分配的节点对象,整棵树和 token 数量线性相关。
2. **`Python/compile.c`**:遍历这棵 AST,为每个节点发射字节码(`LOAD_FAST`/`BINARY_OP`/`STORE_FAST` 等栈式指令)。遍历完才释放 AST。

对比 Lua 的单遍式:

| 维度 | Lua 5.5 | CPython 3.12+ |
|---|---|---|
| **遍数** | 单遍:读 token 同时发射指令 | 两遍:先建 AST,再遍历 AST 发射指令 |
| **中间产物** | 无 AST,只有 `expdesc`(一个栈上结构体)+ `FuncState` 计数器 | 完整 AST(`stmt_ty`/`expr_ty` 节点树,堆分配) |
| **表达式优先级** | 算符优先算法 + 一张 18 行 `priority` 表 | PEG 文法里为每个优先级写一条产生式(`sum: term (('+'|'-') term)*`),优先级嵌在文法结构里 |
| **代码生成耦合度** | 紧耦合:解析器直接调 `luaK_infix/prefix/posfix` 发射指令 | 松耦合:parser 只产 AST,compiler 独立遍历 AST |
| **编译期峰值内存** | O(最大表达式深度 + 函数体大小),无 AST | O(源码大小),整棵 AST 常驻 |
| **指令模型** | 寄存器式(`ADD R3 R0 R2`) | 栈式(`LOAD_FAST`/`BINARY_OP`/`STORE_FAST`) |

两边的取舍很清晰:

- **CPython 选两遍 + AST**,代价是编译期内存更大、多一遍遍历;换来的是 **parser 和 compiler 解耦**——可以独立优化 AST(类型推断、常量折叠、future import 处理)、支持更复杂的语法分析(PEG 能处理 Lua 的 LL(1)+算符优先处理不了的歧义),以及调试工具(纯 AST 操作的工具链、AST 转 Python 源)。
- **Lua 选单遍 + 无 AST**,代价是 parser 和 codegen 紧耦合(改一个要同时懂另一个)、优化只能在解析瞬间做;换来的是**编译期极低内存占用、编译速度快、零中间产物**。这恰好是"嵌入宿主、运行时编译小段脚本"场景最看重的。

Lua 经常在 nginx/Redis 里被宿主在请求路径上 `loadstring` 编译一段脚本,CPython 的 AST 方案在那种场景下内存和延迟都偏高。Lua 的单遍编译让"编译一段脚本"的开销接近"解析一段配置"——这是它能塞进任何宿主的编译期根基之一。

另一个对照点:**算符优先 vs PEG 优先级**。Lua 用一张表 + 一个 `subexpr` 函数搞定所有二元运算符优先级;CPython 的 PEG 文法里,每个优先级是一条独立的规则(`sum`、`term`、`factor`、`power`……),层级等于优先级数。Lua 的做法更紧凑(18 行表 vs PEG 文法里十几条产生式),代价是不如 PEG 直观、不支持更复杂的歧义处理。对 Lua 这种运算符集合固定的语言,算符优先足够;对 Python 这种语法持续演进的语言,PEG 的表达力更重要。

### 回扣主线

这一章在"统一与精简换小而快"里的位置:**精简**在编译期的最高体现。

- **精简**:砍掉 AST 中间层,解析器直接产字节码。一个 `expdesc` + 一个 `FuncState` 就是全部工作内存。算符优先算法用一张 18 行的表,折叠了传统递归下降十几层的子程序。
- **换小**:编译期内存 O(深度)而非 O(源码大小)。嵌入宿主编译脚本,不为 AST 预留堆内存。
- **换快**:单遍编译,少一遍遍历。`loadstring` 的延迟接近配置解析,不是重型编译。
- **统一**:全局变量访问和 `t.field` 是同一条 `VINDEXED` 路径——`_ENV` 只是个普通 upvalue,没有专门的全局表查询机制。这是 P1 主线(一切复合数据都是 Table)在编译期的延续。

下一章 P2-08 进入语法分析下半部分:**作用域与 upvalue 的静态收集**——看 `BlockCnt` 链怎么管 local 变量的生命周期、`singlevaraux` 怎么沿 `FuncState` 链把外层局部变量登记成 upvalue、以及 `needclose`/`OP_CLOSE` 怎么在块退出时把开放 upvalue 闭合。这些是单遍编译"在不建 AST 的前提下,仍然能正确处理词法作用域和闭包捕获"的关键机制。
