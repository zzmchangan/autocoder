# P5-17 对象回收与弱表

> **本书主线**:统一与精简换小而快。**二分法**:编译侧(源码→字节码)↔ 执行侧(VM 执行 + 管值生命周期)。本章在执行侧的 GC 篇。**★对照**:CPython 的 `weakref` + `__del__`。**源码**:lua-5.5.0。**基调**:纯直球,不用比喻。
>
> **承接**:P5-16 讲清了增量三色标记怎么"可中断地"推进;本章回答剩下的那一半——**这一堆灰的、黑的、白的对象,GC 具体怎么认识每一种?弱表和 `__gc` 又是怎么在不破坏三色不变式的前提下被正确回收的?**

## 一、这章解决什么问题

P5-16 给出了三色标记的骨架:对象有白(未达)、灰(已达但子对象未扫完)、黑(已扫完)三态;`propagatemark` 从灰队列取一个对象、扫它的子对象、把自己涂黑。但骨架里留了两个口子,这两个口子正是本章要填的。

**第一个口子:对象种类不止一种,怎么"扫子对象"统一得了?**

Lua 的可回收对象有七类:`Table`、字符串(`TString`,短/长两种)、Lua 闭包(`LClosure`)、C 闭包(`CClosure`)、`UpVal`、线程(`lua_State`)、`Proto`、`userdata`(`Udata`)。每一类的"子对象"长得完全不一样:`Table` 的子对象是 array 部分 + node 部分 + metatable;闭包的子对象是它捕获的 upvalue;线程的子对象是它值栈上 `top` 以下的活元素 + 一串 open upvalue。如果为每一类各写一套互不相干的标记代码,就违背了"统一换小"的主线。

Lua 的做法是给所有可回收对象一个**公共头** `CommonHeader`,所有 GC 操作先通过这个头认对象,再按类型分派。这套分派必须做到:**任何一种被漏标的类型,都意味着那一类对象永不被回收(内存泄漏)或被错收(悬空指针)**。所以"按类型正确遍历"是 GC soundness 的第一根支柱。

**第二个口子:不是所有引用都该让对象活着。**

真实程序里常有这样的需求:我有一个缓存表,`key` 是某个大对象,只要那个大对象本身没人用了,我希望整个缓存项自动消失,而不是被缓存表强行拽着活下去。又比如:`userdata` 被回收时,我想调一个 C 函数去释放它背后的资源(关闭文件、解锁)。

这两种需求分别对应:

- **弱表**(weak table):表的 key 和/或 value 是"弱引用"——指向对象但不让对象活。对象一旦没有强引用,弱表里这条记录就该被清掉。
- **`__gc` 终结器**(finalizer):对象被回收前,先调它的 `__gc` 元方法。

这两个机制直接挑战三色不变式。强引用的世界里,"黑对象指向白对象"是 barrier 要修补的违例;但弱表里,**弱引用指向白对象恰恰是正常的、期望的**——我们就是要让那个白对象死掉。finalizer 更麻烦:对象被判定为"死"了,却在调 `__gc` 时又把控制权交回 Lua 代码,那段代码可能重新引用这个对象(把它从死拉回活),这叫**复活**。

所以本章要回答的核心问题是:

> **增量三色 GC 怎么认识七种对象、怎么在不破坏不变式的前提下回收弱表、怎么保证 `__gc` 复活不悬空?**

下面用 lua-5.5.0 的源码逐一落地。

## 二、源码怎么实现

### 2.1 公共头 `CommonHeader`:七类对象的统一入口

所有可回收对象都以同一个三字段头开头,定义在 `lobject.h:301`:

```c
#define CommonHeader	struct GCObject *next; lu_byte tt; lu_byte marked
```

三个字段:

- `next`:把对象串进 `global_State` 的某条链(`allgc` / `finobj` / `tobefnz` / `fixedgc`)。
- `tt`:类型 + 变体(`LUA_VTABLE` / `LUA_VSHRSTR` / `LUA_VLCL` 等)。
- `marked`:GC 位域——白位(双白)、黑位、finalized 位、age 位(分代用)。

`GCObject` 本身只是这个头的裸壳(`lobject.h:305`):

```c
typedef struct GCObject {
  CommonHeader;
} GCObject;
```

七类对象的定义无一例外地以 `CommonHeader;` 起头。贴几个关键的:

`Table`(`lobject.h:776`):

```c
typedef struct Table {
  CommonHeader;
  lu_byte flags;       /* 1<<p means tagmethod(p) is not present */
  lu_byte lsizenode;   /* log2 of number of slots of 'node' array */
  unsigned int asize;  /* number of slots in 'array' array */
  Value *array;        /* array part */
  Node *node;
  struct Table *metatable;
  GCObject *gclist;
} Table;
```

`Udata`(`lobject.h:491`):

```c
typedef struct Udata {
  CommonHeader;
  unsigned short nuvalue;  /* number of user values */
  size_t len;              /* number of bytes */
  struct Table *metatable;
  GCObject *gclist;
  UValue uv[1];            /* user values */
} Udata;
```

`UpVal`(`lobject.h:679`):

```c
typedef struct UpVal {
  CommonHeader;
  union {
    TValue *p;          /* points to stack or to its own value */
    ptrdiff_t offset;   /* used while the stack is being reallocated */
  } v;
  union {
    struct {            /* (when open) */
      struct UpVal *next;
      struct UpVal **previous;
    } open;
    TValue value;       /* the value (when closed) */
  } u;
} UpVal;
```

`Proto`(`lobject.h:602`)、`TString`(`lobject.h:405`)、`LClosure`/`CClosure`(`lobject.h:706`/`699`,`ClosureHeader` 也是 `CommonHeader; lu_byte nupvalues; GCObject *gclist`)、`lua_State`(`lstate.h:285`)同理。

**这套统一头的回报**:GC 可以用一个通用的 `GCObject *` 指针遍历整条 `allgc` 链,不关心具体类型,只读 `next`/`tt`/`marked` 就能完成"串链、判活、判死、判是否带 finalizer"。只有真正要"扫子对象"时,才按 `tt` 分派到对应类型的遍历函数。这是"统一换小"在 GC 层面的直接体现:一条扫链、一套回收代码,服务七种对象。

对象的诞生也走同一条入口 `luaC_newobjdt`(`lgc.c:297`):

```c
GCObject *luaC_newobjdt (lua_State *L, lu_byte tt, size_t sz, size_t offset) {
  global_State *g = G(L);
  char *p = cast_charp(luaM_newobject(L, novariant(tt), sz));
  GCObject *o = cast(GCObject *, p + offset);
  o->marked = luaC_white(g);   /* 染成当前白 */
  o->tt = tt;
  o->next = g->allgc;          /* 头插进 allgc */
  g->allgc = o;
  return o;
}
```

新对象一出生就被染成**当前白**(双白机制下,这一轮的白在下一轮是"另一种白",即死色)、挂到 `allgc` 链头。这就保证了"凡是被分配的,都在 GC 视野里"。

### 2.2 标记 `reallymarkobject`:按类型分派

当一个白色对象被发现可达(被某个灰/黑对象引用),就调 `reallymarkobject`(`lgc.c:339`)把它"标记"。注意"标记"不等于"涂黑"——有的对象无子可扫,当场涂黑;有的要进灰队列等后续遍历:

```c
static void reallymarkobject (global_State *g, GCObject *o) {
  g->GCmarked += objsize(o);
  switch (o->tt) {
    case LUA_VSHRSTR:
    case LUA_VLNGSTR: {
      set2black(o);  /* nothing to visit */
      break;
    }
    case LUA_VUPVAL: {
      UpVal *uv = gco2upv(o);
      if (upisopen(uv))
        set2gray(uv);  /* open upvalues are kept gray */
      else
        set2black(uv);  /* closed upvalues are visited here */
      markvalue(g, uv->v.p);  /* mark its content */
      break;
    }
    case LUA_VUSERDATA: {
      Udata *u = gco2u(o);
      if (u->nuvalue == 0) {  /* no user values? */
        markobjectN(g, u->metatable);  /* mark its metatable */
        set2black(u);  /* nothing else to mark */
        break;
      }
      /* else... */
    }  /* FALLTHROUGH */
    case LUA_VLCL: case LUA_VCCL: case LUA_VTABLE:
    case LUA_VTHREAD: case LUA_VPROTO: {
      linkobjgclist(o, g->gray);  /* to be visited later */
      break;
    }
    default: lua_assert(0); break;
  }
}
```

逐类看:

- **字符串**(`LUA_VSHRSTR`/`LUA_VLNGSTR`):字符串没有指向其他 GC 对象的指针(它的内容是裸字节),所以**当场涂黑**。这是最便宜的标记——零额外遍历。
- **`UpVal`**:闭合 upvalue(`uv->v.p` 指向自己内部的 `value`)当场涂黑,然后标它包着的值;开放 upvalue(还指向某线程栈槽)**保持灰色**——因为它的值随时可能被栈上别的写操作改,保持灰可以避免反复触发 barrier(详见 P4-14)。
- **`userdata`**:如果 `nuvalue == 0`(纯二进制块,无 uservalue),只标它的 metatable 然后涂黑;否则 fallthrough 进灰队列,留给 `traverseudata` 扫它的 uservalue 数组。
- **其余五类**(两种闭包、Table、线程、Proto):统一 `linkobjgclist(o, g->gray)`——染灰、挂进灰队列,等 `propagatemark` 取出来慢慢扫。

这里有个精妙的递归边界注释值得点出:upvalue 和 userdata 都可能在 `reallymarkobject` 内部递归调用 `reallymarkobject`(标 upvalue 的值、标 userdata 的 metatable),但注释说**递归最多两层**——因为 upvalue 的值不会是另一个 upvalue(只有闭包才持有 upvalue),userdata 的 metatable 必是 table。这个边界保证标记栈不会爆。

### 2.3 遍历 `traverse*`:灰→黑,扫子对象

灰队列的工作由 `propagatemark`(`lgc.c:727`)驱动:

```c
static l_mem propagatemark (global_State *g) {
  GCObject *o = g->gray;
  nw2black(o);
  g->gray = *getgclist(o);  /* remove from 'gray' list */
  switch (o->tt) {
    case LUA_VTABLE: return traversetable(g, gco2t(o));
    case LUA_VUSERDATA: return traverseudata(g, gco2u(o));
    case LUA_VLCL: return traverseLclosure(g, gco2lcl(o));
    case LUA_VCCL: return traverseCclosure(g, gco2ccl(o));
    case LUA_VPROTO: return traverseproto(g, gco2p(o));
    case LUA_VTHREAD: return traversethread(g, gco2th(o));
    default: lua_assert(0); return 0;
  }
}
```

取出一个灰对象、`nw2black`(涂黑)、从灰队列摘下、按 `tt` 分派。返回值是"这次扫了多少槽",给增量步计量价用(P5-16 的 `GCdebt` 控速)。

注意 `getgclist`(`lgc.c:163`)——不是所有对象都有 `gclist` 字段(字符串、upvalue 没有,因为它们不进灰队列),有 `gclist` 的七类里返回各自 `gclist` 字段的地址,供 `linkgclist` 把对象串进某条灰链。

#### traversetable:弱表分流的起点

`traversetable`(`lgc.c:608`)是本章的重心之一,因为弱表的所有逻辑都从这里分流:

```c
static l_mem traversetable (global_State *g, Table *h) {
  markobjectN(g, h->metatable);
  switch (getmode(g, h)) {
    case 0:  /* not weak */
      traversestrongtable(g, h);
      break;
    case 1:  /* weak values */
      traverseweakvalue(g, h);
      break;
    case 2:  /* weak keys */
      traverseephemeron(g, h, 0);
      break;
    case 3:  /* all weak; nothing to traverse */
      if (g->gcstate == GCSpropagate)
        linkgclist(h, g->grayagain);  /* must visit again its metatable */
      else
        linkgclist(h, g->allweak);  /* must clear collected entries */
      break;
  }
  return cast(l_mem, 1 + 2*sizenode(h) + h->asize);
}
```

`getmode`(`lgc.c:595`)查 metatable 的 `__mode` 字段:

```c
static int getmode (global_State *g, Table *h) {
  const TValue *mode = gfasttm(g, h->metatable, TM_MODE);
  if (mode == NULL || !ttisstring(mode))
    return 0;  /* ignore non-string modes */
  else {
    const char *smode = getstr(tsvalue(mode));
    const char *weakkey = strchr(smode, 'k');
    const char *weakvalue = strchr(smode, 'v');
    return ((weakkey != NULL) << 1) | (weakvalue != NULL);
  }
}
```

返回值编码:`& 1` 表示弱 value,`& 2` 表示弱 key。所以 `__mode="v"` → 1,`__mode="k"` → 2,`__mode="kv"` → 3,无 `__mode` 或非字符串 → 0。

**这里有个 5.5 vs 老资料的重要差异**。讲 Lua 5.3 的资料(包括一些书和博客)常说 `struct Table` 里有一个 `lu_byte weak` 字段,在 `__mode` 被设置时一次性算好缓存。但 5.5 的 `struct Table`(`lobject.h:776`)**没有 `weak` 字段**——它只有 `flags`(缓存 tagmethod 缺失位)、`lsizenode`、`asize`、`array`、`node`、`metatable`、`gclist`。弱性是**每次遍历时现查 `__mode`** 得到的。

为什么现查不慢?因为 `gfasttm`(`ltm.h:65`)走的是位缓存快路径:

```c
#define gfasttm(g,mt,e)  \
  (checknoTM(mt, e) ? NULL : luaT_gettm(mt, e, (g)->tmname[e]))
#define checknoTM(mt,e)	((mt) == NULL || (mt)->flags & (1u<<(e)))
```

`Table.flags` 的第 `e` 位为 1,表示"这个 metatable 没有 第 `e` 号 tagmethod"。一旦查过一次发现没有 `__mode`,`luaT_gettm`(`ltm.c:60`)会把对应位置 1:

```c
const TValue *luaT_gettm (Table *events, TMS event, TString *ename) {
  const TValue *tm = luaH_Hgetshortstr(events, ename);
  lua_assert(event <= TM_EQ);
  if (notm(tm)) {  /* no tag method? */
    events->flags |= cast_byte(1u<<event);  /* cache this fact */
    return NULL;
  }
  else return tm;
}
```

注意 `luaT_gettm` 里有个断言 `event <= TM_EQ`——位缓存只对 `TM_INDEX..TM_EQ`(枚举值 0..5)有效。而 `TM_GC=2`、`TM_MODE=3` 都落在这个区间内(`ltm.h:18` 的枚举顺序),所以 `__gc` 和 `__mode` 的查询都吃这个快路径。绝大多数 table 没有 `__mode`,`checknoTM` 直接命中缓存返回 NULL,`getmode` 秒回 0。

### 2.4 三种遍历:strong / weakvalue / ephemeron

`getmode` 返回 0 走 `traversestrongtable`(`lgc.c:576`),返回 1 走 `traverseweakvalue`(`lgc.c:487`),返回 2 走 `traverseephemeron`(`lgc.c:541`),返回 3 不遍历内容、只挂链。三者差别就在"key 和 value 要不要被标活"。

#### strong table:正常的强引用

```c
static void traversestrongtable (global_State *g, Table *h) {
  Node *n, *limit = gnodelast(h);
  traversearray(g, h);
  for (n = gnode(h, 0); n < limit; n++) {  /* traverse hash part */
    if (isempty(gval(n)))  /* entry is empty? */
      clearkey(n);  /* clear its key */
    else {
      lua_assert(!keyisnil(n));
      markkey(g, n);
      markvalue(g, gval(n));
    }
  }
  genlink(g, obj2gco(h));
}
```

`traversearray`(`lgc.c:514`)扫 array 部分,把每个 GC 对象标活;然后扫 node 部分,每个非空 entry 同时标 key 和 value。这是强引用语义:**表里有的就让它活**。`clearkey`(`lgc.c:209`)是给空 entry 的"可回收 key"打 deadkey 标记(防止悬空 GC 指针破坏遍历链,但 entry 本身仍占位)。

#### weak value:只保 key,不保 value

```c
static void traverseweakvalue (global_State *g, Table *h) {
  Node *n, *limit = gnodelast(h);
  int hasclears = (h->asize > 0);
  for (n = gnode(h, 0); n < limit; n++) {
    if (isempty(gval(n)))
      clearkey(n);
    else {
      lua_assert(!keyisnil(n));
      markkey(g, n);  /* key 照常标活 */
      if (!hasclears && iscleared(g, gcvalueN(gval(n))))
        hasclears = 1;  /* value 是白的,待清 */
    }
  }
  if (g->gcstate == GCSpropagate)
    linkgclist(h, g->grayagain);  /* atomic 阶段重扫 */
  else if (hasclears)
    linkgclist(h, g->weak);       /* 有要清的 value,挂 weak 链 */
  else
    genlink(g, obj2gco(h));
}
```

关键差别:**key 用 `markkey` 标活,value 不标**——只检查 value 是不是白的(`iscleared`),是就记 `hasclears`。注意 array 部分"假设可能有白 value"(`hasclears = (h->asize > 0)`),懒得现在扫,留到 atomic 阶段统一清。

弱 value 表的清理推迟到 atomic:`traverseweakvalue` 在 propagate 阶段把表挂进 `grayagain`(atomic 重扫),在 atomic 阶段若有白 value 则挂进 `weak` 链(等 `clearbyvalues` 清理)。

#### ephemeron:弱 key,整个 entry 的命运系在 key 上

弱 key 表是最微妙的——它的语义是 **ephemeron(短命)语义**:**只有 key 活,value 才活;key 死,整个 entry(连同 value)都回收**。这和"弱 key 但强 value"完全不同:后者会让 value 把 key 强行拽活(违反弱的初衷)。Lua 的弱 key 表严格执行 ephemeron 语义。

`traverseephemeron`(`lgc.c:541`):

```c
static int traverseephemeron (global_State *g, Table *h, int inv) {
  int hasclears = 0;  /* true if table has white keys */
  int hasww = 0;  /* true if table has entry "white-key -> white-value" */
  unsigned int i;
  unsigned int nsize = sizenode(h);
  int marked = traversearray(g, h);  /* traverse array part */
  for (i = 0; i < nsize; i++) {
    Node *n = inv ? gnode(h, nsize - 1 - i) : gnode(h, i);
    if (isempty(gval(n)))
      clearkey(n);
    else if (iscleared(g, gckeyN(n))) {  /* key 未标活 */
      hasclears = 1;
      if (valiswhite(gval(n)))
        hasww = 1;  /* 白 key → 白 value,待收敛 */
    }
    else if (valiswhite(gval(n))) {  /* key 活,value 还白 */
      marked = 1;
      reallymarkobject(g, gcvalue(gval(n)));  /* 现在就标 value */
    }
  }
  if (g->gcstate == GCSpropagate)
    linkgclist(h, g->grayagain);
  else if (hasww)
    linkgclist(h, g->ephemeron);   /* 有 ww,需多轮收敛 */
  else if (hasclears)
    linkgclist(h, g->allweak);     /* 有死 key,待清 */
  else
    genlink(g, obj2gco(h));
  return marked;
}
```

仔细读这个分支:

1. **key 已经活**(`!iscleared(g, gckeyN(n))`)且 value 还白:立刻 `reallymarkobject(value)`。这是 ephemeron 的核心规则——**value 的活力来自 key**,key 活就把 value 拉活。
2. **key 还白**(`iscleared` 返回 1):不标 value,只记 `hasclears`;若 value 也白,再记 `hasww`(white→white)。

为什么 `hasww` 单独标记?因为 key 现在白,**不代表它最终会死**——可能在后续标记中,有别的强引用把 key 拉活了,key 一活,它对应的 value 也该活。所以"白 key → 白 value"的 entry 不能现在就判死,必须**多轮收敛**:`convergeephemerons`(`lgc.c:755`)会在 atomic 阶段反复扫 `ephemeron` 链,直到没有新对象被标活为止。

```c
static void convergeephemerons (global_State *g) {
  int changed;
  int dir = 0;
  do {
    GCObject *w;
    GCObject *next = g->ephemeron;
    g->ephemeron = NULL;
    changed = 0;
    while ((w = next) != NULL) {
      Table *h = gco2t(w);
      next = h->gclist;
      nw2black(h);
      if (traverseephemeron(g, h, dir)) {
        propagateall(g);  /* 把新标活的 value 传播出去 */
        changed = 1;
      }
    }
    dir = !dir;  /* 交替方向,加速链式收敛 */
  } while (changed);
}
```

`do...while(changed)`:只要某一轮标活了任何 value,就再扫一遍所有 ephemeron 表——因为那个新活的 value 可能是另一张表某个 entry 的 key。交替方向(`dir = !dir`)是对付"同一张表内 key↔value 链式依赖"的优化:正着扫一遍没收敛完,反着扫可能更快收敛。这套循环保证**所有该活的 entry 最终都被拉活,所有该死的 key 最终都进 `hasclears`**。

#### iscleared:什么算"该清"

`iscleared`(`lgc.c:223`)是判断"一个 GC 对象能否从弱表里清掉"的统一裁决:

```c
static int iscleared (global_State *g, const GCObject *o) {
  if (o == NULL) return 0;  /* non-collectable value */
  else if (novariant(o->tt) == LUA_TSTRING) {
    markobject(g, o);  /* strings are 'values', so are never weak */
    return 0;
  }
  else return iswhite(o);
}
```

两条特殊规则:

1. **非 GC 对象(NULL,如数字、布尔)永不清**——它们不是 GC 管的。
2. **字符串当弱引用时永不清**,而且还要**顺手把它标活**(`markobject(g, o)`)。

第 2 条值得展开。Lua 的设计决定是:**字符串在弱表里按 value 对待,不按 key 对待**。即便你写 `__mode="v"` 把一张表的 value 设成弱,如果 value 是字符串,它也不会被回收。为什么?因为 Lua 的字符串是**驻留的**(短串进 `strt` 哈希表,P1-03 详讲),如果允许弱表把字符串拽死,会出现"同一个字面量,有时在有时不在"的诡异现象。所以弱表对字符串豁免,且显式标活,避免它被其他弱项误伤。

(注:弱 key 表里,字符串 key 仍然可以死——因为字符串作为 key 时,它的死活由字符串表的整体驻留策略决定,弱 key 表只是不阻止它死。这里的不豁免只针对"字符串作为弱 value"。)

### 2.5 atomic 阶段:弱表清理的总枢纽

增量 GC 的标记阶段可以中断,但**清理弱表必须在某个不可中断的瞬间一次性完成**——否则清理到一半被中断,表的内部状态会不一致(一部分 entry 清了、一部分没清)。这个瞬间就是 `atomic`(`lgc.c:1543`):

```c
static void atomic (lua_State *L) {
  global_State *g = G(L);
  GCObject *origweak, *origall;
  GCObject *grayagain = g->grayagain;
  g->grayagain = NULL;
  lua_assert(g->ephemeron == NULL && g->weak == NULL);
  lua_assert(!iswhite(mainthread(g)));
  g->gcstate = GCSatomic;
  markobject(g, L);                    /* mark running thread */
  markvalue(g, &g->l_registry);
  markmt(g);                           /* mark global metatables */
  propagateall(g);                     /* empties 'gray' list */
  remarkupvals(g);
  propagateall(g);
  g->gray = grayagain;
  propagateall(g);                     /* traverse 'grayagain' list */
  convergeephemerons(g);
  /* Clear values from weak tables, before checking finalizers */
  clearbyvalues(g, g->weak, NULL);
  clearbyvalues(g, g->allweak, NULL);
  origweak = g->weak; origall = g->allweak;
  separatetobefnz(g, 0);               /* separate objects to be finalized */
  markbeingfnz(g);                     /* mark objects that will be finalized */
  propagateall(g);                     /* remark, to propagate 'resurrection' */
  convergeephemerons(g);
  /* remove dead objects from weak tables */
  clearbykeys(g, g->ephemeron);
  clearbykeys(g, g->allweak);
  /* clear values from resurrected weak tables */
  clearbyvalues(g, g->weak, origweak);
  clearbyvalues(g, g->allweak, origall);
  luaS_clearcache(g);
  g->currentwhite = cast_byte(otherwhite(g));  /* flip current white */
  lua_assert(g->gray == NULL);
}
```

这是全章最关键的函数,值得逐段拆。它把弱表清理和 finalizer 复活编织成一个**两阶段、防互相干扰**的流程。

**第 1 段(到第一次 `convergeephemerons`):把所有强可达对象标完。**

`markobject(g, L)` 重标运行线程(增量阶段线程可能又长出新栈元素);`markmt` 标全局 metatable 数组 `g->mt[]`(基本类型的元表可能被 C API 改);`propagateall` 把灰队列跑空。然后 `remarkupvals`(`lgc.c:406`)处理一个增量阶段遗留的边角:那些"线程本身没被标活、但它持有 open upvalue"的情况——upvalue 的值可能需要被标活(因为 upvalue 的值栈来源线程虽然要死,但 upvalue 已被别的活对象引用)。

接着把 `grayagain` 接回 `gray` 再 `propagateall`。`grayagain` 是什么?是**那些"propagate 阶段扫过一次、但 atomic 阶段必须再扫一次"的对象**——线程(栈会变)、弱 value 表(value 要现查死活)、弱 key 表(要走 ephemeron 收敛)。把它们在第一次扫描时挂进 `grayagain` 而不是直接涂黑,正是为了给 atomic 留下复访机会。

最后 `convergeephemerons`:第一轮 ephemeron 收敛。

**第 2 段(两次 `clearbyvalues` + `separatetobefnz` + `markbeingfnz` + 第二次 `convergeephemerons`):finalizer 复活。**

先 `clearbyvalues(g, g->weak, NULL)` 和 `clearbyvalues(g, g->allweak, NULL)`——清掉弱 value 表和全弱表里的死 value。`clearbyvalues`(`lgc.c:808`)遍历链上每张表的 array + node,凡是 `iscleared` 的 value,`setempty` 把 entry 置空:

```c
static void clearbyvalues (global_State *g, GCObject *l, GCObject *f) {
  for (; l != f; l = gco2t(l)->gclist) {
    Table *h = gco2t(l);
    Node *n, *limit = gnodelast(h);
    unsigned int i;
    unsigned int asize = h->asize;
    for (i = 0; i < asize; i++) {
      GCObject *o = gcvalarr(h, i);
      if (iscleared(g, o))
        *getArrTag(h, i) = LUA_VEMPTY;  /* remove entry */
    }
    for (n = gnode(h, 0); n < limit; n++) {
      if (iscleared(g, gcvalueN(gval(n))))
        setempty(gval(n));
      if (isempty(gval(n)))
        clearkey(n);
    }
  }
}
```

记下 `origweak`/`origall`(清理前的链头),后面要用。

然后 `separatetobefnz(g, 0)`(`lgc.c:1023`):把 `finobj` 链上**这一轮被判死(白)**的对象搬到 `tobefnz` 链尾:

```c
static void separatetobefnz (global_State *g, int all) {
  GCObject *curr;
  GCObject **p = &g->finobj;
  GCObject **lastnext = findlast(&g->tobefnz);
  while ((curr = *p) != g->finobjold1) {
    lua_assert(tofinalize(curr));
    if (!(iswhite(curr) || all))
      p = &curr->next;
    else {
      if (curr == g->finobjsur)
        g->finobjsur = curr->next;
      *p = curr->next;           /* 从 finobj 摘下 */
      curr->next = *lastnext;    /* 挂到 tobefnz 末尾 */
      *lastnext = curr;
      lastnext = &curr->next;
    }
  }
}
```

注意这里只遍历到 `g->finobjold1`(分代模式下的 OLD1 分界;增量模式 `finobjold1 == NULL`,遍历整条链)。`finobjold1` 之后的对象不会是白的(老对象必非白),省掉遍历。

紧接着 `markbeingfnz(g)`(`lgc.c:388`)把这些刚搬进 `tobefnz` 的对象**全部标活**:

```c
static void markbeingfnz (global_State *g) {
  GCObject *o;
  for (o = g->tobefnz; o != NULL; o = o->next)
    markobject(g, o);
}
```

这就是**复活**:对象在 `separatetobefnz` 里被判定为"这一轮该死"(白),但因为它带 `__gc`,GC 不能直接回收它——得先调它的 `__gc`。而调 `__gc` 时,对象的 `__gc` 函数可能重新引用这个对象(比如把它存进某个全局表)。所以 GC **保守地把所有待 finalize 的对象重新标活**,让它们至少活过这一轮,下一轮(如果还没人强引用)才真正回收。

`propagateall(g)` 把复活传播出去——`__gc` 里引用到的其他对象也得跟着活。然后再 `convergeephemerons(g)`:复活可能让某些原本要死的 key 重新活过来,对应的 ephemeron entry 也得重新评估。

**第 3 段(`clearbykeys` × 2 + `clearbyvalues` × 2):收尾清理。**

`clearbykeys(g, g->ephemeron)`(`lgc.c:789`)清掉弱 key 表里死 key 的 entry:

```c
static void clearbykeys (global_State *g, GCObject *l) {
  for (; l; l = gco2t(l)->gclist) {
    Table *h = gco2t(l);
    Node *limit = gnodelast(h);
    Node *n;
    for (n = gnode(h, 0); n < limit; n++) {
      if (iscleared(g, gckeyN(n)))  /* unmarked key? */
        setempty(gval(n));          /* remove entry */
      if (isempty(gval(n)))
        clearkey(n);
    }
  }
}
```

注意 `clearbykeys` 和 `clearbyvalues` 的顺序:先清 value(第 2 段开头),复活可能让一些 value 重新活,所以第 2 段末尾再 `clearbyvalues(g, g->weak, origweak)`——从 `origweak` 到当前链头这一段(复活过程中新加入 weak 链的表)再清一遍 value,但**不清复活之前已清过的部分**(用 `origweak` 作终点)。这个双清理是防止"复活把已死的 value 又拽回来时漏清"。

最后 `luaS_clearcache(g)` 清字符串 API 缓存(也可能持有死串引用),`g->currentwhite = otherwhite(g)` 翻转双白——下一轮分配的对象用另一种白,这一轮的白对象在下一轮就是死色,等待 sweep 回收。

### 2.6 finalizer `__gc`:复活与二次回收

对象的 `__gc` 元方法是怎么挂上 finalizer 机制的?入口在 `luaC_checkfinalizer`(`lgc.c:1068`),由 C API `lua_setmetatable` 调用(`lapi.c:981`/`989`):

```c
void luaC_checkfinalizer (lua_State *L, GCObject *o, Table *mt) {
  global_State *g = G(L);
  if (tofinalize(o) ||                 /* obj. is already marked... */
      gfasttm(g, mt, TM_GC) == NULL ||    /* or has no finalizer... */
      (g->gcstp & GCSTPCLS))               /* or closing state? */
    return;  /* nothing to be done */
  else {  /* move 'o' to 'finobj' list */
    GCObject **p;
    if (issweepphase(g)) {
      makewhite(g, o);
      if (g->sweepgc == &o->next)
        g->sweepgc = sweeptolive(L, g->sweepgc);
    }
    else
      correctpointers(g, o);
    for (p = &g->allgc; *p != o; p = &(*p)->next) { }
    *p = o->next;          /* 从 allgc 摘下 */
    o->next = g->finobj;   /* 挂到 finobj 头 */
    g->finobj = o;
    l_setbit(o->marked, FINALIZEDBIT);  /* 标记"已进 finalizer 流程" */
  }
}
```

逻辑:对象一旦被设上有 `__gc` 的 metatable,就从 `allgc` 链搬到 `finobj` 链,并置 `FINALIZEDBIT` 位(`tofinalize(o)` 测这个位,`lgc.h:96`)。从此它走独立的回收路径——sweep 时不和普通对象一起被 `freeobj`,而是先进 `tobefnz`、被 `__gc` 调用后,下一轮才回收。

`__gc` 的真正调用发生在 GC 状态机的 `GCScallfin` 阶段(`singlestep` @ `lgc.c:1669`):

```c
case GCScallfin: {  /* call finalizers */
  if (g->tobefnz && !g->gcemergency && luaD_checkminstack(L)) {
    g->gcstopem = 0;
    GCTM(L);  /* call one finalizer */
    stepresult = CWUFIN;
  }
  else {
    g->gcstate = GCSpause;
    stepresult = step2pause;
  }
  break;
}
```

每个 GC step 只调**一个** `__gc`(`GCTM`),避免 finalizer 调用串成一个大停顿。`GCTM`(`lgc.c:968`):

```c
static void GCTM (lua_State *L) {
  global_State *g = G(L);
  const TValue *tm;
  TValue v;
  lua_assert(!g->gcemergency);
  setgcovalue(L, &v, udata2finalize(g));
  tm = luaT_gettmbyobj(L, &v, TM_GC);
  if (!notm(tm)) {
    TStatus status;
    lu_byte oldah = L->allowhook;
    lu_byte oldgcstp  = g->gcstp;
    g->gcstp |= GCSTPGC;  /* avoid GC steps */
    L->allowhook = 0;
    setobj2s(L, L->top.p++, tm);   /* push finalizer... */
    setobj2s(L, L->top.p++, &v);   /* ... and its argument */
    L->ci->callstatus |= CIST_FIN;
    status = luaD_pcall(L, dothecall, NULL, savestack(L, L->top.p - 2), 0);
    L->ci->callstatus &= ~CIST_FIN;
    L->allowhook = oldah;
    g->gcstp = oldgcstp;
    if (l_unlikely(status != LUA_OK)) {
      luaE_warnerror(L, "__gc");
      L->top.p--;
    }
  }
}
```

`udata2finalize`(`lgc.c:947`)把对象从 `tobefnz` 摘下、**挂回 `allgc`**(这就是"复活"在数据结构层面的落地——对象回到了普通回收链):

```c
static GCObject *udata2finalize (global_State *g) {
  GCObject *o = g->tobefnz;
  lua_assert(tofinalize(o));
  g->tobefnz = o->next;
  o->next = g->allgc;
  g->allgc = o;
  resetbit(o->marked, FINALIZEDBIT);  /* "normal" again */
  if (issweepphase(g))
    makewhite(g, o);
  else if (getage(o) == G_OLD1)
    g->firstold1 = o;
  return o;
}
```

`resetbit(o->marked, FINALIZEDBIT)` 清掉 finalizer 标记——意思是"这个对象的 `__gc` 已经调过了,下次再死就直接回收,不再二次 finalize"。这是"复活一次"的精确含义:**带 `__gc` 的对象最多被 finalize 一次,finalize 后即使又被 `__gc` 代码引用,下次死时也不再调 `__gc`**,直接走普通 sweep。

调用 `__gc` 期间,`g->gcstp |= GCSTPGC` 暂停 GC 步进(防止 finalizer 里的内存分配触发新的 GC step,把正在 finalize 的对象又搅进新一轮标记),`L->allowhook = 0` 关 debug hook(finalizer 不该被 hook 打断)。`luaD_pcall` 保护调用,`__gc` 报错不会传播出来,只 `luaE_warnerror` 发个警告。

### 2.7 sweep:真正释放

标记和弱表清理都做完后,进入 sweep 阶段(`GCSswpallgc` → `GCSswpfinobj` → `GCSswptobefnz` → `GCSswpend`,见 `singlestep` @ `lgc.c:1648-1668`)。sweep 走 `allgc` / `finobj` / `tobefnz` 三条链,对每个对象用 `sweeplist`(`lgc.c:892`)判死活:

```c
static GCObject **sweeplist (lua_State *L, GCObject **p, l_mem countin) {
  global_State *g = G(L);
  int ow = otherwhite(g);
  int white = luaC_white(g);
  while (*p != NULL && countin-- > 0) {
    GCObject *curr = *p;
    int marked = curr->marked;
    if (isdeadm(ow, marked)) {  /* 是"另一种白"=这一轮的死色 */
      *p = curr->next;
      freeobj(L, curr);
    }
    else {
      curr->marked = cast_byte((marked & ~maskgcbits) | white | G_NEW);
      p = &curr->next;
    }
  }
  return (*p == NULL) ? NULL : p;
}
```

`otherwhite(g)`(`lgc.h:98`)给出"这一轮的死白位"(当前白的反色)。对象的 marked 里若含死白位,就是这轮该死的——`freeobj`(`lgc.c:836`)按类型分派释放(Table 调 `luaH_free`、Proto 调 `luaF_freeproto`、字符串要从 `strt` 表摘下再 free,等等)。活的对象则重染成**新当前白**、age 重置为 `G_NEW`,准备进入下一轮。

sweep 是增量的——每个 step 只扫 `GCSWEEPMAX=20`(`lgc.c:34`)个对象,把停顿摊平。这和 P5-16 讲的增量主线一脉相承。

## 三、为什么这样设计是 sound 的

这一节集中回答三件事:**为什么按类型分派遍历不漏标、为什么弱表清理不破坏不变式、为什么 finalizer 复活不悬空**。

### 3.1 类型分派遍历的完整性

GC soundness 的第一要求是**所有活对象都被标到**。漏标一个活对象 = 把它错判成死 = sweep 时 `freeobj` 释放它 = 后续访问悬空。

Lua 的保证来自两点:

1. **每个 GC 对象都在 `allgc`(或 `finobj`)链上**(`luaC_newobjdt` 头插)。sweep 扫这两条链,不会漏。
2. **每种类型的遍历函数扫尽了它的全部出边**。逐类核对:
   - `traversetable`:扫 array(`traversearray`)+ node(每个 entry 的 key 和 value)+ metatable。Table 持有 GC 对象的途径就这三处,全覆盖。
   - `traverseudata`:扫 metatable + 全部 uservalue(`uv[0..nuvalue-1]`)。Udata 的出边全在这里。
   - `traverseproto`:扫 source + 常量表 `k` + upvalue 名 + 嵌套 proto `p` + 局部变量名。Proto 的全部 GC 引用都在这五个数组里。
   - `traverseLclosure`:扫 proto `p` + 全部 upvalue `upvals[0..nupvalues-1]`。闭包的子对象就这两个。
   - `traverseCclosure`:扫全部 C upvalue。
   - `traversethread`:扫值栈 `stack..top` 的活元素 + open upvalue 链 + atomic 阶段清死栈片段(`luaD_shrinkstack` + `setnilvalue`)。线程持有的 GC 对象要么在栈上、要么在 open upvalue 里。
   - 字符串/upvalue:无出边(字符串内容是裸字节;upvalue 的值在 `reallymarkobject` 里当场标),不需要 traverse 函数。

只要每类的出边扫全,从根集(main thread + registry + 全局 mt + tobefnz 里的待 finalize 对象,见 `restartcollection` @ `lgc.c:440`)出发的传递闭包就能被完整标黑。这是三色标记"不漏"的根基。

### 3.2 弱表不破坏不变式的关键:推迟到 atomic

弱表的核心矛盾是:**它的弱引用指向白对象是合法的,但三色不变式要求黑对象不能指向白对象**。如果让弱表在 propagate 阶段就涂黑,它持有的白 value 会立刻违例。

Lua 的解法是**把弱表的最终处理全部推迟到 atomic**(那个不可中断的瞬间):

- propagate 阶段扫到弱表,不涂黑,而是挂进 `grayagain`(`traverseweakvalue`/`traverseephemeron` 在 `gcstate == GCSpropagate` 时都 `linkgclist(h, g->grayagain)`)。
- atomic 阶段重扫这些表,此时所有强可达对象已标完,**弱引用指向的白对象是真的没有强引用了**(否则早就被标黑)。
- atomic 一次性 `clearbyvalues`/`clearbykeys` 把死 entry 清空,表涂黑。清空后表不再持有那些白对象,不变式恢复。

这个"推迟"是 sound 的关键:**判断"弱引用对象是否该死"必须在所有强引用都标完之后做**,否则会把"还没标到但其实是活的"对象误判死。增量阶段做不到这一点(标记随时可能被打断),所以必须留到 atomic。

### 3.3 ephemeron 收敛的正确性

弱 key 表的 ephemeron 语义("key 活则 value 活")引入了**间接依赖**:value 的死活不直接由根集决定,而由 key 的死活决定。而 key 的死活又可能被另一张表的 value 决定(那张表的 value 是这张表的 key)。这就形成了**依赖链**,甚至**依赖环**。

`convergeephemerons` 的 `do...while(changed)` 循环正是处理这个:**反复扫所有 ephemeron 表,每轮把"key 已活但 value 还白"的 value 标活并 `propagateall`,直到没有新对象被标活**。这是个不动点迭代——收敛时,所有"传递地被某个活 key 拉活"的 value 都已标黑,剩下的白 key 真的没有活 key 撑腰,可以安全清掉。

为什么是 sound 的?因为"key 活→value 活"这条规则的传递闭包,恰好就是 `convergeephemerons` 的不动点。迭代结束时:

- 任何活 key 对应的 value 都已标黑(被 `reallymarkobject(gcvalue(gval(n)))` 标了)。
- 任何白 key 都传递地没有任何活 key 支撑(否则它会被拉黑)。

所以 `clearbykeys` 清掉的白 key entry,都是**确定无活 key 支撑**的,清掉它们不会误杀任何活对象。交替方向(`dir = !dir`)只是加速收敛,不改变不动点的正确性。

### 3.4 finalizer 复活不悬空

finalizer 的麻烦在于:对象已被 `separatetobefnz` 判死(白),但调 `__gc` 时,`__gc` 代码可能重新引用它。如果 GC 在调 `__gc` 前就把对象释放了,`__gc` 拿到的就是个悬空指针。

Lua 的保证是**三道防线**:

1. **判死后立刻复活**:`separatetobefnz` 把白对象搬进 `tobefnz`,`markbeingfnz` 立刻把它们全标活。从这一刻起,这些对象在 GC 眼里是活的,sweep 不会动它们。
2. **sweep 跳过 tobefnz 的常规清理**:`tobefnz` 链有独立的 sweep 状态(`GCSswptobefnz`),且 `tobefnz` 里的对象都带着 `FINALIZEDBIT`,`sweeplist` 不会在常规 allgc 扫描中碰到它们。
3. **`__gc` 调用时对象已挂回 allgc**:`udata2finalize` 在调 `__gc` 前把对象从 `tobefnz` 摘下、挂回 `allgc`、清 `FINALIZEDBIT`。这意味着即使 `__gc` 里又触发了 GC,对象也在 `allgc` 上被正常标记保护。

**"复活一次"的精确边界**:`udata2finalize` 里 `resetbit(o->marked, FINALIZEDBIT)` 清掉了 finalizer 标记。所以如果 `__gc` 代码把这个对象存进某个全局表(真正强引用),它就继续活;如果 `__gc` 返回后没人引用它,下一轮 GC 它会作为**普通 allgc 对象**被 sweep(因为已无 `FINALIZEDBIT`,`luaC_checkfinalizer` 也不会再把它搬回 `finobj`——`tofinalize(o)` 为假但 `gfasttm(g, mt, TM_GC)` 非空时本来会再搬,但 `luaC_checkfinalizer` 只在 `lua_setmetatable` 时调用,普通 sweep 路径不调)。于是**带 `__gc` 的对象最多被 finalize 一次**,不会无限复活。

(补充:`__gc` 代码里若再次 `lua_setmetatable` 给同一个对象换 metatable,且新 metatable 仍有 `__gc`,`luaC_checkfinalizer` 会因为 `tofinalize(o)` 已被 reset 而再次把它搬进 `finobj`——这是"换元表可以重新注册 finalizer"的合理行为,不算违反"只 finalize 一次"。)

### 3.5 open upvalue 保持灰色的用意

`reallymarkobject` 里 open upvalue 被 `set2gray` 而非涂黑,注释说"to avoid barriers"。展开讲:open upvalue 的值 `uv->v.p` 指向某线程的栈槽,那个栈槽随时可能被 `SETTABLE`/`MOVE` 之类改写。如果 upvalue 涂黑,一旦栈槽被改写成新白对象,就得触发 `luaC_barrier_` 把白对象标活——但 upvalue 不是常规的"黑→白"屏障触发点(它没有 `gclist` 进灰队列的标准路径)。保持灰色,upvalue 永远在灰态,它的值会在 `traversethread` 或 `remarkupvals` 里被复访标活,不需要 barrier。这是个"用一点重复扫描换掉 barrier 复杂度"的精简选择,和 Lua 一贯的"简单优先"一致。

## 四、★对照 CPython:weakref 与 `__del__`

把 Lua 的弱表 + `__gc` 放到 CPython 旁边,能看清两条路线的相对位置。

### 4.1 弱引用机制

| 维度 | Lua 5.5 | CPython |
|---|---|---|
| **弱引用的载体** | Table 的 key/value(通过 `__mode` 元字段声明) | 独立的 `weakref` 对象(`weakref.ref(o)`)+ `WeakValueDictionary`/`WeakKeyDictionary`/`WeakSet` |
| **声明方式** | `setmetatable(t, {__mode="k"})` / `"v"` / `"kv"` | 显式构造 `weakref.ref(obj)` 或用专用容器类 |
| **回收粒度** | entry 级:弱表里单条 entry 在 atomic 阶段被清 | 引用级:单个 weakref 在被指对象回收时变 dead |
| **回调通知** | 无(对象死了,弱表 entry 静默消失) | `weakref.ref` 可带 callback,对象回收时被调 |
| **字符串豁免** | 字符串作弱 value 永不回收(`iscleared` 显式标活) | 字符串同样不被弱引用回收(CPython 字符串驻留 + 弱引用对字符串返回 dead 但字符串本身因驻留仍在) |

关键差别在**架构**:

- **Lua 把弱引用做成 Table 的一种属性**,不引入新类型。一个 Table 是不是弱表,完全由它的 metatable 的 `__mode` 决定(`getmode` 现查)。这是"统一 Table"主线的延伸——弱表不是新数据结构,是普通 Table 的一种回收策略。源码层面,弱表的全部逻辑只是 GC 在 `traversetable` 里多了三个分支(`traverseweakvalue`/`traverseephemeron`/全弱)+ atomic 里的清理,没有新的用户可见类型。
- **CPython 把弱引用做成独立对象和独立容器类**。`weakref.ref` 是个一级对象,有自己的 C 实现;`WeakValueDictionary` 是 `dict` 的包装,内部维护 `self.data` 到 weakref 的映射。弱引用和 dict 是两套机制,各自一套代码。

Lua 这条路的代价:弱表只能以"整张表所有 entry 都弱"的粒度声明,不能像 CPython 那样"单个引用是弱的、放进普通 dict";且回调能力弱(没有 `__del__` 之外的弱引用回调)。回报:数据结构和 GC 代码都更少——又一个"统一换小"。

### 4.2 ephemeron 语义

Lua 的弱 key 表严格执行 ephemeron 语义(key 死则 value 死),由 `convergeephemerons` 的不动点迭代保证。CPython 的 `WeakKeyDictionary` 也是 ephemeron 语义,但实现方式不同:**CPython 依赖引用计数**——key 的 refcount 归零时,CPython 立刻回调 weakref,weakref 把 `WeakKeyDictionary` 里对应 entry 删掉。因为引用计数是即时的,不需要"多轮收敛";但因为引用计数不处理循环引用,`WeakKeyDictionary` 的 key 若陷入循环引用,得等分代 GC 周期才能回收(那时才触发 weakref 回调)。

Lua 没有 refcount,全靠标记清除的 `convergeephemerons` 一次性收敛——**循环引用也能被正确处理**(标记清除天然识别循环不可达)。这是标记清除相对引用计数在弱表语义上的一个优势:不依赖额外的循环检测。

### 4.3 finalizer

| 维度 | Lua 5.5 `__gc` | CPython `__del__` |
|---|---|---|
| **适用对象** | userdata(及有 metatable 的 table,5.4+ 也支持 table 的 `__gc`) | 任何定义了 `__del__` 的类实例 |
| **调用时机** | GC 的 `GCScallfin` 阶段,每 step 调一个 | 引用计数归零时**立刻**调 |
| **复活** | 显式支持:对象标活、挂回 allgc、下轮再回收 | 历史上支持(对象在 `__del__` 里被重新引用会复活),但 CPython 3.4+ 引入 `pep 442` 后复活仍可行但有警告;不会二次调 `__del__` |
| **异常处理** | `__gc` 报错被吞,只发 warning(`luaE_warnerror`) | `__del__` 异常打印到 stderr 但被忽略 |
| **顺序保证** | 无特定顺序(按 `tobefnz` 链顺序) | 无特定顺序(按 refcount 归零时机) |

最深的差别在**调用时机与复活**:

- CPython 的 `__del__` 在 refcount 归零的瞬间调,语义接近"对象正在被销毁"。复活虽然在语言层面可行,但被强烈不推荐(CPython 文档明确说 resurrection 是"不可预测的")。
- Lua 的 `__gc` 在独立的 `GCScallfin` 阶段调,和对象的"逻辑死亡"(被标记为白)是**两个分开的事件**——先在 atomic 阶段判死并复活,稍后在 `GCScallfin` 调 `__gc`,再下轮才真正回收。这种分离让复活成为**一等公民**:Lua 程序惯用 `__gc` 做"对象被 GC 前的最后一次清理",且清楚知道对象此时还在内存里、可被安全访问。

两种设计反映了两种 GC 哲学:CPython 的 refcount 让 finalizer 极及时但复活棘手;Lua 的标记清除让 finalizer 延迟但复活干净。对嵌入式场景(Lua 的主场),延迟 + 干净复活更可控——finalizer 不会在任意内存分配点把控制权抢走,只在 GC 的专属阶段跑。

### 4.4 回扣主线

把弱表和 `__gc` 放回"统一与精简换小而快"的主线上看:

- **统一**:弱表不是新类型,是普通 Table 加 `__mode` 元字段后的回收策略;`__gc` 不是新机制,是 metatable 元方法之一。两者都寄生在已有的"Table + metatable"统一数据模型上,没有为它们引入新的用户可见类型。源码里,弱表的全部实现是 `lgc.c` 里几个遍历分支 + atomic 清理,`__gc` 的全部实现是 `finobj`/`tobefnz` 两条链 + `GCTM`。
- **精简**:弱表清理推迟到 atomic 一个瞬间完成,避免了在增量阶段维护"弱引用对象可能死"的复杂状态;finalizer 的复活机制用"标活 + 挂回 allgc + 清 FINALIZEDBIT"三步搞定,不需要专门的不死区。每一步都在复用已有的三色标记基础设施。

代价是:弱表清理集中在 atomic 会增加这一个瞬间的工作量(虽然 atomic 本身要求尽量短);finalizer 的延迟回收让带 `__gc` 的对象比普通对象多活一轮。这些都是"用一点点延迟换大幅简化的机制"的取舍,和 Lua 一贯的"宁可稍慢也要小而简单"一脉相承。

弱表和 `__gc` 讲完,GC 篇的增量三色主线就闭环了:从根集出发的可达对象被标黑、弱表的死 entry 在 atomic 被清、带 finalizer 的对象被复活一次后下轮回收。剩下的问题是——**能不能让年轻对象死得更快,不必每次都跑完整的三色轮?** 这正是分代 GC 要回答的,下一章见。
