# P3-12 寄存器式精解:为什么比栈式指令少

> **本章主线**:统一与精简换小而快——这是执行侧(P3)的收尾,也是"精简换快"在指令模型上最直接的一组证据。都说 Lua 寄存器式字节码比 Python 栈式更快、更紧凑,但"快"到底快在哪里?"省指令"到底省了几条、省在什么操作上?本章用同一段代码同时反编译 Lua 5.5 和 CPython 3.11+,逐组数指令,把"精简换快"从口号落成可数的事实。**二分法**:执行侧(P3)收尾,汇总 P3-10(指令格式)与 P3-11(解释器循环)。**★对照**:本章整章即对照——CPython 栈式字节码(LOAD_FAST/LOAD_CONST/STORE_FAST/BINARY_OP/BINARY_SUBSCR/LOAD_ATTR/CALL)。**源码**:lua-5.5.0(`lopcodes.h`/`lopnames.h`/`lvm.c`/`lcode.c`)。CPython 用 3.11+ 公开字节码描述(本地无 Python 源码,以 CPython 公开指令集为准)。**基调**:纯直球,不用比喻。

## 一、这章解决什么问题

P0-01 给过一个对照:同样写 `local b = a + 2`,Lua 编出两条核心指令,CPython 编出七八条。当时只给了结论,没数清楚。本章要把这件事彻底数明白。

问题不是"寄存器式和栈式哪个好"——这个结论业界早有共识,脚本语言里 Lua、LuaJIT、Dalvik、V8 的 Ignition、Erlang BEAM 都偏向寄存器式,而老牌的 JVM、CPython、WebAssembly 偏向栈式,各有各的道理。真正值得问的是三个更具体的问题:

1. **省下来的指令,到底省的是什么操作?** 是省了算术本身,还是省了"搬值上下栈"这种和业务逻辑无关的搬运?如果省的是搬运,那"省指令"和"快"之间的因果链就清楚了:脚本语言的解释器开销大头不在那条算术指令本身(那只是一次 CPU 加法),而在每条指令都要经历的**取指、译码、分发**这一套固定开销。少一条指令,就少一次取指译码分发;省下来的不是运算,是解释器的呼吸。
2. **省下来的幅度有多大?** 是省一两条还是省一大半?对不同结构(算术、字段访问、表写、调用、拼接)省的比例一样吗?
3. **寄存器式有没有代价?** 栈式凭什么还活着?如果寄存器式全方位碾压,为什么 CPython、JVM 不换?一定有它付的代价,这个代价要诚实讲清。

本章的讲法很直接:挑五种最常见的代码结构,每种给一段最短的 Lua 代码,同时给出它在 Lua 5.5 和 CPython 3.11+ 上的大致字节码,逐条数,做对照表。然后从机制上解释指令数差异从哪里来,各自付什么代价。最后把 Lua 放进"寄存器 vs 栈"的业界谱系里定位。

一个前置说明:本章反编译用"大致编译成"的措辞。Lua 5.5 的精确反编译以 `luac -l` 为准,CPython 以 `dis.dis()` 为准;不同小版本在指令选择上会有细微差异(尤其 CPython 3.11 把 `BINARY_ADD` 重构为 `BINARY_OP`,3.12 又调整了 `LOAD_ATTR`),但指令条数级差稳定。本章的对照表反映的是 5.5.0 与 CPython 3.11+ 的典型输出。

## 二、前置:寄存器式与栈式的本质区别

在数指令之前,先把两种指令模型的定义钉死。差异不在"有没有寄存器",而在**操作数怎么给定**。

**栈式(stack-based)**:指令不带显式操作数,操作数隐含在"一个共享的求值栈的栈顶"。要做 `a + b`,必须先用 `LOAD a` 把 `a` 压到栈顶,再用 `LOAD b` 把 `b` 压到栈顶,然后发一条加法指令——这条加法指令本身没有操作数,它默认从栈顶弹出两个值相加,把结果压回栈顶。最后要存结果,再发一条 `STORE` 把栈顶弹出来写进变量。每一次"值从一个地方挪到求值栈"都是一条独立的指令(`LOAD`/`STORE`),每一次"用栈顶的值"都隐含一次弹栈。CPython、JVM、WebAssembly 都是这种。

**寄存器式(register-based)**:指令的操作数直接是"寄存器号"。Lua 里每个函数有一组自己的寄存器,本质是它那段值栈上的槽位(P4-13 详讲调用栈与寄存器的关系)。要做 `a + b`,一条指令就能说"`R_c = R_a + R_b`"——它同时完成了"取操作数、运算、存结果"三件事,中间值根本不经过一个独立的求值栈来回。Lua、LuaJIT、Dalvik、V8 Ignition、Erlang BEAM 都是这种。

注意一个容易被老资料带偏的点:所谓"寄存器"在 Lua 里**不是 CPU 寄存器**,而是 Lua 解释器自己维护的、函数私有的值栈槽位。每个 Lua 函数被调用时,会在所属线程的值栈上划出 `MAX_FSTACK`(`lopcodes.h:210`)以内的若干个槽位给自己用,这些槽位的编号就是寄存器号。这个上限是 255:

```c
/* lopcodes.h:106 */
#define MAXARG_A	((1<<SIZE_A)-1)   /* SIZE_A = 8, 所以 MAXARG_A = 255 */
...
/* lopcodes.h:206-215 */
/*
** Maximum size for the stack of a Lua function. It must fit in 8 bits.
** The highest valid register is one less than this value.
*/
#define MAX_FSTACK	MAXARG_A
/*
** Invalid register (one more than last valid register).
*/
#define NO_REG		MAX_FSTACK
```

因为寄存器号 A 占 8 位(`SIZE_A = 8`,`lopcodes.h:47`),一个函数最多有 255 个槽位,最高有效寄存器号是 254(`NO_REG = 255` 表示"无效")。扣掉参数、upvalue 临时占用,实际可用的局部变量/临时寄存器通常在 200 左右。这就是"Lua 单函数寄存器数有上限(约 250)"的物理来源——它是 32 位指令位宽的直接约束。P2-09 讲过,这是寄存器式付的代价之一,本章第三节会再权衡。

把定义钉死后,下面进入对照实验。每组都按这个结构走:给出 Lua 源码 → Lua 5.5 大致字节码(标注操作码在 `lopcodes.h` 的真实语义注释)→ CPython 3.11+ 大致字节码 → 数指令 → 对照表 → 点出差异来源。

## 三、对照实验:五种代码结构逐组数指令

### 第 1 组:二元算术 `local b = a + 2`(立即数加法)

最简单的起点。Lua 源码:

```lua
local a = 1
local b = a + 2
```

**Lua 5.5 大致编译成**(操作码语义见 `lopcodes.h:236` `OP_LOADI`、`lopcodes.h:261` `OP_ADDI`):

```
LOADI   R0  1        -- R0 := 1            (a)
ADDI    R1  R0  2     -- R1 := R0 + 2       (b)
MMBINI  R1  R0  2  0  -- 元方法兜底(下文解释)
RETURN0
```

这里要先讲一个 5.5 的关键设计,否则后面所有算术对照都会看不懂。`ADDI` 是"乐观算术":它假设 `R0` 是数字(整数或浮点),直接走快速路径相加。如果 `R0` 真是数字,相加成功,**解释器在执行时会 `pc++` 跳过紧跟的 `MMBINI`**(因为 `op_arith_aux` 在 `lvm.c:992` 整型快路径成功分支里有 `pc++`),元方法兜底那条指令形同虚设。如果 `R0` 不是数字(比如是个表,定义了 `__add` 元方法),`ADDI` 的快速路径失败,**不跳 `pc++`**,于是下一条 `MMBINI` 被执行,它去调对应的元方法。

这是 5.5 相对老资料(讲 5.3/5.4 的书/博客)的一个**硬差异**:老资料里算术指令的常量操作数靠 RK 位编码(操作数高位为 1 表示这是个常量索引而非寄存器号),一条 `OP_ADD` 既能加寄存器也能加常量。**5.5 把这条路拆了**:算术改成专用指令——`OP_ADD`(寄存器+寄存器,`lopcodes.h:278`)、`OP_ADDK`(寄存器+常量表,`lopcodes.h:263`)、`OP_ADDI`(寄存器+小整数立即数,`lopcodes.h:261`)三套,各自配一条 `MMBIN`/`MMBINK`/`MMBINI` 兜底。常量加用 `ADDK`,小整数加用 `ADDI`,纯寄存器加用 `ADD`。编译器在 `lcode.c` 里按操作数类型分流:`codebini`(`lcode.c:1521`)发 `ADDI`,`codebinK`(`lcode.c:1533`)发 `ADDK`,`codebinexpval`(`lcode.c:1506`)发 `ADD`。

所以严格数指令:Lua 这段是 **2 条业务指令**(`LOADI` + `ADDI`)+ 1 条几乎总是被跳过的元方法兜底(`MMBINI`)+ 1 条 `RETURN0`。真正在热路径上跑的是 2 条。

**CPython 3.11+ 大致编译成**:

```
RESUME        0
LOAD_CONST    1 (1)      ; 压入整数 1
STORE_FAST    0 (a)      ; 弹出,存入局部变量 a
LOAD_FAST     0 (a)      ; 把 a 压回栈顶
LOAD_CONST    2 (2)      ; 压入整数 2
BINARY_OP     0 (+)      ; 弹出栈顶两个,相加,结果压回
STORE_FAST    1 (b)      ; 弹出结果,存入局部变量 b
RETURN_CONST  0 (None)
```

数一下:`LOAD_CONST` + `STORE_FAST` + `LOAD_FAST` + `LOAD_CONST` + `BINARY_OP` + `STORE_FAST` = **6 条业务指令**(另加 `RESUME` 和 `RETURN_CONST` 两条框架指令)。其中真正"做事"的只有 `BINARY_OP` 一条,其余 5 条全是**搬运**:`LOAD_CONST` 把字面量从常量表搬上栈,`STORE_FAST` 把栈顶搬进局部变量槽,`LOAD_FAST` 又把局部变量搬回栈。`a` 这个值先被 `STORE_FAST` 从栈顶搬走,紧接着又被 `LOAD_FAST` 搬回来——来回白搬了一次。

对照表:

| | Lua 5.5 | CPython 3.11+ |
|---|---|---|
| 业务指令 | 2(LOADI + ADDI) | 6(LOAD_CONST + STORE_FAST + LOAD_FAST + LOAD_CONST + BINARY_OP + STORE_FAST) |
| 其中"搬运"指令 | 0 | 5(LOAD_CONST × 2 + STORE_FAST × 2 + LOAD_FAST) |
| 其中"运算"指令 | 1(ADDI) | 1(BINARY_OP) |
| 元方法兜底 | 1(MMBINI,几乎总被 pc++ 跳过) | 0(元方法在 BINARY_OP 内部分支处理) |

差异的来源一眼可见:**CPython 那 5 条搬运指令,在 Lua 里一条都没有**。Lua 的 `ADDI R1 R0 2` 把"取 R0、加 2、存 R1"三件事压进了一条 32 位指令;CPython 必须把 `a`、`2` 分别压上栈、用完再分别弹下栈,每次压弹都是一条指令。寄存器式省的就是这些搬运。

### 第 2 组:寄存器+寄存器加法 `local c = a + b`

立即数加法是最优情况(右操作数是字面量,能塞进 8 位立即数 `sC`)。现在看右操作数也是变量的情况,这是寄存器式最"正常"的形态。Lua 源码:

```lua
local a = 1
local b = 2
local c = a + b
```

**Lua 5.5 大致编译成**(`OP_ADD` 语义见 `lopcodes.h:278`):

```
LOADI   R0  1
LOADI   R1  2
ADD     R2  R0  R1     -- R2 := R0 + R1
MMBIN   R2  R0  6      -- __add 兜底(TM_ADD = 6)
RETURN0
```

业务指令 3 条(2 条 LOADI + 1 条 ADD),其中搬运 0 条、运算 1 条。注意 `ADD R2 R0 R1` 的三个操作数全是寄存器号,一条指令就把"取 R0、取 R1、相加、存 R2"四件事做完。

**CPython 3.11+ 大致编译成**:

```
RESUME        0
LOAD_CONST    1 (1)
STORE_FAST    0 (a)
LOAD_CONST    2 (2)
STORE_FAST    1 (b)
LOAD_FAST     0 (a)
LOAD_FAST     1 (b)
BINARY_OP     0 (+)
STORE_FAST    2 (c)
RETURN_CONST  0 (None)
```

业务指令 8 条,其中搬运 7 条(`LOAD_CONST` × 2 + `STORE_FAST` × 3 + `LOAD_FAST` × 2)、运算 1 条。

对照表:

| | Lua 5.5 | CPython 3.11+ |
|---|---|---|
| 业务指令 | 3 | 8 |
| 搬运指令 | 0 | 7 |
| 运算指令 | 1 | 1 |

这一组更能体现差异:CPython 为了把 `a` 和 `b` 都送到栈顶供 `BINARY_OP` 取用,各发了一条 `LOAD_FAST`;而这两个值此前刚被 `STORE_FAST` 写进局部变量槽——存进去马上又读出来,这是栈式模型天然的来回开销。Lua 的 `a`、`b` 自打被 `LOADI` 写进 `R0`、`R1` 就待在那里不动,`ADD` 直接用,没有任何重复搬运。

### 第 3 组:字段读 `local x = t.k`

算术是最直接的对照,字段访问更能看出两种模型对"复合操作"的不同切分。Lua 源码:

```lua
local t = { k = 10 }
local x = t.k
```

**Lua 5.5 大致编译成**(`OP_GETFIELD` 语义见 `lopcodes.h:250`):

```
NEWTABLE  R0  0  1       -- R0 := {}(预分配 1 个哈希槽)
LOADI     R1  0          -- 临时(下面 EXTRAARG 模式忽略,简化表示)
...                       -- 实际 5.5 用 SETFIELD 写入 k=10,此处简化
GETFIELD  R1  R0  K0     -- R1 := R0[K0],K0 是字符串常量 "k"
MMBIN     R1  R0  K0     -- __index 兜底(若 t 无 k 且有元表)
RETURN0
```

精确反编译会更长(建表要 `NEWTABLE` + 写入字段),但聚焦在"读字段"这一步:`GETFIELD R1 R0 K0` 一条指令完成。注意 5.5 的一个细节——`GETFIELD` 的 C 操作数必须是**短字符串常量**(`lopcodes.h:250` 注释 `K[C]:shortstring`),不是任意常量。编译器在 `lcode.c:1265` 有 `isKstr` 检查:只有当键是短字符串常量时才发 `GETFIELD`,否则退化成 `GETTABLE`(键在寄存器里)。这是 5.5 的优化——字符串字段访问是 Lua 里最高频的操作(面向对象 `obj.method`、配置 `cfg.option` 全走这条路),给它配一条专用指令、键直接编进指令字,省一次寄存器分配。

**CPython 3.11+ 大致编译成**(建表部分简化,聚焦读字段):

```
BUILD_MAP        1             ; 建一个 dict,预分配 1 项
LOAD_CONST       1 (10)        ; 压入值 10
LOAD_CONST       2 ('k')       ; 压入键 'k'
BUILD_TUPLE      1             ; 组成 ('k', 10) 写入 map(具体指令因版本而异)
STORE_FAST       0 (t)
LOAD_FAST        0 (t)         ; 把 t 压上栈
LOAD_ATTR        2 ('k')       ; 从栈顶对象取属性 'k',结果压回栈顶
STORE_FAST       1 (x)         ; 弹出存入 x
```

CPython 3.12+ 把 `LOAD_ATTR` 做了进一步优化(会把"取值还是取方法"的判断也压进这一条),但骨架不变:`LOAD_FAST` 把表压栈 → `LOAD_ATTR` 取字段 → `STORE_FAST` 弹栈存结果,三步。

聚焦"读字段"一步的对照:

| | Lua 5.5 | CPython 3.11+ |
|---|---|---|
| 读字段指令 | 1(GETFIELD) | 3(LOAD_FAST + LOAD_ATTR + STORE_FAST) |
| 其中搬运 | 0 | 2(LOAD_FAST + STORE_FAST) |

差异和算术组同构:CPython 必须先把表搬上栈、字段读出来再搬下栈;Lua 的表在 `R0` 不动,`GETFIELD` 直接把结果写进 `R1`。

### 第 4 组:表写 `t[i] = v`

读字段是表当源,写表是表当目的,指令切分更复杂,差异更明显。Lua 源码:

```lua
local t = {}
local v = 42
t[1] = v
```

**Lua 5.5 大致编译成**(`OP_SETI` 语义见 `lopcodes.h:254`):

```
NEWTABLE  R0  0  0       -- R0 := {}
LOADI     R1  42         -- R1 := 42 (v)
SETI      R0  1  R1  0   -- R0[1] := R1(k 位=0 表示 R1 是寄存器)
MMBIN     R0  1  R1      -- __newindex 兜底
RETURN0
```

聚焦"写表"这一步:`SETI R0 1 R1 0` 一条指令。这里要看清 5.5 的 RK 机制——`SETI` 的 C 操作数既可以是寄存器号也可以是常量号,由 k 位区分(`lopcodes.h:222` 注释 `RK(x) == if k(i) then K[x] else R[x]`)。编译器在 `lcode.c:1095` 的 `codeABRK` 里调用 `exp2RK`(`lcode.c:1085`)决定右值塞寄存器还是常量:如果右值是个能放进常量表的表达式且索引在 `MAXINDEXRK` 以内,就当常量编进 C 操作数并置 k 位,省掉一条把右值搬进寄存器的指令。

注意一个 5.5 vs 老资料的微妙点:**5.5 只有"写表"系列(SETTABLE/SETI/SETFIELD/SETTABUP)和 SELF 还保留 RK 位**;读表和算术都改成了专用 K 指令(GETFIELD/GETI/GETTABLE 各自独立,ADDK/SUBK/... 各自独立)。老资料常把 RK 说成 5.3/5.4 的通用机制,5.5 已经收窄了它的使用范围。

**CPython 3.11+ 大致编译成**(聚焦写表):

```
BUILD_MAP        0             ; R0 := {}
STORE_FAST       0 (t)
LOAD_CONST       1 (42)
STORE_FAST       1 (v)
LOAD_FAST        0 (t)         ; 压入 t(目的对象)
LOAD_FAST        1 (v)         ; 压入 v(要写的值)
LOAD_CONST       2 (1)         ; 压入键 1
STORE_SUBSCR                  ; t[栈顶-1] = 栈顶
```

CPython 的 `STORE_SUBSCR` 是一条三操作数指令(从栈顶依次取键、值、对象),它本身比 `LOAD_ATTR` 信息量大,但前面仍需三条 `LOAD` 把三个操作数都搬上栈。

聚焦"写表"一步的对照:

| | Lua 5.5 | CPython 3.11+ |
|---|---|---|
| 写表指令 | 1(SETI) | 4(LOAD_FAST + LOAD_FAST + LOAD_CONST + STORE_SUBSCR) |
| 其中搬运 | 0 | 3(三条 LOAD) |

Lua 的 `SETI R0 1 R1 0` 把"表、键、值"三个操作数外加一个 k 位全编进 32 位;CPython 必须把三者分别压栈。差距还是来自搬运。

### 第 5 组:函数调用 `local r = f(a, b)`

调用是最能体现寄存器式省指令的场景,因为一次调用涉及"函数对象 + 多个参数 + 多个返回值",信息量很大。Lua 源码:

```lua
local function f(x, y) return x + y end
local a = 1
local b = 2
local r = f(a, b)
```

聚焦"调用 `f(a, b)`"这一步。**Lua 5.5 大致编译成**(`OP_CALL` 语义见 `lopcodes.h:320`):

```
; f 已在 R0,a 已在 R1,b 已在 R2(参数紧挨函数放)
CALL  R0  3  2     -- 调用 R0(R1, R2),期望 1 个返回值(C=2 表示 C-1=1)
                  ; 返回值落在 R0..R0(即 R0 := f(a,b))
RETURN0
```

一条 `CALL R0 3 2` 把整个调用说清楚:`A=R0` 是函数对象所在寄存器,`B=3` 表示参数个数加 1(即 2 个参数,且参数必须在 `R1`、`R2` 连续排布),`C=2` 表示返回值个数加 1(即 1 个返回值,落在 `R0` 起)。这是寄存器式调用的精髓——**函数、参数、返回值全用寄存器号指代,一条指令编码整个调用的形状**。

执行侧(`lvm.c:1720`)的逻辑印证了这点:`vmcase(OP_CALL)` 里先 `int b = GETARG_B(i)`、`int nresults = GETARG_C(i) - 1`,如果 `b != 0` 就 `L->top.p = ra + b` 把参数个数告诉被调函数,然后 `luaD_precall` 进入新调用帧。整个过程没有一次"搬值上下栈"的额外指令。

**CPython 3.11+ 大致编译成**(聚焦调用):

```
LOAD_FAST        0 (f)       ; 压入函数对象
LOAD_FAST        1 (a)       ; 压入参数 a
LOAD_FAST        2 (b)       ; 压入参数 b
PRECALL          2           ; 3.11+ 的调用前置(检查可调用性等)
CALL             2           ; 调用,栈顶两个是参数,下面是函数
STORE_FAST       3 (r)       ; 弹出返回值存 r
```

CPython 3.11+ 把老的 `CALL_FUNCTION` 拆成了 `PRECALL` + `CALL`(3.12 又进一步调整),但骨架是:先 `LOAD` 函数、再逐个 `LOAD` 参数、再 `CALL`、最后 `STORE` 结果。N 个参数就要 N 条 `LOAD`。

对照表:

| | Lua 5.5 | CPython 3.11+ |
|---|---|---|
| 调用指令 | 1(CALL) | 5(LOAD_FAST×3 + PRECALL + CALL + STORE_FAST,实际 6 条) |
| 其中搬运 | 0 | 4(LOAD_FAST×3 + STORE_FAST) |

参数越多,这个差距越大。CPython 调用 `f(a, b, c, d)` 要 4 条 `LOAD`;Lua 永远是一条 `CALL`,只是 B 操作数从 3 变成 5。这是寄存器式在高元调用上最占便宜的地方。

### 第 6 组:字符串拼接 `local s = "x" .. y .. z`

这一组用来说明寄存器式还能把"批量操作"压成一条指令。Lua 源码:

```lua
local y = "hello"
local z = "world"
local s = "x" .. y .. z
```

**Lua 5.5 大致编译成**(`OP_CONCAT` 语义见 `lopcodes.h:301`):

```
LOADK    R0  K0       ; R0 := "hello" (y)
LOADK    R1  K1       ; R1 := "world" (z)
LOADK    R2  K2       ; R2 := "x"
MOVE     R3  R0       ; R3 := R0 (把 y 挪到拼接序列起点,编译器安排)
MOVE     R4  R1       ; R4 := R1 (z)
CONCAT   R2  3        -- R2 := R2 .. R3 .. R4(一条指令拼连续的 3 个寄存器)
MMBIN    R2  R0  9    -- __concat 兜底(TM_CONCAT)
RETURN0
```

精确反编译会因编译器对操作数顺序的安排略有不同(实际 5.5 编译器对 `..` 链有专门处理,见 `lcode.c:1777` 附近的 `OP_CONCAT` 发射逻辑),但关键点是:**`CONCAT R2 3` 一条指令,把从 `R2` 起连续 3 个寄存器(`R2`、`R3`、`R4`)全部拼接,结果落在 `R2`**。这是寄存器式"一条指令干一串事"的极致体现——操作数 B 是"参与拼接的寄存器个数",所以无论拼 2 个还是拼 20 个,都是一条 `CONCAT`(B 操作数变一下)。

执行侧(`lvm.c:1626` `vmcase(OP_CONCAT)`)调用 `luaV_concat(L, n)`(`lvm.c:684`),这个函数会从最高位的寄存器往低位扫,逐个拼接,中间遇到非字符串会触发 `__concat` 元方法。

**CPython 3.11+ 大致编译成**(简化表示,聚焦拼接):

```
LOAD_CONST       1 ('hello')
STORE_FAST       0 (y)
LOAD_CONST       2 ('world')
STORE_FAST       1 (z)
LOAD_CONST       3 ('x')
LOAD_FAST        0 (y)
BINARY_OP        0 (+)        ; 字符串拼接在 CPython 里也走 + 或专用 FORMAT/BUILD_STRING
LOAD_FAST        1 (z)
BINARY_OP        0 (+)
STORE_FAST       2 (s)
```

CPython 对字符串拼接有不同的优化路径(3.11+ 对 `f"..."` 用 `FORMAT_VALUE`,对显式 `..` 等价的 `+` 链用 `BINARY_OP`;某些版本会合并成 `BUILD_STRING`)。但无论走哪条,骨架都是:**每多拼一个字符串,就多一对 `LOAD + BINARY_OP`(或 `LOAD + FORMAT_VALUE`)**。拼 N 个字符串,搬运指令数线性增长。

对照表:

| | Lua 5.5 | CPython 3.11+ |
|---|---|---|
| 拼接 3 个串的指令 | 1(CONCAT,B=3) | 约 5(LOAD + BINARY_OP + LOAD + BINARY_OP + STORE_FAST) |
| 拼接 N 个串的指令 | 1(CONCAT,B=N) | 约 2N−1(线性增长) |

这一组最能看出寄存器式的设计取向:**它不只是省搬运,还能把"一个操作作用于一组连续操作数"压成一条**。栈式做不到这点,因为栈顶永远只有一个值,要做 N 元操作必须 N−1 次弹出压入。

### 对照小结

把六组数据汇总:

| 代码结构 | Lua 5.5 业务指令 | CPython 3.11+ 业务指令 | 倍数 |
|---|---|---|---|
| 立即数加法 `b = a + 2` | 2 | 6 | 3.0× |
| 寄存器加法 `c = a + b` | 3 | 8 | 2.7× |
| 字段读 `x = t.k` | 1(聚焦读) | 3(聚焦读) | 3.0× |
| 表写 `t[i] = v` | 1(聚焦写) | 4(聚焦写) | 4.0× |
| 调用 `r = f(a, b)` | 1(聚焦调用) | 6 | 6.0× |
| 拼接 `s = "x"..y..z` | 1(聚焦拼接) | 5 | 5.0× |

CPython 的指令数稳定是 Lua 的 3 到 6 倍。而且越往复杂结构(调用、拼接)走,倍数越大——因为栈式对"多操作数"的指令数是线性增长的,寄存器式是常数。这就是 P0-01 那句"寄存器式指令少、紧凑"的可数证据。

## 四、为什么 sound:指令数差异的机制根源

数完指令,要回答的是:这个 3 到 6 倍的差距,机制上从哪里来?为什么寄存器式必然省搬运?

根源在**操作数隐式 vs 显式**这一条。

栈式的求值栈是一个共享的、单端访问的结构。任何一条指令要用一个值,这个值必须在栈顶——而要把它弄到栈顶,就得有一条 `LOAD` 把它从某个地方(局部变量槽、常量表、对象属性)搬上来。任何一条指令产出一个值,这个值会落在栈顶——而要把它存走,又得有一条 `STORE` 把它从栈顶搬走。**每一次"值跨过求值栈的边界"都是一条独立的指令**。一个值从局部变量进入算术、再回到局部变量,要跨两次边界(`LOAD` 进、`STORE` 出),就是两条搬运指令。这些搬运指令和业务逻辑(算术本身)毫无关系,纯粹是栈式模型的"过路费"。

寄存器式没有这个共享求值栈。每个值待在自己的寄存器里,指令直接用寄存器号引用它——`ADD R2 R0 R1` 里的 `R0`、`R1` 就是值所在的地方,不必搬。结果 `R2` 也就是值落下的地方,不必再搬。**值从不跨边界,所以没有搬运指令**。寄存器式的全部指令都是"做事"的指令:算术、字段访问、调用、跳转——没有一条纯粹为了搬运而存在(除了 `MOVE`,但那是显式的变量赋值,不是模型强加的过路费)。

这就解释了为什么六组对照里,Lua 的搬运指令数恒为 0,CPython 的搬运指令数占了总指令数的大半(第 1 组 5/6、第 2 组 7/8、第 4 组 3/4、第 5 组 4/6)。**栈式的指令里有相当一部分是在"伺候栈",而不是在"算业务"。**

再往深一层:为什么"伺候栈"的指令会拖慢执行?因为脚本语言的解释器,每条指令的执行成本不是"那条指令做了多少运算",而是"取指 + 译码 + 分发"这一套固定开销。Lua 的解释器循环 `luaV_execute`(`lvm.c`)是一个大的 `switch`,每条指令都要走一遍:从 `pc` 取出 32 位指令 → 解码操作码 → 跳到对应 `vmcase` 分支 → 执行 → 回到循环顶。这套开销里,取指是内存访问(指令在 `Proto->code` 数组里)、译码是位运算和比较、分发是间接跳转(分支预测不友好)。**这部分开销和指令做的事无关,只和指令的条数有关**。少一条指令,就少一次这套开销。

所以寄存器式"省指令 = 快"的因果链是:省搬运指令 → 总指令数少 → 取指/译码/分发次数少 → 解释器开销低 → 执行快。省下来的不是 CPU 加法(CPU 加法一周期,谁都一样),是解释器自己每条指令的固定呼吸。

Lua 团队在《The Implementation of Lua 5.0》(Roberto Ierusalimschy, 2005)这篇论文里实测过:从 4.0 的栈式改成 5.0 的寄存器式后,在基准测试上提速明显(论文报告平均约 30%–50% 的提升,具体因测试而异),且字节码体积也没有变大反而略小——因为虽然单条指令信息量更大,但指令条数少得更多,总字节数下降。这是 Lua 选寄存器式的直接动机,也是后来 LuaJIT、Dalvik、V8 Ignition 纷纷跟进的经验依据。

## 五、寄存器式的代价(诚实讲)

上一节讲了寄存器式为什么省、为什么快。但栈式活了三十年没死(JVM、CPython、WebAssembly 都是),说明寄存器式一定付了代价。代价有三条,都要讲清。

### 代价 1:编译器要做寄存器分配

这是最显眼的代价。栈式编译器几乎不用动脑子:见到表达式就 `LOAD` 压栈,见到运算符就发运算指令弹栈相加,结果自然在栈顶。它不需要决定"这个中间结果放哪个槽"——栈顶就是它的位置,天经地义。

寄存器式编译器必须做寄存器分配:决定每个局部变量占哪个寄存器、每个中间表达式结果落在哪个寄存器、用完何时释放。教科书式的寄存器分配是图着色,是 NP 难问题。Lua 当然不做图着色——它用线性分配(P2-09 详讲):局部变量按声明顺序占低槽位,临时表达式结果从 `freereg` 往后顺序占用,用完即释放。整个 `lcode.c` 在 5.5.0 是 1971 行,寄存器分配相关的逻辑(`luaK_reserveregs` `lcode.c:488`、`freereg` `lcode.c:499`、`luaK_exp2nextreg` `lcode.c:999`)加起来也就几百行。

但这仍是栈式没有的复杂度。CPython 的编译器(`compile.c`)不需要维护 `freereg`、不需要决定表达式结果落哪个槽——它只需要维护一个"当前栈深度"用于跳转修正。所以**寄存器式把一部分复杂度从执行期挪到了编译期**:编译器更复杂,执行器更简单(因为不必每条指令都弹栈压栈)。

这个取舍对 Lua 是赚的:脚本程序的编译只发生一次(或者被缓存成字节码后零次),但执行可能发生千万次。编译期多花的几十微秒,换来执行期每条逻辑省下的几条指令,长期收益巨大。

### 代价 2:寄存器号占操作数位宽,单函数寄存器数有上限

寄存器号要编进 32 位指令字。Lua 的 A 操作数是 8 位(`SIZE_A = 8`,`lopcodes.h:47`),所以一个函数最多 255 个寄存器(`MAX_FSTACK = MAXARG_A = 255`,`lopcodes.h:210`)。扣掉参数、闭包 upvalue 占用,实际可用的局部变量和临时寄存器大约 200 个。

绝大多数函数远远用不到这个上限——一个普通函数十几个局部变量绰绰有余。但确实存在极端情况:一个自动生成的、有几百个局部变量的函数(比如某些 DSL 生成的查表代码),会撞上这个限制,Lua 会报 "too many local variables" 错误。这是寄存器式的硬约束,栈式没有(栈式的栈深度只受内存限制)。

不过这个限制在实践中几乎不构成问题——任何有几百个局部变量的函数本身就需要重构。Lua 的设计哲学就是"为正常代码优化,不为病态代码留口子"。

### 代价 3:单条指令信息量大,对位宽和译码更挑剔

寄存器式的指令要编码多个操作数(`ADD R2 R0 R1` 要编三个寄存器号),所以单条指令的位宽需求比栈式大。Lua 用 32 位字长,刚够编出 `iABC` 格式(Op 7 + A 8 + k 1 + B 8 + C 8 = 32 位,见 `lopcodes.h:21`)。这迫使人手不够时做一些妥协:比如寄存器号只能 8 位(255 上限)、立即数只能 8 位有符号(`sC`,范围约 ±127)、常量索引也受 8 位限制(超了要用 `EXTRAARG` 扩展,多一条指令)。

CPython 的指令字宽是 16 位(操作码 8 位 + 操作数 8 位,3.6+ 用"指令 + 缓存"的 wordcode 格式,每条指令 2 字节,操作数可以跨多条指令扩展)。栈式只需要编码"做什么"(操作码)和"对谁"(一个操作数,通常是局部变量号或常量索引),所以 16 位够用,字节码更紧凑(按字节算)。

所以一个微妙的权衡:**寄存器式按条数算省(指令少),但按字节算不一定省**(单条 32 位 vs 单条 16 位)。Lua 论文实测字节码体积也没变大,是因为条数少得足够多,抵消了单条变大的影响。但如果一个程序的逻辑主要是"顺序赋值"(每条赋值在两种模型下都差不多是一条指令),寄存器式的字节码可能反而略大。这是真实存在的边界情况。

## 六、栈式的优势(给 CPython 说话)

讲完寄存器式的代价,要把天平摆平:栈式有自己的、不可替代的优势,这就是它至今没被淘汰的原因。

### 优势 1:编译器实现极简

栈式编译器是递归下降最自然的搭档。每解析一个表达式节点,就 `LOAD` 操作数、发运算指令,不需要任何状态追踪(除了一个用于跳转修正的栈深度计数器)。整个 CPython 的编译器(`compile.c`)在概念上就是把 AST 走一遍、每个节点发对应指令,逻辑极其线性。

这对一门要快速演进、快速加新特性的语言是巨大的红利。CPython 每个版本都在加新语法(async/await、海象运算符、模式匹配、except* 等),栈式编译器让这些新增特性的编译逻辑都很直白——加一个 AST 节点、对应一段 emit 代码,完事。如果 CPython 是寄存器式,每加一个特性都要考虑它怎么和现有的寄存器分配交互,工程负担重得多。

### 优势 2:指令位宽小,字节码紧凑

上一节说过,栈式指令 16 位,寄存器式 32 位。在"按字节传输/存储字节码"的场景(网络分发、嵌入式刷写),栈式的字节码体积优势明显。JVM 的 class 文件要在网络上传输,WebAssembly 的 wasm 要尽量小,这两个场景栈式都是合理选择。

注意这不是绝对的——Lua 论文证明寄存器式的总字节码体积也能更小(条数少抵消单条大)。但在"指令密度均匀、没有大量可省搬运"的程序上,栈式按字节算是省的。

### 优势 3:语义简单,验证/反编译/JIT 都更容易

栈式字节码的语义是"线性流":顺着 pc 走,每条指令的效果就是"压几个、弹几个、算一个",很容易静态验证(检查栈深度是否平衡、类型是否匹配)。JVM 的字节码验证器就依赖这点——它能静态保证一段字节码不会栈下溢或类型错配。

寄存器式就难得多:寄存器号的合法性依赖运行时上下文(这个寄存器此刻是否已被赋值),静态验证要跟踪每个寄存器的活跃状态,接近于做一遍抽象解释。这也是为什么 JVM 选择栈式(要保证跨机器安全加载任意 class 文件),而 Lua 不在乎(它的字节码从不跨机器分发,只在本地编译本地执行)。

所以两种模型各有其合理的生态位:**栈式适合"要简单编译、要紧凑传输、要安全验证"的场景;寄存器式适合"要执行快、编译在本地、不需要跨机分发"的场景**。Lua 是后者,所以选寄存器式是必然。

---

## 七、★对照 CPython:把 Lua 放进寄存器 vs 栈的业界谱系

本章整章都是对照,这里做收束性的定位。

先看业界谱系。寄存器式 VM 的代表:

- **Lua 5.0+**:最早把脚本语言 VM 从栈式改成寄存器式的主流实现(2003 年的 5.0,论文 2005 年发表),并且公开了实测数据,直接影响了后来的设计者。
- **LuaJIT**:Lua 的另一个实现,寄存器式 + 追踪 JIT,性能比官方 Lua 高一个数量级。
- **Dalvik**(Android 的 VM,被 ART 取代前):明确是寄存器式,设计文档里直接引用了 Lua 5.0 论文作为依据。Android 团队的理由和 Lua 一样:寄存器式指令少、解释执行快(早期 Android 设备 CPU 弱、没有 JIT,解释器性能生死攸关)。
- **V8 Ignition**(Chrome/Node.js):Google 在 2016 年把 V8 的解释器从基于栈的 Crankshaft 改成寄存器式的 Ignition,理由同样是减少解释器开销、让字节码更紧凑(Ignition 的字节码也是寄存器式,每个函数有一组累加器和寄存器)。
- **Erlang BEAM**:寄存器式,每个函数有一组虚拟寄存器(x/y registers)。

栈式 VM 的代表:

- **JVM**:Java 字节码是栈式,理由前述(跨机器安全验证、字节码紧凑传输)。
- **CPython**:Python 字节码是栈式,理由前述(编译器简单、快速演进)。
- **WebAssembly**:栈式,理由是验证简单 + 体积小(浏览器要快速下载和验证任意来源的 wasm)。
- **.NET CLR**:栈式(虽然有自己的优化,但基础模型是栈式)。

一个有趣的观察:**早期(1990s–2000s)的脚本/字节码 VM 多是栈式**,因为那时候编译器简单性、字节码可移植性是首要考量(JVM 的跨平台承诺是核心卖点)。**2000s 中期之后新设计或大改的 VM 多是寄存器式**(Lua 5.0、Dalvik、Ignition、BEAM 的演进),因为那时候硬件变快、JIT 普及,但**解释器本身的取指/译码/分发开销**成了新瓶颈,减少指令数成了提速的主战场。Lua 5.0 的论文在这个转向里是里程碑式的——它用实测数据证明了"对解释执行的脚本语言,寄存器式值得",后来者多多少少都受了它的影响。

把 Lua 放在这个谱系里:它是"寄存器式脚本 VM"的开创者和最纯粹的样本。LuaJIT、Dalvik、Ignition 都在它之后,且都比它复杂(LuaJIT 加了 JIT、Dalvik 面向移动端、Ignition 是巨型 V8 的一部分)。Lua 是其中最小、最干净的一个——整个解释器循环 `luaV_execute` 在 5.5.0 的 `lvm.c` 里不到 2000 行,每个 `vmcase` 分支都短到能一眼看完。这使得它成为理解"寄存器式 VM 到底怎么工作"的最佳教学样本:没有 JIT 的干扰、没有内联缓存的复杂、没有多层的解释器 tiers,就是最朴素的 switch 分发 + 寄存器号操作数。

回到和 CPython 的对照。两者是同生态位(嵌入式脚本语言 VM)的正反两面:

| 维度 | Lua 5.5 | CPython 3.11+ |
|---|---|---|
| 指令模型 | 寄存器式,32 位字 | 栈式,16 位 wordcode |
| 同段代码指令数 | 1× | 3–6× |
| 搬运指令占比 | ~0 | 大半 |
| 编译器复杂度 | 中(要做寄存器分配,~2000 行 lcode.c) | 低(只压栈弹栈,但 compile.c 因语言特性也不小) |
| 字节码跨机分发 | 不支持(本地编译本地执行) | 不强调(虽然有 .pyc,但同版本) |
| 解释器开销 | 低(指令少) | 高(指令多,但 3.11+ 有内联缓存和专用化指令优化) |

CPython 团队并非不知道寄存器式的优势——事实上 CPython 3.11 引入的"专用化指令"(specializing adaptive interpreter,把 `LOAD_ATTR`、`BINARY_OP` 等替换成针对常见类型的快速版本)就是在不改变栈式骨架的前提下,用"内联缓存"减少每条指令的译码和类型检查开销,部分弥补栈式的指令数劣势。这是一种"在栈式框架里挤性能"的路线,和 Lua"换模型省指令"是两条不同的路。两者都在演进,各自在自己的约束里做到最好。

## 八、收束:精简换快在指令模型上的最终落点

本章把 P0-01 的那句"寄存器式比栈式指令少"数清楚了。六个对照实验的数据稳定显示:同一段代码,Lua 5.5 的指令数是 CPython 3.11+ 的 1/3 到 1/6,而且省下来的全是和业务无关的搬运指令。差异的机制根源是操作数隐式(栈式)vs 显式(寄存器式)——栈式每跨一次求值栈边界就要一条指令,寄存器式的值从不跨边界,所以没有搬运。

这件事不是免费的。Lua 付了三条代价:编译器要做寄存器分配(但只付一次,执行期一直省)、寄存器号占 8 位所以单函数上限 255(实践中几乎不构成约束)、单条指令信息量大所以位宽 32 位(但总字节码体积因条数少而更小)。栈式也用自己的优势(编译器简单、字节码紧凑、易验证)证明它没被淘汰是有道理的。

把这个取舍放进全书的主线:**Lua 用编译期的一次寄存器分配,换执行期每条逻辑的更少指令**。这正是"精简换快"在指令模型上的具体含义——多花一点编译期的功夫,把执行期能省的搬运全砍掉。而执行期省下的不是 CPU 运算(那是一周期的加法,谁都一样),是解释器每条指令的取指、译码、分发这一套固定呼吸。脚本语言的瓶颈从来不在运算本身,而在解释器的呼吸次数。Lua 把呼吸次数压到最低,这就是它"快得不像脚本"的指令层根源。

P3 这一篇(P3-10 指令格式、P3-11 解释器循环、P3-12 本章)到此收束。接下来 P4 进入函数、闭包与调用——本章数过的 `CALL` 指令,在 P4-13 里会被拆开讲清楚:`luaD_precall` 怎么建立新的调用帧、参数怎么从调用者的寄存器搬到被调函数的寄存器(其实不搬,只是改了 base 指针)、返回值怎么落回。寄存器式调用的"一条 CALL 搞定一切",在 P4-13 里会看到它在执行侧是怎么被一层调用栈机制支撑的。
