# P6-20 协程:协作式多任务的栈切换

> **本书主线**:统一与精简换小而快。**二分法**:编译侧(源码→字节码)↔ 执行侧(VM 执行 + 管值生命周期)。**本章落点**:执行侧——协程是多个独立 `lua_State` 共享一个 `global_State` 时的栈切换,是"少量机制换多任务"的极致体现。**★对照**:CPython 的 generator + asyncio。**源码**:lua-5.5.0。**基调**:纯直球,不用比喻。

---

## 一、这章解决什么问题

Lua 是一门单线程的语言——整个内核里没有一行创建 OS 线程的代码,`pthread_create`、`CreateThread` 都不出现。可它却有一套完整的"并发"原语:`coroutine.create`、`coroutine.resume`、`coroutine.yield`、`coroutine.wrap`、`coroutine.status`。一个协程能跑到一半主动让出,把控制权交回它的调用者,过一会儿又被恢复继续跑,中间还能在两边来回传值。

这就抛出本章要回答的三个具体问题:

1. **没有 OS 线程,协程到底"并发"在哪?** 答案是:它不并发,它是**协作式**(cooperative)的。多个协程共用一个 OS 线程,谁在跑谁就独占这个线程;只有正在跑的那个主动调 `coroutine.yield`,控制权才会转给别人。没有抢占(preemption),没有时间片轮转,没有调度器在背后打断。
2. **resume/yield 是怎么在"两个执行流"之间切换的?** 既然没有 OS 线程切换(那要保存寄存器、切内核上下文),那 Lua 到底切的是什么?答案是:切 **`lua_State`**。每个协程是一个独立的 `lua_State`(源码里叫 thread),有自己的值栈和调用链。resume 把当前正在跑的 `lua_State` 换成协程的那个,yield 再换回来。这一切换纯粹是 C 函数调用的进入与返回,不涉及任何内核态。
3. **为什么 Lua 的手册反复强调"不能跨 C 调用边界 yield"?** 一个用 `lua_pushcfunction` 注册的 C 函数,在它内部直接调 `lua_yield` 会报错 `attempt to yield across a C-call boundary`。这条约束的物理根源是什么?为什么 CPython 的 generator 没有这条限制,而 Lua 的协程有?

要回答这三个问题,得先回到 P1-02 看过的那个布局:一个 `global_State` 可以挂多个 `lua_State`。协程的全部机制都建立在这个布局之上。下面进源码。

## 二、源码怎么实现

### 2.1 协程的物质基础:一个 `global_State` 多个 `lua_State`

先复习 P1-02 的核心结构。`lua_State`(`lstate.h:285`)是"每线程"状态:

```c
struct lua_State {
  CommonHeader;
  lu_byte allowhook;
  TStatus status;
  StkIdRel top;  /* first free slot in the stack */
  struct global_State *l_G;
  CallInfo *ci;  /* call info for current function */
  StkIdRel stack_last;  /* end of stack (last element + 1) */
  StkIdRel stack;  /* stack base */
  UpVal *openupval;  /* list of open upvalues in this stack */
  StkIdRel tbclist;  /* list of to-be-closed variables */
  GCObject *gclist;
  struct lua_State *twups;  /* list of threads with open upvalues */
  struct lua_longjmp *errorJmp;  /* current error recover point */
  CallInfo base_ci;  /* CallInfo for first level (C host) */
  ...
  l_uint32 nCcalls;  /* number of nested non-yieldable or C calls */
  ...
};
```

注意三组关键字段。第一组是值栈:`stack`/`stack_last`/`top`,这是一个协程自己独立的、动态增长的 `StackValue` 数组。第二组是调用链:`ci` 指向当前正在执行的那个 `CallInfo`(`base_ci` 是根,代表 C 宿主那一层),每个 `CallInfo` 记录一个函数调用的 `func`/`top`/`savedpc`。第三组是 `l_G`,指向全局状态。

`global_State`(`lstate.h:327`)是"进程级"状态,被同一进程里所有 `lua_State` 共享:内存分配器、字符串驻留表、GC 状态机、注册表、基本类型的元表。它里面还有一个内嵌的 `mainth`:

```c
typedef struct global_State {
  lua_Alloc frealloc;  /* function to reallocate memory */
  void *ud;
  ...
  stringtable strt;  /* hash table for strings */
  TValue l_registry;
  ...
  struct lua_State *twups;  /* list of threads with open upvalues */
  ...
  LX mainth;  /* main thread of this state */
} global_State;
```

`lua_newstate`(`lstate.c:336`)只创建一个 `global_State`,主线程就是 `g->mainth.l`(`lstate.c:342`)。协程就是在这个 `global_State` 之下,再创建出额外的 `lua_State`。每个新 `lua_State` 有**自己的栈和调用链**,但**共用 `global_State` 里的字符串表、GC、注册表、内存分配器**。

这是协程能存在的物质基础:既然每个 `lua_State` 都是自洽的一套执行状态(有栈、有调用链、有 `savedpc`),那把"当前在跑哪个 `lua_State`"换一下,就等于换了一个执行流。两个执行流之间共享的、不变的东西(字符串、GC、内存堆),正好放在 `global_State` 里,不用随切换而搬动。

P0-01 里留过一个伏笔:"可以共享同一个 `global_State` 的多个 `lua_State`"——那正是协程的物理基础。这里落实。

### 2.2 创建一个协程:`lua_newthread` + `coroutine.create`

`coroutine.create(f)`(`lcorolib.c:96`)做三件事:检查参数是函数,调 `lua_newthread` 建一个新 `lua_State`,把函数搬进新 `lua_State` 的栈底。

```c
static int luaB_cocreate (lua_State *L) {
  lua_State *NL;
  luaL_checktype(L, 1, LUA_TFUNCTION);
  NL = lua_newthread(L);
  lua_pushvalue(L, 1);  /* move function to top */
  lua_xmove(L, NL, 1);  /* move function from L to NL */
  return 1;
}
```

核心是 `lua_newthread`(`lstate.c:273`):

```c
LUA_API lua_State *lua_newthread (lua_State *L) {
  global_State *g = G(L);
  GCObject *o;
  lua_State *L1;
  lua_lock(L);
  luaC_checkGC(L);
  /* create new thread */
  o = luaC_newobjdt(L, LUA_TTHREAD, sizeof(LX), offsetof(LX, l));
  L1 = gco2th(o);
  /* anchor it on L stack */
  setthvalue2s(L, L->top.p, L1);
  api_incr_top(L);
  preinit_thread(L1, g);
  L1->hookmask = L->hookmask;
  L1->basehookcount = L->basehookcount;
  L1->hook = L->hook;
  resethookcount(L1);
  /* initialize L1 extra space */
  memcpy(lua_getextraspace(L1), lua_getextraspace(mainthread(g)),
         LUA_EXTRASPACE);
  luai_userstatethread(L, L1);
  stack_init(L1, L);  /* init stack */
  lua_unlock(L);
  return L1;
}
```

逐行看。

`luaC_newobjdt(L, LUA_TTHREAD, sizeof(LX), offsetof(LX, l))`:在 GC 堆里分配一个 `LX`(`lstate.h:318`)。`LX` 是 `lua_State` 加上用户可配置的 `LUA_EXTRASPACE` 的包装结构:

```c
typedef struct LX {
  lu_byte extra_[LUA_EXTRASPACE];
  lua_State l;
} LX;
```

新分配的对象的类型标记是 `LUA_TTHREAD`(8,见 `lua.h:72`),所以一个协程在 Lua 里是一个一等(first-class)值,可以存进 Table、传给函数、被 GC 回收。

`preinit_thread(L1, g)`(`lstate.c:225`)做不分配内存的预初始化——把 `L1->l_G` 设成和 `L` 同一个 `g`,清空 `ci`/`nci`/`hook` 等:

```c
static void preinit_thread (lua_State *L, global_State *g) {
  G(L) = g;
  L->stack.p = NULL;
  L->ci = NULL;
  L->nci = 0;
  L->twups = L;  /* thread has no upvalues */
  L->nCcalls = 0;
  L->errorJmp = NULL;
  L->hook = NULL;
  L->hookmask = 0;
  L->basehookcount = 0;
  L->allowhook = 1;
  resethookcount(L);
  L->openupval = NULL;
  L->status = LUA_OK;
  L->errfunc = 0;
  L->oldpc = 0;
  L->base_ci.previous = L->base_ci.next = NULL;
}
```

`G(L) = g` 是关键一行——**新协程共享主线程的 `global_State`**,字符串、GC、注册表、内存分配器全共用。

`stack_init(L1, L)`(`lstate.c:158`)才真正分配新协程自己的值栈:

```c
static void stack_init (lua_State *L1, lua_State *L) {
  int i;
  /* initialize stack array */
  L1->stack.p = luaM_newvector(L, BASIC_STACK_SIZE + EXTRA_STACK, StackValue);
  L1->tbclist.p = L1->stack.p;
  for (i = 0; i < BASIC_STACK_SIZE + EXTRA_STACK; i++)
    setnilvalue(s2v(L1->stack.p + i));  /* erase new stack */
  L1->stack_last.p = L1->stack.p + BASIC_STACK_SIZE;
  /* initialize first ci */
  resetCI(L1);
  L1->top.p = L1->stack.p + 1;  /* +1 for 'function' entry */
}
```

`BASIC_STACK_SIZE` 是 `2*LUA_MINSTACK = 40`(`lstate.h:156`、`lua.h:79`),加 `EXTRA_STACK = 5`(`lstate.h:142`)。注意分配栈用的 `luaM_newvector` 的第一个参数是 `L`(创建者)而不是 `L1`——因为此刻 `L1` 的 GC 还没完全建好,内存分配出错时要由 `L` 的错误处理机制兜底。

`resetCI(L1)`(`lstate.c:146`)把调用链重置成只有一个根帧:

```c
static void resetCI (lua_State *L) {
  CallInfo *ci = L->ci = &L->base_ci;
  ci->func.p = L->stack.p;
  setnilvalue(s2v(ci->func.p));  /* 'function' entry for basic 'ci' */
  ci->top.p = ci->func.p + 1 + LUA_MINSTACK;  /* +1 for 'function' entry */
  ci->u.c.k = NULL;
  ci->callstatus = CIST_C;
  L->status = LUA_OK;
  L->errfunc = 0;
}
```

栈底第 0 槽留给"function entry"——这是 C 宿主那一层的虚拟函数位置,目前是 nil。协程第一次 `resume` 时,`coroutine.create` 塞进来的那个 Lua 函数会放到这里。

回到 `luaB_cocreate`。建完 `lua_newthread` 后,`lua_pushvalue(L, 1)` 把参数里的函数复制到 `L` 栈顶,`lua_xmove(L, NL, 1)` 把它从 `L` 的栈搬到 `NL`(新协程)的栈。`lua_xmove`(`lapi.c:126`)是两个共享同一 `global_State` 的 `lua_State` 之间的值搬运:

```c
LUA_API void lua_xmove (lua_State *from, lua_State *to, int n) {
  int i;
  if (from == to) return;
  lua_lock(to);
  api_checkpop(from, n);
  api_check(from, G(from) == G(to), "moving among independent states");
  api_check(from, to->ci->top.p - to->top.p >= n, "stack overflow");
  from->top.p -= n;
  for (i = 0; i < n; i++) {
    setobjs2s(to, to->top.p, from->top.p + i);
    to->top.p++;
  }
  lua_unlock(to);
}
```

`api_check(from, G(from) == G(to), ...)`——只有共享同一 `global_State` 的两个 `lua_State` 才能 `xmove`。这是 `xmove` 与"跨进程拷贝"的根本区别:字符串、Table、函数这些 GC 对象的指针在两个 `lua_State` 之间直接通用,因为它们都指向同一个 GC 堆。

此刻新协程 `NL` 的栈状态:栈底 `func` 槽放着一个 Lua 函数,`top` 指向栈底 +1(函数之上),`status == LUA_OK`,`ci == &base_ci`。协程处于"刚创建,还没跑"的初始状态。

### 2.3 协程的状态:`auxstatus` 四态

`coroutine.status` 通过 `luaB_costatus`(`lcorolib.c:150`)返回状态字符串,真正判定在 `auxstatus`(`lcorolib.c:128`):

```c
#define COS_RUN		0
#define COS_DEAD	1
#define COS_YIELD	2
#define COS_NORM	3

static const char *const statname[] =
  {"running", "dead", "suspended", "normal"};

static int auxstatus (lua_State *L, lua_State *co) {
  if (L == co) return COS_RUN;
  else {
    switch (lua_status(co)) {
      case LUA_YIELD:
        return COS_YIELD;
      case LUA_OK: {
        lua_Debug ar;
        if (lua_getstack(co, 0, &ar))  /* does it have frames? */
          return COS_NORM;  /* it is running */
        else if (lua_gettop(co) == 0)
            return COS_DEAD;
        else
            return COS_YIELD;  /* initial state */
      }
      default:  /* some error occurred */
        return COS_DEAD;
    }
  }
}
```

这里有个常被老资料混淆的点:**Lua 协程的状态机不是靠 `status` 字段的一个枚举,而是靠 `lua_State::status`(`TStatus`)+ 调用栈是否空的组合判定**。`lua_status` 返回的就是 `L->status`,可能的值见 `lua.h:48-53`:`LUA_OK=0`、`LUA_YIELD=1`、`LUA_ERRRUN=2`、`LUA_ERRSYNTAX=3`、`LUA_ERRMEM=4`、`LUA_ERRERR=5`。

四个状态怎么落:

- **running**(`COS_RUN`):查询的协程就是当前正在跑的 `L`。
- **suspended**(`COS_YIELD`):两种情况合一种——`status == LUA_YIELD`(挂起过),或 `status == LUA_OK` 但栈非空(刚 `create` 完,还没第一次 `resume`)。后者是"初始状态",栈底放着函数但还没调用过。
- **normal**(`COS_NORM`):`status == LUA_OK` 且 `lua_getstack(co, 0, &ar)` 能取到第 0 层帧。意思是它此刻正在跑——只是不是查询者自己。当一个协程 A `resume` 了协程 B,B 在跑的时候,B 看 A 的状态就是 normal。
- **dead**(`COS_DEAD`):`status == LUA_OK` 且栈空(`lua_gettop(co) == 0`,函数跑完返回了,栈被清空),或者 `status` 是任何错误码(出错死掉)。

注意 `lua_State::status` 本身只存 `LUA_OK`/`LUA_YIELD`/错误码四种取值;**"running/normal" 这两个状态不是字段,是从调用关系推出来的**。这是 5.5(以及 5.x)的状态机设计:用最小字段(一个 `TStatus` 字节)表达四个逻辑状态。

### 2.4 resume 协程:`lua_resume` 全程

`coroutine.resume(co, ...)`(`lcorolib.c:57`)走 `luaB_coresume` → `auxresume` → `lua_resume`。`auxresume`(`lcorolib.c:33`)负责搬运参数和结果:

```c
static int auxresume (lua_State *L, lua_State *co, int narg) {
  int status, nres;
  if (l_unlikely(!lua_checkstack(co, narg))) {
    lua_pushliteral(L, "too many arguments to resume");
    return -1;  /* error flag */
  }
  lua_xmove(L, co, narg);
  status = lua_resume(co, L, narg, &nres);
  if (l_likely(status == LUA_OK || status == LUA_YIELD)) {
    if (l_unlikely(!lua_checkstack(L, nres + 1))) {
      lua_pop(co, nres);  /* remove results anyway */
      lua_pushliteral(L, "too many results to resume");
      return -1;  /* error flag */
    }
    lua_xmove(co, L, nres);  /* move yielded values */
    return nres;
  }
  else {
    lua_xmove(co, L, 1);  /* move error message */
    return -1;  /* error flag */
  }
}
```

`lua_xmove(L, co, narg)` 把 resume 的参数从调用者 `L` 搬到协程 `co`。`lua_resume(co, L, narg, &nres)` 是真正干活的——**注意 5.5 的签名有四个参数**(见 2.7 的 5.5 vs 老资料差异)。返回后,如果是正常结束或 yield,把 `co` 栈上的结果用 `lua_xmove(co, L, nres)` 搬回 `L`;如果是错误,搬一个错误对象回 `L`。`luaB_coresume`(`lcorolib.c:57`)再在前面塞个 `true`,变成 `resume` 返回的 `true, ...results` 或 `false, errormsg`。

进入 `lua_resume`(`ldo.c:966`):

```c
LUA_API int lua_resume (lua_State *L, lua_State *from, int nargs,
                                      int *nresults) {
  TStatus status;
  lua_lock(L);
  if (L->status == LUA_OK) {  /* may be starting a coroutine */
    if (L->ci != &L->base_ci)  /* not in base level? */
      return resume_error(L, "cannot resume non-suspended coroutine", nargs);
    else if (L->top.p - (L->ci->func.p + 1) == nargs)  /* no function? */
      return resume_error(L, "cannot resume dead coroutine", nargs);
  }
  else if (L->status != LUA_YIELD)  /* ended with errors? */
    return resume_error(L, "cannot resume dead coroutine", nargs);
  L->nCcalls = (from) ? getCcalls(from) : 0;
  if (getCcalls(L) >= LUAI_MAXCCALLS)
    return resume_error(L, "C stack overflow", nargs);
  L->nCcalls++;
  luai_userstateresume(L, nargs);
  api_checkpop(L, (L->status == LUA_OK) ? nargs + 1 : nargs);
  status = luaD_rawrunprotected(L, resume, &nargs);
   /* continue running after recoverable errors */
  status = precover(L, status);
  if (l_likely(!errorstatus(status)))
    lua_assert(status == L->status);  /* normal end or yield */
  else {  /* unrecoverable error */
    L->status = status;  /* mark thread as 'dead' */
    luaD_seterrorobj(L, status, L->top.p);  /* push error message */
    L->ci->top.p = L->top.p;
  }
  *nresults = (status == LUA_YIELD) ? L->ci->u2.nyield
                                    : cast_int(L->top.p - (L->ci->func.p + 1));
  lua_unlock(L);
  return APIstatus(status);
}
```

分段读。开头是状态校验。协程要么 `status == LUA_OK`(第一次启动),要么 `status == LUA_YIELD`(从挂起恢复),其他都是错误——`resume_error`(`ldo.c:899`)返回 `LUA_ERRRUN` 并塞个错误串。这里有两个判据常被忽略:

- `L->ci != &L->base_ci`:意思是这个协程还有未完成的调用帧(不在最底层),那它一定不是 suspended。这用来挡住"一个协程正在跑(被别人 resume 了),又被第三个 resume"的情形。
- `L->top.p - (L->ci->func.p + 1) == nargs`:栈上除了参数没有别的东西——即函数已经跑完、栈被清空了。这是"协程正常结束已死"的判定。

`L->nCcalls = (from) ? getCcalls(from) : 0;` 这行很关键——**协程继承调用者的 C 调用深度**。`nCcalls` 是嵌套 C 调用计数(P4-13 见过),resume 一个协程本质上是 C 函数 `lua_resume` 的一次调用,要把调用者的 C 栈深度延续下去,否则协程里递归 C 调用会突破 `LUAI_MAXCCALLS`(`ldo.h:63`,默认 200)的栈深度保护而不自知。接着 `L->nCcalls++` 计自己这一次。

然后是核心:`status = luaD_rawrunprotected(L, resume, &nargs);`。`luaD_rawrunprotected`(`ldo.c:160`)是 Lua 的"pcall 骨架",一个 setjmp 包装:

```c
TStatus luaD_rawrunprotected (lua_State *L, Pfunc f, void *ud) {
  l_uint32 oldnCcalls = L->nCcalls;
  lua_longjmp lj;
  lj.status = LUA_OK;
  lj.previous = L->errorJmp;  /* chain new error handler */
  L->errorJmp = &lj;
  LUAI_TRY(L, &lj, f, ud);  /* call 'f' catching errors */
  L->errorJmp = lj.previous;  /* restore old error handler */
  L->nCcalls = oldnCcalls;
  return lj.status;
}
```

`LUAI_TRY` 在 ISO C 下就是 `if (setjmp((c)->b) == 0) ((f)(L, ud))`(`ldo.c:105`)。它给 `resume` 这个函数调用装上一个错误恢复点:协程里任何 `luaD_throw` 都会 `longjmp` 回到这里。注意 `lua_longjmp` 通过 `previous` 串成链(`lj.previous = L->errorJmp`),协程的 `errorJmp` 形成自己的错误恢复链——这是协程内错误能被 `pcall` 风格捕获的物理基础。

真正执行协程体的是 `resume`(`ldo.c:916`):

```c
static void resume (lua_State *L, void *ud) {
  int n = *(cast(int*, ud));  /* number of arguments */
  StkId firstArg = L->top.p - n;  /* first argument */
  CallInfo *ci = L->ci;
  if (L->status == LUA_OK)  /* starting a coroutine? */
    ccall(L, firstArg - 1, LUA_MULTRET, 0);  /* just call its body */
  else {  /* resuming from previous yield */
    lua_assert(L->status == LUA_YIELD);
    L->status = LUA_OK;  /* mark that it is running (again) */
    if (isLua(ci)) {  /* yielded inside a hook? */
      lua_assert(ci->callstatus & CIST_HOOKYIELD);
      ci->u.l.savedpc--;
      L->top.p = firstArg;  /* discard arguments */
      luaV_execute(L, ci);  /* just continue running Lua code */
    }
    else {  /* 'common' yield */
      if (ci->u.c.k != NULL) {  /* does it have a continuation function? */
        lua_unlock(L);
        n = (*ci->u.c.k)(L, LUA_YIELD, ci->u.c.ctx); /* call continuation */
        lua_lock(L);
        api_checknelems(L, n);
      }
      luaD_poscall(L, ci, n);  /* finish 'luaD_call' */
    }
    unroll(L, NULL);  /* run continuation */
  }
}
```

两条分支。

**第一次 resume**(`L->status == LUA_OK`):`ccall(L, firstArg - 1, LUA_MULTRET, 0)`。`firstArg - 1` 是栈上函数的位置(coroutine.create 时塞进去的那个 Lua 函数),`ccall`(`ldo.c:757`)是统一的调用入口:

```c
l_sinline void ccall (lua_State *L, StkId func, int nResults, l_uint32 inc) {
  CallInfo *ci;
  L->nCcalls += inc;
  if (l_unlikely(getCcalls(L) >= LUAI_MAXCCALLS)) {
    checkstackp(L, 0, func);
    luaE_checkcstack(L);
  }
  if ((ci = luaD_precall(L, func, nResults)) != NULL) {  /* Lua function? */
    ci->callstatus |= CIST_FRESH;  /* mark that it is a "fresh" execute */
    luaV_execute(L, ci);  /* call it */
  }
  L->nCcalls -= inc;
}
```

第三个参数 `inc` 传 0——注意,`0` 意味着这次调用既不算 C 调用也不算 non-yieldable 调用(对比 `luaD_call` 传 `1`,`luaD_callnoyield` 传 `nyci = 0x10000|1`)。这是协程体可以被 yield 的前提:`inc=0` 不增加 non-yieldable 计数,所以 `yieldable(L)` 在协程体内仍为真。`CIST_FRESH` 标记这个 `luaV_execute` 是协程的顶层执行帧,后面 yield 恢复时靠它辨认。

`luaD_precall`(`ldo.c:715`)为这个 Lua 函数建一个新的 `CallInfo`,设好 `savedpc = p->code`(从函数字节码起点开始),然后 `luaV_execute(L, ci)`(P3-11 讲过的取指-译码-执行循环)就开始跑协程体了。

**从挂起恢复**(`L->status == LUA_YIELD`):`L->status = LUA_OK` 先把状态翻成"运行中"。然后分两种 yield 来源。如果是 `isLua(ci)` 且 `CIST_HOOKYIELD`——这是在调试钩子里 yield 的特殊路径(钩子里不能正常 yield 值,见 2.6),`savedpc--` 撤回一条指令,`luaV_execute(L, ci)` 接着跑。常见的"协程体里 `coroutine.yield`"走 `else` 分支:如果 yield 发生在 C 函数里(通过 `lua_yieldk` 带了续体 `k`),调续体 `(*ci->u.c.k)(L, LUA_YIELD, ci->u.c.ctx)`;然后 `luaD_poscall(L, ci, n)` 收尾这次被 yield 打断的 C 调用。最后 `unroll(L, NULL)` 把可能堆在调用链上的多层续体跑完。

`unroll`(`ldo.c:866`)是个循环,把所有被 yield 挂起的帧一层层跑完:

```c
static void unroll (lua_State *L, void *ud) {
  CallInfo *ci;
  UNUSED(ud);
  while ((ci = L->ci) != &L->base_ci) {  /* something in the stack */
    if (!isLua(ci))  /* C function? */
      finishCcall(L, ci);  /* complete its execution */
    else {  /* Lua function */
      luaV_finishOp(L);  /* finish interrupted instruction */
      luaV_execute(L, ci);  /* execute down to higher C 'boundary' */
    }
  }
}
```

`while ((ci = L->ci) != &L->base_ci)`——只要调用栈还没退到根,就继续。C 帧用 `finishCcall` 把它的续体跑完,Lua 帧用 `luaV_execute` 接着解释。这是 yield 恢复后"把控制权沿调用链一路返回到协程顶层"的机制。

### 2.5 yield 让出:`lua_yieldk` 与 `coroutine.yield`

`coroutine.yield(...)`(`lcorolib.c:113`)极简:

```c

}
```

`lua_yield(L,n)` 是宏(`lua.h:317`):

```c

```

真正干活的是 `lua_yieldk`(`ldo.c:1006`):

```c
LUA_API int lua_yieldk (lua_State *L, int nresults, lua_KContext ctx,
                        lua_KFunction k) {
  CallInfo *ci;
  luai_userstateyield(L, nresults);
  lua_lock(L);
  ci = L->ci;
  api_checkpop(L, nresults);
  if (l_unlikely(!yieldable(L))) {
    if (L != mainthread(G(L)))
      luaG_runerror(L, "attempt to yield across a C-call boundary");
    else
      luaG_runerror(L, "attempt to yield from outside a coroutine");
  }
  L->status = LUA_YIELD;
  ci->u2.nyield = nresults;  /* save number of results */
  if (isLua(ci)) {  /* inside a hook? */
    lua_assert(!isLuacode(ci));
    api_check(L, nresults == 0, "hooks cannot yield values");
    api_check(L, k == NULL, "hooks cannot continue after yielding");
  }
  else {
    if ((ci->u.c.k = k) != NULL)  /* is there a continuation? */
      ci->u.c.ctx = ctx;  /* save context */
    luaD_throw(L, LUA_YIELD);
  }
  lua_assert(ci->callstatus & CIST_HOOKED);  /* must be inside a hook */
  lua_unlock(L);
  return 0;  /* return to 'luaD_hook' */
}
```

逐行。`api_checkpop(L, nresults)` 确认栈上有足够的值要 yield。然后是最关键的一行——**yieldable 检查**:

```c
if (l_unlikely(!yieldable(L))) {
  if (L != mainthread(G(L)))
    luaG_runerror(L, "attempt to yield across a C-call boundary");
  else
    luaG_runerror(L, "attempt to yield from outside a coroutine");
}
```

`yieldable(L)` 是宏(`lstate.h:104`):

```c
/* true if this thread does not have non-yieldable calls in the stack */
#define yieldable(L)		(((L)->nCcalls & 0xffff0000) == 0)
```

这行就是 2.6 节要深讲的"不能跨 C 边界 yield"的全部判据。先把它的位运算看清楚:`nCcalls` 是 `l_uint32`(32 位无符号),`0xffff0000` 取高 16 位。如果高 16 位非 0,说明调用栈上至少有一个"non-yieldable"调用——那这个 yield 就不合法,报错。

通过检查后,`L->status = LUA_YIELD` 把协程状态置为挂起。`ci->u2.nyield = nresults` 把这次 yield 要传出的值数存进当前 `CallInfo` 的 `u2.nyield` 字段(`lstate.h:205`)——这个字段稍后会被 `lua_resume` 读出来,作为 `*nresults` 返回给 resume 的调用者:

```c

                                  : cast_int(L->top.p - (L->ci->func.p + 1));
```

(这一行在 `lua_resume` 里,`ldo.c:994`。)

然后 `luaD_throw(L, LUA_YIELD)`。`luaD_throw`(`ldo.c:125`)走 `LUAI_THROW`,即 `longjmp` 回到最近的 `errorJmp`:

```c
l_noret luaD_throw (lua_State *L, TStatus errcode) {
  if (L->errorJmp) {  /* thread has an error handler? */
    L->errorJmp->status = errcode;  /* set status */
    LUAI_THROW(L, L->errorJmp);  /* jump to it */
  }
  ...
}
```

这个最近的 `errorJmp`,正是 `lua_resume` 调 `luaD_rawrunprotected` 时设的那个(`ldo.c:160` 的 `L->errorJmp = &lj`)。longjmp 把控制权瞬间弹回 `lua_resume`,绕过了中间所有 C 函数的栈帧。

回到 `lua_resume`:`luaD_rawrunprotected` 返回 `LUA_YIELD`(因为 `lj.status` 被设成了 `LUA_YIELD`)。`precover`(`ldo.c:955`)对 yield 状态(`LUA_YIELD < LUA_ERRRUN`,不满足 `errorstatus`)直接返回不动。然后:

```c

  lua_assert(status == L->status);  /* normal end or yield */
```

`status == L->status`——此时都是 `LUA_YIELD`,自洽。`*nresults = L->ci->u2.nyield` 把 yield 的值数交还调用者。`lua_resume` 返回 `APIstatus(LUA_YIELD)` = 1。

注意整个 yield 的"切换"动作:**根本没有切 OS 线程,也没有保存/恢复寄存器集**。它做的事是:(1) 在协程的 `lua_State` 上记下 `status = LUA_YIELD` 和 `nyield`;(2) longjmp 弹回 `lua_resume`(它本来就在调用者 `L` 的 C 栈上等着);(3) `lua_resume` 返回到 `auxresume`,`lua_xmove` 把 yield 的值搬到调用者栈。所谓"切回 resume 者",在 C 层面就是一次函数返回。

### 2.6 为什么不能跨 C 边界 yield

这是本章的核心 why。看 `yieldable` 的全部含义。`nCcalls` 的位布局在 `lstate.h:94-100` 有注释:

```c
/*
** About 'nCcalls':  This count has two parts: the lower 16 bits counts
** the number of recursive invocations in the C stack; the higher
** 16 bits counts the number of non-yieldable calls in the stack.
** (They are together so that we can change and save both with one
** instruction.)
*/

#define yieldable(L)		(((L)->nCcalls & 0xffff0000) == 0)
#define getCcalls(L)	((L)->nCcalls & 0xffff)
#define incnny(L)	((L)->nCcalls += 0x10000)
#define decnny(L)	((L)->nCcalls -= 0x10000)
#define nyci	(0x10000 | 1)
```

- 低 16 位:嵌套 C 调用的深度(`getCcalls` 取它)。
- 高 16 位:non-yieldable 调用的深度。

`ccall` 的第三个参数 `inc` 决定一次调用加多少。三种用法:

- `luaD_call`(`ldo.c:775`):`ccall(L, func, nResults, 1)`——加 1,只算 C 调用深度,不增加 non-yieldable。Lua 函数可以从这里 yield。
- `luaD_callnoyield`(`ldo.c:783`):`ccall(L, func, nResults, nyci)`——加 `0x10000|1`,既算 C 调用又算 non-yieldable。这种调用期间禁止 yield。
- 协程体的首次调用(`resume` 里 `ccall(L, firstArg - 1, LUA_MULTRET, 0)`):加 0,两个都不算。

什么场景会用 `luaD_callnoyield`?P4-13 讲过的 metamethod 调用、`__close` 调用、某些 finalizer——它们的共同点是:**这些调用不能 yield,因为 yield 之后无法安全地把控制权交还给它们**。

要理解这一点,设想一个具体场景。宿主 C 代码这么写:

```c
int my_cfunc(lua_State *L) {
  lua_call(L, 2, 1);      /* 调一个 Lua 函数,期望它返回 */
  do_something();          /* ← 这一行依赖 lua_call 已返回 */
  return 1;
}
```

`lua_call` 是同步的——它进入被调函数,跑完,返回。`do_something()` 这行在 `lua_call` 返回后才执行,这是 C 语义的保证。现在假设被调的那个 Lua 函数内部 `coroutine.yield` 了。如果 Lua 允许它 yield,会发生什么?控制权会 longjmp 弹出 `lua_call`,回到 `lua_resume`,协程挂起。可 `my_cfunc` 的 C 栈帧被一起弹掉了——`do_something()` 永远不会被执行,C 函数的状态(局部变量、循环计数器、已分配的资源)全部丢失。等到下一次 resume 这个协程,Lua 试图从 yield 点继续,但 `my_cfunc` 已经不存在了,它的后半截代码成了悬空(dead)的逻辑。

这就是"跨 C 边界 yield"的物理后果:**留下一个执行到一半、再也回不去的 C 帧**。C 函数没有"续体"(continuation)机制——不像 Lua 函数有 `savedpc` 可以记下来下次接着跑。一个 C 函数要么跑完返回,要么没跑完;没有"跑到一半挂起,过会儿从中间某行继续"这种事。

所以 Lua 用 `nCcalls` 高 16 位显式标记"这一段调用不允许 yield"。只要调用栈上还有任何 non-yieldable 帧,`yieldable(L)` 就是假,yield 会报错。报错信息分两种:如果是主线程(`L == mainthread(G(L))`),报"attempt to yield from outside a coroutine"(主线程压根不是协程);否则报"attempt to yield across a C-call boundary"(在协程里但跨了 C 边界)。

那些**确实需要 yield 的 C 函数**,Lua 给了续体机制:`lua_callk`/`lua_pcallk` + `lua_KFunction k`。它们带一个续体函数指针,当被调函数 yield 后再 resume 时,Lua 调用 `k` 而不是回到原 C 函数(`resume` 里 `(*ci->u.c.k)(L, LUA_YIELD, ci->u.c.ctx)` 那行,`ldo.c:936`)。续体函数从零开始重建状态(用 `ctx` 传上下文),相当于"yield 之后的下半截代码"重写成一个独立 C 函数。这是 Lua 把"不可中断的 C 调用"改造成"可中断"的唯一方式——**不是真的让 C 函数挂起,而是把它拆成 yield 前和 yield 后两段**。

### 2.7 resume/yield 的值传递:对称的值管道

resume 和 yield 的值传递是对称的,这是 Lua 协程设计上很干净的一点。

- `coroutine.resume(co, a, b, c)` 的 `a, b, c`:第一次 resume 时成为协程体的参数;后续 resume 时成为协程内上一次 `coroutine.yield(...)` 的返回值。
- `coroutine.yield(x, y, z)` 的 `x, y, z`:成为对应 `coroutine.resume` 的返回值(在 `true` 之后)。

值怎么走的,在源码里是 `lua_xmove` 两次。resume 方向:`auxresume` 里 `lua_xmove(L, co, narg)`(`lcorolib.c:39`),把参数从调用者 `L` 搬到协程 `co`,协程体把它们当参数或 yield 返回值用。yield 方向:yield 的值留在协程 `co` 栈上,`lua_resume` 返回后,`auxresume` 里 `lua_xmove(co, L, nres)`(`lcorolib.c:47`)把它们搬到调用者 `L`,成为 resume 的返回值。

`lua_xmove` 之所以能直接搬值(而不是序列化/反序列化),是因为两个 `lua_State` 共享同一个 `global_State`——同一个 GC 堆、同一张字符串表。一个 `TValue` 里装的 `Table*`/`TString*`/`Closure*` 指针,在两个 `lua_State` 的栈上都指向同一块内存,直接拷贝 `TValue` 即可。这是协程值传递零拷贝(除了 `TValue` 本身那 16 字节)的根源。

注意一个边界:`lua_xmove` 的 `api_check(from, G(from) == G(to), "moving among independent states")`(`lapi.c:131`)。两个**独立**的 `lua_State`(各自 `lua_newstate` 创建的,不共享 `global_State`)之间不能 `xmove`——它们的 GC 堆是隔离的,指针不通用。`xmove` 只在"同一进程内、协程之间"合法。

### 2.8 错误传播:协程内出错

协程体内的错误怎么处理?Lua 的回答是:**协程内的未捕获错误,作为 `coroutine.resume` 的返回值传给 resume 者**,而不是抛到宿主。这是 pcall 风格的错误传播——resume 本身就在 `luaD_rawrunprotected` 的保护下运行。

看 `lua_resume` 的错误分支(`ldo.c:989`):

```c
else {  /* unrecoverable error */
  L->status = status;  /* mark thread as 'dead' */
  luaD_seterrorobj(L, status, L->top.p);  /* push error message */
  L->ci->top.p = L->top.p;
}
```

`L->status = status`(此时 status 是某个 `LUA_ERRxxx`)——**协程一旦未捕获出错,就永久变 dead**,不能再 resume。错误对象被 `luaD_seterrorobj` 推到协程栈顶,然后 `auxresume` 里 `lua_xmove(co, L, 1)`(`lcorolib.c:51`)搬到调用者,`luaB_coresume` 返回 `false, errormsg`。

还有一种"可恢复错误"——协程内有 `pcall`,pcall 捕获了错误但 pcall 自己又 yield 了。这种复杂情况由 `precover`(`ldo.c:955`)处理:

```c
static TStatus precover (lua_State *L, TStatus status) {
  CallInfo *ci;
  while (errorstatus(status) && (ci = findpcall(L)) != NULL) {
    L->ci = ci;  /* go down to recovery functions */
    setcistrecst(ci, status);  /* status to finish 'pcall' */
    status = luaD_rawrunprotected(L, unroll, NULL);
  }
  return status;
}
```

`findpcall`(`ldo.c:884`)沿调用链找一个带 `CIST_YPCALL` 标记的帧(就是 pcall 的恢复点)。找到就把控制权下放到那一层,重新装一个 `luaD_rawrunprotected`,让 pcall 的错误处理跑起来。这保证协程在"pcall 内 yield 后又遇到错误"这种交叉场景下也能正确恢复——`CIST_RECST`(`lstate.h:262`)那 3 个位就是用来在多次 yield/close 之间保存原始错误状态的。

错误传播的 sound 性(不泄漏)靠两点:(1) 协程出错后 `status` 被永久置成错误码,`lua_resume` 开头的 `else if (L->status != LUA_YIELD)` 会挡住后续 resume,返回 "cannot resume dead coroutine";(2) 协程内的 `to-be-closed` 变量和 upvalue 由 `luaE_resetthread`(`lstate.c:310`)在协程死亡时关闭——`lua_closethread`(`lstate.c:324`)和 `luaD_closeprotected`(`ldo.c:1059`)负责把 `__close` 元方法跑完、把 upvalue 收掉。

### 2.9 `coroutine.wrap`:函数化的协程

`coroutine.wrap(f)`(`lcorolib.c:106`)是 `create` + 一个包装:

```c
static int luaB_cowrap (lua_State *L) {
  luaB_cocreate(L);
  lua_pushcclosure(L, luaB_auxwrap, 1);
  return 1;
}
```

它先 `create` 一个协程(协程对象作为 upvalue),然后把 `luaB_auxwrap` 包成一个 C 闭包返回。调用这个闭包就等于 `resume` 那个协程。区别于 `coroutine.resume`:`wrap` 返回的函数**直接返回 yield 的值**(出错时 propagate 错误而不是返回 `false, err`)。看 `luaB_auxwrap`(`lcorolib.c:74`):

```c
static int luaB_auxwrap (lua_State *L) {
  lua_State *co = lua_tothread(L, lua_upvalueindex(1));
  int r = auxresume(L, co, lua_gettop(L));
  if (l_unlikely(r < 0)) {  /* error? */
    int stat = lua_status(co);
    if (stat != LUA_OK && stat != LUA_YIELD) {  /* error in the coroutine? */
      stat = lua_closethread(co, L);  /* close its tbc variables */
      lua_assert(stat != LUA_OK);
      lua_xmove(co, L, 1);  /* move error message to the caller */
    }
    if (stat != LUA_ERRMEM &&  /* not a memory error and ... */
        lua_type(L, -1) == LUA_TSTRING) {  /* ... error object is a string? */
      luaL_where(L, 1);  /* add extra info, if available */
      lua_insert(L, -2);
      lua_concat(L, 2);
    }
    return lua_error(L);  /* propagate error */
  }
  return r;
}
```

出错时 `lua_closethread(co, L)` 先把协程的 to-be-closed 变量关掉(避免泄漏),再把错误对象加上调用位置信息(`luaL_where`),最后 `lua_error(L)` 把错误重新抛到 wrap 的调用者——表现成一个普通 Lua 错误。这是 wrap 与 resume 在错误语义上的根本区别:resume 把错误当返回值,wrap 把错误当异常。

### 2.10 一次完整的 create→resume→yield→resume 栈切换

把上面所有部件串起来,走一遍一个最小例子的栈切换全过程。Lua 代码:

```lua
local co = coroutine.create(function(a)
  local b = coroutine.yield(a + 1)   -- 第一次 yield
  return b * 2                        -- 最终返回
end)

local r1 = coroutine.resume(co, 10)   -- r1 == true, 11
local r2 = coroutine.resume(co, 20)   -- r2 == true, 40
local r3 = coroutine.resume(co)       -- r3 == false, "cannot resume dead coroutine"
```

**步骤 0·create**:`coroutine.create(f)`。`lua_newthread` 建协程 `co`(自己的栈 `co->stack`、调用链只有 `&base_ci`、`status=LUA_OK`)。`lua_xmove` 把 `f` 搬到 `co->stack[0]`(func 槽)。此刻 `co` 状态:suspended(初始,`status==LUA_OK` 但栈非空)。

**步骤 1·第一次 resume(co, 10)**:调用者 `L` 是主线程。

- `auxresume`:`lua_xmove(L, co, 1)` 把 `10` 搬到 `co` 栈(`co->stack[1]`)。
- `lua_resume(co, L, 1, &nres)`:`co->status == LUA_OK` 且 `ci == &base_ci`,通过校验。`co->nCcalls = getCcalls(L) + 1`。
- `luaD_rawrunprotected(co, resume, ...)`:装上 errorJmp。
- `resume`:`co->status == LUA_OK` 走首次分支。`ccall(co, firstArg-1, LUA_MULTRET, 0)`——`firstArg` 是 `co->stack[1]`(参数 10),`firstArg-1` 是 `co->stack[0]`(函数 f)。`inc=0` 不增加 non-yieldable。
- `luaD_precall`:为 f 建新 `CallInfo ci1`,`ci1->func = &co->stack[0]`,`ci1->savedpc = f->p->code`。参数 10 落到 f 的第一个形参寄存器。
- `luaV_execute(co, ci1)`:开始解释 f 的字节码。执行 `a + 1`(= 11),然后碰到 `coroutine.yield(a+1)`。

**步骤 2·yield(11)**:`coroutine.yield(11)` → `lua_yield(co, 1)` → `lua_yieldk(co, 1, 0, NULL)`。

- `yieldable(co)`:此刻 `co->nCcalls` 高 16 位是 0(因为 ccall 用 inc=0),为真,通过。
- `co->status = LUA_YIELD`。`ci->u2.nyield = 1`(yield 一个值)。
- `isLua(ci)` 为真但 `ci` 是协程体的 Lua 帧(不是 hook),走 `else`:`ci->u.c.k = NULL`(没续体),`luaD_throw(co, LUA_YIELD)`。
- `luaD_throw` → `LUAI_THROW` → longjmp 弹回 `luaD_rawrunprotected`(`ldo.c:166` 的 `LUAI_TRY` 那行)。`lj.status = LUA_YIELD`。

**步骤 3·lua_resume 返回第一次**:`luaD_rawrunprotected` 返回 `LUA_YIELD`。`precover` 对 yield 不动。`*nresults = co->ci->u2.nyield = 1`。`lua_resume` 返回 1(LUA_YIELD)。

- 回到 `auxresume`:`lua_xmove(co, L, 1)` 把 11 搬到 `L` 栈。返回 1。
- `luaB_coresume`:塞 `true`,返回 `true, 11`。Lua 里 `r1 == true, 11`。

此刻协程 `co` 的状态:`status == LUA_YIELD`,`ci` 还指向 `ci1`(f 的帧),`ci1->savedpc` 停在 yield 那条指令之后,`co->stack` 上 f 的局部变量(`a=10`、中间结果)都原封不动。**协程的"挂起"就是把整套执行状态留在它自己的 `lua_State` 里,等下次 resume**。

调用者 `L`(主线程)此刻在跑,`co` 完全静默——它们共享一个 OS 线程,谁在跑谁独占。

**步骤 4·第二次 resume(co, 20)**:

- `lua_xmove(L, co, 1)` 把 20 搬到 `co` 栈顶。
- `lua_resume`:`co->status == LUA_YIELD`,通过校验。
- `resume` 走 `else`(非首次):`co->status = LUA_OK`。`ci->u.c.k == NULL`(上次 yield 没续体),跳过续体调用。`luaD_poscall(co, ci, n)`——这里 n 是 resume 传进来的参数数(1),把 20 作为上一次 yield 的返回值落到 f 的字节码里(`local b = coroutine.yield(...)` 的 b 收到 20)。
- `unroll(co, NULL)`:`co->ci` 此刻还是 `ci1`(isLua 为真),`luaV_finishOp` 完成 yield 那条指令的收尾,`luaV_execute(co, ci1)` 接着跑 f 的剩余字节码。
- f 执行 `b * 2`(= 40),`return`。

**步骤 5·f 返回**:f 的 `RETURN` 指令走 `luaD_poscall`,把 40 落到结果位置,`co->ci` 退回 `&base_ci`。`unroll` 的 `while ((ci = L->ci) != &L->base_ci)` 退出(已经到根)。`resume` 返回。`luaD_rawrunprotected` 返回 `LUA_OK`。

- `lua_resume`:`status == LUA_OK`,`*nresults = co->top - (co->ci->func + 1) = 1`(栈上是 40)。返回 0(LUA_OK)。
- `auxresume`:`lua_xmove(co, L, 1)` 搬 40。返回 1。
- `luaB_coresume` 返回 `true, 40`。Lua 里 `r2 == true, 40`。

此刻 `co->status == LUA_OK`,栈已空(`ci == &base_ci`,`top == func+1`)——协程进入 dead。

**步骤 6·第三次 resume(co)**:`lua_resume` 里 `co->status == LUA_OK` 但 `co->top - (co->ci->func + 1) == 0 == nargs`,命中 `resume_error(L, "cannot resume dead coroutine", nargs)`。返回 `LUA_ERRRUN`。`luaB_coresume` 返回 `false, "cannot resume dead coroutine"`。Lua 里 `r3 == false, "..."`。

整个过程中,**没有任何 OS 线程的创建或切换**。所有"并发"都是:协程的 `lua_State` 独立保留执行状态(栈、ci、savedpc),resume 用 C 函数调用进入协程体,yield 用 longjmp 弹出。C 调用栈本身在 resume 期间增长了一层(`lua_resume` → `ccall` → `luaV_execute`),yield 时这一层被 longjmp 整体弹掉。

## 三、为什么这样设计是 sound 的

### 3.1 栈隔离:一个协程崩溃不连累别的

每个协程有独立的 `stack` 数组(`stack_init` 里 `luaM_newvector` 分配)和独立的 `ci` 链。一个协程栈溢出,只触发自己的 `luaD_growstack`(`ldo.c:353`)或自己的栈溢出错误;不会写到别的协程的栈上。一个协程的 `luaD_throw`,longjmp 弹的是它自己的 `errorJmp` 链(每个 `lua_State` 有自己的 `errorJmp` 字段,`lstate.h:298`),不会跳到别的协程的恢复点。

值隔离 + 状态隔离,使得"一个协程崩溃"是一个局部事件。错误要么被协程内的 pcall 接住,要么传给 resume 者(协程变 dead),不会污染共享的 `global_State` 或别的协程。共享的部分(GC 堆、字符串表)是只读或受 GC 保护的,不会因为一个协程的错误而损坏。

### 3.2 yield 的 C 边界约束:不留下悬空 C 帧

2.6 已详述。核心 sound 性:`yieldable(L)` 用 `nCcalls` 高 16 位显式标记调用栈上是否有 non-yieldable 帧。任何会留下"执行到一半的 C 帧"的路径(metamethod、`__close`、finalizer、同步 `lua_call`),都用 `luaD_callnoyield` 加 `nyci` 标记。yield 前检查这个标记,有就报错——**宁可报错,也不产生悬空帧**。

需要 yield 的 C 调用,用 `lua_callk`/`lua_pcallk` 显式提供续体 `k`,把"yield 之后的逻辑"重写成一个独立函数。这不是绕过约束,而是**把一个不可中断的 C 函数,重构成"yield 前 + yield 后"两段可中断的代码**。约束因此被满足:yield 发生时,C 栈上没有任何"还没跑完"的帧——要么已经跑完(返回了),要么本来就有续体等着接手。

### 3.3 错误传播:不泄漏

协程出错时,`lua_resume` 的 `else` 分支(`ldo.c:989`)做三件事:把 `status` 永久置成错误码(协程 dead)、把错误对象推到栈顶、调整 `ci->top`。这保证:(1) 后续 resume 被开头校验挡住,不会重复触发错误;(2) 错误对象通过 `auxresume` 的 `lua_xmove` 传给调用者,不丢;(3) 协程死亡时 `lua_closethread`/`luaE_resetthread` 关闭它的 to-be-closed 变量和 upvalue,资源不泄漏。

`precover` 处理"pcall 内 yield 后又出错"的交叉场景,靠 `CIST_RECST` 那 3 个位在多次 yield/close 之间保存原始错误状态,保证 pcall 的语义在协程里和普通代码里一致。

### 3.4 C 调用深度继承:不突破栈保护

`lua_resume` 里 `L->nCcalls = (from) ? getCcalls(from) : 0`(`ldo.c:978`)——协程继承调用者的 C 栈深度。这保证协程内递归 C 调用也会被 `LUAI_MAXCCALLS`(`ldo.h:63`,默认 200)挡住。如果不继承,协程内的深层 C 递归会绕过保护,真正撑爆 C 栈(那是一次进程级崩溃,不是 Lua 错误)。继承让协程的 C 栈深度是"宿主深度 + 协程内深度"的累计,保护始终有效。

### 3.5 共享 global_State:不多花一份机制

协程之间共享 `global_State` 的设计,使得字符串驻留、GC、注册表、内存分配器都只有一份。创建一个协程的开销就是一个 `lua_State` 结构 + 一个初始栈(45 个 `StackValue`,约 720 字节)+ 一个 `CallInfo`——没有复制全局状态、没有独立的 GC 堆、没有字符串表副本。这是"少量机制换多任务"的精确含义:用 `lua_State` 这一个结构(本就存在的)的复制,换到一个完整的协作式执行流。

## 四、★对照 CPython:generator + asyncio vs Lua 协程

CPython 也有"协程",但它的演化路径和 Lua 完全不同,值得对照看清 Lua 的取舍。

### 4.1 CPython 的 generator:只能在自己的帧里 yield

CPython 的 generator 是一个带独立栈帧(`PyFrameObject`,3.11 后是 `executor` + `PyCodeObject`)的对象。`yield` 只能在 generator 函数自己的帧里发生——yield 弹出当前 generator 帧的控制权,回到调用者,但**调用者不能在自己的帧里 yield**。换句话说,Python 的 generator 是"单层的":yield 必须直接发生在 generator 函数体内,不能跨函数边界(一个 generator 函数调了普通函数,普通函数里不能 yield 回到 generator 的调用者)。

CPython 字节码层面,generator 靠 `GEN_START`/`RESUME`/`YIELD_VALUE`/`RETURN_VALUE` 这几条指令配合。`YIELD_VALUE` 把栈顶值交给调用者,`frame` 状态被保留(`f_lasti` 记下一条要执行的指令)。

Lua 的协程不同:`coroutine.yield` 可以发生在协程调用栈的**任意深度**。协程体调了一个函数,那个函数再调一个函数,第三层里 `coroutine.yield`——完全合法,整个调用链(三层 `CallInfo`)被原样保留在协程的 `lua_State` 里,下次 resume 从第三层那条 `savedpc` 继续。Lua 的 yield 是"穿透调用栈"的,因为 yield 走 longjmp 弹到 `lua_resume` 这个协程的顶层恢复点,中间不管多少层都被保留状态。

这是两者在能力上的根本差异:**Python 的 generator 是"一个可挂起的函数",Lua 的协程是"一个可挂起的执行流(完整线程)"**。

### 4.2 CPython 的 asyncio:event loop 驱动

CPython 3.5+ 用 `async`/`await` 引入协程语法,底层是 `asyncio` 库。但 `async def` 定义的协程,本质仍是 generator(基于 `CO_COROUTINE` flag 的 generator),它的"并发"靠 **event loop**(`asyncio.get_event_loop()`)调度——event loop 维护一个就绪队列,`await` 一个 future 时让出,loop 调度别的就绪任务。

关键区别:asyncio 的"让出"是协作式的(必须 `await`),但**调度是中心化的**——有一个显式的 event loop 在背后决定下一个跑谁。event loop 还管 IO 多路复用(select/epoll)、定时器、callback。

Lua 没有内置 event loop。`coroutine.resume`/`yield` 只提供原语——谁 resume 谁,完全由用户代码决定。Lua 的协程是更底层的"协作式线程"原语,要做成 asyncio 那样的调度器,得用户自己写(或用第三方库)。这是 Lua"精简"的体现:不把 event loop 这种高层抽象塞进内核,只提供最小机制(独立栈 + resume/yield),需要调度的人自己在上面搭。

### 4.3 数据对比表

| 维度 | Lua 5.5 协程 | CPython generator + asyncio |
|---|---|---|
| **yield 能穿透的深度** | 任意深度(整个协程调用栈) | 仅 generator 函数自身那一层(generator);asyncio 的 `await` 也只能 await 别的 awaitable,不能穿透普通函数 |
| **调度模型** | 无内置调度,resume/yield 是原语,用户自己编排 | event loop 中心化调度,就绪队列 + IO 多路复用 |
| **状态机** | 4 态(running/normal/suspended/dead),靠 `status` 字段 + 调用栈是否空判定 | generator: `GEN_CREATED`/`GEN_SUSPENDED`/`GEN_RUNNING`/`GEN_CLOSED`;asyncio Task: 又一层状态 |
| **跨 C 边界 yield** | 禁止(`yieldable` 检查),需用 `lua_callk` 续体 | CPython C API 里 similarly 受限(`PyIter_Next` 不能跨 C 帧 yield 到别处),但 generator 的 yield 本身只在 Python 帧内 |
| **值传递** | resume↔yield 对称传值(`lua_xmove`) | `send()` 传值进 generator,`yield` 传值出;asyncio 用 future/result |
| **实现机制** | 独立 `lua_State` + longjmp 切换,共享 `global_State` | 独立 frame 对象 + 指令级状态保留,共享 interpreter |
| **错误传播** | 协程内未捕获错误→resume 返回 `false, err`,协程 dead | generator 内异常→抛给 `next`/`send` 的调用者;asyncio Task 异常存在 task 里 |

Lua 协程的相对位置:**比 Python generator 更强(完整线程,任意深度 yield),比 asyncio 更原始(无内置调度)**。这正是"统一与精简"的体现——用 `lua_State` 这一个已存在的结构,加上 `resume`/`yield` 两个 API,换到一个完整的协作式多任务原语,不引入 event loop、不引入就绪队列、不引入 IO 多路复用。需要那些的人,在协程之上自己搭。

## 五、回扣主线

协程是本书主线"统一与精简换小而快"在执行侧的一个极致样本。

- **统一**:`lua_State` 这个结构本就是 Lua 的核心(一个 `lua_State` 就是一个 VM 实例,P0-01 见过)。协程没有引入新的"执行流"类型——一个协程就是一个 `lua_State`,类型标记 `LUA_TTHREAD`,和主线程是同一种对象,走同一套 GC、同一套栈管理、同一套调用链。统一的数据结构,服务了"主 VM 实例"和"协程"两种角色。
- **精简**:resume/yield 的全部机制,就是 `lua_newthread`(建独立栈)+ `lua_resume`(C 函数调用进入)+ `lua_yield`(longjmp 弹出)+ `lua_xmove`(共享堆上零拷贝传值)。没有 OS 线程、没有调度器、没有就绪队列、没有寄存器集保存/恢复。"并发"这个通常很重的概念,在 Lua 里被压成了几次 C 函数调用和一次 longjmp。
- **换到的能力**:完整的协作式多任务——任意深度 yield、对称值传递、独立栈隔离、pcall 风格错误传播。这些能力加起来,够支撑状态机、生成器、迭代器、协作式调度(用户在协程之上自己写 event loop)。

而约束("不能跨 C 边界 yield")不是缺陷,是 sound 的代价——它保证 yield 不留下悬空 C 帧,使得"协作式挂起"在 C 实现层面是安全的。Lua 用 `nCcalls` 高 16 位这一个计数器,把这条约束物化成一行检查(`yieldable` 宏),又是一处"少量机制换正确性保证"。

读完本章,协程不再是魔法:它就是几个共享 `global_State` 的 `lua_State`,在 C 函数调用和 longjmp 之间换手。Lua 的"并发"被精简到了这个程度,仍然是一门完整的、可跑生产逻辑的语言——这正是"用更少的机制换更多的能力"在协程这一招上的落地。
