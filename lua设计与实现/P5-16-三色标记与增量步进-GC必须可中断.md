# P5-16 三色标记与增量步进:GC 必须可中断

> **本书主线**:统一与精简换小而快。**二分法**:执行侧。**★对照**:CPython(引用计数 + 分代 STW)。**源码**:lua-5.5.0 `lgc.c` / `lgc.h` / `lstate.h`。**基调**:纯直球,不用比喻。
>
> **本章落点**:全书主线第三招"增量式三色 GC,可中断"的具体落实。前两招(统一 Table、寄存器式字节码)讲的是"小"和"快";这一招讲的是"**适合嵌入**"。一个游戏引擎每帧 16 毫秒、Redis 单线程跑事务脚本,都不能接受 GC 把宿主堵住几百毫秒。Lua 的回答是:把一次完整回收切成无数小步,每分配一小段内存只做一小段 GC 工作,把停顿摊平到几微秒一次。这一章就是把这套"可中断、按步推进、还不丢正确性"的机制,一行行源码讲清楚。

## 一、这章解决什么问题

先说清楚一个脚本语言 VM 为什么需要 GC,以及"可中断"为什么对 Lua 这种嵌入型 VM 几乎是生死攸关。

Lua 的所有复合对象(`Table`、闭包、`userdata`、`Proto`、线程、`UpVal`)都是堆上分配的 GC 对象,挂在 `global_State` 的 `allgc` 链上(`lstate.h:345`)。VM 执行字节码时不断创建这些对象,但"什么时候回收、谁来回收"是个问题:Lua 没有 C++ 那种确定性析构,也没有 Rust 那种所有权检查,它选了**追踪式(tracing)GC**——从一组根(root)出发,沿着指针链把所有可达对象标成活的,剩下的就是垃圾。

朴素做法是一次性扫完整堆。这在小脚本里没问题,堆一大就糟了:一个停顿可能几百毫秒,期间整个 VM 完全冻结。对桌面脚本也许还能忍,对 Lua 的典型宿主却是灾难——

- **游戏引擎**:每帧 16 毫秒预算,GC 停顿超过一两帧玩家就明显感觉到卡顿。
- **Redis / nginx**:单线程事件循环,GC 把主循环堵住就等于把所有连接都堵住。
- **嵌入式设备**:不仅卡,还可能直接撑爆有限的响应时限。

所以 Lua 必须让 GC **可中断**:做一小段工作就立刻把控制权还给宿主/解释器,让业务继续跑,过一会儿再来做下一段。难点在于——GC 中途放下了控制权,业务代码可能在这段空隙里改了对象图(新建对象、改指针指向),GC 凭什么还能保证回收正确?既不漏(该回收的没回收,内存泄漏)、也不重(不该回收的回收了,悬空指针)?

Lua 的解法是**增量式三色标记清除**:

1. 给每个对象上色(白/灰/黑),用颜色编码"已处理到哪一步"。
2. 维护一条核心不变式:**黑对象绝不指向白对象**。只要这条成立,回收就不会误杀活对象。
3. 业务代码改指针破坏不变式时,用**写屏障**当场补救。
4. 把整个回收切成若干阶段,每阶段再切成小步,由内存分配量驱动步进。

下面把这套机制在 `lgc.c` 里的实现一行行走一遍,最后讲清楚它为什么是 sound 的。

## 二、源码怎么实现

### 2.1 三色与核心不变式

三种颜色编码在 GCObject 公共头的 `marked` 字段里(`lobject.h:301` 的 `CommonHeader` = `next` + `tt` + `marked`),三个 bit:

```c
// lgc.h:79-84
#define WHITE0BIT    3  /* object is white (type 0) */
#define WHITE1BIT    4  /* object is white (type 1) */
#define BLACKBIT     5  /* object is black */
#define FINALIZEDBIT 6  /* object has been marked for finalization */
```

注意有两套白(`WHITE0BIT`/`WHITE1BIT`),只有一位黑(`BLACKBIT`)。颜色的判定宏在 `lgc.h:91-94`:

```c
#define iswhite(x)  testbits((x)->marked, WHITEBITS)        /* 任一白位置位 */
#define isblack(x)  testbit((x)->marked, BLACKBIT)
#define isgray(x)  /* neither white nor black */
    (!testbits((x)->marked, WHITEBITS | bitmask(BLACKBIT)))
```

为什么有两套白?这是为了在清扫(sweep)阶段区分"本轮该死"和"本轮新生"。任意时刻 `global_State.currentwhite`(`lstate.h:333` 附近,`otherwhite` 宏 `lgc.h:98` 翻转之)指明当前用的是哪一套白。一次回收的 atomic 步会**翻转**当前白:上一轮活下来的对象标的是旧白,本轮新建对象标的是新白;清扫时凡是带旧白色的对象就是本轮要回收的。这套双白方案让"区分本轮死物 vs 下一轮新生物"只需看位,不必每次重置全堆颜色。

三色的语义在 `lgc.h:17-29` 的注释里说得很直白:

- **白**:未标记。本轮可能可回收(具体死活看是不是"当前白")。
- **灰**:已标记可达,但它的子对象(它指向的对象)还没遍历完。
- **黑**:已标记,且所有子对象都已经遍历过。

核心不变式(`lgc.h:20-23`)是整个回收正确性的命脉:

> **一个黑对象绝不指向白对象。**

只要这条成立,任何白对象都一定不可达(否则它要么在灰链上、要么被某个灰/黑对象指向并最终被标灰)。回收白对象就不会误杀。增量回收的全部复杂度,都围绕"如何在业务代码穿插运行的情况下维持这条不变式"展开。

### 2.2 GC 状态机

增量 GC 不是一条直线走完的,而是一个状态机。5.5 的状态定义在 `lgc.h:35-43`,这是和 5.3/5.4 老资料最容易出差异的地方:

```c
// lgc.h:35-43 (lua-5.5.0)
#define GCSpropagate    0
#define GCSenteratomic  1   /* 5.5 新增:进入原子步前的过渡态 */
#define GCSatomic       2
#define GCSswpallgc     3
#define GCSswpfinobj    4
#define GCSswptobefnz   5
#define GCSswpend       6
#define GCScallfin      7
#define GCSpause        8
```

**★5.5 vs 老资料差异(重要)**:讲 5.3/5.4 的资料里状态通常是 `GCSpropagate / GCSatomic / GCSsweepstring / GCSsweep / GCSfinalize`。5.5 把清扫拆得更细了——清扫分成扫 `allgc`、扫 `finobj`、扫 `tobefnz`、收尾(`swpend`)四个子状态;终结(finalizer)调用是独立的 `GCScallfin`;还多了一个 `GCSenteratomic` 作为进入原子步的过渡状态。另外 5.3 那个"先单独扫字符串表"的 `GCSsweepstring` 在 5.5 已经合并进通用清扫(字符串表只在 `GCSswpend` 的 `checkSizes` 里顺手收缩),老资料这条状态名已过时。

另外有两个判断宏贯穿全局(`lgc.h:46-58`):

```c
#define issweepphase(g)  \
    (GCSswpallgc <= (g)->gcstate && (g)->gcstate <= GCSswpend)
#define keepinvariant(g)  ((g)->gcstate <= GCSatomic)
```

- `keepinvariant(g)` 为真,表示"现在必须维持黑不指白的不变式"——即 `propagate`、`enteratomic`、`atomic` 三个状态。进入清扫阶段后,不变式**允许被破坏**(清扫会把活对象重新标白,这时白对象可能指向黑对象,没关系,因为清扫只看"是否旧白"来决定死活,不依赖不变式),直到清扫全部结束、所有对象重新回到白,不变式才在下轮标记开始时重建。这条注释见 `lgc.h:50-57`。
- `issweepphase(g)` 为真,表示当前在清扫阶段——写屏障在这个阶段有另一套处理(见 2.6)。

状态机由 `singlestep` 驱动,每次推进一个状态。`singlestep` 是整个增量 GC 的心脏,放在 `lgc.c:1617`。

### 2.3 增量的心脏:`singlestep`

```c
// lgc.c:1617-1686
static l_mem singlestep (lua_State *L, int fast) {
  global_State *g = G(L);
  l_mem stepresult;
  lua_assert(!g->gcstopem);  /* collector is not reentrant */
  g->gcstopem = 1;  /* no emergency collections while collecting */
  switch (g->gcstate) {
    case GCSpause: {
      restartcollection(g);
      g->gcstate = GCSpropagate;
      stepresult = 1;
      break;
    }
    case GCSpropagate: {
      if (fast || g->gray == NULL) {
        g->gcstate = GCSenteratomic;  /* finish propagate phase */
        stepresult = 1;
      }
      else
        stepresult = propagatemark(g);  /* traverse one gray object */
      break;
    }
    case GCSenteratomic: {
      atomic(L);
      if (checkmajorminor(L, g))
        stepresult = step2minor;
      else {
        entersweep(L);
        stepresult = atomicstep;
      }
      break;
    }
    case GCSswpallgc: {  /* sweep "regular" objects */
      sweepstep(L, g, GCSswpfinobj, &g->finobj, fast);
      stepresult = GCSWEEPMAX;
      break;
    }
    case GCSswpfinobj: {  /* sweep objects with finalizers */
      sweepstep(L, g, GCSswptobefnz, &g->tobefnz, fast);
      stepresult = GCSWEEPMAX;
      break;
    }
    case GCSswptobefnz: {  /* sweep objects to be finalized */
      sweepstep(L, g, GCSswpend, NULL, fast);
      stepresult = GCSWEEPMAX;
      break;
    }
    case GCSswpend: {  /* finish sweeps */
      checkSizes(L, g);
      g->gcstate = GCScallfin;
      stepresult = GCSWEEPMAX;
      break;
    }
    case GCScallfin: {  /* call finalizers */
      if (g->tobefnz && !g->gcemergency && luaD_checkminstack(L)) {
        g->gcstopem = 0;  /* ok collections during finalizers */
        GCTM(L);  /* call one finalizer */
        stepresult = CWUFIN;
      }
      else {  /* no more finalizers ... */
        g->gcstate = GCSpause;  /* finish collection */
        stepresult = step2pause;
      }
      break;
    }
    default: lua_assert(0); return 0;
  }
  g->gcstopem = 0;
  return stepresult;
}
```

几个要点:

1. **一次调用只做一个状态的一小段工作**。`propagate` 阶段每次只调 `propagatemark` 遍历**一个**灰对象;清扫阶段每次最多扫 `GCSWEEPMAX`(=`20`,`lgc.c:34`)个对象;终结阶段每次只调**一个** finalizer。
2. **返回值 `stepresult` 是这一步做了多少"工作量"**(单位是遍历了多少 slot/对象)。上层 `incstep` 用它来扣减"这一轮 step 该做多少工作"的预算,预算耗尽就交还控制权。
3. **特殊返回值**:三个负数(`step2pause=-3`/`atomicstep=-2`/`step2minor=-1`,`lgc.c:1612-1614`)是给上层发信号——这轮到原子步了、或者整轮结束了、或者从增量模式切到分代小回收了。这些信号让 `incstep` 知道"原子步这种不可中断的步骤做完后,要不要立刻停下来"(见 2.7)。
4. **`g->gcstopem`** 是防止重入的标志:GC 自己运行期间不再触发紧急 GC(`lgc.c:1621`),避免递归。

把这一段走完,就是一轮完整的增量 GC。但"什么时候调一次 `singlestep`"是上层 `incstep`/`luaC_step` 管的事,2.7 详讲。先看每个状态具体干什么。

### 2.4 标记根:`restartcollection`

GC 从 `GCSpause` 进入 `GCSpropagate` 的瞬间,先标根。根有四类(`lgc.c:440`):

```c
// lgc.c:440-447
static void restartcollection (global_State *g) {
  cleargraylists(g);
  g->GCmarked = 0;
  markobject(g, mainthread(g));         /* 主线程 */
  markvalue(g, &g->l_registry);         /* 注册表(全局环境) */
  markmt(g);                            /* 基本类型的元表数组 */
  markbeingfnz(g);                      /* 上轮残留待终结对象 */
}
```

- `mainthread(g)`:`global_State` 里内嵌的主线程(见 P1-02),是所有协程的根。
- `g->l_registry`:注册表 `Table`,全局变量、加载的模块全挂在它的链条上。
- `markmt(g)`(`lgc.c:378`):遍历 `g->mt[LUA_NUMTYPES]`,这是基本类型(string、number 等)共享的元表。
- `markbeingfnz(g)`(`lgc.c:388`):上轮标记为"待调 finalizer"的对象。它们虽然逻辑上要死,但 finalizer 还没跑完前必须当活物处理(finalizer 可能"复活"它们)。

根都被 `markobject`/`markvalue` 调 `reallymarkobject`(`lgc.c:339`)处理。`reallymarkobject` 的关键行为是按类型分流:

```c
// lgc.c:339-372 (节选)
static void reallymarkobject (global_State *g, GCObject *o) {
  g->GCmarked += objsize(o);
  switch (o->tt) {
    case LUA_VSHRSTR:
    case LUA_VLNGSTR: {
      set2black(o);  /* nothing to visit */   // 字符串没有子对象,直接标黑
      break;
    }
    case LUA_VUPVAL: {
      UpVal *uv = gco2upv(o);
      if (upisopen(uv))
        set2gray(uv);  /* open upvalues are kept gray */   // 开放 upvalue 保持灰
      else
        set2black(uv);  /* closed upvalues are visited here */
      markvalue(g, uv->v.p);  /* mark its content */
      break;
    }
    case LUA_VUSERDATA: {
      Udata *u = gco2u(o);
      if (u->nuvalue == 0) {  /* no user values? */
        markobjectN(g, u->metatable);
        set2black(u);  /* nothing else to mark */
        break;
      }
      /* else... */
    }  /* FALLTHROUGH */
    case LUA_VLCL: case LUA_VCCL: case LUA_VTABLE:
    case LUA_VTHREAD: case LUA_VPROTO: {
      linkobjgclist(o, g->gray);  /* to be visited later */   // 复杂对象进灰链
      break;
    }
    default: lua_assert(0); break;
  }
}
```

两类对象直接标黑,不再进灰链:

- **字符串**:没有指向其他 GC 对象的指针,标完即结束。
- **无 uservalue 的 `userdata`**:只有一个可选元表,顺手标了就完。
- **closed upvalue**:内容即一个值,标了内容就完。

而 **Table、闭包、Proto、线程、带 uservalue 的 userdata、open upvalue** 这些"有子对象"的,进 `g->gray` 灰链,留给 `propagate` 阶段慢慢遍历。`linkobjgclist`(`lgc.c:197`)在链入的同时把对象涂成灰:

```c
// lgc.c:186-191
static void linkgclist_ (GCObject *o, GCObject **pnext, GCObject **list) {
  lua_assert(!isgray(o));  /* cannot be in a gray list */
  *pnext = *list;
  *list = o;
  set2gray(o);  /* now it is */
}
```

注意 `linkgclist_` 那条断言 `!isgray(o)`——一个对象不会同时挂在两个灰链上。这是后续多种灰链(`gray`/`grayagain`/`weak`/`ephemeron`/`allweak`)能正确运作的前提。

### 2.5 增量传播:`propagatemark`

`GCSpropagate` 是增量 GC **最主要的增量段**。每次 `singlestep` 在这个状态下只调一次 `propagatemark`(`lgc.c:727`),处理灰链头部那一个对象:

```c
// lgc.c:727-740
static l_mem propagatemark (global_State *g) {
  GCObject *o = g->gray;
  nw2black(o);                       // 先把自己涂黑
  g->gray = *getgclist(o);           // 从灰链摘下自己
  switch (o->tt) {
    case LUA_VTABLE: return traversetable(g, gco2t(o));
    case LUA_VUSERDATA: return traverseudata(g, gco2u(o));
    case LUA_VLCL: return traverseLclosure(g, gco2lcl(o));
    case LUA_VCCL: return traverseCclosure(g, gco2ccl(o));
    case LUA_VPROTO: return traverseproto(g, gco2p(o));
    case LUA_VTHREAD: return traversethread(g, gco2th(o));
    default: lua_assert(0); return 0;
  }
}
```

模式很规整:**先把灰对象涂黑,再遍历它的子对象(把白的子对象标灰、推进灰链)**。涂黑在前是关键——这保证处理完这个对象后,它和它的所有子引用都已"上色",不会再被当成未处理。各种 `traverse*` 函数做的就是"沿这个对象的所有指针走,对每个白色子对象调 `reallymarkobject` 把它推进灰链"。

以 `traversetable`(`lgc.c:608`)为例:

```c
// lgc.c:608-625
static l_mem traversetable (global_State *g, Table *h) {
  ...
  switch (h->flags & maskcolors) {
    case 0:                                   // 强表
      traversestrongtable(g, h);
      break;
    case 1:                                   // 弱值表
      traverseweakvalue(g, h);
      break;
    case 2:                                   // ephemeron 表(弱键)
      traverseephemeron(g, h, 0);
      break;
    case 3:  /* all weak; nothing to traverse */
      if (g->gcstate == GCSpropagate)
        linkgclist(h, g->grayagain);
      break;
  }
  ...
}
```

这里出现了一个增量 GC 的关键设计:弱表的处理需要特殊对待。

- **强表**(`traversestrongtable`,`lgc.c:576`):正常遍历,所有值直接标灰。
- **弱值表**(`traverseweakvalue`,`lgc.c:487`):值是弱的,不该因为表活着就让值活。但表的遍历结果在增量期间可能被业务修改,所以 `lgc.c:502` 把它**同时挂到 `grayagain` 链**,留到 atomic 步再扫一遍。
- **ephemeron 表(弱键)**(`traverseephemeron`,`lgc.c:541`):键的活死决定值的活死,但键的活死又可能反过来被值影响(值引用了键),存在循环依赖。这类表的标记不能在一次扫描里定论,需要收敛。`lgc.c:564-568` 把它同时挂到 `grayagain` 和 `ephemeron` 两条链,留到 atomic 步用 `convergeephemerons`(`lgc.c:755`)反复传播直到收敛。

弱表和 ephemeron 表是 GC 篇最难的一块,细节留到 [P5-17 对象回收与弱表](P5-17-对象回收与弱表.md) 展开。这里只需记住:**增量传播阶段对弱表只是"初扫 + 自挂 grayagain",真正的判定推迟到不可中断的 atomic 步**——因为弱表的活死判定要求在某一瞬间看到完整的可达图,增量期间业务还在改,初扫结果不可信。

线程的遍历(`traversethread`,`lgc.c:695`)同样有个增量细节:

```c
// lgc.c:695-699 (节选)
static l_mem traversethread (global_State *g, lua_State *th) {
  UpVal *uv;
  StkId o = th->stack.p;
  if (isold(th) || g->gcstate == GCSpropagate)
    linkgclist(th, g->grayagain);  /* insert into 'grayagain' list */
  ...
}
```

线程的值栈在增量期间会被业务持续改写(VM 执行就在改栈),所以即便 `propagatemark` 已经遍历过它的栈、把它涂黑了,也得**重新挂回 `grayagain`**,在 atomic 步再扫一次栈,确保没有遗漏被业务新压进来的值。这正是 `keepinvariant` 注释里说的"增量期间业务可能修改对象图,某些对象必须被重访"。

整个 `GCSpropagate` 阶段就是不断 `propagatemark`,直到 `g->gray` 空了,`singlestep` 才把状态推进到 `GCSenteratomic`(`lgc.c:1629-1637`)。这一步是可中断的——每次只处理一个对象,处理完立刻返回,上层可以选择继续还是把控制权还给业务。

### 2.6 原子步:`atomic`

进入 `GCSenteratomic` 后,`singlestep` 直接调 `atomic(L)`(`lgc.c:1640`)。这一步**不可中断**——一次跑完所有剩余的标记工作。为什么必须原子?下面讲。先看它干了什么(`lgc.c:1543`):

```c
// lgc.c:1543-1581
static void atomic (lua_State *L) {
  global_State *g = G(L);
  GCObject *origweak, *origall;
  GCObject *grayagain = g->grayagain;  /* save original list */
  g->grayagain = NULL;
  lua_assert(g->ephemeron == NULL && g->weak == NULL);
  lua_assert(!iswhite(mainthread(g)));
  g->gcstate = GCSatomic;
  markobject(g, L);                  /* mark running thread */
  /* registry and global metatables may be changed by API */
  markvalue(g, &g->l_registry);
  markmt(g);                         /* mark global metatables */
  propagateall(g);                   /* empties 'gray' list */
  /* remark occasional upvalues of (maybe) dead threads */
  remarkupvals(g);
  propagateall(g);                   /* propagate changes */
  g->gray = grayagain;
  propagateall(g);                   /* traverse 'grayagain' list */
  convergeephemerons(g);
  /* at this point, all strongly accessible objects are marked. */
  /* Clear values from weak tables, before checking finalizers */
  clearbyvalues(g, g->weak, NULL);
  clearbyvalues(g, g->allweak, NULL);
  origweak = g->weak; origall = g->allweak;
  separatetobefnz(g, 0);             /* separate objects to be finalized */
  markbeingfnz(g);                   /* mark objects that will be finalized */
  propagateall(g);                   /* remark, to propagate 'resurrection' */
  convergeephemerons(g);
  /* at this point, all resurrected objects are marked. */
  /* remove dead objects from weak tables */
  clearbykeys(g, g->ephemeron);
  clearbykeys(g, g->allweak);
  /* clear values from resurrected weak tables */
  clearbyvalues(g, g->weak, origweak);
  clearbyvalues(g, g->allweak, origall);
  luaS_clearcache(g);
  g->currentwhite = cast_byte(otherwhite(g));  /* flip current white */
  lua_assert(g->gray == NULL);
}
```

`atomic` 干的事按顺序:

1. **重标当前线程、注册表、元表**(`lgc.c:1551-1554`):业务代码在增量期间可能改了它们(比如 C API 往注册表塞了新值),这一步把可能被漏掉的根重新标一遍。
2. **清空 `gray` 链**(`propagateall`,`lgc.c:743`):把增量传播剩下的灰对象全扫完。
3. **`remarkupvals`**(`lgc.c:406`):处理可能死掉的线程上的开放 upvalue——这些 upvalue 引用的对象可能还没被标,得补标。注意它顺带把已经扫过的线程从 `twups` 链摘掉。
4. **扫 `grayagain`**(`lgc.c:1559-1560`):这是关键。增量期间所有"需要重访"的对象(Table、线程、弱表等)都挂在这里,这一步把它们再扫一遍,确保完整。
5. **`convergeephemerons`**(`lgc.c:755`):处理 ephemeron 表的循环依赖,反复传播标记直到不再变化。
6. **清弱表的死值**(`clearbyvalues`,`lgc.c:1564-1565`):弱值表里值是死的,这里清掉(`P5-17` 详讲)。
7. **`separatetobefnz` + `markbeingfnz` + `propagateall`**(`lgc.c:1567-1569`):把带 `__gc` 元方法、本轮要死的对象"分离"出来挂到 `tobefnz` 链(它们要先调 finalizer 才能回收),并把这些对象重新标记成活的——这就是**复活(resurrection)**。finalizer 执行时,这些对象及其引用链都还是活的,得保证不被本轮清扫。
8. **再收敛 ephemeron + 清死键**(`lgc.c:1570-1577`):复活动作可能让某些 ephemeron 的键又活了,再扫一遍;然后清掉真正死的键。
9. **翻转当前白**(`lgc.c:1579`):`g->currentwhite = otherwhite(g)`。这是 atomic 步最具标志性的动作——从这一刻起,"当前白"变成另一套,后续新建的对象都用新白;本轮清扫将回收所有"旧白"对象。

为什么 atomic 必须不可中断?核心原因:**它要重建一个"瞬间一致"的可达图**。增量传播期间,业务一直在改对象图,GC 看到的图是流动的;弱表判定、复活判定、白色翻转这些操作,如果允许被打断,业务代码一插手就可能让"已经判定为死/活"的对象状态失效,导致回收出错。原子步保证这几步在一个不被打扰的瞬间完成,锁死本轮回收的判定结果。`lua_assert(g->gray == NULL)` 在结尾确认所有灰对象都处理完了,不变式干净地重建。

代价是:这一步是整轮 GC 里**唯一的真正停顿点**。但它的工作量是受控的——因为增量传播已经把绝大多数对象标过了,atomic 只剩"剩余传播 + 弱表/复活处理",通常远小于一次完整标记。这就是增量 GC 的核心权衡:**把标记的大头摊平到增量步,把必须原子的尾巴留在 atomic**。

atomic 完成后,`entersweep(L)`(`lgc.c:1643` → `lgc.c:1504`)把状态推进到清扫:

```c
// lgc.c:1504-1509
static void entersweep (lua_State *L) {
  global_State *g = G(L);
  g->gcstate = GCSswpallgc;
  lua_assert(g->sweepgc == NULL);
  g->sweepgc = sweeptolive(L, &g->allgc);
}
```

### 2.7 增量清扫:`sweeplist` / `sweepstep`

清扫阶段把 `allgc` 链上的旧白对象(本轮垃圾)释放掉,把活对象重新标白(为下一轮准备)。核心是 `sweeplist`(`lgc.c:892`):

```c
// lgc.c:892-909
static GCObject **sweeplist (lua_State *L, GCObject **p, l_mem countin) {
  global_State *g = G(L);
  int ow = otherwhite(g);
  int white = luaC_white(g);  /* current white */
  while (*p != NULL && countin-- > 0) {
    GCObject *curr = *p;
    int marked = curr->marked;
    if (isdeadm(ow, marked)) {  /* is 'curr' dead? */
      *p = curr->next;  /* remove 'curr' from list */
      freeobj(L, curr);  /* erase 'curr' */
    }
    else {  /* change mark to 'white' and age to 'new' */
      curr->marked = cast_byte((marked & ~maskgcbits) | white | G_NEW);
      p = &curr->next;  /* go to next element */
    }
  }
  return (*p == NULL) ? NULL : p;
}
```

死活判定靠 `isdeadm(ow, marked)`:只要对象带的是**上一轮的白**(atomic 翻转后,`otherwhite` 指向的就是旧白),就是死的,释放之;否则是活的,清掉所有颜色位和 age 位、重新涂上当前白、age 设为 `G_NEW`,为下一轮做准备。

清扫是增量的——每次最多扫 `GCSWEEPMAX=20` 个对象(`sweepstep`,`lgc.c:1588`):

```c
// lgc.c:1588-1596
static void sweepstep (lua_State *L, global_State *g,
                       lu_byte nextstate, GCObject **nextlist, int fast) {
  if (g->sweepgc)
    g->sweepgc = sweeplist(L, g->sweepgc, fast ? MAX_LMEM : GCSWEEPMAX);
  else {  /* enter next state */
    g->gcstate = nextstate;
    g->sweepgc = nextlist;
  }
}
```

`sweepgc` 是个 `GCObject **` 游标,记住当前扫到 `allgc`/`finobj`/`tobefnz` 哪条链的哪个位置。一条链扫完(`sweeplist` 返回 `NULL`)就切到下一个状态、下一条链。整个清扫分四步(`singlestep` 里的 `GCSswpallgc`/`GCSswpfinobj`/`GCSswptobefnz`/`GCSswpend`),分别扫普通对象链、带 finalizer 对象链、待终结对象链,最后 `swpend` 里 `checkSizes`(`lgc.c:935`)顺手收缩过大的字符串表。

为什么清扫能增量而标记大部分能增量、唯独 atomic 不能?因为清扫**不依赖不变式**:清扫时业务代码可以照常跑,新建对象都用当前(新)白,不会被误回收(死活只看旧白);清扫把活对象重新标白,这一刻白对象可能指向黑对象——但这没关系,因为本轮不再做标记了,下轮开始前所有对象都会回到白,不变式自动重建。`keepinvariant` 在清扫阶段为假(`lgc.h:58` 的 `(g)->gcstate <= GCSatomic` 把清扫排除在外),写屏障因此也走另一条路径(见 2.9)。

清扫完进入 `GCScallfin`,逐个调用 `tobefnz` 链上对象的 `__gc` finalizer。每个 `singlestep` 只调一个(`lgc.c:1669`):

```c
// lgc.c:1669-1681
case GCScallfin: {  /* call finalizers */
  if (g->tobefnz && !g->gcemergency && luaD_checkminstack(L)) {
    g->gcstopem = 0;  /* ok collections during finalizers */
    GCTM(L);  /* call one finalizer */
    stepresult = CWUFIN;
  }
  else {
    g->gcstate = GCSpause;  /* finish collection */
    stepresult = step2pause;
  }
  break;
}
```

`GCTM`(`lgc.c:968`)从 `tobefnz` 取一个对象、找它的 `__gc` 元方法、用 `luaD_callnoyield` 调它。finalizer 是用户 Lua 代码,可能分配内存、改对象图,所以调它时临时打开 GC(`g->gcstopem = 0`)。每个 finalizer 算 `CWUFIN=10` 个工作量单位(`lgc.c:40`)。finalizer 全调完,状态回到 `GCSpause`,一轮回收结束。下一轮什么时候开始由 `setpause`(`lgc.c:1122`)决定——见 2.10。

### 2.8 步长控制:`incstep` 与 `luaC_step`

现在回答"什么时候调 `singlestep`、调多少次"。入口是 `luaC_step`(`lgc.c:1740`):

```c
// lgc.c:1740-1760
void luaC_step (lua_State *L) {
  global_State *g = G(L);
  lua_assert(!g->gcemergency);
  if (!gcrunning(g)) {  /* not running? */
    if (g->gcstp & GCSTPUSR)  /* stopped by the user? */
      luaE_setdebt(g, 20000);
  }
  else {
    luai_tracegc(L, 1);
    switch (g->gckind) {
      case KGC_INC: case KGC_GENMAJOR:
        incstep(L, g);
        break;
      case KGC_GENMINOR:
        youngcollection(L, g);
        setminordebt(g);
        break;
    }
    luai_tracegc(L, 0);
  }
}
```

增量模式下(`KGC_INC`)走 `incstep`(`lgc.c:1710`):

```c
// lgc.c:1710-1728
static void incstep (lua_State *L, global_State *g) {
  l_mem stepsize = applygcparam(g, STEPSIZE, 100);
  l_mem work2do = applygcparam(g, STEPMUL, stepsize / cast_int(sizeof(void*)));
  l_mem stres;
  int fast = (work2do == 0);  /* special case: do a full collection */
  do {  /* repeat until enough work */
    stres = singlestep(L, fast);
    if (stres == step2minor)
      return;
    else if (stres == step2pause || (stres == atomicstep && !fast))
      break;  /* end of cycle or atomic */
    else
      work2do -= stres;
  } while (fast || work2do > 0);
  if (g->gcstate == GCSpause)
    setpause(g);  /* pause until next cycle */
  else
    luaE_setdebt(g, stepsize);
}
```

几个关键量:

- `stepsize = applygcparam(g, STEPSIZE, 100)`:这一步的"预算字节量"(由 `STEPSIZE` 参数控制,默认 `LUAI_GCSTEPSIZE = 200*sizeof(Table)`,`lgc.h:201`)。
- `work2do = applygcparam(g, STEPMUL, stepsize / sizeof(void*))`:把预算换算成"工作量单位",再乘上 `STEPMUL`(步长倍率,默认 `LUAI_GCMUL=200`,`lgc.h:198`)。意思是每分配一字节(以指针大小为单位)的工作量,GC 做 `STEPMUL/100` 个工作量单位。默认 200% 意味着 GC 跑得比分配快一倍,这样才能逐步追上并回收掉垃圾。
- `do { singlestep } while (work2do > 0)`:反复调 `singlestep`,每次扣减它返回的工作量,直到预算用完。
- **遇到原子步或整轮结束就停**(`lgc.c:1719`):`atomicstep` 信号表示刚做完不可中断的 atomic,这时无论如何先停一下,把控制权还给业务——这正是"atomic 之后的停顿点是 GC 让出 CPU 的关键时刻"。
- 预算用完但回收还没结束,`luaE_setdebt(g, stepsize)`(`lgc.c:1727`)把 GC 债务设为 `stepsize`,意思是"再分配这么多字节才会触发下一次 `luaC_step`"。

那 `luaC_step` 本身什么时候被调?这就是"分配驱动 GC"的核心——`luaC_condGC` 宏(`lgc.h:233`):

```c
// lgc.h:233-235
#define luaC_condGC(L,pre,pos) \
    { if (G(L)->GCdebt <= 0) { pre; luaC_step(L); pos;}; \
      condchangemem(L,pre,pos,0); }

#define luaC_checkGC(L)  luaC_condGC(L,(void)0,(void)0)
```

逻辑很简洁:**只要 GC 债务 `GCdebt` 降到 0 以下,就做一步 GC**。`pre`/`pos` 是可选的前后处理(比如保存 PC、修正栈顶),让"GC 可能在任意分配点插入"这件事对调用者透明。

`GCdebt` 怎么变?`luaM_realloc_` 等分配器每次分配/释放字节时,会增减 `GCtotalbytes`(已分配字节数)和 `GCdebt`(`lmem.c`,`gettotalbytes = GCtotalbytes - GCdebt`,`lstate.h:435`)。分配多了 `GCdebt` 变负,触发 `luaC_step`;`luaC_step` 里 `incstep` 做完一步把 `GCdebt` 设回正数,GC 就"睡着等"下一次分配把它再次推过零。

这套机制把 GC 完全和"内存分配量"绑定:**宿主不主动调度 GC,业务分配内存本身就是在驱动 GC**。这正是 Lua 嵌入性的精髓——宿主不需要任何 GC 线程、不需要定时器,只要业务在分配内存,GC 就在悄悄推进。

调用点遍布解释器(`lvm.c:1178` 的 `checkGC` 宏,在字节码主循环里)、C API(`lapi.c` 里 `lua_createtable`/`lua_pushstring` 等十多处 `luaC_checkGC`)、字符串驻留(`llex.c`/`lstring.c`)、编译器(`lparser.c:848`)。任何一个可能分配内存的入口都在收尾处 `checkGC`,确保 GC 债务不会无限累积。

`lvm.c:1178` 那个 `checkGC` 宏值得特别看一眼:

```c
// lvm.c:1178-1181
#define checkGC(L,c)  \
    { luaC_condGC(L, (savepc(ci), L->top.p = (c)), \
                         updatetrap(ci)); \
       luai_threadyield(L); }
```

`pre` 参数是 `(savepc(ci), L->top.p = (c))`——在调 `luaC_step` 之前先把当前 PC 存下来、把栈顶修正到正确位置(因为 `luaC_step` 内部可能调用 finalizer,会动栈)。`pos` 是 `updatetrap(ci)`,GC 回来后刷新陷阱状态。这是"GC 可能嵌套执行用户代码(finalizer)、却对解释器主循环完全透明"的工程处理。

### 2.9 写屏障:维持黑不指白

现在讲最关键的正确性机制——**写屏障**。增量 GC 期间,业务代码(VM 执行字节码、C API)随时可能执行"`黑对象 o` 新指向 `白对象 v`"这种赋值。一旦发生,不变式"黑不指白"就被破坏,如果不补救,`v` 会在本轮被误回收(它的引用者 `o` 是黑的,GC 不会再扫 `o`,于是 `v` 永远不会被标到)。

补救有两条路,对应两个屏障宏(`lgc.h:241-252`):

```c
// lgc.h:241-252
#define luaC_objbarrier(L,p,o) (  \
    (isblack(p) && iswhite(o)) ? \
        luaC_barrier_(L,obj2gco(p),obj2gco(o)) : cast_void(0))

#define luaC_barrier(L,p,v) (  \
    iscollectable(v) ? luaC_objbarrier(L,p,gcvalue(v)) : cast_void(0))

#define luaC_objbarrierback(L,p,o) (  \
    (isblack(p) && iswhite(o)) ? luaC_barrierback_(L,p) : cast_void(0))

#define luaC_barrierback(L,p,v) (  \
    iscollectable(v) ? luaC_objbarrierback(L, p, gcvalue(v)) : cast_void(0))
```

注意两个宏都先判 `isblack(p) && iswhite(o)`——只有在真的发生"黑指白"时才真正动手,否则是空操作。这是性能优化:绝大多数赋值并不触发屏障。

**前向屏障(forward barrier)`luaC_barrier_`**(`lgc.c:246`):

```c
// lgc.c:246-261
void luaC_barrier_ (lua_State *L, GCObject *o, GCObject *v) {
  global_State *g = G(L);
  lua_assert(isblack(o) && iswhite(v) && !isdead(g, v) && !isdead(g, o));
  if (keepinvariant(g)) {  /* must keep invariant? */
    reallymarkobject(g, v);  /* restore invariant */
    ...
  }
  else {  /* sweep phase */
    ...
    if (g->gckind != KGC_GENMINOR)
      makewhite(g, o);  /* mark 'o' as white to avoid other barriers */
  }
}
```

策略是**把被指的白对象 `v` 标记(标灰/黑),让它不再白**。这样 `o → v` 就不是"黑指白"了,不变式恢复。代价是要立刻遍历 `v` 的子对象(在 `reallymarkobject` 里),工作量较大。所以前向屏障用在"知道 `v` 是个简单对象、或者遍历它代价可控"的场景。

**后向屏障(back barrier)`luaC_barrierback_`**(`lgc.c:268`):

```c
// lgc.c:268-279
void luaC_barrierback_ (lua_State *L, GCObject *o) {
  global_State *g = G(L);
  lua_assert(isblack(o) && !isdead(g, o));
  ...
  if (getage(o) == G_TOUCHED2)  /* already in gray list? */
    set2gray(o);
  else  /* link it in 'grayagain' and paint it gray */
    linkobjgclist(o, g->grayagain);
  ...
}
```

策略反过来:**把黑父对象 `o` 涂灰、挂回 `grayagain` 链**,让 atomic 步重扫它。这样不用立刻遍历 `v` 的子对象——把工作推迟到 atomic。代价是 `o` 会被重扫(它原本已经标黑了,现在要再扫一次)。后向屏障用在"`o` 有很多子对象、立刻遍历代价大"的场景,典型是 **Table**:往一个黑 Table 里塞新元素时,不知道塞了多少个,与其每次都前向标记每个新值,不如直接把 Table 自己涂灰,atomic 时一次性重扫。

所以源码里:

- `luaC_barrier` 用在"父对象子引用较少、可知"的场景,如闭包设置 upvalue、`Proto` 设置常量。
- `luaC_barrierback` 用在 Table 改动(`luaH_set` 系列往表里写值),因为 Table 可能被反复写,前向标记代价高。

这就是"两策"的取舍:**前向省的是父对象重扫、花的是子对象立刻遍历;后向省的是子对象立刻遍历、花的是父对象重扫**。具体用哪个,看哪边代价小。

清扫阶段的屏障走另一条路(`lgc.c:256-260`):不变式此时本就不维持,所以把黑父对象 `o` 涂成白(`makewhite`),这样它就不再是黑的了,后续对它的赋值不会反复触发屏障——这是清扫阶段的性能优化。

### 2.10 GC 参数:pause、stepmul、stepsize

Lua 的 GC "多激进"由几个参数控制,5.5 把它们收敛进一个 `gcparams[LUA_GCPN]` 数组(`lstate.h:338`),用 `LUA_GCPARAM` 这个统一 API 读写(`lua.h:340`):

```c
// lua.h:347-357
#define LUA_GCPMINORMUL    0  /* control minor collections */
#define LUA_GCPMAJORMINOR  1  /* control shift major->minor */
#define LUA_GCPMINORMAJOR  2  /* control shift minor->major */
#define LUA_GCPPAUSE       3  /* size of pause between successive GCs */
#define LUA_GCPSTEPMUL     4  /* GC "speed" */
#define LUA_GCPSTEPSIZE    5  /* GC granularity */
#define LUA_GCPN           6
```

**★5.5 vs 老资料差异(重要)**:讲 5.3/5.4 的资料里,`lua_gc` 有 `LUA_GCSETPAUSE` 和 `LUA_GCSETSTEPMUL` 两个独立的 set 选项。5.5 删了这两个常量,改成统一的 `LUA_GCPARAM`(传 param 编号 + value,见 `lapi.c:1229`)。老资料里那些 `lua_gc(L, LUA_GCSETPAUSE, 200)` 的写法在 5.5 编不过,得改成 `lua_gc(L, LUA_GCPARAM, LUA_GCPPAUSE, 200)`。

增量模式相关的三个参数:

- **`PAUSE`**(默认 `LUAI_GCPAUSE=250`,`lgc.h:191`):两轮回收之间的间隔。一轮结束后,`setpause`(`lgc.c:1122`)算出下一轮的触发阈值 `≈ 本轮标记字节数 × PAUSE/100`。默认 250% 意味着"等到内存涨到上轮活对象的 2.5 倍才开始下一轮"——给业务留出足够喘息空间,GC 不至于频繁启动。
- **`STEPMUL`**(默认 `LUAI_GCMUL=200`,`lgc.h:198`):步长倍率。每分配一字节(指针大小为单位),GC 做 `STEPMUL/100` 个工作量单位。默认 200% 让 GC 跑得比分配快,逐步追平垃圾。
- **`STEPSIZE`**(默认 `LUAI_GCSTEPSIZE = 200*sizeof(Table)`,`lgc.h:201`):每一步 `luaC_step` 的预算字节量。控制单步的粒度——太大单步停顿久,太小 GC 调度开销大。

参数本身用一个**浮点字节**(floating-point byte,1 字节存 1~16384 范围的值)压缩存储,编码在 `luaO_codeparam`(`lobject.c:62`)、解码在 `luaO_applyparam`(`lobject.c:89`)。这是一种经典的紧凑编码:用 4 位尾数 + 4 位指数表示,把一个本该用 int 存的参数压进 1 字节。`applygcparam(g, PAUSE, x)` 就是 `luaO_applyparam(g->gcparams[LUA_GCPPAUSE], x)`,返回 `参数值 × x / 100`。

`setpause` 把 PAUSE 翻译成实际债务:

```c
// lgc.c:1122-1127
static void setpause (global_State *g) {
  l_mem threshold = applygcparam(g, PAUSE, g->GCmarked);
  l_mem debt = threshold - gettotalbytes(g);
  if (debt < 0) debt = 0;
  luaE_setdebt(g, debt);
}
```

`g->GCmarked` 是本轮标记到的活字节数(`restartcollection` 时清零,`reallymarkobject` 时累加)。阈值 = 活字节 × PAUSE%,debt = 阈值 − 当前已分配字节。这意味着:**GC 不会在内存还没涨到阈值前启动下一轮**,业务可以安心跑一段时间。

这套参数让宿主能精确调"GC 多激进":游戏引擎可以把 PAUSE 调高、STEPMUL 调低,换取更平滑的帧率;批处理脚本可以调激进些,尽快回收内存。Lua 默认值(250/200)是个偏保守的平衡点。

## 三、为什么这样设计是 sound 的

把上面的机制放在一起,验证它满足三条正确性要求:**不漏(垃圾被回收)、不重(活物不被误杀)、不留悬空指针**。

### 3.1 不漏:三色标记 + 翻转白保证旧白即死物

一轮回收的判定基准是 atomic 步翻转后的"旧白"。任何在 atomic 时刻可达的对象,要么在 atomic 之前已被增量传播标到(灰→黑)、要么在 atomic 步内被补标(`markobject(L)`/`markvalue(registry)`/`remarkupvals`/`propagateall(grayagain)`/`convergeephemerons`)。atomic 结束时 `lua_assert(g->gray == NULL)` 保证所有可达对象都已变黑或被处理。清扫时,凡是还带旧白色的对象就是 atomic 时不可达的,即垃圾,被 `freeobj` 释放。所以**只要 atomic 不被打断、所有可达对象都标到位,旧白必死、必被回收,不会漏**。

### 3.2 不重:写屏障 + atomic 重建维持黑不指白

增量期间业务可能让黑对象新指向白对象,破坏不变式。如果不补救,这个白对象会因"引用者是黑、GC 不再扫黑"而永远不被标到,被误当垃圾回收。写屏障两策正是来补救的:

- 前向屏障把被指白对象当场标记,它就不再白,不变式恢复。
- 后向屏障把黑父对象涂灰挂回 `grayagain`,atomic 步重扫它,届时它指向的所有白对象都会被标到。

无论哪策,atomic 开始时不变式都已恢复(`keepinvariant` 在 propagate/enteratomic/atomic 三个状态为真)。atomic 步内不受业务干扰,一次性把所有灰对象扫完、补标所有被屏障挂回的对象。所以**活物在 atomic 结束时一定不是旧白,清扫不会误杀**。

### 3.3 不留悬空指针:复活 + 分离 tobefnz

带 `__gc` 的对象本轮要死,但 finalizer 没跑前不能直接释放(finalizer 可能访问它、甚至让它重新可达)。atomic 步用 `separatetobefnz` 把这些对象从 `allgc` 移到 `tobefnz` 链、再用 `markbeingfnz` 把它们重新标活——这就是**复活**。它们和它们引用的对象本轮都不会被清扫。finalizer 在 `GCScallfin` 阶段逐个调用,调完一个对象才真正进下一轮的待回收队列。这保证了**任何 finalizer 执行时,它操作的对象及其引用链都还在,不会悬空**。

### 3.4 增量步的可中断性:原子步最小化

整套设计的精髓在于:**只有 atomic 一步不可中断,其余都切成小步**。propagate 每次一个对象、sweep 每次 20 个、callfin 每次一个 finalizer,每步之间业务都可以插进来跑。atomic 之所以必须原子,是因为它要做"瞬间一致"的判定(弱表、复活、翻白);但它的工作量被增量传播预先压到最小——大头标记都在 propagate 阶段摊掉了。结果就是:**GC 的停顿被切成无数次几微秒的小停顿 + 一次可控的 atomic 停顿,宿主几乎感受不到**。

这正是"GC 必须可中断才能嵌入"在源码层面的落实:不是靠某种调度魔法,而是靠**三色不变式 + 写屏障**让增量期间业务能安全穿插,再靠**状态机 + debt 步长**把工作切成受控的小步。

## 四、★对照 CPython + 回扣主线

| 维度 | Lua 5.5 增量三色标记 | CPython 引用计数 + 分代标记 |
|---|---|---|
| **主回收机制** | 追踪式:从根出发沿指针链标记可达,扫剩余 | 引用计数:每个对象带 refcnt,赋值即增减,归零即回收 |
| **回收时机** | 按步推进,延迟回收(本轮 atomic 后才清扫) | 即时:refcnt 归零当场释放 |
| **循环引用** | 三色标记天然处理(环上对象不可达就一起白) | refcnt 处理不了,靠独立的分代标记周期(那是一次 STW) |
| **停顿特征** | 增量小步 + 一次 atomic,停顿摊平到微秒级 | 每次指针赋值有 refcnt 开销(纳秒级,但极频繁);分代周期是一次完整停顿 |
| **对嵌入的友好度** | 高:停顿可控、可调参、业务穿插运行 | 中:refcnt 开销散布在每次赋值;分代 STW 在大堆上可能几十毫秒 |
| **实现复杂度** | 三色状态机 + 写屏障 + 多种灰链,约 1800 行 | refcnt 简单,但分代标记 + 弱引用处理同样复杂 |

两种 GC 体现了两种取舍:

- **CPython 的 refcnt** 追求**回收即时性**——对象一不可达立刻释放,内存占用峰值低,finalizer(`__del__`)时机确定。代价是每次指针赋值都要改 refcnt(高频小开销),且循环引用必须靠独立的分代周期兜底(那就是一次 STW,大堆上不可忽视)。
- **Lua 的增量三色**追求**停顿可控性**——把回收摊平,绝不出现长停顿,天生适合嵌入对延迟敏感的宿主(游戏、Redis)。代价是回收不即时(本轮垃圾要等本轮清扫才回收),且需要三色状态机 + 写屏障维持增量期间的正确性。

对 Lua 的生态位(嵌入游戏引擎、基础设施、嵌入式设备),**停顿可控比回收即时更重要**——一个游戏帧卡 200 毫秒是事故,垃圾晚几毫秒回收无所谓。所以 Lua 选增量三色是必然:它把"GC 永远不成为宿主的负担"做进了语言运行时的骨子里。

**回扣主线**:全书第三招"增量式三色 GC,可中断"在这一章落地。Lua 用三色不变式 + 写屏障 + 状态机 + debt 步长,把"回收内存"这件本来要长停顿的事,拆成无数可中断的小步,每步和内存分配绑定、由分配量自然驱动。这恰好是"适合嵌入"的硬要求:**宿主不需要为 GC 付出任何调度代价,GC 自己藏在每次分配的背后悄悄推进**。统一与精简换小而快——这里的"精简"不是代码少,而是**把 GC 对宿主的干扰压到最小**,让 Lua 能安心住进任何宿主。下一章 [P5-17 对象回收与弱表](P5-17-对象回收与弱表.md)继续讲清扫阶段的细节:弱表怎么清、`__gc` finalizer 怎么调、对象怎么真正被释放。
