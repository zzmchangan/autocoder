# 全书第一性原理(贯穿始终,反复回扣)

> **把 hyper 的"一个 Service 处理一个请求"升级成"路由分发到 handler、提取器自动反序列化请求、响应器自动序列化响应"的 Web 框架——底层仍是 hyper+Tower 的 Service 链,框架的魔力在 Handler/FromRequest/IntoResponse 用泛型+宏把任意 async fn 编译期变 Service。**

每写一章,先问自己:这一章讲的机制,是在**路由与分发这一面**(Router/PathRouter/matchit/MethodRouter),还是在**提取与响应这一面**(Handler trait/FromRequest/FromRequestParts/IntoResponse/提取器链)?这个机制 hyper/Tower 是怎么支撑的、对照 actix/rocket/go 怎么做?把答案写进"为什么"里。读者要的不是"axum API 怎么用",而是"它凭什么把 hyper 的 Service 链包装成这么好写的 Web 框架、源码里那些 Handler trait、impl_handler! 宏、FromRequest/FromRequestParts、Router 的 matchit 双层匹配、IntoResponse 到底在干什么"。

# 写作方法命脉:动机(why) + 技巧(how)双线 ⚠️

这是本书区别于其他系列的关键。axum 源码(Rust)充斥高级技巧(`Handler<T, S>` trait 的 T 是 coherence 占位 tuple、`impl_handler!` + `all_the_tuples!` 宏对 0~16 参数全部 impl Handler、FromRequestParts 只读 parts 可多次跑 vs FromRequest 消费 body 只能一次、ViaParts marker 桥接、`impl Future` 非 async-trait、Router 用 Arc&lt;RouterInner&gt; + PathRouter 双层、Node 包 matchit::Router&lt;RouteId&gt; + 两个 HashMap 双向映射、MethodRouter 按 method 持 MethodEndpoint、Route 是 BoxCloneSyncService 类型擦除……)。读者"一知半解"的缺口**不在动机**(好懂),**在技巧**(看不懂)。所以每个关键代码点必须讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?hyper/Tower 怎么支撑/对照 actix/rocket/go 怎么做/不这样会怎样?
2. **实现技巧(how 的精妙)**:axum 用了什么 Rust 手段(泛型 + 宏展开 + trait 约束 + 类型擦除)?为什么妙?**不这么写会怎样**(孤儿规则冲突 / 类型推断失败 / 提取器顺序错乱 / body 重复消费)?

**每章单开"技巧精解"小节**(正文后、小结前),挑本章**最硬核 1~2 个技巧**,配真实源码 + 行号,单独拆透,必须有**反面对比**("hyper/Tower/actix 怎么做 / 朴素地写会撞什么墙")。**讲不清 Handler trait 的 T 参数 / impl_handler! 宏展开 / FromRequestParts vs FromRequest 二元划分 / Router 的 matchit 双层匹配 = 没讲 axum。**

# ⚠️ 承接《hyper》《Tower》《Tokio》铁律(本书最大特色,动手前必读)

axum 直接长在 hyper+Tower 上,全异步跑在 Tokio。**承接铁律**:hyper/Tower/Tokio 已讲透的,一句带过 + 指路,篇幅全留 axum 独有。

- **承接《hyper》[[hyper-series-project]]/[[hyper-source-facts]]**:协议机(HTTP/1 状态机/HTTP/2 via h2)、连接管理(accept/keep-alive/连接池)、Service trait 本身(`call(&self) -> Future`,为什么 axum 的 Router/Route 都 impl 它)、Body as Stream、buffered IO、bytes 零拷贝、协议升级(websocket)——这些《hyper》已拆透,**一句带过 + 指路**(协议机指 P2-P3、Service 指 P1-02、连接管理指 P4-P5),别当新东西重复。axum 收到的 `Request<Body>` 已经是 hyper 解析好的。
- **承接《Tower》**(成网后,目前以"axum 怎么用 + 对照 hyper 怎么用"视角讲):Service/Layer trait 本身、ServiceBuilder 叠中间件、poll_ready 语义(axum 的 Router/Route poll_ready 无条件 Ready,忽略中间件背压)、BoxCloneSyncService 类型擦除的内部原理、SyncWrapper——这些 Tower 讲透的(成网后)一句带过。目前涉及 Tower 内部机制诚实标注"在 tower crate,本书引用其用法 + 对照 hyper 怎么用 Tower"。
- **承接《Tokio》[[tokio-source-facts]]**:运行时(task 调度/AsyncRead/AsyncWrite/timer/mpsc channel/budget),axum 全异步跑在 Tokio 上——Tokio 讲透的一句带过。Sse 用 channel、超时中间件用 timer、budget 让出,都指路 Tokio。
- **横连《gRPC》[[grpc-series-project]]**:axum 中间件链对照 gRPC filter stack(都是 Service 套娃),axum 的 Json 提取器对照 gRPC protobuf 反序列化,HTTP/2 承接一句带过。

**禁止把上面讲透的当新东西重复**——承接铁律硬伤,违者返工。

# 写前必做:源码与技巧核查(防止编造,这一步不能省)

1. **每个要引用的函数 / 结构体 / 行号,先用 Grep / Read 在本地 `../axum/` 核实**——**严禁凭记忆**。axum 演进快(0.7→0.8 有 breaking changes,main 在做 0.9),凭印象写必错。已核实的非显然事实(供参考,但仍要自己核):
   - `Handler<T, S>` trait 的 `T` 是 coherence 占位 tuple(`Handler<((),)>`、`Handler<(M, T1, T2,)>` 等),不是 handler 的某个具体类型。
   - `impl_handler!` 宏 + `all_the_tuples!` 宏展开 0~16 参数的 impl Handler,**最后一个参数 `$last: FromRequest<S, M>`**(可消费 body),其余 `$ty: FromRequestParts<S>`(只读 parts)。
   - `FromRequest<S, M = ViaRequest>` 带 marker,`FromRequestParts<S>` 无 marker;`impl FromRequest<S, ViaParts> for T where T: FromRequestParts` 是桥接(只读 parts 的提取器也能当 FromRequest 用)。
   - `Router<S>` 的 `S` 是"缺失"的状态类型,`with_state(state)` 消耗 S 变 `Router<()>`,只有 `Router<()>` 能交给 `serve`。
   - `PathRouter { routes: Vec<Endpoint<S>>, node: Arc<Node> }`,`Node { inner: matchit::Router<RouteId>, route_id_to_path, path_to_route_id }`,`RouteId(usize)` 索引 Vec。
   - `Router::route` 只接 `MethodRouter`(0.8 变动),`Router::route_service` 接任意 Service。
   - 路径参数 0.8 用 `{foo}`/`{*foo}`,0.7 的 `:foo`/`*foo` 会 panic(除非 `without_v07_checks`)。
   - `Router`/`Route`/`MethodRouter` 的 `poll_ready` 无条件 `Poll::Ready(Ok(()))`(忽略 Tower 中间件背压)。
   - `Route` 内部是 `BoxCloneSyncService<Request, Response, E>`(类型擦除的 Tower Service)。
   - `from_request`/`from_request_parts` 用 `impl Future`(非 async-trait,RPITIT)。
2. **每个代码块必须从真实文件摘录**(允许 `...` 省略,保留部分逐字一致);简化展示必须标注"(简化示意,非源码原文)"。
3. **涉及 hyper/Tower/Tokio 承接**:读对应 source-facts 确认真值,一句带过别讲错(尤其 hyper 的 Service trait/Tokio 的 Future/Poll)。
4. **每个技巧(Handler trait T 参数/impl_handler! 宏/FromRequestParts vs FromRequest/Router 双层匹配等),要能讲清"为什么 sound"**:为什么不重复消费 body、为什么编译期保证提取器顺序、为什么类型推断不爆、为什么 merge 不冲突、为什么 nest 路径拼接正确。
