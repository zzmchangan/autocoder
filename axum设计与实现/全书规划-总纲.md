## 一、这本书的定位

### 目标读者

用 axum 写过 Web、用 `Router::new().route("/", get(handler))` 起过服务、知道底层是 hyper+Tower、看过几篇 axum 源码分析,但**一知半解**:知道有 Handler、FromRequest、IntoResponse、Router、MethodRouter、State、`#[axum::debug_handler]`,却讲不清一次 axum 请求从 hyper 把 `Request` 交上来,到 `Router` 经 matchit 路径匹配 → `MethodRouter` 按方法分发 → `Handler::call` 用 `impl_handler!` 宏展开的 tuple 提取器链逐个 `from_request_parts` → handler fn 真正被调用 → 返回值 `into_response` 写回,中间怎么和 hyper 的 `Service`、Tower 的 `Layer`、Tokio 的运行时无缝拼起来。和"读过 hyper 源码但不理解"、"翻过 gRPC 源码却一知半解"是同一种病——**"动机好懂,技巧看不懂"**。

### 全书一句话主旨

> **axum 做的事,本质是把 hyper 的"一个 Service 处理一个请求"升级成"路由分发到 handler、提取器自动反序列化请求、响应器自动序列化响应"的 Web 框架——但底层仍是 hyper+Tower 的 Service 链,框架的魔力在 Handler/FromRequest/IntoResponse 这些 trait 怎么用 Rust 的泛型 + 宏展开,把任意 `async fn` 在编译期变成一个满足 `tower::Service` 的对象。**

每个组件,都是"Web 框架 × hyper/Tower"拼合的一个环节。读完一本书,读者不是"多记了几个 axum API",而是能**在脑子里放映出一次 axum 请求的全过程**:hyper 把 `Request` 交给 `Router::call` → `PathRouter::call_with_state` 用 `matchit` 字典树匹配路径拿 `RouteId` → 索引 `Vec<Endpoint>` 拿到 `MethodRouter` → `MethodRouter::call_with_state` 按 HTTP method 选 handler → `Handler::call`(由 `impl_handler!` 宏对 tuple 展开生成)把 `Request` 拆成 `parts + body`,逐个跑 `FromRequestParts::from_request_parts` 提取器链,最后一个参数走 `FromRequest::from_request`(可能消费 body)→ 调用真正的 `async fn` handler → 返回值 `IntoResponse::into_response` 拼成 `Response` 交回 hyper——以及这每一步底下,hyper/Tower/Tokio 是怎么被用起来的(承接)。

### 为什么用"路由与分发 vs 提取与响应"当骨架(而非按模块平铺)

axum 的模块(routing/handler/extract/response/middleware)高度耦合:一次请求要穿过路由匹配、被 handler 的提取器链拆解、handler fn 执行、返回值经 IntoResponse 编码回 Response,中间 middleware 还要插一层 Tower Service。按模块百科式平铺,读者会觉得每块都是孤岛。用"**路由与分发(怎么把 URL+method 路由到那个 handler fn) vs 提取与响应(handler fn 的参数怎么自动从 Request 来、返回值怎么自动变成 Response)**"这条主轴当骨架,每个模块都是"为了让 Web 框架好写而必须出现的角色"。

## 二、写作方法:动机(why) + 技巧(how)双线 ⚠️ 本系列命脉

> **这一节是本书与数学系列、其他源码系列最关键的方法论,所有写作者必读、必守。**

axum 源码(Rust)里充斥高级技巧:`Handler<T, S>` trait 用类型参数 `T`(coherence 占位 tuple)绕开孤儿规则,对 1~16 个参数的 `async fn` 全部实现 `Handler`(`impl_handler!` 宏 + `all_the_tuples!` 宏的递归展开);提取器链用 `FromRequestParts`(只读 `&mut Parts`、可多次跑)与 `FromRequest`(消费 body、只能跑一次)的二元划分,且 `impl FromRequest<S, ViaParts> for T where T: FromRequestParts` 这个桥接让"只读 parts 的提取器也能当 FromRequest 用";`Router` 内部用 `Arc<RouterInner<S>>` + `PathRouter { routes: Vec<Endpoint>, node: Arc<Node> }`,`Node` 包 `matchit::Router<RouteId>` + 两个 `HashMap` 做 `RouteId ↔ path` 双向映射;`MethodRouter` 按 HTTP method 持有 `MethodEndpoint`;`Route` 是 `BoxCloneSyncServiceLayer` 的类型擦除……用户自己读过源码,"一知半解"的缺口往往**不在**"它要解决什么问题"(动机好懂),而**在**"它怎么用这么巧的 Rust 类型系统手段解决的"(技巧看不懂)。

所以本书的每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?hyper/Tower 是怎么提供支撑的、不这样会怎样?
2. **实现技巧(how 的精妙)**:axum 用了什么 Rust 手段(泛型 + 宏展开 + trait 约束 + 类型擦除)?为什么妙?**不这么写会怎样**(类型推断失败 / 孤儿规则冲突 / 性能损失 / 提取器顺序错乱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果朴素地实现 / 别的系统(actix-web / rocket / go net/http)怎么做",让妙处显形。

一句话:**讲不清 Handler trait 的 `T` 参数、`impl_handler!` 宏展开、FromRequestParts/FromRequest 二元划分、Router 的 matchit 双层匹配、MethodRouter 按 method 分发,等于没讲 axum。**

> **⚠️ 深度标杆(用户强调,本书命脉)**:本书篇幅标杆拉到对标《SQLite》《RocksDB》《gRPC》《Envoy》《hyper》《MySQL》实际深度——**每章正文约 50000~68000 字符(招牌章奔 70000+)**。用户原话:"不要吝啬文字和篇幅,内容可以更丰富一点,要把问题讲解清楚,深入浅出,好理解又有深度。"所以每个机制至少讲透三层(动机 → hyper/Tower 怎么支撑/不这样会怎样 → axum 怎么实现 + 为什么 sound),宁可多铺反例、对照(actix-web 的 Handler / rocket 的 request guard / go net/http 的 ServeMux、hyper 直接写 Service vs axum 包一层)、图,把问题讲到读者"原来如此",也不要一笔带过。与《gRPC》《TiKV》《RocksDB》《JVM》《hyper》一致。

## 三、全书主线、二分法、主比喻、承接

### 主线

> **把 hyper 的"一个 Service 处理一个请求"升级成"路由分发到 handler、提取器自动反序列化、响应器自动序列化"的 Web 框架——底层仍是 hyper+Tower 的 Service 链,框架的魔力在 Handler/FromRequest/IntoResponse 用泛型+宏把任意 async fn 编译期变 Service。**

任何一处看不懂某个机制,回到这句问:"这是在**路由与分发这一面**(URL+method 怎么路由到 handler fn:Router/PathRouter/matchit/MethodRouter),还是在**提取与响应这一面**(handler fn 的参数怎么从 Request 来、返回值怎么变成 Response:Handler trait/FromRequest/FromRequestParts/IntoResponse/提取器链)?这个机制 hyper/Tower 是怎么支撑的?"

### 二分法

本书的二分法:

> **路由与分发(Router/PathRouter 的 matchit 路径匹配 + MethodRouter 的按方法分发) vs 提取与响应(Handler trait 的编译期魔法 + FromRequest/FromRequestParts 提取器链 + IntoResponse 响应器)。**

- **路由与分发这一面**:`routing/mod.rs`(Router/RouterInner/Endpoint/Fallback)、`routing/path_router.rs`(PathRouter + Node 包 matchit::Router<RouteId> + 两个 HashMap 双向映射)、`routing/method_routing.rs`(MethodRouter 按 HTTP method 持有 MethodEndpoint)、`routing/route.rs`(Route = 类型擦除的 BoxCloneSyncService)、`routing/method_filter.rs`(MethodFilter)、`routing/strip_prefix.rs`、`routing/url_params.rs`、`routing/nest` 的 StripPrefix+SetNestedPath——这些决定"一个请求按 URL+method 该交给哪个 handler"。
- **提取与响应这一面**:`handler/mod.rs`(Handler trait + `impl_handler!` 宏 + Layered/HandlerService/HandlerWithoutStateExt)、`axum-core/src/extract/mod.rs`(FromRequest/FromRequestParts 两个 trait + ViaRequest/ViaParts marker + Result/Option 桥接)、`axum-core/src/extract/request_parts.rs`、`axum-core/src/extract/tuple.rs`(tuple 的 FromRequestParts 组合)、`axum/src/extract/`(Path/Query/State/Form/Json/multipart/ws 等具体提取器)、`axum-core/src/response/into_response.rs`、`into_response_parts.rs`、`axum/src/response/`(Redirect/Sse)——这些决定"handler fn 的参数怎么自动从 Request 提取出来、返回值怎么自动变成 Response"。

### 承接《hyper》(本书最大特色,务必遵守)

axum 直接长在 hyper 上,承接最紧密。**承接铁律**:hyper 已讲透的,一句带过 + 指路 [[hyper-series-project]]/[[grpc-source-facts 不,是 hyper]],篇幅全留 axum 独有。具体承接点:

- **每个连接一个 hyper task**:axum 用 `axum::serve(listener, router)`(内部 hyper-util)accept 连接,每连接 spawn task + 跑 HTTP 协议机。hyper 的 accept/连接管理/HTTP/1 状态机/HTTP/2 via h2([[hyper-source-facts 不,hyper]])**一句带过指路《hyper》P5/P2-P3**。
- **HTTP 协议机**:axum 收到的 `Request<Body>` 已经是 hyper 解析好的(`http::Request`),body 是 `axum_core::body::Body`(包 hyper body)。**协议机一句带过指路《hyper》第 2~3 篇**。
- **Service 模型**:axum 的 `Router`、`MethodRouter`、`Route`、`HandlerService` 全部实现 `tower::Service<Request>`,承 hyper 的 Service trait 模型(`call(&self) -> Future`)。**hyper 的 Service 一句带过指路《hyper》P1-02**。
- **Serve 与 hyper-util**:`axum::serve` 内部走 `hyper-util` 的 `server::conn::http1/http2`,承 hyper 的 server 端口。一句带过。

**禁止把 hyper 讲透的(协议机/连接池/keep-alive/buffered IO/Service trait 本身/body Stream)当新东西重复**——那是承接铁律的硬伤,违者返工。

### 承接《Tower》(中间件与 Service 抽象)

axum 的中间件就是 Tower Layer,axum 的所有可路由对象都是 `tower::Service<Request>`。**承接铁律**:Tower 讲透的(Service/Layer/SyncWrapper/ServiceBuilder/poll_ready 语义)一句带过 + 指路。具体:

- **中间件 = Tower Layer**:`Router::layer` / `route_layer` / `MethodRouter::layer` / `Handler::layer` 全部接 `tower_layer::Layer<Route>`,中间件链就是 Tower 的 Service 套娃。axum 的 `middleware::from_fn` / `from_extractor` 只是把闭包/提取器包成 Layer。
- **Service 套娃**:`Route` 内部是 `BoxCloneSyncService<Request, Response, E>`(类型擦除的 Tower Service)。承 Tower 的 `Service`/`Layer`/`util`。一句带过。
- **ServiceExt/ServiceBuilder**:axum 重导出 `tower::ServiceExt`、`tower::ServiceBuilder`。一句带过。

> **注**:《Tower》成网后单独成书,目前本书以"axum 怎么用 Tower"的视角讲,涉及 Tower 内部机制(BoxCloneSyncService 的类型擦除怎么做、Layer 套娃的 poll_ready 语义)诚实标注"在 tower crate 内部,本书引用其用法 + 对照 hyper 怎么用 Tower"。

### 承接《Tokio》

axum 全异步,所有 handler 是 `async fn`,所有提取器返回 `Future`,serve 跑在 Tokio 运行时上。**一句带过指路 [[tokio-source-facts]]**:task 调度/AsyncRead/AsyncWrite/同步原语(mpsc channel 给 Sse 用)/timer(超时中间件)/budget——Tokio 讲透的一句带过。

### 横连《gRPC》(HTTP/2 + filter chain 对照)

- axum 中间件链对照 gRPC 的 interceptor/filter stack(都是 Service 套娃,但 gRPC 是 C++ callback→Promise 大重构,axum 是 Tower Layer)。一句带过对照。
- axum 的 `Json` 提取器对照 gRPC 的 protobuf 反序列化(都是"请求体自动反序列化",但 axum 用 serde,gRPC 用 protobuf C++ codegen)。

### 主比喻决策:直球为主,比喻点睛(沿用系列)

> **直球为主**:绝大多数地方第一性原理直给,不套比喻。**比喻点睛**:仅在开篇(P0-01)做一次性点位睛——hyper 是"快递公司的收发室"(收每一个连接的包裹),Tower 是"公司内部的流程规范"(谁套谁、谁审批谁),axum 是"前台调度员+翻译官"(看收件地址 URL 选部门、看方法 method 选工位、把包裹内容 Request 翻译成 handler 能懂的参数、把 handler 的回复翻译成 Response 包裹发出去)。全书不建立贯穿重比喻。沿用《RocksDB》《gRPC》《hyper》定调。

### 第 0 篇 · 开篇:为什么 hyper 之上需要 axum(定调,1 章)

- **第 1 章 · 第一性原理:为什么 hyper 之上需要 axum**(`P0-01`):hyper 提供了"一个连接一个 Service",但写 Web 要手写路由(if path==)、手写反序列化、手写每条路径的 Service 套娃——又长又易错。axum 干的就是把这些机械化的事用 Rust 类型系统+宏做成零成本抽象:`Router` 管 URL+method 分发,`Handler` trait 用宏把任意 `async fn` 编译期变 Service,`FromRequest` 把参数自动从 Request 提,`IntoResponse` 把返回值自动变成 Response。承 hyper(一句带过协议机/Service)、承 Tower(一句带过 Layer)、对照 actix-web/rocket/go net/http、点出 axum 是 Tokio 官方团队出品的 Web 框架。**【样本章节,全书风格标杆】**

### 第 3 篇 · 提取与响应:Handler trait 的编译期魔法(5 章,提取招牌)

> 这是二分法的"提取与响应"一面,本书精华中的精华。Handler trait 怎么用宏把任意 async fn 变 Service,是 axum 最值得讲也最容易看不懂的地方。

- **第 9 章 · Handler trait:把 async fn 变 Service**(`P3-09`,★★双招牌章)——为什么随便一个 `async fn(State, Path<i32>) -> String` 就能当 handler?技巧:★**Handler<T, S> trait 的 T 参数**(coherence 占位 tuple,绕开孤儿规则)、★**`impl_handler!` + `all_the_tuples!` 宏展开**(对 0~16 个参数全部 impl Handler,最后一个参数 FromRequest、其余 FromRequestParts)、★**call(self, req, state) 内部把 req 拆 parts+body 逐个跑提取器链**、`HandlerService`、`Layered`(Handler::layer 把 Tower Layer 套到单个 handler)。源码:`axum/src/handler/mod.rs`(核心)、`handler/service.rs`、`handler/future.rs`、`macros.rs`(all_the_tuples)。归属:`提取(招牌)`。
- **第 10 章 · FromRequest 与 FromRequestParts:提取器的二元划分**(`P3-10`,★招牌章)——为什么 Path/Query/State 只读 parts,Json/Form 要消费 body?技巧:★**FromRequestParts(只读 &mut Parts,可多次跑)vs FromRequest(消费 body,只能跑一次)**、★**ViaRequest/ViaParts marker + `impl FromRequest<S, ViaParts> for T where T: FromRequestParts`** 这个桥接、`from_request` 用 `impl Future`(非 async-trait,0.x 起的 RPITIT 改造)、`Result<T, Rejection>`/`Option<T>` 的提取器自动桥接。源码:`axum-core/src/extract/mod.rs`(核心)、`extract/request_parts.rs`、`extract/option.rs`、`extract/tuple.rs`。归属:`提取(招牌)`。
- **第 11 章 · 提取器实战:Path/Query/State/Json/Form**(`P3-11`)——那些内置提取器具体怎么实现?技巧:**Path(serde 反序列化 URL 参数)**、**Query(query string 反序列化)**、**State(从 state 拿)**、★**Json(消费 body + serde_json + Content-Type 校验 + 限制大小)**、Form、`rejection` 怎么变 Response。源码:`axum/src/extract/path/`、`extract/query.rs`、`extract/state.rs`、`axum/src/json.rs`、`axum/src/form.rs`、`axum-core/src/extract/rejection.rs`。归属:`提取`。
- **第 12 章 · IntoResponse:返回值怎么变成 Response**(`P3-12`)——为什么 `String`/`&str`/`StatusCode`/`(StatusCode, Json<T>)`/`Result<T, E>` 都能当返回值?技巧:★**IntoResponse trait + into_response**、★**tuple 组合响应**(`(HeaderMap, impl IntoResponse)`、`(StatusCode, T)`、`(Parts, T)` 的链式拼装)、`IntoResponseParts`(先写 parts 再写 body)、`append_headers`、具体实现(String 设 text/plain、Json 设 application/json + serde 序列化)、`Response` 的 body 是 `axum_core::body::Body`。源码:`axum-core/src/response/into_response.rs`、`into_response_parts.rs`、`append_headers.rs`、`axum/src/json.rs`(IntoResponse for Json)、`axum/src/response/redirect.rs`、`response/sse.rs`。归属:`响应`。
- **第 13 章 · 自定义提取器与 #[axum::debug_handler]**(`P3-13`)——怎么写自己的 FromRequestParts?handler 报类型错怎么办?技巧:**自定义 FromRequestParts 的步骤**、★**`#[axum::debug_handler]` 宏**(`axum-macros/src/debug_handler.rs`,把 handler 包成命名函数改类型错信息)、`FromRef`(从 state 派生子状态)、为什么自定义提取器要 `Send`。源码:`axum-macros/src/debug_handler.rs`、`axum-core/src/extract/from_ref.rs`。归属:`提取`。

### 第 4 篇 · 中间件:from_fn 与 Tower Layer(3 章)

> axum 中间件就是 Tower Layer,但 axum 提供了 from_fn/from_extractor 这些便利层。

- **第 14 章 · from_fn:把闭包变中间件**(`P4-14`,★招牌章)——鉴权/日志/压缩这些横切怎么不侵入业务?技巧:★**`middleware::from_fn`**(`middleware/from_fn.rs`,把 `async fn` 闭包变 Layer)、★**`from_fn_with_state`/`map_request`/`map_response`**、from_fn 内部怎么用 `Service::call` 套娃、对照 actix-web 中间件/go middleware。源码:`axum/src/middleware/from_fn.rs`、`map_request.rs`、`map_response.rs`、`middleware/mod.rs`。归属:`中间件(招牌)`。
- **第 15 章 · from_extractor:把提取器当中件**(`P4-15`)——用提取器做请求校验当中间件?技巧:**`from_extractor`**(`middleware/from_extractor.rs`,把 FromRequest 包成 Layer 提前校验)、与 from_fn 的区别、拒绝时怎么变 Response。源码:`axum/src/middleware/from_extractor.rs`、`response_axum_body.rs`。归属:`中间件`。
- **第 16 章 · 中间件链与 ServiceBuilder**(`P4-16`)——多个中间件怎么叠?顺序重要吗?技巧:**Tower ServiceBuilder 叠中间件**、`Router::layer`(全局含 fallback)vs `route_layer`(只匹配的路由不含 fallback)vs `MethodRouter::layer` vs `Handler::layer` 四种作用域、中间件顺序(先加的在外层)、错误处理(`HandleErrorLayer`)。源码:`axum/src/routing/mod.rs`(layer/route_layer)、`error_handling/`。归属:`中间件`。

### 第 5 篇 · 服务、错误处理与高级(3 章)

- **第 17 章 · serve 与监听器:从 Router 到上线**(`P5-17`)——`axum::serve` 内部怎么把 Router 跑起来?技巧:**`serve::serve` 函数 + Listener trait**(`serve/mod.rs`、`serve/listener.rs`,抽象监听器支持 TcpListener/UnixListener)、hyper-util 的 `auto::Builder` 自动协商 HTTP/1+2、graceful shutdown(`Serve::with_graceful_shutdown`)、`Connected` trait 提取对端信息。源码:`axum/src/serve/mod.rs`、`serve/listener.rs`、`extract/connect_info.rs`。归属:`总览`。
- **第 18 章 · 错误处理:Infallible 与 HandleError**(`P5-18`)——为什么 Router 的 Service Error 是 Infallible?中间件出错了怎么办?技巧:★**Router/Route 的 Error 永远是 Infallible**(axum 在框架层把所有错误转成 Response)、★**HandleErrorLayer/HandleError**(中间件若返回非 Infallible 错误要用 HandleError 兜底转 Response)、`Response` 的错误怎么自定义、panic 怎么处理(tower-http catch_panic)。源码:`axum/src/error_handling/`、`routing/mod.rs`(impl Service Error=Infallible)。归属:`总览`。
- **第 19 章 · WebSocket、SSE 与流式响应**(`P5-19`)——长连接/服务端推送怎么在 axum 里写?技巧:**WebSocket 提取器**(`extract/ws.rs`,hyper 协议升级承《hyper》P2-07)、**Sse 响应器**(`response/sse.rs`,Stream of Event)、流式 body(`Body::from_stream`,承《hyper》Body as Stream)、对照 tonic 的双向流。源码:`axum/src/extract/ws.rs`、`axum/src/response/sse.rs`、`axum-core/src/body/`。归属:`响应`。

### 第 6 篇 · 演进、对照与收束(2 章)

- **第 20 章 · axum 0.7→0.8 演进 + 0.9 展望**(`P6-20`)——为什么 0.8 把 `Router::route` 限定只接 MethodRouter?路径参数语法为什么从 `:foo` 改成 `{foo}`?技巧:★**0.8 关键变动**(`Router::route` 只接 MethodRouter + 新增 `route_service`、路径参数 `{foo}`/`{*foo}` 取代 `:foo`/`*foo` 且 `without_v07_checks` 兼容旧路径、nest 在 `/` 不再支持、`MethodRouter::fallback`、axum-core 拆分)、★**为什么这么改**(API 清晰化:route=handler 语义、route_service=service 语义)、0.9 main 分支展望(0.8.x 分支才是 crates.io 版本)。源码:`axum/CHANGELOG.md`、`routing/mod.rs`(`without_v07_checks`)、`routing/path_router.rs`(`validate_v07_paths`)。归属:`总览`。
- **第 21 章 · 全书收束:axum 在 Rust 异步栈的位置**(`P7-21`)——把 axum 在 Rust 异步栈的位置钉死(Tokio→hyper→Tower→axum/tonic/Pingora);★对照 actix-web(actor 模型 + 自实现运行时 vs axum 全 Tokio)、rocket(过程宏 request guard vs axum trait 提取器)、go net/http(ServeMux vs axum matchit);★对照 tonic(gRPC 框架,Service 模型同源但协议/序列化不同);axum 生态(tower-http/axum-extra/axum-login)展望。归属:`总览`。
