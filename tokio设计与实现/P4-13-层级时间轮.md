# 第 13 章 · 层级时间轮(hierarchical timing wheel)

> **核心问题**:tokio 要管**成千上万个** `sleep` / 超时,凭什么管得动?最直觉的答案是"用一个最小堆(`BinaryHeap`),谁先到期谁在堆顶"——可 tokio 偏偏不用堆,而用一套**层级时间轮**:6 层、每层 64 个槽,用一个 64 位整数的**不同位段**直接定位"这个定时器该放进第几层第几号槽"。这玩意儿凭什么能把海量定时器的"插入"和"到期检查"都压到 O(1)?
>
> 这一章我们死盯一件事:**时间轮的数据结构本身**。sleep 怎么挂进轮、轮怎么转、定时器怎么到期,留到第 14 章;本章只把"轮"这个结构讲透。
>
> **读完本章你会明白**:
> - 为什么不用最小堆、不用"每毫秒扫一遍所有定时器"——这两条朴素路各自撞在哪堵墙上,而层级轮是怎么同时躲掉两面墙的。
> - "层级"为什么是 6 层 64 槽这么个怪数字——它跟"用整数位段定位槽"是同一件事的两面:64 = 2⁶,所以一层正好吃掉 6 个比特。
> - tokio 怎么用**位运算**(`leading_zeros` / `trailing_zeros` / 移位 / 掩码)把"第几层、第几号槽"算出来,既快又对——这是本章技巧精解的主角。
> - "槽"里的定时器为什么是**侵入式双向链表**(intrusive linked list),以及一个 `u64` 的 `occupied` 位图怎么让"找下一个该处理的槽"也是 O(1)。
>
> 难度提示:这章的位运算(`level_for` 里那行 `elapsed ^ when | SLOT_MASK`)第一眼看像天书,不慌。

> **如果一读觉得太难**:先只记住三件事——① 定时器按"到期时刻"分进不同层级的槽,近的进低层(毫秒槽)、远的进高层(秒/分/时/天槽);② 插入和到期检查都是 O(1),不随定时器总数增长;③ 到期处理时,高层槽里的定时器会"降级"重新散到低层,直到最终落到毫秒层被叫醒。位运算的细节可以回头再啃。

## 章首·一句话点破

> **时间轮不是一个"轮",而是一摞"格子货架":货物(定时器)按"它该在多久后被处理"的远近,被分到不同层的不同格子里;处理时只看"当前时间对应的格子",不扫全部货物。而"分到哪层哪格"这件事,用到期时刻的整数二进制位直接切出来——所以又快又省。**

这是**结论**。这一章倒过来拆:先看两条朴素路(最小堆、全量扫描)各自死在哪,再一层一层把层级轮搭起来,最后用源码里那几行位运算把"为什么是 O(1)"钉死。

第 1 篇讲过 `Future::poll` 的两态契约——返回 `Pending` 时必须留好 Waker。timer 的活,就是"在到点那一刻,把 Waker 叫醒"。可"叫醒"的前提是"我得先知道谁到点了、它在哪"——这正是时间轮要解决的问题:**在成千上万个挂起的定时器里,O(1) 地找出下一个该响的**。

---

## 一、先看反面:管理海量定时器的两条朴素路

要理解 tokio 为什么非造个层级轮,得先看清另外两条最直觉的路各自死在哪。

### 反面一:最小堆(BinaryHeap)

最经典的定时器数据结构是**最小堆**(按到期时间排的小顶堆)。堆顶永远是最近到期的那个;到点后 pop 堆顶,继续等下一个。

```rust
// 简化示意,非源码原文:最小堆管定时器
struct TimerHeap(BinaryHeap<Reverse<(Instant, Waker)>>);

// 插入:把定时器塞进堆
heap.push(Reverse((deadline, waker)));

// 到期检查:看堆顶
let next = heap.peek();   // O(1) 看一眼
// 到点了:pop 堆顶并唤醒
while heap.peek().map(|t| t.0 <= now).unwrap_or(false) {
    let Reverse((_, waker)) = heap.pop();   // O(log n) 弹出
    waker.wake();
}
```

这条路**没错**,很多系统(Go 的 runtime timer 早期、Java 的 `DelayQueue`)都用过。它的问题在于:**每个操作的代价是 O(log n)**,而 n 是定时器总数。

> **不这样会怎样**:设想一台服务器,每秒新建立 10 万条连接,每条连接都有一个读超时(`timeout`)、一个心跳定时器(`interval`)——轻松百万级定时器在轮里转。用最小堆:
> - 每插入一个定时器:O(log 1,000,000) ≈ 20 次比较 + 堆化(可能涉及 20 次内存访问,cache 不友好)。
> - 每次到期处理:pop 是 O(log n),pop 完还得再 log n 调整。
>
> 一秒 10 万次插入 × 每次 20 步 = **每秒 200 万次堆操作**,且每次都在"树"上跳着访问内存,cache 命中率惨。更要命的是:**堆是全局一个结构,所有线程争一把锁**(或得拆分多个堆)。在百万定时器 + 多核场景下,堆的 log n 这一项,会成为吞吐天花板。

> **钉死这件事**:最小堆的命门是"**所有操作都和 n 有关**"。n 越大,每次插入/到期越贵。而 tokio 想要的是:不管轮里有 10 个还是 100 万个定时器,**插入和到期检查都一样快**(O(1))。这,就是时间轮存在的全部理由。

### 反面二:全量扫描(every ms, scan all)

另一个朴素想法更暴力:既然定时器都带个到期时刻,那就**每隔 1 毫秒扫一遍所有定时器**,看谁到点了。

```rust
// 简化示意,非源码原文:灾难级全量扫描
loop {
    sleep(1ms);
    let now = Instant::now();
    for timer in &all_timers {        // 每毫秒扫一遍全部!
        if timer.deadline <= now {
            timer.waker.wake();
        }
    }
}
```

> **不这样会怎样**:一百万个定时器,每毫秒扫一遍 = **每秒扫 10¹² 个定时器**。CPU 全烧在"扫"上,一个正经任务都跑不了。这是把第 1 章讲的"忙等轮询"在时间层重演了一遍——而且比忙等还狠,因为它每毫秒触发一次。这条路在工程上是**直接报废**的。

### 两条路,夹出一个需求

| 反面 | 命门 | 想要的 |
|------|------|--------|
| 最小堆 | 插入/到期 O(log n),n 大了就贵 | **操作代价和定时器总数无关**(O(1)) |
| 全量扫描 | 每毫秒扫全部,CPU 烧穿 | **只看"当前时间对应的那一格"**,不扫全部 |

把这两条叠起来,就是层级时间轮要做的事:**用"时间"本身当索引,把定时器按到期时刻分散到固定数量的格子里,处理时只动"当前时间那一格"**。格子的数量是固定的(不随 n 增长),所以无论 n 多大,单次操作都是 O(1)。

> **比喻回到餐厅**:催单闹钟不是把所有订单的催单时间排成一棵树(最小堆),更不是每秒钟挨个问每桌"该催了吗"(全量扫描)。餐厅的做法是:**一面墙上钉着一摞格子,按"几点几分催"分格**——"12:00 该催的"放一格、"12:30 该催的"放一格。服务员瞄一眼当前时间对应那格,有单就拿出来催,没单就走。**格子数量是固定的(就那么几格),订单再多也只是每格变厚,不影响"找格子"的速度。** 这,就是时间轮。

## 二、单层时间轮:把"时间"当数组下标

先把最简单的**单层**时间轮搭起来,看清它的好处和天花板。

### 想法:一个环形数组,下标 = 时间

假设我们的时间粒度是 1 毫秒。开一个**环形数组**(长度 N),数组的每个元素是一个链表(放定时器)。每个定时器按"到期时刻 mod N"放进对应下标的格子(叫 **slot**,槽)。

```
单层时间轮(示意,N=8):
下标:    [0]    [1]    [2]    [3]    [4]    [5]    [6]    [7]
        定时器  定时器   ()    定时器   ()     ()    定时器  ()
         A,B    C              D                    E
当前指针 → [2]
```

- **插入**一个到期时刻为 `t` 的定时器:算 `t mod 8`,塞进 `slots[t mod 8]` 这个链表。**O(1)。**
- **到期检查**:每过 1 毫秒,指针前进一格,处理指针所指那个 slot 里的所有定时器。**O(当前 slot 里的定时器数),和总定时器数 n 无关。**

这就是时间轮的核心红利:**插入是 O(1) 的算下标 + 链表头插;到期检查只动一格,不扫全部**。n 无论多大,都不影响"找格子"的速度。

> **钉死这件事(时间轮的根本红利)**:把"时间"本身当成数组的下标——到期时刻越接近的定时器,落在越靠近当前指针的格子里。处理时只跟着指针一格一格走,**永远只看"现在这一格"**,这就是它能把单次操作压到 O(1) 的全部秘密。

### 单层轮的天花板:跨度有限

可单层轮有个致命问题:**它的最大跨度只有 N 毫秒**。一个 1 秒后到期的定时器,放进长度 8 的轮里,会和"125 毫秒后到期"的撞到同一个 slot(因为 `1000 mod 8 == 0`)——它俩根本不是同一时刻到期的,却挤在一格,处理时就乱了。

要把跨度做大,最笨的办法是把 N 调大:跨度 2 年(≈ 6×10¹⁰ 毫秒),就开一个 6×10¹⁰ 长度的数组。**内存直接爆**——这个数组要占几百 GB。

> **不这样会怎样**:单层轮要么"跨度小"(只能管几秒内的定时器),要么"为了大跨度把数组开得巨大"——内存爆炸。两个都不可接受。

### 出路:别用一层,用"多层"

真实世界早就给出了答案——**时钟**。时钟不是一层,是三层:秒针(60 格)、分针(60 格)、时针(12 格)。秒针转一圈,分针走一格;分针转一圈,时针走一格。三层加起来,12 小时的跨度,每层只 60 格左右。

时间轮借用这个思路,做成**层级时间轮(hierarchical timing wheel)**:好几层轮叠起来,**低层管细粒度(毫秒),高层管粗粒度(秒、分、时、天)**;高层每"转完一圈",低层的指针就推进一格——更准确地说,**高层的定时器,在它的那一格被处理时,会"降级(cascade)"重新散到低层的格子里**。

这就是 tokio 时间轮的形状。下面我们看它具体长什么样。

## 三、tokio 的层级轮:6 层 × 64 槽,2 年跨度

先看 tokio 时间轮在源码里长什么样。真实路径不在 `tokio/src/time/wheel/`(那是个误导),而在 **`runtime/time/wheel/`**——时间轮是 runtime 的一部分,和 reactor 并列。打开 `runtime/time/wheel/mod.rs`:

```rust
/// Timing wheel implementation.
#[derive(Debug)]
pub(crate) struct Wheel {
    /// The number of milliseconds elapsed since the wheel started.
    elapsed: u64,

    /// Timer wheel.
    ///
    /// Levels:
    /// * 1 ms slots / 64 ms range
    /// * 64 ms slots / ~ 4 sec range
    /// * ~ 4 sec slots / ~ 4 min range
    /// * ~ 4 min slots / ~ 4 hr range
    /// * ~ 4 hr slots / ~ 12 day range
    /// * ~ 12 day slots / ~ 2 yr range
    levels: Box<[Level; NUM_LEVELS]>,

    /// Entries queued for firing
    pending: EntryList,
}

/// Number of levels. Each level has 64 slots.
const NUM_LEVELS: usize = 6;

/// The maximum duration of a `Sleep`.
pub(super) const MAX_DURATION: u64 = (1 << (6 * NUM_LEVELS)) - 1;
```

([tokio/src/runtime/time/wheel/mod.rs:22-48](../tokio/tokio/src/runtime/time/wheel/mod.rs#L22-L48))

读这段,六层结构一目了然:

| 层 | 每槽跨度 | 本层总跨度 |
|----|----------|-----------|
| Level 0 | 1 ms | 64 ms |
| Level 1 | 64 ms | ≈ 4 s |
| Level 2 | ≈ 4 s | ≈ 4 min |
| Level 3 | ≈ 4 min | ≈ 4 h |
| Level 4 | ≈ 4 h | ≈ 12 day |
| Level 5 | ≈ 12 day | ≈ 2 yr |

几个关键数字钉死:

- **6 层,每层 64 个 slot**。`NUM_LEVELS = 6`。
- **`MAX_DURATION = (1 << 36) - 1`** —— 2³⁶ 毫秒 ≈ 2 年。tokio 的 `sleep` 最长能管 2 年(`MAX_SAFE_MILLIS_DURATION` 还要更小一点,留出几个哨兵值,见 entry.rs 的 `STATE_DEREGISTERED` 等)。
- **每层 64 = 2⁶**。这不是巧合,是**故意的**——为了让一层正好吃掉 6 个比特,详见下一节的位运算。

> **钉死这件事(为什么是 64)**:每层 64 槽,是因为 64 = 2⁶。一个 64 位的到期时刻(`u64`,单位毫秒),正好可以被切成 6 段、每段 6 位——每段直接当一层轮的 slot 下标用。**"6 层 × 64 槽"和"用整数位段定位槽"是同一件事的两面。** 如果选 100 槽,位运算就失效了,只能用除法/取模,慢且丑。

### `Level`:每层一个 `occupied` 位图 + 64 个链表

每一层长什么样?看 `runtime/time/wheel/level.rs`:

```rust
/// Wheel for a single level in the timer. This wheel contains 64 slots.
pub(crate) struct Level {
    level: usize,

    /// Bit field tracking which slots currently contain entries.
    ///
    /// Using a bit field ... allows avoiding a scan to find entries.
    /// The least-significant bit represents slot zero.
    occupied: u64,

    /// Slots.
    slot: [EntryList; LEVEL_MULT],
}

/// Level multiplier. Being a power of 2 is very important.
const LEVEL_MULT: usize = 64;
```

([tokio/src/runtime/time/wheel/level.rs:6-38](../tokio/tokio/src/runtime/time/wheel/level.rs#L6-L38))

两层信息:

1. **`slot: [EntryList; 64]`** —— 64 个槽,每个槽是一条链表(放落在这一格的定时器)。`EntryList` 是侵入式双向链表(下一节细讲)。
2. **`occupied: u64`** —— 一个 64 位整数当**位图**:第 i 位是 1,表示第 i 个 slot 里有定时器;是 0 表示空槽。

这个 `occupied` 位图,是另一个 O(1) 技巧的根基(技巧精解里拆)。先记住它的作用:**"找下一个非空槽"不用扫 64 个链表,看位图就行**。

### 整张轮的形状(ASCII 布局图)

把六层叠起来,tokio 时间轮长这样:

```
                  tokio 层级时间轮(6 层 × 64 槽)
 ┌─────────────────────────────────────────────────────────────┐
 │ Level 5: 12 day/slot   总跨度 ≈ 2 年     (管 12 天 ~ 2 年) │
 │ [0][1][2] ... [63]   每槽 ≈ 12 天                           │
 ├─────────────────────────────────────────────────────────────┤
 │ Level 4: 4 hr/slot     总跨度 ≈ 12 天      (管 4 小时 ~ 12 天)│
 │ [0][1][2] ... [63]                                          │
 ├─────────────────────────────────────────────────────────────┤
 │ Level 3: 4 min/slot    总跨度 ≈ 4 小时                       │
 │ [0][1][2] ... [63]                                          │
 ├─────────────────────────────────────────────────────────────┤
 │ Level 2: 4 s/slot      总跨度 ≈ 4 分                         │
 │ [0][1][2] ... [63]                                          │
 ├─────────────────────────────────────────────────────────────┤
 │ Level 1: 64 ms/slot    总跨度 ≈ 4 秒                         │
 │ [0][1][2] ... [63]                                          │
 ├─────────────────────────────────────────────────────────────┤
 │ Level 0: 1 ms/slot     总跨度 64 ms  (到期就在这层被叫醒)   │
 │ [0][1][2] ... [63]   ← 当前指针 elapsed 指着这里             │
 └─────────────────────────────────────────────────────────────┘
   每层 64 槽;高层每槽跨度 = 低层总跨度(64 倍递进)
```

到期时刻(毫秒,`u64`)的二进制,被切成 6 段、每段 6 位,分别对应一层:

```
到期时刻(64-bit,单位 ms)的二进制位段 → 层级映射:

 bit:  63                    36 35                 30 29    24 23 18 17 12 11  6 5   0
      ┌─────────────────────┬────────────────────┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┐
      │  (留作哨兵/未来扩展) │   Level 5 槽号      │L4│L3│L2│L1│  Level 0 槽号     │
      └─────────────────────┴────────────────────┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┘
                                                    每 6 bit 一层,共 6 层 → 2^36 ms ≈ 2 年
```

> **钉死这件事(层级递进的数学)**:每层 64 槽,每层总跨度是下一层的 64 倍(`level_range(level) = 64 × slot_range(level)`,见 level.rs)。所以 6 层叠加:64 ms × 64⁵ ≈ 2³⁶ ms ≈ 2.18 年。**这就是 `MAX_DURATION = 2³⁶ - 1` 的来历**——它正好是 6 层 × 6 位能表示的最大毫秒数。

## 四、插入与到期:轮怎么转

数据结构清楚了。现在看两个核心操作:**插入一个定时器**、**处理到期的定时器**。

### 插入:先定层,再定槽,链表头插

`Wheel::insert`:

```rust
pub(crate) unsafe fn insert(
    &mut self,
    item: TimerHandle,
) -> Result<u64, (TimerHandle, InsertError)> {
    let when = unsafe { item.sync_when() };

    if when <= self.elapsed {
        return Err((item, InsertError::Elapsed));   // 已过期,立刻 fire
    }

    // Get the level at which the entry should be stored
    let level = self.level_for(when);

    unsafe {
        self.levels[level].add_entry(item);          // 头插进对应槽
    }
    // ...
    Ok(when)
}
```

([tokio/src/runtime/time/wheel/mod.rs:88-113](../tokio/tokio/src/runtime/time/wheel/mod.rs#L88-L113))

两步:**① 算该放哪层(`level_for`);② 在那一层算该放哪个槽(`slot_for`),链表头插**。两步都是 O(1) 的位运算,下面"技巧精解"会拆透。这里先看 `Level::add_entry`:

```rust
pub(crate) unsafe fn add_entry(&mut self, item: TimerHandle) {
    let slot = slot_for(unsafe { item.registered_when() }, self.level);

    self.slot[slot].push_front(item);     // 链表头插,O(1)

    self.occupied |= occupied_bit(slot);  // 位图标记该槽非空
}
```

([tokio/src/runtime/time/wheel/level.rs:122-128](../tokio/tokio/src/runtime/time/wheel/level.rs#L122-L128))

注意两个细节:

1. **`push_front`**:定时器是**侵入式链表节点**,插入就是把节点接到链表头,不分配内存(O(1))。这点和最小堆(节点要存进堆数组、可能要搬动)截然不同。
2. **`self.occupied |= 1 << slot`**:顺手把位图对应位置 1。这一笔极其便宜,但让"找下一个非空槽"也变成位运算(见技巧精解)。

### 到期处理:跟着 `elapsed` 走,高层定时器"降级"

时间轮最反直觉、也最精妙的地方,在 `Wheel::poll`(被 `process_expiration` 调用)。它不是"扫一层",而是这样的循环:

```rust
pub(crate) fn poll(&mut self, now: u64) -> Option<TimerHandle> {
    loop {
        if let Some(handle) = self.pending.pop_back() {
            return Some(handle);   // 先把"已经判定要 fire"的逐个吐出
        }

        match self.next_expiration() {
            Some(ref expiration) if expiration.deadline <= now => {
                self.process_expiration(expiration);   // 处理这一格
                self.set_elapsed(expiration.deadline); // 推进 elapsed
            }
            _ => {
                self.set_elapsed(now);
                break;
            }
        }
    }
    self.pending.pop_back()
}
```

([tokio/src/runtime/time/wheel/mod.rs:141-165](../tokio/tokio/src/runtime/time/wheel/mod.rs#L141-L165))

关键在 `process_expiration`:它取出某个槽里的所有定时器,然后**根据这个槽在第几层,决定每个定时器的命运**:

```rust
pub(crate) fn process_expiration(&mut self, expiration: &Expiration) {
    let mut entries = self.take_entries(expiration);   // 把这一槽整个摘下来

    while let Some(item) = entries.pop_back() {
        // Try to expire the entry ... updates registered_when.
        match unsafe { item.mark_pending(expiration.deadline) } {
            Ok(()) => {
                // Item was expired —— 真的到点了,丢进 pending 队列准备 fire
                self.pending.push_front(item);
            }
            Err(expiration_tick) => {
                // 还没到点 —— 它是被高层槽"带下来"的,重新散到正确的低层槽
                let level = level_for(expiration.deadline, expiration_tick);
                unsafe {
                    self.levels[level].add_entry(item);
                }
            }
        }
    }
}
```

([tokio/src/runtime/time/wheel/mod.rs:217-250](../tokio/tokio/src/runtime/time/wheel/mod.rs#L217-L250))

读这段,要看清"降级(cascade)"是怎么发生的:

- 一个 1 小时后到期的定时器,最初被插进 **Level 3**(4 分/槽)的某个槽。
- 时间过了快 1 小时,Level 3 那个槽轮到了被处理。`process_expiration` 把它摘下来,调 `mark_pending(deadline)`。
- `mark_pending` 发现"它的真实到期时刻还在 `deadline` 之后"(还差几分钟),返回 `Err(真实 tick)`——意思是"我还没真到点"。
- 于是 `process_expiration` 用 `level_for` 重新算它**现在**该去哪层——因为时间已经推进,它现在离到期只剩几分钟了,该去 **Level 2**(4 秒/槽)。重新插进 Level 2 的某个槽。
- 又过几分钟,Level 2 那个槽轮到,再降级到 Level 1(64 ms/槽),再到 Level 0(1 ms/槽)。
- 最后在 Level 0,`mark_pending` 终于返回 `Ok(())`——真的到点了,丢进 `pending` 队列,等待被 fire、唤醒 Waker。

> **比喻回到餐厅**:一面墙按"几点催"分格。12:00 该催的单,放在"12 点"那个大格子里;可你不会等到 12:00 才去翻那个大格子(里面可能挤了上千张单)。服务员的做法是:11:55 时,把"12 点"大格里的单,**按"具体几分"重新散到 12:00 / 12:01 / 12:02 这些小格子里**;到 12:00 真正那分钟,只动"12:00"那个小格子就行。**高层的粗粒度格子 → 时间临近时降级散到低层细粒度格子**,这就是 cascade。它让"瞬时处理量"永远是小的(只动一个细粒度小格),而定时器总数再大也不影响。

> **钉死这件事(cascade 的意义)**:高层槽的定时器,**不会在高层就 fire**,而是"时间临近"时被取下来、重新散到更精细的低层。这个降级过程把"跨度的压力"逐层摊薄——任何时刻,真正会被 fire 的,只有 Level 0 当前那一毫秒格里的定时器。所以即使 100 万个定时器分散在 2 年跨度里,**到点瞬间的处理量也只是"这一毫秒到期的那些"**,极少。

### 到期处理的代价分析

回到开头的"反面"对照,看层级轮怎么躲掉两面墙:

| 操作 | 最小堆 | 全量扫描 | 层级轮 |
|------|--------|----------|--------|
| 插入 | O(log n) | O(1)(但要扫) | **O(1)** |
| 到期检查(无定时器到期时) | O(1) 看堆顶 | O(n) 扫全部 | **O(1)** 看位图 |
| 到期处理(一批到点) | O(k log n),k 是到点数 | O(n) | **O(k + 降级数)**,与 n 无关 |

最右列每一项都**和定时器总数 n 无关**。这就是层级轮的红利:无论轮里 10 个还是 100 万个定时器,单次插入和单次到期检查都是 O(1)。

## 五、侵入式链表 + 状态机:定时器节点长什么样

最后补一个结构性细节:槽里的定时器,到底是什么?

答案是**侵入式双向链表节点**(`TimerShared`),它本身是 `Sleep` Future 内部状态的一部分,**不单独分配内存**。看 `runtime/time/entry.rs`:

```rust
pub(crate) struct TimerShared {
    /// A link within the doubly-linked list of timers on a particular level and slot.
    /// Valid only if state is equal to Registered.
    pointers: linked_list::Pointers<TimerShared>,   // 链表前后指针,嵌在节点里

    /// The time when the TimerEntry was registered into the Wheel.
    registered_when: AtomicU64,

    /// Current state. (not complete / fired / error ...)
    state: StateCell,

    _p: PhantomPinned,   // 这个节点不可移动(被链表指针指着)
}
```

([tokio/src/runtime/time/entry.rs:331-355](../tokio/tokio/src/runtime/time/entry.rs#L331-L355))

几个关键点:

1. **侵入式(intrusive)**:链表的 `prev` / `next` 指针**直接嵌在 `TimerShared` 这个结构体里**,而不是链表自己分配一个"节点"结构体把 `TimerShared` 包起来。这意味着定时器节点**不额外占内存**——它的存储成本就是 `Sleep` Future 本身那一块。
2. **`PhantomPinned`**:这个节点被链表指针指着,**不能被移动**(移动了,指它的指针就野了)。所以 `TimerEntry` 必须 `Pin` 住才能用。这呼应第 3 章讲的 `Pin`:**凡是会被别人用指针长期指着的结构,都得 Pin**。tokio 在 `Sleep` 上用 `pin_project!` 把这一层 pin 下去。
3. **`state: StateCell`**:一个 `AtomicU64`,既存"到期时刻",又用几个哨兵值(`STATE_DEREGISTERED = u64::MAX`、`STATE_PENDING_FIRE = u64::MAX - 1`)表示"已注销 / 待 fire"。**一个原子字同时编码"时间"和"状态"**,这是第 5 章 task 状态位打包的同款技巧在 timer 上的重演——不展开,记住"timer 也用位/哨兵值打包状态"。

> **钉死这件事(为什么侵入式 + Pin)**:最小堆的节点是"堆数组里的一个元素",插入要拷贝/搬动(堆化)。时间轮的节点是"侵入式链表节点",插入只需改几个指针、零拷贝。代价是:节点必须**钉在原地不能动**(否则链表指针野),所以 `TimerEntry` 必须 `Pin`。这是 O(1) 插入的代价,被 Rust 的 `Pin` 类型系统稳稳兜住。

## 技巧精解:位运算定位槽——为什么插入和到期检查都是 O(1)

这一节是本章硬核。我们拆两个最关键的位运算:**① `level_for`——给定到期时刻,算它该放第几层;② `slot_for`——给定时刻和层,算它该放这一层的第几号槽**;外加 **③ `next_occupied_slot`——用位图找下一个非空槽**。三个加起来,就是时间轮 O(1) 的全部根基。

### 拆解一:`slot_for`——6 位切一层

最简单的是 `slot_for`,先看它热身:

```rust
/// Converts a duration (milliseconds) and a level to a slot position.
fn slot_for(duration: u64, level: usize) -> usize {
    ((duration >> (level * 6)) % LEVEL_MULT as u64) as usize
}
```

([tokio/src/runtime/time/wheel/level.rs:171-173](../tokio/tokio/src/runtime/time/wheel/level.rs#L171-L173))

(`LEVEL_MULT = 64`)

这一行干的事:**把 `duration` 右移 `level × 6` 位,再取低 6 位**。翻译成人话:

- **Level 0**(`level=0`):右移 0 位,取低 6 位 → `duration` 的最低 6 位(0~63)。这是"毫秒位",代表"这一毫秒格"。
- **Level 1**(`level=1`):右移 6 位,取低 6 位 → `duration` 的第 6~11 位。这是"64 毫秒位",代表"第几个 64 毫秒格"。
- **Level 2**(`level=2`):右移 12 位,取低 6 位 → 第 12~17 位。"4 秒位"。
- ……
- **Level 5**(`level=5`):右移 30 位,取低 6 位 → 第 30~35 位。"12 天位"。

**整个 64 位整数,被干净地切成 6 段、每段 6 位,每段对应一层。** 这就是"为什么是 64 槽"的答案——为了让"取一段 6 位"就是一个移位 + 一次掩码(`% 64` 在 2 的幂上等价于 `& 63`),一条指令搞定。

> **不这样会怎样(反面对比)**:如果一层是 100 槽(不是 2 的幂),`slot_for` 就得写 `(duration / 某个基数) % 100`——除法比位运算慢好几倍,且无法和"二进制位段"对齐,层与层之间的换算会变成一堆魔法数字。**选 2 的幂(尤其 2⁶),是让位运算能成立的根本前提。** 源码注释那句 `Being a power of 2 is very important`(level.rs:37)说的就是这个。

### 拆解二:`level_for`——用"最高有效位"定层

`level_for` 是最烧脑的一行,但它把"该放哪层"算得极其精巧。先看源码:

```rust
fn level_for(elapsed: u64, when: u64) -> usize {
    const SLOT_MASK: u64 = (1 << 6) - 1;   // 0b111111,低 6 位全 1

    // Mask in the trailing bits ignored by the level calculation in order to cap
    // the possible leading zeros
    let mut masked = elapsed ^ when | SLOT_MASK;

    if masked >= MAX_DURATION {
        // Fudge the timer into the top level
        masked = MAX_DURATION - 1;
    }

    let leading_zeros = masked.leading_zeros() as usize;
    let significant = 63 - leading_zeros;

    significant / NUM_LEVELS    // NUM_LEVELS = 6
}
```

([tokio/src/runtime/time/wheel/mod.rs:275-291](../tokio/tokio/src/runtime/time/wheel/mod.rs#L275-L291))

一行一行拆,看它到底在算什么。

**目标**:`elapsed` 是"当前已经过去的毫秒数",`when` 是"定时器的到期毫秒数"。我们要算出:`when - elapsed`(距到期还有多久)落在哪一层。

**直觉**:`when - elapsed` 越大,层级越高(因为高层管远期的)。具体说,**`when - elapsed` 的最高有效位(MSB)落在第几个 6 位段,就是层级**。比如 `when - elapsed` 的 MSB 在第 8 位(值在 256~511 之间),它就该去 Level 1(64 ms/槽,总跨度 4 秒)——因为 Level 0 只管 0~63 ms,装不下。

**源码的精巧之处**:它不直接算 `when - elapsed`(减法可能有溢出/边界问题),而是用 `elapsed ^ when`(异或)。异或的特性:**`elapsed` 和 `when` 在高位第一次出现不同的那一位,就是 `(when - elapsed)` 的 MSB 所在位置**(当 `when > elapsed` 时)。这一步用异或代替减法,既避开了减法的边界,又抓住了"层级取决于最高不同位"这个本质。

接着 `| SLOT_MASK`:把低 6 位全置 1。为什么?因为**低 6 位决定的是 Level 0 内部的 slot,不影响层级**;把它们抹成全 1,相当于"向上取整到 6 位边界",保证 `leading_zeros` 算出来的 MSB 落在正确的 6 位段上。

然后 `leading_zeros` + `63 - leading_zeros` = MSB 的位置(0~63)。最后 `/ NUM_LEVELS`(除以 6)= 层号(0~5)。**MSB 在第 0~5 位 → Level 0;第 6~11 位 → Level 1;……;第 30~35 位 → Level 5。**

举个具体例子:`elapsed = 0`,`when = 1000`(1 秒后到期)。

1. `elapsed ^ when = 0 ^ 1000 = 1000`。
2. `1000 | 63 = 1023`(二进制 `1111111111`,10 个 1)。
3. `leading_zeros(1023) = 54`(64 - 10)。
4. `significant = 63 - 54 = 9`(MSB 在第 9 位)。
5. `9 / 6 = 1` → **Level 1**(64 ms/槽,总跨度 4 秒)。✓ 1 秒确实落在 Level 1。

> **不这样会怎样(反面对比)**:朴素写法是 `match when - elapsed { 0..=63 => 0, 64..=4095 => 1, ... }`——一堆魔法数字的 if-else,每加一层得改一堆范围,还容易写错边界。tokio 用"异或 + leading_zeros + 整除 6"三步,**没有任何魔法数字,层级完全由二进制位决定**——加一层只是把 `NUM_LEVELS` 改大,代码不用动。这是位运算的优雅之处:**让数据结构自己长出它的形状,而不是硬编码**。

> **钉死这件事(`level_for` 的核心)**:它没有用减法,而是用 `elapsed ^ when` 找出"两个时刻在二进制上第一个不同的高位"——那个高位在哪一个 6 位段,定时器就该去哪一层。**层级 = 距到期时间的数量级**,而数量级正好对应二进制位数。这一行代码,是"用整数位段编码时间轮"这个设计的浓缩。

### 拆解三:`next_occupied_slot`——位图找下一个非空槽

最后一个技巧,让"到期检查"也 O(1)。前面说过,每层有个 `occupied: u64` 位图,标记哪些槽非空。到期检查时,要找"从当前时间开始,下一个该处理的非空槽"。看 `Level::next_occupied_slot`:

```rust
fn next_occupied_slot(&self, now: u64) -> Option<usize> {
    if self.occupied == 0 {
        return None;   // 整层都空
    }

    // Get the slot for now using Maths
    let now_slot = (now / slot_range(self.level)) as usize;
    let occupied = self.occupied.rotate_right(now_slot as u32);
    let zeros = occupied.trailing_zeros() as usize;
    let slot = (zeros + now_slot) % LEVEL_MULT;

    Some(slot)
}
```

([tokio/src/runtime/time/wheel/level.rs:108-120](../tokio/tokio/src/runtime/time/wheel/level.rs#L108-L120))

这一段用了一个极其漂亮的 CPU 内建指令组合:

1. **`now / slot_range(level)`**:算出"当前时间对应这一层的第几个槽"(`now_slot`)。这是"指针"当前指着的格子。
2. **`occupied.rotate_right(now_slot)`**:把位图**循环右移 `now_slot` 位**。移完之后,**当前槽对应的那一位,跑到了位图的最低位(LSB)**;之后的位,是"未来将要处理的槽"。
3. **`trailing_zeros`**:数移位后位图末尾有几个连续的 0。**末尾第一个 1 的位置,就是"从当前槽开始,下一个非空槽还要走几步"**。
4. **`(zeros + now_slot) % 64`**:换算回原始槽号。

这一段把"在环形结构里找下一个非空位"这件事,压成了一次除法 + 一次循环移位 + 一次 `trailing_zeros`——**全是单周期 CPU 指令**,没有循环、没有扫描。

> **不这样会怎样(反面对比)**:如果没有位图,找下一个非空槽要 `for slot in now_slot..64 { if !self.slot[slot].is_empty() { return slot } }`——最坏扫 64 个链表头。64 个链表头散在内存各处,cache 命中差。换成位图后,**一次 `trailing_zeros` 就跳到答案**,且位图本身只占 8 字节,永远在 cache 里。这是"用 8 字节 + 一条指令,换掉最多 64 次内存访问"的典型空间换时间——而且换得极其划算。

> **钉死这件事(位图的红利)**:`occupied` 位图让"找下一个该处理的槽"从 O(64) 降到 O(1)(单条 `trailing_zeros` 指令)。配合 `level_for` 的定层 O(1)、`slot_for` 的定槽 O(1),**时间轮的"插入 + 到期检查"全链路都是 O(1),且没有任何一处依赖定时器总数 n**。这就是层级轮能扛百万定时器的根本。

## 章末小结

### 用"餐厅服务员"比喻回顾本章

1. **催单闹钟不是排成一棵树(最小堆)**——树越高(定时器越多),每次"翻出最早那个"越贵。也不是每秒挨桌问(全量扫描)。**催单闹钟是一面墙的格子,按"几点几分催"分格**——这是层级时间轮。
2. **格子分好几层**:大格子按"几点"分,中格子按"几分"分,小格子按"几秒"分。**远的催单先放大格子,时间临近时再降级散到小格子**——这是 cascade。
3. **服务员只看当前时间对应那一格**,有单就拿出来催,没单就走——**永远只动一格,不扫全部**。所以订单再多(100 万张),单次"找该催的"都一样快。
4. **格子的数量是固定的**(6 层 × 64 格),不随订单数增长——所以内存开销也是固定的。

### 本章在全书主线中的位置

记住全书的二分法:**调度执行(让就绪的任务跑) vs 事件唤醒(让等待的任务不空耗、就绪了再叫)**。

这一章服务的是**事件唤醒**那一面——而且是"**时间事件**"这一面。第 1 章讲过:运行时三件套里,**timer 负责"到点叫醒"**。可"叫醒"的前提是"我得先知道谁到点了、它在哪"——本章解决的就是这个"在哪"的问题:**在成千上万个挂起的定时器里,O(1) 地定位下一个该响的**。

层级时间轮,是 timer 这件套的**数据结构地基**。下一章,我们会看 timer 的另一半——**这套轮子,谁在拨动它?一个 `sleep(5s)` 怎么挂进轮、挂起后到点怎么被唤醒?** 那是"事件唤醒"在时间层的完整闭环。

### 五个"为什么"清单

1. **为什么不用最小堆管海量定时器?**:最小堆的插入和到期都是 O(log n),n(定时器总数)越大越贵;百万定时器时,每秒几十万次堆操作会成为吞吐天花板。tokio 要的是 O(1),不管 n 多大。
2. **为什么是 6 层、每层 64 槽?**:64 = 2⁶,一层正好吃掉到期时刻(`u64`)的 6 个比特;6 层 × 6 位 = 36 位,2³⁶ ms ≈ 2 年(`MAX_DURATION`)。"6×64"和"用整数位段定位槽"是同一件事的两面。
3. **`level_for` 凭什么一行算出层级?**:它用 `elapsed ^ when`(异或)找出两个时刻在二进制上第一个不同的高位——那个高位落在第几个 6 位段,就是层级。用异或代替减法,避开边界问题,且层级完全由二进制位数决定,没有魔法数字。
4. **高层定时器怎么最终被 fire?**:高层槽的定时器在"时间临近"时被取下,用 `mark_pending` 判定还没真到点,就用 `level_for` 重新散到更精细的低层(cascade)。逐层降级,最终落到 Level 0 的某一毫秒格,才真正被 fire、唤醒 Waker。
5. **`occupied` 位图省了什么?**:让"找下一个非空槽"从"最多扫 64 个链表头"(O(64),cache 差)降到"一次 `trailing_zeros` 指令"(O(1),位图在 cache 里)。配合定层/定槽的位运算,全链路 O(1)。
