# 《Tokio 设计与实现深入浅出:从一根线程到百万并发》—— 全书总纲

> 一本写给"用过 Rust async、甚至读过 tokio 源码,却总觉得一知半解"的人的小书。
>
> 本文是全书的**总纲**:定主线、定方法、定比喻、定分篇分章、定源码策略、定写作约定。
> 与《Linux 内核》《数据库内核》《容器》同源,都是"**第一性原理 + 真实源码**";但本书多一条命脉:**实现技巧精解**——Rust 系统级代码的技巧密度远高于 C 项目,光讲"为什么这么设计"讲不透"实现"。

---

## 一、这本书的定位

### 目标读者

会写 async Rust、用过 `tokio::spawn` / `#[tokio::main]`,甚至翻过 tokio 源码,但**一知半解**:知道有 runtime、task、reactor、timer,却讲不清它们怎么协作、为什么这么设计、源码里那些 `unsafe` / 原子位打包 / `Pin` / 缓存行对齐到底在干什么。和"看过数据库源码但不理解"是同一种病。

### 全书一句话主旨

> **从"一根 OS 线程只会阻塞着干等",到"少量线程高效驱动成千上万个并发任务"——tokio 是怎么把"等待"从"占用线程"里解放出来的,每一步的设计动机和实现技巧都讲透。**

每个组件,都是这条旅程上的一个**驿站**。读完一本书,读者不是"多记了几个 API",而是能**在脑子里放映出 tokio 运转的全过程**:一个 task 怎么被 spawn、被 poll、在 await 点让出、被 reactor 唤醒、被调度器重新捡起——以及每一步底下用了什么巧妙的手段。

### 为什么用"从阻塞到百万并发"当骨架(而非按模块平铺)

tokio 的模块(`runtime` / `task` / `io` / `time` / `sync` / `net`)高度耦合:task 要被 scheduler 调度,scheduler 要靠 reactor 唤醒,reactor 的事件要回灌 scheduler,timer 又是另一种"到点唤醒"。按模块百科式平铺,读者会觉得每块都是孤岛、不知道**为什么要学这个**。用"一根线程如何演变成百万并发"这条旅程当骨架,每个模块都是"为了让旅程继续而必须出现的角色"——读者永远知道**我在哪、为什么需要下一站**。

## 二、写作方法:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与数学系列、乃至其他源码系列最关键的方法论,所有写作者必读、必守。**

tokio 源码里充斥高级技巧:无锁原子位打包、`Pin`、`UnsafeCell`、Chase-Lev 双端队列、`#[repr(align(64))]` 缓存行对齐、`thread_local` 快通道、层级时间轮、budget 让出……用户自己读过源码,"一知半解"的缺口往往**不在**"它要解决什么问题"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书的每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?(沿用系列一贯的"为什么优先")
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果不用这个技巧、朴素地写,会撞上什么墙",让妙处显形。

一句话:**讲不清技巧,等于没讲 tokio。**

## 三、全书主线、二分法、主比喻

### 主线

> **如何用极少的 OS 线程,高效驱动成千上万个并发任务?——核心是把"等待"从"占用线程"里解放出来。**

任何一处看不懂某个机制,回到这句问:"这是在让**就绪的任务跑起来**,还是在让**等待的任务不空耗线程**?"

### 二分法

本书的二分法:

> **调度执行(让就绪的任务跑) vs 事件唤醒(让等待的任务不空耗、就绪了再叫它)。**

- **调度这一面**:task 是什么、scheduler 怎么排队 / 偷工作 / 让出。
- **唤醒这一面**:reactor(I/O 事件)、timer(时间事件)、sync 原语的 notify,都是"等待者先挂起、条件满足时被叫醒"。

tokio 每个组件都落在这两面之一(或两者衔接处)。任何时候迷路,回到这个二分法。

### 主比喻:餐厅服务员

全书统一用**餐厅服务员**比喻。核心对应(背下来):

| 餐厅比喻 | 对应 tokio 概念 | 出现在 |
|------|----------------|--------|
| 服务员(人数少) | **OS worker 线程** | 全书 |
| 客人的订单(海量) | **task**(异步任务) | 全书 |
| 一张订单"等菜"时,服务员不傻站、去服务别桌 | **await 让出**(协作式调度,任务在等时释放线程) | 第 1~2 篇 |
| 厨房喊"3 号桌菜好了" | **reactor / I/O 事件唤醒** | 第 3 篇 |
| 闹钟响"5 号桌的催单到点了" | **timer 唤醒** | 第 4 篇 |
| 餐厅经理(决定哪个服务员接哪单) | **scheduler**(调度器) | 第 2 篇 |
| 忙不过来的服务员,把订单转给闲的同事 | **work-stealing**(偷工作) | 第 2 篇 |
| 每个服务员手边那摞"自己负责的订单" | **本地运行队列(local run queue)** | 第 2 篇 |
| 前台那本"谁都来取"的总订单本 | **全局运行队列(injector)** | 第 2 篇 |
| 服务员没活时,靠着吧台打盹、被叫一声就醒 | **park / unpark**(线程睡眠与唤醒) | 第 2~3 篇 |
| 一张订单本身(折叠好的小纸条,记录做到哪步) | **task = Future 状态机 + 调度壳** | 第 1 篇 |

新概念先给餐厅比喻,再落到源码。可叠加子比喻,但不得与餐厅体系冲突,更不得另起一套。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧**。主线是一条旅程:**阻塞为何低效 → 任务怎么造出来 → 怎么调度 → 怎么等 I/O → 怎么等时间 → 怎么协作 → 进阶真相**。
> 章号全局连续;命名 `P{篇}-{章}-标题.md`,开篇第 1 章 = `P0-01`,第 1 篇第 2 章 = `P1-02`,以此类推。

### 第 0 篇 · 开篇:为什么需要异步运行时(定调,1 章)

- **导言 · 一篇看懂 tokio 全貌**(`P0-00`):用"餐厅服务员"把 task→scheduler→reactor→timer 一口气串一遍,立住"调度执行 vs 事件唤醒"二分法。
- **第 1 章 · 第一性原理:为什么需要异步运行时**(`P0-01`):一个请求一个线程为什么撑不住(线程贵、上下文切换贵、阻塞=浪费)→ async 怎么把"等待"从"占用"解放 → 运行时到底做了哪三件事(scheduler + reactor + timer)。**【样本章节,见后】**

### 第 1 篇 · 地基:任务到底是什么(Future / async-await / Task,4 章)

> 先搞清楚被调度的对象——task——到底是什么。这一篇是全书地基。

- **第 2 章 · Future 与 poll 模型**(`P1-02`)——为什么异步是"状态机 + 轮询"而不是回调;`Future::poll` 的契约;为什么"轮询"反而高效。源码:`std::future::Future`、`Poll`。技巧:poll 返回 `Pending` 时挂起的机制。
- **第 3 章 · async/await 的真相**(`P1-03`)——编译器怎么把一个 `async fn` 编译成一个状态机;`.await` 到底做了什么。**技巧精解:自引用状态机、为什么必须 `Pin`**(移动会野指针)、`Pin`/`Unpin` 的设计。
- **第 4 章 · Waker:谁来唤醒一个挂起的任务**(`P1-04`)——`Waker` 的契约;任务挂起后,凭什么能被重新叫醒。源码:`Waker` / `RawWaker` / vtable。**技巧精解:`RawWaker` 的 vtable + 引用计数布局**(一个 fat pointer 怎么同时携带虚表与数据)。
- **第 5 章 · Task:Future 怎么变成可调度单元**(`P1-05`)——`spawn` 之后发生了什么;task 的内存布局;`JoinHandle`。源码:`tokio/src/task/`、`Header` / `Schedule` trait。**技巧精解:task 状态位打包**(一个 `AtomicUsize` 塞进"引用计数 + scheduled/running/idle + 取消标志"十几个位,无锁位运算更新)、`Header` 布局。

### 第 2 篇 · Runtime:少量线程如何驱动海量任务(4 章)

> 这一篇是 tokio 的心脏:调度器。

- **第 6 章 · Runtime 全貌:scheduler + reactor + timer 三件套**(`P2-06`)——一个 `Runtime` 由哪些部件拼成;它们怎么协作。源码:`runtime/mod.rs`、`runtime/runtime.rs`。技巧:三者如何在一个循环里交替。
- **第 7 章 · work-stealing 调度器**(`P2-07`)——本地队列、全局队列、偷工作;一个 task 从被 spawn 到被执行的全过程。源码:`runtime/scheduler/multi_thread/`(worker / handle / overflow)。**技巧精解:Chase-Lev 无锁双端队列**(owner 端 push/pop、别人 steal 的环形数组,`AtomicUsize` 的 head/tail 怎么做到无锁又正确)。
- **第 8 章 · current-thread vs multi-thread**(`P2-08`)——两种模式各自的取舍;为什么有单线程模式。技巧:LIFO slot(刚唤醒的任务优先跑、提高缓存命中)、injector 队列。
- **第 9 章 · 协作式让出与 budget**(`P2-09`)——为什么不能让一个 task 一直占着线程(budget);yield / 防饿死。**技巧精解:coop budget**(每个 task 默认跑 128 次 poll 就强制让出,这个数字怎么实现、为什么是 128 量级)。

### 第 3 篇 · Reactor 与 I/O:等待不占线程(3 章)

> "等待"的另一半:I/O 事件。线程怎么做到"在等数据时去干别的、数据来了再回来"。

- **第 10 章 · mio 与 epoll:事件驱动的底座**(`P3-10`)——epoll / kqueue / io_uring 是什么;为什么它们是"事件驱动"的关键。源码:`mio` crate。技巧:edge-triggered vs level-triggered、为何 tokio 选 edge。
- **第 11 章 · readiness 模型与 AsyncFd**(`P3-11`)——"准备好读了"再读;注册兴趣、事件来了唤醒。源码:`tokio/src/io/driver.rs`、`AsyncFd`。**技巧精解:注册表 slab + token 映射**(海量 fd 怎么用小整数 token 高效映射回 task)。
- **第 12 章 · reactor 与 scheduler 的握手**(`P3-12`)——I/O 事件怎么把等待中的 task 重新塞回调度队列。

### 第 4 篇 · Timer:成千上万个定时器怎么管(2 章)

> "等待"的另一种:时间。

- **第 13 章 · 层级时间轮(hierarchical timing wheel)**(`P4-13`)——为什么不用最小堆管理海量定时器;层级轮的精妙。源码:`tokio/src/time/wheel/`。**技巧精解:层级时间轮的位运算定位**(用整数的不同位段,直接定位"几号槽",O(1) 插入/到期)。
- **第 14 章 · sleep 的实现与驱动**(`P4-14`)——`sleep` 怎么变成 task;定时器谁来推进(和 reactor 一起 poll)。

### 第 5 篇 · 并发原语:sync 模块(3 章)

> 任务之间怎么协作与同步。

- **第 15 章 · async Mutex / RwLock**(`P5-15`)——为什么不能直接用 `std::sync::Mutex`(会阻塞整个线程);async 版怎么"等锁时不占线程"。技巧:无锁 fast path、争用时入队挂起。
- **第 16 章 · channel:mpsc / oneshot / broadcast**(`P5-16`)——三种 channel 各为哪种场景;有界 mpsc 的背压。源码:`tokio/src/sync/mpsc.rs` 等。**技巧精解:有界 mpsc 的无锁 / 单链表 / slot 复用 + close 语义**。
- **第 17 章 · Notify 与 Semaphore**(`P5-17`)——通知与信号量。技巧:`Notify` 的无锁计数(已到的事件不丢)。

### 第 6 篇 · 网络 I/O(2 章)

- **第 18 章 · AsyncRead / AsyncWrite**(`P6-18`)——把"读写字节"poll 化;和 `Future` 的关系。
- **第 19 章 · TcpListener / TcpStream 的异步实现**(`P6-19`)——accept / connect / read / write 在 readiness 模型下怎么落地。

### 第 7 篇 · 进阶真相(2 章)

- **第 20 章 · spawn / block_on / select 的真相**(`P7-20`)——`spawn` 的完整路径;`block_on` 怎么在当前线程上跑一个 future(不打扰 scheduler);`select!` 宏展开后是什么。
- **第 21 章 · 取消与 shutdown**(`P7-21`)——drop 一个 task 发生什么(取消的传播);`Runtime` 的 graceful shutdown。技巧:引用计数驱动 task 生命周期、`abort` 与 `CancellationToken`。

### 附录

- **A · 全景脉络**:把全书收束成几条哲学(把等待从占用解放、协作式而非抢占式、无锁优先、靠 Rust 类型系统把 unsafe 关进笼子)。
- **B · 源码阅读路线与工具**:`cargo doc` / 源码跳转路线、**`tokio-console`**(运行时观测)、**`loom`**(并发模型测试, tokio 怎么验证无锁代码正确)。

## 六、写作约定(沿用系统系列,微调适配 Tokio)

1. **动机 + 技巧双线**(本书命脉,见第二节):每个关键代码点讲"为什么这么设计" + "用了什么技巧、不这么写会怎样";每章单开"技巧精解"小节。
2. **"为什么"永远先于"怎么做"**:先讲它解决什么本质问题、不这样会怎样,再讲实现。
3. **大白话 + 餐厅服务员比喻**:新概念先给餐厅比喻,再落到源码(落在第三节对照表内)。
4. **关键路径配真实源码**:挑本章最核心的 1~2 处,贴真实片段、逐段讲。**技巧精解**固定在正文后、小结前。
5. **每章标准结构**:① 章首(核心问题 + 读完你会明白 + 难度高的加逃生阀);② 正文若干小节(招牌四段式:**提问 → 不这样会怎样 → 所以这样设计 → 源码/技巧佐证**);③ 技巧精解;④ 章末小结(餐厅比喻回顾 + 回扣"调度执行 vs 事件唤醒"主线 + 五个为什么清单 + 想深入往哪钻 + 引出下一章)。
6. **回扣全局**:每章结尾回到二分法,说明本章服务"调度执行"还是"事件唤醒"那一面,衔接前后章。
7. **语言**:中文,技术名词保留英文(runtime、task、Future、Waker、work-stealing、reactor、epoll 等)。
8. **标点**:**半角 ASCII 逗号 `,`、冒号 `:`、括号 `()`、问号 `?`;全角顿号 `、`、破折号 `——`、引号 `""`、句号 `。`**;中英文之间留空格。
9. **文件命名**:`P{篇}-{章}-标题.md`。第 0 篇第 1 章 = `P0-01-…`,第 2 篇第 7 章 = `P2-07-…`。
