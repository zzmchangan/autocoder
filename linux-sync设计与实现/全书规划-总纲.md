# 《Linux 同步原语深入浅出:RCU 凭什么这么快》—— 全书总纲

> 一本写给"写过 Linux 多线程程序、甚至翻过 `kernel/locking/`、`kernel/rcu/`,却总觉得一知半解"的人的小书。
>
> 本文是全书的**总纲**:定主线、定方法、定分篇分章、定源码策略、定写作约定。
> 与《Tokio》《LevelDB》《etcd》《Go runtime》《内存分配器》《Linux mm》《Linux 调度器》同源,都是"**第一性原理 + 真实源码**";命脉同样是**动机(why) + 技巧(how) 双线**;基调延续《LevelDB》——**直球讲透为主,比喻只在反直觉处点睛**。
> **本书独有特色:★ 对照《Tokio》《Go runtime》《内存分配器》《Linux 调度器》**——锁与无锁不是 Linux 内核独有,Tokio 的 `cmpxchg`/Pin、Go runtime 的原子与 channel、内存分配器的 per-CPU cache、调度器的 `rq->lock` 与延迟抢占,处处是同源思想。把这些钉在一起,才拼得出"并发同步"的完整全栈。

## 一、这本书的定位

### 目标读者

会写 C、读过操作系统课、用过 `pthread_mutex`/`std::sync::atomic`/`sync.Mutex`、甚至翻过 `kernel/locking/mutex.c`、`kernel/locking/rwsem.c`、`kernel/rcu/tree.c`,但**一知半解**:知道有 spinlock、mutex、rwsem、RCU、futex,却讲不清一次 `mutex_lock` 内核的 fast path 用的是 `cmpxchg` 还是自旋、乐观读的 A-D-S 手写为什么不会读到撕裂数据、RCU 的读者为什么可以不加锁而不丢更新、宽限期到底在等什么、tree RCU 的 `rcu_node` 层级怎么把"等所有 CPU 静默"这件事做成 O(log N)、futex 怎么做到"无竞争时不进内核"。和"读过 tokio/go runtime/mm/sched 源码但不理解"是同一种病。

### 全书一句话主旨

> **多核共享可变数据,要既不出错(一致性/可见性/有序性)又不被锁拖死(竞争/上下文切换/缓存行乒乓)。阻塞睡眠锁(mutex/rwsem/futex)等不到就睡,自旋/无锁(spinlock/RCU/per-cpu/seqlock)要么死等要么根本不锁——两条路各有 sound 的理由和要付的代价,内核用几十年的工程把它们磨到了极致。**

每个组件,都是"在正确性与性能之间找平衡"这条路上的一个**驿站**。读完一本书,读者能**在脑子里放映出多核同步的全过程**:一次 `mutex_lock` 怎么走 fast path、争用了怎么睡;一次 RCU 读端为什么零开销、写者怎么延迟回收;一次 rwsem 乐观读的 A-D-S 手写为什么 sound——以及每一步底下用了什么巧妙的手段,**为什么不出错**。

### 为什么用"多核共享数据怎么不出错又不被锁拖死"当骨架(而非按文件平铺)

`kernel/locking/` + `kernel/rcu/` + `kernel/futex/` 加起来几十个 `.c`、一堆内部头、大量宏与体系结构相关代码,高度耦合:原子操作与内存屏障是地基、spinlock 类(ticket/qspinlock)与 seqlock 是不阻塞一极、mutex/rwsem/futex 是阻塞一极、RCU 自成一派(读者不锁、写者延迟回收)、lockdep 是验证工具。按文件百科式平铺,读者会觉得每块都是孤岛。用"共享数据怎么不出错又不被锁拖死"这条主线当骨架——原子与屏障是地基,然后是"要么自旋/无锁、要么睡眠"两条路,RCU 是"读者零开销"的终极解,lockdep 兜底——每个模块都是"为了让这条路走得通且 sound"而出现的角色,读者永远知道**我在哪、为什么需要下一站**。

### 为什么是精品、技巧密度最高

本书是内核招牌技巧密度最高的一本:cmpxchg fast path、MCS 队列锁、优先级继承链、乐观读 A-D-S 手写、per-cpu 计数器、RCU 宽限期 + 静默态 + tree 层级报告、srcu 可睡眠读者、内存序屏障。每一条都既是**性能优化**(把竞争降到极致),又是**正确性陷阱**(少一个 `smp_mb`、写反一个 `acquire`/`release` 就会丢更新/读到撕裂)。本书偏精品,意味着**每章必须讲清"为什么 sound"**——不是"它这么写",而是"它必须这么写,不这么写就会在某条执行序下出错"。这是本书的命脉。

## 二、写作方法

### 命脉:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与一般操作系统教材、源码导读最关键的分野,所有写作者必读、必守。**

Linux 同步原语源码充斥内核级技巧:`cmpxchg` fast path、MCS 队列(qspinlock)、优先级继承(PI 链遍历)、rwsem 乐观读的 A-D-S 手写(`smp_load_acquire`/`smp_store_release` 配对)、per-cpu 无锁计数、RCU 宽限期 + 静默态 + tree `rcu_node` 层级报告、srcu 可睡眠读者、内存序屏障(`smp_mb`/`READ_ONCE`/`WRITE_ONCE`)。读者自己读过源码,"一知半解"的缺口**往往不在动机**(锁解决什么问题好懂),**而在技巧**(为什么这么写、为什么 sound、少一个屏障会怎样)。

所以本书每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?(沿用系列一贯的"为什么优先")
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱——尤其是"为什么会丢更新/死锁/读到撕裂")?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果不用这个技巧、朴素地写,会撞上什么墙(在多核并发下具体的某条执行序出错)",让妙处显形。

> **本书命脉:每个锁/RCU/内存序技巧必须讲清"为什么 sound"**——即"在所有并发执行序下,为什么不会丢更新、不会死锁、不会破坏内存序、不会读到撕裂数据"。这是本书区别于一切"锁怎么用"运维书的关键,也是用户读过源码仍一知半解的最大缺口。

一句话:**讲不清"为什么 sound",等于没讲 Linux 同步原语。**

### 基调:直球讲透为主,比喻只在反直觉处点睛 ⚠️ 延续《LevelDB》

同步原语有一部分反直觉(RCU 读者为什么不锁也不丢更新、乐观读为什么 A-D-S 不会读到撕裂、内存重排为什么和直觉相反),这部分**允许适度比喻点睛**。但同步原语的大部分(`struct mutex` 的 owner 字段编码、MCS 队列、RCU 宽限期状态机、PI 链遍历)**具体可见,直球讲清最好**。

- **默认直球**:能用表 / 时序图 / 源码说清的,不上比喻。
- **比喻只点睛**:RCU 可点"像写者先把老版本挂着,等所有读者交卷了再回收";乐观读可点"像边读边看门牌号,门牌号变了就重读"。点到为止。
- **判断标准**:直球讲不清才上比喻。

### ★ 对照多本(本书独有特色)

本书和系列多本呼应,锁与无锁绝非内核独有。标 ★ 的章在章末加轻量"★对照"小栏,和下列书对读:

- **对照《Tokio》**(异步运行时):Tokio 的无锁队列(`mpsc`/Crossbeam)、`AtomicWaker`、`Pin` 的"读者不搬走内存",和 RCU 的"读者不阻塞写者、写者延迟回收"是同源思想——都是"用某种契约换无锁"。
- **对照《Go runtime》**(语言级并发):Go 的 `sync/atomic`、`sync.Mutex`(fast path 也是 `cmpxchg` + 自旋 + sema 睡眠)、`channel` 底层的 `hchan.buf`(用一把 mutex 保护),和内核 mutex/futex 是同款 fast/slow path 分层。Go 的 GMP 调度(协作抢占、`gogo` 切栈)对照内核的延迟抢占(`TIF_NEED_RESCHED`)。
- **对照《内存分配器》**(tcmalloc/jemalloc):per-CPU cache 的 `thread_local` / per-CPU 链表,是 per-cpu 计数器和 seqlock 的思想源头——"把竞争消灭在结构里"。
- **对照《Linux 调度器》**(姊妹篇):`rq->lock` 是 spinlock、`preempt_count` 抢占计数、`TIF_NEED_RESCHED` 延迟抢占——本系列调度器那本重点讲了这些,本书正面讲它们的实现。两本天然呼应,可互引。

收尾章给一张总对照表,把锁/RCU 与 Tokio/Go/分配器/调度器钉在一起。

## 三、全书主线、二分法

### 主线

> **多核共享可变数据,要既不出错(一致性/可见性/有序性)又不被锁拖死(竞争/上下文切换/缓存行乒乓)。阻塞睡眠锁(mutex/rwsem/futex)等不到就睡,自旋/无锁(spinlock/RCU/per-cpu/seqlock)要么死等要么根本不锁——内核用几十年工程把它们磨到极致,每一步都既快又 sound。**

任何一处看不懂某个机制,回到这句问:"这是在**阻塞睡眠(等不到就睡)**,还是在**自旋/无锁(不等或根本不锁)**,或是在**为这两条路铺地基(原子/屏障/lockdep)**?"把答案写进"为什么"里。

### 二分法(迷路时回到它)

本书的二分法:

> **阻塞睡眠锁(mutex/rwsem/futex,等不到就睡,睡在 wait queue 上) vs 不阻塞自旋/无锁(spinlock/qspinlock 死等、seqlock 读者不阻塞写者、per-cpu 不竞争、RCU 读者根本不锁)。**

- **阻塞睡眠一极**:`mutex_lock` 慢路径(进 wait queue → `schedule()` 睡眠 → 被 unlock 唤醒)、`rwsem` 写者慢路径、rtmutex 的 PI 链、futex 争用进内核(`futex_wait` 睡眠在 `futex_q` 上)。这些回答"等不到怎么办——睡,等别人唤醒"。
- **自旋/无锁一极**:spinlock/qspinlock(等不到就 `cpu_relax()` 自旋)、seqlock(读者用版本号重读,不阻塞写者)、per-cpu(每核一份,根本不竞争)、RCU(读者只关抢占/禁迁移,不取锁,写者延迟回收)。这些回答"怎么不睡也能正确"。

支撑这两极的地基:**原子操作**(`atomic_t`/`cmpxchg`)、**内存屏障**(`smp_mb`/`rmb`/`wmb`、`READ_ONCE`/`WRITE_ONCE`)、**lockdep**(运行时验证锁的正确性)。

同步原语每个机制都落在这两极之一或支撑地基。任何时候迷路,回到这个二分法。

> **为什么这个二分法是命脉**:它直击同步原语的设计分野。睡要付出"上下文切换 + 唤醒延迟"的代价(μs 级),但持锁可以很久(睡着的进程不占 CPU);自旋要付出"白白烧 CPU"的代价,但拿锁延迟极低(ns 级),且只在持锁很短时才划算。RCU 走到极致——读者根本不取锁,代价是写者要延迟回收。**所有同步原语的设计,都是在"睡 vs 自旋 vs 不锁"三选一里权衡**,这就是本书的组织轴。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线是一条旅程:**为什么多核需要同步 → 原子与屏障地基 → 自旋锁类(不阻塞一极) → 阻塞锁类(睡眠一极) → 读写锁 → RCU(读者零开销)→ 应用与收尾**。
> 章号全局连续;命名 `P{篇}-{章}-标题.md`。**全书 18 章 6 篇**(核心全覆盖,RCU 重头戏占 5 章)。

### 第 0 篇 · 开篇:为什么多核需要同步(定调,1 章)

- **第 1 章 · 第一性原理:为什么多核需要同步**(`P0-01`):CPU 是多核共享 + 共享可变数据 + 缓存一致性(MESI)+ 内存重排 → 不加同步会竞争出错、可见性错乱、有序性错乱 → 必须有同步原语既不出错又不被锁拖死 → 立起"阻塞睡眠 vs 自旋/无锁"二分法 + ★对照(Tokio/Go 无锁、分配器 per-cpu、调度器 rq->lock)。**【样本章节】**

### 第 1 篇 · 地基:原子操作、内存屏障、lockdep(3 章)

> 这是支撑两条路的地基。源码主体 `include/linux/atomic/atomic-instrumented.h`、`include/linux/compiler.h`(barrier)、`include/asm-generic/barrier.h`(本地未 sparse,描述 + 引在线 6.9)、`kernel/locking/lockdep.c`、`include/linux/lockdep.h`、`include/linux/lockdep_types.h`。

- **第 2 章 · 原子操作:atomic_t 与 cmpxchg**(`P1-02`)——为什么 `i++` 不是原子的(读-改-写三步)、`atomic_t` 封装、`READ_ONCE`/`WRITE_ONCE`、`cmpxchg`(LL/SC 与 CAS)、原子操作的内存序参数(`acquire`/`release`/`relaxed`/`seq_cst`)。源码:`include/linux/atomic/atomic-instrumented.h`、`include/linux/atomic/atomic-arch-fallback.h`。**技巧精解:cmpxchg 的 LL/SC 实现与 ABA 问题**(反面对比"普通 read-modify-write 在多核会丢更新")。归属:`支撑地基`。
- **第 3 章 · 内存屏障:为什么重排会害死你 + smp_mb/rmb/wmb**(`P1-03`)——CPU 乱序执行 + 编译器重排 → 共享内存上的"可见性/有序性"崩溃 → `barrier()`/`smp_mb`/`smp_rmb`/`smp_wmb`/`smp_store_release`/`smp_load_acquire`,以及 `READ_ONCE`/`WRITE_ONCE` 阻止编译器重排。Store Buffer 与 Invalidate Queue 直觉(MESI)。源码:`include/asm-generic/barrier.h`(本地未 sparse,描述其作用 + 引在线)、`include/linux/compiler.h`(`barrier()` @L85)。**技巧精解:经典消息传递模式(Store-Buffer reordering)与 smp_wmb/smp_rmb 配对**(配反例时序图,讲清"少了屏障为什么会读到新数据但标志位是旧值")。归属:`支撑地基`。
- **第 4 章 · lockdep 与运行时验证**(`P1-04`)——锁的正确性怎么验证:`lock_acquire`/`lock_release` 钩子、依赖图、死锁检测、IRQ 安全性(spin_lock_irqsave 的 `usage` 位图)、`struct held_lock`。源码:`kernel/locking/lockdep.c`(`lock_acquire` @L5719、`lock_release` @L5761、`validate_chain` @L3822)、`include/linux/lockdep.h`、`include/linux/lockdep_types.h`(`struct lock_class_key` @L75、`struct held_lock` @L205)。**技巧精解:依赖图 + 死锁检测**(反面对比"没有 lockdep,死锁只在生产环境炸")。归属:`支撑地基`。

### 第 2 篇 · 自旋锁类:不阻塞一极(3 章)

> 等不到就自旋或根本不阻塞。源码主体 `kernel/locking/spinlock.c`、`kernel/locking/qspinlock.c`、`include/linux/spinlock.h`、`include/linux/seqlock.h`、`include/linux/seqlock_types.h`。

- **第 5 章 · spinlock 演进:从 ticket 到 qspinlock**(`P2-05`)——朴素 TAS 自旋的公平性与缓存乒乓问题 → ticket lock(先到先得)→ **MCS 队列锁**(每个 spinner 自旋在自己的本地变量上,唤醒按队列传)→ **qspinlock**(把 MCS 压进一个原子字)。源码:`kernel/locking/qspinlock.c`(`queued_spin_lock_slowpath` @L316、MCS 注释 @L32)、`kernel/locking/spinlock.c`(`_raw_spin_lock` @L152、`_raw_spin_lock_irqsave` @L160)、`include/linux/spinlock.h`(`spin_lock` 内联 @L349、`spin_lock_irqsave` 宏 @L379)。注:`include/asm-generic/qspinlock.h` 与 `struct qspinlock` 本地未 sparse,描述其结构 + 引在线。**技巧精解:MCS 队列把缓存乒乓消灭在结构里**(反面对比"朴素 TAS,64 核抢一把锁全是缓存行乒乓")。归属:`自旋/无锁一极`。
- **第 6 章 · 顺序锁 seqlock:读者不阻塞写者**(`P2-06`)——读多写少但写者不能等的场景(如 `jiffies`、`ktime`)怎么解:seqlock 读者用版本号重读(`read_seqcount_begin`/`read_seqcount_retry`),写者用 `write_seqlock` 加版本号。读者**根本不阻塞写者**,写者也不等读者——读者最多多读几遍。源码:`include/linux/seqlock.h`(`read_seqbegin` @L770、`read_seqretry` @L790、`__read_seqcount_begin` @L279 重读循环、`write_seqlock` @L820)、`include/linux/seqlock_types.h`(`seqlock_t` @L84、`seqcount_t` @L33)。**技巧精解:奇偶版本号 + READ_ONCE 重读**(反面对比"rwsem 读者要等写者,实时统计被卡"),讲清"为什么重读能保证读到一致快照"。归属:`自旋/无锁一极`。
- **第 7 章 · 中断禁用与 spin_lock_irqsave**(`P2-07`)——为什么持 spinlock 时不能被中断(中断处理里再拿同一把锁就死锁)→ `spin_lock_irqsave`/`spin_lock_bh` 的 IRQ/软中断禁用、与 `preempt_disable` 的关系、IRQ 上下文不能睡眠的铁律(回扣调度器那本的抢占点)。源码:`include/linux/spinlock.h`(`spin_lock_irqsave` @L379)、`kernel/locking/spinlock.c`(`_raw_spin_lock_irqsave` @L160、`_raw_spin_lock_bh` @L176)。**技巧精解:IRQ 上下文不可睡眠 + irqsave 的"保存-恢复"**(反面对比"持 spinlock 被中断抢去调度,直接挂死"),讲清"为什么 sound"。归属:`自旋/无锁一极`。

### 第 3 篇 · 阻塞锁:睡眠一极(3 章)

> 等不到就睡。源码主体 `kernel/locking/mutex.c`、`kernel/locking/mutex.h`、`include/linux/mutex.h`、`include/linux/mutex_types.h`、`kernel/locking/rtmutex.c`、`kernel/locking/rtmutex_api.c`、`kernel/futex/`(本地未解压,引在线 6.9)。

- **第 8 章 · mutex:fast path + 慢路径睡眠**(`P3-08`)——mutex 与 spinlock 的分野(持锁可睡眠)→ fast path:`cmpxchg` 抢 owner 字段(0 → curr)→ 慢路径:进 wait queue、`schedule()` 睡眠、被唤醒后重试 → 乐观自旋(`mutex_optimistic_spin`,持锁者在跑就再等等不睡)→ owner 字段低位编码 waiters/handoff 标志。源码:`kernel/locking/mutex.c`(`mutex_lock` @L281、`__mutex_trylock_fast` @L166 含 `cmpxchg_acquire` @L171、`__mutex_unlock_fast` @L177、`__mutex_lock_common` @L573、`mutex_optimistic_spin` @L440、`mutex_unlock` @L542、`__mutex_add_waiter` @L204)、`include/linux/mutex_types.h`(`struct mutex` @L41,owner 字段 @L42)、`kernel/locking/mutex.h`(`struct mutex_waiter` @L14)。**技巧精解:owner 字段低位编码 + fast/slow path 分层**(反面对比"朴素 mutex 每次都进 wait queue 睡,短临界区白白付上下文切换"),讲清"为什么乐观自旋 + cmpxchg 能消灭绝大多数睡眠开销"。归属:`阻塞睡眠一极`。
- **第 9 章 · rtmutex:优先级继承**(`P3-09`)——优先级反转问题(低优进程持锁,高优进程等它,中优进程抢占低优 → 高优被中优间接拖死)→ rtmutex 让持锁者临时继承等待者的最高优先级(PI 链遍历 `rt_mutex_adjust_prio_chain`)→ rtmutex 是 PREEMPT_RT 上 mutex/spinlock/rwsem 的底层。源码:`kernel/locking/rtmutex.c`(`task_blocks_on_rt_mutex` @L1200、`rt_mutex_adjust_prio_chain` @L675、`rt_mutex_adjust_prio` @L524)、`kernel/locking/rtmutex_api.c`(`rt_mutex_lock` @L69)。**技巧精解:PI 链遍历**(反面对比"无 PI,Jupiter 探测器式的优先级反转会让关键任务错过截止期"),讲清"链上每个锁的持锁者都继承最高优先级,为什么不会成环"。归属:`阻塞睡眠一极`。
- **第 10 章 · futex:用户态锁的 fast path**(`P3-10`)——为什么 `pthread_mutex` 在无竞争时不进内核:futex 用户态先用 `cmpxchg` 试,抢到就完事;抢不到才 `sys_futex(FUTEX_WAIT)` 进内核,睡在 `futex_q` 上、挂在 per-hash-bucket 的 hash 桶上;unlock 时 `sys_futex(FUTEX_WAKE)` 唤醒。内核与用户态的契约:futex word 的原子性。源码:`kernel/futex/core.c`、`kernel/futex/pi.c`、`kernel/futex/waitwake.c`(本地未解压,引在线 6.9;`io_uring/futex.c` 本地有但非核心)。**技巧精解:futex 的双 fast path**(用户态 cmpxchg + 内核态等待队列;反面对比"纯内核 mutex 每次都 syscall,无竞争时白白付系统调用开销")。归属:`阻塞睡眠一极`。

### 第 4 篇 · 读写锁:读多写少场景(2 章)

> 读者与读者不互斥、读者与写者互斥。源码主体 `kernel/locking/rwsem.c`、`include/linux/rwsem.h`、`kernel/locking/percpu-rwsem.c`、`include/linux/percpu-rwsem.h`。

- **第 11 章 · rwsem:乐观读的 A-D-S 三段手写**(`P4-11`)——rwsem(读写信号量)支持多读者、单写者,可睡眠 → 乐观读(Optimistic Read):reader 用 `rwsem_read_trylock` 试(`atomic_long_try_cmpxchg_acquire`),命中就直接走,不进 wait queue;争用才进慢路径。乐观读的核心是 **A-D-S 手写**(Acquire-Do-Release):读者慢路径用 `smp_load_acquire(&waiter.task)` 等被唤醒,唤醒者用 `smp_store_release()` 配对——这对屏障配对保证读者看到的等待状态一致。owner 字段低位编码 `RWSEM_READER_OWNED`/`RWSEM_NONSPINNABLE` 决定能否乐观自旋。源码:`kernel/locking/rwsem.c`(`down_read` @L1523、`rwsem_read_trylock` @L241、`rwsem_down_read_slowpath` @L996 含乐观偷抢块 @L1012-1032 与 for 循环 @L1072 + `smp_load_acquire` @L1074、`rwsem_mark_wake` @L411、`up_read` @L1619、`__up_read` @L1338 含 `add_return_release` @L1347、`rwsem_optimistic_spin` @L819)、`include/linux/rwsem.h`(`struct rw_semaphore` @L48,owner @L55、osq @L57、count @L49)。**技巧精解:乐观读 A-D-S 手写**(反面对比"朴素 rwsem 读者每次都进 wait queue,热路径全在排队",再讲清"smp_load_acquire / smp_store_release 配对为什么不会读到撕裂数据")。归属:`阻塞睡眠一极`(读者睡眠慢路径)+ 混合乐观自旋。
- **第 12 章 · percpu-rwsem:per-cpu 计数器换读者零争用**(`P4-12`)——rwsem 在多读者场景仍有 cache line 乒乓(per-CPU 都改一把 count)→ percpu-rwsem 给每个 CPU 一个本地 `read_count`,读者只 `this_cpu_inc` 本地计数(零争用);写者拿到 write 后,挨个把读者计数和全局状态切换到慢路径(`rcu_sync` + `percpu_down_write` 的 wait + synchronize_rcu)。源码:`kernel/locking/percpu-rwsem.c`(`__percpu_down_read_trylock` @L48 含 `this_cpu_inc(*sem->read_count)` @L50、`__percpu_down_read` 慢路径 @L167、`percpu_down_write` @L224)、`include/linux/percpu-rwsem.h`(`struct percpu_rw_semaphore` @L12、`percpu_down_read` 内联 @L47、`percpu_up_read` 内联 @L97)。**技巧精解:per-cpu 计数器 + rcu_sync**(反面对比"rwsem 64 核读者抢同一 count,缓存行乒乓"),讲清"读者快路径零争用,代价是写者很重(切换 + 等 RCU)"。归属:`自旋/无锁一极` 读者快路径 + 阻塞睡眠 写者。

### 第 5 篇 · RCU:读者零开销的终极解(5 章)⚠️ 核心重头戏

> RCU(Read-Copy-Update):读者不加锁(只关抢占/禁迁移),写者复制一份改、老的延迟回收。代价是写者要等"所有读者都交卷"(宽限期)。源码主体 `kernel/rcu/tree.c`、`kernel/rcu/tree.h`、`kernel/rcu/tree_plugin.h`、`kernel/rcu/update.c`、`kernel/rcu/srcutree.c`、`kernel/rcu/sync.c`、`include/linux/rcupdate.h`、`include/linux/srcu.h`。

- **第 13 章 · RCU 原理:读者无锁,写者延迟回收**(`P5-13`)——RCU 的核心契约:读者只标记自己进入临界区(`rcu_read_lock`,6.9 是 `static __always_inline` @rcupdate.h#L777,只增 preempt 计数)、写者改前复制一份、老的等宽限期后回收(`kfree_rcu`/`call_rcu`)。RCU 只能保护**指针数据结构**(读者拿到一份快照)。为什么 sound:宽限期保证所有"开始时进入的读者"都退出了。源码:`include/linux/rcupdate.h`(`rcu_read_lock` @L777、`rcu_read_unlock` @L808、`rcu_lock_acquire` @L327、`RCU_LOCKDEP_WARN` @L394)、`kernel/rcu/update.c`、`kernel/rcu/tree.c`(`synchronize_rcu` @L3600、`call_rcu` @L2836)。**技巧精解:读者零开销**(反面对比"rwsem 读者要原子加 count,64 核读者仍 cache line 乒乓;RCU 读者只动本地 preempt 计数"),讲清"为什么读者不加锁也不丢更新——读者拿的是快照指针,写者改的是新副本,老的还在"。归属:`自旋/无锁一极`(读者无锁)。
- **第 14 章 · grace period 与 quiescent state:宽限期在等什么**(`P5-14`)——宽限期(Grace Period)的本质:等所有 CPU 都至少经过一次**静默态**(Quiescent State,即不在 RCU 临界区)→ tick 钩子 `rcu_sched_clock_irq`(6.9 改名,`@tree.c#L2275`)检测每个 CPU 是否静默 → 报告给 `rcu_node` 树 → 根节点收集齐 → 宽限期结束 → 触发回调。源码:`kernel/rcu/tree.c`(`rcu_sched_clock_irq` @L2275、`rcu_report_qs_rdp` @L2008、`rcu_report_qs_rnp` @L1905、`rcu_report_qs_rsp` @L1880、`rcu_do_batch` @L2119、`call_rcu` @L2836、`synchronize_rcu` @L3600)。**技巧精解:静默态检测**(反面对比"朴素做法用计数器,但读者可能没退出;RCU 用'每个 CPU 至少经过一次静默'保证所有读者已交卷")。归属:`自旋/无锁一极`。
- **第 15 章 · tree RCU 的层级报告:`rcu_node` 树**(`P5-15`)——多核机器上"等所有 CPU 静默"怎么不慢:把 CPU 分组成树形 `rcu_node`(`@tree.h#L41`),叶子节点管若干 CPU,中间节点聚合,根节点定宽限期完成。报告按层级上传,避免根节点成瓶颈。GP 状态机(`RCU_GP_IDLE`/`RCU_GP_INIT`/`RCU_GP_CLEANUP` 等 @tree.h#L411-L419)。GP 主循环 `rcu_gp_kthread`(@tree.c#L1836,for(;;) @L1839)。源码:`kernel/rcu/tree.c`(`rcu_gp_kthread` @L1836、`rcu_gp_init` @L1428、`rcu_gp_cleanup` @L1718、`rcu_state` @L85)、`kernel/rcu/tree.h`(`struct rcu_node` @L41、`struct rcu_data` @L178、`struct rcu_state` @L328、GP 状态宏 @L411-L419)。**技巧精解:rcu_node 层级树 + 层级化报告**(反面对比"所有 CPU 直接报告给根,根节点成锁瓶颈"),讲清"为什么 sound:树根收集齐等价于所有 CPU 静默"。归属:`自旋/无锁一极`。
- **第 16 章 · srcu:可睡眠的 RCU 读者**(`P5-16`)——普通 RCU 读者不能睡眠(只在关抢占里);srcu(Sleepable RCU)读者可以睡眠(`srcu_read_lock` 内含计数加 1,允许 `schedule`)。代价是 srcu 宽限期更复杂(要扫描两个 counter 槽位)。源码:`kernel/rcu/srcutree.c`(`__srcu_read_lock` @L712、`__srcu_read_unlock` @L728、`srcu_funnel_gp_start` @L989)、`include/linux/srcu.h`。**技巧精解:srcu 双计数槽**(反面对比"普通 RCU 读者睡了,宽限期永远不结束;srcu 用两个 counter 切换"),讲清"为什么 sound:新读者进新槽,等老槽清空就是宽限期"。归属:`自旋/无锁一极`(读者可睡眠但不取锁)。
- **第 17 章 · RCU 同步原语家族:`rcu_sync` 与 Tasks Trace**(`P5-17`)——基于 RCU 构建的轻量同步原语:`rcu_sync`(`kernel/rcu/sync.c`,被 percpu-rwsem 用)做"写者通知读者开始/结束切换"、Tasks Trace RCU(用于 Tracing,读者范围更窄)。源码:`kernel/rcu/sync.c`、`kernel/rcu/tree_plugin.h`。**技巧精解:rcu_sync 的状态机**(只切换一次 RCU 宽限期,避免重复 synchronize_rcu)。归属:`自旋/无锁一极`。

## 七、内核 C 技巧侧重(对标 Tokio Rust / Go runtime / LevelDB C++ / mm/sched 内核 C)

本书要重点拆的 Linux 内核同步技巧(写作时遇到这些,务必钻"怎么做到的 + 为什么 sound"):

- **cmpxchg fast path**:`mutex_lock` 用 `cmpxchg_acquire` 抢 owner 字段(`@mutex.c#L171`),无竞争时不进内核慢路径——fast/slow path 分层。
- **owner 字段低位编码**:mutex/rwsem 的 `atomic_long_t owner` 把 task 指针 + waiters/handoff/READER_OWNED/NONSPINNABLE 标志压进一个原子字(`mutex_types.h#L42`、`rwsem.h#L55`、`rwsem.c#L63-L65`)。
- **MCS 队列锁 / qspinlock**:把"每个 spinner 自旋在自己的本地变量上"压进一个原子字,消灭缓存行乒乓(`qspinlock.c#L316` + MCS 注释 @L32)。
- **优先级继承(PI)链**:`rt_mutex_adjust_prio_chain` 遍历锁等待图,链上每个持锁者继承最高优先级(`rtmutex.c#L675`)。
- **rwsem 乐观读 A-D-S 手写**:reader 慢路径 `smp_load_acquire(&waiter.task)`(@rwsem.c#L1074)与 waker `smp_store_release` 配对,保证读到的等待状态一致——**这是"为什么不会读到撕裂数据"的命脉**。
- **seqlock 奇偶版本号**:读者用 `READ_ONCE(sequence)` 重读(`@seqlock.h#L279`),写者加版本号前后各写——读者重读一致快照。
- **per-cpu 计数器**:percpu-rwsem 读者 `this_cpu_inc(*sem->read_count)`(`@percpu-rwsem.c#L50`)零争用,写者用 `rcu_sync` 切换。
- **RCU 读者零开销**:`rcu_read_lock` 只增 preempt 计数(`@rcupdate.h#L777`),不取锁、不原子、不缓存行乒乓。
- **tree RCU 层级报告**:CPU 分组成 `rcu_node` 树(`@tree.h#L41`),报告按层级上传,根节点收集齐 = 宽限期结束——SMP 可扩展性。
- **GP 状态机**:`RCU_GP_IDLE`/`RCU_GP_INIT`/`RCU_GP_CLEANUP` 等(`@tree.h#L411-L419`),`rcu_gp_kthread` 主循环驱动。
- **静默态检测**:`rcu_sched_clock_irq`(`@tree.c#L2275`,6.9 改名)在 tick 里检测 CPU 是否静默,报告给 `rcu_node`。
- **srcu 双计数槽**:读者可睡眠,用两个 counter 槽位切换,新读者进新槽,等老槽清空 = 宽限期。
- **futex 双 fast path**:用户态 cmpxchg + 内核态 wait queue(`kernel/futex/`,在线 6.9)。
- **内存序屏障**:`smp_mb`/`smp_rmb`/`smp_wmb`/`smp_store_release`/`smp_load_acquire`(`include/asm-generic/barrier.h`,本地未 sparse,描述 + 引在线),Store Buffer/Invalidate Queue 直觉。
- **lockdep 依赖图**:`lock_acquire`/`validate_chain`(`@lockdep.c#L5719`/`L3822`)运行时死锁检测与 IRQ 安全性。
- **IRQ/softirq 上下文不可睡眠**:`spin_lock_irqsave`/`spin_lock_bh`(`@spinlock.h#L379`/`spinlock.c#L160`)与 preempt_disable 的关系。
