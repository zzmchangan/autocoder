> 一本写给"写过 C/C++、调用过 `malloc`/`free`,甚至翻过 ptmalloc 或 jemalloc 源码,却总觉得一知半解"的人的小书。

## 一、这本书的定位

### 目标读者

会写 C/C++、天天用 `malloc`/`free`(或 `new`/`delete`),甚至翻过 glibc 的 `malloc/malloc.c` 或 jemalloc 的 `src/`,但**一知半解**:知道有 arena、chunk、free list、size class、thread cache,却讲不清它们怎么协作、为什么这么设计、源码里那些原子操作、per-CPU slab、radix tree、huge page filler、`madvise(MADV_DONTNEED)` 到底在干什么。和"读过 tokio 源码但不理解""读过 leveldb 源码但不理解"是同一种病——**动机好懂,技巧看不懂**。

### 全书一句话主旨

> **`malloc`/`free` 是一切程序的底座。操作系统给的 `mmap`/`brk` 太粗(按页,4KB 起),glibc ptmalloc 的 arena 锁在高并发下成了瓶颈、碎片也压不住。tcmalloc/jemalloc/mimalloc 这一代分配器的核心,是把"一次 malloc"拆成三层快慢道:线程本地缓存(无锁,O(1),纳秒级)→ 中心自由链表(按大小分级,批量,低争用)→ OS 页堆(整批向系统要/还内存)。快(低延迟、低争用、高吞吐)和省(低碎片、低占用、及时归还 OS)是永恒张力,每一个设计都在两者间权衡。**

每个组件,都是这条旅程上的一个**驿站**。读完一本书,读者不是"多记了几个环境变量",而是能**在脑子里放映出一次 `malloc` 的一生 + 一次 `free` 的归宿**:一个指针怎么被要出来、先在本地缓存里秒拿、缓存空了去中心自由链表批量补货、中心也没有了去页堆切 span、超大块直接 `mmap`、释放时怎么回到自由链表、碎片怎么被合并、空闲内存怎么被 `madvise` 归还 OS、多核下怎么不让锁成瓶颈——以及每一步底下用了什么巧妙的手段。

### 为什么用"一次 malloc 的一生 + 后台归还"当骨架(而非按分配器平铺)

这本书有**四套**分配器。如果按"先讲完 tcmalloc、再讲完 jemalloc……"来组织,读者会在第三套就疲劳、且每个机制都要被讲四遍。更糟的是,这四套解决的是**同一组问题**(怎么快、怎么省、怎么并发),它们的差异恰恰是本书最值钱的部分——**同样的问题,四套给出了不同答案,对比之下,每个设计的"为什么"才真正显形**。

所以本书的骨架是**横切的**:沿着"一次 `malloc` 怎么走完三层快慢道"这条旅程,每个驿站(size class、线程缓存、自由链表、中心链表、页堆、大块、多核并发、碎片归还、工程化)都**四套并排对照**。读者永远知道**我在哪、为什么需要下一站、四套在这里各自怎么选**。这条旅程走完,四套分配器的全貌和取舍也就讲透了。

## 二、写作方法:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与 API 文档、与一般源码导读最关键的分野,所有写作者必读、必守。**

内存分配器源码里充斥 C/C++ 系统级技巧:无锁原子自由链表(把 `next` 指针藏在释放块里)、TLS / per-CPU slab(restartable sequences / rseq)、`acquire`/`release` 内存序、size class 的位运算映射、radix tree 指针反查、放射状 pagemap、huge page filler 的碎片打包、`madvise(MADV_DONTNEED)` vs `MADV_FREE` 的取舍、缓存行对齐防 false sharing、几何采样 profiling、fork 前后的锁状态处理。用户自己读过源码,"一知半解"的缺口往往**不在**"它要解决什么问题"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书的每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?(沿用系列一贯的"为什么优先")
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果不用这个技巧、朴素地写,会撞上什么墙",让妙处显形。

一句话:**讲不清技巧,等于没讲分配器。**

## 三、基调:直球讲透为主,比喻只在反直觉处点睛 ⚠️ 沿用 LevelDB

> 这一条由系列已拍板的方法论决定(LevelDB 起确立):**"也不用非要用比喻,如果不是很抽象的话,能讲清楚就行。"**

内存分配器的机制大多**具体可见**——自由链表就是链表、size class 就是查表、span 就是连续页、radix tree 就是分级索引。这些直球讲清楚最清楚,硬套一个贯穿比喻反而啰嗦、喧宾夺主、还可能不准(这跟 Tokio 的 async 运行时不一样,后者反直觉,必须用"餐厅服务员"贯穿兜住)。

所以本书基调:

- **默认直球**:把数据结构、流程、源码本身讲透。能用一张表、一张图、一段源码说清的,**不上比喻**。
- **比喻只点睛,不贯穿,不另起体系**:只在少数**反直觉或本质抽象**处,用一两句话点一个比喻帮助"啊哈",点到为止。例如:
  - 三层快慢道的整体心智("本地缓存秒拿、中心货架批量补货、总仓库整箱进出"):可点一句"像工位手边的零件盒 → 车间中转货架 → 总仓库"。
  - 线程本地缓存"借了不还"换无锁:可点一句"像每个工人手边自己攒一盒零件,不跟人抢,代价是每人占一份"。
  - 碎片治理 / huge page filler:可点一句"像仓库管理员把零散空位打包进整箱,凑满一箱才退还供应商"。
  - 但**绝不**展开成术语对照表、**绝不**强制每章出现、**绝不**和别的比喻混用。
- **判断标准**:直球讲不清的,才上比喻;讲得清的,就直球。比喻服务于理解,不是装饰,更不是 KPI。

## 四、全书主线、二分法、点睛比喻

### 主线

> **如何让 `malloc`/`free` 又快又省?——快:用线程/CPU 本地缓存换无锁 O(1) 的 fast path;省:用中心页堆批量进出、用合并与 `madvise` 归还 OS、用大页治碎片。每一层都是"快"与"省"的权衡。**

任何一处看不懂某个机制,回到这句问:"这是在让**线程本地分配更快**(fast path、无锁、纳秒级),还是在让**全局内存更省更整齐**(slow path、合并、归还、大页)?"把答案写进"为什么"里。

### 二分法(迷路时回到它)

本书的二分法:

> **局部缓存(线程/CPU 私有,无锁 fast path,要"快") vs 中心堆(全局/按核共享,slow path,管页/合并/归还,要"省")。**

- **局部这一面(fast path)**:线程本地缓存(tcache / thread_cache / mimalloc 的 heap-local free list)、CPU 本地 slab(tcmalloc 新版的 per-CPU cache)、自由链表的 O(1) pop/push。这些都要"快":无锁、不争用、纳秒级。
- **中心这一面(slow path)**:中心自由链表(central free list / arena bins)、页堆(page allocator / extent / segment)、大块 `mmap`、合并(coalesce)、`madvise` 归还、huge page filler。这些都要"省"和"稳":批量、低碎片、及时还给系统。
- **衔接处**:transfer cache / cache miss 回填(局部 ↔ 中心的批量流转)、`madvise` 决策(省到什么程度才还)。每次 `malloc` 都是"先走 fast path,miss 了走 slow path"的降级旅程。

四套分配器每个机制都落在这两面之一(或衔接处)。任何时候迷路,回到这个二分法。

> 注:profiling / fork / 初始化等工程化章节不直接服务"快"或"省",但它们服务于"在快和省的前提下,还能观测、能 fork、能起步"——可以视为这条主线的"基础设施"支线。

### 点睛比喻:工位 — 中转货架 — 总仓库(只点睛,不贯穿)

只在讲"三层快慢道整体心智"时点一下,核心对应:

| 仓库比喻 | 对应分配器概念 | 出现在 |
|------|----------------|--------|
| 工人手边的零件盒(每人一份,随用随拿,不跟人抢) | **线程/CPU 本地缓存**(tcache / per-CPU cache / heap-local) | 第 1~3 篇 |
| 零件按型号分格(不然每次都得翻) | **size class 分级** | 第 1 篇 |
| 车间中转货架(按型号补货给工位,工位退回也先放这) | **中心自由链表 + transfer cache** | 第 1~2 篇 |
| 总仓库(整箱向供应商进货,空了整箱退) | **页堆 / OS 接口**(`mmap`/`brk`) | 第 2 篇 |
| 超大件不走货架,直接让供应商整车送 | **大块直接 `mmap`**(huge/large) | 第 2 篇 |
| 仓库管理员把零散空位凑成整箱再退 | **合并 + `madvise` 归还 + huge page filler** | 第 4 篇 |

新概念默认直球,讲到三层心智或反直觉处才点这个比喻。**不得另起一套,不得每章出现。**

## 五、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线是一条旅程:**为什么 glibc malloc 不够用 → 三层快慢道的共通地基(size class、对齐、自由链表、线程缓存、中心链表)→ 页堆怎么批量向系统要内存 → 多核怎么不让锁成瓶颈 → 碎片怎么治理、内存怎么归还 → 工程化(初始化、fork、profiling)→ 安全调试与四套对照 → 收束**。
> 章号全局连续;命名 `P{篇}-{章}-标题.md`,开篇第 1 章 = `P0-01`,第 1 篇第 2 章 = `P1-02`,以此类推。**全书 21 章**,对标《Tokio》《LevelDB》。

### 第 0 篇 · 开篇:为什么自己写 malloc(定调,1 章)

- **第 1 章 · 第一性原理:为什么 glibc malloc 不够用**(`P0-01`):操作系统给的 `mmap`/`brk` 太粗(按页,4KB 起,每次 syscall 贵);glibc ptmalloc 的 arena 锁在高并发下成瓶颈、bins + consolidate 压不住碎片 → 三层快慢道的直觉(本地缓存 / 中心链表 / 页堆)→ "快 vs 省"的永恒张力 → 立起"局部缓存 vs 中心堆"二分法 → 四套分配器亮相定位。**【样本章节,见后】**

### 第 1 篇 · 共通地基:三层快慢道与 size class(5 章)

> 全书地基。四套分配器在这一层**高度同构**,把共通心智立起来,后面分野才看得清。

- **第 2 章 · size class:把任意大小凑整成分级**(`P1-02`)——为什么不按申请大小原样分配(每个 size 一条 free list 会爆炸);把大小归并成几十个**分级**(class),每个 class 一条链;内部碎片 vs 查表 O(1) 的权衡。四套对照:tcmalloc `size_classes.cc`/`sizemap.cc`(默认 ~88 个 class,新版 pow2 实验)、jemalloc `sc.c`/`sz.c`、mimalloc 的 size class、ptmalloc 的 fastbin/smallbin/largebin 区间。**技巧精解:size → class 的查表 + 位运算**(为什么 jemalloc 能用位运算从 size 算出 class index,不必查表)。
- **第 3 章 · 对齐与缓存行:size class 为什么这么对齐**(`P1-03`)——为什么 class 要对齐到 8/16/32…(硬件访存对齐、`max_align_t`)、对齐到缓存行(64 字节)防 false sharing、pow2 对齐的好处(掩码代替除法);为什么大块要页对齐。源码:tcmalloc `pages.h`/`alignment`、jemalloc `sz.c` 的 size 算法、mimalloc 对齐分配 `alloc-aligned.c`。**技巧精解:对齐到缓存行防 false sharing**(两个线程的 hot data 落同一缓存行会被反复invalidate)。
- **第 4 章 · 自由链表:用释放块本身串链**(`P1-04`)——释放掉的块怎么不额外花 metadata 就串成可分配链(把 `next` 指针**藏进释放块的头几个字节**);push/pop 都是 O(1);LIFO vs FIFO 对缓存与延迟的影响。四套对照:tcmalloc `central_freelist`/`thread_cache` 的 FreeList、jemalloc `cache_bin`、mimalloc page 的 free list、ptmalloc chunk 的 `fd`/`bk`。**技巧精解:无锁单线程 free list + 跨线程的原子 push**(tcache 本地无锁,回收给中心时 CAS)。
- **第 5 章 · 线程本地缓存:fast path 必须无锁**(`P1-05`)——为什么 fast path 必须无锁(每秒百万次 malloc,锁就是灾难);用 TLS(thread-local storage)给每个线程一个私有 cache;cache 满了/空了怎么办。四套对照:tcmalloc `thread_cache.cc`(经典)与 `cpu_cache.cc`(新版 per-CPU,详见第 12 章)、jemalloc `tcache.c`/`cache_bin.c`、mimalloc 的 thread-local heap、ptmalloc 的 tcache(glibc 2.26+ 才加,正好说明"老分配器没有这层"为什么慢)。**技巧精解:TLS 的几种实现**(GCC `__thread`/`thread_local`、jemalloc 的 `tsd_*`、key+dtor)与懒创建。
- **第 6 章 · 中心自由链表与 transfer cache:cache miss 后的批量取还**(`P1-06`)——本地缓存 miss 后去哪(中心自由链表,按 size class 组织,批量取一串补货);回收时批量归还;transfer cache 在线程间直接流转(一个线程归还的块,另一个线程直接拿走,不必退到页堆)。四套对照:tcmalloc `central_freelist.cc` + `transfer_cache.cc`(LFIFO/LIFO 策略)、jemalloc arena 的 `bin`/`cache_bin` 回填、mimalloc `segment`→`page` 的补充、ptmalloc fastbin→smallbin 的迁移。**技巧精解:transfer cache 的批量与争用控制**(一次搬一串平摊锁开销)。

### 第 2 篇 · 页堆:批量向系统要内存(3 章)

> slow path 的下半段:中心链表也没有时,向 OS 要页,按页/span/extent 管理。

- **第 7 章 · 页、span、extent:内存按页管理**(`P2-07`)——中心链表空了,去页堆要一整批连续页;怎么把页组织成可分配单位(tcmalloc 的 **span**——连续 N 页、jemalloc 的 **extent**、mimalloc 的 **segment**→**page**);ptmalloc 的 top chunk / brk 对照。源码:tcmalloc `span.cc`/`page_allocator.cc`、jemalloc `extent.c`/`large.c`/`pa.c`、mimalloc `segment.c`/`page.c`。**技巧精解:span 的"连续页 + 长度"建模**(一个 span 记住起始页和页数,切分/合并都 O(1))。
- **第 8 章 · pagemap 与 rtree:指针 → 所属 span 的 O(1) 反查**(`P2-08`)——`free(p)` 只给了指针,怎么 O(1) 知道 `p` 属于哪个 span/extent、是哪个 size class?两种解法:tcmalloc 的**放射状 pagemap**(用页号直接索引的多级数组)、jemalloc 的 **radix tree**(`rtree.c`/`emap.c`)。源码:tcmalloc `pagemap.cc`/`pagemap.h`、jemalloc `rtree.c` + `include/jemalloc/internal/rtree.h`、mimalloc `segment-map.c`。**技巧精解:放射数组 vs radix tree 的取舍**(放射数组 O(1) 但吃内存、radix tree 省内存但多级)。
- **第 9 章 · 大块分配:huge/large 的单独路径**(`P2-09`)——申请一大块(远超最大 size class)时,走 size class 链太浪费;直接 `mmap` 一整块、单独管理、释放时直接 `munmap`。四套对照:tcmalloc 的 huge 路径、jemalloc `large.c`、mimalloc 直接大 mmap、ptmalloc 的 mmap threshold(`DEFAULT_MMAP_THRESHOLD` ~128KB)。源码:tcmalloc `huge_allocator.cc`/`huge_cache.cc`、jemalloc `large.c`、mimalloc `arena.c`。**技巧精解:mmap threshold 的自适应**(ptmalloc 动态调整阈值,平衡 mmap 开销与碎片)。

### 第 3 篇 · 多核并发:不让锁成瓶颈(3 章)

> 全书性能核心,也是 tcmalloc 与 jemalloc **分野**之处。同样的"减少锁争用"问题,四套给出不同答案。

- **第 10 章 · 争用从哪来、三种解法**(`P3-10`)——锁争用的根源(多线程抢同一个 arena/bin 锁);三种解法总览:① 多 arena 把线程分流(jemalloc/ptmalloc)、② per-CPU 让锁只在核内(tcmalloc 新版)、③ fast path 彻底无锁(都做,靠 tcache)。ptmalloc 作 baseline:它的 main_arena + 动态开 thread arena 为什么仍不够。**本章是第 11、12 章的总纲,二分法落在"局部这一面"。**
- **第 11 章 · jemalloc 的 arena 与 bin**(`P3-11`)——默认开 `4 * ncpus` 个 arena,每个线程绑定一个(按 round-robin),把锁争用摊到多个 arena;每个 arena 内,每个 size class 一个 `bin`、一把锁;extent 操作另有锁分离。源码:`src/arena.c`、`arenas_management.c`、`bin.c`/`bin_info.c`。**技巧精解:arena 数量与线程绑定的权衡**(为什么是 4× 而不是 1× 或 ncpus×;线程迁移 arena 的代价)。对照 ptmalloc 的 arena(`main_arena` + malloc 时按需 `arena_get`)。
- **第 12 章 · tcmalloc 的 per-CPU cache**(`P3-12`)——**新版 tcmalloc 的灵魂**:不再按"线程"缓存,而是按"CPU 核"缓存(`cpu_cache.cc`),每个核一份 slab,线程被调度到哪核就用哪核的 cache。为什么(线程数 ≫ 核数时,per-thread cache 浪费内存且迁移贵;per-CPU 天然按物理核摊开,避免 false sharing)。靠 **rseq**(Linux restartable sequences)做到"被抢占时安全回退"。源码:`cpu_cache.cc`/`cpu_cache.h`、`tcmalloc/internal/` 的 rseq/percpu 支持、`thread_cache.cc`(legacy 对照)。**技巧精解:rseq + per-CPU slab**(restartable sequences 怎么让无锁 per-CPU 操作在被抢占时安全;为什么这是 tcmalloc 新版相对 jemalloc 的代差)。

### 第 4 篇 · 碎片治理与内存归还(3 章)

> "省"这一面。怎么把碎片压低、把空闲内存及时还给 OS。新一代分配器(HPAA)的重头戏。

- **第 13 章 · 碎片的两个来源与合并**(`P4-13`)——碎片分两种:**内部碎片**(申请 17B 给了 32B 的 class,浪费 15B,源于 size class 凑整)和**外部碎片**(页堆里很多零散空闲页,凑不成一整块可用,源于释放/切分)。怎么压:外部碎片靠**合并**(coalesce,释放时和物理相邻的空闲 span/extent 合并成大的)。四套对照:ptmalloc `unlink` + `consolidate`、jemalloc extent 合并(`ehooks`)、tcmalloc span 合并、mimalloc segment 内 page 合并。**技巧精解:合并的 O(1) 邻居查找**(靠 pagemap/rtree 找到前后邻居,合并成大 span)。
- **第 14 章 · 归还 OS:madvise 与 decay purge**(`P4-14`)——空闲内存攒够了,什么时候、用什么方式还给 OS(不还 → RSS 居高不下;乱还 → 反复缺页抖动)。两个 syscall 关键:`madvise(MADV_DONTNEED)`(立刻清零还页)vs `MADV_FREE`(惰性还,内核缺页时才回收)。四套对照:tcmalloc `page_allocator` 的归还、jemalloc `decay.c` + `background_thread.c`(基于时间衰减的 purge)、mimalloc `arena-abandon.c`(整 arena 抛弃)、ptmalloc `trim`/`MADV_DONTNEED`。**技巧精解:`MADV_DONTNEED` vs `MADV_FREE` 的取舍**(延迟、RSS、信号量)与 decay 模型(jemalloc 用时间常数控制 purge 节奏)。
- **第 15 章 · huge page aware allocator(重头戏)**(`P4-15`)——**新一代分配器治碎片的杀手锏**:用 2MB 大页(huge page)为单位管理,降低 TLB miss、把碎片压缩在大页内。tcmalloc 的 **HPAA**(`huge_page_aware_allocator.cc` + `huge_page_filler.h`,把零散对象打包进大页的 filler)和 jemalloc 的 **hpa**(`hpa.c`/`hpa_central.c`/`hpdata.c`)。源码:tcmalloc `huge_page_aware_allocator.cc`/`huge_page_filler.h`/`huge_page_subrelease.h`、jemalloc `hpa.c`/`hpdata.c`/`psset.c`。**技巧精解:huge page filler 的"碎片打包" + subrelease**(怎么把存活对象挤进少数大页、把整页腾出来归还;donated/subreleased 大页的生命周期)。

### 第 5 篇 · 工程化:初始化、线程、fork、观测(3 章)

> 支撑前面一切的工程基建。让分配器在真实程序里"能起步、能 fork、能看见"。

- **第 16 章 · 初始化、TLS 与懒创建**(`P5-16`)——第一次 `malloc` 时分配器自己还没初始化,怎么自举(静态初始化、bootstrap allocator);线程的 tcache/heap 怎么在第一次使用时**懒创建**、线程退出时怎么清理(key + destructor)。四套对照:tcmalloc `static_vars.cc`(静态全局)与 `thread_cache` 懒创建、jemalloc `jemalloc_init.c` + `tsd.c`(`tsd_*` 一套)、mimalloc `init.c` + thread-local heap。**技巧精解:bootstrap——分配器启动前怎么分配自己的内存**(鸡生蛋问题:用静态 buffer / 早期 mmap)。
- **第 17 章 · fork 与多线程**(`P5-17`)——`fork()` 后只保留一个线程,但锁可能停在任意状态(持有锁的线程没复制过来)→ 死锁。分配器怎么处理:在 fork 前后加 hook,`prefork` 抢占所有锁、`postfork` 重置。四套对照:jemalloc `jemalloc_fork.c`(`prefork`/`postfork` 锁全部 arena)、tcmalloc fork handler、ptmalloc `arena_atfork`、mimalloc fork 处理。**技巧精解:fork 锁的处理**(为什么必须在 prefork 拿全部锁、postfork 释放,否则子进程永久死锁)。
- **第 18 章 · 采样、profiling 与统计**(`P5-18`)——怎么在**不拖慢 fast path** 的前提下统计内存使用、抓泄漏。核心是**采样**:不是每次都记账,而是按概率抽。四套对照:tcmalloc `sampler.cc`/`allocation_sampling.cc`(几何采样)、jemalloc `prof.c`/`prof_data.c`(采样 + 调用栈)、mimalloc `stats.c`(统计)。**技巧精解:几何采样**(每次分配以 `1/2^k` 概率被采,大块更容易被采到,用一次随机数预算采多少——`NextGate`/`PickNextSampling`),凭什么不影响 fast path。

### 第 6 篇 · 安全、调试与对照(2 章)

- **第 19 章 · 调试分配器:guarded page、sanitizer、随机化**(`P6-19`)——release 模式下怎么抓内存错误(use-after-free、堆溢出):tcmalloc 的 **guarded page allocator**(GWP-ASAN 风格,`guarded_page_allocator.cc`,概率给某些分配加守卫页)、jemalloc 的 sanitizer 集成(`san.c`/`san_bump.c`)、mimalloc 的随机化布局(`random.c`)与 guard pages。源码:tcmalloc `guarded_page_allocator.cc`、jemalloc `san.c`、mimalloc `random.c`/`segment.c` 的随机化。**技巧精解:guarded page 抓堆溢出/UAF**(在分配前后夹一个不可写页,越界/释放后访问立刻 SIGSEGV;为什么只概率开启——代价是内存)。
- **第 20 章 · 四套对照总表与演进趋势**(`P6-20`)——把四套分配器做一张**全维度对照表**(架构层次、size class 策略、并发模型、fast path 实现、归还策略、大页支持、profiling、调试特性),让读者一眼看清取舍;再讲演进趋势:per-CPU 取代 per-thread、arena-less 设计、huge page aware 成标配、meshing/紧凑化等前沿。本章是"全程对比"的收束,大量用表格,少上新源码。

### 第 7 篇 · 收束(1 章)

- **第 21 章 · 全景脉络与权衡哲学**(`P7-21`)——把全书收束成几条哲学:① 三层快慢道(fast path 无锁、slow path 批量)是分配器的通用骨架;② 用 size class 换 O(1)、用局部缓存换无锁;③ 快与省是永恒张力,碎片治理和归还是"省"的战场;④ 大页(per-CPU、HPAA)是当代代差;⑤ 所有技巧都服务于"纳秒级 fast path + 不泄漏不碎片"。回扣"局部缓存 vs 中心堆"二分法,把 21 章串成一条线。

### 附录

- **A · 全景脉络**:把全书收束成几条贯穿哲学(三层快慢道、无锁 fast path、size class 换 O(1)、局部缓存 vs 全局回收的张力、用大页治碎片)。
- **B · 源码阅读路线与实战**:tcmalloc / jemalloc / mimalloc / ptmalloc 四套的源码阅读地图;实战:`LD_PRELOAD` 换分配器、`MALLOC_CONF`(jemalloc)/`TCMALLOC_*`(tcmalloc)调参、用 `jemalloc` 的 `prof` 和 tcmalloc 的采样抓内存问题。

## 八、C/C++ 技巧侧重(对标 Tokio 的 Rust `unsafe`/原子/`Pin`、LevelDB 的 C++ 技巧)

本书要重点拆的系统级技巧(写作时遇到这些,务必钻"怎么做到的 + 为什么 sound"):

- **无锁与原子**:自由链表的 `next` 指针藏在释放块里;tcache 本地操作无锁,回收给中心用 CAS;`acquire`/`release` 内存序凭什么让"发布一个新挂载的空闲块"对别的线程可见而不数据竞争。
- **per-CPU 与 rseq**:`cpu_cache` 用 Linux **restartable sequences**(rseq)做 per-CPU 无锁操作,被抢占时安全回退——这是 tcmalloc 新版相对 jemalloc 的代差,重点拆"为什么 sound"。
- **TLS / tsd**:`__thread`/`thread_local`、jemalloc 的 `tsd_*`(key + destructor)几种实现,懒创建与线程退出清理。
- **位运算**:size → class index 的位运算映射(jemalloc `sz.c`)、bitmap(`bitmap.c`)、size class 的 pow2 编码。
- **数据结构**:放射状 pagemap(tcmalloc,多级数组 O(1))vs radix tree(jemalloc `rtree`,省内存多级);span/extent 的"连续页 + 长度"建模;huge page filler 的碎片打包。
- **系统调用与页**:`mmap`/`munmap`/`brk`、`madvise(MADV_DONTNEED)` vs `MADV_FREE`、huge pages(`MADV_HUGEPAGE` / transparent huge pages)、缺页与 TLB。
- **缓存行对齐**:`alignas(64)` / `CACHELINE_SIZE` 把 hot data 隔离,防 false sharing(tcmalloc/jemalloc 都大量使用)。
- **生命周期与 fork**:arena/extent 的生命周期;fork 的 `prefork`/`postfork` 锁处理,为什么不在 prefork 拿全部锁就会子进程死锁。
- **采样统计**:几何采样怎么用一次随机数预算采样间隔,凭什么不影响 fast path。
