# 第二十章 · 四套对照总表与演进趋势

> 篇:P6 · 安全、调试与对照
> 主线呼应:前 19 章我们沿着"一次 `malloc` 的一生"走完了三层快慢道的每一个驿站——从 size class、对齐、自由链表、线程/CPU 缓存,到中心链表、页堆、大块、多核并发、合并、归还 OS、大页,再到初始化、fork、profiling、调试。每一站都把四套分配器(tcmalloc / jemalloc / mimalloc / ptmalloc)并排对照过。但那些对照是**按驿站展开**的:你在 P3-12 看到 per-CPU、在 P4-15 看到 HPAA,却很难一眼看清"四套在所有维度上的全貌取舍"。这一章是**横评的收束**:把散落各章的对照表汇总成几张**全维度总表**,让读者一眼看清四套在每个维度上的真实设计选择,再从这张总表里读出**当代分配器的演进趋势**——per-CPU 取代 per-thread、huge page aware 成标配、arena 被淡化、随机化安全、meshing 前沿。读完这一章,前面 19 章的细节会被压缩成一张可检索的地图;读完这本书,你会知道"下一次选分配器,该看哪些维度"。

## 核心问题

**四套分配器在十几个维度上到底各自怎么选?把所有取舍汇总成一张总表后,能不能看出当代内存分配器在往哪个方向演进?**

读完本章你会明白:

1. **全维度对照**:架构层次、size class、对齐、free list、缓存模型、中心层、页堆单位、指针反查、大块、并发、合并、归还、大页、初始化、fork、profiling、调试——17 个维度上四套的真实选择,一格一格对清,每个结论都回指前面已写章节。
2. **取舍地图**:同一组问题(怎么快、怎么省、怎么并发),四套给出了不同答案;对照之下,每个设计的"为什么"才真正显形——为什么 tcmalloc 选 per-CPU、为什么 jemalloc 守 arena、为什么 mimalloc 不要 arena。
3. **五条演进趋势**:per-CPU 取代 per-thread(代差)、huge page aware 成标配(治碎片杀手锏)、arena 被淡化(简化架构)、随机化安全(防攻击兼抓腐败)、meshing 紧凑化(前沿研究方向)。
4. **代差在哪里**:tcmalloc 新版(per-CPU + HPAA)相对 ptmalloc/jemalloc 的代差,落在并发模型和大页两条线上;jemalloc 的工程成熟度(多 arena + witness + decay)是另一极;mimalloc 用"简化 + 随机化"走出第三条路。

> **如果一读觉得太难**:先只记住三件事——① 四套的核心代差在**并发模型**(per-CPU vs 多 arena vs thread-local)和**大页**(HPAA/hpa vs 无)两条线上;② 演进方向是"per-CPU + 大页 + 简化(arena-less)+ 随机化";③ ptmalloc 是 baseline 反衬,它的每一道墙都被新一代分配器补上。

## 20.1 一句话点破

> **四套分配器解决的是同一组问题——怎么快、怎么省、怎么并发、怎么归还——但它们在每一个维度上给出了不同答案。把这些答案汇总成一张总表,你会看到:ptmalloc 是"多 arena + bins + 被动合并"的经典朴素派;tcmalloc 新版是"per-CPU + HPAA 大页"的激进派;jemalloc 是"多 arena + bin 锁细分 + decay + hpa"的工程成熟派;mimalloc 是"thread-local heap + 随机化"的简化派。四派的取舍,折射出当代分配器正在往"per-CPU、大页、arena 淡化、随机化"演进。**

这是结论,不是理由。本章不重拆任何机制(那 19 章已拆透),只做两件事:**把对照表汇总成总表**,让前面散落的对照一目了然;**从总表里读出演进趋势**,把四套的差异收束成"当代方向"。

## 20.2 为什么需要一张总表

前 19 章每章都有一张局部对照表,但它们是**按驿站展开**的——P1-02 对照四套的 size class、P3-12 对照四套的并发模型、P4-15 对照四套的大页策略。问题在于:读者很难从这些分散的表里,**一眼**看清"四套在全维度上的取舍地图"。

打个比方(这是本书少数几次点比喻):这像看一份城市的地铁图——每一站(每一章)你都看清了周边设施,但要把整座城市的全貌看清,得把所有站拼到一张大图上。本章就是这张大图。

> **不这样会怎样**:如果只看局部对照表,你会记住"tcmalloc 用 per-CPU""jemalloc 用多 arena",但很难回答"那在 size class 上 tcmalloc 和 jemalloc 谁更细?在归还 OS 上四套谁更激进?在调试上 mimalloc 和 tcmalloc 谁更主动?"——这些问题需要**全维度横切**才能答。总表就是为了让这种横切成为可能。

总表的纪律:**每个格子都基于前面已写章节的真实结论,不新编**。下面四张大表,每个格子都标了出处(见 P3-12 等),你可以随时回查。本章不引入任何新源码机制,只做收束。

## 20.3 总表一:架构与策略全景

这是最重要的一张表,把四套在**架构层次与核心策略**上的选择一次性摆清。读这张表时,横着看是"四套在某个维度的差异",竖着看是"一套在所有维度的取舍全貌"。

| 维度 | tcmalloc(新版) | jemalloc | mimalloc | ptmalloc(baseline) |
|------|-----------------|----------|----------|---------------------|
| **本地缓存(fast path)** | per-CPU `CpuCache`(新版默认)/ `ThreadCache`(legacy 退化) | per-thread `tcache`(`cache_bin`) | per-thread `mi_heap_t`(thread-local) | per-thread `tcache`(glibc 2.26+ 才补) |
| **中心自由链表** | `CentralFreeList`(每 size class 一把 `SpinLock`) | arena 的 `bin`(每 arena × 每 size class 一把锁) | segment 里的 `page` + delayed free 原子链 | arena 的 `fastbin`/`smallbin`/`largebin` |
| **中转流转层** | **transfer cache**(栈式 LIFO + sharded,显式独立层) | `cache_bin` 的 fill/flush(无独立中转层) | delayed free list(跨线程原子累积) | 无(退直接进 fastbin) |
| **页堆单位** | `Span`(连续 N 页,8KB 默认)+ HugeRange(2MB) | `edata_t`(extent,连续页)+ 分层 dirty/muzzy/retained | `segment`(32MB)切 `slices[]`(64KB) | chunk(自带 header)+ top chunk |
| **指针反查** | 放射状多级数组 **pagemap**(`PageMap2`/`PageMap3`) | **radix tree**(`rtree`)+ emap + 线程本地三级缓存 | 强制 32MiB 对齐 + 位掩码(零元数据) | 每块自带 header(boundary tag,16B/块) |
| **大块路径** | size class=0 的 Span(HPAA 时代废弃独立 huge path) | 独立 extent(`slab=false`)挂 arena large list | 三级:small/large/huge(huge 独占 segment) | 直接 `mmap`(`mmap_threshold` 自适应,封顶 32MB) |
| **并发模型** | **per-CPU(rseq)+ 中心 SpinLock,无 arena** | **多 arena**(默认 `4×ncpu`)+ bin 锁细分 + witness | thread-local heap + delayed free,**无 arena** | 多 arena(上限 `8×ncpu`)+ 每 arena 一把粗 `mutex` |
| **合并策略** | per-hugepage 整合(filler 主动)+ pagemap 查邻居 | 释放即贪婪 `extent_try_coalesce` + rtree 查邻居 | segment 内 slice 数组算术合并 | 被动 `malloc_consolidate`(攒波触发) |
| **归还 OS** | `MADV_DONTNEED`(默认)+ 后台线程 + subrelease | **decay 两段式**:dirty→muzzy 用 `MADV_FREE`,muzzy→retained 用 `MADV_DONTNEED` | `MADV_DONTNEED` + `purge_delay`(固定 10ms) | `MADV_DONTNEED` + 静态 `M_TRIM_THRESHOLD` |
| **大页支持** | **HPAA**(默认页堆,filler 四级优先级打包) | **hpa**(experimental PAI,按需 hugify) | 弱(只支持 1GB huge OS page 预留) | 无显式感知(靠内核 THP) |
| **初始化** | 静态 `Static` 类 + `Arena` 元数据池 + double-checked locking | **五态状态机** + 独立 base allocator(`b0`) | 两阶段(process/thread)+ 静态 `_mi_heap_main`(主线程零分配) | 静态 `main_arena` + `__malloc_initialized` |
| **fork 处理** | **自己零代码**(下沉 Abseil SpinLock 的 atfork handler) | 自己注册 `pthread_atfork`,**9 阶段**按 rank 拿所有锁,child 用 `mutex_init` 重建 | **完全不处理**(赌单线程 + COW,osx 空函数) | 硬编码进 `__libc_fork`(≥2.24,修 BZ#19431) |
| **profiling/采样** | 几何采样(扣减式,默认 512KB 间隔)+ pprof 调用栈 | 几何采样(累加式,默认 512KB)+ pprof 调用栈 | **不做采样**(只粗粒度累加,无调用栈) | **不做采样**(只 `mallinfo2`/`malloc_stats` 外部查询) |
| **调试/安全** | **GWP-ASAN**(512 slot 固定池 + 概率守卫页 + magic byte) | guard page + junk(`0x5b5b...`)+ quarantine(双保险) | **ChaCha20 PRNG + free list 双 key 编码 + segment 守卫页**(安全调试合一) | Safe-Linking(2.32+,地址派生 key,强度低) |

> 出处:架构层次见 P1-06;size class 见 P1-02;对齐见 P1-03;free list 见 P1-04;缓存模型见 P1-05/P3-12;中心层见 P1-06;页堆单位见 P2-07;指针反查见 P2-08;大块见 P2-09;并发见 P3-10/P3-11/P3-12;合并见 P4-13;归还见 P4-14;大页见 P4-15;初始化见 P5-16;fork 见 P5-17;profiling 见 P5-18;调试见 P6-19。

## 20.4 总表二:size class 与对齐的精细对照

size class 是分配器的"地基的地基"(P1-02、P1-03 已拆透),四套在这里的差异最能体现各自的设计哲学。

| 维度 | tcmalloc | jemalloc | mimalloc | ptmalloc |
|------|----------|----------|----------|----------|
| **表怎么生成** | 编译期 `static constexpr` 数组(`size_classes.cc`) | 运行时 `sc_boot` 计算,由几个常量推(`sc.c`) | 运行时按位运算推导(`mi_bin` 纯算法,无表) | 编译期宏(`fastbin`/`smallbin`/`largebin_index_64`) |
| **class 数量** | 默认 **46 个** base class(8KB 页) | 几十~上百(可配) | **73 个 bin**(`MI_BIN_HUGE`) | **128 个 bin**(NSMALLBINS=64 + largebin) |
| **小对象步长** | 8/16 字节 | tiny(2/4/8B)+ quantum(16B 起) | 8 字节(word)起 | **固定 16 字节**(`SMALLBIN_WIDTH`) |
| **大对象步长** | 按 2 倍近似递增 | 按 group(NGROUP=4)递增,delta 每 pow2 翻倍 | ~12.5% 递增 | 多段位运算(`>>6`/`>>9`/`>>12`/`>>15`/`>>18`) |
| **小/大分界** | `kLargeSize = 1024` | `SC_SMALL_MAXCLASS` | `MI_SMALL_SIZE_MAX = 1024` | `MIN_LARGE_SIZE = 1024`(64-bit) |
| **size→class 计算** | 扁平数组查表(377 项,两次位移) | **纯位运算**(大对象,`lg_floor`+分组,零查表)+ 小查表 | 查表(`pages_free_direct[]`,129 项)+ 纯算法 `mi_bin` | 三段手工位运算(嵌套三元 + 位移,5 段魔数) |
| **最小对外对齐** | `kAlignment=8` | `QUANTUM=16`(x86-64) | `MI_MAX_ALIGN_SIZE=16` | `MALLOC_ALIGNMENT=16`(x86-64) |
| **缓存行隔离** | `ABSL_CACHELINE_ALIGNED`(Span/SizeMap/CpuCache) | `JEMALLOC_ALIGNED(CACHELINE)`(sz 表/arena bins) | `mi_decl_cache_align`(arena 全局原子/stats) | 无明显策略 |
| **页大小** | 默认 **8KB**(可配 4/32/256KB) | configure 检测,典型 **4KB** | slice **64KB**、segment **32MB** | **4KB**(运行时) |
| **fastbin 上限** | — | — | — | **160 字节**(`MAX_FAST_SIZE`),中等对象享受不到无锁 |

> 出处见 P1-02、P1-03。

> **钉死这件事**:四套 size class 表从粗到细大致是:tcmalloc 默认(46)< mimalloc(73)< jemalloc(可配,几十~上百)< ptmalloc(128)。一个反直觉的结论:**ptmalloc 的 128 个 bin 最多,恰恰是因为它没有本地缓存**——只能靠更细的 bins 在 arena 层减少锁争用,这是"用分级补偿并发"的典型。反过来 tcmalloc class 数少,是因为它有 per-CPU 缓存兜底,不需要在 size class 上做文章。**每一个数字背后都是一整套取舍**。

对齐上,tcmalloc 简洁(8/128 二段式)、jemalloc 精巧(分组 delta + 纯位运算)、mimalloc 实用(page 起点按 block 微调防 cache aliasing)、ptmalloc 朴素(`2*SIZE_SZ` 起 + 内核 THP)。四套都做缓存行对齐防 false sharing,但只有 tcmalloc/jemalloc 把它系统化(用宏标注每个 hot 结构)。

## 20.5 总表三:并发模型——全书性能核心的对照

并发模型是 tcmalloc 与 jemalloc **分野**之处(P3-10/P3-11/P3-12 已拆透),也是四套代差最明显的地方。这张表值得单独抽出来细看。

| 维度 | tcmalloc | jemalloc | mimalloc | ptmalloc |
|------|----------|----------|----------|----------|
| **arena 机制** | **无 arena**(中心层按 size class 分锁) | **多 arena**,默认 `4×ncpu`,least-loaded 绑定 | **无 arena**(thread-local heap) | **多 arena**,上限 `8×ncpu`,按需懒创建 |
| **小对象中心锁粒度** | 每 size class 一个 `CentralFreeList` + `SpinLock` | 每 arena × 每 size class 一个 `bin->lock`,可 shard | 无中心锁(heap 私有,跨线程走 delayed free) | **每 arena 一把 `mutex`**(粒度最粗) |
| **线程绑定对象** | per-CPU cache(随调度变) | arena(绑死在 tsd,first_null + least-loaded) | thread-local heap(每线程一个) | arena(TLS `thread_arena`) |
| **fast path 无锁机制** | **rseq 临界区**(每核独占 + 原子性,被抢占安全回退) | TLS 独占(本线程私有) | TLS 独占(本线程私有) | TLS 独占(本线程私有) |
| **大块/extent 锁** | page_allocator 内部锁 | `large_mtx` + `ecache->mtx` 分离 | segment/arena 锁 | arena mutex(和分配共享) |
| **锁序防护** | `SpinLockHolder` RAII | **witness**(rank-based 锁序检查,debug build) | 无显式锁序机制 | 无显式锁序机制 |
| **热点 size class 额外分流** | 中心层天然按 size class 分(无热点放大) | **bin shard**(`MALLOC_CONF=bin_shards`) | 无 | 无 |
| **per-CPU 能力** | **是**(rseq,核心创新,Linux 专属) | 实验性(`opt_percpu_arena`,默认关) | 否 | 否 |
| **平台依赖** | Linux rseq(否则退化 per-thread) | 纯用户态,跨平台 | 纯用户态,跨平台 | 纯用户态,跨平台 |
| **线程数 ≫ 核数时** | **最优**(per-CPU 只占核数份缓存) | 较好(arena 数固定,锁争用随线程数升) | 较好(heap 数 = 线程数,占用高但无锁) | **最差**(arena 上限 `8×ncpu`,大量线程挤少量 arena) |

> 出处见 P3-10、P3-11、P3-12。

> **钉死这件事**:并发模型上,四套代表了**三种解法的不同侧重**(P3-10 立的三解法:多 arena 治密度、per-CPU 治跨核、无锁 fast path 治频率):
> - **tcmalloc = per-CPU + 中心 SpinLock(无 arena)**:fast path 用 rseq 彻底治跨核争用,中心锁用批量摊薄,不需要 arena 这一层。这是当代方向。
> - **jemalloc = 多 arena + bin 锁细分 + witness**:用 arena 数量摊开争用,再在每个 arena 内按 size class 细分锁,辅以 witness 防锁序错乱。这是工程成熟派。
> - **ptmalloc = 多 arena + 粗 mutex(baseline)**:arena 数量上限 `8×ncpu`,但每 arena 只有一把粗 mutex,粒度最粗。这是被新一代分配器补的墙。
> - **mimalloc = thread-local heap + delayed free(无 arena)**:每个线程一个 heap,跨线程归还走原子链,根本不需要 arena。这是简化派。

三种解法**正交叠加**,只是侧重不同。tcmalloc 把宝全押在 per-CPU 上(代价是 Linux 专属);jemalloc 三解法都用但 per-CPU 默认关(代价是复杂度最高);mimalloc 用"thread-local + delayed free"绕开 arena 这一层(代价是线程数多时占用高);ptmalloc 三解法都用得最粗(代价是高并发争用)。

## 20.6 总表四:碎片治理与内存归还——"省"的战场

碎片治理和内存归还是"省"这一面的核心(P4-13/P4-14/P4-15 已拆透),也是新一代分配器相对 ptmalloc 最大的进步。

| 维度 | tcmalloc(HPAA) | jemalloc | mimalloc | ptmalloc |
|------|-----------------|----------|----------|----------|
| **外部碎片治理单位** | Span(连续页)+ huge page(2MB) | extent(连续页)+ 分层 dirty/muzzy/retained | segment 内 slice(定长 64KB) | chunk(带 header) |
| **邻居查找索引** | pagemap(3 级放射数组) | rtree(radix tree)+ emap | slice 数组指针算术 | chunk header 的 `prev_size` + 标志位 |
| **O(1) 查找实现** | 页号索引数组(3 次访问) | 地址分级下标(多级 radix) | `slice + slice_count`(纯算术) | 看 PREV_INUSE 位 + 指针算术 |
| **合并主动性** | **高**(filler 主动整合到 huge page) | **高**(释放即贪婪 `extent_try_coalesce`) | 高(释放即并,segment 内) | **低**(被动攒波,`malloc_consolidate`) |
| **合并粒度上限** | 2MB huge page(再大走 HugeCache) | 无硬上限(受 max_size 限制) | 4MB segment | 无硬上限(受 arena 边界) |
| **默认 madvise** | `MADV_DONTNEED`(默认) | dirty→muzzy:`MADV_FREE`;muzzy→retained:`MADV_DONTNEED` | `MADV_DONTNEED`(默认 decommit) | `MADV_DONTNEED`(子堆)/ `brk`(main) |
| **节奏控制** | `background_release_rate`(默认 0=关)+ 自适应 subrelease | **decay**(200 步 smoothstep,默认 dirty 10s / muzzy 0s) | `purge_delay`(默认 10ms 固定) | `M_TRIM_THRESHOLD`(静态阈值) |
| **触发主体** | 后台线程按字节/秒持续还 | 释放时推算 epoch + 后台线程定期 purge | segment 级定时检查 | `free` 末尾(主动)+ `malloc_trim`(手动) |
| **清零责任** | DONTNEED 保证清零 | muzzy 不保证,retained 保证 | DONTNEED 保证清零 | DONTNEED 保证清零 |
| **大页感知** | **显式,核心特性**(HPAA 是默认页堆) | 显式,可选(hpa 是 experimental PAI) | 弱(只支持 1GB huge OS page 预留) | 无显式感知(靠内核 THP) |
| **获取大页方式** | `mmap` + `madvise(MADV_HUGEPAGE)` | `madvise(MADV_HUGEPAGE)`(hugify) | `mmap(MAP_HUGETLB)` 预留 1GB 页 | 无(内核 khugepaged 自动) |
| **填大页策略** | filler 多级 TrackerList(`longest_free_range` + `nallocs`) | psset 按年龄选最年轻 pageslab | N/A(无 2MB 大页填机制) | N/A |
| **凑空归还** | 整页归还 + **subrelease**(半空大页 madvise 子页) | purge(合并相邻 dirty 段省 TLB shootdown) | arena-abandon(整 segment 抛弃) | systrim(4KB 粒度) |
| **大块零头处理** | **donated 标记 + abandoned 统计** | 无(大块走 extent,不做 donated) | N/A | N/A |

> 出处见 P4-13、P4-14、P4-15。

> **钉死这件事**:碎片治理上,四套的差距比并发模型更悬殊。tcmalloc 是"大页优先"派(HPAA 从底层按 2MB 组织,filler 把小对象挤进去,subrelease 处理半空大页);jemalloc 是"4KB 优先、按需 hugify"派(先按 4KB 管,满了再 `madvise(MADV_HUGEPAGE)` 合并,purge 时合并相邻 dirty 段省 TLB shootdown);mimalloc 对 2MB 大页支持弱(只支持 1GB huge OS page);ptmalloc 完全无显式大页感知(靠内核 THP 碰运气)。归还节奏上,jemalloc 是唯一默认走 `MADV_FREE` 的(两段式 dirty→muzzy 用 FREE 抗抖,muzzy→retained 用 DONTNEED 降 RSS);tcmalloc/mimalloc 默认 DONTNEED;ptmalloc 静态阈值最朴素。**这一张表,就是 ptmalloc 在长期运行服务里 RSS 居高不下的根因**。

## 20.7 总表五:工程化与可观测性

初始化、fork、profiling、调试这些"支线"(P5-16/P5-17/P5-18/P6-19 已拆透)不直接服务"快"或"省",但决定了一个分配器能不能在真实程序里"能起步、能 fork、能看见、能抓 bug"。

| 维度 | tcmalloc | jemalloc | mimalloc | ptmalloc |
|------|----------|----------|----------|----------|
| **全局状态存储** | `Static` 类的 `ABSL_CONST_INIT` 静态成员 | 全局 `malloc_init_state` + `b0` 指针 | 静态 `_mi_heap_main` + `tld_main` | 静态 `main_arena` + 全局 `__malloc_initialized` |
| **bootstrap 内存来源** | 静态 buffer + `Arena` 元数据池 | **base allocator(`b0`)**,直接 `mmap` | **`_mi_os_zalloc` 直取 OS**(主线程零分配) | 复用 `main_arena` 自带的静态结构 |
| **初始化阶段** | 单阶段(`SlowInit` 一次性做完) | **五态状态机**(uninitialized → a0 → recursible → initialized) | 两阶段(process init / thread init) | 单阶段 |
| **重入安全** | `Arena` 元数据池自包含 | 显式 `recursible` 阶段 + tsd 重入计数 | `mi_atomic_do_once` 天然防递归 | 依赖 libc 启动顺序,未显式处理 |
| **fork 注册方式** | **不注册**(下沉 Abseil SpinLock) | 自己注册 `pthread_atfork` | **不注册**(osx 空函数) | 硬编码进 `__libc_fork`(≥2.24) |
| **prefork 拿锁** | Abseil SpinLock 自带 prepare handler | 分 **9 阶段**按 rank 拿所有 arena 的所有锁 | 不拿(osx 空函数) | 拿 `list_lock` + 遍历 arena 链逐个拿 mutex |
| **child hook** | Abseil SpinLock 自带 child handler(reset 成 unlocked) | **`malloc_mutex_init` 重建**每把锁(不是 unlock!) | 不做(osx 空函数) | **`__libc_lock_init` 重建**每把锁 |
| **fork 死锁保证** | 强(Abseil 全收、child reset) | 强(prefork 全收、child 重建) | 强(同骨架) | **弱**(赌单线程 + COW) |
| **是否做采样 profiling** | 是,几何采样(扣减式) | 是,几何采样(累加式,`config_prof` 编译开关) | **否**(只粗粒度累加) | **否**(只外部查询) |
| **默认采样间隔** | 512KB(`profile_sampling_interval`) | 512KB(`lg_prof_sample = 19`) | — | — |
| **fast path 采样代价** | 一条 `sub` + 溢出标志位 | 一次 `add` + 无符号比较 | 一次非原子 `+=` | 零(不动) |
| **调用栈维度** | 有(`StackTrace`,dump 出 pprof) | 有(`prof_bt_t`,dump 出 pprof) | **无** | **无**(`mtrace` 有,但全量) |
| **生产可长期开** | 是(fast path 几乎零开销) | 是(同) | 是(更轻,但无栈维度) | 否(`mtrace` 重,`mallinfo` 无栈) |
| **守卫页(单分配)** | **有,GWP-ASAN**(512 slot 固定池 + 概率采样 + quarantine) | 有,挂在 extent 上,采样间隔可配 | 无(只在 segment 级) | 无 |
| **UAF 检测** | 守卫页 `PROT_NONE` 立即 SEGV | **junk(`0x5b5b...`)+ quarantine** | free list 编码会暴露踩坏 | tcache key 检测 double-free(2.34+) |
| **溢出检测** | 守卫页抓大溢出 + **magic byte** 抓小溢出 | 守卫页 | segment 守卫页 + padding(`MI_PADDING`) | chunk 边界检查(有限) |
| **随机化** | slot 随机选取 | 池子内随机 | **ChaCha20 PRNG + free list key + 布局随机** | Safe-Linking 地址派生 key |
| **free list 指针保护** | (无,守卫池外不暴露) | (无显式编码) | `MI_ENCODE_FREELIST`:XOR + 旋转,每 page 双 key | Safe-Linking:地址高位 XOR(2.32+) |
| **抓 bug 哲学** | 概率守卫,撞进 PROT_NONE 立即崩 | 采样守卫 + junk 复查 | 随机化让攻击和腐败自暴露 | 主要靠外部 ASan |

> 出处见 P5-16、P5-17、P5-18、P6-19。

> **钉死这件事**:工程化上,四套的成熟度梯度很清晰。jemalloc 是"完整教科书实现"(五态状态机 + 9 阶段 fork + witness + decay + hpa),每一个工程难点都自己写代码解决;tcmalloc 是"借力派"(fork 下沉 Abseil、profiling 借 Google pprof 生态、GWP-ASAN 是对工业界最大贡献之一);mimalloc 是"简化 + 安全合一"(主线程零分配 + ChaCha20 随机化,代价是无调用栈 profiling、fork 不处理);ptmalloc 是"baseline"(依赖 libc 已初始化好 main_arena、`__libc_fork` 硬编码、Safe-Linking 是 2.32 才补的)。**ptmalloc 在工程化上的每一道墙,都被新一代分配器补上了**。

## 20.8 从总表读出五条演进趋势

五张大表摆完,四套的全貌已经清晰。现在我们从这张总表里读出**当代分配器的演进方向**。这五条趋势不是凭空总结,而是从总表的对照里**直接显形**的——每个趋势都能在表里找到四套的真实选择作证。

### 趋势一:per-CPU 取代 per-thread(代差,已专章拆透)

这是四套里**代差最明显**的趋势。看总表三的"并发模型"行:tcmalloc 用 per-CPU(新版默认),其余三套都还在用 per-thread(tcache / thread-local heap)。这条代差的根因在 P3-12 已拆透——当线程数远超核数时,per-thread cache 有三道墙:

- **墙一总占用爆炸**:per-thread = 线程数 × 囤货(可上百 MB);per-CPU = 核数 × 囤货(固定几十份)。
- **墙二迁移局部性丢失**:per-thread cache 跟线程走,迁核丢 cache;per-CPU slab 跟核走,迁核零代价。
- **墙三 false sharing**:两个线程的 ThreadCache 落同一缓存行会反复 invalidate;per-CPU 每核 slab 独占缓存行。

tcmalloc 新版用 [Linux rseq(restartable sequences)](../tcmalloc/tcmalloc/internal/percpu.cc#L86-L89) 做到"per-CPU 无锁操作在被抢占时安全回退"——这是它相对 jemalloc/ptmalloc/mimalloc 的代差命脉。源码佐证([tcmalloc.cc:1198-1200](../tcmalloc/tcmalloc/tcmalloc.cc#L1198-L1200)):fast path 是 `AllocateFast`(per-CPU slab 一次 Pop),miss 了走 `AllocateSlow`(去中心 transfer cache 批量补货),总开关是 `UsePerCpuCache`。

> **钉死这件事**:per-CPU 不是"换个缓存粒度"这么简单,它要解决"无锁 per-CPU 操作在被抢占时仍然 sound"这件极难的事(靠 rseq 的临界区中断重启)。这是 tcmalloc 新版敢把锁粒度推到"物理核"这一极限的底气。jemalloc 也有 `opt_percpu_arena` 但默认关(因为要跨平台,没法把 rseq 做成默认)——这是两条路在工程哲学上的根本分野。详见 P3-12 整章。

### 趋势二:huge page aware 成标配(治碎片杀手锏,已专章拆透)

看总表四的"大页支持"行:tcmalloc HPAA 是默认页堆,jemalloc hpa 是 experimental(默认部分启用),mimalloc 弱(只 1GB huge OS page),ptmalloc 无。这条趋势的根因在 P4-15 已拆透——用 2MB 大页为单位管理,降低 TLB miss、把碎片压缩在大页内。

两派的工程哲学分野很清晰:

- **tcmalloc 是"大页优先"派**:HPAA 从底层按 2MB 组织,filler([huge_page_filler.h:1100](../tcmalloc/tcmalloc/huge_page_filler.h#L1100))把小对象挤进大页,subrelease 处理半空大页。灵魂是 filler 的四级优先级选择(regular → donated → partial_released → released,见 [huge_page_filler.h:1685](../tcmalloc/tcmalloc/huge_page_filler.h#L1685)),把"碰谁的代价"显式编码。
- **jemalloc 是"4KB 优先、按需 hugify"派**:先按 4KB 管理,满了再 `madvise(MADV_HUGEPAGE)` 合并。独有的 purge 优化:合并相邻 dirty 段省 TLB shootdown([hpdata.c:340](../jemalloc/src/hpdata.c#L340),注释明说 "the expensive part of purging is the TLB shootdowns")。

> **钉死这件事**:大页感知是当代分配器治外部碎片的杀手锏。ptmalloc 完全无显式大页感知(靠内核 khugepaged 碰运气),这是它长期运行服务 RSS 居高不下的根因之一。tcmalloc 和 jemalloc 都把大页做成了核心特性,只是路径不同(tcmalloc 从底层重构,jemalloc 在 4KB 之上叠加)。详见 P4-15 整章。

### 趋势三:arena 被淡化(简化架构,需本章补总览)

这条趋势在前面章节**分散提及但没有专章收束**,这里补上。看总表三的"arena 机制"行:tcmalloc **无 arena**(中心层按 size class 分锁),mimalloc **无 arena**(thread-local heap),只有 jemalloc 和 ptmalloc 还在用多 arena。

三套的"arena 淡化"程度:

- **mimalloc 完全 arena-less**:每个线程一个 `mi_heap_t`([alloc.c:208](../mimalloc/src/alloc.c#L208) `mi_prim_get_default_heap()` 拿 thread-local heap),跨线程归还走 delayed free 原子链,根本不需要 arena 这一层。源码断言保护([alloc.c:186](../mimalloc/src/alloc.c#L186)):"heaps are thread local"——heap 必须线程私有。
- **tcmalloc 半 arena-less**:全局只有一套 `tc_globals`(单例),中心 transfer cache 按 size class 分片(`ShardedTransferCache`),没有 jemalloc 那种"4×ncpu 个独立 arena"。它的策略是:fast path 用 per-CPU 彻底治跨核争用,中心锁用批量摊薄,**不需要 arena 这一层**。
- **jemalloc 仍用多 arena,但 hpa 在向"按物理资源组织"靠拢**:jemalloc 因为要跨平台,没法像 tcmalloc 那样把 rseq 做成默认,所以仍守着多 arena。但它的 hpa(huge page aware)和 `opt_percpu_arena` 都在向"按物理资源(CPU 核 / 大页)组织"靠拢——这是 arena 概念正在被淡化的信号。

> **钉死这件事**:arena 是 ptmalloc 时代为"减少锁争用"发明的层次,但它在新一代分配器里正在被淡化——mimalloc 用 thread-local heap 绕开,tcmalloc 用 per-CPU 替代。为什么?因为 arena 是"按线程数摊开争用"的思路,而当代方向是"按物理核(CPU)组织"——per-CPU 天然按物理核摊开,不需要 arena 这一层抽象。arena 不是被"删除"了,而是被"更底层的物理资源组织"取代了。详见 P3-10、P3-11、P3-12 的对照小节。

### 趋势四:随机化安全(防攻击兼抓腐败,已专章拆透)

看总表五的"调试/安全"行:mimalloc 用 ChaCha20 PRNG + free list 双 key 编码 + segment 守卫页(安全和调试合一),tcmalloc 用 GWP-ASAN(概率守卫),jemalloc 用 guard page + junk 双保险,ptmalloc 用 Safe-Linking(强度最低)。

mimalloc 是这条趋势的代表,核心在 P6-19 第 19.5.2 节已拆透:

- **ChaCha20 密码学级 PRNG**([random.c:20](../mimalloc/src/random.c#L20),20 轮,注释明说"用密码学级 PRNG 不是为性能而是为安全")。
- **安全随机源 + 弱退化升级**:优先用 OS 安全随机源(`getrandom` / `arc4random_buf`,经 `_mi_prim_random_buf` 原语抽象,见 [random.c:181](../mimalloc/src/random.c#L181)),失败才退化到 weak([random.c:167](../mimalloc/src/random.c#L167) ASLR + 时钟),拿到真随机后再升级([random.c:211](../mimalloc/src/random.c#L211))。
- **free list 指针编码(XOR + 旋转,每 page 双 key)**:[internal.h:803](../mimalloc/include/mimalloc/internal.h#L803) `mi_rotl(x ^ keys[1], keys[0]) + keys[0]`,push/pop 都走 encode/decode。
- **segment 守卫页**([segment.c:955/958](../mimalloc/src/segment.c#L955),`MI_SECURE` 模式段头尾各一页)。

> **钉死这件事**:随机化安全是"防御性设计"的趋势——不是为了快、不是为了省,而是为了"在 release 模式下也能抓内存错误 + 防堆攻击"。mimalloc 把它做成了默认开启的轻量特性(密码学编码每次 ~5 条指令);tcmalloc 的 GWP-ASAN 是概率守卫(撞进 PROT_NONE 立即崩,是它对工业界最大的贡献之一);jemalloc 是 guard page + junk 双保险;ptmalloc 的 Safe-Linking 强度最低(地址派生 key,不是独立随机)。这条趋势折射出一个变化:**分配器不再只是性能组件,也是安全组件**。详见 P6-19 整章。

### 趋势五:meshing / 紧凑化(前沿研究方向,本书四套未直接实现)

前四条趋势本书四套都有实现,这条**没有**——meshing 是 dmmalloc/mesh(PLDI 2019,Bobby Powers et al.)的专属概念,本书四套都未直接实现。但它是当代分配器研究的前沿方向,值得用一两段介绍,并指出它与本书已讲机制的关系。

**meshing 的核心思想**:把两个半空的页通过随机化布局"缝合"(mesh)成一个逻辑上更满的页,治外部碎片。具体做法是——

1. 给每个 size class 的 free list 做**随机化布局**(对象在页内的位置随机),这样两个页的活跃对象大概率不重叠。
2. 当一个页只剩少量活跃对象时,把它的活跃对象**重映射**到另一个半空页的空隙里,腾出原来的物理页归还 OS。
3. 因为布局是随机化的,重映射后两个页的活跃对象不会撞(概率上)。

**与本书已讲机制的关系**:

- meshing 治的是**外部碎片**(P4-13 的主题),但走的是"随机化重映射"这条路,而不是 coalesce 的"物理相邻拼接"。coalesce 要求两个空闲块物理相邻才能合并;meshing 不要求相邻,靠随机化让"逻辑上拼满"成为可能。
- meshing 的随机化思想和 P6-19 mimalloc 的随机化是**同一思想家族**——mimalloc 的 free list 随机化([page.c](../mimalloc/src/page.c))和 segment 内 slice 随机选取,可以看作 meshing 的"半成品":它有随机化,但没有"缝合"那一步,所以只能防攻击、不能紧凑化。这是本书四套里离 meshing 最近的一块。

> **钉死这件事**:meshing 是"用随机化换紧凑"的极致——它放弃了"物理相邻才能合并"的约束,转而靠随机化让"逻辑上拼满"成为可能。本书四套都没走到这一步(coalesce 是物理相邻拼接,filler 是把小对象挤进大页但仍是物理连续),但 mimalloc 的随机化布局是朝这个方向迈出的第一步。如果未来某套主流分配器引入 meshing,你会在这本书里找到它所有前置概念(随机化 P6-19、外部碎片 P4-13、huge page filler P4-15)。

## 20.9 四套的"设计哲学"一句话定位

总表和趋势讲完,最后用一句话给四套的**设计哲学**定位——这是选分配器时最先该问的问题:

| 分配器 | 一句话设计哲学 | 最适合的场景 | 最不适合的场景 |
|--------|---------------|-------------|----------------|
| **tcmalloc** | "激进派":per-CPU + HPAA 大页,把锁粒度推到物理核极限,用 rseq 换无锁 | Linux 高并发服务(线程数 ≫ 核数)、对延迟敏感、能用 rseq | 非 Linux 平台、不能依赖 rseq |
| **jemalloc** | "成熟派":多 arena + bin 锁细分 + decay + hpa,每个工程难点都自己解决 | 跨平台、长期运行服务、需要精细调参(`MALLOC_CONF`)、需要调用栈 profiling | 极致低延迟(per-CPU 更优)、内存极度受限(arena 有开销) |
| **mimalloc** | "简化派":thread-local heap + 随机化,不要 arena,安全和调试合一 | 单线程或多线程但线程数适中、需要安全防攻击、追求代码简洁 | 线程数极多(heap 数 = 线程数,占用高)、需要调用栈 profiling |
| **ptmalloc** | "baseline":多 arena + bins + 被动合并,glibc 默认 | 默认场景(不想换分配器)、单线程或低并发、历史兼容 | 高并发(arena 锁争用)、长期运行(碎片压不住)、需要大页 |

> **钉死这件事**:选分配器不是选"最好的",而是选"最适合的"。tcmalloc 在 Linux 高并发上几乎无敌(per-CPU + HPAA),但离开 Linux 就退化;jemalloc 是"万能瑞士军刀"(跨平台 + 工程成熟),但复杂度最高;mimalloc 是"安全优先"的新秀(随机化 + 简化),但线程数多时占用高;ptmalloc 是 baseline,只在"懒得换"时用。这本书横评四套,不是要分高下,而是要让读者看清**每个设计在权衡什么**,从而根据自己的场景做选择。

## 20.10 技巧精解:从总表读"代差落在哪条线上"

这一章是对照章,没有新源码可拆。但我们用一个"技巧精解"小节,把全书最值钱的一个观察讲透:**四套的代差到底落在哪条线上?**

答案是两条:**并发模型(per-CPU vs 多 arena)**和**大页(HPAA/hpa vs 无)**。这两条线,是 tcmalloc 新版相对 ptmalloc/jemalloc 的真正代差所在。

**为什么是这两条,而不是别的?** 看总表的其他维度:size class 四套都做了(只是粗细不同)、free list 四套都用了侵入式链表(只是编码与否)、合并四套都做了(只是主动性不同)、归还 OS 四套都用了 `madvise`(只是节奏不同)——这些维度上,四套是**同构**的,差异是"程度"而非"质变"。但在**并发模型**和**大页**这两个维度上,四套是**质变**:

- **并发模型的质变**:从"per-thread + 多 arena"(ptmalloc/jemalloc/mimalloc)到"per-CPU + rseq"(tcmalloc)。这不是"换个缓存粒度",而是"换一种无锁的 sound 性基础"——rseq 让 per-CPU 操作在被抢占时安全回退,这是 per-thread 时代做不到的。
- **大页的质变**:从"靠内核 THP 碰运气"(ptmalloc/mimalloc)到"显式按 2MB 组织"(tcmalloc HPAA/jemalloc hpa)。这不是"换个页大小",而是"换一种碎片治理单位"——2MB 大页让 TLB miss 下降一个数量级,filler 把碎片压缩在大页内。

> **反面对比**:假设我们只优化 size class、free list、合并、归还这些"同构"维度,能不能追上 tcmalloc?不能。因为 ptmalloc 在这些维度上已经做得不差(128 个 bin、Safe-Linking、consolidate、`MADV_DONTNEED`),差距在"质变"那两条线上——per-CPU 和 HPAA。这就是为什么 tcmalloc 新版敢说自己"代差"——它的进步不在"做得更细",而在"换了一个基础"。

> **钉死这件事**:读完这本书,如果你只记住一件事,记住这个——**当代分配器的代差,不在 size class / free list / 合并这些"同构"维度上,而在 per-CPU(并发模型)和 HPAA(大页)这两条"质变"线上**。ptmalloc 的每一道墙,都被这两条线补上了;tcmalloc 新版的激进,jemalloc 的成熟,mimalloc 的简化,都是围绕这两条线在做不同的取舍。

## 章末小结

这一章是**横评的收束**,我们没有引入任何新机制,只做了两件事:把前 19 章散落的对照表汇总成**五张大总表**(架构全景、size class 与对齐、并发模型、碎片治理与归还、工程化与可观测性),让四套在 17 个维度上的取舍一目了然;再从总表里读出**五条演进趋势**(per-CPU 取代 per-thread、huge page aware 成标配、arena 被淡化、随机化安全、meshing 前沿)。

### 回扣二分法

本章服务的不是"局部缓存"或"中心堆"某一单面,而是**横跨两面**的总览。但总表清晰地印证了全书的二分法:

- **局部这一面(fast path)**:四套的代差落在并发模型上——per-CPU(tcmalloc)vs 多 arena(jemalloc/ptmalloc)vs thread-local heap(mimalloc)。fast path 的"快"靠的是无锁,而无锁的 sound 性基础(per-CPU rseq / TLS 独占)是代差命脉。
- **中心这一面(slow path)**:四套的代差落在大页和归还上——HPAA/hpa(tcmalloc/jemalloc)vs 无(ptmalloc/mimalloc),decay 两段式(jemalloc)vs 单段(tcmalloc/mimalloc)vs 静态阈值(ptmalloc)。slow path 的"省"靠的是碎片治理和归还节奏。
- **衔接处**:transfer cache(tcmalloc 显式独立层)vs cache_bin fill/flush(jemalloc 无独立层)vs delayed free(mimalloc)vs 无(ptmalloc),四套在"局部 ↔ 中心怎么批量流转"上也是分野。

任何时候回头看这五张表,都能立刻定位"四套在某个维度上的取舍",以及"这个取舍服务的是快还是省"。

### 五个"为什么"清单

1. **为什么需要一张总表,而不是每章局部对照就够了?** 局部对照是按驿站展开的,读者很难一眼看清"四套在所有维度上的全貌取舍"。总表是横切,让"四套 × 17 维度"的取舍地图一目了然,每个格子都回指前面章节可深挖。
2. **为什么 ptmalloc 的 size class 有 128 个 bin 最多,反而并发最差?** 因为它没有本地缓存(2.26 才补 tcache),只能靠更细的 bins 在 arena 层减少锁争用——这是"用分级补偿并发"。反过来 tcmalloc class 数少(46 个),是因为有 per-CPU 缓存兜底。**每个数字背后都是一整套取舍**。
3. **为什么 per-CPU 是代差,而 size class 细分不是?** 因为 per-CPU 是"换一种无锁的 sound 性基础"(rseq 让 per-CPU 操作被抢占时安全回退),是质变;size class 细分是"做得更细",是程度。代差落在质变线上,不落在程度上。
4. **为什么 arena 在新一代分配器里被淡化?** arena 是 ptmalloc 时代为"减少锁争用"发明的层次(按线程数摊开),但当代方向是"按物理核组织"(per-CPU)。per-CPU 天然按物理核摊开,不需要 arena 这一层抽象。mimalloc 用 thread-local heap 绕开,tcmalloc 用 per-CPU 替代。
5. **为什么 meshing 本书四套都没实现?** meshing 是"用随机化换紧凑"的极致(放弃物理相邻约束,靠随机化重映射缝合两个半空页),是前沿研究方向。本书四套的合并都是物理相邻拼接(coalesce),filler 是把小对象挤进大页但仍物理连续。mimalloc 的随机化布局是朝 meshing 迈出的第一步(有随机化但没缝合)。
