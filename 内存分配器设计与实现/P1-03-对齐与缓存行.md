# 第 3 章 · 对齐与缓存行:size class 为什么这么对齐

> 篇:P1 共通地基
> 主线呼应:这一章服务"**局部缓存**"这一面——fast path 想要纳秒级命中,要的不只是"无锁",还有"缓存友好"。一个 size class 的边界为什么偏偏落在 8、16、32、64 这些 2 的幂上?为什么 jemalloc 的 size class 越大、对齐步长也越大?为什么 tcmalloc 把 1024 以上的 class 强行对齐到 128?答案全部指向两个硬件真相:**CPU 访存必须对齐**(不对齐要么慢、要么崩),以及**缓存是以 64 字节的"缓存行"为单位搬运的**(两个线程各自频繁写的字段若落进同一行,会让这一行被反复 invalidate——这就是 false sharing,size class 的对齐策略有一半是为了躲它)。读完这一章,你就能解释为什么 `malloc` 返回的指针至少 16 字节对齐、为什么分配器内部到处是 `alignas(64)`,以及为什么大块必须按页对齐。

## 核心问题

**为什么 size class 的边界偏偏是 8、16、32、64…… 这些对齐数,而不是随便取个 13、27、53?为什么 1024 字节以上的 class 突然要按 128 对齐?为什么大块必须 4KB 起步、超大块甚至要 2MB 对齐?四套分配器在这件事上,各自怎么取舍?**

读完本章你会明白:

1. **硬件真相一**:CPU 访问内存有**对齐要求**——一个 8 字节的 `uint64_t` 必须放在 8 的倍数地址上,不对齐要么慢(x86 默默修好但要花两拍)、要么直接 trap(SPARC、某些 ARM 模式)。所以 `malloc` 返回的指针**至少**要对齐到 `max_align_t`(典型 16 字节),这是 ABI 红线。
2. **硬件真相二**:CPU 搬运内存以**缓存行**(cache line,典型 64 字节)为单位。两个线程各自频繁写的字段若挤在同一行,这一行会被反复 invalidate——**false sharing**,性能能塌方一个数量级。
3. **size class 对齐的三个动机**:① 满足硬件对齐(把块切成 8/16/32…,自然保证返回指针合格);② 凑 2 的幂好做**位运算对齐**(用掩码代替除法,纳秒级 fast path 不能容忍 `idiv`);③ 大块按页对齐,方便页堆以页为单位切分、还回 OS、上 huge page。
4. **四套横评**:tcmalloc 的"≤1024 对齐 8、>1024 对齐 128"二段式、jemalloc 的"quantum + 分组 delta"精细化、mimalloc 的"8 字节起、page 起点按 block size 微调"、ptmalloc 的"`2*SIZE_SZ` 起步"baseline——同一组硬件约束,四套答案。

> **如果一读觉得太难**:先只记住三件事——① `malloc` 返回的指针必须对齐(典型 16 字节),否则程序会崩或变慢,这是 ABI;② 缓存行 64 字节,两个线程的 hot data 不能挤一行(false sharing);③ 对齐到 2 的幂是为了用 `(x + mask) & ~mask` 代替除法。其余细节都是这 3 条的展开。

## 3.1 一句话点破

> **size class 的边界对齐到 8/16/32/64…… 不是审美,是物理:CPU 要求访存对齐(不对齐要么慢要么崩)、缓存按 64 字节的行搬运(false sharing 会塌方)、按 2 的幂对齐才能用掩码代替除法(纳秒级 fast path 用不起 `idiv`)。大块按 4KB 页对齐,是为了让页堆能整页切分、整页归还、整页上 huge page。对齐,是分配器在"硬件说了算"的约束下,把每个 size class 安放在地址空间里最不打架的位置。**

这是结论,不是理由。本章倒过来拆:先看"访存为什么必须对齐",再看"缓存行为什么是 64 字节、false sharing 为什么塌方",再看"按 2 的幂对齐为什么省掉除法",最后看四套分配器怎么把这三件事落到 size class 表里。

---

## 3.2 硬件真相一:CPU 访存必须对齐

先把最基础的事钉死:`malloc` 返回的指针为什么必须对齐?

答案在 CPU 这一层。CPU 不是"想读哪个字节就读哪个字节"的——它的 load/store 指令对**地址**有要求。一个 8 字节的 `uint64_t`,在 x86-64 上要从 8 的倍数地址读;一个 `__m256`(32 字节的 AVX 向量),通常要求 32 字节对齐;一个 `long double`(x86 上 10 字节但要求 16 字节对齐),必须落在 16 的倍数地址。

不对齐会怎样?分三种情况,从轻到重:

1. **x86(默许但慢)**:x86 的普通 load/store 允许非对齐访问(unaligned access),硬件会"默默修好"——跨缓存行的访问要拼两拍、跨页的访问还可能触发两次缺页。结果就是**慢**,但程序不会崩。一个 hot 循环里反复 unaligned 访问,实测能慢 30%~200%。
2. **ARM(很多模式直接 trap)**:ARMv7 之前、以及 ARMv8 的某些对齐检查模式下,非对齐访问会**直接抛 `SIGBUS`**。这就是为什么 Android NDK 的文档反复强调"别假设非对齐访问没事"。
3. **SPARC、MIPS(几乎必崩)**:这些架构传统上对非对齐访问零容忍,直接 SIGBUS。

> **钉死这件事**:C/C++ 的 ABI 规定,`malloc` 返回的指针必须满足**最严格的对齐要求**,让任何类型的对象都能放进去。这个"最严格"的标尺,在 C11 里有个名字叫 `max_align_t`,在 glibc/x86-64 上是 16 字节(`long double` 和 `__m128` 的对齐)。所以**`malloc` 返回的指针必须 16 字节对齐**——这不是性能优化,是 ABI 红线,不对齐程序就会崩。

`max_align_t` 到底多大?四套分配器各自确认过:

| 分配器 | 最小保证对齐 | 出处 |
|--------|------------|------|
| ptmalloc(glibc) | `MALLOC_ALIGNMENT = max(2*SIZE_SZ, __alignof__(long double))`,x86-64 上 = **16** | [malloc-alignment.h](https://github.com/glibc/glibc/blob/main/sysdeps/generic/malloc-alignment.h)(新版已抽出独立头) |
| tcmalloc | `kAlignment = 8`(分配器内部最小),对外保证 `alignof(max_align_t)` | [common.h:212](../tcmalloc/tcmalloc/common.h#L212) |
| jemalloc | `QUANTUM = 1 << LG_QUANTUM`,x86-64 上 `LG_QUANTUM=4` → **16** | [quantum.h:22-23](../jemalloc/include/jemalloc/internal/quantum.h#L22-L23) |
| mimalloc | `MI_MAX_ALIGN_SIZE = 16`(默认,SSE 要求) | [types.h:37-39](../mimalloc/include/mimalloc/types.h#L37-L39) |

注意一个细节:**ptmalloc 的 `MALLOC_ALIGNMENT` 用 `__alignof__(long double)` 兜底**。历史上 glibc 曾因返回 8 字节对齐导致 SSE 程序(要求 16 字节对齐)崩溃,2002 年左右的 [sourceware bug #206](https://sourceware.org/bugzilla/show_bug.cgi?id=206) 把它改成了 `max(2*SIZE_SZ, __alignof__(long double))`。这是一个真实教训:**"分配器以为够了"的对齐,未必"程序真的够"**。

> **所以这样设计**:size class 的最小边界是 8 字节(ptmalloc 在 32 位上、tcmalloc 内部)、典型是 16 字节(64 位上的 `max_align_t`)。所有 size class 的边界都落在 8 或 16 的倍数上,这就从源头保证了——**只要从某个 size class 里 pop 出一个块,它的起点地址天然满足最严格的 ABI 对齐**,分配器不必再额外做对齐调整。对齐不是分配时再"凑",而是 size class 设计时就嵌进了。

## 3.3 硬件真相二:缓存行 64 字节,false sharing 塌方

对齐到 8/16 字节解决了"程序不崩"的底线。但对齐到 64 字节(一个缓存行)解决的是另一件事——**多核性能**。

### 缓存行:CPU 搬内存的最小单位

CPU 缓存(L1/L2/L3)不是按字节存的,是按**缓存行**(cache line)存的,典型一行 **64 字节**(x86、ARM 都是;某些老架构 32 或 128)。当你读 `*p` 一个字节,CPU 实际上是把 `p` 所在的整个 64 字节缓存行从内存(或下一级缓存)整个搬进 L1。下次读同一行的字节就命中 L1,纳秒级。

这有个连锁后果:**缓存一致性(coherence)也是以行为单位的**。多核系统里,每个核有自己的 L1/L2,如果核 A 修改了某行的某个字节,这一整行在别的核的 L1 里就"作废"了(invalidate)——别的核下次读这一行(哪怕读的是另一个字节),都得重新从 L3 或内存搬。

### false sharing:两个计数器挤一行

现在看一个经典陷阱。假设你写了一个性能计数器:

```c
struct counters {
    long hits;        // 线程 A 频繁 ++
    long misses;      // 线程 B 频繁 ++
};
```

两个线程,A 疯狂 `++c.hits`,B 疯狂 `++c.misses`。看起来两人井水不犯河水——`hits` 和 `misses` 是**不同字段**。但它们大概率落进**同一个 64 字节缓存行**(两个 `long` 一共 16 字节,挤在 64 字节里绰绰有余)。后果是:

```
 缓存行(64 字节)
 ┌──────────────────────────────────────────────────────────────────┐
 │ hits │ misses │            (48 字节空闲)                          │
 └──────────────────────────────────────────────────────────────────┘
       ↑           ↑
    A 在写      B 在写
```

每次 A 写 `hits`,这一行在 B 的 L1 里 invalidate;B 下次写 `misses` 要先把这行重新拉进 B 的 L1;B 写完又让 A 的副本 invalidate……两个核就这样**交替抢这一行**,缓存命中率塌方,性能能比"两人写不同行"慢 5~10 倍。这就是 **false sharing**:两人写的明明不是同一字节,却因为"挤同一行"被硬件的缓存一致性机制误判成冲突。

> **钉死这件事**:多核性能的第一杀手不是锁,是 false sharing。两个线程各自频繁写的字段,**只要落进同一个 64 字节缓存行**,这一行就会被反复 invalidate,延迟从纳秒级掉到几十纳秒、甚至上百纳秒(看缓存层级)。分配器自己的 hot 字段(每个 CPU 的计数器、每个 arena 的统计)是最容易踩这个雷的地方——因为它们就是"多核各写各的"。

### 分配器怎么防 false sharing:把 hot 字段按缓存行隔离

四套分配器在源码里大量使用"按缓存行对齐"来防 false sharing。手法是:把每个核/每个线程的 hot 字段,用 `alignas(64)` 或等价宏**强制独占自己的缓存行**(甚至隔一个空洞),让别人写不到我这行。

**jemalloc** 把"64 字节"这个常量钉在 [jemalloc_internal_types.h:82-85](../jemalloc/include/jemalloc/internal/jemalloc_internal_types.h#L82-L85):

```c
#define LG_CACHELINE 6
#define CACHELINE 64
#define CACHELINE_MASK (CACHELINE - 1)
#define CACHELINE_CEILING(s) (((s) + CACHELINE_MASK) & ~CACHELINE_MASK)
```

然后 jemalloc 的三张 size class 查找表本身就 `JEMALLOC_ALIGNED(CACHELINE)`(查表是 fast path,必须独占行),见 [sz.c:5-6](../jemalloc/src/sz.c#L5-L6):

```c
JEMALLOC_ALIGNED(CACHELINE) size_t sz_pind2sz_tab[SC_NPSIZES + 1];
JEMALLOC_ALIGNED(CACHELINE) size_t sz_index2size_tab[SC_NSIZES];   // L72
JEMALLOC_ALIGNED(CACHELINE) uint8_t sz_size2index_tab[(SC_LOOKUP_MAXCLASS >> SC_LG_TINY_MIN) + 1];  // L88
```

jemalloc 的 arena `bin` 数组(每个 bin 是一把独立的锁,争用最敏感)也按缓存行对齐,注释直白写在 [arena.h:162-168](../jemalloc/include/jemalloc/internal/arena.h#L162-L168):

```c
/*
 * Enforcing cacheline-alignment to minimize the number of cachelines
 * touched on the hot paths.
 */
JEMALLOC_ALIGNED(CACHELINE) bin_t all_bins[];
```

**tcmalloc** 不自定义 `CACHELINE` 常量,直接用 abseil 的 `ABSL_CACHELINE_ALIGNED`(底层是 `__attribute__((aligned(64)))`)。代表性的 hot 字段隔离:

- 整个 `Span` 对象按缓存行对齐:[span.h:84](../tcmalloc/tcmalloc/span.h#L84) `class ABSL_CACHELINE_ALIGNED Span final`。
- 全局 SizeMap、CpuCache 按缓存行对齐:[static_vars.cc:74](../tcmalloc/tcmalloc/static_vars.cc#L74)、[static_vars.cc:79](../tcmalloc/tcmalloc/static_vars.cc#L79)。
- per-CPU 的 `ResizeInfo` 结构体特意按缓存行对齐,注释明说要防 false sharing:[cpu_cache.h:592](../tcmalloc/tcmalloc/cpu_cache.h#L592) `struct ABSL_CACHELINE_ALIGNED ResizeInfo`。

**mimalloc** 的 cache-line 对齐宏是 [internal.h:30](../mimalloc/include/mimalloc/internal.h#L30) `#define mi_decl_cache_align mi_decl_align(64)`。最有意思的用法在 [arena.c:62-64](../mimalloc/src/arena.c#L62-L64),三个全局原子变量各自按缓存行隔离,注释直接点了 issue #895(真实踩过 false sharing 的雷):

```c
static mi_decl_cache_align _Atomic(mi_arena_t*) mi_arenas[MI_MAX_ARENAS];
static mi_decl_cache_align _Atomic(size_t)      mi_arena_count;
static mi_decl_cache_align _Atomic(int64_t)     mi_arenas_purge_expire;
```

> **不这样会怎样**:如果不把这三个原子变量按缓存行隔离,它们(各 8 字节)会挤进同一行。每个线程 `mmap` 时 `++mi_arena_count`、每个 purge 周期改 `mi_arenas_purge_expire`——这些写操作会让"持有 `mi_arenas` 数组读"的别的核反复 invalidate。mimalloc 真在这个位置踩过坑(见 issue #895),修法就是给每个变量加 `mi_decl_cache_align`。

> **所以这样设计**:分配器内部的每个"被多核频繁各写各的"字段,都要按 64 字节缓存行隔离。这一条不是 size class 表能直接体现的,但它是 size class 对齐策略的**深层动机之一**——某些 size class 的边界(比如 tcmalloc 在 >1024 时把对齐升到 128、jemalloc 的部分 class 落在 64 倍数上)就是为了"让 hot 对象天然不挤同一行"。

## 3.4 技巧铺垫:为什么对齐到 2 的幂——用掩码代替除法

到这里有一个看起来无关但极其关键的小技巧:**所有 size class 的边界都是 2 的幂(或 2 的幂的倍数),不是巧合,是为了能用位运算代替除法**。

纳秒级 fast path 里,**一条 `idiv`(整数除法)指令要 20~40 个周期**,而一条 `and` 指令只要 1 个周期。当一次 `malloc` 要从 size 算出 size class、从地址算出页号、从页号算出 span 时,如果每步都做除法,fast path 就毁了。但如果对齐数是 2 的幂,就能用掩码代替:

- **向上对齐到 N(N 是 2 的幂)**:`(x + (N-1)) & ~(N-1)` —— 两条 ALU 指令,1 周期。
- **向下对齐到 N**:`x & ~(N-1)` —— 一条指令。
- **判断是否对齐**:`(x & (N-1)) == 0` —— 一条指令。
- **取模 N**:`x & (N-1)` —— 一条指令。

四套分配器的对齐辅助函数,清一色这套掩码手法。tcmalloc 的 `RoundUp`/`RoundDown` 在 [system_allocator.h:286-295](../tcmalloc/tcmalloc/internal/system_allocator.h#L286-L295):

```c
// Rounds size down to a multiple of alignment.
inline size_t RoundDown(const size_t size, const size_t alignment) {
  TC_ASSERT(absl::has_single_bit(alignment));   // 必须 2 的幂
  return (size) & ~(alignment - 1);
}

// Rounds size up to a multiple of alignment.
inline size_t RoundUp(const size_t size, const size_t alignment) {
  return RoundDown(size + alignment - 1, alignment);
}
```

注意那个 `TC_ASSERT(absl::has_single_bit(alignment))`——**对齐必须是 2 的幂**,这是掩码手法的隐含前提,断言就是来钉它的。tcmalloc 把页大小向上取整的代码 [tcmalloc.cc:1822](../tcmalloc/tcmalloc/tcmalloc.cc#L1822) `size = (size + page_size - 1) & ~(page_size - 1);` 就是同款套路。

mimalloc 的版本在 [internal.h:384-403](../mimalloc/include/mimalloc/internal.h#L384-L403),专门给"非 2 的幂"留了退路(用真除法),但 fast path 走的是 2 的幂分支:

```c
static inline uintptr_t _mi_align_up(uintptr_t sz, size_t alignment) {
  mi_assert_internal(alignment != 0);
  uintptr_t mask = alignment - 1;
  if ((alignment & mask) == 0) {   // 是 2 的幂?
    return ((sz + mask) & ~mask);  // 快路径:位运算
  } else {
    return (((sz + mask) / alignment) * alignment);  // 慢路径:真除法
  }
}
```

glibc ptmalloc 的对齐掩码是全局宏 `MALLOC_ALIGN_MASK = MALLOC_ALIGNMENT - 1`(因为 `MALLOC_ALIGNMENT` 是 2 的幂,减 1 就是全 1 掩码),满文件都是 `(x + MALLOC_ALIGN_MASK) & ~MALLOC_ALIGN_MASK` 的写法。源码见 [malloc.c](https://github.com/glibc/glibc/blob/main/malloc/malloc.c) 的 `request2size` 宏。

> **钉死这件事**:对齐到 2 的幂 = 用掩码代替除法 = fast path 纳秒级。这不是"代码风格",是**性能生死线**。如果 size class 的边界是随便取的奇数,每一步对齐都要做除法,fast path 就废了。所有 size class 的边界、所有页/缓存行的边界,都必须是 2 的幂——这是 size class 设计的**第一约束**,比对齐本身的硬件动机还更底层。

## 3.5 tcmalloc 的二段式:≤1024 对齐 8、>1024 对齐 128

现在可以看四套分配器各自的 size class 对齐策略了。

**tcmalloc 的策略最直白:分两段。** 源码就两行,在 [sizemap.cc:108-115](../tcmalloc/tcmalloc/sizemap.cc#L108-L115),写在一个叫 `IsValidSizeClass` 的校验函数里(用来检查 size class 表的合法性):

```c
  // Check required alignment
  const size_t alignment = size > SizeMap::kLargeSize
                               ? kLargeSizeAlignment
                               : static_cast<size_t>(kAlignment);
  if ((size & (alignment - 1)) != 0) {
    TC_LOG("%v not aligned properly %v", size, alignment);
    return false;
  }
```

两个常量在 [sizemap.h:58-59](../tcmalloc/tcmalloc/sizemap.h#L58-L59) 钉死:

```c
  static constexpr int kLargeSize = 1024;
  static constexpr int kLargeSizeAlignment = 128;
```

加上 [common.h:212](../tcmalloc/tcmalloc/common.h#L212) 的 `kAlignment = 8`,合起来就是:

- **size ≤ 1024 字节**:必须 8 字节对齐(每 8 字节一个 class)。
- **size > 1024 字节**:必须 **128 字节对齐**(每 128 字节一个 class)。

为什么分两段?

> **不这样会怎样**:假设全段都按 8 字节对齐,class 会爆炸。1024 到 262144(256KB, tcmalloc 最大 class)之间有 26 万字节,按 8 字节分就是 3 万多个 class——每个 class 一条 free list、一份 cache,内存和元数据开销无法承受。所以大 size 段必须把"对齐步长"放大,压缩 class 数量。tcmalloc 选了 128(= 2^7),既能覆盖多数硬件对齐需求(`__m128` 是 16、AVX 是 32、部分 prefetch 友好对齐到 64/128),又能让 1KB~256KB 这段只产生约 2000 个 class,可控。

看 tcmalloc 真实的 size class 表([size_classes.cc](../tcmalloc/tcmalloc/size_classes.cc),`TCMALLOC_PAGE_SHIFT==13` 默认配置),节选几行就能看出二段式的痕迹:

```
{     8,    8192,   32},   // class 1   size 8     align 8      (≤1024 段)
{    16,    8192,   32},   // class 2   size 16    align 8
{    32,    8192,   32},   // class 3   size 32    align 8
{    96,    8192,   32},   // class 6   size 96    align 8      (96=8×12,不是 32 的倍数)
{   128,    8192,   32},   // class 8   size 128   align 8
{  1024,    8192,   32},   // class 20  size 1024  align 8      (= kLargeSize, 临界)
{  1152,   16384,   32},   // class 21  size 1152  align 128    (>1024,1152=128×9)
{  2048,    8192,   32},   // class 24  size 2048  align 128    (2048=128×16)
{  4096,    8192,   16},   // class 27  size 4096  align 128
{ 262144,  262144,    2},  // class 47  size 256KB align 128    (最大 class)
```

注意 class 6 的 size 是 **96**——它**不是** 32 的倍数,只是 8 的倍数。这就点破一个常见误解:tcmalloc 的 size class 边界**不一定**按 16/32/64 排,小于 1024 的段只要满足 8 字节对齐就合格(实际表里混着 8、16、32、48、64、80、96……)。"对齐"是底线要求,不是"每个 class 都按对齐数等距排列"。

> **所以这样设计**:tcmalloc 用"8/128 二段式"在三个目标间取平衡——① 小对象密集对齐到 8(满足 ABI、控制内部碎片);② 大对象稀疏对齐到 128(压缩 class 数、对齐到 prefetch 友好边界);③ 全是 2 的幂(用掩码代替除法)。这套设计在 [sizemap.h:66-68](../tcmalloc/tcmalloc/sizemap.h#L66-L68) 的注释里说得很直白:"Sizes ≤ 1024 have an alignment >= 8… Sizes > 1024 have an alignment >= 128"——并据此用了**两张查找表**:小表按 `ceil(size/8)` 索引、大表按 `ceil(size/128)` 索引,各是密集的位数组。

## 3.6 jemalloc 的精细化:quantum + 分组 delta

jemalloc 比 tcmalloc 更精细,它把 size class 的对齐步长**随 size 增长而增长**——不是粗暴的"二段式",而是"每个 pow2 区间内,对齐步长 = 区间起点 / 4"。源码全部围绕 `lg_base`、`lg_delta`、`ndelta` 三个量展开。

### 三个关键常量

先看三个常量(都在 [sc.h](../jemalloc/include/jemalloc/internal/sc.h)):

- `SC_LG_TINY_MIN = 3`(行 173):tiny 类的最小对齐(8 字节)。
- `SC_LG_NGROUP = 2`(行 172):每个 pow2 区间分 4 个 class(`NGROUP = 2^2 = 4`)。
- `LG_QUANTUM`(在 [quantum.h](../jemalloc/include/jemalloc/internal/quantum.h)):"量子",小对象的基准对齐。x86-64 上是 4(`QUANTUM = 16`),ARMv7 上是 3(`QUANTUM = 8`)。

jemalloc 的设计文档就在 [sc.h:7-167](../jemalloc/include/jemalloc/internal/sc.h#L7-L167) 那一大段注释里,翻译成人话是:

1. **tiny 类(8 ~ QUANTUM/2)**:每个 2 的幂一个 class,`lg_delta = lg_base`。比如 QUANTUM=16 时,tiny 类有 8(对齐 8)。这些是为了"小到没必要 16 字节对齐"的小对象设计的(比如 3 字节的字符串),减少内部碎片。
2. **sub-quantum 到 QUANTUM**:对齐到 `lg_base`(每个 2 的幂一个 class)。
3. **QUANTUM 起,进入"分组"模式**:每个 pow2 区间 `[2^k, 2^(k+1))` 切成 `NGROUP = 4` 个等距 class,**对齐步长 = `2^k / 4 = 2^(k-2)`**。比如区间 `[16, 32)` 切成 16/20/24/28 四个 class,每个隔 4(`lg_delta = 2`);区间 `[256, 512)` 切成 256/320/384/448 四个 class,每个隔 64(`lg_delta = 6`)。

这就是 jemalloc 著名的"**delta doubles per band**"(delta 每个区间翻一倍)机制。源码生成逻辑在 [sc.c:169-210](../jemalloc/src/sc.c#L169-L210) 的 `size_classes` 函数,核心是这一段:

```c
// sc.c:169 起 —— 常规分组循环
while (...) {
    lg_base = lg_base + lg_ngroup;     // 跳到下一个 pow2 区间
    ...
    while (ndelta <= ndelta_limit) {
        ...
        size_class(..., lg_base, lg_delta, ndelta, ...);
        ndelta++;
    }
    ...
    lg_delta++;                          // 关键:每个区间后 delta 翻一倍
}
```

`lg_delta++`(行 209)就是"对齐步长每个区间翻一倍"的字面实现。

### 从 size 算 usable size:位运算之美

这套设计的真正妙处,是**从任意 size 算出对应的 usable size(对齐到 class 后的大小)可以纯位运算完成,不必查表**。jemalloc 的 `sz_s2u`(size to usable size)在 [sz.h:316-336](../jemalloc/include/jemalloc/internal/sz.h#L316-L336),核心是:

```c
size_t x = lg_floor((size << 1) - 1);
size_t lg_delta = (x < SC_LG_NGROUP + LG_QUANTUM + 1)
    ? LG_QUANTUM
    : x - SC_LG_NGROUP - 1;
// ... 用 delta_mask 向上取整 ...
usize = (size + delta_mask) & ~delta_mask;
```

读这段:先 `lg_floor` 算出 size 落在哪个 pow2 区间(`x`),再据此推出 `lg_delta`(对齐步长),最后 `(size + delta_mask) & ~delta_mask` 向上取整到这个步长。**全程位运算,没有除法,没有查表**(只有 size 很小时才走 `sz_s2u_lookup` 直接查一个小表)。

> **不这样会怎样**:如果不设计成"delta 每个 pow2 翻一倍",就没法用 `lg_floor + 一个减法` 算出 delta,只能查表。查表意味着多一次内存访问(表项可能不在 L1),fast path 就毁了。jemalloc 这套"banding"设计的核心动机,就是**让 size→class 的映射可以纯位运算**,在 fast path 上零额外访存。

### sz_sa2u:对齐感知的版本

如果用户额外要求对齐(比如 `posix_memalign(..., 64, ...)`),jemalloc 走 `sz_sa2u`(size + alignment to usable size)在 [sz.h:400-463](../jemalloc/include/jemalloc/internal/sz.h#L400-L463)。这里有个 jemalloc 独有的聪明点,源码注释在 [sz.h:414-425](../jemalloc/include/jemalloc/internal/sz.h#L414-L425) 写得很清楚:

> *"for each small size class, every object is aligned at the smallest power of two that is non-zero in the base two representation of the size. For example: 96 → 32, 144 → 32, 192 → 64."*

翻译:**一个 size class 的对象天然对齐到"它的 size 的最低非零二进制位"对应的 2 的幂**。size 96 = `0b1100000`,最低非零位是 bit 5(32),所以 96 字节的 class 天然 32 字节对齐。这意味着——用户要 32 字节对齐、申请 96 字节时,jemalloc **不需要额外做任何对齐处理**,因为 size class 本身已经满足。这就是 jemalloc 对齐策略的"自洽":size class 的对齐步长已经嵌进了每个 class 的 size,外部对齐要求大多是"白送"的。

## 3.7 mimalloc 的策略:8 字节起、page 起点按 block 微调

mimalloc 的策略介于 tcmalloc 和 jemalloc 之间:小对象 8 字节起、按 ~1.25 倍指数增长;大对象按页对齐。

mimalloc 的 size class 表在 [init.c:55-67](../mimalloc/src/init.c#L55-L67) 那个 `MI_PAGE_QUEUES_EMPTY` 宏里(以 `sizeof(uintptr_t)` 为单位,64 位上乘 8):

```
1,2,3,4,5,6,7,8, 10,12,14,16, 20,24,28,32, 40,48,56,64,
80,96,112,128, 160,192,224,256, 320,384,448,512, ...
```

乘 8 后就是(64 位):**8, 16, 24, 32, 40, 48, 56, 64, 80, 96, 112, 128, 160, 192, 224, 256, ...**。前 8 个 class(8~64)是每 8 字节一个,正好满足"小对象 8 字节对齐"。从 64 之后按 ~1.25 倍指数增长(80, 96, 112, 128,然后 160, 192, 224, 256……)。最大 class 数固定 73 个(`MI_BIN_HUGE = 73`,在 [types.h:208](../mimalloc/include/mimalloc/types.h#L208))。

mimalloc 有一个 jemalloc/tcmalloc 都没有的独门技巧:**page 起点(block 起始地址)按 block size 微调**。源码在 [segment.c:344-369](../mimalloc/src/segment.c#L344-L369) 的 `_mi_segment_page_start_from_slice`:

```c
  if (block_size > 0 && block_size <= MI_MAX_ALIGN_GUARANTEE) {
    // 把 page 起点对齐到 block size 的倍数
    const size_t adjust = block_size - ((uintptr_t)pstart % block_size);
    if (adjust < block_size && psize >= block_size + adjust) {
      start_offset += adjust;
    }
  }
  ...
  start_offset = _mi_align_up(start_offset, MI_MAX_ALIGN_SIZE);
```

这个微调做了两件事:① 让 page 内第一个 block 的地址天然是 `block_size` 的倍数(保证所有 block 都对齐到自己的 size);② 故意偏移一点点,避免不同 page 里同 offset 的 block 落进同一组 set-associative cache 的同一索引(cache aliasing,会引发另一种伪共享)。

mimalloc 的对齐 API 验证逻辑是经典三步:① 必须是 2 的幂;② 至少 `sizeof(void*)`(只在 `posix_memalign` 入口要求);③ 大于 `MI_BLOCK_ALIGNMENT_MAX` 走 dedicated huge page。源码在 [alloc-aligned.c:177-182](../mimalloc/src/alloc-aligned.c#L177-L182):

```c
  if mi_unlikely(alignment == 0 || !_mi_is_power_of_two(alignment)) {
    #if MI_DEBUG > 0
    _mi_error_message(EOVERFLOW, "aligned allocation requires the alignment to be a power-of-two ...");
    #endif
    return NULL;
  }
```

`_mi_is_power_of_two` 是经典位运算技巧 `(x & (x-1)) == 0`(在 [internal.h:374-376](../mimalloc/include/mimalloc/internal.h#L374-L376)),一条指令判定 2 的幂。

> **所以这样设计**:mimalloc 的对齐策略偏"实用主义"——最小对齐 16(`MI_MAX_ALIGN_SIZE`,满足 SSE),小对象 8 字节密集分布(同 tcmalloc),page 起点按 block size 微调(独门,既保证对齐又防 cache aliasing),超大对齐走 dedicated huge segment。它没有 jemalloc 那么精巧的位运算映射,但实际效果接近。

## 3.8 大块对齐:为什么 4KB 起、超大要 2MB

前面讲的都是"小对象 size class"的对齐。但 size class 的最大也就 256KB(tcmalloc)/512KB(jemalloc)/256KB(mimalloc)。再大的块,分配器走"大块路径",**对齐要求陡然升级到页级**。

### 为什么大块必须页对齐

页是操作系统管理内存的最小单位,Linux/x86 上**一页 4KB**。分配器向 OS 要内存,必须按页要(`mmap` 的长度向上取整到页);把内存还回 OS,也必须按页(`munmap`/`madvise` 的粒度是页)。所以分配器内部管理大块,**必须按页对齐**——这样:

1. **切分/合并是 O(1)**:页对齐的地址,取页号就是 `addr >> 12`(位运算)。
2. **归还 OS 是整页**:不会出现"半页空、半页在用"没法还的情况。
3. **`mmap` 返回的指针天然页对齐**:OS 给的内存本来就是页对齐的。

四套分配器的"页大小"常量:

| 分配器 | 页大小常量 | 值(默认) | 出处 |
|--------|----------|----------|------|
| tcmalloc | `kPageSize = 1 << kPageShift` | `1<<13 = 8KB`(默认;可配 4KB/32KB/256KB) | [common.h:108-210](../tcmalloc/tcmalloc/common.h#L108-L210) |
| jemalloc | `PAGE = 1 << LG_PAGE` | configure 检测,x86 上 `LG_PAGE=12` → 4KB | [pages.h:14](../jemalloc/include/jemalloc/internal/pages.h#L14) |
| mimalloc | `MI_SEGMENT_SLICE_SIZE = 1 << MI_SEGMENT_SLICE_SHIFT` | 64KB(`13 + MI_INTPTR_SHIFT`,64 位上) | [types.h:172-184](../mimalloc/include/mimalloc/types.h#L172-L184) |
| ptmalloc | `pagesize` | 运行时 `sysconf(_SC_PAGESIZE)`,x86 上 4KB | [malloc.c](https://github.com/glibc/glibc/blob/main/malloc/malloc.c) |

> **钉死一件事**:tcmalloc 默认**页大小 8KB**(`kPageShift=13`),不是 4KB。这是一个反直觉的设计选择——tcmalloc 故意把"内部页"做得比"硬件页"大(8KB = 2 个硬件页),目的是减少 span 的元数据开销(span 数 = 总内存 / 页大小,页大 → span 少)。tcmalloc 还支持 32KB、256KB 页(对应 `TCMALLOC_INTERNAL_32K_PAGES` / `256K_PAGES` 编译选项),走的是更激进的"大页化管理"。jemalloc 和 mimalloc 则更贴近硬件页:jemalloc 用 4KB,mimalloc 用 64KB slice(更大,因为 mimalloc 的 segment 是 32MB,slice 是 segment 内的子单位)。

### 超大块:2MB huge page 对齐

到了"超大块"(几 MB 起),分配器会进一步对齐到 **2MB huge page**。这是为了:

1. **降低 TLB miss**:普通 4KB 页,1GB 内存要 26 万个页表项,TLB 装不下;2MB huge page,1GB 只要 512 项,TLB 命中率大增。一个 miss 能让访存从纳秒级掉到百纳秒级。
2. **治碎片**:huge page 是整页不可切分的,把它当单位管理,外部碎片被压缩在 huge page 内部。

四套的 huge page 常量:

- **tcmalloc**:`kHugePageShift = 21` → 2MB(x86-64/ARM64/RISC-V)、16MB(PPC64),见 [config.h:171-186](../tcmalloc/tcmalloc/internal/config.h#L171-L186)。tcmalloc 的 HPAA(huge page aware allocator,第 15 章详讲)就是围绕这个 2MB 单位组织的。
- **jemalloc**:`HUGEPAGE = 1 << LG_HUGEPAGE`,`LG_HUGEPAGE` configure 检测,典型 21(2MB),见 [pages.h:24](../jemalloc/include/jemalloc/internal/pages.h#L24)。HPA(huge page allocator)在 [hpa_hooks.c:29-30](../jemalloc/src/hpa_hooks.c#L29-L30) 显式按 huge page 对齐 `pages_map(NULL, size, HUGEPAGE, ...)`。
- **mimalloc**:支持 large OS page(2 或 4MB)和 huge OS page(1GB 级),通过 option `allow_large_os_pages` / `reserve_huge_os_pages` 开关,见 [options.c:129-132](../mimalloc/src/options.c#L129-L132)。
- **ptmalloc**:不主动用 huge page,依赖内核的 transparent huge pages(THP),通过 `madvise(MADV_HUGEPAGE)` 启用,这是 baseline 的"朴素"做法。

> **所以这样设计**:大块按页(4KB/8KB)对齐,是为了让页堆能整页切分、整页归还;超大块按 huge page(2MB)对齐,是为了压 TLB miss、治碎片。**对齐步长越大,管理粒度越粗,元数据越省,但内部碎片也越大**——这是又一个"快与省"的权衡。

## 3.9 四套横评:同一组硬件约束,四套答案

把四套的对齐策略放一张表,对比最清楚:

| 维度 | tcmalloc | jemalloc | mimalloc | ptmalloc(baseline) |
|------|----------|----------|----------|----------|
| **最小对外对齐** | `kAlignment=8`(内部);对外保 `max_align_t` | `QUANTUM=16`(x86-64) | `MI_MAX_ALIGN_SIZE=16` | `MALLOC_ALIGNMENT=max(2*SIZE_SZ, __alignof__(long double))`,x86-64 上 16 |
| **小 size class 步长** | ≤1024:对齐 8,大致每 8~16 字节一个 class | 分组:每 pow2 区间 4 个 class,delta 翻倍 | 8~64 每 8 字节;之后 ~1.25x 指数 | bins 区间(fastbin/smallbin/largebin)非等距 |
| **大 size class 步长** | >1024:对齐 128 | 同上分组规则,继续到 ~512KB | 同上指数规则,到 ~256KB | largebin 按对数分组 |
| **size→class 计算** | 查两张表(`/8` 和 `/128` 索引) | **纯位运算**(`lg_floor` + delta,零查表) | 查表(`MI_PAGES_DIRECT` 直接索引) | 查 bins 表 |
| **缓存行隔离** | `ABSL_CACHELINE_ALIGNED`(Span/SizeMap/CpuCache/ResizeInfo) | `JEMALLOC_ALIGNED(CACHELINE)`(sz 表/arena bins)+ 字段顺序(mutex.h) | `mi_decl_cache_align`(arena 全局原子/stats) | (无明显策略) |
| **页大小** | 默认 8KB(可配 4/32/256KB) | configure 检测,典型 4KB | slice 64KB、segment 32MB | 4KB(运行时) |
| **huge page** | 2MB(`kHugePageSize`,HPAA 围绕它) | 2MB(`HUGEPAGE`,HPA 显式按它对齐) | 2MB/4MB large page + 1GB huge page | 依赖内核 THP(`madvise`) |

这张表里有几个值得反复看的点:

1. **最小对外对齐,四套都是 16 字节**(64 位上),这是 `max_align_t` 的要求,没人敢违反。
2. **size→class 的计算,jemalloc 最精巧**(纯位运算,零查表),tcmalloc/mimalloc 用查表(但表小、在 L1)。这是 jemalloc 在 fast path 上的一个微小优势。
3. **缓存行隔离,四套都重视**,但手法不同:tcmalloc/jemalloc/mimalloc 都有显式的 64 字节对齐宏;ptmalloc 是 baseline,这块没那么讲究。
4. **huge page,tcmalloc 和 jemalloc 是"显式管理"**(各自有 HPAA/HPA 模块),mimalloc 是"按需 mmap",ptmalloc 是"靠内核 THP"——这是新一代分配器相对 baseline 的代差之一。

> **钉死这件事**:四套分配器的对齐策略,**底层动机完全一致**(满足硬件对齐、防 false sharing、用掩码代替除法、大块按页对齐),但落地手法有显著差异。tcmalloc 简洁(二段式)、jemalloc 精巧(分组 delta + 位运算)、mimalloc 实用(page 起点微调)、ptmalloc 朴素(baseline)。这些差异,本质上是同一组硬件约束在不同设计哲学下的解。

## 3.10 技巧精解:缓存行对齐防 false sharing——两个计数器挤一行的反面对比

本章最硬核的技巧,就是把"缓存行隔离"这件事用一个具体例子拆透。

### 反面案例:朴素实现,两个计数器挤一行

假设你写一个简单的多线程计数器,统计两个线程各自处理的请求数:

```c
// ❌ 朴素实现:false sharing 的活教材
struct stats {
    uint64_t requests;    // 线程 A 频繁 ++
    uint64_t errors;      // 线程 B 频繁 ++
};

struct stats s;           // 两个 8 字节字段,大概率落进同一缓存行
```

两个线程,A 在循环里 `++s.requests`,B 在循环里 `++s.errors`。看起来两人井水不犯河水——但是看缓存布局:

```
没有缓存行隔离时的内存布局(64 字节缓存行)
┌─────────────────────────────────────────────────────────────────────┐
│ requests(8B) │ errors(8B) │          48 字节空闲                      │
└─────────────────────────────────────────────────────────────────────┘
       ↑              ↑
    核 A 在写      核 B 在写
    (整行在核 B 的 L1 invalidate)
                   (整行在核 A 的 L1 invalidate)
```

每次 A 写 `requests`,**整行**在 B 的 L1 里被 invalidate;B 下次写 `errors`,要把整行从 A 的 L1(或 L3)重新搬进 B 的 L1;B 写完又 invalidate A 的副本……两个核就这样**交替抢一行**。延迟从 L1 命中的 ~1ns 掉到跨核同步的 ~40ns(看架构),实测一段 1 亿次 ++ 的微基准,朴素版能比"两人写不同行"**慢 5~10 倍**。

### 正解:按缓存行隔离

修复手法就是给每个 hot 字段加 `alignas(64)`(或分配器内的等价宏),让每个字段**独占一行**:

```c
// ✅ 正解:每个 hot 字段独占缓存行
struct stats {
    alignas(64) uint64_t requests;    // 独占一行
    alignas(64) uint64_t errors;      // 独占下一行
};
```

布局变成:

```
有缓存行隔离时的内存布局
缓存行 0                                  缓存行 1
┌──────────────────────────────────┐    ┌──────────────────────────────────┐
│ requests(8B) │  56 字节空洞       │    │ errors(8B)   │  56 字节空洞       │
└──────────────────────────────────┘    └──────────────────────────────────┘
       ↑                                          ↑
    核 A 在写                                  核 B 在写
    (只 invalidate 核 B 的缓存行 0)         (只 invalidate 核 A 的缓存行 1)
    —— 但核 B 根本不读这行!                  —— 核 A 也不读这行!
    → 没有 invalidation 风暴
```

代价是多了 56+56 = 112 字节的"空洞"——这就是"省"和"快"的权衡:**为了多核性能,我们愿意浪费一些内存来换缓存行的独占**。

### 分配器源码里的真实例子

这个技巧在四套分配器里无处不在。挑三个最有代表性的:

**例子 1:tcmalloc 的 `ResizeInfo`**。per-CPU 的 resize 信息(每个核一份、各写各的),[cpu_cache.h:592](../tcmalloc/tcmalloc/cpu_cache.h#L592):

```cpp
struct ABSL_CACHELINE_ALIGNED ResizeInfo {
  ...
};
```

`ABSL_CACHELINE_ALIGNED` 展开是 `__attribute__((aligned(64)))`。每个 CPU 核的 `ResizeInfo` 独占一行,核 A 改它的、核 B 改它的,互不 invalidate。

**例子 2:jemalloc 的 arena `bin` 数组**。每个 bin 是一把独立锁、被不同线程争用,[arena.h:162-168](../jemalloc/include/jemalloc/internal/arena.h#L162-L168):

```c
/*
 * Enforcing cacheline-alignment to minimize the number of cachelines
 * touched on the hot paths.
 */
JEMALLOC_ALIGNED(CACHELINE) bin_t all_bins[];
```

jemalloc 不只是简单加宏——它在 [mutex.h:22-74](../jemalloc/include/jemalloc/internal/mutex.h#L22-L74) 里还**精心安排字段顺序**,把"争用方会修改"的 `prof_data` 放在结构体最前面,把"持锁方会修改"的 mutex 字段放后面。注释写得很直白([mutex.h:26-33](../jemalloc/include/jemalloc/internal/mutex.h#L26-L33)):

> *"prof_data is defined first to reduce cacheline bouncing: the data is not touched by the mutex holder during unlocking, while might be modified by contenders. Having it before the mutex itself could avoid prefetching a modified cacheline (for the unlocking thread)."*

这是一种更精细的"逻辑隔离":**没有用对齐宏,纯靠字段顺序**让"持锁方读的字段"和"争用方写的字段"落在不同缓存行。

**例子 3:mimalloc 的 arena 全局原子**。三个全局原子变量被三个不同的代码路径频繁改写(每线程 mmap 改 `mi_arena_count`、purge 周期改 `mi_arenas_purge_expire`),[arena.c:62-64](../mimalloc/src/arena.c#L62-L64) 各自 `mi_decl_cache_align`:

```c
static mi_decl_cache_align _Atomic(mi_arena_t*) mi_arenas[MI_MAX_ARENAS];
static mi_decl_cache_align _Atomic(size_t)      mi_arena_count;
static mi_decl_cache_align _Atomic(int64_t)     mi_arenas_purge_expire;
```

mimalloc 的注释点出了这是个真实修过的雷(issue #895)。不加这三个 `mi_decl_cache_align`,三变量挤一行,每个 mmap/purge 都会让所有核的相关缓存行 invalidate。

### 一个常被忽视的变种:用 padding 而不是 alignas

有时结构体不能整体 `alignas(64)`(比如它要放进数组、对齐会浪费太多),就用**手填 padding 字段**:

```c
struct per_cpu_stats {
    uint64_t requests;
    uint64_t errors;
    char pad[48];     // 凑满 64 字节,让下一个字段落在新一行
};
```

tcmalloc 的 `huge_page_filler.h` 用更精细的版本——它在 [huge_page_filler.h:107-122](../tcmalloc/tcmalloc/huge_page_filler.h#L107-L122) 用 `static_assert(offsetof(...) <= 2 * ABSL_CACHELINE_SIZE)` 断言"hot 字段必须落在前两个缓存行内",这样布局不对了编译期就报错。这是一种**编译期保证**缓存友好布局的手法,比事后 profile 更可靠。

> **钉死这件事**:false sharing 是多核性能的隐形杀手,但它"看起来不像 bug"(程序结果正确,只是慢)。防它的唯一办法是:**所有被多核频繁各写各的 hot 字段,要么 `alignas(64)` 独占一行,要么精心安排字段顺序让冲突字段落不同行**。四套分配器在这件事上都不敢马虎——你看到源码里到处是 `CACHELINE_ALIGNED`、`mi_decl_cache_align`,不是因为"代码风格",是因为这一行不对齐,fast path 就塌了。

## 章末小结

这一章我们钻进了 size class 的"对齐"细节。表面上看是"边界为什么是 8/16/32",深里看是四件事:

1. **硬件对齐**:`malloc` 返回的指针必须满足 `max_align_t`(典型 16 字节),不对齐要么慢(x86)、要么崩(ARM/SPARC)。size class 的最小边界嵌进 8/16,从源头保证 ABI。
2. **缓存行隔离**:缓存按 64 字节的行搬运,两个线程的 hot 字段若挤一行就是 false sharing,性能塌方。分配器内部所有 hot 字段都按缓存行隔离(`ABSL_CACHELINE_ALIGNED` / `JEMALLOC_ALIGNED(CACHELINE)` / `mi_decl_cache_align`)。
3. **掩码代替除法**:对齐到 2 的幂,就能用 `(x + mask) & ~mask` 代替 `idiv`,fast path 才能纳秒级。这是 size class 边界必须是 2 的幂的最底层动机。
4. **大块按页、超大按 huge page**:4KB 页让页堆能整页管理;2MB huge page 压 TLB miss、治碎片。对齐步长越大,管理越粗、元数据越省、但内部碎片越大——又一个"快与省"的权衡。

四套分配器对这四件事给出了不同答案:**tcmalloc** 简洁(≤1024 对齐 8、>1024 对齐 128 的二段式)、**jemalloc** 精巧(分组 delta + size→class 纯位运算)、**mimalloc** 实用(page 起点按 block 微调防 cache aliasing)、**ptmalloc** 朴素(`2*SIZE_SZ` 起、靠内核 THP)。

### 回扣二分法:这一章服务"局部缓存"

这一章的对齐策略,**几乎全部服务"局部缓存"这一面**(fast path):

- 满足 ABI 对齐 → fast path 返回的指针能直接用,不必再调对齐。
- 缓存行隔离 → fast path 操作的 hot 字段(每个核的 cache、计数器)不互相 invalidate。
- 掩码代替除法 → fast path 的 size→class 计算纳秒级。
- size class 边界对齐 → free list pop 出来的块天然对齐,fast path 不必做对齐调整。

只有"大块按页/huge page 对齐"那一小节偏向"中心堆"(slow path,为了页堆管理),但即使是它,也是为了让 fast path miss 后的 slow path 不至于太慢。

### 五个"为什么"清单

1. **为什么 `malloc` 返回的指针必须 16 字节对齐?** 这是 C ABI 红线(`max_align_t`),`long double`、SSE `__m128` 都要求 16 字节对齐。不对齐,程序在 x86 上慢、在 ARM/SPARC 上直接 `SIGBUS`。glibc 历史上踩过(sourceware bug #206)。
2. **为什么 size class 的边界是 8/16/32……,不能是 13 或 27?** 三个原因:① 满足硬件对齐(边界是 8 的倍数,保证块起点合格);② 2 的幂好做位运算对齐(掩码代替除法);③ 大段对齐到 128 压缩 class 数(否则 class 爆炸)。ptmalloc/tcmalloc/jemalloc/mimalloc 全部遵守。
3. **为什么分配器内部到处是 `alignas(64)` / `CACHELINE_ALIGNED`?** 防 false sharing。两个线程各写各的 hot 字段,若落进同一 64 字节缓存行,这一行会被反复 invalidate,性能能塌方 5~10 倍。把每个 hot 字段按缓存行隔离,代价是多 56 字节空洞,换多核不抢行。
4. **为什么 jemalloc 能纯位运算从 size 算 class,不查表?** 因为它的 size class 设计成"每个 pow2 区间分 4 个等距 class,delta = 区间起点/4"——这样 `lg_delta = lg_floor(size) - SC_LG_NGROUP - 1`,一条 `bsr` + 一个减法搞定。这是 jemalloc 在 fast path 上的设计美学。
5. **为什么大块要按 4KB 页对齐、超大块要按 2MB huge page 对齐?** 页是 OS 管理内存的最小单位,按页对齐才能整页切分/归还/mmap;2MB huge page 大幅减少页表项(1GB 从 26 万项降到 512 项),压 TLB miss,治外部碎片。tcmalloc 的 HPAA、jemalloc 的 HPA 都围绕这个 2MB 单位。
