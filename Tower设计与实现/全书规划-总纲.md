## 一、这本书的定位

### 目标读者

用 axum 写过 Web、用 tonic 写过 gRPC、给 `axum::Router` 套过 `.layer(TimeoutLayer)`、给 reqwest 加过 `tower::timeout`,知道 Tower 是 Rust 异步生态的"通用中间件语言",但**一知半解**:知道有 `Service`、`Layer`、`ServiceBuilder`、timeout/retry/限流/负载均衡,却讲不清 `poll_ready` 到底在背什么压、为什么 `call(&mut self)` 而不是 `&self`、`Layer` 怎么像洋葱一样层层套、`Buffer` 怎么把一个 `!Clone` 的服务变成可克隆的、`Balance` 的 P2C 算法凭什么好用、`ConcurrencyLimit` 和 `RateLimit` 一个控并发一个控速率差别在哪、为什么 axum/tonic/Pingora/reqwest 都用 Tower。和"读过 Tokio 不理解"、"翻过 hyper 一知半解"是同一种病。

### 全书一句话主旨

> **Tower 做的事,本质是把"处理一个请求"抽象成一个 `Service<Request> -> Future<Response>`,再把"装饰这个请求处理"抽象成一个 `Layer`——两者拼起来,就能像洋葱一样层层套出 timeout/retry/限流/负载均衡等无限中间件。它是 Rust 异步生态的"请求处理通用语言",也是 hyper/axum/tonic/reqwest/Pingora 的共同骨架。**

每个组件,都是"Service × Layer"拼合的一个环节。读完一本书,读者不是"多记了几个 Tower API",而是能**在脑子里放映出一次请求穿过 Tower 中间件栈的全过程**:请求进来 → 最外层 `Layer` 包出的 `Service` 接到请求 → 它的 `poll_ready` 先轮询(背压)→ 拿到 `Ready` 后 `call` 把请求往下传、返回一个 `Future` → 下一层 `Service` 同样的流程 → 一路穿到业务最内层 → 响应 `Future` 一层层 resolve 回去——以及每一步底下,Tokio 的 `Future`/`Poll` 怎么被用起来(承接)、hyper 的 `Service` 怎么和 Tower 对接(承接)。

### 为什么用"执行单元 vs 组合单元"当骨架(而非按模块平铺)

Tower 的模块(`tower-service`/`tower-layer` 两个核心 trait crate + `tower` 这个中间件大集合)如果按模块百科式平铺,读者会觉得 timeout/retry/buffer/balance 互不相干。用"**执行单元(Service:`poll_ready` + `call` 把一个请求处理跑起来,含背压)vs 组合单元(Layer:把 Service 装饰成新 Service,洋葱层层套)**"这条主轴当骨架,每个模块都是"为了让中间件能组合/能背压而必须出现的角色":timeout 是个 Layer 包出来的 Service,retry 是个 Layer 包出来的 Service,balance/buffer/concurrency_limit 同理——它们都是同一套 Service×Layer 抽象的不同实例。

## 二、写作方法:动机(why) + 技巧(how)双线 ⚠️ 本系列命脉

> **这一节是本书与数学系列、其他源码系列最关键的方法论,所有写作者必读、必守。**

Tower 源码(Rust)里充斥高级技巧:`Service` trait 为什么是 `call(&mut self)` + `poll_ready(&mut self)` 而不是 `&self`(背压/资源预留/`mem::replace` 取走就绪服务的惯用法)、`Layer` 怎么用类型级洋葱(`Stack<T, L>` 编译期嵌套,零运行时开销)、`ServiceBuilder` 怎么把多层 `Layer` 链成一个 `Stack`、`Buffer` 怎么靠一个 worker task + mpsc 通道把 `!Clone` 的服务变成 `Clone + Send`(承 Tokio `mpsc`/`tokio::spawn`)、`RateLimit` 怎么用令牌桶 + `tokio::time` 实现、`ConcurrencyLimit` 怎么用 `tokio::sync::Semaphore`、`Balance` 怎么用 Power-of-2-Choices(P2C)做负载均衡、`Retry` 的 `Policy` trait 怎么决定重试、`LoadShed` 怎么把 `Pending` 翻成错误、`BoxCloneSyncService` 怎么类型擦除还能 `Clone + Sync`、`pin-project-lite` 怎么手写 Future 状态机……用户自己用过 Tower,"一知半解"的缺口往往**不在**"它要解决什么问题"(动机好懂),而**在**"它怎么用这么巧的 Rust 手段解决的"(技巧看不懂)。

2. **实现技巧(how 的精妙)**:Tower 用了什么 Rust 手段?为什么妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

## 三、全书主线、二分法、主比喻、承接

### 主线

> **把"处理一个请求"抽象成一个 `Service = poll_ready + call -> Future`,把"装饰请求处理"抽象成一个 `Layer = Fn(Service) -> Service`,两者拼出无限可组合的中间件——这是 Rust 异步生态的请求处理通用语言。**

任何一处看不懂某个机制,回到这句问:"这是在**执行单元**(Service:`poll_ready` 背压 + `call` 发请求 + `Future` 收响应),还是在**组合单元**(Layer:把一个 Service 装饰成新 Service、洋葱层层套、`ServiceBuilder` 编译期 `Stack`)?这个机制 Tokio/hyper 是怎么支撑的?"

### 二分法

本书的二分法:

> **执行单元(Service trait:`poll_ready(&mut self)` 背压 + `call(&mut self, req) -> Future`) vs 组合单元(Layer trait:`layer(&self, inner) -> Service` 洋葱装饰 + `ServiceBuilder`/`Stack` 类型级组合 + 生态中间件)。**

- **执行单元这一面**:`tower-service/src/lib.rs` 的 `Service<Request>` trait(`poll_ready`/`call`/`Future`,`&mut self` 语义,背压)、`ServiceExt`(`util/service.rs`,`oneshot`/`ready`/`map_response`/`and_then` 等组合子)、`Buffer`/`SpawnReady` 把 `poll_ready` 异步化的"让服务就绪"中间件、`load_shed` 把 `Pending` 翻成错误的取舍——这些决定"一个请求怎么被一个 Service 跑起来,以及背压怎么传"。
- **组合单元这一面**:`tower-layer/src/lib.rs` 的 `Layer<S>` trait + `Identity`/`Stack<L, Inner>`/`LayerFn`(类型级洋葱)、`tower/src/builder/` 的 `ServiceBuilder`(链式 `Stack` 嵌套)、`util/` 的 map_*/and_then/then 等装饰器、生态中间件(timeout/retry/limit/balance/buffer/hedge/filter/reconnect/discover/steer)各自的 `Layer`——这些决定"怎么把 Service 层层套成中间件栈"。

### 承接《Tokio》(承接铁律,务必遵守)

Tower 的 Service 基于 `Future`/`Poll`(标准库 `core::future`,`Future` 和 `Context`/`Poll` 都在 `core`,不在 tokio 仓),但大量中间件用 Tokio 的同步原语和 timer。**承接铁律**:Tokio 已讲透的,一句带过 + 指路 [[tokio-source-facts]],篇幅全留 Tower 独有。具体承接点:

- **Service 返回 Future**:`Service::call` 返回 `impl Future<Output = Result<Response, Error>>`,`Future`/`Poll`/`Waker` 是 `core::future`/`core::task` 标准库,Tokio 只是执行者。一句带过指路 [[tokio-source-facts]]。
- **Buffer 用 mpsc + spawn**:`tower::buffer` 把 `!Clone` 的服务变成 `Clone + Send`,靠一个 worker task + `tokio::sync::mpsc` 通道把请求转发给原服务。Tokio 的 `mpsc`/`tokio::spawn`/task 调度一句带过指路。
- **ConcurrencyLimit 用 Semaphore**:`tower::limit::concurrency` 用 `tokio::sync::Semaphore` 控并发数。Tokio 的 `Semaphore`/permit 语义一句带过指路。
- **RateLimit/Timeout/Hedge 用 tokio::time**:`tower::limit::rate` 的令牌桶、`tower::timeout`、`tower::hedge` 都用 `tokio::time`(Sleep/Interval)。Tokio 的时间轮(`runtime/time/wheel`)一句带过指路 [[tokio-source-facts]]。
- **budget 让出**:Tower 不自带 budget,承接 axum/hyper 里 Tokio budget=128 的协作式让出,一句带过。

**禁止把 Tokio 讲透的(reactor/scheduler/park/time wheel/task 状态位/Cell/mpsc 内部/Semaphore 内部)当新东西重复**——那是承接铁律的硬伤,违者返工。

### 承接《hyper》(本书最强承接,务必遵守)

hyper 的 `Service` trait 就是 `tower-service::Service` 的简化版。**承接铁律**:hyper P1-02/03 已讲透 Service 入门 + Tower 中间件链,一句带过 + 指路,篇幅全留 Tower 独有。具体承接点:

- **hyper Service = tower-service 简化版**:hyper 的 `service::Service`(`hyper/src/service/service.rs`)把 `tower-service::Service` 的 `poll_ready` **删了**——背压挪到 HTTP/1 的 `in_flight` 单槽 / HTTP/2 的 h2 流控 / client 的 `SendRequest::poll_ready`。本书要讲透**为什么 Tower 保留 `poll_ready` 而 hyper 删了**(协议层 vs 通用抽象层的取舍),这是招牌对照。
- **hyper 的 Tower 入门**:hyper P1-02 讲了 Service trait 把请求抽象成 Future,P1-03 讲了 Tower 与中间件链(对照 gRPC filter)。本书 P0-01/P1-02 复习 + 深化,不重复 hyper 讲过的,补全 Tower 全貌(背压语义、Layer 洋葱、生态中间件技巧)。
- **hyper 用 Tower**:`hyper::Service`/`hyper::body::Body` 与 Tower 的关系、axum 怎么用 Tower Layer 组装路由中间件,在附录 B 讲。

### 横连对照《gRPC》/《Envoy》/《Go》(跨语言对照)

- **gRPC filter stack**:gRPC C++ core 的 filter 链(《gRPC》第 4 篇招牌)和 Tower 的 Layer 洋葱是同一思想的不同语言实现——Tower 用 Rust 类型级 `Stack<T, L>` 编译期组合,gRPC 用 C++ 的 `Call`/`Interceptor` 链表运行期组合。对照讲"洋葱模型在两种语言里怎么落地"。
- **Envoy filter chain**:Envoy HCM(Network/HTTP filter chain,《Envoy》P3)也是洋葱,但 Envoy 是 C++ 运行期组装、Tower 是 Rust 编译期单态化。对照讲"零成本抽象 vs 运行期灵活"。
- **Go middleware**:Go 的 `func(http.Handler) http.Handler` 洋葱(如 chi/gin)是动态运行期链,Tower Layer 是静态编译期 `Stack`,对照讲"为什么 Tower 要把洋葱做进类型系统"。

### 主比喻决策:直球为主,比喻点睛(沿用系列)

> **直球为主**:绝大多数地方第一性原理直给,不套比喻。**比喻点睛**:仅在开篇(P0-01)用"洋葱"+"插座"做一次性点位睛——Service 是"插座上的电器"(每个电器干一件事:处理请求),Layer 是"洋葱皮"(一层层套在电器外面:timeout/retry/限流),ServiceBuilder 是"装修清单"(按顺序贴洋葱皮)。全书不建立贯穿重比喻。沿用《hyper》《RocksDB》《gRPC》定调。

### 第 0 篇 · 开篇:为什么 Rust 异步生态需要 Tower(定调,1 章)

- **第 1 章 · 第一性原理:为什么 Rust 异步生态需要 Tower**(`P0-01`):Tokio 提供了运行时、hyper 提供了 HTTP 协议,但"处理一个请求"在 Rust 生态里没有通用抽象——timeout/retry/限流/负载均衡每个框架各写一套。Tower 干的就是定义这套通用语言:`Service<Request> -> Future<Response>`(执行单元)+ `Layer`(组合单元)。承 Tokio(Future/Poll 是 `core`)、★强承 hyper(它的 Service 就是 Tower Service 的简化版,删了 `poll_ready`)、对照 gRPC filter stack / Envoy filter chain / Go 洋葱 middleware,点出 Tower 是 axum/tonic/reqwest/Pingora 的共同骨架。**【样本章节,全书风格标杆】**

### 第 1 篇 · 核心 trait:Service 与 Layer(3 章,Tower 灵魂)

### 第 2 篇 · 背压类中间件:让 poll_ready 有意义(3 章,执行招牌)

> `poll_ready` 是 Service 的灵魂,但很多底层服务(数据库连接、HTTP client)就绪是异步的。这一篇讲 Tower 怎么用中间件把 `poll_ready` 的复杂性藏起来。**建议顺序读**。

- **第 5 章 · Buffer:把 !Clone 服务变成 Clone+Send**(`P2-05`,★招牌章)——一个持数据库连接的 Service 不能 Clone,怎么在多 task 间共享?技巧:★**Buffer 的 worker task + mpsc 模型**(`tower/src/buffer/{service,worker,message}.rs`,承 Tokio mpsc/spawn,把请求经通道转发给唯一 worker,worker 持有原服务)、为什么这样能让 Service `Clone`(克隆的是 Sender 不是服务)、容量与背压、★0.5.0 修过的 off-by-one bug。源码:`tower/src/buffer/{mod,service,worker,future,layer,message,error}.rs`。归属:`执行(招牌)`。
- **第 6 章 · SpawnReady:后台预热就绪**(`P2-06`)——不想每次请求都阻塞在 `poll_ready` 上,怎么办?技巧:**SpawnReady**(`tower/src/spawn_ready/`),spawn 一个后台 task 反复 poll 内层服务直到 Ready,把"就绪"这件事从请求路径剥离、承 Tokio spawn、`tracing` span 传播。源码:`tower/src/spawn_ready/{mod,service,future}.rs`。归属:`执行`。
- **第 7 章 · LoadShed 与背压的取舍**(`P2-07`)——内层服务满载时,是等(Pending)还是直接拒绝?技巧:**LoadShed**(`tower/src/load_shed/`),把 `poll_ready` 的 `Pending` 翻成 `Overloaded` 错误,主动丢请求保系统、对照 Envoy overload manager(令牌桶 vs LoadShedPoint)。源码:`tower/src/load_shed/{mod,future,error,layer}.rs`。归属:`执行`。

### 第 3 篇 · 限流与超时类中间件:控住流量(3 章)

- **第 10 章 · RateLimit:令牌桶控速率**(`P3-10`)——怎么限制每秒请求数?技巧:**RateLimit**(`tower/src/limit/rate/`),令牌桶(token bucket)算法 + `tokio::time::Interval` 补充令牌、和 ConcurrencyLimit 的差别(并发 vs 速率)、为什么是 `poll_ready` 里扣令牌。源码:`tower/src/limit/rate/{mod,service,rate,layer}.rs`。归属:`执行`。

### 第 4 篇 · 韧性类中间件:失败时怎么办(3 章)

- **第 11 章 · Retry:失败重试与 Policy**(`P4-11`,★招牌章)——请求失败了要不要重试?技巧:★**Retry + Policy trait**(`tower/src/retry/{mod,policy,budget,backoff,future,layer}.rs`),`Policy::retry(&mut self, req, res)` 决定是否再试、★**Budget trait**(0.5.0 把 Budget 从结构体重构成 trait,允许自定义桶,TPS budget)、backoff(指数退避)、重试的 `call` 多次调用的语义、为什么重试要限预算(重试风暴)。源码:`tower/src/retry/{mod,policy,budget/{mod,tps_budget},backoff,future,layer}.rs`。归属:`执行(招牌)`。
- **第 12 章 · Hedge:对冲请求降尾延迟**(`P4-12`)——长尾请求(p99 慢)怎么治?技巧:★**Hedge**(`tower/src/hedge/{mod,latency,select,delay,rotating_histogram}.rs`),发一个请求,等它接近 p99 时再发第二个 hedge 请求,谁先回用谁、★**rotating_histogram**(`hdrhistogram` 滚动直方图估 p99)、对照 Envoy retry/hedge、为什么 hedge 要配 filter(避免无效 hedge)。源码:`tower/src/hedge/{mod,latency,select,delay,rotating_histogram}.rs`。归属:`执行(招牌)`。
- **第 13 章 · Reconnect:断线重连**(`P4-13`)——后端连接断了怎么自动重连?技巧:**Reconnect**(`tower/src/reconnect/`),基于 `MakeService`(工厂模式,`MakeService::Service`),把"造连接"和"用连接"分开,连接失败就再造一个、和 Buffer/balance 配合。源码:`tower/src/reconnect/mod.rs`、`tower/src/make/mod.rs`。归属:`执行`。

### 第 5 篇 · 路由与负载均衡类中间件:多个后端选一个(3 章,招牌)

> 这一篇是 Tower 区别于普通中间件库的地方:它有完整的服务发现 + 负载均衡栈。**建议顺序读**。

- **第 14 章 · Discover 与 ready_cache:服务发现**(`P5-14`)——后端列表怎么动态更新?技巧:**Discover**(`tower/src/discover/`,sealed trait alias for `TryStream<Item = Change>`,承 Tokio Stream)、**ready_cache**(`tower/src/ready_cache/`,缓存"已就绪"的服务、`indexmap` 维护)、`Change::Insert/Remove`。源码:`tower/src/discover/{mod,list}.rs`、`tower/src/ready_cache/mod.rs`。归属:`组合`。
- **第 15 章 · Balance 与 P2C:负载均衡**(`P5-15`,★招牌章)——多个后端怎么挑一个发请求?技巧:★**Power-of-2-Choices(P2C)**(`tower/src/balance/p2c/`),随机抽两个后端挑负载小的(为什么 P2C 比 round-robin/随机好)、★**load 模块**(`tower/src/load/`,`peak_ewma`/`pending_requests`/`constant` 三种负载度量,PEWMA 衰减历史)、`MakeBalance`/`BalanceLayer`、对照 Envoy LB(轮询/最少请求/环哈希)、为什么 P2C 用得最多。源码:`tower/src/balance/{mod,error,p2c/{mod,service,layer,make}}.rs`、`tower/src/load/{mod,peak_ewma,pending_requests,constant,completion}.rs`。归属:`执行(招牌)`。
- **第 16 章 · Steer 与 Filter:按规则分发**(`P5-16`)——不按负载,按请求内容(如 URL path)分发怎么做?技巧:**Steer**(`tower/src/steer/`,按 `Predicate` 把请求路由到一组服务之一)、**Filter**(`tower/src/filter/`,`Predicate`/`AsyncPredicate` 同步异步两种,过滤不合法请求)、和 Balance 的区别(路由 vs 均衡)。源码:`tower/src/steer/mod.rs`、`tower/src/filter/{mod,future,error,layer}.rs`。归属:`组合`。

### 第 6 篇 · 工程化:类型擦除与集成(3 章)

- **第 17 章 · BoxService 家族:类型擦除**(`P6-17`,★招牌章)——`ServiceBuilder` 套出来的类型是巨大嵌套的 `Stack<Timeout<Retry<...>>, ...>`,怎么擦除成统一类型?技巧:★**BoxService/BoxCloneService/BoxCloneSyncService**(`tower/src/util/boxed{,_clone,_sync}/`),用 trait object 把 Service 擦成 `Box<dyn Service>`、★为什么需要 `Clone + Sync`(axum 路由要 Clone,多线程要 Sync)、★0.5.0 让 `BoxService` 变 `Sync`、0.5.2 加 `BoxCloneSyncService`、`UnsyncBoxService` 历史包袱。源码:`tower/src/util/boxed/{mod,sync,unsync}.rs`、`tower/src/util/boxed_clone.rs`、`tower/src/util/boxed_clone_sync.rs`、`tower/src/util/boxed/{layer,layer_clone,layer_clone_sync}.rs`。归属:`组合(招牌)`。
- **第 18 章 · util 组合子与 service_fn:函数当 Service**(`P6-18`)——不想写一个 struct impl Service,直接用闭包/async fn 行不行?技巧:★**service_fn**(`tower/src/util/service_fn.rs`,把 `Fn(Request) -> Future` 包成 Service)、**map_request/map_response/map_err/and_then/then/map_result/map_future**(`tower/src/util/` 各文件,Service 版的 `Iterator` 组合子)、`Either`(两个 Service 二选一)、`optional`(Option<Service>)、`call_all`(Stream 喂 Service)。源码:`tower/src/util/{service_fn,and_then,then,map_err,map_request,map_response,map_result,map_future,either,optional,call_all,rng,future_service,ready,oneshot,boxed}.rs`。归属:`组合`。
- **第 19 章 · Tower 在 axum/tonic/hyper/Pingora 怎么落地**(`P6-19`)——真实框架怎么用 Tower?技巧:★**axum 的 `from_fn`/`Router::layer`**(axum 把 Handler 包成 Tower Service)、**tonic 的 `interceptor`/`Status`**(gRPC interceptor 是 Tower Layer 的特化)、**hyper 1.x 的 Service**(hyper Service 删了 poll_ready 的取舍)、**Pingora 的 proxy 模型**(Pingora 用 Tower 做 proxy filter)、附录 B 详细展开。源码:外部 crate(引用其用法,诚实标注)。归属:`总览`。
