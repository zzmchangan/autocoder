# 全书第一性原理(贯穿始终,反复回扣)

> **把"处理一个请求"抽象成一个 `Service = poll_ready(&mut self) + call(&mut self, req) -> Future`,把"装饰请求处理"抽象成一个 `Layer = Fn(Service) -> Service`,两者拼出无限可组合的中间件——这是 Rust 异步生态的请求处理通用语言。**

每写一章,先问自己:这一章讲的机制,是在**执行单元**(Service:`poll_ready` 背压 + `call` 发请求 + `Future` 收响应),还是在**组合单元**(Layer:把一个 Service 装饰成新 Service、洋葱层层套、`ServiceBuilder` 编译期 `Stack`)?这个机制 Tokio/hyper 是怎么支撑的、对照 gRPC/Envoy/Go 怎么做?把答案写进"为什么"里。读者要的不是"Tower API 怎么用",而是"它凭什么把请求处理抽象成可组合的 Future、源码里那些 Service trait、poll_ready 背压、Layer 洋葱、ServiceBuilder 类型级 Stack、Buffer worker task、Balance P2C、BoxCloneSyncService 到底在干什么"。

# ⚠️ 承接《Tokio》《hyper》铁律(本书最大特色,动手前必读)

Tower 的 Service 基于 `Future`/`Poll`(标准库 `core::future`/`core::task`,不在 tokio 仓),但大量中间件用 Tokio 同步原语和 timer;hyper 的 Service 就是 tower-service 的简化版(删了 poll_ready)。**承接铁律**:Tokio/hyper 已讲透的,一句带过 + 指路,篇幅全留 Tower 独有。

- **承接《Tokio》[[tokio-source-facts]]**:`Future`/`Poll`/`Waker`/`Context` 是 `core::future`/`core::task` 标准库、Buffer 用 `tokio::sync::mpsc` + `tokio::spawn`(worker task)、ConcurrencyLimit 用 `tokio::sync::Semaphore` permit、RateLimit/Timeout/Hedge 用 `tokio::time::Sleep`/`Interval`、Discover 用 `TryStream`、axum/hyper 里 Tokio budget=128 让出、task 状态位/Cell/mpsc 内部/Semaphore 内部/时间轮 `runtime/time/wheel`——这些《Tokio》已拆透,**一句带过 + 指路**,别当新东西重复。
- **承接《hyper》[[hyper-series-project]]**:hyper P1-02 已讲 Service trait 把请求抽象成 Future、P1-03 已讲 Tower 与中间件链(对照 gRPC filter)。★关键对照点贯穿全书:**hyper 的 `service::Service` 删了 `poll_ready`**(背压挪到 H1 `in_flight` 单槽/H2 h2 流控/client `SendRequest::poll_ready`),**Tower 的 `Service` 保留 `poll_ready`**——讲每章时这个对照点一句带过指路,篇幅留 Tower 全貌(背压语义/洋葱组合/生态中间件技巧)。
- **横连《gRPC》[[grpc-source-facts]]**:gRPC C++ filter stack 是运行期链表,Tower Layer 是 Rust 类型级 `Stack` 编译期组合,对照讲"洋葱模型两种语言落地"。
- **横连《Envoy》[[envoy-source-facts]]**:Envoy filter chain(Network/HTTP,HCM)C++ 运行期组装 + overload manager,LossT 做编译期单态化,对照"零成本抽象 vs 运行期灵活";LoadShed 对照 LoadShedPoint。
