# 《TiKV 设计与实现深入浅出:为什么能把一个 Raft 放大成百万个》—— 全书总纲

> 一本写给"读过 etcd/TiDB 源码、用过 TiKV,却总觉得一知半解"的人的小书。
>
> 本文是全书的**总纲**:定主线、定方法、定比喻、定分篇分章、定源码策略、定写作约定。
> 与《Tokio》《LevelDB》《etcd》《gRPC》《Linux 内核》同源,都是"**第一性原理 + 真实源码**";本书最大的特色是**三重承接**——它把读者已经学过的三本拧成一股:**Raft 承接《etcd》、单机引擎 RocksDB 承接《LevelDB》、RPC 层 gRPC 承接《gRPC》**;在此基础上,本书多两条命脉:① **multi-raft 工业级落地**——把 etcd 那个"一个 Raft 组"放大成"百万个 Raft 组",靠 FSM + batch-system actor 模型驱动;② **Percolator 分布式事务**——Raft 只保证单 Region 一致,跨 Region 的 ACID 事务靠 Percolator 两阶段提交拼出来。

## 一、这本书的定位

### 目标读者

读过 etcd 的 Raft、用过 TiDB、部署过 TiKV 集群,甚至翻过 `tikv/tikv` 源码,但**一知半解**:知道有 Region、raftstore、Peer、MVCC、Percolator、scheduler、PD,却讲不清一个 `INSERT` 从 TiDB 发出、经 PD 找到 Region、经 Raft 复制、经 MVCC 编码、经 Percolator 两阶段提交最终落盘的全过程;更讲不清 raftstore 里那些 FSM、batch-system、PeerFsm、ApplyFsm、RaftEngine、hibernate、async_io 到底在干什么。和"读过 etcd 源码但不理解 multi-raft"是同一种病。

### 全书一句话主旨

> **从 etcd 的"一个 Raft 组管全量 KV",到 TiKV 的"百万个 Raft 组分片管海量数据、再用 Percolator 跨分片拼出 ACID 事务"——TiKV 是怎么把一致性从"一组"扩展到"一百万组"、又怎么在跨组的情况下保证事务 ACID 的,每一步的设计动机和实现技巧都讲透。**

每个组件,都是这次旅程上的一个**驿站**。读完一本书,读者不是"多记了几个概念",而是能**在脑子里放映出一条写请求的全过程**:TiDB 把事务拆成 prewrite/commit 两次 gRPC 调用 → TiKV 的 `service/kv.rs` 收到 → `storage/txn/scheduler` 调度、过 latch → 落到对应 Region 的 raftstore → `PeerFsm` 发起 Raft 提议、`entry_storage` 写 RaftEngine、Raft 复制到其他副本 → commit 后 `ApplyFsm` 把命令 apply 到 RocksDB(MVCC 编码、写 default/write/lock 三个 CF)→ 返回。以及每一步底下用了什么巧妙的手段。

### 为什么用"一条写请求的旅程"当骨架(而非按模块平铺)

TiKV 的模块(`raftstore` / `engine_rocks` / `txn_types` / `storage/txn` / `pd_client` / `coprocessor`)高度耦合:一条写请求要被 scheduler 调度、被 latch 排队、被 raftstore 提议、被 Raft 复制、被 apply 到 RocksDB。按模块百科式平铺,读者会觉得每块都是孤岛、不知道**为什么要学这个**。用"一条写请求从 TiDB 到 TiKV 落盘再回来"这条旅程当骨架,每个模块都是"为了让旅程继续而必须出现的角色"——读者永远知道**我在哪、为什么需要下一站**。

## 二、写作方法:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与数学系列、其他源码系列最关键的方法论,所有写作者必读、必守。**

TiKV 是 Rust 写的大型系统,源码里充斥高级技巧:FSM + batch-system 的 actor 模型(一个线程池批量驱动百万 Peer)、Raft 日志的批量 propose/append、RaftEngine 专用日志引擎(替代存 RocksDB)、MVCC 的 key+ts 编码与三 CF 分离、Percolator 的 Primary Key 两阶段提交、scheduler 的 latch 行锁 + pipelined 写、async_io 读写分离、hibernate 空闲 Peer 休眠省 CPU、Region 分裂/迁移的动态再平衡……用户自己读过源码,"一知半解"的缺口往往**不在**"它要解决什么问题"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书的每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?(沿用系列一贯的"为什么优先")
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果不用这个技巧、朴素地写,会撞上什么墙",让妙处显形。

一句话:**讲不清 FSM/batch-system 怎么驱动百万 Peer、讲不清 Percolator 怎么跨 Raft 组拼出 ACID,等于没讲 TiKV。**

> **⚠️ 深度标杆(用户强调,本书命脉)**:本书篇幅标杆拉到对标《LuaJIT》深入浅出第一性原理版——**每章约 12000~18000 字**。用户原话:"不要吝啬文字和篇幅,内容可以更丰富一点,要把问题讲解清楚,深入浅出,好理解又有深度。"所以每个机制至少讲透三层(动机 → 反例 → 技巧 + 为什么 sound),宁可多铺反例、对比、图、选型历史,把问题讲到读者"原来如此",也不要一笔带过。这是本书区别于《gRPC》/《Tokio》(每章 8000~12000 字)的更高标杆。

## 三、全书主线、二分法、主比喻

### 主线

> **如何用成千上万个 Raft 组(multi-raft)分片扛住海量数据,每个 Region 一个 Raft 组保证"不丢不乱",再用 Percolator 跨 Region 拼出分布式 ACID 事务?**

任何一处看不懂某个机制,回到这句问:"这是在**让单个 Region 内部不丢不乱(复制层)**,还是在**让跨 Region 的事务 ACID(事务层)**?"

### 二分法

本书的二分法:

> **复制层(每个 Region 怎么不丢不乱) vs 事务层(跨 Region 怎么拼出 ACID)。**

- **复制层这一面**:Region 分片、multi-Raft(FSM + batch-system)、Raft 日志(RaftEngine)、Apply 到 RocksDB、Region 分裂/迁移/Snapshot——这些决定"数据怎么存、怎么不丢、怎么扩展"。
- **事务层这一面**:Percolator 两阶段提交(prewrite/commit/cleanup)、MVCC 编码(key+ts、default/write/lock 三 CF)、scheduler + latch、TSO 全局时间戳、悲观锁、GC——这些决定"跨 Region 的操作怎么 ACID"。

TiKV 每个模块都落在这两面之一(或两者衔接处:`storage/txn` 是事务层发起写、落到复制层 raftstore 的枢纽)。任何时候迷路,回到这个二分法。

### 主比喻决策:直球为主,比喻点睛(⚠️ 沿用系列定调,区别于 Tokio)

沿用《LevelDB》《etcd》《gRPC》起的系列定调:**直球为主**——绝大多数地方第一性原理直给,不套比喻。**比喻点睛**——仅在开篇(P0-01)用"一群快递员分片送包裹"做一次性点睛(片区 = Region、每个片区一个组长 = 一个 Raft 组的 leader、跨片区的连环单 = 跨 Region 事务要 Percolator 协调),全书不建立贯穿的重比喻体系。这是本书与《Tokio》(餐厅服务员全程贯穿)最显著的风格差异。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线是一条旅程:**单机扛不住 → 切 Region → multi-Raft 复制 → Raft 怎么落地 → 单机引擎 RocksDB+MVCC → Percolator 事务 → PD 协调 → 生产特性 → 收束**。
> 章号全局连续;命名 `P{篇}-{章}-标题.md`,开篇第 1 章 = `P0-01`,第 1 篇第 2 章 = `P1-02`,以此类推。

### 第 0 篇 · 开篇:为什么需要 TiKV(定调,1 章)

- **第 1 章 · 第一性原理:为什么需要 TiKV**(`P0-01`):单机数据库扛不住(数据量爆炸 + 单点故障不可接受)。etcd 的答案是"一个 Raft 组管全量 KV",但它撑不住海量数据(一个 Raft 组是吞吐瓶颈、分裂困难)。TiKV 的跃迁:**把数据按 key range 切成成千上万个 Region,每个 Region 一个独立的 Raft 组(multi-raft)**,于是吞吐随节点数线性扩展;但 Raft 只保证单 Region 一致,**跨 Region 的事务**(一次转账改两个 Region)怎么办?答案:**Percolator 两阶段提交**。点出本书三重承接(Raft←etcd、RocksDB←LevelDB、RPC←gRPC)与架构演进(raftstore-v2 / RaftEngine)。**【样本章节,全书风格标杆,见后】**

### 第 1 篇 · 地基:Region 与 multi-Raft(3 章)

> 先立起"被复制的单位"——Region,以及"怎么让百万个 Raft 组共存"。这一篇是全书地基。

- **第 2 章 · Region:把海量 key 切成一段段**(`P1-02`)——为什么不把全量数据塞一个存储,要切成 Region?Region(key range + 元数据)怎么编号、怎么路由。技巧:**key 编码**(`components/keys/`、`components/codec/`)、Region 元数据、Region 大小为什么是 256MB 量级(分裂/迁移的粒度权衡)。归属:`复制层`。
- **第 3 章 · Raft 库回顾与 multi-Raft 的挑战**(`P1-03`)——TiKV 的 Raft 用的是 `raft` crate(`rust-lang/raft`,即 etcd 的 Raft 算法的 Rust 移植),**算法本体承接《etcd》那本**。本章不讲 Raft 怎么选主/日志(那是 etcd 那本的活),专讲:**一个 Raft 库实例,怎么扩展成百万个?** 朴素做法(每 Region 一个 OS 线程跑 Raft tick)为什么不行(百万线程爆炸)。归属:`复制层`。
- **第 4 章 · batch-system + FSM:一个线程池驱动百万 Peer**(`P1-04`)——TiKV 的招牌解法:**把每个 Region 的副本抽象成一个 Peer,Peer 是一个 FSM(有限状态机);用 batch-system(actor 模型)让少量线程批量轮询百万个 PeerFsm**。技巧:**FSM + 批量消息**(一个线程一次 poll 一批 Peer 的消息,而非一个 Peer 一个线程)、`PeerFsm` / `StoreFsm` / `ApplyFsm` 三种 FSM 的分工。源码:`components/batch-system/src/`、`components/raftstore/src/store/fsm/{peer,store}.rs`。归属:`复制层(招牌)`。

### 第 2 篇 · 复制:raftstore 怎么把 Raft 跑起来(4 章)

> 这一篇是 TiKV 的心脏:百万个 Raft 组在一个进程里怎么真的转起来。

- **第 5 章 · raftstore 全貌:一条写请求的旅程**(`P2-05`)——一条写怎么从 `storage/txn` 到达 raftstore,再被 Raft 复制?技巧:**Propose → Append → Replicate → Commit → Apply** 五步流水线、Peer 怎么发起提议、Raft 消息怎么经 `transport.rs` 发往其他副本。源码:`components/raftstore/src/store/peer.rs`、`store/fsm/peer.rs`、`store/transport.rs`。归属:`复制层/衔接`。
- **第 6 章 · Raft 日志存储:RaftEngine**(`P2-06`)——Raft 日志存哪?**重要演进**:老版本 TiKV 把 Raft log 存在 RocksDB 里,新版本用专用的 **RaftEngine**(`components/raft_log_engine/`)独立存储。技巧:**为什么 Raft 日志要单独存**(顺序追加、高吞吐、回收快,LSM 不划算)、entry 存储、日志截断与回收。源码:`components/raftstore/src/store/entry_storage.rs`、`components/raft_log_engine/src/`。归属:`复制层(招牌)`。
- **第 7 章 · 异步 IO 与 hibernate:省 CPU 的两手**(`P2-07`)——百万 Peer 怎么不被 Raft tick 拖垮 CPU?技巧:**async_io 读写分离**(`store/async_io/`:把日志写/读挪到独立线程,不阻塞 Raft 推进)、**hibernate**(`store/hibernate_state.rs`:空闲 Region 的 leader 进入休眠,不发心跳、省 CPU,有事件再唤醒)。归属:`复制层`。
- **第 8 章 · Region 分裂、迁移与 Snapshot**(`P2-08`)——Region 太大要分裂、节点不均要迁移、加副本要传 Snapshot。技巧:**split**(split_check 算该不该切、切完两个 Region 各自独立 Raft 组)、**transfer leader + move region**(PD 调度)、**Snapshot**(`store/snap.rs`:给新副本传一份 SST 快照,不用回放全部历史日志)。源码:`store/worker/split_check.rs`、`store/snap.rs`、`coprocessor/split_*`。归属:`复制层`。

### 第 3 篇 · 单机引擎:RocksDB 与 MVCC 编码(3 章)—— 承接《LevelDB》

> Raft commit 的命令最终要落盘,落在 RocksDB 里。这一篇承接《LevelDB》那本的 LSM-tree。

- **第 9 章 · RocksDB 引擎:LSM-tree + Column Family**(`P3-09`)——TiKV 的单机引擎是 RocksDB(LevelDB 的工业级后代)。技巧:**Column Family**(TiKV 一个 RocksDB 实例开三个 CF:default 存 value、write 存提交记录、lock 存锁)、`engine_traits` 抽象(屏蔽 RocksDB/Engine 实现)、新特性 **in_memory_engine**(热数据放内存)。源码:`components/engine_rocks/`、`components/engine_traits/`、`components/in_memory_engine/`。归属:`复制层`。
- **第 10 章 · MVCC 编码:key + 时间戳**(`P3-10`)——多版本怎么存?技巧:**key 编码成 `key + ts`**(同一个 key 的多个版本在 RocksDB 里有序排列)、`txn_types`(MvccKey / Write / Lock / Data)、三个 CF 各存什么(default=value、write=提交记录指向 value、lock=未提交的锁)。源码:`components/txn_types/src/lib.rs`、`components/keys/`。归属:`事务层`。
- **第 11 章 · Apply 流水线:Raft 命令怎么落盘**(`P3-11`)——Raft commit 的命令怎么变成 RocksDB 的写?技巧:**ApplyFsm**(`store/fsm/apply.rs`)批量 apply、**cmd batch**(把多个已 commit 的命令攒一批一起 apply,提高 RocksDB 写吞吐)、apply 完回调通知 client。归属:`复制层/衔接`。

### 第 4 篇 · 事务:Percolator 两阶段提交(4 章)—— TiKV 最核心

> Raft 只保证单 Region 一致,但一个事务可能改多个 Region(跨多个 Raft 组)。怎么 ACID?答案:Percolator。这一篇是 TiKV 区别于"纯 KV 存储"的灵魂。

- **第 12 章 · 事务模型全景:scheduler + latch + 双引擎**(`P4-12`)——TiDB 是事务协调者,TiKV 是执行者。技巧:**scheduler**(`storage/txn/scheduler.rs`)怎么调度事务命令、**latch**(`storage/txn/latch.rs`)行级锁避免同一行并发冲突(单 Region 内的本地排队)、**乐观 vs 悲观双引擎**。源码:`storage/txn/{scheduler,latch}.rs`、`storage/txn/commands/`。归属:`事务层`。
- **第 13 章 · Prewrite 预写:选 Primary,加锁**(`P4-13`)——Percolator 第一阶段。技巧:**选一个 Primary Key**、给所有涉及 key 写 **lock**(lock CF)、value 先写 default CF 但不提交;Primary 是事务成败的"锚点"。源码:`storage/txn/actions/prewrite.rs`、`storage/txn/commands/prewrite.rs`。归属:`事务层(招牌)`。
- **第 14 章 · Commit 提交与 Secondary 清理**(`P4-14`)——Percolator 第二阶段。技巧:**Primary Key 先 commit**(write CF 写提交记录、清 lock)→ 事务"成功"了;**Secondary Key 异步清理**(不用等,后续读到 Secondary 的 lock 时按 Primary 状态决定是提交还是回滚)。源码:`storage/txn/actions/{commit,cleanup}.rs`。归属:`事务层(招牌)`。
- **第 15 章 · MVCC 读取与锁的解决**(`P4-15`)——读怎么找到可见版本?读到 lock 怎么办?技巧:**MVCC 读**(`storage/mvcc/reader/`:scanner 扫 write CF 找 ≤ start_ts 的最大版本)、**读到 lock**→ resolve_lock / check_txn_status(查 Primary 决定 wait 还是清锁)、死锁检测(悲观锁)。源码:`storage/mvcc/{txn.rs,reader/}`、`storage/txn/actions/{check_txn_status,resolve_lock,cleanup}.rs`、`src/server/lock_manager/deadlock.rs`。归属:`事务层(招牌)`。

### 第 5 篇 · 协调:PD 与全局时间戳(3 章)

> 事务需要全局序(MVCC 的 ts 从哪来)、Region 需要调度(谁当 leader、副本放哪)。这都是 PD(Placement Driver)的活。

- **第 16 章 · PD 的角色:TSO + 调度 + ID 分配**(`P5-16`)——PD 是 TiKV 集群的"大脑"。技巧:PD 用 Raft 自己保证高可用;**三个职责**:分配全局时间戳 TSO、调度 Region(balance)、分配 Region/Store ID。源码:`components/pd_client/src/`。归属:`事务层/衔接`。
- **第 17 章 · TSO:全局单调递增的时间戳**(`P5-17`)——MVCC 的版本号、事务的 start_ts/commit_ts 都从 TSO 来。技巧:**TSO 怎么做到全局单调**(PD leader 集中分配 + 批量预分配)、**resolved_ts**(`components/resolved_ts/`):算出"这个时间点之前的数据都稳定了"的安全点(给 GC 和读用)。归属:`事务层(招牌)`。
- **第 18 章 · Region 调度:balance 与热点**(`P5-18`)——数据不均、热点怎么自动再平衡。技巧:**balance leader/region**(把 leader/副本均匀分布)、**hot region**(识别热点 key 调度分散)、**split_controller**(自动分裂大 Region/热点 Region)。源码:`store/worker/{pd,split_controller,region}.rs`。归属:`复制层/衔接`。

### 第 6 篇 · 生产:Coprocessor、GC、悲观锁、CDC(3 章)

- **第 19 章 · Coprocessor:把计算下推**(`P6-19`)——为什么把 SQL 的过滤/聚合下推到 TiKV 执行?技巧:**计算下推**(省网络:只回传聚合结果而非全表数据)、`tidb_query_*`(下推的表达式/聚合执行器)、tipb(Coprocessor 的 protobuf 协议,走 gRPC)。源码:`src/coprocessor/`、`components/tidb_query_{expr,aggr,executors}/`。**承接《gRPC》那本**(Coprocessor 走 gRPC)。归属:`事务层/可观测`。
- **第 20 章 · GC 与 flashback:MVCC 老版本回收**(`P6-20`)——MVCC 会堆积海量老版本,怎么清?技巧:**GC safe point**(只保留 safe point 之后的版本)、**compaction filter**(RocksDB 压缩时顺便删过期版本)、**flashback**(新特性:闪回到某个 version)。源码:`src/server/gc_worker/`、`storage/txn/actions/{gc,flashback_to_version}.rs`。归属:`事务层`。
- **第 21 章 · 悲观锁与 CDC**(`P6-21`)——悲观事务怎么加锁、变更怎么捕获。技巧:**悲观锁**(acquire_pessimistic_lock 行锁 + lock_manager 死锁检测 + waiter_manager 锁等待队列)、**CDC**(`components/cdc/`:Change Data Capture,把 TiKV 变更实时推给下游)。源码:`storage/txn/actions/acquire_pessimistic_lock.rs`、`src/server/lock_manager/`、`components/cdc/`。归属:`事务层`。

### 第 7 篇 · 收束(1 章)

## 六、写作约定(沿用《gRPC》《Tokio》系列)

1. **动机 + 技巧双线**(本书命脉,见第二节):每个关键代码点讲"为什么这么设计" + "用了什么技巧、不这么写会怎样";每章单开"技巧精解"小节。
2. **"为什么"永远先于"怎么做"**:先讲它解决什么本质问题、不这样会怎样,再讲实现。
3. **大白话 + 比喻点睛**(直球为主):绝大多数地方第一性原理直给;仅在开篇用"一群快递员分片送包裹"做一次性点睛,不建立贯穿重比喻。
4. **关键路径配真实源码**:挑本章最核心的 1~2 处,贴真实片段、逐段讲。**技巧精解**固定在正文后、小结前。
5. **每章标准结构**:① 章首(核心问题 + 读完你会明白 + 难度高的加逃生阀);② 正文若干小节(招牌四段式:**提问 → 不这样会怎样 → 所以这样设计 → 源码/技巧佐证**);③ 技巧精解;④ 章末小结(回扣"复制层 vs 事务层"主线 + 五个为什么清单 + 想深入往哪钻 + 引出下一章)。
6. **回扣全局**:每章结尾回到二分法,说明本章服务"复制层"还是"事务层"那一面,衔接前后章。
7. **承接前作**:讲 Raft 时回扣《etcd》、讲 RocksDB 时回扣《LevelDB》、讲 RPC 时回扣《gRPC》,不重复造轮子,一句话带过 + 指路即可。
8. **语言**:中文,技术名词保留英文(Region、Raft、Peer、multi-raft、Percolator、MVCC、scheduler、latch、raftstore、CF、TSO、PD、Coprocessor、FSM 等)。
9. **标点**:**半角 ASCII 逗号 `,`、冒号 `:`、括号 `()`、问号 `?`;全角顿号 `、`、破折号 `——`、引号 `""`、句号 `。`**;中英文之间留空格。
10. **文件命名**:`P{篇}-{章}-标题.md`。第 0 篇第 1 章 = `P0-01-…`,第 2 篇第 5 章 = `P2-05-…`。
