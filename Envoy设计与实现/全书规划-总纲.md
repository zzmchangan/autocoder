# 《Envoy 设计与实现深入浅出:为什么把每条流量做成一串可热更新的 filter》—— 全书总纲

> 一本写给"天天在微服务里用 Envoy/Istio、却总觉得只知其表不知其里"的人的小书。
>
> 本文是全书的**总纲**:定主线、定方法、定比喻、定分篇分章、定源码策略、定写作约定。
> 与《Tokio》《gRPC》《TiKV》《Linux 内核》同源,都是"**第一性原理 + 真实源码**";本书最大的特色是**两个灵魂、一个范式**:**filter chain(数据面灵魂)+ xDS(控制面灵魂)= data plane / control plane 分离的服务网格范式**。本书多两条命脉:① **filter chain 精解**——Envoy 把"处理每一条流量"拆成一条可插拔的过滤器链(listener filter → network filter → HCM → http filter → router → cluster),这是它区别于"写死的代理"的根本;② **xDS 动态配置**——filter 怎么配、转发到哪,全由控制面通过 xDS 动态下发、热更新不停机。承接《gRPC》那本(HTTP/2/HPACK + xDS 是 Envoy 定义 gRPC 复用)。

## 一、这本书的定位

### 目标读者

部署过 Envoy 或 Istio、配过 listener/filter/cluster、看过 Envoy 的 YAML 配置,甚至翻过 `envoyproxy/envoy` 源码,但**一知半解**:知道有 listener、filter chain、cluster、xDS、worker,却讲不清一条 HTTP 请求从进 Envoy 到转给后端,中间穿过了哪些 filter、每个 filter 干什么、为什么是"链";更讲不清 xDS 怎么把配置从 Istio 动态推到每个 Envoy、还不停机热更新;以及 Envoy 的 worker 线程模型、thread-local 无锁、hot restart 这些招牌到底在干什么。和"用过 Nginx 但不懂它"是同一种病。

### 全书一句话主旨

> **从 Nginx 那种"配置写死、reload 才生效"的经典代理,到 Envoy 这种"每条流量穿过一串可热更新的 filter、配置由 xDS 动态下发"的可编程数据面——Envoy 是怎么用 filter chain + xDS 这两件套,撑起整个服务网格(Service Mesh)范式的,每一步的设计动机和实现技巧都讲透。**

每个组件,都是这次旅程上的一个**驿站**。读完一本书,读者不是"多记了几个 YAML 字段",而是能**在脑子里放映出一条流量在 Envoy 里的全过程**:请求进 listener → 穿过 listener filter(TLS 检查)→ 到 network filter / HCM → 解码成 HTTP → 穿过 http filter(鉴权/限流/故障注入)→ router 按 route_config 选 cluster → 负载均衡挑一个 endpoint → 经连接池转发 → 响应原路返回;同时,这一整套 filter 和路由表,是由控制面通过 xDS 动态下发、版本协商、热更新的。以及每一步底下用了什么巧妙的手段。

### 为什么用"一条流量的旅程"当骨架(而非按模块平铺)

Envoy 的模块(`listener` / `filter chain` / `HCM` / `cluster` / `load balancing` / `xDS` / `worker`)高度耦合:一条流量要从 listener 一路穿到 endpoint,中间每个 filter 都在前一个的输出上加工。按模块百科式平铺,读者会觉得每块都是孤岛、不知道**为什么要学这个**。用"一条流量从进到出"这条旅程当骨架,每个模块都是"为了让流量继续往前而必须出现的角色"——读者永远知道**我在哪、为什么需要下一站**。

## 二、写作方法:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与数学系列、其他源码系列最关键的方法论,所有写作者必读、必守。**

Envoy 是 C++ 写的大型系统(用 abseil/protobuf/bazel),源码里充斥高级技巧:每个 worker 一个 libevent dispatcher 的单线程事件循环、connection 绑定 worker + thread-local 无锁、filter chain 的责任链 + 解码/编码两向、零拷贝 Buffer(LibcSlice)、xDS 的 resource version 协商与 ACK/NACK、hot restart 零停机(新进程接管 socket)、overload manager 的令牌桶过载保护、ring hash / maglev 一致性哈希负载均衡、outlier detection 异常点检测……用户自己用过 Envoy,"一知半解"的缺口往往**不在**"它要解决什么问题"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书的每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?(沿用系列一贯的"为什么优先")
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果不用这个技巧、朴素地写,会撞上什么墙",让妙处显形。

一句话:**讲不清 filter chain 怎么把一条流量穿成链、讲不清 xDS 怎么动态热更新配置,等于没讲 Envoy。**

> **⚠️ 深度标杆(用户强调,本书命脉)**:本书篇幅标杆拉到对标《LuaJIT》深入浅出第一性原理版——**每章约 12000~18000 字**。用户原话:"不要吝啬文字和篇幅,内容可以更丰富一点,要把问题讲解清楚,深入浅出,好理解又有深度。"所以每个机制至少讲透三层(动机 → 反例 → 技巧 + 为什么 sound),宁可多铺反例、对比、图、选型历史(Envoy vs Nginx/HAProxy 演进、为什么 C++、为什么 filter chain + xDS),把问题讲到读者"原来如此",也不要一笔带过。这与《gRPC》《TiKV》两本保持同一标杆。

## 三、全书主线、二分法、主比喻

### 主线

> **如何把"处理每一条流量"做成一串可插拔的 filter(filter chain),把"配什么 filter、转发到哪"做成可动态下发的 xDS 配置(filter chain + xDS = service mesh 的 data plane)?**

任何一处看不懂某个机制,回到这句问:"这是在**处理每一条流量的数据面**(filter chain / worker 事件循环),还是在**动态下发与热更新配置的控制面**(xDS)?"

### 二分法

本书的二分法:

> **数据面(每条流量怎么被 filter 链处理) vs 控制面(配置怎么被 xDS 动态下发、热更新)。**

- **数据面这一面**:线程模型(worker + thread-local 无锁)、事件引擎(libevent dispatcher)、listener、listener filter、network filter、HCM(HTTP 解码)、http filter、router、cluster、负载均衡、连接池——这些决定"一条流量怎么从进到出"。
- **控制面这一面**:xDS 协议(LDS/RDS/CDS/EDS/SDS)、订阅与传输(grpc streaming / delta / ADS 聚合)、resource version 协商、listener drain 与热更新、hot restart——这些决定"配置怎么动态来、怎么不停机生效"。

Envoy 每个模块都落在这两面之一(或两者衔接处:`config/` + `config_subscription/` 是控制面驱动数据面 filter 的枢纽)。任何时候迷路,回到这个二分法。

### 主比喻决策:直球为主,比喻点睛(⚠️ 沿用系列定调)

沿用《LevelDB》《etcd》《gRPC》《TiKV》起的系列定调:**直球为主**——绝大多数地方第一性原理直给,不套比喻。**比喻点睛**——仅在开篇(P0-01)用"邮局分拣中心"做一次性点睛(邮件 = 流量、分拣传送带 = filter chain、分拣规则表 = xDS 动态下发的配置),全书不建立贯穿的重比喻体系。这是本书与《Tokio》(餐厅服务员全程贯穿)最显著的风格差异。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线是一条旅程:**传统代理的痛 → 线程模型与事件引擎 → downstream(listener/network filter)→ HTTP(HCM/http filter/router)→ upstream(cluster/LB)→ 控制面 xDS → 生产特性 → 收束**。
> 章号全局连续;命名 `P{篇}-{章}-标题.md`,开篇第 1 章 = `P0-01`,第 1 篇第 2 章 = `P1-02`,以此类推。

### 第 0 篇 · 开篇:为什么需要 Envoy(定调,1 章)

- **第 1 章 · 第一性原理:为什么需要 Envoy**(`P0-01`):Nginx 这类经典代理的痛——配置写死、reload 才生效、微服务海量流量难治理、跨语言难统一。Envoy 的回答:**filter chain(数据面:把处理流量做成可插拔的链)+ xDS(控制面:配置动态下发、热更新)**,data plane / control plane 分离,这就是 Service Mesh 范式。★对照 Nginx。承接《gRPC》(xDS 是 Envoy 定义、gRPC 客户端复用)。**【样本章节,全书风格标杆,见后】**

### 第 1 篇 · 地基:线程模型与事件引擎(3 章)

> 先搞清楚"谁来跑 filter chain"——Envoy 的线程模型和事件引擎。这一篇是全书地基。

- **第 2 章 · 线程模型:MainThread + N worker,thread-local 无锁**(`P1-02`)——Envoy 怎么用少量 worker 线程扛住海量连接?技巧:**每个 worker 一个独立 event loop、connection 绑定到一个 worker(不跨线程)**、**thread-local 存储**(stats 等热数据放 thread-local,worker 各自更新自己的副本,main 定期归并,无锁)。源码:`source/common/thread/`、`source/common/thread_local/`、`source/server/worker_impl.cc`。归属:`数据面`。
- **第 3 章 · 事件引擎:libevent dispatcher**(`P1-03`)——一个 worker 怎么同时处理几千个连接?技巧:**libevent/epoll 封装成 dispatcher**(注册 fd + 回调,epoll_wait 一次拿一批事件)、回调驱动的"反转控制"、定时器/延迟任务/信号都走 dispatcher。源码:`source/common/event/`(dispatcher.h, filed.h, timer.h)。归属:`数据面`。
- **第 4 章 · Buffer 与零拷贝**(`P1-04`)——流量在 filter 之间怎么传字节?技巧:**Buffer::Instance**(分片 slice 组成的字节流)、**零拷贝传递**(filter 之间传 buffer 的指针/引用,而非 copy)、**LibcSlice**。源码:`source/common/buffer/`。归属:`数据面`。

### 第 2 篇 · 数据面·Downstream:Listener 与 Network Filter(3 章)

> 流量进来的第一站:listener 接连接、listener filter 做 TCP 层处理、network filter 在字节流上加工。

- **第 5 章 · Listener:监听端口,接受连接**(`P2-05`)——Envoy 怎么监听端口、怎么把连接分给 worker?技巧:**SO_REUSEPORT**(每个 worker 都 listen 同一端口,内核做连接负载均衡到 worker)、**active listener**、listener drain(优雅下线)。源码:`source/server/active_listener_base.h`、`source/common/listener_manager/`、`source/server/instance_impl.cc`。归属:`数据面`。
- **第 6 章 · Listener Filter:TCP 层的过滤器**(`P2-06`)——在 HTTP 解码之前、TCP 层能做什么?技巧:**listener filter 链**(TLS inspect 探测是否 TLS、proxy_protocol 解析、original_dst 还原目的地址)、为什么要在 HTTP 之前做。源码:`source/extensions/filters/listener/`。归属:`数据面`。
- **第 7 章 · Network Filter 与 TCP 代理**(`P2-07`)——字节流层面怎么加工、怎么纯 TCP 代理?技巧:**network filter 链**(tcp_proxy 直接转发、ratelimit、echo)、字节流在 filter 间流转的机制。源码:`source/extensions/filters/network/`、`source/common/tcp_proxy/`。归属:`数据面`。

### 第 3 篇 · 数据面·HTTP:HTTP Connection Manager(4 章)—— Envoy HTTP 招牌

> 这是 Envoy 处理 HTTP 的核心:把 TCP 字节解码成 HTTP,再穿过 HTTP filter 链。

- **第 8 章 · HTTP Connection Manager(HCM)**(`P3-08`)——HCM 怎么把一串 TCP 字节变成结构化的 HTTP 请求?技巧:**HCM 作为特殊的 network filter**、编解码器(codec)插件化、请求/响应的"两向"处理。源码:`source/common/http/`(connection_manager_impl.cc)。归属:`数据面`。
- **第 9 章 · HTTP 编解码:HTTP/1.1、HTTP/2、HTTP/3**(`P3-09`)——三种 HTTP 版本怎么在 Envoy 里统一?技巧:**HTTP/1.1**(文本,逐行解析)、**HTTP/2**(二进制帧,**HPACK 承接《gRPC》那本**、流多路复用)、**HTTP/3**(基于 quiche/QUIC,UDP)。源码:`source/common/http/http1/`、`http/http2/`、`source/common/quic/`。归属:`数据面(承《gRPC》)`。
- **第 10 章 · HTTP Filter chain**(`P3-10`)——HTTP 请求怎么穿过滤滤器链?技巧:**http filter 链**(decoder 解码向 + encoder 编码向,两向链)、典型 filter(router、rate_limit、fault、compressor、wasm)、filter 的 stop/continue 语义。源码:`source/extensions/filters/http/`、`source/common/http/filter/`。归属:`数据面(招牌)`。
- **第 11 章 · Router 与路由匹配**(`P3-11`)——请求按什么规则转发到哪个 cluster?技巧:**route_config**(virtual host + route + cluster)、**matcher**(header/path 匹配)、加权路由、redirect/direct_response。源码:`source/common/router/`、`source/common/rds/`、`source/common/matcher/`。归属:`数据面`。

### 第 4 篇 · 数据面·Upstream:Cluster 与负载均衡(4 章)—— 转发到后端

> 选好 cluster 之后,怎么挑一个后端、怎么保证它健康、怎么过载保护。

- **第 12 章 · Cluster 与 Endpoint:后端集群与连接池**(`P4-12`)——cluster 怎么定义、endpoint 怎么发现、连接怎么复用?技巧:**cluster 类型**(static/strict_dns/logical_dns/eds/original_dst)、**连接池**(HTTP/1/2/3 各自的连接池,复用连接)。源码:`source/common/upstream/`、`source/extensions/clusters/`、`source/common/conn_pool/`。归属:`数据面`。
- **第 13 章 · 负载均衡:挑哪个后端**(`P4-13`)——一次请求在 cluster 里挑哪个 endpoint?技巧:**LB 策略**(round_robin / least_request / random / **ring_hash** 一致性哈希 / **maglev** / **subset** 子集)、**locality-aware**、为什么 ring_hash 能粘性。源码:`source/extensions/load_balancing_policies/`、`source/common/upstream/`。归属:`数据面(招牌)`。
- **第 14 章 · 健康检查与异常点检测**(`P4-14`)——怎么知道一个后端能不能接活?技巧:**active health check**(主动探活)、**outlier detection**(被动:根据失败率/延迟自动把后端踢出,过段时间再加回)、逐出与恢复。源码:`source/extensions/health_checkers/`、`source/common/upstream/outlier_detection`。归属:`数据面`。
- **第 15 章 · 断路器、重试、超时、过载保护**(`P4-15`)——后端扛不住怎么保护?技巧:**circuit breaker**(连接/请求上限熔断)、**retry**(重试优先级、重试预算)、**overload manager**(令牌桶,内存/压力过载时拒绝)、timeout。源码:`source/server/overload_manager_impl.cc`、`source/common/upstream/`、`source/extensions/retry/`。归属:`数据面`。

### 第 5 篇 · 控制面:xDS 动态配置(4 章)—— Envoy 灵魂

> 这一篇是 Envoy 区别于 Nginx 的根本:配置不是静态文件,而是控制面通过 xDS 动态下发、热更新。

- **第 16 章 · xDS 协议总览:LDS/RDS/CDS/EDS/SDS**(`P5-16`)——xDS 是什么、有哪几类?技巧:**五种 xDS**(Listener/Route/Cluster/Endpoint/Secret)、**resource version 协商**(控制面发新版本、数据面 ACK/NACK)、控制面与数据面的契约。源码:`source/common/config/`、`api/`(envoy service discovery proto)。归属:`控制面(招牌)`。
- **第 17 章 · xDS 订阅与传输:grpc streaming / delta / ADS**(`P5-17`)——xDS 怎么传?技巧:**SotW(State of the World)** vs **Delta xDS**(只发变更)、**ADS(Aggregated Discovery Service,聚合订阅)**(一个流拉所有类型,保证顺序)、grpc / file / rest 后端。源码:`source/extensions/config_subscription/`。**承接《gRPC》那本**(xDS 走 gRPC 双向流)。归属:`控制面(招牌)`。
- **第 18 章 · Listener 热更新:drain 与不停机生效**(`P5-18`)——listener 改了怎么不停机生效?技巧:**listener drain**(旧 listener 停止接新连接、等旧连接处理完)、**热更新**(新配置替换旧 filter chain,不影响在途流量)。源码:`source/common/listener_manager/`、`source/server/active_listener_base.h`。归属:`控制面`。
- **第 19 章 · Cluster/Endpoint 动态发现:CDS/EDS**(`P5-19`)——后端实例增减怎么实时反映?技巧:**CDS**(cluster 动态增减)、**EDS**(endpoint 动态发现,服务发现核心:新实例上线 / 下线秒级感知)。源码:`source/common/upstream/`、`source/extensions/config_subscription/`。归属:`控制面`。

### 第 6 篇 · 生产:可观测、安全、扩展(3 章)

- **第 20 章 · 可观测:stats、access log、tracing**(`P6-20`)——线上怎么观测 Envoy?技巧:**stats**(counter/gauge/**histogram** 分桶延迟统计,**thread-local 攒批归并**)、**access log**(结构化访问日志)、**分布式追踪**(request_id + trace 上下文透传)。源码:`source/common/stats/`、`source/common/access_log/`、`source/common/tracing/`。归属:`数据面/可观测`。
- **第 21 章 · 安全:mTLS、TLS 终止、SDS、hot restart**(`P6-21`)——跨网络怎么加密、怎么零停机更新?技巧:**mTLS**(双向 TLS,service mesh 的安全基石)、**TLS 终止**(在 Envoy 解密)、**SDS**(Secret 动态下发,证书轮换不重启)、**hot restart** 零停机重启(新进程通过 fd 传递接管 socket)。源码:`source/extensions/transport_sockets/`、`source/common/ssl/`、`source/common/secret/`、`source/server/hot_restart_impl.cc`。归属:`控制面/安全`。
- **第 22 章 · 扩展:Wasm filter 与 dynamic modules**(`P6-22`)——Envoy 怎么不被内置 filter 限死?技巧:**Wasm filter**(运行时加载 Wasm 字节码扩展 filter,隔离 + 动态)、**dynamic modules**(动态加载 C++ 模块)、自定义 filter 开发。源码:`source/extensions/wasm_runtime/`、`source/extensions/dynamic_modules/`。归属:`数据面/扩展`。

### 第 7 篇 · 收束(1 章)

- **第 23 章 · 全书收束:Envoy 与服务网格的范式**(`P7-23`)——filter chain + xDS 这套,得到了什么(动态、可编程、统一数据面)、付出了什么(filter 链开销、C++ 扩展门槛、配置复杂);**Envoy vs Nginx / HAProxy 对照总表**;service mesh(Istio/Linkerd)如何把 Envoy 当数据面、xDS 如何成为控制面与数据面的通用契约。归属:`总览`。

## 六、写作约定(沿用《gRPC》《TiKV》系列,LuaJIT 级标杆)

1. **动机 + 技巧双线**(本书命脉,见第二节):每个关键代码点讲"为什么这么设计" + "用了什么技巧、不这么写会怎样";每章单开"技巧精解"小节。
2. **"为什么"永远先于"怎么做"**:先讲它解决什么本质问题、不这样会怎样,再讲实现。
3. **大白话 + 比喻点睛**(直球为主):绝大多数地方第一性原理直给;仅在开篇用"邮局分拣中心"做一次性点睛,不建立贯穿重比喻。
4. **关键路径配真实源码**:挑本章最核心的 1~2 处,贴真实片段、逐段讲。**技巧精解**固定在正文后、小结前。
5. **每章标准结构**:① 章首(核心问题 + 读完你会明白 + 难度高的加逃生阀);② 正文若干小节(招牌四段式:**提问 → 不这样会怎样 → 所以这样设计 → 源码/技巧佐证**);③ 技巧精解;④ 章末小结(回扣"数据面 vs 控制面"主线 + 五个为什么清单 + 想深入往哪钻 + 引出下一章)。
6. **回扣全局**:每章结尾回到二分法,说明本章服务"数据面"还是"控制面"那一面,衔接前后章。
7. **承接前作**:讲 HTTP/2/HPACK 回扣《gRPC》、讲 epoll/异步回扣《Tokio》《Linux 内核》,不重复造轮子,一句话带过 + 指路即可。
8. **语言**:中文,技术名词保留英文(Envoy、filter chain、listener、HCM、cluster、endpoint、xDS、LDS/RDS/CDS/EDS/SDS、worker、dispatcher、overload manager、outlier detection、ring hash、hot restart 等)。
9. **标点**:**半角 ASCII 逗号 `,`、冒号 `:`、括号 `()`、问号 `?`;全角顿号 `、`、破折号 `——`、引号 `""`、句号 `。`**;中英文之间留空格。
10. **文件命名**:`P{篇}-{章}-标题.md`。第 0 篇第 1 章 = `P0-01-…`,第 3 篇第 8 章 = `P3-08-…`。
