# 第 4 篇 · 第 14 章 · 健康检查与异常点检测

> **核心问题**:第 13 章讲负载均衡时,反复出现一个前提——"LB 只从 healthy host 里挑"。可 Envoy 怎么知道一个 endpoint 能不能接活?一个后端进程,OS 进程还活着、TCP 端口还在 listen、连上去甚至能握手,但应用层已经卡死、响应 5xx、延迟飙到秒级——这种"半死不活"的状态,LB 若照样把流量打过去,真实用户就遭殃了。Envoy 的回答是两套互补的机制:**active health check(主动探活)**——Envoy 周期性自己发探测请求,据响应标 healthy/unhealthy;**outlier detection(异常点检测,被动)**——不主动探测,而是观察真实业务请求的结果(失败率、5xx、延迟),把统计上偏离正常的后端临时踢出(eject)。一个慢周期兜底、一个实时反应,合起来构成 upstream 的健康治理。

> **读完本章你会明白**:
> 1. 为什么不能"只靠真实请求失败来发现坏后端"——慢、影响真实用户、卡死在应用层的后端 OS 层探不到。active health check 用 Envoy 自己周期性发探测,绕开"等真实请求撞墙"。
> 2. 为什么不能"只靠 active 探活"——探活周期内的突发故障抓不到、探活路径可能与真实请求路径不同。outlier detection 蹲在真实请求的结果流上,统计上偏离正常就踢出。
> 3. outlier detection 的三种触发(consecutive_5xx / success_rate / failure_percentage)各自的统计算法、为什么 success_rate 用"均值减 N 倍标准差"这种统计学阈值而不是固定阈值。
> 4. "踢出(eject)"不是删除,是临时把 host 排除出 LB 候选;过 `base_ejection_time × 倍数` 后自动加回;连续被踢的 host 倍数翻倍(退避),防抖动。
> 5. 两者怎么配合:active 探活成功可以提前把被 outlier 踢出的 host 救回来(successful_active_health_check_uneject_host)。

> **如果一读觉得太难**:先只记住三件事——① **active health check** 是 Envoy 周期性自己发探测(HTTP GET /health、TCP 连接、gRPC health.proto),据响应标 host 的 `FAILED_ACTIVE_HC` flag;② **outlier detection** 是蹲在真实请求结果上,失败率/连续 5xx 偏离正常就把 host 临时踢出(设 `FAILED_OUTLIER_CHECK` flag),过段时间自动加回;③ LB 只从两个 flag 都没设的 host 里挑——本章就是"这两个 flag 怎么被设上、怎么被清掉"。

---

## 〇、一句话点破

> **active health check 是 Envoy 主动去"敲敲门"问后端还在不在,outlier detection 是 Envoy 蹲在真实流量的结果流上"看谁老出错就先晾一边";前者慢周期兜底、后者实时反应,两者都只是给 host 贴/撕健康 flag,真正"不用谁"是负载均衡在第 13 章里干的事。**

这是结论,不是理由。本章倒过来拆:先讲"不探活、只等真实请求失败"会撞什么墙,引出 active;再讲"只靠 active"又会漏掉什么,引出 outlier;然后分别拆 active 的协议实现和 outlier 的三种统计算法;最后拆两者怎么配合、eject 的退避倍增怎么防抖动。

---

## 一、为什么需要"知道后端能不能接活":两个朴素方案各自的墙

第 13 章讲 LB 时,所有策略(round_robin、least_request、ring_hash、maglev……)都有一个共同前提:**只从 healthy host 里挑**。但"healthy"这个判定从哪来?这是本章的核心。先看两个朴素方案各自撞的墙,你就明白为什么需要 active + passive 两套。

### 朴素方案一:不主动探活,等真实请求失败再说

最省事的做法:Envoy 不主动探活,直接按 LB 把请求打过去。如果后端挂了,请求自然失败(连接拒绝、超时、5xx),Envoy 把这个失败的 host 标记一下,下次不再选它。

这听起来简单,但在生产里撞三道墙:

**第一道墙:慢,而且代价由真实用户承担。** 假设一个后端实例在 10:00:00 完全挂掉(进程僵死,不再响应)。Envoy 若不主动探活,只能等下一个真实请求撞上去才知道。假设这个 cluster 的 QPS 是每秒 100,均匀分布到 10 个 host,那大约 0.1 秒后就有真实请求撞到这个坏 host——听起来很快?但这一撞,就是**一个真实用户的请求失败了**。如果挂掉的是个关键路径(支付、下单),这个失败的用户体验就是真金白银的损失。更糟的是,如果故障是"间歇性的"(后端一半的请求超时),LB 可能要撞好几次、攒够失败样本,才意识到这个 host 不行——这几分钟里,源源不断的真实用户被分配到坏 host 上。

> **不这样会怎样**:不主动探活,意味着"发现坏后端"这件事的代价,全部由**真实业务流量**承担。后端什么时候挂、挂多严重,Envoy 只能通过"牺牲一批真实请求"来知道。这在 SLA 敏感的场景(支付、网关入口)是不可接受的——你希望的是"后端一挂,Envoy 马上知道,根本不让真实用户撞上"。

**第二道墙:OS 层活着 ≠ 应用层活着。** 后端进程的"挂掉"有多个层次:

- 最彻底:进程退出,OS 不再有这个进程,端口 listen 套接字关闭。这种 `connect()` 直接 `ECONNREFUSED`,最容易发现。
- 半死:进程还在,端口还在 listen,`connect()` 能成功,但应用层(比如 HTTP handler)已经卡死——可能是死锁、内存溢出、依赖的下游卡住、GC 长停顿。这种 host,纯靠 TCP 连接成功是发现不了的,得真发个 HTTP 请求看它回不回。
- 间歇故障:大部分时候正常,偶发 5xx 或延迟飙升。这种最难,单次探测发现不了,得看统计。

对第二种"半死"状态,如果 Envoy 不主动发应用层探测,就只能等真实用户发请求撞上去——而且这个"撞"可能不是立即失败(连接是成功的),而是请求发出去后**长时间挂起直到超时**,占用 Envoy 的连接和工作线程,放大故障面。

**第三道墙:故障的"发现"和"恢复"都不及时。** 一个 host 挂了,Envoy 靠真实请求失败发现——慢。一个 host 恢复了(比如重启好了),Envoy 怎么知道可以重新往这里打流量?如果靠"再发真实请求试试",那第一个试的请求又是"代价由真实用户承担"。没有主动探活,故障的**恢复**检测同样迟钝。

> **钉死这件事**:不主动探活,等于让真实业务流量当"探测器"——发现故障慢、恢复检测慢、代价全由用户承担。这就是为什么需要 active health check:Envoy **自己**周期性发探测,**自己**承担探测成本,**不把发现故障的代价转嫁给真实请求**。

### 朴素方案二:只靠 active 探活,定期敲敲门就行

既然主动探活能解决"不转嫁用户"的问题,那是不是配上 active health check 就够了?也不行。active 也有它的墙:

**第一道墙:周期内的突发故障抓不到。** active health check 有个 `interval`(默认 1 秒到几秒,看配置)。如果探活间隔是 5 秒,那么在两次探活之间的 5 秒内,后端发生的任何故障,active 都要等下一次探活才知道。一个后端在 t=2s 突然开始 100% 失败,active 要到 t=5s 的下一次探活才发现——这中间 3 秒,真实流量照样被 LB 打到这个坏 host 上(因为 active 还没标它 unhealthy)。

**第二道墙:探活路径 ≠ 真实请求路径。** active 探活通常打一个专门的 `/health` 端点,这个端点往往是应用专门为健康检查写的轻量接口(可能只返回 200,不走数据库、不走业务逻辑)。问题来了:真实业务请求走的是 `/api/order` 这种重路径,要查数据库、调下游——这些环节挂了,`/health` 端点照样返回 200,active 探活根本发现不了。这种"健康检查通过、业务实际挂了"的假阳性,是纯 active 方案的盲区。

> **不这样会怎样**:只靠 active,突发故障有"探活间隔"的盲区(秒级),而"探活端点健康但真实业务挂了"这种深层故障,active 可能完全看不见。你需要一双盯着**真实业务请求结果**的眼睛——这就是 outlier detection。

### 所以:active 兜底,outlier 实时,两者互补

Envoy 的答案是**两套并用**:

- **active health check(主动探活)**:Envoy 周期性(默认间隔可配,常见 1~5 秒)主动向后端发探测请求(HTTP `GET /health`、TCP 连接、gRPC health check proto 等)。这是"Envoy 自己敲门",代价由 Envoy 承担,能发现"OS 层挂掉、端口连不上"这种彻底故障,**慢周期兜底**。
- **outlier detection(异常点检测,被动)**:Envoy 不主动探测,而是**蹲在真实业务请求的结果流上**——每个真实请求结束(成功/失败/超时),router filter 都会把结果报给 outlier detector(`putResult`)。detector 统计每个 host 的失败率、连续 5xx、延迟,一旦某个 host **统计上显著偏离正常**,就把它临时踢出(eject)。这是**实时反应**(每个请求都看),能抓到 active 盲区里的突发故障、深层故障。

一个慢周期兜底(发现彻底挂掉的),一个实时反应(发现慢/间歇故障的),两者覆盖了"后端能不能接活"这个问题的不同侧面。

```
   后端健康判定的两个维度
   ┌─────────────────────────────────────────────────────────┐
   │                                                         │
   │   active health check (主动)        outlier detection (被动)│
   │   ─────────────────────              ───────────────────│
   │   Envoy 周期性自己发探测              蹲在真实请求结果流上   │
   │   代价: Envoy 自己承担               代价: 零(复用真实流量)│
   │   发现: 彻底挂掉(连不上)             发现: 慢/间歇/深层故障 │
   │   速度: 秒级(受 interval 限制)       速度: 每请求即报      │
   │   路径: 专门探活端点                 路径: 真实业务请求     │
   │                                                         │
   │   ↓ 设 FAILED_ACTIVE_HC flag         ↓ 设 FAILED_OUTLIER_CHECK│
   │                                                         │
   │   LB (第13章) 只从两个 flag 都没设的 host 里挑            │
   └─────────────────────────────────────────────────────────┘
```

> **钉死这件事**:active 和 outlier 不是二选一,而是**互补**。active 解决"不转嫁用户、发现彻底故障",outlier 解决"实时性、抓深层故障"。生产配置里两者通常一起开。理解了这一点,后面拆各自的实现就有方向了。

## 二、active health check:Envoy 周期性自己敲门

先拆 active。它的核心是:Envoy 给 cluster 里每个 host 维护一个**探活会话(ActiveHealthCheckSession)**,周期性发探测,据响应把 host 的 `FAILED_ACTIVE_HC` flag 设上或清掉。

### 探活的协议多样性:HTTP / TCP / gRPC / Redis / Thrift

后端协议不同,探活方式自然不同。Envoy 把探活做成**可扩展的 health checker**,每种协议一个实现,都在 `source/extensions/health_checkers/` 下:

- **HTTP**([`http/health_checker_impl.cc`](../envoy/source/extensions/health_checkers/http/health_checker_impl.cc)):周期性发一个 HTTP 请求(比如 `GET /health`),看响应码是否落在 `expected_statuses` 配置的范围(默认 200)。
- **TCP**([`tcp/health_checker_impl.cc`](../envoy/source/extensions/health_checkers/tcp/health_checker_impl.cc)):建立 TCP 连接,可选地发一段字节(`send`)、收一段字节看是否匹配(`receive`)。最简单的形式只要 `connect()` 成功就算健康。
- **gRPC**([`grpc/health_checker_impl.cc`](../envoy/source/extensions/health_checkers/grpc/health_checker_impl.cc)):用标准的 `grpc.health.v1.HealthCheck` proto 发 `Check` 请求,看返回的 `ServingStatus` 是不是 `SERVING`。
- **Redis / Thrift**:同理,各自用协议特有的方式探活。

所有这些实现,都继承自一个公共基类 [`HealthCheckerImplBase`](../envoy/source/extensions/health_checkers/common/health_checker_base_impl.h#L44),它负责**所有协议共性的逻辑**:周期定时器、超时定时器、healthy/unhealthy threshold 计数、健康 flag 切换、回调通知。子类只需要实现"怎么发探测、怎么判定这一次成功还是失败"。

> **所以这样设计**:探活的"周期调度 + threshold 状态机 + flag 切换"是协议无关的,抽到基类;每种协议只实现自己特有的"发什么、怎么判"。这是经典的模板方法模式(template method)——基类定骨架,子类填协议细节。**不这么写会怎样**:每个协议都重写一遍 threshold 状态机和定时器调度,逻辑重复、容易出 bug,而且改一个共性问题(比如加 jitter)要改 N 个文件。

### threshold 状态机:不是一次失败就标 unhealthy

active 探活不是"一次失败就立刻标 unhealthy",而是有 **healthy/unhealthy threshold** 两个阈值,要连续多次才能切换状态。看基类的状态机:

- host 当前 healthy,某次探活失败:不立刻标 unhealthy,而是 `num_unhealthy_++`;只有当 `num_unhealthy_` 达到 `unhealthy_threshold_`(默认 2~5,看配置)才设 `FAILED_ACTIVE_HC` flag、标 unhealthy。中间几次是 `ChangePending`(还没到阈值)。
- host 当前 unhealthy,某次探活成功:同理,`num_healthy_++`,达到 `healthy_threshold_` 才清 `FAILED_ACTIVE_HC` flag、标 healthy。

这是 [`handleFailure`](../envoy/source/extensions/health_checkers/common/health_checker_base_impl.cc#L419) 和 [`handleSuccess`](../envoy/source/extensions/health_checkers/common/health_checker_base_impl.cc#L295) 里的核心逻辑。截一段 `setUnhealthy`(被 `handleFailure` 调用)的源码:

```cpp
// source/extensions/health_checkers/common/health_checker_base_impl.cc:369 (简化示意,非源码原文)
HealthTransition ActiveHealthCheckSession::setUnhealthy(...) {
  num_healthy_ = 0;
  HealthTransition changed_state = HealthTransition::Unchanged;
  if (!host_->healthFlagGet(Host::HealthFlag::FAILED_ACTIVE_HC)) {
    if ((!networkHealthCheckFailureType(type) && !retriable) ||
        ++num_unhealthy_ == parent_.unhealthy_threshold_) {
      host_->healthFlagSet(Host::HealthFlag::FAILED_ACTIVE_HC);  // 达到阈值才标
      parent_.decHealthy();
      changed_state = HealthTransition::Changed;
    } else {
      changed_state = HealthTransition::ChangePending;  // 还没到阈值,挂起
    }
  }
  ...
}
```

> **不这样会怎样**:为什么不用"一次失败就标 unhealthy"?因为探活可能偶发抖动——网络瞬断、后端 GC 停顿、慢请求导致探活超时。一次失败就标 unhealthy,会让健康的 host 频繁进出 unhealthy 状态,引发 LB 抖动、连接重建风暴。threshold 的本质是**给 host 一个"缓冲"**,要连续多次(说明是稳定故障,不是偶发抖动)才动它。同理,恢复也要连续多次成功,避免一次偶然的成功就把还没真好的 host 加回。这是所有健康检查系统的通用设计(Nginx upstream、Spring Boot actuator 都有类似阈值)。

注意源码里一个细节:`networkHealthCheckFailureType(type) && !retriable` 的失败(网络层失败,不可重试)**绕过阈值、立刻标 unhealthy**。这是合理的——网络层直接连不上的 host(`ECONNREFUSED`),基本是进程没了,没必要等阈值。而应用层的失败(比如 HTTP 503),可能是偶发,要走阈值。

### 一次探活的完整时序

把基类的调度逻辑和 HTTP 子类的探测逻辑串起来,一次 HTTP 探活的完整时序是这样的:

```mermaid
sequenceDiagram
    participant T as interval_timer
    participant S as ActiveHealthCheckSession
    participant H as HTTP子类(onInterval)
    participant U as Upstream Host
    participant B as handleSuccess/Failure
    T->>S: onIntervalBase() (周期到)
    S->>H: onInterval() 发 HTTP GET /health
    S->>S: 启动 timeout_timer
    H->>U: 经连接池发请求
    U-->>H: HTTP 响应 (200 / 5xx / 超时)
    alt 响应成功
        H->>B: handleSuccess()
        B->>B: num_healthy_++; 达 healthy_threshold 清 FAILED_ACTIVE_HC
    else 响应失败或超时
        H->>B: handleFailure(ACTIVE/TIMEOUT)
        B->>B: num_unhealthy_++; 达 unhealthy_threshold 设 FAILED_ACTIVE_HC
    end
    B->>T: enableTimer(下一次 interval, 按 healthy/unhealthy 调)
```

注意时序图最后一步:`interval` 不是固定的,可以根据 host 当前是 healthy 还是 unhealthy 切换不同的间隔(`healthy_interval` / `unhealthy_interval`,见基类 [`interval()`](../envoy/source/extensions/health_checkers/common/health_checker_base_impl.h#L139))。这给了一个有用的策略:**对已经 unhealthy 的 host 探活更频繁**,以便更快发现它恢复;**对 healthy 的 host 探活更稀疏**,省资源。

### first_check 快速路径:首次探活成功立即转 healthy

看 [`handleSuccess`](../envoy/source/extensions/health_checkers/common/health_checker_base_impl.cc#L305) 里这个分支:

```cpp
if (host_->healthFlagGet(Host::HealthFlag::FAILED_ACTIVE_HC)) {
  // 如果是这个 host 第一次拿到探活结果,立刻转 healthy,加速启动
  if (first_check_ || ++num_healthy_ == parent_.healthy_threshold_) {
    host_->healthFlagClear(Host::HealthFlag::FAILED_ACTIVE_HC);
    ...
  }
}
```

Envoy 里每个 host 初始带 `PENDING_ACTIVE_HC` flag(还没探过活)。第一次探活(`first_check_ == true`)只要成功,就**绕过 healthy_threshold、立刻转 healthy**。这是个启动加速:新上线一批 host,如果都要老老实实攒够 `healthy_threshold` 次成功才转 healthy,启动会慢。first_check 快速路径用"轻微牺牲可靠性换启动速度"——新 host 通常本来就是健康的(刚启动),第一次探活成功就信它。

> **钉死这件事**:active health check 的本质是——Envoy 给每个 host 维护一个探活会话,周期性(可配 interval、jitter、按健康状态切间隔)发协议特有的探测,据响应在 **threshold 状态机**里推进,达到阈值才设/清 `FAILED_ACTIVE_HC` flag。整套机制协议无关的骨架在基类 `HealthCheckerImplBase`,协议特有的探测细节在各子类。

## 三、outlier detection:蹲在真实请求结果流上的统计警察

active 讲完,现在拆本章的招牌——outlier detection(异常点检测)。它和 active 最大的不同:**不主动探测,而是观察真实业务请求的结果**。

### 数据从哪来:每个真实请求结束都"上报"

outlier detector 的输入,不是 Envoy 主动发的探测,而是**真实业务请求的结果**。具体来说,router filter(第 11 章)在每次请求结束时,都会调用 host 的 outlier detector 的 `putResult` 方法上报这一次的结果。看真实的调用点:

```cpp
// source/common/router/router.cc:1505 (简化示意,非源码原文)
upstream_request.upstreamHost()->outlierDetector().putResult(result, code);
```

`putResult` 接收一个 `Result` 枚举(`LocalOriginTimeout`、`LocalOriginConnectFailed`、`ExtOriginRequestFailed`、`ExtOriginRequestSuccess` 等)和一个可选的 HTTP code。tcp_proxy、redis_proxy、thrift_proxy、udp_proxy 等所有转发 filter 都在各自的关键点上调 `putResult`——grep 一下能找到十几处上报点([`router.cc`](../envoy/source/common/router/router.cc#L1505)、[`tcp_proxy.cc`](../envoy/source/common/tcp_proxy/tcp_proxy.cc#L1070)、[`redis/client_impl.cc`](../envoy/source/extensions/filters/network/common/redis/client_impl.cc#L259) 等)。

> **所以这样设计**:outlier detector 不自己产生流量,而是**寄生在真实业务流量的结果流上**。代价几乎为零(每个请求多调一次 `putResult`,就是几个原子自增),收益是看到了**真实请求的真实表现**——这正是 active 探活(打专门的 `/health` 端点)看不到的。**不这么设计会怎样**:如果 outlier 自己发探测,就重复了 active 的工作,而且探测路径还是和真实请求不同(回到第二节的"探活路径 ≠ 真实请求路径"问题)。寄生在真实流量上,是 outlier 能抓"深层故障"的根本。

### `putResult` → `putHttpResponseCode`:统一成 HTTP code 视角

outlier detector 内部把所有结果统一映射成 HTTP code 来统计。看 [`putResultNoLocalExternalSplit`](../envoy/source/common/upstream/outlier_detection_impl.cc#L145) 的逻辑:它先把 `Result` 通过 [`resultToHttpCode`](../envoy/source/common/upstream/outlier_detection_impl.cc#L108) 映射成 HTTP code(比如 `LocalOriginTimeout` → 504、`ExtOriginRequestFailed` → 500、`ExtOriginRequestSuccess` → 200),再调 [`putHttpResponseCode`](../envoy/source/common/upstream/outlier_detection_impl.cc#L79)。

为什么统一成 HTTP code?因为 outlier 的统计逻辑(连续 5xx、成功率)都是围绕 HTTP code 设计的。这样不管上层是 HTTP、TCP、Redis 还是 Thrift,内部都能用同一套统计引擎。`putHttpResponseCode` 是统计的核心入口:

```cpp
// source/common/upstream/outlier_detection_impl.cc:79 (简化示意,非源码原文)
void DetectorHostMonitorImpl::putHttpResponseCode(uint64_t response_code) {
  external_origin_sr_monitor_.incTotalReqCounter();   // 总请求数 +1
  if (Http::CodeUtility::is5xx(response_code)) {
    // 5xx: 累加连续 5xx 计数,达到阈值就触发 eject
    if (++consecutive_5xx_ == detector->config().consecutive5xx()) {  // 默认 5
      detector->onConsecutive5xx(host_);
    }
    // 顺便处理 gateway failure (502/503/504)
    ...
  } else {
    // 非 5xx: 成功,清零连续 5xx,累加成功计数
    external_origin_sr_monitor_.incSuccessReqCounter();
    consecutive_5xx_ = 0;
  }
}
```

这里就藏着 outlier 的第一种触发:**consecutive_5xx**(连续 N 次 5xx)。每收到一个 5xx,`consecutive_5xx_` 自增;一旦达到配置的阈值(默认 5,见 [`DEFAULT_CONSECUTIVE_5XX`](../envoy/source/common/upstream/outlier_detection_impl.h#L368)),就调 `onConsecutive5xx` 触发 eject。中间只要夹一个非 5xx,计数清零——和 active 的 threshold 一样,要"连续"才动,过滤偶发抖动。

### 三种触发:consecutive_5xx / success_rate / failure_percentage

outlier detection 不只有 consecutive_5xx。看配置类 [`DetectorConfig`](../envoy/source/common/upstream/outlier_detection_impl.h#L301) 和检测统计宏 [`ALL_OUTLIER_DETECTION_STATS`](../envoy/source/common/upstream/outlier_detection_impl.h#L229),一共三种(加 local origin 变体是六种)触发:

**1. consecutive_5xx(连续 5xx)**:某个 host 连续返回 N 次 5xx(默认 N=5),立即 eject。这是**单 host 内部、即时**的触发——不和别人比,自己连续出错就踢。源码就在上面的 `putHttpResponseCode` 里。变体还有 `consecutive_gateway_failure`(只看 502/503/504 这几个 gateway 错误)和 `consecutive_local_origin_failure`(本地连接失败,如 connect refused)。

**2. success_rate(成功率统计)**:这是**跨 host、统计性**的触发。每隔一个 `interval`(默认 10 秒,见 [`DEFAULT_INTERVAL_MS`](../envoy/source/common/upstream/outlier_detection_impl.h#L366)),detector 把所有 host 的成功率收集起来,算出集群**平均成功率**和**标准差**,然后设定一个阈值 = `均值 - N × 标准差`。任何成功率**低于这个阈值**的 host,都被 eject。这是统计学里的"离群点检测"(outlier detection 名字的由来)——不设固定阈值(比如"成功率低于 95% 就踢"),而是**相对于集群当前的整体水平**动态判定。

**3. failure_percentage(失败百分比)**:和 success_rate 类似也是统计性触发,但更简单直接——某个 host 的失败率 `(100 - 成功率)` 超过配置的 `failure_percentage_threshold`(默认 85%),就 eject。它和 success_rate 的区别:failure_percentage 是**绝对阈值**(配多少就是多少),success_rate 是**相对阈值**(相对集群均值减标准差)。

三种触发的对比:

| 触发 | 视角 | 时机 | 阈值性质 | 适用场景 |
|------|------|------|---------|---------|
| consecutive_5xx | 单 host 内 | 即时(每请求) | 固定次数(默认 5) | host 突然持续出错 |
| success_rate | 跨 host 比较 | 周期(默认 10s) | 相对(均值 - N×stdev) | 集群整体还好,个别 host 偏低 |
| failure_percentage | 单 host 内 | 周期(默认 10s) | 绝对(默认 85%) | 不管集群整体,绝对失败率高就踢 |

> **不这样会怎样**:只有 consecutive_5xx,会漏掉"间歇性故障"——一个 host 每 10 个请求错 3 个(失败率 30%),但从来不会连续 5 次 5xx(consecutive_5xx 永远触发不了),可它的失败率明显高于集群其他 host(其他都 99%+),这种就该用 success_rate 抓。只有 success_rate,又会漏掉"集群整体都差"的场景——如果整个集群失败率都 50%(大面积故障),success_rate 的相对阈值会跟着拉低,反而踢不出任何 host(因为大家都差不多),这时候 failure_percentage 的绝对阈值能兜底。**三种触发覆盖不同的故障形态,这正是 outlier detection 设计的周到之处。**

### success_rate 的统计算法:为什么是"均值减 N 倍标准差"

success_rate 触发的核心算法,是 [`successRateEjectionThreshold`](../envoy/source/common/upstream/outlier_detection_impl.cc#L739) 这个静态方法。源码很短但内涵深:

```cpp
// source/common/upstream/outlier_detection_impl.cc:739 (简化示意,非源码原文)
EjectionPair DetectorImpl::successRateEjectionThreshold(
    double success_rate_sum,
    const std::vector<HostSuccessRatePair>& valid_success_rate_hosts,
    double success_rate_stdev_factor) {
  // 用均值和标准差做离群点检测
  // 例: success_rate_data = {50, 100, 100, 100, 100}
  //   sum = 450, mean = 90, variance = 400, stdev = 20
  //   threshold = mean - stdev_factor * stdev = 90 - 19*20 = ... (factor 默认 19)
  double mean = success_rate_sum / valid_success_rate_hosts.size();
  double variance = 0;
  for_each(valid_success_rate_hosts, [&variance, mean](HostSuccessRatePair v) {
    variance += pow(v.success_rate_ - mean, 2);
  });
  variance /= valid_success_rate_hosts.size();
  double stdev = sqrt(variance);
  return {mean, (mean - (success_rate_stdev_factor * stdev))};
}
```

这就是统计学里的**均值 ± k 倍标准差**离群点检测法。阈值不是写死的"95%",而是 `mean - factor × stdev`,其中 `factor` 由配置 `success_rate_stdev_factor` 控制(默认 1900,但单位是千分之一,所以实际是 1.9,见 [`DEFAULT_SUCCESS_RATE_STDEV_FACTOR = 1900`](../envoy/source/common/upstream/outlier_detection_impl.h#L373) 和 [`processSuccessRateEjections`](../envoy/source/common/upstream/outlier_detection_impl.cc#L828) 里的 `/ 1000.0`)。

源码注释里那个例子很说明问题:`{50, 100, 100, 100, 100}`——五个 host,一个成功率 50%(明显有问题),其他四个都 100%。算出来 mean=90, stdev=20, threshold = 90 - 1.9×20 = 52。那个 50% 的 host 低于 52,被踢。整个集群的均值(90)和离散度(stdev=20)自动反映了"当前集群整体水平",阈值随之动态调整。

> **钉死这件事**:success_rate 用"均值减 N 倍标准差"而不是固定阈值,是因为**集群的整体水平是动态的**——白天高峰期可能整体成功率 99%(那一个 95% 的 host 就是异常),凌晨低峰期可能整体只有 90%(这时候 85% 的才算异常)。固定阈值(比如"低于 95% 就踢")在不同场景下要么太严要么太松。相对阈值让 outlier 的判定**自适应集群当前状态**——这是统计学方法比拍脑袋阈值高明的地方,也是"outlier detection"这个名字(统计离群点检测)的精确含义。

### 触发的"前置条件":够多少 host、够多少请求量

success_rate 和 failure_percentage 不是无脑跑的,有前置条件,在 [`processSuccessRateEjections`](../envoy/source/common/upstream/outlier_detection_impl.cc#L769) 开头:

- **集群里要有足够的 host**(`success_rate_minimum_hosts`,默认 5)。只有 2 个 host,算"均值和标准差"没统计意义——你怎么知道是哪个异常?可能两个都正常只是有差异。Envoy 要求至少 5 个 host 才启用 success_rate。这是统计显著性的底线。
- **每个 host 要有足够的请求量**(`success_rate_request_volume`,默认 100)。一个 host 这 10 秒只来了 2 个请求、错了 1 个,失败率 50%——但这数据没意义(样本太小)。Envoy 要求一个 host 在窗口内至少 100 个请求,它的成功率才进入统计。

> **不这样会怎样**:不加前置条件,小集群(2 个 host)或低流量 host(几秒来 1 个请求)会触发大量**误踢**——基于小样本的统计完全不靠谱。这两个 minimum 是统计显著性的硬约束,宁可漏踢(等样本够了再说)也不误踢。outlier detection 的设计哲学是**保守**:宁可让一个真正有问题的 host 多留一会儿,也不要因为统计噪音把好 host 踢出去。

## 四、eject:不是删除,是临时排除 + 退避倍增

outlier detection 触发后,动作是 **eject**(踢出)。eject 这个动作有两层精妙设计,值得单独拆。

### eject 不删除 host,只贴 flag

很多人误以为"踢出"就是把 host 从 cluster 删掉。不是。eject 只是给 host 贴一个 `FAILED_OUTLIER_CHECK` flag,host 还在 cluster 里(EDS 下发的 endpoint 列表不变),只是**被排除出 LB 的候选集**。看 [`DetectorHostMonitorImpl::eject`](../envoy/source/common/upstream/outlier_detection_impl.cc#L52):

```cpp
// source/common/upstream/outlier_detection_impl.cc:52 (简化示意,非源码原文)
void DetectorHostMonitorImpl::eject(MonotonicTime ejection_time) {
  ASSERT(!host_.lock()->healthFlagGet(Host::HealthFlag::FAILED_OUTLIER_CHECK));
  host_.lock()->healthFlagSet(Host::HealthFlag::FAILED_OUTLIER_CHECK);  // 只贴 flag
  num_ejections_++;
  last_ejection_time_ = ejection_time;
}
```

而第 13 章的 LB,在挑 host 时,会跳过任何带 `FAILED_ACTIVE_HC` 或 `FAILED_OUTLIER_CHECK` flag 的 host。看 Host 的 [`healthFlagsGet`](../envoy/source/common/upstream/upstream_impl.h#L421):

```cpp
// source/common/upstream/upstream_impl.h:421 (简化示意,非源码原文)
if (healthFlagsGet(enumToInt(HealthFlag::FAILED_ACTIVE_HC) |
                   enumToInt(HealthFlag::FAILED_OUTLIER_CHECK))) {
  // 这个 host 不健康,不进 healthyHosts(),LB 看不到它
}
```

> **所以这样设计**:eject 是"软隔离"而不是"硬删除"。host 的元信息(地址、权重、EDS 状态)都还在,只是暂时不被 LB 选中。这样设计的好处:① **恢复容易**——host 好了之后清掉 flag 即可,不用重新走 EDS 发现、重建;② **不丢信息**——你能从 stats 看到"这个 host 被 eject 了 N 次",监控、排障信息完整;③ **职责分离**——outlier detector 只管"贴/撕 flag",LB 只管"看 flag 选 host",两者解耦。**不这么设计会怎样**:如果 eject 是真删除,恢复时要重新走 EDS、等控制面下发,慢;而且 host 的 eject 历史(被踢了几次)这些监控信息会丢。

### eject 的上限:不能把整个集群踢光

outlier 不能无限制地踢。配置 `max_ejection_percent`(默认 10%,见 [`DEFAULT_MAX_EJECTION_PERCENT`](../envoy/source/common/upstream/outlier_detection_impl.h#L370))限制了同时被 eject 的 host 比例。看 [`ejectHost`](../envoy/source/common/upstream/outlier_detection_impl.cc#L542):

```cpp
// source/common/upstream/outlier_detection_impl.cc:542 (简化示意,非源码原文)
void DetectorImpl::ejectHost(HostSharedPtr host, OutlierEjectionType type) {
  uint64_t max_ejection_percent = min(100, runtime.getInteger(
      MaxEjectionPercentRuntime, config_.maxEjectionPercent()));  // 默认 10
  double ejected_percent = 100.0 * (ejections_active + 1) / host_monitors_.size();
  bool should_eject = (ejected_percent <= max_ejection_percent);
  if (should_eject) {
    if (enforceEjection(type)) {  // enforce 概率(见下文)
      ejections_active_helper_.inc();
      host_monitors_[host]->eject(time_source_.monotonicTime());
      // 退避倍增 ...
    }
  } else {
    stats_.ejections_overflow_.inc();  // 超过上限,只计数不踢
  }
}
```

> **不这样会怎样**:如果不加上限,极端情况下 outlier 可能踢掉大部分 host——比如集群整体故障(所有 host 都出错),outlier 会把一个接一个踢出去,最后 LB 没有可用 host,所有请求 503(进入 panic 模式,第 12 章讲)。`max_ejection_percent` 是个**熔断保护**——保留至少 `(100 - max_ejection_percent)%` 的 host 给 LB,哪怕它们也不太健康。两害相权取其轻:与其完全没 host 可用(100% 失败),不如保留一部分 host(虽然失败率高,但至少还能服务部分请求)。这是 outlier 设计的**保守底色**:宁可带着问题 host 继续服务,也不要把集群踢瘫。

### enforce 概率:只踢一定比例的"应踢 host"

`ejectHost` 里有个 `enforceEjection(type)`,这是另一层保守机制。每种触发类型有个 `enforcing_*` 配置(0~100 的百分比,见 [`DEFAULT_ENFORCING_CONSECUTIVE_5XX = 100`](../envoy/source/common/upstream/outlier_detection_impl.h#L377)、[`DEFAULT_ENFORCING_SUCCESS_RATE = 100`](../envoy/source/common/upstream/outlier_detection_impl.h#L379) 等)。`enforceEjection` 按这个概率决定"虽然检测到该踢,但要不要真踢"。默认 consecutive_5xx 和 success_rate 是 100%(检测到就踢),而 gateway_failure、failure_percentage 默认是 0%(检测到也不踢,只统计)。

为什么留这个口子?因为有些触发的"假阳性"率可能高(比如 failure_percentage 在小样本下不稳定),运维想先**只观察**(看 `ejections_detected_*` 计数)而不真的踢,等调参有信心了再把 `enforcing_*` 调上去。这是给运维一个**渐进式启用**的旋钮。

### 退避倍增:连续被踢的 host,踢的时间翻倍

这是 eject 最精妙的设计,也是本章技巧精解的重点之一。看 `ejectHost` 里这段:

```cpp
// source/common/upstream/outlier_detection_impl.cc:581 (简化示意,非源码原文)
if ((host_monitors_[host]->ejectTimeBackoff() * base_eject_time) <
    (max_eject_time + base_eject_time)) {
  host_monitors_[host]->ejectTimeBackoff()++;  // 倍数 +1
}
```

每个 host 有个 `eject_time_backoff_` 计数器(初始 0)。每次被 eject,这个计数器 +1(有上限,见条件判断)。下次"该踢多久"的计算是 `base_eject_time × eject_time_backoff_`,在 [`checkHostForUneject`](../envoy/source/common/upstream/outlier_detection_impl.cc#L401):

```cpp
// source/common/upstream/outlier_detection_impl.cc:401 (简化示意,非源码原文)
if ((min(base_eject_time * monitor->ejectTimeBackoff(), max_eject_time) + jitter) <=
    (now - monitor->lastEjectionTime().value())) {
  unejectHost(host);  // 踢的时间到了,加回
}
```

也就是说:**第一次被踢,留 `base_eject_time`(默认 30 秒);第二次被踢,留 `2 × base_eject_time`(60 秒);第三次 90 秒……** 直到 `max_eject_time`(默认 `10 × base_eject_time` = 300 秒,见 [`DEFAULT_MAX_EJECTION_TIME_MS`](../envoy/source/common/upstream/outlier_detection_impl.h#L385))封顶。而且,一个 host 如果**健康地存活了一个 interval**(没被踢),backoff 会**衰减**:

```cpp
// source/common/upstream/outlier_detection_impl.cc:897 (简化示意,非源码原文)
// 在 onIntervalTimer 末尾
if (!host->healthFlagGet(Host::HealthFlag::FAILED_OUTLIER_CHECK)) {
  // host 健康,且距上次 uneject 过了一个 interval
  if (monitor->lastUnejectionTime().has_value() && ...) {
    if (monitor->ejectTimeBackoff() != 0) {
      monitor->ejectTimeBackoff()--;  // 衰减
    }
  }
}
```

> **不这样会怎样**:退避倍增解决的是"反复抖动的坏 host"问题。一个不稳定的 host,被踢 30 秒、加回,马上又开始出错、又被踢 30 秒、加回……无限循环。这种 host 每次加回只存活几秒就又被踢,但它频繁进出 healthy 集合,引发 LB 重新计算、连接池重建、客户端重连——**抖动放大**。退避倍增让"屡教不改"的 host 被踢得越来越久(30s → 60s → 90s → ... → 300s 封顶),给它更长的"冷静期",减少抖动。而衰减机制保证:如果 host 真的好了(连续健康存活),backoff 慢慢回到 0,下次偶发故障不会一上来就踢很久。**这是"惩罚递增 + 改过自新"的工程化**——和 TCP 拥塞控制的"慢启动"、和第 15 章要讲的断路器的"半开恢复",是同一种思想。

## 五、active 与 outlier 的配合:健康 flag 的统一与互相补救

讲到这里,active 和 outlier 各自的机制清楚了。但它们不是两个孤岛——Envoy 让它们**通过统一的 health flag 体系协同**,甚至互相补救。

### 统一的 health flag 体系

Host 上有一组 health flag,定义在 [`upstream.h`](../envoy/envoy/upstream/upstream.h#L150)。和本章相关的几个:

```cpp
// envoy/upstream/upstream.h:151 (HEALTH_FLAG_ENUM_VALUES 宏)
m(FAILED_ACTIVE_HC, 0x1)            // active health check 失败
m(FAILED_OUTLIER_CHECK, 0x02)       // outlier detection 踢出
m(DEGRADED_ACTIVE_HC, 0x08)         // active 探活标为 degraded
m(PENDING_ACTIVE_HC, 0x40)          // 还没探过活(初始)
m(DEGRADED_OUTLIER_DETECTION, 0x400) // outlier 标为 degraded(软踢)
```

LB(第 13 章)看的就是这几个 flag:**只要 `FAILED_ACTIVE_HC` 或 `FAILED_OUTLIER_CHECK` 任一被设,host 就不进 `healthyHosts()`,LB 不选它**。active 和 outlier 各自只管设/清自己的 flag,互不干扰,但对 LB 来说是"或"的关系——任一判你不健康,你就出局。

```
   Host health flags (和本章相关)
   ┌──────────────────────────────────────────────────────┐
   │  FAILED_ACTIVE_HC (0x1)    ← active health check 设  │
   │  FAILED_OUTLIER_CHECK (0x02) ← outlier detection 设  │
   │                                                      │
   │  LB (第13章):  任一被设 → 不进 healthyHosts()        │
   │                                                      │
   │  DEGRADED_ACTIVE_HC (0x08)    ← active 标 degraded   │
   │  DEGRADED_OUTLIER_DETECTION (0x400) ← outlier 标 degraded│
   │                                                      │
   │  LB:  degraded host 还能被选,但优先级降级(少分配)   │
   └──────────────────────────────────────────────────────┘
```

### active 探活成功,可以提前救回被 outlier 踢的 host

这是两者配合最巧妙的一点。outlier 踢出一个 host 后,正常要等 `base_eject_time × backoff` 才自动加回(30 秒到几分钟)。但如果这期间 **active health check 探活成功**了呢?这说明这个 host 其实已经恢复了(至少 active 探活的端点是好的)。Envoy 提供一个配置 `successful_active_health_check_uneject_host`(默认 **true**,见 [`DetectorConfig` 构造函数](../envoy/source/common/upstream/outlier_detection_impl.cc#L293)):active 探活成功时,**立即清掉 `FAILED_OUTLIER_CHECK` flag,把 host 提前救回来**,不用等 eject 时间到。

看这段衔接逻辑,在 outlier detector 的 [`initialize`](../envoy/source/common/upstream/outlier_detection_impl.cc#L344) 里注册到 active health checker 的回调:

```cpp
// source/common/upstream/outlier_detection_impl.cc:344 (简化示意,非源码原文)
if (config_.successfulActiveHealthCheckUnejectHost() && cluster.healthChecker() != nullptr) {
  cluster.healthChecker()->addHostCheckCompleteCb(
      [this](HostSharedPtr host, HealthTransition, HealthState current_check_result) {
        // 如果 host 被 outlier 踢了,但 active 探活成功,提前救回
        if (current_check_result == HealthState::Healthy &&
            !host->healthFlagGet(Host::HealthFlag::FAILED_ACTIVE_HC) &&
            host->healthFlagGet(Host::HealthFlag::FAILED_OUTLIER_CHECK)) {
          host->healthFlagClear(Host::HealthFlag::FAILED_OUTLIER_CHECK);
          unejectHost(host);
        }
      });
}
```

> **所以这样设计**:outlier 是被动的、基于统计的,可能"误踢"——一个 host 只是在统计窗口内运气不好(失败率偏高),其实已经恢复了。让它干等 30 秒~5 分钟才加回,白白浪费一个能用的后端。active 探活(周期性主动敲门)恰好是个**独立的、即时的信号**——如果 active 说它好了,大概率是真好了,赶紧救回来。这把 active 的"主动、即时"和 outlier 的"被动、保守"互补到了极致:**outlier 保守地踢,active 积极地救**。这个跨机制的补救,是 Envoy 健康治理的匠心所在。**不这么设计会怎样**:如果两者完全独立,被 outlier 踢的 host 只能死等 eject 时间到,哪怕它早好了也要被晾着——后端资源利用率下降、恢复慢。active 的补救让"误踢"的代价最小化。

## 六、技巧精解:两个最硬核的设计

本章挑两个最值得钉死的技巧,配源码拆透。

### 技巧一:outlier detection 的成功率统计——双桶无锁轮换

outlier 的 success_rate 触发,需要每个 host 在一个窗口(默认 10 秒)内统计"成功请求数 / 总请求数"。难点在于:**这些统计发生在 worker 线程**(真实请求在 worker 上跑、在 worker 上调 `putResult`),而**判定和 eject 发生在 main 线程**(interval_timer 在 main 上跑)。多线程并发写同一个计数器,怎么避免锁竞争?

Envoy 的解法是 [`SuccessRateAccumulator`](../envoy/source/common/upstream/outlier_detection_impl.h#L63) 的**双桶轮替(double buffering)**。每个 host 的 accumulator 有两个桶(`current` 和 `backup`),worker 线程**无锁地往 current 桶写**(用 atomic 计数器),main 线程在 interval 到了的时候**交换两个桶**,然后读 backup 桶(此时它已经不被写了)算成功率。

看 [`SuccessRateAccumulatorBucket`](../envoy/source/common/upstream/outlier_detection_impl.h#L53),就是两个 `std::atomic<uint64_t>`:

```cpp
// source/common/upstream/outlier_detection_impl.h:53 (简化示意,非源码原文)
struct SuccessRateAccumulatorBucket {
  std::atomic<uint64_t> success_request_counter_;
  std::atomic<uint64_t> total_request_counter_;
};
```

worker 写(在 `SuccessRateMonitor::incTotalReqCounter` / `incSuccessReqCounter`):

```cpp
// source/common/upstream/outlier_detection_impl.h:103 (简化示意,非源码原文)
void incTotalReqCounter() {
  success_rate_accumulator_bucket_.load()->total_request_counter_++;  // 原子自增,无锁
}
```

main 线程在 `onIntervalTimer` 里调 `updateCurrentSuccessRateBucket`(对每个 host),它调 [`SuccessRateAccumulator::updateCurrentWriter`](../envoy/source/common/upstream/outlier_detection_impl.cc#L1009):

```cpp
// source/common/upstream/outlier_detection_impl.cc:1009 (简化示意,非源码原文)
SuccessRateAccumulatorBucket* SuccessRateAccumulator::updateCurrentWriter() {
  // 把 backup 清零,然后和 current 交换
  backup_success_rate_bucket_->success_request_counter_ = 0;
  backup_success_rate_bucket_->total_request_counter_ = 0;
  current_success_rate_bucket_.swap(backup_success_rate_bucket_);  // unique_ptr 交换
  return current_success_rate_bucket_.get();  // 新的 current(旧 backup,已清零)
}
```

然后 main 线程用 [`getSuccessRateAndVolume`](../envoy/source/common/upstream/outlier_detection_impl.cc#L1019) 读**刚才交换过去的 backup**(现在它不被 worker 写了,因为 worker 的指针已经指向新的 current):

```cpp
// source/common/upstream/outlier_detection_impl.cc:1019 (简化示意,非源码原文)
optional<pair<double, uint64_t>> SuccessRateAccumulator::getSuccessRateAndVolume() {
  if (!backup_success_rate_bucket_->total_request_counter_) return nullopt;  // 没请求量
  double success_rate = backup_success_rate_bucket_->success_request_counter_ * 100.0 /
                        backup_success_rate_bucket_->total_request_counter_;
  return {{success_rate, backup_success_rate_bucket_->total_request_counter_}};
}
```

> **不这么写会怎样**:朴素方案一是"worker 加锁写计数器,main 加锁读"——但 `putResult` 在**每个真实请求**的热路径上,加锁会让所有 worker 在这里排队,高 QPS 下成为性能瓶颈。朴素方案二是"每个 worker 维护自己的 thread-local 计数器,main 定期归并所有 worker"——这和 Envoy 的 stats thread-local 攒批(第 1 章和第 20 章)是同一个套路,正确但实现复杂(要维护 N 份副本)。**双桶轮替**是最轻量的解法:① worker 写 current(atomic 自增,无锁);② main 交换双桶(一次 `unique_ptr::swap`,极快);③ main 读旧 current(现 backup,已和 worker 解耦)。巧妙之处在于:**交换的是指针(unique_ptr),不是数据**——worker 写完的桶,指针一交,就成了 main 的只读快照,worker 开始写新桶。无锁、零拷贝、无归并。这是用"时间错位"(worker 写 N 窗口,main 读 N-1 窗口)换"空间隔离"的经典并发技巧,和图形学里的双缓冲、和数据库的多版本并发(MVCC)是同一种思想。

注意这里有个细节:`success_rate_accumulator_bucket_` 是 `std::atomic<SuccessRateAccumulatorBucket*>`(见 [`SuccessRateMonitor`](../envoy/source/common/upstream/outlier_detection_impl.h#L112)),worker 通过 `load()` 拿当前桶指针。交换后,worker 的 `load()` 会拿到新指针(因为 `updateCurrentWriter` 里 `store` 了新指针)。这里有个**微妙的内存顺序**问题:atomic 的 `store` 和 `load` 默认是 `memory_order_seq_cst`,保证了 worker 看到的指针更新和 main 的清零顺序一致——这是这套无锁方案 sound 的根。

### 技巧二:eject 退避倍增的工程权衡

退避倍增(第四节讲过机制)的精妙,在于它几个参数的**协同设计**。把相关常数和逻辑放一起看:

- `base_eject_time` = 30 秒([`DEFAULT_BASE_EJECTION_TIME_MS`](../envoy/source/common/upstream/outlier_detection_impl.h#L367))
- `max_eject_time` = 300 秒 = `10 × base`([`DEFAULT_MAX_EJECTION_TIME_MS`](../envoy/source/common/upstream/outlier_detection_impl.h#L385))
- 倍增条件:`(backoff × base) < (max + base)` 时 `backoff++`(见 [`ejectHost`](../envoy/source/common/upstream/outlier_detection_impl.cc#L581))
- 实际踢时:`min(backoff × base, max)`(见 [`checkHostForUneject`](../envoy/source/common/upstream/outlier_detection_impl.cc#L401))
- 健康存活一个 interval,`backoff--`(衰减)

代入默认值推演一个反复出错的 host:

| 第几次 eject | backoff(自增后) | 实际踢多久 min(backoff×30s, 300s) | 加回后存活 |
|------|------|------|------|
| 1 | 1 | 30s | 又出错 |
| 2 | 2 | 60s | 又出错 |
| 3 | 3 | 90s | 又出错 |
| ... | ... | ... | ... |
| 10 | 10 | 300s(封顶) | 又出错 |
| 11 | 11(但条件 `11×30=330 < 300+30=330` 不成立,不再自增) | 300s | ... |

注意倍增条件的边界:`(backoff × base) < (max + base)`,即 `backoff × 30 < 330`,`backoff < 11`。所以 backoff 最多到 11(第 10 次自增后),第 11 次 eject 时 backoff 不再涨,但实际踢的时间因为 `min` 已经被 max 封顶在 300 秒了。这个设计的意图:**给坏 host 越来越长的冷静期,但有上限**——不会无限增长到把 host 永久踢出去(那等于变相删除,违背"eject 是临时"的设计)。

> **不这么写会怎样**:三种朴素的替代方案各有问题。① **不退避,固定 30 秒**——前面说过,坏 host 反复进出,抖动放大。② **永久踢出(一踢不回)**——太激进,一个 host 偶发故障就被永久剔除,集群规模越来越小;而且要恢复得人工干预或等 EDS 重新下发,运维负担重。③ **退避但无上限**——backoff 指数涨,几次之后就等于永久踢出,退化成方案 2。Envoy 的设计是**线性退避 + 封顶 + 衰减**:线性增长(不是指数,温和)、封顶在 max(不会永久踢)、健康存活会衰减(改过自新能回归)。这套参数协同,既惩罚了持续不稳定的 host,又给了恢复的机会,还保证了不会把集群踢瘫。**这是"流量治理"里典型的工程权衡——不是追求理论最优,而是追求在各种故障形态下都不至于失控**。和第 15 章的断路器(半开恢复)、和 TCP 的拥塞控制(加性增乘性减),是同一种工程哲学。

## 七、章末小结

### 回扣主线

本章服务的是**数据面**这一面——具体说是 upstream 的健康治理。它回答的问题是:第 13 章的 LB"只从 healthy host 里挑",那"healthy"这个判定怎么来?答案是 active health check(主动探活,设 `FAILED_ACTIVE_HC`)和 outlier detection(被动统计,设 `FAILED_OUTLIER_CHECK`)两套互补的机制,它们共同维护 host 的健康 flag,而 LB 只看 flag 选 host。这和 filter chain、xDS 一起,构成了 Envoy "治理每一条流量"的完整画面:filter chain 处理流量、xDS 动态配、健康治理保证流量只打到能接活的后端。

### 五个为什么

1. **为什么不能只靠真实请求失败来发现坏后端?**——慢(发现要等真实请求撞上去)、代价由真实用户承担、应用层卡死但 OS 活着的 host 探不到。需要 active health check 让 Envoy 自己周期性敲门、自己承担探测成本。

2. **为什么不能只靠 active 探活?**——探活间隔内的突发故障抓不到(秒级盲区)、探活端点(轻量 `/health`)健康但真实业务挂了的情况看不见。需要 outlier detection 蹲在真实请求结果流上实时反应。

3. **为什么 outlier 的 success_rate 用"均值减 N 倍标准差"而不是固定阈值?**——集群整体水平是动态的(高峰 99%、低峰 90%),固定阈值在不同场景下忽严忽松。相对阈值(均值 - 1.9×stdev)自适应集群当前状态,这是统计离群点检测的精确含义,也是"outlier detection"名字的由来。

4. **为什么 eject 不删除 host 只贴 flag?**——软隔离便于恢复(清 flag 即可,不走 EDS 重发)、不丢监控信息(eject 次数可查)、职责分离(outlier 贴 flag、LB 看 flag)。配合退避倍增(线性增长 + 封顶 + 衰减),既惩罚不稳定 host,又给恢复机会,还不把集群踢瘫。

5. **为什么 active 探活成功能提前救回被 outlier 踢的 host?**——outlier 是被动统计,可能误踢(运气不好的 host)。active 是独立、即时的主动信号,它说"好了"大概率真好了。`successful_active_health_check_uneject_host` 让 active 积极补救 outlier 的保守误踢,把两者互补到极致。
