# 《Envoy 设计与实现深入浅出:为什么把每条流量做成一串可热更新的 filter》—— 目录与导读

> 一本写给"天天在微服务里用 Envoy/Istio、却总觉得只知其表不知其里"的人的小书。
>
> **一句话主旨**:如何把"处理每一条流量"做成一串可插拔的 filter(filter chain),把"配什么 filter、转发到哪"做成可动态下发的 xDS 配置(filter chain + xDS = service mesh 的 data plane)?
>
> **二分法**(迷路时回到它):**数据面**(每条流量怎么被 filter 链处理:线程模型/listener/network filter/HCM/http filter/router/cluster/LB) vs **控制面**(配置怎么被 xDS 动态下发、热更新)。
>
> **主比喻**:直球为主、比喻点睛。开篇用"邮局分拣中心"做一次性点睛(邮件 = 流量、分拣传送带 = filter chain、分拣规则表 = xDS 动态下发的配置),全书不再建立贯穿重比喻。
>
> **承接**:HTTP/2/HPACK 承接《gRPC》、xDS 是 Envoy 定义 gRPC 复用(承接《gRPC》P6-22)、epoll/异步承接《Tokio》《Linux 内核》。

每章一行:**一句话钩子** —— 技巧标签 —— 二分法归属(`数据面` / `控制面` / `可观测` / `安全` / `总览`)。

旅程:从 Nginx 那种"配置写死、reload 才生效"的经典代理,一路走到"每条流量穿过一串可热更新的 filter、配置由 xDS 动态下发"的可编程数据面。每篇都是这条旅程上的一个驿站——读完你能在脑子里放映出一条流量在 Envoy 里的全过程:请求进 listener → listener filter(TLS 探测)→ network filter / HCM → 解码 HTTP → http filter(鉴权/限流)→ router 选 cluster → LB 挑 endpoint → 连接池转发 → 响应原路返回;同时这一切 filter 与路由表,是控制面通过 xDS 动态下发、热更新不停机的。
