# 第 27 章 · 内存屏障:为什么光有锁还不够

> **前置**:你需要先读过第 25 章(spinlock/mutex——本章反复回扣"`spin_unlock` 里的 release 屏障"、"`mutex_lock` 里的 acquire 屏障")和第 26 章(RCU——`rcu_assign_pointer`/`rcu_dereference` 那对屏障)。本章是把前两章里"偷偷编进去的屏障"挖开给你看。

读完这一章你会明白:**代码里写的顺序,不等于内存真正生效的顺序**。有两个小妖精(编译器、CPU)会偷偷把你的指令重排、缓存、延迟。内存屏障就是用来管住它们的。你也会知道:**日常用锁/RCU 时不用自己加屏障**(锁内部已经编好了),**只有写无锁(lockless)代码时才需要亲手上屏障**——而且该用 acquire/release,而不是老式的 `smp_mb`。

> **如果一读觉得太难**:先只记住四件事——① **代码顺序 ≠ 内存生效顺序**,编译器和 CPU 都会重排;② `READ_ONCE`/`WRITE_ONCE` 挡**编译器**,`smp_mb` 等挡 **CPU**;③ 日常用锁/RCU 时**屏障是锁内部自带的,你不用管**;④ 只有写无锁代码(自己用原子变量/标志位同步)才需要 `smp_store_release`/`smp_load_acquire`。其余细节第二遍配合源码再抠。

## 一、一个让你怀疑人生的例子

先看一段"逻辑上天衣无缝、却会出 bug"的代码:

```c
/* 写者(核 A) */
data = 42;          /* (1) 准备数据 */
ready = true;       /* (2) 置标志位"数据好了" */

/* 读者(核 B) */
while (!ready);     /* (3) 等标志位 */
print(data);        /* (4) 用数据 —— 本该打印 42 */
```

这段代码在任何一本单线程教材里都是对的。可**在多核 + 优化编译下,读者可能打印出一个旧 data(比如 0)**。

不是你写错了,是有两个你看不见的家伙**偷偷动了你的代码顺序**:

### 1.1 第一个妖精:编译器重排

编译器为了优化,会**重排指令**。对单线程程序,只要"结果一样",编译器爱怎么排怎么排。对上面那段,编译器可能:

- 把 (2) 的写 `ready` 重排到 (1) 之前——因为在单线程看来,先写哪个无所谓。
- 把 (4) 的读 `data` 缓存到寄存器、或者重排——因为编译器觉得 `ready` 和 `data` 没关系。

**不这样会怎样?** 编译器一重排,写者变成"先置 ready、后写 data";读者变成"先读 data、后判 ready"。多核下就乱套:读者看到 ready=true,去读 data,可写者的 data 还没写真值 → 读到旧的 0。

### 1.2 第二个妖精:CPU 乱序执行 + store buffer

就算编译器老老实实按顺序生成指令,**CPU 自己也会乱序执行**。现代 CPU 为了让流水线快:

- **乱序执行(out-of-order)**:CPU 不一定按指令顺序执行,只要没依赖,谁先算出来谁先走。
- **store buffer**:CPU 写内存时,**不直接写到内存/缓存**,而是先写到一个本核私有的"写缓冲(store buffer)"里,稍后才真正刷到缓存、被别的核看见。读的时候,本核能立刻看到自己 store buffer 里的新值,但**别的核还看不见**。

**不这样会怎样?** 写者发了"写 data=42"和"写 ready=true"两条指令。由于 store buffer,可能"写 ready"先刷到缓存、被核 B 看见,而"写 data"还停在 store buffer 里没刷出去。核 B 看到 ready=true,去读 data——读到内存里的旧值 0!

> **这就是"内存一致性模型(memory consistency model)"问题**。每个 CPU 架构对"写操作的可见顺序"有不同的承诺(叫它的"内存模型"):x86 是**强序(strongly ordered)**,大部分顺序有保证,但仍有少数情况会乱;ARM、PowerPC、Alpha 是**弱序(weakly ordered)**,基本不保证顺序,不插屏障就乱套。

### 1.3 所以:你需要内存屏障

内存屏障(barrier)就是一条"**到此为止,不许重排、不许延迟**"的指令。它的作用:

- **编译器屏障**(`barrier()`):告诉编译器"屏障之前的内存访问,不许重排到屏障之后;反之亦然。"
- **CPU 屏障**(`smp_mb` 等):生成一条真的 CPU 屏障指令(x86 上是带 `lock` 前缀的指令,或 `mfence`),告诉 CPU"屏障之前的写,必须先于屏障之后的写,被别的核看见。"

**加了屏障后,前面那段代码才真的可靠**:

```c
/* 写者 */
data = 42;
smp_store_release(&ready, true);   /* release 屏障:保证 data 先就绪 */

/* 读者 */
while (!(smp_load_acquire(&ready))); /* acquire 屏障:保证先读到 ready 再读 data */
print(data);
```

## 二、第一道防线:READ_ONCE / WRITE_ONCE 挡住编译器

管住编译器,是第一道、也是最便宜的防线。内核提供 `READ_ONCE` / `WRITE_ONCE`,出自 [include/linux/compiler.h](../linux-6.14/include/linux/compiler.h#L47-L62):

```c
#define __READ_ONCE(x)	(*(const volatile __unqual_scalar_typeof(x) *)&(x))

#define READ_ONCE(x)							\
	( ... __READ_ONCE(x) ... )

#define __WRITE_ONCE(x, val)						\
	(*(volatile __unqual_scalar_typeof(x) *)&(x) = (val))

#define WRITE_ONCE(x, val)						\
	( ... __WRITE_ONCE(x, val) ... )
```

它们到底挡住了什么?

1. **`volatile`**:阻止编译器把这次访问"优化掉"(比如循环里反复读一个标志位,编译器可能只读一次存寄存器,后续都用寄存器旧值——这对多核共享变量是致命的)。`volatile` 强制**每次都真正去内存读/写**。
2. **顺序护栏**:内核版本的 `READ_ONCE`/`WRITE_ONCE` 还保证:编译器不能把它们和**前后**的其它 `READ_ONCE`/`WRITE_ONCE` 重排(虽然普通变量间仍可能重排)。

> **`READ_ONCE`/`WRITE_ONCE` 只管编译器,不管 CPU**。它们是"必要的、但不充分的"——挡住了编译器重排,CPU 那层乱序还得靠下面的 `smp_*` 屏障。但在很多场景(尤其 x86 这种强序架构),光挡住编译器往往就够用了(因为 x86 本身大部分顺序有保证)。

> **心法**:内核里**任何被多个执行流共享访问的变量,至少要用 `READ_ONCE`/`WRITE_ONCE`**。这不是洁癖,是底线——否则编译器优化随时可能给你埋雷。你会在内核代码里看到满地的 `READ_ONCE`,原因就在这。

## 三、第二道防线:smp_* 屏障挡住 CPU

管住 CPU,要靠真正的 CPU 屏障指令。内核提供一整套 `smp_*` 宏,出自 [include/asm-generic/barrier.h](../linux-6.14/include/asm-generic/barrier.h):

| 宏 | 作用 | 比喻 |
|----|------|------|
| `smp_mb()` | 全屏障:之前的读写,必须在之后的读写之前、对别的核可见 | "前后都设一道铁闸" |
| `smp_rmb()` | 读屏障:之前的读,必须在之后的读之前完成 | "只管读这边的顺序" |
| `smp_wmb()` | 写屏障:之前的写,必须在之后的写之前完成 | "只管写这边的顺序" |
| `smp_store_release(p,v)` | 写 + 之前的所有读写都必须先于本次写可见 | "封箱前把货备齐再封" |
| `smp_load_acquire(p)` | 读 + 之后的读写都必须在该读之后才进行 | "确认盖章后再拆后续包裹" |

### 3.1 mb/rmb/wmb 的关系

看 [include/asm-generic/barrier.h](../linux-6.14/include/asm-generic/barrier.h#L99-L107):

```c
#define smp_mb()	do { kcsan_mb(); __smp_mb(); } while (0)
#define smp_rmb()	do { kcsan_rmb(); __smp_rmb(); } while (0)
#define smp_wmb()	do { kcsan_wmb(); __smp_wmb(); } while (0)
```

`__smp_*` 是每个架构自己定义的"真指令"。以 x86 为例,出自 [arch/x86/include/asm/barrier.h](../linux-6.14/arch/x86/include/asm/barrier.h#L53-L56):

```c
#define __smp_mb()	asm volatile("lock; addl $0,-4(%%" _ASM_SP ")" ::: "memory", "cc")
#define __smp_rmb()	dma_rmb()
#define __smp_wmb()	barrier()
```

逐行看 x86 的选择,信息量很大:

- **`__smp_mb()`**:生成一条 `lock; addl $0,-4(%rsp)` 指令。`lock` 前缀在 x86 上是一条**全内存屏障**——它强制本核的 store buffer 全部刷出、并排空读队列。这就是 x86 上"全屏障"的样子(用 `lock` 指令而不是 `mfence`,是历史性能选择)。
- **`__smp_rmb()`**:`dma_rmb()`,对普通 SMP 内存而言基本是空的(`barrier()`)。**因为 x86 不允许"读读"乱序**(硬件本来就保证),所以读屏障在 x86 上啥也不用做!
- **`__smp_wmb()`**:就是 `barrier()`——**只是编译器屏障,连 CPU 指令都没有**!因为 x86 也不允许"写写"乱序(写操作按程序顺序对别的核可见)。

> **这就是"为什么 x86 是强序架构"的具体含义**:它对"写写"和"读读"顺序天然保证,所以 `smp_wmb`/`smp_rmb` 在 x86 上**编译后是空的**(只剩编译器护栏)。只有需要全屏障的 `smp_mb` 才生成真指令。
>
> **但这不意味着你在 x86 上能省掉 `smp_wmb`/`smp_rmb`**!内核是跨架构的。你在 ARM/PowerPC(弱序)上跑,`smp_wmb` 就是真指令——省了就 bug。所以**内核代码必须按"最弱的架构"来写**(假设随时会乱序),该插的屏障一个不能少。这是可移植性的代价,也是为什么屏障看着"多余"却必须写。

### 3.2 不这样会怎样?(典型的发布-订阅模式)

回到开头那个例子,现在用 `smp_*` 写对(经典"发布-订阅"模式):

```c
/* 写者 */
WRITE_ONCE(data, 42);     /* 准备数据 */
smp_wmb();                /* 写屏障:data 必须先于 ready 被看见 */
WRITE_ONCE(ready, true);

/* 读者 */
while (!READ_ONCE(ready));
smp_rmb();                /* 读屏障:确认 ready 后才能读 data */
print(READ_ONCE(data));
```

写者的 `smp_wmb` 保证:任何看到 `ready=true` 的核,**必然也已经看到 `data=42`**。读者的 `smp_rmb` 保证:读了 `ready` 之后才读 `data`,顺序不乱。两个屏障**配对**,消息才能正确传递。

> 这是"用标志位 + 数据"做无锁通信的最基本范式。**少任意一个屏障,在弱序 CPU 上就时灵时不灵。** 这类 bug 在 x86 上永远复现不出来(因为 x86 强序),一到 ARM 服务器/手机上就偶发崩溃——**最难调的跨架构 bug**。

## 四、现代首选:acquire/release 语义

上面用 `smp_wmb`/`smp_rmb` 的写法是对的,但内核现在更推荐一种**更精细、更便宜**的写法:**acquire/release 语义**。

### 4.1 什么是 acquire / release

- **release 写**(`smp_store_release(p, v)`):这次写之前(程序顺序里)的所有读写,**都必须在本次写之前、对别的核可见**。也就是"封箱前把所有东西备齐再封"。
- **acquire 读**(`smp_load_acquire(p)`):这次读之后(程序顺序里)的所有读写,**都必须在本次读之后才进行**。也就是"确认盖章后才能拆后续包裹"。

泛型定义出自 [include/asm-generic/barrier.h](../linux-6.14/include/asm-generic/barrier.h#L138-L156):

```c
#ifndef __smp_store_release
#define __smp_store_release(p, v)					\
do {									\
	compiletime_assert_atomic_type(*p);				\
	__smp_mb();							\
	WRITE_ONCE(*p, v);						\
} while (0)
#endif

#ifndef __smp_load_acquire(p)
#define __smp_load_acquire(p)						\
({									\
	__unqual_scalar_typeof(*p) ___p1 = READ_ONCE(*p);		\
	compiletime_assert_atomic_type(*p);				\
	__smp_mb();							\
	(typeof(*p))___p1;						\
})
#endif
```

`compiletime_assert_atomic_type(*p)` 是个安全护栏:**acquire/release 只能用在标量/原子类型上**(一个指针、一个 int),不能用在结构体上。这强迫你"用一个标量当传递点",而不是想 barrier 整个结构体。

泛型实现里,`__smp_store_release` 用了 `__smp_mb()`(全屏障)兜底——但**每个架构可以提供更便宜的特化版本**。看 x86,出自 [arch/x86/include/asm/barrier.h](../linux-6.14/arch/x86/include/asm/barrier.h#L59-L71):

```c
#define __smp_store_release(p, v)					\
do {									\
	compiletime_assert_atomic_type(*p);				\
	barrier();							\
	WRITE_ONCE(*p, v);						\
} while (0)

#define __smp_load_acquire(p)						\
({									\
	typeof(*p) ___p1 = READ_ONCE(*p);				\
	compiletime_assert_atomic_type(*p);				\
	barrier();							\
	___p1;								\
})
```

> **注意 x86 上 acquire/release 只是 `barrier()`(编译器护栏)——零 CPU 指令!** 因为 x86 的内存模型天然满足 acquire/release 语义(它本来就是强序的)。所以 `smp_store_release`/`smp_load_acquire` 在 x86 上**免费**,而在 ARM 上才是真指令。这就是为什么内核推荐用它们:**它们能在保证正确性的前提下,在每个架构上用最便宜的方式实现**——比一刀切的 `smp_mb` 便宜。

### 4.2 用 acquire/release 重写发布-订阅

```c
/* 写者 */
WRITE_ONCE(data, 42);
smp_store_release(&ready, true);   /* release:data 必须先就绪 */

/* 读者 */
while (!smp_load_acquire(&ready)); /* acquire:确认 ready 后才读 data */
print(READ_ONCE(data));
```

比 `smp_wmb`/`smp_rmb` 版本更清晰(屏障和具体的那次读/写绑在一起,意图明确),而且各架构上更便宜。**现代内核首选这种写法**。

## 五、锁和 RCU 内部的屏障(为什么你平时不用自己加)

讲到这里,你可能慌了:"那我内核代码里到处都在读写共享变量,岂不是每个地方都要加屏障?"

**好消息:绝大多数时候你不用。** 因为锁和 RCU **内部已经把屏障编好了**。你只要正确地用锁/RCU,屏障自动就有了。

### 5.1 spinlock / mutex 内部的 acquire/release

回想第 25 章:

- `queued_spin_unlock` 里是 `smp_store_release(&lock->locked, 0)`——**release 屏障**。它保证:你在临界区里所有写,必须在 unlock 之前、对别的核可见。所以下一个拿锁的人,看到的是你完整的临界区结果。
- `__mutex_trylock_fast` 里是 `atomic_long_try_cmpxchg_acquire(...)`——**acquire 屏障**。它保证:你拿锁之后的读,在拿锁之后才进行。所以你读到的是上一个持锁者完整的临界区结果。

**锁 = acquire/release 对**。拿锁是 acquire、释放是 release,这一对屏障保证了临界区"对别的核而言是个整体,既不漏出去、也不被别人插进来"。你只要规规矩矩用锁,**屏障就是锁送的赠品**,你完全不用操心。

### 5.2 RCU 内部的屏障

第 26 章已经讲透:

- `rcu_assign_pointer` 内部是 `smp_store_release`——release 屏障。
- `rcu_dereference` 内部是 `smp_load_acquire`——acquire 屏障。

所以你只要用 `rcu_assign_pointer`/`rcu_dereference` 这对官方 API,RCU 的"读旧版、写新版"的正确性屏障就自动有了。

### 5.3 那么,什么时候你才真的需要亲手加屏障?

**只有一种情况:你写"无锁(lockless)"代码**——不借助锁/RCU,自己用原子变量、标志位做同步。比如:

- 自己实现的发布-订阅(`data` + `ready` 标志位)。
- 无锁环形缓冲(producer/consumer 用原子头尾指针)。
- `seqlock`(顺序锁,用版本号实现"读时不锁")。

这些场景,锁不帮你背锅,你必须亲手用 `smp_store_release`/`smp_load_acquire`(首选)或 `smp_wmb`/`smp_rmb`(老式)。

> **心法**:**屏障不是"加了更安全"的调料,而是"无锁同步时必须精确配对"的契约**。多加一个不需要的屏障顶多浪费一点性能;**少加或加错**才是灾难。写无锁代码,先想清楚"谁 release、谁 acquire、配对没",再动手。

## 六、smp_mb__before/after_atomic:原子操作配套屏障

最后一个容易踩的点:**原子操作本身不一定带全屏障**。

很多原子操作(比如 `atomic_inc`、`atomic_set`、`clear_bit`)只保证自己**这一次操作原子**,但不保证**前后的内存访问顺序**。看 x86,出自 [arch/x86/include/asm/barrier.h](../linux-6.14/arch/x86/include/asm/barrier.h#L75-L76):

```c
#define __smp_mb__before_atomic()	do { } while (0)
#define __smp_mb__after_atomic()	do { } while (0)
```

在 x86 上这俩是空的——因为 x86 的原子操作(带 `lock` 前缀)本来就是全屏障,自带顺序保证。**但在弱序架构上它们是真屏障**。所以内核提供了 `smp_mb__before_atomic()` / `smp_mb__after_atomic()`(泛型定义在 [include/asm-generic/barrier.h](../linux-6.14/include/asm-generic/barrier.h#L164-L168)),让你在"原子操作前后需要顺序保证"时显式插上。

典型用法:

```c
/* 想保证:写 other_stuff 必须在 clear_bit 之前、对别的核可见 */
other_stuff = x;
smp_mb__before_atomic();
clear_bit(BIT_FLAG, &flags);
```

> **什么时候需要它?** 当你用一个**非 `acquire/release` 语义的原子操作**(普通的 `set_bit`/`clear_bit`/`atomic_inc` 等)来"传递同步信息",且需要它前后的普通变量访问也有顺序保证时。**用 `atomic_fetch_add_release` 这种带语义的新式 API,就不需要它了**(语义自带)。所以现代内核倾向用带 acquire/release 后缀的原子 API,绕开这套 `before/after` 的心智负担。

## 关键源码精读:x86 为什么"白送"这么多屏障

本章最值得逐行看的,是 x86 那几个"看起来啥也没干"的屏障宏。它们浓缩了"内存模型是架构相关"这个全章核心。出自 [arch/x86/include/asm/barrier.h](../linux-6.14/arch/x86/include/asm/barrier.h#L53-L71):

```c
#define __smp_mb()	asm volatile("lock; addl $0,-4(%%" _ASM_SP ")" ::: "memory", "cc")
#define __smp_rmb()	dma_rmb()
#define __smp_wmb()	barrier()
...
#define __smp_store_release(p, v)					\
do {									\
	compiletime_assert_atomic_type(*p);				\
	barrier();							\
	WRITE_ONCE(*p, v);						\
} while (0)
```

逐行对应设计:

1. **`__smp_mb()` 是唯一生成真指令的**——一条 `lock` 前缀指令。因为 x86 是 "TSO(全存储定序)"模型:它**保证"写写"和"读读"顺序**(所以 `rmb`/`wmb` 空),但**不保证"读写"和"写读"的全顺序**(store→load 之间有 store buffer 缓冲),所以需要全屏障时还得真指令。
2. **`__smp_rmb()` = `dma_rmb()`**(普通 SMP 情况下退化为 `barrier()`):**CPU 这层啥也不做**,只留编译器护栏。因为 x86 不允许"读后读"乱序。
3. **`__smp_wmb()` = `barrier()`**:**同样只挡编译器**。因为 x86 保证"写后写"按程序顺序对别的核可见(store buffer 是 FIFO,不会把先发的写排到后写的之后)。
4. **`__smp_store_release` 也只是 `barrier()`**——因为 x86 的 TSO 模型天然满足 release 语义(release 要求"之前的写在本次写之前可见",而 x86 的写后写本来就保序)。所以 release 在 x86 上免费。

> **这几行揭示了一个深刻的事实**:**"内存屏障"在不同架构上是完全不同的东西**。同样的 `smp_store_release`,在 x86 上编译成零指令(只剩编译器护栏),在 ARM 上编译成 `stlr`(一条 release 语义的 store 指令)。**内核的屏障 API 是"跨架构的统一抽象"——它让你写一份代码,在每个架构上都用该架构最便宜、又正确的实现。**
>
> 这也是为什么本章反复强调"按最弱架构写代码":你写 `smp_store_release` 时,要假设它后面藏着真正的 CPU 重排(像 ARM 那样),这样你的代码才能在任何架构上正确。x86 的"白送"只是开发体验上的福利,不是你可以省屏障的理由。

## 章末小结

用"档案室"比喻收一下:

- **代码里写的顺序 ≠ 档案真正生效的顺序**。编译器(为了优化)和 CPU(为了流水线快 + store buffer)都会偷偷重排。
- **`READ_ONCE`/`WRITE_ONCE`** = 告诉档案员"这几份材料的存取,不许你自作主张优化/重排"(挡编译器)。
- **`smp_*` 屏障** = 在档案流程里设"铁闸",强制"之前的手续必须先于之后的,被别的窗口看见"(挡 CPU)。
- **acquire/release**(`smp_store_release`/`smp_load_acquire`)= 更精细、更便宜的"配对铁闸",是现代首选。
- **好消息**:锁(spinlock/mutex)和 RCU **内部已经编好了屏障**——你只要正确用它们,屏障就是赠品。**只有写无锁代码才需要亲手上屏障**。

**回扣全篇**:这一章把第 24~26 章里所有"偷偷藏起来"的屏障补全了——

- 第 24 章:并发有三个来源 → 需要同步。
- 第 25 章:锁圈起临界区 → 但锁的"圈"靠的是 unlock 的 release、lock 的 acquire(本章)。
- 第 26 章:RCU 读者零开销 → 但"读旧版、写新版"的正确性靠 `rcu_assign_pointer`/`rcu_dereference` 那对屏障(本章)。
- 第 27 章(本章):屏障本身——挡住编译器(READ_ONCE/WRITE_ONCE)和 CPU(smp_*/acquire/release),保证"改的顺序被正确看见"。

至此,**并发与同步的四副药全齐了**:认清敌人(并发来源)→ 锁(挡住别人)→ RCU(读者免费)→ 屏障(保证顺序)。它们组合起来,支撑着整个内核在多核、中断、抢占三重夹击下依然正确运行。

**回扣全书主线**:屏障属于"管资源"那一侧的**最底层基础设施**——它管的是"内存访问的可见性顺序",这是所有上层同步原语(锁、RCU、原子操作)的地基。你看不到它,但它在每一把锁、每一次 RCU 替换下面默默工作。

**想继续深入该往哪钻**:
- **内存模型圣经**:`Documentation/memory-barriers.txt`(在 [Documentation/](../linux-6.14/Documentation/) 下)——内核作者写给驱动/子系统开发者的屏障使用规范,几十页,极其详尽。**这是本章的"原始资料",读完它你就算屏障入门了。**
- **lockless 编程范式**:`seqlock`([include/linux/seqlock.h](../linux-6.14/include/linux/seqlock.h))——用版本号实现"读时不锁"的又一经典,综合运用了 `smp_wmb`/`READ_ONCE`。
- **KCSAN(并发净化器)**:内核的数据竞争检测器,打开 `CONFIG_KCSAN` 跑,它会帮你抓"两个执行流无保护地碰同一变量"的隐患——和 lockdep 一样,是学并发最好的老师。
- **跨架构对比**:把 [arch/x86/include/asm/barrier.h](../linux-6.14/arch/x86/include/asm/barrier.h) 和 `arch/arm64/include/asm/barrier.h` 对照看,体会"强序 vs 弱序"在代码上的真实差别——你会立刻理解为什么"按最弱架构写代码"是必须的。
