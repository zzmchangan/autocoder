# 导言 · 一篇看懂"中断、异常与系统调用"全貌

> 这篇是本篇的"地图"。读各章之前先读它,你会对"市民什么时候、为什么必须惊动市政府"建立一个结实的直觉。
> 全文只讲两件事:**内核为什么会被迫介入**,以及**介入的这扇门怎么过**。
> 我会用一个贯穿全篇的比喻——**市政府的门、门铃、报警器、办事窗口**——帮你记住每一个机制。

---

## 一、一句话说清本篇在干嘛

前面进程篇讲过:市民(进程)在自己的世界里闷头跑,调度器在飞速轮换,让每个市民都觉得"我独占了办事窗口(CPU)"。但你有没有想过一个更要命的问题:

> **市民在自己的世界里跑得好好的,内核凭什么能"突然插进来"接管?网卡收到一个包、你按下键盘、进程除以零、进程想读文件——这些事件,是怎么让正在跑的市民"让一让",把内核从后台叫到前台的?**

答案只有三个字:**陷入内核(trap)**。

> ⚠️ **本篇就讲这一件事:市民和政府之间的那道边界,什么时候、为什么、怎么被跨过去。** 内核的所有"主动介入",都发生在这道边界被跨越的瞬间。

跨越这道边界的情况,只有三种。它们表面上八竿子打不着,内核却把它们当成**同一类事**来处理。先记住这张"三种惊动"的总图:

| 是谁在惊动政府? | 怎么惊动的? | 内核概念 | 本篇比喻 |
|---|---|---|---|
| **外部硬件**主动 | 门铃响了(网卡/键盘/时钟) | **中断(IRQ)** | 门铃 |
| **市民自己**闯祸 | 触发了 CPU 的报警(除零、缺页、非法指令) | **异常(exception / trap / fault)** | 报警器 |
| **市民主动**申请 | 排队去窗口办事(读文件、建进程) | **系统调用(syscall)** | 办事窗口 |

记住这张表,本篇的所有复杂设计——IDT、softirq、上半部下半部、hrtimer、`syscall` 指令——全都是在回答:**"这三种惊动,门到底怎么过、过了之后怎么处理、处理完怎么回来。"**

## 二、建立比喻:政府的三道门

市政府([全书导言](全书导言-一篇看懂内核全貌.md)里说了:内核 = 市政府)有三道门,对应三种惊动。我们把它们一个个摆开:

### 门 1:门铃(中断 / IRQ)

外部硬件(网卡、键盘、磁盘、USB)自己没有 CPU 时间,它们就像市政大厅外排着队的来访人,**按门铃**通知政府:"我这儿有事,来处理一下"。

- 门铃一响,政府里**正在办事的市民**必须立刻停下来(不管他办到一半没),让政府去处理门铃带来的事。
- 处理完,市民**从中断的地方继续**——他甚至可能根本不知道自己被打断过。
- 这是最典型的"异步":门铃什么时候响,政府事先**不知道**,得随时待命。

> 这就是为什么你的键盘输入从不会丢:键盘一按,门铃一响,内核立刻冲出来把按键收走存好,等进程来读。**硬件不用等进程"主动来问",它主动拍政府肩膀。**

### 门 2:报警器(异常)

报警器和门铃的区别在于:**它是市民自己触发的**。市民在办事过程中,自己干了件"违法/出格"的事——比如除以零、访问了不存在的地址、执行了非法指令。CPU(而不是外部硬件)**当场**拉响报警,政府自动介入。

- 异常是**同步**的:它一定发生在某条**具体的指令**上,你能精确定位到"就是这条指令惹的祸"。
- 异常分三种命运(后面细讲):
  - **fault(故障)**:比如 [page fault](P2-09-page-fault-虚拟到物理的按需兑现.md)——市民敲门发现门后没房间。政府介入后**可能能修好**(给他配间房),修好让市民**重新执行那条指令**。
  - **trap(陷阱)**:故意触发的,比如调试断点。处理完**继续往下一条指令**。
  - **abort(致命)**:出大问题了(硬件错、内核栈坏了),通常**救不回来**,整个进程或系统挂掉。

> 内存篇的 [page fault 章](P2-09-page-fault-虚拟到物理的按需兑现.md)其实就是异常的典型样本。那一章讲的是"市民敲门要房间"——这正是报警器响了、政府来兑现。本篇第 1 章会把异常这一类讲全。

### 门 3:办事窗口(系统调用)

市民平时不能自己动手碰公共资源——你不能自己跑去磁盘上扒数据、不能自己改别人的内存、不能自己创建新市民。**所有"动用公共资源"的事,都必须去办事窗口排队申请**。

- 系统调用是**同步、自愿**的:市民**故意**发起(`syscall` 指令),政府按规矩办,办完把结果交还市民。
- 这是**安全和隔离的根本**:正是因为"市民不能自己动手,只能申请",才有了进程之间的隔离、才有了权限检查、才有了"内核态 vs 用户态"这道墙。
- 你写程序调用的 `read`/`write`/`fork`/`malloc`(底层 `brk`/`mmap`)……每一个,都是一次"去窗口办事"。

> 一个震撼的事实:一个普通程序,**几乎所有"有意义的动作"最终都是系统调用**。计算、判断、循环在你自己房间里搞(用户态);一旦要碰外部世界(文件、网络、屏幕、新进程),就必须"出门去窗口"。系统调用就是这扇门。

## 三、三种惊动的"统一内核":都是陷入内核

现在最关键的一步:这三种惊动表面上八竿子打不着(硬件、市民闯祸、市民申请),**内核却把它们当成同一类事来处理**。为什么?

因为对 CPU 来说,**它们都是同一件事的化身**:

> **"CPU 正在用户态(市民世界)跑着,突然要切到内核态(政府世界)去执行一段内核代码,完了再切回来。"**

这件事,硬件上要做的"过门动作"是**一模一样**的:

1. **保存现场**:市民办到一半,得把他的"手里那摊事"(寄存器、指令位置)记下来,免得回来接不上。
2. **换世界**:从用户态切到内核态——换栈、换页表基地址(内核地址空间)、把 CPU 的运行级别从"低权限"提到"高权限"。
3. **查表找人**:内核预先登记了一张**"门牌总表"**(x86 上叫 **IDT,Interrupt Descriptor Table**)。每种惊动一个编号,CPU 拿编号去表里查"这事该交给哪个处理函数"。
4. **执行处理函数**:内核代码上场处理。
5. **恢复现场、回门**:把刚才记的现场原样恢复,CPU 切回用户态,市民从中断处继续。

**中断、异常、系统调用,走的是同一条 5 步流水线。** 它们的区别,仅仅在第 0 步——**"是谁、怎么触发的"**:

- 中断:外部硬件在某个管脚上发了个信号,CPU 硬件自动查 IDT。
- 异常:某条指令执行时,CPU 自己检测到状况,硬件自动查 IDT。
- 系统调用:进程故意执行了一条 `syscall` 指令,这条指令"约定俗成"地指向 IDT 里的系统调用入口。

> 这是本篇最重要的一句认知:**三种惊动是"同一扇门的三种敲门方式"。** 内核用同一套基础设施(IDT、上下文保存/恢复、`irqentry_enter/exit`)服务它们。第 4 章会专门把这张统一图收束起来。

## 四、跟着门铃走一遍:中断处理的全貌

三道门里,**中断(门铃)** 是最复杂、最值得细看的——它牵出"上半部 / 下半部""softirq""工作队列"一整套机制。我们顺着门铃从响到处理完,走一遍全程。系统调用和异常是同一条流水线的变种,理解了门铃,它们就好懂了。

### 第 0 步:硬件按门铃

网卡收到一个包,它在主板上的一根**中断线**上发了个电信号。这个信号先到**中断控制器**(现代 x86 是 **APIC,Advanced Programmable Interrupt Controller**),由它汇总、排队、决定优先级,再通过一根专门的管脚去**戳 CPU**。

> **为什么不让设备直接戳 CPU?** 因为设备多得几十上百个,CPU 的管脚就那么几个。中断控制器像个"门卫室",负责把上百个门铃汇总成 CPU 能处理的少数几根线,还能给它们排优先级(时钟中断 > 普通设备中断,免得一个网卡把 CPU 占满,连心跳都跳不动)。

CPU 被戳到后,会**立刻**中断当前正在跑的指令(连一条指令执行到一半都打断),保存现场,查 IDT。

### 第 1 步:查 IDT——门牌总表

CPU 手里有一张表,**IDT**(本仓库 [arch/x86/kernel/idt.c](../linux-6.14/arch/x86/kernel/idt.c) 是它的"登记册")。这张表的每一项是一个**门描述符**,写着"这个编号的中断/异常,入口代码在哪"。

- 表是按**编号(vector)**索引的:0~31 号是 CPU 保留给**异常**的(除零是 0 号、page fault 是 14 号……);32 号往后才是**外部中断**。
- 时钟中断、网卡中断各有自己的编号。CPU 拿到编号,直接查表拿到入口地址,跳过去——**这一步是硬件做的,纯查表,O(1)**。

> **比喻**:IDT 就是市政大厅一进门的那块**"各部门楼层分布图"**。来访者(中断信号)进门先看图:"哦,门铃 #247 是网科的,在 3 楼"。然后直接上楼,不会在大厅里乱转。

### 第 2 步:过门——保存现场 + 切到内核

入口地址指到的是一段极短的**汇编入口代码**(x86 上是 `entry_64.S` 里宏生成的一堆 stub)。这段代码干的就是"过门 5 步"里的前 2 步:

- 把市民的寄存器一股脑压栈(保存现场,内核里叫 **`struct pt_regs`**——你将在系统调用章反复看到它);
- 切到**内核栈**(每个线程都有自己的内核栈,专门为"陷入内核时"准备);
- 切到内核地址空间(CR3 寄存器换成内核页表);
- 把 CPU 提到内核权限级。

> 注意:此刻**中断通常是关掉的**(本 CPU 上)。因为内核正在"进门"的临界状态,不能再被打断,否则现场会乱。这也是后面"为什么硬中断里不能干太久"的根源——**你占着关中断的 CPU,这期间整个核对外界充耳不闻**。

### 第 3 步:通用入口宏 `irqentry_enter`

过门之后,内核有一段**三种惊动共用的骨架代码**:`irqentry_enter`([kernel/entry/common.c](../linux-6.14/kernel/entry/common.c))。它做账务:记账(这次中断耗了多少时间)、进入 RCU 临界区、更新抢占计数(标记"现在我在硬中断里,调度器别来抢我")。这是"统一性"的代码铁证——不管你是中断、异常还是系统调用,进门都得先过它。

### 第 4 步:分发——这个中断归谁管?

到这儿,通用的"过门"做完了,该具体处理了。对**外部设备中断**,内核有一套专门的多层抽象:

- **`irq_desc`**(中断描述符):每一个 IRQ 号对应一个,记录"这个中断的硬件特性、它的处理器(handle_irq)、挂在上面的驱动 action 链"。
- **`handle_irq`(高层流控处理器)**:根据中断是边沿触发还是电平触发,决定"先应答硬件还是先跑驱动"——比如 `handle_fasteoi_irq`、`handle_edge_irq`。
- **`action` 链(驱动处理函数)**:一个 IRQ 可以被多个驱动共享(比如 PCI 设备),内核挨个调用链上的处理函数,看是谁的事。

最终调用到的,就是你写驱动时用 `request_irq` 注册的那个**中断处理函数**。

> **为什么要叠这么多层?** 因为硬件五花八门:有的是电平触发、有的是边沿触发,有的要回写 EOI 寄存器应答、有的不用,有的能共享 IRQ、有的不能。内核用 `irq_desc` + 流控处理器这一层,**把"硬件怎么个脾气"和"驱动要干啥"解耦**——驱动只管写"我收到中断了,该干什么",硬件特性交给流控层。这和内存篇"分层抽象、清晰接口解耦"是一个套路。

### 第 5 步:上半部干完——但活儿没完

你的驱动处理函数跑完了,`irqentry_exit` 恢复现场、开门让市民继续。看起来结束了?

**问题来了**:很多中断的真正处理是**费时**的。比如网卡收到一个包,后半段要把包往协议栈里送、做校验、唤醒等待的进程——这些活儿可能要微秒级。可你现在是**在硬中断上下文、关着中断、占着 CPU**。你要是在这儿慢慢磨:

- 别的中断进不来(键盘、时钟都被堵着);
- 当前市民的指令被卡住(他啥也干不了);
- 最要命的——**硬中断里不能睡眠**(它没有进程上下文可以挂起),连普通的锁都不能等。

**不这样会怎样?** 网卡一次中断就磨几微秒,网络一忙 CPU 全被中断占满,系统卡死;或者中断里等了一把被进程持有的锁,直接死锁。

### 第 6 步:下半部——把费时活儿推迟

所以内核的解法是经典的"**推迟执行**":**上半部(硬中断)只做最要紧的一丁点(把数据从硬件扒下来、记一笔"有活儿要干"),真正费时的活儿推迟到下半部。** 下半部开中断跑,能被抢占,有些还能睡眠。

下半部有三种实现,各管一摊:

- **softirq(软中断)**:最底层、最快。在中断返回时(`irq_exit`)就地检查"有没有挂起的软中断",有就**在同一个 CPU 上、开着中断**跑。延迟最低,但仍然不能睡眠。`handle_softirqs`([kernel/softirq.c](../linux-6.14/kernel/softirq.c))是它的引擎。
- **tasklet**:建立在 softirq 之上的薄封装,**同一 tasklet 不会在多核上同时跑**(对写驱动的人友好),6.x 里在逐步淘汰。
- **工作队列(workqueue)**:把活儿丢给一个**内核线程**去干。因为是在线程上下文里,**可以睡眠**——那些"中断里要分配大内存、要等 IO"的活儿只能用它。

> **回扣比喻**:门铃响了 → 门卫(上半部)冲到门口,先把来访者领进门、登个记(扒数据),**马上把门让出来**(开中断、返回);然后让后厨(下半部)在后台慢慢处理来访者带来的事——后厨可以分批、可以排队、甚至可以打烊轮班(工作队列,能睡眠)。门卫绝不在门口磨蹭。

这就是本篇第 1 章的全部主线:**硬件按门铃 → IDT 查表 → 过门 → 通用骨架 → 分发到驱动 → 上半部扒数据、记一笔 → irq_exit 触发下半部 → 下半部干费时活**。

## 五、一种最特殊的门铃:时钟(本篇第 2 章)

上面这套门铃机制里,有一个门铃**最特殊、最重要**——**时钟中断**。它是全城的"心跳",第 2 章专讲它。

为什么特殊?因为别的外设门铃是"有事才响"(网卡没包就不响),而**时钟门铃是定时响个不停**(比如每秒 250 次/1000 次)。这一个不断响的门铃,驱动了内核里几乎所有"和时间有关"的事:

- **调度**:内核怎么知道"这个市民用窗口用了够久、该换人了"?靠时钟中断周期性地戳一下调度器(`sched_tick`)。
- **定时器**:你 `sleep(1)`、你设了个 100ms 超时——内核怎么知道时间到了?靠时钟中断检查"哪些定时器到点了"。
- **记账**:CPU 时间片用在哪了、进程跑了多久,都靠时钟累计。

现代内核更进一步:**hrtimer(高精度定时器)** 让内核从"傻乎乎每秒滴答 1000 次"进化到"只在下一个定时器到期时才让时钟响一次"——空闲时甚至**停掉心跳(tickless / NO_HZ)省电**。

> 时钟这一章是第 1 章的"特例深挖":它就是一个普通的中断,但它特殊到值得单开一章——因为**它是调度和定时器的共同心脏**,牵出进程篇、同步篇一堆线索。

## 六、办事窗口的全貌:系统调用(本篇第 3 章)

最后一道门是**办事窗口(系统调用)**。它和门铃走同一条"过门 5 步"流水线,但有自己的味道,第 3 章专讲。

关键区别:

- **谁触发**:市民**故意**执行一条 `syscall` 指令(x86-64 上的专用指令)。不是硬件信号,不是 CPU 自动,是进程自愿。
- **怎么传诉求**:市民把"要办什么事"用**系统调用号**放进寄存器(比如 `read` 是几号),把**参数**也放进约定好的寄存器,然后执行 `syscall`。
- **`syscall` 指令替我们干了什么**:它原子地"换世界"——`swapgs` 切 GS 基址、换到内核栈、切 CR3 到内核页表(防 Spectre),然后在栈上构造那个 `struct pt_regs`(市民的现场)。这一切都是**硬件 + 一小段汇编**在指令里完成的,**比老式的 `int 0x80` 快得多**。
- **查表找人**:系统调用号去查 **`sys_call_table`**([arch/x86/entry/syscall_64.c](../linux-6.14/arch/x86/entry/syscall_64.c)),拿到对应的 `sys_xxx` 处理函数(比如 `sys_read`)。
- **回来**:处理完,`sysret` 指令快速切回用户态(比通用的 `iret` 快)。

> **为什么必须从办事窗口走,不能让市民自己动手?** 这是**安全和隔离的根本**。设想如果进程能直接执行"读磁盘扇区"的特权指令:那任何程序都能读别人的文件、能直接改硬件、能绕过权限检查——整个隔离崩塌。所以 CPU 设计上,**特权指令只在内核态能执行**;市民想干这些事,唯一的合法途径就是"去窗口申请,由内核代劳并检查权限"。

这一章你会看到 Linux 最经典的"硬件 + 汇编 + C"协作:一条 `syscall` 指令怎么在纳秒级完成"世界切换",`SYSCALL_DEFINE` 这套宏怎么把一个 C 函数变成表项,`do_syscall_64` 怎么防 Spectre 式越界读。

## 七、收束:本篇的"两句话"

走完全篇,你看到的其实是**一张统一的图**:

```
                  市民在用户态跑(市民世界)
                          │
        ┌─────────────────┼─────────────────┐
   门铃响了            市民闯祸           市民去窗口
  (中断 IRQ)        (异常 fault)      (系统调用 syscall)
        │                 │                 │
        └────────┬────────┴────────┬────────┘
                 ▼                 ▼
         都是"陷入内核"——同一条过门流水线
                 │
        IDT 查表 → 保存现场(pt_regs)→ 切内核态
        → irqentry_enter → 处理 → irqentry_exit → 恢复现场 → 回用户态
                 │
   ┌─────────────┼──────────────────┐
   ▼             ▼                  ▼
 中断:异步、外部、  异常:同步、CPU 自动、  系统调用:同步、自愿、
 关中断上下文、      和当前指令绑定、       有明确"当前进程"、
 可推迟到下半部      通常关联"当前进程"     权限检查严格
```

### 记住这两句话,就抓住了本篇

1. **中断、异常、系统调用,本质都是"陷入内核",区别只在"谁发起、为什么"。** 内核对它们用同一套骨架(IDT / pt_regs / `irqentry_enter`),你理解了其中一种,另两种就是它的变种。
2. **硬中断里"不能干太久、不能睡眠"是被逼出来的硬约束**——它驱动了"上半部 / 下半部""softirq / 工作队列"这一整套推迟执行的设计。当你看到任何一个"为什么要把活儿拆出来异步做"的设计,去找这个根源。
