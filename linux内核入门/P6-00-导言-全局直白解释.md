# 第 6 篇导言 · 一篇看懂文件与存储

> 这篇是本篇的"正门"。读任何一章之前先读它,你会对"市民的资料怎么存、怎么找、怎么不丢"建立结实的直觉。
>
> 全文只讲两件事:**文件系统在干什么**,以及**为什么非得这么干**。我会用城市比喻里的**档案馆 + 地下仓库**,帮你记住每一个机制。

---

## 一、一句话说清文件系统在干嘛

> **文件系统(VFS)是这座城市的档案馆。它干的所有事,就是让每个市民都觉得"世界上一切都是文件"——硬盘是文件、键盘是文件、网卡是文件、连进程信息都是文件。市民用同一套 open/read/write/close 就能摆平一切,完全不知道底下硬盘长什么样。**

而真正存东西的,是档案馆底下的**地下仓库(磁盘/块设备)**。仓库又慢、又笨、还会突然掉电。档案馆的全部技术含量,就在于**在市民的"文件幻觉"和仓库的"笨重现实"之间,架一座又快又安全的桥**。

你任何时候看不懂本篇某个机制,就回来问自己:

> **"这个机制,是在'维护文件的统一幻觉',还是在'对付仓库的笨重和不可靠'?"**

本篇所有章节都在这两件事之一:

- **造幻觉(VFS 侧)**:superblock/inode/dentry/file 四张抽象卡片 + 一张操作表。硬盘、键盘、网卡千差万别,统统被它们伪装成"文件"。这是内核卖的**第三个幻觉**。
- **对付仓库(块设备侧)**:page cache 当前台快取、bio/request 两层排队、blk-mq 多队列调度、writeback 攒批写回、journaling 先记流水。让慢仓库表现得快、笨仓库表现得聪明、会掉电的仓库表现得可靠。

这正好是内存篇"管资源 / 造幻觉"二分法在本篇的复刻。你已经见过内存篇怎么玩这个二分法,这里只是换了套道具。

---

## 二、回顾一下:三个幻觉走到哪了

全书一直在讲内核卖的三套幻觉:

1. **独占 CPU 的幻觉**——调度器飞速轮换,你以为独占了窗口(进程/调度篇)。
2. **独占内存的幻觉**——虚拟内存让你以为全城房子都是你的(内存篇)。
3. **统一的文件/设备世界的幻觉**——硬盘、键盘、网卡千差万别,VFS 让你都觉得是"文件"(**本篇**)。

本篇负责把第三套幻觉讲透。讲完,市民程序赖以生存的三层幻觉就齐了:CPU 是我的、内存是我的、世界是文件。一个程序活在这三层幻觉里,安心地写代码,完全不知道底下市政府忙成什么样。

## 三、建立比喻:档案馆 + 地下仓库

| 城市比喻 | 内核概念 | 一句话 |
|----------|----------|--------|
| 档案馆 | 文件系统(VFS) | 统一接待市民、管档案的地方 |
| 一份档案 | **文件**(file) | 市民眼里的"文件",有名字、有内容 |
| 档案的目录卡 | **dentry** | 文件名 ↔ inode 的映射(目录就是一堆卡) |
| 档案本身的元信息 | **inode** | 这份档案多大、谁建的、存在哪些货架——唯独**没有名字** |
| 整个档案馆的户口本 | **superblock** | 这个文件系统的总账:多大、怎么分块、根目录在哪 |
| 市民手里那张"借阅条" | **struct file**(一个 fd) | 一次具体的打开:从哪读到哪、用什么模式 |
| 档案馆前台快取柜 | **page cache** | 最近读过的文件页先放这儿,下次直接给,别老往仓库跑 |
| 前台快取柜的索引账本 | **address_space** | 一个文件一份账:"第 N 页在不在前台、脏不脏" |
| 地下仓库 | 磁盘 / 块设备 | 真正存东西的地方,慢、笨 |
| 货架 | 磁盘扇区/块 | 仓库的最小存取单位 |
| 取货/存货的运货单 | **bio** | 文件系统发给仓库的原始 I/O 单元 |
| 攒成的大宗运货单 | **request** | blk-mq 把多个 bio 合并、排队后的单子 |
| 仓库调度室 | **块设备层(blk-mq)** | 排队、合并、调度,让仓库别乱 |
| 改账前先记的流水簿 | **journal(日志)** | 先把"要怎么改"写下来,掉电了也能照着改完或撤销 |

这张表是本篇术语的总开关。每遇到新名词,先回来查它的"户口"。

## 四、跟着一次 `read()` 走一遍全篇

整篇的脉络,就是一次市民读文件的全部旅程。我们先把整条路走一遍,每经过一站,对应后面一章。

### 起点:市民说"我要读 /etc/hosts"

市民不能自己动手碰公共资源(这是边界篇讲的系统调用铁律),于是它发起系统调用 `read(fd, buf, 4096)`,排队去办事窗口。内核接管。

### 第 1 站:VFS——管你底下是什么,先按"文件"规矩来(第 28 章)

内核手里拿到的是一个 `fd`(借阅条)。它顺着 `fd → struct file(这次打开)→ dentry(目录卡)→ inode(档案本身)` 一路摸到这份档案。

关键在于:**VFS 不关心这个文件到底是 ext4 上的真文件,还是 /proc 里的虚拟文件,还是设备文件**。它只认四张抽象卡片,以及卡片背后那张**操作表**(一堆函数指针)。`read` 到底执行什么代码,取决于 inode 上挂的是哪套操作表——这就是"一切皆文件"骗术的全部秘密。

> 这一站讲透:**为什么是函数指针表让 VFS 能挂上百种文件系统**。

### 第 2 站:这份档案真身在仓库里长什么样(第 29 章)

VFS 只管抽象,真把数据摆上货架的是 ext4。市民的 `/etc/hosts` 在磁盘上不是连续的一坨,而是被切成块,散落在某个 block group 里;它的元信息(多大、谁建的、数据块在第几个货架)记在一张叫 inode 的小卡片上,和别的几万个 inode 挤在 inode 表里。

> 这一站讲透:**磁盘上的字节是怎么布局的,以及它怎么和 VFS 的内存对象对上**。

### 第 3 站:先问前台快取柜,别急着下仓库(第 30 章)

VFS 摸到 inode 后,第一件事不是冲去仓库,而是问 page cache:"这份文件的第 0 页,前台有没有?"

- 有(命中):直接把内存里的页拷给市民,根本不碰磁盘。**这是 `ls`、`cat` 第二次飞快的真正原因。**
- 没有(miss):这才往下走,去仓库取。而且 readahead 会顺手多取几页——"既然都下仓库了,不如一次取足"。

> 这一站讲透:**page cache 是文件系统和内存管理的交叉点**(内存篇讲过它的内存面,这里讲它的文件面)。

### 第 4 站:真要下仓库,得排队合并调度(第 31 章)

前台没货,page cache 发起一次真正的 I/O。文件系统把"读这 4KB"翻译成"读磁盘扇区 X 到 Y",包成一个 `bio` 交给块设备层。

块设备层(blk-mq)是仓库调度室。它不会傻到每个 bio 都立刻发一辆叉车——它把相邻的 bio **合并**成一个 `request`,攒一批一起送,还按磁盘物理顺序排好队,**让磁头少跑冤枉路**。

> 这一站讲透:**bio → request 两层模型 + 多硬件队列 + plug 机制**,为什么现代 NVMe 能跑满带宽。

### 第 5 站:写回去,而且要保证掉电不丢(第 32 章)

读到此结束。但如果是写呢?市民 `write` 只是把 page cache 里那页**标记成脏**就返回了——真正的落盘是"后来的事"。

谁后来写?**写回线程**。什么时候写?脏页攒到一定比例、定时器到了、或内存紧张被逼着回收——三种触发。怎么写不丢?**journaling**:改档案前,先把"我要怎么改"记进流水簿,真改的时候就算掉电,重启后照流水簿把没改完的补上或撤销,档案永远处于自洽状态。

> 这一站讲透:**写回的平衡之道 + 日志为什么能扛掉电**。

### 终点:数据回到市民手里

走完全程,你现在能完整解释一行 `cat /etc/hosts`:

> shell fork 出 `cat` → `cat` 调 `open` 系统调用 → VFS 顺 fd 摸到 inode → `read` 问 page cache,命中就直接给,miss 就发 bio → 块设备层合并调度后递交磁盘驱动 → 数据回到 page cache → 拷给市民 → `cat` 调 `write` 写到屏幕。

这一行命令,穿过了**系统调用 → VFS → 具体文件系统 → page cache → 块设备 → 磁盘**六层。本篇就是把这条线一层层拆开。

## 五、本篇的两个反复出现的"为什么"

走完全篇你会发现,所有复杂设计都被两个现实逼出来:

1. **磁盘慢,而且慢得离谱**(比内存慢 10 万倍以上)。→ 逼出 page cache(别老下仓库)、逼出 bio/request 合并(下了就一次取足)、逼出预读(顺手多取)。
2. **磁盘会掉电,掉电会改坏档案**。→ 逼出延迟写回(攒批)、逼出 journaling(先记流水)。

而 VFS 那套"四张卡片 + 操作表"的统一幻觉,则是被**第三个现实**逼出来的:

3. **存储设备千差万别**(硬盘、U 盘、网络盘、内存盘、虚拟文件)。→ 逼出 VFS 抽象层,让市民程序不用认识几十种方言。

> **记住这个三角:慢、会掉电、长得不一样。** 本篇所有机制,都是被这三个现实逼出来的工程妥协。当你觉得某个设计"为什么搞这么复杂",去找是哪个现实在作怪。
