# 第 8 章 · 虚拟内存与多级页表:为什么要"骗"进程

> **核心问题**:进程根本不碰物理地址,它看到的是"独占的、连续的、从 0 开始"的虚拟地址空间。这个幻象为什么必要,又是怎么实现的?

> 读完本章你会明白:
> - 为什么内核要费尽心机给每个进程编造一个"整栋楼都是我的"幻觉。
> - 为什么一张平铺的大页表(单层)根本不可行——光是记账就要 512GB。
> - 多级页表如何用"懒建账"把稀疏映射"压扁",只花几页内存就能管理 TB 级空间。
> - 为什么 TLB 是页表的"速查缓存",以及为什么整本书的页表设计都在围着 TLB 转。

> **如果一读觉得太难**:先只记住三件事——① 虚拟内存就是给每个进程发一套**独立门牌表**,靠**页表**把"我的门牌号"翻译成"真实房间",隔离和保护是这套翻译机制免费送的红利;② 千万别用平铺单层页表(光是给一个进程记全就要 512GB),要用**多级页表树**,只为真正用到的地址段建子账,所以几十 KB 就能管 TB 级空间;③ **TLB** 是页表的速查缓存,多级页表(省)和 TLB(快)是一对缺一不可的孪生兄弟。其余细节第二遍配合源码再抠,不影响往下读。

## 一、视角转换:从"切房间"到"造幻觉"

前 7 章,我们一直待在**物理那一侧**:怎么盘点 RAM、怎么搭临时页表、怎么用 memblock 凑合、怎么把空房间交给伙伴系统。一句话——**物理大楼的房间怎么切、怎么管**。

从这一章起,我们跨过中线,进入**虚拟那一侧**。这里要解决的是一个完全不同的问题:

> **怎么让每个进程都觉得"这栋楼是我一个人的"。**

注意这句话的分量。物理大楼只有一栋(物理 RAM),但此刻楼里可能住着一百个进程。如果让每个进程直接面对真实门牌号(物理地址),那 A 进程一写就把 B 进程的数据冲了——整栋楼立刻乱套。

> **不解决会怎样?** 三个灾难:
> 1. **没法隔离**:A、B 进程得自己商量"你用 1000~2000 号房间,我用 2000~3000 号",这种协商既脆弱(谁守规矩?)又僵化(谁先加载谁占好地段)。
> 2. **没法保护**:没有任何机制能阻止一个进程读写别人的房间,或者执行内核的代码。
> 3. **没法超量**:大楼就 16GB,不可能让 100 个进程每个都以为自己有 4TB。

所以内核选择了**"骗"**:给每个进程发一张**自己的门牌表(虚拟地址空间)**,从 0 开始排,一直排到天文数字。进程只认自己手里的门牌号。至于"我的 100 号门牌**实际**对应大楼里哪间房"——那是物业(内核)的**台账(页表)**才记录的真相。

> **大楼比喻(回扣导言)**:每个住户手里都有一张"我的门牌表",从 1 号、2 号……一直排到天文数字。住户只认自己的门牌号。但物业的**台账(页表)**才记录真相:"住户 A 的 100 号门牌 → 实际是物理大楼的 5037 号房间"。每个住户的"100 号",对应的是**不同的真实房间**。

这一章就是讲这本台账是怎么搭起来的,以及为什么它非得长成那个怪样子。

## 二、先回答"为什么":虚拟内存的四个理由

在钻进页表细节之前,先把"为什么要费这么大劲骗进程"讲透。一个设计能存在几十年,一定是它解决了不止一个痛点。虚拟内存至少解决了四个,每一个都足以让设计者非这么干不可。

### 理由一:隔离——让 100 个住户各有各的"100 号"

A、B 两个进程都有自己的"100 号门牌"(虚拟地址),但物理上是**完全不同的两间房**。A 把自己的 100 号写坏了,绝不会污染 B 的 100 号。每个进程都活在自己那套幻觉里,彼此彻底隔绝。

> 不这样会怎样?早期没有虚拟内存的系统(比如裸机 DOS),所有程序共用一套地址,一个程序的野指针就能把系统或别的程序写崩。虚拟内存让"我的地址"和"你的地址"在物理上根本不在同一个地方——**隔离是地址翻译机制免费送的红利**。

### 理由二:保护——台账里给每个门牌标权限

台账上的每条记录,除了"这门牌对应哪间房",还标了**权限**:只读/可写、可不可执行、用户态能不能碰。一旦某次访问越权(比如用户进程想写内核的代码段),MMU(Memory Management Unit,内存管理单元——CPU 里专门管地址翻译的那块硬件)当场拦截,触发一个异常(就是第 9 章要讲的 page fault),内核通常直接把进程杀了(SIGSEGV)。

权限位就藏在页表项里。看 x86 的真实定义([arch/x86/include/asm/pgtable_types.h:51-57](../linux-6.14/arch/x86/include/asm/pgtable_types.h#L51-L57)):

```c
/* arch/x86/include/asm/pgtable_types.h:51 */
#define _PAGE_PRESENT   ...  // 这条记录在不在(有没有配真实房间)
#define _PAGE_RW        ...  // 可读还是可写
#define _PAGE_USER      ...  // 用户态可见,还是仅内核态
#define _PAGE_DIRTY     ...  // 这页被写过(脏了)
#define _PAGE_ACCESSED  ...  // 这页最近被访问过(回收算法靠它)
#define _PAGE_PCD       ...  // 这页要不要禁用 CPU 缓存(给设备 MMIO 用)
```

> 注意 `_PAGE_PRESENT`——这是后面整本书最关键的一位。它为 0,就表示"这门牌此刻**没配真实房间**"。第 9 章的 page fault,正是由这位触发的:**住户敲门,MMU 查台账发现这条记录不存在(present=0),立刻抛异常**。这一位,是"按需兑现"哲学的硬件支点。

### 理由三:超量分配(overcommit)——开空头支票

大楼一共 16GB,但内核敢让 100 个进程每个都以为自己有 4TB——因为它赌:**这 100 个进程不可能同时把各自那 4TB 全部敲响一遍**。绝大多数门牌此刻只是台账上的一个"空指针",并没有真实房间。这就是 overcommit。

> 这条理由的物理基础,是程序的**局部性**:一个程序再大,真正同时活跃的内存就那么一点点(工作集)。虚拟内存让"承诺给进程的地址空间"和"实际占用的物理房间"解耦,从而**把有限 RAM 的利用率推到极致**。

### 理由四:按需分页——只有真敲的门才配房间

顺着第三条:既然大部分门牌是空头的,那什么时候给它配真实房间?答案是——**住户真的去敲那扇门的时候**。访问一个 present=0 的门牌,触发 page fault,内核这才临时去伙伴系统要一页,在台账里把这门牌 ↔ 这间房关联上,让进程重试。这就是"懒加载"。

> 这四个理由是**递进的**:隔离让我们有了独立地址空间,保护让权限可控,overcommit 让承诺远大于物理,按需分页让承诺只在需要时才兑现。**虚拟内存不是一个机制,而是一套互相支撑的设计**。本章先把"台账"本身讲清楚,page fault(兑现时机)留给第 9 章。

## 三、直觉方案:一张平铺的大账本——为什么不可行

既然要给每个门牌记一条"对应哪间房",最直觉的方案是:**搞一张巨大的平铺表,每个门牌号占一条**。这就叫**单层页表**。

我们用 64 位 x86 常见的 48 位虚拟地址(4 级页表模式下进程能用到的地址宽度)算一笔账:

```
  虚拟地址位数:           48 位
  页大小:                 4KB = 2^12  →  低 12 位是"页内偏移"
  需要的页表条目数:        2^48 / 2^12 = 2^36 条 ≈ 680 亿条
  每条 8 字节:            2^36 × 8 = 2^39 字节 = 512 GB
```

> **512GB!** 光是给**一个进程**记这本平铺大账,就要吃掉 512GB 物理内存——比绝大多数机器的全部 RAM 还大。而这账本要**每个进程一份**。这显然荒唐。

问题出在哪?**平铺表是"全款预付"的**:它假设进程会把 48 位地址空间**每一页都用到**,所以提前把每一页的记录都摊开。可现实是——一个进程哪怕自以为有 4TB 地址空间,真正用到的可能就几十 MB。剩下 99.9% 的条目全是空的"未映射",却照样占着 8 字节。

> **一句话**:单层页表把"承诺的空间"和"记账的成本"绑死了——承诺越大,账本越大。这是不可接受的。

## 四、解法:多级页表——只给"真用的段"建子账

多级页表的灵感,和生活里的查地址一模一样:

> 你去查"北京市朝阳区建国路 88 号 SOHO 现代城 A 座 1203 室",绝不会有人给你一张**全国每一间房**的平铺总表让你翻。真实的查法是**分层**的:先查"北京市"(顶层目录)→ "朝阳区"(二级)→ "建国路 SOHO"(三级)→ "A 座 1203"(末级)。每一级只翻很小的一页。

多级页表就是这个思路:**把那本 512GB 的大账,拆成一棵树**。

```
   顶层目录 (PGD)        ← 一页,512 个槽
        │
        ▼ 只有"用到的槽"才指向下一级
   二级目录 (P4D/PUD)    ← 一页,512 个槽
        │
        ▼
   三级目录 (PMD)        ← 一页,512 个槽
        │
        ▼
   末级页表 (PTE)        ← 一页,512 个槽
        │
        ▼
   真实物理页(4KB)
```

**关键魔法在两点:**

### 魔法一:每个目录刚好装满"一页"

设计者刻意让**每一级目录都恰好是 512 个槽**,而每个槽 8 字节,`512 × 8 = 4096 字节 = 4KB = 一页`。看真实定义([arch/x86/include/asm/pgtable_64_types.h:84-97](../linux-6.14/arch/x86/include/asm/pgtable_64_types.h#L84-L97)):

```c
/* arch/x86/include/asm/pgtable_64_types.h:84 */
#define PUD_SHIFT       30
#define PTRS_PER_PUD    512     // 一个 PUD 目录装 512 个槽
#define PMD_SHIFT       21
#define PTRS_PER_PMD    512     // 一个 PMD 目录装 512 个槽
#define PTRS_PER_PTE    512     // 一个末级页表装 512 个槽
```

> **为什么非要 512?** 因为 `512 × 8字节 = 4KB = 一页`。这样每一级目录都能**刚好塞进一页**,于是每一级都可以用伙伴系统**按页分配**——目录本身就是普通的物理页,不需要特殊待遇。这是个让"页表"和"伙伴系统"无缝衔接的精妙约束。每一位移(`SHIFT`)正好差 9 位(`log2(512)`),所以每级目录从虚拟地址里**切走 9 位**来当索引。

### 魔法二:没用到的段,上层就是个"此路不通"的空指针

这是省内存的真正来源。回到那个"自以为 4TB、实际只用几十 MB"的进程:

- 平铺表:无论用不用,**680 亿条全摊开**,512GB。
- 多级页表:顶层 PGD 的 512 个槽里,**只有少数几个**指向了下一级(因为进程只用了地址空间的零头);其余槽位直接是空的(present=0)。**这些空槽不会分配任何下一级目录页**。

> **于是记账成本和"实际使用量"成正比,而不是和"承诺的空间"成正比。** 那个 4TB 幻觉的进程,实际可能只用了几十 KB 的页表。这就是多级页表"压扁稀疏映射"的本质:**它只为真正被访问的地址段建立链路,没碰过的整片区域,在上层一个空指针就打发了。**

### 把地址切开:48 位 = 4×9 + 12

64 位 x86 在 4 级页表模式下,把 48 位虚拟地址切成五段:

```
  47────────39 │ 38────30 │ 29────21 │ 20────12 │ 11────0
     PGD 索引  │  PUD 索引 │  PMD 索引 │  PTE 索引 │ 页内偏移
     (9 位)    │  (9 位)   │  (9 位)   │  (9 位)   │ (12 位)
```

每段 9 位 → 能在 512 个槽里定位一个;末尾 12 位是页内偏移(4KB 页内哪一字节)。MMU 翻译地址时,就是**逐段下钻**:用 PGD 索引查顶层目录 → 拿到 PUD 目录页 → 用 PUD 索引查它 → …… → 末级 PTE → 拿到物理页基址 → 加上页内偏移 → 真实物理地址。

## 五、真实的层级名:Linux 的统一命名 vs x86 硬件名

这里有个容易绊倒新手的坑:**Linux 源码里的层级名,和 x86 手册里的名字不完全一样。**

x86 硬件手册(4 级页表模式)叫:PML4 → PDPT → PD → PT。而 Linux 为了**一套代码跑所有架构**(有的架构 3 级、有的 5 级),发明了统一的抽象名,从上到下永远是这五层:

| Linux 名 | x86 硬件名(4级) | 对应位移 | 这一层一个条目能映射多大 |
|----------|------------------|----------|--------------------------|
| PGD | PML4 | `PGDIR_SHIFT`=39 | 512 GB |
| P4D | *(被折叠掉)* | `P4D_SHIFT`=39 | 512 GB |
| PUD | PDPT | `PUD_SHIFT`=30 | 1 GB |
| PMD | PD | `PMD_SHIFT`=21 | 2 MB |
| PTE | PT | `PAGE_SHIFT`=12 | 4 KB |

> **为什么 Linux 硬要弄五层?** 因为不同架构的页表级数不一样(早些年 x86 是 4 级,后来出了 5 级 LA57;ARM、RISC-V 又是别的级数)。Linux 干脆**永远写五层**(pgd→p4d→pud→pmd→pte),对级数不够的架构,把多余的层"折叠"掉(那一层的 `_offset` 直接返回上一层指针)。这样核心代码(如 `mm/memory.c`)不用为每种架构改写。**这是"用一层抽象换可移植性"的典型设计**——第 3 章里那个"分层抽象"哲学又一次现身。

注意 `P4D_SHIFT` 和 `PGDIR_SHIFT` 在 4 级模式下都是 39——因为 P4D 层在 4 级时被折叠了,它只是个摆设,实际不占位移。等机器开了 5 级页表(LA57,`PGDIR_SHIFT` 变 48),P4D 才真正干活(见 [arch/x86/include/asm/pgtable_64_types.h:51-79](../linux-6.14/arch/x86/include/asm/pgtable_64_types.h#L51-L79)):

```c
/* arch/x86/include/asm/pgtable_64_types.h:51 */
#ifdef CONFIG_X86_5LEVEL
#define PGDIR_SHIFT     pgdir_shift       // 5级:48
#define P4D_SHIFT       39
#else
#define PGDIR_SHIFT     39                // 4级:39,P4D 层被折叠
#endif
```

> **PMD 条目映射 2MB、PUD 条目映射 1GB** —— 这两个尺寸正是第 13 章要讲的"大页/透明大页"的由来:与其让一个 2MB 区域占满一整页 PTE(512 条),不如直接在 PMD 层记一条"这 2MB 都映射到这一大块连续物理内存"。大页的本质,就是**让翻译停在更上层、少下钻几级**,从而减轻 TLB 压力。这是后话。

## 六、页表项里到底装了什么:物理页帧号 + 一堆权限位

一条末级 PTE(页表项)在 x86_64 上是 64 位,内容分两部分:

```
  ┌────────────────────────────┬──────────────────────────┐
  │  高位:物理页帧号(PFN)     │  低位:权限/状态标志位    │
  │  (指向真实房间的"楼+层")   │  (PRESENT/RW/USER/DIRTY…)│
  └────────────────────────────┴──────────────────────────┘
```

物理地址必须是页对齐的(低 12 位恒为 0),所以低位这 12 位"白白浪费"——干脆拿来存标志位,一石二鸟。这就是为什么 `_PAGE_PRESENT`、`_PAGE_RW` 这些位都定义在低位(见 [arch/x86/include/asm/pgtable_types.h:51-57](../linux-6.14/arch/x86/include/asm/pgtable_types.h#L51-L57))。

写一条 PTE,本质上就是"把物理页帧号和权限位或在一起,塞进 64 位":

```c
/* arch/x86/include/asm/pgtable_64.h:65 —— 写一条页表项 */
static inline void native_set_pte(pte_t *ptep, pte_t pte)
{
    WRITE_ONCE(*ptep, pte);
}
```

就这么朴实无华——`WRITE_ONCE` 保证这次 8 字节写入对其他 CPU 是原子可见的(64 位对齐的 8 字节写在 x86 上本身就是原子的)。

> **底层页表项的类型为什么是 `typedef struct { pteval_t pte; }` 而不是直接 `unsigned long`?** 见 [arch/x86/include/asm/pgtable_64_types.h:21-22](../linux-6.14/arch/x86/include/asm/pgtable_64_types.h#L21-L22)。用单成员结构体包一层,是为了**类型安全**——`pte_t`、`pmd_t`、`pud_t` 内部都是整数,但编译器不让它们互相赋值,免得你手滑把一个 PMD 项当 PTE 用。这是个用类型系统防低级错误的细节,内核里到处都是这种"靠编译器把人盯住"的套路。

## 七、TLB:页表的速查缓存——为什么它是页表结构的孪生兄弟

到这里,地址翻译的成本显而易见:每次访问内存,MMU 都要**下钻 4 级目录**,也就是**额外做 4 次内存读取**才能拿到真实物理地址。访存本来一次,现在变五次——CPU 还怎么跑得动?

救场的是 **TLB(Translation Lookaside Buffer,页表缓存)**——CPU 芯片内部的一小块超高速缓存,专门缓存"最近翻译过的虚拟页 → 物理页"映射。

> **大楼比喻**:MMU 每次都要翻 4 本台账才能查到房间号,太慢。于是它在桌角贴了一张便签(TLB),最近查过的几条直接记在便签上,再查同样的门牌,一秒就出结果。

**TLB 改变了整个性能方程**:

- 没有 TLB:每次访存 = 4 次额外访存(读 4 级页表),CPU 慢到不可用。
- 有 TLB:绝大多数访存命中 TLB,**0 次额外访存**,几乎零开销。

> **于是有了一条贯穿全书的铁律:页表结构的几乎所有设计,都在围着 TLB 转。** 举几个后续会反复遇到的例子:
> - **为什么有大页?** 一个 2MB 大页只要一条 TLB 项就覆盖了;若用 4KB 小页,同样 2MB 要占 512 条 TLB 项。TLB 容量极有限(几百到几千条),大页能让 TLB 覆盖的地址范围大几个数量级 → 缺页更少 → 更快。这就是第 13 章 THP 的全部动机。
> - **为什么换页表要刷 TLB?** 进程切换时,新进程的门牌体系完全不同,旧 TLB 全部作废,得重查。刷 TLB 是进程切换开销的重要来源,内核有一堆优化(如 PCID,给每个进程的 TLB 项打标签,切换时不必全刷)。
> - **为什么改了页表项要主动 flush TLB?** 因为 TLB 是缓存,它可能还存着旧翻译。内核改了 PTE(比如把只读改成可写),必须通知 CPU"这条作废,重查",否则硬件还会用旧权限。看真实的刷 TLB 接口:

```c
/* arch/x86/include/asm/tlbflush.h:241 —— 改了一个 VMA 范围内的页表后调用 */
#define flush_tlb_range(vma, start, end)            \
    flush_tlb_mm_range((vma)->vm_mm, start, end, ...)
```

> **TLB 和多级页表是一对孪生兄弟,缺一不可**:多级页表解决了"账本太大",却带来"翻译太慢";TLB 解决了"翻译太慢",又依赖多级页表的局部性(访问聚集在少数页 → TLB 命中率高)。两者一起,才让虚拟内存**又省又快**。

## 八、把"造幻觉"的代价收一收:空间账

讲完结构,给多级页表算一笔**真实的空间账**,体会它有多省。假设一个进程的地址空间里真正映射了连续的 8MB(比如它的代码段+数据段):

```
平铺表(假设性方案):
  8MB / 4KB = 2048 条 PTE,看似不多
  但平铺表是"全空间预付"的 → 还是 512GB,根本用不了

4 级多级页表(真实方案):
  PGD  顶层目录 ........... 1 页(进程一出生就有,4KB)
  PUD  第二层(只 1 个槽用上)→ 1 页(4KB)
  PMD  第三层(8MB / 2MB = 4 个槽)→ 1 页(4KB)
  PTE  末级(8MB / 4KB = 2048 条 = 4 页)→ 4 页(16KB)
  ────────────────────────────
  合计约 7 页 ≈ 28KB,管理 8MB 映射
```

**用 28KB 的台账,管理 8MB 的映射;而那个进程自己以为拥有的 TB 级空间,没碰到的部分一行账都不记。** 这就是多级页表的胜利。如果这 8MB 还用了 2MB 大页(4 个 PMD 条目直接映射),连末级 PTE 那 4 页都省了——大页省的不只是 TLB,还有页表内存。

## 九、关键源码精读:一次完整的页表遍历

理论讲完,看 Linux 软件层面怎么走这棵页表树。最好的教科书级例子,就在 `print_bad_pte` 里——它遇到坏页表项时要打印诊断信息,得自己把整棵树走一遍。这段代码在 [mm/memory.c:498-501](../linux-6.14/mm/memory.c#L498-L501) 清清楚楚展示了四级下钻:

```c
/* mm/memory.c:498 —— 软件层面的页表遍历(精简) */
pgd_t *pgd = pgd_offset(vma->vm_mm, addr);   // ① 顶层:从 mm->pgd 出发
p4d_t *p4d = p4d_offset(pgd, addr);          // ② 第二层
pud_t *pud = pud_offset(p4d, addr);          // ③ 第三层
pmd_t *pmd = pmd_offset(pud, addr);          // ④ 第四层
// 后续再 pte_offset_map(pmd, addr) 拿到末级 PTE
```

每一行的逻辑都一样:**用上一层的指针 + 虚拟地址的某一段,算出下一层目录项的地址**。以第一层为例:

```c
/* include/linux/pgtable.h:168 —— 第一层入口 */
#define pgd_offset(mm, address)   pgd_offset_pgd((mm)->pgd, (address))

/* include/linux/pgtable.h:159 */
static inline pgd_t *pgd_offset_pgd(pgd_t *pgd, unsigned long address)
{
    return (pgd + pgd_index(address));        // 基址 + 索引
}

/* include/linux/pgtable.h:90 —— 索引怎么算 */
#define pgd_index(a)  (((a) >> PGDIR_SHIFT) & (PTRS_PER_PGD - 1))
```

`pgd_index` 的逻辑正是第四节那张"把地址切开"的图:把虚拟地址右移 `PGDIR_SHIFT` 位(丢掉低位),再 `& 511`(取低 9 位)——这就是**从地址里抠出 PGD 索引**。末级 PTE 的索引同理:

```c
/* include/linux/pgtable.h:67 —— 末级索引 */
static inline unsigned long pte_index(unsigned long address)
{
    return (address >> PAGE_SHIFT) & (PTRS_PER_PTE - 1);
}
```

中间层的 `pmd_offset`、`pud_offset` 模式完全一致(见 [include/linux/pgtable.h:144-156](../linux-6.14/include/linux/pgtable.h#L144-L156))——返回"上一层项里指向的下一级页 + 本层索引"。

> **一个关键区别要记住**:软件遍历(像上面这段)和**硬件 MMU 遍历**走的是同一棵树,但目的不同:
> - **硬件 MMU 遍历**:CPU 访问内存时,MMU 自动、透明地下钻四级目录,把虚拟地址翻成物理地址。**进程对此毫无感知**。这是硬件写死在硅片里的逻辑。
> - **软件遍历**:内核自己(比如处理 page fault、复制页表、诊断坏页表)需要"看看某个虚拟地址现在映射到哪",于是用 `pgd_offset → ... → pte_offset` 这套宏手动走一遍。
>
> **为什么要软件也能走?** 因为内核经常需要**读/改**页表(建映射、改权限、回收页),而 MMU 只负责"翻译",不会帮你改页表。所以这套 `*_offset` 宏,是内核操作台账的"读写接口"。

还有一个起点值得点出:`pgd_offset` 的第一个参数是 `mm->pgd`。每个进程的"顶层目录"地址,就存在它的进程描述符里(见 [include/linux/mm_types.h:832](../linux-6.14/include/linux/mm_types.h#L832)):

```c
/* include/linux/mm_types.h:832 —— 每个进程的页表树根 */
pgd_t * pgd;
```

> 进程切换时,CPU 把 `cr3` 寄存器指向新进程的 `mm->pgd`(物理地址),从此 MMU 就开始用新进程那套台账翻译地址——**这就是"换一套幻觉"的硬件动作**,快得几乎只是一次寄存器写入(代价主要在随之而来的 TLB 失效)。

## 十、章末小结

### 用大楼比喻回顾

物理大楼只有一栋,但内核要给一百个住户每人编一个"整栋楼都是我的"的幻觉。它靠的是一本**分层的台账**:

- **为什么不平铺?** 一张覆盖 48 位空间的平铺大表要 512GB——比整机 RAM 还大,因为平铺是"按承诺预付"的。
- **所以分层。** 把账本拆成一棵树(PGD→P4D→PUD→PMD→PTE),每级目录刚好一页(512 槽),**只给真正用到的地址段建子目录**。没用到的整片区域,上层一个"空指针"就打发了。账本大小和"实际用量"成正比,28KB 就能管理 8MB 映射。
- **页表项里**塞了物理页帧号(指向真实房间)+ 一堆权限位(PRESENT/RW/USER/DIRTY…)。`_PAGE_PRESENT=0` 意味着"这门牌此刻没配房间"——这是第 9 章 page fault 的触发点。
- **TLB** 是这本台账的速查缓存。多级页表让翻译变慢(4 次下钻),TLB 让它重新变快;两者互相救赎,缺一不可。**页表后续的一切设计(大页、PCID、flush 时机)都在伺候 TLB。**

### 回扣全书主线

这本书的二分法:**物理一侧管"切房间",虚拟一侧管"造幻觉"。** 这一章是**虚拟那一侧的地基**——它讲清了"幻觉"的物质载体(页表)长什么样、为什么这么长。

但要注意:**本章只造了"台账的格式",还没有任何"兑现"动作。** 到现在为止,门牌表只是张空壳:大部分门牌 present=0,没有任何真实房间。一个门牌从"空头支票"变成"真金白银",要等住户真的去敲那扇门——那就是下一章的主角:**page fault**。

所以你在主线上的位置是:

```
   物理侧:房间怎么切、怎么管      ← 第 1~7 章(已讲完)
   ─────────────────────────────
   虚拟侧:幻觉怎么造              ← 第 8 章(本章:页表 = 台账格式)
   虚拟侧:幻觉怎么兑现            ← 第 9 章(page fault:敲门才配房间)
   虚拟侧:幻觉的范围怎么记账      ← 第 10 章(VMA:门牌预定单)
```
