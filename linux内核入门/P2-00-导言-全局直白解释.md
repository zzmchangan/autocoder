# 导言 · 一篇看懂 Linux 内存管理全貌

> 这篇是"地图"。读各章之前先读它,你会对整个内存子系统建立一个结实的直觉。
> 全文只讲两件事:**它在干什么**,以及**为什么非得这么干**。
> 我会用一个贯穿始终的比喻——**一栋大楼的物业管理**——帮你记住每一步。

---

## 一、一句话说清内存管理在干嘛

> **内核就是 RAM 这栋大楼的物业管理员。它的全部工作,是让所有住户(进程、设备、内核自己)都觉得"这栋楼是我一个人的",而又不互相打架,还尽量不浪费一间空房。**

就这么一句话。后面所有的复杂设计,都是在实现这个目标。你任何时候看不懂某个机制,就回来问自己一个问题:

> **"这个机制,是在'切分物理大楼的房间',还是在'制造住户独占整栋楼的幻觉'?"**

整个内存子系统就这两大块。记住这个二分法,你就永远不会迷路。

- **物理那一侧**:管真实的房间怎么数、怎么分区、怎么分配、怎么回收。
- **虚拟那一侧**:管怎么让每个住户产生"整栋楼都是我的"的幻觉,以及怎么把这个幻觉兑现成真实房间。

---

## 二、建立比喻:RAM 是一栋大楼

我们把物理内存想象成一栋超大的公寓楼:

| 比喻 | 对应的内核概念 |
|------|----------------|
| 整栋大楼 | 物理 RAM |
| 一间房间 | 一个**物理页(page,通常 4KB)** |
| 房间的门牌号(物理) | 物理地址 |
| 物业管理员 | 内核 |
| 住户 | 进程 |
| 住户心里"我家在这栋楼的哪个门牌" | **虚拟地址** |
| 物业台账(住户门牌 ↔ 真实房间) | **页表** |
| 住户敲门发现房间没家具,临时搬家具进去 | **page fault** |
| 楼里满员了,把不常在家的住户东西搬进地下室 | **swap / 回收** |
| 有装卸货需求的特殊房间(一楼、靠近货梯) | **ZONE_DMA** |
| 物业把房间"2 间一组、4 间一组…"打包管理 | **伙伴系统** |
| 预先摆好同款家具的样板间 | **slab** |
| 大楼刚盖好、正式物业入驻前的临时工棚 | **memblock** |

这张表先扫一眼留个印象,后面每一步我都会回来对应。

## 三、跟着时间线走一遍:大楼是怎么开张的

我们从大楼**还没通电**开始,一步步看物业管理是怎么建立起来的。

### 第 0 步:白纸阶段——大楼还黑着

机器刚通电,CPU 处于"实模式",脑子还很笨,只认识最低 1MB 的地址。

这时候真正在管事的是 **BIOS/UEFI**(相当于"建楼施工方")。它干两件事:

1. **盘点大楼**:挨根内存条去数,"哦,这台机器有 16GB,但这些地址段被显存占了、那些被设备占了"。盘点的结果写成一张**内存使用图**(传统 BIOS 叫 **e820**,UEFI 有自己的格式)。
2. **把内核请进来**:BIOS 把内核镜像 `bzImage` 装进大楼的一个角落,然后跳过去执行。

> **关键认知**:此刻谁都没有"拥有"内存。存在的只有一份**"哪些房间能用、哪些被占了"的原始清单**。这就是内核醒来时拿到的东西。

⚠️ **一个反直觉的事实**:内存**不是**"从 0 开始连续 16GB"。中间到处是窟窿——显存、BIOS、设备的 MMIO 寄存器都占走了某些地址段。所以内核拿到的是一张**满是洞的地图**,不是干净的一条。这个事实解释了后面很多设计。

### 第 1 步:内核自举——物业先给自己搭个脚手架

内核一醒来,自己也挤在大楼一个角落里,连个像样的分配器都没有。它的第一件事不是管别人,而是**先把自己撑起来**:

- 用汇编 + 一点点 C([arch/x86/kernel/head_64.S](../linux-6.14/arch/x86/kernel/head_64.S)),给自己搭一张**临时页表**,让自己从"笨模式"切到"聪明模式(长模式)"。
- 这张临时页表只服务一件事:让内核自己的代码能正确运行。

> **比喻**:大楼刚通电,物业还没办公室。他们先在走廊里搭个临时工棚,先把"物业办公室"这个房间装修好、通电、能办公。装修期间用的都是临时材料。

这就是"自举(bootstrap)"——拎着自己的鞋带把自己提起来。

### 第 2 步:memblock——临时分配员上场

问题来了:**正式的分配器(伙伴系统)还没建好,但内核在启动过程中就要分配内存**(给页表、给 per-cpu 数据、给设备树建结构……)。这是个**鸡生蛋问题**:

> 伙伴系统需要内存才能初始化自己;但内存分配又只有伙伴系统这一个出口。怎么办?

答案是先上**临时工**:[memblock](../linux-6.14/mm/memblock.c)。

memblock 的设计哲学就四个字:**极简凑合**。

- 它不管单间,只记"哪些楼层是空着的、哪些楼层被划走了"。
- 要分配?在空楼层里划一段,记一笔。
- 它的存在,纯粹是为了**熬过启动期**。

等正式的伙伴系统盖好了,memblock 把手里**剩下的空房间全部移交**给伙伴系统,自己功成身退(工棚拆除)。

> **为什么这么设计?** 因为启动期需求又急又杂,正儿八经写个通用分配器太重、还可能有 bug。memblock 用几百行代码就解决了"先凑合用"的问题,把复杂性留给真正该管事的伙伴系统。这是**"用临时的、丑的、但够用的方案,换取启动可靠"**的典型工程取舍。

### 第 3 步:ZONE——给房间分区

移交之前,内核做了一个重要的**布局决策**:不把所有房间一视同仁,而是**按物理位置分区**。这就是 **ZONE**(定义在 [include/linux/mmzone.h](../linux-6.14/include/linux/mmzone.h)):

| ZONE | 比喻 | 为什么单独分区 |
|------|------|----------------|
| ZONE_DMA | 一楼靠货梯的房间 | 老 ISA 设备做 DMA,**物理上只能寻址到 16MB 以下**,只能用这些房间 |
| ZONE_DMA32 | 1~3 楼 | 32 位 DMA 设备只能到 4GB 以下 |
| ZONE_NORMAL | 4 楼以上 | 普通内存,绝大部分都在这(64 位) |

> **为什么不直接一个池子?** 因为分配时内核要能表达**"给我一块设备能搬货进来的内存"**这种约束。ZONE 就是这种硬件约束的化身。
>
> 注意**降级回退**规则:要 NORMAL 的请求,如果 NORMAL 没了,可以**借** DMA 区的房间(因为 DMA 区的房间当普通房间用没问题);但反过来不行。所以 ZONE 之间不是平等竞争,而是有优先级。

ZONE 内部还会按"**这些房间能不能搬动**"再分组(migrate type):能搬的住户聚一起、不能搬的聚一起。这是为了将来要腾出**一整层连续房间**(大页)时方便——这是后话。

### 第 4 步:伙伴系统——正式的房间分配管家

[mm/page_alloc.c](../linux-6.14/mm/page_alloc.c)。memblock 移交完毕,伙伴系统正式接管全部空房间。

这是物理那一侧的**核心**。它管的最小单位是**一间房间(一页 4KB)**。

**它怎么管?** 把房间按"2 的幂"打包:

```
1 间   2 间   4 间   8 间 ... 1024 间
(4KB) (8KB) (16KB)(32KB)    (4MB)
```

共 11 档,最大一档 1024 间 = 4MB(这就是伙伴系统一次能给出的"最大连续块")。

**分配**:
- 要 1 间?从"1 间"那档拿一个。
- "1 间"档空了?把"2 间"档的一个对半劈,用一间,剩一间放回"1 间"档。
- 一路劈上去,总能凑出来。

**释放**(精髓在这):释放一间房时,**先看它的"伙伴"(对偶那间)是不是也空着**——空着就**合并**成 2 间档的一块;再看这 2 间的"伙伴块"……一路向上合并。

> **为什么叫"伙伴"?** 因为每块房间都有且只有一个"对偶块",它们合起来正好凑成上一档的一整块。靠这个对偶关系,合并永远是干净利落的。
>
> **这个设计解决了什么?** 外部碎片。频繁分配释放后,内存会变成"到处都是零散空房间、但凑不出一整块大的"那种烂摊子(就像停车场全是零散空位但停不下一辆大卡车)。伙伴系统靠"对半劈 + 对偶合并",自动把零散房间重新聚成大块,**O(1) 完成,几乎不花额外记账成本**。

⚠️ **伙伴系统的天花板**:最小粒度是**一页(4KB)**。所以它适合"我要几页连续内存"。但内核里大量需要的是"几十字节"的小东西——这引出下一步。

### 第 5 步:slab——小物件的样板间

[mm/slub.c](../linux-6.14/mm/slub.c)。内核天天要分配几十字节的小对象:`task_struct`(进程描述符)、`inode`、`dentry`、`file`……

如果每个都找伙伴系统要一页,那就是**用 4KB 的房间装一支铅笔**,浪费到离谱。

slab 的思想是**"面向对象的样板间池化"**:

- 给每一种高频小物件建一个专属的"样板间工厂"(叫 `kmem_cache`,比如 `task_struct` 专用的那个)。
- 这个工厂从伙伴系统拿几页房间,然后把房间**切成一个个固定大小的物件槽位**,每个槽位摆好同款"家具"(预初始化)。
- 分配 = 从样板间拿一个现成的;释放 = 放回样板间,**家具不拆**,下次直接复用。

两个让它特别快的优化:
- **每个 CPU 一个快通道(per-cpu slab)**:绝大多数分配/释放都在自己 CPU 的快通道完成,**几乎无锁**。
- **构造函数只跑一次**:物件第一次创建时初始化好,之后反复复用,省掉反复初始化的开销。

> **一句话**:伙伴系统发"砖(页)",slab 把砖砌成"预制件(对象)"。

### 第 6 步:虚拟内存——让住户产生"独占整栋楼"的幻觉

到这一步,物理那一侧基本齐活了。但**进程根本不碰物理地址**——它看到的是一个**"从 0 开始、连续、独占"的虚拟地址空间**。这就是虚拟内存([mm/memory.c](../linux-6.14/mm/memory.c))。

> **比喻**:每个住户手里都有一张"我的门牌表",从 1 号、2 号、3 号……一直排到天文数字。住户只认自己的门牌号。但物业的**台账(页表)**才记录真相:"住户 A 的 100 号门牌 → 实际是物理大楼的 5037 号房间"。每个住户的 100 号,对应的是**不同的真实房间**。

为什么要费这么大劲"骗"进程?四个理由,每一个都足以让设计者这么干:

1. **隔离**:A、B 两个进程都有"100 号门牌",但物理上是不同房间。A 写坏了也不会污染 B。每个进程都觉得自己独占整栋楼。
2. **保护**:台账上每个门牌都标了权限(只读/可写/可执行/用户态能否访问)。越权访问,物业当场拦下(触发 page fault,通常发 SIGSEGV 杀掉)。
3. **超量分配(overcommit)**:大楼一共 16GB,但 100 个进程每个都以为有 4TB 是自己的——没问题,因为大多数门牌**此刻没对应真实房间**,只是个空头支票。
4. **按需分页**:只有真正被访问的门牌,物业才临时去给它配真实房间(下一个机制)。

**页表为什么是多级?** 算笔账就懂:如果给 48 位地址空间建一张完整大表,光是这本台账就要 **512GB**(比物理内存还大)。多级页表的妙处:**只有实际用到的门牌段,才建立对应的下层账页;没用的,上层就一个"此路不通"的空指针,一行不记。** 稀疏映射天然省账本。

**TLB** 是这本台账的"速查缓存"——CPU 每次访问门牌都要查台账,太慢,于是把最近查过的几条记在芯片里的 TLB 里。页表结构的设计,很大程度上是在**讨好 TLB**。

### 第 7 步:page fault——空头支票的"兑现时刻"

虚拟门牌是"空头支票",那它什么时候变成真金白银?答案是:**住户真的去敲那扇门的时候**。

进程访问一个**还没配真实房间**的门牌号,CPU 触发一个异常(叫 **page fault**),陷进内核([arch/x86/mm/fault.c](../linux-6.14/arch/x86/mm/fault.c) → [mm/memory.c](../linux-6.14/mm/memory.c) 的 `handle_mm_fault`)。内核分情况处理:

- **合法但还没配房间**(比如刚 malloc 还没用过):从伙伴系统要一页,在台账上把这门牌 ↔ 这间房关联上,返回住户重试——这次成功了。
- **写一个"只读"门牌**:如果是写时复制(COW)场景,就**复制一份**再给写权限;否则就是真违法 → 杀进程。
- **完全非法的门牌**:直接 SIGSEGV。

> **视角转变(全书最重要的一句话之一)**:
>
> **page fault 不是"错误",而是"兑现时机"。** 它是虚拟内存"懒加载"哲学的实现机制。理解了这一点,你就理解了为什么 `malloc` 之后内存没真涨、为什么 fork 之后父子进程改之前共享同一份内存、为什么程序可以比物理内存大得多——全靠"先开空头支票,要钱时再兑现"。

### 第 8 步:VMA——住户的"地址账本"

住户的虚拟门牌不是随便排的,而是分块的:代码段、数据段、堆、共享库、栈……每一段是一块连续门牌。内核用 **VMA(`struct vm_area_struct`)** 来描述"这段门牌是什么性质、什么权限、背后是匿名内存还是文件"。

- `malloc` / `mmap` 这种调用,**不立即配真实房间**,只是**在账本上圈一块门牌号范围**(创建一个 VMA)。真实房间要等 page fault。
- 想看一个进程的账本?`cat /proc/<pid>/maps`,每一行就是一个 VMA。

> **比喻**:VMA 是住户的"预定单"——"我预定 100~200 号门牌归我用"。物业记下预定,但**不搬家具**。等住户真去用了(page fault),物业才往里搬。

### 第 9 步:page cache——文件也被搬进大楼住

进程读写文件时,数据其实是在**大楼里的房间**中转的。这就是 **page cache**([mm/filemap.c](../linux-6.14/mm/filemap.c)):

- 读文件 → 先看大楼里有没有这页的缓存,有就直接给(所以**第二次读同一个文件特别快**);没有就从磁盘读进大楼,再给进程。
- 写文件 → 先写到大楼里的房间(变成**脏页**),由内核择机写回磁盘。

> **为什么这么设计?** 磁盘比内存慢几十万倍。Linux 的哲学是:**空闲的房间不如拿来当磁盘缓存**——反正闲着也是闲着。所以你常看到"机器 16GB,free 只剩 200MB"但完全正常,因为大部分内存都在当 page cache,需要时会自动让出来。

### 第 10 步:回收、swap、OOM——大楼满员了怎么办

物理房间终究有限。分配压力大时,内核要**腾房间**([mm/vmscan.c](../linux-6.14/mm/vmscan.c)),策略分三类住户:

1. **干净的文件缓存住户**(内容磁盘上有原件):直接赶走,房间清空。下次要用,重新从磁盘读。
2. **脏住户**(改过还没写回磁盘):先写回磁盘原件,再赶走。
3. **匿名住户**(进程的堆/栈,磁盘上没原件):把他们的东西搬到**地下室(swap 区)**,台账上记一笔"这门牌的东西在地下室"。要用时再搬回来。

后台有个守护进程 **kswapd**,在水位线告急前就开始悄悄打扫,让回收和分配并行。实在一点都腾不出来了,**OOM killer** 登场——挑一个"占房最多又不重要"的住户,直接杀掉腾房。

> **为什么能"超量分配"?** 就是因为内核赌:**100 个住户不可能同时都来敲自己那堆空头支票的门**。赌赢了,16GB 的大楼能服务出几百 GB 的"已分配"额度。

## 四、收束:三层抽象 + 两句话

走完全程,你看到的其实是**三层抽象叠在一起**:

```
     进程看到的:   虚拟地址空间(独占幻觉)
                       ↑ 页表 + page fault 兑现
     内核分配的:   物理页(伙伴系统 / slab)
                       ↑
     真实存在的:   RAM(满是洞的物理地址)
```

每一层都**只解决一类问题**,层与层之间靠**明确的接口**解耦。这就是 Linux 内存管理能既复杂又不出乱子的根本原因。

### 记住这三句话,就抓住了全书

1. **物理一侧管"切房间",虚拟一侧管"造幻觉"。** 任何时候迷路,回到这个二分法。
2. **"懒"是核心哲学。** VMA 只圈地不占地,malloc 只开支票不给钱,page fault 才兑现——用延迟换利用率。
3. **所有复杂机制都是被四个本性逼出来的**:RAM 唯一且平铺、有限、慢、有物理拓扑。当你觉得某个设计"为什么搞这么复杂"时,去找是哪个本性在作怪。
