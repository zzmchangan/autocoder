# 核心写作原则(最重要,反复对照)

1. **"为什么"永远先于"怎么做"。** 这是本书最高原则,每个设计点都要先回答:
   - **不这样会怎样?**(痛点是什么、会出什么问题——把灾难场景具体写出来)
   - **所以这样设计。**(它如何化解痛点)
   - 然后才讲实现。读者理解了"为什么",自然知道该去代码里看什么。

2. **大白话 + 比喻。** 全书统一使用"**RAM = 一栋大楼,内核 = 物业管理员**"的比喻体系(见导言的术语对照表)。新概念先给一个生活化比喻,再落到技术。可用叠加子比喻,但不要与主比喻冲突。

3. **小白友好。** 不假设读者懂内核术语。任何术语首次出现都要一句话解释(如"TLB——页表的速查缓存")。避免堆砌黑话。

4. **关键路径配真实源码。** 在设计的关键处,贴 `linux-6.14/` 里的真实源码片段,标注 `文件:行号`,逐段讲解。代码密度:概念为主,关键处配码。

5. **回扣全局。** 每章结尾回到"物理侧管切房间 / 虚拟侧管造幻觉"这个二分法,说明本章服务哪一侧,并预告/衔接前后章节。

# 每章的标准结构(严格照此组织,顺序不可变)

1. **章首**:
   - 先一行**前置导航**:`> **前置**:你需要先读过第 X 章《……》(理由一句话)。` ——把全书旅程串起来,让读者知道"我是顺着哪条线走到这儿的"。
   - 再一句话点出**本章核心问题** + "读完本章你会明白……"。
2. **正文·若干小节**:每节遵循 **"提出问题 → 不这样会怎样 → 所以这样设计 → 代码佐证"** 四段式,可用小标题分割。
3. **关键源码精读**(固定位置:**正文之后、小结之前**):挑本章最核心的 1~2 个函数,贴片段、逐行/逐段讲清逻辑如何对应前面讲的设计。
4. **章末小结**:用大楼比喻回顾本章;回扣全书主线;给"想继续深入该往哪个函数/文件钻"的指引。

> 难度较高、信息密度大的章节,在"读完本章你会明白"之后,额外加一个**逃生阀**小段:
> > **如果一读觉得太难**:先只记住三件事——①…;②…;③…。其余细节第二遍配合源码再抠,不影响理解后续章节。

# 硬性要求

- **语言**:中文,技术名词保留英文原词(page fault、TLB、slab 等)。
- **篇幅**:不限,但要能把本章主题讲清楚。
- **术语一致性**:与导言、目录、已写章节用词保持一致(页/页帧=page,房间=物理页,台账=页表,兑现=page fault 等)。
- **源码准确性(最易翻车,务必严格遵守,见下方"写前必做")**。
- **源码链接格式**: `[描述](../linux-6.14/路径#L行号)`;**区间一律用 `#L起始-L结束`**(如 `#L202-L245`),不要只标起点。
- **输出**:把完整章节正文写入文件 `docs/P{n}-0X-标题.md`(n 为**篇号**如 P0/P1/P2/P3…,X 为两位章号,标题与总纲一致)。例如:全书第 0 篇第 1 章 = `P0-01-…`,内存篇(第 2 篇)第 5 章 = `P2-05-…`。**全书所有篇一律用此前缀,内存篇也不例外。** 写完用一句话告诉我文件路径和大致字数。

# 写前必做:源码核查(防止编造,这一步不能省)

源码引用**严禁凭记忆**,必须先用 `grep`/读文件确认:

1. **每个要引用的函数/结构体,先 grep 出真实行号再写**。例如要引 `struct zone`,先 `grep -n "^struct zone {" include/linux/mmzone.h`,拿到真实行号再用。
2. **每个要贴的代码块,必须是从真实文件里摘录的原文**(允许为省略无关部分用 `...`,但保留的部分逐字一致)。**绝对不要把"意译/改写/脑补"的代码当成真实源码贴出来。**
   - 如果某段逻辑你想简化展示,要么直接引真实代码,要么**明确标注**"(以下为简化示意,非源码原文)",并写清楚与真实代码的差异。
3. 引用的文件路径必须真实存在;不确定就先 `ls`/`grep` 确认。
4. **不确定精确行号时,宁可只标文件不标行号(`[文件](../linux-6.14/路径)`),也不要猜一个数字。**
