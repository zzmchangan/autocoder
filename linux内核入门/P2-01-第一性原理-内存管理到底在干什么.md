# 第 1 章 · 第一性原理:内存管理到底在干什么

> **核心问题**:为什么要专门搞一个"内存管理"子系统?它到底解决了什么、不解决就不行的问题?
>
> 这一章我们不碰任何机制、不背任何 API。我们只问一件事:**RAM 这东西,天生就带着哪些"麻烦"?这些麻烦逼着谁必须站出来管它?**
>
> **读完本章你会明白**:为什么后面十二章里那些层层叠叠、看似"过度工程"的设计,没有一个是工程师拍脑袋想出来的——它们全是被内存的几个本性**逼**出来的。理解了这一章,你在后面任何一章遇到"为什么搞这么复杂",都能自己找到答案。

---

## 章首·一句话点破

如果用一句话回答"内存管理在干嘛",那就是导言里那句:

> **内核是 RAM 这栋大楼的物业管理员,它让所有住户都觉得"这楼是我的",而又不互相打架,还尽量不浪费一间空房。**

但这句话是**结论**,不是**理由**。这一章我们要倒过来——先把结论收起来,从最底层问起:

- RAM 到底是个什么东西?它好管吗?
- 如果**没人管**,直接让所有程序自己往 RAM 里写,会怎样?
- "管"这件事,为什么非得是一个**复杂的、分层的子系统**,而不能是"一人一本账本"那么简单?

我们一块一块拆。

---

## 一、RAM 的四个"麻烦本性"

要理解为什么需要内存管理,得先看清我们要管的这个对象——物理 RAM——**天生带着哪些毛病**。我把它总结成四个。这四个本性,是后面一切设计的总根源。

### 本性 1:RAM 唯一,且平铺

物理 RAM 在硬件层面就是一根(或几根)内存条,**地址从 0 开始一根线排下去**,所有 CPU、所有设备、内核自己,看到的是**同一根地址线**。

> **比喻**:整栋大楼只有**一栋**,所有住户、所有送货的、连物业自己,要找房间都得在同一栋楼里。没有"另一个平行的大楼"可以各住各的。

**这个"唯一性"是麻烦的起点**:正因为只有一栋楼,而楼里要住**很多很多人**,所以"谁住哪、谁不能碰谁的房间"这件事,**不会自己解决,必须有人来管**。如果楼里只住一个人,那确实不用物业;可一旦住一百个人,又共用同一栋楼,没有物业就一定打架。

**"平铺"则是另一个隐患**:地址是连续的、从 0 编号的,这意味着"地址 100"和"地址 101"在物理上是隔壁两间房。这听起来天经地义,但它给后面埋了个雷——我们后面会看到,**进程希望"我的地址从 0 开始连续排",可物理房间早被各种人占得七零八落**,怎么让两边对上号?这个错位,是虚拟内存要解决的核心矛盾。先记住这个伏笔。

### 本性 2:RAM 有限

这一点最直白,却最致命:**RAM 的容量是有限的,而且远小于大家想要的总量。**

一台 16GB 的机器,同时跑着一两百个进程,光浏览器一个进程就敢张口要几个 GB。如果把每个进程"想要的内存"加起来,轻松超过物理 RAM 一个数量级。

> **比喻**:大楼一共就 16000 间房,但 100 个住户每个都说"我要 4000 间"。加起来 40 万间,**物理上根本不存在**。

**不这样会怎样?** 如果要求"申请多少就必须立刻占多少、占不到就失败",那这台机器连一半进程都跑不起来,大量 RAM 在大部分时间闲着——因为一个进程申请了内存,真正"用起来"的往往只是其中一小部分(你可能 `malloc` 了一个大数组,但只写了前几行)。

所以"有限"这个本性,逼出了一个看起来反常、实则必需的决策:**内核敢于答应超过物理总量的请求**。它凭什么敢赌?凭本性 4 之前的观察——人们申请的内存,大部分时候并不会同时全用上。这个"赌",就是后面 overcommit、按需分页、回收这套机制的起源。

### 本性 3:RAM 慢(相对 CPU 而言)

这一条容易被忽略。RAM 比磁盘快几十万倍没错,但**比起 CPU 自己的速度,RAM 其实慢得离谱**——CPU 几个时钟周期就能算完一道题,可去 RAM 取一个数据要等上百个周期。

**不这样会怎样?** 如果 CPU 每一条访存指令都老老实实等 RAM,那再快的 CPU 也得把大部分时间耗在干等上。于是硬件层面早就上了两道补救:CPU 内部的**缓存(cache)**,以及地址翻译用的 **TLB**(页表的速查缓存——这两个词后面会反复出现,先记住"它们都是用更快的存储来加速对 RAM 的访问")。

这条本性对软件设计的影响是:**内存管理的许多数据结构,不只是要"正确",更要"对缓存和 TLB 友好"。** 为什么页表是多级的、为什么伙伴系统要按 2 的幂对齐、为什么有 per-cpu 的快通道——背后都有"讨好硬件缓存"的影子。我们点到为止,细节留给各章。

### 本性 4:RAM 有"物理拓扑",不是一块均匀的膏药

这是最反直觉、却最重要的一条。

新手脑子里的 RAM 是"从 0 开始、连续、均匀的一大块"。**真相完全不是。** 真实的物理地址空间是这样一幅烂拼图:

- **地址 0 开始的一段被 BIOS 自己占了**;
- **显存**占走了某一段(你的显卡要有一块内存映射进来);
- **各种设备**的寄存器(MMIO)也占走了若干段;
- 真正能当普通内存用的 RAM,被这些"窟窿"切得**断断续续**。

更别提**多核服务器**上,内存条物理上分布在不同的 CPU 插槽旁边——访问"离自己近的"内存条比"离自己远的"快得多(这叫 **NUMA**,非一致内存访问)。

> **比喻**:你以为大楼是从 1 号到 16000 号一字排开。实际上中间好几层被改成了配电房、设备机房;而且这楼是分栋连廊结构,住东栋的人去东栋自己的房间近、跑去西栋拿东西要绕远路。

**不这样会怎样?** 如果内存管理假装"地址是连续均匀的一块",那么:

- 它会把"其实被显存占了"的地址当普通 RAM 发出去 → **写到显存里,系统崩**;
- 它会把"离 CPU 很远"的内存和"很近"的内存一视同仁 → **明明有近的不用,跑去用远的,白白变慢**。

所以这一条本性,逼出了两个设计:第一,内核**绝不**假装内存是连续的,它必须老老实实接收一份"哪些地址能用、哪些被占了"的**地图**(这就是下一章的 e820);第二,它要**按物理位置把内存分区、分节点**(这就是第 5 章的 ZONE 和第 13 章的 NUMA)。

> ⚠️ **一个贯穿全书的事实**:内存**不是**"从 0 开始连续 N 字节"。如果你只记本章一件事,就记这件。它解释了后面至少一半的"为什么"。

## 二、从四个本性,推出四个"根本需求"

RAM 这四个毛病摆在这儿了。现在问:面对这样一堆麻烦货,内核**必须**满足哪些需求,系统才转得起来?顺着四个本性反推,正好得到四个需求。

| RAM 的本性 | 逼出的根本需求 |
|------------|----------------|
| 1. 唯一且平铺(大家共用一栋楼) | **隔离**:每个住户只能碰自己的房间 |
| (同一栋楼,谁都能写) | **保护**:越界、越权必须被拦下 |
| 2. 有限(想要的远超实际) | **按需**:先开空头支票,用到才兑现 |
| 3. 慢 / 4. 有拓扑 | **高效**:数据结构要对缓存友好,分配要就近 |

下面把每个需求讲透:它到底防的是什么"如果不满足就会发生的灾难"。

### 需求 A:隔离——为什么不能让进程直接写物理地址

设想一个**没有隔离**的世界:所有程序直接拿物理地址用。进程 A 被分配了地址 `0x1000`~`0x2000`,进程 B 被分配了 `0x2000`~`0x3000`,大家各写各的。

**不这样会怎样?**

1. **一个 bug 就毁全机**:A 的代码有个 off-by-one,写穿了 `0x2000`,直接改坏了 B 的数据,甚至改坏了内核自己的代码。一个进程崩溃 = 整台机器崩溃。这就是 1970 年代以前"裸机编程"的真实处境,也是为什么那个年代的程序动不动就"把系统写死"。
2. **没法同时跑多个程序**:每个程序都得自己记住"我能用哪段地址",还得和别的程序商量"你用那段、我用这段"。这套协商机制如果让用户程序自己搞,既低效又不可信——谁能保证某个程序不耍赖占走一大段?

所以"隔离"不是锦上添花,而是**多任务系统能存在的先决条件**。它的实现手段,就是后面第 8 章的**虚拟内存**:给每个进程一套**独立的、从 0 开始的地址空间**,让 A 和 B 各自有各自的"100 号门牌",物理上却映射到不同房间。A 写坏了,永远只脏自己那块。

> **回扣比喻**:物业给每个住户发一本**私家门牌本**,住户只认自己本上的门牌号;物业的**台账(页表)**记录"住户 A 的 100 号 → 真实 5037 号房;住户 B 的 100 号 → 真实 8821 号房"。两个 100 号,物理上是两间毫不相干的房间。**隔离由此达成。**

### 需求 B:保护——光隔离还不够,还得拦得住越权

隔离解决了"你不该碰别人的房间"。但还有一种情况:**进程不该碰某些房间,哪怕它逻辑上"有权"知道这个地址**。比如:

- 内核的代码和数据,用户态进程**绝对不能**写,否则就能改掉内核的权限检查,整个安全体系崩溃;
- 代码段通常是**只读**的,防止程序自己把自己改了;
- 某些内存页标记为**不可执行**,防止"把数据当代码跑"(这是抵御缓冲区溢出攻击的关键一道闸)。

**不这样会怎样?** 如果只隔离不保护,那么任何进程只要拿到了一个地址,就能随便读写。恶意程序能改内核、改代码段、把数据当指令跑——这台机器上没有任何东西是可信的。1970~80 年代的蠕虫病毒之所以能横行,很大原因就是早期系统的内存保护太弱。

所以光知道"这间房是谁的"不够,**每间房门上还得挂牌子**:这间只读、这间可写、这间可执行、这间用户态禁入。越权了,硬件当场拦截(触发异常,内核介入,通常直接杀进程)。这套"挂牌子"的机制,就长在**页表条目**里(第 8、9 章详讲)。

> **隔离 vs 保护**:别把这俩混了。隔离是"你我各用各的地址"(防**越界**),保护是"即便你自己的地址,也得守权限"(防**越权**)。前者是"房间划分",后者是"房门上的锁"。

### 需求 C:按需——为什么内存可以"先欠着"

这一条,是 Linux 内存管理最有魅力、也最反直觉的地方。

物理 RAM 有限(本性 2),可需求总量远超物理。如果内核死板地要求"申请即兑现",那要么大量内存被申请了却闲着(浪费),要么机器根本跑不起这么多进程。

**不这样会怎样?** 假设内核严格执行"申请多少立刻占满多少物理页":

- 你 `malloc(1GB)`,内核立刻去占 1GB 物理页。可你大概率只用了其中几 MB。剩下 900 多 MB **白白占着**,别的进程要用却没货。
- 一台 16GB 机器,启动时各种程序申请的额度加起来轻松上百 GB,根本兑现不完 → 大量 `malloc` 直接失败。

观察现实:**程序申请的内存,和它真正访问的内存,差着好几个数量级。** 这就是内核敢于"超量答应"的底气——它赌大家不会同时把空头支票都拿兑现。

**所以这样设计**:申请(比如 `malloc`、`mmap`)时,内核**只在账本上圈一块门牌**(创建 VMA),**不配真实房间**;等进程**真的去访问**某扇门时,硬件发现"这门后头没房间",触发 **page fault**,内核这时才**临时去配一间**(第 9、10 章)。

> **全书最重要的一次视角转变**:
>
> **page fault 不是"错误",而是"兑现时机"。** 它是把"空头支票"变成"真金白银"的那一刻。理解这一点,你就理解了:为什么 `malloc` 之后内存占用没立刻涨、为什么 `fork()` 之后父子进程一开始共享同一份内存、为什么一个 10GB 的程序能在 4GB 的机器上跑——全靠"先欠着,要钱时再给"。

这是"懒"哲学的第一次亮相。它会在后面反复出现:VMA 只圈地不占地、page fault 才兑现、回收时把不用的房间收回来再欠出去……**用延迟换利用率**,是整个虚拟内存的灵魂。

### 需求 D:高效——既要对,还得快

前三个需求(隔离、保护、按需)主要解决"正确性"。但本性 3(RAM 慢)和本性 4(有拓扑)在不停提醒:**光做对不行,还得做得快、做得就近。**

- 每次 CPU 访问内存都要查页表,如果页表结构不友好,TLB 频繁失效 → 访存拖慢几十倍。所以页表要**多级**、要对齐、要"讨好 TLB"(第 8 章)。
- 分配内存是内核最最频繁的操作之一,如果每次都加锁扫一遍全局链表 → 多核上互相等锁。所以伙伴系统要有 **per-cpu 快通道**(第 6 章),slab 要有**每 CPU 一个样板间**(第 7 章),都为了"绝大多数情况无锁"。
- 在 NUMA 机器上,要尽量分配"离当前 CPU 近的"内存(第 13 章),否则跑得动但跑得慢。

"高效"这一条不是单独一个机制,而是**渗透在每一个机制的设计里**的约束。你后面看到任何一处"为什么这里要分快慢路径""为什么要 per-cpu""为什么按 2 的幂",答案多半都能追到这一条。

## 三、四个需求彼此打架:为什么逼出了"分层抽象"

到这一步,我们已经攒齐了:四个本性 → 四个需求。如果故事到此为止,内存管理不过是"建一本账本、配几把锁"的事。可真正的难处在于——**这四个需求是互相打架的**。这才是复杂性的真正来源。

我们看几个冲突:

**冲突一:隔离 ↔ 高效。**
要绝对隔离,最干净的做法是每个进程一份独立的物理内存副本,任何共享都禁止。可这浪费得离谱——100 个进程跑同一个 `ls` 程序,代码段完全一样,凭什么复制 100 份?所以内核在"隔离"和"高效"之间折中:**逻辑上隔离(地址空间独立),物理上能共享就共享**(比如只读的代码段、写时复制的页)。这个折中叫 **COW(copy-on-write)**,是第 9 章的主角之一。

**冲突二:按需 ↔ 性能。**
按需分页(用到才配房间)省内存,但意味着每次首次访问都要 page fault,而 fault 是个相对慢的操作(陷内核、分配、改页表)。如果一个程序一启动就疯狂首次访问每一页,会被 fault 拖垮。所以内核在"懒"和"预读"之间折中:多数情况懒加载,但对**文件顺序读**等模式会**预读**(readahead),一次 fault 多配几页,摊薄开销(第 11 章)。

**冲突三:有限 ↔ 谁都不让步。**
内存满了,要回收,可回收谁?文件缓存页可以丢(磁盘有原件),但丢早了又要重读,慢;匿名页要 swap,swap 更慢;什么都不想动 → 死锁。所以内核发明了**水位线 + LRU 老化 + kswapd 后台回收**(第 12 章),本质是在"留着当缓存"和"腾给新需求"之间动态平衡。

**冲突四:有拓扑 ↔ 通用性。**
要尊重 NUMA 拓扑(就近分配),就得多一套"节点"的概念,分配逻辑变复杂;要尊重设备 DMA 约束(只能用低地址),就得把内存分区(ZONE),分配时要选区。每加一类硬件约束,通用分配器就更复杂一分。这就是为什么 `alloc_pages` 看起来参数那么多——每一个参数都是在替某个硬件约束"说话"。

> **关键结论(本章的高潮)**:
>
> 这四个需求**没有任何一个能单独满足得又好又省**。隔离到底就浪费,懒到底就卡顿,守拓扑就复杂。**唯一能让四个需求同时被"差不多"满足的,不是某一个聪明算法,而是把问题分层——每一层只管一件事,层与层之间靠清晰的接口解耦。**

这就是导言结尾说的**三层抽象**的由来。它不是工程师炫技,是**被冲突逼出来的唯一出路**:

```
   进程看到的:   虚拟地址空间(满足隔离、保护、按需的"幻觉层")
                     ↑  页表 + page fault 把幻觉兑现
   内核分配的:   物理页(满足高效、就近的"实物层":ZONE + 伙伴系统 + slab)
                     ↑
   真实存在的:   RAM(满是洞、有拓扑的物理地址)
```

- **幻觉层**(虚拟内存 + 页表 + VMA)专心造"独占、连续、从 0 开始"的幻觉,把隔离和保护的全活儿揽下来;
- **实物层**(ZONE + 伙伴系统 + slab)专心管"真实的、有限的、有拓扑的房间",把高效和就近做到极致;
- 中间那道**页表 + page fault**,是连接两层的"翻译官 + 兑现窗口"。

每一层都不必知道另一层的全部细节。进程不必关心物理房间在哪儿;伙伴系统不必关心这页给了哪个进程。**正因解耦,才能各自演进、各自优化,合起来还能不出乱子。**

## 四、看一眼代码:这些"抽象"在源码里长什么样

这一章是概念章,本不急着贴代码。但我想给你一个**定心丸**:上面说的"房间""节点""账本",在内核源码里都对应着**真实存在、写得明明白白**的结构体。后面十二章,本质就是在逐个吃透这些结构体上的操作。

我们就看三处,建立"比喻 ↔ 代码"的对应感。**这一节不求懂,只求混个脸熟。**

### 一间房 = `struct page`

前面说"物理 RAM 被切成一间一间 4KB 的房间"。内核里,**每一个物理页都对应一个 `struct page`**——你可以把它理解成"这间房门后头挂的那张房卡":

> [include/linux/mm_types.h:72-400](../linux-6.14/include/linux/mm_types.h#L72-L400)

```c
struct page {
    unsigned long flags;        /* 这间房的状态牌:空闲?脏?锁定?... */
    union {
        struct {                /* 当房间"住着人"时(页缓存/匿名页) */
            union {
                struct list_head lru;   /* 挂在回收用的 LRU 链表上 */
                struct list_head buddy_list; /* 或挂在伙伴系统的空闲链表上 */
                struct list_head pcp_list;   /* 或挂在 per-cpu 快通道上 */
            };
            struct address_space *mapping;  /* 背后是谁(文件?匿名?) */
            ...
        };
        ...
    };
    ...
};
```

注意三件事,它们直接对应前面讲的抽象:

- **`flags`**——这间房"挂牌子"(状态)。需求 B 说的"保护"虽然主要在页表,但房间本身也有自己的状态(空闲、脏、锁定……)。
- **`lru` / `buddy_list` / `pcp_list` 三选一的 `union`**——同一间房,要么"住着人"(挂在 LRU 上等回收),要么"空着"(挂在伙伴系统或 per-cpu 链表上等分配)。**同一块内存,在不同生命阶段扮演不同角色**,这正是第 12 章回收和第 6 章分配交汇的地方。
- 整个 `struct page` 是个**带 `union` 的紧凑结构**——因为物理页动辄几百万个,`struct page` 必须尽量小(一个 `page` 数组就要占不少内存)。这是"高效"约束(需求 D)在最底层的一个体现:连描述房间的账本本身,都要省着用房间。

> 一个 `page` 数组,在内存里长这样(简化):
>
> ```
>  物理页 #0  物理页 #1  物理页 #2  ...  物理页 #N
>      │          │          │              │
>   struct page  struct page struct page  struct page   ← 这整片叫 vmemmap / mem_map
> ```
>
> 内核拿到一个物理地址,做一步简单除法就知道它对应哪个 `struct page`,反过来也一样。**物理房间和它的房卡之间,是 O(1) 的双向定位关系。** 这一点是后面伙伴系统、回收、page fault 兑现能高效运转的地基。

### 一栋楼划成几个区 = `struct zone`

本性 4 说"RAM 有拓扑,有些房间设备才用得上"。这个"分区"在代码里就是 `struct zone`:

> [include/linux/mmzone.h:842-1028](../linux-6.14/include/linux/mmzone.h#L842-L1028)

你不用现在读懂它,只要知道:**DMA、DMA32、NORMAL 这几个区,每一个都是一个 `struct zone` 实例**。伙伴系统的空闲链表、水位线(watermark)、per-cpu 快通道,**都是挂在 `zone` 上的**。换句话说,前面讲的"实物层"那一大坨机制,真正的"宿主"就是 `zone`。第 5 章我们会逐字段拆它。

### 一栋楼(或一个 NUMA 节点)= `struct pglist_data`

再往上,一台 NUMA 机器有多个 CPU 节点,每个节点有自己的本地内存。**每个节点**用一个 `struct pglist_data`(常简写 `pg_data_t`)描述:

> [include/linux/mmzone.h:1322-1495](../linux-6.14/include/linux/mmzone.h#L1322-L1495)

它里面装着"这个节点上有哪几个 zone、各多大、空闲多少"。分配内存时,内核先定**节点**(就近),再定 **zone**(满足硬件约束),最后在 zone 里找空闲页——**三级定位,正好对应三种本性的层层过滤**。

```
   pglist_data (节点:就近)  ──┐  本性4 拓扑
        └─ zone (区:满足设备约束) ──┐  本性4 拓扑
             └─ page (一间房:实际分配)   本性1 平铺
```

这张图你先扫一眼。**整本书的后半部分,基本就是在讲这条链上每一级是怎么动的。**

## 章末小结

让我们用大楼比喻把这一章收个口。

这一章我们做了一件纯粹的事:**搞清楚物业(内核)为什么非存在不可**。答案不在任何一个机制里,而在物理 RAM 本身的四个毛病里:

1. **楼只有一栋、还一字排开**(唯一且平铺)→ 不管就打架;
2. **房间总数有限,想要的远超总数**(有限)→ 不能死板地申请即兑现;
3. **楼虽大,跑腿却慢**(相对 CPU 慢)→ 管理手段得对缓存友好;
4. **楼里到处是配电房、设备机房,还分栋连廊**(有物理拓扑)→ 不能假装内存连续均匀。

这四个毛病,逼出了四个根本需求:**隔离、保护、按需、高效**。而这四个需求又彼此打架,没有任何单一手段能同时摆平它们——**于是唯一出路,是分层**:让"幻觉层"专心造独占的假象(隔离/保护/按需),"实物层"专心伺候真实的房间(高效/就近),中间用页表和 page fault 把两层翻译、兑现接起来。

> **回扣全书主线**:这一章我们站在最顶层,把"物理侧管切房间 / 虚拟侧管造幻觉"这个二分法**从根上立住了**。后面每一章,你要做的只是不断回到这个二分法,问一句:"这一步是给物理侧添砖加瓦,还是给虚拟侧织幻象?"
>
> - 第 2、3、4 章是**物理侧的前奏**:大楼怎么从黑灯瞎火、被盘点出一张满是洞的地图,到搭起临时工棚;
> - 第 5、6、7 章是**物理侧的主力**:ZONE 分区、伙伴系统切房间、slab 切小物件;
> - 第 8、9、10 章是**虚拟侧的核心**:多级页表、page fault 兑现、VMA 账本;
> - 第 11、12、13 章是**两侧的交汇与压力**:page cache 把文件搬进楼、回收/swap/OOM 在压力下腾房、NUMA/大页/memcg 处理现代场景。
