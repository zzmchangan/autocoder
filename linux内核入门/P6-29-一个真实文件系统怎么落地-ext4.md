# 第 29 章 · 一个真实文件系统怎么落地(以 ext4 为例)

> **前置**:你需要先读过**第 28 章《VFS》**(理由:本章讲的就是"谁来填 VFS 那几张空操作表"。VFS 的 superblock/inode/dentry/file 四张卡片是空框架,ext4 是第一个把它们填满的真实例子)。

**本章核心问题**:VFS 只定义了抽象卡片和空操作表,真正把数据摆上磁盘货架的是 ext4 这种具体文件系统。那磁盘上那串冷冰冰的字节到底是怎么组织的?它怎么和 VFS 的内存对象对上号?

**读完本章你会明白**:
- ext4 为什么把磁盘切成一个个 block group,每组都带一套位图 + inode 表 + 数据块;
- 磁盘上的 `ext4_inode`(磁盘格式)和内存里的 `struct inode`(VFS 对象)是怎么互相转换的;
- 一个文件的数据块到底怎么被找到(`i_block[15]` 的直/间/二间/三间指针,以及现代 ext4 的 extent 树);
- ext4 怎么"填 VFS 的操作表",从而真正接入 VFS。

> **如果一读觉得太难**:先只记住三件事——① 磁盘被切成 block group,每组有自己的"小账本"(块位图 + inode 位图 + inode 表),目的是让元数据和数据靠得近;② inode 里那 15 个 `i_block` 槽位是"数据块指针",现代 ext4 用 extent(一段连续区间的记录)而不是一个个单块指针;③ ext4 把自己的读/写/mmap 函数填进 VFS 的 `file_operations` 表,就算"接入了 VFS"。其余细节第二遍配合源码再抠。

## 一、不规划布局会怎样:找个 inode 得全盘扫描

磁盘(块设备)在内核眼里是什么?**一串从 0 开始编号的扇区(或块),平铺连续,没有任何"意义"**。第 0 号块、第 1 号块……就是一排货架,内核自己并不知道"哪个货架存的是哪个文件"。

如果 ext4 不做任何布局规划,把所有文件的数据随便丢在磁盘上,会发生什么?

- **找个 inode 得全盘扫描**:文件 `/etc/hosts` 的 inode 在磁盘第几块?不知道,得从头扫一遍磁盘找。
- **元数据和数据天各一方**:文件的 inode 在磁盘这头,数据块在磁盘那头,读一个文件磁头得跑个对穿。
- **碎片爆炸**:没有位图统一管理"哪些块空闲",分配新文件时怎么知道哪块是空的?

所以**每个文件系统都必须回答一个问题:这串货架,怎么分区、怎么编号、怎么记账、怎么找东西**。这个回答,就是文件系统的**磁盘布局(on-disk layout)**。

ext4 的回答是:**把磁盘切成一个个 block group(块组),每组自带一套"小账本"**。下面我们一层层拆开看。

---

## 二、ext4 的磁盘布局:把仓库切成一个个"小库房"

整个 ext4 分区,长这样(从第 0 块开始):

```
┌─────────────────────────────────────────────────────────────┐
│  Boot block(1024 字节,引导扇区,文件系统不管它)            │
├─────────────────────────────────────────────────────────────┤
│  超级块(superblock)副本 + 组描述符表                       │
├─────────────────────────────────────────────────────────────┤
│  Block Group 0                                              │
│   ├─ 超级块副本(可选,带 sparse_super 时不每组都有)        │
│   ├─ 组描述符(group descriptor)                            │
│   ├─ 块位图(block bitmap):这组里哪些数据块是空闲的         │
│   ├─ inode 位图(inode bitmap):这组里哪些 inode 号是空闲的  │
│   ├─ inode 表(inode table):这组的 inode 全存这儿          │
│   └─ 数据块(data blocks):实际存文件内容的地方             │
├─────────────────────────────────────────────────────────────┤
│  Block Group 1   (同样的结构)                              │
├─────────────────────────────────────────────────────────────┤
│  ... 一直到磁盘末尾                                         │
└─────────────────────────────────────────────────────────────┘
```

为什么要切成 group、每组都带一套位图和 inode 表?**核心动机是 locality(局部性)**:让一个文件的元数据(inode)和它的数据块尽量待在同一个 group 里,磁头少跑路,访问更快。同时,分配新文件时,内核只需看**本组的两张位图**就能决定"用哪个 inode 号、用哪些数据块",不用扫全盘。

我们逐个看这些组件。

### 1. 超级块:整个文件系统的户口本

超级块固定在分区偏移 1024 字节处(ext4 的"老传统",引导扇区占前 1024),它的磁盘格式就是 `struct ext4_super_block`([fs/ext4/ext4.h](../linux-6.14/fs/ext4/ext4.h#L1304-L1348)):

```c
struct ext4_super_block {
/*00*/	__le32	s_inodes_count;		/* Inodes count */
	__le32	s_blocks_count_lo;	/* Blocks count */
	__le32	s_r_blocks_count_lo;	/* Reserved blocks count */
	__le32	s_free_blocks_count_lo;	/* Free blocks count */
/*10*/	__le32	s_free_inodes_count;	/* Free inodes count */
	__le32	s_first_data_block;	/* First Data Block */
	__le32	s_log_block_size;	/* Block size */
	__le32	s_log_cluster_size;	/* Allocation cluster size */
/*20*/	__le32	s_blocks_per_group;	/* # Blocks per group */
	__le32	s_clusters_per_group;	/* # Clusters per group */
	__le32	s_inodes_per_group;	/* # Inodes per group */
	__le32	s_mtime;		/* Mount time */
...
/*38*/	__le16	s_magic;		/* Magic signature */
```

注意几个关键信息:

- `s_magic`:魔数 `0xEF53`。挂载时内核先读这里,**魔数对不上就拒绝挂载**——这是"你到底是不是个 ext4"的身份证。
- `s_blocks_count` / `s_inodes_count`:整个分区有多少块、多少 inode。
- `s_log_block_size`:块大小(通常是 4KB)。`s_blocks_per_group` / `s_inodes_per_group`:每组多少块、多少 inode。**这些参数把磁盘的"骨架尺寸"全定死了**。
- `s_feature_incompat`:不兼容特性集(比如 extent、64bit)。如果内核不认识某个不兼容特性,就拒绝挂载——防止把新格式文件系统读坏。

> 因为超级块这么关键,ext4 在多个 group 里都放了**超级块副本**(备份)。磁盘前几块坏了,还能从副本恢复。

### 2. 组描述符:每个小库房的"目录牌"

紧跟着超级块的,是一张**组描述符表**,每组一条 `struct ext4_group_desc`([fs/ext4/ext4.h](../linux-6.14/fs/ext4/ext4.h#L395-L408)):

```c
struct ext4_group_desc
{
	__le32	bg_block_bitmap_lo;	/* Blocks bitmap block */
	__le32	bg_inode_bitmap_lo;	/* Inodes bitmap block */
	__le32	bg_inode_table_lo;	/* Inodes table block */
	__le16	bg_free_blocks_count_lo;/* Free blocks count */
	__le16	bg_free_inodes_count_lo;/* Free inodes count */
	__le16	bg_used_dirs_count_lo;	/* Directories count */
	__le16	bg_flags;		/* EXT4_BG_flags (INODE_UNINIT, etc) */
	...
	__le16	bg_checksum;		/* crc16(sb_uuid+group+desc) */
```

它就是**每个 block group 的目录牌**:本组的块位图在哪个块、inode 位图在哪个块、inode 表从哪个块开始、本组还剩多少空闲块/inode。内核定位本组任何东西,都从这张牌出发。

注意字段名都带 `_lo`(low)和对应的 `_hi`(high)——这是 ext4 为支持超大磁盘做的"64 位拆分":地址的低 32 位和高 32 位分两个地方存(老 ext2/ext3 只有 32 位,磁盘一大就不够用了)。

### 3. 两张位图 + inode 表 + 数据块

每个 group 内部,按组描述符的指引:

- **块位图(block bitmap)**:一个 bit 代表本组一个数据块,1=已用,0=空闲。分配新数据块时,内核在这里找 0、置 1。
- **inode 位图(inode bitmap)**:一个 bit 代表本组一个 inode 号,同样的用法。
- **inode 表(inode table)**:本组所有 inode 的实际内容,连续存放。第 N 个 inode 就在表里第 N 个位置。
- **数据块**:剩下的全是存文件实际内容的货架。

这套结构的好处立刻体现:**建一个新文件**,内核只需在本组里:在 inode 位图找个空位(分配 inode 号)→ 在 inode 表对应位置写上元信息 → 在块位图找空位(分配数据块)→ 把数据写进去。全程只在本组的"小库房"里转,不用扫全盘。

## 三、磁盘上的 inode:档案本身的"小卡片"

一个文件在磁盘上的"户口",就是 inode 表里的一条 `struct ext4_inode`([fs/ext4/ext4.h#L771-L794](../linux-6.14/fs/ext4/ext4.h#L771-L794)):

```c
struct ext4_inode {
	__le16	i_mode;		/* File mode */
	__le16	i_uid;		/* Low 16 bits of Owner Uid */
	__le32	i_size_lo;	/* Size in bytes */
	__le32	i_atime;	/* Access time */
	__le32	i_ctime;	/* Inode Change time */
	__le32	i_mtime;	/* Modification time */
	__le32	i_dtime;	/* Deletion Time */
	__le16	i_gid;		/* Low 16 bits of Group Id */
	__le16	i_links_count;	/* Links count */
	__le32	i_blocks_lo;	/* Blocks count */
	__le32	i_flags;	/* File flags */
	union {
		...
	} osd1;				/* OS dependent 1 */
	__le32	i_block[EXT4_N_BLOCKS];/* Pointers to blocks */
	__le32	i_generation;	/* File version (for NFS) */
	...
```

对比第 28 章的 VFS `struct inode`,你会发现**字段几乎能一一对应**:`i_mode`、`i_uid`、`i_size`、`i_atime/mtime/ctime`、`i_links_count`……磁盘格式和内存对象本质上是同一份信息的两种表示。

**最大的不同在 `i_block[15]` 这个数组——它是"数据块指针"的入口**,决定了一个文件的数据到底存在磁盘哪些块上。下面专门讲它。

### 注意:磁盘 inode 不存名字

再次强调第 28 章讲的:`ext4_inode` 里**没有文件名字段**。名字存在**目录**里——目录本身也是一个文件,它的数据块内容,就是一堆"`<文件名, inode 号>`"的条目(directory entry)。打开 `/etc/hosts` 时,内核读 `/etc` 这个目录文件的数据块,在里面找到 `hosts → inode#123` 这条记录,才拿到 inode 号。

## 四、`i_block[15]`:数据块怎么被找到

这是本章最关键也最精彩的部分。`i_block` 是个 15 个元素的数组,每个元素 4 字节。它怎么定位文件数据?ext4 有两套机制(老的和新的)。

### 老机制:直/间/二间/三间指针(ext2 风格)

看这几个宏([fs/ext4/ext4.h](../linux-6.14/fs/ext4/ext4.h#L465-L469)):

```c
#define	EXT4_NDIR_BLOCKS		12
#define	EXT4_IND_BLOCK			EXT4_NDIR_BLOCKS
#define	EXT4_DIND_BLOCK			(EXT4_IND_BLOCK + 1)
#define	EXT4_TIND_BLOCK			(EXT4_DIND_BLOCK + 1)
#define	EXT4_N_BLOCKS			(EXT4_TIND_BLOCK + 1)   /* = 15 */
```

所以 `i_block[15]` 被分成四段:

```
i_block[0]  … i_block[11]   →  12 个直接指针(direct),每个指向一个数据块
i_block[12]                  →  一级间接指针(indirect),指向一个"装满指针的块"
i_block[13]                  →  二级间接指针(double indirect)
i_block[14]                  →  三级间接指针(triple indirect)
```

- **前 12 个**直接就是数据块的块号。小文件(≤48KB,假设 4KB 块)用这 12 个就够,直接定位,飞快。
- **第 13 个(一级间接)**指向一个"指针块"——这个块本身不存数据,而是装着更多数据块号(4KB 块能装 1024 个 4 字节指针),于是能多管 1024 个数据块(4MB)。
- **二级、三级**层层套娃,文件可以做到 TB 级。

**为什么不全部用直接指针?** 因为 `i_block` 只有 15 格,全用直接指针,文件最多 60KB。要支持大文件,必须用间接指针"用一块换一千块"。但间接指针每次多一次磁盘读(先读指针块,再读数据块),所以**小的部分用直接(快)、大的部分用间接(慢但能装)**——这是典型的"用分级换取既要快又要大"的权衡,和内存篇多级页表是一个思路。

### 新机制:extent 树(ext4 的主力)

老的直/间/三间指针有个大毛病:**大文件要跟一长串零散的指针块,碎片化时性能崩塌**。现代 ext4 默认开启 `extents` 特性,改用 **extent(范围)** 来记录数据块位置。

一个 extent 大意是:"**这个文件的第 0 到第 N 个逻辑块,连续存在磁盘第 M 到第 M+N 个物理块**"——一句话搞定一大段连续数据,不用一个个块记。

启用 extent 时,`i_block[15]` 这块空间被**重新解释**为一棵小小的 **extent 树的根节点**(`ext4_extent_header` + 若干 `ext4_extent`/`ext4_extent_idx`):叶子节点存"逻辑块范围 → 物理块范围",中间节点存索引。文件越大、越碎,这棵树越深,但绝大多数文件就一个 extent(因为连续),开销极小。

> 这就是为什么 ext4 比老 ext2/ext3 快很多、碎片也少:**它用"范围"代替"单块指针",一段连续数据只需一条记录**。这也是 ext4 的 `s_feature_incompat` 里 `EXT4_FEATURE_INCOMPAT_EXTENTS` 这个特性位存在的意义——老内核不认识 extent 就拒绝挂载,免得读坏。

## 五、从磁盘到内存:`ext4_inode` 怎么变成 `struct inode`

磁盘上的 `ext4_inode` 和 VFS 的 `struct inode` 是两种东西。挂载一个 ext4、或第一次访问一个文件时,内核得**把磁盘格式读进来,翻译成内存对象**。

这个过程的关键函数:

- **挂载时**:`mount` 系统调用 → ext4 的 `ext4_fill_super`([fs/ext4/super.c](../linux-6.14/fs/ext4/super.c))。它读超级块、校验魔数、解析特性位、分配 VFS 的 `struct super_block` 并把 ext4 的 `super_operations` 挂上去;同时把 ext4 自己的私有信息塞进 `super_block->s_fs_info`(就是 `struct ext4_sb_info`,[fs/ext4/ext4.h#L1495](../linux-6.14/fs/ext4/ext4.h#L1495),里面存着从超级块解析出来的各种参数)。
- **读一个 inode 时**:VFS 调 superblock 操作表里的 `alloc_inode` / `read_inode` —— ext4 实现 `ext4_iget`([fs/ext4/inode.c](../linux-6.14/fs/ext4/inode.c)),它**从磁盘把 `ext4_inode` 读出来,字段一一搬到内存的 `struct inode`**:`i_mode` → `inode->i_mode`,`i_size_lo` → `inode->i_size`,`i_atime` → `inode->i_atime_sec`……并且解析 `i_block` 成内存里方便用的 extent 缓存。

转换完成后,这个内存 `struct inode` 就被 VFS 缓存(inode cache),以后再用同一个文件就不用再读磁盘了。

### ext4 填表:真正"接入 VFS"的瞬间

光把数据搬进来还不够,ext4 还得**把 VFS 那几张空操作表填上自己的函数**,VFS 的派发(`file->f_op->read_iter` 之类)才能落到 ext4 的代码上。

看 ext4 怎么填 `file_operations`([fs/ext4/file.c](../linux-6.14/fs/ext4/file.c#L951-L975)):

```c
const struct file_operations ext4_file_operations = {
	.llseek		= ext4_llseek,
	.read_iter	= ext4_file_read_iter,
	.write_iter	= ext4_file_write_iter,
	.iopoll		= iocb_bio_iopoll,
	.unlocked_ioctl = ext4_ioctl,
	...
	.mmap		= ext4_file_mmap,
	.open		= ext4_file_open,
	.release	= ext4_release_file,
	.fsync		= ext4_sync_file,
	...
	.splice_read	= ext4_file_splice_read,
	.splice_write	= iter_file_splice_write,
	.fallocate	= ext4_fallocate,
	.fop_flags	= FOP_MMAP_SYNC | FOP_BUFFER_RASYNC |
			  FOP_DIO_PARALLEL_WRITE,
};
```

看明白了吗?**这就是上一章说的"操作表",ext4 把自己的函数填进去了**。当 VFS 的 `vfs_read` 走到 `file->f_op->read_iter` 时,如果这个 file 是 ext4 文件打开的,执行的就是这里的 `ext4_file_read_iter`。

`ext4_file_read_iter` 自己([fs/ext4/file.c#L130](../linux-6.14/fs/ext4/file.c#L130))其实也没干多少"读"的活——它做一些 ext4 特有的检查(加密?verity?),然后**把真正的读交给通用的 page cache 路径**(`generic_file_read_iter`)。这就是下一章的入口。

> 整个流程串起来:**磁盘 `ext4_inode` → 内存 `struct inode`(填好 ext4 的操作表)→ VFS 派发 `read_iter` → ext4 的函数 → 交给 page cache**。ext4 负责前半段(磁盘 ↔ inode),page cache 负责后半段(inode ↔ 市民)。两者在 `address_space` 这里接头。

## 六、关键源码精读:`i_block` 与 extent 的取舍

我们再聚焦看 `i_block[EXT4_N_BLOCKS]` 这一行([fs/ext4/ext4.h#L794](../linux-6.14/fs/ext4/ext4.h#L794)):

```c
	__le32	i_block[EXT4_N_BLOCKS];/* Pointers to blocks */
```

`EXT4_N_BLOCKS` = 15。这 15 个 4 字节槽位,既可以是老的"12 直 + 1 间 + 2 间 + 3 间",也可以被重新解释为 extent 树根。**到底按哪种解释,取决于 inode 的 `i_flags` 里有没有 `EXT4_EXTENTS_FL` 这个标志位**。

- 老式(无 extent):小文件直接 12 个指针够用;大文件靠多级间接指针,大且碎时性能差。
- 新式(extent):`i_block[0..3]` 当 extent 头 + 第一个叶子节点用,记录"逻辑块范围 → 物理块范围"。连续的大文件,一条 extent 就搞定几百 MB。

这个"同一块 60 字节空间,两种解释"的设计,完美体现了**向后兼容**的工程智慧:ext4 必须能读老的 ext2/ext3 创建的文件(它们用的是间接指针),又得给新文件用更高效的 extent。靠一个标志位区分两套解释路径,既不破坏老数据,又能享受新机制。

> 顺便说一句,`i_flags` 还有别的有趣位:`EXT4_INLINE_DATA_FL`——把**很小的文件直接塞进 `i_block` 这 60 字节里**(inline data),连数据块都不用分配。这是 ext4 为"海量小文件"场景做的优化。一个设计良好的文件系统,总是会在这些细节里塞满针对真实工作负载的妥协。

## 七、本章小结

用档案馆的比喻回顾本章:

- ext4 把**地下仓库(磁盘)**切成一个个 **block group(小库房)**,每个小库房自带**块位图、inode 位图、inode 表、数据块**。这样划分,是为了 **locality**(元数据和数据靠得近)和**快速分配**(在本组里就能决定用哪些块)。这是磁盘布局的"骨架"。
- 仓库里每个文件的"户口"是磁盘上的 **`ext4_inode`**(档案本身的小卡片),它和 VFS 内存里的 `struct inode` 字段几乎一一对应。**它不含文件名**——名字在目录文件的数据块里。
- 文件数据怎么定位?老式用 `i_block[15]` 的"直/间/二间/三间"指针(小文件直接、大文件多级间接);现代 ext4 默认用 **extent 树**(一段连续范围一条记录),又快又省。
- ext4 接入 VFS 的方式:**挂载时把磁盘 `ext4_inode` 翻译成内存 `struct inode`,并把自己的函数填进 VFS 的 `file_operations` 等操作表**。VFS 的派发一旦落到这些函数,ext4 就真正"上线"了。

回到全书主线:本章是**造幻觉那侧**(VFS 抽象)落到**真实硬件**(磁盘)的桥梁。ext4 证明了 VFS 的抽象不是空中楼阁——它确实能被一个真实文件系统填满、跑起来。

但请注意一个伏笔:`ext4_file_read_iter` 把真正的读"甩"给了 page cache。我们到现在为止,看到的还是"文件 → inode"这一层,**数据具体怎么从磁盘进到内存、怎么被市民拿到,还没展开**。这正是下一章的主题。
