# 第 1 章(全书第 20 章)· 中断与异常:硬件怎么拍政府的肩膀

> **前置**:你需要先读过进程篇(市民怎么出生、谁此刻占窗口)和本篇 [导言](P4-00-导言-全局直白解释.md)(建立"三种惊动"的总图)。本章是导言里那道**门铃(中断 IRQ)**和**报警器(异常)**的全展开——它俩走的是同一条"陷入内核"的流水线,所以放在一起讲。
>
> **核心问题**:网卡突然收到一个包、你按下键盘——硬件自己没有 CPU 时间,它凭什么能让正在跑的进程"让一让",把内核叫出来处理?内核又怎么在一片混乱中,**准确**找到"这个中断该交给哪个驱动"?而内核自己,为什么一旦进了中断处理就"不能睡眠、不能磨蹭"?
>
> **读完本章你会明白**:
> - 为什么 CPU 能被一根电信号**瞬间**从用户态拉到内核态(IDT 这张"门牌总表"是干什么的);
> - 为什么 `irq_desc` / `handle_irq` / `action` 这套多层抽象,是被"硬件五花八门的脾气"逼出来的;
> - 为什么**硬中断里不能干太久、不能睡眠**——这个硬约束如何逼出"上半部 / 下半部",以及 softirq / tasklet / 工作队列各自管哪一摊;
> - 异常(除零、page fault、非法指令)和中断共享多少基础设施,又差在哪里。

> **如果一读觉得太绕**:先只记住三件事——① **中断是硬件主动拍肩膀、异常是 CPU 自动报警,两者都靠 IDT 这张门牌表把 CPU 瞬间跳到对应处理函数**;② **硬中断里关着中断、没有进程上下文,所以不能睡眠、不能磨蹭**,于是拆成"上半部扒数据 + 下半部干费时活";③ **下半部有三档:softirq(最快、不睡眠)、tasklet(在退场)、工作队列(能睡眠)**。其余分层细节第二遍配合源码再抠。

## 一、先看清"门铃"和"报警器"长什么样

[导言](P4-00-导言-全局直白解释.md)里我们说,三种惊动本质都是"陷入内核"。中断和异常是其中两道门:

- **中断(IRQ,Interrupt Request)**:外部硬件主动发的电信号。异步——什么时候来,内核事先不知道。
- **异常(exception)**:CPU 执行某条指令时,自己检测到状况(除零、缺页、非法指令),自动触发。同步——绑在那条具体指令上。

两者在"过门 5 步"上几乎完全一样:都是 CPU 把当前现场存好、切到内核态、查表找处理函数、处理、回来。**它们共享同一张门牌表(IDT)、同一套保存/恢复现场的工具、同一段进出门的骨架代码(`irqentry_enter/exit`)。** 第 4 章会把这套统一性收束起来,本章先把它们各自走一遍。

我们从**中断**切入,因为它牵出的机制(上半部/下半部、softirq、工作队列)最多,也是本章的真正主角;异常作为它的"同步兄弟",在关键处对比着讲。

## 二、硬件怎么按门铃:从中断线到 CPU

### 场景

你插着网线的机器,网卡忽然收到一个以太网包。网卡要通知 CPU "有货到了,快来收"。可网卡自己没有 CPU 时间,它只会做一件事:**在一根中断线上发个电信号**。

### 不这样会怎样:如果硬件不能主动通知

设想硬件没有"拍肩膀"的能力。那 CPU 怎么知道网卡有包?只剩一条路:**轮询(polling)**——CPU 死循环去问每个设备"你有事吗?你有事吗?"。

**不这样会怎样?**

- **CPU 被白白烧光**:99% 的时间设备啥事没有,CPU 却在不停问,占满算力。
- **响应慢**:你问完键盘问完磁盘,转一圈再回来问网卡,这期间包可能就丢了。
- **能耗爆炸**:笔记本待机时 CPU 还在疯狂轮询,电池半小时见底。

所以硬件**必须**有"主动打断 CPU"的能力——这就是中断存在的根本理由:**让 CPU 在没事时安心干活,有事时被立刻叫起来,而不是傻乎乎地全程盯着。**

### 中断控制器:上百个门铃,几根线

机器里的设备几十上百个,但 CPU 的中断管脚就那么几个。怎么汇总?靠**中断控制器**。x86 上是 **APIC(Advanced Programmable Interrupt Controller)**——主板上一颗 **I/O APIC**(汇总外设信号)+ 每个 CPU 里一颗 **Local APIC**(负责本地分发、还内置了时钟/性能计数器中断)。

> APIC 像市政大厅的**门卫室**:来访者(设备中断信号)先到门卫室,门卫室给每个人发个**号牌(vector,中断号)**,排队、按优先级决定什么时候戳哪个 CPU。

为什么需要这个号牌?因为 CPU 收到"被戳了"之后,得知道**是谁**戳的、该查哪张表项。APIC 会把对应的 vector 塞给 CPU,CPU 拿 vector 去查 IDT。这一步把"硬件哪根线"和"软件哪个处理函数"解耦——设备驱动只要注册"我的中断来了,调这个函数",不用关心自己挂在 APIC 的哪个管脚。

## 三、IDT:门牌总表,CPU 查表只用一步

### 这张表是什么

CPU 被中断信号戳到后(或异常触发后),它**硬件**做一件事:拿一个编号,**去查 IDT**。IDT 是一张数组,索引是编号(vector),每一项是一个**门描述符**,核心内容就是"这个编号的事,入口代码的地址是几"。

> x86 上 vector 0~31 是 CPU **保留给异常**的(0 号除零、14 号 page fault、13 号通用保护错误 GP……);32 号往后才是外部中断和系统向量(系统向量 sysvec 是 CPU 内部产生的"软中断式"事件,比如时钟中断、IPI 核间中断、SPURIOUS 假中断)。

IDT 在启动时由内核填好——本仓库里这张表的"登记册"是 [arch/x86/kernel/idt.c](../linux-6.14/arch/x86/kernel/idt.c):

```c
void __init idt_setup_traps(void)            /* 登记异常那 0~31 号 */
void __init idt_setup_apic_and_irq_gates(void) /* 登记外部中断和系统向量 */
```

> [arch/x86/kernel/idt.c:232](../linux-6.14/arch/x86/kernel/idt.c#L232) 和 [arch/x86/kernel/idt.c:284](../linux-6.14/arch/x86/kernel/idt.c#L284)。这套登记在 `start_kernel` 那条初始化大游行里很早就完成(内存篇讲过 `start_kernel`),内核先把"门牌总表"挂好,之后随时能接客。

### 不这样会怎样:如果没有 IDT 这张表

**不这样会怎样?** 如果硬件不知道"这个编号的事归谁管",CPU 只能跳到一个**固定的**入口,然后让内核在那里用一大堆 `if/vector == ...` 去判断"是谁来了"。这样既慢(每个中断都要遍历判断),又把"硬件编号"和"软件处理"死死焊在一起,加一个设备就得改内核。

有了 IDT,CPU 查表是**纯硬件、一步、O(1)**:拿到 vector → 取表项 → 跳到入口地址。这就是为什么中断的"分诊"快得惊人——大部分工作硬件替你做了。

> **回扣比喻**:IDT 就是市政大厅进门那块"部门楼层分布图"。来访者进门先看图直接上楼,不在大厅里转。每个编号一个楼层,登记一次,终身受用。

## 四、过门:汇编入口保存现场

查到入口地址后,CPU 跳到那里执行。入口代码是**汇编**写的(x86 上由一组宏在 [arch/x86/entry/entry_64.S](../linux-6.14/arch/x86/entry/entry_64.S) 里展开),干的就是导言说的"过门 5 步"的前两步:保存现场 + 切到内核态。

核心动作:

1. **保存现场**:把用户态那一刻的所有寄存器(RAX、RBX、RIP、RSP……)一股脑压进栈。这一堆东西在内核里有个名字——**`struct pt_regs`**。它就是"市民被打断时的完整快照",内核处理完中断,要靠它原样恢复。
2. **切到内核栈**:用户态的栈不能给内核用(不安全、可能很小)。每个线程都预先准备了一个**内核栈**,专门为"陷入内核时"用。入口代码把 RSP 换成内核栈顶。
3. **切地址空间 + 提权限**:把 CR3 换成内核页表(内核地址空间),把 CPU 运行级从用户态(_RING 3)提到内核态(RING 0)。

> 注意:这一套做完后,**中断在这个 CPU 上通常是关闭的**。原因下面讲——这是本章后半段"上半部/下半部"的祸根。

### 通用骨架:三种惊动共用 `irqentry_enter`

过完门的纯硬件动作,接下来进入**架构无关**的通用代码。这里有一段三种惊动共用的骨架——`irqentry_enter`:

> [kernel/entry/common.c:236](../linux-6.14/kernel/entry/common.c#L236)

```c
noinstr irqentry_state_t irqentry_enter(struct pt_regs *regs)
{
	...
}
```

它干的活是"进门登记":进入 RCU 临界区(内存篇和同步篇会反复讲 RCU)、给当前进程的 CPU 时间记账(这次中断算谁的)、把抢占计数里的 `HARDIRQ_OFFSET` 加一(标记"我现在在硬中断里")。

> 这段代码是"统一性"的铁证——**不管你是外部中断、异常还是系统调用,进了门都得先过它**。它把"进/出内核上下文"这件事统一封装,免得每种惊动各写一套。第 4 章会专门回来看它。

骨架跑完,才分流到具体的"处理函数"。中断和异常在这里**分道扬镳**:外部中断走 IRQ 子系统(`irq_desc` 一套),异常走各自的 trap handler(page fault 走 `do_page_fault`,除零走 `math_error`……)。我们顺着中断这条路走到底。

## 五、中断的"多层分诊":从 vector 到驱动处理函数

到这儿,通用的"过门"做完了。但对**外部设备中断**,内核还套了好几层抽象,才最终调到你驱动里写的那行 `my_interrupt_handler()`。这几层看起来啰嗦,但每一层都被"硬件五花八门的脾气"逼出来。我们一层层拨开。

### 第 1 层:`irq_desc`——一个中断一份档案

内核给每一个 IRQ 号维护一份档案,叫 **`irq_desc`**([include/linux/irqdesc.h](../linux-6.14/include/linux/irqdesc.h)):

> [include/linux/irqdesc.h:67](../linux-6.14/include/linux/irqdesc.h#L67)

```c
struct irq_desc {
	...
	irq_flow_handler_t	handle_irq;   /* 高层流控处理器 */
	struct irq_data		irq_data;      /* 底层硬件数据:挂在哪个芯片、vector 几号 */
	...
	struct irqaction	*action;      /* 驱动注册的处理函数链 */
	...
};
```

记住这三个字段,后面全是它们:

- `irq_data`:这个中断"硬件上是谁"——挂在哪个中断芯片、vector 是几号、触发方式是什么。
- `handle_irq`:这个中断"软件上怎么走流程"——**高层流控处理器**,决定先应答硬件还是先跑驱动。
- `action`:这个中断"挂了哪些驱动"——一条链表,记录所有"我要处理这个中断"的驱动处理函数。

### 第 2 层:`handle_irq`——高层流控处理器

为什么不能一进来就调 `action` 里的驱动函数?因为**硬件触发方式五花八门**,得先按规矩应答硬件,否则中断永远解除不掉(同一个中断会反复响):

- **电平触发(level)**:中断线一直保持"有效电平",直到设备被告知"我收到了"。你不应答,电平不掉,中断会一直响。
- **边沿触发(edge)**:中断线上的一个跳变(0→1)就算一次。边沿来了就来了,要注意别漏。

`handle_irq` 这个字段就是**流控处理器**。内核按硬件类型挑一个现成的实现([kernel/irq/chip.c](../linux-6.14/kernel/irq/chip.c)):

> [kernel/irq/chip.c](../linux-6.14/kernel/irq/chip.c) 里几种典型实现:
> - `handle_level_irq`([628](../linux-6.14/kernel/irq/chip.c#L628)):电平触发——先屏蔽、再跑驱动、再开屏蔽。
> - `handle_edge_irq`([787](../linux-6.14/kernel/irq/chip.c#L787)):边沿触发。
> - `handle_fasteoi_irq`([687](../linux-6.14/kernel/irq/chip.c#L687)):现代硬件最常见的——硬件保证"写 EOI 寄存器就应答",流控只管在合适时机回写 EOI。

我们看 `handle_fasteoi_irq`(MSI/MSI-X、现代 PCI 设备大多走它):

> [kernel/irq/chip.c:687-737](../linux-6.14/kernel/irq/chip.c#L687-L737)(精简)

```c
void handle_fasteoi_irq(struct irq_desc *desc)
{
	...
	if (unlikely(irqd_irq_disabled(&desc->irq_data) || irqd_irq_inprogress(&desc->irq_data))) {
		...                                  /* 这个中断被禁用或在处理中:先记下,别跑 */
		goto out;
	}
	...
	handle_irq_event(desc);                   /* 真正去跑 action 链上的驱动处理函数 */

	cond_unmask_eoi_irq(desc, chip);          /* 最后回写 EOI:告诉硬件"我处理完了,可以再来" */
}
```

它做的事就是**"先检查能不能跑 → 跑驱动 → 最后回写 EOI 应答硬件"**。注意 EOI(end of interrupt)放在最后——**这是关键**:驱动处理函数还在跑的时候,硬件其实还没被"解禁",同号中断暂时进不来(防止同一个中断自己踩自己)。

> **不这样会怎样?** 如果不先把"应答硬件"这件事和"跑驱动"分清楚:电平触发的中断,你还没跑完驱动,硬件电平还挂着,结果同一个中断被反复触发,CPU 100% 卡死在同一个中断上。流控处理器的存在,就是把"硬件脾气"和"驱动逻辑"解耦——驱动只管写"我收到中断了干啥",硬件的应答时序交给这一层。

### 第 3 层:`action` 链——驱动注册的处理函数

流控处理器调到 `handle_irq_event`,它去遍历 `action` 链,挨个调驱动注册的处理函数。核心是 `__handle_irq_event_percpu`:

> [kernel/irq/handle.c:139-189](../linux-6.14/kernel/irq/handle.c#L139-L189)(精简)

```c
irqreturn_t __handle_irq_event_percpu(struct irq_desc *desc)
{
	irqreturn_t retval = IRQ_NONE;
	struct irqaction *action;

	for_each_action_of_desc(desc, action) {       /* 挨个调挂在这个 IRQ 上的驱动处理函数 */
		irqreturn_t res;

		...
		res = action->handler(irq, action->dev_id);   /* ← 你写的中断处理函数在这里被调用 */

		if (WARN_ONCE(!irqs_disabled(), ...))          /* 处理函数里中断必须一直是关的 */
			local_irq_disable();

		switch (res) {
		case IRQ_WAKE_THREAD:
			/* 驱动说"上半部干完了,把费时活儿交给我的内核线程"→ 唤醒中断线程 */
			...
			__irq_wake_thread(desc, action);
			break;
		default:
			break;
		}
		retval |= res;
	}
	return retval;
}
```

两个要点:

1. **一个 IRQ 可以挂多个驱动**(`action` 是个链表)。这叫**中断共享**——PCI 设备经常几个共用一根 IRQ 线。内核挨个问"是你的事吗",谁返回"是我的(`IRQ_HANDLED`)"就算谁的。
2. **处理函数返回 `IRQ_WAKE_THREAD`**:这是**线程化中断(threaded interrupt)**的钩子——驱动可以选择"上半部只记一笔、费时活儿丢给一个专门的内核线程"。这其实就是"上半部/下半部"思想的一种实现(后面会讲)。

### 驱动怎么注册:`request_threaded_irq`

驱动初始化时,用 `request_threaded_irq` 把自己的处理函数挂上去:

> [kernel/irq/manage.c:2141](../linux-6.14/kernel/irq/manage.c#L2141)

```c
int request_threaded_irq(unsigned int irq, irq_handler_t handler,
                         irq_handler_t thread_fn,
                         unsigned long irqflags, const char *devname,
                         void *dev_id)
```

`handler` 就是"上半部处理函数",`thread_fn` 是可选的"线程化下半部"。它内部会分配一个 `struct irqaction`,填好你的两个函数,挂到对应 `irq_desc` 的 `action` 链上。

`struct irqaction` 长这样([include/linux/interrupt.h:122](../linux-6.14/include/linux/interrupt.h#L122)):

```c
struct irqaction {
	irq_handler_t		handler;    /* 上半部 */
	...
	irq_handler_t		thread_fn;  /* 线程化下半部 */
	struct task_struct	*thread;    /* 给下半部用的内核线程 */
	...
	struct irqaction	*next;      /* 链表:下一个共享的驱动 */
};
```

> 到这里你看到了完整的分诊链:**vector → IDT → 入口汇编 → `irqentry_enter` → `irq_desc` → `handle_irq`(流控)→ `handle_irq_event` → `action->handler`(你的驱动函数)**。六层,每一层解耦一类问题:编号解耦、硬件触发方式解耦、共享解耦。这是内存篇"分层抽象"在本篇的重演。

## 六、关键约束:硬中断里不能干太久、不能睡眠

到这里,驱动处理函数被调到了。但有一个**铁律**压在它头上:

> ⚠️ **硬中断处理函数(上半部)里:① 不能睡眠 / 不能调度;② 不能持有需要睡眠才能拿到的锁;③ 必须尽快返回。**

这条铁律不是"建议",是会直接死机的硬约束。理解它,就理解了后面一切设计的来源。

### 为什么不能睡眠

**硬中断没有"进程上下文"**。普通进程想睡眠,内核把它从 CPU 上拿下来、挂到等待队列、记录"它在等什么"——这一切依赖"有一个进程可以被挂起"。可硬中断跑在**中断上下文**——它不是任何一个进程在执行,它借用的是"被中断那一刻正好在这个 CPU 上的进程"的内核栈,但**它不代表那个进程**。

**不这样会怎样?** 如果硬中断里 `sleep`/等一个互斥锁/分配需要换出的内存:内核想把它挂起,却发现"我没有一个真正的进程可以挂起"——当前这个 task_struct 此刻根本不该对中断负责。结果要么直接 oops/panic,要么挂起后再也醒不过来(没有人知道该唤醒谁)。

所以内核里有个检查:在可能睡眠的地方断言 `!in_atomic()`,在中断上下文里睡眠会立刻触发 BUG。

### 为什么不能磨蹭

更现实的约束:**此刻中断是关的(至少本 CPU 上同号中断进不来)**。你磨蹭一微秒,这一微秒里:

- 别的设备中断进不来(键盘、网卡被堵);
- 被中断的进程卡住不动;
- 最严重——**时钟中断**被堵:连调度器都得不到心跳,这个 CPU 上的时间停摆。

**不这样会怎样?** 网卡在高负载下,每次中断都磨几微秒收包进栈,CPU 会被中断处理完全占满,用户进程一个字节都跑不了——这叫"中断风暴(live lock)"。

> **回扣比喻**:门铃响了,门卫(上半部)冲到门口。他绝不能在门口跟来访者泡茶聊天——那会堵死整条进门的通道,别的门铃响了都没人接。门卫的规矩是:**最快把来访者领进门、登个记,立刻把门让出来**。真正要陪来访者办的事(泡茶、谈事),交给后厨(下半部)在后面慢慢搞。

这条"不能磨蹭"的铁律,逼出了 Linux 最经典的一套设计——**上半部 / 下半部分离**。

---

## 七、上半部与下半部:把费时活儿推迟执行

### 痛点

网卡收到一个包。完整处理它要做的事:

1. 从网卡硬件寄存器把包数据扒下来(必须立刻做,否则数据可能被下一个包冲掉);
2. 把包丢进接收队列、做校验、过协议栈、唤醒等待读 socket 的进程——这些**费时**,但晚个一两毫秒也无所谓。

第 1 步必须在上半部做(关中断、抢时间);第 2 步如果也在上半部做,就违反了"不能磨蹭"。

### 所以这样设计:上半部扒数据 + 标记"有活儿",下半部干费时活

解法是**推迟执行(deferred work)**:

- **上半部(硬中断,`action->handler`)**:只做最要紧的——扒数据、记一笔"我这边有活儿要干"。然后**立刻返回**,让中断尽快结束、开门。
- **下半部**:被推迟的费时活儿。它**开中断**跑(能被新中断打断),有的还能睡眠。下半部什么时候开始跑?在 `irq_exit`(中断要返回用户态的前一刻)检查"有没有挂起的软中断",有就就地跑。

这套思想在内核里有三种实现,逐层从"快但受限"到"慢但自由":

---

### 下半部之一:softirq(软中断)

最底层、最快、最原始的下半部机制。

**它怎么跑**:内核维护一个全局的 `softirq_vec` 数组,每项是一个软中断类型(TIMER、NET_RX、NET_TX、TASKLET、RCU……)和它的处理函数。上半部要推迟活儿时,调 `raise_softirq` 把对应类型"置位"——其实就是在 per-CPU 的一个位图里打个标记。

> [kernel/softirq.c:60](../linux-6.14/kernel/softirq.c#L60)

```c
static struct softirq_action softirq_vec[NR_SOFTIRQS] __cacheline_aligned_in_smp;
```

真正跑软中断的引擎是 `handle_softirqs`(老代码里叫 `__do_softirq`,6.14 重构了):

> [kernel/softirq.c:518-592](../linux-6.14/kernel/softirq.c#L518-L592)(精简,保留关键控制流)

```c
static void handle_softirqs(bool ksirqd)
{
	unsigned long end = jiffies + MAX_SOFTIRQ_TIME;   /* 最多跑这么久就得交还给 CPU */
	int max_restart = MAX_SOFTIRQ_RESTART;            /* 最多重跑这么多次 */
	struct softirq_action *h;
	__u32 pending;
	int softirq_bit;

	pending = local_softirq_pending();   /* 读 per-CPU 位图:哪些软中断挂着 */
	...
restart:
	set_softirq_pending(0);               /* 清位图(先清,再开中断,允许新软中断置位) */
	local_irq_enable();                   /* 关键:开中断!软中断跑的时候允许硬中断进来 */

	h = softirq_vec;
	while ((softirq_bit = ffs(pending))) { /* 找到下一个置位的软中断 */
		h += softirq_bit - 1;
		...
		h->action();                     /* 调它的处理函数 */
		h++;
		pending >>= softirq_bit;
	}

	local_irq_disable();
	pending = local_softirq_pending();    /* 跑完再查:有没有新的软中断又置位了 */
	if (pending) {
		if (time_before(jiffies, end) && !need_resched() && --max_restart)
			goto restart;             /* 没超时、没人要 CPU:再跑一轮 */
		wakeup_softirqd();               /* 超时了或别人要 CPU:交给 ksoftirqd 内核线程兜底 */
	}
}
```

读这段,你抓住了 softirq 的灵魂:

1. **开中断跑**:`local_irq_enable()`——下半部不再是"关中断独占",新中断能进来。这是它和上半部的本质区别。
2. **有兜底,不无限霸占**:`max_restart` 和 `MAX_SOFTIRQ_TIME` 两道闸——软中断不能无限重跑。如果软中断太多跑不完,就唤醒 `ksoftirqd` 内核线程**在进程上下文里慢慢跑**(这时候它就是个普通进程,可以被调度、让出 CPU)。这是"既保证下半部及时执行,又不让它饿死用户进程"的精妙平衡。
3. **绑定在本 CPU**:软中断基本在"哪个 CPU raise 的就在哪个 CPU 跑",per-CPU 位图,几乎无锁。

**什么时候触发它跑?** 在 `irq_exit`(中断返回时)检查:

> [kernel/softirq.c:652-661](../linux-6.14/kernel/softirq.c#L652-L661)

```c
static inline void __irq_exit_rcu(void)
{
	...
	preempt_count_sub(HARDIRQ_OFFSET);                 /* 退出硬中断上下文 */
	if (!in_interrupt() && local_softirq_pending())    /* 不在(嵌套的)中断里,且有挂起软中断 */
		invoke_softirq();                              /* 就地跑软中断 */
	...
}
```

注意条件 `!in_interrupt()`:只有"硬中断都退干净了"(不是中断嵌套的场景),才就地跑软中断。否则推迟——避免软中断在中断嵌套深处乱跑。

> **一句话**:softirq = "中断快返回时,顺手把攒下的费时活儿在开中断的状态下跑了;跑不完就丢给 ksoftirqd 线程"。延迟最低,但它**仍然不能睡眠**(还是借用上下文,不是真正进程)。

---

### 下半部之二:tasklet(在退场的老兵)

tasklet 是建在 softirq 之上的一层薄封装。它给写驱动的人一个友好保证:**同一个 tasklet 不会在多个 CPU 上同时执行**(虽然不同 tasklet 可以并行)。这降低了驱动里加锁的心智负担。

> [kernel/softirq.c:933-934](../linux-6.14/kernel/softirq.c#L933-L934)

```c
open_softirq(TASKLET_SOFTIRQ, tasklet_action);
open_softirq(HI_SOFTIRQ, tasklet_hi_action);
```

它本质上就是"软中断 TASKLET_SOFTIRQ 触发时,内核去遍历 tasklet 链表挨个跑"。tasklet 的"同 tasklet 不同时跑"是靠给每个 tasklet 加状态锁实现的。

> **现状提醒**:tasklet 在内核里正在被**逐步淘汰**(官方认为它的语义和性能都不如 softirq / 工作队列,且给多核加锁添乱)。新代码别再用。但你看老驱动、老资料会大量遇到它,所以得认识。它的命运本身就是一个"设计演进"的故事:早期为了方便,后来发现是负担,于是退场。

---

### 下半部之三:工作队列 workqueue(唯一能睡眠的下半部)

如果中断里要干的活儿**必须睡眠**——比如要分配大块内存(可能触发回收换页)、要等磁盘 IO、要拿一个会被进程长期持有的锁——softirq/tasklet 都不行,**只能用工作队列**。

**它怎么跑**:工作队列的本质是"**把活儿丢给一个内核线程去干**"。既然是在线程上下文里,那就能睡眠、能被调度、能持有任意锁——它和一个普通进程没本质区别,只是它在跑内核代码、干内核派发的活儿。

驱动用 `INIT_WORK(work, my_func)` 把一个函数包成 `work_struct`,然后 `queue_work` 丢进队列。内核维护一组**工作者线程(kworker)**,从队列里取活儿来跑。

> [include/linux/workqueue.h:508](../linux-6.14/include/linux/workqueue.h#L508) 是 `alloc_workqueue` 的声明——驱动可以自己建一个专属工作队列(`alloc_workqueue`),也可以用内核默认的 `system_wq`(`queue_work` 默认走它)。`struct work_struct`([include/linux/workqueue.h:114](../linux-6.14/include/linux/workqueue.h#L114))就是"一个待办的活儿"。

> **三种下半部的取舍**:
>
> | 机制 | 能不能睡眠 | 延迟 | 适用 |
> |---|---|---|---|
> | softirq | ❌ | 最低(中断返回时就地跑) | 高频、性能敏感(网络、RCU、定时器) |
> | tasklet | ❌ | 低(在退场) | 老驱动,别用 |
> | 工作队列 | ✅ | 较高(要等线程被调度) | 要分配内存/等 IO/拿可睡眠锁的活儿 |

这就是"上半部/下半部"的全部:**硬约束(不能睡眠、不能磨蹭)逼出推迟执行;推迟执行的快慢,又按"能不能睡眠"分了三档**。

---

## 八、异常:中断的"同步兄弟"

讲完中断,异常就好懂了——它是同一条流水线上的另一种触发方式。区别只在第 0 步:

- **中断**:外部硬件发信号 → CPU 查 IDT。**异步**,不知道何时来。
- **异常**:CPU 执行某条指令时,自己检测到状况(除零、缺页、非法指令、断点)→ CPU **自动**查 IDT。**同步**,绑在那条指令上。

vector 0~31 是异常专属,在 `idt_setup_traps`([arch/x86/kernel/idt.c:232](../linux-6.14/arch/x86/kernel/idt.c#L232))里登记。比如:

- **0 号 `#DE`(除零)**:除法指令分母为 0 → CPU 触发 → 内核走 `math_error`,通常给进程发 `SIGFPE`。
- **14 号 `#PF`(page fault)**:访问的虚拟地址页表里没映射或越权 → 这就是内存篇 [第 9 章](P2-09-page-fault-虚拟到物理的按需兑现.md)的主角。**fault 类型**:内核"修好"(配一页)后,让进程**重新执行那条指令**。
- **13 号 `#GP`(通用保护错误)**:在用户态执行特权指令、或访问不该访问的段 → 内核发 `SIGSEGV`,经典"段错误"。
- **3 号 `#BP`(断点)/1 号 `#DB`**:调试用的 **trap 类型**:处理完**继续下一条指令**。

异常和中断共享多少基础设施?几乎全部:同一种"保存现场(`pt_regs`)"、同一段进门骨架(`irqentry_enter`/`irqentry_exit`)、同样会走到 `irq_exit` 检查软中断。**它俩唯一的真正区别,就是"谁触发的、能不能预测"。** 第 4 章会把这个统一性画成一张大图。

> 一个常被忽略的点:异常**有"当前进程"**(就是执行那条指令的进程),所以异常处理可以合法地做很多事——比如 page fault 里给当前进程配一页、发信号给当前进程。而外部中断**没有"当前进程"可言**(它打断谁算谁的),所以中断处理不能假设自己在为某个特定进程服务,更不能睡眠。这就是"异常能关联进程、中断不能"的根源。

---

## 九、关键源码精读

这一章源码多而散,我们精读两个最能体现设计的:入口宏 `DEFINE_IDTENTRY_SYSVEC`(怎么把一个 vector 变成"过门骨架 + 具体处理")和 `handle_softirqs`(下半部引擎的核心控制流)。把这两段读通,本章就通了。

### 精读一:`DEFINE_IDTENTRY_SYSVEC`——一个宏展开出整个"过门"骨架

> [arch/x86/include/asm/idtentry.h:248-272](../linux-6.14/arch/x86/include/asm/idtentry.h#L248-L272)

```c
#define DEFINE_IDTENTRY_SYSVEC(func)					\
static void __##func(struct pt_regs *regs);				\
									\
static __always_inline void instr_##func(struct pt_regs *regs)		\
{									\
	run_sysvec_on_irqstack_cond(__##func, regs);			\
}									\
									\
__visible noinstr void func(struct pt_regs *regs)			\
{									\
	irqentry_state_t state = irqentry_enter(regs);			\
									\
	kvm_set_cpu_l1tf_flush_l1d();                                   \
	instrumentation_begin();					\
	instr_##func (regs);						\
	instrumentation_end();						\
	irqentry_exit(regs, state);					\
}									\
									\
void fred_##func(struct pt_regs *regs)					\
{									\
	instr_##func (regs);						\
}									\
									\
static noinline void __##func(struct pt_regs *regs)
```

逐段拆开。这个宏是"模板":你写 `DEFINE_IDTENTRY_SYSVEC(sysvec_apic_timer_interrupt)`,它替你展开出**三层**函数:

1. **`func`(如 `sysvec_apic_timer_interrupt`)**——真正的 IDT 入口。它 `__visible noinstr`(对外可见、禁止插桩),干三件事:
   - `irqentry_enter(regs)`:进门登记(进 RCU、记账、`HARDIRQ_OFFSET++`)——**三种惊动共用的骨架**就在这一句。
   - 调 `instr_##func`:转去真正的处理逻辑。
   - `irqentry_exit(regs, state)`:出门,检查软中断、恢复现场。
2. **`instr_##func`**——把真正的处理 `__##func` 放到**专门的硬中断栈**上跑(`run_sysvec_on_irqstack_cond`)。为什么单独换栈?因为硬中断栈独立于进程栈,中断嵌套时不会把进程栈撑爆。
3. **`__##func`**——你真正写的处理逻辑(比如 `__sysvec_apic_timer_interrupt` 调时钟处理)。`static noinline`,强制不内联,保证栈切换干净。

> 这个宏的精妙在于:它把"过门骨架(进出门、换栈、记账)"和"具体业务逻辑"**彻底分离**。你写系统向量处理函数,只要填 `__func` 这一段业务逻辑,通用的骨架由宏自动包好。`noinstr`/`instrumentation_begin/end` 这些是为了让**进出内核这段最敏感的代码不被调试工具插桩**(插桩本身可能引发递归中断/睡眠)。这是"框架替你处理通用难题、你只管业务"的典型工程套路。

举个真实例子,时钟中断的入口就是用它定义的:

> [arch/x86/kernel/apic/apic.c:1049-1059](../linux-6.14/arch/x86/kernel/apic/apic.c#L1049-L1059)

```c
DEFINE_IDTENTRY_SYSVEC(sysvec_apic_timer_interrupt)
{
	struct pt_regs *old_regs = set_irq_regs(regs);

	apic_eoi();                                      /* 应答 Local APIC:这个中断我收到了 */
	trace_local_timer_entry(LOCAL_TIMER_VECTOR);
	local_apic_timer_interrupt();                    /* 真正的时钟处理(下一章主角) */
	trace_local_timer_exit(LOCAL_TIMER_VECTOR);

	set_irq_regs(old_regs);
}
```

`apic_eoi()` 就是回写 EOI——告诉 APIC "这个时钟中断处理完了,可以再发下一个"。这一行体现了第五节讲的"应答硬件":不 EOI,时钟中断不会再来,系统时间停摆。下一章会从 `local_apic_timer_interrupt` 往下钻。

### 精读二:`handle_softirqs`——下半部的"限时重跑 + 兜底"

第六节已经贴过它的核心。这里聚焦它最体现设计的两段:那两道"闸"。

```c
	unsigned long end = jiffies + MAX_SOFTIRQ_TIME;   /* 闸 1:时间上限 */
	int max_restart = MAX_SOFTIRQ_RESTART;            /* 闸 2:重跑次数上限 */
	...
restart:
	set_softirq_pending(0);
	local_irq_enable();                               /* 开中断跑 */
	... while (ffs(pending)) { h->action(); ... }     /* 跑一轮所有挂起的软中断 */
	local_irq_disable();

	pending = local_softirq_pending();                /* 跑完又冒出新的了吗? */
	if (pending) {
		if (time_before(jiffies, end) && !need_resched() && --max_restart)
			goto restart;                         /* 没超时、没人要 CPU:再来一轮 */
		wakeup_softirqd();                            /* 否则:丢给 ksoftirqd 线程兜底 */
	}
```

**不这样会怎样?** 设想去掉这两道闸:软中断处理函数可能又 `raise_softirq` 自己(比如网络 RX 软中断处理时又收到包又 raise)——于是 `goto restart` 永远停不下来,软中断无限霸占 CPU,**用户进程永远得不到执行,系统假死**。

所以内核设了双重保护:

- **`MAX_SOFTIRQ_TIME`**:软中断累计跑了这么久就必须停手。
- **`MAX_SOFTIRQ_RESTART`**:重跑次数上限。

任何一个超了,就 `wakeup_softirqd()`——把剩下的软中断交给 `ksoftirqd` 内核线程。`ksoftirqd` 是个**普通进程**(优先级还比较低),它在进程上下文里跑软中断,**可以被调度、可以让出 CPU 给用户进程**。这就是精妙之处:

> **软中断的"即时性"和"不饿死用户"两种诉求,被拆成了两条路**:少量、及时的,在中断返回时就地跑(快);太多、跑不完的,降级成普通线程慢慢跑(不霸道)。同一个机制,按负载自动在"快"和"公平"之间切换。

## 十、章末小结

用"门铃 + 门卫 + 后厨"的比喻把这一章收口。

这一章我们追着一个网卡中断,走完了它的完整旅程,也看清了"报警器(异常)"和它共享同一条流水线。一句话贯穿:**中断和异常,都是 CPU 被"从用户态瞬间拉到内核态"——硬件靠 IDT 这张门牌总表,一步查到该交给谁。**

完整旅程:

1. **硬件按门铃**:外部设备在中断线上发信号 → APIC 汇总、编号 → 戳 CPU。
2. **查 IDT**:CPU 拿 vector 查门牌总表,一步跳到入口汇编(O(1),纯硬件)。
3. **过门**:入口汇编保存现场(`pt_regs`)、换内核栈、切内核态;通用骨架 `irqentry_enter` 进门登记。
4. **多层分诊**(中断专属):`irq_desc` → `handle_irq` 流控处理器(按硬件触发方式应答)→ `handle_irq_event` → `action->handler`(你的驱动函数)。
5. **上半部铁律**:不能睡眠、不能磨蹭(关中断、无进程上下文)。
6. **推迟到下半部**:中断返回时(`irq_exit` → `invoke_softirq`)就地跑 softirq,跑不完丢给 ksoftirqd;要睡眠的活儿用工作队列丢给内核线程。
7. **异常**走同一条骨架,只是触发者从"硬件信号"变成"CPU 检测到指令异常",且异常关联"当前进程"。

而这一切背后的灵魂,是两个被逼出来的设计:

> **① "分层分诊"是被硬件五花八门的脾气逼出来的。** vector / IDT / `irq_desc` / `handle_irq` / `action` 这一串抽象,每一层解耦一类问题(编号、触发方式、共享、驱动逻辑),和内存篇"分层抽象"一脉相承。
>
> **② "上半部/下半部"是被"硬中断里不能睡眠、不能磨蹭"逼出来的。** 这个硬约束驱动了"扒数据 + 推迟执行"的拆分,以及 softirq / tasklet / 工作队列这三档"按能否睡眠分级"的下半部。

> **回扣全书主线**:这一章是**"内核何时介入"的总钥匙**。进程篇讲了市民怎么跑,但没回答"内核凭什么能插进来"——答案就是中断和异常(以及下一章的系统调用)。没有它俩,调度器得不到时钟心跳、网卡收不到包、按键盘没人理。这一章也是**同步篇的铺垫**:同步篇要讲"多核 + 中断 + 抢占三种并发来源",其中"中断会随时打断你""下半部也在抢占你"这两路,正是本章埋的雷。
>
> - **承前**:它直接消费进程篇的"task_struct / 内核栈"概念(中断要切到内核栈、要记在当前进程账上),也回扣内存篇 [page fault 章](P2-09-page-fault-虚拟到物理的按需兑现.md)(page fault 就是一个典型异常)。
> - **启后**:本章里反复出现的"时钟中断",其实是个特殊到值得单开一章的家伙——它驱动调度、驱动所有定时器。下一章就钻进它。

### 记住这三句话

1. **中断和异常本质都是"陷入内核"**,靠 IDT 一步查表把 CPU 跳到处理函数;区别只在"谁触发(硬件 vs CPU 自动)、异步还是同步"。
2. **硬中断里不能睡眠、不能磨蹭**——这个硬约束逼出"上半部扒数据 + 下半部干费时活",下半部按"能否睡眠"分 softirq / tasklet / 工作队列三档。
3. **中断没有"当前进程"、异常有**——这解释了为什么 page fault 能给当前进程配页、发信号,而硬中断只能推迟到下半部、不能假设在为谁服务。

### 想继续深入,该往哪儿钻

- 顺着调用链通读:[arch/x86/kernel/idt.c](../linux-6.14/arch/x86/kernel/idt.c) 看 IDT 怎么登记;[arch/x86/include/asm/idtentry.h](../linux-6.14/arch/x86/include/asm/idtentry.h) 看入口宏怎么展开骨架;再到 [kernel/irq/chip.c](../linux-6.14/kernel/irq/chip.c) 的 `handle_fasteoi_irq`([687](../linux-6.14/kernel/irq/chip.c#L687))、[kernel/irq/handle.c](../linux-6.14/kernel/irq/handle.c) 的 `__handle_irq_event_percpu`([139](../linux-6.14/kernel/irq/handle.c#L139)) 看分诊;最后 [kernel/softirq.c](../linux-6.14/kernel/softirq.c) 的 `handle_softirqs`([518](../linux-6.14/kernel/softirq.c#L518)) 看下半部。
- 想吃透"上半部/下半部",对比读 `raise_softirq` → `__irq_exit_rcu`([652](../linux-6.14/kernel/softirq.c#L652)) → `handle_softirqs`,再看 `request_threaded_irq`([kernel/irq/manage.c:2141](../linux-6.14/kernel/irq/manage.c#L2141)) 的线程化中断(`thread_fn`)——它是"上半部/下半部"的另一种打包方式。
- 实战观察:`cat /proc/interrupts` 看每个 IRQ 在各 CPU 上的计数(哪个设备最忙、中断是不是集中在某一个核——绑核调优 `smp_affinity`);`cat /proc/softirqs` 看 NET_RX / TIMER / RCU 各类软中断频率;`watch -n1 'grep -E "softirq|Timer" /proc/interrupts'` 观察负载下中断风暴。
