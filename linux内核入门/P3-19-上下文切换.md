# 第 19 章 · 上下文切换:从 A 切到 B 到底换了什么

> **前置**:你需要先读过**第 16 章**(`task_struct` 的 `thread_struct` 字段)、**第 18 章**(`__schedule` → `context_switch`)和**内存篇第 8 章**(页表、TLB)。理由:这一章拆解"切窗口"的三层——寄存器、栈、页表,前两层来自第 16 章的 `thread_struct`,第三层就是内存篇讲的页表与 TLB。

> **核心问题**:第 18 章调度器决定了"从进程 A 切到进程 B"。但这只是个"决定"。真正"把窗口从 A 交给 B",硬件和内核到底动了哪些东西,才让 B 能像"一直在这儿"一样接着跑?切换贵在哪?
>
> **读完本章你会明白**:
> - 一次切换换**三层**:寄存器/CPU 状态、栈指针、地址空间(页表)。
> - 内核线程为什么"借"别人的 `active_mm`(**lazy TLB**)——切页表很贵,能省就省。
> - `switch_to` 为什么是个"诡异的宏":它**不是普通函数调用**——栈指针一换,控制权就跑到另一个栈上了。
> - 切换的真实代价不在搬数据,而在 **TLB/Cache 被打冷**——这才是上下文切换贵在哪。

> **如果一读觉得汇编难懂**:先只记住三件事——① **切换换三层:寄存器(thread_struct)、栈指针(RSP)、页表(switch_mm)**;② **`switch_to` 不是普通函数,它靠改 RSP 把执行流"瞬移"到 B 的栈上**;③ **内核线程借 `active_mm` 实现 lazy TLB,省一次页表重载**。汇编细节第二遍配合源码再抠。

## 章首·"决定切"和"真正切"是两件事

第 18 章末尾,`__schedule` 调用了 `context_switch(rq, prev, next, &rf)`。但要注意:

> **"调度器决定切到 B"和"真正完成这次切换",是两件事。** 前者是 `__schedule` + `pick_next_task` 的逻辑决策,后者是 `context_switch` 要干的**硬件级搬砖**。

这一章我们就钻进 `context_switch`,看"真正切"到底切了什么。

先建立直觉。一个进程在 CPU 上跑,它"占据 CPU"这件事,可以分解成三个层次:

1. **CPU 寄存器**:它的 RAX、RBX、指令指针 RIP、标志位……记着它算到哪了、当前正在算什么。这些是**最当下的状态**。
2. **栈**:函数调用栈,记着"调用了谁、压了哪些局部变量、返回到哪"。RSP 指着栈顶。
3. **地址空间(页表)**:它用的"门牌表"——它的虚拟地址翻译成物理地址靠这本账(CR3 寄存器指着页表根)。

要把 CPU 从 A 交给 B,这三层**每一层都得换**:

- A 的寄存器值存起来,换成 B 的;
- RSP 从 A 的栈顶换成 B 的栈顶;
- CR3 从 A 的页表根换成 B 的(如果 A、B 是不同进程)。

> **回扣比喻**:办事窗口从市民 A 交给市民 B,得做三件事:① 把 A 摊在桌上的草稿纸(RAX 等寄存器)收进 A 的档案夹;② 把窗口的工作台(A 的栈)清出来、换成 B 的工作台;③ 把墙上挂的"门牌对照表"(页表)从 A 的换成 B 的。三件事办完,B 一坐下,就像自己一直在那儿。

这一章按这三层展开,最后讲 lazy TLB 和切换的真实代价。

## 一、总览:`context_switch` 的骨架

先看总入口 `context_switch`([kernel/sched/core.c:5322](../linux-6.14/kernel/sched/core.c#L5322))。它把三层切换组织成一个清晰的流程:

> [kernel/sched/core.c:5322-5383](../linux-6.14/kernel/sched/core.c#L5322-L5383)

```c
context_switch(struct rq *rq, struct task_struct *prev,
	       struct task_struct *next, struct rq_flags *rf)
{
	prepare_task_switch(rq, prev, next);
	...
	arch_start_context_switch(prev);

	/*                            // 切换页表的三种情况:
	 * kernel -> kernel   lazy + transfer active
	 *   user -> kernel   lazy + mmgrab_lazy_tlb() active
	 *
	 * kernel ->   user   switch + mmdrop_lazy_tlb() active
	 *   user ->   user   switch
	 */
	if (!next->mm) {                                // 目标是内核线程
		enter_lazy_tlb(prev->active_mm, next);
		next->active_mm = prev->active_mm;
		if (prev->mm)                           // 来自用户进程
			mmgrab_lazy_tlb(prev->active_mm);
		else
			prev->active_mm = NULL;
	} else {                                        // 目标是用户进程
		membarrier_switch_mm(rq, prev->active_mm, next->mm);
		...
		switch_mm_irqs_off(prev->active_mm, next->mm, next);   // ★ 换页表
		lru_gen_use_mm(next->mm);
		...
	}

	switch_mm_cid(rq, prev, next);
	prepare_lock_switch(rq, next, rf);

	/* Here we just switch the register state and the stack. */
	switch_to(prev, next, prev);                    // ★ 换寄存器 + 换栈
	barrier();

	return finish_task_switch(prev);
}
```

骨架很清楚:**先决定要不要换页表(`if (!next->mm)`),再 `switch_to` 换寄存器和栈。** 注意顺序——**先换页表,后换寄存器/栈**。为什么?因为换页表(`switch_mm_irqs_off`)还在"当前进程的执行上下文"里(CR3 改了,但代码还在跑),而换寄存器/栈(`switch_to`)之后,执行流就"瞬移"到 next 的栈上了,回来时已经是 next。所以必须先做完"还能在 prev 上下文里做的页表切换",再瞬移。

> 还有一个关键细节:**这个函数的"下半截"(`finish_task_switch`)不是由 prev 执行的,而是等以后 prev 重新被切回来时才执行。** 这是 `switch_to` 的"诡异"之处,第三节详谈。

下面逐层展开。

---

## 二、第一层:地址空间——换页表(`switch_mm`)

如果 prev 和 next 是**不同的用户进程**(各有各的 `mm`),它们用的页表不同,必须切换。这由 `switch_mm_irqs_off` 完成([arch/x86/mm/tlb.c:499](../linux-6.14/arch/x86/mm/tlb.c#L499))。

切换页表的核心动作,是改 **CR3 寄存器**——它指向"当前页表的根(PGD)"。把 CR3 从 prev 的 PGD 换成 next 的 PGD,从此 CPU 翻译虚拟地址就用 next 的账本了。

> **回扣内存篇第 8 章**:页表是进程"独占内存幻觉"的台账。两个进程各有各的台账(各有各的 `mm->pgd`)。切 CR3,就是"墙上换挂一本台账"。

### 不这样会怎样:不换页表会出什么事

**不这样会怎样?** 如果切换时不换 CR3:next 一执行,访问虚拟地址 `0x400000`(它的 main 函数所在),CPU 还拿着 prev 的页表翻译——可能翻到 prev 的某个物理页,读到 prev 的数据。**隔离彻底崩塌**:进程能读别人的内存。所以切进程必须切页表。

### 换页表的隐藏代价:TLB 被打冷

换 CR3 本身只是写一个寄存器,一条指令的事。但它带来一个**昂贵的副作用**:**TLB(页表速查缓存)失效了。**

TLB 是 CPU 芯片里缓存"最近翻译过的虚拟→物理地址映射"的硬件(内存篇第 8 章讲过)。prev 刚才把它的热点映射都填进了 TLB,现在一换 CR3,这些映射全是 next 的"错"地址——必须冲掉。冲掉后,next 一上来,**它的每一次访存都要重新走完整的页表查找**(内存篇讲过,多级页表查找可能要 4 次内存访问),慢到爆。

这就是切换"贵"的核心:**不是搬数据贵,是 TLB 被打冷后,next 要经历一阵"冷启动"的访存延迟。** 现代 CPU 用 ASID(地址空间标识)等技术尽量减少 TLB 冲刷(只冲该冲的),但代价依然存在。

## 三、第二层 + 第三层:寄存器和栈——`switch_to` 的"瞬移"

页表换完了,接下来换寄存器和栈。这两件事**绑在一起**做,由 `switch_to` 宏完成。这是整个切换里最"烧脑"的部分。

### `switch_to` 不是普通函数

先看它的定义:

> [arch/x86/include/asm/switch_to.h:49-52](../linux-6.14/arch/x86/include/asm/switch_to.h#L49-L52)

```c
#define switch_to(prev, next, last)					\
do {									\
	((last) = __switch_to_asm((prev), (next)));			\
} while (0)
```

它看着像"调了个函数 `__switch_to_asm`",但这个"函数"干的事,**打破了"函数调用"的全部常识**。`context_switch` 里调用它之后的那行注释一语道破:

> [kernel/sched/core.c:5381](../linux-6.14/kernel/sched/core.c#L5381)

```c
	/* Here we just switch the register state and the stack. */
	switch_to(prev, next, prev);
```

"we just switch the register state and the stack"——**切寄存器 + 切栈**。问题来了:**切栈这一步,让"函数返回"这个概念失效了。**

### 看汇编:`__switch_to_asm` 怎么"瞬移"

> [arch/x86/entry/entry_64.S:177-215](../linux-6.14/arch/x86/entry/entry_64.S#L177-L215)

```asm
SYM_FUNC_START(__switch_to_asm)
	/*
	 * Save callee-saved registers
	 * This must match the order in inactive_task_frame
	 */
	pushq	%rbp
	pushq	%rbx
	pushq	%r12
	pushq	%r13
	pushq	%r14
	pushq	%r15

	/* switch stack */
	movq	%rsp, TASK_threadsp(%rdi)     ; prev 的 RSP 存进 prev->thread.sp
	movq	TASK_threadsp(%rsi), %rsp     ; 从 next->thread.sp 载入新 RSP  ★

	...
	/* restore callee-saved registers */
	popq	%r15
	popq	%r14
	popq	%r13
	popq	%r12
	popq	%rbx
	popq	%rbp

	jmp	__switch_to                    ; 跳到 C 函数收尾
```

逐段拆解这个"瞬移"魔术:

**第 1 步:保存 prev 的寄存器到 prev 的栈。**

`pushq %rbp/%rbx/%r12~%r15`——把这几个 **callee-saved 寄存器**压进**当前栈(还是 prev 的栈)**。注意:**只压这几个,不压全部寄存器**。为什么?这是 x86 函数调用约定的精髓:

> **callee-saved(被调用者保存)寄存器**(`rbp`、`rbx`、`r12~r15`):按约定,函数可以使用这些寄存器,但用完必须恢复原值。所以"切走"时把它们存起来,"切回"时恢复即可。
>
> **caller-saved(调用者保存)寄存器**(`rax`、`rcx`、`rdx` 等等):按约定,调用者自己负责保存。切换发生时,这些寄存器的值在 prev 的栈帧里**已经被 prev 自己(或它的调用链)保存过了**——切换代码不需要再管。

**这就是为什么 `task_struct` 的 `thread_struct` 只存这几个寄存器**——回扣第 16 章"只存该存的,这是优化的精髓"。内核不会傻乎乎地把全部 16 个通用寄存器都存一遍,那纯粹是浪费。

**第 2 步:切换栈指针(魔术的核心)。**

```asm
movq	%rsp, TASK_threadsp(%rdi)     ; 把当前 RSP(prev 的栈顶)存进 prev->thread.sp
movq	TASK_threadsp(%rsi), %rsp     ; 把 next->thread.sp 载入 RSP          ★★★
```

`%rdi` 是第一个参数(prev),`%rsi` 是第二个参数(next)。这两条指令干了件惊天动地的事:

- 第一条:把 prev 当前的栈顶指针 RSP,**存进 prev 的 `task_struct` 的 `thread.sp` 字段**(下次切回 prev 时从这里恢复);
- 第二条:从 next 的 `thread.sp` 载入一个值到 RSP——**这一刻,CPU 的栈从 prev 的栈,瞬间变成了 next 的栈。**

**`thread.sp` 指向哪?** 它指向"next 上次被切走时,压完 callee-saved 寄存器、刚要继续执行的栈顶"。换句话说,它指向 next 的栈里一个**精心准备的栈帧**,这个栈帧的布局就是上面那 6 个 `pushq` 的逆操作——`struct inactive_task_frame`:

> [arch/x86/include/asm/switch_to.h:23-31](../linux-6.14/arch/x86/include/asm/switch_to.h#L23-L31)

```c
struct inactive_task_frame {
#ifdef CONFIG_X86_64
	unsigned long r15;
	unsigned long r14;
	unsigned long r13;
	unsigned long r12;
#else
	unsigned long flags;
```

**第 3 步:从新栈恢复 next 的寄存器。**

栈已经是 next 的了。接下来 `popq %r15~%r12/%rbx/%rbp`——把 next 栈顶那 6 个值弹回寄存器。这些值,正是 next 上次被切走时压进去的。**弹完,next 的 callee-saved 寄存器全部恢复成它被切走时的样子。**

**第 4 步:`jmp __switch_to`(C 函数收尾)。**

跳到 C 函数 `__switch_to`,做一些零碎的架构相关清理:刷新 FS/GS 段寄存器、TLS、FPU 状态、per-cpu 数据等。这些做完,函数"返回"。

### 关键:返回到哪?——这就是"瞬移"的诡异之处

这里是最反直觉的一点。`__switch_to` "返回"时,它**不是返回到 prev 调用它的地方**(因为栈已经换了,返回地址在 next 的栈上),而是**返回到 next 上次"调用了 switch_to 然后被切走"的那个调用点**。

换句话说:

- prev 调 `switch_to(prev, next, prev)` → 执行流跑进 next 的栈;
- 从此 prev 这边"卡住了"(它停在 `switch_to` 调用处),直到将来某天 prev 又被某个进程切回来,执行流才会"从某个 switch_to 调用处返回",继续 prev 的 `context_switch` → `finish_task_switch`。

> **回扣第 16 章的"状态不丢"**:进程被切走时,它算到哪了(RIP)、栈压到哪了(RSP + 栈帧),全存在 `thread.sp` 指向的那个栈帧里。切回来时,从这个栈帧恢复,它就**一丝不差地接着上次的位置往下执行**,就像 `switch_to` 这个调用刚刚才"返回"。**这就是"函数调用"视角下,一个进程能"暂停-恢复"的全部秘密。**

### `last` 参数:为了记住"谁切给了我"

回看宏定义:`((last) = __switch_to_asm((prev), (next)))`。这个 `last`(在 `context_switch` 里就是 `prev`)用来接收一个微妙的值:**"是谁把我切回来的"**。

因为 `__switch_to_asm` 返回时,prev 这个局部变量(切之前的 prev)已经不是"刚才切走我的那个进程"了。`finish_task_switch(prev)` 需要知道"刚才那个让出 CPU 的进程是谁,好给它做收尾(比如如果它要睡觉就唤醒相关等待)"。`last` 就是干这个的——它由 `__switch_to_asm` 通过巧妙的传参实现,让 `finish_task_switch` 拿到正确的"前一个进程"。这个细节不展开,知道"有个 `last` 是为了处理这个错位"即可。

## 四、内核线程的优化:lazy TLB

回到 `context_switch` 开头那个 `if (!next->mm)` 分支。这里藏着一个重要优化——**lazy TLB**。

### 内核线程没有自己的 `mm`

内核线程(比如 kworker、ksoftirqd)在内核态跑,**没有自己的地址空间**——它的 `mm` 是 `NULL`。它不访问用户态内存,只用内核那半地址空间(所有进程共享的内核映射)。

### 不这样会怎样:内核线程也老老实实切页表?

**不这样会怎样?** 如果内核线程切换时也像用户进程那样调 `switch_mm` 换 CR3、刷 TLB:那纯属浪费——内核线程根本不需要"用户态那半页表",它只用内核映射(这部分所有进程都一样)。换了 CR3,TLB 被打冷,等切回用户进程时又得重新填,**白刷两遍 TLB**。

### 所以这样设计:借 `active_mm`,不换页表

`context_switch` 里那段注释把四种情况列得清清楚楚。当目标是内核线程(`!next->mm`)时:

```c
	if (!next->mm) {                                // 目标是内核线程
		enter_lazy_tlb(prev->active_mm, next);  // 标记"懒",不换页表
		next->active_mm = prev->active_mm;      // 借用上一个进程的 mm
		...
	}
```

**不调 `switch_mm_irqs_off`,CR3 不动,TLB 不刷。** 内核线程"借用"上一个用户进程的 `active_mm`(地址空间),继续用那本页表——反正它只用内核映射,借谁的都一样。

> **回扣第 16 章**:当时说 `active_mm` 是"给内核线程用的借用字段,普通进程 `active_mm == mm`"。现在真相大白:普通进程的 `mm` 和 `active_mm` 是同一个;内核线程的 `mm` 是 `NULL`,`active_mm` 是它借来的别人的 `mm`。**`active_mm` 这个字段的存在,就是为了 lazy TLB 这一个优化。**

这就是"懒哲学"在进程篇的一次漂亮回响:**能不换就不换,页表重载和 TLB 冲刷太贵,内核线程能省则省。**

## 五、切换的真实代价:不在搬数据,在冷启动

讲完三层切换,要纠正一个常见误解:很多人以为"上下文切换贵,是因为要存/取那么多寄存器"。

**不全是。** 存取几个 callee-saved 寄存器,加上改 CR3,几十上百个时钟周期就完事了,在现代 CPU 上其实很快。

**切换真正贵在哪?**

1. **TLB 被打冷**(前面讲过):next 上来,它的热点地址映射全没了,每次访存都要走完整页表查找。这是**微秒级**的延迟放大,而且要持续到 TLB 重新预热。这是单次切换最大的直接代价。
2. **CPU Cache 被打冷**:prev 把它的热点数据填满了 L1/L2/L3 cache。next 是另一套工作集,cache 里基本没有它要的数据,**大量 cache miss**,要从内存搬。cache 预热也要时间。
3. **分支预测器状态失效**:CPU 的分支预测器学的是 prev 的分支模式,对 next 不管用,要重新学。

这三者合起来,叫"**冷启动税**"。一次切换后,next 要经历一段"慢热"期,这期间的性能明显低于稳态。**这才是切换的真正代价,而且它不在"切换那一瞬间",而在"切换之后的几百微秒里慢慢还"。**

> **实战印证**:这就是为什么"线程切换比进程切换便宜"——同进程的线程**共享 `mm`**,切换时不换页表、不刷 TLB(只有寄存器和栈要换),冷启动税小得多。这也是为什么调度器不会把时间片设得太短——切太频繁,进程大部分时间都在"还冷启动税",实际干活的时间反而少。

## 六、关键源码精读:`context_switch` 三层切换

把 `context_switch` 作为这一章的核心精读,它把前面所有概念串成一条流程:

> [kernel/sched/core.c:5322-5383](../linux-6.14/kernel/sched/core.c#L5322-L5383)(精简注释版)

```c
context_switch(struct rq *rq, struct task_struct *prev,
	       struct task_struct *next, struct rq_flags *rf)
{
	prepare_task_switch(rq, prev, next);          /* 钩子:统计、记账(收尾在 finish) */
	arch_start_context_switch(prev);

	/* === 第一层:地址空间(页表)=== */
	if (!next->mm) {                               /* 目标是内核线程 → lazy TLB,不换页表 */
		enter_lazy_tlb(prev->active_mm, next);
		next->active_mm = prev->active_mm;     /* 借上一个进程的 mm */
		...
	} else {                                       /* 目标是用户进程 → 换页表 */
		...
		switch_mm_irqs_off(prev->active_mm, next->mm, next);   /* 改 CR3,刷 TLB */
		...
	}
	switch_mm_cid(rq, prev, next);
	prepare_lock_switch(rq, next, rf);

	/* === 第二、三层:寄存器 + 栈 === */
	switch_to(prev, next, prev);                   /* __switch_to_asm:存寄存器、换 RSP、恢复寄存器 */
	barrier();

	return finish_task_switch(prev);               /* 这个"return"其实是 prev 将来被切回时执行的 */
}
```

把这个函数和第二、三节的汇编对照着读,三层切换就彻底通了:

- **第一层(页表)**:用户进程走 `switch_mm_irqs_off`(改 CR3、付 TLB 税);内核线程走 `enter_lazy_tlb`(不换,白嫖)。
- **第二、三层(寄存器+栈)**:`switch_to` → `__switch_to_asm`。**这俩绑在一起**,因为换栈(`movq TASK_threadsp(%rsi), %rsp`)的瞬间,执行流就瞬移了,寄存器必须紧接着从新栈恢复——不能分开。
- **错位的返回**:`finish_task_switch(prev)` 看似紧跟在 `switch_to` 后面,但**它实际是由"将来切回 prev 的那次切换"来执行的**。`barrier()` 防止编译器把这段优化乱。

> **一句话**:`context_switch` 是个"看起来线性、实际上执行流会跳走再跳回"的函数。它的精妙之处,全在 `switch_to` 那个"换栈即瞬移"的汇编魔术里。读懂它,你就读懂了"进程凭什么能暂停-恢复"。

## 七、章末小结

用城市比喻把这一章收口。

这一章我们拆解了"把窗口从市民 A 真正交给市民 B"的全部细节:

1. **切换换三层**:**地址空间(页表,CR3)、寄存器(thread_struct)、栈指针(RSP)**。三层各司其职:页表管隔离,寄存器管"算到哪",栈管"调用到哪"。
2. **`switch_to` 是"瞬移"魔术**:它靠"改 RSP"这一条指令,把执行流从 prev 的栈瞬移到 next 的栈。从此"函数返回"的概念失效——prev 卡在 `switch_to` 调用处,直到将来被切回。这就是进程能"暂停-恢复"的全部秘密。
3. **内核线程用 lazy TLB**:`mm` 是 `NULL`,借 `active_mm`,不换页表、不刷 TLB——"能省则省"。
4. **切换真正贵在冷启动税**:TLB 和 Cache 被打冷,next 要经历一段慢热期。这就是线程切换比进程切换便宜(不换页表)、调度器不把时间片设太短的原因。

> **回扣全书主线**:这一章是进程篇**"分配窗口"那一侧的落地环节**。第 18 章的 `__schedule` 决定"切到谁",这一章的 `context_switch` 负责"怎么切"。**调度决策 + 上下文切换,两者合起来,才完整制造了"独占 CPU"的幻觉。**
>
> 到此,进程篇闭环:进程是什么(16)→ 怎么生(17)→ 谁先用 CPU(18)→ 怎么把 CPU 真正交过去(19)。一个市民从档案建立、出生换芯、排队叫号、到窗口交接,全链路打通。
>
> - **承前**:它消费第 16 章的 `thread_struct`(存恢复寄存器)和 `active_mm`(lazy TLB)、第 18 章的 `__schedule`(它决定调 `context_switch`)、内存篇第 8 章的页表与 TLB;
> - **启后**:`switch_to` 的"瞬移"、切换时持有的 `rq` 自旋锁、切换与中断的关系,都是**第 4 篇(边界:中断/异常/系统调用)**和**第 5 篇(同步)**要深入的主题。比如"为什么中断里不能直接调度""切换时自旋锁怎么和中断配合",都会回到这一章铺的底子。

### 记住这三句话

1. **切换换三层:页表(CR3)、寄存器(thread)、栈(RSP)**——页表管隔离,寄存器管进度,栈管调用链。
2. **`switch_to` 不是普通函数,改 RSP 即瞬移**——进程"暂停-恢复"的秘密全在这一个汇编魔术。
3. **切换贵在 TLB/Cache 冷启动税,不在搬寄存器**——所以内核线程用 lazy TLB 省、线程切换比进程切换便宜、时间片不能太短。
