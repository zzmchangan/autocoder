# 第 3 章(全书第 22 章)· 系统调用:过关的艺术

>
> **核心问题**:市民(进程)不能自己碰公共资源(读文件、建进程、分配内存、写屏幕……),必须"去办事窗口申请"。这扇从用户态到内核态的门,到底是怎么过的——那条改变世界状态的 `syscall` 指令究竟替我们干了什么?为什么必须有这一关,而不是让进程直接执行特权操作?
>
> **读完本章你会明白**:
> - 为什么系统调用是**安全与隔离的根本**(不是性能取舍,是没它整个隔离就崩);

> - 参数怎么传、`SYSCALL_DEFINE` 这套宏如何既生成函数又生成表项、`do_syscall_64` 怎么用 `array_index_nospec` 防 Spectre 式越界读;
> - 为什么不能让用户直接传指针、内核必须用 `copy_from_user`/`copy_to_user`。

> **如果一读觉得太绕**:先只记住三件事——① **系统调用是市民自愿的、唯一的"合法动用公共资源"通道**,没有它隔离和安全全崩;② **一条 `syscall` 指令原子地完成"换世界"**(swapgs+换栈+切CR3+压pt_regs),比老的 `int 0x80` 快;③ **系统调用号查 `sys_call_table`,查表前先 `array_index_nospec` 防越界**。其余汇编细节第二遍配合源码再抠。

## 一、先回答最根本的问题:为什么必须有这扇门

### 市民不能自己动手

进程在自己的房间里(用户态)能干很多事:计算、判断、读写自己的内存。但一旦要碰**公共资源**——读磁盘上的文件、往屏幕写字、建一个新进程、改路由表、绑定一个端口——它就不能自己动手了。它只能**去办事窗口排队申请**,由内核(政府)代劳,并顺便检查"你有没有资格"。

### 不这样会怎样:如果没有系统调用这道墙

CPU 从硬件上把指令分成两类:

- **普通指令**:谁都能执行。加减乘除、读写自己的寄存器和内存。
- **特权指令(privileged)**:只有内核态能执行。读写磁盘控制器的寄存器、改页表(CR3)、关中断(CLI)、配置中断控制器……

CPU 有**运行级别(ring)**:x86 上 ring 0 是内核态(全权)、ring 3 是用户态(受限)。用户态执行特权指令,CPU **当场拒绝**,触发异常(通用保护错误 `#GP`)。

**不这样会怎样?** 假如用户态能直接执行特权指令、能直接读写硬件:

- **隔离崩塌**:任何程序能直接读磁盘上别人的文件、直接读写别的进程的物理内存(绕过虚拟内存的隔离)。
- **权限形同虚设**:普通用户能直接改内核数据结构把自己提成 root。
- **系统随时崩**:一个 bug 程序能直接写坏中断控制器的配置,整台机器死。

所以 CPU 设计上,**特权指令只在 ring 0 能用**。进程想干这些事,唯一的合法出路是:**进入 ring 0,让内核替你做,内核顺便检查权限**。这个"进入 ring 0 让内核代劳"的机制,就是系统调用。

> ⚠️ **系统调用不是性能取舍,是安全与隔离的根本。** 它是"用户态/内核态"这道墙的**唯一合法门**。整本书讲的所有"幻觉"(独占 CPU、独占内存、统一文件世界),都建立在这道墙之上;而墙的通行证,就是系统调用。

### 为什么不用 `int 0x80` 了:从软中断到专用指令

历史上 x86 的系统调用用 `int 0x80`(一条软中断指令,触发 0x80 号向量,查 IDT 走中断那条路)。它有个问题:**慢**。`int` 指令要走完整的"中断过门"流程(查 IDT、保存一大堆、改运行级),开销大。

x86-64 引入了**专用指令 `syscall`**。它不走 IDT,而是靠几个**模型特定寄存器(MSR)**预先登记的入口:

- `MSR_LSTAR`:系统调用入口地址(指向 `entry_SYSCALL_64`)。
- `MSR_STAR`:运行级切换的段信息。
- `MSR_SFMASK`:哪些标志位要清。

CPU 执行 `syscall` 时,**硬件原子地**完成:保存返回地址到 RCX、保存标志到 R11、切到 ring 0、跳到 `MSR_LSTAR` 指向的入口。**比 `int 0x80` 少走一大圈**,这就是为什么现代 Linux 系统调用能做到百纳秒级。

## 二、`syscall` 指令替我们干了什么:精读入口汇编

这是本章最硬的部分。我们逐行读 [arch/x86/entry/entry_64.S](../linux-6.14/arch/x86/entry/entry_64.S) 的 `entry_SYSCALL_64`:

> [arch/x86/entry/entry_64.S:87-122](../linux-6.14/arch/x86/entry/entry_64.S#L87-L122)

```asm
SYM_CODE_START(entry_SYSCALL_64)
	UNWIND_HINT_ENTRY
	ENDBR

	swapgs                                          /* ① 切 GS 基址 */
	/* tss.sp2 is scratch space. */
	movq	%rsp, PER_CPU_VAR(cpu_tss_rw + TSS_sp2)        /* 暂存用户栈指针 */
	SWITCH_TO_KERNEL_CR3 scratch_reg=%rsp                  /* ② 切到内核页表(防 Spectre) */
	movq	PER_CPU_VAR(pcpu_hot + X86_top_of_stack), %rsp  /* ③ 换到内核栈顶 */

	...
	/* Construct struct pt_regs on stack */
	pushq	$__USER_DS				/* pt_regs->ss */
	pushq	PER_CPU_VAR(cpu_tss_rw + TSS_sp2)	/* pt_regs->sp  (刚暂存的用户栈) */
	pushq	%r11					/* pt_regs->flags */
	pushq	$__USER_CS				/* pt_regs->cs */
	pushq	%rcx					/* pt_regs->ip  (syscall 把返回地址存进 RCX) */
	pushq	%rax					/* pt_regs->orig_ax (系统调用号) */

	PUSH_AND_CLEAR_REGS rax=$-ENOSYS                   /* 把其余寄存器全压栈,构造完整 pt_regs */

	/* IRQs are off. */
	movq	%rsp, %rdi                                /* 第一个参数:pt_regs 指针 */
	movslq	%eax, %rsi                                /* 第二个参数:系统调用号(符号扩展) */
	...
	call	do_syscall_64		/* returns with IRQs disabled */   /* ④ 进入 C 代码 */
```

逐段拆:

**① `swapgs`——切 GS 基址**。GS 是个段基址寄存器,内核和用户态各有一个 GS。内核用 per-CPU 数据(每个 CPU 一份的数据,如当前 task 指针、内核栈顶)都通过 GS 访问。`swapgs` 一条指令把"用户态 GS"和"内核态 GS"对换——从此这条指令之后,`PER_CPU_VAR(...)` 访问的就是内核的 per-CPU 数据。这是系统调用能"瞬间找到内核侧上下文"的关键。

**② `SWITCH_TO_KERNEL_CR3`——切页表**。把 CR3 从用户进程的页表换成内核页表。注意:这看似多余(内核空间本就映射在每个进程页表的高半),为什么要换?**为了防 Spectre/Meltdown 类侧信道攻击**。Spectre 出现后,内核默认开启 **KPTI(内核页表隔离)**:用户态运行时,页表里**根本不映射内核空间**,这样投机执行也读不到内核数据。代价就是每次进出内核要切一次 CR3。`syscall` 这里切进去,返回时(`SWITCH_TO_USER_CR3_STACK`)切回来。

**③ 换到内核栈**。`movq PER_CPU_VAR(pcpu_hot + X86_top_of_stack), %rsp`——从 per-CPU 数据里取出"这个核当前线程的内核栈顶",把 RSP 换过去。从此压栈都压在内核栈上。

**④ 构造 `pt_regs`**。接下来一连串 `pushq`,把用户态的现场(ss/sp/cs/ip/flags/orig_ax/通用寄存器)按 `struct pt_regs` 的布局压进内核栈。这一堆压栈的结果,就是第 1 章反复出现的 `pt_regs`。`do_syscall_64` 的第一个参数(`%rdi`=`%rsp`)就是指向它的指针。注意 `orig_ax` 存的是**系统调用号**(用户从 RAX 传进来的)。

> [arch/x86/include/asm/ptrace.h:128-139](../linux-6.14/arch/x86/include/asm/ptrace.h#L128-L139) 里 `orig_ax` 字段的注释说清了它的用途:**入口时它存系统调用号**(对系统调用而言);对普通中断它存的是 vector。这就是为什么 `pt_regs` 能同时服务中断和系统调用——同一个结构,不同语义。

**压栈后 `call do_syscall_64`**——进入 C 代码,真正开始"办事"。注意此时**中断是关的**(`syscall` 进来默认关中断),内核在这里做完敏感的切换后才开。

> 这一整套(swapgs + 切 CR3 + 换栈 + 构造 pt_regs)**全是硬件 + 几十行汇编**在一条 `syscall` 指令触发的入口里完成的,纳秒级。这就是"过门"在系统调用里的极致优化:不查 IDT、不走通用中断框架,专用指令直达。

## 三、`do_syscall_64`:查表找人 + 防越界

进入 C 代码,第一站是 `do_syscall_64`:

> [arch/x86/entry/common.c:76-95](../linux-6.14/arch/x86/entry/common.c#L76-L95)

```c
/* Returns true to return using SYSRET, or false to use IRET */
__visible noinstr bool do_syscall_64(struct pt_regs *regs, int nr)
{
	add_random_kstack_offset();                    /* 给内核栈顶加随机偏移(防栈地址猜测攻击) */
	nr = syscall_enter_from_user_mode(regs, nr);   /* 进门登记:RCU、审计、seccomp 检查 */

	instrumentation_begin();

	if (!do_syscall_x64(regs, nr) && !do_syscall_x32(regs, nr) && nr != -1) {
		/* Invalid system call, but still a system call. */
		regs->ax = __x64_sys_ni_syscall(regs);    /* 无效调用号 → 返回 -ENOSYS */
	}

	instrumentation_end();
	syscall_exit_to_user_mode(regs);               /* 出门:检查信号、是否需要调度 */
	...
}
```

几个要点:

- **`syscall_enter_from_user_mode`**:和第 1 章中断的 `irqentry_enter` 对应——进门的通用登记(进 RCU、审计、**seccomp 过滤**——这是容器/沙箱限制"哪些系统调用能调"的钩子)。
- **`do_syscall_x64`** 是真正"查表"的地方。看它的实现,有一个关键的安全动作:

> [arch/x86/entry/common.c:42-55](../linux-6.14/arch/x86/entry/common.c#L42-L55)

```c
static __always_inline bool do_syscall_x64(struct pt_regs *regs, int nr)
{
	unsigned int unr = nr;

	if (likely(unr < NR_syscalls)) {
		unr = array_index_nospec(unr, NR_syscalls);   /* ★ 防 Spectre 越界读 */
		regs->ax = x64_sys_call(regs, unr);            /* 查 sys_call_table,调对应函数 */
		return true;
	}
	return false;
}
```

**`array_index_nospec(unr, NR_syscalls)`——这一句是 Spectre 攻防的精华。** 它干嘛?

设想没有它:用户传一个**超大**的系统调用号(比如 99999)。`if (unr < NR_syscalls)` 会判断为假,正常路径返回 false。看起来安全。**但 CPU 的投机执行会先乐观地按"可能为真"去执行 `x64_sys_call(regs, unr)`**——用那个超大的 `unr` 去索引 `sys_call_table` 数组,投机地读了一个**越界的内存位置**。虽然事后这条路径会被丢弃,但投机读到的数据可能通过缓存侧信道泄露给用户。

`array_index_nospec` 用一条**屏蔽指令**(`cmp + cmov`/`mask`),让 CPU **在投机执行时也无法越过这个边界**——它把"边界检查"变成了"投机安全"的操作。这是 Spectre V1 之后内核在所有"用户可控数组索引"处的标配防御。

> **不这样会怎样?** 没有这层防护:恶意进程传超大系统调用号,投机执行越过 `sys_call_table` 边界读内核内存,通过侧信道把内核数据偷出来——Spectre 式信息泄露。一行屏蔽指令,堵住了这条路。这是"安全"压在系统调用路径上的硬要求。

`x64_sys_call(regs, unr)` 就是真正的查表——它内部用 `unr` 索引 `sys_call_table` 数组,拿到对应的 `sys_xxx` 函数指针,调用之。返回值写进 `regs->ax`(RAX),这就是系统调用的返回值。

## 四、`sys_call_table` 与 `SYSCALL_DEFINE`:一张表 + 一套宏

### 表是怎么来的

系统调用入口靠一张**函数指针表** `sys_call_table`:

> [arch/x86/entry/syscall_64.c:24-30](../linux-6.14/arch/x86/entry/syscall_64.c#L24-L30)

```c
/*
 * The sys_call_table[] is no longer used for system calls, but
 * ...
 */
const sys_call_ptr_t sys_call_table[] = {
	...
};

long x64_sys_call(const struct pt_regs *regs, unsigned int nr)
{
	...
}
```

> 注意那条注释:`sys_call_table[]` 在现代内核里**已经不直接用于系统调用分发了**(改用 `x64_sys_call` 里一个生成的 switch/数组,更利于优化和安全),但表本身仍在,是"系统调用号 → 处理函数"映射的权威来源。表的每一项按系统调用号顺序排列,内容是各 `sys_xxx` 函数的指针。

### 处理函数怎么写:`SYSCALL_DEFINE`

内核里定义一个系统调用,不用普通的 C 函数,而是用一套宏 `SYSCALL_DEFINEn`:

> [kernel/sys.c:969-972](../linux-6.14/kernel/sys.c#L969-L972) 是最简单的例子:

```c
SYSCALL_DEFINE0(getpid)
{
	return task_tgid_vnr(current);
}
```

`SYSCALL_DEFINE0(getpid)` 看起来像定义了一个叫 `getpid` 的函数,实际上这套宏([include/linux/syscalls.h:217-260](../linux-6.14/include/linux/syscalls.h#L217-L260))展开后做了好几件事:

1. 生成真正的处理函数(名字类似 `__x64_sys_getpid`),签名是 `long __x64_sys_getpid(const struct pt_regs *regs)`——**统一从 `pt_regs` 取参数**,而不是普通 C 的形参。
2. 把它的指针填进 `sys_call_table` 对应位置。
3. 给参数加上类型/边界检查的元数据(配合 `sysfs` 里的系统调用信息)。

**为什么要用宏、为什么参数从 `pt_regs` 取?**

- **统一的入口签名**:`do_syscall_64` 调处理函数时,传的就是 `pt_regs`。不管 `read` 要 3 个参数还是 `getpid` 要 0 个,入口签名都是 `(const struct pt_regs *)`。宏负责"从 pt_regs 的对应寄存器里把参数取出来,转成 C 类型"。这避免了"每个系统调用签名不同、没法用函数指针表"的问题。
- **安全包装**:宏会在外面包一层,做参数的边界检查、地址校验。这是抵御"用户构造畸形参数攻击内核"的一层防线。

`getpid` 之所以是最好懂的例子:它**不要参数、不碰用户指针、几乎没副作用**,就返回当前进程的 PID(`task_tgid_vnr(current)`)。是认识系统调用最干净的起点。

### 系统调用的"号"从哪来

每个系统调用有一个全局唯一的号(在 `include/uapi/asm-generic/unistd.h` / `arch/x86/entry/syscalls/syscall_64.tbl` 里登记)。`read` 是 0 号、`write` 是 1 号、`open` 是 2 号……这些号是**用户态和内核态之间的 ABI 契约**:用户 libc 把"要调 read"翻译成"RAX=0,执行 syscall",内核靠 0 查表找 `sys_read`。号的稳定性比内核内部重构重要——所以号的分配极谨慎,只增不改。

## 五、参数怎么传:为什么不能让用户直接传指针

系统调用要办事,得传参数。`read(fd, buf, count)` 要告诉内核:读哪个文件描述符、读到哪个缓冲区、读多少字节。

### 寄存器传参 + 指针的陷阱

x86-64 约定:**系统调用号放 RAX,最多 6 个参数放 RD RSI RDX R10 R8 R9**。多于 6 个的参数(极少)才用内存。这些都是寄存器传,快。

但参数里常有**指针**——`read` 的 `buf` 是用户空间的一个地址。这里有个要命的陷阱:

> ⚠️ **内核绝不能直接解引用用户传来的指针。**

**不这样会怎样?** 设想内核直接 `memcpy(kdst, buf, count)`,而 `buf` 是用户传的:

- `buf` 可能是个**非法地址**(用户传了个 `0xdeadbeef`),直接解引用触发内核 page fault → Oops/panic。
- `buf` 可能指向**内核地址空间**(用户故意传 `0xffff...` 一个内核地址),直接解引用就**读/写了内核数据**——隔离和安全当场崩溃。这正是很多内核漏洞的根源。
- 用户可能在内核读 `buf` 期间**改它**(TOCTOU):内核刚检查完"这块内存是只读的",用户立刻改成可写,内核拿到的就是不一致的数据。

所以内核提供专门的、安全的用户内存访问函数:

- `copy_from_user(kdst, user_src, n)`:从用户空间**安全地**拷 n 字节到内核。它会检查 `user_src` 是不是合法的用户地址、是不是真的可读,并且不会被投机执行绕过。
- `copy_to_user(user_dst, ksrc, n)`:反向,安全地写回用户。
- `get_user(x, ptr)` / `put_user(x, ptr)`:读写单个简单类型。

> **为什么内存篇 [page fault 章](P2-09-page-fault-虚拟到物理的按需兑现.md) 里 `copy_from_user` 碰到坏指针会触发 `do_kern_addr_fault` 而不是崩?** 正是因为内核访问用户指针时走的是这套"受控"路径:`copy_from_user` 用专门的 `__get_user_asm` 汇编,把可能触发的 page fault 接管成"返回 -EFAULT"而不是 Oops。这就是"内核不直接解引用用户指针"背后的实现机制。

**参数还有别的校验**:文件描述符得是合法的、`count` 不能是负的或离谱大的、flags 不能有非法位……每个 `sys_xxx` 入口都做一遍。这些校验是"用户是不可信输入"这条铁律在系统调用层的落地。

> **回扣比喻**:办事窗口的公务员绝不轻信市民递来的任何材料。市民说"这是我家房产证复印件"(一个用户指针),公务员不会直接拿去归档(直接解引用),而是要**核验真伪、复印留底**(`copy_from_user`)。材料造假(非法地址/越权)当场驳回(`-EFAULT`/`-EINVAL`)。**办事窗口的整个流程,本质就是"把不可信的输入,安全地转成可信的内核操作"。**

## 六、返回:为什么 SYSRET 比 IRET 快

处理函数跑完,返回值在 `regs->ax`。然后 `syscall_exit_to_user_mode(regs)` 做出门准备:检查有没有挂起的信号要处理、要不要重新调度(`TIF_NEED_RESCHED`,回扣第 2 章)、把刚才的现场从 `pt_regs` 恢复。最后回到汇编,切回用户态。

返回用的指令有两条,挑哪条看情况:

- **`sysretq`(SYSRET)**:快路径。它和 `syscall` 是一对,硬件原子地"切回 ring 3、跳到 RCX 保存的返回地址、恢复 R11 的标志"。**比 IRET 快很多**,因为不走完整的"中断返回"框架。
- **`iret`(IRET)**:慢路径,通用。任何不能走 SYSRET 的情况都用它(见汇编里那段判断)。

`do_syscall_64` 末尾有一串检查,决定能不能用快路径 SYSRET:

> [arch/x86/entry/common.c](../linux-6.14/arch/x86/entry/common.c)(紧接 `syscall_exit_to_user_mode` 之后)

```c
	/* SYSRET requires RCX == RIP and R11 == EFLAGS */
	if (unlikely(regs->cx != regs->ip || regs->r11 != regs->flags))
		return false;                        /* 返回 false → 汇编走 IRET 慢路径 */

	/* CS and SS must match the values set in MSR_STAR */
	if (unlikely(regs->cs != __USER_CS || regs->ss != __USER_DS))
		return false;
	...
```

**不这样会怎样?** SYSRET 有个硬件约束:它假设返回地址在 RCX、标志在 R11(因为 `syscall` 就是这么存的)。如果信号处理或别的机制改了这些(比如改了返回地址,或改了标志位),再用 SYSRET 就会跳到错误的地方——**这是一个历史上有过安全漏洞的地方**(SYSRET 到非规范地址触发 #GP 被利用)。所以内核严格检查"RCX==RIP 且 R11==EFLAGS 且 CS/SS 是标准用户段",任何一项不满足,就老老实实走 IRET。**快路径能用就用,不能用绝不冒险。**

汇编侧相应地:

> [arch/x86/entry/entry_64.S:130-145](../linux-6.14/arch/x86/entry/entry_64.S#L130-L145)(节选)

```asm
	ALTERNATIVE "testb %al, %al; jz swapgs_restore_regs_and_return_to_usermode", \
		"jmp swapgs_restore_regs_and_return_to_usermode", X86_FEATURE_XENPV

syscall_return_via_sysret:
	...
	POP_REGS pop_rdi=0                          /* 从 pt_regs 恢复寄存器 */
	...
	SWITCH_TO_USER_CR3_STACK scratch_reg=%rdi   /* 切回用户页表(KPTI) */
	...
	swapgs                                      /* 切回用户态 GS */
	...
	sysretq                                     /* 原子切回 ring 3 */
```

`%al` 就是 `do_syscall_64` 的返回值(true=可走 SYSRET、false=走 IRET)。快路径恢复寄存器、切回用户 CR3、`swapgs`、`sysretq`,一气呵成。

## 七、关键源码精读:`do_syscall_64` 全流程

这一章的精华都浓缩在 `do_syscall_64` 这一个函数里——它把"进门→查表→防越界→办事→出门"全串起来了。我们把它完整读一遍:

> [arch/x86/entry/common.c:76-120](../linux-6.14/arch/x86/entry/common.c#L76-L120)

```c
__visible noinstr bool do_syscall_64(struct pt_regs *regs, int nr)
{
	add_random_kstack_offset();                    /* ① 栈地址随机化(防栈地址猜测) */
	nr = syscall_enter_from_user_mode(regs, nr);   /* ② 进门登记 + seccomp 过滤 */

	instrumentation_begin();

	if (!do_syscall_x64(regs, nr) && !do_syscall_x32(regs, nr) && nr != -1) {
		regs->ax = __x64_sys_ni_syscall(regs);    /* ③ 无效号 → -ENOSYS */
	}

	instrumentation_end();
	syscall_exit_to_user_mode(regs);               /* ④ 出门:信号、调度检查 */
	...
	return true/false;                             /* ⑤ 决定 SYSRET 还是 IRET 返回 */
}
```

逐段对应前面的讲解:

- **① `add_random_kstack_offset`**:每次系统调用,给内核栈顶加一个随机偏移。**不这样会怎样?** 攻击者通过精确测量系统调用耗时,能推测内核栈布局,进而构造栈溢出攻击。随机化让这种侧信道失效。这是"安全"渗透在系统调用最细微处的体现。
- **② `syscall_enter_from_user_mode`**:进门通用骨架(对应中断的 `irqentry_enter`),还含 **seccomp**——容器/沙箱用来限制"这个进程只允许调哪些系统调用"的钩子。一个被 seccomp 限制的进程,在这里被拦下,根本走不到 `do_syscall_x64`。
- **③ 无效调用号**:用户传了非法号,返回 `-ENOSYS`("此功能未实现")。注意还是老老实实返回,不崩——**用户传什么垃圾进来,内核都得优雅拒绝**。
- **④ `syscall_exit_to_user_mode`**:出门。这里检查两件大事:**有没有信号要投递给当前进程**(比如另一个进程给它发了 SIGTERM)、**要不要重新调度**(`TIF_NEED_RESCHED`,第 2 章 `sched_tick` 设的)。如果有,在回用户态前处理掉。这是"系统调用返回点"作为"天然的、安全的内核→用户过渡点"的复用——信号投递、抢占调度,都攒到这个出口一起做。
- **⑤ 返回值决定快慢路径**:能 SYSRET 就 SYSRET,否则 IRET。

> **一句话**:`do_syscall_64` 是系统调用的"海关":栈随机化 + seccomp 是入境安检,`array_index_nospec` + 表分发是"按号办事且防投机越界",`syscall_exit_to_user_mode` 是出境时顺手处理信号和调度。每一个环节都体现了"用户是不可信的、过渡点是稀缺的、安全压倒一切"的设计哲学。

## 八、章末小结

用"办事窗口"的比喻把这一章收口。

[全书导言](全书导言-一篇看懂内核全貌.md)里说,系统调用是市民去政府办事的**办事窗口**。这一章我们跟着一次 `read(fd, buf, count)`,看清了这扇门从敲门到办完回来的全过程。

完整旅程:

1. **用户侧准备**:libc 把系统调用号放进 RAX、参数放进约定寄存器,执行 `syscall`。
2. **`syscall` 指令硬件过门**:`swapgs` 切 GS、`SWITCH_TO_KERNEL_CR3` 切页表(防 Spectre)、换内核栈、压栈构造 `pt_regs`。
3. **`do_syscall_64` 海关**:`add_random_kstack_offset` 栈随机化 → `syscall_enter_from_user_mode`(含 seccomp)→ `do_syscall_x64` 里 `array_index_nospec` 防越界 + 查 `sys_call_table` 调 `sys_xxx`。
4. **处理函数 `sys_xxx`**:用 `copy_from_user`/`copy_to_user` 安全地搬运用户内存,做完事返回值放 `regs->ax`。
5. **`syscall_exit_to_user_mode` 出门**:处理挂起信号、检查 `TIF_NEED_RESCHED` 抢占。
6. **返回**:满足条件走快路径 SYSRET,否则 IRET 慢路径。

而这一章真正要带走的三个设计:

> **① 系统调用是安全与隔离的根本,不是性能取舍。** 用户态/内核态这道墙的**唯一合法门**就是它。没有它,隔离、保护、权限全部形同虚设。每一个"为什么这里要这么繁琐"的设计(栈随机化、seccomp、`array_index_nospec`、`copy_from_user`),根源都是"用户是不可信输入"。
>
> **② 专用 `syscall` 指令把"过门"优化到纳秒级。** swapgs + 切 CR3 + 换栈 + 构造 pt_regs,硬件 + 几十行汇编一气呵成,绕开了通用中断框架。SYSRET 比 IRET 快。这和中断/异常的"通用过门"是同一条流水线的"高性能特化版"。
>
> **③ `SYSCALL_DEFINE` + `sys_call_table` 用"统一签名 + 查表分发"解耦了"调什么"和"怎么调"。** 处理函数统一从 `pt_regs` 取参,号→函数的映射靠表,加系统调用不用动分诊逻辑。

> **回扣全书主线**:这一章完成了本篇"三种惊动"的第三种——**市民自愿的同步陷入**。它和中断(硬件异步)、异常(CPU 自动同步)走的是**同一条"陷入内核"流水线**,区别只在"谁触发、为什么、安全检查严不严"。系统调用因为"自愿、有明确的当前进程、输入不可信",安全检查最重。它也是**全书所有"幻觉"的兑现通道**——进程要兑现任何幻觉(读文件=档案馆、分配内存=房屋局、建进程=户籍科),都得通过这扇窗口。`ls` 那行命令里的每一次 read/write/fork/exec/stat,都是这里讲的一次过关。
>
> - **承前**:它直接消费第 1 章的"过门 5 步"和 `pt_regs`(只是用专用指令优化了),回扣内存篇 `copy_from_user` 的 page fault 处理(为什么访问用户指针安全)。
> - **启后**:本章提到的 `syscall_exit_to_user_mode` 检查 `TIF_NEED_RESCHED`,把系统调用返回点变成了"抢占调度的安全过渡点"之一——这是同步篇"抢占也是并发来源"的伏笔。

### 记住这三句话

1. **系统调用是用户态/内核态墙的唯一合法门,是安全与隔离的根本**——没有它,整个隔离崩塌。
2. **`syscall` 指令 + 入口汇编把"换世界"做到纳秒级**(swapgs/切CR3/换栈/压pt_regs),比老的 `int 0x80` 快;返回优先用 SYSRET。
3. **用户输入不可信**:栈随机化、seccomp、`array_index_nospec` 防 Spectre、`copy_from_user` 拒绝非法指针——每一道都是被攻击逼出来的防线。
