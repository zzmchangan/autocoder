# 第 24 章 · 为什么内核会自己跟自己抢

> **前置**:你需要先读过全书导言(城市比喻)和第 4 篇·边界(中断/异常/系统调用)。本章反复拿"中断随时打断你"说事,不先理解中断,就理解不了"为什么单核也有并发"。

读完这一章你会明白:**内核从来不是一个执行流**。同一个内核,在同一时刻,可能有好几个"自己"在跑;你写的任意一行碰共享数据的代码,都得假设"此刻可能有人正跟我同时摸这份数据"。后续三章(锁、RCU、屏障)全是在治这个病——但开药前,你得先把病看清楚。这一章就是来把病看清楚的。

---

## 0. 一个让人发懵的问题

很多人第一次听到"内核并发"会发懵。理由很充分:

> 内核又不是用户程序那种"我开了 8 个线程"的多线程程序。内核就是**一个**大程序。它跟谁并发?跟自己抢?这不是自相矛盾吗?

这个直觉**错得很典型**。内核的"并发"不是来自"你开了几个线程",而是来自一个更底层的事实:**内核的代码,会被多个独立的"执行流"在多个独立的时机同时执行。** 这些执行流,有些是不同 CPU 核上的,有些是同一个核上被中断/抢占切出来的。

本章只干一件事:把这"多个独立的执行流"到底有哪些、它们怎么会撞到一起、撞在一起会发生什么——**一个一个掰开揉碎讲清楚**。讲清了,你就知道为什么第 25 章非得有锁、第 26 章非得有 RCU、第 27 章非得有内存屏障。

> **如果一读觉得太难**:先只记住三件事——① 内核并发有三个来源:**SMP 多核、中断、抢占**;② 哪怕**单核**也有并发(因为有中断和抢占);③ "**临界区**"不是整段函数,而是**真正摸共享数据的那几行**,锁只锁那几行。其余细节第二遍配合源码再抠。

## 一、并发来源之一:SMP 多核(最直观的那种)

这是最容易理解的一种并发,因为它**物理上**就摆在那儿。

### 1.1 不这样会怎样?

你的机器有 N 个 CPU 核(现在手机都有 8 核,服务器上百核)。这 N 个核**真·物理并行**:它们在同一纳秒、各自执行各自的指令,共享同一套内存。

设想一个全局链表 `my_list`,好多核都往里 `list_add`:

```c
struct list_head my_list = LIST_HEAD_INIT(my_list);

void add_item(struct item *it)
{
    list_add(&it->node, &my_list);
}
```

`list_add` 在源码里看着是"一行",但展开成机器指令后,它要做这几件事:

```
1. 读 my_list.next           (拿到链表头)
2. 让 it->next 指向那个头
3. 让那个头的 prev 指回 it
4. 让 my_list.next 指向 it
5. 让 it->prev 指回 my_list
```

这**五步不是原子的**。现在核 A 和核 B 同时调 `add_item`,把它们各自的 item 都插到链表头:

```
核 A: 步骤1 读到 my_list.next = HEAD
核 B: 步骤1 读到 my_list.next = HEAD   ← 两人读到同一个旧值!
核 A: 步骤4 my_list.next = A
核 B: 步骤4 my_list.next = B           ← B 覆盖了 A
```

结果:`A` 这个 item 被"挤出"了链表——它以为自己插进去了,可 `my_list.next` 指向的是 `B`,`A` 漂浮在内存里没人指它。更糟的情况,两个核的步骤 4/5 交错,可能让 `my_list.next->prev` 指向一个半成品,内核下次遍历链表时 `oops`,指针飞了,**当场 panic**。

> **为什么会乱?** 因为"读链表头 → 改指针 → 写回"这一串不是原子操作,两个核可以**交错(interleave)**执行。只要两个执行流同时进入"改同一份数据"的代码段,就有可能交错出错。

这就是**竞态(race condition)**。它最阴险的地方在于:**99.9% 的时候不发作**(因为交错恰好没撞上),然后**在某次高负载/特定时序下突然炸**。所以内核开发者常说"并发 bug 是最难复现、最难调的 bug,没有之一"。

### 1.2 所以……后面要加锁

治本的办法只有一个:**把 `list_add` 这几步圈起来,一次只准一个核进去。** 别的核想进来,得在外面等。这就是第 25 章要讲的"锁"。

但锁的细节先放一边。本章你要记住的是:**多核,是第一个、也是最直观的并发来源。** 任何被多个核访问的共享数据(全局变量、链表、计数器、页表……),都面临这种"交错"的危险。

## 二、并发来源之二:中断(单核也躲不开的杀手)

这一种是初学者**最容易漏掉**的,也是最阴险的。

### 2.1 "我只有一个核,应该不用锁吧?"

错。即使你的机器只有一个核,并发照样存在,因为还有**中断**。

回忆第 4 篇·边界:CPU 在执行任何一条指令时,**都可能被中断打断**。中断一来,CPU 立刻暂停当前的活,跳去执行中断处理函数(interrupt handler);处理完再回来继续原来的活。

这跟"多核并发"在**效果上一模一样**:

```
你正在执行函数 A,改一份共享数据,改到一半……
   ↑ 时钟中断/网卡中断/磁盘完成中断来了!
   → CPU 跳去执行中断处理函数 handler()
   → handler() 里,恰好也去改这份共享数据!
   → 改完返回,A 继续从中断处往下改
```

虽然物理上只有一个核,**同一时刻只有一段代码在跑**,但**中断处理函数和被中断的代码,交错执行了**。这份共享数据被"两次进入"——一次是被中断的 A,一次是中断里的 handler。这就是并发。

### 2.2 不这样会怎样?

具体例子。设想一个网络统计计数:

```c
int rx_packets = 0;

irqreturn_t nic_irq(int irq, void *dev)   // 网卡中断处理函数
{
    rx_packets++;                          // 收到一个包,计数 +1
    ...
}

ssize_t read_stats(struct file *f, ...)    // 用户态读这个计数
{
    copy_to_user(buf, &rx_packets, ...);
    ...
}
```

`rx_packets++` 是"读-改-写"三步。设想进程正在 `read_stats` 里读 `rx_packets`(已经把值 5 读到寄存器),这时**网卡中断来了**,`nic_irq` 把 `rx_packets` 改成 6。中断返回,`read_stats` 继续……这个例子读操作问题不大(顶多读到旧值),但**反过来**就致命:

设想 `rx_packets` 是个多字段结构体的更新——你刚改了字段 a,还没改字段 b,中断里也改这个结构体。结果这个结构体被读到"a 是新的、b 是旧的"——**半新半旧的不一致状态**。这就是为什么哪怕单核,改共享数据也得防中断。

> **关键认知**:中断把单核"撕"成了事实上的多线程。**单核内核一样有并发问题**,而很多人因为"我就一个核"掉以轻心,写出只在多核高负载下才崩的 bug。这是内核编程与用户态编程最大的心态差异之一。

### 2.3 所以……要么关中断,要么用对锁

治本的思路有两种(第 25 章详述):

- **关中断**:在改这份共享数据的几行里,临时把本地中断关掉(`local_irq_disable` / `spin_lock_irqsave`)。中断关了,这段代码就不会被打断。
- **用 spinlock**:即使是单核,spinlock 也会(在拿锁时)关掉本核的抢占和(必要时)中断,达到同样的"这段代码一次只有一个执行流"的效果。

注意:**中断处理函数里绝不能用 mutex**——因为 mutex 会睡觉,而中断上下文没有"可睡的进程"。这条规矩第 25 章会反复强调。

## 三、并发来源之三:抢占(进程上下文里的"插队")

第三种并发来源,藏在**进程上下文**里,叫**抢占(preemption)**。

### 3.1 抢占是什么?

回忆第 3 篇·进程:调度器为了让每个进程都"觉得独占 CPU",会**飞速轮换**进程。问题是:轮换**什么时候发生**?

- 在**自愿的调度点**(进程主动调 `schedule()`、主动 sleep),这叫**协作式**,没问题。
- 但 Linux 还允许**非自愿抢占**:即使进程没主动让出,只要它的时间片用完、或来了一个更高优先级的进程,内核可以在**很多代码点**强行把它换下去,换另一个进程上来。

这意味着:**两个进程 A、B,跑在同一个核上,A 跑到一半被抢占换成 B,B 也执行访问同一份共享数据的代码——A 和 B 交错执行了。**

### 3.2 不这样会怎样?

设想一个共享的全局变量,进程 A 和进程 B 都会改它(都在**进程上下文**,不在中断):

```c
void update_shared(void)
{
    shared_field_a = 1;     // 改两个字段,本应原子地一起改
    shared_field_b = 2;
}
```

进程 A 调 `update_shared`,刚改完 `shared_field_a`,**时间片到了,被抢占**。调度器换上进程 B,B 也调 `update_shared`,把 `a`、`b` 都改了。等 A 再被调度回来,它继续改 `b`……

结果:这份共享数据经历了一个"`a` 被 A 改、`b` 被 B 改、`a` 又被 B 改、`b` 被 A 改"的混乱序列。任何在这个间隙读它的进程,都可能读到不一致的状态。

> **为什么单独拎出"抢占"?** 因为它和中断不一样:中断是"硬件逼你停",抢占是"**调度器**主动让你停"。抢占**只在进程上下文里发生**(中断上下文不可抢占)。这意味着:你保护数据时,如果只防中断(关中断),**防不住抢占**——因为关中断不关抢占。要防抢占,得用 `preempt_disable()` 或 spinlock(它会顺手关抢占)。

### 3.3 三种并发来源,谁防谁?

把三种来源放在一张表里,你就明白为什么后面 spinlock 有那么多变体(第 25 章):

| 并发来源 | 在什么上下文 | 怎么防 |
|----------|--------------|--------|
| **SMP 多核** | 任意上下文 | 用锁(任一种),让别的核进不来 |
| **中断 / 软中断** | 中断打断进程上下文,或软中断 | 关中断 `local_irq_disable` / 关软中断 `local_bh_disable` |
| **抢占** | 仅进程上下文 | 关抢占 `preempt_disable`(锁默认就关) |

这张表是第 25 章全部锁变体的根源。`spin_lock` 默认关抢占(防抢占);`spin_lock_irqsave` 在关抢占基础上**再关中断**(连中断一起防);`spin_lock_bh` 关的是软中断(下半部)。**内核的原则是:能少关就少关**,因为关掉的力量越多,这个核对系统其他部分的响应就越迟钝。

## 四、把"执行流"和"上下文"两个概念钉死

讲到这儿,有两个术语必须钉死,否则后面三章会看不懂。

### 4.1 执行流(thread of execution)

内核里,**"执行流"**指"一个正在顺序执行代码的东西"。它不一定是"进程":

- 一个**用户进程/线程**在内核态跑,是一个执行流。
- 一个**中断处理函数**在被执行,也是一个执行流(它跟被它打断的进程,是两个不同的执行流,即使跑在同一个核上)。
- 一个**软中断 / tasklet / 工作队列**,也是执行流。
- 一个**内核线程**(kswapd、kworker…),也是执行流。

并发的本质就是:**两个(或更多)执行流,在同一时刻或交错地,执行了访问同一份共享数据的代码。** 不管这两个执行流是"两个核上的进程",还是"一个进程 + 一个中断",在"会撞车"这件事上没区别。

### 4.2 上下文(context):进程上下文 vs 中断上下文

执行流分两大类上下文,这一刀决定了**你能用什么、不能用什么**(尤其能不能睡觉):

| | 进程上下文(task context) | 中断上下文(interrupt context) |
|---|---|---|
| **是什么** | 在执行某个进程/内核线程的代码 | 在执行中断/软中断/NMI 的代码 |
| **有"当前进程"吗** | 有(`current` 指向一个 task_struct) | **没有**(不算谁的账,不关联进程) |
| **能睡觉吗** | **能**(可以 schedule 走、被唤醒回来) | **绝对不能**(睡了没有进程能被叫醒,系统死锁) |
| **能用 mutex 吗** | 能 | **不能** |
| **能用 spinlock 吗** | 能(但临界区要短) | 能(且几乎只能用它) |

内核提供了一个现成的判断宏,见 [include/linux/preempt.h](../linux-6.14/include/linux/preempt.h#L126-L143):

```c
#define in_nmi()		(nmi_count())
#define in_hardirq()		(hardirq_count())
#define in_serving_softirq()	(softirq_count() & SOFTIRQ_OFFSET)
#define in_task()		(...)
#define in_softirq()		(softirq_count())
#define in_interrupt()		(irq_count())
```

这些宏读的是 `preempt_count`(下面会讲)。你随时可以问"我现在在什么上下文",内核会告诉你。很多函数第一行就是 `BUG_ON(in_interrupt())`——它知道自己不能在中断里被调用(因为它会睡觉)。

> **为什么要分这么清?** 因为**"能不能睡觉"是内核里最致命的一条红线**。在中断上下文睡觉 → 没有进程能被唤醒 → 系统死锁 → panic。所以"我现在在什么上下文"直接决定了你能用哪些工具。这是第 25 章选 mutex 还是 spinlock 的根本依据。

## 五、preempt_count:内核追踪"我现在被几层东西挡着"的小账本

上面反复出现 `preempt_count`,这值得专门讲一下,因为它是理解"关中断/关抢占/持有锁"怎么叠加的关键。

### 5.1 一个核一个计数器

每个 CPU 核的当前任务(`task_struct` 的一部分)里,有一个 32 位的计数器,叫 `preempt_count`。它不是简单的"0 或 1",而是**好几段位域叠在一起**,每一段记录一种"挡住抢占"的力量:

```
preempt_count  [大致结构,简化示意,非逐位精确]
┌──────────┬──────────┬──────────┬──────────────┐
│ PREEMPT  │ SOFTIRQ  │ HARDIRQ  │     NMI      │
│  (抢占)  │ (软中断) │ (硬中断) │              │
└──────────┴──────────┴──────────┴──────────────┘
   每加一把锁/关一次中断,对应那一段 +1
   只要整个 count != 0,这个核就不允许被抢占
```

- 你调 `preempt_disable()`:抢占段 +1。见 [include/linux/preempt.h](../linux-6.14/include/linux/preempt.h#L213-L217):

```c
#define preempt_disable()				\
do {							\
	preempt_count_inc();				\
	barrier();					\
} while (0)
```

- 进了硬中断:硬中断段 +1(`__irq_enter` 里 `preempt_count_add(HARDIRQ_OFFSET)`,见 [include/linux/hardirq.h](../linux-6.14/include/linux/hardirq.h#L36))。
- 拿了一把 spinlock:抢占段也会 +1(spinlock 的实现里)。
- 只要 `preempt_count != 0`(或中断被关了),这个核就**不可抢占**(`preemptible()` 为假,见 preempt.h 的定义 `preemptible() (preempt_count() == 0 && !irqs_disabled())`)。

### 5.2 这个小账本回答了两个问题

1. **"我现在能不能被抢占?"** —— 看 `preempt_count` 是不是 0。不是 0 就不能。这就是为什么**拿 spinlock 自动关抢占**:锁把 count +1,你就不会被中途换班,临界区得以完整执行。
2. **"我现在在什么上下文?"** —— `in_hardirq()`/`in_softirq()`/`in_task()` 这些宏,本质上就是看 `preempt_count` 的对应那一段是不是 0。

> **为什么这么设计?** 因为"关中断、关抢占、持锁、在中断里"这几种"挡住调度"的状态会**嵌套**:你可能关了中断、又关了抢占、又拿了锁。用一个计数器(而不是一堆 bool),自然就支持嵌套——退出一层就 -1,全归 0 才解禁。这是内核处理"多层防护"的通用套路。

## 六、什么是"临界区"?(锁到底锁什么)

讲了一堆并发来源,最后把一个最基础的概念钉死:**临界区(critical section)**。

### 6.1 临界区不是整段函数

很多人以为"加了锁就是把整个函数锁起来"。**不是。** 锁只锁**真正摸共享数据的那几行**。

```c
void process_request(struct req *r)
{
    /* ① 这一大段只是算东西、读自己的局部变量,不碰共享数据 —— 不用锁 */
    int x = r->a + r->b;
    do_some_local_work(x);

    /* ② 这几行才是临界区:摸共享链表 */
    spin_lock(&list_lock);
    list_add(&r->node, &shared_list);
    shared_count++;
    spin_unlock(&list_lock);

    /* ③ 后面又是只读局部/不共享的数据 —— 不用锁 */
    log_request(r);
}
```

> **为什么只锁那几行?** 因为锁有代价(尤其自旋锁,别人要忙等)。把不碰共享数据的代码也圈进锁里,等于**让别人白等更久**。优秀的内核代码,临界区**短到极致**——只把非锁不可的几行圈起来,能挪出锁的(比如先在锁外算好、准备好)一律挪出去。这是写高性能并发代码的第一直觉。

### 6.2 临界区里能干什么,取决于你用的什么锁

- 用 **spinlock** 的临界区:**必须短、不能睡觉、不能调用会睡觉的函数**(不能 `kmalloc(GFP_KERNEL)`、不能 `copy_from_user` 在可能缺页时、不能 `mutex_lock`)。
- 用 **mutex** 的临界区:可以长、可以睡觉、可以调用会睡觉的函数——但**必须在进程上下文**(中断里不能用)。

这条规矩的根源就是第 25 章的核心:spinlock 的"忙等"和 mutex 的"睡觉"是两种根本不同的等待方式,各自决定了临界区里允许做什么。

## 七、串起来:三种并发来源 → 三副药

本章的开头说"先把病看清楚"。现在病看清楚了,药方也就呼之欲出——这正好预告后面三章:

| 病(并发来源/问题) | 药(机制) | 哪一章 |
|------|------|------|
| 多核 + 中断 + 抢占,会让"改同一份数据"的代码交错执行 | **锁**:把临界区圈起来,一次一个 | 第 25 章 |
| 读多写少时,读者排队太亏 | **RCU**:读者不锁、读旧版 | 第 26 章 |
| 编译器/CPU 会把"改的顺序"打乱,即使加锁也可能读到半新半旧 | **内存屏障**:强制顺序 | 第 27 章 |

> 注意这三副药**不是互斥**的,而是**对症下药**:同一段数据,写用 spinlock、读用 RCU;spinlock 内部又偷偷编了内存屏障。成熟的内核代码经常**混用**。后面三章会分别讲清每副药的用法、代价和禁忌。

## 关键源码精读:preempt.h 里的"上下文判断"

本章最值得逐行看的,是内核怎么用 `preempt_count` 判断"我在什么上下文"。这段代码虽短,却浓缩了前面所有概念。出自 [include/linux/preempt.h](../linux-6.14/include/linux/preempt.h#L121-L143):

```c
/*
 * Are we doing bottom half work for interrupt?
 * in_nmi()		- We're in NMI context
 * in_hardirq()		- We're in hard IRQ context
 * in_serving_softirq()	- We're in softirq context
 * in_task()		- We're in task context
 */
#define in_nmi()		(nmi_count())
#define in_hardirq()		(hardirq_count())
#define in_serving_softirq()	(softirq_count() & SOFTIRQ_OFFSET)
#define in_task()		(!(preempt_count() & (...)))

/*
 * in_irq()       - Obsolete version of in_hardirq()
 * in_softirq()   - We have BH disabled, or are processing softirqs
 * in_interrupt() - We're in NMI,IRQ,SoftIRQ context or have BH disabled
 */
#define in_softirq()		(softirq_count())
#define in_interrupt()		(irq_count())
```

逐段讲解,对应前面讲的设计:

1. **`in_nmi()` / `in_hardirq()` / `in_serving_softirq()`**:分别读 `preempt_count` 里 NMI、硬中断、软中断那一段。非 0 表示"我此刻正在这种中断里执行"。这一类**全是中断上下文**,不能睡觉、不能用 mutex。
2. **`in_task()`**:全 0,表示"我谁都不是,我就是个普通的进程上下文"——这种地方**可以睡觉、可以用 mutex**。
3. **`in_interrupt()`** = `irq_count()`:只要 NMI/硬中断/软中断任一段非 0 就为真。这是"我现在能不能睡觉"的快速判断——很多函数开头 `BUG_ON(in_interrupt())`,就是在喊"我会在中断里睡死,别在这种地方调我"。
4. **`in_softirq()` vs `in_serving_softirq()` 的细微差别**:`in_softirq()` 为真还包括"只是关了下半部(BH disabled)"的情况(因为 `local_bh_disable` 也会给软中断段加一个标记),而 `in_serving_softirq()` 严格要求"正在执行软中断"。这种细微差别在写并发代码时很重要——它决定了你"防软中断"到底要防到什么程度。

> 这一小段宏,把本章的全部概念串起来了:**执行流的上下文,本质就是 `preempt_count` 各段位域的状态。** 后面你看到任何"为什么这里能用 mutex / 那里不能用",根源都在这几个宏背后的计数器上。

## 章末小结

用本篇的"档案"比喻收一下:

- **内核不是一个人**,而是一座**多个办事窗口同时运作、门铃随时响、办事员还会被换班**的市政府。同一个办事员改一份档案改到一半,可能被另一扇窗口的同事、被门铃催去的中断、被换班上来的接班人,**先后摸上同一份档案**。这就是内核的并发。
- 并发有**三个来源**:**SMP 多核**(物理并行)、**中断**(同一核上的交错)、**抢占**(进程上下文里的换班)。**单核也躲不开**后两种。
- "**临界区**"是真正摸共享数据的那几行,锁只锁那几行,能短就短。
- 一切并发原语,都在回答"**怎么让改同一份数据的代码不交错**",以及权衡"**为了不交错,牺牲多少性能/可睡性**"。

**回扣全书主线**:并发同步属于"管资源"那一侧——它管的是"多个执行流共享同一份数据"这个资源冲突。它和"造幻觉"那侧关系不大,但**幻觉的维护本身处处依赖同步**(比如切换页表要锁、调度要锁)。所以这一篇是全书的"基础设施层"。

**想继续深入该往哪钻**:
- 想看"关中断/关抢占"的真实指令:读 [arch/x86/include/asm/irqflags.h](../linux-6.14/arch/x86/include/asm/irqflags.h) 的 `native_irq_disable`(就是一条 `cli` 指令)。
- 想理解抢占的完整判定:读 [include/linux/preempt.h](../linux-6.14/include/linux/preempt.h) 的 `preemptible()` 和 `preempt_schedule`。
- 想感受"并发 bug 有多难调":去看内核的 **lockdep** 子系统([kernel/locking/lockdep.c](../linux-6.14/kernel/locking/lockdep.c))——它在运行时帮你抓"忘了加锁"和"加错锁"的 bug,正说明这类 bug 有多常见。
