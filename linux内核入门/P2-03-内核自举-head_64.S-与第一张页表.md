# 第 3 章 · 内核自举:head_64.S 与第一张页表

> 物业管理员刚上班,连自己的办公室都还没装修好。这一章讲的是:在"什么都没有"的初始状态下,内核是怎么"拎着自己的鞋带把自己提起来"的。

---

## 章首 · 核心问题

我们的大楼比喻一路推进到第 2 章:BIOS 已经把整栋楼盘点清楚,画了一张满是窟窿的 e820 内存图,然后把内核镜像 `bzImage` 搬进了楼里某个角落,跳过去执行。

于是内核这个"物业管理员"醒了。但请你停下来想一个尴尬的问题:

> **内核自己也是一个程序,它要运行,就需要内存、需要栈、需要一张能正确寻址的页表。可此刻这些东西一样都没有。它凭什么能跑起来?**

更具体地说,这里横着一连串"鸡生蛋"的死结:

- 要跑 C 代码,得有栈;可栈要在内存里,内存要分配就得有分配器,分配器还没建。
- 要让 CPU 在 64 位长模式(long mode)下访问内核代码,得有一张把"内核虚拟地址 ↔ 真实物理地址"对应起来的页表;可这张页表本身也是内存里的数据,谁来建?
- 一张"正确"的页表,地址都得填成最终的内核虚拟地址(类似 `0xffffffff81000000`);可 CPU 此刻还在用早期映射、可能还跑在物理地址上,直接用最终地址会立刻取不到指令。

本章就是来解开这些死结的。**读完本章你会明白:**

- 为什么内核启动的前几千行代码**几乎全是汇编**,只用极少几个 `__head` 标记的 C 函数;
- 为什么需要一张**身份映射(identity map)**——"虚拟地址等于物理地址"的临时映射;
- 早期页表是怎么"一边跑一边修"的:链接时填的假地址,运行时根据真实加载位置逐一修正;
- 为什么在伙伴系统建好之前,内核靠**早期 page fault 自己给自己建页表**来"凭空"变出内存;
- 这一切是怎么为下一章的 memblock、再之后的伙伴系统**铺好第一块砖**的。

一句话概括本章在大楼比喻里的位置:**物业还没正式办公室,先在走廊里搭个临时工棚,用临时材料把"物业办公室"装修到能办公。装修全程用的都是脚手架,等大楼正式运营后再拆掉。**

> **如果一读觉得太难**:先只记住三件事——
> ① **自举期一切内存都靠"编译进镜像的静态预留数组"**(`early_top_pgt` 等),根本没有动态分配,所以开场必须汇编;
> ② **切换到最终页表之前,必须先建"身份映射"(虚拟地址 = 物理地址)**,否则 CPU 在新页表里取不到当前指令,当场死机——这是"为什么不能一步到位"的全部原因;
> ③ **早期 page fault 不报错,反而自动补建页表**——把"提前穷举所有要访问的地址"变成"访问到哪建到哪",这是全书"延迟兑现"哲学的第一次亮相。
>
> 其余细节(load_delta 怎么算、CR4/EFER 怎么配)第二遍配合源码再抠,不影响理解后续章节。

## 一、为什么这段代码这么"狼狈":自举困境

### 不这样会怎样?

设想你是内核的设计者,要在"全裸"状态下启动。你脑子里第一反应可能是:"写个 `main()` 不就行了?"

不行。原因有三个,每一个都是硬约束:

**第一,你还没有栈。** C 函数调用要压栈、局部变量在栈上。可栈是一段内存,这段内存是谁给的?此刻没有任何分配器。CPU 的 `rsp`(栈指针)指向的是 bootloader 留下的、不可靠的一小块。你必须先用汇编把一个像样的栈指针装好,才敢 `call` 任何 C 函数。看 [arch/x86/kernel/head_64.S](../linux-6.14/arch/x86/kernel/head_64.S) 开头:

```asm
startup_64:
    ...
    /* Set up the stack for verify_cpu() */
    leaq    __top_init_kernel_stack(%rip), %rsp
```

([arch/x86/kernel/head_64.S:62](../linux-6.14/arch/x86/kernel/head_64.S#L62))

`__top_init_kernel_stack` 是链接脚本预留的一块静态栈空间——它不是"分配"出来的,而是**编译进内核镜像里的固定区域**。这是第一个关键技巧:**在还没有动态分配之前,一切内存都靠"编译期就钉死好的静态区域"**。脚手架用的不是现砍的木头,是装在内核镜像里自带进来的预制板。

**第二,你不能信任 bootloader 留下的页表。** 注释写得很直白:

```asm
        /*
         * At this point the CPU runs in 64bit mode CS.L = 1 CS.D = 0,
         * and someone has loaded an identity mapped page table
         * for us.
         * ...
         * Since we may be loaded at an address different from what we were
         * compiled to run at we first fixup the physical addresses in our page
         * tables and then reload them.
         */
```

([arch/x86/kernel/head_64.S:40-58](../linux-6.14/arch/x86/kernel/head_64.S#L40-L58))

"someone"(压缩阶段的解压器 `arch/x86/boot/compressed/head_64.S`)确实给 CPU 装了一张身份映射页表,让你能跑进 `startup_64`。但这张表是**别人临时凑的**,地址可能不准、也可能因为 KASLR(内核地址随机化)而整体偏移了。所以内核**第一件事就是自己重做一张页表**,不依赖任何前人。

**第三,你用的所有"地址"都可能是错的。** 内核镜像在编译时,假设自己会运行在一组固定的地址上。但实际加载位置(因为 KASLR)可能完全不同。任何写死的绝对地址都会取错。这逼出了第二个关键技巧:**全程用 RIP 相对寻址(`(%rip)`)**——每条指令只引用"相对于自己当前位置"的偏移,搬到哪里都成立。

### 所以这样设计

这三个约束合起来,决定了自举代码的"长相":

- **必须汇编开场**:装栈指针、装 GDT/IDT、装 CR3(页表基址寄存器),这些是"装零件"的活,C 干不了也用不起。
- **极少量的 C,且必须 `__head`**:少数纯计算的活(比如修页表里的地址),用 C 写更清楚,但必须标记成 `__head`、全程 `RIP_REL_REF()` 访问全局变量——因为普通 C 编译出来的绝对寻址此刻会爆炸。
- **一切内存来自静态预留**:没有 `malloc`,没有伙伴系统,有的只是链接脚本里预留的几个数组(`early_top_pgt`、`init_top_pgt`、`early_dynamic_pgts`……)。

> **比喻**:大楼刚通电,物业要装修办公室。可建材库房还没开、采购员也没到岗。怎么办?物业进场时**自带了一辆装满预制板材的拖车**(编译进镜像的静态数组)。他们用这些自带材料,先搭个能用的工棚,后续装修材料到位后再正式动工。自举代码,就是这辆拖车上的工具。

## 二、身份映射(identity map):为什么"虚拟地址 = 物理地址"

这是本章最难也最关键的一个概念。搞懂它,后面一切顺理成章。

### 先看地址空间的两副面孔

在 64 位 x86 上,内核最终要活在"高位虚拟地址"里。具体说:

```c
#define __START_KERNEL_map	_AC(0xffffffff80000000, UL)
```

([arch/x86/include/asm/page_64_types.h:50](../linux-6.14/arch/x86/include/asm/page_64_types.h#L50))

内核镜像的代码段 `_text`,最终要被映射到 `__START_KERNEL_map + 某个偏移` 这种 `0xffffffff81000000` 一带的高地址上。这就是"内核高位映射"——大楼里物业正式办公室的楼层。

但问题是:**CPU 此刻的指令指针 RIP,指向的是物理地址**(因为 bootloader 装的身份映射把虚拟地址设成了等于物理地址)。你现在的 RIP 大概在 `0x100000` 这种低物理地址上。

### 不这样会怎样?

假设你急于"一步到位",直接把 CR3 装成最终的高位映射页表,然后想跳转到 `common_startup_64` 这个高地址去执行。会发生什么?

1. 你执行 `movq %rax, %cr3`,切换页表。
2. 下一条指令,CPU 拿着当前的 RIP(低物理地址)去查**新页表**——可新页表只映射了高位地址,低物理地址这段在新表里**根本没映射**!
3. 取不到指令,CPU 当场死机。这叫"拉着自己的鞋带想把自己提离地面"——脚还踩在地上就想飞。

这就是文件开头那行著名注释的由来:

```asm
/*
 * We are not able to switch in one step to the final KERNEL ADDRESS SPACE
 * because we need identity-mapped pages.
 */
```

([arch/x86/kernel/head_64.S:31-34](../linux-6.14/arch/x86/kernel/head_64.S#L31-L34))

### 所以这样设计:身份映射

解决办法是搞一张**两边都通的桥**——身份映射。它的含义是:

> **在这张页表里,"虚拟地址 = 物理地址"。** 当 CPU 用物理地址 `0x100000` 当虚拟地址去查表,表里正好把 `0x100000` 指向物理 `0x100000`。于是 CPU 在物理地址上跑得好好的,丝毫感觉不到切换。

有了这张桥,切换流程就成立了:

1. 新页表里**同时**有两组映射:
   - 身份映射:低地址段 `虚拟 = 物理`,让你当前站在的地方依然能取指令;
   - 内核高位映射:把内核镜像也映射到 `0xffffffff81000000` 一带。
2. 你 `mov %cr3` 切到新页表——因为你脚下的物理地址在新表里也有映射,**取指令不中断**。
3. 立刻 `jmp` 到高地址入口 `common_startup_64`——这次表里有高位映射,跳得过去。
4. 跳过去之后,你已经在高地址站稳了,身份映射这一段就可以**拆除**(它只是过渡用的脚手架)。

> **比喻**:物业办公室要从一楼临时工棚搬到 50 楼正式办公室。你不能直接"咻"地瞬移——50 楼还没通电梯,瞬移过去就悬空了。聪明的做法是:先把 1 楼和 50 楼**用一条临时走廊连起来**(身份映射),顺着走廊从 1 楼走到 50 楼,到了 50 楼站稳,再把这条临时走廊拆掉。

代码里这个"走到高地址再拆走廊"的动作非常干净(`①` `②` 为本书所加的讲解标注,源码原文无):

```asm
        /*
         * Switch to early_top_pgt which still has the identity mappings
         * present.
         */
        movq    %rax, %cr3          /* ① 切到新页表(含身份映射) */

        /* Branch to the common startup code at its kernel virtual address */
        jmp     *.Lcommon_startup_64(%rip)   /* ② 跳到高地址 */
```

([arch/x86/kernel/head_64.S:129-137](../linux-6.14/arch/x86/kernel/head_64.S#L129-L137))

注意 `%rax` 里不只是 `early_top_pgt` 的物理地址,还加上了 SME(内存加密)的 mask——那是 AMD SEV 的细节,先忽略,核心是"装新 CR3"。

到了高地址入口 `common_startup_64` 之后(它在 [head_64.S:195](../linux-6.14/arch/x86/kernel/head_64.S#L195)),代码继续做 CR4/EFER 等配置,最后跳进 C 代码。而身份映射的拆除,发生在稍后的 `reset_early_page_tables()`(见第五节)。

## 三、早期页表的两副面孔:身份映射 + 高位映射

现在我们知道了"为什么要有身份映射"。这一节看它**具体长什么样**——也就是内核镜像里自带的那几张静态页表。

### 页表的"原材料"是编译进镜像的空数组

x86_64 的页表是四级(或五级,LA57):PGD → P4D → PUD → PMD → PTE。最顶层的 PGD 是一张 `early_top_pgt`:

```asm
SYM_DATA_START_PTI_ALIGNED(early_top_pgt)
        .fill   511,8,0
        .quad   level3_kernel_pgt - __START_KERNEL_map + _PAGE_TABLE_NOENC
        .fill   PTI_USER_PGD_FILL,8,0
SYM_DATA_END(early_top_pgt)
```

([arch/x86/kernel/head_64.S:604-608](../linux-6.14/arch/x86/kernel/head_64.S#L604-L608))

读这段数据:

- 前 511 项全填 0(`.fill 511,8,0`)——空。
- 第 511 项(最后一项)指向 `level3_kernel_pgt`。这一项就是"内核高位映射"的入口:当虚拟地址在 `0xffffffff80000000` 这一带时,走第 511 个 PGD 项,下钻到 `level3_kernel_pgt`,再一路到 `level2_kernel_pgt`,最终映射整个内核镜像。
- 注意 `level3_kernel_pgt - __START_KERNEL_map`:页表项里存的是**物理地址**,但 `level3_kernel_pgt` 是个虚拟地址符号,所以要减去 `__START_KERNEL_map` 这个偏移,把它换算成"假设加载在 LOAD_PHYSICAL_ADDR 时的物理地址"。这个换算的假设值在运行时还要修正——这就是下一节的重定位。

> **小白补一句**:PGD 项里存的不是"目标虚拟地址",而是**下一级页表的物理地址** + 一组权限位(`_PAGE_TABLE_NOENC` 表示"这是一个指向下一级页表的项,且不做内存加密")。CPU 的 MMU(内存管理单元)就是顺着这些物理地址一层层往下找的。

`level3_kernel_pgt` 再往下:

```asm
SYM_DATA_START_PAGE_ALIGNED(level3_kernel_pgt)
        .fill   L3_START_KERNEL,8,0
        /* (2^48-(2*1024*1024*1024)-((2^39)*511))/(2^30) = 510 */
        .quad   level2_kernel_pgt - __START_KERNEL_map + _KERNPG_TABLE_NOENC
        .quad   level2_fixmap_pgt - __START_KERNEL_map + _PAGE_TABLE_NOENC
SYM_DATA_END(level3_kernel_pgt)
```

([arch/x86/kernel/head_64.S:658-663](../linux-6.14/arch/x86/kernel/head_64.S#L658-L663))

```asm
SYM_DATA_START_PAGE_ALIGNED(level2_kernel_pgt)
        ...
        PMDS(0, __PAGE_KERNEL_LARGE_EXEC, KERNEL_IMAGE_SIZE/PMD_SIZE)
SYM_DATA_END(level2_kernel_pgt)
```

([arch/x86/kernel/head_64.S:665-680](../linux-6.14/arch/x86/kernel/head_64.S#L665-L680))

`PMDS(0, ...)` 是个汇编宏,一口气填出 `KERNEL_IMAGE_SIZE/PMD_SIZE` 个 2MB 大页(PMD)项,把整个内核镜像(最多 1GB,见 [page_64_types.h:96](../linux-6.14/arch/x86/include/asm/page_64_types.h#L96))用 2MB 大页映射好。

> **为什么用 2MB 大页?** 因为此时你还**没有 PTE 那一级页表**——用大页可以省掉最底层那一级,一张 PMD 顶 512 个 PTE,页表更小、更好建、TLB 命中率还更高。早期一切从简,能用大页绝不拆细。

### 身份映射不在静态表里——它是运行时动态建的

注意一个关键的不对称:**内核高位映射是静态、编译期就写好的**(`early_top_pgt[511]` → `level3_kernel_pgt` → ...);但**身份映射不是静态写死的**,因为内核被加载到哪个物理地址要等运行时才知道(KASLR)。所以身份映射在 `__startup_64()` 里**运行时动态拼出来**,拼到 `early_dynamic_pgts` 这组预留空页里:

```c
        /*
         * Set up the identity mapping for the switchover.  These
         * entries should *NOT* have the global bit set!
         */
        pud = &early_pgts[0]->pmd;
        pmd = &early_pgts[1]->pmd;
        ...
        i = (physaddr >> PGDIR_SHIFT) % PTRS_PER_PGD;
        pgd[i + 0] = (pgdval_t)pud + pgtable_flags;
        pgd[i + 1] = (pgdval_t)pud + pgtable_flags;
        ...
        pmd_entry = __PAGE_KERNEL_LARGE_EXEC & ~_PAGE_GLOBAL;
        ...
        pmd_entry += physaddr;

        for (i = 0; i < DIV_ROUND_UP(va_end - va_text, PMD_SIZE); i++) {
            int idx = i + (physaddr >> PMD_SHIFT);
            pmd[idx % PTRS_PER_PMD] = pmd_entry + i * PMD_SIZE;
        }
```

([arch/x86/kernel/head64.c:202-245](../linux-6.14/arch/x86/kernel/head64.c#L202-L245))

读这段逻辑:它取内核实际加载的物理地址 `physaddr`,在 PGD 的对应槽位、PUD、PMD 上各填一条,把 `physaddr` 这一带的物理内存用 2MB 大页映射成"虚拟 = 物理"。映射范围是从 `_text` 到 `_end`,够这段过渡代码用就行。

注释特意强调 **"should *NOT* have the global bit set"**——为什么?因为 `early_pmd_flags` 里清掉了 `_PAGE_GLOBAL`(`head64.c:53`)。全局位(`_PAGE_GLOBAL`)会让这条映射进 TLB 后,即使 CR3 切换也不被刷掉。身份映射是**一次性脚手架**,用完要拆,绝不能赖在 TLB 里不走,所以**故意不带全局位**。这是个非常细节但很能体现"为什么"的点:**每个标志位的选择,都服务于"用完即拆"这个目标。**

## 四、为什么几乎不用 C:RIP 相对寻址与地址重定位

### 不这样会怎样

前面提过:`__startup_64()` 是个 C 函数,但它访问全局变量时**不能用普通的绝对寻址**。普通 C 编译出来访问全局变量,常常生成"取某个固定虚拟地址"的指令。可此刻:

- 内核可能被 KASLR 整体搬到了别处,所有写死的绝对地址全错;
- 更糟的是,这段函数**正在身份映射上运行**,也就是说"现在的虚拟地址"和"最终虚拟地址"还不一样,任何绝对地址更是错上加错。

### 所以这样设计

两个配套的工程手段:

**手段一:全程 RIP 相对寻址。** 在汇编里你已经反复看到 `(%rip)`:

```asm
        leaq    early_top_pgt(%rip), %rcx
```

([arch/x86/kernel/head_64.S:114](../linux-6.14/arch/x86/kernel/head_64.S#L114))

`(%rip)` 的意思是"相对于当前指令指针的偏移"——不管这段代码被搬到哪,这条指令永远能找到 `early_top_pgt`,因为它和当前指令的相对距离不变。这就是为什么启动代码能扛住任意加载位置。

**手段二:C 函数里用 `RIP_REL_REF()` 宏。** 看 `__startup_64` 头上的注释,把这个设计意图写得明明白白:

```c
/* Code in __startup_64() can be relocated during execution, but the compiler
 * doesn't have to generate PC-relative relocations when accessing globals from
 * that function. Clang actually does not generate them, which leads to
 * boot-time crashes. To work around this problem, every global pointer must
 * be accessed using RIP_REL_REF().
 */
```

([arch/x86/kernel/head64.c:139-145](../linux-6.14/arch/x86/kernel/head64.c#L139-L145))

意思是:这段 C 代码可能正跑在"被搬动"的状态下,GCC/Clang 不保证给全局变量生成 RIP 相对重定位,所以**强制**每个全局变量访问都包一层 `RIP_REL_REF()`,显式做成 PC 相对。

**手段三:运行时修正页表里的物理地址。** 这是 `__startup_64` 的核心使命。它先算出"实际加载位置 vs 编译假设位置"的偏移量:

```c
        /*
         * Compute the delta between the address I am compiled to run at
         * and the address I am actually running at.
         */
        load_delta = __START_KERNEL_map + p2v_offset;
        RIP_REL_REF(phys_base) = load_delta;

        /* Is the address not 2M aligned? */
        if (load_delta & ~PMD_MASK)
            for (;;);
```

([arch/x86/kernel/head64.c:167-176](../linux-6.14/arch/x86/kernel/head64.c#L167-L176))

`p2v_offset` 是在汇编里算好传进来的(见 [head_64.S:101-102](../linux-6.14/arch/x86/kernel/head_64.S#L101-L102),用 `common_startup_64` 的运行时地址减去它的链接时地址)。`load_delta` 就是"我实际在哪 - 我以为我在哪"的差值。算完立刻做对齐检查——因为页表用的是 2MB 大页,加载地址必须 2MB 对齐,否则 `for (;;)` 死循环(早期没有 panic 机制,只能死等)。

> **为什么是死循环而不是报错?** 因为此刻连 printk(打印)都还不能用——打印要栈、要缓冲区、要中断,这些都没有。**早期代码出错,唯一能做的就是原地卡死**,让调试者通过"机器没反应"这个现象知道出事了。这也是自举期"极简凑合"哲学的一部分。

然后用 `load_delta` 把页表里所有"链接时假设的物理地址"逐一修正成真实地址:

```c
        /* Fixup the physical addresses in the page table */
        pgd = &RIP_REL_REF(early_top_pgt)->pgd;
        pgd[pgd_index(__START_KERNEL_map)] += load_delta;
        ...
        RIP_REL_REF(level3_kernel_pgt)[PTRS_PER_PUD - 2].pud += load_delta;
        RIP_REL_REF(level3_kernel_pgt)[PTRS_PER_PUD - 1].pud += load_delta;
        ...
```

([arch/x86/kernel/head64.c:184-200](../linux-6.14/arch/x86/kernel/head64.c#L184-L200))

每一句都是"`+= load_delta`"——把页表项里存的物理地址整体平移。修完,这张页表就和真实加载位置吻合了,可以装进 CR3 用了。

> **比喻**:物业从总部拿到了一份"大楼平面图",图上标着"物业办公室在第 5 层"。可这栋楼实际盖出来,因为施工误差,所有楼层号都偏了一层。物业进场第一件事就是**把整张图上的数字统一加 1**,让图和实物对上。`load_delta` 就是这个"统一加的 1"。

## 五、鸡生蛋的真正破解:早期 page fault 自己造页表

到这里,你可能还有一个疑问:早期页表只映射了"内核镜像 + 一小段身份映射",可内核启动过程中要访问的内存远远不止这些(要读 boot_params、要建 per-cpu、要解析设备树……)。**这些地址在页表里没有映射,一访问不就 page fault 了吗?**

会。但内核**故意**把这个 fault 利用起来了。

### 不这样会怎样

如果早期就要访问一段没映射的内存,而你又没有能力处理这个 fault,那 CPU 直接 triple fault 死机。要避免,你就得**提前**把所有可能访问的地址都映射好——可你根本不知道启动过程要访问哪些地址,这是个无底洞。

### 所以这样设计:让 page fault 当"按需泥瓦匠"

内核给早期阶段装了一个**极简的 page fault 处理器**。当访问到一个还没映射的地址时,它不报错,而是**当场补建一条映射**:

```c
void __init do_early_exception(struct pt_regs *regs, int trapnr)
{
    if (trapnr == X86_TRAP_PF &&
        early_make_pgtable(native_read_cr2()))
        return;
    ...
}
```

([arch/x86/kernel/head64.c:370-384](../linux-6.14/arch/x86/kernel/head64.c#L370-L384))

`native_read_cr2()` 读出 CR2 寄存器——CPU 在发生 page fault 时,会把"出错的虚拟地址"写进 CR2。然后 `early_make_pgtable()` 拿这个地址,**顺着 PGD/P4D/PUD/PMD 一路建下来**,把它映射到对应的物理地址:

```c
static bool __init early_make_pgtable(unsigned long address)
{
    unsigned long physaddr = address - __PAGE_OFFSET;
    pmdval_t pmd;

    pmd = (physaddr & PMD_MASK) + early_pmd_flags;

    return __early_make_pgtable(address, pmd);
}
```

([arch/x86/kernel/head64.c:360-368](../linux-6.14/arch/x86/kernel/head64.c#L360-L368))

注意它用的转换:`physaddr = address - __PAGE_OFFSET`。`__PAGE_OFFSET` 是直接映射区(direct map)的起点(`0xffff888000000000`,见 [page_64_types.h:42](../linux-6.14/arch/x86/include/asm/page_64_types.h#L42))。也就是说,**早期内核假设"虚拟地址 `__PAGE_OFFSET + 物理地址` 永远等于那个物理地址"**——这就是后来大名鼎鼎的"直接映射区"的雏形。任何落在直接映射区、却还没建表的地址,fault 一次就自动补一条 PMD(2MB)映射。

`__early_make_pgtable`([head64.c:290-358](../linux-6.14/arch/x86/kernel/head64.c#L290-L358))的实现也是"自举风":它从 `early_dynamic_pgts[]` 这个预留数组里**逐个取空页**当下级页表,用完 64 个就 `reset_early_page_tables()` 全部清零重来。没有分配器,就用"一个固定数组 + 一个递增下标 `next_early_pgt`"硬扛。

> **这个设计的深意**:`do_early_exception` 是**全书"延迟兑现"哲学的第一个实例**。还记得导言里说的吗——"page fault 不是错误,而是兑现时机"。这里更激进:**连页表本身都懒得提前建好,而是访问到哪、建到哪**。用一个极小的 fault 处理器,把"穷举所有可能访问的地址"这件不可能的事,变成了"谁访问谁负责触发"。这就是用延迟换简单。

> **比喻**:物业刚开工,既没人力也没图纸去把整栋楼每间房都登记好。于是立个规矩——**谁要进哪间房,门没登记的,现场补登记**。门卫(fault 处理器)手里只管一个空登记本(`early_dynamic_pgts`),来一个补一个。等正式物业系统上线,这本临时登记本一扔,换正式台账。

## 六、拆除脚手架:身份映射的功成身退

身份映射是一把梯子,踩着它爬上了高地址之后,就该拆掉了。为什么必须拆?

### 不这样会怎样

- 身份映射让"虚拟地址 = 物理地址"那段低地址可被访问,这本身是个**安全隐患**(用户态理论上有途径摸到物理内存的某些映射);而且
- 它占着 `early_dynamic_pgts` 等宝贵的早期页表资源。

更重要的是:**内核一旦进入真正的 C 世界(`x86_64_start_kernel` 之后),就要换成"正式"的 `init_top_pgt` 页表**,身份映射这套临时班子没用了。

### 所以这样设计

拆除动作在进入 C 代码后立刻发生:

```c
asmlinkage __visible void __init __noreturn x86_64_start_kernel(char * real_mode_data)
{
    ...
    cr4_init_shadow();

    /* Kill off the identity-map trampoline */
    reset_early_page_tables();

    clear_bss();
    ...
}
```

([arch/x86/kernel/head64.c:433-454](../linux-6.14/arch/x86/kernel/head64.c#L433-L454))

`reset_early_page_tables()` 干的事:

```c
/* Wipe all early page tables except for the kernel symbol map */
static void __init reset_early_page_tables(void)
{
    memset(early_top_pgt, 0, sizeof(pgd_t)*(PTRS_PER_PGD-1));
    next_early_pgt = 0;
    write_cr3(__sme_pa_nodebug(early_top_pgt));
}
```

([arch/x86/kernel/head64.c:282-287](../linux-6.14/arch/x86/kernel/head64.c#L282-L287))

注释 "except for the kernel symbol map" 一针见血:**它把 `early_top_pgt` 的前 511 项全清零(身份映射、动态建的映射全没了),唯独保留第 511 项**——也就是那条"内核高位映射"。清完立刻 `write_cr3` 重新装载。从此身份映射那座桥拆了,CPU 已经稳稳站在高地址上,不再需要它。

随后,正式页表 `init_top_pgt` 接管:

```c
    /* set init_top_pgt kernel high mapping*/
    init_top_pgt[511] = early_top_pgt[511];
```

([arch/x86/kernel/head64.c:493-494](../linux-6.14/arch/x86/kernel/head64.c#L493-L494))

把 `early_top_pgt[511]`(那条高位映射)搬到 `init_top_pgt[511]`,正式页表继承了"内核高位映射"这条命脉。从此往后,`init_top_pgt` 就是内核的运行时页表(它最终会被 `paging_init()` 填满成完整的直接映射区)。

最后,`x86_64_start_reservations()` 调用 `start_kernel()`:

```c
void __init __noreturn x86_64_start_reservations(char *real_mode_data)
{
    ...
    start_kernel();
}
```

([arch/x86/kernel/head64.c:499-516](../linux-6.14/arch/x86/kernel/head64.c#L499-L516))

`start_kernel()`([init/main.c:896](../linux-6.14/init/main.c#L896))——整个内核的"大主循环"——由此启动。在那里,`setup_arch()` 会解析 e820、初始化 memblock,然后才轮到伙伴系统登场。**自举到此结束,临时工棚拆除,正式物业管理开张。**

> **下一章的伏笔**:可是 `start_kernel()` 里还要分配一大堆内存(建 per-cpu、建页表、解析设备树……)——而伙伴系统此刻还没建好!谁来供这些内存?这就是第 4 章 memblock 要解决的问题:一个**极简的临时分配员**,接替自举期的"静态预留数组",熬到伙伴系统就位。

## 关键源码精读:`__startup_64` 与 `x86_64_start_kernel`

这一章最核心的两个函数,我们把它们的"骨架逻辑"对应回前面的设计。

### 1. `__startup_64()` —— 修页表 + 建身份映射

入口在汇编里:

```asm
        movq    %r15, %rsi          /* boot_params 作为第二个参数 */
        call    __startup_64

        /* Form the CR3 value being sure to include the CR3 modifier */
        leaq    early_top_pgt(%rip), %rcx
        addq    %rcx, %rax
        ...
        movq    %rax, %cr3          /* 装新页表 */
        jmp     *.Lcommon_startup_64(%rip)   /* 跳高地址 */
```

([arch/x86/kernel/head_64.S:110-137](../linux-6.14/arch/x86/kernel/head_64.S#L110-L137))

`__startup_64` 的返回值 `%rax` 是"要写进 CR3 的值"(`early_top_pgt` 物理地址 + 可能的 SME mask)。它的内部职责清晰分两块:

**职责 A:重定位(修页表里的物理地址)**

```c
load_delta = __START_KERNEL_map + p2v_offset;
...
pgd[pgd_index(__START_KERNEL_map)] += load_delta;
RIP_REL_REF(level3_kernel_pgt)[PTRS_PER_PUD - 2].pud += load_delta;
RIP_REL_REF(level3_kernel_pgt)[PTRS_PER_PUD - 1].pud += load_delta;
```

([head64.c:171-197](../linux-6.14/arch/x86/kernel/head64.c#L171))——把高位映射链路上每一项存的物理地址,整体 `+= load_delta`。

**职责 B:动态建身份映射**(见第三节引用的 [head64.c:202-245](../linux-6.14/arch/x86/kernel/head64.c#L202-L245)):用 `physaddr` 在 `early_dynamic_pgts` 里拼出一条"虚拟=物理"的 2MB 大页映射,带 `_PAGE_GLOBAL` 清除。

**职责 C:收紧高位映射**(很容易被忽略,但很关键):

```c
        /* invalidate pages before the kernel image */
        for (i = 0; i < pmd_index(va_text); i++)
            pmd[i] &= ~_PAGE_PRESENT;

        /* fixup pages that are part of the kernel image */
        for (; i <= pmd_index(va_end); i++)
            if (pmd[i] & _PAGE_PRESENT)
                pmd[i] += load_delta;

        /* invalidate pages after the kernel image */
        for (; i < PTRS_PER_PMD; i++)
            pmd[i] &= ~_PAGE_PRESENT;
```

([arch/x86/kernel/head64.c:265-276](../linux-6.14/arch/x86/kernel/head64.c#L265-L276))

`level2_kernel_pgt` 在编译时填了 `KERNEL_IMAGE_SIZE/PMD_SIZE` 个大页(最多 512 个,即整整 1GB),但内核镜像实际只占其中一小段。这三段循环把"镜像之前"和"镜像之后"的 PMD 项**清掉 `_PAGE_PRESENT` 位**(=取消映射),只保留真正覆盖 `_text`~`_end` 的项。注释解释了原因:只确认过镜像占用的区域是可用内存,没确认过的区域如果留着映射,CPU 的**推测执行**可能溜进去,在某些硬件(UV 平台)上会被 BIOS 当错误直接拉停。

> **这个细节再次体现"为什么优先"**:连"哪些项该留、哪些该清"都不是随手写的——它服务于"避免推测执行访问未验证区域"这个安全/稳定性目标。每一个 `~_PAGE_PRESENT` 都有来历。

### 2. `x86_64_start_kernel()` —— 拆脚手架,交棒给 start_kernel

这是自举期的"收尾函数"。它做的事按顺序看,每一步都对应一个"拆/装"动作(**简化示意**:省略了 `BUILD_BUG_ON` 自检的其余几行与 `tdx_early_init`、`load_ucode_bsp` 等,`①`~`⑨` 为本书所加的讲解标注,源码原文无):

```c
asmlinkage __visible void __init __noreturn x86_64_start_kernel(char * real_mode_data)
{
    /* 编译期地址布局自检 */
    BUILD_BUG_ON(MODULES_VADDR < __START_KERNEL_map);
    ...

    cr4_init_shadow();

    /* ① 拆身份映射 */
    reset_early_page_tables();

    /* ② 清 bss */
    clear_bss();

    /* ③ 清 init_top_pgt(为正式页表做准备) */
    clear_page(init_top_pgt);

    sme_early_init();
    kasan_early_init();         /* ④ KASAN 影子内存,需要先有页表 */

    __native_tlb_flush_global(this_cpu_read(cpu_tlbstate.cr4)); /* ⑤ 刷掉 trampoline 残留 TLB */

    idt_setup_early_handler();  /* ⑥ 装早期 IDT,page fault 能被接住了 */
    ...

    copy_bootdata(__va(real_mode_data));   /* ⑦ 把 boot_params 搬进来 */

    /* ⑧ 把高位映射搬到正式页表 */
    init_top_pgt[511] = early_top_pgt[511];

    x86_64_start_reservations(real_mode_data);  /* ⑨ → start_kernel() */
}
```

([arch/x86/kernel/head64.c:433-497](../linux-6.14/arch/x86/kernel/head64.c#L433-L497))

读这条流水线,你能看到自举期的**"小心翼翼的交接"**:

- ①拆身份映射——但拆之前 CPU 已在高地址,所以安全;
- ⑥装早期 IDT——这一步**让第五节那个 `do_early_exception` 真正生效**,从此任何没映射的访问都能被接住、自动补建;
- ⑦`copy_bootdata` 里要访问的内存,就是靠这个早期 fault 机制兜底的;
- ⑧把唯一要保留的高位映射搬到正式页表 `init_top_pgt`,自举页表的历史使命完成;
- ⑨跳进 `start_kernel()`,世界交给"正式物业"。

注意 `__va(real_mode_data)`([head64.c:486](../linux-6.14/arch/x86/kernel/head64.c#L486))——`__va` 把物理地址换算成直接映射区的虚拟地址。这一步能成立,正是因为早期 fault 机制会把直接映射区里访问到的地址自动建好映射。**鸡生蛋问题,就这样被"按需 fault"漂亮地解开了。**

## 章末小结

回到大楼比喻,把这一章串起来:

> 大楼刚通电,物业进场,既没有办公室也没有建材库。他们靠**自带的一辆预制板拖车**(编译进内核镜像的 `early_top_pgt`/`init_top_pgt`/`early_dynamic_pgts` 等静态页表数组),先在走廊里搭个临时工棚(身份映射),让自己能站住脚。可拖车上的图纸是按"总部标准楼层"画的,和实际大楼有偏差,所以他们第一件事是**把图纸统一修正**(`load_delta` 重定位)。站稳之后,他们顺着工棚走到正式办公室楼层(切换 CR3 + 跳高地址),到了之后**立刻把工棚拆掉**(`reset_early_page_tables`)。整个过程中,谁要进还没登记的房间,**门卫现场补登记**(早期 page fault 建页表),用一本小登记本(`early_dynamic_pgts`)硬扛。最后,把唯一值得保留的那条"物业办公室入口"搬到正式台账(`init_top_pgt[511]`)上,临时班子功成身退,正式物业管理开张(`start_kernel()`)。

### 本章在全书主线中的位置

记住导言的二分法:**物理一侧管"切房间",虚拟一侧管"造幻觉"**。本章服务的是哪一侧?

答案是:**两边都是,又都不是**——本章是**为两侧铺设地基**。它造的第一张页表,既是"物理房间能被正确访问"的前提(物理侧的脚手架),又是"虚拟地址幻觉"的第一块砖(虚拟侧的雏形)。具体说:

- 它建立的**身份映射**和**早期直接映射区 fault**,是物理侧"能访问到具体房间"的先决条件——没有它,后面 memblock、伙伴系统都无从谈起;
- 它建立的**内核高位映射**(`early_top_pgt[511]` → `init_top_pgt[511]`),是虚拟侧"内核自己活在高位虚拟地址"这个幻象的起点——第 8 章讲多级页表时,你会再见到这张表的完整版。

### 五个"为什么"清单

如果你只能记住五件事,记这五件:

1. **为什么开场必须用汇编**:没有栈、没有可信页表、地址可能是错的,只能用"装零件"的汇编把基础设施搭起来。
2. **为什么需要身份映射**:切换页表时,CPU 必须在新旧页表里都能取到指令,否则立刻死机。身份映射就是那座"两边都通"的过渡桥。
3. **为什么用 2MB 大页**:早期没有 PTE 那级页表,大页省表省事、TLB 还友好;一切从简。
4. **为什么早期 page fault 不报错反而建表**:把"穷举所有要访问的地址"这个无解问题,变成"访问到哪、建到哪"的按需兑现——这是全书"延迟兑现"哲学的第一次亮相。
5. **为什么身份映射用完必拆、且故意不带全局位**:它是脚手架,带全局位会赖在 TLB 里造成安全/资源问题;每一个标志位选择都服务于"用完即拆"。
