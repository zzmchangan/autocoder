# 第 18 章 · 调度器:谁此刻用窗口

> **前置**:你需要先读过**第 16、17 章**(进程是什么、`fork`/`exec`)。理由:调度器调度的对象就是 `task_struct`(具体是它的 `sched_entity`)。这一章回答"几千个进程造出来之后,谁先用 CPU"。

> **核心问题**:城里几千个市民(进程),办事窗口(CPU)只有几个。内核凭什么决定"此刻叫谁上窗口"?为什么"公平"这件事这么难,Linux 又是怎么用 CFS 逼近它的?
>
> **读完本章你会明白**:
> - 调度的本质:**分配 CPU = 分配时间片**,而"独占 CPU"的幻觉全靠极快轮换实现。
> - 为什么是"调度类"(`sched_class`)的**阶级制**,而不是"一张大名单"——stop > deadline > rt > fair > idle。
> - CFS(完全公平调度器)的核心 **vruntime**:把"实际跑了多久"换算成"虚拟跑了多久",永远挑 vruntime 最小的跑,从而逼近"按权重公平"。
> - 调度什么时候发生:时钟中断、主动让出、唤醒抢占了别人。

> **如果一读觉得 CFS 的算法绕**:先只记住三件事——① **CPU 是时间,调度就是分时间片**;② **CFS 给每个进程一个 `vruntime`(虚拟已运行时间),永远挑最小的那个跑**;优先级高的进程 `vruntime` 涨得慢,于是能多占;③ **`__schedule` 是调度的总入口,`pick_next_task` 负责选下一个**。EEVDF 的细节(本章末会提)第二遍再抠。

## 章首·"独占 CPU"是个彻头彻尾的幻觉

每个写程序的人都默认一件事:**"我的程序在 CPU 上跑着"。** 你写 `while(1) i++;`,你以为它一刻不停在算。

**真相是:你的程序从没一刻不停过。** 在一个跑着几百个进程的系统上,你的 `while(1)` 每隔几毫秒就会被调度器**请下 CPU**,换别的进程上去跑;几毫秒后又被**请回来**。这个轮换每秒发生上千次,**快到你的程序和你的肉眼都察觉不到**——于是你产生了"我独占了 CPU"的幻觉。

> **回扣全书导言**:这是内核造的**第一个、也是最基础的幻觉:独占 CPU**。内存篇造的"独占内存"幻觉,需要"进程"来承载;而"独占 CPU"的幻觉,**全靠调度器一秒上千次的飞速轮换来维持**。

这一章的核心问题就一个:**调度器凭什么决定"此刻叫谁"?** 它要同时满足几个互相打架的目标:

- **公平**:不能让一个进程霸占 CPU,别的饿死;
- **响应快**:交互进程(你敲键盘的那个终端)要立刻响应,不能等;
- **吞吐高**:CPU 密集型任务(编译、渲染)要把 CPU 用满,别在切换上浪费;
- **还要支持实时**:有些任务有死线(deadline),错过就出事。

"公平"尤其难——后面你会看到,Linux 为此发明了 CFS( Completely Fair Scheduler,完全公平调度器),用一个叫 **vruntime** 的量,优雅地逼近了"按权重公平"。

## 一、调度的本质:CPU 是时间

先建立一个最根本的认知:

> **CPU 不是"一个东西",而是"一段段时间流"。调度 CPU,本质是分配"谁能占用哪一段时间"。**

CPU 一秒能执行几十亿条指令,这一秒可以切成上千个"时间片"(time slice),每个几毫秒。调度器做的事,就是决定**每个时间片归谁**。

> **回扣比喻**(导言第五步):办事大厅几千人排队,窗口只有几个。叫号器(调度器)给每个人发一个"办事时长"(时间片),时间到了就换下一个人。轮换快到每个人都没察觉自己被换过——于是每个人都觉得"窗口是我一个人的"。

时间片**不能太长**(一个进程霸着 CPU,别人饿死、交互卡顿),也**不能太短**(切窗口本身有代价,见第 19 章,切太频繁 CPU 全浪费在切换上)。CFS 会根据进程数量和优先级动态算出"合理的"时间片——后面讲。

## 二、调度类:为什么是"阶级"不是"大名单"

几千个进程,需求天差地别:有的要实时(数控机床的控制程序,错过 1ms 就撞车),有的要交互(你的浏览器,点一下要立刻动),有的就是 CPU 密集(编译内核,恨不得占满所有核)。

**不这样会怎样:把所有人塞进一个大名单公平调度?** 那实时任务(必须 1ms 内响应)和编译任务(晚点无所谓)被一视同仁——实时任务可能因为排在编译任务后面而**错过死线**,后果可能是撞车、设备损坏。实时和普通任务,**根本不该在同一个竞技场比公平**。

**所以这样设计:阶级制。** Linux 把进程分成几个**调度类**(`sched_class`),阶级是**严格优先**的:

| 调度类 | 比喻 | 里面是谁 | 调度策略 |
|--------|------|----------|----------|
| **stop** | 城管/特勤,随时插队 | 内核内部用(如迁移任务),最高 | — |
| **deadline** | 预约了死时间的 VIP | 有明确截止时间的实时任务 | EDF(最早死线优先) |
| **realtime(rt)** | VIP 通道 | 实时进程(SCHED_FIFO/RR) | 优先级抢占 |
| **fair(CFS)** | **普通通道(绝大多数)** | **普通进程** | **CFS(vruntime 公平)** |
| **ext(sched_ext)** | 可插拔(用 eBPF 自定义) | 实验性自定义调度器 | 用户定义 |
| **idle** | 没人办事时,清洁工上场 | 空闲进程(每个 CPU 一个) | — |

**阶级的规则极简单:只要高阶有进程可跑,低阶就排不上。** 只要 VIP 通道(deadline/rt)有人,普通通道(fair)就一个也轮不上。绝大多数普通进程(你的浏览器、shell、编译器)都在 fair 通道,互相之间用 CFS 讲公平。

### 调度类是个"函数表"

调度类不是一个简单的优先级数字,而是一个**函数表**——每种调度类实现自己的一套"怎么选下一个、怎么入队、怎么出队":

> [kernel/sched/sched.h:2411-2440](../linux-6.14/kernel/sched/sched.h#L2411-L2440)

```c
struct sched_class {
	...
	void (*enqueue_task)(struct rq *rq, struct task_struct *p, int flags);
	bool (*dequeue_task)(struct rq *rq, struct task_struct *p, int flags);
	void (*yield_task)(struct rq *rq);
	...
	struct task_struct *(*pick_task)(struct rq *rq);          /* 选下一个 */
	...
	void (*put_prev_task)(struct rq *rq, struct task_struct *p, struct task_struct *next);
	void (*set_next_task)(struct rq *rq, struct task_struct *p, bool first);
	...
```

CFS 实现这一套(`fair_sched_class`,[kernel/sched/fair.c:13597](../linux-6.14/kernel/sched/fair.c#L13597)),实时任务实现另一套(`rt_sched_class`)。**调度器的核心代码(`__schedule`)不需要知道每个类内部怎么选——它只是从高到低挨个问:"你这儿有人能跑吗?",第一个说"有"的那个类,用它自己的 `pick_task` 选一个出来。**

> **回扣内存篇第 1 章**:这就是"分层抽象 + 清晰接口解耦"。调度类的接口固定(一组函数指针),实现可换(CFS/rt/deadline 各写各的)。新增一个调度策略,不用动调度器核心,只要写一个新的 `sched_class` 挂进去——`sched_ext` 就是这么做到"用 eBPF 动态加载自定义调度器"的。

### 阶级顺序靠什么保证

阶级顺序不是靠 if-else 硬编码的,而是靠**链接器布局**——把各 `sched_class` 按地址顺序排好:

> [kernel/sched/sched.h:2566-2575](../linux-6.14/kernel/sched/sched.h#L2566-L2575)

```c
#define for_each_class(class) \
	for_class_range(class, __sched_class_highest, __sched_class_lowest)
	...
#define sched_class_above(_a, _b)	((_a) < (_b))
```

`__sched_class_highest`(stop)到 `__sched_class_lowest`(idle),按地址递增排列。`__schedule` 选下一个时,就用 `for_each_class` 从最高往下扫,第一个有可运行任务的类胜出。**地址顺序 = 优先级**,这是个巧妙的零开销设计。

## 三、CFS 的核心:vruntime

绝大多数进程在 fair 通道,用 **CFS**。这一节是本章的灵魂。

### 痛点:"公平"到底怎么定义

最朴素的"公平"是**轮转(RR)**:每个人轮流跑固定时间片。但这样不公平——一个编译进程和一个编辑器进程,用同样的时间片,编辑器响应会变得迟钝(它需要更频繁地上 CPU 才能及时响应你的按键)。

而且我们还要支持**优先级(nice 值)**:nice 值低的高优先级进程,该多占 CPU。怎么"多占"才算公平?

CFS 的思路极其优雅,一句话:

> **理想情况下,如果 N 个进程同时就绪,CPU 时间应该被它们平分——每个进程的"实际运行时间"始终相等。优先级高的,相当于"它的实际运行时间被放大了"(于是它该多占)。**

但要"始终相等",调度器怎么知道谁该跑了?它需要一个量来衡量"谁已经占够了、该让让了"。这个量就是 **vruntime(虚拟运行时间)**。

### vruntime:把"实际跑了多久"换算成"虚拟跑了多久"

每个进程有一个 `vruntime`,存在它的 `sched_entity` 里(第 16 章见过):

> [include/linux/sched.h:569](../linux-6.14/include/linux/sched.h#L569)(`sched_entity` 内)

```c
	u64				vruntime;
```

它的更新规则简单到惊人——进程每跑一段实际时间 `delta_exec`,它的 vruntime 就增加:

> [kernel/sched/fair.c:1226](../linux-6.14/kernel/sched/fair.c#L1226)

```c
	curr->vruntime += calc_delta_fair(delta_exec, curr);
```

关键在那个 `calc_delta_fair`:它把**实际时间**换算成**虚拟时间**:

> [kernel/sched/fair.c:292-298](../linux-6.14/kernel/sched/fair.c#L292-L298)

```c
static inline u64 calc_delta_fair(u64 delta, struct sched_entity *se)
{
	if (unlikely(se->load.weight != NICE_0_LOAD))
		delta = __calc_delta(delta, NICE_0_LOAD, &se->load);

	return delta;
}
```

这个换算的精髓:

- **普通优先级(nice 0)的进程**:`vruntime` 涨速 = 实际时间涨速(1:1)。跑 1ms,vruntime 涨 1ms。
- **高优先级(权重高)的进程**:`__calc_delta` 让它的 `vruntime` 涨得**慢**。跑 1ms,vruntime 可能只涨 0.5ms。
- **低优先级(权重低)的进程**:`vruntime` 涨得**快**。跑 1ms,vruntime 涨 2ms。

### 决策规则:永远挑 vruntime 最小的

有了 vruntime,CFS 的调度决策就一句话:

> **永远挑 vruntime 最小的那个进程跑。**

为什么这就"公平"了?想一下:

- 所有进程同时开始时,vruntime 都是 0,随便挑一个;
- 被挑中的那个跑了一会儿,vruntime 涨上去了,**它就不再是"最小的"了**,于是换 vruntime 还小的那个跑;
- 如此循环,每个进程的 vruntime **你追我赶、始终咬得很紧**——这正是"实际运行时间大致相等"的体现。
- 高优先级进程 vruntime 涨得慢,所以它**总能更久地保持"最小"**,于是**更频繁地被选中、占更多 CPU**——优先级就这么自然地体现出来了。

> **一个直觉**:vruntime 就像每个市民手上的"已办事时长"计时器,只是高优先级市民的表走得慢、低优先级的表走得快。叫号器永远叫"表上数字最小"的人——于是表走得慢的(高优先级)被叫得更多。**用一个量,把"公平"和"优先级"统一起来了。** 这就是 CFS 的全部优雅。

### 不这样会怎样:用固定时间片轮转

**不这样会怎样?** 如果用经典的时间片轮转 + 固定优先级:nice 值每差一级,时间片就得按某个比例硬调,而且当进程数量变化时,每个人分到的时间片都要重算,逻辑复杂、还容易出现"优先级反转"(低优先级进程因为某些巧合反而霸占 CPU)。

CFS 用 vruntime 把它简化成"挑最小"这一个操作,**自然满足公平,自然支持优先级,还自动适应进程数量变化**。这就是"换一个抽象,把问题变简单"的典范。

## 四、CFS 怎么"挑最小":红黑树 + EEVDF

CFS 要频繁做一件事:**在所有就绪进程里,找 vruntime 最小的那个。** 如果用链表,每次 O(N) 扫一遍,进程多了就慢。

### 数据结构:红黑树

CFS 把就绪进程组织成一棵**红黑树**(按 vruntime 排序):

> [include/linux/sched.h:553](../linux-6.14/include/linux/sched.h#L553)(`sched_entity` 内)

```c
	struct rb_node			run_node;
```

红黑树是一种自平衡二叉搜索树,查找最小值(最左叶子)是 O(log N)。CFS 把这棵树叫"时间线"(`tasks_timeline`),vruntime 最小的进程永远在最左边。挑下一个 = 取最左叶子,O(log N)。

### 6.6 之后的新算法:EEVDF

这里有个**重要的现实更新**:Linux 6.6 起,CFS 的核心算法从旧的"纯 vruntime 最左"升级成了 **EEVDF(Earliest Eligible Virtual Deadline First,最早合格虚拟死线优先)**。现在的 `pick` 函数叫 `pick_eevdf`:

> [kernel/sched/fair.c:905-925](../linux-6.14/kernel/sched/fair.c#L905-L925)

```c
static struct sched_entity *pick_eevdf(struct cfs_rq *cfs_rq)
{
	struct rb_node *node = cfs_rq->tasks_timeline.rb_root.rb_node;
	struct sched_entity *se = __pick_first_entity(cfs_rq);
	...
	/* Pick the leftmost entity if it's eligible */
	if (se && entity_eligible(cfs_rq, se)) {
```

EEVDF 在 vruntime 公平的基础上,引入了两个概念:

- **eligibility(合格性)**:只有"还没占够份额"的进程(eligible)才该被选中,防止某些进程在被唤醒后短时间内疯狂抢占;
- **virtual deadline(虚拟死线)**:每个时间片有个"该被调度的最晚时刻",优先调度死线快到的进程,改善延迟。

**但底子还是 vruntime 那一套**——EEVDF 是对 CFS 的精细改良,不是推翻。你理解了 vruntime 公平,就理解了 CFS 的九成。EEVDF 是为了让"唤醒后的进程不要瞬间抢太多""交互延迟更可预测"这些边角更顺滑。本章不展开 EEVDF 细节,记住"它仍在 vruntime 框架内"即可。

### `update_min_vruntime`:维持基准

为了让新加入的进程有个合理的 vruntime 起点不至于饿死别人(或被别人饿死),CFS 给每个 CPU 的运行队列维护一个基准 `min_vruntime`,每次更新当前进程 vruntime 后同步推进:

> [kernel/sched/fair.c:759-780](../linux-6.14/kernel/sched/fair.c#L759-L780)

```c
static void update_min_vruntime(struct cfs_rq *cfs_rq)
{
	u64 vruntime = cfs_rq->min_vruntime;
	...
	cfs_rq->min_vruntime = __update_min_vruntime(cfs_rq, vruntime);
}
```

新进程入队时,它的 vruntime 会被设成 `min_vruntime` 附近——这样它**不会因为 vruntime 太小而瞬间霸占 CPU**(把老进程饿死),也**不会因为太大而永远轮不上**(被饿死)。这是 CFS 公平性的一个关键保护。

## 五、调度什么时候发生:三种触发

到目前为止我们讲了"怎么选下一个",但还没讲"什么时候选"。调度不是随时发生的,它有**三个触发点**:

### 1. 主动让出(blocking)

进程要等磁盘、等网络、等锁——它**主动**调 `schedule()` 把自己挂起(状态变 `TASK_INTERRUPTIBLE`)。这是最常见的:"我没活干了,谁来接手 CPU?"

```c
// 进程等 I/O 时,典型路径:
set_current_state(TASK_INTERRUPTIBLE);
schedule();   // 主动让出 CPU,直到被唤醒
```

`schedule()` 最终调 `__schedule`,把当前进程移出运行队列,挑下一个上 CPU(第 17 章讲过 `exit`,这里 `schedule` 是"睡觉"不是"退场")。

### 2. 时间片用完(时钟中断驱动)

进程不主动让怎么办?它可能是个死循环 `while(1)`。这就靠**时钟中断**:每个 tick(通常 1ms 或 4ms),时钟中断处理函数 `sched_tick` 会检查"当前进程是不是该让位了"。如果当前进程的 vruntime 已经涨够了(时间片用完),就给它打个标记 `TIF_NEED_RESCHED`("需要重新调度")。

> 注意:**打标记 ≠ 立刻切换。** 时钟中断在中断上下文,不能直接调度。它只是设好标记,等中断返回、回到"可以调度"的时机(内核态抢占点、或返回用户态前),才真正调 `__schedule`。这是"延迟到安全时机再调度"的设计——第 19 章和边界篇会详谈。

### 3. 唤醒抢占

一个高优先级进程被唤醒(比如 I/O 完成了、收到网络包了),它被放回运行队列时,调度器会判断:**它的 vruntime 是不是比当前正在跑的进程还小?** 如果是,它"更有资格",于是给当前进程打 `TIF_NEED_RESCHED` 标记,抢占当前进程。这就是为什么你敲键盘,终端进程能"立刻"上 CPU——它被唤醒后抢占了正在跑的编译进程。

### 三者的统一入口:`__schedule`

不管哪种触发,真正干活的总入口都是 `__schedule`([kernel/sched/core.c:6645](../linux-6.14/kernel/sched/core.c#L6645))。它的核心逻辑(精简):

```c
static void __sched notrace __schedule(int sched_mode)
{
	...
	cpu = smp_processor_id();
	rq = cpu_rq(cpu);              /* 拿到当前 CPU 的运行队列 */
	prev = rq->curr;               /* 当前进程 */
	...
	rq_lock(rq, &rf);
	...
	prev_state = READ_ONCE(prev->__state);
	if (!preempt && prev_state)    /* 主动让出且要睡觉 */
		try_to_block_task(rq, prev, prev_state);  /* 把它移出运行队列 */

	next = pick_next_task(rq, prev, &rf);   /* ★ 挑下一个 */
	...
	if (likely(prev != next)) {
		rq->nr_switches++;
		...
		rq = context_switch(rq, prev, next, &rf);  /* ★ 切窗口(第 19 章) */
	}
	...
}
```

三件事:**(1)如果当前进程要睡觉,把它移出队列;(2)`pick_next_task` 挑下一个;(3)如果下一个不是当前进程,调 `context_switch` 切窗口。** 第 (3) 步就是第 19 章的主角。

## 六、关键源码精读:`__schedule` + `update_curr`

这一章把两个最能体现设计的函数读透。

### 精读一:`__schedule`——调度总入口

`__schedule` 的骨架上面已经贴了。逐段对应设计意图:

- **`rq = cpu_rq(cpu)`**:调度是**每个 CPU 独立**的。每个 CPU 有自己的运行队列(`struct rq`),自己挑自己队列里的进程。多核机器上,8 个核就是 8 个独立的"办事大厅",各自叫号。这避免了全局锁——这是多核可扩展性的关键(后面"负载均衡"会把进程在核之间搬,但核心调度是每核独立的)。
- **`prev_state = ...; if (!preempt && prev_state)`**:只有在"主动调度"(非抢占)且进程有非零状态(要睡觉)时,才把它移出队列。如果是抢占(进程还要继续跑,只是让位),进程**留在队列里**,下次还能被挑中。
- **`pick_next_task`**:遍历调度类(`for_each_class`),找第一个有可运行任务的类,用它的 `pick_task` 选一个。这就是前面讲的"阶级制"落地。
- **`context_switch`**:选定下一个后,真正切窗口。这一步留给第 19 章。

> **一句话**:`__schedule` = 锁队列 → (可能)移出当前进程 → `pick_next_task` 选下一个 → `context_switch` 切窗口。**所有调度,无论谁触发,都汇聚到这一个函数。**

### 精读二:`update_curr`——vruntime 的更新

> [kernel/sched/fair.c:1212-1235](../linux-6.14/kernel/sched/fair.c#L1212-L1235)

```c
static void update_curr(struct cfs_rq *cfs_rq)
{
	struct sched_entity *curr = cfs_rq->curr;
	struct rq *rq = rq_of(cfs_rq);
	s64 delta_exec;
	bool resched;

	if (unlikely(!curr))
		return;

	delta_exec = update_curr_se(rq, curr);          /* 算出"自上次更新以来跑了多久" */
	if (unlikely(delta_exec <= 0))
		return;

	curr->vruntime += calc_delta_fair(delta_exec, curr);   /* ★ vruntime 涨 */
	resched = update_deadline(cfs_rq, curr);               /* 顺带判断是否该设 need_resched */
	update_min_vruntime(cfs_rq);                            /* 推进基准 */

	if (entity_is_task(curr)) {
		struct task_struct *p = task_of(curr);
		update_curr_task(p, delta_exec);                /* 统计实际 CPU 用量 */
		...
	}
	...
}
```

这个函数被**频繁调用**——每次进入调度路径、每次时钟 tick,都会调它刷新当前进程的 vruntime。逐行看:

- **`delta_exec = update_curr_se(...)`**:算"自上次更新以来,当前进程实际跑了多久"。这是 vruntime 增量的原始来源。
- **`curr->vruntime += calc_delta_fair(...)`**:本章的灵魂——把实际时间按权重换算成虚拟时间,累加到 vruntime。高优先级进程这里涨得慢。
- **`update_deadline`**:EEVDF 的死线判断,如果时间片用完了,返回需要重调度。
- **`update_min_vruntime`**:维持基准,保护公平。
- **`update_curr_task`**:顺便记账(实际 CPU 用量,给 `top` 显示的 `%CPU` 用)。

**`update_curr` 是 CFS 公平性的"心脏"**——它持续地把"实际跑了多久"翻译成"虚拟跑了多久",让"挑最小"的决策始终基于最新数据。没有它,vruntime 就是个死的初始值,CFS 的公平就无从谈起。

> **一个细节体现设计**:这一趟更新里,**记账、判断重调度、维持基准、推进 vruntime 全在一个函数里顺手做了**——因为这些事在"当前进程跑了 delta_exec"这个事件上必然要一起做。把它们合并成一次调用,避免反复加锁、反复遍历。这是内核高频路径优化的典型思路:**把相关的工作打包到一次遍历里做完。**

## 七、章末小结

用城市比喻把这一章收口。

这一章我们看清了"几千市民抢几个窗口"的全部机制:

1. **CPU 是时间,调度 = 分配时间片**;"独占 CPU"是靠每秒上千次的飞速轮换维持的幻觉。
2. **调度类是阶级制**(stop > deadline > rt > fair > ext > idle):高阶有人,低阶就排不上。绝大多数进程在 fair 通道,用 CFS。调度类是一组函数指针(接口),实现可换——这是分层抽象的典范。
3. **CFS 的核心是 vruntime**:把"实际跑了多久"按权重换算成"虚拟跑了多久",**永远挑 vruntime 最小的跑**。一个量,统一了"公平"和"优先级"。6.6 之后升级为 EEVDF,但仍在 vruntime 框架内。
4. **调度三种触发**:主动让出、时钟中断时间片用完、唤醒抢占——都汇聚到 `__schedule` 这一个总入口。

> **回扣全书主线**:这一章是**"分配窗口"那一侧**的核心。前两章(16、17)造出了进程(市民),这一章决定"谁此刻上窗口"。**调度器造的"独占 CPU"幻觉,和内存篇造的"独占内存"幻觉,是市民能安心写程序的两个基石。**
>
> - **承前**:它直接消费第 16 章的 `sched_entity`(vruntime 就挂在它上面),也依赖第 17 章 `fork` 时给新进程初始化的 vruntime;
> - **启后**:`__schedule` 最后一步调的 `context_switch`,正是第 19 章的主角——"选好了下一个,接下来怎么把窗口从 A 真正交给 B"。

### 记住这三句话

1. **"独占 CPU"是幻觉,真相是飞速轮换**——CPU 是时间,调度就是分时间片。
2. **调度类是阶级 + 函数表**:高阶有人低阶就排不上;绝大多数进程在 fair 通道用 CFS。
3. **CFS 靠 vruntime 逼近公平**:把实际时间按权重换算成虚拟时间,永远挑最小的跑;高优先级进程 vruntime 涨得慢,自然多占。
