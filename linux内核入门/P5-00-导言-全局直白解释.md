# 导言 · 一篇看懂内核并发与同步全貌

> 这篇是"地图"。读各章之前先读它,你会对整个并发同步子系统建立一个结实的直觉。
> 全文只讲两件事:**它在干什么**,以及**为什么非得这么干**。
> 我会用一个贯穿始终的比喻——**市政府多个窗口同时改一份档案**——帮你记住每一步。

> **前置**:你需要先读过全书导言《一篇看懂内核全貌》(了解"城市比喻映射表")和第 4 篇·边界(中断/异常)。本篇反复出现"中断随时会打断你",不先理解中断,就读不懂为什么要关中断。

## 一、一句话说清并发同步在干嘛

> **市政府里有一份共享档案(比如全城住户登记表)。几十个办事窗口(CPU 核)的办事员、随时响起的门铃(中断)、还会中途换班的办事员(抢占),都可能"同时"去翻这份档案、往上面写东西。并发同步的全部工作,就是保证:这份档案不被改乱、不被读到半新半旧,同时尽量不让办事员傻站在门口排队。**

就这么一句话。后面所有复杂设计,都是在实现这个目标。你任何时候看不懂某个机制,就回来问自己一个问题:

> **"这个机制,是在'挡住别人别同时改',还是在'让等的人别瞎忙活',还是在'保证改的顺序被正确看见'?"**

整个并发同步子系统就这三块活儿。记住这个三分法,你就永远不会迷路:

- **挡住别人**(锁):自旋锁、互斥锁——把"改档案的那几步"圈起来,一次只准一个。
- **让读者别瞎忙活**(RCU):读多写少时,干脆让读者**不锁、读旧版**,写者造好新版再替换。
- **保证顺序被正确看见**(内存屏障):光锁还不够,还要防止编译器和 CPU 把"改的顺序"打乱。

## 二、先回答一个扎心的问题:内核为什么会有"并发"?

很多人第一次听到"内核并发"是懵的:同一个内核,又不是多线程的用户程序,它到底在跟谁并发?

答案是:**内核从来就不是"一个执行流"**。它在同一时刻,可能有好几个"自己"在跑:

| 谁在跑 | 比喻 | 怎么产生并发 |
|--------|------|--------------|
| **多个 CPU 核** | 多个办事窗口同时办公 | 窗口 A 和窗口 B 同时去翻那份档案(SMP) |
| **中断 / 软中断** | 门铃响了,办事员被迫放下手头活去应门 | 你改档案改到一半,被门铃打断,中断处理函数**也去改同一份档案** |
| **抢占(preempt)** | 办事员干到一半被换班 | 你改档案改到一半,被换下来,接班的**也去改同一份档案** |

> ⚠️ **关键认知**:哪怕你的机器只有一个核,后两种(中断 + 抢占)照样存在。所以**单核内核照样有并发问题**——这是初学者最容易栽的坑。"我只有一个核,应该不用锁吧?"——错。中断会让单核也变成"事实上的多线程"。

这三股力量,任意两股同时去摸同一份共享数据,就可能出乱子。第 24 章会逐个演示"不防会怎样",这里先给你建立直觉。

## 三、建立比喻:一份档案与三把武器

我们把任何"被多方共享、可读写"的数据(全局链表、计数器、页表、路由表……)想象成**市政府的一份档案**。围绕它,内核准备了三套武器:

| 比喻 | 对应的内核概念 | 一句话 |
|------|----------------|--------|
| 一份共享档案 | 共享数据结构(链表/计数器/页表…) | 所有人都能翻、都能写 |
| 改档案的那几步 | **临界区(critical section)** | 真正摸数据的代码,不是整段函数 |
| 档案室门锁 | **锁(lock)** | 把临界区圈起来,一次只准一人 |
| 没钥匙的人在门口站着等 | **自旋锁(spinlock)** | 原地忙等,适合等很短、且不能睡 |
| 拿号去休息区坐着等广播 | **互斥锁/睡眠锁(mutex)** | 去睡觉被叫醒,适合等很久、可以睡 |
| 读者拿旧复印件随便看,写者造新版再换 | **RCU(Read-Copy-Update)** | 读多写少专用,读者几乎零开销 |
| 档案员交代"这两笔改动必须按顺序被别人看见" | **内存屏障(memory barrier)** | 防止编译器/CPU 把顺序打乱 |
| 关门铃(暂时不接电话) | `local_irq_disable` / `spin_lock_irqsave` | 防中断来捣乱 |
| 暂停换班(不让别人插队) | `preempt_disable` | 防抢占来捣乱 |
| 等旧读者全部把复印件还回来 | **宽限期(grace period)** | RCU 回收旧版的前提 |

这张表先扫一眼留个印象,下面我会带着你从头走一遍。

## 四、跟着问题走一遍:从"抢乱"到"各司其职"

### 第 1 步:先看清"乱"是怎么发生的

想象一份最简单的共享档案:一个全局计数器 `count`,好多窗口都在 `count++`。

`count++` 在源码里是一行,可它**不是一条指令**——它是"读 count → 加 1 → 写回 count"三步。现在两个核同时干这件事:

```
核 A: 读到 count=5
核 B: 读到 count=5      ← 两人读到的都是旧的 5
核 A: 算出 6,写回
核 B: 算出 6,写回       ← count 变成 6,而不是 7!
```

本该是 7,结果成了 6。这就是**竞态(race condition)**。链表更惨:两个核同时 `list_add`,可能把链表指针改成"谁也不是"的样子,内核当场崩。

> **为什么会乱?** 因为"读-改-写"不是原子的,中间能被别人插进来。**治本的办法只有一个:把这几步圈起来,一次只准一个执行流进去。** 这就是"锁"存在的全部理由。

### 第 2 步:第一把武器——锁(自旋 vs 睡眠)

锁的思路朴素得像公共厕所的门锁:进去的人把门反锁,外面的人只能等。

可"等"有两种截然不同的等法,内核因此准备了两把锁:

**(A) 自旋锁 spinlock:原地站着等(忙等)**

- 等的人不离开、不睡觉,就站在门口不停问"好了没好了没"(在一个循环里反复读锁的状态)。一旦门开了,他**零延迟**冲进去。
- **代价**:等的时候 CPU 是满负荷空转的,白费电。所以**只适合"等的时间极短"**——临界区就几条指令。
- **硬性规矩**:拿着自旋锁的时候**绝不能睡觉**。为什么?因为自旋锁的本意是"很快就好",你一睡,外面站着傻等的人可能就等到地老天荒;更糟的是,在 SMP 上,睡下去后如果持有锁的任务没被调度回来,等你的核就死循环(活锁甚至死锁)。第 25 章会把这条规矩讲到底。

**(B) 互斥锁/睡眠锁 mutex:拿号去休息区坐着等**

- 等的人**不傻站**,而是去休息区睡觉,把自己的名字挂到等待队列上。持锁的人一出来,会广播"叫下一个号",被叫到的人才醒来去拿锁。
- **好处**:等的时候 CPU 能去干别的活(甚至这个核整个省电);可以等**很长很长**的时间(等一个慢设备、等一页从磁盘换回来)。
- **代价**:上下文切换(睡下去 → 被唤醒)是有成本的,大概几百到几千个时钟周期。所以**只适合"等的时间值得睡一觉"**的长临界区。
- **硬性规矩**:mutex 涉及睡觉,而**中断处理函数不能睡觉**(中断里没有"可睡的进程上下文")。所以 mutex 用在进程上下文里。

> **怎么选?** 记一句话:**临界区里要睡觉、或临界区很长 → mutex;临界区极短、或在中断上下文 → spinlock。** 第 25 章会展开成一整套判断流程。

### 第 3 步:锁的变体——"我到底在防谁?"

光是 spinlock 还不够。你拿锁圈住了别的核,但**门铃(中断)还能打断你**。设想:

```
核 A: spin_lock() 拿到锁,开始改档案
   ↑ 中断来了!中断处理函数也想 spin_lock() 同一把锁
   → 中断处理函数死等这个锁,而这个锁正被"被中断打断的 A"拿着
   → 死锁。整个核卡死。
```

为了防这种情况,内核给 spinlock 配了一堆变体,本质都是"拿锁的同时,顺手把某股捣乱的力量也关掉":

| 变体 | 拿锁的同时还关了什么 | 防的是谁 |
|------|----------------------|----------|
| `spin_lock` | 关抢占(默认就关) | 别的核 + 抢占 |
| `spin_lock_irq` | 关**本地中断** | 别的核 + 本核中断 |
| `spin_lock_irqsave` | 关中断**且保存中断原状态**(最常用) | 同上,但能嵌套、能还原 |
| `spin_lock_bh` | 关**下半部(软中断)** | 别的核 + 软中断 |

> **为什么这么碎?** 因为"关掉的力量越多种,代价越大"(比如关中断会让这个核对别的中断也不响应)。内核的原则是**只关必要的、能少关就少关**。第 25 章会逐个讲清。

### 第 4 步:锁太重了——RCU 登场

锁解决了"别同时改",但它有个**原罪:读者也得排队**。

设想一份**路由表**,全城几万个市民每秒都在查它,但极少有人改它。如果用锁:

- 每个读者来,先抢锁(哪怕读锁也要做原子操作、刷缓存)→ 几万次/秒的锁竞争,缓存行被打爆,性能塌方。
- 可这份档案明明**只读不改**的时候,大家完全可以一起看,根本不需要互相挡。

RCU 的天才之处:**让读者根本不锁,几乎零开销地读;代价全部转嫁给极少数的写者**。怎么做到?

```
旧版档案 ──┐
           ├─→ 指针 P(读者只通过 P 找档案)
           │
写者要改:不直接改旧版!
  1. 复制一份旧版 → 得到"新版草稿"
  2. 在草稿上随便改(没人在看草稿,不用锁)
  3. 改完,把指针 P 原子地指向新版 ──→ 这一刻起,新读者都看新版
  4. 旧版怎么办?还有"在替换之前就已经开始读"的读者正拿着它!
     → 写者耐心等,等这些旧读者全部退出读临界区(这叫一个"宽限期 grace period")
  5. 宽限期一过,旧读者全走光了,安全销毁旧版
```

> **为什么这么牛?** 读者只需要"进入临界区 → 读 → 退出临界区"三步,中间**没有任何原子操作、没有缓存行争用**(连锁的影子都没有)。写者付出了"复制 + 等宽限期"的代价,但写者本来就少,这笔账划算。
>
> **代价**:RCU 不是万能的。它**只能保护"指针指向的数据"**(因为替换靠换指针);它要求**旧版本在宽限期内不能被改坏或释放**;它的写者有延迟(要等宽限期)。所以它专治**读多写少 + 指针可整体替换**的场景。第 26 章会讲透,并回扣内存篇 `SLAB_TYPESAFE_BY_RCU`。

### 第 5 步:最后一只妖精——CPU 会"乱序"

你以为加了锁/RCU 就稳了?还有一只看不见的妖精:**乱序执行**。

你写了这样的代码:

```c
data = 42;          // (1) 先把数据准备好
ready = true;       // (2) 再把"数据好了"的标志位置上
```

读者这样读:

```c
while (!ready);     // 等标志位
print(data);        // 用数据
```

逻辑上天衣无缝。**可实际跑起来,读者可能打印出一个旧 data**!为什么?

两个捣蛋鬼:
- **编译器**:为了优化,它可能把(1)(2)的顺序对调,或者把 data 的读缓存到寄存器。
- **CPU + store buffer**:为了让流水线快,CPU 也不一定按你写的顺序把写操作真正送到内存;写还可能先停在 CPU 旁边的"store buffer"里,晚一拍才被别的核看见。

结果:写者的标志位 `ready` 可能在 `data` 之前就被读者看见了。

> **怎么治?** 内存屏障就是"强制顺序"的指令:
> - `READ_ONCE` / `WRITE_ONCE`:拦住编译器("这个变量,不许你优化、不许你重排、给我老老实实读写")。
> - `smp_wmb`(写屏障)/ `smp_rmb`(读屏障)/ `smp_mb`(全屏障):拦住 CPU("屏障之前的写,必须先于屏障之后的写,被别的核看见")。
> - `smp_store_release` / `smp_load_acquire`:更精细、更便宜的"配对屏障"——现代内核首选。
>
> 好消息:你**日常用锁/RCU 时不用自己加屏障**——`spin_lock`、`mutex_lock`、`rcu_assign_pointer` 内部都偷偷编好了屏障。**只有当你写"无锁(lockless)"代码**——自己用原子变量和标志位做同步——才需要亲自上屏障。第 27 章会讲清"什么时候你才会真的需要手写屏障"。

## 五、把三把武器放在一起:一个判断流程

面对一段要保护共享数据的代码,你的思考顺序应该是:

```
1. 临界区有多长?里面要睡觉吗?在中断上下文吗?
   ├─ 长 / 要睡 / 进程上下文 ─────────────► mutex
   └─ 极短 / 不睡 / 可能在中断上下文 ─────► spinlock (+ irqsave/bh 变体)

2. 是不是"读极多、写极少、且数据靠指针整体替换"?
   └─ 是 ───────────────────────────────► 优先考虑 RCU

3. 是不是"无锁、自己用原子变量/标志位做同步"?
   └─ 是 ───────────────────────────────► 别忘了内存屏障(acquire/release)
```

记住:这三把武器不是"谁取代谁",而是**针对不同病情的不同药**。成熟的内核代码里,经常**同一份档案的不同操作用不同的武器**:写用 spinlock、读用 RCU、跨核状态用 acquire/release。第 25~27 章会分别把这三种武器讲到底。

## 六、收束:三句话 + 一个心法

走完全篇,你看到的其实是**对"共享"这件事的层层缓解**:

```
   最朴素:大家随便改           →  乱
   + 锁:一次只准一个改        →  不乱了,但读者也排队
   + RCU:读者干脆不锁         →  读多写少时飞快
   + 屏障:保证顺序被正确看见  →  无锁代码也能正确
```

### 记住这三句话,就抓住了全篇

1. **内核并发有三个来源:多核、中断、抢占。** 单核也不安全,因为还有中断和抢占。
2. **锁是"挡住别人别同时改",RCU 是"让读者别瞎忙活",屏障是"保证顺序被正确看见"。** 三个不同的病,三副不同的药。
3. **所有并发原语都在做一个权衡:正确性 vs 性能 vs 可睡眠性。** spinlock 牺牲(睡眠)换性能,mutex 牺牲(切换开销)换可睡,RCU 牺牲(写延迟+内存)换读者极速。当你看不懂某个设计,去找它在牺牲什么、换什么。

### 一个心法

> **写内核代码,默认假设"任何一行都可能被打断、被并发执行"。** 你手里的每一份共享数据,要么明确地用锁/RCU 护住,要么明确地论证"这里无需保护(为什么)"。**"我没保护"必须是一个有理由的选择,而不是"我忘了"。**

这也是为什么 lockdep(内核的锁依赖检测器)如此重要——它会在运行时帮你抓出"忘了保护"和"保护错了"的代码。本篇附录和各章都会提到它。
