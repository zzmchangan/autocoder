# 第 2 章 · 白纸阶段:上电、BIOS 与 e820 地图

> **核心问题**:机器刚通电,内核还没跑,谁在管内存?内核醒来时,它看到的到底是一份什么样的"使用说明书"?
>
> **读完本章你会明白**:
> - 为什么内存**不是**"从 0 开始连续 16GB",而是一张满是窟窿的地图;
> - 为什么内核**不能闭着眼相信**"这台机器有多少 RAM",而必须靠 BIOS/UEFI 亲手把一张内存图递过来;
> - 实模式 → 保护模式 → 长模式那套花样,到底是在解决内存的什么麻烦;
> - 以及——这一切,只是为后面所有内存机制"铺好第一张白纸"。

> **如果一读觉得太难**:先只记住三件事——
> ① 物理内存**不是**"从 0 开始连续一大条",而是**满是窟窿**的地址地图(显存、BIOS、设备 MMIO 占走了大段地址);
> ② BIOS 把这张地图整理成 **e820** 表交给内核,每段标明起始、大小、**类型**(RAM / 保留 / ACPI…);
> ③ 内核**不盲信**这份表,会先**清洗**一遍(排序、消除重叠、冲突时偏向"不可用"),只把 `E820_TYPE_RAM` 的段交给 memblock。
> 三段模式切换、变化点算法的细节,第二遍配合源码再抠,不影响往下读第 3 章。

---

## 章首·为什么单独用一章讲"上电那一瞬间"

上一章我们说过,内存管理本质上是在干一件事:**在一栋满是窟窿、被人共享的物理大楼上,搭出层层抽象**。

但抽象得有个起点。你不能凭空分配房间——你总得先知道**这栋楼到底有几层、哪几层是能住人的、哪几层早被电梯井和配电室占了**。这份"楼情普查",不是内核自己做的,因为内核醒来之前它还不存在。普查是**施工方(BIOS/UEFI)**做的,内核一睁眼,接到的就是施工方递过来的一张表。

这一章讲的就是这张表:**它长什么样、为什么长这样、内核拿到它之后第一件事是干什么。**

它对应导言里大楼时间线的**第 0 步**——白纸阶段。这一步不碰任何分配,只解决一件事:**搞清楚手里到底有什么货**。但你会发现,光是"搞清楚有什么货",就够写一整章,因为现实世界的物理内存,远比"16GB"三个字复杂。

## 一、上电那一刻:谁醒着,谁认识多少地址

### 提出问题

按下电源键的那一瞬间,CPU 并不是"满血复活"的。它处于一个特别原始的状态,脑子里只有最基本的能力。这时候内核还躺在硬盘里,根本没跑。那么——**这一刻到底是谁在管事?它又能管多少内存?**

### 不这样会怎样?

为了理解为什么 CPU 上电时要"装笨",我们先看一个反直觉的事实:

> 一颗能跑 64 位、寻址空间高达 128TB 的现代 CPU,在通电的第一条指令时,**只认识最低的 1MB 地址**,而且**没有任何内存保护、没有页表、没有虚拟地址**。

听起来很荒唐。但这是历史包袱逼出来的。

x86 这颗架构从 1978 年的 8086 一路演化而来。8086 是 16 位的,只能寻址 1MB(`0x00000`–`0xFFFFF`)。Intel 在设计后续芯片(80286、80386……一直到今天的 Core/Xeon)时,为了让几十年前那些为 8086 写的 BIOS 代码、引导代码**还能在新芯片上电时正常跑**,做了一个硬性承诺:

> **不管你这颗 CPU 多先进,刚上电时,必须表现得像一台老 8086。**

这个"装笨"的模式,叫**实模式(real mode)**。CPU 上电后第一条指令就执行在实模式下,地址 `0xFFFFFFF0`(复位向量)那里放着 BIOS 的第一条代码。

如果 CPU 一上电就是 64 位、带页表的"聪明模式",那几十年积累的 BIOS 固件、引导加载器(grub 早期阶段)全都要重写,而且这套固件自己都没法初始化自己(因为它得先有内存分配器才能初始化页表——又一个鸡生蛋)。所以**"装笨"是为了让启动链条能一环扣一环地把自己拎起来**。

### 所以这样设计:三段式切换

启动过程就是 CPU **一路从笨变聪明**的过程,内存的访问方式跟着变三档:

| 模式 | 比喻 | 能力 | 内存怎么访问 |
|------|------|------|--------------|
| **实模式(real mode)** | 大楼刚通电,只有大堂的一盏灯能亮 | 16 位段寄存器 + 偏移,凑出 20 位地址,**最多 1MB** | 程序给出的地址**就是物理地址**,没有任何翻译,没有保护 |
| **保护模式(protected mode)** | 大楼开了几层,开始有门禁卡 | 32 位,可寻址 4GB,引入**段描述符**和**页表** | 地址要先经过段/页翻译,有了权限检查 |
| **长模式(long mode)** | 整栋楼全亮,64 位 | 64 位虚拟地址空间(实际用 48/57 位) | 完整的多级页表,这就是内核长期驻留的模式 |

为什么要逐级切,而不是一步到位?因为每一级的"变聪明"动作,本身都需要内存来放数据结构(段表、页表),而这些数据结构又得在前一级能寻址的范围内布置好。**这是个递进的自举过程,每一级用上一级提供的能力,搭出下一级的脚手架。**

> **子比喻**:这就像盖楼时,先用手推车(实模式)把砖运到工地,搭好脚手架(保护模式/页表)之后,才能开塔吊(长模式)把整层楼吊上去。你不能一上来就开塔吊,因为塔吊自己还没装好。

具体到 x86 的启动代码:BIOS 在实模式跑 → 加载 bootloader(grub)的 stage1,还在实模式 → grub 把内核镜像 `bzImage` 装进内存 → 跳进内核的入口。内核的入口代码([arch/x86/kernel/head_64.S](../linux-6.14/arch/x86/kernel/head_64.S))干的第一批事里,就有"从实模式/保护模式切到长模式、建立第一张临时页表"。**这是下一章的主角**,本章我们只关心:在这一切发生之前,BIOS 干了什么、留了什么。

### 代码佐证(先埋个钩子)

这一章我们还不钻启动汇编,但你可以先记住入口在哪。内核 64 位入口的早期会建立一张**身份映射(identity mapping)**的临时页表,把自己从实模式送进长模式。`setup_arch()` 是 x86 启动的 C 语言大本营,定义在 [arch/x86/kernel/setup.c:732](../linux-6.14/arch/x86/kernel/setup.c#L732)。本章后半段那张"内存使用说明书",最终就是被送进这个函数消化的。

## 二、内存地图为什么满是窟窿

### 提出问题

假设我问你:"一台 16GB 内存的机器,内存地址是从 `0x0` 到 `0x400000000`(16GB)这一整条连续的、能存数据的区域吗?"

绝大多数初学者会点头。**这是个致命的误解**,也是本章最重要要打破的一个幻觉。

### 不这样会怎样?

如果内存真的就是"从 0 开始连续 16GB 一整条干净的区域",那内存管理会简单得多——内核只要知道一个起始地址和一个长度就够了。但现实是:

> **物理地址空间不是"一整条 RAM",而是一张"地址地图",地图上有些地址段是真内存条(RAM),有些段根本不是 RAM,而是别的硬件寄存器。**

那些"不是 RAM"的地址段,典型有:

1. **显存(VGA/帧缓冲)**:从 `0xA0000` 开始的一块,你的显卡把屏幕像素映射在这里。你往 `0xA0000` 写一个字节,屏幕上某个像素就变色。这**不是内存**,是显卡。
2. **BIOS 自己的代码和数据**:`0xF0000` 一带,放着 BIOS 固件本身。
3. **设备的 MMIO(Memory-Mapped I/O)寄存器**:PCIe 显卡、网卡、USB 控制器,各自在地址空间里"圈了一块地",CPU 往这块地址读写,实际上是在跟设备对话,而不是存数据。比如你的显卡的显存可能在 `0xE0000000`–`0xF0000000` 这段。
4. **ACPI 表、NVS(非易失存储)区**:BIOS 留给电源管理的特殊区域。

如果内核傻乎乎地把这些地址段当成 RAM 来用,往显存地址里分配一个 `task_struct`,那它写进去的数据就会被显卡当成像素显示出来,或者被某个网卡当成控制命令——**系统当场崩溃或行为诡异**。

### 所以这样设计:地址空间 ≠ RAM

物理地址空间是一张**统一的地图**,但地图上的"地块"性质不同:

```
物理地址        性质
0x00000000  ┐
            │  真 RAM(可用)
0x0009FFFF  ┘
0x000A0000  ┐  显存 / BIOS(著名的 "640K 边界")
0x000FFFFF  ┘
0x00100000  ┐
            │  真 RAM(可用,从 1MB 开始)
   ...      │
0xXXXXXXXX  ┘
0xE0000000  ┐  显卡 MMIO(不是 RAM!)
0xF0000000  ┘
   ...
```

> **回扣大楼比喻**:物理地址空间是这栋楼的"全部门牌号"。但不是每个门牌号背后都是一间能住人的房间——有的门牌背后是电梯井(MMIO),有的是配电室(显存),有的是物业档案室(ACPI)。**BIOS 在交楼时,必须告诉内核一张清单:哪些门牌是真房间,哪些是设备占的。**

这张清单,就是 **e820 内存图**。

> 顺便提一句那个著名的"640KB 边界":DOS 时代,`0xA0000` 以下的 640KB 是真 RAM,`0xA0000`–`0x100000` 那 384KB 是显存和 BIOS。这导致早期 PC "可用内存上限是 640KB",也是比尔·盖茨那句(被误传的)"640K 对谁都够了"的背景。这个边界至今还在内核代码里,我们在 [e820/types.h](../linux-6.14/arch/x86/include/asm/e820/types.h) 能看到它的化石:`ISA_START_ADDRESS`、`HIGH_MEMORY = 0x00100000`(就是 1MB 那条线)。

## 三、e820:BIOS 递给内核的"房间清单"

### 提出问题

既然物理地址空间是一张性质复杂的地图,那 BIOS 用什么格式把这张地图交给内核?为什么是 e820 这套东西?

### 不这样会怎样?

BIOS 得给内核传达的信息是:"地址 `[a, b)` 这段是 RAM,`[c, d)` 那段是保留给显存的,`[e, f)` 是 ACPI……"

最朴素的方案是:BIOS 直接告诉内核一个数字——"你有 16GB RAM"。但这立刻就撞上前面那个问题:**地址空间不是连续的**,光给一个总量,内核根本不知道中间哪里有窟窿、哪里被设备占了。它没法安全地分配任何内存。

所以 BIOS 必须给一张**分段清单**:每一段标明起始地址、大小、**类型**(这段到底是 RAM 还是别的)。这就是 e820 协议干的事。

> **e820 是什么?** 它是传统 BIOS 提供的一个中断调用(`INT 0x15, AX=0xE820`)。内核(或引导加载器)反复调用这个中断,每次 BIOS 就吐出"地图里的下一段"。早期叫法 "e820" 就是这个中断号。UEFI 时代换成了 EFI 内存图,但 Linux 内核为了统一处理,会把它也"翻译"成 e820 格式。所以 e820 现在更像是**Linux 内部对"BIOS 内存图"的统一抽象**,不一定是真的来自那个老中断。

### 所以这样设计:一张带类型的区间表

e820 的本质,就是一张"区间数组",每个区间用三个字段描述:

```c
struct e820_entry {
	u64			addr;   // 这段从哪个地址开始
	u64			size;   // 这段多大
	enum e820_type		type;   // 这段是什么性质
} __attribute__((packed));
```

这段定义在 [arch/x86/include/asm/e820/types.h:55](../linux-6.14/arch/x86/include/asm/e820/types.h#L55-L59)。注意它被 `__attribute__((packed))` 紧凑打包——因为大机器上这种区间可能有几千个,省一个字节就是省几千字节。

`type` 字段是灵魂。它告诉内核这段地址**能不能当 RAM 用**:

```c
enum e820_type {
	E820_TYPE_RAM		= 1,   // 真·可用内存
	E820_TYPE_RESERVED	= 2,   // 保留,别碰(常是 BIOS/设备)
	E820_TYPE_ACPI		= 3,   // ACPI 表,内核要用但不是普通 RAM
	E820_TYPE_NVS		= 4,   // 非易失存储,睡眠时不能动
	E820_TYPE_UNUSABLE	= 5,   // 坏掉的内存
	E820_TYPE_PMEM		= 7,   // 持久内存(NVDIMM)
	...
};
```

(见 [types.h:10](../linux-6.14/arch/x86/include/asm/e820/types.h#L10-L47)。)

> **比喻对照**:这就是施工方交给物业的那张"楼情普查表"——每行写"几号到几号门牌是真房间(RAM)、几号到几号是电梯井(RESERVED)、几号是档案室(ACPI)"。内核拿着这张表,才知道哪些房间能分给住户。

### 这些区间从哪塞进内核?

引导加载器(grub 等)在把内核装进内存之前,自己先调 BIOS 的 e820 中断,把那张区间表收集好。然后它把这张表塞进内核约定好的一块"启动参数区"(叫 `boot_params`,也就是历史上著名的"零页 zeropage")。内核一醒来,就从这块约定区域里把表读出来。

因为零页空间有限(结构体大小固定),传统 e820 **最多只能塞 128 条区间**(`E820_MAX_ENTRIES_ZEROPAGE`)。128 条对老机器够用,但现代大服务器(几百个 NUMA 节点)轻松上千条——所以内核还有个扩展机制(`e820__memory_setup_extended`,通过 `setup_data` 链表传额外条目,见 [e820.c:725](../linux-6.14/arch/x86/kernel/e820.c#L725))。而内核内部的表预留得更宽:

```c
#define E820_MAX_ENTRIES	(E820_MAX_ENTRIES_ZEROPAGE + 3*MAX_NUMNODES)
```

见 [types.h:89](../linux-6.14/arch/x86/include/asm/e820/types.h#L89)——"每个 NUMA 节点预留 3 条,外加零页那 128 条"。这条注释本身就是一段"为什么这么定"的故事,值得读读源码注释。

## 四、关键转折:为什么内核不能闭着眼相信这张表

### 提出问题

到这里,你也许会想:那简单了——BIOS 把 e820 表递过来,内核照单全收,把所有 `E820_TYPE_RAM` 的段加进自己的可用内存池,完事。

**不行。** 这是本章第二个反直觉点。

### 不这样会怎样?

BIOS 不是神。它是各主板厂商写的固件,质量参差不齐,历史悠久、bug 满天飞。常见的坑有:

- **区间互相重叠**:BIOS 报告的两段区间地址互相覆盖(明明是同一段内存,报了两次,或两段类型不同的区域叠在一起)。
- **该标保留的没标**:某段地址其实被显卡 MMIO 占了,但 BIOS 错误地标成了 RAM。内核一旦往里分配数据,数据就进了显卡寄存器。
- **凭空多出来的"RAM"**:有些 BIOS 把根本不存在的地址段也报成 RAM。
- **顺序混乱、零长段、重复段**:什么样的脏数据都有。

如果内核照单全收,随便一个 BIOS 的低级 bug,都会让内核把数据写进设备寄存器、或者把同一段内存当成两段重复分配——后果是随机崩溃,而且极难调试,因为"Bug 在 BIOS 里,但炸在内核里"。

### 所以这样设计:内核拿到表后,先"清洗"一遍

Linux 的做法很务实:**信任,但要核实。** 内核拿到 e820 表后做的第一件事,不是用它,而是**清洗(sanitize)**它:

1. **排序**:把所有区间按地址从小到大排好。
2. **消除重叠**:用一套"变化点(change-point)"算法,把互相重叠的区间切开、归并,生成一张**不重叠、有序**的新表。
3. **冲突仲裁**:在重叠区,谁的类型"更保守"谁赢。代码里的规则是**类型值大的优先**(类型值越大越"不可用":1 是 RAM,2/3/4 是各种不可用),所以在"RAM 和 RESERVED 重叠"的地方,内核会判它为 RESERVED——**宁可错杀一段可用 RAM,也不能把设备寄存器当 RAM 用**。这是个非常对的安全取向。

这套清洗逻辑的核心函数是 `e820__update_table()`,我们待会儿在源码精读里细看。它会被反复调用——每次内核修改了 e820 表(比如把某段从 RAM 改成 RESERVED),都要重新清洗一遍。

清洗完,内核还干了另一件体现"不信任"的事:**同时维护三张表**,各有各的用途:

```c
static struct e820_table e820_table_init		__initdata;   // 主表:内核实际用的、会被各种修改
static struct e820_table e820_table_kexec_init		__initdata;   // 给 kexec/kdump 用
static struct e820_table e820_table_firmware_init	__initdata;   // 原封不动保留 BIOS 的原始版本

struct e820_table *e820_table __refdata			= &e820_table_init;
```

见 [e820.c:59](../linux-6.14/arch/x86/kernel/e820.c#L59-L65)。

为什么要留一份**原始的 `e820_table_firmware`**?因为内核在启动过程中会不断"动手脚"修改主表(比如把自己代码占的那段标成 RESERVED),但有些场合(kexec 跳到下一个内核、调试)需要看 BIOS **原话**是怎么说的。留一份原件,就是留个底,免得改坏了查无对证。

> **比喻**:物业拿到施工方的普查表后,自己又抄了几份。一份是"工作底稿"(主表,随便改),一份锁进保险柜(`firmware` 表,原样保存)。日常分配用底稿,出事了翻原件对质。

## 五、一张真实的 e820 长什么样

讲了一堆抽象,给你看一眼真实机器开机后 dmesg 里的 e820(`e820__memory_setup` 里 `e820__print_table` 打印的)。一台典型的 16GB 桌面机,大概长这样:

```
BIOS-e820: [mem 0x0000000000000000-0x000000000009fbff] usable
BIOS-e820: [mem 0x000000000009fc00-0x000000000009ffff] reserved
BIOS-e820: [mem 0x00000000000f0000-0x00000000000fffff] reserved
BIOS-e820: [mem 0x0000000000100000-0x000000005cffffff] usable
BIOS-e820: [mem 0x000000005dd00000-0x000000005fffffff] reserved
BIOS-e820: [mem 0x0000000100000000-0x000000043fffffff] usable
```

读这张表,你能看到本章讲的所有现象:

- **不是连续一条**:`usable`(可用 RAM)分成好几段,中间被 `reserved` 切断。
- **640KB 边界**:`0x9fc00` 处开始就是 reserved——这正是 `0xA0000` 那块显存/BIOS 区的前哨。
- **1MB 以下的窟窿**:`0xF0000`–`0xFFFFF` 是 BIOS 自己。
- **中间的设备保留区**:`0x5dd00000`–`0x60000000` 那段 reserved,是 BIOS 留给各种固件/设备的。
- **地址不连续地"跳"了一下**:`0x60000000` 之后到 `0x100000000`(4GB)之间有空洞,然后 RAM 又从 4GB 之上冒出来(`0x100000000`–`0x43fffffff`)。这就是为什么 16GB 的机器,max RAM 那一段往往从 4GB 往上算。

> **印证**:如果你以为"16GB 就是 0 到 `0x400000000` 一整条 usable",这张表当场打脸。真实物理内存就是这样千疮百孔的。内核所有后续机制,都要在这种"满是洞"的现实上工作。

## 六、关键源码精读

现在我们把前面所有"为什么"落到代码上。挑三个最关键的函数。

### 6.1 `e820__memory_setup()`:开机第一口"吃下"BIOS 内存图

这是内核消化 BIOS 内存图的入口。它在 `setup_arch()` 的很早期被调用:

> [arch/x86/kernel/setup.c:839](../linux-6.14/arch/x86/kernel/setup.c#L839) 处,`setup_arch()` 调用 `e820__memory_setup()`。

函数本身在 [arch/x86/kernel/e820.c:1284](../linux-6.14/arch/x86/kernel/e820.c#L1284-L1298):

```c
void __init e820__memory_setup(void)
{
	char *who;

	/* This is a firmware interface ABI - make sure we don't break it: */
	BUILD_BUG_ON(sizeof(struct boot_e820_entry) != 20);

	who = x86_init.resources.memory_setup();

	memcpy(e820_table_kexec, e820_table, sizeof(*e820_table_kexec));
	memcpy(e820_table_firmware, e820_table, sizeof(*e820_table_firmware));

	pr_info("BIOS-provided physical RAM map:\n");
	e820__print_table(who);
}
```

逐段读:

- **`BUILD_BUG_ON(sizeof(struct boot_e820_entry) != 20)`**:编译期断言。`boot_e820_entry` 必须正好 20 字节(addr 8 + size 8 + type 4)。为什么这么较真?因为这是**和 BIOS/引导加载器之间的二进制接口(ABI)**, bootloader 就是按 20 字节往零页里塞的。内核这边结构体差一个字节,整个表就全错位了。这种编译期检查是"接口契约"的硬保证。
- **`x86_init.resources.memory_setup()`**:真正去把 BIOS 表读进来。这是个**函数指针**,默认指向 `e820__memory_setup_default()`,但虚拟机平台(比如 Xen、KVM 的特殊启动路径)可以替换它——这就是那句"platform quirk available for virtual platforms"的含义。
- **两个 `memcpy`**:刚读进来的主表,立刻原样复制一份给 `kexec` 表和 `firmware` 表。这对应前面说的"留原件"。注意时序——**先复制,后修改**:之后所有对主表的改动,都不会污染这两份原件。
- **`e820__print_table(who)`**:把内存图打印到 dmesg。你开机时能在 `dmesg` 开头看到的那些 `BIOS-e820: [mem 0x0000000000000000-0x000000000009fbff] usable` 行,就是这里打的。`who` 这个字符串("BIOS-e820"/"BIOS-88"...)标记这张表是哪种来源。

真正"读表"的活儿在 `e820__memory_setup_default()` 里([e820.c:1246](../linux-6.14/arch/x86/kernel/e820.c#L1246-L1277))。它先尝试 `append_e820_table()` 把零页里的 e820 表搬进内核主表;如果连这个都失败(BIOS 极度残废),就**伪造**一张最简陋的地图——只有 0–640KB 和 1MB–XMB 两段 RAM:

```c
if (append_e820_table(boot_params.e820_table, boot_params.e820_entries) < 0) {
	...
	e820_table->nr_entries = 0;
	e820__range_add(0, LOWMEMSIZE(), E820_TYPE_RAM);
	e820__range_add(HIGH_MEMORY, mem_size << 10, E820_TYPE_RAM);
}

/* We just appended a lot of ranges, sanitize the table: */
e820__update_table(e820_table);
```

注意最后那行 `e820__update_table()`——**不管表是从 BIOS 来的还是伪造的,都立刻清洗一遍**。这就是"信任但要核实"的落地:连正常路径都不放过消毒,何况伪造的。

### 6.2 `e820__update_table()`:那张"满是洞、会重叠"的脏表,是怎么被洗干净的

这是本章最精巧的函数。它解决的核心问题是:**给定一组可能重叠、无序、带各种类型的区间,重写成一张干净的无重叠表**。位于 [arch/x86/kernel/e820.c:322](../linux-6.14/arch/x86/kernel/e820.c#L322-L420)。

它的思路非常巧妙,值得逐段看:

**第一步:先把每个区间拆成"两个变化点"。**

```c
for (i = 0; i < table->nr_entries; i++)	{
	if (entries[i].size != 0) {
		change_point[chg_idx]->addr	= entries[i].addr;
		change_point[chg_idx++]->entry	= &entries[i];
		change_point[chg_idx]->addr	= entries[i].addr + entries[i].size;
		change_point[chg_idx++]->entry	= &entries[i];
	}
}
```

一段区间 `[addr, addr+size)` 被拆成两个端点:**起点**(这段开始生效)和**终点**(这段结束失效)。整张表就变成了一堆端点。

> **比喻**:不要把内存图看成"一段段的房间",而看成"一条数轴上一连串的开关事件"——在某点某个类型"上线"了,在另一点它"下线"了。

**第二步:把所有端点排序。**

```c
sort(change_point, chg_nr, sizeof(*change_point), cpcompare, NULL);
```

按地址从小到大排。排完之后,数轴上的事件就是有序的了。

**第三步:扫一遍排序后的端点,重建一张无重叠表。**

核心循环的逻辑是:维护一个"当前有哪些区间正处于激活状态"的集合(`overlap_list`)。扫到某个区间的起点,就把它加进集合;扫到终点,就移出。

```c
if (change_point[chg_idx]->addr == change_point[chg_idx]->entry->addr) {
	/* 起点:加入激活集合 */
	overlap_list[overlap_entries++] = change_point[chg_idx]->entry;
} else {
	/* 终点:从激活集合移除 */
	...
}

/*
 * 如果有多个区间激活(说明它们重叠),决定用哪个类型:
 * 类型值大的优先(1=可用, 2,3,4+=不可用)
 */
current_type = 0;
for (i = 0; i < overlap_entries; i++) {
	if (overlap_list[i]->type > current_type)
		current_type = overlap_list[i]->type;
}
```

这段就是"冲突仲裁"的灵魂。当多个区间在某段地址重叠时,**取类型值最大的那个**(也就是最"不可用"的那个)。所以 RAM(`1`)和 RESERVED(`2`)打架,RESERVED 赢——这段地址被判为不可用。**宁可少用一段 RAM,也绝不把设备寄存器当 RAM。** 这是个绝对正确的安全取向,也正是这个函数存在的根本理由。

**最后**:把重建好的干净表拷回原处,更新条目数。

```c
memcpy(entries, new_entries, new_nr_entries*sizeof(*entries));
table->nr_entries = new_nr_entries;
```

读完这个函数,你应该能体会到:Linux 对 BIOS 是多么的不信任。**它假设 BIOS 给的表可能脏到没法直接用,于是写了一整套"变化点 + 区间合并"的算法,把任何来源的脏表都规整成一份安全、无歧义的内部表。** 这套清洗是内核内存管理"安全第一"哲学的第一次亮相——后面 ZONE、伙伴系统、页表,都建立在这份"已经洗干净"的内存图之上。

### 6.3 `e820__memblock_setup()`:把干净的内存图,转交给下一章的主角

清洗完的 e820 表,最终要去哪?答案是:**变成 memblock(临时分配员)手里的可用内存池**。这就是从本章(e820)衔接到下一章(memblock)的桥梁。函数在 [arch/x86/kernel/e820.c:1300](../linux-6.14/arch/x86/kernel/e820.c#L1300-L1336):

```c
void __init e820__memblock_setup(void)
{
	int i;
	u64 end;

	memblock_allow_resize();

	for (i = 0; i < e820_table->nr_entries; i++) {
		struct e820_entry *entry = &e820_table->entries[i];

		end = entry->addr + entry->size;
		if (end != (resource_size_t)end)
			continue;

		if (entry->type == E820_TYPE_SOFT_RESERVED)
			memblock_reserve(entry->addr, entry->size);

		if (entry->type != E820_TYPE_RAM && entry->type != E820_TYPE_RESERVED_KERN)
			continue;

		memblock_add(entry->addr, entry->size);
	}

	/* Throw away partial pages: */
	memblock_trim_memory(PAGE_SIZE);

	memblock_dump_all();
}
```

这段代码把前面所有"为什么"收束成几行,值得逐句读:

- **遍历每一条 e820 区间**:现在这张表已经是干净、无重叠的了——感谢上一个函数。
- **`E820_TYPE_SOFT_RESERVED` 单独 reserve**:这是特殊用途内存,先标"已占用",免得 memblock 把它当普通 RAM 分配出去。
- **`if (entry->type != E820_TYPE_RAM ...) continue`**:**只有 RAM 类型的区间,才被加进可用内存池。** 显存、ACPI、设备 MMIO——所有那些"不是 RAM"的窟窿,就在这一行被默默跳过。整章的核心意图——"区分真房间和电梯井"——在这一行落地。
- **`memblock_add(entry->addr, entry->size)`**:把这段真 RAM 注册给 memblock。从这一刻起,**memblock 接过了"管内存"的接力棒**。
- **`memblock_trim_memory(PAGE_SIZE)`**:把不是整页的零头切掉——内存管理最小粒度是页(4KB),边角料没法用。

> 这是 e820 使命的终点:它把施工方的普查表,清洗、整理、过滤,最终变成一份"哪些是可用真 RAM"的清单,亲手交到 memblock 手里。然后它就基本退居二线了(后面只偶尔被查询、被 kexec 用)。

## 章末小结

### 用大楼比喻回顾本章

回到那栋楼。这一章讲的是**大楼通电后的第一件事**:施工方(BIOS)挨层挨间地普查,把"哪些是真房间、哪些是电梯井、哪些是配电室、哪些是档案室"写成一张清单(e820 内存图),递给刚到的物业管理员(内核)。

物业不会傻乎乎照单全收——他知道施工方的清单可能有错、有重叠、有遗漏。所以他做的第一件事是**清洗**:把清单排序、消除重叠、遇到"是房间还是设备"拿不准的地方,一律按"设备"处理(安全第一)。清洗完,他留两份原件底稿,然后用工作底稿开工:把所有"真房间"挑出来,登记到他的临时账本(memblock)上,准备开始分配。

整个过程,他**没有分配出一间房间**——这一章纯粹是"搞清楚手里有什么货"。

### 回扣全书主线

把这一章放回全书的那条主线——**物理一侧管切房间,虚拟一侧管造幻觉**:

- 本章**完全在物理这一侧**,而且是物理侧**最底层的一铲土**:连"切房间"都还没开始,只是先搞清楚"大楼里有几间房、哪些能住人"。
- e820 这张表,是后面**所有物理侧机制的输入**:ZONE 怎么分区、伙伴系统怎么分配、page fault 兑现时去哪要页——都建立在"我们已经知道哪些是真 RAM"这个前提上。没有这张干净的表,后面一切都是空中楼阁。
- 本章也埋了两条引线:
  - **引向下一章**:e820 表最终被 `e820__memblock_setup()` 转交给了 memblock。所以第 3、4 章会讲——内核怎么先把自己撑起来(head_64.S),memblock 又怎么当那个"临时分配员"。
  - **引向第 5 章**:e820 只告诉内核"哪些地址是真 RAM",但没说"真 RAM 之间有什么区别"。而有些 RAM 因为物理位置不同(16MB 以下、4GB 以下),对老设备有特殊意义——这就引出 ZONE 分区。那是后话。
