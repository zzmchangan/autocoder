# 第 1 篇 · 导言 —— 从一片漆黑到第一盏灯

> 这篇是本篇的"地图"。读各章之前先读它,你会对"一座城市是怎么开张的"建立一个结实的直觉。
> 全文只讲两件事:**启动到底在干什么**,以及**为什么非得这么干**。
> 我会继续用全书统一的比喻——**一台机器 = 一座城市,内核 = 市政府**——帮你记住每一步。

---

## 一、一句话说清启动在干嘛

> **启动(booting),就是这座城市从"一片漆黑、谁都没掌管",走到"市政府挂牌、各部门到岗、第一个市民诞生、第一盏灯亮起"的全过程。**

就这么一句话。后面所有的细节,都是在回答这中间的**两个问题**:

1. **市政府自己怎么进城、怎么给自己搭办公室?**(因为建城施工方 BIOS 把内核请进来时,内核连个分配器都没有)
2. **各部门按什么顺序挂上线、第一个市民怎么生出来?**(因为部门之间有依赖,顺序错了就启动不起来)

你任何时候看不懂启动里的某个机制,就回来问自己一个问题:

> **"这一步,是在'给自己搭脚手架'(自举/memblock),还是在'把某个部门挂上线'(start_kernel),还是在'生市民'(0 号进程)?"**

整个启动过程就这三个动作的反复变奏。记住这个三分法,你就永远不会迷路。

---

## 二、城市开张的时间线:六个时刻

我们把启动拆成**六个时刻**,从最黑的时候开始,一步步看城市怎么亮起来。

### 时刻 0:一片漆黑 —— BIOS 清点家底

你按下电源键。此刻 CPU 处在最笨的状态(只认识最低 1MB 地址),操作系统还躺在硬盘里。全城一片漆黑,谁都没掌管。

真正在管事的是 **BIOS/UEFI**——相当于"建城施工方"。它干两件事:

1. **清点这座城市有多少房子、哪些被设备占了**:挨根内存条去数,写成一张"满是窟窿的内存地图"(传统 BIOS 叫 **e820**)。
2. **把内核这个市政府请进城**:把内核镜像 `bzImage` 装进大楼一个角落,然后跳过去执行。

> **关键认知**:内核醒来时,手里只有施工方递来的一张**破破烂烂的城市地图**(满是洞,因为显存、BIOS、设备 MMIO 都占走了某些地址段)。它**还没有任何分配器、没有任何进程、没有任何部门在岗**。这就是启动的起点。

> 📖 详见第 2 章(⭐引用内存篇)。

### 时刻 1:搭工棚自举 —— 内核先把自己撑起来

内核(市政府)一进城,自己挤在大楼一个角落里,连办公室都没有。它的第一件事不是管别人,而是**先把自己撑起来**:

- 用汇编 + 一点点 C([arch/x86/kernel/head_64.S](../linux-6.14/arch/x86/kernel/head_64.S)),给自己搭一张**临时页表**,让自己从"笨模式(实模式)"切到"聪明模式(长模式)"。
- 这张临时页表只服务一件事:让内核自己的代码能正确运行。

> **比喻**:大楼刚通电,物业还没办公室。他们先在走廊里搭个临时工棚,先把"物业办公室"这个房间装修好、通电、能办公。装修期间用的都是临时材料。

这就是"自举(bootstrap)"——拎着自己的鞋带把自己提起来。

> **为什么这一步几乎不用 C?** 因为此时 C 的运行环境(栈、全局变量、堆)还没建好,能依赖的东西极少,只能用最底层的汇编。这跟"用临时的、丑的、但够用的方案换取可靠"是同一个工程哲学。

> 📖 详见第 3 章(⭐引用内存篇)。

### 时刻 2:临时工 memblock 上场 —— 熬过启动期

问题来了:**正式的资源分配部门(房屋管理局=伙伴系统)还没就位,但内核在启动过程中就要分配内存**(给页表、给 per-cpu 数据、给设备树建结构……)。这是个**鸡生蛋问题**:

> 伙伴系统需要内存才能初始化自己;但内存分配又只有伙伴系统这一个出口。怎么办?

答案是先上**临时工**:[memblock](../linux-6.14/mm/memblock.c)。

memblock 的设计哲学就四个字:**极简凑合**——不管单间,只记"哪些楼层空着、哪些被划走了";要分配就在空楼层划一段、记一笔。它纯粹是为了熬过启动期。等伙伴系统盖好,memblock 把手里剩下的空房间**全部移交**,功成身退(工棚拆除)。

> **为什么这么设计?** 启动期需求又急又杂,正儿八经写个通用分配器太重、还可能有 bug。memblock 用几百行代码就解决了"先凑合用"的问题,把复杂性留给真正该管事的伙伴系统。这是"用临时的、丑的、但够用的方案,换取启动可靠"的典型取舍。

> 📖 详见第 4 章(⭐引用内存篇)。

### 时刻 3:start_kernel —— 初始化大游行

memblock 在场、内核能跑 C 代码了,接下来是整个启动最壮观的一幕:**初始化大游行**。

`head_64.S` 把内核带进长模式后,会跳到一段 C 代码([arch/x86/kernel/head64.c](../linux-6.14/arch/x86/kernel/head64.c)),最终调用 [init/main.c](../linux-6.14/init/main.c) 里的 **`start_kernel()`**——这是一个**两百多行、几乎纯线性**的函数,一行调用一个部门,把内核所有子系统一个个挂起来:

```
关中断 → 建陷阱表 → 建内存(伙伴系统) → 建调度器
→ 建 RCU → 建中断 → 建时钟 → 开中断
→ 建控制台(终于能打印!) → 建 VFS → 建 cgroup → ...
```

为什么是**线性**的?因为部门之间有**严格的依赖**:

- **调度器必须先建好**,否则后面创建进程时无处安放;
- **内存(伙伴系统)必须先就位**,否则后面哪个部门要分配内存都拿不到;
- **中断必须先关掉**,因为中断处理依赖的数据结构(IRQ 描述符)还没建,这时候来一个中断就崩;等数据结构就位了才敢开中断。

> **比喻**:就像新城市开张,得先通水、再通电、再通气——顺序错了,后面的部门没水没电就开不了张。`start_kernel` 这条线,本质上是**一张被依赖关系严格约束的施工进度表**。

大游行的**最后**还有一招**群众智慧**:**initcall 机制**。成百上千个驱动的初始化函数,内核不会一个个手写调用,而是让驱动作者用 `module_init`/`core_initcall`/`late_initcall` 这样的宏**自我登记**,内核在游行末尾按 `pure → core → postcore → arch → subsys → fs → device → late` 八个等级**逐级批量调用**。等级就是依赖:`core` 级的驱动依赖的东西少,先跑;`device` 级的驱动要等总线就位,后跑。

> **为什么用 initcall 而不是手写一张大调用表?** 因为内核是模块化的,你编译时塞了多少驱动是动态的,不可能写死一张表。initcall 让每个驱动**自己登记自己的初始化函数**,链接器按等级把它们排进对应的段,内核运行时扫段调用——这是"用一条链表达全局依赖"的工程范本。

> 📖 详见第 5 章(★本篇新写)。

### 时刻 4:亲手生市民 —— rest_init 与 0/1/2 号进程

大游行走完,市政府挂了牌、各部门到岗,可城里**一个市民都还没有**。市民(进程)不能凭空冒出来——必须有人"生"。

于是 `start_kernel` 的**最后一行**调用 [`rest_init()`](../linux-6.14/init/main.c#L697),字面意思"剩下的初始化"。它干一件开天辟地的事:**亲手生出三尊祖宗**:

| PID | 名字 | 谁生的 | 干什么的 | 寿命 |
|-----|------|--------|----------|------|
| **0** | idle / swapper | 没人——**编译期写死** | 全城祖先;没活干时永远睡觉候着 | **永远不死** |
| **1** | init | 0 号生的(`user_mode_thread`) | 所有**用户程序**的祖先 | init 死 = 城市崩 |
| **2** | kthreadd | 0 号生的(`kernel_thread`) | 所有**内核线程**的祖先 | 跟城市同寿 |

这里有个精妙的顺序问题:**为什么 1 号 init 要先于 2 号 kthreadd 出生?** 不是因为 init 更重要,而是因为**1 号要抢 PID 1 这个号**。如果 kthreadd 先生,它就占了 PID 1,那 init 只能拿 PID 2——而 Unix 世界有个铁律:**PID 1 必须是 init**。所以代码里特意先 `user_mode_thread(kernel_init)`、再 `kernel_thread(kthreadd)`,注释里写得很直白:

> "We need to spawn init first so that it obtains pid 1"

> **比喻**:开张那天,市政府做了三件事:① 拿出一份**早就刻在奠基石里的"荣誉市民"档案**(0 号 idle,祖先);② 亲手给这份档案**生出第一个真正意义上的市民**(1 号 init);③ 再生一个**专门负责"招募所有政府临时工"的市民**(2 号 kthreadd)。从此,城里的市民开始繁衍。

> 📖 详见第 6 章(★本篇新写)。

### 时刻 5:第一盏灯 —— init 跑起用户空间

最后,1 号进程 `init` 在 `kernel_init` 里干一件决定性的事:**从内核线程脱胎成用户进程**——执行 `kernel_execve("/sbin/init")`(或 `/etc/init`、`/bin/init`、最次 `/bin/sh`),把一个用户态程序加载进自己的地址空间,从此**它不再生活在内核里,而是成了一个堂堂正正的用户态市民**。

从这一刻起:

- init 接管了启动的下半场(读配置、启动服务、给你一个登录界面);
- 0 号 idle 退居幕后,**没活干时永远在 `do_idle` 里睡觉**,有活被调度器唤醒;
- 2 号 kthreadd **永远候着**,谁想建内核线程(`kthread_create`)都排队找它代生。

城市,灯火通明。

## 三、收束:启动的本质是"一条线"

走完全程,你会发现启动这件事,本质上就是**一条线**:

```
BIOS ──→ 自举 ──→ memblock ──→ start_kernel(大游行)──→ rest_init(生市民)
 黑       工棚      临时工         各部门挂牌              0/1/2 号祖宗
```

这条线上每一个点,**都被它前一个点决定**:

- 没有自举搭的工棚,内核跑不了 C 代码;
- 没有 memblock,内核在启动期分配不了内存;
- 没有 start_kernel 挂上的各部门,创建进程时没有调度器可挂;
- 没有 rest_init 生出的 init,用户空间永远起不来。

> **为什么启动这么"线性"、不能并行?** 因为依赖链是单向的、刚性的。启动期的内核是个**脆弱的婴儿**:每一步都假设前面的步骤已完成,任何一步抢跑都会用到还没初始化的数据结构,直接 panic。所以启动代码宁可慢、宁可丑(一长串顺序调用),也要保证**可读、可预测、不出错**。这是"可靠优先于性能"的工程哲学在启动期的极致体现。

### 记住这三句话,就抓住了本篇

1. **启动 = 自举 + 大游行 + 生市民。** 任何时候迷路,回到这个三分法:这一步是搭脚手架、挂部门,还是生市民?
2. **启动是线性的,因为依赖是刚性的。** 每一步都被前一步决定;顺序错了就 panic。这不是缺陷,而是可靠性的代价。
3. **0 号进程是刻在奠基石里的,1 号和 2 号是它生的。** 所有进程都是这三个祖宗的后代;杀掉 init = 杀掉整座城市。
