# 第 2 章(全书第 21 章)· 时钟与定时器:城市的心跳

> **前置**:你需要先读过 [第 1 章 中断与异常](P4-01-中断与异常-硬件怎么拍政府的肩膀.md)。本章是第 1 章里那个"最特殊的门铃"的深挖——时钟中断就是一个普通的硬件中断,但它特殊到值得单开一章:它同时驱动着**调度器**和**所有定时器**,是全城的"心跳"。
>
> **核心问题**:进程凭什么会"被调度走"?`sleep(1)` 之后进程凭什么能准时醒来?`select` 设了 100ms 超时,内核怎么知道时间到了?这一切,都依赖那个每秒跳几百次的中断。
>
> **读完本章你会明白**:
> - 时钟中断为什么是"全城心跳"——它周期性地打断 CPU,内核借此驱动调度、定时器、记账三件事;
> - 一条从硬件到软件的完整链:`sysvec_apic_timer_interrupt` → `tick_sched_handle` → `update_process_times`,一次心跳怎么扇出到 `sched_tick` / `run_local_timers` / RCU / POSIX 定时器;
> - 为什么老的 jiffies"滴答轮"被 hrtimer(高精度定时器)取代,以及"按需编程下一次中断"为什么比"傻乎乎每秒固定跳"高效;
> - NO_HZ / tickless:为什么空闲时内核要**主动停掉心跳**省电。

> **如果一读觉得太绕**:先只记住三件事——① **时钟中断 = 一个周期性响的门铃**,它驱动 `update_process_times`,后者扇出调度(`sched_tick`)、低精度定时器(`run_local_timers`)、记账等;② **hrtimer 把"固定频率心跳"升级成"只在下一个定时器到期时才响一次"**,更省电、更精确;③ **空闲时内核会 tickless 停掉心跳**。其余链路细节第二遍配合源码再抠。

## 一、为什么需要"心跳"

### 场景

回忆进程篇(以及 [第 1 章](P4-01-中断与异常-硬件怎么拍政府的肩膀.md)):调度器要让每个进程都觉得"我独占了 CPU",办法是飞速轮换。可是——**调度器自己,凭什么会被触发?**

进程 A 正在跑一段死循环 `while(1);`。它绝不会主动让出 CPU(它又没调 `sleep`,也没做系统调用)。如果没有外界干预,**进程 A 会永远霸占这个 CPU**,进程 B 永远得不到执行。

谁来当这个"外界干预"?必须是**一个进程自己控制不了、能强制打断它的力量**——时钟中断。

> ⚠️ **时钟中断是"抢占式调度"得以存在的物理基础。** 没有它,一个不主动让出的进程能永久占住 CPU。这正是它为什么叫"心跳"——全城的活动节奏,都由它一下一下地敲出来。

### 不这样会怎样

**不这样会怎样?** 如果没有时钟中断:

- **抢占失效**:不主动 `sleep` / 不做系统调用的进程永久霸占 CPU。你开个 `while(1)` 死循环,整个核冻死,连 shell 都切不过去。
- **定时器全废**:`sleep(1)` 永远醒不过来——没人来告诉内核"1 秒到了"。`select`/`poll` 的超时形同虚设。TCP 重传、keepalive 全乱套。
- **记账归零**:CPU 时间花在谁身上了?进程跑了多久?没有周期性的"滴答"采样,这些统计无从谈起。

所以时钟中断**必须**存在,而且**必须周期性地**来——它是内核里几乎所有"和时间流逝有关"的事的**唯一触发源**。

### 它从哪儿来

x86 上,时钟中断来自 **Local APIC 内置的定时器**(每个 CPU 一个,独立)。CPU 启动时给 Local APIC 编程一个周期(比如每 4ms 触发一次,对应 250 HZ),之后它会**自动、周期性地**给本 CPU 发中断。

> 配置项 `CONFIG_HZ` 决定了心跳频率:服务器常设 1000 HZ(每毫秒一跳,响应快、但中断开销大),桌面/嵌入式常设 100~250 HZ(省电、省开销)。这是"响应灵敏度" vs "中断开销"的经典权衡。

它和第 1 章讲的外设中断**没有任何本质区别**——同样是硬件发信号、查 IDT、过门、分诊。它的"特殊"完全在于:它是**自己人**(Local APIC 在 CPU 内部,不经过 I/O APIC)、**周期性**(不停响)、且**每个核各跳各的**(per-CPU)。下面我们顺着它从硬件一路钻到软件。

## 二、从中断到 `update_process_times`:一次心跳的完整链路

这是本章的主干。一次时钟中断,从硬件响起到扇出所有软件处理,经过这条链:

```
Local APIC 定时器 (硬件,周期性)
      │ 发 vector
      ▼
sysvec_apic_timer_interrupt        ← 第 1 章那个 DEFINE_IDTENTRY_SYSVEC 入口
      │ apic_eoi() + local_apic_timer_interrupt()
      ▼
event_handler (tick 设备的回调) ─┐
      │                        │ 两种模式之一:
      ▼                        │  ① 周期模式 tick_handle_periodic
tick_sched_handle (高精度/NOHZ) │  ② 高精度模式 hrtimer_interrupt → tick_sched_timer
      │                        │
      ▼                        ┘
update_process_times  ← 这一句是"扇出点"
      ├── account_process_tick  (记账:这次滴答算用户态还是内核态时间)
      ├── run_local_timers      (低精度定时器:老 jiffies 轮)
      ├── rcu_sched_clock_irq   (RCU 的周期性推进)
      ├── sched_tick            (调度器:谁该让出 CPU ← 抢占的心脏!)
      └── run_posix_cpu_timers  (POSIX CPU 时间限制:超时杀进程)
```

下面我们一段段走,看每一段为什么存在。

### 第 1 段:中断入口

时钟中断的 IDT 入口,就是第 1 章那个宏 `DEFINE_IDTENTRY_SYSVEC` 生成的:

> [arch/x86/kernel/apic/apic.c:1049-1059](../linux-6.14/arch/x86/kernel/apic/apic.c#L1049-L1059)

```c
DEFINE_IDTENTRY_SYSVEC(sysvec_apic_timer_interrupt)
{
	struct pt_regs *old_regs = set_irq_regs(regs);

	apic_eoi();                          /* 应答 Local APIC,否则下一个心跳不来 */
	trace_local_timer_entry(LOCAL_TIMER_VECTOR);
	local_apic_timer_interrupt();        /* 真正的处理,往下钻 */
	trace_local_timer_exit(LOCAL_TIMER_VECTOR);

	set_irq_regs(old_regs);
}
```

`DEFINE_IDTENTRY_SYSVEC` 已经替我们包好了"进门骨架"(`irqentry_enter`/`irqentry_exit`,见 [第 1 章](P4-01-中断与异常-硬件怎么拍政府的肩膀.md)精读一)。这里业务逻辑只有两件事:**`apic_eoi()` 应答硬件**(不 EOI,心跳停摆)、**`local_apic_timer_interrupt()` 转发**。

> Local APIC 定时器在内核里被抽象成一个 **clock_event_device**(时钟事件设备)。`local_apic_timer_interrupt` 最终会调这个设备的 `event_handler` 回调。这个回调**不是固定的**——它取决于内核运行在哪种"滴答模式"。下一段解释。

### 第 2 段:两种心跳模式——周期 vs 高精度

历史上时钟心跳只有一种模式:**周期模式(periodic)**。Local APIC 被编成"每 1/HZ 秒固定触发一次",回调是 `tick_handle_periodic`:

> [kernel/time/tick-common.c:108-118](../linux-6.14/kernel/time/tick-common.c#L108-L118)

```c
void tick_handle_periodic(struct clock_event_device *dev)
{
	int cpu = smp_processor_id();
	ktime_t next = dev->next_event;

	tick_periodic(cpu);
	...
}
```

它干的就是"每次心跳 → `tick_periodic` → 推进 jiffies → 调 `update_process_times`"。简单粗暴,但有个浪费:**哪怕没有任何定时器到期、进程不需要被调度,心跳照样每秒跳 1000 次**。空闲的 CPU 也在不停被中断、不停耗电。

现代内核(几乎所有发行版默认)用**高精度模式(HIGHRES)**取代它。高精度模式的核心思想是:

> **不再"固定频率跳",而是"只在下一个定时器该到期的时候,才让 APIC 响一次"。**

具体做法:Local APIC 被编成 **oneshot(单次)模式**——你告诉它"X 纳秒后响一次",它响完就停,等你下次再编。内核用 **hrtimer(高精度定时器)** 管理所有定时器,在 `hrtimer_interrupt` 里找出最近到期的、重编下一次中断:

> [kernel/time/hrtimer.c:1897-1927](../linux-6.14/kernel/time/hrtimer.c#L1897-L1927)(精简)

```c
void hrtimer_interrupt(struct clock_event_device *dev)
{
	struct hrtimer_cpu_base *cpu_base = this_cpu_ptr(&hrtimer_bases);
	...
	raw_spin_lock_irqsave(&cpu_base->lock, flags);
	entry_time = now = hrtimer_update_base(cpu_base);
retry:
	...
	if (!ktime_before(now, cpu_base->softirq_expires_next)) {
		...
		raise_timer_softirq(HRTIMER_SOFTIRQ);   /* 软定时器到期,丢给软中断 */
	}

	__hrtimer_run_queues(cpu_base, now, flags, HRTIMER_ACTIVE_HARD);  /* 跑到期的硬定时器 */
	...
}
```

高精度模式下,**"滴答"本身也变成了一个 hrtimer**——叫 `tick_sched_timer`。它到期时回调 `tick_nohz_handler`,最终一样走到 `tick_sched_handle` → `update_process_times`。也就是说:高精度模式并没有取消心跳,只是把"固定频率"换成"按需编程",中间顺便支持了纳秒级精度、还能在空闲时停跳(NO_HZ)。

> [kernel/time/tick-sched.c:253-272](../linux-6.14/kernel/time/tick-sched.c#L253-L272) 的 `tick_sched_handle` 是两种模式共同的落脚点:

```c
static void tick_sched_handle(struct tick_sched *ts, struct pt_regs *regs)
{
	if (IS_ENABLED(CONFIG_NO_HZ_COMMON) &&
	    tick_sched_flag_test(ts, TS_FLAG_STOPTED)) {
		touch_softlockup_watchdog_sched();   /* 空闲时心跳被停了,喂一下软锁检测狗 */
		...
	}

	update_process_times(user_mode(regs));    /* ← 扇出点!所有软件处理从这里开始 */
	profile_tick(CPU_PROFILING);
}
```

不管哪种模式,最终都汇聚到 `update_process_times(user_mode(regs))`。这一句是整个时钟子系统最重要的"扇出点"。注意它的参数 `user_mode(regs)`——它判断"这次心跳打断时,CPU 在用户态还是内核态",用于后面记账。

### 第 3 段:`update_process_times`——心跳的扇出点

> [kernel/time/timer.c:2508-2522](../linux-6.14/kernel/time/timer.c#L2508-L2522)

```c
void update_process_times(int user_tick)
{
	struct task_struct *p = current;

	/* Note: this timer irq context must be accounted for as well. */
	account_process_tick(p, user_tick);     /* ① 记账 */
	run_local_timers();                     /* ② 低精度定时器 */
	rcu_sched_clock_irq(user_tick);         /* ③ RCU 推进 */
#ifdef CONFIG_IRQ_WORK
	if (in_irq())
		irq_work_tick();                /* ④ irq_work(轻量级"立刻给我跑个函数") */
#endif
	sched_tick();                           /* ⑤ 调度器 */
	if (IS_ENABLED(CONFIG_POSIX_TIMERS))
		run_posix_cpu_timers();         /* ⑥ POSIX CPU 定时器 */
}
```

短短六行,扇出了内核里几乎所有"需要定期被戳一下"的子系统。挨个看为什么它们都需要心跳:

1. **`account_process_tick`——记账**。这次心跳打断时,进程在用户态还是内核态(`user_tick`)?把这一 tick 的时间记到对账上。这就是 `top` 里 `us`/`sy` 那两列数字的来源。
2. **`run_local_timers`——低精度定时器**。内核里老的 `timer_list`(那些"大概几百毫秒后跑一下就行、不要求精度"的定时器)由它驱动。它内部把到期的定时器丢给 `TIMER_SOFTIRQ` 软中断去跑(回扣第 1 章软中断)。
3. **`rcu_sched_clock_irq`——RCU 推进**。RCU(读多写少的极致同步机制,同步篇主角)需要周期性地"推进宽限期",看看哪些读者已经退出。这个推进靠心跳驱动。
4. **`sched_tick`——调度器**。**这是和进程篇最直接的联系**。它检查"当前进程的时间片用完了吗?",用完了就设一个"需要重新调度"的标志(`TIF_NEED_RESCHED`),等中断返回用户态时,调度器趁机换人。这就是抢占的本质。下一节细讲。
5. **`run_posix_cpu_timers`——POSIX CPU 定时器**。比如进程设了"我用 CPU 时间超过 5 秒就杀我"(`setitimer` 的 PROF 模式),靠这里检查并发信号。

> **一个关键认知**:这六件事**共享同一个心跳**。内核没有为每件事单独设一个定时器(那样太浪费中断),而是让一个时钟中断扇出服务所有"需要周期性被戳"的子系统。这是"用一次中断代价,办六件事"的集约化设计——和第 1 章"一次 `irq_exit` 顺手跑掉所有挂起软中断"是同一种思路。

## 三、`sched_tick`:心跳怎么驱动调度

`sched_tick` 是这条链上最关键的一环,它把"心跳"和"进程篇的调度器"焊接在一起。

> [kernel/sched/core.c:5634](../linux-6.14/kernel/sched/core.c#L5634) 是它的定义。

它干的活(精炼地说):

- 拿到当前 CPU 的运行队列(`rq`)和当前进程;
- 更新当前进程的 `vruntime`(CFS 的"虚拟运行时间",进程篇讲过——这是公平调度的核心量);
- 检查当前进程的**时间片用完了吗**;
- 用完了 → 设置 `TIF_NEED_RESCHED` 标志,表示"这个核该重新调度了"。

**注意**:它只是**设标志**,不直接换人。因为此刻还在中断上下文里(第 1 章说过,硬中断不能调度)。真正的换人,发生在中断返回的路径上——`irqentry_exit` 发现"有 `TIF_NEED_RESCHED` 且要回用户态",才调用调度器 `schedule()` 换人。

> **回扣比喻**:心跳每跳一下,调度器(户籍科的叫号员)就被戳一下"看看现在用窗口的人是不是用够时间了"。用够了,就在他桌上贴张条(`TIF_NEED_RESCHED`)。等这个人办完手头这一下、要回自己房间(回用户态)时,看见桌上的条子,叫号员才真的让他让位、叫下一位。**心跳负责"到点提醒",真正的换人在"自然的出口处"完成**——这就是"延迟到安全点才调度"的设计。

> 这一段也回答了进程篇留下的一个问题:**"时间片"是按什么节奏递减的?** 答案就是时钟中断。每次心跳,`sched_tick` 给当前进程的配额减一。减到 0,设 `TIF_NEED_RESCHED`。

## 四、定时器:从 jiffies 轮到 hrtimer

心跳的另一大职责是驱动**定时器**——`sleep(1)`、`select` 超时、TCP 重传,底层都是"在某个时间点之后跑一个回调"。内核有两套定时器子系统,历史地叠在一起:

### 老式低精度定时器:`timer_list`(jiffies 轮)

> [include/linux/timer_types.h:8](../linux-6.14/include/linux/timer_types.h#L8) 定义 `struct timer_list`。

老内核只有这套。它的精度是**一个滴答**(1/HZ 秒,比如 4ms)。工作原理像"轮盘":所有定时器按"还剩多少个 tick"挂到轮子的不同槽位,每来一个心跳(`run_local_timers`),轮子转一格,到期的那批就被丢给 `TIMER_SOFTIRQ` 执行。

**为什么不淘汰它?** 因为它**轻量**。内核里有大量"几百毫秒后跑一下就行、对精度毫无要求"的活儿(比如网卡看门狗、TCP 的某些慢路径)。为这些用重量级的 hrtimer 不划算。所以 `timer_list` 保留至今,由 `run_local_timers` 驱动。

### 新式高精度定时器:hrtimer

> [include/linux/hrtimer_types.h:39-50](../linux-6.14/include/linux/hrtimer_types.h#L39-L50)

```c
struct hrtimer {
	struct timerqueue_node		node;          /* 红黑树节点,按到期时间排序 */
	ktime_t				_softexpires;  /* 软到期时间(可被打折) */
	enum hrtimer_restart		(*function)(struct hrtimer *);  /* 到期回调 */
	struct hrtimer_clock_base	*base;         /* 属于哪个时钟基(CLOCK_MONOTONIC 等) */
	u8				state;
	u8				is_rel;
	u8				is_soft;       /* 软定时器(在软中断里跑回调) */
	u8				is_hard;       /* 硬定时器(在硬中断里跑回调) */
};
```

hrtimer 是本章真正的"现代主角"。它和 `timer_list` 有三个根本区别:

1. **精度纳秒级**。基于 `ktime_t`(纳秒),不受 1/HZ 滴答粒度限制。`nanosleep`、`timerfd`、pthread 的 `cond_timedwait` 都靠它。
2. **组织方式是红黑树**,不是轮盘。所有 hrtimer 按"到期时间"挂在红黑树上——**树的最左节点就是最早到期的那个**。这样 `hrtimer_interrupt` 检查"哪些到期了"只要从最左节点开始摘,O(log N)。
3. **驱动方式是"按需编程中断"**。这正是第二节讲的:内核每次把"最近一个 hrtimer 的到期时间"编进 Local APIC,APIC oneshot 模式下到点响一次。没有 hrtimer 到期,就不编程,APIC 不响。

> **为什么红黑树?** 因为 hrtimer 数量可能很多(一个 busy 的系统几千个),而且要频繁"插入新 timer / 删除到期 timer / 找最早到期的"。红黑树在这三个操作上都是 O(log N),是这种工作负载的合适选择。和内存篇 VMA 用红黑树(那里是"按地址查找一段")是同一种数据结构,但服务于不同目的——一个是"找区间",一个是"找最早"。

### 不这样会怎样:为什么从滴答轮进化到 hrtimer

**不这样会怎样?** 如果只有 jiffies 滴答轮:

- **精度差**:你想 `nanosleep(1ms)` 但 HZ=250(每 4ms 一跳),实际可能睡 4ms 才醒。对实时应用、媒体同步、高精度网络协议,这不可接受。
- **能耗高**:为了"不错过任何定时器",滴答必须以足够高的频率固定跳(比如 1000 HZ)。可 CPU 空闲时也在被每毫秒一次的心跳打醒,根本进不了深睡省电状态。
- **扩展性差**:轮盘的大小和 HZ 绑定,定时器跨度大(有的 1ms 后到期、有的 1 小时后到期)时轮子要么太大要么浪费。

hrtimer + oneshot 编程一举解决:精度纳秒级、空闲时不跳(NO_HZ)、跨度和性能都由红黑树保证。这是"按需"对"固定"的胜利,和内存篇"按需分页(page fault 才兑现)"是同一种"懒"哲学——**没活儿的时候就别瞎忙**。

## 五、NO_HZ / tickless:空闲时停掉心跳

hrtimer 那套"按需编程中断",带来一个意外的巨大收益:**如果当前没有任何定时器要到期、当前进程也不需要被调度(它睡了或者 CPU 空闲),那心跳根本不用跳**。

这就是 **NO_HZ(也叫 tickless)**:

- CPU 空闲进入 idle 时,内核**主动停掉本 CPU 的周期心跳**(把 `tick_sched` 置 `TS_FLAG_STOPPED`)。
- 直到"有定时器到期 / 有进程要跑 / 有中断来"时,才重新编程心跳把它唤醒。

**不这样会怎样?** 没停掉心跳的话,一台 64 核的服务器,即使全部空闲,每毫秒还要被 64 次时钟中断打醒一次(每个核各跳一次)。这些中断:

- 把 CPU 从深度省电状态(C-states)里拽出来,白白耗电;
- 缓存被打扰,影响刚被唤醒的真实工作负载。

数据中心里,NO_HZ 能省下非常可观的电。笔记本待机时,它能显著延长电池。所以本章开头那条链里,`tick_sched_handle` 有这么一段:

```c
	if (IS_ENABLED(CONFIG_NO_HZ_COMMON) &&
	    tick_sched_flag_test(ts, TS_FLAG_STOPPED)) {
		touch_softlockup_watchdog_sched();   /* 心跳停了,但软锁检测狗还得喂,免得误报"CPU 卡死" */
		...
	}
```

> 注意:停掉心跳不能停太久——至少**软锁检测器(softlockup watchdog)**和**RCU 宽限期推进**还需要周期性被打戳。所以即使 NO_HZ,内核也会保留一个"最晚多久必须跳一次"的底线。这是"省电"和"内核子系统仍需周期维护"之间的折中。

## 六、关键源码精读:`update_process_times`

本章的链路很多,但真正"扇出一切"的就是 `update_process_times`。我们把它再读细一点,看清六件事的轻重缓急:

> [kernel/time/timer.c:2508-2522](../linux-6.14/kernel/time/timer.c#L2508-L2522)

```c
void update_process_times(int user_tick)
{
	struct task_struct *p = current;

	/* Note: this timer irq context must be accounted for as well. */
	account_process_tick(p, user_tick);     /* ① 记账 */
	run_local_timers();                     /* ② 低精度定时器 → 软中断 */
	rcu_sched_clock_irq(user_tick);         /* ③ RCU */
#ifdef CONFIG_IRQ_WORK
	if (in_irq())
		irq_work_tick();                /* ④ irq_work */
#endif
	sched_tick();                           /* ⑤ 调度器(可能设 TIF_NEED_RESCHED) */
	if (IS_ENABLED(CONFIG_POSIX_TIMERS))
		run_posix_cpu_timers();         /* ⑥ POSIX CPU 定时器 */
}
```

几个值得琢磨的细节:

- **`current` 就是"被心跳打断的那个进程"**。注意这句隐含的事实:时钟中断关联"当前进程"(像异常,不像外设中断——见第 4 章统一视角)。`account_process_tick` 把这一 tick 记到 `current` 头上;`sched_tick` 检查的也是 `current` 的时间片。所以时钟中断虽然走的是"中断"入口,但它**深深地绑在当前进程上**——这是它和网卡中断的本质区别之一。
- **顺序有讲究**:记账(`account_process_tick`)在最前,因为后面任何一步都可能改变状态,先把"这次滴答算谁的"记准。`sched_tick` 在后段,因为它设的 `TIF_NEED_RESCHED` 要等中断返回才生效,放后面无妨。`run_local_timers` 把老定时器丢给软中断,真正跑在 `irq_exit` 的软中断阶段(回扣 [第 1 章](P4-01-中断与异常-硬件怎么拍政府的肩膀.md))。
- **这六件事都在硬中断上下文里跑**(快速、不睡眠)。费时的部分(老定时器回调、RCU 回收)被 `raise_*_softirq` 推迟到下半部。这是第 1 章"上半部/下半部"在时钟子系统的具体应用:心跳这一下只做"戳一下各子系统",重活丢给软中断。

> **一句话**:`update_process_times` 是时钟中断的"分发台"——一次心跳,扇出记账/定时器/RCU/调度/POSIX 六件事,重活推迟到下半部。它体现了"用一个共享中断源,集约化服务所有需要周期性维护的子系统"。

## 七、章末小结

用"钟楼"的比喻把这一章收口。

[全书导言](全书导言-一篇看懂内核全貌.md)里说,时钟中断是**"城市钟楼"**——它一下一下地敲,全城的节奏由它定。这一章我们钻进钟楼,看清了它从硬件响起到扇出软件的完整链路。

完整旅程:

1. **Local APIC 定时器**周期性(或 oneshot 按需)发中断 → vector → IDT 入口 `sysvec_apic_timer_interrupt`;
2. 入口 `apic_eoi` 应答硬件,转 `local_apic_timer_interrupt` → clock_event_device 的 `event_handler`;
3. 高精度模式走 `hrtimer_interrupt`(红黑树找到期 timer),周期模式走 `tick_handle_periodic`,两者最终都到 `tick_sched_handle`;
4. **`tick_sched_handle` → `update_process_times`**:扇出记账、低精度定时器、RCU、`sched_tick`(调度)、POSIX 定时器六件事;
5. `sched_tick` 给当前进程的配额减一,用完设 `TIF_NEED_RESCHED`,等中断返回用户态时调度器才真换人。

而这一章真正要带走的三个设计:

> **① 时钟中断是"抢占式调度"和"所有定时器"的共同心跳。** 没有它,一个不主动让出的进程能永久霸占 CPU,`sleep` 永远醒不过来。它是"内核何时介入"在时间维度的化身。
>
> **② 从"固定频率滴答"进化到"hrtimer + oneshot 按需编程",是"按需"对"固定"的胜利。** 红黑树组织、纳秒精度、空闲停跳(NO_HZ)——和内存篇 page fault 的"懒兑现"同一种哲学:没活儿就别瞎忙。
>
> **③ 一次心跳扇出六件事,重活推迟到下半部。** 这是"用一次中断代价集约化服务所有周期性需求"+第 1 章"上半部/下半部分离"的结合。

> **回扣全书主线**:这一章是**中断(第 1 章)的特例深挖**,也是**进程篇调度器的物理基础**。进程篇讲了 CFS、vruntime、时间片,但"时间片按什么节奏递减、到点了谁去检查"——答案就是 `sched_tick`,而它由时钟心跳驱动。**调度和定时器,在这一个中断上汇合。**
>
> - **承前**:它直接消费第 1 章的整个中断基础设施(IDT 入口、软中断下半部),并回答进程篇留下的"抢占从何而来"。
> - **启后**:它埋了同步篇的雷——`sched_tick` 设的 `TIF_NEED_RESCHED` 会触发抢占,而"中断里设标志、返回时才调度"这个延迟点,正是同步篇要讲的"抢占也是一路并发"的关键。

### 记住这三句话

1. **时钟中断是全城心跳**,它驱动调度(`sched_tick`)、定时器、记账;没有它,抢占失效、`sleep` 醒不过来。
2. **hrtimer 把"固定频率"换成"按需编程下一次中断"**,精度纳秒级,还让空闲时停跳(NO_HHZ)省电成为可能。
3. **一次心跳扇出六件事**(`update_process_times`),费时部分推迟到软中断下半部——集约化 + 上半部/下半部。
