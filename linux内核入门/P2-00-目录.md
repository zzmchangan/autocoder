## 这本书的"一条主线"

整本书只讲一件事:

> **一片有限的、被所有人共享的物理 RAM,内核是如何在它之上凭空搭建出层层抽象,既隔离又高效地服务所有人。**

所有章节都挂在这条主线上。如果你在某一章迷路了,回到这句话,问自己:**"这一步是在解决'物理资源怎么切',还是'虚拟幻象怎么造'?"** 就能找回方向。

## 目录

### 第 1 章 · 第一性原理:内存管理到底在干什么

- **核心问题**:为什么要专门搞一个"内存管理"子系统?它到底解决了什么不解决就不行的问题?
- **要讲清的"为什么"**:
  - RAM 的四个"麻烦本性":唯一且平铺、有限、慢、有物理拓扑。
  - 由此推出的四个根本需求:隔离、保护、按需、高效。
  - 为什么这些需求彼此打架,逼出了"分层抽象"这个唯一可行的答案。
- **读完你会明白**:为什么后面所有复杂设计都不是"过度工程",而是被这四个本性逼出来的。

### 第 2 章 · 白纸阶段:上电、BIOS 与 e820 地图

- **核心问题**:机器刚通电,内核还没跑,谁在管内存?内核醒来时,它看到的是一份什么样的"使用说明书"?
- **要讲清的"为什么"**:
  - 为什么内存地图不是"从 0 开始连续 N 字节",而是 holes 满天飞(被 BIOS、显存、设备 MMIO 占走)。
  - 为什么内核不能直接信任"有多少 RAM",而要靠 BIOS 传 e820/UEFI 内存图。
  - 实模式 → 保护模式 → 长模式,为什么内存访问方式要这样切换。
- **关键源码**:`e820_table`、`e820__memory_setup()`([arch/x86/kernel/setup.c](../linux-6.14/arch/x86/kernel/setup.c))。

### 第 3 章 · 内核自举:head_64.S 与第一张页表

- **核心问题**:内核自己也是个程序,它启动时连个像样的内存分配器都没有,它是怎么"把自己撑起来"的?
- **要讲清的"为什么"**:
  - 为什么需要"身份映射(identity map)"的早期页表。
  - 为什么这段代码几乎不用 C,要用汇编 + 极少 C(head64.c)。
  - "鸡生蛋"问题:分配器需要内存来初始化自己,内存从哪来——为下一章的 memblock 埋下伏笔。
- **关键源码**:[arch/x86/kernel/head_64.S](../linux-6.14/arch/x86/kernel/head_64.S)、[arch/x86/kernel/head64.c](../linux-6.14/arch/x86/kernel/head64.c)。

### 第 4 章 · memblock:启动期的临时分配员

- **核心问题**:正式的伙伴系统还没建好,但内核启动过程中就要分配内存(建页表、per-cpu、设备树……),怎么办?
- **要讲清的"为什么"**:
  - 为什么 memblock 不管理单个页,只记录"区间"。
  - 为什么它的设计哲学是"极简、能凑合",而不是"通用"。
  - 它如何把剩余内存"移交"给伙伴系统,然后功成身退——临时工棚拆掉,材料搬进大楼。
- **关键源码**:[mm/memblock.c](../linux-6.14/mm/memblock.c)、`memblock_alloc` / `memblock_free`。

### 第 5 章 · ZONE:把物理内存"分区"的理由

- **核心问题**:物理内存明明平铺连续,为什么非要切成 DMA / DMA32 / NORMAL 几个区?直接用一个池子不行吗?
- **要讲清的"为什么"**:
  - 硬件约束:老 ISA 设备只能 DMA 到 16MB 以下、32 位 DMA 设备只能到 4GB 以下。
  - 为什么 ZONE 之间是"降级回退"而不是"随便挑"。
  - migrate type(可移动/不可移动/可回收)为什么存在——为抗碎片和大页服务。
- **关键源码**:[include/linux/mmzone.h](../linux-6.14/include/linux/mmzone.h)、`struct zone`、`enum zone_type`。

### 第 6 章 · 伙伴系统:页级分配的核心算法

- **核心问题**:内核最频繁的操作之一就是"给我几页连续物理内存"和"还回去"。如何做到 O(1)、又自动抵抗碎片?
- **要讲清的"为什么"**:
  - 为什么是"2 的幂"阶(11 阶,最大 4MB)?这个上限是怎么来的。
  - "伙伴(buddy)"到底是什么,为什么靠它就能干净地合并、抵抗外部碎片。
  - 分配的快路径(pcplist / per-cpu)和慢路径(`__alloc_pages_slowpath`)、水位线(watermark)与 kswapd 的关系。
- **关键源码**:[mm/page_alloc.c](../linux-6.14/mm/page_alloc.c)、`__alloc_pages_noprof`、`free_one_page`。

### 第 7 章 · slab/slub:小对象的高效复用

- **核心问题**:伙伴系统最小只发 4KB 一页,但内核天天要分配几十字节的 `task_struct`、`inode`。怎么办?
- **要讲清的"为什么"**:
  - 为什么不能直接用伙伴系统分配小对象(内部碎片、无构造缓存、无锁竞争)。
  - "面向对象的池化"思想:每种高频对象一个 kmem_cache。
  - per-cpu 快通道为什么能几乎无锁;`SLAB_TYPESAFE_BY_RCU` 为什么是为 RCU 量身定做。
- **关键源码**:[mm/slub.c](../linux-6.14/mm/slub.c)、`kmem_cache_alloc`、`__slab_alloc`。

### 第 8 章 · 虚拟内存与多级页表:为什么要"骗"进程

- **核心问题**:进程根本不碰物理地址,它看到的是"独占的、连续的、从 0 开始"的虚拟地址空间。这个幻象为什么必要,又是怎么实现的?
- **要讲清的"为什么"**:
  - 虚拟内存的四个理由:隔离、保护、overcommit、按需分页。
  - 为什么不用单层页表(算一笔账:512GB 元数据),多级页表如何把稀疏映射"压扁"。
  - TLB 是什么,为什么页表结构和 TLB 是一对孪生兄弟。
- **关键源码**:[mm/memory.c](../linux-6.14/mm/memory.c)、`pgd/pud/pmd/pte` 操作、`include/asm-generic/pgtable-nested.h`。

### 第 9 章 · page fault:虚拟到物理的"按需兑现"

- **核心问题**:虚拟页是"空头支票",它什么时候、怎么变成"真金白银"的物理页?
- **要讲清的"为什么"**:
  - page fault 不是错误,而是"兑现时机"——这个视角转变是理解一切的关键。
  - 四种 fault:缺页、写时复制(COW)、写只读页、非法访问,分别怎么处理。
  - 为什么 COW 能让 `fork()` 几乎不复制内存。
- **关键源码**:[arch/x86/mm/fault.c](../linux-6.14/arch/x86/mm/fault.c) `do_user_addr_fault`、[mm/memory.c](../linux-6.14/mm/memory.c) `handle_mm_fault`。

### 第 10 章 · VMA 与进程地址空间:虚拟地址的"账本"

- **核心问题**:进程的虚拟地址空间里,代码、堆、mmap、栈各占其位,内核怎么记账?`malloc` 为什么不立即占内存?
- **要讲清的"为什么"**:
  - VMA 是"圈地"不是"占地":为什么 mmap 只建账本不分配页。
  - 红黑树 + 链表的组织,为什么这样选。
  - 一个进程的地址空间布局(代码/数据/bss/堆/mmap/栈),以及 ASLR 为什么存在。
- **关键源码**:[mm/mmap.c](../linux-6.14/mm/mmap.c)、`struct vm_area_struct`([include/linux/mm_types.h](../linux-6.14/include/linux/mm_types.h))、`do_mmap`。

### 第 11 章 · page cache:文件 IO 的"内存面"

- **核心问题**:读一个文件,为什么第二次快得多?文件数据是怎么和内存管理搅在一起的?
- **要讲清的"为什么"**:
  - page cache(以及 folio)的定位:磁盘文件的内存缓存,读写的真正战场。
  - 脏页、写回(writeback)、回收之间的三角关系。
  - 为什么 Linux 倾向"用空闲内存当缓存",而不是让内存闲着。
- **关键源码**:[mm/filemap.c](../linux-6.14/mm/filemap.c)、`address_space`、folio 的引入动机。

### 第 12 章 · 回收、swap 与 OOM:内存不够时怎么办

- **核心问题**:物理 RAM 终究有限,分配压力大时,内核怎么决定"牺牲谁"?
- **要讲清的"为什么"**:
  - 三类页的回收策略:干净文件页(直接丢)、脏页(写回再丢)、匿名页(swap)。
  - LRU(活跃/非活跃链表)与"双指针"老化机制,为什么不是简单 LRU。
  - 水位线 min/low/high 如何驱动 kswapd 与直接回收(direct reclaim);OOM killer 的"选谁杀"逻辑。
- **关键源码**:[mm/vmscan.c](../linux-6.14/mm/vmscan.c)、`shrink_lruvec`、`try_to_free_pages`。

### 第 13 章 · 进阶:NUMA、大页与 memcg

- **核心问题**:多核服务器、大内存数据库、容器化,这三类现代场景给内存管理出了什么新题?
- **要讲清的"为什么"**:
  - NUMA:为什么"近的内存更快",内核如何倾向于本地节点分配。
  - 透明大页(THP)/ Hugetlb:为什么 4KB 页在大内存时 TLB 压力巨大,大页怎么救场。
  - memcg(内存 cgroup):容器隔离怎么实现的,为什么有 OOM 的"局部版"。
- **关键源码**:[mm/huge_memory.c](../linux-6.14/mm/huge_memory.c)、[mm/hugetlb.c](../linux-6.14/mm/hugetlb.c)、[mm/memcontrol.c](../linux-6.14/mm/memcontrol.c)。
