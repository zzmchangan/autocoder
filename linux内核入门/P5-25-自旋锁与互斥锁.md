# 第 25 章 · 自旋锁与互斥锁:原地转圈 vs 排队睡觉

> **前置**:你需要先读过第 24 章(三种并发来源、进程上下文 vs 中断上下文、preempt_count)。本章全程建立在"我现在在什么上下文、要防谁"的基础上——没读过 24 章,你会看不懂为什么 spinlock 里不能睡觉、为什么 mutex 用不进中断。

读完这一章你会明白:面对一段要保护共享数据的代码,你能立刻判断**该用 spinlock 还是 mutex、要不要带 irqsave/bh 后缀**——并且能讲出每一条判断背后的"为什么",而不是照抄内核里某段看起来差不多的代码。

> **如果一读觉得太难**:先只记住一个决策表——**① 临界区要睡觉 / 很长 / 进程上下文 → mutex;② 临界区极短 / 不睡 / 可能在中断 → spinlock;③ spinlock 在中断可能碰这把锁的地方,加 `_irqsave`;在软中断可能碰的地方,加 `_bh`。** 其余实现细节第二遍配合源码再抠。

## 一、为什么内核需要两把截然不同的锁?

第 24 章把病看清楚了:**并发会让"改同一份数据"的代码交错执行**。治本的药是"把临界区圈起来,一次只准一个执行流进去"——这就是锁。

可"等拿锁"这件事,有两种**截然不同**的等法,而内核**两种都要**:

| 等法 | 比喻 | 实现 | 适合 |
|------|------|------|------|
| **原地忙等(busy-wait)** | 在门口站着不停问"好了没" | **自旋锁 spinlock** | 等**极短**,且**不能睡** |
| **去睡觉排队(sleep-wait)** | 拿号去休息区,被叫号才醒 | **互斥锁 mutex** | 等**很久**,且**可以睡** |

这一节先回答最根本的问题:**为什么不能只用一种?**

### 1.1 为什么不能"全部用自旋锁"?

自旋锁的等法是**忙等**:拿不到锁的 CPU 在一个紧凑循环里反复读锁状态,直到锁被释放。好处是**一旦锁释放,等的人零延迟冲进去**。坏处显而易见:

- **等的时候 CPU 满负荷空转**,白费电、白费算力。
- 如果临界区**很长**(比如要等一页从磁盘换回来,可能毫秒级),等的人就这么**干转几百万个周期**——荒唐。

所以自旋锁**只配用在"临界区极短"**的地方(几十条指令以内)。

### 1.2 为什么不能"全部用 mutex"?

mutex 的等法是**睡觉**:拿不到锁的执行流把自己挂到等待队列上,调 `schedule()` 让出 CPU,去睡觉;持锁的人释放锁时会唤醒队列上的下一个人。好处是**等的时候 CPU 能去干别的活**。坏处是:

- **睡下去和被唤醒要付出"上下文切换"的代价**(几百到几千个周期:保存/恢复寄存器、切换地址空间相关状态、cache 冷启动)。
- 如果临界区**极短**(就改一个计数器,几个周期),用 mutex 等于"为了睡一觉花的代价比干完活还大"——荒唐。

更致命的是:**mutex 涉及睡觉,而睡觉只能在进程上下文**。中断处理函数里**绝对不能**用 mutex(它会睡觉 → 死锁 → panic,见第 24 章)。

### 1.3 所以:两把锁,各管一摊

这就是为什么内核**必须**有两把锁:

- **spinlock**:临界区**短 + 不睡 + 可能在中断**。
- **mutex**:临界区**长 / 要睡 + 必须在进程上下文**。

> **心法**:选锁的依据**不是"哪种更高级"**,而是**"临界区里要不要睡觉、有多长、在不在中断上下文"**。这是本章最重要的一句话,反复回来对照。

## 二、自旋锁 spinlock:它在自旋什么?

先深挖 spinlock 的实现。你会看到,"自旋"这个词背后藏着相当精巧的设计。

### 2.1 最朴素的 spinlock 长什么样

抛开所有优化,一个最朴素的 spinlock 就是一个整数:

```
0 = 空闲,1 = 有人占了
lock():  while (锁 != 0);   锁 = 1;   // 等 + 抢
unlock(): 锁 = 0;
```

**不这样会怎样?** 这个朴素版本有两个致命问题:

1. **"等"和"抢"不是原子的**。两个核可能同时看到锁=0、同时冲进去。所以"等 + 抢"必须是一条**原子指令**(test-and-set / cmpxchg / xadd 之类)。
2. **公平性**:所有等的人一起抢,刚释放就被任意一个核抢走,**可能有人饿死**(永远抢不到)。在 NUMA 上更糟:远端核反复抢,把锁所在的缓存行打成 ping-pong,总线风暴。

现代 x86 用的是 **queued spinlock(qspinlock)**,就是为了治这两个问题。

### 2.2 qspinlock:排队,别一窝蜂抢

qspinlock 的核心思想:**别一窝蜂去抢锁,排成一条队**。每个等锁的 CPU,在锁里记下"我排第几、我前面是谁",然后只盯着自己前面的那个 CPU(而不是盯着锁本身)。

先看快路径(无竞争时),出自 [include/asm-generic/qspinlock.h](../linux-6.14/include/asm-generic/qspinlock.h#L107-L116):

```c
static __always_inline void queued_spin_lock(struct qspinlock *lock)
{
	int val = 0;

	if (likely(atomic_try_cmpxchg_acquire(&lock->val, &val, _Q_LOCKED_VAL)))
		return;

	queued_spin_lock_slowpath(lock, val);
}
```

逐行讲:

- `atomic_try_cmpxchg_acquire(&lock->val, &val, _Q_LOCKED_VAL)`:这是一条**原子比较并交换**指令(x86 上是 `lock cmpxchg`)。它原子的做:"如果 `lock->val` 等于 0(空闲),就把它设成 `_Q_LOCKED_VAL`(已锁),返回 true"。**整个"判断 + 抢"是一条指令,不可被打断**,治好了朴素版的第一个病。
- 如果抢到了(`return`)——这就是**快路径**:无竞争时,拿锁就一条原子指令,极快。
- 没抢到(锁已经被别人占了)——走 `queued_spin_lock_slowpath`,去**排队**。这就是慢路径,实现在 [kernel/locking/qspinlock.c](../linux-6.14/kernel/locking/qspinlock.c)。

> **`_acquire` 后缀是什么?** 这是 acquire 内存屏障语义(第 27 章详讲)。一句话:它保证"拿锁之后的读写在拿锁之后才被别的核看见",防止 CPU 把临界区里的活儿"漏"到锁外面去。spinlock 的正确性不只靠原子性,还靠这条屏障——第 27 章会回扣这里。

慢路径(qspinlock 的精髓)做的事大致是:

1. 看锁里有没有"等待队列"的尾巴指针。没有就建一个,把自己挂上;有就在它后面排。
2. 把锁里记录的"队尾"原子地更新成自己。
3. 现在**只盯着排在自己前面的那个 CPU 的状态**(而不是全局锁)。前面的人释放、轮到我,我才进去。

> **为什么这么设计?** 这叫 **MCS 锁**思路。好处:**每个等的人只在自己私有的一个缓存行上自旋**,不再有几十个核一起把全局锁的缓存行打成 ping-pong。这把"等锁"的缓存争用从 O(N 个核)降到 O(1)。在高核数机器上,这是 spinlock 性能能撑住的关键。

### 2.3 unlock:为什么是 smp_store_release?

释放锁同样要看一眼,出自 [include/asm-generic/qspinlock.h](../linux-6.14/include/asm-generic/qspinlock.h#L122-L131):

```c
static __always_inline void queued_spin_unlock(struct qspinlock *lock)
{
	/*
	 * unlock() needs release semantics:
	 */
	smp_store_release(&lock->locked, 0);
}
```

`locked = 0` 这么简单的事,为什么要包一个 `smp_store_release`(release 屏障)?

**不这样会怎样?** 因为 CPU 会乱序(第 27 章详讲)。设想临界区里你写了 `data = 42; ready = 1;`(假设 ready 是临界区最后一个写)。CPU 可能把"写 ready=1"先于"写 data=42"送到内存。如果 unlock 只是个普通写,那么"unlock(把锁清 0)"也可能被重排到 `data=42` 之前——结果下一个拿到锁的人,看到"锁是我的了,但 data 还是旧值"。**release 屏障强制:临界区里的所有写,必须在 unlock 之前完成并可见。** 这是锁正确性的另一半保证。

### 2.4 spinlock 的硬规矩:临界区绝不能睡觉

这条规矩怎么强调都不过分,因为它是 spinlock **绝大多数误用 bug** 的根源。

**为什么不能睡?** 两个理由,都致命:

1. **违背 spinlock 的设计前提**。spinlock 的本意是"临界区极短,等的人忙等也很快"。你一睡(比如 `kmalloc(GFP_KERNEL)`、`mutex_lock`、会缺页的 `copy_from_user`),持锁时间可能变成毫秒级甚至更久。外面忙等的核**白烧几百万周期**——spinlock 的性能优势荡然无存。
2. **更糟:可能死锁/活锁**。在 SMP 上,你拿着锁睡下去后,如果调度器因为某种原因没把你及时调回来(比如调度器自己也在等别的锁),外面等这把锁的核可能等到天荒地老。在某些配置下(PREEMPT_RT 之前)持有 spinlock 时抢占是被禁的,但睡觉是另一回事——睡觉会主动调 `schedule()`,在持锁状态下这是灾难。

内核有一整套机制帮你抓这种 bug:**`might_sleep()`、`atomic context` 检测、lockdep**。`might_sleep()` 散布在所有可能睡觉的函数开头,如果你在持 spinlock / 关中断状态下调用它们,内核会立刻报警告(甚至 panic,看配置)。

> **实战铁律**:拿到 spinlock 后,你只能做"纯计算 + 改共享数据 + 释放锁"。任何 `*_sleep`、`GFP_KERNEL` 分配、`mutex_*`、`schedule`、`wait_event`、可能缺页的 `*_user` ——**一律不许碰**。要分配内存用 `GFP_ATOMIC`(不会睡)。

### 2.5 spinlock 的变体:我到底在防谁?

第 24 章讲过,并发有三个来源(多核、中断、抢占)。**spinlock 默认只关抢占**——它防住了别的核(通过原子抢锁)和抢占(拿锁时 count+1),但**防不住中断**。

设想这个灾难场景(单核就能复现):

```
进程上下文: spin_lock(&L) 拿到锁,改共享数据
   ↑ 网卡中断来了!
中断处理函数: spin_lock(&L)  → 锁被自己(本核)拿着,死等
   → 这把锁永远释放不了,因为这个核卡在中断里,回不到进程上下文去 unlock
   → 死锁,整个核卡死。
```

这就是为什么 spinlock 有那么一堆变体,本质都是"**拿锁的同时,顺手把某股捣乱的力量也关掉**":

| API | 拿锁时还做了什么 | 防的是谁 |
|-----|------------------|----------|
| `spin_lock` / `spin_unlock` | 关抢占(默认) | 别的核 + 本核抢占 |
| `spin_lock_irq` / `spin_unlock_irq` | 关 + 开**本地中断** | 再加上本核硬中断 |
| `spin_lock_irqsave` / `spin_unlock_irqrestore` | 关中断**且保存原中断状态**,开时还原 | 同上,但**支持嵌套**(最常用) |
| `spin_lock_bh` / `spin_unlock_bh` | 关 / 开**下半部(软中断)** | 别的核 + 本核软中断 |

**为什么有这么多?能不能全用 `irqsave` 一了百了?** 不行,因为**关中断是有代价的**:关中断期间,这个核对所有硬中断都不响应(实时性变差、丢包、丢时钟 tick)。内核的原则是**只关必要的、能少关就少关**:

- 如果这把锁**只有进程上下文**碰(中断里绝不碰),用裸 `spin_lock` 就够——只关抢占,中断照样响应。
- 如果这把锁**进程上下文和硬中断**都会碰,用 `spin_lock_irqsave`——进程上下文那侧拿锁时关中断,中断处理那侧本来就在中断里(中断已关)。
- 如果只跟**软中断**冲突(进程上下文 vs 软中断),用 `spin_lock_bh`——只关下半部,硬中断照样响应(比关全中断代价小)。

> **`_irqsave` 里的 "save" 为什么重要?** 因为中断状态是**嵌套**的。你可能在"中断已经被上层关掉"的情况下又拿一把锁;解锁时不能无脑 `local_irq_enable()`(那会把上层故意关的中断给开了)。`irqsave` 把拿锁前的中断状态存下来,`irqrestore` 还原成原状。这就是它比 `_irq` 更常用、更安全的原因。

## 三、互斥锁 mutex:拿号、去睡、被叫醒

现在看另一极:mutex。它的设计哲学和 spinlock 完全相反——**等很久没关系,但你得睡觉**。

### 3.1 mutex 的结构

mutex 在内核里是个结构体,核心字段(出自 [include/linux/mutex.h](../linux-6.14/include/linux/mutex.h),非 RT 配置下):

- `owner`:一个原子长整型,记录**当前持锁者是谁**(其实是 `current` 的指针,顺带塞了几个标志位)。
- `wait_lock`:一把**spinlock**,用来保护等待队列本身(等待队列也是共享数据,改它得用 spinlock)。
- `wait_list`:等待这把锁的执行流排成的队列。

> **一个反直觉的点**:mutex 内部**用了一把 spinlock**(`wait_lock`)。这说明"睡眠锁"的实现**仍然离不开自旋锁**——管理"谁在排队"这种短临界区,还是得用 spinlock。这不是矛盾,而是分层:mutex 用 spinlock 保护自己的账本,自己再在账本之上提供"可睡"的语义。

### 3.2 快路径:能 cmpxchg 就别睡

mutex 的 `mutex_lock` 出自 [kernel/locking/mutex.c](../linux-6.14/kernel/locking/mutex.c#L256-L265):

```c
void __sched mutex_lock(struct mutex *lock)
{
	might_sleep();

	if (!__mutex_trylock_fast(lock))
		__mutex_lock_slowpath(lock);
}
EXPORT_SYMBOL(mutex_lock);
```

- `might_sleep()`:第一件事就是声明"我会睡觉"——如果你在不能睡的上下文(中断)调它,这里会立刻报警。这是 mutex 的第一道自我保护。
- `__mutex_trylock_fast`:快路径。如果能原子地拿到锁,就直接返回,**完全不睡觉**。

快路径的实现,出自 [kernel/locking/mutex.c](../linux-6.14/kernel/locking/mutex.c#L141-L152):

```c
static __always_inline bool __mutex_trylock_fast(struct mutex *lock)
{
	unsigned long curr = (unsigned long)current;
	unsigned long zero = 0UL;

	if (atomic_long_try_cmpxchg_acquire(&lock->owner, &zero, curr))
		return true;

	return false;
}
```

逐行:把 `lock->owner` 从 0(无人持有)原子地设成"当前进程指针"。成功就是拿到了。**无竞争时,mutex 拿锁也是一条原子指令**——和 spinlock 一样快。这是 mutex 设计的关键:绝大多数时候锁其实没人在抢,走快路径根本不睡。

### 3.3 慢路径:排队、(可能)乐观自旋、睡觉

快路径抢不到,走 `__mutex_lock_slowpath` → `__mutex_lock`。慢路径做三件事:

1. **乐观自旋(optimistic spinning)**:如果持锁者此刻正在跑(没睡),那它很可能**马上就释放**了——与其睡下去再被唤醒(付出切换代价),不如**短暂地忙等一会**,赌它很快就放手。这就是 `mutex_optimistic_spin`(出自 [kernel/locking/mutex.c](../linux-6.14/kernel/locking/mutex.c#L416) 附近)。它借用了 spinlock 的思路,但加了退出条件:持锁者一旦去睡了,我就不傻等了,转去睡觉。

   > **为什么不直接一直 spin?** 因为 mutex 持锁者可能睡很久(临界区是设计来可以睡的)。一直 spin 会浪费 CPU。乐观自旋只在"持锁者看起来马上就放手"时才 spin,是个聪明的折中。它还用了一把 **osq(optimistic spin queue)** 锁([include/linux/osq_lock.h](../linux-6.14/include/linux/osq_lock.h#L10))来组织多个乐观自旋者,避免它们互相把 owner 缓存行打爆——思路和 qspinlock 类似。

2. **真正排队睡觉**:乐观自旋也没等到?那就把自己挂到 `wait_list` 上(用 `wait_lock` 保护),设成 TASK_UNINTERRUPTIBLE(或 INTERRUPTIBLE),调 `schedule()` **真的睡下去**,让出 CPU。
3. **被唤醒**:持锁者 `mutex_unlock` 时,会唤醒 `wait_list` 队首的人。被唤醒的人从 `schedule()` 返回,重新尝试拿锁。

### 3.4 unlock:快路径 + 叫号

`mutex_unlock`([kernel/locking/mutex.c](../linux-6.14/kernel/locking/mutex.c#L517)):

- 如果 `wait_list` 是空的(没人在等),走快路径 `__mutex_unlock_fast`——一条原子指令把 owner 清 0,完事。
- 如果有人在等,慢路径:在 `wait_lock` 保护下,把 owner 清 0,从 `wait_list` 取队首唤醒。

> **为什么 unlock 也分快慢路径?** 同样的道理:绝大多数 unlock 时根本没人在等,一条原子指令解决;只有真有人在排队时才走"叫号"的慢路径。这种"快路径原子、慢路径才上重武器"的设计,是内核锁的通用模式——spinlock、mutex、甚至 RCU 都是这套路。

### 3.5 mutex 的规矩:只能在进程上下文

mutex 的规矩和 spinlock 正好对称:

- ✅ **能睡觉**:临界区里可以 `kmalloc(GFP_KERNEL)`、可以 `copy_from_user`、可以等事件——这是 mutex 存在的全部意义。
- ❌ **绝不能在中断上下文用**:中断/软中断里没有"当前进程"可以睡,调 mutex 必死。
- ❌ **临界区不能太短得不划算**:虽然能跑,但你拿 mutex 去保护一个"改一个 int"的临界区,纯属浪费(上下文切换代价比干活还大)。那种地方用 spinlock。

## 四、一张决策表:面对临界区,怎么选?

把前面所有"为什么"浓缩成一张实战决策表。**这是本章最值得记住的东西**:

```
问:临界区里要不要睡觉?(kmalloc(GFP_KERNEL)/copy_from_user/mutex_lock/wait_event/...)
│
├─ 要睡 ─────────────────────► mutex(且必须在进程上下文)
│
└─ 不睡 ──► 再问:临界区有多长?
            │
            ├─ 很长(微秒以上)──► 也该用 mutex(让等的人去睡,别空转)
            │
            └─ 极短(几条~几十条指令)──► spinlock。再问:这把锁还有谁会碰?
                  │
                  ├─ 只有进程上下文碰 ─────────► spin_lock
                  ├─ 进程上下文 + 硬中断碰 ────► spin_lock_irqsave
                  └─ 进程上下文 + 软中断碰 ────► spin_lock_bh
```

几条配套铁律:

1. **同一个锁,所有访问点用同一套防护级别**。不能这边 `spin_lock_irqsave`、那边 `spin_lock`——后者那侧会被中断打断撞上锁,死锁。lockdep 会帮你抓这种不一致。
2. **持有锁的顺序要全局一致**,否则死锁(A 拿 L1 再要 L2,B 拿 L2 再要 L1 → 死锁)。lockdep 也会抓。
3. **临界区里不要调持有别的锁可能导致你睡死的函数**(对 spinlock 而言)。`might_sleep()` 是你的哨兵。

## 关键源码精读:从 mutex_lock 到睡觉

本章最值得逐行看的,是 mutex 慢路径"决定要不要乐观自旋、要不要睡"的那段逻辑——它浓缩了 mutex 设计的全部巧思。出自 [kernel/locking/mutex.c](../linux-6.14/kernel/locking/mutex.c)(简化摘录,保留真实函数名与控制流):

```c
static int __sched __mutex_lock(struct mutex *lock, ...)
{
	/* 1. 快路径已在 mutex_lock 里试过(原子 cmpxchg owner),失败才到这 */

	/* 2. 乐观自旋:赌持锁者马上放手,先忙等一会,不睡觉 */
	if (mutex_optimistic_spin(lock, ww_ctx, NULL))   /* 见 mutex.c:416 */
		return 0;   /* spin 期间拿到锁了,直接成功,没睡 */

	/* 3. 乐观自旋也没拿到 —— 老老实实排队睡觉 */
	/*    a. 拿 wait_lock(spinlock) 保护等待队列 */
	/*    b. 把自己(struct mutex_waiter)挂到 wait_list 队尾 */
	/*    c. 设 task 状态为 UNINTERRUPTIBLE,调 schedule() 真正睡下去 */
	/*    d. 被 unlock 唤醒后,从这里恢复,重试拿锁 */
	...
}
```

逐段对应前面讲的设计:

1. **快路径**(`mutex_lock` → `__mutex_trylock_fast`):一条 `cmpxchg` 原子抢 owner。无竞争时到此为止,代价和 spinlock 持平。
2. **乐观自旋**(`mutex_optimistic_spin`):这是 mutex 和 spinlock 的"混血"——既然持锁者可能马上放手,先借 spinlock 的忙等思路赌一把。**赌赢(持锁者很快放手)就赚到**(省了上下文切换);**赌输(持锁者睡下去了)就及时退出**,转去真睡。赌不赌、何时退出,是这段代码的精髓。
3. **排队睡觉**(挂 wait_list + `schedule`):乐观自旋失败后的兜底。这时 mutex 才展现出它"可以等很久"的本事——睡下去后 CPU 完全空闲去做别的活,直到被 unlock 唤醒。

> **这一段为什么是 mutex 设计的精华?** 因为它把 spinlock 的优点(快路径快、乐观自旋低延迟)和"可以睡"(长临界区不浪费 CPU)**两种看似矛盾的优点**缝合在了一起。无竞争→原子指令;轻竞争→乐观自旋;重竞争→睡觉排队。**根据竞争强度自动切换策略**,这就是为什么 mutex 既能用在轻负载下不输 spinlock、又能扛住长临界区。现代内核的 `mutex_lock` 已经是这个"三段式"的成熟形态。

## 章末小结

用"档案室"比喻收一下:

- **临界区是"改档案的那几步"**。锁就是档案室的门锁,把这几步圈起来,一次只准一个办事员进去改。
- **spinlock = 在门口站着忙等**。适合"我就改两笔,马上出来"——办事员站着等几秒没关系,但**站在门口不许去上厕所(不能睡)**。怕有人(中断)拍肩膀插队,就先**把门铃关了**(irqsave)。
- **mutex = 拿号去休息区坐着,叫到号才醒**。适合"改档案要花半天"——站着等太亏,不如去睡。但**只有正式办事员(进程上下文)能拿号**,门卫(中断)不行。
- 两把锁的**快路径都是一条原子指令**(无竞争时一样快);区别全在**抢不到时怎么等**。

**选锁就一句话**:**临界区要睡/很长/进程上下文 → mutex;极短/不睡/可能在中断 → spinlock(再按"还有谁碰这锁"选 irqsave/bh 变体)。**

**回扣全书主线**:锁是"管资源"那一侧的基础设施——它管的是"多个执行流共享同一份数据"这个最普遍的冲突。后面整个内核(调度、内存回收、文件系统、网络)处处在用这两把锁。可以说,**读懂这一章,你才真正具备了阅读内核并发代码的资格**。

**想继续深入该往哪钻**:
- spinlock 的灵魂在 **qspinlock** 慢路径:[kernel/locking/qspinlock.c](../linux-6.14/kernel/locking/qspinlock.c) 的 `queued_spin_lock_slowpath`——看它怎么把等待者排成 MCS 队列。
- mutex 的乐观自旋细节:[kernel/locking/mutex.c](../linux-6.14/kernel/locking/mutex.c) 的 `mutex_optimistic_spin` 和 [kernel/locking/osq_lock.c](../linux-6.14/kernel/locking/osq_lock.c)。
- **lockdep**:[kernel/locking/lockdep.c](../linux-6.14/kernel/locking/lockdep.c)。打开 `CONFIG_LOCKDEP` 跑一遍,它会把你代码里所有"忘了加锁""加错锁""锁顺序会死锁"的隐患在运行时报出来——这是学并发最好的老师。
- PREEMPT_RT:在 [kernel/locking/spinlock.c](../linux-6.14/kernel/locking/spinlock.c) 看 RT 配置下 spinlock 怎么变成"可睡"的(rtmutex),理解为什么"实时性"和"spinlock 不能睡"传统教条会冲突。
