# 第 4 章(全书第 23 章)· 三者的统一视角:都是"陷入内核"

> **前置**:这是本篇的收束章,你需要先读完前三章([中断与异常](P4-01-中断与异常-硬件怎么拍政府的肩膀.md)、[时钟与定时器](P4-02-时钟与定时器-城市的心跳.md)、[系统调用](P4-03-系统调用-过关的艺术.md))。本章不再讲新机制,而是把三者摆在一起,看清它们的**共性**与**真正的区别**——把本篇收成一张图。
>
> **核心问题**:中断、异常、系统调用表面上是三件事(硬件拍肩膀、CPU 自动报警、市民主动办事),内核却用**同一套基础设施**(IDT / pt_regs / `irqentry_enter/exit`)处理它们。它们的本质共性是什么?区别到底在哪几条线上?
>
> **读完本章你会明白**:
> - 三者共享同一条"过门 5 步"流水线,共用同一段进出门骨架(`irqentry_enter`/`irqentry_exit`);
> - 区别只在**三条分界线**:谁发起(硬件/CPU自动/进程)、同步还是异步、有没有"当前进程"可不可以阻塞;
> - 为什么"异常和系统调用更像"(都是和当前进程绑定的同步陷入),而"中断是另一个世界"(跑在中断上下文,没有进程可言);
> - 三者各自的数据结构(IRQ desc / trap error code / syscall table + pt_regs)如何在统一骨架上各司其职。

> **如果只想抓重点**:记住一张表——**三者都是"陷入内核",共骨架、共 IDT、共 pt_regs;区别在"谁发起、是否同步、能否关联当前进程、能否阻塞"四条线上**。中断=异步外部、不能阻塞;异常=同步CPU自动、关联当前进程;系统调用=同步自愿、关联当前进程、输入不可信所以检查最严。

## 一、先承认一件事:它们看起来八竿子打不着

回顾一下本篇讲过的三件事,它们表面上确实像是三个完全不同的世界:

- **中断**:网卡、键盘、时钟这些外部设备,通过电信号戳 CPU。和当前进程没关系,随时可能来。
- **异常**:CPU 执行某条指令,自己检测到除零、缺页、非法指令。和当前进程紧紧绑在一起。
- **系统调用**:进程主动执行一条 `syscall` 指令,自愿请求内核办事。完全是进程主导的。

一个是"外部世界突然来事儿",一个是"指令自己出错",一个是"进程主动申请"。**听起来就像三种不相干的事。**

但内核的实现告诉我们一个反直觉的事实:**这三件事,CPU 和内核是把它们当同一类事来处理的。** 本章就是把这张"统一性"摊开给你看。

---

## 二、统一性:三者共用同一条"过门 5 步"流水线

[导言](P4-00-导言-全局直白解释.md)和第 1 章都讲过"过门 5 步"。回头看,**中断、异常、系统调用走的是完全一样的 5 步**:

```
   ① 保存现场         ② 换世界            ③ 查表找人        ④ 执行处理        ⑤ 恢复现场回门
   (压 pt_regs)      (切内核栈/CR3/ring0) (IDT / syscall)  (handler)        (恢复 pt_regs)
        │                  │                    │               │                 │
   ┌────┴──────────────────┴────────────────────┴───────────────┴─────────────────┴──┐
   │                    三者共用同一条流水线、同一套骨架代码                          │
   └──────────────────────────────────────────────────────────────────────────────────┘
```

逐项对应三者的"同一性":

### 共用 1:同一张门牌总表(IDT)

CPU 不管是被中断戳、自己异常、还是遇到 `syscall`(老式 `int 0x80` 路径),**第一反应都是"查一张表找入口"**。这张表就是 **IDT**(本仓库 [arch/x86/kernel/idt.c](../linux-6.14/arch/x86/kernel/idt.c) 登记)。

- vector 0~31 → **异常**(除零 0、page fault 14、GP 13……);
- 32 往后 → **外部中断**和**系统向量**(时钟、IPI);
- 系统调用虽然现代走专用 `syscall` 指令不经 IDT(走 MSR),但**机制本质完全相同**:都是"一个编号 → 查到一个入口地址"。`syscall` 只是把"查 IDT"换成了"查 MSR 预存的入口",换取更快。

> **IDT 是三者统一性的第一根支柱**。它是一张"任何惊动都先查的门牌表",不管是硬件、CPU 自己还是进程主动,进门第一件事都是查它。

### 共用 2:同一种现场快照(pt_regs)

三者进了门,第一件事都是**把用户态(或被打断的前一个状态)的寄存器全部压栈**,形成同一种结构——**`struct pt_regs`**。这是"被打断现场"的统一快照。

- 中断:把被中断进程的寄存器压成 pt_regs;
- 异常:把触发异常那条指令所在进程的寄存器压成 pt_regs;
- 系统调用:把调 `syscall` 的进程的寄存器压成 pt_regs(其中 `orig_ax` 存系统调用号)。

正因为结构统一,内核处理完之后,**恢复现场**的代码也完全一样:从 pt_regs 把寄存器还原,CPU 回到被打断的地方继续。

> 这也解释了为什么 [第 3 章](P4-03-系统调用-过关的艺术.md) 里系统调用号存在 `pt_regs->orig_ax`([arch/x86/include/asm/ptrace.h:128-139](../linux-6.14/arch/x86/include/asm/ptrace.h#L128-L139)):同一个 pt_regs 结构,中断时 `orig_ax` 存 vector,系统调用时存调用号。**一个结构,多种语义,因为它们本就是同一类事。**

### 共用 3:同一段进出门骨架(`irqentry_enter` / `irqentry_exit`)

这是统一性最硬的证据。三种惊动进了门之后,**都要先过同一段通用骨架代码** `irqentry_enter`,出门都要过 `irqentry_exit`:

> [kernel/entry/common.c:236](../linux-6.14/kernel/entry/common.c#L236)

```c
noinstr irqentry_state_t irqentry_enter(struct pt_regs *regs)
{
	irqentry_state_t ret = {
		.exit_rcu = false,
	};

	if (user_mode(regs)) {                   /* 这次惊动是从用户态进来的? */
		irqentry_enter_from_user_mode(regs);
		return ret;
	}
	...                                       /* 从内核态进来的(idtentry嵌套)处理 RCU 细节 */
}
```

它做"进门登记"——RCU 进入、记账、抢占计数加 `HARDIRQ_OFFSET`、给当前 CPU 时间记账。**不管你是中断、异常还是系统调用,这段代码都跑。**

出门侧:

> [kernel/entry/common.c:328-360](../linux-6.14/kernel/entry/common.c#L328-L360)(精简)

```c
noinstr void irqentry_exit(struct pt_regs *regs, irqentry_state_t state)
{
	lockdep_assert_irqs_disabled();

	if (user_mode(regs)) {                    /* 回用户态? */
		irqentry_exit_to_user_mode(regs);     /* → 检查信号、TIF_NEED_RESCHED,然后出栈回用户 */
	} else if (!regs_irqs_disabled(regs)) {
		...
		if (IS_ENABLED(CONFIG_PREEMPTION))
			irqentry_exit_cond_resched(); /* 抢占式内核:回内核态时可趁机重新调度 */
		...
	}
}
```

注意这段代码里有个关键的**统一分诊点**:`user_mode(regs)`——"这次惊动,是从用户态进来的吗?"

- **是(从用户态来)** → 出门时走 `irqentry_exit_to_user_mode`,在回用户态前处理挂起信号、检查 `TIF_NEED_RESCHED`(抢占)。
- **否(从内核态来,即中断嵌套在内核里)** → 走另一条路,抢占式内核里可能在这里重新调度(`irqentry_exit_cond_resched`),但**不发信号**(信号是给用户进程的)。

> **这一个 `user_mode(regs)` 判断,是三者统一骨架里的"分水岭"**。它不关心"你是中断还是异常还是系统调用",只关心"你从哪儿来"。这种"按来源而非按类型分流"的设计,正是统一性的体现。

## 三、统一骨架上的三种"血型":各自的表与上下文

骨架是统一的,但三者各自有自己那本"账"。它们在统一骨架上挂了不同的"分诊数据":

| 维度 | 中断 (IRQ) | 异常 (exception) | 系统调用 (syscall) |
|---|---|---|---|
| **谁触发** | 外部硬件(异步) | CPU 执行指令时自动(同步) | 进程主动(同步、自愿) |
| **入口表** | IDT(vector 32+) | IDT(vector 0~31) | `sys_call_table`(+ MSR 直达) |
| **"请求单"** | IRQ 号 → `irq_desc` | error code(如 [trap_pf.h](../linux-6.14/arch/x86/include/asm/trap_pf.h) 的 fault 标志位) | 系统调用号(RAX) + 参数(寄存器) |
| **关联当前进程?** | ❌ 否(打断谁算谁) | ✅ 是(触发指令的进程) | ✅ 是(调用的进程) |
| **能阻塞/睡眠?** | ❌ 不能(中断上下文) | 多数不能(fault 处理也尽量不睡;少数可) | ✅ 能(有完整进程上下文) |
| **重试行为** | 不重试(返回到下一条) | fault 重试原指令 / trap 跳下一条 | 调完返回(RAX 存返回值) |
| **安全检查** | 几乎没有(硬件/内核自己) | 检查是否非法访问 | **最严**(seccomp/越界/指针校验) |

这张表是本篇的"总账"。下面解释为什么差异集中在这几条线上——它们都不是随意的,而是被每种惊动的"出身"决定的。

---

## 四、第一条分界线:谁发起、同步还是异步

这是三者最直观的区别,也是后续所有差异的根源。

- **中断=异步外部**。它和当前正在执行的指令**没有任何因果关系**。进程 A 正在算 `1+1`,网卡包来了——这个中断不是 A 的指令触发的,A 只是"倒霉正好在场"。所以中断对进程而言是**纯粹的外部干扰**。
- **异常=同步、CPU 自动**。它一定发生在**某条具体指令**上,是那条指令的**直接后果**。进程除以零,异常就发生在那条除法指令上——能精确定位。
- **系统调用=同步、自愿**。进程**故意**执行 `syscall`,主动请求陷入。完全是进程主导的、预期的。

> 这条线决定了"事后怎么处理":中断处理完**返回到被中断处继续**(A 不知道自己被打断过);异常处理完**视情况**——fault 重新执行触发指令(page fault 配好页后重试那条访问)、trap 跳下一条(断点);系统调用处理完**返回到 syscall 之后**(就像普通函数返回)。

---

## 五、第二条分界线:有没有"当前进程"、能不能阻塞

这是最深层、影响最大的区别。

### 中断没有"当前进程"

中断打断进程 A,处理时它"借用"了 A 的内核栈,但它**不代表 A**,它是在**中断上下文**里跑。中断处理函数**不能假设自己在为哪个特定进程服务**——A 只是被打断的倒霉蛋,中断干的活和 A 可能毫无关系(网卡包可能是给进程 B 的)。

**后果**:

- **不能睡眠**:睡眠要"挂起一个进程",可中断不属于任何进程,无进程可挂。
- **不能持有需要睡眠的锁**(如互斥锁),只能用自旋锁(同步篇主角)。
- **不能分配可能触发换页的内存**(那会睡眠)。

这就是 [第 1 章](P4-01-中断与异常-硬件怎么拍政府的肩膀.md) 整个"上半部/下半部"设计的根源。

### 异常和系统调用有"当前进程"

异常和系统调用都是"当前进程正在执行某条指令"时发生的。它们**深深绑在 `current` 上**:

- page fault 要给 `current` 配一页、查 `current` 的 VMA;
- 系统调用要读 `current` 的文件描述符表、改 `current` 的地址空间;
- 异常里发信号,信号发给 `current`(SIGSEGV 杀的就是当前进程)。

**后果**:

- **能睡眠**(有完整进程上下文可挂起)。系统调用里 `sys_read` 等磁盘 IO 时,当前进程睡眠等待,完全合法。
- 能持有任意锁、能分配内存、能做几乎所有"正常进程能做的事"。

> 这就是为什么第 2 章强调"时钟中断虽然走中断入口,却关联当前进程"——它在 `update_process_times` 里给 `current` 记账、给 `current` 减时间片。时钟中断是个**特例**:它名义上是中断(异步、外部、来自 Local APIC),但它干的活(调度、记账)却**紧紧依赖当前进程**。它站在"中断"和"异常"的交界处——这也是它最特殊的原因之一。

### 为什么这条线最重要

这条"有没有当前进程"的分界线,是**同步篇的根**。同步篇要讲"多核 + 中断 + 抢占三种并发来源怎么把共享数据改乱"。其中:

- **中断**这一路并发:它能在任何时刻打断任何进程(包括内核代码),没有任何"约定",所以任何可能被中断访问的共享数据,都要考虑"中断会不会插进来改它"——这正是"关中断""自旋锁要关抢占"的根源。
- **抢占**这一路并发:`TIF_NEED_RESCHED` 在系统调用返回、中断返回时被检查,内核代码可能在任何"安全点"被换下。

理解了"中断没有当前进程、异常/系统调用有",你就理解了为什么内核代码里到处是 `local_irq_disable()`、`preempt_disable()`、`spin_lock_irqsave()`——它们都是在防范这两路并发。本篇给同步篇铺好了这块基石。

## 六、第三条分界线:输入可信度与安全检查的严苛度

三种惊动对内核的"可信度"不同,安全检查的重量也天差地别:

- **中断**:输入来自硬件和内核自己的驱动注册,基本可信(硬件不会"恶意")。几乎不做"权限检查"。
- **异常**:输入是 CPU 自动生成的 error code(描述这次 fault 的性质),也基本可信。检查的是"这次访问合不合法"(有没有 VMA、越不越权)——这是内存篇 [page fault 章](P2-09-page-fault-虚拟到物理的按需兑现.md) 讲的 `access_error`。
- **系统调用**:**输入完全不可信**。系统调用号、参数、用户指针,全是用户空间传进来的、用户完全可控的。所以这一路上**层层防线**:seccomp(能不能调这个)、`array_index_nospec`(防投机越界)、`copy_from_user`(防非法指针)、参数边界检查……这是 [第 3 章](P4-03-系统调用-过关的艺术.md) 反复强调的"用户是不可信输入"。

> **一句话总结这条线**:越靠近"用户主动可控"的惊动,安全检查越严。中断最松(硬件可信),系统调用最严(用户不可信)。这个梯度,本质是"对不可信输入的防御梯度"。

## 八、关键源码精读:`irqentry_enter` / `irqentry_exit`——三者的统一骨架

本章的精华,就浓缩在 `irqentry_enter` / `irqentry_exit` 这对函数里——它们是"三者统一性"的代码铁证。我们再读一遍,这次带着"统一"的视角。

### 进门:`irqentry_enter`

> [kernel/entry/common.c:236-246](../linux-6.14/kernel/entry/common.c#L236-L246)

```c
noinstr irqentry_state_t irqentry_enter(struct pt_regs *regs)
{
	irqentry_state_t ret = {
		.exit_rcu = false,
	};

	if (user_mode(regs)) {                /* ★ 从用户态进来的吗? */
		irqentry_enter_from_user_mode(regs);
		return ret;
	}
	...                                    /* 从内核态进来的(中断嵌套)的 RCU 处理 */
}
```

注意它的分诊逻辑是 `user_mode(regs)`,**不是**"你是中断还是异常还是系统调用"。这就是统一性的精髓:骨架不在乎"哪种惊动",只在乎"你从哪儿来"。`regs` 就是三者共用那个 pt_regs,`user_mode()` 看其中的 CS 段选择子是不是用户态。

### 出门:`irqentry_exit`

> [kernel/entry/common.c:328-357](../linux-6.14/kernel/entry/common.c#L328-L357)(精简)

```c
noinstr void irqentry_exit(struct pt_regs *regs, irqentry_state_t state)
{
	lockdep_assert_irqs_disabled();

	if (user_mode(regs)) {                              /* ★ 回用户态? */
		irqentry_exit_to_user_mode(regs);               /* → 处理挂起信号、TIF_NEED_RESCHED,再出栈 */
	} else if (!regs_irqs_disabled(regs)) {
		...
		if (IS_ENABLED(CONFIG_PREEMPTION))
			irqentry_exit_cond_resched();          /* 抢占式内核:回内核态时可重新调度 */
		...
	}
}
```

同样是 `user_mode(regs)` 分诊。出门侧这一段还揭示了一个统一的关键点:**"回用户态"是一个稀缺的、被复用的安全过渡点**。

- 不管是中断、异常还是系统调用,只要它**最终要回用户态**,出门时都会走 `irqentry_exit_to_user_mode`,在那里**统一处理**:投递挂起的信号、检查 `TIF_NEED_RESCHED` 决定要不要趁机重新调度。
- 这就是为什么信号投递和抢占调度,都"攒在回用户态的出口"做——因为那是唯一安全的、用户态可见状态已经一致的时机。

> **一句话**:`irqentry_enter`/`irqentry_exit` 是三种惊动**共用的进出门海关**。它不区分"哪种惊动",只区分"从哪来、回哪去"。三者的差异,全部封装在骨架**中间**那层各自的"分诊表 + 处理函数"里。这种"统一骨架 + 可替换中段"的结构,是内核处理"本质相同、细节各异"问题的经典手法——和内存篇"分层抽象、清晰接口解耦"一脉相承。

## 九、章末小结(也是本篇的总收束)

用一张"三种惊动"的比喻图把整个第 4 篇收口。

[全书导言](全书导言-一篇看懂内核全貌.md)里说,市民和政府之间有**三道门**——门铃(中断)、报警器(异常)、办事窗口(系统调用)。这四章我们走完了这三道门的全部细节。本章把它们摆在一起,看清了:

> **中断、异常、系统调用,本质都是"陷入内核"——CPU 从用户态切到内核态、保存现场、查表找人、处理、恢复现场返回。三者共用同一条过门 5 步流水线、同一种 pt_regs、同一段 `irqentry_enter/exit` 骨架。**

它们的区别,集中在三条分界线上:

1. **谁发起、同步还是异步**:中断异步外部、异常同步 CPU 自动、系统调用同步自愿。
2. **有没有当前进程、能不能阻塞**:中断没有当前进程、不能睡眠(逼出上半部/下半部);异常和系统调用关联当前进程、能睡眠。
3. **输入可信度与检查严苛度**:中断最松(硬件可信)、系统调用最严(用户不可信,seccomp / `array_index_nospec` / `copy_from_user` 层层设防)。

这三个认知,就是本篇的全部精华:

> **① 三者统一在"陷入内核"上。** IDT 是公共门牌表,pt_regs 是公共现场快照,`irqentry_enter/exit` 是公共进出门骨架。理解了其中一种,另两种就是它的变种。
>
> **② 差异全在"谁触发、有没有当前进程、输入可不可信"。** 这些差异不是随意的,是被每种惊动的"出身"决定的——外部硬件 vs CPU 自动 vs 进程自愿。
>
> **③ "硬中断不能阻塞"是被逼出的硬约束**,驱动了上半部/下半部、softirq/工作队列一整套推迟执行的设计;而"系统调用输入不可信"驱动了层层安全防线。

> **回扣全书主线**:本篇是全书旅程的**第 5 站——市政府的门与门铃**。它回答了一个贯穿前几篇却一直悬而未决的问题:**"内核凭什么能介入正在运行的进程?"** 答案就是陷入内核。没有这三道门,调度器得不到时钟心跳(进程篇)、page fault 无法兑现(内存篇)、`ls` 无法读目录(文件篇)——全书后续每一篇,都建立在本篇这三道门之上。
>
> - **承前**:本篇把内存篇 page fault(一个典型异常)、进程篇调度(靠时钟中断驱动)统一到了一个框架里——它们不是孤立机制,而是"陷入内核"流水线上的具体应用。
> - **启后**:本篇给**同步篇**铺好了全部地基。同步篇要讲"多核 + 中断 + 抢占三种并发来源"——其中中断(会随时打断你)和抢占(在系统调用/中断返回点触发)这两路,正是本篇埋下的。理解了本篇,你就明白为什么内核代码动辄要 `spin_lock_irqsave`(锁 + 关中断)、`preempt_disable`(关抢占)——它们防的,正是本篇这三道门带来的并发。

### 记住这三句话(本篇总纲)

1. **中断、异常、系统调用都是"陷入内核",共骨架(IDT / pt_regs / irqentry_enter)、差在三条线(谁触发、能否关联当前进程、输入可信度)。**
2. **硬中断里不能睡眠、不能磨蹭**——逼出上半部/下半部、softirq/tasklet/工作队列三档。
3. **系统调用是用户态/内核态墙的唯一合法门**——安全与隔离的根本,所以检查最严(seccomp / 防越界 / copy_from_user)。
