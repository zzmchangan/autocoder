## 这一篇的"一条主线"

整篇只讲一件事:

> **市民(进程)什么时候、为什么,必须"惊动"市政府(内核)——以及惊动之后,这扇"从市民世界通往政府世界"的门,到底是怎么过的。**

所有章节都挂在这条主线上。如果你在某一章迷路了,回到这句话,问自己:**"这次惊动,是谁发起的(硬件 / 市民自己 / 市民主动申请)、为了什么、门怎么过的?"** 就能找回方向。

这是全书旅程上的**第 5 站**:前面进程篇讲了市民怎么出生、谁此刻占用办事窗口;这一站要回答的是——**市民平时在自己的世界里闷头跑,凭什么内核能突然插进来接管?答案只有三个字:陷入内核(trap)。** 中断、异常、系统调用,本质都是这三个字的不同化身。

## 目录

### 第 1 章(全书第 20 章)· 中断与异常:硬件怎么拍政府的肩膀

- **核心问题**:网卡突然收到一个包、你按下键盘——硬件自己没有 CPU 时间,它凭什么能让正在运行的进程"让一让",把内核叫出来处理?内核又是怎么在一片混乱中,准确找到"这个中断该交给哪个驱动"的?
- **要讲清的"为什么"**:
  - 为什么不能让中断处理在"中断上下文"里干太久(不能睡眠、不能持有太久锁)——硬约束的根源。
  - 为什么要把处理拆成"上半部 / 下半部":上半部只抢时间关中断地记一笔,真正费时的活推迟到下半部(softirq / tasklet / 工作队列)。
  - IDT(中断描述符表)为什么是"门牌总表":一个编号(vector)如何精确路由到一个处理函数。
  - 中断控制器(APIC)、IRQ 号、vector、`irq_desc` 这一串抽象层为什么要叠这么多层。
- **关键源码**:`DEFINE_IDTENTRY_SYSVEC`/`DEFINE_IDTENTRY_IRQ`([arch/x86/include/asm/idtentry.h](../linux-6.14/arch/x86/include/asm/idtentry.h))、`handle_fasteoi_irq`([kernel/irq/chip.c](../linux-6.14/kernel/irq/chip.c))、`__handle_irq_event_percpu`([kernel/irq/handle.c](../linux-6.14/kernel/irq/handle.c))、`handle_softirqs`([kernel/softirq.c](../linux-6.14/kernel/softirq.c))。

### 第 2 章(全书第 21 章)· 时钟与定时器:城市的心跳

- **核心问题**:进程凭什么会"被调度走"?`sleep(1)` 之后进程凭什么能准时醒来?这一切都依赖一个最特殊的中断——时钟中断。它是怎么既驱动调度、又驱动所有定时器的?
- **要讲清的"为什么"**:
  - 时钟中断为什么是"全城心跳":它周期性地打断 CPU,内核借此知道"时间过了多少、谁该让出窗口、哪个定时器到点了"。
  - clockevents / hrtimer 这套"高精度定时器"为什么取代了老的 jiffies 轮:从"滴答驱动"到"按需编程下一次中断"。
  - `update_process_times` 这条线怎么把一次时钟滴答,扇出到记账、调度(`sched_tick`)、低精度 timer、RCU、POSIX CPU 定时器。
  - NO_HZ / tickless:为什么"空着也是空着"的时候,内核会**主动停掉时钟心跳**省电。
- **关键源码**:`sysvec_apic_timer_interrupt`([arch/x86/kernel/apic/apic.c](../linux-6.14/arch/x86/kernel/apic/apic.c))、`tick_handle_periodic`([kernel/time/tick-common.c](../linux-6.14/kernel/time/tick-common.c))、`hrtimer_interrupt`([kernel/time/hrtimer.c](../linux-6.14/kernel/time/hrtimer.c))、`update_process_times`([kernel/time/timer.c](../linux-6.14/kernel/time/timer.c))、`sched_tick`([kernel/sched/core.c](../linux-6.14/kernel/sched/core.c))。

### 第 3 章(全书第 22 章)· 系统调用:过关的艺术

- **核心问题**:市民(进程)不能自己碰公共资源(读文件、建进程、分配内存……),必须"去办事窗口申请"。这扇从用户态到内核态的门,到底是怎么过的——那条改变世界状态的指令(`syscall`)究竟干了什么?
- **要讲清的"为什么"**:
  - 为什么**必须**有这一关:不是性能取舍,而是**安全与隔离的根本**——没有它,任何程序都能直接读写硬件、改别人的内存。
  - `syscall` 指令到底替我们干了什么(swapgs / 换栈 / 切 CR3 / 构造 pt_regs),以及为什么 SYSRET 能比 IRET 快。
  - 参数怎么传(寄存器传 6 个 + 为什么不能让用户直接传指针)、`SYSCALL_DEFINE` 这套宏如何既定义函数又生成表项、又防越界。
  - `do_syscall_64` 的分诊:为什么先 `array_index_nospec` 再查表(防 Spectre 式越界读)。
- **关键源码**:[arch/x86/entry/entry_64.S](../linux-6.14/arch/x86/entry/entry_64.S) 的 `entry_SYSCALL_64`、[arch/x86/entry/common.c](../linux-6.14/arch/x86/entry/common.c) 的 `do_syscall_64` / `do_syscall_x64`、[arch/x86/entry/syscall_64.c](../linux-6.14/arch/x86/entry/syscall_64.c) 的 `sys_call_table`、`SYSCALL_DEFINE0(getpid)`([kernel/sys.c](../linux-6.14/kernel/sys.c))。

### 第 4 章(全书第 23 章)· 三者的统一视角:都是"陷入内核"

- **核心问题**:中断、异常、系统调用表面上是三件事,内核却用**同一套基础设施**(IDT / 异常入口 / 上下文切换 / `irqentry_enter`)来处理。它们的本质共性是什么?区别到底在哪?
- **要讲清的"为什么"**:
  - 统一性:三者都是"CPU 从用户态切到内核态、保存现场、查表找处理函数、处理完恢复现场返回"的同一条流水线。
  - 三条分界线:① **谁发起**(硬件 / CPU 自动 / 进程主动);② **可不可以预测、可不可以拒绝**(中断异步不可拒、异常同步由当前指令引发、系统调用同步自愿);③ **能不能阻塞 / 上下文能不能睡眠**。
  - 为什么异常和系统调用共用很多代码(都是"和当前进程绑定的同步陷入"),而中断是另一个世界(没有"当前进程"可言,跑在中断上下文)。
  - 用一张大图把三者、连同它们各自的数据结构(IRQ desc / trap_pf error code / pt_regs + syscall table)摆在一起对照。
- **关键源码**:`irqentry_enter` / `irqentry_exit`([kernel/entry/common.c](../linux-6.14/kernel/entry/common.c))——这是三者共用的"进/出内核上下文"骨架。
