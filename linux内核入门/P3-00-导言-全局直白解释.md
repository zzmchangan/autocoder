# 导言 · 一篇看懂进程篇全貌

> 这篇是进程篇的"地图"。读各章之前先读它,你会对"进程是什么、怎么生怎么死、谁此刻用 CPU"建立一个结实的直觉。
> 全文只讲两件事:**进程在干什么**,以及**为什么非得这么管**。
> 我会用一个贯穿本篇的比喻——**进程 = 城市里的市民,CPU = 办事窗口,调度器 = 叫号器**——帮你记住每一个机制。

---

## 一、一句话说清进程管理在干嘛

> **内核是这座城市的市政府,进程是市民。市政府的全部本事,就是让城里几千个市民,每个人都觉得"这台办事窗口(也就是 CPU)是我一个人的",而实际上窗口只有几个、几千人排着队飞速轮换,居然谁都没察觉。**

这是内核造的**第一个、也是最基础的幻觉:独占 CPU**。后面所有的设计——`task_struct`、`fork`、调度、上下文切换——都是在实现和维护这个幻觉。

你任何时候看不懂某个机制,就回来问自己一个问题:

> **"这个机制,是在'管市民的身份和资源',还是在'分配谁此刻去窗口'?"**

进程篇就这两大块。记住这个二分法,你就永远不会迷路:

- **管身份与资源那一侧**:市民是谁、有哪些家当(内存、文件、凭证)、怎么生、怎么死。
- **分配窗口那一侧**:几千市民抢几个窗口,叫号器(调度器)怎么决定"此刻叫谁",以及"把窗口从 A 手里交给 B"到底要动什么。

---

## 二、建立比喻:进程 = 市民

我们沿用全书的城市比喻。本篇涉及的角色:

| 城市比喻 | 对应的内核概念 | 一句话 |
|----------|----------------|--------|
| 市民 | **进程(process)** | 一个正在运行的程序 |
| 市民的分身 | **线程(thread)** | 同一个市民,同时干好几件事 |
| 户籍档案 | **`task_struct`** | 记录市民"是谁、住哪、有多少家当" |
| 市民证编号 | **pid** | 每个市民(含分身)唯一的编号 |
| 户号 | **tgid** | 一家人(同一进程的所有线程)共用的户号 |
| 市民的家当(房子) | **地址空间(`mm_struct`)** | 内存篇造的"独占整栋楼的幻觉" |
| 家当清单(打开的文件、目录位置) | **`files_struct` / `fs_struct`** | 市民手上持有的政府服务凭证 |
| 办事窗口 | **CPU** | 全城最稀缺的资源 |
| 办事大厅的叫号器 | **调度器(scheduler)** | 决定此刻叫哪个市民上窗口 |
| 每人能占窗口多久 | **时间片(time slice)** | 公平分配的"办事时长" |
| 市民累计"已办事时长"(打折扣后) | **vruntime** | CFS 判断"该轮到谁"的依据 |
| 把窗口从 A 交给 B 时换的东西 | **上下文(context)** | 寄存器、栈、页表 |
| 市民睡觉等通知(排号叫到了再说) | **阻塞睡眠(blocked)** | 状态之一 |
| 市民去世、留张死亡证明等家属认领 | **僵尸状态(zombie)** | 退场流程的中间态 |

这张表先扫一眼留个印象,下面我们**跟着一个市民从出生到死亡**走一遍,每一步都回来对应。

## 三、跟着一个市民走一遍:从 `fork` 到 `exit`

### 第 0 步:市民是谁——一个程序怎么就"活"了

你硬盘上有个 `ls` 程序。它躺在那儿,是**死**的——一串字节,什么也不会发生。

它怎么"活"?必须有个**已经在跑的市民**(`shell`)替它办两件事:

1. **`fork`**:shell 把自己"分身"出一个新市民(亲生骨肉);
2. **`exec`**:这个新市民"换一身衣服",把 `ls` 的代码塞进自己身体里,从此它不再是 shell,而是 `ls`。

> **关键认知**:`fork` 造的是**壳**(一个新的 `task_struct`),`exec` 填的是**芯**(新的代码和数据)。这两步合起来,才叫"启动一个新程序"。后面几乎所有程序的诞生都是这个套路。

那这个"市民"在内核眼里到底是个什么?是一份档案:`task_struct`。

### 第 1 步:市民的档案——`task_struct`

内核给每个市民建一份档案,叫 **`task_struct`**(定义在 [include/linux/sched.h](../linux-6.14/include/linux/sched.h))。这份档案大得吓人(几千字节,几百个字段),但分块看就清楚了:

- **身份**:我叫什么(pid)、我家户号是多少(tgid)、我的身份证/工作证是什么(cred:uid/gid/权限)。
- **家当**:我的房子在哪(`mm_struct`,也就是内存篇造的虚拟地址空间)、我打开了哪些文件(`files_struct`)、我的当前目录在哪(`fs_struct`)。
- **调度信息**:我应该归哪类调度(CFS/实时/…)、我已经"办了多久"的事(vruntime)、我的执行实体(`sched_entity`)。
- **硬件状态**:CPU 寄存器的值(`thread_struct`)——我不在窗口时,这些值得存着,下次叫我时恢复。
- **亲属关系**:我爹是谁(parent)、我生了哪些孩子(children)、全市市民链表里我的位置(tasks)。

> **为什么需要这么大一份档案?** 因为市民被"请离窗口"时(每次切换),它的**全部进度**都得有地方记下来:算到哪一步了(寄存器)、堆栈压到哪了(栈指针)、用的哪本门牌表(页表)……下次叫它回来,得能**一丝不差地接着跑**,就像从没离开过。这份档案,就是"市民的全部状态"。

### 第 2 步:市民 vs 分身——进程和线程为什么"是同一种东西"

很多人从教科书里学来一个印象:"进程是程序,线程是轻量级进程"。但在 Linux 内核眼里,这句话要反过来理解:

> **Linux 内核里没有"线程"这个独立的数据结构。线程和进程都是 `task_struct`。区别只在于:它们共享多少家当。**

- 一个**进程**,有自己的房子(`mm`)、自己的文件清单、自己的证件——`fork` 时**全都复制一份**。
- 一个**线程**,是"分身"——`fork` 时带上 `CLONE_VM`/`CLONE_FILES` 这类标志,**房子、文件、证件全共享**,只复制一份新的档案和栈。

所以"线程"在内核里就是个"资源共享版"的进程。一户人家(一个进程)住一个房子,户里的几个分身(线程)共用这个房子,但各有各的档案和栈、各抢各的窗口。

> **这个设计的妙处**:统一。内核调度、切换、管理的代码,**对进程和线程完全一样**——它们都是 `task_struct`,都进同一个调度器排队。没有"进程调度器"和"线程调度器"之分。简单、统一、好维护。代价只是一个"tgid"概念(同一进程的线程共享一个户号),用来对外暴露"进程"的假象。

### 第 3 步:市民的出生——`fork` 凭什么不复制内存

`fork` 要给子进程一份"和父进程一样"的地址空间。父进程可能占好几个 GB。如果老老实实复制——`fork()` 会从毫秒级变成秒级,web 服务器每秒 fork 上百次,直接卡死。

内核的解法,内存篇第 9 章已经揭过:**写时复制(COW)**。

> `fork()` 只复制**档案和页表(账本)**,**一行数据都不复制**。父子的 PTE 指向同一批物理页,且都标成只读。谁先写,谁触发 page fault,内核**当场只复制被写的那一页**。

所以 `fork()` 的代价是 O(页表大小),不是 O(数据大小)。这就是它又快又省的根基。这一篇第 17 章会从"进程出生"的视角,把 `copy_process` 怎么决定"共享还是复制"讲透——这恰恰是进程与线程的分水岭。

### 第 4 步:市民换芯——`exec` 不创建新进程

`fork` 出来的子进程,一开始和父进程长得一模一样(也是 shell)。要让变成 `ls`,得 `exec`。

> **`exec` 的视角转换,是本篇一个反直觉的点**:它**不创建任何新进程**,而是把**现有的进程"洗脑"**——把它的地址空间整个扔掉,换成 `ls` 的代码和数据。但 pid 不变、还是那个市民,只是"灵魂"换了。

为什么要这么设计(而不是"杀掉旧的、创建新的")?因为这样**调用者(`shell`)还能通过 pid 继续管理它**:等它结束、收它的退出码。如果"杀旧建新",这层父子关系就断了。第 17 章会顺着 `do_execveat_common` → `exec_mmap` 看"洗脑"具体怎么发生。

### 第 5 步:几千市民抢几个窗口——调度

市民造出来了,可办事窗口(CPU)只有几个。一个 4 核机器上同时有几百上千个进程想跑,**谁此刻能用?**

**调度器**就是那个叫号器([kernel/sched/](../linux-6.14/kernel/sched/))。它的目标是造出"独占 CPU"的幻觉,办法只有一个:

> **极快地轮换。** 每秒上千次,每个市民用一小片时间(几毫秒),快到市民根本感觉不到自己被换下来过。

"叫谁"可不是随便叫。Linux 把市民分成几个**调度类**,像办事大厅的"VIP 通道/普通通道/没人的话才叫的清洁工":

| 调度类 | 比喻 | 谁在里面 |
|--------|------|----------|
| stop | 城管/特勤,随时能插队 | 内核内部用,优先级最高 |
| deadline | 预约了死时间的 VIP | 实时任务(有截止时间) |
| realtime(rt) | VIP 通道 | 实时进程(SCHED_FIFO/RR) |
| **fair(CFS)** | **普通通道(绝大多数市民)** | **普通进程** |
| idle | 没人办事时,清洁工上场 | 空闲进程(没活干时) |

阶级是**严格的**:只要 VIP 通道有人,普通通道就排不上。绝大多数普通进程都走 **CFS(完全公平调度器)**。

CFS 的核心思想一句话:**给每个市民一个"虚拟已办时长"(vruntime),永远叫 vruntime 最小的那个上窗口。** 谁占便宜多了,vruntime 涨得快,就轮到别人;权重高(优先级高)的市民,vruntime 涨得慢,于是能多占。靠这一个量,就逼近了"按权重公平"。第 18 章会把它讲透。

### 第 6 步:切窗口到底换了什么——上下文切换

调度器决定"从 A 切到 B"之后,真正"换"的活儿叫**上下文切换**(第 19 章)。它换三层:

1. **寄存器/CPU 状态**:A 算到一半的中间结果(RAX、RIP、标志位…)全存进 A 的档案(`thread_struct`),再把 B 档案里的值装回 CPU。
2. **栈指针**:A 的函数调用栈压到哪了,存起来;换成 B 的栈。**这是切换里最"诡异"的一步**——栈指针一换,函数还没"返回",控制权就跑到 B 的栈上了。
3. **地址空间(页表)**:如果 A、B 是不同进程(不同房子),要把 CPU 用的门牌表(页表)从 A 的换成 B 的,顺便把过期的 TLB(门牌速查缓存)冲掉。

切换是有代价的:TLB 和 CPU Cache 被打冷,B 上来会有一阵"冷启动"的慢。所以"频繁切换 = 性能损失",这也是调度器要控制时间片不能太短的原因。

> **一个有趣的优化——lazy TLB**:如果切的是两个**内核线程**(没有自己的房子),它们根本不需要换页表,内核就"偷懒"不换,继续用上一个进程的页表。能省一次昂贵的页表重载。这叫"懒",又是全书"懒哲学"的一个体现。

### 第 7 步:市民的退场——`exit` 和僵尸

市民跑完了,要退场。但内核**不能当场把它拆干净**,原因很现实:

> 市民的"死亡原因"(退出码)还得交给它爹(parent)。它爹通常会调 `wait()` 来收尸。如果内核当场就把子进程的档案删了,它爹就查不到"这孩子怎么死的"了。

所以退场分两步:

1. **`exit`**:市民把房子退回房屋管理局(释放内存)、关掉文件、改状态为 `EXIT_ZOMBIE`(僵尸)——**档案留着**,里面记着退出码和一点统计信息(用了多少 CPU、多少内存)。从此它**不再占窗口、不再排队**,但档案还在。
2. **`wait`**:父进程来"收尸",读到退出码,内核这才把僵尸的档案彻底删除。一个市民才算真正入土。

> **僵尸的危害**:如果爹不调 `wait`(写崩了的父进程),僵尸的档案就永远留着。虽然它不占 CPU、不占内存,但**占一个 pid 和一份 `task_struct`**。僵尸堆多了,系统就建不出新进程了。这就是"僵尸进程"为什么是个问题。

## 四、收束:进程篇回答了什么

走完全程,你看到的进程篇,是在回答两个问题:

```
   "我是谁、有哪些家当"            "我此刻能不能用窗口"
        (身份与资源)                     (调度)
            │                              │
     task_struct                    调度器(sched_class)
     fork / exec                    vruntime / 时间片
     exit / 僵尸                    上下文切换
            │                              │
            └──────── 共同支撑 ────────────┘
                        │
              "独占 CPU 的幻觉"
```

这两个问题,加上内存篇的"独占内存的幻觉",构成了市民能"安心写程序"的全部前提:

- 我有自己的房子(内存篇造的幻觉);
- 我是独立的市民、有自己的家当(本篇的 `task_struct`);
- 我以为这台 CPU 是我一个人的(本篇的调度 + 切换)。

**市民活在这三个幻觉里,完全不知道底下市政府忙成什么样。**

### 记住这三句话,就抓住了本篇

1. **进程和线程在内核里是同一种东西(`task_struct`),区别只在共享多少家当。** `CLONE_*` 标志决定了"是分身还是新市民"。
2. **`fork` 只复制档案和账本(COW),`exec` 不创建新进程只洗脑。** 这两个反直觉点,是理解程序怎么"活"起来的钥匙。
3. **"独占 CPU"是假象,真相是飞速轮换。** CFS 靠 vruntime 决定叫谁,上下文切换换寄存器/栈/页表——切换有代价,所以不能太频繁。
