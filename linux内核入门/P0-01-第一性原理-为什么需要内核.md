# 第 1 章 · 第一性原理:为什么需要内核

> **前置**:这是全书的起点,前置是[全书导言](全书导言-一篇看懂内核全貌.md)——它已经把内核比作"一座城市的市政府",并给出"管资源 vs 造幻觉"的二分法。本章就把那篇导言里轻描淡写的"内核必须存在"这五个字,**追问到底**:凭什么必须存在?没有它会怎样?

> **核心问题**:为什么需要内核?它到底解决了什么"不解决就完全没法用"的问题?
>
> 这一章我们不碰任何具体机制(不调度、不分配、不读文件),只问一件事:**一台 CPU + 一片内存 + 一堆设备,如果上面没有任何"管理者",程序直接跑在裸硬件上,到底会出什么乱子?这个乱子,又为什么非得靠一个"站在所有程序之上、拥有最高权限"的特殊程序来解决?**
>
> **读完本章你会明白**:内核不是"帮程序干活的助手",而是"多个程序能同时活在一台机器上"这个局能成立的**先决条件**;它凭什么站在所有程序之上(特权);它毕生在卖的那**三个幻觉**是什么;以及——内核态和用户态那道生死线,在代码里是怎么落地的。

---

## 章首·一句话点破

全书导言里那句话:

> **内核是一座城市的市政府,它让每个市民(进程)都觉得自己是这座城市唯一的主人。**

但这句话是**结论**,不是**理由**。这一章要倒过来:先把结论收起来,从最底层问起——

- 一台没有内核的裸机,到底长什么样?
- 如果**直接让程序在上面跑**,会出什么事?
- "管"这件事,为什么非得是一个**拥有最高权限、独此一家**的特殊程序,而不能"每个程序自己管自己"?

我们一块一块拆。

---

## 一、先看清"没有内核"的世界长什么样

要理解为什么需要内核,得先看清"没有内核"是个什么处境。

> **比喻**:想象一座**没有政府的城市**。没有警察、没有法院、没有房屋管理局、没有市政服务。每个市民手里都**握着一把万能钥匙**,能打开城里任何一扇门——别人的家门、银行的金库、变电站的电闸、自来水厂的阀门。

这听起来很自由,对吧?但这恰恰是 1970 年代以前"裸机编程"(MS-DOS、早期单片机)的真实处境:**程序直接跑在硬件上,拥有全部权限,能读写全部内存、能直接驱动任何设备。**

我们先不评判,先看这种世界**会出什么乱子**。

### 乱子一:一个 bug,毁掉全城

第一个、也是最致命的问题:**没有任何隔离和保护**。

裸机世界里,程序 A 和程序 B 都在内存里跑,A 写地址 `0x1000`,B 也写地址 `0x1000`——它们写的是**同一间物理房间**。A 的代码有个 off-by-one 错误,写穿了 `0x2000`,而 `0x2000` 正好是 B 的关键数据,甚至是**内核/系统本身**的代码。

> **不这样会怎样**:一个进程的小 bug,直接污染另一个进程,甚至改坏了系统自身的代码。**一个程序崩溃 = 整台机器崩溃。** 这就是为什么那个年代的程序"动不动就把系统写死",为什么你按下 Ctrl-C 经常连机器一起卡死——因为根本没有什么"保护"可言。

### 乱子二:多程序没法共存

第二个问题:**没有仲裁,程序之间必然打架。**

假设你想让两个程序同时跑。它们都得用内存、都得用屏幕、都得用磁盘。可没人规定"你用这段内存、我用那段"——每个程序都得**自己**记住"我能碰哪些地址",还得和别的程序**商量**"你用那段、我用这段"。

> **不这样会怎样**:这套"商量"如果交给应用程序自己做,既低效又不可信——谁能保证某个程序不耍赖、不偷偷占走一大段?谁能保证两个程序对"谁先用磁盘"达成一致?**没有中立的仲裁者,多任务根本无从谈起。** 这就是为什么裸机时代基本上"一次只能干一件事"。

### 乱子三:硬件没法被安全、统一地访问

第三个问题,容易被忽略,却很要命。

一块磁盘、一张网卡、一块显卡,要驱动它们,得往特定的**设备寄存器**里写特定的命令序列——这套"操作手册"极其繁琐、每个设备还都不一样。如果让每个应用程序**自己**去驱动设备,那么:

- 每个程序都得**内置**对每种硬件的驱动代码——你的浏览器得自己会驱动网卡、你的编辑器得自己会驱动磁盘。程序会膨胀到无法维护。
- 更糟的是,设备是**共享**的。两个程序同时往磁盘发命令,磁盘会乱套;两个程序同时改显卡的帧缓冲,屏幕会花。

> **不这样会怎样**:要么每个程序都臃肿到自带所有硬件的驱动(还互相冲突),要么干脆大家都没法用硬件。

### 小结:三个乱子,指向同一个缺口

把三个乱子叠在一起:

| 乱子 | 缺的是什么 |
|------|-----------|
| 一个 bug 毁全城 | **隔离和保护**——谁能碰什么,得有人管 |
| 多程序没法共存 | **仲裁**——资源归谁用、谁先谁后,得有人定 |
| 硬件没法安全访问 | **统一接口 + 专属操作者**——得有人专门伺候硬件 |

三个缺口,合起来指向同一个需求:**这座城市,需要一个站在所有市民之上、拥有最高权限、专门管公共事务的政府。**

这个"政府",就是**内核**。

## 二、为什么必须是"站在所有人之上、独此一家"的特权程序

到这里,你也许会想:行,那给每个程序都发一份"管理公共事务"的权限不就行了?大家都是政府,谁也别欺负谁。

**不行。** 这是理解内核本质最关键的一步。

### 不这样会怎样:特权一旦普及,就不再是特权

特权(能碰任何内存、能驱动任何设备)之所以"特",正因为**绝大多数人没有它**。如果每个程序都拥有最高权限,那:

- "隔离和保护"立刻失效——A 有了特权,就能直接改 B 的内存,谁来拦?
- "仲裁"也失效——大家都持枪,凭什么听你的?

> **比喻**:警察之所以能维持秩序,正因为**只有警察有枪**。如果每个市民都发一把枪,那就没有警察了,只有一片混战。**特权必须被独占和集中**,否则等于没有特权。

### 所以这样设计:把全部特权,集中交给唯一一个程序

于是唯一可行的设计是:

> **挑出一个程序,把全部硬件权限都交给它,只给它。其他所有程序,一律剥夺最高权限,赤手空拳地运行。**

这个拿到全部权限的、独一无二的程序,就是**内核(kernel)**。它是系统里**唯一**能直接碰全部内存、直接驱动全部设备的程序。

- 别的程序想读个文件?自己不能碰磁盘——必须**申请**(系统调用),由内核替它读。
- 别的程序想要点内存?自己不能乱抢——必须**申请**,由内核分配。
- 别的程序想跟网卡说话?必须**申请**,由内核转发。

**内核不是"帮程序干活的助手",它是"唯一持枪的政府"。程序赤手空拳,凡事都得向政府申请,由政府持枪去执行。** 这就是为什么内核在英语里也叫 **supervisor**(监管者)或 **特权程序**(privileged program)——它站在所有程序之上。

> **一个关键推论**:正因为内核拥有全部权限,它必须是**整个系统里最可信、最没 bug 的那部分**。一个普通程序崩了,大不了杀掉它;内核崩了,全城陪葬(这就是"kernel panic")。所以内核代码的严谨性,要求远高于普通程序。这也是为什么内核开发那么难、那么慢、那么挑剔。

## 三、内核卖的三个幻觉(本章高潮)

现在我们知道了:**内核是一个垄断了全部特权的政府。** 但光是"垄断特权"还不够——如果内核只是个高高在上的独裁者,程序们依然没法好好干活。内核真正的高明之处,在于它用这份特权,**给每个程序编织了三个极其舒适的幻觉**。

这三个幻觉,就是内核存在的**全部意义**。整本书后面的每一篇,都是在实现其中一个或几个幻觉。

### 幻觉一:每个程序都以为自己独占了 CPU

**裸现实**:全城只有几个 CPU(几个核),可城里同时住着几千个程序。CPU 是最稀缺的资源。

**不造这个幻觉会怎样**:每个程序都得自己操心"我现在能用 CPU 吗?用多久?什么时候让给别人?"。这套协商如果交给程序自己,既乱又不可信。

**所以内核造这个幻觉**:**调度器**让每个程序都以为"CPU 是我一个人的"。办法是**极快地轮换**——每个程序用一小片时间(几毫秒),然后被悄悄换下、换另一个上,每秒切换上千次。快到程序(以及你)根本感觉不到自己被换下来过。

> 这是进程/调度篇的主角。本章你只需记住:**"我独占 CPU" 是个假象,真相是调度器在飞速分时。**

### 幻觉二:每个程序都以为自己独占了整片内存

**裸现实**:物理内存是全城共享的、有限的、还满是"窟窿"(被设备、BIOS 占走)的一块。

**不造这个幻觉会怎样**:程序得自己记住"我能用哪段物理地址",还得和别的程序商量不撞车——回到乱子一、二。

**所以内核造这个幻觉**:**虚拟内存**给每个程序发一本"私家门牌本",让它以为自己独占了从 0 开始、连续、几乎无限大的整片内存空间。它敲哪扇门,内核才临时给它配一间真房(page fault 兑现)。两个程序的"100 号门牌",物理上是不相干的两间房。

> 这是内存篇(已写)的主角,也是本书目前最完整的部门。它把"独占内存"的幻觉造得最彻底。

### 幻觉三:每个程序都以为"一切都是文件"

**裸现实**:磁盘、键盘、网卡、显卡,千差万别,驱动方式各不相同。

**不造这个幻觉会怎样**:每个程序都得会驱动每种设备(乱子三),代码膨胀、互相冲突。

**所以内核造这个幻觉**:**VFS(虚拟文件系统)** 让每个程序相信"世界上一切都是文件"——硬盘是文件、键盘是文件、网卡是文件、连进程信息都是文件(`/proc`)。不管底下是什么妖魔鬼怪,程序都用同一套"打开 / 读 / 写 / 关闭"的套路去对付。内核在背后负责把这些统一操作翻译成对各种设备的具体命令。

> 这是文件/存储篇的主角。Unix 的那句名言 **"everything is a file"**,说的就是这个幻觉。

### 三个幻觉的本质:用"假象的舒适"换"真实的可控"

把三个幻觉叠起来看,你会发现内核在做同一件事:

> **它在每个程序面前,把"千差万别、互相冲突、危险复杂"的裸硬件,伪装成"独占、统一、安全、好用"的舒适世界。**

程序活在这三个幻觉里,安心地写代码,**完全不知道底下真实的硬件有多乱**。这就是内核这份工作的核心——**不是替程序干活,而是替程序把真实世界挡在门外,只递给它一个干净舒适的幻觉**。

> **回扣全书主线**:导言说的"管共享资源 / 造独占幻觉"二分法,在这一节落到地面了。**造幻觉**(独占 CPU、独占内存、统一文件)是内核面向程序的那一面;**管资源**(分配 CPU 时间、分配物理页、伺候设备)是内核面向硬件的那一面。两个面,一体两面,缺一不可。

## 四、那道生死线:内核态 vs 用户态,特权是怎么真正实现的

讲到这儿,你心里一定有个挥之不去的疑问:

> 内核说"程序没有最高权限,凡事得申请"——**可程序凭什么听话?** 如果一个程序就是赖着不走、非要自己去碰磁盘、自己去改别人的内存,谁来拦它?软件拦软件,谁拦得住?

答案不在软件里,而在**硬件**里。这是理解内核最容易被忽略、却最根本的一层。

### 不这样会怎样:纯软件的特权,是纸糊的

如果"内核有权限、程序没权限"仅仅是个软件约定,那任何程序都能无视它——约定嘛,遵守是情分,不遵守也拿它没办法。那样的话,前面三个幻觉全是纸糊的:一个恶意程序直接绕过去,改内核代码、偷别的程序内存、乱写磁盘,内核毫无办法。

**所以,特权必须由硬件强制执行。**

### 所以这样设计:CPU 内置"特权级",硬件亲自把关

现代 CPU(以 x86 为例)在硬件层面就内置了**特权级(privilege level)**机制,俗称**环(ring)**。x86 有 4 个环(ring 0~3),但 Linux 只用两个:

- **ring 0(内核态)**:**最高权限**。能执行任何指令、访问任何内存、驱动任何设备。**内核就跑在 ring 0。**
- **ring 3(用户态)**:**最低权限**。一大堆"危险指令"(改页表、关中断、直接读写设备端口)在 ring 3 下**硬件直接拒绝执行**——你敢执行,CPU 当场触发异常,把控制权交回内核。**所有普通程序都跑在 ring 3。**

> **比喻**:ring 0 = 市政府(持枪);ring 3 = 普通市民(赤手空拳)。市民想干"持枪才能干的事"(读写磁盘、分配内存),硬件本身就不让他直接干——他必须去办事窗口(系统调用)申请,由 ring 0 的政府替他干。

**这道 ring 0 / ring 3 的线,不是软件约定,是 CPU 硬件画的。** 程序越不过去,因为 CPU 在每条指令执行前都会检查"当前在哪个环?这条指令允许在这个环执行吗?"。这就是内核特权能站得住的真正地基。

### 代码佐证:这道线在源码里长什么样

这道生死线,在内核源码里有一个极简洁、极漂亮的体现。看 x86 段选择子的定义:

```c
#define __KERNEL_CS		(GDT_ENTRY_KERNEL_CS*8)
...
#define __USER_CS		(GDT_ENTRY_DEFAULT_USER_CS*8 + 3)
```

([arch/x86/include/asm/segment.h:135-138](../linux-6.14/arch/x86/include/asm/segment.h#L135-L138))

读这两行,你会惊叹于设计的精妙:

- `__KERNEL_CS`(内核代码段)=`GDT_ENTRY * 8`,**没有加任何东西**。
- `__USER_CS`(用户代码段)= `GDT_ENTRY * 8 + 3`,**多了个 `+ 3`**。

这个 `+ 3` 不是随便加的。段选择子的**最低 3 位**是 **RPL(Requestor Privilege Level,请求特权级)**。`+3` 就是把 RPL 设成 `3`,也就是 **ring 3**;内核段不加,意味着 RPL = 0,也就是 **ring 0**。

> **内核态和用户态的区别,在代码里浓缩成一个 `+3`。** 加了 `+3`,你是赤手空拳的市民(ring 3);没加,你是持枪的政府(ring 0)。整个操作系统的安全大厦,就建立在这个小小的 `+3` 上。

再看旁边那行注释,把"市民是 ring 3"写死在了定义里:

```c
/* User mode is privilege level 3: */
#define USER_RPL		0x3
```

([arch/x86/include/asm/segment.h:47-48](../linux-6.14/arch/x86/include/asm/segment.h#L47-L48))

### 那市民怎么"申请"?——系统调用

市民(ring 3)想干持枪才能干的事(读文件、要内存、生个新市民),怎么办?他不能自己持枪,但他可以**敲办事窗口的玻璃,把要求递进去**,由 ring 0 的政府替他执行。

这个"敲窗口"的动作,就是**系统调用(system call)**。它是一条特殊的 CPU 指令(`syscall`),执行时 CPU **硬件**会:

1. 把当前特权级从 ring 3 **提升到 ring 0**(这是唯一合法的"越权"途径,且由硬件严格管控);
2. 跳到内核预先登记好的入口(系统调用处理函数);
3. 内核在 ring 0 替你把事办了,再把特权级降回 ring 3,把你送回用户态。

> 系统调用是**唯一合法的"市民进政府"通道**,而且全程被硬件监视。这一套机制,是边界篇(中断/异常/系统调用)的主角,本章你只需记住:**那道 ring 0/ring 3 的线,平时谁都越不过去,只有系统调用这一个口子能合法地穿过。**

## 关键源码精读:ring 0 / ring 3 这道线,以及"第一个市民"

本章是概念章,代码点到为止。我们只精读两处,把"特权边界"和"旅程起点"在源码里钉住。

### 1. 特权环:segment.h 里那几行

把刚才引用的几行放在一起看全貌:

```c
/* User mode is privilege level 3: */
#define USER_RPL		0x3
...
#define SEGMENT_RPL_MASK	0x3
...
#define __KERNEL_CS		(GDT_ENTRY_KERNEL_CS*8)
...
#define __USER_CS		(GDT_ENTRY_DEFAULT_USER_CS*8 + 3)
```

([arch/x86/include/asm/segment.h:47-48, 135-138](../linux-6.14/arch/x86/include/asm/segment.h#L47))

逐个对应前面的设计:

- `USER_RPL = 0x3`:**用户态就是 ring 3**,写死。
- `SEGMENT_RPL_MASK = 0x3`:取段选择子最低 2 位(特权级占 2 位,值域 0~3),用来在运行时判断"当前这段是几环"。
- `__KERNEL_CS` 不加 `+3` → ring 0;`__USER_CS` 加 `+3` → ring 3。

**一个 `+3`,划开了政府和市民、划开了"能持枪"和"不能持枪"、划开了"内核态"和"用户态"。** 这是全书所有"隔离、保护、安全"的地基。

> **为什么 x86 有 4 个环(ring 0~3),Linux 只用 2 个?** 因为两环(内核/用户)已经够用,设计上越简单越不容易出错。ring 1、ring 2 在 Linux 里基本闲置。这种"硬件给了 4 档,软件只用 2 档"的取舍,体现了内核"够用就好、KISS"的工程哲学。

### 2. 旅程的起点:第一个市民的诞生

全书旅程从"内核开张、第一个市民诞生"开始。这个"开张"动作,在源码里就一行:

```c
	rest_init();
```

([init/main.c:1099](../linux-6.14/init/main.c#L1099),位于 `start_kernel()` 的最后一行)

`start_kernel()` 是内核初始化的"大游行"——把内存、调度、中断……所有部门一个个挂起来。等所有部门就位,最后一行调用 `rest_init()`,字面意思"剩下的初始化"。它干的最重要的一件事,是**创造出第一个市民**:

```c
static noinline void __ref __noreturn rest_init(void)
{
	...
	pid = user_mode_thread(kernel_init, ...);   /* ① 生出 1 号市民(init) */
	...
	pid = kernel_thread(kthreadd, ...);          /* ② 生出内核线程的祖宗 */
	...
	cpu_startup_entry(CPUHP_ONLINE);             /* ③ 本 CPU 进入调度,开始"叫号" */
}
```

([init/main.c:697](../linux-6.14/init/main.c#L697),精简示意,省略 NUMA/优先级等细节)

注意 `cpu_startup_entry`——从这一刻起,**调度器开始工作了**(叫号器通电)。`rest_init` 本身这个"0 号进程"(`init_task`)是在编译时就静态定义好的:

```c
	set_task_stack_end_magic(&init_task);
```

([init/main.c:901](../linux-6.14/init/main.c#L901))

`init_task` 就是传说中的 **0 号进程(idle)**——全城所有市民的终极祖先。它没有"父亲",是编译期就钉死在内核镜像里的。`rest_init` 由它发起,生下 1 号进程(`init`,所有用户程序的祖先)和 `kthreadd`(所有内核线程的祖先)。

> **回扣旅程**:全书导言里说的"第 1 站:市政府开张",在源码里就是 `start_kernel` → `rest_init` → 第一个市民诞生。下一站(启动篇),我们会顺着这条线,看清楚 0 号进程是怎么从无到有、城市是怎么从一片漆黑亮起灯的。

## 章末小结

### 用城市比喻回顾本章

回到那座城市。这一章我们做了一件纯粹的事:**搞清楚"市政府(内核)为什么非存在不可"。**

答案不在任何一个机制里,而在"裸机世界"的三个乱子里:

1. **没有政府,人人持枪**(每个程序都有全部权限)→ 一个 bug 毁全城;
2. **没有仲裁,谁来分资源** → 多程序没法共存;
3. **没人伺候硬件** → 设备没法被安全、统一地访问。

这三个乱子,逼出了一个设计:**把全部特权集中交给唯一一个程序(内核),其他程序一律赤手空拳(ring 3)**。这个"集中特权"不是独裁的霸道,而是**多程序能共存的前提**——特权必须被独占,否则等于没有。

但内核的高明不止于此。它用这份垄断来的特权,**给每个程序编织了三个舒适的幻觉**:**独占 CPU、独占内存、一切都是文件**。程序活在这三个幻觉里,安心写代码,完全不知道底下真实的硬件有多乱。**这三个幻觉,就是内核存在的全部意义。**

而这一切能站得住,靠的是 **CPU 硬件画的 ring 0 / ring 3 那道生死线**——在代码里,它浓缩成一个 `+3`。

### 本章在全书主线中的位置

记住导言的二分法:**内核一半在"管共享资源",一半在"造独占幻觉"。** 这一章我们把这条主线**从根上立住了**:

- **垄断特权、ring 0/ring 3 这道线**,是"管资源"能成立的地基——没有它,程序能直接抢资源,内核管不了;
- **三个幻觉**,是"造幻觉"的全部分类——后面每一篇,都是这三个幻觉之一的展开。

后面每一章,你要做的只是不断回到这俩锚,问一句:"这章是在**加固特权边界 / 管资源**,还是在**编织某个幻觉**?"

### 五个"为什么"清单

如果你只能记五件事,记这五件:

1. **为什么需要内核**:裸机有三个乱子(bug 毁全机、多程序无法共存、硬件没法安全访问),逼出一个垄断特权的政府。
2. **为什么特权必须集中**:特权一旦普及就不再是特权(人人都持枪=没人持枪);只有集中给内核,隔离和保护才成立。
3. **内核在卖哪三个幻觉**:独占 CPU、独占内存、一切都是文件。这是它存在的意义,也是全书后文的全部主题。
4. **特权靠什么落地**:不是软件约定,是 **CPU 硬件的特权级(ring 0/ring 3)**。代码里就是一个 `+3` 的区别。
5. **程序怎么合法"越权"**:只有**系统调用**这一个口子,由硬件严格管控地把 ring 3 临时提到 ring 0。

### 想继续深入,该往哪钻

- **看特权环的定义**:[arch/x86/include/asm/segment.h](../linux-6.14/arch/x86/include/asm/segment.h),特别是 `__KERNEL_CS`/`__USER_CS` 那个 `+3`,以及 `USER_RPL`、`SEGMENT_RPL_MASK`。
