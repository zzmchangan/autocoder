# 第 6 篇 · 文件与存储:数据住进档案馆

> 本篇对应城市比喻里的**档案馆(文件系统 VFS)+ 地下仓库(磁盘/块设备)**。

## 本篇的"一条主线"

整篇只讲一件事:

> **一份住在又慢又会突然掉电的磁盘上的数据,内核是如何在它之上搭出"一切都是文件"的统一幻觉,既让市民用着舒服,又保证读得快、写得不丢。**

所有章节都挂在这条主线上。迷路时回来问自己一个问题:

> **"这一步,是在'造统一的文件幻觉'(VFS 侧),还是在'和又慢又不可靠的仓库打交道'(块设备侧)?"**

就能找回方向。这正好对应内存篇那个二分法在本篇的复刻:

- **VFS 侧 = 造幻觉**:把硬盘、键盘、网卡、管道……统统伪装成"文件",市民用同一套 open/read/write/close 套路。这是内核卖的**第三个幻觉**(前两个是独占 CPU、独占内存)。
- **块设备侧 = 管资源**:磁盘是真家伙,慢、笨、还会掉电。内核得把"读第 N 页"翻译成"去仓库取第 K 个货架",还要排队、合并、调度、记日志,让仓库别被挤爆、别把数据弄丢。

## 本篇脉络图

```
   市民:  fd = open("/etc/hosts", O_RDONLY);   read(fd, buf, 4096);
                      │  ① 统一的"文件"幻觉
                      ▼
            ┌──── VFS 四大对象 ────┐            ── 第 28 章
            │ superblock / inode   │
            │ dentry / file        │
            │  + 一张操作表(a_ops) │
            └──────────┬───────────┘
                       │  ② 某个真实仓库怎么把对象摆上货架
                       ▼
                 ext4 磁盘布局                ── 第 29 章
            (超级块 / 组描述符 / inode 表 / 数据块)
                       │  ③ 数据其实先到内存
                       ▼
                page cache                  ── 第 30 章
            (address_space:文件页的内存缓存,
             命中就直接返回,miss 才下仓库)
                       │  ④ miss 了,得真去仓库取
                       ▼
            块设备层(blk-mq)              ── 第 31 章
            bio → request → 排队/合并/调度
            → 提交给磁盘驱动
                       │  ⑤ 写回,而且要保证掉电不丢
                       ▼
        写回 writeback + 日志 journaling    ── 第 32 章
        (脏页攒一批再刷;改动先记流水再落盘)
                       │
                       ▼
                   地下仓库(磁盘)
```

走完这条线,你就能完整解释一行 `cat /etc/hosts` 到底经过了什么。

## 目录

### 第 28 章 · VFS:统一的"文件"幻觉

- **核心问题**:硬盘、键盘、网卡、管道、/proc 全不是一回事,为什么内核能让市民觉得"它们都是文件"?这层"统一"是怎么骗出来的?
- **要讲清的"为什么"**:
  - 不统一会怎样:每种设备一套 API,市民程序得认识几十种"方言",移植一个 `ls` 都得重写。
  - VFS 的解法:四张抽象卡片(superblock / inode / dentry / file)+ 每张卡片背后一张"操作表"(函数指针表)。同一套 `open/read/write`,底下指向谁,就调用谁的实现。
  - 为什么是"函数指针表"而不是 `switch-case`:这是 C 语言实现多态的标准手法,也是 VFS 能挂上百种文件系统的根本。
- **关键源码**:[include/linux/fs.h](../linux-6.14/include/linux/fs.h) `struct super_block`、`struct inode`、`struct file`、`struct file_operations`;[fs/read_write.c](../linux-6.14/fs/read_write.c) `vfs_read`。

### 第 29 章 · 一个真实文件系统怎么落地(以 ext4 为例)

- **核心问题**:VFS 只是"卡片和空操作表",真正把数据摆上磁盘货架的是 ext4 这种具体文件系统。磁盘上的字节到底是怎么组织的?它怎么和 VFS 的对象对上?
- **要讲清的"为什么"**:
  - 不规划布局会怎样:磁盘就是一长串扇区,不划分区域,找一个 inode 得全盘扫描。
  - ext4 的布局:把磁盘切成块组(block group),每组一套超级块副本、inode 位图、块位图、inode 表、数据块——为什么这么分( locality:元数据和数据尽量靠近)。
  - 一个 ext4_inode(磁盘上的) ↔ VFS 的 struct inode(内存里的):挂载时怎么读进来、操作表怎么挂上。
- **关键源码**:[fs/ext4/ext4.h](../linux-6.14/fs/ext4/ext4.h) `struct ext4_super_block`、`struct ext4_inode`、`struct ext4_sb_info`;[fs/ext4/super.c](../linux-6.14/fs/ext4/super.c) 挂载入口。

### 第 30 章 · page cache 再访:文件数据的内存面

- **核心问题**:内存篇第 11 章讲过 page cache 是"磁盘文件的内存缓存",这一章站在文件系统的视角看:一次 read/write 是怎么穿过 page cache 的?它和普通匿名内存到底什么关系?
- **要讲清的"为什么"**:
  - 不缓存会怎样:每次 read 都去转一圈慢磁盘,`ls` 都得卡半秒。
  - address_space:一个文件一份页缓存账本(`i_pages` 这棵 xarray),"文件偏移 → folio"的映射就靠它。
  - 读路径:filemap_fault / readahead 的预读策略,为什么"顺手多读几页"反而更快。
  - 写路径:write_begin/write_end 的"借页—拷贝—标记脏"三段式,为什么写文件比读文件复杂得多。
- **关键源码**:[include/linux/fs.h](../linux-6.14/include/linux/fs.h) `struct address_space`、`struct address_space_operations`;[mm/filemap.c](../linux-6.14/mm/filemap.c) `filemap_fault`、`generic_perform_write`。

### 第 31 章 · 块设备层:bio、request 与 I/O 调度

- **核心问题**:page cache 没命中,或写回脏页,真正的"去磁盘取/存"是怎么发起的?这个请求一路怎么排队、合并、调度,最后才递到磁盘驱动手里?
- **要讲清的"为什么"**:
  - 不合并会怎样:每次 4KB 一个请求,磁盘寻道忙死,吞吐量惨不忍睹。
  - bio(生物层,文件系统发出的原始 I/O 单元)→ request(请求层,blk-mq 把 bio 攒成的大单)的两层模型。
  - 为什么是 blk-mq(多硬件队列):现代 NVMe 每核一条队列,谁也不用抢锁;plug(插头)机制把同一进程的连续小 I/O 先攒在本地方再下放。
- **关键源码**:[include/linux/blk_types.h](../linux-6.14/include/linux/blk_types.h) `struct bio`;[include/linux/blk-mq.h](../linux-6.14/include/linux/blk-mq.h) `struct request`;[block/blk-core.c](../linux-6.14/block/blk-core.c) `submit_bio`;[block/blk-mq.c](../linux-6.14/block/blk-mq.c) `blk_mq_submit_bio`。

### 第 32 章 · 写回与掉电安全:脏页怎么回家、journaling 怎么保不丢

- **核心问题**:内存篇说过"写文件只是把页标记脏,真正落盘是后来的事"。那这个"后来"到底是谁、什么时候、按什么规则把脏页写回?更关键的——万一写到一半掉电,文件系统怎么保证不把档案改坏?
- **要讲清的"为什么"**:
  - 立即写回会怎样:每次 write 都同步刷盘,慢得没法用;完全延迟又会丢一大堆数据。
  - 平衡之道:balance_dirty_pages 按比例限流,写回线程 periodically/kick 两种触发,kswapd/回收也能逼着写回。
  - journaling(日志):为什么"先记流水、再改账"能扛掉电;ext4 的 ordered/journal/writeback 三种模式各自拿什么换什么。
- **关键源码**:[mm/page-writeback.c](../linux-6.14/mm/page-writeback.c) `balance_dirty_pages_ratelimited`;[fs/fs-writeback.c](../linux-6.14/fs/fs-writeback.c) `__writeback_single_inode`;[fs/jbd2/transaction.c](../linux-6.14/fs/jbd2/transaction.c) `jbd2_journal_start`。
