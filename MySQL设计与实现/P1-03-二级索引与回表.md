# 第 1 篇 · 第 3 章 · 二级索引与回表

> **核心问题**:上一章讲了"InnoDB 的表就是一棵以主键为顺序的聚簇索引 B+树,叶子页直接存整行"。可真实业务里按主键查只是少数——更常见的是按手机号、按用户名、按订单号查。这些"非主键列"上建的索引叫**二级索引(secondary index)**。问题来了:二级索引的叶子页里,为什么不直接存整行数据、而要存一个"主键值"再回到聚簇索引里把整行捞出来(这步叫**回表**)?这一回多花了什么?什么时候能把它省掉(**覆盖索引**)?说到底,**二级索引的本质,是用"存主键值而非行指针"这个天才决定,换来页分裂时不维护二级索引的便利,代价是多一次 B+树查找**。

> **读完本章你会明白**:
> 1. 二级索引为什么是"一棵独立的 B+树",它的叶子页里到底存了什么——是**索引列值 + 主键值**,不是行指针、也不是整行。
> 2. 为什么是"存主键值"而不是"存行指针"——聚簇索引页分裂/行移动时,行物理位置变了但主键值不变,所以**二级索引不用跟着改**;若存行指针,页分裂就要更新所有二级索引,灾难。
> 3. **回表**到底多花了什么——多一次 B+树查找(从二级索引叶子页里取出主键、回到聚簇索引 B+树再定位一次);源码里是哪个函数、哪一步触发回表、哪一步免回表。
> 4. **覆盖索引(covering index)** 凭什么免回表——查询要的字段全在二级索引里(含主键),就不用去聚簇索引;InnoDB 源码用一个 `need_to_access_clustered` 标志位裁决。

> **逃生阀**:如果一读觉得难,先记三件事——① 二级索引是另一棵 B+树,叶子页存"索引列 + 主键值";② 按二级索引查,先在二级索引 B+树里拿到主键值,再拿这个主键值去聚簇索引 B+树里取整行(这步叫回表);③ 如果要的字段二级索引里全有(含主键),就不用回表,这叫覆盖索引。

---

## 〇、一句话点破

> **二级索引的叶子页存的是"索引列值 + 主键值",不是行指针、也不是整行。所以按二级索引查一行要分两步:先在二级索引 B+树里拿到主键值,再回聚簇索引 B+树用主键取整行——这第二步叫"回表"。为什么这么设计?因为聚簇索引页分裂时行物理位置会变、但主键值不会变,存主键值的二级索引就**不用跟着页分裂维护**,省了海量写放大;代价是回表多一次 B+树查找。把"回表"省掉的,就是覆盖索引。**

这是结论,不是理由。本章倒过来拆:先讲二级索引凭什么是一棵独立的 B+树、它的叶子页里到底装了什么,再讲为什么是"存主键值"这个关键设计,然后讲回表的源码路径,接着讲覆盖索引怎么把回表省掉,最后讲最左前缀和联合索引的伏笔。

## 一、二级索引:一棵独立的 B+树

### 提出问题:P1-02 讲了表 = 主键 B+树,那按非主键查怎么办?

上一章 P1-02 我们讲透了一件事:InnoDB 的表本身就是一棵以主键为序的聚簇索引 B+树,叶子页直接存整行。所以 `SELECT * FROM t WHERE id = 10` 这种按主键的查询极快——一次 B+树定位,叶子页就是整行。

但真实业务里,绝大多数查询**不是按主键**:

```sql
SELECT * FROM users WHERE phone = '13800000000';       -- 按手机号查
SELECT * FROM users WHERE email = 'a@b.com';           -- 按邮箱查
SELECT * FROM orders WHERE order_no = 'ORD20240101001'; -- 按订单号查
```

这些列不是主键。**主键 B+树是按主键排的,在里面按手机号找,只能全表扫一遍**——几亿行的表扫一遍几分钟,业务根本等不了。所以必须给这些"高频查询的非主键列"也配上目录——这就是**二级索引(secondary index,又叫辅助索引)**。

### 不这样会怎样:朴素方案的两个陷阱

朴素地想,有两种方式做二级索引:

**陷阱一:直接给非主键列排序,叶子页存行指针(指向聚簇索引里的物理位置)。**
很多非 InnoDB 引擎就是这么做的(比如 MyISAM,索引叶子页存的是行的物理偏移)。听起来简单,但在 InnoDB 这种"表就是 B+树"的引擎里,有个致命问题——**聚簇索引的页会分裂、行会移动**。一条行原本在页 A,页 A 满了分裂,行被搬到页 B,它的物理位置就变了。如果二级索引存的是行指针,这行一搬,所有指向它的二级索引都得跟着改。一张大表可能有十几个二级索引,一次页分裂就要更新十几处,写放大灾难。

**陷阱二:二级索引叶子页也存整行数据(冗余一份)。**
这样查询是省事(二级索引里直接就是整行,不用回表),但代价更大——一行数据在聚簇索引存一份、在每个二级索引里都存一份,**存储翻几倍**;而且改一行数据要改十几处(聚簇索引 + 所有二级索引的整行副本),写放大同样灾难。

### 所以这样设计:二级索引是独立的 B+树,叶子页存"索引列值 + 主键值"

InnoDB 的选择,避开了两个陷阱:**二级索引是一棵独立的 B+树,叶子页按"索引列"排序,但叶子页里不存整行数据、也不存行指针,而是存「索引列值 + 主键值」**。

```
   表 users(id INT PRIMARY KEY, name VARCHAR, age INT)
   假设在 name 上建了二级索引 idx_name

   聚簇索引(主键 B+树,叶子页存整行):
              [内部节点:按 id 导航]
              /        |        \
       ┌──────────┐ ┌──────────┐ ┌──────────┐
       │ id=1,row │ │ id=2,row │ │ id=3,row │   ← 整行都在叶子页
       └──────────┘ └──────────┘ └──────────┘

   二级索引 idx_name(独立的 B+树,叶子页存 name + id):
              [内部节点:按 name 导航]
              /        |        \
       ┌──────────┐ ┌──────────┐ ┌──────────┐
       │ name="a" │ │ name="b" │ │ name="c" │   ← 只存索引列 + 主键
       │   id=3   │ │   id=1   │ │   id=2   │     没有 age、没有整行
       └──────────┘ └──────────┘ └──────────┘
```

所以一次 `SELECT * FROM users WHERE name = 'b'` 的完整过程是:

```mermaid
flowchart LR
    A["SELECT *<br/>WHERE name='b'"] --> B["① 在二级索引 idx_name<br/>B+树里定位 name='b'"]
    B --> C["② 拿到这行对应的主键值<br/>id=1"]
    C --> D["③ 回聚簇索引 B+树<br/>用 id=1 再查一次"]
    D --> E["④ 在聚簇索引叶子页<br/>拿到整行数据"]
    E --> F["⑤ 返回给客户端"]
    style B fill:#dbeafe
    style D fill:#fee2e2
```

第 ③ 步——拿主键值回聚簇索引再查一次——就叫**回表(bookmark lookup)**。它多了一次完整的 B+树查找(从聚簇索引根节点走到叶子页)。

> **钉死这件事**:二级索引 = 一棵独立的 B+树,叶子页按索引列排序,但叶子页里只有「索引列值 + 主键值」。按二级索引查一行,要"先查二级索引拿主键、再回聚簇索引取整行"两步,后一步叫回表。这是 InnoDB 和堆表引擎(如 MyISAM、PG 的二级索引)在**索引存储内容**上的根本差异:InnoDB 二级索引叶子页存主键值,堆表二级索引叶子页存行指针(指向堆里的位置)。

## 二、为什么存"主键值"而不是"行指针"——本章的灵魂

这一节是本章最重要的洞察。上一节说"二级索引叶子页存主键值",读者第一反应往往是:为什么不存行指针?存指针查起来不是更快(省掉回表那次 B+树查找,直接跳到聚簇索引的物理位置)?这正是 InnoDB 设计的精妙处。

### 提出问题:存指针岂不是更直接?

假设二级索引存的是行指针(指向聚簇索引里这条行的物理位置:`页号 + 页内偏移`),那么查 `WHERE name='b'` 的过程会变成:

1. 在 idx_name B+树里定位 name='b',拿到指针 `(页号=100, 偏移=200)`;
2. 直接按指针去页 100 的偏移 200 处读整行。

**只有一步 B+树查找 + 一次指针解引用**,听起来比"回表再查一次 B+树"快得多。那 InnoDB 为什么不这么做?

### 不这样会怎样:页分裂的连锁维护灾难

问题出在**聚簇索引是 B+树,B+树会分裂**。

聚簇索引按主键序维护。往表里不断插入新行,某个叶子页满了(16KB 装不下),就会**页分裂(page split)**——把页里的一半行搬到新页里。这一搬,原本在页 100 的行,可能被搬到页 105。**行的物理位置(`页号 + 偏移`)变了,但主键值没变**。

```
   页分裂前:行 id=5 在 页100 偏移 200
        ↓ 页 100 满了,分裂
   页分裂后:行 id=5 搬到 页105 偏移 150

   它的主键值还是 id=5(不变)
   它的物理位置 页100:200 → 页105:150(变了)
```

现在看两种存储方式在页分裂时的代价:

**如果二级索引存行指针**(指向页 100 偏移 200):
页分裂后这行搬到了页 105 偏移 150,所有指向它的二级索引**都得更新**。users 表上除了 idx_name,可能还有 idx_email、idx_phone、idx_union_user_id……十几个二级索引,每个索引里凡是指向"被搬动的行"的条目,全都要改。一次页分裂 → 几十上百个二级索引条目更新 → 又触发这些二级索引自己的页分裂/写 redo/写 undo……**写放大指数级爆炸**。

```
   存行指针的噩梦(页分裂时):
   聚簇索引页 100 分裂,行 id=5 搬到页 105
        ↓
   idx_name 里指向 id=5 的条目: 页100:200 → 页105:150  (要改!)
   idx_email 里指向 id=5 的条目: 页100:200 → 页105:150  (要改!)
   idx_phone 里指向 id=5 的条目: 页100:200 → 页105:150  (要改!)
   ... 十几个二级索引全要改
```

**如果二级索引存主键值**(存 id=5):
页分裂后这行搬到页 105,**但它的主键值还是 id=5**。二级索引里存的"主键值"完全不用动——二级索引条目还是 `name='b', id=5`,根本不在乎这行现在物理上在页 100 还是页 105。**零维护成本**。

```
   存主键值的从容(页分裂时):
   聚簇索引页 100 分裂,行 id=5 搬到页 105
        ↓
   它的主键值还是 id=5(不变)
        ↓
   idx_name 里存的 (name='b', id=5) —— 不用改!
   idx_email 里存的 (email='a@b', id=5) —— 不用改!
   ... 所有二级索引都不用改
```

> **不这样会怎样**:存行指针会让"聚簇索引页分裂"演变成"所有二级索引的连锁维护"——一次插入触发十几处索引更新,写放大爆炸。InnoDB 存主键值,把"行的物理位置"这个易变的量,替换成"主键值"这个**永不变**的量,让二级索引与聚簇索引的物理布局**彻底解耦**。聚簇索引随便分裂、随便搬,二级索引岿然不动。**这是 InnoDB 索引设计最精妙的一笔,没有之一。**

### 代价:回表多一次 B+树查找

解耦的代价是:**按二级索引查一行,要回聚簇索引 B+树再查一次**(因为二级索引只存了主键值,没存行的物理位置,拿主键值去聚簇索引找,本来就要走一遍 B+树)。

但这个代价**远小于维护指针的代价**——回表只是一次 B+树查找(几层内部节点 + 一个叶子页,通常 3~4 次内存/IO),而维护指针是每次页分裂都触发的海量写放大。**用"查询时多一次 B+树查找"换"写入时零二级索引维护",在 OLTP(写多读也多、且页分裂频繁)场景下稳赚**。

> **钉死这件事**:InnoDB 二级索引"存主键值而非行指针",本质是用**查询时多一次 B+树查找**的代价,换**写入时二级索引与聚簇索引物理布局解耦**的红利。聚簇索引页分裂时,行物理位置变、主键值不变,所以二级索引不用维护。这个设计让 InnoDB 在"频繁插入 + 多个二级索引"的 OLTP 场景下,写放大保持在可控范围。**反过来说,回表不是缺陷,是这个天才设计的必然代价**——所以才有覆盖索引来对症下药地消除它。

### 源码佐证:二级索引的"主键值追加"在哪发生?

这个"二级索引叶子页存主键值"的设计,在数据字典层面就固化了。当你 `CREATE INDEX idx_name ON users(name)`,InnoDB 在内部构建二级索引的 `dict_index_t` 时,会**自动把聚簇索引的主键列追加到二级索引的字段列表里**——哪怕用户没显式写。这发生在 `dict_index_build_internal_non_clust`:

```c
// storage/innobase/dict/dict0dict.cc
static dict_index_t *dict_index_build_internal_non_clust(
    const dict_table_t *table, dict_index_t *index) {
  ...
  /* Create a new index: 字段数 = 用户定义的字段数 + 1 + 聚簇索引的唯一列数 */
  new_index = dict_mem_index_create(
      table->name.m_name, index->name, index->space, index->type,
      index->n_fields + 1 + clust_index->n_uniq);              // L3177
  ...
  /* Copy fields from index to new_index */
  dict_index_copy(new_index, index, table, 0, index->n_fields); // L3187

  /* 标记 new_index 里已经包含哪些表列(避免主键列重复添加) */
  ...
  /* 把聚簇索引的唯一列(即主键列)逐个追加到二级索引的字段里 */
  for (i = 0; i < clust_index->n_uniq; i++) {                  // L3212
    field = clust_index->get_field(i);
    if (!indexed[field->col->ind]) {
      dict_index_add_col(new_index, table, field->col,         // L3216
                         field->prefix_len, field->is_ascending);
    } else if (dict_index_is_spatial(index)) { ... }
  }
  ...
  if (dict_index_is_unique(index)) {
    new_index->n_uniq = index->n_fields;                       // L3229
  } else {
    new_index->n_uniq = new_index->n_def;                      // L3231
  }
  new_index->n_fields = new_index->n_def;                      // L3237
  ...
}
```

参见 [dict0dict.cc:3150-3242](../mysql-server/storage/innobase/dict/dict0dict.cc#L3150-L3242)。

**这段代码干了三件事**:

1. **L3177**:创建新的内部索引时,字段数特意预留 `index->n_fields + 1 + clust_index->n_uniq`——多出来的位置就是给"主键列"和系统字段留的。这是"二级索引会比用户定义的多几列"的物证。
2. **L3212-3224**:核心循环——遍历聚簇索引的唯一列(即主键的 `n_uniq` 列),**逐个追加**到二级索引的字段里。如果某主键列恰好已经在二级索引里(比如联合索引 `(name, id)` 已经包含 id),就不重复加(`!indexed[field->col->ind]` 判断);空间索引特殊处理。
3. **L3228-3232**:设置 `n_uniq`(决定索引唯一性的列数)。**唯一二级索引**的 `n_uniq = 用户定义字段数`(因为用户声明了 UNIQUE);**非唯一二级索引**的 `n_uniq = n_def`(把追加的主键也算进唯一性判定里——这样 `(name, id)` 里即便 name 重复,加上主键 id 后每条二级索引条目都是唯一的,这对 B+树内部节点定位、对锁的粒度都关键)。

> **钉死这件事**:用户写 `CREATE INDEX idx_name ON users(name)`,InnoDB 内部真正存的索引是 `(name, id)`——主键 id 被悄悄追加。这件事用户看不见,但它在源码里**铁板钉钉**:每一棵二级索引 B+树的叶子页里,必然有"完整的主键值"。这是"回表能发生"的前提——回表用的就是叶子页里存的那个主键值。

`dict_index_t` 结构体里和"字段数"相关的三个核心位域,定义在 [dict0mem.h:1121-1127](../mysql-server/storage/innobase/include/dict0mem.h#L1121-L1127):

```c
unsigned n_uniq : 10;   /*!< 数量上决定索引唯一性的字段数:
                         用于决定 B+树内部节点能否唯一区分一条记录 */
unsigned n_def : 10;    /*!< 目前已定义的字段数(含追加的主键) */
unsigned n_fields : 10; /*!< 索引的实际字段数 */
```

`n_uniq` 是个关键值——B+树的非叶子节点只存前 `n_uniq` 个字段(够唯一区分),`row_build_row_ref_in_tuple` 提取主键时也按这个长度(`dict_index_get_n_unique(clust_index)`)。

## 三、回表:多花的那一次 B+树查找,源码里长什么样

### 提出问题:回表这一步,到底在源码里怎么走?

我们把回表讲清楚了"多一次 B+树查找"。但这次 B+树查找在 InnoDB 源码里具体是怎么发生的?它由什么条件触发、什么条件不触发?这节拆到函数级。

InnoDB 所有 `SELECT`(以及 `UPDATE`/`DELETE` 的 WHERE 定位)走的查询主入口,是 `row_search_mvcc`([row0sel.cc:4431](../mysql-server/storage/innobase/row/row0sel.cc#L4431))。这是个上千行的巨型函数,负责:在指定索引(聚簇或二级)上定位/扫描记录、处理 MVCC 可见性、加锁、ICP 下推、以及——**要不要回表**。

### 回表的触发条件:一个布尔标志位

`row_search_mvcc` 里决定"要不要回表"的,是一个标志位 **`prebuilt->need_to_access_clustered`**(见 [row0mysql.h:561-565](../mysql-server/storage/innobase/include/row0mysql.h#L561-L565)):

```c
unsigned need_to_access_clustered : 1;   /*!< if we are fetching
                               columns through a secondary index
                               and at least one column is not in
                               the secondary index, then this is
                               set to true */
```

注释说得很直白:**当我们通过二级索引取数据、且至少有一列不在二级索引里时,它被置 true**。换句话说——这就是"覆盖索引判定"的反面:`need_to_access_clustered = true` 意味着"二级索引覆盖不全,得回表";`false` 意味着"覆盖了,不用回表"。

这个标志位**在 SQL 准备阶段**就被设好(handler 层根据"要查哪些列"和"二级索引里有哪些列"对比得出),不是查询时动态判断。设它的地方在 `ha_innodb.cc` 的模板构建函数 [ha_innodb.cc:8554-8556](../mysql-server/storage/innobase/handler/ha_innodb.cc#L8554-L8556):

```c
// handler/ha_innodb.cc —— build_template 路径

  prebuilt->need_to_access_clustered = true;     // L8555
}
```

**`templ->rec_field_no == ULINT_UNDEFINED`** 的意思是:**这一列(查询要的某列)在当前索引里找不到对应字段位置**。也就是说,MySQL 要的某一列,这个二级索引里没有——那当然得回聚簇索引去取。整个查询的 `need_to_access_clustered` 只要碰到这么一列,就置 true。

> **钉死这件事**:覆盖索引判定,在源码层面就是这么朴素——**逐列检查"MySQL 要的这列,在选中的索引里有没有字段位置(`rec_field_no`)"**。所有列都有 → `need_to_access_clustered` 保持 false → 不回表(覆盖索引);只要有一列没有 → 置 true → 回表。**没有玄妙的"覆盖索引算法",就是逐列比对**。

而 `ha_innodb.cc:8660` 还有另一处初始化(`change_table_ptr`/索引切换路径):

```c

```

意思更直白:**如果选中的索引就是聚簇索引,初始置 false(本来就在聚簇索引里,没什么可"回表"的);如果是二级索引,初始置 false,然后逐列检查(在 L8554 那里)看要不要翻成 true**。

## 四、覆盖索引:把回表省掉

### 提出问题:回表能不能省?

回表多一次 B+树查找,看起来是固定代价。但有一种情况能省掉它:**如果查询要的所有列,二级索引里全有(含主键),那就不必去聚簇索引取整行——二级索引叶子页里就有全部需要的数据**。这就是**覆盖索引(covering index)**。

看这个例子:

```sql
-- users 表: id PK, name, age, email, phone
-- 假设有索引 idx_name_age (name, age)

SELECT name, age FROM users WHERE name = 'b';   -- 覆盖索引!
```

`idx_name_age` 的叶子页里存的是 `(name, age, id)`(name + age 是索引列,id 是自动追加的主键)。`SELECT name, age` 要的两列,name 和 age 都在二级索引里——**根本不用回表**。

```sql
SELECT name, age, email FROM users WHERE name = 'b';  -- 不覆盖
```

这次要 `email`,而 idx_name_age 里没有 email——**必须回表**去聚簇索引取 email。

### 不这样会怎样:朴素做法的浪费

朴素地看,无论查询要哪些列,都老老实实回表取整行。这在"只需要索引列"的查询里是巨大浪费——本来二级索引叶子页里就有要的数据,却还要多一次 B+树查找去聚簇索引取一堆用不上的列。

InnoDB 的做法(也是大多数 B+树引擎的做法):**在 SQL 准备阶段,逐列检查"查询要的列,选中的索引里有没有"——全有就标记"不用回表"**。前面讲过,这个标记就是 `need_to_access_clustered = false`。

### 所以这样设计:逐列比对,只要一列不在索引里就回表

覆盖索引判定的源码,就是第二节讲过的 [ha_innodb.cc:8554-8556](../mysql-server/storage/innobase/handler/ha_innodb.cc#L8554-L8556)——**逐列检查**。逻辑可以总结成一句:

> 如果"通过二级索引取数"且"至少一列不在二级索引里" → 置 `need_to_access_clustered = true`(要回表)。

反过来,**所有列都在二级索引里 → 保持 false(覆盖索引,不回表)**。

当 `need_to_access_clustered == false`,第二节 L5455 的那个 `if` 条件不成立,直接跳过回表分支,`result_rec` 就指向二级索引记录(`rec`),从二级索引记录里把需要的列抠出来返回 MySQL。这一步完全省掉一次 B+树查找——查询快很多。

> **钉死这件事**:覆盖索引不是一个独立的机制,它就是"回表判定"的反面——`need_to_access_clustered` 这个布尔位为零。SQL 写对了(只取索引列),InnoDB 自动帮你省掉回表;写错了(多取一列不相关的),就老老实实回表。**这是为什么 DBA 强调"只 SELECT 需要的列,别 SELECT *"——少取一列可能就让查询从覆盖索引变成回表,性能差几倍**。

### 一个高频误解:主键是"隐形覆盖"

有个容易踩坑的点:**任何二级索引的叶子页里,都自动包含主键值**(上一节 `dict_index_build_internal_non_clust` 讲的)。这意味着——**只要查询要的列是「索引列 + 主键」,就是覆盖索引**,哪怕用户没显式把主键加进索引。

```sql
-- idx_name(name) 这个索引,叶子页里其实是 (name, id)
SELECT name, id FROM users WHERE name = 'b';   -- 覆盖索引!因为 id(主键)本来就隐含在 idx_name 里
SELECT name, age FROM users WHERE name = 'b';  -- 不覆盖(age 不在 idx_name 里)
```

这个"主键隐含在所有二级索引里"的特性,是 InnoDB 独有的红利(MyISAM/PG 的堆表二级索引存的是行指针,不含主键)。**它让"按二级索引查 + 取主键"这种高频模式天然就是覆盖索引**——比如分页查询里 `SELECT id FROM users WHERE name LIKE 'a%' LIMIT 100`,只要 id 和 name,完全不用回表。

> **钉死这件事**:InnoDB 的二级索引叶子页 = 用户定义的索引列 + 全部主键列(自动追加)。所以"主键列"在任何二级索引里都是免费的——查询只要索引列和主键列,就是覆盖索引。这是 InnoDB 二级索引设计的额外红利,堆表引擎没有。

## 五、联合索引与最左前缀:回表的伏笔

讲完回表和覆盖索引,本节埋一个伏笔——**联合索引(compound index)和最左前缀(leftmost prefix)**。这一节是承上(二级索引机制)启下(P1-04 页结构、P6-21 索引调优),不展开调优细节,只讲清楚"联合索引在二级索引这套机制里是什么"。

### 联合索引:多列排序的二级索引

二级索引的索引列可以是多列——`CREATE INDEX idx_a_b ON t(a, b)`,这叫**联合索引**。它还是一棵 B+树,只是排序规则是"先按 a 排,a 相同按 b 排"。叶子页里存 `(a, b, 主键)`。

```
   联合索引 idx_a_b(a, b) 叶子页里的内容(按 a,b 序):
   ┌──────────────────┐
   │ (a=1,b=1) id=10  │
   │ (a=1,b=3) id=7   │   ← a=1 的连续
   │ (a=1,b=5) id=22  │
   ├──────────────────┤
   │ (a=2,b=2) id=15  │   ← a=2 开始
   │ (a=2,b=4) id=8   │
   ├──────────────────┤
   │ (a=3,b=1) id=30  │   ← a=3 开始
   └──────────────────┘
```

它和单列二级索引在"回表/覆盖索引"机制上**完全一样**——叶子页存索引列 + 主键,`need_to_access_clustered` 同样逐列判定。

### 最左前缀:为什么 WHERE 必须从最左列开始

联合索引 B+树是"先按 a 排,a 相同按 b 排"——这意味着**只有从最左列(a)开始的查询条件,才能利用 B+树的有序性定位**。

```sql
SELECT * FROM t WHERE a = 1 AND b = 3;    -- 走 idx_a_b(命中 a=1 后,在 a=1 段内找 b=3)
SELECT * FROM t WHERE a = 1;              -- 走 idx_a_b(命中 a=1 的整段)
SELECT * FROM t WHERE b = 3;              -- 不走 idx_a_b!(b 在 a=1 段、a=2 段、a=3 段里都有,无法用 a 顺序定位)
```

第三条 `WHERE b = 3`——idx_a_b 是按 a 排的,b 在 B+树里全局无序(只在每个 a 值的小段内有序),所以不能用 idx_a_b,要么全表扫、要么走别的索引。这就是**最左前缀原则**:WHERE 条件必须从联合索引最左列开始连续命中,B+树才能用。

### 与回表的关系:最左前缀决定"能不能用这个二级索引"

最左前缀和回表是两个层面的问题,但容易混:

- **最左前缀**决定"**能不能用**这个二级索引"(B+树能不能定位);
- **覆盖索引**决定"用这个二级索引后,**要不要回表**"。

一个查询能高效走二级索引,要同时满足:① 最左前缀命中(能用);② 最好覆盖(不回表)。这是 P6-21 索引调优的核心,本章只埋伏笔,不展开。

## 六、技巧精解

本章有两个最硬核的技巧值得单独拆透。

### 技巧一:二级索引"存主键值而非行指针"——用不变量换零维护

这是本章的灵魂,前面已经从动机和反例讲过,这里再从"为什么 sound(正确且不漏)"的角度钉死。

**精妙在哪**:InnoDB 把二级索引叶子页设计成存"主键值"而非"行指针",本质是**把一个易变的量(行物理位置)替换成一个永不变的量(主键值)**,从而让二级索引与聚簇索引的物理布局**彻底解耦**。

**为什么 sound(不会查错、不会漏)**:

1. **主键是逻辑标识,永不变**:InnoDB 的主键,一旦行被插入,主键值就不再变(UPDATE 不改主键;DELETE 整行删除,不会留下"主键值还在但行没了"的悬空指针——二级索引会在 purge 时一起清理)。所以二级索引里存的主键值,永远指向一个有效的主键(要么行还在聚簇索引里,要么行被删了二级索引条目也被删)。
2. **回表用主键值在聚簇索引 B+树查找,一定能找到**:聚簇索引 B+树按主键序组织,主键值是唯一的(聚簇索引 = 主键 + UNIQUE)。拿主键值去聚簇索引 B+树做等值查找,一定定位到唯一一条记录(或者发现它已被删除——见 `rec_get_deleted_flag` 处理)。**不会因为页分裂导致指针失效**——因为 B+树查找是逻辑查找(从根按主键比较),不是物理跳转。
3. **页分裂/合并对二级索引透明**:聚簇索引页 100 分裂成页 100 + 页 105,行的物理位置变了,但聚簇索引 B+树的结构(按主键序)没变,主键值还是那些。二级索引里的主键值,去分裂后的聚簇索引 B+树查找,照样找得到。

**反面对比:存行指针为什么不 sound**

如果二级索引存行指针(`页号 + 偏移`):

- 页分裂时,行从页 100 搬到页 105,**指针失效**。要么有机制同步更新所有二级索引的指针(海量写放大),要么允许指针暂时失效(查询出错)。
- 历史上有些引擎(MyISAM)用行指针,但 MyISAM 的表是堆表(无聚簇索引页分裂问题),行一旦插入位置就不变,所以指针稳定。**InnoDB 不能照搬**——它的聚簇索引 B+树天生会分裂,行指针天然不稳。
- 这就是为什么"MyISAM 存指针、InnoDB 存主键"不是随意选择,而是**各自的表组织方式决定的**:堆表 + 行指针稳(行不动);索引组织表 + 行指针不稳(页分裂行搬)→ 必须用主键值。

> **钉死这件事**:"二级索引存主键值"是个看似浪费(查询多一次 B+树查找)、实则天才(写入零维护)的设计。它的核心是**用逻辑标识(主键值)替代物理标识(行指针)**,让二级索引免于聚簇索引页分裂的连锁维护。这个设计的前提是"InnoDB 是索引组织表(聚簇索引按主键序)"——堆表引擎不存在这个问题(行物理位置稳定),所以它们可以存行指针。**两种存储范式(索引组织表 vs 堆表)决定了二级索引该存什么,这是一脉相承的设计**。对照《PG》那本讲的堆表 + 行指针二级索引,正好是 InnoDB 的反面。

### 技巧二:覆盖索引判定的朴素实现——一个布尔位

第二个技巧不那么"硬核",但很能体现 InnoDB 的工程哲学:**最关键的优化,往往用最朴素的判定**。

**精妙在哪**:覆盖索引判定——"查询要不要回表"——InnoDB 没有搞什么复杂的算法,就是**一个 1 比特的标志位 `need_to_access_clustered`,在 SQL 准备阶段逐列比对**([ha_innodb.cc:8554](../mysql-server/storage/innobase/handler/ha_innodb.cc#L8554))。

```c
if (!index->is_clustered() && templ->rec_field_no == ULINT_UNDEFINED) {
  prebuilt->need_to_access_clustered = true;     // L8555
}
```

逐列检查"MySQL 要的这列,在选中的索引里有没有字段位置(`rec_field_no`)"。只要碰到一列没有,整个查询就标记"要回表"。然后查询执行时(row0sel.cc:5455),一个简单的 `if (index != clust_index && prebuilt->need_to_access_clustered)` 决定走不走回表分支。

**为什么妙**:

1. **判定提前到准备阶段**:不是每取一行都判断一次,而是一次性算好、塞进标志位,执行时就是个布尔检查。**零运行时开销**。
2. **和模板系统天然配合**:InnoDB 的 `mysql_row_templ_t` 本来就是"逐列描述 MySQL 要怎么读这行"的结构,`rec_field_no` 是模板里的现成字段。覆盖索引判定只是顺手复用了模板信息,没有为它单独建数据结构。
3. **覆盖索引免的不只是 B+树查找**:回表那一次 B+树查找要拿聚簇索引页的锁、要走 buffer pool、可能触发 IO。覆盖索引省掉的不只是 CPU,还有**锁竞争**(不拿聚簇索引页锁)和 **buffer pool 压力**(不读聚簇索引页)——一举多得。

**反面对比:如果每行动态判断**

朴素地想,也可以在 `row_search_mvcc` 每取到一条二级索引记录时,动态判断"这行要不要回表"。但这意味着每行都跑一遍逐列比对——查询返回一万行就跑一万次。InnoDB 把它前置到准备阶段,执行时就是个布尔位读——**一万行还是一次判定**。

**进一层:模板系统的"双字段号"**

覆盖索引不命中时,InnoDB 有个细节:模板里每个列存了**两个字段号**——`rec_field_no`(在选中的二级索引里的位置)和 `clust_rec_field_no`(在聚簇索引里的位置)。见 [ha_innodb.cc:8901-8906](../mysql-server/storage/innobase/handler/ha_innodb.cc#L8901-L8906):

```c
if (index != clust_index && m_prebuilt->need_to_access_clustered) {
  /* Change rec_field_no's to correspond to the clustered index record */
  for (i = 0; i < m_prebuilt->n_template; i++) {
    mysql_row_templ_t *templ = &m_prebuilt->mysql_template[i];
    ...
  }
}
```

当要回表时,模板的"读这列的位置"要从"二级索引里的位置"切到"聚簇索引里的位置"——因为最终返回 MySQL 的整行是从聚簇索引记录里抠的。**一个列在二级索引和聚簇索引里的存储位置不同(索引列顺序不同),模板用两套字段号适配**。这是个细节,但能让你感受到 InnoDB 在"二级索引记录 vs 聚簇索引记录"之间切换的工程化处理。

> **钉死这件事**:覆盖索引判定,本质就是"逐列检查 + 一个布尔位"。InnoDB 把它做到了极致朴素——准备阶段一次判定,执行阶段零开销。这反映 InnoDB 的工程哲学:**热路径上的优化,宁可朴素到极致,也不搞复杂算法**。覆盖索引能省的是"每行一次 B+树查找",在返回大量行的查询里,这个省下来的累加是巨大的——而判定本身几乎免费。

## 七、一个完整例子:把全章串起来

用一个例子把全章机制跑一遍。表结构:

```sql
CREATE TABLE orders (
    id        BIGINT PRIMARY KEY,        -- 聚簇索引按 id
    order_no  VARCHAR(32),               -- 订单号
    user_id   BIGINT,
    amount    DECIMAL(10,2),
    status    TINYINT,
    INDEX idx_order_no (order_no),       -- 二级索引:叶子页存 (order_no, id)
    INDEX idx_user_status (user_id, status)  -- 联合二级索引:叶子页存 (user_id, status, id)
);
```

**查询一:覆盖索引**

```sql
SELECT order_no, id FROM orders WHERE order_no = 'ORD001';
```

1. 选中 idx_order_no(满足最左前缀,order_no 是最左列)。
2. 在 idx_order_no B+树里定位 order_no='ORD001',拿到叶子页记录 `(order_no='ORD001', id=123)`。
3. 准备阶段判定:要 order_no、id——都在 idx_order_no 叶子页里(order_no 是索引列,id 是自动追加的主键)→ `need_to_access_clustered = false`。
4. 执行:L5455 的 `if` 不成立,不回表。直接从二级索引记录抠 order_no、id 返回。**一次 B+树查找**。

**查询二:回表**

```sql
SELECT * FROM orders WHERE order_no = 'ORD001';
```

1. 选中 idx_order_no。
2. 在 idx_order_no B+树定位,拿到 `(order_no='ORD001', id=123)`。
3. 准备阶段判定:`SELECT *` 要所有列(user_id、amount、status 不在 idx_order_no 里)→ `need_to_access_clustered = true`。
4. 执行:L5455 的 `if` 成立,进回表分支。`row_build_row_ref_in_tuple` 从二级索引记录抠出主键 id=123,`clust_pcur->open_no_init` 用 id=123 在聚簇索引 B+树定位,拿到整行。**两次 B+树查找**。

**查询三:联合索引最左前缀 + 覆盖**

```sql
SELECT user_id, status FROM orders WHERE user_id = 100;
```

1. 选中 idx_user_status(user_id 是最左列,最左前缀命中)。
2. 在 idx_user_status B+树定位 user_id=100,拿到 `(user_id=100, status=?, id=?)`。
3. 准备阶段判定:要 user_id、status——都在 idx_user_status 叶子页里 → 不回表。**一次 B+树查找,覆盖索引**。

**查询四:最左前缀不命中**

```sql
SELECT * FROM orders WHERE status = 2;   -- status 不是 idx_user_status 的最左列
```

idx_user_status 按 (user_id, status) 排序,status 在全局无序,用不了 idx_user_status。优化器只能:走全表扫(聚簇索引),或走别的可用索引,或全索引扫 idx_user_status 再过滤。**索引选错,比没索引还慢**——这是 P6-21 索引调优要讲的。

> **钉死这件事**:把"二级索引是一棵独立 B+树 + 叶子页存索引列和主键 + 回表/覆盖由 `need_to_access_clustered` 裁决 + 最左前缀决定能不能用"这四件事拼起来,你就能解释 InnoDB 里绝大多数"查询走不走索引、走索引后快不快"的现象。这也是为什么 P6-21 索引调优的根基在本章——不懂二级索引和回表,explain 里的 `Extra: Using index`、`Using where`、`rows` 这些字段全是黑话。

## 八、二级索引 vs 堆表的二级索引——对照《PG》

承接《PG》那本(那里详细讲了堆表 + 二级索引)。这里只用一张表点透差异,不重复《PG》的内容:

| 维度 | **InnoDB 二级索引**(索引组织表) | **PG 二级索引**(堆表) |
|------|------------------------|------------------|
| 叶子页存什么 | 索引列值 + **主键值** | 索引列值 + **TID(行指针:页号+偏移)** |
| 找整行怎么做 | 拿主键值,**回聚簇索引 B+树**查一次 | 拿 TID,**直接按物理位置取堆里的行** |
| 聚簇索引页分裂/行移动 | 二级索引**不用维护**(主键值不变) | 堆表行不动(堆无序),**指针天然稳定** |
| 主键查询 | 聚簇索引一跳直达 | 主键索引 → 堆,两跳 |
| 二级索引查询 | 二级索引 → 聚簇索引,两跳(回表) | 二级索引 → 堆,两跳 |

关键洞察:**两种范式都是"二级索引两跳"**(一级索引定位 + 取整行),差别在"第二跳去哪":

- InnoDB 第二跳去聚簇索引 B+树(逻辑查找,主键值定位);
- PG 第二跳去堆文件(物理查找,TID 直接定位)。

InnoDB 的代价是回表那次 B+树查找(B+树有层数),PG 的代价是堆表的"主键查询也要两跳"(因为主键不是聚簇)。**两种范式在不同查询模式下各有优势**,没有绝对赢家——这就是为什么 OLTP(InnoDB 主键点查高频)选索引组织表,而很多分析型/通用型数据库选堆表。

> **承接**:这个对照在《PG》那本讲了堆表视角,本书 P7-23 收束时会再做总表。本章只负责把 InnoDB 二级索引这一面讲透。

## 九、章末小结

### 回扣主线

本章服务二分法的**"存储与索引"**这一面。主线"一条写,InnoDB 用 B+树找到位置、redo 保 crash 不丢、undo/MVCC 保并发读、锁保隔离"——本章讲的是"B+树找到位置"里**按非主键查找**的部分:二级索引是另一棵 B+树,找到的不是整行而是主键值,要回聚簇索引取整行(回表);把回表省掉的就是覆盖索引。

二级索引的设计,是 InnoDB"索引组织表"这个根本决策的延续——正因为表是聚簇索引 B+树(会分裂、行会搬),二级索引才必须存主键值而非行指针。**第 1 篇三章(聚簇索引 → 二级索引 → 页)层层递进,共同回答"InnoDB 怎么存、怎么找"**:

- P1-02 讲了表 = 主键 B+树(聚簇索引),数据在叶子页;
- **本章 P1-03 讲了按非主键查 = 走另一棵 B+树(二级索引)拿主键,再回聚簇索引取整行**;
- P1-04 会讲这些 B+树的页里到底长什么样(16KB 页结构、记录格式、页分裂)。

### 五个为什么

1. **为什么二级索引是独立的 B+树,而不是附在聚簇索引上的"另一套目录"?**——因为 B+树是有序结构,一种排序一棵树。按 name 排和按 id 排是两套不同顺序,只能各存一棵 B+树。这和"一本书按目录查 vs 按索引查"是两套独立检索结构同理。
2. **为什么二级索引叶子页存主键值而非行指针?**——聚簇索引页分裂时,行物理位置变、主键值不变;存主键值让二级索引与聚簇索引物理布局解耦,免去页分裂时对二级索引的连锁维护。代价是回表多一次 B+树查找。
3. **为什么按二级索引查一行要"回表"?**——因为二级索引叶子页只有"索引列 + 主键",没有整行。拿主键值去聚簇索引 B+树再查一次,才能取到整行。这一步在源码里是 `row_build_row_ref_in_tuple`(抠主键)+ `clust_pcur->open_no_init`(聚簇索引 B+树定位)。
4. **为什么覆盖索引能省回表?**——查询要的列全在二级索引叶子页里(含自动追加的主键),`need_to_access_clustered` 保持 false,跳过回表分支,直接从二级索引记录取数据。判定就是逐列检查"这列在不在索引里",朴素到一个布尔位。
5. **为什么主键列在任何二级索引里都"免费"?**——`dict_index_build_internal_non_clust` 建索引时自动把聚簇索引的主键列追加进二级索引(L3212-3224),所以"索引列 + 主键"天然就是覆盖索引。这是 InnoDB 索引组织表的红利,堆表没有。
