# 第 1 篇 · 第 2 章 · 聚簇索引:表的数据就在主键 B+树里

> **核心问题**:你在 PostgreSQL 里建一张表,表是表、索引是索引——表数据存在一个无序的"堆"里,索引只是个指向堆的目录。可到了 InnoDB,你随手 `CREATE TABLE t (id INT PRIMARY KEY, name VARCHAR(50))`,**表里的数据,凭什么就在那棵以 `id` 为顺序的 B+树里**?为什么 InnoDB 死认一个主键、连"没主键也要给你造一个"?这一章,把"索引组织表(IOT)"这个 InnoDB 区别于 PG 堆表的根本存储决策,从设计动机到源码实现,拆到清楚。

> **读完本章你会明白**:
> 1. **索引组织表(IOT) vs 堆表(heap)**:为什么 InnoDB 让"表 = 主键 B+树",而 PG 让"表 = 无序堆 + 独立索引";OLTP 高频主键点查上,这个选择的根本差异在哪里(主键查一跳 vs 两跳)。
> 2. **聚簇索引的本质**:叶子页里存的是**整行数据**而不是行指针——"表"和"主键索引"在 InnoDB 里是同一棵 B+树,源码里这条约定是怎么钉死的(`dict_index_t::is_clustered()`、`dict_table_t::first_index()`)。
> 3. **为什么 InnoDB 没有主键也要给你造一个**:显式 `PRIMARY KEY` → 第一个全列 `NOT NULL` 的 `UNIQUE` 索引 → 都没有就造 6 字节隐式 `DB_ROW_ID` 主键,这条决策链在源码里是怎么走的(server 层升格 vs InnoDB 造 `GEN_CLUST_INDEX`)。
> 4. **为什么强烈推荐自增整型主键**:插入顺序追加少页分裂、主键越小二级索引越省空间——这条"经验口诀"背后的 B+树物理原理。
> 5. 一处被老资料搞混的细节:**8.0 起 InnoDB 内部早就没有 `DICT_CLUSTER` 和 `dict_index_is_clustered()` 这两个老符号了**,它们分别变成了 `DICT_CLUSTERED` 常量和 `is_clustered()` 内联方法。

> **如果一读觉得太难**:先记住一句话——**InnoDB 的表就是一棵按主键排好序的 B+树,整行数据就挂在这棵树的叶子页上**;按主键查一行,顺着树走下去就到了数据,不用再跳第二跳。这是 InnoDB 和 PG 堆表最根本的区别,其他细节(隐式主键、自增主键为什么好)都是这句话的推论。

---

## 〇、一句话点破

> **在 InnoDB 里,"表"和"主键索引"是同一棵 B+树:整行数据就存在主键 B+树的叶子页里,按主键顺序排好。这叫聚簇索引,也叫索引组织表(IOT,Index Organized Table)。**

这是结论,不是理由。本章倒过来拆:先讲"表怎么存"这件事有哪几种范式(InnoDB 的 IOT vs PG 的 heap),为什么 OLTP 选了 IOT;再讲 InnoDB 在源码里怎么把"聚簇索引"这件事钉死;然后回答两个高频追问——**没主键怎么办**(显式主键 → 唯一索引 → 隐式 `DB_ROW_ID`,三级降级)、**主键该怎么选**(为什么自增整型几乎是最优解);最后做一个对照,把 IOT 和 heap 的取舍摊在桌面上。

---

## 一、先看清对手:"表怎么存"有两条根本不同的路

理解聚簇索引,先得跳出"InnoDB 的视角",看清世界上"一张关系表到底怎么存"这件事,本来就有**两条根本不同的路**。这两条路,决定了你后面看到的一切。

### 路线 A:堆表(heap)——表是表,索引是索引

**PostgreSQL** 是堆表的代表。在 PG 里:

- 表的数据,存在一个**无序的堆(heap)**里。堆就是一堆"行"的集合,新行往有空位的地方塞,行的物理位置和它的主键(或任何键)**没有顺序关系**。
- 索引是**独立于堆**的另一个数据结构(PG 默认也是 B+树的一种变体,叫 B-tree)。索引里存的是 `(key, TID)` 二元组——`key` 是你建索引的那个列,`TID`(Tuple ID)是"这行在堆里的物理位置"(哪个页、页里哪个槽)。
- 想按主键查一行?**两跳**:先查主键索引,拿到 `TID`,再拿着 `TID` 去堆里把那行捞出来。

```
   PG 堆表(heap)—— 表和索引是分离的两个结构:

        主键索引(独立 B-tree)              堆(heap,无序)
        ┌──────────────────┐              ┌──────────────────────┐
        │ id=10 → TID=A3   │──┐           │ ...                  │
        │ id=20 → TID=B1   │  │           │ [B1] id=20, name=... │ ← 第二跳:拿 TID 回堆取行
        │ id=30 → TID=A7   │──┼──────────▶│ [A3] id=10, name=... │
        └──────────────────┘  │           │ [A7] id=30, name=... │
                              └──────────▶│ ...                  │
                                  第一跳:查索引拿 TID            │
                                          └──────────────────────┘
                                          (行的物理位置和 id 无关)
```

### 路线 B:索引组织表(IOT)——表本身就是一棵索引树

**InnoDB** 走的是另一条路:

- **整张表的数据,就存在一棵以主键为顺序的 B+树里**。这棵树叫**聚簇索引(clustered index)**。
- B+树的**叶子页**,按主键顺序排列,**直接存整行数据**——不是行指针,是数据本身。
- 想按主键查一行?**一跳**:顺着 B+树从根走到叶子,叶子页里**就是那行数据**,不用再去别处捞。

```
   InnoDB 索引组织表(IOT)—— 表 = 主键 B+树:

                 [根/内部节点:只存主键 key,导航用]
                 /             |              \
          [叶子页]          [叶子页]          [叶子页]
          id=1,2,3..        id=N..2N          id=2N+1..
          row row row       row row row       row row row     ← 叶子页直接存整行
            │                 │                 │               (按主键序)
            └───── 叶子页之间双向链表,范围扫描顺序读 ─────┘
```

**关键洞察**:在 InnoDB 里,**"表"和"主键索引"是同一棵 B+树**。你 `CREATE TABLE` 时就算一个索引都不建,InnoDB 也会给你造一棵(下面第五节会讲怎么造)。这是 InnoDB 区别于 PG 堆表的**根本存储决策**,后面所有的"为什么"——为什么主键这么重要、为什么二级索引要回表、为什么自增主键好——都是从这一个决策长出来的。

### 那么问题来了:为什么要这么存?

这两条路,都有数据库在用(PG/Oracle 默认堆表,InnoDB/SQL Server 默认 IOT),不是谁一定对。但 InnoDB 选 IOT,是有明确动机的——这就是接下来几节要拆的。

> **钉死这件事**:理解聚簇索引的第一步,不是看 B+树怎么转,而是看清"表怎么存"本来就有 IOT 和 heap 两条路。InnoDB 选了 IOT,PG 选了 heap。这个分叉,是本书承接《PostgreSQL 数据库内核》那本 heap 表、并与之分道的起点——《PG》那本里讲的"表是无序堆、索引指向堆"那套,到 InnoDB 这里**整个不成立了**,表本身就是树。

## 二、为什么 OLTP 选 IOT:主键点查只要一跳

现在回到那个最根本的 why:**InnoDB 为什么要选 IOT,把数据塞进主键 B+树的叶子页里?**

### OLTP 最高频的访问模式:按主键点查一行

先看 OLTP(联机事务处理)的真实负载长什么样。电商下单、银行转账、用户登录——这些操作的 SQL,绝大多数长这样:

```sql
SELECT * FROM users    WHERE id   = 10086;   -- 查一个用户
SELECT * FROM orders   WHERE id   = 20240625001;  -- 查一个订单
UPDATE account SET balance = balance - 100 WHERE id = 99;  -- 改一个账户
```

它们的共同点:**按主键(`id`)精确查/改一行**。这是 OLTP 的标志性访问模式——**主键点查(primary key point lookup)**,占了绝大多数事务的绝大多数 IO。一个电商系统一天几十亿次请求,可能 90% 以上都是"按订单号/用户号查一行"。

### 堆表的主键点查:两跳,可能两次 IO

如果用堆表(PG 风格)来扛这种负载,一次主键点查要:

1. **第一跳**:查主键索引(B-tree),从根走到叶子,拿到目标行的 `TID`(堆里的物理位置)。
2. **第二跳**:拿着 `TID`,去堆里把那行数据捞出来。

如果索引和堆都不在内存里,这就是**两次磁盘 IO**(一次读索引页,一次读堆页)。即便都在 buffer pool / page cache 里,也是两次内存寻址、两次可能的缓存未命中。OLTP 一天几十亿次这个操作,**这第二跳的成本,会堆成一个天文数字**。

### IOT 的主键点查:一跳,叶子页就是数据

InnoDB 选 IOT 后,一次主键点查变成:

1. **一跳**:顺着主键 B+树从根走到叶子,**叶子页里就是那行数据**,直接拿到。

省掉了"拿着指针去堆里捞"这一跳。这一跳看似只是少了一次内存寻址,但它的真正价值在**缓存**:

- 堆表里,索引页和堆页是两套独立的页,要同时占用缓存(buffer pool / page cache)。热数据在索引里热、在堆里也热,缓存被"两份热"分摊。
- IOT 里,**热的主键索引页就是热的数据页**,一份缓存装下"索引 + 数据"两件事。同样的内存,能缓存更多热数据。

而且,主键**范围扫描**(`WHERE id BETWEEN 100 AND 200`)在 IOT 里极快——叶子页之间有双向链表,顺着链表顺序读就行,这是 B+树(不是 B 树)的招牌特性。堆表的范围扫描,每一行都得回堆跳一次,IO 模式更碎。

> **不这样会怎样**:如果 InnoDB 学 PG 用堆表,OLTP 最高频的"按主键查一行"就要多一跳——索引查一次、堆查一次。一天几十亿次操作,这第二跳的 IO 和缓存成本,足以让一个 OLTP 引擎的吞吐掉一大截。InnoDB 选 IOT,是**专门为"主键点查 + 主键范围扫是主流"的 OLTP 负载量身定做**的。

> **钉死这件事**:"主键查一跳 vs 两跳"是 IOT 和 heap 最本质的差异。这不是微优化,是**两种存储范式在 OLTP 核心负载上的根本分野**。PG 选 heap 是为了别的取舍(下面对照表会讲),InnoDB 选 IOT 是为了 OLTP 主键点查的极致性能。

## 三、那 IOT 的代价是什么?——二级索引要"回表"

IOT 不是免费的午餐。把整行数据塞进主键 B+树叶子页,带来一个绕不开的代价:**按非主键查,要"回表"**。

### 问题:按非主键(如 name)查一行

考虑这条 SQL:

```sql
SELECT * FROM users WHERE name = '张三';
```

`name` 不是主键。InnoDB 怎么查?它得用 `name` 上的索引(叫**二级索引**,secondary index)。但二级索引的叶子页里,**存的是什么**?

这里有一个 IOT 特有的、极其精妙的设计决策,值得单独钉死:

> **InnoDB 的二级索引,叶子页里存的是"主键值",而不是"行的物理位置(行指针)"。**

也就是说,`name` 索引的叶子页里,存的是 `('张三', id=10086)` 这样的二元组——`name` 的值 + 这行的**主键值**。查到主键值后,再拿着主键值,去**聚簇索引**里走一跳,把整行捞出来。这个过程叫**回表(table lookup by primary key)**。

所以这条 SQL 在 InnoDB 里是:

1. 查 `name` 二级索引,拿到主键值 `id=10086`。
2. 拿着 `id=10086` 去聚簇索引走一跳,拿到整行。

两次 B+树查找。这就是 IOT 的代价:**非主键查询多一次回表**。

### 精妙之处:为什么二级索引存"主键值"而不是"行指针"?

这里有个反直觉、但极其聪明的设计。你可能想:二级索引叶子页里直接存"行在聚簇索引里的物理位置(页号 + 槽号)",不就省了回表的第二次 B+树查找吗?

听起来对,但有个致命问题:**聚簇索引的页会分裂、行会移动**。主键 B+树在插入时,如果一个叶子页满了,会**页分裂**(P1-04 会讲),页里的行会被搬到新的物理页——行的物理位置就变了。如果二级索引存的是"行的物理位置",那每次聚簇索引页分裂、行移动,**所有指向这些行的二级索引都得跟着改**。一张大表上有好几个二级索引,每次页分裂要连锁更新一堆二级索引,写性能会塌。

InnoDB 的解法:**二级索引存主键值(逻辑位置),不存物理位置**。聚簇索引页分裂、行物理位置变了,**行的主键值不变**,所以二级索引一个字都不用改。回表时多走一次 B+树查找(B+树查找是 log 级、很快),换来的是**二级索引在页分裂时的零维护成本**。

> **这是为下一章 P1-03 铺垫的核心技巧,这里只点透**。二级索引的叶子页存主键值、回表、覆盖索引(二级索引里恰好有要查的字段就不用回表),这些是下一章《二级索引与回表》的主角。本章只把"聚簇索引是整行数据的家、二级索引要回这个家"讲透,回表细节往下走。

> **钉死这件事**:IOT 的根本代价是"非主键查询要回表"。但 InnoDB 用一个天才设计——**二级索引存主键值而非行指针**——把页分裂时二级索引的维护成本降到了零。这个设计是 IOT 能高效运转的关键之一,下一章会拆到源码。

## 四、源码里怎么钉死"表 = 聚簇索引":`dict_index_t` 和 `first_index()`

讲了这么多为什么,现在进源码,看 InnoDB 在代码层面是怎么把"聚簇索引"这件事钉死的。这一节是本章的源码重头戏。

### 数据字典的核心三件套:`dict_table_t` / `dict_index_t` / `dict_field_t`

InnoDB 在内存里,用三个核心结构体描述"一张表长什么样":

- [`dict_table_t`](../mysql-server/storage/innobase/include/dict0mem.h#L1925):**一张表的内存表示**。里面有表名、列定义、表空间、以及——**这张表上所有的索引(一个链表)**。
- [`dict_index_t`](../mysql-server/storage/innobase/include/dict0mem.h#L1067):**一个索引的内存表示**。里面有索引的类型(聚簇/唯一/普通)、根页号、字段列表、所属的表。
- `dict_field_t`:索引里的一个字段(哪一列、列前缀长度等)。

它们的关系是:`dict_table_t` 通过一个链表 `indexes` 挂着它所有的 `dict_index_t`;每个 `dict_index_t` 又通过 `table` 反向指针指向所属的 `dict_table_t`。

我们重点看 `dict_index_t` 里和聚簇索引有关的几个字段。先看真实源码(简化,保留逐字片段):

```cpp
// include/dict0mem.h
struct dict_index_t {                              // 第 1067 行
  /** id of the index */
  space_index_t id;
  ...
  /** back pointer to table */
  dict_table_t *table;                             // 第 1081 行 —— 指向所属表
  /** space where the index tree is placed */
  unsigned space : 32;                             // 第 1084 行 —— 表空间号
  /** index tree root page number */
  unsigned page : 32;                              // 第 1087 行 —— 根页号
  ...
  /** index type (DICT_CLUSTERED, DICT_UNIQUE, DICT_IBUF, DICT_CORRUPT) */
  unsigned type : DICT_IT_BITS;                    // 第 1094 行 —— 类型位域
  /** position of the trx id column in a clustered index record ... */
  unsigned trx_id_offset : MAX_KEY_LENGTH_BITS;    // 第 1098 行 —— trx_id 列在聚簇索引记录里的位置
  ...
  /** number of fields in the index */
  unsigned n_fields : 10;                          // 第 1127 行
  ...
  /** array of field descriptions */
  dict_field_t *fields;                            // 第 1175 行
  ...
  /** list of indexes of the table */
  UT_LIST_NODE_T(dict_index_t) indexes;            // 第 1200 行 —— 串进表的索引链表
  ...
  /** read-write lock protecting the upper levels of the index tree */
  rw_lock_t lock;                                  // 第 1244 行 —— 保护树上层节点的读写锁
```

(简化示意,完整结构体延伸到 1400 多行,含大量内联方法。)

注意第 1094 行的 `type`——它是一个**位域**(`unsigned type : DICT_IT_BITS`),用 OR 组合几个标志位来描述索引类型。这里有几个标志:

```cpp
// include/dict0mem.h,第 92-113 行
/** Type flags of an index: OR'ing of the flags is allowed to define a
combination of types */
/** @{ */
/** clustered index; for other than auto-generated clustered indexes, also
 DICT_UNIQUE will be set */
constexpr uint32_t DICT_CLUSTERED = 1;             // 第 97 行 —— 聚簇索引标志
/** unique index */
constexpr uint32_t DICT_UNIQUE = 2;                // 第 99 行 —— 唯一索引标志
/** insert buffer tree */
constexpr uint32_t DICT_IBUF = 8;                  // 第 101 行 —— change buffer 树
```

> **⚠️ 一处老资料搞混的细节(本章要纠正的)**:不少讲老版本(5.7 及更早)的博客和资料,会把聚簇索引标志写成 `DICT_CLUSTER`,把判断函数写成 `dict_index_is_clustered()`。在 **8.0 及之后的代码树里,这两个符号都不存在了**。我 Grep 整个 `include` 目录,`DICT_CLUSTER`(不带 ED)零匹配。新代码里:
> - 常量是 [`DICT_CLUSTERED`](../mysql-server/storage/innobase/include/dict0mem.h#L97)(值为 1)。
> - 判断方法变成了 `dict_index_t` 的内联成员 [`is_clustered()`](../mysql-server/storage/innobase/include/dict0mem.h#L1314)。
>
> 读源码或对照老资料时,务必注意这个版本差异。

### `is_clustered()`:一行判定聚簇索引

判断一个索引是不是聚簇索引,InnoDB 用一个极简的内联方法:

```cpp
// include/dict0mem.h,第 1314-1318 行
  bool is_clustered() const {
    ut_ad(magic_n == DICT_INDEX_MAGIC_N);

    return (type & DICT_CLUSTERED);               // 第 1317 行
  }
```

就一行核心逻辑:`return (type & DICT_CLUSTERED);`——检查 `type` 位域里有没有 `DICT_CLUSTERED` 这一位。这就是 InnoDB 判定"这个索引是不是聚簇索引"的全部代码。

相关的还有两个辅助判定(老风格函数,仍在 `dict0dict.ic`):

```cpp
// include/dict0dict.ic,第 111-113 行
/** Returns true if the index is an auto-generated clustered index. */
inline bool dict_index_is_auto_gen_clust(...) {
  return (index->type == DICT_CLUSTERED);         // type 恰好等于 1,说明是自动生成的
}
```

注意这个判定的精妙:`type == DICT_CLUSTERED`(等于 1),而不是 `type & DICT_CLUSTERED`(包含 1)。区别在哪?

- **显式主键**建出来的聚簇索引,`type` 会有 `DICT_CLUSTERED | DICT_UNIQUE`(= 1 | 2 = 3),既是聚簇、又唯一。
- **InnoDB 自动造的隐式聚簇索引**(下面第五节讲,名字叫 `GEN_CLUST_INDEX`),`type` 只有 `DICT_CLUSTERED`(= 1),**不唯一**——因为它靠一个内部的 6 字节 `DB_ROW_ID` 来唯一标识行,这个列不是用户定义的唯一约束。

所以 `type == 1` 专挑"自动生成的隐式聚簇索引",`type & 1` 挑所有聚簇索引。这种用位域 + 位运算同时表达"是不是"和"是哪一种"的设计,在 InnoDB 源码里随处可见,是 C/C++ 系统编程的常见手法(一个字段装多个维度的信息,省内存、省一次访存)。

### `first_index()`:聚簇索引永远在链表第一个

再看 `dict_table_t` 怎么存它的索引。关键在 [`dict_table_t::indexes`](../mysql-server/storage/innobase/include/dict0mem.h#L2140) 这个链表:

```cpp
// include/dict0mem.h,第 2140 行
struct dict_table_t {
  ...
  /** List of indexes of the table. */
  UT_LIST_BASE_NODE_T(dict_index_t, indexes) indexes;   // 第 2140 行 —— 索引链表头
  ...
};
```

一张表的所有索引,串在一个链表里。然后,InnoDB 有一个**铁律约定**:

> **聚簇索引,永远在这个链表的第一个位置。**

这个约定体现在 `dict_table_t::first_index()` 方法上:

```cpp
// include/dict0mem.h,第 2491-2500 行
  /** @return the clustered index */
  const dict_index_t *first_index() const {      // 第 2491 行
    ut_ad(magic_n == DICT_TABLE_MAGIC_N);
    const dict_index_t *first = UT_LIST_GET_FIRST(indexes);
    return (first);
  }
  /** @return the clustered index */
  dict_index_t *first_index() {                  // 第 2497 行
    return (const_cast<dict_index_t *>(
        const_cast<const dict_table_t *>(this)->first_index()));
  }
```

注意注释:**`@return the clustered index`**——"返回聚簇索引"。代码是"取链表第一个",注释直接说"返回聚簇索引"。这就是 InnoDB 把"表 = 聚簇索引"钉死的地方:**`table->first_index()` 这个调用,在 InnoDB 源码里到处都是,每一次调用,拿到的都是聚簇索引**。整本书你会看到无数次 `first_index()`,记住它的含义:**它就是这张表的"表本身"**。

这个约定为什么 sound(为什么一定成立)?因为 InnoDB 在建表时,有一个更强的保证:**一张 InnoDB 表,有且仅有一个聚簇索引**。建表时,InnoDB 必须先把聚簇索引建出来、放进链表首位置;之后建的所有二级索引,都跟在后面。这从源码上保证了 `first_index()` 永远返回聚簇索引。

handler 层甚至把这个事实写成了一个永远返回 `true` 的函数:

```cpp
// handler/ha_innodb.cc,第 6667-6670 行
/** Determines if the primary key is clustered index.
@return true, i.e. InnoDB always clusters on PK */
bool ha_innobase::primary_key_is_clustered() const { return (true); }
```

注释写得很直白:**"i.e. InnoDB always clusters on PK"**——InnoDB 永远在主键上聚簇。这个函数对 server 层的语义是:"你的主键,在我这儿就是聚簇索引,别想找别的"。这是 IOT 范式对外暴露的接口契约。

> **钉死这件事**:InnoDB 在源码里用三件事把"表 = 聚簇索引"钉死:① `dict_index_t::type` 位域里有 `DICT_CLUSTERED` 标志(第 97 行);② `is_clustered()` 内联方法判定它(第 1314 行);③ `dict_table_t::first_index()` 约定链表首节点永远是聚簇索引,且 `primary_key_is_clustered()` 永远返回 `true`。**在 InnoDB 源码里,你看到 `table->first_index()`,就是在访问"这张表的表本身"**。

## 五、没主键怎么办:显式主键 → 唯一索引 → 隐式 `DB_ROW_ID`

到这里,聚簇索引的本质清楚了:表 = 主键 B+树,整行在叶子页。但马上有个高频追问:

> **如果我建表时没写 `PRIMARY KEY`,InnoDB 怎么造聚簇索引?整行总得按某个顺序排吧?**

这个问题,引出了 InnoDB 一个很有意思的三级降级机制。一句话:**显式 `PRIMARY KEY` → 第一个全列 NOT NULL 的 `UNIQUE` 索引 → 都没有就造 6 字节隐式 `DB_ROW_ID` 主键**。下面把这条决策链拆到源码。

### 第一级:显式 `PRIMARY KEY`

最常见、也最推荐的情况:你建表时写了 `PRIMARY KEY`。

```sql
CREATE TABLE users (
  id   BIGINT NOT NULL AUTO_INCREMENT,
  name VARCHAR(50),
  PRIMARY KEY (id)
);
```

InnoDB 直接用 `id` 这棵 B+树当聚簇索引。整行按 `id` 排序存在叶子页。这是最干净的情况。

### 第二级:没有 PRIMARY KEY,但有"全列 NOT NULL 的 UNIQUE 索引"

如果你没写 `PRIMARY KEY`,但建了一个所有列都 `NOT NULL` 的 `UNIQUE` 索引,InnoDB 会**升格它当聚簇索引**。

```sql
CREATE TABLE users (
  id    BIGINT NOT NULL,           -- 没有 PRIMARY KEY
  email VARCHAR(100) NOT NULL,
  name  VARCHAR(50),
  UNIQUE KEY uk_email (email)      -- 但有一个全列 NOT NULL 的 UNIQUE 索引
);
```

这里 `uk_email` 会被升格成聚簇索引(整行按 `email` 排序)。这是 MySQL 官方文档明确写的规则。

> **⚠️ 一个源码层面的真相(写书时核查出来的,和很多博客讲的不一样)**:
>
> 很多人以为"挑哪个 UNIQUE 索引当聚簇索引"是 **InnoDB 引擎内部**做的决策。我 Grep 了整个 `storage/innobase/dict/` 目录(`dict0dict.cc` / `dict0dd.cc` / `dict0crea.cc`),**没有找到任何"遍历 UNIQUE 索引挑一个升格"的代码**。
>
> 真实情况是:**这个升格决策是 MySQL server 层(`sql/`)做的,不是 InnoDB 做的**。server 层在 `CREATE TABLE` 时,如果没有显式 `PRIMARY KEY`,就在所有 UNIQUE 索引里挑一个(规则:第一个所有列都 NOT NULL 的 UNIQUE 索引),把它标记为 `table_share->primary_key`。然后 InnoDB 通过 server 传下来的 `form->s->primary_key`(一个索引编号)知道"哪个是主键",据此给它打 `DICT_CLUSTERED` 标志。
>
> 所以 InnoDB 看到的世界很简单:**server 告诉我"第 N 个索引是主键",我就把第 N 个索引建成聚簇索引**。"挑 UNIQUE"那一步,在 server 层就消化掉了。这是 MySQL 两层架构(server + 引擎)分工的一个体现——有些语义在 server 层做,引擎只执行。

证据在 `ha_innodb.cc` 的 `create_index` 函数:

```cpp
// handler/ha_innodb.cc,第 12252-12258 行(简化示意)
  if (key_num == form->s->primary_key) {          // server 告诉我这是主键
    ind_type |= DICT_CLUSTERED;                   // 给它打聚簇标志
  }

  if (key->flags & HA_NOSAME) {                   // UNIQUE 约束
    ind_type |= DICT_UNIQUE;
  }
```

`form->s->primary_key` 就是 server 层定好的主键编号。InnoDB 据此 OR 上 `DICT_CLUSTERED`,这个索引就变成聚簇索引。

### 第三级:什么都没有,造 6 字节隐式 `DB_ROW_ID`

最极端:你没写 `PRIMARY KEY`,也没有任何 `UNIQUE` 索引。这种情况,server 层找不到任何东西可升格,`form->s->primary_key == MAX_KEY`(一个哨兵值,表示"没有主键")。这时,**InnoDB 自己造一个隐式聚簇索引**,名字叫 `GEN_CLUST_INDEX`(generated clustered index)。

这个 `GEN_CLUST_INDEX` 长什么样?它没有任何用户列参与排序——它的排序键,是 InnoDB 给每行偷偷加的一个 **6 字节的隐藏列:`DB_ROW_ID`**。每插入一行,InnoDB 给它分配一个全局递增的 `DB_ROW_ID`,整行就按这个 `DB_ROW_ID` 排在 `GEN_CLUST_INDEX` 里。

造 `GEN_CLUST_INDEX` 的源码,在 `dict0dd.cc` 的 `dd_fill_dict_index`(8.0 新数据字典加载表定义时调用):

```cpp
// dict/dict0dd.cc,第 3189-3201 行(简化示意)
  /* Create the keys */
  if (m_form->s->keys == 0 || m_form->s->primary_key == MAX_KEY) {
    /* Create an index which is used as the clustered index;
    order the rows by the hidden InnoDB column DB_ROW_ID. */
    dict_index_t *index = dict_mem_index_create(
        m_table->name.m_name, "GEN_CLUST_INDEX", 0, DICT_CLUSTERED, 0);
    index->n_uniq = 0;

    dberr_t new_err =
        dict_index_add_to_cache(m_table, index, index->page, false);
    ...
  }
```

注意几个细节:
- 名字硬编码成 `"GEN_CLUST_INDEX"`(常量 [`innobase_index_reserve_name`](../mysql-server/storage/innobase/include/dict0mem.h#L1051),值就是字符串 `"GEN_CLUST_INDEX"`)。
- 类型只给 `DICT_CLUSTERED`,**不给 `DICT_UNIQUE`**(这就是上面 `dict_index_is_auto_gen_clust()` 用 `type == 1` 而不是 `type & 1` 判定的原因——隐式聚簇索引不唯一)。
- `n_uniq = 0`:这个索引本身没有任何字段参与唯一性——唯一性完全靠随后注入的 `DB_ROW_ID`。

`GEN_CLUST_INDEX` 这个名字,你用 `SHOW INDEX FROM t` 或 `SHOW CREATE TABLE t` 是看不到的(MySQL 不会暴露它),但它真实存在。一些工具(如某些版本的 `SHOW TABLE STATUS`、information_schema)会显示它。

### 三个隐藏系统列:`DB_ROW_ID` / `DB_TRX_ID` / `DB_ROLL_PTR`

`DB_ROW_ID` 只是 InnoDB 给每行偷偷加的**三个系统隐藏列**之一。三个隐藏列的真实定义在 `data0type.h`:

```cpp
// include/data0type.h,第 177-191 行(逐字)
/** row id: a 48-bit integer */
constexpr uint32_t DATA_ROW_ID = 0;
/** stored length for row id */
constexpr uint32_t DATA_ROW_ID_LEN = 6;

/** Transaction id: 6 bytes */
constexpr size_t DATA_TRX_ID = 1;

/** Transaction ID type size in bytes. */
constexpr size_t DATA_TRX_ID_LEN = 6;

/** Rollback data pointer: 7 bytes */
constexpr size_t DATA_ROLL_PTR = 2;

/** Rollback data pointer type size in bytes. */
constexpr size_t DATA_ROLL_PTR_LEN = 7;

/** number of system columns defined above */
constexpr uint32_t DATA_N_SYS_COLS = 3;
```

> **⚠️ 另一处老资料常搞错的细节**:`DB_TRX_ID`(事务 ID)是 **6 字节**,不是有些博客说的 7 字节。7 字节的是 `DB_ROLL_PTR`(回滚指针)。三个隐藏列长度依次是 **6 / 6 / 7**。源码里甚至有静态断言钉死这件事(`static_assert(DATA_TRX_ID_LEN == 6, ...)`),谁也改不了。

这三个隐藏列是 InnoDB 事务和 MVCC 的命脉:

- `DB_ROW_ID`(6 字节):行的全局唯一 ID,仅当表没有主键也没有唯一索引时,作为隐式聚簇索引的排序键。**注意:只有"造隐式聚簇索引"时才真正用 `DB_ROW_ID` 排序;有显式主键或唯一索引的表,`DB_ROW_ID` 不会被分配**(下面源码会证明)。
- `DB_TRX_ID`(6 字节):**最后修改这一行的事务 ID**。这是 MVCC 判断"这个版本对某个事务可见不可见"的关键(P4 篇会讲)。
- `DB_ROLL_PTR`(7 字节):**回滚指针**,指向这行在 undo log 里的上一个版本。MVCC 的版本链,就是靠它串起来的(P3-10 / P4 篇会讲)。

**注意一个关键事实:`DB_TRX_ID` 和 `DB_ROLL_PTR` 在所有 InnoDB 表的聚簇索引里都有**(它们是 MVCC 和 undo 的命脉);但 `DB_ROW_ID` **只在隐式聚簇索引的表里才有**。这个区别,源码写得清清楚楚。

聚簇索引构造时,这三个隐藏列怎么注入?看 `dict0dict.cc` 的 `dict_index_build_internal_clust`:

```cpp
// dict/dict0dict.cc,第 3027-3042 行(简化示意)
  /* Add system columns */

  size_t trx_id_pos = new_index->n_def;

  /* Add ROW ID */
  if (!dict_index_is_unique(index)) {                       // 聚簇索引不唯一(= 隐式 GEN_CLUST_INDEX)
    dict_index_add_col(new_index, table,
                       table->get_sys_col(DATA_ROW_ID), 0, true);  // 才注入 DB_ROW_ID
    set_phy_pos(table->get_sys_col(DATA_ROW_ID));
    trx_id_pos++;
  }

  /* Add TRX ID */                                           // DB_TRX_ID 无条件注入
  dict_index_add_col(new_index, table,
                     table->get_sys_col(DATA_TRX_ID), 0, true);
  set_phy_pos(table->get_sys_col(DATA_TRX_ID));
```

后面的代码还会无条件注入 `DB_ROLL_PTR`。所以:

- **有显式主键或唯一索引的聚簇索引**:`DB_TRX_ID` + `DB_ROLL_PTR`(2 个隐藏列)。
- **隐式 `GEN_CLUST_INDEX`**:`DB_ROW_ID` + `DB_TRX_ID` + `DB_ROLL_PTR`(3 个隐藏列)。

判定逻辑 `if (!dict_index_is_unique(index))` 决定了"加不加 `DB_ROW_ID`"——显式主键和唯一索引都是 unique(`type & DICT_UNIQUE`),不加;只有 `GEN_CLUST_INDEX` 不 unique(`type == DICT_CLUSTERED`),才加。

### `DB_ROW_ID` 从哪来:全局递增的 `dict_sys->row_id`

最后一个问题:隐式聚簇索引表里,每行的 `DB_ROW_ID` 值怎么生成?

答案:**InnoDB 维护一个全局的、跨所有表共享的、单调递增的计数器**——`dict_sys->row_id`。

```cpp
// include/dict0dict.h,第 1033-1038 行
  /** the next row id to assign; NOTE: we only update the DICT_HDR_ROW_ID header
  once for every DICT_HDR_ROW_ID_WRITE_MARGIN increments, which means that after
  crash recovery the value found in this header might be smaller than the value
  in some of the rows. This is why we add DICT_HDR_ROW_ID_WRITE_MARGIN to it
  after recovery to prevent duplicates. */
  std::atomic<row_id_t> row_id;
```

注意几个细节:
- **`std::atomic<row_id_t>`**:原子变量,支持多线程无锁自增(不用加锁就能并发分配 `DB_ROW_ID`)。
- **跨所有表共享**:不是每张表一个计数器,是全局一个。这意味着 `DB_ROW_ID` 的值,跨表是可能"不连续"的(表 A 拿走了 100,表 B 拿走 101,表 A 再拿就是 102)——但因为 `DB_ROW_ID` 只在单表内做排序键,跨表连续不连续无所谓。
- **每 `DICT_HDR_ROW_ID_WRITE_MARGIN` 次才落盘一次**:为了性能,不是每次自增都把 `row_id` 写回数据字典头(那太慢)。每隔一定次数才刷盘。注释里讲了一个精妙的细节:crash 后,数据字典头里的 `row_id` 可能比实际分配出去的小(因为没刷的丢了),所以恢复时要**加一个 margin**(`DICT_HDR_ROW_ID_WRITE_MARGIN`)——宁可跳过一段 ID 也不用可能重复的 ID。这是数据库"crash 不丢 + 唯一性"的典型权衡:用一点空间(跳号)换正确性(不重复)。

分配新 `DB_ROW_ID` 的代码在 `dict0boot.ic`:

```cpp
// include/dict0boot.ic,第 36-64 行(简化示意)
static inline row_id_t dict_sys_get_new_row_id() {
  row_id_t id = dict_sys->row_id.load();
  while (true) {
    if ((id % DICT_HDR_ROW_ID_WRITE_MARGIN) == 0) {
      // 到了 margin 边界,加锁 + 刷盘
      dict_sys_mutex_enter();
      id = dict_sys->row_id.load();
      if ((id % DICT_HDR_ROW_ID_WRITE_MARGIN) == 0) {
        dict_sys->dict_hdr_flush_row_id();
        dict_sys->row_id.store(id + 1, std::memory_order_release);
        dict_sys_mutex_exit();
        return id;
      }
      ...
      dict_sys_mutex_exit();
    } else if (dict_sys->row_id.compare_exchange_weak(id, id + 1)) {
      return id;                                  // CAS 自增成功
    }
  }
}
```

这是一个典型的 **CAS(compare_exchange_weak)无锁自增 + 边界处加锁刷盘** 的模式:绝大多数情况下(不是 margin 边界),走 CAS 快路径,无锁、快;每 `DICT_HDR_ROW_ID_WRITE_MARGIN` 次,加一次锁、刷一次盘,保证 crash 后能恢复。这种"快路径无锁 + 慢路径加锁"的并发模式,在《Linux 内核》同步原语那本里讲过很多,InnoDB 里也随处可见。

消费这个计数器的地方,在 `row0ins.cc`:

```cpp
// row/row0ins.cc,第 3510-3526 行(简化示意)
  row_id_t row_id;

  ut_ad(node->state == INS_NODE_ALLOC_ROW_ID);

  if (dict_index_is_unique(node->table->first_index())) {
    /* No row id is stored if the clustered index is unique */
    return;                                       // 聚簇索引 unique,根本不分配 DB_ROW_ID
  }

  /* Fill in row id value to row */
  row_id = dict_sys_get_new_row_id();             // 只有隐式聚簇索引才走到这里
  dict_sys_write_row_id(node->row_id_buf, row_id);
```

**这段代码是"显式主键/唯一索引的表不分配 `DB_ROW_ID`"的铁证**:插入时,先看 `first_index()`(聚簇索引)是不是 unique——是,直接 return,不碰 `DB_ROW_ID`;否(隐式聚簇索引),才调 `dict_sys_get_new_row_id()` 分配一个。

> **钉死这件事**:InnoDB 的隐式主键机制,核心是三个源码事实:① server 层决定升格哪个 UNIQUE 索引当主键(InnoDB 只看 `form->s->primary_key`),InnoDB 自己不挑;② server 也没挑出来时,InnoDB 造 `GEN_CLUST_INDEX`,按 6 字节 `DB_ROW_ID` 排序;③ `DB_ROW_ID` 由全局原子计数器 `dict_sys->row_id` 分配,CAS 自增 + margin 刷盘,且有显式主键的表根本不分配。**这条决策链,从 server 层一路进到 InnoDB 内部,每一环都有源码对应,不是"约定俗成",是代码钉死的**。

### 一个实战推论:别让 InnoDB 造隐式主键

讲了这么多源码,有一个实战推论值得点出来:**生产环境,永远别让 InnoDB 造 `GEN_CLUST_INDEX`**。原因:

1. **`DB_ROW_ID` 是全局计数器,跨表争用**:所有"没主键的表"共享一个 `dict_sys->row_id`,高并发插入时,这个全局原子变量会成为热点(虽然 CAS,但仍是全局竞争点)。有主键的表不碰它。
2. **隐式聚簇索引不唯一**:这破坏了"主键天然防重复"的语义,业务层如果误插入重复数据,数据库挡不住。
3. **复制和binlog row 模式下,行用全部列定位**,没有主键的表复制效率低、且容易出问题。
4. **二级索引回表的开销更大**(下一章会讲),因为没有好的主键。

所以 InnoDB 的隐式主键是个**兜底机制**,保证"随便建个表也能存",但生产环境应该永远显式指定主键,而且最好是自增整型主键(下一节讲为什么)。

## 六、主键该怎么选:为什么自增整型几乎是最优解

讲完聚簇索引的本质和没主键的兜底,最后一个高频问题:**主键该怎么选?**

MySQL 老司机都背过一条口诀:"主键用自增整型,别用 UUID,别用字符串"。这条口诀是对的,但光背口诀不懂原理,就是"一知半解"。这一节,把这条口诀背后的 B+树物理原理拆透。

### 核心矛盾:主键的"有序性"决定页分裂频率

聚簇索引是一棵 B+树,叶子页按主键顺序排。B+树有一个根本特性:**当一个叶子页满了,再插入要插进这个页的行,会触发页分裂**(P1-04 会详细讲页结构,这里先点破后果)。

页分裂是什么?一个 16KB 的叶子页装满了,这时来了一个新行,主键落在中间——B+树只能把这个页**劈成两半**,把一半行搬到新页,腾出空间。页分裂的代价很大:

1. **数据移动**:一半的行要物理搬到新页。
2. **页指针更新**:父节点要新增一个指针指向新页。
3. **空间浪费**:分裂后,两个页各只有大约 50% 满,空间利用率下降。
4. **如果所有二级索引存的是物理位置,要连锁更新**(幸好 InnoDB 二级索引存主键值,不用改——这就是上一节讲的那个天才设计)。
5. **随机 IO**:新页通常分配在不连续的磁盘位置,顺序写变随机写。

所以,**让页分裂尽可能少发生**,是 OLTP 写性能的关键之一。

### 自增主键:顺序追加,几乎不分裂

如果你用 `BIGINT AUTO_INCREMENT` 当主键:

- 每次插入,新行的主键 = 上一行 + 1,永远比所有现有行都大。
- 新行永远插入到 B+树**最右边的叶子页**的末尾。
- 最右页满了才分裂(分裂出新的最右页),中间页永远不分裂。
- **写入是顺序的**(永远往右追加),磁盘 IO 是顺序写,对 SSD/HDD 都友好。

```
   自增主键的插入(永远追加到最右页):

   ... ← [页 N-1 满] ← [页 N 半满] ← [新行总往这插]   顺序写,顺序追加
                                              ↑
                                         自增主键的下一个值
```

而且,最右页分裂时,因为是"在末尾分裂",旧页可以保持满,**没有 50% 的空间浪费**(数据不需要从中间搬走,只是新行去新页)。这是 InnoDB 在最右页分裂上的一个优化(顺序插入时的填充因子优化)。

### 随机主键(UUID / 字符串):疯狂页分裂

如果你用 `UUID` 或随机字符串当主键:

- 每次插入,新行的主键是随机的,可能落在 B+树**任何一个叶子页**里。
- 每个叶子页都可能被插满、都可能分裂。
- **写入是随机的**:新行分散到各个页,缓存命中率下降,磁盘 IO 模式恶化。
- **页分裂频繁**:大量页处于半满状态,空间利用率低(典型 UUID 主键表,空间占用比自增主键表多 30%-50%)。

```
   UUID 主键的插入(随机位置,各处分裂):

   [页1]←新行  [页3]←新行  [页5]←新行  [页7]←新行   各处都可能分裂
      分裂 ↗       ↖ 分裂      分裂 ↗       ↖ 分裂
   (大量页半满,空间浪费,随机 IO)
```

### 自增主键小:二级索引更省空间

还有一层好处,承接上一节"二级索引存主键值":

- 每个二级索引的叶子页里,每条记录都存一份主键值(为了回表)。
- 主键越小(`BIGINT` 8 字节 vs UUID 16 字节 vs 字符串可能几十字节),二级索引的每条记录就越小。
- 同样大小的叶子页,能装下更多记录——二级索引更紧凑、缓存命中率更高、磁盘占用更小。

一张有 3 个二级索引、1 亿行的大表,主键从 8 字节(BIGINT)换成 16 字节(UUID),仅二级索引部分就要多花 `(16-8) * 3 * 1亿 ≈ 2.4GB`。这还只是磁盘,内存里缓存的热数据也相应变少。

> **不这样会怎样**:如果用 UUID 或随机字符串当主键,① 写入疯狂页分裂(性能下降、空间浪费);② 二级索引膨胀(每个二级索引的每条记录都存一份大主键)。两害叠加,OLTP 写吞吐和缓存效率都会显著下降。自增整型主键,在 IOT 里几乎是**免费午餐**:顺序写、少分裂、小主键、二级索引省。这就是为什么"InnoDB 强烈推荐自增整型主键"——不是习惯,是 B+树物理特性决定的。

> **钉死这件事**:"主键用自增整型"这条口诀,根子在 B+树的页分裂物理:① 顺序主键 = 顺序追加 = 少分裂 + 顺序 IO;② 小主键 = 二级索引紧凑。这是 IOT 范式下,存储引擎对你最友好的一种主键选择。如果业务允许,永远优先自增整型主键。

## 七、IOT vs heap 全景对照:InnoDB 为什么这么选,PG 为什么那么选

讲了这么多 InnoDB 的选择,一个公平的问题是:**PG 选 heap,是 PG 错了吗?**

当然不是。IOT 和 heap 是**两种各有取舍的范式**,各有擅长的场景。InnoDB 和 PG 的选择,反映的是**两个数据库的设计取向不同**,不是谁对谁错。下面用一个全景对照表把它们摊开:

| 维度 | **InnoDB(IOT 索引组织表)** | **PostgreSQL(heap 堆表)** |
|------|----------------------------|---------------------------|
| 表的数据存哪 | 主键 B+树叶子页(表 = 主键索引) | 无序堆(heap) |
| 主键点查 | **一跳**(树走到叶子就是数据) | **两跳**(索引拿 TID → 堆取行) |
| 非主键查询 | 二级索引 → 回表(走聚簇索引) | 任意索引 → 回堆(都走堆) |
| 主键范围扫 | 叶子页链表顺序读,极快 | 每行回堆,IO 较碎 |
| 没主键时 | 造隐式 `GEN_CLUST_INDEX`(按 `DB_ROW_ID`) | 堆照样能存(TID 唯一) |
| 二级索引存什么 | **主键值**(逻辑位置,页分裂零维护) | **TID 物理位置**(行移动要更新索引) |
| 二级索引大小 | 取决于主键大小(存主键值) | 固定(存 TID) |
| 表的物理重排 | 难(整棵 B+树) | 易(VACUUM 整理堆) |
| 聚簇能力 | 天生聚簇(数据按主键聚簇) | 需 `CLUSTER` 命令显式做,且是一次性的 |
| OLTP 主键点查 | **强**(一跳) | 中(两跳) |
| 跨索引灵活查询 | 弱(都得回主键树) | 中(都回堆,堆是中立位置) |

从这个表能看出两件事:

**1. InnoDB(IOT)在"主键为中心的 OLTP"上是量身定做的。** 主键点查一跳、主键范围扫顺序读、自增主键少分裂——这些都是 OLTP 最高频操作的极致优化。InnoDB 押注"OLTP = 主键点查为主",赢了。

**2. PG(heap)押注的是另一件事:灵活和简单。** 堆表把"数据在哪"和"怎么查"完全解耦——堆就是个无序行集合,你想怎么索引都行,索引之间是平等的(都指向堆,没有谁是"主")。这让 PG 在"没有天然主键、多列查询模式多样、需要灵活建各种索引(甚至 GiST、GIN 这些非 B-tree)"的场景下更自在。PG 还能轻松做"表的物理重排"(VACUUM FULL / CLUSTER),而 InnoDB 要重排一棵 B+树,代价大得多。

> **承接《PostgreSQL 数据库内核》那本**:这一节是本书和《PG》那本的**分水岭**。《PG》那本里讲的"表是无序堆、索引指向 TID、VACUUM 回收死元组"那一整套,是 heap 范式的世界;从本章开始,我们走进 IOT 范式——表就是树、二级索引存主键值、没有 VACUUM 而是 purge(P4-15)。两种范式都是成熟数据库的选择,理解它们的取舍,比记住"谁更好"重要得多。

> **钉死这件事**:IOT 不是"更先进",heap 也不是"过时"。它们是同一个问题——"一张关系表怎么存"——的两个解,各自在最适合自己的负载下最优。InnoDB 选 IOT,是因为它服务 OLTP(P0-01 讲的"数据多 + 并发高 + crash 不丢 + 事务 ACID");PG 选 heap,部分是因为 PG 的设计哲学更看重建索引的灵活性和 MVCC 的另一种实现(用 xmin/xmax 而非 undo 版本链——P4 篇会对照)。**理解一个数据库的存储决策,要回到它要服务的负载和它的整体设计哲学**。

## 八、技巧精解:两个最硬核的洞察

本章正文讲完了聚簇索引的本质。最后,有两个最硬核的洞察值得单独钉死,配真实源码和反面对比,彻底讲透。

### 洞察一:`first_index()` 链表约定——InnoDB 用一个链表首位置,锁死"表 = 聚簇索引"

第一个洞察,是 InnoDB 怎么在源码层面,用一个**数据结构约定**,把"表 = 聚簇索引"这件事钉得死死的。

我们已经知道:`dict_table_t::first_index()` 返回聚簇索引,因为聚簇索引被放在索引链表的第一个位置。但这个约定为什么 sound?为什么链表第一个**永远是**聚簇索引?来看建表时聚簇索引是怎么进链表的。

[`dict_index_add_to_cache`](../mysql-server/storage/innobase/dict/dict0dict.cc)(把一个 index 加入表的缓存)在处理聚簇索引时,有明确的"插到链表头"逻辑。更直接的证据是 `dd_fill_dict_index`(`dict0dd.cc:3189`)——建表时,**第一个**被构造并加入表缓存的索引,就是聚簇索引(无论是显式主键、升格的 UNIQUE,还是 `GEN_CLUST_INDEX`)。所有二级索引,都在聚簇索引之后才构造。这就从**构造顺序**上保证了:链表第一个,永远是聚簇索引。

这个约定的精妙在于:**它不是一个布尔标志(`bool is_clustered`),而是一个位置约定**。整个 InnoDB 源码里,要访问聚簇索引,从来不写"找 `is_clustered() == true` 的那个",而是直接 `table->first_index()`——O(1) 取链表头。这个约定省掉了每次查找的开销,也让"聚簇索引"在源码里成了一个**无处不在、零成本**的概念。

> **不这么设计会怎样**:如果 InnoDB 不用"链表首位置"约定,而是给每个 `dict_index_t` 一个 `bool is_clustered` 标志,然后每次要聚簇索引就遍历链表找标志为 true 的——① 遍历开销(虽然小,但热点路径上累积起来不可忽视);② 多了"两个聚簇索引"或"没有聚簇索引"的非法状态可能(标志能乱设,位置约定不能乱)。位置约定把"表有且仅有一个聚簇索引"这个不变量,**用数据结构本身**保证了——这是用类型系统/数据结构约束状态空间的设计思想(《Linux 内核》那本里讲"用数据结构编码不变量"时反复强调过)。

### 洞察二:全局共享 `DB_ROW_ID` 计数器——用 CAS + margin 刷盘,在并发和持久化之间走钢丝

第二个洞察,是隐式主键的 `DB_ROW_ID` 分配器。我们上面看过 `dict_sys->row_id` 这个全局原子计数器,现在拆透它为什么这么设计。

朴素的设计有两种:

**朴素设计 A:每次自增都加锁 + 立刻刷盘**。正确性最好(任何 crash 都不丢),但性能灾难——每次插入都要加锁、写磁盘,并发插入会被一个全局锁串行化。

**朴素设计 B:只在内存里自增,从不刷盘**。性能最好,但 crash 后 `row_id` 归零,重启后分配的 `DB_ROW_ID` 可能和 crash 前的重复——唯一性被破坏,聚簇索引排序会错乱。

InnoDB 的实际设计,是在这两极之间走钢丝:

```cpp
// include/dict0boot.ic,第 36-64 行(简化示意)
static inline row_id_t dict_sys_get_new_row_id() {
  row_id_t id = dict_sys->row_id.load();
  while (true) {
    if ((id % DICT_HDR_ROW_ID_WRITE_MARGIN) == 0) {
      // 到 margin 边界:加锁,刷盘,自增
      dict_sys_mutex_enter();
      ...dict_sys->dict_hdr_flush_row_id();      // 把当前值刷到数据字典头
      dict_sys->row_id.store(id + 1, ...);
      dict_sys_mutex_exit();
      return id;
    } else if (dict_sys->row_id.compare_exchange_weak(id, id + 1)) {
      return id;                                  // CAS 无锁自增
    }
  }
}
```

它的策略是:

1. **快路径(CAS 无锁)**:绝大多数情况下(`id` 不是 margin 边界),用 `compare_exchange_weak` 做无锁自增。多线程并发分配,各拿各的,不阻塞。
2. **慢路径(加锁刷盘)**:每 `DICT_HDR_ROW_ID_WRITE_MARGIN` 次分配,加一次锁,把当前 `row_id` 刷到数据字典头(`dict_hdr_flush_row_id`)。这一步保证 crash 后能从字典头恢复一个"下界"。
3. **crash 恢复(+margin)**:crash 后,字典头里的 `row_id` 可能比实际分配出去的小(没刷的那些丢了)。重启时,InnoDB 把字典头的值 **+ `DICT_HDR_ROW_ID_WRITE_MARGIN`**,作为新的起点——宁可跳过一段 ID,也不用可能重复的 ID。

这是一个经典的**并发数据结构设计模式:快路径无锁,慢路径加锁,用牺牲一点空间(跳号)换正确性**。同样的思想,你在《Linux 内核》那本的 seqlock、RCU 里见过("读路径无锁,写路径加锁,用版本号保证正确性");在《LevelDB》那本的 SequenceNumber 分配里也见过类似权衡。

> **不这么设计会怎样**:
>
> - 用朴素设计 A(每次加锁刷盘),OLTP 高并发插入会被 `dict_sys->row_id` 这个全局锁串行化——而它本该是个**冷路径**(只对没主键的表才用)。InnoDB 的 CAS 快路径,让这个锁几乎不竞争。
> - 用朴素设计 B(从不刷盘),crash 后 `DB_ROW_ID` 重复,聚簇索引的排序键失效——数据正确性崩。
> - InnoDB 的实际设计,把"并发性(CAS 快路径)"和"持久化(margin 刷盘 + 恢复时 +margin)"两个看似矛盾的需求,用一个 margin 机制调和了。这是数据库"在并发和正确性之间走钢丝"的又一个教科书级案例。

> **钉死这件事**:聚簇索引的两个硬核洞察——① 用链表首位置约定锁死"表 = 聚簇索引"(数据结构编码不变量);② 全局 `DB_ROW_ID` 计数器用 CAS + margin 刷盘调和并发与持久化(快路径无锁 + 慢路径加锁 + 跳号保正确)。这两个洞察,一个是**数据结构设计**的巧思,一个是**并发控制**的巧思——InnoDB 源码里这种级别的技巧,后面每一章都会遇到。

## 九、章末小结

### 回扣主线

本章是第 1 篇"地基"的第一块砖,服务的当然是主线里**"存储与索引"**这一面。

> **二分法提醒**:全书的主线是"**一条写,InnoDB 用 B+树聚簇索引找到位置、redo 保 crash 不丢、undo/MVCC 保并发读、锁保隔离**"。本章讲的是第一句——**"B+树聚簇索引找到位置"**——的存储根基:表怎么存、数据在哪。这一面(B+树聚簇索引 / 二级索引 / 页结构 / buffer pool)决定"数据怎么放、怎么找到";另一面(WAL/redo/undo/2PC + MVCC + 锁)决定"写怎么不丢、并发怎么不乱",是第 3-5 篇的事。

InnoDB 是**索引组织表(IOT)**:表 = 主键 B+树,整行数据在叶子页。这是它区别于 PG 堆表的**根本存储决策**,也是后续所有存储机制(二级索引回表、buffer pool 缓存页、页分裂合并)的起点。

### 五个为什么清单

1. **为什么 InnoDB 是 IOT(索引组织表),而 PG 是 heap(堆表)?** —— 两种"表怎么存"的根本范式。IOT 让主键点查一跳(OLTP 最高频操作),代价是非主键查询要回表;heap 让索引之间平等灵活,代价是主键查询两跳。InnoDB 押注 OLTP 主键点查,选 IOT。

2. **为什么 InnoDB 没主键也要造一个?** —— IOT 必须有聚簇索引(整行得按某个顺序排在 B+树叶子页)。没显式主键,先升格一个全列 NOT NULL 的 UNIQUE 索引;都没有,造 6 字节隐式 `DB_ROW_ID` 的 `GEN_CLUST_INDEX`。源码:`dict0dd.cc:3189` 造 `GEN_CLUST_INDEX`,`row0ins.cc:3510` 分配 `DB_ROW_ID`。

3. **为什么二级索引存主键值,而不是行指针?** —— 聚簇索引页分裂时行物理位置会变,但主键值不变。二级索引存主键值,页分裂时零维护;存物理位置则要连锁更新所有二级索引。代价是回表多一次 B+树查找(log 级,可接受)。这是下一章 P1-03 的主角。

4. **为什么主键强烈推荐自增整型?** —— ① 顺序追加 = 永远插最右页 = 少页分裂 + 顺序 IO;② 主键小 = 二级索引紧凑(二级索引每条记录存主键值)。两害(随机主键的页分裂 + 二级索引膨胀)反向证明:自增整型几乎是 IOT 下最优的主键选择。

5. **为什么 InnoDB 用"链表首位置"约定来标识聚簇索引?** —— 用数据结构(位置)而非标志位编码不变量,保证"表有且仅有一个聚簇索引"这个不变量在源码层面不会被破坏;同时 `table->first_index()` 取链表头是 O(1),聚簇索引访问零成本。源码:`dict0mem.h:2491` `first_index()` 返回聚簇索引。

### 引出下一章

我们搞清楚了"InnoDB 的表是主键 B+树、整行在叶子页"。但 OLTP 不可能只按主键查——你还要按手机号查用户、按邮箱查订单。这些**非主键查询**怎么走?答案是用**二级索引**,但二级索引查到主键值后,**还要回到聚簇索引里把整行捞出来——这叫回表**。二级索引为什么存主键值、回表的代价怎么避免(覆盖索引)、InnoDB 在源码里怎么实现回表,是下一章《P1-03 · 二级索引与回表》的主角。本章埋下的"二级索引存主键值"这个伏笔,下一章拆透。
