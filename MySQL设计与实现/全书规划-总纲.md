## 一、这本书的定位

### 目标读者

天天用 MySQL、建过表、建过索引、`explain` 看过执行计划、背过"RR 解决幻读""索引最左前缀"这些口诀,甚至翻过 `mysql/mysql-server` 的 `storage/innobase/` 源码,但**一知半解**:知道 InnoDB 有 B+树、redo、undo、MVCC、间隙锁、buffer pool,却讲不清一条 `UPDATE` 从执行到落盘,中间到底发生了什么、每一步为什么这么设计;更讲不清聚簇索引为什么"表就是索引"、redo log 8.0.30 重构后变成了什么、间隙锁到底锁的是什么、doublewrite 防的是什么。和"用过 MySQL 但不懂 InnoDB"是同一种病。

### 全书一句话主旨

> **InnoDB 用"B+树聚簇索引存数据 + redo(WAL)保 crash 不丢 + undo(MVCC)保高并发读 + 锁保隔离"这四个灵魂,把一次写做到"既不丢又高并发"——每一步的设计动机和实现技巧都讲透。**

每个组件,都是这条旅程上的一个**工位**。读完一本书,读者不是"多记了几个调优口诀",而是能**在脑子里放映出一条 `UPDATE` 在 InnoDB 里的全过程**:SQL 进 server 层解析优化 → 到 handler 接口进 InnoDB → 在 buffer pool 找到(或读入)所在 B+树页 → 加锁(行锁/间隙锁)→ 改数据 + 写 undo(版本链)→ 生成 redo(mini-transaction)→ 提交时两阶段提交(redo prepare → binlog → redo commit)→ 脏页异步刷盘(经 doublewrite)→ crash 了用 redo 重放 + undo 回滚。以及每一步底下用了什么巧妙的手段。

### 为什么用"一次写的旅程"当骨架(而非按模块平铺)

InnoDB 的模块(B+树/buffer pool/redo/undo/锁/MVCC)高度耦合:一条写要从 buffer pool 找页、加锁、改数据、写 undo、生成 redo、两阶段提交、刷盘。按模块百科式平铺,读者会觉得每块都是孤岛、不知道**为什么要学这个**。用"一条写怎么既不丢又高并发"这条旅程当骨架,每个模块都是"为了让写安全地完成而必须出现的角色"——读者永远知道**我在哪、为什么需要下一站**。

## 二、写作方法:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与数学系列、其他源码系列最关键的方法论,所有写作者必读、必守。**

InnoDB 是 C/C++ 写的成熟引擎(几十年的演进),源码里充斥高级技巧:**B+树页的分裂/合并、聚簇索引(索引组织表)、buffer pool 的改进 LRU(midpoint insertion 防全表扫描冲刷)、自适应哈希索引(AHI)、change buffer(二级索引写合并)、doublewrite buffer(防页撕裂)、WAL + redo(log0write 8.0.30+ 重构)、mini-transaction(mtr 原子写)、两阶段提交(redo/binlog 一致)、MVCC undo 版本链 + read view、间隙锁/临键锁(锁住"不存在的间隙")、crash recovery(log0recv 重放)**……用户自己用过 MySQL,"一知半解"的缺口往往**不在**"它要解决什么问题"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书的每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?(沿用系列一贯的"为什么优先")
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果不用这个技巧、朴素地写,会撞上什么墙",让妙处显形。

一句话:**讲不清聚簇索引为什么"表就是索引"、讲不清 redo 怎么保证 crash 不丢、讲不清间隙锁锁的到底是什么,等于没讲 InnoDB。**

> **⚠️ 深度标杆(用户强调,本书命脉)**:本书篇幅标杆拉到对标《LuaJIT》深入浅出第一性原理版——**每章约 12000~18000 字**。用户原话:"不要吝啬文字和篇幅,内容可以更丰富一点,要把问题讲解清楚,深入浅出,好理解又有深度。"(同 gRPC/TiKV/Envoy/Dubbo 四本)所以每个机制至少讲透三层(动机 → 反例 → 技巧 + 为什么 sound),宁可多铺反例、对比、图、选型历史(InnoDB vs MyISAM、为什么 B+树不是 LSM、8.0.30 redo 重构),把问题讲到读者"原来如此",也不要一笔带过。

## 三、全书主线、二分法、主比喻

### 主线

> **一条写,InnoDB 用四个保证让它"既不丢又高并发":B+树聚簇索引找到位置、redo(WAL)保 crash 不丢、undo(MVCC)保并发读、锁保隔离。**

任何一处看不懂某个机制,回到这句问:"这是在**把数据存好、找到(存储与索引)**,还是在**让写不丢、能并发(事务与并发)**?"

### 二分法

本书的二分法:

> **存储与索引(数据怎么放、怎么找) vs 事务与并发(怎么不丢、怎么并发)。**

- **存储与索引这一面**:B+树聚簇索引、二级索引/回表、页结构、buffer pool、改进 LRU、AHI、change buffer、doublewrite——这些决定"数据怎么放、怎么高效找到、怎么缓存"。
- **事务与并发这一面**:WAL/redo、undo、mtr、两阶段提交、crash recovery、MVCC/read view、锁(记录/间隙/临键)、隔离级别——这些决定"写怎么不丢、并发怎么不乱"。

InnoDB 每个模块都落在这两面之一(或两者衔接处:`buffer pool + mtr` 是存储层通往事务层的枢纽——mtr 生成 redo、刷脏页)。任何时候迷路,回到这个二分法。

### 主比喻决策:直球为主,比喻点睛(⚠️ 沿用系列定调)

沿用系列定调:**直球为主**——绝大多数地方第一性原理直给,不套比喻。**比喻点睛**——仅在开篇(P0-01)用"图书馆"做一次性点睛(聚簇索引 = 按学号排的书架,书本身就按主键排;二级索引 = 按姓名查到学号、再回书架找;WAL = 先在账本记一笔要搬哪本书、再真的去搬,账本比书搬得快且不怕中途断电),全书不建立贯穿的重比喻体系。这是本书与《Tokio》(餐厅服务员全程贯穿)最显著的风格差异。

## 四、全书结构(分篇 + 分章)

> 每章给**章名 + 核心问题 + 关键源码模块 + 本章典型技巧 + 二分法归属**。主线是一条旅程:**为什么需要 InnoDB → 表就是 B+树 → buffer pool 缓存 → WAL 崩溃恢复(redo/undo/2PC)→ MVCC 并发读 → 锁隔离 → 实践 → 收束**。
> 章号全局连续;命名 `P{篇}-{章}-标题.md`,开篇第 1 章 = `P0-01`,第 1 篇第 2 章 = `P1-02`,以此类推。

### 第 0 篇 · 开篇:为什么需要 InnoDB(定调,1 章)

- **第 1 章 · 第一性原理:为什么需要 InnoDB**(`P0-01`):MySQL 的存储引擎架构(server 层 + 可插拔引擎),为什么 InnoDB 成了默认/主流。OLTP 场景的核心矛盾:**既要数据多、并发高,又要 crash 不丢、事务 ACID**。InnoDB 用四个灵魂回答(B+树聚簇索引 + WAL + MVCC + 锁)。★四重承接(↔《PG》heap 表、↔《LevelDB》LSM、↔《Linux mm》page cache、↔《TiKV》分布式事务)。**【样本章节,全书风格标杆,见后】**

### 第 1 篇 · 地基:表就是 B+树(3 章)

> 先搞清楚"InnoDB 怎么存数据"——这是 InnoDB 区别于其他引擎的根本。

- **第 2 章 · 聚簇索引:表的数据就在主键 B+树里**(`P1-02`)——为什么 InnoDB 是"索引组织表(IOT)"而不是"堆表"?技巧:**聚簇索引**(叶子页直接存整行数据)、对照《PG》堆表(heap + 二级索引)、主键为什么这么重要(没有主键 InnoDB 选唯一索引或造隐式主键)。源码:`storage/innobase/btr/`(btr0btr)、`dict/`。归属:`存储与索引(招牌)`。
- **第 3 章 · 二级索引与回表**(`P1-03`)——二级索引为什么不存行数据、要存主键值再回表?技巧:**二级索引存主键值**(不是行指针,这样页分裂不用改二级索引)、**回表**(二级索引→聚簇索引)、**覆盖索引**(二级索引里就有要的字段,免回表)。源码:`btr/`、`row/row0sel.cc`。归属:`存储与索引`。
- **第 4 章 · B+树页与记录:16KB 的页怎么组织**(`P1-04`)——B+树的页(默认 16KB)里长什么样?技巧:**页结构**(File Header + Page Header + 记录 + Page Directory)、**记录格式**(Compact/Redundant/Dynamic)、**页分裂/合并**、**行溢出**(大字段 off-page 到 lob)。源码:`storage/innobase/page/`(page0page)、`rem/`(记录)、`lob/`。归属:`存储与索引`。

### 第 2 篇 · 内存:Buffer Pool(3 章)—— 承接《Linux 内存管理》

> B+树页是存在磁盘的,但每次读写都走磁盘太慢——buffer pool 把热页缓存到内存。

- **第 5 章 · Buffer Pool:页面缓存**(`P2-05`)——为什么 OLTP 离不开 buffer pool?技巧:**buffer pool**(页的缓存池,buf0buf)、命中/未命中、脏页、**free list / LRU list / flush list** 三大链表。源码:`storage/innobase/buf/buf0buf.cc`。归属:`存储与索引(承《Linux mm》)`。
- **第 6 章 · 改进的 LRU:midpoint insertion**(`P2-06`)——朴素 LRU 为什么会被一次全表扫描冲刷干净?技巧:**midpoint insertion**(新读入的页插到 LRU 中部,不是头部,全表扫描的页很快被淘汰,不冲刷热数据)、**young/old 区域**。源码:`buf/buf0lru.cc`。归属:`存储与索引`。
- **第 7 章 · 自适应哈希索引(AHI)与 change buffer**(`P2-07`)——怎么让热点更快、二级索引写更省?技巧:**AHI**(热查询路径自动建哈希索引,`ha/ha0ha`)、**change buffer**(对二级索引的修改先缓存、后台合并,承《LevelDB》写缓冲思想,`ibuf/ibuf0ibuf`)。源码:`ha/`、`ibuf/`。归属:`存储与索引(招牌)`。

### 第 3 篇 · 事务·崩溃恢复:WAL(redo/undo/2PC)(5 章)—— InnoDB 最核心

> 这是 InnoDB 的心脏:**怎么让一次写"crash 了也不丢、而且 redo 和 binlog 一致"**。

- **第 8 章 · WAL 与 redo log**(`P3-08`)——为什么写之前要先记日志(Write-Ahead Logging)?技巧:**WAL**(先写日志再改数据页,崩溃后靠日志重放)、**redo log 是物理日志**(记"哪页哪偏移改成什么",幂等可重放)、**8.0.30+ redo 重构**(log0write/log0buf/log0files_*,老资料讲 log0log.cc 单文件大片过时)。源码:`storage/innobase/log/`(log0log、log0write、log0buf、log0chkp)。归属:`事务与并发(招牌)`。
- **第 9 章 · mini-transaction(mtr):redo 的生成单位**(`P3-09`)——redo 怎么"原子地"生成?技巧:**mtr**(一组对页的修改打包成一个 mini-transaction,要么全写进 redo、要么都不写,保证崩溃一致性)、mtr 与 buffer pool dirty page 的关系。源码:`storage/innobase/mtr/`(mtr0mtr)。归属:`事务与并发`。
- **第 10 章 · undo log:回滚 + MVCC 版本链**(`P3-10`)——undo 不只是回滚?技巧:**undo 是逻辑日志**(记"怎么改回去")、两个用途(事务回滚 + MVCC 的旧版本)、**回滚段 / undo tablespace**。源码:`storage/innobase/trx/`(trx0undo、trx0rec)。归属:`事务与并发(招牌)`。
- **第 11 章 · 两阶段提交(2PC):redo 与 binlog 一致**(`P3-11`)——为什么 InnoDB 和 binlog 要两阶段提交?技巧:**2PC**(redo prepare → 写 binlog → redo commit)、**为什么必须**(否则 crash 后 redo 和 binlog 不一致,主从复制会错)、XA PREPARE。源码:`trx/trx0trx.cc`、`handler/ha_innodb.cc`。归属:`事务与并发(招牌)`。
- **第 12 章 · crash recovery 与 doublewrite**(`P3-12`)——崩溃重启后怎么恢复?技巧:**crash recovery**(log0recv 重放 redo、用 undo 回滚未提交事务)、**doublewrite buffer**(先顺序写一份页到双写区、再随机写到数据文件,防 partial page write 导致页撕裂,buf0dblwr)。源码:`log/log0recv.cc`、`buf/buf0dblwr.cc`。归属:`事务与并发(招牌)`。

### 第 4 篇 · 并发控制:MVCC(3 章)—— 承接《TiKV》

> 写的时候怎么让读不阻塞、且读到一致的快照?

- **第 13 章 · MVCC 全貌:多版本并发控制**(`P4-13`)——为什么"读不加锁、读不阻塞写"?技巧:**MVCC**(一行有多个版本)、**RR/RC 两个隔离级别**(RR 事务开始时建 read view、RC 每条语句建)、对照《TiKV》MVCC(单机 undo 版本链 vs 分布式 MVCC)。源码:`storage/innobase/read/`(read0read)。归属:`事务与并发(承《TiKV》)`。
- **第 14 章 · Read View 与可见性判断**(`P4-14`)——一个事务怎么判断某个版本"看得见不"?技巧:**read view**(活跃事务列表)、**可见性判断算法**(trx_id < min_trx → 可见;trx_id > max_trx → 不可见;在活跃列表 → 不可见)、**顺着 undo 版本链找**。源码:`read/read0read.cc`。归属:`事务与并发`。
- **第 15 章 · undo 版本链与 purge**(`P4-15`)——旧版本堆多了怎么办?技巧:**版本链**(undo log 把旧版本串成链)、**purge**(后台清理"没人再需要"的旧版本,承《TiKV》GC)、purge 与 read view 的关系。源码:`trx/trx0purge.cc`。归属:`事务与并发`。

### 第 5 篇 · 并发控制:锁(4 章)—— InnoDB 锁招牌

> MVCC 解决了"读-写并发",但"写-写并发"还得靠锁。InnoDB 的锁精妙且复杂。

- **第 16 章 · 锁全貌:行锁 + 表锁,两阶段锁协议**(`P5-16`)——InnoDB 有哪些锁?技巧:**行锁**(记录锁 S/X)、**两阶段锁协议**(事务中随时加锁、commit 才统一释放)、锁队列、对照 MyISAM 表锁。源码:`storage/innobase/lock/`(lock0lock)。归属:`事务与并发(招牌)`。
- **第 17 章 · 间隙锁与临键锁:解决幻读**(`P5-17`)——RR 怎么解决幻读?技巧:**间隙锁(gap lock)**(锁住"记录之间不存在的间隙",别的事务插不进来)、**临键锁(next-key lock)**(记录锁 + 前面的间隙)、为什么 RR 用它、RC 为什么不用。源码:`lock/lock0lock.cc`。归属:`事务与并发(招牌)`。
- **第 18 章 · 死锁检测与锁等待**(`P5-18`)——多个事务互相等锁怎么办?技巧:**wait-for graph**(等锁图,检测环=死锁)、**死锁回滚**(挑 undo 量小的事务回滚)、**innodb_lock_wait_timeout**。源码:`lock/lock0lock.cc`(死锁检测)。归属:`事务与并发`。
- **第 19 章 · 隔离级别:RR/RC/Read Uncommitted/Serializable**(`P5-19`)——四个隔离级别怎么用锁 + MVCC 实现?技巧:**RU**(读未提交)、**RC**(读已提交,每条语句新 read view,无间隙锁)、**RR**(可重复读,事务级 read view + 间隙锁)、**Serializable**(串行化)。归属:`事务与并发`。

### 第 6 篇 · 实践与进阶(3 章)

- **第 20 章 · 一条 SQL 的完整旅程**(`P6-20`)——一条 `UPDATE` 怎么从 server 层走到 InnoDB?技巧:**server 层**(连接→解析→预处理→优化→执行,承《PG 数据库内核》讲过的 SQL 旅程)、**handler 接口**(`handler/ha_innodb.cc`)进 InnoDB。源码:`sql/`(server 层)、`storage/innobase/handler/`。归属:`衔接`。
- **第 21 章 · 索引调优与 explain**(`P6-21`)——B+树怎么走、索引怎么选?技巧:**explain**(type/key/rows/Extra)、**索引选择**(选哪个索引、为什么有时不选)、**回表/覆盖/最左前缀**、慢查询排查。归属:`存储与索引/实践`。
- **第 22 章 · 在线 DDL:加字段/索引怎么不锁表**(`P6-22`)——DDL 怎么不阻塞业务?技巧:**in-place alter**(就地改,`handler/handler0alter.cc`)、**online rebuild**(重建表,抄到新表)、**instant DDL**(8.0+ 秒级,只改数据字典)。源码:`storage/innobase/handler/handler0alter.cc`、`dict/`、`ddl/`。归属:`存储与索引`。

### 第 7 篇 · 收束(1 章)

## 五、源码策略

- **源码(已 clone 到本地)**:
  - `../mysql-server/` —— MySQL 主仓(`mysql/mysql-server`),C/C++ 实现,**master @ commit `845d525d49c8027a4d0cdcc43372c96ba295c857`,版本 9.7.0 LTS**(`MYSQL_VERSION` 文件确认;8.4.10 LTS 是生产另一主流)。InnoDB 源码在 `storage/innobase/`。
- **四重承接(本书最大特色)**:
  - **聚簇索引 vs 堆表 → 承接《PostgreSQL 数据库内核》那本**。InnoDB 是索引组织表(表就是 B+树),PG 是堆表(表和索引分离),本书对照讲、不重复 PG 的 SQL 旅程。
  - **B+树 vs LSM → 承接《LevelDB》那本**。两种存储引擎范式(InnoDB 的 B+树更新就地写 vs LevelDB 的 LSM 追加+compaction),对照"写放大/读放大/空间放大"三角。
  - **buffer pool vs page cache → 承接《Linux 内存管理》那本**。InnoDB 自己管 buffer pool(不依赖 OS page cache),LRU 算法对照 kernel 的 page cache LRU。
  - **单机 MVCC/事务 vs 分布式 → 承接《TiKV》那本**。InnoDB 的 undo 版本链 MVCC + 单机锁,对照 TiKV 的分布式 MVCC + Percolator。
- **引用规范**:本地相对路径 `[描述](../mysql-server/路径#L起-L止)`;区间 `#La-Lb`、单行 `#L`;VSCode 预览可点击跳转。
- **版本钉死**:全书以 `mysql-server @ 845d525 (9.7.0 LTS)` 为准,行号统一;若引用处 master 已变动,标注 commit。**注意:生产主流是 8.4 LTS,涉及版本差异处(如 redo log 8.0.30 重构)讲清**。
- **⚠️ 架构演进交代(本书独有,务必处理)**:InnoDB 处于持续演进,几处老资料大片过时:
  1. **redo log 8.0.30 大重构**:老资料讲的 `storage/innobase/log/log0log.cc` 单文件模型**大片过时**,新版拆成 `log0write.cc`/`log0buf.cc`/`log0files_*` 等十几个文件(并发写、无锁优化);`log0pre_8_0_30.cc` 是兼容 8.0.30 之前格式的代码。
  2. **数据字典 8.0 重构**:老资料讲的 `.frm` 文件已废弃,8.0 起用新的 transactional data dictionary(`dict/`,存 InnoDB 表里)。
  3. **instant DDL(8.0+)**:加列等只改数据字典、秒级完成,老资料讲的"DDL 必重建表"过时。

6. **回扣全局**:每章结尾回到二分法,说明本章服务"存储与索引"还是"事务与并发"那一面,衔接前后章。
