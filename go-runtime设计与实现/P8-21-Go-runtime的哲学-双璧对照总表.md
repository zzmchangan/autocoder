# 第二十一章 · Go runtime 的哲学 + 双璧对照总表

> 篇:第 8 篇 · 收尾
> 主线呼应:前面 20 章,我们用一个 goroutine 的一生和一次 GC 的全程,把 `src/runtime` 的主干拆了个遍。GMP 怎么调度、channel 怎么通信、内存怎么分配、GC 怎么并发回收、栈怎么增长、网络怎么不阻塞线程、Mutex/timer 怎么往调度器上接——每一块都讲了"它解决什么本质问题""用了什么技巧""为什么 sound"。这一章不引入新源码,而是退后一步,把这 20 章里反复出现的几条**贯穿哲学**提炼出来,并给一张**双璧对照总表**,把这本书和《Tokio》钉在一起。读完这一章,你手里就有了 Go runtime 的"一张图全貌"——以及,为什么是这张图、而不是别的图。

## 核心问题

**全书 20 章拆了几十个机制,它们背后有没有统一的设计取向?Go runtime 和 Tokio 这两套顶级调度器,到底在哪里同、在哪里不同、为什么不同?**

读完本章你会明白:

1. 贯穿全书 20 章的**五条设计哲学**:语言内置、协作+抢占混合、有 GC 的取舍、分层缓存换无锁快路径、STW 最小化的执念——每条都能回扣到具体章节。
2. 每条哲学"不这样会怎样""所以这样设计""钉死了什么"。
3. Go runtime vs Tokio 在并发单元、调度、work-stealing、channel、GC、网络、timer、抢占八个维度的**同与不同**,以及两种设计取舍的根因(语言内置 vs 库级)。
4. 为什么"没有银弹":两种哲学各有代价,代价落在不同的地方。

> 全书定位:第 21 章 / 第 8 篇 收尾。源码版本 Go 1.27(本地 master @ `6d1bcd10`,Version 常量见 [`src/internal/govversion/govversion.go`](../go/src/internal/govversion/govversion.go#L12) 的 `const Version = 27`)。本章是对前 20 章的收束,所有源码事实均已在前 20 章建立并逐字核实,本章只引用已建立的锚点。

## 21.1 一句话点破

> **Go runtime 的全部技巧,都在为一件事服务——"让 goroutine 像函数一样便宜,让阻塞不阻塞线程,让 GC 不打断业务"。它的设计取向可以收成五句话:把并发做进语言(而不是库)、用协作+抢占混合调度(而不是纯协作)、用并发 GC 换零心智负担(而不是手动管理)、用分层缓存换无锁快路径(而不是一把大锁)、把 STW 压到亚毫秒(而不是周期性停顿)。这五条每一条都和 Tokio 的选择正面对照,合起来就是"语言内置协程"与"库级异步"这两种顶级调度器的全貌。**

这是结论,不是理由。本章倒过来拆:先逐条把这五条哲学立起来(每条回扣具体章节的"不这样会怎样 / 所以这样设计 / 钉死什么"),再用一张总表把 Go runtime 和 Tokio 在八个维度上对齐,最后收束。

## 21.2 五条贯穿哲学

全书 20 章,机制几十个,但反复出现的取向就这五条。每一条都不是孤立的某章,而是横跨多章的设计 DNA。

### 21.2.1 哲学一:把并发做进语言,而不是做成库

> **goroutine 不是库,是语言关键字;channel 不是库,是语言原语。runtime 不是用户可选的依赖,而是和编译器一起 baked into 每一个 Go 程序。**

这条哲学最显眼,也最根本。它决定了后面所有哲学的形态。

**回扣章节**:

- 第 1 章(P0-01):`go func` 是**语言关键字**,编译器把它翻成对 [`newproc`](../go/src/runtime/proc.go#L5334) 的调用。你不用 `import` 任何调度器库,runtime 默认就在。
- 第 8 章(P2-08):`chan` 是**语言内置类型**,`make(chan int)` 直接由编译器翻成 [`makechan`](../go/src/runtime/chan.go#L75),`ch <- x` 翻成 [`chansend`](../go/src/runtime/chan.go#L176)。`select` 语句(P2-09)同样由编译器翻成 `selectgo`。
- 第 13~16 章(P4):写屏障、并发 GC 是 runtime 的一部分,**编译器在每个写指针处插屏障调用**——你写的每行 `p.next = q` 都被编译器改写过,这是语言/编译器/runtime 三者协作。

> **不这样会怎样**:如果 Go 像 Rust 一样,把并发做成库(`tokio::spawn`、`mpsc::channel`),那么:(a) 用户可以选择用不用异步运行时、用哪个,但代价是**所有阻塞式调用(标准库的 `std::net`、阻塞 `read`)都不会自动让出**——你必须显式换成 `tokio::net`、显式写 `async/await`,否则一个阻塞 `read` 就把整条线程钉死;(b) runtime 无法在编译期为 GC 插屏障,要么放弃 GC(走 ownership),要么用保守的运行时扫描。把并发做进语言,换来了"阻塞式 API + 自动让出"的零成本红利。

> **所以这样设计**:编译器 + runtime 协作,让 `go`/`chan`/`select` 成为语法,让 `conn.Read`、`time.Sleep`、`ch <- x` 这些看起来会阻塞的 API **底层都由 runtime 拦截**(park G 让出 M)。用户写的是顺序的、阻塞式的代码,runtime 在背后偷偷异步。这正是 Go 相对 Rust/Tokio 最核心的差异化价值。

> **钉死这件事**:`internal/poll` 那层(P6-19)是这条哲学在 IO 层的落地——`FD.Read` 看起来是阻塞系统调用,实际经 `runtime_pollWait` → [`netpollblock`](../go/src/runtime/netpoll.go) park 进 epoll,数据就绪再唤醒。一个 `conn.Read` 跨越了 `internal/poll` 和 `runtime` 两个世界,用户对此毫无感知。这种"用户写阻塞、runtime 替你让出"的无缝,只有把并发做进语言才做得到。

### 21.2.2 哲学二:协作+抢占混合,而不是纯协作

> **Go 调度器既不是纯协作式(必须 G 主动 yield),也不是纯抢占式(任意时刻打断),而是"默认协作(函数前奏/阻塞点让出)+ 兜底异步抢占(SIGURG 信号)"的混合体。**

这条哲学是理解第 5 章(P1-05)异步抢占和第 7 章(P1-07)sysmon 的钥匙。

**回扣章节**:

- 第 3 章(P1-03):Go 1.14 之前是纯协作式——G 在函数前奏里检查抢占标志,长时间不让出的 G(死循环、紧 CPU 循环)会饿死别人。
- 第 5 章(P1-05):Go 1.14 引入**异步抢占**,sysmon 检测到 G 跑超过 10ms,就给它发 `SIGURG` 信号,在任意安全点强行打断,切入 `mcall` 让出。
- 第 7 章(P1-07):sysmon 是"独立观察者"——它不受 GC/调度约束(不被抢占、不被 STW 阻塞),专门兜底那些调度器自己看不见的问题(长跑 G、长阻塞系统调用、该触发没触发的 GC/网络轮询)。

> **不这样会怎样**:纯协作式的代价是著名的"紧密循环饿死调度器"——一个 `for {}` 不调用任何函数,就不会触发前奏检查,整条线程被它独占,其他 G 全饿死。Go 1.14 之前的真实 bug:一个 goroutine 里写 `for i := 0; i < 1e10; i++ {}`,整个程序在 GC 时卡死(STW 等不到这个 G 到安全点)。纯抢占式的代价是任意点打断要保存完整上下文(包括撕裂的寄存器/栈帧)、要重新恢复,实现极复杂,且 Go 的栈拷贝(P5-17)要求在安全点进行。

> **所以这样设计**:**默认走协作(便宜,99% 的 G 在阻塞点/函数前奏自然让出),只在协作失灵时(死循环、紧 CPU 循环)才动用抢占(贵,发信号、走 `sigtrampgo`、在安全点切栈)**。混合调度 = 协作的便宜 + 抢占的兜底,两者叠加,既不饿死又不过度开销。这是一个典型的"为常见情形留快路径,罕见情形才走慢路径"的设计(这条总原则在第 3、8、11、20 章反复出现)。

> **钉死这件事**:异步抢占靠的是 `SIGURG` 信号——选这个信号是因为它"语义无害"(Go 不用它干别的,POSIX 也无强语义),且能被 `sigtrampgo` 安全处理。抢占要在**安全点**(`preemptM` 后 G 走到下一个函数边界)生效,不能在指令中间打断,否则会和 GC(扫栈)、栈拷贝冲突。这套"信号 + 安全点"机制,是 Go 1.14 最大的调度器进化(第 5 章详拆)。

### 21.2.3 哲学三:有 GC 的取舍——用并发 GC 换零心智,代价是 runtime 复杂

> **Go 选了并发 GC,而不是手动管理(ownership/Raft 式引用计数)。代价是 runtime 必须维护三色不变式、写屏障、mark assist,而且 STW 虽小但非零。**

这条哲学贯穿第 4 篇(P4-13~16)和第 17 章(栈拷贝,因为 GC 要扫栈)。

**回扣章节**:

- 第 13 章(P4-13):三色标记 + 写屏障,堵住"并发标记漏标"——业务改指针、GC 同时扫,凭什么不丢活对象。
- 第 14 章(P4-14):并发 GC 四阶段(标记准备 STW → 并发标记 → 标记终止 STW → 并发清扫),两次 STW 压到亚毫秒。
- 第 15 章(P4-15):mark assist——分配太快的 G 被强制协助标记,把 GC 压力分摊给分配者,防止单靠后台标记跟不上。
- 第 16 章(P4-16):lazy sweep——分配时顺带清扫,而非单独清扫阶段。
- 第 17 章(P5-17):栈拷贝——GC 扫栈时栈不能动,所以增长栈要拷贝整个栈并调整所有指针。

> **不这样会怎样**:没有 GC,要么手动 `free`(C 的方式,内存泄漏/use-after-free 满天飞),要么 ownership + 借用检查(Rust 的方式,编译期保证但心智陡峭,且自引用要 `Pin`)。Rust/Tokio 选了后者——零运行时 GC 开销,但代价是 `async/await` 编译成状态机后,Future 不能移动(否则自引用失效),必须用 `Pin` 类型系统禁止移动(见 17.9 对照栏)。Go 选 GC,换来了用户不用管生命周期、写代码像写顺序程序,但代价是 runtime 要扛三色 + 写屏障 + mark assist + STW 这一大套复杂机器。

> **所以这样设计**:Go 的赌注是——**绝大多数后端程序的 GC 开销,小于手动管理的心智成本 + bug 成本**。所以 Go 把并发 GC 做到"几乎不暂停业务"(亚毫秒 STW),让用户既能享受 GC 的省心,又不用付"周期性秒级停顿"的代价(Java 老时代的痛)。这套"并发 + 写屏障 + mark assist + STW 最小化"的工程,本质是在把 GC 的代价摊薄到"几乎不可见"。

> **钉死这件事**:写屏障(第 13 章)为什么 sound?因为它在每次写堆指针时,顺手把"可能让 GC 看漏"的对象染灰(`shade`),从而把"业务改指针"这件危险事变成 GC 能看见的事——业务无论怎么改指针,GC 都能顺着屏障记录的灰链重新扫到。这是"有 GC"这件事能成立的物理基础,也是 Go runtime 最硬核的正确性论证之一。

### 21.2.4 哲学四:分层缓存换无锁快路径——TCMalloc 思想,而不是一把大锁

> **凡是热路径,都先给一个本地无锁缓存(P 级),缓存不命中才回退到加锁的中心结构(mcentral),再不命中才到全局锁(mheap)。快路径几乎无锁,慢路径才碰锁。**

这条哲学贯穿第 3 篇(P3-10~12)、第 2 章(P1-02 的 P 结构)、第 4 章(P1-04 的 runq),是 Go runtime 高性能的物理根基。

**回扣章节**:

- 第 10~11 章(P3):内存分配三级缓存——mcache(P 本地,无锁)→ mcentral(按 size class 加锁)→ mheap(全局锁)。微小对象还有 tiny allocator 合并。
- 第 2 章(P1-02):P 结构里同时挂着 `runq`(本地无锁 runq)+ `mcache`(本地无锁内存缓存)+ 一堆 GC 缓存(`wbBuf` 写屏障缓冲、`timers` 本地 timer 堆)。
- 第 4 章(P1-04):work-stealing 的 runq 是 256 槽固定环形数组,owner 端 push/pop 无锁,只有 thief 偷的时候才走 CAS。
- 第 20 章(P7-20):Mutex 先自旋(无锁快速尝试)再加锁;timer 寄生在每个 P 的 `timers` 四叉堆(本地),减少全局锁竞争。

> **不这样会怎样**:如果内存分配走一把全局锁(`malloc` 在多线程下排队),那一个高并发 Web 服务器每秒几百万次分配,全卡在这一把锁上——典型的 ptmalloc 多线程扩展性灾难。如果 runq 是全局一个队列,那所有 M 取 G 都要抢同一把锁,work-stealing 就无从谈起。

> **所以这样设计**:**为最常见的情形(本地 P 有活/本地 mcache 有槽)留无锁快路径,罕见情形(本地空了/缓存不命中)才走加锁慢路径**。这条原则在第 11 章(P3-11)被点名是"分配器设计的最纯粹范例",在第 8 章(P2-08)的 channel 快路径、第 4 章(P1-04)的 runq、第 20 章(P7-20)的 Mutex 自旋里反复出现。它是 Go runtime 性能的总开关——**性能不是靠某一处神优化,而是靠"绝大多数热路径不碰锁"这一条贯穿性纪律**。

> **钉死这件事**:这条哲学的代价是**结构复杂**(P 要持有一堆缓存,缓存满了要回退,回退要协调一致性)。但这是值得的——分层缓存换来的"快路径无锁",让 Go 在多核扩展性上几乎线性。读 runtime 源码,凡是看到一个"本地缓存 + 中心结构"的拆分(runq/mcache/wbBuf/timers),都是这条哲学的同一个投影。

### 21.2.5 哲学五:STW 最小化的执念——亚毫秒停顿是设计目标,不是优化结果

> **Go 团队对 STW(stop-the-world)有一种近乎偏执的追求:把两次 STW 压到亚毫秒级。这不是某一次优化的结果,而是从 GC 架构到每一个细节贯穿的执念。**

这条哲学贯穿第 4 篇(P4),尤其第 14 章。

**回扣章节**:

- 第 14 章(P4-14):GC 四阶段,只有"标记准备"(开写屏障、扫栈、开启 GC worker)和"标记终止"(关写屏障、统计)两次 STW,且都压到亚毫秒。标记和清扫都和业务并发。
- 第 13 章(P4-13):写屏障的作用之一就是让"标记能和业务并发"——边标记边跑业务,屏障堵住漏标。
- 第 15 章(P4-15):mark assist 把"标记速率"和"分配速率"耦合,确保标记跟得上,不需要为了追上分配而 STW。
- 第 16 章(P4-16):清扫并发 + lazy,不在单独阶段停顿业务。
- 第 17 章(P5-17):栈拷贝要在安全点做,栈拷贝本身不是 STW 的来源(GC 扫栈才是),但栈拷贝要求"扫描时栈不动",和 GC 协同。

> **不这样会怎样**:老 Java 的 Full GC 是秒级 STW,业务请求全 timeout,这是"GC 打断业务"最痛的形态。如果 Go GC 也这样,那它"高并发后端首选语言"的地位就不成立——一个 SLA 要求 P99 < 10ms 的服务,扛不住一次 50ms 的 STW。

> **所以这样设计**:Go 把"STW 最小化"当成 GC 架构的**第一性约束**,而不是事后优化。三色 + 写屏障是为了"标记能并发";mark assist 是为了"标记速率不依赖业务让步";并发 sweep 是为了"清扫不停业务";栈扫放并发(第 14 章)是为了"扫栈不阻塞所有 G"。每一步都在为"少停一毫秒"服务。

> **钉死这件事**:Go 1.5 把 GC 从 STW 改成并发,是 runtime 历史上最大的一次跃迁;之后的每个版本(1.8 混合屏障、1.14 异步抢占让 GC 能在安全点 STW、1.19 SOFT 内存限制)都在继续压 STW。`GODEBUG=gctrace=1` 打印的每行,你能看到两次 STW 的微秒数(`"pause"` 字段),通常在 100~500 微秒。这不是巧合,是贯穿 10 年的执念。

## 21.3 五条哲学的内在关系

这五条不是孤立的清单,它们互相咬合,构成 Go runtime 的整体:

```
                    哲学一(语言内置)
                    是其他四条的前提
                          │
            ┌─────────────┼─────────────┐
            ▼             ▼             ▼
      哲学二          哲学三          哲学四
   (协作+抢占)      (有 GC)       (分层缓存)
   让 G 必让出     让回收自动      让热路径无锁
            │             │             │
            └─────────────┼─────────────┘
                          ▼
                    哲学五(STW 最小化)
                    是前三条的合力目标
```

- **哲学一(语言内置)是地基**:没有它,`conn.Read` 不会自动 park,channel 不是语法,写屏障不会被编译器插,后面四条都无从谈起。
- **哲学二(协作+抢占)服务调度执行**:确保海量 G 都能被驱动、不让一个死循环卡死。
- **哲学三(有 GC)服务支撑地基**:让内存能被自动回收,用户不用手动管理。
- **哲学四(分层缓存)服务性能**:让调度/分配/写屏障的热路径不碰锁。
- **哲学五(STW 最小化)是合力目标**:哲学二的抢占让 GC 能在安全点 STW,哲学三的并发 GC 让标记不停业务,哲学四的 mcache/wbBuf 让分配和写屏障不和 GC 抢锁。三者合一,STW 才能压到亚毫秒。

一句话:**哲学一是前提,哲学二三四是支柱,哲学五是它们共同支撑起的屋顶。** 这就是 Go runtime 的整张架构图。

## 21.4 一张图:Go runtime 的三层全景

把全书 20 章的子系统摆到"调度执行 / 阻塞唤醒 / 支撑地基"三层上(这是全书的二分法):

```
┌─────────────────────────────────────────────────────────────────┐
│  调度执行层(让就绪的 G 跑起来)                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐         │
│  │ GMP 结构 │→ │ schedule │→ │ work-    │→ │ 异步抢占 │  第1篇  │
│  │ (P1-02)  │  │ findRun  │  │ stealing │  │ (P1-05)  │  P1-02~07│
│  │ runq     │  │ (P1-03)  │  │ (P1-04)  │  │          │         │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘         │
│        ↓ 释放阻塞的 M              ↑ 把就绪的 G 重新入队        │
├─────────────────────────────────────────────────────────────────┤
│  阻塞唤醒层(让阻塞的 G 就绪后被重新调度)                    │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐         │
│  │ netpoll  │  │ sysmon   │  │ timer    │  │ channel  │  唤醒源 │
│  │ (P6-18/19)│  │ (P1-07)  │  │ (P7-20)  │  │ (P2-08/09)│        │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘         │
│        ↑ 都通过 gopark/goready 这对调度器接口接入 ↓             │
├─────────────────────────────────────────────────────────────────┤
│  支撑地基层(让上面两层能运转)                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                      │
│  │ 内存分配 │  │ 并发 GC  │  │ 栈管理   │  第3/4/5篇            │
│  │ mcache/  │  │ 三色+屏障│  │ 连续栈   │  P3-10~12 /           │

│  └──────────┘  └──────────┘  └──────────┘                      │

```

读这张图的方法:任何一个 Go runtime 机制,先问"它在哪一层、服务哪一面"。`schedule` 在调度执行层顶层;`netpoll` 是阻塞唤醒层的一个唤醒源;`mallocgc` 是支撑地基层。三层之间通过两对接口咬合:

- **`gopark`/`goready`**:阻塞唤醒层所有唤醒源(channel/timer/netpoll/sysmon)通往调度执行层的统一接口。一个 G 要阻塞就 `gopark`(交出工作台),就绪了就 `goready`(重新入 runq)。
- **GC ↔ 调度器**:GC 要 STW(暂停所有 G)、要 mark assist(让 G 协助)、要写屏障(编译器在每个写指针处插),这三处都是 GC 和调度执行层/支撑地基层的接缝。

第 20 章(P7-20)收尾已经点出:`Mutex` 和 `timer` 本质都是 `gopark`/`goready` 的使用者。这一章把这张全景图补全——**全书 20 章,就是在填这张图里的每一格**。

## 21.5 双璧对照总表:Go runtime vs Tokio

> 这是全书和《Tokio》的合订索引。Go runtime(语言内置协程)和 Tokio(库级异步)是"调度问题"在两种语言里的顶级解。把两者在八个维度上对齐,能看清"调度"问题的本质,以及两种设计取舍的根因。

下表的每个对照点,都能在前面对应的 ★ 章找到展开。表后讲根因。

| 维度 | Go runtime | Tokio | 对照点(同 / 异) | 根因 |
|---|---|---|---|---|
| **并发单元怎么造** | `go func`(语言关键字,创建 G,2KB 栈)[P0-01] | `tokio::spawn`(库函数,创建 Task,无栈 Future)[P0-01] | 同:都是把一段代码变成可调度单元。异:G 有独立栈,Future 是状态机 | Go 把并发做进语言;Tokio 是库,Future 是普通类型 |
| **并发单元有无栈** | G 有独立可增长栈(初始 2KB,可拷贝增长)[P5-17] | Task 是无栈 Future(编译成状态机) | 异:有栈 vs 无栈。Go 靠栈拷贝处理自引用,Tokio 靠 `Pin` 禁止移动 | 有栈:任意函数调用天然支持协程;无栈:零栈开销但 `async fn` 递归受限 |
| **调度器** | GMP + work-stealing(runtime 内置,P 持本地 runq)[P1-02/03] | multi-thread scheduler + work-stealing(库,worker 持本地队列)[P1-04 对照] | 同:都是 work-stealing 多线程调度器。异:Go 的 P 同时承载 runq+mcache+GC 缓存,Tokio 的 worker 只承载任务队列 | Go runtime 是"调度+内存+GC"合一;Tokio 只管调度 |
| **work-stealing 数据结构** | 256 槽固定环形数组,owner 单写一端、thief 从另一端 CAS 偷一半[P1-04] | Chase-Lev 双端队列(动态容量,经典无锁 work-stealing)[P1-04] | 同:都偷一半、都 owner-thief 协议。异:Go 固定数组(简单缓存友好,满则溢出全局),Tokio 动态 DEQueue(无 ABA 问题因索引单调,但要处理容量) | Go 偏简单+缓存友好;Tokio 偏经典无锁结构 |
| **阻塞怎么不占线程** | runtime 自动 park G(`gopark`)+ netpoll/sysmon 唤醒[P6-18/19] | `Future::poll` 返回 `Pending` + reactor(mio)唤醒[P6-18 对照] | 同:都是"阻塞时不占 worker,就绪后重新入队"。异:Go 是**隐式**(阻塞式 API,runtime 偷偷 park),Tokio 是**显式**(必须 `.await`,编译器编成 poll 状态机) | Go 把并发做进语言,能拦截阻塞 API;Tokio 是库,只能靠 Future poll |
| **channel** | 同步阻塞 API(`ch <- x`、`<-ch`),环形 buf + 等待队列(sudog)[P2-08] | 异步 channel(mpsc/oneshot/broadcast),`.await` 收发[P2-08 对照] | 同:都跨并发单元传值。异:Go 同步阻塞(runtime park),Tokio 显式 async(返回 Future) | 同上:Go 隐式让出,Tokio 显式 await |
| **GC / 内存安全** | 并发三色 GC + 写屏障(运行时回收)[P4-13~16] | 无 GC(ownership + `Pin`,编译期保证)[P5-17 对照] | 异:有 GC vs 无 GC。Go 省心智(有 STW/屏障开销),Tokio 零运行时开销(要手动管理生命周期 + `Pin`) | Go 选零心智;Rust 选零开销。两种语言的根本分歧 |
| **网络 IO** | netpoll 把 epoll 藏进 runtime,用户写阻塞 `conn.Read`[P6-18/19] | mio + AsyncFd 把 epoll 暴露成 Future,用户写 `async read`[P6-18 对照] | 同:都靠 epoll/kqueue 驱动。异:Go 藏进 runtime(用户无感),Tokio 暴露成类型系统(用户显式 async) | 同上:语言内置 vs 库封装 |
| **timer** | 每个 P 一个四叉堆(`timers`),寄生在 `findRunnable`/`checkTimers`[P7-20] | hashed timing wheel(层级时间轮,O(1) 插入/到期)[P7-20 对照] | 同:都把超时管理做进调度器。异:Go 四叉堆(几千~几十万 timer 够用,缓存友好),Tokio 时间轮(百万级 timer 占优,O(1)) | Go timer 数量通常不大,堆够简单;Tokio 面向海量并发任务超时,时间轮摊薄 |
| **抢占** | 协作(函数前奏)+ 异步抢占(SIGURG 信号)混合[P1-05] | 纯协作(`.await` 点即让出点,无信号抢占) | 异:Go 有信号抢占(死循环也能打断),Tokio 纯协作(无 `.await` 的循环无法被打断,要靠 `tokio::task::yield_now`) | Go 的 G 有独立栈,信号抢占可恢复完整上下文;Tokio 的 Future 无栈,只能在 poll 边界让出 |

### 21.5.1 根因:语言内置 vs 库级

这张表的所有"异",最终都能收回到一个根因——**Go 把并发做进语言(编译器 + runtime 协作),Tokio 把并发做成库(纯类型系统 + Future)**。

这个根因分化出三条主线:

1. **能不能拦截阻塞 API**:Go 能(runtime 是 baked in,`conn.Read`/`time.Sleep`/`ch <- x` 都被 runtime 拦截 park);Tokio 不能(库只能管 `tokio::net`/`tokio::time` 这类它提供的 API,标准库的阻塞 `read` 不让出)。这决定了 Go 写**阻塞式**代码、Tokio 写**显式 async**代码。
2. **能不能用 GC**:Go 能(编译器在每个写指针处插屏障);Rust 不行(没有 GC,靠 ownership)。这决定了 Go 省心智但有 STW/屏障开销,Rust 零开销但心智陡峭。
3. **能不能信号抢占**:Go 的 G 有独立栈,信号抢占能保存/恢复完整上下文;Tokio 的 Future 无栈,只能在 `.await` 边界让出。这决定了 Go 能兜底死循环,Tokio 不能。

### 21.5.2 代价落在哪

两种哲学都没有银弹,代价落在不同的地方:

| | Go runtime 的代价 | Tokio 的代价 |
|---|---|---|
| 心智 | 低(写顺序代码,runtime 管异步) | 高(`async/await`、`Pin`、`Send`/`Sync`、取消语义) |
| 性能 | runtime 开销(GC/STW/屏障) | 接近零运行时开销(无 GC) |
| 死循环 | 能兜底(信号抢占) | 不能(必须显式 `yield_now`) |
| 阻塞 API | 自动让出 | 必须换 async 版本 |
| 内存控制 | GC 决定(虽然 1.19 有 SOFT 限制) | 精确(ownership) |

一句话:**Go 用 runtime 复杂换用户心智简单;Tokio 用用户心智复杂换 runtime 简单。** 这是两种哲学的本质 trade-off,没有绝对优劣,各适其域——Go 适合"快速写高并发后端",Rust/Tokio 适合"对延迟/内存极致敏感的系统级编程"。

## 21.6 技巧精解:这套哲学里最 sound 的三个论证

前 20 章讲了大量技巧,这一节挑三个**贯穿全书、最能体现"为什么 sound"**的论证,作为收尾——它们是 Go runtime 这套复杂机器能正确运转的物理基础。

### 21.6.1 写屏障为什么不会漏标活对象(第 13 章)

这是"有 GC"这条哲学能成立的核心论证。

**问题**:并发标记时,GC 已经把对象 A 标黑(认为扫干净了),业务却在 A 上塞了一个 GC 还没扫到的白对象 C 的指针(`A.f = C`)。GC 再也看不见 C,把它当垃圾回收——活对象丢了。

**Go 1.8 混合屏障的论证**:屏障在**每次写堆指针**时,把**被写的旧值染灰 + 新值染灰**(混合屏障 = Dijkstra 插入 + Yuasa 删除的叠加)。这样:

- 业务写 `A.f = C`:屏障把 C 染灰(C 一定被扫到)。
- 业务写 `A.f = C` 覆盖旧值 B:屏障把 B 染灰(B 不会被丢)。

无论业务怎么改指针,所有"可能让 GC 看漏"的对象都被屏障染灰,GC 顺着灰链一定能扫到。这就是"强弱三色不变式"被维持的物理含义——**屏障把"业务改指针"这件危险事,变成了 GC 一定能看见的事**。

> **反面对比**:如果没有写屏障,并发标记会在"白→黑路径上丢失灰"——经典三步漏标。Go 1.8 之前用 Dijkstra 插入屏障(只染新值),但要在栈上也要屏障(代价大);1.8 改混合屏障,只需堆上屏障(栈在 STW 时一次性扫,之后栈上对象默认当灰),把屏障开销从"每次写"降到"只堆写"。这是"有 GC"这件事能又 sound 又快的关键工程。

### 21.6.2 work-stealing 为什么不会丢 G 也不会重复跑(第 4 章)

这是"协作+抢占"和"分层缓存"两条哲学在调度热路径上的论证。

**问题**:两个 P 同时改 runq 的 head/tail,凭什么不丢 G、不重复跑、不死锁?

**论证**:Go 的 runq 是 256 槽固定环形数组,owner 单写 tail(只 owner 一个人改 tail)、thief 和 owner 都改 head 但走 CAS。关键规约:

- owner push/pop:cas 改 head(乐观),失败说明有 thief 在偷,重试或回退。
- thief grab:cas 改 head,把偷的半数 G 拷到自己的本地数组,再 cas 确认——确认前不"声明偷到",确认后才把 G 当自己的。

这样:**一个 G 同一时刻要么在 owner 端、要么在 thief 端,cas 是唯一的状态切换点,不可能同时被两边拿到**(cas 成功的一方独占)。偷一半是为了摊平负载(几轮 steal 就均衡),且让 steal 频率低(head 不会成为热点 CAS)。

> **反面对比**:如果用一把全局锁保护 runq,那所有 M 抢同一把锁,work-stealing 的扩展性红利全没了。如果用朴素的双端队列不做 CAS 协议,会出现"两个 P 都认为自己拿到了同一个 G"——重复执行,或者"G 在两人交接时丢失"——永远跑不到。Go 的 owner 单写 tail + head CAS + 偷一半,是这套论证的最小 sound 实现。

### 21.6.3 栈拷贝为什么不会损坏指针(第 17 章)

这是"有 GC"+"分层缓存"+"语言内置"三条哲学在栈管理上的交汇论证。

**问题**:goroutine 的栈要按需增长(2KB → 4KB → 8KB),增长时要拷贝整个栈到新位置。栈上有指针(指向栈上其他位置、指向堆、指向全局),拷贝后地址全变了,凭什么不损坏?

**论证**:Go 的连续栈拷贝(`copystack`)做三件事:

1. 分配新栈(翻倍大小)。
2. **逐字扫描旧栈**,对每个字判断"这是不是指针"——靠编译器生成的栈指针位图(`stackmap`,编译器知道每个函数的栈帧里哪些槽是指针)。
3. 是指针的,按"指向旧栈 → 调整成指向新栈""指向堆/全局 → 不变"的规则调整。

这样:**栈拷贝只在安全点进行(GC 也要求这个点),栈指针位图由编译器精确提供,所以每个指针都被正确调整**。栈拷贝后,所有指向栈内的指针指向新位置,指向堆的不变——没有悬挂指针、没有漏调。

> **反面对比**:如果像 segmented stack(链式栈)那样,栈分成多段,段间跳转要保存链接,且指针可能跨段——调整极复杂,且有性能抖动(Go 1.3 之前用过,1.4 改连续栈)。Tokio 选了另一条路——Future 是状态机不是栈,但自引用问题用 `Pin` 类型系统禁止移动(见 17.9 对照)。**Go 让栈能搬家(运行时调整指针),Tokio 让 Future 不能搬家(类型系统禁止),两种解法都 sound,代价落在不同地方。**

## 21.8 章末小结

这是全书最后一章,我们退后一步,把 20 章里反复出现的五条哲学提炼出来,并用一张双璧对照总表把这本书和《Tokio》钉在一起。

### 五条贯穿哲学

1. **把并发做进语言**:`go`/`chan`/`select` 是语法,runtime 默认就在,阻塞 API 被自动拦截让出——这是其他四条的前提(回扣第 1、8、13~16 章)。
2. **协作+抢占混合**:默认协作(便宜),死循环才用 SIGURG 信号异步抢占(兜底)——既不饿死又不过度开销(回扣第 5、7 章)。
3. **有 GC 的取舍**:并发三色 GC 换零心智,代价是三色不变式/写屏障/mark assist/STW 这一整套复杂机器(回扣第 4 篇、第 17 章)。
4. **分层缓存换无锁快路径**:热路径本地无锁(P 的 runq/mcache/wbBuf/timers),慢路径才碰锁——这是 Go runtime 高性能的物理根基(回扣第 2、3、4、11、20 章)。
5. **STW 最小化的执念**:亚毫秒停顿是设计目标不是优化结果,贯穿 GC 架构的每一步(回扣第 14 章)。

这五条互相咬合:哲学一是地基,哲学二三四是支柱,哲学五是屋顶。合起来就是 Go runtime 的整张架构图。

### 五个"为什么"清单(收束版)

1. **为什么 Go 把并发做进语言,而不是做成库?** 因为只有语言内置才能拦截阻塞 API(`conn.Read`/`sleep`/`chan`),实现"用户写阻塞、runtime 偷偷异步"的零成本红利。库级(Tokio)做不到,必须显式 `async/await`。
2. **为什么 Go 用协作+抢占混合,而不是纯抢占?** 协作便宜(99% 的 G 在阻塞点自然让出),纯抢占要保存完整上下文极贵。混合 = 协作的便宜 + 抢占的兜底,死循环才动用信号抢占。
3. **为什么 Go 选 GC 而不是 ownership?** 赌注是"绝大多数后端程序,GC 开销小于手动管理的心智+bug 成本"。代价是 runtime 要扛三色/写屏障/mark assist/STW,但 Go 把这套做到"几乎不可见"(亚毫秒 STW)。
4. **为什么 Go 的热路径几乎不碰锁?** 因为分层缓存——P 持 runq/mcache/wbBuf/timers 等本地缓存,本地命中无锁,不命中才回退到 mcentral/sched.lock 等加锁结构。性能不是靠一处神优化,而是靠"绝大多数热路径无锁"的贯穿纪律。
5. **Go runtime 和 Tokio 的根本区别是什么?** Go 是语言内置协程(编译器+runtime 协作,阻塞式 API 自动让出,有 GC,信号抢占),Tokio 是库级异步(纯类型系统+Future,显式 async/await,无 GC 靠 ownership/Pin,纯协作)。同目标(少量线程驱动海量并发),两哲学,代价落在不同地方。

### 收束语

这本书从一句话开始:**"如何用极少的 OS 线程,高效驱动成千上万个 goroutine,让阻塞不阻塞线程、让 GC 不打断业务。"** 20 章之后,这句话不再是口号,而是几百个具体机制的总和——GMP 的结构布局、`schedule` 的多级回退、work-stealing 的无锁环形数组、SIGURG 的异步抢占、`entersyscall` 的 handoff、sysmon 的独立观察、channel 的快路径省 buf、三色 + 写屏障的防漏标、并发 GC 的亚毫秒 STW、mark assist 的反压、连续栈拷贝的指针调整、netpoll 把 epoll 藏进 runtime、Mutex 的自旋+信号量、timer 的四叉堆。

这些机制不是孤岛,它们背后是五条贯穿哲学,而五条哲学合起来,就是"语言内置协程"这种设计的全貌。和《Tokio》的"库级异步"对读,你能看清:调度问题的本质是相通的(少量线程驱动海量并发、阻塞不占线程、负载均衡),但解法因语言哲学而截然不同——Go 用 runtime 复杂换用户心智简单,Tokio 用用户心智复杂换 runtime 简单。

读完这本书,你下次写 `go func`、`ch <- x`、`conn.Read`、`time.Sleep` 时,看到的不再是一行代码,而是 GMP 调度器、channel 等待队列、netpoll、timer 堆在背后运转的整台机器。你也能在性能调优(调 `GOMAXPROCS`/`GOGC`、用 `go tool trace` 找瓶颈)、故障排查(GC 抖动、调度延迟、channel 死锁)、技术选型(该用 Go 还是 Rust/Tokio)时,有 runtime 内部视角的底气。
