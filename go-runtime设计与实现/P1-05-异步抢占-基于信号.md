# 第五章 · 异步抢占:基于信号的协作式→抢占式

> 篇:第 1 篇 · GMP 调度器(全书地基,重头戏)
> 主线呼应:上一章我们把 work-stealing 拆透了——一个 P 本地 runq 空了,会主动去别的 P 那里偷一半 G 回来。但 work-stealing 有个隐含前提:**那个 P 得有机会让出 CPU,进 `schedule` 取下一个 G**。如果某个 G 死循环不让出呢?它占着 P 不放手,本地 runq 里别的 G 排队等着挨饿,全局 GC 的 STW 也卡在它身上(它不进安全点,标记终止就停不下来)。这一章就钻进 Go 1.14 引入的**异步抢占**:sysmon 发现在一个 P 上跑超过 10ms 的 G,就给那条线程投一个 `SIGURG` 信号,让它在几乎任何指令处被打断、被塞回 runq。我们拆透三件事:**为什么 Go 1.14 之前的协作式抢占搞不定死循环、信号凭什么能在任意安全点打断 G 而不破坏 GC 和栈、那段 `asyncPreempt` 汇编到底干了什么**。

## 核心问题

**一个 G 死循环不让出会饿死别人、还会挡住 GC 的 STW。Go 1.14 前的协作式抢占(函数前奏检查 `stackguard`)为什么搞不定死循环?异步抢占怎么靠 `SIGURG` 在任何安全点打断 G,又凭什么这个打断不破坏 GC/栈/内存序?**

读完本章你会明白:

1. 为什么需要抢占:**调度和 GC 都依赖 G 进安全点让出**,一个不让出的 G 会饿死本地 runq 里别的 G、还会让 STW 拖到秒级。抢占是 runtime 让"调度执行"和"并发 GC"成立的硬性前提。
2. 协作式抢占的机制和它的天花板:编译器在每个函数前奏插一段"比较 SP 和 `stackguard0`"的检查,runtime 把 `stackguard0` 改成一个魔法值 `stackPreempt`,G 下次进函数前奏就进 `newstack` 走 `gopreempt_m`。但它**只在 G 调函数时才生效**——一个紧凑的 `for {}` 死循环不调函数,就永远不被打断。
3. 异步抢占怎么补上这个洞:sysmon 通过 `retake` 检测到 G 跑超过 10ms,`preemptone` 置标志并 `preemptM` 给那条 M 的线程投 `SIGURG`;信号到达后内核切到信号栈,`sighandler`→`doSigPreempt` 判安全点,`pushCall` 把 `asyncPreempt` 的地址"塞进"被打断的指令流,G 从信号返回时直接跳进 `asyncPreempt` 这个伪装的"函数调用",保存所有寄存器后进调度器。
4. 安全点判定凭什么 sound:`isAsyncSafePoint` 用编译器打的 `PCDATA_UnsafePoint` 把"原子序列(写屏障)、nosplit 函数、汇编函数、runtime/reflect 包内"全标成不可抢占,加上栈空间检查和保守扫描约定,保证打断处能安全地保守扫描栈和寄存器、能从打断处"回放"恢复执行。

> 逃生阀:这一章真正难的是**安全点判定的正确性**——为什么在某些指令上打断会丢指针、漏标活对象、撕裂原子序列。如果你读到 `isAsyncSafePoint` 那串 `pcdatavalue2` + `funcFlag` 检查觉得绕,先抓主干:**编译器给每条指令打了个"能不能在这打断"的标,`isAsyncSafePoint` 只是个查表器,把"不可打断"的地方一律放过去,等下一拍信号再试**。其余都是它为什么 sound 的展开。

## 5.1 一句话点破

> **抢占不是"调度器主动切换 G",而是"调度器请求目标线程自己切换自己":协作式抢占把这个请求编码进函数前奏的栈检查(只有进函数才看见),异步抢占把这个请求用 `SIGURG` 信号投到目标线程(几乎任何指令都能看见)。安全性的命脉全在"哪些指令上打断是 sound 的"——靠编译器给每条指令打 `PCDATA_UnsafePoint` 标签,信号处理器查这张表,只放行可保守扫描、可回放的安全点。**

这是结论,不是理由。本章倒过来拆:先看为什么 Go 一定要有抢占(死循环的两宗罪),再看协作式抢占怎么工作、为什么够不到死循环,然后拆透异步抢占的信号链路和安全点判定凭什么不破坏 GC/栈,最后讲 `asyncPreempt` 那段汇编是怎么把"信号打断"伪装成"一次普通函数调用"。

## 5.2 为什么一定要有抢占:死循环的两宗罪

第 3 章讲 `schedule` 时,我们一直默认一个前提:G 跑着跑着会**主动回到调度器**,让别的 G 上场。靠什么主动?两类:

- **阻塞让出**:G 阻塞在 channel / 网络 / sleep / syscall,被 park(第 2、6、8 章会展开)。这种 G 是"自愿"让出的。
- **抢占让出**:G 既不阻塞也不结束,但 runtime 强行让它让出。

第二类就是本章主题。先看没有它会撞什么墙。

### 5.2.1 罪一:饿死本地 runq 里的别的 G

设想一段再朴素不过的代码:

```go
go func() { for {} }()      // G1: 死循环,不调任何函数
time.Sleep(time.Second)     // G2: 主 goroutine 想睡 1 秒
```

在单核(`GOMAXPROCS=1`)的机器上,G2 把 CPU 让给 G1 之后,G1 **永远不会把 CPU 还回来**。原因:Go 调度器是**协作式**的——`schedule` 切换 G 的唯一时机是 G 自己阻塞、自己结束、或者在某个**安全点**主动检查"该不该让出"。G1 死循环里既不阻塞也不结束,而它跑的循环体如果连函数都不调(`for {}` 编译出来就是 `JMP` 自己),那连协作式抢占的检查点都进不去。G2 永远醒不过来。

> **不这样会怎样**:这就是 Go 1.14 之前长时间被诟病的"goroutine 调度不保证公平"——只要你不主动让出,就能霸占 P。社区曾经靠 `runtime.Gosched()` 或者"循环里塞个函数调用"来规避。但这不是治本之道:你写不出保证公平的调度器,各种延迟敏感的服务就会出毛刺。

### 5.2.2 罪二:挡住 GC 的 STW

更严重的是,GC 依赖所有 G 进安全点才能停 world。第 4 篇会展开,这里先给一句:并发 GC 在**标记终止**阶段需要一次极短的 STW,把所有 G 的栈扫一遍、确认根集合稳定。这次 STW 要求**每个 G 都不在用户代码的临界区里**(要么 park 了,要么在安全点)。如果一个死循环 G 永远不进安全点,GC 的 STW 就**永远停不完**——`stopTheWorld` 会一直等它。

实际工程里这是个真问题:有人写个紧密计算的 goroutine,跑着跑着 GC 触发,STW 从设计的亚毫秒级飙到几秒甚至几十秒,P99 延迟爆炸。Go 1.14 之前,这类 bug 是排查噩梦。

> **钉死这件事**:Go 调度器和并发 GC 的整个设计,都建立在"任意 G 都会被强制推进到一个安全点"之上。**没有抢占,就没有公平调度,也没有低延迟 GC。** 异步抢占不是优化,是基础。

## 5.3 协作式抢占:把请求塞进函数前奏

理解异步抢占之前,必须先理解它的前身——**协作式抢占**(synchronous preemption)。这两者用同一套标志位(`gp.preempt`、`pp.preempt`、`gp.stackguard0 = stackPreempt`),区别只在"目标 G 什么时候看到这个标志"。

### 5.3.1 编译器在每个函数前奏插的栈检查

Go 编译器为**每一个**非叶子函数生成一段"前奏"(prologue),在函数体执行前先检查"栈还够不够、要不要抢占"。简化后长这样(汇编示意,非源码原文):

```asm
; 函数 f 的前奏(简化示意)
cmp   R14, [gs:stackguard0]   ; 比较 SP 与 stackguard0
jbe   morestack                ; SP <= stackguard0,跳到 morestack
sub   SP, frame_size           ; 否则正常开栈,继续函数体
...
morestack:
    call runtime.morestack     ; 进 runtime 处理
    jmp  f                     ; 处理完回来重试前奏
```

关键在 `cmp R14, stackguard0` 这一行:`R14` 在 amd64 上是当前硬件 SP(实际是 TLS 里读出的 SP,简化成"SP"),`stackguard0` 是 G 里存的"栈底警戒线"。正常情况下 `stackguard0 = gp.stack.lo + stackGuard`,SP 比它小(栈用到底了)就说明栈要溢出,跳 `morestack` 去扩栈(第 17 章详讲)。

注意 `cmp` 之后 `jbe`——**无符号比较,SP <= stackguard0 就跳**。这个"无符号比较"是协作式抢占能复用这条路径的钥匙(见 5.3.2)。

### 5.3.2 投毒 stackguard0:把抢占折叠进栈检查

runtime 要抢占一个 G 时,不另开检查路径,而是**把 `stackguard0` 改成一个比任何真实 SP 都大的魔法值**,让 `cmp` 一定"失败",从而跳进 `morestack`。这个魔法值就是 [`stackPreempt`](../go/src/runtime/stack.go#L133):

```go
// src/runtime/stack.go#L124-L137(节选)
const (
    uintptrMask = 1<<(8*goarch.PtrSize) - 1

    // Goroutine preemption request.
    // 0xfffffade in hex.
    stackPreempt = uintptrMask & -1314
    ...
)
```

`uintptrMask & -1314` 在 64 位上等于 `0xfffffffffffffade`——一个天文数字,比任何真实 SP 都大得多。所以一旦 `stackguard0` 被写成 `stackPreempt`,函数前奏的 `cmp SP, stackguard0`(无符号比较)一定满足"SP <= stackguard0",`jbe` 一定跳 `morestack`。

`morestack` 走到 [`newstack`](../go/src/runtime/stack.go#L1026),后者第一件事是判定"这次进来到底是要扩栈,还是要抢占":

```go
// src/runtime/stack.go#L1079-L1101(节选)
stackguard0 := atomic.Loaduintptr(&gp.stackguard0)

// Be conservative about where we preempt.
preempt := stackguard0 == stackPreempt
if preempt {
    if !canPreemptM(thisg.m) {
        // Let the goroutine keep running for now.
        // gp->preempt is set, so it will be preempted next time.
        gp.stackguard0 = gp.stack.lo + stackGuard
        gogo(&gp.sched) // never return
    }
}
```

`preempt := stackguard0 == stackPreempt`——如果 `stackguard0` 等于那个魔法值,这次 `newstack` 就不是要扩栈,而是要抢占。接着 [`canPreemptM`](../go/src/runtime/preempt.go#L289) 检查"现在能不能安全抢占":

```go
// src/runtime/preempt.go#L288-L291
//go:nosplit
func canPreemptM(mp *m) bool {
    return mp.locks == 0 && mp.mallocing == 0 && mp.preemptoff == "" &&
        mp.p.ptr().status == _Prunning && mp.curg != nil &&
        readgstatus(mp.curg)&^_Gscan != _Gsyscall
}
```

六个条件,缺一不可:`mp.locks == 0`(没持 runtime 锁)、`mp.mallocing == 0`(没在 malloc)、`mp.preemptoff == ""`(没显式禁用抢占,GC STW 期间会设它)、P 状态是 `_Prunning`、有当前 G、当前 G 不在 syscall(syscall 走另一条 handoff 路径,见第 6 章)。如果其中任何一条不满足,抢占请求**取消**——把 `stackguard0` 恢复成正常的栈警戒线,`gogo` 回去继续跑这个 G(它会"下一次"再被检查)。

如果 `canPreemptM` 通过,`newstack` 走到 `gopreempt_m(gp)`,后者把 G 从 `_Grunning` 改成 `_Grunnable` 塞回 runq,然后 `schedule` 取别的 G 来跑——抢占完成。

> **所以这样设计**:把抢占折叠进栈检查,是一个极妙的复用——编译器本来就要在每个函数前奏插栈检查(防栈溢出),抢占只是把"警戒线"换成"魔法值",零额外指令开销。普通路径(SP > stackguard0,前奏只多一条 `cmp + jbe`)几乎免费;抢占路径(走 `newstack`)极少发生。

### 5.3.3 协作式抢占的天花板:够不到死循环

但这里有个致命问题:**前奏检查只在"进函数"时发生**。如果 G 跑的是一个**紧凑循环,不调任何函数**:

```go
for { x++ }   // 编译成 inc + jmp,没有函数调用,没有前奏检查
```

那么这段代码**永远不会**走到函数前奏的 `cmp stackguard0`,`gp.stackguard0 = stackPreempt` 这个"投毒请求"就永远不会被看见。死循环的 G 照样霸占 P。

这就是协作式抢占的本质局限:**它要求 G 主动进入检查点**。一个不愿配合的(或无意中写出来的)紧凑循环,就绕过了它。在 Go 1.14 之前,这是 goroutine 调度公平性的一根硬刺——程序员的循环体里必须确保有函数调用、channel 操作或显式的 `runtime.Gosched()`,否则该 G 可能永远不让出 CPU。社区的 workaround 包括"循环里塞个 atomic add""编译器开 `-N` 关 inline 让循环里多个函数调用"之类,全是治标。

> **钉死这件事**:协作式抢占 = 把抢占请求编码进"函数前奏",但**只有调函数的 G 才看得到**。死循环够不到。Go 1.14 引入异步抢占,就是为了补上"不调函数的 G 也能被打断"这最后一公里。

## 5.4 异步抢占:用 SIGURG 在任意指令打断

异步抢占的思路乍听很简单:**既然协作式抢占的检查点是"函数前奏",那我就用信号强行在"任意指令"处造一个检查点**。具体做法分四步,我们逐一拆。

### 5.4.1 第一步:sysmon 检测超时 G

第 7 章会详讲 sysmon,这里只取和抢占相关的部分。sysmon 是一个**不受 GC/调度约束**的独立后台线程(它跑在自己的 g0 上,不进 runq),每隔最多 10ms 醒来一次,扫一遍所有 P,调用 [`retake`](../go/src/runtime/proc.go#L6681) 检查"每个 P 是否在同一个 `schedtick` 上跑太久":

```go
// src/runtime/proc.go#L6677-L6722(节选)
const forcePreemptNS = 10 * 1000 * 1000 // 10ms

func retake(now int64) uint32 {
    ...
    for i := 0; i < len(allp); i++ {
        pp := allp[i]
        if pp == nil || atomic.Load(&pp.status) != _Prunning {
            continue
        }
        pd := &pp.sysmontick

        schedt := int64(pp.schedtick)
        if int64(pd.schedtick) != schedt {
            pd.schedtick = uint32(schedt)
            pd.schedwhen = now
        } else if pd.schedwhen+forcePreemptNS <= now {
            // ★ 在同一个 schedtick 上跑了超过 10ms,抢占它
            preemptone(pp)
            sysretake = true
        }
        ...
    }
}
```

`schedtick` 是"这个 P 上跑了多少次 schedule"的计数(每次 `schedule` 取一个 G 进来就自增)。`pd.schedtick` 是 sysmon 上次看到的 `schedtick` 快照。如果**两个相等,且距离 `pd.schedwhen` 超过 10ms**,说明这个 P 自上次 sysmon 扫描以来,**一直没有切过 G**——同一个 G 跑了 10ms 还没让出。这就是死循环或者超长计算的特征。

`forcePreemptNS = 10 * 1000 * 1000` = 10ms,这是 Go 的时间片大小。10ms 看起来"很大",但这是**抢占触发阈值**,不是切换间隔——一个 G 如果在 10ms 内自己阻塞/让出了,根本不会被抢占;只有真正霸占 P 超过 10ms 的,sysmon 才介入。

> **为什么 sysmon 是独立线程**:第 7 章详讲。一句话:如果检测超时 G 的逻辑跑在某个被 GC 或被自己要抢占的 G 拖住的 P 上,它就永远跑不起来。sysmon 在自己的线程上独立运行,是抢占能落地的前提。

### 5.4.2 第二步:preemptone 置标志并发信号

[`preemptone`](../go/src/runtime/proc.go#L6917) 做两件事——**置标志**(给协作式抢占看)和**发信号**(给异步抢占看):

```go
// src/runtime/proc.go#L6917-L6946(节选)
func preemptone(pp *p) bool {
    mp := pp.m.ptr()
    if mp == nil || mp == getg().m {
        return false
    }
    gp := mp.curg
    if gp == nil || gp == mp.g0 {
        return false
    }
    if readgstatus(gp)&^_Gscan == _Gsyscall {
        // Don't bother trying to preempt a goroutine in a syscall.
        return false
    }

    gp.preempt = true

    // Every call in a goroutine checks for stack overflow by
    // comparing the current stack pointer to gp->stackguard0.
    // Setting gp->stackguard0 to StackPreempt folds
    // preemption into the normal stack overflow check.
    gp.stackguard0 = stackPreempt

    // Request an async preemption of this P.
    if preemptMSupported && debug.asyncpreemptoff == 0 {
        pp.preempt = true
        preemptM(mp)
    }

    return true
}
```

三处关键:

1. **`gp.preempt = true` + `gp.stackguard0 = stackPreempt`**:这是**协作式抢占的请求**。如果 G 在被信号打断之前,正好进了一个函数前奏,它会通过 `newstack` 走 `gopreempt_m` 让出。这两行就是 5.3 的"投毒"。
2. **`pp.preempt = true`**:把 P 也标上"有抢占请求"。这个标志在 `wantAsyncPreempt` 里和 `gp.preempt` 一起被查(见 5.4.4)——为什么要查两个?因为抢占请求可能在 G 之间切换,**新的 G 可能不知道有未处理的抢占请求**,但 P 的标志会留着。查 P 的标志让"刚切上来的 G"也能看到上一个 G 留下的抢占请求。
3. **`preemptM(mp)`**:这是**异步抢占的核心**——给那条 M(操作系统线程)发一个 `SIGURG` 信号。下一节详拆。

`debug.asyncpreemptoff` 是个调试开关,设成 1 就**完全关闭异步抢占**(只留协作式),用于复现 Go 1.14 之前的行为或者排查异步抢占本身的 bug。`preemptMSupported` 在 Linux/Darwin/FreeBSD 等上是 `true`,在不支持信号机制的平台(如 wasm)上是 `false`。

### 5.4.3 第三步:preemptM 投递 SIGURG

[`preemptM`](../go/src/runtime/signal_unix.go#L369) 是异步抢占的"发令枪"——它通过 `signalM` 给目标线程投一个信号:

```go
// src/runtime/signal_unix.go#L369-L392
func preemptM(mp *m) {
    // On Darwin, don't try to preempt threads during exec.
    // Issue #41702.
    if GOOS == "darwin" || GOOS == "ios" {
        execLock.rlock()
    }

    if mp.signalPending.CompareAndSwap(0, 1) {
        if GOOS == "darwin" || GOOS == "ios" {
            pendingPreemptSignals.Add(1)
        }
        // If multiple threads are preempting the same M, it may send many
        // signals to the same M such that it hardly make progress, causing
        // live-lock problem. Apparently this could happen on darwin. See
        // issue #37741.
        // Only send a signal if there isn't already one pending.
        signalM(mp, sigPreempt)
    }
    ...
}
```

两个细节:

1. **`sigPreempt` 是 `SIGURG`**:见 [`signal_unix.go#L74`](../go/src/runtime/signal_unix.go#L74),`const sigPreempt = _SIGURG`。为什么是 SIGURG?源码注释([`signal_unix.go#L67-L73`](../go/src/runtime/signal_unix.go#L67))说得很清楚——SIGURG 满足抢占信号的所有要求:(1) 默认动作是忽略(不会因为没装 handler 把进程搞死);(2) 应用几乎不会真正用它的"语义"(TCP 带外数据到达,但带外数据基本没人用,且 SIGURG 还不告诉你哪个 socket,语义基本没用);(3) 即使应用真用了 SIGURG,它也得准备好接受"虚假"的 SIGURG(POSIX 允许)。相比之下 SIGUSR1/SIGUSR2 太容易被应用占用,SIGALRM/SIGPROF 各有用途,SIGIO 也比 SIGURG 更可能被真用。所以 Go 工程师挑了 SIGURG 这个"几乎没人要"的信号。

2. **`mp.signalPending.CompareAndSwap(0, 1)` 合并请求**:这是个**单槽**的"有没有未处理信号"标志。如果已经有一个 `SIGURG` 在飞,就不重复发——避免多条线程同时抢占同一个 M,导致那个 M 频繁被信号打断、几乎不前进的 **live-lock**(Darwin 上特别容易触发,issue #37741)。信号在 handler 里处理完会把 `signalPending` 清零(见 5.4.4),所以新的抢占请求总能再发。

`signalM` 在 Linux 上是 `tgkill` 系统调用——给指定线程投指定信号,只打那一个线程,不打整个进程。这是异步抢占"精确"的前提:只打断正在跑那个 G 的 M,不影响别的 M。

### 5.4.4 第四步:信号到达,doSigPreempt 判安全点并改写 PC

信号到达目标线程后,内核把线程挂起,切到该线程的**信号栈**(通常是 `gsignal` 的栈),把当前所有寄存器快照存进一个 `ucontext`,然后跳进 Go 注册的信号入口 `sigtramp`(汇编)→ `sigtrampgo` → `sighandler`。在 `sighandler` 里,如果识别出这是 `sigPreempt`,就走 [`doSigPreempt`](../go/src/runtime/signal_unix.go#L342):

```go
// src/runtime/signal_unix.go#L342-L359
func doSigPreempt(gp *g, ctxt *sigctxt) {
    // Check if this G wants to be preempted and is safe to
    // preempt.
    if wantAsyncPreempt(gp) {
        if ok, newpc := isAsyncSafePoint(gp, ctxt.sigpc(), ctxt.sigsp(), ctxt.siglr()); ok {
            // Adjust the PC and inject a call to asyncPreempt.
            ctxt.pushCall(abi.FuncPCABI0(asyncPreempt), newpc)
        }
    }

    // Acknowledge the preemption.
    gp.m.preemptGen.Add(1)
    gp.m.signalPending.Store(0)
    ...
}
```

两个判定 + 一个改写:

**判定一:`wantAsyncPreempt(gp)`**——这个 G 真的还想被抢占吗?见 [`preempt.go#L369`](../go/src/runtime/preempt.go#L369):

```go
// src/runtime/preempt.go#L369-L372
func wantAsyncPreempt(gp *g) bool {
    // Check both the G and the P.
    return (gp.preempt || gp.m.p != 0 && gp.m.p.ptr().preempt) &&
        readgstatus(gp)&^_Gscan == _Grunning
}
```

为什么要再查一次?因为**信号到达和信号发出之间有窗口**,这个窗口里 G 可能已经被别的方式让出了(比如自己阻塞)、或者抢标志已经被清掉、或者 G 已经不在 `_Grunning`(可能在 GC 扫栈)。这时候再抢占就是错的。所以信号 handler 要"现查一次"。这是个典型的 **recheck 模式**:发信号时置标志,handler 里再确认标志还在,防止"陈旧请求"被误执行。

**判定二:`isAsyncSafePoint(gp, pc, sp, lr)`**——信号打断时这条指令是安全点吗?这是本章的核心正确性,下一节单独拆。

**改写:`ctxt.pushCall(asyncPreempt, newpc)`**——如果两个判定都通过,**不真的去执行 `asyncPreempt`,而是改写 `ucontext` 里的 PC 和 SP**,让它"看起来像"刚刚有人在这里调用了 `asyncPreempt`。然后 `doSigPreempt` 返回,信号 handler 返回,内核用改写后的 `ucontext` 恢复线程——线程一恢复就跳到 `asyncPreempt` 的入口,完全不知道自己刚被信号打断过。

最后两行 `preemptGen.Add(1)` 和 `signalPending.Store(0)` 是**确认接收**——清掉"有信号在飞"的标志,让 `preemptM` 可以再发下一个;`preemptGen` 是个代际计数,用于检测"我发出去的信号到底被处理了几次"(debug 用)。

### 5.4.5 pushCall:把信号打断伪装成一次函数调用

[`pushCall`](../go/src/runtime/signal_amd64.go#L80) 是 amd64 上"伪装函数调用"的实现,短短 6 行,极其精妙:

```go
// src/runtime/signal_amd64.go#L80-L87
func (c *sigctxt) pushCall(targetPC, resumePC uintptr) {
    // Make it look like we called target at resumePC.
    sp := uintptr(c.rsp())
    sp -= goarch.PtrSize
    *(*uintptr)(unsafe.Pointer(sp)) = resumePC
    c.set_rsp(uint64(sp))
    c.set_rip(uint64(targetPC))
}
```

`c.rsp()` / `c.set_rsp()` / `c.set_rip()` 读写的是 `ucontext` 里的寄存器快照(不是当前真实寄存器——handler 跑在信号栈上,真实寄存器是 handler 自己的)。这段代码干的事:

1. `sp := c.rsp()`:读出被信号打断那一刻的 SP。
2. `sp -= goarch.PtrSize`:SP 往下挪一个指针长度(8 字节),腾出一个槽。
3. `*(*uintptr)(sp) = resumePC`:把**被打断的指令地址**(经过安全点修正后的 `newpc`)写进这个新腾出的槽——**这就是伪造的"返回地址"**。
4. `c.set_rsp(sp)`:更新 SP。
5. `c.set_rip(targetPC)`:把 PC 设成 `asyncPreempt` 的入口。

画成图:

```
 信号打断时 ucontext(简化):
 ┌──────────────┐
 │ RIP = PC被打断│ ← 线程恢复后从这里继续
 │ RSP = SP     │
 │ RAX, RBX,... │ ← 14 个通用寄存器
 │ XMM0..15     │ ← 16 个向量寄存器
 └──────────────┘

 pushCall 改写后 ucontext:
 ┌──────────────┐
 │ RIP = asyncPreempt │ ← 线程恢复后跳进 asyncPreempt
 │ RSP = SP - 8       │
 │ RAX, RBX,... │ ← 原封不动(寄存器值由 asyncPreempt 自己保存)
 │ XMM0..15     │
 └──────────────┘

  被打断线程的栈(从信号返回时):

     ┌────────────────┐  ← SP - 8 (pushCall 后的新 SP)
     │ resumePC(=newpc)│ ← 伪造的"返回地址":asyncPreempt 的 RET 会跳回这里
     ├────────────────┤  ← 原 SP
     │  原栈内容        │
     │  ...            │
```

效果是:线程从信号 handler 返回后,硬件按改写后的 `ucontext` 恢复寄存器——SP 是新值、RIP 是 `asyncPreempt` 入口。线程"以为"自己刚刚在 `newpc` 处执行了一次 `CALL asyncPreempt`,CALL 把返回地址 `newpc` 压了栈,然后跳进 `asyncPreempt`。实际上根本没有 CALL 指令执行——是 `pushCall` 手动模拟了 CALL 的效果(压返回地址 + 改 PC)。

`asyncPreempt` 执行完最后会 `RET`——`RET` 弹出栈顶的"返回地址"(也就是 `newpc`),跳回去。于是线程"返回"到 `newpc`,从被信号打断的那条指令重新执行——**对被打断的代码完全透明,它不知道自己被中断过**。

> **钉死这件事**:`pushCall` 把"信号异步打断"伪装成"一次普通的函数调用"——这样 `asyncPreempt` 不需要特殊的"从信号返回"路径,它就是个普通函数:正常保存寄存器、调用 Go 代码、RET 返回。这个伪装的精妙在于它复用了硬件的 CALL/RET 语义:`pushCall` 制造一个假 CALL,`asyncPreempt` 末尾的真 RET 完成对应的"返回"。整个抢占对被抢占代码而言像是一次透明的函数调用。

## 5.5 安全点判定:凭什么在任意指令打断而不出事

异步抢占最硬的正确性问题来了:**凭什么在任意指令处打断一个 G 是 sound 的?** 这一段我们逐条拆 `isAsyncSafePoint` 的判定,每一类"不能打断"都对应一个"如果在这打断会出什么问题"。

### 5.5.1 三类安全点

[`preempt.go#L5-L46`](../go/src/runtime/preempt.go#L5) 的注释把 Go 的安全点分成三类:

1. **阻塞安全点**:G 已经 park / 阻塞 / 在 syscall。这时候它的 CPU 状态最小,GC 完全知道它的栈根。这种不用信号,自然发生。
2. **同步安全点**:5.3 讲的函数前奏检查点。G 在每次函数调用的前奏看 `stackguard0`。代价是要 G 主动进函数。
3. **异步安全点**:**任意**用户指令,只要这条指令上"可以安全暂停 + 保守扫描栈和寄存器能找到所有根 + 有足够栈空间塞 `asyncPreempt` + 不在原子序列里 + 不持 runtime 锁"。这一类是异步抢占要解决的。

异步安全点的核心挑战是"**任意指令**"——不像同步安全点那样有结构(函数前奏),任意指令可能是任何状态:可能正在算一半的指针(寄存器里有个还没存进栈的指针)、可能正在执行原子序列的中间几条(写屏障的"灰指针 + 标记"序列)、可能是 nosplit 函数(没有栈溢出检查、不能扩栈)、可能是汇编函数(没有 Go 的栈帧元数据)。

`isAsyncSafePoint` 的任务就是把这些"不安全"的地方一律否决,只放行安全的。来看它怎么判。

### 5.5.2 isAsyncSafePoint 的六道检查

[`isAsyncSafePoint`](../go/src/runtime/preempt.go#L390) 是个长长的判定函数,核心是六道检查:

```go
// src/runtime/preempt.go#L390-L497(节选)
func isAsyncSafePoint(gp *g, pc, sp, lr uintptr) (bool, uintptr) {
    mp := gp.m
    // 1. 必须是用户 G,不是 g0/gsignal
    if mp.curg != gp {
        return false, 0
    }
    // 2. M 必须可抢占(canPreemptM:无锁、非 malloc、preemptoff 空、P 在跑、非 syscall)
    if mp.p == 0 || !canPreemptM(mp) {
        return false, 0
    }
    // 3. 栈要有足够空间塞 asyncPreempt
    if sp < gp.stack.lo || sp-gp.stack.lo < asyncPreemptStack {
        return false, 0
    }
    // 4. PC 必须是 Go 代码(有 func 信息)
    f := findfunc(pc)
    if !f.valid() {
        return false, 0   // 不是 Go 代码(cgo / 纯汇编),不放行
    }
    // 5. 这条指令不能被编译器标成 UnsafePointUnsafe(原子序列/nosplit 函数体内)
    up, startpc := pcdatavalue2(f, abi.PCDATA_UnsafePoint, pc)
    if up == abi.UnsafePointUnsafe {
        return false, 0
    }
    // 6. 不能是汇编函数(没有 locals pointer map)
    if fd := funcdata(f, abi.FUNCDATA_LocalsPointerMaps); fd == nil || f.flag&abi.FuncFlagAsm != 0 {
        return false, 0
    }
    // 7. 不能是 runtime/reflect 包内的函数
    name := ... // 函数名前缀
    if stringslite.HasPrefix(name, "runtime.") ||
        stringslite.HasPrefix(name, "internal/runtime/") ||
        stringslite.HasPrefix(name, "reflect.") {
        return false, 0
    }
    ...
}
```

每一道都对应一个"为什么不能在这打断"的硬约束。逐个拆:

**第 3 道:栈空间**。`asyncPreempt` 要往栈上写 14 个通用寄存器 + flags + BP + 一个伪造的返回地址([`preempt_amd64.s#L14`](../go/src/runtime/preempt_amd64.s#L14) 里 `ADJSP $112`,共 112 字节),还要走 `asyncPreempt2` 的栈帧。如果被打断时栈快用到底了,塞不进 `asyncPreempt` 就栈溢出。`asyncPreemptStack` 在 `init()` 里算出来(累加 `asyncPreempt` + `asyncPreempt2` + `xRegRestore` 的最大 SP 增量,再加 8 个指针的余量),保证"塞得下"。不够就不抢,等下一拍。

**第 4 道:必须是 Go 代码**。`findfunc(pc)` 查函数元数据表,查不到说明这条 PC 不在 Go 编译的代码里——可能是 cgo 调用的 C 函数、可能是没有元数据的纯汇编。这类地方 runtime 没有"栈帧结构信息",没法判定安全点,一律放行(返回 false)。`asyncPreempt` 不会在这里被注入。

**第 5 道:PCDATA_UnsafePoint 标签**。这是最关键的一道。**编译器在编译每个函数时,给函数内每条指令打了一张表**,表里记录这条指令是不是 unsafe-point。这张表通过 `pcdatavalue2(f, abi.PCDATA_UnsafePoint, pc)` 查询,返回值有几种:

- `UnsafePointSafe`:安全,可以在这打断。
- `UnsafePointUnsafe`:**不安全**——这通常是**原子序列**(写屏障那几条指令)、或者 **nosplit 函数体内**(没有栈检查、不能扩栈)。
- `UnsafePointRestart1` / `UnsafePointRestart2`:**可重启序列**——这是一段"无副作用的、可重头再来的"指令序列(比如某些位操作)。在这种序列中间打断是 OK 的,但恢复执行时要从**序列开头**重跑(因为可能执行了一半)。`isAsyncSafePoint` 返回 `startpc`(序列起始 PC),`pushCall` 用它作为"返回地址",这样 `asyncPreempt` RET 回去时,从序列开头重跑。
- `UnsafePointRestartAtEntry`:从**函数入口**重跑(用在某些特殊函数)。

> **关键洞察:安全点不是 runtime 判出来的,是编译器提前标好的**。编译器知道每条指令在干什么——它知道哪些指令是"原子序列中间不能断"(写屏障、CMPXCHG 之类)、哪些是"无副作用可重跑"。runtime 只是个查表器。这就是为什么异步抢占要**编译器 + runtime 协作**——光靠 runtime 在信号 handler 里分析当前指令是来不及、也不 sound 的,必须编译期就把"可断点"标好。

**第 6 道:不能是汇编函数**。汇编函数没有 Go 风格的 locals pointer map(GC 用来找栈上指针的元数据),保守扫描拿不到栈上哪些槽是指针。在汇编函数里打断、然后 GC 扫这个栈帧,要么漏标活对象(指针被当非指针)、要么误标(非指针当指针)——前者导致 GC 回收活对象,后者导致 GC 读到非法地址。所以汇编函数(通过 `f.flag & abi.FuncFlagAsm` 识别)一律不放行。

**第 7 道:runtime / reflect 包内的函数不放行**。这一道是个保守兜底。`runtime.*` 和 `internal/runtime.*` 的函数,编译器**已经把它们的所有指令标成 UnsafePointUnsafe**(因为 runtime 函数经常持锁、在原子序列里、在调度关键路径上),理论上第 5 道就够了。但注释里写明("these checks are theoretically redundant ... but for some reason, we can't eliminate these checks until issue #72031 is resolved")——实践中发现编译器的标注有遗漏,所以加这道基于函数名的兜底。`reflect.*` 因为 `reflect.makeFuncStub` / `methodValueCall` 这类有 untyped info 在栈上,扫栈会出错。

### 5.5.3 安全点的 sound 性:三道防线

把这六道检查归纳成"为什么 sound"的三道防线:

| 防线 | 防的是什么 | 哪道检查保证 |
|---|---|---|
| **栈和 GC 一致性** | 在 GC 扫栈时,G 的栈帧必须是"可扫描的"——有指针地图、寄存器里的指针都已存栈 | 第 4(有 func)、第 5(非原子序列)、第 6(非汇编)、第 7(非 runtime) |
| **原子性** | 不在写屏障 / CAS 这类多指令原子序列中间打断 | 第 5 的 `UnsafePointUnsafe` 标 |
| **可恢复性** | 重启序列要么完全可重跑(Restart1/2),要么从头重跑(RestartAtEntry) | 第 5 的 Restart 标签 + `pushCall` 用 `startpc` 作返回地址 |

> **反面对比**:假设 `isAsyncSafePoint` 只判"栈够不够",不查 `PCDATA_UnsafePoint`。那么写屏障执行到一半(比如已经把对象标灰、还没把指针写进工作队列)时被信号打断,G 进入 `asyncPreempt` → `gopreempt_m` 让出 CPU。GC 并发标记扫到这个 G 的栈,看到这个对象是灰的(被标过),但工作队列里没有它的子指针——**这个对象的子对象就漏标了,GC 会回收一个还活着的子对象**。这就是为什么第 5 道必须堵住原子序列中间的打断。Go 1.14 之前没有异步抢占,正是因为这个 sound 性极难保证——直到编译器引入 `PCDATA_UnsafePoint` 标注,runtime 才有了"在哪儿打断是安全的"的精确信息。

### 5.5.4 不安全的就放过:等下一拍

注意一个整体设计:**`isAsyncSafePoint` 否决时,抢占不发生,信号被确认接收(`signalPending` 清零),但 G 继续跑**。sysmon 下次扫到这个 P 还是超时,会**再发一次 `SIGURG`**——只要 G 还在跑,抢占请求就会不断重试,直到 G 跑到一个安全点。

这是个"尽力而为"的设计:抢占不保证立即发生,但保证**最终发生**(只要 G 跑到了任何安全点)。对于死循环 `for {}`,它编译成的 JMP 自己不在原子序列里、不在 runtime 包、栈够——是个安全点,所以会被抢。对于"在写屏障里跑很久"的 G,会被一拍一拍地放过,直到它跑出写屏障——但这本来就不该跑很久,真跑很久说明有 bug。

## 5.6 asyncPreempt 汇编:保存一切,调度,恢复一切

现在拆 `asyncPreempt` 这段汇编——它是"信号打断"和"调度器"之间的桥梁,负责**保存所有寄存器**、调用 `asyncPreempt2` 进入调度器、之后**恢复所有寄存器**返回被打断处。完整源码见 [`preempt_amd64.s`](../go/src/runtime/preempt_amd64.s#L8)(由 `mkpreempt.go` 自动生成),我们取关键骨架逐行解释。

### 5.6.1 入口:保存通用寄存器

```asm
// src/runtime/preempt_amd64.s#L8-L31
TEXT ·asyncPreempt(SB),NOSPLIT|NOFRAME,$0-0
    PUSHQ BP              ; BP 入栈
    MOVQ SP, BP           ; BP = SP(标准 Go 函数前奏)
    PUSHFQ                ; flags 入栈(因为后面会 clobber)
    ADJSP $112            ; 栈上腾 112 字节(14 个 GP 寄存器 × 8)
    NOP SP                ; 抑制 vet 的栈检查
    ; 保存 14 个通用寄存器
    MOVQ AX, 0(SP)
    MOVQ CX, 8(SP)
    MOVQ DX, 16(SP)
    MOVQ BX, 24(SP)
    MOVQ SI, 32(SP)
    MOVQ DI, 40(SP)
    MOVQ R8, 48(SP)
    MOVQ R9, 56(SP)
    MOVQ R10, 64(SP)
    MOVQ R11, 72(SP)
    MOVQ R12, 80(SP)
    MOVQ R13, 88(SP)
    MOVQ R14, 96(SP)
    MOVQ R15, 104(SP)
```

几个要点:

- **`NOSPLIT`**:`asyncPreempt` 自己没有栈检查——它已经在响应栈检查/信号了,不能再触发栈检查死循环。所以它必须是 `NOSPLIT`,而且必须有足够栈(由 `isAsyncSafePoint` 的第 3 道检查 `asyncPreemptStack` 保证)。
- **`NOFRAME`**:不生成标准的帧指针记录(因为这是手动管栈帧)。
- **`PUSHQ BP` + `MOVQ SP, BP`**:虽然 `NOFRAME`,但还是手动建了个 BP,方便栈回溯(panic / trace 时能跨过这个帧)。
- **`PUSHFQ`**:保存 flags 寄存器——因为后面的代码可能修改 flags(CMP 等),恢复时不能让被打断代码看到 flags 被改了。
- **`ADJSP $112`**:`ADJSP` 是个伪指令,编译器/汇编器理解它(普通的 SUB SP 它不认,会破坏栈检查元数据)。腾 112 字节 = 14 个 GP 寄存器(AX、CX、DX、BX、SI、DI、R8-R15)× 8 字节。
- **保存顺序**:AX 在 0(SP),CX 在 8(SP),...,R15 在 104(SP)。这个顺序在恢复时反着来。

注意**没保存 SP 本身**——SP 就是当前的栈指针,`asyncPreempt` 不需要"恢复 SP",因为 RET 会自动弹掉它压的东西。

### 5.6.2 保存向量寄存器到 P 的 xRegs

接下来是一段略长的代码,根据 CPU 是否支持 AVX512 / AVX2 / SSE,把 16 个(或 32 个)向量寄存器保存到**P 上的 `xRegs.scratch`** 区域:

```asm
// src/runtime/preempt_amd64.s#L32-L62(SSE 路径,节选)
    get_tls(CX)                                    ; CX = TLS(线程局部存储)
    MOVQ g(CX), R14                                ; R14 = 当前 G
    MOVQ g_m(R14), AX                              ; AX = 当前 M
    MOVQ m_p(AX), AX                               ; AX = 当前 P
    LEAQ (p_xRegs+xRegPerP_scratch)(AX), AX        ; AX = &p.xRegs.scratch
saveSSE:
    MOVUPS X0, 0(AX)
    MOVUPS X1, 64(AX)
    ...                                            ; 16 个 XMM 全存进 scratch
    JMP preempt
```

几个关键设计:

- **向量寄存器不存进栈,存进 P**:`p_xRegs` 是 P 上的一个区域。为什么不存栈?因为栈上没有指向 `xRegs` 的元数据,GC 扫栈时不知道这块是寄存器值还是指针;而 `xRegs` 是**标量**(向量寄存器里只放数值/向量,不含指针),存进 P 后 GC 不会扫它,**避开了"GC 把向量值当指针误标"的陷阱**。这是 `asyncPreempt` 注释里明说的设计:"It saves large additional scalar-only register state ... to an 'extended register state' on the P"。
- **为什么不存进 G**:存进 G 会增加每个 G 的常驻内存(G 的栈都不止 2KB,再加个向量寄存器文件就大了)。存进 P 是因为"任意时刻每个 P 上只有一个活跃的抢占上下文",P 单槽足够。

### 5.6.3 调用 asyncPreempt2,进调度器

```asm
// src/runtime/preempt_amd64.s#L123-L124
preempt:
    CALL ·asyncPreempt2(SB)
```

一行 `CALL asyncPreempt2`——注意这是 `CALL`,不是 `JMP`,所以 `asyncPreempt2` 返回后会回到下面恢复寄存器的代码。

`asyncPreempt2`([`preempt.go#L317`](../go/src/runtime/preempt.go#L317))是 Go 写的延续,它通过 `mcall` 切到系统栈,把 G 标记为 asyncSafePoint,然后:

```go
// src/runtime/preempt.go#L317-L344(节选)
//go:nosplit
//go:nowritebarrierrec
func asyncPreempt2() {
    mcall(func(gp *g) {
        gp.asyncSafePoint = true
        xRegSave(gp)                  ; 把 xRegs.scratch 挪到 G 里(GC 扫栈时不扫这块)
        if gp.preemptStop {
            preemptPark(gp)            ; GC 触发的抢占:把 G 停在 _Gpreempted
        } else {
            gopreempt_m(gp)            ; sysmon 触发的抢占:把 G 塞回 runq
        }
        // 上面两个函数都不返回
    })
    // —— 此处之下不许再增长栈(因为 asyncPreempt 帧里有 untyped 数据)
    gp := getg()
    xRegRestore(gp)                    ; 把 xRegs 从 G 挪回 P.scratch,供恢复用
    gp.asyncSafePoint = false
}
```

两个细节:

1. **`mcall` 切系统栈**:`asyncPreempt` 在 G 的用户栈上跑,栈上有一堆"无类型的"寄存器值(GC 不知道哪些是指针)。如果在这上面 `gopreempt_m` 触发栈扩容或 GC,扫栈会出错。所以 `asyncPreempt2` 第一步 `mcall` 切到 **g0 的系统栈**(系统栈不会被 GC 扫、不会扩容),在系统栈上做"改 G 状态、塞 runq"这些会触发调度的事。
2. **`gopreempt_m` vs `preemptPark`**:`gp.preemptStop` 区分两类抢占——sysmon 的"超时抢占"是 `false`,走 `gopreempt_m`(把 G 塞回 runq 等下次跑);GC STW 的"强制停"是 `true`,走 `preemptPark`(把 G 停在 `_Gpreempted` 状态,等 STW 结束再恢复)。两条路都"不返回"——`asyncPreempt2` 在 `mcall` 的回调里"消失"了。

那 `CALL asyncPreempt2` 后面的代码(恢复寄存器)什么时候执行?**当这个 G 再次被调度、`gogo` 把它从 `_Grunnable` 切回 `_Grunning` 时**。`gogo` 不是从 G 的入口开始,而是从 G 上次被切走时存的 PC 开始——`gopreempt_m` 通过 `mcall` 切走前,`mcall` 把"返回到 `asyncPreempt2` 的 `CALL` 之后"的 PC 存进了 `gp.sched.pc`。所以 G 重新被调度时,从 `xRegRestore` 那一行继续——把向量寄存器从 G 挪回 P.scratch(因为 P 可能换了,需要挪到当前 P),清 `asyncSafePoint`,然后 `asyncPreempt2` 返回,回到汇编的恢复代码。

### 5.6.4 恢复寄存器,RET 回到被打断处

```asm
// src/runtime/preempt_amd64.s#L135-L232(节选,SSE 路径)
restoreSSE:
    MOVUPS 960(AX), X15          ; 从 P.scratch 读回向量寄存器(逆序)
    ...
    MOVUPS 0(AX), X0
    JMP restoreGPs
restoreGPs:
    MOVQ 104(SP), R15            ; 读回通用寄存器(逆序:R15 先,R15 存在 104(SP))
    ...
    MOVQ 0(SP), AX
    ADJSP $-112                  ; 撤回 112 字节
    POPFQ                        ; 恢复 flags
    POPQ BP                      ; 恢复 BP
    RET                          ; ★ 弹出栈顶 = pushCall 伪造的 resumePC,跳回被打断处
```

恢复是保存的严格逆序——向量寄存器先(从 P.scratch),通用寄存器后(从栈),flags 恢复,BP 恢复,然后 **`RET`**。

这个 `RET` 弹出的栈顶是什么?**就是 `pushCall` 在 5.4.5 伪造的那个返回地址 `resumePC`(= `newpc`,被信号打断处的 PC)**。所以 `RET` 跳回 `newpc`——G 从被打断的那条指令继续执行,寄存器、flags、向量寄存器全部恢复成打断前的样子。

> **钉死这件事**:`asyncPreempt` 是个**对称的保存/恢复壳**——入口保存所有寄存器(通用进栈、向量进 P.scratch),`CALL asyncPreempt2` 进调度器;G 被重新调度后,`asyncPreempt2` 返回,汇编恢复所有寄存器,`RET` 跳回 `pushCall` 伪造的"返回地址"=被打断处。整个抢占对被打断代码完全透明——它感知到的是"中间有一段不确定长度的停顿",寄存器和栈帧纹丝不动。

## 5.7 技巧精解:三个最硬的技巧

这一章最值得单独拎出来讲的是三个技巧,它们各自对应一个"朴素地写会撞墙"的反面。

### 5.7.1 技巧一:把信号打断伪装成函数调用(pushCall)

朴素地想"用信号实现抢占",可能写成这样:

```
// 朴素思路(不是 Go 的实现)
signal_handler:
    save_registers_to_some_buffer()       ; 把寄存器存到某个全局缓冲
    call schedule()                       ; 切到调度器
    restore_registers_from_buffer()       ; 恢复
    return_from_signal()                  ; 从信号返回
```

这条路撞两堵墙:

1. **从信号返回和从调度器返回是两套语义**。信号 handler 返回要走 `sigreturn` 系统调用(内核恢复 `ucontext`),而调度器切换走 `gogo`(切栈 + 跳 PC)。两者混在一起,状态机极难理清:调度器切回这个 G 时,是接着 `schedule` 后面跑(那要先 `restore_registers` 再 `sigreturn`),还是从被打断处跑(那要跳过 `restore`)?朴素方案要么漏步骤要么重步骤。
2. **GC 扫栈时找不到这个 G 的真实栈帧**。如果寄存器存到全局缓冲,G 的栈上就没有"被抢占时的状态",GC 扫栈扫不到根(寄存器里的指针不在栈上)——漏标活对象。

`pushCall` 的妙处是**绕开这两堵墙**:它把信号打断**伪装成一次普通的 CALL 指令**——手动模拟"压返回地址 + 改 PC"。这样:

- `asyncPreempt` 就是个**普通函数**:有标准的栈帧(BP)、标准的保存/恢复、标准的 RET。调度器切回它时,走的是和切回任何普通函数一样的 `gogo` 路径,不需要特殊处理。`asyncPreempt` 的 `RET` 自然弹到伪造的返回地址,跳回被打断处——**完全复用硬件的 CALL/RET 语义,没有"信号返回"的特殊路径**。
- 寄存器存在 **G 的栈上**(通用寄存器)和 **P 的 xRegs**(向量),GC 扫栈能扫到通用寄存器(它们在栈上,栈帧有指针地图),向量寄存器在 P 上不扫(它们不含指针)——**根集合完整,不漏标**。

这个"伪装 CALL"的技巧,是 Go 异步抢占最巧妙的一手。它把一个"非对称的、跨了内核和用户态的事件(信号)"**降维成"一个普通的 Go 函数调用"**,从而复用了 runtime 已有的所有机制(栈扫描、调度切换、栈帧元数据)。

### 5.7.2 技巧二:编译器打标签,运行时只查表(PCDATA_UnsafePoint)

朴素地想"判定一条指令能不能被打断",可能写成在信号 handler 里**反汇编当前指令**、判断它是不是在原子序列里、是不是写屏障的一部分——这是个 PhD 级别的难题,而且 runtime 根本来不及在信号 handler 里做这种分析(信号 handler 要快、要确定性)。

Go 的解法是**把判定挪到编译期**:

- 编译器编译每个函数时,**已经完全知道每条指令的语义**——它知道哪些是原子序列的开头和结尾(它自己生成的写屏障代码)、哪些是 nosplit 函数、哪些是无副作用的"可重启序列"。
- 所以编译器给每条指令打一个标签(`UnsafePointSafe` / `UnsafePointUnsafe` / `UnsafePointRestart1/2` / `UnsafePointRestartAtEntry`),存进一张表(`PCDATA_UnsafePoint`),和函数的其它元数据(指针地图等)一起放进二进制。
- runtime 在 `isAsyncSafePoint` 里**只查表**:`pcdatavalue2(f, abi.PCDATA_UnsafePoint, pc)` 一行查到这条 PC 的标签,然后照章处理。

这个分工的精妙在于:

- **编译器有完整信息,运行时几乎零开销**。判定安全点不需要反汇编、不需要启发式——查一张表,O(1)。
- **sound 性由编译器保证**:编译器绝不会把"原子序列中间"标成 Safe(它自己生成的写屏障代码,它当然知道边界)。runtime 不需要"猜",只需要"信"编译器的标注。
- **可重启序列的"返回 PC"由编译器提供**:对于 `Restart1/2`,编译器在表里同时存了"序列起始 PC"(`startpc`),runtime 直接拿它当 `pushCall` 的返回地址——**回放语义也由编译器明示**。

> **反面对比**:假设 runtime 自己在 handler 里分析指令(反汇编)。第一个问题是性能——反汇编慢且复杂,handler 里这么干会让信号处理成毫秒级。第二个问题是 sound 性——runtime 怎么知道一段指令是不是"原子序列"?它得维护一份"所有原子序列的模式表",这比编译器自己标注脆弱得多(编译器升级了序列模式,runtime 得跟着改)。Go 的解法把这个复杂度彻底交给了编译器——它本来就在生成这些代码,标个标签是顺手的事。

### 5.7.3 技巧三:信号合并 + 二次确认(wantAsyncPreempt + signalPending)

朴素地想"给目标线程发抢占信号",可能写成"发现要抢占就立刻 `tgkill` 发 SIGURG"。这条路撞两堵墙:

1. **live-lock**:多条线程同时想抢占同一个 M(比如 sysmon 发一次、GC STW 发一次、还有别的 P 因为 work-stealing 触发 `preemptone`),都给同一个 M 发 SIGURG。那个 M 被信号反复打断,几乎跑不动 `asyncPreempt`——这就是 Darwin 上真实出现过的 issue #37741。
2. **陈旧请求**:信号在飞的过程中,目标 G 可能已经让出了(自己阻塞、被别的机制抢占)。信号到达时 handler 如果不现查一次"还想被抢吗",就会对一个"已经不需要被抢"的 G 执行抢占——错误地把它从 `_Grunning` 拉下来,状态机错乱。

Go 的解法是**两道闸**:

- **`mp.signalPending.CompareAndSwap(0, 1)`**:[`preemptM`](../go/src/runtime/signal_unix.go#L376) 里这个单槽 CAS——只有"没有信号在飞"时才发。发出去后 `signalPending = 1`,handler 处理完清零(5.4.4 的 `gp.m.signalPending.Store(0)`)。这样**同一时刻最多一个信号在飞**,堵掉 live-lock。
- **`wantAsyncPreempt(gp)`** 二次确认:handler 收到信号后,**不直接抢**,而是现查"`gp.preempt` 或 `pp.preempt` 还在吗?G 还在 `_Grunning` 吗?"。如果请求已被取消(标志清了)或 G 已不在运行态,信号被静默确认接收(`preemptGen` 自增、`signalPending` 清零),然后返回——**什么也不做**。堵掉陈旧请求。

这两道闸合起来,把异步抢占变成一个**幂等、可合并、不会误执行**的协议。即使信号偶尔被"浪费"(发出去时还想要,到达时不想要了),也只是多一次 `preemptGen` 自增,不会有正确性问题。

## 章末小结

这一章把 Go 的抢占机制——从协作式到异步式——拆透了。回到二分法,这一章服务**调度执行**:它讲清了"如何强制一个不让出的 G 让出 CPU",这是 work-stealing(上一章)和并发 GC(第 4 篇)能成立的前提。没有抢占,死循环 G 会饿死本地 runq、会挡住 GC 的 STW;有了异步抢占,sysmon 在 10ms 后投一个 `SIGURG`,几乎任何指令处都能把 G 塞回 runq。

### 五个"为什么"清单

1. **为什么 Go 一定要有抢占?** 调度器(让本地 runq 里别的 G 不被饿死)和并发 GC(STW 要所有 G 进安全点)都依赖 G 会让出 CPU。一个不让出的 G(死循环、紧密计算)会同时破坏这两件事——饿死调度、拖死 GC。抢占是基础,不是优化。

2. **协作式抢占为什么搞不定死循环?** 它把抢占请求编码成"`stackguard0 = stackPreempt`",而 `stackguard0` 只在**函数前奏的栈检查**里被看。一个紧凑循环(不调函数)永远走不到前奏,永远看不到这个请求。协作式抢占只能搞定"调函数的 G",够不到"不调函数的 G"。

3. **为什么用 `SIGURG` 而不是别的信号?** SIGURG 的默认动作是忽略(没 handler 不会杀进程)、应用几乎不用它的"真语义"(TCP 带外数据基本没人用,且 SIGURG 不告诉你哪个 socket,语义无用)、即使应用用了也得准备接受虚假 SIGURG。相比之下 SIGUSR1/2 太常被应用占用,SIGALRM/SIGPROF 有专用,SIGIO 更可能被真用。SIGURG 是个"几乎没人要"的信号,适合 runtime 独占。

4. **为什么 `asyncPreempt` 要保存所有寄存器(包括向量)?** 因为信号打断的"任意指令"可能正在用任何寄存器。如果只保存通用寄存器、不保存向量,G 被调度回来后向量寄存器被 `asyncPreempt2` 调度的别的 G 污染了,被抢占代码看到的 XMM 值就错了。所以必须**对称地保存所有可能被使用的寄存器**,恢复时原样还原——对被打断代码完全透明。

5. **`isAsyncSafePoint` 凭什么 sound?** 六道检查筑起三道防线:**栈和 GC 一致性**(有 func 元数据、非汇编、非 runtime——保证 GC 能扫到所有根)、**原子性**(`PCDATA_UnsafePoint` 标签堵住原子序列中间——不破坏写屏障/CAS)、**可恢复性**(Restart 序列要么可重跑要么从头跑——`pushCall` 用 `startpc` 作返回地址)。安全点不是 runtime 判的,是编译器标好的——runtime 只查表。

### 引出下一章

讲透异步抢占之后,我们要处理另一个"让 G 让出 CPU"的场景——**系统调用阻塞**。一个 G 调了一个阻塞的系统调用(比如读文件、`futex`),它不像 channel/sleep 那样被 runtime park(那些走 netpoll 或 timer),也不像死循环那样靠抢占——它**真的卡在内核里**,信号打断它没意义(它在 syscall 里,`isAsyncSafePoint` 第 4 道 `canPreemptM` 就排除了它)。怎么办?Go 的做法是 **handoff P**:这个 G 卡在 syscall 里没关系,把它占的 P 让给别的 M 去跑别的 G(`entersyscall` 释放 P,`handoffp` 转交,`exitsyscall` 抢回 P)。下一章我们钻进系统调用阻塞的 handoff P 机制。
