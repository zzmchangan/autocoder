# 第十二章 · 逃逸分析:堆还是栈

> 篇:第 3 篇 · 内存分配
> 主线呼应:第 11 章把 `mallocgc` 切成了三条道(tiny / small / large),但那个分析留下了一个最根本的问题没回答——**到底什么变量才需要走 `mallocgc`?** 一个看似局部的 `var x int`、一个 `&y`、一个 `make([]byte, 8)`,有的进了堆(走 `mallocgc`、被 GC 追踪、被写屏障盯防),有的却一辈子待在栈上(GC 完全看不见)。决定权不在 runtime,而在**编译器**——具体说,在 `src/cmd/compile/internal/escape/` 这一个包里跑的逃逸分析(escape analysis)。本章离开 runtime,走进编译器,看 Go 怎么用一张赋值流图 + 一个 Bellman-Ford 求最短路的算法,把每个变量钉死在"堆"或"栈"上,以及为什么栈分配的对象能彻底免 GC。

## 核心问题

**编译器凭什么决定一个变量分配在堆上还是栈上?逃逸分析在这张"赋值流图"上具体做了什么?栈分配的对象凭什么能让 GC 彻底忽略它?**

读完本章你会明白:

1. 逃逸分析要保证的两条铁律:**指针不能从栈逃到堆**,且**指向栈的指针不能活过它所属的栈帧**(否则栈帧回收后就是野指针)。
2. Go 的逃逸分析怎么把整个程序画成一张有向加权图(location + hole + edge),用 `derefs` 权重编码 `&`/`*` 计数,再用 Bellman-Ford 求最短路径判定逃逸。
3. 触发逃逸的几条经典规则:返回指针、存进接口、闭包捕获、过大、`go`/`defer` 参数,以及它们在源码里各自对应哪一处。
4. 栈分配为什么"免 GC":栈帧在函数返回时整片回收,不需要扫描;goroutine 栈的增长靠指针调整而非 GC 协助。
5. `paramTag` 的参数标签(param tags)如何让跨函数分析"只要看函数签名就能复用",这是逃逸分析能在编译期跑得起的关键。

> 逃生阀:本章会出现 `location`、`hole`、`derefs`、`loopDepth`、`paramEsc` 这些逃逸分析内部的抽象。看不懂某个细节时,回到这一句:整个分析本质上是回答一个问题——**"有没有一条赋值路径,让这个变量的地址流到了一个比它活得久的地方?"** 有,就堆;没有,就栈。其余全是实现工程。

## 12.1 一句话点破

> **逃逸分析把每个变量画成图上的一个点,把每条赋值画成带权边,然后问:从这变量出发,有没有一条路把它的地址送进堆、送进返回值、送进闭包、送进接口——也就是送进一个"比它活得久"的地方?有就标 `EscHeap` 进堆,没有就标 `EscNone` 留栈;栈上的东西 GC 根本不扫,因为函数一返回整片栈帧就没了,不需要回收。**

这是结论,不是理由。本章倒过来拆:先看逃逸分析到底在防什么(两条铁律),再看它怎么把整个程序变成图、怎么在图上求最短路,最后看几条经典逃逸规则各自落在哪一行代码,以及栈分配为什么能让 GC 真正"省心"。

## 12.2 先卸包袱:逃逸分析要防什么

读逃逸分析源码前,必须先想清楚它**为什么存在**。它的全部使命是保证下面这两条不变式,任何一条被破坏,程序都会崩:

**铁律一:指向栈对象的指针,不能被存进堆。**

```go
var p *int
func leak() {
    x := 42
    p = &x     // x 的地址被存进全局变量 p(全局在堆/BSS)
}
// leak 返回后,x 所在的栈帧被回收;此时 *p 是野指针
```

如果 `x` 留在栈上,`p = &x` 这条赋值一旦执行,`p` 就持有了栈帧内的一块内存;`leak` 一返回,那块栈帧可能马上被下一个函数调用复用,`*p` 读出来的是别人的局部变量——典型的 use-after-free。

**铁律二:指向栈对象的指针,不能活过该对象。**

```go
func rangeLeak() {
    var saved *int
    for i := 0; i < 10; i++ {
        var j = i          // j 在每次循环迭代复用同一块栈槽
        if i == 5 {
            saved = &j     // saved 持有 j 的栈地址
        }
    }
    println(*saved)        // j 早被覆盖,*saved 是错的值
}
```

这是逃逸分析源码注释([escape.go:26-31](../go/src/cmd/compile/internal/escape/escape.go#L26-L31))开门见山讲的两条:

```go
// The two key invariants we have to ensure are: (1) pointers to stack objects
// cannot be stored in the heap, and (2) pointers to a stack object
// cannot outlive that object (e.g., because the declaring function
// returned and destroyed the object's stack frame, or its space is
// reused across loop iterations for logically distinct variables).
```

> **不这样会怎样**:如果编译器不分析、把所有变量无脑留栈,上面两个例子就会在运行时崩——野指针、读到错值、栈被栈增长拷贝后指针失效。这些 bug 不可复现、不可调试,是语言实现里最阴险的一类。
>
> **所以这样设计**:编译器在生成代码前,先做一遍静态数据流分析,**只要它不能证明一个变量的地址"老老实实待在自己栈帧的寿命内",就把它挪到堆上**。这是"宁可错杀(多进堆),不可错放(留栈后崩溃)"的保守策略。逃逸分析的全部精巧,都是为了让"错杀"尽可能少,从而让更多对象留在栈上、省 GC。

这就是逃逸分析存在的理由。下一节看它怎么把这件事做到。

## 12.3 把程序画成图:location、hole、derefs

逃逸分析的核心数据结构是三件套,记牢了后面源码全是它们:

| 抽象 | 类型 | 含义 |
|---|---|---|
| **location** | `struct` | 图的顶点。每个变量声明、每个 `new`/`make`/复合字面量都被映成一个 location([graph.go:43](../go/src/cmd/compile/internal/escape/graph.go#L43))。 |
| **hole** | `struct` | "值要流向哪里"的上下文。比如评估 `p` in `x = **p` 时,hole 是 `{dst: x, derefs: 2}`([graph.go:146-158](../go/src/cmd/compile/internal/escape/graph.go#L146-L158))。 |
| **edge** | `struct` | 图的边,连接两个 location,带 `derefs` 权重([graph.go:113-118](../go/src/cmd/compile/internal/escape/graph.go#L113-L118))。 |

最关键的概念是 **`derefs` 权重**。源码注释([escape.go:54-69](../go/src/cmd/compile/internal/escape/escape.go#L54-L69))用四个例子钉死它:

```
//     p = &q    // derefs = -1  (取地址:权重 -1)
//     p = q     // derefs =  0  (直接赋值)
//     p = *q    // derefs = +1  (解引用一次)
//     p = **q   // derefs = +2  (解引用两次)
//
//     p = **&**&q  // = 2  (& 和 * 抵消)
```

`derefs` 把"& 多一层 / * 多一层"这件事编码成一个整数。**当一条路径上 `derefs` 累加到负数,就说明这条路径在传递某个东西的地址**(weight -1 一次 + 取地址再 -1)。地址沿这条路径流到终点,而终点如果是堆 / 返回值 / 闭包——即"比源 location 活得久",源 location 就得逃逸。

具体怎么"流",看一个最朴素的赋值 `p = &q` 在逃逸分析里的样子([assign.go](../go/src/cmd/compile/internal/escape/assign.go)、[expr.go](../go/src/cmd/compile/internal/escape/expr.go)):

```go
// expr.go —— OADDR 分支(L62-L64)
case ir.OADDR:
    n := n.(*ir.AddrExpr)
    e.expr(k.addr(n, "address-of"), n.X)
```

`k.addr` 就是 `k.shift(-1)`([graph.go:190](../go/src/cmd/compile/internal/escape/graph.go#L190)),把 hole 的 `derefs` 减 1。于是在评估 `&q` 时,传给 `q` 的 hole 比传给 `&q` 的 hole 多了一个 `-1`。最终 `flow(k, q.location)` 把这条带 `derefs = 原 k.derefs - 1` 的边写进图。

再看一个稍微绕的——把值赋给一个全局变量(`p = &x`,p 是全局):

- `p` 是 `PEXTERN`(包级全局),`addr` 对 `PEXTERN` 直接返回 `heapHole()`([assign.go:27-30](../go/src/cmd/compile/internal/escape/assign.go#L27-L30))。这是关键:全局变量在逃逸分析里被建模成"等价于堆"。
- 评估 `&x` 时,`k.addr` 把 hole 的 derefs 减 1,所以传到 `x` 的 hole 是 `heapHole()`(derefs = -1)。
- `flow(heapHole_addr, x.loc)` 走进 [graph.go:199](../go/src/cmd/compile/internal/escape/graph.go#L199),发现 `dst.hasAttr(attrEscapes) && derefs < 0`,直接把 `x.loc` 标成 `attrEscapes`([graph.go:211-226](../go/src/cmd/compile/internal/escape/graph.go#L211-L226))——`x` 逃逸。

整个机制就这一个核心。下面把"怎么判定逃逸"这条算法的源码拆透。

## 12.4 判定算法:walkAll + Bellman-Ford 求最短 derefs

图建好之后,逃逸分析要做的是:对图上每个 location,求出**从它出发到堆 / 返回值 / 闭包等"会逃逸的终点"的最小 `derefs`**。如果这条最小路径的 `derefs` 是负数,就说明它的地址流到了终点——逃逸。

这套"在有负权边的图上求最短路"的经典算法是 **Bellman-Ford**。Go 的实现在 [solve.go:70](../go/src/cmd/compile/internal/escape/solve.go#L70) 的 `walkOne`:

```go
// solve.go —— walkOne(简化,非源码原文,保留关键逻辑)
func (b *batch) walkOne(root *location, walkgen uint32, enqueue func(*location)) {
    // 数据流图有负边(取地址),所以用 Bellman-Ford;
    // 但不用担心无穷负环,因为中间 derefs 下界为 0。
    root.derefs = 0
    todo := newQueue(1); todo.pushFront(root)

    for todo.len() > 0 {
        l := todo.popFront()
        derefs := l.derefs

        addressOf := derefs < 0
        if addressOf {
            // 如果 l 的地址流到了一个比它活得久的 root,l 必须逃逸
            derefs = 0  // 中间下界为 0
            if b.outlives(root, l) {
                // 给 l 打上 attrEscapes,并把它重新入队(让逃逸传播)
                l.attrs |= attrEscapes | attrPersists | attrMutates | attrCalls
            } else if root.hasAttr(attrPersists) {
                l.attrs |= attrPersists
            }
        }

        // 如果 l 是参数,记录它流到堆/返回值的 derefs(用于参数标签)
        if l.param {
            if b.outlives(root, l) { l.leakTo(root, derefs) }
            ...
        }

        // 沿 l 的入边往回传播(Bellman-Ford 松弛)
        for _, edge := range l.edges {
            d := derefs + edge.derefs   // 累加权重
            if edge.src.walkgen != walkgen || edge.src.derefs > d {
                edge.src.derefs = d
                edge.src.dst = l
                todo.pushBack(edge.src) // 松弛成功,重新入队
            }
        }
    }
}
```

几个细节必须讲清:

### (1) 为什么是 Bellman-Ford,不是 Dijkstra

取地址操作 `&x` 产生 `derefs = -1` 的负权边。Dijkstra 在有负权边时是错的(贪心选最短就锁死)。Bellman-Ford 专门处理负权边,靠反复松弛直到收敛。

注释([solve.go:70-74](../go/src/cmd/compile/internal/escape/solve.go#L70-L74))还点出一个关键:"我们不用担心无穷负环,因为中间 `derefs` 下界为 0"——意思是每次松弛时把 `derefs` 钳到 ≥ 0(`if addressOf { derefs = 0 }`),破坏了负环的累积。这让算法在多项式时间内终止。

### (2) `outlives`:判定"活得久"的全部规则

`outlives(root, l)`([solve.go:282-335](../go/src/cmd/compile/internal/escape/solve.go#L282-L335))是逃逸判定的核心谓词,它编码了"一个 location 的地址能不能流到另一个 location"的全部情形。源码里逐条列了:

```go
// solve.go —— outlives(L282-L335,简化)
func (b *batch) outlives(l, other *location) bool {
    if l.hasAttr(attrEscapes) { return true }       // (a) 堆:活过一切

    if l == &b.mutatorLoc || l == &b.calleeLoc { return false } // 伪 location

    if l.paramOut {                                  // (b) 返回值
        // 例外:闭包可以返回外层局部而不逃逸,如果能静态识别所有调用点
        if ir.ContainsClosure(other.curfn, l.curfn) && !l.curfn.ClosureResultsLost() {
            return false
        }
        return true
    }

    if l.curfn == other.curfn && l.loopDepth < other.loopDepth {
        return true                                  // (c) 同函数,l 声明在更外层循环
    }

    if ir.ContainsClosure(l.curfn, other.curfn) {
        return true                                  // (d) l 在外层闭包,other 在内层
    }
    return false
}
```

这五条覆盖了所有逃逸场景:

- **(a) 堆**:地址流进堆,堆活到 GC 回收它,比任何栈帧都久 → 逃逸。
- **(b) 返回值**:地址流进返回值,调用方拿到指针时本函数可能已返回 → 逃逸(除非是确定的内联闭包)。
- **(c) 循环变量地址外溢**:`for { l = new(int) }`,l 在循环外声明、`new(int)` 在循环内,后者地址活过单次迭代 → 逃逸。这一条靠 `loopDepth`(循环深度)判断。
- **(d) 闭包层级**:`var l *int; func() { l = new(int) }()`,内层闭包里 new 的对象地址被外层 location 持有 → 逃逸。

> **钉死这件事**:逃逸判定不是一条规则,而是 `outlives` 这一个函数。所有看似五花八门的"什么情况下逃逸",都收敛到这一句——"地址是否流到一个比源 location 活得久的目标"。读逃逸分析,本质是读这一个函数。

### (3) `walkAll`:从每个 location 各做一次 BFS

`walkAll`([solve.go:19-66](../go/src/cmd/compile/internal/escape/solve.go#L19-L66))对图上每个 location(包括堆、mutator、callee 三个伪 location)各调一次 `walkOne`。注释([solve.go:24-27](../go/src/cmd/compile/internal/escape/solve.go#L24-L27))算了一笔账:每个 location 最多经历 `!persists → persists` 和 `!escapes → escapes` 两次状态迁移,所以总走路次数是 Θ(顶点数)。

一个优化:`enqueue` 把已标 `attrEscapes` 的 location 从队头入([solve.go:36-42](../go/src/cmd/compile/internal/escape/solve.go#L36-L42)),让逃逸状态优先传播,加快收敛。

## 12.5 几条经典逃逸规则:各自落在哪行代码

`outlives` 是判定的最终出口,但触发判定的"赋值流"是从 AST walk 出来的。下面挑读者最关心的几条规则,看它们各自在源码里怎么被建立。

### 规则一:返回指针 → 逃逸

```go
func New() *int { x := 1; return &x }   // x 逃逸
```

这条来自返回值是 `paramOut`,在 `outlives` (b) 里直接返回 true。赋值流是通过 `tagHole` 建立的——被调函数的参数 / 返回值标签(param tag)把实参和返回值的逃逸信息串起来。看 [call.go:388](../go/src/cmd/compile/internal/escape/call.go#L388):

```go
// call.go —— tagHole(L388-L425,简化)
func (e *escape) tagHole(ks []hole, fn *ir.Name, param *types.Field) hole {
    if fn == nil { return e.heapHole() }   // 动态调用:保守,全堆
    ...
    esc := parseLeaks(param.Note)          // 解析参数的逃逸标签
    if x := esc.Heap(); x >= 0 {
        tagKs = append(tagKs, e.heapHole().shift(x))   // 流到堆
    }
    for i := 0; i < numEscResults; i++ {
        if x := esc.Result(i); x >= 0 {
            tagKs = append(tagKs, ks[i].shift(x))      // 流到第 i 个返回值
        }
    }
    return e.teeHole(tagKs...)
}
```

`param.Note` 就是函数签名里那个逃逸标签(下面 12.7 专门讲)。它告诉调用方:"我这个参数会流到堆 / 第 i 个返回值"。调用方据此把实参的 hole 接到正确的终点。**返回指针的本质**是返回值 `paramOut` 在 `outlives` 里判 true,所以 `&x` 沿 `tagHole` 流到返回值 → x 逃逸。

### 规则二:存进接口 → 逃逸

```go
var w io.Writer = &buf{...}   // &buf 逃逸(典型)
```

接口装箱是逃逸分析里最常被吐槽的"暗坑"。源码在 [expr.go:116](../go/src/cmd/compile/internal/escape/expr.go#L116):

```go
// expr.go —— OCONVIFACE(L116-L121)
case ir.OCONVIFACE:
    n := n.(*ir.ConvExpr)
    if !n.X.Type().IsInterface() && !types.IsDirectIface(n.X.Type()) {
        k = e.spill(k, n)              // 新建一个 location,地址流向 k
    }
    e.expr(k.note(n, "interface-converted"), n.X)
```

`spill`([expr.go:337](../go/src/cmd/compile/internal/escape/expr.go#L337))的关键是 `e.flow(k.addr(n, "spill"), loc)`——它给这个新 location 接了一条 `addr`(derefs -1)的边到 k。如果 k 本身是 `heapHole`(典型:接口值存进一个会逃逸的地方),则 `flow` 里 `dst.hasAttr(attrEscapes) && derefs < 0` 直接把这个 location 标 `attrEscapes`。

> **不这样会怎样**:Go 的 interface 是 `{itab, data}` 两字结构。`data` 是一个指针,它必须指向一个**地址稳定**的对象。如果对象在栈上,goroutine 栈增长时会把栈整片拷贝、对象地址变了,`data` 就成了野指针。所以装进 interface 的对象,要么是"直接 interface"(本身已经是指针类型,`IsDirectIface` 为真,不需要额外分配),要么必须进堆,保证地址稳定。这就是 `OCONVIFACE` 逃逸的物理原因。

注意一个例外:`rewriteWithLiterals`([escape.go:609-636](../go/src/cmd/compile/internal/escape/escape.go#L609-L636))会尝试把 `interface{常量}` 的装箱省掉——如果装箱的值是编译期常量,就不分配。这是 Go 1.2X 加的优化,见 12.7。

### 规则三:闭包捕获 → 可能逃逸

```go
func counter() func() int { n := 0; return func() int { n++; return n } }  // n 逃逸
```

闭包捕获在 [escape.go:254-286](../go/src/cmd/compile/internal/escape/escape.go#L254-L286) 的 `flowClosure`:

```go
// escape.go —— flowClosure(L254-L286,简化)
func (b *batch) flowClosure(k hole, clo *ir.ClosureExpr) {
    for _, cv := range clo.Func.ClosureVars {
        loc := b.oldLoc(cv)
        // 捕获方式:≤ 128 字节且不被重赋值 → byval(值拷贝);否则 byref(引用)
        n.SetByval(!loc.addrtaken && !loc.reassigned && n.Type().Size() <= 128)
        if !n.Byval() {
            n.SetAddrtaken(true)
            k = k.addr(cv, "reference")   // 引用捕获:地址流向闭包
        }
        b.flow(k.note(cv, "captured by a closure"), loc)
    }
}
```

闭包捕获分两种:

- **byval(值捕获)**:`n` 的副本塞进闭包记录,原变量不逃逸(只要 ≤ 128 字节且没被重赋值)。这是 Go 的一个优化——`n` 不必逃逸。
- **byref(引用捕获)**:把 `n` 的地址塞进闭包记录。由于闭包记录本身可能逃逸(被返回、被存全局),`n` 也跟着逃逸。

> **为什么是 128 字节**:[escape.go:263](../go/src/cmd/compile/internal/escape/escape.go#L263) 那行 `n.Type().Size() <= 128`。阈值 128 是经验值——大于 128 字节的值拷贝开销太大,不如直接引用。这和栈上 small object 上限是不同的概念,别混淆。

### 规则四:过大 → 逃逸

```go
var big [10 * 1024 * 1024]byte   // 10 MB,逃逸(超过栈上限)
```

这条规则不在赋值流里,而是在 `Batch` 的最后阶段,对每个 location 单独检查。源码是 [utils.go:171](../go/src/cmd/compile/internal/escape/utils.go#L171) 的 `HeapAllocReason`:

```go
// utils.go —— HeapAllocReason(L173-L243,简化)
func HeapAllocReason(n ir.Node) string {
    if n.Op() == ir.ONAME {
        n := n.(*ir.Name)
        if n.Class == ir.PPARAM || n.Class == ir.PPARAMOUT {
            return ""          // 参数永远走栈
        }
    }
    if n.Type().Size() > ir.MaxStackVarSize {
        return "too large for stack"          // 显式变量超过 128 KB
    }
    if n.Type().Alignment() > int64(types.PtrSize) {
        return "too aligned for stack"
    }
    if (n.Op() == ir.ONEW || n.Op() == ir.OPTRLIT) && n.Type().Elem().Size() > ir.MaxImplicitStackVarSize {
        return "too large for stack"          // new(T) 的 T 超过 64 KB
    }
    if n.Op() == ir.OMAKESLICE {
        ...
        if !ir.IsSmallIntConst(r) {
            return ""          // 非常量大小:走混合分配(见下)
        }
        if ir.Int64Val(r) > ir.MaxImplicitStackVarSize/elem.Size() {
            return "too large for stack"      // makeslice 常量 cap 太大
        }
    }
    return ""
}
```

两个阈值([ir/cfg.go:8-19](../go/src/cmd/compile/internal/ir/cfg.go#L8-L19)):

| 常量 | 值 | 含义 |
|---|---|---|
| `MaxStackVarSize` | `128 * 1024`(128 KB) | 显式变量(局部 var)的栈上限 |
| `MaxImplicitStackVarSize` | `64 * 1024`(64 KB) | 隐式变量(`new`、`makeslice`、闭包等)的栈上限 |

> **不这样会怎样**:goroutine 栈初始只有 2 KB(可增长)。一个 128 MB 的局部数组如果留栈,要么撑爆初始栈(每次调用都触发栈增长拷贝),要么直接 OOM。把它们踢到堆,栈保持小而快,是工程上必须的取舍。

`HeapAllocReason` 返回非空时,[escape.go:174-176](../go/src/cmd/compile/internal/escape/escape.go#L174-L176) 把这个 location 强制流向堆:

```go
if why := HeapAllocReason(loc.n); why != "" {
    b.flow(b.heapHole().addr(loc.n, why), loc)
}
```

### 规则五:`go` / `defer` 参数 → 通常逃逸

```go
go slowFunc(x)   // x 的副本逃逸(go 启的 goroutine 生命周期可能超过当前函数)
defer cleanup(x) // 顶层 defer 的 x 不逃逸(见下)
```

这条在 [call.go:236-268](../go/src/cmd/compile/internal/escape/call.go#L236-L268) 的 `goDeferStmt`:

```go
// call.go —— goDeferStmt(L236-L268,简化)
func (e *escape) goDeferStmt(n *ir.GoDeferStmt) {
    k := e.heapHole()                       // 默认:全堆
    if n.Op() == ir.ODEFER && e.loopDepth == 1 && n.DeferAt == nil {
        // 顶层 defer:参数不逃逸到堆,但要活到 defer 执行
        k = e.later(e.discardHole())
        n.SetEsc(ir.EscNever)              // 强制栈分配 defer 记录
    }
    ...
    e.expr(k, call.Fun)
}
```

注意一个**反直觉的优化**:**顶层 defer**(在函数最外层、不在循环里、没有 `DeferAt`)的参数不逃逸——因为 defer 记录本身被强制栈分配(`EscNever`),参数只要活到 defer 执行即可。但**循环里的 defer** 或 **go** 语句,默认全堆,因为它们的生命周期无法在编译期约束。

> **钉死这件事**:这是为什么"`go func(){...}()` 比直接调用慢"的一部分原因——参数和闭包记录都要堆分配。第 2 篇讲 goroutine 创建时会再回到这里,看 `newproc` 怎么吃掉这些堆分配。

### 规则六:`make([]T, n)` 混合栈分配

这是一条 Go 1.2X 加的、很容易被老资料漏掉的规则。当 `makeslice` 的长度是**非常量**时,`HeapAllocReason` 返回空(不强制堆),`walk/builtin.go` 会在 SSA 阶段生成一段**运行时分支**:

```go
// 注释里的伪代码(utils.go:222-235)
// if cap <= K {
//     var backing [K]E          // 栈上固定大小数组
//     s = backing[:len:cap]
// } else {
//     s = makeslice(E, len, cap)  // 堆分配
// }
```

K 来自 `base.Debug.VariableMakeThreshold`([walk/builtin.go:485-486](../go/src/cmd/compile/internal/walk/builtin.go#L485-L486))。这样栈上预留固定空间,小切片不进堆,大切片才进堆。代价是栈占用恒定(无论 cap 多大)。

> **为什么 sound**:逃逸分析在 `OMAKESLICE` 的 `spill` 已经把切片元素流到了 heap([expr.go:150-154](../go/src/cmd/compile/internal/escape/expr.go#L150-L154) 的 backing store spill)。所以即使 backing 数组在栈上,任何写进切片元素的指针仍被正确跟踪——"禁止堆→栈指针"这条铁律不破坏。这是逃逸分析和 SSA walk 协作的范例。

## 12.6 栈分配为什么"免 GC"

逃逸分析把不逃逸的变量留栈,这些变量在 GC 眼里完全不存在。为什么?这要回到 GC 的根集合(root set)和栈的关系。

GC 扫描的根,主要是两类:**全局变量**和**goroutine 栈**。注意——**栈本身是 GC 的根,不是 GC 的回收对象**。GC 扫描栈,是为了找出栈上持有的那些**指向堆**的指针,去标记堆对象;它从不"回收"栈上的对象,因为栈的回收是函数返回时由栈帧机制完成的。

下面这张表把堆对象和栈对象的命运对比清楚:

| 维度 | 堆对象(逃逸) | 栈对象(不逃逸) |
|---|---|---|
| 分配 | `mallocgc` 走 size class / 大对象路径 | 函数栈帧预留槽位(编译期分配) |
| GC 扫描 | 被三色标记追踪,被写屏障盯防 | 不被追踪(它是 GC 根的载体,不是被回收物) |
| GC 回收 | sweep 阶段清扫 | 函数返回时整片栈帧弹出 |
| 写屏障 | 标记期间写它要触发写屏障 | 无(栈上指针更新不走堆写屏障) |
| 内存剖析 | `pprof` 能采到 | 采不到(栈分配无 `mallocgc` 调用) |
| 栈增长 | 无关(它在堆,地址稳定) | 栈拷贝时地址变化,引用它的指针要被遍历调整 |

最后一行值得展开:**栈对象在 goroutine 栈增长时地址会变**。一个 goroutine 的栈从 2 KB 增长到 4 KB,要把整片旧栈拷贝到新栈,然后**遍历栈上的指针,把指向旧栈的指针改指向新栈**。这是第 17 章(可增长栈)的主线。逃逸分析在这里起的作用是:**保证栈上不会有指向别的 goroutine 栈的指针**(那种指针无法在拷贝时被正确调整),所以栈拷贝只需要扫自己栈帧上的指针。

> **钉死这件事**:栈分配"免 GC"有两层含义——(1) 栈对象本身不进 GC 的标记 / sweep 循环;(2) 但栈是 GC 的根,GC 要扫栈找堆指针。这两件事不矛盾:GC 扫栈是为了标堆,不是为了回收栈。逃逸分析的价值就是让"标堆"的工作量尽可能小——栈对象越多,堆对象越少,GC 越快。

## 12.7 技巧精解:param tag——让跨函数分析摊销到签名上

逃逸分析最硬核的工程技巧是 **参数标签(param tags)**。它解决的问题是:**逃逸分析要不要把每个函数的内部都展开?** 如果每次调用都展开,一个递归调用链会指数爆炸,编译期跑不完。

Go 的解法是把每个函数的参数逃逸信息**摘要成一个二进制串**,存进导出数据;调用方只需读这个串,就能知道实参流向,不必展开被调函数体。这套机制的源码在 [leaks.go](../go/src/cmd/compile/internal/escape/leaks.go) 和 [escape.go:420](../go/src/cmd/compile/internal/escape/escape.go#L420) 的 `paramTag`。

### `leaks` 结构:8 字节编码 4 种流向

```go
// leaks.go —— L13-L25
type leaks [8]uint8

const (
    leakHeap = iota     // 流到堆
    leakMutator          // 流到间接赋值的指针操作数
    leakCallee           // 流到调用表达式的被调者操作数
    leakResult0          // 流到第 0 个返回值(后面还有 leakResult1...)
)
```

每个 `leaks` 是 8 字节数组,前 4 个槽分别记录"流到堆 / mutator / callee / 第 i 个返回值"的**最小 derefs**。`-1` 表示没流过去(`get` 返回 `l[i]-1`,0 编码成 -1,见 [leaks.go:61](../go/src/cmd/compile/internal/escape/leaks.go#L61))。

举例:`func F(x *int) *int { return x }`,参数 `x` 流到返回值 0,derefs = 0。它的 leaks 编码后,`leakResult0 = 0`(`Result(0)` 返回 0),其余为 -1。导出时这个串变成 `esc:` 前缀的二进制串([leaks.go:97-114](../go/src/cmd/compile/internal/escape/leaks.go#L97-L114))。

调用方拿到这个串,在 `tagHole`([call.go:404-424](../go/src/cmd/compile/internal/escape/call.go#L404-L424))里反序列化,把实参的 hole 接到正确的终点:

```go
// call.go —— tagHole 解析参数标签(L404-L424,简化)
esc := parseLeaks(param.Note)
if x := esc.Heap(); x >= 0 {
    tagKs = append(tagKs, e.heapHole().shift(x))     // x 流到堆
}
for i := 0; i < numEscResults; i++ {
    if x := esc.Result(i); x >= 0 {
        tagKs = append(tagKs, ks[i].shift(x))        // x 流到第 i 个返回值
    }
}
```

`shift(x)` 是把 derefs 加 x。所以 `esc.Result(0) = 0` 意味着实参以 `derefs = 0` 流到返回值 0——即实参直接成了返回值的一部分。

### 三个典型标签

`paramTag`([escape.go:420-509](../go/src/cmd/compile/internal/escape/escape.go#L420-L509))生成标签时,会调 `reportLeaks` 打印诊断(`-m` flag),常见三种:

1. **`x does not escape`**:参数完全不逃逸。`leaks` 全 -1。调用方传栈对象进来,栈对象不逃逸。
2. **`leaking param: x`**:参数流向堆(`Heap() == 0`)。调用方必须保证实参在堆上。
3. **`leaking param: x to result r level=0`**:参数流向第 r 个返回值。调用方据此递归分析返回值的逃逸。

这就是为什么 `io.Reader.Read(p []byte)` 的 `p` 标 `does not escape`——只要 Read 不把 p 存进堆,你的栈上 buffer 就安全。

> **反面对比**:如果没有参数标签,逃逸分析要么(a)展开每个被调函数,编译时间爆炸;要么(b)保守地假设所有参数都逃逸(全堆),失去逃逸分析的意义。param tag 是"摘要 + 复用"的范例,把跨函数分析的代价摊销到一次签名计算上,后续调用零成本。

### 外部函数的特殊处理

没有函数体的函数(汇编实现、cgo)无法分析,`paramTag` 走 [escape.go:433-471](../go/src/cmd/compile/internal/escape/escape.go#L433-L471) 这条分支:

```go
if len(fn.Body) == 0 {
    // 外部函数:假设不安全,除非 //go:noescape
    if fn.Pragma&ir.Noescape != 0 {
        esc.AddMutator(0); esc.AddCallee(0)   // noescape:不逃逸
    } else {
        esc.AddHeap(0)                          // 默认:逃逸到堆
    }
    ...
}
```

这就是 `//go:noescape` pragma 的作用——告诉编译器"这个汇编/cgo 函数不会让参数逃逸"。runtime 里大量汇编函数(memmove、字符串操作)都标了它,否则它们的栈上实参全会被踢进堆。注释([escape.go:436-440](../go/src/cmd/compile/internal/escape/escape.go#L436-L440))还点出一个有趣的复用:`uintptr` 参数被假定为需要跨调用保活(`fn.Pragma |= ir.UintptrKeepAlive`),这是 `syscall.Syscall` 把指针转 uintptr 调用时不丢指针的关键。

## 12.8 一个例子从头跑到尾

把前面所有机制串起来。看这段代码:

```go
1  func greet(name string, w io.Writer) {
2      buf := []byte("Hello, ")
3      buf = append(buf, name...)
4      w.Write(buf)              // io.Writer 接口
5  }
```

逃逸分析对每一行做什么:

- **L1 `name string`**:`name` 是参数,`paramTag` 看它流向——`name` 只在 L3 被读,不存进堆,标 `does not escape`。
- **L2 `buf := []byte("Hello, ")`**:字面量切片,`OSLICELIT` 走 `spill`([expr.go:200-209](../go/src/cmd/compile/internal/escape/expr.go#L200-L209))。要看 buf 流向哪里。
- **L3 `buf = append(buf, name...)`**:`append` 是个特殊调用([call.go:146-180](../go/src/cmd/compile/internal/escape/call.go#L146-L180))。它建模为"可能分配新 backing store,其地址流向结果"。`name` 通过 `teeHole` 流向 buf 和(若元素含指针)堆;`[]byte` 元素无指针,所以 name 不进堆。
- **L4 `w.Write(buf)`**:`w` 是 interface,`Write` 是接口方法调用。接口调用的实参 buf 通过被调方法(`io.Writer.Write`)的 param tag 决定——`io.Writer.Write` 的 `p []byte` 标 `does not escape`(标准库定义),所以 buf 不逃逸。

最终:buf 不逃逸,栈分配;name 不逃逸,栈分配。整个函数零堆分配。这就是逃逸分析+param tag 协作的结果。

对比一个让 buf 逃逸的版本:

```go
func greetBad(name string) *[]byte {
    buf := []byte("Hello, ")
    buf = append(buf, name...)
    return &buf                    // buf 逃逸:返回值是 *[]byte
}
```

`return &buf` 让 buf 的地址流向返回值(`paramOut` 在 `outlives` 判 true),buf 标 `EscHeap`,走 `mallocgc`。

## 12.9 怎么观测逃逸分析的结果

理论讲完,实操怎么验证?三个工具:

1. **`go build -gcflags='-m'`**:打印每个函数的逃逸诊断(`moved to heap: x`、`x escapes to heap`、`leaking param: x`、`x does not escape`)。这是最直接的。

2. **`go build -gcflags='-m -m'`**:打印更详细的流图(`flow: ... ← &...: from ...`)。本章引用的 `explainFlow`/`explainPath`([solve.go:199-264](../go/src/cmd/compile/internal/escape/solve.go#L199-L264))就是 `-m -m` 输出的来源。

3. **`go build -gcflags='-l'`**:禁用内联。注意:**内联会显著改变逃逸分析的结果**——被内联的函数体直接并入调用方,很多原本逃逸的参数变得不逃逸。所以观测"真实逃逸"要小心内联的影响。

一个常见误区:**`-m` 输出的逃逸信息,是在内联之前还是之后?** 答案是:逃逸分析在内联之后跑([escape.go:131](../go/src/cmd/compile/internal/escape/escape.go#L131) 的 `VisitFuncsBottomUp` 处理的是内联后的函数列表)。所以 `-m` 看到的是内联后的结果,更接近实际生成的代码。

## 章末小结

这一章离开了 runtime,走进编译器的 `cmd/compile/internal/escape`。逃逸分析是内存分配的"前置裁判":它决定每个变量是进堆(走第 11 章的 `mallocgc`、被 GC 追踪)还是留栈(免 GC)。机制可以浓缩成三句话:

1. **画图**:每个变量映成 location,每条赋值映成带 `derefs` 权重的 edge。`&` 让权重 -1,`*` 让权重 +1。
2. **求最短路**:`walkAll` 对每个 location 跑 Bellman-Ford,求它到堆 / 返回值 / 闭包等"会逃逸终点"的最小 `derefs`。负数意味着地址流过去了,`outlives` 判定终点是否比源活得久——是就标 `EscHeap`。
3. **复用**:跨函数分析靠 param tag(`leaks` 8 字节编码 4 种流向),把每个函数的参数逃逸信息摘要进签名,调用方零成本复用。

栈分配"免 GC"的物理基础是:栈是 GC 的根,不是被回收对象;函数返回时栈帧整片弹出,不需要 GC 介入。逃逸分析的价值就是让堆对象尽可能少——堆越瘦,GC 越快,goroutine 越便宜。

本章服务的二分法:**支撑地基(内存)**。逃逸分析是 `mallocgc` 的上游闸门:它放行的变量才进第 11 章的三条道,它拦下的变量留在栈上、对 GC 不可见。没有逃逸分析,要么全堆(GC 压力爆炸),要么全栈(野指针崩溃)——Go 选了"精确分析 + 保守兜底"的中间道路。

### 五个"为什么"清单

1. **为什么逃逸分析要做?** 防止指向栈对象的指针逃到堆或活过栈帧——铁律一防野指针,铁律二防循环变量错值。不分析就只能全堆分配,GC 压力爆炸。
2. **为什么 `derefs` 用负数表示取地址?** 取地址是"反解引用",在赋值流图上产生负权边。Bellman-Ford 处理负权边,负权累加到终点就意味着地址流过去了。
3. **为什么接口装箱会逃逸?** interface 的 data 字段是指针,指向的对象地址必须稳定。栈对象在 goroutine 栈增长时地址会变,所以装进 interface 的非指针对象必须进堆(除非是 `IsDirectIface` 的指针类型本身)。
4. **为什么栈对象能免 GC?** 栈是 GC 的根,不是回收对象。GC 扫栈是为了找堆指针标堆,不是为了回收栈。函数返回时栈帧整片弹出,无需 GC 介入。
5. **为什么需要 param tag?** 跨函数逃逸分析若展开每个被调函数,编译时间指数爆炸。param tag 把每个函数的参数逃逸信息摘要成 8 字节存进签名,调用方零成本复用——"摘要 + 复用"的工程典范。
