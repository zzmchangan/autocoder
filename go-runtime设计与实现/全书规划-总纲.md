> 一本写给"天天写 Go、用满 goroutine/channel,甚至翻过 `src/runtime`,却总觉得一知半解"的人的小书。
>
> 本文是全书的**总纲**:定主线、定方法、定分篇分章、定源码策略、定写作约定。
> 与《Tokio》《LevelDB》《etcd》《数据库内核》同源,都是"**第一性原理 + 真实源码**";命脉同样是**动机(why) + 技巧(how) 双线**;基调延续《LevelDB》——**直球讲透为主,比喻只在反直觉处点睛**。

## 一、这本书的定位

### 目标读者

会写 Go、天天用 `go func` / `chan` / `select` / `sync.Mutex`,甚至翻过 `src/runtime/proc.go`、`chan.go`、`mgc.go`,但**一知半解**:知道有 G/M/P、GC、work-stealing、netpoll,却讲不清一个 goroutine 怎么被创建、怎么被调度、遇到阻塞怎么不卡住线程、GC 怎么并发地回收、channel 怎么做到高效通信;也讲不清 `runtime2.go` 里那些 G/M/P 的几十个字段、`asm_amd64.s` 里的 `mcall`/`gogo`、写屏障到底在防什么。和"读过 tokio/etcd 源码但不理解"是同一种病。

### 全书一句话主旨

> **Go 怎么让"开一个 goroutine 像开一个函数一样便宜",又怎么用少量 OS 线程高效驱动成千上万个 goroutine——GMP 调度 + 阻塞自动让出 + work-stealing + 并发 GC + TCMalloc 内存,全是 runtime 替你干的。**

每个组件,都是这条旅程上的一个**驿站**。读完一本书,读者能**在脑子里放映出一个 goroutine 的全过程 + 一次 GC 的全过程**:一个 `go func` 怎么变成一个 G、怎么被 P 的本地队列承接、怎么被 M 通过 `schedule` 取出执行、遇到 channel/网络阻塞怎么 `park` 让出线程、就绪后怎么被 `netpoll`/`sysmon` 重新唤醒、GC 怎么在后台并发标记不打断业务——以及每一步底下用了什么巧妙的手段。

### 为什么用"一个 goroutine 的一生 + 后台 GC"当骨架

Go runtime 的子系统高度耦合:调度(GMP)要靠 netpoll/sysmon 唤醒阻塞的 G,GC 要靠调度器暂停/辅助,内存分配要和 GC 协作(分配触发 GC、GC 回收释放),channel 又依赖调度器唤醒等待者。按模块百科式平铺,读者会觉得每块都是孤岛。用"一个 goroutine 怎么被创建、调度、阻塞、唤醒、被 GC 回收"这条旅程当骨架,每个子系统都是"为了让旅程继续而必须出现的角色"——读者永远知道**我在哪、为什么需要下一站**。

## 二、写作方法

### 命脉:动机(why) + 技巧(how) 双线 ⚠️ 本系列命脉

> **这一节是本书与 API 文档、一般源码导读最关键的分野,所有写作者必读、必守。**

Go runtime 源码里充斥系统级与编译器协作的技巧:G/M/P 的结构布局(缓存行对齐、字段顺序为减少竞争)、`mcall`/`gogo` 的栈切换汇编、work-stealing(偷一半本地队列)、异步抢占(基于信号 `SIGURG` 打断长时间运行的 G)、`schedule` 的找 G 优先级(本地→全局→netpoll→偷→GC→park)、channel 的无锁环形缓冲 + 等待队列、三色标记 + 写屏障(防并发标记漏标)、并发 GC 的 STW 最小化、TCMalloc 的 mspan/mcache/mcentral/mheap 层级、可增长栈的栈拷贝。用户自己读过源码,"一知半解"的缺口往往**不在**"它要解决什么问题"(动机好懂),而**在**"它怎么用这么巧的手段解决的"(技巧看不懂)。

所以本书每个关键代码点,都要讲**两件事**:

1. **设计动机(why)**:它解决什么本质问题?不这样会怎样?(沿用系列一贯的"为什么优先")
2. **实现技巧(how 的精妙)**:它用了什么手段?为什么这个手段妙?**不这么写会怎样**(性能损失 / 正确性陷阱)?

> **每章单开一个"技巧精解"小节**(固定位置:正文之后、小结之前),挑本章**最硬核的 1~2 个技巧**,配真实源码片段 + 行号,单独拆透。技巧解释必须配**反面对比**——"如果不用这个技巧、朴素地写,会撞上什么墙",让妙处显形。

一句话:**讲不清技巧,等于没讲 Go runtime。**

### 基调:直球讲透为主,比喻只在反直觉处点睛 ⚠️ 延续《LevelDB》

Go runtime 有一部分**反直觉**(为什么 goroutine 这么便宜?阻塞为什么不卡线程?GC 怎么做到几乎不暂停?),这部分**允许适度比喻点睛**。但 runtime 的大部分(G/M/P 结构、channel 环形缓冲、三色标记、mspan 层级)**具体可见,直球讲清最好**。

所以本书基调:
- **默认直球**:把结构、流程、源码本身讲透。能用一张表 / 一张时序图 / 一段源码说清的,**不上比喻**。
- **比喻只点睛,不贯穿,不另起体系**:只在反直觉处用一两句点一个比喻帮助"啊哈"。例如 G/M/P 可点一句"G 是任务、M 是干活的人(OS 线程)、P 是人手边的工作台(本地任务队列+缓存)";阻塞让出可点一句"G 遇到等待就交出工作台去休息,就绪了再回来抢工作台"。**点到为止**,不展开成对照表、不强制每章出现。
- **判断标准**:直球讲不清,才上比喻;讲得清,就直球。

### ★ 双璧对照《Tokio》:本书独有特色

> 这是选 Go runtime 最大的附加值。Go runtime 和 Tokio 是"调度问题"在两种语言里的顶级解——**一个是语言内置协程(编译器+runtime 协作),一个是库级异步 Future(纯库 + 类型系统)**。把两者对读,能看清"调度"问题的本质,以及两种设计的取舍。

本书每讲透一个核心机制,在**章末**用一个轻量"双璧对照"小栏,和 Tokio 的对应物对读(不喧宾夺主,3~5 行):

| Go runtime | Tokio | 对照点 |
|---|---|---|
| G/M/P + work-stealing | scheduler + work-stealing | 两套 work-stealing 的实现差异(偷一半 vs 偷一半,但 G 队列是本地 runq) |
| channel(环形缓冲 + 等待队列) | mpsc/oneshot/broadcast | 同步阻塞通信 vs 异步 Future 通信 |
| 并发三色 GC | 无 GC(靠 ownership + `Pin`) | 有 GC(省心智,有停顿/开销) vs 无 GC(零开销,要手动管理生命周期) |
| netpoll(集成 epoll) | mio + AsyncFd | 网络事件驱动的两种姿势(语言内置 vs 库封装) |
| go func(语言关键字) | tokio::spawn(库函数) | "并发单元"怎么造出来 |
| sysmon 监控线程 | —(Tokio 无独立监控) | Go 特有的后台监控 |

## 三、全书主线、二分法

### 主线

> **如何用极少的 OS 线程(M),高效驱动成千上万个 goroutine(G),并让"阻塞"不阻塞线程、让"垃圾回收"不打断业务——靠 P 承上启下、work-stealing 负载均衡、netpoll/sysmon 唤醒阻塞、并发 GC 后台清扫。**

任何一处看不懂某个机制,回到这句问:"这是在**让就绪的 G 跑起来(调度执行)**,还是在**让阻塞的 G 就绪后被重新调度(阻塞唤醒)**,或是在**支撑这两件事(内存/GC/栈)**?"把答案写进"为什么"里。

### 二分法(迷路时回到它,与 Tokio 高度对称)

本书的二分法:

> **调度执行(GMP:让就绪的 G 跑起来、work-stealing 偷工作) vs 阻塞唤醒(netpoll/sysmon/timer/channel:让阻塞的 G 在就绪后被重新调度)。**

- **调度执行这一面**:G/M/P 结构、`schedule`/`findRunnable`、work-stealing、异步抢占、syscall 的 handoff、sysmon。
- **阻塞唤醒这一面**:netpoll(网络 I/O 就绪)、sysmon(长时间阻塞检测)、timer(定时唤醒)、channel(等待者唤醒)。

支撑这两面的地基:**内存分配**(TCMalloc,给 G 分配栈和对象)、**并发 GC**(回收垃圾,靠写屏障和调度器协作)、**栈管理**(可增长栈,让 G 真正轻量)。

Go runtime 每个机制都落在这两面之一或支撑地基。任何时候迷路,回到这个二分法。

### 第 0 篇 · 开篇:为什么有 goroutine(定调,1 章)

- **第 1 章 · 第一性原理:为什么 goroutine 这么便宜**(`P0-01`):OS 线程为什么贵(创建/切换/内存开销)→ 协程为什么便宜(用户态栈切换、按需增长)→ Go runtime 替你干了哪三件事(调度 + GC + 内存)→ `go func` 一行背后发生了什么 → 立起"调度执行 vs 阻塞唤醒"二分法 + 双璧对照 Tokio。**【样本章节】**

### 第 1 篇 · GMP 调度器:调度执行(6 章)⚠️ 全书重头戏

> 这一篇讲透调度器。源码主体在 `src/runtime/runtime2.go`(结构)、`proc.go`(调度逻辑)、`asm_amd64.s`(切换汇编)。建议顺序读。

- **第 2 章 · G、M、P 结构与全局状态**(`P1-02`)——G(goroutine)、M(OS 线程)、P(处理器,本地队列+缓存)各是什么、它们的字段为什么这么排;全局的 `allgs`/`allm`/`allp`/`sched`。源码:`runtime2.go`。**技巧精解:G/M/P 的结构布局**(P 的 runq、mcache、缓存行考虑;G 的栈、状态机、sudog 等待块)——这是理解后面一切的地基。
- **第 3 章 · 调度循环:schedule 与 findRunnable**(`P1-03`)——M 怎么不断找下一个可跑的 G(`schedule`→`findRunnable`);找 G 的优先级(本地 runq→全局 runq→GC→netpoll→偷→park),为什么这么排。源码:`proc.go`。**技巧精解:findRunnable 的多级回退**(每一级在解决什么"找不到 G"的情况)。
- **第 4 章 · work-stealing:偷工作的艺术**(`P1-04`)——空闲的 P 怎么从别的 P 偷一半 runq;无锁 runq 的实现(环形数组 + head/tail 原子)。源码:`proc.go`(`runqsteal`/`runqgrab`/`runqget`)。**技巧精解:无锁环形 runq + 偷一半**(为什么偷一半而非一个——摊平负载)。**双璧对照 Tokio work-stealing**。
- **第 5 章 · 异步抢占:基于信号的协作式→抢占式**(`P1-05`)——为什么要有抢占(一个 G 死循环不让出会饿死别人);Go 1.14 前的协作式(函数前奏检查)为什么不够;异步抢占怎么靠 `SIGURG` 信号在任何安全点打断 G。源码:`proc.go`(`preemptM`/`signal_recv`)、`asm_amd64.s`。**技巧精解:异步抢占的信号机制 + 安全点**(凭什么能在任意点打断而不破坏 GC/栈)。
- **第 6 章 · 系统调用阻塞:handoff P**(`P1-06`)——G 阻塞在系统调用(非网络)时,M 怎么把 P 让给别的 M 干活(`entersyscall`/`exitsyscall`、`handoffp`);不让一个阻塞系统调用卡住整个 P。源码:`proc.go`(`entersyscall`/`exitsyscall`)、`asm_amd64.s`。**技巧精解:entersyscall 的"释放 P" + exitsyscall 的"抢回 P"**(阻塞不浪费 P)。
- **第 7 章 · sysmon:监控线程**(`P1-07`)——独立的不受 GC/调度约束的后台线程,负责:检测长时间运行的 G(触发抢占)、检测长时间阻塞的系统调用(触发 handoff)、触发网络轮询、触发 GC。源码:`proc.go`(`sysmon`)。**技巧精解:sysmon 作为"独立观察者"**(为什么需要它——调度器自己可能被阻塞,得有个不受约束的线程兜底)。

### 第 2 篇 · channel:CSP 通信(2 章)

- **第 8 章 · channel 结构与发送/接收**(`P2-08`)——`hchan` 的结构(环形缓冲 buf + 等待队列 sendq/recvq + 锁);发送/接收的快路径(无锁/直接)、慢路径(阻塞入队、被唤醒);`close` 的传播。源码:`chan.go`。**技巧精解:channel 的快路径优化**(有 buf 且未满直接写;发送方遇接收方直接拷贝省一次 buf)。**双璧对照 Tokio mpsc**。
- **第 9 章 · select 的实现**(`P2-09`)——`select` 怎么在多个 channel 上"就绪一个就返回";乱序选 case、默认 case;底层 `selectgo` 的实现。源码:`select.go`。**技巧精解:select 的公平性 + 锁顺序**(多个 case 怎么不饿死、不死锁)。

### 第 3 篇 · 内存分配:TCMalloc 思想(3 章)

> 给 G 分配栈和堆对象的地基。

- **第 10 章 · mspan/mcache/mcentral/mheap 层级**(`P3-10`)——Go 的分配器仿 TCMalloc:mspan(定长块)、mcache(P 本地缓存,无锁)、mcentral(全局,按 size class)、mheap(页分配器);为什么这样分层(局部性 + 锁粒度)。源码:`mcache.go`/`mcentral.go`/`mheap.go`/`mspan.go`。**技巧精解:三级缓存的锁粒度递进**(P 本地无锁 → central 加锁 →heap 全局锁,快路径几乎无锁)。
- **第 11 章 · 大小对象分配路径**(`P3-11`)——微小对象(tiny allocator,合并)、小对象(mspan)、大对象(直接 mheap)三条路径;`mallocgc` 的流程。源码:`malloc.go`(`mallocgc`)。**技巧精解:tiny allocator**(把多个微小对象塞进一个块,省内存 + 对齐)。
- **第 12 章 · 逃逸分析:堆还是栈**(`P3-12`)——为什么有些变量分配在堆(逃逸)、有些在栈(省 GC);编译器怎么决定;栈分配的对象怎么免 GC。源码:编译器(`cmd/compile/internal/escape`) + `malloc.go`。**技巧精解:逃逸分析的规则**(变量地址逃逸到堆;不逃逸就栈分配,零 GC 开销)。

### 第 4 篇 · 并发 GC:三色标记(4 章)

- **第 13 章 · 三色标记与写屏障**(`P4-13`)——为什么并发标记会产生"漏标"(白→黑丢失灰);三色不变式;写屏障(Dijkstra/混合)怎么堵住漏标。源码:`mgc.go`/`mbw_amd64.s`。**技巧精解:写屏障防漏标**(反面对比"无屏障并发标记"会丢活对象)。
- **第 14 章 · 并发 GC 的阶段**(`P4-14`)——GC 怎么几乎不暂停业务:标记准备(极短 STW,开启屏障、扫栈)、并发标记(和业务并发)、标记终止(极短 STW)、并发清扫;两次 STW 为什么能做到亚毫秒。源码:`mgc.go`(`gcStart`/`gcMarkDone`/`gcSweep`)。**技巧精解:STW 最小化**(业务和标记并发的工程,凭什么停顿这么短)。
- **第 15 章 · mark assist:G 协助 GC**(`P4-15`)——分配太快的 G 会被要求"帮忙标记"(mark assist),把 GC 压力分摊给分配者,防止单靠后台标记跟不上。源码:`mgcmark.go`(`gcAssistAlloc`)。**技巧精解:mark assist 的反压**(谁分配谁协助,把 GC 和分配速率耦合)。
- **第 16 章 · sweep:清扫与回收**(`P4-16`)——标记完怎么回收:并发清扫、mspan 的清理、空闲页归还。源码:`mgcsweep.go`。**技巧精解:lazy sweep**(分配时顺带清扫,而非单独阶段)。

### 第 5 篇 · 栈管理:让 G 真正轻量(1 章)

- **第 17 章 · 可增长栈与栈拷贝**(`P5-17`)——G 的栈为什么能"按需增长"(初始 2KB,不够就翻倍拷贝);栈拷贝怎么处理栈上指针调整;栈收缩(空闲时缩回)。源码:`stack.go`(`newstack`/`copystack`/`shrinkstack`)。**技巧精解:连续栈拷贝**(凭什么能拷贝一个正在用的栈——调整所有指针)。**双璧对照 Tokio 的 Pin**(Go 靠栈拷贝解决自引用,Tokio 靠 Pin 禁止移动)。

### 第 6 篇 · netpoll:网络不阻塞线程(2 章)

- **第 7 章(18)· netpoll:集成 epoll**(`P6-18`)——网络 I/O 阻塞的 G 怎么不占 M:底层 epoll/kqueue 被 runtime 集成,G 阻塞在 socket 时被 park,数据就绪由 netpoll 唤醒重新入队。源码:`netpoll_epoll.go`/`netpoll.go`。**技巧精解:netpoll 把 epoll 藏进 runtime**(对比裸 epoll 用法)。**双璧对照 Tokio mio/AsyncFd**。
- **第 19 章 · 网络 I/O 的阻塞与唤醒全流程**(`P6-19`)——一次 `conn.Read` 阻塞到唤醒的完整路径:runtime_netpollblock → park → epoll_wait → 唤醒 → 入 runq。源码:`netpoll.go`/`fd_unix.go`。**技巧精解:从系统调用到 netpoll 的衔接**(怎么把"阻塞读"变成"park + 等就绪")。

### 第 7 篇 · sync 原语与 timer(1 章)

- **第 20 章 · Mutex 自旋+信号量、timer 四叉堆**(`P7-20`)——`sync.Mutex` 怎么先自旋再加锁(减少上下文切换)、底层信号量(semaphore);`time.Sleep`/`After` 的 timer 用四叉堆管理(为什么四叉而非二叉)。源码:`src/sync/mutex.go`、`runtime/sema.go`、`runtime/time.go`。**技巧精解:Mutex 的 normal/starvation 模式 + 四叉堆**(自旋换吞吐、饥饿模式防饿死;四叉堆缓存友好)。

### 第 8 篇 · 收尾:权衡哲学与双璧总表(1 章)

## 七、Go runtime 技巧侧重(对标 Tokio Rust / etcd Go / LevelDB C++)

本书要重点拆的 Go runtime 系统级技巧(写作时遇到这些,务必钻"怎么做到的 + 为什么 sound"):

- **G/M/P 结构布局**:字段顺序、缓存行、本地 runq——调度器的高性能根基。
- **栈切换汇编**:`mcall`/`gogo`/`systemstack` 怎么在 goroutine 栈和系统栈间切(对照 Tokio 无栈 Future)。
- **work-stealing**:无锁环形 runq(head/tail 原子)、偷一半。
- **异步抢占**:`SIGURG` 信号在任何安全点打断 G,凭什么不破坏 GC/栈。
- **channel**:无锁环形缓冲 + 等待队列(sudog)、快路径省 buf。
- **并发 GC**:三色 + 写屏障(防漏标)、两次 STW 最小化、mark assist 反压。
- **TCMalloc 分层**:mcache(P 本地无锁)→ mcentral(加锁)→ mheap(全局),快路径几乎无锁。
- **可增长栈**:连续栈拷贝 + 调整栈上指针。
- **netpoll**:把 epoll/kqueue 藏进 runtime,网络阻塞不占 M。
