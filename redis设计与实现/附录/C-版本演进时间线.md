# 附录 C · Redis 版本演进时间线

本书一路把 Redis 8.0 的设计拆开来讲——为什么用单线程事件循环、为什么要分片字典、为什么 IO 多线程要在 8.0 推倒重来。但读者难免会问:这些设计是一开始就有的吗?还是慢慢长成这样的?这条附录就是来回答这个问题的。

我们沿着版本号往上走,每个节点只挑两到四件与本书相关的事,讲清两件事——**它解决了上一版什么痛**,以及**它换了什么设计取向**。拿不准的版本细节我会标"大约""前后",宁可少说,也不编造。

---

## 2.x(2009 起):地基就这么打下了

Salvatore Sanfilippo(antirez)在 2009 年前后放出第一版 Redis。2.x 这一代几乎奠定了 Redis 后面十几年的精神底色:

- **单线程 + 事件循环(ae)**。本书反复讲的 `ae.c` 那一套事件循环,在这一代就立住了:主线程串行处理命令,IO 与计算都不分叉。这不是技术落后,而是一个清醒的取舍——内存操作太快,多线程的锁与上下文切换反而拖后腿。Redis 的"快"从一开始就不是靠多核,而是靠"不等待"。
- **VM(虚拟内存)短暂存在又移除**。早期有过一个把冷 value 换到磁盘的 VM 机制,大约在 2.4/2.6 前后被彻底拿掉。antirez 的结论很硬:用操作系统的 swap 思路管 Redis 内存是错的,内存治理应该用更贴合数据结构的方式(这正是后来 maxmemory + 淘汰策略 + 对象编码优化的方向)。
- **基础数据结构落地**。SDS、dict、ziplist、intset、skiplist 都在这一代成型。本书讲的数据结构"双重身份"(对外是类型,对内是编码),根就在这里。

2.x 的取向可以一句话概括:**先把"单线程也能很快"这件事做到极致,复杂度一律不提前引入。**

---

## 3.0(2015):Cluster 正式 GA

3.0 最大的事是 **Redis Cluster 正式 GA**。在此之前 Cluster 有过多年 RC,到 3.0 才算稳定可用。

- **16384 个 slot + gossip**。本书第 17 章讲的"为什么是 16384 而不是 65536"——slot 数量用 16384 是为了集群节点间消息体大小可控,这是个很务实的工程取舍。节点间用 gossip 协议同步拓扑,故障检测与迁移都建立在它之上。
- **dict 渐进式 rehash 完善**。渐进式 rehash 的思路更早就有了,但 3.x 这一代它在 Cluster、复制、各种内部结构里被普遍依赖,成为一个不能出错的底层契约。本书讲 dict 时强调的"两个 ht 共存、每次操作搬一小段",在 3.x 已经是稳固形态。

3.0 的取向:**承认单机有上限,但分片这件事只在数据层做,集群管理本身仍然尽量轻。**

---

## 4.0(2017):复制、模块、淘汰策略三件套

4.0 是一次"打磨与开窗"并存的版本。

- **PSYNC2**。复制断连后的"部分重同步"在 2.8 就有 PSYNC,但 4.0 的 PSYNC2 大幅改进了重同步的成功率——主从各自记住更多 backlog 偏移与 ID,即使主从角色切换、或中间发生过故障迁移,也能尽量只传增量而不是全量。本书第 14 章讲的重同步,真正成熟是在这里。
- **模块系统(Module API)**。Redis 第一次把内核打开一个口子,允许第三方用 C 写自定义数据类型与命令。这是个取向上的大转弯:以前是"内核只做我认为对的事",现在是"内核提供扩展点"。后来 Redis Stack 的几大组件,很多都建立在 Module API 之上。
- **LFU 淘汰策略**。在原有 LRU 之外,加入了基于访问频率的 LFU(用一个精巧的 Morris 近似计数器+衰减)。本书讲淘汰策略时点过它的"近似 LRU/LFU 随机采样 N 个"做法——它没有维护一个真正的全局排序,因为那太贵。
- **HD dict(High-level Dict)等内部整理**。这一代也在持续优化内存布局,为后面的变化铺路。

4.0 的取向:**内核开始愿意让出一些边界(模块),但主路径仍然保守。**

---

## 5.0(2018):Stream 来了

5.0 的标志是 **Stream 数据类型**。

- **Stream**。这是为日志/消息/事件流场景设计的新类型,本书第 9 章讲过它的主索引建立在 **rax**(基数树)上,ID 是 `<毫秒时间戳>-<序号>` 自增。Stream 把"消费者组、未确认消息、阻塞读"都内化进一种数据结构,是 Redis 第一次明确地把"消息队列"这件事纳入一等公民。
- **listpack 的前奏**。listpack 在这一代开始被引入,目标是替代 ziplist。ziplist 那个"每个 entry 都存前一个的长度、改一个就要连锁更新后面所有 entry"的毛病(本书第 6 章详讲),listpack 用"entry 自己记自己的长度"来根治。5.0 先在部分场景试水,全面铺开是 7.0 的事。
- **新复制协议整理、Cluster bus 收敛**等内部工程。

5.0 的取向:**数据结构层面继续扩张(Stream),同时开始偿还历史债(ziplist)。**

---

## 6.0(2020):IO 多线程第一次试水,权限与协议全面升级

6.0 是 Redis 第一次认真面对"单线程 IO 成了瓶颈"这件事。

- **IO 多线程(io-threads)**。本书第 20 章重点讲过:6.0 的 IO 多线程**只多线程做写回(socket write),后来扩展到读+协议解析,但命令执行始终单线程**。它解决的是"网络 IO 在万兆/十万兆网卡下吃满 CPU"的痛,又用"命令执行不分叉"守住了 Redis 的简单性契约。注意——这一版的设计到 8.0 又被推倒重做了一次,后面会讲。
- **ACL(访问控制列表)**。终于有了用户级别的权限分离,不再只有一个全局密码。可以限制某用户只能跑某些命令、只能访问某些 key。
- **RESP3 协议**。客户端与服务端之间的协议从 RESP2 升级到 RESP3,能表达更丰富的类型(映射、大整数、双精度等),为客户端缓存铺路。
- **客户端缓存(tracking)**。客户端可以在本地缓存 key 的值,服务端在该 key 变更时主动通知失效。这是把缓存这一层"外包"给客户端、用通知维持一致性的设计,取向很 Redis。

6.0 的取向:**承认瓶颈、用最小入侵的方式把耗时外包出去(IO 多线程只动 IO、命令执行不动),同时补齐运维短板(ACL/协议)。**

---

## 7.0(2022):大清理 + 持久化升级 + 一场分叉

7.0 是变化剧烈的一代,既有内部大清理,也有外部的社区震动。

- **listpack 全面取代 ziplist**。本书第 6 章讲过的"连锁更新"根治,在这一代完成:hash、zset、list 的小数据编码从 ziplist 切到 listpack。一个埋了多年的雷终于拆掉。
- **Functions**。本书第 21 章讲脚本时提过:`EVAL` 系列是"临时发一段脚本上来执行",而 Functions 是"把函数注册到服务端、按名调用"——它让脚本能像存储过程一样持久化、像数据一样在主从间复制。这是把脚本从"临时工具"升级为"一等公民"。
- **sharded pubsub**。发布订阅的频道按 slot 分片,订阅消息只在负责该 slot 的节点上流转,不再全集群广播。这是 Cluster 化思路向 pubsub 的延伸。
- **MP-AOF(多部件 AOF)**。本书第 15 章讲过:7.0 把 AOF 拆成 base(全量快照)+ incr(增量日志)多个文件,重写时不再需要 fork 单独搞一份大头,而是直接复用 RDB 作 base。这让 AOF 重启恢复更快、重写更省。
- **授权从 BSD 改为 SSPL/RSALv2**。这一版改了开源协议,直接引发了后来的社区分叉——AWS 等厂商联合分叉出 **Valkey**(Linux 基金会托管)。这件事我们放到 8.0 那节再收尾。

7.0 的取向:**内部还债(listpack)、对外开窗(Functions/MP-AOF),但协议变更埋下了社区分裂的种子。**

---

## 8.0(2025):重回开源,IO 多线程推倒重来

8.0 是本书源码对应的版本,也是变化最大的一代。

- **重回开源(BSD)**。回应 7.0 协议变更引发的社区出走,8.0 重新回到 BSD 许可(详见序章)。Redis 与 Valkey 的关系成了开源世界一个长期话题。
- **IO 多线程重构**。本书第 20 章重点:6.0 那套"io-threads 在主线程的事件循环里分派任务"的做法被推倒重做——**每个 IO 线程跑自己独立的事件循环(ae)**,协议解析下放到 IO 线程。源码里对应新增的 `iothread.c`(本书已交叉验证存在)。这解决了 6.0 设计在连接数极高时的瓶颈,但仍然守住"命令执行单线程"这条底线。
- **kvstore(分片字典)**。本书第 11 章讲过:Cluster 模式下每个 slot 一个 dict 会浪费内存、又要在 rehash 时麻烦;8.0 引入 `kvstore`(对应源码 `kvstore.c`),把多个 slot 的 dict 统一管理,按需 shared、按需 rehash。这是把"分片"这件事下沉到一个专用数据结构里。
- **mstr / keymeta 方向**。本书第 10 章讲过的对象头/key 元信息整理方向,对应 8.0 新增的 `mstr.c`。它代表一种取向:以前 Redis 把"这个 key 是什么、带什么元信息"散落在各处编码里,8.0 开始往统一的 key metadata 结构收敛。这是为后续版本进一步优化(过期、内存记账、分片)打地基。
- **Redis Stack 合入核心**。原本以 Module 形态存在的 Search、JSON、Bloom、TimeSeries 等,合入核心发行版。这让 Redis 从"纯 KV/数据结构服务器"向"多模型数据库"又迈了一步。
- **vector sets 与多项性能/内存优化**。新增面向向量相似度的数据类型,顺应大模型时代的检索需求;同时大量内存与吞吐优化。

8.0 的取向:**在外部回归开源社区,在内部做一次"为下一个十年铺路"的重构(IO 多线程、kvstore、keymeta 都是基础设施级别的更新)。**

## 从演进看 Redis 的设计哲学

把这条时间线拉直了看,会发现 Redis 几乎每个版本都在重复同一套逻辑,可以归纳成三条:

**一、简单优先,复杂度只在瓶颈处才上。** 单线程事件循环从 2.x 一直活到 6.0 才在 IO 层开口子,而且 6.0 只动 IO 不动命令执行——因为命令执行那边还没真正成为瓶颈。Cluster 不是一开始就有,而是单机扛不住了才在 3.0 GA。分片字典 kvstore 不是 Cluster 一上线就有,而是要等到集群规模更大、内存浪费更明显时,才在 8.0 引入。复杂度从来不是设计的起点,而是被性能曲线逼出来的终点。

**二、把耗时的事外包出去。** 网络 IO 慢?交给 IO 线程。客户端要重复读同一个 key?交给客户端缓存,服务端只负责通知失效。脚本每次重发太慢?升级成持久化的 Functions。全量重同步太贵?PSYNC2 让你尽量只传增量。Redis 总是先把"谁最该花这个时间"想清楚,再把活儿派给最合适的人,而不是自己一把抓。

**三、按数据结构而非抽象来思考。** 从最早的 SDS/dict/ziplist,到 listpack 根治连锁更新,到 rax 支撑 Stream,到 kvstore 把分片做成数据结构,再到 mstr/keymeta 把元信息收敛——Redis 的每一次进步,几乎都伴随一种新的或重构过的数据结构落地。它不相信"通用抽象层",更相信"为这个场景量身做一个紧凑、可预测的结构"。

还有一个贯穿全程的细节:**Redis 宁可补丁式地改,也不轻易推翻。** ziplist 的毛病从很早就被诟病,但 listpack 一直到 7.0 才全面取代,中间隔了好几代的试水。6.0 的 IO 多线程设计不理想,8.0 才推倒重做。这种"先忍、忍到真的不能忍再大改"的节奏,正是 Redis 能在十几年里保持接口稳定、同时内部不断演化的原因。

最后一句送给读者:如果你在 8.0 的某个设计面前觉得"这也太绕了吧",不妨往回翻一翻这个版本之前发生了什么。绝大多数时候,那个绕,都是某一代 Redis 为之前省下的复杂度还的债。**复杂度不消灭,只转移**——这句话在 Redis 的版本史里,体现得淋漓尽致。
