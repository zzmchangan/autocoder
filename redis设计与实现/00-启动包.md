## 0. 全书主轴:把 Redis 的整个设计讲清楚

> **主轴:不为单句口号所囿,把 Redis 的每个设计决策讲清楚——为什么这样、不那样,动机(why)+ 精妙(how)双线,并讲清它换来了什么、付出了什么。**

"快"是 Redis 的结果,不是设计目标。Redis 的设计是多元的,不像 PG 能压成"快 vs 不丢不乱"单一张力,也不像 Tokio 能压成"把等待从占用解放"一句。强行用一句话压 23 章会扭曲材料——skiplist 选不选红黑树、编码何时从 listpack 切到 hashtable、cluster 16384 个 slot、repl backlog 环形缓冲,这些"为什么"各不相同,不该被塞进同一个框。

所以本书脊柱回到方法本源(见 [[source-code-series-method]]):**每章讲清这块设计在解决什么问题、为什么是这样不是那样**,不强制每章回扣单一句口号。

### 设计取向图(读者每章可对照的坐标系,非强制回扣清单)

Redis 反复在做的事,大致五条。每章讲完,自然落在其中一条或几条上——诚实优先,不硬贴:

1. **单线程 + 事件循环**:无锁、无并发 bug、可预测。这是"把耗时从主线程解放"那条线的根,贯穿大半本书(ae、dict 渐进 rehash、lazyfree、RDB fork、AOF、IO 多线程、expire/evict)——它依然是最显眼的一条,但**只是五条里的一条,不再是独占主线**。
2. **内存即数据库**:一切常驻内存,为每种访问模式定制数据结构换性能(SDS / dict / listpack / quicklist / skiplist / rax / intset)。P2/P3 的底色。
3. **编码自适应**:对外一套命令,底层多种编码,按数据规模/形态自动切换(`object.c` 的 `OBJ_ENCODING_*` 阈值)。贯穿 P2/P3。
4. **简单优先**:能用简单结构就不用复杂的——skiplist 不用红黑树、SDS 不用 C++ string、自造轮子而非引第三方依赖。这是 Redis 的气质,会在多处点出。
5. **可靠性靠持久化 + 复制**:内存易失,于是 RDB/AOF 兜底持久化,主从 + Cluster 兜底高可用。P4/P5 的主线。

### 结构上的切分(便于组织,不是唯一视角)

篇章仍按 **留在主线程的(P1-P3) vs 解放出去的(P4-P6)** 切两半——因为材料本身就这么分(命令链路/数据结构/内存治理是主线程在干的,持久化/复制/后台是主线程之外的)。这只是组织手段,内容上每章各讲各的设计取向。

每章结尾:**回扣本章真正相关的设计取向(见取向图),或收一句"这个设计换取了什么、牺牲了什么"**——哪条贴哪条,不强求一致。

## 3. 篇章结构(P0–P7,共约 23 章 + 附录)

### P0 · 开篇(1 章)
- **Ch0 序章:为什么单线程的 Redis 这么快——以及,这不只是"快"的故事** — 抛出主轴"讲清整个设计",给设计取向图与全书地图,与 Tokio 书呼应。

### P1 · 留在主线程:命令的一生(3 章)
- **Ch1 RESP 与 networking.c** — 字节流怎么进来;query buffer;铺垫"读"如何被赶出主线程(IO threads 伏笔)。
- **Ch2 ae 事件循环** — Redis 的心脏:epoll reactor、`beforeSleep`/`afterSleep`、为什么选单线程 reactor。
- **Ch3 命令的执行 processCommand** — 命令表、分发表;一条 SET 从字节到落库的完整旅程(回扣 PG 书"一条 SQL 的一生")。

### P2 · 留在主线程:数据结构的执念(让操作常数化)(6 章)
- **Ch4 SDS** — 为什么不用 `char*`;二进制安全、预分配、惰性释放;5 种 header 的内存技巧。
- **Ch5 dict 与渐进式 rehash** ⭐ — 两张表、`rehashidx`、每操作搬一格 + 定时辅助。
- **Ch6 listpack / intset / ziplist** — 紧凑编码的取舍;ziplist 为何被 listpack 取代(连锁更新问题)。
- **Ch7 quicklist** — list 的双层结构 + LZF 压缩;何时 node 存 listpack。
- **Ch8 skiplist(zset)** — 为什么不是红黑树:范围操作友好、实现简单、概率层高(几何直觉讲法)。
- **Ch9 rax 与编码自适应总览** — radix tree 前缀压缩;`object.c` 的 `OBJ_ENCODING_*` 阈值切换作为本篇收口。

### P3 · 留在主线程:内存治理(4 章)
- **Ch10 redisObject 与 keymeta** — `type`/`encoding`/`refcount`/`lru`;8.0 的 keymeta 层带来的变化。
- **Ch11 kvstore:分片字典** — 8.0 cluster 模式下每 slot 一个 dict 的设计;per-slot 渐进 rehash。
- **Ch12 zmalloc 与 defrag** — jemalloc 包装、碎片率;在线碎片整理如何不阻塞。
- **Ch13 过期与淘汰** — `expire`(lazy + active 双策略);`evict`(LRU/LFU 近似采样,而非全扫)。

### P4 · 解放出去:持久化(2 章)
- **Ch14 RDB** — fork 子进程 + COW 的精妙与风险(fork 慢、COW 内存放大)。
- **Ch15 AOF** — 追加写、rewrite(fork)、`appendfsync` 三档(always/everysec/no)与 BIO 线程的配合。

### P5 · 解放出去:复制与集群(3 章)
- **Ch16 主从复制** — 全量(RDB 传输)+ 增量(repl backlog 环形缓冲);psync。
- **Ch17 Cluster** — 16384 slot、crc16、MOVED/ASK 重定向、gossip 协议(`cluster_legacy.c`)。
- **Ch18 Sentinel 哨兵** — 选主、故障转移、quorum(可适度从简)。

### P6 · 解放出去:后台线程与 IO 多线程(2 章)
- **Ch19 BIO 与 lazyfree** — 后台线程做异步 fsync/close/free;`DEL` 大 key 的 `UNLINK`。
- **Ch20 I/O 多线程(8.0)** — 为什么只多线程 read/write、命令执行仍单线程;把网络耗时赶出主线程,把数据结构操作的纯粹留在主线程。

### P7 · 收尾专题(1 章 + 结语)
- **Ch21 那些"看似阻塞"如何不阻塞** — 事务(MULTI-WATCH 乐观锁)、Lua(eval 沙箱与超时)、阻塞命令(BLPOP 的 blocked clients 唤醒机制);主轴收束。

### 附录
- **A 怎么读 Redis 源码** — 编译(`make no-opt` 带 -g)、gdb 关键断点(`aeMain`/`processCommand`/`dictRehash`)、redis-cli DEBUG 命令。
- **B 命令 ↔ 源码函数映射表**。
- **C 版本演进时间线** — 2.x → 7.x → 8.0 关键架构变化(单线程 → IO 多线程、ziplist → listpack、dict → kvstore、引入 keymeta)。

## 4. 写作方法适配(与 Tokio 书的差异)

沿用 [[source-code-series-method]] 的动机+技巧双线、真实 grep 核对、固定四段、全角句号。**针对 Redis/C 的调整**:

| 维度 | Tokio 书(Rust) | Redis 书(C) |
|---|---|---|
| 技巧侧重 | `unsafe` / 原子 / `Pin` / 无锁 / 缓存行对齐 | **指针把玩 / 宏技巧 / 位运算编码 / 紧凑内存布局 / fork-COW / 信号 / 手写无依赖数据结构** |
| 比喻 | 餐厅服务员 | 弱化比喻、直球(遵 [[db-kernel-writing-method]]);仅用"单人收银台 + 后厨异步"做开篇定向,之后转直球 |
| 验证物 | tokio 代码示例跑通 | **redis-benchmark 实测 + gdb 单步看数据结构 + `OBJECT ENCODING`/`DEBUG` 观察编码切换与 rehash**;每章交付前原样重跑,严防 [[mc-sim-reproducibility-pitfall]] |
| 源码事实 | `tokio-source-facts` | 新建 `redis-source-facts`,边写边 grep 核实并记(skiplist MAXLEVEL=32、dict 字段、aeEventLoop 字段、5 种 sdshdr…) |

**每章固定四段**:① 这块要解决什么(动机/why)② 直球讲设计与源码(how,带真实行号)③ 精妙技巧点拨(why 这样不那样)④ 回扣本章相关的设计取向(见 §0 取向图),或收一句"这个设计换取了什么、牺牲了什么",不强求一致。
