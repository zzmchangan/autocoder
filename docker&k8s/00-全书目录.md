## 这本书的"一条主线"

整本书只讲一件事:

> **一个普通的应用,是怎么一步步被打包进标准集装箱(镜像)、装上共享内核的货轮(容器)、最后由航运调度中心(k8s)在几百艘船上调度成几万个的。**

所有章节都挂在这条主线上。迷路时回到一句话问自己:

> **"这一步是在'装箱隔离一个应用'(docker 半本),还是在'调度编排一群箱子'(k8s 半本)?"**

## 目录

### 第 0 篇 · 开篇:容器到底是什么

#### 第 1 章 · 第一性原理:容器不是虚拟机

- **核心问题**:容器到底是什么?它和虚拟机差在哪?凭什么又能隔离、又这么轻?
- **要讲清的"为什么"**:
  - 裸进程有什么不行(能看到/改到宿主的一切)。
  - 为什么 VM 这条路(虚拟硬件 + 独立内核)太重——为了隔离,值不值得每次都启动一个完整 OS。
  - 容器凭什么又轻又隔离:**所有容器共享同一个内核**,隔离靠内核已有的 namespace + cgroup,不虚拟硬件。
  - 最反直觉的真相:**容器不是一类新东西,它就是一个普通的进程**,只是套了两件外套。
- **关键源码**:kernel `clone` 的 `CLONE_NEW*` 标志([include/uapi/linux/sched.h](https://github.com/torvalds/linux/blob/master/include/uapi/linux/sched.h));runc 的 `CloneFlags()`([libcontainer/configs/namespaces_syscall.go](https://github.com/opencontainers/runc/blob/main/libcontainer/configs/namespaces_syscall.go))——证明"容器底层就是 clone 的标志位"。

### 第 1 篇 · 容器的三块基石

#### 第 2 章 · namespace:集装箱的铁皮舱壁

- **核心问题**:同一个内核,怎么让两个进程"看见完全不同的世界"(各自的进程表、网卡、挂载点)?
- **要讲清的"为什么"**:
  - 为什么需要 **8 种** namespace(只隔离进程表不够:还有网络、挂载、主机名、IPC、用户……每种各管一摊)。
  - 为什么是"clone 时带标志"这种设计(而不是另起一套 API)。
  - `/proc/<pid>/ns/` 为什么是理解 namespace 的最佳观测点。
  - user namespace 为什么特殊(让"容器里的 root = 宿主的 nobody",安全基石)。
- **关键源码**:kernel `CLONE_NEW*` 定义、`copy_namespaces()`([kernel/fork.c](https://github.com/torvalds/linux/blob/master/kernel/fork.c));各 namespace 实现([kernel/pid_namespace.c](https://github.com/torvalds/linux/blob/master/kernel/pid_namespace.c)、[net/core/net_namespace.c](https://github.com/torvalds/linux/blob/master/net/core/net_namespace.c) 等);runc 如何配置 namespace([libcontainer/configs/namespaces.go](https://github.com/opencontainers/runc/blob/main/libcontainer/configs/namespaces.go))。

#### 第 3 章 · cgroup:给进程上配额

- **核心问题**:光隔离(看不见)还不够——一个容器还能把宿主的 CPU/内存吃光。怎么限量?
- **要讲清的"为什么"**:
  - 为什么 namespace 不管资源(它只管"看见什么",不管"用多少")。
  - cgroup v1(按控制器分层次)vs v2(统一层级),为什么要有 v2。
  - cpu / memory / blkio 三个最常用控制器各怎么计量、怎么限额。
  - 为什么容器有"局部 OOM"(cgroup memory 超限只杀这个容器,不波及宿主)——回扣内核 OOM 章。
- **关键源码**:kernel cgroup 核心([kernel/cgroup/cgroup.c](https://github.com/torvalds/linux/blob/master/kernel/cgroup/cgroup.c))、memory 控制器([mm/memcontrol.c](https://github.com/torvalds/linux/blob/master/mm/memcontrol.c));runc 应用 cgroup 配置([libcontainer/cgroups](https://github.com/opencontainers/runc/tree/main/libcontainer/cgroups))。

#### 第 4 章 · rootfs 与 pivot_root:换个根

- **核心问题**:容器里的进程,凭什么看到的是"自己的文件系统"而不是宿主的?
- **要讲清的"为什么"**:
  - 为什么光 mount namespace 不够(还得把根目录换掉),否则进程照样能 `/` 看到宿主一切。
  - `chroot` 为什么不安全(只改路径解析,特权进程能逃逸),`pivot_root` 为什么更安全(整体替换根挂载点)。
  - 为什么换根是"容器看起来像个独立系统"的最后一块拼图。
- **关键源码**:kernel `pivot_root` / `chroot` 系统调用([fs/namespace.c](https://github.com/torvalds/linux/blob/master/fs/namespace.c));runc 的 rootfs 准备([libcontainer/rootfs_linux.go](https://github.com/opencontainers/runc/blob/main/libcontainer/rootfs_linux.go))。

#### 第 5 章 · 联合文件系统 overlayfs:分层镜像为什么这么设计

- **核心问题**:一个几百 MB 的镜像,凭什么能秒级启动、还能几十个容器共用大部分内容不占空间?
- **要讲清的"为什么"**:
  - 为什么镜像要**分层**(每次改动一层,公共层复用)——空间和传输的巨大节省。
  - overlayfs 的 lower / upper / merged 三层怎么叠加,只读层怎么共享。
  - **copy-up(写时复制)**:为什么容器改文件时不会污染镜像(改的是 upper 层副本)。
- **关键源码**:kernel overlayfs([fs/overlayfs](https://github.com/torvalds/linux/tree/master/fs/overlayfs));runc 的 overlay 挂载([libcontainer/mount_linux.go](https://github.com/opencontainers/runc/blob/main/libcontainer/mount_linux.go))。

#### 第 6 章 · 动手:不用 docker,手搓一个容器

- **核心问题**:剥掉 docker 所有包装,"容器"的最小实现到底有多简单?
- **要讲清的"为什么"**:
  - 用 `clone(CLONE_NEWPID|NEWNS|NEWNET|...)` 创进程 + 写 cgroup 文件 + `pivot_root` + `exec`,几十行 C/Go 就能造个能跑的容器。
  - 这证明了"容器没有任何神秘力量,它就是这几个系统调用的组合"。
  - 顺带理解 docker 到底替你省了哪些麻烦(镜像、网络、CLI、生命周期)。
- **关键源码**:本章是动手章,自己写最小实现;对照 runc 的 `standard_init_linux.go` 看完整版。

### 第 2 篇 · 镜像与运行时标准

#### 第 7 章 · 镜像的本质:layer + manifest + config

- **核心问题**:一个 image 文件,拆开到底由哪几部分组成?为什么是这样切?
- **要讲清的"为什么"**:
  - 镜像 = 一堆 layer(每层一组文件 diff)+ manifest(层清单)+ config(启动参数、环境变量)。
  - 为什么 config 要和 layer 分开(同样的层,不同启动配置 = 不同镜像)。
  - layer 内容寻址(每层用内容的 hash 命名)为什么能天然去重。
- **关键源码**:OCI image-spec;containerd 的镜像解构([core/mount](https://github.com/containerd/containerd)、`images/...`)。

#### 第 8 章 · OCI 标准:为什么要有运行时标准

- **核心问题**:为什么 Docker 要把自己拆成"镜像格式标准 + 运行时标准"两套公开规范?
- **要讲清的"为什么"**:
  - 没标准会怎样(厂商锁定:你的镜像只能 docker 跑)。
  - OCI image-spec(镜像长什么样)和 runtime-spec(怎么把镜像跑成容器)各管什么。
  - low-level runtime(runc)vs high-level runtime(containerd)的分层,为什么这样切。
- **关键源码**:[OCI specs 仓库](https://github.com/opencontainers/runtime-spec);runc 的 `create`/`start` 命令对应 runtime-spec。

#### 第 9 章 · runc:真正把容器跑起来的那个程序

- **核心问题**:`docker run` 拆到底,是哪个程序真正把容器"砌"出来的?它干了哪几步?
- **要讲清的"为什么"**:
  - runc 的职责:读 runtime-spec → 建 namespace → 设 cgroup → pivot_root → exec 容器进程。
  - `runc create` 和 `runc start` 为什么分开(先建好沙箱,再启动业务进程)。
  - runc 的双进程模型(parent 用 clone 建沙箱,child 是容器进程)为什么这么设计。
- **关键源码**:runc 的初始化主流程([libcontainer/container_linux.go](https://github.com/opencontainers/runc/blob/main/libcontainer/container_linux.go)、[standard_init_linux.go](https://github.com/opencontainers/runc/blob/main/libcontainer/standard_init_linux.go)、[nsenter/nsexec.c](https://github.com/opencontainers/runc/blob/main/libcontainer/nsenter/nsexec.c))。

### 第 3 篇 · docker 与 containerd

#### 第 10 章 · containerd:高层运行时管什么

- **核心问题**:runc 已经能跑容器了,为什么上面还要一层 containerd?
- **要讲清的"为什么"**:
  - runc 只管"把单个容器跑起来",不管镜像拉取、存储、生命周期管理、快照——这些是 containerd 的活。
  - containerd 的几个核心 daemon(containerd、containerd-shim),shim 为什么存在(让容器不绑死在 daemon 上)。
  - CRI(Container Runtime Interface):k8s 怎么通过 containerd 跑容器。
- **关键源码**:containerd 的架构([cmd/containerd](https://github.com/containerd/containerd)、`runtime/v2`)、shim 机制。

#### 第 11 章 · docker:让所有人都会用容器

- **核心问题**:`docker run nginx` 这一串简单命令背后,是怎样一长串调用链?docker 相比直接用 containerd 多做了什么?
- **要讲清的"为什么"**:
  - docker CLI → dockerd → containerd → runc 的完整链路,每一层各加了什么价值。
  - 为什么 docker 把体验做到了"一条命令开箱即用"(镜像仓库、网络、卷、compose)。
  - docker 的历史包袱(daemon 模式、root 权限)和它为什么逐步把底层让给 containerd。
- **关键源码**:docker / moby 的 [daemon](https://github.com/moby/moby) 入口、CLI 到 daemon 的 gRPC。

### 第 4 篇 · 容器网络:集装箱之间的转运通道

#### 第 12 章 · 容器网络基础:veth / bridge / iptables

- **核心问题**:容器怎么有自己的 IP 和网卡?它和宿主、和其他容器怎么通信?
- **要讲清的"为什么"**:
  - network namespace 给容器独立的协议栈/IP/路由表——但它默认是"孤岛"。
  - veth pair(一对虚拟网线)怎么把容器接到宿主,bridge(虚拟交换机)怎么把多个容器连成局域网。
  - iptables NAT 怎么让容器访问外网、怎么把宿主端口映射进容器。
- **关键源码**:kernel veth([drivers/net/veth.c](https://github.com/torvalds/linux/blob/master/drivers/net/veth.c))、bridge([net/bridge](https://github.com/torvalds/linux/tree/master/net/bridge));runc 配网络([libcontainer/network_linux.go](https://github.com/opencontainers/runc/blob/main/libcontainer/network_linux.go))。

#### 第 13 章 · CNI:容器网络接口

- **核心问题**:不同集群的网络需求千差万别(二层、三层、overlay、底层 SDN),怎么做成可插拔?
- **要讲清的"为什么"**:
  - 为什么网络要标准化成 CNI 插件接口(运行时不关心你用 calico 还是 flannel)。
  - CNI 插件干的就几件事:给容器分配 IP、配路由、装策略。
  - bridge(简单)/ calico(BGP 路由)/ flannel(overlay 隧道)三种典型方案的取舍。
- **关键源码**:[CNI 规范](https://github.com/containernetworking/cni);典型插件 [containernetworking/plugins](https://github.com/containernetworking/plugins)。

### 第 5 篇 · 从单机到集群:为什么必须要有 k8s

#### 第 14 章 · 第一性原理:编排为什么必须存在

- **核心问题**:有了 docker 能跑容器,为什么还需要 k8s 这么一整套东西?它到底解决了 docker 解决不了的什么问题?
- **要讲清的"为什么"**:
  - 单机 docker 的四个不够:故障自愈(容器挂了谁拉起)、扩缩容、跨机调度、服务发现。
  - 为什么传统的"脚本 + 监控 + 自动重启"方案撑不住几万个容器。
  - 为什么答案是"**声明期望状态 + 控制器自动调谐**"这套全新范式,而不是"写更多脚本"。
- **本章是 k8s 半本的定调章,概念为主,代码点到为止。**

#### 第 15 章 · k8s 架构:控制平面与节点

- **核心问题**:k8s 这套"航运调度中心",内部到底由哪些部件组成?它们怎么分工?
- **要讲清的"为什么"**:
  - 为什么是"控制平面 + 工作节点"的 master-worker 架构(指挥和干活分开)。
  - apiserver 为什么是**唯一入口**,etcd 为什么是**唯一真相源**(所有部件都通过 apiserver 读写状态)。
  - scheduler / controller-manager / kubelet / kube-proxy 各自一句句话能说清的职责。
  - 为什么一切都是 watch + list(异步事件驱动),而不是轮询。
- **关键源码**:k8s 核心组件 [kubernetes/cmd/kube-apiserver](https://github.com/kubernetes/kubernetes/tree/master/cmd/kube-apiserver)、`cmd/kubelet`、`staging/src/k8s.io/client-go`(informer/watch 机制)。

### 第 6 篇 · k8s 核心抽象

#### 第 16 章 · Pod:为什么不是直接调度容器

- **核心问题**:k8s 明明是管容器的,为什么它的最小调度单位不是容器、而是 Pod?
- **要讲清的"为什么"**:
  - 为什么要把"几个必须紧耦合的容器"(主进程 + sidecar,如 web + 日志收集)绑成一组一起调度。
  - Pod 里的容器共享什么(网络 namespace、存储卷),不共享什么(进程、文件系统)。
  - 为什么 Pod 这个抽象比裸容器更适合编排(原子调度单位)。
- **关键源码**:kubelet 的 Pod 管理([pkg/kubelet](https://github.com/kubernetes/kubernetes/tree/master/pkg/kubelet));Pod 的 pause 容器为什么存在(占住共享的 network namespace)。

#### 第 17 章 · 声明式 API 与 reconcile:k8s 的灵魂

- **核心问题**:为什么 k8s 是"声明我想要 3 个副本"而不是"命令你去启动 3 个"?这套范式为什么这么强大?
- **要讲清的"为什么"**:
  - 命令式(脚本)vs 声明式(期望状态)的本质区别:声明式天然幂等、自愈、可收敛。
  - controller 的 **watch-reconcile 循环**:盯着实际状态,不停把它往期望状态拉。这是 k8s 一切魔力的根源。
  - 为什么这个模式能轻松扩展(写个新 controller 就能管一种新资源——CRD)。
- **关键源码**:k8s controller-runtime([staging/src/k8s.io/client-go/tools/cache](https://github.com/kubernetes/kubernetes/tree/master/staging/src/k8s.io/client-go/tools/cache) 的 informer)、典型 controller([pkg/controller/deployment](https://github.com/kubernetes/kubernetes/tree/master/pkg/controller/deployment))。

#### 第 18 章 · 调度器:Pod 放到哪个节点

- **核心问题**:集群里几百个节点,一个新的 Pod 该放哪个?谁决定、怎么决定?
- **要讲清的"为什么"**:
  - 调度的两阶段:**过滤(filter,哪些节点能放)+ 打分(score,哪个最合适)**。
  - 为什么调度器是**无状态、可预测**的(同样的集群状态,同样的调度结果)——便于调试和扩展。
  - 资源请求/限制(requests/limits)、亲和性/反亲和性、污点/容忍这些调度维度各解决什么问题。
- **关键源码**:k8s 调度器([pkg/scheduler](https://github.com/kubernetes/kubernetes/tree/master/pkg/scheduler))的 filter/score 插件。

#### 第 19 章 · kubelet:节点上的代理人

- **核心问题**:控制平面说"这个节点上要有这个 Pod",谁来真正把它跑起来?节点状态又怎么报回去?
- **要讲清的"为什么"**:
  - kubelet 是每个节点上的"驻港经理":把期望的 Pod 翻译成对 containerd 的 CRI 调用,真正启动容器。
  - 它的**反向同步**:不停把节点和 Pod 的真实状态上报给 apiserver(让 reconcile 有依据)。
  - 为什么 kubelet 自己也是个"小控制器"(POD 期望 vs 容器实际,自己 reconcile)。
- **关键源码**:kubelet 主循环([pkg/kubelet/kubelet.go](https://github.com/kubernetes/kubernetes/blob/master/pkg/kubelet/kubelet.go))、CRI 调用([pkg/kubelet/cri](https://github.com/kubernetes/kubernetes/tree/master/pkg/kubelet/cri))。

#### 第 20 章 · Service 与 kube-proxy:服务发现与负载均衡

- **核心问题**:Pod 是会死会生的(IP 一直在变),别的 Pod 怎么稳定地找到它?
- **要讲清的"为什么"**:
  - 为什么需要一个**稳定门牌(Service)**盖在一组会变的 Pod IP 之上。
  - kube-proxy 怎么用 iptables/ipvs 规则,把对 Service IP 的访问负载均衡到后端 Pod。
  - Service 发现的两种方式(DNS 名、环境变量)各适合什么场景。
- **关键源码**:kube-proxy([pkg/proxy](https://github.com/kubernetes/kubernetes/tree/master/pkg/proxy))的 iptables/ipvs 模式。

### 第 7 篇 · 进阶与边界

#### 第 21 章 · 存储:Volume / PV / PVC

- **核心问题**:容器文件系统是临时的(容器一死数据就没了),怎么给容器接持久化存储?
- **要讲清的"为什么"**:
  - 为什么不直接挂个目录(要解耦:用的人不知道存储在哪、给存储的人不知道谁用)。
  - PVC(我要一块存储)/ PV(我有一块存储)/ StorageClass(自动分配)的三层抽象为什么这样设计。
  - CSI(Container Storage Interface):存储也做成可插拔标准。
- **关键源码**:k8s PV/PVC 控制器([pkg/controller/volume](https://github.com/kubernetes/kubernetes/tree/master/pkg/controller/volume));CSI 规范与实现。

#### 第 22 章 · 安全:Seccomp / AppArmor / capabilities / 容器逃逸

- **核心问题**:容器默认常常以 root 跑,这有多危险?怎么在不影响功能的前提下收紧它的权限?
- **要讲清的"为什么"**:
  - 为什么"容器隔离"不等于"安全"(共享内核,逃逸风险)。
  - capabilities(把 root 的特权拆成几十项,只给需要的)、seccomp(白名单系统调用)、AppArmor/SELinux(强制访问控制)各收紧哪一层。
  - 几个经典的容器逃逸路径(privileged 模式、挂载 docker.sock、内核漏洞)。
- **关键源码**:kernel seccomp([kernel/seccomp.c](https://github.com/torvalds/linux/blob/master/kernel/seccomp.c))、capabilities([kernel/cred.c](https://github.com/torvalds/linux/blob/master/kernel/cred.c));runc 的安全配置([seccomp/](https://github.com/opencontainers/runc/tree/main/libcontainer/seccomp)、[capabilities/](https://github.com/opencontainers/runc/tree/main/capabilities))。

#### 第 23 章 · 容器的边界:它隔离不了什么

- **核心问题**:共享内核换来了轻量,代价是什么?什么时候容器隔离不够、必须上虚拟机?
- **要讲清的"为什么"**:
  - 共享内核的三类风险:内核漏洞全军覆没、/proc 和 /sys 的信息泄漏、资源隔离的盲区(cache、IRQ)。
  - Kata Containers / gVisor 为什么存在(在容器和 VM 之间找平衡:容器体验 + VM 隔离强度)。
  - 安全沙箱的谱系:裸容器 → gVisor(用户态内核)→ Kata(轻量 VM)→ 传统 VM。
- **本章是全书"诚实交代容器局限"的收尾章,呼应第 1 章的"共享内核"。**
