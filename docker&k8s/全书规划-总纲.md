# 《容器深入浅出:从一个进程到几万个进程的调度》—— 全书总纲

> 一本写给"天天用 docker / k8s,却想搞懂它们底下到底在干什么、为什么这么设计"的人的小书。
>
> 本文是全书的**总纲**:定主线、定比喻、定分篇分章、定源码策略、定写作约定。
> 它是《Linux 内核深入浅出》系列的**姊妹篇**——容器没有发明任何新硬件能力,它只是把内核**已有的**能力(namespace 造幻觉、cgroup 管资源)组合、封装、产品化。所以本书的根,扎在内核系列里。

---

## 一、这本书的定位

### 目标读者

会写代码、天天 `docker run` / `kubectl apply`,但没读过 runc / kubernetes 源码;**想搞懂"容器到底是什么、为什么这么设计",而不是"命令怎么敲"**。和内核系列是同一个读者。

### 全书一句话主旨

> **一个普通的应用,是怎么一步步被打包进标准集装箱(镜像)、装上共享内核的货轮(容器)、最后由航运调度中心(k8s)在几百艘船上调度成几万个的。**

每个章节,都是这段旅程上的一个**驿站**。读完一本书,读者不是"背下了一堆命令和 YAML 字段",而是**亲眼看着一个裸进程被一步步隔离、打包、调度**,在每个驿站都搞懂了"这一步为什么必须存在、它解决了什么不解决就不行的问题"。

### 为什么用"旅程"当骨架(而不是按组件平铺)

docker / containerd / runc / kubernetes / CNI / CSI……这套生态组件之多、名词之密,足以把人淹没。如果按"镜像篇、网络篇、调度篇……"百科式平铺,读者会觉得每个组件都是孤岛,而且不知道**为什么要学这个**。

用"从一个进程到几万个进程的调度"这条旅程当骨架,每个组件都是"为了让旅程继续下去而必须出现的角色"——读者永远知道**我在哪、为什么需要下一站**。这正是内核系列"从上电到 `ls`"那条旅程线在容器世界的复刻。

## 二、与内核系列的关系:本书的根,扎在那里

这是本书最重要的一个认知,也是它的"第一性原理":

> **容器不是一个"新东西"。它是一个普通的进程,只是被套上了内核早已准备好的两件外套——namespace(隔离)和 cgroup(限资源)。**

内核系列讲过市政府(内核)给每个市民(进程)造的幻觉:独占 CPU、独占内存、一切是文件。**namespace 和 cgroup,就是内核用来"再造幻觉"和"管资源"的同一套能力,换个姿势使用而已:**

- **namespace**——内核让"两个进程的 100 号门牌指向不同物理房间"的隔离能力,扩展到"两个进程看到完全不同的进程表 / 网卡 / 挂载点 / 主机名"。**造幻觉的同一只手。**
- **cgroup**——内核分配物理页、CPU 时间的那套资源管理能力,加上"按组计量、按组限额"。**管资源的同一只手。**

所以本书的每一章,你都会发现:**它底下都是某个内核机制**。docker / k8s 的全部精妙,在于**怎么把这些内核能力组合、封装、产品化、再规模化调度**。理解了这一点,你就不会被那堆名词吓住——它们都是内核能力穿上了不同马甲。

> **衔接建议**:读者最好先读过内核系列的**进程篇(进程/fork/调度)**和**内存篇的 memcg 章节**。没有也不致命,本书在用到时会一句话点出"这底下是内核的 X 机制",需要深挖时回内核系列查。

## 三、全书主比喻:集装箱与航运物流

"container"这个词,本意就是**集装箱**。Docker 项目正是以此命名(它的 logo 是一条鲸鱼背着一摞集装箱)。本书直接用回这个最贴切的本意——它和容器技术的本质严丝合缝:

> **把应用(货物)打包进标准尺寸的集装箱(镜像),装上共享同一套动力系统的货轮(宿主机内核),再由航运调度中心(k8s)把成千上万个集装箱调度到几百艘船上。**

这套比喻之所以最适合本书,是因为它能一击讲透容器存在的**第一性原理**——为什么是"集装箱"而不是"每件货物配一艘独立的船":

> **虚拟机**:每个应用配一整艘独立的货轮——连船体、轮机、船员、船长都独立一套。隔离最彻底,但贵得离谱(每艘船都是一整个完整的操作系统,启动慢、占资源)。
> **容器**:所有应用装在**同一艘货轮**上,共享船的动力系统(**内核**),只用标准铁皮箱(**namespace + cgroup**)把货物彼此隔开。轻、快、省——代价是共享内核(第 23 章专门讲这个代价)。

**"共享同一艘船的动力系统" = "所有容器共享同一个内核"**,正是容器区别于虚拟机的命门。这个对应,是全书第一性原理的基石。

下面这张表是**全书术语统一的总开关**:

### 全书的二分法

本书的二分法是:

> **打包与隔离(把一个应用装箱、隔绝、限好量) vs 调度与编排(把几万个这样的箱子运到几百艘船上、保活、扩缩容)。**

- **docker 这半本**讲**打包与隔离**:怎么把货物(应用)装进标准集装箱(镜像),怎么用铁皮箱(namespace)隔绝、用配额(cgroup)限量,怎么让装箱和运输一条龙变好用。
- **k8s 这半本**讲**调度与编排**:当这种集装箱多到几万个、散布在几百艘货轮(节点)上,怎么统一调度、保活、扩缩容、让货物互相找得到。

任何时候迷路,回到这个二分法:"这章是在**装箱隔离一个应用**,还是在**调度编排一群箱子**?"

## 四、全书结构(分篇 + 分章)

> 每章只给**章名 + 核心问题**(一句话)。细节在 `00-全书目录.md` 里展开。
> 主线是一条旅程:**裸进程 → 关进隔离世界 → 打包成镜像 → 在集群上调度**。

### 第 0 篇 · 开篇:容器到底是什么(定调,1 章)

- **导言 · 一篇看懂容器全貌**:用"集装箱 / 航运物流"比喻,把 namespace→cgroup→rootfs→镜像→docker→k8s 一口气串一遍,对标内核系列的"导言·全局直白解释"。
- **第 1 章 · 第一性原理:容器不是虚拟机**:裸进程有什么不行 → 为什么 VM 这条路重 → 容器凭什么又轻又隔离(共享内核 + namespace/cgroup)→ 它本质上就是个普通进程。**【样本章节,见后】**

### 第 1 篇 · 容器的三块基石:内核能力的组合(5 章)

> 这一篇是全书地基。讲清楚"容器"拆到底,就是三件内核已有的事。

- **第 2 章 · namespace:集装箱的铁皮舱壁**——为什么需要 8 种 namespace(光隔离进程表不够),每种隔离了什么,clone 的 `CLONE_NEW*` 怎么落地,`/proc/<pid>/ns/` 怎么观测。
- **第 3 章 · cgroup:给进程上配额**——为什么光隔离还不够(还得限资源),cgroup v1 vs v2,cpu/memory/IO 子系统怎么计量和限额,为什么有"局部 OOM"。
- **第 4 章 · rootfs 与 pivot_root:换个根**——为什么容器要"换根目录"(不换则能看到宿主文件),pivot_root vs chroot 的区别(为什么 chroot 不安全)。
- **第 5 章 · 联合文件系统 overlayfs:分层镜像为什么这么设计**——为什么镜像要分层(复用),overlay 的 lower/upper/merged 三层怎么叠加,写时复制(copy-up)怎么省空间。
- **第 6 章 · 动手:不用 docker,手搓一个容器**——用 `clone(CLONE_NEW*)` + cgroup + pivot_root 从零造一个能跑的"容器",证明它没有任何神秘力量。

### 第 2 篇 · 镜像与运行时标准(3 章)

> 把"集装箱"变成可搬运、可复用的标准件。

- **第 7 章 · 镜像的本质:layer + manifest + config**——一个 image 到底由哪几部分组成,为什么是这样切分的,layer 和 overlayfs 的关系。
- **第 8 章 · OCI 标准:为什么要有运行时标准**——image-spec / runtime-spec 是什么,为什么 Docker 要把底层拆出来标准化(避免厂商锁定),low-level vs high-level runtime 的分层。
- **第 9 章 · runc:真正把容器跑起来的那个程序**——runc 干了哪几件事(创 namespace、设 cgroup、pivot_root、exec),它的源码怎么对应这些步骤。

### 第 3 篇 · docker 与 containerd:把一切变得好用(2 章)

- **第 10 章 · containerd:高层运行时管什么**——为什么 runc 之上还要一层(containerd 管镜像拉取、生命周期、快照),它和 runc 的分工。
- **第 11 章 · docker:让所有人都会用容器**——docker CLI / daemon / 分布式存储,以及为什么"docker run 一条命令"背后是一长串调用链。

### 第 4 篇 · 容器网络:小区之间的路(2 章)

- **第 12 章 · 容器网络基础:veth / bridge / iptables**——容器怎么有自己的网卡(namespace net),怎么和宿主、和其他容器通信(veth pair + bridge),怎么 NAT 出网(iptables)。
- **第 13 章 · CNI:容器网络接口**——为什么网络要做成插件标准(CNI),bridge / calico / flannel 这些插件到底干了什么。

### 第 5 篇 · 从单机到集群:为什么必须要有 k8s(2 章)

- **第 6 篇的入口,旅程的转折点:从"一个小区"到"几万个小区"。**

- **第 14 章 · 第一性原理:编排为什么必须存在**——单机 docker 不够用了吗(故障、扩容、调度、服务发现),为什么需要"声明式 + 自动 reconcile"这套全新范式。
- **第 15 章 · k8s 架构:控制平面与节点**——master(apiserver/etcd/scheduler/controller-manager)和 node(kubelet/kube-proxy)怎么分工,为什么 apiserver 是唯一入口、etcd 是唯一真相源。

### 第 6 篇 · k8s 核心抽象(5 章)

> 这一篇是 k8s 的灵魂。每个抽象都先回答"为什么不直接这么做、非要发明这个概念"。

- **第 16 章 · Pod:为什么不是直接调度容器**——为什么 k8s 不直接管容器、非要套一层 Pod(共享网络/存储的紧耦合进程组),Pod 和容器的关系。
- **第 17 章 · 声明式 API 与 reconcile:k8s 的灵魂**——为什么是"声明期望状态"而不是"命令执行",controller 的 watch-reconcile 循环为什么这么强大,这是理解 k8s 一切的关键。
- **第 18 章 · 调度器:Pod 放到哪个节点**——调度的两阶段(filter + score),为什么调度是无状态的、可预测的。
- **第 19 章 · kubelet:节点上的代理人**——kubelet 怎么把"期望的 Pod"变成"真实跑着的容器",它和 containerd/CRI 的关系,为什么它是"反向同步"(节点状态上报)。
- **第 20 章 · Service 与 kube-proxy:服务发现与负载均衡**——为什么 Pod IP 会变、需要一个稳定门牌(Service),kube-proxy 怎么用 iptables/ipvs 实现负载均衡。

### 第 7 篇 · 进阶与边界(3 章)

- **第 21 章 · 存储:Volume / PV / PVC**——容器文件系统是临时的(死了就没了),怎么接持久化存储,PVC/PV 的"消费-提供"解耦为什么这么设计。
- **第 22 章 · 安全:Seccomp / AppArmor / capabilities / 容器逃逸**——容器默认以 root 跑的危险,capabilities 怎么裁权,seccomp 怎么限制系统调用。
- **第 23 章 · 容器的边界:它隔离不了什么**——共享内核的代价(kernel exploit 全军覆没、/proc 和 /sys 的泄漏、资源隔离不彻底),什么时候必须上 VM/Kata/gVisor。

### 附录

- **A · 全景脉络与设计哲学**:把全书"为什么"收束成几条哲学(组合优于发明、声明式优于命令式、分层标准解耦、共享内核换轻量)。
- **B · 源码阅读路线与工具**:runc / containerd / kubernetes 源码怎么看,zread 在线读法,以及 `ctr` / `crictl` / `kubectl debug` / `nsenter` 等观测工具。

## 五、源码策略(已落地)

容器源码分散在多个 GitHub 仓库,体量悬殊(runc 小、kernel/k8s 极大)。本书采用**混合方案**:核心仓库 shallow clone 到**系列根目录**(文章用相对路径 `../<repo>/...` 引用,可离线点击跳转,与内核系列体验一致);超大仓库用 GitHub 链接 + zread 在线核实。

### 本地源码(已 clone,文章用相对路径引用)

- **runc**(容器运行时核心,~21MB):`../runc/` —— 第 1~3 篇主角。引用形如 `[描述](../runc/libcontainer/configs/namespaces_syscall.go#L11-L33)`。
- **containerd**(高层运行时):`../containerd/` —— 第 3 篇主角。引用形如 `[描述](../containerd/<path>#L行号)`。

### 在线源码(用 GitHub 链接 + zread/WebFetch 核实行号)

- **kernel**(namespace/cgroup/overlayfs 定义):[torvalds/linux](https://github.com/torvalds/linux) —— 第 1 篇基石引用。仓库太大(~1.5GB),不本地 clone。
- **kubernetes**(第 5~7 篇主角):[kubernetes/kubernetes](https://github.com/kubernetes/kubernetes) —— 仓库极大(~1GB+),用 zread 在线读 `pkg/`、`staging/src/k8s.io/` 关键文件。

### 引用格式与铁律

- 本地仓库:`[描述](../runc/路径#L行号)`,区间 `#L起-L止`,单行 `#L20`。
- 在线仓库:`[描述](https://github.com/<org>/<repo>/blob/<ref>/路径#L行号)`。
- **严禁凭记忆编造行号或代码**:本地仓库用 Grep/Read 核实,在线仓库用 zread/WebFetch 核实。不确定精确行号时,宁可只标文件不标行号。

## 六、写作约定(沿用内核系列,微调适配容器)

1. **"为什么"永远先于"怎么做"**——全书最高原则。每个设计点先回答"不这样会怎样(痛点/灾难)",再讲"所以这样设计",最后才讲实现。
2. **大白话 + 比喻**——统一用"集装箱 / 航运物流"比喻(见上表),新概念先给航运比喻再落到技术。
3. **小白友好**——不假设读者懂容器术语。任何术语首次出现一句话解释。避免堆砌黑话。
4. **关键路径配真实源码**——runc / kubernetes / kernel 的真实片段,标 GitHub 链接 + 行号,逐段讲解。概念章代码点到为止,实现章配码。
5. **回扣全局**——每章结尾回到"隔离 vs 编排"二分法,说明本章服务哪一侧,衔接前后章节。
6. **每章标准结构**:① 章首"前置导航 + 核心问题 + 读完你会明白";② 正文若干小节(四段式:提问→不这样会怎样→所以这样→代码佐证);③ 关键源码精读(正文后、小结前);④ 章末小结(比喻回顾 + 回扣主线 + 深入指引)。难度高的章节加"逃生阀"。
7. **语言**:中文,技术名词保留英文(namespace、cgroup、Pod、reconcile 等)。
8. **文件命名**:`P{篇号}-{章号}-{标题}.md`。如开篇篇第 1 章 = `P0-01-…`,第 1 篇(基石)第 2 章 = `P1-02-…`。

1. **样本章节选了第 1 章(第一性原理·容器不是虚拟机)**——定调章,对标内核 P0-01。如果你想先看技术核心章(如第 2 章 namespace 或第 6 章手搓容器)作为样本,我可以换。
2. **源码策略默认"GitHub 链接"**(零下载、可验证)。如果你希望像内核系列那样本地有源码树可离线点跳,我可以 shallow clone runc+containerd。
3. **篇幅/深度**:默认对标内核系列(每章 ~6000~10000 字,关键路径配源码)。如果想要更精简或更深入,可调。
4. **k8s 的占比**:目前 docker(第 0~4 篇)和 k8s(第 5~7 篇)大致平衡。如果你的目标读者更偏其中一边,可重新分配章节比重。
5. **进阶篇取舍**:第 7 篇(存储/安全/边界)和附录可按精力决定是否纳入本书还是留"卷二"。
