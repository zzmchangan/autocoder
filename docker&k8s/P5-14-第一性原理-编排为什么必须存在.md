# 第 14 章 · 第一性原理:编排为什么必须存在

> **前置**:你需要先读完 docker 这半本(第 0~4 篇,特别是 [第 11 章《docker》](P2-09-runc-真正把容器跑起来的那个程序.md) 和 [第 10 章《containerd》](P3-10-containerd-高层运行时管什么.md))。前半本的旅程告诉我们:一个普通的应用,已经能被装进标准集装箱(镜像)、装上共享内核的货轮(容器),在一台机器上又轻又安全地跑几十个互不干扰的实例。**这一章,旅程在这里来了一个急转弯**:从一个小区(单机港口),跨到几万个小区(全球航运)。我们要问一个全新的问题——**docker 这套港口,够用了吗?如果不够,差在哪?差的那一块为什么非得是一整套全新的东西(k8s),而不是再多写几个脚本?**

> **核心问题**:有了 docker 能跑容器,为什么还需要 kubernetes 这么一整套东西?它到底解决了 docker 解决不了的什么问题?
>
> 这是 k8s 半本的**定调章**。这一章我们不碰任何 k8s 内部结构(那是下一章的事),只回答一个根本问题:**编排(orchestration)这件事,凭什么必须存在?** 我们不堆名词、不甩 YAML,而是把"单机 docker 在四个地方撑不住"一个一个掰开,再把"为什么传统脚本也救不了"讲透,最后给出一套全新范式的直觉——**声明期望状态 + 控制器自动调谐(reconcile)**。读完你不会觉得 k8s 是花架子,而会真心认同:不发明编排,这事就过不去。
>
> **读完本章你会明白**:
> - 单机 docker 在**故障自愈、扩缩容、跨机调度、服务发现**这四件事上为什么力不从心——不是"它没做好",而是"它根本不该管"。
> - 为什么"脚本 + 监控 + 自动重启"这套老办法,**在容器数量从几百爬到几万时必然崩溃**——脚本天生不幂等、不可收敛、状态散落、无法扩展。
> - 为什么唯一的出路是一套**全新范式**:你只声明"期望状态"(我要 3 个副本),由一个**自动控制器**不停把"实际状态"往"期望状态"拉(reconcile,调谐)——而不是"再写一个更聪明的脚本"。
> - **命令式(imperative)和声明式(declarative)的本质区别**,以及为什么这个区别是 k8s 一切魔力的根源(为第 17 章深挖埋下伏笔)。

> **如果一读觉得太难**:先只记住三件事——① docker 是"一个港口",k8s 是"全球调度中心",港口管不好几百艘船;② "脚本重启"这条路在几万个容器时必然崩,因为它不幂等、不自愈;③ k8s 的核心范式就一句话:**你说"我要 N 个",它盯着实际有几个,少了补、多了杀、挂了重启**。这三句话钉死,后面慢慢往里填。

## 章首·一句话点破

如果你对 k8s 的印象是"一个更复杂的 docker""一个能跑在多台机器上的 docker""一堆花里胡哨的 YAML 字段",请把这种印象先放一边。这是理解 k8s 时最容易绊倒的一个误解,它会把你的好奇心引向"这些字段怎么填",而不是真正重要的问题。

这一章要做的第一件事,是把这个误解连根拔起:

> **k8s 不是"更大号的 docker"。它和 docker 解决的根本不是同一个问题。docker 解决的是"怎么把一个应用装箱、隔离、跑起来";k8s 解决的是"几万个这样的箱子散布在几百台机器上,怎么统一管、怎么保活、怎么扩缩容、怎么让它们互相找得到"。两者是接力,不是替代。**

为什么这么说?我们从一件最朴素的事情开始:**当容器数量从几十个变成几万个,你手里那些好用的工具,为什么会突然全部失灵。**

## 一、转折点:从一个小区,到几万个小区

回忆 docker 半本走过的旅程。到现在为止,你已经能在**一台机器**上做到这些事:

- 把应用装进标准镜像(集装箱),秒级启动几十个互不干扰的实例;
- 用 namespace 隔离、用 cgroup 限量,每个容器都又轻又安全;
- 用 overlayfs 让几十个容器共用大部分内容不占空间;
- 用 veth/bridge/CNI 给容器之间修路,让它们能互相通信;
- 用 docker/containerd 把这一切包装成"一条命令开箱即用"。

一句话:**你能在"一个港口"里,高效地运营几十个集装箱。**

> **比喻**:此时你手里有的是**一个运转良好的港口**。港口里有堆场(镜像仓库)、有港口管理公司(containerd)、有起重机(runc)、有集装箱之间的转运通道(网络)。这个港口,我们把它想象成**一个小区**——一艘货轮(一台宿主机)上的一切,它都能管好。

可是,真实的生产系统长这样:

- 一个中等规模的互联网公司,跑着 **3000~5000 个容器**;
- 一个大厂,跑着 **几万到几十万个容器**;
- 这些容器散布在 **几百到几千台机器**上,机房可能横跨两三个城市;
- 每天有几百次部署、几千次故障、无数次扩缩容。

> **比喻**:这已经不是"一个小区",而是**几万个小区组成的全球航运网络**。几百艘货轮(节点)在海上跑,每艘船上有几十上百个集装箱,每天有船在沉、有新货在加、有港口在升级、有航线在变。**一个港口的管理能力,在这里完全不够用——它管好一艘船没问题,但它管不了几百艘船。**

那么,具体差在哪?我们把"单机 docker 撑不住"的事,一件一件掰开。

## 二、单机 docker 的四个不够

这是全章的核心论证。我们用四个**真实的灾难场景**,把 docker 在单机里"做不了、不该做、也做不好"的四件事,一个一个钉死。

### 不够一:故障自愈——一个容器挂了,谁拉起来?

**场景**:你的 web 服务跑在一个容器里。凌晨 3 点,应用的一个内存泄漏让进程 OOM 退出了。然后呢?

如果你用的是 `docker run`(不配 restart policy),答案是:**没人拉它**。容器进程退出了,容器就进入了 Exited 状态,在那里静静地躺着,直到第二天早上你被值班电话叫醒、ssh 上去、手动 `docker start`。

docker 后来加了 `--restart=always` 这个选项,能在容器退出时自动重启。这看起来解决了问题,但**只解决了最薄的一层**:

- 它能处理**进程级**的崩溃(进程退了,拉起来);
- 它**处理不了机器级**的故障。问一个致命的问题——**如果整台机器(货轮)的内核 panic 了,或者电源烧了,或者网络断了,上面 50 个容器全死了,谁来拉它们?**

  这台机器上的 docker daemon 自己也死了。**一个死人,没法给自己做心肺复苏。** 你指望它重启容器,是不可能的事。

- 它还处理不了**中间状态**:容器在跑,但应用 hang 住了(死锁、活锁、依赖的下游挂了)——`--restart=always` 只看进程是否退出,看不见"应用其实已经不工作了"。这种**活死(livelock)状态**,docker 是不知道的。

**不这样会怎样**:在没有编排的年月,这件事靠"人 + 监控"兜。监控发现"web 不响应了",打电话叫醒工程师,工程师 ssh 上去手动重启。一个容器这样能扛,十个还能扛,**几千个容器呢?** 全靠人盯,根本扛不住。

> **比喻**:这就像一艘货轮上的一个集装箱,自带一个"小发动机",发动机会自己重启。可如果**整艘船沉了**——船上所有发动机都跟着沉——小发动机再能自重启也没用。**你需要的是一个站在别的船上、能看见这艘船沉没的人,把集装箱搬到另一艘还活着的船上重新通电。** 这个人,港口(docker)里没有。

### 不够二:扩缩容——流量来了,谁加副本?

**场景**:你的电商网站平时跑 3 个 web 容器。双十一 0 点,流量翻了 20 倍,3 个容器 CPU 全红、请求堆积、用户开始报错。你必须在几分钟内把"3 个"扩到"30 个"。

如果是单机 docker,你会怎么做?

- **手动方案**:ssh 到 10 台机器,每台 `docker run` 3 个。听起来简单,可问题是——这 10 台机器哪台有空?哪台已经跑了别的服务、不能再加了?你需要**人脑排班表**,对着每台机器的资源占用现算。3 台机器还能算,**300 台呢?**
- **半自动方案**:写个脚本,根据监控告警,自动 ssh 到"某台机器"再 `docker run`。可脚本怎么知道哪台机器资源够?怎么知道这台机器上跑的是别的什么、会不会冲突?**脚本能扩容的前提,是你先手动维护一张"哪台机器空闲"的表——而这张表,在动态变化的集群里永远过时。**

更难的是**缩容**:双十一结束了,30 个副本要缩回 3 个。缩哪个?**杀掉的那 27 个里,有正在处理请求的吗?** 直接 docker kill,用户请求会被中断。你需要先"优雅下线"(标记不接新流量、等老请求处理完),再杀。这种细致活,`docker kill` 帮不了你。

**不这样会怎样**:扩容慢几分钟,就是几百万的损失;缩容粗暴,就是一片用户报错。在容器只有几十个、机器只有几台时,"人 + 脚本"勉强能撑;**容器过千、机器过百时,这件事靠人已经彻底撑不住**。

> **比喻**:这就像港口经理收到电话"今天货要加 10 倍",他得自己盘算:哪艘船还有空位?哪艘船正在卸别的货?怎么调度这批新货上船?**一个港口经理能管一个港口,但他管不了几百艘船组成的全球船队。** 你需要一个站在更高处、看得见所有船的**全球调度中心**。

### 不够三:跨机调度——一个新容器,该放哪台机器?

**场景**:你说"再启动一个 web 容器"。可这一句话背后藏着一连串决策:**这个容器该放哪台机器?**

要回答这个问题,你得同时考虑:

- **资源**:哪台机器还有足够的 CPU、内存、磁盘?
- **亲和性**:这个容器依赖的 Redis 在 node-3 上,把 web 也放 node-3 网络延迟更低?
- **反亲和性**:同一个 web 服务的 3 个副本,**不能全放在同一台机器上**——否则那台机器一挂,3 个副本全军覆没。怎么分散?
- **约束**:有些机器是 SSD、有些是机械盘;有些机器装了 GPU、有些没有;有些机器是给生产用的、有些是给测试用的。这个容器该放哪种?
- **健康度**:有些机器虽然在列表里,但磁盘快满了、网络抖动、已经超卖——这种机器不该再往里塞东西。

docker 在单机上**完全不操心这些**——它只看到"这一台机器"。可当机器有几百台时,**"这个容器放哪台"本身就是一个需要全局视野的决策**。docker 给不了这个视野。

**不这样会怎样**:在没有调度器的世界里,人们靠两种办法:① 人脑排班表(到几百台就崩了);② "随机丢",把容器随机扔到某台机器(资源利用极不均衡——有的机器撑死、有的饿死,故障域也没分散)。

> **比喻**:这就像一个新集装箱到了港口,得有人决定它上哪艘船。一个港口里,港口经理凭经验拍板就行;**可在几百艘船组成的船队里,这个决策需要全局视野**——哪艘船还装得下、哪艘船和哪艘船航线冲突、哪艘船正在维修。这个决策者,必须是站在所有船之上的**调度中心**。

### 不够四:服务发现——容器之间,怎么互相找到?

**场景**:你的 web 容器要访问数据库容器。你怎么写连接字符串?

在单机 docker 里,你可以用 docker 的内置 DNS——同一个 docker network 里的容器,可以用容器名互相找到。听起来挺好。可是放到多机集群里:

- 容器是**会死会生**的:web 容器挂了被重启,IP 变了;扩容了 3 个变 30 个,IP 全是新的;缩容了 30 个变 3 个,27 个 IP 消失了。**IP 一直在变。**
- 跨机通信:web 在 node-1,数据库在 node-3,这俩容器怎么通信?docker 的内置 DNS **只在单机 network 内有效**,跨机它管不了。你需要一个跨机的网络方案(flannel、calico...),但**这还不够**——就算网络通了,IP 还是在变。

如果你直接写"数据库的 IP 是 10.0.3.42",过几天数据库容器被重启,IP 变成 10.0.3.51,你的 web 全部连不上。**你需要一个稳定的"门牌",盖在一组会变的 IP 之上。**

**不这样会怎样**:在没有服务发现的世界里,人们靠两种办法:① 配置文件 + 人肉改 IP(每改一次,改一个集群);② DNS + 短 TTL(DNS 缓存过期前,流量还指向老 IP)。容器只有几个时,这俩都行;**容器上千、每天几百次 IP 变更时,这俩全崩**——配置文件改不过来,DNS 传播延迟让故障雪崩。

> **比喻**:这就像一个集装箱的运单号。**不管这箱货转到哪艘船、换了几次船,运单号不变**——你只认运单号,内部怎么转你不用管。如果没有运单号,你只能每次都问"那批货现在在哪艘船、哪个舱位",这在几百艘船的船队里完全没法追踪。**运单号,就是服务发现要给你的东西。**

### 四个不够的共性:它们都是"跨机的、动态的、需要全局视野的"

把四个不够摆在一起,你会发现一个共性:

> **这四件事,没有一件是"在一台机器上能解决的"。它们都需要一个站在所有机器之上、看得见全局、不被任何一台机器的生死绑住的"东西"。**

- 故障自愈:需要站在别的机器上,看见这台机器死了,把它的活搬到别处。
- 扩缩容:需要看得见所有机器的余量,才能决定往哪儿加、往哪儿减。
- 跨机调度:全局视野是它的核心输入。
- 服务发现:需要一个稳定的门牌,不被任何一台机器的生死绑定。

而 docker,**生来就是一个"单机港口"**——它的整个心智模型,都建立在"我在这一台机器上管好这些容器"。把单机 docker 拼到几百台机器上,这四个问题一个都解决不了。**这不是 docker 没做好,是它根本不该做这件事。**

那好,最直觉的下一步,是"在 docker 之上写点脚本"。这条路到底走不走得通?我们认真看看。

## 三、为什么"脚本 + 监控 + 自动重启"撑不住

面对上面四个不够,一个有经验的运维工程师会本能地给出一个解法:

> **写脚本啊!** 监控发现容器挂了,脚本自动 `docker start`;流量来了,脚本根据监控阈值自动 `docker run` 几个;写个 cron,定期巡检所有机器的容器清单。

这个解法看起来朴素可靠,而且**在容器数量很少时,它确实能跑**。事实上,docker 早期的 `--restart=always`、各种"运维自动化平台",本质上都是这个路子。**问题在于:这套路子有一个致命的天花板,容器数量过了几千就必然撞穿。** 撞穿的原因,不是脚本写得不够多、不够聪明,而是**脚本这种形式本身有四个根本性缺陷**。

### 缺陷一:脚本不幂等

**幂等(idempotent)** 的意思是:一个操作做一次和做一百次,效果一样。`chmod 755 a.sh` 是幂等的——你跑多少遍,文件权限都是 755。`echo "hello" >> a.log` 是**不幂等**的——你跑多少遍,文件就追加多少行 "hello"。

**脚本里的扩容操作,几乎全是不幂等的。** 比如:

```bash
# 一个朴素的"扩容"脚本:发现 web 容器不够,就再起 3 个
docker run -d web
docker run -d web
docker run -d web
```

这个脚本跑一次,起 3 个;跑两次,起 6 个;跑十次,起 30 个。**它不记得"我已经起过几个了"。** 如果你的监控告警触发了两次(网络抖动很容易让告警重复触发),你的容器数量就翻倍了——不是翻到期望的 3 个,而是翻到 6 个、9 个。

运维世界把这种脚本叫"**记得自己执行过命令、却不记得世界现在是什么样**"。**它只关心动作,不关心结果。** 在容器数量少时,你可以人工复核;数量一多,这种"动作正确、结果错误"的事故会让你疲于奔命。

### 缺陷二:脚本不可收敛

**收敛(converge)** 的意思是:不管系统现在处于什么状态,只要不再有新的扰动,系统都会自己慢慢走到"目标状态"。

脚本做不到这一点。考虑这个场景:

- 你期望:集群里有 **3 个** web 容器;
- 现实:集群里有 **5 个**(有人手误多起了 2 个)。

如果你的脚本是"发现不够就起、发现多了不管",那它会无视这多余的 2 个——**它不知道"多了"也是一种偏差**。如果你的脚本是"发现多了就杀",那它怎么判断"哪些是多余的"?按什么顺序杀?杀错了怎么办?**为了让脚本能处理"多了"的情况,你得为每一种异常状态都写一个分支——而异常状态组合爆炸,根本写不完。**

更要命的是**部分失败**:脚本执行到一半,网络断了、机器宕了、磁盘满了——脚本停在中间状态。下次重跑,它不知道上次跑到哪、该从哪续。**你只能人工介入,清理烂摊子,再重跑。** 容器少时人工介入还撑得住,几千个容器时,运维团队每天的工作就是"清理脚本留下的烂摊子"。

### 缺陷三:状态散落,没有"唯一真相"

脚本跑在哪台机器上?通常是一台"运维中控机"。这台机器从 300 台节点上 `ssh` 过去执行命令、收集状态。

可现在问:**"集群里现在到底有几个 web 容器"?** 答案散落在:

- 中控机上的脚本日志(脚本自己以为起了几个);
- 300 台节点上的 docker daemon(每台实际跑了几个);
- 监控系统里(监控看到的几个);
- DNS / 服务注册中心(对外宣称的几个)。

**这四个地方,经常对不上。** 脚本以为起了 3 个,实际有台机器 `docker run` 失败了(磁盘满),只有 2 个;监控因为采集间隔,看到的是 5 分钟前的数据;DNS 缓存还在指向已经死掉的 IP。**每个人眼里都有"真相",但没有一个唯一的、权威的真相。**

在没有"唯一真相源"的世界里,所有自动化都是建立在沙子上——脚本根据一个过时的、不完整的"真相"做决策,做出的决策自然是错的。容器少时这种错偶尔发生,数量一多就是常态。

### 缺陷四:扩展性差,复杂度爆炸

脚本最大的敌人,是**组合爆炸**。

假设你有 5 种服务(web、api、worker、cache、db),每种有 4 种运维操作(部署、扩缩容、重启、回滚),每种操作要处理 3 种异常(超时、部分失败、状态不一致)。光是这 5×4×3=60 种组合,你就要写 60 套脚本,而且它们之间还会互相影响(web 扩容可能挤压 cache 的资源)。

容器到几千个、服务到几十种时,这种组合爆炸会让你的脚本仓库变成一个**没人敢动的怪物**——每个脚本都是某个工程师在某个深夜为了应对某个具体事故写的,后人不敢改、不敢删,只能往上面再叠新脚本。最后,整个系统的复杂度,超过了任何单个工程师的理解能力。**系统从"自动化"退化成"由一堆脚本驱动的、没人能完全理解的黑盒"。**

> **把这四个缺陷摆在一起,结论很清楚**:
>
> **"脚本 + 监控 + 自动重启"这条路,本质上是用"命令式(imperative)"的思维去解决"在动态变化的分布式系统里维持期望状态"的问题。而命令式思维——"我执行什么动作,世界就变成什么样"——天生不适合这种问题。**
>
> 不是脚本写得不够好,是**这种解题范式本身走错了方向**。再多写脚本,只会让复杂度爆炸得更厉害,不会让问题消失。

## 四、出路:换一个范式——声明期望状态,让控制器自动调谐

既然"写更多脚本"走不通,那唯一的出路是**换一个范式**。这个范式,就是 k8s 的灵魂,也是它和 docker、和一切"脚本运维"最本质的区别。

### 这个范式长什么样:你说"我要 N 个",它盯着实际有几个

我们用最朴素的对比,把新旧两种范式摆在一张表上:

| 维度 | 命令式(脚本 / 老 docker) | 声明式(k8s) |
|------|--------------------------|--------------|
| 你说什么 | "**去启动** 3 个 web 容器" | "**我要** 3 个 web 副本" |
| 你关心什么 | 动作(执行了哪些命令) | 结果(最终是不是 3 个) |
| 再跑一次 | 又启动 3 个(变 6 个,不幂等) | 没变化(还是 3 个,幂等) |
| 中间挂了 | 停在半路,要人工清理 | 自己重新算,继续往目标走 |
| 多了怎么办 | 不管(或要单独写脚本杀) | 自动杀掉多余的 |
| 少了怎么办 | 不管(或要单独写脚本补) | 自动补上缺的 |
| 挂了怎么办 | 不管(或要 `--restart`) | 自动在别处重新拉起 |

看出差别了吗?**命令式关心"我做了什么",声明式关心"世界最终该是什么样"。**

> **比喻**:这是两种完全不同的"递单子"方式。
>
> - **命令式**:你给港口经理一张操作清单——"先去 A 码头吊 1 号集装箱上 3 号船,再去 B 码头吊 2 号集装箱上 5 号船,然后……"。**你把每一步动作都规定死了。** 万一 3 号船中途沉了,你的清单就废了——后面的步骤建立在"3 号船还在"的假设上,现在假设不成立,整张清单作废。
>
> - **声明式**:你给全球调度中心一张**目标单**——"**我要这批货在 3 月 5 日前运到 X 港,共 3 箱**"。**你只声明期望,不规定怎么做。** 3 号船沉了?调度中心自己决定把货搬到 5 号船。5 号船也满了?它再找别的船。**不管中间发生什么,它都盯着"3 箱货到 X 港"这个目标,不停调整手段,直到目标达成。**

后者就是 k8s 的范式。它有一个名字,叫 **reconcile(调谐)**——不停地把"实际状态"往"期望状态"拉。

### reconcile 是什么:一个不停运转的循环

我们把这个范式落到一个**最朴素的循环**上(这一章先给直觉,真实源码下一章和第 17 章细讲):

```
   循环 forever:
       期望状态 = 从"唯一真相源"读出:我要 3 个 web 副本
       实际状态 = 从"唯一真相源"读出:现在集群里有几个 web 副本
       if 实际 < 期望:    补上缺的(在合适的节点上起几个新的)
       if 实际 > 期望:    杀掉多余的(按一定策略选几个删掉)
       if 有挂掉的:       在别处重新拉起
       sleep 一小段,再循环
```

注意这个循环的几个**惊人特性**:

1. **它天然幂等**。不管你跑多少遍,"期望 3 个、实际 5 个"这个输入,产出永远是"杀掉 2 个"——结果是确定的 3 个。**幂等性不是写出来的,是从"盯着目标"这个范式里自然长出来的。**

2. **它天然自愈**。容器挂了、机器宕了,下一轮循环就会检测到"实际少了",自动补上。**不需要单独写"故障重启脚本"——reconcile 循环本身就是故障重启。**

3. **它天然可收敛**。不管系统现在处于什么乱七八糟的状态,只要"期望"不再变,循环会一直跑,直到"实际 = 期望",然后保持在那个状态。**任何扰动都会被吸收,而不是被放大。**

4. **它对部分失败免疫**。循环跑到一半挂了?没关系,下一轮重新读"期望"和"实际",从头算一遍。**没有"中间状态",因为每一轮都是从头算的。**

> 这四条,正是脚本的四个缺陷(不幂等、不可收敛、状态散落、扩展性差)的**精确反面**。reconcile 范式不是"把脚本写得更好",而是**换了一个根本性的解题思路**——从"我执行什么动作"换成"我盯什么目标"。

### 谁来跑这个循环:控制器(controller)

这个 reconcile 循环,需要一个**常驻的程序**来跑。这个程序叫**控制器(controller)**。

- 你声明"我要 3 个 web 副本",这个声明被存进一个**唯一真相源**(下一章会讲,在 k8s 里它叫 etcd);
- 有一个控制器,专门盯着"web 副本"这类资源,不停 reconcile——少了补、多了杀、挂了重启;
- 你不需要写这个控制器——k8s 内置了一堆(controller-manager),管各种常见资源(Deployment 管"几个副本"、DaemonSet 管"每台机器一个"、Job 管"跑完就退"……)。

> **关键认知**:**k8s 不是"一个程序",它是"一堆控制器 + 一个真相源 + 一堆节点代理人"的组合体**。其中控制器,就是 reconcile 循环的具体化身。后面几章我们会一个一个拆它们,但**所有控制器的本质,都是同一个 reconcile 循环**——只是盯的资源不同、补/杀的策略不同。

### 这个范式为什么"必须"被发明:回看四个不够

把这个范式套回前面那四个不够,你会发现它们**一一化解**了:

- **故障自愈**:机器宕了,真相源里"期望 3 个"没变,但"实际"少了 3 个(因为节点代理人报不上来了),reconcile 循环检测到偏差,在别的节点上补起 3 个。**不需要人,不需要脚本。**
- **扩缩容**:你把"期望 3 个"改成"期望 30 个",reconcile 循环自动起 27 个。改回去,自动杀 27 个。**一次声明,自动收敛。**
- **跨机调度**:reconcile 在补副本时,要决定"补到哪个节点"——这正是调度器(scheduler)干的事。**调度是 reconcile 的一个子步骤。**
- **服务发现**:每个服务有一个稳定的"门牌"(Service),门牌背后是一组会变的 Pod IP。门牌的更新,也是一个 reconcile 循环在盯着——Pod 变了,门牌自动更新。**还是同一个范式。**

> **四件事,一个范式全包了。** 这就是为什么 k8s 看起来"大而全"——它不是把四件事各写一套,而是**用同一个 reconcile 范式,统一解决所有"维持期望状态"的问题**。这种统一性,正是 k8s 比一堆零散脚本优雅得多的根本原因。

## 五、第一性原理:为什么这套范式非存在不可

到这里,我们可以把这一章的第一性原理钉死了:

> **当容器从几个变成几万个、机器从几台变成几百台,运维的本质问题,从"怎么执行操作"变成了"怎么在动态变化的分布式系统里,持续维持期望状态"。而这个问题,用"命令式(脚本)"是解不开的——因为命令式天生不幂等、不可收敛、对部分失败无能为力。唯一的出路,是换一个范式:声明期望状态,让一个常驻的控制器(reconcile 循环)不停地把实际往期望拉。**

这个范式不是 k8s 凭空"发明"的。它在工业控制论里早就有名字——**控制循环 / 负反馈系统**:测量实际值,和目标值比,有偏差就调整,直到偏差为零。空调恒温器、汽车定速巡航、无人机的姿态稳定,都是这套原理。k8s 做的事,**是把这套工业界古老的负反馈控制原理,搬到了"管理几万个容器"这个问题上**。

> **比喻**:你家里的恒温器,从来不"命令"空调"现在开机、现在关机"。它只做一件事:**盯着温度**——高了就制冷,低了就停。你只设定"我要 22 度"(期望状态),剩下的事它自己 reconcile。**k8s 就是集群的恒温器**:你设"我要 3 个 web 副本"(期望状态),它盯着——少了补、多了杀,不管发生什么扰动,都把实际往期望拉。

一旦你从这个角度看 k8s,它就不再是"一堆花里胡哨的 YAML"。它是一个**极其克制、极其统一的系统**——核心就一个 reconcile 循环,反复跑,管所有资源。所有的 Deployment、DaemonSet、Service、Job、CronJob……都是这个循环的不同化身,盯不同的资源、用不同的策略。**理解了 reconcile,你就理解了 k8s 一切魔力的根源。**

> 这个范式的力量还体现在它的**可扩展性**上:既然所有控制器都是同一个 reconcile 循环,那么**你想管一种新的资源(比如"一个 MySQL 集群"),只需要定义一种新资源 + 写一个新控制器**——这就是 k8s 生态里大名鼎鼎的 **Operator 模式**(第 17 章会细讲)。整个云原生生态那么多项目(Prometheus Operator、Cert-Manager、ArgoCD……),底下都是同一个 reconcile 循环。**统一范式,带来了爆炸性的可扩展性。**

## 关键源码精读:一个 reconcile 循环,在 k8s 里长什么样

这一章是定调章,概念为主,代码点到为止。但为了让上面那个抽象的"reconcile 循环"不至于悬在半空,我们用 k8s 真实源码里**最小的一个证据**,让你看见它的影子——**只看一个文件、一句话**,不展开。

### reconcile 在 k8s 源码里的"指纹"

k8s 的所有控制器,都遵循同一个模式,叫 **Informer 模式**(下一章和第 17 章深挖)。它的核心是 client-go 里的 `tools/cache` 包。这个包提供了一个叫 `SharedInformer` 的东西——它就是 reconcile 循环的"事件源":

- 它从 apiserver **watch**(订阅)某种资源(Pod、Deployment……)的变化;
- 资源一变(新建、更新、删除),它就把变化的事件**推给**注册的回调函数;
- 控制器在自己的回调里,读"期望"和"实际",决定要不要补/杀,然后写回去——**这就是一轮 reconcile**。

这个包在 kubernetes 仓库里的位置,是 [staging/src/k8s.io/client-go/tools/cache](https://github.com/kubernetes/kubernetes/tree/master/staging/src/k8s.io/client-go/tools/cache)。你打开这个目录,会看到几个关键文件:

- [shared_informer.go](https://github.com/kubernetes/kubernetes/blob/master/staging/src/k8s.io/client-go/tools/cache/shared_informer.go)——`SharedInformer` 的实现,所有控制器的"事件源";
- [controller.go](https://github.com/kubernetes/kubernetes/blob/master/staging/src/k8s.io/client-go/tools/cache/controller.go)——一个最朴素的"收到事件就处理"的循环骨架;
- [delta_fifo.go](https://github.com/kubernetes/kubernetes/blob/master/staging/src/k8s.io/client-go/tools/cache/delta_fifo.go)——一个队列,把"资源变化"排成事件流,供控制器消费。

而一个具体的控制器,比如管 Deployment(管"几个副本")的,代码在 [pkg/controller/deployment/deployment_controller.go](https://github.com/kubernetes/kubernetes/blob/master/pkg/controller/deployment/deployment_controller.go)。它的核心方法 `syncDeployment`,就是一轮 reconcile 的具体化身——读出某个 Deployment 的期望副本数,数一下实际有几个 Pod,少了补、多了杀。

> **这里不贴大段代码**——本章是定调章,代码不是主角。你只需要记住一件事:**在 k8s 源码里,reconcile 不是某种抽象口号,它是一个真实存在的、反复运转的循环。** 你打开上面那几个文件,就能亲眼看见它。下一章(第 15 章)会拆 k8s 的整体架构,告诉你这个循环跑在哪个组件里;第 17 章会钻进 informer 的源码,把每一行讲透。

### 用一段简化伪代码,把直觉钉死

最后,为了让你带着"代码感"走出这一章,我们写一段**简化伪代码**(明确标注:**不是源码原文,是 reconcile 范式的最小示意**):

```go
// === 简化示意,非 k8s 源码原文,只用于理解 reconcile 范式 ===

// 一个 Deployment 控制器的核心,本质就是这么一个循环
func reconcileDeployment(deploymentName string) {
    // ① 读期望状态
    desired := etcd.Get(deploymentName).Replicas   // "我要 3 个"

    // ② 读实际状态
    actual := len(etcd.ListPods(deploymentName))   // "现在有几个"

    // ③ 比较偏差,采取行动
    if actual < desired {
        // 少了:补。补到哪个节点,是调度器(scheduler)的事
        createPods(desired - actual)
    } else if actual > desired {
        // 多了:杀。按一定策略(比如最年轻的)选多余的杀掉
        deletePods(actual - desired)
    }
    // 实际 == 期望:什么都不做,这就是"收敛"

    // 这个函数会被反复调用(每次 Deployment 变化、每次 Pod 变化)
    // 不管发生什么扰动,最终都会被拉回 "actual == desired"
}
```

读这段伪代码,记住三件事:

1. **它没有"启动""停止"这种命令**——只有"补""杀"两个动作,而且都是**根据偏差算出来的**,不是预先规定的。
2. **它是幂等的**——不管跑多少遍,"期望 3 个、实际 5 个"的输入,产出永远是"杀 2 个"。
3. **它对扰动免疫**——中间任何一步失败,下一轮重新算一遍就行。

这就是 k8s 一切魔力的根源。**剩下的所有细节(怎么 watch、怎么排队、怎么调度、怎么和 apiserver 通信),都是为了让这个循环跑得高效、跑得可靠。** 但范式本身,就是这么简单。

## 章末小结

### 用航运比喻回顾本章

回到港口。这一章我们做了一件纯粹的事:**搞清楚"为什么一个港口不够,必须有一个全球调度中心"。**

答案分三层,一层比一层接近本质:

1. **一个港口(docker)管好一艘船(单机)上的集装箱(容器)没问题,可一旦这种集装箱多到几万个、散布在几百艘船上,港口就力不从心了**。具体有四件事它做不到:故障自愈(船沉了,谁来搬货)、扩缩容(货多了,谁加船)、跨机调度(新货上哪艘船)、服务发现(货转到哪艘船了,怎么追踪)。**这四件事都需要一个站在所有船之上、有全局视野的角色。**

2. **"再多写点脚本"这条路,在容器数量过千时必然撞穿天花板**。脚本天生不幂等(跑两遍起两倍)、不可收敛(不知道"多了"也是偏差)、状态散落(没有唯一真相)、扩展性差(组合爆炸)。**这不是脚本写得不够好,是命令式思维本身不适合"在动态分布式系统里维持期望状态"。**

3. **唯一的出路,是换一个范式**:你只声明期望状态("我要 3 个副本"),由一个常驻的控制器(reconcile 循环)不停地把实际状态往期望拉——少了补、多了杀、挂了重启。这个范式天然幂等、自愈、可收敛,而且**用同一个循环就能统一解决前面所有四个不够**。**k8s 的全部精妙,都建立在这个范式之上。**

### 本章在全书主线中的位置:旅程的转折点

记住全书的二分法:**打包隔离 vs 调度编排。**

- **前半本(docker,第 0~4 篇)**讲的是**打包隔离**:怎么把一个应用装进集装箱、隔绝、限好量,在一个港口里高效运营几十个箱子。
- **从这一章开始,我们正式跨到"调度编排"这一侧**。这一章是**旅程的转折点**——它论证了"为什么港口不够、为什么必须有一个全球调度中心",立住了 k8s 半本的第一性原理:**编排的本质,是用 reconcile 循环维持期望状态。**

后面六章(第 15~20 章),都是在填这个范式的血肉:

- 第 15 章拆 k8s 的架构——**这个 reconcile 循环,跑在哪个组件里?真相源是什么?节点代理人是谁?**
- 第 16 章讲 Pod——**reconcile 的最小调度单位,为什么不是裸容器**。
- 第 17 章深挖声明式 API + reconcile 的源码——**这一章给的直觉,在那里被逐行钉死**。
- 第 18 章讲调度器——**reconcile 里"补到哪个节点"这一步,具体怎么决策**。
- 第 19 章讲 kubelet——**reconcile 决定"这个节点上要有这个 Pod",谁真正把它跑起来**。
- 第 20 章讲 Service——**服务发现这件事,也是用同一个 reconcile 范式解决的**。

每一章,你都会听到这一章那个范式的回响。

### 五个"为什么"清单

如果你只能记五件事,记这五件:

1. **为什么单机 docker 不够**:它在故障自愈(船沉谁搬货)、扩缩容(流量来谁加副本)、跨机调度(新货上哪艘船)、服务发现(货转到哪艘船了)这四件事上力不从心——这四件事都需要全局视野,而 docker 是单机港口。
2. **为什么"脚本 + 监控 + 自动重启"撑不住**:脚本天生不幂等、不可收敛、状态散落、扩展性差。容器数量过千时必然撞穿天花板——不是脚本写得不够好,是命令式思维本身不适合"在动态分布式系统里维持期望状态"。
3. **k8s 的核心范式是什么**:**声明期望状态 + 控制器自动调谐(reconcile)**。你说"我要 N 个",一个常驻控制器盯着"实际有几个",少了补、多了杀、挂了重启。这套范式天然幂等、自愈、可收敛。
4. **为什么这套范式能统一解决四个不够**:故障自愈是 reconcile 检测到"实际少了"自动补;扩缩容是改"期望"让 reconcile 自动收敛;跨机调度是 reconcile 的子步骤;服务发现也是同一个 reconcile 循环在更新门牌。**四件事,一个范式全包。**
5. **这套范式的力量从哪来**:它不是 k8s 凭空发明的,它是工业界古老的**负反馈控制原理**(恒温器、定速巡航)在"管理几万个容器"问题上的应用。而且因为它统一,所以可扩展——**写个新控制器就能管一种新资源(Operator 模式)**,整个云原生生态都建立在这个范式之上。

### 想继续深入,该往哪钻

- **亲手感受命令式和声明式的差别**:试试用脚本"维持一个目录里始终有 3 个文件"——你会发现自己不停在写"多了删、少了建、被别人删了补"。然后想想:如果把这个逻辑写成一个**循环**(每秒检查一次,根据偏差决定动作),它是不是就幂等、自愈、可收敛了?**这个循环,就是 reconcile。**
- **看 reconcile 在 k8s 源码里的"指纹"**:打开 [staging/src/k8s.io/client-go/tools/cache](https://github.com/kubernetes/kubernetes/tree/master/staging/src/k8s.io/client-go/tools/cache) 这个目录,扫一眼 `shared_informer.go` 和 `controller.go`——你会看见"watch 资源变化 → 排成事件 → 触发处理"这个骨架,它就是 reconcile 循环的"事件驱动版"。
- **看一个真实控制器**:打开 [pkg/controller/deployment/deployment_controller.go](https://github.com/kubernetes/kubernetes/blob/master/pkg/controller/deployment/deployment_controller.go),找到它的 `syncDeployment`(或类似的 sync 方法)——你会看见它读 Deployment 的期望副本数、数实际的 Pod、根据偏差采取行动。**这就是本章那段伪代码的真实版本。**(第 17 章会逐行拆它。)
- **想理解"为什么是负反馈控制"**:可以读一读控制论的基础(维纳的《控制论》任何一本通俗读物都行)。你会发现 k8s 的 reconcile 循环,和恒温器、定速巡航、无人机姿态稳定,是**同一个数学结构**——测量偏差,负反馈调整,收敛到目标。
