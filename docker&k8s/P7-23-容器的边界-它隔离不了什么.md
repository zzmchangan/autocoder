# 第 23 章 · 容器的边界:它隔离不了什么

> **前置**:你需要先读过 [第 1 章《第一性原理:容器不是虚拟机》](P0-01-第一性原理-容器不是虚拟机.md) 和 [第 22 章《安全:Seccomp / AppArmor / capabilities / 容器逃逸》](P7-22-安全-seccomp-capabilities-容器逃逸.md)。第 1 章立了全书最重要的那笔权衡——**"共享内核"既是容器轻量之本,又是容器隔离之限**——但当时只点了题,说"代价留到第 23 章细讲";这一章,我们就来把这笔记账还清。第 22 章给了你一整套"收紧权限"的工具:capabilities 裁权、seccomp 白名单系统调用、AppArmor/SELinux 上强制访问控制。但你用了这些工具之后会发现:它们把口子收紧了,可那道**根本的口子——共享内核——还在**。本章要回答的就是:这道口子到底有多大、什么时候它大到必须换一种隔离方式。

> **核心问题**:共享内核换来了轻量,代价是什么?容器隔离到底够不够用?什么时候容器隔离不够、必须上虚拟机(或它的中间形态)?
>
> 这一章是全书的收尾章。前面 22 章我们一路看着一个裸进程被套上 namespace、打包成镜像、装上货轮、被航运中心调度成几万个——这一章是诚实地交代:**这条旅程有一个边界,共享内核这个轻量化的根基,同时画死了容器隔离的天花板**。我们不回避它,而是讲清楚边界在哪、什么时候够用、什么时候不够、不够的时候工业界给出了哪几种"加高天花板"的办法。
>
> **读完本章你会明白**:
> - 共享内核到底带来了哪**三类风险**——一个内核漏洞为什么能让宿主上所有容器全军覆没、`/proc` 和 `/sys` 默认会泄漏宿主的哪些信息、为什么有些资源(CPU cache、中断)在容器里根本隔离不干净。
> - 为什么第 22 章那套收紧权限的工具**有极限**——capabilities/seccomp/AppArmor 能挡住"配置失误"和"已知漏洞",但挡不住"未知的内核 0day",也挡不住"硬件侧信道"。
> - **Kata Containers** 和 **gVisor** 为什么存在:它们都是在"容器体验"和"VM 隔离强度"之间找平衡——一个给每个 Pod 配一艘迷你小船(轻量 VM)但装卸还像集装箱,一个在箱和轮机之间塞一个"翻译员"(用户态内核)拦掉所有系统调用。
> - **安全沙箱的谱系**:裸容器 → gVisor(用户态内核)→ Kata(轻量 VM)→ 传统 VM,这是一条"隔离强度 vs 开销/启动速度"的取舍曲线,没有银弹,只有按场景选。

## 章首 · 共享内核,到底亏了什么

回到第 1 章钉死的那句话:

> 容器不是虚拟机,它就是个普通进程。它和别的容器共享同一个内核,隔离靠 namespace,限资源靠 cgroup。**这是它轻快省的根本,也是它隔离不如 VM 的根本。**

第 22 章我们已经把"怎么收紧权限"讲了一遍——别用 root、别给 `CAP_SYS_ADMIN`、装 seccomp 挡掉一堆危险系统调用、上 AppArmor。这些工具确实把容器逃逸的口子收得很窄。但本章要追问的是一个更根本的问题:

> **就算你把第 22 章的工具全用上,容器隔离的极限在哪里?有没有什么场景,光靠容器(无论怎么收紧)就是不够,必须上 VM?**

答案是:**有**。而这个"有"的根源,就在第 1 章埋下的那颗钉子——**共享内核**。这一章我们就把"共享内核"这件事拆开,看它到底亏了什么,亏的部分有没有办法补,补到什么程度,以及补不动的时候该怎么办。

> **比喻**:回到航运。前面 22 章我们一直在讲怎么把集装箱造得好、怎么调度得好——舱壁(namespace)够厚、配额(cgroup)够严、安检(seccomp/capabilities)够细。但**所有集装箱终究装在同一艘船的动力系统(内核)上**。这一章我们终于要面对一个早就存在、但一直被绕开的问题:**如果轮机本身出了故障(内核漏洞),或者轮机的仪表盘(/proc、/sys)对每个箱都可见,或者几个箱抢同一根传动轴(cache、中断)——这些事情,集装箱的铁皮舱壁挡得住吗?**

答案分三类风险、三种补法。我们一类一类来。

---

## 一、风险之一:一个内核漏洞,全军覆没

这是共享内核最致命的一类风险,也是和 VM 差别最大的一类。

### 不这样会怎样:VM 里一个漏洞只死一个 VM

先把对照摆出来。VM 的隔离是**物理级**的:每个 VM 有自己独立的内核(运行在 hypervisor 之上)。应用 A 跑在 VM-A 的内核里,应用 B 跑在 VM-B 的内核里。

- 如果 VM-A 的内核被发现一个提权漏洞(比如某个 syscall 能让普通用户拿到 root),攻击者**最多只能攻陷 VM-A**——他能拿到 VM-A 里的 root,但 VM-B 那个独立的内核,他碰都碰不到。
- 想跨 VM 攻击?他得先逃出 VM(突破 hypervisor 的隔离),这叫 "VM escape",难度比内核提权高一个数量级,而且 hypervisor(如 KVM)的代码量比 Linux 内核小得多、攻击面小得多。

这就是 VM 的硬隔离:**每个 VM 是一个独立的"内核域",一个域里的漏洞不波及别的域**。

### 所以容器这里:所有容器共享一个内核,一个 0day 全部沦陷

容器的隔离**没有独立的内核域**——所有容器跑在宿主的同一个内核上。这意味着:

- 容器 A 里的进程,只要能触发一个**内核漏洞**(比如某个系统调用的整数溢出、一个 race condition),就有可能从"普通进程"变成"内核态代码执行"。
- 一旦它进了内核态,**它面对的就是整个宿主机的内核**——所有容器的进程都在这个内核的调度之下,所有 namespace 的隔离都只是这个内核"自愿遵守"的规则。一个能改内核数据结构的攻击者,这些规则形同虚设。
- 这就是所谓的 **容器逃逸(container escape)**:攻击者从容器 A 出发,先拿到容器内的 root(靠内核漏洞提权),再突破 namespace 的边界,最后落到**宿主机**上——拿到宿主的 root。到这一步,宿主上**所有其他容器**都成了他的鱼肉。

> **第 22 章的工具能挡住一部分,但挡不住这一类**。capabilities 把 root 的几十项特权拆碎了,seccomp 把已知的危险系统调用(如 `keyctl`、`bpf`)挡掉了——这些能有效减少**已知的、可枚举的**攻击面。但它们对**未知的内核 0day** 无能为力:0day 的意思就是"我们根本不知道这个漏洞存在",你没办法在 seccomp 白名单里预先挡掉一条你不知道危险的系统调用。一旦 0day 命中,共享内核的后果就是**一损俱损**。

### 不这样会怎样:真实世界的惨案

这不是理论。容器逃逸提权 CVE 是有公开记录的:

- **CVE-2019-5736(runc 逃逸)**:恶意容器通过覆盖宿主上的 runc 二进制(`/proc/self/exe` 指向它),下次宿主用 runc 启动别的容器时,执行的就是攻击者的代码。这是 runc 自己的漏洞,不是内核的,但效果一样——拿到宿主 root。
- **CVE-2022-0185(heap overflow in `fs_context`)**:一个 `fsconfig(2)` 系统调用的整数溢出,普通用户(包括容器内的)能触发内核堆溢出,拿到宿主 root。这种内核级 0day 一旦被不可信容器利用,**宿主上所有容器一起沦陷**。

每一次这种 CVE 出来,整个云原生社区都要紧急升级内核、重启所有节点。因为只要你还在"共享内核"的模型里,这个漏洞就是**对所有容器同时有效**的。

> 这就是共享内核的第一笔亏:VM 里,一个内核漏洞只影响一个 VM;容器里,**一个内核漏洞理论上影响宿主上的所有容器**。隔离的强度,被"共享内核"这一笔硬生生压低了一个等级。

## 二、风险之二:`/proc` 和 `/sys` 默认会泄漏宿主的信息

第二类风险更隐蔽——它不一定要靠漏洞,很多时候是**默认配置就在泄漏**,只是我们习以为常了。

### 不这样会怎样:namespace 没有完全造幻觉

第 1 章我们说 namespace 是"造幻觉"——让进程以为自己是唯一的。但这个幻觉造得**并不彻底**。最典型的漏洞,在 `/proc` 和 `/sys` 这两个伪文件系统上。

`/proc` 是内核暴露进程和系统信息的接口。一个进了 PID namespace 的容器,进程表(`/proc` 里显示的进程)确实只有自己,这点 namespace 做到了。但 `/proc` 里还有大量**系统级**的信息,namespace 根本没隔离它们:

- `/proc/cpuinfo`、`/proc/meminfo`:宿主有几颗 CPU、多大内存,容器里看得清清楚楚。
- `/proc/version`:宿主的内核版本,直接暴露(配合 CVE,攻击者能精准匹配漏洞)。
- `/proc/cmdline`:宿主的内核启动参数。
- `/sys/kernel/`:内核各种参数、地址布局(`/sys/kernel/notes` 会泄漏内核代码段地址,绕过 KASLR 的关键信息)。

这些信息单看每一条都不致命,但合起来,攻击者在一个容器里就能**精准指纹宿主的内核版本、CPU 型号、内存布局**——这等于把宿主的"体检报告"免费送给了潜在的攻击者。在一个多租户环境(你不知道别的容器里跑的是不是恶意程序)里,这是实打实的信息泄漏。

### 所以这样:靠挂载策略和内核补丁打补丁

工业界的应对分几层:

- **挂载时遮蔽**:runc 默认会把 `/proc` 里一些敏感条目(`kcore`、`keys`、`timer_list` 等)用只读的空文件**遮蔽(mask)**掉,避免容器读到。这是"事后的缝补",不优雅但管用。
- **`hidepid` 挂载选项**:让 `/proc` 里只能看到自己的进程,缓解一部分泄漏。
- **内核新增的 namespace**:`time` namespace(CVE-2020-xxxx 一类漏洞催生的)、更细的 `/proc` 隔离……但这些都是**缝缝补补**,内核每加一个隔离,都是在"事后堵漏"。

> 这里的本质是:**namespace 是为"造进程级幻觉"设计的,不是为"造系统级信息隔离"设计的**。VM 里,`/proc/cpuinfo` 显示的是 VM 自己那个虚拟 CPU,和宿主可以完全不同——因为 VM 有独立的内核,它的 `/proc` 就是它自己的。容器没这个条件,它的 `/proc` 天生就是宿主内核的 `/proc`,只能靠后天的挂载策略去"裁剪"。

这是共享内核的第二笔亏:**信息隔离不干净**,默认会泄漏宿主的指纹,只能靠不断打补丁缓解。

## 三、风险之三:有些资源根本隔离不干净(cache、IRQ、NUMA)

第三类风险更"软"——它不直接导致逃逸,但在"性能隔离"上是个永远的盲区。

### 不这样会怎样:cgroup 管得了份额,管不了 cache

cgroup 能干的事情是有限的——它能管 CPU 时间份额、内存上限、IO 带宽,这些都是**可计量、可配额**的资源。但有些资源,cgroup 根本没有干净的控制器:

- **CPU cache(L1/L2/L3)**:这是所有核共享的物理资源。容器 A 在狂跑一个吃 cache 的循环,容器 B 的 cache 命中率就会被挤掉(所谓 "noisy neighbor")。cgroup 的 `cpu` 控制器只管时间片,**不管 cache 占用**。
- **内存带宽**:同上,容器 A 把内存通道塞满,容器 B 的访存延迟飙升。cgroup 的 `memory` 控制器管的是"用了多少内存",管不了"占了多少带宽"。
- **中断(IRQ)**:网卡收到包,硬中断落在哪个 CPU 上?这是由内核的 IRQ 亲和性决定的,不是 cgroup 管的。容器 A 触发的中断可能落在容器 B 的 CPU 上,抢走 B 的时间。
- **NUMA**:多 socket 机器上,跨 NUMA 访问内存慢得多,但容器对 NUMA 的感知很弱。

### 所以这样:只能靠 CPU pinning、CAT、隔离节点这些"重"手段

工业界的应对是:

- **CPU pinning(把容器钉在特定核上)**:用 `cpuset` cgroup,把容器限制在指定的几个核上。但这牺牲了调度的灵活性(第 18 章的调度器假设 CPU 是池子)。
- **CAT(Cache Allocation Technology,Intel)**:硬件级的 cache 分区,把 L3 cache 切成几块,分给不同的容器。但这要硬件支持、且粒度粗。
- ** Dedicated node(独占节点)**:对延迟敏感的容器,干脆给它一整个节点,别的容器都不许调度上来。这回到了"一物一机"的 VM 思路——用资源换隔离。

> VM 在这一类风险上同样不完美(它也共享物理 CPU cache),但有一个关键区别:VM 有**独立的内核**,可以在 VM 内部做更细的调度和隔离决策;而容器和宿主共享内核,所有决策都得在宿主这一层做,粒度受限。更关键的是,在一个多租户环境里,VM 至少保证"这台 VM 里的 CPU 调度、IRQ 分配完全由这个租户的内核说了算",而容器连这个保证都没有——宿主的内核 IRQ 路由可能对每个租户都不透明。

这是共享内核的第三笔亏:**性能隔离有盲区**,cache、带宽、中断这些资源 cgroup 管不干净,在"对延迟/吞吐极敏感"或"强多租户"场景下,这是实实在在的隔离缺口。

---

## 四、三类风险合起来:共享内核的隔离天花板

把上面三类风险摆在一起,你就能看清"共享内核"画的那道隔离天花板:

```
                  隔离强度(从弱到强)
   裸进程 ───────────────► 容器(ns+cgroup+第22章收紧) ───────► ??? ───────► VM

                                                  ▲
                              共享内核的天花板:挡得住配置失误和已知漏洞,
                              挡不住未知的内核 0day、/proc 指纹泄漏、cache 噪声
```

第 22 章的工具(capabilities/seccomp/AppArmor)是把容器往这条线**左侧**的"裸进程"方向推远,让它更靠近天花板,但**推不过天花板**。一旦你的场景需要"超越这条天花板"的隔离强度——典型的是:

- **跑不可信代码**(多租户 SaaS、用户上传的代码执行、CI/CD 跑外部 PR);
- **强隔离的多租户**(一家云厂商的托管 K8s,不同客户的 Pod 跑在同一批节点上);
- **合规要求**(金融、医疗,要求"逻辑隔离不够,要物理/硬件级隔离");

——光靠容器就不够了。这时候,工业界给了两条路:**在容器和 VM 之间,造一个中间地带**。下面两节,我们就看这两条中间路线——**gVisor(用户态内核)** 和 **Kata Containers(轻量 VM)**——它们为什么存在、各自怎么补共享内核的亏。

---

## 五、gVisor:在箱和轮机之间,塞一个翻译员

> **gVisor 是什么**:Google 开源的一个**用户态内核(user-space kernel)**。它自己用 Go 写了一个"简化版 Linux 内核",跑在用户态,作为容器和宿主内核之间的一个**拦截层**。容器里的应用以为自己在调 Linux 内核,实际上它的每一个系统调用都被 gVisor 拦截、由 gVisor **自己重新实现**(而不是转交给宿主内核)。

### 为什么需要 gVisor:把攻击面从"整个内核"缩到"一个 Go 程序"

回想第一节的风险——容器逃逸的根,是**应用直接面对整个宿主内核**,内核有 N 千万行代码、几千个系统调用,攻击面巨大,一个 0day 就全军覆没。

gVisor 的思路是:**不让应用直接面对宿主内核**。在应用和宿主内核之间,塞一层 gVisor 自己实现的"内核"(叫 **Sentry**)。应用发系统调用,先打到 Sentry;Sentry **自己处理掉绝大多数调用**(它用 Go 实现了一套类 Linux 的语义),只在极少数情况下(比如真的要读写文件)才以**最小化的、受控的方式**去请求宿主内核帮忙。

> **比喻**:回到航运。gVisor 相当于**在每个集装箱和轮机之间,加一个"翻译员"(Sentry)**。货物(应用)以为自己直接在跟轮机(宿主内核)对话,但实际上它说的每一句话,都先到翻译员这里;翻译员**自己能处理掉大部分请求**(他手里有一套简化版的轮机手册),只有在极少数情况下,才用最小化的、受控的方式去碰一下真轮机。这样一来,**货物能触达真轮机的路径,从"几千条系统调用"被压缩到"几十条精心筛选的主机调用"——攻击面骤降**。

这套设计的核心收益是:**应用不再直接面对整个 Linux 内核**。哪怕 Linux 内核有 0day,应用也碰不到——因为它只能和 Sentry 打交道,而 Sentry 是用内存安全的 Go 写的、只暴露一个极小的主机调用子集。攻击者要先攻破 Sentry(Go 的内存安全让这非常难),才有机会碰到宿主内核。

### 不这样会怎样:纯 seccomp 拦不住,因为挡的是"名单内"还是"名单外"

你可能会问:第 22 章讲的 seccomp 不就是"拦系统调用"吗?为什么不能光靠 seccomp?

关键区别在这里:

- **seccomp 是"过滤"**:它有一个白/黑名单,系统调用来了,**要么放行给内核,要么直接拒绝**。放行的那个调用,还是**真的进了宿主内核**执行——如果那个调用本身有 0day,seccomp 挡不住(因为你不知道它危险,名单上写的是"放行")。
- **gVisor 是"重新实现"**:Sentry 不是过滤,它是**自己写了一套**。应用要 `open` 一个文件,Sentry 自己的 Go 代码处理这个请求(可能转而去问它的 Gofer 进程要文件描述符),**整个过程根本没碰到宿主内核的 `open` 实现**。宿主内核的 0day,在 gVisor 这层被"绕过"了,而不是"过滤"了。

这就是 gVisor 和 seccomp 的本质区别:**seccomp 是守门的保安(名单对了就放行),gVisor 是把整个衙门搬走了(你想办事,找我这层,根本进不了原来的衙门)**。

### 代价:兼容性和性能

这套设计当然不是免费的:

- **兼容性**:Sentry 只实现了 Linux 系统调用的一个子集。某些冷门的系统调用、某些依赖内核特定行为的程序(比如一些需要 `ptrace`、特定 `ioctl` 的),在 gVisor 下会失败。它不是 100% 兼容 Linux。
- **性能**:每个系统调用都要穿过 Sentry 这层(拦截 + 重新实现),有结构性的开销。gVisor 官方文档把它分成"结构成本"(Sentry 进程的内存占用、系统调用多穿一层)和"实现成本"(Sentry 的网络栈、文件系统不如真内核成熟)。对于系统调用密集型的工作负载(比如高频小 IO),开销明显;对于计算密集型,影响较小。

> gVisor 在 Google 内部(GKE 的 Sandbox)、以及一些多租户场景里大规模使用,定位明确:**牺牲一部分兼容性和性能,换"不可信工作负载"的安全隔离**。它不是用来跑所有容器的,而是用来跑"你不完全信任、但又想享受容器体验"的那些。

---

## 六、Kata Containers:给每个 Pod 配一艘迷你小船

> **Kata Containers 是什么**:一个开源项目(由 Intel、阿里、Red Hat 等联合发起),它的思路更直接——**给每个 Pod 起一个真正的、但极轻量的虚拟机**,容器还是那个容器(OCI 标准),只是它被装进了这个轻量 VM 里跑。这样容器既保持了"OCI 标准镜像 + k8s 调度"的体验,又拿到了"独立内核 + 硬件虚拟化隔离"的强度。

### 为什么需要 Kata:回到 VM 的硬隔离,但甩掉 VM 的重

回忆第 1 章:VM 的隔离最彻底(独立内核),但代价是"每件货物配一整艘船",太重。Kata 想干的事是:**有没有可能把 VM 做得足够轻,让它能像容器一样被 k8s 调度,同时保留 VM 的硬隔离?**

它的实现是三件套:

- **一个轻量 VM**:每个 Pod 一个 VM,VM 里有一个**精简的客户机内核**(不是完整发行版的内核,是裁剪过、专门为容器优化的)。这个 VM 用 **QEMU/KVM、Cloud Hypervisor、Firecracker** 这类 hypervisor 起来。
- **一个宿主侧的 shim(`containerd-shim-kata-v2`)**:它实现 containerd 的 shim v2 接口,让 containerd/k8s **把 Kata 容器当成普通容器对待**——`kubectl apply` 一切照旧,底下偷偷换成 Kata。
- **一个 VM 内的 agent(kata-agent)**:agent 跑在 VM 内部,它才是真正"造容器"的那个——它在 VM 内部用 namespace + cgroup 把容器隔离出来(就像 runc 在宿主上干的事),只是这一切都发生在**一个独立的 VM 内核**里。

> **比喻**:回到航运。Kata 相当于**给某些贵重/危险的货物,单独配一艘迷你小船(轻量 VM)**——这艘小船有自己的小型轮机(独立客户机内核),货物和货轮主机房之间是**物理隔离**的。但关键是:**这艘小船的装卸体验,和普通集装箱一模一样**——起重机还是那个起重机(OCI 标准)、调度中心还是那个调度中心(k8s),只是底下从"装在大船甲板上"变成"装在小船里、小船再停在大船边"。**对航运系统(编排层)完全透明**。

### 不这样会怎样:全 VM 太重,全容器不够硬

把 Kata 放在谱系里看就清楚了:

- 纯 VM(传统云服务器那种):隔离最硬,但启动几十秒、占几 GB 内存,k8s 调度一个 Pod 等于启一个 VM,慢且贵。
- 纯容器(裸 runc):轻快,但共享内核,前面三节那些风险全有。
- **Kata**:用"轻量 VM + 容器体验"在两者中间切了一刀。它启动一个 Pod 还是会起一个 VM(比纯容器慢),但这个 VM 极精简(裁剪内核 + 极小 rootfs),启动时间从传统 VM 的几十秒压到**亚秒到几秒**;它仍然占一份独立内核的内存(几十 MB 级),但比传统 VM 一个 1-2GB 起步要省得多。

Kata 的硬隔离来自那个**独立的客户机内核**:

- 容器 A 在 Kata VM-A 里,它触发一个内核漏洞,最多攻陷 **VM-A 的客户机内核**——和 VM 一样,要影响别的容器,得先逃出 VM,这又是"VM escape"级别的难度。
- `/proc`、`/sys` 泄漏?VM-A 里的 `/proc` 是**客户机内核**的 `/proc`,和宿主的内核是物理隔离的两个内核,泄漏的是 VM-A 自己的信息,不是宿主的。
- cache/IRQ?同样,VM-A 有自己的内核调度,IRQ 路由是 VM-A 内部的事。

换句话说,**Kata 把共享内核这一笔亏,整个还清了**——代价是付出了"每个 Pod 一个轻量 VM"的开销。它换来了 VM 级的硬隔离,但启动和内存成本介于纯容器和纯 VM 之间。

### Kata 的代价:启动慢、密度低、运维复杂

Kata 不是银弹:

- **启动慢于裸容器**:每个 Pod 要起一个 VM,哪怕再轻量,也比"起一个进程"慢。对于需要**极速弹性**的场景(秒级扩容几千个 Pod),这个开销是痛点。
- **密度低**:每个 VM 有固定开销(客户机内核、agent 占的内存),一台物理机上能塞的 Kata Pod 数量,远少于能塞的裸容器数量。"密度"是云厂商的成本命脉。
- **运维复杂**:要管理 hypervisor、客户机内核镜像、virtio 设备、vsock 通信。出问题排查比裸容器难。

> Kata 的典型场景是:**强多租户(云厂商托管 K8s、SaaS 平台)、跑不可信代码、合规要求硬隔离**。在这些场景里,多花的启动时间和内存是"买安全"的合理价格;而在内部可信环境(公司自己的集群、跑自己可信的服务),裸容器 + 第 22 章的收紧工具通常就够了,Kata 反而是过度设计。

## 七、安全沙箱的谱系:一张取舍曲线

把前几节的东西摆到一张图上,你就能看到"容器隔离"这个问题的**完整谱系**——这是一条横轴"隔离强度"、纵轴"开销/启动速度"的取舍曲线:

```
   隔离强度
      ▲
      │                                          ┌─────────────┐
      │  传统 VM                                  │ 独立完整内核 │  启动:几十秒~分钟
      │  (KVM/QEMU,                              │ 硬件虚拟化   │  内存:GB 级
      │   云服务器那种)                           │ 100% 兼容    │  兼容:100%
      │                                          └─────────────┘
      │                    ┌──────────────────┐
      │  Kata Containers    │ 轻量 VM + 独立    │  启动:亚秒~几秒
      │  (containerd-shim   │ 客户机内核        │  内存:几十~百 MB
      │   -kata-v2)         │ OCI/k8s 透明      │  兼容:≈100%
      │                    └──────────────────┘
      │          ┌──────────────────────────┐
      │  gVisor   │ 用户态内核(Sentry)      │  启动:≈裸容器
      │  (runsc)  │ 系统调用拦截+重实现       │  内存:Sentry 开销
      │           │ Go 内存安全              │  兼容:<100%(子集)
      │          └──────────────────────────┘
      │  ┌──────────────────────────────────┐
      │  裸容器(runc)                       │  启动:毫秒
      │  + 第22章收紧(cap/seccomp/AppArmor)  │  内存:≈进程本身
      │  共享宿主内核                         │  兼容:100% Linux
      │  └──────────────────────────────────┘
      │
      └─────────────────────────────────────────────────────► 开销/启动速度(从重到轻)
```

这张图上,每一档都是对前一档"加高隔离天花板"的尝试,也都付出了相应的代价:

| 方案 | 隔离机制 | 隔离强度 | 启动开销 | 内存开销 | 兼容性 | 典型场景 |
|------|---------|---------|---------|---------|-------|---------|
| **裸容器(runc)** | namespace + cgroup(+ 第22章收紧) | 弱(共享内核) | 毫秒 | ≈进程本身 | 100% | 内部可信服务、CI、开发测试 |
| **gVisor(runsc)** | 用户态内核,拦截重实现系统调用 | 中(应用不直接碰宿主内核) | ≈裸容器 | Sentry 进程开销 | <100%(系统调用子集) | 多租户、跑不可信但有界的工作负载 |
| **Kata Containers** | 每个 Pod 一个轻量 VM + 独立客户机内核 | 强(独立内核 + 硬件虚拟化) | 亚秒~几秒 | 几十~百 MB/VM | ≈100% | 强多租户、云厂商托管、合规硬隔离 |
| **传统 VM** | 完整 hypervisor + 完整 guest OS | 最强 | 几十秒~分钟 | GB 级 | 100% | 传统云服务器、需要完全不同的内核/OS |

> **注意这张图上没有银弹**。从左到右,隔离越来越强,但开销越来越大、密度越来越低。**你不可能同时拿到"裸容器的轻"和"VM 的硬隔离"**——Kata 和 gVisor 是在这条曲线上找的**两个甜点**,但它们各自只切到了"比裸容器硬、比 VM 轻"的一部分。

选哪一档,取决于你的**威胁模型**:

- **你完全信任跑在容器里的代码**(自己公司的服务、可信的开源组件):裸容器 + 第 22 章的工具就够了。这是绝大多数内部集群的实际情况,也是为什么 docker/k8s 默认是裸容器。
- **你不完全信任,但攻击面有界**(用户上传的脚本、CI 跑 PR、文档转换服务):gVisor 合适。它兼容性略差、有性能开销,但安全收益大。
- **你不信任,且要硬隔离**(云厂商多租户、金融合规、跑任意第三方代码):Kata 合适。它给你 VM 级隔离,同时不牺牲容器生态的体验。
- **你要跑完全不同的内核/OS**(Windows on Linux、不同发行版):只能上传统 VM。

> 还有一类常被放在一起讲的,是**机密计算(confidential computing)**——基于硬件(如 Intel TDX、AMD SEV-SNP)对 VM 内存做加密,连宿主/hypervisor 都看不到里面的内容。Kata 已经集成了这条路线(它支持 TDX/SEV-SNP guest)。这把隔离强度又往上推了一档:不只是"独立内核",连"宿主也读不到你的内存"。这是比传统 VM 更强的隔离,用于"宿主本身也不可信"的极端多租户(比如在公有云上跑机密数据)。它已经超出"容器 vs VM"的范畴,属于硬件级机密计算的领地了。

## 关键架构精读:两个"加高天花板"的设计,在代码里长什么样

这一节我们做"架构精读"(概念章,代码点到为止),把 gVisor 和 Kata 两个最具代表性的中间路线,在它们各自的代码/配置里看清楚。重点不是贴大段源码,而是看清**它们各自怎么补"共享内核"的亏**。

### 精读一:gVisor 的"系统调用拦截 + 重实现"模型

gVisor 的核心是 **Sentry**(用户态应用内核)和 **Gofer**(文件系统代理)这两个进程。整个沙箱由 `runsc`(gVisor 的 OCI runtime)拉起来。我们看它最关键的几个设计点。

#### (1)平台(Platform):决定怎么"拦截"系统调用

gVisor 把"如何拦截应用的系统调用"抽象成了一个 **Platform 接口**,定义在 [pkg/sentry/platform/platform.go](https://github.com/google/gvisor/blob/master/pkg/sentry/platform/platform.go#L38-L119)。这是整个 gVisor 最底层的抽象——不同的 Platform 用不同的硬件/内核机制来拦截系统调用,但**对上层(Sentry)是透明的**:不管怎么拦截,拦截到的系统调用最后都交给 Sentry 的同一套处理逻辑。

gVisor 目前提供三种 Platform 实现([g3doc/architecture_guide/platforms.md](https://github.com/google/gvisor/blob/master/g3doc/architecture_guide/platforms.md)):

- **Systrap(默认)**:用 seccomp 的 `SECCOMP_RET_TRAP` 机制——应用一发系统调用,内核给它发一个 `SIGSYS` 信号,Sentry 的信号处理器接管。这是纯用户态方案,不需要硬件虚拟化。
- **KVM**:利用硬件虚拟化扩展(VT-x/AMD-V),让 Sentry 同时充当 guest OS 和 VMM。性能更好(尤其在裸机),但需要宿主支持嵌套虚拟化。
- **Ptrace(已废弃)**:用 `PTRACE_SYSEMU`,太慢,保留只为历史兼容。

> 这个 Platform 抽象是 gVisor 的一个高明设计:**"拦截机制"和"系统调用处理"解耦**。换个平台(比如从 Systrap 换 KVM),只是换了"怎么把系统调用截获到 Sentry",Sentry 里那套"用 Go 重新实现 Linux 语义"的代码完全不用动。

#### (2)Sentry:不是过滤,是重实现

这是 gVisor 和 seccomp 最本质的区别。看 gVisor 怎么描述自己的安全模型([g3doc/security_model.md](https://github.com/google/gvisor/blob/master/g3doc/security_model.md)):

> "Sentry 不仅仅过滤系统调用,而是**完全实现它们**。"

应用发 `open`,不是"被放行给宿主内核的 `open`",而是"Sentry 自己的 Go 代码处理这个 open"。系统调用的实现代码在 [pkg/sentry/kernel/syscalls.go](https://github.com/google/gvisor/blob/master/pkg/sentry/kernel/syscalls.go#L69-L86)——这是一个巨大的表,把每个系统调用号映射到 Sentry 用 Go 写的实现函数。**这些函数跑在 Sentry 进程里,根本不碰宿主内核**。

只有当 Sentry 真的需要宿主帮忙时(主要是文件操作),它才通过 **Gofer** 进程去和宿主打交道。Gofer 是一个独立的、权限受限的进程,通过 9P/LISAFS 协议和 Sentry 通信([pkg/sentry/fs](https://github.com/google/gvisor/tree/master/pkg/sentry/fs)、[pkg/gvisor](https://github.com/google/gvisor/tree/master/pkg/gvisor))。Gofer 是 Sentry 能碰宿主文件的**唯一通道**,而且这个通道是受控的——Sentry 不能任意打开宿主文件,只能通过 Gofer 中介。

#### (3)Sentry 自己也在 seccomp 的约束下跑

这是一个容易被忽略但很重要的设计:**Sentry 进程本身,也跑在一个严格的 seccomp 过滤器下**([pkg/sentry/kernel/seccomp.go](https://github.com/google/gvisor/blob/master/pkg/sentry/kernel/seccomp.go#L39-L58))。也就是说,Sentry 只能调一个**极小的主机系统调用子集**——它连"打开文件""创建 socket"这种特权操作都不能直接做(那些必须走 Gofer)。

这构成了 gVisor 的**纵深防御(defense in depth)**:

1. 应用碰不到宿主内核(被 Sentry 拦截重实现);
2. Sentry(应用内核)也碰不到大部分宿主能力(被自己的 seccomp 锁住);
3. Sentry 和 Gofer 分属不同进程,各有各的权限边界。

要攻破 gVisor,攻击者得**连续突破这三层**——这比"突破一个共享内核"难得多。

> **gVisor 在补什么亏**:它主要补的是第一节那个"一个内核 0day 全军覆没"的亏——通过让应用**不直接面对宿主内核**,把攻击面从"整个 Linux 内核"缩到"Sentry 的 Go 实现 + 一个极小的主机调用子集"。它对 `/proc`/`/sys` 泄漏也有缓解(应用看到的是 Sentry 伪造的 `/proc`)。但它**没有独立的客户机内核**,所以隔离强度仍弱于 Kata/VM——理论上 Sentry 自身如果有漏洞,应用还是可能逃逸。它的定位是"比裸容器安全、比 VM 轻"。

### 精读二:Kata 的"轻量 VM + agent"架构

Kata 的核心是把"容器运行时"和"VM 管理"缝合起来,做到对 k8s 透明。我们看它的三件套怎么协作。

#### (1)宿主侧的 shim:`containerd-shim-kata-v2`

kubelet 通过 CRI 调 containerd,containerd 根据 Pod 的 **RuntimeClass**(比如 `kata`)选择对应的 shim。对 Kata 来说,这个 shim 就是 `containerd-shim-kata-v2`([src/runtime/cmd/containerd-shim-kata-v2/](https://github.com/kata-containers/kata-containers/blob/main/src/runtime/cmd/containerd-shim-kata-v2/main.go))。

这个 shim 实现了 containerd 的 **shim v2 API**(一组 ttRPC/gRPC 接口)。对 containerd 来说,它和 `runc` 的 shim(`containerd-shim-runc-v2`)长得一模一样——**都是"给我一个 Pod,我负责把它跑起来、回报状态"**。containerd 不知道、也不关心底下是起了个进程还是起了个 VM。

shim 干的事(参考 [src/runtime/pkg/containerd-shim-v2/service.go](https://github.com/kata-containers/kata-containers/blob/main/src/runtime/pkg/containerd-shim-v2/service.go)):

1. `CreateSandbox`(创建 Pod 沙箱):调内部的 `virtcontainers` 库,启动一个 **hypervisor 进程**(QEMU / Cloud Hypervisor / Firecracker / 内置 Dragonball),这个进程拉起一个轻量 VM。
2. 等 VM 启动好,VM 里的 **kata-agent** 起来,在 VSOCK 上监听。
3. `CreateContainer`(创建 Pod 里的容器):shim 通过 VSOCK 上的 ttRPC 通道,**把容器配置发给 VM 内的 agent**,让 agent 在 VM 内部创建容器。

#### (2)VM 内的 agent:kata-agent

agent 是 Kata 设计的精髓。它跑在**客户机 VM 内部**([src/agent/](https://github.com/kata-containers/kata-containers/blob/main/src/agent/src/rpc.rs)),是一个 Rust 守护进程。它通过 **VSOCK**(虚拟套接字,一种 host-guest 通信通道)和宿主侧的 shim 通信,监听 ttRPC 命令。

agent 收到的命令和 runc 收到的命令**语义上几乎一样**——`CreateContainer`、`StartContainer`、`SignalProcess`、`WaitProcess`(参考 agent 的 RPC 定义 [src/agent/src/rpc.rs](https://github.com/kata-containers/kata-containers/blob/main/src/agent/src/rpc.rs))。区别在于:**runc 在宿主上用 namespace + cgroup 建容器,而 kata-agent 在 VM 内部用 namespace + cgroup 建容器**。

这是个很妙的复用:Kata 没有重新发明"怎么隔离容器",它**在 VM 内部又用了一遍 runc 的那套机制**(agent 内部用了一个叫 `rustjail` 的库,干的就是 runc 在宿主上干的事)。区别只是**作用域从"宿主"换成了"VM 内部"**。容器还是那个容器(OCI 标准),只是它被装进了一个有独立内核的 VM 里。

#### (3)独立的客户机内核:隔离的物理边界

这是 Kata 比 gVisor 更"硬"的地方:**每个 Kata Pod 有一个独立的客户机内核**。这个内核是 Kata 项目专门裁剪、优化的(`tools/packaging/kernel/`),只包含容器需要的子系统,体积小、启动快。

这个独立内核带来的隔离是**物理级**的:

- 容器里的进程触发内核漏洞,攻陷的是**客户机内核**,不是宿主内核。要影响宿主或其他 Pod,得先逃出 VM——这是 hypervisor 级的隔离,难度远高于突破 namespace。
- `/proc`、`/sys` 是**客户机内核**的,泄漏的也是 VM 自己的信息。
- IRQ 路由、CPU 调度都在**客户机内核**的控制下。

换句话说,**Kata 把"共享内核"这一笔亏,用"独立客户机内核"整个补上了**。它换回的是 VM 级的硬隔离,代价是每个 Pod 一个 VM 的开销。

> 一个值得注意的细节:Kata 把**容器体验和 VM 隔离**结合的方式,不是"在 VM 上再装一层容器抽象",而是反过来——**让 k8s/containerd 完全不知道底下是 VM**。从 `kubectl` 到 containerd 到 kubelet,看到的都是标准的 OCI 容器;只有最底下的 shim 知道"其实我起了个 VM"。这种"对上层透明、在底层换隔离机制"的设计,是 Kata 能融入现有云原生生态(而不是另起炉灶)的关键。它呼应了全书一个反复出现的主题:**OCI 标准把"容器长什么样"和"怎么把容器跑起来"解耦**——只要你的 runtime 实现 OCI runtime-spec,上层不关心你是 runc、kata 还是 runsc。第 8 章讲 OCI 的价值,在 Kata 和 gVisor 这里得到了最直接的印证:同一个标准接口,底下可以是截然不同的隔离实现。

## 章末小结

### 用航运比喻回顾本章

回到那片港口。前面 22 章,我们一直在讲怎么把集装箱造得好、怎么调度得好。这一章是诚实地承认:**集装箱的铁皮舱壁,再厚也有个天花板——因为所有集装箱终究装在同一艘船的动力系统(内核)上**。这个天花板,就是"共享内核"的代价。

我们看清了三类风险:

- **一个轮机故障,全军覆没**:共享内核的一个漏洞,理论上能让宿主上所有容器一起沦陷。第 22 章的工具能挡已知漏洞,挡不住未知的 0day。
- **仪表盘对每个箱可见**:`/proc`、`/sys` 默认泄漏宿主的内核指纹、CPU、内存信息,只能靠挂载策略不断打补丁。
- **几个箱抢同一根传动轴**:CPU cache、内存带宽、中断这些资源,cgroup 隔离不干净,在强多租户或延迟敏感场景是实实在在的缺口。

然后我们看了工业界在"容器和 VM 之间"造的两条中间路线:

- **gVisor(翻译员)**:在箱和轮机之间塞一个翻译员(Sentry),应用以为自己直接和轮机对话,实际上每个系统调用都被翻译员拦截、用 Go 重新实现。攻击面从"整个内核"缩到"一个 Go 程序 + 极小的主机调用子集"。
- **Kata Containers(迷你小船)**:给每个 Pod 配一艘迷你小船(轻量 VM),小船有自己的小型轮机(独立客户机内核),货物和主机房物理隔离,但装卸体验和普通集装箱完全一样(OCI 标准 + k8s 透明)。

合起来,我们看到一条**安全沙箱的谱系**:裸容器 → gVisor → Kata → 传统 VM,横轴是隔离强度,纵轴是开销和启动速度。**这条曲线上没有银弹**,只有按威胁模型选——你信任代码就用裸容器,不完全信任但有界就用 gVisor,要硬隔离就用 Kata,要完全不同的内核/OS 就用传统 VM。

### 本章在全书的二分法里站哪边

全书的二分法是:**打包隔离 vs 调度编排**。

这一章,是对**"打包隔离"这一侧的极限的诚实交代**。第 1 章我们立了"容器 = 普通进程 + namespace + cgroup",讲了它又轻又隔离的根本在"共享内核";这一章把那笔账的另一半算清了——**共享内核,既是容器轻量之本,也是容器隔离之限**。从第 1 章到第 22 章,我们一路把"打包隔离"做到了极限:namespace 造幻觉、cgroup 限量、rootfs 换根、overlay 分层、镜像标准化、capabilities 裁权、seccomp 拦截。但到了这一章,我们必须承认:**所有这些,都还停在"共享内核"的天花板之下**。要突破这个天花板,就得往上跨一级——gVisor 用用户态内核,跨半级;Kata 用轻量 VM,跨一级;传统 VM,跨到底。

这也是为什么 gVisor 和 Kata 都拼命保持"OCI 标准 + k8s 透明"——**它们换的是"打包隔离"的实现,但不碰"调度编排"那一侧**。k8s 的调度器、kubelet、Service、PV/PVC,对它们来说都是现成的、照用的。这印证了全书分层解耦的主题:**OCI 把"隔离怎么做"和"怎么调度"切开,让你可以在不动调度层的前提下,替换底下的隔离强度**。

### 全书的收束:一个进程,到这里

这是全书正文的最后一章。让我们回到第 1 章开头那个最朴素的问题——**一个普通的应用,要变成"被调度成几万个"的那个东西,它得经历什么?**

二十几章下来,我们一起走完了这段旅程:

1. 它是个**裸进程**,什么都没有(第 1 章开头)。
2. 被套上 namespace 的铁皮舱壁、cgroup 的配额、pivot_root 的新根——它**被隔离成一个容器**(第 1~6 篇)。
3. 被打包成**标准镜像**(第 2 篇),有了 OCI 标准、runc、containerd、docker 这套工业级的装箱和装卸体系(第 3 篇),修好了集装箱之间的路(第 4 篇)。
4. 当这种集装箱多到几万个、散布在几百艘船上,一个**航运调度中心(k8s)**把它调度、保活、扩缩容(第 5~6 篇)。
5. 中间我们还给它接了**不沉的仓库**(第 21 章存储)、**收紧了安检**(第 22 章安全)。
6. 最后,在这一章,我们**诚实地交代了它的边界**——共享内核画的那道天花板,以及想突破天花板时,有 gVisor 和 Kata 这两条路。

第 1 章我们钉死了一句话:**"容器就是个普通进程"**。这本书想让你带走的,正是这个第一性原理的视角——**理解了"它是个进程",你就不会被 docker/k8s 那堆名词吓住,因为它们底下都是你已经认识的内核能力**。而这一章,我们给那句话补上了它的下半句:

> **容器就是个普通进程——但它有边界。共享内核既是它的轻量之本,也是它的隔离之限。知道边界在哪里,你才敢用;知道有 gVisor 和 Kata 可以把边界往上推,你才知道在什么场景该换一种隔离。**

打包隔离的旅程到这里收口。从这里往后,如果你想继续深入,就该往两个方向钻:一个是**往下**,钻进 Linux 内核(namespace/cgroup/seccomp/hypervisor/KVM 的实现)——那是本书反复说"根扎在那里"的内核系列;另一个是**往侧**,钻进 Kata Containers 或 gVisor 的源码——看它们各自怎么在"容器体验"和"VM 隔离"之间做工程上的精妙权衡。

### 五个"为什么"清单

如果你只能从这一章带走五件事:

1. **共享内核亏在哪(三类风险)**:① 一个内核漏洞理论上让宿主所有容器全军覆没;② `/proc`、`/sys` 默认泄漏宿主指纹;③ cache、带宽、中断这些资源 cgroup 隔离不干净。第 22 章的工具能挡已知漏洞和配置失误,挡不住未知的 0day。
2. **第 22 章为什么有极限**:capabilities/seccomp 是"过滤"——名单内放行、名单外拒绝。放行的调用还是真进了宿主内核;遇到名单上不知道危险的 0day,过滤挡不住。要从根上解决"应用直接面对整个宿主内核",得换隔离机制。
3. **gVisor 为什么存在**:它不是过滤,是用 Go **重实现一套用户态内核(Sentry)**,应用不直接碰宿主内核。攻击面从"整个 Linux 内核"缩到"Sentry + 极小主机调用子集"。代价是兼容性(<100%)和性能(多穿一层)。
4. **Kata 为什么存在**:它给每个 Pod 配一个**轻量 VM(独立客户机内核)**,容器还是 OCI 标准容器。换来 VM 级硬隔离,代价是启动慢于裸容器、密度低、运维复杂。
5. **没有银弹,只有取舍**:裸容器 → gVisor → Kata → 传统 VM,是一条"隔离强度 vs 开销"的取舍曲线。选哪一档取决于威胁模型。这套选择权能存在,得益于 OCI 标准——同一接口,底下可以换截然不同的隔离实现。

### 想继续深入,该往哪钻

- **钻 gVisor**:读 [google/gvisor](https://github.com/google/gvisor) 的 `pkg/sentry/`——[platform.go](https://github.com/google/gvisor/blob/master/pkg/sentry/platform/platform.go) 看 Platform 抽象,[kernel/syscalls.go](https://github.com/google/gvisor/blob/master/pkg/sentry/kernel/syscalls.go) 看系统调用表。配套读官方 [安全模型文档](https://gvisor.dev/docs/architecture_guide/security_model/)。
- **钻 Kata Containers**:读 [kata-containers/kata-containers](https://github.com/kata-containers/kata-containers) 的 `src/runtime/`(宿主侧 shim)和 `src/agent/`(VM 内 agent)。想看机密计算这条更硬的隔离,读 `docs/design/` 下 TDX/SEV-SNP 文档。
- **钻"容器逃逸"的真实 CVE**:CVE-2019-5736(runc)、CVE-2022-0185(内核 fs_context)——读漏洞分析,你会真切理解"共享内核的一个漏洞为什么这么致命"。
- **钻内核侧的 namespace/cgroup**:回到 [torvalds/linux](https://github.com/torvalds/linux) 的 `kernel/pid_namespace.c`、`mm/memcontrol.c`、`kernel/seccomp.c`——看那道铁皮舱壁在内核里到底怎么实现、有哪些已知的缝。
- **亲手试一下**:用 kind 起集群,装 runsc 和 Kata,给 Pod 打不同 `runtimeClassName`——在 gVisor 容器里敲 `uname -r`(看到 Sentry 伪造的版本号)、在 Kata 容器里敲 `dmesg`(看到客户机内核日志)。你会对"OCI 标准把隔离和编排切开"有切肤的理解。
