# 第 1 章 · 第一性原理:容器不是虚拟机

> **前置**:这是全书的起点,前置是[全书导言](P0-00-导言-一篇看懂容器全貌.md)——它已经用"集装箱与航运物流"这个比喻,把 docker 到 k8s 一口气串了一遍。本章就把导言里那句轻描淡写的"容器比虚拟机轻"**追问到底**:轻在哪?凭什么又轻又能隔离?它到底是个什么东西?

> **核心问题**:容器到底是什么?它和虚拟机差在哪?凭什么又能隔离、又这么轻?
>
> 这一章我们不碰任何具体机制(不写 Dockerfile、不敲 kubectl),只问一件事:**如果"容器"不是虚拟机,那它到底是什么?它的隔离是从哪儿来的?为什么能做到"又轻又隔离"这件看起来矛盾的事?**
>
> **读完本章你会明白**:
> - 为什么虚拟机为了隔离,付出了"每件货物配一整艘独立货轮"的代价;而容器能又轻又隔离,靠的不是更厉害的虚拟化,而是**根本不虚拟化**。
> - 容器最反直觉的真相:**它不是一类新东西,就是一个普通的 Linux 进程**——`ps`、`uname`、`/proc` 都会出卖它。
> - namespace 和 cgroup **不是 Docker 的发明**,是内核早就摆在那儿的能力,Docker 只是"把几样现成的东西拼起来"。
> - 为什么"共享内核"既是容器最大的优点(轻),也是它最大的软肋(隔离不如 VM)——这个权衡贯穿全书。

---

## 章首·一句话点破

如果你对容器的印象是"一种轻量级的虚拟机",请把这句话先放一边。这是理解一切容器机制时,最容易绊倒、也最致命的一个误解。

这一章要做的第一件事,就是把这个误解连根拔起:

> **容器根本不是虚拟机。它和虚拟机的差别,是"所有货物装在同一艘船上"和"每件货物配一整艘独立货轮"的差别。**

我们一块一块拆。先从一个最朴素的问题开始:不隔离,会怎样?

---

## 一、先看清"裸进程"的世界:不隔离会怎样

要理解容器为什么存在,得先看清"没有容器"是个什么处境。

> **比喻**:想象一艘**没有集装箱的货轮**。所有货物——汽车、化学品、生鲜、钢琴——**直接堆在甲板上**,没有任何分隔。货物互相能看到、能碰到、能挤占彼此的位置,甚至能碰到船的轮机和舵。

这听起来很乱,但这就是**一个普通 Linux 进程裸跑**时的真实处境。你在机器上随便编译一个程序跑起来,这个进程默认拥有:

- **能看到全部文件**:整个文件系统对它敞开,`/etc/passwd`、别的用户的家目录、`/proc` 里所有其他进程的信息,它想读就读。
- **能看到所有其他进程**:一条 `ps aux`,机器上跑的每一个进程都列得清清楚楚——数据库、SSH 会话、别的应用。
- **能抢占任何资源**:它能把 CPU 跑满、把内存吃光、绑定任何端口,把别的进程活活挤死。
- **甚至能搞坏系统**:如果它有 root 权限,它能改内核参数、卸载磁盘、删掉系统文件。

### 不这样会怎样

假设你想在一台机器上同时跑 10 个不同的应用(你的、同事的、外包的、开源的)。如果没有任何隔离:

- 应用 A 的一个 bug 写穿了内存,改坏了应用 B 的关键数据——**一个程序崩,连累一车**。
- 应用 A 能 `cat /etc/shadow` 读到系统密码、能 `ps` 看到数据库进程、能翻别的应用的家目录——**应用之间毫无秘密可言,根本没法安全地"在一台机器上跑互不信任的程序"**。
- 应用 A 突然狂吃内存,把机器撑爆,所有应用一起死——**没有资源限制,一个流氓应用能搞垮整台机器**。

这就是"裸进程"的三个乱子:**没有隔离(互相能看能改)、没有秘密、没有限额**。它逼出一个需求:得有个办法,让"在同一台机器上跑很多应用"变成安全的、互不干扰的。

最直白、也最彻底的解法,就是虚拟机。我们先看它,再看出它的问题。

---

## 二、虚拟机这条路:为什么太重

为了隔离,人类最先想到的、也是最彻底的办法是**虚拟机(Virtual Machine, VM)**。

思路是这样的:既然应用之间会互相干扰,那我干脆给每个应用**一整套完整的机器**——虚拟的 CPU、内存、磁盘、网卡,然后在上面装**一整套完整的操作系统**(Windows、Linux...)。应用 A 跑在 VM-A 的 OS 里,应用 B 跑在 VM-B 的 OS 里,两个 OS 各有各的内核,物理上彻底分开。

> **比喻**:这就相当于**给每件货物配一整艘独立的货轮**——连船体、轮机、船员、船长都各自独立一套。汽车装一艘船,化学品装另一艘船,彼此绝对碰不到。

### 这么做的好处:隔离最彻底

VM 的隔离是**物理级**的:每个 VM 有自己独立的内核。应用 A 的内核崩了、被入侵了,应用 B 完全不受影响——因为它压根是另一个内核。这是最硬核的隔离。

> 在很长一段时间里,这就是"在一台物理机上跑多个互相隔离的应用"的标准答案。云厂商卖给你的"云服务器",本质上就是一台 VM。

### 不这样会怎样:太重的代价

但"每件货物配一整艘船"贵得离谱。给每个应用一整套 OS,意味着:

- **启动慢**:开一个 VM,要引导一整个操作系统——BIOS 自检、加载内核、起 systemd、起一堆系统服务(sshd、crond、日志……)。**几十秒到几分钟**才能就绪。
- **占资源**:每个 VM 要预留几个 GB 内存、几十 GB 磁盘——哪怕里面只跑一个几百 KB 的小程序。光"维持这艘船空转"就要吃掉大量资源。
- **冗余巨大**:你要跑 100 个应用,就得开 100 个 VM,每个都跑着一整套重复的 OS 和系统服务。**90% 的资源都浪费在"养船"上,而不是"运货"上**。

### 关键洞察:杀鸡用了牛刀

把这些代价摆在一起,你会发现一个尴尬的事实:

> **为了实现"应用之间互不干扰"这一点隔离,VM 把"整个操作系统"原样复制了一遍。**

可问题是:**绝大多数应用,根本不需要不同的内核。** 一个 Node.js 网站和一个 Python 脚本,跑在同一个 Linux 内核上完全没问题——它们要的只是"互不干扰的运行环境",而不是"各自一套内核"。

VM 把整个 OS 都搬来,是为了隔离彻底;但对于 99% 的场景,这份彻底是多余的、是浪费。**这就好比为了让两件货物不互相磕碰,给每件货都造了一整艘船——隔离是做到了,可船的成本远超货物本身。**

于是问题变成了:

> 能不能**不复制内核**,只用更轻的方式,就让应用之间"互不干扰"?——既保住隔离,又甩掉"养一整艘船"的开销?

答案,就是容器。

---

## 三、容器的高明:共享货轮的动力系统,只用铁皮箱隔开

容器的思路,和虚拟机**正好反过来**:

> **不复制内核。所有容器共享宿主机的同一个内核。隔离,靠内核自己早就准备好的两个能力——namespace 和 cgroup。**

> **比喻**:不再"每件货物配一整艘船",而是**把所有货物装在同一艘货轮上,共享船的动力系统(内核),只用标准的铁皮集装箱(namespace + cgroup)把货物彼此隔开**。
>
> - **铁皮舱壁(namespace)**:让每个集装箱里的货物,看不到也碰不到别的箱里的货——它以为自己是这艘船上唯一的货。
> - **载重 / 体积配额(cgroup)**:给每个集装箱限重限量,谁也不能把船的动力吃光。

### 为什么这样就能又轻又隔离

这套做法的精妙在于,**它一点都没"虚拟化"**——它没有造虚拟 CPU、没有造虚拟磁盘、没有起第二个内核。它干的事情,本质上是:

1. **隔离**:用 namespace,让进程"看到的世界"和别的进程不一样(看到自己的进程表、自己的网卡、自己的挂载点)。这层能力,内核里早就有。
2. **限资源**:用 cgroup,给这个进程(组)设个上限——最多用 1 个 CPU、2GB 内存。这层能力,内核里也早就有。

**容器所做的,就是"把这俩现成能力,套到一个进程身上"。** 仅此而已。

因为没造任何虚拟硬件、没起第二个内核,所以:

- **轻**:启动一个容器,本质上就是**启动一个进程**(配上几个隔离标志)。没有 OS 引导、没有系统服务——**毫秒级**就绪,而不是 VM 的几十秒。
- **省**:所有容器共享同一个内核,没有重复的系统服务。一台机器跑几百个容器,内存开销远小于跑几十个 VM。

### 代价:共享内核是命门,也是软肋

但"共享内核"这笔买卖是有代价的,而且这个代价恰恰来自它最大的优点:

> **所有容器共享同一个内核,意味着隔离没有 VM 那么彻底。**

- VM 里,一个内核漏洞只影响那一个 VM;**容器里,一个内核漏洞理论上能让宿主上所有容器全军覆没**(所谓"容器逃逸")。
- 容器之间默认共享同一个内核,所以一些内核资源(`/proc`、`/sys`、CPU cache)的隔离不如 VM 干净。

> **这是全书最重要的一笔权衡**:容器用"共享内核"换了"轻快省",代价是"隔离强度不如 VM"。绝大多数场景下这笔买卖划得来;但在"强隔离、多租户、不可信代码"的场景下,它就是软肋——这正是第 22 章(安全)和第 23 章(容器边界)要专门讲的事,也是 Kata Containers、gVisor 这些"带 VM 隔离强度的容器"存在的原因。
>
> 记住这个权衡,后面每一章你都会看到它在回响。

---

## 四、核心真相:容器就是一个普通进程

到这里,我们要钉死全书**最反直觉、也最根本**的一个事实。如果你只能从这一章带走一句话,就带这一句:

> **容器不是"另一种虚拟机"。它就是一个普通的 Linux 进程。**

这不是夸张,是字面意义上的事实。下面三个"铁证",你随便找台跑着容器的 Linux 机器就能验证:

### 铁证一:宿主机上 `ps` 能看到容器里的进程

在宿主机上敲 `ps aux`,你会看到容器里跑的程序——它赫然在列,有自己的 PID。**它不是"虚拟机里的一个进程",它就是宿主进程表里的一行。** 只不过(因为 PID namespace)它在容器**内部**看到的自己的 PID 是 1,而在宿主上看到的是另一个号。

> **比喻**:集装箱里的货物,和别的货物一起装在同一艘船上。你站在甲板(宿主机)上低头看,能清清楚楚看到每一个集装箱里的每一件货。集装箱没有把货物"传送"到另一艘船上——它只是用铁皮把货物**围起来**,让货物**以为**周围没有别的货。

### 铁证二:`uname -a` 在容器里和在宿主机上完全一样

进到一个容器里敲 `uname -a`,再在宿主机上敲一次。**输出一模一样**——同一个内核版本、同一个主机名架构。为什么?因为**它俩根本就是同一个内核**。容器没有自己的内核,它用的就是宿主的。

> VM 就不一样:VM 里的 `uname` 显示的是 VM 自己那个独立内核的版本,和宿主可以完全不同。这一条对比,就把"容器共享内核、VM 各有内核"的区别,赤裸裸地摆出来了。

### 铁证三:它就是一次 `exec` 出来的程序

容器里那个跑着的应用,不是什么"虚拟机里的 guest",它就是宿主内核调度的一个普通进程,走的是同一套进程调度、同一套内存管理、同一套系统调用。它和你自己 `./myapp` 起的一个进程,**在内核眼里没有任何本质区别**——只是它被多套了几个 namespace、多挂了几个 cgroup 限制而已。

> 这三个铁证合起来,说明一件事:**容器的"魔法"不在于它是一种新东西,而在于"内核早就有这些能力,容器只是把它们组合起来,套到一个普通进程上"。** 理解了这一点,你就不会被 docker / k8s 那一堆名词吓住——它们底下,都是些你已经(或在内核系列里即将)认识的老朋友。

那么,下一个问题自然就来了:**如果容器就是个普通进程,那它的隔离到底是怎么"套"上去的?** 这就引出本章的源码高潮。

---

## 五、铁证如山:namespace 只是 `clone` 的一个标志位

既然"容器 = 普通进程 + 隔离",那"隔离"这个动作,在代码里长什么样?

答案极其朴素,朴素到会让你怀疑人生:**它就是创建进程时,多传了几个标志位。**

### `clone` 系统调用早就有的标志位

Linux 创建进程(线程)的系统调用叫 `clone`。它的参数里有一组**标志位(flags)**,用来决定"新进程和父进程共享什么、不共享什么"。比如 `CLONE_VM` 表示"共享内存地址空间"(这就是线程的本质)。

而其中有这么一批标志位,名字都以 `CLONE_NEW` 开头——它们的意思是:**别共享了,给新进程新建一份**。看内核里它们的定义([include/uapi/linux/sched.h](https://github.com/torvalds/linux/blob/master/include/uapi/linux/sched.h)):

```c
#define CLONE_NEWNS	0x00020000	/* New mount namespace group */
...
#define CLONE_NEWCGROUP	0x02000000	/* New cgroup namespace */
#define CLONE_NEWUTS		0x04000000	/* New utsname namespace */
#define CLONE_NEWIPC		0x08000000	/* New ipc namespace */
#define CLONE_NEWUSER		0x10000000	/* New user namespace */
#define CLONE_NEWPID		0x20000000	/* New pid namespace */
#define CLONE_NEWNET		0x40000000	/* New network namespace */
```

读这几行,注意两件事:

1. **每一个 `CLONE_NEW*`,对应一种 namespace**:mount(挂载点)、cgroup、UTS(主机名)、IPC(信号量/消息队列)、user(用户)、pid(进程号)、net(网络栈)。**容器所谓的"隔离",拆开来看,就是这几样。**
2. **这些标志位,远早于 Docker 诞生就在内核里了。** `CLONE_NEWNS` 是 2002 年进内核的,Docker 2013 年才出现。**Docker 没有发明 namespace,它只是把这些早就存在的标志位,组合起来用。**

> 这就像航运公司没有发明"铁皮箱",铁皮箱(标准集装箱)在码头用了几十年了。航运公司的高明,在于**把标准箱 + 起重机 + 调度规则组合成了一套好用的物流系统**。容器的"新",不在底层零件,而在"组合方式"。

### runc 干的事:把"要哪些 namespace"翻译成"clone 标志位"

如果你还觉得"容器底下一定有什么神秘机制",那我们再看一段铁证——**真正负责创建容器的程序 `runc`**(它是 Docker / containerd 底层那个"起重机"),它的源码里,有一段函数专门干"翻译"的活。看 [runc 的这段代码](../runc/libcontainer/configs/namespaces_syscall.go#L11-L33):

```go
var namespaceInfo = map[NamespaceType]int{
    NEWNET:    unix.CLONE_NEWNET,
    NEWNS:     unix.CLONE_NEWNS,
    NEWUSER:   unix.CLONE_NEWUSER,
    NEWIPC:    unix.CLONE_NEWIPC,
    NEWUTS:    unix.CLONE_NEWUTS,
    NEWPID:    unix.CLONE_NEWPID,
    NEWCGROUP: unix.CLONE_NEWCGROUP,
    NEWTIME:   unix.CLONE_NEWTIME,
}

// CloneFlags parses the container's Namespaces options to set the correct
// flags on clone, unshare. This function returns flags only for new namespaces.
func (n *Namespaces) CloneFlags() uintptr {
    var flag int
    for _, v := range *n {
        if v.Path != "" {
            continue
        }
        flag |= namespaceInfo[v.Type]
    }
    return uintptr(flag)
}
```

读这段代码,你会恍然大悟——**容器运行时,核心就干这么个翻译**:

- 左边 `namespaceInfo` 是一张表:把"用户要的 namespace 类型"(NEWNET、NEWPID…)对应到"clone 的标志位"(`unix.CLONE_NEWNET`、`unix.CLONE_NEWPID`…)。这张表,和上面内核里那几个 `#define` **一一对应**。
- `CloneFlags()` 函数:遍历容器的配置("这个容器要新建哪些 namespace"),把对应的标志位**用按位或(`|`)拼起来**,返回一个整数——这个整数,就是稍后传给 `clone`/`unshare` 系统调用的 flags 参数。

> **就这么几行。** 你以为"启动一个容器"底下是某种深不可测的虚拟化引擎,结果它就是:**读配置 → 查表 → 把几个标志位或在一起 → 传给 clone**。容器运行时最核心的隔离逻辑,浓缩在一个 `map` 和一个 for 循环里。

**这就是本章要立的第一性原理的铁证**:容器没有任何神秘力量。它的隔离,来自内核 `clone` 的几个标志位;它的资源限制,来自往 cgroup 文件里写几个数字(下一章细讲)。docker / k8s 做的所有事,都是**在这两块基石之上盖楼**。

## 关键源码精读:从一个标志位,到"一个容器"

把上面两段代码并在一起,我们就把"容器是怎么被隔离出来的"在源码层面走通了。这里逐段对应前面讲的设计。

### 1. 内核:`CLONE_NEW*` 是隔离的原子能力

回到内核的 `include/uapi/linux/sched.h`。每一个 `CLONE_NEW*` 都是一个**位**(`0x00020000`、`0x20000000`……),它们设计成可以**按位或**拼在一起——这意味着你可以**一次性**给新进程要"好几种隔离":

```c
/* 假设一个容器要同时隔离:挂载、PID、网络、UTS、IPC */
flags = CLONE_NEWNS | CLONE_NEWPID | CLONE_NEWNET | CLONE_NEWUTS | CLONE_NEWIPC;
/* 把 flags 传给 clone / clone3,新进程就同时进了这几个 namespace */
```

> **为什么设计成"标志位"、而不是"一个一个独立的系统调用"?** 因为一个容器通常要**同时**进好几个 namespace(光隔离进程表不够,还得隔离网络、挂载……)。用按位或的标志位,一次 `clone` 调用就能把它们全带上,**原子地**完成组合隔离——既高效,又避免了"先进 PID namespace、还没进 network namespace"这种中间态的不一致。**"用位图表达一组可组合的选项",是 Unix 系统调用一贯的精炼风格。**

### 2. runc:`CloneFlags()` 把"配置"翻译成"标志位"

再看 runc 那段。注意 `CloneFlags()` 里这个细节:

```go
for _, v := range *n {
    if v.Path != "" {
        continue            // ← 这个容器不是"新建",而是"加入"一个已有 namespace
    }
    flag |= namespaceInfo[v.Type]
}
```

`v.Path != ""` 这一行的意思是:有些 namespace,容器不是**新建**一份,而是**加入**一个已经存在的(比如两个容器共享同一个 network namespace,`v.Path` 就指向那个已有的 namespace 文件)。

- **新建**(`Path` 为空):要 `CLONE_NEW*`,从内核那里**现做一个**新的隔离空间。
- **加入**(`Path` 非空):不要新标志位,而是稍后用 `setns` 系统调用,**挤进**别人已有的空间。

> **这个"新建 vs 加入"的区分,正是 k8s 里 Pod 的根基**——Pod 里的几个容器,**共享**同一个 network namespace(它们不是各自新建,而是加入同一个)。这个设计在 runc 源码的这一行 `if v.Path != ""` 里,就已经埋下了伏笔。第 16 章讲 Pod 时你会再见到它。

### 3. 串起来:从一个标志位到"一个容器"

把两段合起来,一个容器被隔离出来的全过程,在源码层面是这样:

```
   用户: docker run nginx
              │
              ▼
   docker / containerd: 解析镜像、准备 rootfs、生成"容器配置"
              │
              ▼
   runc: 读配置 → CloneFlags() 算出标志位 (CLONE_NEWNS|NEWPID|NEWNET|...)
              │
              ▼
   clone(flags=...) : 内核创建新进程,顺便把它塞进这几个新 namespace
              │
              ▼
   设 cgroup、pivot_root 换根、exec nginx : 一个"容器"就这样跑起来了
```

**没有一个步骤是"虚拟化"。** 全是内核已有的系统调用:`clone`(隔离)、写 cgroup 文件(限资源)、`pivot_root`(换根)、`exec`(跑程序)。容器,就是这条调用链的产物。

> 这也是为什么本书开篇要花一整章讲"容器不是虚拟机"——**因为一旦你把它当成 VM,就会期待它有独立内核、期待隔离像 VM 那样彻底,从而在"容器逃逸""性能隔离"等问题上犯根本性的理解错误。** 把"它就是个普通进程"这个认知钉死,后面所有章节才能顺畅地展开。

## 章末小结

### 用航运比喻回顾本章

回到那片港口。这一章我们做了一件纯粹的事:**搞清楚"集装箱(容器)到底是什么"。**

答案分三层,一层比一层接近本质:

1. **它不是"每件货物配一整艘船"(虚拟机)**。VM 为了隔离,把整个操作系统复制了一遍——隔离最彻底,但贵得离谱(启动慢、占资源、冗余巨大)。对于"只要互不干扰"的绝大多数应用,这是杀鸡用牛刀。
2. **它是"所有货物装同一艘船,共享船的动力系统(内核),只用铁皮箱(namespace)隔开、用配额(cgroup)限量"**。容器一点都没虚拟化——它没有造虚拟硬件、没有起第二个内核,它只是**把内核早就有的两个能力,套到一个普通进程身上**。所以它又轻又快又省。
3. **最根本的真相:容器就是一个普通的 Linux 进程**。宿主上 `ps` 能看到它、`uname` 和宿主一模一样、它就是一次 `exec`。它的隔离,来自 `clone` 的几个标志位(`CLONE_NEW*`);它的限额,来自 cgroup。**docker / k8s 的全部精妙,都在"怎么组合这两块基石",而不在底层有什么新发明。**

### 本章在全书主线中的位置

记住全书的二分法:**打包隔离 vs 调度编排。**

这一章,我们把**第一块基石的根基**立住了:

- **"容器 = 普通进程 + namespace + cgroup"** 是全书的地基。后面 docker 半本(第 1~4 篇),讲的都是"怎么把这三件事**组合、封装、产品化**":把隔离做成标准镜像(第 2 篇)、做成好用的 docker/containerd(第 3 篇)、给容器之间修路(第 4 篇网络)。
- 而 **"共享内核"这个权衡**,会在第 22 章(安全)和第 23 章(容器边界)回头收束——那时你会彻底明白,容器隔离的边界到底画在哪里。

后面每一章,你要做的只是不断回到这两个锚,问一句:"这章是在**加深对某块基石的理解**,还是在**往上盖楼(封装/调度)**?"

### 五个"为什么"清单

如果你只能记五件事,记这五件:

1. **为什么需要容器**:裸进程没有隔离(互相能看能改、没有秘密、没有限额),没法安全地在同一台机器上跑很多互不信任的应用。
2. **为什么不用虚拟机**:VM 为了隔离复制了整个操作系统,代价太重(启动慢、占资源、冗余大),对绝大多数"只需互不干扰"的应用是浪费。
3. **容器凭什么又轻又隔离**:它**根本不虚拟化**——所有容器共享同一个内核,隔离靠内核已有的 namespace,限资源靠 cgroup。不造虚拟硬件、不起第二个内核,所以轻快省。
4. **容器到底是什么**:就是一个普通的 Linux 进程。`ps`、`uname`、`/proc` 都会出卖它。它的隔离来自 `clone` 的几个 `CLONE_NEW*` 标志位,根本不是什么新机制。
5. **共享内核的代价**:隔离不如 VM 彻底,一个内核漏洞可能波及所有容器。这是"轻"的代价,第 22、23 章细讲。

### 想继续深入,该往哪钻

- **亲手验证"容器就是个进程"**:在跑着容器的机器上,`docker run -d nginx` 后,`ps aux | grep nginx`——你会看到它就在宿主的进程表里。再 `nsenter` 进去看它的 namespace(下一章的主角)。
- **看 namespace 在内核里的定义**:kernel 的 [include/uapi/linux/sched.h](https://github.com/torvalds/linux/blob/master/include/uapi/linux/sched.h) 里的 `CLONE_NEW*`,以及各 namespace 的实现(`kernel/pid_namespace.c`、`net/core/net_namespace.c` 等)。
- **看 runc 怎么用这些标志位**:[runc/libcontainer/configs/namespaces_syscall.go](../runc/libcontainer/configs/namespaces_syscall.go#L24-L33)(本章引用的 `CloneFlags()`),顺着它往下,看 `clone` 是在哪里被真正调用的。
- **想理解"共享内核的代价"**:可以先翻到第 23 章《容器的边界》,看一个内核漏洞是怎么让所有容器沦陷的——这会让你真正理解本章那个"权衡"的分量。
